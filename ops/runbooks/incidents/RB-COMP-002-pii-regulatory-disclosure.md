# RB-COMP-002: PII 漏えい規制報告・本人通知

本 Runbook は PII 漏えい確定後の規制報告（個人情報保護委員会 / GDPR 監督機関）と本人通知を定める。
[`RB-SEC-005-pii-leak-detection.md`](RB-SEC-005-pii-leak-detection.md) §「影響範囲確定」完了後に起動する。
NFR-E-SIR-002 / NFR-G-CLS-001 / NFR-G-CLS-002 に対応する。
**速報 72 時間 / 確報 30 日は法定期限であり遅延不可**。

## 1. 前提条件

- 実行者は `legal-officer` または CPO で、個人情報保護委員会への報告権限を保持。
- 必要ツール: 個人情報保護委員会報告システムへのアクセス（`https://www.ppc.go.jp/personalinfo/legal/leakAction/`）/ メール送信権限 / 書面通知体制。
- [`RB-SEC-005`](RB-SEC-005-pii-leak-detection.md) §Step 4 が完了し、影響ユーザーリスト（`/tmp/affected_users.txt`）と漏えいデータ種別が確定済み。
- CPO・法務担当との連絡経路が確立済み（[`oncall/contacts.md`](../../oncall/contacts.md)）。
- GDPR 対象の場合: Lead supervisor または各国 DPA の連絡先が手元にあること。
- 通知文書テンプレートとプライバシーポリシーが手元にあること。

## 2. 対象事象

本 Runbook は [`RB-SEC-005-pii-leak-detection.md`](RB-SEC-005-pii-leak-detection.md) の **Step 4（影響範囲確定）** が完了し、個人情報の漏えいが確定した時点で起動する。

個人情報保護法（令和 4 年施行）第 26 条に基づき、以下の要件を満たす場合は **報告義務** が発生:

- 要配慮個人情報が含まれる場合
- 不正アクセスに起因する場合
- 漏えい件数が 1,000 件を超える場合
- 財産的被害が生じるおそれがある場合
- 上記に該当しない場合も、100 件以上は自主報告を推奨

**漏えい確定の起算点**: CPO（個人情報保護責任者）が漏えいを認識した時刻。

タイムライン:

```
T+0h   漏えい確定（CPO 認識）
T+24h  [社内] 情シスマネージャ・監査担当・法務への通知完了
T+72h  [法定] 個人情報保護委員会への速報提出（締切）
T+30d  [法定] 個人情報保護委員会への確報提出（締切）
       [法定] 影響本人への通知（速やかに、確報前まで）
```

## 3. 初動手順（5 分以内）

最初の 5 分で報告義務の判定を完了し、72h タイムラインを起動する。

```bash
# RB-SEC-005 で確定した影響件数を確認
wc -l /tmp/affected_users.txt
# 漏えいデータ種別の確認
cat /tmp/leak-data-types.txt  # RB-SEC-005 §Step 4 で作成
```

ステークホルダー通知（必須、5 分以内）:

- CPO に電話、法務担当に電話、CTO に Slack DM。
- Slack `#incident-sev1` に「個人情報保護委員会報告手続開始」を投稿。
- 72h タイマーを Slack リマインダで設定。

## 4. 原因特定手順

確報（30 日確報）作成のため、以下を根拠資料として準備:

- 漏えいデータの最終確定件数・種別。
- 漏えい経路・原因の技術詳細（[`RB-SEC-005`](RB-SEC-005-pii-leak-detection.md) §「原因特定手順」の結果を引き継ぎ）。
- 二次被害の発生有無・対応履歴（被害者からの問合せ件数、なりすまし詐欺の報告等）。
- 本人通知の実施記録（通知日時・手段・通知数）。
- 再発防止措置の実施計画・実施結果。

エスカレーション: 法務判断が必要な場合は外部法律顧問へ即時依頼。GDPR 対象データが含まれる場合は EU 法務専門家へ。

## 5. 復旧手順

### Step 1: 報告義務の判定（〜1 時間）

法務担当に漏えい内容の詳細を連絡。以下のチェックリストで報告義務を確認:

| チェック項目 | Yes/No | 法的根拠 |
|---|---|---|
| 要配慮個人情報（健康・信条・犯罪歴等）が含まれる | → Yes で即報告義務 | 個人情報保護法 26 条 |
| 不正アクセスが原因と疑われる | → Yes で即報告義務 | 同上 |
| 漏えい件数 ≥ 1,000 件 | → Yes で即報告義務 | 同上 |
| 財産的被害（なりすまし詐欺等）の可能性 | → Yes で即報告義務 | 同上 |
| GDPR 対象の EU 居住者データが含まれる | → 72h 以内に監督機関へ | GDPR 第 33 条 |

### Step 2: 個人情報保護委員会 速報（T+72h 以内）

報告フォームにアクセス: `https://www.ppc.go.jp/personalinfo/legal/leakAction/` → 漏えい等報告。

速報に必要な最低限の情報を記入（確定していない項目は「調査中」で可）:

```
【速報 記載項目】
1. 事業者名・連絡先
2. 漏えい等が発生した個人情報の種類・件数（概算可）
3. 漏えい等が発生した原因（概要）
4. 二次被害の可能性・有無
5. 本人への通知の実施状況
6. 公表の実施状況
7. 再発防止措置の検討状況
```

提出後、受付番号を記録し、法務と CPO に共有する。

### Step 3: 影響本人への通知（速やかに、確報前まで）

通知対象者リストを確定する（`/tmp/affected_users.txt` を参照）。
通知文書テンプレートを使用:

```
件名: 個人情報漏えいに関するお知らせ

[氏名] 様

この度、弊社のサービスにおいて、お客様の個人情報が漏えいした
可能性があることが判明いたしました。

【漏えいした情報の種類】
<種類・件数>

【漏えいの経緯】
<概要>

【弊社の対応】
<封じ込め措置・再発防止策>

【お客様へのお願い】
<不審なメール・連絡への注意喚起等>

【お問い合わせ先】
<privacy@org.example>
```

通知手段: メール（確認可能な場合）/ 書面（重大な場合）/ 公表（個別通知困難な場合）。

### Step 4: GDPR 対象データの場合

GDPR 対象の EU 居住者データが含まれる場合:

- Lead supervisor がいない場合は Ireland DPC を窓口に GDPR 第 33 条に基づき 72h 以内に報告。
- 各国 DPA（Data Protection Authority）への報告を確認。
- 影響者への通知は GDPR 第 34 条に基づき「不当な遅滞なく」実施。

### Step 5: 確報（T+30 日以内）

個人情報保護委員会への確報提出（T+30 日以内）に以下を含める:

- 漏えいの内容（最終確定件数・種別）
- 二次被害の発生状況
- 漏えい原因
- 再発防止策（実施済み + 実施予定）
- 本人通知の実施結果（通知日時・件数・手段）

## 6. 検証手順

対応完了の判定基準:

- 速報受付番号が記録されている（個人情報保護委員会から発行）。
- 速報提出が T+72h 以内に完了（タイムスタンプで確認）。
- 確報提出が T+30 日以内に完了。
- 影響本人への通知が確報提出前に完了（通知率 ≥ 95%、未通知の理由は記録）。
- 個人情報保護委員会の改善指導があった場合、対応計画が期限内に提出済み。
- GDPR 対象の場合、各国 DPA への報告が完了。
- ポストモーテム最終版（30 日確報に基づく）が `postmortems/<YYYY-MM-DD>-RB-SEC-005-final.md` に存在。
- 二次被害（なりすまし詐欺等）の発生件数が記録されている。

## 7. 予防策

- 個人情報保護委員会への確報提出（T+30 日以内）。
- 確報に基づくポストモーテム最終版の作成。
- 個人情報保護委員会の改善指導がある場合は対応計画を策定し、期限内に報告する。
- プライバシーポリシーの改訂が必要な場合は法務と協議し、公開する。
- 四半期 PII スキャン・検知精度の向上（`infra/security/falco/rules/pii.yaml` 更新）。
- 影響者への補償（クレジットモニタリングサービス提供等）を法務判断で検討。
- 月次 Chaos Drill 対象に「PII 漏えい table-top 規制報告」シナリオを追加。
