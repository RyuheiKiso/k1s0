# postmortems — ポストモーテム記録

本ディレクトリはインシデント対応後のポストモーテム記録を保管する。
`docs-postmortem` Skill を経由して起票する（タイプ E、Blameless 原則）。

## 配置

```text
postmortems/
├── README.md                                    # 本ファイル
└── <YYYY-MM-DD>-RB-<カテゴリ>-<NNN>-<slug>.md   # ポストモーテム本体
```

## 命名規則

`<YYYY-MM-DD>-RB-<カテゴリ>-<NNN>-<slug>.md`

例:
- `2026-04-30-RB-DB-002-postgres-oom-kill.md`
- `2026-05-15-RB-SEC-002-cert-renewal-failure.md`
- `2026-06-01-RB-MSG-001-kafka-pvc-full.md`

## 起票期限

[`docs/04_概要設計/55_運用ライフサイクル方式設計/02_インシデント対応方式.md`](../../../docs/04_概要設計/55_運用ライフサイクル方式設計/02_インシデント対応方式.md) §「事後対応」に従う:

| Severity | 起票期限 |
|------|------|
| SEV1 | 24 時間以内 |
| SEV2 | 72 時間以内 |
| SEV3 | 1 週間以内 |

## 必須 10 セクション（`docs-postmortem` Skill 規約）

1. インシデント概要
2. タイムライン
3. 影響範囲
4. 検出
5. 根本原因
6. 対応経緯
7. うまくいったこと
8. うまくいかなかったこと
9. アクションアイテム（Jira チケット紐付け必須）
10. 教訓と Runbook 改善 PR

## Blameless 原則

- 個人の責任追及ではなくシステム改善に焦点を当てる。
- 「なぜミスを犯したのか」ではなく「なぜシステムがミスを許したか」を問う。
- 記述に個人攻撃が含まれていないか起票時にレビュー。

## レビューと承認

- 起票後 48 時間以内に起案者または EM がレビュー。
- アクションアイテムは Jira に同期し、四半期 Product Council で進捗確認。
- 確報（30 日確報、外部報告必要時）はセキュリティ・法務担当が共著。
