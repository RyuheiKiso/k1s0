## 6. framework 提供機能（方針: 汎用性重視）

### framework 再利用の境界（依存グラフ方針）（MUST）

目的: 個別機能チームが「業務ロジックに集中」できるようにしつつ、共通化が肥大化/破綻しない境界を固定する。

#### 何を framework crate に入れるか（固定）

- 横断関心事（observability / logging / tracing / metrics / error / config / auth / retry/timeout など）
- 外部I/Oの共通化（DB接続、共通クライアント、K8s/環境差吸収、共通middleware など）
- 規約を強制する仕組み（例: 設定読み込みの統一、環境変数禁止の代替、標準ログ出力の統一）
- 「業務ドメインに依存しない」抽象（traits/ports、共通DTO、共通ユーティリティ）

#### framework に入れてはいけないもの（固定）

- 特定の feature の業務ドメインロジック（Entity/ValueObject/Usecase）
- 特定サービス固有の要件（例: 画面仕様に引きずられた集約、特定顧客向けの分岐）
- 特定の DB スキーマ前提のロジック（framework 共通テーブルは除く）

#### 依存方向（固定）

- 依存は一方向のみを許可する
	- feature（個別サービス） → framework（共通）: 許可
	- framework → feature: 禁止
- サービス内の層と framework の関係
	- `src/domain/` は原則 framework crate に依存しない（外部I/Oやフレームワーク依存を持ち込まない）
	- `src/application/` / `src/infrastructure/` / `src/presentation/` は、必要に応じて framework crate へ依存してよい
	- 例外が必要な場合は ADR で理由と代替案を明記する

#### 公開 API の安定性（固定）

- framework crate の「公開 API」は破壊的変更を原則禁止する
	- Rust: `pub` で公開される型/関数/trait は SemVer の互換性対象とする
	- `pub(crate)` や非公開モジュールは互換性対象外（内部実装）
- 非推奨化の段階移行を基本とする
	- 互換性維持のため、削除ではなく `deprecated` 化 → 移行期間 →（例外として MAJOR で削除）
	- 移行手順は `UPGRADE.md` またはリリースノートに必ず記載する
- 実験的 API は明示する
	- `experimental` feature flag 等で opt-in とし、既定で利用しない
	- 実験的 API の互換性は保証しない（ただし意図をドキュメント化する）

### framework 強化方針（提供する標準機能の固定）

目的: 「各サービスが毎回実装して事故る」領域を framework が肩代わりし、“ブレの余地”を減らす。

MUST（必須）

- feature サービスは、原則として framework が提供する共通実装を利用する（例外は ADR で理由と代替案を明記する）。
- templates は、次の標準機能を“最初から使っている状態”を既定とする（導入作業を個別チームに残さない）。

提供する標準機能（固定）

1. 設定/秘密情報ランタイムの標準化
	- YAML（非機密）+ secrets（ファイル）+ DB（動的設定）を前提に、共通ローダ・バリデーション・マージ規則（優先順位）を提供する。
	- `*_file` を解決して読み込む仕組みを framework が持ち、アプリが独自に読む実装を禁止する（例外は ADR）。

2. 観測性のデフォルト完備
	- 構造化ログ（JSON）/トレース伝播/メトリクス/リクエストID相関を、共通ミドルウェアで初期化・強制する。
	- ログ/メトリクスの必須フィールドを固定し、「入れ忘れ」「形式違い」を起こさない。

3. gRPC クライアント/サーバ共通基盤
	- deadline 必須、retry 原則 0、エラー分類、OTel 計装、メタデータ付与（trace/tenant 等）を“作法込み”で提供する。
	- サービス間通信の標準値（timeout/下限/上限）を framework の共通ビルダーで強制し、値の逸脱は設定バリデーション/起動時失敗で早期に落とす。
	- metadata（例: trace/tenant）付与は interceptor で自動化し、個別サービスが“付け忘れ/形式違い”を起こせない形にする。
	- 例外の入口（opt-in + ADR）を用意し、例外は「見える」形（コード上の明示 + lint/CI の検知）で管理する。

4. エラーモデル統一
	- 内部のエラー分類（入力不備/依存障害/一時障害/権限/不変条件違反）と、外部表現（gRPC status / HTTP problem details）を統一する。
	- `error_code` を“安定した識別子”として必須化し、運用で「見れば分かる」を実現する。
	- 内部分類 → `error_code` → REST（problem details）/ gRPC（status + metadata）への変換は framework が自動で行い、サービスが個別に変換ロジックを持たない。
	- 変換時に `trace_id` 等の相関情報を必ず付与し、調査の入口（ログ/トレース）へ確実につながる状態を既定にする。

5. DB アクセス標準（Rust/Go）
	- コネクション管理、トランザクション境界、リトライ可否、エラー変換、クエリ計測（遅延/件数）を共通化する。
	- マイグレーション実行も“入口”を固定し（CLI/scripts から呼べる runner を共通提供）、サービスごとの作法の分岐を作らない。

6. 認証/認可の共通部品
	- JWT/OIDC 検証、権限制御（ポリシー評価）、監査ログ（誰が何をしたか）を framework が提供する。
	- 各サービスが独自に認証/認可を実装しない（例外は ADR）。

7. 入力バリデーション＆スキーマ駆動
	- API 境界でのバリデーション、エラーの返し方、OpenAPI/proto と整合するルールを提供する。
	- “動くけど曖昧”を減らすため、テンプレはバリデーションの入口を必ず持つ。

8. 依存先呼び出しのガードレール（レジリエンス）
	- タイムアウト上限/下限、同時実行制限、バルクヘッド、必要時のみのサーキットブレーカを共通ユーティリティとして提供する。
	- 障害時の雪崩（retry 増幅/無限待ち/輻輳）を“設計で入りにくい形”にする。

### フロントエンド（React）framework の提供範囲（固定）

目的: 個別機能チームが「画面の中身」以外（ナビゲーション/レイアウト/デザイン/権限制御/設定読込/観測）を再実装せずに済む状態を既定にする。

MUST（必須ルール）

- UI コンポーネントは MUI（Material UI）を標準とする（独自 UI ライブラリの乱立を防ぐ）
- Header / Footer / Menu（サイドバー/ドロワー含む）は framework が提供する
- 画面遷移（routes/menus/flows）は「設定で制御」できることを既定とする（ノーコード化）
	- ここでのノーコード化は「画面の配線（ルート/メニュー/遷移条件）をコードから追い出す」ことを意味する
	- 画面コンポーネント自体（ドメインに依存する UI）は個別機能チームが実装する
- ナビゲーション設定は `config/{env}.yaml`（非機密）を正本とし、スキーマ/バリデーションは framework が提供する
	- 権限（permission）や feature flag による表示/遷移制御も設定で表現できること

- API クライアントは framework が提供する共通実装を利用する（生成 SDK + 認証付与 + retry/timeout + 計測を一体化）
	- 画面/機能ごとに fetch/axios の流儀が増えない
	- エラー表現（problem details 相当）と `error_code` の扱いを統一する
	- problem details を“そのままJSONで出す”のではなく、標準 UI（エラー表示/リトライ導線/trace_id 表示）として提供し、画面ごとの差を消す

- デザイン/UX の標準（テーマ/レイアウト/フォーム/エラー表示/通知/ダイアログ）を framework の共通部品として提供する
	- 画面ごとの“独自流儀”を防ぎ、レビュー観点と実装コストを固定する

SHOULD（推奨）

- 共通テーマ（色/タイポ/spacing）を framework の MUI theme として提供し、アプリ側での上書きは最小限にする
- 画面追加の最短手順を「1画面=1モジュール + 設定追記」の形に寄せ、CLI で雛形生成できるようにする
