# 手順書：開発基盤チームが新機能を安全に開発する（k1s0）

対象読者：開発基盤チーム（framework / templates / CLI / CI ゲート整備）

目的：新機能追加・規約追加・テンプレ更新・アップグレード支援の変更を、**後方互換性と運用安全性を崩さず**にリリースできるよう、判断と作業を手順化する。

本手順書は `work/構想/` の方針（MUST/SHOULD）を前提に、実務のチェックリストとして具体化したもの。

---

## 0. この手順書のスコープ

### 対象（開発基盤チームの変更）

- `framework/` の共通 crate / 共通サービス雛形の追加・変更
- `templates/` の雛形追加・変更
- `CLI/`（`k1s0 init/new-feature/lint/upgrade`、generator、schemas）の追加・変更
- CI の必須チェック定義（lint / contract / 生成一致 / upgrade check など）
- 共通テーブル定義（`framework/database/table/*.sql`）や所有サービスのマイグレーション

### 対象外（この手順書では扱わない）

- 個別機能チームが `feature/` 配下で行う、業務ロジック実装の手順
- Kubernetes クラスタ等の運用構築（Collector/Trace UI/Grafana 等の実環境整備）

---

## 1. 安全の定義（k1s0 の前提）

「安全」とは、次を同時に満たすこと。

- **互換性**：SemVer の定義に沿い、MAJOR を濫用しない（段階移行が基本）
- **境界**：CLI が更新できる領域（managed）と、業務コード領域（protected）が壊れない
- **検知**：規約はドキュメントでなく、lint/CI で落ちる（逸脱が main に入らない）
- **観測**：障害時に `error_code` と `trace_id` で一次切り分けできる
- **秘密**：secrets を YAML/DB/環境変数に直書きしない（ファイル参照を既定）
- **契約**：proto/OpenAPI の破壊が CI で止まる（API First を運用に落とす）

---

## 2. 変更タイプ分類（最初にやること）

まず変更を分類する（複数に該当することが多い）。

A. framework crate の追加・変更（config/error/observability/grpc-client 等）
B. framework 共通サービス（auth/config/endpoint）や共通テーブルの変更
C. templates の構造・必須ファイル・README 規約の変更
D. CLI（生成/lint/upgrade/差分適用/出力/終了コード）の変更
E. 規約（MUST/SHOULD）そのものの変更、または例外（allowlist）の追加
F. 契約管理（buf/OpenAPI diff/生成一致）の導線・必須化の変更

分類したら、以降のセクションの該当チェックリストを全部通す。

---

## 3. 事前判断（ADR と SemVer）

### 3.1 ADR が必須な変更

次のいずれかに該当する場合、ADR を作る（例外の入口を増やさない）。

- MUST ルールの追加・変更（例：環境変数禁止の例外追加、リトライ例外の新ルール）
- 互換性ポリシーに影響（SemVer の判断が割れる、段階移行が必要）
- Service Mesh / discovery / mTLS 等、運用境界が変わる
- 既存の標準実装を置き換える（観測、エラー、設定、通信）
- lint が新しく落とす対象を増やす（既存プロジェクトに影響）

ADR には最低限：目的、背景、選択肢、決定、影響範囲、移行計画（段階移行）、ロールバックを書く。

### 3.2 SemVer（リポジトリ単一バージョン）判定

k1s0 は **リポジトリ全体で単一バージョン**を持つ（CLI / generator / templates / framework まとめて整合させる）。

- PATCH：後方互換な修正（バグ修正、テンプレ安全修正、互換維持の依存更新）
- MINOR：後方互換な機能追加（任意フィールド追加、新コマンド追加、deprecate 導入）
- MAJOR：破壊的変更（契約破壊、公開 API 破壊、テンプレ構造破壊、DB 破壊）

破壊的変更は原則禁止。必要なら「追加→移行→切替→削除」の段階移行を基本にする。

MAJOR を行う場合は同一 PR で必須：ADR、`UPGRADE.md`（影響と移行手順）、自動/半自動の upgrade 手段（check/パッチ/変換）。

## 4. 実装フロー（必ずこの順で進める）

### 4.1 変更仕様を「検知」まで落とす

実装に入る前に、必ず「どうやって逸脱を落とすか」を決める。

- どの規約を守らせたいか（MUST/SHOULD のどれか）
- その規約はどこで固定するか
	- templates（置き場/必須ファイル/起動引数の明示）
	- framework（API 形状で作法を強制：deadline 必須、retry 既定0、error mapping 等）
	- CLI lint（静的検査：必須ファイル、禁止事項、例外の可視化）
	- CI（マージゲート：lint/contract/生成一致/upgrade check）

「ドキュメントに書いた」だけで終わる変更は禁止。

### 4.2 先にテンプレ・スキーマ・lint を更新する

安全の既定は、サービス実装ではなく「雛形」と「検査」に寄せる。

- まず templates に “正しい置き場と最小骨格” を反映
- 次に CLI schemas / manifest schema を更新（必要なら）
- 次に lint ルールを更新（落とす/案内する）

### 4.3 framework 実装（作法を API 形状で固定）

- 設定：環境変数を使わず、YAML + secrets(file) + DB(setting) の優先順位を守る
- 秘密：値そのものを持たず参照のみ（`*_file`）
- 観測：ログ/トレース/メトリクスの必須出力が「入れ忘れない」入口を提供
- エラー：`error_code` と `trace_id` が必ず外へ出る（変換は framework に集約）
- 通信：gRPC deadline 必須、retry 既定 0、例外は opt-in + ADR + lint 検知

### 4.4 upgrade 支援（壊さず更新できる境界の維持）

テンプレ変更が入る場合、必ず upgrade の観点で検討する。

- `.k1s0/manifest.json` により managed/protected 境界が保持されるか
- managed に含めるべき変更（deploy/config/openapi/CI 等）か
- protected に影響しないか（`src/domain/` / `src/application/` は原則非管理）
- 境界領域（`src/main.rs` / `src/presentation` / `src/infrastructure`）は原則 diff 提示に留められるか

## 5. チェックリスト（必須）

### 5.1 templates（生成物の安全性）

- [ ] ディレクトリ規約に従う（`framework/` / `feature/` / `CLI/` の責務分離）
- [ ] 生成直後に `k1s0 lint` が通る前提である
- [ ] backend は `src/{application,domain,infrastructure,presentation}` を必ず持つ
- [ ] `config/{default,dev,stg,prod}.yaml` がある（機密値直書き無し、`*_file` 参照のみ）
- [ ] K8s 対象なら `deploy/base` と `deploy/overlays/{dev,stg,prod}` がある
- [ ] 契約の正本がある
	- gRPC：`proto/` + `buf.yaml`/`buf.lock`
	- REST：`openapi/openapi.yaml`
- [ ] README 規約（見出し順）を満たす
- [ ] dev scripts（dev-up/down/seed/check）がテンプレ方針に沿って同梱される（該当する場合）

### 5.2 設定（config）と秘密情報（secrets）

- [ ] 環境変数に依存しない（禁止：`envFrom` / `secretKeyRef` / `std::env` / `os.Getenv`）
- [ ] 優先順位：CLI 引数（参照先）→ YAML → DB（`fw_m_setting`）
- [ ] 機密値はファイルとして配布し、YAML/DB には参照のみ（`*_file`）
- [ ] 既定パスが明示される（例：`/etc/k1s0/config/{env}.yaml`、`/var/run/secrets/k1s0/`）
- [ ] config-service 障害時挙動（A/B/C）のどれを採用するかが明記される

### 5.3 観測性（observability）

- [ ] ログ：JSON 1行、必須フィールド（`timestamp`, `level`, `service_name`, `env`, `trace_id`, `span_id`, `message`）
- [ ] メトリクス：命名 `k1s0.<service>.<component>.<metric>`、最低限の成功/失敗/レイテンシが出る
- [ ] トレース：W3C Trace Context 伝播、外部依存（DB/Redis/他サービス）で span を切る
- [ ] `scripts/dev-check.ps1` の合格条件に「観測が届く」が含まれる

### 5.4 エラーモデル（運用が“見れば分かる”）

- [ ] 外部に返す失敗は `error_code` を必ず付与（安定識別子、変更しない）
- [ ] REST：problem+json（RFC7807互換）で `error_code` と `trace_id` を必須
- [ ] gRPC：canonical status code + `error_code` を metadata 等で付与（方式は framework で統一）
- [ ] 変換（内部分類→外部表現）は framework に集約し、サービス独自変換を持たない

### 5.5 gRPC クライアント/サーバ（通信事故を作法で潰す）

- [ ] deadline（timeout）必須（無制限禁止）
- [ ] timeout の下限/上限を固定し、逸脱は設定バリデーション/起動時失敗で落ちる
- [ ] retry 既定 0（原則禁止）
- [ ] retry 例外は opt-in のみ。ADR 参照＋回数/バックオフ/ジッター/最大待ちが必須
- [ ] 例外（retry opt-in 等）は lint/CI が検知できる
- [ ] discovery は K8s DNS + Service 名（論理名）を正本とし、環境差は config に閉じ込める

### 5.6 契約管理（API First の運用）

- [ ] 正本：gRPC は `proto/`、REST は `openapi/openapi.yaml`
- [ ] `buf lint` / `buf breaking` が CI 必須（比較は `main` または直近タグ）
- [ ] OpenAPI diff が CI 必須（破壊的変更は失敗）
- [ ] 生成物（`gen/`, `openapi/gen/`）は Git 管理外でも、生成一致チェックでズレを検知できる

### 5.7 共通 DB（所有者・変更手順）

共通テーブルを変更する場合は必ず同一 PR で次を更新する。

- [ ] `framework/database/table/<table>.sql`（DDL 正本）
- [ ] 所有サービスの `migrations/`（適用用正本）
- [ ] 所有サービスの `README.md`（変更点、影響、ロールバック）

破壊的変更（列削除・型変更等）は原則禁止。必要なら段階移行＋ADR。

### 5.8 CLI / lint / upgrade

- [ ] `k1s0 init/new-feature/lint/upgrade` の出口（終了コード/出力）が標準化されている
- [ ] 失敗時は「原因/対象/次アクション」を必ず出す
- [ ] `.k1s0/manifest.json` は CLI 管理で、人手編集不要（かつ不要な手編集を前提にしない）
- [ ] `upgrade --check` で影響範囲と衝突が一覧化される
- [ ] `upgrade` は managed のみ更新し、衝突は明示して停止できる
- [ ] `migrations/` は「追加のみ」が原則（適用済みの改変は禁止）

## 6. PR 手順（テンプレ）

### 6.1 PR を切る前（設計）

- [ ] 変更タイプ分類（A〜F）
- [ ] ADR 必要性判断（必要なら先に書く）
- [ ] SemVer 判定（PATCH/MINOR/MAJOR）
- [ ] upgrade 影響の有無（managed/protected/boundary のどこが変わるか）

### 6.2 実装（順序固定）

1. templates（置き場/必須ファイル/README）
2. schemas（manifest 等）
3. lint/CI（落とす/案内する）
4. framework（共通実装）
5. upgrade（差分提示・限定適用）
6. ドキュメント更新（docs と work の整理。確定事項は docs へ）

### 6.3 検証（最低限）

- [ ] `k1s0 lint`（または同等の `scripts/lint.ps1`）が通る
- [ ] 契約：`buf lint` / `buf breaking` / OpenAPI diff が想定通りに動く
- [ ] 生成一致：CI 相当の生成手順で差分が出ない
- [ ] dev-check：ヘルス/依存疎通/観測の合格条件が満たせる
- [ ] upgrade --check：影響範囲が説明できる形で出る（衝突時に止まれる）

### 6.4 リリース（破壊を入れない）

- [ ] `k1s0-version.txt`（単一バージョン）を更新
- [ ] 変更点（MINOR/PATCH）は release note 相当を残す（`UPGRADE.md` または docs）
- [ ] deprecate を入れた場合は移行期限と置換手段（自動/半自動）を明記

## 7. よくある事故と防止策（短縮版）

- 事故：テンプレ更新で既存サービスが壊れる
	- 防止：managed/protected を守り、`upgrade --check` で影響を見せてから適用

- 事故：secrets が YAML/環境変数に混入
	- 防止：`*_file` 参照のみ許可＋lint で検知

- 事故：無限待ち/リトライ増幅で障害が拡大
	- 防止：deadline 必須、retry 既定0、例外は opt-in + ADR + lint

- 事故：error 表現がバラつき、運用が詰む
	- 防止：`error_code` と `trace_id` を framework が強制し、変換を集約

- 事故：proto/OpenAPI の破壊が混入
	- 防止：buf breaking / openapi diff を CI 必須にし、main へ入れない
