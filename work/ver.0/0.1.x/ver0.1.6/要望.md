# 要望

バックエンドマイクロサービス間でドメインイベントを疎結合に発行・購読したい。
注文確定→在庫引当、ユーザー登録→通知送信など、サービスをまたぐビジネスプロセスを、直接的なサービス間呼び出しではなくイベント駆動で実現したい。

## 回答（実装方針）

### 概要

1. **k1s0-domain-event（Rust crate）**: ドメインイベントの定義・発行・購読の共通基盤
2. **k1s0-domain-event（Go module）**: Go 版の同等実装

### 技術選定

| 機能 | Rust | Go | 理由 |
|------|------|----|------|
| イベント定義 | trait `DomainEvent` | interface `DomainEvent` | 型安全なイベント契約 |
| インメモリバス | tokio::broadcast | channel | テスト・単体サービス用 |
| 永続化（Outbox） | k1s0-db 連携 | k1s0-db 連携 | トランザクション保証 |
| 外部ブローカー | アダプタ（NATS/Kafka/Redis Streams） | 同左 | 実行環境に応じて選択 |

### 設計方針

- **Outbox パターン**: ドメインイベントをビジネストランザクションと同一トランザクションで `outbox` テーブルに書き込み、リレーがブローカーへ転送
- **At-least-once delivery**: 重複配信を許容し、購読側でべき等処理を行う
- **Clean Architecture 準拠**: domain 層でイベント型を定義、infrastructure 層でブローカー実装

## 機能 1: ドメインイベント定義

### Rust 版 (`k1s0-domain-event`)

```rust
use k1s0_domain_event::{DomainEvent, EventMetadata};
use serde::{Serialize, Deserialize};
use chrono::{DateTime, Utc};

/// ドメインイベント trait
pub trait DomainEvent: Serialize + Send + Sync + 'static {
    /// イベントタイプ（例: "order.confirmed"）
    fn event_type(&self) -> &str;

    /// 集約ID（イベント発生元エンティティの識別子）
    fn aggregate_id(&self) -> &str;

    /// 集約タイプ（例: "Order"）
    fn aggregate_type(&self) -> &str;

    /// イベントバージョン（スキーマバージョニング用）
    fn version(&self) -> u32 { 1 }
}

/// イベントメタデータ（自動付与）
#[derive(Debug, Clone, Serialize, Deserialize)]
pub struct EventMetadata {
    pub event_id: String,           // UUID v7
    pub timestamp: DateTime<Utc>,
    pub correlation_id: Option<String>,
    pub causation_id: Option<String>,
    pub source_service: String,
    pub trace_id: Option<String>,   // k1s0-observability 連携
}

/// エンベロープ（イベント + メタデータ）
#[derive(Debug, Clone, Serialize, Deserialize)]
pub struct EventEnvelope<E: DomainEvent> {
    pub metadata: EventMetadata,
    pub payload: E,
}
```

### Go 版 (`k1s0-domain-event`)

```go

// EventMetadata はイベントに自動付与されるメタデータ

    EventID       string     `json:"event_id"`
    Timestamp     time.Time  `json:"timestamp"`

    CausationID   string     `json:"causation_id,omitempty"`

type EventEnvelope[E DomainEvent] struct {
    Metadata EventMetadata `json:"metadata"`

}
```

## 機能 2: イベント発行（Publisher）

### Rust 版

```rust
use k1s0_domain_event::{EventPublisher, DomainEvent, PublishError};

/// イベント発行インターフェース
#[async_trait]
pub trait EventPublisher: Send + Sync {
    /// 単一イベントを発行
    async fn publish<E: DomainEvent>(&self, event: E) -> Result<(), PublishError>;

    /// 複数イベントを一括発行
    async fn publish_batch<E: DomainEvent>(&self, events: Vec<E>) -> Result<(), PublishError>;
}

// 使用例: 注文確定ユースケース
pub struct ConfirmOrderUseCase {
    order_repo: Arc<dyn OrderRepository>,
    publisher: Arc<dyn EventPublisher>,
}

impl ConfirmOrderUseCase {
    pub async fn execute(&self, order_id: &OrderId) -> Result<(), AppError> {
        let order = self.order_repo.find_by_id(order_id).await?;
        let event = order.confirm()?; // OrderConfirmed イベントを生成

        // ビジネスロジックとイベント発行
        self.order_repo.save(&order).await?;
        self.publisher.publish(event).await?;

        Ok(())
    }
}
```

## 機能 3: イベント購読（Subscriber）

### Rust 版

```rust
use k1s0_domain_event::{EventSubscriber, EventHandler, EventEnvelope, SubscribeError};

/// イベントハンドラ trait
#[async_trait]
pub trait EventHandler<E: DomainEvent>: Send + Sync {
    async fn handle(&self, envelope: EventEnvelope<E>) -> Result<(), HandlerError>;
}

/// イベント購読インターフェース
#[async_trait]
pub trait EventSubscriber: Send + Sync {
    /// 特定イベントタイプを購読
    async fn subscribe<E, H>(
        &self,
        event_type: &str,
        handler: H,
    ) -> Result<SubscriptionHandle, SubscribeError>
    where
        E: DomainEvent + for<'de> Deserialize<'de>,
        H: EventHandler<E> + 'static;

    /// 購読を解除
    async fn unsubscribe(&self, handle: SubscriptionHandle) -> Result<(), SubscribeError>;
}

/// 購読ハンドル（解除用）
pub struct SubscriptionHandle {
    id: String,
}

// 使用例: 在庫引当ハンドラ
pub struct AllocateInventoryHandler {
    inventory_service: Arc<dyn InventoryService>,
}

#[async_trait]
impl EventHandler<OrderConfirmed> for AllocateInventoryHandler {
    async fn handle(&self, envelope: EventEnvelope<OrderConfirmed>) -> Result<(), HandlerError> {
        let event = &envelope.payload;
        self.inventory_service
            .allocate(&event.order_id, &event.items)
            .await
            .map_err(HandlerError::from)
    }
}
```

### Go 版

```go
package domainevent

import "context"

// EventHandler はイベントハンドラインターフェース

    Handle(ctx context.Context, envelope EventEnvelope[E]) error
}

// EventSubscriber はイベント購読インターフェース
type EventSubscriber interface {
    Subscribe(ctx context.Context, eventType string, handler any) (SubscriptionHandle, error)
    Unsubscribe(ctx context.Context, handle SubscriptionHandle) error
}

// SubscriptionHandle は購読解除用ハンドル
type SubscriptionHandle struct {
    ID string
}
```

## 機能 4: Outbox パターン（トランザクション保証）

### Rust 版

```rust
use k1s0_domain_event::outbox::{OutboxStore, OutboxRelay, OutboxEntry};
use k1s0_db::tx::Transaction;

/// Outbox テーブルへの書き込み
#[async_trait]
pub trait OutboxStore: Send + Sync {
    /// トランザクション内でイベントを outbox に記録
    async fn store<E: DomainEvent>(
        &self,
        tx: &mut Transaction,
        event: E,
    ) -> Result<OutboxEntry, OutboxError>;

    /// 未送信エントリを取得
    async fn fetch_pending(&self, limit: usize) -> Result<Vec<OutboxEntry>, OutboxError>;

    /// 送信済みマーク
    async fn mark_delivered(&self, entry_id: &str) -> Result<(), OutboxError>;
}

/// Outbox エントリ
#[derive(Debug, Clone)]
pub struct OutboxEntry {
    pub id: String,
    pub event_type: String,
    pub aggregate_id: String,
    pub payload: serde_json::Value,
    pub metadata: EventMetadata,
    pub created_at: DateTime<Utc>,
    pub delivered_at: Option<DateTime<Utc>>,
    pub retry_count: u32,
}

/// Outbox リレー（ポーリングで未送信イベントをブローカーへ転送）
pub struct OutboxRelay {
    store: Arc<dyn OutboxStore>,
    publisher: Arc<dyn EventPublisher>,
    poll_interval: Duration,
    batch_size: usize,
}

impl OutboxRelay {
    pub fn new(
        store: Arc<dyn OutboxStore>,
        publisher: Arc<dyn EventPublisher>,
    ) -> Self;

    /// リレーを開始（バックグラウンドタスク）
    pub async fn start(&self, shutdown: tokio::sync::watch::Receiver<bool>);
}

// 使用例: Outbox 統合ユースケース
pub struct ConfirmOrderUseCase {
    order_repo: Arc<dyn OrderRepository>,
    outbox: Arc<dyn OutboxStore>,
    db: Arc<dyn Database>,
}

impl ConfirmOrderUseCase {
    pub async fn execute(&self, order_id: &OrderId) -> Result<(), AppError> {
        let mut tx = self.db.begin().await?;

        let order = self.order_repo.find_by_id_tx(&mut tx, order_id).await?;
        let event = order.confirm()?;

        // 同一トランザクションで永続化 + outbox 記録
        self.order_repo.save_tx(&mut tx, &order).await?;
        self.outbox.store(&mut tx, event).await?;

        tx.commit().await?;
        Ok(())
    }
}
```

### Go 版

## 機能 5: インメモリイベントバス（テスト・単体サービス用）

### Rust 版

```rust
use k1s0_domain_event::bus::InMemoryEventBus;

/// インメモリイベントバス（tokio::broadcast ベース）
pub struct InMemoryEventBus {
    sender: broadcast::Sender<SerializedEvent>,
    buffer_size: usize,
}

impl InMemoryEventBus {
    pub fn new(buffer_size: usize) -> Self;
}

#[async_trait]
impl EventPublisher for InMemoryEventBus {
    async fn publish<E: DomainEvent>(&self, event: E) -> Result<(), PublishError> {
        let serialized = serde_json::to_value(&event)?;
        self.sender.send(SerializedEvent {
            event_type: event.event_type().to_string(),
            payload: serialized,
            metadata: EventMetadata::new_auto(),
        })?;
        Ok(())
    }
}

// テストでの使用例
#[cfg(test)]
mod tests {
    use super::*;

    #[tokio::test]
    async fn test_order_confirmed_event() {
        let bus = InMemoryEventBus::new(16);

        bus.subscribe::<OrderConfirmed, _>("order.confirmed", handler.clone()).await.unwrap();
        bus.publish(OrderConfirmed { order_id: "123".into(), items: vec![] }).await.unwrap();

        // ハンドラが呼ばれたことを検証
        assert_eq!(handler.call_count(), 1);
    }
}
```

## k1s0-realtime（フロントエンド）との連携

```
[バックエンド]                          [フロントエンド]
ドメイン層                              @k1s0/realtime
  OrderConfirmed                        │
       │                                │
  EventPublisher                        │
       │                                │
  OutboxRelay ──→ メッセージブローカー    │
                       │                │
              WebSocket/SSE Gateway ──→ useSSE / useWebSocket
                                        │
                                   UI 自動更新
```

バックエンドの `k1s0-domain-event` で発行されたイベントは、OutboxRelay → ブローカー → Gateway を経由してフロントエンドの `@k1s0/realtime` / `k1s0_realtime` に到達する。

## 期待される効果

1. **疎結合**: サービス間を直接呼び出さず、イベント経由で連携
2. **信頼性**: Outbox パターンによるトランザクション保証（イベント消失防止）
3. **拡張性**: 新しい購読者を追加しても発行側を変更不要
4. **テスト容易性**: InMemoryEventBus でユニットテストが容易
5. **観測性**: k1s0-observability との連携で trace_id を伝搬

---

## 注意事項

- **べき等性**: 購読側は同一イベントの重複配信に対応すること（at-least-once）
- **順序保証**: 同一 aggregate_id 内のイベント順序のみ保証、異なる集約間の順序は非保証
- **スキーマ進化**: イベントバージョンを使い、後方互換性を維持すること
- **Outbox テーブル肥大化**: 配信済みエントリの定期削除が必要
- **環境変数禁止**: ブローカー接続情報は `config/*.yaml` で管理（K020 準拠）
