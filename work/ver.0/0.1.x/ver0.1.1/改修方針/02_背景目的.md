## 背景・目的

### 背景

現行の k1s0 は以下の2層構造を採用している:

```
framework/  →  feature/
(共通基盤)     (個別業務機能)
```

しかし、大規模組織での運用では以下の課題がある:

1. **責任境界の曖昧さ**: framework と feature の間に「業務ドメイン共通」という概念が欠落
2. **チーム分割の困難さ**: 複数の業務チームが共通で使う機能を feature に置くか framework に置くか判断が難しい
3. **依存関係の複雑化**: feature 間で共通機能を共有しようとすると、依存関係が複雑になる

### 目的

1. **3層構造の導入**: framework → domain → feature の明確な階層化
2. **チーム責任の明確化**: 各層を担当するチームを明確に分離
3. **依存関係の制御**: 層間の依存方向を強制し、設計の一貫性を維持

### 各層の役割定義

| 層 | 役割 | 内容 | 例（製造業） |
|----|------|------|-------------|
| **framework** | システム共通の動き | 業務内容に依存しない、システム全体で共通して使用する基盤機能 | 認証・認可、ログ出力、エラーハンドリング、DB接続、設定管理、ヘルスチェック |
| **domain** | 各業務部門の共通の動き | 特定の業務領域（ドメイン）内で複数の機能が共通して使用するビジネスロジック | 生産管理ドメイン：BOM（部品表）、工程マスタ、生産能力計算ロジック |
| **feature** | 特定業務でのみの処理 | 個別の業務機能に閉じた処理。他の機能からは参照されない | 製造指示発行、作業日報入力、検査成績書出力 |

### 層の判断基準

コードをどの層に配置するか迷った場合の判断フロー:

```
Q1: その機能は業務内容に依存しますか？
    │
    ├─ No → framework（技術基盤）
    │        例: ログ出力、認証、DB接続
    │
    └─ Yes
        │
        Q2: 同じ業務領域の複数機能で使いますか？
            │
            ├─ Yes → domain（業務部門共通）
            │        例: 生産管理の「BOM（部品表）」は製造指示でも作業日報でも使う
            │
            └─ No → feature（特定業務専用）
                     例: 製造指示書のPDF生成は製造指示機能だけで使う
```

### 想定するチーム構成

| チーム                 | 担当レイヤー | 責務                         |
| ---------------------- | ------------ | ---------------------------- |
| 開発基盤チーム         | framework/   | システム共通基盤の開発・保守 |
| ドメインアドオンチーム | domain/      | 業務ドメイン共通機能の開発   |
| 特定業務アドオンチーム | feature/     | 個別業務機能の開発           |
