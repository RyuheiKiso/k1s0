# 要望1

本システムはシステム共通基盤ならば
システム共通基盤　→　業務部門がドメイン共通機能をアドオン　→　特定業務用の機能をアドオン
という３層形式で開発したい。

開発基盤チーム
ドメインアドオンチーム
特定業務アドオンチーム
という風にしたい。

## 回答（実装方針）

### 現状のk1s0構造（2層）

```
framework/  →  feature/
(共通基盤)     (個別業務機能)
```

### 要望の3層構造への対応案

```
framework/  →  domain/  →  feature/
(共通基盤)     (ドメイン共通)  (特定業務)
```

#### 1. ディレクトリ構造の拡張

```
k1s0/
├── framework/                    # Tier 1: システム共通基盤（開発基盤チーム管理）
│   ├── backend/
│   │   ├── rust/crates/         # k1s0-error, k1s0-config 等
│   │   └── go/pkg/
│   └── frontend/
│       ├── react/packages/      # @k1s0/ui, @k1s0/auth-client 等
│       └── flutter/packages/
│
├── domain/                       # Tier 2: ドメイン共通機能（ドメインアドオンチーム管理）【新規追加】
│   ├── backend/
│   │   ├── rust/{domain_name}/  # 例: domain/backend/rust/finance/
│   │   └── go/{domain_name}/    # 例: domain/backend/go/logistics/
│   └── frontend/
│       ├── react/{domain_name}/
│       └── flutter/{domain_name}/
│
├── feature/                      # Tier 3: 特定業務機能（特定業務アドオンチーム管理）
│   ├── backend/
│   │   ├── rust/{feature_name}/
│   │   └── go/{feature_name}/
│   └── frontend/
│       ├── react/{feature_name}/
│       └── flutter/{feature_name}/
```

#### 2. 依存関係ルール

| レイヤー | 依存可能な対象 | 依存禁止 |
|---------|---------------|---------|
| framework (Tier 1) | 外部ライブラリのみ | domain, feature |
| domain (Tier 2) | framework | feature, 他のdomain |
| feature (Tier 3) | framework, 自身が属するdomain | 他のfeature, 他のdomain |

#### 3. CLIコマンドの拡張

```bash
# ドメイン共通機能の生成（新規）
k1s0 new-domain --type backend-rust --name finance

# 特定業務機能の生成（既存拡張）
k1s0 new-feature --type backend-rust --name invoice-processing --domain finance
```

#### 4. manifest.jsonの拡張

```json
{
  "name": "invoice-processing",
  "type": "backend-rust",
  "layer": "feature",           // "framework" | "domain" | "feature"
  "domain": "finance",          // feature層の場合、所属するdomain名
  "version": "1.0.0"
}
```

#### 5. Lintルールの追加

| ID | 説明 |
|----|------|
| K040 | domain層からfeature層への依存禁止 |
| K041 | feature層から異なるdomain層への依存禁止 |
| K042 | framework層からdomain/feature層への依存禁止 |

#### 6. 実装優先度

1. **Phase 1**: `domain/` ディレクトリ構造の追加
2. **Phase 2**: `k1s0 new-domain` コマンドの実装
3. **Phase 3**: manifest.json スキーマ拡張（layer, domain フィールド）
4. **Phase 4**: 依存関係チェックのLintルール実装（K040-K042）
5. **Phase 5**: ドキュメント・ADR作成
