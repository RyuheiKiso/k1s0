## 全体像

```
開発者のPC                    CI/CD                         Kubernetes
┌──────────────────┐     ┌──────────────────┐     ┌──────────────────┐
│ k1s0 new-feature │     │ docker build     │     │ kubectl apply    │
│ ↓                │     │ ↓                │     │ (kustomize)      │
│ docker compose   │ PR  │ docker push      │ CD  │                  │
│ up (ローカル)     │ ──→ │ (レジストリ)      │ ──→ │ Deployment       │
│ ↓                │     │ ↓                │     │ Service          │
│ 開発・テスト      │     │ lint / test      │     │ ConfigMap        │
│ ↓                │     │ upgrade --check  │     │ Secrets          │
│ git push         │     │                  │     │                  │
└──────────────────┘     └──────────────────┘     └──────────────────┘
```

## ローカル開発フロー

### Step 1: サービス生成

```bash
# リポジトリルートで実行
k1s0 new-feature --type backend-rust --name order-processing --with-db --with-grpc

# 生成されるもの（v0.2.3 で追加される Docker 関連ファイルを含む）:
# feature/backend/rust/order-processing/
# ├── Cargo.toml
# ├── Dockerfile                # NEW: マルチステージビルド
# ├── .dockerignore             # NEW: Docker ビルド除外設定
# ├── docker-compose.yml        # NEW: ローカル開発用
# ├── src/
# │   ├── main.rs
# │   ├── domain/
# │   ├── application/
# │   ├── infrastructure/
# │   └── presentation/
# ├── config/
# │   ├── default.yaml
# │   ├── dev.yaml
# │   ├── stg.yaml
# │   └── prod.yaml
# ├── deploy/
# │   ├── base/
# │   └── overlays/{dev,stg,prod}/
# ├── proto/
# ├── migrations/
# ├── secrets/
# │   └── db_password.txt       # NEW: 初期値 "changeme"（.gitignore 対象）
# └── .k1s0/
#     └── manifest.json
```

### Step 2: ローカル環境の起動

```bash
cd feature/backend/rust/order-processing

# 全サービスをバックグラウンドで起動（app + PostgreSQL + Redis）
docker compose up -d --build

# または k1s0 CLI ラッパーを使用
k1s0 docker compose up -d --build
```

起動されるコンテナ:

| サービス | イメージ           | ポート      | 用途             |
| -------- | ------------------ | ----------- | ---------------- |
| app      | ローカルビルド     | 8080, 50051 | アプリケーション |
| db       | postgres:16-alpine | 5432        | PostgreSQL       |
| redis    | redis:7-alpine     | 6379        | Redis キャッシュ |

### Step 3: 動作確認

```bash
# ヘルスチェック
curl http://localhost:8080/healthz

# コンテナ状態確認
docker compose ps
# または
k1s0 docker status

# ログ確認
docker compose logs -f app
# または
k1s0 docker compose logs -f app
```

### Step 4: 開発サイクル

```bash
# コード修正後、アプリだけリビルド
docker compose up -d --build app

# DB マイグレーション追加後
docker compose down -v   # ボリューム削除（DB リセット）
docker compose up -d --build

# 規約チェック
k1s0 lint

# 全サービス停止
docker compose down
```

### Step 5: ローカル開発での設定管理

設定は `config/dev.yaml` をボリュームマウントで反映する。環境変数は使用しない。

```yaml
# config/dev.yaml
server:
  host: "0.0.0.0"
  port: 8080
database:
  host: db            # docker-compose のサービス名で接続
  port: 5432
  name: order_processing
  user: postgres
  password_file: /run/secrets/db_password
cache:
  host: redis          # docker-compose のサービス名で接続
  port: 6379
```

```bash
# 起動時に環境を明示的に指定（k1s0 の規約に従う）
# アプリの CMD/ENTRYPOINT で --env dev を指定
```

## CI/CD フロー

### PR 作成時（自動実行）

開発者が PR を作成すると、以下のチェックが自動実行される。

```
PR 作成
  ↓
┌─────────────────────────────────────────────┐
│ CI パイプライン                               │
│                                             │
│ 1. k1s0 lint                  → 規約チェック  │
│ 2. cargo test / go test       → ユニットテスト │
│ 3. buf lint / buf breaking    → API 契約検証  │
│ 4. docker build               → ビルド検証    │
│ 5. docker compose up → test   → 統合テスト    │
│ 6. k1s0 upgrade --check       → 差分確認      │
│                                             │
│ 全パス → マージ可能                            │
└─────────────────────────────────────────────┘
```

### 開発者が PR 前に行うこと

```bash
# 1. 規約チェック
k1s0 lint

# 2. テスト実行
cd feature/backend/rust/order-processing
cargo test --all

# 3. Docker ビルド確認（CI と同じビルドが通るか）
docker build -t order-processing:test .
# または
k1s0 docker build --tag order-processing:test

# 4. 統合テスト（compose で依存サービス込み）
docker compose up -d --build
curl http://localhost:8080/healthz
docker compose down -v

# 5. API 契約チェック（gRPC 使用時）
buf lint
buf breaking --against '.git#branch=main'

# 6. コミット & PR
git add .
git commit -m "feat: 注文処理サービスの初期実装"
git push origin feature/order-processing
```

### main マージ後（自動実行）

```
main マージ
  ↓
┌─────────────────────────────────────────────┐
│ CD パイプライン                               │
│                                             │
│ 1. docker build                             │
│ 2. docker tag {feature_name}:{version}      │
│ 3. docker push {registry}/{feature_name}    │
│ 4. kustomize build deploy/overlays/dev      │
│ 5. kubectl apply（dev 環境）                  │
│                                             │
└─────────────────────────────────────────────┘
```

## ステージング・本番デプロイフロー

### dev → stg → prod の昇格フロー

```
dev 環境              stg 環境              prod 環境
┌──────────┐      ┌──────────┐      ┌──────────┐
│ 自動デプロイ│ 承認  │ 手動承認  │ 承認  │ 手動承認  │
│ main merge│ ──→  │ 後デプロイ │ ──→  │ 後デプロイ │
│           │      │           │      │           │
│ config/   │      │ config/   │      │ config/   │
│ dev.yaml  │      │ stg.yaml  │      │ prod.yaml │
└──────────┘      └──────────┘      └──────────┘
```

### Kubernetes デプロイ（既存の deploy/ テンプレート活用）

```bash
# dev 環境デプロイ（CI/CD が自動実行）
kustomize build deploy/overlays/dev | kubectl apply -f -

# stg 環境デプロイ（承認後）
kustomize build deploy/overlays/stg | kubectl apply -f -

# prod 環境デプロイ（承認後）
kustomize build deploy/overlays/prod | kubectl apply -f -
```

### Docker イメージと Kubernetes の関係

```yaml
# deploy/base/deployment.yaml（既存テンプレート生成物）
spec:
  containers:
    - name: order-processing
      image: order-processing:latest    # CI/CD でレジストリURLに置換
      args:
        - --env
        - $(ENV)
        - --config
        - /etc/k1s0/config
        - --secrets-dir
        - /var/run/secrets/k1s0
```

| 段階     | Docker イメージ       | 設定ファイル                    | シークレット                   |
| -------- | --------------------- | ------------------------------- | ------------------------------ |
| ローカル | ローカルビルド        | config/dev.yaml（volume mount） | secrets/db_password.txt        |
| dev      | レジストリ:dev-{sha}  | config/dev.yaml（ConfigMap）    | K8s Secret（ファイルマウント） |
| stg      | レジストリ:stg-{sha}  | config/stg.yaml（ConfigMap）    | K8s Secret（ファイルマウント） |
| prod     | レジストリ:v{version} | config/prod.yaml（ConfigMap）   | K8s Secret（ファイルマウント） |

### 開発者が環境ごとに意識すること

| 項目         | ローカル（compose）                     | Kubernetes                         |
| ------------ | --------------------------------------- | ---------------------------------- |
| 設定ファイル | config/ をボリュームマウント            | ConfigMap にマウント               |
| シークレット | secrets/ ディレクトリ（Docker secrets） | K8s Secret にマウント              |
| DB 接続先    | `db`（compose サービス名）              | K8s Service 名                     |
| Redis 接続先 | `redis`（compose サービス名）           | K8s Service 名                     |
| 環境指定     | `--env dev`                             | `--env` は overlay の ENV から注入 |

**開発者が行うべきこと**: config/*.yaml のホスト名を環境ごとに適切に設定する。接続先以外のロジックは全環境共通。

## フロントエンド開発者のフロー

### React アプリケーション

```bash
# サービス生成
k1s0 new-feature --type frontend-react --name admin-dashboard

# ローカル開発（ホットリロード）
cd feature/frontend/react/admin-dashboard
pnpm install
pnpm dev    # Vite 開発サーバー（推奨: Docker を使わない）

# Docker ビルド確認（本番同等の Nginx 配信を確認したい時）
docker compose up -d --build
# → http://localhost:3000 でアクセス

# Docker の停止
docker compose down
```

### フロントエンドの開発時 Docker 使い分け

| シーン             | 方法                            | 理由                                     |
| ------------------ | ------------------------------- | ---------------------------------------- |
| 日常の開発         | `pnpm dev`（Vite 開発サーバー） | ホットリロードが効く、高速               |
| 本番同等の表示確認 | `docker compose up --build`     | Nginx 配信、gzip、SPA ルーティングの確認 |
| CI/CD              | `docker build`                  | 本番デプロイ用イメージのビルドテスト     |

## よくある操作シナリオ

### シナリオ 1: 新しいバックエンドサービスを作って動かすまで

```bash
# 1. 生成
k1s0 new-feature --type backend-go --name user-management --with-db --with-cache --with-grpc

# 2. 起動
cd feature/backend/go/user-management
docker compose up -d --build

# 3. 確認
curl http://localhost:8080/healthz
docker compose ps

# 4. 開発
# src/ 配下のコードを編集
# ...

# 5. リビルド
docker compose up -d --build app

# 6. テスト
go test -v -race ./...

# 7. 規約チェック
k1s0 lint

# 8. 停止
docker compose down -v

# 9. PR
git add .
git commit -m "feat: ユーザー管理サービスの初期実装"
git push
```

### シナリオ 2: 既存サービスに Docker サポートを追加する

```bash
# upgrade で Docker ファイルを取得
cd feature/backend/rust/existing-service

# 差分確認
k1s0 upgrade --check

# 適用（Dockerfile, .dockerignore, docker-compose.yml が追加される）
k1s0 upgrade

# secrets ディレクトリを手動作成（with_db の場合）
mkdir -p secrets
echo "changeme" > secrets/db_password.txt

# 動作確認
docker compose up -d --build
```

### シナリオ 3: DB マイグレーションを追加する

```bash
cd feature/backend/rust/order-processing

# マイグレーションファイルを追加
# migrations/0002_add_status_column.up.sql
# migrations/0002_add_status_column.down.sql

# DB をリセットして再起動（ローカル開発時）
docker compose down -v
docker compose up -d --build

# マイグレーションが適用されたことを確認
docker compose exec db psql -U postgres -d order_processing -c "\dt"
```

### シナリオ 4: 複数サービスを連携させてローカルテストする

```bash
# サービス A を起動
cd feature/backend/rust/order-processing
docker compose up -d --build

# サービス B を起動（別ターミナル）
cd feature/backend/go/user-management
docker compose up -d --build

# 注意: ポート競合を避けるため、2つ目のサービスのポートを変更する必要がある
# docker-compose.yml の ports を編集（unmanaged ファイルなので自由に変更可能）:
#   ports:
#     - "8081:8080"
#     - "5433:5432"

# サービス間通信は Docker ネットワーク経由ではなく localhost 経由
# config/dev.yaml で接続先を localhost:8080 / localhost:8081 に設定
```

### シナリオ 5: CI で失敗した Docker ビルドをローカルで再現する

```bash
cd feature/backend/rust/order-processing

# CI と同じコマンドでビルド
docker build -t order-processing:test .

# ビルドが成功したらコンテナ起動テスト
docker run --rm -d --name test-run order-processing:test
sleep 5
docker logs test-run
docker stop test-run

# compose での統合テスト
docker compose up -d --build
curl http://localhost:8080/healthz
docker compose down -v
```

## 禁止事項と注意点

### 環境変数の使用禁止（k1s0 規約 K020）

```yaml
# NG: docker-compose.yml でアプリに環境変数を渡す
services:
  app:
    environment:
      DATABASE_URL: postgres://...     # 禁止
      REDIS_HOST: redis                # 禁止

# OK: config ファイルをマウントして --env で指定
services:
  app:
    volumes:
      - ./config:/app/config:ro
```

docker-compose.yml の `environment` は、PostgreSQL や Redis など外部イメージの初期設定のみに使用する。アプリケーションの設定は必ず `config/*.yaml` で管理する。

### シークレットの平文禁止（k1s0 規約 K021）

```yaml
# NG: パスワードを直接書く
services:
  db:
    environment:
      POSTGRES_PASSWORD: mysecretpassword   # 禁止

# OK: Docker secrets を使用
services:
  db:
    environment:
      POSTGRES_PASSWORD_FILE: /run/secrets/db_password
    secrets:
      - db_password
secrets:
  db_password:
    file: ./secrets/db_password.txt
```

### secrets/ ディレクトリの Git 管理禁止

```gitignore
# .gitignore に含まれる
secrets/
```

`secrets/` ディレクトリは各開発者がローカルで作成する。Git にコミットしない。

### Dockerfile の直接編集について

| ファイル           | 管理区分                      | 編集可否                                |
| ------------------ | ----------------------------- | --------------------------------------- |
| Dockerfile         | protected（テンプレート管理） | `k1s0 upgrade` で更新。直接編集は非推奨 |
| .dockerignore      | protected                     | 同上                                    |
| docker-compose.yml | unmanaged（チーム管理）       | 自由に編集可能                          |
| nginx.conf         | protected                     | `k1s0 upgrade` で更新                   |
| secrets/*          | unmanaged                     | ローカル管理、Git 対象外                |

### docker-compose.yml のカスタマイズ例

docker-compose.yml は unmanaged なので、チームが自由にカスタマイズできる。

```yaml
# チームが追加する例: メール送信サービスの追加
services:
  app:
    # ...（テンプレート生成物）

  mailhog:
    image: mailhog/mailhog
    ports:
      - "8025:8025"
      - "1025:1025"
```

## まとめ: 開発者のデイリーコマンド

```bash
# 朝: 開発環境を起動
cd feature/backend/rust/my-service
docker compose up -d

# 開発中: コード変更後にリビルド
docker compose up -d --build app

# 開発中: ログ確認
docker compose logs -f app

# 開発中: 規約チェック
k1s0 lint

# 退勤前: 環境を停止
docker compose down

# PR 前: 最終確認
k1s0 lint
cargo test --all
docker build -t my-service:test .
docker compose up -d --build
curl http://localhost:8080/healthz
docker compose down -v
git push
```
