# 要望

現行の lint エンジン（K001-K060）は全ルールが grep/正規表現ベースの行単位マッチングで実装されており、AST（抽象構文木）レベルの解析に強化したい。

## 背景

| 課題 | 説明 |
|------|------|
| 偽陽性が多い | コメント・文字列リテラル・ドキュメント内のパターンを誤検出する（例: K029 が `/// .unwrap()` をフラグ） |
| 偽陰性がある | 複数行にまたがるパターンや間接的な違反を検出できない（例: K050 で変数経由の SQL インジェクション） |
| コメント除外が不完全 | 行頭プレフィックスのみで判定しており、インラインコメント・複数行コメント・docstring に対応していない |
| 型情報を活用できない | 型エイリアスやトレイト境界経由のプロトコル型依存を検出不可（K026） |
| 制御フロー解析不可 | 到達不能コードや条件分岐内のパターンを区別できない |
| スコープ認識なし | テストコード内の `unwrap()` を除外するのにファイル名ヒューリスティックに依存（K029） |

## 現行実装の問題例

### K029（Panic 検出）— 偽陽性

```rust
// 現行: line.contains(".unwrap()") で判定
/// This method returns None if the value is missing. Use .unwrap() only in tests.
fn get_value() -> Option<i32> { Some(42) }
// ↑ ドキュメントコメント内の .unwrap() が誤検出される
```

### K050（SQL インジェクション）— 偽陰性

```rust
// 現行: 単一行の format!("SELECT ...") を検出
let table = user_input;
let query = format!("SELECT * FROM {}", table);  // 変数名だけでは検出困難
db.execute(&query).await?;
// ↑ format! と execute が別行にあるため検出漏れ
```

### K022（Clean Architecture 違反）— 偽陰性

```rust
// 現行: use 文のパターンマッチのみ
mod helper {
    pub use crate::infrastructure::db::Connection;  // re-export
}
use helper::Connection;  // 間接的な infrastructure 依存
// ↑ re-export 経由の違反を検出できない
```

## 要望内容

### AST ベース解析エンジンの導入

各言語に対応した AST パーサーを導入し、構文木レベルでコード構造を理解した上で lint ルールを適用する。

### 期待される改善

| 項目 | grep ベース | AST ベース |
|------|-----------|-----------|
| コメント・文字列の除外 | 行頭プレフィックスのみ | 構文ノード種別で完全除外 |
| 複数行パターン | 検出不可 | ノード探索で検出可能 |
| 型情報の活用 | 不可 | インポート解決・型推論（限定的） |
| スコープ認識 | ファイル名ヒューリスティック | テスト属性・モジュール構造で判定 |
| 誤検出率 | 高 | 大幅に低減（推定 30-50% 削減） |

## 影響

- 既存の lint ルール番号（K001-K060）と報告フォーマットは変更しない
- grep ベースのフォールバックを維持し、パーサーが対応していない言語は従来通り動作
- `k1s0 lint` の実行時間は増加するが、`--fast` オプションで grep モードに切替可能
- CI/CD ワークフローへの影響なし（出力フォーマット互換）
