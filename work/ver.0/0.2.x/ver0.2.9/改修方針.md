# k1s0 v0.2.9 改修方針

## 概要

k1s0 の lint エンジンを grep/正規表現ベースから AST（抽象構文木）ベースに段階的に移行する。tree-sitter を統一パーサーとして採用し、全対応言語で構文木レベルの解析を実現する。

---

## パーサー選定

### 選定: tree-sitter

| 候補 | 長所 | 短所 | 採否 |
|------|------|------|:----:|
| **tree-sitter** | 全言語統一API、Rust バインディング成熟、増分パース対応、エラー耐性 | 型推論不可、言語固有の意味解析は限定的 | **採用** |
| 言語固有パーサー (syn, go/ast 等) | 完全な型情報・意味解析 | 言語ごとに別実装が必要、メンテコスト大 | 不採用 |
| Roslyn / rustc API | 最高精度の解析 | 外部ツールチェーン依存、起動コスト大 | 不採用 |

### tree-sitter 採用理由

1. **統一 API**: 全言語で同一の `Tree` / `Node` / `Query` インターフェースを使用
2. **Rust ネイティブ**: k1s0 CLI は Rust 製のため、`tree-sitter` crate との親和性が高い
3. **増分パース**: ファイル変更時に差分のみ再パースでき、LSP 統合に有利
4. **エラー耐性**: 不完全なコードでもパースを続行し、可能な限り構文木を構築
5. **実績**: GitHub Linguist、Neovim、Helix 等で広く採用

### 対応言語と tree-sitter グラマー

| 言語 | crate | 対応状況 | ノード種別例（lint で使用） |
|------|-------|---------|--------------------------|
| Rust | `tree-sitter-rust` | ✅ 対応 | `call_expression`, `macro_invocation`, `use_declaration`, `line_comment`, `block_comment`, `string_literal`, `attribute_item` |
| Go | `tree-sitter-go` | ✅ 対応 | `call_expression`, `import_declaration`, `comment`, `interpreted_string_literal` |
| TypeScript | `tree-sitter-typescript` | ✅ 対応 | `call_expression`, `import_statement`, `comment`, `string`, `template_string` |
| Python | `tree-sitter-python` | ✅ 対応 | `call`, `import_from_statement`, `comment`, `string` |
| C# | `tree-sitter-c-sharp` | ✅ 対応 | `invocation_expression`, `using_directive`, `comment`, `string_literal` |
| Kotlin | `tree-sitter-kotlin-ng` | ✅ 対応 | `call_expression`, `import_header`, `comment`, `string_literal` |
| Dart | N/A | ⛔ 未対応 | 互換性のある tree-sitter バインディングが存在しない |
| YAML | `tree-sitter-yaml` | ⛔ 未対応 | tree-sitter 0.20 ABI のため互換性なし |
| Dockerfile | `tree-sitter-dockerfile` | ⛔ 未対応 | tree-sitter 0.20 ABI のため互換性なし |

**注記:**
- Dart, YAML, Dockerfile は tree-sitter 0.25 (ABI version 15) に対応したクレートが存在しないため除外
- Kotlin は `tree-sitter-kotlin-ng` を使用（コミュニティ製、ABI version 15 対応）

### コア抽象

```rust
/// AST 解析コンテキスト
pub struct AstContext {
    tree: tree_sitter::Tree,
    source: Vec<u8>,
    language: Language,
}

impl AstContext {
    /// ノードがコメント・文字列リテラル・ドキュメント内かを判定
    pub fn is_non_code(&self, node: &Node) -> bool;

    /// ノードがテスト関数/モジュール内かを判定
    pub fn is_in_test(&self, node: &Node) -> bool;

    /// tree-sitter Query を実行し、マッチしたノードを返す
    pub fn query_matches(&self, query: &Query) -> Vec<QueryMatch>;

    /// 特定のノード種別の祖先を探索
    pub fn find_ancestor(&self, node: &Node, kind: &str) -> Option<Node>;
}

/// 言語列挙（対応言語のみ）
pub enum Language {
    Rust, Go, TypeScript, Python, CSharp, Kotlin,
}
```

**重要:** Dart, YAML, Dockerfile は ABI 互換性の問題により除外されました。

## ルール別 AST 移行方針

### 優先度: 高（偽陽性の削減効果大）

#### K029: Panic/unwrap/expect 検出

| 項目 | grep ベース（現行） | AST ベース（移行後） |
|------|-------------------|-------------------|
| 検出方法 | `line.contains(".unwrap()")` | `(call_expression function: (field_expression field: (field_identifier) @name (#eq? @name "unwrap")))` |
| コメント除外 | 行頭 `//` のみ | `line_comment`, `block_comment` ノードの子孫を除外 |
| 文字列除外 | なし | `string_literal`, `raw_string_literal` ノードの子孫を除外 |
| テスト除外 | ファイル名 `_test` | `#[cfg(test)]` / `#[test]` 属性を持つモジュール・関数内を除外 |
| `macro_rules!` | 誤検出 | `macro_definition` ノード内を除外 |

**tree-sitter Query (Rust):**
```scheme
;; unwrap() 呼び出しを検出
(call_expression
  function: (field_expression
    field: (field_identifier) @method_name)
  (#match? @method_name "^(unwrap|expect)$"))

;; panic! マクロ呼び出しを検出
(macro_invocation
  macro: (identifier) @macro_name
  (#match? @macro_name "^(panic|todo|unimplemented|unreachable)$"))
```

#### K050: SQL インジェクション検出

| 項目 | grep ベース（現行） | AST ベース（移行後） |
|------|-------------------|-------------------|
| 検出方法 | `line.contains("format!(\"SELECT")` | `macro_invocation` + 引数内の SQL キーワード + 補間式の存在を検出 |
| 変数経由 | 検出不可 | `format!` の戻り値が DB 実行関数に渡されるパターンを限定的に追跡 |
| パラメータ化クエリ | 区別不可 | `sqlx::query!` 等の安全なマクロを除外 |

**tree-sitter Query (Rust):**
```scheme
;; format! マクロ内の SQL 文字列補間を検出
(macro_invocation
  macro: (identifier) @macro_name
  (#eq? @macro_name "format")
  (token_tree
    (string_literal) @sql_string
    (#match? @sql_string "(SELECT|INSERT|UPDATE|DELETE|DROP|ALTER)")))
```

#### K022: Clean Architecture 依存違反

| 項目 | grep ベース（現行） | AST ベース（移行後） |
|------|-------------------|-------------------|
| 検出方法 | `use crate::infrastructure` パターンマッチ | `use_declaration` ノードのパスを解析し、レイヤー境界を判定 |
| re-export | 検出不可 | `pub use` の転送先を追跡（1段階） |
| ワイルドカード | 部分的 | `use crate::infrastructure::*` のグロブインポートも検出 |

### 優先度: 中

#### K026: Domain 層でのプロトコル型使用

- AST で `use` 文を解析し、HTTP/gRPC 関連の型インポートを検出
- 型エイリアス経由の間接利用は、エイリアス定義の解決（1段階）で対応

#### K020: 環境変数使用禁止

- `call_expression` で `std::env::var` 等の関数呼び出しをノードレベルで検出
- `build.rs` やテストファイルは AST コンテキストで除外

#### K053: センシティブデータのログ出力

- `macro_invocation` (`log::info!`, `tracing::info!` 等) の引数を解析
- 引数内の変数名にセンシティブキーワードが含まれるかを判定

### 優先度: 低（AST 移行の効果が限定的）

| ルール | 理由 |
|--------|------|
| K001-K003 | JSON パース（既に構造化解析済み） |
| K010-K011 | ファイルシステムチェック（AST 不要） |
| K021 | YAML パース（既に構造化解析済み） |
| K025 | ファイル名チェック（AST 不要） |
| K060 | Dockerfile FROM チェック（tree-sitter-dockerfile で改善可能だが効果小） |
| K040-K047 | manifest.json ベースのレイヤー検証（AST 不要） |

## grep フォールバック戦略

AST パーサーが利用できない場合（未対応言語、パースエラー等）に従来の grep ベース検出にフォールバックする。

```rust
pub fn check_rule(file: &Path, source: &str, rule: &Rule) -> Vec<Diagnostic> {
    // AST パースを試行
    match AstContext::parse(file, source) {
        Ok(ctx) => rule.check_ast(&ctx),       // AST ベース検出
        Err(_) => rule.check_grep(source),      // grep フォールバック
    }
}
```

### `--fast` モード

```bash
# AST パースをスキップし、grep ベースで高速実行
k1s0 lint --fast
```

CI での高速チェックやエディタ保存時の即時フィードバック用途。

## LSP 統合

tree-sitter の増分パースを活用し、k1s0-lsp に以下の機能を追加する。

| 機能 | 説明 |
|------|------|
| リアルタイム診断 | ファイル保存時ではなく編集中に lint 違反を表示 |
| Quick Fix | AST ノード情報を利用した正確なコード修正提案 |
| ホバー情報 | 違反箇所にホバーで詳細な説明を表示 |

※ LSP 統合は v0.2.9 の後続バージョンでの実装を想定。

## パフォーマンス設計

### パーサーキャッシュ

```rust
/// 言語ごとに tree-sitter パーサーを再利用
pub struct ParserPool {
    parsers: HashMap<Language, tree_sitter::Parser>,
}
```

- `tree_sitter::Parser` はステートフルだが軽量。言語ごとに 1 インスタンスをプール。
- グラマーの読み込みは初回のみ。

### Query コンパイルキャッシュ

```rust
/// コンパイル済み Query を言語×ルールでキャッシュ
pub struct QueryCache {
    cache: HashMap<(Language, RuleId), tree_sitter::Query>,
}
```

- tree-sitter Query のコンパイルは比較的コストが高いため、初回コンパイル後にキャッシュ。

### ベンチマーク目標

| 指標 | grep ベース | AST ベース | 許容上限 |
|------|-----------|-----------|---------|
| 単一ファイル（500行） | ~1ms | ~5ms | 10ms |
| プロジェクト全体（100ファイル） | ~100ms | ~500ms | 2s |
| `--fast` モード | ~100ms | N/A（grep） | 200ms |

## 依存関係の追加

### Cargo.toml (k1s0-generator)

```toml
[dependencies]
tree-sitter = "0.25"                    # ABI version 15 対応のため 0.24 から 0.25 にアップグレード
tree-sitter-rust = "0.24"
tree-sitter-go = "0.23"
tree-sitter-typescript = "0.23"
tree-sitter-python = "0.23"
tree-sitter-c-sharp = "0.23"
tree-sitter-kotlin-ng = "1.0"           # tree-sitter-kotlin から tree-sitter-kotlin-ng に変更（ABI 15 対応）
streaming-iterator = "0.1"              # tree-sitter 0.24+ の QueryMatches API に対応
```

**削除された依存:**
- `tree-sitter-dart` — ABI version 15 対応のクレートが存在しない
- `tree-sitter-yaml` — tree-sitter 0.20 ABI を使用、互換性なし
- `tree-sitter-dockerfile` — tree-sitter 0.20 ABI を使用、互換性なし

### ABI バージョンについて

tree-sitter は C ライブラリの ABI バージョン管理を行っており、主要言語のパーサーは以下の ABI を要求します:

- **ABI version 15**: tree-sitter 0.25+ が必要
- **ABI version 14**: tree-sitter 0.24 が必要
- **ABI version 13**: tree-sitter 0.20-0.23 が必要

Rust/Go/TypeScript/Python/C#/Kotlin の最新パーサーは ABI version 15 を要求するため、tree-sitter 0.25 が必須です。

### バイナリサイズへの影響

- 各言語グラマーは ~200KB-1MB の C コード（コンパイル時に静的リンク）
- 対応6言語で推定 +2-4MB のバイナリサイズ増加
- `--no-default-features` + feature flags で不要な言語を除外可能

```toml
[features]
default = ["ast"]
ast = ["dep:tree-sitter", "dep:tree-sitter-rust", "dep:tree-sitter-go", "dep:tree-sitter-typescript", "dep:tree-sitter-python", "dep:tree-sitter-c-sharp", "dep:tree-sitter-kotlin-ng", "dep:streaming-iterator"]
```

**注記:** 言語別の feature flag は実装せず、単一の `ast` feature で全対応言語を有効化します。

## 実装順序

1. **Phase 1: AST 基盤構築** — `ast/` モジュール新設、`AstContext`・`ParserPool`・`QueryCache` の実装
2. **Phase 2: K029 (Panic 検出) の AST 移行** — 最も偽陽性が多く効果が大きいルール
3. **Phase 3: K050 (SQL インジェクション) の AST 移行** — セキュリティルールの精度向上
4. **Phase 4: K022 (Clean Architecture) の AST 移行** — アーキテクチャ強制の改善
5. **Phase 5: K020, K026, K053 の AST 移行** — 残りの中優先度ルール
6. **Phase 6: `--fast` モード実装** — grep フォールバックの CLI オプション化
7. **Phase 7: テスト・ベンチマーク** — 偽陽性/偽陰性の比較、パフォーマンス計測
8. **Phase 8: ドキュメント更新** — lint 設計ドキュメント、CLAUDE.md の更新

## 注意事項

- **後方互換性**: ルール番号・出力フォーマット・exit code は変更しない
- **段階的移行**: 各ルールを個別に AST 版に移行し、grep 版は一定期間維持
- **feature flags**: 各言語の tree-sitter グラマーを feature flag で制御し、不要な言語を除外可能に
- **unsafe コード禁止**: tree-sitter の C バインディング部分を除き、unsafe を使用しない
- **clippy/pedantic 準拠**: 新規コードは既存の `#![warn(clippy::pedantic)]` 設定に準拠
