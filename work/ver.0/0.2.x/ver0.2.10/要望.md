# 要望

現行の k1s0 プラットフォームにはバックプレッシャー制御が体系的に整備されておらず、高負荷時にシステムが劣化する。包括的なバックプレッシャー制御機構を追加したい。

## 背景

| 課題 | 説明 |
|------|------|
| レート制限アルゴリズムがない | トークンバケット・スライディングウィンドウ等の標準的なレート制限が未実装。外部 API やクライアントからの過剰リクエストを制御できない |
| gRPC ストリーミングのバックプレッシャーがない | サーバー/クライアント共に `onReady()` によるフロー制御やストリームバッファリングが未実装。高速プロデューサーが低速コンシューマーを圧倒する |
| 非同期チャネルにデマンドシグナリングがない | `tokio::sync::mpsc` や `broadcast::channel` で容量制限はあるが、コンシューマーからの `request(n)` 型の需要通知がない |
| DB コネクションプールのバックプレッシャーがない | プール枯渇時の待機キュー制限やリジェクトポリシーが未実装。全リクエストが無制限に待機しタイムアウトする |
| フロントエンド API クライアントのスロットリングがない | `@k1s0/api-client`、`k1s0_http`、`k1s0-http` にリクエストレート制限がなく、バックエンドを過負荷にする可能性がある |
| キュー深度メトリクスが非公開 | 内部キュー（Write-Behind、Domain Event Bus、Offline Queue）の深度が Prometheus に公開されておらず、ボトルネックの可視化が不可能 |

## 現行実装の問題例

### gRPC ストリーミング — バックプレッシャーなし

```go
// framework/backend/go/k1s0-grpc-server/config.go
// 現行: MaxConcurrentStreams のみ。ストリーム内のフロー制御なし
func DefaultServerConfig() *ServerConfig {
    return &ServerConfig{
        MaxConcurrentStreams: 1000,
    }
}

// 問題: サーバーがクライアントの処理速度を無視して送信し続ける
func (s *Server) StreamData(req *pb.StreamRequest, stream pb.Service_StreamDataServer) error {
    for _, item := range items {
        // クライアントのバッファが溢れても送信を止めない
        if err := stream.Send(item); err != nil {
            return err
        }
    }
    return nil
}
```

### Domain Event Bus — ラグ時のイベント消失

```rust
// framework/backend/rust/crates/k1s0-domain-event/src/bus/in_memory.rs
// 現行: broadcast::channel(256) — 遅延サブスクライバーはイベントを消失する
let (tx, _) = broadcast::channel(256);

// 問題: サブスクライバーが 256 イベント以上遅延すると Lagged エラーで消失
match rx.recv().await {
    Ok(event) => handle(event),
    Err(broadcast::error::RecvError::Lagged(n)) => {
        // n 件のイベントが消失 — リカバリ手段なし
        tracing::warn!("Lost {} events", n);
    }
}
```

### Write-Behind Cache — メトリクス非公開

```rust
// framework/backend/rust/crates/k1s0-cache/src/patterns/write_behind.rs
// 現行: bounded channel だが、キュー深度が不透明
let (tx, rx) = tokio::sync::mpsc::channel(max_queue_size);

// 問題: チャネルが満杯に近づいても外部から観測できない
// Prometheus メトリクスなし、アラート設定不可
```

### フロントエンド — 無制限リクエスト

```typescript
// framework/frontend/react/packages/api-client/
// 現行: リクエスト送信にレート制限なし
class ApiClient {
    async get<T>(url: string): Promise<T> {
        // 同時リクエスト数やレートの制限なし
        const response = await fetch(url, this.config);
        return response.json();
    }
}
```

### DB コネクションプール — 無制限待機

```rust
// framework/backend/rust/crates/k1s0-db/
// 現行: プール枯渇時に全リクエストが待機キューに入る
let pool = PgPoolOptions::new()
    .max_connections(20)
    // max_waiting は未設定 — 数千リクエストが待機可能
    .connect(&url)
    .await?;
```

## 要望内容

### 1. gRPC ストリーミングバックプレッシャー

サーバー/クライアント双方のストリームにフロー制御を実装する。`onReady()` コールバックによる送信制御、ストリームバッファサイズの制限、スロープロデューサー/コンシューマー検出を含む。

### 2. レート制限（k1s0-rate-limit 新規パッケージ）

全バックエンド言語（Rust, Go, C#, Python, Kotlin）で統一的なレート制限 API を提供する。トークンバケット・スライディングウィンドウアルゴリズムを実装し、YAML 設定で制御可能にする。

### 3. Domain Event Bus のバックプレッシャー改善

`broadcast::channel` のラグ時消失問題を解決する。遅延サブスクライバーへの通知・リカバリ機構、バッファリング戦略（Drop Oldest / Block / Reject）の選択を可能にする。

### 4. キュー深度の可観測性

内部キュー（Write-Behind、Domain Event Bus、Offline Queue、DB コネクションプール）の深度・容量・待機数を Prometheus メトリクスとして公開する。

### 5. フロントエンド API クライアントスロットリング

React（`@k1s0/api-client`）、Flutter（`k1s0_http`）、Android（`k1s0-http`）にリクエストレート制限と同時接続数制限を追加する。

### 6. DB コネクションプールのバックプレッシャー

プール枯渇時の待機キューに上限を設け、上限超過時にリジェクト（即座にエラー返却）するポリシーを設定可能にする。

## 期待される改善

| 項目 | 現行 | バックプレッシャー導入後 |
|------|------|----------------------|
| gRPC ストリーム高負荷時 | クライアントバッファ溢れ、OOM リスク | フロー制御で送信速度を自動調整 |
| 外部 API 呼び出し | 無制限で429/503を受ける | トークンバケットで事前にスロットリング |
| Domain Event 遅延 | イベント消失（Lagged） | バッファリング + リカバリ戦略で保護 |
| キュー監視 | 不可能 | Prometheus + Grafana ダッシュボードで可視化 |
| フロントエンド負荷 | バックエンドを圧倒する可能性 | クライアント側でレート制限 |
| DB プール枯渇 | 全リクエスト無制限待機 | 待機上限 + 即座エラー返却で早期失敗 |
| レイテンシ劣化パターン | 負荷増加で全リクエストが遅延 | 超過分を早期リジェクトし正常リクエストを保護 |

## 影響

- `k1s0-rate-limit` は新規パッケージとして全バックエンド言語に追加（Tier 2: k1s0-observability に依存）
- 既存の `k1s0-resilience`（セマフォ、サーキットブレーカー）との棲み分けを明確化する必要がある
- `k1s0-grpc-server` / `k1s0-grpc-client` に破壊的変更なし（オプトイン API 追加のみ）
- フロントエンドパッケージへの影響は設定追加のみ（API 互換性維持）
- YAML 設定スキーマの拡張（`rate_limit`, `backpressure` セクション追加）
- Prometheus メトリクス追加に伴い Grafana ダッシュボードの更新が必要
- CI/CD ワークフローへの影響なし
