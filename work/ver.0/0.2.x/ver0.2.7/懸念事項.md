## 致命的な懸念（対処必須）

### C-01: Tera テンプレートと Kotlin 文字列テンプレートの構文競合

**深刻度: 高**

Tera の `{{ }}` と Kotlin の `${}` / `$variable` が競合。`build.gradle.kts.tera` や Kotlin ソースファイルのテンプレート化時に問題。

| 競合パターン | 期待する出力 |
|-------------|-------------|
| `"${project.name}"` | Kotlin/Gradle の文字列テンプレート |
| `implementation("io.ktor:ktor-server-core:$ktorVersion")` | Gradle の変数参照 |
| `"Hello, $name"` | Kotlin の文字列テンプレート |
| `R.string.app_name` | Android リソース参照（問題なし） |

**対処**:
1. `{% raw %}...{% endraw %}` ブロックで Kotlin/Gradle の `$` をエスケープ
2. k1s0 テンプレート変数は `{{ feature_name }}` 形式で明確に区別
3. 全 `.tera` ファイルのレンダリングテストで有効な Kotlin/Gradle コードであることを検証

---

### C-02: Gradle ビルドの CI 実行時間

**深刻度: 高**

Gradle の依存解決が遅く、CI が長くなる。Android ビルドはさらに AGP + SDK ダウンロードが加わる。

| テンプレート | 予想 CI 時間（初回） | キャッシュあり |
|-------------|:-------------------:|:------------:|
| backend-kotlin | ~3.5分 | ~2分 |
| frontend-android(Kotlin) | ~5分 | ~3分 |

**対処**:
1. `gradle/actions/setup-gradle@v4` のキャッシュ機能を活用
2. `--no-daemon` でメモリ使用量を抑制
3. `org.gradle.parallel=true` + `org.gradle.configuration-cache=true`
4. Android: `android.useAndroidX=true` + `android.enableJetifier=false` で不要な処理を削減

---

### C-03: Kotlin / Gradle / AGP バージョン互換性

**深刻度: 高**

Kotlin, Gradle, JDK, AGP の4要素の互換性管理。Android では Compose Compiler も加わり5要素に。

| 組み合わせ | 問題 |
|-----------|------|
| Kotlin 2.0 + Gradle 7.x | 非対応 |
| AGP 8.x + Gradle 7.x | 非対応 |
| Compose Compiler + Kotlin バージョン不一致 | ビルド失敗 |
| JDK 22+ + Kotlin 2.0.0 | 一部非対応 |

**対処**:
1. `gradle/libs.versions.toml` で全バージョンを一元管理
2. Gradle Wrapper でバージョン固定
3. Kotlin 2.0+ では Compose Compiler が Kotlin プラグインに統合されており、バージョン不整合が解消

---

### C-04: Android SDK の環境依存

**深刻度: 高**

Android ビルドには Android SDK が必要。CI 環境と開発者ローカル環境の両方でセットアップが必要。

**対処**:
1. **CI**: `ubuntu-latest` には Android SDK がプリインストール済み（GitHub Actions）
2. **ローカル**: `k1s0 doctor` で `ANDROID_HOME` の存在と必要な SDK バージョンを確認
3. **K020 除外**: `JAVA_HOME`, `ANDROID_HOME`, `GRADLE_HOME` はビルドツール設定のため除外
4. **commandlinetools**: SDK Manager による自動セットアップガイドを提供

## 設計上の懸念

### D-01: Ktor vs Spring Boot の選定

**深刻度: 中**

Spring Boot は最も普及した JVM フレームワークだが、テンプレートとして Ktor を選定。

| 観点 | Ktor | Spring Boot |
|------|------|-------------|
| Kotlin 親和性 | 高（Kotlin-first） | 中（Java 由来） |
| 軽量性 | 高（必要な機能のみ） | 低（多機能バンドル） |
| 起動速度 | ~1秒 | ~5秒 |
| コルーチン | ネイティブ | 部分的 |
| Docker イメージ | ~100MB | ~200MB+ |

**対処**: Ktor をテンプレートに採用。Spring Boot は `migrate` で既存プロジェクト取り込み対応。

---

### D-02: Android テンプレートの View ベース vs Compose

**深刻度: 中**

Android には従来の View ベースと Jetpack Compose の2つの UI パラダイムがある。

**対処**:
1. **テンプレートは Compose のみ**: Google 公式推奨、k1s0 の宣言的 UI 方針と合致
2. **View ベースは `migrate` で対応**: 既存の View ベースプロジェクトは migrate で取り込み
3. **minSdk 26**: Compose の実用的な最低要件。Android 8.0 (API 26) 以上をサポート

---

### D-03: パッケージ名の Java 互換性

**深刻度: 中**

Kotlin パッケージ名にハイフンが使えない。feature 名（kebab-case）→ パッケージ名の変換が必要。

**対処**: `feature_name_package` 変数でハイフン除去小文字を使用（`order-processing` → `orderprocessing`）。

---

### D-04: backend と frontend で同一言語 "kotlin"

**深刻度: 中**

`manifest.schema.json` の `language` フィールドが `"kotlin"` で共通。lint ルールで backend/frontend の区別が必要。

**対処**:
1. `service.type` (`"backend"` / `"frontend"`) で区別
2. lint ルールは `language` + `type` の組み合わせでパターンを選択
3. Android 固有パターン（`BuildConfig.`, `Log.d`, `android.*`）は `type == "frontend"` の場合のみ適用

---

## 互換性の懸念

### B-01: 既存コマンド・テンプレートへの影響

**深刻度: 低**

新規テンプレート追加のみ。既存テンプレート・コマンドへの影響なし。

---

### B-02: lint ルールの Kotlin 固有パターン

**深刻度: 中**

| Kotlin 固有機能 | lint への影響 |
|----------------|-------------|
| 文字列テンプレート `"${expr}"` | K050 検出精度に影響 |
| `!!` (non-null assertion) | K029 新パターン |
| `BuildConfig.` (Android) | K020 Android 固有 |
| `Log.d/e/i/w` (Android) | K053 Android 固有 |

**対処**: 正規表現パターンマッチで実装。Android 固有は `service.type` で分岐。

---

## 運用上の懸念

### O-01: JDK + Android SDK のインストール

**深刻度: 中**

**対処**:
- `k1s0 doctor`: `backend-kotlin` → JDK 21+ チェック、`frontend-android(Kotlin)` → JDK 21+ + Android SDK チェック
- Gradle Wrapper をテンプレートに含め、グローバル Gradle インストール不要

---

### O-02: Android の Docker 非対応

**深刻度: 低**

Android アプリは APK/AAB としてビルドされ、Docker コンテナ化は通常行わない。

**対処**:
1. frontend-android(Kotlin) テンプレートには Dockerfile を含めない
2. `k1s0 docker build` は frontend-android(Kotlin) では非対応として警告
3. `docker.yml` の matrix には含めない

---

### O-03: Gradle マルチモジュール

**深刻度: 低**

Android は `app/` モジュールを持つマルチモジュール構成が標準。backend-kotlin は単一モジュール。

**対処**:
- Android: `settings.gradle.kts` に `include(":app")` + ルートプロジェクト
- backend-kotlin: 単一モジュール（ルートプロジェクトのみ）
- `managed_paths` / `protected_paths` の調整

---

## テストの懸念

### T-01: テンプレートレンダリングの検証

**深刻度: 中**

**対処**: 全 `.tera` ファイルのレンダリングテスト + `$` エスケープ検証。

---

### T-02: Android テスト環境

**深刻度: 中**

Android UI テスト（Compose Test）は Android Emulator が必要。CI ではユニットテストのみ実行。

**対処**:
1. CI: `testDebugUnitTest` のみ実行（エミュレータ不要）
2. UI テスト: ローカル開発環境で実行（`connectedDebugAndroidTest`）
3. 将来的に Firebase Test Lab 統合を検討

---

### 設計確定事項

1. **バックエンド**: Ktor 3.x（Spring Boot は migrate で対応）
2. **Android**: Jetpack Compose（View ベースは migrate で対応）
3. **ビルド**: Gradle 8.x + Kotlin DSL（共通）
4. **JDK**: OpenJDK 21 LTS（共通）
5. **言語フィールド**: `"kotlin"` で共通、`service.type` で backend/frontend 区別
6. **Tera 競合**: `{% raw %}` ブロックで回避
7. **パッケージ名**: backend `dev.k1s0.feature.{name}`, Android `dev.k1s0.app.{name}`
8. **Android Docker**: 非対応（APK/AAB ビルド）
9. **K020 除外**: `JAVA_HOME`, `ANDROID_HOME`, `GRADLE_HOME`
10. **Android テンプレート**: マルチモジュール（`app/` モジュール）
