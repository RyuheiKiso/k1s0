# 要望

Kotlin によるバックエンドサービスおよび Android フロントエンドアプリを k1s0 で生成・管理できるようにしたい。

## 背景

現在 k1s0 が対応しているのはバックエンド4言語（Rust, Go, C#, Python）とフロントエンド2種（React, Flutter）。しかし、Kotlin/JVM エコシステムは企業開発で広く採用されており、以下の課題がある：

| 課題 | 説明 |
|------|------|
| JVM バックエンド未対応 | Java/Kotlin は企業バックエンドの大きなシェアを持つが、k1s0 で管理できない |
| Android ネイティブ未対応 | Flutter でクロスプラットフォームは対応済みだが、Android ネイティブ（Jetpack Compose）を扱えない |
| Spring Boot / Ktor の普及 | JVM で最も普及したフレームワーク群。既存チームの多くが使用 |
| Kotlin の現代性 | Null 安全、コルーチン、data class 等が Clean Architecture と親和性が高い |
| gRPC エコシステム | grpc-kotlin は公式サポートされており、k1s0 の gRPC 戦略と整合する |
| 企業導入の障壁 | Kotlin バックエンドや Android アプリを開発するチームが k1s0 を導入できない |

## 要望内容

### 1. `backend-kotlin` テンプレートの新設

```bash
# Feature サービスの生成
k1s0 new-feature --type backend-kotlin --name order-processing --with-grpc --with-db

# Domain ライブラリの生成
k1s0 new-domain --type backend-kotlin --name finance
```

### 2. `frontend-android(Kotlin)` テンプレートの新設

```bash
# Android アプリの生成
k1s0 new-feature --type frontend-android(Kotlin) --name inventory-app

# 画面の追加
k1s0 new-screen --type frontend-android(Kotlin) --screen user-profile

# Domain ライブラリの生成
k1s0 new-domain --type frontend-android(Kotlin) --name shared-ui
```

### 既存言語との対応関係

#### バックエンド

| 観点 | Rust | Go | C# | Python | **Kotlin** |
|------|------|----|----|--------|------------|
| Web フレームワーク | axum | Gin/Echo | ASP.NET Core | FastAPI | **Ktor** |
| gRPC | tonic | grpc-go | Grpc.AspNetCore | grpcio | **grpc-kotlin** |
| DB | sqlx | pgx | EF Core | SQLAlchemy | **Exposed** |
| ビルド | Cargo | Go Modules | dotnet | uv | **Gradle (Kotlin DSL)** |
| テスト | cargo test | go test | xUnit | pytest | **JUnit 5 + kotest** |
| Lint | clippy | golangci-lint | dotnet format | Ruff | **ktlint / detekt** |

#### フロントエンド

| 観点 | React | Flutter | **Android** |
|------|-------|---------|-------------|
| UI フレームワーク | React (Material-UI) | Flutter (Material 3) | **Jetpack Compose (Material 3)** |
| 言語 | TypeScript | Dart | **Kotlin** |
| 状態管理 | Zustand/Context | Riverpod | **ViewModel + StateFlow** |
| ナビゲーション | React Router | go_router | **Navigation Compose** |
| HTTP | Axios/Fetch | Dio | **Ktor Client / OkHttp** |
| ビルド | pnpm | pub | **Gradle (Kotlin DSL)** |
| テスト | Jest/Vitest | flutter test | **JUnit 5 + Compose Test** |
| Lint | ESLint | dart analyze | **ktlint / detekt** |

## 影響

- k1s0 の対応がバックエンド5言語 + フロントエンド3種に拡大
- JVM エコシステム（バックエンド + Android）を包括的にカバー
- Kotlin の言語統一性: バックエンド（Ktor）とフロントエンド（Compose）で同一言語を使用できる
- 既存の4バックエンド + 2フロントエンドと同等の機能（scaffold 生成、lint、upgrade、docker、playground）が利用可能
- `migrate analyze` が Kotlin プロジェクト（`build.gradle.kts`）と Android プロジェクトを自動検出できるようになる

## 回答（実装方針）

### 概要

- `backend-kotlin` テンプレート（feature / domain）を新設 — Ktor 3.x ベース
- `frontend-android(Kotlin)` テンプレート（feature / domain / screen）を新設 — Jetpack Compose ベース
- `framework/backend/kotlin/` に Kotlin バックエンド用 framework パッケージ群（12パッケージ）を実装
- `framework/frontend/android/` に Android 用 framework パッケージ群（8パッケージ）を実装
- CLI の lint ルール（K020, K022, K029 等）に Kotlin パターンを追加
- CI/CD ワークフロー（`kotlin.yml`, `frontend-android(Kotlin).yml`）を追加
- `manifest.schema.json` の `language` enum に `"kotlin"` を追加
- `migrate analyze` の種別推定に Kotlin/Android パターンを追加

### 技術選定

#### バックエンド (backend-kotlin)

| 項目 | 選定 | 理由 |
|------|------|------|
| 言語 | Kotlin 2.0+ | K2 コンパイラ安定化 |
| Web | Ktor 3.x | 軽量、コルーチンネイティブ、Kotlin-first |
| gRPC | grpc-kotlin 1.4+ | 公式 Kotlin gRPC |
| DB | Exposed 0.50+ | Kotlin DSL、型安全クエリ |
| ビルド | Gradle 8.x (Kotlin DSL) | Kotlin 標準 |
| DI | Koin 4.x | 軽量、アノテーション不要 |
| JDK | 21 (LTS) | 最新 LTS |

#### フロントエンド (frontend-android(Kotlin))

| 項目 | 選定 | 理由 |
|------|------|------|
| 言語 | Kotlin 2.0+ | K2 コンパイラ安定化 |
| UI | Jetpack Compose + Material 3 | Android 公式宣言的 UI |
| アーキテクチャ | MVVM + Clean Architecture | Android 公式推奨 |
| 状態管理 | ViewModel + StateFlow/SharedFlow | Kotlin コルーチン統合 |
| ナビゲーション | Navigation Compose | Jetpack 公式 |
| HTTP | Ktor Client | Kotlin-first、コルーチン対応 |
| DI | Koin 4.x (Android) | バックエンドと統一 |
| ビルド | Gradle 8.x (Kotlin DSL) + AGP | Android 標準 |
| テスト | JUnit 5 + Compose UI Test | Android 公式テストフレームワーク |
| minSdk | 26 (Android 8.0) | Compose の実用的な最低要件 |

### 実装順序

1. **Phase 1: バックエンドテンプレート** — `CLI/templates/backend-kotlin/` の feature/domain テンプレート
2. **Phase 2: バックエンド Framework** — `framework/backend/kotlin/` に Tier 1-3 パッケージ
3. **Phase 3: Android テンプレート** — `CLI/templates/frontend-android(Kotlin)/` の feature/domain/screen テンプレート
4. **Phase 4: Android Framework** — `framework/frontend/android/` にパッケージ
5. **Phase 5: CLI 統合** — lint ルール拡張、manifest スキーマ更新、種別推定追加
6. **Phase 6: CI/CD + Docker** — ワークフロー、Dockerfile
7. **Phase 7: テスト + ドキュメント**

## 期待される効果

1. JVM エコシステムのチームが k1s0 を導入可能になる（バックエンド + Android）
2. Kotlin によるバックエンド・フロントエンド言語統一が可能
3. 対応7テンプレートにより、k1s0 のプラットフォームとしての汎用性が大幅に向上
4. Android ネイティブアプリ開発チームのオンボーディング
5. 既存の Kotlin/Android プロジェクトの `migrate` による取り込みも可能
