## 致命的な懸念（対処必須）

### C-01: 既存コードの破損リスク

**深刻度: 高**

`migrate apply` はディレクトリの移動・ファイルの書き換えを伴う破壊的操作。誤ったステップの実行により、既存プロジェクトのビルドが壊れる可能性がある。

| リスクシナリオ | 影響 | 発生可能性 |
|--------------|------|:----------:|
| ディレクトリ移動後に import パスが壊れる | ビルド不能 | 高 |
| config.yaml 変換ミスで設定値が欠落 | ランタイムエラー | 中 |
| apply 途中でエラー発生、中途半端な状態 | ビルド不能 | 中 |
| バックアップなしで apply を実行 | 復旧不能 | 低（UI で防止） |

**対処**:

1. **バックアップの強制化**: `apply` 実行前に `.k1s0/backup/{timestamp}/` にプロジェクト全体をコピー。`--skip-backup` は非推奨警告付き
2. **dry-run の提供**: `apply --dry-run` で実行内容を事前確認
3. **段階的実行**: `apply --phase phase-2` で特定フェーズのみ実行可能
4. **進捗保存**: 各ステップ完了後に `migration-plan.json` を更新。中断からの再開が可能
5. **ロールバック案内**: 失敗時にバックアップからの復元コマンドを表示

```rust
fn create_backup(project_path: &Path) -> Result<PathBuf> {
    let timestamp = chrono::Local::now().format("%Y%m%d-%H%M%S");
    let backup_dir = project_path.join(format!(".k1s0/backup/{}", timestamp));

    // .k1s0/backup/ と .git/ を除外してコピー
    copy_dir_recursive(project_path, &backup_dir, &[".k1s0/backup", ".git", "target", "node_modules"])?;

    output().success(&format!("バックアップを作成しました: {}", backup_dir.display()));
    Ok(backup_dir)
}
```

### C-02: プロジェクト種別の誤推定

**深刻度: 高**

自動推定が誤った場合、間違ったテンプレートタイプに基づいて移行計画が生成される。

| 誤推定パターン | 例 | 結果 |
|--------------|---|------|
| ライブラリ vs サービス | `Cargo.toml` + `src/lib.rs` のみ | backend-rust と推定するがサービスではない |
| モノレポの部分一致 | ルートに `Cargo.toml` + `package.json` 両方 | 言語の判定が曖昧 |
| フレームワーク未使用 | 標準ライブラリのみの Go プロジェクト | フレームワーク不明 |
| テストプロジェクト | `src/main.rs` がテストランナー | 通常のサービスと誤推定 |

**対処**:

1. **確信度(confidence)の表示**: 推定結果に確信度を付与し、低い場合は警告
2. **--type ヒント**: ユーザーが明示的にタイプを指定可能
3. **確認プロンプト**: analyze/plan で推定結果をユーザーに確認

```rust
fn detect_project_type(path: &Path, type_hint: Option<&str>) -> Result<DetectedProjectType> {
    let detected = auto_detect(path)?;

    if let Some(hint) = type_hint {
        if hint != detected.template_type {
            output().warning(&format!(
                "検出結果 '{}' と指定タイプ '{}' が異なります。指定タイプを優先します。",
                detected.template_type, hint
            ));
            return Ok(DetectedProjectType { template_type: hint.to_string(), confidence: 1.0, ..detected });
        }
    }

    if detected.confidence < 0.7 {
        output().warning(&format!(
            "プロジェクトタイプの推定確信度が低い ({:.0}%) です。\n\
             --type オプションで明示的に指定することを推奨します。\n\
             検出結果: {} ({})",
            detected.confidence * 100.0,
            detected.template_type,
            detected.framework.as_deref().unwrap_or("framework unknown")
        ));
    }

    Ok(detected)
}
```

### C-03: import/use パスの自動更新

**深刻度: 高**

ディレクトリ移動後、ソースコード内の `use`/`import`/`#include` 文のパスが壊れる。

| 言語 | import 形式 | 移動時の影響 |
|------|-----------|-------------|
| Rust | `use crate::handlers::*;` | モジュールパス変更が必要 |
| Go | `import "myproject/db"` | パッケージパス変更が必要 |
| C# | `using MyProject.Handlers;` | namespace 変更が必要 |
| Python | `from .db import repository` | 相対インポート変更が必要 |
| TypeScript | `import { api } from '../api'` | 相対パス変更が必要 |
| Dart | `import 'package:app/db/repo.dart'` | パッケージパス変更が必要 |

**対処**:

1. **移動前にインポートマップを構築**: ディレクトリ移動対象のファイルを参照しているファイルを事前にスキャン
2. **ReplaceInFile ステップの自動生成**: 各移動ステップに対応する import 書き換えステップを生成
3. **移動後のビルド確認ステップ**: Phase 5 で `cargo build` / `go build` 等を実行し、未修正のインポートを検出

```rust
fn generate_import_fix_steps(
    move_step: &MigrationStep,
    project_path: &Path,
    language: &str,
) -> Vec<MigrationStep> {
    let MigrationAction::MoveDirectory { from, to } = &move_step.action else { return vec![] };

    // 移動元パスを参照しているファイルをスキャン
    let old_module = path_to_module(from, language);
    let new_module = path_to_module(to, language);

    let referencing_files = find_files_referencing(project_path, &old_module, language);

    referencing_files.iter().map(|file| {
        MigrationStep {
            action: MigrationAction::ReplaceInFile {
                path: file.clone(),
                pattern: old_module.clone(),
                replacement: new_module.clone(),
                occurrences: count_occurrences(file, &old_module),
            },
            auto: true,
            risk: StepRisk::Medium,
            ..Default::default()
        }
    }).collect()
}
```

4. **完全な自動化が困難なケース**: 動的インポート、リフレクション、文字列ベースのモジュール参照は `ManualAction` として計画に含める

### C-04: .env → config.yaml 変換の正確性

**深刻度: 中**

環境変数の用途をファイル名だけから正確に推定することは困難。

| 困難なパターン | 例 | 問題 |
|--------------|---|------|
| 独自の変数名 | `MY_APP_SPECIAL_FLAG=true` | config.yaml のどのフィールドに対応するか不明 |
| URL 形式の値 | `DATABASE_URL=postgres://...` | パース可能だが複雑 |
| 複合的な値 | `ALLOWED_ORIGINS=http://a.com,http://b.com` | 配列への変換が必要 |
| 秘密値の判定 | `API_KEY` vs `API_VERSION` | KEY は秘密値だが VERSION は違う |

**対処**:

1. **既知パターンの辞書**: `DATABASE_URL`, `REDIS_URL`, `PORT`, `HOST` 等の一般的な変数名は正確に変換
2. **秘密値の推定ルール**: 変数名に `SECRET`, `PASSWORD`, `KEY`, `TOKEN`, `CREDENTIAL` を含む場合は `*_file` に変換
3. **不明な変数のフォールバック**: `app.{variable_name_lowercase}` に配置し、`ManualAction` で確認を促す
4. **URL パーサーの提供**: `DATABASE_URL` 等の URL 形式は構造体に自動分解

```rust
fn convert_env_to_config(env_vars: &[(String, String)]) -> (ConfigYaml, Vec<MigrationStep>) {
    let mut config = ConfigYaml::default();
    let mut manual_steps = Vec::new();

    for (key, value) in env_vars {
        match classify_env_var(key) {
            EnvVarClass::ServerPort => config.server.port = value.parse().unwrap_or(8080),
            EnvVarClass::DatabaseUrl => {
                if let Ok(parsed) = parse_database_url(value) {
                    config.database = parsed;
                    // password は *_file に変換
                    config.database.password_file = Some("/var/run/secrets/k1s0/db_password".into());
                }
            }
            EnvVarClass::Secret(field_name) => {
                config.set_secret_file(&field_name, &format!("/var/run/secrets/k1s0/{}", field_name));
            }
            EnvVarClass::Unknown => {
                config.app.insert(key.to_lowercase(), value.clone());
                manual_steps.push(MigrationStep {
                    action: MigrationAction::ManualAction {
                        description: format!(
                            "環境変数 {} の config.yaml 内での適切な配置を確認してください。\n\
                             現在 app.{} に仮配置されています。",
                            key, key.to_lowercase()
                        ),
                        ..Default::default()
                    },
                    auto: false,
                    risk: StepRisk::Low,
                    ..Default::default()
                });
            }
        }
    }

    (config, manual_steps)
}
```

## 設計上の懸念

### D-01: 分析の精度 vs 実装コスト

**深刻度: 中**

AST 解析を使えば高精度な分析が可能だが、各言語のパーサーを実装するコストが大きい。

| アプローチ | 精度 | 実装コスト | メンテナンスコスト |
|-----------|:----:|:---------:|:----------------:|
| AST 解析 | 高 | 非常に高（言語ごとにパーサー必要） | 高 |
| 正規表現パターンマッチ | 中 | 低 | 低 |
| 既存 lint ルールの再利用 | 中〜高 | 低（既に実装済み） | 低 |
| Language Server Protocol | 高 | 高（LSP サーバー起動が必要） | 中 |

**対処**: v0.2.6 では正規表現ベース + 既存 lint ルールの再利用で実装する。

- 既存の K020, K021, K022, K050, K053, K060 ルールは正規表現ベースで動作しており、十分な精度を持つ
- 誤検知のリスクは `ManualAction` ステップでカバー（自動修正せず人間に委ねる）
- 将来バージョンで AST 解析への段階的移行が可能（analyzer.rs のインターフェースは同一）

### D-02: Clean Architecture 層の推定精度

**深刻度: 中**

既存プロジェクトのディレクトリ名は多様で、k1s0 の4層（domain/application/infrastructure/presentation）への正確なマッピングが困難。

| 既存の一般的な名前 | k1s0 マッピング | 確信度 |
|-------------------|----------------|:------:|
| `models/`, `entities/` | domain/ | 高 |
| `services/`, `usecases/` | application/ | 高 |
| `handlers/`, `controllers/`, `routes/`, `api/` | presentation/ | 高 |
| `db/`, `repositories/`, `external/` | infrastructure/ | 高 |
| `utils/`, `helpers/` | 判断困難 | 低 |
| `lib/`, `common/`, `shared/` | 判断困難 | 低 |
| `src/` (フラット構造、層分離なし) | 全て手動仕分け | 低 |

**対処**:

1. **名前ベースの推定辞書**: 一般的なディレクトリ名 → k1s0 層のマッピング辞書を用意
2. **ファイル内容ベースの補助推定**: HTTP ハンドラ関数を含む → presentation、SQL クエリを含む → infrastructure
3. **推定不可能な場合**: `ManualAction` ステップとして計画に含め、ユーザーに判断を委ねる
4. **フラット構造の場合**: 全ファイルを一覧表示し、インタラクティブに仕分けを提案

```rust
fn map_directory_to_layer(dir_name: &str, file_contents: Option<&str>) -> Option<ArchitectureLayer> {
    // 名前ベースの推定
    match dir_name.to_lowercase().as_str() {
        "domain" | "models" | "entities" | "value_objects" => Some(ArchitectureLayer::Domain),
        "application" | "services" | "usecases" | "use_cases" => Some(ArchitectureLayer::Application),
        "infrastructure" | "db" | "database" | "repositories" | "external" | "adapters" | "persistence"
            => Some(ArchitectureLayer::Infrastructure),
        "presentation" | "handlers" | "controllers" | "routes" | "api" | "grpc" | "rest" | "web"
            => Some(ArchitectureLayer::Presentation),
        _ => {
            // ファイル内容ベースの補助推定
            if let Some(content) = file_contents {
                if contains_http_handler_patterns(content) { return Some(ArchitectureLayer::Presentation); }
                if contains_sql_patterns(content) { return Some(ArchitectureLayer::Infrastructure); }
                if contains_business_rule_patterns(content) { return Some(ArchitectureLayer::Domain); }
            }
            None
        }
    }
}
```

### D-03: 移行計画 JSON の可読性と編集性

**深刻度: 低**

移行計画 JSON はチームレビューに使われるため、人間が読み書きできる形式であるべき。

**対処**:

1. JSON は `serde_json::to_string_pretty()` でフォーマット
2. 各ステップに `description` を日本語で付与
3. `ManualAction` にはドキュメント参照先（`reference` フィールド）を含める
4. `--format yaml` オプションで YAML 出力も検討（将来）

### D-04: 大規模プロジェクトのスキャン性能

**深刻度: 低**

大規模プロジェクト（10,000ファイル以上）のスキャンに時間がかかる可能性。

**対処**:

1. `.gitignore` パターンに従ってスキャン対象を絞る
2. `target/`, `node_modules/`, `.git/`, `vendor/` 等のビルド成果物ディレクトリを除外
3. ファイルサイズ上限（1MB）を超えるファイルはスキップ
4. 進捗表示（スピナー + ファイル数カウント）

```rust
const EXCLUDED_DIRS: &[&str] = &[
    "target", "node_modules", ".git", "vendor", "bin", "obj",
    "__pycache__", ".venv", "venv", ".dart_tool", "build",
    ".k1s0/backup",
];

const MAX_FILE_SIZE: u64 = 1_048_576; // 1 MB
```

## 互換性の懸念

### B-01: 既存コマンドへの影響

**深刻度: 低**

`migrate` は完全に新規のサブコマンド。既存コマンドへの影響なし。

| 影響範囲 | 影響 |
|---------|------|
| `k1s0 lint` | 影響なし。migrate apply 完了後にそのまま lint 可能 |
| `k1s0 upgrade` | 影響なし。migrate で生成した manifest.json は upgrade 互換 |
| `k1s0 new-feature` | 影響なし |
| `k1s0 playground` | 影響なし |
| `k1s0 doctor` | 影響なし |

**対処**: 統合テストで既存コマンドとの共存を確認。

---

### B-02: lint エンジンの拡張互換性

**深刻度: 中**

analyzer が lint エンジンを「非 k1s0 プロジェクト」に適用するため、lint エンジン内の前提（manifest.json の存在、feature/ 配下のパス等）に引きずられる可能性。

**対処**:

1. `AnalyzerConfig` の `relaxed_mode` で manifest/構造前提のルール（K001-K011, K040-K047）をスキップ
2. 各 lint ルールの `check()` メソッドにパスのルートを注入可能にする
3. k1s0-generator の lint API に `scan_arbitrary_path()` を追加

```rust
// 既存 API（k1s0 プロジェクト前提）
pub fn lint(project_root: &Path, rules: &[String]) -> Vec<LintDiagnostic>;

// 新規 API（任意パス対応）
pub fn scan_violations(
    scan_root: &Path,
    rules: &[String],
    config: &AnalyzerConfig,
) -> Vec<Violation>;
```

## 運用上の懸念

### O-01: バックアップのディスク使用量

**深刻度: 中**

大規模プロジェクトのバックアップは数 GB に達する可能性。

| プロジェクト規模 | ソースコード | 依存関係込み | バックアップサイズ |
|----------------|:----------:|:----------:|:----------------:|
| 小規模（100ファイル） | ~5 MB | ~500 MB | ~5 MB |
| 中規模（1,000ファイル） | ~50 MB | ~2 GB | ~50 MB |
| 大規模（10,000ファイル） | ~500 MB | ~5 GB | ~500 MB |

**対処**:

1. `target/`, `node_modules/`, `.git/`, `vendor/` 等のビルド成果物を除外
2. バックアップ前にサイズ見積もりを表示
3. ディスク空き容量チェック（バックアップサイズの2倍を確保）
4. `apply --skip-backup` オプション（非推奨警告付き）
5. 古いバックアップの自動削除は v0.2.6 では対象外（将来 `migrate cleanup` で対応）

---

### O-02: 複数回の apply 実行

**深刻度: 中**

ユーザーが `apply` を複数回実行した場合の挙動。

| シナリオ | 望ましい挙動 |
|---------|-------------|
| 全ステップ完了後に再実行 | 「全ステップ完了済み」と表示して終了 |
| 途中で失敗して再実行 | 完了済みステップをスキップし、失敗ステップから再開 |
| plan を再生成して apply | 新しい計画で最初から実行 |
| apply 後にユーザーが手動変更 | 手動変更を壊さないよう、完了済みステップはスキップ |

**対処**:

1. `migration-plan.json` の各ステップに `status` フィールドを持たせる
2. `apply` 実行時に `status: Completed` のステップをスキップ
3. `status: Failed` のステップは再実行を試みる
4. ファイルハッシュの比較で、計画生成後の変更を検出（警告表示）

---

### O-03: Git ワークフローとの統合

**深刻度: 低**

`apply` によるファイル移動は Git の履歴追跡に影響する。

**対処**:

1. `apply` 実行前に `git status` で未コミットの変更がないことを確認
2. 未コミットの変更がある場合は警告（`--force` で無視可能）
3. ディレクトリ移動は `git mv` を使用（Git の追跡を維持）
4. apply 完了後のコミットメッセージ例を提示

```rust
fn execute_move(from: &Path, to: &Path) -> Result<()> {
    // Git リポジトリ内の場合は git mv を使用
    if is_git_repo(from) {
        Command::new("git")
            .args(["mv", &from.to_string_lossy(), &to.to_string_lossy()])
            .status()?;
    } else {
        fs::rename(from, to)?;
    }
    Ok(())
}
```

## テストの懸念

### T-01: テストフィクスチャの多様性

**深刻度: 中**

実際の既存プロジェクトの構造は多様で、テストフィクスチャでカバーしきれない。

**対処**:

1. 最低限のフィクスチャセット:
   - `rust-axum-basic/` — Clean Architecture 未適用の典型的な Rust プロジェクト
   - `rust-clean-architecture/` — 既に Clean Architecture 適用済み（高スコア期待）
   - `go-gin-basic/` — Go の典型的なプロジェクト
   - `python-fastapi-basic/` — Python の典型的なプロジェクト
   - `flat-structure/` — 層分離なしのフラット構造
2. 各フィクスチャに期待される分析結果（expected.json）を付与
3. フィクスチャは最小限のファイルで構成（ビルド不要、構造のみ）

---

### T-02: apply のべき等性テスト

**深刻度: 中**

`apply` を複数回実行してもプロジェクトが壊れないことを保証する必要がある。

**対処**:

1. 統合テストで「apply → apply → ビルド確認」のパターンをテスト
2. 2回目の apply では全ステップが `Completed` としてスキップされることを確認
3. `execute_move()` で移動先が既に存在する場合のハンドリング
