# k1s0 v0.2.6 改修方針

## 概要

`k1s0 migrate` コマンドを新設し、以下の機能を実装する：

- `migrate analyze` — 既存プロジェクトの k1s0 準拠度を非破壊的に診断
- `migrate plan` — 分析結果から具体的な移行計画 JSON を生成
- `migrate apply` — 計画に基づいてディレクトリ構造変換・ファイル生成を実行
- `migrate status` — 移行の進捗と残りの作業を表示

## 機能目的

1. 既存プロジェクトを k1s0 管理下に段階的に取り込む
2. `analyze` 単独でも規約準拠度の診断ツールとして機能させる
3. 移行の安全性を保証する（バックアップ + dry-run + ロールバック）
4. 移行計画を JSON で出力し、チームレビュー・CI 統合を可能にする
5. `new-feature` のグリーンフィールド前提を補完する

## プロジェクト種別推定

### 検出ロジック

既存プロジェクトのルートディレクトリから、ビルドファイル・設定ファイルのパターンマッチで言語とフレームワークを推定する。

| 検出ファイル | 推定結果 | 確信度 |
|-------------|---------|:------:|
| `Cargo.toml` + `src/main.rs` | backend-rust | 95% |
| `Cargo.toml` + `src/lib.rs` (のみ) | backend-rust (library) | 80% |
| `go.mod` + `cmd/` | backend-go | 95% |
| `go.mod` + `main.go` | backend-go | 85% |
| `*.sln` + `*.csproj` | backend-csharp | 95% |
| `pyproject.toml` + `src/` | backend-python | 90% |
| `requirements.txt` + `main.py` | backend-python (legacy) | 70% |
| `package.json` + `tsconfig.json` + `src/App.tsx` | frontend-react | 95% |
| `package.json` + `tsconfig.json` + `src/main.ts` | frontend-react | 80% |
| `pubspec.yaml` + `lib/` | frontend-flutter | 95% |

### フレームワーク推定（バックエンド）

Rust の場合、`Cargo.toml` の依存関係を解析してフレームワークを特定する。

```rust
struct FrameworkDetection {
    name: String,           // "axum", "actix-web", "tonic" 等
    version: String,
    k1s0_compatible: bool,  // k1s0 framework crate との互換性
}
```

| 依存クレート | 推定フレームワーク | k1s0 互換性 |
|-------------|-------------------|:-----------:|
| `axum` | axum (推奨) | 高 |
| `actix-web` | Actix Web | 中（移行ガイド付き） |
| `tonic` | gRPC (tonic) | 高 |
| `sqlx` | SQLx | 高 |
| `diesel` | Diesel | 低（SQLx への移行推奨） |
| `sea-orm` | SeaORM | 中 |

Go, C#, Python についても同様のパターンマッチを実装する。

#### 1.3 分析結果データ構造

```rust
#[derive(Debug, Serialize, Deserialize)]
pub struct AnalysisResult {
    pub project_path: PathBuf,
    pub detected_type: DetectedProjectType,
    pub scores: ComplianceScores,
    pub structure: StructureAnalysis,
    pub violations: Vec<Violation>,
    pub dependencies: DependencyAnalysis,
    pub migration_estimate: MigrationEstimate,
    pub analyzed_at: String,
}

#[derive(Debug, Serialize, Deserialize)]
pub struct DetectedProjectType {
    pub template_type: String,       // "backend-rust", "backend-go" 等
    pub language: String,            // "rust", "go", "csharp", "python", "typescript", "dart"
    pub framework: Option<String>,   // "axum", "gin", "asp.net-core", "fastapi" 等
    pub build_system: String,        // "cargo", "go-modules", "dotnet", "uv", "pnpm", "pub"
    pub confidence: f64,             // 0.0 - 1.0
}

#[derive(Debug, Serialize, Deserialize)]
pub struct ComplianceScores {
    pub total: u32,                  // 0-100
    pub structure: u32,              // 0-100
    pub convention: u32,             // 0-100
    pub compatibility: u32,          // 0-100
}

#[derive(Debug, Serialize, Deserialize)]
pub struct StructureAnalysis {
    pub layers: Vec<LayerStatus>,
    pub required_dirs: Vec<DirStatus>,
    pub required_files: Vec<FileStatus>,
    pub unnecessary_files: Vec<PathBuf>,  // .env 等
}

#[derive(Debug, Serialize, Deserialize)]
pub struct LayerStatus {
    pub name: String,                // "domain", "application", "infrastructure", "presentation"
    pub expected_path: String,       // "src/domain/"
    pub actual_path: Option<String>, // "src/models/" (検出された実際のパス)
    pub status: LayerDetectionStatus,
}

#[derive(Debug, Serialize, Deserialize)]
pub enum LayerDetectionStatus {
    Correct,                         // 正しいパスに存在
    Renamed { suggested_move: String }, // 別名で存在（移動が必要）
    Missing,                         // 存在しない
    Merged { with: String },         // 他の層と混在
}

#[derive(Debug, Serialize, Deserialize)]
pub struct Violation {
    pub rule_id: String,             // "K020", "K021" 等
    pub severity: String,            // "error", "warning"
    pub file: PathBuf,
    pub line: Option<usize>,
    pub message: String,
    pub auto_fixable: bool,          // migrate apply で自動修正可能か
    pub fix_suggestion: Option<String>,
}

#[derive(Debug, Serialize, Deserialize)]
pub struct MigrationEstimate {
    pub phases: Vec<MigrationPhaseEstimate>,
    pub total_auto_fixable: usize,
    pub total_manual_fixes: usize,
    pub total_file_moves: usize,
    pub total_file_generations: usize,
}
```

  │     │   └─ Step: deploy/ ディレクトリの生成（Kustomize テンプレート）
  │     │
  │     ├─ Phase 4: 規約違反の自動修正

  │     │   ├─ Step: ハードコード秘密値 → *_file 参照への書き換え（K021）
  │     │   └─ Step: Dockerfile FROM の image pinning（K060）
  │     │
  │     └─ Phase 5: 検証
  │         ├─ Step: k1s0 lint の実行
  │         ├─ Step: ビルド確認
  │         └─ Step: 残存する手動修正項目の一覧出力
  │
  ├─ 4. ステップ間の依存関係解決
  │     └─ Phase 2 → Phase 3 → Phase 4 → Phase 5 の順序制約
  │
  └─ 5. 計画ファイル出力
        ├─ migration-plan.json に書き出し
        └─ サマリーを標準出力に表示

#### 2.3 移行計画データ構造

```rust
#[derive(Debug, Serialize, Deserialize)]
pub struct MigrationPlan {
    pub version: String,             // "1.0"
    pub project_path: PathBuf,
    pub feature_name: String,        // kebab-case
    pub template_type: String,
    pub analysis: AnalysisResult,    // 元の分析結果を保持
    pub phases: Vec<MigrationPhase>,
    pub created_at: String,
}

#[derive(Debug, Serialize, Deserialize)]
pub struct MigrationPhase {
    pub id: String,                  // "phase-1", "phase-2" 等
    pub name: String,                // "バックアップ", "ディレクトリ構造の整理" 等
    pub description: String,
    pub steps: Vec<MigrationStep>,
    pub depends_on: Vec<String>,     // 先行フェーズ ID
}

#[derive(Debug, Serialize, Deserialize)]
pub struct MigrationStep {
    pub id: String,                  // "step-2-1", "step-2-2" 等
    pub action: MigrationAction,
    pub description: String,
    pub auto: bool,                  // apply で自動実行可能か
    pub risk: StepRisk,              // "low", "medium", "high"
    pub status: StepStatus,          // "pending", "completed", "skipped", "failed"
}

#[derive(Debug, Serialize, Deserialize)]
#[serde(tag = "type")]
pub enum MigrationAction {
    Backup {
        source: PathBuf,
        destination: PathBuf,
    },
    CreateDirectory {
        path: PathBuf,
    },
    MoveDirectory {
        from: PathBuf,
        to: PathBuf,
    },
    MoveFile {
        from: PathBuf,
        to: PathBuf,
    },
    GenerateFile {
        path: PathBuf,
        template: String,            // テンプレート名
        context: serde_json::Value,  // Tera コンテキスト
    },
    ReplaceInFile {
        path: PathBuf,
        pattern: String,             // 検索パターン（正規表現）
        replacement: String,
        occurrences: usize,          // 期待される置換回数
    },
    DeleteFile {
        path: PathBuf,
        reason: String,              // ".env は config/dev.yaml に移行済み" 等
    },
    RunCommand {
        command: String,
        args: Vec<String>,
        working_dir: PathBuf,
        description: String,
    },
    ManualAction {
        description: String,
        file: Option<PathBuf>,
        line: Option<usize>,
        reference: Option<String>,   // ドキュメント参照先
    },
}

#[derive(Debug, Serialize, Deserialize)]
pub enum StepRisk {
    Low,      // ディレクトリ作成、ファイル生成
    Medium,   // ディレクトリ移動、ファイル内置換
    High,     // ファイル削除、大規模リファクタリング
}

#[derive(Debug, Serialize, Deserialize)]
pub enum StepStatus {
    Pending,
    Completed,
    Skipped,
    Failed { error: String },
}
```

### 3. `migrate apply`

#### 3.1 引数定義

```rust
#[derive(Args, Debug)]
pub struct ApplyArgs {
    /// 対象プロジェクトパス
    #[arg(short = 'p', long, default_value = ".")]
    pub path: PathBuf,

    /// 移行計画ファイル
    #[arg(long, default_value = "migration-plan.json")]
    pub plan: PathBuf,

    /// 特定のフェーズのみ実行
    #[arg(long)]
    pub phase: Option<String>,

    /// dry-run（変更せずに何が行われるか表示）
    #[arg(long)]
    pub dry_run: bool,

    /// 確認プロンプトをスキップ
    #[arg(short = 'y', long)]
    pub yes: bool,

    /// バックアップをスキップ（非推奨）
    #[arg(long)]
    pub skip_backup: bool,
}
```

#### 3.2 処理フロー

```
apply --plan migration-plan.json
  │
  ├─ 1. 計画ファイル読み込み
  │     ├─ migration-plan.json のパース
  │     ├─ バージョン互換性チェック
  │     └─ 既に完了したステップの除外
  │
  ├─ 2. 事前検証
  │     ├─ 計画作成後にファイルが変更されていないか確認
  │     │   └─ project_path のファイルハッシュと plan 作成時のハッシュを比較
  │     ├─ 移動先パスの競合チェック
  │     └─ ディスク空き容量チェック（バックアップ分）
  │
  ├─ 3. dry-run 表示（--dry-run 時）
  │     ├─ 実行予定のステップを一覧表示
  │     └─ 早期リターン
  │
  ├─ 4. 確認プロンプト（--yes 未指定時）
  │     ├─ 変更内容のサマリー表示
  │     ├─ リスクの高いステップをハイライト
  │     └─ 続行確認
  │
  ├─ 5. バックアップ（Phase 1）
  │     ├─ .k1s0/backup/{timestamp}/ にプロジェクト全体をコピー
  │     ├─ バックアップの完了確認
  │     └─ バックアップパスを表示
  │
  ├─ 6. フェーズ順に実行
  │     │
  │     ├─ 各ステップの実行
  │     │   ├─ auto: true → 自動実行
  │     │   ├─ auto: false → スキップ + 手動作業リストに追加
  │     │   ├─ 成功 → status を Completed に更新
  │     │   └─ 失敗 → status を Failed に更新、後続ステップを中断
  │     │
  │     ├─ 各フェーズ完了後に migration-plan.json を更新（進捗保存）
  │     │
  │     └─ 失敗時のロールバック提案
  │         └─ "バックアップからの復元: cp -r .k1s0/backup/{timestamp}/* ."
  │
  └─ 7. 結果サマリー
        ├─ 完了したステップ数 / 全ステップ数
        ├─ 残りの手動作業リスト
        ├─ 次のアクション案内（k1s0 lint, k1s0 migrate status）
        └─ migration-plan.json を更新保存
```

## 分析エンジン設計（analyzer.rs）

### アーキテクチャ

分析エンジンは `k1s0-generator` クレートに配置する。既存の lint エンジンを再利用しつつ、非 k1s0 プロジェクトにも適用できるよう拡張する。

```
k1s0-generator/src/
├── analyzer.rs              # 新規: 分析エンジン本体
├── analyzer/
│   ├── mod.rs               # 分析エンジンモジュール
│   ├── detector.rs          # プロジェクト種別推定
│   ├── structure.rs         # ディレクトリ構造分析
│   ├── convention.rs        # 規約違反スキャン（lint 再利用）
│   ├── dependency.rs        # 依存関係分析
│   └── scorer.rs            # スコア算出
├── lint/                    # 既存: lint エンジン
│   ├── rules/               # 既存: K001-K060
│   └── ...
└── template.rs              # 既存: テンプレートエンジン
```

### lint エンジンとの統合

既存の lint ルール（K020, K021, K022, K025, K029, K050, K053, K060）を非 k1s0 プロジェクトに適用する。ただし、以下の調整が必要：

| lint ルール | 現在の前提 | migrate での調整 |
|------------|-----------|-----------------|
| K001-K011 | manifest.json 必須 | スキップ（migrate の目的が manifest 生成のため） |
| K020 | feature/ 配下のファイル | 任意パスのファイルに適用 |
| K021 | config/*.yaml | 任意の YAML/TOML/JSON に適用 |
| K022 | src/ 配下の Clean Architecture 構造 | 検出された構造に対して適用 |
| K030-K032 | gRPC 設定 | 検出時のみ適用 |
| K040-K047 | manifest.json のドメイン依存 | スキップ |
| K050 | SQL を含むファイル | そのまま適用 |
| K060 | Dockerfile | そのまま適用 |

    pub skip_rules: Vec<String>,
    /// ファイルスキャンのルートパス（通常は project_path）

}

## 環境変数 → config.yaml 自動変換

### .env ファイルからの変換

`analyze` で検出した `.env` ファイルと `std::env::var()` 呼び出しから、`config/default.yaml` を自動生成する。

```
.env ファイル:
  PORT=8080
  DATABASE_URL=postgres://user:pass@localhost:5432/mydb
  REDIS_URL=redis://localhost:6379
  JWT_SECRET=my-secret-key

→ 変換後 config/default.yaml:
  server:
    host: "0.0.0.0"
    port: 8080
  database:
    driver: postgres
    host: localhost
    port: 5432
    name: mydb
    user: user
    password_file: /var/run/secrets/k1s0/db_password
  cache:
    driver: redis
    host: localhost
    port: 6379
  auth:
    jwt_secret_file: /var/run/secrets/k1s0/jwt_secret
```

### 変換ルール

| .env パターン | config.yaml マッピング | 秘密値処理 |
|--------------|----------------------|:----------:|
| `PORT` | `server.port` | - |
| `HOST` | `server.host` | - |
| `DATABASE_URL` | `database.*` (パース) | password → `*_file` |
| `DB_HOST`, `DB_PORT` 等 | `database.*` | - |
| `REDIS_URL` | `cache.*` (パース) | - |
| `*_SECRET`, `*_KEY` | 対応するフィールド | → `*_file` に変換 |
| `*_PASSWORD` | 対応するフィールド | → `*_file` に変換 |
| その他 | `app.{env_name_lowercase}` | - |

## 注意事項

- `analyze` と `plan` は完全に非破壊的。ファイルシステムの読み取りのみ（plan は JSON 出力のみ）
- `apply` は必ずバックアップを作成する（`--skip-backup` で回避可能だが非推奨として警告）
- 移行計画 JSON は人間が読み書き可能。手動でステップの追加・削除・順序変更が可能
- 環境変数 → config.yaml の変換は完全自動化を目指すが、複雑なケースは `ManualAction` として計画に含める
- AST 解析は v0.2.6 では行わない。正規表現ベースのパターンマッチで実装し、精度向上は将来バージョンで対応
- ファイル移動時は `use` 文 / `import` 文の更新を `ReplaceInFile` ステップとして計画に含める
- Windows / macOS / Linux でのパス区切り文字の違いに注意（PathBuf を使用）
- `apply` の実行途中で失敗した場合、`migration-plan.json` にはそこまでの進捗が記録される。再実行で完了済みステップはスキップされる
