# 要望

既存のプロジェクト（Rust/Go/C#/Python/React/Flutter）を k1s0 管理下に取り込む仕組みがほしい。

## 背景

現在 k1s0 で管理下のサービスを作成するには `k1s0 new-feature` でゼロからスキャフォールドを生成する必要がある。しかし、多くのチームは既にコードベースを持っており、以下の課題がある：

| 課題 | 説明 |
|------|------|
| グリーンフィールド前提 | `new-feature` は空のディレクトリにのみ生成可能。既存コードがある場所では使えない |
| 手動移行コストが大きい | 既存プロジェクトを k1s0 構造に手動で合わせるには、ディレクトリ再配置・config 移行・manifest 作成が必要 |
| 規約違反の把握が困難 | 既存コードが k1s0 の規約（環境変数禁止、Clean Architecture 依存方向等）にどの程度違反しているか事前に把握できない |
| 段階的移行ができない | 全か無かの選択を迫られる。一部だけ k1s0 管理下に入れることが難しい |
| オンボーディングの障壁 | k1s0 を導入したいチームが既存資産を捨てて新規に書き直す必要がある |

## 要望内容

### `k1s0 migrate` コマンドの新設

```bash
# 分析のみ（変更なし）— 既存プロジェクトの k1s0 準拠度を診断
k1s0 migrate analyze --path ./existing-project

# 移行計画の生成 — 具体的な変更内容を計画ファイルとして出力
k1s0 migrate plan --path ./existing-project --type backend-rust

# 移行の実行 — 計画に基づいて段階的にファイルを移動・生成
k1s0 migrate apply --path ./existing-project --plan migration-plan.json

# 移行状態の確認
k1s0 migrate status --path ./existing-project
```

### `new-feature` との差異

| 観点 | `new-feature` | `migrate` |
|------|---------------|-----------|
| 目的 | ゼロからスキャフォールド生成 | 既存プロジェクトの k1s0 構造への変換 |
| 入力 | テンプレートタイプ + 名前 | 既存のソースコード |
| 前提 | 空のディレクトリ | 既存コードが存在するディレクトリ |
| 出力 | 完全な k1s0 構造 | 移行計画 + 段階的な構造変換 |
| 既存コードの扱い | なし | 保持したまま構造を整理 |
| manifest.json | 自動生成 | 既存コードを分析して生成 |
| lint | 生成後に実行可能 | 移行前に準拠度を診断 |
| リスク | なし（新規生成） | 既存コードの破損リスクあり → バックアップ + dry-run で対処 |

## 影響

- k1s0 の導入障壁が劇的に下がる（既存資産を捨てる必要がない）
- 段階的な移行により、チームのペースで k1s0 規約に準拠できる
- `migrate analyze` だけでも既存コードの品質診断ツールとして有用
- 企業での大規模導入シナリオ（数十〜数百の既存サービス）に対応可能
- k1s0 のコンサルティング・導入支援の起点として機能する

## 回答（実装方針）

### 概要

- CLI に `migrate` サブコマンド（analyze/plan/apply/status）を追加
- `analyze` は既存コードを非破壊的にスキャンし、k1s0 規約への準拠度を数値化
- `plan` は分析結果から具体的な移行ステップを JSON 計画ファイルとして出力
- `apply` は計画ファイルに基づいて段階的に変換を実行（バックアップ付き）
- `status` は移行の進捗と残りの作業を表示
- 既存の `k1s0-generator` の lint エンジンと `TemplateRenderer` を再利用

### サブコマンド構成

| サブコマンド | 破壊的操作 | 説明 |
|-------------|:----------:|------|
| `analyze` | なし | 既存プロジェクトの構造・規約準拠度を診断 |
| `plan` | なし | 移行計画を JSON ファイルとして生成 |
| `apply` | あり（バックアップ付き） | 計画に基づいてディレクトリ構造変換・ファイル生成 |
| `status` | なし | 移行の進捗表示 |

### analyze の分析項目

| 分析カテゴリ | 検出内容 | 使用する既存機能 |
|-------------|---------|-----------------|
| プロジェクト種別推定 | 言語・フレームワーク・ビルドシステム自動検出 | 新規実装 |
| ディレクトリ構造 | Clean Architecture 準拠度、層の分離状況 | K022 ルール拡張 |
| 設定管理 | 環境変数使用箇所、config ファイル有無 | K020, K021 ルール |
| 依存関係 | 外部クレート/パッケージの分析、framework 互換性 | 新規実装 |
| セキュリティ | ハードコード秘密値、SQL インジェクションリスク | K021, K050 ルール |
| API 定義 | proto/OpenAPI ファイルの有無と構造 | buf/Spectral 連携 |
| Docker | Dockerfile/docker-compose.yml の有無と構造 | K060 ルール |
| テスト | テストファイルの配置と構造 | 新規実装 |

### 実装順序

1. **Phase 1: コマンド基盤 + analyze** — MigrateArgs/Action 定義、プロジェクト種別推定、ディレクトリ構造分析（~500 LOC）
2. **Phase 2: lint 統合 + 準拠度スコア** — 既存 lint ルールを非 k1s0 プロジェクトに適用、スコア算出（~400 LOC）
3. **Phase 3: plan 生成** — 分析結果から移行計画 JSON を生成、ステップ順序の最適化（~500 LOC）
4. **Phase 4: apply 実行** — バックアップ作成、ディレクトリ構造変換、manifest 生成、config 移行（~600 LOC）
5. **Phase 5: status + UX 仕上げ** — 進捗表示、dry-run モード、インタラクティブモード、テスト（~400 LOC）

### 技術選定

| 項目 | 選定 | 理由 |
|------|------|------|
| コマンド体系 | clap サブコマンド (`#[command(subcommand)]`) | `docker.rs`, `playground.rs` と同一パターン |
| プロジェクト推定 | ファイルパターンマッチング | `Cargo.toml` → Rust、`go.mod` → Go 等の確定的推定 |
| 構造分析 | AST 解析なし、ファイルパス + grep パターン | 軽量実装優先。AST は将来拡張 |
| lint 統合 | `k1s0-generator::lint` の再利用 | 既存の K020-K060 ルールをそのまま適用 |
| 移行計画 | JSON ファイル (`migration-plan.json`) | 再現性 + 編集可能 + CI 統合 |
| バックアップ | `.k1s0/backup/{timestamp}/` にコピー | apply 前に自動実行、rollback 可能 |
| ディレクトリ移動 | `fs::rename` + シンボリックリンク互換チェック | クロスプラットフォーム対応 |
| プロンプト | inquire クレート | 既存の `prompts/` モジュール準拠 |

## 期待される効果

1. 既存プロジェクトを持つチームが k1s0 に段階的に移行できる
2. `analyze` だけでも規約違反の把握ツールとして単独で有用
3. 移行計画の JSON 出力により、レビュー・承認フローに組み込める
4. バックアップ + dry-run により、安全な移行を保証
5. 企業レベルの大規模導入シナリオに対応（既存サービス群の一括分析）
6. k1s0 の「既存コードは捨てなくていい」というメッセージが明確になる
7. `analyze` → `plan` → `apply` の段階的ワークフローにより、移行の意思決定を支援
