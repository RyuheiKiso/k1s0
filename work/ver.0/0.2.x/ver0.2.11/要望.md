# 要望

現行の k1s0 プラットフォームにはマイクロサービス間のコンセンサスプロトコルが体系的に整備されておらず、分散環境での排他制御やワークフロー管理が各チームの独自実装に依存している。統一的なコンセンサスプロトコル機構を追加したい。

## 背景

| 課題 | 説明 |
|------|------|
| リーダー選出が未実装 | 複数インスタンスでのシングルトンタスク（バッチ処理、スケジューラ等）をどのノードが担当するか決定する仕組みがない。各チームが独自に DB ロック等で代替しており、フェイルオーバーの信頼性にばらつきがある |
| 分散ロックが未標準化 | 排他制御が必要な処理（在庫引当、決済処理等）で統一された分散ロック API がない。DB の `SELECT FOR UPDATE` や Redis の `SET NX` を直接使う実装が散在し、デッドロックや TTL 管理の不備によるロック漏れが発生している |
| Saga パターンが未サポート | 複数サービスにまたがるトランザクション（注文→在庫→決済→配送）を管理する仕組みがない。障害時の補償トランザクションが手動実装され、中途半端な状態のまま放置されるケースがある |
| 分散ワークフローの可観測性がない | リーダー選出の状態変更、ロック取得/解放、Saga の進行状況を Prometheus/Grafana で監視する仕組みがなく、問題発生時の原因特定が困難 |
| 言語間で実装品質にばらつき | Rust サービスと Go サービスでロック取得のリトライ戦略が異なる等、言語をまたぐと動作が予測しにくい |

## 現行実装の問題例

### リーダー選出 — 独自 DB ロック

```rust
// 各チームが独自に実装している典型例
// 問題: ハートビート失敗時のフェイルオーバーが不安定
async fn try_become_leader(pool: &PgPool, node_id: &str) -> Result<bool, Error> {
    let result = sqlx::query!(
        "INSERT INTO leader_lock (key, holder, expires_at)
         VALUES ('scheduler', $1, NOW() + INTERVAL '30 seconds')
         ON CONFLICT (key) DO UPDATE SET holder = $1, expires_at = NOW() + INTERVAL '30 seconds'
         WHERE leader_lock.expires_at < NOW()",
        node_id
    )
    .execute(pool)
    .await?;
    Ok(result.rows_affected() > 0)
}
// ハートビート、watch 通知、フェイルオーバー検出はすべて自前で実装が必要
```

### 分散ロック — フェンシングトークンなし

```go
// 問題: フェンシングトークンがなく、GC 停止等でリースが期限切れした後に
// 古いクライアントがロック保護された操作を続行するリスクがある
func acquireLock(ctx context.Context, rdb *redis.Client, key string) (bool, error) {
    ok, err := rdb.SetNX(ctx, "lock:"+key, "holder", 30*time.Second).Result()
    return ok, err
}
// TTL 延長、フェンシングトークン検証、デッドロック検出はすべて未実装
```

### Saga — 手動補償

```python
# 問題: 途中で障害が発生すると補償が不完全になる
async def process_order(order_id: str):
    await inventory_service.reserve(order_id)     # ステップ 1
    try:
        await payment_service.charge(order_id)    # ステップ 2
    except PaymentError:
        await inventory_service.release(order_id) # 手動補償
        raise
    try:
        await shipping_service.dispatch(order_id) # ステップ 3
    except ShippingError:
        await payment_service.refund(order_id)    # 手動補償
        await inventory_service.release(order_id) # 手動補償
        raise
# 補償漏れ、順序間違い、冪等性の保証はすべてチーム依存
```

## 要望内容

### 1. リーダー選出（Leader Election）

DB ベースのリース方式によるリーダー選出を実装する。ハートビートによる死活監視、自動フェイルオーバー、リーダー変更の watch 通知を含む。全バックエンド言語で統一 API を提供する。

### 2. 分散ロック（Distributed Lock）

DB ベースと Redis ベースの2種類の分散ロックを実装する。TTL による自動解放、フェンシングトークンによる安全なロック検証、デッドロック検出を含む。

### 3. Saga オーケストレーション

ステップ定義→正方向実行→失敗時補償（逆順）のオーケストレーション型 Saga を実装する。状態の永続化、リトライ、デッドレターキューを含む。

### 4. Saga コレオグラフィ

既存の `k1s0-domain-event` と連携したイベント駆動型 Saga を実装する。イベント発行→購読→次ステップ実行の非同期ワークフローをサポートする。

## 期待される改善

| 項目 | 現行 | コンセンサスプロトコル導入後 |
|------|------|--------------------------|
| シングルトンタスク管理 | 各チーム独自の DB ロック実装 | 統一 API でリーダー選出、自動フェイルオーバー |
| 排他制御 | `SET NX` / `SELECT FOR UPDATE` の直接利用 | フェンシングトークン付き分散ロック、TTL 自動管理 |
| 分散トランザクション | 手動補償、補償漏れリスク | Saga オーケストレーターが自動補償（逆順実行） |
| ワークフロー可観測性 | 不可能 | Prometheus + Grafana ダッシュボードで状態遷移を可視化 |
| 言語間一貫性 | ばらつきあり | 全言語で同一の抽象化と動作保証 |
| 障害時リカバリ | 手動介入が必要 | デッドレターキュー + 自動リトライで回復 |
| デバッグ容易性 | Saga の進行状況が不透明 | ステップごとの状態永続化とトレーシング |

## 影響

- `k1s0-consensus` は新規パッケージとして全バックエンド言語に追加（Tier 2: `k1s0-error`, `k1s0-config`, `k1s0-db`, `k1s0-domain-event` に依存）
- DB マイグレーション追加（`fw_m_leader_lease`, `fw_m_distributed_lock`, `fw_m_saga_instance`, `fw_m_saga_step` テーブル）
- 既存の `k1s0-domain-event` との統合（Saga コレオグラフィでイベントバスを利用）
- 既存の `k1s0-db` との統合（DB ベースリーダー選出・分散ロックでコネクションプールを利用）
- 既存の `k1s0-resilience` との棲み分え: `k1s0-resilience` は「障害への耐性」（リトライ、サーキットブレーカー）、`k1s0-consensus` は「分散合意」（リーダー選出、排他制御、分散トランザクション）
- YAML 設定スキーマの拡張（`consensus` セクション追加）
- Prometheus メトリクス追加に伴い Grafana ダッシュボードの更新が必要
- CI/CD ワークフローへの影響なし（既存の言語別ワークフローでカバー）
