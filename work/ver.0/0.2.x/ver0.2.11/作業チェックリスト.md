## 備考

### k1s0-resilience との棲み分け

| パッケージ | 責務 | 例 |
|-----------|------|-----|
| `k1s0-resilience` | 障害への耐性 | サーキットブレーカー、リトライ、タイムアウト、セマフォ、バルクヘッド |
| `k1s0-rate-limit` | 負荷の制御 | トークンバケット、スライディングウィンドウ |
| `k1s0-consensus` | 分散合意 | リーダー選出、分散ロック、Saga（オーケストレーション/コレオグラフィ） |

### Tier 2 依存ルールの確認

`k1s0-consensus` は Tier 2 に配置し、以下の依存を持つ:
- Tier 1: `k1s0-error`, `k1s0-config`（許可）
- Tier 2: `k1s0-db`, `k1s0-domain-event`, `k1s0-observability`（Tier 2 同士の依存は許可）
- 外部: `prometheus` 系、`redis`、`uuid`、`chrono`

### 環境変数禁止の遵守

コンセンサス設定（リース期間、ロック TTL、Saga リトライポリシー等）は全て `config/*.yaml` から読み込む。Redis 接続情報もシークレットファイル参照を使用する。

### テスト方針

- リーダー選出のフェイルオーバーテスト: `tokio::time::pause()` (Rust) / `fakeclock` (Go) 等のモック時計を使用
- DB テスト: testcontainers で PostgreSQL コンテナを起動
- Redis テスト: testcontainers で Redis コンテナを起動（または `mini-redis` モック）
- Saga テスト: インメモリ実装でステップ実行/補償の動作を検証
- 冪等性テスト: 同一 Saga ステップを複数回実行しても副作用が1回のみであることを検証

### 冪等性ガイドライン

Saga の各ステップは冪等に設計する必要がある:
- `execute`: 一意キー（saga_id + step_index）で重複実行を検出
- `compensate`: 補償済みかどうかを DB で確認してからロールバック実行
- ドキュメントにパターン例を記載する
