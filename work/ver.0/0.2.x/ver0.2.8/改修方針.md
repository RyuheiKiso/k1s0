# k1s0 v0.2.8 改修方針

## 概要

k1s0 リポジトリに `observability/` ディレクトリを新設し、OTEL Collector + Jaeger + Loki + Prometheus + Grafana による可観測性基盤を構築する。

### 前提条件: Docker 必須

可観測性スタックの起動には **Docker Desktop（または Docker Engine）** が必須。Docker がインストールされていない環境では、CLI が検出時に以下のように通知する。

#### `k1s0 doctor` での検出

```
$ k1s0 doctor

✓ Rust 1.85.0
✓ Node 20.x
✓ pnpm 9.15.4
✗ Docker ... Not found
  └─ 可観測性スタック（Grafana, Jaeger 等）を利用するには Docker のインストールが必要です。
     サービスの開発自体は Docker なしで可能です。
     インストール: https://docs.docker.com/desktop/install/windows-install/
```

#### `k1s0 docker compose up` 実行時

```
$ k1s0 docker compose up

Error: Docker が検出されませんでした。

可観測性スタック（OTEL Collector, Jaeger, Loki, Prometheus, Grafana）の
起動には Docker Desktop が必要です。

サービスの開発・ビルド・テストは Docker なしで実行可能です。
可観測性が不要な場合、このエラーは無視できます。

Docker Desktop のインストール:
  https://docs.docker.com/desktop/install/windows-install/
```

#### 設計方針

| 項目 | 方針 |
|------|------|
| Docker 未検出時の動作 | エラーではなく**警告**として通知（`k1s0 doctor` では `warning`） |
| サービス開発への影響 | なし。ビルド・テスト・lint は Docker 不要で動作 |
| 通知タイミング | `k1s0 doctor`, `k1s0 docker compose up`, `k1s0 observability *`（将来） |
| メッセージの多言語対応 | k1s0 の既存 i18n 方針に準拠 |

## コンポーネント詳細

### 1. OTEL Collector

テレメトリデータの受信・加工・転送を行う中央ハブ。

#### パイプライン構成

```
receivers:
  otlp:                           # gRPC (4317) + HTTP (4318)
    protocols:
      grpc:
        endpoint: 0.0.0.0:4317
      http:
        endpoint: 0.0.0.0:4318

processors:
  batch:                          # バッチ化（パフォーマンス向上）
    timeout: 5s
    send_batch_size: 1024
  memory_limiter:                 # メモリ制限
    check_interval: 1s
    limit_mib: 512
  resource:                       # リソース属性付与
    attributes:
      - key: deployment.environment
        action: upsert
        value: "${ENV}"

exporters:
  otlp/jaeger:                    # Traces → Jaeger
    endpoint: jaeger:4317
    tls:
      insecure: true
  prometheusremotewrite:          # Metrics → Prometheus
    endpoint: http://prometheus:9090/api/v1/write
  loki:                           # Logs → Loki
    endpoint: http://loki:3100/loki/api/v1/push

service:
  pipelines:
    traces:
      receivers: [otlp]
      processors: [memory_limiter, batch]
      exporters: [otlp/jaeger]
    metrics:
      receivers: [otlp]
      processors: [memory_limiter, batch]
      exporters: [prometheusremotewrite]
    logs:
      receivers: [otlp]
      processors: [memory_limiter, batch]
      exporters: [loki]
```

#### 環境別差分

| 設定項目 | dev | stg | prod |
|---------|-----|-----|------|
| サンプリング率 | 100%（全件） | 50% | 10-20% |
| バッチサイズ | 256 | 512 | 1024 |
| バッチタイムアウト | 1s | 3s | 5s |
| メモリ制限 | 256 MiB | 512 MiB | 1024 MiB |
| デバッグログ | 有効 | 無効 | 無効 |

### 2. Jaeger

分散トレーシングバックエンド。

| 設定項目 | dev | prod |
|---------|-----|------|
| ストレージ | in-memory / Badger | Elasticsearch |
| 保持期間 | 24h | 7-30日 |
| UI ポート | 16686 | 16686 |
| OTLP 受信 | 4317 (gRPC) | 4317 (gRPC) |

### 3. Loki

ログ集約バックエンド。

| 設定項目 | 値 |
|---------|---|
| ストレージ | ローカル filesystem（dev）/ S3（prod） |
| 保持期間 | 30-90日 |
| インデックス | BoltDB Shipper |
| 受信 | HTTP 3100 |
| ラベル設定 | `service_name`, `deployment_environment`, `level` |

### 4. Prometheus

メトリクス保存・アラート。

#### スクレイプ対象

| ターゲット | エンドポイント | 間隔 |
|-----------|-------------|------|
| OTEL Collector | otel-collector:8888/metrics | 15s |
| 各サービス（k1s0-health） | {service}:{port}/metrics | 15s |

#### レコーディングルール

| ルール名 | 目的 |
|---------|------|
| `k1s0:request_duration:p99` | リクエストレイテンシ P99 |
| `k1s0:error_rate:5m` | 5分間エラー率 |
| `k1s0:request_rate:5m` | 5分間リクエストレート |

#### アラートルール

| アラート名 | 条件 | 重大度 |
|-----------|------|:------:|
| HighErrorRate | エラー率 > 5%（5分間） | critical |
| HighLatency | P99 > 2s（5分間） | warning |
| ServiceDown | ヘルスチェック失敗（1分間） | critical |
| HighMemoryUsage | メモリ使用率 > 85% | warning |

### 5. Grafana

統合可視化ダッシュボード。

#### データソース

| データソース | 用途 | URL |
|-------------|------|-----|
| Jaeger | トレース検索・表示 | http://jaeger:16686 |
| Prometheus | メトリクスクエリ | http://prometheus:9090 |
| Loki | ログ検索 | http://loki:3100 |

#### ダッシュボード定義

| ダッシュボード | パネル内容 |
|---------------|----------|
| overview.json | サービス数、全体エラー率、全体レイテンシ P50/P99、アクティブトレース数 |
| service-health.json | サービス別 UP/DOWN、ヘルスチェック結果、依存サービス状態 |
| grpc-metrics.json | gRPC メソッド別レイテンシ、ステータスコード分布、ストリーム数 |
| http-metrics.json | HTTP パス別レイテンシ、ステータスコード分布、リクエスト数 |
| db-metrics.json | 接続プール使用率、クエリ実行時間、スロークエリ数 |

#### Grafana 設定

| 設定 | 値 |
|------|---|
| 管理者パスワード | `admin`（dev）/ Secrets Manager 参照（prod） |
| 匿名アクセス | 有効（dev）/ 無効（prod） |
| プロビジョニング | dashboards / datasources / alerting を自動読込 |

## docker-compose.yml 構成

```yaml
services:
  otel-collector:
    build: ./otel/collector
    ports:
      - "4317:4317"   # OTLP gRPC
      - "4318:4318"   # OTLP HTTP
      - "8888:8888"   # Collector metrics
    depends_on:
      - jaeger
      - loki
      - prometheus

  jaeger:
    image: jaegertracing/jaeger:1.x  # ピン留めバージョン
    ports:
      - "16686:16686" # Jaeger UI
      - "4317"        # OTLP gRPC (内部)

  prometheus:
    build: ./prometheus
    ports:
      - "9090:9090"   # Prometheus UI + API
    volumes:
      - prometheus-data:/prometheus

  loki:
    build: ./loki
    ports:
      - "3100:3100"   # Loki API

  grafana:
    build: ./grafana
    ports:
      - "3000:3000"   # Grafana UI
    depends_on:
      - jaeger
      - prometheus
      - loki
    volumes:
      - grafana-data:/var/lib/grafana

volumes:
  prometheus-data:
  grafana-data:
```

### ポートマッピング一覧

| ポート | サービス | 用途 |
|-------|---------|------|
| 3000 | Grafana | ダッシュボード UI |
| 4317 | OTEL Collector | OTLP gRPC 受信（各サービスから） |
| 4318 | OTEL Collector | OTLP HTTP 受信 |
| 8888 | OTEL Collector | Collector 自身のメトリクス |
| 9090 | Prometheus | Prometheus UI + API |
| 3100 | Loki | Loki API |
| 16686 | Jaeger | Jaeger UI |

## k1s0 CLI 連携

### `k1s0 docker compose` との統合

可観測性スタックは各 feature サービスの `docker-compose.yml` とは独立して管理する。

```bash
# 可観測性スタック起動
cd observability && docker compose up -d

# feature サービスは OTEL_EXPORTER_OTLP_ENDPOINT を設定して起動
# ※ feature 側の docker-compose.yml で network を共有
```

### 将来的な `k1s0 observability` コマンド

```bash
k1s0 observability up       # 可観測性スタック起動
k1s0 observability down     # 停止
k1s0 observability status   # 各コンポーネントの状態確認
k1s0 observability open     # Grafana をブラウザで開く
```

※ v0.2.8 ではディレクトリ構造と設定ファイルの整備を優先。CLI コマンドは将来バージョンで検討。

## 環境別設定方針

k1s0 の `default/dev/stg/prod` 規約に準拠。

| 環境 | 方針 |
|------|------|
| default | 全環境共通のパイプライン定義・エクスポーター設定 |
| dev | 全量サンプリング、in-memory ストレージ、デバッグログ有効、匿名アクセス許可 |
| stg | 中間設定、本番に近い構成で検証 |
| prod | 低サンプリング率、永続ストレージ、認証必須、アラート有効 |

1. **Phase 1: ディレクトリ構造作成** — 全ディレクトリとプレースホルダファイルの作成
2. **Phase 2: OTEL Collector 設定** — パイプライン定義（default + dev）

9. **Phase 9: deploy/** — Kubernetes / Helm 定義
10. **Phase 10: ドキュメント** — README、運用ガイド

## 注意事項

- **k1s0 設定規約への準拠**: OTEL Collector の設定は `default/dev/stg/prod` の4環境パターンに従う
- **シークレット管理**: Grafana のパスワード等は `_file` サフィックスパターンで外部参照（K021 準拠）
- **Dockerfile ピン留め**: 全 Dockerfile で `:latest` を使用せずバージョンまたは SHA をピン留め（K060 準拠）
- **ネットワーク分離**: 可観測性スタックは独立した Docker ネットワークで動作し、feature サービスから接続
- **リソース制限**: 各コンテナに `mem_limit` / `cpus` を設定し、ローカル環境での過負荷を防止
