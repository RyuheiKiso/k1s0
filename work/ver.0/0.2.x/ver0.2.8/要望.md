# 要望

k1s0 プロジェクトに可観測性（Observability）基盤を構築し、マイクロサービスのトレース・ログ・メトリクスを一元的に収集・可視化できるようにしたい。

## 背景

k1s0 は各言語の framework パッケージとして `k1s0-observability` を提供しているが、テレメトリデータの収集・保存・可視化を行うインフラ基盤が未整備である。

| 課題 | 説明 |
|------|------|
| テレメトリ収集基盤がない | 各サービスが OTEL SDK で計装しても、データの送信先（Collector）が存在しない |
| 分散トレーシング不可 | サービス間のリクエストフローを追跡できず、レイテンシのボトルネック特定が困難 |
| ログ集約なし | 各サービスのログが分散しており、横断的なログ検索ができない |
| メトリクス監視なし | CPU・メモリ・リクエスト数・エラー率等の時系列データを蓄積・アラートできない |
| ダッシュボード未整備 | システム全体の健全性を一目で把握できるUIがない |
| ローカル開発環境での観測不可 | docker compose で可観測性スタックを起動する手段がない |

## 要望内容

### 可観測性スタックの構築

OTELの3本柱（Traces / Logs / Metrics）を以下のコンポーネントで実現する：

```
OTEL Collector（収集・転送）
  ├── Traces  → Jaeger（分散トレーシング）
  ├── Logs    → Loki（ログ集約）
  └── Metrics → Prometheus（メトリクス）
                    └── 全て Grafana で統合可視化
```

### コンポーネント選定理由

| コンポーネント | 役割 | 選定理由 |
|---------------|------|---------|
| OTEL Collector | テレメトリ収集・加工・転送 | ベンダー非依存の標準プロトコル、パイプライン設計による柔軟な処理 |
| Jaeger | 分散トレーシング | CNCF graduated、単体UIが優秀、OTLP ネイティブ対応 |
| Loki | ログ集約 | ログのインデックスではなくラベルベースで軽量、Grafana との深い統合 |
| Prometheus | メトリクス保存・アラート | CNCF graduated、事実上の標準、PromQL の強力なクエリ言語 |
| Grafana | 統合ダッシュボード | 全データソースを統合可視化、プロビジョニングによるコード管理 |

### 各コンポーネントの役割整理

| | Jaeger | Loki | Prometheus |
|---|--------|------|------------|
| 対象データ | トレース（スパン） | ログ（テキスト行） | メトリクス（数値時系列） |
| 答える問い | リクエストはどのサービスを通り、どこで遅延した？ | エラー時にサービスは何を出力した？ | エラー率・レイテンシ・リソース使用量は？ |
| データ形式 | 親子関係を持つスパンツリー | タイムスタンプ付きラベルドテキスト | タイムスタンプ付きラベルド数値 |
| 保存期間 | 短〜中期（7-30日） | 中期（30-90日） | 中〜長期（90-365日） |

## 影響

- ローカル開発環境で `docker compose up` により可観測性スタック全体を起動可能
- 各言語の `k1s0-observability` パッケージが送信するテレメトリを即座に確認可能
- Grafana ダッシュボードでサービス横断的な健全性を可視化
- Kubernetes デプロイ時にも同一構成を適用可能（Helm / Kustomize）
- 将来的にアラート設定やSLO監視にも拡張可能
