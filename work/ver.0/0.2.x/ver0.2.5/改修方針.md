# k1s0 v0.2.5 改修方針

## 概要

`k1s0 playground` コマンドを新設し、以下の機能を実装する：

- `playground start` — テンプレートからサンプルコード付き環境を一時生成し、Docker Compose またはローカルプロセスで起動
- `playground stop` — playground 環境を停止しファイルを削除
- `playground status` — 稼働中の playground 一覧を表示
- `playground list` — 利用可能なテンプレート一覧を表示

## 機能目的

1. k1s0 を初めて触る開発者のオンボーディング体験を改善する
2. `k1s0 init` 不要で、どこからでも実行可能にする
3. 空のスケルトンではなく、動作する CRUD エンドポイントを提供する
4. Docker がなくてもローカルモードで体験可能にする
5. `playground stop` で環境とファイルを自動クリーンアップする

---

## 実行モード

playground は2つの実行モードを提供する。Docker の有無に応じて自動選択される。

### モード比較

| 項目 | Docker モード | ローカルモード |
|------|:------------:|:------------:|
| コマンド | `--mode docker` (デフォルト) | `--mode local` |
| 起動方法 | docker compose up | cargo run / go run 等 |
| DB | PostgreSQL コンテナ | SQLite ファイル |
| キャッシュ | Redis コンテナ | インメモリ HashMap |
| 前提ツール | Docker + Docker Compose | 各言語ツールチェーン |
| ネットワーク | Docker ネットワーク | localhost |
| クリーンアップ | docker compose down + ファイル削除 | プロセス kill + ファイル削除 |

### 自動モード検出

```rust
fn detect_mode(explicit_mode: Option<&str>) -> PlaygroundMode {
    match explicit_mode {
        Some("docker") => PlaygroundMode::Docker,
        Some("local") => PlaygroundMode::Local,
        None => {
            if check_docker().is_ok() && check_docker_compose().is_ok() {
                PlaygroundMode::Docker
            } else {
                output().warning(
                    "Docker が検出されませんでした。ローカルモードで起動します。\n\
                     Docker モードを使用するには Docker Desktop をインストールしてください。"
                );
                PlaygroundMode::Local
            }
        }
    }
}
```

```rust
    #[arg(long)]
    pub with_cache: bool,

    /// ポートオフセット（複数 playground 並行起動用、0-999）
    #[arg(long, default_value = "0", value_parser = clap::value_parser!(u16).range(0..1000))]
```

#### 1.2 処理フロー（Docker モード）

```

  │

  │     └─ 未インストール → ローカルモードへフォールバック

  ├─ 2. 引数解決
  │     ├─ --type 未指定 → インタラクティブ選択（inquire::Select）

  │
  ├─ 3. ポート競合チェック
  │     └─ TcpListener::bind() で使用状況を確認
  │     └─ 競合時: エラー + --port-offset の案内
  │
  ├─ 4. playground ディレクトリ決定
  │     ├─ .k1s0/ 存在 → .k1s0/playground/{name}/
  │     └─ 未存在 → $HOME/.k1s0/playground/{name}/
  │
  ├─ 5. テンプレートレンダリング（3段階マージ）
  │     ├─ Step 1: ベーステンプレート展開
  │     │   └─ TemplateRenderer::new(CLI/templates/{type}/feature/)
  │     │       .render_directory(output_dir, context)
  │     ├─ Step 2: playground オーバーレイ上書き
  │     │   └─ TemplateRenderer::new(CLI/templates/playground/{type}/)
  │     │       .render_directory(output_dir, context)  // 同名ファイル上書き
  │     └─ Step 3: 共通オーバーレイ追加
  │         └─ TemplateRenderer::new(CLI/templates/playground/common/)
  │             .render_directory(output_dir, context)
  │
  ├─ 6. シークレットファイル生成（--with-db 時）
  │     └─ secrets/db_password.txt に "playground" を書き込み
  │
  ├─ 7. メタデータ保存 (.playground.json)
  │
  ├─ 8. docker compose up -d --build
  │
  ├─ 9. ヘルスチェック待機（最大 120 秒、2 秒間隔）
```

### 2. `playground stop`

#### 2.1 引数定義

```rust
#[derive(Args, Debug)]
pub struct StopArgs {
    /// 停止する playground 名（省略時は全て停止）
    #[arg(short = 'n', long)]
    pub name: Option<String>,

    /// ボリュームも削除（Docker モード時）
    #[arg(short = 'v', long)]
    pub volumes: bool,

    /// 確認プロンプトをスキップ
    #[arg(short = 'y', long)]
    pub yes: bool,
}
```

#### 2.2 処理フロー

```
stop 実行
  │
  ├─ 1. playground ディレクトリ走査
  │     ├─ .k1s0/playground/ と $HOME/.k1s0/playground/ の両方を走査
  │     └─ .playground.json から PlaygroundMetadata を読み込み
  │
  ├─ 2. --name 指定時はフィルタ、未指定時は全て対象
  │
  ├─ 3. 確認プロンプト（--yes 未指定時）
  │
  ├─ 4. モードに応じた停止処理
  │     ├─ Docker モード:
  │     │   └─ docker compose -f {path}/docker-compose.yml down [-v]
  │     └─ ローカルモード:
  │         ├─ SIGTERM (Unix) / taskkill (Windows) でプロセス終了
  │         ├─ 3秒待機
  │         └─ プロセスが残っていれば SIGKILL / taskkill /F
  │
  ├─ 5. playground ディレクトリ再帰削除
  │
  └─ 6. 完了メッセージ
```

## テンプレート設計

### オーバーレイマージ方式

ベーステンプレートに playground オーバーレイを上書きマージする。`TemplateRenderer::render_directory()` を複数回呼び出すことで実現する（既存の `WriteResult::Overwritten` を活用）。

```
レンダリング順序:

1. ベーステンプレート    CLI/templates/{type}/feature/         → output/
2. playground オーバーレイ CLI/templates/playground/{type}/      → output/ (上書き)
3. ローカルモード差分     CLI/templates/playground/{type}-local/ → output/ (上書き、ローカル時のみ)
4. 共通オーバーレイ       CLI/templates/playground/common/       → output/ (追加)
```

### DB ドライバ抽象化（サンプルコードの設計）

同一のサンプルコードが PostgreSQL（Docker モード）と SQLite（ローカルモード）の両方で動作する。infrastructure 層でドライバを抽象化する。

#### config/default.yaml（Docker モード）

```yaml

  host: "0.0.0.0"
  port: 8080

database:
  driver: postgres
  host: db
  port: 5432
  name: playground
  user: postgres
  password_file: /var/run/secrets/db_password

cache:
  driver: redis
  host: redis
  port: 6379
```

#### config/default.yaml（ローカルモード — オーバーレイで差し替え）

```yaml
server:
  host: "0.0.0.0"
  port: {{ rest_port }}

  driver: sqlite
  path: ./data/playground.db

  driver: memory
```

#### Rust サンプルコード: リポジトリ実装

```rust
// src/infrastructure/repositories/item_repository.rs
// config の driver フィールドに応じて接続先を切り替え

use crate::domain::entities::item::{Item, CreateItem, UpdateItem};

pub struct ItemRepository {
    pool: DatabasePool,
}

enum DatabasePool {
    Postgres(sqlx::PgPool),
    Sqlite(sqlx::SqlitePool),
}

impl ItemRepository {
    pub async fn new(config: &DatabaseConfig) -> Result<Self> {
        let pool = match config.driver.as_str() {
            "postgres" => {
                let password = std::fs::read_to_string(&config.password_file)?;
                let url = format!(
                    "postgres://{}:{}@{}:{}/{}",
                    config.user, password.trim(),
                    config.host, config.port, config.name
                );
                DatabasePool::Postgres(sqlx::PgPool::connect(&url).await?)
            }
            "sqlite" => {
                std::fs::create_dir_all("./data")?;
                let pool = sqlx::SqlitePool::connect(&format!("sqlite:{}", config.path)).await?;
                sqlx::migrate!("./migrations").run(&pool).await?;
                DatabasePool::Sqlite(pool)
            }
            other => return Err(anyhow!("Unknown database driver: {}", other)),
        };
        Ok(Self { pool })
    }

    pub async fn list(&self) -> Result<Vec<Item>> {
        match &self.pool {
            DatabasePool::Postgres(pool) => {
                Ok(sqlx::query_as!(Item, "SELECT * FROM items ORDER BY id")
                    .fetch_all(pool).await?)
            }
            DatabasePool::Sqlite(pool) => {
                Ok(sqlx::query_as!(Item, "SELECT * FROM items ORDER BY id")
                    .fetch_all(pool).await?)
            }
        }
    }
    // create, get, update, delete も同様のパターン
}
```

この設計は k1s0 の「設定駆動」思想のデモとして教育的価値が高い。

## PlaygroundMetadata

`.playground.json` に保存するメタデータ。Docker モードとローカルモードの両方に対応。

```rust
#[derive(Debug, Serialize, Deserialize)]
pub struct PlaygroundMetadata {
    pub name: String,
    pub template_type: String,
    pub mode: String,              // "docker" | "local"
    pub options: PlaygroundOptions,
    pub port_offset: u16,
    pub ports: PortConfig,
    pub pid: Option<u32>,          // ローカルモード時のみ
    pub created_at: String,        // ISO 8601
    pub dir: PathBuf,
}

#[derive(Debug, Serialize, Deserialize)]
pub struct PlaygroundOptions {
    pub with_grpc: bool,
    pub with_rest: bool,
    pub with_db: bool,
    pub with_cache: bool,
}

#[derive(Debug, Serialize, Deserialize)]
pub struct PortConfig {
    pub rest_port: u16,
    pub grpc_port: u16,
    pub db_port: u16,
```

## テスト方針

### テスト3層構成

| テスト層 | Docker 必要 | CI 実行 | 内容 |
|---------|:-----------:|:------:|------|
| ユニットテスト | 不要 | 常時 | ユーティリティ関数、メタデータ、ポート計算、モード検出 |
| テンプレートテスト | 不要 | 常時 | 各テンプレートのレンダリング結果検証（Docker/ローカル両方） |
| 統合テスト（ローカル） | 不要 | 常時 | テンプレート生成 → ビルド確認のドライラン |
| 統合テスト（Docker） | 必要 | オプション | Docker ライフサイクル全体（フィーチャーゲート） |

- playground.rs: 80% 以上
- テンプレートレンダリング: 各テンプレート × 各モードで 1 ケース以上

## 注意事項

- playground テンプレートのサンプルコードでも k1s0 の規約を遵守する（環境変数禁止、Clean Architecture 依存方向）
- `config/default.yaml` を使用し、`std::env::var` / `os.Getenv` / `process.env` 等は使用しない
- ローカルモードの DB ドライバ抽象化は教育的なデモとして機能する（設定駆動の実例）
- Docker 未インストール時はローカルモードへ自動フォールバックし、明確なメッセージを表示する
- ローカルモードでも言語ツールチェーンが未インストールの場合は明確なエラーメッセージ + インストール案内を表示する
- ポートオフセットは 0-999 の範囲に制限し、ポート衝突を最小化する
- Windows / macOS / Linux の全プラットフォームでプロセス管理（PID 追跡、停止）を正しく動作させる
