# 要望

k1s0 を初めて触る開発者が、1コマンドで動作するサービス環境を体験できる仕組みがほしい。

## 背景

現在 k1s0 で新しいサービスを立ち上げるには、以下のステップが必要：

1. `k1s0 init` でリポジトリを初期化
2. `k1s0 new-feature --type <type> --name <name>` でスキャフォールド生成
3. Docker/Docker Compose でビルド・起動
4. 生成されたコードを理解し、ビジネスロジックを実装

この手順は本番運用には適切だが、以下の課題がある：

| 課題 | 説明 |
|------|------|
| 初回体験のハードル | init 済みリポジトリが前提。気軽に試せない |
| 空のスケルトン | `new-feature` は空の Clean Architecture 構造を生成するため、動作確認できない |
| 手動操作の多さ | scaffold 生成→Docker 起動→ヘルスチェック確認を個別に実行する必要がある |
| クリーンアップ | 試した後のファイル削除が手動。`feature/` や `manifest.json` に登録されるため消しにくい |
| 複数スタック比較 | Rust/Go/C#/Python を比較検証するには、それぞれ feature を作成する必要がある |

## 要望内容

### `k1s0 playground` コマンドの新設

```bash
k1s0 playground start                          # インタラクティブに選択して起動
k1s0 playground start --type backend-rust      # テンプレート指定で起動
k1s0 playground start --type backend-rust --with-grpc --with-db
k1s0 playground stop                           # 環境を停止・クリーンアップ
k1s0 playground status                         # 稼働中の playground 一覧
k1s0 playground list                           # 利用可能なテンプレート一覧
```

### `new-feature` との差異

| 観点 | `new-feature` + `docker compose up` | `playground` |
|------|--------------------------------------|-------------|
| 目的 | 本番用スキャフォールド生成 | 体験・学習・検証 |
| 出力先 | `feature/{type}/{name}/` (永続) | `.k1s0/playground/{name}/` (一時的) |
| manifest 登録 | あり | なし |
| lint 対象 | 対象 | 対象外 |
| init 前提 | 必要 | **不要（どこでも実行可能）** |
| サンプルコード | 空の Clean Architecture 構造 | **動作する CRUD エンドポイント付き** |
| クリーンアップ | 手動 | `playground stop` で自動削除 |

## 影響

- k1s0 のオンボーディング体験が劇的に改善される
- 開発者が各テンプレートの出力を即座に確認・比較できる
- framework パッケージの動作デモとしても機能する
- `playground export` で正式な feature に昇格できれば、試行→採用のワークフローが完成する

## 回答（実装方針）

### 概要

- CLI に `playground` サブコマンド（start/stop/status/list）を追加
- Docker モード（docker compose）とローカルモード（直接プロセス起動）の2モード対応
- Docker 未インストール時はローカルモードに自動フォールバック
- 既存の `TemplateRenderer` + `docker.rs` パターンを再利用
- playground 専用テンプレート（サンプルハンドラ、シード SQL、README）+ ローカルモード差分を追加
- `.k1s0/playground/` ディレクトリに一時展開し、manifest には登録しない

### 実行モード

| 項目 | Docker モード | ローカルモード |
|------|:------------:|:------------:|
| コマンド | `--mode docker` (デフォルト) | `--mode local` |
| 起動方法 | docker compose up | cargo run / go run 等 |
| DB | PostgreSQL コンテナ | SQLite ファイル |
| キャッシュ | Redis コンテナ | インメモリ HashMap |
| 前提ツール | Docker + Docker Compose | 各言語ツールチェーン |
| 自動選択 | Docker 検出時 | Docker 未検出時にフォールバック |

### 実装順序

1. **Phase 1: コマンド基盤 + モード検出** — PlaygroundArgs/Action 定義、自動モード検出、ツールチェーン確認（~400 LOC）
2. **Phase 2: playground テンプレート** — backend-rust 向けサンプルコード（DB ドライバ抽象化含む）+ ローカルモード差分（~500 LOC）
3. **Phase 3: Docker + ローカル統合** — Docker Compose / ローカルプロセス起動、ヘルスチェック、プロセス管理（~400 LOC）
4. **Phase 4: 他テンプレート対応** — backend-go, backend-python, backend-csharp, frontend-react, frontend-flutter（~600 LOC）
5. **Phase 5: UX 仕上げ** — インタラクティブモード、エラーハンドリング、テスト3層（~300 LOC）

### 技術選定

| 項目 | 選定 | 理由 |
|------|------|------|
| コマンド体系 | clap サブコマンド (`#[command(subcommand)]`) | `docker.rs` と同一パターン |
| プロンプト | inquire クレート | 既存の `prompts/` モジュール準拠 |
| テンプレート | Tera (`.tera` ファイル) | `k1s0-generator` 既存基盤 |
| Docker 操作 | `std::process::Command` | `docker.rs` 既存パターン |
| ローカルプロセス | `std::process::Command::spawn()` | PID 取得 + バックグラウンド実行 |
| ローカル DB | SQLite (sqlx) | ゼロ設定、ファイルベース、ポータブル |
| ローカルキャッシュ | インメモリ HashMap | 外部依存ゼロ |
| 一時ディレクトリ | `.k1s0/playground/` | manifest 管理外 |
| プロセス管理 | PID ファイル + SIGTERM/taskkill | クロスプラットフォーム対応 |

## 期待される効果

1. 新規開発者が 1 コマンドで k1s0 の出力を体験できる
2. **Docker がなくても体験可能**（ローカルモード自動フォールバック）
3. 各言語テンプレートの比較検証が容易になる
4. framework パッケージの統合テスト環境としても機能する
5. k1s0 のデモ・プレゼンテーションが容易になる（Docker なしの会議室 PC でも可能）
6. DB ドライバ抽象化のサンプルが k1s0 の「設定駆動」思想のデモとして教育的に機能する
7. `playground export` による試行→本番採用のワークフロー確立（将来）
