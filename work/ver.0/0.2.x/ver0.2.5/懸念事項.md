## 致命的な懸念（対処必須）

### C-01: オーバーレイマージの信頼性 ✅ 対処済み

**深刻度: 高** → **対処済み（Phase 1 で対応）**

playground テンプレートは「ベーステンプレート展開 → オーバーレイで上書き」の2段階で動作する。`TemplateRenderer::render_directory()` は現在1ディレクトリからの単純展開のみをサポートしており、オーバーレイマージ機能が存在しない。

| 操作 | 現在の TemplateRenderer | playground で必要 |
|------|------------------------|------------------|
| 単一ディレクトリ展開 | 対応済み | - |
| 2ディレクトリのマージ展開 | 未対応 | 必須 |
| 同名ファイルの上書き | `WriteResult::Overwritten` で対応済み | 必須 |
| 条件付きファイル追加 | 対応済み（Tera 条件分岐） | - |

**対処**: 案 B を採用 — `render_directory()` を2回呼び出す。

`TemplateRenderer` の `write_file()` は既に `WriteResult::Overwritten` を返す仕組みがあり、同名ファイルの上書きは動作する。playground.rs 内で以下の順序でレンダリングする:

```rust
// 1. ベーステンプレートを展開
let base_renderer = TemplateRenderer::new(&base_template_dir)?;
let base_result = base_renderer.render_directory(&output_dir, &context)?;

// 2. playground オーバーレイで上書き（同名ファイルは Overwritten）
let overlay_renderer = TemplateRenderer::new(&overlay_template_dir)?;
let overlay_result = overlay_renderer.render_directory(&output_dir, &context)?;

// 3. 共通オーバーレイを追加
let common_renderer = TemplateRenderer::new(&common_template_dir)?;
let common_result = common_renderer.render_directory(&output_dir, &context)?;
```

`k1s0-generator` クレートの変更は不要。将来的に `render_directory_with_overlays()` メソッドに昇格可能。

### C-02: init 不要で動作させるためのパス解決 ✅ 対処済み

**深刻度: 高** → **対処済み（段階的対応）**

通常の k1s0 コマンドは `.k1s0/` ディレクトリの存在を前提とする。playground は「どこからでも実行可能」を目指すが、テンプレートの参照パスがバイナリに同梱されていない。

| 実行環境 | .k1s0/ | テンプレート参照 |
|----------|--------|----------------|
| init 済みリポジトリ | 存在 | .k1s0/ 経由で参照可能 |
| init 未実行ディレクトリ | 未存在 | 参照不可 |
| $HOME 直下 | 未存在 | 参照不可 |

**対処**: 段階的アプローチを採用。

**Phase 1（即時）**: テンプレートパスの解決順序を実装:

```rust
fn resolve_template_dir() -> Result<PathBuf> {
    // 1. 環境に応じたバイナリ同梱テンプレートディレクトリ
    //    (exe の隣の templates/ を探す)
    let exe_dir = std::env::current_exe()?.parent();
    if exe_dir.join("templates").exists() {
        return Ok(exe_dir.join("templates"));
    }

    // 2. .k1s0/ ディレクトリ内のテンプレート
    if let Some(root) = find_k1s0_root() {
        return Ok(root.join("CLI/templates"));
    }

    // 3. $HOME/.k1s0/templates/ (レジストリからDL済み)
    let home_templates = dirs::home_dir().join(".k1s0/templates");
    if home_templates.exists() {
        return Ok(home_templates);
    }

    Err(anyhow!("テンプレートが見つかりません。k1s0 init を実行するか、k1s0 registry fetch でテンプレートを取得してください。"))
}
```

**Phase 5（将来）**: `include_dir!` マクロでバイナリ埋め込み。リリースビルドではテンプレートがバイナリに含まれ、外部ファイル依存がゼロになる。

playground ディレクトリの解決:

```rust
fn resolve_playground_base_dir() -> Result<PathBuf> {
    // 1. .k1s0/ が存在 → .k1s0/playground/
    if let Some(root) = find_k1s0_root() {
        return Ok(root.join(".k1s0/playground"));
    }

    // 2. 存在しない → $HOME/.k1s0/playground/
    let home = dirs::home_dir()
        .ok_or_else(|| anyhow!("HOME ディレクトリが取得できません"))?;
    Ok(home.join(".k1s0/playground"))
}
```

### C-03: Docker Compose ファイルのポートオフセット ✅ 対処済み

**深刻度: 中** → **対処済み（Phase 2 で対応）**

テンプレートの `docker-compose.yml.tera` はポートが固定値（8080, 50051, 5432 等）でハードコードされている。

**対処**: playground オーバーレイの `docker-compose.yml.tera` で Tera 変数を使用する。

```yaml
services:
  app:
    ports:
      - "{{ rest_port }}:8080"
      {% if with_grpc %}- "{{ grpc_port }}:50051"{% endif %}
  {% if with_db %}
  db:
    ports:
      - "{{ db_port }}:5432"
  {% endif %}
  {% if with_cache %}
  redis:
    ports:
      - "{{ redis_port }}:6379"
  {% endif %}
```

コンテナ内部のポートは固定（8080, 5432 等）のままで、ホスト側マッピングのみオフセットを適用する。これにより設定ファイルの変更は不要。

### C-04: Docker 未インストール環境への対応 ✅ 対処済み

**深刻度: 高** → **対処済み（Phase 1 で対応）**

開発者の端末に Docker がインストールされていないケースがある。Docker 前提では playground の「どこでも試せる」というコンセプトが成立しない。

| 環境 | Docker | Docker Compose | 想定割合 |
|------|:------:|:--------------:|:--------:|
| Docker Desktop インストール済み | あり | あり | 60% |
| Docker Engine のみ | あり | なし or 別途 | 10% |
| Docker なし（会社ポリシー等） | なし | なし | 20% |
| CI 環境（Docker 利用制限） | 状況次第 | 状況次第 | 10% |

**影響**: Docker 必須にすると、20-30% の開発者が playground を利用できない。

**対処**: `--mode` フラグでローカル実行モードを提供する。

```bash
# Docker モード（デフォルト: Docker が検出された場合）
k1s0 playground start --type backend-rust --with-db

# ローカルモード（Docker 不要）
k1s0 playground start --type backend-rust --mode local

# Docker が検出されない場合は自動的にローカルモードにフォールバック
```

#### ローカルモードの仕様

| 項目 | Docker モード | ローカルモード |
|------|:------------:|:------------:|
| サービス起動 | docker compose up | cargo run / go run / dotnet run / uvicorn |
| データベース | PostgreSQL コンテナ | SQLite (ファイル) |
| キャッシュ | Redis コンテナ | インメモリ HashMap |
| ポート管理 | Docker ポートマッピング | 直接 bind |
| ビルド前提 | Docker のみ | 各言語ツールチェーン (rustc, go, dotnet, python) |
| クリーンアップ | docker compose down + ファイル削除 | プロセス kill + ファイル削除 |
| gRPC | 完全対応 | 完全対応 |
| ネットワーク分離 | Docker ネットワーク | localhost のみ |

#### ローカルモードの処理フロー

```
k1s0 playground start --type backend-rust --mode local
  │
  ├─ 1. 言語ツールチェーン確認
  │     ├─ backend-rust: rustc --version && cargo --version
  │     ├─ backend-go: go version
  │     ├─ backend-csharp: dotnet --version
  │     ├─ backend-python: python --version && uv --version
  │     ├─ frontend-react: node --version && pnpm --version
  │     └─ frontend-flutter: flutter --version
  │
  ├─ 2. テンプレートレンダリング（Docker モードと同一）
  │     ├─ ベーステンプレート展開
  │     ├─ ローカルモード用オーバーレイ追加
  │     │   └─ config/default.yaml を SQLite / インメモリに差し替え
  │     └─ docker-compose.yml は生成しない
  │
  ├─ 3. 依存関係インストール
  │     ├─ backend-rust: cargo build
  │     ├─ backend-go: go mod download
  │     ├─ backend-csharp: dotnet restore
  │     ├─ backend-python: uv sync
  │     ├─ frontend-react: pnpm install
  │     └─ frontend-flutter: dart pub get
  │
  ├─ 4. DB 初期化（--with-db 指定時）
  │     └─ SQLite ファイル作成 + seed.sql 実行
  │
  ├─ 5. サービス起動（バックグラウンドプロセス）
  │     ├─ backend-rust: cargo run -- --env dev --config config/
  │     ├─ backend-go: go run ./cmd/ --env dev --config config/
  │     ├─ backend-csharp: dotnet run --project src/*.Presentation/
  │     ├─ backend-python: uvicorn {name}.presentation.main:app --port 8080
  │     ├─ frontend-react: pnpm dev --port 3000
  │     └─ frontend-flutter: flutter run -d chrome
  │     └─ PID を .playground.json に保存
  │
  ├─ 6. ヘルスチェック待機（Docker モードと同一ロジック）
  │
  └─ 7. 起動完了メッセージ
```

#### ローカルモード用 config/default.yaml の差分

```yaml
# Docker モード
database:
  host: db              # docker compose のサービス名
  port: 5432
  driver: postgres
  name: playground
  user: postgres
  password_file: /var/run/secrets/db_password

# ローカルモード（オーバーレイで差し替え）
database:
  driver: sqlite                          # SQLite にフォールバック
  path: ./data/playground.db              # ローカルファイル
```

```yaml
# Docker モード
cache:
  host: redis
  port: 6379

# ローカルモード
cache:
  driver: memory                          # インメモリにフォールバック
```

#### ローカルモードの stop 処理

```rust
fn stop_local(metadata: &PlaygroundMetadata) -> Result<()> {
    // 1. 保存された PID でプロセスを終了
    if let Some(pid) = metadata.pid {
        #[cfg(unix)]
        {
            nix::sys::signal::kill(Pid::from_raw(pid), Signal::SIGTERM)?;
        }
        #[cfg(windows)]
        {
            Command::new("taskkill").args(["/PID", &pid.to_string(), "/F"]).status()?;
        }
    }

    // 2. SQLite ファイル削除
    // 3. playground ディレクトリ削除
    fs::remove_dir_all(&metadata.dir)?;
    Ok(())
}
```

#### 自動モード検出

```rust
fn detect_mode(explicit_mode: Option<&str>) -> PlaygroundMode {
    match explicit_mode {
        Some("docker") => PlaygroundMode::Docker,
        Some("local") => PlaygroundMode::Local,
        None => {
            // Docker が利用可能か自動検出
            if check_docker().is_ok() && check_docker_compose().is_ok() {
                PlaygroundMode::Docker
            } else {
                output().warning(
                    "Docker が検出されませんでした。ローカルモードで起動します。"
                );
                PlaygroundMode::Local
            }
        }
        _ => unreachable!(),
    }
}
```

## 設計上の懸念

### D-01: playground テンプレートのメンテナンスコスト ✅ 対処済み

**深刻度: 中** → **対処済み（段階的リリース）**

6テンプレート × オーバーレイファイル × 2モード（Docker/ローカル）でメンテナンスが増大するリスク。

**対処**:

1. **Phase 2 では backend-rust のみ実装**。他テンプレートは Phase 4 で需要に応じて追加
2. **モード差分は config/default.yaml のみ**に集約。サンプルコード本体はモード間で共通
3. **CI にテンプレートレンダリングテストを追加**し、ベーステンプレート変更時の破壊を自動検知

```
テンプレート階層（メンテナンスコスト最小化設計）:

CLI/templates/playground/
├── common/                     # 全テンプレート共通（2ファイル）
├── common-local/               # ローカルモード共通（1ファイル: config差分）
├── backend-rust/               # Rust 固有（9ファイル: サンプルコード）
├── backend-rust-local/         # Rust ローカル差分（1ファイル: config/default.yaml のみ）
└── ...
```

モード差分ファイルは `config/default.yaml` の1ファイルのみに抑える。サンプルコード（handlers.rs 等）はモード間で完全共通。

### D-04: ローカルモードのDB抽象化 ✅ 対処済み

**深刻度: 中** → **対処済み（Phase 2 で対応）**

同一のサンプルコードが PostgreSQL（Docker モード）と SQLite（ローカルモード）の両方で動作する必要がある。

**対処**: サンプルコードの infrastructure 層でドライバを抽象化する。

```rust
// src/infrastructure/repositories/item_repository.rs

pub enum DbDriver {
    Postgres(PgPool),
    Sqlite(SqlitePool),
}

impl ItemRepository {
    pub async fn new(config: &DatabaseConfig) -> Result<Self> {
        match config.driver.as_str() {
            "postgres" => {
                let pool = PgPool::connect(&config.connection_string()).await?;
                Ok(Self { driver: DbDriver::Postgres(pool) })
            }
            "sqlite" => {
                let pool = SqlitePool::connect(&config.path).await?;
                sqlx::migrate!("./migrations").run(&pool).await?;
                Ok(Self { driver: DbDriver::Sqlite(pool) })
            }
            _ => Err(anyhow!("Unknown database driver: {}", config.driver)),
        }
    }
}
```

この設計は k1s0 の「設定駆動」の思想とも合致し、サンプルコードとして教育的価値が高い。

### B-01: 既存コマンドへの影響 ✅ 確認済み

**深刻度: 低** → **確認済み**

| 影響範囲 | 影響 |

**深刻度: 中** → **対処済み**

### O-02: ポート競合の検出 ✅ 対処済み

**深刻度: 中** → **対処済み**

**対処**: `start` 実行前に対象ポートの使用状況をチェックする。

```rust
fn check_port_available(port: u16) -> bool {
    std::net::TcpListener::bind(("127.0.0.1", port)).is_ok()
}

fn check_ports(ports: &PortConfig) -> Result<()> {
    let mut conflicts = Vec::new();

    if !check_port_available(ports.rest_port) {
        conflicts.push(format!("REST: {} (使用中)", ports.rest_port));
    }
    if ports.grpc_port > 0 && !check_port_available(ports.grpc_port) {
        conflicts.push(format!("gRPC: {} (使用中)", ports.grpc_port));
    }
    // ... db_port, redis_port

    if !conflicts.is_empty() {
        output().warning(&format!(
            "以下のポートが使用中です:\n  {}\n\n--port-offset を指定して別のポートを使用してください:\n  k1s0 playground start --port-offset 100",
            conflicts.join("\n  ")
        ));
        return Err(anyhow!("ポート競合が検出されました"));
    }
    Ok(())
}
```

Docker モード・ローカルモードの両方で同一のチェックロジックを使用。

### O-03: ローカルモードのプロセス管理 ✅ 対処済み

**深刻度: 中** → **対処済み**

ローカルモードではバックグラウンドプロセスとしてサービスを起動する。プロセスの追跡と確実な停止が必要。

**対処**:

1. `.playground.json` に PID を保存
2. `playground status` で PID の生存確認（kill -0 / tasklist）
3. `playground stop` で SIGTERM → 3秒待機 → SIGKILL のグレースフルシャットダウン
4. PID ファイルが残っているがプロセスが死んでいる場合は「stopped (orphaned)」と表示

```rust
fn is_process_alive(pid: u32) -> bool {
    #[cfg(unix)]
    {
        nix::sys::signal::kill(Pid::from_raw(pid as i32), None).is_ok()
    }
    #[cfg(windows)]
    {
        Command::new("tasklist")
            .args(["/FI", &format!("PID eq {}", pid), "/NH"])
            .output()
            .map(|o| String::from_utf8_lossy(&o.stdout).contains(&pid.to_string()))
            .unwrap_or(false)
    }
}
```

## テストの懸念

### T-01: Docker 依存のテスト ✅ 対処済み

**深刻度: 中** → **対処済み**

**対処**: テストを3層に分離する。

| テスト層 | Docker 必要 | CI 実行 | 内容 |
|---------|:-----------:|:------:|------|
| ユニットテスト | 不要 | 常時 | ユーティリティ関数、メタデータ、ポート計算 |
| テンプレートテスト | 不要 | 常時 | レンダリング結果の検証 |
| 統合テスト（ローカル） | 不要 | 常時 | ローカルモードのライフサイクル |
| 統合テスト（Docker） | 必要 | オプション | Docker モードのライフサイクル |

```rust
#[test]
fn test_resolve_ports() { /* 常に実行 */ }

#[test]
fn test_template_rendering() { /* 常に実行 */ }

#[test]
#[cfg_attr(not(feature = "docker-tests"), ignore)]
fn test_docker_lifecycle() { /* Docker 利用可能時のみ */ }
```

ローカルモードの統合テストは Docker 不要で実行できるため、CI カバレッジが大幅に向上する。
