## 全体像

```
開発者が k1s0 を初めて触る
  │
  ├─ Docker あり → k1s0 playground start --type backend-rust --with-db
  │   (PostgreSQL + axum サービスが Docker Compose で起動)
  │
  ├─ Docker なし → k1s0 playground start --type backend-rust --with-db
  │   (自動検出でローカルモードにフォールバック)
  │   (SQLite + cargo run でサービスが直接起動)
  │
  └─ モード明示 → k1s0 playground start --type backend-rust --mode local
      (Docker があってもローカルモードを使用)
```

```
モード自動検出フロー:

k1s0 playground start (--mode 未指定)
  │
  ├─ Docker + Docker Compose 検出 → Docker モードで起動
  │
  └─ Docker 未検出
      ├─ 言語ツールチェーン検出 → ローカルモードで起動（警告メッセージ付き）
      └─ ツールチェーンも未検出 → エラー + 両方のインストール案内
```

## 初回体験フロー — Docker あり

### Step 1: playground を起動する

```bash
# k1s0 CLI がインストール済みであること
k1s0 --version

# 引数なしで実行するとインタラクティブモードになる
k1s0 playground start

# 以下の選択肢が表示される:
#   ? テンプレートを選択してください:
#   > backend-rust    — Rust (axum) backend service
#     backend-go      — Go backend service
#     backend-csharp  — C# (ASP.NET Core) backend service
#     backend-python  — Python (FastAPI) backend service
#     frontend-react  — React (Material-UI) frontend app
#     frontend-flutter — Flutter frontend app
#
#   ? 実行モードを選択してください:
#   > Docker (推奨)   — Docker Compose で起動。PostgreSQL/Redis をコンテナで実行
#     ローカル         — 直接プロセスを起動。DB は SQLite を使用
#
#   ? 有効にする機能を選択してください:
#   > [x] gRPC support
#     [x] Database (PostgreSQL)
#     [ ] Cache (Redis)
```

### Step 2: 起動を確認する

```bash
# 起動完了メッセージが表示される:
#
#   Playground 'playground-20260131-120000' is running!
#   Mode: docker
#
#   REST API:  http://localhost:8080
#   gRPC:      localhost:50051
#   Database:  PostgreSQL localhost:5432
#
#   Try:
#     curl http://localhost:8080/health
#     curl http://localhost:8080/api/v1/items

# ヘルスチェック
curl http://localhost:8080/health
# => {"status":"ok"}

# サンプルデータ取得
curl http://localhost:8080/api/v1/items
# => [{"id":1,"name":"Sample Item 1","description":"..."},{"id":2,...}]
```

### Step 3: API を試す

```bash
# 新しいアイテムを作成
curl -X POST http://localhost:8080/api/v1/items \
  -H 'Content-Type: application/json' \
  -d '{"name":"My Item","description":"Created from playground"}'
# => {"id":3,"name":"My Item","description":"Created from playground","created_at":"..."}

# ID 指定で取得
curl http://localhost:8080/api/v1/items/3

# 更新
curl -X PUT http://localhost:8080/api/v1/items/3 \
  -H 'Content-Type: application/json' \
  -d '{"name":"Updated Item","description":"Modified"}'

# 削除
curl -X DELETE http://localhost:8080/api/v1/items/3
# => 204 No Content
```

### Step 4: 停止する

```bash
k1s0 playground stop
# => Playground 'playground-20260131-120000' を停止・削除しました。
```

## 初回体験フロー — Docker なし（ローカルモード）

Docker がインストールされていない環境（会社ポリシーで Docker 禁止、軽量 Linux 環境など）での体験フロー。

### Step 1: 前提ツールの確認

```bash
# Rust の場合、rustc と cargo が必要
rustc --version   # => rustc 1.85.0
cargo --version   # => cargo 1.85.0

# Go の場合
go version        # => go1.22.0

# Python の場合
python --version  # => Python 3.12.0
uv --version      # => uv 0.5.0
```

### Step 2: playground を起動する

```bash
# --mode local を明示、または Docker 未検出時に自動フォールバック
k1s0 playground start --type backend-rust --with-db --mode local

# Docker 未検出時の自動フォールバック:
#   ⚠ Docker が検出されませんでした。ローカルモードで起動します。
#     Docker モードを使用するには Docker Desktop をインストールしてください。
#
# ビルドが実行される:
#   Building playground... (cargo build --release)
#   ████████████████████████████████████ 100%
#
# 起動完了メッセージ:
#
#   Playground 'playground-20260131-120000' is running!
#   Mode: local (PID: 12345)
#
#   REST API:  http://localhost:8080
#   Database:  SQLite ./data/playground.db
#
#   Try:
#     curl http://localhost:8080/health
#     curl http://localhost:8080/api/v1/items
#
#   Logs: .k1s0/playground/playground-20260131-120000/logs/server.log
#   Stop: k1s0 playground stop
```

### Step 3: API を試す（Docker モードと同一）

```bash
# 同じ API エンドポイントが同じレスポンスを返す
curl http://localhost:8080/health
# => {"status":"ok"}

curl http://localhost:8080/api/v1/items
# => [{"id":1,"name":"Sample Item 1","description":"..."},{"id":2,...}]

curl -X POST http://localhost:8080/api/v1/items \
  -H 'Content-Type: application/json' \
  -d '{"name":"My Item","description":"Created in local mode"}'
# => {"id":3,"name":"My Item",...}
```

### Step 4: ログを確認する

```bash
# ローカルモードではログファイルにサーバー出力が記録される
tail -f .k1s0/playground/playground-20260131-120000/logs/server.log
```

### Step 5: 停止する

```bash
k1s0 playground stop
# => プロセス (PID: 12345) を停止しました。
# => Playground 'playground-20260131-120000' を削除しました。
```

### Docker モードとの違い（開発者が意識すべき点）

| 項目 | Docker モード | ローカルモード |
|------|:------------:|:------------:|
| API エンドポイント | 同一 | 同一 |
| レスポンス形式 | 同一 | 同一 |
| データベース | PostgreSQL (コンテナ) | SQLite (ファイル) |
| 初回起動 | Docker イメージビルド | cargo build |
| ログ確認 | `docker compose logs` | `tail -f logs/server.log` |
| データの永続性 | Docker ボリューム | `./data/playground.db` |

**重要**: API インターフェースは両モードで完全に同一。DB ドライバの違いは infrastructure 層で吸収されるため、アプリケーション/ドメイン層のコードはモードに依存しない。

## 技術検証フロー（テックリード）

チームのテックリードが Rust と Go のどちらを採用するか検証するシナリオ。

### Docker あり: 並行起動で比較

```bash
# Rust playground をデフォルトポートで起動
k1s0 playground start --type backend-rust --with-grpc --with-db

# Go playground をポートオフセット 100 で並行起動
k1s0 playground start --type backend-go --with-grpc --with-db --port-offset 100

# 稼働状況を確認
k1s0 playground status
# NAME                          TYPE           MODE     OPTIONS      STATUS    PORTS
# playground-20260131-120000    backend-rust   docker   grpc,db      running   8080,50051
# playground-20260131-120100    backend-go     docker   grpc,db      running   8180,50151
```

### Docker なし: ローカルモードで比較

```bash
# Rust playground（ローカルモード）
k1s0 playground start --type backend-rust --with-db --mode local

# Go playground（ポートオフセットでポート分離）
k1s0 playground start --type backend-go --with-db --mode local --port-offset 100

# 稼働状況
k1s0 playground status
# NAME                          TYPE           MODE     OPTIONS    STATUS    PORTS       DISK
# playground-20260131-120000    backend-rust   local    db         running   8080        12 MB
# playground-20260131-120100    backend-go     local    db         running   8180        8 MB
```

### 比較検証

```bash
# 同一 API を比較
curl http://localhost:8080/api/v1/items   # Rust
curl http://localhost:8180/api/v1/items   # Go
# 同一の Item CRUD インターフェースなので直接比較可能
```

### クリーンアップ

```bash
# 全ての playground を一括停止
k1s0 playground stop --yes
```

## playground から本番への移行フロー

### Step 1: リポジトリを初期化（未実行の場合）

```bash
k1s0 init
```

### Step 2: 正式な feature を生成

```bash
# playground で検証したのと同じオプションで feature を生成
k1s0 new-feature --type backend-rust --name order-processing --with-grpc --with-db
```

### Step 3: playground のコードを参考にビジネスロジックを実装

```
playground のサンプルコード → 本番コードへの対応:

  playground: Item エンティティ    → 本番: Order エンティティ
  playground: CRUD ユースケース    → 本番: ビジネスロジック付きユースケース
  playground: ItemRepository      → 本番: OrderRepository（PostgreSQL のみ）
  playground: SQLite/PostgreSQL   → 本番: PostgreSQL のみ（DB 抽象化は不要）
```

### Step 4: playground を停止

```bash
k1s0 playground stop
```

> **将来（v0.2.6 以降予定）**: `k1s0 playground export` で playground のコードを直接 feature に昇格できるようになる予定。

## よくある操作シナリオ

### シナリオ 1: フロントエンド + バックエンドの統合検証

```bash
# バックエンドを起動（Docker またはローカル）
k1s0 playground start --type backend-rust --with-db --name api-server

# フロントエンドを起動（ポートオフセットで分離）
k1s0 playground start --type frontend-react --name web-app --port-offset 1000

# ブラウザで確認: http://localhost:9080
```

### シナリオ 2: Docker なし環境での開発研修

```bash
# 研修用 PC に Docker がない場合でもローカルモードで研修可能
k1s0 playground start --type backend-python --with-db --mode local
# → Python + SQLite で即座に動作確認

# 研修終了後
k1s0 playground stop --yes
```

### シナリオ 3: CI でのテンプレート検証

```bash
# CI パイプラインで Docker を使わずにテンプレートの動作確認
k1s0 playground start --type backend-rust --with-db --mode local --yes
curl --retry 5 --retry-delay 2 http://localhost:8080/health
k1s0 playground stop --yes
```

### シナリオ 4: デモ・プレゼンテーション

```bash
# Docker ありの環境で本番に近いデモ
k1s0 playground start --type backend-rust --with-grpc --with-db --yes
# デモ実施...
k1s0 playground stop --yes

# Docker なしの会議室 PC でのデモ
k1s0 playground start --type backend-rust --with-db --mode local --yes
# デモ実施...
k1s0 playground stop --yes
```

## 禁止事項と注意点

### playground のコードを本番にそのまま使わない

playground はデモ・学習用であり、以下の点で本番コードとは異なる：

- エラーハンドリングが簡略化されている
- 認証・認可が含まれていない
- ログ出力が最小限
- テストが含まれていない
- DB ドライバ抽象化は playground 用（本番では PostgreSQL 固定で良い）

```bash
# NG: playground ディレクトリを直接 feature/ にコピー
cp -r .k1s0/playground/playground-xxx/ feature/backend/rust/my-service/

# OK: new-feature で正式に生成し、playground を参考に実装
k1s0 new-feature --type backend-rust --name my-service --with-db
```

### playground で環境変数を使わない

playground のサンプルコードも k1s0 の規約に準拠している。設定は `config/default.yaml` 経由で行う。

```yaml
# OK: config/default.yaml で設定（Docker モード）
database:
  driver: postgres
  host: db
  port: 5432
  password_file: /var/run/secrets/db_password

# OK: config/default.yaml で設定（ローカルモード）
database:
  driver: sqlite
  path: ./data/playground.db
```

```rust
// NG: 環境変数を使用
let port = std::env::var("PORT").unwrap();
let db_url = std::env::var("DATABASE_URL").unwrap();

// OK: k1s0-config パターン
let config = load_config("config/default.yaml")?;
let port = config.server.port;
let db = ItemRepository::new(&config.database).await?;
```

### playground の長時間放置に注意

```bash
# 使い終わったら必ず停止
k1s0 playground stop

# 稼働中の playground とディスク使用量を確認
k1s0 playground status
# Total: 2 playgrounds (2 running)
# Disk usage: 543 MB

# ローカルモードでも放置するとプロセスとファイルが残る
```

### ローカルモードのビルドキャッシュ

ローカルモードでは `target/`（Rust）や `node_modules/`（React）がディスクを消費する。`playground stop` で自動削除されるが、ビルド中に中断した場合は手動削除が必要な場合がある。

```bash
# 残骸の確認
ls $HOME/.k1s0/playground/

# 手動削除（必要な場合のみ）
rm -rf $HOME/.k1s0/playground/playground-xxx/
```

## まとめ: 日常のコマンド

```bash
# 朝: playground で新しいアイデアを試す
k1s0 playground start --type backend-rust --with-db
# → Docker あり: PostgreSQL + Docker Compose
# → Docker なし: SQLite + cargo run（自動フォールバック）

# 開発中: API を確認（モードに関わらず同じ）
curl http://localhost:8080/api/v1/items

# 採用決定: 正式な feature を作成
k1s0 new-feature --type backend-rust --name my-service --with-db

# 帰宅前: playground をクリーンアップ
k1s0 playground stop
```
