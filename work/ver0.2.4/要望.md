# 要望

Python と C# のバックエンドフレームワークに不足している4パッケージを追加したい。

現在、Rust と Go では全12パッケージが実装済みだが、Python と C# では以下の4パッケージが未実装のため、本番サービス構築に制約がある。

## 背景

- Rust: 12パッケージ全て実装済み（k1s0-auth, k1s0-cache, k1s0-domain-event, k1s0-resilience 含む）
- Go: 12パッケージ全て実装済み
- Python: 8パッケージ実装済み、**4パッケージ未実装**
- C#: 8パッケージ実装済み、**4パッケージ未実装**

## 未実装パッケージ

| パッケージ | Tier | 説明 | Python | C# |
|-----------|------|------|:------:|:--:|
| k1s0-auth / K1s0.Auth | 3 | JWT/OIDC認証、ポリシーエンジン、ミドルウェア | 未実装 | 未実装 |
| k1s0-cache / K1s0.Cache | 2 | Redis キャッシュ、キャッシュパターン（Cache-Aside, Write-Through 等） | 未実装 | 未実装 |
| k1s0-domain-event / K1s0.DomainEvent | 2 | ドメインイベント、Outbox パターン、Pub/Sub | 未実装 | 未実装 |
| k1s0-resilience / K1s0.Resilience | 2 | サーキットブレーカー、タイムアウト、バルクヘッド、リトライ | 未実装 | 未実装 |

## 影響

これらのパッケージがないと、Python/C# バックエンドサービスで以下ができない:

- JWT/OIDC 認証付きエンドポイントの構築
- Redis を使ったキャッシュ戦略の実装
- ドメインイベントによるサービス間連携
- 障害耐性パターン（サーキットブレーカー等）の適用

## 回答（実装方針）

### 概要

8パッケージ（Python 4 + C# 4）を新規実装する。Rust/Go の既存実装をリファレンスとし、各言語のエコシステムに適した技術を選択する。

### 実装順序（複雑度の低い順）

1. **k1s0-domain-event** — 最小（Rust ~611 LOC）。インターフェース中心で移植容易
2. **k1s0-resilience** — 中（Rust ~1,682 LOC）。外部依存なし、自己完結
3. **k1s0-cache** — 高（Rust ~5,103 LOC）。k1s0-health に依存（optional）
4. **k1s0-auth** — 最高（Rust ~5,484 LOC）。Tier 3、k1s0-cache/k1s0-db に依存（optional）

### 技術選定

| パッケージ | Python | C# |
|-----------|--------|-----|
| domain-event | abc.ABC + dataclasses + asyncio | interfaces + records + async/await |
| resilience | asyncio + Semaphore | Polly 互換 or 自前実装 |
| cache | redis-py + connection pool | StackExchange.Redis |
| auth | PyJWT + python-jose + httpx | Microsoft.AspNetCore.Authentication.JwtBearer |

### スコープ決定

k1s0-auth について、Rust 版はフル機能（JWT + OIDC + ポリシーエンジン + 監査ログ + ブラックリスト + リフレッシュトークン）だが、Go 版は最小構成（JWT + OIDC + ミドルウェア）。

**Python/C# は Rust 版のフル機能を目標とする。** Go 版は軽量版として位置づけ、Python/C# は Rust 版と同等の機能パリティを持たせる。

## 期待される効果

1. **言語パリティの達成**: 全4言語で同等のフレームワーク機能を提供
2. **Python/C# サービスの本番投入可能化**: 認証・キャッシュ・イベント・耐障害性の標準パターンが利用可能に
3. **開発者体験の統一**: どの言語を選んでも同じ抽象化レベルで開発可能
