
# k1s0 技術的特徴（PR資料）

## 1. サマリー（ひとことで）

k1s0 は、**テンプレ・CLI・共通 framework** をセットで提供し、マイクロサービス開発の「立ち上げ」「規約準拠」「契約管理」「アップグレード」を **人の記憶ではなく仕組みで固定**するためのモノレポ基盤です。

狙いは 1 つです。

- 「各サービスが頑張って同じことをやる」をやめ、
- 「頑張らなくても同じやり方で安全に動く」を既定にする

---

## 2. 解決する課題（現場の痛み）

マイクロサービスが増えるほど、次が“地味に”積み上がって開発速度を落とします。

- サービスごとに構成・起動手順・設定の読み方が微妙に違い、初速が出ない
- 規約がドキュメント止まりで、レビューの主観に依存する
- proto / OpenAPI の破壊的変更が混入し、下流が壊れてから発覚する
- secrets が YAML や環境変数に散らばり、事故・監査・ローカル再現が難しい
- 観測（ログ/トレース）がバラつき、障害対応が「まず形式理解」から始まる
- 共通基盤のアップグレードが怖くて止まり、結果的に分岐・フォーク・属人化する

k1s0 は、この“積み上がる負債”を **テンプレと自動検査（lint/CI）と upgrade 支援**で抑え込みます。

---

## 3. 3つの構成要素（セットで効く）

### 3.1 templates（雛形の固定）

サービスの最小構造（ディレクトリ/必須ファイル/契約置き場/デプロイ置き場）をテンプレとして同梱します。

- 例: `config/{default,dev,stg,prod}.yaml`
- 例: `deploy/base` と `deploy/overlays/{dev,stg,prod}`
- 例: `proto/`（gRPC 正本）、`openapi/openapi.yaml`（REST 正本）

### 3.2 CLI（操作の固定）

人間の入口を CLI に寄せ、作業を手順ではなくコマンドにします。

- `k1s0 init`（初期化）
- `k1s0 new-feature`（雛形生成）
- `k1s0 lint`（規約検査）
- `k1s0 upgrade --check` / `k1s0 upgrade`（差分提示と限定適用）

### 3.3 framework（実装の固定）

「各サービスが毎回実装して事故る」領域を共通化し、同じやり方で安全に動く標準実装を提供します。

- 設定/秘密情報の読み込み（環境変数に依存しない）
- 観測性（ログ/トレースの初期化と規約）
- エラー表現の統一（原因が分かる形）
- エラー表現の統一（内部分類 → `error_code` → REST/gRPC への自動変換）
- gRPC クライアント/サーバ基盤（共通ビルダーで deadline 必須・retry 既定 0・timeout 下限/上限・OTel 計装・メタデータ付与を強制）
- gRPC のリトライ例外管理（冪等 RPC のみ opt-in + lint 検知 + ADR 連携）
- Service discovery の統一（K8s DNS 前提の論理名指定。環境差は config で吸収）
- “ローカルは Mesh なし”を既定にしても同じコードで成立する入口（h2c を許容し、Kubernetes では Mesh/Proxy で mTLS を担保）
- DB アクセス標準（接続/Tx/エラー変換/クエリ計測 + マイグレーション起動の定型）
- 認証/認可の共通部品（JWT/OIDC + policy + audit）
- 入力バリデーション（API 境界の共通ルール + エラー表現の統一）
- 依存先呼び出しのガードレール（timeout 上下限/同時実行制限/必要時のみ CB）
- フロントエンド（React）の共通 UI/レイアウト/ナビゲーション（MUI + AppShell + 設定駆動）
- フロントエンド（React）の API クライアント統一（生成SDK + 認証 + retry/timeout + 計測）
- フロントエンド（React）の標準エラー表示（problem details 表示 + `error_code`/`trace_id` の露出 + リトライ導線）
- フロントエンド（React）の Design/UX 標準（テーマ/フォーム/通知/ダイアログ等）

## 4. 技術的特徴（差別化ポイント）

### 特徴A: 規約を「書く」から「落とす」に変える（lint + CI で強制）

- 必須ファイル/構造、manifest の整合性、禁止事項を機械的に検査
- ルールはレビューの好みではなく、CI のマージゲートになる
- 失敗は「原因/対象/次アクション」を必ず出す（運用で迷子にしない）

### 特徴B: `.k1s0/manifest.json` で“更新境界”を保存し、upgrade を成立させる

テンプレ更新で最も危険なのは「どこまで上書きしてよいか」が曖昧なことです。
k1s0 は生成時に manifest を残し、次を固定します。

- 生成元テンプレ（name/version/path/fingerprint）
- 更新してよい領域（managed）
- 触らない領域（protected）
- 衝突時は止まる（安全に止まるが既定）

### 特徴C: 契約（proto/OpenAPI）を“正本”として扱い、破壊を早期に止める

- gRPC: `buf lint` / `buf breaking` を前提に置き場を固定
- REST: OpenAPI を正本にし、差分検知の導線を持つ
- 生成物は Git 管理外でも、再現性（生成一致）を CI で担保できる設計

### 特徴D: secrets を「環境変数」ではなく「ファイル参照」に固定する

- 設定 YAML に値を直書きしない（参照キー `*_file` を標準に）
- Kubernetes では Secret を volume mount、ローカルも同形に写像
- “うっかりコミット”と“環境差分”を構造的に減らす

### 特徴E: 観測性をオプションにしない（最低限を既定で揃える）

- JSON 1 行ログの必須フィールドを揃え、トレースと相関
- サービス間呼び出しは trace context を伝播
- 障害対応が「形式理解」から始まらない

### 特徴F: 通信品質を“API 形状”で担保する（deadline 必須・retry 原則禁止）

- gRPC の無制限待ちは“実装ミス”ではなく“入りにくい設計”に寄せる
- retry は既定 0、例外は明示（冪等性・回数・バックオフ等）
- 例外（retry opt-in）は lint で検知し、ADR とセットで管理できる（“例外が見える”）
- timeout は下限/上限を設け、共通ビルダーで強制する（値の逸脱は設定バリデーションで落とす）
- trace/tenant 等のメタデータ付与を interceptor で自動化する

### 特徴F.1: Service discovery と Mesh 境界を固定する（環境差は config に閉じ込める）

- 依存先は論理名を正本にし、K8s DNS 前提で解決する（`{service}.{namespace}.svc.cluster.local` 等）
- ローカルは Mesh なし/h2c を既定にしつつ、同じ client builder で Kubernetes（Mesh あり/なし）へ持ち込める

### 特徴H: エラーモデル統一で運用を強くする（動く/動かないではなく“見れば分かる”）

- 内部のエラー分類（入力不備/依存障害/一時障害/権限/不変条件違反）を固定
- 外部表現（gRPC status / HTTP problem details）を統一し、`error_code` を安定識別子として必須化
- 変換（内部分類→`error_code`→外部表現）は framework が自動化し、サービスが独自変換を持たない

### 特徴I: DB アクセス標準で“遅い/壊れる”を観測できる

- 接続/Tx/エラー変換/クエリ計測を framework 側に寄せ、サービス差を消す
- 遅延・失敗がメトリクス/トレースに自動で出る（当番が即判断できる）
- マイグレーション起動の入口も共通化し、サービスごとに“やり方の違い”が生まれない

### 特徴J: 認証/認可を共通化し、監査ログまで既定で揃う

- JWT/OIDC 検証とポリシー評価を共通部品化
- “誰が何をしたか”の監査ログが標準で残る

### 特徴K: 入力バリデーションを統一し、“動くけど曖昧”を減らす

- API 境界のバリデーションとエラー返却形式を共通化
- OpenAPI/proto と整合する（破壊的なバリデーション強化は段階移行）

### 特徴L: 依存先呼び出しのガードレールで雪崩を防ぐ

- timeout の上限/下限、同時実行制限、バルクヘッドを標準提供
- サーキットブレーカは必要時のみ（既定OFF）で、使う場合も共通実装に寄せる

### 特徴G: フロントの“配線”をノーコード化する（設定駆動ナビ + 共通レイアウト）

- UI は MUI（Material UI）を標準採用し、見た目/実装パターンの分岐を抑える
- Header / Footer / Menu は framework が提供し、個別チームはページ本体に集中できる
- 画面遷移（routes/menus/flows）を `config/{env}.yaml` 等の設定で制御し、配線変更をコード変更から切り離す
	- 権限（permission）や feature flag による表示/遷移制御も設定で表現できる

狙い: 「画面を足すたびにルーティング/メニュー/レイアウトの儀式を繰り返す」をやめ、工数を最小化する。

### 特徴M: フロントの API クライアントを統一し、実装と運用の差を消す

- 生成SDK（OpenAPI/gRPC-web 等）を共通 client で包み、認証付与/timeout/retry/計測をセットで提供
- 画面ごとに fetch/axios の流儀が増えない
- problem details を標準 UI で表示し、`error_code` と `trace_id` が自然に露出する（調査が速い）

### 特徴N: Design/UX 標準で“独自流儀”を防ぐ

- テーマ、フォーム/エラー表示、通知/ダイアログを共通部品として提供
- UIの品質差・実装差を減らし、レビュー観点と実装工数を固定する

## 5. 技術仕様の骨子（PRで語れる「固定」）

### 5.1 ディレクトリ規約（置き場の固定）

- backend: `feature/backend/rust/{feature_name}/`
- frontend: `feature/frontend/flutter/{feature_name}/`, `feature/frontend/react/{feature_name}/`

backend（Rust）の最小構造（lint 対象）

```
src/
	application/
	domain/
	infrastructure/
	presentation/
config/
	default.yaml
	dev.yaml
	stg.yaml
	prod.yaml
deploy/
	base/
	overlays/
		dev/
		stg/
		prod/
proto/
openapi/
	openapi.yaml
```

### 5.2 設定/秘密情報（暗黙をやめる）

- 起動引数で `--env` / `--config` / `--secrets-dir` を明示（暗黙選択しない）
- YAML は非機密の参照束、機密はファイルに置く

### 5.3 upgrade（上書きではなく“境界のある更新”）

- `k1s0 upgrade --check` が影響範囲と衝突を一覧化
- `k1s0 upgrade` は managed を中心に安全に適用し、衝突は止める

## 6. 想定ユースケース（導入後の見え方）

- 新規サービス追加: `k1s0 new-feature` で雛形が揃い、生成直後から `k1s0 lint` が通る
- 契約変更: proto/OpenAPI の破壊が CI で止まり、意図しない破壊が入りにくい
- 秘密情報: “値を持たない運用”が既定となり、レビューが安全になる
- アップグレード: 差分が見える/止まれるので、基盤更新が継続できる
- フロント画面追加: 画面コンポーネント実装 + 設定追記だけで、メニュー/遷移/レイアウトに接続できる

## 8. 付録: 実装ロードマップ（要約）

実装は「MVP で“落とせる”ところまで」→「運用品質の固定を増やす」の順で進めます。

- MVP: CLI 骨格、backend テンプレ、manifest、lint（必須構造/禁止事項の最低限）、CI 必須化
- 拡張: framework の充実（config/error/observability/grpc-client）、契約管理/生成一致、upgrade 精度向上
