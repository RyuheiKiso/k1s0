
# k1s0 技術的特徴（PR資料）

## 1. サマリー（ひとことで）

k1s0 は、**テンプレ・CLI・共通 framework** をセットで提供し、マイクロサービス開発の「立ち上げ」「規約準拠」「契約管理」「アップグレード」を **人の記憶ではなく仕組みで固定**するためのモノレポ基盤です。

狙いは 1 つです。

- 「各サービスが頑張って同じことをやる」をやめ、
- 「頑張らなくても同じやり方で安全に動く」を既定にする

---

## 2. 解決する課題（現場の痛み）

マイクロサービスが増えるほど、次が“地味に”積み上がって開発速度を落とします。

- サービスごとに構成・起動手順・設定の読み方が微妙に違い、初速が出ない
- 規約がドキュメント止まりで、レビューの主観に依存する
- proto / OpenAPI の破壊的変更が混入し、下流が壊れてから発覚する
- secrets が YAML や環境変数に散らばり、事故・監査・ローカル再現が難しい
- 観測（ログ/トレース）がバラつき、障害対応が「まず形式理解」から始まる
- 共通基盤のアップグレードが怖くて止まり、結果的に分岐・フォーク・属人化する

k1s0 は、この“積み上がる負債”を **テンプレと自動検査（lint/CI）と upgrade 支援**で抑え込みます。

---

## 3. 3つの構成要素（セットで効く）

### 3.1 templates（雛形の固定）

サービスの最小構造（ディレクトリ/必須ファイル/契約置き場/デプロイ置き場）をテンプレとして同梱します。

- 例: `config/{default,dev,stg,prod}.yaml`
- 例: `deploy/base` と `deploy/overlays/{dev,stg,prod}`
- 例: `proto/`（gRPC 正本）、`openapi/openapi.yaml`（REST 正本）

### 3.2 CLI（操作の固定）

人間の入口を CLI に寄せ、作業を手順ではなくコマンドにします。

- `k1s0 init`（初期化）
- `k1s0 new-feature`（雛形生成）
- `k1s0 lint`（規約検査）
- `k1s0 upgrade --check` / `k1s0 upgrade`（差分提示と限定適用）

### 3.3 framework（実装の固定）

「各サービスが毎回実装して事故る」領域を共通化し、同じやり方で安全に動く標準実装を提供します。

- 設定/秘密情報の読み込み（環境変数に依存しない）
- 観測性（ログ/トレースの初期化と規約）
- エラー表現の統一（原因が分かる形）
- gRPC クライアント初期化（deadline 必須・retry 原則禁止の設計）

## 4. 技術的特徴（差別化ポイント）

### 特徴A: 規約を「書く」から「落とす」に変える（lint + CI で強制）

- 必須ファイル/構造、manifest の整合性、禁止事項を機械的に検査
- ルールはレビューの好みではなく、CI のマージゲートになる
- 失敗は「原因/対象/次アクション」を必ず出す（運用で迷子にしない）

### 特徴B: `.k1s0/manifest.json` で“更新境界”を保存し、upgrade を成立させる

テンプレ更新で最も危険なのは「どこまで上書きしてよいか」が曖昧なことです。
k1s0 は生成時に manifest を残し、次を固定します。

- 生成元テンプレ（name/version/path/fingerprint）
- 更新してよい領域（managed）
- 触らない領域（protected）
- 衝突時は止まる（安全に止まるが既定）

### 特徴C: 契約（proto/OpenAPI）を“正本”として扱い、破壊を早期に止める

- gRPC: `buf lint` / `buf breaking` を前提に置き場を固定
- REST: OpenAPI を正本にし、差分検知の導線を持つ
- 生成物は Git 管理外でも、再現性（生成一致）を CI で担保できる設計

### 特徴D: secrets を「環境変数」ではなく「ファイル参照」に固定する

- 設定 YAML に値を直書きしない（参照キー `*_file` を標準に）
- Kubernetes では Secret を volume mount、ローカルも同形に写像
- “うっかりコミット”と“環境差分”を構造的に減らす

### 特徴E: 観測性をオプションにしない（最低限を既定で揃える）

- JSON 1 行ログの必須フィールドを揃え、トレースと相関
- サービス間呼び出しは trace context を伝播
- 障害対応が「形式理解」から始まらない

### 特徴F: 通信品質を“API 形状”で担保する（deadline 必須・retry 原則禁止）

- gRPC の無制限待ちは“実装ミス”ではなく“入りにくい設計”に寄せる
- retry は既定 0、例外は明示（冪等性・回数・バックオフ等）

## 5. 技術仕様の骨子（PRで語れる「固定」）

### 5.1 ディレクトリ規約（置き場の固定）

- backend: `feature/backend/rust/{feature_name}/`
- frontend: `feature/frontend/flutter/{feature_name}/`, `feature/frontend/react/{feature_name}/`

backend（Rust）の最小構造（lint 対象）

```
src/
	application/
	domain/
	infrastructure/
	presentation/
config/
	default.yaml
	dev.yaml
	stg.yaml
	prod.yaml
deploy/
	base/
	overlays/
		dev/
		stg/
		prod/
proto/
openapi/
	openapi.yaml
```

### 5.2 設定/秘密情報（暗黙をやめる）

- 起動引数で `--env` / `--config` / `--secrets-dir` を明示（暗黙選択しない）
- YAML は非機密の参照束、機密はファイルに置く

### 5.3 upgrade（上書きではなく“境界のある更新”）

- `k1s0 upgrade --check` が影響範囲と衝突を一覧化
- `k1s0 upgrade` は managed を中心に安全に適用し、衝突は止める

---

## 6. 想定ユースケース（導入後の見え方）

- 新規サービス追加: `k1s0 new-feature` で雛形が揃い、生成直後から `k1s0 lint` が通る
- 契約変更: proto/OpenAPI の破壊が CI で止まり、意図しない破壊が入りにくい
- 秘密情報: “値を持たない運用”が既定となり、レビューが安全になる
- アップグレード: 差分が見える/止まれるので、基盤更新が継続できる

---

## 7. メッセージ例（資料/スライドに転記しやすい短文）

- 「規約はドキュメントではなく、CI が裁く」
- 「テンプレの更新は上書きではなく、境界のある更新」
- 「契約は正本、生成物は再現性で守る」
- 「secrets は環境変数ではなくファイル参照」
- 「観測性は追加オプションではなく最低限の礼儀」

---

## 8. 付録: 実装ロードマップ（要約）

実装は「MVP で“落とせる”ところまで」→「運用品質の固定を増やす」の順で進めます。

- MVP: CLI 骨格、backend テンプレ、manifest、lint（必須構造/禁止事項の最低限）、CI 必須化
- 拡張: framework の充実（config/error/observability/grpc-client）、契約管理/生成一致、upgrade 精度向上
