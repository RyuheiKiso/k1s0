### Phase 1.2: manifest.json スキーマ拡張

#### タスク 1.2.1: layer フィールドの追加

**背景:**
3層構造を manifest.json で表現するため、各コンポーネントがどの層に属するかを示す `layer` フィールドが必要です。

**作業内容:**
`CLI/schemas/manifest.schema.json` を編集し、`layer` フィールドを追加してください。このフィールドは以下の仕様とします：
- 型: string（enum）
- 許可される値: `"framework"`, `"domain"`, `"feature"`
- 必須: 新規作成時は必須、既存の manifest との互換性のため未指定の場合は `"feature"` として扱う

JSON Schema の定義例：
```json
"layer": {
  "type": "string",
  "enum": ["framework", "domain", "feature"],
  "description": "コンポーネントが属するレイヤー。未指定の場合は feature として扱われます。"
}
```

**成果物:**
- 更新された `CLI/schemas/manifest.schema.json`

**依存タスク:**
なし

**完了条件:**
- JSON Schema に `layer` フィールドの定義が存在すること
- `"framework"`, `"domain"`, `"feature"` 以外の値がバリデーションエラーになること

#### タスク 1.2.2: domain フィールドの追加

**背景:**
feature 層のコンポーネントがどの domain に属するかを示すフィールドが必要です。

**作業内容:**
`CLI/schemas/manifest.schema.json` に `domain` フィールドを追加してください。このフィールドは以下の仕様とします：
- 型: string または null
- 必須: いいえ（domain に属さない feature も存在するため）
- 用途: feature 層のコンポーネントが所属する domain 名を指定
- domain 層や framework 層では null または未指定

JSON Schema の定義例：
```json
"domain": {
  "type": ["string", "null"],
  "pattern": "^[a-z][a-z0-9]*(-[a-z0-9]+)*$",
  "description": "feature 層の場合、所属する domain 名（kebab-case）。domain 層や framework 層では null または未指定。"
}
```

**成果物:**
- 更新された `CLI/schemas/manifest.schema.json`

**依存タスク:**
- タスク 1.2.1 が完了していること

**完了条件:**
- JSON Schema に `domain` フィールドの定義が存在すること
- kebab-case 以外の値（例: `PascalCase`）がバリデーションエラーになること

#### タスク 1.2.3: version フィールドの追加

**背景:**
domain 層は独自のバージョンを持ち、SemVer で管理されます。framework の単一バージョンとは別に、各 domain が独立してバージョンを管理できるようにする必要があります。

**作業内容:**
`CLI/schemas/manifest.schema.json` に `version` フィールドを追加してください。このフィールドは以下の仕様とします：
- 型: string
- フォーマット: SemVer（例: `"1.2.0"`, `"0.1.0-beta"`）
- 必須: domain 層では必須、feature 層では任意
- 用途: domain のバージョンを記録し、feature からの依存解決に使用

JSON Schema の定義例：
```json
"version": {
  "type": "string",
  "pattern": "^(0|[1-9]\\d*)\\.(0|[1-9]\\d*)\\.(0|[1-9]\\d*)(?:-((?:0|[1-9]\\d*|\\d*[a-zA-Z-][0-9a-zA-Z-]*)(?:\\.(?:0|[1-9]\\d*|\\d*[a-zA-Z-][0-9a-zA-Z-]*))*))?(?:\\+([0-9a-zA-Z-]+(?:\\.[0-9a-zA-Z-]+)*))?$",
  "description": "domain のバージョン（SemVer 形式）"
}
```

**成果物:**
- 更新された `CLI/schemas/manifest.schema.json`

**依存タスク:**
- タスク 1.2.1 が完了していること

**完了条件:**
- JSON Schema に `version` フィールドの定義が存在すること
- SemVer 形式以外の値がバリデーションエラーになること

#### タスク 1.3.2: LayerType enum の定義

**背景:**
layer フィールドの値を型安全に扱うため、enum を定義する必要があります。

**作業内容:**

```rust
use serde::{Deserialize, Serialize};

#[derive(Debug, Clone, Copy, PartialEq, Eq, Serialize, Deserialize, Default)]
#[serde(rename_all = "lowercase")]
pub enum LayerType {
    Framework,
    Domain,
    #[default]
    Feature,
}

impl LayerType {
    pub fn as_str(&self) -> &'static str {
        match self {
            LayerType::Framework => "framework",
            LayerType::Domain => "domain",
            LayerType::Feature => "feature",
        }
    }
}
```

`#[default]` 属性により、未指定時は Feature として扱われます。

```rust
fn determine_output_path(
    template_type: &str,
    name: &str,
    output: Option<&Path>,
) -> Result<PathBuf> {
    if let Some(path) = output {
        return Ok(path.to_path_buf());
    }

    // テンプレートタイプから backend/frontend と言語を判定
    let (category, lang) = match template_type {
        "backend-rust" => ("backend", "rust"),
        "backend-go" => ("backend", "go"),
        "frontend-react" => ("frontend", "react"),
        "frontend-flutter" => ("frontend", "flutter"),
        _ => anyhow::bail!("未知のテンプレートタイプ: {}", template_type),
    };

    // domain/{category}/{lang}/{name} を返す
    Ok(PathBuf::from(format!("domain/{}/{}/{}", category, lang, name)))
}
```

fn validate_domain(template_type: &str, domain_name: &str) -> Result<PathBuf> {
    let (category, lang) = parse_template_type(template_type)?;
    let domain_path = PathBuf::from(format!("domain/{}/{}/{}", category, lang, domain_name));

    if !domain_path.exists() {
        anyhow::bail!(
            "domain '{}' が見つかりません。\n\
             期待されるパス: {}\n\
             まず 'k1s0 new-domain --type {} --name {}' で domain を作成してください",
            domain_name,
            domain_path.display(),
            template_type,
            domain_name
        );

    // domain の manifest.json も確認
    let manifest_path = domain_path.join(".k1s0/manifest.json");
    if !manifest_path.exists() {
        anyhow::bail!(
            "domain '{}' の manifest.json が見つかりません: {}",
            domain_name,
            manifest_path.display()
        );
    }

    Ok(domain_path)
}

fn get_domain_version_constraint(domain_path: &Path) -> Result<String> {
    let manifest_path = domain_path.join(".k1s0/manifest.json");
    let manifest: Manifest = serde_json::from_str(&std::fs::read_to_string(&manifest_path)?)?;

## Phase 2.5: ドメインバージョン管理

### 背景と目的

Phase 2.5 では、domain のバージョン管理機能を実装します。domain は独自のバージョン（SemVer）を持ち、feature はバージョン制約を指定して domain に依存します。これにより、domain の破壊的変更時に feature への影響を制御できます。

（タスク 2.5.1.1 〜 2.5.6.4 の詳細は、Phase 2.1 と同様の形式で記載します。主要なコマンドと機能を以下に示します：）

### 実装するコマンド

1. **`k1s0 domain version`**: domain のバージョンを bump する
   - `--name`: 対象 domain 名
   - `--bump`: major / minor / patch
   - `--breaking`: メジャーバージョンアップ時の破壊的変更の説明

2. **`k1s0 domain list`**: 全 domain の一覧を表示
   - 名前、バージョン、タイプを表示
   - `--json`: JSON 形式出力

3. **`k1s0 domain dependents`**: 特定 domain に依存する feature を一覧表示
   - `--name`: 対象 domain 名
   - 各 feature の domain_version 制約も表示

4. **`k1s0 domain impact`**: domain のバージョンアップ影響を分析
   - `--name`: 対象 domain 名
   - `--to`: 目標バージョン
   - 影響を受ける feature を一覧表示

5. **`k1s0 feature update-domain`**: feature の domain_version を更新
   - `--name`: 対象 feature 名
   - `--domain`: domain 名
   - `--version`: 新しいバージョン制約（例: `^2.0.0`）

## Phase 3: Lint ルール実装

### 背景と目的

Phase 3 では、3層構造の依存関係ルールを強制する Lint ルール（K040-K047）を実装します。これらのルールにより、誤った層間依存を自動的に検出できます。

（タスク 3.1.1 〜 3.10.8 の詳細は、各ルールの検出ロジック実装です。主要な内容を以下に示します：）

### 実装するルール

| ID   | 説明                                | 検出対象                                                |
| ---- | ----------------------------------- | ------------------------------------------------------- |
| K040 | domain → feature 依存禁止           | domain 内のコードが feature をインポート                |
| K041 | feature → 異なる domain 依存禁止    | feature が所属しない domain をインポート                |
| K042 | framework → domain/feature 依存禁止 | framework が domain または feature をインポート         |
| K043 | domain → 他 domain 依存禁止         | domain が他の domain をインポート                       |
| K044 | feature → 他 feature 依存禁止       | feature が他の feature をインポート                     |
| K045 | 存在しない domain バージョン参照    | manifest の domain_version が存在しないバージョンを指定 |
| K046 | domain バージョン制約違反           | 実際の domain バージョンが制約を満たさない              |
| K047 | 非推奨 domain バージョン依存        | deprecated な domain への依存                           |

### 各ルールの実装方針

1. **依存グラフの構築**: Cargo.toml / package.json / go.mod からインポートを抽出
2. **layer の判定**: パスから layer を判定（framework/, domain/, feature/）
3. **違反の検出**: インポート元とインポート先の layer を比較
4. **エラーメッセージ**: 具体的な解決方法を含むメッセージを生成
