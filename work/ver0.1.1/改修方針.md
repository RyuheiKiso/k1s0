## 概要

k1s0 ver0.1.1 では、現行の2層構造（framework → feature）を3層構造（framework → domain → feature）に拡張し、組織的な開発体制に対応する。

### 主要変更点

| 項目             | 変更内容                                       |
| ---------------- | ---------------------------------------------- |
| ディレクトリ構造 | `domain/` レイヤーの新設                       |
| CLI コマンド     | `k1s0 new-domain` の追加、`new-feature` の拡張 |
| manifest.json    | `layer`, `domain` フィールドの追加             |
| Lint ルール      | K040-K042（層間依存チェック）の追加            |
| テンプレート     | domain 層用テンプレートの追加                  |

## 背景・目的

### 背景

現行の k1s0 は以下の2層構造を採用している:

```
framework/  →  feature/
(共通基盤)     (個別業務機能)
```

しかし、大規模組織での運用では以下の課題がある:

1. **責任境界の曖昧さ**: framework と feature の間に「業務ドメイン共通」という概念が欠落
2. **チーム分割の困難さ**: 複数の業務チームが共通で使う機能を feature に置くか framework に置くか判断が難しい
3. **依存関係の複雑化**: feature 間で共通機能を共有しようとすると、依存関係が複雑になる

### 目的

1. **3層構造の導入**: framework → domain → feature の明確な階層化
2. **チーム責任の明確化**: 各層を担当するチームを明確に分離
3. **依存関係の制御**: 層間の依存方向を強制し、設計の一貫性を維持

### 各層の役割定義

| 層 | 役割 | 内容 | 例（製造業） |
|----|------|------|-------------|
| **framework** | システム共通の動き | 業務内容に依存しない、システム全体で共通して使用する基盤機能 | 認証・認可、ログ出力、エラーハンドリング、DB接続、設定管理、ヘルスチェック |
| **domain** | 各業務部門の共通の動き | 特定の業務領域（ドメイン）内で複数の機能が共通して使用するビジネスロジック | 生産管理ドメイン：BOM（部品表）、工程マスタ、生産能力計算ロジック |
| **feature** | 特定業務でのみの処理 | 個別の業務機能に閉じた処理。他の機能からは参照されない | 製造指示発行、作業日報入力、検査成績書出力 |

### 層の判断基準

コードをどの層に配置するか迷った場合の判断フロー:

```
Q1: その機能は業務内容に依存しますか？
    │
    ├─ No → framework（技術基盤）
    │        例: ログ出力、認証、DB接続
    │
    └─ Yes
        │
        Q2: 同じ業務領域の複数機能で使いますか？
            │
            ├─ Yes → domain（業務部門共通）
            │        例: 生産管理の「BOM（部品表）」は製造指示でも作業日報でも使う
            │
            └─ No → feature（特定業務専用）
                     例: 製造指示書のPDF生成は製造指示機能だけで使う
```

### 想定するチーム構成

| チーム                 | 担当レイヤー | 責務                         |
| ---------------------- | ------------ | ---------------------------- |
| 開発基盤チーム         | framework/   | システム共通基盤の開発・保守 |
| ドメインアドオンチーム | domain/      | 業務ドメイン共通機能の開発   |
| 特定業務アドオンチーム | feature/     | 個別業務機能の開発           |

| feature   | 所属 domain    | ✓ 許可   | 自身が属するドメインの機能を使用可   |
| feature   | 他の domain    | ✗ 禁止   | ドメイン境界を超えた依存を防止       |
| feature   | 他の feature   | ✗ 禁止   | feature 間の密結合を防止             |

### 5.1 CLI コマンドの変更

#### 新規コマンド: `k1s0 new-domain`

```bash
k1s0 new-domain [OPTIONS] --type <TYPE> --name <NAME>
```

**オプション:**

| オプション       | 説明                     | 必須 | デフォルト             |
| ---------------- | ------------------------ | ---- | ---------------------- |
| `--type <TYPE>`  | テンプレートタイプ       | ✓   | -                      |
| `--name <NAME>`  | ドメイン名（kebab-case） | ✓   | -                      |
| `--output <DIR>` | 出力先ディレクトリ       | -    | `domain/{type}/{name}` |
| `--force`        | 既存ディレクトリを上書き | -    | false                  |

**使用例:**

```bash
# production（生産管理）ドメインを作成（Rust バックエンド）
k1s0 new-domain --type backend-rust --name production

# inventory（在庫管理）ドメインを作成（React フロントエンド）
k1s0 new-domain --type frontend-react --name inventory
```

**出力構造（backend-rust の場合）:**

```
domain/backend/rust/production/
├── .k1s0/
│   └── manifest.json
├── Cargo.toml
├── src/
│   ├── lib.rs                    # クレートエントリポイント
│   ├── domain/
│   │   ├── mod.rs
│   │   ├── entities/             # ドメインエンティティ（BOM、工程等）
│   │   ├── value_objects/        # 値オブジェクト（ロット番号等）
│   │   └── services/             # ドメインサービス（生産能力計算等）
│   ├── application/
│   │   ├── mod.rs
│   │   └── services/             # アプリケーションサービス
│   └── infrastructure/
│       ├── mod.rs
│       └── repositories/         # リポジトリ実装
└── tests/
```

#### 既存コマンド拡張: `k1s0 new-feature`

```bash
k1s0 new-feature [OPTIONS] --type <TYPE> --name <NAME>
```

**追加オプション:**

| オプション        | 説明           | 必須 | デフォルト                   |
| ----------------- | -------------- | ---- | ---------------------------- |
| `--domain <NAME>` | 所属ドメイン名 | -    | なし（ドメインなし feature） |

**使用例:**

```bash
# production ドメインに属する work-order（製造指示）を作成

```

**ドメイン指定時の動作:**

1. 指定された domain が存在するか検証
2. manifest.json に `domain` フィールドを設定
3. Cargo.toml / package.json に domain クレート/パッケージへの依存を追加

**追加ルール:**

| ルール | 説明                                             |
| ------ | ------------------------------------------------ |
| K040   | domain 層から feature 層への依存を検出           |
| K041   | feature 層から異なる domain 層への依存を検出     |
| K042   | framework 層から domain/feature 層への依存を検出 |
| K043   | domain 層から他の domain 層への依存を検出        |
| K044   | feature 層から他の feature 層への依存を検出      |

### 5.2 ドメインバージョン管理

#### バージョン管理の方針

| レイヤー  | バージョン管理方式                 | 説明                   |
| --------- | ---------------------------------- | ---------------------- |
| framework | 単一バージョン（k1s0-version.txt） | リポジトリ全体で統一   |
| domain    | **独自バージョン（SemVer）**       | ドメインごとに独立管理 |
| feature   | 独自バージョン（SemVer）           | feature ごとに独立管理 |

#### なぜ domain は独自バージョンか

1. **チームの独立性**: ドメインアドオンチームが自律的にリリースサイクルを管理できる
2. **破壊的変更の制御**: domain の breaking change が全 feature に即座に影響しない
3. **段階的移行**: feature チームが自分のタイミングで新バージョンに移行できる

#### バージョン管理の仕組み

```
domain/backend/rust/production/
├── .k1s0/
│   └── manifest.json          # layer: "domain", version: "1.2.0"
├── Cargo.toml                 # version = "1.2.0"
├── CHANGELOG.md               # バージョン履歴
└── src/
```

**manifest.json（domain）:**

```json
{
  "name": "production",
  "type": "backend-rust",
  "layer": "domain",
  "version": "1.2.0",
  "min_framework_version": "0.1.0",
  "breaking_changes": {
    "2.0.0": "EntityA のフィールド名変更",
    "1.0.0": "初回リリース"
  }
}
```

**Cargo.toml（domain）:**

```toml
[package]
name = "production"
version = "1.2.0"
edition = "2024"
```

#### feature から domain への依存指定

**manifest.json（feature）:**

```json
{
  "name": "work-order",
  "type": "backend-rust",
  "layer": "feature",
  "domain": "production",
  "domain_version": "^1.2.0",
  "dependencies": {
    "framework": ["k1s0-error", "k1s0-config"],
    "domain": {
      "production": "^1.2.0"
    }
  }
}
```

**Cargo.toml（feature）:**

```toml
[dependencies]
# domain 依存（パス指定 + バージョン制約）
production = { path = "../../../../domain/backend/rust/production", version = "^1.2.0" }
```

#### バージョン互換性ルール

| バージョン変更        | 互換性     | feature への影響               |
| --------------------- | ---------- | ------------------------------ |
| patch (1.2.0 → 1.2.1) | 後方互換   | 自動適用可                     |
| minor (1.2.0 → 1.3.0) | 後方互換   | 自動適用可（新機能追加のみ）   |
| major (1.2.0 → 2.0.0) | **破壊的** | feature 側の明示的な更新が必要 |

#### CLI コマンドの拡張

```bash
# domain のバージョンを上げる
k1s0 domain version --name production --bump minor
# => production 1.2.0 → 1.3.0

# 破壊的変更を伴うメジャーバージョンアップ
k1s0 domain version --name production --bump major --breaking "EntityA のフィールド名変更"
# => production 1.3.0 → 2.0.0
# => CHANGELOG.md に自動追記

# feature が使用する domain バージョンを更新
k1s0 feature update-domain --name work-order --domain production --version "^2.0.0"
```

#### 依存関係の検証（Lint）

| ルール | 説明                                                |
| ------ | --------------------------------------------------- |
| K045   | domain_version が存在しない domain バージョンを参照 |
| K046   | domain_version の制約を満たさないバージョンを使用   |
| K047   | 非推奨（deprecated）の domain バージョンに依存      |

**K046 エラー例:**

```
[K046] error: domain バージョンの制約違反
  対象: feature/backend/rust/work-order/.k1s0/manifest.json
  制約: production ^1.2.0
  実際: production 2.0.0
  ヒント: manifest.json の domain_version を "^2.0.0" に更新してください
```

#### バージョン確認コマンド

```bash
# domain のバージョン一覧
k1s0 domain list
# => production      1.2.0  (backend-rust)
# => inventory    0.5.0  (backend-rust)
# => quality      1.0.0  (frontend-react)

# 特定 domain の依存元を表示
k1s0 domain dependents --name production
# => work-order  (^1.2.0)
# => daily-report     (^1.0.0)  ← 古いバージョン制約

# domain の更新影響を確認
k1s0 domain impact --name production --to 2.0.0
# => 破壊的変更の影響を受ける feature:
# =>   - work-order (現在: ^1.2.0)
# =>   - daily-report (現在: ^1.0.0)
```

#### ワークフロー例

```
1. ドメインアドオンチームが production を開発
   └─ production 1.0.0 リリース

2. 特定業務チームが work-order を作成
   └─ k1s0 new-feature --type backend-rust --name work-order --domain production
   └─ 自動的に domain_version: "^1.0.0" が設定される

3. ドメインアドオンチームが production に機能追加
   └─ k1s0 domain version --name production --bump minor
   └─ production 1.1.0 リリース
   └─ work-order は ^1.0.0 なので自動的に 1.1.0 を使用

4. ドメインアドオンチームが production に破壊的変更
   └─ k1s0 domain version --name production --bump major --breaking "API 変更"
   └─ production 2.0.0 リリース
   └─ work-order は ^1.0.0 なので 1.x 系を継続使用
   └─ 特定業務チームが準備できたら domain_version を ^2.0.0 に更新
```

### 5.3 manifest.json スキーマ拡張

#### 現行スキーマ（feature）

```json
{
  "name": "order-service",
  "type": "backend-rust",
  "template_version": "1.0.0",
  "fingerprint": "a1b2c3d4...",
  "created_at": "2026-01-28T10:00:00Z"
}
```

#### 拡張スキーマ

```json
{
  "$schema": "https://k1s0.dev/schemas/manifest.json",
  "name": "work-order",
  "type": "backend-rust",
  "layer": "feature",
  "domain": "production",
  "template_version": "1.0.0",
  "fingerprint": "a1b2c3d4...",
  "created_at": "2026-01-28T10:00:00Z",
  "dependencies": {
    "framework": ["k1s0-error", "k1s0-config", "k1s0-grpc-server"],
    "domain": ["production"]
  }
}
```

**新規フィールド:**

| フィールド               | 型                                     | 説明                                   | 必須             |
| ------------------------ | -------------------------------------- | -------------------------------------- | ---------------- |
| `layer`                  | `"framework" \| "domain" \| "feature"` | 所属レイヤー                           | ✓（新規作成時） |
| `domain`                 | `string \| null`                       | 所属ドメイン名（feature 層のみ）       | -                |
| `dependencies`           | `object`                               | 明示的な依存関係                       | -                |
| `dependencies.framework` | `string[]`                             | 使用する framework クレート/パッケージ | -                |
| `dependencies.domain`    | `string[]`                             | 使用する domain（feature 層のみ）      | -                |

## マイグレーション方針

### 既存プロジェクトへの影響

1. **後方互換性の維持**
   - `layer` フィールドが未指定の場合は `feature` として扱う
   - `domain` フィールドが未指定の場合は「ドメインなし」として扱う
   - 既存の feature は変更なしで動作

2. **段階的な移行**
   - 新規作成時のみ layer 指定を推奨
   - 既存 feature の domain への移行は任意

### マイグレーション手順

#### 1. 既存 feature を domain に昇格させる場合

```bash
# 1. domain ディレクトリを作成
mkdir -p domain/backend/rust/production

# 2. feature から domain にコードを移動
mv feature/backend/rust/production-common/* domain/backend/rust/production/

# 3. manifest.json を更新
# layer: "domain" に変更
# domain: null に変更

# 4. 依存元の feature を更新
# Cargo.toml のパスを修正
# manifest.json の dependencies.domain を追加
```

#### 2. 新規 feature を既存 domain に紐付ける場合

```bash
# --domain オプションを指定して作成
k1s0 new-feature --type backend-rust --name work-order --domain production
```

| 課題                      | 詳細                                                                                           | 深刻度 | 解決策                                                                                                    |

| **domain 境界の決定**     | 何を domain にするか、どこで境界を引くかの判断基準が不明確。production と procurement（購買）は別？同じ？ | 高     | 「Domain Boundary Guidelines」を策定。ビジネスケイパビリティマップを作成し、組織の業務領域と1:1対応させる |
| **チーム間の調整コスト**  | domain チームと feature チームの優先度が異なる場合、domain の機能追加が遅延する可能性          | 高     | 週次「Domain Sync Meeting」開催。feature チームが domain に PR を出せる「Contribution Model」を導入       |
| **責任の押し付け合い**    | 「これは domain に入れるべき」「いや feature で十分」という議論が発生                          | 中     | 判断フローチャート作成：「2つ以上の feature で使う？」→ Yes → domain                                      |
| **domain チームの過負荷** | 複数 feature チームからの要望が集中し、domain チームがボトルネック化                           | 高     | Inner Source モデル採用。feature チームが domain への PR を作成し、domain チームはレビューに専念          |
| **スキルセットの分断**    | domain 開発者と feature 開発者で必要なスキルが異なり、人材流動性が低下                         | 中     | ローテーション制度導入。四半期ごとに domain ↔ feature 間で人員交流                                        |

### 2. 技術運用

| 課題                       | 詳細                                                                     | 深刻度 | 解決策                                                                                     |
| -------------------------- | ------------------------------------------------------------------------ | ------ | ------------------------------------------------------------------------------------------ |
| **循環依存の検出限界**     | Lint は直接依存を検出できるが、間接的な循環（A→B→C→A）の検出が困難       | 中     | `k1s0 lint --deep` オプションで依存グラフ全体を解析。CI で定期実行                         |
| **ビルド時間の増大**       | 3層構造により依存関係が深くなり、フルビルド時間が増加                    | 中     | Cargo workspace の共有ターゲット活用。sccache / mold linker 導入。増分ビルド最適化         |
| **パス解決の複雑化**       | `../../../../domain/backend/rust/production` のような深いパスが可読性を低下 | 低     | workspace の `[patch]` セクションでエイリアス定義。`k1s0-domain-production` のような名前で参照可能に |
| **IDE サポートの不完全性** | 層をまたいだコード補完・定義ジャンプが正しく動作しない可能性             | 中     | rust-analyzer / tsserver の workspace 設定ガイド提供。推奨 IDE 設定を `.vscode/` に同梱    |
| **テスト戦略の複雑化**     | domain のテストと feature のテストの境界、モック戦略の標準化が必要       | 中     | テストピラミッドガイドライン策定。domain は単体テスト中心、feature は統合テスト中心        |

### 3. バージョン管理・リリース運用

| 課題                             | 詳細                                                                       | 深刻度 | 解決策                                                                                              |
| -------------------------------- | -------------------------------------------------------------------------- | ------ | --------------------------------------------------------------------------------------------------- |
| **バージョン同期の困難さ**       | manifest.json の `domain_version` と Cargo.toml の `version` の二重管理    | 高     | `k1s0 sync-versions` コマンド実装。CI で自動検証し、不整合があれば PR をブロック                    |
| **破壊的変更の伝播遅延**         | domain 2.0.0 リリース後も feature が 1.x を使い続け、技術的負債が蓄積      | 高     | 「Migration Deadline Policy」導入。major リリースから6ヶ月で旧バージョンを deprecated、12ヶ月で削除 |
| **複数 domain バージョンの共存** | 同じ domain の複数バージョンがビルドに含まれる可能性（diamond dependency） | 高     | workspace レベルで単一バージョンを強制。`k1s0 lint` で複数バージョン検出時に警告                    |
| **リリース順序の制約**           | domain をリリースしてから feature をリリースする順序強制が CI/CD を複雑化  | 中     | 依存グラフに基づく自動リリース順序決定。`k1s0 release-plan` コマンドで順序を可視化                  |
| **ロールバックの複雑化**         | feature のロールバックが domain バージョンとの整合性を壊す可能性           | 高     | ロールバック前に `k1s0 rollback-check` で整合性検証。旧 domain バージョンが存在することを確認       |
| **CHANGELOG の分散**             | 各 domain/feature に CHANGELOG があり、全体の変更履歴が追いにくい          | 低     | `k1s0 changelog --aggregate` で全層の CHANGELOG を統合表示。リリースノート自動生成                  |

### 4. 開発者体験（DX）

| 課題                       | 詳細                                                                         | 深刻度 | 解決策                                                                            |
| -------------------------- | ---------------------------------------------------------------------------- | ------ | --------------------------------------------------------------------------------- |
| **学習コストの増大**       | 2層から3層への概念の増加、新人のオンボーディング負荷                         | 中     | 「k1s0 Tutorial」インタラクティブガイド作成。30分で3層構造を理解できるハンズオン  |
| **ローカル開発の複雑化**   | domain を変更したい feature 開発者が domain もクローン・ビルドする必要       | 中     | `k1s0 dev-setup --feature work-order` で必要な domain を自動セットアップ  |
| **デバッグの困難さ**       | 問題が domain 起因か feature 起因か切り分けが難しい                          | 中     | エラーにスタックトレースと層情報を付与。`[domain:production]` のようなプレフィックス |
| **コマンドの増加**         | `new-domain`, `domain version`, `domain list` 等のコマンド増加による認知負荷 | 低     | `k1s0` のみ実行で対話型メニュー表示。よく使うコマンドを学習して提案               |
| **エラーメッセージの理解** | K040-K047 のエラーが初見では意味が分かりにくい可能性                         | 低     | エラーに「詳細は `k1s0 explain K041` を参照」リンク追加。具体例付きの解説を表示   |

### 5. ガバナンス・ルール運用

| 課題                          | 詳細                                                                   | 深刻度 | 解決策                                                                                        |
| ----------------------------- | ---------------------------------------------------------------------- | ------ | --------------------------------------------------------------------------------------------- |
| **例外処理の扱い**            | 「一時的に K041 を無視したい」といった例外申請のワークフロー未定義     | 中     | `.k1s0/exceptions.json` で有効期限付き除外を管理。期限切れは CI で自動検出                    |
| **domain 昇格/降格の基準**    | feature を domain に昇格させる基準、逆に domain を廃止する基準が不明確 | 中     | 基準明文化：「3つ以上の feature から参照」で昇格検討、「6ヶ月間新規参照なし」で降格検討       |
| **レビュープロセス**          | domain の変更は誰がレビューするか、feature チームの承認は必要か        | 高     | CODEOWNERS で domain ごとにレビュアー指定。破壊的変更は依存 feature チームに通知必須          |
| **非推奨（deprecation）運用** | domain バージョンの非推奨化から削除までのタイムライン未定義            | 中     | Deprecation Policy：通知 → 3ヶ月警告期間 → 6ヶ月後に削除。`k1s0 deprecate` コマンドで管理     |
| **Lint 除外の乱用**           | `// k1s0-ignore: K041` のようなコメントが乱用される可能性              | 中     | ignore コメントには理由と Issue 番号を必須化。`k1s0 lint --report-ignores` で除外一覧を可視化 |

### 6. スケーラビリティ

| 課題                 | 詳細                                                                      | 深刻度 | 解決策                                                                                 |
| -------------------- | ------------------------------------------------------------------------- | ------ | -------------------------------------------------------------------------------------- |
| **domain 数の爆発**  | 細分化しすぎて domain が数十個に増え、管理が困難に                        | 中     | domain 作成は Architecture Review Board の承認制。上限目安を設定（例：15 domain まで） |
| **feature 数の増加** | 1つの domain に数十の feature が紐づき、依存関係が複雑化                  | 中     | 1 domain あたり feature 上限を推奨（例：20 feature）。超過時は domain 分割を検討       |
| **モノレポの肥大化** | framework + domain + feature の全てが1リポジトリで、クローン/ビルドが重い | 高     | Git sparse checkout ガイド提供。将来的に domain 単位での polyrepo 分割オプションを検討 |
| **CI/CD の実行時間** | 全層のテスト実行で CI が数時間かかる可能性                                | 高     | `k1s0 affected` で変更影響範囲を特定。影響のある domain/feature のみテスト実行         |

### 7. マイグレーション運用

| 課題                                | 詳細                                                       | 深刻度 | 解決策                                                                                      |
| ----------------------------------- | ---------------------------------------------------------- | ------ | ------------------------------------------------------------------------------------------- |
| **既存 feature の domain 割り当て** | 既存の feature をどの domain に紐付けるか判断が難しい      | 高     | 「Domain Mapping Workshop」開催。ビジネス部門と共同で業務領域マップを作成し、feature を分類 |
| **段階的移行の長期化**              | 「domain なし」の feature が残り続け、新旧混在が長期化     | 中     | マイグレーション期限設定（例：6ヶ月）。期限後は `k1s0 lint` で警告 → エラーに昇格           |
| **移行中のブランチ管理**            | 移行作業中に通常開発が並行すると、マージコンフリクトが多発 | 中     | 移行専用 `migration/*` ブランチ運用。短期集中（1-2週間）で一括移行を推奨                    |
| **ドキュメントの陳腐化**            | 移行期間中にドキュメントが現実と乖離しやすい               | 低     | `k1s0 docs-check` でドキュメント内のパス・コマンド例を自動検証。CI で毎日実行               |
