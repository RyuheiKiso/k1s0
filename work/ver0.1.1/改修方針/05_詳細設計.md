## 詳細設計

### 5.1 CLI コマンドの変更

#### 新規コマンド: `k1s0 new-domain`

```bash
k1s0 new-domain [OPTIONS] --type <TYPE> --name <NAME>
```

**オプション:**

| オプション       | 説明                     | 必須 | デフォルト             |
| ---------------- | ------------------------ | ---- | ---------------------- |
| `--type <TYPE>`  | テンプレートタイプ       | ✓   | -                      |
| `--name <NAME>`  | ドメイン名（kebab-case） | ✓   | -                      |
| `--output <DIR>` | 出力先ディレクトリ       | -    | `domain/{type}/{name}` |
| `--force`        | 既存ディレクトリを上書き | -    | false                  |

**使用例:**

```bash
# production（生産管理）ドメインを作成（Rust バックエンド）
k1s0 new-domain --type backend-rust --name production

# inventory（在庫管理）ドメインを作成（React フロントエンド）
k1s0 new-domain --type frontend-react --name inventory
```

**出力構造（backend-rust の場合）:**

```
domain/backend/rust/production/
├── .k1s0/
│   └── manifest.json
├── Cargo.toml
├── src/
│   ├── lib.rs                    # クレートエントリポイント
│   ├── domain/
│   │   ├── mod.rs
│   │   ├── entities/             # ドメインエンティティ（BOM、工程等）
│   │   ├── value_objects/        # 値オブジェクト（ロット番号等）
│   │   └── services/             # ドメインサービス（生産能力計算等）
│   ├── application/
│   │   ├── mod.rs
│   │   └── services/             # アプリケーションサービス
│   └── infrastructure/
│       ├── mod.rs
│       └── repositories/         # リポジトリ実装
└── tests/
```

#### 既存コマンド拡張: `k1s0 new-feature`

```bash
k1s0 new-feature [OPTIONS] --type <TYPE> --name <NAME>
```

**追加オプション:**

| オプション        | 説明           | 必須 | デフォルト                   |
| ----------------- | -------------- | ---- | ---------------------------- |
| `--domain <NAME>` | 所属ドメイン名 | -    | なし（ドメインなし feature） |

**使用例:**

```bash
# production ドメインに属する work-order（製造指示）を作成
k1s0 new-feature --type backend-rust --name work-order --domain production

# ドメインに属さない独立 feature を作成（従来互換）
k1s0 new-feature --type backend-rust --name standalone-service
```

**ドメイン指定時の動作:**

1. 指定された domain が存在するか検証
2. manifest.json に `domain` フィールドを設定
3. Cargo.toml / package.json に domain クレート/パッケージへの依存を追加

#### コマンド: `k1s0 lint` の拡張

```bash
k1s0 lint [OPTIONS] [PATH]
```

**追加ルール:**

| ルール | 説明                                             |
| ------ | ------------------------------------------------ |
| K040   | domain 層から feature 層への依存を検出           |
| K041   | feature 層から異なる domain 層への依存を検出     |
| K042   | framework 層から domain/feature 層への依存を検出 |
| K043   | domain 層から他の domain 層への依存を検出        |
| K044   | feature 層から他の feature 層への依存を検出      |

### 5.2 ドメインバージョン管理

#### バージョン管理の方針

| レイヤー  | バージョン管理方式                 | 説明                   |
| --------- | ---------------------------------- | ---------------------- |
| framework | 単一バージョン（k1s0-version.txt） | リポジトリ全体で統一   |
| domain    | **独自バージョン（SemVer）**       | ドメインごとに独立管理 |
| feature   | 独自バージョン（SemVer）           | feature ごとに独立管理 |

#### なぜ domain は独自バージョンか

1. **チームの独立性**: ドメインアドオンチームが自律的にリリースサイクルを管理できる
2. **破壊的変更の制御**: domain の breaking change が全 feature に即座に影響しない
3. **段階的移行**: feature チームが自分のタイミングで新バージョンに移行できる

#### バージョン管理の仕組み

```
domain/backend/rust/production/
├── .k1s0/
│   └── manifest.json          # layer: "domain", version: "1.2.0"
├── Cargo.toml                 # version = "1.2.0"
├── CHANGELOG.md               # バージョン履歴
└── src/
```

**manifest.json（domain）:**

```json
{
  "name": "production",
  "type": "backend-rust",
  "layer": "domain",
  "version": "1.2.0",
  "min_framework_version": "0.1.0",
  "breaking_changes": {
    "2.0.0": "EntityA のフィールド名変更",
    "1.0.0": "初回リリース"
  }
}
```

**Cargo.toml（domain）:**

```toml
[package]
name = "production"
version = "1.2.0"
edition = "2024"
```

#### feature から domain への依存指定

**manifest.json（feature）:**

```json
{
  "name": "work-order",
  "type": "backend-rust",
  "layer": "feature",
  "domain": "production",
  "domain_version": "^1.2.0",
  "dependencies": {
    "framework": ["k1s0-error", "k1s0-config"],
    "domain": {
      "production": "^1.2.0"
    }
  }
}
```

**Cargo.toml（feature）:**

```toml
[dependencies]
# domain 依存（パス指定 + バージョン制約）
production = { path = "../../../../domain/backend/rust/production", version = "^1.2.0" }
```

#### バージョン互換性ルール

| バージョン変更        | 互換性     | feature への影響               |
| --------------------- | ---------- | ------------------------------ |
| patch (1.2.0 → 1.2.1) | 後方互換   | 自動適用可                     |
| minor (1.2.0 → 1.3.0) | 後方互換   | 自動適用可（新機能追加のみ）   |
| major (1.2.0 → 2.0.0) | **破壊的** | feature 側の明示的な更新が必要 |

#### CLI コマンドの拡張

```bash
# domain のバージョンを上げる
k1s0 domain version --name production --bump minor
# => production 1.2.0 → 1.3.0

# 破壊的変更を伴うメジャーバージョンアップ
k1s0 domain version --name production --bump major --breaking "EntityA のフィールド名変更"
# => production 1.3.0 → 2.0.0
# => CHANGELOG.md に自動追記

# feature が使用する domain バージョンを更新
k1s0 feature update-domain --name work-order --domain production --version "^2.0.0"
```

#### 依存関係の検証（Lint）

| ルール | 説明                                                |
| ------ | --------------------------------------------------- |
| K045   | domain_version が存在しない domain バージョンを参照 |
| K046   | domain_version の制約を満たさないバージョンを使用   |
| K047   | 非推奨（deprecated）の domain バージョンに依存      |

**K046 エラー例:**

```
[K046] error: domain バージョンの制約違反
  対象: feature/backend/rust/work-order/.k1s0/manifest.json
  制約: production ^1.2.0
  実際: production 2.0.0
  ヒント: manifest.json の domain_version を "^2.0.0" に更新してください
```

#### バージョン確認コマンド

```bash
# domain のバージョン一覧
k1s0 domain list
# => production      1.2.0  (backend-rust)
# => inventory    0.5.0  (backend-rust)
# => quality      1.0.0  (frontend-react)

# 特定 domain の依存元を表示
k1s0 domain dependents --name production
# => work-order  (^1.2.0)
# => daily-report     (^1.0.0)  ← 古いバージョン制約

# domain の更新影響を確認
k1s0 domain impact --name production --to 2.0.0
# => 破壊的変更の影響を受ける feature:
# =>   - work-order (現在: ^1.2.0)
# =>   - daily-report (現在: ^1.0.0)
```

#### ワークフロー例

```
1. ドメインアドオンチームが production を開発
   └─ production 1.0.0 リリース

2. 特定業務チームが work-order を作成
   └─ k1s0 new-feature --type backend-rust --name work-order --domain production
   └─ 自動的に domain_version: "^1.0.0" が設定される

3. ドメインアドオンチームが production に機能追加
   └─ k1s0 domain version --name production --bump minor
   └─ production 1.1.0 リリース
   └─ work-order は ^1.0.0 なので自動的に 1.1.0 を使用

4. ドメインアドオンチームが production に破壊的変更
   └─ k1s0 domain version --name production --bump major --breaking "API 変更"
   └─ production 2.0.0 リリース
   └─ work-order は ^1.0.0 なので 1.x 系を継続使用
   └─ 特定業務チームが準備できたら domain_version を ^2.0.0 に更新
```

### 5.3 manifest.json スキーマ拡張

#### 現行スキーマ（feature）

```json
{
  "name": "order-service",
  "type": "backend-rust",
  "template_version": "1.0.0",
  "fingerprint": "a1b2c3d4...",
  "created_at": "2026-01-28T10:00:00Z"
}
```

#### 拡張スキーマ

```json
{
  "$schema": "https://k1s0.dev/schemas/manifest.json",
  "name": "work-order",
  "type": "backend-rust",
  "layer": "feature",
  "domain": "production",
  "template_version": "1.0.0",
  "fingerprint": "a1b2c3d4...",
  "created_at": "2026-01-28T10:00:00Z",
  "dependencies": {
    "framework": ["k1s0-error", "k1s0-config", "k1s0-grpc-server"],
    "domain": ["production"]
  }
}
```

**新規フィールド:**

| フィールド               | 型                                     | 説明                                   | 必須             |
| ------------------------ | -------------------------------------- | -------------------------------------- | ---------------- |
| `layer`                  | `"framework" \| "domain" \| "feature"` | 所属レイヤー                           | ✓（新規作成時） |
| `domain`                 | `string \| null`                       | 所属ドメイン名（feature 層のみ）       | -                |
| `dependencies`           | `object`                               | 明示的な依存関係                       | -                |
| `dependencies.framework` | `string[]`                             | 使用する framework クレート/パッケージ | -                |
| `dependencies.domain`    | `string[]`                             | 使用する domain（feature 層のみ）      | -                |

### 5.4 Lint ルール詳細設計

#### K040: domain → feature 依存禁止

**検出対象:**

```rust
// domain/backend/rust/production/src/lib.rs
use work_order::something;  // NG: feature への依存
```

```typescript
// domain/frontend/react/production/src/index.ts
import { something } from '@feature/work-order';  // NG
```

**エラーメッセージ:**

```
[K040] error: domain 層から feature 層への依存が検出されました
  対象: domain/backend/rust/production/src/lib.rs:5
  依存先: work-order (feature)
  ヒント: domain 層は feature 層に依存できません。
         共通機能は domain 層に移動するか、インターフェースを使用してください。
```

#### K041: feature → 異なる domain 依存禁止

**検出対象:**

```rust
// feature/backend/rust/work-order/src/main.rs
// manifest.json: { "domain": "production" }
use inventory::tracking;  // NG: 異なる domain への依存
```

**エラーメッセージ:**

```
[K041] error: feature 層から異なる domain 層への依存が検出されました
  対象: feature/backend/rust/work-order/src/main.rs:3
  所属 domain: production
  依存先 domain: inventory
  ヒント: feature は自身が属する domain にのみ依存できます。
         異なる domain の機能が必要な場合は、BFF 層または共通 API を検討してください。
```

#### K042: framework → domain/feature 依存禁止

**検出対象:**

```rust
// framework/backend/rust/crates/k1s0-config/src/lib.rs
use production::settings;  // NG: domain への依存
```

**エラーメッセージ:**

```
[K042] error: framework 層から domain/feature 層への依存が検出されました
  対象: framework/backend/rust/crates/k1s0-config/src/lib.rs:10
  依存先: production (domain)
  ヒント: framework 層は業務ロジックに依存できません。
         必要な機能はインターフェースとして定義し、実装を注入してください。
```

#### K043: domain → 他の domain 依存禁止

**検出対象:**

```rust
// domain/backend/rust/production/src/lib.rs
use inventory::common;  // NG: 他の domain への依存
```

**エラーメッセージ:**

```
[K043] error: domain 層から他の domain 層への依存が検出されました
  対象: domain/backend/rust/production/src/lib.rs:3
  依存元: production
  依存先: inventory
  ヒント: domain 間の直接依存は禁止されています。
         共通機能は framework 層に移動してください。
```

#### K044: feature → 他の feature 依存禁止

**検出対象:**

```rust
// feature/backend/rust/work-order/src/main.rs
use daily_report::client;  // NG: 他の feature への依存
```

**エラーメッセージ:**

```
[K044] error: feature 層から他の feature 層への依存が検出されました
  対象: feature/backend/rust/work-order/src/main.rs:5
  依存元: work-order
  依存先: daily-report
  ヒント: feature 間の直接依存は禁止されています。
         サービス間連携は gRPC/REST API を使用してください。
```
