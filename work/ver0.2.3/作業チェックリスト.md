# k1s0 v0.2.3 作業チェックリスト

## 概要

Docker と docker-compose を k1s0 プラットフォームにファーストクラスサポートとして採用する。

- **Dockerfile テンプレート全言語対応**: Rust / Go に Dockerfile.tera 新設、C# / Python 改善
- **フロントエンド Dockerfile**: React / Flutter テンプレートに Dockerfile.tera 新設
- **docker-compose.yaml テンプレート**: feature 生成時に compose.yaml を自動生成
- **`k1s0 docker` サブコマンド実装**: up / down / logs / build / ps + --tag / --push / --platform
- **docker.yml CI ワークフロー**: hadolint + ビルド + Trivy + マルチアーキテクチャ
- **ルートレベル compose.yaml**: `k1s0 init` 時に共通インフラサービス定義を生成
- **manifest.json / manifest.schema.json 更新**: Docker メタデータ + with_cache 対応
- **lint ルール K010/K011 更新**: Dockerfile を必須ファイルに追加
- **`k1s0 upgrade` 移行対応**: 既存サービスへの Docker ファイル追加
- **コンテナレジストリ / 命名規則 / マルチアーキテクチャ**: --tag / --push / --platform
- **`--with-docker` / `--no-docker` フラグ**: Docker ファイル生成のオプトアウト

## 受け入れ基準

### 機能要件（Dockerfile テンプレート: バックエンド）

- [ ] `k1s0 new-feature --type backend-rust --name X` で Dockerfile + .dockerignore が生成される
- [ ] `k1s0 new-feature --type backend-go --name X` で Dockerfile + .dockerignore が生成される
- [ ] 全 4 言語の Dockerfile が Multi-stage build を採用している
- [ ] 全 4 言語の Dockerfile が non-root ユーザーで実行される
- [ ] `--with-grpc` で gRPC ポート（50051）が EXPOSE される
- [ ] hadolint で全 Dockerfile が警告なし（.hadolint.yaml 適用）

### 機能要件（Dockerfile テンプレート: フロントエンド）

- [ ] `k1s0 new-feature --type frontend-react --name X` で Dockerfile + .dockerignore が生成される
- [ ] `k1s0 new-feature --type frontend-flutter --name X` で Dockerfile + .dockerignore が生成される
- [ ] React / Flutter 共に Multi-stage build（ビルド → nginx 配信）
- [ ] /healthz エンドポイントで HEALTHCHECK 動作

### 機能要件（compose.yaml テンプレート）

- [ ] `k1s0 new-feature --type backend-rust --name X --with-db` で compose.yaml に postgres が含まれる
- [ ] `k1s0 new-feature --type backend-rust --name X --with-cache` で compose.yaml に redis が含まれる
- [ ] 全依存サービスにヘルスチェックが設定されている
- [ ] `docker compose config` でバリデーション可能
- [ ] otel-collector は profiles: observability で分離されている

### 機能要件（k1s0 docker サブコマンド）

- [ ] `k1s0 docker up` でサービスが起動する
- [ ] `k1s0 docker down` でサービスが停止する
- [ ] `k1s0 docker logs` でログが表示される
- [ ] `k1s0 docker build` でイメージがビルドされる
- [ ] `k1s0 docker build --tag v0.1.0` で命名規則準拠のタグ付けができる
- [ ] `k1s0 docker build --push` でレジストリへ Push できる
- [ ] `k1s0 docker build --platform linux/amd64,linux/arm64` でマルチアーキテクチャビルドができる
- [ ] `k1s0 docker ps` でステータスが表示される
- [ ] Docker 未インストール時にリカバリ提案が表示される
- [ ] compose.yaml 未検出時にリカバリ提案が表示される
- [ ] `--json` モードで構造化出力（ps コマンド）

### 機能要件（--with-docker / --no-docker）

- [ ] `k1s0 new-feature --no-docker` で Docker ファイルが生成されない
- [ ] デフォルト（`--with-docker`）では Docker ファイルが生成される

### 機能要件（init 統合）

- [ ] `k1s0 init` でルート compose.yaml が生成される
- [ ] `k1s0 init` で .hadolint.yaml が生成される
- [ ] `k1s0 init` でルート .dockerignore が生成される
- [ ] 既存ファイルがある場合はスキップ（上書き防止）

### 機能要件（manifest / lint）

- [ ] manifest.json の managed_paths に Docker ファイルが含まれる
- [ ] manifest.schema.json に with_cache プロパティが定義されている
- [ ] `k1s0 lint` で Dockerfile がない feature に K010/K011 エラーが出る（feature 層のみ）
- [ ] domain 層では Dockerfile チェックが行われない

### 機能要件（upgrade 移行）

- [ ] `k1s0 upgrade` で v0.2.2 以前のサービスに Docker ファイル追加が提案される
- [ ] `k1s0 upgrade --check` で変更内容のプレビューが正しい
- [ ] 既に Docker ファイルが存在するサービスでは追加提案されない

### 機能要件（healthcheck エンドポイント）

- [ ] 全バックエンドテンプレートの生成コードに `/healthz` エンドポイントが含まれる
- [ ] Docker HEALTHCHECK と `/healthz` エンドポイントの連携で healthy 判定される

### 機能要件（Docker Secrets / K021 整合）

- [ ] compose.yaml の POSTGRES_PASSWORD に開発専用コメント + Docker Secrets サンプルが含まれる
- [ ] K021 lint が compose.yaml を検査対象外とする

### 機能要件（L3 統合サービス名衝突回避）

- [ ] `k1s0 docker up --all` で複数 feature を起動しても app サービス名が衝突しない
- [ ] include の project_directory 指定でプロジェクト分離されることを確認

### 機能要件（doctor 統合）

- [ ] `k1s0 doctor` に Docker Engine チェックが含まれる
- [ ] `k1s0 doctor` に Docker Compose V2 チェックが含まれる
- [ ] `k1s0 doctor --json` で Docker チェック結果が JSON に含まれる
- [ ] Docker 未インストール時にプラットフォーム別（Windows / macOS / Linux）のインストール案内が表示される

### 機能要件（CI/CD）

- [ ] docker.yml ワークフローが正常に動作する
- [ ] hadolint で Dockerfile lint が実行される
- [ ] Trivy でセキュリティスキャンが実行される
- [ ] マルチアーキテクチャビルド（QEMU + Buildx）が CI で動作する

### 非機能要件

- [ ] `cargo check` パス
- [ ] `cargo clippy --all-targets --all-features -- -D warnings` パス
- [ ] `cargo test --all` 全テストパス
- [ ] 既存コマンドの後方互換性が維持される
- [ ] `--json` / 非 TTY 環境で適切に動作する
