# k1s0 v0.2.3 改修方針

## 概要

Docker と docker-compose を k1s0 プラットフォームにファーストクラスサポートとして採用する。

- **Dockerfile テンプレート全言語対応**: Rust / Go バックエンドに Dockerfile.tera を新設（C# / Python は改善）
- **docker-compose.yaml テンプレート**: feature 生成時にサービス単体 + 依存サービスの compose.yaml を自動生成
- **`k1s0 docker` サブコマンド実装**: up / down / logs / build / ps の 5 コマンドを CLI に追加
- **docker.yml CI ワークフロー**: hadolint + ビルド + Trivy セキュリティスキャン
- **ルートレベル compose.yaml**: `k1s0 init` 時に共通インフラサービス定義を生成

## 機能目的

- 全バックエンド言語（Rust / Go / C# / Python）でコンテナ化を標準化する
- `k1s0 docker up` 一発でローカル開発環境を起動可能にする
- CI で Docker ビルド検証とセキュリティスキャンを自動化する
- docker-compose により複数サービスの統合テストを容易にする

## モノレポ × Docker の課題と対策

k1s0 のモノレポ構成と Docker は本質的に相性が悪い部分がある。以下にその課題と、本バージョンで採る対策を整理する。

### 課題一覧

| # | 課題 | 詳細 |
|---|------|------|
| 1 | **ビルドコンテキストの肥大化** | Docker の `COPY` はビルドコンテキスト内のファイルしか参照できない。feature サービスのディレクトリをコンテキストにすると、`framework/` や `domain/` の共有コードを COPY できない。リポジトリルートをコンテキストにすると数 GB のコンテキスト転送が発生する。 |
| 2 | **共有ライブラリへの依存** | feature サービスは `framework/` や `domain/` のクレート・パッケージに依存する。Docker ビルド時にこれらをコンテナ内で解決する方法が必要。 |
| 3 | **共有コード変更時の連鎖リビルド** | `framework/` の変更が全サービスの Docker キャッシュを無効化し、CI ビルド時間が爆発する。 |
| 4 | **Dockerfile の相対パス地獄** | `feature/backend/rust/order-processing/` から `framework/backend/rust/crates/` を参照するには `../../../../` のような深い相対パスが必要。 |
| 5 | **compose.yaml の散在** | 各 feature サービスに compose.yaml が散在し、プロジェクト全体の起動が複雑になる。 |
| 6 | **レジストリ公開前の循環依存** | framework パッケージがレジストリ（crates.io / NuGet / PyPI）未公開の場合、Docker ビルド内でローカルパス参照が必要になり Dockerfile が複雑化する。 |

### 対策

#### 対策 1: ビルドコンテキスト戦略（2 モード提供）

**モード A: サービス単体ビルド（デフォルト）**

- ビルドコンテキスト = feature サービスディレクトリ
- framework / domain はレジストリ経由で取得（crates.io / NuGet / PyPI / Go modules）
- 前提: framework パッケージがレジストリに公開済み
- メリット: コンテキストが小さく高速、Dockerfile がシンプル
- 用途: 本番デプロイ、CI リリースビルド

```dockerfile
# Mode A: レジストリ依存（デフォルト）
FROM rust:1.85-slim AS builder
WORKDIR /app
COPY . .
RUN cargo build --release
```

**モード B: モノレポルートビルド（開発・CI 統合テスト用）**

- ビルドコンテキスト = リポジトリルート
- `.dockerignore` でコンテキストを最小化（.git/, node_modules/, target/ 等を除外）
- Dockerfile 内で相対パスを使って framework / domain をコピー
- メリット: レジストリ未公開でもビルド可能、ローカル変更を即反映
- 用途: ローカル開発、CI 統合テスト

```dockerfile
# Mode B: モノレポルートコンテキスト
FROM rust:1.85-slim AS builder
WORKDIR /app

# framework クレートをコピー
COPY framework/backend/rust/crates/ /app/framework/
# domain クレートをコピー（該当するもののみ）
COPY domain/backend/rust/ /app/domain/
# feature サービスをコピー
COPY feature/backend/rust/{{ feature_name }}/ /app/service/

WORKDIR /app/service
RUN cargo build --release
```

compose.yaml は `context` と `dockerfile` を切り替えるだけで対応:

```yaml
services:
  app:
    build:
      # Mode A（デフォルト）: サービスディレクトリ
      context: .
      dockerfile: Dockerfile
      # Mode B（モノレポ）: ルートコンテキスト
      # context: ../../../..
      # dockerfile: feature/backend/rust/{{ feature_name }}/Dockerfile.monorepo
```

#### 対策 2: .dockerignore によるコンテキスト最小化

モノレポルートをコンテキストにする Mode B 用に、ルートレベルの `.dockerignore` を提供:

```
# .dockerignore（リポジトリルート）
.git
**/.git
**/node_modules
**/target
**/.venv
**/bin
**/obj
**/__pycache__
**/.mypy_cache
**/.ruff_cache
docs/
work/
.github/
*.md
!README.md
.claude/
```

これにより、コンテキスト転送量を大幅に削減する（推定: 数 GB → 数十 MB）。

#### 対策 3: レイヤーキャッシュ戦略

共有コード変更時の連鎖リビルドを緩和するため、依存解決とソースコードコピーを分離:

```dockerfile
# Step 1: 依存定義のみコピー（変更頻度: 低）
COPY Cargo.toml Cargo.lock ./
COPY framework/backend/rust/crates/*/Cargo.toml ./framework/
RUN cargo chef cook --release --recipe-path recipe.json

# Step 2: ソースコードコピー（変更頻度: 高）
COPY . .
RUN cargo build --release
```

これにより framework の Cargo.toml が変わらない限り、依存ビルドのキャッシュが効く。

#### 対策 4: compose.yaml の階層設計

散在する compose.yaml を整理するため、3 階層の Compose 構成を採用:

```
k1s0/
├── compose.yaml                    # L1: 共通インフラ（PostgreSQL, Redis, OTEL）
├── compose.override.yaml           # L1: ローカル開発用オーバーライド（任意）
├── feature/backend/rust/
│   └── order-processing/
│       └── compose.yaml            # L2: サービス単体 + 依存インフラ
└── compose.services.yaml           # L3: マルチサービス統合（k1s0 docker が自動生成）
```

- **L1（ルート）**: `k1s0 init` で生成。共通インフラのみ。
- **L2（feature）**: `k1s0 new-feature` で生成。サービス単体の開発用。
- **L3（統合）**: `k1s0 docker up --all` 時に CLI が動的生成。複数サービスの同時起動用。L2 の compose.yaml を `include` で集約する。

#### 対策 5: `k1s0 docker` による抽象化

Docker の複雑さを `k1s0 docker` コマンドで隠蔽する:

- `k1s0 docker up` → 適切な compose.yaml を自動検出・選択
- `k1s0 docker up --all` → L3 統合 compose を自動生成して起動
- `k1s0 docker build --mode monorepo` → Mode B でルートコンテキストビルド
- `k1s0 docker build` → Mode A でサービス単体ビルド（デフォルト）

ユーザーはモノレポ構成の複雑さを意識せず、k1s0 CLI を通じて Docker を操作できる。

#### 対策 6: framework 未公開時のフォールバック

framework パッケージがレジストリ未公開の場合（初期開発段階）:

1. `k1s0 doctor` が「framework パッケージがレジストリに未公開」を警告
2. `k1s0 docker build` が自動的に Mode B（モノレポルートコンテキスト）にフォールバック
3. Dockerfile.monorepo テンプレートを別途生成し、ローカルパス参照に対応

### 採用しなかった代替案

| 代替案 | 不採用理由 |
|--------|-----------|
| Bazel / Pants によるビルドシステム統合 | 学習コスト大、k1s0 のシンプルさと矛盾 |
| Docker BuildKit マウント（`--mount=type=bind`） | Compose V2 では YAML で表現しにくい、CI 互換性に懸念 |
| 全サービスを 1 つの Dockerfile でビルド | サービスの独立性が失われる |
| Git submodule で framework を分離 | モノレポの利点を放棄することになる |
| コンテナレスな開発のみサポート | ローカル開発は可能だが CI 統合テスト・本番デプロイに Docker は不可避 |

## 機能 1: Dockerfile テンプレート全言語対応

### 1.1 設計方針

- 全言語で Multi-stage build を採用し、最終イメージを最小化
- non-root ユーザー実行（セキュリティ）
- `config/` ディレクトリのコピー（k1s0 設定規約準拠）
- HEALTHCHECK ディレクティブで `/healthz` エンドポイントを監視
- `.dockerignore` で不要ファイルを除外

### 1.2 backend-rust Dockerfile（Mode A: レジストリ依存）

```dockerfile
# {{ feature_name }} - Multi-stage Docker build (Mode A)
# Rust {{ rust_version }}+
# Build context: feature/backend/rust/{{ feature_name }}/

# ---------------------------------------------------------------------------
# Stage 1: Chef (dependency cache)
# ---------------------------------------------------------------------------
FROM rust:{{ rust_version }}-slim AS chef
RUN cargo install cargo-chef
WORKDIR /app

# ---------------------------------------------------------------------------
# Stage 2: Planner
# ---------------------------------------------------------------------------
FROM chef AS planner
COPY . .
RUN cargo chef prepare --recipe-path recipe.json

# ---------------------------------------------------------------------------
# Stage 3: Builder
# ---------------------------------------------------------------------------
FROM chef AS builder
COPY --from=planner /app/recipe.json recipe.json
RUN cargo chef cook --release --recipe-path recipe.json
COPY . .
RUN cargo build --release --bin {{ feature_name }}

# ---------------------------------------------------------------------------
# Stage 4: Runtime
# ---------------------------------------------------------------------------
FROM debian:bookworm-slim AS runtime
WORKDIR /app

RUN apt-get update && apt-get install -y --no-install-recommends \
    ca-certificates \
    && rm -rf /var/lib/apt/lists/*

RUN groupadd -r appuser && useradd -r -g appuser -s /sbin/nologin appuser

COPY --from=builder /app/target/release/{{ feature_name }} /app/{{ feature_name }}
COPY config/ /app/config/

EXPOSE 8080
{% if with_grpc %}
EXPOSE 50051
{% endif %}

HEALTHCHECK --interval=10s --timeout=3s --start-period=5s --retries=3 \
    CMD ["/app/{{ feature_name }}", "healthcheck"] || exit 1

USER appuser
ENTRYPOINT ["/app/{{ feature_name }}"]
```

```dockerfile

EXPOSE 50051
{% endif %}

HEALTHCHECK --interval=10s --timeout=3s --start-period=5s --retries=3 \
    CMD ["/app/{{ feature_name }}", "healthcheck"] || exit 1

USER appuser
ENTRYPOINT ["/app/{{ feature_name }}"]
```

### 1.7 backend-csharp Dockerfile 改善（Mode A: レジストリ依存）

既存の `Dockerfile.tera` に HEALTHCHECK を追加する改善版。

```dockerfile
# {{ feature_name }} - Multi-stage Docker build (Mode A)
# ASP.NET Core 8.0
# Build context: feature/backend/csharp/{{ feature_name }}/

# ---------------------------------------------------------------------------
# Stage 1: Build
# ---------------------------------------------------------------------------
FROM mcr.microsoft.com/dotnet/sdk:8.0 AS build
WORKDIR /src

# Copy solution and project files first for layer caching
COPY {{ feature_name_pascal }}.sln .
COPY Directory.Build.props .
COPY Directory.Packages.props .
COPY src/{{ feature_name_pascal }}.Domain/{{ feature_name_pascal }}.Domain.csproj src/{{ feature_name_pascal }}.Domain/
COPY src/{{ feature_name_pascal }}.Application/{{ feature_name_pascal }}.Application.csproj src/{{ feature_name_pascal }}.Application/
COPY src/{{ feature_name_pascal }}.Infrastructure/{{ feature_name_pascal }}.Infrastructure.csproj src/{{ feature_name_pascal }}.Infrastructure/
COPY src/{{ feature_name_pascal }}.Presentation/{{ feature_name_pascal }}.Presentation.csproj src/{{ feature_name_pascal }}.Presentation/

# Restore dependencies
RUN dotnet restore src/{{ feature_name_pascal }}.Presentation/{{ feature_name_pascal }}.Presentation.csproj

# Copy all source code
COPY src/ src/
{% if with_grpc %}
COPY proto/ proto/
{% endif %}

# Build
RUN dotnet publish src/{{ feature_name_pascal }}.Presentation/{{ feature_name_pascal }}.Presentation.csproj \
    -c Release \
    -o /app/publish \
    --no-restore

# ---------------------------------------------------------------------------
# Stage 2: Runtime
# ---------------------------------------------------------------------------
FROM mcr.microsoft.com/dotnet/aspnet:8.0 AS runtime
WORKDIR /app

# Security: run as non-root
RUN groupadd -r appuser && useradd -r -g appuser -s /sbin/nologin appuser

COPY --from=build /app/publish .
COPY config/ /app/config/

# ASP.NET Core settings
ENV ASPNETCORE_URLS=http://+:8080
ENV DOTNET_EnableDiagnostics=0

EXPOSE 8080
{% if with_grpc %}
EXPOSE 50051
{% endif %}

HEALTHCHECK --interval=10s --timeout=3s --start-period=10s --retries=3 \
    CMD curl -f http://localhost:8080/healthz || exit 1

USER appuser
ENTRYPOINT ["dotnet", "{{ feature_name_pascal }}.Presentation.dll"]
```

1. `k1s0 docker up`（ルートで実行 → 共通インフラ起動）
2. `cd feature/backend/rust/order-processing && k1s0 docker up`（個別サービス起動）
   または `k1s0 docker up --all`（全サービス一括起動）

## 機能 10: `k1s0 upgrade` の Docker ファイル移行対応

### 10.1 設計方針

v0.2.3 以前に生成された既存サービスに対して `k1s0 upgrade` を実行した際、
Dockerfile / compose.yaml / .dockerignore が存在しなければ新規追加する。

### 10.2 upgrade ロジック

```rust
// upgrade 時の Docker ファイル移行
fn upgrade_docker_files(service_path: &Path, manifest: &Manifest) -> Result<Vec<UpgradeAction>> {
    let mut actions = vec![];

    // Dockerfile が存在しなければ追加提案
    if !service_path.join("Dockerfile").exists() {
        actions.push(UpgradeAction::AddFile {
            path: "Dockerfile".into(),
            reason: "v0.2.3: Docker コンテナ化対応".into(),
        });
    }

    // Dockerfile.monorepo が存在しなければ追加提案
    if !service_path.join("Dockerfile.monorepo").exists() {
        actions.push(UpgradeAction::AddFile {
            path: "Dockerfile.monorepo".into(),
            reason: "v0.2.3: モノレポビルド対応".into(),
        });
    }

    // .dockerignore, compose.yaml, compose.monorepo.yaml 同様
    // ...

    // manifest.json の managed_paths に Docker ファイルを追加
    if !manifest.managed_paths.contains(&"Dockerfile".to_string()) {
        actions.push(UpgradeAction::UpdateManifest {
            field: "managed_paths".into(),
            add: vec!["Dockerfile", "Dockerfile.monorepo", ".dockerignore",
                       "compose.yaml", "compose.monorepo.yaml", "deploy/docker/"],
        });
    }

    Ok(actions)
}
```

### 10.3 upgrade 出力例

```

  + compose.yaml                  (v0.2.3: Docker Compose 対応)
  + compose.monorepo.yaml         (v0.2.3: モノレポビルド対応)
  + deploy/docker/nginx.conf      (v0.2.3: フロントエンドのみ)
```

## オプション機能: プロキシ対応

### 背景

企業ネットワーク環境ではインターネットアクセスに HTTP/HTTPS プロキシが必要な場合が多い。
Docker ビルド時の依存パッケージ取得（cargo, go mod, pip, dotnet restore, npm）や、
Docker デーモン自体のイメージ Pull がプロキシ未設定により失敗するケースがある。
この機能はオプションとし、プロキシ環境でなければ一切影響しない設計とする。

### 対象レイヤーと対策

| レイヤー | プロキシが必要な場面 | 対策 |
|---------|---------------------|------|
| **Docker デーモン** | `docker pull` でのイメージ取得 | `k1s0 doctor` でプロキシ設定状況を検出・案内 |
| **Dockerfile ビルド** | `cargo install`, `go mod download`, `pip install`, `dotnet restore` | ビルド引数（`--build-arg`）でプロキシを注入 |
| **コンテナ実行時** | アプリからの外部 API 呼び出し | compose.yaml の `environment` で設定 |
| **k1s0 CLI** | `k1s0 docker build` 等の CLI 経由ビルド | CLI がホスト環境のプロキシ設定を自動検出し `--build-arg` に転送 |

### Dockerfile テンプレートへの組み込み

全言語の Dockerfile.tera に ARG を追加（Mode A / Mode B 共通）:

```dockerfile
# ---------------------------------------------------------------------------
# Proxy support (optional)
# Usage: docker build --build-arg HTTP_PROXY=http://proxy:8080 ...
# ---------------------------------------------------------------------------
ARG HTTP_PROXY
ARG HTTPS_PROXY
ARG NO_PROXY
ARG http_proxy
ARG https_proxy
ARG no_proxy
```

大文字・小文字の両方を定義する理由: ツールによって参照する環境変数名が異なる（curl は小文字、一部の Go ツールは大文字）。

ARG はビルド時のみ有効でイメージには残らないため、プロキシ認証情報の漏洩リスクはない。

#### Rust Dockerfile の例

```dockerfile
FROM rust:1.85-slim AS builder
ARG HTTP_PROXY
ARG HTTPS_PROXY
ARG NO_PROXY
ARG http_proxy
ARG https_proxy
ARG no_proxy

# cargo-chef は環境変数からプロキシを自動認識
RUN cargo install cargo-chef
```

#### Go Dockerfile の例

```dockerfile
FROM golang:1.22-alpine AS builder
ARG HTTP_PROXY
ARG HTTPS_PROXY
ARG NO_PROXY
ARG http_proxy
ARG https_proxy
ARG no_proxy

# go mod download は GOPROXY 環境変数も考慮
ARG GOPROXY
```

Go の場合、`GOPROXY`（Go モジュールプロキシ）も追加で受け付ける。

### compose.yaml テンプレートへの組み込み

compose.yaml.tera の app サービスに `build.args` セクションを追加:

```yaml
services:
  app:
    build:
      context: .
      dockerfile: Dockerfile
      args:
        - HTTP_PROXY=${HTTP_PROXY:-}
        - HTTPS_PROXY=${HTTPS_PROXY:-}
        - NO_PROXY=${NO_PROXY:-}
        - http_proxy=${http_proxy:-}
        - https_proxy=${https_proxy:-}
        - no_proxy=${no_proxy:-}
{% if language == "go" %}
        - GOPROXY=${GOPROXY:-}
{% endif %}
```

`${VAR:-}` 構文により、ホスト環境にプロキシ設定がなければ空文字が渡り、影響なし。

### `k1s0 docker build` のプロキシ自動転送

```rust
fn build_args_for_proxy() -> Vec<String> {
    let proxy_vars = [
        "HTTP_PROXY", "HTTPS_PROXY", "NO_PROXY",
        "http_proxy", "https_proxy", "no_proxy",
        "GOPROXY",
    ];
    proxy_vars.iter()
        .filter_map(|var| {
            std::env::var(var).ok().map(|val| {
                format!("--build-arg={}={}", var, val)
            })
        })
        .collect()
}
```

`k1s0 docker build` 実行時にホストのプロキシ環境変数を自動検出し、`docker compose build --build-arg` に転送する。ユーザーが手動で `--build-arg` を指定する必要はない。

### `k1s0 docker` サブコマンドへのオプション追加

```rust
Build {
    // ... 既存フィールド ...

    /// プロキシ自動転送を無効化
    #[arg(long)]
    no_proxy_passthrough: bool,
},

Up {
    // ... 既存フィールド ...

    /// プロキシ自動転送を無効化
    #[arg(long)]
    no_proxy_passthrough: bool,
},
```

デフォルトでプロキシ自動転送が有効。`--no-proxy-passthrough` で明示的に無効化可能。

### doctor への プロキシ診断追加

```
$ k1s0 doctor

  ✓ Docker Engine 24.0.7
  ✓ Docker Compose v2.24.0
  ⚠ HTTP_PROXY が設定されていますが、Docker デーモンにプロキシが未設定の可能性があります
    確認: docker info | grep -i proxy
    設定: https://docs.docker.com/config/daemon/systemd/#httphttps-proxy
  ⚠ NO_PROXY に localhost, 127.0.0.1 が含まれていません
    ヒント: ローカルサービス間通信がプロキシを経由する可能性があります
```

チェック項目:
- ホスト環境のプロキシ環境変数の有無
- Docker デーモンのプロキシ設定との整合性
- `NO_PROXY` に `localhost`, `127.0.0.1`, Docker ネットワークが含まれているか

### 設定ファイルによるプロキシ固定化（オプション）

プロジェクト単位でプロキシを固定する場合、`.k1s0/config.yaml` に設定可能:

```yaml
# .k1s0/config.yaml（任意）
docker:
  proxy:
    http: http://proxy.corp.example.com:8080
    https: http://proxy.corp.example.com:8080
    no_proxy: localhost,127.0.0.1,.corp.example.com
```

CLI はこの設定を環境変数より優先して `--build-arg` に渡す。
設定ファイルがなければ環境変数にフォールバックする。

### テスト方針

| テスト対象 | テスト内容 |
|-----------|-----------|
| `build_args_for_proxy()` | 環境変数あり → `--build-arg` 生成確認 |
| `build_args_for_proxy()` | 環境変数なし → 空リスト確認 |
| `--no-proxy-passthrough` | プロキシ転送が無効化されることを確認 |
| Dockerfile ARG | テンプレート展開後に ARG 行が含まれることを確認 |
| compose.yaml build.args | テンプレート展開後に build.args が含まれることを確認 |
| doctor プロキシ診断 | プロキシ環境変数設定時に警告が出ることを確認 |
| `.k1s0/config.yaml` | 設定ファイルのプロキシ値が環境変数より優先されることを確認 |

```rust
pub fn health_routes() -> Router {

}
```

#### Go（net/http）

```go
// cmd/server/main.go または internal/presentation/routes.go に追加
mux.HandleFunc("/healthz", func(w http.ResponseWriter, r *http.Request) {
    w.WriteHeader(http.StatusOK)
```

#### C#（ASP.NET Core）

```csharp
// src/{Name}.Presentation/Program.cs に追加
app.MapGet("/healthz", () => Results.Ok("ok"));
```

#### Python（FastAPI）

```python
# src/{feature_name_snake}/presentation/main.py に追加
```

#### React / Flutter（nginx）

## 補足: Docker Secrets と K021 規約の整合性

### 課題

compose.yaml テンプレートで `POSTGRES_PASSWORD: k1s0_dev_password` とハードコードしているが、k1s0 の K021 規約（シークレットのハードコード禁止）と矛盾する。

### 対応方針

- compose.yaml テンプレートに**開発専用である旨のコメント**を明記
- 本番環境では Docker Secrets パターンを使う旨を注釈
- lint の K021 ルールは `compose.yaml` を検査対象外とする（Docker Compose の `environment` は Docker レベルの設定であり、アプリケーション設定ではないため）

```yaml
# compose.yaml テンプレートに追加するコメント
  postgres:
    environment:
      # ⚠ 開発専用のデフォルト値。本番環境では Docker Secrets を使用すること:
      #   POSTGRES_PASSWORD_FILE: /run/secrets/db_password
      # secrets:
      #   db_password:
      #     file: ./secrets/db_password.txt
      POSTGRES_PASSWORD: k1s0_dev_password
```

### K021 lint との関係

K021 ルールの検査対象は `config/*.yaml`（アプリケーション設定ファイル）のみ。`compose.yaml` は Docker Compose 定義であり、K021 の検査対象外とする。この判断を `docs/conventions/config-and-secrets.md` に明記する。

## 補足: L3 統合時のサービス名衝突回避

### 課題

`k1s0 docker up --all` で複数 feature の compose.yaml を `include` で集約する際、全 feature が `app` サービス名を使用していると名前が衝突する。

### 対応方針

Compose の `include` ディレクティブで `project_directory` と暗黙のプロジェクト名分離を利用する。

#### 方式 1: include の project 分離（採用）

```yaml
# compose.services.yaml（L3 統合 compose）
include:
  - path: feature/backend/rust/order-processing/compose.yaml
    project_directory: feature/backend/rust/order-processing
    env_file: []
  - path: feature/backend/go/user-management/compose.yaml
    project_directory: feature/backend/go/user-management
    env_file: []
```

Compose V2 の `include` は各インクルードを独立したプロジェクトとして扱い、サービス名にプロジェクト名プレフィックスが自動付与されるため衝突しない。

#### 方式 2: feature_name をサービス名に使用（不採用）

```yaml
services:
  {{ feature_name }}:  # "app" ではなく feature_name
    build: ...
```

不採用理由: 単体起動時に `docker compose exec app` のような汎用的なコマンドが使えなくなる。

### compose_aggregator.rs への反映

```rust
// L3 統合 compose 生成時
fn generate_l3_compose(features: &[FeatureInfo]) -> String {
    let mut yaml = String::from("# Generated by k1s0 CLI\ninclude:\n");
    for feature in features {
        yaml.push_str(&format!(
            "  - path: {}/compose.yaml\n    project_directory: {}\n    env_file: []\n",
            feature.relative_path, feature.relative_path
        ));
    }
    yaml
}
```

## 補足: manifest.schema.json の service.features スキーマ

### 課題

`manifest.schema.json` に `service.features` スキーマが未定義であり、`with_cache` だけでなく `with_grpc` / `with_db` も定義されていない。

### 対応方針

`with_cache` 追加と同時に既存の `with_grpc` / `with_db` もスキーマに追加する。

```json
{
  "service": {
    "type": "object",
    "properties": {
      "service_name": { "type": "string" },
      "language": { "type": "string" },
      "type": { "type": "string", "enum": ["feature", "domain"] },
      "features": {
        "type": "object",
        "properties": {
          "with_grpc": { "type": "boolean", "default": false },
          "with_db": { "type": "boolean", "default": false },
          "with_cache": { "type": "boolean", "default": false },
          "with_docker": { "type": "boolean", "default": true }
        },
        "additionalProperties": false
      }
    }
  }
}
```

`with_docker` も追加し、`--no-docker` で生成された場合は `false` となる。
