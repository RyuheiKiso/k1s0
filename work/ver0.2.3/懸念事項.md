## 致命的な懸念（対処必須）

### C-01: ヘルスチェックのポートとパスの不一致 ✅ 対処済み

**深刻度: 高** → **対処済み（改修方針・作業チェックリストに反映）**

改修方針の Dockerfile では `curl -f http://localhost:8080/health` を HEALTHCHECK に設定しているが、実際のテンプレート設定は異なる。

**config/default.yaml.tera（Rust テンプレート）の実態:**

```yaml
health:
  port: 9090                   # 8080 ではない
  liveness_path: /health/live  # /health ではない
  readiness_path: /health/ready
  startup_path: /health/startup
```

**deploy/base/deployment.yaml.tera の実態:**

```yaml
livenessProbe:
  httpGet:
    path: /healthz            # /health でも /health/live でもない
    port: http                # 8080
```

3箇所で使用されるヘルスチェックのポート・パスが統一されていない。

| 箇所 | ポート | パス |
|------|-------|------|
| config/default.yaml.tera | 9090 | /health/live |
| deploy/base/deployment.yaml.tera | 8080 | /healthz |
| 改修方針 Dockerfile HEALTHCHECK | 8080 | /health |

**影響**: Dockerfile の HEALTHCHECK が常に失敗し、コンテナが unhealthy になる。docker-compose の `depends_on: condition: service_healthy` が永遠に待機する。

**対処案**: Dockerfile の HEALTHCHECK を K8s deployment と同じ `http://localhost:8080/healthz` に統一するか、アプリの実装に合わせて確定する。全テンプレート（Rust/Go/C#/Python）で統一が必要。

---

### C-02: config/default.yaml のホスト名がコンテナ環境に合わない ✅ 対処済み

**深刻度: 高** → **対処済み（Phase 0.3 で dev.yaml にコンテナ用ホスト名を追記する方針に確定）**

config/default.yaml.tera では DB と Redis の接続先が `localhost` にハードコードされている。

```yaml
database:
  host: localhost     # コンテナ環境では「db」
cache:
  redis:
    url: "redis://localhost:6379"   # コンテナ環境では「redis」
```

Docker Compose では各サービスが別コンテナで動作するため、`localhost` では接続できない。

**影響**: `docker compose up` 後、アプリが DB・Redis に接続できず起動失敗する。

**対処案**:
- config/dev.yaml.tera にコンテナ用の接続先（`db`, `redis`）を追記する
- docker-compose.yml でアプリ起動時に `--env dev --config /app/config` を指定する
- ただし default.yaml はホスト直接実行用に `localhost` を維持する

**補足**: dev.yaml.tera の現状を確認したところ、DB/Redis のホスト上書きは存在しない。docker-compose 導入に合わせて dev.yaml にコンテナ用設定を追加するか、docker-compose 専用の config/docker.yaml を新設する必要がある。後者は `--env` の固定4値（default/dev/stg/prod）に `docker` を追加することになり、規約変更を伴う。

---

### C-03: Dockerfile の ENTRYPOINT に `--env` / `--config` 引数がない ✅ 対処済み

**深刻度: 高** → **対処済み（ENTRYPOINT + CMD 分離パターンを全 Dockerfile に適用する方針に確定）**

K8s deployment.yaml.tera では起動引数が明示されている:

```yaml
args:
  - --env
  - $(ENV)
  - --config
  - /etc/k1s0/config
  - --secrets-dir
  - /var/run/secrets/k1s0
```

しかし改修方針の Dockerfile では:

```dockerfile
ENTRYPOINT ["/app/order-processing"]   # 引数なし
```

アプリケーションが config ファイルを読み取れない。

**影響**: コンテナ起動後に設定ファイルが読み込まれず、デフォルト値やパニックが発生する。

**対処案**: Dockerfile の ENTRYPOINT または CMD で `--env dev --config /app/config` を指定する。docker-compose.yml の `command:` で上書きする方法もあるが、Dockerfile 単体でも動作する設計が望ましい。

```dockerfile
ENTRYPOINT ["/app/{{ feature_name }}"]
CMD ["--env", "default", "--config", "/app/config"]
```

## 設計上の懸念

### D-01: Dockerfile の managed/protected 分類の妥当性

**深刻度: 中**

改修方針では Dockerfile を protected（テンプレート管理、直接編集非推奨）としているが、実運用では以下のカスタマイズ需要がある。

- 追加の apt パッケージインストール（例: imagemagick, ffmpeg）
- ビルド引数の追加（`ARG`, `--build-arg`）
- マルチステージの中間ステージ追加
- ランタイムの環境固有設定

manifest.json の update_policy 定義:

```json
"update_policy": {
  "deploy/": "auto",
  "config/": "suggest_only"
}
```

**影響**: Dockerfile を protected にすると `k1s0 upgrade` で自動上書きされ、チームのカスタマイズが消える可能性がある。

**対処案**: update_policy を `"Dockerfile": "suggest_only"` にし、diff を表示して手動適用を促す。docker-compose.yml は unmanaged で問題ない。

---

### D-02: docker-compose.yml 内の `environment` と K020 ルールの境界

**深刻度: 中**

改修方針では docker-compose.yml の `environment` は「DB初期化等の外部イメージ設定のみ許可」としているが、K020 lint ルール（`CLI/crates/k1s0-generator/src/lint/env_vars.rs`）は `.yml` / `.yaml` ファイル内の環境変数パターンもチェック対象にしている可能性がある。

```yaml
# docker-compose.yml
services:
  db:
    environment:
      POSTGRES_DB: order_processing      # K020 でフラグされる？
      POSTGRES_PASSWORD_FILE: /run/secrets/db_password
```

**影響**: lint が docker-compose.yml の `environment` セクションを検出して K020 違反を報告する可能性がある。

**対処案**: K020 ルールの除外パスに `docker-compose.yml` を追加する。または、K020 がスキャンする対象ファイルを `src/` 配下のアプリケーションコードのみに限定する。現状の実装を精査して判断が必要。

---

### D-03: secrets/ ディレクトリと K021 ルールの関係

**深刻度: 低**

`secrets/db_password.txt` に初期値 `changeme` を書き込む設計だが、K021（シークレットハードコード検出）がこのファイルをスキャンする可能性がある。

**対処案**: K021 のスキャン対象から `secrets/` ディレクトリを除外する。`.gitignore` 対象のファイルはそもそもリポジトリに入らないため、CI での lint は問題にならないが、ローカル `k1s0 lint` で検出される可能性がある。

---

### D-04: ヘルスチェック専用ポート（9090）とDockerの相性

**深刻度: 中**

config/default.yaml.tera ではヘルスチェックが専用ポート 9090 で提供される設計になっている。しかし Dockerfile の `EXPOSE` には 8080 と 50051 のみ記載されており、9090 がない。

```yaml
health:
  port: 9090
```

```dockerfile
EXPOSE 8080
EXPOSE 50051   # with_grpc
# 9090 がない
```

docker-compose.yml のヘルスチェックも 8080 を想定しており、9090 にアクセスできない。

**影響**: ヘルスチェックが設計通りに動作しない。

**対処案**:
1. Dockerfile に `EXPOSE 9090` を追加し、docker-compose のヘルスチェックも 9090 に変更する
2. または、ヘルスチェックを 8080 の `/healthz` エンドポイントに統一し、専用ポートを廃止する

---

### D-05: CLI docker コマンドと `std::process::Command` のクロスプラットフォーム対応

**深刻度: 中**

k1s0 CLI は Windows でも使用される（現在の開発環境が Windows）。`k1s0 docker build` は内部で `docker` コマンドを呼び出すが:

- Windows: `docker.exe` / `docker compose` （Docker Desktop）
- Linux/Mac: `docker` / `docker compose`
- 古い環境: `docker-compose`（v1、ハイフン区切り）

**影響**: Windows 環境でパス解決やプロセス起動が失敗する可能性がある。

**対処案**: `which` / `where` でコマンドを探索し、見つからない場合は doctor で検知・リカバリ提案を行う。Docker Compose v1（`docker-compose`）はサポート対象外であることを明記する。

---

### D-06: frontend-react の Dockerfile ビルドで pnpm-lock.yaml が必須

**深刻度: 低**

改修方針の frontend-react Dockerfile:

```dockerfile
COPY package.json pnpm-lock.yaml ./
RUN pnpm install --frozen-lockfile
```

`--frozen-lockfile` は `pnpm-lock.yaml` が存在しないとエラーになる。`k1s0 new-feature --type frontend-react` で生成直後は `pnpm-lock.yaml` が存在しない（`pnpm install` 実行前）。

**影響**: 生成直後に `docker build` するとビルド失敗する。

**対処案**:
1. `k1s0 new-feature` 生成後に `pnpm install` を自動実行するか、手順として明記する
2. または Dockerfile を `RUN pnpm install` に変更し、`--frozen-lockfile` は CI 用に docker-compose で上書きする

## 互換性の懸念

### B-01: 既存サービスへの upgrade で Dockerfile が追加される際の影響

**深刻度: 中**

v0.2.3 より前に `k1s0 new-feature` で生成された既存サービスには Dockerfile がない。`k1s0 upgrade` で Dockerfile が追加される想定だが:

- manifest.json の managed_paths に Dockerfile が追加されるため、manifest のスキーマバージョン差分が発生する
- 既存の手動作成 Dockerfile がある場合、衝突が発生する

**影響**: upgrade 時にファイル衝突エラーまたはユーザーの Dockerfile が上書きされる。

**対処案**:
- 新規ファイル追加の場合は衝突チェックを行い、既存ファイルがあれば diff 表示 + 確認プロンプトを出す
- manifest.json の managed_paths 追加は MINOR バージョン変更として扱う

---

### B-02: template_version の変更判定

**深刻度: 低**

Dockerfile / docker-compose.yml の追加は template_version のどのレベルの変更に該当するか。

- PATCH: バグ修正のみ → 不適切（新機能追加）
- MINOR: 後方互換な機能追加 → 適切（Dockerfile は追加ファイル、既存ファイルへの影響は HEALTHCHECK のみ）
- MAJOR: 破壊的変更 → 不適切（既存機能を壊さない）

**対処案**: MINOR バージョンアップとし、`k1s0 upgrade --check` で追加ファイル一覧を表示する。

---

### B-03: 既存 C# / Python Dockerfile への HEALTHCHECK 追加が破壊的でないかの確認

**深刻度: 低**

既存の C# / Python テンプレートで生成されたサービスが `k1s0 upgrade` した際、HEALTHCHECK と `apt-get install curl` が追加される。

- `apt-get install curl` によりイメージサイズが増加する（数MB）
- HEALTHCHECK 追加自体は破壊的ではない（Docker が定期的にヘルスチェックを実行するだけ）

**対処案**: PATCH レベルの変更として扱い、upgrade 時に diff を表示する。

## 運用上の懸念

### O-01: 複数サービス同時起動時のポート競合

**深刻度: 中**

開発者が複数のサービスを同時に `docker compose up` した場合、以下のポートが競合する。

| ポート | 用途 | 競合リスク |
|-------|------|-----------|
| 8080 | HTTP | 全バックエンドサービスで競合 |
| 50051 | gRPC | 全 gRPC サービスで競合 |
| 5432 | PostgreSQL | 全 DB 付きサービスで競合 |
| 6379 | Redis | 全 Cache 付きサービスで競合 |
| 9090 | ヘルスチェック | 全バックエンドサービスで競合 |
| 3000 | フロントエンド | React サービスで競合 |

**影響**: 2つ目のサービスの `docker compose up` が `port already allocated` で失敗する。

**対処案**:
- docker-compose.yml は unmanaged なのでポート変更をドキュメントに明記する
- 「開発者の振る舞い.md」のシナリオ 4 で言及済みだが、より具体的なガイドが必要
- 将来的には `k1s0 docker compose up` でポートオフセットを自動計算する機能を検討

---

### O-02: Docker イメージサイズの肥大化

**深刻度: 低**

マルチステージビルドにより最終イメージは軽量になるが、言語ごとに差がある。

| テンプレート | ベースイメージ | 推定サイズ |
|------------|-------------|----------|
| backend-rust | debian:bookworm-slim + バイナリ | ~100MB |
| backend-go | alpine:3.19 + バイナリ | ~30MB |
| backend-csharp | aspnet:8.0 | ~220MB |
| backend-python | python:3.12-slim + パッケージ | ~200MB |
| frontend-react | nginx:alpine + 静的ファイル | ~40MB |

**対処案**: 特に対処不要。マルチステージビルドにより十分に軽量。将来的に distroless イメージへの移行を検討する余地はある。

---

### O-03: Docker Desktop ライセンス

**深刻度: 情報提供**

Docker Desktop は商用利用で有料（従業員250人以上または年間売上1,000万ドル以上の企業）。

**対処案**: ドキュメントにライセンスに関する注意書きを追加する。代替として Rancher Desktop / Podman の使用可能性についても言及する。ただし、k1s0 の docker コマンドは `docker` CLI を前提としており、Podman 互換は保証しない。

---

### O-04: Docker ビルドキャッシュの効率

**深刻度: 低**

Rust テンプレートのダミー main.rs テクニック:

```dockerfile
RUN mkdir src && echo "fn main() {}" > src/main.rs \
    && cargo build --release \
    && rm -rf src
```

これは依存のキャッシュには有効だが、`Cargo.lock` を COPY していない場合、依存バージョンが非再現的になる。

**影響**: CI とローカルでビルド結果が異なる可能性がある。

**対処案**: `COPY Cargo.toml Cargo.lock ./` で Cargo.lock も含める。ただし、生成直後は Cargo.lock が存在しない可能性があるため、`COPY Cargo.lock* ./` のようなワイルドカード対応を検討する。

---

### O-05: Windows 環境での volume マウント性能

**深刻度: 中**

Windows + Docker Desktop（WSL 2 バックエンド）で `./config:/app/config:ro` のようなバインドマウントはパフォーマンスが低い（特にファイル数が多い場合）。

**影響**: config/ のマウントは数ファイルなので問題ないが、ソースコード全体をマウントするホットリロード構成を将来追加する場合に問題になる。

**対処案**: v0.2.3 では config/ のみのマウントなので対処不要。ドキュメントに注意書きを追加する程度でよい。

## テストの懸念

### T-01: CI での Docker ビルドテストの実行時間

**深刻度: 中**

5テンプレート × Docker ビルド + docker-compose テストは、CI ランナーのリソースを大量に消費する。

| テスト | 推定時間 |
|-------|---------|
| backend-rust docker build | 5-10分（Rust コンパイル） |
| backend-go docker build | 1-2分 |
| backend-csharp docker build | 2-3分 |
| backend-python docker build | 1-2分 |
| frontend-react docker build | 1-2分 |
| docker-compose up（各テンプレート） | 1-3分 |

合計: 15-30分（並列実行時）。

**影響**: CI パイプライン全体の実行時間が大幅に増加する。

**対処案**:
- docker.yml は Docker 関連ファイル変更時のみトリガー（path フィルター）
- matrix で並列実行
- Docker レイヤーキャッシュ（`actions/cache` + `docker/build-push-action` のキャッシュ機能）を活用
- E2E テスト（compose up → test → down）は nightly ビルドに分離する案も検討

---

### T-02: テンプレート生成直後の Docker ビルド成功が保証できない

**深刻度: 高**

`k1s0 new-feature` で生成されたスキャフォールドは最小限のコードしか含まない。テンプレートタイプごとに、Docker ビルドが即座に成功するかの確認が必要。

| テンプレート | 生成直後のビルド可否 | 課題 |
|------------|-------------------|------|
| backend-rust | 要確認 | `src/main.rs` に最小限の axum サーバーがあるか |
| backend-go | 要確認 | `cmd/main.go` がコンパイルできるか |
| backend-csharp | 可能（既存 Dockerfile で動作確認済み） | - |
| backend-python | 可能（既存 Dockerfile で動作確認済み） | - |
| frontend-react | 要確認 | `pnpm-lock.yaml` 不在（D-06 参照）、`pnpm build` でビルドできるか |

**影響**: 「生成してすぐ docker compose up」の体験が壊れる。

**対処案**: 各テンプレートの `src/main.*` が最小限のヘルスチェックエンドポイントを含み、Docker ビルド + 起動まで成功することをテンプレートの品質基準とする。CI の統合テストで `new-feature → docker build → docker run → healthcheck` を検証する。

---

### T-03: docker-compose テストで外部イメージのプル失敗

**深刻度: 低**

CI 環境で `postgres:16-alpine` や `redis:7-alpine` のプルがレートリミットやネットワーク障害で失敗する可能性がある。

**対処案**: CI でイメージキャッシュを利用する。GitHub Actions の `docker/setup-buildx-action` + レジストリキャッシュを活用する。

---

### T-04: ヘルスチェックの待機時間によるテストタイムアウト

**深刻度: 中**

HEALTHCHECK の設定:

```dockerfile
HEALTHCHECK --interval=10s --timeout=3s --retries=3
```

最悪の場合、healthy になるまで 10s × 3 = 30 秒かかる。docker-compose の `depends_on: condition: service_healthy` がこの間ブロックされる。

DB の起動 + アプリの起動を含めると、全体で 60 秒以上かかる可能性がある。

**影響**: CI テストがタイムアウトする可能性がある。

**対処案**:
- CI テストのタイムアウトを十分に長く設定する（3-5分）
- HEALTHCHECK の interval を CI 用に短くする（5s）ことは Docker の推奨に反するため避ける
- テスト側で `docker compose up` 後にポーリングで healthy を待つスクリプトを用意する

---
