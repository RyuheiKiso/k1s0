# 要望

Docker と docker-compose を k1s0 プラットフォームに本格採用したい。
現在の k1s0 は Kubernetes（Kustomize）ベースのデプロイメントを前提としているが、ローカル開発環境や CI での統合テストで docker-compose が欠かせない。また、Dockerfile テンプレートが C# と Python にしか存在せず、Rust / Go バックエンドには未対応である。

## 背景

- Dockerfile テンプレートが `backend-csharp` と `backend-python` にのみ存在し、`backend-rust` と `backend-go` には未対応
- docker-compose.yaml のテンプレートや生成機能が存在しない
- `k1s0 docker` サブコマンド群（up/down/logs/build/ps）が CLAUDE.md に記載されているが未実装
- docker.yml CI ワークフローが未作成
- ローカル開発で PostgreSQL / Redis / OTEL Collector などの依存サービスを手動で起動する必要がある
- 複数の feature サービスを連携テストする際に、統一的な起動方法がない

### モノレポ × Docker の課題認識

k1s0 のモノレポ構成と Docker には本質的な相性問題がある。Docker の `COPY` 命令はビルドコンテキスト内のファイルしか参照できないため、feature サービスのディレクトリをコンテキストにすると `framework/` や `domain/` の共有コードをコピーできない。リポジトリルートをコンテキストにすると数 GB の転送が発生する。この課題を解決する設計が必要である。

主な課題:
1. **ビルドコンテキストの肥大化**: ルートコンテキストだと巨大、サービスコンテキストだと共有コードが取れない
2. **共有ライブラリの参照**: feature は framework / domain に依存するが、Docker 内での解決方法が必要
3. **共有コード変更時の連鎖リビルド**: framework の変更が全サービスのキャッシュを無効化
4. **相対パス地獄**: `feature/backend/rust/X/` から `framework/backend/rust/crates/` へは `../../../../` が必要
5. **compose.yaml の散在**: 各 feature に compose.yaml が散在し、統合起動が複雑
6. **レジストリ未公開時の循環**: framework がレジストリ未公開だと Docker ビルドが困難

## 回答（実装方針）

### 概要

13の機能を実装し、Docker / docker-compose をファーストクラスサポートにする。

1. **全バックエンドテンプレートへの Dockerfile 追加**: Rust / Go テンプレートに Dockerfile.tera を新設、C# / Python に HEALTHCHECK 追加
2. **フロントエンドテンプレートへの Dockerfile 追加**: React / Flutter テンプレートに Dockerfile.tera を新設（nginx 配信）
3. **docker-compose.yaml テンプレート生成**: feature 生成時にサービス単体 + 依存サービスの docker-compose.yaml を自動生成
4. **`k1s0 docker` サブコマンド実装**: up/down/logs/build/ps の 5 コマンドを CLI に追加（--tag / --push / --platform 対応）
5. **docker.yml CI ワークフロー追加**: Dockerfile の hadolint + ビルド + Trivy セキュリティスキャン + マルチアーキテクチャ
6. **ルートレベル docker-compose.yaml 生成**: `k1s0 init` 時にプロジェクト全体の共通インフラサービス定義を生成
7. **manifest.json への Docker メタデータ追加**: managed_paths / update_policy に Docker ファイルを追加
8. **lint ルール K010/K011 更新**: Dockerfile を feature 層の必須ファイルに追加
9. **hadolint 設定ファイル**: .hadolint.yaml を init 時に生成
10. **`k1s0 upgrade` 移行対応**: 既存サービスへの Docker ファイル自動追加
11. **コンテナレジストリ / 命名規則 / マルチアーキテクチャ**: イメージ命名規則と .k1s0/settings.yaml 連携
12. **`--with-docker` / `--no-docker` フラグ**: Docker ファイル生成のオプトアウト
13. **manifest.schema.json の with_cache 対応**: スキーマに with_cache プロパティ追加

### 技術選定

| 機能 | 使用技術 | 理由 |
|------|---------|------|
| コンテナランタイム | Docker Engine 24+ | デファクトスタンダード |
| マルチコンテナ定義 | Docker Compose V2 (compose.yaml) | docker-compose v1 は非推奨、V2 は Docker CLI 統合 |
| Dockerfile lint | hadolint | Dockerfile ベストプラクティスの自動検査 |
| セキュリティスキャン | Trivy | コンテナイメージの脆弱性検出 |
| CLI 統合 | std::process::Command | 外部コマンド実行（既存パターン） |

### 設計方針

- **2 モードビルド戦略**: Mode A（サービス単体コンテキスト、レジストリ依存）と Mode B（モノレポルートコンテキスト、ローカル参照）を提供。デフォルトは Mode A、`--mode monorepo` で Mode B に切り替え
- **`k1s0 docker` による抽象化**: モノレポの複雑さを CLI が隠蔽し、ユーザーは Docker の細部を意識しない
- **3 階層 compose 設計**: L1（ルート共通インフラ）→ L2（feature 単体）→ L3（統合起動、CLI が動的生成）
- **Compose V2 準拠**: `compose.yaml`（docker-compose.yaml ではなく）を正式ファイル名として採用（後方互換のため docker-compose.yaml も読み込み可能）
- **テンプレート変数活用**: 既存の Tera テンプレート変数（`feature_name`, `with_grpc`, `with_db` 等）を docker-compose.yaml にも適用
- **依存サービス自動構成**: `--with-db` で PostgreSQL、`--with-cache` で Redis を自動追加
- **プロファイルベース分離**: 開発用（dev）・テスト用（test）・CI 用（ci）をプロファイルで切り替え
- **ヘルスチェック統合**: 既存の k1s0-health パッケージと Docker HEALTHCHECK を連携
- **ルート .dockerignore**: Mode B 時のコンテキスト転送を最小化（数 GB → 数十 MB）

## 機能 1: 全バックエンドテンプレートへの Dockerfile 追加

### 対象

| テンプレート | 現状 | v0.2.3 |
|-------------|------|--------|
| backend-rust | Dockerfile なし | 新規追加 |
| backend-go | Dockerfile なし | 新規追加 |
| backend-csharp | Dockerfile.tera あり | .dockerignore 改善 |
| backend-python | Dockerfile.tera あり | .dockerignore 改善 |

### Rust Dockerfile 方針

- Multi-stage build: cargo-chef でキャッシュ最適化 → release ビルド → distroless/debian runtime
- non-root ユーザー実行
- config/ ディレクトリのコピー

### Go Dockerfile 方針

- Multi-stage build: go build → scratch / distroless runtime
- CGO_ENABLED=0 で静的リンク
- non-root ユーザー実行

## 機能 2: docker-compose.yaml テンプレート生成

### 生成タイミング

- `k1s0 new-feature` 実行時にサービス単位の `compose.yaml` を自動生成
- `--with-db`, `--with-cache` フラグに応じて依存サービスを追加

### 生成例

```yaml
# feature/backend/rust/order-processing/compose.yaml
services:
  app:
    build:
      context: .
      dockerfile: Dockerfile
    ports:
      - "8080:8080"
    volumes:
      - ./config:/app/config:ro
    depends_on:
      postgres:
        condition: service_healthy
    environment:
      - K1S0_ENV=dev

  postgres:
    image: postgres:16-alpine
    ports:
      - "5432:5432"
    volumes:
      - postgres_data:/var/lib/postgresql/data
    environment:
      POSTGRES_DB: order_processing
      POSTGRES_USER: k1s0
      POSTGRES_PASSWORD_FILE: /run/secrets/db_password
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U k1s0"]
      interval: 5s
      timeout: 3s
      retries: 5

volumes:
  postgres_data:
```

## 機能 3: `k1s0 docker` サブコマンド

### コマンド一覧

| コマンド | 説明 | 内部動作 |
|---------|------|---------|
| `k1s0 docker up` | サービス起動 | `docker compose up -d` |

| `k1s0 docker ps` | ステータス表示 | `docker compose ps` |

### 追加オプション

```
k1s0 docker up [--profile dev|test|ci] [--service <name>] [--build]
k1s0 docker down [--volumes]
k1s0 docker logs [--service <name>] [--tail <n>]
k1s0 docker build [--no-cache] [--service <name>]
k1s0 docker ps [--all]
```

| frontend-react | Dockerfile なし | 新規追加（node + nginx） |

### React Dockerfile 方針

## 機能 11: プロキシ対応（オプション）

企業ネットワーク等でプロキシ経由のインターネットアクセスが必要な環境向け。

### 対応範囲

| 場面 | 対策 |
|------|------|
| Docker ビルド時のパッケージ取得 | Dockerfile に `ARG HTTP_PROXY` 等を定義、`k1s0 docker build` がホスト環境変数を自動転送 |
| docker compose build | compose.yaml の `build.args` でプロキシ変数を注入（`${VAR:-}` 形式、未設定時は空） |
| プロキシ設定の診断 | `k1s0 doctor` がプロキシ環境変数の存在・Docker デーモン設定との整合性をチェック |
| プロジェクト固定設定 | `.k1s0/config.yaml` の `docker.proxy` セクションで環境変数より優先して設定可能 |

### 動作仕様

```
# プロキシ環境では追加操作不要（自動検出）
$ export HTTP_PROXY=http://proxy.corp:8080
$ k1s0 docker build
  ℹ プロキシ設定を検出しました: HTTP_PROXY=http://proxy.corp:8080
  ビルド引数に自動転送します

# 明示的に無効化
$ k1s0 docker build --no-proxy-passthrough
```

### 設計原則

- **ゼロコンフィグ**: プロキシ環境変数があれば自動転送、なければ何もしない
- **セキュリティ**: `ARG` はビルド時のみ有効でイメージには残らない。プロキシ認証情報の漏洩リスクなし
- **オプトアウト可能**: `--no-proxy-passthrough` で明示的に無効化

## 注意事項

- **環境変数の使用制限**: compose.yaml 内の `environment` は Docker レベルの設定のみ許可。アプリケーション設定は `config/*.yaml` 経由
- **シークレット管理**: `POSTGRES_PASSWORD` 等は Docker Secrets (`*_FILE` パターン) を使用。開発用 compose.yaml のハードコード値には開発専用コメントを明記し、K021 lint は compose.yaml を検査対象外とする
- **後方互換性**: 既存の Kustomize デプロイメントは維持。docker-compose はローカル開発・CI 用途
- **CI互換**: `--json` モードでは docker コマンドの出力もJSON形式に統一
- **モノレポ × Docker**: Mode A / Mode B の 2 モード提供で対応。詳細は改修方針の「モノレポ × Docker の課題と対策」を参照
- **framework 未公開時**: Mode B（モノレポルートコンテキスト）で回避。`k1s0 doctor` が未公開を警告し Mode B を推奨
- **healthcheck エンドポイント**: Dockerfile の HEALTHCHECK が `/healthz` を叩く。各言語テンプレートのアプリケーションコードに `/healthz` エンドポイントが実装されていること（未実装なら追加）
- **L3 統合時のサービス名衝突**: `k1s0 docker up --all` で複数 feature を include する際、Compose V2 の include project 分離機能でサービス名衝突を回避
- **Windows / macOS 対応**: Docker Desktop（商用ライセンス）または Rancher Desktop / Colima が必要。`k1s0 doctor` でプラットフォーム別のインストール案内を提供
- **manifest.schema.json**: with_cache 追加と同時に、未定義だった with_grpc / with_db / with_docker もスキーマに追加
