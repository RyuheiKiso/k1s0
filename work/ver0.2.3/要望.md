# 要望

Docker と docker-compose をファーストクラスでサポートしたい。
現在のk1s0はテンプレート生成でClean Architectureのスキャフォールドを提供しているが、コンテナ化は C# と Python テンプレートのみ Dockerfile が存在し、Rust・Go・React にはDockerサポートがない。また、docker-compose による複数サービスのローカル開発環境構築がどのテンプレートにも用意されていない。

## 背景

- backend-rust、backend-go テンプレートに Dockerfile が存在しない
- frontend-react テンプレートに Dockerfile が存在しない（SPA のコンテナ配信不可）
- どのテンプレートにも docker-compose.yml が存在しない
- ローカル開発で PostgreSQL・Redis 等の依存サービスを個別に用意する必要がある
- CI/CD ワークフローで Docker イメージのビルド・プッシュが標準化されていない
- `k1s0 new-feature` で生成後、すぐに `docker compose up` で動作確認できない

## 回答（実装方針）

### 概要

4つの機能を実装する。

1. **全バックエンドテンプレートに Dockerfile を追加**: backend-rust、backend-go に Dockerfile.tera と .dockerignore を追加（C#・Python は既存を改善）
2. **frontend-react テンプレートに Dockerfile を追加**: Nginx ベースのマルチステージビルド
3. **docker-compose.yml テンプレートを全テンプレートに追加**: アプリ + PostgreSQL + Redis のローカル開発構成
4. **CLI docker コマンドの追加**: `k1s0 docker build` / `k1s0 docker compose` のラッパーコマンド

### 技術選定

| 機能 | 使用技術 | 理由 |
|------|---------|------|
| Rust ビルド | `rust:1.85-slim` + `debian:bookworm-slim` | マルチステージでバイナリのみコピー、軽量ランタイム |
| Go ビルド | `golang:1.22-alpine` + `alpine:3.19` | 静的バイナリ + 最小ランタイム |
| React ビルド | `node:20-slim` + `nginx:alpine` | SPA ビルド + Nginx 配信 |
| C# ビルド | `mcr.microsoft.com/dotnet/sdk:8.0`（既存） | 変更なし |
| Python ビルド | `python:3.12-slim`（既存） | 変更なし |
| Compose | docker-compose v2（`docker compose`） | Docker Desktop 標準、v1 は非推奨 |
| DB | PostgreSQL 16 | k1s0 標準 DB |
| Cache | Redis 7 | k1s0 標準キャッシュ |

### 設計方針

- **マルチステージビルド必須**: 全 Dockerfile でビルドステージとランタイムステージを分離
- **非 root ユーザー必須**: 全コンテナで `appuser` として実行
- **ヘルスチェック内蔵**: 各コンテナに `HEALTHCHECK` 命令を追加（ポート・パスは既存テンプレートと統一）
- **レイヤーキャッシュ最適化**: 依存ファイルを先にコピーし、ソースコードは後でコピー
- **環境変数ゼロ**: k1s0 の規約に従い、設定は `config/*.yaml` をコンテナにマウント
- **Tera テンプレート**: 既存テンプレートと同様に `.tera` 拡張子で変数埋め込み対応
- **ENTRYPOINT + CMD 分離**: ENTRYPOINT にバイナリ、CMD にデフォルト引数（`--env default --config /app/config`）を指定し、docker-compose で上書き可能にする
- **プロキシ対応**: 企業ネットワーク環境での `docker build` 時にプロキシ設定を利用可能にする（`--build-arg` での HTTP_PROXY/HTTPS_PROXY 伝播）
- **k1s0 ルートフォルダの保全**: `k1s0 docker` コマンドの中間ファイル（ビルドコンテキスト等）はシステムの temp ディレクトリを使用し、リポジトリルートを汚さない

## Dockerfile テンプレート仕様

### backend-rust

```dockerfile
# Stage 1: Build
FROM rust:1.85-slim AS build
WORKDIR /app
COPY Cargo.toml Cargo.lock ./
RUN mkdir src && echo "fn main() {}" > src/main.rs && cargo build --release && rm -rf src
COPY src/ src/
RUN cargo build --release

# Stage 2: Runtime
FROM debian:bookworm-slim AS runtime
RUN apt-get update && apt-get install -y --no-install-recommends ca-certificates && rm -rf /var/lib/apt/lists/*
RUN groupadd -r appuser && useradd -r -g appuser -s /sbin/nologin appuser
COPY --from=build /app/target/release/{feature_name} /app/{feature_name}
COPY config/ /app/config/
EXPOSE 8080
USER appuser
ENTRYPOINT ["/app/{feature_name}"]
CMD ["--env", "default", "--config", "/app/config"]
```

### backend-go

```dockerfile
# Stage 1: Build
FROM golang:1.22-alpine AS build
WORKDIR /app
COPY go.mod go.sum ./
RUN go mod download
COPY . .
RUN CGO_ENABLED=0 go build -o /app/server ./cmd/

# Stage 2: Runtime
FROM alpine:3.19 AS runtime
RUN apk add --no-cache ca-certificates
RUN addgroup -S appuser && adduser -S appuser -G appuser
COPY --from=build /app/server /app/server
COPY config/ /app/config/
EXPOSE 8080
USER appuser
ENTRYPOINT ["/app/server"]
CMD ["--env", "default", "--config", "/app/config"]
```

### frontend-react

```dockerfile
# Stage 1: Build
FROM node:20-slim AS build
WORKDIR /app
RUN corepack enable pnpm
COPY package.json pnpm-lock.yaml ./
RUN pnpm install --frozen-lockfile
COPY . .
RUN pnpm build

# Stage 2: Runtime
FROM nginx:alpine AS runtime
COPY --from=build /app/dist /usr/share/nginx/html
COPY deploy/nginx.conf /etc/nginx/conf.d/default.conf
EXPOSE 80
```

## docker-compose.yml テンプレート仕様

### バックエンドサービス共通

```yaml
services:
  app:
    build:
      context: .
      dockerfile: Dockerfile
    ports:
      - "8080:8080"
    command: ["--env", "dev", "--config", "/app/config"]
    volumes:
      - ./config:/app/config:ro
    depends_on:
      db:
        condition: service_healthy
      redis:
        condition: service_healthy

  db:
    image: postgres:16-alpine
    ports:
      - "5432:5432"
    environment:
      POSTGRES_DB: ${feature_name}
      POSTGRES_USER: postgres
      POSTGRES_PASSWORD_FILE: /run/secrets/db_password
    volumes:
      - db-data:/var/lib/postgresql/data
      - ./migrations:/docker-entrypoint-initdb.d
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U postgres"]
      interval: 5s
      timeout: 3s
      retries: 5
    secrets:
      - db_password

  redis:
    image: redis:7-alpine
    ports:
      - "6379:6379"
    healthcheck:
      test: ["CMD", "redis-cli", "ping"]
      interval: 5s
      timeout: 3s
      retries: 5

volumes:
  db-data:

secrets:
  db_password:
    file: ./secrets/db_password.txt
```

## 期待される効果

1. **開発開始の高速化**: `k1s0 new-feature` 後すぐに `docker compose up` で動作確認可能
2. **環境差異の排除**: 全開発者が同一のコンテナ環境で開発
3. **本番同等のローカル環境**: DB・Redis を含む完全なサービス構成をローカルで再現
4. **CI/CDとの一貫性**: ローカルと同じ Dockerfile を CI でも使用
5. **セキュリティ**: 非 root 実行、マルチステージビルド、シークレット管理の標準化

## 注意事項

- **環境変数禁止**: docker-compose.yml の `environment` は DB 初期化等の外部イメージ設定のみ許可。アプリ設定は `config/*.yaml` マウント
- **シークレット管理**: Docker secrets または `*_file` パターンを使用。平文の環境変数にパスワードを入れない
- **flutter は対象外**: モバイルアプリのため Docker 化の対象外
- **既存テンプレートとの互換性**: Dockerfile/docker-compose は追加ファイルのみ。既存のディレクトリ構造に変更なし
- **プロキシ環境**: `docker build` 時に `--build-arg HTTP_PROXY` / `--build-arg HTTPS_PROXY` を指定可能。Dockerfile 内で `ARG HTTP_PROXY` / `ARG HTTPS_PROXY` を宣言
- **temp ディレクトリの使用**: `k1s0 docker` コマンドが生成する一時ファイルはシステム temp ディレクトリ配下に作成し、リポジトリルートには一切ファイルを残さない
