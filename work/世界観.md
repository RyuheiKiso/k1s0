
# k1s0 が成す世界観（関係者向けPR資料）

## 0. ひとことで

k1s0 は、**テンプレ・CLI・共通 framework** をセットで提供し、マイクロサービス開発の「立ち上げ」「規約準拠」「契約管理」「アップグレード」を **人の記憶ではなく仕組みで固定**するためのモノレポ基盤です。

狙いは 1 つです。

- 「各サービスが頑張って同じことをやる」をやめ、
- 「頑張らなくても同じやり方で安全に動く」を既定にする

---

## 1. この世界観が解決したい“現場の痛み”

マイクロサービスが増えるほど、次の“地味な差”が積み上がり、速度と品質を奪います。

- サービスごとに構成・起動手順・設定の読み方が微妙に違い、初速が出ない
- 規約がドキュメント止まりで、レビューが主観と経験に依存する
- proto / OpenAPI の破壊的変更が混入し、下流が壊れてから発覚する
- secrets が YAML や環境変数に散らばり、事故・監査・ローカル再現が難しい
- 観測（ログ/トレース）がバラつき、障害対応が「まず形式理解」から始まる
- 共通基盤のアップグレードが怖くて止まり、分岐・フォーク・属人化する

k1s0 はこの負債を、テンプレと自動検査（lint/CI）と upgrade 支援で「増やしにくい形」にします。

---

## 2. k1s0 が実現する“当たり前”

### 2.1 新規サービス追加が「雛形生成」から始まる

- `k1s0 new-feature` で必要な置き場が揃う（設定/契約/デプロイ/README/構造）
- 生成直後に `k1s0 lint` が通る状態が既定

### 2.2 規約は「書く」ではなく「落とす」

- 規約逸脱はレビューではなく `k1s0 lint` / CI の必須チェックで落とす
- 失敗時は「原因/対象/次アクション」が必ず出る

### 2.3 契約は正本、生成物は再現性で守る（API First）

- gRPC: `proto/` を正本、`buf lint` / `buf breaking` を必須化
- REST: `openapi/openapi.yaml` を正本、差分検知を必須化
- 生成物は原則 Git 管理外でも、生成一致チェックでズレを検知する

### 2.4 secrets は環境変数ではなく「ファイル参照」を既定にする

- YAML/DB に機密値を直書きしない
- Kubernetes Secret 等を volume mount し、`*_file` 参照で読む

### 2.5 観測性をオプションにしない

- JSON 1 行ログの必須フィールド、トレース伝播、メトリクス命名を固定
- dev-check の合格条件に「観測が届く」を含める

### 2.6 アップグレードが「怖い作業」ではなく「差分が見える作業」になる

- `.k1s0/manifest.json` が生成元テンプレと更新境界（managed/protected）を保存
- `k1s0 upgrade --check` が影響範囲と衝突を一覧化
- `k1s0 upgrade` は安全に適用できる範囲だけを適用し、衝突は止まる

## 3. 世界の登場人物（責務の固定）

### 3.1 開発基盤チーム

目的: “個別チームが同じ事故を繰り返さない”ための既定（framework/テンプレ/CLI）を作る。

- `framework/`（共通 crate / 共通サービス雛形）
- `templates/`（生成されるサービスの型）
- `CLI/`（生成・lint・upgrade）
- CI の必須ゲート定義（lint/contract 等）

### 3.2 個別機能チーム

目的: “業務ロジック”に集中する。

- `feature/` 配下でサービスを実装（機能単位 = 1 マイクロサービス）
- 契約（proto/OpenAPI）互換性ルールを守る
- サービス間呼び出しで deadline 必須、retry 原則禁止を守る（例外は明示・可視化）

---

## 4. 3つの構成要素（セットで効く）

| 要素 | 役割 | 期待効果 |
| --- | --- | --- |
| templates | 置き場と最小構造を固定する | 初速・レビュー観点・CI対象が揃う |
| CLI | 人間の入口を固定する（手順をコマンド化） | 立ち上げ/更新が属人化しない |
| framework | 実装の作法を固定する（安全な既定を提供） | 通信/設定/観測/エラーが揃う |

---

## 5. アーキテクチャの世界観（方針を“運用品質”まで落とす）

### 5.1 マイクロサービス（API First）+ クリーンアーキテクチャ

- サービス粒度は機能単位（複数機能をまとめない）
- 層の依存方向を固定する（例: `presentation` → `application` → `domain`）
- `domain` は外部I/Oや特定フレームワークに依存しない

### 5.2 通信方式（固定）

- 内部 API: gRPC を基本
- 外部公開が必要な場合のみ REST（HTTP + OpenAPI）

### 5.3 サービス間通信の“実運用設計”を既定にする

方針だけで終わらせず、テンプレと framework で品質を固定します。

#### MUST（必須）

- すべての gRPC 呼び出しで deadline（timeout）必須（無制限禁止）
- retry 既定 0（原則禁止）。例外は冪等 RPC のみ、opt-in で明示
- OTel 伝播と計測の既定有効（trace/metrics/log を相関）
- Service discovery は K8s DNS + Service 名を前提に統一（環境差は config に閉じ込める）

#### 例外管理（“増やさない”設計）

- retry 等の例外は「opt-in + ADR + lint 検知」で管理し、例外が“見える”状態にする

---

## 6. 設定と秘密情報（暗黙を捨て、事故を減らす）

### 6.1 設定の優先順位（固定）

1. CLI 引数（参照先指定に限定: `--env` / `--config` / `--secrets-dir`）
2. YAML（`config/{env}.yaml`。非機密の静的設定）
3. DB（`fw_m_setting`。feature 固有の動的設定）

### 6.2 secrets の扱い（環境変数禁止の代替）

- secrets はファイルとして配布し、YAML/DB には参照のみを置く（例: `*_file`）
- Kubernetes では Secret を volume mount（例: `/var/run/secrets/k1s0/`）
- ConfigMap は `/etc/k1s0/config/{env}.yaml` に mount（値直書き禁止）

---

## 7. 観測性（運用が“見れば分かる”を既定にする）

### 7.1 標準出力（固定）

- ログ: JSON（1行1イベント）。必須: `timestamp`, `level`, `service_name`, `env`, `trace_id`, `span_id`, `message`
- メトリクス: OTel Metrics。命名: `k1s0.<service>.<component>.<metric>`
- トレース: W3C Trace Context を伝播し、外部依存（DB/Redis/他サービス）は span を切る

### 7.2 dev-check の合格条件（固定）

- ヘルス: HTTP なら `GET /healthz`、gRPC のみなら Health/Check
- 依存疎通: DB/Redis/config-service（利用する場合）
- 観測: 起動後に最低 1 件のトレース/メトリクスが Collector に到達

### 7.3 エラー規約（固定）

- 外部へ返す失敗は安定した `error_code` を必須化
- REST: problem details（RFC7807 互換）を基本とし、`trace_id` を必須化
- gRPC: canonical status code を使用し、`error_code` を metadata 等で付与
- 変換（内部分類 → `error_code` → 外部表現）は framework に集約し、サービス独自変換を持たない

---

## 8. 契約管理（API First を“運用”にする）

### 8.1 正本（source of truth）の置き場（固定）

- gRPC: `proto/`
- REST: `openapi/openapi.yaml`

### 8.2 CI の必須チェック（固定）

- gRPC: `buf lint` / `buf breaking`（比較対象は `main` または直近タグ）
- REST: OpenAPI 差分検知（破壊的変更は CI 失敗）
- 生成一致チェック: CI で生成し、ズレがあれば失敗

### 8.3 破壊的変更（MAJOR）の必須成果物

破壊的変更は原則禁止。例外的に MAJOR を行う場合は同一 PR で次を必須とする。

- ADR（なぜ必要か、代替案、影響範囲、移行計画）
- `UPGRADE.md`（影響一覧、移行手順、ロールバック）
- `k1s0 upgrade --check` で破壊箇所/衝突/移行対象を検知できること

---

## 9. ディレクトリ世界（置き場が固定され、迷子にならない）

### 9.1 リポジトリの三領域（固定）

- `framework/`: 共通化の唯一の置き場
- `feature/`: 個別機能チームの実装領域
- `CLI/`: 雛形生成・lint・upgrade

### 9.2 サービス粒度（固定）

- 機能単位 = 1 マイクロサービス
- 例: `feature/backend/rust/{feature_name}/`

### 9.3 禁止される共通化（固定）

- `feature/` 配下に `common/` を作らない（共通化は `framework/` へ移す）

---

## 10. アップグレードの世界観（止まらない基盤へ）

### 10.1 単一バージョン（リポジトリ全体で SemVer）

k1s0 はリポジトリ全体で単一バージョンを持ちます（CLI / generator / templates / framework すべてを 1 セットとして整合させる）。

### 10.2 upgrade を成立させる“境界”

- `.k1s0/manifest.json` を CLI が管理し、人手編集しない
- `managed_paths`: CLI が更新してよい領域
- `protected_paths`: CLI が触らない領域
- 衝突は検知して止まる（安全に止まるが既定）

---

## 11. ローカル開発の世界観（Windows/VSCode を前提に“差を消す”）

### 11.1 前提

- OS: Windows
- エディタ: VSCode

### 11.2 独立した開発環境（MUST）

各チームが追加作業なしで「依存起動・シード・疎通・観測」まで揃うことを、テンプレと framework で固定します。

- Dev Container + Docker Compose を標準
- 依存起動/停止/seed/check は `scripts/` のラッパーでワンコマンド化（方針）

---

## 12. 成功条件（何ができたら“世界が変わった”と言えるか）

- `k1s0 new-feature` で雛形サービスを生成でき、生成直後に `k1s0 lint` が通る
- 規約逸脱（必須ファイル削除、禁止事項、依存方向違反）が PR 時点で検知される
- 契約（buf/OpenAPI）に対して lint/breaking/差分検知の枠があり、CI で必須化される
- `.k1s0/manifest.json` により、upgrade が差分提示/限定適用できる
- 障害時に `error_code` と `trace_id` だけで一次切り分けができる

---

## 13. 実装ロードマップ（MVP → 拡張）

### 13.1 MVP（最短で“落とせる”ところまで）

1. CLI 骨格 + 出力/終了コード標準化 + ファイル操作ユーティリティ
2. backend テンプレ（置き場固定）
3. fingerprint/manifest 生成 + `init` / `new-feature`
4. `k1s0 lint`（必須構造/manifest）
5. framework 最小（config/error/observability）
6. CI 必須化（lint/contract）

### 13.2 拡張（運用品質と自動化を増やす）

- 禁止事項/依存方向の強化
- gRPC client/server ガードレール（deadline/metadata/計測/例外管理）
- 共通サービス（auth/config/endpoint）と DB 所有の確定
- 契約管理と生成一致の実運用レベル化
- upgrade の精度向上

## 15. 補足（誤解されやすい点）

- k1s0 は「特定ドメインに最適化された共通業務ロジック」ではない。共通化の対象は横断関心事と“事故が起きやすい型”である。
- 破壊的変更は原則禁止。必要な場合は段階移行が基本で、例外が MAJOR。
- ローカルは Mesh なしを既定にしつつ、同じコードが Kubernetes（Mesh あり/なし）へ持ち込める入口を framework に寄せる。
