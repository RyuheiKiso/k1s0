
# k1s0 世界観（構想から実現される“運用されている日常”）

この文書は、[work/構想.md](work/%E6%A7%8B%E6%83%B3.md) の方針が **実際の開発・運用の日常**として根付いたとき、チームやサービスがどんな空気で動くかを描く。
技術仕様の羅列ではなく、「その世界にいると何が当たり前で、何が起きなくなるか」を言語化する。

---

## 1. この世界の当たり前

### 1.1 立ち上げは“個人の工夫”ではなく“テンプレの責務”

新しいサービスを触るとき、最初に聞く言葉は「セットアップ手順は？」ではない。
代わりにこう言う。

- 「`dev-up` 叩けば動くよ」
- 「`dev-check` が通れば疎通も観測も OK」

開発者の初速は、環境差や手順の記憶量ではなく、テンプレと framework が肩代わりしている。

フロントエンド（React）も同じだ。

- UI は MUI を前提に揃っていて、見た目や部品の“選定会議”が起きにくい
- Header / Footer / Menu は最初から入っていて、「まずレイアウトを作る」時間が消える

### 1.2 規約は“ドキュメント”ではなく“落とす仕組み”

規約を守っているかは、レビューの主観ではなく `lint` と CI が決める。
規約違反は「指摘される」ではなく「マージできない」という形で自然に消える。

### 1.3 契約（proto / OpenAPI）は“正本”であり、コードは従う

API は「実装が先」ではない。
proto / OpenAPI が真実で、生成物は再現可能で、差分が出れば CI が止める。
破壊的変更は“気づかず入る”ことができない。

### 1.4 秘密は環境変数ではなく“ファイルとして存在する”

秘密情報は値そのものではなく「ファイルへの参照」として扱う。
設定 YAML にベタ書きしない。
環境変数で受け取らない。
秘密は、置き場と扱い方が最初から固定されている。

### 1.5 観測性は“追加オプション”ではなく“最低限の礼儀”

ログは JSON で、必要なフィールドが揃い、トレースと相関できる。
サービスが増えても、見方が増えない。
障害の調査は「このサービスはどうログ出すんだっけ？」から始まらない。

### 1.6 エラーは“返し方”ではなく“同じ言語”になる

この世界では、エラーはサービスごとに違う顔をしない。

- REST は problem details で返り、`error_code` と `trace_id` が必ず入っている
- gRPC は status code が揃い、同じ `error_code` がメタ情報として付いてくる

当番が見るのは「どのサービスのどの `error_code` か」だ。
“このサービスのエラー形式を覚える”が仕事から消える。

さらに重要なのは、REST/gRPC の変換がサービス実装ではなく framework 側で自動化されていることだ。
その結果、サービスが増えても「エラーの返し方が増える」が起きにくい。

### 1.7 DB は“つながる/つながらない”ではなく“遅い/壊れている”まで見える

DB へのアクセスは framework の標準入口を通る。
結果として、クエリ遅延や失敗は最初から計測され、トレースに現れる。

マイグレーションも同じ入口から起動でき、サービスごとに“やり方の違い”が生まれない。

「なんか遅い」は言わない。
「このクエリの p95 が跳ねてる」と言う。

### 1.8 認証/認可は“各サービスの工夫”ではなく“基盤の責務”

- JWT/OIDC の検証は共通部品を使う
- 権限判定は policy として評価される
- 監査ログ（誰が/いつ/何を/結果）が当たり前に残る

認可の事故は「実装漏れ」ではなく「例外の儀式（ADR）」として可視化される。

### 1.9 入力バリデーションは“曖昧さ”を残さない

API 境界のバリデーションは統一され、失敗は同じ形で返る。
“動くけどよく分からない”状態が減り、仕様と実装のズレが早く見つかる。

### 1.10 依存先呼び出しは雪崩れない（ガードレールが先にある）

timeout の上下限、同時実行制限、バルクヘッドが既定で効く。
リトライは既定で 0。

障害時に起きるのは「全体が遅延で溶ける」ではなく、
「守るべき境界で抑制が効いて、観測される」だ。

### 1.11 フロントは“配線だけ”ではなく“呼び出し方とUX”まで揃う

- ルート/メニュー/表示制御は設定で決まり、配線はノーコード
- API 呼び出しは共通クライアントに寄り、認証/timeout/計測が揃う
- problem details の標準表示により、`error_code` と `trace_id` が自然に見える
- テーマ、フォーム/エラー表示、通知/ダイアログが共通部品で揃う

その結果、「画面ごとに作法が違う」が起きにくい。

## 2. 登場人物と役割分担（チームの空気）

### 2.1 開発基盤チーム（framework / templates / CLI）

基盤チームの仕事は「便利機能を増やす」よりも先に、
“どのサービスでも同じやり方で安全に動く”という地面を固めることにある。

- 規約は README ではなくテンプレと CLI/CI に埋め込む
- 例外を作るなら、例外が“見える”入口を作る（ADR、allowlist、check の分岐）
- 互換性は精神論ではなく、差分検知と upgrade 支援で守る

基盤チームの成果は、各サービスのリポジトリに散らばらない。
「みんな頑張って同じことをやる」ではなく、「同じことを頑張らずに済む」を作る。

### 2.2 個別機能チーム（feature サービス）

機能チームが集中するのは業務ロジックだ。
インフラの細部、観測、契約管理の儀式、セットアップの罠を学ぶことは、仕事の中心ではない。

- 迷ったら `lint` が正解を示す
- 依存先との契約は proto/OpenAPI に従う
- 通信は deadline 必須、retry は原則禁止（例外は明示）
- secrets はファイル、設定は参照
- フロントの画面遷移/メニュー/表示制御は設定で決まり、配線に時間を溶かさない

この世界では、優秀さは「規約を暗記していること」ではなく、
規約に従ってもなお残る“意思決定”に集中できることとして現れる。

## 3. 主人公（開発者）の一日

### 朝：新しいサービスに触る

開発者は、別チームが作ったサービスを初めて触る。
README を眺めて手順の差分に怯える必要はない。

1. 依存を起動する（ワンコマンド）
2. マイグレーション・シードを当てる（ワンコマンド）
3. 疎通と観測の合格を確認する（ワンコマンド）

ここで「動かない理由」が出たとしても、出力は“原因/対象/次アクション”を最低限含む。
迷子にならない。

### 昼：機能追加をする

実装はクリーンアーキテクチャの層に沿って進む。
domain がインフラに引きずられそうになったら、コンパイルか lint が止める（止められる設計が選ばれている）。
結果として、レビューで “雰囲気 Clean Architecture” を議論する時間は減り、業務ロジックの議論が増える。

フロントの画面追加でも、同じ空気がある。

- 画面コンポーネントを実装し、画面IDを登録する
- ルート/メニュー/遷移（ウィザードの流れ等）は `config/{env}.yaml` を編集する

「Router どこ？ Menu どこ？ Layout どこ？」から始まらない。

### 夕方：他サービスとつなぐ

gRPC を呼ぶときは deadline が必須で、無制限呼び出しは入りにくい。
トレースは伝播し、失敗は観測され、どこで遅いかが追える。

依存先が不調でも、呼び出し側が巻き込まれて雪崩れない。
ガードレールが先にあり、抑制が効いたこと自体がメトリクスで見える。

「たまに遅いんだよね」ではなく、
「この RPC の p95 がこの条件で跳ねている」と話せる。

## 4. PR と CI の儀式（人間の善意に依存しない）

### 4.1 PR は“動く/動かない”ではなく“守る/守れない”を先に裁く

この世界の PR では、最初に問われるのは「動いた？」ではなく「規約を壊してない？」だ。
CI は次を最低限チェックし、失敗したらマージできない。

- `k1s0 lint`（または同等のラッパー）
- 契約 lint（buf/OpenAPI）
- 破壊的変更の検知（buf breaking / OpenAPI diff）
- 生成一致（生成物の差分が出ないこと）

その結果、レビューは“検出できる違反”を探す時間を失い、
設計と業務要件の議論へ時間を再配分できる。

### 4.2 レビューで消える会話

次のような会話は減る。

- 「このサービスのログ形式、他と違わない？」
- 「環境変数どれだっけ？」
- 「proto 変えたけど、下流大丈夫かな？」
- 「config に秘密が入ってない？」
- 「このエラー、どんな形式で返すの？」
- 「DB が遅いとき、どこを見るんだっけ？」
- 「認可って結局どこでやってる？」

なぜなら、それらはレビューの“指摘”ではなく CI の“拒否”に移るからだ。

## 5. 設定と秘密情報の“手触り”

### 5.1 設定は値ではなく“参照の束”として扱う

設定ファイル（ConfigMap 相当）には、機密値そのものが入らない。
入るのは参照（例: `*_file`）であり、実体は secrets ディレクトリにある。

これにより、次が同時に成立する。

- 設定レビューが安全になる（うっかり秘密をコミットしにくい）
- ローカルでも同じ形で動く（secrets を差し替えるだけ）
- “環境変数禁止”が現実的になる（代替が先に用意されている）

### 5.2 優先順位が固定され、揉めない

設定の優先順位は「人が決める」ではなく、世界のルールとして固定されている。

1. 起動引数（参照先指定）
2. YAML（非機密の静的設定）
3. DB（動的設定）

議論が必要なのは“何をどこに置くか”ではなく、
“その設定は静的か動的か、秘密か非秘密か”という本質だけになる。

---

## 6. 契約管理が“作業”ではなく“呼吸”になる

### 6.1 proto / OpenAPI が正本であることの効用

正本が決まっていると、実装は迷わない。
「コードが動いたからOK」ではなく「契約が正しいからOK」に移る。

### 6.2 破壊は“検知される”のではなく“意図しない限り起きない”

buf breaking や OpenAPI diff の価値は、
「壊れたときに気づける」だけではなく「壊すことを選べない」状態を作る点にある。

破壊的変更は、たいてい「急いでいる」「気づいてない」「後で直す」の混合から生まれる。
この世界では、それらが CI で物理的に止まる。

---

## 7. 通信のルールが“ライブラリ”として体に入る

### 7.1 deadline（timeout）は必須で、既定がある

gRPC を呼ぶたびに「timeout どうする？」と考えない。
既定があり、必要なら設定で上書きする。
その結果、無限待ちは“実装ミス”ではなく“できない状態”に近づく。

### 7.2 retry は原則禁止で、例外は明示

retry は正義ではない。
成功率を上げる代わりに、負荷と遅延と障害時の増幅を運んでくる。
この世界では、retry は既定で 0。
許すなら、冪等性・回数・バックオフ・ジッターが設計として説明される。

---

## 8. upgrade が怖くない（“安全に止まる”が先）

### 8.1 upgrade は上書きではなく“境界のある更新”

テンプレの更新は、どこでも上書きしていいわけではない。
世界は領域を分けて扱う。

- managed（CLI が更新してよい）
- protected（CLI が触らない）
- 境界（提示・衝突検知が中心）

この境界は運用で決め直すものではなく、manifest に保存される。
「たぶんここは上書きしても大丈夫」は許されない。

### 8.2 まず `--check` が成立し、変更は一覧で見える

適用より先に、影響範囲と衝突が列挙される。
その出力は、更新の“意思決定”のためにある。
変更の中身を当てずっぽうでレビューする文化は消える。

---

## 9. “起きなくなること”の一覧（この世界の効能）

- サービスごとに `config` の読み方が違って、引数や環境変数の解釈で事故る
- secrets が YAML やログに紛れて、あとから大掃除する
- gRPC の deadline を忘れて無限待ちし、障害時に雪崩れる
- 契約変更が気づかれず入り、下流が壊れてから発覚する
- 観測がサービスごとにバラバラで、障害対応が「まずログ形式の理解」から始まる
- 規約がレビューの“好み”になり、議論が再発する
- フロントで「ルーティング修正」「メニュー追加」「レイアウト調整」が毎回の儀式になって疲弊する

---

## 10. “それでも起きうること”と、世界の対処

### 10.1 破壊的変更（MAJOR）が必要になった

この世界では、破壊は例外であり、例外には儀式がある。

- ADR が必要（なぜ、代替案、影響、移行計画）
- UPGRADE 情報が必要（手順、ロールバック、影響一覧）
- `upgrade --check` で“壊れる場所”が列挙される

破壊は「勇気」ではなく「手順」によって管理される。

### 10.2 規約が現場に合わない

例外は “隠れる” のではなく “記録される”。
例外を許すなら、理由と代替案をドキュメントに残し、lint の allowlist の入口を明確にする。
例外は増やすためではなく、削るために可視化される。

### 10.3 開発環境が壊れた／遅い

この世界では「誰のマシンでは動いた」ではなく、
“同じ入口を叩けば同じ結果になる”ことが優先される。
原因の切り分けは、個人の頑張りではなく、入口（dev-up/dev-check）のログと観測で行う。
再現性がない問題は、まず入口の固定を疑う（入口を増やさない、分岐させない）。

---

## 11. この世界のデザイン原則（短い合言葉）

- **暗黙にしない**：`--env` / `--config` / `--secrets-dir` は明示する
- **値を持たない**：秘密は“参照”として扱う（`*_file`）
- **正本を決める**：契約は proto / OpenAPI、生成物は再現性で担保する
- **落とせる形にする**：規約は lint/CI で拒否できるところまで落とす
- **観測できる形にする**：失敗はログ/メトリクス/トレースに出る
- **壊すなら儀式**：MAJOR は ADR と UPGRADE と upgrade 支援が前提

## 12. 技術的な固定（“世界観”を支える仕様の骨）

ここから先は、世界観を「気分」ではなく「実装と運用で再現できるもの」にするための、最小の技術的な固定事項をまとめる。
詳細はテンプレ/CLI/規約ドキュメントに委譲するが、世界の前提として共有される。

### 12.1 ディレクトリ規約（置き場が固定される）

feature サービスの置き場は揺れない。

- backend: `feature/backend/rust/{feature_name}/`
- frontend: `feature/frontend/flutter/{feature_name}/`, `feature/frontend/react/{feature_name}/`

backend/rust サービスの最小構造は次を必須とする（lint 対象）。

```
src/
	application/
	domain/
	infrastructure/
	presentation/
config/
	default.yaml
	dev.yaml
	stg.yaml
	prod.yaml
deploy/
	base/
	overlays/
		dev/
		stg/
		prod/
proto/
openapi/
	openapi.yaml
```

### 12.2 設定/秘密情報の既定パス（環境変数の代替）

サービスは暗黙に環境を推測せず、起動引数で参照先を受け取る。

- `--env {dev|stg|prod}`
- `--config /etc/k1s0/config/{env}.yaml`（Kubernetes）
- `--secrets-dir /var/run/secrets/k1s0/`（Kubernetes）

ローカル開発では、同じ形をローカルパスに写像する（例: `./secrets/dev/`）。

YAML には機密の“値”を置かず、参照（例: `db.password_file`）のみを置く。

### 12.3 CLI の責務と I/F（人間向けの標準入口）

CLI は「生成」「初期化」「規約検査」「アップグレード支援」を担う。
世界の最低限の入口は次。

- `k1s0 init`
- `k1s0 new-feature {feature_name}`
- `k1s0 lint`
- `k1s0 upgrade --check`
- `k1s0 upgrade`

CLI の失敗は“何が悪いか”を必ず示す（原因/対象/次アクション）。
終了コードは、少なくとも「lint 違反」「ユーザー入力不備」「I/O 失敗」を区別できることが望ましい。

### 12.4 `.k1s0/manifest.json`（世界の境界を保存する）

テンプレ由来と更新境界は、プロジェクト側に必ず保存される。
最小キーは次（例）。

```json
{
	"k1s0_version": "vX.Y.Z",
	"template": {
		"name": "backend-rust",
		"version": "vX.Y.Z",
		"path": "templates/backend-rust/project",
		"fingerprint": "sha256:..."
	},
	"service": {
		"service_name": "{feature_name}",
		"language": "rust"
	},
	"managed_paths": ["deploy/", "config/", "proto/", "openapi/"],
	"protected_paths": ["src/domain/", "src/application/"],
	"update_policy": "suggest_only"
}
```

この manifest があるから、upgrade は「上書き」ではなく「境界のある更新」になる。

### 12.5 lint が見るもの（“世界を守る”最小チェック）

lint は「テンプレ規約との差分」と「禁止事項」を見る。
世界観に直結する最小セットは次。

- 必須ファイル/ディレクトリの存在（構造の固定）
- `.k1s0/manifest.json` の整合性（キー、境界、バージョン）
- 禁止事項（例）
	- アプリ実装での環境変数参照（例: Rust の `std::env`）
	- `config/*.yaml` への機密直書き（例: `password`, `token`, `secret` などの値）
	- domain 層からの不適切な依存（方針に応じて lint/コンパイルで防ぐ）

lint は“落とすだけ”ではなく、置換案や次アクションを伴うのがこの世界の作法。

### 12.6 観測性の最低限（ログ/トレースが揃う）

ログは 1 行 JSON を基本とし、最低限のフィールドを揃える（例）。

- `timestamp`, `level`, `service_name`, `env`, `trace_id`, `span_id`, `message`

サービス間呼び出しは trace context を伝播し、依存呼び出し（DB/他サービス）は span を切る。
観測は “入れてほしい” ではなく、テンプレと framework で “入っている状態” を既定にする。

### 12.7 契約管理の最低限（buf / OpenAPI diff）

契約の置き場とチェックは固定される。

- gRPC: `proto/` を正本にして `buf lint` / `buf breaking`
- REST: `openapi/openapi.yaml` を正本にして差分検知（破壊的変更は CI 失敗）
- 生成物は原則 Git 管理外だが、生成一致（再現性）は CI で担保する

### 12.8 scripts が提供する“入口”（Dev Container 外でも迷わない）

テンプレは、最低限の運用入口を `scripts/` に同梱する。
Windows/VSCode 前提の世界では、入口が PowerShell で揃っていることに意味がある。

- `scripts/dev-up.ps1`（依存起動）
- `scripts/dev-down.ps1`（依存停止）
- `scripts/dev-seed.ps1`（マイグレーション + 初期データ）
- `scripts/dev-check.ps1`（疎通 + 観測の合格判定）
- `scripts/lint.ps1`（`k1s0 lint` のラッパー）

これにより、Dev Container を使う/使わないの差分が「入口の違い」にならず、
チーム間のハンズオンコストを押し下げる。

### 12.9 生成物の扱い（Git 管理外だが、ズレは検知される）

生成物は原則 Git 管理外（`.gitignore`）とし、代わりに再現性の指標を Git 管理する。
例として、次のような置き方を許す。

- gRPC
	- 正本: `proto/`
	- 生成先（Git 管理外）: `gen/`
	- 再現性指標（Git 管理）: `gen/.k1s0-gen.sha256`
- OpenAPI
	- 正本: `openapi/openapi.yaml`
	- 生成先（Git 管理外）: `openapi/gen/`
	- 再現性指標（Git 管理）: `openapi/openapi.fingerprint`

CI は「生成を実行→指標を再計算→差分があれば失敗」を既定にできる。
生成物をコミットしない自由と、ズレを放置しない厳しさが両立する。

### 12.10 エラー/出力の最小規約（CLI もサービスも“原因が分かる”）

この世界では、失敗は必ず“扱える形”で出る。
人間向けの最小要素は次。

- 何が起きたか（要約）
- 何に対してか（対象パス、サービス名、ルールID等）
- 次に何をすべきか（推奨アクション）

将来 `--json` を導入しても破綻しないよう、内部的には「エラーコード（分類）」を持つのが望ましい。
例: `INVALID_ARGS`, `LINT_VIOLATION`, `IO_ERROR`, `TEMPLATE_CONFLICT`。

### 12.11 依存方向の強制（domain を汚さない）

依存方向は “レビューで注意する” では弱い。
世界が採る現実的な強制方法は次のいずれか（または併用）。

- コンパイルで強制（推奨）: crate 分割により domain から infra/framework を参照できない形にする
- lint で強制（MVP）: `src/domain/**` からの禁止 import を検出する（例: `tokio`, `sqlx`, `reqwest`, `tonic` 等）

どちらに寄せるかは実装フェーズで確定するが、
“強制される”こと自体が世界観の一部である。

## 付録: “世界観が守れている”サイン

この世界が実現しているかは、雰囲気ではなく兆候で分かる。

- 新規サービスの立ち上げで、手順質問がほぼ出ない
- 規約指摘がレビューではなく CI に集約される
- secrets がレビュー対象から自然に外れる（参照だけが見える）
- 契約変更が CI で止まり、下流破壊が事前に分かる
- 障害対応で「ログ形式が違う」が話題にならない
- upgrade の議論が「怖い」ではなく「差分はこれ、衝突はこれ」になる
