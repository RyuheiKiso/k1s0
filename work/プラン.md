

# k1s0 実装プラン（正式版 / フェーズ形式・細分化）

本ドキュメントは、[work/構想.md](work/%E6%A7%8B%E6%83%B3.md) の方針を「実装順序」と「成果物・完了条件」に落とした、k1s0 モノレポ（framework / templates / CLI / CI）の実装計画である。

重要: 本プランは **コード/テンプレ/CLI/CI 設定の作成に特化**し、環境構築（Dev Container / Docker Compose / 実クラスタ準備など）は対象外とする。

---

## フェーズ0: スコープと前提の固定

### 対象（このプランで作るもの）

- CLI（Rust）: `k1s0 init`, `k1s0 new-feature`, `k1s0 lint`, `k1s0 upgrade --check`, `k1s0 upgrade`
- templates: backend / frontend のサービス雛形、framework 共通サービス雛形
- framework: 共通 crate（config/observability/error/grpc-client など）と共通サービス（auth/config/endpoint）
- 規約 lint: manifest/必須ファイル/禁止事項/依存方向の逸脱検知
- 契約管理の枠: `proto/` + buf、`openapi/` と差分検知、生成一致チェックの枠
- CI 側の必須チェック定義（lint/contract を必須化）

### 対象外（このプランでは作らないもの）

- Dev Container / Docker Compose / ローカル依存起動スクリプト（`dev-up` 等）の整備
- Kubernetes クラスタ/namespace 払い出し等の運用
- Observability スタック（Collector/Trace UI/Grafana 等）の実環境構築

### 完了条件（全体）

- `k1s0 new-feature` で雛形サービスを生成でき、生成直後に `k1s0 lint` が通る
- 規約逸脱（必須ファイル削除、禁止事項、依存方向違反）が PR 時点で検知される
- 契約（buf/OpenAPI）に対して lint/breaking/差分検知の“枠”があり、CI で必須化できる
- `.k1s0/manifest.json` により、upgrade が安全に差分提示/限定適用できる

## フェーズ19.1: framework crate（http/gRPC error mapping の完成）

目的: エラー表現を「仕様」ではなく「実装」で統一し、運用で“見れば分かる”状態にする。

### 成果物

- REST: `application/problem+json`（RFC7807 互換）への変換ヘルパ
- gRPC: Canonical Status Code + `error_code` 付与（metadata を含む）の統一インターセプタ
- ログ/メトリクスへ `error.kind` / `error.code` / status code を必ず出す仕組み
- 「内部分類 → `error_code` →（REST/gRPC）外部表現」への変換を framework 側に集約し、サービスが独自変換を持たない方針をテンプレで固定

### 完了条件

- 典型エラー（入力不備/認証/認可/依存障害/一時障害）が、サービス間で同じ分類・同じ表現になる
- 運用が `error_code` と `trace_id` だけで一次判断できる（サービスごとの差が出ない）

## フェーズ21: framework crate（gRPC client）

目的: deadline 必須・retry 原則禁止などの通信ルールを API で固定する。

### 成果物

- 共通ビルダーでの client 初期化（各サービスが生の client/channel を直接組まない）
- deadline（timeout）必須化（デフォルト値 + 設定で上書き）
- timeout の下限/上限をバリデーションし、逸脱は起動時に失敗させる
- trace 伝播（OTel 計装の既定有効）
- metadata（例: trace/tenant）付与の自動化
- 共通 interceptors（trace context 伝播 / `error_code` 受け渡し / request_id 相関 / tenant 等）
- retry は既定 0（例外のみ opt-in）
- Service discovery の統一（K8s DNS 前提の論理名指定 + config で環境差吸収）
- “ローカルは Mesh なし”を既定にしても同じコードで成立する入口（h2c / http を許容し、Kubernetes では Mesh/Proxy で mTLS を担保）

### 完了条件

- “無制限呼び出し”が入りにくい API になる
- 生成直後のテンプレが「共通ビルダー経由」で必ず gRPC client を作る

## フェーズ21.3: gRPC リトライ例外管理（opt-in + lint + ADR 連携）

- CI で必須化できる出力（原因/対象/次アクション）

### 完了条件
