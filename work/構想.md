# 構想

---

## ゴール

- 高速な開発サイクルを実現するframeworkの構築
- 各機能をモジュール化し、再利用性を高める
- チームごとに独立した開発環境を提供し、開発効率を向上させる
- 最新技術を取り入れ、将来の拡張性を確保する
- 個別機能チームの実装負担を軽減し、ビジネスロジックに集中できるようにする

### 独立した開発環境（MUST）

“namespace で解決”だけでなく、各チームが追加作業なしで「依存起動・シード・疎通・観測」まで揃うことをテンプレと framework で固定する。

#### ローカル開発の標準（固定）

- **Dev Container + Docker Compose を標準**とする（Windows/VSCode 前提で環境差を最小化する）。
	- Dev Container を使えない場合のみ例外としてローカル実行を許可し、差分は `scripts/` のラッパーで吸収する。

#### 依存（DB/Redis/観測）起動と疎通（ワンコマンド・固定）

- テンプレは次を必ず同梱し、**1コマンドで起動できる**状態を既定とする。
	- 依存起動：`scripts/dev-up.ps1`
	- 依存停止：`scripts/dev-down.ps1`
	- マイグレーション + 初期データ投入：`scripts/dev-seed.ps1`
	- 疎通チェック（HTTP/gRPC/DB）：`scripts/dev-check.ps1`
- ローカルで起動する依存は最小構成を固定する（例：PostgreSQL / Redis / OTel Collector / Trace UI）。
- 各サービスは起動引数で `--env dev` と `--config` を明示し、暗黙選択しない。

#### 規約の自動検査（lint）（MUST）

規約を「ドキュメント」だけで終わらせず、テンプレ/CLI/CIで **自動的に逸脱を検知して落とす**。

- リポジトリおよび各サービスは、次のいずれか（または両方）を必ず提供する。
	- `k1s0 lint`（CLIコマンド）
	- `scripts/lint.ps1`（Windows向けラッパー。Dev Container 外でも動くこと）
- `lint` の検査対象（最低限）
	- ディレクトリ構造・必須ファイルの存在（テンプレ規約との差分）
		- 例：`config/{default,dev,stg,prod}.yaml` / `deploy/base` / `deploy/overlays/{dev,stg,prod}` / `src/{application,domain,infrastructure,presentation}`
	- `.k1s0/manifest.json` の整合性（存在・必須キー・managed/protected/suggest_onlyの妥当性）
	- 禁止事項の検出（例）
		- アプリ実装での環境変数参照（例：Rustの `std::env`、Goの `os.Getenv` 等）
		- Kubernetesマニフェストでの `envFrom` / `secretKeyRef` によるSecret注入（原則禁止）
		- ConfigMap（`config/{env}.yaml`）への機密値直書き（`*_file` 参照のみ許可）

#### CIでの必須化（固定）

- CIは `k1s0 lint`（または同等の `scripts/lint.ps1`）を必須チェックとして実行し、失敗時はマージ不可にする（人間レビュー依存を減らす）。

#### env 払い出し（Kubernetes・固定）

- **チームごとに dev 用 namespace を払い出す**（例：`k1s0-{team}-dev`）。
- `deploy/overlays/dev/` に namespace・ホスト名・依存先などの差分を閉じ込め、サービス名で名前解決する。
- **DB 分離**
	- 原則：マイクロサービスごとに DB を分離する（dev でも同じ）。
	- 共有 DB クラスタを使う場合でも、最低限「namespace（チーム）× service」で DB（または schema）を分離し、衝突しない命名規則を固定する。
- **Secret 配布（環境変数禁止の代替）**
	- Secret は Kubernetes Secret 等から volume mount し、`/var/run/secrets/k1s0/` 配下のファイルとして配布する。
	- ConfigMap は `/etc/k1s0/config/{env}.yaml` に mount し、YAML には `*_file`（参照）だけを置く。

## アーキテクチャ
- マイクロサービス(API First)
- クリーンアーキテクチャ

マイクロサービスの粒度は機能単位とする、複数機能をまとめることはない
例：ログイン機能、データベース機能、ログ機能

### 通信方式（MUST）

- **サービス間通信は gRPC を基本とする**（内部API）
- **外部システムへ情報提供する場合のみ REST API（HTTP + OpenAPI）を提供する**
	- 外部公開の要件が無いサービスは、原則 gRPC のみでよい

### サービス間通信の実運用設計（MUST）

gRPC を「方針」で終わらせず、サービス間通信の運用品質（可用性 / 安全性 / 観測性）をテンプレと framework で固定する。

#### 責務分担（固定）

- **開発基盤チーム（framework / テンプレの責務）**
	- **名前解決（Service Discovery）**
		- Kubernetes を前提とし、K8s DNS + Service 名で解決する（例：`{service_name}.{namespace}.svc.cluster.local`）。
		- アプリケーションが独自に Consul 等へ依存して解決しない（必要なら ADR で例外化）。
	- **mTLS（サービス間）**
		- Service Mesh で mTLS を強制し、証明書配布/ローテーションは Mesh 側に寄せる（アプリは証明書管理をしない）。
		- Mesh の選定候補：Istio / Linkerd（採用は別途 ADR で固定）。
	- **LB / レジリエンス**
		- 負荷分散・アウトライヤー検知・接続プール等は Mesh/Proxy 側を第一選択とする（全サービスで一貫させるため）。
		- リトライ/タイムアウトの標準値とルールをテンプレに同梱し、逸脱を検知できる形にする。
	- **観測性（必須）**
		- gRPC の分散トレース（TraceContext）を標準で伝播し、メトリクス/ログと相関できるようにする（OTel）。
		- 通信失敗（timeout / retry / circuit breaker）をメトリクスとして標準出力する。

- **個別機能チーム（各サービス実装の責務）**
	- gRPC 呼び出しは必ず deadline（timeout）を指定する（無制限禁止）。
	- リトライは冪等（idempotent）な RPC のみ許可し、RPC 単位で明示する（禁止が既定）。
	- 失敗時の扱い（フォールバック / エラー返却）は application 層のユースケースで定義する。
	- 依存先サービスの契約（proto）と互換性ルールを守る（`buf lint` / `buf breaking` を通す）。

#### リトライ/タイムアウト/サーキットブレーカのルール（固定）

- **Timeout（必須）**
	- すべての gRPC 呼び出しで deadline を設定する。
	- 既定値はテンプレで固定し、設定（YAML/DB設定）で上書き可能にする。
- **Retry（原則禁止・例外のみ）**
	- 既定は 0 回（retry なし）。
	- 例外として許可するのは読み取り系など冪等な RPC のみ。
	- 許可する場合も、指数バックオフ + ジッター、最大回数/最大待ち時間を必ず設定する。
- **Circuit Breaker（Mesh 優先）**
	- 可能な限り Mesh/Proxy 側のアウトライヤー検知（連続失敗時の一時遮断）を利用する。
	- アプリ側で実装が必要な場合は framework crate が提供する共通実装を利用し、サービスごとの独自実装は禁止（必要なら ADR）。

#### 実装の固定（テンプレ / framework が提供）

- framework crate に共通 gRPC クライアント初期化を用意する（例：コネクション / OTel / timeout の標準設定）。
- 依存先アドレス指定は「サービス名 + namespace」等の論理名を基本とし、環境差は config で吸収する。
- Mesh を使う場合、アプリの gRPC は原則「平文（h2c）+ sidecar で mTLS」等、運用方式をテンプレに明記して統一する（方式は ADR で確定）。

#### Service Mesh とローカル開発の境界（固定）

Service Mesh（Istio / Linkerd 等）は本番運用の品質に効く一方、ローカル（Docker Compose）では Mesh 無しが既定となるため、責務の境界を固定してブレを防ぐ。

- **ローカル開発（Docker Compose）**
	- Mesh なしを既定とする
	- gRPC/HTTP は TLS なし（h2c / http）を許容する（ローカルで証明書配布に依存しない）
	- 観測性（OTel）は **アプリ側で必ず有効** にし、Collector に送る（後述の標準出力を満たす）
	- レジリエンス（timeout / retry / circuit breaker）は **アプリ側のルール** を適用する（Mesh 依存を前提にしない）

- **Kubernetes（dev/stg/prod）**
	- Mesh 導入時は mTLS を Mesh 側で強制し、アプリは証明書管理をしない
	- LB/アウトライヤー検知/接続プール等は Mesh/Proxy 側を第一選択とする（全サービスで一貫）
	- アプリ側では、deadline（timeout）とトレース伝播（OTel）を必須とする（Mesh だけに寄せない）

- **例外（ADR 必須）**
	- ローカルで TLS/mTLS が必須な要件がある場合
	- アプリ側で独自の Service Discovery/リトライ制御等を持たせる必要がある場合

### 契約管理（MUST）

API First を成立させるため、gRPC / REST(OpenAPI) の「契約（スキーマ）」を正本として管理し、**後方互換性のルールと破壊検知をCIで強制**する。

#### 契約管理の実体（テンプレに組み込む・必須運用）

本方針を「口約束」にせず、各サービスのテンプレに次を **必ず同梱** する。

- 正本（source of truth）の置き場を固定する（サービス内）
	- gRPC：`proto/` 配下の `*.proto`
	- REST：`openapi/openapi.yaml`
- 生成物の置き場を固定する（サービス内）
	- gRPC：`gen/`（例：`gen/rust/`、`gen/ts/`）
	- REST：`openapi/gen/`（例：サーバスタブ/SDK/型定義）
	- 生成物は原則 Git 管理対象外（`.gitignore` に含める）。ただし「生成物の再現性検証」を CI で必須にする
- CIで差分検知を必須にする
	- gRPC：`buf lint` / `buf breaking`（比較対象は `main` or 直近タグ）
	- REST：openapi-diff 等で `main` or 直近タグとの差分比較（破壊的変更はCI失敗）
	- 生成一致チェック：CIで生成を実行し、生成物差分が出る場合は失敗（＝テンプレ/生成手順の再現性を担保）

テンプレは最低限、次のファイル/ディレクトリを持つ（言語・ツールは将来差し替え可能だが「置き場」は固定）。

- `proto/`（gRPC契約）
- `openapi/openapi.yaml`（REST契約）
- `openapi/gen/`（REST生成物の出力先。Git管理外）
- `gen/`（gRPC生成物の出力先。Git管理外）
- `buf.yaml` / `buf.lock`（gRPC契約検証・生成の基盤。サービス直下）
- `scripts/` または `Makefile` / `justfile` 等（契約lint/breaking/生成を1コマンド化。OS差はラッパーで吸収）

#### 破壊的変更の例外（MAJOR）時に必須の成果物

破壊的変更が必要な場合（例外としてMAJORで実施）は、テンプレ/運用として次を必須とする。

- `docs/adr/` に ADR を追加（なぜ必要か、代替案、影響範囲、段階移行 or 一括移行の計画）
- `UPGRADE.md`（またはリリースノート）を追加
	- 影響一覧（API/DB/設定/テンプレ）
	- 移行手順（手動手順/ロールバック手順）
	- CLIで自動/半自動化できる範囲
- CLIにアップグレード支援を追加
	- `k1s0 upgrade --check` が検知できること（破壊箇所、衝突、移行対象）
	- `k1s0 upgrade` が安全に止まれること（中途半端に壊さない）

#### gRPC（Protocol Buffers）

- 正本は各サービスの `proto/` 配下の `*.proto` とする（ソースコードではなくスキーマを契約とみなす）
- 後方互換性ルール（原則）
	- フィールド追加はOK（既存フィールドの意味を変えない）
	- フィールド削除・再採番・型変更はNG
	- 使わなくなったフィールドは削除せず `reserved`（フィールド番号とフィールド名の両方）を必ず宣言する
	- `oneof` の破壊的変更（既存ケースの削除/意味変更）はNG
	- サービス名/メソッド名/パッケージ名の変更は原則NG（必要なら段階移行：新規追加→移行→切替→削除）
	- 非推奨化は `deprecated = true` を用い、削除は原則行わない（例外はMAJOR）
- 破壊検知（CI）
	- PRでは `buf lint` を必須とする
	- PRでは `buf breaking` を必須とする（比較対象は `main` もしくは直近リリースタグ）
	- 破壊が検出された場合はCIを失敗させ、例外としてMAJOR変更を行う場合は ADR / UPGRADE 手順 / 移行計画 を同一PRで必須とする

#### REST（OpenAPI）

- 正本は各サービスの `openapi/openapi.yaml` とする
- 生成物の扱い
	- `openapi/openapi.yaml` は手書き（契約の正本）とし、生成物は `openapi/gen/` などに出力して Git 管理対象外（または最小限）とする
	- SDK/クライアント生成を行う場合でも、生成物のレビューは「元のOpenAPI差分」を中心に行う
- 後方互換性ルール（原則）
	- エンドポイント/operation の削除・パス変更・HTTPメソッド変更はNG
	- リクエスト/レスポンスからのフィールド削除・型変更はNG
	- 任意→必須（required追加）、バリデーション強化（許容値の縮小）は原則NG（段階移行で実施）
	- フィールド追加はOK（任意のみ）。必須フィールド追加は原則NG
- 破壊検知（CI）
	- PRでは openapi-diff 等を用いて `main`（または直近タグ）との差分を比較し、破壊的変更があればCIを失敗させる
	- 例外としてMAJOR変更を行う場合は、gRPCと同様に ADR / UPGRADE 手順 / 移行計画 を同一PRで必須とする

## frameworkのアップグレード支援（互換性ポリシー）

CLIが「自動アップグレード支援」を成立させるため、互換性の定義と破壊的変更の扱いを固定する。

### バージョンの単位

- k1s0 はリポジトリ全体で単一バージョン（SemVer）を持つ（例：`v1.2.3`）
	- 対象：CLI / generator / templates / framework crates / framework共通サービス雛形
	- 目的：導入・アップグレード時に「整合する組み合わせ」を常に1つにする

### SemVer（互換性）の定義

- **MAJOR**：破壊的変更（後方互換なし）
	- 例：公開APIの破壊（HTTP/gRPC/GraphQL の契約変更、主要なレスポンス形/必須フィールド変更）
	- 例：framework crates の公開API破壊（公開関数/型/featureの削除、互換性のない挙動変更）
	- 例：テンプレの構造的変更（移動/削除により既存サービスがビルド不能になる等）
	- 例：DBの破壊（列削除・型変更など「即時互換性を失う」変更）
- **MINOR**：後方互換ありの機能追加
	- 例：任意フィールド追加、エンドポイント追加、非推奨化（deprecate）の導入
- **PATCH**：後方互換ありの不具合修正/小改善
	- 例：バグ修正、依存更新（互換性維持）、テンプレのコメント/軽微な安全修正

### 破壊的変更の扱い（必須運用）

- 破壊的変更は原則禁止。必要な場合は **段階移行（追加→移行→切替→削除）** を基本とし、例外のみMAJORで実施する
- MAJOR を行う場合、同一PRで次を必須とする
	- ADR（なぜ必要か、代替案、影響範囲、移行計画）
	- `UPGRADE.md` もしくはリリースノート（対応手順、ロールバック手順、影響一覧）
	- 自動/半自動アップグレード手段（CLIのパッチ適用、変換スクリプト、チェックコマンドの追加等）

### テンプレ差分適用の戦略（CLIの責務）

CLIはアップグレード時に「上書きしてよい領域」と「チームの業務コード領域」を分離して扱う。

- **原則：管理対象（CLIが更新する）**
	- `deploy/`（Kustomize）
	- `config/`（環境別yaml）
	- `openapi/`（雛形・共通ルールに関する部分）
	- `migrations/`（生成/追加のみ。既存の適用済みファイルは改変しない）
	- ビルド/静的解析/CIに関する共通設定（例：rootの共通ファイル、サービスの定型設定）
- **原則：非管理対象（CLIは変更しない）**
	- `src/domain/` / `src/application/`（業務ロジックの中心）
- **境界（衝突しやすいので要注意）**
	- `src/main.rs` / `src/presentation/` / `src/infrastructure/` は配線・I/Oが集中するため、CLIは「更新候補の提示（diff出力）」を基本とし、衝突時は手動解決とする

差分適用方式は「テンプレの旧版→新版差分」をプロジェクトへ当てる方針とし、衝突時は

- `k1s0 upgrade --check`：影響箇所と衝突を事前検出
- `k1s0 upgrade`：安全に適用できる変更のみ自動適用し、衝突は明示して停止（中途半端に壊さない）

### upgrade成立のためのメタ情報（プロジェクト側に保存・必須運用）

CLIが「テンプレの差分」を安全に適用するため、プロジェクト側に **メタ情報を必ず保存** する。

- 保存先：`.k1s0/manifest.json`（CLI管理。人手編集しない）
- 目的
	- そのプロジェクトが「どのテンプレ/どのバージョン」から生成されたかを確定する
	- CLIが「管理対象ファイル」を特定し、上書き可否/差分適用/衝突検知を機械的に行えるようにする
	- upgrade時に「どの生成手順（契約生成/CI設定等）を期待するか」を固定する

#### `.k1s0/manifest.json` の内容（案）

最低限、次の情報を持つ。

- `k1s0_version`：リポジトリ全体のバージョン（例：`1.2.3`）
- `template`
	- `name`：テンプレ名（例：`backend-rust/feature`）
	- `version`：テンプレのバージョン（原則 `k1s0_version` と一致）
	- `source`：テンプレの出自（例：`RyuheiKiso/k1s0`、またはテンプレ配布元URL/registry名）
	- `path`：テンプレのパス（例：`CLI/templates/backend-rust/feature`）
	- `revision`：テンプレ内容を特定できるリビジョン（例：git commit SHA / tag）。`version` と一致しない場合がある（ホットフィックス等）
	- `fingerprint`：テンプレ内容の指紋（例：テンプレディレクトリを正規化して sha256）。`revision` が無い配布形態でも「旧テンプレ」を一意に特定できるようにする
- `generated_at`：生成日時（監査用。挙動には使わない）
- `service`
	- `service_name`：サービス名（= feature_name）
	- `language`：`rust` / `go` 等
- `managed_paths`：CLIが管理する領域（glob可）
	- 例：`deploy/**`, `config/**`, `openapi/openapi.yaml`, `proto/**`, `scripts/**`, `.github/workflows/**`
- `protected_paths`：CLIが変更しない領域（チェックに使う）
	- 例：`src/domain/**`, `src/application/**`
- `update_policy`：衝突時の方針（自動適用/提示/停止）
	- 例：`src/main.rs` / `src/presentation/**` / `src/infrastructure/**` は `suggest_only`
- `checksums`（任意だが推奨）
	- 管理対象ファイルのハッシュ（例：sha256）を記録し、ユーザー改変を検知できるようにする
	- 改変検知は `k1s0 upgrade --check` で警告/停止に利用する

（任意だが推奨）「旧テンプレ→新テンプレ差分」を安定して当てるための追加メタ情報

- `template_snapshot`（任意だが推奨）
	- 生成時点のテンプレ（またはレンダリング後の管理対象ファイル一式）を `.k1s0/template_snapshot/` 等に保存する方式
	- 差分適用の基準点（old）を確実に再現できるため、`revision` や `fingerprint` だけでは不足する配布形態でも upgrade の成功率が上がる
	- 容量増を避けたい場合は「スナップショットは保存せず `revision` + `fingerprint` を必須」にする、または「最新N世代だけ保持」等のポリシーを設ける

#### CLIの挙動（メタ情報をどう使うか）

- `k1s0 init` / `k1s0 new-feature`：テンプレ展開後に `.k1s0/manifest.json` を生成
- `k1s0 upgrade --check`
	- `manifest.json` の `template.version`（および `template.revision` / `template.fingerprint`）と現在の `k1s0` の差を計算
	- 旧テンプレの特定に必要な情報が不足している場合は「自動差分適用は不可」として停止し、移行手順（例：snapshot再生成、手動upgrade）を案内する
	- `managed_paths` の改変有無（`checksums`）を検知し、衝突リスクを事前提示
	- 破壊的変更（MAJOR）対象なら `ADR` / `UPGRADE.md` の存在をチェック
- `k1s0 upgrade`
	- 旧テンプレ（`template_snapshot` または `template.revision` / `template.fingerprint` で特定）→新テンプレの差分を作り、`managed_paths` のみを安全に更新
	- `suggest_only` は diff を提示して停止（自動で壊さない）
	- `protected_paths` は原則触らない（差分検知のみ）

### マイグレーション自動適用の可否

- **dev**：任意で自動適用を許可（例：`k1s0 upgrade --apply-migrations`）
- **stg/prod**：自動適用は禁止（生成のみ）。適用は運用手順（CI/CDや手動承認）で行う
- 自動適用の有無に関わらず、マイグレーションファイルは差分として追加し、ロールバック方針をREADMEに必ず記載する

## frameworkが提供する設定読み込み機能

frameworkで設定読み込み機能を提供する。(configモジュールとconfigサービス)
`framework\database\table\fw_m_setting.sql`

framework自身の設定はyamlファイルで制御するが、feature固有の設定値はDBで管理する。
環境変数は使用しない。

### 設定の優先順位と耐障害（固定）

YAML（ConfigMap 等で配布するファイル設定）と DB（`fw_m_setting`）を併用する場合、優先順位と障害時挙動を固定し、サービスごとに解釈差が出ないようにする。

#### 優先順位（override 順）（固定）

同一キーが複数ソースで定義される場合、**後勝ち** とし、優先順位は次の順で固定する。

1. **CLI 引数**（例：`--config` / `--env` / `--secrets-dir` など。設定の「参照先」を指定する用途に限定）
2. **YAML**（`config/{env}.yaml`。非機密の静的設定）
3. **DB（`fw_m_setting`）**（feature 固有の動的設定）

※ 原則として「機密値そのもの」を YAML/DB へ置かず、参照（`*_file` / secret_ref）に留める。

#### config-service 障害時の挙動（固定）

feature 固有設定（DB/設定サービス）が取得できない場合の挙動を固定し、ローカル/本番で差が出ないようにする。

- **起動時**
	- 取得に失敗した場合は、次のいずれかを設定単位で選べるようにする（既定は A）。
		- A: **キャッシュがあれば起動可**（キャッシュなしなら起動不可）
		- B: **フェイルオープン**（DB設定がなくても起動し、YAML 既定値で動作）
		- C: **起動不可**（設定取得が必須なサービス/機能に適用）

- **稼働中**
	- 取得失敗時は直前のキャッシュ（メモリ/ローカル永続）を使い、一定時間後にリトライする
	- キャッシュの TTL と最大保持世代数は framework で既定値を持ち、YAML/DB 設定で上書き可能にする
	- 失敗時は必ずメトリクス/ログ/トレースで観測できるようにする（後述）

#### setting のスキーマ進化（固定）

`fw_m_setting` のキー/型はサービスの公開 API と同等に扱い、破壊的変更を最小化する。

- **キー変更・廃止**
	- 既存キーのリネームは原則禁止（互換層を設ける）。必要な場合は「新キー追加 → 移行期間 → 旧キー廃止」の段階移行とする
	- 廃止は `status=0`（無効）等で段階的に行い、即時削除しない
	- 廃止時は `README.md` の設定一覧を更新し、移行手順を明記する

- **型変更**
	- 既存キーの `value_type` 変更は原則禁止
	- 変更が必要なら新キーを追加し、旧キーは互換のために一定期間維持する

- **デフォルト値**
	- DB 設定が存在しない場合の既定値は YAML もしくはアプリ側の既定（framework で固定）とし、どちらが正かを README で明示する

### 秘密情報の扱い（環境変数禁止の代替）

方針：**秘密情報（DBパスワード、APIキー、JWT秘密鍵等）は「ファイルとして配布」し、YAML/DBには値そのものを置かず「参照（ファイルパス/参照名）」のみを保持する。**

#### MUST（必須ルール）

- ConfigMap（`config/{env}.yaml`）に秘密情報を直接書かない
- 秘密情報は Kubernetes Secret 等から **volume mount** してコンテナ内ファイルとして配布する
- アプリは秘密情報を **環境変数で受け取らない**（envFrom/secretKeyRefを使わない）
- 秘密情報の既定の配置先を固定する（例：`/var/run/secrets/k1s0/`）
- YAML には「*_file」キー（ファイルパス）を置き、値は起動時にファイルから読み込む

#### 推奨（Git管理する場合）

- 秘密情報をGitに置く必要がある場合は **SOPS（age/GPG）等で暗号化したファイル**として管理し、CI/CDで復号してK8s Secretを生成する
	- 例：`secrets/{service_name}/{env}.enc.yaml`（暗号化） → デプロイ時に復号 → Secret作成
- Gitに置かない運用の場合は、External Secrets Operator / Secret Store CSI Driver 等を使い、クラウドのSecret Managerから同期しvolume mountする

#### K8sでの標準配布方式（例）

- `config/{env}.yaml`：ConfigMapとして配布（非機密）
- `secret/`：Kubernetes Secretとして配布（機密）
- Podでは次を行う
	- ConfigMapを `/etc/k1s0/config/{env}.yaml` にmount
	- Secretを `/var/run/secrets/k1s0/` にmount
	- 起動引数で `--env {env}` と `--config /etc/k1s0/config/{env}.yaml` を渡す（暗黙選択しない）

#### ローカル開発での標準配布方式（例）

- `config/{env}.yaml` をそのまま参照（非機密）
- 機密値は `secrets/.gitignore` 配下に置き、起動時に `--secrets-dir ./secrets/{env}` のようにディレクトリを指定する
	- 例：`secrets/dev/db_password`, `secrets/dev/jwt_private_key.pem`

#### YAMLキー例（値を置かず参照のみ）

- DB接続
	- `db.host`, `db.port`, `db.name`, `db.user`：非機密（YAML）
	- `db.password_file`：機密（Secretファイルのパス）
- JWT
	- `auth.jwt_private_key_file` / `auth.jwt_public_key_file`
- 外部API
	- `integrations.some_api.token_file`

※ feature固有の設定値（`fw_m_setting`）も同様に、機密値は原則「参照（*_file / secret_ref）」で持たせ、値そのものを格納しない（必要ならADRで例外化）。

## 観測性（標準出力・疎通チェック）（MUST）

OTel Collector/Trace UI の起動を「用意した」で終わらせず、各サービスが必ず満たすべき出力と、`dev-check` の合格条件を固定する。

### 標準出力（固定）

- **ログ**
	- 形式：JSON（1行1イベント）を既定とする
	- 必須フィールド：`timestamp`, `level`, `service_name`, `env`, `trace_id`, `span_id`, `message`
	- 追加推奨：`request_id`（外部リクエスト単位）, `grpc.method` / `http.route`, `error.kind`

- **メトリクス**
	- エクスポート：OTel Metrics（Collector 経由）を既定とする
	- 命名規約：`k1s0.<service>.<component>.<metric>` を基本形にする
	- 必須メトリクス（最低限）：
		- リクエスト数/失敗数（HTTP/gRPC）
		- レイテンシ（p50/p95/p99 を算出可能な histogram）
		- 外部依存（DB/Redis/他サービス）呼び出しの失敗数/レイテンシ
		- 設定取得（config-service/DB）の成功/失敗数

- **トレース**
	- 伝播：W3C Trace Context を既定とし、サービス間で必ず伝播する
	- 必須属性（最低限）：`service.name`, `deployment.environment`, `rpc.system`/`rpc.method` もしくは `http.method`/`url.path`
	- 外部依存呼び出し（DB/Redis/他サービス）は span を切る（相関できること）

### 疎通チェック（`scripts/dev-check.ps1`）の合格条件（固定）

テンプレの疎通は「起動できた」ではなく「開発が始められる状態」を確認する。

- **プロセス/ヘルス**
	- HTTP を提供する場合：`GET /healthz` が 200 を返す
	- gRPC のみの場合：`grpc.health.v1.Health/Check` が `SERVING` を返す

- **依存疎通**
	- DB：マイグレーション適用済みで、簡単な read（例：`SELECT 1`）が成功する
	- Redis：ping 相当が成功する（利用するサービスのみ）
	- config-service を利用する場合：設定取得の最小 API が成功する（キャッシュ運用の場合はその状態も出力）

- **観測**
	- 起動後に少なくとも 1 件のトレース/メトリクスが Collector に到達していること（Collector の受信を簡易に確認できること）

## ディレクトリ構成図

### 方針

#### 目的

- リポジトリ内の「どこに何を書くべきか」を固定し、読み手/書き手の解釈差をなくす
- 生成（CLI）・共通化（framework）・個別開発（feature）を衝突なく並行させる

#### MUST（必須ルール）

- monorepo とし、責務ごとに次の3領域へ分離する
	- `framework/`：開発基盤チームが提供する共通部品（再利用のための唯一の置き場）
	- `feature/`：個別機能チームのサービス実装（機能単位=マイクロサービス）
	- `CLI/`：雛形生成・導入・アップグレード

- サービス粒度
	- **機能単位=1マイクロサービス**（例：ログイン機能、ログ機能、DB機能）
	- 1ディレクトリ=1サービス（例：`feature/backend/rust/{feature_name}/`）
	- サービス間の“共通化”は `feature/` 配下で行わない。共通にしたいものは `framework/` へ移す

- frameworkが提供する共通サービス
	- frameworkの `auth-service` / `config-service` / `endpoint-service` は **共通マイクロサービス**であり、`feature` 配下のサービスと同じ運用単位（ビルド/デプロイ/監視/DB）として扱う
	- 共通サービスの置き場所は固定：`framework/backend/{lang}/services/{service_name}/`
	- `{service_name}` は kebab-case かつ `-service` で終える（例：`auth-service`）
	- **ディレクトリ/ファイル構成は固定**（担当者の裁量で増減しない）
		- 必須：`Cargo.toml`, `README.md`, `config/`, `deploy/`, `src/`
		- 必須：`config/{default,dev,stg,prod}.yaml`
		- 必須：`deploy/base/` と `deploy/overlays/{dev,stg,prod}/`
		- 必須：`src/` は Clean Architecture 構成（`application/`,`domain/`,`infrastructure/`,`presentation/`）
		- 条件付き必須：公開APIを持つ場合は次を必須とする
			- REST(HTTP) を提供する場合：`openapi/openapi.yaml`
			- gRPC を提供する場合：`proto/`（`*.proto`）

- framework共通DBの責務（所有者を固定）
	- framework共通テーブルは **所有サービスを固定**し、所有サービス以外はスキーマ変更を行わない
		- `auth-service` 所有：`fw_m_user`, `fw_m_role`, `fw_m_permission`, `fw_m_user_role`, `fw_m_role_permission`
		- `config-service` 所有：`fw_m_setting`
		- `endpoint-service` 所有：`fw_m_endpoint`
	- DDLの正本は `framework/database/table/*.sql` とする（設計・レビュー対象）
	- 適用用マイグレーションの正本は各サービスの `framework/backend/{lang}/services/{service_name}/migrations/` とする（実行・運用対象）
	- テーブル定義を変更する場合は **必ず同一PRで** 次を同時に更新する
		- `framework/database/table/<table>.sql`（最新の完全定義）
		- 所有サービスの `migrations/`（差分として適用可能なSQL）
		- 所有サービスの `README.md`（変更点・影響・ロールバック方針）
	- `migrations/` の命名規則は固定：`0001_*.sql`（4桁連番）
	- 互換性ルールは固定：本番適用後の破壊的変更（列削除・型変更等）は原則禁止。必要なら「段階移行（追加→移行→切替→削除）」とし、ADRで例外承認する

- サービス `README.md` 規約（全サービス共通）
	- 対象：`feature/backend/*/{feature_name}/` および `framework/backend/*/services/{service_name}/` および BFF（存在する場合）
	- `README.md` は必ず次の見出しをこの順番で持つ（見出し名も固定）
		- `# {service_name}`
		- `## 概要`（何のサービスか/何をしないか）
		- `## 責務`（箇条書き3〜7個）
		- `## 公開API`（gRPC/REST(HTTP) のいずれか。proto/OpenAPI への参照を含む）
		- `## 依存`（利用するframework crate/外部サービス: DB/Redis/Queue等）
		- `## 設定`（YAMLキー一覧 + DB設定キー一覧。キー、型、デフォルト、説明）
		- `## DB`（所有テーブル/参照テーブル/マイグレーション運用/ロールバック方針）
		- `## 認証・認可`（必要な場合：要求トークン、要求権限キー、認可方式）
		- `## 監視`（メトリクス/ログ/トレースの出し方、重要アラート条件）
		- `## 起動方法`（ローカル起動手順、`--env` の指定例、依存の起動順）
		- `## リリース`（バージョニング方針、互換性、破壊的変更手順）
	- `公開API` が無いサービスは作らない（内部サービス化が必要ならBFF/featureの責務を見直す）。例外はADRに記録する

- 命名規則（固定）
	- `{feature_name}` は **kebab-case**（例：`auth` / `user` / `log`）
	- `{service_name}` は **{feature_name}と同一**（DB設定スコープ・監視名・権限スコープと一致させるため）
	- `{env}` は **固定の4値のみ**：`default` / `dev` / `stg` / `prod`

- Clean Architecture（依存方向を固定）
	- `presentation` → `application` → `domain`
	- `infrastructure` は外部I/O（DB/HTTP/Cache/Queue/OTel等）の実装置き場
	- `domain` は原則として外部フレームワーク/DB/HTTPに依存しない

- 層ごとの「置くもの」を固定（これ以外に業務コードを置かない）
	- `src/domain/`：Entity / ValueObject / DomainError / Repository traits（ports）
	- `src/application/`：Usecase（入力→業務→出力）とApplication Service
	- `src/infrastructure/`：Repository実装、DB/Cache/Queue/Config/Logging/Telemetry
	- `src/presentation/`：HTTP/gRPC/WebSocket のルーティング、DTO、バリデーション、エラー→HTTP変換

- 型の流出ルール（層またぎのデータ型を固定）
	- `domain` 型（Entity/VO）を `presentation` のレスポンスDTOとして直接返さない
	- `infrastructure` のDB行モデル（Row/Record）を `domain` に漏らさない
	- DTO ↔ Usecase入出力 ↔ Domain の変換は **presentation側**（I/O境界）で行う

- 設定ルール（環境変数は使わない）
	- framework自身の設定：`config/{env}.yaml`（K8s ConfigMap等で配布）
	- feature固有の設定：DB（`fw_m_setting`）
	- アプリ起動時に `--env {env}` のように環境を明示し、`config/{env}.yaml` を選択する（暗黙選択しない）
	- `fw_m_setting.setting_key` 命名：`category.name`（ドット区切り、小文字+数字+アンダースコアのみ）
		- 例：`http.timeout_ms` / `db.pool_size` / `auth.jwt_ttl_sec` / `feature.flag_x`

- APIドキュメント/デプロイ配置（固定）
	- APIドキュメント：各サービスの `openapi/`（REST）および `proto/`（gRPC）配下に集約
	- デプロイ：各サービスの `deploy/` 配下に集約（Kustomize: `base/` + `overlays/{env}/`）

#### SHOULD（推奨ルール）

- `docs/` には確定した規約/運用/ADRを置き、検討中の草案は `work/` に置く
- `openapi/` は手書きと生成物を分ける（例：手書き `openapi/openapi.yaml`、生成 `openapi/gen/`）
- `migrations/` は連番SQL（例：`0001_init.sql`）で管理し、適用ツールは各言語標準に合わせる

#### 禁止事項

- `feature/` 配下に `common/` を作るなど、共通化を “各チームの裁量” にしない
- `domain` から `infrastructure` を直接 import しない
- 設定の読み替えや上書きに環境変数を利用する実装を入れない（必要ならADRで例外化）
