# カオスエンジニアリング設計

k1s0 におけるカオスエンジニアリングの全体方針・障害注入・自動回復・Game Day 運用の仕様を定義する。
Tier アーキテクチャの詳細は [tier-architecture.md](../overview/tier-architecture.md) を参照。

## 基本方針

- **Netflix Principles of Chaos Engineering** に基づき、本番環境の信頼性を実験的に検証する
- "信頼性は証明するもの、仮定するものではない" を基本哲学とする
- k1s0 の全 Tier（system / business / service）を対象とし、Tier ごとに適切な Blast Radius を設定する
- **Steady State → Hypothesis → Experiment → Observe → Learn** のサイクルを継続的に回す
- カオス実験は Feature Flag（k1s0-featureflag）と連動し、即時停止可能な安全設計とする
- 可観測性基盤（Prometheus / Grafana / Loki / Jaeger）と統合し、実験結果を定量的に評価する

## D-200: カオスエンジニアリング全体設計

### アーキテクチャ概要

![カオスエンジニアリングアーキテクチャ概要](images/chaos-engineering-overview.svg)

カオスエンジニアリング基盤は以下のコンポーネントで構成される。

| コンポーネント | 役割 | デプロイ先 Namespace |
| --- | --- | --- |
| Litmus ChaosCenter | カオス実験の管理・スケジュール・ダッシュボード | `chaos-engineering` |
| Chaos Mesh | 補完的な障害注入エンジン（IO Chaos・DNS Chaos） | `chaos-engineering` |
| Recovery Operator | 自動回復アクションの実行（Rust controller-rs） | `chaos-engineering` |
| chaos-client ライブラリ | アプリケーション内障害注入 SDK（4言語対応） | 各サービス内組込 |
| Prometheus / Grafana | 実験メトリクスの収集・可視化 | `observability` |
| Feature Flag Server | 障害注入の ON/OFF 制御 | `k1s0-system` |

### Chaos Engineering の 5 原則と k1s0 への適用

| 原則 | Netflix 定義 | k1s0 での適用 |
| --- | --- | --- |
| 定常状態の仮説を立てる | ビジネスメトリクスの正常範囲を定義 | SLO（D-108）で定義した可用性・レイテンシ・エラーレートを定常状態とする |
| 実世界のイベントを変化させる | 本番に近い障害を注入 | Pod Kill・Network Delay・Kafka Broker Down 等の実際の障害パターンを再現 |
| 本番環境で実験する | 本番環境での実験を推奨 | 成熟度 Level 2 以上で Production 実験を段階導入。Blast Radius を制御 |
| 自動化して継続的に実行する | 手動ではなく自動化 | ChaosSchedule CRD で定期実行。CI/CD パイプライン（D-101）にも統合 |
| Blast Radius を最小化する | 被害範囲を制御 | Tier 別の Blast Radius 制御戦略を適用。Feature Flag で即時停止 |

### 成熟度モデル

カオスエンジニアリングの導入を 5 段階で進める。

| Level | 名称 | 内容 | 対象環境 | 自動化度 |
| --- | --- | --- | --- | --- |
| 0 | 未導入 | カオスエンジニアリング未実施 | - | なし |
| 1 | 手動実験 | 手動で障害注入、手動で復旧確認 | Dev | 手動 |
| 2 | 自動実験 | ChaosSchedule による定期自動実行。Staging 環境に拡大 | Dev / Staging | 半自動 |
| 3 | 自動回復 | Recovery Operator による自動回復。SLO 連動トリガー | Dev / Staging / Production（限定） | 自動 |
| 4 | 継続的カオス | CI/CD 統合。Game Day 自動実行。全環境対象 | 全環境 | 完全自動 |

> **注記**: k1s0 の初期リリースでは Level 1 を目標とし、四半期ごとに Level を引き上げる計画とする。

### Blast Radius 制御戦略

段階的に Blast Radius を拡大し、リスクを最小化する。

| 段階 | 環境 | Blast Radius | 対象 | 承認フロー |
| --- | --- | --- | --- | --- |
| 1 | Dev | 単一 Pod | 個別サービスの単一 Pod | 開発者が自律的に実行 |
| 2 | Staging | 単一サービス | 特定サービスの全 Pod（最大 50%） | チームリードの承認 |
| 3 | Production（限定） | 単一 Tier 内の単一サービス | 特定サービスの一部 Pod（最大 30%） | SRE チームの承認 + Game Day 計画 |
| 4 | Production（拡大） | 複数サービス / 依存関係 | Tier 横断の障害シナリオ | SRE リード + プロダクトオーナー承認 |

### 実験分類

| 分類 | 対象 | 障害タイプ例 | k1s0 での適用場面 |
| --- | --- | --- | --- |
| Infrastructure | Kubernetes Node / Pod / Container | Pod Kill、Node Drain、Disk Fill | サービスのセルフヒーリング検証 |
| Application | アプリケーションプロセス | HTTP Fault Injection、gRPC Error、Memory Leak | アプリケーション層の耐障害性検証 |
| Network | ネットワーク通信 | Latency、Packet Loss、Partition、Bandwidth | Istio サービスメッシュ（D-116）の耐障害性検証 |
| Dependency | 外部依存 | Kafka Broker Down、Redis Timeout、PostgreSQL Failover | サーキットブレーカー・Retry の動作検証 |
| DNS | 名前解決 | DNS Error、DNS Random Response | サービスディスカバリの耐障害性検証 |

## D-201: 障害注入フレームワーク

### ツール選定

| 比較項目 | Litmus ChaosCenter | Chaos Mesh | Gremlin |
| --- | --- | --- | --- |
| ライセンス | Apache 2.0（OSS） | Apache 2.0（OSS） | 商用 |
| CNCF ステータス | Graduated | Incubating | - |
| Kubernetes ネイティブ | CRD ベース | CRD ベース | Agent ベース |
| UI ダッシュボード | ChaosCenter UI | Dashboard | Web Console |
| CI/CD 統合 | GitHub Actions Plugin | GitHub Actions Plugin | API 連携 |
| IO Chaos | 限定的 | 強力（fuse ベース） | 対応 |
| DNS Chaos | 限定的 | 対応 | 対応 |
| Application Chaos | LitmusProbe | HTTPChaos | Process Chaos |
| GameDay 機能 | Workflow（Argo 統合） | Workflow | Attack Scenario |
| RBAC 連携 | Kubernetes RBAC | Kubernetes RBAC | 独自 RBAC |
| コスト | 無料 | 無料 | 有料（$$$） |

> **注記**: Litmus ChaosCenter をプライマリツールとして採用し、IO Chaos・DNS Chaos など Litmus の弱い領域を Chaos Mesh で補完する。Gremlin は商用コストが高く OSS 優先の方針に合わないため不採用。

| 障害タイプ | 説明 | ツール | 影響範囲 | 推奨 Blast Radius |
| --- | --- | --- | --- | --- |
| pod-kill | 対象 Pod を即座に Kill | Litmus | 対象 Pod | single-pod |
| pod-failure | Pod を一時的に停止（削除ではなく fail 状態） | Litmus | 対象 Pod | single-pod |
| container-kill | 特定コンテナのみ Kill（sidecar 等の検証） | Litmus | 対象コンテナ | single-pod |

| network-loss | パケットロスの注入 | Chaos Mesh | loss: 30%, correlation: 25% | single-service |
| network-partition | ネットワーク分断（特定サービス間の通信遮断） | Chaos Mesh | direction: both, target: auth-server | single-service |

### k1s0 障害注入ライブラリ設計

アプリケーション層からの障害注入を行うクライアントライブラリを 4 言語（Rust / Go / TypeScript / Dart）で提供する。Feature Flag と連動し、アプリケーション内部での障害シミュレーションを実現する。

#### 公開 API（全言語共通契約）

| 型・トレイト | 種別 | 説明 |
| --- | --- | --- |
| `ChaosClient` | トレイト/インターフェース | 障害注入の制御インターフェース（inject, is_active, stop） |
| `ChaosConfig` | 構造体/クラス | 障害注入の設定（fault_type, probability, duration, target） |
| `FaultType` | enum | 障害タイプ（LatencyInjection, ErrorInjection, ResourceExhaustion, CircuitOpen） |
| `ChaosExperiment` | 構造体/クラス | 実行中の実験（id, config, status, started_at, metrics） |
| `ChaosMetrics` | 構造体/クラス | 実験メトリクス（requests_affected, errors_injected, latency_added） |
| `ChaosError` | enum/クラス | エラー型（ExperimentNotFound, FeatureFlagDisabled, InvalidConfig） |
| `NoopChaosClient` | 構造体/クラス | 本番用の何もしない実装（障害注入無効時のデフォルト） |
| `InMemoryChaosClient` | 構造体/クラス | テスト用インメモリ実装 |

**主要 API**:

```rust
use async_trait::async_trait;
use std::collections::HashMap;
use std::time::Duration;

/// 障害タイプ
#[derive(Debug, Clone, PartialEq, serde::Serialize, serde::Deserialize)]
pub enum FaultType {
    /// レイテンシ注入（指定ミリ秒の遅延を追加）
    LatencyInjection { delay_ms: u64 },
    /// エラー注入（指定確率でエラーを返す）
    ErrorInjection { status_code: u16, message: String },
    /// リソース枯渇（メモリ / CPU の模擬的な枯渇）
    ResourceExhaustion { resource: String, limit: u64 },
    /// サーキットブレーカー強制オープン
    CircuitOpen { target_service: String },
}

/// 障害注入設定
#[derive(Debug, Clone, serde::Serialize, serde::Deserialize)]
pub struct ChaosConfig {
    pub fault_type: FaultType,
    /// 障害発生確率（0.0 - 1.0）
    pub probability: f64,
    /// 障害注入の持続時間
    pub duration: Duration,
    /// 対象サービス / エンドポイント
    pub target: String,
    /// Feature Flag キー（このフラグが有効な場合のみ障害注入）
    pub feature_flag_key: String,
    /// 追加属性
    pub attributes: HashMap<String, String>,
}

/// カオスクライアントトレイト
#[async_trait]
#[cfg_attr(feature = "mock", mockall::automock)]
pub trait ChaosClient: Send + Sync {
    /// 障害注入を開始する
    async fn inject(&self, config: &ChaosConfig) -> Result<ChaosExperiment, ChaosError>;

    /// 指定した実験がアクティブかどうかを確認する
    async fn is_active(&self, experiment_id: &str) -> Result<bool, ChaosError>;

    /// 障害注入を停止する
    async fn stop(&self, experiment_id: &str) -> Result<(), ChaosError>;

    /// 全アクティブ実験を停止する（緊急停止用）
    async fn stop_all(&self) -> Result<(), ChaosError>;

    /// 実験メトリクスを取得する
    async fn metrics(&self, experiment_id: &str) -> Result<ChaosMetrics, ChaosError>;

    /// リクエスト処理前に障害を適用する（ミドルウェア用）
    async fn apply_fault(&self, target: &str) -> Result<Option<FaultEffect>, ChaosError>;
}

/// 障害適用結果
#[derive(Debug, Clone)]
pub enum FaultEffect {
    /// 遅延を追加
    Delay(Duration),
    /// エラーレスポンスを返す
    Error { status_code: u16, message: String },
    /// 影響なし
    None,
}
```

**Feature Flag 連動の使用例**:

```rust
use k1s0_chaos_client::{ChaosClient, ChaosConfig, FaultType, FaultEffect};
use k1s0_featureflag::{FeatureFlagClient, EvaluationContext};
use std::sync::Arc;
use std::time::Duration;

/// axum ミドルウェアでの障害注入
async fn chaos_middleware(
    chaos_client: Arc<dyn ChaosClient>,
    featureflag_client: Arc<dyn FeatureFlagClient>,
    target: &str,
    request_context: &EvaluationContext,
) -> Result<Option<FaultEffect>, Box<dyn std::error::Error>> {
    // Feature Flag が無効なら即座にスキップ
    let chaos_enabled = featureflag_client
        .is_enabled("chaos.global.enabled", request_context)
        .await
        .unwrap_or(false);

    if !chaos_enabled {
        return Ok(None);
    }

    // 障害適用を試行
    match chaos_client.apply_fault(target).await? {
        Some(FaultEffect::Delay(duration)) => {
            tracing::warn!(
                target = %target,
                delay_ms = %duration.as_millis(),
                "chaos: latency injection applied"
            );
            tokio::time::sleep(duration).await;
            Ok(Some(FaultEffect::Delay(duration)))
        }
        Some(FaultEffect::Error { status_code, message }) => {
            tracing::warn!(
                target = %target,
                status_code = %status_code,
                "chaos: error injection applied"
            );
            Ok(Some(FaultEffect::Error { status_code, message }))
        }
        _ => Ok(None),
    }
}

/// 障害注入の開始例
async fn start_chaos_experiment(
    chaos_client: Arc<dyn ChaosClient>,
) -> Result<(), Box<dyn std::error::Error>> {
    let config = ChaosConfig {
        fault_type: FaultType::LatencyInjection { delay_ms: 500 },
        probability: 0.3,
        duration: Duration::from_secs(300),
        target: "task-server".to_string(),
        feature_flag_key: "chaos.task-server.latency".to_string(),
        attributes: Default::default(),
    };

    let experiment = chaos_client.inject(&config).await?;
    tracing::info!(experiment_id = %experiment.id, "chaos experiment started");

    Ok(())
}
```

#### Go 実装

**配置先**: `regions/system/library/go/chaos-client/`

```go
package chaosclient

import (
	"context"
	"time"
)

// FaultType は障害タイプを定義する。
type FaultType string

const (
	FaultTypeLatencyInjection    FaultType = "latency_injection"
	FaultTypeErrorInjection      FaultType = "error_injection"
	FaultTypeResourceExhaustion  FaultType = "resource_exhaustion"
	FaultTypeCircuitOpen         FaultType = "circuit_open"
)

// ChaosConfig は障害注入の設定を保持する。
type ChaosConfig struct {
	FaultType      FaultType         `json:"fault_type"`
	Probability    float64           `json:"probability"`
	Duration       time.Duration     `json:"duration"`
	Target         string            `json:"target"`
	FeatureFlagKey string            `json:"feature_flag_key"`
	Attributes     map[string]string `json:"attributes,omitempty"`
	// LatencyInjection 用パラメータ
	DelayMs int64 `json:"delay_ms,omitempty"`
	// ErrorInjection 用パラメータ
	StatusCode int    `json:"status_code,omitempty"`
	Message    string `json:"message,omitempty"`
}

// ChaosExperiment は実行中の実験を表す。
type ChaosExperiment struct {
	ID        string       `json:"id"`
	Config    ChaosConfig  `json:"config"`
	Status    string       `json:"status"`
	StartedAt time.Time   `json:"started_at"`
	Metrics   ChaosMetrics `json:"metrics"`
}

// ChaosMetrics は実験メトリクスを保持する。
type ChaosMetrics struct {
	RequestsAffected int64 `json:"requests_affected"`
	ErrorsInjected   int64 `json:"errors_injected"`
	LatencyAddedMs   int64 `json:"latency_added_ms"`
}

// FaultEffect は障害適用結果を表す。
type FaultEffect struct {
```

## D-202: 自動回復（Self-Healing）設計

### アーキテクチャ

![Self-Healing パイプライン](images/self-healing-pipeline.svg)

自動回復パイプラインは以下のフローで障害を検知・回復する。

| ステップ | コンポーネント | 役割 |
| --- | --- | --- |
| 1. 検知 | Prometheus | メトリクスベースの異常検知（SLO 違反・リソース枯渇） |
| 2. 通知 | Alertmanager | アラートのルーティング・グルーピング・抑制 |
| 3. 判定 | Recovery Operator | RecoveryPolicy に基づく回復アクションの決定 |
| 4. 実行 | Recovery Operator | 回復アクションの実行（Pod 再起動・スケールアウト等） |
| 5. 検証 | Recovery Operator | 回復後の SLO 充足を検証 |
| 6. 記録 | RecoveryHistory CRD | 回復履歴の永続化（ポストモーテム用） |

### 障害検知パイプライン

#### 検知ルール定義

```yaml
# infra/kubernetes/monitoring/recovery-alerting-rules.yaml
apiVersion: monitoring.coreos.com/v1
kind: PrometheusRule
metadata:
  name: self-healing-alerting-rules
  namespace: observability
  labels:
    prometheus: k8s
    role: alerting-rules
spec:
  groups:
    - name: self-healing.rules
      rules:
        # Pod の再起動ループ検知
        - alert: PodCrashLooping
          expr: |
            rate(kube_pod_container_status_restarts_total[15m]) * 60 * 15 > 3
          for: 5m
          labels:
            severity: critical
            recovery_action: pod-restart
            tier: "{{ $labels.namespace }}"
          annotations:
            summary: "Pod {{ $labels.namespace }}/{{ $labels.pod }} is crash looping"
            recovery_policy: auto-pod-restart

        # サービスの高エラーレート検知
        - alert: HighErrorRate
          expr: |
            sum(rate(http_requests_total{status=~"5.."}[5m])) by (service, namespace)
            / sum(rate(http_requests_total[5m])) by (service, namespace)
            > 0.05
          for: 3m
          labels:
            severity: critical
            recovery_action: circuit-breaker-activate
            tier: "{{ $labels.namespace }}"
          annotations:
            summary: "Service {{ $labels.service }} error rate > 5%"
            recovery_policy: auto-circuit-breaker

        # レイテンシ SLO 違反検知
        - alert: HighLatencySLOViolation
          expr: |
            histogram_quantile(0.99, sum(rate(http_request_duration_seconds_bucket[5m])) by (le, service, namespace))
            > 1.0
          for: 5m
          labels:
            severity: warning
            recovery_action: scale-out
            tier: "{{ $labels.namespace }}"
          annotations:
            summary: "Service {{ $labels.service }} P99 latency > 1s"
            recovery_policy: auto-scale-out

        # メモリ使用量の急上昇
        - alert: MemoryUsageCritical
          expr: |
            container_memory_working_set_bytes{container!=""}
            / container_spec_memory_limit_bytes{container!=""}
            > 0.9
          for: 5m
          labels:
            severity: critical
            recovery_action: pod-restart
          annotations:
            summary: "Container {{ $labels.container }} memory usage > 90%"
            recovery_policy: auto-pod-restart

        # Kafka Consumer Lag の急上昇
        - alert: KafkaConsumerLagCritical
          expr: |
            kafka_consumer_group_lag_sum > 10000
          for: 10m
          labels:
            severity: warning
            recovery_action: scale-out
          annotations:
            summary: "Kafka consumer group {{ $labels.consumergroup }} lag > 10000"
            recovery_policy: auto-scale-out
```

#### 多段エスカレーション設計

| エスカレーション段階 | 条件 | アクション | 通知先 | タイムアウト |
| --- | --- | --- | --- | --- |
| L0: 自動回復 | RecoveryPolicy に合致 | Recovery Operator が自動実行 | Slack 通知（info） | 5 分 |
| L1: 自動回復失敗 | L0 が 2 回失敗 | 別の回復アクションを試行 | Slack 通知（warning） | 10 分 |
| L2: 手動介入要求 | L1 が失敗 | オンコール担当に PagerDuty 通知 | PagerDuty + Teams | 30 分 |
| L3: インシデント宣言 | L2 タイムアウト | インシデントコマンダーに報告 | Teams + メール | - |

### Recovery Action カタログ

| Action | 条件 | 対象 | 自動/手動 | 優先度 | 副作用 |
| --- | --- | --- | --- | --- | --- |
| Pod 再起動 | CrashLoopBackOff / OOMKilled | 障害 Pod | 自動 | 高 | 一時的なリクエスト損失 |
| スケールアウト | 高レイテンシ / 高 CPU | Deployment | 自動 | 中 | リソースコスト増 |
| Circuit Breaker 活性化 | 高エラーレート（> 5%） | 障害サービスへの通信 | 自動 | 高 | 一部機能の一時停止 |
| トラフィックシフト | 特定バージョンの障害 | Istio DestinationRule | 自動 | 高 | カナリアリリースの中断 |
| Rollback トリガー | デプロイ後の SLO 違反 | Deployment | 半自動（承認後） | 高 | 新機能のロールバック |
| Database フェイルオーバー | Primary DB の応答なし | PostgreSQL Cluster | 自動 | 最高 | 一時的な書き込み不可 |
| Kafka Consumer リバランス | Consumer Lag 急上昇 | Consumer Group | 自動 | 中 | 一時的なメッセージ遅延 |
| DNS フェイルオーバー | サービスの DNS 解決失敗 | CoreDNS | 自動 | 高 | キャッシュ済みエントリへのフォールバック |

#### Recovery Operator 実装（Rust controller-rs）

```rust
use kube::{
    api::{Api, ListParams, Patch, PatchParams},
    client::Client,
    runtime::controller::{Action, Controller},
    CustomResource, Resource,
};
use schemars::JsonSchema;
use serde::{Deserialize, Serialize};
use std::sync::Arc;
use tokio::time::Duration;
use tracing::{info, warn, error};

/// RecoveryPolicy のスペック定義
#[derive(CustomResource, Debug, Serialize, Deserialize, Clone, JsonSchema)]
#[kube(
    group = "k1s0.io",
    version = "v1",
    kind = "RecoveryPolicy",
    namespaced,
    status = "RecoveryPolicyStatus",
    shortname = "rp"
)]
pub struct RecoveryPolicySpec {
    pub target_ref: TargetRef,
    pub conditions: Vec<RecoveryCondition>,
    pub actions: Vec<RecoveryActionSpec>,
    pub cooldown: String,
    pub escalation: Option<EscalationConfig>,
}

#[derive(Debug, Serialize, Deserialize, Clone, JsonSchema)]
pub struct TargetRef {
    pub kind: String,
    pub name: String,
    pub namespace: String,
}

#[derive(Debug, Serialize, Deserialize, Clone, JsonSchema)]
pub struct RecoveryCondition {
    pub alert_name: String,
    pub severity: String,
    pub duration: String,
}

#[derive(Debug, Serialize, Deserialize, Clone, JsonSchema)]
pub struct RecoveryActionSpec {
    #[serde(rename = "type")]
    pub action_type: String,
    pub order: i32,
    pub timeout: String,
    pub max_retries: i32,
    pub params: serde_json::Value,
}

#[derive(Debug, Serialize, Deserialize, Clone, JsonSchema)]
pub struct EscalationConfig {
    pub max_auto_retries: i32,
    pub notify_on_failure: String,
}

#[derive(Debug, Serialize, Deserialize, Clone, JsonSchema)]
pub struct RecoveryPolicyStatus {
    pub last_executed: Option<String>,
    pub execution_count: i64,
    pub last_result: String,
}

/// Recovery Operator のメインコントローラー
pub struct RecoveryOperator {
    client: Client,
    metrics: RecoveryMetrics,
}

impl RecoveryOperator {
    pub fn new(client: Client, metrics: RecoveryMetrics) -> Self {
        Self { client, metrics }
    }

    /// Alertmanager Webhook を受信し、RecoveryPolicy に基づいて回復アクションを実行する
    pub async fn handle_alert(
        &self,
        alert: &AlertmanagerAlert,
    ) -> Result<(), RecoveryError> {
        let policy = self.find_matching_policy(alert).await?;

        // クールダウン期間チェック
        if self.is_in_cooldown(&policy).await? {
            info!(
                policy = %policy.metadata.name.as_deref().unwrap_or("unknown"),
                "skipping recovery: cooldown period active"
            );
            return Ok(());
        }

        // アクションを優先度順に実行
        let mut actions = policy.spec.actions.clone();
        actions.sort_by_key(|a| a.order);

        for action_spec in &actions {
            match self.execute_action(&policy, action_spec, alert).await {
                Ok(()) => {
                    self.metrics.record_recovery_success(
                        &action_spec.action_type,
                        &policy.spec.target_ref.name,
                    );
                    // SLO 検証
                    if self.verify_slo_recovery(&policy).await? {
                        info!("recovery successful, SLO verified");
                        return Ok(());
                    }
                }
                Err(e) => {
                    warn!(
                        action = %action_spec.action_type,
                        error = %e,
                        "recovery action failed, trying next action"
                    );
                    self.metrics.record_recovery_failure(
                        &action_spec.action_type,
                        &policy.spec.target_ref.name,
                    );
                }
            }
        }

        // 全アクション失敗時のエスカレーション
        if let Some(escalation) = &policy.spec.escalation {
            self.escalate(alert, escalation).await?;
        }

        Err(RecoveryError::AllActionsFailed)
    }
}
```

### SLO 連動自動回復

エラーバジェット消費率に基づき、回復アクションを段階的にトリガーする。SLO 設計（D-108）で定義したバーンレートアラートと連動する。

| エラーバジェット消費率 | バーンレート | 回復アクション | 自動/手動 |
| --- | --- | --- | --- |
| < 50% | < 3x | なし（通常運用） | - |
| 50% - 75% | 3x - 6x | スケールアウト + アラート通知 | 自動 |
| 75% - 90% | 6x - 14.4x | Circuit Breaker 活性化 + トラフィックシフト | 自動 |
| > 90% | > 14.4x | Rollback トリガー + インシデント宣言 | 半自動（承認後） |

```yaml
# SLO 連動回復ポリシー
apiVersion: k1s0.io/v1
kind: RecoveryPolicy
metadata:
  name: slo-budget-recovery
  namespace: k1s0-system
spec:
  targetRef:
    kind: Deployment
    name: auth-server
    namespace: k1s0-system
  conditions:
    - alertName: SLOBurnRateWarning
      severity: warning
      duration: "30m"
    - alertName: SLOBurnRateCritical
      severity: critical
      duration: "5m"
  actions:
    # バーンレート 6x: スケールアウト
    - type: scale-out
      order: 1
      timeout: "5m"
      maxRetries: 1
      params:
        triggerBurnRate: 6.0
        maxReplicas: 10
        scaleUpPercent: 50
    # バーンレート 14.4x: Circuit Breaker + トラフィックシフト
    - type: circuit-breaker
      order: 2
      timeout: "1m"
      maxRetries: 1
      params:
        triggerBurnRate: 14.4
        targetServices: ["dependent-service-a", "dependent-service-b"]
    - type: traffic-shift
      order: 3
      timeout: "3m"
      maxRetries: 1
      params:
        triggerBurnRate: 14.4
        stableVersion: "v1.2.3"
        canaryWeight: 0
    # バーンレート 14.4x 持続: Rollback
    - type: rollback
      order: 4
      timeout: "10m"
      maxRetries: 1
      params:
        triggerBurnRate: 14.4
        triggerDuration: "15m"
        requireApproval: true
  cooldown: "30m"
  escalation:
    maxAutoRetries: 2
    notifyOnFailure: "pagerduty://sre-oncall"
```

## D-203: Game Day 運用設計

### Game Day 概要と目的

Game Day は、計画的な障害注入イベントを通じて、システムの耐障害性とチームの対応力を検証する取り組みである。

| 項目 | 内容 |
| --- | --- |
| 目的 | 本番障害への対応力向上、未知の脆弱性の発見、ランブック（対応手順書）の検証 |
| 対象 | 全 Tier のサービスとインフラ |
| 頻度 | 月次（小規模）/ 四半期（大規模） |
| 参加者 | 開発チーム、SRE チーム、プロダクトオーナー |
| 前提条件 | 可観測性基盤が稼働中、ランブックが整備済み、ロールバック手順が確立済み |

### Game Day メトリクス

Game Day の効果を定量的に追跡するメトリクス。

| メトリクス | 定義 | 目標値 | 計測方法 |
| --- | --- | --- | --- |
| MTTD（Mean Time To Detect） | 障害注入から検知までの平均時間 | < 2 分 | Prometheus alert firing time - experiment start time |
| MTTR（Mean Time To Recover） | 障害注入から SLO 復旧までの平均時間 | < 10 分 | SLO recovery time - experiment start time |
| 発見された脆弱性数 | Game Day で発見された新規脆弱性の数 | 減少傾向 | 報告書の脆弱性カウント |
| 脆弱性修正率 | 前回発見された脆弱性の修正完了率 | > 80% | 修正済み数 / 前回発見数 |
| シナリオ合格率 | 期待値を満たしたシナリオの割合 | > 90% | passed + passed_with_findings / total |
| 自動回復成功率 | Recovery Operator による自動回復の成功率 | > 95% | auto recovery success / auto recovery total |
| 対応手順カバレッジ | ランブックでカバーされた障害パターンの割合 | > 90% | ランブック対応数 / 全障害パターン数 |

```yaml
# Game Day メトリクスの Prometheus Recording Rule
apiVersion: monitoring.coreos.com/v1
kind: PrometheusRule
metadata:
  name: gameday-metrics-rules
  namespace: observability
  labels:
    prometheus: k8s
    role: recording-rules
spec:
  groups:
    - name: gameday.metrics
      interval: 30s
      rules:
        # MTTD: 障害注入から検知までの時間
        - record: gameday:mttd_seconds:avg
          expr: |
            avg(
              chaos_recovery_triggered_total_timestamp
              - chaos_experiment_total_timestamp
            ) by (experiment_name)

        # MTTR: 障害注入から回復までの時間
        - record: gameday:mttr_seconds:avg
          expr: |
            avg(chaos_recovery_duration_seconds) by (experiment_name, recovery_action)

        # 自動回復成功率
        - record: gameday:auto_recovery_success_rate
          expr: |
            sum(chaos_recovery_triggered_total{trigger="auto", result="success"})
            /
            sum(chaos_recovery_triggered_total{trigger="auto"})
```

## 技術スタック選定理由

| コンポーネント | 技術 | 選定理由 | 代替案 |
| --- | --- | --- | --- |
| カオス実験管理 | Litmus ChaosCenter 3.x | CNCF Graduated。CRD ベースで Kubernetes ネイティブ。Argo Workflow 統合で複雑なシナリオに対応 | Chaos Mesh（補完利用）、Gremlin（商用） |
| IO/DNS 障害注入 | Chaos Mesh 2.x | CNCF Incubating。IO Chaos（fuse ベース）と DNS Chaos が Litmus より高機能 | Litmus（IO Chaos が限定的） |
| Recovery Operator | controller-rs（kube-rs） | Rust で Kubernetes Operator を実装。k1s0 の Rust 基盤と統合。メモリ安全性と高パフォーマンス | kubebuilder（Go）、Operator SDK（Go） |
| メトリクス連携 | Prometheus + Grafana | 既存の可観測性基盤（D-107）を活用。SLO（D-108）との直接連携が可能 | Datadog（商用）、New Relic（商用） |
| Feature Flag 連動 | k1s0-featureflag | 既存ライブラリで障害注入の ON/OFF を制御。テナント単位・ユーザー単位の制御が可能 | LaunchDarkly（商用）、Unleash |
| サービスメッシュ連携 | Istio | 既存のサービスメッシュ（D-116）でトラフィックシフト・Fault Injection を実現 | Linkerd |
| CRD 定義 | Kubernetes CRD | RecoveryPolicy / RecoveryAction / RecoveryHistory / GameDay の 4 つの CRD でカオス運用を宣言的に管理 | ConfigMap ベース（柔軟性が不足） |

## ローカル開発環境

### docker-compose でのカオステスト実行手順

ローカル開発環境では、docker-compose の `chaos` プロファイルでカオステスト用コンポーネントを起動する。

```yaml
# infra/docker/docker-compose.chaos.yaml
services:
  chaos-client-test:
    build:
      context: ../../regions/system/library/rust/chaos-client
      dockerfile: Dockerfile.test
    environment:
      - CHAOS_ENABLED=true
      - FEATURE_FLAG_ENDPOINT=http://featureflag-server:8080
      - PROMETHEUS_ENDPOINT=http://prometheus:9090
      - RUST_LOG=k1s0_chaos_client=debug
    depends_on:
      - featureflag-server
      - prometheus
    profiles:
      - chaos

  chaos-experiment-runner:
    image: litmuschaos/litmus-checker:latest
    environment:
      - CHAOS_ENGINE_NAME=local-pod-kill
      - APP_NAMESPACE=default
      - CHAOS_NAMESPACE=default
    volumes:
      - /var/run/docker.sock:/var/run/docker.sock
    profiles:
      - chaos
```

### ミニマルなカオス実験のデモ手順

```bash
# 1. カオステスト環境を起動
docker compose --profile infra --profile observability --profile chaos up -d

# 2. Feature Flag で障害注入を有効化
curl -X PUT http://localhost:8080/api/v1/flags/chaos.global.enabled \
  -H "Content-Type: application/json" \
  -d '{"enabled": true, "variants": [{"name": "on", "value": "true", "weight": 100}]}'

# 3. レイテンシ注入実験を開始
curl -X POST http://localhost:8090/api/v1/experiments \
  -H "Content-Type: application/json" \
  -d '{
    "fault_type": {"LatencyInjection": {"delay_ms": 500}},
    "probability": 0.3,
    "duration_secs": 300,
    "target": "task-server",
    "feature_flag_key": "chaos.task-server.latency"
  }'

# 4. メトリクスを確認
curl -s http://localhost:9090/api/v1/query?query=chaos_experiment_active | jq .

# 5. 実験を停止
curl -X POST http://localhost:8090/api/v1/experiments/stop-all

# 6. Feature Flag を無効化
curl -X PUT http://localhost:8080/api/v1/flags/chaos.global.enabled \
  -H "Content-Type: application/json" \
  -d '{"enabled": false}'
```

### 必要な環境変数・設定

| 環境変数 | 説明 | デフォルト値 | 必須 |
| --- | --- | --- | --- |
| `CHAOS_ENABLED` | カオス機能の有効/無効 | `false` | はい |
| `CHAOS_ENDPOINT` | chaos-client API エンドポイント | `http://localhost:8090` | いいえ |
| `FEATURE_FLAG_ENDPOINT` | Feature Flag サーバーエンドポイント | `http://localhost:8080` | はい |
| `PROMETHEUS_ENDPOINT` | Prometheus エンドポイント | `http://localhost:9090` | いいえ |
| `CHAOS_DEFAULT_PROBABILITY` | デフォルトの障害発生確率 | `0.0` | いいえ |
| `CHAOS_MAX_DURATION_SECS` | 最大実験持続時間（秒） | `3600` | いいえ |
| `CHAOS_ABORT_ON_SLO_BREACH` | SLO 違反時の自動中止 | `true` | いいえ |
| `RUST_LOG` | ログレベル | `info` | いいえ |

```toml
# config/chaos.toml（ローカル開発用の設定例）
[chaos]
enabled = true
default_probability = 0.1
max_duration_secs = 600
abort_on_slo_breach = true

[chaos.feature_flags]
global_key = "chaos.global.enabled"
prefix = "chaos."

[chaos.metrics]
enabled = true
endpoint = "http://localhost:9090"

[chaos.experiments]
# ローカル開発用のデフォルト実験定義
[[chaos.experiments.presets]]
name = "latency-injection"
fault_type = "LatencyInjection"
delay_ms = 200
probability = 0.2
target = "*"

[[chaos.experiments.presets]]
name = "error-injection"
fault_type = "ErrorInjection"
status_code = 503
message = "Service Unavailable (chaos)"
probability = 0.1
target = "*"
```
