# モジュール成熟度ロードマップ

## 概要
k1s0プラットフォームの各モジュールは `modules.yaml` の `maturity` フィールドで成熟度を管理する。
本ドキュメントは成熟度レベルの定義、昇格基準、プロセスを定める。

## 成熟度レベル定義

### template-only
- テンプレートとしてのみ存在し、スキャフォールド生成に使用される
- CIでのビルド・テストは実行されない
- ディレクトリ構成・型定義のみが実装されている

### experimental
- 積極開発中。インターフェースは不安定で破壊的変更が発生する可能性がある
- CIでは `continue-on-error: true` で実行。失敗しても全体ビルドをブロックしない
- インフラ依存（DB, Kafka 等）が未設定でも起動可能（インメモリフォールバック）

### beta
- 機能として完成しているが、本番利用前の検証段階
- CIで必ずリント・テスト・ビルドが実行される（failure はブロック）
- 少なくとも1つのステージング環境での稼働実績が必要
- 本番移行前の最終確認フェーズ

### production
- 本番利用可能。SLAが適用される
- CI で必ずリント・テスト・ビルドが実行される
- インフラ設定（DB, Kafka 等）が未指定の場合は起動を拒否する（infra_guard）
- 下位互換性の維持が必要（破壊的変更はDeprecationプロセスを経る）

## 昇格基準

### experimental → beta
- [ ] ユニットテストカバレッジ 80% 以上
- [ ] E2Eテストが主要シナリオをカバー
- [ ] API設計書（docs/servers/ または docs/libraries/）が整備済み
- [ ] セキュリティレビューが完了（認証・認可・入力検証）
- [ ] ステージング環境で1週間以上の安定稼働実績

### beta → production
- [ ] 本番同等環境での負荷テスト完了（想定トラフィックの2倍）
- [ ] SLO定義（可用性・レイテンシ・エラー率）が文書化済み
- [ ] 運用手順書（デプロイ・ロールバック・障害対応）が整備済み
- [ ] データベースマイグレーション戦略が確認済み（必要な場合）
- [ ] セキュリティ脆弱性スキャンがクリーン（HIGH/CRITICAL ゼロ）
- [ ] oncall/障害対応体制が確認済み
- [ ] `infra_guard` 設定が有効（stable環境でのインフラ必須化）

## 昇格プロセス

1. **PR作成**: `modules.yaml` の `maturity` フィールドを更新するPRを作成
2. **チェックリスト確認**: 上記の昇格基準を満たすことをPRで明示
3. **レビュー**: テックリード + セキュリティ担当のレビューが必要
4. **ドキュメント同期**: 設計書の `status` セクションも同時に更新（同一コミット）
5. **マージ後**: CI/CDで新しい成熟度に応じた設定が自動適用される

## 降格プロセス

運用上の問題が発見された場合、成熟度を下げることができる:
- `production → beta`: 重大なSLO違反、セキュリティ脆弱性発見時
- `beta → experimental`: 設計見直しが必要と判断された場合

降格は通常の機能PRと同様のプロセスを経る。
