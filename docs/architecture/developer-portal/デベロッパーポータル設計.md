# デベロッパーポータル設計

k1s0 における開発者ポータルの設計を定義する。開発者の認知負荷を最小化し、自律的なサービス開発を可能にすることを目的とする。
Tier アーキテクチャの詳細は [tier-architecture.md](../overview/tier-architecture.md) を参照。

## 基本方針

- **認知負荷の最小化**: 200+ の設計ドキュメント・25 System Servers・50+ ライブラリを Single Pane of Glass で集約する
- **Backstage ベース**: CNCF Incubating プロジェクトである Backstage をベースに、k1s0 固有の拡張プラグインを実装する
- **既存システム統合**: service-catalog サーバー（Rust axum）・api-registry サーバーとのリアルタイム統合
- **セルフサービス**: k1s0 CLI のテンプレートエンジンと連携し、ポータル UI からのサービス作成・インフラプロビジョニングを実現する
- **品質の可視化**: サービススコアカード・DORA メトリクス・Tech Radar で継続的な品質改善を促進する

## D-220: デベロッパーポータル全体設計

### アーキテクチャ概要

![デベロッパーポータル全体アーキテクチャ](images/developer-portal-overview.svg)

Backstage をコアとし、k1s0 固有のカスタムプラグインで機能を拡張する。フロントエンドは React（TypeScript）、バックエンドは Node.js で構成し、PostgreSQL をカタログデータベースとして使用する。

### Backstage 選定理由

| 項目 | Backstage | Port | Cortex | 自社開発 |
| --- | --- | --- | --- | --- |
| ライセンス | Apache 2.0（OSS） | 商用 SaaS | 商用 SaaS | N/A |
| CNCF ステータス | Incubating | N/A | N/A | N/A |
| プラグインエコシステム | 200+ コミュニティプラグイン | 限定的 | 限定的 | ゼロから構築 |
| カスタマイズ性 | プラグイン API で自由に拡張 | API 連携のみ | API 連携のみ | 完全自由 |
| Software Catalog | 標準搭載（Entity Model） | 搭載 | 搭載 | 自社実装が必要 |
| Software Templates | 標準搭載（Scaffolder） | テンプレート機能あり | なし | 自社実装が必要 |
| TechDocs | 標準搭載（MkDocs 統合） | ドキュメント機能あり | なし | 自社実装が必要 |
| セルフホスト | 可能（オンプレミス対応） | SaaS のみ | SaaS のみ | 可能 |
| 運用コスト | 中（OSS + 自社運用） | 高（SaaS 利用料） | 高（SaaS 利用料） | 非常に高 |
| 初期構築コスト | 低〜中 | 低 | 低 | 非常に高 |
| コミュニティ | 活発（Spotify 主導、CNCF） | 限定的 | 限定的 | なし |
| k1s0 統合難易度 | 低（プラグイン開発） | 高（API 制約） | 高（API 制約） | 中（自由だが工数大） |

> **注記** k1s0 はオンプレミス環境での運用を前提としているため、SaaS 製品は選択肢から除外される。Backstage はセルフホスト可能かつプラグインエコシステムが充実しており、k1s0 の既存サーバー群との統合が最も容易である。

### 認証統合（Keycloak OIDC 連携）

Backstage の認証プロバイダーとして Keycloak を使用し、既存の認証基盤と統一する。

#### Backstage Auth Provider 設定

```typescript
// packages/backend/src/plugins/auth.ts
import { createBackendModule } from '@backstage/backend-plugin-api';
import {
  authProvidersExtensionPoint,
  createOAuthProviderFactory,
} from '@backstage/plugin-auth-node';
import { oidcAuthenticator } from '@backstage/plugin-auth-backend-module-oidc-provider';

export const k1s0AuthModule = createBackendModule({
  pluginId: 'auth',
  moduleId: 'k1s0-oidc',
  register(reg) {
    reg.registerInit({
      deps: {
        providers: authProvidersExtensionPoint,
      },
      async init({ providers }) {
        providers.registerProvider({
          providerId: 'k1s0-oidc',
          factory: createOAuthProviderFactory({
            authenticator: oidcAuthenticator,
            async signInResolver({ profile }, ctx) {
              if (!profile.email) {
                throw new Error('Login failed: email is required');
              }
              return ctx.signInWithCatalogUser({
                filter: {
                  'spec.profile.email': profile.email,
                },
              });
            },
          }),
        });
      },
    });
  },
});
```

#### RBAC: チーム・ロールベースのアクセス制御

Backstage の Permission Framework と Keycloak のロール情報を連携し、k1s0 の RBAC モデルに準拠したアクセス制御を実現する。

| ロール | 権限 | 対象 |
| --- | --- | --- |
| `viewer` | カタログ閲覧、ドキュメント閲覧、スコアカード閲覧 | 全ユーザー |
| `developer` | テンプレート実行、サービス登録・更新、API スキーマ登録 | 開発チームメンバー |
| `team-lead` | 承認ワークフロー、リソースクォータ管理、チーム設定 | チームリーダー |
| `platform-admin` | 全機能、プラグイン設定、Tech Radar 更新、グローバル設定 | プラットフォームチーム |

```typescript
// packages/backend/src/plugins/permission.ts
import { createBackendModule } from '@backstage/backend-plugin-api';
import {
  PolicyDecision,
  AuthorizeResult,
} from '@backstage/plugin-permission-common';
import {
  PermissionPolicy,
  PolicyQuery,
} from '@backstage/plugin-permission-node';
import { BackstageIdentityResponse } from '@backstage/plugin-auth-node';

class K1s0PermissionPolicy implements PermissionPolicy {
  async handle(
    request: PolicyQuery,
    user?: BackstageIdentityResponse,
  ): Promise<PolicyDecision> {
    // Keycloak ロールに基づくアクセス制御
    const userRoles = user?.identity?.ownershipEntityRefs ?? [];

    // platform-admin は全権限
    if (userRoles.some(ref => ref.includes('platform-admin'))) {
      return { result: AuthorizeResult.ALLOW };
    }

    // テンプレート実行は developer 以上
    if (request.permission.name === 'scaffolder.task.create') {
      if (userRoles.some(ref =>
        ref.includes('developer') || ref.includes('team-lead')
      )) {
        return { result: AuthorizeResult.ALLOW };
      }
      return { result: AuthorizeResult.DENY };
    }

    // カタログ閲覧は全ユーザーに許可
    if (request.permission.name.startsWith('catalog.entity.read')) {
      return { result: AuthorizeResult.ALLOW };
    }

    return { result: AuthorizeResult.ALLOW };
  }
}

export const k1s0PermissionModule = createBackendModule({
  pluginId: 'permission',
  moduleId: 'k1s0-policy',
  register(reg) {
    reg.registerInit({
      deps: {},
      async init() {
        // PermissionPolicy の登録
      },
    });
  },
});
```

## D-221: サービスカタログ設計

### サービスカタログ全体像

![サービスカタログアーキテクチャ](images/service-catalog-architecture.svg)

Backstage の Software Catalog と k1s0 の service-catalog サーバーを双方向同期する。Backstage 側は Entity モデルで管理し、service-catalog サーバー側はサービスメタデータ・依存関係・ヘルスステータスを管理する。

### Backstage Software Catalog と k1s0 service-catalog の統合

| 項目 | Backstage Software Catalog | k1s0 service-catalog サーバー |
| --- | --- | --- |
| データモデル | Entity（Component, System, API, Resource, Group, User） | Service（メタデータ、依存関係、ヘルス、スコアカード） |
| データソース | catalog-info.yaml（Git） | REST/gRPC API |
| 同期方向 | Git → Backstage Catalog | Backstage Catalog ↔ service-catalog サーバー |
| 更新頻度 | 10 分間隔（GitHub プロバイダー） | 5 分間隔（k1s0-catalog プラグイン） |
| 用途 | 開発者向け UI 表示 | プログラマティックなサービス管理 |

#### catalog-info.yaml 標準定義

各サービスのリポジトリルート（またはモノリポ内のサービスディレクトリ）に `catalog-info.yaml` を配置する。k1s0 の 3 ティア構造に対応した Entity 定義を使用する。

#### k1s0 の 3 ティア対応 Entity 定義

| Backstage Entity | k1s0 での用途 | 対応する Tier |
| --- | --- | --- |
| System | Tier を表現するグルーピング | system / business / service |
| Component | サーバー・クライアント・ライブラリ | 全 Tier |
| API | REST/gRPC/GraphQL エンドポイント | 全 Tier |
| Resource | データベース・Kafka トピック・Redis | 全 Tier |
| Group | 開発チーム・部門 | N/A |
| User | 開発者個人 | N/A |

#### Entity 自動検出

GitHub リポジトリをスキャンし、`catalog-info.yaml` を自動検出して Backstage カタログに登録する。

```yaml

          repository: "k1s0"
        schedule:
          frequency: { minutes: 30 }
```

#### k1s0 CLI のテンプレートエンジン（Tera）との橋渡し設計

k1s0-scaffolder プラグインは Backstage の Scaffolder バックエンドにカスタムアクション `k1s0:cli:generate` を登録する。このアクションは k1s0 CLI サーバーの API を呼び出し、Tera テンプレートエンジンでコードを生成する。

```typescript
// plugins/k1s0-scaffolder/src/actions/generate.ts
import { createTemplateAction } from '@backstage/plugin-scaffolder-node';

export const createK1s0GenerateAction = () => {
  return createTemplateAction<{
    template: string;
    parameters: Record<string, unknown>;
  }>({
    id: 'k1s0:cli:generate',
    description: 'k1s0 CLI テンプレートエンジンでコードを生成する',
    schema: {
      input: {
        type: 'object',
        required: ['template', 'parameters'],
        properties: {
          template: {
            type: 'string',
            title: 'テンプレート名',
            description: 'k1s0 CLI のテンプレート名（例: server-rust）',
          },
          parameters: {
            type: 'object',
            title: 'テンプレートパラメータ',
          },
        },
      },
    },
    async handler(ctx) {
      const { template, parameters } = ctx.input;

      // k1s0 CLI サーバー API を呼び出してコード生成
      const response = await fetch(
        `${ctx.config.getString('k1s0.cli.endpoint')}/api/v1/generate`,
        {
          method: 'POST',
          headers: { 'Content-Type': 'application/json' },
          body: JSON.stringify({
            template,
            parameters,
            output_dir: ctx.workspacePath,
          }),
        },
      );

      if (!response.ok) {
        throw new Error(
          `k1s0 CLI generation failed: ${response.statusText}`,
        );
      }

      const result = await response.json();
      ctx.logger.info(
        `Generated ${result.files_created} files using template: ${template}`,
      );
    },
  });
};
```

## D-223: サービススコアカード設計

### スコアカード概要

![スコアカード概要](images/scorecard-overview.svg)

サービススコアカードは各サービスの品質を定量的に評価し、継続的な改善を促進する。5 つのカテゴリに分類されたチェック項目を自動収集し、0-100 のスコアで可視化する。

### スコアリングカテゴリ

| カテゴリ | 重み | チェック項目 | データソース |
| --- | --- | --- | --- |
| 信頼性 | 30% | SLO 達成率、エラーバジェット残量、MTTR、可用性（30 日間） | Prometheus、service-catalog サーバー |
| セキュリティ | 25% | 脆弱性スキャン結果（Critical/High/Medium）、依存関係更新頻度、mTLS 設定、コンテナイメージスキャン | Trivy、GitHub Dependabot、Istio |
| 可観測性 | 20% | メトリクス定義数、アラートルール数、ダッシュボード有無、構造化ログ対応、トレース有効化 | Prometheus、Grafana API、Loki |
| 品質 | 15% | テストカバレッジ（%）、リントパス率、ドキュメント充実度、コードレビュー率 | SonarQube、GitHub API、catalog-info.yaml |
| 運用性 | 10% | オーナー設定、オンコール設定、ランブック有無、インシデント対応手順 | catalog-info.yaml、service-catalog サーバー |

### スコア計算ロジック

#### カテゴリ別スコア計算

各カテゴリのスコアは 0-100 で計算し、重み付き合計で総合スコアを算出する。

**総合スコア** = (信頼性スコア x 0.30) + (セキュリティスコア x 0.25) + (可観測性スコア x 0.20) + (品質スコア x 0.15) + (運用性スコア x 0.10)

### DORA メトリクス統合

#### 4 つのキーメトリクス

| メトリクス | 定義 | Elite | High | Medium | Low |
| --- | --- | --- | --- | --- | --- |
| デプロイ頻度 | 本番環境へのデプロイ回数（週あたり） | 日次以上 | 週 1-6 回 | 月 1-4 回 | 月 1 回未満 |
| 変更リードタイム | コミットから本番デプロイまでの時間 | 1 時間未満 | 1 日未満 | 1 週間未満 | 1 ヶ月以上 |
| 変更失敗率 | デプロイ後にロールバックまたは修正が必要になった割合 | 0-5% | 5-10% | 10-15% | 15% 超 |
| MTTR | 障害発生から復旧までの平均時間 | 1 時間未満 | 1 日未満 | 1 週間未満 | 1 ヶ月以上 |

#### 収集方法と計算ロジック

| メトリクス | データソース | 計算方法 |
| --- | --- | --- |
| デプロイ頻度 | GitHub Actions（deploy ワークフロー完了イベント） | 過去 7 日間の production デプロイ成功数 |
| 変更リードタイム | GitHub API（PR マージ日時 → deploy ワークフロー完了日時） | PR マージからデプロイ完了までの中央値（過去 30 日間） |
| 変更失敗率 | Argo Rollouts API（ロールバックイベント）+ GitHub API（hotfix PR 数） | (ロールバック数 + hotfix 数) / 総デプロイ数 x 100（過去 30 日間） |
| MTTR | Prometheus アラート履歴（firing → resolved の時間差） | アラート firing から resolved までの平均時間（過去 30 日間） |

```yaml
# DORA メトリクス収集用 Prometheus Recording Rules
groups:
  - name: dora-metrics
    interval: 1h
    rules:
      # デプロイ頻度（過去 7 日間の本番デプロイ数）
      - record: dora:deploy_frequency:7d
        expr: |
          count(
            github_actions_workflow_run_completed_total{
              workflow="deploy",
              environment="production",
              conclusion="success"
            }
          ) by (service)

      # MTTR（アラート firing → resolved の平均時間）
      - record: dora:mttr:30d
        expr: |
          avg(
            ALERTS_FOR_STATE{alertstate="resolved"}
            - ALERTS_FOR_STATE{alertstate="firing"}
          ) by (service)
```

| `k1s0 portal scorecard <name>` | `GET /api/proxy/k1s0-catalog/api/v1/services/<id>/scorecard` | スコアカードの表示 |

| `k1s0 portal tech-radar` | `GET /api/proxy/k1s0-tech-radar/api/v1/entries` | Tech Radar の表示 |

### テンプレート同期メカニズム

k1s0 CLI のテンプレート（50+）と Backstage の Software Templates を同期する。

| 同期方向 | トリガー | 処理内容 |
| --- | --- | --- |
| CLI → Backstage | Git push（templates/ ディレクトリの変更） | GitHub Actions で template.yaml を自動生成し、Backstage カタログに登録 |
| Backstage → CLI | テンプレート実行時 | Backstage Scaffolder が k1s0 CLI サーバー API を呼び出し |

```yaml
# .github/workflows/sync-templates.yaml
name: Sync Templates to Backstage
on:
  push:
    paths:
      - 'templates/**'
      - 'CLI/templates/**'

jobs:
  sync:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Generate Backstage Template Entities
        run: |
          # k1s0 CLI テンプレートから Backstage template.yaml を生成
          k1s0 template sync --format backstage --output ./backstage-templates/
      - name: Register Templates
        run: |
          # Backstage カタログに登録
          for template in ./backstage-templates/*.yaml; do
            curl -X POST \
              "${BACKSTAGE_URL}/api/catalog/locations" \
              -H "Authorization: Bearer ${BACKSTAGE_TOKEN}" \
              -H "Content-Type: application/json" \
              -d "{\"type\": \"url\", \"target\": \"${GITHUB_SERVER_URL}/${GITHUB_REPOSITORY}/blob/main/${template}\"}"
          done
```
