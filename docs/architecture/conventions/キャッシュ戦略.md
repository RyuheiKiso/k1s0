# キャッシュ戦略

## 概要

k1s0-cache ライブラリ（`regions/system/library/rust/cache/`）が提供する2種類のバックエンドの使い分け方針を定義する。
誤ったバックエンド選択はデータ不整合やスケーリング問題を引き起こすため、本ドキュメントを参照すること。

## バックエンド選択基準

| バックエンド | 対象 | 適用場面 | 制限 |
|------------|------|---------|------|
| **InMemory** (`InMemoryCacheClient`) | プロセスローカル | 単一インスタンス、テスト、短命な参照データ | プロセス再起動でデータ消失 |
| **Redis** (`RedisCacheClient`) | 分散共有 | マルチインスタンス、セッション共有、大容量データ | Redis インフラが必要 |

## CacheClient トレイト API

```rust
// k1s0-cache ライブラリの CacheClient トレイト定義（全バックエンド共通）
pub trait CacheClient: Send + Sync {
    // キーに対応する値を取得する（存在しない場合は None）
    async fn get(&self, key: &str) -> Result<Option<String>, CacheError>;
    // キーに値をセットする（ttl: None で期限なし）
    async fn set(&self, key: &str, value: &str, ttl: Option<Duration>) -> Result<(), CacheError>;
    // キーを削除する（削除成功: true、存在しない: false）
    async fn delete(&self, key: &str) -> Result<bool, CacheError>;
    // キーの存在確認
    async fn exists(&self, key: &str) -> Result<bool, CacheError>;
    // キーが存在しない場合のみセット（原子的操作、分散ロックの基盤）
    async fn set_nx(&self, key: &str, value: &str, ttl: Duration) -> Result<bool, CacheError>;
    // キーの TTL を更新する（スライディングウィンドウ用）
    async fn expire(&self, key: &str, ttl: Duration) -> Result<bool, CacheError>;
}
```

## キャッシュ用途別 TTL ガイドライン

| データ種別 | 推奨 TTL | バックエンド | 例 |
|-----------|---------|------------|-----|
| JWKS 公開鍵 | 5分 | InMemory | 認証ライブラリ内部 |
| マスターデータ | 10〜15分 | Redis | 商品カテゴリ、設定値 |
| ユーザーセッション | 30分〜2時間 | Redis（セッションサーバーが管理） | BFF セッション |
| フィーチャーフラグ | 30秒〜1分 | InMemory | 頻繁に変更されない場合 |
| 集計・計算結果 | 1〜5分 | Redis | ダッシュボード集計 |
| API レスポンスキャッシュ | 30秒 | Redis | 外部 API 呼び出し結果 |

## キャッシュ無効化戦略

### 1. TTL ベース（推奨デフォルト）
有効期限が切れたら自動的に無効化される。実装が最もシンプル。

### 2. Write-through（マスターデータ更新時）
```rust
// マスターデータ更新時は即座にキャッシュを無効化する
async fn update_config(&self, key: &str, value: &str) -> Result<(), Error> {
    // まず DB に書き込む
    self.repository.update(key, value).await?;
    // キャッシュを即座に削除して次回読み込み時に最新値を取得させる
    self.cache.delete(&format!("config:{}", key)).await?;
    Ok(())
}
```

### 3. イベント駆動（Kafka 連携）
Kafka の設定変更イベントを受信したサービスが自身のキャッシュを無効化する。
複数サービスが同じデータをキャッシュしている場合に有効。

## テスト方針

```rust
// テスト環境では InMemory バックエンドを使用する（Redis 依存を排除）
#[cfg(test)]
mod tests {
    use k1s0_cache::memory::InMemoryCacheClient;

    #[tokio::test]
    async fn test_cache_logic() {
        // InMemory バックエンドでユニットテストを実行する
        let cache = Arc::new(InMemoryCacheClient::new());
        // ... テストロジック
    }
}

// mock feature を使用したモックテスト
// Cargo.toml に features = ["mock"] を追加することで MockCacheClient が生成される
```

## 4言語での利用

詳細な使用例は `docs/libraries/_common/how-to-use.md` を参照すること。

| 言語 | ライブラリ | バックエンド切り替え方法 |
|------|---------|-------------------|
| Rust | `k1s0-cache` | `features = ["redis"]` / InMemory はデフォルト |
| Go | `cache` | 構造体の実装を DI で切り替え |
| TypeScript | `@k1s0/cache` | コンストラクタ引数で切り替え |
| Dart | `k1s0_cache` | コンストラクタ引数で切り替え |
