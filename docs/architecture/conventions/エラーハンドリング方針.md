# エラーハンドリング方針

k1s0 プロジェクトにおける Rust のエラーハンドリング方針を定義する。`anyhow` と `thiserror` の使い分けを明確にし、エラー型の一貫性を保つ。

## 基本原則

| # | 原則 | 説明 |
|---|------|------|
| 1 | ライブラリは型付きエラー | 呼び出し元がパターンマッチでハンドリングできるよう `thiserror` を使用 |
| 2 | アプリケーションは柔軟に | サーバーのエントリーポイント・ハンドラーでは `anyhow::Result` を使用可 |
| 3 | エラーチェーンの保持 | `cause` / `source` を失わない変換を徹底する |
| 4 | コンテキスト付与 | 外部エラーには `.context()` で発生箇所の情報を付与する |

## ライブラリ（library/）: thiserror を使用

呼び出し元が型によるエラーハンドリングを行えるよう、型付きエラーを定義する。

### 定義パターン

```rust
// thiserror で列挙型エラーを定義する
#[derive(Debug, thiserror::Error)]
pub enum AuthError {
    // トークンの有効期限切れ
    #[error("トークンが有効期限切れです: {0}")]
    TokenExpired(String),

    // トークンの署名検証失敗
    #[error("トークンの署名が無効です")]
    InvalidSignature,

    // 外部サービスとの通信エラー（source を保持）
    #[error("認証プロバイダへの接続に失敗しました")]
    ProviderConnection(#[from] reqwest::Error),
}
```

### ルール

- `#[derive(thiserror::Error)]` で列挙型エラーを定義する
- 各バリアントは具体的なエラー原因を表現する
- 外部クレートのエラーは `#[from]` で自動変換を提供する
- エラーメッセージは日本語で記述する

## アプリケーション（server/）: anyhow を使用可

サーバーのエントリーポイント、ハンドラー等では `anyhow::Result` を使用できる。

### 使用パターン

```rust
use anyhow::{Context, Result};

// ハンドラーでの anyhow 使用例
async fn initialize_server(config: &Config) -> Result<()> {
    // 外部エラーに .context() でコンテキストを付与する
    let pool = PgPool::connect(&config.database_url)
        .await
        .context("データベースへの接続に失敗しました")?;

    // ライブラリの型付きエラーも anyhow に変換される
    let auth = AuthService::new(&config.auth)
        .context("認証サービスの初期化に失敗しました")?;

    Ok(())
}
```

### ルール

- `ServiceError` 等の型付きエラーが定義済みの場合はそちらを優先する
- 外部エラー（DB, HTTP, etc.）は `.context()` でコンテキストを付与する
- `unwrap()` は禁止（`clippy::unwrap_used = "deny"` で強制）

## 境界での変換

ライブラリエラーからサーバーエラーへの `From` 変換を実装し、エラーチェーンを保持する。

```rust
// ライブラリエラー → サーバーエラーへの変換
impl From<AuthError> for ServiceError {
    fn from(err: AuthError) -> Self {
        match err {
            // クライアント起因のエラーは 401 にマッピング
            AuthError::TokenExpired(_) | AuthError::InvalidSignature => {
                ServiceError::Unauthorized(err.to_string())
            }
            // インフラ起因のエラーは 500 にマッピング（source を保持）
            AuthError::ProviderConnection(e) => {
                ServiceError::Internal(anyhow::Error::new(e).context("認証プロバイダエラー"))
            }
        }
    }
}
```

## service / business 層のドメイン Repository トレイト

service 層および business 層のドメイン Repository トレイトでは、`anyhow::Result` ではなく専用のドメインエラー型を返すこと。

### 方針

| 層 | Repository トレイトの戻り値 | 説明 |
|----|--------------------------|------|
| domain/repository | `Result<T, TaskError>` 等のドメインエラー型 | 呼び出し元（usecase 層）がエラー種別に応じた処理（404 返却・ドメイン例外スロー等）を行えるようにする |
| usecase / adapter | `anyhow::Result` 使用可 | ユースケース境界ではエラーを `anyhow` に変換して伝播してよい |

### 対応するドメインエラー型

| サービス | ドメインエラー型 | ファイル |
|---------|--------------|---------|
| task | `TaskError` | `domain/error.rs` |
| board | `BoardError` | `domain/error.rs` |
| activity | `ActivityError` | `domain/error.rs` |

### 実装例

```rust
// domain/repository.rs — Repository トレイトはドメインエラー型を使用する
#[async_trait]
pub trait TaskRepository: Send + Sync {
    // ドメインエラー型を返すことで呼び出し元がエラー種別を判定できる
    async fn find_by_id(&self, id: &TaskId) -> Result<Option<Task>, TaskError>;
    async fn save(&self, task: &Task) -> Result<(), TaskError>;
}

// thiserror でドメインエラー型を定義する
#[derive(Debug, thiserror::Error)]
pub enum TaskError {
    // タスクが見つからない場合（呼び出し元で 404 にマッピング）
    #[error("タスクが見つかりません: {0}")]
    NotFound(TaskId),

    // ステータス遷移が不正な場合（呼び出し元で 422 にマッピング）
    #[error("ステータス遷移が不正です: {from:?} -> {to:?}")]
    InvalidStatusTransition { from: TaskStatus, to: TaskStatus },

    // DB 等のインフラエラー（呼び出し元で 500 にマッピング）
    #[error("データベースエラー: {0}")]
    Database(#[from] sqlx::Error),
}
```

## 段階的移行計画

現在のコードベースでは `anyhow` と `thiserror` の使い分けが混在している箇所がある。

| フェーズ | 対象 | 方針 |
|----------|------|------|
| 即時 | 新規コード | 上記方針に従って実装する |
| 段階的 | 既存コード | 該当箇所を変更する際に方針に合わせて移行する |
| 将来 | 全体 | clippy カスタムルールで方針違反を検出する |

### 移行時の注意事項

- 既存の `anyhow::Result` をライブラリから除去する際は、呼び出し元の変更も同時に行う
- `From` 実装を追加する際は、エラーチェーン（`source()`）が途切れないことを確認する
- 移行対象が多い場合は `tasks/todo.md` に追跡項目を記載する
