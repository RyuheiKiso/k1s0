# 新規サービス監視設定ガイド

新しいサービスをk1s0の監視基盤に組み込む手順を示す。
CLIによる自動生成から手動設定まで、必要な手順をステップバイステップで説明する。

---

## 前提条件

- サービスに `/metrics` エンドポイントが実装済みであること
- telemetry ライブラリが組み込まれていること（[トレーシング設計](./トレーシング設計.md) 参照）
- サービスの Tier が決定していること（system / business / service）

---

## チェックリスト

以下の手順をすべて実施すること:

```
□ Step 1: /metrics エンドポイントの確認
□ Step 2: CLI でアラートルール・ServiceMonitor を生成
□ Step 3: 生成ファイルを確認・調整
□ Step 4: Grafana ダッシュボードにサービスを追加
□ Step 5: Runbook を作成
□ Step 6: ローカル環境での動作確認
□ Step 7: PR でレビュー → マージ
```

---

## Step 1: /metrics エンドポイントの確認

サービスが Prometheus 形式のメトリクスを公開しているか確認する:

```bash
# ローカル環境での確認
curl -sf http://localhost:{port}/metrics | head -20

# 以下のメトリクスが含まれていることを確認
# - http_requests_total
# - http_request_duration_seconds_bucket
# - process_cpu_seconds_total
# - process_resident_memory_bytes
```

telemetry ライブラリを使用している場合、これらは自動的に公開される:
- Go: `regions/system/library/go/telemetry/`
- Rust: `regions/system/library/rust/telemetry/`
- TypeScript: `regions/system/library/typescript/telemetry/`
- Dart: `regions/system/library/dart/telemetry/`

---

## Step 2: CLI でアラートルール・ServiceMonitor を生成

```bash
# アラートルールの生成（Tier を指定する）
k1s0 generate observability \
  --service-name {service-name} \
  --namespace k1s0-{system|business|service} \
  --tier {system|business|service}

# ServiceMonitor の生成
k1s0 generate servicemonitor \
  --service-name {service-name} \
  --namespace k1s0-{system|business|service} \
  --tier {system|business|service}
```

生成されるファイル:
- `infra/observability/{service-name}/prometheusrule.yaml`
- `infra/observability/{service-name}/servicemonitor.yaml`

---

## Step 3: 生成ファイルを確認・調整

### アラートルール（prometheusrule.yaml）の確認ポイント

```yaml
# 生成されたファイルを確認し、必要に応じて調整する
# 特に以下を確認:
# 1. 閾値が Tier に合っているか（system: 1%, business/service: 5%）
# 2. runbook_url が正しいか
# 3. サービス固有のアラートが必要な場合は追加する
```

Tier 別の期待する閾値:

| Tier | エラー率 critical | レイテンシ warning |
|------|-----------------|-----------------|
| system | > 1%（5分継続） | P99 > 500ms（5分継続） |
| business | > 5%（5分継続） | P99 > 1s（5分継続） |
| service | > 5%（5分継続） | P99 > 1s（5分継続） |

### ServiceMonitor（servicemonitor.yaml）の確認ポイント

```yaml
# scrape interval が Tier に合っているか確認:
# - system Tier: interval: 15s
# - business Tier: interval: 30s
# - service Tier: interval: 30s
```

### ローカル Prometheus への追加（開発環境のみ）

```yaml
# infra/docker/prometheus/prometheus.yaml に追加
scrape_configs:
  # ...
  - job_name: '{service-name}'
    static_configs:
      - targets: ['{service-name}:{metrics-port}']
    metrics_path: /metrics
```

---

## Step 4: Grafana ダッシュボードにサービスを追加

### 方法 A: CLI で自動生成（推奨）

```bash
k1s0 generate grafana \
  --service-name {service-name} \
  --tier {system|business|service}
```

生成されるファイル:
- `infra/observability/grafana/dashboards/{service-name}-overview.json`

ローカル環境用:
- `infra/docker/grafana/dashboards/{service-name}-overview.json`

### 方法 B: 既存ダッシュボードに変数を追加

概要ダッシュボード（`overview.json`）は全サービスのメトリクスを自動表示する設計のため、
追加作業は不要な場合がある。

---

## Step 5: Runbook を作成

サービス固有の対応手順が必要な場合、Runbook を作成する:

```bash
# テンプレートをコピー
cp docs/architecture/observability/runbooks/_template.md \
   docs/architecture/observability/runbooks/services/{service-name}/overview.md

# 以下の内容を記入:
# - サービス固有の確認コマンド
# - よくある障害パターンと対処法
# - エスカレーション先
```

サービス固有の対応手順がない場合は、共通 Runbook（`runbooks/common/`）で十分。

---

## Step 6: ローカル環境での動作確認

```bash
# 可観測性スタックを起動
docker compose --profile infra --profile observability up -d

# サービスを起動
docker compose --profile {service-profile} up -d

# Prometheus でメトリクスが取得できているか確認
# http://localhost:9090 → Status → Targets → {service-name} が UP か確認

# Grafana でダッシュボードが表示されるか確認
# http://localhost:3200 → Dashboards → {service-name}

# アラートルールが読み込まれているか確認
# http://localhost:9090 → Alerts → {service-name} のアラートが表示されるか確認
```

---

## Step 7: PR でレビュー → マージ

PR に以下を含めること:

```
## 監視設定の追加

### 追加したファイル
- infra/observability/{service-name}/prometheusrule.yaml
- infra/observability/{service-name}/servicemonitor.yaml
- infra/docker/grafana/dashboards/{service-name}-overview.json（ローカル用）
- infra/docker/prometheus/prometheus.yaml（ローカル用 scrape 追加）

### 確認済み内容
- [ ] /metrics エンドポイントが正常に動作している
- [ ] Prometheus でメトリクスが取得できている
- [ ] アラートルールの構文が正しい（promtool check rules で確認）
- [ ] Grafana ダッシュボードが表示されている
- [ ] Tier 別の閾値が正しく設定されている
```

---

## トラブルシューティング

### Prometheus でメトリクスが取得できない

```bash
# ServiceMonitor が正しく設定されているか確認
kubectl get servicemonitor -n observability
kubectl describe servicemonitor {service-name} -n observability

# サービスのラベルが ServiceMonitor のセレクターと一致しているか確認
kubectl get svc -n {namespace} -l app={service-name}
```

### アラートが発火しない

```bash
# PrometheusRule が読み込まれているか確認
kubectl get prometheusrule -n {namespace}

# Prometheus のアラートルールを確認
# http://prometheus.k1s0.internal/rules → {service-name} のルールを確認
```
