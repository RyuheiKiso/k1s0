# インシデント管理設計

k1s0 システムにおけるインシデント対応ワークフロー、エスカレーションパス、
オンコールローテーション、ポストモーテムプロセスを定義する。

関連ドキュメント: [可観測性設計.md](./可観測性設計.md) | [監視アラート設計.md](./監視アラート設計.md) | [SLO設計.md](./SLO設計.md)

## インシデント重要度分類

### SEV レベル定義

| SEV | 名称 | 基準 | 対応開始 SLA | 例 |
|-----|------|------|-------------|---|
| **SEV1** | 重大インシデント | サービス全体停止 / SLO バーンレート Critical | 15分以内 | auth-server ダウン、全エラー率 > 50% |
| **SEV2** | 重大障害 | 主要機能の部分障害 / SLO バーンレート Warning | 30分以内 | 特定エンドポイントのエラー率急増、DB プール枯渇 |
| **SEV3** | 軽微な問題 | SLO への影響がない劣化 / 軽微な機能障害 | 翌営業日 | warning アラート、証明書期限 30日前 |

### SEV レベルへのマッピング（アラート → SEV）

| アラート重要度 | バーンレート | 自動 SEV 判定 |
|-------------|------------|------------|
| critical + バーンレート > 14.4x | - | SEV1 |
| critical | - | SEV1 または SEV2（影響範囲で判断） |
| warning + バーンレート > 6x | - | SEV2 |
| warning | - | SEV3 |

## インシデント対応フロー

### 全体フロー図

```
[アラート発火]
    │
    ▼
[オンコール担当が受信]
    │
    ├─── SEV1/SEV2 → [初動対応 開始（SLA 内）]
    │                    │
    │                    ▼
    │               [Runbook 参照]
    │               https://github.com/k1s0/k1s0/blob/main/
    │               docs/architecture/observability/runbooks/
    │                    │
    │                    ▼
    │               [影響確認・原因特定]
    │                    │
    │                    ├─── 解決見通しあり → [復旧作業]
    │                    │                        │
    │                    │                        ▼
    │                    │                   [復旧確認・アラート解消]
    │                    │                        │
    │                    │                        ▼
    │                    │                   [ポストモーテム作成]
    │                    │
    │                    └─── 解決見通しなし → [エスカレーション]
    │
    └─── SEV3 → [チケット起票 → 翌営業日対応]
```

### 対応ステップ詳細

#### Step 1: 初動確認（アラート受信後 5分以内）

1. アラートの内容を確認（何が、どこで、いつから）
2. Grafana ダッシュボードで現在のメトリクス状況を確認
3. SEV レベルを判定する
4. SEV1/SEV2 の場合: 対応開始を Teams に報告

```
# Teams への初動報告テンプレート（SEV1/SEV2）
[インシデント開始]
- アラート: {アラート名}
- SEV: SEV1/SEV2
- 影響サービス: {サービス名}
- 対応者: {自分の名前}
- 状況: 調査中
```

#### Step 2: 調査・復旧（SEV1: 15分以内、SEV2: 30分以内）

1. 対応する Runbook を開く（アラート通知の `runbook_url` から）
2. Runbook の「初動対応」セクションの手順を実施
3. 原因が特定できたら「復旧手順」を実施
4. 定期的（15〜30分ごと）に状況を Teams に更新

```
# Teams への進捗報告テンプレート
[インシデント更新] {時刻}
- 状況: {調査中/復旧作業中/復旧確認中}
- 判明したこと: {原因や状況}
- 次のアクション: {次に行うこと}
- 解決見通し: {あり/なし/調査中}
```

#### Step 3: 復旧確認

1. アラートが resolve されたことを確認
2. 影響を受けていたメトリクスが正常値に戻ったことを確認
3. エラーバジェット残量を確認（[SLO ダッシュボード](http://grafana.k1s0.internal/d/k1s0-slo/slo-dashboard)）

```
# Teams への解消報告テンプレート
[インシデント解消] {時刻}
- アラート: {アラート名} 解消
- 原因: {根本原因}
- 対応内容: {実施した対応}
- エラーバジェット消費: 約 {X} 分
- ポストモーテム: {SEV1 は必須 / SEV2 は任意 / SEV3 は不要}
```

#### Step 4: ポストモーテム（SEV1 は必須、SEV2 は任意）

- インシデント解消後 24時間以内にポストモーテムドキュメントを作成する
- テンプレート: [runbooks/postmortem-template.md](./runbooks/postmortem-template.md)
- 保存先: `docs/architecture/observability/postmortems/YYYY-MM-DD-{アラート名}.md`

## エスカレーションパス

### エスカレーション条件

以下の条件に該当する場合、次のレベルにエスカレーションする:

| 条件 | エスカレーション先 |
|------|------------------|
| 15分以内に復旧の見通しが立たない（SEV1） | チームリード |
| 30分以内に復旧の見通しが立たない（SEV2） | チームリード |
| 複数サービスに連鎖障害が発生 | チームリード + アーキテクト |
| データ損失・セキュリティ侵害の可能性 | マネージャー + セキュリティチーム |
| SLA 違反が確定（月間 SLA 未達） | マネージャー |

### エスカレーション時の情報伝達

エスカレーション時は以下の情報を伝える:

```
【エスカレーション報告】
- インシデント開始時刻: {時刻}
- SEV レベル: SEV{1/2}
- 影響サービス: {サービス名}
- 現在の状況: {状況}
- これまでに試みた対応: {対応内容}
- 解決を阻んでいる問題: {問題点}
- 必要な支援: {何を手伝ってほしいか}
```

## オンコールローテーション設計

### 体制

- **Primary**: 最初に対応する担当者（アラートを最初に受信）
- **Secondary**: Primary が15分以内に応答しない場合のバックアップ

### ローテーション

- 1週間ごとにローテーション（月曜 09:00 に交代）
- ローテーション前に引き継ぎチェックリストを実施

### 引き継ぎチェックリスト

```
□ 現在進行中のインシデントがないか確認
□ 直近の SLO バーンレート状況を確認
  - Grafana → SLO ダッシュボード
□ 前週のポストモーテムがあれば読む
□ 監視設定の変更が最近あったか確認
  - git log --oneline infra/observability/ -10
□ Teams の #alert-critical / #alert-warning チャンネルを確認
□ 連絡先（Primary/Secondary）を Teams に告知
```

### オンコール中の注意事項

- Teams 通知を確認できる状態を維持する
- 対応開始時は必ず Teams で報告する
- 独力で解決できないと判断したら早めにエスカレーション

## 通知チャネル設計

### 現在の通知構成

```
Alertmanager
    ├── critical → prometheus-msteams → Teams #alert-critical
    ├── warning  → prometheus-msteams → Teams #alert-warning
    └── default  → prometheus-msteams → Teams #alert-default
```

### 今後の拡張計画

深夜帯の critical アラートを見逃さないため、以下の通知チャネルの追加を検討する:

| チャネル | 用途 | 優先度 |
|---------|------|--------|
| PagerDuty / Opsgenie | SEV1 の電話エスカレーション | 高 |
| Slack | Teams の補完（任意） | 中 |
| メール | SEV3 のチケット通知 | 低 |

> 実際の導入時は `infra/observability/alertmanager/alertmanager-config.yaml` の receiver を拡張する。

## SLA 違反時の対応

[SLO設計.md](./SLO設計.md) の SLA 違反エスカレーション条件を参照:

| 条件 | アクション |
|------|-----------|
| リアルタイム可用性低下 | 即座にオンコール担当に通知（Alertmanager → Teams） |
| 月間 SLA 未達 | 月次レビューで原因分析と再発防止策を検討、ポストモーテム実施 |

月次レビュー:
- エラーバジェット消費量の確認
- SLA 未達サービスの根本原因分析
- アクションアイテムの追跡

## ポストモーテムプロセス

### 実施タイミング

| SEV | ポストモーテム |
|-----|-------------|
| SEV1 | 必須（解消後 24時間以内に作成） |
| SEV2 | 任意（再発リスクがある場合は実施） |
| SEV3 | 不要 |

### 保存先と命名規則

```
docs/architecture/observability/postmortems/
├── YYYY-MM-DD-{alert-name}-{short-description}.md
例:
├── 2026-03-21-SystemServiceHighErrorRate-payment-service-db-connection.md
└── 2026-03-15-SLOBurnRateCritical-auth-server-memory-leak.md
```

### ポストモーテムテンプレート

[runbooks/postmortem-template.md](./runbooks/postmortem-template.md) を使用する。
