# ADR-0014: auth-db マイグレーション往復リネームの記録と修正方針

## ステータス

承認済み

## コンテキスト

`regions/system/database/auth-db/migrations/` の 009〜013 番マイグレーション群において、
`audit_logs` テーブルのカラムに対して以下の往復リネームが発生した。

| マイグレーション番号 | 内容 |
|---------------------|------|
| 009 `align_audit_log_columns` | `detail → metadata`、`created_at → recorded_at` へリネーム |
| 010 `fix_audit_log_columns` | 009 のリネームを元に戻す（`metadata → detail`、`recorded_at → created_at`）|

この往復により、マイグレーション履歴が汚染された状態となった。
既に本番・ステージング環境で 009 と 010 が適用済みであったため、単純なマイグレーション削除は不可能であった。

013 番マイグレーション `squash_audit_columns` は、009/010 の往復が正しく完了し
現在のカラム名（`detail`、`created_at`）が元の設計通りであることを確認する no-op として追加された。

## 決定

1. **009 と 010 を no-op 化する**: 両マイグレーションの内容を `DO $$ BEGIN END $$;` に置き換え、
   既存 DB への適用済みフラグを保持しつつスキーマ変更を無効化した。
2. **013 で no-op 確認を記録する**: `SELECT 1;` による no-op マイグレーションで、
   往復が完了し現在のスキーマが正しい状態であることをバージョン履歴に刻む。
3. **マイグレーション番号は欠番にしない**: 履歴の連続性を維持するため、009〜013 の番号をすべて存在させる。

## 理由

- 既に適用済み環境が存在するため、009/010 を物理削除するとマイグレーションツールが
  「未知の適用済みマイグレーション」を検知してエラーを返す恐れがある。
- no-op 化することで、適用済み環境でも新規環境でも `migrate up` が冪等に成功する。
- 013 による確認 no-op は、将来の開発者が履歴を読んだときに往復の経緯と
  現在のスキーマ状態を把握できるようにするためのドキュメントとしての役割を持つ。

## 影響

**ポジティブな影響**:

- 既存 DB への後方互換性を維持しつつ、新規 DB でも一貫したスキーマが得られる
- `migrate up` が全環境で冪等に成功する
- 往復の経緯が no-op コメントとして履歴に残り、将来の開発者が混乱しない

**ネガティブな影響・トレードオフ**:

- マイグレーション履歴（009〜013）は「何もしない」エントリを複数含む。
  ファイル数が見かけ上増え、履歴が若干読みづらくなる
- no-op マイグレーションは純粋にスキーマ変更の追跡という観点では意味を持たないため、
  ツールによっては警告が出る場合がある

## 代替案

| 案 | 概要 | 採用しなかった理由 |
|----|------|-----------------|
| 009/010 を物理削除 | ファイルを削除して番号を飛ばす | 適用済み環境でマイグレーションツールがエラーを返すリスクがある |
| 009/010 を revert して新規マイグレーションに統合 | 既存番号を上書きし squash する | Git 履歴の改ざんに相当し、チーム開発で混乱を招く |
| down マイグレーションでロールバック後に削除 | 本番 DB に対して `migrate down` を実行 | 本番へのダウンタイムが発生し、リスクが高い |

## 教訓

- **マイグレーションは原子的に計画すること**: カラムリネームを行う前に、
  新旧カラム名の影響範囲（アプリケーションコード・API・ドキュメント）を一括レビューする。
- **往復リネームは避けること**: 一度リネームしたカラムを元に戻すと、
  履歴が汚染され修正コストが増大する。設計段階で確定させること。
- **スキーマ変更は小さく、頻度を減らすこと**: 一つのマイグレーションで複数カラムを
  同時にリネームする場合は特に慎重にレビューを行うこと。
