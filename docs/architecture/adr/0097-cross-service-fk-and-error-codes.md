# ADR-0097: クロスサービス FK 不在の設計根拠と Proto エラーコード方針（M-002 + L-014 監査対応）

## ステータス

承認済み

## コンテキスト

### クロスサービス FK 不在（M-002）

外部技術監査（M-002）において、マイクロサービス間でクロス DB の外部キー制約が存在しないことが指摘された。具体的には以下のようなケースが挙げられた:

- `tenant_members.user_id` が `auth.users` テーブルを参照する外部キーを持たない
- `activity.task_id` が `task.tasks.id` を参照する外部キーを持たない
- `board.task_id` が `task.tasks.id` を参照する外部キーを持たない

これらは各サービスが独立した PostgreSQL スキーマ（または将来的に独立した DB）を持つ設計に起因する。

### Proto エラーコード不統一（L-014）

外部技術監査（L-014）において、各サービスの gRPC Proto ファイルで定義されるエラーコードが統一された enum として管理されていないことが指摘された。具体的には、各サービスが `SYS_{SERVICE}_{ERROR}` 形式で独自にエラーコードを定義しており、共通 enum が存在しない。

## 決定

### クロスサービス FK は意図的に設けない

マイクロサービスアーキテクチャの原則に従い、クロスサービス FK 制約を DB レベルで設けない。サービス間の参照整合性は **Saga パターン** および **イベント駆動アーキテクチャ** によってアプリケーション層で維持する。

### Proto エラーコードは各サービスが独自定義を維持する

各サービスは `SYS_{SERVICE}_{ERROR}` 形式のエラーコードを独自に定義・管理する。全サービス横断の共通 enum 化は将来の検討課題とする。

## 理由

### クロスサービス FK を設けない理由

1. **サービス独立性の維持**: 別 DB・別スキーマ間の FK 制約はサービスの独立したデプロイを阻害する。サービス A の DB マイグレーションがサービス B に依存するようになると、独立したリリースサイクルが実現できない。

2. **DB 分割・スケールの困難化**: PostgreSQL のクロスデータベース FK は標準 SQL では不可能である。同一 DB 内のクロススキーマ FK は技術的には可能だが、将来の DB 分割（マルチ DB 化・シャーディング）の際に大きな障壁になる。

3. **マイクロサービスアーキテクチャの原則**: 各境界付きコンテキスト（Bounded Context）は自身のデータを完全に所有し管理する責任を持つ。外部の DB テーブルへの直接依存は境界の侵食を引き起こす。

4. **Saga パターンによる補完**: 本システムでは Outbox パターン（[ADR: outbox ライブラリ設計](../../libraries/messaging/outbox.md)）および Saga ワークフローによって、分散トランザクションの整合性を確保している。イベントの発行・消費により、サービス間の結果整合性が保証される。

5. **アプリケーション層バリデーション**: クロスサービス参照の妥当性チェックは API 層で実施する。例えば `task_id` の存在確認は task-server への gRPC 呼び出しで検証する。

### Proto エラーコードを各サービス独自定義にする理由

1. **サービス自律性**: 各サービスのエラー条件はドメイン固有であり、全サービス共通の enum に押し込めると変更の独立性が失われる。

2. **段階的統一の困難さ**: 全 33 サービスのエラーコードを一度に統一 enum 化するには大規模なリファクタリングが必要であり、現時点でのリターンに対してコストが過大である。

3. **命名規約による暗黙的統一**: `SYS_{SERVICE}_{ERROR}` の命名規約が全サービスに適用されているため、ある程度の一貫性は確保されている。

## 影響

**ポジティブな影響**:

- 各サービスを独立してデプロイ・スケールできる
- 将来の DB 分割・マルチクラウド対応への道を開いておける
- 各サービスチームが独立してスキーマ変更を実施できる
- Saga/Outbox パターンの採用により、分散環境での結果整合性を宣言的に管理できる

**ネガティブな影響・トレードオフ**:

- DB レベルの参照整合性が保証されないため、孤立レコード（orphan record）が発生しうる
- データ不整合の検出はアプリケーション層またはバッチ検証ジョブに依存する
- Saga の補償トランザクション（compensating transaction）の設計・実装コストが発生する
- 共通エラーコード enum がないため、クライアント側でのエラーハンドリングにサービス固有の知識が必要

## 代替案

| 案 | 概要 | 採用しなかった理由 |
|----|------|-----------------|
| クロススキーマ FK の設置 | 同一 DB 内でのクロススキーマ FK 制約を設ける | DB 分割・サービス独立デプロイが困難になる。将来の疎結合化を阻害する |
| マテリアライズドビューによる整合性確認 | クロスサービスデータをマテリアライズドビューで結合し整合性チェック | ビューのリフレッシュタイミングと整合性確認の周期に依存し、根本解決にならない |
| 共通エラーコード proto ファイル | 全サービス共通の `common_errors.proto` を作成し enum を一元管理 | 全 33 サービスの移行コストと、共通プロトのバージョン管理複雑度が高い。将来の改善候補として保留 |
| DB レベルのサービスメッシュ（FDW）| PostgreSQL の Foreign Data Wrapper でクロス DB 参照を実現 | 運用複雑度が高く、スケール時の性能特性が不透明。本番採用例が限定的 |
