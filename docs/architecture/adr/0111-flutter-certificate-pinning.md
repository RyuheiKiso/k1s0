# ADR-0111: Flutter モバイルクライアントの証明書ピンニング実装方針

## ステータス

承認済み

## コンテキスト

k1s0 Flutter モバイルクライアントは BFF-Proxy（bff-proxy）と HTTPS 通信を行っている。
現状では TLS 証明書の検証を OS・Flutter ランタイムの標準実装に委ねており、
中間者攻撃（MITM）に対する追加防御が実装されていない。

外部技術監査（2026-04-06）において、以下のリスクが指摘された:

1. **企業プロキシ・ルート証明書インストール攻撃**: 端末に不正なルート CA をインストールされた場合、
   OS の TLS 検証を通過してしまう
2. **クラックツールによる SSL ストリッピング**: モバイルアプリ解析ツール（Charles Proxy, mitmproxy 等）に
   よるトラフィック傍受が容易
3. **動的インスツルメンテーション**: Frida 等のツールによる実行時 TLS 検証バイパス

Flutter のデフォルト実装（`dart:io` の `HttpClient`）は証明書ピンニングを提供しない。

## 決定

Flutter モバイルクライアントに証明書ピンニングを実装する。
実装方式として `http_certificate_pinning`（または `ssl_pinning_plugin`）パッケージを採用し、
公開鍵ハッシュ（SPKI フィンガープリント）によるピンニングを行う。

### 実装方針

#### 1. ピンニング対象

BFF-Proxy（`https://bff.k1s0.example.com`）の証明書公開鍵を SHA-256 ハッシュでピンニングする。

#### 2. ピンの設定方法

```dart
// MEDIUM-FE-003 監査対応: BFF-Proxy の公開鍵ハッシュをピンニングする
// 証明書更新時はこのリストを同時に更新すること（バックアップピンを含む）
const List<String> _pinnedPublicKeyHashes = [
  // プライマリ証明書の SHA-256 フィンガープリント（Base64 エンコード）
  'sha256/AAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAA=',
  // バックアップ証明書（証明書ローテーション時の継続性保証）
  'sha256/BBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBB=',
];
```

#### 3. バックアップピン

証明書ローテーション時のサービス断を防ぐため、常に最低2つのピン（プライマリ + バックアップ）を保持する。
バックアップピンは次回更新予定の証明書公開鍵、または中間 CA 証明書のハッシュとする。

#### 4. 失敗時の動作

ピンニング検証失敗時はネットワークリクエストを中断し、ユーザーに接続エラーを表示する。
サイレントフォールバック（HTTP フォールバック等）は実装しない。

## 理由

- **公開鍵ハッシュ（SPKI）ピンニングを選択した理由**: 証明書ピンニングと比較して、
  証明書更新（有効期限切れによる再発行）時に公開鍵を維持した場合はアプリ更新不要となる。
  公開鍵を差し替える際はバックアップピンがあれば継続性を担保できる。

- **`http_certificate_pinning` パッケージを選択した理由**: Flutter エコシステムで広く使用されており、
  Android（OkHttp）と iOS（NSURLSession）両方のネイティブ実装をラップする。
  `ssl_pinning_plugin` と比較してメンテナンスが活発である。

- **HSTS のみでは不十分な理由**: HSTS はブラウザ向けのメカニズムであり、
  モバイルアプリの MITM 攻撃（特に企業プロキシやルート CA インストール攻撃）に対する
  防御効果は限定的である。証明書ピンニングと組み合わせることで多層防御を実現する。

## 影響

**ポジティブな影響**:

- BFF-Proxy との通信に対する MITM 攻撃リスクの大幅な低減
- 不正ルート CA インストール・SSL ストリッピング攻撃への耐性向上
- 外部監査指摘事項（MEDIUM-FE-003）の解消

**ネガティブな影響・トレードオフ**:

- **証明書更新時の運用コスト増加**: 証明書を更新する際は、バックアップピンを事前にリリースし、
  古い証明書が失効する前にプライマリピンを更新するアプリをリリースする必要がある。
  手順を誤るとユーザーがアプリを使用できなくなる（強制アップデート or 再インストールが必要）。

- **開発・デバッグ環境への影響**: Charles Proxy / mitmproxy を使用したデバッグができなくなる。
  開発環境では `--dart-define=DISABLE_CERTIFICATE_PINNING=true` フラグで無効化する仕組みを設ける。

- **アプリ更新必須**: 証明書公開鍵を変更する場合は必ずアプリ更新が必要となる。
  Google Play Store / Apple App Store のレビュー時間を考慮した証明書ローテーション計画が必要。

### 証明書更新手順

1. 新しい証明書をバックアップピンとして既存アプリにリリース
2. 全ユーザーが新バージョンに移行するのを待つ（最低2週間を推奨）
3. 新しい証明書を BFF-Proxy にデプロイ
4. バックアップピンをプライマリに昇格させた最終アプリをリリース

## 代替案

| 案 | 概要 | 採用しなかった理由 |
|----|------|-----------------|
| HSTS のみ | HTTP Strict Transport Security で証明書の強制 | モバイルアプリの MITM に対する効果が限定的 |
| TrustKit（iOS 専用） | NSURLSession に統合された証明書ピンニング | Flutter クロスプラットフォームに対応しない |
| Network Security Config（Android 専用） | Android の証明書ピンニング設定 | iOS に対応しない（Flutter は両方必要） |
| 証明書ピンニング + ルート CA ピンニング | 証明書と CA の両方をピン | 過剰な制約。CA 証明書変更時の対応コストが高すぎる |
| 無効化（対応見送り） | 現状維持 | 外部監査指摘事項であり、MITM リスクが残存する |
