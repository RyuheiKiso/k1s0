# ADR-0027: DBアプリケーションユーザー権限分離

## ステータス

承認済み

## コンテキスト

全 Rust サービスが `config.yaml` にて `user: k1s0` でデータベース接続を試みていた。しかし PostgreSQL に `k1s0` ロールが存在しなかったため、全 Rust サービスが起動不可能な状態だった。

回避策として `dev` スーパーユーザーでの接続を試みた場合、PostgreSQL の `BYPASSRLS` 属性により Row Level Security (RLS) ポリシーが完全に無効化される。テナント分離・データアクセス制御の根幹を担う RLS が機能しない状態は、セキュリティ要件を満たさない。

## 決定

`k1s0` ロールを以下の属性で作成し、必要なスキーマとテーブルへの最小権限のみ付与する。

- `NOSUPERUSER`: スーパーユーザー権限なし
- `NOBYPASSRLS`: RLS ポリシーを必ず適用
- `LOGIN`: アプリケーションからのログイン許可
- 各スキーマへの `USAGE` 権限
- 各テーブルへの `SELECT`、`INSERT`、`UPDATE`、`DELETE` 権限
- シーケンスへの `USAGE` 権限

## 理由

RLS ポリシーが正常に機能するためには非特権ロールによる接続が必須条件である。スーパーユーザーや `BYPASSRLS` 属性を持つロールは RLS を無視するため、マルチテナント分離が保証できない。最小権限の原則（Principle of Least Privilege）に従い、アプリケーションロールには業務上必要な最小限の権限のみを付与する。

## 影響

**ポジティブな影響**:

- RLS ポリシーが全 Rust サービスで正常に適用される
- 最小権限の原則を満たし、誤操作・不正アクセスによる被害範囲を限定できる
- アプリケーションロールがスキーマ・DDL 操作を行えないため、意図しないテーブル変更を防止
- セキュリティ監査での指摘事項を解消

**ネガティブな影響・トレードオフ**:

- 新規スキーマ・テーブル追加時に `GRANT` 文の追加が必要（管理コスト）
- 初期セットアップに `infra/docker/init-db/00-create-app-user.sql` の実行が必要

## 代替案

| 案 | 概要 | 採用しなかった理由 |
|----|------|-----------------|
| 案 A: dev スーパーユーザーを継続使用 | 既存の `dev` スーパーユーザーでアプリが接続 | `BYPASSRLS` により RLS が完全無効化され、テナント分離が機能しない。セキュリティ要件を満たさないため不可 |
| 案 B: スキーマオーナーとしてのロール | `k1s0` ロールにスキーマの所有権を付与 | スキーマオーナーは DDL 操作が可能になるため最小権限の原則に反する |
