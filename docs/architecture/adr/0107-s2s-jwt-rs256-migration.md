# ADR-0107: サービス間 JWT の RS256 非対称鍵への移行

## ステータス

承認済み

## コンテキスト

現在、Kong ゲートウェイのサービス間通信（S2S: Service-to-Service）では
JWT プラグインを使用した Consumer 認証を行っている。

しかし、現行の設定には以下のセキュリティ上の問題が存在する:

1. **HS256 対称鍵の使用**: HS256（HMAC-SHA256）は対称鍵アルゴリズムであり、
   署名と検証に同一の共有シークレットを使用する。
   この共有シークレットが漏洩した場合、任意の JWT を偽造できる。

2. **予測可能な `key` フィールド**: Kong JWT Consumer の `key` フィールドが
   静的文字列 `"service-to-service"` に設定されており、高度に予測可能な状態である。
   `key` フィールドは JWT の `iss`（issuer）クレームと照合されるため、
   攻撃者が `iss: "service-to-service"` で署名した JWT を試行できる。

これらの問題により、内部ネットワーク内での S2S トークン偽造リスクが存在する。

## 決定

段階的に HS256 から RS256 非対称鍵方式に移行する。移行は3フェーズで実施する。

### Phase 1（即時対応）: key フィールドを UUID に変更

Kong JWT Consumer の `key` フィールドを予測不可能な UUID に変更する。

```yaml
# 変更前
key: "service-to-service"

# 変更後
key: "a1b2c3d4-e5f6-7890-abcd-ef1234567890"  # ランダム UUID
```

JWT 生成側（各サービス）の `iss` クレームも対応する UUID に更新する。
Vault の `infra/vault/auth/auth.tf` で管理される S2S 用シークレットに UUID を設定する。

### Phase 2（RS256 移行）: 非対称鍵ペアの生成と Kong 設定

1. Vault で RSA 2048-bit 鍵ペアを生成する（`vault write pki/issue/s2s-jwt`）
2. Kong JWT Consumer に公開鍵（`rsa_public_key`）を登録する
3. 各サービスが Vault から秘密鍵を取得して JWT に RS256 署名する
4. Kong が公開鍵で検証する

### Phase 3（HS256 削除）: 対称鍵設定の削除

全サービスが RS256 に移行完了後、Kong から HS256 関連設定を削除する。

## 理由

- **最小権限の原則**: RS256 では署名（秘密鍵）と検証（公開鍵）が分離される。
  各サービスは自身の秘密鍵のみを持ち、他サービスの秘密鍵にアクセスできない。

- **侵害の局所化**: HS256 では共有シークレットの漏洩が全サービスへの影響を与えるが、
  RS256 では特定サービスの秘密鍵漏洩の影響を局所化できる。

- **ゼロトラストアーキテクチャとの整合**: 内部ネットワークの信頼を前提としない設計において、
  非対称鍵による S2S 認証はベストプラクティスである。

- **Phase 1 の即時対応**: `key` フィールドの UUID 化は設定変更のみで即時適用でき、
  RS256 移行完了前の暫定的なセキュリティ向上として有効である。

## 影響

**ポジティブな影響**:

- S2S JWT 偽造リスクが大幅に低下する
- 秘密鍵の漏洩影響が局所化され、サービス単位でのローテーションが可能になる
- `key` フィールドの予測可能性がなくなり、ブルートフォース試行が困難になる

**ネガティブな影響・トレードオフ**:

- Phase 2 では Vault の PKI 設定と各サービスの JWT 生成ロジック変更が必要
- RS256 の署名・検証は HS256 より計算コストが高い（S2S 頻度では許容範囲内）
- 鍵ペアのローテーション手順を整備する必要がある
- Phase 1 から Phase 3 の移行期間中は HS256 と RS256 の並行稼働が必要

## 代替案

| 案 | 概要 | 採用しなかった理由 |
|----|------|-----------------|
| 案 A: HS256 を維持し、シークレットのみ強化する | `key` を UUID に変更し、共有シークレットを定期ローテーションする | 対称鍵の根本的な問題（共有シークレット漏洩リスク）が残る。ローテーションの運用コストも発生する |
| 案 B: mTLS による S2S 認証に完全移行する | JWT を廃止し、相互 TLS 証明書による S2S 認証に切り替える | Kong での設定変更が大規模になる。既存の JWT ベースの認可ロジックとの統合が複雑になる |
| 案 C: OIDC クライアントクレデンシャルフローを使用する | Keycloak の client_credentials グラントで S2S トークンを発行する | Keycloak の可用性に S2S 認証が依存するリスクがある。JWT 発行の遅延が発生する可能性がある |
