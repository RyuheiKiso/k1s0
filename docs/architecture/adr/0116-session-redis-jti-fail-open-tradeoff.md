# ADR-0116: session-redis jti 失効チェックの fail-open 設計

## ステータス
承認済み

## コンテキスト
session サービスの `is_jti_revoked()` 関数は、Redis 接続エラー時に `false`（失効していない）を返す。
つまり Redis が障害中は、ログアウト済み JWT トークンが再利用可能になる（fail-open）。

対立する要件:
- セキュリティ優先（fail-closed）: Redis 障害時にセッション検証を失敗させ、全ユーザーをログアウト
- 可用性優先（fail-open）: Redis 一時障害でもサービスを継続し、全ユーザーへの影響を最小化

## 決定
fail-open（`unwrap_or(false)`）を維持する。Redis 障害時はセッション検証をスキップし、
サービスの継続稼働を優先する。

## 理由
1. JWT の有効期限（exp クレーム）による基本的な失効制御は Redis と独立して機能する
2. Redis 一時障害は数分以内に回復することが多く、その間に全ユーザーをログアウトさせるリスクは大きい
3. JWT 再利用攻撃には Redis 障害のタイミングに依存する難易度の高さがある
4. Redis Sentinel 構成により可用性を向上させることで、fail-open のリスクを低減できる

## 影響

**ポジティブな影響**:

- Redis 一時障害時のサービス継続性が保たれる
- ユーザー体験への影響が最小化される

**ネガティブな影響・トレードオフ**:

- Redis 障害中（最大数分）にログアウト済みトークンが理論的に再利用可能
- セキュリティ基準が厳格な環境では許容できない場合がある

## 代替案

| 案 | メリット | デメリット |
|----|---------|----------|
| fail-closed（`unwrap_or(true)`） | セキュリティ最高 | Redis 障害時に全ユーザーログアウト |
| fail-open（現状） | 可用性優先 | 障害中は失効チェックをスキップ |
| Redis Sentinel/Cluster 高可用性構成 | 障害頻度を激減 | インフラコスト増 |
