# ADR-0085: ライブラリの言語パリティ方針（TypeScript/Dart の意図的非対称）

## ステータス

承認済み

## コンテキスト

k1s0 モノリポは Go/Rust/TypeScript/Dart の 4 言語に対応するライブラリ群を
`regions/system/library/{go,rust,typescript,dart}/` に提供している。

外部技術監査（M-012）において、以下のライブラリが Go/Rust にのみ存在し
TypeScript/Dart には存在しないことが指摘された:

- `secret-store`（シークレット管理）
- `statestore`（状態ストア）
- `pubsub`（Pub/Sub メッセージング）
- `binding`（外部サービスバインディング）

監査では、この非対称が意図的な設計なのか、実装漏れなのかを明確化することが求められた。

## 決定

`secret-store`・`statestore`・`pubsub`・`binding` の 4 ライブラリは
**Go/Rust のサーバーサイド専用**とし、TypeScript/Dart への移植は行わない。

TypeScript/Dart はクライアントサイド（React SPA・Flutter モバイル）で使用するため、
これらのサーバーサイドインフラストラクチャライブラリは不要である。

## 理由

### サーバーサイド専用ライブラリの根拠

| ライブラリ | サーバーサイド理由 |
|----------|----------------|
| `secret-store` | Vault/Kubernetes Secrets へのアクセスはサーバープロセスのみが許可される。クライアントが直接シークレットにアクセスすることはセキュリティ上禁止される。 |
| `statestore` | Redis 等のサーバーサイド状態ストアへの接続はサーバープロセスが管理する。クライアントは BFF 経由でのみ状態にアクセスする。 |
| `pubsub` | Kafka/RabbitMQ 等のメッセージブローカーはサーバー間通信に使用する。クライアントはリアルタイム通信に WebSocket（BFF 経由）を使用する。 |
| `binding` | 外部サービス（SMS・メール送信等）への直接バインディングはサーバーサイドのみ。クライアントはすべて BFF API 経由でアクセスする。 |

### アーキテクチャ上の役割分担

```
[React / Flutter] ──HTTP/WS──> [BFF-proxy / API Gateway] ──gRPC/HTTP──> [Go/Rust Services]
                                                                              ↓
                                                                  [secret-store / pubsub / statestore / binding]
```

クライアントは BFF または API Gateway を経由してのみバックエンドサービスにアクセスする。
クライアントが直接 Vault・Kafka・Redis に接続することはアーキテクチャ上想定されていない。

## 影響

**ポジティブな影響**:

- セキュリティ境界が明確になる（クライアントがインフラストラクチャに直接アクセスしない）
- ライブラリの責務が明確になり、不必要な移植コストが発生しない
- Go/Rust のサーバーサイドライブラリに集中してメンテナンスできる

**ネガティブな影響・トレードオフ**:

- TypeScript/Dart ライブラリの非対称性が生じるが、これは意図的設計である
- 新規開発者が非対称を見て混乱する可能性がある（本 ADR で説明を提供）

## 代替案

| 案 | 概要 | 採用しなかった理由 |
|----|------|-----------------|
| TypeScript/Dart にも移植する | すべての言語で同じライブラリを提供する | クライアントからインフラへの直接アクセスはセキュリティ違反。実用的なユースケースが存在しない。 |
| 共通抽象レイヤーを作成する | 言語横断の抽象 API を定義する | 不必要な複雑性を導入する。サーバー/クライアントの役割分担で十分に対処できる。 |
