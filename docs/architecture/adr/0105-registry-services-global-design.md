# ADR-0105: レジストリサービスのシステムグローバル設計

## ステータス

承認済み

## コンテキスト

k1s0 プラットフォームは多くのサービスがテナント分離（RLS + `tenant_id` カラム）を採用している。
しかし、以下の3つのレジストリサービスについてはテナント分離を適用するかどうかの判断が必要だった:

- **service-catalog**: 組織内で利用可能なサービスのカタログ
- **api-registry**: API スキーマ（OpenAPI / Proto）の保管・配信
- **app-registry**: モバイル・デスクトップアプリの配布管理

これらのサービスは「組織全体で共有されるインフラ」として機能しており、
テナント別に独立したカタログ・スキーマ・配布を持つ設計が本当に必要かどうかを検討した。

## 決定

service-catalog・api-registry・app-registry の3サービスを **システムグローバル設計** とし、
テナント分離（`tenant_id` カラム・RLS ポリシー）を導入しない。

アクセス制御はテナント分離ではなく **RBAC（ロールベースアクセス制御）** で実施する。
チームや組織単位のスコープが必要な場合は、`owner_team` や `visibility` フィールドで管理する。

## 理由

### テナント分離が不要な理由

1. **共通インフラの性質**: サービスカタログ・APIスキーマ・アプリ配布は組織全体で共通のインフラである。
   テナントごとに別々のカタログを持つことは通常のユースケースに合致しない。

2. **重複管理の回避**: テナント分離を導入すると、同一サービスのエントリを各テナントに複製する必要が生じ、
   メンテナンスコストが指数的に増大する。

3. **一貫したスキーマ管理**: api-registry にバージョン管理された API スキーマは全テナントで共通であるべきで、
   テナントごとに異なるスキーマを持つことは互換性問題を引き起こす。

4. **アプリ配布の一元管理**: app-registry はモバイル・デスクトップアプリのリリースを管理するものであり、
   テナント別に配布バージョンを分けることは運用の複雑化につながる。

### RBAC によるアクセス制御

テナント分離の代わりに、以下のロールで細粒度アクセス制御を実施する:

| ロール | 権限 |
|-------|------|
| `registry:admin` | 全エントリの読み書き・削除 |
| `registry:publisher` | 自チーム所有エントリの作成・更新 |
| `registry:viewer` | 全エントリの読み取り |

所有チームのスコープは `owner_team` フィールドで管理し、Keycloak のグループ・ロールと連携させる。

## 影響

**ポジティブな影響**:

- データモデルが単純化され、テナント分離のための migration・RLS ポリシーが不要になる
- 全テナントで一貫したサービスカタログ・APIスキーマ・アプリ配布が提供できる
- サービス間の API スキーマ参照が単純化される（テナント ID を意識しなくてよい）
- 運用・メンテナンスコストが大幅に削減される

**ネガティブな影響・トレードオフ**:

- 将来的に完全なテナント分離が必要になった場合、migration と大規模なリファクタリングが必要になる
- RBAC の設計・実装に追加コストが生じる
- テナントが完全に独立したカタログを持ちたいという要件が発生した場合には対応が困難

## 代替案

| 案 | 概要 | 採用しなかった理由 |
|----|------|-----------------|
| 案 A: 他サービスと同様にテナント分離を導入する | `tenant_id` カラムと RLS ポリシーを全テーブルに追加する | 共通インフラとしての性質に反する。同一スキーマをテナントごとに複製する二重管理となりオーバーエンジニアリング |
| 案 B: テナント分離と RBAC を併用する | `tenant_id` カラムを持ちながら、デフォルト tenant で共有エントリを管理する | 設計が複雑化し、「system テナント」と「ユーザーテナント」の概念が混在して認知コストが高い |
| 案 C: 可視性フラグのみで制御する（フラットなアクセス制御） | `is_public` フラグで公開・非公開を切り替える | チームスコープの細粒度制御ができず、将来的な要件変化に対応しにくい |
