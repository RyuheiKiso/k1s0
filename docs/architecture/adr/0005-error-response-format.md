# ADR-0005: エラーレスポンス体系の統一

## ステータス

承認済み

## コンテキスト

k1s0 は Go/Rust/TypeScript/Dart の4言語でサーバーを実装しており、各言語・各サービスで
エラーレスポンスの形式が統一されていなかった。技術品質監査（2026-03-21）の指摘（L-002）により、
クライアント側でのエラーハンドリングが複雑になっているという問題が指摘された。

具体的な問題:
- REST API では `{ "error": "...", "message": "..." }` 形式と `{ "code": "...", "detail": "..." }` 形式が混在
- gRPC では標準の Status コードを使用しているが、詳細メッセージの形式が統一されていない
- HTTP ステータスコードのマッピング（バリデーションエラー → 400 vs 422 等）が一貫していない

## 決定

全サービスで以下のエラーレスポンス形式を採用する:

```json
{
  "code": "SVC_TASK_NOT_FOUND",
  "message": "Task '12345' not found",
  "details": []
}
```

- `code`: サービス固有のエラーコード（例: `SVC_TASK_NOT_FOUND`）。機械的な判別に使用する
- `message`: 人間が読めるエラーメッセージ（英語）。ログ・デバッグ用途
- `details`: 追加情報の配列（バリデーションエラーのフィールド詳細等）

HTTP ステータスコードのマッピング:

| エラー種別 | HTTP | gRPC |
|----------|------|------|
| バリデーションエラー | 400 | INVALID_ARGUMENT |
| 認証エラー | 401 | UNAUTHENTICATED |
| 認可エラー | 403 | PERMISSION_DENIED |
| リソース未発見 | 404 | NOT_FOUND |
| 競合（楽観ロック等） | 409 | ABORTED |
| サーバー内部エラー | 500 | INTERNAL |

## 理由

- `code` フィールドにより、クライアントは文字列マッチングなしに型安全なエラーハンドリングが可能
- `details` 配列によりバリデーションエラーの複数フィールドを一度に返せる
- `k1s0_server_common::error::ServiceError` 型として実装することで、全 Rust サービスが共通の型を使用できる

## 影響

**ポジティブな影響**:

- クライアント（React/Flutter）のエラーハンドリングが統一され、コードが簡素化される
- エラーコード一覧がドキュメント化できる（API設計.md のエラーコードカタログ）
- ログ集計時にエラーコードでフィルタリングが容易になる

**ネガティブな影響・トレードオフ**:

- 既存の異なる形式を返しているエンドポイントの移行コストが発生する
- 後方互換性のためのバージョニングが必要な場合がある

## 代替案

| 案 | 概要 | 採用しなかった理由 |
|----|------|-----------------|
| RFC 7807 (Problem Details) | 標準仕様に準拠したエラー形式 | k1s0 固有の `code` フィールドの追加が必要で、標準との乖離が生じる |
| gRPC Status のみ | gRPC の標準エラーのみ使用 | REST API に適用できない |
