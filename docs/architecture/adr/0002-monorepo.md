# ADR-0002: モノリポ採用

## ステータス

承認済み

## コンテキスト

k1s0 プラットフォームは複数のサービス・ライブラリ・クライアントで構成される。具体的には以下のコンポーネントが存在する:

- **サーバー**: Go / Rust 製の複数マイクロサービス（auth, config, saga, DLQ, featureflag, ratelimit, tenant 等）
- **共通ライブラリ**: 4 言語（Go / Rust / TypeScript / Dart）で実装された 30 以上の共通ライブラリ
- **クライアント**: React（TypeScript）/ Flutter（Dart）製のフロントエンドアプリケーション
- **CLI**: Rust 製の対話式 CLI ツール
- **インフラ**: Docker Compose、Kubernetes マニフェスト、CI/CD 設定

これらを複数リポジトリで管理するか、単一リポジトリで管理するかを決定する必要があった。

## 決定

全コードを単一の Git リポジトリ（モノリポ）で管理する。

## 理由

**横断的変更の容易さ**: 共通ライブラリの API 変更やプロトコルの更新が発生した場合、影響を受ける全サービス・クライアントを単一の PR で変更できる。複数リポジトリでは複数 PR の連携管理が必要になり、変更の原子性が失われる。

**依存関係の一元管理**: 共通ライブラリをパスベース（`path = "../../system/library/rust/auth"` 等）で参照できるため、バージョン管理の複雑さを排除できる。

**CI/CD の統一**: 単一の CI/CD パイプラインで全コンポーネントのビルド・テスト・リントを管理できる。影響範囲の自動検出とテストの選択的実行が容易になる。

**コード共有の容易さ**: Proto 定義・設定スキーマ・テストフィクスチャ等を言語横断で共有しやすい。

**一貫した開発体験**: `justfile` によるタスクランナーを使って、言語に関係なく統一されたコマンド（`just test`、`just lint` 等）で開発できる。

## 影響

**ポジティブな影響**:

- 横断的変更が 1 つの PR で完結する
- 全コンポーネントのコードを同時に参照できる
- 共通の開発規約・Linter 設定を一元管理できる
- ライブラリのバージョン齟齬が発生しない

**ネガティブな影響・トレードオフ**:

- リポジトリのサイズが大きくなり、フルクローンに時間がかかる（sparse-checkout で対応）
- 全コンポーネントへのアクセス権を持つ必要がある（アクセス制御の粒度が粗くなる）
- CI/CD の影響範囲判定が複雑になる

**大規模モノリポへの対応**:

sparse-checkout を活用し、開発者は自分が担当する Tier のコードのみをチェックアウトできる。CLI ツールがインタラクティブに対象 Tier を選択し、必要なパスのみを取得する設計としている。

## 代替案

| 案 | 概要 | 採用しなかった理由 |
|----|------|-----------------|
| マルチリポジトリ | サービスごとに独立したリポジトリを持つ | 横断的変更の管理コストが高く、ライブラリのバージョン管理が複雑になる |
| ハイブリッド | コアライブラリのみモノリポ、サービスは個別リポジトリ | 境界の決定が難しく、結局ライブラリ変更と連動したサービス変更の管理が必要になる |
