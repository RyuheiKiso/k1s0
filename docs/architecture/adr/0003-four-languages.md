# ADR-0003: 4 言語（Go / Rust / TypeScript / Dart）採用

## ステータス

承認済み（2026-03-22 改訂: Tier 別言語方針を追記 — H-05 / L-05 監査対応）

## コンテキスト

k1s0 プラットフォームはサーバー・クライアント・CLI と多様なコンポーネントで構成される。各コンポーネントの特性（パフォーマンス要件・エコシステム・デプロイ形態）が異なるため、単一言語での実装が最適とは言えない状況があった。

一方で、言語数を増やすほど開発者の学習コスト・ライブラリの保守コストが増大する。採用言語を最小限に絞りつつ、各コンポーネントに適した言語を選択する必要があった。

技術品質監査（H-05 / L-05）において、Service Tier および Business Tier でのサーバー実装言語の選択基準が明文化されていないと指摘された。本改訂でその基準を明示する。

## 決定

以下の 4 言語をプラットフォーム標準言語として採用する:

| 言語 | 主な用途 |
|------|---------|
| **Go** | gRPC プロキシ・ミドルウェア・管理系サービス（System Tier） |
| **Rust** | 高パフォーマンスが必要なコアサービス・Business / Service Tier・CLI |
| **TypeScript** | Web クライアント（React）・BFF |
| **Dart** | モバイル・デスクトップクライアント（Flutter） |

さらに、共通ライブラリ（auth, config, telemetry, pagination 等）は全 4 言語で実装し、言語の違いによる API の不整合を防ぐ。

### Tier 別言語方針（H-05 / L-05 監査対応）

Tier ごとにサーバー実装言語の採用基準を以下のとおり定める。

| Tier | サーバー実装言語 | 方針 |
|------|----------------|------|
| **System Tier** | Go + Rust | ミドルウェア・プロキシ・管理系は Go、高パフォーマンスのコアサービスは Rust で実装する。両言語を併用する唯一の Tier。 |
| **Business Tier** | Rust 優先 | ドメインロジックを持つサービス（project-master 等）は Rust で実装する。Go はプレースホルダーとして存在する場合があるが、正式実装は Rust。 |
| **Service Tier** | Rust 優先 | task / board / activity 等のサービス固有ロジックは Rust で実装する。Go はプレースホルダーとして存在する場合があるが、正式実装は Rust。 |

#### Service Tier / Business Tier で Rust を優先する理由

- **パフォーマンス要件**: task（タスク管理）・board（ボード管理）・activity（アクティビティ）は低レイテンシ・高スループットが求められるため、GC ポーズのない Rust が適している
- **安全性**: ビジネスドメインではメモリ安全・型安全による実行時エラーの排除が重要
- **ドメインモデルの表現力**: Rust の `enum`（代数的データ型）と所有権システムは DDD のドメインモデルを型レベルで表現しやすい
- **System Tier との一貫性**: System Tier の高パフォーマンスサービスが Rust で実装されており、上位 Tier でも同一の言語・ツールチェーンを使用することで認知コストを抑えられる

#### ライブラリの言語カバレッジ方針

| Tier | ライブラリ言語カバレッジ |
|------|----------------------|
| **System Tier** | 全 4 言語（Rust / Go / TypeScript / Dart）で実装。`modules.yaml` の `library_parity` で管理 |
| **Business Tier** | 当該 Tier で使用する言語のみ実装（現状は Rust 実装が正式、Go はプレースホルダー） |
| **Service Tier** | 当該 Tier で使用する言語のみ実装（現状は Rust 実装が正式、Go はプレースホルダー） |

> **注意**: Go プレースホルダーの取り扱い方針は [go-stub-policy.md](../../infrastructure/cicd/go-stub-policy.md) を参照。

## 理由

### Go を採用した理由

- **高速起動・低メモリ**: gRPC プロキシや BFF サービスはリクエスト数が多く、軽量なランタイムが必要
- **シンプルな並行処理**: goroutine による並行処理が書きやすく、ミドルウェア層の実装に適している
- **デプロイの容易さ**: シングルバイナリとして配布でき、コンテナイメージを小さく保てる
- **gRPC エコシステム**: `protoc-gen-go-grpc` による protobuf 連携が成熟している

### Rust を採用した理由

- **メモリ安全**: ガベージコレクターなしでメモリ安全性を保証し、C/C++ 相当のパフォーマンスを実現する
- **ゼロコスト抽象化**: 高レベルな抽象化を書いてもランタイムのオーバーヘッドが発生しない
- **CLI 開発**: `dialoguer`・`clap` 等の豊富なエコシステムでインタラクティブな CLI を構築できる
- **WebAssembly 対応**: Rust は WebAssembly へのコンパイルが容易で、将来のエッジコンピューティング対応に有利

### TypeScript を採用した理由

- **型安全な Web 開発**: JavaScript の柔軟性を維持しつつ、静的型付けでバグを早期に検出できる
- **React エコシステム**: Web クライアント開発において React / Next.js が事実上の標準
- **フルスタック共有**: サーバー（BFF）とクライアントで型定義・バリデーションロジックを共有できる

### Dart を採用した理由

- **クロスプラットフォーム**: Flutter により iOS・Android・Web・デスクトップを単一コードベースでカバーできる
- **高い UI 表現力**: Flutter のウィジェットシステムにより一貫した UI をクロスプラットフォームで実現できる
- **型安全**: Dart の強い型システムにより大規模アプリケーションの保守性が高い

### 統一ライブラリを 4 言語で提供する理由

認証（authlib）・設定（config）・可観測性（telemetry）・ページネーション（pagination）等の横断的関心事を 4 言語すべてで同一の API 契約に基づいて実装することで:

- サーバーとクライアントで同じ概念・命名を使用できる
- 言語をまたいだドキュメントの理解コストが下がる
- 新しいサービスを追加するときにライブラリの使い方を再学習する必要がない

## 影響

**ポジティブな影響**:

- 各コンポーネントに最適な言語を選択できる
- 全言語でパフォーマンス・安全性・開発効率のバランスが取れている
- 統一ライブラリにより言語をまたいだ一貫した開発体験を提供できる

**ネガティブな影響・トレードオフ**:

- 4 言語のツールチェーン・依存関係を管理する必要がある（devcontainer で統一バージョンを管理して対応）
- 開発者が全言語に精通するのは難しい（担当コンポーネントの言語のみに集中できる設計にしている）
- 共通ライブラリを 4 言語で実装・保守するコストが発生する（設計を言語共通にして個別実装量を最小化）

## 代替案

| 案 | 概要 | 採用しなかった理由 |
|----|------|-----------------|
| Go のみ | 全コンポーネントを Go で実装 | サーバービルドは可能だが、モバイルクライアント（Flutter/Dart）への対応が困難 |
| Rust のみ | 全コンポーネントを Rust で実装 | CLI・サーバーは優秀だが、Web/モバイル UI 開発のエコシステムが未成熟 |
| Node.js + TypeScript のみ | 全コンポーネントを TypeScript で実装 | サーバーパフォーマンスが Go/Rust に劣り、バイナリ配布が困難 |
| Kotlin Multiplatform | サーバー（Kotlin）+ モバイル（KMP）| JVM の起動時間・メモリ消費が大きく、CLI ツールに不向き |
