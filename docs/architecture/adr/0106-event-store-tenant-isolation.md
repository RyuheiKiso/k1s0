# ADR-0106: event-store テナント分離実装

## ステータス

承認済み

## コンテキスト

event-store サービスは CQRS/ES（コマンドクエリ責務分離 / イベントソーシング）の中核を担い、
以下のエンティティを管理している:

- `StoredEvent`: 永続化されたドメインイベント
- `EventStream`: 集約ルートに紐づくイベントストリーム
- `Snapshot`: 集約状態のスナップショット

現状、これらのエンティティに `tenant_id` カラムが存在せず、
すべてのテナントのイベントが同一テーブルに混在している。
これにより、あるテナントのアプリケーションが別テナントのイベントを取得・参照できる状態となっている。

DB 層の RLS ポリシーは migration 006 で設定済みだが、アプリケーション層で
`set_config('app.current_tenant_id', ...)` を呼び出してセッション変数を設定する実装が欠落しており、
RLS が実質的に機能していない。

また、`saga` サービスの `find_incomplete` メソッドは全テナントをまたいだサーガの回復処理を行うが、
通常の RLS テナント分離とは異なる管理者専用の接続が必要となる点も考慮が必要である。

## 決定

event-store サービスのテナント分離を次の方式で実装する:

### 1. エンティティへの tenant_id 追加

`StoredEvent`・`EventStream`・`Snapshot` のすべてのエンティティに `tenant_id: String` フィールドを追加する。

### 2. アプリケーション層での set_config パターン適用

すべてのリポジトリメソッドのトランザクション開始直後に以下を実行する:

```sql
SELECT set_config('app.current_tenant_id', $1, true)
```

第3引数 `true` は `SET LOCAL`（トランザクションスコープ）を意味し、
トランザクション終了時にセッション変数がリセットされることを保証する。

### 3. migration でのテナント分離適用

migration 006 で設定済みの RLS ポリシーが正しく機能するよう、
アプリケーション側のセッション変数設定を必須とする。

### 4. saga find_incomplete の管理者専用コネクション

`saga` サービスの `find_incomplete` メソッドは全テナントのサーガを回復するため、
RLS をバイパスする管理者専用 DB コネクション（`app.is_admin = 'true'` または別ロール）を使用する。
通常の操作との混同を防ぐため、管理者コネクションは明示的に分離して保持する。

### 実装対象

| エンティティ | テーブル | 対応内容 |
|------------|---------|---------|
| `StoredEvent` | `event_store.events` | `tenant_id` カラム追加・RLS 有効化 |
| `EventStream` | `event_store.streams` | `tenant_id` カラム追加・RLS 有効化 |
| `Snapshot` | `event_store.snapshots` | `tenant_id` カラム追加・RLS 有効化 |

## 理由

- テナント分離なしでは、マルチテナント環境においてテナント間のデータ漏洩リスクが存在する
- RLS ポリシーは migration 006 で既に定義済みであり、アプリケーション層の実装のみが欠落している
- `set_config` パターンは他サービス（notification・config・session 等）で実績があり、一貫性を保てる
- イベントソーシングの特性上、`tenant_id` はイベントの生成元テナントを不変的に記録する役割も担う

## 影響

**ポジティブな影響**:

- テナント間のイベントデータ漏洩が防止される
- RLS ポリシーが実際に機能するようになり、DB レベルでのテナント境界が保証される
- `tenant_id` の追加により、テナント別の集約イベント一覧取得が可能になる

**ネガティブな影響・トレードオフ**:

- 既存データへの `tenant_id` バックフィルが必要（データ移行コスト）
- `tenant_id` なしで既存データを投入した場合、デフォルト値の設定が必要
- `find_incomplete` の管理者コネクション分離による実装コストが増加する
- すべてのリポジトリメソッドにトランザクション＋`set_config` の追加が必要

## 代替案

| 案 | 概要 | 採用しなかった理由 |
|----|------|-----------------|
| 案 A: イベントストリーム ID にテナントプレフィックスを付与する | `stream_id = "tenant_xxx/aggregate_yyy"` のような命名規則を使用する | アプリケーション依存のフィルタリングであり DB レベルの保護がない。既存ストリーム ID との互換性も失われる |
| 案 B: テナントごとに別スキーマを使用する | `tenant_abc.events`・`tenant_def.events` のように動的スキーマを作成する | スキーマ数がテナント数に比例して増加し、migration 管理が困難になる |
| 案 C: アプリケーション層のみでフィルタリングする（RLS なし） | WHERE 句に `tenant_id = ?` を明示的に追加する | クエリへの追加漏れや将来的な SQL インジェクションリスクがある。RLS による DB レベル保護が失われる |
