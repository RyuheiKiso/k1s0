# ADR-0083: task ステータス遷移の検証レイヤー方針

## ステータス

承認済み

## コンテキスト

task サービスでは、タスクのステータスが `todo → in_progress → done` などの有限状態機械に従う。
この遷移ルールをどのレイヤーで強制するかが設計上の論点となった。

候補として以下の2つのアプローチが存在する:

1. **DB トリガー方式**: PostgreSQL のトリガーで遷移ルールを強制する
2. **アプリケーション層方式**: ドメインエンティティ内の `can_transition_to()` / `transition_to()` メソッドで検証する

また、外部技術監査（M-008）において、有効なステータス値そのものの制約（CHECK 制約）が欠落している点も指摘された。
C-005 対応として、有効ステータス値の CHECK 制約をマイグレーションで追加する。

## 決定

アプリケーション層（ドメインエンティティ）のみで遷移検証を行う。DB トリガーは使用しない。

有効なステータス値の制約は CHECK 制約として DB にも追加し、データ整合性の最終防衛線とする。

## 理由

- **ドメインロジックの集中管理**: `can_transition_to()` / `transition_to()` をドメインエンティティに持つことで、遷移ルールがコードで明示的に表現される。テストも容易である。
- **トリガーの複雑性回避**: DB トリガーはデバッグ困難であり、マイグレーション管理が煩雑になる。ドメインルールがコードと DB に分散すると整合性維持のコストが増大する。
- **既存設計との一貫性**: 他の検証（ページサイズ制限・日付フォーマット等）も全てアプリケーション層で行う方針（ADR-0068 準拠）。
- **CHECK 制約の補完的役割**: アプリケーション層のバグや直接 DB 操作に備え、有効値の CHECK 制約を DB に追加することで多層防御を実現する。

## 影響

**ポジティブな影響**:

- 遷移ルールがドメインエンティティに集約され、可読性・保守性が向上する
- ユニットテストで遷移ルールを網羅的に検証できる
- DB 依存なしにドメインロジックをテストできる

**ネガティブな影響・トレードオフ**:

- 直接 DB を操作した場合（マイグレーションスクリプト・管理ツール等）に無効な遷移が混入する可能性がある（CHECK 制約で有効値は担保するが遷移順序は担保しない）
- アプリケーション層をバイパスした更新には遷移バリデーションが効かない

## 代替案

| 案 | 概要 | 採用しなかった理由 |
|----|------|-----------------|
| DB トリガー方式 | PostgreSQL トリガーで遷移ルールを強制 | デバッグ困難・マイグレーション煩雑・ドメインロジック分散 |
| DB チェック制約のみ | 有効値のみ CHECK 制約で制限 | 遷移順序（todo→done の直接遷移防止等）を表現できない |
| アプリ層 + トリガー併用 | 両方で二重チェック | 保守コスト増大・ルール変更時の同期ミスリスク |
