# ADR-0092: AES-GCM-SIV 移行検討（M-002 監査対応）

## ステータス

提案中

## コンテキスト

現在のシステムでは AES-256-GCM（96-bit ランダムノンス）を暗号化アルゴリズムとして採用している（ADR-0066、ADR-0090 参照）。

外部技術監査（M-002）において、Birthday paradox によるノンス衝突リスクが指摘された。具体的には、同一鍵で 2^48 回（約 281 兆回）の暗号化を行うと 50%の確率でノンスが衝突する。ノンス衝突が発生した場合、暗号文の機密性と完全性（認証タグ）が破綻する。

AES-GCM-SIV（RFC 8452）はこの問題を解決する派生アルゴリズムであり、ノンスが衝突しても機密性が完全に失われる代わりに、完全性（認証タグ）のみが損なわれる設計（nonce-misuse resistance）を採用している。これにより、ランダムノンスを使用した場合のリスクを大幅に低減できる。

## 決定

**現時点では AES-GCM-SIV への移行を行わない。**

ただし、将来の移行に備えてリスク指標を定義し、トリガー条件を設ける。

## 理由

### 移行しない理由

1. **現実的なリスク水準**: 本システムの暗号化頻度は数十万回/日以下であり、2^48 の衝突確率に達するには数百万年以上を要する。短中期における現実的なリスクは許容範囲内である。

2. **ADR-0090 による攻撃表面の削減**: AAD（Additional Authenticated Data）の導入により、ノンス衝突が発生した場合でも文脈外での暗号文流用（replay attack・context confusion）を防いでいる。これにより実質的な被害リスクは大幅に低下している。

3. **クレートの成熟度**: Rust における `aes-gcm-siv` クレートは `aes-gcm` クレートと比較して採用実績・メンテナンス状況の確認が必要であり、本番投入には安定性の検証が求められる。

4. **移行コスト**: 既存の暗号化済みデータは新旧アルゴリズムが異なるため、再暗号化が必要となる。ローリング移行の設計・実装・テストは非自明であり、相応のコストを要する。

### 将来の移行トリガー

以下のいずれかの条件を満たした場合に、本 ADR を再評価して移行を検討する:

- **暗号化頻度の急増**: 同一鍵での月間暗号化回数が **1 億回** を超えた場合
- **クレート安定化**: `aes-gcm-siv` クレートが安定版（1.x 以上）に達し、`RustCrypto` コミュニティによる十分なレビューが確認された場合
- **鍵管理の変更**: 鍵ローテーション周期が延長される等、鍵の長期利用が発生する変更があった場合

## 影響

**ポジティブな影響**:

- 現時点では実装変更なし。リリースリスクゼロ。
- 暗号化回数の可視化（監視メトリクス追加）により将来の意思決定根拠を整備できる。

**ネガティブな影響・トレードオフ**:

- ノンス衝突リスクは理論上残存する。ただし前述の通り現実的なリスク水準は低い。
- 将来移行する際には、暗号化済みデータの再暗号化フェーズが必要となる。

### 推奨対応（今後のアクション）

- `regions/system/` の各サービスで暗号化操作をカウントする Prometheus メトリクス（`encryption_operations_total{key_id="..."}`）の追加を推奨する。
- 鍵ごとの累積暗号化回数が 10^8 に近づいた場合に Alertmanager でアラートを発火させること。

## 代替案

| 案 | 概要 | 採用しなかった理由 |
|----|------|-----------------|
| AES-GCM-SIV への即時移行 | `aes-gcm-siv` クレートへ全面切り替え | クレートの安定性・移行コスト・現実的リスクが低いため時期尚早 |
| カウンターベースノンス管理 | 分散環境でのノンスカウンターをDBで管理 | 実装複雑度が高く、カウンター管理自体が単一障害点になるリスクがある |
| 鍵ローテーション短縮 | ローテーション周期を短縮してノンス空間の使用量を抑制 | ADR-0008 で既に適切な周期を設定済み。さらなる短縮は運用コスト増加 |
