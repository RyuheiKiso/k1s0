# ADR-0008: JWT 秘密鍵ローテーション手順

## ステータス

承認済み

## コンテキスト

k1s0 では認証基盤として Keycloak を採用し、RS256 アルゴリズムで署名された JWT（JSON Web Token）を
発行している。RS256 は非対称鍵暗号方式であり、秘密鍵（署名鍵）と公開鍵（検証鍵）のペアで構成される。

技術品質監査（M-21）において、以下の問題が指摘された:

- 秘密鍵ローテーション手順がドキュメント化されていない
- 鍵ローテーション中に既存の JWT が無効化されるリスクへの対処方針が不明確である
- 障害発生時や鍵漏洩時に即時ローテーションを実施する手順が存在しない

JWT の秘密鍵は長期間同一のものを使い続けると、漏洩リスクが累積するため、定期的なローテーションが必要である。
また、ローテーション中に有効な JWT を持つユーザーへのサービス提供が途絶えないよう、
無停止での切り替えが要件となる。

## 決定

Keycloak の鍵プロバイダー機能を用いた **新旧鍵の重複有効期間（Grace Period）** によるローテーション手順を採用する。
具体的には以下の方針とする:

1. **定期ローテーション**: 6ヶ月ごとに実施する
2. **重複有効期間**: 新鍵追加後、旧鍵は JWT の最大有効期限（1時間）＋バッファ（30分）= 90分間は「検証のみ可能な状態」として維持する
3. **手順の管理場所**: 詳細な操作手順は `infra/keycloak/` ディレクトリに配置する

## 理由

### 新旧鍵の重複有効期間を設ける理由

鍵を即時に切り替えると、旧鍵で署名された有効な JWT（最大1時間）を持つユーザーが
認証エラーとなりサービスが利用できなくなる。Keycloak は複数の鍵プロバイダーを
同時に有効化できる機能を持つため、新鍵で署名しつつ旧鍵でも検証できる期間を設けることで
無停止切り替えが実現できる。

### Keycloak Admin UI / REST API による操作を採用する理由

Keycloak の鍵管理は Admin UI または REST API（`/admin/realms/{realm}/components`）
で操作できる。Terraform（terraform-provider-keycloak）でも管理可能だが、
鍵素材そのものは Terraform State に含まれるべきでないため、
運用時のローテーションは Admin UI / REST API 経由で実施する。
初期構築時の Realm 設定のみ Terraform で管理する。

## 影響

**ポジティブな影響**:

- 手順が明文化されることで、担当者交代時も安全にローテーションを実施できる
- 新旧鍵の重複期間により、ユーザーセッションへの影響なく鍵を切り替えられる
- 漏洩時の緊急ローテーション手順が整備され、インシデント対応時間を短縮できる

**ネガティブな影響・トレードオフ**:

- ローテーション期間中は Keycloak が2つの鍵を保持するため、わずかにリソース消費が増加する
- 90分の重複有効期間中に漏洩した旧鍵は引き続き有効なため、漏洩検知から無効化まで最大90分のウィンドウが生じる
- 緊急ローテーション時は重複期間を短縮できるが、その場合はオンラインユーザーの一部が再認証を求められる

## 代替案

| 案 | 概要 | 採用しなかった理由 |
|----|------|-----------------|
| 案 A: 即時切り替え | 旧鍵を即時無効化し新鍵に切り替える | 有効な JWT を持つユーザーが認証エラーになるため無停止要件を満たせない |
| 案 B: Terraform のみで管理 | terraform-provider-keycloak で鍵を完全管理 | 鍵素材が State ファイルに記録され、Terraform State の漏洩が鍵漏洩に直結するリスクがある |
| 案 C: 外部 KMS（AWS KMS 等）連携 | Keycloak を KMS バックエンドで構成 | オンプレミス環境への依存度が高く、現在のインフラ構成（自社 DC + Kubernetes）では導入コストが過大 |
