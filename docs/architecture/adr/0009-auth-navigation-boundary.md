# ADR-0009: Auth サービスと Navigation サービスの責務境界

## ステータス

承認済み

## コンテキスト

k1s0 システムには以下の2つのサービスが存在し、責務の重複が指摘されていた（M-24指摘）:

- **Auth サービス** (`regions/system/server/rust/auth`): 認証・認可を担うシステムサービス。
  JWT 発行、パーミッションチェック、ロール管理を提供する。
  設計書（`docs/servers/auth/server.md`）では `/api/v1/auth/permissions/check` エンドポイントを定義している。
- **Navigation サービス** (`regions/system/server/rust/navigation`): 画面遷移・メニュー構造を担うシステムサービス。
  ユーザーのロールに基づいてアクセス可能なメニュー項目を返す。
  Navigation サービスの設計書にも permission チェックへの言及があり、正規オーナーが曖昧だった。

この設計上の曖昧さにより:

- クライアント実装者がどちらのエンドポイントを使うべきか判断できない
- 将来的に `/permissions/check` が二重実装されるリスクがある
- Auth サービスと Navigation サービスの間に循環依存が生じる可能性がある

## 決定

**`/api/v1/auth/permissions/check` エンドポイントの正規オーナーは Auth サービスとする。**

各サービスの責務を以下のとおり明確に分離する:

| サービス | 責務 | 責務対象外 |
|---------|------|----------|
| Auth サービス | JWT 発行・検証、ロール管理、個別パーミッションチェック（`/permissions/check`） | メニュー構造の管理、画面遷移ルールの定義 |
| Navigation サービス | メニュー構造の管理、画面遷移権限の管理（どのロールがどの画面に遷移できるか）、ナビゲーションツリーの返却 | 個別リソースへのアクセス可否判定 |

Navigation サービスは画面遷移権限の判断が必要な場合、Auth サービスの
`/api/v1/auth/permissions/check` に問い合わせる（委譲）。
Navigation サービスが独自に権限チェックロジックを実装してはならない。

## 理由

### Auth サービスが permissions/check を所有すべき理由

パーミッションチェックはビジネスロジックに依存しない横断的な認可ロジックであり、
単一の真実源（Single Source of Truth）として Auth サービスが一元管理すべきである。
Navigation サービスが独自に権限チェックを行うと:

1. 権限定義の更新が Auth・Navigation の両方に波及し、整合性の維持が困難になる
2. 監査時に権限チェックの実装が分散しており、セキュリティレビューが複雑になる

### Navigation サービスが画面遷移権限のみを管理する理由

「どの画面に遷移できるか」はメニュー構造という UI の関心事であり、Auth サービスではなく
Navigation サービスが管理するのが自然な分担である。ただし、個別リソースへの
アクセス可否（例: 特定テナントのデータを読み取れるか）は Auth の責務であるため、
Navigation はこの判断を Auth に委譲する。

## 影響

**ポジティブな影響**:

- 権限チェックの実装が Auth サービスに集約され、セキュリティ監査が容易になる
- クライアントは権限確認に Auth エンドポイントを一貫して使用するため、実装の迷いがなくなる
- Navigation サービスの責務が明確になり、テスタビリティが向上する

**ネガティブな影響・トレードオフ**:

- Navigation サービスが Auth サービスに同期的に問い合わせる場合、レイテンシが増加する可能性がある（キャッシュで緩和）
- Navigation サービスの設計書内の permission 関連の記述を更新する必要がある

## 代替案

| 案 | 概要 | 採用しなかった理由 |
|----|------|-----------------|
| 案 A: Navigation サービスが permissions/check を所有 | メニュー構造と権限チェックをまとめて Navigation が管理 | 認可ロジックが Auth 外に漏れ出し、権限チェックが分散する。認証基盤の一元化原則に反する |
| 案 B: 両サービスが独立して実装 | Auth と Navigation がそれぞれ権限チェックを実装 | 権限定義の二重管理が発生し、不整合が生じやすい。禁止とする |
| 案 C: 権限チェック専用サービスを新設 | Policy サービス（既存）に完全委譲 | Policy サービスは RBAC ポリシーの CRUD を担当しており、リアルタイムな権限チェックとは役割が異なる。Auth サービスが Policy サービスを参照してチェックする現行設計を維持する |
