# ADR-0082: GraphQL ページネーション統一方針

## ステータス

承認済み

## コンテキスト

現在の GraphQL スキーマには 4 種類の混在したページネーションスタイルが存在する:

1. **cursor ベース**: `first` / `after` / `edges` / `pageInfo` 形式（Relay 仕様準拠）
2. **offset-like ベース**: `limit` / `offset` 形式
3. **offset ベース**: `page` / `limit` 形式（page は 1 始まりだが limit を使用）
4. **page ベース**: `page` / `pageSize` 形式（設計書準拠）

外部技術監査（M-015, 2026-04-04）において、クライアントの実装コストが高く、一貫性のない API 設計であると指摘された。

REST API および Proto 定義では `page` / `page_size` ベースに統一されているが（ADR-0068 準拠、ページネーション設計.md 参照）、GraphQL スキーマのみ複数スタイルが混在している。

## 決定

GraphQL スキーマのページネーションを `page` / `pageSize` ベースに統一する。

- リクエスト: `page: Int` (デフォルト 1), `pageSize: Int` (デフォルト 20, 上限 100)
- レスポンス: `pagination { totalCount page pageSize hasNext }` 形式で統一

既存クライアントへの影響を最小化するため、段階的移行（deprecated フィールド維持 → 削除）を行う。

## 理由

- **REST/Proto との一貫性**: 既存の設計書（ページネーション設計.md）および Proto 定義と統一することでクライアント実装コストを削減する
- **DoS 防止**: page_size の上限 100 制限を GraphQL でも統一適用する（M-002/M-004 対応と同方針）
- **クライアント実装コスト削減**: 複数スタイルへの対応が不要になる
- **cursor ベース不採用の理由**: k1s0 のユースケースでは全件数表示（totalCount）が必要なため、cursor ベースとの相性が悪い

## 影響

**ポジティブな影響**:

- GraphQL クライアントの実装が単純化される
- REST/Proto/GraphQL 全体での API 一貫性が確保される
- ページサイズ上限 100 による DoS 防止が GraphQL にも適用される

**ネガティブな影響・トレードオフ**:

- 既存の cursor / offset-like スタイルを使用するクライアントは移行が必要（段階的廃止で対応）
- Relay クライアントとの互換性が失われる（k1s0 では Relay は未使用のため影響なし）

## 移行計画

| フェーズ | 内容 | 時期 |
|---------|------|------|
| Phase 1 | `page` / `pageSize` フィールドを全スキーマに追加（既存フィールドは `@deprecated` 付きで残す） | 次スプリント |
| Phase 2 | クライアントを新フィールドへ移行 | Phase 1 完了後 1 スプリント |
| Phase 3 | deprecated フィールドを削除 | Phase 2 完了後 |

## 代替案

| 案 | 概要 | 採用しなかった理由 |
|----|------|-----------------|
| cursor ベースへ統一 | Relay 仕様準拠の cursor ページネーション | totalCount が必要なユースケースと相性が悪い |
| 混在スタイル維持 | 現状維持 | クライアント実装コストが高く、DoS 防止の統一適用が困難 |
| offset ベースへ統一 | `limit` / `offset` 形式 | Proto/REST の `page` / `page_size` と乖離が生じる |
