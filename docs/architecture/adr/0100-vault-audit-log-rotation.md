# ADR-0100: Vault 監査ログローテーション管理方針

## ステータス

承認済み

## コンテキスト

Vault は監査ログ（audit log）を有効化すると、全リクエスト・レスポンスをファイルシステムに記録する。
監査ログが無効化されると Vault はリクエストを処理しなくなるため（`audit_required` 設定に依存）、
PVC がディスクフルになった場合に Vault 全体がブロックするリスクがある。

ローカル開発環境では `/var/log/vault/audit.log` に書き込まれるが、ローテーション設定がないと
長期運用でディスクを圧迫する。本番 Kubernetes 環境では PVC の容量上限を超えるとポッドが
`OOMKilled` または書き込みエラーで停止する。

## 決定

Vault 監査ログの管理に以下の方式を採用する:

1. **開発環境**: logrotate ConfigMap を Vault のサイドカーコンテナとして配置し、
   `daily` ローテーション・`7日間保持` で管理する。

2. **本番環境**: syslog 転送（`/sys/audit/syslog` エンドポイント）を使用して
   外部ログ集約基盤（Loki 等）に転送する。PVC へのファイル書き込みを廃止し、
   ディスクフルリスクを根本から除去する。

3. **Kubernetes PVC サイズ**: ファイル出力を使用する場合は PVC に
   最低 10Gi を確保し、使用率 80% でアラートを設定する。

## 理由

- syslog 転送は PVC 依存を排除できるため最も安全だが、本番ログ基盤の整備が前提。
- logrotate による管理は既存インフラに変更を加えず導入できるため、開発環境に適している。
- `audit_required = true` 設定時は監査デバイスが1つも応答できない状態が Vault の停止に直結するため、
  ディスクフル対策は必須である。

## 影響

**ポジティブな影響**:

- PVC フルによる Vault ブロックリスクを排除できる。
- 監査ログが外部に転送されることでセキュリティ監査の追跡性が向上する。
- ローテーション設定により長期運用でのディスク逼迫を防止できる。

**ネガティブな影響・トレードオフ**:

- syslog 転送にはログ集約基盤（Loki/Elasticsearch 等）の整備が必要。
- logrotate サイドカーはポッド構成を複雑化させる。
- ローテーション後のアーカイブファイルの保持期間・保存場所を別途定義する必要がある。

## 代替案

| 案 | 概要 | 採用しなかった理由 |
|----|------|-----------------|
| 案 A: PVC 容量を増加するだけ | PVC サイズを大きくして問題を先送りにする | 根本解決にならず、最終的にディスクフルは発生する |
| 案 B: `audit_required = false` に設定 | 監査デバイスが停止しても Vault は動作し続ける | セキュリティ要件（コンプライアンス）に違反する |
| 案 C: ファイル出力 + cron で定期削除 | cron ジョブで古いログを削除する | ログを消去する前に外部転送する保証がなく、監査証跡が消える |
