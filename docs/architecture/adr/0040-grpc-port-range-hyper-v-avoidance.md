# ADR-0040: gRPC ホストポートを Hyper-V 動的除外範囲外に移動

## ステータス

承認済み

## コンテキスト

Windows の Hyper-V は起動時に TCP ポートの一部をシステム用に予約・除外する。この除外範囲はシステム起動のたびに変動するが、一般的に 50060-50159 の範囲が含まれる。

k1s0 の以下のサービスのデフォルト gRPC ホストポートがこの除外範囲に含まれていた:

| サービス | 旧デフォルトポート |
|---------|-----------------|
| event-monitor-rust | 50060 |
| master-maintenance-rust | 50061 |
| navigation-rust | 50062 |
| policy-rust | 50063 |
| rule-engine-rust | 50064 |
| session-rust | 50065 |

これにより、Windows + Docker Desktop (Hyper-V バックエンド) 環境でこれらのサービスの gRPC ポートをホストにバインドしようとした際に以下のエラーが発生する:

```
Error response from daemon: driver failed programming external connectivity
on endpoint: Bind for 0.0.0.0:50060 failed: port is already allocated
```

外部技術監査（2026-03-28）で HIGH 問題として検出された。

## 決定

gRPC ホストポートのデフォルト値を 50060-50065 から 50200-50205 に変更する。

各サービスは `docker-compose.yaml` の環境変数デフォルト値を変更することで対応する:

```yaml
# 変更前
- "${EVENT_MONITOR_GRPC_HOST_PORT:-50060}:50051"
# 変更後
- "${EVENT_MONITOR_GRPC_HOST_PORT:-50200}:50051"
```

環境変数オーバーライドの仕組みは維持するため、既に `.env` でカスタムポートを設定している開発者には影響しない。

## 理由

### 採用した案（案A: デフォルト値変更）

- コンテナ内部ポート（50051）は変更不要
- 環境変数オーバーライドの仕組みで個別対応も可能
- 50200-50205 は Hyper-V 除外範囲（一般的に 50060-50159）の外に位置する
- 開発者が追加の設定をしなくても Windows 環境でそのまま動作する

### 棄却した案（案B: 各開発者が .env でオーバーライド）

- 新規開発者が必ず詰まる問題であり、ドキュメントに記載しても見落とされやすい
- セットアップコストが高い

## 影響

**ポジティブな影響**:

- Windows + Docker Desktop 環境で追加設定なしに gRPC ポートが起動できる
- 新規開発者の初回セットアップ失敗率が低下する

**ネガティブな影響・トレードオフ**:

- 既存の開発者が外部から gRPC ポートを直接参照している場合（テストスクリプト等）はポート番号の更新が必要
- 環境変数を明示的に設定している開発者には影響しない

## 代替案

| 案 | 概要 | 採用しなかった理由 |
|----|------|-----------------|
| 案A（採用）| デフォルト値を 50200 帯に変更 | 追加設定不要で問題を根本解決する |
| 案B | 開発者が .env でオーバーライドする手順をドキュメント化 | 新規開発者が必ず詰まるため根本解決にならない |
| 案C | gRPC をホストに公開しない（Docker 内部ネットワークのみ） | 外部ツールからの gRPC テストが困難になる |
