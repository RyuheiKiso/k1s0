# インシデント対応フロー

本ドキュメントは、k1s0 におけるインシデント発生時の対応フローを定義する。

## 1. インシデント重大度定義

| 重大度 | 定義 | 例 |
|--------|------|-----|
| **P1 (Critical)** | 全サービス停止、データ損失リスク | 本番 DB 障害、全 Pod ダウン |
| **P2 (High)** | 主要機能の障害、性能大幅劣化 | 認証サービス障害、50% 以上のエラー |
| **P3 (Medium)** | 一部機能の障害、代替手段あり | 特定 API のエラー、軽微な性能劣化 |
| **P4 (Low)** | 軽微な問題、ユーザー影響小 | ログ出力の異常、非本質的な警告 |

## 2. 対応フロー概要

```
┌─────────────────┐
│  インシデント検知  │
└────────┬────────┘
         ▼
┌─────────────────┐
│   初期トリアージ   │ ← 重大度判定（5分以内）
└────────┬────────┘
         ▼
┌─────────────────┐
│  チャンネル開設   │ ← Slack #incident-YYYY-MMDD-XX
└────────┬────────┘
         ▼
┌─────────────────┐
│    調査・対応     │ ← 原因特定、復旧作業
└────────┬────────┘
         ▼
┌─────────────────┐
│    復旧確認      │ ← サービス正常性確認
└────────┬────────┘
         ▼
┌─────────────────┐
│  インシデント終了  │ ← 終了宣言、レポート作成
└─────────────────┘
```

## 3. Phase 1: 検知と初期対応（5分以内）

### 3.1 インシデント検知

インシデントの検知元:

- アラート（PagerDuty、Slack）
- 監視ダッシュボード
- ユーザー報告
- 内部発見

### 3.2 初期トリアージ

```markdown
## インシデント初期情報
- 検知時刻: YYYY-MM-DD HH:MM:SS JST
- 検知方法: [アラート名/報告者]
- 影響範囲: [影響を受けるサービス/機能]
- 重大度: P1 / P2 / P3 / P4
- 担当者: [自分の名前]
```

### 3.3 チェックリスト

- [ ] 重大度を判定
- [ ] Slack #alerts に初報を投稿
- [ ] P1/P2 の場合、オンコール担当に連絡
- [ ] インシデントチャンネルを開設

## 4. Phase 2: 調査（目標: 30分以内）

### 4.1 情報収集

```bash
# サービス状態確認
kubectl get pods -n k1s0
kubectl get events -n k1s0 --sort-by='.lastTimestamp' | tail -20

# ログ確認
kubectl logs -l app={service_name} --tail=200 | jq 'select(.level=="ERROR")'

# メトリクス確認
# Grafana ダッシュボードを確認

# 最近のデプロイ確認
kubectl rollout history deployment/{service_name}
```

### 4.2 原因候補の絞り込み

| 症状 | 確認項目 |
|------|---------|
| 全 Pod ダウン | ノード障害、リソース不足、設定エラー |
| 高エラーレート | 依存サービス障害、コード障害、設定変更 |
| 高レイテンシ | DB 負荷、外部 API 遅延、リソース逼迫 |
| 接続エラー | ネットワーク障害、DNS 障害、証明書期限 |

### 4.3 タイムライン記録

インシデントチャンネルに時系列で記録:

```
10:00 アラート検知: HighErrorRate on user-service
10:02 初期調査開始
10:05 DB 接続エラーを確認
10:08 DB Pod の状態を確認 → CrashLoopBackOff
...
```

## 5. Phase 3: 対応・復旧

### 5.1 復旧アクション

**サービス再起動:**
```bash
kubectl rollout restart deployment/{service_name}
kubectl rollout status deployment/{service_name}
```

**ロールバック:**
```bash
kubectl rollout undo deployment/{service_name}
```

**スケールアウト:**
```bash
kubectl scale deployment/{service_name} --replicas={数}
```

**設定変更:**
```bash
kubectl edit configmap {service_name}-config
kubectl rollout restart deployment/{service_name}
```

### 5.2 エスカレーション基準

以下の場合はエスカレーション:

- [ ] 30 分以上原因不明
- [ ] 復旧の見込みが立たない
- [ ] 影響範囲が拡大
- [ ] 専門知識が必要（DB、インフラ、セキュリティ）

### 5.3 エスカレーション先

| 問題領域 | エスカレーション先 | 連絡方法 |
|----------|-------------------|---------|
| アプリケーション | サービスオーナー | Slack DM / 電話 |
| インフラ | インフラチーム | #infra-oncall |
| データベース | DBA | #dba-oncall |
| セキュリティ | セキュリティチーム | #security-oncall |

## 6. Phase 4: 復旧確認

### 6.1 確認チェックリスト

- [ ] 全サービスが正常稼働
- [ ] エラーレートが正常値に復帰
- [ ] レイテンシが正常値に復帰
- [ ] ユーザーからの報告が止まった
- [ ] 監視アラートがクリア

### 6.2 確認コマンド

```bash
# Pod 状態
kubectl get pods -n k1s0

# ヘルスチェック
for svc in user-service order-service; do
  kubectl exec -it $(kubectl get pod -l app=$svc -o jsonpath='{.items[0].metadata.name}') -- curl -s http://localhost:8080/healthz
done

# エラーレート確認
curl 'http://prometheus:9090/api/v1/query?query=sum(rate(k1s0_request_failures_total[5m]))/sum(rate(k1s0_request_total[5m]))'
```

## 7. Phase 5: インシデント終了

### 7.1 終了宣言

```markdown
## インシデント終了宣言

- インシデント番号: INC-YYYY-MMDD-XX
- 終了時刻: YYYY-MM-DD HH:MM:SS JST
- 総対応時間: X時間Y分
- 根本原因: [簡潔な説明]
- 暫定対応: [実施した対応]
- 恒久対応: [今後の対応予定]
```

### 7.2 終了チェックリスト

- [ ] #alerts に終了を通知
- [ ] インシデントチャンネルをアーカイブ
- [ ] インシデントレポートのチケットを作成
- [ ] 翌営業日にポストモーテム予定を設定（P1/P2 の場合）

## 8. コミュニケーション

### 8.1 社内通知テンプレート

**初報:**
```
:rotating_light: [インシデント発生] P{重大度}
影響: {影響を受けるサービス/機能}
状況: {現在の状況}
担当: {担当者名}
チャンネル: #incident-YYYY-MMDD-XX
```

**中間報告（30分ごと）:**
```
:information_source: [インシデント更新] P{重大度}
現在の状況: {調査/対応中/復旧作業中}
進捗: {具体的な進捗}
次のアクション: {予定}
```

**終了報告:**
```
:white_check_mark: [インシデント終了] P{重大度}
終了時刻: HH:MM JST
総対応時間: X時間Y分
根本原因: {簡潔な説明}
恒久対応: {予定}
```

### 8.2 外部通知（P1 の場合）

ステータスページ更新:

1. インシデント発生を通知
2. 30 分ごとに状況更新
3. 復旧を通知

## 9. ポストモーテム

### 9.1 実施基準

| 重大度 | ポストモーテム |
|--------|--------------|
| P1 | 必須（72 時間以内） |
| P2 | 必須（1 週間以内） |
| P3 | 推奨 |
| P4 | 任意 |

### 9.2 ポストモーテムテンプレート

```markdown
# ポストモーテム: INC-YYYY-MMDD-XX

## サマリ
- 発生日時: YYYY-MM-DD HH:MM - HH:MM JST
- 影響範囲: [影響を受けたサービス/ユーザー数]
- 重大度: P{X}

## タイムライン
| 時刻 | イベント |
|------|---------|
| HH:MM | [イベント] |
| ... | ... |

## 根本原因
[詳細な原因分析]

## 影響
- ユーザー影響: [具体的な影響]
- ビジネス影響: [売上/評判への影響]

## 対応
### 検知
[どのように検知されたか]

### 対応
[実施した対応]

### 復旧
[復旧までの流れ]

## 教訓
### うまくいったこと
- [項目]

### 改善が必要なこと
- [項目]

## アクションアイテム
| アクション | 担当 | 期限 |
|-----------|------|------|
| [恒久対応] | [担当者] | YYYY-MM-DD |
| [監視改善] | [担当者] | YYYY-MM-DD |
```

## 10. 役割分担

### 10.1 インシデントコマンダー

- インシデント対応の指揮
- コミュニケーションの統括
- エスカレーション判断

### 10.2 テクニカルリード

- 技術的な調査・対応
- 復旧作業の実行
- 技術的な判断

### 10.3 コミュニケーションリード

- 社内外への通知
- タイムラインの記録
- ステークホルダーとの調整
