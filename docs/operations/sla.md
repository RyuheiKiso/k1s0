# SLA 定義

本ドキュメントは、k1s0 における SLA（Service Level Agreement）と SLO（Service Level Objective）を定義する。

## 1. 用語定義

| 用語 | 定義 |
|------|------|
| **SLA** | サービスレベル合意（外部へのコミットメント） |
| **SLO** | サービスレベル目標（内部目標、SLA より厳しく設定） |
| **SLI** | サービスレベル指標（測定可能なメトリクス） |
| **エラーバジェット** | 許容される障害時間/エラー数 |

## 2. 可用性目標

### 2.1 環境別可用性

| 環境 | SLO | SLA | 月間許容ダウンタイム |
|------|-----|-----|---------------------|
| prod | 99.95% | 99.9% | 4.38 分（SLO）/ 43.8 分（SLA） |
| stg | 99.5% | - | 3.6 時間 |
| dev | 95% | - | 36 時間 |

### 2.2 可用性の計算方法

```
可用性 = (成功リクエスト数 / 総リクエスト数) × 100
```

**成功リクエストの定義:**
- HTTP: ステータスコード 2xx, 3xx, 4xx（クライアントエラーは成功扱い）
- gRPC: OK, INVALID_ARGUMENT, NOT_FOUND, ALREADY_EXISTS, PERMISSION_DENIED

**失敗リクエストの定義:**
- HTTP: ステータスコード 5xx
- gRPC: INTERNAL, UNAVAILABLE, DEADLINE_EXCEEDED, UNKNOWN

### 2.3 計測除外

以下は可用性計算から除外:

- 計画メンテナンス（事前通知済み）
- 外部依存（クラウドプロバイダ障害等）の影響
- DDoS 攻撃等の不正アクセス

## 3. レスポンスタイム目標

### 3.1 API レイテンシ目標

| 分類 | P50 | P95 | P99 |
|------|-----|-----|-----|
| 軽量 API（認証、ヘルスチェック等） | 10ms | 50ms | 100ms |
| 標準 API（CRUD 操作） | 50ms | 200ms | 500ms |
| 複雑 API（集計、検索等） | 200ms | 500ms | 1000ms |

### 3.2 バッチ処理目標

| 処理タイプ | 目標時間 |
|-----------|---------|
| 日次バッチ | 1 時間以内 |
| 月次バッチ | 4 時間以内 |

### 3.3 測定方法

```prometheus
# P99 レイテンシ
histogram_quantile(0.99,
  sum(rate(k1s0_{service}_request_duration_seconds_bucket[5m])) by (le)
)
```

## 4. データ保持期間

### 4.1 ログ

| 環境 | 保持期間 | 備考 |
|------|---------|------|
| prod | 90 日 | 法令要件に応じて延長可能 |
| stg | 30 日 | |
| dev | 7 日 | |

### 4.2 メトリクス

| 解像度 | 保持期間 |
|--------|---------|
| 15 秒（Raw） | 7 日 |
| 1 分（Aggregated） | 30 日 |
| 5 分（Aggregated） | 90 日 |
| 1 時間（Aggregated） | 1 年 |

### 4.3 トレース

| 環境 | 保持期間 |
|------|---------|
| prod | 14 日 |
| stg | 7 日 |
| dev | 3 日 |

### 4.4 バックアップ

| データ | 頻度 | 保持期間 |
|--------|------|---------|
| データベース | 日次フル + 時間毎増分 | 30 日 |
| 設定ファイル | 変更時 | 無期限（Git） |

## 5. エラーバジェット

### 5.1 月間エラーバジェット

| SLO | エラーバジェット | 計算方法 |
|-----|----------------|---------|
| 99.95% | 0.05% | 月間リクエストの 0.05% |
| 99.9% | 0.1% | 月間リクエストの 0.1% |

### 5.2 エラーバジェット消費時のアクション

| 消費率 | アクション |
|--------|-----------|
| 50% 未満 | 通常運用 |
| 50-75% | 新機能リリースの慎重な検討 |
| 75-100% | 新機能リリース凍結、安定性改善に注力 |
| 100% 超過 | リリース停止、緊急対応 |

### 5.3 エラーバジェットの確認

```prometheus
# 月初からのエラーバジェット消費率
(
  sum(increase(k1s0_request_failures_total[30d]))
  /
  sum(increase(k1s0_request_total[30d]))
) / 0.001 * 100  # 99.9% SLO の場合
```

## 6. インシデント対応時間

### 6.1 重大度別対応目標

| 重大度 | 定義 | 初動 | 復旧目標 |
|--------|------|------|---------|
| P1 (Critical) | 全サービス停止、データ損失リスク | 15 分 | 1 時間 |
| P2 (High) | 主要機能の障害、性能大幅劣化 | 30 分 | 4 時間 |
| P3 (Medium) | 一部機能の障害、代替手段あり | 2 時間 | 24 時間 |
| P4 (Low) | 軽微な問題、ユーザー影響小 | 24 時間 | 1 週間 |

### 6.2 オンコール体制

| 時間帯 | 体制 |
|--------|------|
| 平日 9:00-18:00 | 通常運用チーム |
| 平日夜間・休日 | オンコール担当（ローテーション） |

## 7. 計画メンテナンス

### 7.1 メンテナンスウィンドウ

| 頻度 | 時間帯 | 事前通知 |
|------|--------|---------|
| 定期 | 毎週日曜 2:00-6:00 JST | 3 営業日前 |
| 緊急 | 随時 | 可能な限り事前通知 |

### 7.2 ダウンタイム許容

| メンテナンス種別 | 許容ダウンタイム |
|----------------|----------------|
| パッチ適用 | 0（ローリングアップデート） |
| マイナーバージョンアップ | 5 分 |
| メジャーバージョンアップ | 30 分 |
| DB マイグレーション | 15 分（計画による） |

## 8. 報告とレビュー

### 8.1 定期報告

| 報告 | 頻度 | 内容 |
|------|------|------|
| SLO レポート | 週次 | 可用性、レイテンシ、エラーバジェット消費 |
| インシデントレビュー | 月次 | インシデント数、MTTR、根本原因分析 |
| SLA レビュー | 四半期 | SLA 達成状況、改善計画 |

### 8.2 SLO ダッシュボード

Grafana で以下を常時可視化:

- 現在の可用性（30 日ローリング）
- P99 レイテンシ
- エラーバジェット残量
- インシデント数

## 9. SLA 違反時の対応

### 9.1 社内対応

1. インシデントレポート作成
2. 根本原因分析（RCA）
3. 再発防止策の策定と実施
4. ステークホルダーへの報告

### 9.2 補償（外部 SLA の場合）

| 月間可用性 | 補償 |
|-----------|------|
| 99.9% 未満 99.0% 以上 | クレジット 10% |
| 99.0% 未満 95.0% 以上 | クレジット 25% |
| 95.0% 未満 | クレジット 50% |
