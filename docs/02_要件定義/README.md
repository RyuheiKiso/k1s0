# k1s0 要件定義書

本ディレクトリは、マイクロサービス基盤プラットフォーム **k1s0** (けいいちぜろ) の要件定義資料を格納する。

企画書 ([`../01_企画/企画書.md`](../01_企画/企画書.md)) が「何を、なぜ作るのか」を経営層に訴求するための資料であるのに対し、本要件定義書は「そのプラットフォームが具体的にどうあるべきか」を、部外者 (情シス部門以外の関係者・新規参画者・パートナー企業担当者) でも理解できる粒度で整理したものである。

---

## 本要件定義書の特徴

- **部外者にも読めるように書く** — 専門用語には必ず用語集で解説を付ける。前提知識を仮定しない。
- **企画書と重複しない** — 課題背景やアーキテクチャ詳細は企画ドキュメントに委ね、本書は「要件」に集中する。
- **Phase 1 (MVP) を主対象とする** — 現時点で実装を約束できる範囲に絞る。将来拡張は「スコープ外」として明記する。
- **要件ごとに一意の ID を付与する** — トレーサビリティ (後工程の設計書・テスト仕様書との対応付け) のため `BR-xxx` / `FR-xxx` / `NFR-xxx` の形式で管理する。

## 要件 ID 命名規則

| プレフィックス | 種別 | 例 |
|---|---|---|
| `BR-xxx` | Business Requirement (業務要件) | BR-001: 横断的関心事を統一 API 化すること |
| `FR-xxx` | Functional Requirement (機能要件) | FR-010: SSO による認証統合 |
| `NFR-xxx` | Non-Functional Requirement (非機能要件) | NFR-001: tier1 API の p99 レイテンシ 500 ms 以内 |
| `CON-xxx` | Constraint (制約) | CON-005: オンプレミス環境で完結すること |
| `ASM-xxx` | Assumption (前提) | ASM-002: Kubernetes クラスタは社内で構築可能であること |
| `RISK-xxx` | Risk (リスク) | RISK-003: 起案者離脱によるバス係数 1 問題 |

## 優先度の表記

各要件には以下の優先度を付与する。

| 記号 | 意味 | MVP 扱い |
|---|---|---|
| **MUST** | 必達。達成できない場合は MVP としてリリース不可 | 必須 |
| **SHOULD** | 強く推奨。達成できない場合は代替策を明記すること | 極力対応 |
| **COULD** | 対応できれば望ましい | Phase 2 以降で検討 |
| **WON'T** | 今回は対応しない (スコープ外の明示) | スコープ外 |

本記法は MoSCoW 分析 (Must / Should / Could / Won't have) に準拠している。
