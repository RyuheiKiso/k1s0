# 要件トレーサビリティマトリクス (RTM)

## 1. 本章の目的

要件トレーサビリティマトリクス (Requirements Traceability Matrix, RTM) は、業務要件 (BR) → 機能要件 (FR) → 非機能要件 (NFR) → 制約 (CON) → 前提 (ASM) → リスク (RISK) の「なぜこの要件が必要か」を一気通貫で追跡するための索引である。レビュー時に「この FR はどの BR に紐づくか」「この NFR が崩れるとどの BR が影響を受けるか」を即座に確認できることが目的であり、要件の抜け漏れや孤立した要件を検出する手段でもある。

本章の RTM は以下の 3 つのビューで構成される。

1. **BR → FR マッピング** — 業務要件を実現する機能要件の対応関係
2. **FR → NFR/CON マッピング** — 機能要件に課される品質・制約の対応関係
3. **横断トレース** — BR から RISK までの全連鎖を一覧するサマリ

---

## 2. BR → FR マッピング

各業務要件がどの機能要件によって実現されるかを示す。1 つの BR が複数の FR で実現されるケースが大半であり、逆に 1 つの FR が複数の BR に貢献することもある。この多対多の関係が k1s0 の「横断的関心事を統一 API で集約する」というコア設計思想の反映である。

## 5. 孤立要件の検出

RTM 作成時に「どの BR にも紐づかない FR」や「どの FR にも紐づかない NFR」が見つかった場合、それは要件体系の不整合か、記載漏れの兆候である。現時点で検出された孤立候補はない。四半期レビュー時に本マトリクスを更新し、新規追加・削除された要件の紐づけを確認すること。

---

## 6. 本マトリクスの運用ルール

- **更新タイミング**: 要件の追加・変更・削除が発生した際に、対応する RTM のセルも更新する
- **レビュー**: Phase ゲートレビュー時に RTM の整合性を確認項目に含める
- **責任者**: 要件変更の起案者が RTM 更新を同時に行う。レビュー時に未更新の RTM は差し戻し対象とする
