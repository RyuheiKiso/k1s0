# テスト戦略

## 目的

k1s0 の tier1 内部実装 (Go + Rust ハイブリッド) / tier1 公開 API (多言語クライアントライブラリ) / tier2・tier3 サービスのテスト方針を定義する。「何を」「どのレベルで」「どの環境で」テストするかを明文化し、CI パイプラインへの組み込み方針を示す。

---

## 1. テスト設計の原則

1. **テストピラミッドに従う** — ユニットテストを最多、統合テストを中程度、E2E テストを最少にする
2. **tier1 のテストは tier1 チームが責任を持つ** — tier2 / tier3 は tier1 公開 API の正しさを前提にできる
3. **Dapr 依存のテストを分離する** — Dapr sidecar が必要なテストと不要なテストを明確に分ける
4. **CI で自動実行できないテストは書かない** — 手動テストは探索的テストに限定する
5. **雛形生成 CLI がテスト基盤も生成する** — 新規サービスに最初からテストが存在する状態にする

---

## 2. テストレベルの定義

### 2.1 ユニットテスト

| 対象 | テスト範囲 | 外部依存の扱い | 実行環境 |
|---|---|---|---|
| tier1 Go サービス | 個別関数 / メソッド | Dapr SDK をモック | GHA runner Pod |
| tier1 Rust サービス | 個別関数 / モジュール | ZEN Engine は実インスタンス (in-process) | GHA runner Pod |
| tier1 クライアントライブラリ (C# / Go / TS) | API ラッパーのシリアライズ / バリデーション | gRPC スタブ | GHA runner Pod |
| tier2 / tier3 サービス | 業務ロジック | tier1 公開 API をモック | GHA runner Pod |

ユニットテストの基準:

- **カバレッジ目標**: tier1 は 80% 以上、tier2 / tier3 は 70% 以上
- **実行時間**: 全ユニットテストが **3 分以内** に完了すること
- **Dapr sidecar 不要**: ユニットテストは Dapr を一切起動しない

### 2.2 統合テスト

| 対象 | テスト範囲 | 外部依存 | 実行環境 |
|---|---|---|---|
| tier1 Go ファサード ↔ Dapr | Dapr building block の呼び出しと応答 | Dapr sidecar (daprd) + 軽量バックエンド | k8s 上の test namespace |
| tier1 Go ↔ Rust | gRPC 経由のサービス間通信 | 実 Rust サービス | k8s 上の test namespace |
| tier1 クライアント ↔ tier1 サービス | 公開 API の疎通と応答形式 | 実 tier1 サービス + Dapr | k8s 上の test namespace |
| tier2 ↔ tier1 | 業務サービスから tier1 API を実際に呼ぶ | 実 tier1 + Dapr + 軽量 infra | k8s 上の test namespace |

統合テストの基準:

- **実行時間**: **10 分以内** に完了すること
- **テスト環境**: 専用の `test` namespace に一時的にデプロイし、テスト後に破棄する
- **Dapr Component**: テスト用の Component (in-memory state store / in-memory pubsub) を使用し、Kafka / PostgreSQL への依存を避ける

### 2.3 E2E テスト

| 対象 | テスト範囲 | 実行環境 |
|---|---|---|
| ユーザーシナリオ | 配信ポータルからのアプリ起動 → tier1 API 呼び出し → 監査ログ記録 | staging 相当の k8s クラスタ |
| CI/CD パイプライン | GHA → ビルド → スキャン → Harbor push → Argo CD 同期 | 実環境 |
| 認証フロー | Keycloak ログイン → SSO で Backstage / Argo CD / Harbor にアクセス | 実環境 |

E2E テストの基準:

- **実行頻度**: PR ごとではなく、**main ブランチマージ後 / 日次** で実行
- **実行時間**: **30 分以内**
- **失敗時の扱い**: 即座のロールバックではなく、アラート通知 → 手動判断

## 3. tier1 固有のテスト課題と対処

### 3.1 Go + Rust 境界のテスト

tier1 内部で Go サービスと Rust サービスが gRPC で通信する。この境界のテストには特別な配慮が必要。

| 課題 | 対処 |
|---|---|
| 2 言語のビルドが必要 | CI ワークフローで Go / Rust を並列ビルドし、統合テスト step で合流 |
| Protobuf 契約の整合性 | `buf breaking` で `.proto` の後方互換を CI で自動検証 |
| デバッグの困難さ | 統合テスト失敗時は gRPC のリクエスト / レスポンスをログ出力して原因特定を容易にする |

### 3.2 Dapr 依存テストの分離

| テスト種別 | Dapr の扱い | 使用するバックエンド |
|---|---|---|
| ユニットテスト | Dapr SDK をモック | なし |
| 軽量統合テスト | `daprd` を `--resources-path` でローカル起動 | in-memory state / in-memory pubsub |
| フル統合テスト | k8s 上の Dapr sidecar injection | Valkey (state) / Kafka (pubsub) |

軽量統合テス��は GHA runner Pod 内で `daprd` プロセスを直接起動することで、k8s への依存なしにテストできる。

### 3.3 多言語クライアントライブラリのテスト

| 言語 | テストフレームワーク | テスト内容 |
|---|---|---|
| C# | xUnit | シリアライズ / デシリアライズ、gRPC スタブ応答検証、エラーハンドリング |
| Go | `testing` + `testify` | 同上 |
| TypeScript | Vitest | 同上 |

各言語のクライアントライブラリは、**Protobuf から自動生成されたコード + 薄いラッパー** で構成されるため、テスト対象はラッパー層のロジックに限定する。

## 4. テスト環境

### 4.1 テスト環境の種別

| 環境 | 用途 | ライフサイクル | infra 依存 |
|---|---|---|---|
| ローカル + Tilt (開発者 PC) | 開発中の即時確認・探索的開発 | 常時 | ローカル k3s + Dapr (in-memory) |
| ローカル (開発者 PC) | ユニットテストの実行 | 常時 | なし |
| GHA runner Pod | ユニットテスト + 軽量統合テスト | PR 単位で起動・破棄 | `daprd` のみ |
| test namespace (k8s) | 統合テスト | テスト実行時に作成・終了後に破棄 | Dapr + in-memory バックエンド |
| staging (k8s) | E2E テスト | 常時稼働 (Phase 3 以降) | フルスタック |

### 4.2 MVP でのテスト環境

MVP (Phase 1) では staging 環境を持たない。テストは以下の 2 段階で実施する。

1. **GHA runner Pod**: ユニットテスト + 軽量統合テスト
2. **本番 k8s の test namespace**: tier1 リファレンス実装の統合テスト

E2E テストは Phase 2 以降で本格導入する。MVP では手動での動作確認をテスト手順書として TechDocs に記録する。

MVP-1a から **Tilt によるローカル開発環境** を導入し、CI を回す前に手元で即時動作確認を可能にする。詳細は [`04_ローカル開発環境.md`](./04_ローカル開発環境.md) を参照。

## 5. CI パイプラインへの統合

### 5.1 PR 時 (全 tier 共通)

| ステージ | テスト種別 | 失敗時の動作 |
|---|---|---|
| Lint / フォーマット | — | PR ブロック |
| ユニットテスト | ユニット | PR ブロック |
| 軽量統合テスト (tier1 のみ) | 統合 (Dapr in-memory) | PR ブロック |
| `buf breaking` (tier1 のみ) | 契約互換性 | PR ブロック |
| Apicurio 互換性チェック (Phase 2 以降) | イベント契約互換性 | PR ブロック |

### 5.2 main マージ後

| ステージ | テスト種別 | 失敗時の動作 |
|---|---|---|
| フル統合テスト | 統合 (k8s test namespace) | アラート通知、Argo CD 同期を一時停止 |
| E2E テスト (Phase 2 以降) | E2E | アラート通知 |

---

## 6. テストデータ管理

| 方針 | 内容 |
|---|---|
| テストデータは fixture として管理 | `testdata/` ディレクトリにテスト用 JSON / YAML を配置 |
| 本番データをテストに使用しない | 個人情報 / 業務データの混入を防止 |
| ZEN Engine のテスト | 決定表 (DMN JSON) のテストケースを `testdata/decisions/` に配置 |
| DB テスト | CloudNativePG のテスト用 DB をテスト開始時に作成、終了後に DROP |

---

## 7. Phase 別のテスト拡充

| Phase | テスト拡充内容 |
|---|---|
| Phase 1 (MVP) | ユニットテスト + 軽量統合テスト + 手動動作確認 + Tilt によるローカル開発環境 |
| Phase 2 | フル統合テスト自動化 + 多言語クライアントのクロスバージョンテスト + **Litmus による Chaos Engineering** (PostgreSQL / Valkey / Dapr Sidecar の縮退動作自動検証、週次 CronChaosEngine 実行) |
| Phase 3 | E2E テスト自動化 + パフォーマンステスト (k6) + Litmus のカスケード障害テスト拡張 (ネットワーク障害注入・本番 namespace 対象) |
| Phase 4 | セキュリティテスト (DAST) + 負荷テストの定期実行 |
