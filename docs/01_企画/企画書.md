# k1s0

## 個人開発のクラウドネイティブ
## アプリケーションプラットフォーム

<br>

**ライセンス**: Apache 2.0
**著作者**: 起案者（個人）
**版**: v1.0

---

# k1s0 とは何か

- **何を**: クラウドネイティブ技術を統合した **OSS のアプリケーションプラットフォーム**
- **誰が作ったか**: 起案者が業務外で個人的に開発している
- **何を解決するか**: 「古い単一技術のモノリス集合体」「横断的関心事のコピペ実装」「商用基盤のベンダーロックイン」を 1 つの基盤で解消する
- **会社との関係**: 会社（起案者の雇用主）が業務システムとして利用している。会社の予算・工数で開発されたものではなく、会社が自発的に採用している
- **位置づけ**: Linux / Docker / Vue.js のように個人発の OSS が結果として企業に採用された系譜のなかにある

---

# k1s0 が個人開発である理由

- **段階導入は企業プロジェクトの作法**: 組織体力をマネジメントする仕組みであり、個人開発には不要
- **設計の純度を保つ**: 段階移行のための後方互換やオプショナルな実装を入れず、最初から本来あるべき形を実装する
- **時間軸の主権**: 期日も納期もない。納得するまで作り込んでから世に出す
- **美意識の貫徹**: 既存の業務システム（.NET Framework モノリス集合）への個人的な不満が動機の根底にある

詳細: [`01_背景と目的/06_個人開発OSSの系譜.md`](./01_背景と目的/06_個人開発OSSの系譜.md)

# 1. 解決する課題

## 技術的負債のスパイラル

古い技術で作りこむ → 新技術への移行が困難 → 新技術を提案しても「よくわからないからダメ」 → 学習する若手が報われない → 古参が古い技術にしがみつく → さらに新技術への移行が困難に (ループ)

**結果**: 単一技術ごとのモノレポの集合が「崩れかけのジェンガ」状態になる

# 3. k1s0 の設計方針

> **「OSS 積み上げで、レガシーと共存できる、ベンダーに縛られない開発プラットフォーム」**

## 5 つの設計特性

1. **無償で始められる** — Apache 2.0 OSS、ライセンス費ゼロ
2. **レガシー (.NET Framework) と共存** — 段階的に取り込める
3. **オンプレ / VM で完結** — クラウド依存なし
4. **言語を尊重** — tier2 / tier3 は C# / Go / MAUI / React 自由
5. **採用ハードルを下げる** — Apache 2.0 ライセンス + 雛形 + ドキュメント整備

# 7. tier1 の核心: Dapr 隠蔽

## 課題
- Dapr は強力だが、tier2 / tier3 で直接使うと **Dapr に縛られる**
- バージョン更新 / 破壊的変更 / 将来差し替え時の影響が広範に

## 解決策
**tier1 公開 API (多言語ライブラリ) で Dapr を完全に隠蔽 / 内部は Go ファサードが stable Dapr Go SDK を直接利用**

```csharp
// tier2/tier3 のコード — Dapr を一切意識しない
await k1s0.Log.Info("注文を受領", new { orderId });
await k1s0.State.SaveAsync("orders", orderId, order);
await k1s0.PubSub.PublishAsync("order-events", "created", order);
await k1s0.Audit.RecordAsync("ORDER_CREATED", userId, orderId);
```

→ Dapr のバージョン更新 / 差し替えの影響を **tier1 内に閉じ込められる**

詳細: [`02_tier1設計/01_設計の核/01_Dapr隠蔽方針.md`](../02_構想設計/02_tier1設計/01_設計の核/01_Dapr隠蔽方針.md)

# 7. tier1 の核心: バラつきを防ぐ多層防御

> **「機械で遮断できる逸脱はツールで、残余は人のレビューで」**

静的言語 (Go / Rust / C#) の禁止 import は ①〜④ で機械的に検出できるが、動的言語 (TypeScript / Python) の動的ロード・リフレクション経由の Dapr SDK 呼び出しや、Opinionated API の意図に反する「型は合うが設計思想を外した」使い方は CI 単独では検出しきれない。残余は ⑤ の PR レビューで捕捉し、頻出パターンを ⑥ の内製 analyzer に吸い上げる多段構造で、完全自動化ではなく継続的な捕捉率向上を狙う。

| 段 | 施策 |
|---|---|
| ① | **雛形生成 CLI** — ゼロから書かせない |
| ② | **Opinionated API** — やり方を 1 通りに絞る |
| ③ | **CI ガード** — 静的言語の禁止 import を機械検出 (Dapr SDK / Kafka クライアント直接利用 等) |
| ④ | **リファレンス実装** — 模範サービスを 1 本提供 |
| ⑤ | **PR チェックリスト** — 動的言語・設計思想逸脱を人のレビューで補完 |
| ⑥ | **内製 analyzer** — 頻出逸脱を吸い上げて成長する |

詳細: [`02_tier1設計/02_API契約/03_API設計原則.md`](../02_構想設計/02_tier1設計/02_API契約/03_API設計原則.md)

# 10. アプリ配信ポータル: 端末設定コピー

## 解決する課題
PC リプレース時、旧端末の **アプリ一覧 + 設定** を新端末へ復元するのに情シスが消耗

## 仕組み
- tier1 公開 API `k1s0.Settings` で各アプリが設定を端末横断で同期
- 新端末ログイン時に **「旧端末からコピー」** を 1 クリックで実行
- コピー元: ① 自分の旧端末 / ② 同僚 (テンプレート化) / ③ 部署標準テンプレート

## コピー可否の設計原則

**権限の引き継ぎではなく利便性の引き継ぎ** — 申請制アプリの権限は新端末で再評価。コピー対象は tier1 `k1s0.Settings` スキーマで **`copyable: true` を明示したフィールドのみ** (デフォルト拒否)。除外対象を明文化:

- **認証値**: パスワード / OAuth トークン / API キー / セッション・デバイストークン
- **個人・端末固有値**: 端末 ID / MAC アドレス / ライセンスキー / 生体認証ハッシュ
- **一時キャッシュ**: 検索履歴の一部 / ワンショット UI 状態

全コピー操作を tier1 監査ログに「誰が・いつ・どのフィールドを・どの端末間で」記録する。

# 11. セキュリティと非機能要件

## セキュリティモデル

| 層 | 対策 |
|---|---|
| 認証 | Keycloak OIDC で全コンポーネントを SSO 統一 |
| 通信暗号化 | Istio Ambient ztunnel HBONE (サービス間 mTLS) + TLS (外部) |
| ネットワーク分離 | k8s NetworkPolicy で tier 間の不正通信を拒否 |
| 認可 | Keycloak RBAC + k8s RBAC + Istio AuthorizationPolicy (L7 は waypoint proxy 経由) |
| サプライチェーン | Trivy スキャン + Cosign 署名 + Kyverno + SLSA L3 (hermetic runner + Rekor 登録) |
| 監査 | tier1 `k1s0.Audit` で全操作を記録、改ざん防止 (ハッシュチェーン) |

## 非機能目標値

| 分類 | 目標値 |
|---|---|
| tier1 API レイテンシ (p99) | 全 API < 500 ms |
| 可用性 (業務時間帯) | 99% |
| バックアップ RPO | PostgreSQL: 数秒、etcd: 24 時間 |
| バックアップ RTO | PostgreSQL: 15 分、クラスタ全壊: 4 時間 |
| CVE 対応 (Critical) | 48 時間以内 |

# 13. ワークフロー 2 基盤の棲み分け

## なぜ 2 つあるか

k1s0 は **Dapr Workflow**（Dapr 同梱、tier1 Go ファサード経由）と **Temporal**（別サーバ、tier1 Go ファサード経由）の 2 つのワークフロー基盤を併用する。「**短期・簡易フロー用 (Dapr Workflow) と長期・複雑フロー用 (Temporal) で棲み分け、tier2/3 からは同一 `k1s0.Workflow` API で抽象化**」する設計を採っている。

## 棲み分け基準

| 軸 | Dapr Workflow | Temporal |
|---|---|---|
| 想定実行時間 | 秒〜数分 (例: 注文 Saga / 在庫引当) | 数時間〜数週間 (例: 申請承認 / 個人情報削除 / 新人オンボーディング) |
| 想定ステップ数 | 10 以下 | 数十〜数百 |
| リトライ / タイマー | 単純な指数バックオフ | 複雑なスケジュール・人的承認待ち・SLA タイマー |
| 可視化 / 監査 UI | 最小 (ログ + Tempo Trace) | Temporal Web UI で 1 ワークフローを時系列で可視化 |
| バックエンド | Dapr State Store (Postgres) | Temporal 専用 Postgres |

tier2/3 からは `k1s0.Workflow.StartAsync(workflowType, input)` のみを叩く。tier1 Go ファサードが workflowType の属性 (YAML 設定) を見て Dapr Workflow と Temporal に自動振り分けする。詳細は [`02_tier1設計/04_診断と補償/05_ワークフロー振り分け基準.md`](../02_構想設計/02_tier1設計/04_診断と補償/05_ワークフロー振り分け基準.md) を参照。

> **要点**: tier2/3 からは 1 つの API に見え、振り分けは tier1 内部に閉じている。Dapr Workflow 単独では数週間の承認フロー・GDPR 30 日猶予の時限実行が表現できず、Temporal 単独では短期 Saga に過剰な学習コストを強いる。

# 14. 個人開発の前提と限界

k1s0 は個人開発の OSS であり、以下の前提のもとで運用されることを想定する。

## 前提

- **バス係数 = 1**: 起案者が継続的に維持する。会社利用者は自身でフォーク・内製化する余地を確保する責任を持つ
- **SLO / SLA は提供しない**: OSS としての品質保証はない。会社が業務利用する以上、運用責任は会社側にある
- **dogfooding で品質を担保**: 起案者が日常利用し、会社が業務利用することで継続的にフィードバックが得られる構造
- **OSS としての公開**: GitHub で Apache 2.0 ライセンスとして公開済。社外利用・コントリビュートを排除しない

## 既存資産との関係

- 既存 .NET Framework アプリは **tier3 の拡張ポイント** として共存
- 共存方式: **サイドカー方式** または **API Gateway 経由**
- 移行を強制しない

詳細: [`01_背景と目的/03_新技術導入リスクへの回答.md`](./01_背景と目的/03_新技術導入リスクへの回答.md)

# 15. ライセンスと法務

## k1s0 のライセンス

- **本体**: Apache 2.0
  - 商用利用・改変・再配布が自由
  - 明示的特許許諾条項あり
  - ZEN Engine (MIT) と整合
- **依存 OSS の AGPL コンポーネント**: Grafana / Loki / MinIO 等を `infra/` レイヤで利用
  - k1s0 本体のライセンスには伝播しない構成
  - 採用側が AGPL 義務（ソース改変時の公開義務）の発動条件を確認する責任を負う

## 採用側が確認すべき法務論点

| 論点 | 確認事項 |
|---|---|
| AGPL 依存 OSS の利用 | `infra/` 内部で動作する 6 本（Grafana / Loki / MinIO 等）。FSF / Grafana Labs 公式 FAQ により通常利用では義務発動なし、改変時は採用側で確認 |
| 個人情報保護法 / マイナンバー / 電帳法 | k1s0 は業務データを保持しない。必要な技術要件 (監査ログ / 暗号化 / RBAC / ハッシュチェーン) は tier1 公開 API で基盤提供 |
| 知財帰属 | k1s0 の著作権は **起案者**。Apache 2.0 ライセンスにより誰でも利用・改変・再配布が可能 |
| 監査応答 | SBOM / CVE / ライセンス判定 / 四半期レポートを CI 自動生成、`infra/compliance-reports` で 5 年保管 |

詳細: [`05_法務サマリ/`](./05_法務サマリ/) / [`../02_構想設計/05_法務とコンプライアンス/`](../02_構想設計/05_法務とコンプライアンス/)
