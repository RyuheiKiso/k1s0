# 競合マップ

## 目的

k1s0 が対象とする「開発者プラットフォーム / マイクロサービス基盤」領域の既存製品・事例を分類し、各製品と k1s0 の関係 (競合 / 採用 / 参考) を俯瞰する。個別の評価は [`01_主要製品別評価.md`](./01_主要製品別評価.md)、差別化の論旨は [`02_差別化ポイント.md`](./02_差別化ポイント.md) を参照。

## 注意事項

本資料および関連資料の各製品情報は一般に公開されている情報に基づく概観である。企画書に数値や機能詳細を引用する場合は、公式ドキュメント最新版での再確認を必須とする。

## 1. 競合の分類

| 分類 | 代表例 | k1s0 との関係 |
|---|---|---|
| OSS 型 IDP (Internal Developer Platform) | Backstage | **採用 (開発者ポータル領域)** |
| 商用 IDP / PaaS | Humanitec / Port / Cortex | 機能重複 (有償) |
| 商用 k8s ディストリビューション | OpenShift / Tanzu | 機能重複 (有償・ベンダー依存) |
| OSS k8s 管理プラットフォーム | Rancher | 採用候補 |
| マイクロサービス building blocks | **Dapr** | **tier1 と最も機能重複する (採用して隠蔽)** |
| サービスメッシュ / Gateway 商用プラットフォーム | Gloo Platform (Solo.io) | 機能重複 (有償) |
| CNCF アプリ配信 / 抽象化 | KubeVela / Crossplane | 思想的参考 |
| ポリシー / ガバナンス | Nirmata / Kyverno | 部分競合 / 採用候補 |
| レガシー PaaS | Cloud Foundry / Tanzu Application Service | 移行元 (既存利用企業向け) |
| マイクロサービスプラットフォーム (商用) | Mia-Platform | 機能重複 (有償) |
| 国内大手の自社開発 PaaS | 各社テックブログで公開されている事例 | k1s0 アプローチの妥当性事例 |

---

## 2. k1s0 との関係パターン

製品との関係は 5 つに大別できる。

| パターン | 意味 | 該当製品 |
|---|---|---|
| **採用** | k1s0 の構成要素として取り込む | Backstage (開発者ポータル) / Dapr (tier1 内部、隠蔽) / Kyverno (ポリシー候補) / Rancher (infra 層候補) |
| **機能重複 (商用)** | 同じ問題を有償製品で解いているため、OSS 積み上げの無償代替として提案 | OpenShift / Tanzu / Gloo Platform / Mia-Platform / Humanitec |
| **思想的参考** | 考え方を取り入れるが直接は採用しない | KubeVela (OAM) / Crossplane / Score |
| **移行元** | 既存導入企業向けの移行パス提示対象 | Cloud Foundry / Tanzu Application Service |
| **方向性の妥当性事例** | 「大企業が自社用に k8s 基盤を内製する」潮流の裏付け | 国内大手テック企業の公開事例 |

## 3. Dapr を競合とせず採用する論旨の要点

Dapr との機能重複は大きく、企画書では必ず明示的に触れる必要がある。k1s0 の方針は 2026-04-12 に決定済み。

| 論点 | k1s0 の回答 |
|---|---|
| Dapr と機能が重複するのでは? | **採用する**。ただし tier1 の内部実装として組み込み、tier2 / tier3 からは隠蔽する |
| tier2 / tier3 開発者は Dapr を意識するか? | 一切しない。tier1 公開 API (多言語クライアントライブラリ) のみを使う |
| Dapr の破壊的変更・差し替えリスク | tier1 内に閉じ込める。tier2 / tier3 には波及しない |
| Rust SDK が Alpha の問題 | Daprファサードは Go 実装 (stable Go SDK)。Rust は Dapr 非依存の自作領域に限定 |

詳細は [`../03_tier1設計/01_Dapr隠蔽方針.md`](../03_tier1設計/01_Dapr隠蔽方針.md) / [`../03_tier1設計/02_内部言語ハイブリッド.md`](../03_tier1設計/02_内部言語ハイブリッド.md) を参照。

## 4. 国内大手事例の位置付け

国内大手テック企業 (Web / モバイル系) の多くが、k8s をベースにした自社マイクロサービスプラットフォームを内製している。ただし各社の具体情報を企画書に引用する際は、**公開テックブログ・登壇資料から最新事例を個別に収集する必要がある**。

一般的な傾向として以下が確認できる。

- 共通基盤チームを設置し、アプリチームが業務ロジックに集中できる体制を敷く事例が多い
- 公開資料の多くはクラウド (AWS / GCP) マネージド前提で、オンプレ / VM 上での k8s 運用ノウハウは相対的に情報量が少ない
- 事例の多くは **Web / モバイル企業** が中心で、**JTC 情シス部門による内製事例は相対的に少ない**

### 示唆

- 「大企業が自社用に k8s ベースの基盤を作る」は業界のトレンドとして既に成立しており、k1s0 のアプローチは奇抜ではない
- k1s0 の独自性は「JTC 情シス部門でも運用できること」「.NET Framework 等のレガシー共存」「オンプレ / VM 前提」にある
