# 差別化ポイント

## 目的

既存製品では解けない (または解く気のない) JTC 情シス固有の要件を k1s0 がどう埋めるかを整理する。ペルソナ別の価値提案は [`../01_背景と目的/02_解決する価値.md`](../01_背景と目的/02_解決する価値.md) と対で読むこと。

---

## 1. 軸別の比較

既存の製品 / 事例の大多数と k1s0 の立ち位置を軸で並べる。

| 軸 | 既存製品 / 事例 | k1s0 |
|---|---|---|
| コスト | 商用製品は有償サブスクリプション | OSS 積み上げで無償 |
| ベンダー依存 | OpenShift / Tanzu / Gloo / Mia-Platform はロックインあり | 回避を設計原則とする |
| 対象ユーザー層 | Web 系テック企業 / 大規模 SaaS 向けが多い | **JTC 情シス部門** |
| レガシー共存 | 考慮されないケースが多い | **.NET Framework 等の古い資産との共存を第一級で扱う** |
| 実行環境 | クラウドマネージド前提が多い | **オンプレ / VM を第一級で扱う** |
| 言語の自由度 | プラットフォーム指定の言語が中心 | **tier2 / tier3 は C# / Go / TS / MAUI / React など自由。既存資産の言語を尊重** |
| Dapr の扱い | tier2 / tier3 が直接 Dapr SDK / annotation を扱う前提 | **tier1 内部実装として隠蔽。tier2 / tier3 は Dapr を一切意識しない** |
| レガシー所有者の動機付け | 移行を強制 / 無視 | 運用コスト削減・デプロイ速度などの金銭的メリットで **「作り直してでも参加したくなる」** 設計 |
| 人材要件 | k8s / サービスメッシュの専門家前提 | 未経験チームでも始められる雛形・ドキュメント提供 |
| 基盤言語 | Go / TypeScript / Java 中心 | **Rust で長期保守性を最優先** (自作領域) + **Go** (Daprファサード) |

---

## 2. JTC 情シス部門固有の選定軸

k1s0 は一般的な開発者プラットフォームとは異なる「JTC 情シス部門」という特殊な市場を対象とする。この市場に特有の選定軸は以下のとおり。

| 軸 | JTC 情シスでの重要度 | 既存製品の典型的な立ち位置 | k1s0 の立ち位置 |
|---|---|---|---|
| **稟議が通るか** | 極めて高 | 商用は数百万円〜で稟議ハードル高 | **無償 OSS なので稟議不要** |
| **情報システム子会社でも運用できるか** | 高 | 専門家前提で運用困難 | 雛形・ドキュメント・段階導入で対応 |
| **既存ベンダー契約との整合** | 高 | OpenShift 等は IBM / Red Hat 契約が前提 | 既存契約に影響しない |
| **セキュリティ監査・ISO への適合** | 高 | 個別製品ごとに監査資料を用意 | OSS 積み上げ構成で監査は tier 単位 |
| **日本語サポート・ドキュメント** | 中 | 商用は日本法人サポートあり | 自社内のノウハウ蓄積に依存 |
| **既存 .NET Framework 資産の扱い** | 極めて高 | 通常は考慮されない | **第一級で対応** |
| **オンプレ DC / VMware 環境での動作** | 高 | クラウドマネージド前提が多い | **前提条件として明示** |

## 3. k1s0 の勝ち筋 5 点セット

**同じ機能を提供する製品は既に多数存在する**。機能そのものでは差別化できない。特に Dapr とは機能的な重複が大きい。機能ではなく **ターゲット市場と運用可能性** で勝ち筋を成立させる。

| # | 勝ち筋 | 要点 |
|---|---|---|
| 1 | **JTC 情シスが無償で始められる** | 稟議のハードルがない。OSS 積み上げで初期ライセンス費ゼロ |
| 2 | **レガシーと共存できる** | .NET Framework 等の既存資産を捨てずに段階移行できる ((C) ラップ配信フォーム / tier2 サイドカー共存) |
| 3 | **ベンダー依存しない** | オンプレ / VM で完結し、クラウド接続必須・ベンダー契約必須の構造を持たない |
| 4 | **言語を尊重する** | tier2 / tier3 の言語を縛らず、既存資産を活かせる。tier1 公開 API が各言語向けに提供される |
| 5 | **レガシー所有者に金銭的メリット** | 運用コスト削減・デプロイ速度改善で自発的な参加を促す。強制移行ではない |

企画書ではこの 5 点を一貫した主張として繰り返す。

---

## 4. 既存 OSS との非敵対設計

既存 OSS (Rancher / Backstage / Kyverno 等) とは敵対せず、**組み合わせて使う前提** で設計する。開発スコープを現実的に抑えるための重要な方針。

| 既存 OSS | k1s0 での扱い |
|---|---|
| Backstage | 開発者ポータルとして正式採用 |
| Dapr | tier1 内部実装として採用し、tier2 / tier3 から隠蔽 |
| Rancher | infra 層の k8s 管理 GUI の採用候補 |
| Kyverno | セキュリティ / ポリシー層の採用候補 |
| OPA | セキュリティ / 認可ポリシーで ZEN Engine と棲み分け |
| Istio / Envoy Gateway | infra 層として採用 |
| Kafka / Loki / Prometheus / Jaeger / Grafana | infra 層として採用 |

---

## 5. 商用競合との位置付け

商用競合 (Gloo Platform / Mia-Platform / Humanitec / OpenShift / Tanzu) は同じ問題意識から既に製品化されている。k1s0 はそれらを **無償の OSS 積み上げで代替することを目指す**。

- 「Gloo Platform と同等のものを OSS 積み上げで無償実現」
- 「Mia-Platform の OSS 版」
- 「OpenShift の JTC 向けオンプレ最小構成」

これらはすべて企画書での対比説明として有効な表現である。ただし、開発者ポータル機能や GUI の洗練度では商用製品が先行している点は正直に認める。k1s0 の勝ち筋は「機能の最先端」ではなく「JTC で稟議を通せる・運用できる・既存資産を活かせる」にある。
