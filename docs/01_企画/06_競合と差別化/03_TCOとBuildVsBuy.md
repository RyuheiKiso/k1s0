# TCO と Build vs Buy

## 目的

k1s0 を内製する (Build) か商用製品を購入する (Buy) かの選択は企画書の核心論点となる。本資料では 5 年スパンでの TCO を概念比較し、両選択肢のリスクを整理する。

> **注意**: 本資料の表は概念レベルの比較である。実数値による 5 年 TCO 試算は [`../08_定量試算/01_TCO5年試算.md`](../08_定量試算/01_TCO5年試算.md) に移管した (従業員 1,000 / 3,000 / 10,000 名の 3 規模で実数比較)。本資料は意思決定の論旨構成と概念理解の入口として位置づけ、数的根拠は 08_定量試算 を参照。

## 1. TCO ざっくり比較 (5 年スパン、概念レベル)

| 項目 | 商用 k8s ディストリビューション (OpenShift 等) | 商用 IDP (Humanitec 等) | k1s0 (OSS 積み上げ + 内製) |
|---|---|---|---|
| ライセンス費 | ○ (高) | ○ (中〜高) | - |
| 初期構築費 | △ (設定のみ) | △ (設定のみ) | ◎ (開発コスト) |
| 運用保守費 | ○ (ベンダー保守) | ○ (ベンダー保守) | △ (自社運用) |
| 人材育成費 | △ (研修あり) | △ (研修あり) | ◎ (ノウハウ内製化) |
| 撤退コスト / ロックイン解消費 | ◎ (非常に高) | ○ (高) | - (ロックインなし) |
| **5 年 TCO の傾向** | **高** | **中〜高** | **中** (初期高、運用段階で下がる) |

凡例: `◎` コスト大 / `○` コスト中 / `△` コスト小 / `-` 不要

### 示唆

- 初期開発コストは k1s0 の方が高いが、ライセンス費と撤退コストを考慮すると **5 年スパンでは k1s0 が優位になる可能性が高い**
- 企画書ではこの概念を具体的な試算に落とす必要がある
- 特に「撤退コスト / ロックイン解消費」は商用製品で非常に大きくなるが、見落とされがち。企画書で明示的に触れる

## 2. Build (k1s0 を内製) のリスクと対処

| リスク | 発生確率 | 影響度 | 対処方針 |
|---|---|---|---|
| 個人開発ゆえの属人化 | 高 | 高 | **ドキュメント充実 + コード公開** でバス係数を下げる |
| 開発が途中で頓挫する | 中 | 高 | MVP スコープを最小化して早期に価値提供 |
| 新技術追従の工数が読めない | 中 | 中 | OSS 積み上げなのでコミュニティ更新に追従するだけ |
| セキュリティ脆弱性の対応遅延 | 中 | 高 | 採用 OSS の CVE 監視と tier1 の修正体制を事前に設計 |
| 所属企業の評価 / 支援が得られない | 中 | 中 | 無償提供の時点で ROI を明示して合意を取る |

---

## 3. Buy (商用製品を購入) のリスク

| リスク | 発生確率 | 影響度 | 対処方針 |
|---|---|---|---|
| ライセンス費が予算を圧迫 | 高 | 高 | k1s0 なら回避 |
| ベンダーロックイン | 高 | 極高 | k1s0 なら回避 |
| ベンダー都合の仕様変更 / 値上げ (例: VMware Broadcom 事例) | 中 | 極高 | k1s0 なら回避 |
| 自社要件 (.NET Framework 共存等) が製品側でサポートされない | 高 | 高 | k1s0 なら自由に対応 |
| 日本法人のサポート品質が担保されない | 中 | 中 | k1s0 は自社運用で品質自己決定 |

---

## 4. リスクの構造的な違い

Build と Buy のリスクは **質が異なる** ことを理解することが重要。

| 観点 | Build (k1s0) | Buy (商用) |
|---|---|---|
| リスクの源 | 「個人開発起点である」という組織的事情 | 「契約構造・ベンダー支配力」という構造的事情 |
| 解消可能性 | 段階的に会社として引き取る体制を整えれば **低減できる** | 契約締結後の解消は **困難** (撤退コストが大きい) |
| 時間軸 | 初期ほどリスクが高く、運用段階で下がる | 初期は安全に見え、契約更新ごとにリスクが顕在化する |

### 示唆

Build 側のリスクは「個人開発起点」に由来するものが主で、段階的に会社として引き取る体制を整えれば低減できる。Buy 側のリスクは「構造的なロックイン」に由来するものが多く、契約締結後の解消は困難。

**k1s0 の企画書では Build のリスクへの対処方針を具体的に示すこと** が説得力の鍵となる。

---

## 5. 企画書での論旨構成

決裁者向けの答弁は以下の順で組み立てる。

1. **機能重複の事実を認める** — 既存製品で同等の機能は実現できる
2. **商用製品では JTC 固有要件が埋まらない** — レガシー共存 / オンプレ / 稟議文化 / 言語自由 ([`02_差別化ポイント.md`](./02_差別化ポイント.md))
3. **5 年 TCO で見ると k1s0 が優位** — 初期は高いが運用段階で下がる / ロックイン解消費が不要
4. **Build のリスクは対処可能** — ドキュメント・MVP スコープ・OSS 積み上げ方針で個人開発リスクを段階的に解消する
5. **Buy のリスクは構造的で対処困難** — 契約更新ごとに顕在化し、撤退コストが膨大
