# パフォーマンス帰属設計

## 目的

tier1 の Dapr 隠蔽と多層アーキテクチャにより、パフォーマンス問題発生時に「どの tier が原因か」を客観的に特定する手段が不足している。本資料では、**人間の判断に依存せずレイテンシの帰属先を自動的に示す**ための 2 つの仕組みを定義する。

- **レイテンシ帰属メトリクス** — 各 tier のクライアントライブラリが自動エクスポートするメトリクスにより、事後分析でウォーターフォールを組み立てる
- **セルフサービストレース探索** — tier3 開発者が Grafana Tempo の操作スキルなしにボトルネック tier を特定できる UI / CLI を提供する

設計原則: 「誰のせいか」をチーム間の議論ではなく、データが自動的に示す状態を作る。

---

## 1. 背景となる課題

### 1.1 既存の診断機能のカバー範囲

[`08_診断機能設計.md`](./08_診断機能設計.md) で定義した診断コンテキストは **tier1 内部の動作**を可視化するが、リクエスト全体のうち各 tier が消費した時間を示す機能は含まない。

| 既存機能 | カバー範囲 | 不足 |
|---|---|---|
| `Diagnostics.LatencyMs` | tier1 内部の処理時間 | tier2 / tier3 の処理時間は含まない |
| `Diagnostics.Source` | キャッシュヒットかバックエンド直アクセスか | 同上 |
| `Error.Layer` | エラー発生時の帰属 tier | エラーなしの「遅い」ケースには使えない |
| `k1s0.Diag.HealthCheckAsync()` | tier1 バックエンドの健全性 | 「どの tier が全体レイテンシの何 % を消費しているか」は示さない |

### 1.2 ウォーターフォール組み立ての制約

リクエストの全体ウォーターフォール (tier3 → tier2 → tier1 → infra) を**リアルタイムの API レスポンスに含める**ことはできない。tier1 は自身より上流の tier2 / tier3 の処理時間を知らないため、レスポンスヘッダーに全 tier のレイテンシ内訳を付与する方式は成立しない。

ウォーターフォールは **Prometheus メトリクスの事後集計** または **Grafana Tempo のスパン分析** で組み立てる。

---

## 2. レイテンシ帰属メトリクス

### 2.1 概要

tier1 クライアントライブラリが Prometheus メトリクスを自動エクスポートし、「どのサービスがどの API をどのくらいの時間で呼んだか」を記録する。tier2 / tier3 開発者のコード変更は不要。

### 2.2 自動エクスポートメトリクス

tier1 クライアントライブラリが全 API 呼び出しで以下のメトリクスを自動記録する。

| メトリクス名 | 型 | ラベル | 説明 |
|---|---|---|---|
| `k1s0_client_call_duration_seconds` | Histogram | `caller_service`, `target_api`, `status` | tier1 API の呼び出し所要時間 |
| `k1s0_client_call_total` | Counter | `caller_service`, `target_api`, `status` | tier1 API の呼び出し回数 |

`caller_service` ラベルは tier1 クライアントライブラリが `k1s0.config.yaml` の `service.name` から自動注入する。tier2 / tier3 開発者がコードでラベルを設定する必要はない。

### 2.3 既存メトリクスとの関係

[`../02_アーキテクチャ/02_可用性と信頼性/05_SLOとエラーバジェット.md`](../02_アーキテクチャ/02_可用性と信頼性/05_SLOとエラーバジェット.md) で定義済みの `k1s0_api_call_total` と `k1s0_api_call_duration_seconds` は tier1 Go ファサード側でエクスポートされるプラットフォーム全体の SLI メトリクスである。本施策で追加するメトリクスは **クライアントライブラリ側** から呼び出し元サービス名付きでエクスポートするもので、用途が異なる。

| メトリクスの出力元 | 用途 | メトリクス名 |
|---|---|---|
| tier1 Go ファサード (既存) | プラットフォーム全体の SLO 追跡 | `k1s0_api_call_*` |
| tier1 クライアントライブラリ (本施策) | 呼び出し元サービス別のレイテンシ帰属 | `k1s0_client_call_*` |

名前空間を `k1s0_api_*` (サーバー側) と `k1s0_client_*` (クライアント側) で分離し、混同を防ぐ。

### 2.4 tier2 サービス間呼び出しのメトリクス

tier2 → tier2 のサービス間呼び出しは `k1s0.Service.Invoke` 経由で行われる。この呼び出しにも同じメトリクスが記録されるため、tier2 間のレイテンシ帰属も追跡可能。

```
k1s0_client_call_duration_seconds{
  caller_service="order-ui",
  target_api="Service.Invoke:payment-service",
  status="ok"
}
```

### 2.5 Grafana ダッシュボード

Prometheus メトリクスから以下のパネルを持つ Grafana ダッシュボードを自動生成する。

| パネル | 表示内容 | データソース |
|---|---|---|
| **自サービスの E2E レイテンシ** | リクエスト受信から応答までの全体時間 | Istio / Dapr サイドカーメトリクス |
| **依存先 tier2 のレイテンシ** | `Service.Invoke` 経由の呼び出し所要時間 | `k1s0_client_call_duration_seconds{target_api=~"Service.Invoke:.*"}` |
| **tier1 API のレイテンシ** | `State` / `PubSub` 等の呼び出し所要時間 | `k1s0_client_call_duration_seconds{target_api!~"Service.Invoke:.*"}` |
| **ボトルネック表示** | 全体レイテンシに対する各 tier の時間比率 | 上記パネルの値を積み上げ棒グラフで表示 |

ダッシュボードは `k1s0.config.yaml` の `service.name` に基づき、Grafana のプロビジョニング機構で自動生成する。サービスごとの手動設定は不要。

### 2.6 Backstage サービスダッシュボードとの統合

Backstage 導入後 (Phase 3) は、サービスカタログの各サービスページから上記 Grafana ダッシュボードへ直接リンクする。tier3 開発者は Backstage 上で自サービスの依存先レイテンシを確認でき、Grafana の操作方法を新たに学ぶ必要はない。

| 操作 | 手順 |
|---|---|
| 自サービスのレイテンシ確認 | Backstage → サービスページ → 「パフォーマンス」タブ |
| 特定の依存先の深掘り | パネルをクリック → CorrelationId 付きで Tempo に遷移 |

### 2.7 本番 Diagnostics ポリシーとの整合

本施策のメトリクスは **Prometheus のスクレイピング** で収集されるため、API レスポンスに追加のフィールドは不要。[`08_診断機能設計.md`](./08_診断機能設計.md) 2.4 節の「本番は `CorrelationId` のみ」というポリシーに影響を与えない。

## 3. セルフサービストレース探索

### 3.1 概要

メトリクスによる帰属分析 (2 章) は傾向と統計値を示す。個別リクエストの詳細な時間内訳を確認するには分散トレース (Grafana Tempo) が必要だが、tier3 開発者に Tempo の操作スキルを要求するのは学習コストが高い。**環境に応じた適切な UI** を提供し、Tempo を直接操作させない。

| 環境 | 提供するツール | 認証・ネットワーク |
|---|---|---|
| ローカル (Tilt) | `k1s0 dev trace` CLI | Tilt 内の Tempo に直接アクセス。認証不要 |
| ステージング / 本番 | Backstage トレースビューア | Backstage が Tempo へプロキシ。Backstage の認証で制御 |

### 3.2 ローカル環境: `k1s0 dev trace` の拡張

[`11_ローカル開発CLI設計.md`](./11_ローカル開発CLI設計.md) 2.3 節で定義済みの `k1s0 dev trace <correlation-id>` コマンドを拡張する。

#### 既存コマンドの出力拡張

出力を**tier 別のウォーターフォール形式**に変更する。各スパンに tier ラベルを付与し、ボトルネック tier を自動判定する。

```
トレース: abc-123-def
─────────────────────────────────────────────
  order-ui (tier3)              12ms
  └─ order-service (tier2)      45ms
     ├─ k1s0.State.Get (tier1)   5ms
     └─ k1s0.PubSub.Pub (tier1)  3ms

合計: 68ms
最大消費: order-service (tier2) — 全体の 66%
```

tier の判定は `k1s0.config.yaml` の `service.tier` フィールドから取得する。Backstage 導入後はサービスカタログのメタデータとも同期する。

#### 遅延リクエスト一覧コマンドの追加

```bash
k1s0 dev trace slow --service order-ui --threshold 500ms
```

```
order-ui の遅延リクエスト (閾値: 500ms)
─────────────────────────────────────────────
  CorrelationId     合計     最大消費
  abc-123-def       820ms    payment-service (tier2) 89%
  ghi-456-jkl       510ms    k1s0.State.Get (tier1)  72%

  詳細: k1s0 dev trace <correlation-id>
```

Tilt 内のローカル Tempo に対して TraceQL でクエリする。

### 3.3 ステージング / 本番: Backstage トレースビューア

Backstage にトレースビューアプラグインを追加する。tier3 開発者は Grafana Tempo を直接操作せず、Backstage 上で以下の操作を行う。

| 操作 | 手順 |
|---|---|
| CorrelationId でトレース検索 | Backstage → トレースビューア → CorrelationId 入力 → ウォーターフォール表示 |
| 自サービスの遅延リクエスト検索 | Backstage → サービスページ → 「遅延リクエスト」タブ → 閾値指定 → 一覧表示 |

Backstage が Tempo API へのプロキシを担い、認証は Backstage 自体の認証 (Keycloak 連携) で制御する。tier3 開発者に Tempo のエンドポイントやクエリ言語を公開しない。

#### アクセス制御

| ロール | 許可される操作 |
|---|---|
| tier3 開発者 | 自サービスが関与するトレースの閲覧 |
| tier2 開発者 | 自ドメインのサービスが関与するトレースの閲覧 |
| tier1 / infra チーム | 全トレースの閲覧 |

RBAC は Backstage の Permission Framework で実装する。

### 3.4 非同期パターンへの対応

`k1s0.PubSub` や `k1s0.Workflow` を使う非同期処理は、同期のリクエスト-レスポンス形式のウォーターフォールでは表現できない。非同期パターンでは**イベントフロー表示**に自動切り替えする。

#### 判定基準

トレース内のスパンに `messaging.system` 属性 (OpenTelemetry Semantic Conventions) が含まれる場合、非同期パターンと判定してイベントフロー表示に切り替える。

#### イベントフロー表示

```
トレース: xyz-789-abc (非同期)
─────────────────────────────────────────────
  order-ui (tier3)
  └─ PUBLISH "order.created"               12ms
     ... 850ms (キュー待機) ...
     payment-service (tier2)
     └─ SUBSCRIBE "order.created"           45ms
        └─ k1s0.State.Set (tier1)            5ms

合計 (発行〜最終処理完了): 912ms
キュー待機: 850ms (93%)
処理時間: 62ms (7%)
```

キュー待機時間と処理時間を分離して表示することで、「遅いのは処理なのかキューなのか」を即座に判別できる。

## 4. 段階的導入

| フェーズ | 施策 | 前提条件 |
|---|---|---|
| Phase 2 前半 | クライアントライブラリに `k1s0_client_call_duration_seconds` を追加 | tier1 クライアントライブラリの実装 |
| Phase 2 後半 | `k1s0 dev trace` のウォーターフォール出力と `k1s0 dev trace slow` を実装 | ローカル Tempo の動作 |
| Phase 2 後半 | Grafana に呼び出し元サービス別レイテンシダッシュボードを追加 | Recording Rule の設定 |
| Phase 3 | Backstage サービスページからの Grafana ダッシュボードリンク | Backstage 導入 |
| Phase 3 | Backstage トレースビューア (CorrelationId 検索 + 遅延リクエスト一覧) | Backstage + Tempo 連携 |
| Phase 3 | 非同期パターンのイベントフロー表示 | PubSub / Workflow の本格運用 |
