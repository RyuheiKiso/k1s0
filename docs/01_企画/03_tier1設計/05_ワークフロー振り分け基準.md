# ワークフロー振り分け基準

## 目的

tier1 公開 API `k1s0.Workflow.*` の内部実装として **Temporal** と **Dapr Workflow** を併用する方針を採っている ([`../04_技術選定/12_マニフェストとワークフロー.md`](../04_技術選定/12_マニフェストとワークフロー.md))。本資料は、個々のワークフローをどちらのエンジンに振り分けるかの**定量的な判断基準**を定義する。

tier2 / tier3 開発者はこの判断に関与しない。`k1s0.Workflow.*` API を呼び出すだけであり、バックエンドが Temporal か Dapr Workflow かは tier1 内部で決定する。

---

## 1. 設計原則

> **「迷ったら Temporal」**

Temporal は Dapr Workflow の上位互換に近い機能セットを持つ。振り分けの判断に迷う場合は Temporal を選択する。Dapr Workflow を選択するのは、明確にそのメリットが活きるケースに限定する。

---

## 2. 定量的振り分け基準

### 2.1 判断フローチャート

以下の質問に順番に回答し、最初に該当した選択肢を採用する。

| # | 質問 | Yes の場合 | No の場合 |
|---|---|---|---|
| 1 | 補償トランザクション (Saga) が必要か? | **Temporal** | 次へ |
| 2 | 人間の介入待ち (承認・レビュー等) が発生するか? | **Temporal** | 次へ |
| 3 | 実行時間が 5 分を超える見込みか? | **Temporal** | 次へ |
| 4 | ステップ数が 5 を超えるか? | **Temporal** | 次へ |
| 5 | 子ワークフローの起動が必要か? | **Temporal** | 次へ |
| 6 | 実行中のワークフローに外部からシグナルを送る必要があるか? | **Temporal** | 次へ |
| 7 | ワークフローの実行状態を外部からクエリする必要があるか? | **Temporal** | 次へ |
| 8 | 上記全て No | **Dapr Workflow** | — |

### 2.2 定量的な閾値

| 指標 | Dapr Workflow の適用範囲 | Temporal の適用範囲 |
|---|---|---|
| **実行時間** | 数秒 - 5 分 | 5 分 - 数週間 |
| **ステップ数** | 2 - 5 ステップ | 5 ステップ以上 |
| **補償トランザクション** | 不要 | 必要 |
| **人間の介入待ち** | なし | あり |
| **子ワークフロー** | 不要 | 必要 |
| **Signal / Query** | 不要 | 必要 |
| **実行頻度** | 高頻度 (100+ 回/日) の短期処理 | 低〜中頻度の長期処理 |
| **可視性要件** | ログで十分 | Web UI での状態確認が必要 |

## 3. ユースケース別の振り分け

### 3.1 Temporal を選択するユースケース

| ユースケース | 理由 | 該当指標 |
|---|---|---|
| **稟議承認フロー** | 人間の介入待ち (数日〜数週間)。承認者変更のシグナル対応 | 実行時間 > 5 分、人間介入あり、Signal 必要 |
| **経費精算ワークフロー** | 申請 → 上長承認 → 経理確認 → 完了の多段階。却下時の補償あり | ステップ数 > 5、補償トランザクション必要 |
| **月次バッチ処理** | 長時間実行。リトライ・チェックポイントが必須 | 実行時間 > 5 分 |
| **ユーザー退職処理** | Keycloak 無効化 → 各 tier2 論理削除 → 30 日猶予 → 物理削除のSaga | 補償トランザクション必要、実行時間 > 5 分 |
| **アプリ配信の段階ロールアウト** | カナリア → 拡大 → 全展開の多段階。各段階で確認待ち | 人間介入あり、ステップ数 > 5 |
| **年次データアーカイブ** | 大量データの移行。チェックポイント・リプレイが必須 | 実行時間 > 5 分 |

### 3.2 Dapr Workflow を選択するユースケース

| ユースケース | 理由 | 該当指標 |
|---|---|---|
| **通知の順次送信** | メール → Slack → 監査ログの 3 ステップ。補償不要 | ステップ数 < 5、実行時間 < 5 分 |
| **キャッシュウォームアップ** | 外部 API → 変換 → Valkey 書き込みの短期パイプライン | ステップ数 < 5、実行時間 < 5 分 |
| **tier1 内部のデータ同期** | 設定値の複数ストア同期など tier1 内部の短期処理 | ステップ数 < 5、Temporal 依存を増やさない |
| **ヘルスチェック集約** | 複数コンポーネントの死活確認を順次実行 | ステップ数 < 5、高頻度 |

## 4. 境界ケースの判断指針

### 4.1 実行時間が不確定なケース

外部 API 呼び出しやバッチ処理の実行時間が事前に予測できない場合:

| 条件 | 判断 |
|---|---|
| 95 パーセンタイルで 5 分以内に完了する | Dapr Workflow |
| 5 パーセンタイルでも 5 分を超える可能性がある | Temporal |
| 過去の実行データがない (新規ユースケース) | **Temporal** (安全側に倒す) |

### 4.2 ステップ数が増減するケース

ワークフローの設計段階ではステップ数が少なくても、将来的に拡張される可能性がある場合:

| 条件 | 判断 |
|---|---|
| 現時点で 3 ステップ、今後の拡張予定なし | Dapr Workflow |
| 現時点で 3 ステップだが、Phase 3 以降で拡張予定 | **Temporal** (移行コストを回避) |

### 4.3 Dapr Workflow → Temporal への移行

Dapr Workflow で開始したワークフローが要件拡大で Temporal に移行する場合:

| 項目 | 影響 |
|---|---|
| tier2 / tier3 コード | **変更なし** (`k1s0.Workflow.*` API は共通) |
| tier1 内部実装 | Go ファサード内のルーティング先を変更 |
| 実行中のワークフロー | Dapr Workflow 側の実行中インスタンスは完了まで Dapr で処理。新規インスタンスから Temporal にルーティング |
| 移行コスト | tier1 内部で完結。1-2 人日 |

## 5. ルーティングの実装

### 5.1 tier1 Go ファサード内のルーティング

`k1s0.Workflow.Start` が呼ばれた際、Go ファサードはワークフロー定義 (workflowType) に基づいてバックエンドを決定する。

| ルーティング方法 | 概要 |
|---|---|
| **ワークフロー定義ベース** | ワークフロー定義時に `engine: temporal` または `engine: dapr` を指定。tier1 の設定ファイルで管理 |
| **デフォルト** | 明示的な指定がない場合は **Temporal** をデフォルトとする |

### 5.2 設定ファイルの形式

```yaml
# tier1 内部設定 (tier2/tier3 には非公開)
workflows:
  routing:
    default_engine: temporal
    overrides:
      - workflow_type: "notification.send"
        engine: dapr
      - workflow_type: "cache.warmup"
        engine: dapr
      - workflow_type: "healthcheck.aggregate"
        engine: dapr
```

### 5.3 tier2 / tier3 からの透過性

tier2 / tier3 は以下の API のみを使用する。バックエンドの違いは一切意識しない。

```go
// tier2 のコード例 — バックエンドを意識しない
result, err := k1s0.Workflow.Start(ctx, "expense.approval", input)
status, err := k1s0.Workflow.GetStatus(ctx, workflowID)
```

Signal / Query API は Temporal バックエンドでのみ動作する。Dapr Workflow バックエンドのワークフローに対して Signal / Query を呼び出した場合、tier1 は `K1s0WorkflowUnsupportedOperationException` を返す。

## 6. Temporal のスケーリング考慮

### 6.1 PostgreSQL バックエンドの負荷

Temporal は全ワークフローの状態を PostgreSQL に永続化する。ワークフロー数の増加に伴い PostgreSQL の負荷が増加する。

| Phase | 想定ワークフロー数 (月間) | PostgreSQL 負荷 | 対策 |
|---|---|---|---|
| Phase 2 | 100 - 500 | 低 | all-in-one モードで十分 |
| Phase 3 | 500 - 2,000 | 中 | Temporal Server のコンポーネント分離を検討 |
| Phase 5 | 2,000 - 10,000 | 高 | Temporal 専用 PostgreSQL インスタンスの分離を検討 |

### 6.2 パフォーマンスベンチマーク

Phase 2 の導入時に以下のベンチマークを実施し、実測値を記録する。

| 指標 | 計測内容 | 許容値 |
|---|---|---|
| Workflow Start レイテンシ | `client.ExecuteWorkflow` の呼び出しから Workflow の開始まで | < 100 ms (p99) |
| Activity 実行レイテンシ | Activity の dispatch から Worker での実行開始まで | < 50 ms (p99) |
| PostgreSQL クエリ負荷 | Temporal が発行する PostgreSQL クエリの QPS | 現行 QPS の 30% 以内の増加 |
| History サイズ | 1 ワークフローあたりの History Event 数 | < 10,000 events |

### 6.3 History サイズの管理

Temporal のワークフローは実行履歴 (History) を PostgreSQL に蓄積する。長期実行ワークフローでは History が肥大化し、パフォーマンスが劣化する。

| 対策 | 方法 |
|---|---|
| **Continue-As-New** | History が一定サイズ (5,000 events) に達した時点でワークフローを再起動し、History をリセットする。Temporal の標準パターン |
| **Archival** | 完了したワークフローの History を MinIO にアーカイブし、PostgreSQL の容量を管理する。Phase 3 以降で導入 |

## 7. 段階導入計画

| フェーズ | 施策 |
|---|---|
| **Phase 1 (MVP)** | `k1s0.Workflow.*` API のインタフェース定義。スタブ実装 (Start → 即座に Complete を返す) |
| **Phase 2** | Temporal Server デプロイ。振り分けルーティング実装。tier2 サンプルで Temporal ワークフローを 1 本実装。Dapr Workflow の短期処理を 1 本実装。パフォーマンスベンチマーク実施 |
| **Phase 3** | 稟議承認フロー / ユーザー退職処理を Temporal で実装。Temporal Server のコンポーネント分離を検討。History Archival の導入 |
| **Phase 4** | 業務ワークフローの本格展開。振り分け基準のレビューと更新 |

---

## 8. リスクと対処

| リスク | 影響 | 対処 |
|---|---|---|
| 振り分け基準の曖昧さによるバックエンド乱立 | メンテナンス対象が 2 系統に分散 | 本資料の判断フローを厳守。四半期ごとにワークフロー一覧をレビューし、Dapr Workflow → Temporal への統合を検討 |
| Temporal の PostgreSQL 負荷が想定以上 | ワークフローの応答遅延 | Phase 2 でベンチマーク実施。負荷が閾値を超えた場合は Temporal 専用 PostgreSQL インスタンスの分離を前倒し |
| Dapr Workflow の将来的な機能拡張で Temporal との差が縮まる | 2 系統の維持理由が薄れる | Phase 終了時のレトロスペクティブで評価。差が縮まった場合は Temporal への一本化を検討 |
| Temporal Server 障害時の影響 | Saga / 長期ワークフローが停止 | [`../02_アーキテクチャ/02_可用性と信頼性/03_グレースフルデグラデーション.md`](../02_アーキテクチャ/02_可用性と信頼性/03_グレースフルデグラデーション.md) の Workflow 縮退動作に準拠。Temporal の状態永続化により復旧後にリプレイで自動再開 |
