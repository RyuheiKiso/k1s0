# tier1 API 設計原則

## 目的

tier2 / tier3 開発者が tier1 を利用するときに、コードの書き方がばらつくことを防ぐ。「ドキュメントに書いて守らせる」ではなく「**規律はツールで強制する**」を方針とする。

---

## 1. 背景となる懸念

tier1 公開 API (多言語クライアントライブラリ) を提供するだけでは、以下の理由でコードがばらつく。

| 懸念 | 具体例 |
|---|---|
| API の使い方が複数ある | `Log.Info()` と `Logger.Write()` のような重複導線 |
| エラーハンドリングが各人の流儀 | try / catch の有無、リトライ判断の差 |
| 初期化コードがコピペで分岐 | DI 登録・設定読み込みの書き方が散る |
| Dapr / infra を直接呼ぶ抜け道 | ライブラリを迂回されると隠蔽が崩れる |
| オプション引数の使い方 | タイムアウトやリトライ回数を各人が任意指定 |

これらは tier1 の長期保守性と Dapr 隠蔽の前提を破壊する。

---

## 2. 中核原則

> **「規律はドキュメントではなくツールで強制する」**

ドキュメントに「こう書いてください」と書くだけでは守られない。**雛形生成 / CI チェック / Kyverno Admission / analyzer で「違反したコードもリソースも存在できない」状態を作る** ことを唯一の解とする。

---

## 3. 多層防御 (6 段 + オプション)

### ① 雛形生成 CLI で初期状態を均一化

新規サービスは必ず CLI で生成する。CLI は Rust で実装し、テンプレートを `tera` 等で外出しする。生成物には以下が組み込み済み:

- tier1 ライブラリの初期化コード (DI 登録 / 設定読み込み)
- ログ / トレース / メトリクスの計装が完了済み
- Dockerfile / k8s manifest / Dapr annotation
- ヘルスチェックエンドポイント
- 共通 Protobuf 契約 (`src/tier1/contracts/`) への参照
- サンプル業務エンドポイント (削除して書き換える前提)

雛形は対象サービスの種類に応じて複数系統を出力する。

| 雛形種別 | 用途 | 言語 |
|---|---|---|
| tier1 Go サービス雛形 | tier1 内部の Daprファサード系サービス | Go |
| tier1 Rust サービス雛形 | tier1 内部の自作領域 (決定エンジン / 固有機能) | Rust |
| tier2 / tier3 サービス雛形 | 業務ドメインサービス / 個別アプリ | C# / Go / TS 等 |

両 tier1 雛形に共通の規約:

- 構造化ログのフィールド名 / レベル名 / trace_id 注入位置を言語横断で同一にする
- エラーコード体系 (Protobuf enum) を全 tier1 サービスで共有
- `CLAUDE.md` の規約 (1 ファイル 300 行以内 / 日本語コメント必須) を雛形時点で守った状態で出力

→ 「ゼロから書く」を不可能にする。バラつきの大半はこれだけで消える。

### ② API 設計を Opinionated にする (やり方を 1 通りに絞る)

ライブラリ側で「やり方が一つしかない」ようにする。選択肢を意図的に減らす。

設計ガイドライン:

- オプション引数を削る。タイムアウト・リトライは tier1 がポリシー決定する
- 設定はコードからではなく **環境変数 / ConfigMap 固定** にする (tier1 が読む)
- 「便利関数」を作らない。最短経路を 1 本だけ提供する
- メソッド名は業務語彙 (`Audit.Record` / `Workflow.Start`) に寄せる。インフラ語彙 (`PublishToTopic`) を避ける

悪い例・良い例:

| 悪い例 | 良い例 |
|---|---|
| `Log.WriteAsync(level, message, fields, source, timestamp)` (自由度が高すぎる) | `Log.Info("注文を受領", new { orderId, userId })` (型安全・引数固定) |
| `PubSub.PublishWithRetry(topic, msg, retry: 3, timeout: 5s)` (呼び出し側で決めるな) | `PubSub.Publish("order-events", "created", order)` (ポリシーは tier1 が決める) |

### ③ Lint / CI ガードで抜け道を物理的に塞ぐ

CI で禁止ルールを enforce する。違反した瞬間にビルドが落ちる。

| 禁止対象 | 検出方法 |
|---|---|
| `using Dapr.*` (C#) / `import "github.com/dapr/..."` (Go) | Roslyn analyzer / `golangci-lint` で禁止 import チェック |
| `dapr.io/*` annotation を直接書く | k8s manifest の YAML lint |
| `Microsoft.Extensions.Logging` 直接利用 | analyzer で禁止 (`k1s0.Log` のみ許可) |
| Kafka / Redis クライアント直接利用 | 同上 |
| `HttpClient` でのサービス間直接通信 | 同上 (`k1s0.Service.Invoke` 必須) |
| infra コンポーネントの URL ハードコード | 正規表現ベースの YAML / コードスキャン |

### ④ リファレンス実装

「どう書くべきか」を文字で説明するより、**完全に動く模範サービス** を 1 本置く。tier2 開発者は「そのコードを真似る」のが正解になる。

- `examples/tier2-reference-service/` — 全 API の典型的な使い方を網羅
- レビュー時の合言葉: 「reference と書き方が違うのはなぜか」

### ⑤ コードレビューチェックリスト

PR テンプレートに以下を埋め込む。

- tier1 公開 API 以外の import がない
- 雛形生成 CLI の出力を改変していない (初期化部分)
- ログ / エラーハンドリングは reference service と同じ書き方
- k1s0 設定値はコード内にハードコードされていない
- 新しい依存パッケージを追加していない (追加が必要な場合は tier1 チームに相談)

### ⑦ (オプション) 内製 analyzer

更に強化するなら、tier1 チームが各言語向けの analyzer を提供する。

- 「`k1s0.Log.Info` の第 1 引数は文字列リテラルでなければならない」(動的文字列はログメッセージ統一の妨げ)
- 「`catch(Exception)` で握りつぶし禁止」
- 「`async` メソッドに `CancellationToken` がない場合警告」
- 「`k1s0.State.Save` の第 1 引数 (store 名) は定数定義から参照すること」

## 4. 段階的導入

JTC 情シスの体力を踏まえ、以下の順で段階導入する。

| フェーズ | 施策 | 投資コスト | 効果 |
|---|---|---|---|
| MVP-0 | ① 雛形生成 CLI + ④ リファレンス実装 | 中 | バラつきの大半が消える |
| MVP-1b | ⑥ Kyverno Admission ポリシー (PSS + イメージ制御 + ラベル強制) | 低 | k8s レベルで違反リソースの投入自体を拒否 |
| Phase 2 | ② Opinionated API + ③ CI ガード (禁止 import) + ⑥ cosign 署名検証追加 | 中 | ソース + クラスタの両面で抜け道を機械的に防ぐ |
| Phase 3 | ⑤ レビューチェックリスト + ⑦ analyzer | 高 | 微細な逸脱まで矯正 |

---

## 5. 企画書での位置付け

本原則は tier1 が「ライブラリを置くだけ」で終わらないための仕組み。企画書では以下の文脈で参照する。

- Dapr を tier2 / tier3 から隠蔽するという設計判断の実現手段として
- 「言語を尊重する」原則と矛盾しない理由として (各言語のイディオムは保ちつつ、k1s0 利用パターンだけを統一する)
- 長期保守性の根拠として (人が入れ替わってもコードが均質に保たれる)
