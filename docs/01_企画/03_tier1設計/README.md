# 03_tier1設計

k1s0 の存在意義そのものである tier1 — Dapr 隠蔽・Go+Rust ハイブリッド・多言語クライアントライブラリ・雛形生成 CLI — の設計思想を扱う章。tier1 が tier2/tier3 からどれだけの複雑性を引き受けるかが k1s0 全体の開発者体験を決める。本章は「tier1 がサボると tier2 が苦労する」前提で書かれており、引き受ける範囲・引き受けない範囲を明示することを目的にしている。

## 想定読者

tier1 実装を担うシステム基盤チーム（アーキテクト級）、クライアントライブラリ設計者、tier3 向け開発者体験の設計者。業務ロジック実装者（tier2 開発者）は [`14_tier3開発者体験設計.md`](./14_tier3開発者体験設計.md) 周辺のみ参照でよい。

## 前章と後章

- 前章: [`../02_アーキテクチャ/`](../02_アーキテクチャ/) — 本章は tier1 の内部構造。
- 後章: [`../04_技術選定/`](../04_技術選定/) — 本章の決定事項を OSS にマッピングする / [`../05_CICDと配信/`](../05_CICDと配信/) — クライアントライブラリの配布導線

## 04 章との責務境界

本章は「tier1 が何を引き受け、どう設計するか」を書く。**選定した OSS の候補比較・定量根拠・フェーズ別導入計画は [`../04_技術選定/`](../04_技術選定/) の責務であり、本章では扱わない**。例えば「Go + Rust ハイブリッド」という判断そのものは [`02_内部言語ハイブリッド.md`](./02_内部言語ハイブリッド.md) に書くが、Go / Rust の工数・性能比較の数値は [`../04_技術選定/18_言語選定の定量分析.md`](../04_技術選定/18_言語選定の定量分析.md) に置く。読者が「なぜそう決めたか」を数値で詰めたいときは 04 章、「どう設計したか」を詰めたいときは本章、という分業である。この分業が崩れていると感じた箇所は Issue を起こしてほしい。
