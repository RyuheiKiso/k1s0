# デバッグ支援ツール

## 目的

tier1 API の診断機能 ([`08_診断機能設計.md`](08_診断機能設計.md) はコードレベルの観測性を提供する。本資料では、それを補完する**開発環境向けの可視化ツール** (Tilt ダッシュボード) と**運用時のトラブルシューティング手順** (ランブック) を定義する。

---

## 1. Tilt ローカル診断ダッシュボード

### 1.1 背景

ローカル開発環境 (Tilt) で問題が発生した場合、tier2 開発者に Grafana / Loki / Tempo の操作スキルを要求するのは学習コストが高い。Tilt Web UI 内で最低限の診断情報を確認できる拡張を提供する。

### 1.2 提供する機能

| 機能 | 表示内容 | 用途 |
|---|---|---|
| **API コールログ** | tier1 API 呼び出しのリアルタイムストリーム (API 名 / 所要時間 / 成否 / 診断コンテキスト) | 「自分のコードが tier1 を正しく呼べているか」の確認 |
| **PubSub フロー** | イベントの publish / subscribe をサービス間で可視化 | 「イベントが届いていない」問題の即座の特定 |
| **State ブラウザ** | Valkey 内のキーと値をブラウザで閲覧・検索 | State の読み書きが期待通りか確認 |
| **ヘルスステータス** | tier1 バックエンド (Dapr / Valkey / Kafka) の状態一覧 | 「tier1 側に問題があるか」の即座の判定 |
| **Dapr サイドカーログ** | Dapr サイドカーのログを集約表示 | ローカル環境限定で Dapr 内部の動作を確認 |

### 1.3 実装方針

Tilt は Extension 機構 (`load('ext://...', ...)`) によりカスタムリソースを追加できる。以下の方針で実装する。

| コンポーネント | 実装方式 | 備考 |
|---|---|---|
| API コールログ | tier1 ライブラリが stdout に構造化 JSON を出力 → Tilt がストリーム表示 | 追加コンテナ不要 |
| PubSub フロー | tier1 内部に軽量な Admin API を追加 → Tilt Extension が polling | 開発環境限定 |
| State ブラウザ | Valkey の SCAN / GET を Tilt Extension 内で実行 → Web UI に表示 | 読み取り専用 |
| ヘルスステータス | `k1s0.Diag.HealthCheckAsync()` を Tilt Extension が定期呼び出し | Diag API ([`08_診断機能設計.md`](08_診断機能設計.md) 3 章) を利用 |
| Dapr サイドカーログ | `kubectl logs` を Tilt が集約 (Tilt 標準機能) | 設定追加のみ |

### 1.4 雛形生成 CLI との統合

雛形生成 CLI が出力する `Tiltfile` に診断拡張を自動組み込む。tier2 開発者は追加セットアップなしで `tilt up` するだけで診断ダッシュボードが使える状態になる。

```python
# Tiltfile (雛形生成 CLI が自動生成する内容の一部)

# tier1 診断拡張の読み込み
load('ext://k1s0-diag', 'k1s0_diag_dashboard')

# 診断ダッシュボードの有効化
k1s0_diag_dashboard(
    services=['my-service'],
    enable_state_browser=True,
    enable_pubsub_flow=True,
)
```

### 1.5 ローカル環境限定の理由

本ダッシュボードはローカル開発環境 (Tilt) のみで動作する。ステージング・本番では Grafana / Loki / Tempo を使用する。

| 環境 | 診断ツール | 理由 |
|---|---|---|
| ローカル (Tilt) | Tilt 診断ダッシュボード | 学習コストゼロで使える簡易ツール |
| ステージング / 本番 | Grafana + Loki + Tempo | 大量データの検索・長期保存に対応 |

tier2 開発者は L1–L3 の学習段階では Tilt ダッシュボードのみで十分に開発可能。Grafana の習得は L4 以降で段階的に行う ([オンボーディング戦略](../03_開発者体験/07_tier2オンボーディング戦略.md)と連動)。

## 2. トラブルシューティングランブック

### 2.1 概要

「症状 → 原因候補 → 確認方法 → 対処」のパターンで逆引きできるランブック集を整備する。tier1 API のエラーオブジェクトに含まれる `RunbookUrl` ([`08_診断機能設計.md`](08_診断機能設計.md) 4.3 節) からリンクされる。

### 2.2 ランブックの構成

各ランブックは以下の固定構造に従う。

```markdown
# {エラーコード}: {症状の要約}

## 症状
{tier2 開発者が観測する現象}

## 原因候補
### 候補 1: {原因名}
- **確認方法**: {具体的なコマンドまたは API 呼び出し}
- **対処**: {修正手順}
- **担当**: {tier2 / tier1 / infra}

### 候補 2: {原因名}
- **確認方法**: ...
- **対処**: ...
- **担当**: ...

## エスカレーション基準
{この手順で解決しない場合の連絡先と提供すべき情報}
```

### 2.3 初期ランブック一覧

Phase 2 開始時に以下のランブックを整備する。以降はインシデント発生時に追加していく。

| エラーコード | 症状 | 主な原因候補 |
|---|---|---|
| `STATE_GET_TIMEOUT` | State 取得がタイムアウト | Valkey の高負荷 / ネットワーク遅延 / キーサイズ過大 |
| `STATE_SAVE_CONFLICT` | State 保存で楽観ロック競合 | 同一キーへの同時書き込み / リトライによる二重実行 |
| `PUBSUB_PUBLISH_TIMEOUT` | イベント発行がタイムアウト | Kafka ブローカー不調 / メッセージサイズ超過 / シリアライズエラー |
| `PUBSUB_SUBSCRIBE_LAG` | イベント受信が大幅に遅延 | コンシューマーの処理速度不足 / パーティション偏り |
| `WORKFLOW_START_DUPLICATE` | ワークフロー開始が重複エラー | 冪等性キーの重複 / リトライによる再実行 |
| `WORKFLOW_STEP_TIMEOUT` | ワークフローステップがタイムアウト | アクティビティの実行時間超過 / 外部サービス応答遅延 |
| `AUTH_VERIFY_EXPIRED` | トークン検証が期限切れエラー | クライアント側のリフレッシュ不備 / Keycloak の時刻ずれ |
| `AUTH_VERIFY_INVALID` | トークン検証が不正エラー | 署名鍵のローテーション / Realm 設定の不一致 |
| `SERVICE_INVOKE_UNAVAILABLE` | サービス間呼び出しが到達不能 | 宛先サービスの Pod 停止 / Dapr サイドカー未起動 |
| `DIAG_HEALTH_DEGRADED` | ヘルスチェックで縮退検出 | バックエンド (Valkey / Kafka / Temporal) の部分障害 |

### 2.4 ランブックの配置と提供形態

| フェーズ | 配置先 | 理由 |
|---|---|---|
| Phase 2 (Backstage 未導入時) | リポジトリ内 `docs/runbooks/` | Git で管理・レビュー可能 |
| Phase 3 (Backstage 導入後) | Backstage TechDocs | 検索・タグ付け・サービスカタログとの連携 |

両フェーズともランブックは Markdown 形式で記述する。Backstage 移行時はファイル配置を変更するだけで済む。

### 2.5 ランブックの運用ルール

| ルール | 内容 |
|---|---|
| **新規作成のトリガー** | 新エラーコード追加時に tier1 チームがランブックも同時に作成する |
| **更新のトリガー** | インシデント対応で新しい原因候補が判明した場合、ポストモーテムの一環で追記する |
| **レビュー** | ランブックの PR は tier1 + tier2 の両チームがレビューする |
| **鮮度維持** | 四半期に 1 回、全ランブックの有効性を確認する (確認課題はスプリントに組み込む) |

## 3. 段階的導入

| フェーズ | 施策 | 備考 |
|---|---|---|
| Phase 2 開始前 | 初期ランブック 10 件 (2.3 節) を整備 | tier2 開発者参画の前提条件 |
| Phase 2 | Tilt 診断ダッシュボード (API コールログ + ヘルスステータス) を実装 | 最小構成で提供開始 |
| Phase 2 後半 | PubSub フロー + State ブラウザを追加 | Kafka / Valkey 本格運用に合わせる |
| Phase 3 | ランブックを Backstage TechDocs に移行 | Backstage 導入に合わせる |
| Phase 3 | Dapr サイドカーログ集約を追加 | Tilt 標準機能の設定追加のみ |
