# クライアントライブラリ配布戦略

## 目的

tier1 公開 API は多言語クライアントライブラリ (C# / Go / TypeScript) として tier2 / tier3 に配布する。本資料ではライブラリの生成・配布・バージョン更新の仕組みと、tier1 チームの保守負荷を最小化する戦略を定義する。

---

## 1. 基本方針

### 1.1 .proto を Single Source of Truth とする

クライアントライブラリのコードを手書きしない。`src/tier1/contracts/` 配下の `.proto` ファイルを唯一の正とし、全言語のクライアントコードを **`buf generate` で自動生成** する。

| 要素 | 手書き / 自動生成 | 担当 |
|---|---|---|
| `.proto` ファイル | 手書き | tier1 チーム |
| `buf.gen.yaml` (生成設定) | 手書き (初回のみ) | tier1 チーム |
| C# gRPC クライアントコード | **自動生成** | CI |
| Go gRPC クライアントコード | **自動生成** | CI |
| TypeScript gRPC クライアントコード | **自動生成** | CI |
| DI 統合層 (各言語) | 手書き (薄い) | tier1 チーム |
| API リファレンスドキュメント | **自動生成** | CI |

### 1.2 選定の根拠

クライアントライブラリの配布方式として 3 つの選択肢を検討し、Protobuf 自動生成方式を採用した。

| 方式 | 概要 | 不採用の理由 |
|---|---|---|
| 手書きクライアントライブラリ | 各言語のクライアントを個別に手書き | 3 言語分の保守コストが tier1 チームに集中する |
| サイドカー方式 | クライアントライブラリの代わりに Pod 内サイドカーで通信 | Istio + Dapr に加え 3 つ目のサイドカーとなりリソース消費が過大 |
| **Protobuf 自動生成方式** | `.proto` から全言語のコードを自動生成 | — (**採用**) |

---

## 2. クライアントライブラリの構成

### 2.1 自動生成される層と手書きの層

クライアントライブラリは「自動生成コード」と「DI 統合層」の 2 層で構成する。

**自動生成コード** (API の追加・変更のたびに再生成):

- gRPC サービスクライアント
- リクエスト / レスポンスのメッセージ型
- エラーコード enum

**DI 統合層** (手書き、API 追加では原則変更不要):

- `AddK1s0()` 等の DI 登録拡張メソッド
- tier1 サーバーへの接続先解決 (k8s 内 DNS)
- 構造化ログ / OTel 計装の初期設定

DI 統合層は各言語で数十行程度の薄いコードであり、新しい API を `.proto` に追加しても DI 統合層の変更は発生しない。

### 2.2 言語別のパッケージ構成

| 言語 | パッケージ名 | 内容 |
|---|---|---|
| C# | `K1s0.Client` | 自動生成コード + DI 統合層 (NuGet) |
| Go | `go.k1s0.dev/client` | 自動生成コード + 初期化ヘルパー (Go module) |
| TypeScript | `@k1s0/client` | 自動生成コード + 初期化ヘルパー (npm) |

## 3. 自動生成パイプライン

### 3.1 トリガーからパッケージ公開までの流れ

`.proto` ファイルの変更が `main` にマージされたとき、以下の処理が CI で自動実行される。

| ステップ | ツール | 失敗時の動作 |
|---|---|---|
| Protobuf スタイル検証 | `buf lint` | PR をブロック |
| 後方互換性検証 | `buf breaking --against '.git#branch=main'` | PR をブロック |
| コード生成 (3 言語) | `buf generate` | ビルド失敗 |
| 契約テスト実行 | 後述 (3.2) | ビルド失敗 |
| パッケージ公開 | NuGet / Go module / npm push | 手動リトライ |
| tier2 / tier3 への更新 PR 生成 | Renovate | 各チームが確認・マージ |

### 3.2 `buf breaking` による後方互換性の自動検証

`buf breaking` は PR の時点で以下の破壊的変更を自動検出しブロックする。

| 検出対象 | 例 |
|---|---|
| フィールドの削除 | `string key = 2;` の行を削除 |
| フィールド番号の変更 | `string key = 2;` → `string key = 5;` |
| フィールドの型変更 | `string key = 2;` → `int32 key = 2;` |
| RPC の削除 | `rpc Save(...)` の削除 |
| RPC の戻り値型変更 | `returns (SaveResponse)` → `returns (SaveResponseV2)` |

これにより**うっかり互換性を壊す変更が main に入ること自体が不可能**になる。意図的な破壊的変更は [`04_APIバージョニング戦略.md`](./04_APIバージョニング戦略.md) のプロセスに従う。

## 4. バージョン差への対処

### 4.1 前提となる問題

tier2 / tier3 の各サービスがクライアントライブラリを個別に参照するため、バージョンにばらつきが生じる。

| サービス | クライアントバージョン |
|---|---|
| tier2 経費サービス | `K1s0.Client` 1.5.0 |
| tier2 在庫サービス | `K1s0.Client` 1.3.0 |
| tier3 配信ポータル | `K1s0.Client` 1.5.0 |

tier1 サーバーは全バージョンのクライアントからの通信を同時に受け付ける必要がある。

### 4.2 Protobuf のフィールド追加による自然な後方互換

Protobuf3 の仕様上、フィールドの追加は後方互換である。

```protobuf
// v1.3 時点の定義
message SaveRequest {
  string store = 1;
  string key = 2;
  bytes value = 3;
}

// v1.5 で ttl_seconds を追加
message SaveRequest {
  string store = 1;
  string key = 2;
  bytes value = 3;
  int32 ttl_seconds = 4;  // v1.5 で追加
}
```

| クライアント | 送信内容 | サーバーの受信 |
|---|---|---|
| v1.3 (古い) | `store`, `key`, `value` のみ | `ttl_seconds` はゼロ値 (0) |
| v1.5 (新しい) | `store`, `key`, `value`, `ttl_seconds` | 全フィールド |

古いクライアントが新しいフィールドを送らなくても、サーバーはゼロ値として受け取る。**どちらのバージョンでも通信が成立する**。

### 4.3 サーバー側の実装原則: バージョン判定ではなくフィールド存在で分岐

サーバー側はクライアントのバージョンを判定しない。フィールドに値が設定されているかどうかで挙動を分岐する。

| 方針 | サーバー側の実装 |
|---|---|
| **採用**: フィールド存在で分岐 | `if req.TtlSeconds > 0 { useTTL(req.TtlSeconds) }` |
| **不採用**: バージョンで分岐 | `if clientVersion >= "1.5" { ... }` |

この原則により、サーバー側にバージョン別のコードパスが蓄積することを防ぐ。

### 4.4 サポートウィンドウ

全てのバージョンを無期限にサポートすると、サーバー側の互換コードが蓄積し保守性が低下する。以下のサポートウィンドウを設ける。

| バージョン種別 | サポート範囲 | 強制手段 |
|---|---|---|
| MINOR バージョン | 現行 MINOR から **3 バージョン前** まで | Kyverno / CI でデプロイ時に最低バージョンを検証 |
| MAJOR バージョン | 現行 MAJOR から **1 つ前** まで (最大 6 ヶ月) | [`04_APIバージョニング戦略.md`](./04_APIバージョニング戦略.md) 3.2 に従う |

サポート範囲外のクライアントバージョンを含むコンテナイメージは、Kyverno Admission ポリシーでデプロイを拒否する。

## 5. .proto のライフサイクル管理

### 5.1 フィールドの追加

新しいフィールドは常に optional として追加する。古いクライアントからはゼロ値として受信されるため、サーバー側でデフォルト挙動を定義する。

```protobuf
message SaveRequest {
  string store = 1;
  string key = 2;
  bytes value = 3;
  // v1.5 で追加。未指定時はサーバーデフォルト (0 = 無期限)
  int32 ttl_seconds = 4;
}
```

### 5.2 フィールドの非推奨化

削除予定のフィールドには `deprecated` オプションを付与する。自動生成されたクライアントコードに非推奨警告が付与され、tier2 / tier3 開発者に移行を促す。

```protobuf
message SaveRequest {
  string store = 1;
  string key = 2;
  bytes value = 3;
  int32 ttl_seconds = 4;
  // v1.3 で非推奨。v2.0 で削除予定
  string deprecated_format = 5 [deprecated = true];
}
```

### 5.3 フィールドの削除

削除済みのフィールド番号とフィールド名を `reserved` で予約し、将来の再利用を防ぐ。フィールドの削除は MAJOR バージョン変更に該当する。

```protobuf
message SaveRequest {
  string store = 1;
  string key = 2;
  bytes value = 3;
  int32 ttl_seconds = 4;
  // フィールド 5 は削除済み。番号と名前の再利用を禁止
  reserved 5;
  reserved "deprecated_format";
}
```

### 5.4 ライフサイクルのまとめ

| 操作 | SemVer への影響 | `buf breaking` | 手順 |
|---|---|---|---|
| フィールド追加 (optional) | MINOR | 通過 | `.proto` に追加し、サーバー側でデフォルト挙動を実装 |
| フィールド非推奨化 | MINOR | 通過 | `deprecated = true` を付与。最低 1 MINOR バージョン並行 |
| フィールド削除 | **MAJOR** | **ブロック** (`--except` で明示的に許可が必要) | RFC → 移行期間 → MAJOR バージョン |
| メソッド追加 | MINOR | 通過 | `.proto` に `rpc` を追加 |
| メソッド削除 | **MAJOR** | **ブロック** | RFC → 移行期間 → MAJOR バージョン |

## 6. 契約テストの自動化

### 6.1 互換性テストマトリクス

CI で以下の組み合わせを自動テストし、バージョン間の互換性を保証する。

| テスト対象 | 内容 | 実行タイミング |
|---|---|---|
| 現行サーバー × 最新クライアント | 新機能の正常動作確認 | `.proto` 変更時 |
| 現行サーバー × 1 つ前のクライアント | 後方互換性の確認 | `.proto` 変更時 |
| 現行サーバー × 2 つ前のクライアント | サポートウィンドウ内の互換性確認 | `.proto` 変更時 |

### 6.2 テストの実装方針

- 各クライアントバージョンの自動生成コードを Git タグで保持する
- テストは Testcontainers で tier1 サーバーを起動し、各バージョンのクライアントから実際に gRPC 通信する
- テスト失敗は PR をブロックする

## 7. ドキュメントの自動生成

`.proto` ファイルのコメントから API リファレンスを自動生成する。別途ドキュメントを手書きしない。

```protobuf
// 指定されたストアに状態を保存する。
// 同一キーが存在する場合は上書きする。
rpc Save(SaveRequest) returns (SaveResponse);
```

| 生成物 | ツール | 配信先 |
|---|---|---|
| API リファレンス (HTML) | `protoc-gen-doc` / Buf Schema Registry | Backstage プラグイン |
| 変更履歴 | `buf breaking` の差分レポート | CHANGELOG に自動追記 |

## 8. tier1 チームの作業範囲

### 8.1 API 追加時に tier1 チームが行う作業

| 作業 | 手作業 / 自動 | 備考 |
|---|---|---|
| `.proto` ファイルの設計・記述 | 手作業 | メッセージ定義とコメント |
| サーバー側の実装 (Go / Rust) | 手作業 | tier1 の本来業務 |
| 契約テストの追加 | 手作業 | テストケースの定義 |
| 3 言語のクライアントコード生成 | **自動** | `buf generate` |
| 後方互換性の検証 | **自動** | `buf breaking` |
| パッケージの公開 (NuGet / Go module / npm) | **自動** | CI パイプライン |
| tier2 / tier3 への更新 PR 生成 | **自動** | Renovate |
| API リファレンスドキュメント | **自動** | `protoc-gen-doc` |

### 8.2 DI 統合層の保守

DI 統合層は API 追加のたびに変更する必要がない設計とする。

- gRPC チャネルの接続先は k8s 内 DNS で自動解決する (`tier1-state.tier1.svc.cluster.local`)
- 新しいサービスクライアントの DI 登録はリフレクション / ソースジェネレータで自動化する
- DI 統合層の変更が必要になるのは、接続方式の根本的な変更 (gRPC → 別プロトコル等) の場合のみ
