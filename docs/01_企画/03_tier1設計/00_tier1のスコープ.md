# tier1 のスコープ

## 目的

k1s0 の核心である tier1 層が、何を担当し何を担当しないかを明確にする。tier1 の責務は [`../02_アーキテクチャ/01_基礎/01_レイヤ構成と責務.md`](../02_アーキテクチャ/01_基礎/01_レイヤ構成と責務.md) にも概要があるが、本資料では設計判断のレベルで深掘りする。

---

## 1. tier1 の存在意義

tier1 は「システム全体で共通利用する機能」を集中管理することで、以下の目的を達成する。

| 目的 | 効果 |
|---|---|
| tier2 / tier3 からインフラ層の詳細を隠蔽する | Kafka / Loki / state store 等の差し替えを tier1 内に閉じ込められる |
| OSS の利用を tier1 に限定し、ラッパーで隔離する | OSS の有償化・メンテ停止・破壊的変更の影響を tier1 内に閉じ込められる |
| ログ・トレース・監査ログの形式を統一する | 現場チームの実装差異で崩れない |
| 横断的関心事をコピペさせない | 業務ロジックに集中できる |
| Dapr の採用益と抽象化を両立させる | Dapr に縛られずに Dapr の機能を享受する |

---

## 2. tier1 の 3 つの構成要素

tier1 は以下の 3 要素から成る。どれか 1 つでも欠けると tier1 の価値が成立しない。

### 2.1 tier1 公開 API (多言語クライアントライブラリ)

- tier2 / tier3 が利用する **唯一の接点**
- C# / Go / TypeScript 向けに提供
- 薄いラッパーであり、内部では tier1 内部の Go / Rust サービスに gRPC で振り分ける
- Dapr SDK や infra コンポーネントを一切露出させない

### 2.2 tier1 内部実装 (Go + Rust ハイブリッド、不可視)

| 構成要素 | 言語 | 役割 |
|---|---|---|
| Dapr Control Plane + サイドカー | — | building blocks の実行基盤 |
| Go サービス (Daprファサード) | Go | stable Dapr Go SDK を直接利用 |
| Rust サービス (自作領域) | Rust | ZEN Engine / JTC 固有機能 / 雛形 CLI |

詳細は [`02_内部言語ハイブリッド.md`](./02_内部言語ハイブリッド.md) を参照。

### 2.3 雛形生成 CLI

- 新規サービスは必ず CLI で生成する
- 生成物には tier1 ライブラリ初期化・Dockerfile・k8s manifest・Dapr annotation・ヘルスチェック・OTel 計装が組み込み済み
- Rust で実装し、テンプレートは `tera` 等で外出しする
- Go / Rust 両方のサービス雛形を出力できる

雛形生成 CLI の役割は [`04_雛形生成CLI.md`](./04_雛形生成CLI.md) で詳しく扱う予定 (現時点では API 設計原則の章に内包)。

---

## 3. tier1 が **扱わない** こと

境界を明確にするため、扱わないものを明示する。

| 扱わないもの | 理由 |
|---|---|
| 業務ドメインロジック | tier2 の責務 |
| エンドユーザー UI | tier3 の責務 |
| k8s クラスタ管理 | infra チームの責務 |
| GitHub Actions のワークフロー定義の内製 | `CI_CD_パイプライン構成.md` の雛形に従う |
| 業務ルールの詳細 | ZEN Engine の JDM ファイルで業務担当が編集 |

---

## 4. tier1 が解決する 3 つの技術課題

tier1 を自前実装することの合理性は、以下の 3 点に限定される。

### 4.1 インフラ層アクセスの統一

- Dapr を基本、足りない箇所を tier1 自前実装で補完
- 詳細: [`01_Dapr隠蔽方針.md`](./01_Dapr隠蔽方針.md)

### 4.2 ログ標準化

- tier1 が各言語向けに構造化ログライブラリを提供
- アプリ内部のログ形式・フィールド・集約先を統一する
- Dapr 単体では対応できない領域 (tier1 の存在意義が最も明確)

### 4.3 OpenTelemetry 標準化

- Dapr サイドカーはサービス境界のトレースを自動計装するが、アプリ内部の span 生成は対象外
- tier1 が各言語向けに OTel SDK をラップして提供し、アプリ内部計装を統一する
- Dapr と tier1 の両方が必要な補完関係

### 4.4 Feature Flag による段階的ロールアウト

- 新機能の有効化 / 無効化をコードデプロイと分離し、部署単位・ユーザー割合での段階的ロールアウトを可能にする
- 内部では flagd (CNCF) に gRPC で接続するが、tier2 / tier3 は `k1s0.Feature` API のみを利用し、flagd の存在を意識しない
- レガシー共存 (Phase 4) において、旧サービスと新 tier2 サービスの切替を部署単位で制御する「制御弁」として機能する
- 障害時は参照系 API と同じ cache-fallback パターンを適用する

---

## 5. Dapr と tier1 の並立構成

tier1 は「Dapr を採用する」か「自前実装する」かの二者択一ではなく、**並立構成** を取る。

- **Dapr を基本**: state / pub-sub / secrets / workflow / bindings は Dapr を使う
- **自前実装は補完**: ログ標準化 / OTel アプリ内部計装 / JTC 固有機能 / 決定エンジンに限定
- **tier2 / tier3 は Dapr を意識しない**: tier1 公開 API のみを使う

詳細な判断基準と機能マッピングは [`01_Dapr隠蔽方針.md`](./01_Dapr隠蔽方針.md) を参照。

---

## 6. なぜ tier1 を作るのか (企画書での答弁案)

決裁者から「tier1 を作らず Dapr をそのまま使わせれば良いのでは」と問われた際の回答構成:

1. **Dapr は採用する**。ただし tier1 の内部実装として組み込み、tier2 / tier3 からは隠蔽する
2. tier2 / tier3 開発者は Dapr を一切意識しない。tier1 公開 API のみを使う
3. これにより Dapr のバージョン更新・破壊的変更・差し替えの影響を tier1 内に閉じ込められる
4. JTC 固有要件 (ログ標準化 / 監査改ざん防止 / 個人情報マスキング) は Dapr 単体では対応できない。tier1 で補完する
5. 結果として「Dapr の機能的恩恵」と「Dapr に縛られない長期保守性」を同時に達成できる
