# 診断機能設計

## 目的

tier1 が Dapr / Kafka / Valkey を隠蔽する設計は、正常時の開発体験を向上させる一方で、障害時に「内部で何が起きたか」を tier2 開発者が確認する手段を失わせる。本資料では**抽象化を維持したまま観測性を確保する**ための 3 つの機能を定義する。

- **診断コンテキスト** — 全 API レスポンスに内部動作の情報を付与する
- **k1s0.Diag API** — tier1 の内部状態を能動的に照会する専用 API
- **エラー時の自動切り分けガイド** — 例外オブジェクトに「次に何を調べるべきか」を含める

設計原則: tier2 開発者にインフラコンポーネントを**操作させない**が、**観測はできる**ようにする。

---

## 1. 背景となる課題

tier1 公開 API の隠蔽レベルが高いことにより、tier2 開発者は以下の状況で問題の切り分けができない。

| 状況 | tier2 開発者が直面する困難 |
|---|---|
| API 呼び出しが遅い | キャッシュヒットなのかバックエンド直アクセスなのか不明 |
| イベントが届かない | Kafka のコンシューマーラグか、パーティション割り当てか、tier1 内部のエラーか不明 |
| State 保存が失敗 | Valkey の障害か、ネットワークか、リトライ上限超過か不明 |
| エラーが返された | 問題が tier1 内部なのか自サービスのコードなのか切り分け不能 |

---

## 2. 診断コンテキスト (全 API レスポンスへの付与)

### 2.1 概要

tier1 公開 API の全レスポンスに `Diagnostics` プロパティを付与する。正常系・異常系を問わず、tier1 内部で何が起きたかを構造化データで返す。

### 2.2 レスポンス構造

```csharp
// State API の例
var result = await k1s0.State.GetAsync<Order>("orders", orderId);

result.Value;          // Order オブジェクト (従来通り)
result.Diagnostics;    // 診断コンテキスト
```

`Diagnostics` オブジェクトの共通フィールド:

| フィールド | 型 | 説明 |
|---|---|---|
| `Source` | string | データ取得元 (`"cache-hit"` / `"backend"` / `"fallback"`) |
| `LatencyMs` | int | tier1 内部処理時間 (ms) |
| `Retries` | int | 実行されたリトライ回数 |
| `BackendType` | string | 使用されたバックエンド (`"valkey"` / `"kafka"` / `"temporal"`) |
| `Degraded` | bool | 縮退モードで動作したか |
| `CorrelationId` | string | 分散トレース用 ID (Grafana Tempo で検索可能) |

### 2.3 API 種別ごとの追加フィールド

| API 種別 | 追加フィールド | 説明 |
|---|---|---|
| `State` | `CacheTtlRemaining` | キャッシュの残存 TTL (秒) |
| `PubSub` | `Partition`, `Offset` | メッセージが書き込まれた Kafka パーティションとオフセット |
| `Workflow` | `WorkflowStatus`, `CurrentStep` | ワークフローの現在状態とステップ |
| `Feature` | `EvaluationReason` | フラグ評価の根拠 (`"targeting-match"` / `"default-value"`) |

### 2.4 環境別の動作

| 環境 | Diagnostics の動作 | 理由 |
|---|---|---|
| ローカル (Tilt) | **全フィールド有効** | 開発時のデバッグに必要 |
| ステージング | **全フィールド有効** | 結合テスト・負荷テスト時の問題調査に必要 |
| 本番 | **`CorrelationId` のみ** | パフォーマンスへの影響回避。詳細はログに記録済み |

本番環境で詳細な診断情報が必要な場合は、`CorrelationId` を使って Grafana Loki / Tempo でログ・トレースを確認する。

### 2.5 Protobuf 定義

```protobuf
message Diagnostics {
  string source = 1;
  int32 latency_ms = 2;
  int32 retries = 3;
  string backend_type = 4;
  bool degraded = 5;
  string correlation_id = 6;
  // API 種別固有の追加フィールド
  map<string, string> extra = 7;
}
```

`extra` フィールドを `map<string, string>` とすることで、API 種別固有のフィールドを Protobuf 定義の変更なく追加できる。

## 3. k1s0.Diag 診断 API

### 3.1 概要

tier1 の内部状態を能動的に照会するための専用 API を新設する。通常の API 呼び出しに付随する診断コンテキスト (2 章) とは異なり、**特定の問題を調査するときに明示的に使う**ものである。

### 3.2 ヘルスチェック

tier1 が管理する全バックエンドの健全性を一括確認する。

```csharp
var health = await k1s0.Diag.HealthCheckAsync();
```

レスポンス:

| フィールド | 値の例 | 説明 |
|---|---|---|
| `State.Status` | `"healthy"` / `"degraded"` / `"unavailable"` | Valkey の状態 |
| `State.LatencyMs` | `3` | 直近のヘルスチェック応答時間 |
| `PubSub.Status` | `"degraded"` | Kafka の状態 |
| `PubSub.ConsumerLag` | `1523` | 未処理メッセージ数 |
| `Workflow.Status` | `"healthy"` | Temporal の状態 |
| `Workflow.ActiveCount` | `12` | 実行中ワークフロー数 |
| `Auth.Status` | `"healthy"` | Keycloak の状態 |

tier2 開発者は「自分の API が遅い → tier1 バックエンドの問題か自分のコードの問題か」を即座に切り分けられる。

### 3.3 イベント追跡

特定のイベントの配信状態を追跡する。「イベントを publish したが、購読者に届いていない」場合の調査に使う。

```csharp
var trace = await k1s0.Diag.TraceEventAsync("order-events", eventId);
```

レスポンス:

| フィールド | 値の例 | 説明 |
|---|---|---|
| `Published` | `"2026-04-13T10:00:00Z"` | 発行日時 |
| `Partition` | `3` | Kafka パーティション |
| `Delivered` | `true` | 少なくとも 1 購読者に配信済みか |
| `Consumers[].Service` | `"inventory-service"` | 購読サービス名 |
| `Consumers[].Acked` | `false` | 処理完了したか |
| `Consumers[].Reason` | `"timeout"` | 未完了の理由 (Acked=false 時) |

### 3.4 ワークフロー状態照会

実行中のワークフローの進行状況を確認する。

```csharp
var status = await k1s0.Diag.GetWorkflowStatusAsync(workflowId);
```

レスポンス:

| フィールド | 値の例 | 説明 |
|---|---|---|
| `Status` | `"running"` / `"completed"` / `"failed"` / `"compensating"` | 現在状態 |
| `CurrentStep` | `"payment-authorization"` | 実行中のステップ名 |
| `CompletedSteps` | `["order-validation", "inventory-check"]` | 完了済みステップ |
| `FailedStep` | `null` | 失敗したステップ (失敗時のみ) |
| `FailureReason` | `null` | 失敗理由 |

### 3.5 アクセス制御

| 環境 | アクセス権 | 備考 |
|---|---|---|
| ローカル (Tilt) | 全機能制限なし | 開発時 |
| ステージング | 全機能制限なし | テスト時 |
| 本番 | **読み取り専用 + RBAC** | `k1s0-diag-reader` ロールが必要 |

Diag API は純粋な照会 API であり、状態の変更や操作は一切できない。

## 4. エラー時の自動切り分けガイド

### 4.1 概要

tier1 API がエラーを返す際、標準的な例外に加えて**次に何を調べるべきか**を構造化して含める。エラーオブジェクトの `RunbookUrl` フィールドは [`09_デバッグ支援ツール.md`](./09_デバッグ支援ツール.md) で定義するランブックにリンクされる。

### 4.2 エラーオブジェクトの構造

```csharp
try
{
    await k1s0.PubSub.PublishAsync("order-events", "created", order);
}
catch (K1s0Exception ex)
{
    ex.Code;           // "PUBSUB_PUBLISH_TIMEOUT"
    ex.Layer;          // "tier1" — 問題の発生レイヤー
    ex.CorrelationId;  // "abc-123" — Grafana Tempo 検索用
    ex.RunbookUrl;     // トラブルシューティング手順への URL
    ex.Suggestion;     // 次に取るべきアクションの自然言語説明
}
```

### 4.3 フィールド定義

| フィールド | 型 | 説明 |
|---|---|---|
| `Code` | string | 機械可読なエラーコード。`{API}_{OPERATION}_{CAUSE}` 形式 |
| `Layer` | string | 問題の発生レイヤー (`"tier1"` / `"tier2"` / `"infra"`) |
| `CorrelationId` | string | 分散トレース用 ID。Grafana Tempo でこの値を検索すると関連スパンが表示される |
| `RunbookUrl` | string | 対応するトラブルシューティングランブックの URL |
| `Suggestion` | string | 次に取るべきアクションの自然言語説明 |

### 4.4 Layer 判定ロジック

`Layer` フィールドにより「誰の問題か」を即座に切り分ける。

| Layer 値 | 意味 | tier2 開発者の対応 |
|---|---|---|
| `"tier2"` | リクエスト内容の不備 (バリデーション失敗等) | 自サービスのコードを修正 |
| `"tier1"` | tier1 内部の処理失敗 (リトライ上限超過等) | `k1s0.Diag.HealthCheckAsync()` で状態確認後、tier1 チームにエスカレーション |
| `"infra"` | インフラ層の障害 (Kafka / Valkey 停止等) | インフラチームにエスカレーション。tier1 の縮退動作で一部機能は継続 |

### 4.5 エラーコード体系

```
{API}_{OPERATION}_{CAUSE}

例:
STATE_GET_TIMEOUT         — State.Get でタイムアウト
STATE_SAVE_CONFLICT       — State.Save で楽観ロック競合
PUBSUB_PUBLISH_TIMEOUT    — PubSub.Publish でタイムアウト
PUBSUB_PUBLISH_OVERSIZE   — メッセージサイズ上限超過
WORKFLOW_START_DUPLICATE   — 同一ワークフロー ID の重複開始
AUTH_VERIFY_EXPIRED        — トークン有効期限切れ
```

エラーコードは Protobuf enum として `src/tier1/contracts/` に定義し、全言語で共有する。

### 4.6 Suggestion の自動生成

`Suggestion` フィールドの内容はエラーコードとランタイム情報から自動生成する。

| エラーコード | Suggestion 例 |
|---|---|
| `STATE_GET_TIMEOUT` | `"k1s0.Diag.HealthCheckAsync() で State の状態を確認してください (CorrelationId: abc-123)"` |
| `PUBSUB_PUBLISH_OVERSIZE` | `"メッセージサイズが 1.2MB です (上限: 1MB)。ペイロードを分割するか、State に保存してキーのみを publish してください"` |
| `AUTH_VERIFY_EXPIRED` | `"トークンの有効期限が切れています。クライアント側でトークンリフレッシュを確認してください"` |

## 5. 段階的導入

| フェーズ | 施策 | 備考 |
|---|---|---|
| Phase 2 | 診断コンテキスト (2 章) を全 API レスポンスに追加 | tier2 開発開始の前提条件 |
| Phase 2 | エラー切り分けガイド (4 章) を全例外に適用 | 同上 |
| Phase 2 | `k1s0.Diag.HealthCheckAsync()` を実装 | 最低限の切り分け手段 |
| Phase 2 後半 | `k1s0.Diag.TraceEventAsync()` を実装 | Kafka 本格運用に合わせる |
| Phase 3 | `k1s0.Diag.GetWorkflowStatusAsync()` を実装 | Temporal 本格運用に合わせる |
| Phase 3 | 本番環境の RBAC 制限を適用 | チーム拡大に合わせてアクセス制御を強化 |
