# 内製知財戦略

## 目的

k1s0 プラットフォーム自体 (tier1 公開 API / 雛形生成 CLI / Dapr Component バリデータ / Rust 自作領域 / tier1 Go ファサード / アプリ配信ポータル / ドメインロジック) が生み出す **JTC 自社の知的財産** について、以下を定義する。

- k1s0 プラットフォームに適用するライセンス方針
- 知財の所有権と利用許諾の範囲
- グループ会社間 / 委託業者との取り扱い
- 将来の外部公開 (OSS 化 / コンサルティング活用) の余地

本資料が埋める空白は、[`00_OSS法務対応.md`](./00_OSS法務対応.md) が「採用 OSS 側」の法務を扱うのに対し、「k1s0 自作コード側」の知財取り扱いが未定義であった点である。稟議段階では「k1s0 は社内資産か、OSS 化するのか、関連会社に配布するのか」という問いに答える必要がある。

---

## 1. k1s0 の知財範囲

k1s0 が生成する知的財産を以下の 6 カテゴリに分類する。カテゴリごとに扱いが異なる。

| カテゴリ | 具体例 | 特徴 |
|---|---|---|
| (A) 基盤コード | tier1 公開 API 仕様 (`k1s0.Log` / `k1s0.State` 等)、Go ファサード実装、Rust 自作領域実装 | k1s0 の核。他社にも通用する汎用性 |
| (B) 設定・マニフェスト | Kustomize / Helm values / OpenTofu / Kyverno ポリシー | 環境固有の情報を含むが、抽象化すれば汎用化可 |
| (C) 開発者ツール | 雛形生成 CLI (`k1s0-scaffold`)、デバッグツール | 他社 JTC にも横展開可能 |
| (D) ドメインロジック | 申請ワークフロー JDM、権限判定 JDM、アプリ配信ポータル業務ロジック | JTC 固有の業務ルール・業務フロー |
| (E) ドキュメント | 本企画書配下の全 md / drawio 図 | 思想・設計・選定根拠 |
| (F) 運用ノウハウ | Runbook、障害対応履歴、SLO 達成記録 | 運用を通じて蓄積される暗黙知 |

## 2. カテゴリ別のライセンス方針

### 2.1 (A) 基盤コード — **社内独占 + 将来 OSS 化の余地**

| 観点 | 方針 |
|---|---|
| 初期ライセンス | **Proprietary (JTC 社内独占)** |
| 将来の外部公開 | Phase 5 以降で Apache 2.0 への OSS 化を選択肢として検討 |
| グループ会社での利用 | 無償でグループ内利用可 (関連会社契約の範囲内) |
| 外部委託業者の扱い | 開発委託契約内でコード利用可、委託終了後は返却 / 削除義務 |

基盤コードを初期段階で OSS 化しない理由:

1. **初期は JTC の競争優位性**: k1s0 が稼働して業務改革を実現する期間 (Phase 0〜4) は社内資産として扱う
2. **品質の担保**: Phase 5 以降の安定稼働を待たずに OSS 化すると、品質が低い状態で外部に晒される
3. **サポート体制**: OSS 化するとコミュニティからの Issue 対応・PR レビュー負担が発生する。情シスチーム 10 名規模では対応不可能

OSS 化を検討するタイミング:

- Phase 5 (全社ロールアウト) 完了後の運用が 1 年以上安定
- チーム体制が 20 名以上に成長
- 業界内で他社 JTC からの利用要望が複数件寄せられる
- JTC 経営層が外部プレゼンス向上の戦略的価値を認定

### 2.2 (B) 設定・マニフェスト — **社内独占 + サンプルの OSS 化検討**

| 観点 | 方針 |
|---|---|
| 初期ライセンス | Proprietary (JTC 社内独占) |
| サンプル版の外部公開 | Phase 3 以降で「リファレンス実装」として MIT で公開検討 |
| 本番 values / Secret | **絶対に外部公開しない** (Sealed Secrets / OpenBao で管理) |

本番環境の設定は JTC の業務情報を含むため、外部公開しない。ただし、業界標準のリファレンスとしての value 構造を示す「サンプル values」は Phase 3 以降で外部公開を検討可能。

### 2.3 (C) 開発者ツール — **社内独占 + 積極的な OSS 化検討**

| 観点 | 方針 |
|---|---|
| 初期ライセンス | Proprietary (JTC 社内独占) |
| 将来の外部公開 | Phase 2 完了後 Apache 2.0 で OSS 化を積極検討 |

雛形生成 CLI (`k1s0-scaffold`) やデバッグツールは、他社 JTC が類似の基盤を構築する際に直接利用可能。OSS 化することで「JTC 向けプラットフォーム」のデファクトとなり、コミュニティ拡大・人材獲得で JTC にメリットが生じる。OSS 化の優先度は (A) 基盤コードより高い。

### 2.4 (D) ドメインロジック — **社内独占の維持**

| 観点 | 方針 |
|---|---|
| ライセンス | **Proprietary (JTC 社内独占)**、永続的 |
| グループ会社での利用 | 関連会社契約の範囲内で共有可 |
| 外部公開 | 原則なし |

申請ワークフロー / 権限ポリシー / 業務アプリ配信ポータルの業務ロジックは、JTC の内部業務慣習・組織階層・稟議ルールを反映するため、外部には意味をなさない。外部公開することでむしろ組織構造を類推されるリスクがあるため、永続的に社内独占とする。

### 2.5 (E) ドキュメント — **社内独占 + 抽象版の外部公開検討**

| 観点 | 方針 |
|---|---|
| 初期ライセンス | Proprietary (JTC 社内独占) |
| 将来の外部公開 | Phase 5 以降で CC BY-SA 4.0 での段階公開を検討 |

設計思想・選定根拠は、他社 JTC にとっても参考価値が高い。企画書の核心部分は JTC 固有の事情 (ペルソナ / TCO / 組織体制) を含むが、技術選定方針や tier 構造の考え方は抽象化すれば外部公開可能。

### 2.6 (F) 運用ノウハウ — **社内独占の維持**

| 観点 | 方針 |
|---|---|
| ライセンス | Proprietary (JTC 社内独占)、永続的 |
| 文書化 | Runbook として社内 Wiki で管理 |

障害対応の具体的な手順や SLO 達成ノウハウは、JTC の運用体制と密接に結びつく。外部公開する意義が薄く、かつセキュリティインシデントの詳細を含むため、永続的に社内独占とする。

## 3. 知財所有権の帰属

### 3.1 開発体制と所有権の原則

| 開発者種別 | コードの所有権 | 根拠 |
|---|---|---|
| JTC 社員 (雇用契約下) | **JTC 法人が所有** | 職務著作 (著作権法 15 条) |
| 協力会社 SES (派遣) | **JTC 法人が所有** (派遣契約で譲渡を明記) | SES 契約条項で指定 |
| 業務委託 (準委任 / 請負) | **JTC 法人が所有** (業務委託契約で譲渡を明記) | 業務委託契約条項で指定 |
| 複数 JTC グループ会社 | 複雑 (次節で詳述) | グループ会社間契約の個別合意 |

### 3.2 業務委託契約に必須の条項

OSS ミックス構成のプラットフォーム開発を外部委託する場合、契約書に以下を明記する。

- **著作権譲渡**: 成果物の著作権は JTC に譲渡する旨を明記 (著作者人格権不行使特約を含む)
- **派生物の取り扱い**: 成果物から派生するコードの扱い
- **OSS 利用の義務**: [`00_OSS法務対応.md`](./00_OSS法務対応.md) の allow list 以外の OSS を無断採用しない
- **秘密保持**: 業務ロジック / 業務情報 / 認証情報の第三者開示禁止
- **業務終了後の返却 / 削除**: コード・ドキュメント・認証情報の返却または削除義務

### 3.3 グループ会社間の取り扱い

JTC の多くは複数のグループ会社で構成される。k1s0 をグループ会社で横展開する際は以下を整理する。

| シナリオ | 推奨対応 |
|---|---|
| 親会社で開発 → 子会社で利用 | 親会社が所有権を保持、子会社へはライセンス許諾 |
| 複数グループ会社で共同開発 | 開発の主担当会社が所有権、他社はライセンス許諾 |
| グループ外の取引先 (サプライヤー) での利用 | 原則として不可。必要時は個別契約で限定 |

ライセンス許諾の範囲はグループ会社間の具体的な契約条項で定める。本企画の段階ではフレームワークのみ示し、具体的な契約は Phase 1c 完了時点で法務部門と詳細設計する。

## 4. AGPL 採用 OSS との構造的分離

[`00_OSS法務対応.md`](./00_OSS法務対応.md) セクション 2.4 で示した通り、AGPL OSS (Grafana / Tempo / Loki / MinIO / Pyroscope / Renovate) と k1s0 自作コードは **プロセス分離** により法的に切り離されている。これは知財戦略として以下の意味を持つ。

1. k1s0 自作コード (Proprietary) を将来 OSS 化する際、AGPL の copyleft 条項に引きずられない
2. 業務委託契約で成果物の著作権を JTC が受け取る際、AGPL OSS の義務が JTC に継承されない
3. グループ会社への配布時、AGPL OSS は別途各社でインストール運用する前提となり、k1s0 本体の配布物に AGPL ライセンスが混入しない

この構造を維持することは知財戦略の前提条件であり、[`00_OSS法務対応.md`](./00_OSS法務対応.md) セクション 2.4 の 3 原則 (プロセス分離 / ライブラリリンクなし / ABI 分離) を守ることが必須となる。

## 5. 商標とブランディング

### 5.1 "k1s0" の商標化

プラットフォーム名 "k1s0" の商標出願の是非は以下で判断する。

| シナリオ | 推奨対応 |
|---|---|
| 社内利用のみ、外部公開なし | 商標出願不要 |
| 将来 OSS 化を視野に入れる | **Phase 3 完了時点で商標出願を検討** |
| 外部へのコンサルティング活用 | 商標出願必須 |

現段階 (Phase 0) は商標出願不要。Phase 3 完了時点で再評価する。

### 5.2 ロゴ・ブランディング資産

ロゴデザインや UI コンポーネントの意匠は、作成時に以下を明記する。

- 作成者 (デザイナー) の帰属
- JTC への著作権譲渡
- 派生物作成の許諾範囲

これらを契約書で明記しない場合、ロゴを利用した社外プレゼン・広報資料で予期せぬ紛争が発生する可能性がある。

## 6. 特許とリスク管理

### 6.1 k1s0 自体の特許出願

k1s0 のアーキテクチャ / tier 構造 / Dapr 隠蔽方式 等は、JTC 固有の工夫を含む可能性がある。ただし以下の理由で特許出願を積極推進しない。

- **既存技術の組み合わせが主**: k8s + Istio + Dapr の組み合わせは業界で既知であり、新規性が低い
- **出願コストと回収見込みの乖離**: 特許出願には国内 30〜50 万円、国際出願では 200〜500 万円がかかる。JTC 内部利用のみで回収は困難
- **OSS 化との不整合**: 将来 OSS 化する場合、特許を持っていると「特許無償提供」の明示が必要となり手続きが複雑化

ただし、ZEN Engine と連携した業務ルール評価の独自手法や、監査改ざん防止の独自プロトコルで特別な新規性が生まれた場合は個別検討する。

### 6.2 第三者特許の侵害リスク

k1s0 が採用する OSS (Dapr / Kubernetes / Kafka 等) は、いずれも CNCF 等の財団ガバナンス下で運営され、特許プールへの参加・特許非主張 (Patent Non-Assertion) のコミットメントがある。このため、OSS 由来の機能で第三者特許侵害を受けるリスクは構造的に低い。

k1s0 自作コード (tier1 自作領域 / 雛形 CLI 等) は、既存の特許データベースでのサーチを Phase 3 完了時点で一度実施し、重大な侵害リスクがないことを確認する。

## 7. 公表・プレゼンテーションの扱い

### 7.1 社内プレゼンテーション

企画書本文 (`企画書.md`) および配下ドキュメントは、JTC 内部の稟議・プロジェクト説明会・開発者オンボーディングで無制限に利用可能。

### 7.2 社外プレゼンテーション

技術カンファレンス (JANOG / CloudNative Days / Japan Container Days / Go Conference 等) での登壇でのプレゼンは、以下の原則で判断する。

| 公開対象 | 判断 |
|---|---|
| 技術選定の考え方 (Dapr 隠蔽 / tier 構造) | **原則公開可** (業界貢献として推奨) |
| 具体的な実装詳細 (コードレベル) | Phase 2 完了後に条件付き公開可 |
| 業務ロジック (JDM 内容) | **公開不可** (企業秘密) |
| TCO 試算の絶対額 | **公開不可** (企業財務情報) |
| TCO 試算の比率 (20% 削減) | 公開可 |
| 採用 OSS リスト | 公開可 |
| 組織体制・人員配置 | 公開不可 |

### 7.3 論文・書籍執筆

プロジェクトメンバーが論文 / 書籍を執筆する場合、以下を事前に確認する。

- 公開対象が上記セクション 7.2 の基準に合致するか
- JTC 広報部門の承認が必要か
- 共著者に協力会社 / 業務委託が含まれる場合の権利関係

## 8. 5 年タイムラインでの知財戦略サマリ

| Phase | 知財アクション |
|---|---|
| Phase 0〜1 (2026) | 業務委託契約の雛形整備 + OSS 採用ライセンス監査 CI 構築 |
| Phase 2 (2027 前半) | 雛形生成 CLI の OSS 化 RFC 起案、他社 JTC での試用打診 |
| Phase 3 (2027 後半) | 商標 "k1s0" の出願検討、特許サーチ実施、外部カンファレンス発表開始 |
| Phase 4 (2028) | tier1 公開 API の OSS 化検討 (Apache 2.0 予定) |
| Phase 5 (2029 以降) | 基盤コード全体の OSS 化検討 + コミュニティ形成 |

各 Phase のアクションは、当該 Phase の完了時点で経営層 + 法務部門 + tier1 チームで再評価する。

---

## 9. 本資料の限界

- 本資料は JTC 情シスが法務部門・経営層と協議するための論点整理資料であり、法的助言ではない
- 業務委託契約・グループ会社契約の具体的な条項は、各 JTC の法務部門 / 契約雛形で個別設計する必要がある
- 知財戦略は JTC の経営戦略・IR 戦略とも連動するため、経営層の方針が優先する
- 特許・商標の実務は専門家 (弁理士) への相談が必須
