# 知財帰属と業務委託

## 目的

本資料は、稟議で問われる「k1s0 は誰の知財か」に対し、職務著作の原則・外部開発者の扱い・OSS 化の将来オプション・特許商標の扱いを整理する。[`README.md`](./README.md) 第 1.3 節の即答を、契約条項の雛形と将来オプションまで含めて展開する。

---

## 1. 職務著作の原則

起案者が業務として作成する k1s0 の全成果物は、**著作権法 15 条 (職務著作) により JTC 法人に帰属** する。これは OSS 採用 / 非採用に関係なく成立する原則であり、起案者個人が著作権を留保する構造はとらない。この点は **稟議承認以前の既定事項** であり、個別契約は不要。

したがって稟議の場で「起案者が辞めたら著作権を持って行かれるのか」という問いは前提が誤っている。職務著作の発動条件 (法人の発意・業務として作成・法人名義で公表・契約就業規則に別段の定めがない) はすべて満たされる運用を前提とする。

---

## 2. 外部開発者の扱い

業務委託・SES・グループ会社から開発者が参画する場合は、各契約雛形で以下を標準化する。

- 成果物の著作権を JTC へ譲渡する条項 (著作者人格権不行使特約を含む)
- 成果物の OSS ライセンスを allow list に限定する条項
- 秘密保持義務 (業務ロジック・認証情報の第三者開示禁止)
- 業務終了後のコード・ドキュメント・認証情報の返却・削除義務

契約雛形の整備は Phase 0〜1 の法務部門タスクとして事前着手する。詳細条項は [`../../02_構想設計/05_法務とコンプライアンス/01_内製知財戦略.md`](../../02_構想設計/05_法務とコンプライアンス/01_内製知財戦略.md) 第 3.2 節を参照。

稟議の場で「委託先との契約が甘いと後で揉めるのでは」と問われた際には、本節の 4 項目を雛形として整備済みと即答できる。

---

## 3. OSS 化の将来オプション

稟議の場で「職務著作で法人に帰属する知財を、将来 OSS として公開する選択肢はあるのか」「競業他社に使われないか」と問われた際、**選択肢を明示した上で判断は Phase 5 以降に保留する** のが k1s0 の方針である。選択肢を曖昧にすると「戦略なく公開して後悔するリスク」と「選択肢を潰して横展開機会を失うリスク」の両方を取りに行くことになる。したがって本節では 3 シナリオを構造的に比較し、Phase ごとの判断軸を明確にする。詳細な契約条項と判断フレームは [`../../02_構想設計/05_法務とコンプライアンス/01_内製知財戦略.md`](../../02_構想設計/05_法務とコンプライアンス/01_内製知財戦略.md) 第 2 章を参照。

### 3.1 3 つの OSS 化シナリオ

k1s0 自作コード (雛形生成 CLI / tier1 Rust 自作領域 / アプリ配信ポータル) の知財活用方法は、大別して以下 3 シナリオに整理できる。現時点 (Phase 0) では **シナリオ ① 社内専有を既定値とし、Phase 5 以降でシナリオ ② ③ を再評価する** 。

#### シナリオ ① 社内専有 (既定値、Phase 0〜4 の方針)

全コードを社内独占とし、法人帰属のまま外部公開しない。競合他社に使われるリスクはゼロだが、OSS コミュニティからのコントリビュート獲得・採用ブランディングの機会も失う。

- **経営インパクト**: ポジティブ面は知財の独占、ネガティブ面は「社内 OSS だけでは外部エンジニアの採用訴求力が弱い」。競合優位は認められるが、5 年以降の陳腐化リスクはある
- **技術インパクト**: OSS コミュニティからの PR がないため、バグ発見と機能追加は全て自社リソースに依存。CVE 対応も自力
- **法務インパクト**: 最もシンプル。社内配布のみのため AGPL OSS の公開義務も発動しない。GDPR / マイナンバー法の答弁も社内完結

#### シナリオ ② Apache 2.0 で GitHub 公開 (Phase 5 以降の有力オプション)

雛形生成 CLI と tier1 Go ファサードの一部を Apache 2.0 で公開する。ZEN Engine / Dapr と同じライセンスで整合性が取れ、商用利用・改変・再配布が自由なため OSS コミュニティの受容性が高い。

- **経営インパクト**: 採用ブランディングに直接寄与 (Rust / Go 経験者の流入増)。グループ企業・パートナー企業にも無償提供可能で、エコシステムを形成しやすい。競合他社も使える点は「インフラは共通化・差別化は業務ドメインで」という割り切りに立脚する
- **技術インパクト**: OSS コミュニティからの PR / Issue 流入で開発速度が向上する一方、後方互換性の維持義務が生じる。コア開発者が 2〜3 名体制では対応負荷が跳ね上がるため、Phase 5 で専任の OSS メンテナンスチーム (0.5 FTE) が最低限必要
- **法務インパクト**: Apache 2.0 は特許条項 (§3) で貢献者の特許権を相互ライセンスするため、**k1s0 関連の特許取得戦略と整合** を取る必要がある。商標 "k1s0" は OSS 化前に防衛的登録を検討する。契約雛形は第 2 章の業務委託条項を「OSS 化後の貢献者契約 (CLA)」に拡張する

#### シナリオ ③ Dual License (商用オプション有)

中核コードは AGPL-3.0 または GPL-3.0 で公開し、商用利用には別途商用ライセンスを有償提供する。MongoDB / MariaDB / Grafana が採用するモデル。OSS の拡散と商用収益の両立を目指す。

- **経営インパクト**: 商用ライセンス収益が立てば k1s0 の開発費を部分的に回収可能。ただし JTC 情シスでこのモデルを運用する事例は国内に前例がなく、**営業・契約・法務の体制整備に初期投資 2,000〜5,000 万円 + 専任 1〜2 名** を要する。新規事業化の意思決定とセットでしか採算が合わない
- **技術インパクト**: AGPL/GPL 特有の「ネットワーク経由利用でも公開義務」が競合他社の採用ハードルを高めるため、採用層を限定する。コア機能と商用機能の境界設計に別途コストが発生
- **法務インパクト**: 3 シナリオで最も複雑。CLA (Contributor License Agreement) の運用、特許権の取扱い、商用契約の雛形整備、ライセンス違反時の法的対応体制まで必要。稟議承認段階では本シナリオは **明示的に「Phase 5 で再評価 / 新規事業化判断と同時」** と位置付ける

### 3.2 Phase 別の判断タイミング

OSS 化判断は一度固めると後戻りが難しい (公開した版を取り消せない) ため、**Phase 進行に応じて段階的に判断する** 。

- **Phase 0〜2**: シナリオ ① 社内専有で固定。OSS 化の議論は行わない。CLA 等の事前整備もこの段階では不要
- **Phase 3〜4**: tier1 公開 API が安定し tier2/3 のユーザーが定着した段階で、**雛形生成 CLI の部分 OSS 化 (シナリオ ②)** を検討開始。判断材料は「社内の OSS 運営体制を 0.5 FTE 確保できるか」「公開する価値のあるレベルに整っているか」の 2 点
- **Phase 5 以降**: 全社ロールアウト完了後に、経営層とのセットで **シナリオ ②/③ を再評価** する。特に新規事業化・グループ企業への有償提供を検討する場合はシナリオ ③ を選択肢に含める

稟議の場で深掘りされた場合は「Phase 0 時点ではシナリオ ① を既定値として進める。シナリオ ② ③ は Phase 5 以降の経営判断事項として選択肢を保持する」と即答できる。選択肢を放棄せず、かつ今は判断しないという構えを明示することが、稟議での刺さりを最小化する。

## 4. 特許と商標

k1s0 の個別要素は「k8s + Istio + Dapr」の組み合わせが主で、特許取得に値する新規性は現時点では識別されない。商標 "k1s0" も社内利用のみでは出願不要。両者とも Phase 3 完了時点で再評価する。稟議承認段階では追加費用計上は不要。

社外への提供・商用化を検討する段階になった場合、特許出願費用・商標登録費用・維持費用を別途予算化する。現時点での試算 ([`../04_定量試算/01_TCO5年試算.md`](../04_定量試算/01_TCO5年試算.md)) には含めない。
