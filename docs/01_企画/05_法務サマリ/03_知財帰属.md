# 知財帰属とライセンス

## 目的

本資料は、k1s0 を採用する際に問われる「k1s0 は誰の知財か」「ライセンス上の利用条件は何か」「採用側に何の責任が及ぶか」に対する答えを整理する。k1s0 は **起案者個人の著作物** であり、**Apache 2.0 ライセンス** で GitHub に公開されている OSS である。本章では帰属の構造、Apache 2.0 が採用側に与える権利、外部コントリビューションの扱い、特許・商標の扱いを整理する。

---

## 1. 帰属とライセンスの構造

k1s0 の **共通基盤資産**（tier1 公開 API 仕様、Go ファサード、Rust 自作領域、雛形生成 CLI、汎用マニフェスト、汎用ポータル部品、汎用設計ドキュメント）は、**起案者の著作物** である。起案者は業務外で個人的にこれを開発し、Apache 2.0 ライセンスで GitHub に公開している。

Apache 2.0 ライセンスにより、採用側（個人・企業を問わず）は以下の権利を取得する。

- **複製・実行**: コードベースを取得し、自由にビルド・実行できる
- **改変**: ソースコードを改変し、派生物を作成できる
- **再配布**: 改変版を含めて再配布できる（ライセンス表記の保持義務あり）
- **商用利用**: 商業目的で利用できる
- **特許利用**: コントリビュータが保有する関連特許の明示的許諾を受ける

採用側に課される義務は以下に限定される。

- ライセンステキスト（`LICENSE`）と告知（`NOTICE`）を派生物に同梱する
- 改変箇所を明示する（推奨）
- k1s0 の商標・トレードドレスを許可なく利用しない（Apache 2.0 第 6 条）

会社が k1s0 を業務利用していても、Apache 2.0 の原則として **改変版を社内に閉じて運用する場合のソース公開義務はない**（AGPL とは異なる）。社外公開する派生物については `LICENSE` と `NOTICE` の保持義務が発動する。

---

## 2. AGPL 依存 OSS との切り分け

k1s0 本体は Apache 2.0 だが、依存 OSS として AGPL の Grafana / Loki / MinIO 等を `infra/` レイヤで利用している。これは **k1s0 のコードベース自体を AGPL 化するわけではない**。AGPL OSS は独立したコンポーネントとしてプロセス分離されたうえで動作する構成を採っている。

| ライセンス区分 | 対象 | 採用側の義務 |
|---|---|---|
| Apache 2.0（k1s0 本体） | tier1 / 雛形 CLI / Go ファサード / Rust 自作領域 | `LICENSE` / `NOTICE` 保持、商標尊重 |
| AGPL（依存 OSS） | Grafana / Loki / MinIO 等 | 改変時のソース公開義務（FSF / Grafana Labs 公式 FAQ により通常利用では発動しない） |
| MIT（ZEN Engine 等） | ルールエンジン | ライセンス表記保持のみ |
| MPL 2.0（OpenBao / OpenTofu 等） | シークレット管理 / IaC | 改変ファイル単位での MPL 維持 |

AGPL 義務の発動条件（ソース改変時の公開義務）は、採用側が当該 OSS を改変するか否かで決まる。k1s0 の依存 OSS をそのまま利用する限り発動しないが、Grafana のダッシュボード自作やプラグイン作成は AGPL の射程に入る場合がある。詳細は [`01_OSSライセンス適合.md`](./01_OSSライセンス適合.md) を参照。

採用側はこの切り分けを自社の法務で確認する責任を負う。k1s0 のメンテナ（起案者）は AGPL 義務の発動可否について個別判断する責任は持たない。

---

## 3. 外部コントリビューションの扱い

k1s0 は GitHub で公開されており、外部開発者からのプルリクエスト・Issue を受け付ける。コントリビューションは以下のルールで運用する。

- **DCO（Developer Certificate of Origin）**: 全コミットに `Signed-off-by:` を必須とし、コントリビュータが正当な権利者であることを宣言してもらう
- **Apache 2.0 のもとで投稿される**: コントリビュータは投稿コードを Apache 2.0 でライセンスすることに同意する（CLA は採らず DCO で済ませる方針）
- **コントリビュータの著作権はコントリビュータ自身に残る**: k1s0 のコードベースは複数の著作権者の Apache 2.0 ライセンスコードの集合体になる
- **特許許諾**: Apache 2.0 第 3 条により、コントリビュータが保有する関連特許は採用側に明示的に許諾される

業務委託・SES・グループ会社から開発者が参画する場合、その雇用契約・業務委託契約と DCO の整合性は雇用主・委託元側で確認する責任を持つ。k1s0 として個別審査はしない。

---

## 4. 採用側との関係

k1s0 を業務利用する組織（会社）と、k1s0 の著作権者（起案者）の関係は以下のとおり。

- **k1s0 は会社の開発成果物ではない**: 会社は k1s0 に開発費を支払っておらず、起案者にも k1s0 開発の業務時間は割り当てられていない
- **会社は外部の任意ユーザと同じ立場**: 会社が k1s0 を採用するのは、原理的には GitHub から OSS を採用する任意の利用者と同じ。Apache 2.0 ライセンスの権利・義務がそのまま適用される
- **起案者の継続関与は保証されない**: 起案者は OSS のメンテナとして善意で開発を続けているが、契約上の継続義務はない
- **会社利用者は内製化の余地を確保する責任を持つ**: バス係数は本質的に 1。起案者が開発を止めた場合に備え、フォーク・内製化の準備を会社側で行うことが採用条件

「起案者が辞めたら著作権を持って行かれるのか」と問われた場合、**起案者は権利者のままだが Apache 2.0 ライセンスは取消不能なため運用は止まらない** と即答できる。Apache 2.0 のライセンスは付与時点で取消不能（irrevocable）であり、起案者が会社を辞めても・k1s0 の開発を止めても、既に取得した利用権は有効である。

---

## 5. 特許と商標

k1s0 の個別要素は「k8s + Istio + Dapr」の組み合わせが主で、特許取得に値する新規性は現時点では識別されない。商標 "k1s0" は GitHub プロジェクト名として保持しているのみで、商標登録はしていない。

Apache 2.0 ライセンス第 3 条により、コントリビュータが保有する関連特許は採用側に明示的に許諾される。これにより採用側はコントリビュータからの特許訴訟リスクを構造的に軽減できる。

将来的に商用化・有償サポートを検討する段階になった場合、特許出願費用・商標登録費用・維持費用は別途検討する。Apache 2.0 ライセンスを変更する予定はない（[`../01_背景と目的/06_個人開発OSSの系譜.md`](../01_背景と目的/06_個人開発OSSの系譜.md) 第 4 節参照）。

---

## 6. 法的助言の限界

本資料は採用側が k1s0 のライセンス構造を把握するための一般情報であり、個別事例の法的助言ではない。採用側が業務利用する際は自社の法務部門で以下を確認することを推奨する。

- 自社のソフトウェア著作権ポリシーと Apache 2.0 の整合
- 派生物作成・社外提供時の `LICENSE` / `NOTICE` 保持義務の運用
- AGPL 依存 OSS の改変時の公開義務発動条件
- ISMS / SOC 2 / FISC 等の監査フレームワーク内での OSS 採用記録の管理
