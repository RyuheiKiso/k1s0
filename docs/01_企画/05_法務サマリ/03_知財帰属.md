# 知財帰属と利用許諾

## 目的

本資料は、稟議で問われる「k1s0 は誰の知財か」に対し、**起案者帰属 + 所属企業への無償利用許諾** を前提に、外部開発者の扱い・OSS 化の将来オプション・特許商標の扱いを整理する。[`README.md`](./README.md) 第 1.3 節の即答を、契約条項の雛形と将来オプションまで含めて展開する。

---

## 1. 採用方針

k1s0 の **共通基盤資産**（tier1 公開 API 仕様、Go ファサード、Rust 自作領域、雛形生成 CLI、汎用マニフェスト、汎用ポータル部品、汎用設計ドキュメント）は、**起案者帰属** とする。そのうえで所属企業には、**無償・非独占・取消不能の社内利用許諾** を付与する。

この利用許諾に含める権利は以下のとおり。

- 社内での複製・実行・改変・バックアップ
- 障害対応・保守・監査対応のための利用
- 委託先 / SES / 指定グループ会社への開示と作業委託
- 継続運用に必要な派生物の社内利用

一方で、**JTC 固有の業務データ / 本番秘密情報 / 個別業務ルール / テナント固有設定** は社内管理とし、外部公開や再ライセンスの対象に含めない。

したがって稟議の場で「起案者が辞めたら著作権を持って行かれるのか」と問われた際には、**起案者は権利者のままだが、所属企業は取消不能の社内利用権を持つため運用は止まらない** と即答できる。

---

## 2. 外部開発者の扱い

業務委託・SES・グループ会社から開発者が参画する場合は、各契約雛形で以下を標準化する。

- **共通基盤への成果物** は、起案者へ権利帰属させるか、少なくとも起案者に再利用・再配布可能な包括ライセンスを付与する
- **所属企業固有の成果物**（業務ルール、機密設定、運用記録）は所属企業管理とし、共通基盤とは分離する
- 採用 OSS は allow list に限定し、無断で copyleft を混入させない
- 秘密保持義務（業務ロジック・認証情報・構成情報の第三者開示禁止）
- 業務終了後のコード・ドキュメント・認証情報の返却・削除義務

契約雛形の整備は Phase 0〜1 の法務部門タスクとして事前着手する。詳細条項は [`../../02_構想設計/05_法務とコンプライアンス/01_内製知財戦略.md`](../../02_構想設計/05_法務とコンプライアンス/01_内製知財戦略.md) 第 3.2 節を参照。

稟議の場で「委託先との契約が甘いと後で揉めるのでは」と問われた際には、本節の 5 項目を雛形として整備済みと即答できる。

## 3. OSS 化の将来オプション

稟議の場で「起案者帰属の知財を、将来 OSS として公開する選択肢はあるのか」「競業他社に使われないか」と問われた際、**選択肢を明示した上で判断は Phase 5 以降に保留する** のが k1s0 の方針である。起案者帰属を採ることで公開権限は明確になるが、公開可否はなお **所属企業の機密レビュー・商標レビュー・運用体制** とセットで判断する必要がある。

### 3.1 3 つの OSS 化シナリオ

k1s0 の共通基盤資産（雛形生成 CLI / tier1 Rust 自作領域 / tier1 Go ファサード / 汎用ポータル部品）の知財活用方法は、大別して以下 3 シナリオに整理できる。現時点（Phase 0）では **シナリオ ① 社内利用限定を既定値** とし、Phase 5 以降でシナリオ ② ③ を再評価する。

#### シナリオ ① 社内利用限定（既定値、Phase 0〜4 の方針）

起案者が権利を保持したまま、所属企業へ無償利用許諾のみを与え、外部公開は行わない。競合他社に使われるリスクはゼロだが、OSS コミュニティからのコントリビュート獲得・採用ブランディングの機会も失う。

- **経営インパクト**: 起案者の継続関与を前提にしつつ、所属企業は運用継続権を確保できる。外部プレゼンスは限定的
- **技術インパクト**: バグ発見と機能追加は社内リソース依存。CVE 対応も自力
- **法務インパクト**: 最もシンプル。社内利用のみのため AGPL OSS の公開義務も発動しない

#### シナリオ ② Apache 2.0 で GitHub 公開（Phase 5 以降の有力オプション）

起案者が権利者として、所属企業の機密レビューを経た共通基盤部分を Apache 2.0 で公開する。ZEN Engine / Dapr とライセンス整合が取りやすく、商用利用・改変・再配布が自由なため OSS コミュニティの受容性が高い。

- **経営インパクト**: 採用ブランディングに直接寄与。所属企業は自社実績を対外発信しやすい
- **技術インパクト**: OSS コミュニティからの PR / Issue 流入で開発速度が向上する一方、後方互換性の維持義務が生じる
- **法務インパクト**: 公開前に「会社固有情報を含まないこと」「商標利用をどう扱うか」「委託開発分の権利チェーンが閉じていること」を確認する必要がある

#### シナリオ ③ Dual License（商用オプション有）

中核コードは AGPL-3.0 または GPL-3.0 で公開し、商用利用には別途商用ライセンスを有償提供する。権利者が起案者であるため制度設計は可能だが、所属企業のブランド・サポート範囲・収益配分まで含めた合意が前提となる。

- **経営インパクト**: 収益化余地はあるが、JTC 情シスで直ちに採るモデルではない
- **技術インパクト**: OSS の拡散と商用オプションの両立には境界設計が必要
- **法務インパクト**: 3 シナリオで最も複雑。商用契約・特許・CLA の整備が必要

### 3.2 Phase 別の判断タイミング

OSS 化判断は一度固めると後戻りが難しい（公開した版を取り消せない）ため、**Phase 進行に応じて段階的に判断する**。

- **Phase 0〜2**: シナリオ ① 社内利用限定で固定。まずは所属企業内で価値を証明する
- **Phase 3〜4**: tier1 公開 API が安定し利用実績が積み上がった段階で、**雛形生成 CLI の部分 OSS 化（シナリオ ②）** を検討開始
- **Phase 5 以降**: 全社ロールアウト完了後に **シナリオ ②/③ を再評価**。公開可否は起案者と所属企業の共同判断とする

稟議の場で深掘りされた場合は「Phase 0 時点では社内利用限定で進める。外部公開は将来オプションとして保持する」と即答できる。

---

## 4. 特許と商標

k1s0 の個別要素は「k8s + Istio + Dapr」の組み合わせが主で、特許取得に値する新規性は現時点では識別されない。商標 "k1s0" も社内利用のみでは出願不要。両者とも Phase 3 完了時点で再評価する。

社外への提供・商用化を検討する段階になった場合、特許出願費用・商標登録費用・維持費用を別途予算化する。公開主体が起案者であっても、**所属企業名義での対外発信や導入事例公表には別途合意が必要** である。
