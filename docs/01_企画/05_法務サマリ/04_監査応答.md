# 監査応答の準備

## 目的

本資料は、稟議で問われる「5 年後に監査で説明できるか」に対し、自動生成される証跡と ISMS / ISO 27001 / SOC 2 監査で繰り返し問われる典型 5 問への即答を整理する。[`README.md`](./README.md) 第 1.4 節の即答を、証跡の生成タイミングと保管場所まで含めて展開する。

---

## 1. 自動生成される証跡

k1s0 は監査応答に必要な証跡を **CI パイプラインで自動生成** し、人手による証跡作成を排除する。以下は Phase 1c (MVP-1b) 時点で全件稼働する。

| 証跡 | 生成タイミング | 生成ツール | 保管場所 |
|---|---|---|---|
| SBOM (CycloneDX 形式) | 全ビルド時 | Syft | Harbor + `infra/compliance-reports` |
| CVE 検出レポート | 全ビルド時 | Grype + Trivy | 同上 |
| ライセンス判定結果 | 全 PR | Syft + allow list 照合 | PR コメント + `infra/compliance-reports` |
| 四半期ライセンスレポート | 四半期自動 | `operation` namespace のジョブ | `infra/compliance-reports` への自動 PR |
| AGPL 隔離状況 | 四半期自動 | Envoy ルート定義 + Keycloak クライアント定義のスナップショット | 同上 |

これらを **最低 5 年間** Git 管理することで、監査要求された時点で数分以内に当該時点のスナップショットを復元できる。人手による証跡作成が発生しないため、担当者交代や退職による証跡欠落のリスクも排除される。

---

## 2. 監査時の典型 5 問への即答

ISMS / ISO 27001 / SOC 2 / 業種特化監査で繰り返し問われる 5 問に対し、本章の記述だけで即答可能な状態を目指す。

| 質問 | 即答 | 根拠資料 |
|---|---|---|
| 利用中の OSS 一覧を提示してください | SBOM (CycloneDX) を Harbor から即座に出力 | 本章 1 節 |
| 各 OSS のライセンスを提示してください | SBOM にライセンス情報埋め込み済み | [`01_OSSライセンス適合.md`](./01_OSSライセンス適合.md) 第 1 節 |
| AGPL OSS の利用でソース提供義務を果たしていますか | 内部利用のため義務発動なし。FSF / Grafana Labs 公式 FAQ と隔離 5 原則を根拠として提示 | [`01_OSSライセンス適合.md`](./01_OSSライセンス適合.md) 第 2 節 |
| ライセンス違反検出の仕組みはありますか | CI に自動ライセンス監査を組み込み、違反時は PR ブロック | 本章 1 節 |
| 四半期ごとの OSS 状況を確認していますか | 四半期レポートを自動生成し、法務部門へ提示 | 本章 1 節 |

より詳細な監査対応フロー (RFC 起案 → ライセンス判定 → 法務レビュー → 採用決定) は [`../../02_構想設計/05_法務とコンプライアンス/00_OSS法務対応.md`](../../02_構想設計/05_法務とコンプライアンス/00_OSS法務対応.md) 第 5 章。

## 3. 監査応答を「自動化しておく」理由

監査応答を人手で準備する運用は、担当者不在時に破綻する。k1s0 はこの運用リスクを「証跡は常時自動生成されている」状態で構造的に排除する。監査チームが要求した時点で最新スナップショットが存在し、過去 5 年分は Git 履歴に残るため、監査応答担当者が特別な準備なしに回答できる。

この設計は、単に監査対応を楽にするためだけではない。Phase 2 以降にバス係数が上がり、特定の法務・運用担当者に依存しない組織構造を確立するための基盤でもある。[`../../02_構想設計/05_法務とコンプライアンス/00_OSS法務対応.md`](../../02_構想設計/05_法務とコンプライアンス/00_OSS法務対応.md) で詳述する監査対応フローは、この自動化を前提に設計されている。
