# OSS ライセンス適合

## 目的

本資料は、k1s0 が採用する 40+ OSS のライセンス義務を採用検討視点で要約する。[`README.md`](./README.md) 第 1.1 節で提示した「AGPL OSS は大丈夫か」の即答に対し、採用ライセンスの分布・AGPL-3.0 の扱い・MPL 2.0 の扱い・ライセンス変更リスクへの備えを、それぞれ論点化される前に封じ込める粒度で展開する。

---

## 1. 採用 OSS のライセンス分布

採用予定 40+ OSS のうち、ライセンスの種類は 8 種。採用検討で論点化するのは AGPL-3.0 (6 本) と MPL 2.0 (2 本) に限定され、残り 30+ 本 (Apache 2.0 / MIT / BSD / PostgreSQL / ISC) は **内部利用では義務なし**。

| ライセンス | 本数 | 代表 OSS | 内部利用での義務 |
|---|---|---|---|
| Apache 2.0 | 30 本超 | Kubernetes / Istio / Envoy / Kafka / Dapr / Keycloak / Argo CD / Backstage / Prometheus | NOTICE 表示のみ |
| MIT | 数本 | TypeScript ライブラリ / 一部 Rust crate | ライセンス文言添付のみ |
| BSD-3-Clause | 数本 | 一部 npm パッケージ | 同左 + 組織名の無断利用禁止 |
| PostgreSQL License | 1 本 | PostgreSQL 本体 | ライセンス文言添付のみ |
| ISC | 数本 | 一部 npm パッケージ | ライセンス文言添付のみ |
| MPL 2.0 | 2 本 | OpenTofu / OpenBao | 無改変利用なら義務なし |
| AGPL-3.0 | 6 本 | Grafana / Loki / Tempo / Pyroscope / MinIO / Renovate | 本章第 2 節で詳述 |
| デュアル (MIT OR Apache) | 数本 | sqlx 等 Rust crate | いずれかを選択 (MIT 選択) |

重要なのは Apache 2.0 + MIT + BSD 系で全体の約 8 割を占める事実である。この層は 採用側組織の情シスが気にする必要のない「通常の OSS」であり、論点化するのは AGPL と MPL の合計 8 本のみ。**論点の絶対数は小さい**。

---

## 2. AGPL-3.0 の扱い: 隔離 5 原則による内部利用

AGPL-3.0 OSS (Grafana / Loki / Tempo / Pyroscope / MinIO / Renovate) は infra 層で **プロセス分離された独立コンポーネント** として動作する。AGPL-3.0 第 13 条のソース提供義務は「外部ユーザーへのネットワーク越し提供」で発動する条項であり、社内利用 (採用側組織の認証ユーザーに限定) では FSF / Grafana Labs 公式 FAQ が「義務発動なし」と明言している。

採用側は以下 5 原則の遵守を証跡化することで、AGPL OSS の内部利用が義務発動条件に該当しないことを示せる。

1. プロセス分離 (AGPL OSS と自作コードは独立 Pod で動作)
2. ライブラリリンクなし (公開 API のみ使用、SDK 直接 import なし)
3. 無改変利用 (fork・プラグイン自作・ソース変更を禁止)
4. 外部境界遮断 (Envoy Gateway + Keycloak OIDC で 採用側組織の認証ユーザー以外を物理遮断)
5. 監査ログの可検証性 (AGPL Pod への全アクセスを Kafka 監査トピックに記録)

5 原則の詳細と監査時の回答スクリプトは [`../../02_構想設計/05_法務とコンプライアンス/00_OSS法務対応.md`](../../02_構想設計/05_法務とコンプライアンス/00_OSS法務対応.md) 第 2.5.3 節。

## 3. MPL 2.0 (OpenTofu / OpenBao) の扱い

MPL 2.0 はファイル単位の copyleft で、**対象 OSS を fork して改変した場合のみ** 改変ファイルのソース公開義務が発生する。k1s0 は両者とも無改変利用であり、義務は発生しない。採用検討の場ではこの 1 行で完結する。

k1s0 自作コードは OpenTofu のプロバイダ・OpenBao のシークレットバックエンドと **API 境界で接する** のみで、OSS 本体と静的/動的リンクしない。この境界設計により、将来 MPL 2.0 の運用方針が変わった場合でも影響範囲はファサード層に閉じる。

---

## 4. ライセンス変更リスクへの備え

OSS ライセンスは業界全体で頻繁に変更される。直近の事例は Redis (2024 / RSALv2 へ)、Terraform (2023 / BUSL へ)、Elastic (2021 / SSPL へ) で、いずれも「同等 OSS への fork 移行」が発生した。k1s0 は以下の体制で備える。

- CI に Syft + Grype による自動ライセンス検出を組み込み、採用 OSS のライセンス変更を即検知する
- allow list (Apache / MIT / BSD / PostgreSQL / ISC / MPL 2.0 / AGPL-3.0) 外への変更が検出された時点で、Renovate が PR ブロック
- 代替 OSS への移行計画は [`../../02_構想設計/03_技術選定/01_俯瞰/17_OSS長期戦略.md`](../../02_構想設計/03_技術選定/01_俯瞰/17_OSS長期戦略.md) で OSS ごとに事前定義 (Redis → Valkey の移行は実施済み)

これらの仕組みは k1s0 のリリース時点で全件稼働させ、「ライセンス変更を人手で検知する」状態を排除する。採用検討の場で「5 年後にライセンスが変わったらどうするのか」と問われた際、本節の自動検知 + 事前定義された代替 OSS を根拠に即答できる状態を維持する。
