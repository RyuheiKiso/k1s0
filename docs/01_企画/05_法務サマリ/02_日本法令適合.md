# 日本法令適合

## 目的

本資料は、稟議で問われる「個人情報保護法・マイナンバー法・電子帳簿保存法・業種特化規制は大丈夫か」に対し、k1s0 がプラットフォーム側で提供する技術要件と、業務アプリ (tier3) に委ねる責任範囲を明示する。[`README.md`](./README.md) 第 1.2 節の即答を、各法令の要件と k1s0 の実装要素に写像する粒度で展開する。

---

## 1. プラットフォームと業務アプリの責任分界

k1s0 自体は業務データを保持しない「プラットフォーム」であり、個人情報・マイナンバー・電子帳簿はすべて **業務アプリ (tier3)** が扱う。法令適合の責任は業務アプリにあるが、k1s0 は適合に必要な技術要件を **tier1 公開 API** として提供することで、業務アプリ側の実装負荷を下げる。

この分界により、「k1s0 は法令違反製品か」という稟議論点は成立しない。論点は「業務アプリが法令要件を満たすために必要な基盤機能が k1s0 に揃っているか」に書き換わる。以下、主要 4 法令について揃っている要件を整理する。

---

## 2. 個人情報保護法 (2022 改正) 適合

改正個人情報保護法は、保有個人データの開示請求対応・漏えい時の本人通知・仮名加工情報の取扱いなど、業務アプリ側の対応を大幅に増やした。k1s0 はこれらに必要な基盤機能を以下で担保する。

| 法令要件 | k1s0 の提供機能 | 参照 |
|---|---|---|
| アクセスログの保管 (第 23 条 安全管理措置) | `k1s0.Audit` で全 API コールを Kafka 監査トピック + 改ざん防止ハッシュチェーンで記録 | [`../../02_構想設計/01_アーキテクチャ/03_セキュリティ/01_セキュリティモデル.md`](../../02_構想設計/01_アーキテクチャ/03_セキュリティ/01_セキュリティモデル.md) |
| 通信経路の暗号化 | Istio mTLS (サービス間) + TLS (外部) を全通信で強制 | 同上 |
| データ保存時の暗号化 | OpenBao Transit Engine による透過的暗号化・鍵ローテーション | [`../../02_構想設計/03_技術選定/01_俯瞰/04_選定一覧.md`](../../02_構想設計/03_技術選定/01_俯瞰/04_選定一覧.md) |
| 本人確認・アクセス制御 | Keycloak OIDC で全コンポーネントを SSO 統一、RBAC で業務単位の権限制御 | 同上 |
| 開示請求時のデータ所在特定 | Backstage Software Catalog で保有データのオーナーを一覧化 | [`../../02_構想設計/04_CICDと配信/01_開発者ポータル_Backstage.md`](../../02_構想設計/04_CICDと配信/01_開発者ポータル_Backstage.md) |

## 3. マイナンバー法 (番号利用法) 適合

マイナンバーを取り扱う業務アプリは厳格な安全管理措置を要するが、k1s0 は「マイナンバーを扱う tier3 を隔離する」構造を提供する。

- 専用 namespace (`tier3-mynumber`) を分離し、NetworkPolicy で他 namespace からのアクセスを拒否
- OpenBao Transit Engine で常時暗号化、アクセスログを `k1s0.Audit` で別トピックに記録
- 閲覧可能な運用者は Keycloak の専用 RBAC ロールで限定、アクセスは全件記録

この構造により、マイナンバーを扱う業務アプリが満たすべき「物理・技術・組織・人的」4 類型の安全管理措置のうち、**技術的措置** が k1s0 で包括的に満たされる。

---

## 4. 電子帳簿保存法 (2022 改正) 適合

電子帳簿保存法は「真実性」「可視性」「検索性」の 3 要件を要求する。k1s0 が提供する機能は以下。

| 要件 | k1s0 の実装 |
|---|---|
| 真実性 (改ざん防止) | `k1s0.Audit` のハッシュチェーンに記録された時点で改ざん検出可能 |
| 真実性 (タイムスタンプ) | Kafka 監査トピックへの記録時刻が信頼できる基準時刻として機能。必要に応じて TSA (Timestamping Authority) 連携を tier1 で追加 |
| 可視性 (ディスプレイでの確認) | Backstage + Grafana で閲覧可能なダッシュボードを標準提供 |
| 検索性 (日付・金額・取引先) | tier1 `k1s0.Search` (Phase 2) で業務データの横断検索を提供 |

電子帳簿保存法の要求は業務アプリ固有の内容も含むため、完全な法令適合は業務アプリ側の責任である。ただし「技術的要件」に関しては k1s0 の基盤機能で半自動的に満たせる。

---

## 5. 業種特化規制 (金融・医療・公共)

業種特化規制 (FISC 安全対策基準 / 医療情報システムの安全管理ガイドライン / 政府情報システムのためのセキュリティ評価制度 ISMAP) は、JTC の業種によって要件が大きく異なる。k1s0 の基盤機能 (mTLS / 監査ログ / 暗号化 / RBAC / ネットワーク分離) はこれら基準の「技術的要件」を広くカバーするが、**業種特化の運用手続き** (暗号鍵管理体制・監査委員会・セキュリティ評価受審) は各 JTC の運用組織側で整備する必要がある。

Phase 1b 以降、業種特化規制の詳細マッピングを [`../../03_要件定義/30_セキュリティ_データ/10_compliance/`](../../03_要件定義/30_セキュリティ_データ/10_compliance/) で採番する。本章は稟議承認時点での「技術基盤側の準備済み項目」を示す。
