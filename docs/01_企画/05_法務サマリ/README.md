# 05_法務サマリ

k1s0 を稟議の場で否決しうる **法務・コンプライアンス 4 論点**（OSS ライセンス適合・日本法令適合・知財帰属・監査応答可能性）を、決裁 1 時間以内に読み切れる粒度で要約する章。詳細根拠と実装規定は [`../../02_構想設計/05_法務とコンプライアンス/`](../../02_構想設計/05_法務とコンプライアンス/) に委ねるが、「稟議の場で論点が出た瞬間に答えを返せる」水準を本章で担保する。

本章が埋める空白は、従来 `02_構想設計/05_...` に押し込まれていた法務論点を **稟議の場で直接参照できる深さに圧縮** した点である。法務部門が稟議に陪席しない JTC でも、本章の 4 論点即答（下記第 1 節）と各章ファイルを通読すれば、決裁者は主要論点を自答できる。

## 想定読者

第一読者は決裁者（情シス管理職・役員）。第二読者は法務部門（詳細確認のエントリーポイントとして）。起案者自身が稟議の場で参照する一次答弁ソースでもある。

## 1. 稟議で刺される 4 論点と即答

稟議の場では法務関連の質問は概ね以下 4 つの形で出る。それぞれについて、本節では 2〜3 行の即答を掲げ、詳細と根拠は配下の章ファイルに委ねる。

### 1.1 「AGPL の OSS を使うのは大丈夫か」

**即答**: Phase 1a (MVP-0) では AGPL OSS を一切採用しない。Phase 1b 以降で採用する 6 本 (Grafana / Loki / Tempo / Pyroscope / MinIO / Renovate) はすべて `infra` namespace 内部のみで動作し、JTC 認証ユーザー以外は物理的にアクセスできない。AGPL-3.0 第 13 条のソース提供義務は「外部ユーザーへのネットワーク越し提供」で発動する条項であり、社内利用では FSF / Grafana Labs 公式 FAQ が「義務発動なし」と明言している。

つまり **稟議承認の時点では AGPL 論点はゼロ**、Phase 1b 開始の法務レビュー時点で個別確認するプロセスに分割している。詳細と根拠は [`01_OSSライセンス適合.md`](./01_OSSライセンス適合.md) 第 2 節、および [`../../02_構想設計/05_法務とコンプライアンス/00_OSS法務対応.md`](../../02_構想設計/05_法務とコンプライアンス/00_OSS法務対応.md) 第 2.5 節。

### 1.2 「個人情報・マイナンバー・電帳法は大丈夫か」

**即答**: k1s0 はプラットフォームであり個人情報そのものを保持しない設計である。業務アプリ (tier3) が個人情報を扱う場合、tier1 の監査ログ・暗号化・アクセス制御 API を経由することで、個人情報保護法・マイナンバー法・電子帳簿保存法の各適合要件を **プラットフォーム側で一律に担保** する。個別の業務アプリで要件を重複実装する必要はない。適合性の具体的な写像は [`02_日本法令適合.md`](./02_日本法令適合.md) を参照。

### 1.3 「k1s0 は誰の知財か」

**即答**: **起案者が業務として作成する成果物は、職務著作 (著作権法 15 条) により JTC 法人に帰属する**。起案者個人が個別契約で保有権を持つ構造はとらない。業務委託・SES・グループ会社が開発に加わる場合も、各契約雛形で著作権を JTC へ譲渡する条項を標準化する。OSS 採用 OSS の copyleft 条項 (MPL 2.0 / AGPL-3.0) は [`01_OSSライセンス適合.md`](./01_OSSライセンス適合.md) 第 3 節のプロセス分離により k1s0 自作コードに伝播しない。詳細は [`03_知財帰属.md`](./03_知財帰属.md) および [`../../02_構想設計/05_法務とコンプライアンス/01_内製知財戦略.md`](../../02_構想設計/05_法務とコンプライアンス/01_内製知財戦略.md)。

### 1.4 「5 年後に監査で説明できるか」

**即答**: 採用 OSS の SBOM (Syft 自動生成)・四半期ライセンスレポート (自動 PR 起票)・ライセンス判定履歴を `infra/compliance-reports` リポジトリで Git 管理し、最低 5 年間保管する。ISMS / ISO 27001 / SOC 2 監査で要求される典型 5 問 (利用 OSS 一覧 / ライセンス一覧 / AGPL 義務履行 / 違反検出の仕組み / 四半期確認) は [`04_監査応答.md`](./04_監査応答.md) の対応表で即答可能。

## 2. 残存リスクと想定限界

本章は JTC 情シスが **稟議承認の時点で法務論点を封じ込める** ための要約書である。以下の限界があり、詳細対応は `02_構想設計/05_...` および法務部門の個別判断に委ねる。

- 本章の記述は法的助言ではない。AGPL-3.0 の解釈や業種特化規制の適用範囲は、最終的に各 JTC の法務部門が判断する
- OSS のライセンスは頻繁に変更される (Redis 2024 / Terraform 2023 / Elastic 2021)。本章の記述は 2026-04 時点のスナップショットであり、半期ごとの更新が必要
- 業種特化規制 (FISC / 医療情報ガイドライン / ISMAP) の詳細マッピングは本章では扱わず、Phase 1b 以降の要件定義層 ([`../../03_要件定義/30_セキュリティ_データ/10_compliance/`](../../03_要件定義/30_セキュリティ_データ/10_compliance/)) で採番する
- 業務委託契約・グループ会社契約の個別条項は、各 JTC の契約雛形との突き合わせが必要
