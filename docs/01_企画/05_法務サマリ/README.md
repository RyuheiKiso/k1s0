# 05_法務サマリ

k1s0 を採用する際に問われる **法務・コンプライアンス 4 論点**（OSS ライセンス適合・日本法令適合・知財帰属・監査応答可能性）を、採用判断者が 1 時間以内に読み切れる粒度で要約する章。詳細根拠と実装規定は [`../../02_構想設計/05_法務とコンプライアンス/`](../../02_構想設計/05_法務とコンプライアンス/) に委ねるが、「採用検討の場で論点が出た瞬間に答えを返せる」水準を本章で担保する。

k1s0 は **Apache 2.0** で公開されている個人開発の OSS であり、本章はその前提のもとで採用側が確認すべき法務観点を整理する。

## 想定読者

第一読者は k1s0 を採用しようとする組織の管理職・エンジニアリングマネージャ。第二読者は法務部門（詳細確認のエントリーポイントとして）。

## 1. 採用検討で問われる 4 論点と即答

採用検討の場では法務関連の質問は概ね以下 4 つの形で出る。それぞれについて、本節では 2〜3 行の即答を掲げ、詳細と根拠は配下の章ファイルに委ねる。

### 1.1 「AGPL の OSS が含まれているのは大丈夫か」

**即答**: k1s0 本体は Apache 2.0 ライセンスで、AGPL の伝播はない。依存 OSS として AGPL の Grafana / Loki / Tempo / Pyroscope / MinIO / Renovate を `infra/` レイヤで利用しているが、これらはプロセス分離された独立コンポーネントとして動作する。AGPL-3.0 第 13 条のソース提供義務は「外部ユーザーへのネットワーク越し提供」で発動する条項であり、社内利用では FSF / Grafana Labs 公式 FAQ が「義務発動なし」と明言している。改変時の公開義務発動条件は採用側で確認する責任を負う。詳細と根拠は [`01_OSSライセンス適合.md`](./01_OSSライセンス適合.md) 第 2 節、および [`../../02_構想設計/05_法務とコンプライアンス/00_OSS法務対応.md`](../../02_構想設計/05_法務とコンプライアンス/00_OSS法務対応.md) 第 2.5 節。

### 1.2 「個人情報・マイナンバー・電帳法は大丈夫か」

**即答**: k1s0 はプラットフォームであり個人情報そのものを保持しない設計である。業務アプリ (tier3) が個人情報を扱う場合、tier1 の監査ログ・暗号化・アクセス制御 API を経由することで、個人情報保護法・マイナンバー法・電子帳簿保存法の各適合要件を **プラットフォーム側で一律に担保** する。個別の業務アプリで要件を重複実装する必要はない。適合性の具体的な写像は [`02_日本法令適合.md`](./02_日本法令適合.md) を参照。

### 1.3 「k1s0 は誰の知財か」

**即答**: **k1s0 の著作権は起案者個人に帰属し、Apache 2.0 ライセンスで GitHub に公開されている**。採用側は Apache 2.0 の権利（複製・改変・再配布・商用利用・特許許諾）を取得し、義務として `LICENSE` / `NOTICE` の保持・改変箇所の明示・k1s0 商標の尊重を負う。Apache 2.0 ライセンスは取消不能（irrevocable）なため、起案者が開発を止めても採用側の利用権は失われない。詳細は [`03_知財帰属.md`](./03_知財帰属.md) および [`../../02_構想設計/05_法務とコンプライアンス/01_内製知財戦略.md`](../../02_構想設計/05_法務とコンプライアンス/01_内製知財戦略.md)。

### 1.4 「5 年後に監査で説明できるか」

**即答**: 採用 OSS の SBOM (Syft 自動生成)・四半期ライセンスレポート (自動 PR 起票)・ライセンス判定履歴を `infra/compliance-reports` リポジトリで Git 管理し、最低 5 年間保管する。ISMS / ISO 27001 / SOC 2 監査で要求される典型 5 問 (利用 OSS 一覧 / ライセンス一覧 / AGPL 義務履行 / 違反検出の仕組み / 四半期確認) は [`04_監査応答.md`](./04_監査応答.md) の対応表で即答可能。

## 2. 残存リスクと想定限界

本章は採用側が **採用判断時に法務論点を確認する** ための要約書である。以下の限界があり、詳細対応は `02_構想設計/05_...` および採用側の法務部門の個別判断に委ねる。

- 本章の記述は法的助言ではない。AGPL-3.0 の解釈や業種特化規制の適用範囲は、最終的に採用側の法務部門が判断する
- OSS のライセンスは頻繁に変更される (Redis 2024 / Terraform 2023 / Elastic 2021)。本章の記述は 2026-04 時点のスナップショットであり、半期ごとの更新が必要
- 業種特化規制 (FISC / 医療情報ガイドライン / ISMAP) の詳細マッピングは本章では扱わず、要件定義層 ([`../../03_要件定義/30_非機能要件/`](../../03_要件定義/30_非機能要件/)) で採番する
- 業務委託契約・グループ会社契約の個別条項は、採用側の契約雛形との突き合わせが必要
