# イベントスキーマレジストリ

## 目的

k1s0 のイベント駆動アーキテクチャ ([`../../02_アーキテクチャ/04_非機能とデータ/02_データアーキテクチャ.md`](../../02_アーキテクチャ/04_非機能とデータ/02_データアーキテクチャ.md)) において、Kafka トピックのイベントスキーマを中央管理し、破壊的変更を CI / ランタイムで機械的に検出・拒否する仕組みを導入する。

---

## 1. 解決する問題

データアーキテクチャでは CloudEvents v1.0 / トピック命名規則 / イベント型バージョニングを定義しているが、これらは **ドキュメント上の規約** であり、実行時に違反を検出する仕組みがない。

| 問題 | 発生シナリオ | 影響 |
|---|---|---|
| スキーマ破壊的変更 | サービス A がフィールドの型を変更 | サービス B のデシリアライズが失敗し本番障害 |
| スキーマドリフト | 複数チームが同じイベント型を独自に拡張 | 消費側で予期しないフィールドが混入 |
| 契約の不可視 | どのサービスがどのイベントを発行・消費しているか分からない | 変更影響が事前に把握できない |
| 検知の遅延 | スキーマ不整合がデプロイ後に判明 | ロールバックと原因調査で運用負荷が増大 |

tier1 の API 設計原則 ([`../../03_tier1設計/02_API契約/03_API設計原則.md`](../../03_tier1設計/02_API契約/03_API設計原則.md)) は「規律はドキュメントではなくツールで強制する」を基本思想とする。gRPC 契約には `buf breaking` があるが、**イベント契約に相当する検証手段がない**。スキーマレジストリはこのギャップを埋める。

---

## 2. 候補比較

| 候補 | 採否 | 評価 |
|---|---|---|
| **Apicurio Registry** | 採用 | Red Hat 主導の OSS。Apache 2.0。JSON Schema / Avro / Protobuf 対応。REST API / Maven / Gradle プラグインで CI 統合可能 |
| Confluent Schema Registry (Community) | 次点 | Confluent Community License (非 OSI)。機能は充実するが k1s0 のライセンス方針に不適合 |
| Karapace | 次点 | Aiven 発の OSS (Apache 2.0)。Confluent 互換 API。コミュニティ規模が小さい |
| Buf Schema Registry | 対象外 | Protobuf 専用。JSON Schema / Avro 非対応。gRPC 契約は `buf breaking` で十分 |
| 自前実装 (Git + CI) | 却下 | スキーマの互換性検証ロジックを自前で実装・保守する負担が大きい |

---

## 3. 採用理由 (Apicurio Registry)

1. **Apache 2.0 ライセンス** — k1s0 の OSS ライセンス方針に完全適合。Confluent Schema Registry (非 OSI) を排除した上での最適解
2. **JSON Schema をネイティブサポート** — k1s0 の CloudEvents は `datacontenttype: application/json` を標準とする。Avro / Protobuf への将来移行も可能
3. **互換性ルールの自動検証** — `BACKWARD` / `FORWARD` / `FULL` の互換性モードを artifact 単位で設定可能。CI で `apicurio-registry-maven-plugin` (または REST API) を呼び出してスキーマ互換性を自動検証
4. **PostgreSQL バックエンド対応** — CloudNativePG 共有クラスタ上に `apicurio` DB を作成し、既存インフラに統合
5. **REST API が充実** — Go / Rust / TypeScript から HTTP で操作可能。tier1 Go ファサードに統合しやすい
6. **Kafka SerDe 統合** — Kafka プロデューサー / コンシューマーがシリアライズ / デシリアライズ時にレジストリを参照し、不整合を実行時にも検出可能
7. **Red Hat の継続的サポート** — Red Hat build of Apicurio Registry として商用サポートも利用可能

## 4. k1s0 における役割

### 4.1 tier1 アーキテクチャへの統合

Apicurio Registry は **tier1 内部実装** として配置する。tier2 / tier3 はレジストリの存在を意識しない (Dapr 隠蔽と同じ原則)。

| 通信パス | 説明 |
|---|---|
| tier2/tier3 → `k1s0.PubSub.Publish()` → Go ファサード → Apicurio 検証 → Kafka | 発行時にスキーマバリデーションを実行 |
| Kafka → Dapr PubSub → Go ファサード → Apicurio 検証 → tier2/tier3 | 受信時にデシリアライズ検証を実行 |
| CI (GHA) → Apicurio REST API | PR 時にスキーマ互換性を自動検証 |
| Backstage → Apicurio REST API | イベント API カタログとしてスキーマを可視化 |

### 4.2 スキーマ管理の責務

| 作業 | 責任者 | 手段 |
|---|---|---|
| スキーマの初期登録 | 各サービスの開発チーム | `schemas/` ディレクトリに JSON Schema を配置、CI で自動登録 |
| 互換性モードの設定 | システム基盤チーム | Apicurio の artifact ルール設定 (デフォルト: `BACKWARD`) |
| スキーマの互換性検証 | CI パイプライン | PR 時に Apicurio REST API で検証 |
| スキーマカタログの参照 | 全開発者 | Backstage プラグインまたは Apicurio UI |

### 4.3 互換性モードの適用方針

| イベント分類 | 互換性モード | 根拠 |
|---|---|---|
| ドメインイベント (tier2) | `BACKWARD` | コンシューマーが旧スキーマでも読める。プロデューサー先行リリースが可能 |
| 監査イベント (tier1) | `FULL` | プロデューサー・コンシューマー双方の互換性を保証。改ざん防止要件 |
| Saga コマンド (tier1) | `BACKWARD` | Workflow オーケストレーターの先行リリースが可能 |

## 5. CI / CD パイプラインへの統合

既存の GHA ワークフローに **スキーマ互換性チェック** ステージを追加する。

| ステージ | 追加位置 | 動作 | 失敗時 |
|---|---|---|---|
| スキーマ互換性チェック | `buf breaking` の直後 | `schemas/` 内の JSON Schema を Apicurio REST API に送信し、`BACKWARD` 互換性を検証 | PR ブロック |

### CI フロー (PR 時、イベントスキーマ変更を含む場合)

1. Lint / フォーマット
2. ユニットテスト
3. `buf breaking` (gRPC 契約)
4. **Apicurio 互換性チェック (イベント契約)** — 追加
5. コンテナビルド → スキャン → push → GitOps

### スキーマファイルの配置

雛形生成 CLI が以下のディレクトリ構造を生成する。

| パス | 内容 |
|---|---|
| `schemas/events/{event-type}/v{N}.json` | イベントの JSON Schema 定義 |
| `schemas/events/{event-type}/testdata/` | スキーマのテスト用サンプルデータ |

## 6. Backstage との統合

Backstage の Software Catalog でイベント API を可視化する。

| 統合方法 | 内容 |
|---|---|
| Apicurio Registry API の呼び出し | 登録済みスキーマの一覧・詳細を Backstage から参照 |
| `catalog-info.yaml` へのアノテーション追加 | `apicurio/artifact-group` でサービスとイベントスキーマを紐付け |
| イベントの発行元・消費先の可視化 | どのサービスがどのイベントを発行・消費しているかをカタログで表示 |

## 7. 段階導入計画

| フェーズ | スコープ |
|---|---|
| Phase 2 | Apicurio Registry を `infra` namespace にデプロイ。PostgreSQL バックエンドで CloudNativePG 共有クラスタに `apicurio` DB を作成。tier1 監査イベントのスキーマを登録。CI にスキーマ互換性チェックを追加。Go ファサード内で発行時バリデーションを実装 |
| Phase 3 | tier2 ドメインイベントのスキーマ登録を開始。Backstage 連携 (イベント API カタログ)。受信時バリデーションの追加 |
| Phase 4 | 全イベントのスキーマ登録を義務化。Apicurio UI でのスキーマブラウジングを業務担当に公開 |

### MVP での扱い

MVP では Kafka を導入しないため、Apicurio Registry も導入しない。Phase 2 で Kafka (Strimzi) と同時に導入する。

---

## 8. リスクと受容判断

| リスク | 影響 | 緩和策 |
|---|---|---|
| Apicurio のコミュニティ規模が Confluent より小さい | 機能追加やバグ修正の速度 | Red Hat 主導で開発は活発。商用サポートも利用可能 |
| スキーマ検証のレイテンシ増加 | `k1s0.PubSub.Publish()` の応答時間 | スキーマをローカルキャッシュ (Valkey) し、レジストリへの問い合わせを最小化 |
| JSON Schema の表現力の限界 | 複雑な制約を記述できない場合がある | 必要に応じて Avro への移行を検討。Apicurio は複数フォーマットをサポート |
| レジストリ障害時のイベント発行 | スキーマ検証ができない | グレースフルデグラデーション: レジストリ不達時はキャッシュ済みスキーマでバリデーション。キャッシュもない場合はバリデーションをスキップしてログ警告 (fire-and-forget 的扱い) |
