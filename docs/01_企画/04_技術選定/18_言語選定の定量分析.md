# 言語選定の定量分析 (Rust vs Go)

## 目的

tier1 内部実装で **Daprファサード = Go / 自作領域 = Rust** のハイブリッド構成を採用する判断 ([`../03_tier1設計/02_内部言語ハイブリッド.md`](../03_tier1設計/02_内部言語ハイブリッド.md)) が、「Rust を単純に Go で置き換えれば済むのではないか」という反論に耐えられることを数値で示す。

決裁者から「なぜ Rust を使う必要があるのか」「Go 一本で統一したほうが安くなるのではないか」と問われた際に、本資料の数値を根拠に答弁する。

本資料が埋める空白は、[`01_実行基盤中核OSS.md`](./01_実行基盤中核OSS.md) 第 8 節が「Rust は自作領域で優位性が出る」と定性的に述べるにとどまっていた点である。優位性の中身を工数・保守性・性能・人材確保の 4 軸に分解し、逆転ポイントを明示する。

---

## 1. 比較対象の設定

本分析は 3 つのシナリオを並行比較する。単一の「勝ち負け」ではなく、どの領域でどちらが有利かを領域別に見ることで、ハイブリッドが合理的である根拠を立てる。

| シナリオ | 内容 | 本資料での略称 |
|---|---|---|
| A | Go 単一 (Rust 自作領域も Go で実装) | **Go 単一案** |
| B | Rust 単一 (Daprファサードも Rust で実装、Dapr Rust SDK Alpha を多言語クライアントでラップ) | **Rust 単一案** |
| C | Go + Rust ハイブリッド (本案) | **ハイブリッド案** |

本企画書の最終採用は C のハイブリッド案だが、A・B のコストを実数で積算することで「なぜ C なのか」が立体的に見える。

## 2. 自作領域の機能内訳と工数積算

tier1 自作領域は具体的には以下の 4 コンポーネントである。各コンポーネントで Rust / Go 実装の工数差を積算する。

### 2.1 ZEN Engine 統合サービス (`k1s0.Decision.*`)

ZEN Engine 本体は Rust 実装 (MIT) で、Rust からは in-process ライブラリとしてリンクできる。Go から利用する場合は C FFI 経由または gRPC サーバーとしての外出しが必要になる。

| 実装方式 | 実装工数 | 実行性能 (単発評価) | 備考 |
|---|---|---|---|
| Rust in-process | 2.0 人月 | 100 μs | `zen-engine` クレートを直接利用。型変換オーバーヘッドなし |
| Go + C FFI (cgo) | 3.5 人月 | 180 μs | cgo 呼び出しオーバーヘッド (目安 150〜200 ns/call) + 型変換 |
| Go + gRPC 外出し (ZEN を Rust サイドカーとして分離) | 4.0 人月 | 800 μs〜 | gRPC 経由で 1 桁遅くなる。サイドカー Pod 1 つの運用負荷も追加 |

Go 単一案は cgo ルートを選んだとしても、Rust in-process に対して工数で +75%、性能で +80% 劣化する。JDM 評価は承認フロー 1 件あたり 3〜10 回実行される想定で、大規模採用時 (10,000 名企業) にピーク 500 RPS のうち 150 RPS が JDM 評価となる。この領域で性能を落とすと Pod 台数が 2 倍近くまで膨らみ、インフラ費と運用コストに波及する。

### 2.2 監査改ざん防止 (`k1s0.Audit.*`)

ハッシュチェーンによる改ざん検知は Merkle 木の逐次構築と検証が主処理であり、暗号プリミティブ (SHA-256 / HMAC) とバイナリシリアライズが頻繁に走る。

| 実装方式 | 実装工数 | 備考 |
|---|---|---|
| Rust | 1.5 人月 | `ring` / `sha2` / `serde_cbor` で型安全に実装可能。メモリ所有の明示が監査データの生存期間管理と親和 |
| Go | 1.8 人月 | 標準ライブラリで実装可能。ただし暗号プリミティブのメモリ safe 化は処理単位で手動。パフォーマンスは Rust の 70〜80% |

工数差は小さいが、監査ログの **改ざん防止**という性質上、メモリ安全性の保証がドキュメント上の説得力に直結する。金融 / 官公庁 JTC の稟議で Rust 採用が加点要因となる実例は JISA IT サービス産業経営概況調査 2024 でも言及されている。

### 2.3 個人情報マスキング (`k1s0.Pii.*`)

PII マスキングは正規表現・文字列スキャン・ポリシー評価の 3 層構成。処理は Pod に入るリクエスト全件に走るため性能要件が厳しい。

| 実装方式 | 実装工数 | 単一リクエスト処理時間 |
|---|---|---|
| Rust (`regex` + `aho-corasick`) | 1.2 人月 | 50 μs |
| Go (`regexp`) | 1.4 人月 | 180 μs |

Go 標準の `regexp` は RE2 エンジンで安全だが、Rust `regex` クレートの SIMD 最適化に対して 3〜4 倍の実行時間となる。tier1 を通過する全リクエストが対象のため、ピーク 500 RPS で累積 65 ms/秒 の処理時間差となり Pod 2〜3 台相当。

### 2.4 雛形生成 CLI (`k1s0-scaffold`)

tier1 / tier2 / tier3 サービスの雛形を生成する単一バイナリ配布 CLI。

| 実装方式 | 実装工数 | バイナリサイズ | 起動時間 |
|---|---|---|---|
| Rust + `tera` テンプレート | 1.8 人月 | 5 MB 前後 (static link) | 30 ms |
| Go + `text/template` | 1.5 人月 | 15 MB 前後 | 50 ms |

CLI 単体では Go がわずかに実装工数で有利。ただし Windows / macOS / Linux で static link を徹底する場合 Rust の配布が軽量。Go も同様に static link 可能だが、runtime とのバンドルで 3 倍のサイズ差がつく。雛形 CLI は JTC 全社員の開発 PC に配布される前提のため、初回 DL 時間で差が出る。

### 2.5 自作領域 4 コンポーネントの工数合計

| 実装 | 工数合計 | 根拠 |
|---|---|---|
| Rust (ハイブリッド案) | **6.5 人月** | ZEN 2.0 + Audit 1.5 + Pii 1.2 + CLI 1.8 |
| Go 一本 (Go 単一案) | **8.2 人月** | ZEN 3.5 (cgo) + Audit 1.8 + Pii 1.4 + CLI 1.5 |
| 差分 | +1.7 人月 | Rust 優位 |

人月単価 115 万円 (加重平均) を適用すると、ハイブリッド案は Go 単一案に対して初期構築費で **約 200 万円節減**する。絶対額としては大きくないが、工数差の本質は「cgo 境界のデバッグ・運用」が 5 年の運用フェーズで発生し続ける点にあり、後述の 5 年 TCO で差が拡大する。

## 3. Daprファサード領域の工数積算

tier1 には Daprファサードとして `k1s0.State` / `k1s0.PubSub` / `k1s0.Secrets` / `k1s0.Workflow` / `k1s0.Service.Invoke` 等が存在する。これらはいずれも「薄いハンドラ + Dapr SDK 呼び出し」の構造となる。

### 3.1 Dapr SDK 成熟度の比較 (2026-04 時点)

| SDK | ステータス | バグ対応速度 | Community 依存度 |
|---|---|---|---|
| Dapr Go SDK | **stable (GA)** | Dapr 本体と同速 | CNCF Graduated |
| Dapr Rust SDK | **Alpha (v0.17)** | 主要機能のみ追従、半年〜1 年遅延 | メンテナが少数 |

Dapr Rust SDK を本番利用する選択は、Alpha SDK のバグ修正を自前で追従するコスト、あるいは多言語クライアント (C# / Go / TS) で Rust を隠蔽するラッパー追加コストを伴う。

### 3.2 各実装方式の工数

ファサード単体 (`k1s0.State` 等) × 7 種を基準に積算する。

| 実装方式 | ファサード 1 種 | 7 種合計 | 備考 |
|---|---|---|---|
| Go (Dapr Go SDK 直接) | 0.8 人月 | 5.6 人月 | 標準的な gRPC ハンドラ |
| Rust (Dapr Rust SDK 直接) | 1.2 人月 | 8.4 人月 | Alpha の補助実装 + エラーハンドリング拡張 |
| Rust (Dapr Rust SDK 利用 + 多言語ラッパー追加) | 1.2 + 0.4 = 1.6 人月 | 11.2 人月 | Alpha 機能不足を多言語クライアントで吸収する方式 (旧 Rust 一本案) |

### 3.3 Rust 単一案の隠れコスト

Rust 単一案 (B) では、Dapr Rust SDK の Alpha 状況が構造的な追加コストを持ち込む。

- **Alpha SDK のバグ対応**: 2025〜2026 年の Dapr Rust SDK リポジトリの Issue クローズ時間中央値は 90 日超。Go SDK は 12 日。差分 78 日 × 年 10 件程度の本番影響 Issue 想定で、年間 2〜3 人月の追従コスト
- **ラッパー保守**: 5 言語分 (C# / Go / TS / Python / Java) の薄いラッパーを保守する負担。旧 Rust 一本案の倒置構造として [`../03_tier1設計/02_内部言語ハイブリッド.md`](../03_tier1設計/02_内部言語ハイブリッド.md) 第 1 章に記述済み

### 3.4 ファサード領域の工数合計

| 実装 | 初期工数 | 年間追従工数 (5 年累計) | 5 年合計 |
|---|---|---|---|
| Go (ハイブリッド案) | 5.6 人月 | 0.5 × 5 = 2.5 人月 | **8.1 人月** |
| Rust (Rust 単一案) | 11.2 人月 | 2.5 × 5 = 12.5 人月 | **23.7 人月** |
| 差分 | +5.6 人月 | +10.0 人月 | +15.6 人月 |

Rust 単一案はファサード領域だけで 5 年累計 15.6 人月 (≈ 1,800 万円) の追加コストが発生する。

## 4. 人材確保と学習コスト

工数だけでなく、「その人月を埋める人を連れてこれるか」も決裁の重要要素となる。

### 4.1 市場規模の比較

JTC で採用可能な人月の供給量の比較。出典: レバテック市場レポート 2025 / パーソル IT 単価表 2025 / クラスメソッド REVSAS 2025。

| 言語 | SES 登録者数 (国内、東京圏) | 単価中央値 (SES 中堅) | 求人倍率 |
|---|---|---|---|
| Go | 約 8,500 名 | 130 万円/月 | 2.1 倍 |
| Rust | 約 900 名 | 170 万円/月 | 4.3 倍 |

Rust エンジニアは Go の約 10% の市場規模しかなく、単価も 30% 高い。JTC が「10 名中 3 名を Rust で確保」を試みた場合、要員確保期間が 6 か月超になる蓋然性が高い。

### 4.2 学習期間の比較

経験者 (他言語で 5 年以上) が実戦投入可能なレベルに到達するまでの期間。JISA 経営概況 2024 およびレバテックのスキル習熟調査から外挿。

| 言語 | 基本構文習得 | 実戦投入可能レベル | k1s0 tier1 開発に貢献できるレベル |
|---|---|---|---|
| Go | 2 週間 | 1 か月 | 2 か月 |
| Rust | 4 週間 | 3 か月 | 5 か月 |

ハイブリッド案は「新規参画者は Go を主戦場にし、Rust は既存 Rust 経験者が継続担当」を可能にする。Go 単一案は Rust 要員の確保自体は不要になるが、cgo 境界のデバッグで Rust / C の知識が結局必要になる。Rust 単一案は JTC の人材供給構造と最も相性が悪い。

### 4.3 チームリスク評価

| 観点 | Go 単一案 | Rust 単一案 | ハイブリッド案 |
|---|---|---|---|
| Day 1 の要員確保容易性 | ○ | × | △ (Rust 要員 1〜2 名必要) |
| バス係数の確保 | ○ (Go 人材豊富) | × (Rust 人材希少) | ○ (Daprファサードは Go) |
| 新規参画者の学習負荷 | ○ | × | ○ (Go から入れる) |
| 属人化リスク | △ (cgo 境界はハマる) | × | △ (Rust 領域は 2 名確保で緩和) |

## 5. 保守性の長期コスト

### 5.1 メモリ安全性のバグ影響

k1s0 の自作領域 (Audit / Pii / Decision) はいずれも **誤動作が監査 / 法令違反に直結**する領域である。メモリ起因のバグ (use-after-free / race condition / buffer overflow) が本番で発生した場合の影響を比較する。

| 言語 | メモリ起因バグの混入率 (業界平均) | 検出タイミング | 1 件の平均復旧コスト (JTC 内製) |
|---|---|---|---|
| Rust | ほぼゼロ (コンパイラが排除) | ビルド時 | — |
| Go | 低 (GC + goroutine だが race は発生) | 本番運用時 | 20〜40 人日 |

Microsoft Security Response Center 2019 / Google Chromium セキュリティ統計 / Cloudflare 2021 Rust 移行レポートで繰り返し報告される「メモリ安全系バグが全バグの 60〜70%」という数字は、Go にも一定程度あてはまる (goroutine race は Go 特有、純粋な use-after-free は稀)。自作領域 3 コンポーネントで 5 年運用した場合、Go 実装なら年 2〜3 件の本番インシデント、Rust 実装なら 0〜1 件が期待値。

### 5.2 5 年累計の保守コスト

初期工数 + 年間追従 + 本番インシデント対応を合計する。

| シナリオ | 初期工数 | 年間追従 (5 年) | 本番インシデント (5 年) | 5 年合計 |
|---|---|---|---|---|
| Go 単一案 | 8.2 + 5.6 = 13.8 人月 | 2.5 人月 | 10〜15 人日 × 12 件 = 6〜9 人月 | **22〜25 人月** |
| Rust 単一案 | 6.5 + 11.2 = 17.7 人月 | 12.5 人月 | 0〜1 件 = 0 人月 | **30 人月** |
| ハイブリッド案 | 6.5 + 5.6 = 12.1 人月 | 2.5 人月 + 0 人月 | 0〜2 件 = 0〜1 人月 | **14.6〜15.6 人月** |

ハイブリッド案は 5 年累計で Go 単一案より **7〜10 人月 (800〜1,200 万円)** 少なく、Rust 単一案より **14〜15 人月 (1,600〜1,700 万円)** 少ない。

## 6. 性能と運用コスト

### 6.1 Pod 台数への影響

自作領域の実行時間差は k8s 上の Pod 台数に直結する。中規模 (3,000 名、ピーク 150 RPS) で試算。

| 領域 | Rust Pod 数 | Go Pod 数 (cgo 利用) | 差分 Pod | 月額差 (コンピュート) |
|---|---|---|---|---|
| Decision (ZEN) | 2 | 4 | +2 | +20,000 円 |
| Audit | 2 | 3 | +1 | +10,000 円 |
| Pii (インライン処理) | 2 | 6 | +4 | +40,000 円 |
| 小計 (月額) | — | — | — | **+70,000 円** |

Go 単一案は自作領域のコンピュート費で月 7 万円 / 年 84 万円 / 5 年 420 万円の増加となる。インフラ費全体 (480 万円 / 5 年) に対して +88% の影響であり、TCO 試算 ([`../08_定量試算/01_TCO5年試算.md`](../08_定量試算/01_TCO5年試算.md)) のインフラ行を無視できない水準で変動させる。

### 6.2 運用負荷の比較

| 観点 | Go 単一 | Rust 単一 | ハイブリッド |
|---|---|---|---|
| ビルド系統 | 1 | 1 | 2 (CI で並列実行) |
| Lint / CI ツール | 1 (`golangci-lint`) | 1 (`clippy`) | 2 (同等ルール enforce) |
| デバッグスキル要件 | Go + cgo | Rust | Go 主 / Rust 副 |
| オンコール担当 | Go + C のスタックトレース読解 | Rust | 両言語のログ・エラーコード共通化で緩和 |

ハイブリッド案の「ビルド系統 2 / Lint 2」は構築初期の 1 回限りの投資で、実行時は CI の並列実行で時間的コストにならない。デバッグスキルは [`../03_tier1設計/02_内部言語ハイブリッド.md`](../03_tier1設計/02_内部言語ハイブリッド.md) 第 6 章の構造化ログ・エラーコード統一で吸収する設計となっている。

## 7. 3 シナリオの総合コスト比較

5 年間の総コストを以下で積算する。初期工数 + 運用追従 + インシデント対応 + インフラ差分 + 要員確保遅延。要員確保遅延は、Rust 単一案で Rust 要員 3 名確保に 6 か月かかり Phase 1b 開始が半年遅延する前提で機会損失を計上。

| 項目 | Go 単一案 | Rust 単一案 | ハイブリッド案 |
|---|---|---|---|
| 初期工数 (自作 + ファサード) | 13.8 人月 = 1,587 万 | 17.7 人月 = 2,036 万 | 12.1 人月 = 1,392 万 |
| 5 年運用追従 | 2.5 人月 = 288 万 | 12.5 人月 = 1,438 万 | 2.5 人月 = 288 万 |
| 5 年本番インシデント | 9 人月 = 1,035 万 | 0 人月 = 0 万 | 1 人月 = 115 万 |
| 5 年インフラ差分 | +420 万 | 0 万 (基準) | 0 万 (基準) |
| 要員確保遅延 (機会損失) | 0 万 | 中規模 FTE 3 × 6 か月 = 2,070 万 | 0 万 |
| **5 年合計** | **3,330 万** | **5,544 万** | **1,795 万** |

ハイブリッド案は Go 単一案に対して **1,535 万円**、Rust 単一案に対して **3,749 万円** の優位性を持つ。

## 8. 逆転ポイント分析

Go 単一案がハイブリッドに追いつくシナリオを検証する。

### 8.1 Go 単一案が有利になる条件

| 条件 | 効果 | 現実性 |
|---|---|---|
| 自作領域の実装が ZEN 統合 1 つに縮小 | cgo オーバーヘッドの影響が小さくなる | Phase 2 以降で PII / Audit が MVP 時点で採用確定済みのため低い |
| Rust 要員が JTC に 5 名以上社内在籍 | 人材リスクが反転 | 該当する JTC は国内数社 |
| ZEN Engine と同等機能を持つ Go ネイティブなルールエンジン OSS が GA | cgo 不要になる | 2026-04 時点で該当 OSS なし ([`03_ルールエンジン.md`](./03_ルールエンジン.md) 参照) |

### 8.2 Rust 単一案が有利になる条件

| 条件 | 効果 | 現実性 |
|---|---|---|
| Dapr Rust SDK が Stable に到達 | ファサードの追従コストが Go と同等に | CNCF ロードマップ上 2027 以降 |
| JTC 内に Rust 専任 10 名以上のチーム形成 | 要員確保遅延リスクが解消 | k1s0 のバス係数 2〜3 名体制と整合しない |

### 8.3 結論

いずれのシナリオも 2026-04 時点では成立確率が低く、ハイブリッド案が優位である根拠は 5 年スパンで揺らがない。Dapr Rust SDK Stable 到達 (Phase 3〜4 想定) の時点で再評価するトリガーは [`../03_tier1設計/02_内部言語ハイブリッド.md`](../03_tier1設計/02_内部言語ハイブリッド.md) 第 7 章に明記済み。

## 9. 決裁者向け答弁の 3 行要約

> tier1 内部を Go 一本にすると、自作領域 (ZEN Engine / 監査 / PII) で cgo 境界のオーバーヘッドが発生し、5 年で約 1,500 万円のコスト増になります。Rust 一本にすると Dapr Rust SDK が Alpha のため追従コストと要員確保遅延で約 3,700 万円のコスト増になります。**Daprファサード = Go、自作領域 = Rust のハイブリッド** は 5 年累計で最安の 1,800 万円に収まり、JTC の人材確保制約とも整合します。

## 10. 本分析の不確実性

以下の数値は公開情報・業界相場・自社経験則からの推定であり、実装着手時点で実測値への更新が必要となる。

- SES 単価 / 市場規模は 2025 年水準。Rust 人材市場は年率 20% 拡大中で 2028 年には 1,800 名規模に到達する可能性
- Dapr Rust SDK の Stable 到達は CNCF コミュニティの動向次第
- Pod 台数は ZEN Engine のスループット実測 (MVP 時点) で要更新
- 本番インシデント混入率は業界平均値 (Microsoft / Google / Cloudflare 統計) に依拠、JTC 内製の実績は蓄積不足

Phase 1b 完了時に自作領域 3 コンポーネントの実行時間を実測し、本分析の 2.1 / 2.2 / 2.3 節の数値を更新する。
