# フェーズ別 OSS 採用タイムライン

## 目的

[`04_選定一覧.md`](./04_選定一覧.md) が「何を採用するか」を責務別にまとめた静的インデックスであるのに対し、本資料は「いつ導入するか」を時系列で示す。Phase 0 (企画承認) から Phase 3 (全社展開) まで、各 OSS がどのフェーズで初期導入され、どのフェーズで本番運用水準に達するかを一枚で俯瞰できるようにする。これにより「Phase 1a の VM 1 台で本当にデモできるのか」「Phase 1b に進むには何が増えるのか」を稟議の場で即答できる。

個々の OSS の選定根拠は [`04_選定一覧.md`](./04_選定一覧.md) からリンクされる詳細ファイル (`01_実行基盤中核OSS.md` / `02_周辺OSS.md` / `03_ルールエンジン.md` / `07_追加採用OSS.md` 以降) を参照すること。本資料はタイムラインの一元インデックスに徹する。

---

## 1. フェーズ区分の前提

[`../07_ロードマップと体制/00_フェーズ計画.md`](../07_ロードマップと体制/00_フェーズ計画.md) のフェーズ定義と整合させる。各フェーズで「初期導入」「本番水準」「保留」のいずれかに分類する。

- **初期導入 (Intro)**: 稼働するが冗長化や運用手順は最低限。検証利用が主。
- **本番水準 (Prod)**: SLO 準拠・Runbook 整備済み・バックアップ検証済み。JTC 本番業務で依存して良い状態。
- **保留 (Hold)**: 採用確定だが導入しない。後続フェーズで順次解除。

Phase 1a の VM 1 台制約では「本番水準」に到達する OSS は存在しない。Phase 1b 以降で順次昇格させる。

## 3. フェーズ別の累計 OSS 数

各フェーズに到達した時点で、「本番水準」に達している OSS の累計数。稟議の場で「このフェーズで何が動いているか」の規模感を即答するための指標。

| フェーズ | 本番水準 OSS 数 | Intro 段階 OSS 数 | 主な新規追加 |
|---|---|---|---|
| Phase 1a (MVP-0) | 0 | 11 | k8s / Dapr / Go / Prometheus / Grafana / OTel / Keycloak / CNPG / OpenTofu / Tilt / GHA runner |
| Phase 1b (MVP-1a) | 0 | 26 | Istio / Kafka / Rust / Loki / Tempo / Argo CD / Harbor / Valkey / ZEN Engine / Kyverno / flagd / Longhorn / MetalLB / kube-vip / Cosign/Syft/Grype |
| Phase 1c (MVP-1b) | 36 | 6 | Backstage / MinIO / Litmus / Renovate / Argo Events / pgvector / pg_partman / OpenBao / Argo Rollouts / Temporal / KEDA / ESO / Kubeshark (on-demand) |
| Phase 2 | 42 | 2 | Pyroscope / Tekton |
| Phase 3 | 44 | 0 | 残存する Intro の本番昇格のみ |

Phase 1c での本番 OSS 数 36 は「JTC パイロット業務 1 つを運用できる最低水準」であり、Phase 2 への判定の前提となる数値。Phase 1b までの 0 件は「動くが運用条件を満たさない」状態であることを明示する。

## 4. フェーズ間で必ず検証すること

各フェーズに進む前のゲート条件として、以下の検証を実施する。[`../07_ロードマップと体制/02_体制と役割.md`](../07_ロードマップと体制/02_体制と役割.md) の KPI と突き合わせる。

### 4.1 Phase 1a → Phase 1b ゲート

- MVP-0 デモ (起案者単独、VM 1 台) が決裁者に対して完走すること
- VM 3 台 + 協力者 1 名のアサイン合意が取れていること
- Phase 1b で新規導入する 15 OSS の Helm Chart / Operator 版数を確定していること (Renovate 未導入のため、手動版数ロック)

### 4.2 Phase 1b → Phase 1c ゲート

- バス係数 2 の実証 (協力者が単独で環境再構築できること)
- tier1 公開 API の 5 主要 API (`Log` / `Telemetry` / `State` / `PubSub` / `Service.Invoke`) が Phase 1b で動作すること
- パイロット業務 1 本が配信ポータル経由で稼働していること
- SBOM 生成率 50% 以上

### 4.3 Phase 1c → Phase 2 ゲート

- 本番水準 OSS が 36 件揃っていること (本資料第 3 節)
- Runbook 充足率 15/15 (tier1 主要障害シナリオ全件)
- フルリストア訓練が協力者単独で完了していること (起案者不在)
- ライセンス自動監査 CI が有効化されていること ([`../09_法務とコンプライアンス/00_OSS法務対応.md`](../09_法務とコンプライアンス/00_OSS法務対応.md) 第 4 節)

### 4.4 Phase 2 → Phase 3 ゲート

- 全フェーズ共通 KPI がすべて「必達」水準 ([`../07_ロードマップと体制/02_体制と役割.md`](../07_ロードマップと体制/02_体制と役割.md) 第 6 節)
- システム基盤チーム以外のチームが独立して新規サービスをリリースできること (tier1 依存がブラックボックス化できる水準)

## 5. 本資料の限界

- OSS 版数は本資料で管理しない。[`17_OSS長期戦略.md`](./17_OSS長期戦略.md) と Renovate / Dependabot の自動更新結果を信頼する
- 採用判断が変わった場合 (例: Kafka → NATS の切り替え検討) は [`17_OSS長期戦略.md`](./17_OSS長期戦略.md) の差し替えシナリオに沿って追加の RFC が必要
- Phase 境界は目安であり、KPI 未達時は前フェーズに留まることもあり得る
