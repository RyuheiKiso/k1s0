# 分散トレーシングとパーティション管理 OSS の選定

## 目的

分散トレーシングバックエンドの最終選定 (Grafana Tempo) と、PostgreSQL 大規模テーブルの自動パーティション管理 (pg_partman) を整理する。前者は観測性スタック ([`09_ネットワークとテレメトリ基盤.md`](./09_ネットワークとテレメトリ基盤.md) のテレメトリパイプラインの受け手) を完成させ、後者は監査ログ / イベントログの長期保持戦略を支える。対象カテゴリは以下のとおり。

- W. 分散トレーシングバックエンド
- X. テーブルパーティション管理

---

## W. 分散トレーシングバックエンド

| 候補 | 採否 | 評価 |
|---|---|---|
| **Grafana Tempo** | 採用 | Grafana Labs 開発。OTLP ネイティブ。オブジェクトストレージ (S3 互換) をバックエンドに使用し、既存の MinIO を活用可能。Loki / Prometheus / Grafana と同一エコシステムで LGTM スタックを完成させる |
| Jaeger | 不採用 (変更) | CNCF Graduated。実績豊富だが、バックエンドに Elasticsearch / Cassandra が必要であり、2 名体制での追加コンポーネント運用が困難。Red Hat OpenShift が 2025 年 11 月に Jaeger サポートを終了し Tempo 移行を公式推奨している |
| Zipkin | 却下 | Jaeger 同様にバックエンドストレージの運用負荷が高い。コミュニティ規模も Jaeger に劣る |
| SigNoz | 却下 | Apache 2.0 で機能豊富だが、独自の UI/分析スタックを持ち Grafana エコシステムと重複する |

### 旧選定 (Jaeger) からの変更理由

初期選定では Jaeger を「CNCF Graduated の信頼性」と「実績」を根拠に採用していた。以下の理由で Grafana Tempo に変更する。

1. **バックエンドストレージの運用負荷** — Jaeger の本番運用には Elasticsearch または Cassandra が必要であり、これら自体が JVM ベースの重量級コンポーネントである。2 名体制で新たな JVM 系コンポーネントを追加運用することは現実的でない。Tempo は既に採用済みの MinIO (S3 互換) をそのままバックエンドに使用できる
2. **LGTM スタックの統一** — Loki (ログ) / Grafana (可視化) / Prometheus (メトリクス) は既に採用済み。Tempo を加えることで Grafana エコシステムの LGTM スタックが完成し、Logs → Traces → Metrics の三方向相互リンクがネイティブに実現する。Jaeger は Grafana との統合がプラグイン経由であり、統合品質が劣る
3. **Red Hat の公式判断** — Red Hat OpenShift は 2025 年 11 月 3 日で Jaeger のサポートを終了し、Tempo Operator への移行を公式推奨している。エンタープライズ市場における方向性が Tempo に傾いている
4. **ストレージコスト** — Tempo はトレースデータを Parquet 形式でオブジェクトストレージに圧縮保存する。Jaeger + Elasticsearch はインデックス + レプリカで同じデータ量に対し 5〜10 倍のディスクを消費する。オンプレ 3 ノード環境ではこの差は致命的
5. **メモリ消費** — Tempo monolithic モードは 0.5〜1 GB で動作する。Jaeger Collector + Query + Elasticsearch (3 ノード推奨) は合計 4〜8 GB 以上を消費する

### 採用理由 (Grafana Tempo)

1. **MinIO (既存) をバックエンドにそのまま利用** — 追加のストレージ基盤が不要。Tempo はオブジェクトストレージ (S3 互換) をネイティブバックエンドとして使用するため、既に採用済みの MinIO で完結する
2. **OTLP ネイティブデータモデル** — Tempo は設計時から OTLP をネイティブデータ形式として採用しており、OTel Collector から受信したトレースデータをそのまま Parquet 形式で保存する。データモデル変換のオーバーヘッドがゼロ
3. **Grafana との三方向相互リンク** — Loki のログに含まれる TraceID クリックで Tempo のトレースにジャンプ、Tempo のスパンから Prometheus の Exemplars にリンク、Prometheus のグラフから該当トレースへジャンプ。全てネイティブ機能として標準提供
4. **TraceQL による高度なトレース検索** — Tempo 独自のクエリ言語 TraceQL で、スパン属性・期間・エラー状態を組み合わせた検索が Grafana Explore 画面から直接実行可能
5. **運用コンポーネントが少ない** — monolithic モード (単一バイナリ) で起動可能。スケール時は microservices モードに切替。Jaeger + Elasticsearch のような複数コンポーネント管理が不要
6. **OTel Collector との整合** — 既に MVP-1a で導入する OTel Collector Agent (DaemonSet) から、OTLP exporter でそのまま Tempo に送信できる。Jaeger 専用 exporter への切替が不要

### ライセンスについて

Grafana Tempo のライセンスは AGPL-3.0 である。k1s0 の選定方針は「OSI 承認済みライセンス」であり、AGPL-3.0 は OSI 承認である。k1s0 はプラットフォーム内部利用 (SaaS 提供しない) のため、AGPL-3.0 のコピーレフト伝播条件に該当しない。また Loki / Grafana / MinIO / Renovate で既に AGPL-3.0 を受容済みであり、ライセンス方針との矛盾はない。

### Jaeger v2 との比較

2024 年 11 月に Jaeger v2 がリリースされ、OpenTelemetry Collector をコアに統合する大幅な刷新が行われた。OTLP ネイティブ対応により OTel 互換性は改善されたが、**ストレージバックエンド (Elasticsearch / Cassandra / Badger) の問題は v2 でも変わっていない**。k1s0 にとっての最大のデメリット (追加ストレージ基盤の運用負荷) が未解消であるため、v2 を考慮しても Tempo の優位性は変わらない。

### デプロイモデル

| モード | 構成 | 用途 | 導入時期 |
|---|---|---|---|
| Monolithic | 単一 Deployment (1〜2 レプリカ) | MVP 規模。単一バイナリで distributor / ingester / querier / compactor を内包 | Phase 2 |
| Microservices | 各コンポーネントを個別 Deployment | Phase 3 以降。スケールが必要になった場合 | Phase 3 |

### テレメトリ経路 (Phase 2)

```
サービス Pod (OTel SDK)
    ↓ OTLP (gRPC)
OTel Collector Agent (DaemonSet)
    ↓ OTLP (gRPC)
OTel Collector Gateway (Deployment)
    ├──→ Prometheus (メトリクス)
    ├──→ Loki (ログ)
    └──→ Tempo (トレース) ──→ MinIO (S3 バックエンド)
```

### Grafana での統合

| 連携方向 | 仕組み | 効果 |
|---|---|---|
| Loki → Tempo | ログ内の `traceID` フィールドからトレースにジャンプ | エラーログから即座にリクエスト全体を追跡 |
| Tempo → Prometheus | スパンから関連メトリクスへリンク (Exemplars) | トレースからシステム全体の負荷状況を確認 |
| Prometheus → Tempo | メトリクスグラフ上の Exemplar ポイントからトレースにジャンプ | レイテンシスパイクの原因リクエストを特定 |

### メモリ概算

| 構成 | メモリ |
|---|---|
| Tempo monolithic (1 レプリカ) | 0.5 GB |
| Tempo monolithic (2 レプリカ HA) | 1.0 GB |

Jaeger + Elasticsearch (4〜8 GB) と比較して大幅に軽量。

### MVP スコープ

- **Phase 2 で導入** (OTel Collector Gateway 導入と同時)
- `infra` namespace にデプロイ (Deployment、monolithic モード)
- バックエンド: MinIO (既存) の `tempo` バケット
- Receiver: OTLP (gRPC / HTTP)
- Grafana に Tempo データソースを追加
- Loki → Tempo のトレースリンクを設定
- Phase 2 ではサンプリング戦略として head-based sampling (OTel Collector Gateway) を採用
- Phase 3 で tail-based sampling への移行を検討

### トレードオフ

- CNCF プロジェクトではない (Grafana Labs 主導)。ただし Grafana Labs は Loki / Grafana / Mimir / Alloy 等の主要 OSS を継続的にメンテナンスしており、実質的な継続性リスクは低い
- 単体 UI を持たない (Grafana が必須)。k1s0 は Grafana 採用済みのため影響なし
- Jaeger (2017年〜) と比較して歴史が短い (2020年〜)。ただし急速に普及しており、Red Hat の公式採用で信頼性は十分

## X. テーブルパーティション管理 (PostgreSQL)

| 候補 | 採否 | 評価 |
|---|---|---|
| **pg_partman** | 採用 | PostgreSQL 拡張。時間ベースの自動パーティション作成・保持・アーカイブを提供。`CREATE EXTENSION` で有効化するだけで追加インフラ不要 |
| 手動パーティション管理 | 却下 | `CREATE TABLE ... PARTITION BY` で手動作成は可能だが、パーティションの自動作成・保持期間管理・アーカイブを自前で実装する必要がある |
| TimescaleDB | 却下 | 時系列データに特化した PostgreSQL 拡張だが、ライセンスが Timescale License (非 OSI) に変更されており選定方針に反する |
| パーティションなし | 却下 | 監査ログの保持期間 7 年 (法的要件) で数億行に達し、クエリ性能が劣化する |

### 採用理由 (pg_partman)

1. **監査ログの 7 年保持要件への対応** — [`../02_アーキテクチャ/04_非機能とデータ/02_データアーキテクチャ.md`](../02_アーキテクチャ/04_非機能とデータ/02_データアーキテクチャ.md) で監査ログの保持期間は 7 年 (法的要件) と定義されている。単一テーブルでは数億行に達し、INSERT 性能・SELECT 性能・VACUUM 性能の全てが劣化する。pg_partman の時間ベースパーティショニングで月次 (または日次) に自動分割し、クエリ性能を維持する
2. **既存インフラへの追加のみ** — CloudNativePG の PostgreSQL インスタンスに `CREATE EXTENSION pg_partman;` を実行するだけで利用可能。追加の DB サーバー・Operator・ストレージが不要。pgvector と同様の導入パターン
3. **パーティションのライフサイクル自動管理** — 新しいパーティションの事前作成 (pre-creation)、古いパーティションの保持期間管理 (retention)、アーカイブテーブルスペースへの移動を自動化する。手動パーティション管理の運用負荷をゼロにする
4. **バックアップ・リストアとの整合** — CloudNativePG の barman-cloud バックアップにパーティションテーブルもそのまま含まれる。パーティション単位での `pg_dump` によるアーカイブも可能
5. **PostgreSQL License (OSI 承認)** — 選定方針に完全に適合

### 対象テーブル

| テーブル | DB | パーティションキー | 分割単位 | 保持期間 | 導入時期 |
|---|---|---|---|---|---|
| `audit_logs` | `k1s0_audit` | `created_at` (timestamp) | 月次 | 7 年 (84 パーティション) | Phase 2 |
| tier2 業務ログ系テーブル | 各 tier2 DB | `created_at` (timestamp) | 月次 | サービス定義 | Phase 3 |

### パーティション戦略

| 設定項目 | 値 | 理由 |
|---|---|---|
| パーティション方式 | `RANGE` (時間ベース) | 監査ログは時系列データ。範囲パーティションが最適 |
| 分割単位 | 月次 (`monthly`) | 日次では 7 年で 2,555 パーティション (過多)。月次なら 84 パーティション |
| 事前作成数 | 3 パーティション先行作成 | パーティション未作成による INSERT エラーを防止 |
| 保持期間 | 84 ヶ月 (7 年) | 法的要件。保持期間経過後は pg_partman が自動 detach |
| アーカイブ | detach 後に別テーブルスペースへ移動 (Phase 3) | アクティブストレージの容量を節約 |

### pg_partman の自動メンテナンス

pg_partman は `pg_partman_bgw` (Background Worker) または `pg_cron` 経由で定期的にメンテナンス関数を実行する。

```
pg_partman_bgw (Background Worker)
    ↓ 定期実行 (デフォルト: 3600 秒ごと)
run_maintenance()
    ├── 新パーティションの事前作成
    ├── 保持期間超過パーティションの detach
    └── detach 済みパーティションの drop (設定による)
```

### CloudNativePG との統合

CloudNativePG のカスタム PostgreSQL イメージに pg_partman を含める。コミュニティが提供する PostGIS / pgvector 同梱イメージと同様に、pg_partman 同梱イメージを Dockerfile で構築し Harbor にホストする。

```
CloudNativePG Cluster CRD
    └── imageName: harbor.k1s0.local/infra/postgresql:16-pgvector-pgpartman
```

### MVP スコープ

- **Phase 2 で導入** (tier1 Rust サービスの本格実装と同時)
- CloudNativePG の PostgreSQL イメージに pg_partman 拡張を追加
- `k1s0_audit` DB の `audit_logs` テーブルに月次パーティションを適用
- `pg_partman_bgw` を有効化し、パーティションの自動作成・保持管理を開始
- Phase 3 でアーカイブテーブルスペース (別ストレージ) への移動を実装

### トレードオフ

- カスタム PostgreSQL イメージのビルド・保守が必要。ただし pgvector でも同様の対応が必要であり、追加の運用コストは最小限
- pg_partman_bgw が Background Worker スロットを 1 つ消費する。MVP 規模では影響なし
- 既存のパーティションなしテーブルからパーティションテーブルへのマイグレーションは一括ではなく段階的に行う必要がある。Phase 2 で新規作成するテーブルから適用し、既存データのマイグレーションは Phase 3 で実施
