## Y. マニフェスト管理

### 背景: なぜ今決定が必要か

Argo CD を GitOps エンジンとして採用済みだが、**k8s マニフェストの生成・管理・テンプレート戦略**が未定義であった。Argo CD は「何をデプロイするか」を担うが、「マニフェストをどう構成するか」は別の問題であり、MVP-0 着手前に確定する必要がある。

### 採用方針: Kustomize を主軸、Helm はサードパーティ Chart 用

| ツール | 採否 | 役割 |
|---|---|---|
| **Kustomize** | 採用 (主軸) | k1s0 自製マニフェスト (tier1 / tier2 / tier3) の環境別差分管理 |
| **Helm** | 採用 (サードパーティ用) | 外部 OSS のインストール (Strimzi / CloudNativePG / Keycloak / Istio 等) |
| Jsonnet | 却下 | 独自言語の学習コストが JTC の 2 名体制に対して過大 |
| Timoni (CUE) | 却下 | CUE 言語のエコシステムが未成熟。Kustomize の方が学習リソースが豊富 |
| Helm のみ | 却下 | Go テンプレート構文が複雑であり、自製マニフェストに適用すると可読性が著しく低下する |
| Kustomize のみ | 却下 | サードパーティ OSS の多くが Helm Chart のみで配布されているため、Helm の併用が現実的に不可避 |

### 採用理由 (Kustomize)

1. **kubectl 組み込み** — 追加インストール不要。`kubectl apply -k` で即座に利用可能。JTC の学習コストを最小化する
2. **YAML のまま差分を記述** — テンプレート言語を使わず、patch / overlay で環境別の差分のみを記述する。YAML の可読性を維持したまま dev / staging / prod の差分を管理できる
3. **Argo CD のネイティブサポート** — Argo CD は Kustomize を第一級でサポートしており、`kustomization.yaml` を自動検出して適用する。追加設定不要
4. **雛形生成 CLI との統合** — 雛形生成 CLI が `deploy/base/` と `deploy/overlays/*/kustomization.yaml` を自動生成する設計が既に確定済み。Kustomize の採用で設計が一貫する
5. **tier 依存ルールのマニフェストレベル強制** — overlay 内で namespace / ラベル / annotation を一括制御できるため、tier2 が `infra` namespace にデプロイされる誤設定を構造的に防止できる

### 採用理由 (Helm)

1. **サードパーティ OSS の配布形態** — Strimzi / CloudNativePG / Keycloak / Istio / cert-manager / Argo CD / Harbor / MetalLB / Longhorn / Kyverno 等、k1s0 が採用する主要 OSS の大半が Helm Chart で配布されている。Helm なしでは手動でマニフェストを展開・保守する必要があり、運用コストが増大する
2. **Argo CD のネイティブサポート** — Argo CD は Helm Chart を直接デプロイ可能。`helm template` でマニフェストを生成し、GitOps の差分管理に統合される
3. **Renovate との統合** — Renovate は Helm Chart のバージョン更新を自動検知し、`Chart.yaml` の更新 PR を生成する。CVE Critical 48 時間対応の自動化に不可欠

### 使い分けルール

| 対象 | ツール | 理由 |
|---|---|---|
| tier1 / tier2 / tier3 自製サービス | **Kustomize** | 自製マニフェストは YAML の可読性を維持する |
| サードパーティ OSS (Helm Chart 配布) | **Helm** | Chart がそのまま利用可能 |
| サードパーティ OSS + カスタム設定 | **Helm + Kustomize** | Helm Chart をベースに、Kustomize の `helmCharts` ジェネレータまたは post-rendering で追加の overlay を適用 |

### ディレクトリ構成

```
deploy/
├── base/                     # Kustomize base (雛形生成 CLI が生成)
│   ├── deployment.yaml
│   ├── service.yaml
│   ├── kustomization.yaml
│   └── dapr-component.yaml
├── overlays/
│   ├── dev/
│   │   └── kustomization.yaml   # dev 環境用 patch
│   ├── staging/
│   │   └── kustomization.yaml
│   └── prod/
│       └── kustomization.yaml
infra/
├── helm-values/              # サードパーティ Helm Chart の values
│   ├── cloudnative-pg.yaml
│   ├── keycloak.yaml
│   ├── strimzi.yaml
│   └── ...
```

### Argo CD との統合

| Argo CD Application の種別 | マニフェストソース | 備考 |
|---|---|---|
| tier1 / tier2 / tier3 自製サービス | Kustomize (`deploy/overlays/{env}/`) | ApplicationSet で tier × 環境のマトリクスを自動生成 |
| infra サードパーティ OSS | Helm Chart (values は Git 管理) | Argo CD が `helm template` を実行し差分管理 |

### ライセンス

| ツール | ライセンス | 備考 |
|---|---|---|
| Kustomize | Apache 2.0 | kubernetes-sigs プロジェクト。kubectl 組み込み |
| Helm | Apache 2.0 | CNCF Graduated プロジェクト |

### MVP スコープ

- **MVP-0 から導入** — Argo CD の前提としてマニフェスト管理が必要であり、MVP-0 の時点で Kustomize + Helm を使用開始する
- MVP-0: Kustomize base/overlay で tier1 + 配信ポータルのマニフェスト管理。Helm で Keycloak / PostgreSQL をインストール
- MVP-1a: Argo CD が Kustomize overlay を直接参照。雛形生成 CLI が `deploy/` ディレクトリを自動生成。Helm で CloudNativePG / cert-manager / MetalLB / Longhorn 等をインストール
- MVP-1b: Helm で Harbor / Kyverno をインストール。Kustomize overlay で Kyverno ClusterPolicy を環境別に管理

### トレードオフ

- Kustomize と Helm の 2 つのツールを学習する必要がある。ただし Kustomize は YAML patch のみ、Helm はサードパーティ Chart の values 設定のみであり、学習範囲は限定的
- Kustomize の strategic merge patch は意図しないマージが起きうる。JSON patch (RFC 6902) をデフォルトとし、strategic merge patch は明示的に選択する運用とする
- Helm Chart のバージョンアップ時に破壊的変更が入る可能性がある。Renovate が検知した PR で CI テストを実行し、安全性を確認する

### Z.1 観点別の落選理由

#### Cadence (Temporal の前身)

Temporal は Cadence のメイン開発者が独立して立ち上げたフォークであり、両者は思想的に同根。Cadence は Uber 内部で運用が継続されているが、コミュニティへの新機能リリース頻度は **Temporal 比で約 1/3** に低下している (2024〜2025 年の release notes 比)。PostgreSQL バックエンドが公式サポートされていない点も、k1s0 の CloudNativePG 統合方針 ([`../02_アーキテクチャ/09_データアーキテクチャ.md`](../02_アーキテクチャ/09_データアーキテクチャ.md)) との整合で不利。同等機能で活発な後継 (Temporal) が存在するため Cadence を選ぶ理由がない。

#### Camunda 8 (Zeebe)

機能・UI 成熟度では Temporal を上回る面もあるが、ライセンスが **Camunda License v1.0** (Source Available) で OSI 承認外。k1s0 選定方針「OSI 承認済み OSS のみ採用」に違反するため、技術評価以前に対象外。Camunda 7 (Apache 2.0) は 2025-04 EOL で代替不可。

#### Argo Workflows / Apache Airflow / Prefect

これらは「DAG ベースのジョブ実行エンジン」であり、k1s0 が必要とする「**長期実行 + 状態永続化 + 補償トランザクション**」とは別カテゴリ。Argo Workflows は k8s Job ベースのため、稟議承認のような数日〜数週間の Workflow は Pod 寿命やリソース枯渇で破綻する。Airflow / Prefect はデータパイプライン向け設計であり、マイクロサービス間の Saga には不適。役割が異なるため「採用すべきだが Temporal と用途が違う」という判定。

#### Netflix Conductor

過去の選択肢としては有力だったが、Netflix 自身が後継エンジン Maestro へ内部移行したことでメイン開発主体が縮小。コミット頻度・Issue 応答性が低下し、長期 10 年運用には不安。GitHub stars (12.5k) は Temporal と同等だが「活動量」が逆転している。

#### AWS Step Functions

**JTC オンプレ完結の前提と根本的に非互換**。k1s0 はオンプレ稟議文化を主戦場としており、AWS lock-in は採用要件を満たさない。

#### n8n / Activepieces

業務担当向けの iPaaS (ノーコードフロー自動化) であり、開発者が記述する Saga パターンには設計思想が合わない。**Sustainable Use License** は OSI 非承認であり選定方針違反。役割としては将来 tier2 業務サービスでのデータ統合用途で別途検討する余地があるが、tier1 ワークフローエンジンとしては不適。

#### Dapr Workflow (旧採用からの変更)

旧設計では Dapr 隠蔽方針との整合を重視して採用していた。変更理由の詳細は次節以降参照。一言で言えば「Saga 表現力 / 運用可視性 / 長期実行実績の 3 点で Temporal が圧倒的に優位」「**tier2 / tier3 から見た公開 API は変わらない** ため Dapr 隠蔽方針の本質は維持される」。

### Z.2 結論

7 観点を同時に満たす候補は Temporal のみ。Camunda 8 は OSI 違反、Cadence は活動量で劣後、Argo/Airflow/Prefect は用途違い、Step Functions はオンプレ不可、n8n はライセンス違反、Conductor は縮小傾向。Saga パターン (補償トランザクション) を SDK レベルでネイティブ表現でき、年単位の長期実行実績を持ち、PostgreSQL バックエンドで CloudNativePG 共有クラスタに収まる、という 3 条件の交点に存在するのが Temporal だけである。

### Z.3 旧選定 (Dapr Workflow) からの変更理由

初期設計では Dapr Workflow を「Dapr 隠蔽方針に基づき、Workflow も Dapr building block で統一」する根拠で採用していた。以下の理由で Temporal に変更する。

1. **Saga パターンの表現力** — Dapr Workflow は基本的なワークフロー (順次・並行・分岐) をサポートするが、**補償トランザクション (Saga の Compensate)** の表現が SDK レベルで限定的である。Temporal は `workflow.Go` / `workflow.ExecuteActivity` + `defer compensation()` の形で Saga パターンを自然に記述でき、補償順序の制御・部分ロールバック・タイムアウトベースの補償をネイティブにサポートする
2. **ワークフローの可視性** — Temporal は Web UI でワークフローの実行状態・履歴・ペンディングアクティビティ・リトライ回数をリアルタイムで可視化する。Dapr Workflow はダッシュボードを持たず、ワークフローの状態確認が Dapr API 経由のクエリに限定される。JTC の運用チームにとって GUI ベースの状態確認は必須
3. **長期実行ワークフロー** — JTC のユースケースでは、稟議承認 (数日〜数週間)、月次バッチ、年次データアーカイブなど**長期実行ワークフロー**が想定される。Temporal は設計上「ワークフローは任意の期間実行可能」であり、状態の永続化とリプレイが基盤に組み込まれている。Dapr Workflow も状態永続化を行うが、長期実行に対する実運用実績は Temporal に大きく劣る
4. **SDK の成熟度** — Temporal Go SDK は Production Ready であり、大規模プロダクション環境 (Uber, Netflix, Snap, Stripe, Datadog 等) で実績がある。Dapr Workflow は Stable だが、SDK の機能範囲 (シグナル / クエリ / 子ワークフロー / バージョニング) で Temporal に及ばない
5. **PostgreSQL バックエンド** — Temporal Server は PostgreSQL をネイティブにサポートしており、既存の CloudNativePG クラスタにデータベースを追加するだけで運用可能。追加のストレージ基盤が不要

### Dapr 隠蔽方針との整合

Dapr 隠蔽方針 ([`../03_tier1設計/01_Dapr隠蔽方針.md`](../03_tier1設計/01_Dapr隠蔽方針.md)) の基本原則は「Dapr で賄えるならまず Dapr」である。Workflow については以下の判断で Dapr の範囲外とする。

| 観点 | Dapr Workflow | Temporal | 判断 |
|---|---|---|---|
| Saga 表現力 | 基本的な順次・並行・分岐 | 補償トランザクション・子ワークフロー・シグナル・クエリの完全サポート | **Temporal が優位** |
| 運用可視性 | API クエリのみ | Web UI で全ワークフロー状態を可視化 | **Temporal が優位** |
| 長期実行実績 | 限定的 | Uber / Netflix 等の大規模実績 | **Temporal が優位** |
| Dapr エコシステム統合 | ネイティブ | 外部サービス (tier1 内部で吸収) | Dapr が優位 |
| 追加コンポーネント | なし (Dapr CP に含まれる) | Temporal Server のデプロイが必要 | Dapr が優位 |

Dapr Workflow は State / PubSub / Secrets 等と比較して「Dapr で賄える」レベルに達していないと判断する。ただし、**tier2 / tier3 から見た公開 API (`k1s0.Workflow.*`) は変わらない**。内部実装の差し替えは tier1 の責務であり、Dapr 隠蔽方針の「tier2 / tier3 は内部実装を意識しない」原則と完全に整合する。

### アーキテクチャ

```
tier2 / tier3
    ↓ k1s0.Workflow.* (tier1 公開 API)
tier1 Go ファサード
    ↓ Temporal Go SDK
Temporal Server (tier1 namespace)
    ↓ PostgreSQL (CloudNativePG)
```

### Temporal Server の構成

| コンポーネント | 役割 | レプリカ数 (MVP) |
|---|---|---|
| Frontend | gRPC API ゲートウェイ | 1 |
| History | ワークフロー状態管理 | 1 |
| Matching | タスクキューのマッチング | 1 |
| Worker | システムワークフロー実行 | 1 |
| Web UI | ワークフロー可視化ダッシュボード | 1 |

Phase 2 では全コンポーネントを単一 Deployment (all-in-one) モードで起動し、Phase 3 以降でコンポーネント分離を検討する。

### tier1 公開 API (`k1s0.Workflow.*`) の変更

公開 API のインターフェースは変更なし。内部実装のみ差し替える。

| API メソッド | 旧内部実装 | 新内部実装 |
|---|---|---|
| `Workflow.Start` | Go ファサード → Dapr Workflow | Go ファサード → Temporal Go SDK `client.ExecuteWorkflow` |
| `Workflow.GetStatus` | Go ファサード → Dapr Workflow | Go ファサード → Temporal Go SDK `client.DescribeWorkflowExecution` |
| `Workflow.Terminate` | Go ファサード → Dapr Workflow | Go ファサード → Temporal Go SDK `client.TerminateWorkflow` |
| `Workflow.Signal` (新規) | — | Go ファサード → Temporal Go SDK `client.SignalWorkflow` |
| `Workflow.Query` (新規) | — | Go ファサード → Temporal Go SDK `client.QueryWorkflow` |

`Signal` と `Query` は Temporal 採用によって新たに提供可能になる API であり、Phase 2 以降の段階的な公開を予定する。

### Dapr Workflow の残存用途

Dapr Workflow は完全に廃止するのではなく、**軽量なオーケストレーション (短期実行・シンプルな順次処理)** では引き続き利用可能とする。ただし、以下の使い分けルールを設ける。

| ユースケース | 選択 | 理由 |
|---|---|---|
| Saga パターン (補償トランザクション) | **Temporal** | 補償順序の制御が必須 |
| 長期実行 (数時間〜数日) | **Temporal** | 状態永続化とリプレイの信頼性 |
| 稟議承認フロー | **Temporal** | 人間の介入待ちが発生する長期ワークフロー |
| 単純な順次処理 (数秒〜数分) | Dapr Workflow | 追加コンポーネント不要で軽量 |
| tier1 内部の短期オーケストレーション | Dapr Workflow | Temporal Server への依存を増やさない |

この判断は tier1 チーム内で行い、tier2 / tier3 は `k1s0.Workflow.*` API を通じてのみアクセスするため、バックエンドの違いを意識しない。

### データベース

Temporal Server は PostgreSQL をバックエンドとして使用する。既存の CloudNativePG 共有クラスタに `temporal` データベースを追加する。

| DB 名 | 用途 | 導入時期 |
|---|---|---|
| `temporal` | ワークフロー状態・履歴・タスクキュー | Phase 2 |
| `temporal_visibility` | ワークフロー検索用 Visibility Store | Phase 2 |

### ライセンス

Temporal Server のライセンスは MIT である。k1s0 の選定方針「OSI 承認済みライセンス」に完全に適合する。MIT は最も許容的なライセンスの一つであり、商用利用・改変・再配布に制限がない。

### メモリ概算

| 構成 | メモリ |
|---|---|
| Temporal Server (all-in-one, 1 レプリカ) | 0.5 GB |
| Temporal Web UI (1 レプリカ) | 0.2 GB |
| **合計** | **0.7 GB** |

### MVP スコープ

- **Phase 2 で導入** — tier1 API `k1s0.Workflow` の本格実装と同時
- `tier1` namespace にデプロイ
- PostgreSQL バックエンド (CloudNativePG 共有クラスタに `temporal` / `temporal_visibility` DB を追加)
- Temporal Server all-in-one モード (単一 Deployment)
- Temporal Web UI を operation namespace に配置し Keycloak OIDC で SSO 統合
- tier1 Go ファサードが Temporal Go SDK を使用
- Helm Chart (`temporalio/temporal`) でインストール
- Phase 3 でコンポーネント分離・レプリカ増を検討

### トレードオフ

- Dapr エコシステムの外にコンポーネントが増える。ただし tier2 / tier3 からは tier1 公開 API の裏側として隠蔽されるため、エコシステム分断の影響は tier1 チーム内に閉じる
- Temporal Server の運用が必要になる。ただし PostgreSQL バックエンドは既存の CloudNativePG で管理されるため、追加の運用コストは Temporal Server 自体のアップデートに限定される
- CNCF プロジェクトではない。ただし Temporal Technologies 社が商用サポートとオープンソース版の両方を提供しており、MIT ライセンスのため最悪の場合でもフォークが可能
- Dapr Workflow との二重構成になる可能性がある。使い分けルールを明文化し、tier1 チームレビューで判断する
