# ネットワークとテレメトリ基盤の選定

## 目的

[`07_追加採用OSS.md`](./07_追加採用OSS.md) (L〜P) に続き、追加で採用が決定した OSS の選定根拠を整理する。対象カテゴリは以下のとおり。

- Q. ベアメタルロードバランシング
- R. Control Plane 高可用性 VIP
- S. テレメトリパイプライン

---

## Q. ベアメタルロードバランシング

| 候補 | 採否 | 評価 |
|---|---|---|
| **MetalLB** | 採用 | CNCF Sandbox。オンプレミス k8s で Service type=LoadBalancer を実現する唯一のデファクト。L2 / BGP 両モード対応 |
| kube-vip (LB モード) | 併用検討 | Control Plane VIP として採用する kube-vip にも LB 機能があるが、MetalLB の方が成熟度・事例・ドキュメントが豊富 |
| 外部ハードウェア LB | 却下 | 調達・稟議が必要で「無料で始められる」原則に反する |
| NodePort のみ | 却下 | ポート管理が煩雑。Envoy Gateway / Keycloak / Harbor / Grafana 等の外部公開エンドポイント増加に耐えられない |

### 採用理由 (MetalLB)

1. **オンプレミスで Service type=LoadBalancer を動作させる必須要件** — クラウドマネージド k8s を採用しない k1s0 では、cloud-provider の LB が存在しない。MetalLB なしでは Envoy Gateway / Keycloak / Grafana 等の外部公開に NodePort を強いられ、運用が破綻する
2. **L2 モードで MVP から即利用可能** — ARP ベースの L2 モードは BGP 環境不要で、既存の閉域ネットワークにそのまま投入できる。JTC のネットワーク管理者との調整が最小限で済む
3. **CNCF Sandbox プロジェクト** — コミュニティ活発。kubeadm / k3s いずれの構成でも動作確認済み
4. **Envoy Gateway との整合** — MetalLB が払い出す VIP を Envoy Gateway の Service に割り当てることで、外部 → クラスタの入口が一本化される

### L2 モードと BGP モードの使い分け

| モード | 用途 | 導入時期 |
|---|---|---|
| L2 (ARP) | MVP〜Phase 2。BGP 不要で即導入可能。フェイルオーバーは ARP GARP で数秒 | MVP-1a |
| BGP | Phase 3 以降。マルチクラスタ / 大規模環境でトラフィック分散が必要になった場合 | Phase 3 |

### MVP スコープ

- Phase 1 (MVP-1a) から導入
- `infra` namespace にデプロイ (speaker は DaemonSet、controller は Deployment)
- L2 モードで IP プール (5〜10 IP) を定義
- Envoy Gateway の Service type=LoadBalancer に VIP を割り当て

### トレードオフ

- L2 モードではリーダーノードに全トラフィックが集中する (単一ノードがボトルネック)。MVP の規模では問題にならない
- BGP モードへの移行は IPAddressPool と BGPPeer の追加のみで、アプリケーションへの影響はない

## R. Control Plane 高可用性 VIP

| 候補 | 採否 | 評価 |
|---|---|---|
| **kube-vip** | 採用 | 軽量。static Pod として Control Plane ノードで動作。VIP によるリーダー選出で API Server の HA を実現 |
| haproxy + keepalived | 次点 | 枯れた構成だが 2 つのコンポーネントの管理が必要。kubespray のデフォルト構成 |
| 外部ロードバランサ | 却下 | 追加のハードウェアまたは VM が必要。コスト増 |
| DNS ラウンドロビン | 却下 | フェイルオーバーが DNS TTL に依存し遅延する |

### 採用理由 (kube-vip)

1. **Control Plane HA の VIP を最小構成で実現** — 追加の VM やハードウェア LB が不要。static Pod 1 つで API Server の VIP を提供し、リーダー選出による自動フェイルオーバーを実現する
2. **haproxy + keepalived より運用が単純** — kubespray のデフォルト (haproxy + keepalived) は 2 コンポーネントの管理と設定が必要。kube-vip は単一バイナリ / 単一設定で完結する
3. **kubespray との統合実績** — kubespray は kube-vip を HA メカニズムの選択肢として公式サポートしている (`kube_vip_enabled: true`)
4. **MetalLB との役割分担が明確** — kube-vip は Control Plane VIP のみ、MetalLB は Service LB を担当。責務が分離され、障害時の切り分けが容易

### kube-vip と MetalLB の棲み分け

| コンポーネント | 対象 | 仕組み | 配置 |
|---|---|---|---|
| kube-vip | k8s API Server の VIP | ARP ベースのリーダー選出。Control Plane ノード間で VIP をフェイルオーバー | static Pod (各 Control Plane ノード) |
| MetalLB | Service type=LoadBalancer | ARP (L2) / BGP でクラスタ外部に VIP を広報 | DaemonSet (speaker) + Deployment (controller)、`infra` namespace |

両者はレイヤが異なる (kube-vip = Control Plane レベル、MetalLB = Service レベル)。kube-vip の LB 機能は使用せず、MetalLB に統一する。

### MVP スコープ

- Phase 1 (MVP-1a) から導入 (kubeadm 3 ノード HA 構成と同時)
- kubespray の `kube_vip_enabled: true` で自動構成
- VIP アドレス 1 つを Control Plane 用に割り当て
- kubeconfig の endpoint を VIP アドレスに設定

### トレードオフ

- kube-vip 自体が停止しても API Server は各ノードで直接アクセス可能。VIP が解決できなくなるだけで、kubelet や既存の Pod に即時影響はない
- haproxy + keepalived より実績は少ないが、kubespray の公式サポートと CNCF エコシステムでの採用実績で十分

## S. テレメトリパイプライン

| 候補 | 採否 | 評価 |
|---|---|---|
| **OpenTelemetry Collector** | 採用 | CNCF (OpenTelemetry プロジェクトの一部)。ベンダー中立のテレメトリパイプライン。受信・処理・エクスポートを統一的に扱える |
| 各サービスからバックエンドへ直接送信 | 却下 | バックエンド変更時に全サービスの再デプロイが必要。tier1 の「インフラ隠蔽」思想と矛盾する |
| Fluentd / Fluent Bit のみ | 次点 | ログ転送に特化。メトリクスとトレースは別途パイプラインが必要になり管理が分散する |
| Vector | 次点 | Rust 製で高性能だが、OTel エコシステムとの統合度で Collector に劣る |

### 採用理由 (OpenTelemetry Collector)

1. **テレメトリ経路を tier1 の抽象化層に統合** — tier1 は Dapr / Kafka / Valkey 等のインフラを tier2/tier3 から隠蔽する。テレメトリバックエンド (Prometheus / Loki / Grafana Tempo) も同様に隠蔽すべきであり、OTel Collector がその中継地点となる。バックエンド変更時にサービスの再デプロイが不要になる
2. **メトリクス・トレース・ログを単一パイプラインに統合** — 3 種のテレメトリシグナルを 1 つのコンポーネントで受信・加工・転送。Fluentd (ログ) + Prometheus (メトリクス) + トレースバックエンドを個別管理する必要がなくなる
3. **処理パイプラインで PII マスキング・属性付与が可能** — JTC 固有要件である個人情報マスキングや監査ログの属性統一を、Collector のプロセッサレベルで一元実施できる。各サービスのコードを変更する必要がない
4. **段階的なバックエンド追加が容易** — MVP-1a では Prometheus exporter のみで開始し、Phase 2 で Grafana Tempo exporter (OTLP) を追加するだけ。アプリケーション側の変更はゼロ
5. **CNCF OpenTelemetry プロジェクトの公式コンポーネント** — OTel SDK と Collector は同一プロジェクト内で整合性が保証される。tier1 が各言語向けに OTel SDK をラップする設計と完全に整合する

### デプロイモデル

| モード | 構成 | 用途 | 導入時期 |
|---|---|---|---|
| Agent (DaemonSet) | 各ノードに 1 Pod | ノードローカルでテレメトリを収集し、バッファリング・バッチ送信 | MVP-1a |
| Gateway (Deployment) | 集中 Pod (2 レプリカ) | Agent から受信し、tail-based sampling / PII マスキング等の高度な処理を実行してバックエンドに転送 | Phase 2 |

MVP-1a では Agent モードのみ。Agent が各ノードの Pod からメトリクスとログを収集し、Prometheus に転送する。Phase 2 で Gateway を追加し、トレース (Grafana Tempo) と高度な処理を実現する。

### テレメトリ経路 (MVP-1a)

```
サービス Pod (OTel SDK)
    ↓ OTLP (gRPC)
OTel Collector Agent (DaemonSet)
    ↓ remote write
Prometheus
    ↓
Grafana
```

### テレメトリ経路 (Phase 2)

```
サービス Pod (OTel SDK)
    ↓ OTLP (gRPC)
OTel Collector Agent (DaemonSet)
    ↓ OTLP (gRPC)
OTel Collector Gateway (Deployment)
    ├──→ Prometheus (メトリクス)
    ├──→ Loki (ログ)
    └──→ Grafana Tempo (トレース) ──→ MinIO (S3 バックエンド)
```

### tier1 公開 API との関係

tier1 の `k1s0.Telemetry.*` API は各言語向けに OTel SDK をラップしている。この SDK が送信先として OTel Collector Agent (ノードローカル) の OTLP エンドポイントを指す。tier2/tier3 開発者は以下を意識しない。

- OTel Collector の存在
- バックエンドの種類 (Prometheus / Grafana Tempo / Loki)
- バックエンドの接続先 / 認証情報

これにより、Phase 2 で Grafana Tempo を追加しても、Phase 3 以降でバックエンドを変更しても、tier2/tier3 のコードは一切変わらない。

### MVP スコープ

- Phase 1 (MVP-1a) から導入
- `infra` namespace にデプロイ (DaemonSet)
- Receiver: OTLP (gRPC / HTTP)
- Processor: batch / memory_limiter
- Exporter: prometheusremotewrite
- Phase 2 で Gateway モード追加、OTLP exporter (Grafana Tempo 向け) / Loki exporter / attributes processor (PII マスキング) を構成

### トレードオフ

- DaemonSet のため各ノードにメモリ消費が発生する (約 0.15 GB/ノード)。ただし個別の Fluentd / Prometheus agent を排除できるため、全体では増加しない
- MVP-1a 時点では Prometheus への転送のみで、Collector なしでも Prometheus の scrape で賄える。しかし Phase 2 での Grafana Tempo / Loki 追加時にサービス側の変更をゼロにするために、MVP-1a で経路を確立しておく価値がある
