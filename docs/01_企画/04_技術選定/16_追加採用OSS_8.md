# 追加採用 OSS の選定 (8)

## 目的

[`15_追加採用OSS_7.md`](./15_追加採用OSS_7.md) (AE〜AF) に続き、追加で採用が決定した OSS の選定根拠を整理する。対象カテゴリは以下のとおり。

- AG. Continuous Profiling (Grafana Pyroscope)

可観測性スタックの完成を目的とする。[`../02_アーキテクチャ/06_プラットフォーム自己監視.md`](../02_アーキテクチャ/06_プラットフォーム自己監視.md) の監視戦略を補強し、パフォーマンス問題の根本原因特定能力を飛躍的に向上させる。

## AG. Continuous Profiling

| 候補 | 採否 | 評価 |
|---|---|---|
| **Grafana Pyroscope** | 採用 | Grafana Labs 開発。マルチテナント対応の Continuous Profiling データベース。オブジェクトストレージ (S3 互換) をバックエンドに使用し、既存の MinIO を活用可能。Loki / Grafana / Tempo / Prometheus と同一エコシステムで可観測性スタックを完成させる |
| Parca | 次点 | Polar Signals 社開発。Apache 2.0。eBPF ベースのエージェントで言語非依存のプロファイリングを提供するが、Grafana エコシステムとの統合が外部プラグイン経由であり、Trace-to-Profile のネイティブ連携ができない |
| Elastic APM (Profiling) | 却下 | ライセンスが非 OSI (Elastic License 2.0 / SSPL)。k1s0 の選定方針に反する |
| 手動 pprof / perf | 不十分 | Go の `net/http/pprof` や Linux `perf` で個別プロファイリングは可能だが、常時収集・蓄積・検索・比較ができない。障害発生後に「そのときのプロファイル」を取得できない |

### 背景: なぜ Continuous Profiling が必要か

k1s0 の可観測性スタックは現在 **LGTM** (Loki / Grafana / Tempo / Mimir(Prometheus)) で構成されている。メトリクス・ログ・トレースの 3 本柱は揃っているが、**「なぜ遅いのか」「なぜ CPU / メモリを消費しているのか」** への回答能力が欠落している。

| 可観測性シグナル | 回答できる問い | 回答できない問い |
|---|---|---|
| メトリクス (Prometheus) | 「何が遅い？」(p99 レイテンシ) | 「なぜ遅い？」 |
| ログ (Loki) | 「何が起きた？」(エラーメッセージ) | 「どの関数で時間を消費した？」 |
| トレース (Tempo) | 「どのサービス間で遅延が発生した？」 | 「そのサービス内部のどの関数が原因？」 |
| **プロファイル (Pyroscope)** | **「どの関数が CPU / メモリを消費した？」** | — |

Continuous Profiling は上記 3 シグナルでは到達できない **コードレベルの根本原因** に到達する唯一の手段である。

### 具体的なユースケース

| シナリオ | 既存 LGTM のみ | LGTM + Pyroscope |
|---|---|---|
| tier1 Go ファサードの p99 レイテンシが SLO (500ms) を超過 | Tempo でスパン分解まで。「Dapr 呼び出しが遅い」で止まる | Trace-to-Profile で該当スパンの CPU フレームグラフを確認。JSON シリアライズの GC 圧力が原因と特定 |
| tier1 Rust サービス (ZEN Engine) のメモリ使用量が漸増 | Prometheus で `container_memory_working_set_bytes` の上昇を検知。原因不明 | Pyroscope のメモリプロファイルで特定の JDM 評価パスにアロケーションリークを特定 |
| デプロイ後に全体スループットが低下 | メトリクスで低下を確認、ログにエラーなし | デプロイ前後の CPU プロファイルを diff 比較。新規追加コードの O(n²) ループを特定 |
| エラーバジェットの急速消費 | バーンレートアラート発火。どのサービスの何が原因か不明 | Pyroscope でサービス別の CPU / メモリ消費推移を確認し、回帰を即座に特定 |

### 採用理由 (Grafana Pyroscope)

1. **LGTM スタックの自然な拡張 (LGTMP)** — Pyroscope は Grafana Mimir / Loki / Tempo と同一のアーキテクチャ設計思想で構築されており、MinIO (S3 互換) をバックエンドとする。既存の LGTM スタックに Pyroscope を加えることで可観測性の 4 本柱が完成し、メトリクス → トレース → プロファイルの三方向相互リンクがネイティブに実現する
2. **Trace-to-Profile 連携** — Grafana Tempo のトレースビューから直接 Pyroscope のフレームグラフにジャンプできる。`pyroscope.profile.id` 属性がスパンに自動付与され、「このリクエストの処理中にどの関数が CPU を消費していたか」をワンクリックで確認可能。[`11_追加採用OSS_3.md`](./11_追加採用OSS_3.md) で採用した Tempo との統合が完成する
3. **Go / Rust / C# の全 tier1-3 言語をサポート** — Go は `runtime/pprof` ネイティブ統合で本番環境でのオーバーヘッドが極めて低い (CPU 使用率 1〜3% 増)。Rust は `pprof-rs` クレート経由。C# は `dotnet-trace` / Pyroscope .NET SDK。tier2/tier3 の C# サービスでもプロファイリングが利用可能
4. **OpenTelemetry Profiling 統合** — Pyroscope は OpenTelemetry eBPF プロファイラをサポートし、アプリケーションコードの変更なしで CPU プロファイリングが可能。また OTel SDK との統合ライブラリ (`otel-profiling-go` 等) によりスパンプロファイルの自動収集を実現する
5. **MinIO (既存) をバックエンドにそのまま利用** — Tempo と同様、追加のストレージ基盤が不要。Pyroscope はオブジェクトストレージ (S3 互換) をネイティブバックエンドとして使用する
6. **monolithic / microservices デュアルモード** — Tempo と同じ運用モデル。Phase 2 では monolithic (単一 Deployment) で十分。Phase 3 以降でスケールが必要になった場合に microservices モードに移行可能
7. **デプロイ前後の diff 比較** — Pyroscope のフレームグラフ diff 機能により、デプロイ前後の CPU / メモリ消費パターンを視覚的に比較できる。パフォーマンス回帰の検出と原因特定が即座に可能

### Parca を選ばない理由

Parca は Apache 2.0 で技術的には健全だが、以下の理由で k1s0 では Pyroscope を優先する。

| 比較軸 | Pyroscope | Parca |
|---|---|---|
| Grafana 統合 | ネイティブデータソース。Traces-to-Profiles / Metrics-to-Profiles がビルトイン | 外部プラグイン経由。Trace 連携は追加設定が必要 |
| Tempo 連携 | `pyroscope.profile.id` スパン属性でワンクリック遷移 | Tempo との直接連携なし |
| ストレージ | MinIO (S3 互換) をそのまま利用。Tempo / Loki と統一 | 独自ストレージまたは S3。Grafana エコシステムとの運用統一感に劣る |
| アーキテクチャ | Mimir / Loki / Tempo と同一設計。運用パターンが統一 | 独自アーキテクチャ。運用ノウハウが別体系 |
| 言語別 SDK | Go / Rust / C# / Java / Python / Ruby / Node.js | eBPF エージェントが中心。SDK は Go のみ成熟 |
| コミュニティ規模 | GitHub Star 10k+。Grafana Labs がフルタイムで開発 | GitHub Star 4k+。Polar Signals 社が開発 |

### ライセンスについて

Grafana Pyroscope のライセンスは AGPL-3.0 (サーバー) / Apache 2.0 (クライアント SDK) である。k1s0 の選定方針は「OSI 承認済みライセンス」であり、AGPL-3.0 は OSI 承認である。k1s0 はプラットフォーム内部利用 (SaaS 提供しない) のため、AGPL-3.0 のコピーレフト伝播条件に該当しない。また Grafana / Grafana Tempo / Loki / MinIO / Renovate で既に AGPL-3.0 を受容済みであり、ライセンス方針との矛盾はない。

### デプロイモデル

| モード | 構成 | 用途 | 導入時期 |
|---|---|---|---|
| Monolithic | 単一 Deployment (1〜2 レプリカ) | MVP 規模。単一バイナリで distributor / ingester / querier / compactor を内包 | Phase 2 |
| Microservices | 各コンポーネントを個別 Deployment | Phase 3 以降。スケールが必要になった場合 | Phase 3 |

Tempo と同一の運用パターンであり、運用ノウハウが共有される。

### テレメトリ経路 (Phase 2)

```
サービス Pod (Pyroscope SDK / OTel eBPF Profiler)
    ↓ HTTP Push (Pyroscope ingestion API)
Pyroscope Server (monolithic, Deployment)
    └──→ MinIO (S3 バックエンド)
```

OTel Collector は Pyroscope のプロファイルデータ経路には含まれない。Pyroscope SDK がサーバーに直接 Push する方式である。ただし OTel Profiling SIG の成熟に伴い、将来的に OTel Collector 経由のパイプラインに統一される可能性がある。

### Grafana での統合

| 連携方向 | 仕組み | 効果 |
|---|---|---|
| Tempo → Pyroscope | スパンの `pyroscope.profile.id` 属性からフレームグラフにジャンプ (Traces-to-Profiles) | スロートレースの原因関数を即座に特定 |
| Pyroscope → Prometheus | フレームグラフから関連メトリクスへリンク | CPU 消費関数とシステム負荷の相関確認 |
| Grafana ダッシュボード | Pyroscope データソースによるフレームグラフ・テーブルパネル | サービス別の CPU / メモリ消費推移を常時監視 |
| diff 比較 | 2 つの時間範囲のフレームグラフを重ね合わせ | デプロイ前後のパフォーマンス回帰を視覚的に検出 |

### tier1 での計装方法

| 言語 | 計装手段 | オーバーヘッド | 備考 |
|---|---|---|---|
| Go (tier1 ファサード) | Pyroscope Go SDK (`github.com/grafana/pyroscope-go`)。`runtime/pprof` ベース。CPU / メモリ / goroutine / mutex / block プロファイル | CPU 1〜3% 増 | 本番環境で常時有効化可能。OTel 統合 (`otel-profiling-go`) でスパンプロファイル自動収集 |
| Rust (tier1 自作領域) | `pprof-rs` クレート + Pyroscope ingestion API への HTTP Push | CPU 1〜2% 増 | CPU プロファイルのみ。メモリプロファイルは `jemalloc-ctl` 等で補完 |
| C# (tier2/tier3) | Pyroscope .NET SDK (`Pyroscope.Agent`)。`EventPipe` ベース。CPU / メモリ / 例外プロファイル | CPU 2〜5% 増 | .NET 6+ が対象。.NET Framework は非対応 (Phase 4 のレガシー共存スコープ外) |

### tier1 公開 API との関係

Continuous Profiling は `k1s0.Telemetry.*` API の一部として tier2/tier3 に提供する。

| 方式 | 説明 |
|---|---|
| **自動計装 (推奨)** | 雛形生成 CLI が生成するサービステンプレートに Pyroscope SDK の初期化コードを組み込む。tier2/tier3 開発者は何も意識せずにプロファイリングが有効化される |
| 手動計装 | `k1s0.Telemetry.Profile.*` API でカスタムプロファイルラベル (業務コンテキスト等) を付与。必要な場合のみ |

雛形生成 CLI が SDK 初期化を自動生成するため、tier2/tier3 開発者が Pyroscope の存在を意識する必要はない。Dapr 隠蔽と同様の「透過的な計装」パターンである。

### サンプリング戦略

| フェーズ | 戦略 | 理由 |
|---|---|---|
| Phase 2 | 全サービス常時プロファイリング (100 Hz CPU サンプリング) | サービス数が少なく、ストレージ / CPU オーバーヘッドが許容範囲内 |
| Phase 3 | サービス単位でサンプリングレート調整 (高負荷サービスは 50 Hz に低減) | サービス数増加に伴うストレージ最適化 |
| Phase 5 | OTel eBPF Profiler による言語非依存の常時プロファイリング + SDK ベースの補完 | 大規模環境での統一プロファイリング |

### メモリ概算

| 構成 | メモリ |
|---|---|
| Pyroscope monolithic (1 レプリカ) | 0.5 GB |
| Pyroscope monolithic (2 レプリカ HA) | 1.0 GB |

Tempo monolithic (0.5〜1.0 GB) と同等の軽量さ。

### MVP スコープ

- **Phase 2 で導入** — Grafana Tempo (Phase 2) + MinIO (MVP-1b) が前提
- `infra` namespace にデプロイ (Deployment、monolithic モード)
- バックエンド: MinIO (既存) の `pyroscope` バケット
- Grafana に Pyroscope データソースを追加
- Tempo → Pyroscope の Traces-to-Profiles リンクを設定
- 雛形生成 CLI のテンプレートに Pyroscope SDK 初期化コードを追加
- tier1 Go ファサードに Pyroscope Go SDK を組み込み
- tier1 Rust サービスへの計装は Phase 2 中期 (`pprof-rs` の安定性評価後)

### 段階的な計装拡大

| 段階 | 対象 | 計装手段 |
|---|---|---|
| Phase 2 初期 | tier1 Go ファサード (全サービス) | Pyroscope Go SDK + OTel 統合 |
| Phase 2 中期 | tier1 Rust サービス | `pprof-rs` + HTTP Push |
| Phase 2 後期 | tier2 サンプルサービス (C# / Go) | 雛形生成 CLI のテンプレートに SDK 初期化を追加 |
| Phase 3 | tier2/tier3 全サービス | 雛形生成 CLI で自動計装 |

### トレードオフ

- Pyroscope SDK の組み込みにより、各サービスのバイナリサイズが若干増加する (Go: 約 2 MB、Rust: 約 1 MB)。コンテナイメージサイズへの影響は無視できる水準
- CPU プロファイリングのオーバーヘッド (1〜5%) は本番環境では許容範囲だが、リソース制約の厳しい MVP 環境ではモニタリングが必要。オーバーヘッドが許容範囲を超えた場合はサンプリングレートを低減する
- Rust のプロファイリングサポート (`pprof-rs`) は Go ほど成熟していない。CPU プロファイルは安定だが、メモリプロファイルは `jemalloc-ctl` 等の別手段が必要。Phase 2 中期に安定性を評価してから計装を拡大する
- .NET Framework は Pyroscope 非対応。Phase 4 のレガシー共存で .NET Framework アプリのプロファイリングは対象外とする (.NET 6+ のみサポート)
- CNCF プロジェクトではない (Grafana Labs 主導)。ただし Grafana Labs は Loki / Grafana / Tempo 等の主要 OSS を継続的にメンテナンスしており、[`11_追加採用OSS_3.md`](./11_追加採用OSS_3.md) の Grafana Tempo と同一のリスク評価が適用される
