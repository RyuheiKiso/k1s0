# シークレット管理

## 目的

k1s0 プラットフォームにおけるシークレット (パスワード / API キー / TLS 秘密鍵 / 暗号鍵) の一元管理・動的生成・ローテーション・監査の手段を定義する。

---

## Q. シークレット管理

| 候補 | 採否 | 評価 |
|---|---|---|
| **OpenBao** | 採用 | HashiCorp Vault の LF fork。MPL 2.0。OpenTofu / Valkey と同じ構図 (ライセンス変更への LF 対応) |
| HashiCorp Vault | 却下 | BSL 1.1 (非 OSI 承認)。k1s0 選定方針 [`00_選定方針.md`](./00_選定方針.md) の前提条件 3 に違反 |
| SealedSecrets | 補完的に採用 | GitOps で Secret を暗号化して Git に格納する用途に限定。Secret の動的生成・ローテーション・監査は対象外 |
| External Secrets Operator | 採用 | OpenBao の Secret を k8s Secret に自動同期。インフラ層コンポーネント (CloudNativePG / Keycloak / Argo CD / Harbor ��) が直接参照する k8s Secret の一元管理に必須。詳細は [`13_追加採用OSS_5.md#AC`](./13_��加採用OSS_5.md) を参照 |
| k8s Secrets (素の利用) | 却下 | base64 エンコードのみで暗号化ではない。ローテーション・監査機能なし |

### 採用理由 (OpenBao)

1. **OSI 承認ライセンス (MPL 2.0)** — k1s0 選定方針の前提条件に完全適合。OpenTofu (Terraform fork) / Valkey (Redis fork) と同じ構図であり、選定の一貫性を保てる
2. **動的シークレット** — PostgreSQL / Valkey 等のバックエンドに対して一時的な認証情報を動的生成 (TTL 付き)。漏洩時の影響を時間で制限する
3. **自動ローテーション** — 静的パスワードの手動ローテーションを排除。リース更新で無停止ローテーションを実現する
4. **Transit Engine (アプリケーション層暗号化)** — tier1 `k1s0.Pii` API の暗号化/復号処理を OpenBao に委譲し、鍵管理の責務を k1s0 チームから除去する
5. **PKI Engine (中間 CA)** — cert-manager の Issuer として OpenBao PKI を接続し、JTC オンプレ環境における自前 CA チェーンを構築する
6. **全アクセスの監査ログ** — 誰がいつどの Secret にアクセスしたかを自動記録。tier1 `k1s0.Audit` と突合してセキュリティインシデント調査を支援する
7. **Dapr Secret Store として統合可能** — Dapr の `secretstores.hashicorp.vault` Component で接続。tier1 `k1s0.Secrets` API の裏側を k8s Secrets から OpenBao に差し替えるだけで、tier2 / tier3 への影響ゼロ
8. **Vault との API 互換** — HashiCorp Vault の既存ドキュメント・エコシステム・ツール群をそのまま活用可能

### k1s0 における位置付け

| 観点 | 内容 |
|---|---|
| 配置 | `infra` namespace |
| 管理責任 | インフラチーム |
| tier1 との統合 | Dapr Secret Store Component のバックエンドとして接続。tier1 `k1s0.Secrets` API の内部実装 |
| tier2 / tier3 からの可視性 | **不可視** (Dapr 隠蔽と同じ原則) |
| Transit Engine の利用者 | tier1 Rust サービス (`k1s0.Pii` / `k1s0.Audit`) |
| PKI Engine の利用者 | cert-manager (Issuer として接続) |

### SealedSecrets との棲み分け

| 観点 | OpenBao | SealedSecrets |
|---|---|---|
| 目的 | シークレットの一元管理・動的生成・ローテーション・監査 | GitOps で Secret を暗号化して Git に格納 |
| 格納先 | OpenBao の暗号化ストレージ | Git リポジトリ (暗号化済み) |
| 動的シークレット | 対応 | 非対応 |
| ローテーション | 自動 | 手動 (再暗号化 + commit) |
| 監査ログ | 全アクセス記録 | なし |
| 適用範囲 | ランタイムの Secret 取得 | GitOps マニフェスト内の Secret 定義 |

SealedSecrets は MVP-1b で導入済みの既存手段として維持する。OpenBao 導入後は、SealedSecrets の用途を「Git 管理が必要な初期設定値 (OpenBao 自体の bootstrap 設定等)」に限定し、ランタイムの Secret 取得は OpenBao に移行する。

## Dapr Secret Store との統合

OpenBao 導入前後で tier1 公開 API (`k1s0.Secrets.*`) の仕様は一切変わらない。変更は Dapr Component YAML のバックエンド指定のみ。

```yaml
# Before (MVP-1b): k8s Secret をバックエンド
apiVersion: dapr.io/v1alpha1
kind: Component
metadata:
  name: secret-store
  namespace: tier1
spec:
  type: secretstores.kubernetes

# After (Phase 2): OpenBao をバックエンド
apiVersion: dapr.io/v1alpha1
kind: Component
metadata:
  name: secret-store
  namespace: tier1
spec:
  type: secretstores.hashicorp.vault
  metadata:
    - name: vaultAddr
      value: "http://openbao.infra.svc.cluster.local:8200"
    - name: vaultTokenMountPath
      value: "/var/run/secrets/openbao/token"
```

この差し替えは **Dapr 隠蔽方針が正しかったことを実証する最初の事例** となる。

## 利用する Engine

| Engine | 用途 | 導入フェーズ |
|---|---|---|
| **KV Secrets Engine v2** | 静的シークレットの格納 (Keycloak 管理者パスワード等) | Phase 2 |
| **Database Secrets Engine** | PostgreSQL 動的認証情報の生成 (TTL: 1 時間) | Phase 2 |
| **Transit Secrets Engine** | tier1 `k1s0.Pii` API の暗号化/復号委譲 | Phase 2 |
| **PKI Secrets Engine** | 中間 CA。cert-manager Issuer として接続 | Phase 3 |

## リスクと対処

| リスク | 対処 |
|---|---|
| OpenBao 自体の障害で全 Secret が取得不能 | 参照系 API と同じ cache-fallback パターンを適用 (起動時キャッシュ + プロセス内メモリ保持)。HA 構成 (3 Pod Raft) で可用性を確保 |
| Unseal 鍵の管理 | Shamir Secret Sharing (3-of-5) で分散。鍵ホルダーを起案者 + 協力者 + 管理者に分散し、単独での Unseal を防止 |
| HashiCorp からの法的リスク | OpenBao は LF 傘下であり、OpenTofu と同じく LF が法的防御を提供。2025 年時点で HashiCorp からの訴訟はなし |
| Vault との機能乖離 | コア機能 (KV / Database / Transit / PKI) は API 互換を維持。k1s0 が利用するのはこれらのコア機能に限定されるため、乖離リスクは低い |
| 運用負荷 (Unseal / 鍵ローテーション) | Auto-unseal (Transit 方式) を Phase 3 以降で検討。MVP-1b / Phase 2 では手動 Unseal (頻度: Pod 再起動時のみ) |

## リソース見積

| 項目 | 値 |
|---|---|
| メモリ | 256 MB - 512 MB (3 Pod Raft 構成で合計 0.8 - 1.5 GB) |
| CPU | 0.5 vCPU / Pod |
| ストレージ | 1 GB (Raft ストレージ、Longhorn PV) |

MVP-1b の VM 3 台 (合計 24 GB) のメモリ余裕 (約 3.0 GB) 内に収まる。

## 段階導入計画

| フェーズ | 施策 |
|---|---|
| **MVP-1b** | OpenBao デプロイ (HA 3 Pod Raft)。KV Engine で Keycloak / PostgreSQL / Argo CD の静的 Secret を��納。SealedSecrets との並行運用。**External Secrets Operator を導入し、OpenBao → k8s Secret の自動同期を開始** |
| **Phase 2** | Dapr Secret Store バックエンドを k8s Secrets → OpenBao に移行。Database Engine で PostgreSQL 動的認証情報を生成。Transit Engine で `k1s0.Pii` の暗号化を委譲 |
| **Phase 3** | PKI Engine で中間 CA を構築し cert-manager と連携。Auto-unseal の検討。SealedSecrets の用途を bootstrap 設定のみに縮小 |
