# 運用工数試算

## 目的

Phase 2 以降の k1s0 運用に必要な年間工数を、採用 OSS ごとに分解して試算する。[`02_開発工数試算.md`](./02_開発工数試算.md) が「作る工数」であるのに対し、本資料は「保守する工数」を扱う。CVE 対応・バージョン追従・障害対応・リファレンス更新の 4 活動について OSS 単位で積み上げ、年間の運用人員 FTE を導出する。本試算の数字は [`01_TCO5年試算.md`](./01_TCO5年試算.md) の Build 側コスト構造に直接入力される。

本資料で導く結論は以下の三点に集約される。まず、中規模 (3,000 名) で採用 OSS 全体の年間保守工数は 18 人月 = 1.5 FTE となり、商用 IDP で自社運用が不要になる訳ではない（3,000 名で 3 FTE の社内運用人員は商用でも必要）という事実を明らかにする。次に、保守工数は OSS の数に比例せず、tier1 自作領域（Go + Rust + ZEN Engine + 雛形 CLI）が全体の 30% を占める構造となる。最後に、規模による保守工数の伸びは人員に比例せず（小 70% / 中 100% / 大 150%）、規模効果が効くことを示す。

---

## 1. 試算方針

### 1.1 保守工数の 4 活動

採用 OSS の年間保守工数を以下の 4 活動に分解する。

- **CVE 対応**: 採用 OSS に公表された脆弱性への対処（影響評価・パッチ適用・検証）。CVSS スコア 7.0 以上は原則 30 日以内に対応する運用想定
- **バージョン追従**: マイナー・メジャーバージョンアップへの追従。Renovate により PR 自動生成されるが、動作確認・統合テスト・Runbook 更新は人手
- **障害対応**: 本番障害の一次調査・Runbook 実行・根本原因分析・再発防止策
- **リファレンス更新**: 新機能や設定変更に伴うドキュメント更新・リファレンス実装の改修

### 1.2 見積もりの単位

各 OSS について、4 活動の年間工数を 0.1 人月（約 2 人日）刻みで見積もる。見積もりの根拠は以下の 3 種類を採用する。

- CNCF Landscape の成熟度（Graduated / Incubating / Sandbox）
- 過去 24 ヶ月の CVE 公表頻度（NVD 参照）
- 採用 OSS のメジャーバージョン更新頻度（GitHub Releases 参照）

### 1.3 規模スケール係数

OSS 保守工数は規模に線形比例せず、「同じ OSS を運用するコスト」の基盤部分は規模に依らず発生する。本試算では規模スケール係数を以下で設定する。

- 小 (1,000 名): × 0.7 — パイロット業務 1 本、ピーク RPS 50、障害対応頻度が低い
- 中 (3,000 名): × 1.0 — 基準値
- 大 (10,000 名): × 1.5 — パイロット業務 3 本、ピーク RPS 500、障害対応頻度が増え、リファレンス改修も複数部門向けに拡張

## 3. 中規模 (3,000 名) の年間合計

| カテゴリ | 年間人月 | 比率 |
|---|---|---|
| k8s 基盤系 | 2.8 | 15% |
| サービスメッシュ / ゲートウェイ | 1.3 | 7% |
| 可観測性 (LGTMP + OTel) | 1.7 | 9% |
| データストア | 1.8 | 10% |
| CI/CD | 2.0 | 11% |
| セキュリティ / IAM | 2.7 | 14% |
| イベント / ワークフロー | 2.3 | 12% |
| tier1 自作領域 | 3.8 | 20% |
| 開発者ポータル | 1.2 | 6% |
| **合計** | **18.6 人月/年** | **100%** |

年間 18.6 人月 = 1.55 FTE。内訳では「tier1 自作領域（20%）」と「k8s 基盤系（15%）」が主要で、この 2 カテゴリだけで 35% を占める。商用製品では前者が不要になる（tier1 ファサードが商用機能に置換されれば）が、後者は残存する（K8s 運用自体は商用でも自社負担）。この構造が「商用に切り替えても運用工数は半減しない」という [`01_TCO5年試算.md`](./01_TCO5年試算.md) の前提を支える。

## 4. 規模別の年間保守工数

### 4.1 スケール適用後の集計

| 規模 | 係数 | 年間人月 | FTE |
|---|---|---|---|
| 小 (1,000 名) | × 0.7 | 13.0 | 1.08 |
| 中 (3,000 名) | × 1.0 | 18.6 | 1.55 |
| 大 (10,000 名) | × 1.5 | 27.9 | 2.33 |

保守工数は 18.6 人月を基準として規模で伸縮するが、これは「OSS を動かし続けるだけ」の数字である。tier2 オンボーディング支援・業務ロジック実装・アプリ配信ポータル運用など「k1s0 の価値を顧客（社内部門）に届ける」活動は本試算に含まれず、[`02_開発工数試算.md`](./02_開発工数試算.md) 第 8 章の年間人員想定に追加分として計上されている。

### 4.2 規模別の総人員（保守 + 価値提供）内訳

| 規模 | 保守 (本資料) | 価値提供 (02 章 8) | 合計 FTE |
|---|---|---|---|
| 小 (1,000 名) | 1.08 | 1.92 | 3.0 |
| 中 (3,000 名) | 1.55 | 3.45 | 5.0 |
| 大 (10,000 名) | 2.33 | 7.67 | 10.0 |

価値提供側の工数が保守工数の 2〜3 倍となる点は、k1s0 が「運用負担を下げて業務への貢献を増やす」設計思想と整合する。

## 5. 商用 IDP 採用時との比較

商用 IDP (OpenShift / Tanzu) を採用した場合、本資料のカテゴリの一部がベンダー保守に置換される。ただし完全に不要となる訳ではなく、「OSS が閉鎖的になる分、社内からの要望対応とカスタマイズ工数が増える」側面もある。

| カテゴリ | k1s0 (自社運用) | OpenShift 採用時 | 差分 |
|---|---|---|---|
| k8s 基盤系 | 2.8 | 1.5 (ベンダー保守の残り運用工数) | -1.3 |
| サービスメッシュ | 1.3 | 0.5 | -0.8 |
| 可観測性 | 1.7 | 1.2 (OpenShift 内蔵部分はベンダー保守) | -0.5 |
| データストア | 1.8 | 1.8 (CloudNativePG はベンダー非対応で残る) | 0 |
| CI/CD | 2.0 | 1.2 (OpenShift Pipelines 相当) | -0.8 |
| セキュリティ / IAM | 2.7 | 2.0 (Keycloak は残る) | -0.7 |
| イベント / ワークフロー | 2.3 | 2.3 (OpenShift 範囲外) | 0 |
| tier1 自作領域 | 3.8 | 4.5 (商用 API ラッパ追加で増加) | +0.7 |
| 開発者ポータル | 1.2 | 0.5 (OpenShift Dev Console) | -0.7 |
| **合計** | **18.6** | **15.5** | **-3.1 人月/年** |

OpenShift 採用で年間 3.1 人月（約 370 万円）を削減できるが、同時にライセンス費 1,500 万〜4,800 万円/年が加算される（規模依存）。人件費削減効果がライセンス費を下回る構造は [`01_TCO5年試算.md`](./01_TCO5年試算.md) で明示する。

## 6. 不確実性

本試算の精度に関する主な不確実性は以下。

- CVE 公表頻度は過去 24 ヶ月平均で見積もっており、Zero-Day が集中した場合は工数が 1.5 倍に増える
- 採用 OSS がライセンス変更・主要メンテナー離脱した場合、代替 OSS への移行工数が別途発生する（前例: Redis → Valkey）
- tier1 自作領域の工数は社内ユーザー数に強く依存し、大規模では要望対応が急増する可能性がある
- 業務部門からの機能要望の優先度判断（プロダクトマネジメント）工数は本試算に含まれていない
