# 開発工数試算

## 目的

k1s0 の初期開発（Phase 0〜1c）および基盤拡張（Phase 2 以降）に必要な開発工数を、機能単位・スキル別・期間別に試算する。[`../07_ロードマップと体制/00_フェーズ計画.md`](../07_ロードマップと体制/00_フェーズ計画.md) で定義された Phase スコープを量的に裏付け、稟議における「何人で何ヶ月かかるか」の答弁を支える。本試算は [`00_試算前提.md`](./00_試算前提.md) の人月単価を使用する。

本資料で導く結論は以下の三点に集約される。まず、Phase 0〜1c の初期構築は合計 14.2 人月・約 1,850 万円で完遂できる。次に、Phase 1a は起案者単独の集中稼働で 2.5 人月を要し、当初計画の「3〜4 週間」は既存業務と並行する前提では達成不可能である。最後に、Phase 2 以降の年間拡張・運用は規模により 3,700 万円〜1.4 億円/年のレンジに収まり、商用 IDP のライセンス費を下回る水準を維持できる。

---

## 1. 試算方針

### 1.1 工数分解の単位

開発工数は、Phase 計画で列挙された採用コンポーネント単位に分解する。各コンポーネントに対して「セットアップ（導入 / 構成 / 動作確認）」「統合（既存スタックへの組込み）」「ドキュメント」の 3 工程で人月を割当てる。これにより、コンポーネント増減に対して工数が機械的に再試算できる構造とする。

### 1.2 スキル区分

人月をスキル別に分解することで、「どんな人材を何人確保すべきか」の採用・アサイン判断に直接つなげる。本試算では以下の 6 区分を採用する。

- **k8s 運用**: kubeadm / Helm / Kustomize / ネットワーク / ストレージの構築・運用
- **Go 開発**: tier1 Go ファサード / 雛形生成 CLI / Temporal Worker
- **Rust 開発**: tier1 Rust サービス / ZEN Engine 統合（Phase 2 以降）
- **Web / Frontend**: アプリ配信ポータル / Backstage カスタマイズ
- **IaC / GitOps**: OpenTofu / Argo CD / GHA ワークフロー / Tilt
- **セキュリティ / IAM**: Keycloak / OpenBao / cert-manager / Kyverno
- **ドキュメント**: Runbook / アーキテクチャ文書 / リファレンス実装解説

### 1.3 稼働率の前提

起案者および協力者はフェーズ中の稼働率を 100%（既存業務から完全離脱）で試算する。JTC の実態として Phase 0 承認時に起案者を他業務から解放する稟議が同時に通る前提であり、もし 50% 稼働（兼任）となる場合は実期間が 2 倍に伸長する。この感度は 7 章で示す。

## 2. Phase 0 (企画承認): 0.5 人月

Phase 0 は本企画書とその配下ドキュメント群の整備、決裁者への説明、パイロット業務の選定着手からなる。起案者（シニア）0.5 FTE × 1 ヶ月 = 0.5 人月を想定する。本試算前はこの工数は既に消化済み（2026 年 4 月時点でドキュメント整備が完了しつつある状態）であり、稟議通過後に再消化されることはない。

| 項目 | 人月 | スキル | 備考 |
|---|---|---|---|
| 企画書・配下ドキュメント整備 | 0.3 | ドキュメント | 本資料を含む |
| 決裁者・ステークホルダー説明 | 0.1 | ドキュメント | Marp スライド使用 |
| パイロット業務の選定 | 0.1 | ドキュメント | 業務部門ヒアリング |
| **合計** | **0.5** | | 起案者シニア 150 万/月 × 0.5 = **75 万円** |

---

## 3. Phase 1a (MVP-0): 2.5 人月

Phase 1a は起案者単独、VM 1 台で「動くもの」を構築しデモする。フェーズ計画では「3〜4 週間」と記載されているが、下記コンポーネントの積み上げで 2.5 人月（1 名フルタイムで 10 週間、2.5 ヶ月）が妥当な見積もりとなる。「3〜4 週間」は既存業務と並行する前提ではなく、集中稼働であっても達成困難であるため、本試算では実行可能な 2.5 ヶ月を採用する。

### 3.1 機能別工数

| 機能 | 人月 | スキル |
|---|---|---|
| kubeadm 単一ノード構築 + Dapr Control Plane | 0.3 | k8s 運用 |
| Kustomize + Helm 雛形整備 | 0.2 | IaC / GitOps |
| tier1 Go サービス (`k1s0.Log` のみ) | 0.3 | Go |
| 雛形生成 CLI (最小テンプレート) | 0.3 | Go |
| Keycloak + PostgreSQL (単一) 構築 | 0.2 | セキュリティ / IAM |
| アプリ配信ポータル (SSO + 一覧 + 起動リンク) | 0.5 | Web / Frontend |
| サンプル Web アプリ (パイロット業務簡易版) | 0.3 | Web / Frontend |
| 動作確認・デモ準備 | 0.2 | ドキュメント |
| デモ用ドキュメント・セットアップ手順 | 0.2 | ドキュメント |
| **合計** | **2.5** | |

### 3.2 金額換算

起案者（社員シニア 150 万円/月）× 2.5 人月 = **375 万円**。VM は既存開発用 VM を流用する想定で、ハードウェア費は発生しない。

### 3.3 Phase 1a 完了条件との対応

Phase 1a の完了条件は「決裁者へのデモを実施し、動くものを確認できること」である。上記工数により、アプリ配信ポータル経由でサンプル Web アプリが Keycloak SSO で起動し、tier1 Go サービスがログを出力する一連のフローを実演できる状態に達する。

## 4. Phase 1b (MVP-1a): 6.8 人月

Phase 1b は 2 名体制で kubeadm HA クラスタを構築し、パイロット業務 1 本を稼働させる段階。起案者と協力者が並行稼働し、実期間 3.4 ヶ月を想定する。当初計画の「3 ヶ月」と概ね整合する。

### 4.1 機能別工数

| 機能 | 人月 | スキル |
|---|---|---|
| kubeadm 3 ノード HA (kubespray) + OpenTofu | 0.5 | k8s 運用 / IaC |
| kube-vip (Control Plane VIP) | 0.3 | k8s 運用 |
| MetalLB (L2 モード) | 0.3 | k8s 運用 |
| Dapr 拡張 + tier1 Go (`k1s0.Log` + `k1s0.Telemetry`) | 0.8 | Go |
| Keycloak HA + CloudNativePG (プライマリ + レプリカ) | 0.5 | セキュリティ / k8s 運用 |
| Argo CD + GHA self-hosted runner | 0.7 | IaC / GitOps |
| Prometheus + Grafana | 0.3 | k8s 運用 |
| OTel Collector (Agent / DaemonSet) | 0.3 | k8s 運用 |
| cert-manager + 社内 CA | 0.2 | セキュリティ |
| Tilt (ローカル開発環境) + Tiltfile 生成 | 0.4 | Go / IaC |
| Longhorn (分散ストレージ) | 0.3 | k8s 運用 |
| CloudNativePG Pooler (PgBouncer) | 0.2 | k8s 運用 |
| Renovate | 0.2 | IaC / GitOps |
| Testcontainers (DinD sidecar) | 0.3 | IaC / GitOps |
| Headlamp (K8s Web UI) | 0.2 | k8s 運用 |
| Kubeshark (eBPF API ビューア) | 0.1 | k8s 運用 |
| アプリ配信ポータル拡張 (監査ログ) | 0.5 | Web / Frontend |
| リファレンス実装 1 本 (パイロット業務) | 0.5 | Go / Web |
| Runbook・アーキテクチャ文書 | 0.5 | ドキュメント |
| **合計** | **6.8** | |

### 4.2 金額換算

- 起案者（社員シニア 150 万円/月）× 3.4 ヶ月 = 510 万円
- 協力者（社員中堅 100 万円/月）× 3.4 ヶ月 = 340 万円
- **小計: 850 万円**
- VM 3 台 (4 vCPU / 16 GB / 200 GB SSD): 新規調達なら 3 台 × 100 万円 = 300 万円、既存流用なら 0 円

### 4.3 Phase 1b 最重要完了条件との対応

「起案者以外の協力者が独立して環境を再構築できること（バス係数 2 の実証）」が完了条件。工数内訳の OpenTofu・Runbook・リファレンス実装・Tilt に合計 2.0 人月を割当てており、再現性ドキュメントへの投資比率（30%）が完了条件を支える。

## 5. Phase 1c (MVP-1b): 4.4 人月

Phase 1c は新機能追加を禁じ、運用に耐える状態へ引き上げる。2 名体制で 2.2 ヶ月、計画の「2 ヶ月」と整合する。

### 5.1 機能別工数

| 機能 | 人月 | スキル |
|---|---|---|
| Harbor + Trivy (push 時スキャン、Critical 拒否) | 0.3 | IaC / セキュリティ |
| Loki (ログ集約) | 0.3 | k8s 運用 |
| Sealed Secrets | 0.2 | セキュリティ |
| Kyverno (PSS Restricted + イメージソース制限) | 0.5 | セキュリティ |
| MinIO (S3 互換オブジェクトストレージ) | 0.4 | k8s 運用 |
| OpenBao (HA 3 Pod Raft + KV Engine) | 0.6 | セキュリティ |
| External Secrets Operator | 0.3 | セキュリティ / IaC |
| CI/CD パイプライン完成 (Lint / UT / FS Scan / Harbor / Argo CD) | 0.5 | IaC / GitOps |
| バックアップ手順 + フルリストア訓練 | 0.5 | k8s 運用 / ドキュメント |
| 運用 Runbook + アラート設定 | 0.5 | ドキュメント |
| Chaos 初期訓練 (手動 PostgreSQL フェイルオーバー等) | 0.3 | k8s 運用 |
| **合計** | **4.4** | |

### 5.2 金額換算

- 起案者 × 2.2 ヶ月 = 330 万円
- 協力者 × 2.2 ヶ月 = 220 万円
- **小計: 550 万円**

### 5.3 Phase 1c 完了条件との対応

「協力者が起案者不在で障害シミュレーション → Runbook 参照 → 対処を完結できること」。Runbook 整備とリストア訓練に合計 1.0 人月を投下し、協力者のドキュメント駆動運用を実証する。

## 6. Phase 0〜1c 合計および金額まとめ

### 6.1 工数合計

| Phase | 人月 | 実期間 | 体制 |
|---|---|---|---|
| Phase 0 | 0.5 | 1.0 ヶ月 | 起案者シニア × 0.5 FTE |
| Phase 1a | 2.5 | 2.5 ヶ月 | 起案者シニア × 1.0 FTE |
| Phase 1b | 6.8 | 3.4 ヶ月 | 起案者 + 協力者 × 2.0 FTE |
| Phase 1c | 4.4 | 2.2 ヶ月 | 起案者 + 協力者 × 2.0 FTE |
| **合計** | **14.2** | **9.1 ヶ月** | |

### 6.2 スキル別集計

| スキル | 人月 | 比率 | Phase 分布 |
|---|---|---|---|
| k8s 運用 | 4.6 | 32% | 1a: 0.3 / 1b: 2.3 / 1c: 2.0 |
| Go 開発 | 2.2 | 15% | 1a: 0.6 / 1b: 1.2 / 1c: 0.0 / その他 0.4 |
| Web / Frontend | 1.5 | 11% | 1a: 0.8 / 1b: 0.5 / その他 0.2 |
| IaC / GitOps | 2.3 | 16% | 1a: 0.2 / 1b: 1.4 / 1c: 0.7 |
| セキュリティ / IAM | 2.3 | 16% | 1a: 0.2 / 1b: 0.2 / 1c: 1.9 |
| ドキュメント | 1.3 | 10% | Phase 0 0.5 + 1a 0.4 + 1b 0.5 - 1c 1.0 (一部重複調整) |

最多スキルは k8s 運用（32%）であり、協力者の必須スキルは「kubernetes に実務で触れたことがある」水準であることを示す。Go 開発は 15% に留まるため、起案者が単独でカバー可能。

### 6.3 金額合計

| Phase | 人件費 | ハードウェア費 | 小計 |
|---|---|---|---|
| Phase 0 | 75 万 | 0 | 75 万 |
| Phase 1a | 375 万 | 0 (既存 VM) | 375 万 |
| Phase 1b | 850 万 | 0〜300 万 | 850〜1,150 万 |
| Phase 1c | 550 万 | 0 | 550 万 |
| **合計** | **1,850 万** | **0〜300 万** | **1,850〜2,150 万** |

規模に関わらず Phase 0〜1c の初期構築費は約 1,850〜2,150 万円で一定となる（MVP の技術スコープが規模非依存）。この総額は決裁者が稟議で扱う「初期投資額」に相当し、商用 IDP の初期導入費（ライセンス 1 年目 + 設定費）と同オーダーに収まる。

## 7. 感度分析

### 7.1 稼働率による伸長

起案者・協力者が既存業務と兼任し稼働率 50% となった場合、実期間は倍の 18.2 ヶ月（約 1.5 年）となる。人月は不変でも、デモから全社ロールアウトまでが 3 年以上延びるため、Phase 0 の稟議時点で「フルタイムアサイン」を条件として明記することが重要。

### 7.2 協力者の離任

Phase 1b 途中で協力者が離任し、起案者単独に戻った場合、残工数 5 人月を起案者が消化する必要があり実期間が 5 ヶ月延長する。この事象への備えとして、Phase 1b のドキュメント投資比率を 30% に確保している。

### 7.3 外部 SES 活用

協力者を社員ではなく協力会社 SES 中堅（130 万/月）に置き換えた場合、Phase 1b の人件費は 850 万 → 952 万円（+ 12%）に増加。バス係数の実証が担保できるなら、社員離任リスクを減らす選択肢として検討可能。

## 8. Phase 2 以降の年間拡張工数（規模別）

Phase 2 以降は基盤拡張と運用の並行フェーズとなり、規模により必要人員が変動する。詳細な運用工数の内訳は [`03_運用工数試算.md`](./03_運用工数試算.md) に譲り、本章では人月・金額サマリのみを示す。

| 規模 | Phase 2 以降の開発 + 運用人員 | 年間人月 | 年間人件費 |
|---|---|---|---|
| 小 (1,000 名) | 3 名 FTE (社員中堅換算) | 36 人月 | 3,600 万 |
| 中 (3,000 名) | 5 名 FTE (社員 4 + 協力会社 1 想定、加重 115 万/月) | 60 人月 | 6,900 万 |
| 大 (10,000 名) | 10 名 FTE (社員 7 + 協力会社 3 想定、加重 115 万/月) | 120 人月 | 13,800 万 |

### 8.1 人員内訳の考え方

中規模 (3,000 名) を基準とした内訳:
- tier1 チーム（Go + Rust + ZEN Engine）: 2 名
- インフラ運用（k8s + 可観測性 + バックアップ + セキュリティ）: 1 名
- tier2 推進・オンボーディング: 1 名
- 開発者ポータル + ドキュメント: 1 名

小規模では tier2 推進を内製開発が兼任し、3 名に圧縮。大規模では tier2 推進とドメイン別担当者が 4 名、インフラ 2 名、tier1 チーム 3 名、開発者ポータル 1 名に拡張。
