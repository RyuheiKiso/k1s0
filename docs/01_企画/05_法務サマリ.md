# 法務サマリ

## 目的

本資料は、稟議の場で **法務・コンプライアンス観点から k1s0 を否決しうる 4 つの論点**（OSS ライセンス適合・個人情報法令適合・知財帰属・監査応答可能性）を、決裁 1 時間以内に読み切れる粒度で整理する。詳細根拠と実装規定は [`../02_構想設計/05_法務とコンプライアンス/`](../02_構想設計/05_法務とコンプライアンス/) に委ねるが、「稟議の場で論点が出た瞬間に答えを返せる」水準の要約を本書で担保する。

本書が埋める空白は、従来 `02_構想設計/05_...` に押し込まれていた法務論点を **稟議の場で直接参照できる深さに圧縮** した点である。法務部門が稟議に陪席しない JTC でも、本書 1 枚を通読すれば決裁者は主要論点を自答できる。

---

## 1. 稟議で刺される 4 論点と即答

稟議の場では法務関連の質問は概ね以下 4 つの形で出る。それぞれについて、本章後段で根拠と実装を示す。

### 1.1 「AGPL の OSS を使うのは大丈夫か」

**即答**: Phase 1a (MVP-0) では AGPL OSS を一切採用しない。Phase 1b 以降で採用する 6 本 (Grafana / Loki / Tempo / Pyroscope / MinIO / Renovate) はすべて `infra` namespace 内部のみで動作し、JTC 認証ユーザー以外は物理的にアクセスできない。AGPL-3.0 第 13 条のソース提供義務は「外部ユーザーへのネットワーク越し提供」で発動する条項であり、社内利用では FSF / Grafana Labs 公式 FAQ が「義務発動なし」と明言している。

つまり **稟議承認の時点では AGPL 論点はゼロ**、Phase 1b 開始の法務レビュー時点で個別確認するプロセスに分割している。詳細と根拠は [`../02_構想設計/05_法務とコンプライアンス/00_OSS法務対応.md`](../02_構想設計/05_法務とコンプライアンス/00_OSS法務対応.md) 第 2.5 節。

### 1.2 「個人情報・マイナンバー・電帳法は大丈夫か」

**即答**: k1s0 はプラットフォームであり個人情報そのものを保持しない設計である。業務アプリ (tier3) が個人情報を扱う場合、tier1 の監査ログ・暗号化・アクセス制御 API を経由することで、個人情報保護法・マイナンバー法・電子帳簿保存法の各適合要件を **プラットフォーム側で一律に担保** する。個別の業務アプリで要件を重複実装する必要はない。適合性の具体的な写像は本書第 3 節を参照。

### 1.3 「k1s0 は誰の知財か」

**即答**: **起案者が業務として作成する成果物は、職務著作 (著作権法 15 条) により JTC 法人に帰属する**。起案者個人が個別契約で保有権を持つ構造はとらない。業務委託・SES・グループ会社が開発に加わる場合も、各契約雛形で著作権を JTC へ譲渡する条項を標準化する。OSS 採用 OSS の copyleft 条項 (MPL 2.0 / AGPL-3.0) は後述 2.3 節のプロセス分離により k1s0 自作コードに伝播しない。詳細は [`../02_構想設計/05_法務とコンプライアンス/01_内製知財戦略.md`](../02_構想設計/05_法務とコンプライアンス/01_内製知財戦略.md)。

### 1.4 「5 年後に監査で説明できるか」

**即答**: 採用 OSS の SBOM (Syft 自動生成)・四半期ライセンスレポート (自動 PR 起票)・ライセンス判定履歴を `infra/compliance-reports` リポジトリで Git 管理し、最低 5 年間保管する。ISMS / ISO 27001 / SOC 2 監査で要求される典型 5 問 (利用 OSS 一覧 / ライセンス一覧 / AGPL 義務履行 / 違反検出の仕組み / 四半期確認) は本書第 4 節の対応表で即答可能。

## 2. OSS ライセンス適合

### 2.1 採用 OSS のライセンス分布

採用予定 40+ OSS のうち、ライセンスの種類は 8 種。稟議で論点化するのは AGPL-3.0 (6 本) と MPL 2.0 (2 本) に限定され、残り 30+ 本 (Apache 2.0 / MIT / BSD / PostgreSQL / ISC) は **内部利用では義務なし**。

| ライセンス | 本数 | 代表 OSS | 内部利用での義務 |
|---|---|---|---|
| Apache 2.0 | 30 本超 | Kubernetes / Istio / Envoy / Kafka / Dapr / Keycloak / Argo CD / Backstage / Prometheus | NOTICE 表示のみ |
| MIT | 数本 | TypeScript ライブラリ / 一部 Rust crate | ライセンス文言添付のみ |
| BSD-3-Clause | 数本 | 一部 npm パッケージ | 同左 + 組織名の無断利用禁止 |
| PostgreSQL License | 1 本 | PostgreSQL 本体 | ライセンス文言添付のみ |
| ISC | 数本 | 一部 npm パッケージ | ライセンス文言添付のみ |
| MPL 2.0 | 2 本 | OpenTofu / OpenBao | 無改変利用なら義務なし |
| AGPL-3.0 | 6 本 | Grafana / Loki / Tempo / Pyroscope / MinIO / Renovate | 本章 2.2 で詳述 |
| デュアル (MIT OR Apache) | 数本 | sqlx 等 Rust crate | いずれかを選択 (MIT 選択) |

重要なのは Apache 2.0 + MIT + BSD 系で全体の約 8 割を占める事実である。この層は JTC 情シスが気にする必要のない「通常の OSS」であり、論点化するのは AGPL と MPL の合計 8 本のみ。**論点の絶対数は小さい**。

### 2.2 AGPL-3.0 の扱い：Phase 分割で稟議論点を先送りする

AGPL-3.0 の義務解釈を稟議の場で持ち込むと、論点が拡散する。k1s0 はこれを構造的に回避するため、**Phase 1a (MVP-0) では AGPL OSS を 1 本も使わない**。Grafana の代わりに Prometheus 標準 UI、Loki の代わりに `kubectl logs`、MinIO の代わりにローカルボリュームで MVP-0 を成立させる。

この分割により、稟議承認 (Phase 0 通過) の時点では **AGPL 論点はゼロ** となる。Phase 1b 移行のタイミングで法務部門と個別確認し、以下 5 原則の遵守を証跡化する。

1. プロセス分離 (AGPL OSS と自作コードは独立 Pod で動作)
2. ライブラリリンクなし (公開 API のみ使用、SDK 直接 import なし)
3. 無改変利用 (fork・プラグイン自作・ソース変更を禁止)
4. 外部境界遮断 (Envoy Gateway + Keycloak OIDC で JTC 認証ユーザー以外を物理遮断)
5. 監査ログの可検証性 (AGPL Pod への全アクセスを Kafka 監査トピックに記録)

5 原則の詳細と監査時の回答スクリプトは [`../02_構想設計/05_法務とコンプライアンス/00_OSS法務対応.md`](../02_構想設計/05_法務とコンプライアンス/00_OSS法務対応.md) 第 2.5.3 節。

### 2.3 MPL 2.0 (OpenTofu / OpenBao) の扱い

MPL 2.0 はファイル単位の copyleft で、**対象 OSS を fork して改変した場合のみ** 改変ファイルのソース公開義務が発生する。k1s0 は両者とも無改変利用であり、義務は発生しない。稟議の場ではこの 1 行で完結する。

### 2.4 ライセンス変更リスクへの備え

OSS ライセンスは業界全体で頻繁に変更される。直近の事例は Redis (2024 / RSALv2 へ)、Terraform (2023 / BUSL へ)、Elastic (2021 / SSPL へ) で、いずれも「同等 OSS への fork 移行」が発生した。k1s0 は以下の体制で備える。

- CI に Syft + Grype による自動ライセンス検出を組み込み、採用 OSS のライセンス変更を即検知する
- allow list (Apache / MIT / BSD / PostgreSQL / ISC / MPL 2.0 / AGPL-3.0) 外への変更が検出された時点で、Renovate が PR ブロック
- 代替 OSS への移行計画は [`../02_構想設計/03_技術選定/01_俯瞰/17_OSS長期戦略.md`](../02_構想設計/03_技術選定/01_俯瞰/17_OSS長期戦略.md) で OSS ごとに事前定義 (Redis → Valkey の移行は実施済み)

## 3. 日本の法令適合性

### 3.1 プラットフォームと業務アプリの責任分界

k1s0 自体は業務データを保持しない「プラットフォーム」であり、個人情報・マイナンバー・電子帳簿はすべて **業務アプリ (tier3)** が扱う。法令適合の責任は業務アプリにあるが、k1s0 は適合に必要な技術要件を **tier1 公開 API** として提供することで、業務アプリ側の実装負荷を下げる。

この分界により、「k1s0 は法令違反製品か」という稟議論点は成立しない。論点は「業務アプリが法令要件を満たすために必要な基盤機能が k1s0 に揃っているか」に書き換わる。以下、主要 4 法令について揃っている要件を整理する。

### 3.2 個人情報保護法 (2022 改正) 適合

改正個人情報保護法は、保有個人データの開示請求対応・漏えい時の本人通知・仮名加工情報の取扱いなど、業務アプリ側の対応を大幅に増やした。k1s0 はこれらに必要な基盤機能を以下で担保する。

| 法令要件 | k1s0 の提供機能 | 参照 |
|---|---|---|
| アクセスログの保管 (第 23 条 安全管理措置) | `k1s0.Audit` で全 API コールを Kafka 監査トピック + 改ざん防止ハッシュチェーンで記録 | [`../02_構想設計/01_アーキテクチャ/03_セキュリティ/01_セキュリティモデル.md`](../02_構想設計/01_アーキテクチャ/03_セキュリティ/01_セキュリティモデル.md) |
| 通信経路の暗号化 | Istio mTLS (サービス間) + TLS (外部) を全通信で強制 | 同上 |
| データ保存時の暗号化 | OpenBao Transit Engine による透過的暗号化・鍵ローテーション | [`../02_構想設計/03_技術選定/01_俯瞰/04_選定一覧.md`](../02_構想設計/03_技術選定/01_俯瞰/04_選定一覧.md) |
| 本人確認・アクセス制御 | Keycloak OIDC で全コンポーネントを SSO 統一、RBAC で業務単位の権限制御 | 同上 |
| 開示請求時のデータ所在特定 | Backstage Software Catalog で保有データのオーナーを一覧化 | [`../02_構想設計/04_CICDと配信/01_開発者ポータル_Backstage.md`](../02_構想設計/04_CICDと配信/01_開発者ポータル_Backstage.md) |

### 3.3 マイナンバー法 (番号利用法) 適合

マイナンバーを取り扱う業務アプリは厳格な安全管理措置を要するが、k1s0 は「マイナンバーを扱う tier3 を隔離する」構造を提供する。

- 専用 namespace (`tier3-mynumber`) を分離し、NetworkPolicy で他 namespace からのアクセスを拒否
- OpenBao Transit Engine で常時暗号化、アクセスログを `k1s0.Audit` で別トピックに記録
- 閲覧可能な運用者は Keycloak の専用 RBAC ロールで限定、アクセスは全件記録

この構造により、マイナンバーを扱う業務アプリが満たすべき「物理・技術・組織・人的」4 類型の安全管理措置のうち、**技術的措置** が k1s0 で包括的に満たされる。

### 3.4 電子帳簿保存法 (2022 改正) 適合

電子帳簿保存法は「真実性」「可視性」「検索性」の 3 要件を要求する。k1s0 が提供する機能は以下。

| 要件 | k1s0 の実装 |
|---|---|
| 真実性 (改ざん防止) | `k1s0.Audit` のハッシュチェーンに記録された時点で改ざん検出可能 |
| 真実性 (タイムスタンプ) | Kafka 監査トピックへの記録時刻が信頼できる基準時刻として機能。必要に応じて TSA (Timestamping Authority) 連携を tier1 で追加 |
| 可視性 (ディスプレイでの確認) | Backstage + Grafana で閲覧可能なダッシュボードを標準提供 |
| 検索性 (日付・金額・取引先) | tier1 `k1s0.Search` (Phase 2) で業務データの横断検索を提供 |

電子帳簿保存法の要求は業務アプリ固有の内容も含むため、完全な法令適合は業務アプリ側の責任である。ただし「技術的要件」に関しては k1s0 の基盤機能で半自動的に満たせる。

### 3.5 業種特化規制 (金融・医療・公共)

業種特化規制 (FISC 安全対策基準 / 医療情報システムの安全管理ガイドライン / 政府情報システムのためのセキュリティ評価制度 ISMAP) は、JTC の業種によって要件が大きく異なる。k1s0 の基盤機能 (mTLS / 監査ログ / 暗号化 / RBAC / ネットワーク分離) はこれら基準の「技術的要件」を広くカバーするが、**業種特化の運用手続き** (暗号鍵管理体制・監査委員会・セキュリティ評価受審) は各 JTC の運用組織側で整備する必要がある。

Phase 1b 以降、業種特化規制の詳細マッピングを `03_要件定義/30_セキュリティ_データ/10_compliance/` で採番する。本書は稟議承認時点での「技術基盤側の準備済み項目」を示す。

## 4. 知財帰属と業務委託

### 4.1 職務著作の原則

起案者が業務として作成する k1s0 の全成果物は、**著作権法 15 条 (職務著作) により JTC 法人に帰属** する。これは OSS 採用 / 非採用に関係なく成立する原則であり、起案者個人が著作権を留保する構造はとらない。この点は **稟議承認以前の既定事項** であり、個別契約は不要。

### 4.2 外部開発者の扱い

業務委託・SES・グループ会社から開発者が参画する場合は、各契約雛形で以下を標準化する。

- 成果物の著作権を JTC へ譲渡する条項 (著作者人格権不行使特約を含む)
- 成果物の OSS ライセンスを allow list に限定する条項
- 秘密保持義務 (業務ロジック・認証情報の第三者開示禁止)
- 業務終了後のコード・ドキュメント・認証情報の返却・削除義務

契約雛形の整備は Phase 0〜1 の法務部門タスクとして事前着手する。詳細条項は [`../02_構想設計/05_法務とコンプライアンス/01_内製知財戦略.md`](../02_構想設計/05_法務とコンプライアンス/01_内製知財戦略.md) 第 3.2 節を参照。

### 4.3 OSS 化の将来オプション

k1s0 自作コードのうち **雛形生成 CLI** は Phase 2 完了後に OSS 化 (Apache 2.0) を検討する。**基盤コード本体** と **業務ロジック (JDM)** は社内独占を維持する。この方針は経営層の判断事項であり、稟議承認段階では「Phase 5 以降で再評価」と位置付けて判断を保留する。詳細は [`../02_構想設計/05_法務とコンプライアンス/01_内製知財戦略.md`](../02_構想設計/05_法務とコンプライアンス/01_内製知財戦略.md) 第 2 章。

### 4.4 特許と商標

k1s0 の個別要素は「k8s + Istio + Dapr」の組み合わせが主で、特許取得に値する新規性は現時点では識別されない。商標 "k1s0" も社内利用のみでは出願不要。両者とも Phase 3 完了時点で再評価する。稟議承認段階では追加費用計上は不要。

## 5. 監査応答の準備

### 5.1 自動生成される証跡

k1s0 は監査応答に必要な証跡を **CI パイプラインで自動生成** し、人手による証跡作成を排除する。以下は Phase 1c (MVP-1b) 時点で全件稼働する。

| 証跡 | 生成タイミング | 生成ツール | 保管場所 |
|---|---|---|---|
| SBOM (CycloneDX 形式) | 全ビルド時 | Syft | Harbor + `infra/compliance-reports` |
| CVE 検出レポート | 全ビルド時 | Grype + Trivy | 同上 |
| ライセンス判定結果 | 全 PR | Syft + allow list 照合 | PR コメント + `infra/compliance-reports` |
| 四半期ライセンスレポート | 四半期自動 | `operation` namespace のジョブ | `infra/compliance-reports` への自動 PR |
| AGPL 隔離状況 | 四半期自動 | Envoy ルート定義 + Keycloak クライアント定義のスナップショット | 同上 |

これらを **最低 5 年間** Git 管理することで、監査要求された時点で数分以内に当該時点のスナップショットを復元できる。

### 5.2 監査時の典型 5 問への即答

ISMS / ISO 27001 / SOC 2 / 業種特化監査で繰り返し問われる 5 問に対し、本書の記述だけで即答可能な状態を目指す。

| 質問 | 即答 | 根拠資料 |
|---|---|---|
| 利用中の OSS 一覧を提示してください | SBOM (CycloneDX) を Harbor から即座に出力 | 本書 5.1 節 |
| 各 OSS のライセンスを提示してください | SBOM にライセンス情報埋め込み済み | 本書 2.1 節 |
| AGPL OSS の利用でソース提供義務を果たしていますか | 内部利用のため義務発動なし。FSF / Grafana Labs 公式 FAQ と隔離 5 原則を根拠として提示 | 本書 2.2 節 |
| ライセンス違反検出の仕組みはありますか | CI に自動ライセンス監査を組み込み、違反時は PR ブロック | 本書 5.1 節 |
| 四半期ごとの OSS 状況を確認していますか | 四半期レポートを自動生成し、法務部門へ提示 | 本書 5.1 節 |

より詳細な監査対応フロー (RFC 起案 → ライセンス判定 → 法務レビュー → 採用決定) は [`../02_構想設計/05_法務とコンプライアンス/00_OSS法務対応.md`](../02_構想設計/05_法務とコンプライアンス/00_OSS法務対応.md) 第 5 章。

## 6. 残存リスクと想定限界

本書は JTC 情シスが **稟議承認の時点で法務論点を封じ込める** ための要約書である。以下の限界があり、詳細対応は `02_構想設計/05_...` および法務部門の個別判断に委ねる。

- 本書の記述は法的助言ではない。AGPL-3.0 の解釈や業種特化規制の適用範囲は、最終的に各 JTC の法務部門が判断する
- OSS のライセンスは頻繁に変更される (Redis 2024 / Terraform 2023 / Elastic 2021)。本書の記述は 2026-04 時点のスナップショットであり、半期ごとの更新が必要
- 業種特化規制 (FISC / 医療情報ガイドライン / ISMAP) の詳細マッピングは本書では扱わず、Phase 1b 以降の要件定義層 ([`../03_要件定義/30_セキュリティ_データ/10_compliance/`](../03_要件定義/30_セキュリティ_データ/10_compliance/)) で採番する
- 業務委託契約・グループ会社契約の個別条項は、各 JTC の契約雛形との突き合わせが必要
