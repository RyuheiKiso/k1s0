# k1s0 全体構成図 (インフラ + アーキテクチャ)

## 目的

k1s0 が作り出すプラットフォームの全体像を **1 枚の図** で把握できるようにする。物理的な実行基盤 (k8s クラスタ / ネットワーク / ストレージ / セキュリティ) と、その上に乗る 4 実行レイヤ (`infra` / `tier1` / `tier2` / `tier3`) と横断運用レイヤ (`operation`) の関係を、1 つの構成図に統合する。

詳細な分割図 (レイヤ構成のみ / 依存ルールのみ / 配置形態のみ) は [`01_アーキテクチャ/`](../02_構想設計/01_アーキテクチャ/) に既に存在するため、本資料はそれらを **横断的に俯瞰する 1 枚** として位置付ける。

## 図の読み方

図は上から下に向かって、外部 → エッジ → クラスタ内部 (上位 tier → 下位 tier) という順序で配置している。横軸方向は左から「業務トラフィック」、右側に「運用 / 開発者向けの operation 列」を独立させている。

### 1. 外部とエッジ

エンドユーザーの端末 (PWA / MAUI / WinUI) は **社内 DNS / VPN** を経由してクラスタの **MetalLB (L2 モード)** に到達する。MetalLB が払い出した VIP を **Envoy Gateway** が受け、TLS 終端と OIDC 認可 (ext_authz → Keycloak) を行ってからクラスタ内部にルーティングする。クラウド LB / マネージド k8s には依存しないため、閉域ネットワーク要件を満たしつつ稟議のハードルを下げる。

`cert-manager` は Edge / ポータルで利用する TLS 証明書を自動発行・自動更新し、運用者の手で証明書を扱う場面を排除する。

開発者 / 運用担当者は同じ Envoy Gateway 経由で `operation` namespace に到達し、Backstage / Argo CD / Keycloak 等のポータル群へ OIDC SSO で入る。

### 2. クラスタと kube-vip

k8s 実装は **kubeadm をメインプラン、k3s をサブプラン** とするセルフマネージド構成を採用する。クラウドマネージド k8s (EKS / AKS / GKE) は閉域ネットワーク要件と整合しないため非採用とする。

Control Plane の高可用性 VIP は **kube-vip** が静的 Pod として各 Master ノードに配置され担保する。kube-vip は namespace に属さないため、図上では cluster の上端に注記として配置している。

### 3. tier3 → tier2 → tier1 → infra (依存ルール)

クラスタ内部の左 / 中央列は、上から **tier3 → tier2 → tier1 → infra** の順で積み上げている。これは k1s0 の **1 方向依存ルール** をそのまま視覚化したものである。各 tier 間を結ぶ縦の矢印は「上位が下位を参照する」流れを示す。

- **tier3** は業務アプリ Pod / アプリ配信ポータル Pod / レガシー .NET ラップ Pod を持つ。すべて `app container + Dapr sidecar` の 2 コンテナ構成だが、**sidecar の管理責任は tier1 にある** ため、tier3 開発者は Dapr を意識しない。
- **tier2** はドメインサービス Pod 群と、画面 / IF / 抽象クラスを定義する仕様レイヤから成る。tier3 が準拠する標準を提供する。
- **tier1** は最も濃い情報量を持つ層で、4 つの構成要素 (公開 API / Dapr Control Plane / tier1 Go サービス / tier1 Rust サービス) と雛形生成 CLI から成る。**`k1s0 公開 API` 帯** が tier2/tier3 から見える唯一の接点であり、その下にある内部実装 (Go / Rust の使い分け) は完全に隠蔽される。
- **infra** はサービスメッシュ・メッセージング・観測性・データストア・セキュリティを 5 つの小グループに整理して配置している。tier1 のみが infra に直接アクセスでき、tier2 / tier3 は infra コンポーネントの URL や SDK を直接参照してはならない。

### 4. operation 列 (横断)

右列の `operation` namespace は実行レイヤではなく、全レイヤに横断的に作用する運用プロセス層である。Backstage が開発者ポータル、Argo CD が GitOps デプロイの起点、Keycloak が OIDC 認可、GitHub Actions self-hosted runner が CI エンジン、Renovate が依存自動更新 PR、Argo Events がイベント駆動 (Phase 2)、Litmus が Chaos Engineering をそれぞれ担う。

operation の各コンポーネントは tier1〜tier3 のどこにでも作用しうるため、図上では cluster 内に縦に独立した列として配置し、業務トラフィックとは線を引かずに視覚的に切り離している。

## マルチテナント分離モデル

k1s0 は「単一の情シス部門が複数部門・複数グループ会社の業務を同居運用する」ユースケースを第一級で扱う。テナント境界が曖昧なままでは、稟議で「別部門の情報が別テナントに漏れないか」「グループ企業に横展開できるか」と問われた際に即答できない。本節ではテナント分離を 4 層に分解し、どの層でどの OSS が何を保証するかを明示する。詳細な設計は [`20_品質特性/05_tenant_isolation/`](../03_要件定義/20_品質特性/05_tenant_isolation/) と [`01_アーキテクチャ/03_セキュリティ/01_セキュリティモデル.md`](../02_構想設計/01_アーキテクチャ/03_セキュリティ/01_セキュリティモデル.md) を参照。

### 分離の 4 層

テナント分離は「名前空間」「ネットワーク」「認可」「データ」の 4 層すべてで同時に担保する。どれか 1 層でも抜けるとテナント越境が成立し、監査で致命的な指摘を受ける。

- **名前空間分離 (k8s Namespace + ResourceQuota + LimitRange)**: テナントごとに `tenant-{id}` 名前空間を割当て、CPU / メモリ / Pod 数の上限を ResourceQuota で強制する。1 テナントのリソース暴走が他テナントの Pod を退避させない設計とする。
- **ネットワーク分離 (k8s NetworkPolicy + Istio AuthorizationPolicy)**: NetworkPolicy で namespace 間の Pod-to-Pod 通信を既定拒否とし、tier1 公開 API への到達のみを許可する。Istio AuthorizationPolicy は mTLS の SPIFFE ID (`spiffe://cluster.local/ns/tenant-a/sa/xxx`) を principal として認可し、tenant-a の ServiceAccount が tenant-b のサービスに到達することを拒否する。
- **認可分離 (Keycloak Realm + RBAC + OPA/ZEN Engine)**: Keycloak は **テナントごとに Realm を分離** し、ユーザー・ロール・クライアントをテナント境界を越えて共有しない。tier1 公開 API は全リクエストで JWT の `tenant_id` クレームを検証し、ZEN Engine の決定表で「この操作を、この tenant_id が、このリソースに対して行えるか」を宣言的に判定する。
- **データ分離 (PostgreSQL スキーマ + MinIO バケット + Kafka トピック)**: PostgreSQL は **スキーマ単位でテナント分離** し、Row-Level Security (RLS) で `tenant_id` 列に一致する行以外を SELECT 不可にする。MinIO は `tenant-{id}/` プレフィックスと IAM ポリシーで分離、Kafka は `tenant-{id}.*` トピック名前空間と ACL で分離する。監査ログ (`k1s0.Audit`) も同様に `tenant_id` を必須フィールドとする。

### 越境検知と封じ込め

分離設計が運用中に破れる典型パターンは 3 つある: ①開発者が namespace を誤って共有する、②JWT 検証のバグで `tenant_id` クレームが欠落する、③DB マイグレーションで RLS が外れる。k1s0 はこれらを **運用中に機械的に検知する仕組み** を持つ。

- **Kyverno ポリシー**: Pod に `tenant-id` ラベル必須、namespace に ResourceQuota 必須を admission 時に強制
- **Istio AuthorizationPolicy のデフォルト拒否**: 明示的に許可されない tenant 間通信は全て 403
- **`k1s0.Audit` の tenant_id 必須検証**: `tenant_id=null` で監査ログを書き込んだ場合、tier1 が 500 を返し CI が落ちる
- **Litmus Chaos 実験での越境シミュレーション**: 月次で「tenant-a の Pod が tenant-b の Secret を盗めるか」を自動検証

### グループ企業展開時の分離ポリシー

グループ企業・子会社に k1s0 を横展開する際は、テナント分離だけではなくクラスタ分離を選択肢として持つ。以下の判断基準で分離レベルを決定する。

- **同一クラスタ + テナント分離**: 同一法人内の部門・子会社。監査要件が共通で、運用チームが 1 つで済む場合
- **クラスタ分離 (Phase 3 マルチクラスタ前提)**: 法人が異なる / 監査基準が異なる / データレジデンシー要件が異なる場合。詳細は次節「リージョンとデータレジデンシー戦略」を参照

## リージョンとデータレジデンシー戦略

k1s0 は閉域オンプレ前提で設計されているが、「グローバル展開する顧客が使えるか」「日本の個人情報を海外に出さない保証はあるか」は稟議の必出論点となる。本節ではリージョン構成とデータレジデンシーを Phase 別に明示する。詳細な設計は [`02_可用性と信頼性/04_マルチクラスタ戦略.md`](../02_構想設計/01_アーキテクチャ/02_可用性と信頼性/04_マルチクラスタ戦略.md) と [`30_セキュリティ_データ/07_data_residency/`](../03_要件定義/30_セキュリティ_データ/07_data_residency/) を参照。

### Phase 別のリージョン戦略

Phase 1a〜2 までは単一データセンター (または単一クラウドリージョン) に留め、Phase 3 でマルチクラスタ化、Phase 5 で多リージョン化に進む。最初からマルチリージョンを設計に組み込むと MVP-0 の 2 週間デモが破綻するため、段階的導入を明確化する。

- **Phase 1a〜1c**: 単一 VM / 単一クラスタ / 単一リージョン。DR は PostgreSQL の論理バックアップのみ。
- **Phase 2**: 単一クラスタだが MinIO / PostgreSQL のバックアップを **同一リージョン内の別 AZ / 別 DC** へ複製開始。RPO 目標 15 分 / RTO 4 時間。
- **Phase 3**: **2 クラスタ体制 (staging / prod)** を同一リージョン内で構築。Argo CD が両クラスタを管理し、prod 障害時は staging への昇格手順を Runbook 化する。
- **Phase 5**: **マルチリージョン (東日本 DC + 西日本 DC)** に拡張。Keycloak は master realm レプリケーション、PostgreSQL は Streaming Replication、Kafka は MirrorMaker 2、MinIO は Bucket Replication でリージョン間同期する。

### データレジデンシーの保証

JTC 顧客・グループ企業の要件として最も頻出するのは「日本の個人情報を国外に出さない」「監査ログは国内で 5 年保管」の 2 つ。k1s0 はデータ種別ごとにレジデンシーを明示し、保管先を技術的に強制する。

- **個人情報 (氏名 / マイナンバー / 生体情報)**: `k1s0.Pii` API 経由のみアクセス可能。保管先は MinIO `pii-jp/` バケット固定、OpenBao Transit Engine で暗号化。国外レジデンシーの MinIO / RDBMS への書込みを Kyverno で拒否する。
- **監査ログ (`k1s0.Audit`)**: `audit-jp/` バケット固定、WORM (Write-Once Read-Many) モード、保管期間 5 年、ハッシュチェーンで改ざん検知。
- **業務データ (`k1s0.State` / `k1s0.PubSub`)**: テナントのレジデンシー設定 (Keycloak Realm 属性) に従って配置先リージョンを自動選択する。
- **運用ログ (Loki) / メトリクス (Prometheus) / トレース (Tempo)**: PII を含まない前提で自由配置。OTel Collector の `redaction` プロセッサで PII を除去してから転送する。

### GDPR / 地方法令への対応

EU 顧客を想定する場合、GDPR 第 44 条 (第三国への転送) への回答が必要となる。k1s0 は以下 3 点で答弁する。

- **右を削除 (RTBF)**: `k1s0.Pii.Delete` API が Kafka の tombstone イベントを全 tier に配信し、Postgres / MinIO / Loki から論理削除 + 物理削除 (crypto-shredding: OpenBao Transit Engine の鍵削除) を実施。削除完了は `k1s0.Audit` で証跡化する。
- **データポータビリティ**: `k1s0.Pii.Export` API が JSON-LD 形式で個人データをエクスポート。
- **データ保護影響評価 (DPIA)**: テナント新規作成時に DPIA チェックリストを Backstage の Software Template で自動生成。法務部門の承認をもって Realm 作成を許可する。

## 設計信念との対応

> **tier1 は tier2 を楽にするために、tier2 は tier3 を楽にするために存在する。そうすることで開発効率が向上する。**

この信念は本図にも色濃く現れている。

| 層 | 想定開発者レベル | 図上の表現 |
|---|---|---|
| `tier1` | アーキテクト級 | 公開 API / Dapr CP / Go / Rust / 雛形 CLI が並ぶ最も情報密度の高い帯 |
| `tier2` | ドメイン上級 | サービス Pod と仕様定義の 2 ブロックのみ。ドメインに集中できる |
| `tier3` | 初級〜中級 | 業務アプリ / 配信ポータル / レガシーラップの 3 ブロックのみ。Dapr / OSS を一切意識しない |

下位 tier ほど複雑性を引き受け、上位 tier ほど参入コストが下がる。これは偶然ではなく設計上の意図であり、組織全体の開発スループットを最大化するための構造である。
