# k1s0 全体構成図 (インフラ + アーキテクチャ)

## 目的

k1s0 が作り出すプラットフォームの全体像を **1 枚の図** で把握できるようにする。物理的な実行基盤 (k8s クラスタ / ネットワーク / ストレージ / セキュリティ) と、その上に乗る 4 実行レイヤ (`infra` / `tier1` / `tier2` / `tier3`) と横断運用レイヤ (`operation`) の関係を、1 つの構成図に統合する。

詳細な分割図 (レイヤ構成のみ / 依存ルールのみ / 配置形態のみ) は [`01_アーキテクチャ/`](../02_構想設計/01_アーキテクチャ/) に既に存在するため、本資料はそれらを **横断的に俯瞰する 1 枚** として位置付ける。

## 図の読み方

図は上から下に向かって、外部 → エッジ → クラスタ内部 (上位 tier → 下位 tier) という順序で配置している。横軸方向は左から「業務トラフィック」、右側に「運用 / 開発者向けの operation 列」を独立させている。

### 1. 外部とエッジ

エンドユーザーの端末 (PWA / MAUI / WinUI) は **社内 DNS / VPN** を経由してクラスタの **MetalLB (L2 モード)** に到達する。MetalLB が払い出した VIP を **Envoy Gateway** が受け、TLS 終端と OIDC 認可 (ext_authz → Keycloak) を行ってからクラスタ内部にルーティングする。クラウド LB / マネージド k8s には依存しないため、閉域ネットワーク要件を満たしつつ稟議のハードルを下げる。

`cert-manager` は Edge / ポータルで利用する TLS 証明書を自動発行・自動更新し、運用者の手で証明書を扱う場面を排除する。

開発者 / 運用担当者は同じ Envoy Gateway 経由で `operation` namespace に到達し、Backstage / Argo CD / Keycloak 等のポータル群へ OIDC SSO で入る。

### 2. クラスタと kube-vip

k8s 実装は **kubeadm をメインプラン、k3s をサブプラン** とするセルフマネージド構成を採用する。クラウドマネージド k8s (EKS / AKS / GKE) は閉域ネットワーク要件と整合しないため非採用とする。

Control Plane の高可用性 VIP は **kube-vip** が静的 Pod として各 Master ノードに配置され担保する。kube-vip は namespace に属さないため、図上では cluster の上端に注記として配置している。

### 3. tier3 → tier2 → tier1 → infra (依存ルール)

クラスタ内部の左 / 中央列は、上から **tier3 → tier2 → tier1 → infra** の順で積み上げている。これは k1s0 の **1 方向依存ルール** をそのまま視覚化したものである。各 tier 間を結ぶ縦の矢印は「上位が下位を参照する」流れを示す。

- **tier3** は業務アプリ Pod / アプリ配信ポータル Pod / レガシー .NET ラップ Pod を持つ。すべて `app container + Dapr sidecar` の 2 コンテナ構成だが、**sidecar の管理責任は tier1 にある** ため、tier3 開発者は Dapr を意識しない。
- **tier2** はドメインサービス Pod 群と、画面 / IF / 抽象クラスを定義する仕様レイヤから成る。tier3 が準拠する標準を提供する。
- **tier1** は最も濃い情報量を持つ層で、4 つの構成要素 (公開 API / Dapr Control Plane / tier1 Go サービス / tier1 Rust サービス) と雛形生成 CLI から成る。**`k1s0 公開 API` 帯** が tier2/tier3 から見える唯一の接点であり、その下にある内部実装 (Go / Rust の使い分け) は完全に隠蔽される。
- **infra** はサービスメッシュ・メッセージング・観測性・データストア・セキュリティを 5 つの小グループに整理して配置している。tier1 のみが infra に直接アクセスでき、tier2 / tier3 は infra コンポーネントの URL や SDK を直接参照してはならない。

### 4. operation 列 (横断)

右列の `operation` namespace は実行レイヤではなく、全レイヤに横断的に作用する運用プロセス層である。Backstage が開発者ポータル、Argo CD が GitOps デプロイの起点、Keycloak が OIDC 認可、GitHub Actions self-hosted runner が CI エンジン、Renovate が依存自動更新 PR、Argo Events がイベント駆動 (Phase 2)、Litmus が Chaos Engineering をそれぞれ担う。

operation の各コンポーネントは tier1〜tier3 のどこにでも作用しうるため、図上では cluster 内に縦に独立した列として配置し、業務トラフィックとは線を引かずに視覚的に切り離している。

## 設計信念との対応

> **tier1 は tier2 を楽にするために、tier2 は tier3 を楽にするために存在する。そうすることで開発効率が向上する。**

この信念は本図にも色濃く現れている。

| 層 | 想定開発者レベル | 図上の表現 |
|---|---|---|
| `tier1` | アーキテクト級 | 公開 API / Dapr CP / Go / Rust / 雛形 CLI が並ぶ最も情報密度の高い帯 |
| `tier2` | ドメイン上級 | サービス Pod と仕様定義の 2 ブロックのみ。ドメインに集中できる |
| `tier3` | 初級〜中級 | 業務アプリ / 配信ポータル / レガシーラップの 3 ブロックのみ。Dapr / OSS を一切意識しない |

下位 tier ほど複雑性を引き受け、上位 tier ほど参入コストが下がる。これは偶然ではなく設計上の意図であり、組織全体の開発スループットを最大化するための構造である。
