# ペルソナ

## 目的

k1s0 が想定する利害関係者を具体的な人物像として定義する。機能やスコープの判断時に「この人の問題を解くか」という基準として使う。

> 以下のペルソナはすべて架空のテンプレートであり、特定の個人を指すものではない。実際の組織で紹介資料を提示する際は、現場の実態に合わせて年齢・スキル・痛みを調整すること。

---

## 記載項目の凡例

- **名前 (仮)**: 呼称用の仮名
- **年齢 / 役職**: 想定属性
- **スキル**: 技術経験
- **業務上の痛み**: 現状抱える不満・課題
- **k1s0 への態度**: 推進 / 中立 / 抵抗
- **k1s0 で得られる価値**: 導入後の改善

## ペルソナ間の関係図

k1s0 導入の意思決定は、以下のような力学で進む。

| 役割 | 方向性 | 採用判断者への作用 |
|---|---|---|
| 挑戦者 (森田) | 推進 | 新技術提案の起点 |
| 守護者 (田辺) | 抵抗 | 変化への拒否反応 |
| ドメイン開発 (長谷川) | 支持 | 挑戦者への援護 |
| 要件定義 (青木) | 潜在支持 | 業務価値から採用判断者を援護 |
| 運用 (石田) | 中立 | 運用観点での加点 |
| 採用判断者 (西尾) | 条件付き | 最終判断 |

採用判断者の判断は、**推進側の業務価値・運用価値と、守護者の不安** の両方を同時に受ける構造となる。

---

## まとめ

- k1s0 の紹介資料は「**守護者タイプを排除せず、挑戦者タイプを報いる**」という両立構造を取る必要がある。
- 採用判断者タイプが首肯できる **根拠 (運用コスト / リードタイム / 撤退リスク)** を別途用意する。
- ドメイン開発者・要件定義担当・運用担当は「日常業務が楽になる」という実感で味方にできる。
- 守護者への対抗手段は「既存資産の共存保証」と「学び直しを強制しない」こと。
