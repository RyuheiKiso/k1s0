# 新技術導入リスクへの回答

## 目的

[`00_背景と課題.md`](./00_背景と課題.md) セクション 3 で提起した「新技術を無秩序に取り入れた場合の構造的課題」に対し、k1s0 がどのような設計判断で回答するかを詳細に説明する。

本資料は決裁者・守護者ペルソナ ([`01_ペルソナ.md`](./01_ペルソナ.md)) が「新技術導入のリスク」を指摘した際の回答源として機能する。

---

## 指摘の要約

| # | 指摘 | 本質的な懸念 |
|---|---|---|
| 1 | 新しい技術を取り入れても 5 年後に継続的にメンテナンスするのか | 技術の賞味期限とメンテナンス責任の所在 |
| 2 | 新しい技術や言語を使用して個別にアプリを開発すると組織としての統一感が無くなる | 個別最適の蓄積による全体最適の崩壊 |
| 3 | A さんだけ作れる、メンテナンスできる。ではダメ | 属人化のリスクは技術の新旧を問わない |

これらは正当な懸念であり、k1s0 はこれらを「起きないように祈る」のではなく、**アーキテクチャで構造的に防止する**方針を取る。

---

## 回答 1: 5 年後もメンテナンスし続けられるのか

### 1.1 懸念の構造

新技術導入に対する「5 年後どうするのか」という問いには、実際には 3 つの小問題が含まれている。

| 小問題 | 具体例 |
|---|---|
| OSS プロジェクト自体の存続 | コミュニティの縮小、資金枯渇による開発停止 |
| ライセンスの変更 | 無償 → 有償化 (Redis / Terraform の前例) |
| 破壊的変更への追従 | メジャーバージョンアップによる API 非互換 |

これらすべてに対し、「発生しない」前提で設計してはならない。

### 1.2 k1s0 の回答: 3 層の防御構造

k1s0 は OSS リスクに対し、以下の 3 層で防御する。

#### 第 1 層: OSS の選定段階でリスクを最小化する

採用する OSS は [`../../02_構想設計/03_技術選定/01_俯瞰/00_選定方針.md`](../../02_構想設計/03_技術選定/01_俯瞰/00_選定方針.md) の前提条件を満たすものに限定する。特に以下の 2 点が 5 年後のリスクを構造的に下げる。

| 条件 | 効果 |
|---|---|
| OSI 承認ライセンスであること | BSL / SSPL / RSALv2 のような「ソース公開だが制限あり」のライセンスを排除する |
| CNCF / Apache Foundation / Linux Foundation 等の財団ガバナンス下であること | 単一企業の判断でライセンス変更ができない構造を確保する |

CNCF Graduated プロジェクトが有償化した事例は過去に一度もない。この実績は「5 年後の存続性」に対する最も強い根拠となる。

#### 第 2 層: tier1 封じ込めで影響範囲を局所化する

仮に OSS のリスクが顕在化しても、その影響が組織全体に波及しない構造を作る。

k1s0 では OSS の直接利用を **tier1 のみに限定** し、tier2 / tier3 は tier1 公開 API を経由してのみ OSS の機能を利用する ([`../../02_構想設計/03_技術選定/01_俯瞰/00_選定方針.md`](../../02_構想設計/03_技術選定/01_俯瞰/00_選定方針.md) セクション 5.2)。

```text
tier3 (個別アプリ)
  ↓ tier1 公開 API のみ利用
tier2 (ドメインサービス)
  ↓ tier1 公開 API のみ利用
tier1 (共通基盤) ← OSS の直接利用はここだけ
  ↓
infra (k8s / Kafka / Istio 等)
```

この構造により、OSS の差し替えが必要になっても **tier1 内部の修正だけで完了** し、tier2 / tier3 のコードは一切変更不要となる。

#### 第 3 層: OSS ごとの出口戦略を事前に定義する

すべての OSS を同じ方針で扱わない。OSS の性質に応じて 3 つの戦略を使い分ける ([`../../02_構想設計/03_技術選定/01_俯瞰/17_OSS長期戦略.md`](../../02_構想設計/03_技術選定/01_俯瞰/17_OSS長期戦略.md))。

| 戦略 | 適用対象 | 例 |
|---|---|---|
| A. 永続利用 | CNCF Graduated 等、商用化リスクが極めて低い | Kubernetes / PostgreSQL / Kafka |
| B. 代替 OSS への差し替え | tier1 封じ込めにより差し替え可能で代替候補が存在する | Dapr / Temporal / Keycloak |
| C. 自作への置き換え | k1s0 固有のドメインロジックで規模が小さい | ZEN Engine / 雛形生成 CLI |

各 OSS がどの戦略に該当するかを個別に判定し、代替候補を含めて文書化している。「5 年後に困ったらどうするか」への回答は「今この瞬間に出口を用意済み」である。

### 1.3 差し替えの実例

tier1 封じ込めによる差し替えの手順は以下の通りである。

1. 対象 OSS の代替候補を [`../../02_構想設計/03_技術選定/01_俯瞰/17_OSS長期戦略.md`](../../02_構想設計/03_技術選定/01_俯瞰/17_OSS長期戦略.md) の代替候補表から選定
2. tier1 内部のラッパー実装を差し替え先の API に接続し直す
3. tier2 / tier3 の統合テストで公開 API の挙動が変わらないことを検証
4. Argo Rollouts による Canary リリースで段階的に切り替え

**tier2 / tier3 の開発者は差し替えが起きたことすら知る必要がない**。これが tier1 封じ込めの最大の効果である。

### 1.4 定期的な健全性確認

出口戦略を用意するだけでは不十分であり、定期的に見直す仕組みが必要となる。

| タイミング | 確認内容 |
|---|---|
| 半期ごと | 各 OSS のライセンス・ガバナンス状況の確認、代替候補表の更新 |
| 新規 OSS 採用時 | 戦略分類 (A / B / C) の判定と文書への追記 |
| リスク顕在化時 | 該当 OSS の差し替え手順の即時発動 |

## 回答 2: 組織としての統一感が無くなるのではないか

### 2.1 懸念の構造

「各チームが好きな言語・技術で作ると統一感がなくなる」という指摘は、以下の具体的な問題を内包している。

| 問題 | 結果 |
|---|---|
| 認証の実装がアプリごとに異なる | セキュリティホールが特定アプリにだけ存在する |
| ログのフォーマットがアプリごとに異なる | 障害調査時に横断的な原因追跡ができない |
| 監視の仕組みがアプリごとに異なる | 運用チームがアプリごとに異なる手順を覚える必要がある |
| デプロイ方式がアプリごとに異なる | CI/CD パイプラインが個別に乱立する |

これらは「言語の自由」が原因ではなく、**横断的関心事の管理責任が不在**であることが原因である。

### 2.2 k1s0 の回答: 自由にしてよい層と統一すべき層の分離

k1s0 は tier 構造によって、**技術選択の自由度** と **組織としての統一性** を両立する ([`../../02_構想設計/01_アーキテクチャ/01_基礎/01_レイヤ構成と責務.md`](../../02_構想設計/01_アーキテクチャ/01_基礎/01_レイヤ構成と責務.md))。

| 層 | 統一 / 自由 | 理由 |
|---|---|---|
| infra | **統一** (k8s / Istio / Kafka / LGTMP) | 実行基盤の分裂は運用コストを指数関数的に増大させる |
| tier1 | **統一** (Go + Rust) | 横断的関心事を 1 箇所で管理するための言語統一 |
| tier2 | **自由** (C# / Go) | 既存の業務ドメイン資産の言語を尊重する |
| tier3 | **自由** (C# / Go / MAUI / React) | 業務要件に最適な技術を選択する |

#### 統一される部分 (tier2 / tier3 の開発者が意識しなくても統一される)

以下の横断的関心事は tier1 が一括で提供するため、tier2 / tier3 がどの言語を使っても自動的に統一される。

| 関心事 | tier1 公開 API | 統一の効果 |
|---|---|---|
| 認証 | `k1s0.Auth` | 全アプリが同一の OIDC フロー (Keycloak) を使用する |
| 構造化ログ | `k1s0.Log` | 全アプリのログが同一フォーマットで Loki に集約される |
| 分散トレース / メトリクス | `k1s0.Telemetry` | 全アプリの OTel データが Grafana で一元的に可視化される |
| State 管理 | `k1s0.State` | KV ストアへのアクセスパターンが統一される |
| シークレット | `k1s0.Secrets` | 全アプリが OpenBao 経由で動的シークレットを取得する |
| 監査ログ | `k1s0.Audit` | 全アプリの監査証跡が同一形式で記録される |
| Feature Flag | `k1s0.Feature` | 段階的ロールアウトの仕組みが組織共通となる |

#### 自由にされる部分 (tier2 / tier3 の開発者が選択する)

| 選択可能な事項 | 理由 |
|---|---|
| サービスの実装言語 (C# / Go) | 既存資産の言語を尊重し、移行コストを発生させない |
| クライアントの技術 (MAUI / React) | 業務要件に応じた最適な選択を許容する |
| ドメインモデリングの手法 | 業務の性質に応じて DDD / CQRS 等を開発チームが判断する |

### 2.3 統一を構造で強制する仕組み

「統一すべき」と定めるだけでは形骸化する。k1s0 は以下の 5 層防御で統一を強制する ([`../../02_構想設計/02_tier1設計/02_API契約/03_API設計原則.md`](../../02_構想設計/02_tier1設計/02_API契約/03_API設計原則.md))。

| 防御層 | 手段 | 効果 |
|---|---|---|
| 雛形生成 CLI | `k1s0-scaffold` で生成されるコードに tier1 API への依存が組み込み済み | 初期状態で統一されている |
| Opinionated Default | tier1 API のデフォルト設定が k1s0 の規約に準拠 | 意識しなくても統一される |
| CI ガード | Dapr SDK / Kafka クライアント等の直接 import を CI で検出・拒否 | 統一を迂回するコードが混入しない |
| リファレンス実装 | 模範的な tier2 サービスの実装例を提供 | 「正しい作り方」が明示される |
| コードレビューチェックリスト | tier1 API の利用方法が正しいかを確認する項目 | 人間によるダブルチェック |

### 2.4 「言語を縛らない」と「統一感」は矛盾しない

一見すると「言語自由」と「統一感」は矛盾するように見える。しかし k1s0 が統一するのは**言語ではなく、横断的関心事の実装**である。

| 観点 | 言語を統一するアプローチ | k1s0 のアプローチ |
|---|---|---|
| 統一の対象 | 言語そのもの | 認証・ログ・監視・デプロイの仕組み |
| 統一の手段 | 「全員 Go を使え」 | 「全員 tier1 API を使え」 |
| 既存資産 | 全て書き直し | そのまま共存 |
| 開発者の学習コスト | 新言語の習得 | tier1 API の習得 ([`../../02_構想設計/02_tier1設計/03_開発者体験/07_tier2オンボーディング戦略.md`](../../02_構想設計/02_tier1設計/03_開発者体験/07_tier2オンボーディング戦略.md)) |
| 統一の強制力 | コードレビューに依存 | 雛形 + CI + 構造で自動強制 |

## 回答 3: A さんしか作れない・メンテナンスできない。ではダメ

### 3.1 懸念の構造

属人化の問題は 2 つのレイヤで発生する。

| レイヤ | 問題 | 結果 |
|---|---|---|
| プラットフォーム自体 | k1s0 を構築・運用できるのが起案者 1 名だけ | 起案者が退職すると基盤が崩壊する |
| 個別アプリ | 特定の技術で作られたアプリを担当者 1 名だけが保守する | 担当者の異動・退職で保守不能になる |

k1s0 はこの両方に対して構造的な対策を持つ。

### 3.2 プラットフォーム自体の属人化への向き合い

#### k1s0 のバス係数は本質的に 1 である

k1s0 は起案者個人が業務外で開発している OSS であり、バス係数は本質的に 1 である。これは妥協ではなく設計判断であり、k1s0 を企業プロジェクトとして組織化することを意図的に避けている（[`06_個人開発OSSの系譜.md`](./06_個人開発OSSの系譜.md) 参照）。「k1s0 のバス係数を 2 以上に上げる」ことを目標にすると、k1s0 は段階導入や体制計画を必要とする企業プロジェクトに変質してしまうため、別の構造で属人化リスクを軽減する。

#### Apache 2.0 OSS 公開で採用側の選択肢を制度保証する

属人化リスクの軽減は、組織体制の拡大ではなく以下 3 点の構造で行う。

- **GitHub 公開 + Apache 2.0 ライセンス**: 採用側は起案者が開発を止めた場合もフォーク可能。Apache 2.0 ライセンスは取消不能（irrevocable）のため、既得の利用権は失われない。コードベースを完全に手元に持てる構造が、ベンダーロックイン回避と属人化リスク軽減の両方に効く
- **dogfooding の継続**: 起案者個人が日常的に使い続け、会社が業務利用することで rrrspec / Hystrix のような「使われなくなる罠」を構造的に回避する（[`06_個人開発OSSの系譜.md`](./06_個人開発OSSの系譜.md) 第 3 節）
- **採用側の内製化準備**: 採用側（会社）は自身でフォーク・内製化する余地を確保する責任を持つ。k1s0 は Apache 2.0 OSS としてこの選択肢を制度的に保証する

会社が k1s0 を業務利用する以上、運用責任は会社側にある。SLO/SLA は k1s0 として保証しない。

#### 起案者と採用側の役割分離

起案者と採用側（会社）の役割は固定的で、時間軸に応じて変化するものではない。

| 主体 | 役割 |
|---|---|
| 起案者 | k1s0 本体の設計・実装・改善・OSS としての公開維持 |
| 採用側（会社） | k1s0 の業務適用・運用・SLO 保証・障害対応・自社用カスタマイズ・内製化準備 |

採用側の運用体制は採用側で設計する。k1s0 として体制計画は提供しない。

### 3.3 個別アプリの属人化防止

tier 構造は個別アプリレベルの属人化も構造的に緩和する。

#### 横断的関心事の属人化が消える

従来は各アプリの開発者が認証・ログ・監視を個別に実装していた。この部分は**担当者しか理解できないコード**になりやすく、属人化の温床となっていた。

k1s0 では横断的関心事はすべて tier1 が提供するため、個別アプリの開発者はビジネスロジックだけを実装する。結果として以下の変化が起きる。

| 観点 | 従来 | k1s0 |
|---|---|---|
| 認証の実装 | アプリごとに独自実装。担当者しか分からない | `k1s0.Auth` を呼ぶだけ。全アプリ共通 |
| ログ設計 | アプリごとにフォーマットが異なる | `k1s0.Log` のフォーマットに統一 |
| 新メンバーの引き継ぎ | 認証・ログ・監視の仕組みから説明が必要 | tier1 API の使い方は共通知識。ビジネスロジックだけ引き継げばよい |

#### 雛形生成による初期構造の統一

`k1s0-scaffold` で生成されるサービスの構造は全て同一のパターンに従う。これにより「A さんが独自に作った構造」が生まれにくくなる。

新しいメンバーが既存のサービスを保守する場合、雛形から生成された共通構造を基点に理解を始められるため、**サービス固有の構造を 1 から学ぶコストが大幅に下がる**。

#### オンボーディング戦略による参入障壁の低減

[`../../02_構想設計/02_tier1設計/03_開発者体験/07_tier2オンボーディング戦略.md`](../../02_構想設計/02_tier1設計/03_開発者体験/07_tier2オンボーディング戦略.md) では、tier2 開発者の習熟度を 5 段階 (L1〜L5) に分け、各段階に対応するサンプル・橋渡しドキュメント・確認課題を提供する。

これにより「A さんに教えてもらわないと分からない」状態を排除し、**ドキュメントとサンプルを通じて誰でも自力で習得できる**環境を整備する。

#### チャンピオン制度による知識の分散

[`../../02_構想設計/02_tier1設計/03_開発者体験/12_開発者アダプション戦略.md`](../../02_構想設計/02_tier1設計/03_開発者体験/12_開発者アダプション戦略.md) で定義するチャンピオン制度は、各チームに 1 名以上の「k1s0 を使いこなせる推進者」を配置する。これにより技術知識が特定個人に集中することを防ぎ、チーム内での知識の自然な伝播を促進する。

### 3.4 OSS の活用自体がバス係数を上げる

自作コンポーネントはチーム内でしか保守できないが、OSS はコミュニティ全体がメンテナンスを担う。つまり OSS のバス係数は事実上無限大である ([`../../02_構想設計/03_技術選定/01_俯瞰/17_OSS長期戦略.md`](../../02_構想設計/03_技術選定/01_俯瞰/17_OSS長期戦略.md) セクション 2.2)。

k1s0 が OSS を積極的に活用する理由の一つは、**自作範囲を最小化することでプロジェクト全体のバス係数を維持する**ことにある。

| アプローチ | 自作範囲 | バス係数 |
|---|---|---|
| すべて自作 | 最大 | チーム人数に依存 (1〜数名) |
| OSS + 最小限の自作 | 最小 | 自作部分のみチーム依存、残りはコミュニティ |
| 商用製品 | ゼロ (ブラックボックス) | ベンダーに完全依存 |

k1s0 は 2 番目のアプローチを取り、自作する領域を戦略 C に分類される ZEN Engine 統合と雛形生成 CLI に限定している。いずれも数千〜数万行の規模であり、チームで保守可能な範囲に収まる。

## まとめ: 3 つの指摘への構造的回答

| 指摘 | k1s0 の回答 | 根拠となる設計判断 |
|---|---|---|
| 5 年後もメンテナンスし続けられるのか | OSS の選定基準 + tier1 封じ込め + OSS ごとの出口戦略で「5 年後に困らない構造」を今作る | 選定方針 / OSS 長期戦略 / tier1 限定の原則 |
| 組織としての統一感が無くなる | tier 構造で「統一すべき層」と「自由にしてよい層」を分離する。言語は自由だが横断的関心事は統一 | レイヤ構成と責務 / API 設計原則 (5 層防御) |
| A さんしか作れない。ではダメ | k1s0 本体は個人 OSS としてバス係数 1 を許容し Apache 2.0 公開で採用側の内製化余地を保証。個別アプリは雛形・オンボーディング・チャンピオン制度で参入障壁を下げる | 個人開発 OSS の系譜 / オンボーディング戦略 / 開発者アダプション戦略 |

これらの回答は「個別の対策を積み重ねた」ものではなく、**tier 構造という単一のアーキテクチャ判断から自然に導出される**。tier1 が横断的関心事と OSS リスクを一手に引き受ける構造が、5 年後の継続性・組織の統一感・属人化の防止のすべてに効いている。
