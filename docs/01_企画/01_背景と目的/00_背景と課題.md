# 背景と課題

## 目的

k1s0 が解決しようとしている JTC (Japanese Traditional Company) 情報システム部門の技術的負債と構造的な課題を整理し、「なぜ新しいプラットフォームが必要なのか」の根拠を明示する。

---

## 1. 技術的負債のスパイラル

多くの JTC 情シス部門では、以下の負のループが長年にわたり繰り返されている。

1. 古い単一技術でシステムを作り込む
2. 新技術への移行が困難になる
3. 新技術を提案しても「よくわからないからダメ」で却下される
4. 学習投資をした若手が評価されない
5. 古参が既存技術に固執する
6. 結果としてさらに新技術への移行が困難になる

このループは時間が経つほど強化され、**単一技術ごとのモノレポの集合** が組織の資産として積み上がっていく。それぞれは動作しているが全体を俯瞰すると「崩れかけのジェンガ」に似た状態となり、1 箇所の変更が予期しない箇所の故障を誘発する。

---

## 2. 典型的な 4 つの痛み

| 痛み | 影響 |
|---|---|
| レガシー .NET Framework 資産が動き続けている | 捨てるに捨てられず、新規開発の足を引っ張る |
| 横断的関心事 (認証 / ログ / 監視) のコピペ実装 | 各プロジェクトが同じコードを量産し、業務ロジックに集中できない |
| 端末への手動アプリインストール | PC リプレース時に情シスが数人月単位で消耗する |
| 商用基盤の高額ライセンス / ベンダーロックイン | 稟議が通らない、あるいは契約後の撤退コストが膨大 |

これらは個々に見ると「よくある話」だが、**組み合わせが JTC 情シス部門特有の状況** を形成している。単一問題ではないため、部分的な改善では効果が出にくい。

---

## 3. 新技術導入の裏に潜むもう一つの罠

セクション 1 ～ 2 では「古い技術に固執し続ける」ことの問題を述べた。しかし逆に、新技術を無秩序に取り入れた場合にも別の構造的課題が生じる。JTC 情シスが新技術導入を躊躇する背景には、以下の正当な懸念がある。

### 3.1 5 年後もメンテナンスし続けられるのか

新しい技術やフレームワークを採用しても、5 年後にその技術のエコシステムが存続している保証はない。OSSプロジェクトの停滞、メジャーバージョンの破壊的変更、コミュニティの縮小といったリスクに対して「誰がどのように追従し続けるのか」を答えられなければ、導入は単なるリスクの先送りになる。

### 3.2 個別最適が組織の統一感を破壊する

チームや担当者がそれぞれ好みの技術・言語でアプリケーションを開発すると、短期的には生産性が上がったように見える。しかし中長期では、認証・ログ・監視・デプロイなどの横断的関心事がアプリごとにバラバラとなり、組織としての技術基盤が存在しない状態に陥る。結果として運用コストが膨れ上がり、障害対応やセキュリティ対策が後手に回る。

### 3.3 属人化の再生産 — 「A さんしか作れない」問題

新技術を導入しても、その技術を扱えるのが特定の 1 名だけでは、レガシー時代の属人化をそのまま繰り返すことになる。「A さんが退職したらメンテナンスできない」状態は、使用技術が新しいか古いかに関係なく等しくリスクである。技術選定と同時に「複数人が保守できる体制」を設計しなければ、属人化は形を変えて再発する。

### 3.4 k1s0 はこの罠をどう回避するか

k1s0 はこれらの懸念に対し、以下の構造で応える。

| 懸念 | k1s0 の回答 |
|---|---|
| 5 年後の継続性 | tier1 が OSS の選定・追従・置換を一手に担い、tier2 / tier3 は公開 API のみに依存するため影響を局所化できる ([`../04_技術選定/17_OSS長期戦略.md`](../04_技術選定/17_OSS長期戦略.md)) |
| 組織の統一感 | 横断的関心事はすべて tier1 に集約し、tier2 / tier3 は言語自由だが基盤は統一される ([`../02_アーキテクチャ/01_レイヤ構成と責務.md`](../02_アーキテクチャ/01_レイヤ構成と責務.md)) |
| 属人化の再生産 | tier1 の内部言語は Go + Rust に限定し、クライアントライブラリ + ドキュメント + 雛形生成で参入障壁を下げる ([`../03_tier1設計/07_tier2オンボーディング戦略.md`](../03_tier1設計/07_tier2オンボーディング戦略.md)) |

つまり「言語を縛らない」と「統一感を保つ」は矛盾しない。tier 構造によって **自由にしてよい層** と **統一すべき層** を分離することが、k1s0 のアーキテクチャ上の核心的な判断である。各懸念への詳細な回答は [`03_新技術導入リスクへの回答.md`](./03_新技術導入リスクへの回答.md) を参照。

## 4. 既存の選択肢が JTC に刺さらない理由

マイクロサービス基盤や Internal Developer Platform (IDP) 製品は既に多数存在するが、JTC 情シス部門に直接は刺さらない。理由は以下のとおり。

| 既存製品の前提 | JTC 情シス部門の実情 |
|---|---|
| クラウドマネージド前提 (EKS / AKS / GKE 等) | オンプレ / VMware / 閉域ネットワークが主流 |
| サブスクリプション契約が稟議を通る | 高額な年次契約は稟議のハードルが極めて高い |
| 言語を Go / TypeScript に統一できる | .NET Framework / C# 資産を捨てられない |
| 開発者向け UI を重視 | 運用担当・要件定義担当・決裁者の目線が揃わない |
| 内製開発文化を前提 | 情報システム子会社・ベンダー委託が混在 |

既存 OSS IDP (Backstage 等) は優秀だが、**「実行基盤側を別途構築する」** 前提となっており、JTC 情シスが単体で導入しても効果を引き出しきれない。

## 5. 解決の方向性

上記の課題に対し、k1s0 は以下の原則で応える。詳細は [`02_解決する価値.md`](./02_解決する価値.md) で展開する。

- **OSS 積み上げで無償** — 稟議のハードルをゼロに近づける
- **オンプレ / VM で完結** — クラウド接続を前提にしない
- **レガシー共存を第一級で扱う** — .NET Framework 資産を捨てさせない
- **言語を縛らない** — 既存の開発資産 (C# / Go / React / MAUI 等) を尊重する
- **金銭的メリットで参加を促す** — 強制移行ではなく、運用コスト削減を訴求する
