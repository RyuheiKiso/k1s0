# k1s0 が解決する価値

## 目的

[`00_背景と課題.md`](./00_背景と課題.md) で整理した JTC 情シス部門の課題に対し、k1s0 がどのような価値を提供するかを説明する。ペルソナ別の訴求ポイントも併記する。

---

## 1. k1s0 のコアメッセージ

> **「OSS 積み上げで、JTC 情シスでも運用できる、レガシーと共存できる、ベンダーに縛られない開発プラットフォーム」**

k1s0 はこの 1 文を一貫して繰り返し、機能ではなく価値観で支持者を増やすことを方針とする。

---

## 2. 5 つの勝ち筋

| # | 勝ち筋 | 響く対象 | 根拠 |
|---|---|---|---|
| 1 | **無償で始められる** | 決裁者 / 情シス課長 | 稟議のハードルゼロ。OSI 承認 OSS のみで構成 |
| 2 | **レガシーと共存できる** | 守護者 / 現場 | .NET Framework 資産を tier3 拡張ポイントとして共存 |
| 3 | **オンプレ / VM で完結する** | 運用 / セキュリティ担当 | クラウド依存なし。閉域ネットワーク対応 |
| 4 | **言語を尊重する** | 開発者 / 守護者 | tier2 / tier3 は C# / Go / MAUI / React 自由 |
| 5 | **金銭的メリットで参加を促す** | 守護者 / レガシー所有者 | 運用コスト削減・リードタイム短縮を訴求 |

これらは個別に見ると地味だが、**5 点セットで競合製品の差別化要素** を成す。詳細は [`../06_競合と差別化/02_差別化ポイント.md`](../06_競合と差別化/02_差別化ポイント.md) を参照。

---

## 3. 誰が何を得るか

ペルソナ別に、k1s0 から得られる具体的な価値を整理する。ペルソナの定義は [`01_ペルソナ.md`](./01_ペルソナ.md) を参照。

### 3.1 守護者 (田辺) が得るもの

- 既存 .NET Framework モジュールを捨てずに共存できる
- 学習投資を強制されない
- アプリ配信ポータル経由で既存資産が配信対象となり、評価対象に残る

### 3.2 挑戦者 (森田) が得るもの

- 新技術を使った貢献が正当に評価される
- プライベート学習が業務成果に直結する
- Rust / Go / k8s の実務経験を積める場が社内にできる

### 3.3 決裁者 (西尾) が得るもの

- 運用コスト削減を数値で説明できる
- ベンダーロックインを回避でき、将来の撤退コストがゼロ
- 属人化解消により退職リスクを下げられる
- 「商用 IDP を買うより k1s0 を内製する方が TCO が低い」を示せる (詳細は [`../06_競合と差別化/03_TCOとBuildVsBuy.md`](../06_競合と差別化/03_TCOとBuildVsBuy.md))

### 3.4 運用担当 (石田) が得るもの

- 可観測性が統一され、ログ・トレース・メトリクスが一元管理される
- 夜間対応の属人化が解消される
- 端末リプレース時の設定復元が自動化される ([`../05_CICDと配信/02_アプリ配信ポータル.md`](../05_CICDと配信/02_アプリ配信ポータル.md) 参照)

### 3.5 ドメイン開発者 (長谷川) が得るもの

- tier1 が認証 / ログ / 監視 / 監査を提供するため、業務ロジックに集中できる
- 雛形生成 CLI でセットアップが数分で終わる
- コードレビューで「横断的関心事」の議論が消える

### 3.6 要件定義担当 (青木) が得るもの

- マイクロサービス単位での機能追加によりリードタイムが短縮する
- 決定表 (ZEN Engine) により業務ルールを自分で編集できる余地が生まれる
- 優先度の変更がシステム実装に素早く反映される

## 4. 価値を提供するメカニズム

k1s0 がこれらの価値を提供できる構造的な根拠は以下の 4 点である。いずれも具体的な設計上の仕掛けに裏打ちされている。

### 4.1 tier 構造と依存方向の 1 方向化

- tier3 → tier2 → tier1 → infra の 1 方向依存
- 横断的関心事はすべて tier1 に閉じ込める
- 詳細: [`../02_アーキテクチャ/01_レイヤ構成と責務.md`](../02_アーキテクチャ/01_レイヤ構成と責務.md)

### 4.2 Dapr の隠蔽と多言語クライアントライブラリ

- Dapr は tier1 の内部実装として組み込み、tier2 / tier3 からは完全に隠蔽する
- tier2 / tier3 は tier1 公開 API (C# / Go / TS) のみを利用
- 詳細: [`../03_tier1設計/01_Dapr隠蔽方針.md`](../03_tier1設計/01_Dapr隠蔽方針.md)

### 4.3 決定表 (ZEN Engine) によるルールの外出し

- 権限ポリシー・稟議承認・配信可視化などを JSON Decision Model で宣言する
- 業務担当や情シス担当が直接編集できる領域を意図的に広げる
- 詳細: [`../04_技術選定/03_ルールエンジン.md`](../04_技術選定/03_ルールエンジン.md)

### 4.4 アプリ配信ポータル

- 情シスの端末訪問を撤廃し、ユーザーが自分でインストールする UX を提供する
- PC リプレース時の設定復元を自動化する
- 詳細: [`../05_CICDと配信/02_アプリ配信ポータル.md`](../05_CICDと配信/02_アプリ配信ポータル.md)

---

## 5. 提供しないもの (スコープ外)

k1s0 は万能ではない。以下は意図的に扱わない。

| 扱わないもの | 理由 |
|---|---|
| クラウドマネージドサービスの代替 | 前提が「オンプレ / VM」 |
| Windows Active Directory の完全置換 | 既存 AD への連携で十分 |
| BI / データ分析基盤 | k1s0 は業務アプリ実行基盤 |
| エンドユーザーコンピューティング (RPA) | 業務アプリから呼び出す形で連携 |
| セキュリティ監査製品の置換 | tier1 監査ログ + 既存の SIEM 連携で対応 |

スコープを絞ることで、開発リソースを「**JTC 情シス固有の痛み**」に集中させる。
