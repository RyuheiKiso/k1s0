# 障害復旧とバックアップ

## 目的

k1s0 プラットフォームを構成する各コンポーネントのバックアップ戦略・復旧手順・RTO (Recovery Time Objective) / RPO (Recovery Point Objective) を定義する。JTC 情シスが運用可能な手順レベルに落とし込むことを目標とする。

---

## 1. 設計原則

1. **GitOps が最大の復旧手段** — k8s マニフェスト / 設定はすべて Git に格納。クラスタが全壊しても Git + IaC から再構築可能にする
2. **ステートフルコンポーネントのみバックアップ対象** — ステートレスサービスは再デプロイで復旧
3. **自動化を前提** — 手動バックアップは運用漏れの温床。CronJob / Operator で自動化する
4. **復旧手順はテスト済みでなければ信頼しない** — 定期的にリストア訓練を実施する
5. **ストレージ冗長性は Longhorn で担保** — ローカル PV のノード障害リスクを排除。Longhorn の 3 ノードレプリケーションで PV の可用性を確保する

---

## 2. コンポーネント別のバックアップ戦略

### 2.1 PostgreSQL (CloudNativePG)

PostgreSQL は k1s0 における最重要のステートフルコンポーネント。Keycloak / Backstage / Argo CD / Harbor / 業務サービスのデータを保持する。

| 項目 | 方針 |
|---|---|
| バックアップツール | CloudNativePG の組み込みバックアップ (`barman-cloud`) |
| バックアップ種別 | 物理バックアップ (base backup) + WAL アーカイブ (継続的) |
| バックアップ先 | MinIO (S3 互換)。MinIO の PV は Longhorn がレプリケーションで保護 |
| 頻度 | base backup: 日次、WAL: 継続的 (秒単位) |
| RPO | **数秒〜数十秒** (WAL アーカイブ遅延分) |
| RTO | **15 分以内** (base backup からのリストア + WAL リプレイ) |
| 保持期間 | base backup: 7 日分、WAL: 7 日分 |
| 暗号化 | バックアップファイルの AES 暗号化 (barman-cloud 標準機能) |

### 2.2 etcd (k8s クラスタ状態)

| 項目 | 方針 |
|---|---|
| バックアップツール | `etcdctl snapshot save` |
| バックアップ先 | ローカルディスク + 別ノードへのコピー |
| 頻度 | 日次 (CronJob) |
| RPO | **24 時間** (日次バックアップ間隔) |
| RTO | **30 分〜1 時間** (スナップショットリストア + k8s リカバリ) |
| 保持期間 | 7 日分 |
| 補足 | k8s マニフェストは Git にあるため、etcd の完全消失でも Git + IaC からクラスタ再構築が可能。etcd バックアップは「時間短縮のための保険」 |

### 2.3 Apache Kafka (Strimzi)

| 項目 | 方針 |
|---|---|
| 耐障害性 | Kafka 自体のレプリケーション (`replication.factor=3`) が主な防御策 |
| バックアップ | MirrorMaker 2 による別クラスタへのミラーリング (Phase 3 以降) |
| RPO | **レプリケーション遅延分** (通常ミリ秒単位) |
| RTO | **ブローカー再起動: 5 分以内** / クラスタ全壊: 30 分 (トピック再作成 + Consumer リセット) |
| 保持期間 | トピックの `retention.ms` に従う (デフォルト 7 日) |
| MVP での方針 | 3 broker KRaft 構成で 1 broker 障害に耐える。バックアップツールは Phase 2 以降 |

### 2.4 Harbor (コンテナレジストリ)

| 項目 | 方針 |
|---|---|
| バックアップ対象 | Harbor DB (PostgreSQL) + イメージストレージ |
| バックアップ方法 | DB: CloudNativePG バックアップに含まれる / ストレージ: MinIO (S3 互換) にイメージを保存。MinIO 自体のデータは別ノードへのレプリケーション or PV スナップショットで保護 |
| RPO | DB: 数秒 (WAL) / ストレージ: 日次 |
| RTO | 30 分 |
| 補足 | イメージは CI で再ビルド可能なため、ストレージの完全消失は致命的ではない |

### 2.5 Keycloak

| 項目 | 方針 |
|---|---|
| バックアップ対象 | Keycloak の DB (PostgreSQL 内の `keycloak` DB) + Realm Export |
| バックアップ方法 | DB: CloudNativePG バックアップに含まれる / Realm: `kc.sh export` を CronJob で日次実行 |
| RPO | DB: 数秒 / Realm Export: 24 時間 |
| RTO | 15 分 (DB リストア) / Realm Export からのリカバリ: 30 分 |
| 補足 | Realm Export は JSON ファイルとして Git に格納し、バージョン管理する |

### 2.6 OpenBao (シークレット管理)

| 項目 | 方針 |
|---|---|
| バックアップ対象 | OpenBao の Raft ストレージ (全 Secret・ポリシー・Engine 設定) |
| バックアップ方法 | `bao operator raft snapshot save` を CronJob で日次実行 |
| バックアップ先 | MinIO (S3 互換)。暗号化状態のまま保存 |
| 頻度 | 日次 |
| RPO | **24 時間** (日次スナップショット間隔) |
| RTO | **15 分** (スナップショットリストア + Unseal) |
| Unseal 鍵 | Shamir Secret Sharing (3-of-5) で分散保管。鍵自体は OpenBao の外部で安全に管理 |
| 補足 | Raft 構成 (3 Pod) で 1 Pod 障害は自動復旧。全 Pod 消失時のみスナップショットリストアが必要 |

### 2.7 Backstage

| 項目 | 方針 |
|---|---|
| バックアップ対象 | Backstage の DB (PostgreSQL 内の `backstage` DB) |
| バックアップ方法 | CloudNativePG バックアップに含まれる |
| RPO / RTO | PostgreSQL と同一 |
| 補足 | Software Catalog の定義は `catalog-info.yaml` として Git にあるため、DB 消失時は再インデックスで復旧可能 |

## 3. バックアップ先の設計

### 3.1 ストレージ構成

| バックアップ先 | 用途 | 導入時期 |
|---|---|---|
| Longhorn (分散ブロックストレージ) | CloudNativePG / MinIO / Harbor / etcd スナップショットの PV | Phase 1 |
| MinIO (オンプレオブジェクトストレージ) | PostgreSQL barman-cloud、Harbor ストレージ | Phase 1 |
| 外部 NFS / NAS | 長期保管 (コールドバックアップ) | Phase 2 |

### 3.2 3-2-1 ルールの適用

- **3 コピー**: 本番データ + MinIO バックアップ + 外部 NFS
- **2 種類の媒体**: SSD (本番) + オブジェクトストレージ (MinIO) + NAS (外部)
- **1 つは別拠点**: Phase 3 以降でリモートサイトへの複製を検討

MVP では MinIO を同一クラスタ上で稼働させ、最低限の冗長性を確保する。

## 4. 障害シナリオと復旧手順

### 4.1 シナリオ別の対応

| シナリオ | 影響範囲 | 復旧手順 | 目標 RTO |
|---|---|---|---|
| 単一 Pod の障害 | 該当サービスのみ | k8s が自動再起動 (ReplicaSet) | 1 分以内 |
| 単一ノードの障害 | そのノード上の全 Pod | k8s がスケジューリングで別ノードに再配置 | 5 分以内 |
| PostgreSQL プライマリ障害 | DB 依存の全サービス | CloudNativePG が自動フェイルオーバー (レプリカ昇格) | 30 秒以内 |
| Kafka ブローカー 1 台障害 | メッセージング遅延 | レプリケーションで自動復旧 | 数秒 |
| etcd クォーラム喪失 | k8s API 全停止 | etcd スナップショットからのリストア | 30 分〜1 時間 |
| k8s クラスタ全壊 | 全サービス停止 | OpenTofu で VM 再構築 → Git からマニフェスト適用 | 2〜4 時間 |
| ストレージ障害 (データ消失) | 該当 PV のサービス | barman-cloud からの PostgreSQL リストア | 15 分〜30 分 |

### 4.2 クラスタ全壊時の復旧手順 (概要)

1. `tofu apply` で VM 3 台を再構築
2. kubespray (kubeadm) で k8s クラスタを再構築 (サブプラン: k3s)
3. etcd スナップショットがあればリストア (なければ空クラスタから)
4. Argo CD をインストールし、GitOps リポを登録
5. Argo CD が全マニフェストを同期 → 全サービスが再デプロイ
6. PostgreSQL を barman-cloud バックアップからリストア
7. Keycloak / Backstage / Harbor のデータが復旧
8. 動作確認 (Keycloak SSO → 各ツール → サンプルサービス)

この手順は Phase 1 完了時に TechDocs として整備し、**起案者以外の人が実行可能** な状態にする。
