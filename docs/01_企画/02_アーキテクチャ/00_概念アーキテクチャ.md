# 概念アーキテクチャ

## 目的

k1s0 の全体像を 1 枚の図で把握できるようにする。企画書の導入部に流用する。本資料は概念レベルに留め、物理ノード配置・ストレージ構成・ネットワーク詳細は扱わない。

---

## 1. 全体俯瞰

k1s0 は 4 つの実行レイヤ (`infra` / `tier1` / `tier2` / `tier3`) と、それらを横断する運用レイヤ (`operation`) から成る。

![レイヤ構成](./img/レイヤ構成.svg)

### レイヤの役割 (概要)

| レイヤ | 役割 | 担当チーム |
|---|---|---|
| `infra` | k8s クラスタと周辺コンポーネント (Istio / Kafka / 観測基盤) | インフラチーム |
| `tier1` | 共通基盤サービス。公開 API と内部実装 (Go + Rust ハイブリッド) からなる | システム基盤チーム |
| `tier2` | 業務ドメインロジック | ドメイン開発チーム |
| `tier3` | エンドユーザー向け UI / API / アプリ配信ポータル | 個別アプリ開発チーム |
| `operation` | 運用手順・監視・オンコール・リリース管理 (横断) | 運用チーム + システム基盤チーム |

詳細な責務と管轄ディレクトリは [`01_レイヤ構成と責務.md`](./01_レイヤ構成と責務.md) で展開する。

---

## 2. 依存ルール

k1s0 は tier 間の依存を **1 方向のみ** に制約する。

![依存ルール](./img/依存ルール.svg)

- tier3 は原則 tier2 を経由して tier1 を利用する
- tier2 が存在しない場合に限り、tier3 は tier1 を直接利用できる
- **infra への直接依存は tier1 のみに許可する**
- tier2 / tier3 は Dapr SDK / infra コンポーネント (Kafka クライアント等) を直接 import / 呼び出してはならない

この 1 方向依存により、将来的な infra 置換 (Kafka → NATS、Loki → Tempo 等) や Dapr のバージョン更新の影響を **tier1 内に閉じ込められる**。詳細は [`02_依存ルールと通信経路.md`](./02_依存ルールと通信経路.md) を参照。

---

## 3. 配置形態

k8s クラスタ内では、以下の 5 つの namespace に各レイヤを配置する。

![配置形態](./img/配置形態.svg)

- 各 namespace は `infra` / `tier1` / `tier2` / `tier3` / `operation` に対応
- クラウドマネージド (EKS / AKS / GKE) は採用しない。セルフマネージドな k8s (`kubeadm` / `k3s` / `RKE2` 等) を想定する
- tier2 / tier3 の Pod には Dapr sidecar が自動注入されるが、sidecar の管理責任は tier1 チームにある

詳細は [`03_配置形態.md`](./03_配置形態.md) を参照。
