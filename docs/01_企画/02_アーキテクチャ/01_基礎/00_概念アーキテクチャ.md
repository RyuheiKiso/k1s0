# 概念アーキテクチャ

## 目的

k1s0 の全体像を 1 枚の図で把握できるようにする。企画書の導入部に流用する。本資料は概念レベルに留め、物理ノード配置・ストレージ構成・ネットワーク詳細は扱わない。

---

## 1. 全体俯瞰

k1s0 は 4 つの実行レイヤ (`infra` / `tier1` / `tier2` / `tier3`) と、それらを横断する運用レイヤ (`operation`) から成る。

![レイヤ構成](./img/レイヤ構成.svg)

### レイヤの役割 (概要)

| レイヤ | 役割 | 担当チーム | 想定開発者レベル |
|---|---|---|---|
| `infra` | k8s クラスタと周辺コンポーネント (Istio / Kafka / 観測基盤) | インフラチーム | — |
| `tier1` | 共通基盤サービス。公開 API と内部実装 (Go + Rust ハイブリッド) からなる | システム基盤チーム | アーキテクト級 |
| `tier2` | 業務ドメインロジック | ドメイン開発チーム | ドメイン寄りの上級 |
| `tier3` | エンドユーザー向け UI / API / アプリ配信ポータル | 個別アプリ開発チーム | 初級〜中級 |
| `operation` | 運用手順・監視・オンコール・リリース管理 (横断) | 運用チーム + システム基盤チーム | — |

詳細な責務と管轄ディレクトリは [`01_レイヤ構成と責務.md`](./01_レイヤ構成と責務.md) で展開する。

## 2. 設計信念

> **tier1 は tier2 を楽にするために、tier2 は tier3 を楽にするために存在する。そうすることで開発効率が向上する。**

k1s0 のレイヤ構造は「上位層が下位層を利用する」だけの技術的な依存関係ではない。各 tier は **自分の上位 tier を楽にすることを存在目的** とする。

- **tier1 → tier2 を楽にする**: 横断的関心事 (認証 / ログ / 監視 / 監査 / state 管理) を tier1 が引き受けることで、tier2 の開発者は業務ドメインロジックだけに集中できる
- **tier2 → tier3 を楽にする**: 業務ロジックが tier2 に整理されていることで、tier3 の開発者は UI / UX の実装に専念できる。ビジネスルールの重複実装や不整合が起きない

この信念は「どの tier を優先的に改善すべきか」の判断基準にもなる。tier1 の API が使いづらければ tier2 の全チームが苦しみ、tier2 のドメインモデルが不明瞭なら tier3 の全アプリが混乱する。**下位 tier の品質が上位 tier の開発効率を決定する** という因果関係を、チーム全員が共有する。

### 開発者レベルの勾配

この設計信念は、各 tier に求められる開発者レベルにも直結する。

| tier | 想定開発者レベル | 理由 |
|---|---|---|
| `tier1` | アーキテクト級 | 横断的関心事の抽象化・infra 層との統合・多言語クライアント設計など、システム全体を見通す判断力が必要 |
| `tier2` | ドメイン寄りの上級 | 業務ドメインの深い理解と、それを適切にモデリングする設計力が必要。infra の詳細は tier1 が隠蔽するため、ドメインに集中できる |
| `tier3` | 初級〜中級 | tier1 と tier2 が複雑性を吸収するため、tier3 の開発者は UI / UX の実装に専念できる。参入障壁を下げることで開発リソースをスケールさせやすくなる |

**tier の番号が上がるほど、必要な開発者レベルが下がる**。これは偶然ではなく、設計上の意図である。下位 tier が複雑性を引き受けることで、上位 tier の参入コストを下げ、組織全体の開発スループットを最大化する。

## 3. 依存ルール

k1s0 は tier 間の依存を **1 方向のみ** に制約する。

![依存ルール](./img/依存ルール.svg)

- tier3 は原則 tier2 を経由して tier1 を利用する
- tier2 が存在しない場合に限り、tier3 は tier1 を直接利用できる
- **infra への直接依存は tier1 のみに許可する**
- tier2 / tier3 は Dapr SDK / infra コンポーネント (Kafka クライアント等) を直接 import / 呼び出してはならない

この 1 方向依存により、将来的な infra 置換 (Kafka → NATS、Loki → Tempo 等) や Dapr のバージョン更新の影響を **tier1 内に閉じ込められる**。詳細は [`02_依存ルールと通信経路.md`](./02_依存ルールと通信経路.md) を参照。

## 4. 配置形態

k8s クラスタ内では、以下の 5 つの namespace に各レイヤを配置する。

![配置形態](./img/配置形態.svg)

- 各 namespace は `infra` / `tier1` / `tier2` / `tier3` / `operation` に対応
- クラ��ドマネージド (EKS / AKS / GKE) は採用しない。セルフマネージドな k8s (`kubeadm` メイン / `k3s` サブプラン) を想定する
- tier2 / tier3 の Pod には Dapr sidecar が自動注入されるが、sidecar の管理責任は tier1 チームにある

詳細は [`03_配置形態.md`](./03_配置形態.md) を参照。
