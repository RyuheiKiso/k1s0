# セキュリティモデル

## 目的

k1s0 プラットフォーム全体のセキュリティ設計を整理する。認証・認可・ネットワーク境界・データ分類・脅威モデリングの各観点で、tier 構成に即した防御策を定義する。

---

## 1. セキュリティ設計の原則

1. **ゼロトラスト前提** — ネットワーク境界を信頼せず、全通信を認証・暗号化する
2. **最小権限** — 各サービス / ユーザーは業務に必要な最小限の権限のみを保持する
3. **多層防御** — 単一の防御策の突破が全体の侵害に直結しない構造を作る
4. **監査可能性** — 全ての認証・認可イベントをログに記録し、事後追跡を可能にする
5. **tier 依存ルールの厳守** — セキュリティ設定も tier 間の 1 方向依存に従う

---

## 2. 認証基盤

### 2.1 Keycloak を中心とした認証統一

| 対象 | 認証方式 | Keycloak との関係 |
|---|---|---|
| エンドユーザー (配信ポータル) | OIDC (Authorization Code Flow + PKCE) | Keycloak Realm `k1s0` で認証 |
| 開発者 (Backstage / Argo CD / Harbor) | OIDC (Authorization Code Flow) | 同一 Realm で SSO |
| サービス間通信 (tier1 ↔ tier2 ↔ tier3) | mTLS (Istio 自動付与) | Keycloak 非依存。証明書は Istio CA が管理 |
| tier1 公開 API 呼び出し | サービスアカウントトークン (OIDC Client Credentials) | Keycloak でサービスアカウント発行 |
| 外部クライアント → Envoy Gateway | OIDC トークン検証 | Envoy Gateway の ext-authz で Keycloak を参照 |

### 2.2 Phase 別の認証拡張

| Phase | 認証機能 |
|---|---|
| Phase 1 (MVP) | Keycloak ローカル DB、手動ユーザー登録、OIDC SSO |
| Phase 2 | Active Directory / LDAP 連携、グループ同期 |
| Phase 3 | MFA (TOTP)、条件付きアクセスポリシー |
| Phase 4 | 外部 IdP 連携 (Azure AD / Google Workspace) |

## 3. 認可モデル (RBAC)

### 3.1 ロール体系

k1s0 は **Keycloak の Realm Role + Client Role** を組み合わせて RBAC を実現する。

| ロール種別 | スコープ | 例 |
|---|---|---|
| Realm Role | プラットフォーム全体 | `k1s0-admin`, `k1s0-developer`, `k1s0-operator`, `k1s0-viewer` |
| Client Role | 個別コンポーネント | `harbor-admin`, `argocd-deployer`, `backstage-editor` |
| tier1 アプリケーションロール | 業務サービス | `order-manager`, `approval-requester` |

### 3.2 tier 別の権限境界

| tier | 許可される操作 | 禁止される操作 |
|---|---|---|
| infra | k8s クラスタ管理、namespace 作成、CRD 適用 | 業務データへの直接アクセス |
| tier1 | Dapr Component 管理、tier1 namespace 内のリソース操作 | tier2/tier3 namespace のリソース変更 |
| tier2 | 自 namespace 内の業務データ CRUD | 他 tier2 サービスの DB への直接アクセス |
| tier3 | 自 namespace 内のリソース操作、tier1 公開 API 呼び出し | tier2 DB / infra コンポーネントへの直接アクセス |

### 3.3 k8s RBAC との統合

- 各 tier の namespace に `Role` / `RoleBinding` を定義し、ServiceAccount の権限を制限する
- `ClusterRole` は infra チームのみが管理する
- tier2 / tier3 の Pod は自 namespace 外のリソースを参照できない

## 4. ネットワークポリシー

### 4.1 namespace 間通信制御

k8s NetworkPolicy で tier 間の通信を依存ルールどおりに制限する。

| 送信元 | 送信先 | 許可 | 手段 |
|---|---|---|---|
| tier3 | tier2 | 許可 | Istio mTLS + NetworkPolicy |
| tier3 | tier1 | 条件付き許可 (tier2 未経由時のみ) | NetworkPolicy の podSelector |
| tier2 | tier1 | 許可 | Istio mTLS + NetworkPolicy |
| tier1 | infra | 許可 | NetworkPolicy |
| tier2 / tier3 | infra | **拒否** | NetworkPolicy で明示的に deny |
| 外部 | tier3 | Envoy Gateway 経由のみ | NetworkPolicy + Istio Gateway |

### 4.2 Istio AuthorizationPolicy

NetworkPolicy (L3/L4) に加え、Istio AuthorizationPolicy (L7) でサービス単位のアクセス制御を行う。

- 各 tier1 サービスは呼び出し元の ServiceAccount を検証する
- tier2 サービス間の通信は `AuthorizationPolicy` で許可リストを管理する
- `DENY` ポリシーをデフォルトとし、明示的に許可したもののみ通信可能にする

## 5. データ分類と保護

### 5.1 データ分類

| 分類 | 定義 | 例 | 保護要件 |
|---|---|---|---|
| 機密 | 漏洩時に法的・経済的損害が発生 | 個人情報、認証情報、暗号鍵 | 暗号化 (保存時 + 転送時)、アクセスログ必須、PII マスキング |
| 社外秘 | 社内限定の業務情報 | 業務データ、設定値、監査ログ | 転送時暗号化、RBAC による制限 |
| 一般 | 漏洩しても影響が軽微 | サービスカタログ、TechDocs | 転送時暗号化 |

### 5.2 保護の実装

| 保護対象 | 手段 |
|---|---|
| 転送中データ | Istio mTLS (サービス間)、cert-manager 管理の TLS 証明書 (外部通信) |
| 保存データ (PostgreSQL) | PostgreSQL の TDE またはボリューム暗号化 |
| 保存データ (etcd) | k8s の etcd 暗号化設定 (`EncryptionConfiguration`) |
| Secrets | Dapr Secret Store → OpenBao (MPL 2.0、Vault の LF fork)。動的シークレット / 自動ローテーション / 全アクセス監査。詳細は [`../../04_技術選定/03_周辺OSS/08_シークレット管理.md`](../../04_技術選定/03_周辺OSS/08_シークレット管理.md) |
| 個人情報 | tier1 `k1s0.Pii` API によるマスキング (Rust 実装)。暗号化/復号は OpenBao Transit Engine に委譲 |
| 監査ログ | tier1 `k1s0.Audit` API による改ざん防止 (ハッシュチェーン) |

## 6. 脅威モデリング (STRIDE 概観)

本節は STRIDE の概観のみを示す。信頼境界図・主要データフロー別の脅威列挙・残余リスクは [`14_脅威モデル詳細.md`](../03_セキュリティ/02_脅威モデル詳細.md) に分離して詳述する。稟議で「最悪系のシナリオ」を問われた際は、まず同ファイルの信頼境界図と残余リスク節を提示する運用とする。

| 脅威カテゴリ | k1s0 における脅威例 | 対処 |
|---|---|---|
| **S**poofing (なりすまし) | 偽造トークンによる tier1 API 呼び出し | Keycloak OIDC トークン検証 + Istio mTLS |
| **T**ampering (改ざん) | 監査ログの事後改ざん | ハッシュチェーンによる改ざん検知 (tier1 Audit) |
| **R**epudiation (否認) | 操作の否認 | 全操作を監査ログに記録、Keycloak の認証イベントログ |
| **I**nformation Disclosure (情報漏洩) | tier2 DB への不正アクセス | NetworkPolicy + RBAC + データ暗号化 |
| **D**enial of Service (サービス拒否) | API への過負荷攻撃 | Envoy Gateway の Rate Limiting + Istio Circuit Breaker |
| **E**levation of Privilege (権限昇格) | Pod からの k8s API 不正利用 | ServiceAccount の最小権限 + Kyverno ClusterPolicy |

### Phase 別の脅威対策

| Phase | 対策 |
|---|---|
| MVP-1a | mTLS、NetworkPolicy、RBAC、Keycloak OIDC、cert-manager (TLS 証明書自動管理) |
| MVP-1b | Kyverno ClusterPolicy (PSS Restricted 相当 + イメージ制御 + ラベル強制) |
| Phase 2 | Cosign 署名 + Kyverno 未署名拒否、SBOM |
| Phase 3 | Falco (ランタイムセキュリティ)、OpenBao PKI Engine (中間 CA + cert-manager 連携)、Auto-unseal |

## 7. サプライチェーンセキュリティ

| 段階 | 対策 | 導入時期 |
|---|---|---|
| ソースコード | GitHub Branch Protection + PR レビュー必須 | MVP-1a |
| 依存パッケージ | Trivy FS スキャン (GHA step) | MVP-1b |
| コンテナイメージ | Harbor 内蔵 Trivy (Critical 拒否) | MVP-1b |
| TLS 証明書 | cert-manager による自動発行・更新 (期限切れ防止) | MVP-1a |
| Admission ポリシー | Kyverno で Harbor 以外のイメージ pull 拒否 + `:latest` タグ禁止 | MVP-1b |
| イメージ署名 | Cosign + Kyverno 未署名ブロック | Phase 2 |
| SBOM | Trivy SBOM 生成 + Harbor 保管 | Phase 2 |
| 証跡 | Tekton Chains (代替フロー採用時) | Phase 3 |
