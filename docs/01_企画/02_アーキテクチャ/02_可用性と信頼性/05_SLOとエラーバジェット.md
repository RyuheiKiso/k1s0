# SLO とエラーバジェット

## 目的

k1s0 プラットフォームの SLO (Service Level Objective) を定義し、エラーバジェットに基づく運用判断の仕組みを設計する。[`07_非機能要件.md`](../04_非機能とデータ/01_非機能要件.md) で可用性の数値目標を定義しているが、本資料はその目標を**運用可能な制度**に落とし込む。

---

## 1. SLI / SLO / SLA の関係

| 概念 | 定義 | k1s0 での役割 |
|---|---|---|
| **SLI** (Service Level Indicator) | サービスの健全性を測る定量指標 | Prometheus メトリクスとして計測 |
| **SLO** (Service Level Objective) | SLI の目標値 (内部コミットメント) | 開発 / 運用チームの品質基準 |
| **SLA** (Service Level Agreement) | SLO を含む外部契約 | k1s0 は社内基盤のため Phase 3 以降で検討 |

k1s0 は社内プラットフォームであり、MVP 時点で外部向け SLA は不要。SLO を内部基準として定義し、エラーバジェットで「品質 vs 速度」のトレードオフを定量的に管理する。

---

## 2. SLO 定義

### 2.1 tier1 公開 API

| SLO 名 | SLI | 計算式 | 目標値 (Phase 1) | 目標値 (Phase 3) | 目標値 (Phase 5) |
|---|---|---|---|---|---|
| **可用性** | 成功率 | `(成功レスポンス数) / (全レスポンス数)` ※ HTTP 5xx を失敗とみなす | 99% | 99.5% | 99.9% |
| **レイテンシ (高速 API)** | p99 応答時間 | `k1s0.Log` / `k1s0.Telemetry` / `k1s0.Auth.ValidateToken` の p99 | < 100 ms | < 50 ms | < 50 ms |
| **レイテンシ (標準 API)** | p99 応答時間 | `k1s0.State` / `k1s0.PubSub` / `k1s0.Decision` の p99 | < 500 ms | < 200 ms | < 200 ms |
| **レイテンシ (重量 API)** | p99 応答時間 | `k1s0.Workflow.Start` / `k1s0.Service.Invoke` の p99 | < 1000 ms | < 500 ms | < 500 ms |

### 2.2 infra コンポーネント

| SLO 名 | SLI | 計算式 | 目標値 (Phase 1) | 目標値 (Phase 3) |
|---|---|---|---|---|
| **k8s API Server 可用性** | API Server の応答成功率 | `(成功レスポンス数) / (全レスポンス数)` | 99.5% | 99.9% |
| **PostgreSQL 可用性** | プライマリへの接続成功率 | `(成功クエリ数) / (全クエリ数)` | 99.5% | 99.9% |
| **Kafka 可用性** | プロデュース ACK 成功率 | `(ACK 成功数) / (プロデュース試行数)` | 99% | 99.5% |

### 2.3 CI/CD パイプライン

| SLO 名 | SLI | 計算式 | 目標値 |
|---|---|---|---|
| **ビルド成功率** | PR ビルドの成功率 (コードエラー起因を除く) | `(インフラ起因でない成功ビルド数) / (全ビルド数)` | 95% |
| **ビルド速度** | PR → ビルド完了の所要時間 | p95 | < 10 分 |
| **デプロイ速度** | main マージ → Pod 更新完了の所要時間 | p95 | < 20 分 |

## 3. エラーバジェット

### 3.1 エラーバジェットとは

SLO が 99% の場合、30 日間で許容されるダウンタイムは `30 日 x 24 時間 x (1 - 0.99) = 7.2 時間`。この 7.2 時間が**エラーバジェット**であり、新機能デプロイ・実験的変更・メンテナンスに「使える」障害予算となる。

### 3.2 Phase 別のエラーバジェット

| SLO | Phase 1 (99%) | Phase 3 (99.5%) | Phase 5 (99.9%) |
|---|---|---|---|
| 月間エラーバジェット | **7.2 時間** | 3.6 時間 | 43 分 |
| 週間エラーバジェット | 1.68 時間 | 50 分 | 10 分 |

### 3.3 エラーバジェットの消費追跡

Prometheus の recording rule でエラーバジェットの残量をリアルタイムに計算する。

```yaml
# 概念例: 30 日間のエラーバジェット残量
groups:
  - name: slo
    rules:
      - record: k1s0:tier1_api_error_budget_remaining
        expr: |
          1 - (
            (1 - (sum(rate(k1s0_api_call_total{status="success"}[30d]))
                  / sum(rate(k1s0_api_call_total[30d]))))
            / (1 - 0.99)
          )
```

`k1s0:tier1_api_error_budget_remaining` が 0 以下になった時点でエラーバジェットが枯渇。

## 4. バーンレートアラート

### 4.1 バーンレートとは

エラーバジェットの消費速度。バーンレート 1.0 は「このペースなら SLO ウィンドウ (30 日) の最終日にちょうどバジェットが尽きる」速度。バーンレート 14.4 は「このペースなら 2 日でバジェットが尽きる」速度。

### 4.2 アラート定義

Google SRE Workbook のマルチウィンドウ・マルチバーンレート手法を採用する。

| 重要度 | バーンレート | 短期ウィンドウ | 長期ウィンドウ | 意味 |
|---|---|---|---|---|
| **Critical (P1)** | 14.4x | 5 分 | 1 時間 | 2 日でバジェット枯渇。即時対応 |
| **Warning (P2)** | 6x | 30 分 | 6 時間 | 5 日でバジェット枯渇。当日中に対応 |
| **Info (P3)** | 1x | 6 時間 | 3 日 | 30 日でバジェット枯渇。次回メンテナンスで対応 |

短期ウィンドウと長期ウィンドウの両方が閾値を超えた場合にのみ発火する (短期的なスパイクのみでは発火しない)。

### 4.3 Grafana ダッシュボード

「SLO Overview」ダッシュボードを Grafana に配置し、以下を可視化する。

| パネル | 内容 |
|---|---|
| エラーバジェット残量 (%) | 30 日ウィンドウでの残量をゲージ表示。50% 以下で黄色、10% 以下で赤色 |
| バーンレート推移 | 直近 24 時間のバーンレートを時系列グラフで表示 |
| SLI 推移 | 可用性 / レイテンシの SLI を SLO 目標線と重ねて表示 |
| エラーバジェット消費イベント | バジェットを消費したインシデント / デプロイをアノテーションで表示 |

## 5. エラーバジェットポリシー

### 5.1 バジェット残量に応じた運用ルール

| バジェット残量 | 状態 | 許可される操作 | 禁止される操作 |
|---|---|---|---|
| **50% 以上** | 正常 | 通常の開発・デプロイ・実験的変更 | なし |
| **25% - 50%** | 注意 | 通常デプロイは許可。リスクの高い変更 (infra 更新等) は tier1 チームリーダーの承認必須 | 実験的変更 |
| **10% - 25%** | 警告 | バグ修正 / セキュリティパッチのみ。新機能デプロイは凍結 | 新機能デプロイ / infra 変更 / Chaos Engineering |
| **0% 以下** | 枯渇 | **フリーズ**: バグ修正 / セキュリティパッチのみ。全新機能を凍結し、信頼性改善に注力 | 新機能デプロイ / 実験的変更 / infra 変更 / 計画メンテナンス |

### 5.2 フリーズの解除条件

エラーバジェット枯渇によるフリーズは、以下の**全て**が成立した時点で解除する。

1. バジェット残量が 25% 以上に回復した
2. 枯渇の原因となったインシデントの Postmortem が完了した
3. Postmortem で特定されたアクションアイテムの Critical 項目が完了した

### 5.3 例外処理

| 例外 | 扱い |
|---|---|
| **計画メンテナンス** | メンテナンスウィンドウ内の停止はエラーバジェットから除外しない (SLA にメンテナンス除外を設けない方針に準拠)。計画メンテナンスもバジェットの「投資」として扱う |
| **外部依存 (ネットワーク障害等)** | k1s0 の制御外の障害もバジェットから控除する。ただし Postmortem で「制御可能だったか」を評価し、対策が可能ならアクションアイテムに追加 |
| **Chaos Engineering** | Litmus による計画的な障害注入で発生したエラーはバジェットから除外する (staging 環境でのみ実行が前提) |

## 6. SLO レビュープロセス

### 6.1 定期レビュー

| 頻度 | 内容 | 参加者 |
|---|---|---|
| **週次** | バーンレートの確認。バジェット残量の報告。前週のインシデント振り返り | tier1 チーム |
| **月次** | SLI の傾向分析。SLO 目標値の妥当性評価。バジェットポリシーの遵守状況 | tier1 + インフラ + 運用 |
| **Phase 終了時** | SLO 目標値の引き上げ判断。次 Phase の SLO 策定 | 全チーム |

### 6.2 SLO 変更の判断基準

| 条件 | 判断 |
|---|---|
| 3 か月連続でバジェット残量が 80% 以上 | SLO を引き上げてよい (次 Phase の目標値を適用) |
| 3 か月連続でバジェットが枯渇 | SLO が現実と乖離している。目標値を一時的に引き下げ、信頼性改善に注力 |
| tier2 / tier3 チームから可用性に対する苦情が月 3 件以上 | SLO の見直しまたは縮退設計の改善が必要 |

## 7. Prometheus / Grafana の実装

### 7.1 必要な Recording Rule

| Rule 名 | 目的 |
|---|---|
| `k1s0:tier1_api_availability_30d` | 30 日間の可用性 SLI |
| `k1s0:tier1_api_latency_p99_5m` | 5 分ウィンドウの p99 レイテンシ |
| `k1s0:tier1_api_error_budget_remaining` | エラーバジェット残量 (0-1) |
| `k1s0:tier1_api_burn_rate_1h` | 1 時間ウィンドウのバーンレート |
| `k1s0:tier1_api_burn_rate_6h` | 6 時間ウィンドウのバーンレート |

### 7.2 必要な Alerting Rule

| Alert 名 | 条件 | 重要度 |
|---|---|---|
| `K1s0SLOBurnRateCritical` | バーンレート 14.4x が 5 分 + 1 時間の両方で超過 | Critical |
| `K1s0SLOBurnRateWarning` | バーンレート 6x が 30 分 + 6 時間の両方で超過 | Warning |
| `K1s0SLOBudgetExhausted` | エラーバジェット残量が 0% 以下 | Critical |
| `K1s0SLOBudgetLow` | エラーバジェット残量が 25% 以下 | Warning |

## 8. 段階導入計画

| フェーズ | 施策 |
|---|---|
| **Phase 1 (MVP)** | tier1 公開 API のメトリクス (`k1s0_api_call_total` / `k1s0_api_call_duration_seconds`) を公開。SLO 99% を内部基準として設定。Grafana に SLI 推移パネルを追加 |
| **Phase 2** | Recording Rule + Alerting Rule を Prometheus に追加。バーンレートアラートを Grafana に設定。エラーバジェットポリシーを文書化し運用開始。週次 SLO レビューを開始 |
| **Phase 3** | SLO を 99.5% に引き上げ。月次 SLO レビューを開始。tier2 チームへの SLO 報告を開始。SLA の策定を検討 |
| **Phase 5** | SLO を 99.9% に引き上げ。SLA を tier2 チームとの間で締結 (内部合意)。エラーバジェットの自動レポート生成 |

---

## 9. MVP での最小実装

Phase 1 (MVP) では以下の最小セットで SLO 運用を開始する。

| 項目 | 実装内容 |
|---|---|
| SLI 計測 | `k1s0_api_call_total{status}` と `k1s0_api_call_duration_seconds` を tier1 Go ファサードが公開 |
| SLO 目標 | tier1 API 可用性 99% (月間ダウンタイム 7.2 時間以内) |
| 可視化 | Grafana に「SLI 推移」パネルを 1 つ追加 (可用性の時系列) |
| アラート | `k1s0_api_unavailable{api="..."} == 1` が 5 分継続で Critical (既存のグレースフルデグラデーションアラートを流用) |
| バジェット追跡 | 手動 (月次の SLI 集計から計算)。Phase 2 で自動化 |

バーンレートアラート・エラーバジェットポリシー・フリーズ制度は Phase 2 以降で導入する。MVP では「SLI を計測し、SLO を意識する文化を作る」ことに注力する。
