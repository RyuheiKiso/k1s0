# レイヤ構成と責務

## 目的

k1s0 の 4 実行レイヤと横断運用レイヤの責務・管轄範囲・実装言語を定義する。全体俯瞰は [`00_概念アーキテクチャ.md`](./00_概念アーキテクチャ.md) を参照。

---

## 1. infra 層

### 責務

- 物理的な実行基盤。k8s クラスタと周辺コンポーネント全般を担う
- サービスメッシュ・観測基盤・メッセージング・レジストリなどの実行基盤を提供する

### 主要コンポーネント

| カテゴリ | 採用 OSS | 備考 |
|---|---|---|
| コンテナオーケストレーション | Kubernetes (`kubeadm` メイン / `k3s` サブプラ���) | セルフマネージド前提 |
| サービスメッシュ | Istio | mTLS / トラフィック制御 |
| API Gateway | Envoy Gateway | k8s Gateway API 準拠 |
| メッセージング | Apache Kafka (Strimzi Operator) | イベントソーシング前提 |
| メトリクス | Prometheus | CNCF Graduated |
| ログ | Loki | 軽量・低コスト |
| 分散トレース | OpenTelemetry + Grafana Tempo | ベンダー中立。LGTMP スタック統一 |
| Continuous Profiling | Grafana Pyroscope | Tempo 連携 (Traces-to-Profiles)。CPU / メモリ消費の根本原因をコードレベルで特定 |
| 可視化 | Grafana | 統合ダッシュボード |
| RDBMS | CloudNativePG + PostgreSQL | 共有 DB クラスタ。Keycloak / Backstage / 業務サービスが利用 |
| 接続プーリング | PgBouncer (CloudNativePG Pooler) | PostgreSQL 接続数の制御。CloudNativePG Pooler CRD で宣言的管理 |
| ポリシーエンジン | Kyverno | Admission Controller。Pod セキュリティ / イメージ制御 / ラベル強制 |
| 証明書管理 | cert-manager | TLS 証明書の自動発行・自動更新 |
| オブジェクトストレージ | MinIO | S3 互換。Harbor バックエンド / バックアップ保存先 |
| シークレット管理 | OpenBao | Vault の LF fork (MPL 2.0)。動的シークレット / 自動ローテーション / Transit 暗号化 / PKI |
| 分散ストレージ | Longhorn | CNCF Incubating。k8s 環境でのブロックストレージ。レプリケーション / スナップショット |
| ベアメタル LB | MetalLB | CNCF Sandbox。オンプレミスで Service type=LoadBalancer を実現。L2 / BGP 対応 |
| テレメトリパイプライン | OpenTelemetry Collector | OTel プロジェクト公式。メトリクス / トレース / ログの統一パイプライン |
| イベント駆動オートスケーラー | KEDA | CNCF Graduated。Kafka Consumer Lag / Temporal Task Queue / Prometheus メトリクスに基づく Pod オートスケーリング |

kube-vip は static Pod として各 Control Plane ノードに配置され、API Server の高可用性 VIP を提供する。namespace には属さない。

選定根拠は [`../04_技術選定/01_実行基盤中核OSS.md`](../04_技術選定/01_実行基盤中核OSS.md) / [`../04_技術選定/09_ネットワークとテレメトリ基盤.md`](../04_技術選定/09_ネットワークとテレメトリ基盤.md) / [`../04_技術選定/14_追加採用OSS_6.md`](../04_技術選定/14_追加採用OSS_6.md) / [`../04_技術選定/16_追加採用OSS_8.md`](../04_技術選定/16_追加採用OSS_8.md) を参照。

### 管轄

- 担当チーム: インフラチーム
- ディレクトリ: `infra/`

## 2. tier1 層 (共通基盤サービス)

### 責務

システム全体で共通利用する機能をサービス / ライブラリとして提供する。tier1 は以下の 3 要素から成る。

| 要素 | 役割 |
|---|---|
| **tier1 公開 API (多言語クライアントライブラリ)** | tier2 / tier3 が利用する唯一の接点。C# / Go / TS 向けに提供 |
| **tier1 内部実装 (不可視)** | Dapr / Go ファサード / Rust 自作サービスの並立。tier2 / tier3 からは隠蔽される |
| **雛形生成 CLI** | サービス雛形を生成し、Dapr annotation や tier1 ライブラリ依存を自動で組み込む |

### 提供する機能カテゴリ

| カテゴリ | 対応する tier1 公開 API |
|---|---|
| 構造化ログ | `k1s0.Log` |
| 分散トレース / メトリクス | `k1s0.Telemetry` |
| State 管理 | `k1s0.State` |
| Pub / Sub | `k1s0.PubSub` |
| Secrets | `k1s0.Secrets` |
| Workflow | `k1s0.Workflow` |
| 認証 | `k1s0.Auth` |
| 監査ログ | `k1s0.Audit` |
| 決定評価 (BRE) | `k1s0.Decision` |
| 設定同期 (端末横断) | `k1s0.Settings` |
| Feature Flag | `k1s0.Feature` |

これらの API は tier1 内部で Dapr Go SDK または Rust 自作サービスに振り分けられる。tier2 / tier3 開発者は振り分け先を知る必要はない。詳細な隠蔽方針は [`../03_tier1設計/01_Dapr隠蔽方針.md`](../03_tier1設計/01_Dapr隠蔽方針.md) を参照。

### 重要原則

- **tier2 / tier3 は Dapr を直接意識しない**。Dapr SDK の import、`dapr.io/*` annotation、Component YAML、Dapr のエラーメッセージ、いずれも tier2 / tier3 のコードに現れてはならない。これらは tier1 公開 API / 雛形生成 CLI / tier1 チームが吸収する。

### 内部言語の使い分け

tier1 内部サービスは用途で言語を選ぶ。

| 用途 | 言語 | 理由 |
|---|---|---|
| Dapr building block のファサード (state / pub-sub / secrets / bindings) | Go | Dapr Go SDK が stable |
| ワークフローエンジン統合 (Saga / 長期実行) | Go | Temporal Go SDK が Production Ready。Dapr Workflow は軽量短期処理に限定 |
| 認証プロキシ | Go | oauth2-proxy 自体が Go 製 |
| OTel 集約・Loki クライアント | Go | エコシステムが Go 中心 |
| 決定エンジン (ZEN Engine 統合) | Rust | ZEN Engine が Rust ネイティブ、in-process 埋め込み可 |
| JTC 固有機能 (crypto / 監査改ざん防止 / 個人情報マスキング) | Rust | 安全性とパフォーマンスが効く |
| 雛形生成 CLI | Rust | 単一バイナリ配布・起動高速 |
| Dapr Component バリデータ | Rust | 雛形生成 CLI と同梱可能 |

詳細は [`../03_tier1設計/02_内部言語ハイブリッド.md`](../03_tier1設計/02_内部言語ハイブリッド.md) を参照。

### 管轄

- 担当チーム: システム基盤チーム
- ディレクトリ: `src/tier1/`
- 実装言語: 公開 API = 対象言語 (C# / Go / TS)、内部実装 = Go + Rust

## 3. tier2 層 (ドメインサービス)

### 責務

- 業務ドメインロジックを実装するマイクロサービス群
- tier1 公開 API のみを利用し、infra コンポーネントに直接触れない

### 管轄

- 担当チーム: ドメイン開発チーム
- ディレクトリ: `src/tier2/`
- 実装言語: C# / Go 自由 (既存資産の言語を尊重)

### 制約

- Dapr SDK / Kafka クライアント / state store クライアントを直接 import / 呼び出してはならない
- ログ・トレース・エラーハンドリングは tier1 が提供する API でのみ行う
- 設定値はコード内にハードコードせず、tier1 の Configuration API 経由で取得する

---

## 4. tier3 層 (個別アプリケーション)

### 責務

- エンドユーザー向けの UI / API を持つアプリケーション
- アプリ配信ポータル (k1s0 自製) も tier3 として実装される

### 配信形態

| 形態 | 用途 | 備考 |
|---|---|---|
| Web (PWA) | 第一選択 | インストール不要。UX を統一できる |
| ネイティブ (MAUI / WinUI) | ハードウェア連携 / オフライン業務 | 必要最小限に限定 |
| レガシー .NET Framework exe のラップ | 移行期のみ | ClickOnce / MSIX でラップ配信 |

詳細は [`../05_CICDと配信/02_アプリ配信ポータル.md`](../05_CICDと配信/02_アプリ配信ポータル.md) を参照。

### 管轄

- 担当チーム: 個別アプリケーション開発チーム
- ディレクトリ: `src/tier3/`
- 実装言語: サーバー = C# / Go、クライアント = MAUI / React (TSX)

---

## 5. operation 層 (運用 / 横断)

### 責務

実行レイヤではなく、全レイヤに横断的に作用する運用プロセス層。運用手順・監視設定・オンコール対応・リリース管理を担う。

### 主要コンポーネント

| コンポーネント | 役割 |
|---|---|
| Backstage | 開発者ポータル (Software Catalog / TechDocs / Software Templates) |
| Argo CD | GitOps による継続的デプロイ |
| Keycloak | OIDC / 認証基盤 |
| GitHub Actions self-hosted runner | CI エンジン |
| Harbor | コンテナレジストリ + 脆弱性スキャン |
| Litmus | Chaos Engineering (縮退動作の自動検証) |
| Renovate | 依存パッケージの自動更新 PR 生成 (GHA runner 上で実行) |
| Argo Events | イベント駆動の自動化 (GitHub Webhook / Harbor Webhook / Kafka) |

詳細は [`../05_CICDと配信/`](../05_CICDと配信/) と [`../04_技術選定/02_周辺OSS.md`](../04_技術選定/02_周辺OSS.md) を参照。

### 管轄

- 担当チーム: 運用チーム + システム基盤チーム (Backstage 構築)
- ディレクトリ: `operation/`
