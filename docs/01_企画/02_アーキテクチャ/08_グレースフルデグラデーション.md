# グレースフルデグラデーション

## 目的

tier1 公開 API の障害時に tier2 / tier3 サービスが受ける影響と、クライアント側の縮退動作を定義する。tier1 は全サービスの共通基盤であり、障害時の影響範囲が広い。「tier1 が落ちたら全てが止まる」状態を避けるため、API ごとの縮退方針をクライアントライブラリに組み込む。

---

## 1. 設計原則

### 1.1 基本方針

> **「観測系は落としても業務を止めるな。認証系はキャッシュで耐えろ。状態系は呼び出し元に判断させろ」**

tier1 API はその性質によって障害時の振る舞いが異なる。全 API を一律に「リトライして待つ」のは誤りであり、API の特性に応じた縮退戦略を取る。

### 1.2 縮退分類

tier1 公開 API を障害時の影響度に基づき 3 分類する。

| 分類 | 定義 | 障害時の動作原則 | 該当 API |
|---|---|---|---|
| **観測系 (Observability)** | データの欠損は許容できるが業務停止は許容できない | **fire-and-forget**: 送信失敗を握りつぶし、業務処理を継続する | `Log`, `Telemetry`, `Audit` |
| **参照系 (Cached Reference)** | 最新値でなくても古い値で業務を継続できる | **cache-fallback**: ローカルキャッシュの値で継続する | `Auth`, `Secrets`, `Decision`, `Settings`, `Feature` |
| **状態系 (Stateful)** | 障害時にデータの整合性が崩れる可能性がある | **fail-fast + caller decides**: 即座にエラーを返し、呼び出し元に判断を委ねる | `State`, `PubSub`, `Workflow`, `Service.Invoke` |

### 1.3 禁止パターン

| パターン | 理由 |
|---|---|
| 無制限リトライ | tier1 復旧まで呼び出し元がブロックされ、スレッド枯渇・カスケード障害を引き起こす |
| 全 API 一律タイムアウト | 観測系に 5 秒待つのは無駄、状態系に 100ms で打ち切るのは早すぎる |
| サイレント失敗 (状態系) | State や PubSub の失敗を握りつぶすとデータ不整合が発生する |
| 独自リトライロジック (tier2/tier3 側) | リトライポリシーは tier1 クライアントライブラリが制御する。呼び出し元が二重にリトライすると雪崩を起こす |

## 2. API 別の縮退仕様

### 2.1 観測系 API

#### k1s0.Log.*

| 項目 | 仕様 |
|---|---|
| 内部依存 | tier1 ライブラリ内 → Loki (バックエンド) |
| 障害時の影響 | ログが欠損する。業務処理への影響はない |
| 縮退動作 | Loki 送信失敗時、**メモリバッファに一時保持** (上限 1,000 件)。バッファ溢れ時は古い順に破棄。送信失敗を呼び出し元に返さない |
| 復旧後の動作 | バッファ内のログを非同期でフラッシュする |
| クライアントライブラリの責務 | `Log.Info()` は常に即座に返却する。例外を throw しない |
| 監視への影響 | ログベースのアラートが発火しなくなる可能性がある。[`06_プラットフォーム自己監視.md`](./06_プラットフォーム自己監視.md) のメタ監視で Loki 自体の死活を検知する |

#### k1s0.Telemetry.*

| 項目 | 仕様 |
|---|---|
| 内部依存 | tier1 ライブラリ内 OTel SDK → OTel Collector → Grafana Tempo / Prometheus |
| 障害時の影響 | トレース / メトリクスが欠損する。業務処理への影響はない |
| 縮退動作 | OTel Collector への送信失敗時、**バッチを破棄**する。OTel SDK の BatchSpanProcessor が標準でこの動作を行う |
| 復旧後の動作 | 欠損期間のデータは復旧しない (設計上許容) |
| クライアントライブラリの責務 | テレメトリ送信の失敗を呼び出し元に伝搬しない |

#### k1s0.Audit.*

| 項目 | 仕様 |
|---|---|
| 内部依存 | tier1 Rust サービス → PostgreSQL (永続化) / Loki (保管) |
| 障害時の影響 | 監査ログが記録されない。法的要件に関わるため、**観測系の中で最も重要** |
| 縮退動作 | Rust サービスへの送信失敗時、**ローカルファイルに WAL 形式で書き出す** (Pod ローカル PV)。バッファとファイルの二段構成で欠損を最小化する |
| 復旧後の動作 | ローカル WAL を読み出し、Rust サービスに再送する。再送完了後にローカルファイルを削除する |
| クライアントライブラリの責務 | `Audit.Record()` は常に成功を返す。ただし、**ローカル WAL への書き込みすら失敗した場合は stderr にフォールバック出力し、メトリクス `k1s0_audit_fallback_total` をインクリメント**する |
| 運用上の注意 | `k1s0_audit_fallback_total > 0` はCritical アラートとする。監査ログの欠損は法的リスクに直結するため、Audit サービスの復旧を最優先とする |

### 2.2 参照系 API

#### k1s0.Auth.ValidateToken

| 項目 | 仕様 |
|---|---|
| 内部依存 | Go サービス → Keycloak (JWKS エンドポイント) |
| 障害時の影響 | 新規トークン検証ができなくなる |
| 縮退動作 | **JWKS キャッシュで継続**。クライアントライブラリは JWKS を TTL 付きでキャッシュする (TTL: 1 時間、最大延命: 4 時間)。Keycloak 障害中もキャッシュ内の公開鍵で署名検証を行う |
| キャッシュ失効後 | キャッシュが最大延命を超えた場合、**新規リクエストを拒否** (HTTP 503) する。既に検証済みのセッションは影響しない |
| 復旧後の動作 | JWKS を再取得し、キャッシュを更新する |
| 設計判断 | fail-open (キャッシュ期限内は通す) を採用。fail-close (検証不能なら全拒否) は業務停止の影響が大きすぎるため不採用。ただし、キャッシュの最大延命 (4 時間) を超えた場合は安全側に倒して拒否する |

#### k1s0.Secrets.Get

| 項目 | 仕様 |
|---|---|
| 内部依存 | Go ファサード → Dapr → OpenBao (MVP-1b 以降) |
| 障害時の影響 | Secret の取得ができなくなる |
| 縮退動作 | **起動時に取得した値をプロセス内メモリにキャッシュ**。Dapr / OpenBao 障害中はキャッシュ値を返す |
| キャッシュ更新 | 正常時は設定間隔 (デフォルト 5 分) で再取得する |
| OpenBao HA | Raft 構成 (3 Pod) で 1 Pod 障害は自動復旧。全 Pod 障害時のみキャッシュフォールバック |
| 制約 | OpenBao Database Engine の動的認証情報はリース期間 (TTL: 1 時間) 内であればキャッシュ値で継続可能。リース期限を超えた場合は DB 接続が切断される可能性があるため、OpenBao の復旧を優先する |

#### k1s0.Decision.Evaluate

| 項目 | 仕様 |
|---|---|
| 内部依存 | tier1 Rust サービス → ZEN Engine (in-process) |
| 障害時の影響 | ビジネスルール評価ができなくなる |
| 縮退動作 | ZEN Engine はin-process のため、Rust サービス自体が落ちない限り障害は発生しにくい。**Rust サービスへの通信障害時は、最後に評価成功したルールバージョンのキャッシュ結果を返す** (同一入力パラメータの場合) |
| キャッシュミス時 | 同一入力のキャッシュがない場合は呼び出し元にエラーを返す。**デフォルト値での自動フォールバックは行わない** (ビジネスルールの誤適用を防ぐため) |
| 復旧後の動作 | 最新ルールファイルを再読み込みする |

#### k1s0.Settings.*

| 項目 | 仕様 |
|---|---|
| 内部依存 | Go ファサード → Dapr State / Configuration → Valkey |
| 障害時の影響 | 設定値の取得 / 更新ができなくなる |
| 縮退動作 | **ローカルキャッシュで継続**。最後に取得成功した設定値を返す |
| 設定更新 | 障害中の設定変更は反映されない。復旧後に最新値を再取得する |
| クライアントライブラリの責務 | `Settings.Get()` はキャッシュ値を返す際にログレベル WARN で記録する (キャッシュ値であることを運用者が認識できるようにする) |

#### k1s0.Feature.*

| 項目 | 仕様 |
|---|---|
| 内部依存 | Go ファサード → flagd gRPC |
| 障害時の影響 | Feature Flag の評価ができなくなる。段階的ロールアウトの制御が効かなくなる |
| 縮退動作 | **ローカルキャッシュで継続**。最後に評価成功したフラグ値を返す |
| キャッシュミス時 | 一度も評価成功していないフラグについては、フラグ定義のデフォルト値を返す。デフォルト値も存在しない場合は `false` (無効) を返す |
| 設計判断 | flagd 障害時に全フラグを `true` (有効) にするのは危険。未検証の新機能が全ユーザーに露出する。安全側に倒して `false` (無効 = 既存動作の維持) をデフォルトとする |
| 復旧後の動作 | flagd に再接続し、全フラグの最新状態を取得してキャッシュを更新する |

### 2.3 状態系 API

#### k1s0.State.*

| 項目 | 仕様 |
|---|---|
| 内部依存 | Go ファサード → Dapr → Valkey |
| 障害時の影響 | 状態の読み書きができなくなる。業務トランザクションが失敗する可能性がある |
| 縮退動作 | **即座にエラーを返す**。リトライはクライアントライブラリ内で最大 3 回 (exponential backoff: 100ms / 200ms / 400ms)。全リトライ失敗後、`K1s0StateUnavailableException` を throw する |
| 呼び出し元の責務 | 例外をキャッチし、業務要件に応じて以下のいずれかを選択する: (A) ユーザーにリトライを促す (B) 代替処理に分岐する (C) 処理を中断しエラーを返す |
| 設計判断 | State のサイレント失敗はデータ不整合の原因となるため、fail-fast を強制する。呼び出し元が意図的に例外を握りつぶすことは許可するが、その場合 tier1 チームのレビューを必須とする |

#### k1s0.PubSub.*

| 項目 | 仕様 |
|---|---|
| 内部依存 | Go ファサード → Dapr → Kafka |
| 障害時の影響 | イベントの発行ができなくなる。非同期処理が滞留する |
| 縮退動作 (Publish) | リトライ (最大 3 回、exponential backoff)。全失敗後、`K1s0PubSubUnavailableException` を throw する |
| 縮退動作 (Subscribe) | Kafka Consumer は Dapr sidecar が管理する。sidecar 再起動後に Consumer Group のオフセットから再開する (Kafka の標準動作)。メッセージの欠損はないが、**重複配信の可能性がある**。tier2 / tier3 の Subscribe ハンドラは**冪等に実装**する必要がある |
| 呼び出し元の責務 | Publish 失敗時: 業務要件に応じて (A) 同期処理にフォールバック (B) ローカルキューに退避して後でリトライ (C) エラーを返す |
| 冪等性の強制 | 雛形生成 CLI が Subscribe ハンドラに冪等性チェックのボイラープレートを含める。リファレンス実装でパターンを示す |

#### k1s0.Workflow.*

| 項目 | 仕様 |
|---|---|
| 内部依存 | Go ファサード → Temporal Go SDK (Saga / 長期実行) / Dapr Workflow (軽量短期処理) |
| 障害時の影響 | Saga / 長期トランザクションの開始・進行ができなくなる |
| 縮退動作 (Start) | リトライ (最大 2 回)。失敗後、`K1s0WorkflowUnavailableException` を throw する |
| 縮退動作 (実行中 Workflow) | Temporal Server は状態を PostgreSQL に永続化しているため、Temporal 復旧後に中断箇所から再開する (Temporal のリプレイ機構)。Dapr Workflow 側も同様に状態永続化により復旧後に再開する。**手動介入は不要** |
| 呼び出し元の責務 | Start 失敗時は同期処理で代替するか、エラーを返す。実行中 Workflow の中断は Temporal / Dapr の復旧を待つ |

#### k1s0.Service.Invoke

| 項目 | 仕様 |
|---|---|
| 内部依存 | Go ファサード → Dapr → Istio (mTLS) |
| 障害時の影響 | サービス間同期呼び出しが失敗する |
| 縮退動作 | Istio の retry + circuit breaker を第一防御とする。Istio レベルで回復しない場合、クライアントライブラリが `K1s0ServiceUnavailableException` を throw する |
| サーキットブレーカー | Istio DestinationRule で設定。連続 5 回失敗で 30 秒間 open (呼び出しを遮断)。open 中の呼び出しは即座に例外を返す |
| 呼び出し元の責務 | 宛先サービスの障害であり、tier1 自体の障害ではないケースが多い。呼び出し元はタイムアウトとフォールバックを業務要件に基づいて設計する |

## 3. クライアントライブラリの共通実装

### 3.1 ヘルスステート

クライアントライブラリは tier1 各 API の接続状態を内部的に追跡する。

| ステート | 定義 | 動作 |
|---|---|---|
| **Healthy** | 直近の呼び出しが成功 | 通常動作 |
| **Degraded** | 直近の呼び出しが失敗したが、キャッシュ / フォールバックで継続中 | WARN ログを出力。メトリクス `k1s0_api_degraded{api="..."}` = 1 |
| **Unavailable** | キャッシュも枯渇し、機能提供不能 | ERROR ログを出力。メトリクス `k1s0_api_unavailable{api="..."}` = 1 |

### 3.2 メトリクス

クライアントライブラリが自動で公開するメトリクス:

| メトリクス名 | 型 | 説明 |
|---|---|---|
| `k1s0_api_call_total` | Counter | API 呼び出し回数 (ラベル: `api`, `status`) |
| `k1s0_api_call_duration_seconds` | Histogram | API 呼び出しレイテンシ |
| `k1s0_api_degraded` | Gauge | 縮退状態フラグ (ラベル: `api`) |
| `k1s0_api_unavailable` | Gauge | 利用不能状態フラグ (ラベル: `api`) |
| `k1s0_api_cache_hit_total` | Counter | キャッシュヒット回数 (参照系のみ) |
| `k1s0_api_fallback_total` | Counter | フォールバック発動回数 |
| `k1s0_audit_fallback_total` | Counter | 監査ログのフォールバック発動回数 (Critical アラート対象) |

### 3.3 Grafana ダッシュボード

上記メトリクスを集約した「tier1 API Health」ダッシュボードを Grafana に配置する。全 API の Healthy / Degraded / Unavailable 状態を一覧表示し、障害の波及範囲を即座に把握できるようにする。

## 4. カスケード障害の防止

### 4.1 障害の連鎖パターン

tier1 内部のコンポーネント障害がどのように波及するかを整理する。

| 障害元 | 直接影響を受ける tier1 API | 間接影響 |
|---|---|---|
| **Valkey 障害** | State, Settings | PubSub (Dapr State の内部利用がある場合) |
| **Kafka 障害** | PubSub | Audit (Kafka 経由のログ保管を使う場合) |
| **Keycloak 障害** | Auth | 全 API (認証が通らなければ全 API が呼べない) |
| **PostgreSQL 障害** | Audit, Auth (セッション管理) | Keycloak, Backstage, Harbor (全て PostgreSQL 依存) |
| **OpenBao 障害** | Secrets | Dapr Secret Store キャッシュで継続。動的認証情報の新規発行・リース更新が停止 |
| **Dapr Control Plane 障害** | 新規 Pod の Dapr sidecar 注入が失敗 | 既存 Pod は影響なし (sidecar は Pod ローカル) |
| **Temporal Server 障害** | Workflow (Saga / 長期実行)。実行中のワークフローは一時停止するが、Temporal 復旧後にリプレイ機構で中断箇所から自動再開 | Dapr Workflow (軽量短期処理) は影響なし |
| **Dapr Sidecar 障害 (個別 Pod)** | 該当 Pod の State, PubSub, Secrets, Workflow (軽量短期処理), Service.Invoke | 他の Pod は影響なし |
| **tier1 Go サービス障害** | State, PubSub, Secrets, Workflow, Service.Invoke, Auth, Settings | k8s ReplicaSet で自動復旧 |
| **tier1 Rust サービス障害** | Audit, Decision | k8s ReplicaSet で自動復旧 |
| **flagd 障害** | Feature | キャッシュで継続。キャッシュミス時はデフォルト値 (false) |
| **KEDA Operator 障害** | 新規のスケールアウト/インが停止。既存 Pod は影響なし (現在のレプリカ数で固定) | Kafka Consumer Lag / Temporal Task Queue の滞留が自動解消されなくなる |
| **Istio Control Plane 障害** | 新規の mTLS 証明書発行が停止 | 既存証明書の有効期限内は通信継続 |

### 4.2 最も危険な障害: PostgreSQL + Keycloak

PostgreSQL の全面障害は最大の影響範囲を持つ。Keycloak が DB を失うと認証基盤が停止し、JWKS キャッシュの延命 (最大 4 時間) を超えると全サービスが認証不能になる。

対処:
- CloudNativePG の自動フェイルオーバー (RTO: 30 秒) が第一防御
- PostgreSQL 障害は [`05_障害復旧とバックアップ.md`](./05_障害復旧とバックアップ.md) で最優先復旧対象とする
- Auth の JWKS キャッシュ延命 (4 時間) はフェイルオーバー完了までの猶予として十分な時間を確保している

### 4.3 バルクヘッド (障害隔離)

tier1 内部の Go サービスと Rust サービスを別 Deployment として分離し、一方の障害が他方に波及しないようにする。

| Deployment | 含まれる機能 | レプリカ数 (MVP) |
|---|---|---|
| `tier1-go-facade` | State, PubSub, Secrets, Workflow, Service.Invoke, Settings | 2 |
| `tier1-go-auth` | Auth (認証は独立性が高いため分離) | 2 |
| `tier1-rust-core` | Audit, Decision | 2 |

Auth を Go ファサードから分離する理由: 認証はプラットフォーム全体のゲートであり、State/PubSub の負荷や障害から隔離する必要がある。

### 4.4 レート制限 (過負荷防止)

tier2 / tier3 の暴走 (無限ループ等) が tier1 を過負荷にするパターンは、バルクヘッド (障害隔離) だけでは防げない。tier1 Go ファサードにサービス単位のレート制限を設け、1 つの tier2 の暴走が他サービスに波及しない構造を作る。

詳細は [`10_レート制限.md`](./10_レート制限.md) を参照。

## 5. 障害時の運用対応

### 5.1 アラートと縮退状態の対応表

| メトリクス条件 | 重要度 | 対応 |
|---|---|---|
| `k1s0_api_unavailable{api="Auth"} == 1` が 5 分継続 | **Critical** | Keycloak / PostgreSQL の状態を確認。フェイルオーバーが動いているか検証 |
| `k1s0_api_unavailable{api="State"} == 1` が 5 分継続 | **Critical** | Dapr sidecar / Valkey の状態を確認 |
| `k1s0_api_degraded{api="Auth"} == 1` | **Warning** | JWKS キャッシュで延命中。4 時間以内に Keycloak を復旧する必要あり |
| `k1s0_api_degraded{api="Secrets"} == 1` | **Warning** | キャッシュ値で継続中。Secret ローテーション中でないか確認 |
| `k1s0_audit_fallback_total > 0` | **Critical** | 監査ログが正規経路で記録されていない。Rust サービスと PostgreSQL を即座に確認 |
| `k1s0_api_degraded{api="Decision"} == 1` | **Warning** | ルール評価がキャッシュ結果を返している。Rust サービスの状態を確認 |

### 5.2 Runbook への統合

各アラートに対応する Runbook を TechDocs に整備する。Runbook は以下の構成とする:

1. **発生条件**: どのメトリクスがどの閾値を超えたか
2. **影響範囲**: どの tier2 / tier3 サービスが影響を受けるか
3. **確認手順**: kubectl / Grafana での確認コマンド
4. **復旧手順**: 段階的な対応手順
5. **エスカレーション**: 復旧しない場合の連絡先と判断基準

## 6. Phase 別の導入計画

| フェーズ | 施策 |
|---|---|
| **Phase 1 (MVP)** | 観測系 API の fire-and-forget 実装。Auth の JWKS キャッシュ。状態系 API の fail-fast + 基本リトライ。メトリクス `k1s0_api_call_total` / `k1s0_api_unavailable` の公開 |
| **Phase 2** | Audit のローカル WAL フォールバック。Decision のキャッシュ。Feature のキャッシュ + デフォルト値フォールバック。Grafana「tier1 API Health」ダッシュボード。Runbook 初版。**Litmus による縮退動作の自動検証開始** (PostgreSQL / Valkey / Dapr Sidecar の 3 パターン、週次 CronChaosEngine) |
| **Phase 3** | 全参照系 API のキャッシュ戦略実装。バルクヘッド (Deployment 分離)。**Litmus のカスケード障害テスト拡張** (ネットワーク障害注入、本番 namespace 対象) |

## 7. テスト方針

縮退動作は通常のテストでは検証しにくい。以下の方法で検証する。

| テスト種別 | 内容 | 実施時期 |
|---|---|---|
| **ユニットテスト** | クライアントライブラリの各縮退パス (キャッシュヒット / フォールバック / 例外 throw) を mock で検証 | 全 Phase |
| **統合テスト** | Dapr sidecar を意図的に停止し、クライアントライブラリが正しく縮退するか検証 | Phase 2 以降 |
| **Chaos Engineering (Litmus)** | ChaosExperiment CRD で PostgreSQL / Valkey / Dapr Sidecar の障害を自動注入し、縮退動作が設計通りに機能するか検証。CronChaosEngine で週次自動実行 | Phase 2 以降 |
| **障害注入テスト** | Istio の Fault Injection (delay / abort) を使い、tier1 API の応答遅延 / エラーを注入 | Phase 3 以降 |
| **カスケードテスト** | Litmus で PostgreSQL / Valkey / Kafka を順に停止し、影響範囲が想定通りか確認 | Phase 3 以降 |

### Chaos Engineering との関係

本章で定義した縮退動作は「設計上の期待」であり、実装が設計通りに動くかは Chaos Engineering で継続的に検証する。Litmus の採用根拠は [`../04_技術選定/02_周辺OSS.md`](../04_技術選定/02_周辺OSS.md) を参照。上述のカスケード障害パターン (セクション 4) を Litmus の ChaosExperiment として定義し、週次で自動実行する。
