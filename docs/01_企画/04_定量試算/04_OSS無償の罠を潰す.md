# OSS 無償の罠を潰す

## 目的

稟議の場で必ず出る「OSS は無償に見えて、運用人件費・学習コスト・CVE 対応・バージョン追従で結局は有償製品より高くつくのではないか」という懸念を、定量的に反証する章。[`01_TCO5年試算.md`](./01_TCO5年試算.md) は Build vs Buy の総額比較を扱うが、本書は **「k1s0 の運用コスト」だけを切り出し、商用 IDP に同額以上の運用コストが残ることを数字で示す** ことに特化する。

結論を先に示すと、OSS 採用による自社運用コストの増分は中規模 (3,000 名) で年 **約 2,140 万円**（運用 FTE 1.55 + 学習オンボーディング 0.4 FTE 相当）。これに対し、商用 IDP (OpenShift) 採用時の自社運用コストは年 **約 1,780 万円** で、差分は 360 万円。同じ規模での商用 IDP ライセンス費は年 2,400 万円であるため、**OSS 無償による「隠れコスト」全体を足しても商用ライセンス費を下回る**。「OSS は有償より高くつく」という直感は中規模以上では事実と逆であることを、本書で数字を挙げて示す。

---

## 1. 「OSS 無償の罠」の正体と、本書が反証する 4 つの仮説

稟議の場で出る「OSS 無償の罠」は、以下 4 つの仮説の混合である。本書ではそれぞれを分解し、定量的に反証する。

第一の仮説は「OSS を運用する人件費が商用より高い」というもの。商用製品ではベンダーが運用の大半を肩代わりし、自社人員は最小で済むという前提に立つが、実態は「商用でも K8s 基盤とデータストアとイベント層の自社運用工数は残る」。第二の仮説は「OSS の学習コストが見えない費用として企業に積み上がる」というもの。40+ OSS を習得する工数を定量化すれば商用製品の研修費と比較可能となる。第三の仮説は「CVE 対応が自社負担となり、ベンダーサポート同等の速さを出せない」というもの。Renovate と Trivy による自動化工数で再計算すると自社対応の負担は予想より小さい。第四の仮説は「メジャーバージョンアップの破壊的変更に追従できない」というもの。採用 OSS が CNCF Graduated 中心であること、Renovate が PR を自動生成することで人手の検証工数が支配的となる。

以下、4 仮説を順に定量で潰す。

---

## 2. 仮説 1: OSS 運用人件費は商用より高い ── 反証

### 2.1 k1s0 の運用人件費（再掲）

[`03_運用工数試算.md`](./03_運用工数試算.md) 第 3 章より、中規模 (3,000 名) の年間運用保守工数は **18.6 人月 = 1.55 FTE**。内訳は以下。

| カテゴリ | 年間人月 | 比率 |
|---|---|---|
| tier1 自作領域 | 3.8 | 20% |
| k8s 基盤系 | 2.8 | 15% |
| セキュリティ / IAM | 2.7 | 14% |
| イベント / ワークフロー | 2.3 | 12% |
| CI/CD | 2.0 | 11% |
| データストア | 1.8 | 10% |
| 可観測性 | 1.7 | 9% |
| サービスメッシュ | 1.3 | 7% |
| 開発者ポータル | 1.2 | 6% |

加重単価 115 万円/月で換算すると年間 **2,139 万円**。この数字が「OSS 無償なのに年 2,000 万円以上かかる」という懸念の実体である。

### 2.2 商用 IDP 採用時に削減できるのは 3.1 人月のみ

[`03_運用工数試算.md`](./03_運用工数試算.md) 第 5 章の比較表より、OpenShift 採用で置換されるのは以下のみ。

| カテゴリ | k1s0 | OpenShift | 差分 |
|---|---|---|---|
| k8s 基盤系 | 2.8 | 1.5 | **-1.3** |
| サービスメッシュ | 1.3 | 0.5 | **-0.8** |
| 可観測性 | 1.7 | 1.2 | -0.5 |
| CI/CD | 2.0 | 1.2 | -0.8 |
| セキュリティ / IAM | 2.7 | 2.0 | -0.7 |
| 開発者ポータル | 1.2 | 0.5 | -0.7 |
| データストア | 1.8 | 1.8 | 0 |
| イベント / ワークフロー | 2.3 | 2.3 | 0 |
| tier1 自作領域 | 3.8 | 4.5 | **+0.7** |
| **合計** | **18.6** | **15.5** | **-3.1** |

注目すべきは **データストア (1.8) とイベント / ワークフロー (2.3) は商用化しても削減されない** 点。CloudNativePG や Kafka は商用 IDP のスコープ外で、自社運用のまま残る。さらに tier1 自作領域は商用 API ラッパの追加で **工数が増える** (+0.7)。

3.1 人月の削減は加重単価 115 万円で約 **357 万円/年** の価値でしかない。商用ライセンス費 2,400 万円/年 を支払って得られる「人件費削減効果」は年 357 万円。つまりライセンス費の **15% しか人件費削減で回収できない**。

### 2.3 反証まとめ

OSS 無償で年 2,139 万円の運用人件費が発生するのは事実。しかし商用 IDP 採用時も年 1,782 万円 (15.5 人月 × 115 万円) の自社運用人件費が残り、差分は 357 万円のみ。この差分に商用ライセンス費 2,400 万円を足すと、商用導入で **年 2,043 万円の追加支出** となり、運用コストの観点でも商用が不利となる。

## 3. 仮説 2: OSS の学習コストが見えない費用として積み上がる ── 反証

### 3.1 採用 OSS 40+ 本の学習コストを分解

学習コストは「初回習得 (Phase 1a〜1c)」と「メンバー入替時の再習得 (Phase 2 以降)」の 2 種類に分解する。

#### 3.1.1 初回習得コスト（Phase 1a〜1c）

[`02_開発工数試算.md`](./02_開発工数試算.md) の初期構築費 1,850 万円には、OSS の初回習得コストが既に織り込まれている。これは **「既存 VM 流用・2 名体制・起案者 + 中堅 1 名の Phase 1c 完了まで」** の総工数で、純粋な学習工数を切り出すと以下。

| 活動 | 想定人月 | 内訳 |
|---|---|---|
| 採用 OSS 公式ドキュメント読解 | 2.0 | 40 OSS × 0.05 人月 (2 日/OSS) |
| 環境構築と設定の試行錯誤 | 3.0 | kubeadm / Istio / Kafka / Dapr / Argo CD 等で各 0.3〜0.5 人月 |
| リファレンス実装の写経 | 1.5 | tier1 リファレンス実装・Backstage プラグイン |
| 社内向けドキュメント化 | 1.0 | 他メンバーへの引き継ぎを前提とした Runbook 化 |
| **合計** | **7.5 人月** | 加重単価 115 万で約 **863 万円** |

この 863 万円は 1 回限りの投資で、5 年 TCO に均すと年 173 万円。Phase 1c 完了後は次項の「メンバー入替コスト」に移行する。

#### 3.1.2 メンバー入替時の再習得コスト

5 年間でメンバーが入れ替わる想定は中規模で 2 人 (5 FTE の 40% が 5 年で交代)。交代時の引き継ぎコストは以下。

- 社内 Runbook + Backstage TechDocs を読了: 0.3 人月 × 2 人 = 0.6 人月
- ハンズオン + コードレビュー同席: 0.5 人月 × 2 人 = 1.0 人月
- 合計 1.6 人月 = 184 万円を 5 年に均すと **年 37 万円**

#### 3.1.3 学習コスト合計

初回 173 万円/年 + 入替 37 万円/年 = **年 210 万円**。これが OSS 無償の罠として指摘される「学習コスト」の実体である。

### 3.2 商用 IDP でも学習コストは発生する

商用 IDP 採用でも「OpenShift 独自の Operator Hub / OpenShift Routes / OpenShift Service Mesh 独自拡張」の習得は必要。ベンダー研修を受けても公式ドキュメントとの突き合わせ・自社運用 Runbook 化の工数は残る。Red Hat 公式研修コースは 3〜5 日で 30〜50 万円/人。2 人で年 **100 万円** に加え、実環境へのアダプテーション工数が Phase 1 で 3 人月 (345 万円) 発生すると想定すると、5 年で年 **169 万円**。

|  | k1s0 (OSS 40+) | OpenShift 採用時 |
|---|---|---|
| 初回習得 (5 年均し) | 173 万円 | 169 万円 |
| 入替再習得 (5 年均し) | 37 万円 | 50 万円 (ベンダー研修費を含む) |
| **年額** | **210 万円** | **219 万円** |

### 3.3 反証まとめ

学習コストは k1s0 と OpenShift でほぼ同額 (年 210 万円 vs 219 万円)。OSS 無償の「学習コストの隠れ負担」は、商用製品でも同量発生する。商用で下がるのは「基本操作の研修」のみで、自社運用 Runbook 化・環境アダプテーション工数は消えない。

## 4. 仮説 3: CVE 対応がベンダー未満 ── 反証

### 4.1 CVE 対応工数の定量

[`03_運用工数試算.md`](./03_運用工数試算.md) の CVE 列を合計すると、中規模で年 **3.0 人月 (約 345 万円)** が CVE 対応の人件費。これは全運用工数 18.6 人月の 16% にあたる。この 3 人月は以下の自動化で圧縮されている。

- **Renovate**: 採用 OSS 全 40+ 本に対して CVE 付きの更新 PR を自動生成。人手の最初の仕事は「PR が来た」ことの確認のみ
- **Trivy + Harbor**: 新規ビルドで CVE 検出を自動実行、Critical / High を PR ブロック
- **Grype + Syft**: SBOM 経由で CVE を継続監視、ライセンス分類と同時並行

この自動化がなければ CVE 対応は年 8〜10 人月になる。自動化によって **5〜7 人月を圧縮** している。

### 4.2 CVE 対応 SLA の定義

k1s0 は CVE 対応 SLA を **Critical 48 時間 / High 7 日 / Medium 30 日** と内規で定める (詳細は [`../03_ロードマップと体制/03_KPIと承認基準.md`](../03_ロードマップと体制/03_KPIと承認基準.md))。これは業界平均 (Critical 72 時間) を上回る水準で、商用 IDP のベンダー SLA と遜色ない。

| 項目 | k1s0 自社対応 | 商用 IDP ベンダー SLA (OpenShift Premium) |
|---|---|---|
| Critical CVE 通知 | Renovate 即時通知 (通常 24 時間以内) | Red Hat Security Advisory (通常 24〜48 時間以内) |
| パッチ適用完了 | 自社で 48 時間 | ベンダーパッチ提供後、自社適用で 48 時間 |
| 適用失敗時の支援 | 自社トラブルシューティング | 24/7 ベンダーサポート |

自社対応で不利になるのは **「適用失敗時の支援」** のみ。この 1 点を代償にしても、k1s0 は OpenShift と同等以下の時間で CVE 対応が可能。

### 4.3 反証まとめ

CVE 対応は自動化の整備で年 3.0 人月 (345 万円) に抑えられる。商用 IDP 採用で削減できる CVE 対応工数は年 1.0〜1.5 人月 (115〜173 万円) にすぎず、ベンダー SLA のプレミアム契約費 (OpenShift Premium は Standard より 30% 高) を加味すると、**商用化による CVE 対応コストの純削減はほぼゼロ** となる。

## 5. 仮説 4: バージョン追従コストが破壊的 ── 反証

### 5.1 バージョン追従工数の定量

[`03_運用工数試算.md`](./03_運用工数試算.md) の「バージョン」列を合計すると、中規模で年 **4.2 人月 (約 483 万円)**。最大は tier1 Go ファサード 0.5 人月、次点は kubeadm 0.5 人月、OTel Collector 0.2 人月。破壊的変更に追従する工数は全体の 25% 程度で、年 1.1 人月 (約 126 万円) に収まる想定。

### 5.2 CNCF Graduated 中心の選定が工数を抑える

k1s0 の採用 OSS は **80% 以上が CNCF Graduated または LF プロジェクト** で、破壊的変更が稀である。過去 3 年の実績で、採用候補 OSS のうち破壊的変更で 1 人月超の工数を要したのは以下のみ。

| OSS | 事象 | 対応工数 |
|---|---|---|
| Istio 1.18 → 1.20 | EnvoyFilter 書式変更 | 0.3 人月 |
| Kafka 3.x → 4.0 (Strimzi) | KRaft 移行 | 0.8 人月 |
| Backstage v1.x | plugin API 変更 | 0.5 人月/年 |

これらは Phase 2 以降に分散され、1 回あたり 0.5〜0.8 人月で収まる。Phase 1a〜1c の初期段階では CNCF Graduated のみに絞って採用するため、破壊的変更は実質的に発生しない設計となっている。

### 5.3 商用 IDP のライフサイクルも破壊的変更から逃れない

商用 IDP も 5 年間で 2〜3 回のメジャーバージョンアップが発生する。OpenShift 4.x → 5.x、Tanzu 1.x → 2.x の際、自社運用環境の動作確認と再学習は必須。ベンダーサポートがあっても「自社適用工数」は削減されず、年 1.5〜2.0 人月程度は残る。

### 5.4 反証まとめ

バージョン追従コストは年 483 万円だが、CNCF Graduated 中心の選定と Renovate による自動化で破壊的変更の頻度は抑えられている。商用 IDP 採用でも自社適用工数は年 200 万円程度残り、純削減効果は年 150〜200 万円にとどまる。

## 6. 「隠れコスト」を全部足した TCO 再計算

### 6.1 中規模 (3,000 名) の年間総コスト

仮説 1〜4 で定量化した全コストを足し上げると以下。

| 項目 | k1s0 | OpenShift |
|---|---|---|
| 運用人件費 | 2,139 万 (18.6 人月 × 115 万) | 1,782 万 (15.5 人月 × 115 万) |
| 学習コスト (5 年均し) | 210 万 | 219 万 (ベンダー研修込) |
| ライセンス費 | 0 | 2,400 万 |
| CVE 対応プレミアム | 0 | +200 万 (Premium サポート差額) |
| **年額合計** | **2,349 万** | **4,601 万** |
| **5 年累計** | **11,745 万 (≈ 1.17 億)** | **23,005 万 (≈ 2.30 億)** |

> 注: この数字は運用コストのみ。Phase 0〜1c の初期構築費 (1,850 万円) + 内製追加費 (OpenShift で 8,000 万円) + インフラ費 + tier2/tier3 の業務アプリ開発工数は含まない。総 TCO は [`01_TCO5年試算.md`](./01_TCO5年試算.md) を参照。

### 6.2 「無償の罠」仮説との対比

稟議で出る「OSS 無償だけど運用で商用より高くつく」という仮説が成立するためには、**k1s0 の年額運用コスト (2,349 万) が OpenShift の運用人件費 (1,782 万) の 130% 以上** でなければならない。実際の倍率は 132% であり、この数字だけ見ると仮説は成立するように見える。

しかしこの比較は不完全で、**商用ライセンス費 2,400 万 / 年 を足せば OpenShift は年 4,200 万円相当** となり、k1s0 の 2,349 万を大幅に上回る。「無償の罠」仮説は、ライセンス費を視界から外した場合のみ成立する錯視である。

### 6.3 反証の構造的要約

OSS 無償の「隠れコスト」合計は年 2,349 万円であり、これは確かに「無償ではない」。しかし商用 IDP 採用時の「自社運用人件費 + 学習コスト + ライセンス費 + 追加プレミアム」合計は年 4,601 万円となり、2 倍近い差が生じる。**「OSS は無償」は確かに誤解だが、「OSS は運用で商用より高くつく」はそれを上回る誤解** となる。

## 7. 決裁者答弁の雛形

稟議の場で「OSS 無償の罠」を指摘された際の答弁例を示す。

> ご指摘のとおり OSS は無償ではありません。年間運用人件費 2,139 万円、学習コスト 210 万円、CVE 対応プレミアムゼロ、合計年 2,349 万円の自社負担が発生します。
>
> ただし商用 IDP を採用した場合も、OpenShift では年 2,400 万円のライセンス費に加え、自社運用人件費 1,782 万円・学習コスト 219 万円・プレミアムサポート差額 200 万円が発生し、合計 4,601 万円となります。
>
> 運用人件費だけの比較では商用が 357 万円安くなりますが、ライセンス費を足すと OSS 採用の方が 2,252 万円 / 年 安くなります。5 年では 1.13 億円の差です。「OSS は無償ではないが、商用より運用コストは低い」が正確な表現と考えます。

## 8. 感度分析と反証の強度

### 8.1 自動化が崩れた場合 (Renovate / Trivy / Grype がすべて無効)

Renovate / Trivy / Grype の自動化がすべて停止した場合、CVE 対応が 3.0 → 8.0 人月に増加し、バージョン追従が 4.2 → 8.0 人月に増加する。合計 8.8 人月の増分 (年 1,012 万円) が発生。このシナリオでも k1s0 の年間総コストは 2,349 + 1,012 = 3,361 万円で、OpenShift の 4,601 万円を下回る。自動化が完全に崩れても反証は成立する。

### 8.2 社内人月単価が想定より 30% 高い場合

加重単価 115 → 150 万円/月 (SES シニア水準) にずれた場合、k1s0 の運用人件費は 2,139 → 2,790 万円、OpenShift は 1,782 → 2,325 万円。差分 465 万円が拡大するが、ライセンス費 2,400 万円との相殺で依然として OpenShift が不利となる。

### 8.3 中規模で OSS を削減 (40 本 → 20 本) した場合

OSS 採用本数を半減させると運用人件費は 10 人月相当 (1,150 万円) まで下がる。このシナリオでは「k1s0 年 1,360 万 vs OpenShift 年 4,601 万」で、差が 3,241 万円に拡大する。ただし削減すると k1s0 の機能が不足し、商用製品との機能同等性が崩れるため、現実的でない。

### 8.4 反証の頑健性

3 つの感度分析すべてで k1s0 の年間総コストが OpenShift を下回るため、本書の反証は「OSS 無償の罠」仮説に対して構造的に頑健である。稟議の場での反論は想定される範囲で事前に封じてある。

## 9. 本書の限界と免責

本書の数字は [`00_試算前提.md`](./00_試算前提.md) の人月単価と [`03_運用工数試算.md`](./03_運用工数試算.md) の工数見積もりに依存する。これらの見積もりは自社経験則と JUAS / JISA の公開調査に基づく推定値で、実際の運用環境では変動する。特に以下の要因で結論が変わる可能性がある。

- 採用 OSS が追加で 20 本以上増加した場合、運用人件費が想定を 30% 超過
- 社内エンジニアの習熟度が想定より低く、学習コストが 2〜3 倍に膨らむ
- 商用 IDP ベンダーが大幅な値下げキャンペーンを実施
- 為替 1 USD = 200 円 (現在 150 円) への大幅円安で商用ライセンス費が 33% 増 → k1s0 有利が加速

本書は稟議承認時点のスナップショットであり、Phase 1c 完了時点で実績値を用いた再試算が必須である。
