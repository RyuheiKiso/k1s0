# 運用工数試算

## 目的

Phase 2 以降の k1s0 運用に必要な年間工数を、採用 OSS ごとに分解して試算する。[`02_開発工数試算.md`](./02_開発工数試算.md) が「作る工数」であるのに対し、本資料は「保守する工数」を扱う。CVE 対応・バージョン追従・障害対応・リファレンス更新の 4 活動について OSS 単位で積み上げ、年間の運用人員 FTE を導出する。本試算の数字は [`01_TCO5年試算.md`](./01_TCO5年試算.md) の Build 側コスト構造に直接入力される。

本資料で導く結論は以下の四点に集約される。まず、中規模 (3,000 名) で採用 OSS 全体の年間保守工数は 20.4 人月 = 1.70 FTE となり、商用 IDP で自社運用が不要になる訳ではない（3,000 名で 3 FTE の社内運用人員は商用でも必要）という事実を明らかにする。次に、保守工数は OSS の数に比例せず、tier1 自作領域（Go + Rust + ZEN Engine + 雛形 CLI）とポータル群（Backstage + アプリ配信ポータル）の 2 つで全体の 30% を占める構造となる。第三に、**2 ワークフロー基盤 (Dapr Workflow + Temporal) 併用ゆえの固有工数 0.5 人月/年・2 ポータル (Backstage + アプリ配信ポータル) 併用ゆえの固有工数 0.3 人月/年** を独立項目として計上し、「併用は運用を二重化するのでは」という稟議論点に定量で応答できる構造にする。最後に、規模による保守工数の伸びは人員に比例せず（小 70% / 中 100% / 大 150%）、規模効果が効くことを示す。

---

## 1. 試算方針

### 1.1 保守工数の 4 活動

採用 OSS の年間保守工数を以下の 4 活動に分解する。

- **CVE 対応**: 採用 OSS に公表された脆弱性への対処（影響評価・パッチ適用・検証）。CVSS スコア 7.0 以上は原則 30 日以内に対応する運用想定
- **バージョン追従**: マイナー・メジャーバージョンアップへの追従。Renovate により PR 自動生成されるが、動作確認・統合テスト・Runbook 更新は人手
- **障害対応**: 本番障害の一次調査・Runbook 実行・根本原因分析・再発防止策
- **リファレンス更新**: 新機能や設定変更に伴うドキュメント更新・リファレンス実装の改修

### 1.2 見積もりの単位

各 OSS について、4 活動の年間工数を 0.1 人月（約 2 人日）刻みで見積もる。見積もりの根拠は以下の 3 種類を採用する。

- CNCF Landscape の成熟度（Graduated / Incubating / Sandbox）
- 過去 24 ヶ月の CVE 公表頻度（NVD 参照）
- 採用 OSS のメジャーバージョン更新頻度（GitHub Releases 参照）

### 1.3 規模スケール係数

OSS 保守工数は規模に線形比例せず、「同じ OSS を運用するコスト」の基盤部分は規模に依らず発生する。本試算では規模スケール係数を以下で設定する。

- 小 (1,000 名): × 0.7 — パイロット業務 1 本、ピーク RPS 50、障害対応頻度が低い
- 中 (3,000 名): × 1.0 — 基準値
- 大 (10,000 名): × 1.5 — パイロット業務 3 本、ピーク RPS 500、障害対応頻度が増え、リファレンス改修も複数部門向けに拡張

#### 2.7.1 2 ワークフロー基盤併用固有工数

Dapr Workflow (短期フロー用) と Temporal (長期フロー用) を併用する構成は、個別 OSS の運用工数だけでは捉えきれない**併用ゆえの固有コスト** を発生させる。企画書 13.5 章で棲み分け基準を定めたが、運用段階では基準の境界ケース (10 ステップ・数分ちょうど等) が必ず出現し、振り分けの判断と再設定を継続して要する。稟議で「なぜ 1 つに統合しないのか」と問われた際、単に「機能が違うから」と答えるのではなく、併用コストを定量化した上で「この工数を払ってでも 2 基盤構成が合理的」と説明する必要がある。

併用固有工数は以下の 3 活動で発生する。いずれも Dapr Workflow 単体・Temporal 単体のいずれかで運用する構成では発生しない増分コストである。

- **振り分け設定の保守**: tier1 Go ファサードが workflowType (YAML 設定) を見て自動振り分けする仕組みを維持する工数。新しい workflowType の追加・既存タイプの基盤変更 (Dapr Workflow → Temporal への昇格) の際、ルーティングテーブルの更新と回帰試験が必要
- **障害時切り分け Runbook と演習**: `k1s0.Workflow.StartAsync` が失敗した時に「Dapr Workflow 側で詰まったのか Temporal 側で詰まったのか」を判別する Runbook を整備し、四半期ごとの障害演習で両基盤のシミュレーションを実施
- **tier2/3 開発者への教育二重化**: 「Saga パターンは Dapr Workflow、人的承認が絡むのは Temporal」という判断基準を tier2/3 開発者に継続的に周知する (新規 OJT・四半期リファレンス更新)

| 活動 | 年間工数 | 根拠 |
|---|---|---|
| 振り分け設定保守 | 0.2 人月 | 新規 workflowType 追加 2 件/四半期 × 0.05 人月/件 = 0.2 人月/年 (過去の tier1 API 追加ペースから外挿) |
| 障害時切り分け Runbook・演習 | 0.2 人月 | Runbook 更新 0.1 人月 + 四半期演習 4 回 × 0.025 人月 = 0.2 人月/年 |
| tier2/3 開発者教育 (OJT + 四半期ドキュメント更新) | 0.1 人月 | ドキュメント更新 0.05 人月 + 新規メンバーへの OJT 平均 3 回/年 × 0.02 人月 = 0.11 人月/年 |
| **2 基盤併用固有工数 合計** | **0.5 人月/年** | |

#### 2.7.2 併用の合理性の再確認 (稟議答弁用)

併用固有工数 0.5 人月/年 ≒ **58 万円/年** (社員単価 115 万/月換算) は、1 基盤構成で失う価値と比較して判断する。Dapr Workflow 単独では数週間にわたる稟議承認フローを表現できず (タイマーと人的承認の制御が弱い)、Temporal 単独では「短期 Saga で Temporal Server の運用学習コストを払う」という過剰さが生じる。企画書 13.5 章の棲み分け表 (秒〜数分 vs 数時間〜数週間 / 10 ステップ以下 vs 数十〜数百ステップ) にあるとおり、JTC の業務フローは両極端が混在しており、単一基盤では構造的にカバーできない。

併用構成で得られる具体的価値は [`../../02_構想設計/02_tier1設計/04_診断と補償/05_ワークフロー振り分け基準.md`](../../02_構想設計/02_tier1設計/04_診断と補償/05_ワークフロー振り分け基準.md) 第 3 節に詳述されている。年 58 万円の併用コストで「稟議承認 3 か月フローの人的承認タイマー制御」「個人情報削除 GDPR 30 日猶予期間の時限実行」など、1 基盤では表現できない長期フローを成立させる、というのが答弁の核。

#### 2.9.3 2 ポータル併用固有工数

2 ポータル併用は「同じ認証基盤を使いながら対象ユーザー層が異なる」運用を継続することに固有のコストを発生させる。単一ポータル構成ではこれらは発生しない。稟議で「Backstage に統合できないか」と問われた場合、本節の工数と引き換えに失う価値を答弁する必要がある。

- **認証統合保守 (Keycloak Realm 間)**: 同一 Keycloak に 2 つのクライアント (Backstage / アプリ配信ポータル) を登録し、それぞれの Role Mapping が退行しないよう四半期ごとに監査。テナント追加時の両クライアント同時更新も要する
- **導線整合 (相互リンク)**: アプリ配信ポータルのアプリ詳細ページから Backstage のサービスカタログへの相互リンク、開発者が業務担当目線の UX を確認するための導線維持
- **ドキュメント分離の維持**: 「業務担当向けは配信ポータル」「開発者向けは Backstage」という役割分担を崩さないよう、情シス内のドキュメント配置を継続的に整理 (混同するとどちらのポータルを使うべきかの問い合わせが増える)

| 活動 | 年間工数 | 根拠 |
|---|---|---|
| 認証統合保守 (Keycloak Realm / Client 間の Role Mapping 監査) | 0.1 人月 | 四半期監査 4 回 × 0.025 人月 = 0.1 人月/年 |
| 導線整合 (相互リンク保守・UX 確認) | 0.1 人月 | リンク URL 変更対応 + 月次 UX 確認 12 回 × 0.008 人月 = 0.1 人月/年 |
| ドキュメント分離の維持 (情シス内の問い合わせ対応含む) | 0.1 人月 | 問い合わせ対応 月 2 回 × 0.004 人月 × 12 か月 = 0.1 人月/年 |
| **2 ポータル併用固有工数 合計** | **0.3 人月/年** | |

#### 2.9.5 併用の合理性の再確認 (稟議答弁用)

2 ポータル併用固有工数 0.3 人月/年 ≒ **35 万円/年** (社員単価 115 万/月換算) は、統合案で失う価値と比較して判断する。Backstage に統合する案は業務担当ユーザーを Backstage の開発者向け UI に露出させるため、業務担当向けの簡易 UX (アプリストア風・端末設定コピー) が成立しない。単一ポータル案で Backstage を業務担当向けに簡素化するカスタマイズを行うと、逆に 0.5〜0.8 人月/年 の独自プラグイン保守工数が発生し、分離運用より高くつく。

この分析は [`../../02_構想設計/04_CICDと配信/02_アプリ配信ポータル.md`](../../02_構想設計/04_CICDと配信/02_アプリ配信ポータル.md) 第 2 節の統合 vs 分離の比較に基づく。年 35 万円の併用コストで「業務担当が開発者 UI を見ずに済む」「端末設定コピーを 1 クリックで実現する」という tier3 固有体験を成立させる、というのが答弁の核。

| ポータル群 (2 ポータル併用含む) | 2.5 | 12% | うち 2 ポータル併用固有工数 0.3 |
| **合計** | **20.4 人月/年** | **100%** | 2 併用固有工数 計 0.8 (全体の 4%) |

年間 20.4 人月 = 1.70 FTE。内訳では「tier1 自作領域 (19%)」「k8s 基盤系 (14%)」「イベント / ワークフロー (14%)」の 3 カテゴリで 47% を占める。**2 併用固有工数 (ワークフロー 0.5 + ポータル 0.3 = 0.8 人月/年)** は全体の 4%。商用製品では「tier1 自作領域 19%」が不要になる（tier1 ファサードが商用機能に置換されれば）が、k8s 基盤系・イベント基盤・ポータル群の保守は商用でも自社負担として残存する。この構造が「商用に切り替えても運用工数は半減しない」という [`01_TCO5年試算.md`](./01_TCO5年試算.md) の前提を支える。

### 3.1 併用固有工数の位置付け

2 ワークフロー基盤併用 (0.5) と 2 ポータル併用 (0.3) の固有工数 0.8 人月/年 ≒ **92 万円/年** は、以下の価値と引き換えに発生する投資である。稟議で「併用は二重化コストではないか」と問われた際、この 0.8 人月/年 を以下の効果と結びつけて答弁する。

- **2 ワークフロー基盤併用の効果** — 稟議承認 3 か月フロー・個人情報削除 GDPR 30 日猶予など、単一基盤では表現できない長期フローを成立させる (2.7.2 節)
- **2 ポータル併用の効果** — 業務担当ユーザーに開発者 UI を露出させず、tier3 固有の簡易 UX (アプリストア風・端末設定コピー) を成立させる (2.9.5 節)

いずれも統合案で実現するには、独自カスタマイズ工数が併用固有工数を上回る。併用は「楽をしているから二重化している」のではなく「統合するより安い」という定量的合理性を持つ。

## 4. 規模別の年間保守工数

### 4.1 スケール適用後の集計

| 規模 | 係数 | 年間人月 | FTE |
|---|---|---|---|
| 小 (1,000 名) | × 0.7 | 14.3 | 1.19 |
| 中 (3,000 名) | × 1.0 | 20.4 | 1.70 |
| 大 (10,000 名) | × 1.5 | 30.6 | 2.55 |

保守工数は 20.4 人月を基準として規模で伸縮するが、これは「OSS を動かし続ける + 2 併用固有工数」の数字である。tier2 オンボーディング支援・業務ロジック実装・業務要件対応など「k1s0 の価値を顧客（社内部門）に届ける」活動は本試算に含まれず、[`02_開発工数試算.md`](./02_開発工数試算.md) 第 8 章の年間人員想定に追加分として計上されている (総 FTE は 3.0 / 5.0 / 10.0 のまま変わらず、内訳のうち保守側の精度が上がった形)。

### 4.2 規模別の総人員（保守 + 価値提供）内訳

| 規模 | 保守 (本資料) | 価値提供 (02 章 8) | 合計 FTE |
|---|---|---|---|
| 小 (1,000 名) | 1.19 | 1.81 | 3.0 |
| 中 (3,000 名) | 1.70 | 3.30 | 5.0 |
| 大 (10,000 名) | 2.55 | 7.45 | 10.0 |

総 FTE は [`01_TCO5年試算.md`](./01_TCO5年試算.md) と整合する 3.0 / 5.0 / 10.0 を維持する。本資料で保守側を 1.55 → 1.70 FTE に精緻化した結果、価値提供側が 3.45 → 3.30 FTE に再配分される。この再配分は 2 併用固有工数 (0.8 人月/年 ≒ 0.07 FTE 相当) が保守に移動した分に相当する。価値提供側の工数が保守工数の 2 倍前後となる点は、k1s0 が「運用負担を下げて業務への貢献を増やす」設計思想と整合する。

## 5. 商用 IDP 採用時との比較

商用 IDP (OpenShift / Tanzu) を採用した場合、本資料のカテゴリの一部がベンダー保守に置換される。ただし完全に不要となる訳ではなく、「OSS が閉鎖的になる分、社内からの要望対応とカスタマイズ工数が増える」側面もある。

| カテゴリ | k1s0 (自社運用) | OpenShift 採用時 | 差分 |
|---|---|---|---|
| k8s 基盤系 | 2.8 | 1.5 (ベンダー保守の残り運用工数) | -1.3 |
| サービスメッシュ | 1.3 | 0.5 | -0.8 |
| 可観測性 | 1.7 | 1.2 (OpenShift 内蔵部分はベンダー保守) | -0.5 |
| データストア | 1.8 | 1.8 (CloudNativePG はベンダー非対応で残る) | 0 |
| CI/CD | 2.0 | 1.2 (OpenShift Pipelines 相当) | -0.8 |
| セキュリティ / IAM | 2.7 | 2.0 (Keycloak は残る) | -0.7 |
| イベント / ワークフロー (2 WF 併用含む) | 2.8 | 2.8 (OpenShift 範囲外、併用構成は維持) | 0 |
| tier1 自作領域 | 3.8 | 4.5 (商用 API ラッパ追加で増加) | +0.7 |
| ポータル群 (2 ポータル併用含む) | 2.5 | 1.8 (OpenShift Dev Console が Backstage を部分代替、配信ポータル自製は残る) | -0.7 |
| **合計** | **20.4** | **17.3** | **-3.1 人月/年** |

OpenShift 採用で年間 3.1 人月（約 370 万円）を削減できるが、同時にライセンス費 1,500 万〜4,800 万円/年が加算される（規模依存）。人件費削減効果がライセンス費を下回る構造は [`01_TCO5年試算.md`](./01_TCO5年試算.md) で明示する。なお 2 併用固有工数 (0.8 人月/年) は商用採用後も維持される (Temporal は OpenShift 範囲外、アプリ配信ポータルは商用で代替不能) ため、両者の差分計算には影響しない。

## 6. 不確実性

本試算の精度に関する主な不確実性は以下。

- CVE 公表頻度は過去 24 ヶ月平均で見積もっており、Zero-Day が集中した場合は工数が 1.5 倍に増える
- 採用 OSS がライセンス変更・主要メンテナー離脱した場合、代替 OSS への移行工数が別途発生する（前例: Redis → Valkey）
- tier1 自作領域の工数は社内ユーザー数に強く依存し、大規模では要望対応が急増する可能性がある
- 業務部門からの機能要望の優先度判断（プロダクトマネジメント）工数は本試算に含まれていない
