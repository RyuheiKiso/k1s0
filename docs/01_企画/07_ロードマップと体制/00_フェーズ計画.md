# フェーズ計画

## 目的

k1s0 の段階的な導入計画を整理する。MVP (Phase 1) の詳細は [`01_MVPスコープ.md`](./01_MVPスコープ.md)、体制は [`02_体制と役割.md`](./02_体制と役割.md) を参照。

---

## 1. 全体フェーズ俯瞰

| フェーズ | 位置付け | 主な成果物 |
|---|---|---|
| **Phase 0** | 事前資料 / 企画承認 (**現在ここ**) | 企画書 / 技術選定 / 競合分析 / MVP スコープ定義 |
| **Phase 1 (MVP)** | 最小構成の稼働、起案者単独で立ち上げ可能 | infra 最小構成 + tier1 の基本 API + Backstage + アプリ配信ポータル (Web 配信のみ) |
| **Phase 2** | 機能拡張、2〜3 名の協力体制 | tier1 state/pub-sub/workflow + tier2 サンプル + 端末台帳 + 権限ポリシー + Backstage プラグイン |
| **Phase 3** | エンドユーザー体験の拡充 | tier3 サンプル + ネイティブ配信 (MSIX) + 端末設定コピー + ZEN Engine 本格利用 |
| **Phase 4** | 業務運用フェーズ | 申請ワークフロー / レガシー .NET Framework 共存 / 業務担当による JDM 編集 |
| **Phase 5** | 全社ロールアウト | 本番展開 / マルチクラスタ / レガシーラップ配信 / 業務ルール大半の宣言化 |

---

## 2. フェーズ別の詳細

### Phase 0: 企画承認 (現在)

- 本企画書および配下ドキュメントの整備
- 決裁者への説明と承認取得
- パイロット業務の選定開始

### Phase 1 (MVP)

**目的**: 最小構成で「k1s0 の 5 つの勝ち筋 ([`../06_競合と差別化/02_差別化ポイント.md`](../06_競合と差別化/02_差別化ポイント.md))」を小規模でも示す。

- infra 最小構成 (k8s + Istio + Envoy Gateway + Kafka + Loki + Prometheus + Jaeger)
- Dapr Control Plane
- tier1 最小 API (`k1s0.Log` / `k1s0.Telemetry` / 雛形生成 CLI)
- Backstage (Software Catalog + TechDocs + SSO)
- Keycloak (ローカル DB)
- Harbor + Trivy
- Argo CD (tier1 向け ApplicationSet)
- GitHub Actions self-hosted runner
- アプリ配信ポータル (Web アプリへの起動リンク + 監査ログ)
- ZEN Engine は API スタブのみ (実装は Phase 2)

詳細は [`01_MVPスコープ.md`](./01_MVPスコープ.md)。

### Phase 2

- tier1 API 拡張 (`k1s0.State` / `k1s0.PubSub` / `k1s0.Workflow` / `k1s0.Secrets` / `k1s0.Decision`)
- tier2 サンプルサービス (業務ドメインロジックの最小例)
- アプリ配信ポータル: 端末台帳 / 権限ポリシー / 部門フィルタ / 利用統計 / お知らせ
- Backstage Software Templates (雛形生成 CLI 統合) + Kubernetes / Jaeger / Grafana / GitHub Actions / Argo CD プラグイン
- Cosign 署名 + Kyverno 未署名ブロック
- Argo Rollouts によるカナリア / Blue-Green
- Trivy DB のオフラインミラー運用 (JTC 環境向け)
- **ZEN Engine 本格稼働** (tier1 Rust サービスに埋め込み、初期 JDM 3〜5 件)
- Keycloak AD 連携の検討開始

### Phase 3

- tier3 サンプルアプリ
- ネイティブインストーラ配信 (MSIX)
- 自動更新
- レビュー / 評価機能
- **端末設定コピー** (旧端末→新端末の一括復元)
- マルチクラスタ (staging / prod 分離)
- External Secrets Operator + Keycloak バックアップ
- ZEN Engine を権限ポリシー / 申請ワークフローで本格利用
- Backstage に JDM Editor プラグイン追加

### Phase 4

- 申請ワークフロー / 稟議承認の電子化
- 評価ランキング / レコメンド
- 同僚 / 部署テンプレートからのコピー
- **レガシー .NET Framework 共存** (サイドカー方式 / API Gateway 経由)
- **業務担当が JDM Editor で直接編集** (情シス稟議ルールの画面編集)

### Phase 5

- 本番展開 / 全社ロールアウト
- レガシー .NET Framework アプリのラップ配信 (ClickOnce / MSIX / 配信エージェント)
- tier2 業務サービスで ZEN Engine 本格採用 (経費・勤怠等の自動判定)
- 業務ロジックの大半が宣言的に外出しされる

## 3. フェーズ移行の判断基準

各フェーズから次フェーズへの移行は、**技術的成熟** と **組織合意** の両方が揃った時点で行う。

| 遷移 | 技術的判断基準 | 組織的判断基準 |
|---|---|---|
| Phase 0 → 1 | MVP スコープが定義済み / 試行環境が確保 | 企画承認 |
| Phase 1 → 2 | tier1 リファレンス実装で GHA フロー疎通 / MVP が 1 業務で稼働 | 2〜3 名の協力体制組成 / パイロット業務が成功 |
| Phase 2 → 3 | tier1 公開 API の主要 building block が安定 | tier2 サンプルの運用実績 |
| Phase 3 → 4 | アプリ配信ポータルがエンドユーザーに受容 | 業務担当 / 情シスが JDM 編集に関心を示す |
| Phase 4 → 5 | 申請ワークフローが 1 業務で稼働 | 全社ロールアウトの稟議承認 |
