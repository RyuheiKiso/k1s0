# Build vs Buy のリスク構造

## 目的

k1s0 を内製する (Build) か商用製品を購入する (Buy) かの選択は企画書の核心論点である。本資料は **リスクの質的構造** に絞って Build / Buy を対比する。金額そのもの（5 年 TCO・年額ライセンス・撤退コスト）は [`../04_定量試算/01_TCO5年試算.md`](../04_定量試算/01_TCO5年試算.md) を **単一の一次ソース** として扱い、本資料に数値表を重複して持たない。

決裁者が本章を読んで得るべき結論は、「Build と Buy はリスクの量ではなく質が違う」という理解である。数値差は次章に委ね、本章はその数値差が生まれる構造的理由を言語化する。

---

## 1. 金額比較の結論（単一ソースへの入口）

5 年 TCO の実数比較（1,000 / 3,000 / 10,000 名の 3 規模）は [`../04_定量試算/01_TCO5年試算.md`](../04_定量試算/01_TCO5年試算.md) に集約した。結論のみ再掲すると、k1s0 は 3 規模すべてで最安となり、次点の商用案に対して 15〜20% の TCO 優位を維持する。

金額の出典（人月単価・規模別クラスタ構成・ライセンス単価・為替前提）は [`../04_定量試算/00_試算前提.md`](../04_定量試算/00_試算前提.md) に集約している。数字を問われた場合は常に定量試算章を参照する運用とし、本章で数値を保持して二重管理するのは避ける。

---

## 2. Build (k1s0 を内製) のリスクと対処

Build 側のリスクは「個人開発起点である」という組織的事情に由来するものが主で、組織として段階的に引き取る体制を整えれば低減できる。以下はそれぞれのリスクと、稟議承認時点で既に計画済みの対処。

| リスク | 発生確率 | 影響度 | 対処方針 |
|---|---|---|---|
| 個人開発ゆえの属人化 | 高 | 高 | **ドキュメント充実 + コード公開** でバス係数を下げる。MVP-0 → MVP-1 の 2 段階分割で協力者獲得を構造化 |
| 開発が途中で頓挫する | 中 | 高 | MVP スコープを最小化して早期に価値提供。[`../03_ロードマップと体制/01_MVPスコープ.md`](../03_ロードマップと体制/01_MVPスコープ.md) で MVP-0 (2 週間デモ) を先行させる |
| 新技術追従の工数が読めない | 中 | 中 | OSS 積み上げなのでコミュニティ更新に追従するだけ。内製フォークを禁止して追従コストを制御 |
| セキュリティ脆弱性の対応遅延 | 中 | 高 | 採用 OSS の CVE 監視と tier1 の修正体制を事前に設計。Renovate + Trivy で検知自動化 |
| 所属企業の評価 / 支援が得られない | 中 | 中 | 無償提供の時点で ROI を明示して合意を取る。[`../04_定量試算/01_TCO5年試算.md`](../04_定量試算/01_TCO5年試算.md) で数値化 |

---

## 3. Buy (商用製品を購入) のリスク

Buy 側のリスクは「契約構造・ベンダー支配力」という構造的事情に由来するものが多く、契約締結後の解消は困難である。k1s0 を採用することで、以下リスクの大半は発生自体を回避できる。

| リスク | 発生確率 | 影響度 | 回避可否 |
|---|---|---|---|
| ライセンス費が予算を圧迫 | 高 | 高 | k1s0 なら回避 |
| ベンダーロックイン | 高 | 極高 | k1s0 なら回避 |
| ベンダー都合の仕様変更 / 値上げ (例: VMware Broadcom 事例) | 中 | 極高 | k1s0 なら回避 |
| 自社要件 (.NET Framework 共存等) が製品側でサポートされない | 高 | 高 | k1s0 なら自由に対応 |
| 日本法人のサポート品質が担保されない | 中 | 中 | k1s0 は自社運用で品質自己決定 |

ベンダーロックインを「高・極高」と位置付ける根拠は、2023〜2024 の VMware Broadcom 買収事例・Red Hat Free 版打ち切り事例で、**契約後の解消コストが初期ライセンス費を上回る** 実例が複数観測されたためである。本資料は個別事例の詳細に踏み込まないが、定量試算章の「撤退コスト」項目で数値化している。

---

## 4. リスクの構造的な違い

Build と Buy のリスクは **質が異なる** ことを理解することが、本章の最終目的である。以下の 3 軸で対比する。

| 観点 | Build (k1s0) | Buy (商用) |
|---|---|---|
| リスクの源 | 「個人開発起点である」という組織的事情 | 「契約構造・ベンダー支配力」という構造的事情 |
| 解消可能性 | 段階的に会社として引き取る体制を整えれば **低減できる** | 契約締結後の解消は **困難** (撤退コストが大きい) |
| 時間軸 | 初期ほどリスクが高く、運用段階で下がる | 初期は安全に見え、契約更新ごとにリスクが顕在化する |

Build のリスクは時間とともに **減衰する構造** を持つのに対し、Buy のリスクは時間とともに **顕在化が蓄積する構造** を持つ。両者の TCO 差が 5 年スパンで広がる理由も、この時間軸上の非対称性から説明される。

---

## 5. 企画書での論旨構成

決裁者向けの答弁は以下の順で組み立てる。機能論・差別化論・金額論・リスク論の順で積み上げ、最後に時間軸を導入する。

1. **機能重複の事実を認める** — 既存製品で同等の機能は実現できる
2. **商用製品では JTC 固有要件が埋まらない** — レガシー共存 / オンプレ / 稟議文化 / 言語自由 ([`02_差別化ポイント.md`](./02_差別化ポイント.md))
3. **5 年 TCO で見ると k1s0 が優位** — 初期は高いが運用段階で下がる / ロックイン解消費が不要 ([`../04_定量試算/01_TCO5年試算.md`](../04_定量試算/01_TCO5年試算.md))
4. **Build のリスクは対処可能** — ドキュメント・MVP スコープ・OSS 積み上げ方針で個人開発リスクを段階的に解消する（本章第 2 節）
5. **Buy のリスクは構造的で対処困難** — 契約更新ごとに顕在化し、撤退コストが膨大（本章第 3〜4 節）

この 5 段階のうち、3 段目（金額優位）は **定量試算章を単一ソースとして参照** し、本章では数値の重複掲載を避ける。数値が揺れた場合の唯一の修正先を定量試算章に固定することで、情報ドリフトを構造的に防ぐ。
