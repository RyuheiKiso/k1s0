# Build vs Buy のリスク構造

## 目的

k1s0 を内製する (Build) か商用製品を購入する (Buy) かの選択は企画書の核心論点である。本資料は **リスクの質的構造** に絞って Build / Buy を対比する。金額そのもの（5 年 TCO・年額ライセンス・撤退コスト）は [`../04_定量試算/01_TCO5年試算.md`](../04_定量試算/01_TCO5年試算.md) を **単一の一次ソース** として扱い、本資料に数値表を重複して持たない。

加えて、Build 側にも k1s0 とは別の構成 (素 K8s + Backstage + Dapr SDK 直叩きで tier1 facade を持たない最小内製案、以下 Build-lite) が成立することを認識する必要がある。商用製品との比較だけで結論すると「商用に勝つだけなら Build-lite で十分」という反論を受けるため、本資料では **k1s0 vs Build-lite vs Buy** の三項対立として構造化する。

決裁者が本章を読んで得るべき結論は、「Build (k1s0) と Build-lite と Buy はそれぞれ質の異なるリスクを引き受ける選択であり、量ではなく質で判断する必要がある」という理解である。数値差は次章に委ね、本章はその数値差が生まれる構造的理由を言語化する。

---

## 1. 金額比較の結論（単一ソースへの入口）

5 年 TCO の実数比較（1,000 / 3,000 / 10,000 名の 3 規模）は [`../04_定量試算/01_TCO5年試算.md`](../04_定量試算/01_TCO5年試算.md) に集約した。結論のみ再掲すると、k1s0 は 3 規模すべてで最安となり、次点の商用案に対して 15〜20% の TCO 優位を維持する。

金額の出典（人月単価・規模別クラスタ構成・ライセンス単価・為替前提）は [`../04_定量試算/00_試算前提.md`](../04_定量試算/00_試算前提.md) に集約している。数字を問われた場合は常に定量試算章を参照する運用とし、本章で数値を保持して二重管理するのは避ける。

---

## 2. Build (k1s0 を内製) のリスクと対処

Build 側のリスクは「個人開発起点である」という組織的事情に由来するものが主で、組織として段階的に引き取る体制を整えれば低減できる。以下はそれぞれのリスクと、稟議承認時点で既に計画済みの対処。

| リスク | 発生確率 | 影響度 | 対処方針 |
|---|---|---|---|
| 個人開発ゆえの属人化 | 高 | 高 | **ドキュメント充実 + コード公開** でバス係数を下げる。MVP-0 → MVP-1 の 2 段階分割で協力者獲得を構造化 |
| 開発が途中で頓挫する | 中 | 高 | MVP スコープを最小化して早期に価値提供。[`../03_ロードマップと体制/01_MVPスコープ.md`](../03_ロードマップと体制/01_MVPスコープ.md) で MVP-0 (2 週間デモ) を先行させる |
| 新技術追従の工数が読めない | 中 | 中 | OSS 積み上げなのでコミュニティ更新に追従するだけ。内製フォークを禁止して追従コストを制御 |
| セキュリティ脆弱性の対応遅延 | 中 | 高 | 採用 OSS の CVE 監視と tier1 の修正体制を事前に設計。Renovate + Trivy で検知自動化 |
| 所属企業の評価 / 支援が得られない | 中 | 中 | 無償利用許諾の時点で ROI と利用範囲を明示して合意を取る。[`../04_定量試算/01_TCO5年試算.md`](../04_定量試算/01_TCO5年試算.md) で数値化 |

---

## 3. Buy (商用製品を購入) のリスク

Buy 側のリスクは「契約構造・ベンダー支配力」という構造的事情に由来するものが多く、契約締結後の解消は困難である。k1s0 を採用することで、以下リスクの大半は発生自体を回避できる。

| リスク | 発生確率 | 影響度 | 回避可否 |
|---|---|---|---|
| ライセンス費が予算を圧迫 | 高 | 高 | k1s0 なら回避 |
| ベンダーロックイン | 高 | 極高 | k1s0 なら回避 |
| ベンダー都合の仕様変更 / 値上げ (例: VMware Broadcom 事例) | 中 | 極高 | k1s0 なら回避 |
| 自社要件 (.NET Framework 共存等) が製品側でサポートされない | 高 | 高 | k1s0 なら自由に対応 |
| 日本法人のサポート品質が担保されない | 中 | 中 | k1s0 は自社運用で品質自己決定 |

ベンダーロックインを「高・極高」と位置付ける根拠は、2023〜2024 の VMware Broadcom 買収事例・Red Hat Free 版打ち切り事例で、**契約後の解消コストが初期ライセンス費を上回る** 実例が複数観測されたためである。本資料は個別事例の詳細に踏み込まないが、定量試算章の「撤退コスト」項目で数値化している。

## 4. Build (k1s0) vs Build-lite の質的差

商用 IDP との比較ばかりに焦点を当てると、決裁の場で「商用に勝つだけなら tier1 facade を持たない Build-lite (素 K8s + Backstage + Dapr SDK 直叩き) で十分では?」という反論を受ける。本節ではこの反論への構造的回答を提示する。金額差 (中規模 5 年で k1s0 が Build-lite に対して 0.72 億上乗せ) は [`../04_定量試算/01_TCO5年試算.md`](../04_定量試算/01_TCO5年試算.md) 第 6 章を参照。

Build-lite は機能的には素直な「素 K8s + Backstage」案だが、k1s0 が解決する 6 つの構造課題を引き受けない。下記 6 点は [`../01_背景と目的/00_背景と課題.md`](../01_背景と目的/00_背景と課題.md) で論じた JTC 情シスの構造課題に対する k1s0 の核心的回答であり、Build-lite を選ぶことは「これらの解決を諦める」ことを意味する。

| 項目 | Build-lite が引き受けるリスク | k1s0 の構造的回答 |
|---|---|---|
| Dapr バージョン追従 | tier2/3 の全サービスを一斉に書き換える必要 (Dapr v1.13 Workflow API 改訂のような破壊的変更が直撃) | tier1 facade で変更を tier1 内に閉じ込め、tier2/3 のコードを変えない |
| アプリ配信ポータル | Backstage Catalog の URL リンク列挙のみ。端末台帳・部門フィルタ・設定コピー・レビュー・お知らせ機能なし | JTC 固有要件 (PC リプレース時の情シス工数圧縮 / エンドユーザー導線) を構造的に解決 |
| 業務ルール宣言化 | 稟議承認・経費判定・権限ポリシーを情シスがコード化、業務担当は要件を伝えるだけ | ZEN Engine + Phase 4 の JDM Editor で業務担当が画面編集できる将来へ進める |
| 横断ガード (規律のツール化) | 各 tier2/3 チームが Dapr の使い方をバラバラに実装、属人化が再発 | Opinionated API + 雛形強制 + CI ガードで「規律はドキュメントではなくツールで強制」 |
| 監査ログの統一 | 各サービスが独自ログ実装、ISMS / SOC 2 監査時に統合できない | `k1s0.Audit` で全サービスのログ形式・改ざん防止 (ハッシュチェーン) を統一 |
| クライアントライブラリ | tier3 が C# / TypeScript / Java で Dapr SDK 直接呼び、各言語にボイラープレートが量産される | 4 言語クライアント SDK で「言語を尊重しつつ統一感を保つ」レイヤ設計を維持 |

### 4.1 Build-lite が合理的となる条件

以下に該当する場合、Build-lite が合理的選択となる。本企画では合致しないと判断するが、決裁の場で同等の検討を求められた場合の比較軸として本節を引用する。

- 業務担当向けアプリ配信ポータルを必要としない (例: 全員開発者の組織)
- 業務ルールはコードで書ききる方針で、業務担当の画面編集を将来も求めない
- Dapr バージョン追従コストを各 tier2/3 チームが個別に引き受けられる組織体力がある
- 監査要件が緩く、各サービス独自のログ形式でも対応可能

### 4.2 k1s0 が必須となる条件

逆に、以下に該当する場合は Build-lite では JTC 情シスの構造課題を解決できず、k1s0 が必須となる。

- PC リプレース時の情シス工数 (端末訪問・設定再現) を削減することが稟議の主要約束に含まれる
- 業務担当が稟議ルールを直接編集できる将来 ([`../03_ロードマップと体制/00_フェーズ計画.md`](../03_ロードマップと体制/00_フェーズ計画.md) Phase 4) を成立させたい
- ISMS / SOC 2 / FISC 等の監査でログ統一が必須要件として課される
- 複数言語 (C# / Go / TypeScript / Java) を尊重しつつ、横断的関心事をプラットフォーム側で統一したい

## 5. リスクの構造的な違い

Build と Buy のリスクは **質が異なる** ことを理解することが、本章の最終目的である。以下の 3 軸で対比する。

| 観点 | Build (k1s0) | Buy (商用) |
|---|---|---|
| リスクの源 | 「個人開発起点である」という組織的事情 | 「契約構造・ベンダー支配力」という構造的事情 |
| 解消可能性 | 段階的に会社として引き取る体制を整えれば **低減できる** | 契約締結後の解消は **困難** (撤退コストが大きい) |
| 時間軸 | 初期ほどリスクが高く、運用段階で下がる | 初期は安全に見え、契約更新ごとにリスクが顕在化する |

Build のリスクは時間とともに **減衰する構造** を持つのに対し、Buy のリスクは時間とともに **顕在化が蓄積する構造** を持つ。両者の TCO 差が 5 年スパンで広がる理由も、この時間軸上の非対称性から説明される。

---

## 6. 企画書での論旨構成

決裁者向けの答弁は以下の順で組み立てる。機能論・差別化論・Build-lite 論・金額論・リスク論の順で積み上げ、最後に時間軸を導入する。

1. **機能重複の事実を認める** — 既存製品で同等の機能は実現できる
2. **商用製品では JTC 固有要件が埋まらない** — レガシー共存 / オンプレ / 稟議文化 / 言語自由 ([`02_差別化ポイント.md`](./02_差別化ポイント.md))
3. **真の対抗馬は Build-lite であることを認める** — 商用比較で勝つだけでは不十分。Build-lite に対する k1s0 の上乗せ投資 (中規模 5 年 0.72 億) を JTC 固有要件 6 点で正当化する（本章第 4 節）
4. **5 年 TCO で見ると k1s0 が商用に対し優位 / Build-lite に対し JTC 固有要件分を上乗せ** — 初期は高いが運用段階で下がる / ロックイン解消費が不要 ([`../04_定量試算/01_TCO5年試算.md`](../04_定量試算/01_TCO5年試算.md))
5. **Build のリスクは対処可能** — ドキュメント・MVP スコープ・OSS 積み上げ方針で個人開発リスクを段階的に解消する（本章第 2 節）
6. **Buy のリスクは構造的で対処困難** — 契約更新ごとに顕在化し、撤退コストが膨大（本章第 3 節・第 5 節）

この 6 段階のうち、4 段目（金額優位）は **定量試算章を単一ソースとして参照** し、本章では数値の重複掲載を避ける。数値が揺れた場合の唯一の修正先を定量試算章に固定することで、情報ドリフトを構造的に防ぐ。
