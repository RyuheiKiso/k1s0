# Build vs Buy のリスク構造

## 目的

商用 IDP / k8s ディストリビューションを採用する（Buy）か、k1s0 のような OSS プラットフォームを採用する（Build）かは、企業がアプリケーション基盤を選ぶ際の核心論点である。本資料は **リスクの質的構造** に絞って Build / Buy を対比し、「k1s0 を採用するとは何を引き受け、何を回避することか」を整理する。

加えて、Build 側にも k1s0 とは別の構成（素 K8s + Backstage + Dapr SDK 直叩きで tier1 facade を持たない最小内製案、以下 Build-lite）が成立することを認識する必要がある。商用製品との比較だけで結論すると「商用に勝つだけなら Build-lite で十分」という反論が成立するため、本資料では **k1s0 vs Build-lite vs Buy** の三項対立として構造化する。

採用側がこの章を読んで得るべき結論は、「Build (k1s0) と Build-lite と Buy はそれぞれ質の異なるリスクを引き受ける選択であり、量ではなく質で判断する必要がある」という理解である。

---

## 1. Build (k1s0 を採用) の特性

k1s0 は個人開発の OSS であるため、採用側の引き受けるリスクと利点は商用製品とは質的に異なる。

| 特性 | 内容 |
|---|---|
| ライセンス費 | **ゼロ**（Apache 2.0） |
| サポート | k1s0 として公式 SLA は提供しない。GitHub Issues / Discussion で起案者および外部コミュニティが対応 |
| バス係数 | 1（起案者）。採用側はフォーク・内製化する余地を確保する責任を持つ |
| ベンダーロックイン | なし。コードベースが OSS のため、採用側で完全制御可能 |
| カスタマイズ自由度 | 高（フォーク・改変・派生物作成が自由） |
| レガシー共存 | 第一級で扱う設計（.NET Framework サイドカー / API Gateway 共存） |
| 言語の自由度 | tier2 / tier3 は C# / Go / TypeScript / Java など自由 |

採用側が引き受けるべきリスクは「個人開発の OSS であることに由来する持続可能性の不確実さ」である。k1s0 は dogfooding（起案者自身の継続利用）と GitHub 公開（外部コミュニティの参画余地）で持続性を確保する設計を採っているが、商用 SLA のような契約的な保証は提供しない。

---

## 2. Buy (商用製品を購入) のリスク

Buy 側のリスクは「契約構造・ベンダー支配力」という構造的事情に由来するものが多く、契約締結後の解消は困難である。k1s0 を採用することで、以下リスクの大半は発生自体を回避できる。

| リスク | 発生確率 | 影響度 | k1s0 採用での扱い |
|---|---|---|---|
| ライセンス費が予算を圧迫 | 高 | 高 | 回避（Apache 2.0） |
| ベンダーロックイン | 高 | 極高 | 回避（OSS） |
| ベンダー都合の仕様変更 / 値上げ（VMware Broadcom 事例） | 中 | 極高 | 回避 |
| 自社要件（.NET Framework 共存等）が製品側でサポートされない | 高 | 高 | 自由に対応可能 |
| 日本法人のサポート品質が担保されない | 中 | 中 | 自社運用で品質自己決定 |

ベンダーロックインを「高・極高」と位置付ける根拠は、2023〜2024 の VMware Broadcom 買収事例・Red Hat Free 版打ち切り事例で、**契約後の解消コストが初期ライセンス費を上回る** 実例が複数観測されたためである。HashiCorp / Terraform の MPL → BSL 1.1 ライセンス変更（2023/8）も、商用ベンダー支配力に対するリスク事例として挙げられる。

## 3. Build (k1s0) vs Build-lite の質的差

商用 IDP との比較ばかりに焦点を当てると、「商用に勝つだけなら tier1 facade を持たない Build-lite（素 K8s + Backstage + Dapr SDK 直叩き）で十分では?」という反論が成立する。本節ではこの反論への構造的回答を提示する。

Build-lite は機能的には素直な「素 K8s + Backstage」案だが、k1s0 が解決する 6 つの構造課題を引き受けない。下記 6 点は [`../01_背景と目的/00_背景と課題.md`](../01_背景と目的/00_背景と課題.md) で論じた現場の構造課題に対する k1s0 の核心的回答であり、Build-lite を選ぶことは「これらの解決を諦める」ことを意味する。

| 項目 | Build-lite が引き受けるリスク | k1s0 の構造的回答 |
|---|---|---|
| Dapr バージョン追従 | tier2/3 の全サービスを一斉に書き換える必要（Dapr v1.13 Workflow API 改訂のような破壊的変更が直撃） | tier1 facade で変更を tier1 内に閉じ込め、tier2/3 のコードを変えない |
| アプリ配信ポータル | Backstage Catalog の URL リンク列挙のみ。端末台帳・部門フィルタ・設定コピー・レビュー・お知らせ機能なし | 業務固有要件（PC リプレース時の情シス工数圧縮 / エンドユーザー導線）を構造的に解決 |
| 業務ルール宣言化 | 申請承認・経費判定・権限ポリシーをコード化、業務担当は要件を伝えるだけ | ZEN Engine + JDM Editor で業務担当が画面編集できる設計 |
| 横断ガード（規律のツール化） | 各 tier2/3 チームが Dapr の使い方をバラバラに実装、属人化が再発 | Opinionated API + 雛形強制 + CI ガードで「規律はドキュメントではなくツールで強制」 |
| 監査ログの統一 | 各サービスが独自ログ実装、ISMS / SOC 2 監査時に統合できない | `k1s0.Audit` で全サービスのログ形式・改ざん防止（ハッシュチェーン）を統一 |
| クライアントライブラリ | tier3 が C# / TypeScript / Java で Dapr SDK 直接呼び、各言語にボイラープレートが量産される | 4 言語クライアント SDK で「言語を尊重しつつ統一感を保つ」レイヤ設計 |

### 3.1 Build-lite が合理的となる条件

以下に該当する場合、Build-lite が合理的選択となる。

- 業務担当向けアプリ配信ポータルを必要としない（例: 全員開発者の組織）
- 業務ルールはコードで書ききる方針で、業務担当の画面編集を将来も求めない
- Dapr バージョン追従コストを各 tier2/3 チームが個別に引き受けられる組織体力がある
- 監査要件が緩く、各サービス独自のログ形式でも対応可能

### 3.2 k1s0 が必須となる条件

逆に、以下に該当する場合は Build-lite では現場の構造課題を解決できず、k1s0 が必須となる。

- PC リプレース時の情シス工数（端末訪問・設定再現）を削減したい
- 業務担当が申請ルールを直接編集できる将来を成立させたい
- ISMS / SOC 2 / FISC 等の監査でログ統一が必須要件として課される
- 複数言語（C# / Go / TypeScript / Java）を尊重しつつ、横断的関心事をプラットフォーム側で統一したい

## 4. リスクの構造的な違い

Build と Buy のリスクは **質が異なる**。以下の 3 軸で対比する。

| 観点 | Build (k1s0) | Buy (商用) |
|---|---|---|
| リスクの源 | 「個人開発 OSS の持続可能性」という運用面の不確実さ | 「契約構造・ベンダー支配力」という構造的事情 |
| 解消可能性 | フォーク・内製化・コミュニティ参画で **採用側が能動的に低減できる** | 契約締結後の解消は **困難**（撤退コストが大きい） |
| 時間軸 | OSS の実績・コミュニティ規模が育つほどリスクが下がる | 初期は安全に見え、契約更新ごとにリスクが顕在化する |

Build のリスクは時間とともに **減衰する構造** を持つ（OSS の利用実績・外部コントリビュータ・dogfooding が積み上がる）のに対し、Buy のリスクは時間とともに **顕在化が蓄積する構造** を持つ（ライセンス値上げ・仕様変更・撤退コストの肥大）。

---

## 5. 採用判断の論旨構成

採用判断は以下の順で組み立てるのが構造的に堅い。

1. **機能重複の事実を認める** — 既存の商用製品で同等の機能はおおむね実現できる
2. **商用製品では現場固有要件が埋まらない** — レガシー共存 / オンプレ / 言語自由（[`02_差別化ポイント.md`](./02_差別化ポイント.md)）
3. **真の対抗馬は Build-lite であることを認める** — 商用比較で勝つだけでは不十分。Build-lite に対する k1s0 の上乗せ価値を本章第 3 節の 6 点で評価する
4. **Build のリスクは採用側が能動的に低減できる** — フォーク・コミュニティ参画・内製化の余地を確保する（本章第 1 節）
5. **Buy のリスクは構造的で対処困難** — 契約更新ごとに顕在化し、撤退コストが膨大（本章第 2 節・第 4 節）
