[`00_競合マップ.md`](./00_競合マップ.md) で挙げた主要な競合・参考製品について、1 製品ずつ評価を整理する。k1s0 との関係を明確にすることで、企画書での答弁根拠として利用する。

## 6. Dapr (Distributed Application Runtime)

> 調査日: 2026-04-12 時点。Dapr 公式ドキュメント (v1.17 系) を確認。

### 基本情報

| 項目 | 内容 |
|---|---|
| 種別 | Distributed Application Runtime (マイクロサービス building blocks) |
| 提供元 | Microsoft が公開、現在は CNCF **Graduated** プロジェクト (2024 年 11 月卒業) |
| 現行バージョン | v1.17 系 |
| ライセンス | Apache 2.0 |
| デプロイ形態 | サイドカー方式が標準。加えて Dapr Shared (DaemonSet / Deployment 方式) も提供 |
| 制御プレーン | `dapr-operator` / `dapr-sidecar-injector` / `dapr-sentry` (mTLS 用 CA, SPIFFE 準拠) |

### Building Blocks と tier1 との対応

Dapr は以下の building blocks を API として提供する。k1s0 の tier1 設計と機能的に大きく重複する。

| Building Block | 内容 | tier1 での位置付け |
|---|---|---|
| Service invocation | gRPC/HTTP ベースのサービス間通信 (mTLS, リトライ, 名前解決) | Istio との併用で重複小 |
| State management | 多様な state store (Redis / PostgreSQL 等) への統一 API | **tier1 公開 API `k1s0.State.*` の裏側で利用** |
| Pub/Sub | 多様なブローカー (Kafka / NATS / RabbitMQ 等) への統一 API | **tier1 公開 API `k1s0.PubSub.*` の裏側で利用** |
| Secrets | 多様な secret store への統一 API | **tier1 公開 API `k1s0.Secrets.*` の裏側で利用** |
| Bindings (input/output) | 外部システム連携の統一抽象 | 必要に応じて tier1 で対応 |
| Actors | 仮想アクターモデル | k1s0 では想定外 |
| Workflow | コードベースの durable workflow エンジン | **tier1 公開 API `k1s0.Workflow.*` の裏側で利用** |
| Observability | 分散トレース / メトリクス / ログの統合 | OTel 採用で重複小 |
| Configuration | 設定の動的更新 API | **tier1 公開 API `k1s0.Settings.*` の裏側で利用** |
| Distributed lock | 分散ロック API | 必要に応じて tier1 で対応 |

### 調査で判明した重要事実

**(1) Rust SDK は Alpha (未成熟)**

> 公式ドキュメント v1.17 より引用: "The Dapr Rust-SDK is currently in Alpha. Work is underway to bring it to a stable release and will likely involve breaking changes."

- 現行バージョンは `dapr = "0.16"` (Cargo.toml)
- 破壊的変更を今後含むことが明言されている
- **Rust SDK に依存する実装は採用できない**。これが tier1 内部を「Daprファサードは Go、自作領域は Rust」のハイブリッドにする一次根拠となっている
- C# / Go / Python / Java / JavaScript SDK は安定しているため、tier2 / tier3 が Dapr を利用するのは技術的には問題ないが、**k1s0 では隠蔽方針により tier2 / tier3 が Dapr に触れることを禁止する**

**(2) Dapr Shared でサイドカー方式のオーバーヘッド軽減が可能**

Dapr は標準ではサイドカー方式だが、**Dapr Shared** という拡張により DaemonSet (ノードあたり 1 つ) / Deployment (クラスタあたり 1 つ) 方式も選べる。「Dapr はサイドカーで重い」という従来の批判は部分的に解消される。

**(3) Dapr Workflow は durable execution 対応**

Dapr Workflow は code-first workflow エンジンで、長時間実行 (数日〜数週間) / 耐障害性 (クラッシュ後の再開) / 外部イベント待ち / タイマー / 承認フロー / Saga パターン (補償トランザクション) をネイティブにサポート。**tier1 の Saga オーケストレータを自前実装するのと機能的に同等**のため、自前実装は不要。

### k1s0 との関係: 採用して隠蔽 (2026-04-12 決定)

**Dapr を tier1 内部実装として組み込み、tier2 / tier3 からは完全に隠蔽する**。

| 隠蔽対象 | tier2 / tier3 で禁止される表れ | 吸収する場所 |
|---|---|---|
| Dapr SDK の import | `using Dapr.Client;` 等が現れない | tier1 公開ライブラリ内に閉じ込める |
| Pod annotation | `dapr.io/*` annotation を書かない | 雛形生成 CLI が自動付与 |
| Component YAML | `apiVersion: dapr.io/v1alpha1` を扱わない | tier1 チームが集中管理 |
| Dapr のエラーメッセージ | アプリログに Dapr 用語が現れない | tier1 ライブラリで k1s0 統一エラーに変換 |

隠蔽の詳細設計と 5 層防御は [`../../02_構想設計/02_tier1設計/01_設計の核/01_Dapr隠蔽方針.md`](../../02_構想設計/02_tier1設計/01_設計の核/01_Dapr隠蔽方針.md) / [`../../02_構想設計/02_tier1設計/02_API契約/03_API設計原則.md`](../../02_構想設計/02_tier1設計/02_API契約/03_API設計原則.md) を参照。

## 12. 商用競合の定量比較（稟議答弁用）

本章までの各製品評価は「k1s0 との関係」と「機能構成」に軸を置いていたが、稟議の場では **「年額・サポート SLA・撤退コスト」** の 3 軸で数値が求められる。この 3 軸について、商用競合 4 製品 (OpenShift / Tanzu / Humanitec / Mia-Platform) を直接比較する。各セルは [`../04_定量試算/01_TCO5年試算.md`](../04_定量試算/01_TCO5年試算.md) の前提数値と各ベンダー公開情報・業界ヒアリングに基づく。

### 12.1 中規模 (従業員 3,000 名) の年額ライセンス費

中規模想定 (80 コア / 内製エンジニア 10 名) のライセンス費を、ベンダー公開価格を基に定量化する。為替 150 円/USD 固定。

| 製品 | 課金モデル | 単価 | 年額 (中規模) | 出典 |
|---|---|---|---|---|
| **OpenShift Container Platform + Application Foundations** | コア数 | 2,000 USD/コア/年 (OCP Standard 1,500 + App Foundations 500) | **2,400 万円** | Red Hat 公開価格 2025 + 代理店見積もり |
| **VMware Tanzu Application Platform** | CPU 数 | 800 USD/CPU/年 | **960 万円** | Broadcom 2025 国内取引例 (変動可能性あり) |
| **Humanitec** | ユーザー数 | 基本 60,000 USD/年 (10 ユーザー) | **900 万円** | Humanitec 公開価格 2025 |
| **Mia-Platform** | モジュール + ノード | 要見積もり (中規模で 80,000〜150,000 EUR/年 程度と推定) | **1,440〜2,700 万円** | Mia-Platform 欧州金融機関事例からの推定 |

OpenShift が最も高額で、Tanzu / Humanitec が中帯、Mia-Platform は契約内容で大きく振れる。k1s0 はライセンス費 **0 円**。5 年累計では OpenShift 1.20 億、Tanzu 4,800 万、Humanitec 4,500 万に対し k1s0 ゼロとなる。

### 12.2 サポート SLA の水準比較

SLA は「応答時間 (Response Time)」と「復旧時間 (Resolution Time)」で評価する。応答時間はベンダーが第一報を返すまでの時間、復旧時間はワークアラウンド提供までの時間。

| 製品 | プラン | 重大度 1 応答 | 重大度 1 復旧 | 稼働率保証 | 出典 |
|---|---|---|---|---|---|
| **OpenShift Container Platform** | Premium | 1 時間 (24/7) | 4 時間 (Resolution 無保証、Best effort) | クラウド版 99.95%、自己運用は適用外 | Red Hat SLA 公開資料 |
| **OpenShift Container Platform** | Standard | 2 時間 (営業時間) | 同上 | 同上 | 同上 |
| **VMware Tanzu Application Platform** | Production | 1 時間 (24/7) | 4 時間 | クラウド版のみ稼働率保証、自己運用対象外 | Broadcom 2024 サポート条件 |
| **Humanitec** | Enterprise | 4 時間 (営業時間) | 24 時間 | SaaS 版 99.9% | Humanitec サポート公開条件 |
| **Mia-Platform** | Enterprise | 2 時間 (24/7) | 要個別契約 | 要個別契約 | Mia-Platform 取引事例 |
| **k1s0 (自社運用)** | ─ | 社内対応 (オンコール 30 分以内目標) | 4 時間 (クリティカル)、Runbook 整備後は大半を自社解決 | 業務時間帯 99.5% 目標 (MVP-1 時点) | [`../../02_構想設計/01_アーキテクチャ/02_可用性と信頼性/05_SLOとエラーバジェット.md`](../../02_構想設計/01_アーキテクチャ/02_可用性と信頼性/05_SLOとエラーバジェット.md) |

数字を読むと、商用製品の Premium プランと k1s0 の自社対応の応答時間は拮抗している。商用の優位は「24/7 対応の体制維持コストがベンダー側に寄る」点のみで、応答時間そのものは自社オンコール体制で同水準が達成できる。一方で **稼働率保証** は商用のクラウド版 (SaaS) のみ適用され、JTC 情シスが前提とする **オンプレ / 自己運用では商用製品も稼働率保証の対象外** となる。「商用の SLA が安心」という直感は、オンプレ前提の JTC では実は成立しない。

### 12.3 撤退コストの定量比較

撤退コストは「契約解除時に発生する一括費用」と「代替製品への移行工数」に分解できる。5 年契約で 3 年目に撤退する想定で比較する。

| 項目 | k1s0 | OpenShift | Tanzu | Humanitec | Mia-Platform |
|---|---|---|---|---|---|
| 契約違約金 | 0 | 残契約期間の 20〜50% (Red Hat 規約) | 残期間のほぼ全額 (Broadcom 2024 以降の厳格化) | 残契約期間全額 (Enterprise SaaS 標準) | 残契約期間の 50〜100% |
| 独自 CRD / 独自仕様からの脱却 | 0 (独自 CRD なし) | OpenShift Routes / OpenShift Service Mesh を標準 k8s に書き換え 3〜6 人月 | TAP GUI / Supply Chain を置換 3〜6 人月 | Score 仕様の書き換え 1〜3 人月 | Mia 独自 UI / マーケットプレイスを代替ツールに移行 5〜10 人月 |
| データエクスポート | 0 (標準 OSS のみ使用) | etcd / イメージレジストリは標準 OSS 形式で抽出可 | 同上 | クラウド API 経由でのエクスポートが必要 (形式は独自 JSON) | 独自 DB / マーケットプレイス DB の再構築必須 |
| **撤退コスト合計 (中規模 3,000 名, 3 年目撤退)** | **0 円** | 約 **3,600〜4,800 万円** (違約金 2,400 万 + 移行工数 1,200〜2,400 万) | 約 **2,000〜3,400 万円** | 約 **1,500〜2,300 万円** | 約 **2,500〜4,700 万円** |

k1s0 は撤退コストがゼロとなる最大の理由は「採用 OSS が標準 k8s の範疇で動作し、独自 CRD を使わない」点にある。OpenShift / Tanzu / Mia-Platform はそれぞれ独自 CRD / 独自 UI / 独自マーケットプレイスを導入するため、撤退時にそれらを剥がす工数が発生する。これは単なる工数ではなく、**「既に稼働している業務アプリが動かなくなる」リスクを伴う移行** であり、多くの JTC が「事実上撤退できない」状態に陥るのはこのためである。

本節の数値は 2026-04 時点のベンダー契約慣行に基づく推定で、実契約では数値が変動する。ただし **「k1s0 は撤退コストゼロ、商用は 1,500〜4,800 万円」** という桁違いの構造は、契約内容の詳細によらず成立する。

### 12.4 3 軸統合の稟議答弁用サマリ

年額・SLA・撤退コストを統合し、中規模 (3,000 名) で 5 年運用した場合の数値を並列表示する。

| 項目 | k1s0 | OpenShift | Tanzu | Humanitec | Mia-Platform |
|---|---|---|---|---|---|
| 年額ライセンス | 0 円 | 2,400 万 | 960 万 | 900 万 | 1,440〜2,700 万 |
| 5 年累計ライセンス | 0 円 | 1.20 億 | 4,800 万 | 4,500 万 | 0.72〜1.35 億 |
| Severity 1 応答 | 社内オンコール 30 分 | Premium で 1 時間 | 1 時間 | 4 時間 (営業時間) | 2 時間 |
| オンプレ稼働率保証 | なし (目標値のみ) | **なし** (SLA はクラウド版限定) | **なし** (同上) | — (SaaS 中心) | 個別契約 |
| 3 年目撤退時コスト | 0 | 3,600〜4,800 万 | 2,000〜3,400 万 | 1,500〜2,300 万 | 2,500〜4,700 万 |
| **5 年 Total (本書の範囲のみ)** | **0 円** | **1.56〜1.68 億** | **6,800〜8,200 万** | **6,000〜6,800 万** | **0.97〜1.82 億** |

この表は **ライセンス費と撤退コストのみ** を扱い、運用人件費・インフラ費・内製追加費は含まない (それらは [`../04_定量試算/01_TCO5年試算.md`](../04_定量試算/01_TCO5年試算.md) で別途加算)。稟議の場で「商用を買った方が安いのでは」と問われた際、本表 1 行で「商用は最安でも 6,000 万円の追加支出、最大で 1.68 億円」と即答できる。

### 12.5 決裁者答弁の雛形

> 中規模 (3,000 名) で 5 年運用した場合、商用 IDP のライセンス費と撤退コストだけで最低 6,000 万円、最大で 1.68 億円が必要となります。OpenShift は稟議で最も挙がる選択肢ですが、5 年ライセンスが 1.20 億円、3 年目撤退時の違約金と移行工数が 3,600〜4,800 万円、合計 1.56〜1.68 億円です。
>
> 一方 k1s0 はライセンスゼロ、撤退コストゼロで、採用 OSS はすべて標準 k8s の範疇に収まっており、別の OSS ディストリビューションへ切り替える自由が構造的に担保されています。
>
> サポート応答時間は商用 Premium (1 時間) と k1s0 のオンコール体制 (30 分以内目標) で拮抗します。稼働率保証は商用でもオンプレ運用では適用外のため、自己運用という点では条件は同等です。
