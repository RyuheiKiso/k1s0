# 主要製品別評価

## 目的

[`00_競合マップ.md`](./00_競合マップ.md) で挙げた主要な競合・参考製品について、1 製品ずつ評価を整理する。k1s0 との関係を明確にすることで、採用検討時の根拠として利用する。

各製品情報は一般に公開されている情報に基づく概観であり、紹介資料に引用する際は公式ドキュメント最新版で再確認すること。

## 6. Dapr (Distributed Application Runtime)

> 調査日: 2026-04-12 時点。Dapr 公式ドキュメント (v1.17 系) を確認。

### 基本情報

| 項目 | 内容 |
|---|---|
| 種別 | Distributed Application Runtime (マイクロサービス building blocks) |
| 提供元 | Microsoft が公開、現在は CNCF **Graduated** プロジェクト (2024 年 11 月卒業) |
| 現行バージョン | v1.17 系 |
| ライセンス | Apache 2.0 |
| デプロイ形態 | サイドカー方式が標準。加えて Dapr Shared (DaemonSet / Deployment 方式) も提供 |
| 制御プレーン | `dapr-operator` / `dapr-sidecar-injector` / `dapr-sentry` (mTLS 用 CA, SPIFFE 準拠) |

### Building Blocks と tier1 との対応

Dapr は以下の building blocks を API として提供する。k1s0 の tier1 設計と機能的に大きく重複する。

| Building Block | 内容 | tier1 での位置付け |
|---|---|---|
| Service invocation | gRPC/HTTP ベースのサービス間通信 (mTLS, リトライ, 名前解決) | Istio との併用で重複小 |
| State management | 多様な state store (Redis / PostgreSQL 等) への統一 API | **tier1 公開 API `k1s0.State.*` の裏側で利用** |
| Pub/Sub | 多様なブローカー (Kafka / NATS / RabbitMQ 等) への統一 API | **tier1 公開 API `k1s0.PubSub.*` の裏側で利用** |
| Secrets | 多様な secret store への統一 API | **tier1 公開 API `k1s0.Secrets.*` の裏側で利用** |
| Bindings (input/output) | 外部システム連携の統一抽象 | 必要に応じて tier1 で対応 |
| Actors | 仮想アクターモデル | k1s0 では想定外 |
| Workflow | コードベースの durable workflow エンジン | **tier1 公開 API `k1s0.Workflow.*` の裏側で利用** |
| Observability | 分散トレース / メトリクス / ログの統合 | OTel 採用で重複小 |
| Configuration | 設定の動的更新 API | **tier1 公開 API `k1s0.Settings.*` の裏側で利用** |
| Distributed lock | 分散ロック API | 必要に応じて tier1 で対応 |

### 調査で判明した重要事実

**(1) Rust SDK は Alpha (未成熟)**

> 公式ドキュメント v1.17 より引用: "The Dapr Rust-SDK is currently in Alpha. Work is underway to bring it to a stable release and will likely involve breaking changes."

- 現行バージョンは `dapr = "0.16"` (Cargo.toml)
- 破壊的変更を今後含むことが明言されている
- **Rust SDK に依存する実装は採用できない**。これが tier1 内部を「Daprファサードは Go、自作領域は Rust」のハイブリッドにする一次根拠となっている
- C# / Go / Python / Java / JavaScript SDK は安定しているため、tier2 / tier3 が Dapr を利用するのは技術的には問題ないが、**k1s0 では隠蔽方針により tier2 / tier3 が Dapr に触れることを禁止する**

**(2) Dapr Shared でサイドカー方式のオーバーヘッド軽減が可能**

Dapr は標準ではサイドカー方式だが、**Dapr Shared** という拡張により DaemonSet (ノードあたり 1 つ) / Deployment (クラスタあたり 1 つ) 方式も選べる。「Dapr はサイドカーで重い」という従来の批判は部分的に解消される。

**(3) Dapr Workflow は durable execution 対応**

Dapr Workflow は code-first workflow エンジンで、長時間実行 (数日〜数週間) / 耐障害性 (クラッシュ後の再開) / 外部イベント待ち / タイマー / 承認フロー / Saga パターン (補償トランザクション) をネイティブにサポート。**tier1 の Saga オーケストレータを自前実装するのと機能的に同等**のため、自前実装は不要。

### k1s0 との関係: 採用して隠蔽 (2026-04-12 決定)

**Dapr を tier1 内部実装として組み込み、tier2 / tier3 からは完全に隠蔽する**。

| 隠蔽対象 | tier2 / tier3 で禁止される表れ | 吸収する場所 |
|---|---|---|
| Dapr SDK の import | `using Dapr.Client;` 等が現れない | tier1 公開ライブラリ内に閉じ込める |
| Pod annotation | `dapr.io/*` annotation を書かない | 雛形生成 CLI が自動付与 |
| Component YAML | `apiVersion: dapr.io/v1alpha1` を扱わない | tier1 チームが集中管理 |
| Dapr のエラーメッセージ | アプリログに Dapr 用語が現れない | tier1 ライブラリで k1s0 統一エラーに変換 |

隠蔽の詳細設計と 5 層防御は [`../../02_構想設計/02_tier1設計/01_設計の核/01_Dapr隠蔽方針.md`](../../02_構想設計/02_tier1設計/01_設計の核/01_Dapr隠蔽方針.md) / [`../../02_構想設計/02_tier1設計/02_API契約/03_API設計原則.md`](../../02_構想設計/02_tier1設計/02_API契約/03_API設計原則.md) を参照。

## 12. 商用競合との構造的な対比

商用 4 製品 (OpenShift / Tanzu / Humanitec / Mia-Platform) と k1s0 の差は、ライセンス費・サポート形態・撤退時の障壁という 3 つの構造的軸で読める。具体的なベンダー価格表・訴訟記録・本番採用実績は [`04_外部根拠_商用価格とロックイン.md`](./04_外部根拠_商用価格とロックイン.md) に一次情報を集約しているため、本節では商用と k1s0 のリスク構造の差異だけを整理する。

### 12.1 ライセンス構造の差

商用 4 製品は **コア数 / CPU 数 / ユーザー数 / モジュール数** いずれかを単位にサブスクリプション課金する。スケールに比例してコストが線形に増加し、為替変動と価格改定が予算リスクとして毎年発生する。Broadcom / Red Hat の値上げ事例は [`04_外部根拠_商用価格とロックイン.md`](./04_外部根拠_商用価格とロックイン.md) 第 1〜2 節で訴訟記録ベースに裏付けている。

k1s0 はそもそも **個人開発の OSS を Apache 2.0 で公開** している。採用側はライセンス費を支払わず、自社で運用する。スケールしてもライセンス費が増加する経路がない。

### 12.2 サポート形態の差

商用は Premium プランで Severity 1 応答 1 時間 (24/7) を提供するが、**稼働率保証はクラウド版 (SaaS) のみに適用** され、採用側組織の情シスが前提とするオンプレ / 自己運用では商用も保証対象外となる。「商用の SLA が安心」という直感は、オンプレ前提では成立しない。

k1s0 は個人開発 OSS のためベンダーサポートは存在しない。採用側は自社オンコール体制で対応する。応答時間そのものは社内オンコール 30 分以内を目標値として設計可能で、オンプレ前提では商用 Premium と構造的に拮抗する ([`../../02_構想設計/01_アーキテクチャ/02_可用性と信頼性/05_SLOとエラーバジェット.md`](../../02_構想設計/01_アーキテクチャ/02_可用性と信頼性/05_SLOとエラーバジェット.md) 参照)。

### 12.3 撤退時の障壁

商用 4 製品はそれぞれ独自 CRD / 独自 UI / 独自マーケットプレイス / 独自仕様を導入する。撤退時に「既に稼働している業務アプリが動かなくなる」リスクを伴うため、契約上の違約金以前に **技術的に撤退できない構造** が形成される。Broadcom 永久ライセンス廃止・Red Hat ソース制限・AT&T 1,050% 値上げ訴訟など、撤退困難性が顕在化した事例は [`04_外部根拠_商用価格とロックイン.md`](./04_外部根拠_商用価格とロックイン.md) を参照。

k1s0 は **採用 OSS が標準 k8s の範疇で動作し、独自 CRD を導入しない** 設計選択を貫く。Apache 2.0 ライセンスは取消不能 (irrevocable) なため、起案者が開発を止めても採用側の利用権は失われない。Build vs Buy のリスク構造比較は [`03_BuildVsBuy.md`](./03_BuildVsBuy.md) で詳述する。
