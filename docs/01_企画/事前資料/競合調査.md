# 競合調査

## 目的
k1s0 が対象とする「開発者プラットフォーム / マイクロサービス基盤」領域の既存製品・事例を整理し、k1s0 の立ち位置 (差別化ポイント) を明確にする。

## 注意事項
本資料の各製品情報は一般に公開されている情報に基づく概観である。
企画書に数値や機能詳細を引用する場合は、公式ドキュメント最新版での再確認を必須とする。

| 分類 | 代表例 | k1s0 との関係 |
|---|---|---|

| 商用 IDP / PaaS | Humanitec / Port / Cortex | 機能重複 (有償) |

| OSS k8s 管理プラットフォーム | Rancher | 採用候補 |

## 6. Dapr (Distributed Application Runtime)

> **調査日**: 2026-04-12 時点。Dapr 公式ドキュメント (v1.17 系) を context7 経由で確認。

### 基本情報

| 項目 | 内容 |
|---|---|
| 種別 | Distributed Application Runtime (マイクロサービス building blocks) |
| 提供元 | Microsoft が公開、現在は CNCF **Graduated** プロジェクト (2024 年 11 月卒業) |
| 現行バージョン | v1.17 系 |
| ライセンス | Apache 2.0 |
| デプロイ形態 | **サイドカー方式が標準**。加えて Dapr Shared (DaemonSet / Deployment 方式) も提供。 |
| 制御プレーン | `dapr-operator` / `dapr-sidecar-injector` / `dapr-sentry` (mTLS 用 CA, SPIFFE 準拠) |

### Building Blocks (提供される共通機能)

Dapr は以下の building blocks を API として提供する。k1s0 の tier1 設計と**機能的に大きく重複する**。

| Building Block | 内容 | tier1 での対応 |
|---|---|---|
| Service invocation | gRPC/HTTP ベースのサービス間通信 (mTLS, リトライ, 名前解決) | Istio が担当するため重複小 |
| **State management** | 多様な state store (Redis / PostgreSQL / Cosmos DB 等) への統一 API | tier1 で独自実装を想定 |
| **Pub/Sub** | 多様なブローカー (Kafka / NATS / RabbitMQ 等) への統一 API | **tier1 Kafka クライアントと直接重複** |
| Secrets | 多様な secret store (Vault / k8s secrets / AWS SM 等) への統一 API | tier1 で独自実装を想定 |
| Bindings (input/output) | 外部システム連携 (DB / SaaS / IoT 等) の統一抽象 | 必要に応じて tier1 で対応 |
| Actors | 仮想アクターモデル (単一実行保証・状態保持) | k1s0 では想定外 |
| **Workflow** | **コードベースの durable workflow エンジン**。長時間実行・耐障害性・外部イベント待ち・タイムアウト・Saga 風オーケストレーションに対応 | **tier1 の Saga オーケストレータと直接重複** |
| Observability | 分散トレース / メトリクス / ログの統合 | OTel 採用で重複小 |
| Configuration | 設定の動的更新 API | tier1 で独自実装を想定 |
| Distributed lock | 分散ロック API | 必要に応じて tier1 で対応 |

### 今回の調査で判明した重要事実

**(1) Rust SDK は Alpha (未成熟)**

> 引用: *"The Dapr Rust-SDK is currently in Alpha. Work is underway to bring it to a stable release and will likely involve breaking changes."* (公式ドキュメント v1.17)

- 現行バージョンは `dapr = "0.16"` (Cargo.toml)
- **破壊的変更を今後含む**ことが明言されている
- **k1s0 の tier1 が Rust 固定である以上、Dapr を tier1 で直接利用することは現時点では現実的ではない**
- C# / Go / Python / Java / JavaScript SDK は安定しているため、tier2 / tier3 が Dapr を利用するのは問題ない

**(2) Dapr Shared でサイドカー方式のオーバーヘッド軽減が可能**

Dapr は標準ではサイドカー方式だが、**Dapr Shared** という拡張により以下の方式も選べる:
- **DaemonSet** (ノードあたり 1 つの Dapr インスタンス)
- **Deployment** (クラスタあたり 1 つの Dapr インスタンス)

これにより Function / FaaS 系ワークロードや、サイドカーあたりのメモリコストを嫌うユースケースに対応できる。**「Dapr はサイドカーで重い」という従来の批判は、部分的に Dapr Shared で解消される**。

**(3) Dapr Workflow は durable execution 対応**

Dapr Workflow は code-first workflow エンジンで、以下をネイティブにサポート:
- 長時間実行 (数日〜数週間)
- 耐障害性 (クラッシュ後の再開)
- 外部イベント待ち (`waitForExternalEvent`)
- タイマー (`createTimer`)
- 承認フロー、タイムアウト付きワークフロー
- Saga パターン (補償トランザクション)

→ **tier1 の Saga オーケストレータを自前実装するのと、Dapr Workflow を採用するのは機能的に同等の選択肢**になる。

### k1s0 との関係: 採用方針 (2026-04-12 決定)

**採用方針**: **Dapr を tier1 内部実装として組み込み、tier2 / tier3 からは完全に隠蔽する**

#### 3 層構造

```
tier2 / tier3 (C# / Go / Java / TS / Python)
        ↓ 利用するのは tier1 公開 API のみ
tier1 公開 API (多言語クライアントライブラリ) ← k1s0 固有の統一 API
        ↓ 内部呼び出し (tier2 / tier3 からは不可視)
tier1 内部実装
  ・Dapr Control Plane + サイドカー + 各言語 安定 Dapr SDK
  ・Rust 実装サービス (JTC 固有機能 / Component 管理 / 雛形生成 CLI)
        ↓
infra 層 (k8s / Istio / Kafka / Loki 等)
```

#### 隠蔽の徹底

| 隠蔽対象 | tier2 / tier3 で禁止される表れ | 吸収する場所 |
|---|---|---|
| Dapr SDK の import | `using Dapr.Client;` 等が現れない | tier1 公開ライブラリ内に閉じ込める |
| Pod annotation | `dapr.io/*` annotation を書かない | 雛形生成 CLI が自動付与 |
| Component YAML | `apiVersion: dapr.io/v1alpha1` を扱わない | tier1 チームが集中管理 |
| Dapr のエラーメッセージ | アプリログに Dapr 用語が現れない | tier1 ライブラリで k1s0 統一エラーに変換 |

#### Rust SDK Alpha 問題の解消

- **tier1 Rust 実装は Dapr を直接呼ばない**。Rust 部分は JTC 固有機能 / Component 管理 / 雛形生成 CLI に限定する。
- **tier1 多言語クライアントライブラリ (C# / Go / Java / TS / Python)** は各言語の**安定 Dapr SDK** を内部で利用する。
- これにより Dapr Rust SDK の Alpha ステータスは k1s0 全体に影響しない。

#### 採用の利点

- **Dapr の機能 (state / pub-sub / workflow / secrets 等) を業界標準実装として享受**できる
- tier2 / tier3 のコードは「Dapr を知らない」まま長期保守できる
- **将来的な Dapr 差し替え・バージョン更新・互換性破壊の影響を tier1 内に閉じ込められる**
- Dapr Rust SDK が安定化したら、内部実装を Rust に置き換える選択肢も残る

### 企画書での答弁案

決裁者から「なぜ Dapr を採用 (もしくは隠蔽) するのか」と問われた際の回答構成:

1. **Dapr は採用する**。ただし tier1 の内部実装として組み込み、tier2 / tier3 からは隠蔽する。
2. **tier2 / tier3 開発者は Dapr を一切意識しない**。tier1 公開 API (多言語クライアントライブラリ) のみを使う。
3. これにより Dapr のバージョン更新・破壊的変更・将来的な差し替えの影響を **tier1 内に閉じ込められる**。
4. tier1 Rust 実装側は Dapr を直接呼ばない (Rust SDK が Alpha のため)。各言語の安定 SDK は tier1 公開ライブラリ内でのみ使用する。
5. 結果として「Dapr の機能的恩恵」と「Dapr に縛られない長期保守性」を両立できる。

## 12. 国内大手企業の自社開発プラットフォーム事例

k1s0 の方向性 (大企業が自社用マイクロサービス基盤を内製する) には国内でも先行事例がある。
ただし**本セクションの具体情報は一般論レベルに留めており、企画書に引用する際は各社の公開テックブログ・登壇資料から最新事例を個別に収集する必要がある**。

一般的な傾向として、以下の点が既に業界内で確認できる。

- Web / モバイル系の大手テック企業の多くが、k8s をベースにした自社マイクロサービスプラットフォームを内製している。
- 共通基盤チームを設置し、アプリチームが業務ロジックに集中できる体制を敷いている事例が多い。
- 公開資料の多くはクラウド (AWS / GCP) マネージド前提であり、オンプレ / VM 上での k8s 運用ノウハウは相対的に情報量が少ない。

**示唆**:
- 「大企業が自社用に k8s ベースの基盤を作る」は業界のトレンドとして既に成立しており、k1s0 のアプローチは奇抜ではない。
- ただし事例の多くは **Web / モバイル企業** が中心で、**JTC の情シス部門による内製事例は相対的に少ない**。
- k1s0 の独自性は「JTC 情シス部門でも運用できること」「.NET Framework 等のレガシー共存」「オンプレ / VM 前提」にある。

## k1s0 の差別化ポイント

| 軸 | 既存製品 / 事例 | k1s0 |
|---|---|---|
| コスト | 商用製品は有償サブスクリプション | OSS 積み上げで無償 |
| ベンダー依存 | OpenShift / Tanzu / Gloo / Mia-Platform はロックインあり | 回避を設計原則とする |
| 対象ユーザー層 | Web 系テック企業 / 大規模 SaaS 向けが多い | **JTC 情シス部門** |
| レガシー共存 | 考慮されないケースが多い | **.NET Framework 等の古い資産との共存を第一級で扱う** |
| 実行環境 | クラウドマネージド前提が多い | **オンプレ / VM を第一級で扱う** |
| 言語の自由度 | プラットフォーム指定の言語が中心 | **tier2 / tier3 は C# / Go / MAUI / React など自由。既存資産の言語を尊重** |
| Dapr の扱い | tier2 / tier3 が直接 Dapr SDK / annotation を扱う前提 | **tier1 内部実装として隠蔽。tier2 / tier3 は Dapr を一切意識しない** |
| レガシー所有者の動機付け | 移行を強制 / 無視 | **運用コスト削減・デプロイ速度などの金銭的メリットで「作り直してでも参加したくなる」設計** |
| 人材要件 | k8s / サービスメッシュの専門家前提 | 未経験チームでも始められる雛形・ドキュメント提供 |
| 基盤言語 | Go / TypeScript / Java 中心 | **Rust で長期保守性を最優先** |

## JTC 情シス部門固有の選定軸

k1s0 は一般的な開発者プラットフォームとは異なる「JTC 情シス部門」という特殊な市場を対象とする。
この市場に特有の選定軸は以下のとおり。

| 軸 | JTC 情シスでの重要度 | 既存製品の典型的な立ち位置 | k1s0 の立ち位置 |
|---|---|---|---|
| **稟議が通るか** | 極めて高 | 商用は数百万円〜で稟議ハードル高 | **無償 OSS なので稟議不要** |
| **情報システム子会社でも運用できるか** | 高 | 専門家前提で運用困難 | 雛形・ドキュメント・段階導入で対応 |
| **既存ベンダー契約との整合** | 高 | OpenShift 等は IBM / Red Hat 契約が前提 | 既存契約に影響しない |
| **セキュリティ監査・ISO への適合** | 高 | 個別製品ごとに監査資料を用意 | OSS 積み上げ構成で監査は tier 単位 |
| **日本語サポート・ドキュメント** | 中 | 商用は日本法人サポートあり | 自社内のノウハウ蓄積に依存 |
| **既存 .NET Framework 資産の扱い** | 極めて高 | 通常は考慮されない | **第一級で対応** |
| **オンプレ DC / VMware 環境での動作** | 高 | クラウドマネージド前提が多い | **前提条件として明示** |

## Build vs Buy リスク分析

k1s0 を内製する (Build) か、商用製品を購入する (Buy) かの選択は企画書の核心論点となる。

### Build (k1s0 を内製) のリスク

| リスク | 発生確率 | 影響度 | 対処方針 |
|---|---|---|---|
| 個人開発ゆえの属人化 | 高 | 高 | **ドキュメント充実 + コード公開** でバス係数を下げる |
| 開発が途中で頓挫する | 中 | 高 | MVP スコープを最小化して早期に価値提供 |
| 新技術追従の工数が読めない | 中 | 中 | OSS 積み上げなのでコミュニティ更新に追従するだけ |
| セキュリティ脆弱性の対応遅延 | 中 | 高 | 採用 OSS の CVE 監視と tier1 の修正体制を事前に設計 |
| 所属企業の評価 / 支援が得られない | 中 | 中 | 無償提供の時点で ROI を明示して合意を取る |

### Buy (商用製品を購入) のリスク

| リスク | 発生確率 | 影響度 | 対処方針 |
|---|---|---|---|
| ライセンス費が予算を圧迫 | 高 | 高 | k1s0 なら回避 |
| ベンダーロックイン | 高 | 極高 | k1s0 なら回避 |
| ベンダー都合の仕様変更 / 値上げ (例: VMware Broadcom 事例) | 中 | 極高 | k1s0 なら回避 |
| 自社要件 (.NET Framework 共存等) が製品側でサポートされない | 高 | 高 | k1s0 なら自由に対応 |
| 日本法人のサポート品質が担保されない | 中 | 中 | k1s0 は自社運用で品質自己決定 |

**示唆**: Build 側のリスクは「個人開発起点」に由来するものが主で、段階的に会社として引き取る体制を整えれば低減できる。Buy 側のリスクは「構造的なロックイン」に由来するものが多く、契約締結後の解消は困難。**k1s0 の企画書では Build のリスクへの対処方針を具体的に示すこと**が説得力の鍵となる。

## 結論

- **同じ機能を提供する製品は既に多数存在する**。機能そのものでは差別化できない。特に Dapr とは機能的な重複が大きく、企画書では必ず明示的に言及して差別化を語る必要がある。
- k1s0 の勝ち筋は以下 5 点セット。
  1. **JTC 情シスが無償で始められる** (稟議のハードルがない)
  2. **レガシー (.NET Framework 等) と共存できる**
  3. **ベンダー依存しない (オンプレ / VM で完結)**
  4. **言語を尊重する** — tier2 / tier3 の言語を縛らず、既存資産を活かせる
  5. **レガシー所有者に金銭的メリットを提示する** — 運用コスト削減・デプロイ速度改善で自発的な参加を促す
- 企画書ではこの 5 点を一貫した主張として繰り返す。
- 既存 OSS (Rancher / Backstage / Kyverno 等) とは敵対せず、**組み合わせて使う前提**で設計することで開発スコープを現実的に抑える。
- **Dapr との関係** (2026-04-12 決定):
  - **Dapr は tier1 の内部実装として組み込み、tier2 / tier3 からは完全に隠蔽する**
  - tier2 / tier3 は **tier1 公開 API (多言語クライアントライブラリ) のみ** を利用し、Dapr SDK / annotation / Component YAML / Dapr エラーメッセージのいずれも触れない
  - tier1 公開ライブラリ内では各言語の **安定 Dapr SDK** (C# / Go / Java / TS / Python) を内部利用する
  - tier1 Rust 実装は Dapr を直接呼ばない (Dapr Rust SDK が Alpha のため)。Rust 部分は JTC 固有機能 / Component 管理 / 雛形生成 CLI に限定する
  - これにより Dapr のバージョン更新・破壊的変更・将来的な差し替えの影響を **tier1 内に閉じ込められる**
  - 詳細は `tier1_vs_Dapr詳細比較.md` を参照
- 商用競合 (Gloo Platform / Mia-Platform / Humanitec) は同じ問題意識から既に製品化されている。k1s0 はそれらを無償の OSS 積み上げで代替することを目指す。
