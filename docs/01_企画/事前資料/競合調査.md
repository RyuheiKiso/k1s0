# 競合調査

## 目的
k1s0 が対象とする「開発者プラットフォーム / マイクロサービス基盤」領域の既存製品・事例を整理し、k1s0 の立ち位置 (差別化ポイント) を明確にする。

## 注意事項
本資料の各製品情報は一般に公開されている情報に基づく概観である。
企画書に数値や機能詳細を引用する場合は、公式ドキュメント最新版での再確認を必須とする。

---

## 競合の分類

| 分類 | 代表例 | k1s0 との関係 |
|---|---|---|
| OSS 型 IDP (Internal Developer Platform) | Backstage | 補完関係 |
| 商用 IDP / PaaS | Humanitec / Port / Cortex | 機能重複 (有償) |
| 商用 k8s ディストリビューション | OpenShift / Tanzu | 機能重複 (有償・ベンダー依存) |
| OSS k8s 管理プラットフォーム | Rancher | 採用候補 |
| マイクロサービス building blocks | **Dapr** | **tier1 と最も機能重複する** |
| サービスメッシュ / Gateway 商用プラットフォーム | Gloo Platform (Solo.io) | 機能重複 (有償) |
| CNCF アプリ配信 / 抽象化 | KubeVela / Crossplane | 思想的参考 |
| ポリシー / ガバナンス | Nirmata / Kyverno | 部分競合 / 採用候補 |
| レガシー PaaS | Cloud Foundry / Tanzu Application Service | 移行元 (既存利用企業向け) |
| マイクロサービスプラットフォーム (商用) | Mia-Platform | 機能重複 (有償) |
| 国内大手の自社開発 PaaS | メルカリ / LINE 等 | k1s0 アプローチの妥当性事例 |

| 提供元 | Spotify が公開、CNCF Incubating |
| 主な機能 | サービスカタログ、ソフトウェアテンプレート、TechDocs、プラグインによる拡張 |

| 強み | 圧倒的なエコシステム。大手企業での採用事例多数。 |

| k1s0 との関係 | **補完関係**。k1s0 は実行基盤寄り、Backstage はポータル寄り。将来的には k1s0 に Backstage を組み込む選択肢もある。 |

## 6. Dapr (Microsoft 発 / CNCF)

| 項目 | 内容 |
|---|---|
| 種別 | Distributed Application Runtime (マイクロサービス building blocks) |
| 提供元 | Microsoft が公開、現在は CNCF 傘下 (CNCF Graduated) |
| 主な機能 | サイドカー経由で提供される共通機能 (state management / pub-sub / secrets / bindings / actors / service invocation / distributed tracing / workflow) |
| ライセンス | Apache 2.0 |
| 強み | **言語非依存** (gRPC/HTTP API で呼び出すため C# / Go / Rust / Python すべて等価に利用可能)。tier1 が提供したい共通機能の大半を既に実装済み。CNCF Graduated で信頼性高い。 |
| 弱み | サイドカー方式によるリソース消費。学習コストあり。Dapr に依存することで「Dapr ロックイン」の懸念。独自の抽象モデルを理解する必要がある。 |
| k1s0 との関係 | **最も機能重複する競合** — tier1 が提供しようとしている「共通基盤サービス」の多くは Dapr が既に提供している。以下の 3 つの選択肢がある。 |

**選択肢の検討**:

| 選択肢 | メリット | デメリット |
|---|---|---|
| (a) tier1 をすべて自前実装 | 完全な制御、Rust で統一、ロックイン回避 | 開発コスト最大、車輪の再発明 |
| (b) Dapr を tier1 の一部として組み込む | 開発コスト削減、業界標準の採用 | Dapr への依存、サイドカー方式による複雑性、tier1 の Rust 統一が崩れる |
| (c) tier1 で Dapr 互換 API を提供 | 将来的な Dapr 移行パスを残しつつ Rust で統一 | API 設計の制約、実装負荷は (a) と同等 |

**示唆**: Dapr の存在は k1s0 の企画で**必ず言及する必要がある**。「なぜ Dapr を採用せず tier1 を自前実装するのか」の問いは決裁者からほぼ確実に出る。以下が想定される答弁:
- Rust で統一することで長期保守性・メモリ安全性を最優先する
- サイドカー方式のリソース消費を避ける (オンプレ / VM 前提ではリソース効率が重要)
- tier1 の設計を k1s0 固有の業務要件 (JTC の認証基盤連携等) に最適化できる
- **ただし将来的な Dapr 互換化の余地は残す** (選択肢 (c) の方向性)

## 12. 国内大手企業の自社開発プラットフォーム事例

k1s0 の方向性 (大企業が自社用マイクロサービス基盤を内製する) には国内でも先行事例がある。
ただし**本セクションの具体情報は一般論レベルに留めており、企画書に引用する際は各社の公開テックブログ・登壇資料から最新事例を個別に収集する必要がある**。

一般的な傾向として、以下の点が既に業界内で確認できる。

- Web / モバイル系の大手テック企業の多くが、k8s をベースにした自社マイクロサービスプラットフォームを内製している。
- 共通基盤チームを設置し、アプリチームが業務ロジックに集中できる体制を敷いている事例が多い。
- 公開資料の多くはクラウド (AWS / GCP) マネージド前提であり、オンプレ / VM 上での k8s 運用ノウハウは相対的に情報量が少ない。

**示唆**:
- 「大企業が自社用に k8s ベースの基盤を作る」は業界のトレンドとして既に成立しており、k1s0 のアプローチは奇抜ではない。
- ただし事例の多くは **Web / モバイル企業** が中心で、**JTC の情シス部門による内製事例は相対的に少ない**。
- k1s0 の独自性は「JTC 情シス部門でも運用できること」「.NET Framework 等のレガシー共存」「オンプレ / VM 前提」にある。

## k1s0 の差別化ポイント

| 軸 | 既存製品 / 事例 | k1s0 |
|---|---|---|
| コスト | 商用製品は有償サブスクリプション | OSS 積み上げで無償 |
| ベンダー依存 | OpenShift / Tanzu / Gloo / Mia-Platform はロックインあり | 回避を設計原則とする |
| 対象ユーザー層 | Web 系テック企業 / 大規模 SaaS 向けが多い | **JTC 情シス部門** |
| レガシー共存 | 考慮されないケースが多い | **.NET Framework 等の古い資産との共存を第一級で扱う** |
| 実行環境 | クラウドマネージド前提が多い | **オンプレ / VM を第一級で扱う** |
| 言語の自由度 | プラットフォーム指定の言語が中心 | **tier2 / tier3 は C# / Go / MAUI / React など自由。既存資産の言語を尊重** |
| サイドカー方式 | Dapr はサイドカー前提でリソース消費大 | **tier1 はライブラリ / サービス混成でリソース効率を重視** |
| レガシー所有者の動機付け | 移行を強制 / 無視 | **運用コスト削減・デプロイ速度などの金銭的メリットで「作り直してでも参加したくなる」設計** |
| 人材要件 | k8s / サービスメッシュの専門家前提 | 未経験チームでも始められる雛形・ドキュメント提供 |
| 基盤言語 | Go / TypeScript / Java 中心 | **Rust で長期保守性を最優先** |

## JTC 情シス部門固有の選定軸

k1s0 は一般的な開発者プラットフォームとは異なる「JTC 情シス部門」という特殊な市場を対象とする。
この市場に特有の選定軸は以下のとおり。

| 軸 | JTC 情シスでの重要度 | 既存製品の典型的な立ち位置 | k1s0 の立ち位置 |
|---|---|---|---|
| **稟議が通るか** | 極めて高 | 商用は数百万円〜で稟議ハードル高 | **無償 OSS なので稟議不要** |
| **情報システム子会社でも運用できるか** | 高 | 専門家前提で運用困難 | 雛形・ドキュメント・段階導入で対応 |
| **既存ベンダー契約との整合** | 高 | OpenShift 等は IBM / Red Hat 契約が前提 | 既存契約に影響しない |
| **セキュリティ監査・ISO への適合** | 高 | 個別製品ごとに監査資料を用意 | OSS 積み上げ構成で監査は tier 単位 |
| **日本語サポート・ドキュメント** | 中 | 商用は日本法人サポートあり | 自社内のノウハウ蓄積に依存 |
| **既存 .NET Framework 資産の扱い** | 極めて高 | 通常は考慮されない | **第一級で対応** |
| **オンプレ DC / VMware 環境での動作** | 高 | クラウドマネージド前提が多い | **前提条件として明示** |

## Build vs Buy リスク分析

k1s0 を内製する (Build) か、商用製品を購入する (Buy) かの選択は企画書の核心論点となる。

### Build (k1s0 を内製) のリスク

| リスク | 発生確率 | 影響度 | 対処方針 |
|---|---|---|---|
| 個人開発ゆえの属人化 | 高 | 高 | **ドキュメント充実 + コード公開** でバス係数を下げる |
| 開発が途中で頓挫する | 中 | 高 | MVP スコープを最小化して早期に価値提供 |
| 新技術追従の工数が読めない | 中 | 中 | OSS 積み上げなのでコミュニティ更新に追従するだけ |
| セキュリティ脆弱性の対応遅延 | 中 | 高 | 採用 OSS の CVE 監視と tier1 の修正体制を事前に設計 |
| 所属企業の評価 / 支援が得られない | 中 | 中 | 無償提供の時点で ROI を明示して合意を取る |

### Buy (商用製品を購入) のリスク

| リスク | 発生確率 | 影響度 | 対処方針 |
|---|---|---|---|
| ライセンス費が予算を圧迫 | 高 | 高 | k1s0 なら回避 |
| ベンダーロックイン | 高 | 極高 | k1s0 なら回避 |
| ベンダー都合の仕様変更 / 値上げ (例: VMware Broadcom 事例) | 中 | 極高 | k1s0 なら回避 |
| 自社要件 (.NET Framework 共存等) が製品側でサポートされない | 高 | 高 | k1s0 なら自由に対応 |
| 日本法人のサポート品質が担保されない | 中 | 中 | k1s0 は自社運用で品質自己決定 |

**示唆**: Build 側のリスクは「個人開発起点」に由来するものが主で、段階的に会社として引き取る体制を整えれば低減できる。Buy 側のリスクは「構造的なロックイン」に由来するものが多く、契約締結後の解消は困難。**k1s0 の企画書では Build のリスクへの対処方針を具体的に示すこと**が説得力の鍵となる。

## 結論

- **同じ機能を提供する製品は既に多数存在する**。機能そのものでは差別化できない。特に Dapr とは機能的な重複が大きく、企画書では必ず明示的に言及して差別化を語る必要がある。
- k1s0 の勝ち筋は以下 5 点セット。
  1. **JTC 情シスが無償で始められる** (稟議のハードルがない)
  2. **レガシー (.NET Framework 等) と共存できる**
  3. **ベンダー依存しない (オンプレ / VM で完結)**
  4. **言語を尊重する** — tier2 / tier3 の言語を縛らず、既存資産を活かせる
  5. **レガシー所有者に金銭的メリットを提示する** — 運用コスト削減・デプロイ速度改善で自発的な参加を促す
- 企画書ではこの 5 点を一貫した主張として繰り返す。
- 既存 OSS (Rancher / Backstage / Kyverno 等) とは敵対せず、**組み合わせて使う前提**で設計することで開発スコープを現実的に抑える。
- **Dapr との関係**は特に重要: 「なぜ Dapr を採用しないのか」に明確な答えを用意する。現状の方針は "Rust 統一 / サイドカー回避 / JTC 固有業務への最適化" だが、**将来的な Dapr 互換化の余地は残す**。
- 商用競合 (Gloo Platform / Mia-Platform / Humanitec) は同じ問題意識から既に製品化されている。k1s0 はそれらを無償の OSS 積み上げで代替することを目指す。
