# tier1 vs Dapr 詳細比較

## 目的

本資料は、k1s0 の tier1 層が担う 3 つの目的について、Dapr と機能レベルで比較し、どちらに寄せるべきか (または両立させるべきか) の判断材料を提供する。

## tier1 の 3 つの目的 (前提)

本比較では tier1 の目的を以下の 3 点に限定して扱う。

1. **tier2 / tier3 がインフラ層 (Kafka / Loki / Prometheus / Secret Store 等) にアクセスする方法の統一化**
2. **ログの標準化** (構造化ログの形式統一、出力先統一)
3. **OpenTelemetry の標準化** (計装方法・コンテキスト伝搬・エクスポート先の統一)

この 3 点は、`docs/99_壁打ち/解決したい課題.md` に示された k1s0 の方針 (tier1 は基盤的な共通機能を提供する) を具体化したものである。

---

## 1. インフラ層アクセスの統一

### 比較表

| 観点 | tier1 (k1s0) | Dapr |
|---|---|---|
| 目的 | tier2 / tier3 が Kafka / state store / secret store 等に直接触れず tier1 経由で利用 | アプリが state store / pub-sub / secret 等に直接触れず Dapr 経由で利用 |
| 統一の単位 | tier1 が提供する API (Rust 実装 + 各言語クライアントライブラリ) | Dapr HTTP / gRPC API (言語非依存) |
| 抽象の粒度 | k1s0 固有の業務要件に最適化可能 | 業界標準の抽象 (state / pub-sub / secrets / bindings) |
| 実装形態 | ライブラリ (in-process) + サービス (out-of-process) の混成 | サイドカー (標準) / DaemonSet / Deployment (Dapr Shared) |
| バックエンド切替 | tier1 実装次第。設定ファイルで切替可能に設計する必要あり | **Dapr Component で宣言的に切替可能**。アプリコードの変更不要 |
| 言語サポート | tier1 が提供する各言語クライアントに依存 | **言語非依存** (HTTP / gRPC で呼ぶだけ) |
| 追加インフラ | tier1 サービスが必要な場合あり | Dapr Control Plane + サイドカー / 共有ランタイム |
| 学習コスト | tier1 独自 API の習得 | Dapr 概念 (building blocks / component) の習得 |
| 実装コスト | **高** (ゼロから実装) | **低** (既存実装を利用) |
| JTC 固有要件への適合 | 自由にカスタマイズ可能 | カスタムコンポーネントとして拡張可能だが学習コストあり |

### 評価

**Dapr が優位な点**:
- 既に実装済みで成熟している (CNCF Graduated)
- 言語非依存なので tier2 / tier3 の言語の自由度を保ったまま利用できる
- Component による宣言的切替で Kafka → NATS 等の移行が容易
- サイドカーが mTLS / リトライ / サーキットブレーカー等の信頼性機能を既に提供

**tier1 が優位な点**:
- JTC 固有の業務要件 (社内認証基盤との連携、監査ログ要件等) に最適化できる
- サイドカーを強制しない選択肢 (ライブラリ呼び出し) により、リソース制約のある環境にも適合
- Rust 実装で安全性・性能を統一できる (ただし tier1 → Dapr 併用の場合、この優位性は限定的)

**結論**: インフラ層アクセス統一という目的に限れば、**Dapr が機能面で優位**。tier1 を自前実装する合理性は、**JTC 固有要件への最適化が必要な範囲に限定**される。

## 2. ログ標準化

### 比較表

| 観点 | tier1 (k1s0) | Dapr |
|---|---|---|
| ログ出力方式 | tier1 ログライブラリ経由で構造化ログを強制 | アプリが通常通り標準出力にログを出す。Dapr サイドカーは独自にサイドカー自身のログを出す |
| ログ形式の強制力 | **tier1 ライブラリを使う限り形式が統一される** | **アプリ内部ログ形式はアプリ任せ** (Dapr は関与しない) |
| 構造化フィールド | tier1 が service_name / trace_id / user_id 等を自動注入 | アプリが自前で構造化する必要あり |
| 集約先 | Loki (tier1 が抽象化) | 任意のログバックエンド (Loki / ELK / Datadog 等、Dapr 経由ではなくクラスタのログ集約機構に依存) |
| tracing コンテキストの埋め込み | tier1 ログライブラリが trace_id / span_id を自動注入 | Dapr は tracing を OpenTelemetry に統合するが、**アプリ側ログへの trace_id 伝搬は別途実装が必要** |
| JTC 監査要件への適合 | 自由にカスタマイズ可能 (例: 個人情報マスキング、監査ログフォーマット) | カスタマイズ余地は限定的 |

### 評価

**ここは tier1 が明確に優位**。

Dapr の本質は「サービス間通信の抽象化」であり、**アプリケーション内部のログ形式までは強制しない**。一方 tier1 がログライブラリを各言語向けに提供することで、アプリ内部のログ形式・フィールド・集約先を統一できる。

特に JTC の情報システム部門では以下の要件が発生しやすく、Dapr のみでは対応困難:
- 個人情報のマスキング / ハッシュ化
- 監査ログの固定フォーマット (経産省ガイドライン準拠等)
- 長期保存ログと短期保存ログの分離
- 特定業務カテゴリのログに対する改ざん防止

これらは tier1 ログライブラリで強制しないと、現場チームの実装ばらつきで崩れる。

**結論**: ログ標準化は **tier1 の存在意義が明確に出る領域**。Dapr を併用しても tier1 ログライブラリは必要。

## 3. OpenTelemetry 標準化

### 比較表

| 観点 | tier1 (k1s0) | Dapr |
|---|---|---|
| OTel 計装 | tier1 が各言語向け初期化コードを提供し、アプリ起動時に埋め込む | **Dapr サイドカーがアプリ外側で自動計装**。アプリコードの変更最小 |
| サービス間通信のトレース | tier1 HTTP / gRPC ミドルウェアで trace header 伝搬 | Dapr service invocation が自動で trace header を伝搬 |
| アプリ内部の span 生成 | tier1 OTel ライブラリで明示的に span を生成 | **アプリが自前で OTel SDK を使う必要がある** (Dapr は関与しない) |
| カスタム属性の付与 | tier1 が標準属性 (tenant_id / user_id 等) を自動注入 | アプリ実装に依存 |
| エクスポート先 | Jaeger (tier1 が抽象化、差し替え可能) | 任意のバックエンド (OTel Collector 経由) |
| メトリクス | tier1 メトリクスライブラリ経由で Prometheus に公開 | Dapr 自身のメトリクスを自動公開、アプリメトリクスは別途 |
| 計装の網羅性 | アプリ開発者が tier1 API を使う範囲で網羅 | サービス境界は網羅されるがアプリ内部は網羅されない |

### 評価

**棲み分け可能だが両方必要**。

Dapr の OTel 計装は「**サービス境界 (サイドカー通過時)**」に強く、tier1 の OTel 計装は「**アプリケーション内部**」に強い。以下のような分業ができる:

```
┌─────────────────────────────────────────────────────────┐
│ Dapr サイドカー (service invocation / pub-sub トレース) │  ← Dapr が自動計装
├─────────────────────────────────────────────────────────┤
│ アプリケーション (ビジネスロジック / DB アクセス)         │  ← tier1 OTel ライブラリ
└─────────────────────────────────────────────────────────┘
```

両方の計装結果を同じ OTel Collector → Jaeger に流すことで、エンドツーエンドのトレースが一本化される。

**結論**: OTel 標準化は **Dapr と tier1 の両方で補完的に実施**する。どちらか一方では不完全。

## 総合判断

### 目的別サマリ

| 目的 | tier1 単独 | Dapr 単独 | tier1 + Dapr 併用 | 結論 |
|---|---|---|---|---|
| (1) インフラ層アクセス統一 | ○ (実装コスト高) | **◎** | ○ (二重管理の懸念) | **Dapr に寄せる余地大** |
| (2) ログ標準化 | **◎** | △ (アプリ内部は対象外) | ◎ | **tier1 が必須** |
| (3) OTel 標準化 | ○ (アプリ内部のみ) | ○ (サービス境界のみ) | **◎** | **両方必要 (補完関係)** |

### tier1 のスコープ再定義案

この比較結果を踏まえ、tier1 のスコープを以下のように再定義する選択肢を検討する価値がある。

**従来案 (これまでの想定)**:
tier1 = インフラ抽象化 + 共通機能実装 (Rust 自前)

**再定義案 (本比較を反映)**:
tier1 = 以下の 4 つに集約
- (A) **Dapr Component の構成管理** — tier2 / tier3 が使う pub/sub / state / secrets 等を tier1 が宣言的に管理。アプリチームは tier1 が提供する Component 定義を参照するだけ
- (B) **ログ標準化ライブラリ** — 各言語向けに構造化ログライブラリを提供。Dapr が対象としないアプリ内部ログを強制統一
- (C) **OTel 計装標準化ライブラリ** — Dapr のサービス境界計装を補完するアプリ内部計装ライブラリ
- (D) **JTC 固有機能** — 社内認証基盤連携、監査ログ、規制対応等、Dapr の標準コンポーネントで対応できない部分

### 再定義案のメリット

1. **tier1 の実装規模が劇的に縮小**する。Kafka クライアント / secret store クライアント / state store クライアント等を自前実装する必要がなくなる
2. MVP スコープが現実的な範囲に収まる
3. k1s0 の「車輪の再発明を避ける」原則と整合
4. アプリチームは業界標準の Dapr API を学べばよく、k1s0 独自 API の習得コストが減る
5. Dapr Rust SDK の Alpha 問題を回避できる (tier1 自体は Dapr を直接呼ばない、tier2 / tier3 が安定 SDK で呼ぶ)

### 再定義案のデメリット・リスク

1. **Dapr への依存**が発生する。Dapr Control Plane の運用が必須になる
2. Dapr の方向転換 (機能廃止、互換性破壊) の影響を受ける
3. 「tier1 は Rust 実装で統一」という当初の思想からの後退
4. サイドカー方式によるリソース消費 (Dapr Shared で軽減可能)
5. Dapr の学習コストがアプリチームに発生する

### 従来案と再定義案の比較

| 観点 | 従来案 (tier1 自前実装) | 再定義案 (tier1 + Dapr 併用) |
|---|---|---|
| 初期開発コスト | **大** | 中 |
| 長期保守コスト | 中 | 中 |
| Dapr への依存 | なし | あり |
| Rust 統一性 | 高 | 中 |
| アプリチームの学習コスト | k1s0 独自 API | Dapr + k1s0 ログ / OTel ライブラリ |
| 業界標準への適合 | 弱 | 強 |
| JTC 固有要件への対応 | 自由 | Dapr 拡張で対応 |
| MVP の現実性 | やや困難 | **現実的** |

## 決定事項 (2026-04-12)

本比較を踏まえ、以下の方針で確定した。

### 基本方針: Dapr を基本、足りない箇所を tier1 の Rust 自前実装で補完

tier1 層は **「Dapr」と「Rust 自前実装」の並立構成** とする。
ただし **tier2 / tier3 は Dapr を一切意識しない**。以下の 3 層構造で隠蔽する。

```
┌──────────────────────────────────────────────────────────┐
│  tier2 / tier3 コード (C# / Go / Java / TS / Python)       │
│  → tier1 公開 API のみを import / 呼び出し                  │
└──────────────────────────────────────────────────────────┘
                        ↓
┌──────────────────────────────────────────────────────────┐
│  tier1 公開 API (多言語クライアントライブラリ)             │
│  k1s0 固有の統一 API を提供                                 │
│  ・log / otel / state / pub-sub / secrets / workflow / 認証 │
└──────────────────────────────────────────────────────────┘
                        ↓ (内部呼び出し, tier2/tier3 からは不可視)
┌──────────────────────────────────────────────────────────┐
│  tier1 内部実装                                            │
│  ・Dapr サイドカー / 各言語 Dapr SDK                        │
│  ・Rust 実装サービス (JTC 固有機能 / 雛形生成 / Component 管理)│
└──────────────────────────────────────────────────────────┘
                        ↓
┌──────────────────────────────────────────────────────────┐
│  infra 層 (k8s / Istio / Envoy Gateway / Kafka / Loki 等)  │
└──────────────────────────────────────────────────────────┘
```

**利用判断のフローチャート (tier1 内部の設計判断)**:

```
tier1 公開 API の実装者が判断:
  ├─ 機能が Dapr の building block で賄える?
  │   ├─ Yes → tier1 公開 API の内部で Dapr SDK (各言語安定版) を呼ぶ
  │   └─ No  → tier1 公開 API の内部で tier1 Rust サービスを呼ぶ
  └─ (補完対象の典型例: ログ標準化 / OTel アプリ内部計装 / JTC 固有要件)
```

tier2 / tier3 開発者はこの判断に関与しない。Dapr の存在を知る必要がない。

### tier2 / tier3 から Dapr を隠蔽するための施策

| 項目 | 隠蔽方法 |
|---|---|
| コード import | tier2 / tier3 は tier1 ライブラリのみ import。Dapr SDK の import は tier1 ライブラリ内部に閉じる |
| Pod annotation (`dapr.io/enabled` 等) | 雛形生成 CLI が自動付与。開発者は手書きしない |
| Dapr Component YAML | tier1 チームが `src/tier1/` で管理。tier2 / tier3 リポジトリには置かない |
| エラーメッセージ | tier1 ライブラリが Dapr エラーをラップし k1s0 固有例外に変換 |
| ログ / トレース属性 | tier1 ライブラリが `k1s0.component` 等の標準属性を付与。Dapr 由来の属性は内部で正規化 |
| 設定ファイル | tier1 ライブラリが k1s0 固有の設定キーを公開。Dapr 固有の設定は tier1 内で変換 |

### tier1 の構成要素

すべての機能は **tier1 公開 API (多言語クライアントライブラリ)** を通して tier2 / tier3 に提供される。
内部実装が Dapr か Rust サービスかは、tier2 / tier3 から見えない。

| カテゴリ | 内部実装 | tier1 公開 API での扱い |
|---|---|---|
| Service invocation | Dapr (内部) | tier1 ライブラリの `invoke_service()` 等でラップ |
| State management | Dapr (内部) | tier1 ライブラリの `state.save/get/delete` 等でラップ |
| Pub / Sub | Dapr (内部) | tier1 ライブラリの `pubsub.publish/subscribe` 等でラップ |
| Secrets | Dapr (内部) | tier1 ライブラリの `secrets.get` 等でラップ |
| Bindings | Dapr (内部) | tier1 ライブラリの binding API でラップ |
| Actors | Dapr (内部) | tier1 ライブラリの actor API でラップ (必要に応じて) |
| Workflow | Dapr Workflow (内部) | tier1 ライブラリの workflow API でラップ |
| Configuration | Dapr (内部) | tier1 ライブラリの config API でラップ |
| Distributed lock | Dapr (内部) | tier1 ライブラリの lock API でラップ |
| **ログ標準化** | **tier1 Rust ライブラリ内蔵 or 各言語ネイティブ実装** | tier1 ライブラリの `log.info/warn/error` 等 |
| **OTel 計装 (アプリ内部)** | **tier1 各言語ライブラリで OTel SDK をラップ** | tier1 ライブラリの `trace.span/event` 等 |
| **JTC 固有機能** | **tier1 Rust サービス** | tier1 ライブラリ経由で呼び出し |
| **Dapr Component 構成管理** | **tier1 チーム (YAML / CRD)** | tier2 / tier3 には見えない (tier1 リポジトリに格納) |
| **雛形生成 CLI** | **tier1 Rust ツール** | サービス雛形に Dapr annotation と tier1 ライブラリ依存を自動で組み込む |

### Dapr の階層配置

- **Dapr は tier1 の構成要素として位置付ける** (infra ではない)。
- 理由: Dapr が提供する機能はまさに「tier1 が提供すべき共通基盤サービス」そのものであり、**Dapr は tier1 の実装手段の一つ**と見なすのが自然。
- これにより「infra 層への直接依存は tier1 のみ」というルールを維持できる。
- tier2 / tier3 が Dapr API を呼ぶことは「tier1 を呼ぶ」の一形態とみなす。

### 受け入れるリスクと対処方針

| リスク | 対処方針 |
|---|---|
| Dapr の方向転換・互換性破壊 | tier1 の雛形生成 CLI で Dapr API の利用を間接化し、破壊的変更時は CLI 出力の更新で吸収 |
| Dapr Control Plane の運用コスト | tier1 チームが Dapr の運用を担当。MVP 段階で運用手順書を整備 |
| Dapr Rust SDK が Alpha | **tier1 Rust 実装は Dapr を直接呼ばない**方針を堅持。tier2 / tier3 は各言語の安定 SDK を使用 |
| 2 系統 (Dapr / tier1) の判断基準の曖昧さ | tier1 ドキュメントで「Dapr で賄えるならまず Dapr」という原則を明文化。判断に迷う事例は tier1 チームがレビュー |
| サイドカー方式のリソース消費 | 必要に応じて Dapr Shared (DaemonSet / Deployment 方式) を検討 |

### 課題メモとの整合

`docs/99_壁打ち/解決したい課題.md` における tier1 の定義「ライブラリやサービスを提供する」は、本決定後も有効。Dapr は tier1 が提供する「サービス」の実装手段の一つとなる。
