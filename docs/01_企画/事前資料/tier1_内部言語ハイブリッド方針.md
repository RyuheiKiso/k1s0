# tier1 内部言語ハイブリッド方針 (Go + Rust)

## 目的

tier1 内部実装の言語選択について、**「Daprファサードは Go、自作領域は Rust」のハイブリッド構成** を採用する判断と、その境界・規約・運用方針を明文化する。

## 決定日

2026-04-12

---

## 1. 背景 (なぜハイブリッドにするか)

### 従来案の構造的問題

`技術選定比較表.md` 9 章では tier1 実装言語を **Rust 一本** とし、`tier1_vs_Dapr詳細比較.md` では **「tier1 Rust 実装は Dapr を直接呼ばない」** という制約を採用していた。これは Dapr Rust SDK が Alpha (`競合調査.md` Section 6 で確認済み) であることへの回避策だが、結果として次の倒置が発生していた。

```
Rust 一本ありき
   ↓
Dapr Rust SDK が Alpha
   ↓
tier1 Rust から Dapr を直接呼べない
   ↓
多言語クライアントで Dapr を隠蔽する必要が生まれる
   ↓
5 言語分の薄いラッパーを保守する負担
```

この鎖は **「Rust を採用したい」という前提のために発生している複雑性** であり、用途で言語を選び直せば解消できる。

### ハイブリッドが解消する点

- Daprファサードは **Dapr Go SDK (stable)** を直接呼ぶ。Alpha 回避のための間接化が不要になる
- Rust は **本当に Rust が活きる用途** に限定される (ZEN Engine 統合 / JTC 固有機能 / 雛形生成 CLI 等)
- tier2 / tier3 から見えるのは **クライアントライブラリ呼び出しと gRPC エンドポイントのみ**で、内部言語の多様性は不可視
- JTC で集めにくい Rust 経験者を Daprファサード側に配置する必要がなくなる

## 2. 基本方針

> **tier1 内部のサービスは「用途」で言語を選ぶ。Daprファサードは Go、自作領域は Rust を既定とする。**

### 言語選択の判断フロー

```
新規 tier1 内部サービスの言語選択:
  ├─ 主たる責務が Dapr building block のラップか?
  │   ├─ Yes → Go (stable Dapr Go SDK を直接利用)
  │   └─ No  → 次の判断へ
  ├─ ZEN Engine / OS バイナリ / crypto / parser など Rust の優位性が出るか?
  │   ├─ Yes → Rust
  │   └─ No  → 次の判断へ
  ├─ 単一バイナリ配布 (CLI / バリデータ) が必要か?
  │   ├─ Yes → Rust
  │   └─ No  → Go (既定)
  └─ 迷った場合は Go (人材確保が容易、エコシステムが広い)
```

### サービス分類 (初期想定)

| サービス | 言語 | 理由 |
|---|---|---|
| State / Pub-Sub / Secrets / Workflow / Bindings ファサード | **Go** | Dapr SDK 安定、HTTP / gRPC ハンドラが薄い |
| ログ / OTel 集約サービス (独立サービス化する場合) | **Go** | OTel Collector / Loki クライアント等のエコシステムが Go 中心 |
| 認証プロキシ (Keycloak / oauth2-proxy 連携) | **Go** | oauth2-proxy 自体が Go 製 |
| **決定エンジン (ZEN Engine 統合)** | **Rust** | ZEN Engine が Rust ネイティブ。in-process 呼び出しが最速 |
| **JTC 固有機能** (監査ログ改ざん防止 / 個人情報マスキング / 規制対応) | **Rust** | crypto / parser 系で Rust の安全性が活きる |
| **雛形生成 CLI** | **Rust** | 単一バイナリ配布、起動高速 |
| **Dapr Component YAML バリデータ / 管理 CLI** | **Rust** | 雛形生成 CLI と同梱可能 |

### 言語境界の規約

サービス境界をまたぐ呼び出しは以下を必須とする。

| 規約 | 内容 |
|---|---|
| **通信プロトコル** | gRPC (HTTP/JSON は禁止) |
| **契約定義** | Protobuf (`.proto`) のみ |
| **契約の保管場所** | `src/tier1/contracts/` に集中管理。Go / Rust どちらも同じ `.proto` から生成 |
| **生成タイミング** | CI で自動生成。手書き禁止 |
| **エラー型** | tier1 共通エラーコード (Protobuf enum) を全サービスで共有 |
| **観測属性** | `service.name` / `service.language` / `k1s0.tier` を OTel 属性で必須付与 |

## 3. tier1 公開 API (多言語クライアントライブラリ) との関係

tier1 公開 API (C# / Go / Java / TS / Python) は **従来通り提供する**。公開 API は次のいずれかの経路で内部サービスに到達する。

```
tier2 / tier3 のコード (C# / Go / Java / TS / Python)
    ↓ tier1 公開クライアントライブラリ
tier1 公開 API (各言語ラッパー)
    ↓ gRPC (tier1 内部の Service Discovery 経由)
    ├──→ tier1 Go サービス (Daprファサード) ──→ Dapr サイドカー / Control Plane
    └──→ tier1 Rust サービス (自作領域)        ──→ ZEN Engine / JTC 固有機能 / etc.
```

- tier2 / tier3 から見える接点は **クライアントライブラリの関数呼び出しのみ**
- 内部が Go か Rust かは **完全に不可視**
- 公開 API のラッパー実装側は、用途に応じて Go サービスか Rust サービスのどちらかに gRPC で振り分ける

## 4. 雛形生成 CLI への影響

雛形生成 CLI は **2 種類のサービス雛形** を出力する責務を持つ。

| 雛形種別 | 出力内容 |
|---|---|
| **Go サービス雛形** | `go.mod` / 主要ハンドラ / Dapr Go SDK 初期化 / Dockerfile (golang base) / k8s manifest / Dapr annotation / OTel 計装 / ヘルスチェック |
| **Rust サービス雛形** | `Cargo.toml` / 主要ハンドラ / tier1 Rust 共通ライブラリ依存 / Dockerfile (distroless or alpine) / k8s manifest / OTel 計装 / ヘルスチェック |

両雛形に共通の規約:

- **共通 Protobuf 契約への参照**を最初から含む
- **構造化ログのフォーマットを言語横断で同一**にする (フィールド名 / レベル名 / trace_id 注入位置)
- **エラーコード体系を言語横断で同一**にする (Protobuf enum を共有)
- **CLAUDE.md の規約 (1ファイル300行以内 / 日本語コメント必須)** を雛形時点で守った状態で出力

雛形生成 CLI 自体は Rust で実装するが、テンプレート部分は外出し (`tera` 等のテンプレートエンジン) し、テンプレート編集だけなら Go 経験者でも対応可能にする。

## 5. 受け入れるトレードオフ

| トレードオフ | 影響 | 対処 |
|---|---|---|
| ビルド / CI / Lint / 依存管理が 2 系統 | 構築初期コストが上がる | 雛形生成 CLI で初期構成を均一化。手書き禁止 |
| 言語境界の契約管理 | Go ↔ Rust 間で型不整合バグの温床 | Protobuf gRPC を必須化、`.proto` 集中管理、CI で生成物を verify |
| コーディング規約の二重保守 | 規約逸脱が片言語で起きやすい | golangci-lint / clippy 双方で同等のルール enforce、CI で reject |
| オンコール担当が両言語を読める必要 | 障害時の初動が遅れる可能性 | 構造化ログ / エラーコード / OTel 属性を言語横断で同一にし、デバッグの初手は言語非依存にする |
| 雛形生成 CLI 自体は Rust 一本 | Go 担当が CLI 拡張に貢献しにくい | テンプレートを `tera` で外出し。テンプレート編集は誰でも可 |
| Rust 経験者と Go 経験者の役割分担 | 役割固定で属人化する | tier1 チームのスキルマトリクスを明示、片方の言語のみ担当者を作らない方針 |

## 6. 決定を見直す条件

以下のいずれかが成立した場合、本方針を再評価する。

1. **Dapr Rust SDK が Stable に到達** — Daprファサードを Rust に統一する選択肢が現実的になる
2. **JTC 内に Rust 経験者が一定数 (5 名以上) 集まった** — Daprファサードも Rust にする余地が出る
3. **Go / Rust 間の Protobuf 契約管理が破綻した** — 単一言語に戻す議論が必要になる
4. **tier1 内部サービスが 30 を超えた** — 言語多様性の運用コストが許容範囲を超える可能性

各条件は Phase 終了時のレトロスペクティブで定量的にチェックする。

## 7. 企画書での答弁案

決裁者から「なぜ tier1 を 1 言語に統一しないのか」と問われた際の回答構成:

1. **tier2 / tier3 からは内部言語が完全に不可視**。tier1 公開クライアントライブラリ越しに使うため、内部実装言語が何であっても影響しない
2. **Daprファサードは Dapr Go SDK が安定しているため Go が最適**。Rust SDK が Alpha のため Rust で書くと隠蔽の追加コストが発生する
3. **自作領域は Rust が活きる用途に限定** (ZEN Engine 統合 / 監査ログ改ざん防止 / 雛形 CLI 等)。長期保守でメモリ安全性が効く
4. **JTC で人材が集めやすい Go を多用するサイドに置く**ことで属人化リスクを下げる
5. **言語境界は Protobuf gRPC で厳密に管理**。型不整合バグは構造的に防止される
6. 結果として **「Rust 採用根拠の明確化」と「人材確保リスク低減」と「Dapr 直接利用の自然さ」を同時に達成**できる
