# 技術選定比較表

## 目的
k1s0 が採用する技術スタックについて、選定候補と採用理由・却下理由を整理する。
課題メモに記載された技術選定の妥当性を裏付け、企画書での説明責任を果たすための資料。

## 選定の前提条件
- オンプレミス / クラウド VM で動作すること (マネージドサービス非依存)
- ベンダーロックインを回避できること
- OSS であること (商用製品は補助的)
- コミュニティが活発で長期運用に耐えること
- JTC の情報システム部門が扱える習得難易度であること

---

## 1. コンテナオーケストレーション

| 候補 | 採用可否 | 評価 |
|---|---|---|
| **Kubernetes (k8s)** | ◎ 採用 | デファクトスタンダード。エコシステム最大。学習コストは高いが情報源豊富。 |
| Docker Swarm | △ 却下 | シンプルだが機能不足。k8s に比べ新規採用事例・情報量が乏しい。 |
| HashiCorp Nomad | △ 却下 | 軽量で優秀だがコミュニティ規模が小さく人材確保困難。 |

**採用理由**: マイクロサービス基盤としての機能成熟度、エコシステム、人材市場性で他候補を圧倒する。

---

## 2. API ゲートウェイ

### 候補一覧

| 候補 | 採用可否 | 概評 |
|---|---|---|
| **Envoy Gateway** | ◎ 採用 | Istio と同じ Envoy データプレーンで統一できる。Kubernetes Gateway API ネイティブ準拠。 |
| Kong Gateway | ○ 次点 | プラグインエコシステム最大。OSS 版のみで本番運用可能だが Istio との二重プロキシ問題がある。 |
| Traefik | △ 次点 | k8s 親和性高い。プラグインエコシステムは Kong より弱く、Envoy ほど高性能でもない。 |
| NGINX Ingress | △ 却下 | Ingress としては実績十分だが API Gateway 機能 (認証・変換・レート制限) が弱い。 |

### Kong vs Envoy Gateway 詳細比較

| 観点 | Kong Gateway | Envoy Gateway |
|---|---|---|
| **ベース技術** | NGINX / OpenResty (Kong Gateway 3 系は Lua + Go プラグインを併用) | Envoy Proxy (C++) |
| **設定モデル** | 独自 CRD (Kong Ingress Controller) または DB-less モード。Kubernetes Gateway API にも対応中。 | **Kubernetes Gateway API ネイティブ準拠**。標準 CRD のみで操作。 |
| **パフォーマンス** | 実用十分だが Envoy ほどの高スループットは出ない。 | Envoy の C++ 実装により**高スループット・低レイテンシ**。大規模 API 基盤での実績豊富。 |
| **プラグイン / 拡張** | **OSS だけで数十種類のプラグイン** (OIDC, JWT, rate-limit, transformation, OpenTelemetry 等)。Lua / Go で独自拡張可。 | Envoy Filter (Wasm / Lua) で拡張。既製プラグインは Kong より少ないが、Wasm でほぼ何でも書ける。 |
| **認証機能** | OIDC / JWT / OAuth2 / Basic / Key Auth などプラグインで即利用可。 | 基本認証は対応。OIDC 等は **別途 ext_authz サーバー (例: oauth2-proxy) との連携** が必要。 |
| **Istio との共存** | Kong (NGINX) + Istio (Envoy) で**プロキシが 2 種類混在**。運用監視が二重管理になる。 | Envoy Gateway + Istio Ingress Gateway は**共に Envoy**。計装・メトリクス・デバッグが統一される。 |
| **成熟度 / コミュニティ** | 2015 年〜、GitHub Star 39k+。事例豊富。 | CNCF 配下、2023 年 GA。Envoy 自体は CNCF Graduated (2018)。比較的新しいが基盤技術の信頼性は高い。 |
| **商用版の位置付け** | OSS 版 (Kong Gateway) と商用版 (Kong Konnect / Enterprise) で機能差あり。**エンタープライズ向け機能の一部は有償**。 | 商用依存なし。純粋な OSS。 |
| **ドキュメント / 学習コスト** | 長年の蓄積で Qiita / Zenn / 書籍の情報も多い。 | 新しいため情報は相対的に少ない。公式ドキュメントは整備済み。 |
| **Kubernetes Gateway API 対応** | 対応中 (v3 系で実装進行)。従来は独自 CRD が中心。 | **Gateway API のリファレンス実装に位置付けられており標準性が高い**。 |

### Kong を選んだ場合のメリット / デメリット

**メリット**:
- 認証・レート制限・変換などのプラグインが**即座に使える**。tier1 で独自実装する範囲が最小化される。
- 事例・日本語情報が豊富で、JTC 情シス部門でも導入しやすい。
- OIDC プラグインで SSO 統合が設定ベースで完結する。

**デメリット**:
- Istio が採用する Envoy と異なるプロキシ技術 (OpenResty / NGINX) が混在し、**二重管理**になる。
- OSS 版と商用版の機能差を常に意識する必要がある。
- Envoy Gateway に比べ Kubernetes Gateway API への準拠は遅れ気味。

### Envoy Gateway を選んだ場合のメリット / デメリット

**メリット**:
- **Istio と同じ Envoy 系で統一される**ため、プロキシ設定・メトリクス・トレース・デバッグ手順が一貫する。運用チームの認知負荷が下がる。
- Kubernetes Gateway API ネイティブで、将来的な標準化の恩恵を受けやすい。
- 商用版に縛られず、純粋な OSS で完結する。
- 高スループット要件が出たときもボトルネックになりにくい。

**デメリット**:
- 認証・レート制限・変換などの**高レベル機能は ext_authz サーバーや Wasm フィルタを自前で組み合わせる必要**がある。tier1 側の実装負担が増える。
- 日本語情報が少なく、トラブル時は一次情報 (英語) への依存度が高い。
- プロダクトとしての成熟度は Kong より若い。

### 採用理由

現状の k1s0 のアーキテクチャ (**Istio をサービスメッシュとして採用**) を踏まえ、**Envoy Gateway を採用**する。理由は以下。

1. **プロキシ技術が Envoy に統一される**ため、運用・監視・デバッグの手順が一本化できる。課題メモにある「未経験チームでも運用できる」という目標に合致する。
2. **Kubernetes Gateway API への準拠**が進んでおり、将来的に別の Gateway 実装に乗り換える際の移行コストが低い。ベンダーロックイン回避の原則とも整合する。
3. **商用版と OSS 版の機能差を気にしなくて良い**。Kong の OSS / Enterprise の切り分けを監視する必要がなくなる。
4. 認証・レート制限などの高レベル機能は、いずれにせよ **tier1 で共通ライブラリ / サービスを提供する方針**なので、Kong のプラグインに頼らなくても同等の機能は tier1 で実現できる。むしろ tier1 に集約したほうが k1s0 の抽象化原則に沿う。

### トレードオフと対処方針

- **初期開発コスト**: 認証・レート制限・変換等を tier1 で実装する必要があり、Kong のプラグイン流用と比べ初期コストが高い → MVP では認証を最優先、他は後続フェーズに送る。
- **日本語情報の少なさ**: Envoy Gateway の日本語情報は相対的に少ない → 公式ドキュメント (英語) を一次情報として扱い、tier1 開発チームで運用手順書を内製する。
- **ext_authz サーバー**: OIDC / SSO 連携では ext_authz サーバー (oauth2-proxy 等) との組み合わせを前提とする。tier1 の認証サービスが ext_authz インタフェースを満たす設計とする。

## 3. サービスメッシュ

| 候補 | 採用可否 | 評価 |
|---|---|---|
| **Istio** | ◎ 採用 | 機能最多。運用ノウハウ豊富。オーバーヘッドは大きい。 |
| Linkerd | ○ 次点 | 軽量でシンプル。Rust 製データプレーン。機能は Istio より絞られる。 |
| Consul Connect | △ 却下 | HashiCorp 依存。k8s 前提なら Istio が合う。 |

**採用理由**: トラフィック制御・mTLS・観測性を一括で提供。CNCF Graduated で信頼性が高い。
Linkerd も有力だが、機能面で Istio の方が JTC の長期要件に応えやすい。

## 4. 分散トレーシング

| 候補 | 採用可否 | 評価 |
|---|---|---|
| **OpenTelemetry + Jaeger** | ◎ 採用 | OTel は計装の業界標準。Jaeger はバックエンドとして実績十分。 |
| Zipkin | △ 却下 | Jaeger と同等だが k8s との親和性で Jaeger が優位。 |
| Grafana Tempo | ○ 次点 | Loki / Grafana と統合しやすい。将来的に移行検討余地あり。 |

**採用理由**: OTel は計装コードをベンダーロックインから解放する。バックエンドは差し替え可能。

## 5. ログ基盤

| 候補 | 採用可否 | 評価 |
|---|---|---|
| **Loki** | ◎ 採用 | 軽量。Grafana 統合。インデックスを持たず低コスト。 |
| Elasticsearch (ELK) | △ 却下 | 高機能だがリソース消費が大きい。オンプレ小規模環境にはオーバースペック。 |
| ClickHouse + Fluentd | ○ 次点 | 大規模向け。学習コスト高。 |

**採用理由**: オンプレ前提でリソース効率が重要。Loki は運用負荷が低い。

## 7. 分散トランザクション

| 候補 | 採用可否 | 評価 |
|---|---|---|
| **Saga パターン** | ◎ 採用 | マイクロサービスの標準解。実装負荷はあるが疎結合を保てる。 |
| 2PC (Two-Phase Commit) | × 却下 | 可用性を下げる。マイクロサービスに不向き。 |
| TCC (Try-Confirm-Cancel) | △ 却下 | Saga より複雑。学習コスト高。 |

**採用理由**: マイクロサービスでの結果整合性を許容する JTC 業務は多い。Saga の実装を tier1 で支援する。

## 8. メッセージング / イベントバス

| 候補 | 採用可否 | 評価 |
|---|---|---|
| **Apache Kafka** | ◎ 採用 | 実績豊富。エコシステム最大。イベントソーシング / ストリーム処理に対応。永続化・リプレイ機能が堅牢。 |
| NATS (JetStream) | ○ 次点 | 軽量・高速で運用容易。ただしエコシステムと実績で Kafka に劣る。 |
| RabbitMQ | △ 却下 | 実績はあるが、スループットとイベントソーシング適性で Kafka に劣る。 |

**採用理由**: 長期運用での実績・エコシステム・イベントソーシング対応を重視。Saga パターンにおけるイベント履歴のリプレイ性が堅牢で、JTC の業務系ワークフローとの親和性が高い。Strimzi Operator により k8s 上での運用負荷も軽減できる。

**補足 (運用負荷のトレードオフ)**: Kafka は Zookeeper 依存を外した KRaft モードや Strimzi Operator の成熟により、以前ほどの運用負荷はない。ただし NATS と比べれば依然としてリソース消費は大きいため、tier1 側で共通クライアントライブラリを提供し、利用者が直接 Kafka API に触れない設計とする。

## 9. tier1 実装言語

| 候補 | 採用可否 | 評価 |
|---|---|---|
| **Rust** | ◎ 採用 | メモリ安全・高性能・長期保守性。基盤用途に最適。 |
| Go | ○ 次点 | 学習コスト低く生産性高い。tier2 / tier3 で採用可。 |
| C# (.NET) | △ 却下 | .NET 6+ は Linux も十分実用的。ただし基盤用途で Rust ほどのメモリ安全性・並行性の静的保証が得られない。tier2 / tier3 では積極的に採用。 |

**採用理由**: 基盤は安全性と性能が最優先。Rust は長期保守での退行を抑えられる。

## 付録: 検討対象外としたカテゴリ

- **クラウドマネージドサービス全般** (EKS / AKS / GKE, Cloud Run, Lambda, RDS 等)
  → 「オンプレ / VM 動作」「ベンダーロックイン回避」の前提に反するため一括で対象外。
- **商用プラットフォーム** (OpenShift, Tanzu, Rancher Prime 等)
  → OSS 版の Rancher は採用候補だが、本表では個別 OSS を積み上げる方針のため対象外。

---

## 結論

- tier1 = Rust、実行基盤 = k8s + Istio + Envoy Gateway、可観測性 = OTel + Jaeger + Loki + Prometheus、メッセージング = Kafka の組み合わせを採用する。
- いずれも CNCF Graduated または広く採用されている OSS に限定し、特定ベンダーに依存しない構成とする。
- 将来的な差し替え (Jaeger → Tempo、Kafka → NATS 等) を想定し、OpenTelemetry のような抽象層や tier1 クライアントライブラリを挟むことで退路を確保する。
