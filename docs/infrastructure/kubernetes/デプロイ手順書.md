# デプロイ/ロールバック手順書

本番リリースの手順と障害時のロールバック方法を定義する。

関連設計: [CI/CD設計](../cicd/CI-CD設計.md) / [プログレッシブデリバリー設計](../../architecture/deployment/プログレッシブデリバリー設計.md)

## 1. 通常デプロイフロー

### 1-1. 自動デプロイ（dev/staging）

main ブランチへのマージで GitHub Actions が自動実行される。

```
build-and-push
  └─→ deploy-dev（自動、self-hosted: dev）
        └─→ deploy-staging（自動、self-hosted: staging）
              └─→ deploy-prod（手動承認待ち）
```

各ステップで実行されるコマンド（`_service-deploy.yaml` より）:

```bash
# Helm upgrade（--atomic: 失敗時は自動ロールバック、--wait: Pod Ready まで待機）
helm upgrade --install <service-name> \
  ./infra/helm/services/<helm-chart-path> \
  -n <namespace> \
  -f ./infra/helm/services/<helm-chart-path>/values-<env>.yaml \
  --set image.tag=<version>-<sha> \
  --atomic --wait --timeout 5m
```

### 1-2. prod 手動承認

1. GitHub Actions の workflow 画面で `deploy-prod` ジョブが **Waiting** 状態になる
2. `Settings → Environments → prod → Required reviewers` に登録された **2名以上** が承認する
3. 承認後、自動的に prod デプロイが開始される
4. デプロイ後スモークテスト（`/healthz` への疎通確認）が成功すれば完了

### 1-3. デプロイ状態の確認

```bash
# Helm リリース一覧
helm list -n k1s0-system
helm list -n k1s0-business
helm list -n k1s0-service

# デプロイ履歴の確認
helm history <release-name> -n <namespace>

# Pod の稼働状態
kubectl get pods -n <namespace> -l app=<service-name>

# Rollout 状態（Argo Rollouts 使用時）
kubectl argo rollouts get rollout <service-name> -n <namespace>
```

## 2. ロールバック手順

### 判断フロー

```
デプロイ後に問題発生
  │
  ├─ --atomic フラグで自動ロールバック済み？
  │    Yes → Helm が自動的に前バージョンへ戻している（手順不要）
  │    No ↓
  │
  ├─ Argo Rollouts によるカナリアデプロイ中？
  │    Yes → 「2-2. Argo Rollouts ロールバック」へ
  │    No ↓
  │
  └─ 既に prod デプロイ完了済み？
       Yes → 「2-1. Helm 手動ロールバック」または「2-3. 緊急ロールバック」へ
```

### 2-1. Helm 手動ロールバック（推奨）

```bash
# 1. リビジョン履歴を確認する
helm history <release-name> -n <namespace>
# 例:
# REVISION  STATUS      CHART           APP VERSION
# 1         superseded  auth-0.1.0      1.2.3
# 2         deployed    auth-0.1.0      1.2.4  ← 問題のあるバージョン

# 2. ロールバックを実行する（前リビジョンへ戻す）
helm rollback <release-name> <revision> -n <namespace> --wait --timeout 5m
# 例: helm rollback auth 1 -n k1s0-system --wait --timeout 5m

# 3. ロールバック完了を確認する
kubectl rollout status deployment/<service-name> -n <namespace>
helm list -n <namespace>  # REVISION が増加し STATUS が deployed になることを確認
```

### 2-2. Argo Rollouts ロールバック

```bash
# カナリア進行を中止する（stable 版へ即時切り戻し）
kubectl argo rollouts abort <rollout-name> -n <namespace>

# 前バージョンへ戻す（abort 後に安定版へ確実に切り戻す）
kubectl argo rollouts undo <rollout-name> -n <namespace>

# 状態を確認する
kubectl argo rollouts get rollout <rollout-name> -n <namespace> --watch
```

### 2-3. 緊急ロールバック（kubectl rollout undo）

Helm が使用できない緊急時のみ使用。Helm の State と乖離するため後でsync が必要。

```bash
# Deployment を前リビジョンへ戻す
kubectl rollout undo deployment/<service-name> -n <namespace>

# 特定リビジョンへ戻す場合
kubectl rollout history deployment/<service-name> -n <namespace>
kubectl rollout undo deployment/<service-name> -n <namespace> --to-revision=<N>

# 状態確認
kubectl rollout status deployment/<service-name> -n <namespace>
```

**注意**: `kubectl rollout undo` 使用後は必ず Helm の State を修正すること。

```bash
# Helm State を現状に合わせて再sync する
helm upgrade <release-name> ./infra/helm/services/<path> \
  -n <namespace> \
  -f ./infra/helm/services/<path>/values-prod.yaml \
  --set image.tag=<stable-version> \
  --reuse-values
```

## 4. デプロイ後確認チェックリスト

```bash
# Pod が全て Running であること
kubectl get pods -n <namespace>

# エラーログがないこと（直近5分）
# Grafana → Explore → Loki: {app="<service-name>", level="error"} | 5m

# ヘルスチェックエンドポイントが正常応答すること
# kubectl exec deploy/<service-name> -n <namespace> -- /busybox wget -qO- http://127.0.0.1:8080/healthz

# SLO バーンレートが正常範囲内であること
# Grafana: SLO ダッシュボード → 該当サービス
```
