# デプロイ/ロールバック手順書

本番リリースの手順と障害時のロールバック方法を定義する。

関連設計: [CI/CD設計](../cicd/CI-CD設計.md) / [プログレッシブデリバリー設計](../../architecture/deployment/プログレッシブデリバリー設計.md)

## 1. 通常デプロイフロー

### 1-1. 自動デプロイ（dev/staging）

main ブランチへのマージで GitHub Actions が自動実行される。

```
build-and-push
  └─→ deploy-dev（自動、self-hosted: dev）
        └─→ deploy-staging（自動、self-hosted: staging）
              └─→ deploy-prod（手動承認待ち）
```

各ステップで実行されるコマンド（`_service-deploy.yaml` より）:

```bash
# Helm upgrade（--atomic: 失敗時は自動ロールバック、--wait: Pod Ready まで待機）
helm upgrade --install <service-name> \
  ./infra/helm/services/<helm-chart-path> \
  -n <namespace> \
  -f ./infra/helm/services/<helm-chart-path>/values-<env>.yaml \
  --set image.tag=<version>-<sha> \
  --atomic --wait --timeout 5m
```

### 1-1b. 手動デプロイ（MED-014 対応: image.tag の明示指定）

緊急対応や CI をスキップして特定バージョンを直接デプロイする場合は、
`--set image.tag=<version>` を必ず明示すること。未指定の場合は Helm の values 既定値が使われ、
意図しない古いイメージが起動する恐れがある。

```bash
# image.tag の形式: <version>-<sha12>（CI 自動デプロイと同じ形式）
# 例: 1.2.3-abc123def456
helm upgrade --install <service-name> \
  ./infra/helm/services/<helm-chart-path> \
  -n <namespace> \
  -f ./infra/helm/services/<helm-chart-path>/values-<env>.yaml \
  --set image.tag=<version>-<sha12> \
  --atomic --wait --timeout 5m

# 特定バージョンタグのみを指定する場合（SHA なし）
helm upgrade --install <service-name> \
  ./infra/helm/services/<helm-chart-path> \
  -n <namespace> \
  -f ./infra/helm/services/<helm-chart-path>/values-<env>.yaml \
  --set image.tag=<version> \
  --atomic --wait --timeout 5m
```

**注意**: `<version>` は `git describe --tags --abbrev=0 | sed 's/^v//'` で取得できる。
Harbor レジストリに対象タグが存在することを事前に確認すること。

### 1-2. prod 手動承認

1. GitHub Actions の workflow 画面で `deploy-prod` ジョブが **Waiting** 状態になる
2. `Settings → Environments → prod → Required reviewers` に登録された **2名以上** が承認する
3. 承認後、自動的に prod デプロイが開始される
4. デプロイ後スモークテスト（`/healthz` への疎通確認）が成功すれば完了

### 1-3. デプロイ状態の確認

```bash
# Helm リリース一覧
helm list -n k1s0-system
helm list -n k1s0-business
helm list -n k1s0-service

# デプロイ履歴の確認
helm history <release-name> -n <namespace>

# Pod の稼働状態
kubectl get pods -n <namespace> -l app=<service-name>

# Rollout 状態（Argo Rollouts 使用時）
kubectl argo rollouts get rollout <service-name> -n <namespace>
```

## 2. ロールバック手順

### 判断フロー

```
デプロイ後に問題発生
  │
  ├─ --atomic フラグで自動ロールバック済み？
  │    Yes → Helm が自動的に前バージョンへ戻している（手順不要）
  │    No ↓
  │
  ├─ Argo Rollouts によるカナリアデプロイ中？
  │    Yes → 「2-2. Argo Rollouts ロールバック」へ
  │    No ↓
  │
  └─ 既に prod デプロイ完了済み？
       Yes → 「2-1. Helm 手動ロールバック」または「2-3. 緊急ロールバック」へ
```

### 2-1. Helm 手動ロールバック（推奨）

```bash
# 1. リビジョン履歴を確認する
helm history <release-name> -n <namespace>
# 例:
# REVISION  STATUS      CHART           APP VERSION
# 1         superseded  auth-0.1.0      1.2.3
# 2         deployed    auth-0.1.0      1.2.4  ← 問題のあるバージョン

# 2. ロールバックを実行する（前リビジョンへ戻す）
helm rollback <release-name> <revision> -n <namespace> --wait --timeout 5m
# 例: helm rollback auth 1 -n k1s0-system --wait --timeout 5m

# 3. ロールバック完了を確認する
kubectl rollout status deployment/<service-name> -n <namespace>
helm list -n <namespace>  # REVISION が増加し STATUS が deployed になることを確認
```

### 2-2. Argo Rollouts ロールバック

```bash
# カナリア進行を中止する（stable 版へ即時切り戻し）
kubectl argo rollouts abort <rollout-name> -n <namespace>

# 前バージョンへ戻す（abort 後に安定版へ確実に切り戻す）
kubectl argo rollouts undo <rollout-name> -n <namespace>

# 状態を確認する
kubectl argo rollouts get rollout <rollout-name> -n <namespace> --watch
```

### 2-3. 緊急ロールバック（kubectl rollout undo）

Helm が使用できない緊急時のみ使用。Helm の State と乖離するため後でsync が必要。

```bash
# Deployment を前リビジョンへ戻す
kubectl rollout undo deployment/<service-name> -n <namespace>

# 特定リビジョンへ戻す場合
kubectl rollout history deployment/<service-name> -n <namespace>
kubectl rollout undo deployment/<service-name> -n <namespace> --to-revision=<N>

# 状態確認
kubectl rollout status deployment/<service-name> -n <namespace>
```

**注意**: `kubectl rollout undo` 使用後は必ず Helm の State を修正すること。

```bash
# Helm State を現状に合わせて再sync する
helm upgrade <release-name> ./infra/helm/services/<path> \
  -n <namespace> \
  -f ./infra/helm/services/<path>/values-prod.yaml \
  --set image.tag=<stable-version> \
  --reuse-values
```

## 3.5. デプロイ前チェックリスト（MED-009 対応）

本番デプロイ前に以下を必ず確認すること。

### etcd 暗号化キーの確認

etcd の Secret 暗号化設定（EncryptionConfiguration）が正しく注入されているか確認する。
etcd 暗号化キーが未設定または古いキーのまま上書きされると、
既存の暗号化済み Secrets が読み取り不能になる恐れがある（MED-009 監査対応）。

```bash
# 1. EncryptionConfiguration が kube-apiserver Pod に適用されているか確認する
kubectl get pod -n kube-system -l component=kube-apiserver -o yaml | \
  grep -A 5 "encryption-provider-config"

# 2. etcd 暗号化キー ConfigMap/Secret が最新状態か確認する
kubectl get secret etcd-encryption-config -n kube-system -o jsonpath='{.metadata.resourceVersion}'

# 3. 現在の暗号化プロバイダーで既存 Secrets が復号できることを確認する（例: vault-operator の Secrets）
kubectl get secret -n k1s0-system -o json | \
  jq '.items[].metadata.name' | head -5

# 上記コマンドが正常終了（エラーなし）なら復号できている。
# エラーが出る場合は暗号化キーのローテーション手順（別途 runbook 参照）に従うこと。
```

> **重要**: 暗号化キーのローテーション時は必ず旧キーを `decryptionKeys` に残したまま
> 新キーを `encryptionKeys` の先頭に追加すること。旧キーを削除してから新キーに切り替えると
> 既存 Secrets が失われる（非可逆的障害）。

## 4. デプロイ後確認チェックリスト

```bash
# Pod が全て Running であること
kubectl get pods -n <namespace>

# エラーログがないこと（直近5分）
# Grafana → Explore → Loki: {app="<service-name>", level="error"} | 5m

# ヘルスチェックエンドポイントが正常応答すること
# kubectl exec deploy/<service-name> -n <namespace> -- /busybox wget -qO- http://127.0.0.1:8080/healthz

# SLO バーンレートが正常範囲内であること
# Grafana: SLO ダッシュボード → 該当サービス
```

## 5. kube-system コンポーネント異常時の調査手順（K8S-MED-004 対応）

### kube-scheduler / kube-controller-manager の高頻度再起動調査

`kube-scheduler` や `kube-controller-manager` が高頻度で再起動している場合、
クラスタの安定性に直結するため、以下の手順で原因を特定すること。

```bash
# 1. kube-system の Pod 一覧と再起動回数を確認する
kubectl get pod -n kube-system

# 2. 対象 Pod の詳細イベントを確認する（OOMKill・Liveness probe 失敗など）
kubectl describe pod -n kube-system kube-scheduler-<node-name>
kubectl describe pod -n kube-system kube-controller-manager-<node-name>

# 3. ログで直近のエラーを確認する
kubectl logs -n kube-system kube-scheduler-<node-name> --previous
kubectl logs -n kube-system kube-controller-manager-<node-name> --previous

# 4. ノードリソース（CPU/メモリ）の枯渇を確認する
kubectl top node
kubectl describe node <node-name> | grep -A 10 "Conditions:"

# 5. Metrics Server が正常に稼働しているか確認する（HPA 連携）
kubectl get pod -n kube-system | grep metrics-server
kubectl logs -n kube-system deployment/metrics-server
```

### よくある原因と対処

| 原因 | 確認コマンド | 対処 |
|-----|------------|------|
| OOMKill（メモリ不足） | `kubectl describe pod ... \| grep OOMKilled` | ノードの増設またはスケジューラのリソース上限緩和 |
| Liveness probe タイムアウト | `kubectl describe pod ...` の Events | リーダー選出の遅延が原因の場合はタイムアウト値を調整 |
| etcd 接続断 | `kubectl logs ... \| grep etcd` | etcd ヘルスチェックと TLS 証明書の期限を確認 |
| TLS 証明書期限切れ | `kubeadm certs check-expiration` | `kubeadm certs renew all` で更新 |
| 静的 Pod マニフェスト変更 | `/etc/kubernetes/manifests/` の mtime 確認 | 意図しない変更がないか確認 |

> **注意**: kube-scheduler・kube-controller-manager は静的 Pod として起動するため、
> `kubectl delete pod` で削除しても kubelet が自動的に再作成する。
> 恒久的な修正は `/etc/kubernetes/manifests/` 内のマニフェストファイルを変更すること。
