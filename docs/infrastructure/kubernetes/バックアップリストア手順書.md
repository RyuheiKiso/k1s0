# バックアップ/リストア手順書

各コンポーネントのバックアップスケジュールと、障害時のリストア手順を定義する。

---

## バックアップスケジュール一覧

| コンポーネント | スケジュール (UTC) | 保持期間 | 方式 | 保存先 |
|--------------|------------------|---------|------|--------|
| PostgreSQL (dev/staging) | 毎日 03:00 | 30日 | CronJob → PVC | backup-pvc:/backup/postgres/ |
| PostgreSQL (prod) | 毎日 03:00 | S3管理 | Terraform → S3 | s3://\<backup_bucket\>/postgresql/ |
| MySQL (prod) | 毎日 03:00 | S3管理 | Terraform → S3 | s3://\<backup_bucket\>/mysql/ |
| Vault Raft | 毎日 01:00 | 30日 + S3 | CronJob → PVC + S3 | backup-pvc:/backup/vault/ + S3 |
| etcd | 毎日 02:00 | 30日 + S3 | CronJob → PVC + S3 | backup-pvc:/backup/etcd/ + S3 |
| Harbor DB | 毎週日曜 04:00 | 90日 | CronJob → PVC | backup-pvc:/backup/harbor/ |
| Consul State | 毎日（Terraform管理） | 7世代 | CronJob → S3 | Ceph RGW |

実装ファイル:
- `infra/kubernetes/backup/` — dev/staging 用 CronJob (5件)
- `infra/terraform/modules/database/backup.tf` — prod 用 PostgreSQL/MySQL
- `infra/terraform/modules/consul-backup/` — Consul スナップショット

## 1. PostgreSQL リストア手順

### 1-1. dev/staging（PVC 方式）

```bash
# 1. バックアップファイルを確認する
kubectl exec -n k1s0-system deploy/postgres-client -- \
  ls -la /backup/postgres/

# 2. リストア先の DB に接続できることを確認する
kubectl exec -n k1s0-system deploy/postgres-client -- \
  psql -h postgresql.k1s0-system.svc.cluster.local -U k1s0_user -c '\l'

# 3. リストアを実行する（pg_restore は既存テーブルへの上書きに -c オプションが必要）
kubectl exec -n k1s0-system deploy/postgres-client -- \
  pg_restore \
    -h postgresql.k1s0-system.svc.cluster.local \
    -U k1s0_user \
    -d k1s0_system \
    -Fc \
    -c \
    /backup/postgres/k1s0_system-<TIMESTAMP>.dump

# 4. 他のデータベースも同様にリストアする
# 対象 DB: k1s0_system, config_db, dlq_db
```

### 1-2. prod（S3 方式）

```bash
# 1. S3 から最新バックアップを取得する
aws s3 ls s3://<backup_bucket>/postgresql/ \
  --endpoint-url <S3_ENDPOINT> | sort | tail -5

aws s3 cp \
  s3://<backup_bucket>/postgresql/pg-<YYYYMMDD>.sql.gz \
  /tmp/pg-restore.sql.gz \
  --endpoint-url <S3_ENDPOINT>

# 2. 解凍してリストアする
gunzip -c /tmp/pg-restore.sql.gz | \
  psql -h postgresql.k1s0-system.svc.cluster.local \
       -U k1s0_user \
       -d k1s0_system

# 3. リストア完了を確認する
psql -h postgresql.k1s0-system.svc.cluster.local \
     -U k1s0_user \
     -c "SELECT count(*) FROM information_schema.tables WHERE table_schema='public';" \
     k1s0_system
```

## 2. MySQL リストア手順（prod）

```bash
# 1. S3 から取得する
aws s3 cp \
  s3://<backup_bucket>/mysql/mysql-<YYYYMMDD>.sql.gz \
  /tmp/mysql-restore.sql.gz \
  --endpoint-url <S3_ENDPOINT>

# 2. 解凍してリストアする
gunzip -c /tmp/mysql-restore.sql.gz | \
  mysql -h mysql.k1s0-system.svc.cluster.local \
        -u root \
        -p<MYSQL_ROOT_PASSWORD>
```

## 3. Vault リストア手順

### 3-1. PVC からのリストア（dev/staging）

```bash
# 1. バックアップファイルを確認する
kubectl exec -n k1s0-system vault-0 -- \
  ls -la /backup/vault/

# 2. Vault を sealed 状態にする
kubectl exec -n k1s0-system vault-0 -- vault operator seal

# 3. スナップショットをリストアする
kubectl exec -n k1s0-system vault-0 -- \
  vault operator raft snapshot restore \
    /backup/vault/vault-raft-<TIMESTAMP>.snap

# 4. Vault を unseal する（各 Master ノードで実行）
kubectl exec -n k1s0-system vault-0 -- vault operator unseal <unseal_key_1>
kubectl exec -n k1s0-system vault-0 -- vault operator unseal <unseal_key_2>
kubectl exec -n k1s0-system vault-0 -- vault operator unseal <unseal_key_3>

# 5. 状態確認
kubectl exec -n k1s0-system vault-0 -- vault status
```

### 3-2. S3 からのリストア（prod）

```bash
# 1. S3 から最新スナップショットを取得する
aws s3 ls s3://<S3_BUCKET>/vault/ \
  --endpoint-url <S3_ENDPOINT> | sort | tail -5

aws s3 cp \
  s3://<S3_BUCKET>/vault/vault-raft-<TIMESTAMP>.snap \
  /tmp/vault-restore.snap \
  --endpoint-url <S3_ENDPOINT>

# 2. スナップショットをコピーして Pod 内でリストアする
kubectl cp /tmp/vault-restore.snap k1s0-system/vault-0:/tmp/vault-restore.snap

kubectl exec -n k1s0-system vault-0 -- \
  vault operator raft snapshot restore /tmp/vault-restore.snap
```

**注意**: Vault リストア後は関連サービスが Vault に再認証が必要になる場合がある。サービスの再起動を検討すること。

## 4. etcd リストア手順

etcd のリストアはクラスタを停止して実施する。**必ず全 Master ノードで実行すること。**

```bash
# 1. S3 から最新スナップショットを取得する
aws s3 cp \
  s3://<S3_BUCKET>/etcd/etcd-snapshot-<TIMESTAMP>.db \
  /tmp/etcd-restore.db \
  --endpoint-url <S3_ENDPOINT>

# 2. スナップショットを検証する
ETCDCTL_API=3 etcdctl snapshot status /tmp/etcd-restore.db --write-out=table

# 3. 全 Master ノードで kube-apiserver を停止する
# /etc/kubernetes/manifests/kube-apiserver.yaml を一時的に退避する
mv /etc/kubernetes/manifests/kube-apiserver.yaml /tmp/

# 4. 各 Master ノードで etcd をリストアする（ノードごとに initial-cluster-token を変更すること）
ETCDCTL_API=3 etcdctl snapshot restore /tmp/etcd-restore.db \
  --name=master-1 \
  --initial-cluster="master-1=https://10.0.200.11:2380,master-2=https://10.0.200.12:2380,master-3=https://10.0.200.13:2380" \
  --initial-cluster-token=etcd-cluster-restored \
  --initial-advertise-peer-urls=https://10.0.200.11:2380 \
  --data-dir=/var/lib/etcd-restore

# 5. 既存の etcd データを退避し、リストアデータを適用する
mv /var/lib/etcd /var/lib/etcd.bak
mv /var/lib/etcd-restore /var/lib/etcd

# 6. kube-apiserver を復元して再起動する
mv /tmp/kube-apiserver.yaml /etc/kubernetes/manifests/

# 7. クラスタ正常性を確認する
kubectl get nodes
kubectl get pods -n kube-system
```

## 5. Harbor リストア手順（dev/staging）

```bash
# 1. バックアップファイルを確認する
kubectl exec -n k1s0-system deploy/harbor-client -- \
  ls -la /backup/harbor/

# 2. Harbor サービスを停止する（DBマイグレーション中の書き込み防止）
kubectl scale deployment harbor-core -n harbor --replicas=0

# 3. Harbor DB にリストアする
kubectl exec -n k1s0-system deploy/harbor-client -- \
  pg_restore \
    -h harbor-database.harbor.svc.cluster.local \
    -U postgres \
    -d registry \
    -Fc \
    -c \
    /backup/harbor/harbor-db-<TIMESTAMP>.dump

# 4. Harbor サービスを再起動する
kubectl scale deployment harbor-core -n harbor --replicas=1
kubectl rollout status deployment/harbor-core -n harbor
```

## 6. Consul リストア手順（Terraform State 復旧）

Terraform State ロックが残留したり State が破損した場合に使用する。

```bash
# 1. S3 から最新スナップショットを取得する
aws s3 ls s3://<S3_BUCKET>/consul-backup/ \
  --endpoint-url <S3_ENDPOINT> | sort | tail -5

aws s3 cp \
  s3://<S3_BUCKET>/consul-backup/<latest-snapshot>.snap \
  /tmp/consul-restore.snap \
  --endpoint-url <S3_ENDPOINT>

# 2. スナップショットをリストアする
export CONSUL_HTTP_ADDR="https://consul.internal.example.com:8500"
export CONSUL_HTTP_TOKEN="<token>"  # Consul ACL トークン

consul snapshot restore /tmp/consul-restore.snap

# 3. リストア後に Terraform の State 整合性を確認する
cd infra/terraform/environments/prod
terraform plan  # 差分が最小であることを確認する

# 4. 差分がある場合は内容を確認してから適用する
# 自動で発生した差分と意図的な差分を区別すること
```

## 7. 定期リストアテスト手順（staging 環境）

**目的**: バックアップからのリストアが実際に機能することを定期的に確認する

**頻度**: 四半期に1回

```bash
# 1. staging 環境のバックアップから一時 DB を起動する
kubectl run restore-test --image=postgres:17-alpine \
  -n k1s0-system --restart=Never \
  --env="PGPASSWORD=<password>" \
  -- sleep 3600

# 2. バックアップをリストアする
kubectl exec -n k1s0-system restore-test -- \
  pg_restore -h postgresql.k1s0-system.svc.cluster.local \
             -U k1s0_user -d k1s0_system_restore -Fc \
             /backup/postgres/<latest>.dump

# 3. 基本クエリで整合性を確認する
kubectl exec -n k1s0-system restore-test -- \
  psql -h postgresql.k1s0-system.svc.cluster.local \
       -U k1s0_user -d k1s0_system_restore \
       -c "SELECT count(*) FROM users; SELECT count(*) FROM sessions;"

# 4. テスト後のクリーンアップ
kubectl delete pod restore-test -n k1s0-system
```
