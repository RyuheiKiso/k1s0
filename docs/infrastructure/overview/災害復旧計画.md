# 災害復旧（DR）計画

本番環境の重大障害時に秩序立てて復旧するための計画を定義する。

関連ドキュメント: [バックアップリストア手順書](../kubernetes/バックアップリストア手順書.md) / [デプロイ手順書](../kubernetes/デプロイ手順書.md) / [Terraform運用手順書](../terraform/Terraform運用手順書.md)

---

## RPO / RTO 目標

| コンポーネント | RPO (目標復旧時点) | RTO (目標復旧時間) | 優先度 |
|--------------|------------------|------------------|--------|
| etcd | 2時間 | 2時間 | 最高 |
| PostgreSQL (system DB) | 24時間 | 4時間 | 高 |
| Vault | 24時間 | 2時間 | 高 |
| Consul (TF State) | 24時間 | 1時間 | 高 |
| PostgreSQL (業務 DB) | 24時間 | 6時間 | 中 |
| Harbor | 1週間 | 8時間 | 中 |
| Kafka | N/A（揮発）| 30分 | 高 |

---

## 復旧優先順位

クラスタ障害後は以下の順序で復旧する。各ステップが完了してから次へ進む。

```
1. クラスタ基盤（etcd → Master ノード → Worker ノード）
2. 共通インフラ（Vault → Consul → Cert-Manager）
3. データ層（PostgreSQL → MySQL → Kafka → Harbor）
4. system tier（auth → config → その他 system サービス）
5. business tier（domain-master）
6. service tier（order → inventory → payment）
7. ゲートウェイ（Kong → Nginx Ingress）
8. 可観測性スタック（Prometheus → Grafana → Loki）
```

**根拠**: system tier が認証・設定を提供するため、business/service より先に復旧する必要がある。Vault は JWT 署名キーを管理するため auth より先に復旧が必要。

## 障害シナリオ別復旧手順

### シナリオ A: 単一 Worker ノード障害

**症状**: Pod が Pending 状態になる、特定ノードの Pod が落ちる

```bash
# 1. 障害ノードを確認する
kubectl get nodes

# 2. 障害ノードを drain する（Kubernetes が自動的に他ノードへ Pod を再スケジュール）
kubectl drain <node-name> --ignore-daemonsets --delete-emptydir-data

# 3. ノードを修復・再起動する（Ansible で実行）
ansible-playbook -i inventory/prod infra/ansible/playbooks/node-restart.yml \
  --limit <node-name>

# 4. ノードを復旧後、uncordon する
kubectl uncordon <node-name>

# 5. Pod が正常に再スケジュールされたことを確認する
kubectl get pods -o wide | grep <node-name>
```

**所要時間目安**: 30分〜1時間

---

### シナリオ B: Master ノード障害（1台）

prod は 3 Master の HA 構成のため、1台障害でもクラスタは稼働継続する。

```bash
# 1. etcd クラスタの健全性を確認する（残り 2 台の Master で確認）
kubectl exec -n kube-system etcd-<healthy-master> -- \
  etcdctl --endpoints=https://127.0.0.1:2379 \
          --cacert=/etc/kubernetes/pki/etcd/ca.crt \
          --cert=/etc/kubernetes/pki/etcd/healthcheck-client.crt \
          --key=/etc/kubernetes/pki/etcd/healthcheck-client.key \
          endpoint health

# 2. 障害 Master を etcd クラスタから除外する
kubectl exec -n kube-system etcd-<healthy-master> -- \
  etcdctl --endpoints=https://127.0.0.1:2379 \
          --cacert=... --cert=... --key=... \
          member remove <member-id>

# 3. 障害 Master を修復後、Ansible で再参加させる
ansible-playbook -i inventory/prod infra/ansible/playbooks/master-rejoin.yml \
  --limit <failed-master>

# 4. クラスタに再参加したことを確認する
kubectl get nodes
```

**所要時間目安**: 1〜2時間

---

### シナリオ C: 全クラスタ障害（壊滅的障害）

etcd が完全に失われた場合など。**最も深刻なシナリオ。**

```bash
# 1. 全 Master ノードを停止する
# 2. etcd スナップショットから復旧する（バックアップリストア手順書を参照）
# → 「4. etcd リストア手順」を参照

# 3. Master ノードを順番に再起動する
ansible-playbook -i inventory/prod infra/ansible/playbooks/cluster-restart.yml

# 4. Worker ノードを再起動する
ansible-playbook -i inventory/prod infra/ansible/playbooks/worker-restart.yml

# 5. Vault を unseal する（各 Master の Vault Pod）
kubectl get pods -n k1s0-system -l app=vault
kubectl exec -n k1s0-system vault-0 -- vault operator unseal <key1>
kubectl exec -n k1s0-system vault-0 -- vault operator unseal <key2>
kubectl exec -n k1s0-system vault-0 -- vault operator unseal <key3>

# 6. システムサービスの復旧状態を確認する
kubectl get pods -n k1s0-system
kubectl get pods -n k1s0-business
kubectl get pods -n k1s0-service
```

**所要時間目安**: 4〜8時間

---

### シナリオ D: Ceph ストレージ障害

PVC が使用不能になった場合。

```bash
# 1. Ceph クラスタの健全性を確認する
ceph health detail
ceph osd status
ceph df

# 2. 障害 OSD を確認する
ceph osd out <osd-id>  # 障害OSDを「out」にする
ceph osd down <osd-id>

# 3. データが他の OSD に再配置されるまで待つ（HEALTH_WARN → HEALTH_OK）
watch ceph -s

# 4. 障害OSDを交換後、新 OSD を追加する
ceph osd in <new-osd-id>

# 5. データ整合性を確認する
ceph osd repair <pg-id>
```

**所要時間目安**: ストレージ交換含め2〜24時間（データ量による）

---

### シナリオ E: Vault 障害

JWT 署名キーが取得できず全認証が失敗するため、**最優先で復旧する。**

```bash
# バックアップリストア手順書の「3. Vault リストア手順」を参照

# Vault 復旧後、依存サービスを再起動する
kubectl rollout restart deployment/auth -n k1s0-system
kubectl rollout restart deployment/config -n k1s0-system
```

## 連絡体制

| フェーズ | 通知先 | 手段 |
|---------|--------|------|
| 障害検知 | インフラチーム全員 | Microsoft Teams #alert-critical（自動） |
| SEV1 宣言 | インフラリード + サービスオーナー | 電話・Teams |
| 復旧着手 | 開発チームリード | Teams #incident |
| 復旧完了 | 全関係者 | Teams #announce |
| ポストモーテム | 全チーム | 72時間以内に開催 |

## DR 演習計画

**頻度**: 四半期に1回（3月・6月・9月・12月の第2水曜日）

**演習内容**:

| 演習 | 対象環境 | 内容 |
|-----|---------|------|
| バックアップリストアテスト | staging | PostgreSQL のリストアが成功することを確認（毎月） |
| etcd 障害シミュレーション | staging | Worker ノード1台を意図的に停止、自動復旧を確認 |
| 全手順のドライラン | staging | この計画書の手順を実際に辿り、コマンドが動作することを確認 |
| Vault フェイルオーバー | staging | Vault を停止し、サービスへの影響と復旧を確認 |

**演習後のアクション**:
1. 手順書のコマンドを最新状態に更新する
2. RTO 目標を実績値で見直す
3. ポストモーテムテンプレートで改善点を記録する
