# ゲートウェイ責務分離

k1s0 の通信経路は 3 層のゲートウェイで構成される。各層の責務を理解することで、設定変更の判断と障害調査が正確に行える。

---

## トラフィックフロー

```
クライアント（モバイル / Web / CLI）
  │
  ▼
[Nginx Ingress Controller]  ← 外部インターネット境界
  │  TLS 終端・外部 LB・L7 ルーティング
  ▼
[Kong API Gateway]          ← API 管理層
  │  認証・認可・レート制限・リクエスト変換
  ▼
[Istio Sidecar (Envoy)]     ← サービスメッシュ層
  │  mTLS・サービス間認可・トラフィック制御
  ▼
Backend Service（auth / config / order など）
```

---

## 各層の責務

### 1. Nginx Ingress Controller

**責務**: 外部トラフィックをクラスタ内に受け入れる

| 機能 | 詳細 |
|------|------|
| TLS 終端 | 外部クライアントとの HTTPS 接続を終端する。内部は HTTP に変換 |
| L7 ロードバランシング | ホスト名・パスプレフィックスで Kong へルーティング |
| MetalLB との統合 | `LoadBalancer` type Service に外部 IP を付与（オンプレ環境） |
| レート制限（粗粒度） | DoS 対策として IP ベースの粗粒度レート制限（Kong の前段） |

**設定ファイル**: `infra/kubernetes/ingress/`

**この層で設定すること**:
- 外部ドメインの TLS 証明書（cert-manager）
- ホスト名ベースのルーティング
- クライアント IP 保持（`X-Forwarded-For` ヘッダー）

### 2. Kong API Gateway

**責務**: API の横断的な関心事を一元管理する

| 機能 | 詳細 |
|------|------|
| JWT 認証 | Keycloak 発行の JWT を検証し、認証されていないリクエストを拒否 |
| レート制限（細粒度） | ユーザー・サービス・Tier 別のレート制限（Redis 連携） |
| リクエスト変換 | ヘッダー追加・削除、リクエストボディ変換 |
| ルーティング | Service/Route 設定でバックエンドへのルーティングを定義 |
| プラグイン | CORS・ログ・メトリクス等のクロスカッティングな機能 |

**設定ファイル**: `infra/kong/`、`infra/helm/services/system/kong/`

**この層で設定すること**:
- 認証が必要なルートの定義
- API ごとのレート制限値
- CORS 設定
- リクエスト・レスポンスの変換

### 3. Istio Sidecar (Envoy)

**責務**: サービス間通信のセキュリティとトラフィック制御

| 機能 | 詳細 |
|------|------|
| mTLS | サービス間通信をすべて相互 TLS で暗号化・認証 |
| AuthorizationPolicy | L7 レベルのサービス間認可（HTTP メソッド・パス・ヘッダーで制御） |
| トラフィック分割 | VirtualService / DestinationRule でカナリア・A/B テスト |
| リトライ・タイムアウト | サービス間通信の信頼性向上 |
| サーキットブレーカー | 障害の連鎖を防止 |

**設定ファイル**: `infra/istio/`、`infra/helm/charts/k1s0-common/templates/_istio-resources.tpl`

**この層で設定すること**:
- サービス間の通信許可・拒否ルール（AuthorizationPolicy）
- カナリアデプロイのトラフィック比率
- Tier 間の通信制御（system → business → service の一方向性）

---

## 設定変更の判断フロー

```
変更したい内容は何か？
  │
  ├─ 外部ドメイン・TLS 証明書・ホスト名ルーティング
  │    → Nginx Ingress の設定 (infra/kubernetes/ingress/)
  │
  ├─ 認証の有無・レート制限値・CORS・リクエスト変換
  │    → Kong の設定 (infra/kong/)
  │
  ├─ サービス間の通信許可・拒否
  │    → Istio AuthorizationPolicy (infra/istio/authorizationpolicy.yaml)
  │
  ├─ カナリアデプロイ・A/B テストのトラフィック比率
  │    → Istio VirtualService / DestinationRule (infra/istio/)
  │         または Argo Rollouts (Rollout CRD)
  │
  └─ L3/L4 レベルの Namespace 間通信許可
       → Kubernetes NetworkPolicy (infra/kubernetes/network-policies/)
```

---

## NetworkPolicy と AuthorizationPolicy の使い分け

k1s0 では L3/L4 と L7 の 2 層でネットワーク制御を行う。

| 制御層 | リソース | 対象 | 粒度 |
|-------|----------|------|------|
| L3/L4 | Kubernetes NetworkPolicy | Namespace / Pod 間の IP/ポート | IP・ポート・ラベルセレクタ |
| L7 | Istio AuthorizationPolicy | HTTP メソッド・パス・ヘッダー | HTTP リクエスト全体 |

**ルール**:
- **Namespace 間の通信可否** → NetworkPolicy で制御（`infra/kubernetes/network-policies/`）
- **HTTP レベルの認可** → AuthorizationPolicy で制御（`infra/istio/`）
- 両方に設定する必要はなく、重複設定は避ける

---

## Tier 別の通信制御

k1s0 の Tier 依存方向は **一方向** に強制される（system が最上位、service が最下位）。

```
system tier（auth, config, saga 等）
  ↑ system は business/service を呼ばない
business tier（project-master 等）
  ↑ business は service を呼ばない（原則）
service tier（task, board, activity 等）
```

この制約は以下の 2 層で実施される:
- **NetworkPolicy** (`infra/kubernetes/network-policies/`): IP レベルで強制
- **AuthorizationPolicy** (`infra/istio/authorizationpolicy.yaml`): HTTP レベルで強制

## 障害時の切り分け手順

```bash
# 1. Nginx まで届いているか確認する
curl -v https://<external-domain>/healthz

# 2. Kong まで届いているか確認する（クラスタ内から）
kubectl exec -n k1s0-system deploy/debug -- \
  curl -v http://kong.k1s0-system.svc.cluster.local:8000/healthz

# 3. Backend Service まで届いているか確認する（Kong バイパス）
kubectl exec -n k1s0-system deploy/debug -- \
  curl -v http://auth.k1s0-system.svc.cluster.local:8080/healthz

# 4. Istio サイドカーを経由しているか確認する（mTLS チェック）
kubectl exec -n k1s0-system deploy/bff-proxy -c istio-proxy -- \
  pilot-agent request GET /stats | grep downstream_cx_active
```

## Kong JWT 設定の初期化手順（ローカル開発環境）（C-2 対応）

Kong の DBless モードでは `kong.dev.yaml` に JWT 検証用の RSA 公開鍵を直接記述する必要がある。
Keycloak は起動のたびにキーペアを生成するため、初回セットアップ時またはキーローテーション後に以下の手順を実行すること。

### 自動セットアップ（推奨）

```bash
# 1. infra プロファイルを起動（Keycloak が healthy になるまで待機）
docker compose --profile infra up -d

# 2. Keycloak が起動したら公開鍵を自動取得して kong.dev.yaml を更新する
./infra/kong/setup-kong-jwt.sh

# 3. Kong を再起動して新しい設定を反映する
docker compose --profile infra restart kong
```

スクリプト `infra/kong/setup-kong-jwt.sh` は以下を自動実行する:
1. Keycloak の JWKS エンドポイント（`http://localhost:8180/realms/k1s0/protocol/openid-connect/certs`）から RSA 公開鍵を取得
2. JWK 形式（n, e パラメータ）から PEM 形式（`BEGIN PUBLIC KEY`）に変換
3. `kong.dev.yaml` のプレースホルダーを実際の公開鍵に置換

> **注意**: 公開鍵が既に設定済みの場合（プレースホルダーが存在しない場合）はスキップされる（べき等実行）。
