# データベースマイグレーション設計プロセス

## 概要
データベースマイグレーションは本番データに直接影響するため、設計・レビュー・実行の各フェーズで厳格なプロセスを適用する。

## マイグレーションファイルの命名規則
```
{連番3桁}_{説明}.{up|down}.sql
例: 007_add_index_to_users_email.up.sql
    007_add_index_to_users_email.down.sql
```

## 設計チェックリスト

### 作成前（設計フェーズ）
- [ ] **後方互換性確認**: デプロイ中（旧バージョンと新バージョンが同時稼働）に問題が生じないか
- [ ] **ダウンタイム評価**: ALTER TABLEなど長時間ロックを伴う操作はないか
- [ ] **データ量見積もり**: 大量データに対する操作（FULL TABLE SCAN等）は許容できるか
- [ ] **ロールバック計画**: down.sql が安全に実行できるか（データロストなし）
- [ ] **外部キー整合性**: CASCADE/RESTRICT の選択は正しいか（デフォルトは RESTRICT）

### レビュー前
- [ ] up.sql と down.sql がペアで作成されている
- [ ] down.sql を実行することで up.sql 実行前の状態に戻せることを確認
- [ ] 大量データ操作の場合: 本番相当データでの実行時間見積もりが記載されている
- [ ] 既存データの変換を含む場合: 変換クエリの冪等性が確保されている
- [ ] 事前確認クエリ（`-- VERIFY:` コメント）が記載されている（オーナーID変換等）

### 本番適用前（オペレーションフェーズ）
- [ ] ステージング環境での適用完了（本番相当データで検証）
- [ ] バックアップが取得済み（またはスナップショット）
- [ ] メンテナンスウィンドウが確保されている（必要な場合）
- [ ] ロールバック手順を口頭で説明できる

## 破壊的変更の対処パターン

### カラム削除
```sql
-- 手順1: アプリケーションコードから該当カラムへの参照を全て削除してデプロイ
-- 手順2: カラムを NOT NULL から NULL 許容に変更（データ保護）
-- 手順3: 数週間後にカラムを削除
```

### カラム名変更
```sql
-- 手順1: 新カラムを追加
-- 手順2: トリガーで旧カラムと新カラムを同期
-- 手順3: アプリケーションを新カラム使用に更新してデプロイ
-- 手順4: トリガー削除 + 旧カラム削除
```

### NOT NULL 制約追加（データが存在する場合）
```sql
-- 手順1: DEFAULT値付きでカラム追加 or UPDATE で既存データを埋める
-- 手順2: 全行にNULL以外の値が設定されたことを確認（VERIFY クエリ）
-- 手順3: NOT NULL 制約を追加
```

## 外部キー制約のデフォルト

**デフォルトは RESTRICT（CASCADE は禁止）**

理由: CASCADE は予期しない連鎖削除を引き起こし、本番インシデントの原因となる。
削除を連鎖させる必要がある場合は、アプリケーション層で明示的に実装する。

```sql
-- ✓ 推奨
FOREIGN KEY (definition_id) REFERENCES workflow_definitions(id) ON DELETE RESTRICT

-- ✗ 禁止（特別な承認なしに使用しない）
FOREIGN KEY (definition_id) REFERENCES workflow_definitions(id) ON DELETE CASCADE
```

## レビュー基準

### 必須レビュアー
- **通常マイグレーション**: チームメンバー1名
- **既存データ変換あり**: テックリード + DBオーナー
- **スキーマ変更（テーブル追加/削除）**: テックリード + DBオーナー
- **本番データ修正**: テックリード + DBオーナー + セキュリティ担当
