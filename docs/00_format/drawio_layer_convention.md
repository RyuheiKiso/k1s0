# drawio 図解レイヤ記法規約

本ドキュメントは、k1s0 リポジトリにおける drawio 図解の**レイヤ表現規約**を定義する。
同じビジュアル (サイドカー等) で責務レイヤの異なるコンポーネントを描くと、読み手が論理的差異を視覚的同一性に覆い隠されて誤解する (例: Dapr サイドカー vs Istio サイドカー)。本規約は色・線種・配置の 3 軸で責務レイヤを常に明示的に分離することを強制する。

---

## 背景

k1s0 の初期版 `docs/01_企画/img/全体構成図.drawio` において、Dapr サイドカー (アプリから明示呼び出し) と Istio サイドカー (ネットワーク層で透過処理) を同じ「Pod 内の青/赤箱」として描いたため、責務レイヤの差異が読み手に伝わらなかった。本規約はこの再発を防止する。

関連:

- ADR-0001 (Istio Ambient Mesh 採用) — サイドカー競合の構造的解消
- ADR-0002 (本規約採用の意思決定記録)
- `CLAUDE.md` — drawio 全般規約 (矢印重なり禁止 / 白背景 / 等)

---

## 4 レイヤの定義

k1s0 のコンポーネントは以下の 4 レイヤに分類する。**レイヤが異なるコンポーネントは、必ず色・配置・線種のいずれかで視覚的に区別する。**

| レイヤ | 定義 | k1s0 での例 |
|--------|------|-------------|
| アプリ層 | アプリが**明示的に呼び出す**抽象。アプリから見える層 | tier3 業務アプリ / tier2 ドメインサービス / tier1 ファサード / Dapr (daprd) / k1s0 Client ライブラリ |
| ネットワーク層 | アプリから**透過的に効く**ネットワーク機構。アプリから見えない層 | Istio Ambient (ztunnel / waypoint / istiod) / Envoy Gateway / MetalLB / kube-vip / CoreDNS |
| インフラ層 | Kubernetes とノード自体の実行基盤 | kubelet / CRI / CNI / Node OS / etcd / Control Plane |
| データ層 | 永続化 / メッセージング / シークレット等の状態保持コンポーネント | PostgreSQL / Valkey / Kafka / MinIO / OpenBao / Longhorn |

運用基盤 (Argo CD / Backstage / Grafana / Keycloak 等) は原則「アプリ層の管理プレーン」として扱い、暖色系で描く。ただし Keycloak がネットワーク層の認可 (ext_authz) に効く場面など、文脈で責務が変わる場合は**文脈における責務**で色を選ぶ。

## 色システム

### レイヤ基本色

| レイヤ | 塗色 (`fillColor`) | 枠色 (`strokeColor`) | 用途 |
|--------|-------------------|----------------------|------|
| アプリ層 | `#fff2cc` (薄暖色) | `#d79b00` (暖色) | コンポーネント本体 |
| ネットワーク層 | `#dae8fc` (薄寒色) | `#6c8ebf` (寒色) | コンポーネント本体 |
| インフラ層 | `#f5f5f5` (中性灰) | `#666666` (灰) | コンポーネント本体 |
| データ層 | `#e1d5e7` (薄紫) | `#9673a6` (紫) | コンポーネント本体 |

### namespace / 領域境界色 (背景)

namespace や領域を示す大きな背景矩形には、上記より**さらに薄い色**を使う (例: `fillColor=#fff8e6` / `strokeColor=#d6b656`)。コンポーネント本体との対比で境界を読みやすくする。

### 状態色 (補助)

| 状態 | 色 | 用途 |
|------|-----|------|
| 成功・正常 | `#e8f5e9` / `#2e7d32` | Chaos 実験の成功パス等 |
| 失敗・警告 | `#ffebee` / `#c62828` | Chaos 実験の失敗パス / 禁止関係 |
| 注記・強調 | `#fff3e0` / `#ef6c00` | CI/CD パイプライン / サンプルフロー |

### 禁止

- **白文字を白/薄背景に使わない**。fontColor が省略された場合は `#000000` とみなす。
- **白矢印は禁止** (CLAUDE.md 規約)。
- **同一レイヤ内で色を散らばせない**。塗色のバリエーションは「濃淡」でのみ許容 (例: アプリ層本体は `#fff2cc`、アプリ層の子要素は `#fffbe6`)。

## 線種システム

矢印は呼び出し関係の**性質**を線種で区別する。

| 線種 | スタイル (`drawio style`) | 意味 |
|------|--------------------------|------|
| 実線太 (2.0 以上) | `strokeWidth=2` (dashed なし) | 明示的な同期呼び出し (gRPC / HTTP / SDK) |
| 実線細 (1.0〜1.5) | `strokeWidth=1.5` (dashed なし) | 内部補助ルーティング / tier1 内部呼び出し |
| 点線細 | `dashed=1;dashPattern=1 2` | 透過捕捉 (iptables / eBPF / ztunnel が Pod トラフィックを捕捉) |
| 破線 (橙) | `dashed=1;strokeColor=#ef6c00` | CI/CD パイプライン / GitOps デプロイ |
| 破線 (緑) | `dashed=1;strokeColor=#2e7d32` | GitOps sync / 承認済みデプロイ |
| 破線 (紫) | `dashed=1;strokeColor=#9673a6` | 非同期イベント (Kafka / PubSub) |
| 双方向太実線 | `startArrow=classic;endArrow=classic;strokeWidth=2` | 双方向の同期通信 |

### レイヤ跨ぎ矢印の太さ

- レイヤ内 (同レイヤ内の通信): 細線 (strokeWidth=1.5)
- レイヤ跨ぎ (上位レイヤ → 下位レイヤへの呼び出し): 太線 (strokeWidth=2)
- ネットワーク層の透過捕捉 (Pod → ztunnel): 点線細で表現 (「アプリが呼んでいるのではなく、透過的に捕捉されている」ことを明示)

## レーン構造 (swim lane)

大規模な全体構成図では、**水平 4 レーン** (アプリ層 / ネットワーク層 / インフラ層 / データ層) の swim lane を基本骨格とする。

```text
┌────────────────────────────────────────────────┐
│ アプリ層 (暖色)     tier3 / tier2 / tier1 / daprd │
├────────────────────────────────────────────────┤
│ ネットワーク層 (寒色) ztunnel / waypoint / Envoy   │
├────────────────────────────────────────────────┤
│ インフラ層 (灰)      Node / kubelet / CNI / etcd   │
├────────────────────────────────────────────────┤
│ データ層 (紫)        Postgres / Kafka / Valkey / MinIO │
└────────────────────────────────────────────────┘
```

### ルール

1. **レーン境界は横一直線**に引く (厳密な分離)
2. 各レーンの**左端**にレーン名と色サンプルを配置する
3. レーンを跨ぐコンポーネントは存在してはならない。複数レイヤにまたがる責務を持つ場合は、**各レイヤのビューを別枠で描く** (例: Keycloak は「ネットワーク層認可 (ext_authz)」と「アプリ層 SSO」で 2 箇所に現れてよい。ただし同じ名前で参照できること)
4. 矢印は上位レーンから下位レーンへは**垂直方向**、同一レーン内は**水平方向**を基本とする

### 小規模図での簡易運用

レーン構造が過剰な場合 (個別フロー図など) は、**色システムと線種システムのみ適用**してよい。ただし図内に必ず**凡例ブロック**を置き、どの色がどのレイヤを表しているかを明示する。

---

## 凡例ブロックの規約

すべての drawio に**凡例ブロック** (`legend`) を必ず配置する。

### 配置

- **右下固定** (`x = pageWidth - 480 - 60`, `y = pageHeight - 200 - 60` を目安)
- 全図で**同一サイズ** (幅 480 / 高さ 100〜180)
- 枠色 `#666666` / 塗色 `#f5f5f5` (中性灰で目立たない)

### 最低記載項目

1. レイヤ色サンプル (使ったレイヤのみ / 暖色=アプリ / 寒色=ネットワーク / 灰=インフラ / 紫=データ)
2. 線種サンプル (使った線種のみ / 実線=明示呼び出し / 点線=透過捕捉 / 破線=非同期)
3. 特殊記号の説明 (使用時のみ / 例: 橙◯=ステップ番号)

### 文字サイズ

- タイトル (「凡例」): `fontSize=11;fontStyle=1`
- 本文: `fontSize=10`

## チェックリスト (drawio 作成・更新時)

新規作成または既存更新の前に以下を確認すること。

1. [ ] 登場するコンポーネントを 4 レイヤに分類した
2. [ ] レイヤごとに本規約の色システムを適用した
3. [ ] 同一ビジュアル (同じ形・色) で違うレイヤのコンポーネントを並べていない
4. [ ] 矢印の線種がコンポーネント間関係の性質を正確に表している (明示呼び出し / 透過捕捉 / 非同期)
5. [ ] 凡例ブロックを右下に配置し、レイヤ色と線種の説明を記載した
6. [ ] CLAUDE.md の drawio 規約 (白矢印禁止 / 矢印重なり禁止 / 白背景) を遵守した
7. [ ] SVG エクスポート後、第三者視点で「どのコンポーネントがどのレイヤか」を色だけで識別できるか確認した

## 適用範囲

- 本規約は `docs/**/*.drawio` に適用する
- `docs/90_knowledge/` 配下の技術学習用図解は、個別技術の構成説明が主目的のため、4 レイヤのすべてが登場するとは限らない。ただし**複数レイヤが登場する場合は本規約に従う**
- 本規約に違反する既存図は、採用側の全社展開期 監査 (`docs/99_壁打ち/drawio_layer_audit_*.md`) で順次是正する
