## 1. サマリ

1〜2 段落で、何が起こったかを経営視点でまとめる。技術者以外の読者（Product Council、監査）が読んで理解できる水準。

例: 「YYYY/MM/DD HH:MM に tier1 State API で 5xx エラーが急増し、約 40 分間、経費精算アプリの書込みが失敗しました。原因は CloudNativePG Primary の Disk 障害で、自動フェイルオーバーは成功したものの Replica 再構築待ちでレプリカ数が一時的に不足しました。業務影響は経費精算 120 件の一時保存失敗でしたが、tier2 側の自動リトライで全件復旧しました。」

## 2. 影響範囲

- **業務影響**: どの業務・何件・どれくらいの金額・顧客影響
- **SLO 影響**: エラーバジェット消費率、影響期間の SLI 値
- **データ損失**: 有無、範囲
- **監査報告の要否**: 個情法、金融庁等

## 4. 根本原因

技術的根本原因を 5 Whys / Fishbone などで分析し、以下の観点で記述。

### 4.1 直接原因

具体の技術的事象（例: Node の NVMe ディスクの物理故障）

### 4.2 組織的原因

プロセス・体制の問題（例: Disk SMART 監視アラートが未設定だった）

### 4.3 検知遅延の原因

アラート設定が不十分、メトリクス可視性がない等

### 4.4 対応遅延の原因

Runbook 不備、権限不足、担当者スキル等

## 5. うまくいった点

Blameless 原則の一環で、チームがうまく動いた点を先に挙げる。

- {例: Runbook RB-001 に従い、初動から自動フェイルオーバーまで 8 分で対応できた}
- {例: tier2 側のリトライ機構で自動復旧でき、手動再処理が不要だった}

## 6. うまくいかなかった点

改善余地を事実ベースで挙げる。個人批判は避ける。

- {例: Disk SMART 監視が設定されていなかったため、予兆検知できなかった}
- {例: Replica 再構築の ETA が業務部門に伝わらず問合せが集中した}

## 8. 再発防止

アクションアイテムを受けて、具体的に以下を明示。

- **検知強化**: どのアラート追加・調整したか
- **Runbook 更新**: どの Runbook をどう更新したか
- **設計変更**: 必要なら ADR を追加・更新
- **トレーニング**: 必要なら BC-TRN に追加項目

## 9. 学び（Lessons Learned）

組織として持ち帰るべき教訓を言語化。将来の設計判断に活きる水準で書く。

- {例: 自動フェイルオーバーだけではレプリカ回復まで時間がかかる。冗長度目標を Replica=3 ではなく Replica=4 に見直すかの検討が必要}

## ポストモーテム作成ルール

- Severity 1/2 は必須、S3/S4 は SRE 判断
- インシデント収束から 5 営業日以内に Draft 作成
- Draft は関係者 + SRE Lead でレビュー、10 営業日以内に Approved
- Approved 後は `docs/40_運用ライフサイクル/postmortems/` に配置（予定）
- 四半期に全ポストモーテムを集約レビュー、組織学習の材料にする
- 個人責任追及の文言は削除、「チームが」「プロセスが」に書き直す
