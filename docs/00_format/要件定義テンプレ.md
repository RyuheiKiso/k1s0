# 要件定義テンプレ

本ドキュメントは、`docs/03_要件定義/` 配下の個別要件ファイルが従うべき**記述フォーマット** を定義する。フォーマットの目的は、要件を「受け入れ可否が二値で判定できる」粒度に揃えることであり、散文の自由度と構造の一貫性を両立させる。

本テンプレは `CLAUDE.md` の「要件ドキュメントの記述」節と「テーブルの品質基準」節を踏襲する。両者と矛盾が生じた場合は `CLAUDE.md` が優先する。

---

## ファイル構造

個別要件ファイル（例: `10_アーキテクチャ/01_tier1.md`）は以下の 5 ブロック構造を持つ。

1. **導入段落** — この章が解決する課題と読み方
2. **前提・参照** — 依存する他要件 / ADR / 構想設計章
3. **要件本体** — ID ごとに散文＋受け入れ基準の塊を並べる
4. **関連図** — 重要な関係性は drawio の SVG を埋め込む
5. **章末サマリ表** — ID 一覧 / 優先度分布 / Phase 達成度

---

## 個別要件ブロックのテンプレート

各要件は以下の見出し構造で記述する。1 要件 = 1 つの `###` セクションとする。

```markdown
### <要件 ID>: <要件タイトル>

- 優先度: MUST / SHOULD / MAY（判定理由を併記）
- Phase: Phase 1a / 1b / 1c / 2 / 3（達成が要求される最も早い Phase）
- 関連: 依存する要件 ID / 参照 ADR / 参照企画・構想設計章

<現状段落：現在の世界ではどうなっているか、何が困っているか>

<要件達成後段落：この要件が満たされた世界では何が可能になるか>

<崩れた時段落：この要件が未達の場合、どういう事業・技術・法務影響が出るか>

**受け入れ基準**

- <検証可能な条件 1>
- <検証可能な条件 2>
- <検証可能な条件 3>

**検証方法**

- <テスト / 監査 / 運用観測など、どの手段で受け入れ基準を検証するか>
```

### 散文 3 段落の使い分け

要件の説得力は「**現状 → 達成後 → 崩れた時**」の 3 段構成で発生する。崩れた時の段落は稟議・監査・リリース判断で最も参照される。したがって「未達時に誰が何に困るか」を具体的な主体と影響で書く。たとえば「p99 500ms を超過すると tier2 バックオフィス業務のバッチ締め処理が翌営業日に食い込む」といった形で、抽象語を避ける。

### 優先度判定理由の同居

優先度は単に MUST / SHOULD / MAY と書くのではなく、判定理由を併記する。

- `MUST（稟議通過の前提。未達で Phase 0 承認不可）`
- `SHOULD（Phase 1a ではバイパス容認、Phase 1b で必達）`
- `MAY（Phase 2 の拡張余地。現時点では保留）`

### Phase 記号

Phase は企画 [`../01_企画/03_ロードマップと体制/00_フェーズ計画.md`](../../01_企画/03_ロードマップと体制/00_フェーズ計画.md) の定義に準拠する。要件ファイル単体では Phase 記号の定義を繰り返さず、ルート `README.md` と企画を参照させる。

## 章末サマリ表のフォーマット

各ファイルの末尾には、そのファイル内で定義した要件の一覧サマリを表形式で置く。これは `CLAUDE.md` が許容する「章末サマリ」の唯一の例外である。

### ID 一覧

| ID | タイトル | 優先度 | Phase |
|---|---|---|---|
| COM-GLO-001 | 用語「tier1」の定義 | MUST | 1a |
| COM-GLO-002 | 用語「tier2」の定義 | MUST | 1a |

### 優先度分布（任意）

| 優先度 | 件数 | 代表 ID |
|---|---|---|
| MUST | 12 | COM-GLO-001, COM-GLO-005, ... |
| SHOULD | 4 | COM-GLO-003, ... |
| MAY | 2 | COM-GLO-010, ... |

### Phase 達成度（任意）

| Phase | 必達件数 | 達成見込み | 未達影響 |
|---|---|---|---|
| 1a | 8 | 稟議通過の必要条件 | 稟議差し戻し |
| 1b | 3 | パイロット業務開始の必要条件 | パイロット延期 |

## テーブルの品質基準（再掲）

`CLAUDE.md` の規約を要件定義でも踏襲する。要点を再掲する。

- ラベル羅列のテーブルは不可。各セルに根拠・数値・出典を入れ、1 行読んで納得できる密度にする。
- 優先度セルは判定理由を同居させる。
- 数値セルは根拠を併記する（例: 「48 時間以内（業界平均 72 時間を参考に短縮目標）」）。
- 期待効果セルは前後比較を書く（例: 「年次ライセンス費 2,000 万円削減、稟議期間 3 か月 → 0」）。
- 事例参照が可能な場合は事例そのものをセルに入れる。

## 図表

重要な関係性（要件間依存・フロー・構造）は drawio で作図する。作図規約は [`drawio_layer_convention.md`](./drawio_layer_convention.md) と `CLAUDE.md` の drawio 節に従う。要件 ID 相互の依存を可視化する場合は、ノードに ID を書き、依存方向を矢印で示す。

## 変更の追跡

要件の変更は git 履歴で追う。ただし「**なぜ変更したか**」が commit メッセージだけでは追えない規模の判断変更は、ADR として別途記録する（例: 「SEC-AUD-005 の受け入れ基準を週次同期から日次同期に強化」）。要件の廃止は ID を保持したままステータスを `Deprecated` に変更し、代替要件の ID を `Superseded by` として追記する。
