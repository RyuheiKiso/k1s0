# ドキュメント標準

本書は k1s0 プロジェクトで作成するドキュメントの標準書式・原則・運用ルールを定める。[CLAUDE.md](../../CLAUDE.md) のプロジェクト規約を具体的なドキュメント記述ガイドラインに展開したもの。

## 基本方針

1. **部外者が読める品質**: 初見の読み手（監査、新規採用者、外部レビュワー）が独立して理解できる
2. **散文と図の両立**: 表の羅列は禁止、各章節は散文の導入段落 + 図 + 詳細で構成
3. **具体性**: 数値・出典・事例を入れ、ラベル羅列を禁止
4. **追跡可能性**: 要件 ID、ADR ID、Issue ID で関連を追える

## 文書タイプ別書式

### タイプ A: 要件定義書（`03_要件定義/` 配下）

- 各要件は「現状」「達成後の世界」「崩れた時の影響」を散文で段落展開
- ID 体系: `BR-` / `FR-` / `NFR-A〜I-` / `OR-` / `DX-` / `BC-` / `RISK-`
- 章末に ID 一覧、段階達成度、優先度分布の集計表
- 受け入れ基準は箇条書きで列挙
- 関連 ADR / 関連要件へのリンク必須

### タイプ B: ADR（Architecture Decision Record）

- 書式: [ADR-xxxx.md](ADR-xxxx.md) テンプレ準拠
- ID 体系: `ADR-DOMAIN-NNN-short-name.md`（例: ADR-DATA-001-cloudnativepg.md）
- 5 段構成: コンテキスト / 決定 / 検討した選択肢 / 帰結 / 実装タスク
- 検討肢は最低 3 件、採用しなかった選択肢にも理由を明記
- 帰結はポジティブ / ネガティブ両方
- Status: Proposed / Accepted / Deprecated / Superseded by

### タイプ C: Runbook（運用手順）

- 5 段構成: 検出 / 初動 / 復旧 / 根本原因調査 / 事後処理
- 具体的な kubectl / SQL コマンドを記載
- 判定閾値を明示
- アラートの `runbook_url` から直接参照可能

### タイプ D: 技術説明書・設計書

- 冒頭に「何を解決するか」の導入段落
- 概念図 → 構造図 → シーケンス図 → ER 図 の順で可視化
- 実装詳細は参照リンクに留め、本書は論理レベル

## 図の作成規約

### 使用ツール

- drawio（ローカル版）を必須、draw.io Desktop を利用
- Mermaid は Markdown 内で軽量図にのみ許可（シーケンス図・フローチャート）
- アスキーアート、罫線図は一切禁止

### drawio 運用

- `docs/{path}/` 配下に `img/` サブフォルダ作成
- `.drawio`（編集用）と `.svg`（埋込用）の両方を保管
- ファイル命名: `{概念}_{対象}.drawio`（例: `seq_tier1_patterns.drawio`）

### レイヤ色分離

[drawio_layer_convention.md](drawio_layer_convention.md) 準拠。

- アプリ層: `#fff2cc` / `#d79b00`（暖色）
- ネットワーク層: `#dae8fc` / `#6c8ebf`（寒色）
- インフラ層: `#f5f5f5` / `#666666`（中性灰）
- データ層: `#e1d5e7` / `#9673a6`（薄紫）

### 品質チェック

- 背景は白矩形を最初の要素として配置（SVG export の透明背景対策）
- 矢印はボックス・テキストに重ならない
- ラベルは矢印を覆わない
- ダークテーマで確認

## Markdown 記述規約

### 基本

- UTF-8、改行は LF
- 見出しは `#` 1 階層から開始
- リンクは相対パス推奨（`./`, `../`）
- 画像は `![alt](path)` 形式
- コードブロックは言語指定必須（` ```rust `、` ```go ` 等）

### 見出しレベル

- `#`: 文書タイトル
- `##`: 章（1 章 1 概念）
- `###`: 節（具体論）
- `####`: 項（詳細・例示）

### 表の使用基準

- 情報密度が散文より高くなる場合のみ使用
- セル内は必ず根拠・数値・出典を含む
- ラベル羅列の表（例: 「○ / △ / ×」のみ）は禁止
- 要件本体を表セルに入れることは禁止

### 引用と参考文献

- 内部ドキュメントはファイル名リンク
- 外部資料は「タイトル」+ URL または書籍名 + 著者
- OSS プロジェクトは公式ドメイン（例: `opentelemetry.io`）

## ファイル名規約

### 日本語

- 要件定義書・設計書は日本語ファイル名許可（例: `01_業務背景.md`）
- 番号プレフィックス: `{01..99}_` で順序付け

### 英数字

- ADR、テンプレート、設定ファイルは英数字
- ハイフン区切り（`ADR-DATA-001-cloudnativepg.md`）
- スペース・全角英数字は禁止

## リンクとトレーサビリティ

### 必須リンク

- 要件 → 実装する ADR
- ADR → 影響する要件
- Runbook → 対応する FMEA 項目
- 業務フロー → 関連する要件 ID

### トレーサビリティマトリクス

- [80_トレーサビリティ/](../03_要件定義/80_トレーサビリティ/) で管理
- PR で要件・ADR 追加/変更時は必ずマトリクス更新

## ライフサイクル管理

### バージョニング

- ドキュメントは Git で履歴管理
- Major 変更は PR コメントに Before/After 要約
- CHANGELOG.md（[00_format/CHANGELOG.md](CHANGELOG.md)）を必要に応じて更新

### 鮮度維持

- 半期ごとに全ドキュメントの最終更新日棚卸し
- 1 年以上更新のないドキュメントは `Deprecated` 検討
- インシデント後は関連 Runbook を必ず更新

### 削除と Deprecation

- ADR は `Status: Deprecated` / `Superseded by ADR-XXX` に変更、削除しない
- 要件は削除ではなく `Status: 廃止` マーク
- Runbook は手順が古くなっても削除せず、版管理で上書き

## 用語統一

- 用語集: [00_要件定義方針/04_用語集.md](../03_要件定義/00_要件定義方針/04_用語集.md) 参照
- 固有名詞の表記揺れを避ける（例: 「Kafka」「kafka」の混在禁止）
- 略語は初出時に正式名称を併記

## レビュー体制

- ドキュメント PR のレビュアーは [review_checklist.md](review_checklist.md) を参照
- 新規カテゴリ追加は Tech Lead 承認必須
- ADR は Technical Lead + 関連チームリード承認
