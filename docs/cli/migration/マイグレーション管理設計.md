# マイグレーション管理設計

DB マイグレーションの作成・適用・ロールバック・状態確認を言語を意識せず統合管理する。
本機能は CLIフローのメインメニューに「マイグレーション管理」として追加する。

> メインメニューの全体像は [CLIフロー](../flow/CLIフロー.md) を参照。
> ローカル開発環境での自動マイグレーション実行は [ローカル開発環境設計](../dev/ローカル開発環境設計.md) を参照。

## 基本方針

- Go（golang-migrate）と Rust（sqlx-cli）のマイグレーションツール差異を CLI が吸収する
- 連番の自動採番（既存ファイルの最大番号 +1）で命名ミスを防止する
- up / down の両ファイルを必ずペアで生成し、ロールバック可能性を保証する
- 適用状態の一覧表示で「どこまで適用済みか」を即座に確認できる
- `k1s0 ci-check` に未適用マイグレーションの検出を統合する

## 対話フロー

```
? 操作を選択してください
  ...
> マイグレーション管理
  ...
```

```
[1] 操作を選択してください
    > 新規作成
      適用 (up)
      ロールバック (down)
      状態確認
      修復
```

---

### 新規作成

```
[1] > 新規作成

[2] 対象サービスを選択してください
    > order-server (service/rust)  — migrations/ に 3 ファイル
      payment-server (service/go) — migrations/ に 2 ファイル
      auth-server (system/rust)   — migrations/ に 5 ファイル
      ...

[3] マイグレーション名を入力してください（英数字・アンダースコア）: ____
    例: add_payment_status_column

[確認] 以下のファイルを作成します。よろしいですか？

    対象: order-server
    ファイル:
      migrations/004_add_payment_status_column.up.sql
      migrations/004_add_payment_status_column.down.sql
    連番: 004（既存最大: 003）

    > はい
      いいえ（前のステップに戻る）
      キャンセル（メインメニューに戻る）
```

#### 生成されるファイル

**up.sql**:
```sql
-- migrations/004_add_payment_status_column.up.sql
-- マイグレーション: add_payment_status_column
-- 作成日: 2026-03-06

-- TODO: ここにスキーマ変更を記述してください

```

**down.sql**:
```sql
-- migrations/004_add_payment_status_column.down.sql
-- ロールバック: add_payment_status_column
-- 作成日: 2026-03-06

-- TODO: ここに up.sql の変更を元に戻す処理を記述してください

```

#### 命名規則

```
{NNN}_{description}.{up|down}.sql
```

| 要素 | ルール |
|------|-------|
| `NNN` | 3 桁ゼロ埋め連番（001, 002, ...） |
| `description` | 英小文字・数字・アンダースコアのみ（`[a-z0-9_]+`） |
| 方向 | `.up.sql`（適用）と `.down.sql`（ロールバック）のペア |

#### 連番の自動採番

```
既存ファイル:
  001_create_schema.up.sql
  002_create_users.up.sql
  003_create_orders.up.sql

→ 次の連番: 004
```

既存ファイルの最大番号を検出し、+1 した値を自動採番する。
番号にギャップがある場合（例: 001, 002, 005）は、最大番号 +1（006）を採番する。

### 適用 (up)

```
[1] > 適用 (up)

[2] 対象サービスを選択してください
    > order-server (service/rust)  — 1 件未適用
      auth-server (system/rust)   — すべて適用済み
      ...

[3] 適用範囲を選択してください
    > すべての未適用マイグレーションを適用
      指定したマイグレーションまで適用

[4] (指定適用の場合)
    どのマイグレーションまで適用しますか？
    > 004_add_payment_status_column
      005_add_audit_log_table

[5] 接続先データベースを選択してください
    > ローカル開発 (localhost:5432/order_db)
      カスタム接続文字列を入力

[確認] 以下のマイグレーションを適用します。よろしいですか？

    対象: order-server → order_db (localhost:5432)
    適用:
      004_add_payment_status_column.up.sql

    > はい
      いいえ（前のステップに戻る）
      キャンセル（メインメニューに戻る）
```

#### 実行中の表示

```
[migrate up] マイグレーション適用中...

  004_add_payment_status_column.up.sql
    ALTER TABLE orders ADD COLUMN payment_status VARCHAR(20)...
    ✓ 成功 (0.05s)

[migrate up] 完了: 1 件適用
  現在のバージョン: 004
```

#### 適用方法（言語別）

CLI は対象サービスの言語を自動検出し、適切なマイグレーションツールを使用する。

| 言語 | 検出方法 | マイグレーションツール | 実行コマンド |
|------|---------|---------------------|-------------|
| Rust | `Cargo.toml` の存在 | sqlx-cli | `sqlx migrate run --source {dir} --database-url {url}` |
| Go | `go.mod` の存在 | golang-migrate | `migrate -path {dir} -database {url} up {N}` |

**接続文字列の構築**:

ローカル開発を選択した場合:
```
postgresql://app:password@localhost:5432/{database_name}?sslmode=disable
```

`.k1s0-dev/state.json` からポート番号とデータベース名を取得する。ローカル開発環境が起動していない場合はエラーを表示する。

```
[migrate up] エラー: ローカル開発環境が起動していません。
  → k1s0 → ローカル開発 → 起動 を先に実行してください。
  → または「カスタム接続文字列を入力」を選択してください。
```

### ロールバック (down)

```
[1] > ロールバック (down)

[2] 対象サービスを選択してください
    > order-server (service/rust)  — 現在: 004

[3] ロールバック範囲を選択してください
    > 直前の 1 件をロールバック
      指定したバージョンまでロールバック
      すべてロールバック（初期状態に戻す）

[4] (指定ロールバックの場合)
    どのバージョンまで戻しますか？
    > 003_create_orders
      002_create_users
      001_create_schema

[5] 接続先データベースを選択してください
    > ローカル開発 (localhost:5432/order_db)
      カスタム接続文字列を入力

[確認] ⚠ 以下のマイグレーションをロールバックします。データが失われる可能性があります。

    対象: order-server → order_db (localhost:5432)
    ロールバック:
      004_add_payment_status_column.down.sql

    > はい
      いいえ（前のステップに戻る）
      キャンセル（メインメニューに戻る）
```

#### 実行中の表示

```
[migrate down] ロールバック実行中...

  004_add_payment_status_column.down.sql
    ALTER TABLE orders DROP COLUMN payment_status...
    ✓ 成功 (0.03s)

[migrate down] 完了: 1 件ロールバック
  現在のバージョン: 003
```

#### 安全策

- ロールバック実行前に必ず確認プロンプトを表示する
- `すべてロールバック` を選択した場合は追加の確認を行う

```
[確認] ⚠ 本当にすべてのマイグレーションをロールバックしますか？
       すべてのテーブルとデータが削除されます。
       確認のため "rollback-all" と入力してください: ____
```

### 修復

マイグレーション状態テーブルが不整合になった場合に修復する。

```
[1] > 修復

[2] 対象サービスを選択してください
    > order-server (service/rust)

[3] 修復操作を選択してください
    > ダーティ状態をクリア（失敗したマイグレーションのフラグをリセット）
      バージョンを強制設定（特定のバージョンに状態を上書き）
```

#### ダーティ状態のクリア

マイグレーション実行中にエラーが発生すると「ダーティ」状態になり、以降のマイグレーションが実行できなくなる。この操作でダーティフラグをリセットする。

```
[migrate fix] order-server のダーティ状態をクリアしました。
  現在のバージョン: 003（004 はダーティ状態でした）

  ⚠ 004_add_payment_status_column.up.sql が部分的に適用されている可能性があります。
    手動でスキーマを確認し、必要に応じてロールバックしてください。
```

#### バージョン強制設定

```
[3] > バージョンを強制設定

[4] 設定するバージョンを入力してください（番号）: ____
    例: 3（003 に設定）

[確認] ⚠ マイグレーションバージョンを強制的に 003 に設定します。
       実際のスキーマ状態との整合性は保証されません。

    > はい
      いいえ
```

## マイグレーションファイルの検出

### ディレクトリ検出ルール

対象サービスのプロジェクトルートから `migrations/` ディレクトリを検出する。

```
regions/{tier}/{domain}/server/{language}/{service}/migrations/
regions/{tier}/server/{language}/{service}/migrations/
regions/{tier}/{domain}/database/{db-name}/migrations/
regions/{tier}/database/{db-name}/migrations/
```

### ファイル走査

```
1. migrations/ 配下の全 .sql ファイルを走査
2. ファイル名から連番・説明・方向（up/down）を抽出
   例: 003_create_orders.up.sql
       → 番号: 3, 説明: create_orders, 方向: up
3. up/down のペアが揃っているか検証
4. 連番の順序が正しいか検証
```

### ペア不整合の検出

```
[migrate] ⚠ マイグレーションファイルの不整合を検出しました:

  003_create_orders.up.sql     — ✓ 存在
  003_create_orders.down.sql   — ✗ 欠落

  down.sql が欠落しているとロールバックできません。
  ファイルを作成してください。
```

## 接続先データベースの解決

### ローカル開発環境

`.k1s0-dev/state.json` が存在し、PostgreSQL が起動中の場合に利用可能。

```json
{
  "dependencies": {
    "postgres": {
      "port": 5432,
      "databases": ["order_db", "payment_db"]
    }
  }
}
```

→ 接続文字列: `postgresql://app:password@localhost:5432/order_db?sslmode=disable`

### config.yaml からの解決

ローカル開発環境が利用できない場合、`config/config.dev.yaml` から接続情報を読み取る。

```yaml
database:
  host: "localhost"
  port: 5432
  name: "order_db"
  user: "app"
  password: "password"
```

### カスタム接続文字列

手動入力の場合、以下の形式を受け付ける。

```
postgresql://{user}:{password}@{host}:{port}/{database}?sslmode={mode}
mysql://{user}:{password}@tcp({host}:{port})/{database}
```

## RDBMS 別の対応

### PostgreSQL

```sql
-- up.sql の例
ALTER TABLE orders ADD COLUMN payment_status VARCHAR(20) NOT NULL DEFAULT 'pending';
CREATE INDEX idx_orders_payment_status ON orders (payment_status);

-- down.sql の例
DROP INDEX IF EXISTS idx_orders_payment_status;
ALTER TABLE orders DROP COLUMN IF EXISTS payment_status;
```

### MySQL

```sql
-- up.sql の例
ALTER TABLE orders ADD COLUMN payment_status VARCHAR(20) NOT NULL DEFAULT 'pending';
CREATE INDEX idx_orders_payment_status ON orders (payment_status);

-- down.sql の例
DROP INDEX idx_orders_payment_status ON orders;
ALTER TABLE orders DROP COLUMN payment_status;
```

### SQLite

```sql
-- up.sql の例（ALTER TABLE の制約あり）
CREATE TABLE orders_new (
    id TEXT PRIMARY KEY,
    user_id TEXT NOT NULL,
    total INTEGER NOT NULL,
    payment_status TEXT NOT NULL DEFAULT 'pending',
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP
);
INSERT INTO orders_new SELECT id, user_id, total, 'pending', created_at FROM orders;
DROP TABLE orders;
ALTER TABLE orders_new RENAME TO orders;

-- down.sql の例
CREATE TABLE orders_old (
    id TEXT PRIMARY KEY,
    user_id TEXT NOT NULL,
    total INTEGER NOT NULL,
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP
);
INSERT INTO orders_old SELECT id, user_id, total, created_at FROM orders;
DROP TABLE orders;
ALTER TABLE orders_old RENAME TO orders;
```

## CI 統合

### 未適用マイグレーションの検出

`k1s0 ci-check` に統合し、ファイルレベルで未適用マイグレーションの存在を検出する。

```bash
# CI パイプラインでの実行
k1s0 ci-check --migrations
```

#### CI チェック内容

| チェック | 説明 | 重大度 |
|---------|------|-------|
| ペア整合性 | up.sql と down.sql がペアで存在するか | Error |
| 連番整合性 | 連番にギャップや重複がないか | Warning |
| down.sql 非空チェック | down.sql が空（TODO のまま）でないか | Warning |
| SQL 構文チェック | 基本的な SQL 構文エラーがないか | Error |

#### CI 出力例

```
[migrations] マイグレーションファイルを検証中...

[migrations] order-server:
  ✓ 001_create_schema          up/down ペア OK
  ✓ 002_create_users           up/down ペア OK
  ✓ 003_create_orders          up/down ペア OK
  ✗ 004_add_payment_status     down.sql が空です（TODO のまま）
  ✓ 005_add_audit_log          up/down ペア OK

[migrations] 結果: 1 warning
```

### GitHub Actions 設定例

```yaml
# .github/workflows/ci.yaml
- name: マイグレーションチェック
  run: k1s0 ci-check --migrations --format github-actions
```

## エラーハンドリング

| エラー | メッセージ | 対処 |
|-------|----------|------|
| migrations/ 未検出 | `マイグレーションディレクトリが見つかりません: {path}` | サービスの配置パスを確認するよう案内 |
| ファイル名形式エラー | `ファイル名が命名規則に一致しません: {file}` | `{NNN}_{description}.{up\|down}.sql` の形式を案内 |
| 連番重複 | `連番が重複しています: {number} ({file1}, {file2})` | ファイル名を修正するよう案内 |
| DB 接続失敗 | `データベースに接続できません: {url} — {error}` | 接続情報と DB の起動状態を確認するよう案内 |
| マイグレーション実行エラー | `マイグレーション失敗: {file} — {sql_error}` | SQL を修正するか、`修復` で状態をリセットするよう案内 |
| ダーティ状態 | `マイグレーションがダーティ状態です (version: {n})` | `修復 → ダーティ状態をクリア` を案内 |
| マイグレーションツール未インストール | `{tool} がインストールされていません` | インストールコマンドを表示 |
| ローカル環境未起動 | `ローカル開発環境が起動していません` | `k1s0 → ローカル開発 → 起動` を案内 |

## マイグレーションツールのインストール確認

CLI は初回実行時にマイグレーションツールの存在を確認する。

| 言語 | ツール | インストールコマンド |
|------|-------|-------------------|
| Rust | sqlx-cli | `cargo install sqlx-cli --no-default-features --features postgres` |
| Go | golang-migrate | `go install -tags 'postgres' github.com/golang-migrate/migrate/v4/cmd/migrate@latest` |

```
[migrate] sqlx-cli が見つかりません。インストールしますか？
    > はい（自動インストール）
      いいえ（手動でインストールします）
```

「はい」を選択した場合、CLI がバックグラウンドでインストールを実行する。
