# ローカル開発環境設計

対象サービスの依存関係を解析し、必要な外部サービス（PostgreSQL・Kafka・Redis 等）を一括起動する。
本機能は CLIフローのメインメニューに「ローカル開発」として追加する。

> メインメニューの全体像は [CLIフロー](../flow/CLIフロー.md) を参照。
> 各サービスの config.yaml スキーマは [config設計](../config/config設計.md) を参照。

## 基本方針

- サービスの `config/config.yaml` から依存する外部サービス（DB・Kafka・Redis）を自動検出する
- 検出結果に基づき `docker-compose.yaml` を動的生成して起動する
- DB マイグレーションの自動実行とシードデータ投入をサポートする
- 認証はスキップモード（Keycloak 不要）をデフォルトとし、必要に応じて有効化できる
- `k1s0 dev down` で全コンテナ・ボリュームを完全クリーンアップする

## 依存検出ロジック

### 検出ソース

各サービスの `config/config.yaml` を解析し、有効なセクションから依存を検出する。

| config セクション | 検出結果 | Docker イメージ |
|------------------|---------|----------------|
| `database` | PostgreSQL / MySQL / SQLite | `postgres:17` / `mysql:8.4` / なし（ファイルDB） |
| `kafka` | Kafka ブローカー + Schema Registry | `confluentinc/cp-kafka:7.7` |
| `redis` | Redis | `redis:7` |
| `redis_session` | Redis（セッション用・別インスタンス） | `redis:7` |

### 検出フロー

```
1. 対象サービスの config/config.yaml を読み込み
2. 各セクションの存在を確認
   - database セクションあり → has_database = true
   - kafka セクションあり   → has_kafka = true
   - redis セクションあり   → has_redis = true
3. database.name から DB 名を取得（例: task_db）
4. kafka.topics.publish / subscribe からトピック一覧を取得
5. 複数サービス選択時は依存を統合（重複排除）
```

### 複数サービス選択時の統合

複数サービスを同時に選択した場合、依存する外部サービスを統合する。

```
task-server:     PostgreSQL (task_db) + Kafka + Redis
activity-server: PostgreSQL (activity_db) + Kafka

→ 統合結果:
  PostgreSQL: task_db, activity_db（同一インスタンスに2データベース作成）
  Kafka: 1 ブローカー（トピックは統合）
  Redis: 1 インスタンス
```

## 動的 docker-compose 生成

### 生成先

```
.k1s0-dev/docker-compose.yaml
```

`.k1s0-dev/` ディレクトリは CLI が自動作成する。`.gitignore` に追加を推奨する。

### 生成テンプレート

#### PostgreSQL

```yaml
services:
  postgres:
    image: postgres:17
    container_name: k1s0-dev-postgres
    environment:
      POSTGRES_USER: app
      POSTGRES_PASSWORD: password
      POSTGRES_DB: "{{ databases[0].name }}"
    ports:
      - "5432:5432"
    volumes:
      - k1s0_dev_postgres_data:/var/lib/postgresql/data
      - ./.k1s0-dev/init-db.sql:/docker-entrypoint-initdb.d/init.sql
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U app"]
      interval: 5s
      timeout: 5s
      retries: 5
```

複数 DB が必要な場合、`init-db.sql` で追加 DB を作成する。

```sql
-- .k1s0-dev/init-db.sql（自動生成）
CREATE DATABASE task_db;
CREATE DATABASE activity_db;
```

#### Kafka

```yaml
  kafka:
    image: confluentinc/cp-kafka:7.7.0
    container_name: k1s0-dev-kafka
    environment:
      KAFKA_NODE_ID: 1
      KAFKA_PROCESS_ROLES: broker,controller
      KAFKA_LISTENERS: PLAINTEXT://0.0.0.0:9092,CONTROLLER://0.0.0.0:9093
      KAFKA_ADVERTISED_LISTENERS: PLAINTEXT://localhost:9092
      KAFKA_CONTROLLER_QUORUM_VOTERS: 1@localhost:9093
      KAFKA_CONTROLLER_LISTENER_NAMES: CONTROLLER
      KAFKA_LISTENER_SECURITY_PROTOCOL_MAP: PLAINTEXT:PLAINTEXT,CONTROLLER:PLAINTEXT
      KAFKA_OFFSETS_TOPIC_REPLICATION_FACTOR: 1
      KAFKA_AUTO_CREATE_TOPICS_ENABLE: "true"
      CLUSTER_ID: "k1s0-dev-cluster"
    ports:
      - "9092:9092"
    volumes:
      - k1s0_dev_kafka_data:/var/lib/kafka/data
    healthcheck:
      test: ["CMD-SHELL", "kafka-broker-api-versions --bootstrap-server localhost:9092"]
      interval: 10s
      timeout: 10s
      retries: 5
```

dev 環境では KRaft モード（ZooKeeper なし）で起動する。

#### Redis

```yaml
  redis:
    image: redis:7
    container_name: k1s0-dev-redis
    ports:
      - "6379:6379"
    volumes:
      - k1s0_dev_redis_data:/data
    healthcheck:
      test: ["CMD", "redis-cli", "ping"]
      interval: 5s
      timeout: 3s
      retries: 5
```

#### 管理 UI

```yaml
  pgadmin:
    image: dpage/pgadmin4:latest
    container_name: k1s0-dev-pgadmin
    environment:
      PGADMIN_DEFAULT_EMAIL: admin@example.com
      PGADMIN_DEFAULT_PASSWORD: admin
      PGADMIN_CONFIG_SERVER_MODE: "False"
    ports:
      - "5050:80"
    depends_on:
      postgres:
        condition: service_healthy

  kafka-ui:
    image: provectuslabs/kafka-ui:latest
    container_name: k1s0-dev-kafka-ui
    environment:
      KAFKA_CLUSTERS_0_NAME: k1s0-dev
      KAFKA_CLUSTERS_0_BOOTSTRAPSERVERS: kafka:9092
    ports:
      - "8081:8080"
    depends_on:
      kafka:
        condition: service_healthy
```

#### Keycloak（認証モード有効時のみ）

```yaml
  keycloak:
    image: quay.io/keycloak/keycloak:26.0
    container_name: k1s0-dev-keycloak
    command: start-dev --import-realm
    environment:
      KC_DB: dev-file
      KEYCLOAK_ADMIN: admin
      KEYCLOAK_ADMIN_PASSWORD: admin
    ports:
      - "8180:8080"
    volumes:
      - ./.k1s0-dev/keycloak/realm-export.json:/opt/keycloak/data/import/realm-export.json
    healthcheck:
      test: ["CMD-SHELL", "exec 3<>/dev/tcp/localhost/8080 && echo -e 'GET /health/ready HTTP/1.1\\r\\nHost: localhost\\r\\n\\r\\n' >&3 && cat <&3 | grep -q '200'"]
      interval: 10s
      timeout: 5s
      retries: 10
```

> Keycloak の realm 設定は [認証認可設計](../../architecture/認証認可設計.md) を参照。

#### volumes セクション

```yaml
volumes:
  k1s0_dev_postgres_data:
  k1s0_dev_kafka_data:
  k1s0_dev_redis_data:
```

## DB マイグレーション自動実行

### 実行タイミング

docker-compose の全サービスが `healthy` になった後、自動的にマイグレーションを実行する。

### マイグレーションファイルの検出

```
regions/{tier}/**/migrations/*.up.sql
```

対象サービスのディレクトリ配下にある `migrations/` を自動検出する。

### 実行方法

| 言語 | マイグレーションツール | コマンド |
|------|---------------------|---------|
| Rust | sqlx-cli | `sqlx migrate run --source {migrations_dir} --database-url {url}` |
| Go | golang-migrate | `migrate -path {migrations_dir} -database {url} up` |

**接続文字列の自動構築**:
```
postgresql://app:password@localhost:5432/{database_name}?sslmode=disable
```

config.yaml の `database.name` からデータベース名を取得し、ローカル開発用の固定認証情報で接続する。

### マイグレーション失敗時

```
[dev up] DB マイグレーション実行中...
  ✓ 001_create_schema.up.sql
  ✗ 002_create_orders.up.sql
    エラー: relation "users" does not exist

  マイグレーションに失敗しました。
  → ファイルを修正して再実行: k1s0 dev migrate
  → マイグレーションをスキップして続行するには: k1s0 dev up --skip-migrate
```

## シードデータ

### シードファイルの配置

```
regions/{tier}/**/seed/
├── *.sql         — SQL ファイル（INSERT 文）
└── *.json        — JSON ファイル（fixture データ）
```

### SQL シード

`seed/` ディレクトリ内の `.sql` ファイルをファイル名の辞書順で実行する。

```sql
-- seed/001_users.sql
INSERT INTO users (id, name, email) VALUES
  ('usr-001', 'Test User', 'test@example.com'),
  ('usr-002', 'Admin User', 'admin@example.com');
```

### JSON シード

`.json` ファイルはテーブル名をファイル名から推定し、自動的に INSERT 文を生成する。

```json
// seed/tasks.json
[
  {
    "id": "tsk-001",
    "user_id": "usr-001",
    "total": 1500,
    "status": "pending"
  }
]
```

→ ファイル名 `tasks.json` から `tasks` テーブルへの INSERT として実行。

### シードデータの無効化

```bash
k1s0 dev up --skip-seed
```

## 認証スキップモード

### 動作

認証スキップモードでは、以下の設定でサービスを起動する。

```yaml
# .k1s0-dev/config.dev-local.yaml（自動生成）
auth:
  skip_verification: true
  mock_claims:
    sub: "dev-user-001"
    realm_access:
      roles: ["sys_admin"]
    tier_access: ["system", "business", "service"]
```

- JWT 検証ミドルウェアをバイパスし、固定の Claims を注入する
- Keycloak コンテナを起動しない
- すべてのエンドポイントにフルアクセス可能

### Keycloak モード

Keycloak モードでは、以下の追加コンテナを起動する。

- Keycloak 本体（`localhost:8180`）
- realm 設定を自動インポート

テスト用ユーザーが自動作成される。

| ユーザー | パスワード | ロール |
|---------|----------|-------|
| `admin` | `admin` | `sys_admin` |
| `user` | `user` | `svc_user` |
| `auditor` | `auditor` | `sys_auditor` |

## ポート管理

### デフォルトポート割り当て

| サービス | ポート | 変更可否 |
|---------|-------|---------|
| PostgreSQL | 5432 | 可 |
| Kafka | 9092 | 可 |
| Redis | 6379 | 可 |
| Redis (session) | 6380 | 可 |
| pgAdmin | 5050 | 可 |
| Kafka UI | 8081 | 可 |
| Keycloak | 8180 | 可 |

### ポート競合時の処理

```
[dev up] ポート競合を検出しました:
  5432: 既に使用中（PID: 12345, postgres）

  対処方法を選択してください:
  > 別のポートを使用（5433）
    既存プロセスを確認して手動で解決
    中止
```

ポートを変更した場合、生成する `config.dev-local.yaml` のポート番号も自動的に更新する。

## 設定ファイル

### .k1s0-dev/ ディレクトリ構造

```
.k1s0-dev/
├── docker-compose.yaml          # 動的生成（毎回上書き）
├── config.dev-local.yaml        # ローカル接続情報（毎回上書き）
├── init-db.sql                  # 複数 DB 初期化スクリプト（毎回上書き）
├── .env                         # docker-compose 用環境変数
├── keycloak/                    # Keycloak モード時のみ
│   └── realm-export.json        # realm 設定
└── state.json                   # 起動状態の記録
```

### state.json

CLI が起動状態を管理するための内部ファイル。

```json
{
  "version": 1,
  "started_at": "2026-03-06T10:30:00Z",
  "services": ["task-server"],
  "dependencies": {
    "postgres": { "port": 5432, "databases": ["task_db"] },
    "kafka": { "port": 9092 },
    "redis": { "port": 6379 }
  },
  "auth_mode": "skip",
  "migration_status": {
    "task_db": { "applied": 3, "total": 3 }
  },
  "seed_status": {
    "task_db": { "applied": true }
  }
}
```

## サブコマンド

メインの `dev up` / `dev down` に加え、以下のサブ操作をサポートする。

### dev migrate

マイグレーションのみを再実行する。

```
k1s0 → ローカル開発 → マイグレーション再実行

[dev migrate] マイグレーション実行中...
  - 001_create_schema.up.sql       既に適用済み（スキップ）
  ✓ 002_create_tasks.up.sql        0.1s
  ✓ 003_create_task_items.up.sql   0.1s
```

### dev seed

シードデータの投入のみを再実行する。

```
k1s0 → ローカル開発 → シードデータ再投入

[dev seed] 既存データをクリアしますか？
    > はい（TRUNCATE 実行後に再投入）
      いいえ（追加投入）

[dev seed] シードデータ投入中...
  ✓ seed/001_users.sql        2 rows
  ✓ seed/tasks.json           5 rows
```

### dev reset

全データを削除してマイグレーション＋シードを最初から実行する。

```
k1s0 → ローカル開発 → リセット

[dev reset] ⚠ すべてのデータが削除されます。よろしいですか？
    > はい
      いいえ

[dev reset] ボリューム削除中...
[dev reset] コンテナ再作成中...
[dev reset] マイグレーション実行中...
[dev reset] シードデータ投入中...
[dev reset] 完了
```

## エラーハンドリング

| エラー | メッセージ | 対処 |
|-------|----------|------|
| Docker 未起動 | `Docker Engine が起動していません。Docker Desktop を起動してください。` | Docker 起動を案内 |
| config.yaml 未検出 | `config/config.yaml が見つかりません: {path}` | パスを確認するよう案内 |
| ポート競合 | `ポート {port} は既に使用中です (PID: {pid})` | 別ポートの提案または手動解決を案内 |
| マイグレーション失敗 | `マイグレーションに失敗しました: {file} - {error}` | ファイル修正後の再実行コマンドを表示 |
| シードデータ失敗 | `シードデータの投入に失敗しました: {file} - {error}` | 該当ファイルをスキップして続行 |
| コンテナ起動タイムアウト | `{service} の起動がタイムアウトしました（60秒）` | ログ確認コマンドを表示 |
| ディスク容量不足 | `Docker ボリュームの作成に失敗しました` | `docker system prune` を案内 |
