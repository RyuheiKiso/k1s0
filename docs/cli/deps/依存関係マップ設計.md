# 依存関係マップ設計

サービス間の依存関係を自動解析し、Tier 間の依存方向違反を検出する。
本機能は CLIフローのメインメニューに「依存関係マップ」として追加する。

> メインメニューの全体像は [CLIフロー](../flow/CLIフロー.md) を参照。

## 基本方針

- proto ファイル・Kafka トピック定義・import 文・config.yaml から依存関係を自動解析する
- Tier 間の依存方向（system <- business <- service の一方向）の違反を検出する
- テキスト表示・Mermaid ダイアグラム出力の両方をサポートする
- `k1s0 ci-check` にも統合し、CI パイプラインで依存方向違反を自動検出する

## 対話フロー

```
? 操作を選択してください
  ...
> 依存関係マップ
  ...
```

```
[1] 表示対象を選択してください
    > 全体
      Tier を指定
      サービスを指定

[2] (Tier を指定の場合)
    Tier を選択してください
    > system
      business
      service

[2] (サービスを指定の場合)
    サービスを選択してください（複数選択可）
    > [x] auth-server (system)
      [x] order-server (service)
      [ ] payment-server (service)
      [ ] ...

[3] 出力形式を選択してください
    > ターミナル表示
      Mermaid ファイル出力
      両方

[4] (Mermaid 出力の場合)
    出力先パスを入力してください: ____
    デフォルト: docs/diagrams/dependency-map.md

[実行] 解析中...
```

### ターミナル表示の例

```
=== 依存関係マップ ===

[system tier]
  auth-server
    <- (gRPC)  config-server
    <- (REST)  api-registry
    -> (Kafka) notification-server  k1s0.system.auth.login.v1
  saga-server
    -> (gRPC)  auth-server
    -> (gRPC)  notification-server
  graphql-gateway
    -> (gRPC)  auth-server
    -> (gRPC)  config-server
    -> (gRPC)  api-registry

[business tier]
  order-server
    -> (gRPC)  auth-server [system]
    -> (Kafka) k1s0.business.order.created.v1
    <- (Kafka) k1s0.business.inventory.reserved.v1

[service tier]
  order-bff
    -> (GraphQL) order-server [business]
    -> (gRPC)    auth-server [system]

=== 違反検出 ===

  ✗ payment-server (service) -> order-server (business)
    違反: service tier から business tier への gRPC 直接呼び出し
    場所: regions/service/payment/server/rust/payment-server/api/proto/...
    推奨: system tier の GraphQL Gateway または Kafka 経由で連携してください

  ✗ inventory-server (business/logistics) -> order-server (business/sales)
    違反: 異なるドメインコンテキスト間の gRPC 同期呼び出し
    場所: regions/business/logistics/server/go/inventory-server/...
    推奨: Kafka メッセージング経由で非同期連携してください

=== サマリー ===
  解析対象:  12 サービス
  依存関係:  34 件
  違反:       2 件
  警告:       1 件
```

### 警告表示（違反ではないが注意が必要なケース）

```
  ⚠ order-server (business) -> payment-server (business)
    警告: 同一 Tier 内の同期呼び出し（同一ドメイン内であれば許可）
    確認: 両サービスが同一ドメインコンテキストに属するか確認してください
```

## 解析対象

### 1. gRPC 依存（proto import 解析）

proto ファイルの `import` 文からサービス間の依存を検出する。

**解析対象パス**:
```
api/proto/k1s0/{tier}/{domain}/v{n}/*.proto
regions/{tier}/server/{language}/{service}/api/proto/**/*.proto
```

**解析ロジック**:
```
1. 全 proto ファイルを走査
2. import 文からパッケージ名を抽出
   例: import "k1s0/system/auth/v1/auth.proto"
       → 依存先: auth (system tier)
3. service 定義の rpc メソッドから呼び出し関係を特定
4. 依存元サービスの Tier と依存先の Tier を比較して方向を検証
```

**パッケージ → サービス解決ルール**:
```
k1s0.system.{domain}.v{n}    → system tier の {domain}-server
k1s0.business.{domain}.v{n}  → business tier の {domain}-server
k1s0.service.{domain}.v{n}   → service tier の {domain}-server
k1s0.system.common.v{n}      → 共通型定義（依存としてカウントしない）
```

### 2. Kafka 依存（config.yaml トピック解析）

各サービスの `config/config.yaml` から publish/subscribe トピックを解析する。

**解析対象パス**:
```
regions/{tier}/**/config/config.yaml
regions/{tier}/**/config/config.dev.yaml
```

**解析ロジック**:
```
1. config.yaml の kafka.topics セクションを読み込み
2. publish トピックの命名規則から発行元サービスを特定
   例: k1s0.service.order.created.v1
       → 発行元: order (service tier)
3. subscribe トピックから購読先サービスを特定
   例: k1s0.service.payment.completed.v1
       → 購読先: payment (service tier)
4. 発行元と購読先のペアで依存関係を構築
```

**トピック命名規則**:
```
k1s0.{tier}.{domain}.{event-type}.v{n}
```

### 3. REST/GraphQL 依存（コード解析）

ソースコード内の HTTP クライアント・GraphQL クエリからサービス間依存を検出する。

**解析対象パス**:
```
regions/{tier}/**/src/**/*.{rs,go,ts,dart}
```

**検出パターン**:

| 言語 | パターン | 例 |
|------|---------|-----|
| Go | `http.NewRequest`, `url` 文字列リテラル | `"http://auth-server.k1s0-system:8080"` |
| Rust | `reqwest::Client`, `hyper::Uri` | `"http://config-server.k1s0-system:8080"` |
| TypeScript | `fetch`, `axios` | `"${AUTH_SERVER_URL}/api/v1/..."` |
| Dart | `Dio`, `http.Client` | `"http://auth-server.k1s0-system:8080"` |

**サービス名解決ルール**:
```
{service-name}.k1s0-{tier}.svc.cluster.local → {service-name} ({tier})
{service-name}.k1s0-{tier}                   → {service-name} ({tier})（短縮形）
```

### 4. ライブラリ依存（パッケージ定義解析）

各サービスのパッケージ定義ファイルからシステムライブラリの依存を解析する。

**解析対象ファイル**:

| 言語 | ファイル | 検出パターン |
|------|---------|-------------|
| Rust | `Cargo.toml` | `k1s0-{lib-name} = { path = "../../system/library/rust/..." }` |
| Go | `go.mod` | `require github.com/org/k1s0/regions/system/library/go/...` |
| TypeScript | `package.json` | `"@k1s0/{lib-name}": "..."` |
| Dart | `pubspec.yaml` | `k1s0_{lib_name}: path: ...` |

## Tier 間依存ルール

### 許可される依存方向

```
service  → business  → system    （下位 → 上位）
service  → system                （system は共通基盤のため直接参照可）
```

### 同一 Tier 内のルール

| Tier | 同期通信（gRPC/REST） | 非同期通信（Kafka） |
|------|---------------------|--------------------|
| system | 許可（共通基盤間の連携） | 許可 |
| business | 同一ドメインのみ許可 | 許可（異なるドメイン間はこちら） |
| service | 禁止 | 許可 |

### 禁止される依存方向

```
system   → business    （上位 → 下位）
system   → service     （上位 → 下位）
business → service     （上位 → 下位）
```

### 違反の重大度

| 重大度 | 条件 | CI での扱い |
|-------|------|-----------|
| Error | 上位 → 下位の依存 | ビルド失敗（exit 1） |
| Error | service tier 間の同期通信 | ビルド失敗（exit 1） |
| Warning | business tier 異なるドメイン間の同期通信 | 警告表示（exit 0） |
| Info | 同一ドメイン内の同期通信 | 情報表示のみ |

## Mermaid 出力

### 出力ファイル形式

````markdown
# 依存関係マップ

> 自動生成: {生成日時}
> 解析対象: {サービス数} サービス

```mermaid
graph TD
    subgraph system["System Tier"]
        auth[auth-server]
        config[config-server]
        saga[saga-server]
        notif[notification-server]
    end

    subgraph business["Business Tier"]
        order[order-server]
        inventory[inventory-server]
    end

    subgraph service["Service Tier"]
        order-bff[order-bff]
    end

    config -->|gRPC| auth
    saga -->|gRPC| auth
    saga -->|gRPC| notif
    auth -.->|Kafka| notif
    order -->|gRPC| auth
    order -.->|Kafka| inventory
    order-bff -->|GraphQL| order

    %% 違反
    linkStyle 7 stroke:red,stroke-width:3px
    payment -->|gRPC| order
```
````

### 凡例

| 線種 | 意味 |
|------|------|
| `-->` 実線矢印 | 同期通信（gRPC / REST / GraphQL） |
| `-.->` 破線矢印 | 非同期通信（Kafka） |
| 赤色実線 | Tier 間依存方向違反 |

## CI 統合

`k1s0 ci-check` に依存関係チェックを統合する。

### 実行コマンド

```bash
# CI パイプラインでの実行
k1s0 ci-check --deps

# 依存関係チェックのみ単独実行
k1s0 deps --check
```

### CI での出力例

```
[deps] 依存関係を解析中...
[deps] proto ファイル: 24 件解析
[deps] config.yaml: 12 件解析
[deps] ソースコード: 156 ファイル解析
[deps] ライブラリ依存: 12 件解析
[deps]
[deps] Error: payment-server (service) -> order-server (business)
[deps]   service tier から business tier への gRPC 直接呼び出しは禁止されています
[deps]   場所: api/proto/k1s0/service/payment/v1/payment.proto:5
[deps]         import "k1s0/business/order/v1/order.proto"
[deps]
[deps] 結果: 1 error, 0 warnings
[deps] exit code: 1
```

### GitHub Actions 設定例

```yaml
# .github/workflows/ci.yaml
- name: 依存関係チェック
  run: k1s0 deps --check --format github-actions
```

`--format github-actions` を指定すると、GitHub Actions のアノテーション形式で出力する。

```
::error file=api/proto/k1s0/service/payment/v1/payment.proto,line=5::Tier間依存方向違反: service -> business
```

## 解析キャッシュ

大規模リポジトリでの解析パフォーマンスを確保するため、解析結果をキャッシュする。

### キャッシュファイル

```
.k1s0/deps-cache.json
```

### キャッシュ構造

```json
{
  "version": 1,
  "generated_at": "2026-03-06T10:30:00Z",
  "file_hashes": {
    "api/proto/k1s0/system/auth/v1/auth.proto": "sha256:abc123...",
    "regions/service/order/server/rust/order-server/config/config.yaml": "sha256:def456..."
  },
  "dependencies": [
    {
      "source": { "service": "order-server", "tier": "service" },
      "target": { "service": "auth-server", "tier": "system" },
      "type": "grpc",
      "locations": [
        "api/proto/k1s0/service/order/v1/order.proto:3"
      ]
    }
  ],
  "violations": []
}
```

### キャッシュ無効化

以下のいずれかに該当する場合、キャッシュを再構築する。

- 解析対象ファイルのハッシュが変更された
- `--no-cache` フラグが指定された
- キャッシュファイルが存在しない
- キャッシュバージョンが現在の CLI バージョンと一致しない

## エラーハンドリング

| エラー | メッセージ | 対処 |
|-------|----------|------|
| proto ファイル解析エラー | `proto ファイルの解析に失敗しました: {path} ({error})` | 該当ファイルをスキップして続行 |
| config.yaml 読み込みエラー | `config.yaml の読み込みに失敗しました: {path}` | 該当サービスをスキップして続行 |
| Mermaid 出力先が存在しない | `出力先ディレクトリが存在しません: {path}` | ディレクトリを自動作成 |
| 解析対象サービスなし | `解析対象のサービスが見つかりませんでした` | regions/ 配下の構成を確認するよう案内 |
