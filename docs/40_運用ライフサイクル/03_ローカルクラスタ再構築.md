# 03. ローカルクラスタ再構築 Runbook (ADR-POL-002 SoT 復旧経路)

本 Runbook は kind ベースのローカル本番再現スタック `k1s0-local` を**完全に破棄して再構築する**手順を扱う。日常運用の起動 / 停止は [`01_ローカル本番再現スタック.md`](01_ローカル本番再現スタック.md) で扱い、本書はそれと並列の関係にある。

本書が扱う再構築は ADR-POL-002（local-stack を Single Source of Truth とする）の運用周期の中で発生する破壊的操作であり、`down.sh` で PV ごと state を全消失させた後に `up.sh` で canonical 構成から起こし直す。state を持つコンポーネント（gitea / CNPG / Keycloak / OpenBao）の喪失を許容できる範囲を確定させた上で実行する必要があり、判断と手順を間違えると Argo CD の sync 元 git や開発中の DB データを丸ごと失う事故に直結する。本書はその事故を防ぐため、5 段（検出 / 初動 / 復旧 / 原因調査 / 事後処理）に沿って判断点を分離する。

トレーサビリティ:

- 上流: [ADR-POL-002](../02_構想設計/adr/ADR-POL-002-local-stack-single-source-of-truth.md) / IMP-DEV-POL-006
- 関連 Runbook: [01_ローカル本番再現スタック.md](01_ローカル本番再現スタック.md) / [02_WSL2_distributionバックアップ.md](02_WSL2_distributionバックアップ.md)
- 関連 ADR: [ADR-CICD-001](../02_構想設計/adr/ADR-CICD-001-argocd.md) / [ADR-CICD-003](../02_構想設計/adr/ADR-CICD-003-kyverno.md) / [ADR-POL-001](../02_構想設計/adr/ADR-POL-001-kyverno-dual-ownership.md)
- 実行スクリプト: `tools/local-stack/down.sh` / `up.sh` / `backup-cluster.sh` / `verify-cluster.sh` / `known-releases.sh`

## 1. 検出（どのような時に本 Runbook を使うか）

本 Runbook の起動条件は以下の 4 種類に整理できる。判断者は SRE / Platform/Build / dev role 担当者のいずれでもよいが、起動前に必ず判断根拠を Slack `#k1s0-local-stack` に明文で残す。判断根拠の記録が無いと「なぜ rebuild したか」が後で追跡できず、事後処理段階の SHIP_STATUS 更新が形骸化する。

| 起動条件 | 典型的な兆候 | 緊急度 |
|---|---|---|
| CI drift-check 失敗 | `.github/workflows/drift-check.yml` の sync-check または integration job が fail | 中（PR をブロックしない限り計画的に対処） |
| Kyverno admission 拒否多発 | `kubectl logs -n kyverno deploy/kyverno-admission-controller` に block-non-canonical-helm-releases の deny ログ複数件 | 中 |
| 計画的なバージョンアップ | up.sh の `readonly *_VERSION` 群を一括更新する PR の前段クリーン起点 | 低（メンテナンス窓で実施） |
| 開発者個人のリセット要求 | 「ephemeral namespace が肥大化した」「PV が枯渇した」等の自己診断 | 低（個人作業時間内で実施可） |

CI drift-check 失敗の場合、再構築前に**まず PR 内でドリフトを正規化する選択肢**を検討する。新規 helm release が必要なら `up.sh` の apply\_\* 関数追加 + Kyverno policy allow-list 更新 + `known-releases.sh` 確認の 3 点を同一 PR で揃え、CI を pass させてから rebuild する。rebuild は最後の手段であり、ドリフトの本質的な解決ではない（次回再発する）ことを忘れない。

## 2. 初動（rebuild 開始前の必須準備）

破壊的操作前に必ず以下を完了させる。順序は最低限の前提依存に揃えており、飛ばすと state 喪失事故に直結する。

### 2.1 環境前提の確認

```bash
# kubectl context が kind-k1s0-local であること
kubectl config current-context

# 必須ツールの存在確認（全て ~/.local/bin/ 想定）
for t in kind kubectl helm istioctl docker python3 git curl; do
    which "$t" >/dev/null && echo "$t: OK" || echo "$t: 未インストール"
done
```

`istioctl` が未インストールの場合は先に導入する（バージョンは [`up.sh`](../../tools/local-stack/up.sh) の Istio 系 chart バージョンと揃える）。

### 2.2 関係者通知

Slack `#k1s0-local-stack` に以下を投稿。観測中の dev role 担当者は本通知を見て個人作業を pause する判断ができる。

> ローカルクラスタ `k1s0-local` を再構築します（推定 30〜40 分）。
> 起動条件: `CI fail / 計画 upgrade / 個人リセット / その他` のいずれか
> 想定影響: argocd / tier1〜3 / Backstage / Grafana / gitea が一時停止
> 再構築完了予定: `YYYY-MM-DD HH:MM`

### 2.3 完全バックアップ取得

```bash
# 既定の出力先 /tmp/k1s0-backup-YYYYMMDD-HHMMSS/ を使う
./tools/local-stack/backup-cluster.sh

# integrity 確認（gitea-data 約 440MB / 他は 1〜10MB が正常レンジ）
ls -lh /tmp/k1s0-backup-*

# tar.gz アーカイブ（任意）
LATEST=$(ls -dt /tmp/k1s0-backup-2026* | head -1)
tar -czf "${LATEST}.tar.gz" -C /tmp "$(basename ${LATEST})"
```

`backup-cluster.sh` は CRITICAL 4 種（gitea / pg-state / keycloak / openbao）と IMPORTANT 4 種（helm releases / argocd Apps / 手動 apply / critical secrets）と INFO 2 種（namespace dumps / cluster metadata）の合計 10 カテゴリを退避する。出力ディレクトリの整合性は `du -sh` で確認できるサイズ感（合計 ~470MB が標準）から逸脱しないこと。Gitea data が極端に小さい（数 MB 以下）場合、apply\_gitea が `infra/gitops/local-stack/gitea-deployment.yaml` の PVC を作る前に bootstrap_gitea_content が走った可能性があるため、`down.sh` 前に状況を再確認する。

## 3. 復旧（再構築の実行）

### 3.1 down

```bash
./tools/local-stack/down.sh
```

`down.sh` は `kind delete cluster --name k1s0-local` を実行し、PV を含めて全 state を消失させる。本 Runbook で扱う唯一の不可逆操作であり、実行前に 2.3 のバックアップが完了していることを再確認する。

### 3.2 up

```bash
# 通常運用は full + strict（infra-ops 相当の全レイヤ + Kyverno enforce）
./tools/local-stack/up.sh --role full --mode strict

# 検証用に軽量で起こす場合
./tools/local-stack/up.sh --role tier1-rust-dev --mode dev

# 観測性のみ後追い
./tools/local-stack/up.sh --no-cluster --layers observability
```

`up.sh` は内部で 22 レイヤの apply\_\* を順次呼ぶ。`apply_argocd` の中で `bootstrap_gitea_content`（gitea に admin user 作成 + 現 working tree を push）と `apply_argocd_appsets`（GitHub URL → local gitea URL 変換 + 8 ApplicationSet 適用）が連続して走るため、`apply_argocd` だけは他レイヤより数分長い。`apply_kyverno` は `--mode strict` で `block-non-canonical-helm-releases` を `Enforce` に patch する。Kyverno 起動完了後の helm install はすべて canonical allow-list に対する admission 検査を経るため、`up.sh` 自身が許可されないリリースを `helm upgrade --install` しないことが前提となる（`tools/local-stack/known-releases.sh` の出力と Kyverno allow-list が一致していることを CI が検証する）。

### 3.3 state 復元

#### 3.3.1 gitea repo 内容

`apply_argocd` 内の `bootstrap_gitea_content` が現 working tree の `HEAD` を push するため、`up.sh` 完了時点で gitea には commit 済み内容が反映されている。**未 commit の変更は失われる**ため、rebuild 前に必要な変更は `git commit` を済ませておく。バックアップから復元したい場合は以下:

```bash
# 旧 gitea data から差分があれば extract して push
LATEST=$(ls -dt /tmp/k1s0-backup-2026* | head -1)
ls "${LATEST}/gitea-data/git/repositories/argocd/k1s0.git/"
# 必要に応じて bare repo として clone → 内容比較 → push
```

#### 3.3.2 Keycloak realm

H2 ファイルを hot-copy で退避してあるため、Keycloak が起動してから以下で復元する。Keycloak 起動中は H2 が file lock を握っているため、Keycloak Pod を一旦停止してからファイルを差し替え、再起動する。

```bash
LATEST=$(ls -dt /tmp/k1s0-backup-2026* | head -1)
KC_POD=$(kubectl -n keycloak get pod -l app=keycloak -o jsonpath='{.items[0].metadata.name}')

# Keycloak を一時停止
kubectl -n keycloak scale deployment keycloak --replicas=0
kubectl -n keycloak wait --for=delete pod -l app=keycloak --timeout=60s

# H2 ファイルを差し替え（ephemeral pod 経由）
# ※ Keycloak が PVC を使っていない場合は復元不要（state は emptyDir で消えるため）
# ※ 復元が必要な実運用は別途 chart 側で PVC を有効化する PR を起票する

# 再開
kubectl -n keycloak scale deployment keycloak --replicas=1
```

リリース時点の Keycloak chart は dev mode で H2 + emptyDir のため、復元は最低限の master realm のみ意味がある。realm/user の本格運用は production の外部 IdP に委譲する設計（[ADR-SEC-001](../02_構想設計/adr/ADR-SEC-001-keycloak.md)）であり、ローカル H2 復元の優先度は低い。

#### 3.3.3 pg-state（孤児リソース）

監査の結果 `k1s0-tier1/pg-state` はリポジトリ内の chart から参照されておらず、過去の手動実験の残骸であることが確認された（[ADR-POL-002](../02_構想設計/adr/ADR-POL-002-local-stack-single-source-of-truth.md) F1）。`up.sh` は再構築時に `pg-state` を作らない。dev tooling 用 DB は `cnpg-system/k1s0-postgres`（apply\_cnpg が `manifests/60-cnpg/k1s0-cluster.yaml` を apply）が単一の正規 DB となる。バックアップに `dbs/pg-state/dapr.dump` が残っているが、Dapr が再構築後に同 DB に依存していなければ復元不要。

### 3.4 argocd Applications の Sync 待ち

```bash
# ApplicationSet が Application を generate するまで（数分）
kubectl -n argocd get applicationset -w

# 各 Application が Synced + Healthy になるまで（10〜20 分）
kubectl -n argocd get applications.argoproj.io -w
```

未 Synced が長時間続く場合は `kubectl -n argocd logs deploy/argocd-application-controller` を確認する。gitea repo の内容と Application の `spec.source.path` が一致していない（`bootstrap_gitea_content` の push が 古い revision で止まっている等）が典型的な原因。

## 4. 原因調査（drift 起因の rebuild の場合のみ）

drift 起因の rebuild では「**なぜ canonical 経路を通らない helm install が発生したか**」を必ず追跡する。追跡を省略すると同じ drift が次回 PR で再発し、CI fail → rebuild → 再発のループに陥る。

調査の起点は次の 3 点に絞る:

1. **どの release が drift だったか** — `backup-cluster.sh` 出力の `helm-releases/_inventory.txt` と `tools/local-stack/known-releases.sh` の diff を取る。drift release は canonical 集合外の release 名として現れる。
2. **誰がいつ install したか** — `helm history <release-name> -n <namespace>` で revision 1 のタイムスタンプとシェル履歴を突き合わせる。
3. **なぜ up.sh 経路を通らなかったか** — 該当開発者にヒアリングし、(a) `up.sh` に layer が足りなかった / (b) 急ぎで `helm install` を選んだ / (c) 一時的な検証のつもりが定着した、のどれかを特定する。

(a) の場合は `up.sh` への apply\_\* 追加 PR を起票する。(b) (c) の場合は `tmp-*` / `dev-*` ephemeral namespace の使い方を当該開発者にオンボーディング（runbook 末尾の「補足」節を案内）する。

## 5. 事後処理（再構築完了後の確認と通知）

### 5.1 機械的な検証

```bash
# verify-cluster.sh が PASS で終わること
./tools/local-stack/verify-cluster.sh
echo "exit=$?"

# drift-check の sync-check を local 再現
diff <(./tools/local-stack/known-releases.sh | sort -u) \
     <(python3 -c '
import yaml
with open("infra/security/kyverno/block-non-canonical-helm-releases.yaml") as f:
    for d in yaml.safe_load_all(f):
        if d and d.get("kind")=="ClusterPolicy":
            for r in d["spec"]["rules"]:
                for c in r.get("validate",{}).get("deny",{}).get("conditions",{}).get("all",[]):
                    if c.get("operator")=="NotIn":
                        for v in c["value"]: print(v)
' | sort -u)
```

`verify-cluster.sh` は 7 カテゴリの check（context / cluster / drift / 重要 ns / Kyverno policy / argocd Apps / 主要 endpoint）を順に走らせ、FAIL 0 件で exit 0 を返す。drift-check の sync-check は `known-releases.sh` の出力と Kyverno policy allow-list が同一であることを diff で確認する。CI 側の同検査と等価。

### 5.2 SHIP_STATUS 更新

`docs/SHIP_STATUS.md` の「ローカル本番再現スタック」節（または対応する H 系列項目）に、再構築日 / 起動条件 / 投入した PR 番号 / 検証結果（verify-cluster.sh の PASS/FAIL カウント）を追記する。記録形式は既存の H1〜H3 エントリに揃える。

### 5.3 関係者への完了通知

Slack `#k1s0-local-stack` に以下を投稿。

> ローカルクラスタ `k1s0-local` の再構築が完了しました。
> 起動条件: `2 で記載したもの`
> 結果: verify-cluster.sh `PASS=N WARN=N FAIL=N`
> argocd Applications: `<Synced 件数> / <総数>` Healthy
> 影響範囲: `停止していた範囲`
> 個人作業の再開可否: 再開可（ephemeral namespace を使った実験は `tmp-*` / `dev-*` で）

### 5.4 drift 起因の場合の追加対応

drift 起因の rebuild では、4 で特定した原因に応じた追加 PR を本日中に起票する。`up.sh` への apply\_\* 関数追加 / Kyverno policy allow-list 更新 / オンボーディング資料更新のいずれかが典型。追加 PR を当日中に起票しないと再発防止策の優先度が落ちることが過去事例から判明しているため、再構築の事後処理は PR 起票までを 1 セットとする。

## 補足: ephemeral namespace の使い方

開発者個人の探索的 `helm install` / `kubectl apply` は ADR-POL-002 で明示的に許可されているが、対象 namespace を `tmp-*` または `dev-<username>-*` に限定する必要がある。本 Runbook の防止策（Kyverno policy + CI）はこれらの namespace を検査対象から除外しているため、開発者は以下のように使う。

```bash
# 一時 namespace を作る（毎回 unique にする）
NS="tmp-$(date +%s)"
kubectl create namespace "${NS}"

# helm install は自由
helm install foo bitnami/foo -n "${NS}"

# 終わったら必ず破棄（CI が tmp-*/dev-* の長期残存を fail させる想定）
kubectl delete namespace "${NS}"
```

破棄を忘れた `tmp-*` / `dev-*` namespace は次回 rebuild で消える（PV ごと down される）が、それ以前にも `tools/local-stack/cleanup-ephemeral.sh`（リリース後に提供予定）で個別に掃除する習慣をつける。
