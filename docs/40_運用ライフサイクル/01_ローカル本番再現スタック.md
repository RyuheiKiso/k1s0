# 01. ローカル本番再現スタック Runbook

本 Runbook は `tools/local-stack/` で配備される kind ベースのローカル本番再現スタックの**起動・停止・状態確認・トラブルシュート**を一括で扱う。対象スタックは Argo CD・cert-manager・MetalLB・Istio Ambient・Kyverno・SPIRE・Dapr operator・flagd・CNPG・Strimzi Kafka・MinIO・Valkey・OpenBao・Backstage・Grafana/Loki/Tempo・Keycloak の 17 レイヤで、配備順は接頭辞数値で固定されている（[`tools/local-stack/manifests/README.md`](../../tools/local-stack/manifests/README.md) 参照）。

## 1. 通常運用（incident でない場合）

### 1.1 起動

```bash
# 役割を選んで kind + 役割別配備セットを一括起動
./tools/local-stack/up.sh --role tier1-rust-dev

# 別役を試す場合は切替（symlink + cone 両方を更新）
./tools/sparse/checkout-role.sh tier3-web-dev
./tools/local-stack/up.sh --role tier3-web-dev

# 特定レイヤのみ追加配備（例: 観測性のみ後追い）
./tools/local-stack/up.sh --role tier1-rust-dev --layers observability

# 既存 cluster に対して manifest だけ当て直す（kind 再作成しない）
./tools/local-stack/up.sh --no-cluster --layers cert-manager,argocd
```

役割と配備レイヤの対応は `up.sh` の `ROLE_LAYERS` 連想配列で固定されている。`docs-writer` は `cni + cert-manager` のみで起動が早い。`infra-ops` / `full` は observability + Keycloak まで全載せで、初回 5〜10 分かかる。

### 1.2 状態確認

```bash
./tools/local-stack/status.sh
```

このコマンドは次の 5 種類を順に表示する: kind cluster 一覧 / nodes / k1s0.io/layer ラベルの付いた namespace / 起動途中の Pod (Phase != Running, Succeeded) / helm releases / NodePort で公開されているサービス。NodePort 列は `argocd/argocd-server:30080,80,443` のように出力され、host ブラウザで `http://localhost:30080` に到達できることを意味する。

### 1.3 停止と完全リセット

```bash
# kind cluster を破棄（PV ごと消える）
./tools/local-stack/down.sh
```

PV を残したい場合は、`docker volume ls | grep kind` で kind 関連 volume を特定して `kind delete cluster` 前に export する。デフォルト構成では PV は emptyDir 相当のため `down.sh` で全消失する前提で運用する。

### 1.4 アクセスポイント

| URL | 用途 | 初期認証 |
|---|---|---|
| `http://localhost:30080` | Argo CD UI | `admin` / `kubectl -n argocd get secret argocd-initial-admin-secret -o jsonpath='{.data.password}' \| base64 -d` |
| `http://localhost:30700` | Backstage | basic auth（dev image 既定） |
| `http://localhost:30300` | Grafana | `admin` / `k1s0-local-dev-password` |
| `http://localhost:8200` | OpenBao（kind 経由ではない場合） | token: `dev-root-token` |

NodePort は kind の `extraPortMappings` で host にバインドされている（[`tools/local-stack/kind-cluster.yaml`](../../tools/local-stack/kind-cluster.yaml) §`extraPortMappings`）。`30080` が他のプロセスで占有されている場合は kind 起動時にエラーが出るため §3.4 を参照。

## 2. アクセス経路の切り戻し（Dev Container 内から）

Dev Container は `docker-outside-of-docker` Feature で host docker.sock を共有しているため、container 内の `kubectl` は host の kind cluster をそのまま操作する。`KUBECONFIG` が空でも `kind export kubeconfig --name k1s0-local` を一度実行すれば `~/.kube/config` に context が書かれる。

```bash
kind export kubeconfig --name k1s0-local
kubectl config use-context kind-k1s0-local
kubectl get nodes
```

## 3. トラブルシュート（5 段構成）

各サブセクションは「検出 / 初動 / 復旧 / 根本原因調査 / 事後処理」の 5 段で書く。アラートが将来接続された時点で `runbook_url` の深リンクが各見出しを指す。

### 3.1 kind cluster が起動しない

#### 検出

- `./tools/local-stack/up.sh --role <role>` 実行時に `ERROR: failed to create cluster` で停止
- `kind create cluster` 単独実行でも `failed to apply kubeadm config` 等が出る
- `docker info` は成功するが `kind get clusters` が空のまま

#### 初動

```bash
# kind 既存 cluster の残骸確認
kind get clusters
docker ps -a --filter "label=io.x-k8s.kind.cluster"
# 残骸がある場合は強制削除
kind delete cluster --name k1s0-local
docker ps -a --filter "label=io.x-k8s.kind.cluster=k1s0-local" -q | xargs -r docker rm -f
```

#### 復旧

```bash
# host 側の前提を再確認
docker info | grep -i "Cgroup Driver"   # systemd or cgroupfs
# WSL2 の場合は systemd が立っているか
ps -p 1 -o comm=                          # 期待値: systemd

# 再作成
./tools/local-stack/up.sh --role <role>
```

WSL2 で `ps -p 1` が `init` や `docker-init` の場合、`/etc/wsl.conf` に `[boot] systemd=true` が無い。Windows host で `wsl --shutdown` 後に再接続。詳細は [`docs/05_実装/50_開発者体験設計/05_ローカル環境基盤/01_WindowsWSL2環境構成.md`](../05_実装/50_開発者体験設計/05_ローカル環境基盤/01_WindowsWSL2環境構成.md) §3.2 参照。

#### 根本原因調査

- **inotify limit 不足**: `cat /proc/sys/fs/inotify/max_user_instances` が 128 等の小さな値だと kind 起動時に `too many open files` で失敗する。`/etc/sysctl.conf` に `fs.inotify.max_user_instances=8192` / `fs.inotify.max_user_watches=524288` を追加して `sysctl -p`
- **port 80/443/30080 の占有**: kind の `extraPortMappings` が host 側で衝突している。`ss -tlnp \| grep -E ':(80|443|30080)'` で占有プロセスを特定
- **kindest/node イメージ未取得**: `docker images \| grep kindest/node` が空だと初回 pull が長く、タイムアウトに見える。`docker pull kindest/node:v1.31.4` で先 pull

#### 事後処理

- 同一症状が 3 回以上発生した場合は本ファイルに新セクションを追加
- inotify / port / image pull のいずれかが原因なら、`postCreate.sh` で起動前チェックを追加して再発防止（`tools/devcontainer/postCreate.sh` §preflight）

### 3.2 cert-manager / Argo CD 等の Pod が長時間 Pending

#### 検出

- `kubectl get pods -A` で `STATUS=Pending` の Pod が 5 分以上継続
- `kubectl describe pod` の Events に `0/4 nodes are available: insufficient memory` 等

#### 初動

```bash
kubectl describe pod -n cert-manager cert-manager-<pod>
kubectl top nodes 2>/dev/null    # metrics-server があれば
docker stats --no-stream         # ホスト側のメモリ・CPU 余裕
```

#### 復旧

```bash
# 一時的な解放: 観測性 / Backstage を停止
helm uninstall grafana loki tempo -n observability 2>/dev/null
helm uninstall backstage -n backstage 2>/dev/null

# 必要なら kind worker ノードを再起動
docker restart k1s0-local-worker k1s0-local-worker2 k1s0-local-worker3
```

#### 根本原因調査

- **48GB ホスト RAM の枯渇**: Rust の cargo build + rust-analyzer + kind 全レイヤで上限到達。`free -h` の `available` が 8GB を切っていれば該当
- **kind ノードへの Pod 偏り**: 特定の worker に Pod が集中。`kubectl get pods -A -o wide \| sort -k8` でノード分布を確認
- **PVC が provision されない**: ストレージドライバの問題。`kubectl get pvc -A` で `STATUS=Pending` を確認

#### 事後処理

- 役割別配備セット（`up.sh` の `ROLE_LAYERS`）から重い Pod を除外する案を検討
- 48GB 越えの解消が頻発するなら、`.wslconfig` の `memory` を調整する Runbook（[`02_WSL2_distributionバックアップ.md`](02_WSL2_distributionバックアップ.md) と同階層に追記予定）を参照

### 3.3 Dapr sidecar が pod に注入されない

#### 検出

- pod を `dapr.io/enabled: true` annotation 付きで起動しても sidecar コンテナが現れない
- アプリが `localhost:3500` に接続できず connection refused

#### 初動

```bash
# Dapr sidecar injector の状態
kubectl get pods -n dapr-system
kubectl logs -n dapr-system deployment/dapr-sidecar-injector --tail=200

# 対象 pod の admission 経由イベント
kubectl get events --sort-by=.lastTimestamp | grep -i dapr
```

#### 復旧

```bash
# Dapr 再インストール
./tools/local-stack/up.sh --no-cluster --layers dapr
# pod を再起動して再度 admission を通す
kubectl rollout restart deployment <your-app>
```

#### 根本原因調査

- **mTLS 証明書の期限切れ**: dev mode では root cert が pod 起動時に再生される。`kubectl get secret dapr-trust-bundle -n dapr-system -o yaml` の `tls.crt` の `notAfter` を確認
- **Webhook が Kyverno と衝突**: Kyverno の admission policy が Dapr webhook より先に reject している。`kubectl get clusterpolicies` で関連ポリシーを確認

#### 事後処理

- 衝突が再発するなら、Kyverno の policy exclusion に `dapr-system` namespace を加える policy パッチを `tools/local-stack/manifests/35-kyverno/` に追記

### 3.4 ホスト port 30080 / 30700 / 30300 が衝突

#### 検出

- `up.sh` 後にブラウザで `http://localhost:30080` 等にアクセスすると別アプリが応答する
- `ss -tlnp` で kind 由来でないプロセスがそのポートを LISTEN

#### 初動

```bash
ss -tlnp | grep -E ':(30080|30700|30300)'
# 該当 PID を確認
sudo lsof -i :30080
```

#### 復旧

衝突プロセスを停止できない場合は kind 側を別ポートに逃がす。`tools/local-stack/kind-cluster.yaml` の `extraPortMappings` を編集し、`hostPort` を `40080` 等に変更。`up.sh` 再実行時は `down.sh` でクラスタを作り直す（kind は extraPortMappings の動的変更を許さない）。

#### 根本原因調査

- 別の kind cluster や開発サーバが同 port を使っている。常用エディタ・dev server・他プロジェクトの kind を停止
- WSL2 と Windows 双方で 30080 が LISTEN している場合、Windows 側を停止するか `.wslconfig` の `localhostForwarding=false` を一時的に切替

#### 事後処理

- 30080 / 30700 / 30300 を共通占有することがチームで多発するなら、各役割で衝突しない port 番号体系を `kind-cluster.yaml` に固定し ADR で明文化

### 3.5 Argo CD の管理者パスワードが分からない / リセットしたい

#### 検出

- `kubectl get secret argocd-initial-admin-secret -n argocd` が `not found`（既に削除済み）

#### 初動

```bash
# argocd-server の bcrypt パスワードを直接書き換える経路で reset
# 1. 任意のパスワードを bcrypt 化
NEW_PW='admin-local-dev-password'
NEW_HASH=$(htpasswd -nbBC 10 "" "$NEW_PW" | tr -d ':\n' | sed 's/\$2y/\$2a/')

kubectl -n argocd patch secret argocd-secret \
  -p "{\"stringData\": {\"admin.password\": \"$NEW_HASH\", \"admin.passwordMtime\": \"$(date +%FT%T%Z)\"}}"
kubectl -n argocd rollout restart deployment argocd-server
```

#### 復旧

```bash
# パスワード変更が反映されるまで 1 分待つ
sleep 60
argocd login localhost:30080 --username admin --password "$NEW_PW" --insecure
```

#### 根本原因調査

- 初期 secret の自動削除タイミングを忘れた場合に発生
- 本番では Keycloak OIDC 連携前提なので、ローカル運用のみで起こる事象

#### 事後処理

- 初期パスワードを Backstage 等に表示するようにすれば再発防止になるが、ローカル開発専用かつ単一開発者運用なら Runbook を引く運用で十分
