# ADR-0015: レート制限フレームワーク（k1s0-rate-limit）

## ステータス

承認済み（Accepted）

## 日付

2026-01-30

## コンテキスト

マイクロサービス間の通信において、過負荷からサービスを保護するためのレート制限が必要である。既存の `k1s0-resilience` は障害発生**後**の回復パターン（サーキットブレーカー、リトライ、タイムアウト）を提供しているが、障害**前**の負荷制御機構が欠如していた。

## 決定

Tier 2 パッケージ `k1s0-rate-limit` を新設し、2つのアルゴリズムを全5バックエンド言語で提供する。

1. **トークンバケット** - ロックフリー CAS（`AtomicU64`）実装。バースト許容、O(1) 計算量
2. **スライディングウィンドウ** - タイムスタンプベースの精密制御。バーストなし

ミドルウェア統合: gRPC インターセプター（HTTP 429 + `Retry-After` / gRPC `RESOURCE_EXHAUSTED`）。キー抽出戦略として `ip`, `user_id`, `api_key` を設定可能。

### k1s0-resilience との棲み分け

| パッケージ | 役割 | タイミング |
|-----------|------|-----------|
| k1s0-rate-limit | 負荷制御（レート制限） | 障害**前** |
| k1s0-resilience | 障害回復（サーキットブレーカー等） | 障害**後** |

セマフォ（同時実行数制限）は「時間あたりのリクエスト数」ではなく「同時リクエスト数」を制限するため `k1s0-resilience` に残す。

### 選択肢の比較

| 選択肢 | 長所 | 短所 |
|--------|------|------|
| A. トークンバケット + スライディングウィンドウ（採用） | 用途に応じた選択が可能 | 2アルゴリズムの理解が必要 |
| B. トークンバケットのみ | シンプル | 精密な制御が不可能 |
| C. k1s0-resilience に統合 | パッケージ数を削減 | 責務が混在、パッケージが肥大化 |

### 採用理由

- ロックフリー CAS によりホットパス上のロック競合を回避
- tonic/axum への直接依存を避け、軽量な依存ツリーを維持
- YAML 設定駆動で環境変数を使用しない（k1s0 規約準拠）

## 帰結

### 正の帰結

- API ゲートウェイから内部 gRPC まで、統一的なレート制限を適用可能
- `k1s0_rate_limit_*` メトリクスにより Prometheus/Grafana で負荷状況を可視化
- パイプライン構成: `Request -> Rate Limit -> Circuit Breaker -> Timeout -> Processing`

### 負の帰結

- 分散レート制限（Redis 共有カウンター等）は初期スコープ外
- アルゴリズム選択のガイダンスが必要

### リスクと軽減策

| リスク | 軽減策 |
|--------|--------|
| インスタンス単位のレート制限では分散環境で不正確 | 将来の Redis バックエンド拡張を設計に考慮済み |
| 不適切な設定値による過剰制限 | 設計書にユースケース別の推奨アルゴリズムガイドを記載 |
