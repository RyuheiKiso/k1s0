# ADR-0017: AST ベース Lint エンジンへの移行

## ステータス

承認済み（Accepted）

## 日付

2026-01-30

## コンテキスト

k1s0 の Lint エンジンは grep/正規表現ベースでパターン検出を行っていたが、以下の誤検知が問題になっていた：

- コメント内の `unwrap()` が K029 違反として検出される
- ドキュメント内の SQL 例が K050 違反として検出される
- テストコード内の `panic!` が K029 違反として検出される
- 文字列リテラル内の `process.env` が K020 違反として検出される

grep ベースのアプローチでは、ソースコードの構文的コンテキスト（コメント/文字列/テスト）を区別できないことが根本原因である。

## 決定

tree-sitter を用いた AST（抽象構文木）ベースの Lint エンジンに移行する。以下の6ルールを AST 解析のデフォルト対象とする：

- **K020** - 環境変数参照の禁止
- **K022** - Clean Architecture 依存方向違反
- **K026** - Domain 層でのプロトコル型使用
- **K029** - 本番コードでの panic/unwrap/expect
- **K050** - SQL インジェクションリスク
- **K053** - 機密データのログ出力

対応言語: Rust, Go, TypeScript, Python, C#, Kotlin。

### 選択肢の比較

| 選択肢 | 長所 | 短所 |
|--------|------|------|
| A. tree-sitter AST（採用） | 誤検知排除、構文認識、テスト/コメント区別 | パース処理のオーバーヘッド、一部言語未対応 |
| B. grep/正規表現（従来） | 高速、実装が単純 | コメント/文字列/テスト内の誤検知 |
| C. 言語固有の AST パーサー | 最高精度の解析 | 言語ごとに別実装、保守コスト大 |

### 採用理由

- tree-sitter は多言語対応の統一 API を提供し、6言語を1つのフレームワークで扱える
- `AstContext` による `is_non_code()` / `is_in_test()` 判定で誤検知を構造的に排除
- パーサープーリングとクエリキャッシュにより実行性能を最適化
- `--fast` フラグで grep フォールバックを提供し、CI での高速実行を維持

## 帰結

### 正の帰結

- 誤検知の大幅削減により、開発者の lint 結果への信頼性が向上
- テストコード・コメント・文字列内のパターンを正しく無視
- tree-sitter のインクリメンタルパーシングにより LSP 統合（リアルタイム診断）への拡張が可能

### 負の帰結

- tree-sitter 0.25（ABI version 15）への依存が発生
- Dart, YAML, Dockerfile は ABI バージョン不一致のため AST 未対応（grep フォールバック）
- 型推論、クロスファイル解析、re-export 追跡は対象外

### リスクと軽減策

| リスク | 軽減策 |
|--------|--------|
| tree-sitter の ABI 互換性問題 | ABI バージョンチェックをビルド時に実施。未対応言語は grep フォールバック |
| AST パースによる実行時間増加 | `--fast` フラグで grep ベースに切り替え可能。コンパイル時に `--no-default-features` で AST を無効化可能 |
