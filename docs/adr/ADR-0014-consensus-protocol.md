# ADR-0014: コンセンサスプロトコル機構（k1s0-consensus）

## ステータス

承認済み（Accepted）

## 日付

2026-01-30

## コンテキスト

マイクロサービス環境では、リーダー選出、分散ロック、分散トランザクション（Saga）といったコンセンサスパターンが必要になる。しかし、etcd や ZooKeeper のような専用コンセンサスシステムを導入すると、インフラの複雑性とオペレーションコストが増加する。

k1s0 では PostgreSQL と Redis が既に標準インフラとして存在しており、これらを活用したコンセンサス機構が求められていた。

## 決定

Tier 2 パッケージ `k1s0-consensus` を新設し、以下の3つの機構を全5バックエンド言語（Rust, Go, C#, Python, Kotlin）で提供する。

1. **リーダー選出** - PostgreSQL ベースのリース機構（`fw_m_leader_lease` テーブル）。ハートビートによる生存確認と自動フェイルオーバー
2. **分散ロック** - DB ベースと Redis ベースの2実装。TTL による自動解放とフェンシングトークンによる整合性保証
3. **Saga パターン** - オーケストレーション（DB 永続化 + 補償 + Dead Letter Queue）とコレオグラフィ（`k1s0-domain-event` EventBus 経由）の両方式

依存先: `k1s0-db`, `k1s0-domain-event`, `k1s0-observability`（Tier 2 間依存として例外的に許可）。

### 選択肢の比較

| 選択肢 | 長所 | 短所 |
|--------|------|------|
| A. PostgreSQL + Redis ベース（採用） | 追加インフラ不要、既存スキルで運用可能 | 大規模クラスタでのスケーラビリティに限界 |
| B. etcd / ZooKeeper | 高いスループット、Raft 合意の信頼性 | インフラ追加、運用コスト増大 |
| C. 各チームが独自実装 | 要件に最適化可能 | 品質・実装のばらつき、車輪の再発明 |

### 採用理由

- PostgreSQL と Redis は k1s0 の必須インフラであり、追加の運用負荷がない
- フェンシングトークンにより GC ポーズやネットワーク遅延に対する整合性を保証
- Saga の両パターン提供により、集中制御型と疎結合型の両ユースケースに対応

## 帰結

### 正の帰結

- 新規インフラの追加なしに分散コンセンサスパターンを利用可能
- Saga のべき等性ガイドライン（`saga_id` + `step_index` による重複排除）を標準化
- 全5言語で統一 API を提供し、言語間の学習コストを削減

### 負の帰結

- Tier 2 間の依存（k1s0-db, k1s0-domain-event）を例外的に許可する必要がある
- PostgreSQL ベースのリーダー選出は、数千ノード規模には不向き

### リスクと軽減策

| リスク | 軽減策 |
|--------|--------|
| DB ベースリース更新の遅延 | ハートビート間隔・リースTTL を設定可能にし、環境に応じた調整を許可 |
| Redis 分散ロックの一貫性（Redlock 問題） | 単一 Redis インスタンスでの利用を推奨。強い一貫性が必要な場合は DB ロックを案内 |
