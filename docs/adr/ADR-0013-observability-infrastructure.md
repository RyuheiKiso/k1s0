# ADR-0013: 可観測性基盤の技術選定

## ステータス

承認済み（Accepted）

## 日付

2026-01-30

## コンテキスト

k1s0 プラットフォームでは、全サービスに OpenTelemetry ベースの観測性（ログ/トレース/メトリクス）を標準として求めている。しかし、開発者がローカル環境でテレメトリデータを確認し、本番同等の可観測性を体験できる統合基盤が存在しなかった。

各チームが個別に観測ツールをセットアップすると、以下の問題が発生する：

- 設定の不統一により、環境間でのデバッグが困難になる
- 本番環境と異なるツールチェーンを使うことで、運用スキルが育たない
- ダッシュボード・アラートルールの共有ができない

## 決定

以下のコンポーネントで構成される Observability スタックを `observability/` ディレクトリに一元管理する。

| コンポーネント | 役割 | 選定理由 |
|--------------|------|---------|
| **OTEL Collector** | テレメトリデータの受信・処理・転送 | OpenTelemetry 標準。ベンダーロックインを回避 |
| **Jaeger** | 分散トレースの保存・検索・可視化 | CNCF Graduated。OTel との親和性が高い |
| **Loki** | ログの集約・検索 | Grafana エコシステムとの統合。軽量 |
| **Prometheus** | メトリクスの収集・保存・クエリ | 業界標準。豊富なアラートルール |
| **Grafana** | ダッシュボード・アラート・統合可視化 | 全データソースを統合可視化。プロビジョニング対応 |

### 選択肢の比較

| 選択肢 | 長所 | 短所 |
|--------|------|------|
| A. OTEL + Jaeger + Loki + Prometheus + Grafana（採用） | CNCF 標準、ベンダーニュートラル、プロビジョニング可能 | コンポーネント数が多い |
| B. Datadog / New Relic 等 SaaS | セットアップが簡単 | ベンダーロックイン、ローカル開発で使いにくい、コスト |
| C. Elastic Stack (ELK) | 統合的なログ・メトリクス | リソース消費が大きい、ライセンス変更リスク |

### 採用理由

- CNCF プロジェクトの組み合わせで標準的かつベンダーニュートラル
- docker compose で即座にローカル起動でき、開発体験が良い
- Grafana のプロビジョニング機能でダッシュボード・アラートをコード管理可能
- 本番環境でも同一ツールチェーンを使用でき、スキルの移転性が高い

## 帰結

### 正の帰結

- ローカル・ステージング・本番で同一の観測ツールチェーンを使用
- ダッシュボード・アラートルールをコードとしてバージョン管理
- 環境別設定（サンプリング率等）を `observability/otel/collector/config/` で一元管理
- 言語別の接続設定例を `observability/otel/instrumentation/` で提供

### 負の帰結

- Docker Desktop が必要（ただしサービス開発自体には不要）
- コンポーネント数が多く、初期理解のコストがある

### リスクと軽減策

| リスク | 軽減策 |
|--------|--------|
| ローカルでのリソース消費 | docker compose で必要なサービスのみ起動可能にする |
| Jaeger の将来的な非推奨化 | OTEL Collector を中心に据え、バックエンド差し替えを容易にする |
