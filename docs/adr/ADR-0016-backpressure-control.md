# ADR-0016: バックプレッシャー制御機構

## ステータス

承認済み（Accepted）

## 日付

2026-01-30

## コンテキスト

k1s0 プラットフォームの複数レイヤーでバックプレッシャー制御が欠如しており、以下の問題が発生していた：

- gRPC ストリーミングでバッファ溢れによる OOM リスク
- ドメインイベントバスで購読者遅延時のサイレントなイベント消失（Lagged エラー）
- DB コネクションプール待ちの無制限キューイング
- フロントエンドからバックエンドへの無制限リクエスト送信

## 決定

プラットフォーム全体で6つのバックプレッシャー制御機構を導入する。

1. **gRPC ストリーミング** - `FlowControlledStream`（セマフォベースの送信バッファリング + 遅延プロデューサー検出）
2. **レート制限** - `k1s0-rate-limit` パッケージ（[ADR-0015](ADR-0015-rate-limiting.md) 参照）
3. **ドメインイベントバス** - `BackpressureStrategy` 列挙型（`DropOldest`/`Block`/`Reject`/`DropNewest`）+ リカバリバッファ
4. **キュー深度オブザーバビリティ** - 全内部キューの Prometheus メトリクス公開
5. **フロントエンド API クライアント** - `RequestThrottle`（React/Flutter/Android 共通。`maxRequestsPerSecond`, `maxConcurrent`, `maxQueueSize` を設定可能）
6. **DB コネクションプール** - `BackpressuredPool` ラッパー（`max_waiting` キュー上限と即時拒否）

### 選択肢の比較（イベントバスオーバーフロー戦略）

| 選択肢 | 長所 | 短所 |
|--------|------|------|
| DropOldest（デフォルト採用） | 最新データを優先、バッファ有界 | 古いイベントの消失 |
| Block | イベント消失なし | タイムアウトまでブロック、スループット低下 |
| Reject | 即時フィードバック | 送信側でのリトライ実装が必要 |
| DropNewest | 既存バッファを保全 | 最新イベントの消失 |

### 採用理由

- オプトイン設計で後方互換性を維持（既存 API の変更なし）
- 全コンポーネントが `k1s0_` プレフィクスの Prometheus メトリクスを公開し、可観測性を確保
- 多層防御: フロントエンドスロットル → レート制限 → サーキットブレーカー → プールバックプレッシャー

## 帰結

### 正の帰結

- OOM リスクの排除とサイレントなデータ消失の防止
- 全レイヤーで統一的な負荷制御と可視化
- YAML 設定駆動でチームが環境に応じたチューニングが可能

### 負の帰結

- 各フレームワークパッケージへの変更が必要（影響範囲が広い）
- バックプレッシャー戦略の選択にはワークロード特性の理解が必要

### リスクと軽減策

| リスク | 軽減策 |
|--------|--------|
| デフォルト DropOldest による重要イベントの消失 | 重要イベントにはリカバリバッファを提供。用途に応じた戦略選択ガイドを文書化 |
| バックプレッシャー設定のミスチューニング | Grafana ダッシュボードでキュー深度・拒否率をリアルタイム監視 |
