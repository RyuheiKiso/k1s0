# 04. 教育とトレーニング

本書は k1s0 の導入・普及を支える教育プログラムを定義する。tier2/tier3 開発者、運用チーム、管理者、経営層のそれぞれに対してカリキュラムと到達目標を設ける。

## 本書の位置付け

プラットフォーム導入の失敗原因の上位は「教育不足で組織が使いこなせない」である。技術的に完成していても、使う人が育っていなければ稼働率は上がらない。本書は組織の k1s0 習熟度を段階的に引き上げる体系を定める。

## BC-TRN-001: tier2/tier3 開発者向け基礎

**現状**: 開発者教育が「ドキュメント読んで」で終わると、自走までの時間が数か月に及ぶ。

**要件達成後**: 以下のカリキュラムを提供する。

- **Level 1: Getting Started（4 時間）**: k1s0 概要、tier 階層、Backstage 操作、雛形生成、Hello World デプロイ
- **Level 2: Dapr Building Block（8 時間）**: tier1 11 API の使い方、Service Invoke / State / PubSub / Workflow を実装演習
- **Level 3: 運用と観測（4 時間）**: OpenTelemetry 計装、Grafana ダッシュボード、Runbook 参照、障害対応基礎
- **Level 4: 応用（8 時間、採用後の運用拡大時）**: カスタム CRD、Feature Flag 運用、JDM ルール設計、カオステスト

Level 1〜3 は リリース時点 時点で完備、半年に 1 回の頻度で開催。Level 4 は 採用後の運用拡大時 から。

**崩れた時**: 開発者が試行錯誤で時間を浪費、プラットフォームの効用を体感できない。

**受け入れ基準**:

- リリース時点 で Level 1〜3 コンテンツ完成
- Level 1 修了者 100 名（リリース時点）、300 名（採用後の運用拡大時 終了時）
- 修了後サーベイで満足度 4.0 以上（5 段階）

**優先度**: MUST

## BC-TRN-002: 運用チーム向け

**現状**: 運用チームが k1s0 を扱えないと、引継ぎ後に障害対応が機能しない。

**要件達成後**: 以下のカリキュラムを運用チーム向けに提供する。

- **Operator Onboarding（16 時間）**: Kubernetes 基礎、Argo CD、Istio、PostgreSQL、Kafka、Observability スタック、インシデント対応
- **Runbook Walkthrough（4 時間）**: 主要 Runbook を実演、トラブルシュート練習
- **オンコール訓練（4 時間）**: 月次机上演習、四半期実働訓練（OR-INC-006）
- **セキュリティ対応（4 時間）**: Falco アラート解釈、脆弱性対応、監査対応

運用チームメンバー全員が リリース時点 着手前に完了。

**崩れた時**: 運用引継ぎが形骸化、事故時に対応できない状態。

**受け入れ基準**:

- リリース時点 で教材完成
- 運用チーム 100% 修了（リリース時点 開始時）
- 四半期に訓練実施

**優先度**: MUST

## BC-TRN-003: 管理者向け

**現状**: 情シスマネージャ・CIO が k1s0 の価値と限界を理解できないと、意思決定が遅い。

**要件達成後**: 以下の管理者向けコンテンツを提供する。

- **Executive Briefing（2 時間）**: k1s0 の事業価値、コスト、リスク、他 採用側組織の事例
- **Product Council 参加ガイド**: 月次 Product Council での意思決定項目、データの読み方
- **四半期レビュー資料**: DORA Four Keys、可用性、コスト、テナント満足度の定型レポート

経営層への説明は定期的に実施、数値で価値を実証。

**崩れた時**: 管理者が自信を持って k1s0 を推進できず、横展開が遅延。

**受け入れ基準**:

- リリース時点 で Executive Briefing 完成
- 四半期ごとに経営層向けレビュー実施
- 満足度調査で「k1s0 を理解している」が 80% 以上

**優先度**: SHOULD

## BC-TRN-004: 教材形式

**現状**: 集合研修のみだと、時間拘束でスケールしない。

**要件達成後**: 以下の形式で教材を提供する。

- **録画コンテンツ**: 各カリキュラムの録画を社内 LMS に配置、24/7 視聴可
- **ハンズオン手順書**: 手元で再現可能な Step by Step ガイド
- **Live セッション**: 月 1 回、リアルタイムで質疑応答可能な開催
- **認定制度**: 各 Level 修了時にバッジ付与（採用後の運用拡大時）
- **社内コミュニティ**: Slack `#k1s0-learners` で Q&A、先輩開発者のサポート

**崩れた時**: 集合研修の枠が限られ、受講できないエンジニアが多数発生。

**受け入れ基準**:

- リリース時点 で録画コンテンツ完成
- リリース時点 で LMS に配置
- 月次受講者数を可視化

**優先度**: SHOULD

## BC-TRN-005: 継続学習

**現状**: 初期教育で完了とすると、新機能・ベストプラクティスが浸透しない。

**要件達成後**: 以下で継続学習を仕組み化する。

- **Tech Talk（月次）**: 開発者による技術共有、録画公開
- **Post-mortem 共有**: インシデントから学んだ内容を社内展開（匿名化の上）
- **External Conference Report**: KubeCon、GopherCon 等の参加者が報告会実施
- **書籍購入制度**: k1s0 技術領域の書籍購入を会社負担
- **OSS Day**: 四半期に半日、OSS コントリビューション促進

**崩れた時**: 学習が一過性で終わり、組織の技術水準が停滞。

**受け入れ基準**:

- リリース時点 で Tech Talk 定例開始
- Post-mortem 共有を全 Severity 1/2 で実施
- 書籍購入制度の利用者 50 名以上（年間）

**優先度**: SHOULD

## BC-TRN-006: 外部教育との連携

**現状**: 社内教育だけでは一般的な Kubernetes / Dapr / Istio の体系知識が不足。

**要件達成後**: 以下の外部教育を活用する。

- **CNCF 認定（CKA、CKAD、CKS）**: 受験費用を会社負担、取得者に報奨金
- **Dapr / Istio Training**: 公式トレーニング受講支援
- **Udemy for Business / Pluralsight**: 法人契約で全エンジニアがアクセス
- **ベンダー研修**: 必要に応じて Red Hat、SUSE、Microsoft 等のトレーニング

**崩れた時**: 社内教育に閉じた知識で、業界標準と乖離した誤った習慣が広まる。

**受け入れ基準**:

- リリース時点 で CNCF 認定支援制度開始
- 認定取得者を四半期ごとに可視化
- エンジニア採用時の要件にも反映

**優先度**: SHOULD
