# 02. 国際化 i18n/l10n

本書は k1s0 の国際化（Internationalization, i18n）と地域化（Localization, l10n）要件を定義する。採用側組織の初期展開は日本国内のみだが、採用側のマルチクラスタ移行時にアジア圏の海外拠点展開を視野に入れ、i18n を設計段階から織り込む。

## 本書の位置付け

i18n を後付けで実装すると、文字列リテラルがコードに散在し、全コードベースを書き換える工数が発生する。リリース時点 から i18n 対応を前提とした設計とすることで、将来の海外展開コストを 1/10 に抑える。

## BC-I18N-001: 対応言語

**現状**: 採用側組織の国内業務は日本語のみ、海外拠点は英語または現地語が必要。

**要件達成後**: 以下の言語対応を段階的に進める。

- **リリース時点〜2**: 日本語のみ
- **採用側のマルチクラスタ移行時**: 日本語 + 英語（海外拠点展開時）
- **採用後の長期運用**: 日本語 + 英語 + 中国語簡体字 + 中国語繁体字 + タイ語（アジア拠点需要に応じて）

言語切替は利用者アカウントの設定、ブラウザの Accept-Language ヘッダ、URL パラメタの優先順で決定。

**崩れた時**: 海外展開時に 6 か月の i18n 対応プロジェクトが発生、展開スケジュールを圧迫。

**受け入れ基準**:

- リリース時点 で文字列外部化を徹底（コードにリテラル置かない）
- 採用側のマルチクラスタ移行時 で英語対応完成
- 翻訳品質の第三者レビュー

**優先度**: SHOULD

## BC-I18N-002: 文字列外部化

**現状**: コードに日本語リテラルを直接書くと、後から翻訳する際に漏れや文脈不明のトラブル。

**要件達成後**: 全利用者向け文字列をリソースファイル（JSON / YAML / gettext .po）に外部化する。

- **Go**: `go-i18n` ライブラリ、YAML でメッセージ管理
- **Rust**: `fluent-rs`（Mozilla Fluent）、拡張性と複数形対応
- **TypeScript**: `react-i18next`（React）
- **キー設計**: `<domain>.<context>.<message>` 例: `error.auth.invalid_token`
- **補完情報**: 翻訳者向けのコンテキスト注釈を必須

ログメッセージ・内部エラーは英語で固定（運用者と開発者のため）、利用者向けメッセージのみ多言語化。

**崩れた時**: 文字列が散在し、翻訳漏れで UI に原文日本語が混入。

**受け入れ基準**:

- リリース時点 で外部化ルール適用
- CI で未外部化文字列を検出（Lint ルール）
- 翻訳ファイルの一貫性を自動検証

**優先度**: MUST

## BC-I18N-003: タイムゾーンと日付時刻

**現状**: サーバは UTC、ユーザ表示は現地時刻の分離ができていないと、夏時間・時差で混乱。

**要件達成後**: 以下の原則で時刻を扱う。

- **内部保管**: 全時刻は UTC で保存、ISO 8601 形式（例: `2026-04-19T12:34:56Z`）
- **表示**: 利用者のタイムゾーンに変換して表示、タイムゾーンを併記（例: `2026-04-19 21:34 JST`）
- **タイムゾーン取得**: ブラウザの IANA タイムゾーン（例: `Asia/Tokyo`）をユーザプロフィールに保存
- **日付フォーマット**: ロケールに応じて `YYYY-MM-DD`（ISO）、`YYYY年MM月DD日`（日本）等を切り替え
- **相対時刻**: 「3 分前」「昨日」などは利用者言語で表示

PostgreSQL は `timestamptz` 型を使用、アプリは常に UTC で操作。

**崩れた時**: 夏時間切り替わり時の夜間バッチ処理で時刻ずれ事故。

**受け入れ基準**:

- リリース時点 でタイムゾーン取扱規約を確定
- API 契約書（Protobuf）で時刻フィールドは全て UTC 規定
- 利用者プロフィールにタイムゾーン設定を提供

**優先度**: MUST

## BC-I18N-004: 数値・通貨・単位フォーマット

**現状**: 1,000 と 1.000 の表記が混在すると、桁区切りと小数点の誤読が発生する。

**要件達成後**: 以下のフォーマットを言語・ロケールで切り替える。

- **数値桁区切り**: 日本語は `,`、ドイツ語は `.`、スイスは `'`（ICU でロケール別制御）
- **小数点**: 日本語・英語は `.`、ドイツ語・フランス語は `,`
- **通貨**: ISO 4217 コードで内部管理、表示時にロケール別記号（¥、$、€）と位置（前／後）
- **単位**: 重量・長さ・面積は SI 単位を基本、必要に応じてヤードポンド
- **複数形**: 英語 is/are、日本語は単複同形（CLDR 複数形ルールに従う）

ICU4X / ICU4J など確立されたライブラリを活用。

**崩れた時**: 金額表示ミスで業務影響、契約トラブルの原因。

**受け入れ基準**:

- リリース時点 でフォーマットライブラリ採用と統一
- 通貨計算は Decimal 型で実施（浮動小数点禁止）
- テストで各ロケールのフォーマットを検証

**優先度**: MUST

## BC-I18N-005: RTL（右から左）対応

**現状**: アラビア語・ヘブライ語などの RTL 言語への展開は現時点想定外。

**要件達成後**: RTL 言語対応は 採用側の全社ロールアウト 以降の検討事項。現段階では logical property（`margin-inline-start` 等）を CSS で採用し、RTL 対応時の改修を最小化する。

**崩れた時**: RTL 展開決定時に UI 全面改修が必要。

**受け入れ基準**:

- CSS で logical property を リリース時点 から採用
- RTL 対応は 採用側の全社ロールアウト で再評価

**優先度**: COULD（採用側の全社ロールアウト）

## BC-I18N-006: 翻訳ワークフロー

**現状**: 翻訳が属人化すると、品質のばらつきと更新漏れが発生する。

**要件達成後**: 以下のワークフローで翻訳を管理する。

- **翻訳プラットフォーム**: Crowdin、Weblate、Lokalise など OSS または商用を採用
- **原文（source）**: 英語を master とする想定（採用側組織で日本語優先の場合は日本語を master）
- **翻訳プロセス**: 開発者が原文を追加 → 翻訳者が各言語対応 → ネイティブレビュー → マージ
- **機械翻訳補助**: DeepL API / Google Cloud Translation を一次翻訳に活用、人間が最終レビュー
- **用語集**: プラットフォーム用語・製品名を用語集で統制（「テナント」「ワークフロー」等）

**崩れた時**: 翻訳者チームの手配と翻訳品質のトラブルで、展開スケジュールが遅延。

**受け入れ基準**:

- 採用側のマルチクラスタ移行時 で翻訳プラットフォーム選定と導入
- 用語集を リリース時点 で初版作成
- 翻訳カバレッジを CI で可視化

**優先度**: SHOULD（採用側のマルチクラスタ移行時+）
