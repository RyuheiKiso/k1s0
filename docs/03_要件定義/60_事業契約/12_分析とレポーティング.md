# 12. 分析とレポーティング

本書は k1s0 プラットフォームの利用状況・成果・品質に関する分析とレポーティング要件を定義する。誰が、いつ、何を、どの粒度で報告するかを明確にし、経営層・運用部門・テナントの各ステークホルダーが必要な情報を継続的に入手できる体制を作る。

## 本書の位置付け

分析・レポーティングが不在だと、プラットフォームの成否を語る言葉が感覚論に陥る。定量データで価値を示し、改善サイクルを回すために、誰にどの報告を出すかを事前に設計する。

## BC-ANA-001: ステークホルダー別レポート

**現状**: 全員に同じ報告書を配ると、読まれずに形骸化する。

**要件達成後**: 以下の粒度でレポートを出し分ける。

- **経営層向け（四半期、CIO / 情シスマネージャ）**: KPI サマリ（可用性、TCO、DORA Four Keys、テナント数、満足度）、主要インシデント、次期計画
- **運用チーム向け（月次）**: 運用指標詳細（SLO 達成率、インシデント件数、MTTR、Runbook 実行数）、改善アクション
- **テナント向け（月次、部門単位）**: 部門別リソース使用量、SLA 達成状況、発生事故、コスト
- **開発者向け（月次）**: DORA Four Keys、CI/CD 時間、テストカバレッジ、PR サイクルタイム
- **法務・監査向け（半期）**: 監査ログサマリ、コンプライアンス遵守状況、ライセンス状況

各レポートは Backstage 経由で配信、自動生成を原則とする。

**崩れた時**: 読まれないレポートを作る工数の無駄、または必要な情報が届かない。

**受け入れ基準**:
- Phase 1c で自動レポート基盤構築
- 四半期・月次・半期の定例配信開始
- レポート受領者の満足度調査で 4.0 以上（5 段階）

**優先度**: MUST

## BC-ANA-002: KPI とビジネスインディケータ

**現状**: 測定する KPI が曖昧だと、取得データが活用されない。

**要件達成後**: 以下の KPI を定義する。

- **プラットフォーム KPI**
  - 可用性（月次・テナント別）
  - MTTA / MTTR / MTBF
  - SLO 達成率
  - オンボーディング所要時間
  - テナント数・稼働アプリ数
- **事業 KPI**
  - ROI（BC-COST-006 と整合）
  - TCO 実績 vs 計画
  - ライセンス削減額
- **開発者体験 KPI**
  - DORA Four Keys（DX-MET-001）
  - SPACE Framework（DX-MET-002）
  - NPS（DX-MET-003）
- **セキュリティ KPI**
  - 脆弱性対応 SLA 達成率
  - セキュリティインシデント件数
  - 監査指摘事項件数
- **コンプライアンス KPI**
  - 個人情報インシデント件数（目標 0）
  - 監査合格率

四半期ごとに KPI レビュー、改善アクション設定。

**崩れた時**: 何を改善すべきかの議論が空転、投資配分が適切に行えない。

**受け入れ基準**:
- Phase 1b で KPI ダッシュボード基盤構築
- Phase 1c で全 KPI 可視化
- 四半期 KPI レビューを Product Council で実施

**優先度**: MUST

## BC-ANA-003: 利用状況分析

**現状**: テナントの実利用状況が不可視だと、使われていない機能を作り続ける。

**要件達成後**: 以下を分析する。

- **機能別利用率**: tier1 API 別の呼び出し数、ユニークテナント数
- **時間帯分析**: ピーク時間、業務時間との相関
- **不使用機能**: 3 か月以上利用のない機能を特定、非推奨化検討
- **成長曲線**: テナント数・リクエスト数・データ量の月次推移
- **離脱分析**: テナント離脱時の理由カテゴリ化（EOL、移行、満足度低下）

分析結果は Product Council で月次レビュー、Phase 2 以降は機能投資優先度に反映。

**崩れた時**: 使われない機能を増築、本当に必要な機能が未実装。

**受け入れ基準**:
- Phase 1c で利用状況ダッシュボード
- 月次分析レポート公開
- 不使用機能の非推奨判断を定例化

**優先度**: SHOULD

## BC-ANA-004: 障害トレンド分析

**現状**: 個別障害対応で終わると、組織として学習しない。

**要件達成後**: 以下を分析する。

- **障害件数トレンド**: 月次 / 四半期で Severity 別件数
- **根本原因分類**: ソフトウェアバグ、人的ミス、ハードウェア障害、外部要因、仕様不備
- **影響コンポーネント**: どのコンポーネントでの障害が多いか
- **MTTR トレンド**: 平均復旧時間の短縮状況
- **再発率**: 同種事故の再発件数、ポストモーテム効果測定

半期ごとにトレンドレポート公開、Product Council で改善投資判断。

**崩れた時**: 同じ原因で繰り返し事故、学習サイクルが回らない。

**受け入れ基準**:
- Phase 1c でインシデントデータ集計基盤
- 半期トレンドレポート公開
- 再発率 10% 以下を目標

**優先度**: MUST

## BC-ANA-005: 外部ベンチマーク比較

**現状**: 内部数値だけだと、業界水準から乖離しているか判断できない。

**受け入れ基準**:
- 年次 DORA State of DevOps レポートとの比較
- CNCF Technology Radar 参照で採用技術動向確認
- KubeCon / SRECon 等の業界イベント参加レポート

**要件達成後**: 以下の外部比較を年次実施する。

- **DORA State of DevOps**: Google が毎年発表する業界ベンチマーク、Four Keys の位置づけ確認
- **CNCF State of Cloud Native**: クラウドネイティブ採用状況の比較
- **Gartner Magic Quadrant**: Kubernetes ディストリビューション、API Gateway 等の市場動向
- **業界事例**: 類似規模 JTC のプラットフォーム事例（匿名化ケーススタディ、カンファレンス発表）

**崩れた時**: 自社評価が過大 or 過小評価になり、投資判断を誤る。

**優先度**: SHOULD

## BC-ANA-006: 監査レポート

**現状**: 監査対応の証跡が散在すると、監査対応で数週間を要する。

**要件達成後**: 以下の証跡を常時保全し、監査時に即座に提出可能とする。

- **アクセスログ**: 全 Severity 1 操作、prod 変更、監査 API アクセス
- **変更ログ**: 全マニフェスト変更、ADR、Product Council 議事録
- **インシデントレポート**: Severity 1/2 ポストモーテム、対応タイムライン
- **脆弱性対応ログ**: CVE 通知から対応完了までの記録
- **教育記録**: トレーニング修了証、参加者リスト
- **RACI 記録**: 役割責任の履歴
- **監査法人向けパッケージ**: 上記を四半期ごとに集約した監査パッケージ

**崩れた時**: 監査で証跡不足の指摘、再監査で時間とコスト浪費。

**受け入れ基準**:
- Phase 1c で監査証跡集約基盤
- 四半期監査パッケージ自動生成
- 年次外部監査で指摘なし

**優先度**: MUST
