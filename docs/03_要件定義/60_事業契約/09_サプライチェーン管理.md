# 09. サプライチェーン管理

本書は k1s0 のソフトウェアサプライチェーンを保全するための要件を定義する。依存 OSS のセキュリティ、ベンダー評価、調達経路の検証、SLSA / in-toto / Sigstore などの基準適用を対象とする。

## 本書の位置付け

SolarWinds、Log4j、PyPI・npm の悪意ある依存など、近年のサプライチェーン攻撃は「自社のコードは安全でも依存先から侵入される」リスクを浮き彫りにした。k1s0 は OSS 積み上げ型のため、依存先への攻撃が最大のリスク面となる。本書は「何を、どこから、どう検証して」取り込むかを統制する。

## BC-SC-001: SLSA 準拠

**現状**: ビルドプロセスが改ざん検出なしでは、悪意あるコードが紛れ込んでも気付けない。

**要件達成後**: SLSA（Supply-chain Levels for Software Artifacts）Level 3 を目標とする。

- **Level 1**: ビルドが自動化されていること（GitHub Actions / Argo Workflows）
- **Level 2**: ビルドサービスが認証済みで、provenance を発行
- **Level 3**: ビルド環境が独立・隔離され、再現可能ビルド
- **Level 4 (将来目標)**: 複数名以上レビュー、プラットフォーム完全性確認

provenance は OCI Artifact として各イメージに添付。

**崩れた時**: ビルドサーバ侵害でマルウェアが混入、検出できない。

**受け入れ基準**:
- リリース時点 で SLSA Level 1/2 達成
- 採用後の運用拡大時 で Level 3 目標
- 年次で SLSA 監査実施

**優先度**: MUST

## BC-SC-002: in-toto によるビルド検証

**現状**: ビルドステップ間でアーティファクト改ざんがあっても検出できない。

**要件達成後**: in-toto でビルドステップの整合性を検証する。

- **Layout 定義**: ビルドステップを in-toto Layout として定義（source → build → test → sign → publish）
- **各ステップの成果物署名**: 開発者・CI・テストランナー・署名者がそれぞれ成果物に署名
- **最終検証**: デプロイ時に全署名の整合性を確認
- **失敗時**: 不整合検出で自動デプロイブロック

実装は witness（TestifySec）や SLSA GitHub Generator を活用。

**崩れた時**: ビルド途中での改ざんが本番に到達するリスク。

**受け入れ基準**:
- 採用後の運用拡大時 で in-toto 導入
- 主要ビルドパイプラインで適用
- 検証失敗時の自動ブロック

**優先度**: SHOULD（採用後の運用拡大時）

## BC-SC-003: Sigstore（cosign）署名

**現状**: コンテナイメージに署名がないと、レジストリ侵害時に差し替えを検出できない。

**要件達成後**: Sigstore を使ってコンテナイメージ、SBOM、provenance を全て署名する。

- **cosign でイメージ署名**: CI のビルド最終ステップで自動署名
- **Keyless 署名**: OIDC（GitHub Actions / Argo 組み込み）を利用、秘密鍵の管理不要
- **Rekor 透明性ログ**: 署名を公開透明性ログに記録
- **検証**: Argo CD デプロイ時に cosign verify を実行、未署名イメージは拒否
- **Kyverno ポリシー**: イメージ署名検証ポリシーを Kubernetes 全クラスタに適用

**崩れた時**: レジストリ侵害で悪意あるイメージが配置され、署名検証がないため検出できない。

**受け入れ基準**:
- リリース時点 で全 k1s0 イメージの署名
- リリース時点 で Kyverno による検証強制
- Rekor 記録の 100% 達成

**優先度**: MUST

## BC-SC-004: 依存 OSS の調達経路

**現状**: 公開レジストリから直接取得すると、ネットワーク問題や改ざんレジストリのリスク。

**要件達成後**: 社内プロキシレジストリを経由する調達経路を整備する。

- **コンテナイメージ**: 社内 Harbor / Quay レジストリがプロキシ、公開イメージをキャッシュ
- **Go module**: Athens プロキシまたは社内 GOPROXY
- **Rust crate**: cargo registry mirror
- **npm package**: Verdaccio または Nexus Repository
- **Helm chart**: ChartMuseum または OCI レジストリ
- **スキャン**: プロキシレジストリに入る前に Trivy / Grype でスキャン、重大脆弱性を含むものは取り込み拒否

**崩れた時**: タイポスクワッティング攻撃、公開レジストリ侵害で悪意あるパッケージ導入。

**受け入れ基準**:
- リリース時点 で全調達経路を社内プロキシ化
- スキャンを調達時点で実施
- 脆弱性含むパッケージの検出率 100%

**優先度**: MUST

## BC-SC-005: ベンダー評価

**現状**: OSS コミュニティ以外のベンダー（商用製品提供企業）の信頼性が未評価だと、リスク未把握。

**要件達成後**: 以下のベンダー評価を実施する。

- **セキュリティ体制**: SOC 2 Type II、ISO 27001、SBOM 提供、脆弱性情報提供
- **経営安定性**: 企業財務、事業継続性、買収リスク
- **サポート体制**: サポート品質、障害対応実績、言語対応（日本語）
- **データ取扱**: データセンター所在、データ暗号化、越境規制対応
- **契約条件**: SLA、賠償責任、契約終了条件
- **代替性**: ベンダーロックインリスク、代替候補の存在

評価は リリース時点 の OSS 選定時と、商用ベンダー採用時に実施。2 年ごとに再評価。

**崩れた時**: ベンダー倒産・買収で製品が使えなくなる、またはセキュリティ体制が脆弱で事故発生。

**受け入れ基準**:
- リリース時点 でベンダー評価チェックリスト作成
- 採用前の評価を必須化
- 評価結果を ADR または Backstage で公開

**優先度**: MUST

## BC-SC-006: 脆弱性対応ポリシー

**現状**: 脆弱性対応の優先度が曖昧だと、重大脆弱性への対応が遅延する。

**要件達成後**: CVSS スコアに応じた対応ポリシーを以下に定める。

- **Critical（CVSS 9.0〜10.0）**: 24 時間以内に影響評価、48 時間以内にパッチ適用または Workaround
- **High（CVSS 7.0〜8.9）**: 1 週間以内にパッチ適用
- **Medium（CVSS 4.0〜6.9）**: 1 か月以内にパッチ適用
- **Low（CVSS 0.1〜3.9）**: 次回定期メンテナンスで適用

Exploitable な脆弱性（EPSS スコア参照）は優先度を上げて対応。

**崩れた時**: Critical 脆弱性を放置し、実際の攻撃被害に遭う。

**受け入れ基準**:
- リリース時点 で脆弱性対応ポリシー確定
- Trivy / Grype / Snyk で継続的スキャン
- 対応 SLA 達成率 90% 以上

**優先度**: MUST

## BC-SC-007: インシデント情報共有

**現状**: 業界でのサプライチェーン事故情報を知らないと、同種事故が自社でも起きる。

**要件達成後**: 以下の情報源をモニタリングする。

- **GitHub Security Advisories**: 採用 OSS の CVE アラート
- **CISA KEV カタログ**: 実際に攻撃されている脆弱性（Known Exploited Vulnerabilities）
- **JPCERT / IPA**: 国内の脆弱性情報
- **Cloud Native Security Reports**: CNCF TAG-Security、SLSA などの業界動向
- **ベンダー公式セキュリティ情報**: Red Hat、SUSE 等の情報配信購読

モニタリング結果は週次で情報共有ミーティングを実施、影響評価を即時。

**崩れた時**: 公開済み脆弱性の情報を見落とし、対応遅延。

**受け入れ基準**:
- リリース時点 で情報源の購読開始
- 週次セキュリティレビュー定例化
- 新脆弱性の初動応答 24 時間以内

**優先度**: MUST
