# 06. 課金メータリング

本書は k1s0 内部におけるテナント別課金（Chargeback）と使用量計測（Metering）の要件を定義する。外部顧客向け課金ではなく、JTC 内部の部門間コスト按分（社内振替）を対象とする。

## 本書の位置付け

プラットフォーム費用を情シスが全額吸収すると、利用側の部門はリソース消費に無頓着になりやすい。一方、完全従量課金にすると小さな部門がプラットフォームを敬遠する。本書は中間の「可視化は全て、請求は閾値超過のみ」の Chargeback モデルで、コスト意識と利用促進のバランスを取る。

## BC-BIL-001: メータリング基盤

**現状**: 使用量計測がないと、Chargeback もコスト改善指導もできない。

**要件達成後**: 以下の指標をテナント別に計測する。

- **CPU 使用量**: vCPU・時間（Pod CPU Usage をノード単位で積分）
- **メモリ使用量**: GB・時間（Pod Memory Usage を積分）
- **ストレージ使用量**: GB・月（PVC / MinIO バケット容量のスナップショット）
- **ネットワーク使用量**: GB（Ingress / Egress、Cilium / Envoy メトリクス）
- **API 呼び出し数**: tier1 API 別のリクエスト数（Envoy アクセスログから集計）
- **イベント数**: Kafka トピック別メッセージ数
- **ストレージ操作数**: PostgreSQL クエリ数（将来的に）

計測は OpenCost + Prometheus + 自作 exporter で実装。計測データは 3 年保持。

**崩れた時**: 使用量不明で、コスト配分の根拠が作れない。

**受け入れ基準**:
- Phase 1c で基本指標（CPU/Memory/Storage/Network）計測
- Phase 2 で API 呼び出し数計測
- 計測精度 95% 以上を実測検証

**優先度**: MUST

## BC-BIL-002: Chargeback モデル

**現状**: 課金ルールがないと、按分の正当性をテナントが受け入れない。

**要件達成後**: 以下の Chargeback モデルを採用する。

- **基本料**: テナント固定費として月額 × 種別（Express / Standard / Enterprise）で算定
- **従量料**: 使用量閾値超過分を按分
  - CPU 超過: 追加 vCPU・時間 × 単価
  - ストレージ超過: 追加 GB・月 × 単価
  - ログ量超過: 追加 GB × 単価
- **共通基盤費**: プラットフォームチーム人件費、共通インフラ費用をテナント数または使用量で按分
- **電力費**: 電力単価 × CPU/メモリ使用量で按分（Phase 5+、ESG 対応時）

単価は年次で見直し、Product Council で承認。

**崩れた時**: 按分ルールが恣意的と感じられ、テナント側の不信感。

**受け入れ基準**:
- Phase 2 で Chargeback モデル確定
- 単価見直しプロセスを年次化
- 不服申し立て窓口を設置

**優先度**: SHOULD（Phase 2+）

## BC-BIL-003: 可視化ダッシュボード

**現状**: 使用量がテナント側に見えないと、改善インセンティブが働かない。

**要件達成後**: Backstage に以下のダッシュボードを提供する。

- **テナントダッシュボード**: 自テナントの CPU/Memory/Storage/Network 使用量、ランキング、トレンド
- **比較ダッシュボード**: 同種別テナント中での位置づけ（匿名化）
- **コストレポート**: 月次で Chargeback 金額、内訳、前月比
- **予測ダッシュボード**: 次月の想定請求額、予算超過アラート
- **最適化推奨**: Right-Sizing や未使用リソース削除の推奨アクション

プライバシー配慮として、他テナントの絶対値は非表示、順位と相対値のみ表示。

**崩れた時**: コスト削減行動が起きず、全テナントが無制限にリソース消費。

**受け入れ基準**:
- Phase 2 で可視化ダッシュボード提供
- 月次で閲覧率 70% 以上
- 改善アクション実行率を計測

**優先度**: SHOULD

## BC-BIL-004: 予算と上限設定

**現状**: テナント予算超過を検知できないと、月末に予期せぬ請求が発生。

**受け入れ基準**:
- Phase 2 で予算アラート機能提供
- 閾値 80% で警告、100% で停止（ソフトリミット）
- ハードリミット超過時の停止手順を Runbook 化

**要件達成後**: 以下の予算管理を提供する。

- **テナント月次予算設定**: Chargeback ベースの月次上限を設定可能
- **段階的アラート**: 予算 80% / 90% / 100% で Slack 通知
- **ソフトリミット**: 予算超過時も即座に停止せず、警告のみ
- **ハードリミット（オプション）**: 予算超過で自動的にリソース制限（開発環境推奨）
- **繰越制度**: 未使用予算の翌月繰越（Product Council 承認）

**崩れた時**: 請求が予算を大幅に超過、テナント側の信頼低下。

**優先度**: SHOULD（Phase 2+）

## BC-BIL-005: 請求書発行

**現状**: 内部振替の請求プロセスが属人化すると、経理部門の手作業が発生。

**要件達成後**: 以下で請求書発行を自動化する。

- **月次集計**: 月末締め、翌月 5 営業日以内に確定
- **請求書フォーマット**: 社内振替フォーマット（部門コード、費目、金額、内訳）
- **経理システム連携**: CSV / API で経理システムに自動連携
- **差異調整**: 計測エラーや障害による調整を翌月請求に反映
- **監査ログ**: 請求金額の変更履歴を保全

**崩れた時**: 経理部門が手作業で集計、毎月の締めで数日浪費。

**受け入れ基準**:
- Phase 2 で請求書自動生成
- 経理部門の確認工数 4 時間以内 / 月
- 差異調整プロセスを Runbook 化

**優先度**: SHOULD（Phase 2+）

## BC-BIL-006: 無料利用枠と補助

**現状**: 小規模部門が Chargeback を恐れて k1s0 採用を避けると、プラットフォーム普及が停滞。

**要件達成後**: 以下の無料利用枠を設ける。

- **Express 種別の新規テナント**: 初回 3 か月は Chargeback 免除
- **小規模テナント**: 月間リソース消費が基準以下（例: 1 vCPU・10GB メモリ・50GB ストレージ）は Chargeback 免除、共通費のみ
- **PoC・実験枠**: Product Council 承認で最大 6 か月の無料利用
- **教育利用**: トレーニング・ハンズオン目的の利用は無料

**崩れた時**: 小部門の採用が阻まれ、プラットフォームの普及速度が低下。

**受け入れ基準**:
- Phase 2 で無料枠制度確定
- 無料枠利用テナント数を月次追跡
- 有償移行の卒業率を可視化

**優先度**: COULD（Phase 2+）
