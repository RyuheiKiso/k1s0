# 01. UX 方針

本書は k1s0 が提供する管理 UI（Backstage、Grafana、JDM エディタ、Admin Console）の UX 原則を定義する。tier1 は公開 API 主体であり UI 面積は小さいが、利用者接点となる管理 UI の体験が貧弱だと、機能の良さが伝わらず採用率が下がる。

## 本書の位置付け

JTC 既存業務システムの UX は 2000 年代水準で止まっており、tier2/tier3 開発者・運用者は「社内システムは使いにくいもの」と諦めている。k1s0 はこの諦めを覆す UX を提供することで、プラットフォームのプロダクト性を打ち出す。UX の差別化は OSS 積み上げでも達成可能であり、Backstage プラグインの組み合わせと統一デザインシステムで実現する。

## BC-UX-001: デザインシステム

**現状**: 各 UI（Backstage、Grafana、JDM エディタ）で色・タイポグラフィ・コンポーネントがバラつくと、横断利用時の認知負荷が高い。

**要件達成後**: k1s0 デザインシステムを定義し、全 UI で統一する。

- **カラーパレット**: プライマリ（信頼感のある青）、セカンダリ（アクション強調のオレンジ）、グレースケール 6 段階、セマンティック色（成功緑、警告黄、エラー赤、情報青）
- **タイポグラフィ**: 日本語は Noto Sans JP、英数は Inter、コードは JetBrains Mono
- **コンポーネント**: ボタン、フォーム、テーブル、カード、ダイアログ、ナビゲーションの 6 基本要素
- **アイコン**: Lucide Icons を採用、補足で独自アイコン追加可
- **ダークモード**: 全 UI でダークモード対応

Backstage は Material UI ベース、Grafana は既存スタイル、JDM エディタは独自 UI だが、色・タイポグラフィを統一する。

**崩れた時**: 利用者が UI ごとに学習し直す必要があり、学習コスト増。

**受け入れ基準**:
- デザインシステム仕様書を Phase 1b で策定
- 主要 3 UI でプライマリカラー・タイポグラフィ統一
- Storybook 等でコンポーネントを視覚化

**優先度**: SHOULD

## BC-UX-002: アクセシビリティ

**現状**: 社内アプリのアクセシビリティ不備で、色覚多様性・視覚障害・聴覚障害のある社員の利用を阻害する。

**要件達成後**: 管理 UI で WCAG 2.1 AA を目標とする。tier3 アプリ向けガイドラインも提供。

- **キーボード操作**: 全機能がキーボードのみで操作可能
- **スクリーンリーダー**: aria 属性、ラベル、landmark を適切に配置
- **コントラスト比**: テキスト 4.5:1 以上、大きなテキスト 3:1 以上
- **色だけに依存しない情報伝達**: アイコンやラベルを併用
- **フォーカス可視**: キーボードフォーカスが明確に表示される
- **動画・音声**: 字幕・トランスクリプト提供（該当があれば）

Phase 1c でアクセシビリティ監査、Phase 2 で WCAG 2.1 AA 相当を認証機関または内部監査で確認。

**崩れた時**: 多様性ある社員が業務に参加できず、人権リスクと法的リスク（日本では努力義務）。

**受け入れ基準**:
- WCAG 2.1 AA チェックリストを Phase 1c で完成
- axe-core または Lighthouse で自動検査を CI に組み込み
- Phase 2 で監査完了

**優先度**: SHOULD

## BC-UX-003: 情報アーキテクチャ

**現状**: メニュー階層や画面遷移が深いと、利用者が目的の画面に辿り着けない。

**要件達成後**: 以下の原則で情報アーキテクチャを設計する。

- **3 クリックルール**: 主要機能まで 3 クリック以内
- **グローバルナビゲーション**: 最上位に常時配置、現在地をブレッドクラム表示
- **検索優先**: Backstage はグローバル検索を前面配置、Cmd+K ショートカット
- **コンテキスト保持**: タブ切替でも状態保持、URL にクエリ反映

利用者テスト（Phase 1c 以降）で合計所要時間を計測、改善サイクルを回す。

**崩れた時**: 利用者が迷子になり、サポート問い合わせが増加。

**受け入れ基準**:
- 主要機能の 3 クリックルール達成
- グローバル検索を Phase 1b で提供
- 四半期ごとに利用者テスト

**優先度**: SHOULD

## BC-UX-004: パフォーマンス体験

**現状**: 画面表示が 3 秒以上かかると、利用者が「重い」と感じて離脱する。

**要件達成後**: 以下のパフォーマンス目標を設ける。

- **初回表示**: LCP（Largest Contentful Paint）< 2.5 秒
- **操作反応**: INP（Interaction to Next Paint）< 200ms
- **レイアウト安定**: CLS（Cumulative Layout Shift）< 0.1
- **API 呼び出し**: 画面初期ロードで 1 秒以内にデータ表示

Core Web Vitals を Grafana で可視化、悪化時はアラート。

**崩れた時**: 「Backstage は重い」という評判が広まり、利用率低下。

**受け入れ基準**:
- Real User Monitoring を Phase 1c で導入
- 主要画面で Core Web Vitals が Good 評価
- 月次レビューで改善サイクル

**優先度**: SHOULD

## BC-UX-005: フィードバック収集

**現状**: 利用者のフィードバックが不在だと、改善投資が的外れになる。

**要件達成後**: 以下でフィードバックを常時収集する。

- **フィードバックボタン**: 全画面右下に常駐、改善要望・バグ報告を 1 クリックで送信
- **ユーザテスト**: 四半期に 5 名の利用者を対象にテスト実施（Phase 2+）
- **アンケート**: 半期に 1 回の利用者満足度調査
- **ヒートマップ**: プライバシー配慮の上、主要画面の利用頻度を計測

改善要望は Backstage の Product Backlog に自動起票、Product Council でトリアージ。

**崩れた時**: 使われていない機能を増築、本当に必要な機能が未実装。

**受け入れ基準**:
- Phase 1c でフィードバックボタン配備
- Phase 2 で定期ユーザテスト
- フィードバック対応率 70% 以上（6 か月以内）

**優先度**: SHOULD
