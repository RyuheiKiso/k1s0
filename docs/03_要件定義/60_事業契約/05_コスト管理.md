# 05. コスト管理

本書は k1s0 プラットフォームの総保有コスト（TCO）算定、予算執行管理、コスト最適化プロセスを定義する。企画書の 5 年間 3.68 億円（中規模）試算を基点とし、実績管理と予実差異の対応を手続化する。

## 本書の位置付け

プラットフォームのコストが「なんとなく」で推移すると、稟議通過時の約束（年間 2,349 万円）が破綻した際の説明責任を果たせない。本書は四半期単位の予実管理と、テナント別の可視化で、経営層への継続的な説明可能性を担保する。

## BC-COST-001: コスト構造の把握

**現状**: プラットフォームコストの内訳が不可視だと、どこを削減すべきかの判断ができない。

**要件達成後**: 以下の分類でコストを可視化する。

- **インフラコスト**: VM 基盤利用料（社内按分）、ストレージ、ネットワーク帯域
- **ライセンスコスト**: 商用 OSS（Aqua、Datadog 等採用時）、監視ツール、バックアップツール
- **人件費**: プラットフォーム SRE チーム、新規開発工数、運用対応工数
- **教育費**: トレーニング、書籍、認定試験、外部研修
- **契約費**: ベンダー契約、法務レビュー費、外部監査費
- **電力・ファシリティ**: DC 電力（社内按分）、空調、物理設置工事（該当時）

年間コスト見通しを Phase 毎に更新、稟議通過時の試算と比較。

**崩れた時**: コスト超過が発覚しても、どの領域で超過しているか特定できない。

**受け入れ基準**:
- Phase 1a でコスト分類表完成
- 四半期ごとに実績値を集計
- 予実差異 10% 以内を維持目標

**優先度**: MUST

## BC-COST-002: FinOps 導入

**現状**: FinOps（財務運用）の概念がないと、リソース増減の判断が感覚論になる。

**要件達成後**: FinOps Foundation のフレームワークを適用する。

- **Inform（可視化）**: OpenCost や Kubecost で Kubernetes コストをテナント別に可視化
- **Optimize（最適化）**: 未使用リソースの削除、Right-Sizing、スポット利用検討（オンプレでは Overcommit）
- **Operate（運用）**: 月次コストレビューを Product Council で実施、削減アクション実行

Phase 2 以降で FinOps ロール（兼任可）を定め、責任を明確化。

**崩れた時**: リソース過剰配分で TCO 悪化、逆に不足で性能問題。

**受け入れ基準**:
- Phase 1c で OpenCost 導入
- 月次コストレポートを Product Council に提出
- 削減アクション完了率 70% 以上

**優先度**: MUST

## BC-COST-003: リソース Right-Sizing

**現状**: CPU / Memory の Request/Limit が過大だと、VM が無駄に占有され TCO 悪化。

**要件達成後**: 以下のプラクティスで適正化する。

- **初期見積**: サービス起動時は過大目に設定（安全側）、Phase 1b の実測後に縮小
- **VPA（Vertical Pod Autoscaler）**: Recommendation モードで推奨値を常時提示、開発者は四半期に見直し
- **HPA（Horizontal Pod Autoscaler）**: CPU / Memory / カスタムメトリクスで水平スケール
- **KEDA**: イベント駆動スケール（Kafka の lag、HTTP RPS）で無駄なリソースを削減
- **ノード集約**: Pod Density（ノードあたりの Pod 数）を定期監視、過疎ノードをドレイン

**崩れた時**: リソース無駄でインフラコストが膨張。

**受け入れ基準**:
- Phase 1c で VPA 導入、Recommend モードで推奨値公開
- Phase 2 で HPA 全 tier1 API 適用
- 四半期ごとのリソースレビュー

**優先度**: MUST

## BC-COST-004: テナント別コスト

**現状**: テナント別コストが不明だと、大量消費するテナントへの改善依頼や内部課金の根拠が作れない。

**要件達成後**: 以下でテナント別コストを算出する。

- **CPU / Memory / Storage**: 実使用量に応じて按分
- **ネットワーク**: Envoy / Cilium のメトリクスから Ingress / Egress 計測
- **ログ / メトリクス**: Loki / Prometheus の取り込みバイト数で按分
- **共通基盤コスト**: テナント数で等分または使用量比例（按分ルールを Phase 1c で確定）

可視化は Backstage のテナントページで利用者も閲覧可能。月次レポートとして送付。

**崩れた時**: 一部テナントの過剰利用が他テナントに影響、不公平感が広がる。

**受け入れ基準**:
- Phase 1c で按分ルール確定
- Phase 2 でテナント別月次レポート自動配信
- 上位消費テナントに改善相談を実施

**優先度**: SHOULD

## BC-COST-005: 予算執行プロセス

**現状**: 予算執行が年次で一括承認だと、柔軟性に欠ける。

**要件達成後**: 以下のプロセスで予算を管理する。

- **年次予算計画**: 前年 Q4 に次年度予算を策定、CIO 承認
- **四半期執行レビュー**: 各四半期末に予実差異レビュー、必要に応じて再配分
- **月次トラッキング**: コスト実績を月次で集計、Product Council で確認
- **緊急枠**: 年間予算の 10% を緊急対応枠として確保、承認プロセスを短縮
- **調達予約**: 大型購入（VM 追加、ストレージ拡張）は 3 か月前から予約

**崩れた時**: 予算超過が発覚してから対応が間に合わない、または不足で新機能開発が止まる。

**受け入れ基準**:
- Phase 1a で年次予算テンプレート確定
- 四半期レビューを Product Council で定例化
- 予算差異 ±15% 以内を維持

**優先度**: MUST

## BC-COST-006: ROI 実証

**現状**: プラットフォーム投資の回収が見えないと、継続予算承認が難しくなる。

**要件達成後**: 以下の ROI 指標を定期報告する。

- **ライセンス削減額**: 商用 PaaS 採用と比較した年間削減額
- **開発生産性向上**: 新規アプリの開発リードタイム短縮時間 × 開発者時給
- **運用効率化**: MTTR 短縮、運用工数削減の定量化
- **品質向上**: 障害件数減、個人情報漏えい件数ゼロ維持
- **ベンダーロックイン解除効果**: 撤退可能性確保の価値（商用 PaaS 交渉力）

四半期レビューで稟議時の約束（5 年間 7,600 万円削減）との比較を報告。

**崩れた時**: ROI 証明が弱く、Phase 3 以降の予算承認が難航。

**受け入れ基準**:
- Phase 1c で ROI 計測開始
- 年次経営報告で定量報告
- 稟議約束値の達成率を追跡

**優先度**: MUST
