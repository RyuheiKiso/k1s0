# 07. OSS ライセンス義務

本書は k1s0 が採用する 100 以上の OSS について、各ライセンスの義務遵守のプロセスを定義する。Apache License 2.0、MIT、BSD、LGPL、GPL、AGPL、MPL、Elastic License v2 など多様なライセンスに対応し、商用配布を行わない社内利用プラットフォームでも満たすべき義務を整理する。

## 本書の位置付け

OSS ライセンス違反は法的・評判の両面で致命的である。特に AGPL は「サービス提供者がソースを公開する義務」を持つため、クラウドサービスを売らない JTC 社内利用でも波紋が大きい。本書は採用 OSS を自動スキャンし、義務を機械的に履行する仕組みを要求する。

## BC-LIC-001: ライセンス分類

**現状**: OSS のライセンス区分が曖昧だと、義務の履行漏れが発生する。

**要件達成後**: 採用 OSS を以下のカテゴリに分類する。

- **Permissive（緩和型）**: Apache License 2.0、MIT、BSD-2-Clause、BSD-3-Clause、ISC。帰属表示が主な義務。
- **Weak Copyleft（弱いコピーレフト）**: LGPL、MPL 2.0。改変部分の公開義務あり、リンク利用は免除。
- **Strong Copyleft（強いコピーレフト）**: GPL v2 / v3。派生物全体のソース公開義務。
- **Network Copyleft（AGPL）**: AGPL v3。サービス提供時もソース公開義務。
- **Source-Available（ソース公開ライセンス）**: Elastic License v2、BUSL、SSPL。商用配布制限あり、社内利用は概ね可。
- **独自/Unknown**: 個別評価。

各 OSS のライセンス区分は SPDX 識別子で管理、SBOM に記録する。

**崩れた時**: ライセンス区分の誤認で、AGPL コードを不用意に組み込み公開義務違反。

**受け入れ基準**:
- Phase 1a で分類表完成
- SPDX ライセンス識別子を全 OSS に付与
- 月次で新規採用 OSS のライセンス確認

**優先度**: MUST

## BC-LIC-002: SBOM 生成と保全

**現状**: 依存 OSS が不可視だと、脆弱性対応もライセンス遵守も手付かず。

**要件達成後**: SBOM（Software Bill of Materials）を全アーティファクトで生成する。

- **フォーマット**: CycloneDX（CNCF 標準）または SPDX
- **生成タイミング**: CI/CD のビルド時に自動生成
- **含む情報**: コンポーネント名、バージョン、ライセンス、サプライヤ、依存関係、ハッシュ
- **保管**: OCI レジストリの Artifact として保存（OCI Artifact Spec）
- **署名**: Sigstore（cosign）で SBOM に署名
- **配布**: Backstage からダウンロード可能、API で外部連携可

**崩れた時**: 依存 OSS の脆弱性調査・ライセンス確認が手動となり、対応遅延。

**受け入れ基準**:
- Phase 1a で SBOM 生成自動化
- 全イメージに SBOM が添付されていることを CI で検証
- 外部監査での SBOM 提出が即座に可能

**優先度**: MUST

## BC-LIC-003: 帰属表示（Attribution）

**現状**: Permissive ライセンスでも著作権表示とライセンス文の配布義務がある。

**要件達成後**: 以下の形式で帰属表示を実施する。

- **Backstage に NOTICE ページ**: 採用 OSS の名前、著作権者、ライセンス文、ソース URL を一覧
- **配布物（CLI、Dev Container）に同梱**: `LICENSES/` ディレクトリにライセンス原文を格納
- **自動生成**: SBOM から帰属情報を自動生成、手動メンテナンス不要
- **更新**: 新規 OSS 採用時に自動追加

**崩れた時**: 帰属表示漏れで Apache License 2.0 等の義務違反。

**受け入れ基準**:
- Phase 1b で NOTICE ページ自動生成
- 新規 OSS 追加時の自動更新
- 監査で帰属表示漏れゼロ

**優先度**: MUST

## BC-LIC-004: AGPL 分離アーキテクチャ

**現状**: AGPL 採用 OSS（Grafana / Loki / Tempo / Pyroscope / MinIO / Renovate 等）を tier1 内部に直接統合すると、「リンク」と「ネットワーク経由の提供」の解釈次第で tier1 コード全体が AGPL-3.0 セクション 13 のソース開示義務対象になる懸念。境界が曖昧なまま実装に進むと、法務レビューが実装後に発生し、稟議通過済みアーキテクチャの差し戻しリスクが残る。

**要件達成後**: ADR-0003（AGPL 隔離アーキテクチャ）に従い、以下 4 レイヤごとの技術判定基準で「許容」と「逸脱」を明示する。逸脱が検出された場合は CI ブロックまたは本番デプロイ拒否（Kyverno ポリシー）で機械的に封じる。

- **L1. 同一 OS プロセス境界**: 許容は別プロセス（別バイナリ）のみ。同一プロセス内への AGPL 成果物の動的／静的リンクは禁止。Go plugin / C 共有ライブラリとしての取り込みも禁止（BC-LIC-005 の個別承認で例外化しない限り）。判定: Trivy + ScanCode で同一バイナリ内の AGPL 依存を CI スキャン、検出時 PR ブロック
- **L2. 同一 Pod 境界**: 許容は別 Pod のみ。同一 Pod 内のサイドカーコンテナとしての配置も「プロセス共有の外観」を避けるため禁止。AGPL OSS 用の Deployment / StatefulSet は別 manifest で管理。判定: Kyverno で AGPL ラベル（`license=agpl`）付き Pod との同居を禁止
- **L3. ネットワーク境界（社内限定）**: 許容は Istio NetworkPolicy で「クラスタ内 + VPN 越しの社員アクセス」のみ。外部インターネット・未認証テナントからの到達可能性は禁止。Grafana / Loki 等は Backstage SSO 経由の内部アクセスのみ。判定: Kyverno で `Service type=LoadBalancer` の AGPL ラベル Pod を禁止、Istio AuthorizationPolicy で外部 CIDR を明示拒否
- **L4. データ境界（DB 接続は許容、改変は禁止）**: PostgreSQL / Kafka / MinIO への AGPL OSS からの接続は「ネットワーク越しのクライアント利用」であり、無改変利用として義務を発生させない。ただし AGPL OSS 自体を改変した場合はセクション 13 の改変物開示義務が発動する。判定: AGPL OSS のイメージは公式タグのみ許容、自社フォークは禁止（Kyverno image allowlist）

tier1 自作コード（Rust / Go）は AGPL OSS の成果物を「利用する側」として位置付け、プロセス・Pod・ネットワーク・データの 4 レイヤで独立する。AGPL OSS → tier1 への呼び出しは行わず、tier1 → AGPL OSS への単方向呼び出し（HTTP / gRPC）に限定する。

法務サインオフ対象の具体判断は U-004（00_要件定義方針/03_前提と制約.md の未決事項）で管理し、2026-06-30 までに法務リード + プラットフォームリードの双方で承認する。

AGPL 採用 OSS の例: Grafana（AGPL v3）、MongoDB（SSPL、独自）、Redis 後継（RSAL、独自）。代替手段（Valkey 等）を優先的に採用する（ADR-DATA-004）。

**崩れた時**: AGPL 義務が拡大解釈され、自作コード公開義務発生の法務リスク。稟議で約束した「オンプレ OSS 積み上げで tier1 自作部分は非公開」の前提が崩れ、競合優位性が毀損する。

**受け入れ基準**:
- Phase 1a で AGPL 採用 OSS を棚卸し（Grafana / Loki / Tempo / Pyroscope / MinIO / Renovate を初期リスト）
- 分離アーキテクチャを ADR-0003 で承認、本節 L1〜L4 の判定基準と 1 対 1 対応
- 法務部との合意を U-004 経由で文書化
- L1 / L2 / L4 は CI + Kyverno で機械チェック、L3 は四半期毎に NetworkPolicy 監査

**優先度**: MUST

**関連**: NFR-E-NW-003（AGPL 外部境界遮断）、RISK-LGL-001、ADR-0003、00_要件定義方針/03_前提と制約.md U-004

## BC-LIC-005: 禁止ライセンスリスト

**現状**: 許容ライセンス基準が曖昧だと、個人の判断でライセンス違反リスクのある OSS が混入する。

**要件達成後**: 以下のライセンスを禁止または個別承認とする。

- **禁止**: GPL v2 / v3（派生物全体への公開義務がプラットフォーム性格に合わない）
- **個別承認**: AGPL、SSPL、BUSL、独自ライセンス（法務レビュー必須）
- **許容**: Apache 2.0、MIT、BSD（2/3 Clause）、ISC、MPL 2.0、LGPL
- **条件付き許容**: Elastic License v2（社内利用のみ）

新規 OSS 採用時は CI で自動チェック、禁止ライセンスを含む場合はマージ不可。

**崩れた時**: GPL 汚染で tier1 コード公開義務発生のリスク。

**受け入れ基準**:
- Phase 1a で禁止リスト確定
- CI で自動ライセンススキャン（ScanCode / FOSSology）
- 例外承認を ADR で文書化

**優先度**: MUST

## BC-LIC-006: 商用 OSS のライセンス管理

**現状**: 商用 OSS（Elastic、MongoDB 等）のライセンス更新漏れは突然のアクセス拒否を招く。

**要件達成後**: 以下で商用 OSS ライセンスを管理する。

- **ライセンス台帳**: 全商用 OSS のライセンス番号、有効期限、購入履歴を一元管理
- **更新アラート**: 期限 3 か月前に自動アラート、調達部門に通知
- **アクセスキー保管**: ライセンスキーを OpenBao に保管、アクセス監査
- **利用実績**: ライセンス範囲内の利用を四半期ごとに確認

商用 OSS は極力避け、OSS 代替があれば優先採用する方針。

**崩れた時**: ライセンス切れで稼働停止、緊急対応で計画外コスト発生。

**受け入れ基準**:
- Phase 1b でライセンス台帳整備
- 期限アラート自動化
- 更新漏れゼロを維持

**優先度**: MUST

## BC-LIC-007: OSS コントリビューション

**現状**: OSS コミュニティへの還元なしに利用し続けると、長期的に OSS エコシステムが衰退、自社リスク。

**要件達成後**: 以下で OSS コントリビューションを促進する。

- **コントリビューション方針**: 採用 OSS の改善・バグ修正を会社業務として実施可能とする
- **CLA（Contributor License Agreement）管理**: プロジェクト別の CLA を法務レビュー、締結サポート
- **Intellectual Property 規則**: 業務外のコントリビューションの取り扱いを就業規則で明確化
- **貢献予算**: OSS コントリビューション時間を月次で一定枠確保
- **表彰制度**: 年次で OSS 貢献表彰

**崩れた時**: OSS を使うだけの「フリーライダー」と業界で認識され、採用や提携で不利。

**受け入れ基準**:
- Phase 1c で方針策定
- Phase 2 で貢献予算導入
- 年次で貢献実績を公開

**優先度**: SHOULD
