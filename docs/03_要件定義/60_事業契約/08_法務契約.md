# 08. 法務契約

本書は k1s0 開発・運用にかかわる法務契約要件を定義する。ベンダー契約、秘密保持、業務委託、データ取扱、責任分界、個人情報保護法対応など、法務部レビューの対象となる項目を整理する。

## 本書の位置付け

技術的に優れた基盤でも、法務リスクが不明確だと採用検討通過は難しい。本書は 採用側組織の法務部門が k1s0 を承認できる状態を作ることを目的とし、関連する契約・法令・規程への対応を明文化する。

## BC-LGL-001: 適用法令・規程の確認

**現状**: 適用される法令・社内規程が洗い出されていないと、違反リスクが残存する。

**要件達成後**: 以下を事前棚卸しし、遵守計画を策定する。

- **個人情報保護法（2022 改正）**: 個人データ取扱、漏えい時の 3 日／30 日通知、越境移転
- **マイナンバー法**: 特定個人情報の取扱（該当あれば）
- **サイバーセキュリティ基本法**: 重要インフラ該当の確認、所管省庁ガイドライン
- **労働安全衛生法**: 開発現場の労働環境（リモート含む）
- **下請法**: ベンダー契約の公正性
- **不正アクセス禁止法**: 内部侵害対応の法的手続
- **電子帳簿保存法**: 監査ログ・業務証跡の保存要件（該当あれば）
- **GDPR / CCPA（海外展開時）**: 採用側のマルチクラスタ移行時、越境データ移転時の規制

法令の詳細対応は [30_非機能要件/E_セキュリティ.md](../30_非機能要件/E_セキュリティ.md) と連動。

**崩れた時**: 法令違反が発覚、監督官庁対応と信用毀損。

**受け入れ基準**:

- リリース時点 で適用法令一覧を法務部と合意
- 年次で更新（新法制定、改正対応）
- 遵守状況を四半期ごとに確認

**優先度**: MUST

## BC-LGL-002: 秘密保持契約（NDA）

**現状**: 外部ベンダー・監査業者・コンサルとの情報共有で秘密保持が曖昧だと、情報漏えいリスク。

**要件達成後**: 以下の場面で NDA を締結する。

- **ベンダー契約**: k1s0 関連のハードウェア・ソフトウェア・サービス調達時
- **コンサル契約**: 技術コンサル、法務コンサル、会計コンサル
- **監査契約**: セキュリティ監査、ペネトレーションテスト、内部監査（外部委託時）
- **共同研究**: 他企業・大学との技術連携（採用後の運用拡大時）
- **採用面接**: 技術面接で機密情報を開示する場合

NDA テンプレートは法務部が定めた標準を使用、逸脱は個別レビュー。

**崩れた時**: 秘密情報漏えいで損害発生、訴訟リスク。

**受け入れ基準**:

- 標準 NDA テンプレートを リリース時点 で用意
- 新規外部接触時の NDA 締結を必須化
- 締結台帳を法務部が管理

**優先度**: MUST

## BC-LGL-003: ベンダー契約

**現状**: ハードウェア・ソフトウェア・サービスベンダーとの契約条件が不統一だと、リスク対応が属人化。

**要件達成後**: 以下の条項を含む標準契約を使用する。

- **契約範囲**: 提供するハードウェア・ソフトウェア・サービスの詳細
- **SLA**: 可用性、サポート対応時間、パフォーマンス保証
- **賠償責任**: 上限、除外事項、免責
- **知的財産**: 著作権、特許、商標の帰属
- **秘密保持**: NDA 相当の条項、契約終了後の期間
- **データ取扱**: 個人情報・機密情報の取扱、サブプロセッサ管理
- **監査権**: 採用側組織側の監査実施権
- **契約終了**: 解除条件、契約終了時のデータ移行・返還
- **準拠法・管轄**: 日本法、東京地裁を標準

契約締結は法務部レビュー必須、金額や契約期間によって承認権者を差別化。

**崩れた時**: 不利な条項で損害発生、または紛争時に対応困難。

**受け入れ基準**:

- リリース時点 で標準契約書テンプレート作成
- ベンダー契約締結時の法務レビューを義務化
- 契約台帳を維持、更新期限を管理

**優先度**: MUST

## BC-LGL-004: 個人情報取扱

**現状**: 個人情報取扱の契約条項が不備だと、個人情報保護法違反リスクが残る。

**要件達成後**: 個人情報取扱に関して以下を文書化する。

- **取扱範囲**: k1s0 が扱う個人情報の種類・項目・目的
- **委託**: 外部ベンダーへの処理委託（個人情報保護法 25 条）の契約締結
- **サブプロセッサ**: 委託先がさらに委託する場合の承認フロー
- **保管場所**: 国内保管の徹底、海外データセンター利用禁止（特別な合意がある場合を除く）
- **越境移転**: 海外拠点への移転は個人情報保護法 28 条対応、本人同意または十分性認定国制限
- **削除**: 本人の削除要求への対応（保有期間終了または本人申出）
- **情報漏えい対応**: OR-INC-005 と整合、個人情報保護委員会通知プロセス

**崩れた時**: 個人情報保護法違反で改善命令・罰金・信用毀損。

**受け入れ基準**:

- リリース時点 で個人情報取扱台帳作成
- プライバシーポリシーに反映（採用側のマルチクラスタ移行時、社外アクセス時）
- 年次で取扱内容をレビュー

**優先度**: MUST

## BC-LGL-005: 責任分界（社内 SLA）

**現状**: 事故発生時の責任所在が不明だと、復旧対応と補償交渉で混乱。

**要件達成後**: テナント部門とプラットフォーム部門の責任分界を以下で定義する。

- **プラットフォーム部門の責任**: tier1 API の可用性、tier1 が管理する OSS の運用、インフラ基盤、共通セキュリティ
- **テナント部門の責任**: tier2/tier3 アプリのコード、業務ロジック、アプリ固有のデータ取扱、Feature Flag 運用
- **共同責任**: テナント間通信の認可設計、データ移行、インシデント情報共有
- **対応窓口**: Severity 1/2 の IC（OR-INC-003）が最終調整
- **補償**: 社内振替のため金銭補償は原則なし、MTTR 短縮努力義務

**崩れた時**: 事故対応で互いの責任を押し付け合い、復旧遅延。

**受け入れ基準**:

- リリース時点 で責任分界マトリクス確定
- テナントオンボーディング時に合意書に反映
- インシデント後レビューで責任分界を見直し

**優先度**: MUST

## BC-LGL-006: 業務委託契約

**現状**: 外部開発者への業務委託契約が不備だと、知的財産の帰属や秘密保持でトラブル。

**要件達成後**: 以下の条項を含む業務委託契約を締結する。

- **成果物の帰属**: 採用側組織のに帰属、著作権・特許の譲渡
- **秘密保持**: NDA 相当、契約終了後 5 年
- **下請け禁止**: 採用側組織の事前承認なく再委託禁止
- **報酬**: 金額、支払条件、遅延時の利息
- **契約期間**: 明示、自動更新条件
- **解除**: 解除事由、解除時の成果物引渡
- **損害賠償**: 上限、免責事項
- **競業避止**: 契約終了後 1 年間は競合他社での類似業務禁止（限度あり）

下請法・労働者派遣法に抵触しない形式（請負契約または準委任契約）を選択。

**崩れた時**: 知的財産が外部に帰属してしまい、プラットフォーム継続に支障。

**受け入れ基準**:

- リリース時点 で標準契約書整備
- 外部開発者起用時の契約締結を必須化
- 下請法チェックリストでコンプライアンス確認

**優先度**: MUST

## BC-LGL-007: 保険と賠償対応

**現状**: 重大事故時の経済的備えがないと、企業体力が不足する恐れ。

**要件達成後**: 以下の保険・対応を検討する。

- **情報漏えい保険**: 個人情報漏えい時の通知費用・損害賠償・調査費用を補償
- **サイバー保険**: サイバー攻撃による損害、業務停止、復旧費用を補償
- **ビジネスインターラプション**: 事業中断による機会損失補償（オプション）
- **内部賠償ルール**: 社内振替のため金銭補償はないが、経営陣謝罪・改善計画・再発防止を Core 対応

保険加入の要否は CRO / 法務部門と協議、採用後の運用拡大時に本格検討。

**崩れた時**: 重大事故で企業財務に想定外のインパクト。

**受け入れ基準**:

- 採用後の運用拡大時 で保険加入の費用対効果検討
- 賠償対応フローを Runbook 化
- 年次でリスク評価更新

**優先度**: COULD（採用後の運用拡大時）
