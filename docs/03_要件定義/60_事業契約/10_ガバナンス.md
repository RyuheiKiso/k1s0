# 10. ガバナンス

本書は k1s0 プロジェクトの意思決定機構、ADR（Architecture Decision Records）プロセス、変更管理、ステージゲート、役割責任（RACI）を定義する。

## 本書の位置付け

プラットフォーム開発の成否は、技術力よりも意思決定の速さと一貫性で決まる。個人の判断が蓄積して矛盾すると、3 年後に「なぜこの設計なのか」の説明がつかない基盤ができる。本書はこの「意思決定の品質」を組織的に担保する仕組みを定義する。

## BC-GOV-001: Product Council

**現状**: 技術方針の意思決定者が不明確だと、同じ議論が 3 回繰り返され意思決定が遅延。

**要件達成後**: Product Council を月次で開催し、以下の意思決定を行う。

- **参加者**: 情シスマネージャ（議長）、プラットフォーム SRE リード、tier2/tier3 代表 2 名、法務代表、セキュリティ代表
- **議題**: ロードマップレビュー、ADR 承認、テナント受入判断、予算執行、重大インシデントレビュー、新技術採用
- **意思決定ルール**: コンセンサス優先、決着しない場合は議長判断
- **議事録**: Backstage に全議事録を公開、Action Item の追跡
- **臨時開催**: Severity 1 事故発生時、緊急議題時

**崩れた時**: 意思決定が属人化、担当者不在時に停滞。

**受け入れ基準**:

- リリース時点 で Product Council 発足
- 月次定例開催、参加率 80% 以上
- 議事録公開と Action Item 完了率 80% 以上

**優先度**: MUST

## BC-GOV-002: ADR プロセス

**現状**: 技術判断の理由が残らないと、3 年後に同じ議論を繰り返す。

**要件達成後**: ADR（Architecture Decision Records）を以下のルールで運用する。

- **ADR の記述**: コンテキスト、決定、結果、代替案、トレードオフ、反対意見を散文で記述
- **番号管理**: 横断判断は `ADR-NNNN`、領域別判断は `ADR-<DOMAIN>-NNN` で採番し、削除せず superseded で更新
- **承認**: Product Council でレビュー、コンセンサス取得後に Accepted ステータス
- **公開**: Backstage に全 ADR を公開、検索可能
- **参照**: 要件定義書・構想設計・基本設計から ADR を参照し、判断の背景を辿れる

ADR テンプレートは Michael Nygard 版を基礎とする。

**崩れた時**: 判断理由が闇の中となり、リプレース時に 2 倍の検討時間。

**受け入れ基準**:

- リリース時点 で ADR テンプレート確定
- 主要技術判断を全て ADR 化
- 四半期ごとに ADR の陳腐化レビュー

**優先度**: MUST

## BC-GOV-003: ステージゲート

**現状**: 段階間の判断基準が曖昧だと、やるべきでないタイミングで次段階 に突入し失敗。

**要件達成後**: 各段階 間にステージゲートを設け、以下で通過判定する。

- **リリース時点 → 1a**: 採用検討、予算確保、プラットフォーム SRE チーム結成
- **リリース時点 → 1b**: 採用初期 サンプルアプリ稼働、デモ合格
- **リリース時点 → 1c**: NFR-D-PLN-001 パイロット合格基準達成
- **リリース時点 → 2**: 運用体制確立、RTO 4 時間達成、Runbook 15 本整備、継続判断承認
- **採用後の運用拡大時 → 3**: 複数テナント稼働、DORA Four Keys 改善実証、TCO 予実一致
- **採用側のマルチクラスタ移行時 → 4 以降**: マルチクラスタ安定稼働、外部監査合格

各ゲート通過基準は 段階ゲート文書に明文化、Product Council で判定。

**崩れた時**: 時期尚早な 段階進行で事故頻発、信頼失墜。

**受け入れ基準**:

- 段階ゲート文書を リリース時点 で整備
- 各段階 移行時に Product Council で判定
- 未達時の是正計画 or 撤退判断（OR-EXIT-001 と整合）

**優先度**: MUST

## BC-GOV-004: 役割責任（RACI）

**現状**: 責任分担が曖昧だと、重要タスクが落ちる。

**要件達成後**: 主要業務について RACI マトリクスを整備する。

RACI: Responsible（実行）、Accountable（最終責任）、Consulted（協議）、Informed（通知）

主要業務の例:

- **tier1 API 仕様策定**: プラットフォーム SRE（R/A）、tier2/tier3（C）、Product Council（I）
- **OSS 採用決定**: プラットフォーム SRE（R）、Product Council（A）、tier2/tier3・法務・セキュリティ（C）
- **Severity 1 インシデント対応**: インシデントコマンダー（R）、情シスマネージャ（A）、CIO（I）
- **個人情報漏えい対応**: 情シスマネージャ（R）、CIO（A）、法務部（C）、個人情報保護委員会（I）
- **ベンダー契約締結**: 調達部門（R）、法務部（A）、プラットフォームリード（C）、CIO（I）

RACI マトリクスは Backstage で公開、四半期ごとに見直し。

**崩れた時**: 責任の所在が不明で、重要業務の遅延や漏れ。

**受け入れ基準**:

- リリース時点 で主要業務の RACI 完成
- オンボーディング時に RACI を説明
- 年次で見直し

**優先度**: MUST

## BC-GOV-005: 変更管理

**現状**: 変更が無秩序に行われると、追跡不能で事故時の原因特定が困難。

**要件達成後**: OR-ENV-004（ITIL 変更管理）と整合した変更管理を実施する。

- **Standard Change**: 事前承認済み、Argo CD で自動適用
- **Normal Change**: CAB 承認、計画実施
- **Emergency Change**: 事後 CAB 報告

全ての変更は Git 履歴で追跡可能、Deployment ログは Backstage で可視化。

**崩れた時**: 無秩序な変更で本番事故、原因特定に数日。

**受け入れ基準**:

- リリース時点 で変更分類確定
- 月次変更レポート公開
- 緊急変更の比率 10% 以下を目標

**優先度**: MUST

## BC-GOV-006: リスクガバナンス

**現状**: リスクの識別と対応が属人的だと、重大リスクが放置される。

**要件達成後**: リスクレジスタ（[00_要件定義方針/07_リスクレジスタ.md](../00_要件定義方針/07_リスクレジスタ.md) 予定）を Product Council で四半期レビュー。

- **リスク識別**: 技術リスク、運用リスク、セキュリティリスク、法務リスク、人的リスク、財務リスク
- **評価**: 発生確率（Low/Med/High）× 影響度（Low/Med/High）で 9 マトリクス化
- **対応**: Accept / Avoid / Mitigate / Transfer の 4 戦略
- **モニタリング**: 四半期レビューで状態更新、新規リスク追加

**崩れた時**: リスクが顕在化してから対応を考え始め、対応遅延。

**受け入れ基準**:

- リリース時点 でリスクレジスタ初版
- 四半期ごとにレビュー
- リスク識別漏れを年次監査で確認

**優先度**: MUST

## BC-GOV-007: 倫理と DEI（Diversity, Equity, Inclusion）

**現状**: 組織文化の不備は優秀な人材の離職を招く。

**要件達成後**: 以下の原則を運用する。

- **行動規範**: Contributor Covenant 準拠の Code of Conduct
- **多様性採用**: 性別・年齢・出身・障害の多様性を意識した採用
- **心理的安全性**: 失敗をプロセス改善に繋げる Blameless 文化
- **労働時間**: 36 協定遵守、オンコール負担の公平化
- **教育機会**: 全員が平等にトレーニング・認定取得機会

**崩れた時**: 優秀人材が離職、採用ブランド毀損。

**受け入れ基準**:

- リリース時点 で Code of Conduct 策定
- 離職率・満足度を半期ごとに追跡
- ハラスメント相談窓口を明示

**優先度**: SHOULD
