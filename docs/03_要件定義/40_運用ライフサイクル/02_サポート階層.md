# 02. サポート階層

本書は k1s0 の利用者（tier2/tier3 開発者と tier3 エンドユーザ）からの問い合わせを受け止めるサポート体制の階層定義である。L1/L2/L3 の責任分界、受付チャネル、対応 SLA、FAQ・ドキュメント整備、エスカレーション経路を確定する。

## 本書の位置付け

問い合わせの受付体制が曖昧だと、tier2 開発者は Slack で運用チームに直接質問し、運用チームは夜間の Severity 1 対応と日中の Q&A が同一チームに集中して疲弊する。本書は問い合わせを階層化し、一次回答の多くを FAQ とセルフサービスで解決する構造を定める。目標は「運用チームへのエスカレーション率 30% 以下」。

## OR-SUP-001: 3 層サポートモデル

**現状**: 問い合わせが全て運用チームに集中する体制では、運用チームが本来の可用性維持活動に時間を割けない。

**要件達成後**: L1/L2/L3 の 3 層サポートを以下のように定義する。

- **L1（セルフサービス + FAQ + Backstage）**: 一般的な使い方、テンプレート利用、トラブルシューティングの FAQ。利用者が自ら解決する層。Backstage の TechDocs と Ask Backstage（将来的には AI チャット導入）で対応。
- **L2（tier2/tier3 コミュニティチャネル）**: Slack `#k1s0-users` でコミュニティが Q&A。プラットフォーム SRE チームの一部が輪番でファシリテート。
- **L3（プラットフォーム SRE チーム）**: L2 で解決しない技術問題、バグ、機能要望、障害影響調査。チケット起票で正式受付。

各層の解決率目標: L1 50%、L2 30%、L3 20%（エスカレーション率 20%）。

**崩れた時**: 全ての質問が L3 に直行し、SRE チームが疲弊してインシデント対応能力が低下する。

**受け入れ基準**:
- L1 FAQ 50 エントリ以上を リリース時点 で整備
- Slack チャネル `#k1s0-users` を リリース時点 で開設、輪番ファシリテータを設定
- L3 チケット起票フォーム（Backstage 連携）を リリース時点 で提供
- 四半期ごとにエスカレーション率をモニタリング

**優先度**: MUST

## OR-SUP-002: 対応 SLA

**現状**: 問い合わせの返答時間が不定だと、tier2 開発者が k1s0 採用に躊躇する。

**要件達成後**: 問い合わせ種別ごとの対応 SLA を以下に確定する。

- **L3 バグ報告（Severity High）**: 初動応答 4 時間（営業時間内）、解決目標 3 営業日
- **L3 バグ報告（Severity Medium/Low）**: 初動応答 1 営業日、解決目標 10 営業日
- **L3 機能要望**: 初動応答 3 営業日、Product Council でのトリアージ 10 営業日
- **L3 技術質問**: 初動応答 1 営業日、解決目標 3 営業日
- **L2 コミュニティ質問**: ベストエフォート（中央値 4 時間を目標）

営業時間は平日 9:00〜18:00。休日夜間の L3 対応は Severity 1/2 のインシデント対応に限定。

**崩れた時**: 返答遅延で tier2 開発者が迂回ワークアラウンドを実装、k1s0 の推奨プラクティスから乖離する。

**受け入れ基準**:
- SLA 未達時のエスカレーションフローを定義
- 月次で SLA 達成率を Grafana ダッシュボード化
- SLA 達成率 90% 以上を維持

**優先度**: MUST

## OR-SUP-003: セルフサービスドキュメント

**現状**: ドキュメントが陳腐化すると、利用者が古い情報で実装し、後工程で手戻り。

**要件達成後**: セルフサービスドキュメントを以下の構造で整備する。

- **Getting Started**: 初回利用時の 30 分チュートリアル（tier2 開発者向け / tier3 開発者向け の 2 種）
- **API リファレンス**: tier1 11 API の Protobuf IDL から自動生成、バージョン別に並列公開
- **チュートリアル**: ユースケース別（イベント駆動ワークフロー構築、RBAC 実装、tenant_id 運用など）15 種以上
- **リファレンス**: 設計決定、ADR、制約事項、トレードオフ解説
- **FAQ**: 質問と回答のペア、タグ検索可能

ドキュメントは Backstage TechDocs に集約、ソースはリポジトリで管理して PR ベースで更新。

**崩れた時**: 古いドキュメントが残り、利用者の信頼を損なう。

**受け入れ基準**:
- ドキュメント鮮度 90% 以上（直近 6 か月以内に検証済みの割合、NFR-C-MNT-001 と整合）
- リリース時点で Getting Started とチュートリアル 10 本 / 15 本
- 四半期ごとに鮮度レビュー、陳腐化コンテンツを削除または更新

**優先度**: MUST

## OR-SUP-004: 問い合わせトリアージ

**現状**: L3 に上がった問い合わせが整理されずに溜まると、重要案件が埋もれる。

**要件達成後**: L3 受付チケットは以下のトリアージプロセスを経る。

1. **受付**: Backstage フォームまたはメールでチケット起票、自動で Jira（または同等）に転記
2. **一次トリアージ（1 営業日以内）**: 当番 SRE が内容を読み、分類（バグ／要望／質問）と初期 Priority（P1/P2/P3）を付与
3. **割当**: P1 は即日、P2 は 3 営業日以内、P3 は Backlog に配置
4. **解決**: 対応実施、利用者への返信、チケットクローズ
5. **振り返り**: 月次で全チケットを集計、FAQ 化できる質問を抽出

**崩れた時**: 当番不在で受付放置、利用者から「質問したのに返事が来ない」の不満が発生。

**受け入れ基準**:
- 当番 SRE のローテーション表を四半期ごとに公開
- 一次トリアージ 1 営業日 SLA 達成率 95% 以上
- 月次レポートで解決率と FAQ 化件数を可視化

**優先度**: MUST

## OR-SUP-005: 受付チャネルと連絡先統制

**現状**: Slack DM や口頭で直接問い合わせが来ると、記録が残らず対応漏れが発生する。

**要件達成後**: 以下の受付チャネルのみを正式窓口とし、他は全て転送・誘導する。

- **Slack `#k1s0-users`**: L2 コミュニティ質問（オープン議論）
- **Backstage Ask フォーム**: L3 チケット起票（正式記録）
- **メール `k1s0-support@jtc.example.com`**: メール利用者向け、自動で Backstage に転記
- **緊急連絡（PagerDuty オンコール）**: Severity 1/2 のインシデント疑義のみ

Slack DM や個人メールでの問い合わせは「正式チャネルへ誘導」で統一する。

**崩れた時**: 問い合わせの総量と性質が可視化できず、改善計画が立てられない。

**受け入れ基準**:
- 運用チームメンバーが DM を受けた際のテンプレート応答（正式チャネル案内）を共有
- 受付チャネル統制を Backstage の FAQ と採用ガイドで周知
- 四半期ごとに受付チャネル別件数を集計

**優先度**: MUST

## OR-SUP-006: 利用状況とサティスファクションスコア

**現状**: サポート品質を主観で評価すると、改善サイクルが回らない。

**要件達成後**: 以下の指標を定量測定する。

- **サポート満足度（CSAT）**: チケットクローズ時に 5 段階評価を利用者から取得、四半期平均 4.0 以上
- **NPS（Net Promoter Score）**: 半期ごとに tier2/tier3 開発者に「k1s0 を同僚に推奨するか」調査、リリース時点+40、採用後の運用拡大時60 を目標
- **ドキュメント閲覧ヒートマップ**: Backstage の閲覧ログから頻出質問領域を特定
- **解決までの平均時間**: Priority 別に可視化、四半期ごとに短縮目標を設定

**崩れた時**: サポート体制の改善優先度が感覚論で決まり、投資配分が適切に行えない。

**受け入れ基準**:
- CSAT 測定機構を リリース時点 で導入
- NPS 調査を 採用後の運用拡大時 開始時にベースライン取得
- 月次サポートレポートで上記指標を全て可視化

**優先度**: SHOULD
