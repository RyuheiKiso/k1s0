# 06. FMEA 分析

本書は k1s0 プラットフォームの主要 15 コンポーネントについて、故障モード・影響・原因分析（FMEA: Failure Mode and Effects Analysis）を実施し、各コンポーネントが故障した際の影響度・発生頻度・検出容易性を RPN（Risk Priority Number）で数値化する。OR-INC-007（FMEA 実施）の具体化であり、インシデント対応フロー（OR-INC-002）の Runbook 整備の基礎資料となる。

## 本書の位置付け

プラットフォームが稼働を始めると、コンポーネントのどこが落ちても「それに対する初動が決まっていない」事態が許されない。Runbook 整備（NFR-A-REC-002）の前提として「何が起きうるか」を網羅的に列挙する必要がある。FMEA は信頼性工学の標準手法で、製造業・航空宇宙・医療機器で使われてきた体系的な故障分析プロセスである。本書はこれをソフトウェアプラットフォームに適用し、tier1 API・データ層・ネットワーク層・コントロールプレーンの代表的故障モードを整理する。

RPN は 1〜1000 のスケールで、以下 3 つの評価軸の積で算出する。

- **Severity（影響度）1〜10**: 故障が発生した時の業務影響の大きさ
- **Occurrence（発生頻度）1〜10**: その故障モードが発生する頻度
- **Detection（検出容易性）1〜10**: 故障発生を早期検知できる容易さ（10 は検出困難、1 は即座検知）

RPN が高いほど優先的な対策が必要。100 以上は要即時対策、50〜99 は要計画的対策、49 以下は監視継続で許容可能と分類する。

## 優先対策リスト（RPN 100 以上）

RPN 100 超はオンコール現場での再現性と影響範囲の両面で看過できない水準であり、リリース時点 までに対策計画を確定して リリース時点 までに運用へ組み込む。本節は個別故障モードを単なる対策行で終わらせず、「なぜこの RPN になったか」「どの業務が崩れるか」「対策がどう効くか」を 2〜3 文で示す。表や箇条書きのラベル羅列では、対策担当者が着手時に意図を読み取れないためである。

- **Temporal Determinism 違反（RPN 135）**: Workflow リプレイ時に非決定処理（時刻取得、ランダム、外部呼出）が混入していると再実行結果が原本と食い違い、業務データの状態が発散する。対策は CI に replay test を必須化し、Determinism 違反検知ツール（`go test` + Temporal SDK の replayer）を PR パイプラインで走らせること。リリース時点で 仕組みを導入し / 違反ゼロを合格ライン化する
- **ZEN Engine ルール評価バグ（RPN 126）**: 決定表の評価結果が誤っても呼出側は誤り気づけないため、業務判定が静かに壊れる。ゴールデンケース回帰テスト（ルール × 入力の期待出力マトリクス）を リリース時点 から積み上げ、段階リリース（canary 1% → 10% → 100%）で本番差分を検出する。評価 trace は常時記録し、事後の原因特定に用いる
- **OpenBao 権限ポリシー不備（RPN 120）**: 過剰権限で漏洩時の影響範囲が指数的に拡大する。policy as code（HCL をコード管理、PR レビュー必須）、Kyverno での事前検証（禁止パターンを manifest 段階で弾く）、四半期ごとの権限棚卸しの 3 層で抑止する
- **ZEN Engine ルールバージョン不整合（RPN 90）**: tier2 が旧バージョンで評価を継続するとロールアウト直後に業務判定が分岐する。ルール ID とバージョンのピン止めを強制し、rollback 手順書で「ピンを 1 つ前のバージョンに戻す」だけで収束させる
- **Istio Ambient ポリシー誤設定（RPN 84）**: L7 認可ポリシーの誤記で正当な内部呼出が拒否され、サービス連鎖で業務断が連鎖する。Kyverno での事前検証と staging 先行適用（最短 24 時間）で、本番到達前に拒否パターンを検出する
- **Kafka コンシューマラグ拡大（RPN 84）**: Consumer の処理遅延がイベント駆動業務の遅延に直結し、Workflow のタイムアウト閾値を超えると業務が失敗扱いになる。ラグ閾値アラート（Sev3、5 分以上継続で Sev2 昇格）と HPA 連動でコンシューマ並列度を自動拡大する
- **MinIO バケットポリシー誤り（RPN 84）**: Public 化誤設定でオブジェクト漏洩が発生する。Kyverno でバケット Public 化を禁止（allow list 外は作成拒否）、月次で全バケットを走査して意図外の公開状態を検知する
- **PostgreSQL レプリ遅延（RPN 81）**: レプリ遅延の放置はフェイルオーバ時の RPO を満たせなくする。`pg_stat_replication` を Prometheus で継続監視し、5 秒以上の遅延を Sev3、30 秒超で Sev2 にエスカレートする
- **Temporal デッドロック（RPN 80、tier1 goroutine/async）**: Workflow Worker の goroutine リークや lease 更新失敗で処理が静止する。goroutine dump 自動取得（`SIGQUIT` ハンドラ）と lease timeout（既定 30 秒）で可視化と自動復旧を実現する

## FMEA の運用と更新

FMEA は一度作って終わりではなく、以下のトリガーで更新する。

- **新コンポーネント追加時**: ADR 策定と同時に FMEA を更新
- **インシデント発生後**: ポストモーテム（OR-INC-004）の結果を反映、既存故障モードの Sev/Occ/Det を調整、新しい故障モードを追加
- **四半期レビュー**: Product Council で全 RPN を見直し、RPN 100 以上の項目の対策進捗を確認
- **外部脅威動向**: CVE 公開・業界インシデント（SolarWinds、Log4Shell 等）を参考に Occ を見直し

FMEA 結果は Runbook（NFR-A-REC-002）の「故障モードと初動」セクションに反映され、オンコールチームの訓練（OR-INC-006）の教材になる。

## 要件 ID

- **OR-FMEA-001**: 主要 15 コンポーネントに対する FMEA を実施、RPN を算出（MUST、リリース時点）
- **OR-FMEA-002**: RPN 100 以上の故障モードに対して対策計画を確定（MUST、リリース時点）
- **OR-FMEA-003**: FMEA 結果を Runbook に反映（MUST、リリース時点）
- **OR-FMEA-004**: 四半期ごとに FMEA を更新、インシデント結果を反映（MUST、継続）
- **OR-FMEA-005**: 新コンポーネント追加時の FMEA 策定を PR テンプレートに含める（SHOULD、リリース時点）

## メンテナンス

本書の Sev/Occ/Det は主観評価を含む。数値の調整には Product Council のコンセンサスを要する。業界共通の FMEA 評価基準（AIAG-VDA FMEA Handbook 2019）を参考にしつつ、k1s0 固有の業務文脈で調整する。
