# OPS-INC: インシデント対応要件

本ファイルは、k1s0 のインシデント検知から復旧・ポストモーテムまでの一連を**定量化された応答時間とロール責任**として要件化する。平常時の当番体制は [`00_operation.md`](./00_operation.md)、SLO 閾値は [`../20_品質特性/03_sla_slo.md`](../20_品質特性/03_sla_slo.md) を参照。

インシデント対応の「判断遅延」は MTTR の 70% を占めるというのが業界共通の経験則であり、本ファイルは **Sev 判定 → 初期応答 → コマンダー指名 → 復旧 → ポストモーテム** の各段階にストップウォッチを掛ける構造を固定する。

## 要件本体

### OPS-INC-001: Sev1 / Sev2 / Sev3 の定義

- 優先度: MUST（Sev 判定がないと応答時間ゲートが動作しない。全てのインシデント要件の土台）
- Phase: Phase 1c
- 関連: OPS-INC-002 / QUA-SLA-001

現状、「障害の重大度」を判断する共通基準がなく、起案者の主観で対応優先度が決まる。他の仕事で手が離せない時に深夜の障害を「軽微」として翌朝対応にしたことで、業務影響が拡大した事例がある。

要件達成後の世界では、k1s0 は以下の 3 段階で Sev を定義する。**Sev1 = 全面停止**（tier1 API 全体が応答不可 / 全テナントの稟議フロー停止 / 個人情報漏洩確定）、**Sev2 = 主要機能停止**（単一テナント影響 / 稟議承認機能のみ停止 / SLO 違反検知）、**Sev3 = 軽微**（単一ユーザー / 部分機能の劣化 / エラーバジェット消費 10% 以内）。Prometheus Alertmanager のアラートには Sev ラベルが必須付与され、受信側（PagerDuty）が Sev 別のルーティングを実行する。判定は Runbook `sev-triage.md` のフローチャートに従い、SRE 当番が 5 分以内に確定する。

崩れた時、Sev 判定がブレると初期応答時間の監視が意味をなさない。監査で「SLA 違反時の応答時間を測定せよ」と問われて数字を出せず、稟議 / 契約更新でのリスクとなる。

**受け入れ基準**

- Runbook `sev-triage.md` にフローチャートと 6 パターンの具体例が記載
- 全 Prometheus アラートに `severity` ラベル（sev1/sev2/sev3）が必須
- Sev 判定が 5 分以内（SRE 当番の最初の Slack 書き込み時点で確定）
- Sev1/2 は PagerDuty に即時エスカレ、Sev3 は翌営業日 Slack `#ops-queue`
- 四半期で Sev 分布（件数 × 種別）をレビュー、想定外の Sev1 頻発時は根本原因分析

**検証方法**

- PagerDuty Incident ログから Sev 判定時刻と最初の確定時刻の差分を月次集計
- ゲームデー（OPS-INC-005）で模擬障害の Sev 判定を訓練、正答率 90% 以上

### OPS-INC-002: 初期応答時間（Sev1:15分 / Sev2:1h / Sev3:4h）

- 優先度: MUST（SLA 契約上の応答時間保証。未達は tier3 顧客への SLA クレジット発生）
- Phase: Phase 1c
- 関連: OPS-INC-001 / OPS-OPR-001

現状、深夜 Sev1 の応答時間は起案者個人の在宅状況に依存し、最悪 2 時間の無応答が発生しうる。

要件達成後の世界では、PagerDuty のエスカレーションポリシーで Sev1 は 15 分以内、Sev2 は 1 時間以内、Sev3 は 4 時間以内の初期応答を強制する。初期応答とは「SRE 当番が PagerDuty で Acknowledge + Slack `#incident` に `state: investigating` を宣言」するまで。Sev1 で 15 分応答なし → 代替者（L2 ローテ）へ自動エスカレ → さらに 10 分なしで SRE リード / tier1 リードへ同時通知。全 Sev1 インシデントは Zoom の `incident-war-room` 常設ルームで 30 分以内に合流点を設置。

崩れた時、SLA 違反により tier3 顧客との契約で SLA クレジット（契約月額の 10〜25% 返金）が発動する。中規模 3,000 名テナントで月額 300 万円契約の場合、1 インシデントで 75 万円のキャッシュアウト。

**受け入れ基準**

- PagerDuty エスカレーションポリシー `k1s0-production-v1` が 3 段階（当番 → L2 → リード）
- 月次で Sev1 の初期応答 P95 が 15 分以内、Sev2 は 1 時間以内、Sev3 は 4 時間以内
- 応答時間違反件数は月次レビューで公開、違反時は対応改善 Action Item 化
- war room 設置時刻（Sev1 発動から 30 分）を自動計測
- 初期応答から復旧完了までの MTTR を Sev 別に集計、Sev1 は 4h / Sev2 は 8h / Sev3 は 3 営業日以内

**検証方法**

- PagerDuty Incident API で応答時間 P95 を月次集計、Grafana `incident-response-time` に常設
- 四半期のランダム抜き打ちテストで Sev1 模擬発動、応答時間が基準以内を確認

### OPS-INC-003: インシデントコマンダー制

- 優先度: MUST（複数人対応時の指揮系統が無いと復旧判断が迷走し MTTR が倍増する）
- Phase: Phase 1c
- 関連: OPS-INC-001 / OPS-OPR-002

現状、Sev1 発動時に起案者含め複数人が Slack で同時に動き始めるが、「誰が何をしているか」を統合する役割が不在。復旧操作が重複したり抜けたりする。

要件達成後の世界では、Sev1 / Sev2 発動時に **インシデントコマンダー（IC）** を 5 分以内に指名する。IC は tier1 リード / SRE リード / 当番の中から事前定義された優先順で自動指名され、Slack `#incident` チャンネルで `!ic claim` 発言により正式就任。IC の責任は (1) war room の設置、(2) 対応担当の割振り、(3) Sev 昇降格判断、(4) コミュニケーション（ステークホルダーへの 30 分ごとの状況更新）、(5) 復旧宣言、の 5 点。IC は**手を動かさず判断に集中**し、オペレーションは担当者が実施する。

崩れた時、指揮系統が不明のまま「個別最適の復旧操作」が並行し、たとえば片方が Pod 再起動しようとしている時にもう片方が PVC 削除を実行する、という事故が起きる。Phase 2 以降の複数業務同時インシデントでは IC 不在は復旧不能を意味する。

**受け入れ基準**

- IC 指名が Sev1/2 発動から 5 分以内、Slack `!ic claim` で記録
- IC 役割定義が Runbook `incident-commander.md` に整理、四半期 IC 研修を実施
- IC 候補者（tier1 / SRE リード + 当番）のローテ表が Confluence で公開、常時 5 名以上
- war room ログ（Slack `#incident-{id}`）が 365 日保管、ポストモーテム参照用
- 30 分ごとのステークホルダー通知が IC 主導で発信される

**検証方法**

- ポストモーテムレビューで IC 就任時刻 / 退任時刻 / 発言件数を集計
- 年次 IC ドリルで候補者全員の実地訓練

### OPS-INC-004: ポストモーテム 48 時間以内公開

- 優先度: MUST（学習組織としての成立条件。ポストモーテムなしでは同一障害が再発する）
- Phase: Phase 1c
- 関連: OPS-INC-003 / OPS-OPR-002

現状、過去のインシデントは Slack ログが散逸するのみで、根本原因分析も再発防止策の進捗管理もない。「同じ障害を 3 回やった」という事例が他社でも多発している。

要件達成後の世界では、Sev1/2 インシデントの復旧宣言から **48 時間以内** にポストモーテムのドラフトを IC が Confluence の `postmortems/` 配下に作成する。ドラフトは Blameless 原則（個人非難禁止）に従い、以下 6 セクション必須: (1) タイムライン（分単位）、(2) 直接原因、(3) 根本原因（5 Why）、(4) 影響範囲（ユーザー数 / データ量 / 金額換算）、(5) 再発防止策（担当者と期限付き）、(6) 教訓。復旧 7 日以内に tier1 チーム全員レビューを経て Action Items が JIRA `postmortem-action-*` に登録される。Action Items は四半期進捗レビューで完了率 80% 以上を維持。

崩れた時、学習されない組織となり、Phase 2 以降の複雑な障害で同種の失敗を繰り返す。稟議で「前回と同じ原因の障害が再発した」と指摘されるとプラットフォームの信頼が失墜する。

**受け入れ基準**

- Sev1/2 の復旧宣言から 48h 以内にポストモーテムドラフト公開
- 6 セクション必須テンプレート `postmortem-template.md` を使用
- Action Items の完了率が四半期 80% 以上、期限超過は SRE リードにエスカレ
- Blameless 原則のトレーニングを年 1 回全員受講
- 四半期の「過去同種障害の再発率」を可視化、5% 未満を維持

**検証方法**

- Confluence `postmortems/` の作成日時と Incident 復旧時刻の差分を月次集計
- JIRA `postmortem-action-*` の完了率を四半期レビュー

### OPS-INC-005: ゲームデー四半期実施

- 優先度: SHOULD（実地訓練なしでは体制が紙面のまま腐敗する）
- Phase: Phase 2（稼働業務 2 本以上で意味が生じる）
- 関連: OPS-INC-001 / OPS-INC-003 / QUA-DR-001

現状、インシデント対応手順が Runbook に記載されていても、実際に当番がそれを参照しながら正しく動けるかは未検証。

要件達成後の世界では、四半期ごとに**ゲームデー**（意図的な障害注入訓練）を staging で実施する。Litmus Chaos Engineering を使い、事前予告ありの「計画ゲームデー」と事前予告なしの「抜き打ちゲームデー」を年 2 回ずつ（合計 4 回）実施。シナリオは「Dapr sidecar 全停止」「PostgreSQL Primary Down」「Istio 証明書失効」「Kafka Broker 3 台のうち 2 台停止」など、実本番で起こりうる複合障害を含む。ゲームデー後は通常のポストモーテムと同じテンプレートで記録し、Runbook と IC 手順の不備を抽出する。

崩れた時、本番で初めて遭遇する障害パターンが増え、IC の判断時間と復旧時間が大幅に伸びる。実事例で「無訓練チームは初回対応で 2 倍の MTTR」というのが業界データ。

**受け入れ基準**

- ゲームデー実施回数が年 4 回以上（計画 2 + 抜き打ち 2）
- シナリオは SRE リードが四半期前月に Runbook から選定
- ゲームデー記録 `gamedays/YYYY-QX.md` が Confluence に公開
- Runbook の陳腐化 / 欠損を 3 件以上検出、翌月までに更新
- 抜き打ちゲームデーでの Sev 判定正答率が 90% 以上

**検証方法**

- ゲームデー記録件数と改善 Action 件数を年次レビュー
- 抜き打ち時の Sev 判定正答率を集計
