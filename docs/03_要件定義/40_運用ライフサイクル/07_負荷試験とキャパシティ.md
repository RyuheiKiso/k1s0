# 07. 負荷試験とキャパシティプランニング

本書は k1s0 の負荷試験シナリオとキャパシティプランニングを定義する。非機能要件 B（性能拡張性）の p99 < 500ms・100 RPS 初期・1,000 RPS 採用後の運用拡大時 といった数値目標は、実測で裏付けられなければただの希望値にとどまる。本書は負荷試験の手順・シナリオ・合格基準と、観測結果に基づくハードウェア見積の算出方法を定める。

## 本書の位置付け

申請書では「物理 6〜10 ノードで 100 RPS 処理可能」と試算した。この数値の根拠は「推測」ではなく「実測」に基づく必要がある。パイロット稼働前に負荷試験で 1.5〜2 倍のヘッドルームを確保し、実運用で採用検討通過条件を割り込まないことを保証する。また将来のテナント増加時にハードウェア追加をいつ発注すべきかを予測できるキャパシティプランニングが無ければ、ハードウェア調達リードタイム（2〜3 か月）を踏まえた先行投資判断ができない。

本書は k6（Grafana Labs）と Locust を主ツールとし、tier1 API を中心にシナリオを組む。負荷試験は リリース時点 後半から開始し / のパイロット合格基準（NFR-D-PLN-001）の前提条件として合格する必要がある。

## 負荷試験シナリオ

シナリオは業務特性を反映した 4 種類を定義する。単一 RPC の最大スループット試験だけでは実業務の負荷パターンを捉えきれないため、複数 API を組み合わせた複合シナリオを含める。

### シナリオ A: tier1 Service Invoke スループット試験

最も単純な API のスループット上限を測定する基準シナリオ。

- 対象: tier1 Service Invoke API
- 負荷パターン: 100→500→1,000→2,000 RPS の段階上昇、各段階 10 分保持
- 測定項目: p50/p95/p99 レイテンシ、エラー率、tier1 Pod CPU/メモリ
- 合格基準: 1,000 RPS で p99 < 500ms、エラー率 < 0.1%
- 備考: tier1 ファサード層単体の性能上限を把握、以降のシナリオの基準点

### シナリオ B: State API 書込競合試験

ETag による楽観的排他が競合下でも正しく動作することを検証する。

- 対象: tier1 State API（Valkey + PostgreSQL）
- 負荷パターン: 同一キーに 10 並列 Writer、5 分間継続
- 測定項目: 衝突検出率、再試行成功率、書込スループット
- 合格基準: 衝突検出 100%（データ喪失ゼロ）、再試行後成功率 > 95%
- 備考: 業務で「同一発注の同時更新」のような競合シーンを模擬

### シナリオ C: 業務複合シナリオ（FR 全体）

実業務を模した複数 API の連鎖呼出で、エンドツーエンドの応答性能を測定する。

- 対象: tier1 Service Invoke → State Get → Decision Evaluate → PubSub Publish → Workflow Start
- 負荷パターン: ユーザー 200 並列、各ユーザーが 1 秒間隔で業務シーケンスを実行、30 分継続
- 測定項目: シーケンス全体の p99 レイテンシ、各 API の寄与、tier2/tier3 側の処理時間
- 合格基準: シーケンス全体で p99 < 2 秒（各 API < 500ms の積算）
- 備考: パイロット業務の実パターンに近づけるため、tier2 側の模擬実装を含める

### シナリオ D: ソーク試験（長時間負荷）

メモリリーク・接続プール枯渇など時間経過で顕在化する問題を検出する。

- 対象: プラットフォーム全体
- 負荷パターン: 100 RPS を 48 時間継続
- 測定項目: メモリ使用量推移、goroutine/thread 数推移、GC 頻度、コネクション数
- 合格基準: メモリ増加率 < 1%/時（48 時間で 50% 未満）、エラー率 < 0.05%、レイテンシ劣化 < 20%
- 備考: 短時間のスパイクテストでは見つからない慢性的問題を検出

## ストレステストとチャオス試験

負荷試験とは別に、異常条件下での振る舞いを検証するストレステストとチャオス試験を実施する。OR-INC-006（訓練）および DX-TEST-006（Chaos Mesh）と連動する。

- **CPU Saturation**: Node 1 台の CPU を 80% に固定、HPA・PDB が正しく動作するか
- **Memory Pressure**: Pod にメモリ制限を強制しつつ高負荷、OOM Kill と再起動の挙動確認
- **Network Partition**: Istio 経由の通信を 5 秒遮断、リトライ・タイムアウト動作確認
- **Disk Pressure**: ディスク使用率 95% で Postgres が書込拒否するか、アラート発報タイミング
- **Pod Kill**: ランダムに Pod を kill、リカバリ時間（MTTR）測定

## キャパシティプランニング

キャパシティは「単位リソース当たり処理能力」と「想定負荷」から逆算する。シナリオ A〜C の測定結果を基にユニットサイズを決定し、以下の式で必要リソースを算出する。

- 必要 CPU = 目標 RPS × 1 リクエストあたり CPU 時間 / CPU 効率（80%）
- 必要メモリ = 基本メモリ + 同時接続数 × 接続あたりメモリ
- 必要ストレージ = 初期データ + 月次増加量 × 保管期間

### ユニットサイズ実測値（目標値、リリース時点 で確定予定）

負荷試験完了時点で以下を確定させる。現時点は計画値。

- tier1 ファサード Pod: 2 CPU / 4GB メモリで 200 RPS 処理想定
- PostgreSQL（CloudNativePG）: 4 CPU / 16GB メモリで 1,000 トランザクション/秒想定
- Kafka ブローカー: 2 CPU / 8GB メモリで 10,000 メッセージ/秒想定
- Valkey: 1 CPU / 4GB メモリで 50,000 ops/秒想定

### ノード見積（リリース時点 パイロット規模、100 RPS 想定）

- tier1 ファサード: 2 Pod（冗長性 + HPA ヘッドルーム）
- tier2 アプリ: 3 Pod
- PostgreSQL: 3 Pod（Primary + Standby 2）
- Kafka: 3 Pod（replication factor 3）
- Valkey: 3 Pod（Sentinel 構成）
- その他（Keycloak, OpenBao, Argo CD, Kyverno, Grafana）: 各 1〜2 Pod

これらを集約した Kubernetes Node は物理 6 台（各 16 CPU / 64GB）で収容可能と試算。各 Node に 80% までを目安に配置し、1 台停止時も残り 5 台で継続運用可能な冗長性を確保。

### スケール閾値（いつハードを増やすか）

- **CPU 使用率 平均 60% 超が 1 週間継続**: Node 1 台追加の発注申請を起票
- **メモリ使用率 平均 70% 超**: 同上
- **ストレージ使用率 70% 超**: PVC 拡張 → 容量追加の発注申請起票
- **HPA が maxReplicas に張り付く時間 > 週間 10%**: Pod 増加ではなく Node 増加検討

### 段階別キャパシティ見込み

- **採用初期**: 100 RPS、6 Node 物理
- **リリース時点（採用初期, パイロット）**: 200 RPS 想定、8 Node 物理（パイロット業務+余裕）
- **リリース時点（採用初期, 運用体制）**: 500 RPS 想定、10 Node 物理
- **採用後の運用拡大時（テナント拡大）**: 1,000〜3,000 RPS、15〜20 Node 物理
- **採用側のマルチクラスタ移行時（マルチクラスタ）**: 5,000 RPS 超、複数クラスタ構成

## 負荷試験のスケジュールと責任

負荷試験はプラットフォームチームが実施、tier2/tier3 開発チームが業務シナリオを提供する。リリース時点の終盤に月次 1 回のペースで開始し、採用初期（パイロット合格前）は週次に頻度を上げ、以降は大規模変更ごとに実施する。

- **月次基礎試験**: シナリオ A のみ、定点観測でリグレッションを検出
- **四半期包括試験**: シナリオ A〜D 全部、レポートを Product Council へ提出
- **パイロット前試験**: 本番ハードで全シナリオ + ストレステスト、合格基準 100% 通過
- **大規模変更後**: ADR-0001 等のインフラ変更、ADR-TIER1-003 等のアーキテクチャ変更時に全シナリオ再実行

## 負荷試験環境

本番と可能な限り同一の構成を staging で維持し、負荷試験はそこで実施する。

- **staging 物理構成**: 本番と同等ハード構成（ノード数は 50〜75% 規模）
- **データ量**: 本番想定の 50% 相当の模擬データを投入
- **外部連携**: 可能な範囲でモック化、DOTNET 連携はサイドカー実装をそのまま利用
- **観測**: Grafana LGTM で本番同等のダッシュボードを維持

負荷試験専用環境を別途立てる案もあるが、コストが倍増するため staging と兼用とする。負荷試験時は staging での他の作業（デプロイ検証等）と時間帯を分離する運用。

## 要件 ID

- **OR-LOAD-001**: 負荷試験シナリオ A〜D を定義、負荷試験ツール（k6）で実施可能な形で整備（MUST、リリース時点）
- **OR-LOAD-002**: 採用初期（パイロット）合格判定前に全シナリオの合格を必達条件に設定（MUST、リリース時点）
- **OR-LOAD-003**: 月次基礎試験・四半期包括試験を定期実行（MUST、リリース時点）
- **OR-CAP-001**: キャパシティプランニングのスケール閾値を運用に組み込み（MUST、リリース時点）
- **OR-CAP-002**: ハードウェア発注申請のリードタイム（2〜3 か月）を考慮した先行発注プロセスを確立（MUST、リリース時点）
- **OR-CAP-003**: 段階別キャパシティ見込みを年次で再評価（MUST、継続）
- **OR-LOAD-004**: ストレステスト・チャオス試験を四半期包括試験に含める（SHOULD、リリース時点）
- **OR-LOAD-005**: 負荷試験結果を Grafana ダッシュボードに永続化、リグレッション検出（SHOULD、リリース時点）

## メンテナンス

負荷試験シナリオは業務パターンの変化に応じて見直す。パイロット業務から横展開する際、業務特性が異なるテナントが参加した時点でシナリオ C を拡張する。キャパシティプランニングのユニットサイズは四半期ごとの実測値で更新、計画値と実測値の乖離が 30% を超えたら見積モデルを再構築。
