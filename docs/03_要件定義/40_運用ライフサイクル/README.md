# 40_運用ライフサイクル

本カテゴリは k1s0 が本稼働した後の「運用の契約」を定義する。30_非機能要件の C 運用保守性と重なる部分はあるが、本カテゴリはより手続的で、運用部門・サポート窓口・利用部門との間で誰が何をいつやるのかの責任分界まで踏み込む。30 系は要求仕様、40 系は運用契約と理解してほしい。

## 本カテゴリの狙い

情シス運用部門の稟議通過を左右するのは「本稼働後に自分たちが抱える負荷がどこまでか」の明示である。具体的にはインシデント発生時の一次受け、夜間休日の呼び出し、問い合わせの受け皿、非推奨機能の扱い、そして失敗時の撤退経路。これらが曖昧だと「なんとなく k1s0 全部が運用部門の責任」という認識になり、情シス運用部門長の反対で稟議が止まる。本カテゴリはこの認識齟齬を閉じるための文書である。

## 読み順

通読する場合は 01 → 02 → 03 → 04 → 05 → 06 → 07 → 09 → 08 の順に読む。01 〜 02 は「事故と問い合わせをどう捌くか」の日次運用、03 は環境を健全に保つ継続運用、04 〜 05 は長期的なライフサイクル管理。06 〜 07 は Runbook 整備の根拠となる故障分析と容量試験、09 が Runbook カタログ（15 本の目録）、08 がその中の代表 5 本の詳細サンプルという関係である。
