# 40_運用ライフサイクル: 運用と変更管理

本カテゴリは、k1s0 プラットフォームの**運用プロセス・変更管理・インシデント対応・撤退戦略** を要件化する。平常時の運用タスク、リリースと環境管理、インシデント発生時の初動と復旧、そしてシステムを段階的に廃止・撤退する際の手順までを対象とする。

稟議資料で宣言した「運用工数 3〜10 FTE（規模別）」は見込みであり、本カテゴリで**運用タスクの内訳と自動化レベル** を要件化することで、稟議通過後の運用体制の積算根拠となる。インシデント対応と撤退戦略は、事業継続性と OSS ライセンス適合の両面で監査対象となる。

## 運用要件の書き方

運用要件は「**誰が / 何を契機に / どのツールで / どの頻度で / 何を達成するか**」の 5 点を明示する。たとえば「監視を行う」ではなく「**SRE 当番が Grafana のダッシュボード `tier1-slo` で毎朝 9 時にエラーバジェット残を確認し、30% 以下であれば翌日のデプロイ判断を SRE リードに持ち込む**」のように書く。これは `CLAUDE.md` の「受け入れ基準や手順など条件の束は散文の後の箇条書きで列挙」規約に準拠する。
