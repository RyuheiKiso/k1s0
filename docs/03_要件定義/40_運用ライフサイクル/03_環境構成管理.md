# 03. 環境構成管理

本書は k1s0 における環境（dev / staging / prod）の分離方針、環境間の昇格フロー、設定管理（GitOps）、変更管理プロセスを定義する。30_非機能要件/C_運用保守性 で要求した GitOps 化の内側を、具体的なブランチ戦略・環境別差分管理・承認フローまで踏み込んで確定する。

## 本書の位置付け

環境構成が属人化すると、staging で動いたものが prod で動かない現象が頻発し、リリースごとに夜間作業が必要になる。本書は「全ての環境差分を Git 上で表現する」を原則とし、環境昇格を機械的に実施できる状態を要求する。

## OR-ENV-001: 環境分離と命名規則

**現状**: 環境が混在すると、開発者が誤って本番を操作する事故が起きる。

**要件達成後**: 以下の環境を明確に分離する。

- **dev**: 開発者のローカル + 共有開発クラスタ。データは合成データのみ、個人情報を置かない。破壊可能。
- **staging**: 本番相当の構成、本番と同一のスペックを 1/3 規模で保持。匿名化された本番スナップショットを利用可能。リリース前検証専用。
- **prod**: 本番環境。採用側のマルチクラスタ移行時はマルチクラスタ（primary + dr）。
- **sandbox（採用後の運用拡大時）**: PoC・カオス実験用の独立クラスタ。prod データ流入禁止。

命名規則は Kubernetes namespace レベル、Istio トラフィックポリシー、Backstage の Component タグで強制する。

**崩れた時**: 環境名の揺れで「dev なのか staging なのか分からない」状態が発生、誤操作の温床になる。

**受け入れ基準**:

- 環境別クラスタ／ネームスペース命名規則を ADR で確定
- 操作対象環境を CLI 起動時にプロンプトで確認（`kubectx` + 警告プロンプト）
- prod 操作は踏み台経由で操作ログを監査ログに記録

**優先度**: MUST

## OR-ENV-002: GitOps による設定管理

**現状**: kubectl apply の手動オペレーションでは差分追跡が不可能で、構成ドリフトが発生する。

**要件達成後**: Argo CD によって Git リポジトリの状態をクラスタに同期する。Git リポジトリが単一のソースオブトゥルースとなり、クラスタ側の直接変更は禁止。変更は Pull Request 経由のみ。

- **リポジトリ構成**: `k1s0-platform-infra`（プラットフォーム基盤）と `k1s0-apps`（アプリ）の 2 層
- **環境別ディレクトリ**: `envs/dev/`、`envs/staging/`、`envs/prod/` に Kustomize overlay を配置
- **共通ベース**: `base/` にマニフェストのベースライン、overlay で環境差分を表現
- **シークレット管理**: 平文を Git に置かず、External Secrets Operator + OpenBao で差し込み

**崩れた時**: クラスタ側とリポジトリ側で差分（ドリフト）が発生、本番で原因不明の挙動が発生する。

**受け入れ基準**:

- Argo CD で全環境の Sync Status が Synced かつ Healthy を常時維持
- ドリフト検知時は自動 Self-Heal または人手修正（30 分以内）
- 直接 kubectl apply を禁止する RBAC を リリース時点 で導入

**優先度**: MUST

## OR-ENV-003: 環境昇格フロー

**現状**: dev で確認したものが staging をスキップして prod に入ると、本番事故のリスクが上がる。

**要件達成後**: 全ての変更は dev → staging → prod の順で昇格する。昇格は以下のフローで行う。

1. **開発者が feature branch で変更**: アプリコード／マニフェスト／OSS バージョン更新等
2. **Pull Request 作成**: CI で lint・unit test・build・security scan・policy check を実行
3. **dev への自動デプロイ**: PR マージ前にプレビュー環境で動作確認（ephemeral namespace）
4. **PR マージ（main）**: 自動で dev 環境にデプロイ
5. **staging 昇格（自動 PR）**: dev で 24 時間安定稼働後、環境 Overlay を staging に反映する PR を自動生成、運用チームレビュー後マージ
6. **staging 検証**: E2E テスト実行、手動回帰テスト（リリース時点 は手動多め、2+ は自動化）
7. **prod 昇格（承認必須）**: staging で 3 営業日安定後、運用リード承認で prod へ PR マージ

dev → staging → prod の間は必ず段階を踏む。緊急パッチのみ例外扱いで staging 省略可、ただし情シスマネージャ承認必須。

**崩れた時**: prod 固有バグが事故発生時に発見され、段階昇格を無視した責任追及が発生する。

**受け入れ基準**:

- 昇格フロー図を Backstage に公開
- staging → prod 昇格時の承認ログを監査ログに保存
- 緊急パッチの例外適用件数を四半期ごとに集計、多すぎる場合は昇格フロー見直し

**優先度**: MUST

## OR-ENV-004: 変更管理と ITIL 連携

**現状**: 既存情シスの ITIL ベース変更管理（CAB）と整合しないと、採用検討通過後も組織運用で摩擦。

**要件達成後**: 以下の変更カテゴリで ITIL 変更管理と連携する。

- **Standard Change（定型変更）**: 事前承認済み変更（パッチ適用、Helm chart 更新など）。Argo CD の自動適用で記録のみ。
- **Normal Change（通常変更）**: 新機能追加、メジャーアップデート、構成変更。CAB 承認必須、ただし k8s Operator による変更は簡易 CAB で対応。
- **Emergency Change（緊急変更）**: Severity 1/2 インシデント対応に伴う変更。事後 CAB 報告で可。

CAB は週次開催とし、Normal Change の承認を待つ時間を最小化する。

**崩れた時**: 既存 CAB のリードタイム（1 〜 2 週間）が k1s0 の Release Velocity を殺す。

**受け入れ基準**:

- Standard Change カタログを 30 件以上整備（リリース時点）
- CAB 承認リードタイム中央値を 3 営業日以内に短縮
- 変更管理ログを Backstage で可視化、監査対応可能

**優先度**: MUST

## OR-ENV-005: Secrets と Config の分離

**現状**: Secret と Config が混在すると、誤って Secret を Git にコミットするリスク。

**要件達成後**: 以下の原則で厳密に分離する。

- **Config（ConfigMap / 環境変数）**: Git リポジトリに平文で管理。環境別の差分も全て Git に表現。
- **Secrets**: OpenBao に保管、External Secrets Operator が Kubernetes Secret として差し込み。Git には Secret の参照パスのみを記述。
- **Feature Flag**: flagd サーバが管理、Git リポジトリで宣言的に定義可能（Backstage UI からも変更可）。

`.gitignore` に `.env`、`*secret*` 等のパターンを登録し、pre-commit hook で Secret の誤コミットを検出する。

**崩れた時**: Secret が Git 履歴に残存、監査で脆弱性指摘。

**受け入れ基準**:

- pre-commit hook で Secret 検出（truffleHog / gitleaks）を 100% 実行
- Secret ローテーションの自動化（OpenBao + cert-manager で証明書、手順化で API キー）
- 監査で Secret 誤コミット 0 件を維持

**優先度**: MUST

## OR-ENV-006: 設定スキーマバリデーション

**現状**: マニフェストや Helm values のスキーマ違反が本番適用時まで検出されないと、デプロイ失敗でダウンタイム。

**要件達成後**: CI で以下のバリデーションを PR 時点で実行する。

- **Kubernetes マニフェスト**: `kubeval` または `kubeconform` でスキーマ検証
- **Kustomize overlay**: `kustomize build` の成功を確認
- **Helm chart**: `helm lint`、`helm template` 成功を確認
- **OPA/Kyverno ポリシー**: `conftest` または Kyverno CLI で Policy 準拠を確認
- **SAST（Semgrep）**: コード脆弱性スキャン
- **SBOM 差分**: 新規依存の追加を可視化

これらを全て通過しないと PR はマージ不可とする。

**崩れた時**: 本番デプロイ直前にエラー、夜間対応で作業が延びる。

**受け入れ基準**:

- CI 通過率 95% 以上（PR 時点）
- ポリシー違反検出から修正までの平均時間 1 日以内
- 月次で CI 失敗パターンを分析、頻発するものは開発者教育へフィードバック

**優先度**: MUST
