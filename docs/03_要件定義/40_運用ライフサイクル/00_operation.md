# OPS-OPR: 平常時運用要件

本ファイルは、k1s0 プラットフォームの**平常時**（インシデント発生前）における運用タスクと当番体制、手順書の管理、自動化水準を要件化する。インシデント対応時の動作は [`05_incident_response.md`](./05_incident_response.md) で、リリース時の動作は [`02_release.md`](./02_release.md) で扱う。

稟議で合意した「中規模 3,000 名で運用 5.0 FTE、うち保守 1.70 FTE」は見込みであり、本ファイルは**日次・週次・月次で誰が何をするか**を ID 単位で明示する。運用工数の積み上げ根拠は [`../../01_企画/04_定量試算/03_運用工数試算.md`](../../01_企画/04_定量試算/03_運用工数試算.md) に委ね、本ファイルはその工数が実際に配分される運用タスクの定義に限定する。

## 要件本体

### OPS-OPR-001: SRE 当番体制と引き継ぎ

- 優先度: MUST（Phase 1c で業務稼働を始める前提。当番不在では SLO 違反時の初動が担保できない）
- Phase: Phase 1c（業務 1 本稼働時点で最小 2 名ローテ必須）/ Phase 2 で 9-17 時 + 夜間 on-call へ拡張
- 関連: OPS-INC-001 / OPS-INC-002 / QUA-SLO-001

現状、kubeadm + Dapr 基盤は起案者 1 名のみが障害対応可能な状態であり、バス係数が 1 に張り付いている。起案者が休暇を取った日に Pod 再起動が必要になっても、一次対応する人間がいない。

要件達成後の世界では、tier1 チーム内で **SRE 当番** を 1 週間ローテーションで回し、平日 9-17 時は当番が一次窓口、夜間・休日は on-call アサインが PagerDuty 通知を受ける体制が成立する。当番は週明け月曜 10:00 の引き継ぎミーティング（15 分）で前週の未解決チケット・エラーバジェット残・実施予定メンテを口頭共有し、Confluence の「SRE 当番引継ぎ」ページに 5 項目テンプレートで書き戻す。

この要件が崩れた場合、起案者休暇日の障害は 30 分以上応答できず、QUA-SLO-SLA（可用性 99.5%）のエラーバジェットを 1 日で使い切る恐れがある。加えて、Phase 2 以降の 2-3 業務並行稼働で当番不在は「稼働業務停止の属人リスク」として監査指摘対象となる。

**受け入れ基準**

- 当番表が Confluence の `sre-oncall-schedule` に 8 週間先まで公開され、週次で自動リマインドが当番本人へ Slack DM で届く
- 当番引継ぎテンプレート 5 項目（未解決チケット ID / エラーバジェット残 / 予定メンテ / 注意 OSS / 前週教訓）が毎週月曜 10:15 までに更新されている
- 当番不在時（休暇・体調不良）は 24 時間前までに代替者をアサインし、当番表にログが残る
- Phase 2 以降、夜間 on-call は PagerDuty で 15 分以内に一次応答可能であることを四半期テストで確認
- バス係数 2 以上を維持（対応可能者が常時 2 名以上）

**検証方法**

- 四半期ごとに「当番不在日に模擬アラートを発砲し、代替者が 15 分以内に Slack 応答」訓練を実施し、成功率を監査レポートに記録
- 月次で引継ぎテンプレートの更新率を SRE リードが集計し、80% 未満の月は翌月レビューで是正

### OPS-OPR-002: Runbook の Git 管理と更新責任

- 優先度: MUST（Runbook なしでは当番が属人作業に陥り、OPS-OPR-001 の当番体制が空洞化する）
- Phase: Phase 1c（業務稼働開始時点で最低 10 本必須、Phase 2 で 30 本まで拡充）
- 関連: OPS-INC-003 / OPS-OPR-003

現状、運用手順は起案者の頭の中にあるか、散逸した `メモ.md` に記録されているのみで、変更履歴が追えず、他者が再現できない。PostgreSQL の WAL が溢れた時の対処、Istio mTLS 証明書の再発行手順などを当番が即座に参照できない。

要件達成後の世界では、全 Runbook が GitHub リポジトリ `k1s0/runbooks` 配下に Markdown で格納され、PR ベースで更新される。各 Runbook は冒頭に「対象事象」「前提条件」「所要時間目安」「ロールバック手順」を持ち、Backstage TechDocs 経由で当番が **1 クリックで検索可能**。Runbook の更新は対応完了後 48 時間以内に PR 提出必須とし、未提出のインシデントはポストモーテム承認が下りない。

崩れた時は、同じ事象に対して当番ごとに異なる対応が実施され、復旧時間が平均 2 倍に延びる（業界 SRE 事例で Runbook 未整備チームは MTTR が 1.8〜2.2 倍）。加えて、OPS-INC で要求するポストモーテムでも「手順が存在しないから起こった」と結論付けできず、再発防止策が空回りする。

**受け入れ基準**

- Runbook リポジトリ `k1s0/runbooks` に 10 本以上の Runbook が Markdown で格納され、Backstage TechDocs に自動連携
- 各 Runbook は 4 項目（対象事象 / 前提 / 所要時間 / ロールバック）を冒頭に備える
- インシデント対応から 48 時間以内に新規 Runbook または更新 PR が提出され、SRE リードレビューでマージされる
- 全 Runbook は四半期に 1 回の棚卸しで陳腐化チェック、180 日更新がないものは「要検証」ラベルが自動付与
- Runbook リンク切れは weekly CI で検知し、自動 Issue 化

**検証方法**

- 四半期レビューで Runbook 総数・陳腐化率・インシデント起点の追記率を集計し、Backstage Scorecard に表示
- 無作為抽出した Runbook 3 本を当番が実環境で再現し、記載手順で復旧可能か検証

### OPS-OPR-003: 日次運用タスクと自動化率 80%

- 優先度: MUST（手動タスクが 20% を超える規模では、運用工数試算 1.70 FTE を維持できない）
- Phase: Phase 1c で 60%、Phase 2 で 80%、Phase 3 で 90% を目標
- 関連: OPS-OPR-004 / OPS-CID-001

現状、日次で実施すべき「デプロイ前 SLO 確認」「エラーバジェット残チェック」「新規 CVE 確認」「バックアップ成否確認」「証明書残期限確認」の 5 タスクは、担当者が思い出したタイミングで手作業実施しているのみ。タスクの実施漏れは翌日のアラートでしか気付けない。

要件達成後の世界では、SRE 当番が毎朝 9:00 に Slack の `#sre-daily` チャンネルで BOT 通知を受け取る。BOT は前日の SLO 達成率、エラーバジェット残、NVD/GHSA から取得した新規 CVE（CVSS 7.0 以上）、CloudNativePG バックアップの成否、cert-manager 管理証明書で残期限 30 日未満のものを集計してポストする。当番は各項目に Slack 絵文字リアクション（`:eyes:` = 確認済、`:warning:` = 要対応）で応答し、応答が 10:00 までにないものはリマインドされる。

崩れた場合、日次タスクの実施率が 50% を割り、Phase 2 以降でエラーバジェットの枯渇に気付くのが翌日以降となる。CVE 検知の遅れは「CVE Critical 48 時間以内対応」という SEC 要件に直結し、SEC-CVE 違反時の監査リスクに接続する。

**受け入れ基準**

- 日次 5 タスクのうち 4 つ以上（SLO / エラーバジェット / CVE / バックアップ / 証明書）が BOT 通知で自動集計（自動化率 80%）
- Slack `#sre-daily` の BOT 投稿は平日 9:00 ± 5 分で発火
- 当番リアクション応答率が月平均 95% 以上（未応答は翌日朝に SRE リードへエスカレ）
- 手動タスク（残り 20% 以下）は Runbook に明記され、所要時間の合計が 30 分/日を超えない
- 自動化率は月次レビューで可視化し、Phase ごとの目標値（60 → 80 → 90%）を満たす

**検証方法**

- 自動化率は「BOT 投稿項目数 / 日次タスク総数」で算出し、Grafana ダッシュボード `ops-daily-automation` に表示
- 月次でリアクション応答率を集計し、95% 未満の月は SRE リードが原因分析レポートを提出

### OPS-OPR-004: 運用工数の月次可視化とレビュー

- 優先度: SHOULD（稟議で宣言した 1.70 FTE の妥当性を経営層に説明するため。未達でも Phase 承認は可能だが、Phase 2 で経営レビューが通らない恐れ）
- Phase: Phase 1c（初回計測）/ Phase 2 で月次レポート定常化
- 関連: OPS-OPR-003 / OPS-INC-004

現状、運用工数は稟議資料の試算値 1.70 FTE のみが存在し、実測値との乖離を把握する手段がない。「本当に 1.70 FTE で回っているのか」を経営層が問うた時、根拠を示せない。

要件達成後の世界では、tier1 チーム全員が JIRA の `ops-*` プロジェクトにタスクチケットを切り、実工数（人時）を週次で入力する。毎月第 1 営業日に SRE リードが前月の工数を JIRA API から集計し、Backstage の運用ダッシュボード `ops-workload-monthly` に「活動別（CVE / バージョン / 障害 / ドキュメント / 併用固有）× カテゴリ別（k8s / データ / IAM / tier1 自作 / ポータル）」の積み上げグラフとして表示する。月次レビュー（第 2 火曜 14:00 / 60 分）で SRE リードと tier1 リードが稟議試算との差分を確認し、±20% を超えた項目は原因分析を経営レビュー資料に反映する。

崩れた場合、Phase 2 計画時の経営レビューで「運用工数の実績値が存在せず、スケール後の FTE 試算が机上の空論」と指摘され、Phase 2 予算承認が 1 四半期遅延するリスクがある。

**受け入れ基準**

- JIRA `ops-*` プロジェクトの工数入力率（対 稼働時間）が月平均 85% 以上
- 月次レポートが毎月第 2 営業日までに Backstage ダッシュボードに反映される
- 稟議試算（20.4 人月/年 × スケール係数）との差分が ±20% 以内、超過時は原因分析レポート（A4 2 枚以内）を添付
- 2 併用固有工数（ワークフロー 0.5 / ポータル 0.3）は独立項目として集計し、監視 KPI（§2.7.2 の 3 指標）と連動
- 月次レビュー議事録が Confluence に残り、経営報告 deck から参照可能

**検証方法**

- 四半期ごとに JIRA 工数合計と実際の稼働時間（勤怠データ）の整合を監査で確認
- 年次で稟議試算 1.70 FTE と実績 FTE の差分を TCO 資料にフィードバック
