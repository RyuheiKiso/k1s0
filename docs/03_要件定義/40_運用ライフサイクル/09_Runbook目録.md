# 09. Runbook 目録

本書は NFR-A-REC-002「Runbook 充足率」で Phase 1c 必達とされている **15 本の Runbook** を一覧化する目録である。個別 Runbook の実体は [08_Runbookサンプル.md](08_Runbookサンプル.md) および tier1 プラットフォームチームの運用リポジトリで管理するが、本書は「どの 15 本があり、いつまでに整備し、誰が実行権限を持つか」を単一の真実源として管理する。

## 本書の位置付け

Runbook 整備は「何となく資料が増えた」状態では検収の対象にならない。稟議書（[../../01_企画/04_定量試算/02_KPIと承認基準.md](../../01_企画/04_定量試算/02_KPIと承認基準.md)）では Phase 1b 時点で 10 本、Phase 1c 時点で 15 本という明示的な数値コミットを提示しており、Product Council はこの数値をもってパイロット合格判定を下す。本目録が存在しないと、「何をもって 15 本とカウントするか」が恣意的になり、実装は終わったのに検収で落ちる、あるいは検収が通ったのに実運用で穴が発覚する、といった事故が起きる。

本目録は以下の 4 つの役割を担う。第一に 15 本の ID と対象シナリオを確定する。第二に各 Runbook の Phase 整備期限を決める。第三に実行権限（単独実行可能な役割）を定める。第四に Runbook 間のカバレッジ（セキュリティ 5 本以上、DR 1 本以上など NFR-E-SIR-001 で課される条件）を俯瞰する。

## 計数範囲の定義

NFR-A-REC-002 の「15 本」は **インシデント対応および定常運用で単独実行可能な Runbook** に限定する。以下は計数対象外とする。混同すると形式的に本数を積み増せてしまい、REC-002 の実効性が失われる。

- **開発者ガイド系**（Runbook-DEV-ONBOARD-001、Runbook-DEV-TEST-001 等）: 新規参画者向け・開発手元手順であり、インシデント対応ではない。DX 領域で別途管理する。
- **撤退戦略関連**（Runbook-EXIT-001、Runbook-EXIT-DATA-001〜005、Runbook-EXIT-ORG-001）: 撤退判断・データ退避は通常運用外のシナリオ。[05_撤退戦略.md](05_撤退戦略.md) 側で別枠で管理する。
- **テナント特化**（Runbook-OPS-OFF-001 テナントオフボーディング）: テナントライフサイクル管理の一部であり、基盤のインシデント対応とは性質が異なる。[../60_事業契約/03_テナントオンボーディング.md](../60_事業契約/03_テナントオンボーディング.md) 側で管理する。

これら計数対象外の Runbook も作成・保守する必要はあるが、REC-002 の「15 本」には含めない。

## 15 本の目録

以下に 15 本を整備優先順で列挙する。各 Runbook は「対象シナリオ（何が起きたとき）」「Phase 整備期限」「実行権限」「失敗時の影響」「関連要件・サンプル参照」の 5 項目を必ず記述する。書き味が「手順を詰めれば動ける」水準にならない限り、REC-002 の受け入れ基準は満たさない。

### グループ A: 運用ベース Runbook (Phase 1b 必達 3 本)

パイロット稼働の最低条件として、インシデント発生時の入口と共通運用基盤を担う 3 本。これが無いと、後続 12 本がいくら揃っていても初動対応が発散する。

**Runbook-OPS-INC-001: Severity 判定フローチャート**

対象シナリオ: アラート受領直後の Severity 1/2/3 分類判定。対外 SLA 影響、テナント越境、個人情報関与の 3 軸でフローを回し、30 分以内に分類決定するためのチェックリスト。Phase 整備期限は Phase 1a 着手 2 週間以内（他 Runbook の前提となるため最優先）。実行権限は情シス SRE（オンコール当番）単独で完了可能。失敗時の影響は Severity 誤判定により P1 インシデントが P3 扱いになり、対外通知遅延・顧客信頼失墜。関連要件は NFR-A-REC-002 / NFR-E-SIR-001、詳細サンプルは [01_インシデント対応詳細.md](01_インシデント対応詳細.md) §Severity 判定。

**Runbook-OPS-INC-EXT-001: 緊急連絡先・エスカレーション**

対象シナリオ: Severity 1 認定後の社内外連絡（法務部、情報セキュリティ委員会、個人情報保護委員会、OSS コミュニティ、SES ベンダー、テナント代表）。連絡先の陳腐化を防ぐため、Runbook に電話番号・メール・チャット ID を格納せず、対象組織と連絡経路の「種類」のみ定義し、実連絡先は別 Git リポジトリのシークレットストアから参照する。Phase 整備期限は Phase 1a 着手 2 週間以内（OPS-INC-001 と同時）。実行権限は情シス SRE + 情シスマネージャの 2 名体制必須（単独送信禁止）。失敗時の影響は法令報告期限（24h 速報）超過で個人情報保護法違反。

**Runbook-OPS-ERR-001: エラーコード別アラートポリシー**

対象シナリオ: tier1 API が返す標準エラーコード（[../00_要件定義方針/06_エラータクソノミ.md](../00_要件定義方針/06_エラータクソノミ.md)）に対応するアラート閾値・通知先・抑止ルール集中管理。Phase 整備期限は Phase 1a 末（エラータクソノミと同時）。実行権限は tier1 PF チーム（閾値変更は PR レビュー必須）。失敗時の影響はアラート疲れ（過剰通知）またはアラート見逃し（過少通知）による MTTR 悪化。

### グループ B: インフラ障害対応 Runbook (Phase 1b 必達 5 本)

Phase 1b パイロットで稼働する tier1 基盤の主要構成要素（PostgreSQL / Kafka / Workflow / 機材廃棄）のインシデント対応。これらは FMEA ([06_FMEA分析.md](06_FMEA分析.md)) で RPN 100 以上と評定された重大シナリオに対応する。

**Runbook-OPS-DB-001: PostgreSQL Primary ダウン**

対象シナリオ: CloudNativePG Primary が応答しない状態（FMEA RPN 140）。State API / Workflow API / Audit API が巻き込み被害を受ける。Phase 整備期限は Phase 1b 着手時。実行権限は SRE + DB 担当 tier1 エンジニア。失敗時の影響は tier2 書き込み全停止、対外 SLA 違反。サンプル詳細は [08_Runbookサンプル.md §RB-001](08_Runbookサンプル.md)。

**Runbook-OPS-MSG-001: Kafka Broker ダウン (3 台中 1 台)**

対象シナリオ: Strimzi Kafka Broker の 1 台障害。PubSub API の可用性が劣化、DLQ 流入増加。Phase 整備期限は Phase 1b 着手時。実行権限は SRE 単独。失敗時の影響は非同期処理の遅延蓄積、DLQ 枯渇による損失。サンプル詳細は [08_Runbookサンプル.md §RB-002](08_Runbookサンプル.md)。

**Runbook-OPS-MSG-002: DLQ 滞留**

対象シナリオ: DLQ 件数が閾値超過（500 件/30 分）。アプリ側バグまたはスキーマ変更起因の継続失敗。Phase 整備期限は Phase 1b パイロット前。実行権限は SRE（分析） + tier2 開発者（原因アプリのホットフィックス）。失敗時の影響はメッセージロス、監査証跡の欠損。サンプル詳細は [08_Runbookサンプル.md §RB-005](08_Runbookサンプル.md)。

**Runbook-OPS-WF-001: Temporal Workflow Determinism 違反**

対象シナリオ: Temporal Worker が `NonDeterministicWorkflowError` を記録。Workflow 定義の変更をバージョニングなしで適用した場合に多発。Phase 整備期限は Phase 1b パイロット前。実行権限は Workflow 担当 tier1 エンジニア + 該当 tier2 開発者。失敗時の影響は Workflow 途中停止、業務フロー不整合。サンプル詳細は [08_Runbookサンプル.md §RB-004](08_Runbookサンプル.md)。

**Runbook-OPS-DISPOSE-001: 機材・ストレージ廃棄手順**

対象シナリオ: VM / ディスクの計画撤去またはリプレース。暗号消去（cryptographic erasure）と物理破壊の証跡を監査可能な形で保持する。Phase 整備期限は Phase 1b（最初のリプレース発生前）。実行権限は調達 + インフラ運用の 2 名体制必須。失敗時の影響は個人情報残置、PCI DSS / 社内規程違反。関連要件は NFR-F-ECO-002、[../60_事業契約/13_サステナビリティ.md](../60_事業契約/13_サステナビリティ.md) §廃棄手順。

### グループ C: セキュリティ対応 Runbook (Phase 1b 2 本 + Phase 1c 3 本 = 5 本)

NFR-E-SIR-001 が要求する「Runbook 15 本のうち 5 本以上がセキュリティ関連」を満たす 5 本。Phase 1b で即応系 2 本、Phase 1c で深度対応系 3 本を整備する。

**Runbook-SEC-MESH-001: mTLS 証明書期限切れ / SPIFFE ID 不一致**

対象シナリオ: Istio Ambient ztunnel の mTLS 証明書が期限切れ、または SPIFFE ID の発行不整合で tier1 間通信が失敗。Phase 整備期限は Phase 1b パイロット前。実行権限はセキュリティ担当 SRE。失敗時の影響はサービス間通信停止、広域障害。サンプル詳細は [08_Runbookサンプル.md §RB-003](08_Runbookサンプル.md)。

**Runbook-SEC-LEAK-001: 個人情報漏えい検知・初動対応**

対象シナリオ: Audit API での異常アクセス検知、Secrets API での不正取得、外部通報による漏えい疑い。検知から情シスマネージャと監査担当への 24 時間以内通知までの初動手順。Phase 整備期限は Phase 1b パイロット前。実行権限はセキュリティ担当 SRE + 監査担当の 2 名体制必須。失敗時の影響は報告遅延、被害拡大、刑事告発リスク。関連要件は NFR-E-SIR-002、NFR-G-PRV-003。

**Runbook-SEC-LEAK-002: 漏えい等報告義務対応（72h 速報・30 日確報）**

対象シナリオ: LEAK-001 で漏えい確認後の個人情報保護委員会への速報（72 時間以内）・確報（30 日以内）提出プロセス。Audit API + tenant_id 検索で対象データを特定し、報告様式に落とし込む。Phase 整備期限は Phase 1c 必達。実行権限は監査担当 + 法務部の 2 名体制必須。失敗時の影響は個人情報保護法違反、監督官庁からの指導・公表。関連要件は NFR-E-SIR-002、個人情報保護法 2022 年改正。

**Runbook-SEC-TENANT-001: テナント越境検知・封じ込め**

対象シナリオ: tenant_id 境界を超えた読取・書込の検知（NFR-E-AC-003 / NFR-E-NW-002 で定義）。即時の認可ルール tightening、影響テナントへの通知、Audit ログの 90 日以上保全。Phase 整備期限は Phase 1c 必達（Phase 1b は 1 テナント運用のため単体発生リスクは低いが、Phase 1c 以降は複数テナントで顕在化）。実行権限はセキュリティ担当 SRE + 情シスマネージャ。失敗時の影響は重大な信頼破壊、テナント契約違反、賠償。

**Runbook-SEC-AUTH-001: 不正ログイン試行 / 異常 Secret アクセスパターン**

対象シナリオ: Keycloak の連続失敗ログイン、OpenBao Secret の短時間大量読取、通常時間外の管理 API 呼び出し。Severity 2 相当の封じ込め対応（アカウントロック、Secret ローテーション）。Phase 整備期限は Phase 1c 必達。実行権限はセキュリティ担当 SRE 単独で完了可能（Severity 1 未到達のため情シスマネージャ召集不要）。失敗時の影響はアカウント乗っ取りから Severity 1 昇格。

### グループ D: 災害復旧・法的対応 Runbook (Phase 1c 2 本)

Phase 1c の本稼働体制確立で必須となる DR と法的開示対応。Phase 1b では訓練限定、Phase 1c で本運用を開始する。

**Runbook-DR-001: クラスタ全壊 DR (RTO 4 時間、RPO 24 時間)**

対象シナリオ: kubeadm クラスタが全 Node 停止（物理障害、NW 分断、人的誤操作）。etcd スナップショット + MinIO バックアップからの復元、PostgreSQL WAL リプレイ、tier1 API の再導入、tier2 アプリの接続確認まで。Phase 整備期限は Phase 1c 必達。実行権限は SRE + インフラ運用の 2 名体制必須、年 2 回の DR 訓練で単独実行可能性を検証。失敗時の影響は RTO 超過、対外 SLA 違反、Phase 1c ゲート差戻し。関連要件は NFR-A-DR-001、サンプル詳細は [08_Runbookサンプル.md §RB-001](08_Runbookサンプル.md) の DR 拡張版として整備予定。

**Runbook-LEGAL-DISCLOSE-001: 法的開示対応（捜査・差押・訴訟）**

対象シナリオ: 捜査機関からの捜査関係事項照会、裁判所の差押命令、民事訴訟の証拠保全命令。Audit API / Pii API のログ抽出範囲決定、法務部・情シスマネージャの承認、開示ログの保全と訴訟対応用のチェーンオブカストディ記録。Phase 整備期限は Phase 1c 必達。実行権限は法務部 + 監査担当の 2 名体制必須、情シス SRE 単独での開示実行は禁止。失敗時の影響は開示拒否による司法侮辱、あるいは過剰開示によるプライバシー侵害の二次被害。関連要件は NFR-G-PRV-001、[../30_非機能要件/G_データ保護とプライバシー.md](../30_非機能要件/G_データ保護とプライバシー.md) §法的開示。

## Phase 整備期限の俯瞰

上記 15 本を Phase ごとに集約すると以下になる。Phase 1b で 10 本、Phase 1c で 15 本という REC-002 の数値コミットと一致することを確認する。数値が合わない場合は、本目録の記述または REC-002 の本文を是正する。

| Phase | 整備完了 Runbook | 累積本数 |
|---|---|---|
| Phase 1a | OPS-INC-001 / OPS-INC-EXT-001 / OPS-ERR-001 | 3 本 |
| Phase 1b | グループ A 3 本 + グループ B 5 本 + グループ C 前半 2 本（SEC-MESH-001, SEC-LEAK-001） | 10 本 |
| Phase 1c | 上記 + グループ C 後半 3 本（SEC-LEAK-002, SEC-TENANT-001, SEC-AUTH-001）+ グループ D 2 本（DR-001, LEGAL-DISCLOSE-001） | 15 本 |

## Runbook 共通テンプレート

15 本すべてが [08_Runbookサンプル.md](08_Runbookサンプル.md) §Runbook の思想 で定義した 5 段構成（検出 / 初動 15 分以内 / 復旧 60 分以内 / 根本原因調査 / 事後処理）を踏襲する。本目録で各 Runbook の「実行権限」を明記しているのは、この 5 段構成を **どのロールが単独で完了できるか** を事前に固定するためである。実行権限は「単独」「2 名必須」「3 名合議」の 3 水準で分類し、2 名以上が必要な Runbook は訓練時にペアローテーションを必ず行う（属人化排除）。

## 鮮度レビュー運用

NFR-A-REC-002 の受け入れ基準として「鮮度レビューで 90 日以上未更新の Runbook を検出、改訂」が定められている。本目録に対応する具体運用は以下とする。鮮度が崩れた Runbook は形式的に 15 本のカウントに入っていても実運用では信用できず、REC-002 の実質的違反となる。

- 各 Runbook はメタデータとして `last_reviewed_at` / `last_executed_at` を持ち、90 日以上どちらも更新されていない場合は CI で警告を出す
- 四半期レビュー（Product Council 四半期確認と同期）で、未更新 Runbook を検出次第、担当者をアサインし 2 週間以内に改訂
- 実行機会がない Runbook は年 2 回の訓練（NFR-A-REC-001）で意図的に発動し、`last_executed_at` を更新する
- 改訂履歴は Git で管理、**何が変わったか** がコミットメッセージから追跡可能であること（「軽微修正」等の曖昧な記述を禁止）

## 計数対象外 Runbook の管理先

本目録の 15 本には含めないが、プロジェクトとして整備義務のある Runbook の管理先を明記する。計数範囲の定義で排除した Runbook 群は、それぞれ以下の場所で独立に管理する。

- **開発者ガイド系** (DEV-ONBOARD-001, DEV-TEST-001): [../50_開発者体験/02_ゴールデンパス.md](../50_開発者体験/02_ゴールデンパス.md)、[../50_開発者体験/01_テスト戦略.md](../50_開発者体験/01_テスト戦略.md) で引用
- **撤退戦略** (EXIT-001, EXIT-DATA-001〜005, EXIT-ORG-001): [05_撤退戦略.md](05_撤退戦略.md) で整備期限と責任者を管理
- **テナントライフサイクル** (OPS-OFF-001): [../60_事業契約/03_テナントオンボーディング.md](../60_事業契約/03_テナントオンボーディング.md) で管理

これら計数対象外の本数が過度に膨らんでいる場合は「REC-002 本数を増やさずに見えないコストを積んでいる」状態であり、Product Council レビューで統廃合を議題化する。
