# 04. 非推奨と EOL

本書は k1s0 の機能・API・採用 OSS の非推奨化（Deprecation）と廃止（End of Life）のプロセスを定義する。長期運用では新機能の追加と同時に「古いものを整理する」活動が不可欠で、整理しなければ技術的負債が累積して運用コストが指数的に膨らむ。

## 本書の位置付け

プラットフォーム製品は「非破壊的な進化」が信頼の源泉である。一方で完全な互換性を永久維持するとコードが肥大化し、セキュリティパッチの適用や OSS バージョンアップの追従が不可能になる。本書は「変更の破壊的影響を抑えつつ、計画的に整理する」ための手続きを定める。ここを曖昧にすると、tier2/tier3 開発者は「いつ壊されるか分からない」不安を抱き、k1s0 採用に躊躇する。

## OR-EOL-001: バージョニングポリシー

**現状**: API のバージョン管理が暗黙だと、後方互換性の約束が曖昧になる。

**要件達成後**: tier1 公開 API は SemVer 2.0 に準拠する。

- **Major バージョン**: 破壊的変更を含む。移行ガイドと移行期間（12 か月）が必要。
- **Minor バージョン**: 後方互換な機能追加。既存利用に影響なし。
- **Patch バージョン**: 後方互換なバグフィックス・性能改善。

Protobuf / gRPC の場合は以下を追加制約とする。

- Field number の再利用禁止（削除時も番号を予約）
- Field の rename は既存名を `deprecated = true` として残置しつつ新名を追加
- メッセージ構造の破壊的変更は新パッケージ（例: `k1s0.state.v1` → `v2`）で提供

**崩れた時**: 暗黙の破壊的変更で既存 tier2 アプリが稼働中にエラーを起こす。

**受け入れ基準**:
- バージョニングポリシーを ADR で確定、Contributor Guide に記載
- CI で API 互換性検査（`buf breaking`）を PR 時に実行
- メジャーバージョンアップの頻度を年 1 回以下に制限（Phase 3 まで）

**優先度**: MUST

## OR-EOL-002: 非推奨化ライフサイクル

**現状**: 非推奨機能の扱いが曖昧だと、利用者は「まだ使える」と誤解して移行が進まない。

**要件達成後**: 以下のライフサイクルで非推奨化を進める。

1. **Active**: 通常提供。新規利用を推奨。
2. **Deprecated（非推奨）**: 新規利用を非推奨、既存利用は継続可。ドキュメント・API レスポンスヘッダ・ログで警告を発信。最短 6 か月、通常 12 か月の猶予。
3. **Sunset（サンセット）**: 利用統計からほとんど使われていないことを確認、最終告知を発出。3 か月の最終猶予。
4. **Removed（廃止）**: 機能完全除去。該当 API 呼び出しは 410 Gone エラー。

Deprecated 以降は RFC 8594（Sunset HTTP Header）に準拠した応答ヘッダを返し、利用者側で自動検知可能にする。

**崩れた時**: 移行が進まず、Removed 予定日に突然動かなくなる事故が発生する。

**受け入れ基準**:
- 非推奨化カレンダーを Backstage に公開（予告スケジュール）
- Sunset Header を全 tier1 API で実装（Phase 1c）
- 利用統計を Grafana で可視化、Sunset 判断に利用

**優先度**: MUST

## OR-EOL-003: 告知と移行支援

**現状**: 非推奨告知が開発者に届かないと、気付かないうちに廃止日を迎える。

**要件達成後**: 非推奨化時に以下の告知と支援を行う。

- **告知チャネル**: Backstage の `Changes` ページ、社内メーリングリスト、`#k1s0-announce` Slack チャネル、API レスポンスヘッダ
- **移行ガイド**: 非推奨時点で新旧の差分と移行手順を Backstage に公開。コード例を旧新両方併記。
- **移行ツール**: 可能な範囲で自動移行ツール（AST 書き換え、リクエスト変換アダプタ）を提供
- **移行期間中のサポート**: プラットフォーム SRE が移行チケット（OR-SUP-002 の L3）を優先対応
- **使用状況モニタリング**: 残り利用者を Grafana で可視化、移行が滞るテナントには個別リーチアウト

**崩れた時**: tier2 開発者が「いつの間にか壊された」と感じ、k1s0 への信頼低下。

**受け入れ基準**:
- 告知から廃止まで全利用者に到達（Slack リアクション等で確認）
- 移行ガイド完備率 100%
- 廃止時点で残留利用者 5% 以下を目標

**優先度**: MUST

## OR-EOL-004: OSS バージョン EOL 対応

**現状**: 採用 OSS が EOL を迎えるとセキュリティパッチが停止、脆弱性を抱え続けるリスク。

**要件達成後**: 採用 OSS 全件について EOL カレンダーを保守する。

- **監視対象**: Kubernetes、PostgreSQL、CloudNativePG、Strimzi Kafka、Kafka、Istio、Argo CD、Argo Rollouts、Keycloak、OpenBao、Dapr、Temporal、ZEN Engine 等 40 種以上
- **EOL 前 6 か月で移行計画**: 次バージョンへの移行 PoC 実施、検証環境で確認
- **EOL 前 3 か月で本番移行開始**: staging → prod 段階昇格
- **EOL 後は即座にサポート対象外**: 利用禁止、新規プロジェクトでの採用禁止

EOL カレンダーは `docs/02_構想設計/03_技術選定/` と連動し、変更発生時に 30_非機能要件 の関連項目も改訂する。

**崩れた時**: EOL 済み OSS でセキュリティインシデント発生、監査指摘。

**受け入れ基準**:
- EOL カレンダーを Backstage で可視化（Phase 1b）
- EOL 前 6 か月アラートを自動化（Phase 1c）
- EOL 超過 OSS ゼロを年次監査で維持

**優先度**: MUST

## OR-EOL-005: 機能フラグによる段階廃止

**現状**: 機能廃止を一斉に実施すると、利用者が気付かないうちに障害が発生するリスク。

**要件達成後**: 非推奨機能は Feature Flag（flagd）で段階的に無効化できる構成を必須とする。

1. **Deprecated 開始時**: 全利用者で機能有効（Flag on）
2. **Sunset 中盤**: 新規テナントでは無効（Flag off for new）
3. **Sunset 終盤**: 全テナントで無効、特定テナントのみ延長可（テナント単位 Flag）
4. **Removed**: コード削除、Flag も削除

これにより、問題発生時に即座に Flag を戻すことで復旧可能とする。

**崩れた時**: 廃止直後に問題発覚し、緊急復旧デプロイが必要になる。

**受け入れ基準**:
- Feature Flag で制御できる機能廃止を Phase 1c+ の標準プラクティスとする
- 段階廃止テンプレートを Backstage で提供
- 廃止ロールバック手順を Runbook 化

**優先度**: SHOULD

## OR-EOL-006: 互換性マトリクスの維持

**現状**: tier1 API のバージョンと、Go/Rust SDK のバージョン、tier2 クライアントのバージョンの互換組み合わせが不明だと、アップデート時にランタイムエラー。

**要件達成後**: バージョン互換性マトリクスを公開し、メジャーアップデートのたびに更新する。

- **tier1 API バージョン × Go/Rust SDK バージョン**: 対応関係を表形式で公開
- **tier1 API バージョン × Kubernetes バージョン**: 依存するクラスタバージョンを明記
- **tier1 API バージョン × 依存 OSS バージョン**: CloudNativePG、Kafka 等の互換範囲

マトリクスは Git リポジトリで管理、Backstage でレンダリング。リリースノートに最新版を添付。

**崩れた時**: 互換範囲外の組み合わせで本番運用、障害時に原因調査が難航。

**受け入れ基準**:
- 互換性マトリクスを毎リリース更新
- CI でマトリクス範囲外のデプロイを拒否
- 利用者向けチェックリストを Backstage で提供

**優先度**: MUST
