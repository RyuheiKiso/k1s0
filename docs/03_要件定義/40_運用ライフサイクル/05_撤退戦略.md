# 05. 撤退戦略

本書は k1s0 プロジェクトが想定どおり進まなかった場合の撤退判断基準、代替選択肢、データ退避計画を定義する。成功を想定した計画のみで申請を通すのではなく、失敗時の出口を明示することで、情シス部門長・CIO・調達部門の不安を解消する。

## 本書の位置付け

「失敗したらどうするのか」に答えられないプロジェクトは採用検討で止まる。実際、採用側組織の基幹に関わる判断は、失敗した時のダメージがどの程度か、代替にいくらかかるか、データは救出できるか、の 3 点で決まる。本書はこれらに明確な答えを用意し、撤退可能性を担保することで逆説的に完遂可能性を高める。

## OR-EXIT-001: 撤退判断基準

**現状**: 撤退判断が曖昧だと、失敗プロジェクトが何年も延命し、予算と人材を食い潰す。

**要件達成後**: 段階ごとに以下の撤退判断基準を設ける。

- **採用初期 (デモ段階)**: サンプルアプリのデモで `#k1s0-demo` 視聴者から拒絶 80% 以上、または技術検証で致命的な技術的障害（例: Dapr Go ファサードが Go stable SDK と統合不能）。該当時は構想設計から見直し。
- **採用初期 (パイロット段階)**: NFR-D-PLN-001（パイロット合格基準）の未達が 2 サイクル連続。該当時はパイロット拡張せず撤退。
- **採用初期 (運用引継段階)**: 運用引継ぎで SRE チームが手を挙げない、または RTO 4 時間が達成できない状態が 3 か月続く。該当時は採用後の運用拡大に進まず撤退。
- **採用後の運用拡大時**: TCO 予算の 150% 超過、または主要ステークホルダーの離脱（情シスマネージャ・tier2 部門長の両者が反対）。

**崩れた時**: 撤退基準がないため、失敗プロジェクトが継続し、採用側組織の技術ブランドが毀損する。

**受け入れ基準**:

- 撤退基準を 段階ゲート文書に明文化
- 四半期ごとに基準接近度を評価、情シスマネージャに報告
- 撤退判断フローを Runbook 化（Runbook-EXIT-001）

**優先度**: MUST

## OR-EXIT-002: 代替選択肢の事前準備

**現状**: 撤退時に代替を一から検討すると、数か月のブランクで業務が停滞する。

**要件達成後**: 以下の代替選択肢を事前に評価済みとして準備する。

- **Red Hat OpenShift Platform Plus**: 商用 Kubernetes、Istio 代わりに OpenShift Service Mesh、採用検討通過は比較的容易だが年間ライセンス 2,000 万円規模
- **VMware Tanzu Kubernetes Grid**: 既存 VMware 資産と親和性高い、ただし Broadcom 買収後の価格高騰リスク
- **SUSE Rancher**: 中立系 OSS ディストリビューション、商用サポートは SUSE
- **既存 VMware + 現行手作業**: k1s0 で得た知見は破棄、完全に元のやり方に戻る。最悪シナリオ。

各候補について TCO 試算、移行期間、リスクを文書化しておく。代替選択時の比較検討時間を 1 か月以内に収める。

**崩れた時**: 撤退判断から代替導入まで 1 年以上のブランクが発生、業務停滞で実害。

**受け入れ基準**:

- 代替候補 3 種の TCO 試算と移行シナリオを [90_付録/03_撤退代替比較.md](../80_トレーサビリティ/03_構想設計マトリクス.md) に記述（リリース時点前）
- 四半期ごとに商用製品の価格・機能更新を反映
- 段階ゲート時に代替候補との比較を再実施

**優先度**: MUST

## OR-EXIT-003: データ退避計画

**現状**: 撤退時にデータが k1s0 に閉じ込められると、新基盤への移行が不可能になる。

**要件達成後**: 以下のデータについて、撤退時の退避手順を事前定義する。

- **業務データ（PostgreSQL）**: 標準 SQL ダンプまたは pg_dump で平文エクスポート、新基盤の PostgreSQL に pg_restore で復旧可能
- **イベントデータ（Kafka）**: Kafka MirrorMaker 2 で他 Kafka クラスタに複製、または Avro / JSON でファイル化
- **ファイルデータ（MinIO）**: S3 互換 API で新基盤の S3 互換ストレージ（MinIO / 他）に mc mirror で転送
- **認証データ（Keycloak）**: Keycloak Export で JSON/LDIF に出力、新 IdP にインポート
- **ワークフロー状態（Temporal）**: Temporal の Export 機能または History の Archive で退避、ただし実行中ワークフローは停止が前提
- **設定（Git リポジトリ）**: k1s0 固有の CRD 依存を剥がせばそのまま他 Kubernetes で利用可能

退避後の新基盤起動までの想定時間を各データ種別で明示する（例: DB 100GB で 4 時間）。

**崩れた時**: データ退避に数週間かかり、業務停止期間が許容を超える。

**受け入れ基準**:

- データ退避手順を Runbook-EXIT-DATA-001〜005 で整備（リリース時点）
- 年 1 回の退避リハーサルを staging 環境で実施
- 退避所要時間を実測値で更新

**優先度**: MUST

## OR-EXIT-004: 技術的依存の棚卸し

**現状**: k1s0 固有機能への依存が蓄積すると、退避コストが指数的に膨らむ。

**要件達成後**: 以下の「ロックイン要因」を定期的に棚卸しし、最小化する。

- **tier1 カスタム CRD**: k1s0 固有の CRD をできる限り使わず、標準 Kubernetes オブジェクトで代替可能な設計を優先
- **Dapr Building Block 拡張**: Dapr OSS の標準仕様から逸脱する拡張は ADR で justify、撤退時の代替実装難易度をメモ
- **ZEN Engine JDM 依存**: ルールエンジンの表現が ZEN 固有である場合、DMN（OMG 標準）への変換可能性を保持
- **独自 API**: tier1 API のうち、dapr 標準や Kubernetes 標準で代替不可能なものはリストアップ

棚卸し結果は撤退時の「技術的退避コスト」として保守する。

**崩れた時**: ロックインが進みすぎて実質的に撤退不能になり、サプライヤ交渉力を失う。

**受け入れ基準**:

- 年次でロックイン棚卸しレビュー（運用蓄積後）
- ロックイン要因 10 件以下を維持目標
- 新規独自拡張時は ADR で「なぜ標準で不足か」を必須記述

**優先度**: SHOULD

## OR-EXIT-005: 撤退時の組織・契約対応

**現状**: 撤退判断が技術的に正しくても、関係者への説明・契約解除・人員再配置が混乱すると、実行が数か月遅れる。

**要件達成後**: 撤退決定から 30 営業日以内に以下を完了する計画を保持する。

- **ステークホルダー通知**: CIO・情シスマネージャ・tier2/tier3 部門長・調達部門・法務部への正式通知（Day 1〜3）
- **契約解除対応**: 関連ベンダー契約、サポート契約、OSS コンサル契約の解除または代替契約締結（Day 5〜20）
- **人員再配置**: k1s0 専任エンジニアの代替プロジェクトへの移籍計画（Day 10〜30）
- **利用者への移行支援**: tier2/tier3 で稼働中のアプリの代替基盤移行支援窓口を設置（Day 1〜）
- **監査対応**: 撤退理由・経緯・費用の監査報告、今後の教訓の文書化（Day 60 以内）

**崩れた時**: 撤退移行期間に業務アプリがサポート切れ状態になり、さらなる信頼毀損。

**受け入れ基準**:

- 撤退時組織対応フローを Runbook-EXIT-ORG-001 で整備
- 関係者連絡先リストを四半期ごとに更新
- 撤退リハーサルは実施不要だが、CIO と情シスマネージャと撤退手順を年 1 回 walk-through

**優先度**: MUST

## OR-EXIT-006: 撤退時の学習資産保全

**現状**: 失敗プロジェクトは「なかったこと」にされがちだが、学習資産を保全しないと次の挑戦も同じ失敗を繰り返す。

**要件達成後**: 撤退時に以下の学習資産を保全する。

- **技術的知見**: ADR・Runbook・構想設計文書を社内 Wiki に保全、削除禁止
- **失敗理由の文書化**: Blameless 振り返りで 3 階層以上の根本原因分析、社内公開
- **定量データ**: 稼働期間中のパフォーマンス・可用性・コストの実績値を保全、次プロジェクトの見積根拠に使える形式で整理
- **人的知見**: プロジェクトメンバーからインタビュー（Exit Interview）を実施、組織学習として記録

**崩れた時**: 組織の学習サイクルが止まり、次期プラットフォーム検討が一から始まる。

**受け入れ基準**:

- 学習資産保全の責任者を撤退開始時に指名
- 撤退完了から 90 日以内に学習レポート公開
- レポート公開先を事前に合意（社内 Confluence 等）

**優先度**: SHOULD
