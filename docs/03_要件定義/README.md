# k1s0 要件定義ドキュメント

本ディレクトリは、k1s0 プラットフォームの**要件**を 7 カテゴリ構造で採番・管理する層である。企画 ([`../01_企画/`](../01_企画/)) で合意した方針と構想設計 ([`../02_構想設計/`](../02_構想設計/)) で裏付けた技術判断を、**実装・検証が可能な最小単位の要件** に落とし込み、後続の概要設計 ([`../04_概要設計/`](../04_概要設計/)) に引き渡すことを目的とする。

稟議資料でも設計書でもなく、「**この要件を満たしているか / 満たしていないか** が二値で判定できる記述」を志向する。受け入れ基準と優先度の明示、根拠の同居、Phase ゲートとの紐付けを通じて、要件自体が検収とリリース判断の根拠として機能することを狙う。

---

## 本ディレクトリの位置づけ

企画と構想設計は「**何を作り、なぜ作るか**」の合意形成層であり、文書の説得対象は決裁者と開発主導エンジニアである。一方で要件定義は「**何を満たせば完成と呼べるか**」の規格化層であり、読者は実装者 / テスト設計者 / 監査担当 / 受け入れ承認者である。両者の役割が異なるため、記述のトーンも次の点で切り替える。

- **散文は要件の動機と影響に限定し、機能の詳細は ID 単位の箇条書きに寄せる**。企画では物語、構想設計では判断、要件定義では検収。散文量は要件 1 件あたり 1〜2 段落に抑え、受け入れ基準を列挙可能な形で続ける。
- **表は章末のサマリ（ID 一覧 / Phase 達成度 / 優先度分布）に限定する**。要件本体を表セルに圧縮する書き方は禁止する。これは `CLAUDE.md` の「要件ドキュメントの記述」規約に準拠する。
- **ID を振ったうえで相互参照する**。採番規則は後述する。要件間の依存や排他は ID を介して明示する。

## 7 カテゴリ構造

要件は以下の 7 カテゴリに分類する。カテゴリ番号は **2 桁刻み** で、後続拡張の余地を残している（例: 25_ や 35_ など将来カテゴリ挿入用）。各カテゴリは独立した README を持ち、配下のサブ項目の採番方針・散文の書き方・図表要件を定義する。

| パス | 位置づけ | 主要サブ項目 |
|---|---|---|
| [`00_共通/`](./00_共通/) | 全カテゴリで共有する前提・制約・用語 | glossary / stakeholder / assumption / constraint / risk / error_taxonomy / 横断要件 |
| [`10_アーキテクチャ/`](./10_アーキテクチャ/) | 構造要件。tier 境界・通信契約・内部基盤 | infra / tier1 / tier2 / tier3 / integration / api_sdk_contract / rule_engine / eventing |
| [`20_品質特性/`](./20_品質特性/) | 非機能要件。SLO / SLA の数値化 | performance / availability / observability / sla_slo / tenant_isolation / DR_BCP / accessibility |
| [`30_セキュリティ_データ/`](./30_セキュリティ_データ/) | 安全性・法令適合・データ管理 | security / IAM / kms_crypto / audit / data / privacy / data_residency / backup_restore / artifact_integrity / compliance |
| [`40_運用ライフサイクル/`](./40_運用ライフサイクル/) | 運用・CI/CD・リリース・撤退 | operation / CICD / release / environment_config / migration / incident_response / support / deprecation_EOL / exit_strategy |
| [`50_開発者体験/`](./50_開発者体験/) | 内外の開発者の生産性 | test / devex / feature_management |
| [`60_事業_契約/`](./60_事業_契約/) | 事業性・契約・ガバナンス | UX / i18n_l10n / tenant_onboarding / training / cost / licensing / billing_metering / legal_contract / vendor_supplychain / governance / ai_governance / analytics / sustainability |

2 桁目の番号はカテゴリ内での**読み順** を意味する。例えば 10_アーキテクチャ 配下の `01_tier1.md` は `00_infra.md` を前提とした要件を記載する。カテゴリ内で依存ループが発生する場合は、依存先をサブ項目として独立させ、参照を一方向に保つ。

## 要件 ID 採番規則

要件 ID は `<カテゴリ記号>-<サブ項目記号>-<連番>` の形式で採番する。1 ID = 1 要件とし、複数の受け入れ基準を束ねる場合でも、検証単位が分離できるものは別 ID に分割する。

### カテゴリ記号

| カテゴリ | 記号 |
|---|---|
| 00_共通 | `COM` (common) |
| 10_アーキテクチャ | `ARC` (architecture) |
| 20_品質特性 | `QUA` (quality) |
| 30_セキュリティ_データ | `SEC` (security / data) |
| 40_運用ライフサイクル | `OPS` (operations) |
| 50_開発者体験 | `DEV` (developer experience) |
| 60_事業_契約 | `BIZ` (business) |

### サブ項目記号

各カテゴリ README に一覧を掲載する。例: `ARC-T1-xxx` は tier1、`SEC-IAM-xxx` は IAM を示す。サブ項目記号は原則 2〜3 文字の英大文字で統一する。

### 連番

連番は **001 から 3 桁ゼロ埋め** で採番する。欠番は許容し、後から要件を追加する際に隙間に入れてもよい（ただし要件の意味的なグループを壊さないこと）。廃止された要件は削除せず、ステータスを `Deprecated` に変更したうえで ID を保持する。

### 採番例

- `COM-GLO-001` — 用語「tier1」の定義
- `ARC-T1-012` — tier1 内部通信は Protobuf gRPC 必須
- `QUA-PRF-003` — tier1 API p99 レイテンシ 500ms 以下
- `SEC-AUD-005` — 監査ログのハッシュチェーンによる改ざん検知
- `OPS-CICD-001` — 全 PR は GHA パイプラインでライセンス検査を通過すること

## 要件の記述フォーマット

個別要件ファイルは [`../00_format/要件定義テンプレ.md`](../00_format/要件定義テンプレ.md) のテンプレートに従う。各要件は最低限、以下 7 項目を備える。

1. **要件 ID** — 前節の採番規則に従う
2. **要件タイトル** — 30 文字以内の主語付き命題
3. **優先度** — MUST / SHOULD / MAY の 3 段階。判定理由を同居させる
4. **Phase** — Phase 1a / 1b / 1c / 2 / 3 のいずれか。複数 Phase にまたがる場合は段階を分割
5. **現状・要件達成後・崩れた時** — 散文 1〜3 段落で記述。表セルに詰めるのは禁止
6. **受け入れ基準** — 箇条書きで検証可能な条件を列挙
7. **関連** — 依存する他要件 ID / 参照 ADR / 企画・構想設計の該当章

### 優先度判定の原則

優先度は「**未達時の稟議 / リリース / 監査 への影響**」で判定する。MUST は未達で Phase 承認不可、SHOULD は未達でも承認可能だが次 Phase で解消必須、MAY は将来課題として保留可能を意味する。判定理由の同居は `CLAUDE.md` 「優先度セルは判定理由を同居させること」に従う。

### Phase との紐付け

Phase 記号は企画 [`../01_企画/03_ロードマップと体制/00_フェーズ計画.md`](../01_企画/03_ロードマップと体制/00_フェーズ計画.md) に準拠する。要件は原則として「達成が要求される最も早い Phase」に紐付ける。段階的達成が前提の要件は、サブ要件として分割する。

## 隣接ドキュメントとの関係

| パス | 位置づけ |
|---|---|
| [`../01_企画/`](../01_企画/) | 稟議向け方針。本要件群の動機と前提 |
| [`../02_構想設計/`](../02_構想設計/) | 技術裏付け。本要件群の実現手段の妥当性根拠 |
| [`../02_構想設計/adr/`](../02_構想設計/adr/) | 確定技術決定。要件の実装制約として参照 |
| [`../04_概要設計/`](../04_概要設計/) | 要件を実装単位に落とす層（後続フェーズ） |

要件定義から上流に変更が必要になった場合（例: 新しい制約が発覚）、まず ADR を起票して判断を記録し、その後に該当要件を更新する。要件の変更履歴は git 履歴で追跡する。

## 記載方針

- 図表は drawio で作成し、同階層の `img/` に `.drawio` と `.svg` を両方格納する。md からは svg を埋め込む。
- drawio は [`../00_format/drawio_layer_convention.md`](../00_format/drawio_layer_convention.md) の 4 レイヤ色分け規約に従う。
- 日付は `YYYY-MM-DD` 形式で記載する。
- 1 ファイル 300 行以内。超過する場合はサブ項目を分割する。
- 要件本体を表セルに詰めない。表は ID 一覧 / Phase 達成度 / 優先度分布などの**インデックス用途に限定**する。
