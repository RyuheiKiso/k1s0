# H. アーティファクト完全性とコンプライアンス

本書は k1s0 のソフトウェアアーティファクト（コンテナイメージ、Helm chart、SBOM、設定ファイル等）の完全性保証と、適用される法令・業界標準・監査要件の遵守を定義する。サプライチェーン攻撃が近年増加する中、アーティファクトの真正性と改ざん検出は必須要件となった。

## 本章の位置付け

アーティファクト完全性は「作ったものが、そのまま本番に届く」を保証する要件群である。[../60_事業契約/09_サプライチェーン管理.md](../60_事業契約/09_サプライチェーン管理.md) が事業契約の観点でベンダー評価・調達経路を定めるのに対し、本章は技術的な完全性保証（署名・検証・attestation）に特化する。コンプライアンスは、k1s0 が外部規制・業界標準とどう整合するかを定める。

## H1. アーティファクト完全性

### NFR-H-INT-001: コンテナイメージ署名

**現状**: 署名のないコンテナイメージは、レジストリ侵害時の差し替えを検出できない。

**要件達成後**: 以下で全コンテナイメージに署名する（BC-SC-003 と整合）。

- **署名ツール**: Sigstore cosign
- **署名方式**: Keyless（OIDC 連携）、秘密鍵管理不要
- **記録**: Rekor 透明性ログに公開
- **検証**: Argo CD デプロイ時および Kyverno ポリシーで署名検証を強制
- **未署名拒否**: 署名のないイメージはクラスタで起動禁止

**崩れた時**: レジストリ侵害で悪意ある差し替えが検出できず、本番侵入。

**受け入れ基準**:
- Phase 1b で全 k1s0 イメージ署名
- Phase 1c で Kyverno による検証強制
- Rekor 記録率 100%

**優先度**: MUST

### NFR-H-INT-002: SBOM 添付

**現状**: SBOM がないと、脆弱性発見時に影響範囲の特定が不可能。

**要件達成後**: 全アーティファクトに SBOM を添付する（BC-LIC-002 と整合）。

- **フォーマット**: CycloneDX（CNCF 標準）
- **生成**: CI/CD のビルドステップで自動生成（Syft 等）
- **添付**: OCI Artifact としてイメージに添付
- **署名**: SBOM も Sigstore で署名
- **検索**: Backstage でイメージ別 SBOM を閲覧可能、依存パッケージの検索可

**崩れた時**: Log4j のような脆弱性発生時、どのイメージが該当するかを手動調査で数日。

**受け入れ基準**:
- Phase 1a で SBOM 自動生成
- 全イメージに SBOM が添付されていることを CI で検証
- SBOM 検索機能を Backstage で提供

**優先度**: MUST

### NFR-H-INT-003: ビルド Provenance（SLSA）

**現状**: ビルド環境の真正性が証明できないと、ビルドサーバ侵害を検出できない。

**要件達成後**: SLSA Level 3 相当の provenance を生成する（BC-SC-001 と整合）。

- **内容**: ビルドステップ、入力（ソース、依存）、ビルド環境、ビルドツールのバージョン
- **形式**: in-toto Statement、SLSA Provenance v1
- **署名**: Sigstore で署名
- **検証**: デプロイ時に provenance 検証、未検証は拒否

**崩れた時**: 侵害されたビルドサーバから悪意あるアーティファクトが本番に到達、検出不可。

**受け入れ基準**:
- Phase 1b で SLSA Level 1/2 達成
- Phase 2 で Level 3 目標
- Provenance 検証を自動化

**優先度**: MUST

### NFR-H-INT-004: 再現可能ビルド

**現状**: ビルドが毎回異なるハッシュを生成すると、同じソースから同じアーティファクトを再生成して検証できない。

**要件達成後**: 以下を目指す。

- **決定論的ビルド**: タイムスタンプ・ランダム値を固定化（`SOURCE_DATE_EPOCH`、`GOFLAGS` 等）
- **依存固定**: go.sum、Cargo.lock、package-lock.json でバージョン固定
- **ビルドイメージ固定**: ベースイメージも digest で固定
- **検証**: 同じソースからのビルドで同じハッシュが出ることを CI で確認

Phase 2 以降での完成を目指す（完全な再現可能性は難易度が高い）。

**崩れた時**: ビルドサーバ侵害の検出が困難。

**受け入れ基準**:
- Phase 2 で主要イメージの再現性検証
- Phase 3 で全イメージ対応
- 差分発生時は原因調査プロセス

**優先度**: SHOULD（Phase 2+）

## H2. 鍵と署名の管理

### NFR-H-KEY-001: 鍵ライフサイクル管理

**現状**: 署名鍵のライフサイクル管理がないと、漏洩時の対応や定期交換ができない。

**要件達成後**: 以下で鍵を管理する。

- **発行**: OpenBao で自動発行、有効期限付き
- **ローテーション**: 90 日ごとに自動ローテーション
- **取消**: 漏洩検知時の即時取消プロセス、Rekor の無効化記録
- **監査**: 鍵発行・利用・取消の全ログを監査保全
- **Keyless 署名**: 秘密鍵の長期保持を避け、OIDC 連携で都度発行

**崩れた時**: 鍵漏洩で署名偽造、完全性保証が崩れる。

**受け入れ基準**:
- Phase 1b で鍵ライフサイクル Runbook
- Phase 1c でローテーション自動化
- 年次で鍵管理の監査

**優先度**: MUST

## H3. コンプライアンス

### NFR-H-COMP-001: 個人情報保護法遵守

**現状**: 2022 改正個人情報保護法の要求事項を満たさないと、法令違反。

**要件達成後**: 以下を遵守する（G 章、BC-LGL-004 と統合）。

- **適切な取扱義務**: 利用目的明示、目的達成後消去、第三者提供制限
- **漏えい通知**: 3 日 / 30 日の個人情報保護委員会通知、本人通知
- **越境移転**: 本人同意または十分性認定国制限
- **開示・訂正・削除請求**: 30 日以内対応
- **要配慮個人情報**: 取得時の明示的同意

**崩れた時**: 改善命令・罰金・信用毀損。

**受け入れ基準**:
- Phase 1a で遵守計画確定
- 年次で法令変更対応
- 法務部の定期レビュー

**優先度**: MUST

### NFR-H-COMP-002: J-SOX 対応

**現状**: k1s0 が財務報告関連業務を支える場合、J-SOX（金融商品取引法に基づく内部統制報告制度）対応が必要。

**要件達成後**: 以下で J-SOX 要件を満たす。

- **監査ログ**: 財務関連業務のアクセス・変更を 7 年保全
- **変更管理**: 本番変更の承認フロー、ITGC 整備
- **アクセス管理**: 職務分離、特権アカウント管理
- **データ完整性**: バックアップ、整合性チェック
- **評価**: 内部統制評価の自己評価、外部監査対応

対象業務範囲は JTC 経理・内部統制部門と協議。

**崩れた時**: 内部統制評価で「重要な欠陥」指摘、有価証券報告書の訂正。

**受け入れ基準**:
- Phase 2 以降で適用範囲を確定
- 内部監査部門との連携
- 年次の外部監査対応

**優先度**: SHOULD（該当業務）

### NFR-H-COMP-003: ISO 27001 / ISO 27017 整合

**現状**: 情報セキュリティ管理の国際標準 ISO 27001 との整合で、対外的信頼性確保。

**要件達成後**: ISO 27001（情報セキュリティ管理システム）および ISO 27017（クラウドセキュリティ）の要件と整合する管理策を実装する。

- **組織的管理策**: BC-GOV（ガバナンス）、BC-TRN（教育）
- **人的管理策**: 採用時のバックグラウンドチェック、退職時の資産返却
- **物理的管理策**: DC 物理セキュリティ（JTC 既存 DC に準ずる）
- **技術的管理策**: E 章、G 章、H 章の全要件

JTC 全社で ISO 27001 認証取得済みの場合、k1s0 をスコープに含める対応を実施（Phase 2+）。

**崩れた時**: ISO 認証維持の障害、対外的信頼性低下。

**受け入れ基準**:
- Phase 2 で ISO 27001 スコープ調整
- ギャップ分析と改善計画
- 年次内部監査

**優先度**: SHOULD（Phase 2+）

### NFR-H-COMP-004: 業界ガイドライン整合

**現状**: JTC の業種に応じた業界ガイドライン（金融: FISC、医療: HIPAA、製造: IEC 62443）があれば遵守必要。

**要件達成後**: JTC の業種に応じたガイドラインとの整合を確認する。

- **FISC 安全対策基準**: 金融業であれば遵守
- **電子帳簿保存法**: 電子取引データの 7 年保存
- **IEC 62443**: 産業用制御システム連携があれば考慮
- **業界特有**: その他業界ガイドライン

JTC 業種が定まり次第、個別に適用する（本要件はプレースホルダ）。

**崩れた時**: 業界規制違反で事業影響。

**受け入れ基準**:
- Phase 1a で適用ガイドラインを法務部と確定
- 年次で遵守状況レビュー

**優先度**: MUST（該当業種）

## H4. 監査対応

### NFR-H-AUD-001: 監査ログの完整性

**現状**: 監査ログが改ざん可能だと、監査の信頼性が崩れる。

**要件達成後**: 以下で監査ログの完整性を保証する。

- **WORM（Write Once Read Many）**: 書き込み後の変更禁止、MinIO Object Lock 活用
- **ハッシュチェーン**: 前イベントのハッシュを次イベントに含めて連結、改ざん検出
- **署名**: 日次でハッシュを署名、タイムスタンプを記録（RFC 3161）
- **アクセス制限**: 監査ログの閲覧は監査官・セキュリティ責任者のみ
- **保管期間**: 7 年（J-SOX、個人情報保護法）

**崩れた時**: 監査証跡の信頼性が崩れ、監査合格が困難に。

**受け入れ基準**:
- Phase 1c で WORM 実装
- Phase 2 でハッシュチェーン実装
- 四半期で完整性検証を実行

**優先度**: MUST

### NFR-H-AUD-002: 外部監査対応

**現状**: 外部監査（会計監査、セキュリティ監査、ISO 監査）対応が場当たりだと、工数過大。

**要件達成後**: 以下で対応する。

- **監査パッケージ**: 四半期ごとの定型監査パッケージを自動生成（変更ログ、アクセスログ、インシデントレポート、ADR、RACI）
- **監査窓口**: プラットフォーム SRE + 法務 + 情報セキュリティ委員会で対応チーム編成
- **監査ツール**: 監査人向けに読み取り専用アクセス権限を提供（必要に応じて）
- **指摘対応**: 指摘事項を課題管理システムで追跡、期限内是正

**崩れた時**: 監査対応で数週間の工数、他業務に影響。

**受け入れ基準**:
- Phase 1c で監査パッケージ自動生成
- 年次外部監査で指摘事項ゼロを目標
- 是正率 100%

**優先度**: MUST

## H5. 透明性と情報公開

### NFR-H-TRN-001: プラットフォーム状態の透明性

**現状**: プラットフォーム状態がブラックボックスだと、テナント側の不信感が生じる。

**要件達成後**: 以下を公開する。

- **Status Page**: 主要サービスの稼働状況をリアルタイム公開
- **インシデント履歴**: 過去 Severity 1/2 事故の概要（機密情報除外）
- **メンテナンス予定**: 計画メンテナンスを事前公開
- **リリースノート**: 新機能・変更内容を公開
- **セキュリティ対応**: 重大脆弱性への対応状況（CVE 別）

Backstage に集約、社内全員がアクセス可。

**崩れた時**: 障害時の情報不足でテナントからの問い合わせ殺到、運用チーム疲弊。

**受け入れ基準**:
- Phase 1c で Status Page 公開
- リリースノートの定期発行
- 月次でアクセス数と満足度を計測

**優先度**: SHOULD
