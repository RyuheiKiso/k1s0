# D. 移行性

本書は IPA 非機能要求グレード 2018 の「D. 移行性」に準拠し、k1s0 を導入する際の移行時期、移行方式、移行対象、移行計画の要件を定義する。既存 .NET Framework 資産との共存が稟議の差別化ポイントであり、段階的・非破壊的な移行設計が Phase 1b パイロット合格の前提となる。

## 本章の位置付け

移行性は「レガシー資産の全量書き換え」ではなく「段階的共存と緩やかな更新」を設計する。k1s0 は既存業務システムをリプレースする製品ではなく、新規業務機能を k1s0 で作り、既存機能は共存させる方針。これにより初期導入コストと業務リスクを最小化する。

## D1. 移行時期

### NFR-D-TIM-001: フェーズ別移行スケジュール

**現状**: 既存業務システムの全量書き換えは数年単位で不可能、段階移行の時期設計が曖昧だとリソース配分が読めない。

**要件達成後**: 以下の段階で移行を進める。

- **Phase 1a（MVP-0）**: tier2 サンプルアプリ 1 本を k1s0 ネイティブで構築、業務影響なしのデモ
- **Phase 1b（MVP-1a）**: 新規業務アプリ 1 本を k1s0 で本稼働、既存業務システムは触らない
- **Phase 1c（MVP-1b）**: パイロット 1 本の運用体制確立、k1s0 継続判断
- **Phase 2**: 新規業務アプリ 2〜5 本を k1s0 で構築、既存との共存検証
- **Phase 3**: tier2/tier3 部門横展開、既存からの段階リプレース開始（1 業務アプリ単位）
- **Phase 4 以降**: 既存の順次リプレース、JDM エディタによる業務運用移行

**崩れた時**: 移行時期が曖昧で予算配分が立たず、中途半端な状態で k1s0 と既存の二重保守が常態化する。

**受け入れ基準**:
- 各 Phase のゲート条件と移行判断基準を明文化
- 企画書ロードマップ（[../../01_企画/03_ロードマップと体制/](../../01_企画/03_ロードマップと体制/)）と整合

**優先度**: MUST

### NFR-D-TIM-002: 業務時間影響の最小化

**現状**: 移行作業が業務時間内に実施され、利用者体感の悪化を招くことがある。

**要件達成後**: 移行に伴うデータ移行・DNS 切替・ルーティング変更等の業務影響作業は、土日夜間のメンテナンスウィンドウ（土曜 22:00〜日曜 6:00）で実施する。平日業務時間内の移行は原則禁止。

**崩れた時**: 業務時間中の影響で苦情が増え、k1s0 展開のスピードが落ちる。

**受け入れ基準**:
- 移行作業の計画時期を 1 か月前に通知
- 業務時間内の緊急移行は情シスマネージャ承認を必須

**優先度**: MUST

## D2. 移行方式

### NFR-D-MTH-001: 既存 .NET Framework との共存方式

**現状**: 既存 .NET Framework 4.x アプリを k1s0 の基盤に繋げる方式が明確でないと、Phase 1b パイロット合格判定が曖昧になる。

**要件達成後**: 以下 2 方式を正式採用する。

- **サイドカー方式**（FR-EXT-DOTNET-001）: 既存アプリの Pod にサイドカーコンテナを同居、HTTP/1.1 で k1s0 API にアクセス
- **API Gateway 方式**（FR-EXT-DOTNET-002）: VM 直接稼働の既存アプリは Envoy Gateway 経由で k1s0 API にアクセス

**崩れた時**: レガシー資産の扱いが個別判断となり、部門ごとに実装方式がバラつく。

**受け入れ基準**:
- 両方式のテンプレートを Backstage で提供
- パイロット業務で両方式を検証（Phase 1b）

**優先度**: MUST

### NFR-D-MTH-002: 段階リリース（Canary / Blue-Green）

**現状**: 新バージョンの本番適用で全ユーザ一斉影響が発生し、事故時のインパクトが大きい。

**要件達成後**: Argo Rollouts で Canary（10% → 50% → 100%）または Blue-Green デプロイを標準化する。異常検知時（NFR-B-PERF / NFR-A-CONT のメトリクス悪化）は自動ロールバック。

**崩れた時**: 一斉リリース事故で影響ユーザ数が数千人規模になり、情シスマネージャへの説明責任が発生する。

**受け入れ基準**:
- Argo Rollouts テンプレートで Canary / Blue-Green を選択可能
- Phase 1b で基本 Canary、Phase 1c で自動ロールバック実装

**優先度**: MUST

### NFR-D-MTH-003: データ移行ツール

**現状**: 既存 DB から k1s0 の PostgreSQL へのデータ移行は手作業で、手戻りが多い。

**要件達成後**: データ移行のパターン別にツールと Runbook を整備する。

- SQL Server → PostgreSQL: pgloader + カスタム変換スクリプト
- ファイル → MinIO: MinIO Client（mc）でバッチアップロード
- 既存 Kafka / IBM MQ → k1s0 Kafka: MirrorMaker または自作 Producer / Consumer

初回は個別対応、2 回目以降はテンプレート化。

**崩れた時**: データ移行の都度ゼロから設計となり、工数が膨れる。

**受け入れ基準**:
- 主要パターン 3 種の Runbook 整備
- Phase 2 で初回実施、Phase 3 でテンプレート化
- 優先度 SHOULD（Phase 1b スコープ外、Phase 2+ で整備）

**優先度**: SHOULD

## D3. 移行対象

### NFR-D-OBJ-001: 既存監査ログの集約

**現状**: 既存業務システムの監査ログが個別フォーマットで散在、k1s0 Audit API との連携が無い。

**要件達成後**: 既存業務システムから k1s0 Audit API へ監査イベントを集約する。REST エンドポイント（`POST /audit/events`）を tier1 が提供し、既存アプリは単発 HTTP 呼び出しで記録する。Phase 2 以降で段階移行。

**崩れた時**: 監査ログが散在したまま、J-SOX 対応などで統合クエリができない。

**受け入れ基準**:
- 既存アプリ向け REST エンドポイント（Phase 2）
- 集約後の統合監査ビューを Backstage で提供

**優先度**: SHOULD

### NFR-D-OBJ-002: 既存業務アプリのメトリクス集約

**現状**: 既存業務システムのメトリクスは Zabbix / SCOM に散在、k1s0 Grafana との統合が無い。

**要件達成後**: 既存監視基盤から Prometheus Exporter 経由でメトリクスを取得、Grafana で統合表示する。Phase 2 以降で段階実装。

**崩れた時**: 運用チームが複数ダッシュボードを往復する負荷が継続する。

**受け入れ基準**:
- Zabbix Exporter / SCOM Exporter 等の採用・自作
- Grafana で統合ダッシュボード
- 優先度 COULD（Phase 2+）

**優先度**: COULD

### NFR-D-OBJ-003: 既存認証基盤の統合

**現状**: 既存 AD / LDAP / SAML IdP との統合が無いと、k1s0 新規ログインが負担。

**要件達成後**: Keycloak Federation で既存認証基盤を上流に置く（FR-EXT-IDP-001、002）。Phase 1a で AD / LDAP、Phase 1b で SAML IdP。

**崩れた時**: k1s0 の展開で ID 重複管理が発生、運用負荷増大。

**受け入れ基準**:
- Phase 1a で AD / LDAP Federation 稼働
- Phase 1b で SAML IdP 採用判定

**優先度**: MUST

## D4. 移行計画

### NFR-D-PLN-001: パイロット合格基準

**現状**: パイロット実施の成功条件が曖昧だと、Phase 1c → 2 の継続判断が紛糾する。

**要件達成後**: Phase 1b パイロット合格基準を以下に定義する。

- tier1 公開 API の 11 機能が利用可能（Phase 1b スコープ）
- tier1 API p99 < 500ms を 1 週間連続維持（NFR-B-PERF-001）
- Severity 1 インシデント 0 件（または発生時の復旧が RTO 4 時間以内）
- Runbook 10 本以上整備（NFR-A-REC-002）
- .NET Framework 共存検証（サイドカー方式 or API Gateway 方式のどちらか実績）
- SSO ログイン成功率 99% 以上

**崩れた時**: Phase 1c 継続判断で根拠が曖昧、稟議延長が発生する。

**受け入れ基準**:
- 全 6 基準を達成
- 未達項目は改訂プロトコルで是正計画を作成

**優先度**: MUST

### NFR-D-PLN-002: リハーサル実施

**現状**: 初回本番リリースのリハーサル不十分で事故が起きる。

**要件達成後**: Phase 1b 本稼働前に staging 環境で 2 週間の定常運用リハーサルを実施、RTO / RPO / Runbook 実行を検証する。

**崩れた時**: 本番運用初日に想定外のトラブルが連発し、信頼失墜する。

**受け入れ基準**:
- staging で 2 週間の定常負荷試験
- 全 Severity 別訓練を 1 回ずつ実施

**優先度**: MUST

### NFR-D-PLN-003: 撤退計画

**現状**: 基盤変更の撤退判断が曖昧だと、失敗プロジェクトが撤退できず予算を消耗し続ける。

**要件達成後**: Phase ごとに撤退基準と代替シナリオ（OpenShift 等への移行）を明文化する。Phase 1b パイロット合格基準未達が 2 サイクル続いた場合、Phase 1c 着手せずに撤退判断。撤退時のデータ移行パスも事前定義。

**崩れた時**: 失敗基盤が運用継続され、予算・人材が消耗する。

**受け入れ基準**:
- 撤退基準を ADR または Phase ゲート文書に記述
- 撤退時の代替候補を 90_付録/02_非機能要求グレード判定.md に記述

**優先度**: MUST
