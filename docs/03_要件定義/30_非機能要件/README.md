# 30_非機能要件

本カテゴリは、k1s0 の非機能要件を IPA「非機能要求グレード 2018」の 6 大項目に準拠して定義する。機能要件（20_機能要件）が「何ができるか」を規定するのに対し、本カテゴリは「どの水準で満たすか」を規定する。

## 本カテゴリの位置付け

非機能要件は稟議で外部約束する数値（SLA 稼働率 99%、API p99 < 500ms、RTO 4 時間等）の根拠文書であり、後続の設計・実装・運用で「要件として約束した水準」の判定基準となる。数値の根拠を丁寧に書かないと、レビュー時に「なぜ 99% なのか」「なぜ 500ms なのか」の質問で時間を浪費する。

稼働率とレイテンシの数値は **対外 SLA（本カテゴリ A/B 等）** と **内部 SLO（[I_SLI_SLO_エラーバジェット.md](I_SLI_SLO_エラーバジェット.md)）** の 2 層で運用する。SLA は対テナント契約で違反時にペナルティを伴う水準、SLO は SLA より厳しく設定した内部アラート閾値である。tier1 全般で SLA 99%（月 7.2 時間）／SLO 99.9%（月 43 分）、マージン 約 6.5 時間 を体質改善の余地として運用する。本カテゴリ内に現れる 99% / p99 500ms は SLA 値、I_SLI_SLO 内の 99.9% / 99.95% は SLO 値として読む。

## IPA 非機能要求グレード 2018 との対応

本カテゴリは IPA 非機能要求グレード 2018 の 6 大項目・約 230 小項目を骨格として採用する。k1s0 の社会的影響度判定（モデルシステム①②③相当）は [../90_付録/02_非機能要求グレード判定.md](../90_付録/02_非機能要求グレード判定.md) に集約する。本判定結果に基づき、各大項目のグレード水準を決定する。

### 6 大項目

- **A: 可用性**（[A_可用性.md](A_可用性.md)）: 継続性、耐障害性、災害対策、回復性
- **B: 性能・拡張性**（[B_性能拡張性.md](B_性能拡張性.md)）: 業務処理量、性能目標、リソース拡張、性能品質保証
- **C: 運用・保守性**（[C_運用保守性.md](C_運用保守性.md)）: 通常運用、保守運用、障害時運用、運用環境、サポート体制、運用管理方針
- **D: 移行性**（[D_移行性.md](D_移行性.md)）: 移行時期、移行方式、移行対象、移行計画
- **E: セキュリティ**（[E_セキュリティ.md](E_セキュリティ.md)）: 前提制約、リスク分析、アクセス制御、データ秘匿、不正監視、ネットワーク対策、マルウェア対策、Web 対策、インシデント対応
- **F: システム環境・エコロジー**（[F_システム環境エコロジー.md](F_システム環境エコロジー.md)）: システム制約、特性、適合規格、設置環境、環境管理

### k1s0 拡張項目（IPA 標準外、本プロジェクト追加）

k1s0 特有の要件として、IPA 標準 6 大項目で扱いが浅い以下 2 項目を追加する。

- **G: データ保護とプライバシー**（[G_データ保護とプライバシー.md](G_データ保護とプライバシー.md)）: データ分類、暗号化、データ所在地、保管期間、本人権利対応、プライバシー・バイ・デザイン
- **H: アーティファクト完全性とコンプライアンス**（[H_アーティファクト完全性とコンプライアンス.md](H_アーティファクト完全性とコンプライアンス.md)）: 署名、SBOM、SLSA、鍵管理、個人情報保護法／J-SOX／ISO 27001 整合、監査対応

## 要件 ID 体系

非機能要件は `NFR-<大項目>-<サブカテゴリ>-<通番>` で採番する。大項目は A〜F、サブカテゴリは中項目の英略称。ID 体系の詳細は [../00_要件定義方針/05_要件トレーサビリティ方針.md](../00_要件定義方針/05_要件トレーサビリティ方針.md) を参照。

## ファイル分割方針

各大項目 1 ファイルを基本とする。300 行を超えた場合に限り、中項目単位のサブフォルダに分割する。本書執筆時点では全大項目を単一ファイルで運用し、必要に応じて段階的に分割する。

## 数値根拠の原則

すべての数値（レイテンシ、可用率、容量、期間、回数等）には根拠を併記する。企画書の定量試算、構想設計の実測値、業界平均、社内規定、法令要件のいずれかを出典として明示する。根拠なき数値は要件として受入不可とする。

## 優先度の原則

優先度は MUST / SHOULD / COULD / WON'T の 4 段階で、各要件 1 つを判定理由併記で付与する。非機能要件では以下が目安となる。

- **MUST**: 稟議提示 SLO の達成に直結する要件、法令・社内規定で必須の要件
- **SHOULD**: 未達だと Phase ゲート通過後の手動運用負荷が 50% 以上増える要件
- **COULD**: 余裕があれば実装、未達でも他手段で回避可能
- **WON'T**: 本バージョンでは実施しない
