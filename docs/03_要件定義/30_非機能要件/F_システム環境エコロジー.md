# F. システム環境・エコロジー

本書は IPA 非機能要求グレード 2018 の「F. システム環境・エコロジー」に準拠し、k1s0 のシステム制約、システム特性、適合規格、設置環境、環境マネージメントの要件を定義する。企画書で制約した「オンプレ・閉域・VM ベース」の設計条件を具体水準化する。

## 本章の位置付け

システム環境要件は、物理・運用条件の境界を明示する。これは後続の構想設計・基本設計フェーズで「どの範囲で考えればよいか」の境界条件となり、設計者が環境制約を勝手に拡張して予算オーバーするのを防ぐ。

## F1. システム制約・前提条件

### NFR-F-SYS-001: オンプレミス完結

**現状**: クラウド依存は JTC のデータ主権・コスト予測可能性・閉域要件と整合しない。

**要件達成後**: k1s0 はオンプレ VM 上で完結する。クラウドマネージドサービス（RDS、SQS、CloudFront 等）への依存を禁止する。外部インターネット接続は OSS パッケージ取得とセキュリティ更新のみ、ビジネスデータの外部送信は禁止（Binding API の allowlist で制御）。

**崩れた時**: C-VEND-001（外部商用 SaaS 依存禁止）と C-LEGAL-001（個人情報保護法データ主権）に抵触する。

**受け入れ基準**:
- クラウドマネージドサービスへの依存コードが存在しない
- インターネット送信のログを監視

**優先度**: MUST

### NFR-F-SYS-002: 閉域ネットワーク

**現状**: 外部からの直接アクセスは JTC 社内規程で禁止される。

**要件達成後**: k1s0 は社内 LAN / VLAN 内で動作し、外部からの直接アクセスは許容しない。社員のリモートアクセスは既存 VPN / リモートデスクトップ経由。

**崩れた時**: JTC 社内規程違反、セキュリティ監査指摘。

**受け入れ基準**:
- 外部からの Ingress が NetworkPolicy / Firewall で拒否
- リモート接続は既存 VPN のみ

**優先度**: MUST

### NFR-F-SYS-003: VM ベース

**現状**: ベアメタル調達はリードタイムが長く、既存 VM 基盤の活用が現実的。

**要件達成後**: k1s0 は VMware ESXi / KVM / Hyper-V 等の VM 上で動作する。ベアメタル最適化は実施しない。VM ハイパーバイザの選択は JTC 既存資産に依存（C-INF-001）。

**崩れた時**: 既存 VM 基盤を活用できず、新規ハード調達が必要となる。

**受け入れ基準**:
- VM ハイパーバイザ非依存の Kubernetes 構成
- 既存 VM 基盤で動作確認

**優先度**: MUST

## F2. システム特性

### NFR-F-CHR-001: リソース規模（Phase 1a/1b）

**現状**: 必要リソースが曖昧だと、VM 調達の承認が得られない。

**要件達成後**: Phase 別に必要リソースを以下に確定する。

- **Phase 1a**: VM 1 台（4 vCPU / 16 GB / 200 GB SSD）、オールインワン構成
- **Phase 1b**: VM 3 台（各 4 vCPU / 16 GB / 200 GB SSD）、k8s HA 最小構成
- **Phase 1c**: Phase 1b と同構成を継続運用
- **Phase 2**: 規模別に VM 5〜10 台、詳細は Phase 2 キックオフで再確定
- **Phase 3**: マルチクラスタ（staging + prod）、規模別に VM 10〜20 台
- **Phase 5**: マルチリージョン、管理クラスタ追加で VM 15〜30 台

**崩れた時**: 調達不足で性能目標を達成できない、または過剰調達で予算超過。

**受け入れ基準**:
- 各 Phase のリソース見積を調達部門と合意
- Phase 移行時に実測に基づき再確定

**優先度**: MUST

### NFR-F-CHR-002: ストレージ要件

**現状**: ストレージ要件が曖昧だと、必要容量と IOPS が決められない。

**要件達成後**: 以下のストレージ要件を確定する。

- **PostgreSQL（CloudNativePG）**: SSD、IOPS 1,000 以上、容量は業務データ量に応じて拡張（Phase 1b 100 GB 初期）
- **Kafka（Strimzi）**: SSD 推奨、IOPS 500 以上、容量 50 GB / ブローカー（Phase 1b）
- **MinIO**: HDD 可（RAID 構成）、容量 500 GB〜（Phase 1b、バックアップ集約先）
- **Valkey**: SSD、メモリ主体（永続化ディスク 20 GB）
- **etcd**: SSD、IOPS 500 以上、容量 10 GB（k8s 管理）
- **Longhorn**: ボリューム管理、レプリケーション 3

**崩れた時**: I/O ボトルネックで p99 が悪化、または過剰なストレージ調達で予算超過。

**受け入れ基準**:
- ストレージ要件の調達部門合意
- 四半期ごとに実使用量モニタリング

**優先度**: MUST

### NFR-F-CHR-003: ネットワーク要件

**現状**: ネットワーク帯域と VLAN 構成が確定しないと、設計が進まない。

**要件達成後**: 以下のネットワーク要件を確定する。

- 社内 LAN 1 Gbps 以上（Phase 1b / 1c）、10 Gbps 推奨（Phase 2+）
- VLAN 分離（tier1 インフラ、tier2 アプリ、tier3 UI の最低 3 セグメント）
- 社内 NTP / DNS 参照
- 既存プロキシ経由のインターネット接続（OSS パッケージ取得用）
- MetalLB L2 モード（Phase 1b）、BGP モード（Phase 3+）

**崩れた時**: ネットワーク設計が後付けとなり、VLAN 再編やプロキシ設定で手戻り発生。

**受け入れ基準**:
- ネットワーク要件の情シスインフラ運用部門合意
- Phase 1b キックオフ前に構成図確定

**優先度**: MUST

## F3. 適合規格

### NFR-F-STD-001: クラウドネイティブ標準準拠

**現状**: 独自仕様で構築すると、OSS コミュニティの恩恵を受けにくい。

**要件達成後**: 以下の業界標準に準拠する。

- **Kubernetes** (CNCF): オーケストレーション
- **OCI Image / Distribution Spec**: コンテナイメージ
- **OpenTelemetry** (CNCF): 計装
- **SPIFFE**: ワークロード ID（Istio Ambient 経由）
- **W3C Trace Context**: 分散トレース伝搬
- **OpenFeature** (CNCF): Feature Flag（flagd 経由）
- **OCI Artifact** / **SPDX / CycloneDX**: SBOM

**崩れた時**: 独自拡張で OSS の恩恵を受けられず、採用 OSS の入れ替え困難。

**受け入れ基準**:
- 採用 OSS が業界標準準拠
- ADR で独自拡張の合理性を記述

**優先度**: MUST

### NFR-F-STD-002: アクセシビリティ（tier3 UI）

**現状**: 社内アプリのアクセシビリティ対応が曖昧だと、多様な社員の利用を阻害する。

**要件達成後**: tier3 アプリは WCAG 2.1 AA 相当を目標とする。k1s0 tier1 は UI を持たないため直接の対応範囲外だが、tier3 向けガイドラインを提供する。

**崩れた時**: 視覚・聴覚等に配慮が必要な社員が業務アプリを使えない。

**受け入れ基準**:
- tier3 向けアクセシビリティガイドライン（Phase 1b）
- Backstage / Grafana 等の管理 UI も WCAG 2.1 AA 目標

**優先度**: SHOULD

## F4. 機材設置環境条件

### NFR-F-FAC-001: データセンター環境

**現状**: JTC 既存 DC の環境（温湿度、電源、ラック）を前提とする。

**要件達成後**: 既存 DC の環境条件に依存する。専用設置工事は不要。ラック U 数、電力容量は既存 VM 基盤の範囲内で収める。

**崩れた時**: DC 改修・電源増設で予算超過。

**受け入れ基準**:
- 既存 DC の環境範囲内で完結
- 増設が必要な場合は調達部門と事前合意

**優先度**: MUST

### NFR-F-FAC-002: ハードウェア耐用年数

**現状**: 既存 VM 基盤の耐用年数を超えてのリースや保守切れは運用負荷。

**要件達成後**: VM 基盤のリース期間（通常 5 年）を考慮した計画を維持する。保守切れハードでの運用は禁止。

**崩れた時**: 保守切れハードの故障で長時間停止、復旧不能のリスク。

**受け入れ基準**:
- 調達部門とハードウェア更新計画を共有
- リース更新時期に合わせた移行計画

**優先度**: MUST

## F5. 環境マネージメント

### NFR-F-ECO-001: 電力効率

**現状**: 電力コストは DC 運用費の主要項目。過剰な VM 割当は電力コスト増加に直結。

**要件達成後**: HPA / KEDA による自動スケールインで、業務時間外の電力消費を最小化する。未使用 VM のパワーダウン手順を整備（Phase 3+）。

**崩れた時**: 電力コストが想定を超過、環境配慮の観点でも不利。

**受け入れ基準**:
- HPA / KEDA による自動縮退
- Phase 3 以降で電力効率指標（kWh / 業務処理）の記録

**優先度**: SHOULD

### NFR-F-ECO-002: 廃棄時のデータサニタイズ

**現状**: ハードウェア廃棄時のデータ消去手順が不徹底だと、情報漏えいリスク。

**要件達成後**: 廃棄時のストレージサニタイズ（NIST SP 800-88 Rev.1 Purge 相当）を標準手順化する。廃棄証明書を 7 年保管。

**崩れた時**: 廃棄データ漏えいでコンプライアンス違反、監督官庁対応。

**受け入れ基準**:
- 廃棄手順 Runbook（Runbook-OPS-DISPOSE-001）
- 廃棄証明書の保管

**優先度**: MUST

### NFR-F-ECO-003: カーボンフットプリント報告（Phase 5 以降）

**現状**: CO2 排出量報告義務が拡大傾向で、JTC でも ESG 報告対応が必要になりうる。

**要件達成後**: Phase 5 以降、DC 単位の電力消費量と CO2 換算値を年次レポートに含める。Grafana ダッシュボードで運用中のエネルギー効率を可視化。

**崩れた時**: ESG 報告対応で後手となり、経営層からの説明要求に応えられない。

**受け入れ基準**:
- Phase 5 で方針確定、具体水準は未確定
- 現時点では参考扱い

**優先度**: COULD（Phase 5）
