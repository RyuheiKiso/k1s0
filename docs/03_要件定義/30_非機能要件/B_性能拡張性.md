# B. 性能・拡張性

本書は IPA 非機能要求グレード 2018 の「B. 性能・拡張性」に準拠し、k1s0 の業務処理量、性能目標値、リソース拡張性、性能品質保証の要件を定義する。企画書提示の SLO（API p99 < 500ms、定常 150 RPS 中規模）を根拠ある数値として具体化する。

## 本章の位置付け

性能要件は「どれだけ速いか」ではなく「何人までどんな負荷で捌けるか」を定義する。試算前提で定めた 3 段階（小 1,000 名 / 中 3,000 名 / 大 10,000 名）に対応する業務処理量を想定し、中規模を第一スコープとする。ピーク RPS 150（中規模試算値）で p99 500ms を達成することを MUST とし、採用検討で外部約束した数値を裏付ける。

## B1. 業務処理量

### NFR-B-WL-001: 想定規模と RPS

**現状**: 規模別の想定処理量が不明確なまま設計すると、リリース時点 で「実は 10 倍必要だった」という手戻りが発生する。

**要件達成後**: 試算前提に基づき、規模別のピーク RPS を以下に固定化する。

- 小規模（1,000 名）: ピーク RPS 50
- 中規模（3,000 名）: ピーク RPS 150
- 大規模（10,000 名）: ピーク RPS 500

本値は企画書の規模区分（[../../01_企画/企画書.md](../../01_企画/企画書.md)）に基づく。中規模を リリース時点 の性能試験基準とし、採用後の運用拡大時の 段階ゲートで実測を踏まえて再確定する。

**崩れた時**: 想定外の負荷で degrade 発生、tier2/tier3 利用部門から苦情が出る。

**受け入れ基準**:

- リリース時点 性能試験で中規模 150 RPS を定常負荷として実施
- 試算前提の出典を引用（査読可能）
- 採用後の運用拡大時 キックオフ時に実測値で再確定

**優先度**: MUST

### NFR-B-WL-002: 業務時間帯トラフィック分布

**現状**: 業務時間帯とそれ以外のトラフィック比率が不明だと、バッチ処理・定期実行の設計時期判断が曖昧になる。

**要件達成後**: 業務時間帯（平日 8:00〜20:00 JST）にトラフィックの 95% が集中、深夜 / 週末は 5% 以下と想定する。バッチ処理は深夜ウィンドウ（2:00〜6:00）で実行し、業務時間帯に重ねない。

**崩れた時**: バッチ実行が業務時間帯に重なり、p99 が悪化する。

**受け入れ基準**:

- バッチ処理は深夜ウィンドウで実行（FR-T1-BINDING-004 の cron スケジュール）
- Prometheus で時間帯別トラフィックを記録、想定比率との乖離を監視

**優先度**: SHOULD

## B2. 性能目標値

### NFR-B-PERF-001: tier1 API p99 レイテンシ 500ms

**現状**: 既存業務システムのレスポンスタイムは、社員体感で「遅いと感じる 500ms 以上」が散発的に発生している。新基盤でこの水準を下回ることは最低限の期待値。

**要件達成後**: tier1 公開 API の全エンドポイントで、定常負荷（中規模 150 RPS）において p99 < 500ms を達成する。下記の成分分解（層別内訳）に従い、各層での割当時間を目標とする。

**崩れた時**: tier2/tier3 アプリの UX が悪化、エンドユーザーから「新基盤は遅い」と評価される。採用検討提示 SLO の違反で経営層への説明責任が発生する。

#### 成分分解と根拠

p99 < 500ms は tier2 呼び出し元から tier1 ファサード → バックエンド → 応答までのエンドツーエンド値である。採用検討レビュアーや基本設計担当者が「どの層をどこまで削れば目標に達するか」を判断するため、層別の目標値・算出根拠・計測方法を以下に明示する。目標値合計は 530ms（30ms のマージン込み）であり、採用初期 での eBPF 実測後に再配分する。

| 層 | コンポーネント | 目標 p99 | 算出根拠 | 計測方法 | 担当 |
|---|---|---|---|---|---|
| tier2 アプリ層 | ビジネスロジック処理 | 200ms | 複雑業務ルール評価最大値（社内既存システム実測の 95 パーセンタイル） | OTel span `tier2.handler` | tier2 チーム |
| tier1 ファサード入口 | Dapr SDK + k1s0 SDK オーバーヘッド | 30ms | stable Dapr Go SDK の gRPC 呼出オーバヘッド（Dapr ベンチ実測）+ interceptor 処理 | eBPF probe + Tempo span `k1s0.facade.ingress` | tier1 チーム |
| tier1 バックエンド呼出 | 実サービス/バックエンド呼出 | 80ms | 既存 .NET Framework 応答中央値 50ms + 長尾 30ms（社内既存ベンチ） | Tempo span `k1s0.facade.upstream` | tier1 チーム |
| tier1 観測層 | OTel Collector 処理 | 20ms | span 作成 + attributes 追加 + バッチ送信のノンブロッキング許容時間 | OTel SDK internal trace | Observability チーム |
| tier1 監査層 | Audit API イベント発行 | 50ms | HashChain 計算（5ms 実測）+ MinIO WORM 非同期 write キュー投入 10ms + 冗長 35ms | Tempo span `k1s0.audit.emit` | Security チーム |
| インフラ層 | ネットワーク + データベース往復 | 150ms | 社内クラウド NW レイテンシ 20ms × 最大 5 往復 + DB 処理 50ms（PostgreSQL 実測 p99） | Prometheus `network_latency_bucket` + `db_query_duration` | Platform チーム |
| **合計** | - | **530ms** | 目標 500ms + マージン 30ms（リリース時点 実測で再配分） | - | - |

マージン 30ms（目標 500ms と合計 530ms の差）は、各層の実測値が個別の目標から外れても全体として達成可能とする緩衝である。ただし特定層が 20% 以上目標を超過する場合は、他層のマージンを食い潰す前に ADR で是正計画を記録する。

#### リリース時点 での実測手順

1. eBPF トレーシング（Parca / Pixie）で各層のレイテンシを 1 時間連続記録
2. 実測値と目標値の乖離を層別に算出、10% 以上の乖離があればチューニング対象
3. チューニング後も 10% 乖離が残る場合は、NFR-B-PERF-001 の層別目標値を改訂し、関連 ADR（例: ADR-TIER1-001、ADR-RULE-002）に記録
4. Grafana ダッシュボード `tier1-perf-breakdown` で層別 p99 を常時可視化

**受け入れ基準**:

- リリース時点 性能試験で p99 < 500ms を 1 週間連続維持
- 層別目標（上表）の各層 p99 も 1.2 倍以内で維持（個別層の暴発を検出）
- 未達時は 採用初期 の eBPF 実測を踏まえた層別再配分で数値調整、ADR 化
- Grafana で p50 / p90 / p99 / p99.9 および層別 p99 を可視化、SLO アラート設定

**優先度**: MUST

### NFR-B-PERF-002: tier1 API スループット 150 RPS（中規模）

**現状**: スループット目標が不明だと、水平拡張設計の基準が定まらない。

**要件達成後**: 中規模 3,000 名のピーク 150 RPS を 6 vCPU × 12 GB（tier1 API サーバ 2 レプリカ合計）で処理可能とする。CPU 使用率 60% 以下、メモリ使用率 70% 以下を維持。

**崩れた時**: リソース不足で degrade が頻発、増設対応に予算超過が発生する。

**受け入れ基準**:

- リリース時点 性能試験で 150 RPS 定常、CPU / メモリ使用率の記録
- 200 RPS（中規模 1.3 倍）でも p99 < 500ms を維持（余裕分）
- Grafana でリソース使用率を可視化

**優先度**: MUST

### NFR-B-PERF-003: State API Get p99 < 10ms

**現状**: キャッシュ用途の Get 操作が 100ms を超えると、キャッシュの意味が薄れる。

**要件達成後**: Valkey バックエンドの State API Get 操作は p99 < 10ms を達成する。Valkey は同一 Node に配置、ネットワーク経由でも単一 hop で完結。

**崩れた時**: State API が遅すぎてキャッシュとして機能せず、tier2 アプリが別のキャッシュ実装に走る。

**受け入れ基準**:

- リリース時点 で State API Get の p99 < 10ms 実測
- 書き込み p99 < 20ms（書き込みは Valkey のディスク永続化待ち）

**優先度**: MUST

### NFR-B-PERF-004: Decision 評価 p99 < 1ms

**現状**: ビジネスルール評価が遅いと、高頻度呼び出し（例: 全画面で権限判定）で UX に影響する。

**要件達成後**: ZEN Engine による JDM 決定表評価は in-process 実行で p99 < 1ms（マイクロ秒オーダ目標）。評価対象決定表のサイズと複雑度次第だが、標準的な規模（100 ルール以内）で本水準を達成する。

**崩れた時**: Decision API が業務ロジックのボトルネックになり、tier2 が if/else ハードコードに逃げる。

**受け入れ基準**:

- リリース時点 デモで p99 < 1ms を確認
- 大規模決定表（1,000 ルール以上）の場合の劣化も計測

**優先度**: MUST

### NFR-B-PERF-005: PubSub Publish p99 < 50ms

**現状**: Publish が遅いと、イベント駆動の業務処理全体が詰まる。

**要件達成後**: Kafka Publish の p99 < 50ms（acks=all 設定、同期待ち込み）を達成する。

**崩れた時**: PubSub API のレイテンシが業務処理全体の p99 を引き上げる。

**受け入れ基準**:

- リリース時点 性能試験で実測
- acks=all 維持、at-least-once 保証を優先

**優先度**: MUST

### NFR-B-PERF-006: Log / Telemetry 計装オーバヘッド < 10ms

**現状**: 計装コードが重すぎると、tier2 アプリが計装を省略する誘因となる。

**要件達成後**: tier1 Log / Telemetry API の計装オーバヘッドは tier2 業務処理 p99 に対し 10ms 以下。非ブロッキング送信（バックグラウンド flush）で吸収する。

**崩れた時**: 計装を避けた tier2 アプリが増え、分散トレースのカバレッジが低下、障害調査が困難になる。

**受け入れ基準**:

- リリース時点 で計装あり / なしの p99 差分を計測
- 差分 10ms 以下を維持

**優先度**: MUST

### NFR-B-PERF-007: Feature Flag 評価 p99 < 10ms

**現状**: Feature Flag 評価が遅いと、全画面で行う権限判定や機能 ON/OFF チェックが UX に影響する。

**要件達成後**: flagd バックエンドの Feature API 評価は、キャッシュヒット時 p99 < 10ms。キャッシュ TTL 30 秒で 99% 以上がヒット。

**崩れた時**: Feature Flag 評価がボトルネックとなり、tier2 が Flag 呼び出しを削減して段階ロールアウトの粒度が粗くなる。

**受け入れ基準**:

- リリース時点 で実測
- キャッシュヒット率の Grafana 可視化

**優先度**: MUST

## B3. リソース拡張性

### NFR-B-RES-001: tier1 API サーバの水平拡張

**現状**: 想定規模を超える負荷に対して、スケールアウト手順が確立されていないと、緊急増強で手作業が発生する。

**要件達成後**: tier1 API サーバは HPA（Horizontal Pod Autoscaler）で 2〜10 レプリカの自動拡縮を構成する。CPU 使用率 70% 超過で scale out、30% 以下で scale in。KEDA を使う場合は Kafka Consumer Lag / Prometheus メトリクスも評価条件に加える。

**崩れた時**: 負荷スパイク時に手動スケールアウトが必要となり、対応が間に合わず degrade が発生する。

**受け入れ基準**:

- HPA / KEDA 設定で 2〜10 レプリカの自動拡縮
- リリース時点 で負荷スパイク試験（定常の 5 倍）を実施、自動拡縮の動作確認

**優先度**: MUST

### NFR-B-RES-002: Valkey / PostgreSQL / MinIO の垂直・水平拡張

**現状**: データバックエンドのスケールアウト・スケールアップが予測不能だと、予算・リソース計画が立たない。

**要件達成後**: 各バックエンドの拡張ポリシーを明文化する。

- Valkey: 垂直（CPU / メモリ増）で対応、採用側のマルチクラスタ移行時 で Sentinel による水平化検討
- PostgreSQL: 垂直（CloudNativePG 設定変更）で対応、読み取りは Read Replica で水平化
- MinIO: 水平拡張（ノード追加）で対応、採用後の運用拡大時 で検討

**崩れた時**: バックエンドが飽和し、tier1 API 全体が degrade する。

**受け入れ基準**:

- 各バックエンドの拡張手順書を Runbook に含める
- Grafana でリソース使用率トレンドを可視化、閾値超過でアラート

**優先度**: SHOULD

### NFR-B-RES-003: Workflow 並列実行の水平拡張

**現状**: ワークフロー並列実行数が制限されると、業務ピーク時に詰まる。

**要件達成後**: Dapr Workflow / Temporal Worker を HPA で自動拡縮する。Kafka Consumer Lag ベースで KEDA 拡縮をサブ条件に追加。

**崩れた時**: Workflow 実行キューが詰まり、業務プロセスが遅延する。

**受け入れ基準**:

- リリース時点 で HPA 設定と負荷試験
- Workflow キューの滞留を Grafana で可視化

**優先度**: SHOULD

## B4. 性能品質保証

### NFR-B-QA-001: リリース時点 性能試験

**現状**: 性能試験が無いと、採用検討提示の SLO が紙の約束で終わり、実運用時に違反する。

**要件達成後**: リリース時点 で中規模 150 RPS 定常負荷試験を実施し、p99 / スループット / リソース使用率を計測する。本試験は 採用初期 の KPI 検収に組み込む。

**崩れた時**: 本番稼働後に初めて SLO 違反が判明、信頼失墜となる。

**受け入れ基準**:

- 負荷試験ツール（k6 / Gatling 等）の標準テンプレート整備
- 試験結果は Backstage でアーカイブ、継続比較可能
- 試験結果に基づき 90_付録/02_非機能要求グレード判定.md を更新

**優先度**: MUST

### NFR-B-QA-002: 定期性能退行検知

**現状**: コード変更・OSS バージョンアップで性能退行が発生しても、気づけないまま累積する。

**要件達成後**: CI パイプラインに軽量ベンチマーク（主要 API の p99 計測）を組み込み、基準値から 10% 以上の退行を PR ブロックする。月次で 150 RPS 定常負荷試験を実施する。

**崩れた時**: 性能退行が累積し、半年後に SLO 違反に至る。原因コミット特定が困難。

**受け入れ基準**:

- CI ベンチマークの基準値を リリース時点 で確定
- 月次負荷試験の自動化スクリプト
- 優先度 SHOULD、リリース時点 で実装

**優先度**: SHOULD
