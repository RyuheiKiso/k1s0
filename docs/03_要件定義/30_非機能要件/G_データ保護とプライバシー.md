# G. データ保護とプライバシー

本書は k1s0 が扱うデータの保護水準（暗号化・分類・保管・越境・削除）と、個人情報・要配慮個人情報のプライバシー要件を詳細定義する。IPA 非機能要求グレード 2018 の E セキュリティ（E_セキュリティ.md）で扱う広範なセキュリティ要件のうち、データ取扱に特化した要件を本書で深掘りする。

## 本章の位置付け

個人情報保護法 2022 改正と GDPR 等の国際標準を満たすために、データ分類・暗号化・保管場所・越境移転・削除・本人権利の全てを手続化する必要がある。E 章が横断的なセキュリティ要件であるのに対し、本書は「データ個々」に対する取扱要件を定める。この 2 章は E → G の順で読むと、全体 → 個別の流れで理解しやすい。

## G1. データ分類と識別

### NFR-G-CLS-001: 4 段階機密度分類

**現状**: データ機密度が揃っていないと、同じストレージに公開情報と個人情報が混在する。

**要件達成後**: 全データを 4 段階に分類する（[../00_要件定義方針/09_データ辞書.md](../00_要件定義方針/09_データ辞書.md) と整合）。

- **Public**: 社外公開可（SBOM、公開 API ドキュメント）
- **Internal**: 社内限定（構成情報、ADR、運用手順）
- **Confidential**: 業務機密（業務データ、設計書、取引情報）
- **Restricted**: 最機密（個人情報、認証情報、KMS 鍵、要配慮個人情報）

レベルに応じて暗号化・アクセス制御・保管期間・バックアップを差別化する。

**崩れた時**: 機密データが不適切なストレージに流入、個人情報保護法違反リスク。

**受け入れ基準**:
- Phase 1a で分類ルール確定
- 全データセットに分類ラベル付与
- 四半期ごとに分類レビュー

**優先度**: MUST

### NFR-G-CLS-002: PII 検出と自動タグ付け

**現状**: 個人情報がどこに入り込むか不可視だと、漏えい時の影響範囲特定に時間がかかる。

**要件達成後**: 以下で PII を検出・タグ付けする。

- **入力時検出**: tier1 PII API でリクエストボディ中の個人情報パターン（氏名、メール、電話、マイナンバー等）を検出
- **定期スキャン**: PostgreSQL、MinIO、ログに対して月次で PII スキャン実行、発見時にアラート
- **タグ保全**: テーブル・カラム・オブジェクトに PII タグをメタデータとして保存
- **監査**: PII を含むデータへのアクセスは必ず監査ログに記録（通常データより厳格）

**崩れた時**: 個人情報が非暗号化ストレージに流入、漏えい時の通知範囲特定が困難。

**受け入れ基準**:
- Phase 1b で tier1 PII API 稼働
- Phase 1c で定期 PII スキャン
- 月次で PII 含有率レポート

**優先度**: MUST

## G2. 暗号化

### NFR-G-ENC-001: 保存時暗号化（at-rest）

**現状**: ストレージ盗難・マルウェア侵入時の情報漏えいを防ぐため保存時暗号化が必須。

**要件達成後**: 以下で保存時暗号化を実施する。

- **データベース（PostgreSQL）**: TDE（Transparent Data Encryption）相当、または FDE（Full Disk Encryption）
- **MinIO**: Server-Side Encryption（SSE-S3 相当）、鍵は KMS 管理
- **Kafka**: ブローカーディスク暗号化、将来的には topic 内暗号化（Phase 3+）
- **Valkey**: メモリ暗号化は基本不要、永続化ディスクは FDE
- **バックアップ**: 全バックアップを AES-256 で暗号化、鍵は別保管
- **ログ・メトリクス**: Restricted データを含むログは暗号化、それ以外は FDE 依存

鍵管理は OpenBao + KMS Operator（cert-manager + 自作 controller）で実施、90 日ローテーション。

**崩れた時**: ストレージ盗難で個人情報漏えい、暗号化されていれば漏えいとみなさない救済措置を得られない。

**受け入れ基準**:
- Phase 1a で FDE 全 VM 適用
- Phase 1b で PostgreSQL TDE 相当、MinIO SSE 有効化
- 鍵ローテーションの自動化

**優先度**: MUST

### NFR-G-ENC-002: 転送時暗号化（in-transit）

**現状**: 通信経路での盗聴・改ざん防止のため転送時暗号化必須。

**要件達成後**: 以下で転送時暗号化を実施する。

- **tier1 ↔ tier2 ↔ tier3**: Istio Ambient mesh の mTLS 自動適用
- **tier1 ↔ データベース**: TLS 1.3 必須
- **tier1 ↔ 外部（Keycloak 等）**: TLS 1.3 必須
- **Ingress**: TLS 1.2 最低、1.3 推奨、cert-manager で Let's Encrypt 自動取得
- **Egress**: Envoy で TLS 検証強制

TLS 1.0 / 1.1 / SSL は禁止、弱い暗号スイート（RC4、3DES）も禁止。

**崩れた時**: 中間者攻撃で通信盗聴、個人情報漏えい。

**受け入れ基準**:
- Phase 1a で全通信 TLS 化
- 弱い暗号スイート禁止を Kyverno で強制
- TLS バージョン監査を四半期実施

**優先度**: MUST

### NFR-G-ENC-003: アプリケーションレベル暗号化

**現状**: 特に機密度の高いデータはストレージ暗号化だけでなくアプリ層でも暗号化が必要。

**要件達成後**: 以下のデータはアプリケーションで暗号化する。

- **要配慮個人情報**: 健康、信条、犯罪歴等、ストレージ暗号化に加えアプリ層で暗号化
- **マイナンバー**: 法令要件に基づく暗号化
- **認証情報**: Argon2id / bcrypt でのハッシュ（平文保存禁止）
- **フィールドレベル暗号化**: PostgreSQL の pgcrypto を活用、特定カラムのみ暗号化

鍵は OpenBao に保管、アプリは KMS API 経由でアクセス。

**崩れた時**: DBA や SRE が業務で見るべきでない個人情報を誤って閲覧。

**受け入れ基準**:
- Phase 1c でアプリレベル暗号化を必要データに実装
- 鍵アクセスは MFA + 承認フロー
- 年次で暗号化実装の監査

**優先度**: MUST

## G3. アクセス制御（データ中心）

### NFR-G-AC-001: 最小権限の原則

**現状**: 開発者・運用者に過度な本番データ参照権があると、内部不正・誤操作リスク。

**要件達成後**: 以下の原則で権限を最小化する。

- **本番データ参照**: 業務上必要な場合のみ、2 人承認で時限的に付与
- **DBA 権限**: 構造変更とバックアップのみ、データ SELECT は原則禁止
- **テナント越境アクセス**: デフォルト禁止、例外は監査ログに残す
- **ログアクセス**: PII を含むログは特定ロールのみ閲覧可
- **鍵管理**: KMS 鍵の利用は cell（暗号化単位）別に RBAC

全ての特権昇格は監査ログに記録、四半期で棚卸し。

**崩れた時**: 特権者の数が増加、セキュリティ態勢の弱点となる。

**受け入れ基準**:
- Phase 1b で本番アクセス承認フロー稼働
- 特権アカウント数を四半期レビュー
- 不要権限の自動削除（90 日未使用で失効）

**優先度**: MUST

### NFR-G-AC-002: データマスキング

**現状**: 開発・テスト環境で本番データをそのまま使うと、個人情報の流出リスク。

**要件達成後**: 以下のマスキング戦略を適用する。

- **非本番環境**: 本番データのコピーは禁止、合成データまたはマスク済みデータのみ
- **マスキング手法**: 氏名は仮名、メールは `masked_<hash>@example.com`、電話は `XXX-XXXX-XXXX`、住所は都道府県のみ
- **匿名化**: 統計分析には k-匿名性（k=5 以上）を保証
- **ツール**: 自作マスキングパイプライン（pgloader + カスタム変換）を Phase 2 で整備

**崩れた時**: 開発者 PC 経由で本番個人情報が漏えい。

**受け入れ基準**:
- Phase 1c でマスキング基準確定
- Phase 2 でマスキングパイプライン稼働
- 非本番環境での本番データ検出を定期監査

**優先度**: MUST

## G4. データ所在地（Data Residency）

### NFR-G-RES-001: 国内保管の徹底

**現状**: 個人情報の海外サーバ保管は本人同意や個人情報保護委員会対応が必要。

**要件達成後**: 以下の原則で国内保管を徹底する。

- **全 tier1 API のデータ**: JTC 国内 DC に保管、海外クラウド・海外 DC 利用禁止
- **バックアップ**: 国内の別 DC（DR サイト）に保管
- **ログ・メトリクス**: 国内保管
- **Secret（KMS）**: 国内保管
- **例外**: 海外拠点展開（Phase 4+）で現地データは現地保管可、越境時は個人情報保護法 28 条対応

データ所在地の定期監査を実施、海外流出の有無を確認。

**崩れた時**: 越境移転の法令違反、本人同意取得漏れ。

**受け入れ基準**:
- Phase 1a で所在地ポリシー確定
- 四半期監査で全データ所在を確認
- 海外拠点展開時の越境フロー整備（Phase 4+）

**優先度**: MUST

## G5. 保管期間とライフサイクル

### NFR-G-LIF-001: 標準保管期間の適用

**現状**: データ保管期間が一律だと、不要なデータまで保持してストレージコストと漏えいリスクが増加。

**要件達成後**: データ種別ごとに保管期間を定める（09_データ辞書.md と整合）。

- **業務データ**: 業務要件に応じる（例: 取引データ 10 年、顧客データ 5 年）
- **監査ログ**: 7 年（J-SOX、個人情報保護法、社内規程）
- **アプリケーションログ**: Hot 90 日、Warm 1 年
- **メトリクス**: 高解像度 15 日、中 1 年、低 3 年
- **バックアップ**: 日次 30 日、月次 1 年、年次 7 年

保管期間超過データは自動削除、または匿名化処理。

**崩れた時**: 不要データの蓄積で漏えい時の影響範囲拡大。

**受け入れ基準**:
- Phase 1c で自動削除パイプライン稼働
- 削除実行ログを監査ログに保全
- 年次で保管期間見直し

**優先度**: MUST

### NFR-G-LIF-002: テナント退会時の完全削除

**現状**: テナント退会後にデータ残存すると、個人情報保護法の目的達成後消去義務違反。

**要件達成後**: テナント退会時に以下を実施する。

- **T+30 日**: テナント側でデータ退避完了
- **T+90 日**: 名前空間内全データ削除（PostgreSQL 行、MinIO オブジェクト、Valkey キー）
- **T+90 日**: バックアップからのテナントデータ削除（次回バックアップ世代から不在）
- **T+180 日**: 監査ログからのテナント名前解除（監査ログ自体は 7 年保持、テナント識別は仮名化）
- **削除証明**: 完全削除を証明する監査ログ、テナントに送付

**崩れた時**: 退会テナントのデータが残存し、万一漏えいで大問題。

**受け入れ基準**:
- Phase 1c で退会削除フロー Runbook 化
- 年次で削除フローのリハーサル
- 削除証明の法務レビュー

**優先度**: MUST

## G6. 本人権利対応

### NFR-G-PRV-001: 開示請求対応

**現状**: 個人情報保護法で本人の開示請求権があり、対応手順が必要。

**要件達成後**: 以下の手順で開示請求に対応する。

- **受付**: 法務部が窓口、本人確認を厳格実施
- **検索**: 本人識別子（user_id）でデータ横断検索（PostgreSQL、MinIO、ログ）
- **抽出**: 本人データを構造化形式（JSON / CSV）で抽出
- **提供**: 暗号化 PDF または閲覧可能リンク、本人確認後に提供
- **期限**: 受付から 30 日以内

検索ツールは Backstage で運用、SRE が実行。

**崩れた時**: 法定期限内に応答できず、法令違反。

**受け入れ基準**:
- Phase 1c で横断検索ツール整備
- Runbook-LEGAL-DISCLOSE-001 で手順確立
- リハーサルを半期実施

**優先度**: MUST

### NFR-G-PRV-002: 訂正・削除請求対応

**現状**: 本人の訂正・削除請求に対応できないと、法令違反。

**要件達成後**: 以下で対応する。

- **訂正請求**: データ訂正を実施、履歴は監査ログに残す
- **削除請求**: 個人情報保護法の条件を満たす場合に削除、監査ログは仮名化して保持
- **利用停止請求**: 特定の処理（マーケティング等）の停止、タグ付与で管理

受付から 30 日以内に対応。

**崩れた時**: 本人権利の尊重ができず、訴訟リスク。

**受け入れ基準**:
- Phase 1c で対応フロー確立
- 法令遵守の外部監査を年 1 回
- 対応実績を月次報告

**優先度**: MUST

### NFR-G-PRV-003: 個人情報漏えい通知

**現状**: 漏えい時の法定通知プロセスが機能しないと、監督官庁対応で企業信用毀損。

**要件達成後**: [../40_運用ライフサイクル/01_インシデント対応詳細.md](../40_運用ライフサイクル/01_インシデント対応詳細.md) OR-INC-005 と整合して対応。

- **T+24 時間**: 個人情報保護委員会への初報
- **T+72 時間**: 詳細報告、本人通知開始
- **T+30 日**: 最終報告、再発防止策提出

**崩れた時**: 法令違反で改善命令、罰金、信用毀損。

**受け入れ基準**:
- Phase 1b で通知フロー確立
- 年次の机上演習で確認
- 通知テンプレート 3 種（初報、詳細、本人）を法務承認済み

**優先度**: MUST

## G7. プライバシー・バイ・デザイン

### NFR-G-DES-001: 最小化原則

**現状**: 収集時に「念のため」のデータを集めると、保護対象が膨張。

**要件達成後**: 以下の最小化を実装する。

- **収集**: 業務目的に必要な最小限のデータのみ
- **保持**: 保管期間を明示、超過は自動削除
- **利用**: 収集時の目的範囲内で利用、新用途は本人同意
- **提供**: 第三者提供は本人同意または法定要件

Phase 1b 時点でデータ収集の意図を全 tier2/tier3 テナントから棚卸し。

**崩れた時**: 不必要な個人情報を保持、漏えい時の影響拡大。

**受け入れ基準**:
- Phase 1b で最小化ガイドライン確定
- テナントオンボーディング時の収集意図レビュー
- 年次棚卸し

**優先度**: SHOULD

### NFR-G-DES-002: 同意管理

**現状**: 同意取得と管理の仕組みがないと、利用目的範囲外の処理で法令違反。

**要件達成後**: 以下で同意を管理する。

- **同意レコード**: 誰が、いつ、何について、どのバージョンの利用規約に同意したかを記録
- **撤回機能**: 本人が同意撤回可能、撤回時は該当処理を停止
- **透明性**: プライバシー通知を Backstage で公開、変更時は本人に通知
- **子供の個人情報**: 16 歳未満の場合は保護者同意（該当があれば）

**崩れた時**: 同意範囲外の処理で監督官庁指摘。

**受け入れ基準**:
- Phase 1c で同意管理機能実装
- プライバシー通知の法務レビュー
- 同意撤回率をモニタリング

**優先度**: SHOULD（該当するシステム）
