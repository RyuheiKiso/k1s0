# A. 可用性

本書は IPA 非機能要求グレード 2018 の「A. 可用性」に準拠し、k1s0 の継続性・耐障害性・災害対策・回復性の要件を定義する。採用側組織の情シスが tier1 基盤を使い続けられる水準を、採用検討で外部約束した稼働率を対外 SLA、SRE 運用上の目標値を SLO として 2 層で管理する。

## 本章の位置付け

可用性は「落ちないこと」ではなく「落ちても回復できること」の設計である。k1s0 は 採用初期 まで単一クラスタ・単一拠点の構成で運用するため、理論上の上限は構成由来で決まる。本書では構成上到達可能な水準を明示し、それを超える要求（例: 99.99% マルチリージョン）は 採用側の全社ロールアウト 以降に送る。

グレード判定は「モデルシステム②（業務時間帯運用・重大事故対策）」相当を採用する。24 × 365 の完全無停止は リリース時点〜4 ではスコープ外とし、夜間メンテナンスウィンドウを許容する。

## A カテゴリ定義（CONT / FT / DR / REC の区分）

本章は 4 カテゴリ（継続性 CONT、耐障害性 FT、災害対策 DR、回復性 REC）で要件を整理する。過去に `NFR-A-CONT-002` を「レプリケーション RPO」として参照する亡霊 ID が発生した（[01_要件ID索引.md](../80_トレーサビリティ/01_要件ID索引.md):214-221 に履歴あり）。同種の混同を防ぐため、各カテゴリの定義・典型シナリオ・識別キーワードを以下に明示する。新規要件を追加する際は、本表を参照してカテゴリを確定してから ID を採番する。

| カテゴリ | 接頭辞 | 定義 | 典型シナリオ | 識別キーワード | 他カテゴリとの境界 |
|---|---|---|---|---|---|
| **継続性** | CONT | サービスが計画どおり稼働し続ける能力。対外 SLA・degrade 稼働・影響範囲の局所化を扱う。 | 月 7.2 時間の SLA 達成、Secret 基盤障害時の 1 時間 degrade、バックエンド障害の API 単位限定 | SLA、稼働率、degrade、影響範囲、業務断の回避 | 個別コンポーネント障害の **復旧** は FT、拠点災害は DR、**訓練・手順整備** は REC に属す |
| **耐障害性** | FT | 個別コンポーネント障害が発生したときの自動復旧・フェイルオーバー能力。 | Pod 自動再起動、Kafka 3 ブローカー化、PostgreSQL レプリカ昇格、ztunnel Liveness Probe | レプリカ、フェイルオーバー、Probe、Liveness、Readiness、自動復旧 | **対外約束** は CONT、**拠点全壊** は DR、**訓練の整備度** は REC に属す |
| **災害対策** | DR | クラスタ全壊・拠点災害時の事業継続。RTO（目標復旧時間）と RPO（目標復旧時点）で定量化する。 | RTO 4 時間、PostgreSQL WAL アーカイブ RPO 数秒、etcd 日次バックアップ RPO 24 時間、採用側の全社ロールアウト マルチリージョン | RTO、RPO、バックアップ、アーカイブ、PITR、拠点、全壊 | **個別 Pod・ブローカー障害** は FT、**訓練実施頻度** は REC、**対外 SLA** は CONT に属す |
| **回復性** | REC | 障害復旧プロセスの成熟度。訓練・Runbook・属人化排除で「復旧できる状態」を維持する。 | 年 2 回の復旧訓練、Runbook 15 本整備、鮮度レビュー、単独実行可能性 | 訓練、Runbook、ポストモーテム、鮮度、単独実行、属人化 | **復旧手段の実装** は FT/DR、**稼働率そのもの** は CONT に属す |

ID 採番時の判断フローは「①まず対外約束か内部機構かを判定（対外 = CONT、内部 = FT/DR/REC） → ②内部機構なら範囲を判定（単一コンポーネント = FT、クラスタ全壊 = DR、プロセス成熟度 = REC）」の 2 段階とする。判断に迷う場合は Product Council に諮り、本表に具体例を追記する。

## SLA / SLO 2 層運用の前提

本章が定める稼働率は対外約束（SLA）であり、SRE が日常運用で追う内部目標（SLO）は [I_SLI_SLO_エラーバジェット.md](I_SLI_SLO_エラーバジェット.md) で別管理する。SLA は契約違反時にペナルティを伴う水準、SLO は SLA より厳しく設定した内部アラート閾値である。両者の数値を混同すると、運用判断（リリース凍結やエスカレーション）が発散するため、本章内の数値は SLA 系、I_SLI_SLO 内の数値は SLO 系として読む。

### 数値根拠表（SLA / SLO の導出）

採用検討で約束した SLA と SRE 内部目標 SLO を「どの数式からどう導いたか」を以下に明示する。採用検討レビュアーが「99% はどう計算したのか」「6.5 時間のマージンは何に充てるのか」を問うた際、本表を根拠として直接提示できる。本表が I_SLI_SLO_エラーバジェット.md の SLO 値と整合していなければ、リリース凍結ルールが発散するため、両ファイル間の数値一致は四半期レビューで必ず検証する。

| 指標 | 水準 | 月間換算 | 計算根拠 | 管轄ファイル | 出典・備考 |
|---|---|---|---|---|---|
| tier1 全般 SLA（対外約束） | 稼働率 99% | 720 h × 1% = **月間 7.2 時間のダウンタイム許容** | 24 h × 30 日 = 720 h／年間 8,760 h × 1% = 87.6 h ≒ 月 7.2 h | 本章 NFR-A-CONT-001 | [企画書.md](../../01_企画/企画書.md):307、[試算前提.md](../../01_企画/企画書.md) |
| tier1 全般 SLO（内部目標） | 稼働率 99.9% | 720 h × 0.1% = **月間 43 分のダウンタイム許容** | 720 h × 0.1% = 0.72 h = 43.2 min | [I_SLI_SLO_エラーバジェット.md](I_SLI_SLO_エラーバジェット.md) NFR-I-SLO-001 | Google SRE Workbook の Elite 水準 |
| マージン（SLA − SLO） | 0.9 ポイント | **月間 約 6.5 時間**（7.2 h − 43 min） | 7.2 h − 0.72 h = 6.48 h | 本章 + I_SLI_SLO | 体質改善の余地として確保 |

### マージン 6.5 時間の層別割当（内部運用指針）

月間 6.5 時間のマージンは、SLO 違反（月 43 分）が累積しても SLA 違反（月 7.2 時間）に到達する前に体質改善を起動するための緩衝である。層別割当の目安は以下のとおり（実測後に再確定）。

| 割当先レイヤ | 想定消費時間/月 | 消費シナリオ | 消費時の是正契機 |
|---|---|---|---|
| tier1 ファサード Pod 再起動 | 60 分 | OOMKill / Rolling Update 遅延 | NFR-A-FT-001 の再起動時間チューニング |
| Valkey / PostgreSQL フェイルオーバー | 120 分 | プライマリ停止時の昇格時間 | NFR-A-FT-003 の HA 化（運用蓄積後） |
| etcd バックアップ復元 | 30 分 | コンポーネント状態の軽微復元 | NFR-A-DR-003 のバックアップ間隔見直し |
| Kafka ブローカー復旧 | 60 分 | 採用初期 の 1 ブローカー構成での手動復旧 | NFR-A-FT-002 の 3 ブローカー化（採用側のマルチクラスタ移行時） |
| ネットワーク・DNS 揺らぎ | 30 分 | ztunnel 再起動、CoreDNS 再起動 | NFR-A-FT-004 の ztunnel POC 結果反映 |
| 計画外メンテ（セキュリティパッチ等） | 90 分 | CVE 対応の緊急パッチ適用 | NFR-E-MON-003 のパッチ適用時間最適化 |
| 予備（未分類障害） | 30 分 | ポストモーテム対象 | OR-INC-004 のルートコーズ特定 |
| **合計** | **420 分 = 7.0 時間** | - | - |

合計は 420 分で月間マージン 390 分（6.5 時間）を若干超過する配分となっているが、これは「全レイヤで同時に消費することはない」前提の保険値である。実際の消費はこのうち 50〜60% に収まる想定（試算前提の社内既存システム実績に基づく）。リリース時点 パイロット 1Q 運用後に実測値で再配分する。

数値対応の全体像は以下のとおり。

- **tier1 全般 SLA**: 24h 基準 99%（月間ダウンタイム許容 7.2 時間）— 本章 NFR-A-CONT-001
- **tier1 全般 SLO**: 24h 基準 99.9%（月間ダウンタイム許容 43 分）— I_SLI_SLO NFR-I-SLO-001 以下
- **マージン**: 月 約 6.5 時間（SLA − SLO）を体質改善の余地として運用

## A1. 継続性

### NFR-A-CONT-001: tier1 対外 SLA 稼働率 99%

**現状**: 採用側組織の情シスの既存業務システムは自社運用で稼働率 98〜99% 程度。商用 PaaS（OpenShift 等）採用時は SLA 99.9% の約束が可能だが、5 年 TCO で 5 億円超となり申請を通過しない。

**要件達成後**: k1s0 tier1 基盤は 24 時間 365 日基準で稼働率 99% を維持する対外 SLA を提示する。月間ダウンタイム許容は 720 時間 × 1% = 7.2 時間（計画メンテナンスを除く）。本数値は企画書 04_定量試算/試算前提.md で外部約束した水準であり、採用初期 パイロット 1Q 運用後に実測値で再確定する。SRE が日常追うのは 0.9 ポイント厳しい SLO（99.9%、I_SLI_SLO 参照）であり、SLA 違反前に SLO 違反で体質改善が起動する。

**崩れた時**: 長時間停止が発生し、tier2/tier3 の業務アプリ提供が滞る。SLA 違反は対テナント契約（BC-LGL-005）のペナルティ条項を発動させる。情シスマネージャが経営層に説明責任を負い、次年度の継続投資判断で逆風となる。

**受け入れ基準**:
- 月次稼働率 99% 以上（24h 基準、月間ダウンタイム 7.2 時間以内）を 3 か月連続で達成
- 測定は Prometheus で tier1 公開 API の 5xx / mTLS 失敗 / タイムアウトを集計（SLI 定義は NFR-I-SLI-001 と整合）
- Dead Man's Switch（FR-T1-TELEMETRY-003 連携）で監視基盤自身の稼働を 24 時間監視
- 未達月は翌月初にポストモーテムを実施、是正計画を ADR に記録

**優先度**: MUST（採用検討提示 SLA）

### NFR-A-CONT-002: OpenBao 障害時の degrade 稼働

**現状**: Secret 管理基盤が停止すると、tier2/tier3 アプリが PostgreSQL 動的 Secret を取得できず、新規接続が軒並みエラーになる。

**要件達成後**: OpenBao 障害時、tier2 は直近取得した動的 Secret の TTL 残時間（最大 1 時間）は継続稼働可能。新規接続は Secret キャッシュ満了後に失敗するが、既存接続の保持で数十分の degrade 稼働を確保する。

**崩れた時**: Secret 基盤障害が即座に全アプリ停止に波及し、RTO 目標（4 時間）の達成が困難になる。

**受け入れ基準**:
- tier2 SDK で動的 Secret のキャッシュ（TTL 残時間内）を維持
- キャッシュ失効時は `K1s0Error.SecretsUnavailable` を返し、ログに記録
- リリース時点で SDK 実装 / カオス試験で検証

**優先度**: SHOULD

### NFR-A-CONT-003: Valkey / Kafka / PostgreSQL 障害時の影響限定

**現状**: 単一データバックエンドの障害が全 tier2 アプリに波及し、業務断が発生する。

**要件達成後**: 各データバックエンドの障害は対応する tier1 API のみに影響を限定する。例: Valkey 障害は State API のみ失敗、Kafka 障害は PubSub API のみ失敗。tier2 アプリは該当 API 以外は稼働継続できる設計とする。tier1 SDK は失敗通知を明示的に返し、tier2 側でフォールバックを選択可能にする。

**崩れた時**: 1 つのバックエンド障害で tier2 全アプリが停止し、障害の影響範囲が不必要に拡大する。

**受け入れ基準**:
- 各 tier1 API の失敗は対応バックエンドのみに限定（tier1 ファサード側で切り分け）
- Grafana ダッシュボードで「どの tier1 API が degrade しているか」が可視化
- カオス試験（リリース時点）で各バックエンド単独障害を再現

**優先度**: MUST

### NFR-A-CONT-004: Workflow 実行の永続化と再開性

**現状**: Workflow 実行状態の永続化・再開性が個別実装だと、Pod 再起動やローリングデプロイのたびに業務フローが中断し、手動リカバリが必要になる。

**要件達成後**: Workflow API の短期実行（Dapr Workflow）と長期実行（Temporal）は、いずれも実行状態をバックエンド永続化し、Worker Pod の再起動・再配置・デプロイ後も `workflow_id` を維持したまま途中ステップから再開できる。再開性は「最後に永続化済みのステップから安全に継続できること」と定義し、Replay / Resume 検証を CI と演習で継続確認する。

**崩れた時**: 採用検討・督促・在庫引当などの長時間フローが途中で失われ、補償トランザクションと業務オペレーションが破綻する。影響は単一 API 障害ではなく業務状態不整合として長く残存する。

**受け入れ基準**:
- Dapr Workflow は状態を State Store に、Temporal は Workflow History を永続ストアに保持し、インメモリ専用状態を持たない
- 実行中 Workflow の Worker Pod を再起動しても、60 秒以内に同一 `workflow_id` で再開する
- リリース時点 のカオス試験または DR 演習で「実行中 Workflow 中断 → Pod 再起動 → 途中再開」を検証する

**優先度**: MUST

### NFR-A-CONT-005: Collector 障害時の tier2 稼働継続

**現状**: OpenTelemetry Collector が停止すると、計装送信のリトライやバッファ膨張が tier2 業務処理へ波及し、観測基盤障害がアプリ停止へ連鎖するおそれがある。

**要件達成後**: Log / Telemetry の送信経路は非ブロッキングとし、Collector 障害時も tier2 業務処理は継続する。計装データは一定時間ローカルバッファに保持し、あふれた分のみ破棄する。破棄は可視化対象とし、業務処理の可用性を観測性より優先する。

**崩れた時**: Collector 障害が全 tier2 アプリのレイテンシ悪化や OOM を誘発し、障害調査のための基盤が新たな障害源になる。

**受け入れ基準**:
- tier2 SDK / Agent は Collector 送信を非ブロッキングとし、Collector 無応答時も業務 API を待たせない
- Collector が 60 秒停止しても、tier2 業務処理 p99 への追加影響は 10ms 以下に抑える
- バッファあふれ時は破棄件数をメトリクス化し、Collector 復旧後は通常送信へ自動復帰する

**優先度**: MUST

### NFR-A-CONT-006: flagd 障害時のデフォルト値フォールバック

**現状**: Feature Flag バックエンド flagd が停止すると、tier1 Feature API の評価失敗がそのまま tier2 / tier3 の業務処理失敗に波及する。

**要件達成後**: flagd 応答失敗時は tier1 Feature API がローカルキャッシュまたは呼出側が指定した `default_value` を返し、`K1s0Error` を返さずに業務処理を継続させる。Feature Flag は可用性を下げる依存先ではなく、障害時の安全側制御点として振る舞う。

**崩れた時**: Feature Flag 基盤の障害が業務 API の停止点となり、段階ロールアウトや kill switch のための仕組みが逆に障害ドメインを拡大する。

**受け入れ基準**:
- `Evaluate` / `EvaluateAll` は flagd の timeout / unavailable 時に `default_value` を返し、業務呼出を失敗させない
- 直近キャッシュ値の利用は TTL 30 秒以内に限定し、キャッシュ不在時は必ず `default_value` にフォールバックする
- flagd 応答失敗率とフォールバック発生件数を Telemetry で可視化し、閾値超過時は SEV2 で通知する

**優先度**: MUST

## A2. 耐障害性

### NFR-A-FT-001: 単一 Pod 障害時の自動復旧

**現状**: k8s 単一 Pod の障害は ReplicaSet で自動復旧するが、復旧中の in-flight リクエストが失敗する。

**要件達成後**: tier1 API サーバ Pod は最小 2 レプリカで稼働し、1 Pod 障害時も他 Pod がトラフィックを吸収する。k8s の Liveness Probe / Readiness Probe で 30 秒以内に障害検知、新 Pod 起動は 60 秒以内。in-flight リクエストは Dapr / Istio の retry で最大 3 回リトライされる。

**崩れた時**: Pod 障害時のトラフィック断が p99 を直撃し、SLO 違反が累積する。

**受け入れ基準**:
- tier1 API サーバの最小レプリカ数 2（運用蓄積後）
- Liveness / Readiness の閾値設定を Component で統一
- カオス試験（リリース時点、月次）で Pod ランダム kill を実施、稼働率への影響を計測

**優先度**: MUST

### NFR-A-FT-002: Kafka ブローカー 1 台障害時の継続性

**現状**: Kafka 単一ブローカーの障害で PubSub API が停止する。

**要件達成後**: Kafka は 3 ブローカー構成（Strimzi KRaft モード）で運用し、1 ブローカー障害時もメッセージの publish / consume を継続する。レプリケーションファクター 3、min.insync.replicas 2 で設定する。採用側のマルチクラスタ移行時で本構成に移行、採用初期 は 1 ブローカー構成で運用（障害時は手動復旧、RTO 1 時間以内）。

**崩れた時**: 採用側のマルチクラスタ移行時で Kafka ブローカー障害が業務断に直結し、Publish 失敗がイベント駆動業務を停止させる。

**受け入れ基準**:
- 採用初期: 1 ブローカー構成、手動復旧手順 Runbook 整備
- 採用側のマルチクラスタ移行時: 3 ブローカー構成、自動フェイルオーバー確認
- カオス試験でブローカー 1 台 kill 時の publish / consume 継続を確認

**優先度**: MUST（採用側のマルチクラスタ移行時）、SHOULD（採用初期）

### NFR-A-FT-003: PostgreSQL フェイルオーバー

**現状**: PostgreSQL プライマリ障害でトランザクション中の業務データが失われる、または長時間の復旧で業務断となる。

**要件達成後**: CloudNativePG の Operator で HA 構成（プライマリ 1 + レプリカ 2）を構築。プライマリ障害時、30 秒以内にレプリカがプライマリ昇格、in-flight トランザクションは失敗するが新規接続は継続可能。リリース時点 は単一構成 / で HA 移行。

**崩れた時**: PostgreSQL 障害が業務システム全停止に波及し、復旧まで数時間を要する。

**受け入れ基準**:
- リリース時点: 単一構成、PITR（継続アーカイブバックアップ）で RPO 数秒
- リリース時点: HA 構成、フェイルオーバー 30 秒以内
- カオス試験でプライマリ強制停止時のフェイルオーバー時間を計測

**優先度**: MUST（運用蓄積後）

### NFR-A-FT-004: Istio Ambient ztunnel 障害時の Pod 通信

**現状**: ztunnel が障害停止した場合、その Node 上の Pod は mTLS 通信が不可能になる。

**要件達成後**: ztunnel は DaemonSet で各 Node に配置、Liveness Probe で自動再起動。再起動中の 1〜2 秒は通信不可になるが、Dapr / Istio のリトライで吸収する。採用後の運用拡大時で POC 実施し、挙動を確認する。

**崩れた時**: サービスメッシュ自体が SPOF となり、p99 に影響する。

**受け入れ基準**:
- ztunnel Liveness Probe での自動再起動 30 秒以内
- 採用後の運用拡大時 の Ambient POC で ztunnel kill 時の影響時間を計測
- 影響時間が 5 秒を超える場合、Envoy Gateway + Dapr-mTLS の代替案にフォールバック

**優先度**: SHOULD（採用後の運用拡大時）

## A3. 災害対策

### NFR-A-DR-001: RTO 4 時間（クラスタ全壊）

**現状**: クラスタ全壊（VM 3 台すべて停止）は確率的に低いが発生時の復旧時間が読めない。

**要件達成後**: RTO 4 時間（企画書提示値）を目標とし、以下の復旧手順を Runbook 化する。

1. 予備 VM の調達・起動（社内プール or 新規調達）: 1 時間
2. kubeadm による新規クラスタ構築: 1 時間
3. Argo CD から tier1 基盤の復元: 30 分
4. PostgreSQL PITR による業務データ復元: 1 時間
5. 動作確認と業務再開: 30 分

**崩れた時**: 長時間業務停止で社内業務全般が影響を受け、経営層への説明責任が発生する。

**受け入れ基準**:
- 復旧訓練（運用蓄積後、年 2 回）で RTO 4 時間を達成
- 手順書 Runbook-DR-001 が 15 本の Runbook のうちに含まれる
- 予備 VM の調達 SLA を調達部門と合意

**優先度**: MUST

### NFR-A-DR-002: RPO 数秒（PostgreSQL）

**現状**: PostgreSQL 障害時、失われるデータ量が不明確。

**要件達成後**: CloudNativePG の WAL アーカイブを 5 秒間隔で MinIO に転送する設定で、RPO 数秒を達成する。本値は企画書提示値と整合する。

**崩れた時**: 業務データの数分〜数時間分が失われ、業務データの再入力対応が発生する。

**受け入れ基準**:
- WAL アーカイブ間隔 5 秒以下
- 復旧訓練で PITR による RPO 数秒を確認
- MinIO 側の WAL 保管期間を 7 日以上に設定

**優先度**: MUST

### NFR-A-DR-003: RPO 24 時間（etcd）

**現状**: k8s クラスタの etcd 障害時、クラスタ状態の復元が困難。

**要件達成後**: etcd の日次バックアップを MinIO に転送、24 時間 RPO（企画書提示値）で復元可能。復元手順は Runbook 化。

**崩れた時**: クラスタ状態（Deployment、Service、ConfigMap）の再構築に手作業が必要となり、RTO 4 時間を超過する。

**受け入れ基準**:
- etcd 日次バックアップを MinIO に転送
- 復旧訓練で etcd 復元を実施
- 保管期間 30 日

**優先度**: MUST

### NFR-A-DR-004: マルチリージョン DR（採用側の全社ロールアウト）

**現状**: 単一拠点構成では拠点全壊（火災、災害）時の復旧不可能。

**要件達成後**: 採用側の全社ロールアウト で東京 ↔ 大阪のマルチリージョン DR を検討する。Argo CD マルチクラスタ管理、PostgreSQL 論理レプリケーション、Kafka MirrorMaker による非同期レプリケーション。

**崩れた時**: 拠点災害時の復旧不可能、事業継続性が限定的。

**受け入れ基準**:
- 採用側の全社ロールアウト で方針を確定、具体水準は未確定
- 現時点では参考扱い

**優先度**: WON'T（リリース時点〜4）

## A4. 回復性

### NFR-A-REC-001: バックアップ復旧訓練（年 2 回）

**現状**: 既存業務システムのバックアップ訓練は年 1 回の定期演習で、本番障害時に実際に復旧できるか毎回緊張する。

**要件達成後**: 運用蓄積後、半年に 1 回、事前通告なしに担当 SRE 1 名を抽出し、単独で復旧完了まで実行できるかを計測する。対象は PostgreSQL フルクラスタ全壊、etcd 復元、Kafka クラスタ再構築の 3 シナリオ。

**崩れた時**: 本番障害時の復旧が属人的となり、担当者不在時に業務断が長期化する。

**受け入れ基準**:
- 年 2 回の訓練実施
- 訓練結果（RTO 実績、Runbook 更新点）をポストモーテムで記録
- Runbook の更新が 1 週間以内に完了

**優先度**: MUST

### NFR-A-REC-002: Runbook 充足率

**現状**: インシデント対応手順が部門ごとにバラバラで、後輩への引継ぎが属人的。

**要件達成後**: リリース時点で 10 本 / 15 本（採用検討提示値）の Runbook を整備し、全シナリオで SRE が単独実行可能な状態とする。Runbook は Git 管理、四半期ごとに鮮度レビュー。15 本の具体的 ID・対象シナリオ・段階整備期限・実行権限は [../40_運用ライフサイクル/09_Runbook目録.md](../40_運用ライフサイクル/09_Runbook目録.md) を単一の真実源とする。

**崩れた時**: 新人 SRE の立ち上がりが遅れ、オンコール体制が少数の熟練者に過度に依存する。

**受け入れ基準**:
- リリース時点: 10 本（採用初期 の KPI）
- リリース時点: 15 本（採用初期 必達）
- 鮮度レビューで 90 日以上未更新の Runbook を検出、改訂

**優先度**: MUST
