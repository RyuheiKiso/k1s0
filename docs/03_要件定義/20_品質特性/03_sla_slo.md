# QUA-SLO: SLO / SLA 要件

本ファイルは、k1s0 プラットフォームの **内部 SLO（運用チーム合意）と顧客向け SLA（契約書に反映される数値）** を分離し、両者のエラーバジェット運用・違反時のエスカレーション・デプロイ凍結判断を定義する。SLO はチームの品質基準、SLA は法的拘束力を伴う外部約束であり、同じ数値でも扱いが異なる。

SLO は `QUA-PRF`（レイテンシ）と `QUA-AVL`（稼働率）の数値を具体的な目標値として運用に落とし込むための仕組みである。単に「p99 500ms 以下」を壁の目標にするのではなく、**達成度を毎週レビューし、未達時は自動的にデプロイを凍結する** ポリシーを採番する。これにより「いつの間にか劣化していた」事態を防ぐ。

SLA は Phase 3（本番全社運用）以降で顧客契約に反映される。Phase 1a / 1b はパイロット運用であり、顧客との SLA は内部 SLO の 1 段階緩い数値（例: 内部 SLO 99.9% に対し SLA 99.5%）で合意する。この差分が内部チームのバッファとなり、SLA 違反の発生頻度を抑える。

## 要件本体

### QUA-SLO-001: 内部 SLO の合意プロセス

- 優先度: MUST（内部品質基準の根幹。合意なしに運用は成立しない）
- Phase: Phase 1a
- 関連: `QUA-PRF-001`, `QUA-AVL-001`, 構想設計 05_SLOとエラーバジェット 2 節

現状、tier1 API の数値目標（p99 500ms、稼働率 99%）は構想設計で提案されているが、tier1 チームと infra チーム、および tier2 側の代表者による明示的合意プロセスは未整備。合意なしの目標値は形骸化し、違反しても誰も責任を問われない状態になる。

本要件が満たされた世界では、Phase 1a 開始時点で全 SLO が tier1 / infra / tier2 代表者のサインオフを経て確定し、月次 SLO レビュー会議で達成度を報告・議論する運用が成立する。結果として SLO は単なる数値ではなく、組織のコミットメントとなる。

未達時、SLO が形骸化し「数値は出しているが誰も見ていない」状態となり、違反を放置 → 顧客報告 → 信頼失墜の連鎖が発生する。

**受け入れ基準**

- SLO 確定会議（Phase 1a 開始 -2 週）で以下を合意・署名
  - tier1 公開 API 可用性 99%（Phase 1b）→ 99.5%（Phase 3）→ 99.9%（Phase 5）
  - tier1 外向き API p99 レイテンシ 500ms（`QUA-PRF-001`）
  - CI/CD パイプライン PR ビルド p95 < 10 分（構想設計 2.3 節）
- 月次 SLO レビュー会議を Phase 1a 開始と同時に発足、毎月第 1 営業日 10:00 開催
- SLO 変更には tier1 / infra / tier2 代表の三者合意が必要
- 許容違反頻度: 2 か月連続未達は SLO 引き下げ再合意の対象（構想設計 6.2 節）

**検証方法**

- SLO 合意書を Git 管理し、署名済み PDF を格納
- 月次 SLO レビュー議事録を Backstage TechDocs に保存

### QUA-SLO-002: エラーバジェット運用ルール

- 優先度: MUST（「品質 vs 速度」の定量的意思決定の根拠）
- Phase: Phase 1a（手動計算）→ Phase 1b（自動化）
- 関連: `QUA-SLO-001`, 構想設計 05_SLOとエラーバジェット 3〜5 節

現状、SLO が定まっていても、未達時に何を止め・何を継続するかの運用ルールが無ければ、場当たり的な判断で混乱する。「99% だけど 98.5%、まだ許容範囲か？」の議論が毎回発生する。

本要件が満たされた世界では、エラーバジェット残量に応じたフェーズ別運用ルール（50% 以上 = 通常 / 25〜50% = 注意 / 10〜25% = 警告 / 0% 以下 = フリーズ、構想設計 5.1 節）が自動判定され、新機能デプロイの可否が客観的に決まる。チーム内の政治的駆け引きが不要になる。

未達時、「99.5% を下回ったがリリース迫っているので強行」が繰り返され、SLO は形骸化。本番運用で顧客が先にバグを踏む。

**受け入れ基準**

- エラーバジェット計算: SLO 30 日ウィンドウ。残量 = 1 - （実エラー率 / 許容エラー率）
- 運用ポリシー（構想設計 5.1 節に準拠）:
  - 50% 以上: 通常運用（新機能デプロイ・実験的変更 OK）
  - 25〜50%: tier1 リーダー承認必須の高リスク変更制限
  - 10〜25%: バグ修正 / セキュリティパッチのみ、新機能凍結
  - 0% 以下: フリーズ（Postmortem 完了 + 残量 25% 以上回復まで）
- Phase 1a は手動計算、Phase 1b 以降 Prometheus recording rule `k1s0:tier1_api_error_budget_remaining` で自動化
- バーンレートアラート（Critical 14.4x / Warning 6x / Info 1x）をマルチウィンドウで設定（構想設計 4.2 節）

**検証方法**

- 月次 SLO レビューでバジェット残量を報告
- フリーズ判定の事例を四半期ごとに Postmortem として記録

### QUA-SLO-003: SLA 違反時の顧客補償条項

- 優先度: MUST（Phase 3 本番契約の必須条項。未達で契約不成立）
- Phase: Phase 3（本番全社）
- 関連: `QUA-SLO-001`, `QUA-AVL-001`, [`../60_事業_契約/07_legal_contract.md`](../60_事業_契約/07_legal_contract.md)

現状、Phase 1a / 1b はパイロット運用で SLA は不要。Phase 3 以降、JTC 顧客との本契約に SLA を反映する際、違反時の補償条項（サービスクレジット）が未定義。

本要件が満たされた世界では、Phase 3 開始時点で SLA 違反階層（99.5% 未満→月額 10% クレジット、99.0% 未満→月額 25% クレジット、98.0% 未満→月額 50% クレジットなど、競合 Mia-Platform や AWS SLA の業界平均を参考）が契約書に明記される。顧客は定量的な保護を得て、k1s0 チームは自主的な SLO 改善インセンティブを持つ。

未達時、SLA 補償条項が無い契約は JTC 顧客から「信用できない」として拒絶され、パイロット顧客との本契約移行が失敗する。

**受け入れ基準**

- SLA 数値は内部 SLO の 1 段階緩い設定（SLO 99.9% → SLA 99.5%）
- 補償階層（業界平均を参考）:
  - 99.0〜99.5%: 月額料金の 10% サービスクレジット
  - 98.0〜99.0%: 月額料金の 25% サービスクレジット
  - 98.0% 未満: 月額料金の 50% サービスクレジット + 解約権
- クレジット上限は月額の 100%（AWS SLA 業界標準）
- SLA 対象は tier1 公開 API のみ（tier2 / tier3 は tier2 サービス提供者の個別 SLA）
- 測定単位・地点・期間は `QUA-AVL-001` に準拠
- 計画停止は SLA 計算から除外しない（`QUA-AVL-004` と整合）

**検証方法**

- 法務レビュー済み SLA 契約テンプレートを `../60_事業_契約/` に格納
- 月次 SLO レビューで SLA 違反シミュレーション

### QUA-SLO-004: SLO 未達時のデプロイ凍結判断

- 優先度: MUST（SLO を実効化するための自動制御。手動判断に委ねると形骸化）
- Phase: Phase 1b（自動化）
- 関連: `QUA-SLO-002`, 構想設計 5.1 節

現状、エラーバジェット枯渇時のデプロイ凍結は運用ルール上規定されていても、GitOps（Argo CD）で物理的に止まる仕組みが無ければ「緊急だから例外」が繰り返される。

本要件が満たされた世界では、Argo CD の Sync が Prometheus recording rule の `error_budget_remaining < 0` を検知すると自動的に Paused 状態に遷移し、tier1 リーダーの明示的 unpause なしにデプロイが進まなくなる。結果として SLO 劣化時の無配慮なデプロイで状況を悪化させる事態が防がれる。

未達時、バジェット枯渇時にデプロイが強行され、バグデプロイで SLO が -5% さらに悪化、回復不能のインシデントに発展する。

**受け入れ基準**

- Argo CD の Pre-Sync Hook で Prometheus に問い合わせ、`k1s0:tier1_api_error_budget_remaining` < 0 の場合は sync を中断
- フリーズ解除条件（構想設計 5.2 節）:
  1. バジェット残量が 25% 以上に回復
  2. 枯渇の原因となったインシデントの Postmortem 完了
  3. Postmortem で特定された Critical アクションアイテム完了
- 緊急セキュリティパッチはフリーズ除外（tier1 リーダー + CISO 双方の承認書が必要）
- フリーズ発動・解除は監査ログ（`SEC-AUD-*`）に記録

**検証方法**

- 四半期に 1 回、意図的に SLO を悪化させフリーズ発動を検証
- Postmortem の Critical 項目完了トラッキングを Backstage で実装

### QUA-SLO-005: SLO レビューと Phase 引き上げ判断

- 優先度: SHOULD（SLO の時系列的な成熟を担保）
- Phase: Phase 1b
- 関連: `QUA-SLO-001`, 構想設計 6 節 レビュープロセス

現状、SLO は Phase 1b 99% → Phase 3 99.5% → Phase 5 99.9% と段階引き上げが設計されているが、引き上げ判断基準が定性的で「Phase 移行と同時に引き上げる」運用だと実力が伴わず即座に違反する。

本要件が満たされた世界では、SLO 引き上げは「3 か月連続でバジェット残量 80% 以上」という客観的基準（構想設計 6.2 節）で判定され、実力ベースで段階引き上げされる。結果として、引き上げ直後の即時違反が防がれる。

未達時、Phase 移行と同時に SLO を引き上げると、実力が 99% しかないのに 99.5% を宣言してしまい、月次レビューで毎月違反という不名誉な状態が続く。

**受け入れ基準**

- SLO 引き上げ条件: 3 か月連続でバジェット残量 80% 以上
- SLO 引き下げ条件: 3 か月連続でバジェット枯渇（構想設計 6.2 節）
- Phase 移行と SLO 引き上げは **別の判断**。Phase 移行だけでは SLO は自動引き上げしない
- 引き上げ判断は月次 SLO レビューで合議、四半期に 1 回の定例で確定
- tier2 / tier3 からの可用性苦情が月 3 件以上で SLO 見直し対象

**検証方法**

- 月次 SLO レビュー議事録で引き上げ判断の根拠を記録
- Phase 移行時の SLO 状態を Git タグに記録
