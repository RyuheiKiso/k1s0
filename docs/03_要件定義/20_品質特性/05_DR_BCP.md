# QUA-DRB: 災害復旧 / BCP 要件

本ファイルは、k1s0 プラットフォームの **壊滅的障害（データセンター全損 / Kafka クラスタ全損 / etcd 全損など）** に対する復旧要件を定義する。`QUA-AVL` が平常時の単一ノード障害までを扱うのに対し、本カテゴリは「複数のノード / 1 AZ / 1 リージョン全体が同時損失するような壊滅的事象」を対象とする。

JTC 基幹系プラットフォームにおいて DR / BCP は法的・契約的義務でもある。金融庁ガイドライン・FISC 安全対策基準・個人情報保護委員会のガイドライン等で「バックアップの確保」「RTO / RPO の設定」「定期的な復旧訓練」が要求される。本カテゴリはこれらの法令対応を具体的数値として採番する。

要件は 5 系統。**RPO（データ損失許容量）**、**RTO（復旧時間許容量）**、**地理的冗長の採否**、**BCP 発動基準**、**復旧訓練頻度**。前 2 者は数値目標、後 3 者はプロセス要件である。企画書 11 章の数値（PostgreSQL RPO 数秒 / RTO 15 分、クラスタ全壊 RTO 4 時間）はすべて本カテゴリで要件化する。

## 要件本体

### QUA-DRB-001: RPO（Recovery Point Objective）目標値

- 優先度: MUST（データ損失許容量はデータ種別ごとの契約必須条項）
- Phase: Phase 1b
- 関連: `QUA-AVL-003`, 構想設計 01_障害復旧とバックアップ

現状、RPO（壊滅的障害時に許容できるデータ損失時間）はデータ種別ごとに構想設計で提案されているが（PostgreSQL WAL 数秒、etcd 24 時間、Kafka レプリケーション遅延）、要件として採番されていない。顧客契約時に「データが何秒分まで失われ得るか」を数値で説明できない。

本要件が満たされた世界では、業務データ（PostgreSQL）は RPO 15 分以内、クラスタ状態（etcd）は RPO 24 時間以内、イベント（Kafka）は RPO レプリケーション遅延（通常ミリ秒）以内と明文化される。結果として顧客との SLA 合意で「何秒分のデータ損失まで許容されるか」を契約条項に書き込める。

未達時、顧客への説明で「バックアップはあります」としか言えず、具体的な損失時間が不明なため金融庁監査で「BCP が不十分」と判定されるリスク。

**受け入れ基準**

- 測定単位: 直近成功バックアップ時点から壊滅時点までの経過時間（分）
- 測定地点: CloudNativePG の `last_wal_archive_time`、etcd snapshot の最新作成時刻、Kafka replication lag
- 測定期間: 壊滅的障害発生時点
- 閾値:
  - PostgreSQL（業務データ）: RPO 15 分以内（企画書 11 章「数秒」より保守的に設定、WAL 転送遅延の最悪ケース想定）
  - etcd（クラスタ状態）: RPO 24 時間以内（日次 snapshot、k8s マニフェストは Git なので etcd 完全消失でも GitOps 再構築可能）
  - Kafka（イベント）: RPO 0 秒目標（replication.factor=3、min.insync.replicas=2 で acks=all）
  - 監査ログ: RPO 0 秒（書き込み同期前に commit 返さない。`SEC-AUD-*` で別途採番）
- 許容違反頻度: RPO 超過は即時 Critical インシデント扱い、年 0 件目標

**検証方法**

- 月次で「最新バックアップ時点」と現在時刻の差分を Prometheus で監視
- 年 2 回の復旧訓練（`QUA-DRB-005`）で実測 RPO を計測

### QUA-DRB-002: RTO（Recovery Time Objective）目標値

- 優先度: MUST（復旧時間許容量は業務継続の直接指標）
- Phase: Phase 1b
- 関連: `QUA-AVL-001`, 構想設計 01_障害復旧とバックアップ

現状、RTO は企画書 11 章で「PostgreSQL: 15 分、クラスタ全壊: 4 時間」と提案されているが、手順 Runbook の整備と訓練による実測未検証。

本要件が満たされた世界では、PostgreSQL プライマリ故障時は 15 分以内、Kafka ブローカー 1 台故障時は 5 分以内、クラスタ全壊時は 4 時間以内で復旧することが Runbook と年 2 回の訓練で保証される。結果として金融庁監査や顧客 BCP 審査で「実測値付きの RTO」を提示できる。

未達時、実測されていない RTO は「机上計算」とみなされ、JTC 顧客の BCP 審査で「根拠なし」として信用を失う。

**受け入れ基準**

- 測定単位: 障害発生検知時点から、tier1 API リクエストが正常応答に復帰するまでの分数
- 測定地点: 復旧訓練での Runbook 実行時間記録、または実際の障害 Postmortem の時刻記録
- 閾値:
  - PostgreSQL プライマリ故障: RTO 15 分以内（企画書 11 章）。CloudNativePG のレプリカ昇格は 30 秒、アプリ接続切替 5 分で設計
  - Kafka ブローカー 1 台故障: RTO 5 分以内（構想設計 2.3 節、KRaft ブローカー再起動時間）
  - etcd 全損: RTO 1 時間（etcd snapshot リストア + k8s 整合性確認）
  - クラスタ全壊（ノード全損）: RTO 4 時間（Git + IaC からの再構築、企画書 11 章）
  - リージョン全損: RTO 24 時間（別リージョンへの手動フェイルオーバー、`QUA-DRB-003` 参照）
- 許容違反頻度: 年 0 件目標、訓練で RTO 超過が発生したら翌月までに Runbook 改善
- RTO は検知から復旧までの全工程（検知 + オンコール招集 + 診断 + 復旧実行）を含む

**検証方法**

- 年 2 回の DR 訓練（`QUA-DRB-005`）で各シナリオの復旧時間を実測
- 実障害 Postmortem で RTO 達成 / 未達を記録

### QUA-DRB-003: 地理的冗長の採否判断

- 優先度: SHOULD（Phase 1 / 2 はオンプレ単一ロケーション。Phase 3 で地理冗長を採否判断）
- Phase: Phase 3
- 関連: `QUA-DRB-002`, 構想設計 07_リージョン戦略とデータレジデンシー

現状、k1s0 は Phase 1 / 2 でオンプレ単一データセンター運用を前提としている。地理的冗長（別 DC / 別リージョン）は BCP 強化の切り札だが、インフラ費用が 1.5〜2 倍、運用複雑度が 3 倍になる。Phase 3 以降で採否を判断する事項。

本要件が満たされた世界では、Phase 3 開始前にビジネス要件（顧客の BCP 水準要求、法令要件、DC 停止時の業務影響額）を評価し、地理冗長を採用するか・別の BCP 手段（コールドスタンバイ / 復旧代替拠点契約）で済ませるかを定量根拠付きで決定する。

未達時、Phase 3 時点で地理冗長の採否判断が未実施だと、JTC 大手顧客（金融・保険・通信）との契約で BCP 水準不足として失注する。

**受け入れ基準**

- Phase 3 開始 -6 か月時点で「地理冗長採否評価レポート」を作成
- 評価項目: 1) 顧客の BCP 要求水準、2) 法令要件（金融庁ガイドライン 4.2 節等）、3) インフラ費用差分、4) 運用複雑度増、5) RTO / RPO 短縮効果
- 採用判断基準: 顧客契約の 30% 以上が BCP 水準「ティア 1（地理冗長必須）」を要求する場合は採用
- 採用時の RPO / RTO（Phase 3 の地理冗長適用時）:
  - アクティブ-アクティブ: RPO 1 分 / RTO 15 分
  - アクティブ-スタンバイ: RPO 15 分 / RTO 4 時間
- 不採用時は代替 BCP 策（復旧代替拠点契約、クラウドバーストなど）を定義

**検証方法**

- 評価レポートを `../02_構想設計/01_アーキテクチャ/02_可用性と信頼性/07_リージョン戦略とデータレジデンシー.md` に反映
- Phase 3 開始時点で採否決定が ADR として記録されている

### QUA-DRB-004: BCP 発動基準と Runbook

- 優先度: MUST（発動判断の遅れが最大のコスト。基準の明文化で即時発動）
- Phase: Phase 1b
- 関連: `QUA-DRB-001`, `QUA-DRB-002`, 構想設計 06_壊滅的障害シナリオ

現状、どのレベルの障害で BCP 発動（代替拠点への切替 / フリーズ解除権限委譲 / 経営陣への即時報告など）を行うかの客観基準が未整備。発動判断に 2 時間かかると、それだけで RTO 4 時間の半分を消費する。

本要件が満たされた世界では、BCP 発動基準が 3 段階（Level 1 単一コンポーネント障害 / Level 2 クラスタ全体障害 / Level 3 リージョン全損）で定義され、Level 2 以上は自動的に SRE リーダー + CTO + CISO の 3 者連絡・30 分以内に発動判断が行われる。結果として RTO 消費が発動判断で食われない。

未達時、障害発生から発動判断までに時間を消費し、RTO 超過で顧客 SLA 違反が発生する。

**受け入れ基準**

- BCP 発動レベル定義（構想設計 06_壊滅的障害シナリオ に準拠）:
  - Level 1: 単一コンポーネント障害（tier1 API 部分停止）— オンコール SRE が対応、BCP 未発動
  - Level 2: クラスタ全体障害（kube-apiserver 到達不能 / etcd 過半数損失）— SRE リーダー + CTO 報告、30 分以内に発動判断
  - Level 3: リージョン全損 / データセンター全損— CEO + CTO + CISO 3 者連絡、即時発動
- Runbook を壊滅的障害シナリオ別に整備（Kafka 全損 / etcd 全損 / Istio 全損 / PostgreSQL 全損 / DC 全損）
- Runbook は Backstage TechDocs に格納、四半期レビュー必須
- BCP 発動・解除は監査ログ（`SEC-AUD-*`）に記録

**検証方法**

- Runbook の存在と最終更新日を CI で検証
- 年 2 回の DR 訓練で Runbook 実行結果をレビュー

### QUA-DRB-005: 復旧訓練の頻度

- 優先度: MUST（Runbook は実行されないと信頼できない。訓練頻度が品質を決める）
- Phase: Phase 1b
- 関連: `QUA-DRB-002`, `QUA-DRB-004`

現状、Runbook は整備されても実行訓練が行われず、実障害時に「Runbook 通りにやったが復旧しなかった」事態が発生する。金融庁ガイドラインでも年 1 回以上の BCP 訓練が推奨されている。

本要件が満たされた世界では、年 2 回（業界平均 1 回を上回る頻度で設定、JTC 基幹系の保守性重視）の DR 訓練が staging 環境で実施され、各 Runbook の実効性が実測される。訓練で RTO 超過が判明したら翌月までに Runbook 改善するサイクルが回る。

未達時、Runbook が陳腐化し実障害時に動かない。結果として RTO 4 時間目標が実測 8 時間になり、顧客 SLA 違反・事業影響が拡大。

**受け入れ基準**

- 実施頻度: 年 2 回（上半期・下半期各 1 回、業界平均 1 回を上回る設定）
- 実施環境: staging 環境を本番相当の構成に揃え、本番データ相当量のダミーデータで実施
- 訓練シナリオ: Kafka 全損 / etcd 全損 / PostgreSQL プライマリ故障 / クラスタ全壊 / DC 全損 の 5 シナリオから毎回 2 つ以上
- 訓練結果レポート: RTO 実測値、RPO 実測値、Runbook 改善アクション
- 許容違反: 年 1 回未満の実施は監査指摘対象

**検証方法**

- 訓練結果レポートを Backstage TechDocs に保管
- 監査部門による年次レビュー

### QUA-DRB-006: バックアップの完全性検証

- 優先度: MUST（バックアップがあっても壊れていたら意味がない）
- Phase: Phase 1a
- 関連: `QUA-DRB-001`, 構想設計 01_障害復旧とバックアップ

現状、バックアップは日次で取得されるが、「バックアップから実際に復旧できるか」の検証が未実施。バックアップファイルの破損・暗号化キー紛失などで復旧失敗するリスクがある。

本要件が満たされた世界では、毎週 1 回、直近バックアップから staging 環境への自動リストア検証が実行され、整合性が確認される。結果として「バックアップはあるが復旧できない」事態が予防される。

未達時、実障害時に初めてバックアップの欠損に気付き、RTO どころか復旧不能に陥る。

**受け入れ基準**

- 測定単位: バックアップからのリストア成功 / 失敗（週次）
- 測定地点: staging 環境への CronJob 実行ログ
- 頻度: 週次（毎週月曜 03:00 など業務時間外）
- 対象: PostgreSQL 本番バックアップを staging に復元、データ件数 + checksum を比較
- 許容違反頻度: 月 1 件までは Warning、2 件連続失敗で Critical（バックアップ設計の見直し対象）

**検証方法**

- 週次自動リストアジョブの成功率を Prometheus で監視
- 四半期に 1 回、復元データの業務整合性を手動検証
