# QUA-ACC: アクセシビリティ要件

本ファイルは、k1s0 プラットフォームが提供する **エンドユーザー向け UI（配信ポータル / Backstage / tier3 業務アプリの雛形部分）** のアクセシビリティ要件を定義する。JIS X 8341-3:2016（高齢者・障害者等配慮設計指針—情報通信における機器、ソフトウェア及びサービス）の **レベル AA** 相当を下限とする。

JTC（日本の伝統的大企業）の情シス部門が全社配信する業務アプリは、法令上は総務省「みんなの公共サイト運用ガイドライン」や「障害者差別解消法」に基づく合理的配慮を求められる。加えて、大手顧客の多くが「アクセシビリティ対応済み」を RFP の必須条件として挙げるようになっている。アクセシビリティを満たさない配信ポータル・Backstage は、それ単独で契約失注要因になり得る。

本カテゴリは 4 系統で構成する。**JIS X 8341-3 AA 相当の適合**、**キーボード操作完全対応**、**スクリーンリーダー互換性**、**カラーコントラスト基準**。いずれも数値または標準準拠で採番する。`CLAUDE.md` 規約上、表の羅列は禁止のため、要件ごとに散文と検証方法を明示する。

---

## 前提

- JIS X 8341-3:2016 レベル AA 準拠（WCAG 2.0 AA に対応）
- [`../60_事業_契約/00_UX.md`](../60_事業_契約/00_UX.md) UX 方針
- `COM-GLO-007` 配信ポータルの定義
- 配信ポータル / Backstage / tier3 雛形 UI の 3 つが本カテゴリの対象

---

## 要件本体

### QUA-ACC-001: JIS X 8341-3 AA 相当の適合

- 優先度: MUST（大手 JTC 顧客 RFP の必須要件。未達で契約失注）
- Phase: Phase 1b
- 関連: `COM-GLO-007`, `BIZ-GOV-*`

現状、配信ポータル・Backstage・tier3 雛形のアクセシビリティは検証されておらず、「そのうち対応する」状態。大手 JTC 顧客（金融・保険・官公庁・地方自治体）は RFP でアクセシビリティ適合を必須条件として要求する傾向が強まっており、未対応だと初期スクリーニングで失注する。

本要件が満たされた世界では、k1s0 が提供する全エンドユーザー向け UI が JIS X 8341-3 AA 相当を満たし、第三者監査レポートを提示できる。結果として大手顧客の RFP スクリーニングを通過し、競合（OpenShift / Tanzu）と同等の条件で評価フェーズに進める。

未達時、RFP のアクセシビリティ項目で「不適合」とマークされ、技術評価に進む前に脱落。事業計画の「年間 2 社獲得」の上位顧客枠（売上 70% 占有）を失う。

**受け入れ基準**

- 適合基準: JIS X 8341-3:2016 レベル AA（WCAG 2.0 AA 相当、達成基準 38 項目中 AA までの全 13 項目）
- 対象 UI:
  - 配信ポータル（`COM-GLO-007`）
  - Backstage（開発者ポータル）
  - tier3 雛形 CLI が生成する UI スケルトン
- 測定単位: Axe DevTools または Deque Lighthouse の自動監査での Critical / Serious 違反件数
- 測定期間: PR 毎の CI 実行、四半期に 1 回の第三者監査
- 閾値: 自動監査での Critical 違反 0 件、Serious 違反 0 件
- 許容違反頻度: 検出された違反は 2 週間以内に修正（法的リスクの伴う違反は即時 hotfix）
- 第三者監査: 年 1 回、JIS X 8341-3 認証機関または同等の専門会社（ミツエーリンクス / インフォアクシア 等の国内事業者）

**検証方法**

- GitHub Actions で `axe-core` / `pa11y` の自動スキャンを PR 毎実行
- 年 1 回の第三者監査レポートを Backstage TechDocs に保管
- 個別 RFP の精査チェックリストを雛形化し、営業・法務が参照

### QUA-ACC-002: キーボード操作完全対応

- 優先度: MUST（視覚障害者・運動障害者の利用可能性の基盤）
- Phase: Phase 1b
- 関連: `QUA-ACC-001`

現状、配信ポータル・Backstage ともに React/Vue 等のモダンフロントエンドだが、独自実装の UI 部品でキーボードフォーカスが飛ばせない / Tab 順が崩れている箇所が混在する可能性がある。マウス操作前提の UI は視覚障害者（スクリーンリーダー併用）と運動障害者の両方を排除する。

本要件が満たされた世界では、全画面の全操作がキーボードのみで完結し、視覚障害者・運動障害者・パワーユーザーが等しく業務を完遂できる。結果として JIS X 8341-3 達成基準 2.1.1（キーボード）に適合する。

未達時、キーボード非対応箇所が発生すると、JIS X 8341-3 の必須項目で不適合となり、法的義務のある公共機関顧客で納入不能になる。

**受け入れ基準**

- 全インタラクティブ要素（ボタン / リンク / フォーム / モーダル / ドロップダウン）が Tab キー・Enter キー・Space キー・矢印キーで操作可能
- フォーカスリングが視覚的に明示される（`outline: none` の一律適用禁止、カスタムフォーカススタイルを適用）
- Tab 順が視覚順序と論理的に一致する（`tabindex` の不用意な指定禁止、自然な DOM 順を維持）
- キーボードトラップが存在しない（モーダルなどで Tab 循環が閉じないこと）
- ショートカットキーの競合検出（ブラウザ標準ショートカット・スクリーンリーダーショートカットとの衝突チェック）
- 測定単位: Axe の `keyboard-accessible` ルール違反件数 0

**検証方法**

- 自動テスト: Playwright での Tab 順序テスト、フォーカスリング可視性テスト
- 手動テスト: マウスを外した状態で全業務フローを完遂するテストを四半期実施

### QUA-ACC-003: スクリーンリーダー互換性

- 優先度: MUST（視覚障害者の業務参加を確保。適切な ARIA 属性で大部分が解決する）
- Phase: Phase 1b
- 関連: `QUA-ACC-001`

現状、配信ポータル・Backstage の UI 部品で ARIA 属性（role / aria-label / aria-describedby 等）の付与が不徹底であり、スクリーンリーダー（NVDA / JAWS / VoiceOver / TalkBack）で「ボタン」と認識されず「クリック可能な要素」として読み上げられるなどの問題が発生し得る。

本要件が満たされた世界では、主要スクリーンリーダー 3 種（NVDA + JAWS + VoiceOver）で全画面・全操作が読み上げられ、視覚障害者が配信ポータルからアプリインストール、Backstage での申請などを自立的に行える。結果として JIS X 8341-3 の達成基準 4.1.2（名前・役割・値）に適合する。

未達時、公共機関顧客（地方自治体・官公庁）への納入で「アクセシビリティ不適合」として最終段階で失注。

**受け入れ基準**

- ARIA 属性の適切な付与（role / aria-label / aria-labelledby / aria-describedby / aria-live / aria-expanded 等）
- ランドマーク構造（`<header>` / `<nav>` / `<main>` / `<aside>` / `<footer>`）が全ページで明示
- 動的コンテンツ変更（モーダル表示 / トースト通知 / 非同期ロード）で `aria-live` リージョンによる読み上げトリガが設定
- 画像・アイコンに適切な代替テキスト（装飾目的は `alt=""` / 意味を持つ画像は説明的 alt）
- フォーム入力には `<label>` が必ず関連付け
- 互換性検証: NVDA（Windows / Chrome）、JAWS（Windows / Chrome）、VoiceOver（macOS Safari / iOS Safari）の 3 種で基本フロー読み上げ成立
- 許容違反頻度: Critical 違反は翌スプリントまでに修正

**検証方法**

- Axe DevTools の `aria-*` ルール違反を CI で検出
- 四半期に 1 回、実際のスクリーンリーダーでの操作動画を録画しレビュー

### QUA-ACC-004: カラーコントラスト基準

- 優先度: MUST（色覚異常者・低視力者の可読性を保証）
- Phase: Phase 1b
- 関連: `QUA-ACC-001`

現状、UI カラーパレットは「ブランドカラー優先」で決められており、文字と背景のコントラスト比が JIS X 8341-3 AA 基準（通常テキスト 4.5:1、大きいテキスト 3:1）に達していない可能性がある。

本要件が満たされた世界では、全画面のテキスト・UI 要素がコントラスト比 AA 基準を満たし、色覚異常者（日本人男性の約 5%）や低視力者でも可読な表示が保証される。ダークモード対応時も同基準を維持する。

未達時、JIS X 8341-3 の達成基準 1.4.3（コントラスト、最低限）に不適合で、法令準拠ラベルを取得できない。

**受け入れ基準**

- 通常テキスト（18pt 未満、または太字 14pt 未満）: コントラスト比 4.5:1 以上
- 大きいテキスト（18pt 以上、または太字 14pt 以上）: コントラスト比 3:1 以上
- UI 要素（アイコン / ボタン境界 / フォーカスリング等）: コントラスト比 3:1 以上
- 測定ツール: Chrome DevTools の Contrast 機能 / Stark / axe-core
- 色のみに依存した情報伝達を禁止（エラー表示は赤色 + アイコン + テキストの 3 重冗長）
- ダークモード対応時も同基準を満たす
- 許容違反頻度: Critical 違反は翌スプリント修正、ブランドガイド改訂時は事前 UX レビュー必須

**検証方法**

- CI で全画面のコントラスト比を自動チェック（pa11y + custom rule）
- ブランドガイドに基準値付きカラーパレットを定義し、UX レビューで都度確認

### QUA-ACC-005: 多言語化とアクセシビリティの両立

- 優先度: SHOULD（Phase 1 は日本語のみ、Phase 2 以降で英語対応時に顕在化）
- Phase: Phase 2
- 関連: `QUA-ACC-003`, `BIZ-I18N-*`

現状、Phase 1 は日本語のみ対応だが、Phase 2 以降の多言語化（英語）時にスクリーンリーダーでの言語切替（`lang` 属性）が適切に設定されていないと誤読上げが発生する。

本要件が満たされた世界では、多言語切替時に HTML の `lang` 属性が動的更新され、スクリーンリーダーが適切な発音で読み上げる。結果として英語話者の視覚障害者も業務遂行可能になる。

未達時、英語コンテンツを日本語発音で読み上げられ、英語話者の視覚障害者が業務遂行不能。

**受け入れ基準**

- HTML `<html lang="...">` が現在の表示言語を反映
- 部分的に言語が異なる箇所は `<span lang="...">` で明示
- Phase 2 開始時点で英語版の JIS X 8341-3 AA 相当（WCAG 2.0 AA）適合を確認
- 許容違反頻度: 多言語追加時に言語属性設定を PR チェックリストで必須化

**検証方法**

- Axe の `html-has-lang` / `html-lang-valid` ルール CI 検証
- スクリーンリーダーでの多言語読み上げ検証を半年に 1 回
