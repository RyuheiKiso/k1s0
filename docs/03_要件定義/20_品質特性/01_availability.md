# QUA-AVL: 可用性要件

本ファイルは、k1s0 プラットフォームの **稼働率・冗長構成・単一障害点排除・計画停止の扱い** を数値で定義する。企画書 11 章で宣言した「可用性 99%（業務時間帯）」は Phase 1a の暫定値であり、Phase 3 以降で 99.9% への段階引き上げが合意されている（構想設計 04_非機能とデータ 3.1 節）。本カテゴリは、各 Phase の稼働率目標・冗長構成・計画停止の事前告知ルールを採番し、SLO（`QUA-SLO-*`）との整合を保つ。

稼働率は「サービスがリクエストを受け付け、正しいレスポンスを返している時間の割合」と定義する。単にプロセスが生きているかではなく、**HTTP 5xx / タイムアウト / 誤応答を失敗としてカウントした成功率** を SLI とする。計画メンテナンスを稼働率の計算から除外しない点も重要な合意であり、「計画停止も障害予算から引く」ことで停止の乱発を抑制する。

本カテゴリは `QUA-DRB`（DR/BCP）と対になる。可用性要件は「平常時にどこまで落ちない設計をするか」、DR/BCP は「壊滅的障害時の復旧時間」を扱う。両者の境界は、単一ノード障害までを `QUA-AVL`、データセンター全損・Kafka 全損・etcd 全損を `QUA-DRB` で扱う。

## 要件本体

### QUA-AVL-001: tier1 公開 API の月次稼働率

- 優先度: MUST（SLA 合意の基盤。未達で顧客補償・稟議再審議が発生）
- Phase: Phase 1b
- 関連: `QUA-SLO-001`, `COM-RSK-001`, 企画書 11 章

現状、tier1 API の稼働率は設計目標 99%（月間許容停止 7.2 時間）のみ宣言され、実運用での SLI 測定基盤が未整備。Phase 1a で SLI を計測する基盤が無いため「稼働率 99%」が達成されたかの客観判定ができない。

本要件が満たされた世界では、tier1 外向き API の月次稼働率が Phase 1b で 99%、Phase 3 で 99.5%、Phase 5（本番全社）で 99.9% と段階引き上げされ、各 Phase の開始条件として前 Phase の実績が 2 か月連続で達成されていることが求められる。顧客との SLA 合意はこの SLO に連動する。

未達時、Phase 1b パイロット運用で稼働率 99% を割り、パイロット顧客が離脱 → 稟議当初の「年間 2 社獲得」事業計画が破綻する。

**受け入れ基準**

- 測定単位: 成功レスポンス数 / 全レスポンス数（HTTP 5xx と 60 秒超タイムアウトを失敗とみなす）
- 測定地点: tier1 外向き API（`k1s0.*`）。Envoy Gateway 通過後の L7 ログから集計
- 測定期間: 月次（暦月）、30 日スライディングウィンドウも併用
- 閾値: Phase 1b 99.0%（月間許容停止 7.2 時間、構想設計 4 節の MVP 基準）、Phase 3 99.5%（3.6 時間、競合 Mia-Platform 99.5% 契約と同等水準）、Phase 5 99.9%（43 分、JTC 基幹系業界平均）
- 許容違反頻度: 各 Phase で 2 か月連続未達は SLO 引き下げ再合意の対象
- 計画メンテナンスは稼働率計算から除外しない（構想設計 3.3 節に準拠）

**検証方法**

- Prometheus recording rule `k1s0:tier1_api_availability_30d`
- 月次 SLO レビュー会議（`QUA-SLO-001`）で報告

### QUA-AVL-002: コントロールプレーン冗長化（3 ノード以上）

- 優先度: MUST（単一障害点排除は稓働率 99% 達成の必須条件）
- Phase: Phase 1a
- 関連: `QUA-AVL-001`, `ARC-INF-*`

現状、k8s コントロールプレーン（kube-apiserver / etcd / scheduler / controller-manager）の冗長化構成は構想設計上「3 ノード推奨」とされているが、要件化されていない。稟議図面上は 3 ノード前提で描画されているが、MVP-0 段階で 1 ノードでの稼働を許容する記述もあり、運用上の誤解リスクがある。

本要件が満たされた世界では、MVP-0 承認時点で既に 3 ノード構成が必達となり、1 ノード障害でも API 応答が継続する。etcd のクォーラム（過半数）が 2/3 で維持されるため、単一ノード損失で書き込みが停止しない。

未達時、コントロールプレーンが 1 ノード構成のまま Phase 1a が承認されると、Phase 1b 初日のノード故障で全クラスタ停止、信用失墜と再設計コスト数人月が発生する。

**受け入れ基準**

- 測定単位: コントロールプレーン Pod の Ready 状態ノード数
- 測定地点: `kubectl get nodes -l node-role.kubernetes.io/control-plane` の Ready 数
- 測定期間: 常時（Prometheus で 1 分間隔監視）
- 閾値: Ready ノード 3 以上。1 ノード Down 時も Ready 2 以上を維持
- 許容違反頻度: 計画メンテナンスを除き、Ready 2 未満は Critical アラート即時発火
- ノードは別ラック / 別電源系統に配置（構想設計 01_障害復旧とバックアップ 1 節の設計原則）

**検証方法**

- Prometheus `up{job="kube-apiserver"}` の count アラート
- Chaos Mesh で四半期に 1 回、1 ノード障害注入訓練

### QUA-AVL-003: データストアのレプリケーション

- 優先度: MUST（ステートフルコンポーネントの単一障害点排除は企業基盤の要件）
- Phase: Phase 1a
- 関連: `QUA-AVL-001`, `QUA-DRB-001`, 構想設計 01_障害復旧とバックアップ

現状、PostgreSQL / Kafka / Valkey / MinIO といったステートフルコンポーネントは構想設計で冗長化が推奨されているが、要件として明文化されていない。「コスト都合で単一インスタンス」の抜け道が残っている。

本要件が満たされた世界では、全ステートフルコンポーネントが実時間レプリケーションで冗長化され、プライマリ障害時のレプリカ昇格によってデータ欠損ゼロの自動フェイルオーバーが実現する。結果として tier1 API の稼働率 99.9%（Phase 5）が物理的に達成可能になる。

未達時、PostgreSQL 単一インスタンス構成でディスク故障時に tier1 全停止、RTO 15 分どころか数時間の停止となり、`QUA-DRB-002` の 4 時間 RTO を達成しても `QUA-AVL-001` の月次稼働率が崩壊する。

**受け入れ基準**

- PostgreSQL: CloudNativePG で 1 プライマリ + 2 レプリカ（同期 1 / 非同期 1）。レプリカ遅延は非同期側で p99 5 秒以内
- Kafka: replication.factor=3、min.insync.replicas=2（acks=all でプロデュース時の 2 レプリカ ACK 担保）
- Valkey: 3 ノードのクラスタモードまたは 1 プライマリ + 2 レプリカ
- MinIO: Erasure Coding 4+2（読み書きに 4 ディスク必要、2 ディスク損失まで復旧可能）
- 測定期間: 常時（Prometheus で 1 分間隔監視）
- 許容違反頻度: 単一レプリカ損失は 5 分以内に Warning、15 分以内に復旧または再レプリカ生成

**検証方法**

- 四半期に 1 回、各データストアのプライマリ Kill → フェイルオーバー時間計測
- レプリケーション遅延を Prometheus で常時監視

### QUA-AVL-004: 計画停止の事前告知と窓

- 優先度: MUST（顧客との運用合意の基礎。未達で突発停止が頻発し信頼崩壊）
- Phase: Phase 1b
- 関連: `QUA-SLO-001`, 構想設計 04_非機能とデータ 3.3 節

現状、計画メンテナンスの告知ルールは構想設計 3.3 節で「月 1 回、平日夜間 22:00〜翌 6:00」と示されているが、告知期間と告知経路が未定義。突発的なメンテナンスで tier2 バッチ処理が巻き込まれる可能性がある。

本要件が満たされた世界では、計画停止は 48 時間前までに配信ポータル・メール・Backstage の 3 経路で告知され、tier2 / tier3 が自身のバッチ計画を調整できる。計画停止は稓働率計算から除外しない（構想設計 3.3 節合意）ため、停止時間も SLO 予算から引かれ、停止の乱発は自動的に抑制される。

未達時、告知なしの停止で tier3 業務アプリの夜間バッチが 2 時間中断、翌朝の締め処理が間に合わず経理部門が手作業修正を強いられる。

**受け入れ基準**

- 計画停止は 48 時間前まで（業界平均の 72 時間を短縮、JTC 情シスの運用リズムに合わせて 2 営業日前ルールを採用）に告知
- 告知経路: 配信ポータル通知 + メール + Backstage の少なくとも 2 経路以上
- メンテナンスウィンドウ: 月 1 回、平日夜間 22:00〜翌 6:00（構想設計 3.3 節）
- 緊急メンテナンス（セキュリティパッチ等）は 4 時間前告知を許容、ただし月次 1 回まで
- 計画停止時間も稼働率 SLI 計算に含める

**検証方法**

- 配信ポータルの告知履歴と実停止時刻の突合を月次監査
- SLI 計算に計画停止が含まれていることを Prometheus の `up` メトリクスで検証

### QUA-AVL-005: PodDisruptionBudget とローリングアップデート

- 優先度: MUST（ローリング更新時の無停止を担保。未達でデプロイのたびに SLO 消費）
- Phase: Phase 1a
- 関連: `QUA-AVL-001`, 構想設計 3.2 節

現状、tier1 Pod のローリングアップデート時に PDB が未設定だと、Deployment 更新時に一時的に 0 Pod 状態となりリクエストが失敗する可能性がある。

本要件が満たされた世界では、tier1 / tier2 の主要サービスに PDB が設定され、ローリング更新時も常に replicas-1 以上が稼働、無停止デプロイが保証される。結果として月 4 回の tier1 デプロイによる稼働率消費がゼロに抑えられる。

未達時、デプロイ毎に 5 秒〜30 秒の瞬断が発生し、月 4 回 × 30 秒 = 2 分の稼働率消費、SLO 予算の 0.5% を食う。

**受け入れ基準**

- tier1 Deployment: `minAvailable: replicas - 1`（最低 replicas-1 を常時稼働）
- tier2 主要サービス: `minAvailable: 1`（最低 1 Pod 稼働）
- rollingUpdate: `maxUnavailable: 0`, `maxSurge: 1`（旧 Pod 終了前に新 Pod 起動）
- HTTP Readiness Probe の failureThreshold を考慮（平均起動時間 + 10 秒の余裕）
- 許容違反頻度: デプロイ時の瞬断 0 を目標、例外時も 1 秒以内

**検証方法**

- Argo CD デプロイ時に Synthetic Monitor（k6 cloud）で連続リクエストを流し、5xx 発生 0 を確認
- `kubectl get poddisruptionbudget` で全 Deployment の PDB 設定を CI で検証

### QUA-AVL-006: グレースフルデグラデーション

- 優先度: SHOULD（tier1 部分障害時も業務継続。Phase 1a は必要最小限、Phase 1b で強化）
- Phase: Phase 1b
- 関連: 構想設計 03_グレースフルデグラデーション

現状、tier1 の部分障害（Kafka 全損など）時の挙動は「呼び出し全失敗」が既定で、部分機能での業務継続の余地が無い。

本要件が満たされた世界では、Kafka 障害時は同期 API のみ継続（PubSub は失敗通知）、Valkey 障害時は State は PostgreSQL フォールバック、といった部分縮退が tier2 から観測可能な形で提供され、業務停止が最小化される。

未達時、1 コンポーネント障害で全 API 停止、tier2 業務アプリが全機能停止、業務影響が最大化する。

**受け入れ基準**

- tier1 Dapr ファサードは各バックエンド別にヘルスチェックを持ち、障害時は `k1s0_api_unavailable{api="..."}` メトリクスを 1 に設定
- 縮退時の `k1s0.*` API は 503 Service Unavailable + Retry-After ヘッダを返す（一律のタイムアウトではない）
- 縮退対象と継続対象の組み合わせが構想設計 03_グレースフルデグラデーション と整合

**検証方法**

- Chaos Mesh でバックエンド単体障害を注入し、tier1 API の挙動を検証
- 縮退メトリクスが Grafana で観測可能であることを E2E テストで確認
