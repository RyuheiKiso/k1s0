# QUA-PRF: パフォーマンス要件

本ファイルは、k1s0 プラットフォームの **レイテンシ / スループット / リソース効率** を数値で定義する。企画書 11 章で宣言した「tier1 API p99 < 500ms」「可用性 99%」などは机上積算の見込み値であり、前回の技術査読で High リスク（`COM-RSK-001`）として「実測未検証」が指摘された。本カテゴリはその積み残しを解消するため、各要件に **測定単位・測定地点・測定期間・許容違反頻度** の 4 点を埋め込み、実装前に合意された閾値として採番する。

要件は大きく 3 系統に分かれる。第一に tier2/tier3 が経験する tier1 外向き API のレイテンシ、第二に tier1 内部の Dapr ファサード・Rust 自作領域・gRPC 相互呼び出しのレイテンシ、第三にスループット・負荷耐性・リソース効率である。外向きと内向きで閾値を分けることで、どのレイヤが p99 を押し上げているかを帰属分析できる。

本カテゴリの前提は「**p99 を実測で合意する**」ことである。Phase 1a（MVP-0）で負荷試験を必須化し（`QUA-PRF-005`）、実測結果を `QUA-SLO-*` に取り込む。未達時は企画書 11 章に記された段階的フォールバック（目標値再設定 / 監査非同期化 / API 別 SLO）で処理する。

## 要件本体

### QUA-PRF-001: tier1 外向き API の p99 レイテンシ

- 優先度: MUST（稟議通過時に企画書 11 章で宣言した数値の根幹。未達で SLA 違反と顧客補償が発生）
- Phase: Phase 1a
- 関連: `COM-RSK-001`, `QUA-PRF-005`, `QUA-SLO-001`, 企画書 11 章

現状、tier1 の外向き API レイテンシは **業務ロジック 200ms + Dapr サイドカー 80ms + OTel インストルメンテーション 20ms + ハッシュチェーン監査 50ms + ネットワーク・DB 150ms = 500ms** の机上積算のみで、実機での p99 は測定されていない。この状態ではパイロット顧客との SLO 合意も受け入れ試験も実施できない。

本要件が満たされた世界では、tier2 / tier3 から見た tier1 `k1s0.*` API の p99 レイテンシが 5 分間ウィンドウで 500ms 以下であり、業務アプリが「業務ロジック 200ms + tier1 往復 300ms 以内」という前提でユーザー体験を設計できる。結果として tier2 バックオフィスのバッチ処理や tier3 のオンライン入力画面が 1 秒以内のレスポンスを維持できる。

未達時、tier2 のバッチ締め処理が翌営業日に食い込み業務停止を引き起こす。さらに SLA 違反の顧客補償（1 月分のライセンス相当、`QUA-SLO-003`）が発生し、Phase 2 本番運用開始前に SLO の再設定 → 事業計画の見直しへ波及する。

**受け入れ基準**

- 測定単位: HTTP レスポンス完了までの往復時間（ミリ秒）、クライアント側（tier2 呼び出し元）計測
- 測定地点: tier2 から見た tier1 `k1s0.State.*` / `k1s0.PubSub.*` / `k1s0.Decision.*` / `k1s0.Service.Invoke` のクライアント側。Dapr サイドカー + Istio ambient + ハッシュチェーン監査を含むフルパス
- 測定期間: 直近 30 日、5 分間ウィンドウの p99 を Prometheus recording rule で算出
- 閾値: 500ms 以下（企画書 11 章の机上積算上限）
- 許容違反頻度: 月次 0.1% 以内（= 7.2 時間の SLO 予算の範囲内で吸収可）
- API カテゴリ別の内訳は `QUA-PRF-002` に準拠（高速 / 標準 / 重量の 3 区分）

**検証方法**

- MVP-0 で k6 による負荷試験スクリプトを実装し、CI 毎 PR で 100 RPS × 10 分のベースライン比較を実施（`QUA-PRF-005`）
- MVP-1a で Grafana Pyroscope + eBPF により各レイヤのレイテンシ寄与を分解測定し、机上積算の妥当性を再確認
- 本番では Grafana Tempo の 100% トレースで外れ値を継続監視

### QUA-PRF-002: API カテゴリ別のレイテンシ階層

- 優先度: MUST（全 API を一律 500ms では高速 API の体感が損なわれる。階層化は UX 上必須）
- Phase: Phase 1a
- 関連: `QUA-PRF-001`, 構想設計 04_非機能とデータ 2.1 節

現状、tier1 API はすべて「500ms 以下」の一律基準で語られており、`k1s0.Log.*` のような fire-and-forget API にも `k1s0.Workflow.Start` のような重量 API にも同じ閾値が当てられている。これは高速 API の性能余力を無駄にし、重量 API の必要コストを過小評価する。

本要件が満たされた世界では、tier1 API は 3 階層（高速 / 標準 / 重量）に分類され、それぞれ独立した p99 閾値で監視される。高速 API がキャッシュヒットで 50ms を出している事実が、重量 API の 500ms に隠されて見えなくなる問題が解消される。

未達時、一律 500ms を基準とすると、`k1s0.Log` が 300ms かかっても「基準内」と判定され、実はライブラリの劣化で本来の 5ms から 60 倍遅くなっている事実を検知できない。

**受け入れ基準**

- 測定単位: HTTP p99 レイテンシ（ミリ秒）、tier2 クライアント側計測
- 測定地点: API 名別のラベル付きメトリクス `k1s0_api_call_duration_seconds{api="..."}`
- 測定期間: 直近 30 日、5 分間ウィンドウの p99
- 閾値（構想設計 04_非機能とデータ 2.1 節より、MVP 時点）:
  - 高速 API（`k1s0.Log.*` / `k1s0.Telemetry.*` / `k1s0.Auth.ValidateToken`）: p99 100ms 以下
  - 標準 API（`k1s0.State.*` / `k1s0.PubSub.Publish` / `k1s0.Decision.Evaluate` / `k1s0.Secrets.Get` / `k1s0.Audit.Record`）: p99 500ms 以下
  - 重量 API（`k1s0.Workflow.Start` / `k1s0.Service.Invoke`）: p99 1000ms 以下
- 許容違反頻度: 月次 0.1% 以内（各階層独立）

**検証方法**

- Prometheus recording rule `k1s0:tier1_api_latency_p99_5m` を API 別ラベル付きで集計
- 階層ごとに Grafana ダッシュボード「SLO Overview」のパネルを分割

### QUA-PRF-003: tier1 内部 gRPC の p99 レイテンシ

- 優先度: MUST（外向き 500ms を達成するには内部の余力が必要。未達で `QUA-PRF-001` が連鎖崩壊）
- Phase: Phase 1a
- 関連: `QUA-PRF-001`, `ARC-T1-*`

現状、tier1 内部（Dapr ファサード Go ⇔ 自作領域 Rust）の gRPC 通信のレイテンシは定義されていない。外向きが 500ms のうち業務ロジック 200ms + ネットワーク・DB 150ms を除いた **残り 150ms で Dapr サイドカー・OTel・監査・内部 gRPC をすべて吸収** する必要があるが、内部 gRPC の予算配分が無いため積算が検証不能。

本要件が満たされた世界では、tier1 内部の gRPC 往復が p99 50ms 以下に保たれ、外向き 500ms に対して十分な余力（残 450ms）を残す。結果として業務ロジックや DB アクセスに 400ms 近く使えるため、ZEN Engine のルール評価（ARC-RUL）や監査ログのハッシュチェーンが揺れても全体予算を食い潰さない。

未達時、tier1 内部で 200ms を超える呼び出しが発生し、外向き 500ms の達成が机上で不可能になる。Dapr ファサードと ZEN Engine 間の gRPC が遅いという原因が外部計測では見えず、帰属分析で初めて発見される。

**受け入れ基準**

- 測定単位: gRPC unary リクエストの RTT（ミリ秒）、サーバ側 Pod 内計測
- 測定地点: Go Dapr ファサード ⇔ Rust 自作領域（ZEN Engine / crypto）間の社内 gRPC。同一ノード上の localhost 通信
- 測定期間: 直近 7 日、1 分間ウィンドウの p99
- 閾値: 50ms 以下
- 許容違反頻度: 週次 0.5% 以内（短期ウィンドウ許容は tier1 外向きより緩く設定。内部通信は即時 tier2 影響ではないため）
- 内部 gRPC のオーバーヘッド（Protobuf エンコード / OTel 計装）を除いた純正往復は p99 20ms 以下

**検証方法**

- Grafana Pyroscope の CPU プロファイルで内部 gRPC の CPU 時間を計測
- eBPF `bpftrace` で TCP RTT 分布を採取しベースラインと比較

### QUA-PRF-004: ZEN Engine ルール評価レイテンシ

- 優先度: MUST（ルール評価は全 `k1s0.Decision.*` 呼び出しの中核。未達で Decision API 全体が崩壊）
- Phase: Phase 1a
- 関連: `QUA-PRF-002`, `COM-GLO-006`, `ARC-RUL-*`

現状、ZEN Engine の JDM ルール評価レイテンシは Rust バインディング経由で in-process 実行する前提だが、実測値はない。GoRules の公式ベンチマークはあるが、k1s0 固有の JDM スキーマ（JTC 固有の稟議・承認フロー）でのレイテンシは未検証。

本要件が満たされた世界では、ZEN Engine のルール評価が p99 3ms 以下で完了し、`k1s0.Decision.Evaluate` 全体 100ms 予算のうち 97ms を業務ロジック・gRPC・監査に充当できる。結果として複雑な承認フロー（20 段階条件分岐）でも 1 リクエスト内で評価完結する。

未達時、ZEN Engine の評価が遅延すると Decision API が 100ms 予算を突破し、標準 API の p99 500ms も連鎖違反する。代替として Rego（OPA）への移行が発生するが、ルール DSL の書き換えコストが数人月発生する。

**受け入れ基準**

- 測定単位: ZEN Engine `evaluate()` 関数の呼び出し時間（ミリ秒）、Rust 関数内計測
- 測定地点: tier1 Rust 自作領域内、ZEN Engine Rust バインディングの `evaluate()` 直後
- 測定期間: 直近 7 日、1 分間ウィンドウの p99
- 閾値: 3ms 以下（業界一般の in-process ルール評価エンジンのベンチマーク平均 2ms を参考に、JDM のスキーマ検証オーバーヘッドを加味して 3ms に設定）
- 許容違反頻度: 週次 0.5% 以内
- ルールセットの規模は標準 JDM（100 ノード、10 条件分岐）での測定

**検証方法**

- ZEN Engine 評価関数に OTel スパンを張り、Tempo で p99 を計測
- ベンチマーク CI で代表的な JDM スキーマを使い PR 毎に回帰検知

### QUA-PRF-005: 負荷試験の必須化（MVP-0 時点）

- 優先度: MUST（実測なしに p99 500ms を宣言することは査読指摘事項。MVP-0 承認の前提条件）
- Phase: Phase 1a
- 関連: `COM-RSK-001`, `QUA-PRF-001`

現状、tier1 の p99 500ms は机上積算のみで、Phase 1a（MVP-0）承認時点での実測が存在しない。前回の技術査読で「実装前の予測値」であることが High リスクとして指摘された。

本要件が満たされた世界では、MVP-0 段階で k6 / JMeter による負荷試験が CI に組み込まれ、tier1 外向き API の p99 が実測で 500ms 以下であることが PR ごとに検証される。結果として MVP-0 承認ゲートで「実測値を伴った性能合意」が成立し、稟議説得力が担保される。

未達時、MVP-0 が机上値だけで承認され、Phase 1b パイロット開始時に実測すると 800ms〜1200ms が判明、SLO 再設定と開発計画の差し戻しが発生する。

**受け入れ基準**

- 負荷試験ツールは k6（MVP）または JMeter（DMBOK 準拠が必要な場合）を使う
- 試験シナリオ: 100 RPS × 10 分（MVP）、500 RPS × 10 分（Phase 2）
- RPS 根拠: 構想設計 04_非機能とデータ 1 節「MVP ピーク 50 RPS」の 2 倍をベースラインとした。本番 Phase 2 の 500 RPS は構想設計 Phase 5 想定値
- 受け入れゲート: p99 500ms 以下（外向き）、p99 50ms 以下（内部 gRPC）、エラー率 0.1% 以下
- 負荷試験のシナリオは Git 管理、結果は 30 日保持
- MVP-0 承認時点で少なくとも 1 回の負荷試験レポートを提示

**検証方法**

- GitHub Actions で nightly に負荷試験ジョブを実行
- 結果を Grafana ダッシュボードで週次レビュー

### QUA-PRF-006: スループットと同時接続

- 優先度: SHOULD（Phase 1a は MVP 50 RPS で十分。Phase 2 以降の本番運用で必達）
- Phase: Phase 1b
- 関連: `QUA-PRF-005`, 構想設計 04_非機能とデータ 1 節

現状、tier1 がどこまでの同時接続数・RPS に耐えるかの上限は未定義。水平スケール（HPA）で伸ばす設計だが、単一 Pod の処理能力の実測値が無い。

本要件が満たされた世界では、tier1 の 1 Pod あたり 50 RPS を安定処理できることが実測され、HPA の ReplicaSet 計算根拠が数値で保証される。Phase 2 の 500 RPS は 10 Pod で水平分散できる。

未達時、HPA が設定通り動かず、ピーク時に Pod が足りずレイテンシ急増、もしくは過剰プロビジョニングでコスト肥大化する。

**受け入れ基準**

- 測定単位: 1 Pod あたりの持続 RPS（5 分間平均）
- 測定地点: tier1 Dapr ファサード Pod
- 閾値: MVP 時点 50 RPS / Pod（CPU 使用率 70% で飽和する上限）、Phase 2 で 50 RPS / Pod を維持しつつ 10 Pod 水平分散で 500 RPS
- 許容違反頻度: Pod 単体のスパイク 5% 許容、クラスタ全体の RPS はスパイク不許容（HPA で吸収）

**検証方法**

- k6 で単 Pod 負荷試験（HPA 無効化）を実施し CPU / RPS 飽和点を計測
- HPA 有効の全体試験でスパイク時の Pod 増加応答時間を計測

### QUA-PRF-007: リソース効率

- 優先度: SHOULD（TCO と密接。Phase 2 の 3,000 名規模で顕在化）
- Phase: Phase 1c
- 関連: `BIZ-COST-*`, 企画書 13 章 TCO 比較

現状、tier1 Pod の CPU / メモリ要求値は設計時の勘で決まっており、実測に基づくリソース最適化が行われていない。過剰割り当てで TCO が膨らむリスクがある。

本要件が満たされた世界では、tier1 の CPU request が実測の p95 使用量から算出され、ノード集約率が上がり 5 年 TCO 3.68 億円の想定を維持できる。

未達時、Pod リソース過剰設定で Phase 2 時点のノード数が想定 5 から 10 へ倍増し、インフラコストが年間 1,000 万円級で上振れする。

**受け入れ基準**

- 測定単位: Pod 単位の CPU request / limit（millicore）、メモリ request / limit（MiB）
- 測定地点: Prometheus `container_cpu_usage_seconds_total` / `container_memory_working_set_bytes`
- 閾値: request は p95 使用量の 1.2 倍以内、limit は p99 使用量の 1.5 倍以内
- 許容違反頻度: 四半期に 1 回のリソースチューニング時点で逸脱が 10% 以内

**検証方法**

- VPA（Vertical Pod Autoscaler）の推奨値と手動設定の乖離を月次レビュー
- コスト部門と連携した四半期 TCO レポート
