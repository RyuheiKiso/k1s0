# QUA-TNT: テナント分離要件

本ファイルは、k1s0 プラットフォーム上で複数テナント（JTC 顧客企業、または 1 社内の複数事業部）が同一クラスタを共用する際の **性能隔離・リソース上限・ノイジーネイバー対策** を定義する。セキュリティ観点の分離（データ機密性・認可境界）は `30_セキュリティ_データ/04_data.md` と `01_IAM.md` で別途扱い、本カテゴリはあくまで「性能干渉」に限定する。

JTC 基幹系プラットフォームでテナント分離が崩れるとどうなるか。例えば経理部門のバッチ処理が暴走し CPU を占有、その結果として営業部門のオンライン入力画面が 3 秒待ちになる。テナント A の Kafka プロデュースが突出しブローカーのディスク IO を飽和させ、テナント B の PubSub 遅延が 10 倍悪化する。これらは `QUA-PRF` と `QUA-AVL` の SLO を形式的には満たしていても、実際には **隣の住人が居ない時だけ達成できる数値** という脆弱な状態を意味する。本カテゴリは、他テナントの挙動に依存しない安定性を保証する。

要件は 4 系統で構成される。CPU / メモリ上限（`QUA-TNT-001`）、ネットワーク帯域の公平割り当て（`QUA-TNT-002`）、ストレージ IOPS 上限（`QUA-TNT-003`）、テナント跨ぎ性能干渉の検知（`QUA-TNT-004`）である。前者 3 つは予防策、最後の 1 つは検知と抑制の仕組みである。

## 要件本体

### QUA-TNT-001: テナント別 CPU / メモリ上限

- 優先度: MUST（ノイジーネイバー対策の基本。未達で 1 テナントの暴走が全体に波及）
- Phase: Phase 1b（マルチテナント化開始）
- 関連: `QUA-PRF-001`, `QUA-AVL-001`

現状、tier2 / tier3 の Pod は個別 ResourceRequest / Limit を持つが、テナント全体としての CPU / メモリ上限が定義されていない。1 テナントが無制限に Pod を増やすと、他テナントの Pod が優先度劣化で Evict される可能性がある。

本要件が満たされた世界では、各テナントが Kubernetes の Namespace + ResourceQuota で CPU / メモリ上限を付与され、テナント A の暴走（バッチの想定外スケール）がテナント B のリソースを侵食しない。結果として営業部門のオンライン業務が経理部門の月次バッチの影響を受けない安定性が実現する。

未達時、1 テナントの Pod 増殖で他テナントの Pod が OOMKilled / Evict され、無関係な業務が停止。原因調査に数日かかり、マルチテナント化そのものの信頼性が否定される。

**受け入れ基準**

- 測定単位: Namespace 単位の `requests.cpu` / `limits.cpu` / `requests.memory` / `limits.memory`（ResourceQuota）
- 測定地点: `kube-apiserver` の ResourceQuota 状態
- 測定期間: 常時（Prometheus で 1 分間隔）
- 閾値: テナント契約プランに応じた上限（例: Small プラン 10 core / 40 GiB、Medium プラン 40 core / 160 GiB、Large プラン 100 core / 400 GiB）
- プラン別閾値の根拠: 業界標準の Kubernetes マルチテナント構成（OpenShift Projects / GKE Namespace Quota）のベストプラクティスを踏襲し、Small は業務時間 100 req/s × 0.1 core = 10 core で算出
- 許容違反頻度: 上限 90% 到達で Warning、100% 到達で Pod 作成拒否（即時）

**検証方法**

- ResourceQuota を GitOps で管理し、手動変更を CI で拒否
- 四半期に 1 回、意図的に上限到達を発生させフェイルセーフ動作を確認

### QUA-TNT-002: ネットワーク帯域の公平割り当て

- 優先度: MUST（Istio Ambient / ノード NIC の輻輳によるテナント干渉を防ぐ）
- Phase: Phase 1c
- 関連: `QUA-PRF-001`, Istio Ambient 設計

現状、テナント間のネットワーク通信は Istio Ambient Mesh で論理分離されているが、物理帯域（ノード NIC 10Gbps）は共有されている。1 テナントが大容量データ転送を行うとノード NIC が飽和し、他テナントのレイテンシが悪化する。

本要件が満たされた世界では、各テナントに Kubernetes NetworkPolicy と組み合わせた egress 帯域制限（tc または Cilium BandwidthManager）が適用され、ノード NIC 10Gbps が契約プラン比で公平分割される。結果として大量データ転送テナントがいても、他テナントのレイテンシ p99 が設定値内に収まる。

未達時、テナント A のログ転送暴走でノード NIC が 95% 飽和、テナント B の tier1 API 往復が 50ms → 500ms に悪化、SLO 違反が群発する。

**受け入れ基準**

- 測定単位: Pod 単位の egress Mbps（kubernetes.io/egress-bandwidth アノテーション）
- 測定地点: Cilium BandwidthManager または tc qdisc の統計
- 閾値: Small プラン 100 Mbps / Pod、Medium 500 Mbps / Pod、Large 2 Gbps / Pod
- 根拠: ノード 10 Gbps NIC を前提に、ノード上に Small プラン 100 Pod が共存できる設計（10 Gbps / 100 Pod = 100 Mbps）
- 許容違反頻度: ノード NIC 使用率 80% 超過が 5 分継続で Warning

**検証方法**

- iperf3 による帯域テストを四半期に 1 回実施
- Prometheus `container_network_transmit_bytes_total` を Pod 別に監視

### QUA-TNT-003: ストレージ IOPS 上限

- 優先度: MUST（PostgreSQL / MinIO の IO 飽和によるテナント干渉を防ぐ）
- Phase: Phase 1c
- 関連: `QUA-PRF-001`, Longhorn / MinIO 設計

現状、ストレージ（Longhorn 経由の PV）は全テナント共有で、IOPS 上限が未設定。1 テナントの大量 Write 処理が Longhorn の 3 レプリカ同期を飽和させ、他テナントの DB クエリが遅延する可能性がある。

本要件が満たされた世界では、PVC 単位で IOPS 上限がプラン別に設定され（Longhorn の `diskPressurePercentage` と組み合わせ）、他テナントのストレージ性能に干渉しない。結果として、テナント A のインポートジョブ（大量 INSERT）がテナント B の SELECT レイテンシを悪化させない。

未達時、共有ストレージの IOPS 飽和で PostgreSQL 読み取り遅延が p99 5ms → 500ms に 100 倍悪化、`QUA-PRF-003` 違反。

**受け入れ基準**

- 測定単位: PVC 単位の IOPS（Read / Write 別）
- 測定地点: Longhorn の per-volume metrics `longhorn_volume_actual_size_bytes` と `node_disk_io_time_seconds_total`
- 閾値: Small プラン 500 IOPS / PVC、Medium 2000 IOPS / PVC、Large 8000 IOPS / PVC
- 根拠: オンプレ SSD の典型 IOPS 20,000 を前提に、ノード上に Small 40 PVC 共存可能な設計
- 許容違反頻度: IOPS 上限 90% 到達で Warning、ノード全体 IOPS 使用率 80% 超過で Node Pressure 発動

**検証方法**

- fio による IO ベンチマークを四半期に 1 回実施
- Longhorn メトリクスを Prometheus で監視

### QUA-TNT-004: テナント跨ぎ性能干渉の検知

- 優先度: MUST（予防策の取りこぼしを検知。干渉の事実を客観計測）
- Phase: Phase 1b
- 関連: `QUA-TNT-001`, `QUA-TNT-002`, `QUA-TNT-003`

現状、予防策（CPU / 帯域 / IOPS 上限）を設定しても、実運用で想定外の干渉が発生する可能性がある（例: カーネルレベルのコンテキストスイッチ増加、共有キャッシュの汚染など）。干渉を検知する仕組みが無いと、顧客クレームで初めて気付くことになる。

本要件が満たされた世界では、テナント別に p99 レイテンシの時系列を比較監視し、テナント A の負荷スパイクとテナント B のレイテンシ悪化に相関（Pearson 係数 > 0.7）が検出されるとアラートが発火する。結果として、予防策の抜け漏れを実測で発見できる。

未達時、干渉が発生していることにチームが気付かず、顧客 B から「最近遅い」の苦情が来てから対応を始める。原因特定に数日かかり、その間も干渉が継続する。

**受け入れ基準**

- 測定単位: テナント別 p99 レイテンシ（5 分間ウィンドウ、`tenant_id` ラベル付き）
- 測定地点: tier1 API メトリクス `k1s0_api_call_duration_seconds{tenant_id="..."}`
- 検知ルール: テナント X の RPS スパイク（前週同時刻比 +300%）とテナント Y の p99 悪化（前週同時刻比 +50%）が同時発生で Warning
- 許容違反頻度: 月次 2 件以内（それ以上は予防策見直し）

**検証方法**

- Prometheus recording rule でテナント別 SLI を算出
- Chaos Mesh で片方テナントに意図的に負荷を注入し、他方への影響を四半期測定

### QUA-TNT-005: テナント単位のキャパシティプランニング

- 優先度: SHOULD（事前のキャパ計画でテナント追加時の干渉リスクを低減）
- Phase: Phase 2
- 関連: `QUA-TNT-001`, `BIZ-COST-*`

現状、新規テナント追加時のキャパシティ影響評価プロセスが無く、「空いているノードに追加」という場当たり運用になっている。テナント数が 5 を超えると干渉リスクが顕在化する。

本要件が満たされた世界では、新規テナント追加前に既存テナントの SLO バジェット消費予測が計算され、既存テナントの SLO を悪化させない確証を得てから追加される。結果として、テナント追加のたびに既存顧客の体験が劣化する事態が防がれる。

未達時、テナント追加で既存テナントの p99 が 20% 悪化、SLA 違反でクレジット発動。

**受け入れ基準**

- 新規テナント追加前に「キャパシティ影響評価レポート」を作成
- 評価指標: 既存テナントの CPU / メモリ / 帯域 / IOPS 使用率が追加後も 70% 以下に収まる予測
- Phase 2 時点のテナント上限: 5 テナント / クラスタ（業界標準の k8s マルチテナント設計）
- 上限到達時は新クラスタ追加（`QUA-DRB-003` のマルチクラスタ戦略と連動）

**検証方法**

- キャパシティ影響評価レポートを Backstage TechDocs で管理
- 月次キャパシティレビューで既存テナントの使用率トレンドを報告
