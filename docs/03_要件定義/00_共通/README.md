# 00_共通: 横断要件と全カテゴリ前提

本カテゴリは、他の 6 カテゴリ（10〜60）が共通して参照する**前提条件 / 用語 / リスク / 関係者 / 横断要件** を集約する。要件定義全体の「基盤層」として機能し、ここが不安定だと上位カテゴリの要件が互いに矛盾する。

用語の揺れ（tier1 が何を指すのか、監査ログとは何を含むのか）、前提の非明示（全 JTC テナントが Kubernetes を運用可能と仮定してよいか）、制約の不一致（Rust Edition 2024 を全自作領域で使うのか）といった問題は、要件定義の後半フェーズほど修正コストが跳ね上がる。本カテゴリを最初に固定することで、後続カテゴリの議論を要件の本質にフォーカスさせる。

---

## サブ項目とファイル構成

| ファイル | サブ項目記号 | 内容 |
|---|---|---|
| [`00_glossary.md`](./00_glossary.md) | `GLO` | 用語定義。tier1/2/3、自作領域、Dapr ファサード、ZEN Engine など |
| [`01_stakeholder.md`](./01_stakeholder.md) | `STK` | ステークホルダーと役割。稟議決裁者 / 情シス部門 / 業務部門 / 法務 / 監査 |
| [`02_assumption.md`](./02_assumption.md) | `ASM` | 前提条件。Kubernetes 運用体制 / ネットワーク接続性 / OSS 利用方針 |
| [`03_constraint.md`](./03_constraint.md) | `CON` | 制約条件。技術スタック / ライセンス / 社内規程 / 法令 |
| [`04_risk.md`](./04_risk.md) | `RSK` | リスク台帳。発生確率 × 影響度でスコアリング |
| [`05_error_taxonomy.md`](./05_error_taxonomy.md) | `ERR` | エラー分類と tier 間伝播ルール |
| [`06_横断要件.md`](./06_横断要件.md) | `XRQ` | 複数カテゴリに跨がる要件。命名規約 / バージョニング / 言語選定原則 |

サブ項目記号は `COM-<記号>-<連番>` の形で ID 採番に使う。例: `COM-GLO-001` は用語集の 1 番目の定義。

---

## 読み方

初読時は `00_glossary.md` → `01_stakeholder.md` → `03_constraint.md` → `02_assumption.md` → `04_risk.md` → `05_error_taxonomy.md` → `06_横断要件.md` の順で読むと、依存関係が自然に積み上がる。用語と関係者を先に押さえると、制約の意味が具体的な主体と結びついて理解できる。
