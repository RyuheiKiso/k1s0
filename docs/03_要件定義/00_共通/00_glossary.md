# COM-GLO: 用語定義

k1s0 プロジェクト内の用語を定義する。要件定義全体の語彙の揺れを抑え、後続カテゴリの要件同士が同じ語を同じ意味で使うことを保証する。

用語の定義は**曖昧さ排除** と **参照一元化** の 2 点を目的とする。前者は「tier1 に業務ロジックを入れるかどうか」のような論点で、定義が揺れると要件間で矛盾が発生する。後者は、構想設計や企画で繰り返し現れる用語をここに集約し、各要件ファイルから参照できるようにする。

本ファイルは他カテゴリから最も頻繁に参照される。用語追加時は**代替用語 / 同義語 / 似て非なる用語** を並記して誤用を予防する。

## 要件本体

### COM-GLO-001: 用語「tier1」の定義

- 優先度: MUST（tier 境界の全要件がこの定義に依存。未達で要件全体が崩壊）
- Phase: Phase 1a
- 関連: `ARC-T1-*`, [`../../02_構想設計/02_tier1設計/`](../../02_構想設計/02_tier1設計/)

tier1 という語が企画・構想設計・要件定義の各所で登場するが、具体的に「何が tier1 であり何が tier1 でないか」の境界が定義されていないと、内部実装の言語選定や API 公開範囲の判断が属人化する。

本要件が満たされた世界では、tier1 は「**業務ロジックを持たず、Dapr ファサードとしてランタイム抽象を提供する層**」として参照され、tier2 のドメインサービスや tier3 の業務アプリとは責務・実装言語・変更頻度の 3 軸で明確に区別される。

定義が崩れた場合、tier1 に業務要件が混入し、tier2 以降で再実装する羽目になる、もしくは tier1 の内部リファクタが tier2/3 の契約破壊に波及する。

**受け入れ基準**

- tier1 の責務として「業務ロジックを持たない」「Dapr / ランタイム抽象を提供する」「tier2 に向けた gRPC / クライアントライブラリを公開する」の 3 点が明記されている
- tier1 と tier2 の境界が 1 つの決定ツリー（業務依存有無）で判定可能である
- 全要件ファイルから `COM-GLO-001` を参照可能である

**検証方法**

- 要件定義内の「tier1」用例を grep し、本定義と矛盾する用法がないことを確認

### COM-GLO-002: 用語「tier2」の定義

- 優先度: MUST（tier 境界の全要件がこの定義に依存）
- Phase: Phase 1a
- 関連: `ARC-T2-*`

tier2 は「**業務ドメインを持つサービス群**」と定義する。tier1 が公開する gRPC / クライアントライブラリを経由して tier1 のランタイム抽象を利用し、tier3 から見た業務機能のエンドポイントを提供する層である。

本要件が満たされた世界では、tier2 の新規サービス追加が tier1 の改修なしに可能であり、かつ tier2 が tier1 の内部言語（Rust / Go）に依存しないため、tier2 自体の実装言語は JTC 顧客の既存資産（Java / Python / Node.js など）に合わせて自由に選択できる。

定義が崩れた場合、tier2 と tier1 の境界が曖昧化し、tier1 の内部言語変更が tier2 に波及するリスクが発生する。

**受け入れ基準**

- tier2 は tier1 の内部言語を知らない
- tier2 は tier1 の公開 gRPC / クライアントライブラリのみを利用する
- tier2 同士の通信は原則 tier1 経由とする（直接通信は例外運用）

**検証方法**

- tier2 のビルド成果物の依存解析で tier1 内部パッケージが含まれないこと

---

### COM-GLO-003: 用語「tier3」の定義

- 優先度: MUST
- Phase: Phase 1a
- 関連: `ARC-T3-*`

tier3 は「**顧客・現場ユーザー向けの業務アプリ**」と定義する。tier2 のドメインサービスを組み合わせ、業務 UI・バッチ処理・帳票出力などを提供する。雛形 CLI で生成されたスケルトンを出発点とし、配信ポータル経由でパッケージとして配布される。

tier3 が明確に区別されることで、「**雛形から外れた独自実装**」を CI 段階で検知でき、運用工数の増加を早期に抑制できる。

**受け入れ基準**

- tier3 は雛形 CLI で生成されたスケルトンから派生する
- tier3 は配信ポータルを経由して配布される
- tier3 は原則として tier1 に直接依存せず、tier2 を経由する

---

### COM-GLO-004: 用語「自作領域」の定義

- 優先度: MUST（Rust / Go の境界判定に直結）
- Phase: Phase 1a
- 関連: `ARC-T1-003`, [`../../02_構想設計/03_技術選定/`](../../02_構想設計/03_技術選定/)

自作領域は「**k1s0 プロジェクトが自力で設計・実装し、OSS に還元しないか、還元しても独立プロジェクトとして成立する機能群**」と定義する。具体的には以下 4 領域を指す。

- ZEN Engine 統合（ルール DSL と評価エンジンの糊付け）
- crypto モジュール（暗号化・署名・監査ログのハッシュチェーン）
- 雛形 CLI（tier3 スケルトン生成）
- JTC 固有機能（社内 IdP 連携・電子帳簿保存法対応などの個別要件）

本定義は Rust 採用範囲の決定根拠になる。自作領域以外（Dapr ファサード層）は Go で実装し、ZEN Engine / crypto は Rust で実装する。

**受け入れ基準**

- 自作領域 4 つが個別に列挙されている
- 各領域の Rust / Go の採用根拠が明示されている
- 新規機能追加時に「自作領域か否か」の判定基準が 1 つのフローチャートに収まる

---

### COM-GLO-005: 用語「Dapr ファサード」の定義

- 優先度: MUST
- Phase: Phase 1a
- 関連: `ARC-T1-002`, [`../../02_構想設計/02_tier1設計/01_設計の核/`](../../02_構想設計/02_tier1設計/01_設計の核/)

Dapr ファサードは「**tier1 のうち Dapr を隠蔽する Go 実装層**」と定義する。tier2/tier3 から見ると、Dapr ではなく tier1 の gRPC / クライアントライブラリを呼んでいるように見える。Dapr が将来別のランタイムに置き換わっても、tier2/tier3 への契約は維持される。

この定義は、企画書 13 章「Dapr 差し替え可能性」の技術的実現根拠となる。

**受け入れ基準**

- Dapr ファサードは Go で実装され、daprd を直接呼ぶ責務を持つ
- tier2/tier3 から daprd を直接呼ぶコードは検出時に CI で拒否される
- Dapr Workflow / Temporal のどちらを使うかは Dapr ファサード内部で振り分ける

---

### COM-GLO-006: 用語「ZEN Engine」の定義

- 優先度: MUST
- Phase: Phase 1a
- 関連: `ARC-RUL-*`, [`../../02_構想設計/03_技術選定/02_中核OSS/`](../../02_構想設計/03_技術選定/02_中核OSS/)

ZEN Engine は「**GoRules 社がオープンソースとして提供する JDM ベースのビジネスルール評価エンジン**」と定義する。k1s0 は ZEN Engine を Rust バインディング経由で統合し、ルール DSL の評価基盤として利用する。ZEN Engine のライセンスは MIT であり、k1s0 の内部利用および OSS 還元に制約はない。

**受け入れ基準**

- ZEN Engine のバージョン固定ポリシーが定義されている
- JDM スキーマのバージョニング方針が定義されている
- ZEN Engine への独自パッチが発生した場合の上流還元ルールが定義されている

---

### COM-GLO-007: 用語「配信ポータル」の定義

- 優先度: MUST
- Phase: Phase 1b
- 関連: `ARC-T3-002`, [`../../02_構想設計/04_CICDと配信/02_アプリ配信ポータル.md`](../../02_構想設計/04_CICDと配信/02_アプリ配信ポータル.md)

配信ポータルは「**tier3 業務アプリのパッケージを配布し、端末設定（copyable フラグで許可された設定のみ）をコピー可能な形で提供する Web UI**」と定義する。Backstage（開発者向けポータル）とは別物であり、主な利用者は業務部門のエンドユーザー・情シス担当者である。

**受け入れ基準**

- 配信ポータルは Backstage とは別 URL / 別 UI で提供される
- copyable フラグによる端末設定の公開制御が実装されている
- 端末設定のコピーは監査ログに記録される

---

### COM-GLO-008: 用語「監査ログ」の定義

- 優先度: MUST
- Phase: Phase 1a
- 関連: `SEC-AUD-*`

監査ログは「**ユーザー主体の操作 / 権限変更 / データアクセス / システム設定変更の 4 種を対象とした、ハッシュチェーンで改ざん検知可能な時系列記録**」と定義する。アプリケーションのエラーログやパフォーマンスログとは別物で、用途・保管期間・保管場所が異なる。

**受け入れ基準**

- 監査対象イベントの 4 分類が明示されている
- 各イベントのスキーマが定義されている
- ハッシュチェーンの検証手順が文書化されている

---

### COM-GLO-009: 用語「Phase」の定義

- 優先度: MUST
- Phase: Phase 1a
- 関連: 全要件

Phase は企画の [`../../01_企画/03_ロードマップと体制/00_フェーズ計画.md`](../../01_企画/03_ロードマップと体制/00_フェーズ計画.md) に定義された時系列区分を指す。具体的には以下 5 区分を使う。

- Phase 1a: MVP-0（稟議通過の最小動作）
- Phase 1b: MVP-1a（パイロット業務開始）
- Phase 1c: MVP-1b（機能完全性の確保）
- Phase 2: 本番運用（複数テナント受け入れ）
- Phase 3: 拡張フェーズ（OSS 化 / 外部公開）

全要件で「達成が要求される最も早い Phase」を記載する。要件間の依存でより早い Phase で達成が必要になった場合は、要件を分割する。

**受け入れ基準**

- 5 区分が全要件ファイルで同じ意味で使われている
- Phase 跨ぎの依存が存在する場合、要件が分割されている
