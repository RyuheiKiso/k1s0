# COM-XRQ: 横断要件

複数カテゴリに跨がる要件を集約する。特定カテゴリに帰属させると重複記述が発生する、もしくは責任分界が不明瞭になる要件を本ファイルで一元管理する。

横断要件の典型は「命名規約」「バージョニング方針」「言語選定原則」「文書規約」など。これらはカテゴリ横断で遵守される必要があり、個別カテゴリに埋めると参照漏れが発生しやすい。

## 要件本体

### COM-XRQ-001: 命名規約の統一

- 優先度: MUST
- Phase: Phase 1a
- 関連: 全カテゴリ

プロジェクト全体で以下の命名規約を統一する。

- コード（ファイル / 関数 / 変数 / パッケージ）は各言語の慣習に従う（Rust: snake_case、Go: camelCase、Protobuf: snake_case）
- Kubernetes リソース名は `<tier>-<service>-<component>` 形式
- 環境変数は `K1S0_<category>_<key>` 形式
- 監査ログのイベント名は `<actor>.<action>.<resource>` 形式

**受け入れ基準**

- 各規約が 1 つの参照ドキュメントに集約されている
- CI で静的検査が実施される

---

### COM-XRQ-002: バージョニング方針

- 優先度: MUST
- Phase: Phase 1a
- 関連: `ARC-API-*`, `OPS-REL-*`

k1s0 自体のバージョニングは SemVer (MAJOR.MINOR.PATCH) を採用する。API / SDK のバージョニングはカレンダーバージョン（YYYY.MM.PATCH）と SemVer の併用を tier1 設計で選択する（`ARC-API-001` で詳細決定）。

**受け入れ基準**

- リリースタグが SemVer 遵守
- 破壊的変更時の MAJOR 昇格ルールが明文化

---

### COM-XRQ-003: 言語選定原則

- 優先度: MUST
- Phase: Phase 1a
- 関連: `ARC-T1-003`

tier1 自作領域は Rust、Dapr ファサード層は Go。tier2 は顧客自由（JTC 既存資産を尊重）。tier3 は雛形 CLI が生成する言語（初期は Go / TypeScript の 2 種を提供）。

言語選定の例外は ADR で記録し、要件定義を更新する。

**受け入れ基準**

- 各 tier の使用言語が明文化されている
- 新規言語追加は ADR 必須

---

### COM-XRQ-004: 文書規約の遵守

- 優先度: MUST
- Phase: Phase 1a
- 関連: `CLAUDE.md`

`CLAUDE.md` のコーディング規約（コメント必須、ファイル 300 行以内）、ドキュメント規約（drawio 必須、散文必須、表の品質基準）、drawio レイヤ記法規約を全成果物で遵守する。

**受け入れ基準**

- PR テンプレートに規約チェックリストが含まれている
- 規約違反は CI / レビューで指摘される

---

### COM-XRQ-005: ADR による技術決定の記録

- 優先度: MUST
- Phase: Phase 1a
- 関連: `BIZ-GOV-001`

確定した技術決定は `docs/00_format/ADR-xxxx.md` のフォーマットに従って ADR として記録する。決定済み ADR は原則書き換えず、変更時は新規 ADR を起票し旧 ADR を `Superseded` に更新する。

**受け入れ基準**

- ADR 連番が欠番なく採番されている
- Phase ゲートで ADR レビューを実施

---

### COM-XRQ-006: プロジェクト構造の 3 層原則

- 優先度: MUST
- Phase: Phase 1a
- 関連: `ARC-T1-*`, `ARC-T2-*`, `ARC-T3-*`

tier1 / tier2 / tier3 の 3 層構造を要件定義・実装・配布の全段階で維持する。3 層外の「tier0」「tier4」を作らない。ランタイム基盤（Kubernetes / Istio / Kafka）は tier1 から見た下位レイヤであり、tier 番号を振らない。

**受け入れ基準**

- 要件定義内で tier0 / tier4 の表現が出現しない
- 新規コンポーネントは 3 層のいずれかに配置される

---

### COM-XRQ-007: 時刻と日付の扱い

- 優先度: MUST
- Phase: Phase 1a
- 関連: `BIZ-I18-003`, `SEC-AUD-*`

全システム内部の時刻は UTC で持ち、ユーザー表示時に JST に変換する。日付フォーマットは `YYYY-MM-DD`、時刻は ISO 8601。監査ログはナノ秒精度で記録する。

**受け入れ基準**

- DB スキーマの時刻型が UTC 保持前提で定義されている
- クライアント表示のタイムゾーン変換が集中管理されている
