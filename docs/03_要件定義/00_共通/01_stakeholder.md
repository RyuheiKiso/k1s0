# COM-STK: ステークホルダー定義

k1s0 の要件定義に影響を与える/受けるステークホルダーを定義し、各要件の優先度判定・合意形成プロセス・受け入れ承認者を明確化する。

要件定義を単独のドキュメントとして完結させるのではなく、**受け入れ承認の責任主体** を明示することで、後続の運用・監査・事業判断の各局面で判断者が一意に定まる。ステークホルダーの定義が曖昧だと、SLO 合意や法務適合の判断が誰に帰属するのか不明瞭になり、変更時の意思決定が停滞する。

## 要件本体

### COM-STK-001: 稟議決裁者

- 優先度: MUST（稟議通過の必要条件）
- Phase: Phase 1a
- 関連: `BIZ-GOV-*`

稟議決裁者は、Phase 1a / 1b / 2 / 3 の各境界で Phase ゲートの承認可否を判断する役割を持つ。通常は情シス担当役員またはその委任を受けた部長クラス。

本要件が明示されていない状態では、Phase ゲートの承認ルートが属人化し、Phase 完了判定が遅延する。決裁者を明示することで、要件の MUST / SHOULD 判定が「未達時に誰が承認を止めるか」で一意に判断できるようになる。

**受け入れ基準**

- Phase ごとの決裁者ロールが明文化されている
- 決裁者が不在時の代理権限ルートが明示されている

---

### COM-STK-002: 情シス部門

- 優先度: MUST
- Phase: Phase 1a
- 関連: `OPS-*`

情シス部門は、k1s0 の運用責任主体。Kubernetes クラスタの運用、tier1 の保守、tier2/tier3 開発者のサポート窓口を担う。SRE 当番、リリース承認、インシデント対応の第一線責任を持つ。

**受け入れ基準**

- 情シス部門の人員規模（Phase 別）が記載されている
- 運用タスクの責任範囲が `40_運用ライフサイクル/` の各要件と整合している

---

### COM-STK-003: 業務部門（tier3 開発者 / エンドユーザー）

- 優先度: MUST
- Phase: Phase 1b
- 関連: `ARC-T3-*`, `BIZ-UX-*`

業務部門は、tier3 業務アプリの開発者と、配信ポータルからアプリを利用するエンドユーザーの総称。雛形 CLI の使いやすさ、配信ポータルの UX、オンボーディング速度に関する受け入れ承認を行う。

**受け入れ基準**

- 代表業務部門（パイロット対象 3〜5 部門）が特定されている
- 受け入れフィードバックの収集チャネルが定義されている

---

### COM-STK-004: 法務 / コンプライアンス部門

- 優先度: MUST（法令適合・契約条項の承認必須）
- Phase: Phase 1a
- 関連: `SEC-CMP-*`, `BIZ-LEG-*`, `BIZ-LIC-*`

法務 / コンプライアンス部門は、OSS ライセンス適合、日本法令適合、顧客契約条項、知財帰属の承認主体。AGPL 内部利用の根拠、個人情報保護法改正への対応、SLA 条項の適法性を確認する。

**受け入れ基準**

- 法務レビューが必要な要件が特定され、ラベル付けされている
- レビュー期間の SLA が設定されている（例: 10 営業日以内）

---

### COM-STK-005: 監査部門（内部監査 / 外部監査対応窓口）

- 優先度: MUST
- Phase: Phase 1b
- 関連: `SEC-AUD-*`, `SEC-CMP-*`

監査部門は、監査ログの保全、CSV 搬出、内部統制の有効性テスト、外部監査応答を担う。tier1 自作領域の監査ログ実装、ハッシュチェーンの検証フロー、WORM 保管の実装根拠を確認する。

**受け入れ基準**

- 監査対応責任者が特定されている
- 外部監査発生時のエスカレーションパスが定義されている

---

### COM-STK-006: セキュリティ部門（SOC / CSIRT）

- 優先度: MUST
- Phase: Phase 1a
- 関連: `SEC-SEC-*`, `OPS-INC-*`

セキュリティ部門は、脆弱性管理、インシデント検知、SOC 運用、CSIRT 対応を担う。CVE スキャン基準、脆弱性 SLA、Sev1 インシデントのエスカレーション先。

**受け入れ基準**

- インシデント発生時の連絡先と SLA が明示されている
- 脆弱性対応の責任範囲が運用部門と重ならない形で分担されている

---

### COM-STK-007: 開発主導エンジニア（tier1 コア開発チーム）

- 優先度: MUST
- Phase: Phase 1a
- 関連: `ARC-T1-*`, `DEV-*`

tier1 コア開発チームは、Go ファサードと Rust 自作領域を実装する内部チーム。要件定義の技術的実現可能性を査読し、実装期間の妥当性を評価する責任を持つ。

**受け入れ基準**

- チーム規模と Rust / Go 人材構成が Phase 別に明示されている
- Rust 育成 / 採用の進捗を Phase ゲートで確認する
