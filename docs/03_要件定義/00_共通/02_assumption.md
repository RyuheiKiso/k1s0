# COM-ASM: 前提条件

k1s0 の要件群が成立するための**前提** を列挙する。前提は「この条件が変わったら要件群自体を再評価する必要がある」性質のものに限定する。

前提は制約（`03_constraint.md`）とは区別する。制約は「変えられないルール」だが、前提は「現時点で成立しているが崩れうる仮定」である。前提が崩れた際の影響が大きい要件は、関連する Phase ゲートで**前提の再確認** を行う。

## 要件本体

### COM-ASM-001: Kubernetes 運用体制が確保されている

- 優先度: MUST（前提が崩れると k1s0 自体の成立が困難）
- Phase: Phase 1a
- 関連: `ARC-INF-*`, `OPS-OPR-*`

本プロジェクトは、顧客または情シス部門が Kubernetes を運用できる体制（人員 / スキル / 予算）を持つことを前提とする。現状、日本企業では Kubernetes 運用人材が不足しており、自社運用ではなくマネージド K8s の利用でこの前提が崩れる可能性がある。

要件達成後の世界では、全顧客に対して k1s0 のインストール時に「Kubernetes 運用前提」のチェックリストが提示され、前提を満たさない場合は SES 契約またはマネージド K8s 利用に誘導される。

前提が崩れた場合、k1s0 のインストール後の運用が顧客側で回らず、サポート負荷が情シス部門に集中する。最悪のシナリオでは、k1s0 が「運用できないプラットフォーム」として評価されリコール対象になる。

**受け入れ基準**

- Kubernetes 運用前提のチェックリストが整備されている
- 前提不成立時の誘導パス（SES / マネージド K8s）が明示されている
- 前提再確認を Phase ゲートで実施する

---

### COM-ASM-002: 社内ネットワーク接続性が制御下にある

- 優先度: MUST
- Phase: Phase 1a
- 関連: `ARC-INF-005`, `OPS-ENV-*`

JTC 環境での corporate proxy、社内 DNS、SSL 復号ゲートウェイの存在を前提とする。k1s0 の初期セットアップ時にこれらのネットワーク制約が明示的に解決されていることを想定する。

前提が崩れた場合、MVP-0 の期間見積（企画書の 3〜4 週間）がさらに遅延し、前回の技術査読で指摘した High リスク（MVP-0 過小見積もり）が現実化する。

**受け入れ基準**

- corporate proxy / 社内 DNS の設定手順が MVP-0 セットアップに含まれる
- 不明確な場合は SES 契約での対応が契約条項に明記されている

---

### COM-ASM-003: OSS 利用方針が法務承認済み

- 優先度: MUST
- Phase: Phase 1a
- 関連: `BIZ-LIC-*`, `SEC-CMP-*`

企画 [`../../01_企画/05_法務サマリ/`](../../01_企画/05_法務サマリ/) で整理した OSS 利用方針（AGPL Phase 分割、MPL 個別対応など）が法務部門の承認を Phase 1a 開始前に得ていることを前提とする。

前提が崩れた場合、Phase 1a 開始後に OSS 採用を再評価することになり、MVP-0 期間が大幅に延伸する。

**受け入れ基準**

- 法務承認記録が保管されている
- AGPL / MPL / GPL 系 OSS の採用可否表が最新化されている

---

### COM-ASM-004: Rust 人材確保パスが存在する

- 優先度: SHOULD（前提崩れ時は SES でフォールバック可能だが TCO に影響）
- Phase: Phase 1b
- 関連: `BIZ-VEN-003`, `BIZ-TRN-002`

tier1 自作領域の Rust 実装を担う人材（社内育成 + 中途採用 + SES）の確保パスが複線で存在することを前提とする。企画書 16.5 章の「Rust 育成成功率 70%」は推定値で出典不明であり、前回の技術査読で Medium リスクとして指摘されている。

前提が崩れた場合、Phase 2 の本番運用時に自作領域の保守人員が不足し、SES 依存が高まってコスト増と知財流出リスクが顕在化する。

**受け入れ基準**

- Phase 1c 完了時点で Rust 実装可能な社内人員が 2 名以上確保されている
- SES フォールバック契約のコスト上限が TCO 試算に反映されている

---

### COM-ASM-005: Phase ゲートで技術前提を再評価する

- 優先度: MUST
- Phase: Phase 1a
- 関連: 全要件

各 Phase ゲート（1a → 1b、1b → 1c、1c → 2、2 → 3）で、本ファイルの前提が依然として成立していることを確認するプロセスを前提とする。

**受け入れ基準**

- Phase ゲートチェックリストに「前提再確認」項目がある
- 再確認結果が ADR として記録される
