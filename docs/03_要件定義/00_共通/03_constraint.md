# COM-CON: 制約条件

k1s0 の要件群が遵守すべき**絶対条件** を列挙する。制約は技術・ライセンス・社内規程・法令の 4 系統に分類し、それぞれの根拠と未遵守時の影響を明示する。

制約は前提（`02_assumption.md`）と区別する。前提は変化しうる仮定だが、制約は**意思決定者が変えられないルール**。制約違反の要件は採番自体を拒否する（MUST / SHOULD / MAY のラベル付与以前に、採番対象にならない）。

## 要件本体

### COM-CON-001: 技術スタック制約（Rust Edition 2024 / Go Dapr SDK）

- 優先度: MUST
- Phase: Phase 1a
- 関連: `ARC-T1-*`, [`../../02_構想設計/adr/`](../../02_構想設計/adr/)

tier1 自作領域は Rust Edition 2024 を使用する。Dapr ファサード層は Go の stable Dapr SDK を使用する。両言語の選定根拠は構想設計で ADR として記録されており、本要件定義内では選定判断を覆さない。

制約違反の具体例としては、「Rust Edition 2021 を使いたい」「Go Dapr SDK の alpha を採用したい」などの要求が発生した場合、要件採番を拒否し、必要であれば ADR の差し戻しを経てから再審議する。

**受け入れ基準**

- CI で Rust Edition と Go SDK バージョンが固定されている
- 例外申請は ADR として記録されることが明文化されている

---

### COM-CON-002: 内部通信は Protobuf gRPC 必須

- 優先度: MUST
- Phase: Phase 1a
- 関連: `ARC-T1-004`, `ARC-API-*`

tier1 内部のサービス間通信は Protobuf + gRPC を必須とする。JSON / REST / MessagePack などの他プロトコルを内部で利用することは禁止する。本制約は、Rust と Go の言語境界で型整合を機械的に検証できる唯一の手段が Protobuf スキーマであることに由来する。

**受け入れ基準**

- tier1 内部の全サービスが Protobuf スキーマを公開している
- CI で Protobuf スキーマの後方互換チェックが実施される
- JSON / REST による内部通信の検出は CI 拒否

---

### COM-CON-003: tier2/tier3 から内部言語は不可視

- 優先度: MUST
- Phase: Phase 1a
- 関連: `ARC-T2-001`, `ARC-T3-*`

tier2 / tier3 は tier1 の内部実装言語（Rust / Go）を知らない。tier1 が公開するのはクライアントライブラリと gRPC エンドポイントのみ。本制約は、tier1 の内部リファクタ（例: Go → Rust の移動）が tier2/3 の契約破壊に波及しない保証を与える。

**受け入れ基準**

- tier1 の公開成果物一覧（クライアントライブラリ / gRPC スキーマ）が明示されている
- tier2 のビルド成果物依存解析で tier1 内部パッケージが含まれないこと

---

### COM-CON-004: OSS ライセンス制約（AGPL / GPL / MPL の扱い）

- 優先度: MUST
- Phase: Phase 1a
- 関連: `BIZ-LIC-*`, [`../../01_企画/05_法務サマリ/01_OSSライセンス適合.md`](../../01_企画/05_法務サマリ/01_OSSライセンス適合.md)

AGPL-3.0 / GPL-2.0+/-3.0 / MPL-2.0 系 OSS の採用は、Phase 1a では原則禁止し、Phase 1b 以降で個別に法務承認を得た範囲で採用可能とする。Phase 1a の代替手段（Prometheus 素 UI / kubectl logs / PostgreSQL WORM）は前回技術査読で Medium リスクとして指摘済みで、MVP の可視性低下を受け入れるトレードオフとする。

**受け入れ基準**

- 依存 OSS の SBOM が法務承認済みライセンス一覧と照合される
- AGPL 採用時は SaaS 提供への影響が明示的に評価される
- ライセンス義務（ソース公開 / 派生物表示）の対応手順が CI に組み込まれている

---

### COM-CON-005: 日本法令への適合必須

- 優先度: MUST
- Phase: Phase 1a
- 関連: `SEC-CMP-*`, `SEC-PRV-*`

個人情報保護法（最新改正）、電子帳簿保存法、金融商品取引法、会社法の該当条項への適合を必須とする。業界法令（FISC 安対基準、医療情報ガイドラインなど）は顧客セグメント別に個別対応。

**受け入れ基準**

- 適合対象法令が一覧化され、各法令と関連要件がクロスリファレンスされている
- 法改正時の影響評価プロセスが定義されている

---

### COM-CON-006: データ国内保管原則

- 優先度: MUST（顧客契約上の絶対条件）
- Phase: Phase 1a
- 関連: `SEC-RES-*`

データ（顧客業務データ / 監査ログ / バックアップ / ログ）は国内リージョンで保管する。クロスボーダー移転は顧客明示同意を必須とする。

**受け入れ基準**

- データフロー図で国境を跨がないことが可視化されている
- クラウド事業者のリージョン指定が運用手順に含まれる

---

### COM-CON-007: ファイルサイズ制約（300 行以内）

- 優先度: MUST
- Phase: Phase 1a
- 関連: [`../../CLAUDE.md`](../../../CLAUDE.md)

ソースコードファイルは 1 ファイル 300 行以内とする（docs は例外）。本制約は `CLAUDE.md` に記載された開発規約であり、要件定義時のコードベース構造計画で考慮する必要がある。

**受け入れ基準**

- CI で行数チェックが実施される
- 300 行超過の例外は ADR で記録される
