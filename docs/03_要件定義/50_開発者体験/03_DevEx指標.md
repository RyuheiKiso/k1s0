# 03. DevEx 指標

本書は k1s0 の開発者体験を定量的に測定するための指標を定義する。DORA Four Keys、SPACE Framework、DevEx Survey の 3 フレームを組み合わせ、感覚論ではなく数値で改善サイクルを回す。

## 本書の位置付け

開発者体験は曖昧な概念であり、「開発が楽になった」の実感だけでは経営層を説得できない。本書は測定可能な指標を定め、定期的な計測と改善で生産性向上を実証する。DevEx 指標は Phase 1c から本格運用し、Phase 2 以降の稟議で「投資対効果」として利用する。

## DX-MET-001: DORA Four Keys

**現状**: ソフトウェア配信の生産性が測定されていないと、改善投資の優先度が立てられない。

**要件達成後**: DORA の 4 指標を Phase 1c から計測する。

- **Deployment Frequency（デプロイ頻度）**: 本番デプロイの頻度。目標は「オンデマンド」（1 日複数回）。Phase 1b は週 1 回、Phase 1c は 1 日 1 回、Phase 2 は 1 日複数回。
- **Lead Time for Changes（変更リードタイム）**: コミットから本番デプロイまでの時間。Phase 1b は 1 日、Phase 1c は 6 時間、Phase 2 は 1 時間を目標。
- **Change Failure Rate（変更失敗率）**: 本番デプロイが失敗して緊急修正が必要になる割合。15% 以下（Elite 基準）。
- **Time to Restore Service（復旧時間）**: 本番障害発生から復旧までの時間。1 時間以内（Elite 基準）。

計測は GitHub / Jira / Argo CD / PagerDuty のデータから自動集計、Grafana で可視化。

**崩れた時**: 改善投資が的外れになり、開発チームの時間が空転する。

**受け入れ基準**:
- Phase 1c で自動計測基盤構築
- Grafana の Four Keys ダッシュボードを全チーム閲覧可能
- 月次レビューで各指標のトレンドを追跡

**優先度**: MUST

## DX-MET-002: SPACE Framework

**現状**: DORA は配信速度のみを測るため、開発者の満足度や協働は取りこぼす。

**要件達成後**: Microsoft Research の SPACE Framework で補完する。

- **Satisfaction（満足度）**: 半期ごとの開発者サーベイ、10 段階評価で 7 以上を目標
- **Performance（パフォーマンス）**: 機能リリース数、不具合率、顧客影響
- **Activity（活動量）**: コミット数、PR 数、レビュー数（参考値、単体で KPI にはしない）
- **Communication（コミュニケーション）**: PR レビュー時間、チーム間コラボレーション頻度
- **Efficiency（効率）**: フロータイム（Focus Time / 割り込み時間比率）

単一指標の KPI 化は避け、組み合わせで解釈する。Goodhart の法則（測定されたものは操作される）への配慮。

**崩れた時**: DORA のみだと「早くデプロイした」だけで満足し、開発者が疲弊している状態を見逃す。

**受け入れ基準**:
- Phase 1c で半期サーベイを開始
- Phase 2 で PR レビュー時間の自動計測
- サーベイ回答率 80% 以上

**優先度**: SHOULD

## DX-MET-003: 開発者サーベイ

**現状**: 開発者の本音が上に届かないと、改善の優先度を外す。

**要件達成後**: 半期ごとに以下のサーベイを実施する。

- **k1s0 全体満足度（NPS）**: 「同僚に k1s0 を推奨するか」0〜10 で回答
- **カテゴリ別満足度**: テンプレート、ドキュメント、ローカル開発、CI/CD、サポート、テストの 6 カテゴリを 5 段階評価
- **痛み領域**: 自由記述、テキストマイニングで頻出キーワード抽出
- **改善優先度**: 回答者が改善してほしい項目を 3 つ選択
- **利用時間内訳**: コーディング / レビュー / ミーティング / 障害対応の割合

回答は匿名化、結果は全社公開。改善アクションは Product Council で優先付け。

**崩れた時**: 上意下達で改善が進み、現場の実感と乖離。

**受け入れ基準**:
- Phase 1c で初回サーベイ実施
- 半期ごとに継続、トレンド追跡
- 改善アクション項目の完了率 70% 以上

**優先度**: MUST

## DX-MET-004: CI/CD パイプライン時間

**現状**: CI/CD が遅いと、PR 投入を躊躇し、結果として大きな PR が増えてレビューコストが上がる。

**要件達成後**: 以下の時間目標を設ける。

- **PR チェック完了時間**: 10 分以内（P50）、15 分以内（P95）
- **本番デプロイ完了時間**: 30 分以内（main マージから prod 適用まで）
- **フィードバックループ**: コミットから PR コメント表示まで 15 分以内

CI ジョブは並列化、キャッシュを積極活用。遅い job は定期的に分析し、ボトルネックを解消。

**崩れた時**: 開発者が PR を貯める癖がつき、レビューが重くなる悪循環。

**受け入れ基準**:
- Phase 1b で目標設定と計測開始
- 月次で P50/P95 を Grafana 可視化
- 目標超過が 2 か月続いたら改善スプリント起票

**優先度**: MUST

## DX-MET-005: 新規参画者の立ち上がり時間

**現状**: 新規参画者の立ち上がり時間が長いと、採用コストと機会損失が増大。

**要件達成後**: 以下を計測する。

- **First PR Time**: 参画から初 PR マージまでの日数。目標 5 営業日以内。
- **Productivity Time**: 参画からチーム平均並みのアウトプット到達までの週数。目標 4 週間。

10 分ルール（DX-GP-003）と連動、オンボーディングドキュメントの改善サイクルに活用。

**崩れた時**: 新規参画者の離職率上昇、採用コスト回収不能。

**受け入れ基準**:
- Phase 1c で計測開始
- First PR Time 中央値 5 営業日以内
- オンボーディング改善サイクルを四半期で回す

**優先度**: SHOULD

## DX-MET-006: Flow Time（集中時間）

**現状**: 会議・割り込み・障害対応で開発者のコーディング時間が減ると、生産性は急減する。

**要件達成後**: 以下を計測する。

- **Focus Time**: 2 時間以上の中断なし時間の 1 週間合計。目標 15 時間以上。
- **割り込み件数**: Slack メンション・ミーティング招待の 1 日件数。目標 10 件以下。
- **Context Switching**: 同時進行タスク数。目標 3 件以下。

計測は Google Calendar API、Slack API などから自動取得（対象者の同意前提）。

**崩れた時**: 生産性数値が下がり、原因が「忙しい感」以外に特定できない。

**受け入れ基準**:
- Phase 2 で計測基盤導入
- 対象者の同意ベースで匿名集計
- チーム平均 Focus Time が 15 時間以上を維持

**優先度**: COULD（Phase 2+）
