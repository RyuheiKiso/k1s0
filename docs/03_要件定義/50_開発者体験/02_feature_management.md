# DEV-FLG: フィーチャーフラグ / 機能管理要件

本ファイルは、k1s0 プラットフォーム上で運用されるフィーチャーフラグ基盤、フラグのライフサイクル、カナリアリリースの段階評価、A/B 実験の統計的健全性、古いフラグの自動検知を要件化する。フィーチャーフラグはリリース戦略（`OPS-REL-*`）とは別レイヤの関心であり、「コードは本番に出ているが、挙動はテナント単位・ユーザー単位で切り替えられる」状態を支える機構である。

フラグ運用で最も頻発する失敗は、「一度追加したフラグが削除されないまま半永久的に残る」ことである。経験的に、1 年放置されたフラグが分岐条件の真理値表を指数関数的に膨らませ、レグレッション検知を不能にする。本要件群は**追加ポリシーより撤去ポリシー**に比重を置き、3 ヶ月未変更フラグの自動検知を MUST として課す。

また、k1s0 は Dapr ファサードの差し替え可能性という長期戦略を掲げる以上、フラグ基盤自体も**ベンダロックインを避ける**必要がある。`OpenFeature` 標準を採用し、プロバイダ実装（flagd / Unleash / LaunchDarkly 等）は切り替え可能な状態を維持する。

## 要件本体

### DEV-FLG-001: OpenFeature 準拠のフィーチャーフラグ基盤

- 優先度: MUST（基盤選定は後から変更不可能に近いため Phase 1a で確定させる必要がある）
- Phase: Phase 1a
- 関連: `OPS-REL-*`, `COM-GLO-005`

現状、フィーチャーフラグ基盤は未選定である。既存 SaaS（LaunchDarkly / Split.io）は機能豊富だが JTC の閉域ネットワーク要件と衝突し、年間数百万円単位のランニングコストを増やす。自作は避ける一方で、特定 OSS への硬直的な依存も避けたい。

達成後は、k1s0 のフィーチャーフラグ API は **OpenFeature 標準**（CNCF incubating、ベンダ中立仕様）のクライアント SDK を採用し、プロバイダ実装として OSS の **flagd** を初期選定する。tier2/3 開発者は OpenFeature SDK を通じてフラグを参照し、プロバイダ差し替え時にアプリケーションコードを変更する必要がない。flagd は `operation` namespace で稼働し、PostgreSQL にフラグ定義を永続化する。

崩れた場合、特定ベンダの SDK に tier2/3 のコードが直結し、JTC 顧客のオンプレ要件が満たせなくなる。あるいは自作に走って運用工数を押し上げ、企画書の「1.5 名体制」運用前提が崩壊する。

**受け入れ基準**

- tier1 公開 SDK（C# / Go / TS）が OpenFeature 1.x クライアント API を提供
- プロバイダ実装として flagd が `operation` namespace で稼働し、SLO 99.9% を満たす
- フラグ定義スキーマが JSON Schema で管理され、バリデーションが CI で実行
- プロバイダ差し替え（flagd → 他 OpenFeature 準拠）がアプリコード修正なしで可能であることを設計レビューで確認

**検証方法**

- Phase 1a 完了時に tier2 リファレンス実装で 3 つ以上のフラグを運用し、OpenFeature API のみで参照していることをコード監査で確認
- flagd の可用性を Grafana ダッシュボードで常時計測

### DEV-FLG-002: フラグ命名規約とライフサイクル定義

- 優先度: MUST（フラグの命名揺れは削除判定を属人化させ、技術的負債を蓄積する）
- Phase: Phase 1a
- 関連: `DEV-FLG-001`, `DEV-FLG-005`

フラグ名に一貫性がないと、「このフラグは何のために追加したのか」が時間経過で失われ、削除判断が不能になる。また、フラグにはライフサイクル（Release / Experiment / Ops / Permission の 4 種）があり、種別ごとに削除タイミングが異なる。

達成後は、全フラグが **`<種別>.<ドメイン>.<機能名>`** の 3 階層命名（例: `release.billing.new-invoice-pdf` / `experiment.onboarding.wizard-v2` / `ops.kill-switch.bulk-export` / `permission.admin.audit-export`）に従う。各フラグには必須メタデータ（owner / created_at / expected_retirement_at / jira_ticket）が登録され、expected_retirement_at を超えたフラグは `DEV-FLG-005` の自動検知対象となる。

崩れた場合、フラグが無秩序に増殖し、PR レビュー時に「このフラグは消せるか」の判定に数時間かかる。レビュワーがコスト増を嫌ってフラグ追加を拒むようになり、本来フラグで守るべきリリースが直接リリースされて事故を生む。

**受け入れ基準**

- 命名規約違反フラグを追加する PR が CI で自動ブロック（`flagd` の CRD バリデーションまたは Pre-commit フック）
- 必須メタデータ 4 項目（owner / created_at / expected_retirement_at / jira_ticket）が全フラグに存在
- フラグ種別（Release / Experiment / Ops / Permission）ごとに標準的な寿命（Release: 2 週間 / Experiment: 実験期間 + 1 週間 / Ops: 恒久 / Permission: 恒久）が文書化
- 命名規約と寿命基準が Backstage TechDocs で公開

**検証方法**

- 月次でフラグ一覧を自動スキャンし、規約違反件数をダッシュボード化

### DEV-FLG-003: カナリアリリースの段階評価基準

- 優先度: MUST（カナリア判定基準がないと実質的に「全展開」と等価になり事故を防げない）
- Phase: Phase 1b
- 関連: `OPS-REL-001`, `QUA-PRF-001`, `QUA-OBS-*`

構想設計で「既定のリリース戦略はカナリア」と定められたが、段階移行の**判定基準**は未定義である。「感覚で GO」では 3 名体制の運用者によって判断がぶれ、事故時の責任所在が曖昧になる。

達成後は、カナリア段階は **5% → 25% → 100% の 3 段階**で、各段階に最低観測期間（5%: 30 分 / 25%: 2 時間）を設ける。移行判定は以下の 4 指標すべてが基準内であることを求める: (1) HTTP 5xx 率 0.1% 以下（ベースライン比 2 倍以内）、(2) p99 レイテンシ `QUA-PRF-001` の 500ms を超過しない、(3) エラーバジェット（`QUA-SLA-*`）の今期消費が 20% 以下、(4) 新規エラーログのパターンがベースラインと統計的有意差なし。1 指標でも失敗時は自動ロールバック（`OPS-REL-002`）。

崩れた場合、カナリア判定が属人化し、夜間リリースで「しんどいから 100% まで進めた」判断が事故を生む。JTC 顧客契約の SLA 違反に直結し、賠償金と信用失墜の両方を招く。

**受け入れ基準**

- Argo Rollouts（または同等）の AnalysisTemplate に上記 4 指標が定義
- 5% / 25% 段階で最低観測期間が厳守される設定
- 基準未達時の自動ロールバック動作を Phase 1b 完了前に Chaos エンジニアリング試験で検証
- 判定指標の 4 つが Grafana カナリアダッシュボードに統合表示

**検証方法**

- Phase 1b の本番相当環境で意図的にエラー率を上げたカナリアを流し、自動ロールバックが作動することを確認

### DEV-FLG-004: A/B 実験の統計的有意性基準

- 優先度: SHOULD（Phase 1b までは不要、Phase 2 で業務改善実験を始める時点で必達）
- Phase: Phase 2
- 関連: `DEV-FLG-003`, `BIZ-ANA-*`

A/B 実験は「効果があるように見えた」だけで本採用すると、ランダムゆらぎを効果と誤認し、実は悪化している機能を全展開する事故を起こす。統計的有意性の判定基準を事前合意しないと、実験結果に事後解釈の余地が残り、判断が政治化する。

達成後は、全 A/B 実験に**事前登録された仮説 / サンプルサイズ / 有意水準**が必要となる。最小サンプルサイズは `pwr` パッケージ相当の検出力計算（検出力 0.8、有意水準 0.05、最小検出効果 5%）で事前決定する。実験中の途中覗き見（p-hacking）を防ぐため、中間解析は事前登録された時点のみで実施し、アルファ消費補正（O'Brien-Fleming 境界等）を適用する。

崩れた場合、実験結果の解釈がマーケ部門や開発者の主観に引きずられ、本採用決定が信頼できなくなる。JTC 顧客への「データドリブンで改善しています」の訴求根拠が内部から崩壊する。

**受け入れ基準**

- A/B 実験登録フォーム（Backstage 上）に仮説 / サンプルサイズ / 有意水準の 3 項目が必須
- 実験開始前の事前登録がない限り flagd で `experiment.*` フラグを有効化できない
- 結果レポートが `BIZ-ANA-*` の分析基盤（Superset / Metabase 等）で自動生成
- p-hacking 検知（実験期間延長回数 / ルール変更回数）を監査ログに記録

**検証方法**

- Phase 2 開始時に 1 件以上の A/B 実験を登録〜完了し、レポート自動生成が機能することを確認

### DEV-FLG-005: 古いフラグの自動検知と撤去促進

- 優先度: MUST（フラグ技術的負債は指数関数的に蓄積し、放置は回復不能）
- Phase: Phase 1b
- 関連: `DEV-FLG-002`, `OPS-DEP-*`

フラグの追加は簡単だが削除は意識的な取り組みがないと進まない。経験則として「3 ヶ月未変更のリリース系フラグ」の大半は、既に本番で常時 ON または常時 OFF になっており、削除可能である。このゴミ掃除を自動化せずに手動運用に頼ると、全員が掃除の先送りをして負債が雪だるま式に増える。

達成後は、`expected_retirement_at` を超過したフラグ、および **3 ヶ月以上値が変更されていない Release / Experiment フラグ**を週次スキャナが自動検知し、GitHub Issue を owner にアサインして起票する（ラベル `flag-retirement`）。2 回の通知でも対応されない場合、自動 PR でフラグ参照コードの削除パッチを提案する（Phase 2 以降の拡張）。Ops / Permission 系は恒久フラグとして例外扱いし、命名規約の prefix で判別する。

崩れた場合、フラグ数が線形ではなく指数関数的に増え、コード内の分岐条件が真理値表で管理不能になる。過去他社事例では、1,000 件超の未使用フラグが QA テストケースを爆発させ、リリースサイクルが 2 週間から 6 週間に伸びている。

**受け入れ基準**

- 週次スキャナが GHA cron で稼働し、検知結果を Grafana ダッシュボードに可視化
- 3 ヶ月以上未変更の Release / Experiment フラグを自動検知
- GitHub Issue の自動起票（テンプレート `flag-retirement`）が動作
- 撤去済みフラグ数の月次推移がダッシュボードで追跡可能
- Ops / Permission 系の恒久フラグは明示的な `permanent: true` メタデータで除外

**検証方法**

- Phase 1b 完了時に未撤去フラグが 5 件以下であることを確認
- 四半期ごとに撤去済み件数 / 残存件数を稟議レビューに提出

### DEV-FLG-006: フラグ変更の監査ログ化

- 優先度: MUST（権限系フラグの変更は `COM-GLO-008` の監査対象 4 分類に該当）
- Phase: Phase 1a
- 関連: `COM-GLO-008`, `SEC-AUD-*`, `DEV-FLG-001`

フラグ変更は実質的な挙動変更であり、特に Permission 系フラグは権限変更そのものである。監査ログ要件（`COM-GLO-008`）の「権限変更」カテゴリに該当するため、flagd 上の操作が tier1 監査ログに連携される必要がある。

達成後は、flagd の管理 UI / API から行われる全フラグ操作（create / update / delete / toggle）が監査ログに記録され、ハッシュチェーンで改ざん検知可能となる。記録項目は `SEC-AUD-*` で定義するスキーマに従い、who（主体）/ when（時刻）/ what（フラグ名・変更前後の値）/ why（チケット ID）の 4 点を含む。

崩れた場合、監査時に「誰がいつどのフラグを変えたか」が追跡不能となり、`COM-GLO-008` の監査ログ定義が部分的に破綻する。内部不正によるフラグ操作（例: 特定ユーザへのアクセス権をこっそり付与）を事後検知できなくなる。

**受け入れ基準**

- flagd から tier1 監査ログ API への連携が実装
- 監査イベントが `COM-GLO-008` のハッシュチェーンに接続
- 記録項目 4 点（who / when / what / why）が全イベントに存在
- 監査ログの保管期間が電子帳簿保存法準拠（7 年以上）の要件に整合

**検証方法**

- flagd で意図的にフラグを変更し、tier1 監査ログ API で記録を確認
- Phase 1a 完了時に監査ログのハッシュチェーン検証を実行
