# 05. ローカル開発環境

本書は tier2/tier3 開発者が手元のラップトップで k1s0 アプリを開発・動作確認するための環境要件を定義する。Dev Container、kind / minikube の標準設定、tier1 API のモック、ホットリロード、デバッグ環境を確定する。

## 本書の位置付け

ローカル開発が遅い、壊れやすい、本番と乖離しているといった状況は、開発者の「k1s0 嫌い」を生む最大要因である。本書は「朝 PC を開いてから 5 分でコーディング開始」を標準とし、本番と同一の体験でローカル開発できる環境を要求する。

## DX-LD-001: Dev Container 標準化

**現状**: 開発環境のセットアップが `brew install` の列挙手順だと、依存バージョンの差でトラブルが頻発する。

**要件達成後**: VS Code Dev Container（または JetBrains Gateway）で標準開発環境を提供する。

- **ベースイメージ**: Microsoft 公式 Dev Container + k1s0 カスタムレイヤ
- **含むツール**: Go、Rust、Node.js、kubectl、kind、Helm、Argo CD CLI、Buf、OpenTelemetry Collector、k1s0 CLI
- **シェル設定**: zsh + starship、補完、エイリアス事前設定
- **VS Code 拡張**: Go、rust-analyzer、Kubernetes、YAML、Protobuf、GitLens を自動インストール
- **ポートフォワード**: 開発クラスタへの主要ポートを自動マッピング

Dev Container は SemVer で管理、Backstage から選択可能とする。

**崩れた時**: 「私のローカルでは動く」問題が頻発、サポート負荷が増大。

**受け入れ基準**:
- Phase 1a で Dev Container v1 提供
- Phase 1b でサンプルアプリ全種がそのまま動く
- Phase 1c で JetBrains Gateway 版も提供

**優先度**: MUST

## DX-LD-002: ローカル Kubernetes クラスタ

**現状**: 開発者がそれぞれ minikube / kind / k3d / Docker Desktop を使い分けると、動作確認結果のばらつきが発生する。

**要件達成後**: kind を標準とし、`k1s0 dev up` コマンドで 3 分以内にクラスタ起動・k1s0 基盤セットアップが完了する。

- **クラスタ構成**: kind シングルノード、control-plane 兼 worker
- **プリインストール**: Istio Ambient、Argo CD、PostgreSQL（CloudNativePG 1 node）、Kafka（Strimzi 1 broker）、Valkey、MinIO、flagd、Keycloak 開発用
- **リソース**: 最低 8GB RAM、4 vCPU、50GB ディスク
- **永続性**: `k1s0 dev down` で削除、`--persist` で永続化可能

minikube / k3d も動作サポートするが、公式は kind。

**崩れた時**: ローカルクラスタ起動失敗で数時間デバッグ、開発時間を浪費。

**受け入れ基準**:
- kind 標準構成を Phase 1a で提供
- `k1s0 dev up` 所要時間 3 分以内（初回は 10 分以内）
- macOS / Windows / Linux で動作保証

**優先度**: MUST

## DX-LD-003: tier1 API モック

**現状**: tier1 API を実起動せずに tier2 アプリの単体テストを書けないと、テスト実行が重くなる。

**要件達成後**: tier1 全 API に対してインプロセスモック SDK を提供する。

- **State API モック**: インメモリストレージ、単体テストで即座に利用可能
- **PubSub API モック**: goroutine / tokio タスクで subscribe callback を呼び出す実装
- **Decision API モック**: 固定 JDM 結果を返すスタブ
- **Workflow API モック**: 同期実行する fake
- **Log / Telemetry モック**: No-op 実装

tier2 アプリの単体テストでは、これらを依存注入で差し替えることで高速テストが可能。

**崩れた時**: 単体テストでも kind クラスタ起動が必要になり、テスト時間が 10 倍に。

**受け入れ基準**:
- Phase 1b で全 tier1 API のモック提供
- モック動作が実装と一致（契約テストで検証、DX-TEST-004）
- モック利用ドキュメントを Backstage で公開

**優先度**: MUST

## DX-LD-004: ホットリロード

**現状**: コード変更のたびにコンテナビルド・デプロイが必要だと、1 変更あたり 3 分かかり集中が切れる。

**要件達成後**: 以下のツールで高速フィードバックを実現する。

- **Go**: `air` でファイル変更検知と自動再起動（ローカル直接実行）
- **Rust**: `cargo-watch` + `cargo run` でコンパイル後再起動
- **Next.js**: Next.js Dev Server のホットリロード
- **Kubernetes デプロイ**: `tilt` または `skaffold` でコード変更を Pod に即反映、1 変更あたり 10 秒以内

Dev Container + Tilt の組み合わせを標準とし、雛形に事前組み込み。

**崩れた時**: 変更反映待ちが積み重なり、1 日あたり 1 時間のロス。

**受け入れ基準**:
- Phase 1b で Tilt 連携を提供
- ホットリロード時間 10 秒以内
- ドキュメントで使い方を明示

**優先度**: SHOULD

## DX-LD-005: デバッグ支援

**現状**: 本番相当の Kubernetes 環境でのデバッグが困難だと、ログ追加 → 再デプロイの泥臭いループに。

**要件達成後**: 以下のデバッグツールを提供する。

- **リモートデバッグ**: Go は `dlv`、Rust は `lldb-server` または `gdbserver`、Pod 内に attach 可能
- **ログ閲覧**: `stern` で複数 Pod のログをストリーム、Backstage 経由で Loki クエリ
- **トレース閲覧**: Tempo の trace を Backstage から確認、Jaeger UI も利用可
- **kubectl debug**: エフェメラルコンテナで Pod 内にデバッグシェル投入
- **port-forward 補助**: `k1s0 dev connect <service>` で主要サービスへの接続を自動化

**崩れた時**: デバッグが勘と printf の世界に戻り、問題解決時間が倍増。

**受け入れ基準**:
- Phase 1c で全ツール動作確認済み
- デバッグガイドを Backstage で公開
- 主要 IDE（VS Code、GoLand、RustRover）でリモートデバッグ手順を整備

**優先度**: SHOULD

## DX-LD-006: オフライン開発対応

**現状**: 出張・リモート環境でインターネット不安定時に開発が止まると、開発者のフラストレーションが高まる。

**要件達成後**: オフライン状態でもローカル開発が継続できる。

- **OSS パッケージキャッシュ**: Go module、crate、npm パッケージを Dev Container にプリキャッシュ
- **コンテナイメージキャッシュ**: 主要イメージを手元に pull 済み
- **ドキュメント**: Backstage TechDocs をローカルミラー（docusaurus ビルド）で閲覧可能
- **プラットフォーム API**: モック SDK（DX-LD-003）で tier1 API はオフラインでも動作

CI / デプロイはオンライン時にまとめて実行する前提。

**崩れた時**: 移動中に開発できず、生産性が低下。

**受け入れ基準**:
- Phase 1c で OSS キャッシュ対応
- ドキュメントのオフライン閲覧を提供
- オフライン動作確認シナリオを Runbook 化

**優先度**: COULD

## DX-LD-007: ラップトップ要件

**現状**: 低スペックラップトップでは k1s0 ローカル開発が動かず、開発生産性に格差が発生する。

**要件達成後**: 以下を推奨スペックとする。

- **CPU**: Apple Silicon M2 以上 / Intel Core i7 以上 / AMD Ryzen 7 以上
- **RAM**: 32GB 以上（16GB は最低限、kind + 主要 OSS 稼働で 12GB 使用）
- **ストレージ**: SSD 512GB 以上
- **OS**: macOS 13+ / Windows 11 + WSL2 / Ubuntu 22.04+

情シス部門と調達部門で標準機種選定時に反映。既存低スペック機の利用者にはリモート開発環境（GitHub Codespaces 互換のリモート Dev Container、Phase 2+）を代替提供。

**崩れた時**: スペック不足でローカル開発が遅く、開発者が k1s0 を敬遠。

**受け入れ基準**:
- 推奨スペックを情シス調達に共有
- 低スペック機向けリモート開発環境を Phase 2 で検討
- ラップトップ性能チェックを `k1s0 doctor` に統合

**優先度**: SHOULD
