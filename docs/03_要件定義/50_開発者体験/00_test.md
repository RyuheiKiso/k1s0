# DEV-TST: テスト要件

本ファイルは、k1s0 の tier1 自作領域（Rust）/ tier1 Dapr ファサード（Go）/ tier1 公開 API（多言語クライアントライブラリ）/ tier2・tier3 サービスに対するテスト要件を定義する。単体・統合・契約・E2E・負荷・プロパティ・ミューテーションの 7 階層にわたって、「何を」「どの水準で」「どの CI ステージで」検証するかを**受け入れ可否が二値で判定できる粒度**に揃える。

テスト要件の説得力は、品質の絶対値よりも「レグレッションが main にマージされる前にどこで止まるか」で決まる。k1s0 は tier2/3 から tier1 内部言語（Rust/Go）を不可視にする以上、契約破壊を tier1 側で先回りして検出できなければ、tier2/3 の開発者が tier1 の内部事情に引きずられる。本要件群はそのレグレッション封じ込め網を、定量基準と CI ステージの責務分担で定義する。

また、crypto と監査ログのハッシュチェーンのような**不変条件を持つ自作領域**は、例示ベースのテストだけでは網羅不能である。プロパティベーステストとミューテーションテストを要件として明示することで、自作領域のバグ検出力を稟議段階から測定可能にする。

## 要件本体

### DEV-TST-001: 自作領域（Rust）の単体テストカバレッジ下限

- 優先度: MUST（自作領域は OSS に還元する前提のため、低カバレッジでは外部公開に耐えない）
- Phase: Phase 1a
- 関連: `COM-GLO-004`, `OPS-CID-001`

現状、Rust 自作領域（ZEN Engine 統合 / crypto / 雛形 CLI / JTC 固有機能）は企画段階であり、カバレッジ基準が存在しない。基準がないまま実装を進めると、PR レビュー時に「十分にテストされているか」の判断が属人化し、MVP-0 で crypto に回帰バグが残ったまま稟議通過する危険がある。

本要件達成後の世界では、Rust 自作領域は **行カバレッジ 85% 以上**（構想設計 03 章の 80% から、自作領域に限り強化）で CI に組み込まれ、`cargo llvm-cov` がカバレッジレポートを artifact として出力し、閾値未達の PR は自動ブロックされる。新規 PR でカバレッジが前回 main より 1% 以上低下した場合も警告する。

崩れた場合、crypto の不変条件違反や ZEN Engine バインディングのメモリリークが本番で顕在化し、監査ログの改ざん検知が機能不全となる。結果として電子帳簿保存法対応を含む SEC カテゴリ要件の根拠が失われ、JTC 顧客への納品が不可能になる。

**受け入れ基準**

- `cargo llvm-cov --fail-under-lines 85` が CI で実行され、未達で PR ブロック
- カバレッジレポートが GHA artifact に保存され、Backstage TechDocs からリンク可能
- 自作領域 4 サブディレクトリ（zen / crypto / scaffold / jtc）ごとに個別集計
- main ブランチのカバレッジ推移が週次で Grafana ダッシュボードに可視化される

**検証方法**

- GHA ワークフロー定義に `cargo llvm-cov` ステップが含まれ、`--fail-under-lines 85` が指定されていることを監査
- 意図的に未テスト関数を追加した実験 PR がブロックされることを四半期ごとに確認

### DEV-TST-002: Go ファサードの単体テストカバレッジ下限

- 優先度: MUST（Dapr ファサードは tier2/3 から見た唯一の接点のため、契約破壊が即事故）
- Phase: Phase 1a
- 関連: `COM-GLO-005`, `DEV-TST-001`

Dapr ファサードは「tier2/3 から daprd を見せない」責務を持つ薄い層であり、実装ロジックは Rust 自作領域より少ない。一方で tier2/3 からの全呼び出しがこの層を通るため、**境界条件のバグがそのまま全テナントに波及**する。構想設計 03 章では tier1 全体を 80% と定めたが、ファサード層は境界ロジックの密度を踏まえて 75% を下限とする。

達成後は `go test -cover` が 75% 未満で PR をブロックし、`go.opencensus.io` 相当の計測で分岐網羅も確認される。Rust 領域より低い目標とするのは、Go ファサードが「Dapr SDK の薄いラッパー」で本質的なロジックが少ないためである。低すぎる目標でも形骸化するため、境界エラーハンドリング（タイムアウト / 認証失敗 / Dapr sidecar 非存在）に対する個別テストを必須化する。

崩れた場合、Dapr SDK のバージョンアップで SDK 側のデフォルト挙動が変わった際に検知できず、tier2/3 全サービスが同時に破壊される。tier2/3 側には原因が見えないため、初動切り分けだけで半日を失う。

**受け入れ基準**

- `go test -coverprofile -covermode=atomic` でカバレッジ 75% 未達時に PR ブロック
- タイムアウト / 認証失敗 / Dapr sidecar 未起動の 3 つは個別テストケースが存在
- `go vet` と `staticcheck` が CI で並走し、警告も PR ブロック対象
- カバレッジは Rust 自作領域と同じ Grafana ダッシュボードに並列表示

**検証方法**

- GHA ワークフローに `go test -coverprofile` と閾値チェックが含まれていることを監査
- 境界ケーステストの存在をリポジトリ検索（`TestDaprSidecarDown` 等の命名規約）で確認

### DEV-TST-003: Protobuf 契約テストの必須化

- 優先度: MUST（tier1-tier2 の契約破壊は企画書「Dapr 差し替え可能性」の前提を崩す）
- Phase: Phase 1a
- 関連: `ARC-T1-004`, `COM-GLO-005`, `DEV-TST-002`

tier1 内部および tier1 公開 API は Protobuf gRPC が必須（`CLAUDE.md` 技術スタック）。しかし Protobuf スキーマが変更された際、生成コードはコンパイル通過しても**セマンティクス互換性**は担保されない。フィールド番号の使い回しや enum 値の意味変更は典型的な事故パターンである。

達成後は、全 `.proto` 変更 PR で `buf breaking`（後方互換検査）と **Pact 契約テスト**（プロバイダ/コンシューマ双方の期待値検証）が CI で実行され、破壊的変更を含む PR は自動ブロックされる。Pact ブローカは tier1 Go ファサードを唯一のプロバイダとし、tier1 公開クライアントライブラリ（C# / Go / TS）をコンシューマとして登録する。

崩れた場合、tier2 サービスが tier1 の新バージョンで静かに壊れ、ステージングのない MVP Phase では本番で初めて発覚する。過去に他社で発生した事例では、enum 値の意味変更によるバッチ不整合の是正に 2 週間を要している。

**受け入れ基準**

- `buf breaking --against '.git#branch=main'` が全 PR で実行される
- Pact ブローカが `operation` namespace に常設され、tier1 公開 API の契約が登録済み
- コンシューマ側契約テストが全クライアントライブラリ（C# / Go / TS 初期 3 種）で PR ごとに実行
- スキーマ破壊 PR が自動ブロックされ、`release-candidate` ラベル付与時のみ承認バイパス可能

**検証方法**

- ダミーの破壊的 `.proto` 変更 PR を発行して CI が赤になることを MVP-0 受け入れ時に確認
- Pact ブローカの契約一覧を月次でレビューし、未登録のコンシューマがないことを確認

### DEV-TST-004: E2E テストは tier2 クリティカルパスに限定する

- 優先度: SHOULD（Phase 1b では手動確認で代替可能だが、Phase 2 以降の複数テナント時に必達）
- Phase: Phase 1b
- 関連: `OPS-CID-001`, `QUA-AVL-*`

E2E テストは実行時間と環境維持コストが高く、網羅性を志向すると CI が詰まる。構想設計 03 章は「main マージ後 / 日次 / 30 分以内」を上限としており、この枠内で守れる範囲を明示する必要がある。

達成後は、E2E の検証対象を **tier2 クリティカルパスの 3 シナリオ**（配信ポータルからの起動 → 業務 API 実行 → 監査ログ確認 / Keycloak SSO / tier2-tier1 間の Dapr PubSub）に限定し、それ以外は統合テスト（Testcontainers）に寄せる。シナリオ単位で失敗時のオンコール通知先と切り分け手順を TechDocs に明記する。

崩れた場合、E2E が肥大化して常時赤となり、アラート疲れで誰も見なくなる。結果として「E2E が通っていれば大丈夫」という心理的拠り所が失われ、リリース判断が属人的になる。

**受け入れ基準**

- E2E 対象シナリオが 3 件以内で、各シナリオの成功判定条件が明文化
- E2E の実行時間が 30 分以内（構想設計 03 章の上限と同値）
- 失敗時の一次通知先（SRE チームの Slack チャネル）と切り分け手順書（Runbook）へのリンクが TechDocs に存在
- E2E 非対象領域は Testcontainers 統合テストでカバーされていることがトレーサビリティ表に記録

**検証方法**

- Phase 1b 完了時に 30 日間の E2E 実行ログを集計し、実行時間中央値 / 失敗率 / 誤検知率を評価

### DEV-TST-005: MVP-0 での負荷試験実施

- 優先度: MUST（`QUA-PRF-001` p99 500ms の検証が稟議通過の根拠）
- Phase: Phase 1a
- 関連: `QUA-PRF-001`, `QUA-AVL-001`

現状、性能要件は構想設計段階で p99 500ms と合意されているが、実測値の裏付けがない。稟議段階で「目標値を置いたが測っていない」状態は、決裁者にとって最も拒絶されやすいパターンである。

達成後は、MVP-0 完了時点で **k6 による負荷試験**が実行され、tier1 公開 API の代表 3 エンドポイント（`k1s0.Log` / `k1s0.State.Get` / `k1s0.PubSub.Publish`）に対して「100 rps を 10 分間」の試験で p99 500ms 以下を満たすことが CI パイプラインの artifact として保存される。MVP-0 では本番規模ではなく、`QUA-PRF-001` の下限値のみを検証する。

崩れた場合、稟議突破後の Phase 1b でパフォーマンス問題が顕在化し、アーキテクチャ変更を強いられる。Dapr sidecar の CPU 制限値や PostgreSQL 接続プールの再設計は数週間の手戻りを意味し、ロードマップが遅延する。

**受け入れ基準**

- k6 スクリプトが `testdata/perf/` 配下に保存され、Phase 1a 完了時点で実行
- 試験結果レポートが GHA artifact と Backstage TechDocs から参照可能
- p99 レイテンシ / エラー率 / スループットの 3 指標が Grafana ダッシュボードに自動連携
- 目標未達時のチューニング方針（PostgreSQL 接続プール / Go ファサード GOMAXPROCS）を Runbook に記録

**検証方法**

- Phase 1a 完了レビューで k6 レポートをレビュー対象とし、稟議委員会に提示

### DEV-TST-006: テストデータは本番 PII マスキング前提で管理する

- 優先度: MUST（個人情報保護法 / 電子帳簿保存法の監査で本番データ混入は即 NG）
- Phase: Phase 1a
- 関連: `SEC-PRV-*`, `SEC-DAT-*`

テストデータに本番データを直接コピーすると、個人情報保護法の開示義務範囲が CI 環境まで拡大し、GHA runner Pod のログまで監査対象になる。構想設計 03 章は「本番データをテストに使用しない」と明記しているが、実運用では「一部だけ使いたい」圧力が必ず発生する。

達成後は、**テストデータ生成は合成データまたは PII マスキング済みデータに限定**し、`testdata/` 配下に本番由来データが混入することを CI で静的検出する（`pii-scanner` 相当を dry-run し、マッチすれば PR ブロック）。マスキングのルール（氏名 → `[NAME_MASKED]`、マイナンバー → `[MYNUMBER_MASKED]` 等）は `SEC-PRV-*` で定義した分類に従う。

崩れた場合、監査時に CI ログ / artifact / Backstage キャッシュから本番データが発見され、個人情報保護委員会への報告義務が発生する。加えて、JTC 顧客契約の守秘義務違反として賠償責任を問われる。

**受け入れ基準**

- `testdata/` 配下のデータが合成データであることを宣言する manifest ファイルが存在
- PII スキャナ（`gitleaks` 相当 + 自作 PII パターン）が全 PR で実行
- 合成データ生成用の faker ライブラリが雛形 CLI で自動セットアップ
- 違反検出時の対応 Runbook（secret rotation / 影響範囲通知）が SEC カテゴリと整合

**検証方法**

- Phase 1a 完了時に `testdata/` の全ファイルを手動監査し、以降は PII スキャナで継続検出

### DEV-TST-007: crypto / 監査ログ領域のプロパティベーステスト必須化

- 優先度: MUST（crypto バグは監査ログの改ざん検知を無効化し SEC-AUD の根拠を破壊）
- Phase: Phase 1a
- 関連: `SEC-AUD-*`, `SEC-KMS-*`, `DEV-TST-001`

crypto / 監査ログのハッシュチェーンは「任意の入力に対して特定の代数的性質を満たす」ことが品質基準である。例示テストでは不変条件の反例を見つけにくく、例えば「空のメッセージ」「UTF-8 の境界バイト列」「チェーン長 1 の特殊ケース」などを人間が網羅するのは非現実的である。

達成後は、Rust 自作領域の crypto / 監査ログモジュールに対して **`proptest` クレートによるプロパティベーステスト**が必須化される。具体的な性質として「署名-検証の往復一致」「ハッシュチェーンの連結順序不変」「暗号文復号の恒等性」「鍵ローテーション後の古い署名検証可能性」を最低限カバーする。

崩れた場合、crypto 実装に潜む不変条件違反が本番まで温存され、監査ログのハッシュチェーン連結ミスで改ざん検知が機能しない状態が長期間継続する。監査時に初めて発覚するため、過去全期間の監査ログの信頼性が失われる。

**受け入れ基準**

- `proptest` が Rust 自作領域の crypto / 監査ログクレートに導入済み
- 最低 4 つの性質テスト（署名検証往復 / ハッシュチェーン不変 / 復号恒等 / 鍵ローテ互換）が実装
- `proptest` の探索回数は PR 時 1,000 ケース、nightly は 100,000 ケース
- カウンターエグザンプル発見時の Minimal Reproducible Example（MRE）が自動保存

**検証方法**

- `cargo test --features proptest` が CI で実行され、カウンターエグザンプルを artifact に保存
- 四半期ごとに nightly 実行結果をレビュー

### DEV-TST-008: 自作領域のミューテーションテスト方針

- 優先度: SHOULD（Phase 1b で導入。MVP-0 ではカバレッジで代替可能）
- Phase: Phase 1b
- 関連: `DEV-TST-001`, `DEV-TST-007`

カバレッジが 85% であっても、テストが「通っただけで実際には何も検証していない」ケースがある（いわゆるアサーションなしテスト）。ミューテーションテストはコードに意図的な変更を加え、テストが落ちるかを測ることで「テストの実質的な検知力」を測定する。

達成後は、Rust 自作領域に対して `cargo mutants` を週次実行し、**Mutation Score（殺したミュータント÷生成ミュータント）70% 以上**を目標とする。MVP-0 では必須化せず、Phase 1b 以降で目標達成状況をレビューする。閾値未達時は即座に CI をブロックするのではなく、未達ミュータントの一覧を GitHub Issue として自動起票し、技術的負債として可視化する。

崩れた場合、カバレッジは高いが実質検知力の低いテストが蓄積し、Phase 2 以降のリファクタで回帰を見落とす。自作領域を OSS 還元する際に、外部コントリビュータからの「テストが形骸化している」指摘で信頼を失う。

**受け入れ基準**

- `cargo mutants` が週次 GHA cron で実行
- Mutation Score が Grafana ダッシュボードに公開
- 未達ミュータントが GitHub Issue として自動起票（テンプレート / ラベル `mutation-debt`）
- Phase 2 以降で Mutation Score 70% 未達時の改善アクションが運用 Runbook に明記

**検証方法**

- Phase 1b 完了時に 3 ヶ月分の Mutation Score 推移をレビュー

### DEV-TST-009: 雛形 CLI 生成直後の CI 通過保証

- 優先度: MUST（生成直後に CI が落ちる雛形は開発者体験の入口を破壊）
- Phase: Phase 1a
- 関連: `DEV-DEX-001`, `DEV-TST-001`, `DEV-TST-002`

雛形 CLI で生成したスケルトンが初回 CI で赤の場合、tier3 開発者は「そもそも何をどう直せばよいか」から詰まる。オンボーディング指標（30 分以内に初回デプロイ）が初回 CI 失敗で崩壊する。

達成後は、**雛形 CLI が生成した初期状態のリポジトリは PR 作成時に CI 100% グリーン**となることが保証される。雛形 CLI 自体の CI に「生成 → 即 push → CI 結果 assert green」のシナリオを組み込み、雛形テンプレートの破壊が即座に検知される。

崩れた場合、tier3 開発者が最初の PR でつまずき、雛形 CLI への信頼を失う。口コミで「k1s0 は入口でつまずく」と広がり、Phase 2 以降の社内普及が遅滞する。

**受け入れ基準**

- 雛形 CLI の CI に「生成 → git init → push → CI green」のエンドツーエンドテストが存在
- 初期生成物に含まれるユニットテスト / 契約テストのダミーが全て pass
- 雛形テンプレート 2 種（Go / TypeScript）それぞれで上記が担保
- 生成直後の CI 実行時間が 3 分以内（フィードバック速度維持）

**検証方法**

- 雛形 CLI リポジトリの CI が週次 cron で「生成→CI」シナリオを自動実行
