# 01. テスト戦略

本書は k1s0 プラットフォーム自体のテスト戦略と、tier2/tier3 が k1s0 上で構築する業務アプリのテスト戦略を定義する。Mike Cohn のテストピラミッドをベースに、単体・結合・契約・E2E・カオスの 5 階層でどこにどれだけ投資するかを定量化する。

## 本書の位置付け

テスト戦略が不在だと、tier2 開発者は「k1s0 上のアプリをどうテストすればいいのか分からない」状態になり、独自の（多くは貧弱な）テスト手法を発明する。結果、本番事故が増え、k1s0 全体の信頼が低下する。本書は tier2/tier3 の全チームが同じテスト基準で動けるように、テンプレートと受け入れ条件まで踏み込む。

## DX-TEST-001: テストピラミッド

**現状**: テスト投資が E2E に偏ると、テスト実行が重くなりフィードバックサイクルが 1 時間単位になる。逆に単体偏重ではインテグレーション問題を検出できない。

**要件達成後**: 以下の比率を目標テストポートフォリオとする。

- **単体テスト**: 全テストの 70%、実行時間 PR あたり 2 分以内
- **結合テスト（tier1 サービス間、外部依存モック）**: 20%、実行時間 PR あたり 5 分以内
- **契約テスト（tier1 ↔ tier2 SDK の API 契約）**: 5%、実行時間 PR あたり 3 分以内
- **E2E テスト（staging 環境、本番相当シナリオ）**: 4%、実行時間 30 分以内（並列実行）
- **カオステスト（本番類似環境、障害注入）**: 1%、採用後の運用拡大時 実施、週次または月次

PR タイムでは単体＋結合＋契約（合計 10 分以内）を必須、E2E は main ブランチマージ後の deployment pipeline で実行。

**崩れた時**: テスト実行時間が長くなり、tier2 開発者が PR を小さく分割できず、レビューが滞る。

**受け入れ基準**:

- 全テスト層を リリース時点 までに整備
- PR 時点での合計テスト時間 10 分以内を維持
- 四半期ごとにテスト実行時間と失敗率を Grafana で可視化

**優先度**: MUST

## DX-TEST-002: 単体テスト基準

**現状**: 単体テストのカバレッジ基準がないと、重要コードにテストがない状態が蓄積する。

**要件達成後**: 以下のカバレッジ基準を定める。

- **tier1 API 実装（Go/Rust）**: 80% 以上（ステートメントカバレッジ）、重要経路は分岐カバレッジ 90% 以上
- **共通ライブラリ（Rust crate、Go module）**: 85% 以上
- **tier2 テンプレート（Backstage Software Template 出力コード）**: 雛形時点で 60% 以上のテストを含める

カバレッジ測定は Go は `go test -coverprofile`、Rust は `cargo llvm-cov` を使用。結果は Codecov 的な可視化ツールで PR に表示。

カバレッジは目安であり、品質の代替指標ではない。コア機能のミューテーションテスト（`mutest` / `gremlin` 等）で補完する（採用後の運用拡大時）。

**崩れた時**: 重要コードが未テストで本番リリースされ、インシデント発生で原因調査が難航。

**受け入れ基準**:

- CI でカバレッジ基準未達時に警告（リリース時点）、基準未達でブロック（リリース時点+）
- ミューテーションテストを 採用後の運用拡大時 で導入、重要モジュールで実施
- カバレッジレポートを PR コメントで可視化

**優先度**: MUST

## DX-TEST-003: 結合テスト基準

**現状**: 結合テストで外部依存を全てモックすると、実環境との乖離が発生する。逆に全部実依存だとテストが不安定になる。

**要件達成後**: 以下の原則で結合テストを設計する。

- **内部依存**: 同一 tier1 内のサービス間呼び出しは実依存（テスト用 k8s クラスタ起動）
- **外部 OSS**: Testcontainers で PostgreSQL / Kafka / Valkey / MinIO を起動、Mock は最小限
- **tier1 外部 API（Keycloak、OpenBao 等）**: WireMock または専用 fake 実装で代替
- **時刻・ランダム性**: 決定論的にするため注入可能に設計

結合テストは `make test-integration` で誰でも手元実行できる。CI では並列化。

**崩れた時**: モック地獄で「テストは通るが本番で落ちる」状態になる。

**受け入れ基準**:

- 結合テスト実行時間 PR あたり 5 分以内（並列込み）
- Testcontainers 採用で実依存テストを標準化
- 手元実行手順を Runbook-DEV-TEST-001 に記述

**優先度**: MUST

## DX-TEST-004: 契約テスト（Consumer Driven Contracts）

**現状**: tier1 API が破壊的に変更されると、tier2 のクライアントが気付かないうちに壊れる。

**要件達成後**: Pact（または類似の Consumer Driven Contracts ツール）で tier1 ↔ tier2/tier3 SDK の契約テストを実装する。

- **Consumer 側（tier2/tier3 SDK）**: 期待する API レスポンスを契約として記述、Pact ファイルを発行
- **Provider 側（tier1 API 実装）**: Pact ファイルを取り込み、全契約を満たすことを CI で検証
- **契約破壊検出**: Consumer が期待する契約を Provider が満たせない場合、CI で赤くなる
- **バージョン管理**: Pact Broker で契約バージョンを管理、互換性マトリクス（OR-EOL-006）と連動

**崩れた時**: 破壊的変更が tier2 の本番デプロイ時まで検出されず、リリース事故。

**受け入れ基準**:

- 契約テストを リリース時点 で導入、採用後の運用拡大時 で全 tier1 API 対応
- Pact Broker を Backstage と連動、契約カバレッジを可視化
- 契約違反 CI 失敗件数を月次レポート

**優先度**: SHOULD

## DX-TEST-005: E2E テスト

**現状**: E2E テストを全シナリオで書くと、実行時間が膨大になり CI がボトルネック。

**要件達成後**: E2E テストは以下の原則で選定する。

- **ゴールデンパス**: 代表的なユースケース 10〜20 本に絞る。全機能網羅は狙わない。
- **staging 環境で実行**: 本番と同一構成の 1/3 規模 staging に対して実行
- **並列実行**: シナリオ独立性を保ち、並列実行で 30 分以内完了
- **不安定テスト（flaky test）の排除**: 連続 3 回成功しないシナリオは quarantine、修正後復帰
- **フレームワーク**: Go 側は `godog`（BDD）、Rust 側は `cucumber-rs`、tier3 UI は Playwright

E2E は deployment pipeline で実行、PR 時点では実行しない（時間の観点）。

**崩れた時**: E2E が遅すぎて PR 投入が躊躇され、結果として main ブランチが不安定になる。

**受け入れ基準**:

- E2E シナリオを リリース時点で 10 本 / 15 本、採用後の運用拡大時 で 20 本
- 実行成功率 95% 以上（flaky test 除外後）
- 失敗時の再現可能性を Runbook で担保

**優先度**: MUST

## DX-TEST-006: カオステスト

**現状**: 計画的な障害注入なしに可用性 99% を保証することは、実証上不可能。

**要件達成後**: 採用後の運用拡大時で Chaos Mesh または LitmusChaos を導入する。

- **ネットワーク障害**: pod 間のパケットロス、レイテンシ増加、パーティション
- **ポッド障害**: ランダムな Pod 終了、CPU/メモリ枯渇
- **ノード障害**: ノードドレイン、kubelet 停止
- **ストレージ障害**: Longhorn ボリュームの一時切断、I/O エラー
- **依存 OSS 障害**: Kafka ブローカー 1 台停止、PostgreSQL プライマリ停止

カオス実験は以下のワークフローで実施。

1. **仮説定義**: 「X が起きた時、Y の SLO は維持される」を事前に明文化
2. **blast radius 限定**: 最初は小さな範囲で実施、徐々に拡大
3. **実行**: staging で自動化、本番は off-peak 時間帯で運用チーム立ち会い
4. **観測**: Grafana で SLO への影響を計測
5. **学習**: 結果を FMEA（OR-INC-007）にフィードバック

**崩れた時**: 想定外の障害で大規模事故、事前に検証していれば防げた状況が発生。

**受け入れ基準**:

- 採用後の運用拡大時 でカオスツール導入、3 シナリオ実施
- 採用側のマルチクラスタ移行時 で月次カオス実施を定例化
- カオス実験レポートを全社公開

**優先度**: SHOULD（採用後の運用拡大時）

## DX-TEST-007: セキュリティテスト

**現状**: セキュリティテストが手動監査のみだと、脆弱性検出が年次レベルでしか起きない。

**要件達成後**: 以下のセキュリティテストを CI/CD に組み込む。

- **SAST（Static Application Security Testing）**: Semgrep / CodeQL、PR 時実行
- **DAST（Dynamic Application Security Testing）**: OWASP ZAP、staging 環境で定期実行
- **SCA（Software Composition Analysis）**: Trivy / Snyk、依存 OSS の既知脆弱性検出
- **コンテナイメージスキャン**: Trivy でイメージ内脆弱性検出、重大度 High 以上でデプロイブロック
- **IaC セキュリティ**: Checkov / kube-score で Kubernetes マニフェストの脆弱性検出
- **Secret スキャン**: gitleaks で Secret 誤コミット検出

ペネトレーションテストは年 1 回、外部専門家に依頼（採用後の運用拡大時）。

**崩れた時**: 脆弱性を抱えたまま稼働し、監査指摘や実際の攻撃被害。

**受け入れ基準**:

- リリース時点 で SAST/SCA/Secret スキャン導入
- リリース時点 で DAST/IaC セキュリティ導入
- 重大度 High 以上の脆弱性ゼロを目標、1 営業日以内対応

**優先度**: MUST

## DX-TEST-008: テストデータ管理

**現状**: テストデータが属人化すると、同じバグの再現ができなくなる。

**要件達成後**: テストデータを以下の原則で管理する。

- **合成データ（Synthesized）**: テストで使うデータは合成データが基本、個人情報・業務機密を含めない
- **本番類似データ**: 本番データを使う場合は匿名化（Aircloak Insights または自前の匿名化パイプライン）
- **フィクスチャ管理**: Git でバージョン管理、テストコードから参照
- **プロパティベーステスト**: Go は `gopter`、Rust は `proptest` でランダムデータ生成、境界値検証

**崩れた時**: 本番データがテストに流入し、個人情報漏えいリスク。匿名化手順が属人化。

**受け入れ基準**:

- 匿名化パイプラインを リリース時点 で整備
- 合成データ生成ライブラリを Backstage で提供
- プロパティベーステストを重要モジュールで採用

**優先度**: SHOULD
