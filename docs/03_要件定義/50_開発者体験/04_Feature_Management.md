# 04. Feature Management

本書は k1s0 における Feature Flag（機能フラグ）の運用方針を定義する。FR-T1-FEATURE-001〜004 で定義した tier1 Feature API の上位にある運用プラクティスであり、flagd（OpenFeature 準拠）を中心に、フラグの命名、ライフサイクル、A/B テスト、段階ロールアウト、負債管理を確定する。

## 本書の位置付け

Feature Flag は「デプロイとリリースの分離」を実現し、安全な段階リリースと迅速なロールバックを可能にする。一方、運用ルールがないと Flag が数千件に膨れ、コードは `if flag.isEnabled(...)` だらけになり保守不能に陥る。本書は Flag を資産として運用するためのガードレールを定める。

## DX-FM-001: Feature Flag の種別

**現状**: フラグの目的が混在すると、運用ライフサイクルが統一できない。

**要件達成後**: Flag を以下の 4 種別に分類する。

- **Release Flag**: 開発中機能の段階リリース用。機能が安定したら削除対象（最長 3 か月）。
- **Experiment Flag**: A/B テスト用。実験終了後に削除（最長 6 か月）。
- **Ops Flag**: 緊急停止・トラブルシュート用。長期保有（永続）だが、四半期レビュー必須。
- **Permission Flag**: テナント別・ユーザ別の機能有効化。長期保有（永続）。

各 Flag 登録時に種別を必須メタデータとする。

**崩れた時**: Flag の削除が進まず、コードが複雑化、テストマトリクスが爆発。

**受け入れ基準**:
- Flag 登録時の種別選択を リリース時点 で必須化
- 種別別の Flag 件数を Grafana で可視化
- Release/Experiment Flag の在籍期間を監視

**優先度**: MUST

## DX-FM-002: 命名規則

**現状**: 命名が自由だと、似た Flag が重複登録されたり、目的が読み取れない。

**要件達成後**: 命名規則を以下とする。

- **形式**: `<tier>-<service>-<purpose>-<version>` 例: `tier2-order-new-pricing-v2`
- **種別プレフィックス**: Release は `rel-`、Experiment は `exp-`、Ops は `ops-`、Permission は `perm-`
- **テナントスコープ**: テナント限定 Flag は `tenant:<id>` タグ付与
- **オーナー**: オーナーチームをメタデータで必須記述

命名規則違反は Backstage の Flag 作成フォームでバリデーション。

**崩れた時**: 意図不明な Flag が増え、保守性が低下。

**受け入れ基準**:
- 命名規則ドキュメントを Backstage で公開
- Flag 作成時の自動チェック
- 既存 Flag の名前変更は非推奨化ライフサイクルで対応

**優先度**: MUST

## DX-FM-003: Flag ライフサイクル

**現状**: Flag の削除が後回しになると、技術的負債として累積する。

**要件達成後**: 以下のライフサイクル管理を実施する。

- **作成**: Backstage フォームで種別・オーナー・期限（Release/Experiment のみ）を登録
- **運用**: PR で Flag 評価コードを追加、本番でロールアウト
- **完了判定**: Release は機能が全テナントで安定、Experiment は分析完了
- **削除準備**: 削除 PR を自動生成、レビュー後マージ
- **削除**: flagd 設定とコード内参照を除去

期限超過 Flag は毎週レポート、オーナーに通知。3 か月以上超過で Product Council に escalate。

**崩れた時**: 削除忘れが蓄積、6 か月で 100 件の死んだ Flag がコードに残存。

**受け入れ基準**:
- ライフサイクル管理ダッシュボードを リリース時点 で提供
- 期限超過率 10% 以下を維持
- Flag 削除 PR を Backstage が自動生成（採用後の運用拡大時）

**優先度**: MUST

## DX-FM-004: 段階ロールアウトのパターン

**現状**: 新機能を一斉公開すると事故時の影響が大きい。

**要件達成後**: 以下のロールアウトパターンを標準化する。

- **Dogfooding**: 社内開発チームのみ有効化（Day 0〜7）
- **Tenant Pilot**: パイロットテナント 1 件で限定有効化（Day 7〜14）
- **Canary**: 全テナントの 10% で有効化（Day 14〜21）
- **Gradual**: 10% → 25% → 50% → 100% と段階的に拡大（Day 21〜35）
- **General Availability**: 全テナント有効化（Day 35+）

Argo Rollouts の Canary / Blue-Green と連動し、メトリクス悪化時に自動ロールバック。

**崩れた時**: 一斉公開で数万ユーザに影響、事故対応で大量のリソースが必要。

**受け入れ基準**:
- ロールアウトテンプレートを Backstage で提供
- リリース時点 で自動ロールバック機構連携
- 段階ロールアウト使用率（Release Flag の）80% 以上

**優先度**: MUST

## DX-FM-005: A/B テスト基盤

**現状**: A/B テストが属人的な実装だと、結果の信頼性が担保できない。

**要件達成後**: A/B テスト基盤を以下で整備する。

- **割当**: ユーザ / テナント ID をシードに一貫ハッシュで割当、同一ユーザは常に同じグループ
- **メトリクス**: OpenTelemetry Metrics と連動、グループ別に KPI を計測
- **統計的有意性**: 実験期間と最小サンプルサイズを事前計算、p-value < 0.05 で判定
- **実験管理**: Backstage で実験を登録、仮説・メトリクス・期間を必須記載
- **偏り検査**: A/A テストで割当の偏りを事前検証

**崩れた時**: 誤った結論で意思決定、業務改善が空振り。

**受け入れ基準**:
- 採用後の運用拡大時 で A/B テスト基盤整備
- 統計的有意性判定を自動化
- 実験結果の共有ダッシュボードを Backstage に提供

**優先度**: SHOULD（採用後の運用拡大時）

## DX-FM-006: Flag 評価のパフォーマンス

**現状**: Flag 評価のレイテンシが高いと、業務 API のレイテンシを悪化させる。

**要件達成後**: 以下の性能目標を定める。

- **Flag 評価レイテンシ**: p99 < 10ms（tier1 Feature API）
- **キャッシュヒット率**: 90% 以上（SDK 内でキャッシュ）
- **フラグ変更の反映時間**: 5 秒以内（flagd push 通知）

アプリ側 SDK はローカルキャッシュを保持、flagd からの push で更新。ネットワーク障害時はキャッシュから継続評価（フェイルセーフ）。

**崩れた時**: Flag 評価が遅く、全 API のレイテンシ悪化。

**受け入れ基準**:
- リリース時点 で性能計測と改善
- Grafana で Flag 評価レイテンシを可視化
- 性能目標 3 か月連続達成

**優先度**: MUST

## DX-FM-007: 監査とガバナンス

**現状**: Flag の変更履歴が不可視だと、本番事故時の原因特定が難航。

**要件達成後**: 以下の監査ログを保全する。

- **変更ログ**: 誰が・いつ・どの Flag を・どう変更したか（監査 API に記録、7 年保持）
- **承認フロー**: Production 環境の Ops Flag 変更は 2 人承認必須
- **ロールバック記録**: Flag による機能無効化も監査ログに記録
- **アクセス制御**: Release/Experiment は開発チーム権限、Ops は SRE 権限、Permission は管理者権限

**崩れた時**: Flag 変更が原因の事故で「誰が変えたか分からない」状態。

**受け入れ基準**:
- リリース時点 で監査ログ統合
- 変更履歴を Backstage で検索可能
- 四半期に監査レビュー実施

**優先度**: MUST
