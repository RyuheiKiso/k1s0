# 02. ゴールデンパス

本書は tier2/tier3 の開発者が k1s0 で業務アプリを構築する際の「標準経路（Golden Path）」を定義する。Backstage Software Template、雛形 CLI、推奨スタックを標準として用意し、逸脱するには ADR の justification を求めることで、組織全体の一貫性を保つ。

## 本書の位置付け

Netflix / Spotify が示したように、自由放任の内部プラットフォームは 1000 パターンの実装を生み、運用不可能になる。k1s0 は 採用側組織の規模で数十チームが使うため、ゴールデンパスを強く推すことで「悩まずに済む」開発体験を提供する。同時に強制はしない。逸脱する自由を残すが、逸脱には理由と責任を付与する。

## DX-GP-001: Backstage Software Template

**現状**: 新規アプリ開始時のボイラープレート生成が属人化すると、1 週間のセットアップ作業が全チームで繰り返される。

**要件達成後**: 以下のテンプレートを Backstage Software Template として提供する。

- **tier2 ドメインサービス（Go）**: Go + Dapr Go SDK + gRPC + PostgreSQL + Kafka の標準構成
- **tier2 ドメインサービス（Rust）**: Rust + tonic（gRPC）+ SeaORM + Kafka の標準構成
- **tier3 アプリ（React）**: React (Vite) + tier2 API クライアント + tier1 API SDK のフロントエンド
- **tier3 アプリ（Web Components）**: 軽量フロントエンド向け
- **JDM ルール集**: ZEN Engine JDM エディタ向けテンプレート
- **バッチジョブ**: CronJob + Dapr Workflow の標準構成
- **.NET Framework サイドカー**: 既存 .NET Framework アプリへの k1s0 統合用サイドカー

各テンプレートは Backstage から 1 クリックで生成でき、10 分以内にローカルで `Hello World` が動く。

**崩れた時**: テンプレートがないと、チームごとに独自の雛形が発生し、運用側の把握が追いつかない。

**受け入れ基準**:
- リリース時点 で 3 テンプレート（Go tier2、React tier3、バッチ）
- リリース時点 で 5 テンプレート
- リリース時点 で全 7 テンプレート
- 月次でテンプレート利用率を計測、未使用テンプレートは整理

**優先度**: MUST

## DX-GP-002: 雛形 CLI

**現状**: Backstage が使えない環境（オフライン、個人ブランチ切り替え中など）では雛形生成が滞る。

**要件達成後**: Rust 製の `k1s0` CLI を提供し、Backstage テンプレートと同等の雛形を CLI から生成可能とする。

- `k1s0 new app tier2-go my-app`: tier2 Go アプリを生成
- `k1s0 new app tier3 my-ui`: tier3 UI アプリを生成
- `k1s0 dev up`: ローカル開発クラスタ起動
- `k1s0 dev down`: ローカル開発クラスタ停止
- `k1s0 doctor`: 開発環境の健全性診断（必要ツールがインストールされているか等）
- `k1s0 login`: Keycloak 経由でログイン、短期 Token を取得

CLI は Windows / macOS / Linux で動作、Homebrew / Scoop / apt でインストール可能とする。

**崩れた時**: Backstage 障害時に全チームの新規アプリ開発が止まる。

**受け入れ基準**:
- リリース時点 で CLI v0.1 リリース
- リリース時点 で CLI v1.0 リリース、SemVer で管理
- CLI 自動アップデート機構を実装

**優先度**: MUST

## DX-GP-003: 10 分ルール（Onboarding Velocity）

**現状**: 新規開発者が業務アプリを開発開始するまでのオンボーディング時間が数日かかると、採用加速が阻害される。

**要件達成後**: 以下の 10 分ルールを達成する。

- **ラップトップ受領後 10 分**: 社内 SSO 連携、Dev Container 起動、Backstage カタログ閲覧可能
- **テンプレート選択から 10 分**: 新規アプリの雛形生成 → ローカル起動 → `Hello World` が tier1 API 経由で応答
- **コード変更から 10 分**: 変更コミット → CI 通過 → dev 環境デプロイ → 動作確認

これらを達成するためのチェックポイントを Runbook-DEV-ONBOARD-001 で詳述、新規参画者の所要時間を計測する。

**崩れた時**: 新規参画者が既存 .NET 開発に逃げ、k1s0 採用の勢いが失速。

**受け入れ基準**:
- リリース時点 で 10 分ルール達成、計測を開始
- 新規参画者 10 名のオンボーディング時間中央値 10 分以下
- 改善機会を月次レビュー

**優先度**: SHOULD

## DX-GP-004: 推奨技術スタック

**現状**: 自由な技術選択は短期的な最適化を生むが、組織的な運用保守負荷を増やす。

**要件達成後**: 以下を推奨スタックとして定める。逸脱には ADR での justification を求める。

- **tier2 言語**: Go（デフォルト）、Rust（性能要求がある場合）
- **tier3 言語**: TypeScript + React (Vite)（デフォルト）、Web Components（軽量要件）
- **データベース**: PostgreSQL（tier2 透過的に利用）
- **メッセージング**: Kafka（Dapr PubSub 経由）
- **キャッシュ**: Valkey（Dapr State 経由）
- **ワークフロー**: Temporal（複雑な業務フロー）、Dapr Workflow（軽量）
- **IaC**: Helm + Kustomize + Argo CD
- **観測**: OpenTelemetry SDK（全言語）、Prometheus、Grafana、Loki、Tempo
- **認証**: Keycloak OIDC、OAuth2 Proxy または Istio JWT Validation
- **ビルド**: Buildpacks（Paketo）または Dockerfile、マルチステージビルド必須

推奨外技術を使う場合は、ADR で (a) 推奨スタックで不足する点、(b) 運用保守責任の所在、(c) 撤退時のコストを記述する。

**崩れた時**: 技術が乱立、運用チームが全てに精通する必要が出て疲弊。

**受け入れ基準**:
- 推奨スタックを Backstage TechDocs に明示
- ADR フォーマットに「推奨逸脱理由」欄を追加
- 四半期ごとに推奨スタック採用率を計測

**優先度**: MUST

## DX-GP-005: ドキュメント・サンプル充実

**現状**: ドキュメントが API リファレンスのみだと、実際にどう使うかの手探りが続く。

**要件達成後**: 以下のドキュメントとサンプルを提供する。

- **Quickstart**: 各テンプレートの 15 分チュートリアル
- **Cookbook**: ユースケース別の実装例（イベント駆動ワークフロー、トランザクショナルアウトボックス、RBAC、マルチテナント、バックグラウンド処理、etc）15 本以上
- **Reference App**: フル機能を使った参照アプリ（ToDo アプリ、注文処理、承認ワークフロー）3 本
- **Anti-patterns**: やってはいけない実装パターン集
- **Migration Guide**: .NET Framework からの移行手順

ドキュメントは Backstage TechDocs、サンプルコードは Git リポジトリ（k1s0-examples）で管理。

**崩れた時**: 「どう書けばいいか分からない」が採用阻害の最大要因となる。

**受け入れ基準**:
- リリース時点 で Quickstart 全テンプレート、Cookbook 10 本
- リリース時点 で Cookbook 15 本、Reference App 3 本
- サンプルコードは CI で動作確認、壊れた状態で公開しない

**優先度**: MUST

## DX-GP-006: Inner Source とコントリビューション

**現状**: プラットフォームチームがボトルネックになり、tier2 開発者が待ち時間で非効率化。

**要件達成後**: k1s0 本体（tier1 API、共通ライブラリ、テンプレート）への Pull Request を tier2/tier3 の開発者も受け付ける Inner Source モデルを採用する。

- **Contributing Guide**: コントリビューション手順、コードスタイル、テスト基準、レビュー基準を明記
- **Code of Conduct**: Inner Source でも適用
- **メンテナ制**: プラットフォームチームが最終レビュー権を持つ、追加メンテナは実績に基づき任命
- **PR テンプレート**: 変更内容、動機、テスト、ドキュメント更新の項目
- **メンター制度**: 初回 PR の開発者にはメンターをアサイン、レビュープロセスを支援

**崩れた時**: プラットフォームチームがボトルネックになり、tier2 の改善要望が 6 か月溜まる。

**受け入れ基準**:
- Contributing Guide を リリース時点 で整備
- 四半期あたり Inner Source PR 10 本以上を目標（採用後の運用拡大時）
- 初回 PR からマージまでの平均日数 10 営業日以内

**優先度**: SHOULD（採用後の運用拡大時）
