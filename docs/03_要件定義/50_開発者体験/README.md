# 50_開発者体験

本カテゴリは tier2/tier3 の開発チームが k1s0 基盤を利用して業務アプリを構築する際の生産性基準を定義する。k1s0 が稟議通過後に実際に使われるかは、この「開発者体験（Developer Experience, DevEx）」の品質で決まる。どれだけ技術的に優れた基盤でも、学習コストが高く、ローカル開発が重く、テストが書きにくければ、tier2/tier3 は迂回して既存 .NET スタックに留まる。

## 本カテゴリの狙い

k1s0 の敵は「既存のやり方の慣性」である。既存 .NET Framework 開発は、VS + IIS + SQL Server の枠内で 10 年以上続いた開発文化が完成しており、摩擦が少ない。k1s0 への移行は、テンプレ生成からデプロイまで 10 分以内、ローカルで tier1 API をモックで触れる、テストは CI で 5 分以内に完了、という開発者ハピネスを実装することで慣性を超える。本カテゴリはその定量的基準を定める。

## ドキュメント構成

- [01_テスト戦略.md](01_テスト戦略.md): 単体・結合・E2E・契約テスト・カオステストの階層、テストピラミッド、カバレッジ目標

- [05_ローカル開発環境.md](05_ローカル開発環境.md): Dev Container、kind / minikube 連携、tier1 API モック、ホットリロード

## 読み順

通読する場合は 01 → 02 → 03 → 04 → 05 の順。01 は品質保証、02 は標準化、03 は測定、04 は運用、05 は手元の道具。
