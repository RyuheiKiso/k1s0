# ARC-API: API / SDK 契約要件

本ファイルは、tier1 が tier2 / tier3 / 外部パートナーに公開する API / SDK の契約管理、バージョニング、互換性ポリシー、契約テストの要件を定義する。

tier1 の存在意義は「Dapr の内部実装を隠蔽した安定した公開契約の提供」であり、この契約が安定しない限り tier2 / tier3 の開発者は毎月の基盤リリースに追従する負担を背負うことになる。本ファイルの要件は、tier2 / tier3 側が「tier1 の API は少なくとも半年は動く」と信頼できる状態を作ることを目的とする。

論点は 5 つに集約される。バージョニング方針（SemVer + 日付併記）、Deprecation の告知期間（6 ヶ月）、Protobuf スキーマの後方互換規則、SDK の多言語展開ポリシー、契約テストの CI 必須化である。これらが揃うことで、tier1 の基盤改善が tier2 / tier3 を巻き込まずに進む体制になる。

## 要件本体

### ARC-API-001: API バージョニングポリシー（SemVer + カレンダーバージョン併記）

- 優先度: MUST（バージョニングが曖昧だと tier2 側で「どのバージョンが壊れるか」の判断ができない）
- Phase: Phase 1a
- 関連: `COM-GLO-009`, `ARC-API-003`

現状、API バージョニングを単一ルール（SemVer のみ or 日付のみ）で運用すると、「Phase 2 で受け入れた SemVer v2.0.0 が Phase 1b のカレンダー v2026.04 と等価か」が読み取れない。

本要件が満たされた世界では、tier1 の公開 API は SemVer（`v2.3.1`）とカレンダーバージョン（`2026-04-01`）を併記し、両者の対応表を公開する。SemVer のメジャーアップは破壊的変更、マイナーは後方互換機能追加、パッチはバグ修正に限定する。カレンダーバージョンは Phase との整合性を可視化する用途に使う。

崩れた場合、tier2 側で「v1.9.3 から v2.0.0 への移行が何月のどの Phase に必要か」の判断ができず、tier1 のメジャーリリースが tier2 のロードマップに奇襲的に影響する。

**受け入れ基準**

- 全公開 API に SemVer + カレンダーバージョンが併記される
- 対応表が Backstage TechDocs で公開される
- メジャー / マイナー / パッチの変更基準が明文化されている

**検証方法**

- Backstage の API カタログ監査
- リリース PR の CI でバージョン整合を自動検査

### ARC-API-002: Deprecated API の移行猶予期間（6 ヶ月）

- 優先度: MUST（告知なしの削除は tier2 / tier3 の予期せぬ障害を引き起こす）
- Phase: Phase 1b
- 関連: `OPS-EOL-*`, `ARC-T2-001`

現状、Deprecated API を短期間（1〜2 ヶ月）で削除すると、tier2 / tier3 の改修が基盤リリースサイクルに追従できず、本番障害を引き起こす。

本要件が満たされた世界では、Deprecated となった API は最低 6 ヶ月間の移行猶予期間を経てから削除される。猶予期間中、API 応答には `Deprecation` ヘッダ（RFC 9745 相当）と `Sunset` ヘッダを含め、SDK のクライアント側でログ警告を出す。Deprecation 開始時点で代替 API が提供され、移行手順が Backstage TechDocs に掲載される。

崩れた場合、tier2 開発チームが改修時間を確保できず、tier1 リリース時の緊急対応（hotfix）を強いられる。信頼失墜により tier2 チームが tier1 SDK を使わず Protobuf 直接呼び出しに回避するようになり、ARC-T2-001 の前提が崩れる。

**受け入れ基準**

- Deprecation 通知から削除まで最低 6 ヶ月の猶予がある
- Deprecated API に `Deprecation` / `Sunset` ヘッダが付与される
- 代替 API が Deprecation 開始時点で提供されている

**検証方法**

- Deprecation 通知日と削除日の記録を CI で検証
- 四半期の Deprecation レビュー会議

### ARC-API-003: Protobuf スキーマの後方互換規則

- 優先度: MUST（`COM-CON-002` を実効化する運用ルール）
- Phase: Phase 1a
- 関連: `COM-CON-002`, `ARC-T1-004`

現状、Protobuf スキーマのフィールド番号再利用・型変更・必須化などの変更は、Rust / Go / Java / Python の各クライアントで沈黙的に挙動が変わり、本番で初めて露呈する。

本要件が満たされた世界では、tier1 の `.proto` ファイルは buf の breaking lint を CI で必須通過とし、以下の変更を禁止する: フィールド番号の再利用、フィールド型の変更、必須化（`optional` → `required` 相当）、oneof の再編成、ネストメッセージの削除。メジャーバージョンアップ時のみこれらの変更を許可する。

崩れた場合、tier2 のビルドは通るが実行時に Marshal エラーでサービスダウンする。原因特定に数時間を要する。

**受け入れ基準**

- buf breaking の CI 通過が必須である
- 禁止変更 5 種が明文化されている
- メジャーバージョンアップの承認フローが定義されている

**検証方法**

- CI の buf breaking 通過率モニタリング
- Protobuf リポジトリのスキーマ差分レビュー

### ARC-API-004: SDK の多言語サポート範囲と同期リリース

- 優先度: MUST（言語間で SDK のバージョンが不整合だと tier2 の本番挙動が言語で変わる）
- Phase: Phase 1a
- 関連: `ARC-T1-007`, `ARC-T2-003`

現状、Go SDK と Java SDK のリリースタイミングがずれると、tier2 A（Go）は最新挙動、tier2 B（Java）は旧挙動となり、Pub/Sub で渡されるイベント構造に差分が出る。

本要件が満たされた世界では、Phase 1a は Go / Java の 2 言語、Phase 1b で Python / TypeScript の計 4 言語を同一バージョン番号・同一リリース日でリリースする。各言語の SDK は同一 Protobuf スキーマから自動生成され、言語固有の手書きコードは「リトライ・認証クライアント・エラーマッピング」の 3 領域に限定する。リリース遅延が発生する場合は該当言語のみをホールドせず、全言語同時リリースを原則とする。

崩れた場合、tier2 A と tier2 B の挙動差分で数時間単位の障害調査が発生する。

**受け入れ基準**

- Phase 1a で Go / Java SDK が同一バージョンでリリースされる
- Phase 1b で Python / TypeScript を追加し計 4 言語が同期リリースされる
- 言語固有コードが 3 領域に限定される

**検証方法**

- SDK リリースパイプラインのバージョンタグ整合検査
- 多言語間の挙動差分 E2E テスト

### ARC-API-005: 契約テストの CI 必須化

- 優先度: MUST（契約テストなしの tier1 改修は tier2 / tier3 の後付け検知に頼る）
- Phase: Phase 1a
- 関連: `DEV-TST-*`, `ARC-T2-001`

現状、tier1 の API 変更が tier2 / tier3 のクライアントに与える影響を手動検証に頼ると、検知漏れで本番障害が発生する。

本要件が満たされた世界では、tier1 の全公開 API に対する契約テスト（Pact または同等の CDC テスト）が CI で必須通過となる。tier2 / tier3 側は pact ブローカーに期待契約を登録し、tier1 側はこれを検証する。契約違反時は tier1 のメインブランチへの merge が自動拒否される。

崩れた場合、tier1 の無害に見える変更（例: エラーコード文字列の微修正）が tier3 のパターンマッチ分岐を破壊し、本番インシデントを引き起こす。

**受け入れ基準**

- 全公開 API に CDC テストが存在する
- Pact ブローカー（または同等）が運用されている
- CI で契約違反時に自動拒否される

**検証方法**

- Pact ブローカーの検証結果を PR チェックで確認
- 四半期の契約テストカバレッジレビュー

---

### ARC-API-006: API 命名規約（Dapr 中立・ドメイン語彙）

- 優先度: SHOULD（Dapr 固有用語の漏出は ARC-T1-002 の Dapr 隠蔽原則を実質的に崩す）
- Phase: Phase 1a
- 関連: `ARC-T1-002`

現状、API 名に `PublishEvent` / `InvokeBinding` など Dapr 固有用語を使うと、tier2 開発者に「Dapr を知らないと tier1 を使えない」という誤解を生む。

本要件が満たされた世界では、公開 API の命名はドメイン中立な語彙（例: `EmitBusinessEvent` / `CallExternalSystem`）に統一され、Dapr 固有用語は内部実装に限定される。API レビューガイドラインに命名語彙リストが含まれる。

崩れた場合、Dapr 差し替え時（`COM-RSK-006`）に API 名の大規模改名が必要となり、破壊的変更が連鎖する。

**受け入れ基準**

- 公開 API 名に Dapr 固有用語が含まれない
- 命名語彙リストがレビューガイドラインに含まれる

**検証方法**

- API 命名の静的解析（禁止語彙リストマッチ）
