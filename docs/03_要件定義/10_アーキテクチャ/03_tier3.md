# ARC-T3: tier3 要件

本ファイルは tier3 業務アプリの要件を定義する。tier3 は顧客・現場ユーザー向けの業務 UI・バッチ処理・帳票出力などを提供する層であり、雛形 CLI で生成されたスケルトンを出発点として、配信ポータル経由で配布される。

tier3 の設計で最も重要な論点は「雛形からの派生を CI で検知すること」「配信ポータルを経由しない配布を禁止すること」「tier2 経由で業務機能を利用し tier1 への直接依存を避けること」「copyable whitelist による端末設定の監査可能な公開制御」の 4 点である。これらが崩れると、tier3 アプリの独自実装が運用工数を押し上げ、企画書で宣言した「2 名運用」前提が達成できない。

本ファイルの要件は、tier3 を作る顧客側開発者（JTC 既存エンジニア）が読者として想定される。雛形 CLI の使い方や配信ポータルの操作は構想設計および `DEV-*` 要件で扱い、本ファイルでは構造契約に限定する。

## 要件本体

### ARC-T3-001: tier3 は雛形 CLI で生成されたスケルトンから出発する

- 優先度: MUST（雛形外の独自実装は運用工数の主要因。未達で 2 名運用前提が崩壊）
- Phase: Phase 1a
- 関連: `COM-GLO-003`, `COM-GLO-004`

現状、tier3 の新規作成が手作業で行われると、ディレクトリ構造・Dockerfile・CI 設定・監査ログ送信コードが顧客ごとに独自化する。これにより基盤チームが顧客ごとのアプリを個別サポートする必要が生まれ、企画書で宣言した「2 名運用」前提が 5 顧客程度で破綻する。

本要件が満たされた世界では、tier3 アプリは必ず雛形 CLI（`k1s0 init` コマンド）で生成されたスケルトンを出発点とする。スケルトンには監査ログ送信・tier2 クライアント呼び出し・Helm Chart・CI パイプラインがプリセットで含まれる。雛形から外れた実装は `.k1s0/ancestry.json` のハッシュ検証で CI が検知し、差分が許容範囲外の場合は PR を拒否する。

崩れた場合、顧客 A のバグ修正が顧客 B では適用されず、同一の根本原因のインシデントが顧客数だけ重複発生する。

**受け入れ基準**

- tier3 の初期生成が雛形 CLI 経由のみである
- `.k1s0/ancestry.json` によるテンプレートハッシュ検証が CI で実行される
- 雛形差分の許容範囲がポリシーとして定義されている

**検証方法**

- 新規 tier3 プロジェクトの CI ログで雛形検証ステップの通過を確認
- 四半期 1 回の雛形更新時の差分伝播時間を計測

### ARC-T3-002: tier3 の配布は配信ポータル経由に限定

- 優先度: MUST（未経由配布は監査ログ・ライセンス義務（`COM-CON-004`）・端末設定管理の前提を崩す）
- Phase: Phase 1b
- 関連: `COM-GLO-007`, `OPS-REL-*`

現状、tier3 アプリを個別にビルドして顧客にパッケージ手渡しすると、配布履歴が Git 管理外になり、「いつ・誰が・どの端末に・どのバージョンを」配布したかの監査が成立しない。電子帳簿保存法対応（`COM-CON-005`）の監査証跡要件と整合しない。

本要件が満たされた世界では、tier3 の配布は必ず配信ポータルを経由し、配布履歴は監査ログ（`COM-GLO-008`）として記録される。配信ポータルは Backstage（開発者向け）と別 URL / UI で提供され、業務部門のエンドユーザーや情シス担当者が利用する。`copyable` フラグで許可された端末設定のみがコピー可能となる。

崩れた場合、未承認のバージョンが本番端末に入り込み、法的な監査応答で証跡が欠落する。稟議通過済みの範囲外機能が現場に展開されるコンプライアンスインシデントが発生する。

**受け入れ基準**

- 配信ポータル以外の配布経路が運用ポリシーで禁止されている
- 配布履歴が監査ログ（`COM-GLO-008`）に記録される
- Backstage と配信ポータルが別 URL / UI で提供されている

**検証方法**

- 配信ポータル側の配布履歴と監査ログの突合
- 四半期 1 回のコンプライアンス監査

### ARC-T3-003: 端末設定の copyable whitelist

- 優先度: MUST（誤公開で顧客機密情報漏洩、未実装で情シス運用手順が確立できない）
- Phase: Phase 1b
- 関連: `COM-GLO-007`, `SEC-DAT-*`

現状、端末設定（プロキシ設定・認証情報・固有パラメータなど）を自由にコピー可能にすると、本来公開してはいけない値（例: 顧客固有 API キー・社内 LDAP パスワード）が他顧客環境に露出する。逆にコピー不可にすると、情シスが端末ごとに手動設定する運用負荷が発生する。

本要件が満たされた世界では、端末設定ごとに `copyable: true/false` フラグを必ず付与し、`true` の設定のみが配信ポータルでコピー可能となる。デフォルトは `false` とし、明示的に true へ切り替える際は配信ポータルの管理者権限と承認ワークフローを経る。コピー操作は監査ログに記録される。

崩れた場合、個人情報保護法上の漏洩事故として報告対象となり、JTC 顧客の社内監査で契約違反判定を受ける。

**受け入れ基準**

- 全端末設定に `copyable` フラグが付与されている
- デフォルト値が `false` である
- `true` への切替が承認ワークフローを経る
- コピー操作が監査ログに記録される

**検証方法**

- 配信ポータルの設定 DB スキーマレビュー
- 承認ワークフローの Runbook 訓練

### ARC-T3-004: tier3 は tier1 に直接依存しない（tier2 経由を原則）

- 優先度: MUST（直接依存は tier1 リファクタが tier3 の広範囲改修に波及する経路を作る）
- Phase: Phase 1a
- 関連: `COM-GLO-003`, `ARC-T2-001`

現状、tier3 から tier1 の SDK を直接利用すると、tier1 の API バージョン変更が tier3 全顧客に影響する。tier2 経由であれば tier2 がバッファとなり、tier3 への波及を限定できる。

本要件が満たされた世界では、tier3 は原則として tier2 の API のみを呼び、tier1 SDK の直接利用は例外とする。例外として許容される範囲は、雛形に組み込まれた監査ログ送信・tier1 認証クライアント・設定取得の 3 機能に限定する。それ以外で tier1 SDK を使う場合は ADR を記録する。

崩れた場合、tier1 SDK の破壊的変更時に全 tier3 顧客の改修が同時に必要となり、基盤チームのリリース凍結が発生する。

**受け入れ基準**

- tier3 の依存解析で tier1 SDK 直接利用が 3 機能に限定されている
- 3 機能以外の利用は ADR 登録必須である
- 例外 ADR の一覧が四半期レビューされる

**検証方法**

- tier3 ビルド時の静的解析による依存検査
- 例外 ADR 件数の推移モニタリング

---

### ARC-T3-005: tier3 のサンドボックス制約

- 優先度: SHOULD（tier3 の誤動作が基盤全体に波及しないための最低限の隔離）
- Phase: Phase 1b
- 関連: `SEC-IAM-*`, `ARC-INF-006`

現状、tier3 アプリが Kubernetes 権限として namespace 管理者権限や cluster 権限を要求すると、tier3 のバグが基盤全体を破壊しうる。

本要件が満たされた世界では、tier3 は自 namespace のみに閉じ、PodSecurityStandard `restricted` を enforce され、ServiceAccount には tier2 / tier1 の API 呼び出しに必要な最小権限のみを付与する。ネットワーク的には Istio Ambient の waypoint proxy 経由でのみ tier2 に到達し、直接外部へ egress することは禁止する。

崩れた場合、tier3 のバグで namespace を跨いだファイル破壊やクラスタ設定変更が発生する。

**受け入れ基準**

- tier3 namespace に `restricted` PodSecurityStandard が enforce されている
- ServiceAccount 権限が最小である
- egress が waypoint proxy 経由に限定されている

**検証方法**

- RBAC 設定の静的解析
- Istio Ambient のトラフィックフロー監視
