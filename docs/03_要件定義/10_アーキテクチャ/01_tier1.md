# ARC-T1: tier1 要件

本ファイルは、k1s0 の存在意義そのものである tier1 ファサード層の要件を定義する。tier1 は「業務ロジックを持たない Dapr 隠蔽層」として tier2 / tier3 に抽象 API を提供する。ここに業務依存が混入すると、tier1 の内部リファクタが tier2/3 の契約破壊に波及し、プロジェクト全体の保守性が崩壊する。

本ファイルの要件は、tier1 の責務スコープ、Dapr 隠蔽原則、Go と Rust のハイブリッド実装境界、内部通信契約（Protobuf gRPC 必須）、公開 API の言語バッジ制度、ワークフロー振り分けの 6 つの論点に集中している。

読み方としては、まず `ARC-T1-001`（スコープ定義）と `ARC-T1-002`（Dapr 隠蔽）を理解した後、`ARC-T1-003`（Go/Rust 境界）へ進むのが自然である。内部通信の `ARC-T1-004` は `COM-CON-002` と対になり、公開 API の `ARC-T1-005` は `ARC-API-*` の前提となる。

## 要件本体

### ARC-T1-001: tier1 の責務スコープ（業務ロジックを持たない）

- 優先度: MUST（tier 境界の根本定義。未達で tier2/3 の契約設計が成立しない）
- Phase: Phase 1a
- 関連: `COM-GLO-001`, `ARC-T2-001`

現状、tier1 の責務が曖昧なまま実装を始めると、tier2 側の業務要件（例: 承認フロー・与信チェック）が tier1 にじわじわ混入する。一度混入すると tier1 の内部リファクタが業務仕様変更を引き起こし、「基盤改修のたびに業務エンドユーザーへ説明が必要」という最悪の運用負荷が生まれる。

本要件が満たされた世界では、tier1 は「Dapr / ランタイム抽象の提供」「crypto / 監査基盤」「tier2 クライアントライブラリ公開」「雛形 CLI 提供」の 4 機能に限定される。業務ドメインに関する if 文は 0 個、顧客ごとの振る舞いの分岐も 0 個を徹底する。業務知識を必要とする処理はすべて tier2 に引き剥がす。

崩れた場合、tier1 のリポジトリが JTC 顧客ごとのカスタムコードで膨張し、tier1 の PR 1 本が 5 つの顧客セグメントに影響を与える。基盤チームが業務要件の QA まで引き受けることになり、2 名運用が破綻する。

**受け入れ基準**

- tier1 リポジトリ内に業務ドメイン用語（顧客名 / 業界名 / 取引種別）が含まれていない
- tier1 PR の CI で業務ドメイン用語のブラックリスト検出が有効化されている
- tier1 の責務が「Dapr 抽象 / crypto / クライアントライブラリ / 雛形 CLI」の 4 点に列挙されている

**検証方法**

- CI 上の用語ブラックリスト検査
- 四半期 1 回のリポジトリレビューで業務依存コードの有無を目視確認

### ARC-T1-002: Dapr 隠蔽原則（daprd への直接アクセス禁止）

- 優先度: MUST（企画書 13 章「Dapr 差し替え可能性」の技術的実現根拠）
- Phase: Phase 1a
- 関連: `COM-GLO-005`, `ARC-INF-003`, ADR-0001

現状、tier2/tier3 のコードから daprd の HTTP API（`:3500`）や gRPC API（`:50001`）を直接呼び出すパターンが放置されると、Dapr を将来別ランタイム（例: Orleans / Akka / 自作）に差し替える際に tier2/3 コードの大規模改修が発生する。

本要件が満たされた世界では、tier2/3 は tier1 が公開する gRPC / クライアントライブラリのみを経由し、daprd の存在自体を知らない。Dapr ファサード（Go）は daprd を内部で呼び、tier1 の公開 API は Dapr 中立な語彙（例: `PublishEvent` ではなく `EmitBusinessEvent`）で設計される。Dapr Workflow / Temporal の振り分けも tier1 内部で完結する。

崩れた場合、Dapr の EOL / ライセンス変更（`COM-RSK-006`）で tier2/3 全 PR の改修が必要となる。また Istio Ambient 下でも Dapr ポートへの直接呼び出しは waypoint proxy 経由で可視化されず、監査の穴になる。

**受け入れ基準**

- tier2/3 コードから daprd のポート（3500 / 50001 / 50002）への直接アクセスが CI で拒否される
- tier1 公開 API の命名に Dapr 固有用語が含まれていない
- Dapr 差し替え可能性を担保するアダプタ層が Go で実装されている

**検証方法**

- CI 上の依存解析・静的解析（Dapr SDK import を tier1 以外で検出した場合に fail）
- API 仕様書の用語レビュー

### ARC-T1-003: Go ファサードと Rust 自作領域の境界判定ツリー

- 優先度: MUST（Rust / Go の境界が曖昧だと採番フェーズで要件の言語属性が決まらない）
- Phase: Phase 1a
- 関連: `COM-GLO-004`, `COM-CON-001`

現状、新機能追加時に「Rust で書くか Go で書くか」が個別判断になると、同じ自作領域内で言語が混在し、ビルド基盤・CI・デバッグ体験が二重管理になる。

本要件が満たされた世界では、機能ごとの言語選択は 1 つのフローチャートで判定される。判定軸は「自作領域か（COM-GLO-004 の 4 領域に該当するか）」「暗号・パース・評価など CPU バウンドか」「Dapr SDK を直接呼ぶか」の 3 つ。該当が自作領域 × CPU バウンドなら Rust、Dapr SDK を直接呼ぶ層なら Go、ツーリング / 雛形 CLI は Rust、連携・配管コードは Go を原則とする。例外採用は ADR として記録する。

崩れた場合、ZEN Engine 統合を Go で書いたうえで Dapr ファサードを Rust で再実装するような構成が発生し、FFI / CGO 呼び出しが両方向に発生して p99 レイテンシ予算を食い潰す。

**受け入れ基準**

- 言語選択のフローチャートが 1 枚の drawio で定義され `COM-GLO-004` から参照されている
- 自作領域 4 つ（ZEN Engine 統合 / crypto / 雛形 CLI / JTC 固有機能）の Rust 適用範囲が明示されている
- 例外採用の ADR テンプレートが存在する

**検証方法**

- 新機能 PR のレビューでフローチャートとの照合を実施
- リポジトリ言語統計の四半期レビュー

### ARC-T1-004: tier1 内部通信は Protobuf gRPC 必須

- 優先度: MUST（Rust / Go 言語境界の型整合を機械検証できる唯一の手段）
- Phase: Phase 1a
- 関連: `COM-CON-002`, `ARC-API-003`

現状、Rust と Go のサービス間で JSON REST を採用すると、Go 側で `interface{}` / Rust 側で `serde_json::Value` として中間データを保持することになり、スキーマ変更の検知が実行時まで遅延する。

本要件が満たされた世界では、tier1 内部の全サービス間通信が Protobuf + gRPC で統一される。`.proto` スキーマは単一リポジトリに集約し、buf / protoc-gen-* で Go / Rust の両 stub を自動生成する。後方互換チェックを CI で実施し、フィールド削除・型変更を PR 段階で拒否する。

崩れた場合、Rust-Go 境界で型が無言で不整合を起こし、本番障害の原因特定に数時間を要する。`COM-CON-002` の制約違反として要件採番自体が拒否される。

**受け入れ基準**

- tier1 内部の全 gRPC サービスが `.proto` スキーマを公開している
- CI で buf breaking 後方互換チェックが通過している
- JSON REST による内部通信の検出が CI 上で fail する

**検証方法**

- buf breaking / buf lint の CI 結果を PR 必須化
- Istio Ambient の L7 メトリクスで HTTP/JSON トラフィックが tier1 内部に存在しないことを確認

### ARC-T1-005: 公開 API の言語バッジ制度（[Go] / [Rust]）

- 優先度: SHOULD（内部言語を完全に隠蔽するのが理想だが、ローカル開発 CLI のデバッグ時には言語特性が現れるため可視化する）
- Phase: Phase 1b
- 関連: `COM-CON-003`, `ARC-T1-003`

現状、クライアントライブラリの挙動が Go 側実装か Rust 側実装かで、panic 時のスタックトレース形式や GC 停止挙動が異なる。tier2/3 開発者がデバッグ時にこれを知らないと、原因特定が遅延する。

本要件が満たされた世界では、tier1 公開 API のドキュメント（Backstage TechDocs）で各エンドポイントに `[Go]` / `[Rust]` バッジを明示する。これは tier2/3 がバッジを条件分岐に使うためではなく、デバッグ時の観測対象ログの切り替え判断に使うための情報提供である。バッジは API 自体の契約ではなく、ドキュメントレベルの付加情報として扱う。

崩れた場合、tier2/3 のデバッグ時に「なぜログ形式が違うのか」が不明となり、問い合わせが基盤チームに集中する。

**受け入れ基準**

- 全公開 API にバッジが付与されている
- バッジは API 契約の一部ではなくドキュメント上のメタ情報として明示される
- バッジの自動生成ツールが CI に統合されている

**検証方法**

- Backstage TechDocs レンダリング結果の目視確認
- API 仕様書の自動生成パイプラインのメタデータ検査

### ARC-T1-006: ワークフロー振り分け（Dapr Workflow / Temporal）

- 優先度: MUST（振り分け複雑化は `COM-RSK-002` として High リスク認定済み）
- Phase: Phase 1b
- 関連: `COM-RSK-002`, `ARC-EVT-003`

現状、Dapr Workflow と Temporal の双方を tier1 が同時採用する前提では、どのワークフローをどちらで動かすかの判断が属人化しやすい。運用工数 0.5 人月/年の前提が下振れする確度が 70% と査読で指摘されている。

本要件が満たされた世界では、振り分け基準は tier1 内部の振り分けサービスが OPA / Rego 記述のポリシーで決定する。基準は「実行時間 5 分以上 or 人的承認あり → Temporal、それ以外 → Dapr Workflow」を起点にし、境界ケース（例: 5 分 Saga + 人的承認）は明示的な拡張ルールとして追加する。tier2 からは単一 API（`StartWorkflow`）で起動し、背後の基盤は不可視にする。

崩れた場合、tier2 開発者が両基盤の API を使い分ける必要が生じ、tier1 の存在意義が失われる。振り分け判断のレビュー件数が月次で運用工数を圧迫する。

**受け入れ基準**

- 振り分けポリシーが OPA / Rego で記述され Git 管理されている
- MVP-1a で 20+ の実ワークフローパターンをシミュレーションした結果が文書化されている
- tier2 側は単一 API で起動し基盤は不可視である

**検証方法**

- ポリシーの単体テスト（OPA の conftest）
- 月次の振り分け監視ダッシュボードで判断件数をレビュー

### ARC-T1-007: SDK 多言語展開方針

- 優先度: SHOULD（JTC 顧客の既存資産に合わせるため複数言語対応は必須だが、Phase 1a では Go / Java の 2 言語に限定する）
- Phase: Phase 1b
- 関連: `COM-GLO-002`, `ARC-API-004`

現状、tier2 の実装言語を JTC 顧客ごとに自由選択（Java / Python / Node.js / Go）すると、tier1 SDK をすべての言語で同時提供するコストが膨大になる。

本要件が満たされた世界では、Phase 1a は Go / Java の 2 言語に SDK を限定し、Phase 1b で Python / TypeScript を追加する。SDK 生成は Protobuf スキーマから自動化し、言語ごとのサンプル・エラーハンドリング・リトライ戦略を統一する。

崩れた場合、SDK バージョン間で挙動差分が発生し、tier2 開発者が tier1 のリリースノートを信頼できなくなる。

**受け入れ基準**

- Phase 1a サポート言語が Go / Java の 2 言語である
- Phase 1b で Python / TypeScript 追加計画が Roadmap に記載されている
- SDK 生成パイプラインが Protobuf スキーマから自動化されている

**検証方法**

- SDK リリースパイプラインの自動化状況を CI 履歴で検証
