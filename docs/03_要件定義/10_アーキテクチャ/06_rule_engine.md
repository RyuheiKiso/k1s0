# ARC-RUL: ルールエンジン要件

本ファイルは、k1s0 の中核コンポーネントである ZEN Engine の統合、ルール DSL / JDM の記述規約、ルール配布（GitOps 経路）、バージョニングと A/B 評価、評価レイテンシ上限に関する要件を定義する。

ルールエンジンが重要な理由は単純である。JTC 顧客の業務ロジックの多くは「審査基準」「与信閾値」「承認ルート」などのルール駆動で表現でき、これらを tier3 業務アプリのコードに直書きすると、ルール改定のたびに全顧客の tier3 を再デプロイする運用が発生する。ZEN Engine を tier1 基盤として採用し、ルールを JDM（JSON Decision Model）で表現・配布することで、業務部門がルールを更新するだけで全 tier3 に即座に反映できる世界を作る。

本ファイルの要件は、ZEN Engine 固定方針・JDM 記述規約・GitOps 配布パス・A/B 評価・レイテンシ上限 3ms・ロールバック手順の 6 論点をカバーする。

## 要件本体

### ARC-RUL-001: ZEN Engine をルール評価エンジンとして採用

- 優先度: MUST（`COM-GLO-006` で定義済み。別エンジン採用は ADR 差し戻しが必要）
- Phase: Phase 1a
- 関連: `COM-GLO-006`, `COM-CON-004`

現状、ルール評価エンジンを自作するか、Drools / Easy Rules などの代替を採用するかの判断は、Phase 1a 着手時点で確定していないと実装チームが動けない。

本要件が満たされた世界では、k1s0 のルール評価エンジンは ZEN Engine（MIT ライセンス、GoRules 社提供）に固定される。Rust バインディング経由で tier1 内部に統合され、ルール DSL は JDM（JSON Decision Model）で記述される。独自パッチが発生した場合は上流還元を原則とし、独自 fork は回避する。バージョンは Phase 1a 着手時点で確定し、四半期 1 回の更新レビューで上げる。

崩れた場合、自作ルールエンジン開発で 3〜6 ヶ月の工数が追加発生し、稟議時の「自作領域を 4 つに絞る」前提が崩れる。Drools（AGPL 懸念）を選ぶと OSS ライセンス制約（`COM-CON-004`）に抵触する。

**受け入れ基準**

- ZEN Engine のバージョンが Phase 1a 着手時点で固定されている
- Rust バインディングが tier1 内部に統合されている
- 独自パッチの上流還元ルールが定義されている

**検証方法**

- Cargo.toml / Cargo.lock のバージョン固定確認
- 独自パッチ件数の四半期レビュー

### ARC-RUL-002: ルール DSL / JDM の記述規約

- 優先度: MUST（規約なしでは業務部門のルール改定が基盤チームのレビュー待ちになる）
- Phase: Phase 1a
- 関連: `ARC-RUL-001`, `BIZ-GOV-*`

現状、JDM の書き方が属人化すると、同じ与信判定ロジックを A チームは `decision-table`、B チームは `function` で書くような分裂が発生し、ルール監査の可読性が落ちる。

本要件が満たされた世界では、JDM の記述規約が定義され、業務部門が直接編集する層と基盤チームがレビューする層の責務境界が明確になる。規約は「決定表は `decision-table` ノード、式評価は `expression` ノード、他ルール呼び出しは `decision` ノード」の原則を含み、ネスト階層は最大 3 階層に制限する。命名規約（日本語ラベル + ASCII key）を統一する。

崩れた場合、同じロジックが複数記法で書かれ、監査部門がルール整合性を確認できず、内部統制上の弱点として指摘される。

**受け入れ基準**

- JDM 記述規約が Backstage TechDocs で公開されている
- ネスト階層 3 階層制限が lint で検証される
- 命名規約が自動検査されている

**検証方法**

- ルール lint の CI 通過
- 月次のルール可読性レビュー

### ARC-RUL-003: ルール配布（GitOps / 配信ポータル経路）

- 優先度: MUST（配布経路の分散は監査証跡の欠落を招く）
- Phase: Phase 1b
- 関連: `COM-GLO-007`, `OPS-REL-*`, `SEC-AUD-*`

現状、ルールを tier2 / tier3 のリポジトリに直接コミットすると、ルール改定のたびにアプリの再デプロイが必要となり、「業務部門がルール更新できる」という本来の価値が失われる。

本要件が満たされた世界では、ルール JDM は専用の Git リポジトリで管理され、ArgoCD（GitOps）経由で tier1 のルール配信サービスに同期される。配信サービスは tier2 / tier3 に対して gRPC で評価 API を公開する。業務部門はルールエディタ UI（Backstage プラグイン or 配信ポータル拡張）から編集し、PR 承認フローを経て本番反映される。全ルール変更は監査ログ（`COM-GLO-008`）に記録される。

崩れた場合、ルール改定が Git 管理外で行われ、誰がいつ何を変えたかの追跡が不可能になる。監査対応で契約違反判定を受けるリスクが発生する。

**受け入れ基準**

- ルールが専用 Git リポジトリで管理されている
- ArgoCD による GitOps 配布が運用されている
- 全ルール変更が監査ログに記録される

**検証方法**

- Git 履歴と監査ログの突合
- 四半期のルール配布訓練

### ARC-RUL-004: ルールバージョニングと A/B 評価

- 優先度: SHOULD（A/B 評価は Phase 1b 以降の価値。Phase 1a では単純バージョニングで十分）
- Phase: Phase 1b
- 関連: `ARC-RUL-003`, `ARC-API-001`

現状、ルール改定の影響を事前に測らずに本番投入すると、予期しない判定結果で顧客業務が止まるリスクがある。

本要件が満たされた世界では、ルールは SemVer 準拠のバージョン管理下にあり、新バージョンは A/B 評価（カナリア配布）で限定トラフィックに適用した後、全面ロールアウトされる。A/B 評価は tier1 のルール配信サービスが ZEN Engine の評価結果をシャドウ実行し、旧バージョンと新バージョンの差分を比較する。差分が想定外の場合は自動ロールバックする。

崩れた場合、ルール改定が即座に全トラフィックに影響し、誤判定による顧客業務影響がインシデント化する。

**受け入れ基準**

- ルールが SemVer バージョン管理下にある
- A/B 評価機能がカナリアトラフィック向けに動作する
- シャドウ実行による差分比較が実装されている

**検証方法**

- A/B 評価の E2E テスト
- カナリア判定差分の検知ログレビュー

### ARC-RUL-005: ルール評価のレイテンシ上限（p99 < 3ms）

- 優先度: MUST（`COM-RSK-001` の tier1 p99 500ms 目標を単独で食い潰さないため）
- Phase: Phase 1a
- 関連: `COM-RSK-001`, `QUA-PRF-001`

現状、tier3 業務アプリは 1 リクエストあたり 3〜5 回のルール評価を呼び出すことが想定される。1 回の評価が 10ms 超過すると、最悪で 50ms がルール評価に占有され、tier1 API p99 500ms 目標を単独で食い潰す計算になる。

本要件が満たされた世界では、ZEN Engine の評価レイテンシは p99 < 3ms を目標とし、実測で 3ms を超えた場合は評価結果キャッシュ（Redis レイヤ）の導入を発動条件とする。キャッシュ導入は運用コンポーネントを 1 つ増やすため、発動閾値を超えない限り採用しない。測定は Phase 1a の load test で実施し、結果を SLO ダッシュボードで常時監視する。

崩れた場合、tier1 API p99 が 500ms 予算を超過し、SLA 補償条項（`COM-RSK-001` の軽減策）が発動する。最悪ケースで Phase 2 本番運用開始前に SLO の再設定が必要となり事業計画に波及する。

**受け入れ基準**

- ZEN Engine 評価の p99 < 3ms が Phase 1a の load test で検証される
- 3ms 超過時のキャッシュ導入判定基準が運用 Runbook に記載される
- SLO ダッシュボードでレイテンシが常時監視される

**検証方法**

- k6 / JMeter による Phase 1a load test
- Grafana での継続監視

### ARC-RUL-006: ルールのロールバック手順

- 優先度: MUST（誤ルール適用時の復旧時間が長引くと業務停止が拡大）
- Phase: Phase 1b
- 関連: `ARC-RUL-003`, `OPS-OPR-*`

現状、ルール誤適用時のロールバック手順が曖昧だと、復旧判断で数十分を浪費する。

本要件が満たされた世界では、ルールのロールバックは前バージョンへの Git revert + ArgoCD 同期で 5 分以内に完了する。ロールバック操作は配信ポータルの UI から実行可能で、実行者・対象バージョン・実行時刻が監査ログに記録される。

崩れた場合、誤ルール適用で業務停止が 30 分以上継続し、顧客 SLA 違反となる。

**受け入れ基準**

- ロールバック手順が Runbook 化されている
- 配信ポータル UI からロールバック操作が可能である
- ロールバックが 5 分以内に完了する

**検証方法**

- ロールバック訓練を四半期 1 回実施し時間を計測
