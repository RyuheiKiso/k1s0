# ARC-T2: tier2 要件

本ファイルは、tier2 ドメインサービス層の要件を定義する。tier2 は k1s0 において「業務ドメインを持つサービス群」であり、tier1 が提供するランタイム抽象を消費しながら tier3 業務アプリへ業務機能のエンドポイントを供給する。

tier2 の設計で最も重要な論点は「tier1 の内部言語を知らない」「tier1 経由でしか他 tier2 と通信しない」「tier3 の雛形から呼ばれる側として API 安定性を守る」の 3 点である。本ファイルの要件はこの 3 つを検証可能な条件に落とし込む。

tier2 は JTC 顧客の既存資産（Java / Python / Node.js など）との整合性を優先するため、言語選定は tier1 と異なり自由度が高い。ただし自由度には tier1 クライアントライブラリ経由必須・配布導線の統一といった制約が付く。

## 要件本体

### ARC-T2-001: tier2 は tier1 の内部言語に依存してはならない

- 優先度: MUST（`COM-CON-003` と対。違反は要件採番自体を拒否する制約レベル）
- Phase: Phase 1a
- 関連: `COM-CON-003`, `COM-GLO-002`, `ARC-T1-002`

現状、tier2 のビルド時に tier1 の Rust / Go の内部クレートやパッケージを import することを放置すると、tier1 の内部リファクタ（例: モジュール名変更・crate 分割）が tier2 の PR を直ちに破壊する。この結合は tier1 の自由な改善を阻害し、結果として基盤の進化速度が落ちる。

本要件が満たされた世界では、tier2 は tier1 が公開する「SDK ライブラリ」または「Protobuf 経由の gRPC エンドポイント」のみを経由する。tier1 の内部パッケージ名・クレート名・内部構造体は tier2 のビルドグラフに現れない。CI の依存解析で `tier1/internal/*` や `tier1-core/*` への import が検知された場合は PR が自動拒否される。

崩れた場合、tier1 の内部モジュール名変更が tier2 の全顧客セグメントのビルドを破壊し、基盤チームが各顧客の tier2 コードの修正 PR を出す事態になる。2 名運用の前提が崩壊する。

**受け入れ基準**

- tier2 のビルド成果物依存解析に tier1 内部パッケージが含まれないこと
- CI で tier1 内部パッケージへの import が検知時に fail すること
- tier2 向けに公開される成果物の一覧が tier1 リポジトリの `PUBLIC_API.md` に列挙されていること

**検証方法**

- `go mod graph` / `cargo tree` / `mvn dependency:tree` による依存解析
- 月次の公開成果物一覧レビュー

### ARC-T2-002: tier2 間通信は tier1 経由を原則とする

- 優先度: MUST（直接通信は tier2 間結合を強め、tier 構造の存在意義を損なう）
- Phase: Phase 1a
- 関連: `ARC-T1-001`, `ARC-EVT-002`

現状、tier2 同士が HTTP で直接通信し合うと、tier2 の接続マトリクスが N×N に膨れ、1 つの tier2 の障害が直接伝播する。また Istio Ambient 下での AuthorizationPolicy を tier2 ペアごとに記述する運用負荷が発生する。

本要件が満たされた世界では、tier2 間通信は原則として tier1 の Pub/Sub（`ARC-EVT-002`）または tier1 の Service Invocation 抽象を経由する。直接通信は tier1 で受け切れない同期応答要件（例: 数十 ms の低レイテンシ要求）がある場合のみ例外扱いとし、ADR で記録する。直接通信を採用する場合でも、tier1 が発行する mTLS 証明書を使用し、Istio Ambient の waypoint proxy で AuthorizationPolicy を必ず経由させる。

崩れた場合、tier2 A の障害が tier2 B / C / D へ直接連鎖し、部分的な機能縮退（`QUA-*` グレースフルデグラデーション）が実現できなくなる。

**受け入れ基準**

- tier2 間の通信経路が tier1 経由を原則とすることが文書化されている
- 直接通信の例外が ADR で記録され、リストが四半期レビューされている
- Istio Ambient の AuthorizationPolicy で tier2 間直接通信がデフォルト deny である

**検証方法**

- Istio Ambient の L7 メトリクスで tier2 間直接通信を観測し、例外リストと照合
- 直接通信 ADR の件数推移を四半期レビュー

### ARC-T2-003: tier2 の言語選定自由度（JTC 既存資産との整合）

- 優先度: MUST（tier2 の言語を固定すると JTC 顧客の既存エンジニア資産を活かせない）
- Phase: Phase 1a
- 関連: `ARC-T1-007`, `BIZ-*`

現状、tier2 の言語を基盤側が固定すると、Java 中心の JTC 顧客 A・Python 中心の顧客 B・Node.js 中心の顧客 C のいずれかで学習コストが急激に上がる。一方で全言語の SDK を基盤側で同時提供するのも負荷が大きい。

本要件が満たされた世界では、tier2 の言語は Phase 1a 時点で Go / Java の 2 言語、Phase 1b で Python / TypeScript を追加する（`ARC-T1-007` と対応）。顧客の既存資産に合わせた言語選択が可能となり、tier2 のオンボーディング時間が半減する。サポート言語外（例: Ruby / Kotlin）で書かれた tier2 は tier1 SDK を利用できず、代替として Protobuf gRPC エンドポイントの直接利用が許可される。

崩れた場合、JTC 顧客の既存エンジニアが言語習得から始める必要があり、パイロット期間が 3 ヶ月延び、稟議時の期待効果（TCO 削減）が未達となる。

**受け入れ基準**

- Phase 1a で Go / Java の SDK が GA 品質で提供されている
- Phase 1b で Python / TypeScript SDK の追加計画が Roadmap に反映されている
- サポート言語外での Protobuf 直接利用手順が文書化されている

**検証方法**

- SDK のリリース状況を四半期レビューで確認
- tier2 オンボーディング実績（新規プロジェクトのキックオフから初回デプロイまでの日数）を計測

### ARC-T2-004: tier2 の配布方式（コンテナイメージ / Helm / ArgoCD）

- 優先度: MUST（配布方式の不統一は運用チームの手作業を増やし `OPS-*` 要件に波及）
- Phase: Phase 1a
- 関連: `OPS-CICD-*`, `COM-GLO-007`

現状、tier2 の配布を顧客ごとに独自スクリプトで行うと、デプロイ履歴が Git 管理外になり、監査・ロールバックが困難になる。

本要件が満たされた世界では、tier2 はコンテナイメージとして社内コンテナレジストリにタグ付き push され、Helm Chart または Kustomize マニフェストで ArgoCD が同期する GitOps 構成を前提とする。tier3 と異なり配信ポータル（`COM-GLO-007`）は経由せず、Phase 1a 時点は ArgoCD 画面から直接プロモートする。

崩れた場合、tier2 のバージョンが環境間で不整合となり、障害時のロールバックが 30 分以上かかる。

**受け入れ基準**

- tier2 の配布経路が ArgoCD GitOps で統一されている
- Helm Chart / Kustomize のテンプレートが基盤チームから提供されている
- ロールバック手順が Runbook に記載されている

**検証方法**

- ArgoCD 同期履歴の月次レビュー
- ロールバック訓練を四半期 1 回実施

### ARC-T2-005: tier2 の業務ドメイン境界明示

- 優先度: SHOULD（複数 tier2 が同じドメインを扱うと責務重複が発生）
- Phase: Phase 1b
- 関連: `ARC-T2-001`, `BIZ-GOV-*`

現状、tier2 のドメイン境界が暗黙のまま複数チームが tier2 を作ると、例えば「注文」ドメインが tier2-A と tier2-B の両方に存在し、データ整合性の責務がどちらか判然としなくなる。

本要件が満たされた世界では、tier2 ごとに境界づけられたコンテキスト（Bounded Context）を 1 つ以上所有し、それらを統合カタログ（Backstage）で一覧可能にする。新規 tier2 のコンテキストは既存と重複しないようレビュー承認を経て登録される。

崩れた場合、同一ドメインの業務イベントが 2 つの tier2 から二重発行され、tier3 側で重複処理が発生する。

**受け入れ基準**

- 全 tier2 のドメイン境界が Backstage のカタログに登録されている
- 新規 tier2 登録時に既存ドメインとの重複チェックが実施される

**検証方法**

- Backstage カタログのレビュー
- ドメインイベントの発行元監査
