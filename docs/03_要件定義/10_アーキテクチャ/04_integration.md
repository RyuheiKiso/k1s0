# ARC-INT: 外部連携要件

本ファイルは、k1s0 プラットフォームと外部システム（IoT エッジデバイス・既存業務システム・SaaS）との接続方式に関する要件を定義する。外部境界はセキュリティ信頼境界と一致するため、30_セキュリティ_データ の要件との整合が特に重要である。

外部連携で最もリスクが高いのは「プロトコルの多様性を tier2 / tier3 に露出させること」である。IoT は MQTT / gRPC、既存業務システムは SFTP / SOAP / DB 直結、SaaS は OAuth / REST と、プロトコルが分散している。これを tier1 の境界アダプタで吸収せず tier2 / tier3 へ透過すると、業務アプリごとにプロトコル固有のリトライロジック・認証実装・データ変換が乱立する。

本ファイルでは外部連携を 4 系統（IoT / 既存業務システム / SaaS / egress 制御）に分類し、それぞれの接続プロトコル・認証方式・データ変換層の配置・Egress ポリシーを要件として固定する。

## 要件本体

### ARC-INT-001: IoT エッジデバイスの接続プロトコル

- 優先度: MUST（IoT 連携は JTC 顧客の主要ユースケース。未達で Phase 2 の事業性が成立しない）
- Phase: Phase 2
- 関連: `ARC-T1-002`, `ARC-EVT-002`

現状、IoT エッジデバイスとの接続を tier3 業務アプリが直接担う構成にすると、デバイスごとの通信プロトコル（MQTT / gRPC / CoAP / Modbus over TCP）の差異がアプリ内部まで到達し、業務ロジックが通信詳細で汚染される。

本要件が満たされた世界では、IoT エッジとの通信は tier1 境界に配置される IoT ゲートウェイ（MQTT Broker / gRPC アダプタ）で終端される。ゲートウェイは受信データを Dapr Pub/Sub（`ARC-EVT-002`）の共通イベントスキーマに正規化し、tier2 以降は正規化済みイベントのみを受け取る。デバイス認証は X.509 クライアント証明書を原則とし、証明書の発行・失効は tier1 の crypto 基盤が担う。

崩れた場合、IoT デバイスのプロトコル変更（例: MQTT 3.1.1 → 5.0）が tier3 業務アプリの全顧客改修を引き起こし、ロードマップが 1 四半期遅延する。

**受け入れ基準**

- IoT ゲートウェイが tier1 境界で MQTT / gRPC を終端する
- 受信データが Dapr Pub/Sub の共通イベントスキーマに正規化される
- デバイス認証が X.509 クライアント証明書である

**検証方法**

- IoT ゲートウェイのログで正規化後のイベントスキーマを検査
- 証明書発行・失効の運用訓練

### ARC-INT-002: 既存業務システム（オンプレ）との接続方式

- 優先度: MUST（JTC の既存資産との共存が企画書の主要価値提案）
- Phase: Phase 1b
- 関連: `ARC-INF-005`, `COM-RSK-003`, [`../../02_構想設計/01_アーキテクチャ/05_接続と共存/02_レガシー共存パターン.md`](../../02_構想設計/01_アーキテクチャ/05_接続と共存/02_レガシー共存パターン.md)

現状、JTC の既存業務システムは SOAP / SFTP / DB 直結 / MQ（IBM MQ / RabbitMQ）など多様なプロトコルで稼働しており、これらを tier2 から直接呼ぶと tier2 実装が JTC 顧客ごとに分岐する。さらに corporate proxy 経由の到達性確保が Phase 1a の過小見積もりリスク（`COM-RSK-003`）に直結する。

本要件が満たされた世界では、既存業務システム連携は「アダプタパターン」「レプリカパターン」「共存パターン」「置換パターン」の 4 つに整理され、顧客ごとにパターンを選択する。Phase 1b ではアダプタパターン（tier1 境界に SFTP / SOAP 変換アダプタを配置）を優先し、レガシー側の改修を要求しない。アダプタは tier1 の Go 実装として配置し、tier2 からは gRPC 経由で統一される。

崩れた場合、tier2 側で SFTP 接続・XML パース・キャラクタコード変換を個別実装する羽目になり、顧客ごとに類似バグが散発する。JTC の稟議説得力（「既存資産を活かせる」）が失われる。

**受け入れ基準**

- 4 共存パターンが文書化されアダプタ実装テンプレートが提供される
- アダプタが tier1 境界で終端し tier2 からは gRPC で利用できる
- corporate proxy 経由の接続検証が Phase 1b 完了条件に含まれる

**検証方法**

- アダプタテンプレートの CI 自動検証
- 実環境 corporate proxy 下での疎通テスト

### ARC-INT-003: SaaS 連携の認証・認可（OAuth 2.1 / OIDC）

- 優先度: MUST（SaaS 認証トークン管理は漏洩時の影響が甚大）
- Phase: Phase 1b
- 関連: `SEC-IAM-*`, `SEC-KMS-*`

現状、SaaS（例: Salesforce / Microsoft 365 / Slack）連携のアクセストークンを tier2 / tier3 が個別保管すると、鍵管理の責任が分散し、失効・ローテーションの運用が属人化する。

本要件が満たされた世界では、SaaS 連携の OAuth 2.1 / OIDC フローは tier1 の連携基盤が一元実行し、アクセストークン / リフレッシュトークンは tier1 の KMS（`SEC-KMS-*`）で暗号化保管される。tier2 / tier3 は「SaaS 呼び出し抽象 API」を tier1 に対して呼び、tier1 側でトークン取り出し・リフレッシュ・失効監視を行う。

崩れた場合、トークン漏洩インシデントで tier2 / tier3 全箇所の調査が必要になり、インシデント対応時間が SLA を超過する。

**受け入れ基準**

- OAuth 2.1 / OIDC フローが tier1 連携基盤に集約されている
- トークンが tier1 KMS で暗号化保管される
- トークン失効時の tier2 / tier3 への通知経路が定義されている

**検証方法**

- トークン保管場所の監査
- 失効通知の疎通テスト

### ARC-INT-004: 外部境界におけるデータ変換層

- 優先度: MUST（外部の多様なスキーマを内部に透過すると tier2 / tier3 が汚染される）
- Phase: Phase 1b
- 関連: `ARC-INT-001`, `ARC-INT-002`, `ARC-EVT-004`

現状、外部システムの XML / CSV / Fixed-Length / JSON 各形式を tier2 / tier3 が個別解釈すると、同じ顧客情報が複数のスキーマで内部流通する。

本要件が満たされた世界では、データ変換（パース・スキーマ変換・バリデーション）は tier1 境界のアダプタで完結し、tier2 以降は Protobuf 定義の共通スキーマのみを受け取る。変換ルールは ZEN Engine（`ARC-RUL-001`）で記述可能にし、非エンジニアも変換ロジックの確認・改修ができる状態を目指す。

崩れた場合、同一業務エンティティ（例: 取引先マスタ）が tier2 A では XML パース済み、tier2 B では生 CSV のまま流通し、tier3 のレポート出力でデータ不整合が発生する。

**受け入れ基準**

- 外部データ変換が tier1 境界アダプタで完結する
- 変換後スキーマが Protobuf で統一される
- 変換ルールが ZEN Engine で記述可能である

**検証方法**

- アダプタのスキーマ検査テスト
- tier2 / tier3 で生データ（XML / CSV）が流通していないことを依存解析で確認

### ARC-INT-005: Egress 制御（外部宛通信のホワイトリスト）

- 優先度: MUST（無制限 egress は監査・セキュリティ・コスト（データ転送）の 3 方向でリスク）
- Phase: Phase 1a
- 関連: `ARC-INF-004`, `SEC-*`, `COM-CON-006`

現状、Kubernetes デフォルトでは Pod から任意の外部アドレスへ egress 可能であり、tier2 / tier3 の内部ロジックが想定外の外部サービスにデータを送信するリスクがある。国内保管原則（`COM-CON-006`）との整合性も確保されない。

本要件が満たされた世界では、全 tier の egress が Istio Ambient の waypoint proxy（または専用 egress Gateway）経由に限定され、許可された外部ホスト名のみが通過する。許可リストは GitOps で管理され、追加時はレビュー必須。国外 IP への egress は明示的な顧客同意がある場合のみ許可される。

崩れた場合、tier3 のバグで顧客データが外部解析サービス（例: 未承認のクラッシュレポート送信先）に送出されるインシデントが発生する。

**受け入れ基準**

- egress が waypoint proxy または egress Gateway 経由に限定される
- 許可リストが GitOps で管理されている
- 国外 egress が明示的承認制である

**検証方法**

- Istio Ambient の egress ログ分析
- 月次の許可リストレビュー
