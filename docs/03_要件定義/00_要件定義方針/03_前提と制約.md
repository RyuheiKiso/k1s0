# 前提と制約

本書は k1s0 要件定義フェーズで所与とする前提条件と制約条件を整理する。前提（前提として真であるもの）と制約（遵守しなければならないもの）を混同すると、要件の優先度判定と工数見積が破綻するため、区別して記述する。

## なぜ前提と制約を明文化するか

要件定義が進むと、記述者ごとに「何を所与と見なすか」が揺れ、ある要件では「オンプレであること」を所与として使い、別の要件では「オンプレか否かも含めて検討」してしまうことがある。このズレは稟議レビューで「この要件はクラウド前提では不要では」と指摘された時に露呈し、要件の解釈を全面再検討する羽目になる。

本章で前提と制約をまとめて固定化することで、要件本体の記述では「本書 00_要件定義方針/03_前提と制約.md の前提 P-BIZ-001 による」と参照するだけで根拠説明が完結する。前提自体が誤りだった場合は、この章の改訂をトリガとして影響要件を再評価するルールを 05_要件トレーサビリティ方針.md と合わせて運用する。

## 前提条件

前提条件は「要件定義を進める上で真として扱う事実・合意」である。前提が崩れた場合、該当前提に依存する要件は再検討が必要となる。前提条件には ID `P-<カテゴリ>-<番号>` を付与し、トレーサビリティを担保する。

### P-BIZ-001: 事業目的は企画書で凍結済み

JTC 情シスが OSS 積み上げで商用 PaaS 比 31% 安のオンプレマイクロサービス基盤を構築することは、企画書（01_企画/企画書.md）で合意済みである。本要件定義書は事業目的を再定義しない。事業目的が変更された場合は企画書の改訂を先行させ、本書はそれを受けて改訂する。

### P-BIZ-002: 5 年 TCO と年額費用は稟議提示値を基準とする

5 年 TCO 3.68 億円（規模中 3,000 名想定）、Phase 2 以降の年額 2,349 万円（人件費 2,139 万円 + 学習 210 万円）は稟議提示値であり、要件の優先度判定は本値を上回らない範囲で実施する。機能追加による費用超過は重大改訂として扱う。

### P-BIZ-003: 想定規模は小 1,000 / 中 3,000 / 大 10,000 名の 3 段階

試算前提に基づき、3 段階の規模別に非機能要件を記述する。中規模 3,000 名を第一スコープとし、小規模は「縮退可能であること」、大規模は「水平拡張可能であること」を要件化する。単一規模への最適化はしない。

### P-TEC-001: オンプレミス・閉域ネットワーク完結

k1s0 はクラウドマネージドサービスを利用せず、オンプレ VM 上で完結する。クラウド連携（例: 外部 SaaS へのイベント配信）は tier3 の責務であり、tier1 基盤はクラウド非依存を維持する。

### P-TEC-002: tier1 内部実装は tier2/tier3 から不可視

tier1 の Dapr / Rust / Go の実装詳細は、公開 API（クライアントライブラリと Protobuf gRPC エンドポイント）を通じてのみ tier2/tier3 から見える。tier1 は内部実装を差し替える自由を維持する。

### P-TEC-003: 採用 OSS は構想設計で選定済み

Dapr、ZEN Engine、Istio Ambient、Kubernetes（kubeadm）、Keycloak、OpenBao、CloudNativePG、Apache Kafka（Strimzi）、Valkey、MinIO、Grafana / Loki / Tempo / Pyroscope、Argo CD 等の採用 OSS は構想設計で選定済み。要件定義では各 OSS の機能・非機能を要件文に落とし込むが、別 OSS への置換検討は本フェーズでは行わない。

### P-TEC-004: Dapr ファサードは Go 実装、自作領域は Rust 実装

tier1 内部言語はハイブリッド構成とし、stable な Dapr Go SDK を使うファサード層を Go で、ZEN Engine 統合・crypto・JTC 固有機能・雛形 CLI を Rust で実装する。Dapr Rust SDK の Alpha 段階の不安定性を本フェーズでは許容しない。

### P-TEC-005: tier1 内部のサービス間通信は gRPC 必須

Protobuf gRPC を唯一の内部通信プロトコルとする。HTTP/JSON、GraphQL、独自バイナリは禁止する。契約定義は `src/tier1/contracts/` に集中管理し、CI で自動生成する。

### P-ORG-001: Phase 0〜1b は小規模体制

Phase 0 は起案者単独（FTE 0.5）、Phase 1a は起案者単独（FTE 0.5）、Phase 1b は 2 名体制（FTE 2.0、人月 6.8）を前提とする。本規模で実装可能な範囲を要件のスコープとする。Phase 2 以降の体制拡大は参考扱いとする。

### P-ORG-002: Rust 人材は育成 + 中途 + SES の 3 経路で確保

Rust 経験者の社内枯渇を前提に、Go 経験者の 6 か月育成（70%）、中途採用（30%）、SES 並走（補填）の 3 経路で人材確保する。確保確度 99% 以上を前提とする。

### P-ORG-003: JTC 稟議文化を尊重する

年次稟議サイクル、定量根拠の要求、監査部門の関与を所与とする。k1s0 の導入自体がこれら文化を破壊することを目指さず、むしろ ZEN Engine + JDM エディタで稟議ルール自動化を支援する（Phase 4 以降）方向で設計する。

## 制約条件

制約条件は「遵守しなければならない外部条件」である。法律・規格・既存資産・外部契約など、k1s0 側で選択できないものを列挙する。制約 ID は `C-<カテゴリ>-<番号>` を付与する。

### C-LEGAL-001: 個人情報保護法 2022 年改正版準拠

JTC の内部業務システムを支える基盤として、個人情報保護法（令和 2 年改正、令和 4 年施行）を遵守する。仮名加工情報・匿名加工情報の取扱い、漏えい等報告義務（3 日以内の速報、30 日以内の確報）、外国にある第三者提供時の情報提供義務に準拠する。

### C-LEGAL-002: OSS ライセンス義務の完全遵守

採用 OSS のライセンス義務を逸脱しない。AGPL-3.0 の 6 本（Grafana / Loki / Tempo / Pyroscope / MinIO / Renovate）はネットワーク経由の提供がソース開示義務を発生させるため、外部境界遮断 + プロセス分離で無改変利用する。MPL 2.0（OpenTofu / OpenBao）はファイル単位の改変義務があるため無改変利用する。Apache 2.0 系は NOTICE 継承義務のみ。詳細は 01_企画/05_法務サマリ/OSSライセンス適合.md を参照。

### C-LEGAL-003: 下請法（SES 並走時）

Rust 人材 SES 並走の契約は下請法（親事業者の義務）を遵守する。書面交付、支払期日、不当な買いたたき禁止を契約に明記する。

### C-REG-001: J-SOX 内部統制報告制度

経営者評価対象の IT 全般統制（ITGC: Information Technology General Controls）の範囲に k1s0 が入る場合、アクセス管理・変更管理・運用管理の統制を整備する。監査証跡（tier1 Audit API のハッシュチェーン）を年次監査で提出可能とする。

### C-REG-002: 情報セキュリティ方針（社内規定）

JTC 社内規定（情報セキュリティポリシー、システム開発標準）を遵守する。具体条項は社内規定参照のためここでは抽象化するが、要件定義 30_非機能要件/E_セキュリティ では社内規定の主要要求（認証の多要素化、監査ログ保管期間、秘密情報の暗号化）を個別要件として再掲する。

### C-INF-001: VM ベース・既存ハード資産活用

Phase 1a は既存 VM 1 台を活用、Phase 1b は VM 3 台（4 vCPU / 16 GB / 200 GB SSD）を新規または流用で調達する。ベアメタル・新規ハード購入はしない。GPU も持たない。

### C-INF-002: ネットワーク要件（社内 LAN / VLAN）

既存社内 LAN の上に構築する。VLAN 分離、社内 NTP、社内 DNS（CoreDNS と連携）、既存プロキシ経由のインターネット接続（外部ライセンス検証用）を前提とする。専用線調達・外部 DC 移設はしない。

### C-EXIST-001: 既存 .NET Framework 資産との共存

JTC の既存業務システムは .NET Framework 4.x / ASP.NET Web Forms / WCF で構築されており、全量書き換えは数年単位で不可能である。k1s0 は tier3 拡張ポイントで .NET Framework を「サイドカー方式」または「API Gateway 経由方式」で共存させる。既存資産の強制リプレースはしない。

### C-EXIST-002: 既存認証基盤との統合

JTC 社内には既存の AD / LDAP / SAML IdP が稼働している。k1s0 の Keycloak を IdP プロキシ（Federation）として動作させ、既存認証基盤を上流に置く構成を採る。既存 IdP の廃止は Phase 5 以降の別議論とする。

### C-EXIST-003: 既存監視・バックアップ基盤との併存

JTC 社内には既存のバックアップ基盤（Commvault 等）、既存監視基盤（Zabbix / SCOM 等）が稼働している。k1s0 は独自の監視スタック（Prometheus / Grafana Tempo / Loki）を持つが、Severity 1 のアラートは既存監視基盤にも転送する設計を要件化する（FR-EXT-MON-001 で記述。バックアップ基盤との併存は FR-EXT-BACKUP-001）。

### C-VEND-001: 外部商用 SaaS 依存禁止（運用に関して）

k1s0 の運用に不可欠な商用 SaaS（PagerDuty、Datadog、New Relic、CrowdStrike 等）への依存は禁止する。これは C-EXIST-003 と整合する。SaaS を採用する場合は tier3 の業務アプリ側の選択であり、k1s0 基盤の必須依存にしない。

### C-TIME-001: Phase 0 稟議は 2026 年 4Q までに通過させる

会計年度の稟議サイクルに合わせ、Phase 0 稟議通過を 2026 年 4Q（〜2026-12）までに達成する。Phase 1a 着手は 2027-01 目安。本制約は事業スポンサーとの口頭合意に基づく。

## 前提崩れ時のプロトコル

前提が実測・外部変化で崩れた場合、以下の手順で要件を再評価する。このプロトコルを先に固定化しないと、前提崩壊時にアドホックな意思決定が連発し、要件定義書の内部整合性が失われる。

1. 前提 ID を起点に 80_トレーサビリティ/01_要件ID索引.md の逆引きで影響要件を列挙する
2. 列挙要件の優先度と検収条件を、崩れた前提の下で再評価する
3. 要件の変更が企画書の事業目的に波及する場合、企画書改訂を先行させる
4. 要件のみ変更で済む場合、本書の改訂プロセス（01_要件定義プロセス.md の改訂フロー）を適用する
5. 変更履歴は ADR-0000 台帳と本書末尾の改訂ログに追記する

## 未決事項

本章の記述時点で未決である前提・制約を明記する。各項目は「責任者（役割）／決着期限（絶対日付）／判定基準／暫定値」を 1 セットで管理する。決着までは関連要件を「仮」として扱うが、暫定値は Phase 0 稟議通過の前提として合意済みのものを採用する。期限超過は 07_リスクレジスタ に RISK として昇格させる。

### U-001: Phase 2 以降の規模（中 3,000 名継続 or 大 10,000 名拡大）

**責任者**: 企画スポンサー（事業部長クラス）／ 構想設計リード
**決着期限**: 2026-10-31（Phase 1b パイロット 1Q 運用完了時点）
**判定基準**: Phase 1b パイロットでの tier2 アダプション実績（MAU、アプリ登録数）が KPI 目標比 120% 以上で「大 10,000 名拡大」、80〜120% で「中 3,000 名継続」、80% 未満で「スコープ縮小 → 撤退戦略」を起動
**暫定値**: 中 3,000 名を第一スコープ（P-BIZ-003）。B_性能拡張性の RPS 試算もこの前提
**影響要件**: P-BIZ-003、NFR-B-WL-001〜003、NFR-F-CHR-002、70_プロジェクト管理/02_WBSと工程表

### U-002: 既存 IdP との Federation 方式（SAML vs OIDC ブリッジ）

**責任者**: 情シス認証基盤担当 ／ セキュリティリード
**決着期限**: 2026-07-31（Phase 1a 後半、NFR-E-AC-001 実装着手前）
**判定基準**: 現行 IdP（Active Directory Federation Services）が出す SAML 2.0 アサーションで JWT クレーム（tenant_id、roles）が表現可能なら SAML、属性マッピングに制約がある場合は Keycloak を OIDC ブリッジとして挟む構成
**暫定値**: Keycloak を OIDC ブリッジとして使用（C-EXIST-002、方式非依存の記述）
**影響要件**: C-EXIST-002、FR-EXT-IDP-001〜002、NFR-E-AC-001、NFR-D-OBJ-003

### U-003: J-SOX 対象範囲（k1s0 が IT 全般統制の対象に含まれるか）

**責任者**: 経理財務部 J-SOX 統制担当 ／ 法務リード
**決着期限**: 2026-09-30（Phase 1b テナント受入開始前、監査年度開始 3 か月前）
**判定基準**: tier2 配下に売上・仕入・会計処理に直接関与する業務アプリが 1 つでも載る計画がある場合は IT 全般統制の対象。BC-LGL-005（責任分界）の契約条項と 60_事業契約/10_ガバナンス の統制記述に影響する
**暫定値**: 対象に含まれる前提で要件記述（C-REG-001）。範囲外となれば監査系要件（NFR-E-MON-001 の保管期間、H-COMP-003 の証跡保全）を緩和可能
