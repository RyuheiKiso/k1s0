# 09. データ辞書

本書は k1s0 プラットフォームで扱う主要データエンティティとその属性を一元定義する。tier1 API のレスポンス、内部データベーススキーマ、監査ログ、メトリクスで共通して使われる識別子・メタデータ・型を本辞書で統制し、用語のばらつきや型不整合を防ぐ。

## 本書の位置付け

プラットフォームの各所で `user_id` と `userId` と `UserId` が混在すると、連携バグの温床になる。本書は命名規則・型・意味を標準化し、Protobuf / SQL / JSON / OpenTelemetry 属性で一貫した表現を可能にする。

## 命名規則

- **JSON / YAML**: `snake_case`（例: `tenant_id`）
- **Protobuf メッセージフィールド**: `snake_case`（例: `tenant_id`）
- **Protobuf メッセージ名**: `UpperCamelCase`（例: `TenantInfo`）
- **SQL テーブル・カラム**: `snake_case`（例: `audit_log.tenant_id`）
- **URL パス**: `kebab-case`（例: `/audit-log/events`）
- **Go 構造体フィールド**: `UpperCamelCase`（例: `TenantID`）、JSON タグで `snake_case` にマッピング
- **Rust 構造体フィールド**: `snake_case`
- **OpenTelemetry 属性**: `dot.case`（例: `k1s0.tenant.id`）

tier1 API の公開 IDL（Protobuf）では `snake_case` を正とし、クライアント SDK が言語慣習に合わせて変換する。

## 標準識別子

### tenant_id

テナントを一意に識別する文字列。マルチテナント分離の基軸となる最重要識別子。

- **型**: string
- **長さ**: 3〜64 文字
- **文字種**: `a-z`、`0-9`、`-`（ハイフン）のみ
- **例**: `tenant-acme`、`tenant-hr`、`tenant-finance`
- **用途**: 全 tier1 API リクエストに必須、ログ・メトリクス・トレースに付与
- **発行**: Backstage テナントオンボーディング時に発行、以後変更不可
- **関連**: FR-T1-INVOKE-001〜005、FR-T1-STATE-001〜005、FR-T1-PUBSUB-001〜005、FR-T1-SECRETS-001〜004、FR-T1-BINDING-001〜004、FR-T1-WORKFLOW-001〜005、FR-T1-LOG-001〜004、FR-T1-TELEMETRY-001〜004、FR-T1-DECISION-001〜004、FR-T1-AUDIT-001〜003、FR-T1-PII-001〜002、FR-T1-FEATURE-001〜004、BC-ONB-001〜006

### user_id

利用者を一意に識別する文字列。Keycloak の subject claim に対応。

- **型**: string（UUID 形式を推奨）
- **長さ**: 固定 36 文字（UUID v4）
- **例**: `01234567-89ab-cdef-0123-456789abcdef`
- **用途**: 認証済みコンテキストでの操作主体識別
- **発行**: Keycloak 発行
- **関連**: NFR-E-AC-001〜005、FR-T1-AUDIT-001〜003

### correlation_id

単一の業務トランザクションを横断するリクエスト識別子。クライアント発行。

- **型**: string（ULID 推奨）
- **長さ**: 26 文字
- **例**: `01J9T8QW8B3Y6MZJ47V4H4T8K2`
- **用途**: tier3 → tier2 → tier1 → 外部システムを通じた追跡、障害時の横串調査
- **発行**: 最上流（通常 tier3 UI）、途中のコンポーネントは引き継ぐ
- **関連**: FR-T1-LOG-001〜004、FR-T1-TELEMETRY-001〜004、06_エラータクソノミ.md

### trace_id / span_id

W3C Trace Context に準拠した分散トレース識別子。

- **trace_id 型**: string（16 バイト hex、32 文字）
- **span_id 型**: string（8 バイト hex、16 文字）
- **用途**: OpenTelemetry による分散トレース、Tempo で可視化
- **発行**: 最上流のアプリまたはサービスメッシュ
- **関連**: FR-T1-TELEMETRY-001〜004

### idempotency_key

冪等性保証のためのクライアント発行キー。重複実行を防ぐ。

- **型**: string（ULID 推奨）
- **長さ**: 26 文字
- **用途**: PubSub publish、Workflow 起動、Bindings 送信等の非冪等操作
- **有効期限**: 24 時間（Valkey に保存）
- **関連**: FR-T1-PUBSUB-001〜005、FR-T1-WORKFLOW-001〜005

## メタデータ

### 時刻

全ての時刻は UTC で保管、ISO 8601 形式で表現する。

- **型**: string（ISO 8601 UTC）
- **例**: `2026-04-19T12:34:56.789Z`
- **精度**: ミリ秒
- **保管**: PostgreSQL の `timestamptz`、Protobuf の `google.protobuf.Timestamp`
- **表示変換**: 利用者タイムゾーンでの表示は UI 層で変換
- **関連**: BC-I18N-003

### タグ・ラベル

リソースに付与するキー・バリュー形式のメタデータ。

- **キー**: `snake_case`、1〜63 文字、英数字と `-`・`_`
- **バリュー**: 任意文字列、0〜256 文字
- **数量制限**: リソースあたり 64 組まで
- **予約キー**: `k1s0.*` は k1s0 システム予約、ユーザ定義は `tenant.*` プレフィックス推奨
- **用途**: リソースの分類、フィルタ、コスト按分

### バージョン

SemVer に準拠したバージョン表現。

- **型**: string
- **形式**: `<MAJOR>.<MINOR>.<PATCH>`、必要に応じて `-alpha.1`、`+build.42`
- **例**: `1.2.3`、`2.0.0-rc.1`
- **関連**: OR-EOL-001

## イベントとログの共通属性

### AuditEvent

監査ログに含める標準属性。

- `event_id`: ULID（26 文字）、イベントの一意識別子
- `event_type`: 文字列、イベント種別（`tenant.created`、`user.login.success` など）
- `event_version`: SemVer 形式、イベントスキーマのバージョン
- `occurred_at`: ISO 8601、イベント発生時刻
- `tenant_id`: 関連テナント
- `actor_type`: `user` / `service` / `system`
- `actor_id`: 実行主体の識別子
- `target_type`: 操作対象リソース種別（`state`、`secret`、`workflow` 等）
- `target_id`: 操作対象リソース識別子
- `action`: 操作種別（`create`、`read`、`update`、`delete`、`invoke`）
- `outcome`: `success` / `failure` / `partial`
- `correlation_id`: 関連付け ID
- `trace_id`: 分散トレース ID
- `client_ip`: 発信元 IP（マスク済みで保管）
- `user_agent`: クライアント種別
- `extra`: JSON オブジェクト、イベント固有の追加情報

関連: FR-T1-AUDIT-001〜003、30_情報要件.md

### LogEntry

構造化ログの標準フィールド。

- `timestamp`: ISO 8601
- `severity`: `DEBUG` / `INFO` / `WARN` / `ERROR` / `FATAL`
- `message`: 人間可読メッセージ、英語
- `tenant_id`: 関連テナント
- `service`: 発生サービス名
- `module`: モジュール識別子（エラータクソノミと整合）
- `trace_id`: 分散トレース ID
- `span_id`: スパン ID
- `correlation_id`: 関連付け ID
- `error.code`: エラーコード（エラー時のみ）
- `error.category`: エラーカテゴリ（同上）
- `error.retriable`: 再試行可否（同上）
- `attributes`: JSON オブジェクト、任意属性

関連: FR-T1-LOG-001〜004、06_エラータクソノミ.md

### Metric

OpenTelemetry メトリクスの標準ラベル。

- `service_name`: サービス名
- `service_version`: バージョン
- `service_instance_id`: インスタンス識別子（Pod 名相当）
- `tenant_id`: テナント（テナント別メトリクスの場合）
- `k8s.namespace`: Kubernetes 名前空間
- `k8s.pod.name`: Pod 名
- `method`: 呼び出しメソッド（HTTP/gRPC）
- `status_code`: レスポンスコード

関連: FR-T1-TELEMETRY-001〜004

## データ分類と取扱

### 機密度レベル

全データに以下の機密度レベルを付与する。

- **Public**: 社外公開可（Backstage の公開ページ、SBOM 等）
- **Internal**: 社内限定（構成情報、内部ドキュメント）
- **Confidential**: 機密（業務データ、個人情報を除く）
- **Restricted**: 最機密（個人情報、認証情報、KMS 鍵）

レベルに応じて暗号化・アクセス制御・保管期間を差別化する。詳細は [../30_非機能要件/E_セキュリティ.md](../30_非機能要件/E_セキュリティ.md) および 30_セキュリティ系のデータ保護要件を参照。

### 個人情報タグ

個人情報を含むデータには PII タグを付与する。

- **PII**: 個人情報を含む
- **PII-Sensitive**: 要配慮個人情報（健康、信条等）
- **Pseudonymous**: 仮名化済み（復元可能）
- **Anonymized**: 匿名化済み（復元不能）
- **Non-PII**: 個人情報を含まない

タグに応じて暗号化・アクセスログ・保管期間を自動制御する。

関連: FR-T1-PII-001〜002、BC-LGL-004

### 保管期間

標準保管期間を以下に定める。個別要件で延長・短縮可。

- **ビジネスデータ**: 業務要件に応じる（通常 5 〜 10 年）
- **監査ログ**: 7 年（J-SOX、個人情報保護法、社内規程に合わせて）
- **アプリケーションログ**: 90 日（Hot）、1 年（Warm、圧縮済み）
- **メトリクス**: 15 日（高解像度）、1 年（中解像度）、3 年（低解像度）
- **トレース**: 30 日（サンプリング後）
- **バックアップ**: 30 日（日次）、1 年（月次）、7 年（年次）
- **Secret**: ローテーション間隔に応じる（90 日ローテーション標準）

関連: NFR-G-LIF-001、NFR-C-NOP-002、FR-T1-AUDIT-003

## データ整合性ルール

- **tenant_id 必須**: ビジネスデータは全て tenant_id を持つ、tier1 API の強制チェック
- **論理削除優先**: 物理削除は監査要件を満たす場合のみ、通常は `deleted_at` カラムで論理削除
- **参照整合性**: 外部キー制約を RDB で強制、整合性が崩れるケースは tenant_id 境界のみ
- **タイムスタンプ**: 全テーブルに `created_at`、`updated_at` を持つ、自動更新トリガ
- **楽観ロック**: 並行更新があるエンティティは `version` カラムで楽観ロック

## スキーマ進化

データスキーマ変更は以下の原則で進める。

- **後方互換な追加**: 新規カラム追加（NULL 許容または DEFAULT 値付き）は即時適用
- **後方互換な削除**: 古いカラムを `deprecated` コメント付きで残置、次メジャーバージョンで削除
- **型変更**: Blue-Green 方式で新旧両方を並行書き込み、移行完了後に旧フィールド削除
- **Protobuf フィールド番号**: 再利用禁止、削除時は `reserved` で番号予約
- **CI バリデーション**: `buf breaking` で破壊的変更を PR 時点で検出

関連: OR-EOL-001
