# 要件トレーサビリティ方針

本書は、k1s0 要件定義書における要件 ID 体系と、上流（企画）・下流（構想設計・基本設計・詳細設計・テスト）との双方向トレースの運用ルールを定義する。トレーサビリティを機械的に検証可能にすることで、要件変更時の影響範囲評価と、監査時のエビデンス提示を軽量化する。

## なぜトレーサビリティが必要か

要件定義書は執筆時点で内部整合していても、半年後に「企画のこのゴールに対応する要件はどれか」「この要件は構想設計のどのコンポーネントに落ちたのか」「テストはこの要件をカバーしているか」を手作業で辿ると、3〜5 人日かかる。採用検討レビューや年次監査の都度この労力をかけると、改訂が滞り、ドキュメント鮮度が劣化する。

要件 ID を一意に付与し、上流・下流のドキュメントからその ID を引くだけで対応が取れる状態を維持する。これは単なる管理コスト削減ではなく、「要件ベース検収」（仕様に合致したかではなく、合意した要件を満たしたかで検収する）を実現する前提条件である。トレーサビリティが崩れると、検収が個人の記憶と印象に依存し、採用初期 パイロット合格判定で紛糾する。

## 要件 ID 体系

### ID の構成

要件 ID は `<種別>-<カテゴリ>[-<サブカテゴリ>]-<通番>[枝番]` の可変長で構成する。種別は 2〜3 文字の英大文字、カテゴリ・サブカテゴリは 1〜12 文字の英大文字・数字、通番は 3 桁ゼロ埋め整数とする。既存要件を後方互換を保ったまま分割する場合のみ、通番の末尾に小文字 1 文字の枝番を付けて `001a` / `001b` のように表す。ハイフンで区切る。

FR / NFR は `<種別>-<カテゴリ>-<サブカテゴリ>-<通番>[枝番]`、BR / OR / DX / BC / MIG は `<種別>-<カテゴリ>-<通番>[枝番]` を基本形とする。

```text
FR-T1-STATE-001
^   ^  ^     ^
|   |  |     +-- 通番（3 桁ゼロ埋め）
|   |  +-------- サブカテゴリ
|   +----------- カテゴリ
+--------------- 種別
```

```text
BC-BIL-001a
^   ^   ^^^^
|   |   +------ 通番 + 枝番（後方互換を保った分割時のみ）
|   +---------- カテゴリ
+-------------- 種別
```

### 種別

7 種類の種別を定義する。

**FR（Functional Requirement、機能要件）**: 20_機能要件/ 配下の要件。tier1 公開 API、外部連携、情報モデルに関する要件。

**NFR（Non-Functional Requirement、非機能要件）**: 30_非機能要件/ 配下の要件。IPA 非機能要求グレード拡張の A〜I に対応。

**BR（Business Requirement、業務要件）**: 10_業務要件/ 配下の要件。tier2/tier3 開発業務、運用業務、配信業務に関する要件。

**OR（Operations Requirement、運用ライフサイクル要件）**: 40_運用ライフサイクル/ 配下の要件。インシデント、EOL、Runbook、負荷試験など運用プロセスに関する要件。

**DX（Developer Experience Requirement、開発者体験要件）**: 50_開発者体験/ 配下の要件。テスト戦略、ゴールデンパス、DevEx 指標、Feature Management、ローカル開発に関する要件。

**BC（Business Contract Requirement、事業契約要件）**: 60_事業契約/ 配下の要件。UX、法務、課金、ガバナンス、サステナビリティなど対外約束・事業運用に関する要件。

**MIG（Migration Requirement、移行計画要件）**: 70_プロジェクト管理/05_移行計画.md 配下の要件。段階移行、波次、撤退判断、並行運用に関する要件。

### カテゴリ・サブカテゴリ

種別ごとにカテゴリ・サブカテゴリを定義する。カテゴリは本書の章構成と一致させ、サブカテゴリは章内の節またはコンポーネント名に対応させる。

**FR のカテゴリ**:

- `T1`: tier1 公開 API（FR-T1-STATE-001 など）
- `EXT`: 外部連携（FR-EXT-DOTNET-001 など）
- `INFO`: 情報要件（FR-INFO-LOGMODEL-001 など）

**FR のサブカテゴリ（T1 下）**:

- `INVOKE` / `STATE` / `PUBSUB` / `SECRETS` / `BINDING` / `WORKFLOW` / `LOG` / `TELEMETRY` / `DECISION` / `AUDIT` / `PII` / `FEATURE`

**NFR のカテゴリ**:

- `A`: 可用性（NFR-A-CONT-001 など）
- `B`: 性能・拡張性
- `C`: 運用・保守性
- `D`: 移行性
- `E`: セキュリティ
- `F`: システム環境・エコロジー
- `G`: データ保護・プライバシー
- `H`: アーティファクト完整性・コンプライアンス
- `I`: SLI / SLO / エラーバジェット

**NFR のサブカテゴリ**: 各大項目内の中項目を英略称で表す。例: `CONT`（継続性）、`FT`（耐障害性）、`DR`（災害対策）、`REC`（回復性）、`WL`（業務処理量）、`PERF`（性能目標値）、`RES`（リソース拡張性）、`QA`（性能品質保証）、`NOP`（通常運用）、`MNT`（保守運用）、`IR`（障害時運用）、`ENV`（運用環境）、`SUP`（サポート体制）、`MGMT`（運用管理方針）、`TIM`（移行時期）、`MTH`（移行方式）、`OBJ`（移行対象）、`PLN`（移行計画）、`PRE`（前提と制約 / セキュリティ）、`RSK`（リスク分析）、`AC`（アクセス制御）、`ENC`（データ秘匿）、`MON`（不正追跡監視）、`NW`（ネットワーク対策）、`AV`（マルウェア対策）、`WEB`（Web 対策）、`SIR`（セキュリティインシデント対応）、`SYS`（システム制約）、`CHR`（システム特性）、`STD`（適合規格）、`FAC`（設置環境）、`ECO`（環境マネージメント）、`CLS`（分類）、`LIF`（ライフサイクル）、`PRV`（プライバシー）、`DES`（廃棄）、`INT`（完整性）、`KEY`（鍵管理）、`COMP`（コンプライアンス）、`AUD`（監査）、`TRN`（トレーニング）、`SLI` / `SLO` / `EB` / `SLA`。

**BR のカテゴリ**:

- `PLATUSE`: tier2/tier3 開発者の利用業務
- `PLATOPS`: 情シス運用業務
- `PLATGOV`: 情シス管理業務

**OR のカテゴリ**:

- `INC` / `SUP` / `ENV` / `EOL` / `EXIT` / `FMEA` / `LOAD` / `CAP`

**DX のカテゴリ**:

- `TEST` / `GP` / `MET` / `FM` / `LD`

**BC のカテゴリ**:

- `UX` / `I18N` / `ONB` / `TRN` / `COST` / `BIL` / `LIC` / `LGL` / `SC` / `GOV` / `AI` / `ANA` / `SUS`

**MIG のカテゴリ**:

- `STG` / `INV` / `WAVE` / `PATTERN` / `PILOT` / `WAVE2` / `DATA` / `ROLLBACK` / `PAR` / `LGL`

### 通番

FR / NFR はカテゴリ × サブカテゴリごと、BR / OR / DX / BC / MIG はカテゴリごとに 001 から採番する。削除された ID は再利用しない（欠番のまま残す）。既存 ID の意味を崩さずに分割する場合のみ、`001a` / `001b` の枝番を許可する。

### ID の命名例

- `FR-T1-STATE-001`: State API の基本機能（Get/Set/Delete）
- `FR-T1-WORKFLOW-005`: Workflow API の長期実行 Saga サポート
- `NFR-A-CONT-001`: tier1 対外 SLA 稼働率 99%（24h 基準、月 7.2 時間以内）
- `NFR-I-SLO-006`: Workflow API の SLO（永続性 / Determinism / 開始レイテンシ）
- `NFR-E-AC-003`: JWT tenant_id クレーム強制検証
- `NFR-B-PERF-002`: tier1 API p99 < 500ms
- `BR-PLATUSE-001`: tier2 開発者が SDK ワンラインで State を呼べる
- `BR-PLATOPS-004`: 運用 SRE が Dead Man's Switch でメトリクス途絶を検知できる
- `BC-BIL-001a`: 課金メータリングの基本 4 メトリック
- `MIG-STG-001`: Strangler Fig 段階移行戦略

## 双方向トレースの対象

トレースは以下 4 方向で維持する。方向ごとに維持責任者と整合確認頻度を定める。

### 上流: 企画 → 要件

企画書の事業目的・KPI・ロードマップ項目から、それを実現する要件 ID を引ける状態を維持する。起案者は 80_トレーサビリティ/02_企画要件マトリクス.md を執筆し、四半期ごとに整合確認する。企画改訂時は本マトリクスで影響要件を特定する。

### 下流: 要件 → 構想設計

要件 ID から、それを実装する構想設計のコンポーネント（tier1 内部モジュール、採用 OSS、ADR）を引ける状態を維持する。起案者は 80_トレーサビリティ/03_構想設計マトリクス.md を執筆し、構想設計改訂（新規 ADR、OSS 採択変更）時に更新する。

### 下流: 要件 → 基本設計以降

基本設計以降で作成される設計書・テスト仕様書・コードは、要件 ID を参照する形で執筆する。構想設計と基本設計の境界では、構想設計 ADR が要件 ID を引用する。基本設計以降では、PR の CHANGELOG エントリに要件 ID を記述する規約を導入する（リリース時点 着手時に CI チェック追加予定）。

### 横断: 前提・制約 → 要件

00_要件定義方針/03_前提と制約.md で定義した前提 ID（P-*-*）と制約 ID（C-*-*）から、依存要件を引ける状態を維持する。前提崩壊時の影響評価で参照する。80_トレーサビリティ/01_要件ID索引.md に逆引き表を格納する。

## 維持ルール

### 新規要件追加時

新規要件 ID を採番する際、以下 5 項目を同時に埋めることを必須とする。抜けがある状態でマージしない。

1. 本要件の根拠となる上流（企画ゴールまたは前提・制約）
2. 本要件を実装する下流コンポーネント（構想設計 ADR または採用 OSS）
3. 受け入れ基準（検収方法、測定可能な表現）
4. 優先度（MUST / SHOULD / COULD / WON'T + 判定理由）
5. 想定される 段階（採用初期 / 採用後の運用拡大時 / 採用側のマルチクラスタ移行時 以降）

### 要件変更時

既存要件 ID の意味を変更する場合、変更後の内容が旧内容と後方互換か否かで扱いを分ける。後方互換の拡張（例: 対応言語追加）は同 ID で改訂、非互換の変更（例: 検収基準の数値悪化）は旧 ID を `DEPRECATED` マークした上で新 ID を採番する。ID の意味ドリフトを避けるための措置である。

### 要件削除時

削除時は ID を `DEPRECATED` 状態で残し、削除理由（前提変更、スコープ外判定、統合等）を本書末尾の改訂ログに記す。再利用しない。

### 索引の再生成

80_トレーサビリティ/01_要件ID索引.md は、段階ゲート（1a → 1b 移行、採用初期 → 1c 移行等）の都度、`tools/validate_requirement_ids.ps1` で全要件を機械的にスキャンして再生成する。手作業の維持では鮮度が劣化するため、CI でも同スクリプトを実行して未定義参照を検出する。

## 受け入れ基準の書き方

受け入れ基準は「測定可能」「単一の検収者が単独で判定可能」「二値的」の 3 条件を満たす散文 + 箇条書きで記述する。曖昧な表現は要件としての検収不能を招くため禁止する。

**悪い例**: 「API は十分に高速であること」「運用が容易であること」「セキュアであること」

**良い例**: 「tier1 公開 API の全エンドポイントが、定常負荷（中規模 150 RPS）において p99 < 500ms を 1 週間連続で維持する。計測は tier1 側の OTel 計装で取得し、Prometheus で可視化する。」

受け入れ基準の粒度は、要件 1 件あたり 3〜7 箇条を目安とする。これを下回る場合は要件の粒度が粗すぎ、上回る場合は複数要件に分割すべきと判断する。

## 優先度の判定基準

優先度は要件 1 件あたり 1 つを割り当てる。複数優先度の併記は禁止する。判定理由を常に併記する。

**MUST**: 未達で 段階のゲート通過が不可能な要件。採用検討通過、リリース時点 デモ成功 / パイロット合格 / 運用体制受入のいずれかが直接崩れる水準。例: 「tier1 API p99 < 500ms（未達で SLO 約束が崩れ、採用検討差し戻し）」。

**SHOULD**: ゲート通過には必須ではないが、未達だと次段階の作業量が 50% 以上増える要件。例: 「CI 自動 SBOM 生成（未達だと 採用後の運用拡大時 監査対応で数人月の作業発生）」。

**COULD**: 余裕があれば実施する要件。未達でも後続段階で追加可能。例: 「tier1 API の Python クライアント SDK（採用後の運用拡大時 に繰り越せる）」。

**WON'T**: 本バージョンでは実施しない。将来バージョンでの実施可能性は参考扱い。例: 「マルチリージョン DR（採用側の全社ロールアウト 以降、当面 WON'T）」。

判定理由は 30 文字程度で、未達時の影響を具体的に書く。例: 「MUST（採用検討通過の前提。未達で リリース時点 承認不可）」「SHOULD（採用後の運用拡大時 で自動化未対応だと年 2 人月の手動対応が発生）」。

## 改訂ログの維持

本書末尾に改訂ログ節を設け、要件定義書全体の重大改訂（中規模・重大改訂）を時系列で記録する。軽微改訂は Git ログで十分なため記録しない。ログの記述形式は以下とする。

**以下のコードブロックは将来形式の例示であり、現時点の実改訂ではない。** `2026-XX-XX` という日付プレースホルダ、`FR-T1-WORKFLOW-005` や `FR-T1-DECISION-008` といった ID はいずれもフォーマット提示のためのサンプルであり、実在する要件 ID / ADR ID として索引やマトリクスから参照してはならない。実改訂時は本コードブロックを置き換えるのではなく、本書末尾の実改訂ログ節に追記する運用とする（下流設計チームがサンプル ID を実 ID と誤読する事故が 2026-04 に発生したため、本注記で構造的に再発防止する）。

```text
## 改訂ログ

### 2026-XX-XX - v0.2（リリース時点 キックオフ時）
- FR-T1-WORKFLOW-005 を追加（Temporal 採択に伴う）
- NFR-A-CONT-001 の数値を 99.0% → 99.5% に改訂（実測結果反映）
- DEPRECATED: FR-T1-DECISION-003（JDM 表現不可能と判明、代替要件 FR-T1-DECISION-008 へ）
```

改訂ログは 80_トレーサビリティ/ 配下のマトリクス更新とセットで commit する。ログのみ、マトリクスのみの片方更新は禁止する。

## ツーリング

要件 ID の抽出、双方向マトリクスの整合確認、受け入れ基準の曖昧語検出（「十分に」「適切に」「容易に」等のブラックリスト）は、リリース時点 着手時に CI スクリプトを導入する。本書時点では手運用とし、整合確認は 段階ゲートでのレビューで実施する。

具体のスクリプト仕様は、構想設計リードと起案者で別途決める。ツーリング実装までの過渡期は、起案者が四半期ごとに手作業で整合確認を行う。
