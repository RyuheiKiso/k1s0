# 07. リスクレジスタ

本書は k1s0 プロジェクトのリスクを一元管理するリスクレジスタである。発生確率 × 影響度で評価し、対応戦略（Avoid / Mitigate / Transfer / Accept）と責任者を明記する。Product Council で四半期ごとにレビュー、状態を更新する。

## 本書の位置付け

リスクの識別・評価・対応を属人運用すると、重大リスクが放置される。本書は全リスクを単一レジスタに集約し、「誰が、いつまでに、どう対応するか」を機械的に追跡する基盤となる。

## リスク評価基準

### 発生確率

- **High**: 6 か月以内に発生する可能性が高い（50% 以上）
- **Medium**: 1 年以内に発生する可能性がある（20〜50%）
- **Low**: 1 年以内の発生は考えにくい（20% 未満）

### 影響度

- **High**: プロジェクト停止、重大損害（数千万円以上）、個人情報漏えい、法令違反
- **Medium**: Phase 遅延、中程度損害、主要機能停止 1 日以上
- **Low**: 運用負荷増、軽微な損害、一時的な部分影響

### 優先度マトリクス

- **P1（最優先）**: High × High、High × Medium、Medium × High
- **P2（高）**: High × Low、Medium × Medium、Low × High
- **P3（中）**: Low × Medium、Medium × Low
- **P4（低）**: Low × Low

## 対応戦略

- **Avoid（回避）**: リスクそのものを発生させない選択（該当プロジェクト要素の中止・代替）
- **Mitigate（軽減）**: 発生確率または影響度を下げる（多くの技術リスク対応はここ）
- **Transfer（移転）**: 保険、契約条項、外部ベンダーへ責任移転
- **Accept（受容）**: コスト対効果で対応不要と判断、定期モニタリングは継続

## 技術リスク

### RISK-TEC-001: 採用 OSS の重大脆弱性

依存 OSS（Kafka、PostgreSQL、Istio 等）で Critical 脆弱性が発生、緊急パッチ適用による想定外のダウンタイム。

- **確率**: Medium、影響度: Medium、優先度: P2
- **戦略**: Mitigate
- **対応**: BC-SC-006（脆弱性対応 SLA 遵守）、BC-SC-007（情報共有）、Blue-Green デプロイで迅速適用
- **責任者**: プラットフォーム SRE リード
- **次回レビュー**: 四半期

### RISK-TEC-002: ZEN Engine / JDM の実装バグ

ZEN Engine（CBOR Rust OSS）に実装バグがあり、ルール評価結果が想定と異なる。

- **確率**: Medium、影響度: High、優先度: P1
- **戦略**: Mitigate
- **対応**: 採用 OSS のコミュニティ活発度監視、E2E テストで業務ロジック検証、フォールバック（ルール無効化＋手動）を準備
- **責任者**: tier1 Decision API 担当
- **次回レビュー**: Phase 1b ゲート時

### RISK-TEC-003: Istio Ambient Mesh の成熟度

Istio Ambient は比較的新しい機能で、本番事故リスクが sidecar 版より高い。

- **確率**: Medium、影響度: Medium、優先度: P2
- **戦略**: Mitigate
- **対応**: Phase 1b 時点で小規模検証、不具合時は sidecar 版への切替計画を保持（ADR で記載）
- **責任者**: プラットフォーム SRE
- **次回レビュー**: Phase 1b 終了時

### RISK-TEC-004: Dapr Go SDK 仕様変更

Dapr の upstream で破壊的変更が発生、k1s0 ファサード層の改修が必要になる。

- **確率**: Medium、影響度: Medium、優先度: P2
- **戦略**: Mitigate
- **対応**: Dapr stable SDK 採用（RC/alpha 回避）、Dapr のロードマップ監視、4 週間ごとの依存更新サイクル
- **責任者**: Dapr ファサード担当
- **次回レビュー**: 四半期

## 運用リスク

### RISK-OPS-001: 運用チームの bus factor

特定個人が Runbook の中核知識を独占、その人が不在時に重大対応が停滞。

- **確率**: High、影響度: High、優先度: P1
- **戦略**: Mitigate
- **対応**: NFR-C-NOP-003 の bus factor 2 以上、ペア運用、Runbook 品質向上、知識共有セッション定例化
- **責任者**: 情シス運用部門長
- **次回レビュー**: 月次

### RISK-OPS-002: オンコール疲弊

Severity 1/2 の夜間対応頻発で運用チームが疲弊、離職リスク。

- **確率**: Medium、影響度: High、優先度: P1
- **戦略**: Mitigate
- **対応**: ローテーション公平化、MTTR 短縮で対応時間削減、訓練（OR-INC-006）で対応効率化、人員補強
- **責任者**: プラットフォーム SRE リード
- **次回レビュー**: 月次

### RISK-OPS-003: 環境ドリフト

GitOps 管理にもかかわらず人手変更が混入、クラスタと Git リポジトリの差分が発生。

- **確率**: Medium、影響度: Medium、優先度: P2
- **戦略**: Mitigate
- **対応**: Argo CD Self-Heal、kubectl 直接実行の RBAC 禁止、週次ドリフト検査
- **責任者**: プラットフォーム SRE
- **次回レビュー**: 四半期

## セキュリティリスク

### RISK-SEC-001: 個人情報漏えい

tier1 API を経由して個人情報が漏えい、個人情報保護法違反。

- **確率**: Low、影響度: High、優先度: P1
- **戦略**: Mitigate
- **対応**: KMS 暗号化、RBAC 厳格化、監査ログ、アクセス最小化、PII タグ運用、年次ペネトレーションテスト
- **責任者**: 情報セキュリティ委員会
- **次回レビュー**: 月次

### RISK-SEC-002: サプライチェーン攻撃

公開レジストリ侵害で悪意ある依存が混入。

- **確率**: Medium、影響度: High、優先度: P1
- **戦略**: Mitigate
- **対応**: BC-SC-001（SLSA）、BC-SC-003（Sigstore）、BC-SC-004（社内プロキシ）、BC-SC-006（脆弱性対応）
- **責任者**: プラットフォーム セキュリティ担当
- **次回レビュー**: 四半期

### RISK-SEC-003: 内部不正

特権アカウントによる不正アクセス・データ持ち出し。

- **確率**: Low、影響度: High、優先度: P1
- **戦略**: Mitigate
- **対応**: 特権昇格の 2 人承認、全操作の監査ログ、異常検知（Falco）、退職時の速やかなアクセス取消
- **責任者**: 情報セキュリティ委員会
- **次回レビュー**: 四半期

## 法務・コンプライアンスリスク

### RISK-LGL-001: AGPL ライセンス違反

AGPL OSS（Grafana 等）の使用範囲拡大でソース公開義務発生。

- **確率**: Low、影響度: High、優先度: P1
- **戦略**: Avoid + Mitigate
- **対応**: BC-LIC-004 の AGPL 分離アーキテクチャ、採用 OSS の AGPL 回避を優先、法務レビュー必須
- **責任者**: 法務部 + プラットフォームリード
- **次回レビュー**: OSS 新規採用時

### RISK-LGL-002: 個人情報保護法改正対応漏れ

個人情報保護法の改正で追加義務発生、対応遅延で違反。

- **確率**: Medium、影響度: High、優先度: P1
- **戦略**: Mitigate
- **対応**: 個人情報保護委員会の情報購読、年次法令レビュー、改正時の迅速対応プロセス
- **責任者**: 法務部
- **次回レビュー**: 半期

## 事業リスク

### RISK-BUS-001: 稟議否決・予算削減

Phase 1 稟議否決、または Phase 2 以降で予算削減。

- **確率**: Low（稟議通過前）→ Medium（Phase 2 以降）、影響度: High、優先度: P1
- **戦略**: Mitigate
- **対応**: ROI 実証（BC-COST-006）、定量データで経営層説得、企画書差別化の継続アピール
- **責任者**: プロジェクトスポンサー（情シスマネージャ）
- **次回レビュー**: Phase ゲート時

### RISK-BUS-002: テナント採用停滞

tier2/tier3 が k1s0 採用を敬遠、期待したテナント数に到達しない。

- **確率**: Medium、影響度: High、優先度: P1
- **戦略**: Mitigate
- **対応**: BC-TRN-001（開発者教育）、DX-GP-001〜003（ゴールデンパス）、BC-ONB-005（オンボーディング教育）、Product Council でテナント満足度追跡
- **責任者**: プラットフォームリード
- **次回レビュー**: 月次

### RISK-BUS-003: キーマン離職

プロジェクト主要メンバー（tier1 リード、SRE リード等）の離職。

- **確率**: Medium、影響度: High、優先度: P1
- **戦略**: Mitigate
- **対応**: Bus factor 2 以上、Knowledge Transfer Session の継続、採用市場価値に見合う処遇、ワークライフバランス
- **責任者**: 情シス人事・情シスマネージャ
- **次回レビュー**: 半期

## 財務リスク

### RISK-FIN-001: TCO 超過

計画 TCO（5 年 3.68 億円中規模）を大幅超過、経営層の信頼毀損。

- **確率**: Medium、影響度: Medium、優先度: P2
- **戦略**: Mitigate
- **対応**: BC-COST-001〜006（FinOps）、四半期予実レビュー、Right-Sizing
- **責任者**: プラットフォームリード + 財務部門
- **次回レビュー**: 四半期

### RISK-FIN-002: 通貨・エネルギー価格変動

電力・ハードウェア価格高騰で運用コスト膨張。

- **確率**: Medium、影響度: Medium、優先度: P2
- **戦略**: Accept + Mitigate
- **対応**: 調達契約での固定単価（長期リース）、電力効率化（BC-SUS-001）、予備枠確保
- **責任者**: 調達部門
- **次回レビュー**: 半期

## 人的リスク

### RISK-HUM-001: 文化的抵抗

既存 .NET 開発者の「新技術への抵抗」で採用率低下。

- **確率**: Medium、影響度: Medium、優先度: P2
- **戦略**: Mitigate
- **対応**: BC-TRN（教育プログラム）、Success Story の共有、早期成功事例の可視化、抵抗者との対話
- **責任者**: プラットフォームリード + 情シスマネージャ
- **次回レビュー**: 四半期

### RISK-HUM-002: スキルギャップ

Kubernetes / Rust / Go / Istio 等の新技術スキル不足で採用ペース遅延。

- **確率**: High、影響度: Medium、優先度: P1
- **戦略**: Mitigate
- **対応**: BC-TRN-006（外部教育）、CNCF 認定支援、ペアプログラミング、メンター制度、採用強化
- **責任者**: 人事 + プラットフォームリード
- **次回レビュー**: 四半期

## リスクレビュープロセス

四半期ごとに Product Council でリスクレジスタ全体をレビューする。

1. **状態更新**: 各リスクの最新状況を責任者が報告
2. **新規リスク追加**: 新たに識別されたリスクを登録
3. **クローズ判断**: 対応完了またはリスクが消失したものをクローズ
4. **優先度見直し**: 発生確率・影響度を再評価
5. **対応アクション**: P1 リスクの対応アクションを確認、未完了は担当者に確認

レビュー議事録は Backstage に公開、全ステークホルダーが閲覧可能とする。
