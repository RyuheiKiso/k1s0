# 06. エラータクソノミ

本書は k1s0 プラットフォーム全体で統一的に使われるエラーコード体系（エラータクソノミ）を定義する。tier1 API の全エラー応答、tier2/tier3 アプリのエラー伝搬、観測系（ログ・トレース）のエラー分類に本タクソノミを適用し、障害対応の一貫性を担保する。

## 本書の位置付け

エラーコードが各 API・各コンポーネントで異なる体系だと、観測・アラート・Runbook の全てで個別対応が発生し、運用負荷が爆発する。本書はエラーコード体系を単一ルールに統一し、分類・命名・HTTP マッピング・再試行可否を機械可読な形で提供する。障害対応時に「このエラーが何を意味し、どう対応すべきか」が即座に判明する状態を作る。

## エラータクソノミの設計原則

エラータクソノミは「発生カテゴリ」「発生モジュール」「エラー特性」の 3 軸を持つ。これにより、1 つのエラーを多面的に検索・集約できる。発生カテゴリは対応方針（再試行・フェイル・エスカレーション）を決定し、発生モジュールは責任所在を決定し、エラー特性は Severity を決定する。

## エラーコード形式

全エラーは以下の形式で識別する。

`E-<CATEGORY>-<MODULE>-<NUMBER>`

- **CATEGORY**: 発生カテゴリ 4 文字（VALID、AUTHZ、AUTHN、RSRC、DEPS、DATA、QUOT、CONF、INTL、EXTL）
- **MODULE**: 発生モジュール 6 文字以内（STATE、PUBSUB、WFLOW、DECSN、AUDIT、BINDG、SECRT、FFLAG、TELMY 等）
- **NUMBER**: モジュール内の通番 3 桁

例: `E-AUTHZ-STATE-001` は State API における認可失敗エラー。

## エラーカテゴリ

### VALID（バリデーションエラー）

入力値の形式・範囲・整合性に関するエラー。クライアント側の問題。HTTP 400 系に対応。

- 再試行: 同一リクエストでは **不可**（入力を変える必要）
- Severity: 通常 Info〜Warn
- 例: 必須フィールド欠落、フォーマット不正、範囲超過

### AUTHN（認証エラー）

認証失敗。クライアントの資格情報が無効または期限切れ。HTTP 401 に対応。

- 再試行: トークン再取得後に可
- Severity: Warn
- 例: トークン期限切れ、署名不正、IdP からの拒否

### AUTHZ（認可エラー）

認証済みだが権限不足。HTTP 403 に対応。

- 再試行: **不可**（権限変更が必要）
- Severity: Warn
- 例: RBAC 拒否、テナント越境アクセス、読み取り専用権限での書き込み

### RSRC（リソース不存在）

指定したリソースが存在しない。HTTP 404 に対応。

- 再試行: 可（作成後）
- Severity: Info
- 例: 存在しないキー、削除済みエンティティ、別テナントのリソース参照

### DEPS（依存関係エラー）

依存先の OSS やサービスとの通信・整合性に関するエラー。HTTP 502/503/504 に対応。

- 再試行: 基本的に **可**（指数バックオフ推奨）
- Severity: Error
- 例: PostgreSQL 接続切断、Kafka ブローカ不在、Istio サイドカー通信失敗

### DATA（データ整合性エラー）

データベース・ストレージの整合性違反。HTTP 409 または 500 に対応。

- 再試行: 判定後（競合時は再試行可、整合性違反は不可）
- Severity: Error
- 例: 重複キー、楽観ロック衝突、参照整合性違反

### QUOT（クォータ超過）

テナント別・API 別のレート・容量制限超過。HTTP 429 に対応。

- 再試行: **可**（バックオフ必須）
- Severity: Warn
- 例: レート制限、ストレージ容量制限、同時実行数制限

### CONF（設定エラー）

設定ミス・構成不整合に起因するエラー。HTTP 500 に対応。

- 再試行: **不可**（設定修正後）
- Severity: Error
- 例: 必須環境変数未設定、Secret 参照失敗、設定スキーマ違反

### INTL（内部エラー）

予期しない内部エラー。HTTP 500 に対応。

- 再試行: 冪等性が保証されている場合のみ可
- Severity: Error
- 例: パニック、アサーション失敗、予期せぬ NULL、型変換失敗

### EXTL（外部サービスエラー）

外部 IdP、外部監視系、レガシー .NET などの外部システム起因。HTTP 502/504 に対応。

- 再試行: 可（間隔長めで）
- Severity: Error
- 例: Keycloak Federation 応答遅延、Zabbix Exporter 接続失敗、外部 SAML IdP の証明書エラー

## モジュール識別子

tier1 公開 11 API とそれを支える内部コンポーネントにモジュール識別子を付与する。

- `STATE`: State API
- `PUBSUB`: PubSub API
- `SECRT`: Secrets API
- `BINDG`: Binding API
- `WFLOW`: Workflow API
- `LOG`: Log API
- `TELMY`: Telemetry API
- `DECSN`: Decision API
- `AUDIT`: Audit API
- `PII`: PII API
- `FFLAG`: Feature Flag API
- `INVOK`: Service Invoke API
- `IAM`: 認証認可基盤
- `GATEW`: Gateway（Envoy / Istio Ambient）
- `PLTFM`: プラットフォーム共通

## 再試行ポリシー

エラーに対する再試行可否は `retriable` フラグで機械可読に表現する。再試行可の場合は以下のポリシーを標準とする。

- **初期バックオフ**: 100ms
- **最大バックオフ**: 10s
- **バックオフ倍率**: 2 倍（指数バックオフ）
- **ジッタ**: ±20%（thundering herd 回避）
- **最大試行回数**: 5 回
- **全体タイムアウト**: 60s

この標準から逸脱する場合は API 仕様で明示する。

## エラー応答フォーマット

全 tier1 API は以下の JSON フォーマットでエラーを返却する。gRPC の場合は google.rpc.Status の details に同等構造を含める。

```json
{
  "error": {
    "code": "E-AUTHZ-STATE-001",
    "message": "tenant_id mismatch: requesting tenant cannot access target state store",
    "category": "AUTHZ",
    "module": "STATE",
    "retriable": false,
    "correlation_id": "01J9T8QW...",
    "trace_id": "4bf92f3577b34da6a3ce929d0e0e4736",
    "tenant_id": "tenant-acme",
    "timestamp": "2026-04-19T12:34:56Z",
    "documentation_url": "https://k1s0.jtc.internal/docs/errors/E-AUTHZ-STATE-001"
  }
}
```

クライアントには機械可読な情報（code、category、retriable）と人間可読な情報（message、documentation_url）の両方を提供する。内部詳細（スタックトレース、内部ID）は応答に含めず、ログと observability 基盤にのみ出力する。

## ログとトレースへの反映

エラー発生時は OpenTelemetry のスパン属性に `error.code`、`error.category`、`error.module`、`error.retriable` を付与する。構造化ログ（JSON）にも同等フィールドを含める。Grafana / Tempo / Loki でカテゴリ別のエラー率を可視化できる。

## アラートとの連携

エラーコードごとのアラートポリシーは Runbook-OPS-ERR-001 で集中管理する。以下の基本ルールを適用する。

- **CATEGORY=INTL または DATA で発生率急増**: P1 オンコール呼出
- **CATEGORY=DEPS で発生率急増**: P2、営業時間外は翌営業日対応
- **CATEGORY=QUOT で発生率急増**: テナントへの是正連絡
- **CATEGORY=VALID で発生率急増**: 該当 tier2/tier3 の実装ミス疑い、開発チームへ通知

## エラーコードのライフサイクル

新規エラーコードの追加は以下のフローで行う。

1. API 設計時または新機能実装時にエラーコード必要性を識別
2. エラーコード定義 PR（Protobuf エラー列挙、ドキュメント、Runbook エントリ）を作成
3. レビュアが命名規則・カテゴリ妥当性を確認
4. マージ後、リリースノートに記載
5. Backstage のエラーカタログに自動登録

エラーコードの削除は OR-EOL 系（非推奨化ライフサイクル）に従う。

## エラーコードカタログ運用

全エラーコードは Backstage のエラーカタログに集約し、以下で検索可能とする。

- コード文字列（E-AUTHZ-STATE-001）
- カテゴリ、モジュール別の絞り込み
- 発生頻度（直近 30 日のログから集計）
- 関連 Runbook リンク
- 再現手順（テスト用）

カタログは tier1 API Protobuf 定義から自動生成されるため、コード定義とカタログの乖離が発生しない。

## 責任分界

エラーカテゴリに応じて対応責任を明確化する。

- VALID / AUTHZ / RSRC / QUOT → 発生元テナント（tier2/tier3 開発者）が修正
- AUTHN → テナント + 認証基盤運用（IAM モジュール担当）
- DEPS / CONF / INTL → プラットフォーム SRE チームが調査・修正
- DATA → モジュール担当 + プラットフォーム SRE で協調調査
- EXTL → 外部ベンダー・IdP 担当と協調

この責任分界は RACI マトリクス（[../60_事業契約/10_ガバナンス.md](../60_事業契約/10_ガバナンス.md) BC-GOV-004）と整合する。
