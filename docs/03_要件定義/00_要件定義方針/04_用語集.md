# 用語集

本書は k1s0 要件定義書で使用する用語の正式定義を集約する。同一語が異なる意味で使われる事故（例: 「API」が tier1 公開 API を指すのか Dapr の building block を指すのか）を防ぐため、要件文内で揺れやすい語と、外部読者に馴染みのない固有語を優先的に収録する。

## 用語集の運用ルール

要件文の初出時には、本用語集の見出し語と完全一致する記法で書く。本用語集にない語で初出時に定義が必要なものは、本章に追加してから要件文を更新する。同義語は見出し語に集約し、別名からは `同義語参照先` で誘導する。用語の版管理は Git で行い、大きな意味変更時は ADR に経緯を残す。

略語は「正式名（略語）」の順で初出を書き、以降は略語のみ可とする。例: Service Level Objective（SLO）→ 以降は SLO。

## プラットフォーム層構造

### tier1

k1s0 基盤層。Dapr ファサード（Go）と自作 Rust 領域の 2 種類の実装で構成される。tier2/tier3 に対し 11 種の公開 API を提供する。内部実装は tier2/tier3 から不可視。

### tier2

tier1 の公開 API を使って業務ドメインロジックを実装する層。言語は C# / Go 等自由。採用側組織の業務システム（人事、経費、見積、在庫等）のビジネスロジックを担う。

### tier3

エンドユーザー向け UI とアプリ配信ポータルを実装する層。MAUI / React / C# / Go 等自由。

### infra

k8s / Istio / Kafka / Longhorn / MetalLB / CoreDNS / 観測基盤等の基盤 OSS 群。tier1 から見てインフラとなる層。

## 採用 OSS と標準

### Dapr

Distributed Application Runtime。CNCF プロジェクト。k1s0 では tier1 ファサード層で Go SDK を直接利用し、State / PubSub / Workflow / Secrets / Bindings / Service Invocation の各 building block を隠蔽して公開する。

### ZEN Engine

GoRules 社 OSS。Rust 実装のビジネスルール・決定表エンジン。JSON Decision Model（JDM）で決定表を記述し、マイクロ秒オーダで評価する。k1s0 では tier1 Rust 領域に in-process 組込し、権限ポリシー・申請ワークフロー・配信可視化ルール等を宣言的に扱う。

### Istio Ambient Mesh

サービスメッシュ。従来のサイドカー型 Istio に対し、ztunnel（DaemonSet）が Node 単位で L4 mTLS / HBONE を担当し、waypoint proxy が必要 namespace/service にのみ L7 機能を提供する方式。k1s0 では Dapr サイドカーとの二重サイドカー衝突回避を理由に採用。ADR-0001 参照。

### ztunnel

Istio Ambient の zero-trust tunnel コンポーネント。各 Node の DaemonSet で動作し、Pod 間通信に L4 mTLS を自動適用する。HBONE（HTTP-Based Overlay Network Environment）プロトコルで再カプセル化する。

### Keycloak

オープンソース IdP。k1s0 では OIDC / SAML / OAuth 2.0 の中央認証基盤として採用。既存 AD / LDAP の Federation を上流に置く構成。

### OpenBao

HashiCorp Vault の MPL 2.0 フォーク（Linux Foundation 配下）。動的シークレット発行、PKI、Transit 暗号化、Database Engine による PostgreSQL パスワード TTL 自動ローテーション（TTL 1 時間）を担う。

### Valkey

Redis の Linux Foundation フォーク（BSD ライセンス）。Redis 7.2 互換。k1s0 では State API のバックエンドおよびキャッシュとして採用。

### CloudNativePG

PostgreSQL の Kubernetes Operator。高可用構成、自動フェイルオーバー、継続アーカイブバックアップ（PITR）、PgBouncer Pooler CRD を提供する。pgvector 拡張でベクトル検索対応。

### Strimzi

Apache Kafka の Kubernetes Operator。KRaft モード（Zookeeper 不要）で Kafka ブローカーを運用する。

### Apache Kafka

分散ストリーミングプラットフォーム。追記専用・時系列・リプレイ可能・at-least-once 保証。k1s0 では PubSub API および監査イベント配信のバックエンド。

### MinIO

S3 互換オブジェクトストレージ（AGPL-3.0）。バックアップ、コンテナイメージ、ログアーカイブの格納先。外部境界遮断 + プロセス分離で無改変利用。

### Grafana / Loki / Tempo / Pyroscope

LGTM スタック。Grafana は可視化、Loki はログ、Tempo は分散トレース、Pyroscope は Continuous Profiling。いずれも AGPL-3.0（Grafana Labs 改ライセンス後）。外部境界遮断 + プロセス分離で無改変利用。

### OpenTelemetry（OTel）

分散トレース・メトリクス・ログの標準仕様と実装。k1s0 では各言語 SDK を tier1 がラップして `k1s0.Telemetry.*` API で提供する。Collector は DaemonSet で配置。

### W3C Trace Context

分散トレース伝搬の W3C 標準。HTTP / gRPC ヘッダで traceparent / tracestate を受け渡す。k1s0 各言語ライブラリで自動注入。

### flagd

CNCF OpenFeature の参照実装（Feature Flag エバリュエータ）。k1s0 では `k1s0.Feature.*` API のバックエンドとして採用。

### Temporal

長期実行ワークフロー・Saga 実装のための OSS。k1s0 では Dapr Workflow では表現しきれない長期実行 Saga を Temporal Go SDK 経由で補完する。

### Kyverno

Kubernetes Policy Engine。Pod セキュリティ、イメージ署名検証、ラベル強制、tenant-id 必須化を宣言的に管理する。

### cert-manager

Kubernetes 向け TLS 証明書自動発行・更新 OSS。

### Longhorn

Kubernetes 向け分散ブロックストレージ（CNCF）。レプリケーション、スナップショット、バックアップを備える。

### MetalLB

Kubernetes 向けオンプレ用ロードバランサ実装。L2 モード/ BGP モード（採用側のマルチクラスタ移行時）。

### kubeadm

Kubernetes クラスタ構築・運用ツール（公式）。k1s0 ではコントロールプレーン HA 構成（kube-vip 併用）を kubeadm で構築する。

### k3s

軽量 Kubernetes ディストリビューション。k1s0 ではサブプラン（小規模拠点・DR サイト）で採用候補。

### Argo CD / Argo Rollouts

GitOps ツール群。Argo CD は ApplicationSet による staging→prod 順序同期、Argo Rollouts は Canary / Blue-Green デプロイを担う。

### Renovate

依存関係自動更新 OSS（AGPL-3.0）。k1s0 内で自社の OSS バージョン追従を自動化する。

### OpenTofu

Terraform の MPL 2.0 フォーク（Linux Foundation 配下）。IaC 記述で採用。

## k1s0 固有用語

### k1s0 公開 API（11 種）

tier1 が tier2/tier3 向けに公開するクライアントライブラリと gRPC エンドポイント。Service Invoke、State、PubSub、Secrets、Binding、Workflow、Log、Telemetry、Decision、Audit / Pii、Feature の 11 種。20_機能要件/10_tier1_API要件/ 参照。

### JDM（JSON Decision Model）

ZEN Engine の決定表記述フォーマット。GoRules 独自規格。決定表を JSON で宣言的に表現する。

### 決定表

条件と結論の対応関係を表形式で記述するビジネスルール表現。k1s0 では ZEN Engine で評価する。

### バス係数（Bus Factor）

チームから何人が抜けると開発が停止するかを示す指標。値が大きいほど属人化が少ない。k1s0 では 採用初期 で 2、採用後の運用拡大時 で 3 以上を目標とする。

### Runbook

インシデント対応手順書。k1s0 では リリース時点 で 15 種類のシナリオを整備し、運用 SRE が訓練込みで実行可能な状態とする。

### Dead Man's Switch

一定時間メトリクスが届かないと自動的にアラートを発する監視手法。k1s0 では Watchdog alert として Prometheus 依存しない外部 CronJob ヘルスチェックで実装する。監視基盤自体の障害検知に使う。

### SBOM（Software Bill of Materials）

ソフトウェア構成表。依存 OSS の一覧と版数を機械可読で記述する。k1s0 では 採用初期 で生成率 100% を目標とする。

### 採用側組織

k1s0 を取り込んで自社プラットフォームとして運用する組織。年次予算申請、定量根拠の要求、監査部門の関与、情シス部門での内製運用を特徴とする日本企業を中心想定とする。k1s0 自体は OSS としてライセンス条件を満たす任意の組織が利用できる。

## IPA 標準用語

### 共通フレーム 2013（SLCP-JCF2013）

IPA が発行するソフトウェアライフサイクルプロセスの業界標準。要件定義プロセス、システム開発プロセス、運用保守プロセス等を規定する。ISO/IEC 12207 日本版に相当。

### 非機能要求グレード 2018

IPA 発行の非機能要件記述ガイド。可用性・性能拡張性・運用保守性・移行性・セキュリティ・システム環境エコロジーの 6 大項目・約 230 小項目で構造化。社会的影響度に応じたモデルシステム①〜③のグレード判定を提供する。

### モデルシステム①〜③

非機能要求グレードが提示する典型システム。①は社会的影響度が極めて高いシステム（24 × 365、SIE 重大事故防止）、②は影響度が限定的なシステム（業務時間帯、重大事故対策）、③は影響度が特定領域に限定（業務時間帯、通常事故対策）。k1s0 の判定は 90_付録/02_非機能要求グレード判定.md を参照。

### RTO（Recovery Time Objective）

目標復旧時間。障害発生から復旧完了までの許容時間。

### RPO（Recovery Point Objective）

目標復旧時点。障害発生時に失われてもよいデータ量（時間換算）。

### MTTR（Mean Time To Repair）

平均復旧時間。

### SLA / SLO / SLI

Service Level Agreement / Objective / Indicator。SLA は顧客との合意、SLO は内部目標、SLI は計測指標。

## セキュリティ用語

### SPIFFE / SPIRE

Secure Production Identity Framework For Everyone。ワークロード ID の標準仕様（SPIFFE）と実装（SPIRE）。k1s0 では Istio Ambient が SPIFFE ID を発行する。

### mTLS（Mutual TLS）

双方向 TLS 認証。k1s0 では Istio Ambient ztunnel が自動適用する。

### HBONE（HTTP-Based Overlay Network Environment）

Istio Ambient で使われるトンネリングプロトコル。HTTP/2 CONNECT をベースにする。

### AuthorizationPolicy

Istio の L7 認可ポリシー。SPIFFE ID ベースでアクセス制御する。

### ハッシュチェーン

各イベントに前イベントのハッシュを含めることで、途中改ざんを検知可能にするデータ構造。k1s0 では Audit API で採用。

### PII（Personally Identifiable Information）

個人を識別可能な情報。個人情報保護法の対象。k1s0 では tier1 の Pii API でマスキング / 仮名化を実施する。

## 業務プロセス用語

### 適用段階（リリース時点 / 採用初期 / 採用後の運用拡大時 / 採用側のマルチクラスタ移行時 / 採用側の全社展開期 / 採用側の全社ロールアウト）

k1s0 が想定する利用ステージ。OSS リリース時点では全機能を一気通貫で本格実装した状態で公開する。採用初期は採用側組織が単一クラスタで運用を立ち上げる時期、採用後の運用拡大時は同一テナントでの利用範囲拡大、採用側のマルチクラスタ移行時は staging / prod 分離・マルチクラスタ展開、採用側の全社展開期はグループ会社横断展開、採用側の全社ロールアウトは長期運用段階に対応する。詳細は `01_企画/` を参照。

### 稟議

採用側組織が運用する業務システム（経費精算・購買承認・契約決裁等）における意思決定文書。起案者が回付し、関係者印（電子承認）を集めて決裁する。本書では k1s0 が代替対象とする業務システムの一例として参照する。

### MUST / SHOULD / COULD / WON'T

要件優先度（MoSCoW 法）。MUST は必達、SHOULD は強く推奨、COULD は余裕があれば、WON'T は今回やらない。本書では MUST / SHOULD / COULD / WON'T の 4 段階を使う。

### Severity 1 / 2 / 3

インシデント重大度。Severity 1 はサービス全停止相当（応答 30 分以内）、Severity 2 は機能縮退（応答 2 時間以内）、Severity 3 は単発不具合（応答 1 営業日）。

## 未収録語の扱い

本用語集に未収録の語が要件文で必要になった場合、執筆者は PR で本用語集への追加を先行させる。未収録のまま要件文に登場させることは禁ずる。略語のみ使って用語集を更新しない状態は、3 か月経過後に整合性レビューで指摘する。
