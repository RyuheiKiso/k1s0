# 機能一覧

本書は、機能要件カテゴリ（20_機能要件/）に存在するすべての要件 ID のインデックスであり、同時に各 API の スコープ判定と業務根拠の対応を明示する「機能範囲の単一真実源」である。散文主体の要件詳細は各 API ファイル（[10_tier1_API要件/](10_tier1_API要件/)）・[20_外部連携要件.md](20_外部連携要件.md)・[30_情報要件.md](30_情報要件.md) で記述し、本書では API グループ単位で「なぜこの 段階に配置するのか」「どの業務根拠 (BR-*) から導出されたのか」を散文で説明したうえで、索引としての ID 一覧を各グループ章末に置く。

## 本書の読み方

採用検討説明資料・監査エビデンス・構想設計着手時の機能範囲確認で本書を参照する。要件 ID から詳細ファイルへ辿るのは索引としての用途、段階別の達成度把握は「段階到達度」節を見る用途、個別要件の 段階判定理由は API グループ冒頭の散文で確認する用途である。本書と詳細ファイルが乖離した場合は詳細ファイル側を優先するが、段階・優先度・業務根拠 ID の改訂は本書と詳細ファイルの両方で同時に更新する運用とする。

本書では以下の表記規約を採用する。

- **段階判定理由**: 優先度セルに括弧書きで判定根拠を添える。例: `MUST（採用検討提示 SLO、リリース時点 未達で採用検討通過不可）`
- **業務根拠**: 各要件の業務要件（BR-\*）との対応を「業務根拠」列に列挙する
- **複数段階またがり**: 段階遷移で優先度が変わる要件は、リリース時点 代替案 / 採用側のマルチクラスタ移行時 本装案のように分離記述する

## tier1 公開 API 機能要件

### Service Invoke API（FR-T1-INVOKE-001〜005）

**段階判定の論理**: Service Invoke は tier2 間の同期呼び出しを tier1 ファサード越しに成立させる最短経路の API であり、これが無ければ リリース時点 で「tier2 アプリが 1 つでも動く」という 採用初期 の最小成立条件が崩れる。そのため同期 gRPC（INVOKE-001）・タイムアウトリトライ（INVOKE-003）・認証トークン伝搬（INVOKE-005）の 3 件は リリース時点 MUST とし、他の 2 件はレガシー互換（INVOKE-002 HTTP/1.1）・運用安定性（INVOKE-004 サーキットブレーカ）として リリース時点 に送る。HTTP/1.1 互換プロキシを リリース時点 に前倒ししない判断は、.NET Framework 4.x 資産の移行を 運用蓄積後で扱う企画書ロードマップと整合させた結果である。リリース時点 で HTTP/1.1 を入れると リリース時点 スコープが膨張し、採用検討通過に要する 2 週間の時間的制約（C-TIME-001）を超過する。

**業務根拠**: tier2 開発者が Dapr 実装詳細を意識せず API を呼び出せること（BR-PLATUSE-001）、認証・トレース・テナント分離が透過的に行われサービス開発に集中できること（BR-PLATUSE-002）、.NET Framework 資産との共存で移行コストを抑えること（BR-PLATUSE-003）。

|---|---|---|---|---|---|
| FR-T1-INVOKE-001 | 同期サービス呼び出し（gRPC） | 採用初期 | MUST（リリース時点 採用初期 の最小成立条件、これが無いと tier2 間呼出が 1 本も通らない） | BR-PLATUSE-001, 002 | [01_Service_Invoke_API.md](10_tier1_API要件/01_Service_Invoke_API.md) |
| FR-T1-INVOKE-002 | HTTP/1.1 互換プロキシ | 採用初期 | SHOULD（.NET Framework 4.x 資産の移行を リリース時点 に合わせる。リリース時点 前倒しは採用検討通過時間制約 C-TIME-001 を超過） | BR-PLATUSE-003 | 同上 |
| FR-T1-INVOKE-003 | タイムアウト・リトライ制御 | 採用初期 | MUST（同期呼出の p99 SLO 500ms を守る前提、SDK に既定値が無いと tier2 がバラバラに実装） | BR-PLATUSE-002 | 同上 |

### State API（FR-T1-STATE-001〜005）

**段階判定の論理**: 基本 Get/Set/Delete（STATE-001）と TTL 制御（STATE-002）は tier2 のセッション管理とキャッシュが動く最小条件であり、リリース時点 MUST。Bulk 操作（STATE-003）は N 回往復による p99 悪化の回避策、楽観ロック（STATE-004）は並行更新事故の防止策で、いずれも リリース時点 には無くても単機能アプリなら動くため リリース時点 SHOULD に送る。トランザクション（STATE-005）は代替手段（アプリ側リトライ）でも成立し、tier2 のユースケース顕在化を待ってから リリース時点 で判断するため COULD。

**業務根拠**: tier2 が接続文字列・認証・プール設定を個別管理せず済むこと（BR-PLATUSE-001）、tenant_id の自動プレフィックスで越境事故を構造的に防ぐこと（BR-PLATGOV-001）。

|---|---|---|---|---|---|
| FR-T1-STATE-001 | 基本 Get/Set/Delete | 採用初期 | MUST（tier2 のセッション・軽量キャッシュが動く最小条件） | BR-PLATUSE-001, BR-PLATGOV-001 | [02_State_API.md](10_tier1_API要件/02_State_API.md) |
| FR-T1-STATE-002 | TTL 制御 | 採用初期 | MUST（TTL なしでは Valkey メモリ圧迫、tier2 毎バラつき発生） | BR-PLATUSE-001 | 同上 |
| FR-T1-STATE-003 | Bulk 操作 | 採用初期 | SHOULD（N 回往復による p99 悪化の回避策、リリース時点 では tier2 ループ実装で代替可） | BR-PLATUSE-002 | 同上 |
| FR-T1-STATE-004 | 楽観的ロック（ETag） | 採用初期 | SHOULD（並行更新事故の防止策、リリース時点 では単一アクター前提で許容） | BR-PLATOPS-002 | 同上 |
| FR-T1-STATE-005 | トランザクション | 採用初期 | COULD（tier2 ユースケース顕在化待ち、アプリ側リトライで代替可能） | BR-PLATGOV-002 | 同上 |

### PubSub API（FR-T1-PUBSUB-001〜005）

**段階判定の論理**: 同期呼出（Service Invoke）だけでは業務プロセスの疎結合化ができず、イベント駆動が リリース時点 の段階で必要となる。Publish/Subscribe 基本機能（PUBSUB-001）・at-least-once 配信保証（PUBSUB-002）・Consumer Group 管理（PUBSUB-003）・トピック命名規約によるテナント分離（PUBSUB-005）の 4 件は リリース時点 MUST で揃って初めてイベント駆動アプリが成立する。DLQ（PUBSUB-004）は運用の品質向上で、初期は手動対応で代替できるため SHOULD。PubSub を リリース時点 に含めない判断は、Kafka の単一ブローカー構成での運用リスク（NFR-A-FT-002 の リリース時点 代替案）と、tier2 の初期ユースケースが同期呼出中心である企画想定に基づく。

**業務根拠**: 業務プロセスの疎結合化による tier2 アプリ同士の独立リリース（BR-PLATUSE-004）、テナント分離強制による越境イベント配信事故の防止（BR-PLATGOV-001）。

| ID | タイトル | 適用段階 | 優先度（判定理由） | 業務根拠 | 参照 |
|---|---|---|---|---|---|
| FR-T1-PUBSUB-001 | Publish / Subscribe 基本機能 | 採用初期 | MUST（リリース時点 のイベント駆動成立の最小条件） | BR-PLATUSE-004 | [03_PubSub_API.md](10_tier1_API要件/03_PubSub_API.md) |
| FR-T1-PUBSUB-002 | at-least-once 配信保証 | 採用初期 | MUST（業務イベント欠損は請求漏れ等の金銭的影響を引き起こすため リリース時点 で必達） | BR-PLATGOV-002 | 同上 |
| FR-T1-PUBSUB-003 | Consumer Group 管理 | 採用初期 | MUST（スケールアウト時の Consumer 分散が必須、単一 Consumer では水平拡張不可） | BR-PLATOPS-001 | 同上 |
| FR-T1-PUBSUB-004 | Dead Letter Queue | 採用初期 | SHOULD（初期は手動対応で代替可、運用品質向上の位置付け） | BR-PLATOPS-002 | 同上 |
| FR-T1-PUBSUB-005 | トピック命名規約強制（tenant 分離） | 採用初期 | MUST（テナント越境イベント配信は漏えい重大インシデントのため構造的防止が必須） | BR-PLATGOV-001 | 同上 |

### Secrets API（FR-T1-SECRETS-001〜004）

**段階判定の論理**: 静的 Secret 取得（SECRETS-001）は tier2 の外部サービス接続（SaaS API キー等）が成立する最小条件として リリース時点 MUST。動的 Secret 発行（SECRETS-002）は DB 接続の短寿命化による漏えいリスク低減で、初期は静的 DB 接続文字列で代替できるため SHOULD。Transit 暗号化（SECRETS-003）は tier2 のフィールド暗号化ユースケース依存で リリース時点 COULD。自動ローテーション（SECRETS-004）は リリース時点で は手動 Runbook で代替 / 自動化判定。リリース時点 に Secrets を含めない判断は、OpenBao の立ち上げと KMS 連携の設計深度を リリース時点 まで延ばした企画判断と整合する。

**業務根拠**: Secret 直書きコードの撲滅（BR-PLATGOV-003）、漏えい時の影響範囲限定（BR-PLATGOV-004）。

| ID | タイトル | 適用段階 | 優先度（判定理由） | 業務根拠 | 参照 |
|---|---|---|---|---|---|
| FR-T1-SECRETS-001 | 静的 Secret 取得 | 採用初期 | MUST（リリース時点 の tier2 外部連携成立の最小条件） | BR-PLATGOV-003 | [04_Secrets_API.md](10_tier1_API要件/04_Secrets_API.md) |
| FR-T1-SECRETS-002 | 動的 Secret 発行（PostgreSQL 等） | 採用初期 | SHOULD（DB 接続の短寿命化、リリース時点 では静的接続文字列で代替可） | BR-PLATGOV-004 | 同上 |
| FR-T1-SECRETS-003 | Transit 暗号化（API 経由） | 採用初期 | COULD（tier2 のフィールド暗号化ユースケース顕在化待ち） | BR-PLATGOV-003 | 同上 |
| FR-T1-SECRETS-004 | Secret ローテーション自動化 | 採用後の運用拡大時 | SHOULD（リリース時点 は手動 Runbook で代替、運用蓄積後で自動化判定） | BR-PLATGOV-004 | 同上 |

### Binding API（FR-T1-BINDING-001〜004）

**段階判定の論理**: MinIO Output（BINDING-001）は監査ログの外部保管・業務添付ファイルの保管で リリース時点 必達。SMTP（BINDING-002）・HTTP（BINDING-003）は通知・外部 API 連携で リリース時点 中頃に必要だが、初期はアドホックな tier2 実装で代替可能なため SHOULD。Cron Input（BINDING-004）は k8s CronJob で代替できるため リリース時点 COULD。Binding を リリース時点 に含めない判断は、外部連携の allowlist 管理・Audit 記録の統一を リリース時点 で一括整備する方針と整合する。

**業務根拠**: tier2 の外部通知・ファイル保管の統一化（BR-PLATUSE-005）、外部連携の監査証跡統一（BR-PLATGOV-004）。

| ID | タイトル | 適用段階 | 優先度（判定理由） | 業務根拠 | 参照 |
|---|---|---|---|---|---|
| FR-T1-BINDING-001 | MinIO Output Binding | 採用初期 | MUST（監査ログ外部保管の基盤、リリース時点 の Audit 長期保管に必要） | BR-PLATGOV-004 | [05_Binding_API.md](10_tier1_API要件/05_Binding_API.md) |
| FR-T1-BINDING-002 | SMTP Output Binding | 採用初期 | SHOULD（通知は初期アドホック tier2 実装で代替可、統一化は品質向上） | BR-PLATUSE-005 | 同上 |
| FR-T1-BINDING-003 | HTTP Output Binding | 採用初期 | SHOULD（外部 API 連携の allowlist 統一管理、リリース時点 中頃に必要） | BR-PLATUSE-005 | 同上 |
| FR-T1-BINDING-004 | Input Binding（定期実行） | 採用後の運用拡大時 | COULD（k8s CronJob で代替可、統一 UI は リリース時点 品質向上） | BR-PLATOPS-003 | 同上 |

### Workflow API（FR-T1-WORKFLOW-001〜005）

**段階判定の論理**: 短期ワークフロー（WORKFLOW-001）・タイマー（WORKFLOW-004）は リリース時点 の業務ユースケース（採用検討、督促通知）の最小条件で MUST。長期 Saga（WORKFLOW-002）・補償ロジック（WORKFLOW-003）は複数システム跨りの整合性確保で SHOULD、初期は単一システム完結で代替可能。外部イベント待ち受け（WORKFLOW-005）は Signal 機構の実装コストが高いため リリース時点 COULD。ADR-RULE-002 により、短期ワークフロー / タイマーは Dapr Workflow、長期実行 Saga は Temporal を採用済みとし、リリース時点 の実装範囲は WORKFLOW-001〜004 で固定する。

**業務根拠**: 業務プロセス（申請・督促）の長期実行と状態保持（BR-PLATUSE-006）、複数システム跨りの整合性確保（BR-PLATGOV-002）。

|---|---|---|---|---|---|
| FR-T1-WORKFLOW-001 | 短期ワークフロー（Dapr Workflow） | 採用初期 | MUST（リリース時点 の採用検討・督促通知ユースケースの最小条件） | BR-PLATUSE-006 | [06_Workflow_API.md](10_tier1_API要件/06_Workflow_API.md) |
| FR-T1-WORKFLOW-002 | 長期実行 Saga（Temporal） | 採用初期 | SHOULD（長期実行ユースケース向け。短期ユースケースは WORKFLOW-001/004 で代替可） | BR-PLATUSE-006 | 同上 |
| FR-T1-WORKFLOW-003 | 補償ロジック（Compensating Transactions） | 採用初期 | SHOULD（Saga 採用時の必須条件、SHOULD 据え置きは単一完結 UX 優先のため） | BR-PLATGOV-002 | 同上 |
| FR-T1-WORKFLOW-004 | タイマー・遅延実行 | 採用初期 | MUST（督促メールなどの遅延処理で リリース時点 必達、tier2 独自実装は Pod 再起動で失われる） | BR-PLATUSE-006 | 同上 |

### Log API（FR-T1-LOG-001〜004）

**段階判定の論理**: 構造化ログ出力（LOG-001）・W3C Trace Context 注入（LOG-002）・tenant_id 必須（LOG-003）は リリース時点 MUST。ここで手を抜くと、リリース時点 から記録されたログが後続段階 での障害調査・監査で使えず、過去ログの再構築が不可能になる（ログは時間経過と共に失われる性質）。動的ログレベル変更（LOG-004）は debug レベル有効化に Pod 再起動を伴う代替手段があるため リリース時点 SHOULD。

**業務根拠**: 障害調査の効率化（BR-PLATOPS-004）、監査ログの一貫性確保（BR-PLATGOV-004）、テナント別ログ可視化（BR-PLATGOV-001）。

|---|---|---|---|---|---|
| FR-T1-LOG-001 | 構造化ログ出力 | 採用初期 | MUST（リリース時点 から記録されないと過去ログ再構築不可、時間経過で失われる性質） | BR-PLATOPS-004 | [07_Log_API.md](10_tier1_API要件/07_Log_API.md) |
| FR-T1-LOG-002 | W3C Trace Context 自動注入 | 採用初期 | MUST（リリース時点 からトレース相関が無いと後続段階 での障害調査が不可能） | BR-PLATOPS-004 | 同上 |
| FR-T1-LOG-003 | tenant_id 必須フィールド強制 | 採用初期 | MUST（リリース時点 から欠落するとテナント別集計が後付けで不可、ログ再処理コスト大） | BR-PLATGOV-001 | 同上 |
| FR-T1-LOG-004 | ログレベル動的変更 | 採用初期 | SHOULD（Pod 再起動で代替可、運用品質向上の位置付け） | BR-PLATOPS-004 | 同上 |

### Telemetry API（FR-T1-TELEMETRY-001〜004）

**段階判定の論理**: Metrics（TELEMETRY-001）・Traces（TELEMETRY-002）・Collector 配信（TELEMETRY-003）は リリース時点 MUST。リリース時点 から計装されていないと / パイロット時に性能 SLO 達成判定の基準データが存在しない。Profiling（TELEMETRY-004）は Pyroscope のエージェント組み込みに追加工数を要するため リリース時点 SHOULD。

**業務根拠**: SLO 達成判定基盤（BR-PLATOPS-005）、パフォーマンス問題の根本原因特定（BR-PLATOPS-004）。

| FR-T1-TELEMETRY-003 | OpenTelemetry Collector 経由配信 | 採用初期 | MUST（計装データの収集基盤、SDK 直送は単一点障害を tier2 に波及させる） | BR-PLATOPS-001 | 同上 |

### Decision API（FR-T1-DECISION-001〜004）

**段階判定の論理**: JDM 決定表評価（DECISION-001）は リリース時点 MUST。ZEN Engine 統合は本プロジェクトの差別化要素（ADR-0002）であり、リリース時点 デモ段階でルール駆動業務の PoC が無いと採用検討説明で弱い。バージョン管理（DECISION-002）・Audit 連携（DECISION-003）は リリース時点 で業務投入時に必須 / は単一バージョン・記録なしで許容。ホットリロード（DECISION-004）は Pod 再起動で代替できるため リリース時点 COULD。

**業務根拠**: ビジネスルール変更の迅速化（BR-PLATUSE-007）、ルール評価の監査証跡（BR-PLATGOV-004）。

### Audit / Pii API（FR-T1-AUDIT-001〜003、FR-T1-PII-001〜002）

**段階判定の論理**: ハッシュチェーン監査ログ（AUDIT-001）は リリース時点 MUST。リリース時点 から改ざん不可能な監査が組み込まれていないと、後付けでの監査対応で「過去の操作の改ざんが無いこと」を証明できない。改ざん検知クエリ（AUDIT-002）は リリース時点 の監査部門向け UI で MUST。長期保存 7 年（AUDIT-003）は個人情報保護法・J-SOX の保管義務で リリース時点 MUST / は 3 年保持で運用開始し リリース時点 で 7 年に延長する段階的拡張とする。PII マスキング（PII-001）は リリース時点 の業務投入時に MUST / は tier2 が手動マスキング実装で許容。PII 仮名化（PII-002）は分析用途の限定シナリオで リリース時点 SHOULD。

**業務根拠**: 個人情報保護法・J-SOX の保管義務準拠（BR-PLATGOV-004）、漏えい時のレピュテーション保護（BR-PLATGOV-003）。

### Feature API（FR-T1-FEATURE-001〜004）

**段階判定の論理**: Feature Flag 評価（FEATURE-001）は リリース時点 MUST。リリース時点 で業務投入する段階で、新機能の段階ロールアウト・カナリアデプロイの制御点が無いと、障害時の切戻が Pod 再起動依存となる。段階ロールアウト（FEATURE-002）は % ベースのロールアウト制御で SHOULD（リリース時点 ロールアウト戦略）。Circuit Breaker ルール連動（FEATURE-003）は運用高度化で リリース時点 SHOULD、A/B テスト（FEATURE-004）は分析用途で 採用後の運用拡大時 COULD。Feature を リリース時点 に含めない判断は / は単一構成での動作確認が主で段階ロールアウトが不要であることに基づく。

**業務根拠**: 新機能の段階ロールアウト（BR-PLATOPS-006）、障害時の迅速な切戻（BR-PLATOPS-002）。

## 段階別達成判定とスコープ遷移

各段階 ゲートで何が揃っていれば 段階通過とみなすかを、API グループ単位で明示する。本節は採用検討レビューや基本設計のマイルストーン評価で直接引用する運用とする。

### リリース時点（採用初期、採用検討通過直後）

リリース時点 は「tier2 アプリが 1 つ動く」最小条件の成立を目標とする。以下の 17 件の MUST 要件が揃った時点で リリース時点 通過とみなす。

- Service Invoke: INVOKE-001, 003, 005（同期呼出・タイムアウト・認証伝搬）
- State: STATE-001, 002（基本操作・TTL）
- Log: LOG-001, 002, 003（構造化・Trace Context・tenant_id）
- Telemetry: TELEMETRY-001, 002, 003（Metrics・Traces・Collector）
- Decision: DECISION-001（JDM 評価の PoC）
- Audit: AUDIT-001（ハッシュチェーン、後付け不可）
- 外部連携: IDP-001（AD/LDAP、認証動作の前提）
- 情報要件: LOGMODEL-001, AUDITMODEL-001, TENANTMODEL-001（データモデル基盤）

### リリース時点（採用初期、パイロット投入）

リリース時点 は「パイロット 1 チームが業務投入できる」条件の成立を目標とする。追加で以下の 14 件の MUST が揃う。

- Service Invoke: INVOKE-002（HTTP/1.1 プロキシ）※ SHOULD 扱いだが実質必須
- PubSub: PUBSUB-001, 002, 003, 005（基本・at-least-once・Consumer Group・テナント分離）
- Secrets: SECRETS-001（静的 Secret）
- Binding: BINDING-001（MinIO Output）
- Workflow: WORKFLOW-001, 004（短期・タイマー）
- Audit: AUDIT-002（改ざん検知）
- PII: PII-001（マスキング）
- Decision: DECISION-003（Audit 連携）
- Feature: FEATURE-001（Flag 評価）
- 外部連携: DOTNET-001（サイドカー）、MON-001（監視連携）

### リリース時点（採用初期、運用定着）

リリース時点 は「SRE オンコール体制で継続運用できる」条件の成立を目標とする。追加で AUDIT-003（7 年保存）と NFR 側の HA 化・カオス試験が揃う。機能要件としての新規 MUST は AUDIT-003 の 1 件のみ。

### 採用後の運用拡大時（拡大）

A/B テスト（FEATURE-004）、高度 PII 仮名化（PII-002）など、業務投入後の分析・品質高度化を扱う。

## 要件総数と優先度分布

本カテゴリの要件総数は tier1 公開 API 49 件・外部連携 6 件・情報要件 4 件の合計 59 件。優先度分布の再計算は、本書のテーブル内「優先度（判定理由）」列から `MUST` / `SHOULD` / `COULD` を機械抽出して集計する（単純な MUST/SHOULD/COULD カウントではなく、判定理由が記録されていることを検証する CI スクリプトを将来的に整備）。

| 適用段階 | MUST | SHOULD | COULD | 計 |
|---|---|---|---|---|
| 採用初期 | 17 | 0 | 0 | 17 |
| 採用初期 | 14 | 18 | 2 | 34 |
| 採用後の運用拡大時 | 1 | 3 | 3 | 7 |
| 2+ | 0 | 0 | 1 | 1 |
| **合計** | **32** | **21** | **6** | **59** |

リリース時点 MUST の 17 件は API 側 13 件（INVOKE-001/003/005、STATE-001/002、LOG-001〜003、TELEMETRY-001〜003、DECISION-001、AUDIT-001）に外部連携 1 件（IDP-001）と情報要件 3 件（LOGMODEL-001、AUDITMODEL-001、TENANTMODEL-001）を加えたもの。EXT / INFO を集計から落とさないこと。採用段階別の累計は、リリース時点 17 件 → 採用初期（パイロット）51 件（+34）→ 採用初期（運用体制確立）58 件（+7）→ 採用後の運用拡大時 59 件（+1）となる。採用段階ゲート通過判定は累計に対する達成率 95% 以上を基準とする。

## 不足要件の検出

本書のインデックスに登場しないが、業務要件（[10_業務要件/](../10_業務要件/)）や構想設計で言及されている機能は、要件の記述漏れである可能性がある。四半期ごとの整合レビューで、業務要件 → 機能要件のリンクが切れている BR-\* を検出し、本書への追加を検討する。業務根拠列に未確定や仮置きが発生した要件は、次回 Product Council レビューで BR 側の追記または FR 側の優先度見直しを判定する。現行版では未確定項目は存在しない。
