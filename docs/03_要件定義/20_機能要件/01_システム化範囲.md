# システム化範囲

本書は、機能要件として含める範囲と意図的に除外する範囲を明示する。10_業務要件/02_業務スコープ.md が「業務」として扱うかを決めたのに対し、本書は「業務スコープ内の業務を、どこまで機能として実装するか」を決める。業務スコープと機能スコープを分けるのは、「運用対象だが自動化しない」「業務対象だが機能化せず手順書で回す」といった判断を明示するためである。

## スコープ設定の原則

k1s0 の機能スコープは、以下 4 原則で判定する。原則を個別機能に適用した結果は本書の「含める」「除外」「将来検討」の 3 分類で整理する。

**原則 1: tier1 公開 API の責務に対応する機能か**。tier1 が State / PubSub / Workflow / Decision / Audit 等の公開 API を提供するために必要な機能は原則含める。tier2/tier3 固有の業務機能は含めない。

**原則 2: 複数利用者で共有されるか**。tier2 複数アプリ、tier3 複数配信チャネルで共有される機能は含める。1 利用者のためだけの機能は除外する。

**原則 3: 既存資産との共存を崩さないか**。.NET Framework 4.x との共存を崩す機能（例: .NET 4.x では呼べないプロトコルの強制）は除外する。

**原則 4: 採用検討で約束した範囲内か**。企画書で明示した約束を超える機能は将来検討とし、採用検討改訂を要する。

## 機能として含める範囲

### tier1 公開 API 群（リリース時点 スコープ）

以下 5 API を リリース時点 の最小機能として含める。

- **Service Invoke API**: tier2/tier3 サービス間の同期呼び出し。gRPC をネイティブ対応、HTTP/1.1 互換プロキシを補助提供。
- **State API**: Valkey バックエンドでの Get / Set / Delete / Bulk 操作、TTL 制御、トランザクション。
- **Log API**: 各言語 SDK で構造化ログを出力。W3C Trace Context の自動注入、`tenant_id` 必須フィールドの強制。
- **Telemetry API**: OpenTelemetry SDK のラッパとして metrics / traces / logs の計装 API を提供。
- **Decision API**: ZEN Engine の JDM 決定表を in-process 評価。業務ルール評価の基本機能。

### tier1 公開 API 群（リリース時点 スコープ追加）

リリース時点 で以下 6 API を追加する。

- **PubSub API**: Kafka バックエンドでの Publish / Subscribe、consumer group 管理、at-least-once 配信保証。
- **Workflow API**: Dapr Workflow 経由の短期ワークフロー、Temporal 経由の長期実行 Saga。
- **Secrets API**: OpenBao 経由の秘密取得、Database Engine 動的シークレット（PostgreSQL パスワード TTL 1 時間）、Transit 暗号化。
- **Binding API**: MinIO / SMTP / HTTP 外部バインディング。
- **Audit / Pii API**: Rust 自作領域。ハッシュチェーン改ざん防止ログ、PII マスキング / 仮名化。
- **Feature API**: flagd バックエンドでの Feature Flag 評価、段階ロールアウト、circuit breaker ルール。

### プラットフォーム運用機能

tier1 基盤を運用するための機能を含める。

- Backstage カタログによる サービス一覧 / API 一覧 / ランブック導線
- Argo CD GitOps での staging → prod 順序同期
- Argo Rollouts での Canary / Blue-Green デプロイ
- Prometheus / Grafana Tempo / Loki / Pyroscope の統合監視
- Dead Man's Switch（Watchdog alert）による監視基盤の自己監視
- cert-manager による TLS 証明書自動発行・更新
- Renovate による OSS バージョン追従 PR 自動生成
- SBOM 生成（CycloneDX / SPDX）
- Cosign 署名 + Kyverno 検証（採用後の運用拡大時）
- Keycloak による SSO（既存 AD / LDAP の Federation）
- Kyverno / NetworkPolicy によるテナント隔離強制
- Litmus Chaos による定期的な隔離境界検証（採用後の運用拡大時、月次）

### 開発者体験機能

tier2/tier3 開発者が k1s0 API を使うための補助機能を含める。

- 雛形 CLI（`k1s0 init` 相当）による Go / Rust / C# / Python プロジェクト scaffold
- Protobuf スキーマの集中管理と CI 自動生成
- ローカル開発環境の立ち上げスクリプト（docker-compose または kind）
- Backstage プラグインによるサービス登録 / ネームスペース発行 / ロール設定

## 機能として除外する範囲

### tier2/tier3 の業務ロジック

経費精算、承認フロー、在庫管理等の具体業務ロジックは tier2 の責務。k1s0 は ZEN Engine で決定表の評価機能を提供するが、決定表の内容（どの条件でどの承認者）は tier2 が業務要件として定義する。

### tier3 の UI コンポーネントライブラリ

社内 UI フレームワーク（デザインシステム）は tier3 の責務。k1s0 は認証統合（Keycloak OIDC）、Feature Flag 評価、監査ログ記録の基盤 API を提供するが、ボタン / フォーム / 画面遷移は tier3 が実装する。

### マルチクラウド・マルチリージョン DR

リリース時点 のオンプレ・閉域完結前提では除外。採用側の全社ロールアウト 以降に参考検討。

### 業務アプリのデータベーススキーマ設計

tier2 の業務アプリは CloudNativePG の PostgreSQL を利用できるが、スキーマ設計・マイグレーション・インデックスチューニングは tier2 の責務。k1s0 は PgBouncer による接続プーリングと Row-Level Security による tenant_id フィルタの強制のみ提供する。

### 外部 SaaS との直接連携

k1s0 基盤は外部 SaaS（PagerDuty、Slack、GitHub 等）への直接依存を持たない。tier3 が業務アプリとして必要な場合に個別実装する。社内通知は社内 Alertmanager → メールサーバへの転送で統一する。

### 商用 PaaS 互換レイヤ

OpenShift / Tanzu / GKE Anthos の API 互換は目指さない。k1s0 独自 API 体系（`k1s0.*`）で閉じる。

### エンドユーザートレーニング機能

研修 LMS、試験、認定制度は k1s0 のスコープ外。録画ビデオと Hands-on サンプルで自学自習をサポートするが、それらは機能としての要件化をしない。

### 公開 API のマーケットプレイス

社外向けの API 公開・課金・利用者管理は 採用側の全社ロールアウト 以降の別事業判断。現時点で除外。

## 将来検討（採用後の運用拡大時）

以下は将来的に取り込む可能性があるが、リリース時点〜採用後の運用拡大時の要件としては未確定。

- **AI / LLM 推論基盤**: pgvector による RAG は 採用側のマルチクラスタ移行時、推論基盤（GPU + モデルサービング）は 採用側の全社展開期。
- **マルチクラスタ管理クラスタ**: 採用側の全社ロールアウト。Backstage / 監視基盤 / Argo CD を管理クラスタに集約。
- **JDM エディタの Backstage プラグイン化**: 採用側の全社展開期。稟議ルール画面更新を実現。
- **Chaos Engineering の本番適用**: 採用後の運用拡大時。Litmus Chaos の Canary 実行を本番で月次実施。
- **SLO ベース SRE プラクティス**: 採用側のマルチクラスタ移行時。Error Budget ベースのリリース判断。

将来検討の方針は変わる可能性があるため、本書では箇条書きの方針のみ残し、段階ゲート時に改訂で昇格・変更・除外を判断する。

## スコープ境界の検証

スコープ判定で迷った場合、以下の質問を順に適用する。Yes / No の 2 値に分解することで、判断の再現性を確保する。

1. 本機能は tier1 公開 API の責務か
2. 本機能は複数の tier2/tier3 利用者で共有されるか
3. 本機能は既存 .NET Framework 資産との共存を崩さないか
4. 本機能は 5 年 TCO 枠内で工数確保できるか
5. 本機能は採用検討で外部約束した範囲内か

いずれかが No の場合、含めない（または改訂プロトコル経由で追加判断）。すべて Yes の場合、含める。境界が不明瞭な場合は [00_要件定義方針/02_ステークホルダー.md](../00_要件定義方針/02_ステークホルダー.md) のエスカレーションパスに従う。
