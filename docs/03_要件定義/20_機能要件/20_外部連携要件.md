# 外部連携要件

本書は、k1s0 が外部システム（既存 .NET Framework 資産、既存 IdP、既存監視・バックアップ基盤）と連携する機能要件を定義する。既存資産との共存は採用検討で約束した主要差別化ポイントであり、連携方式の曖昧さは リリース時点 パイロット合格判定を直接左右する。

## 連携の設計方針

k1s0 は新規技術の全量導入ではなく、既存資産との「段階的共存」を前提に設計する。連携方式は「サイドカー方式」「API Gateway 経由方式」「Federation 方式」の 3 パターンを使い分け、既存資産側の改修量を最小化する。新規連携先ごとに方式を議論し、採用パターンを明示する。

## .NET Framework 連携

### FR-EXT-DOTNET-001: .NET Framework サイドカー連携

**現状**: 既存 ASP.NET Web Forms / WCF サービスは .NET Framework 4.x 上で稼働し、gRPC クライアントライブラリのサポートが貧弱。k1s0 の tier1 公開 API を gRPC で呼ぶには、既存プロジェクトのビルド環境全面刷新が必要。

**要件達成後**: 既存 .NET Framework アプリの Pod に、Go または .NET 6 の k1s0 SDK を実装したサイドカーコンテナを同居させる。既存アプリはサイドカーに HTTP/1.1 で REST 呼び出しし、サイドカーが gRPC で tier1 を呼ぶ。既存アプリのコードは最小改修（HTTP クライアント追加のみ）で済む。

**崩れた時**: 既存 .NET Framework 資産を k1s0 から利用できず、新旧システム共存が リリース時点 の目標水準（共存可能性のパイロット検証）を満たせない。

**受け入れ基準**:

- サイドカー Helm Chart またはマニフェストテンプレート提供
- 既存アプリのコード改修量が 100 行以下（1 か所あたり）
- サイドカー経由の呼び出し p99 < 600ms（tier1 p99 500ms + 100ms オーバヘッド）
- k1s0 の認証トークンがサイドカーで自動付与

### FR-EXT-DOTNET-002: .NET Framework API Gateway 連携

**現状**: サイドカーを追加できないレガシーアプリ（例: VM 直接稼働）に対しては、サイドカーが適用できない。

**要件達成後**: Envoy Gateway に k1s0 API への変換ルートを設定し、レガシーアプリが外部 URL として k1s0 API を呼び出せるようにする。認証は API Gateway 側で既存トークン（AD 統合済み）から Keycloak トークンに変換する ext-authz 連携で吸収。

**崩れた時**: VM 直接稼働のレガシー資産を k1s0 から隔離せざるを得ず、基盤の統合価値が限定的になる。

**受け入れ基準**:

- Envoy Gateway ルート設定のテンプレート提供
- ext-authz による認証変換サポート
- レイテンシは p99 < 700ms（サイドカー方式 + 100ms）
- 優先度 SHOULD、リリース時点 で実装判定

## 既存 IdP 連携

### FR-EXT-IDP-001: 既存 AD / LDAP Federation

**現状**: 採用側組織の社内には既存の Active Directory（Windows Server）および LDAP ディレクトリが稼働しており、全社員の認証情報を管理している。k1s0 が独自認証基盤を持つと、ID 重複管理や SSO 体験の崩壊が発生する。

**要件達成後**: Keycloak を IdP プロキシとして設置し、既存 AD / LDAP を Federation（User Federation 機能）で上流に置く。k1s0 利用者は既存の社員 ID とパスワードでログイン、Keycloak がトークンを発行する。AD / LDAP の属性（部門、役職、メール）は Keycloak で取得し、JWT クレームとして tier2/tier3 に伝搬する。

**崩れた時**: 新規認証情報の発行・管理が必要となり、運用負荷が増大する。SSO 体験が崩れ、ユーザーから苦情が出る。

**受け入れ基準**:

- Keycloak User Federation 設定で AD / LDAP バインド
- 既存社員 ID での Keycloak ログイン成功
- AD 属性（部門コード、役職）が JWT クレームに含まれる
- パスワード変更は AD / LDAP 側で実施、Keycloak は透過

### FR-EXT-IDP-002: SAML IdP Federation

**現状**: 一部部門で既存 SAML IdP（ADFS 等）が稼働している場合、Keycloak の上流としての SAML Federation が必要。

**要件達成後**: Keycloak の Identity Broker 機能で SAML IdP を Federation する。エンドユーザーは既存 SAML IdP のログイン画面で認証し、Keycloak がトークンを発行する。

**崩れた時**: 既存 SAML IdP ユーザーが k1s0 を利用できず、基盤展開範囲が限定的になる。

**受け入れ基準**:

- SAML 2.0 IdP との Federation 設定
- SP-initiated / IdP-initiated 両方のフロー対応
- 属性マッピング設定可能
- 優先度 SHOULD、リリース時点 で判定

## 既存監視基盤連携

### FR-EXT-MON-001: 既存監視基盤への Severity 1 転送

**現状**: 既存監視基盤（Zabbix / SCOM 等）が 24 時間オンコール体制を運用している。k1s0 の監視アラートが既存基盤に届かないと、オンコール担当者が k1s0 アラートを見逃す。

**要件達成後**: k1s0 の Alertmanager から Severity 1 アラートを既存監視基盤に転送する Webhook / メール連携を実装。既存基盤の通知ルールに従って担当者へ通知される。Severity 2 以下は k1s0 側の通知経路で完結する。

**崩れた時**: オンコール担当者が既存基盤の通知に反応する体制下で、k1s0 の Severity 1 が見逃される。インシデント対応が遅延する。

**受け入れ基準**:

- Alertmanager Receiver 設定で Webhook / メール転送
- Severity 1 の全アラートが既存基盤に 1 分以内に届く
- 重複通知防止（k1s0 側と既存基盤の両方で同じアラート）の排除ルール
- 転送失敗時は k1s0 側でリトライ + SRE への別経路通知

## 既存バックアップ基盤連携

### FR-EXT-BACKUP-001: 既存バックアップ基盤との併存

**現状**: 採用側組織の既存バックアップ基盤（Commvault / Veritas NetBackup 等）が全社データのバックアップ運用を担当している。k1s0 が独自のバックアップ運用を持つと、運用二重化でコストと複雑度が増す。

**要件達成後**: k1s0 の重要データ（PostgreSQL フルバックアップ、MinIO スナップショット）を既存バックアップ基盤にも転送する。既存基盤のスケジュール・保管期間・リストア手順に従う。日次増分を k1s0 側で、週次フルを既存基盤側で運用する分担。

**崩れた時**: バックアップ運用の二重化でコストが増し、リストア時にどちらを使うべきか判断が遅れる。

**受け入れ基準**:

- 既存バックアップ基盤の CLI / SDK 経由でバックアップデータ転送
- 転送成功・失敗の監視と Severity 2 アラート
- リストア手順書（既存基盤使用時と k1s0 使用時）を 2 本保持
- 優先度 SHOULD、リリース時点 で判定

## 外部サービス連携の原則

本カテゴリで扱う以外の外部サービス連携（例: PagerDuty、Slack、GitHub）は、k1s0 基盤の必須依存として扱わない。tier3 の業務アプリが個別に必要とする場合は、Binding API（FR-T1-BINDING-003 HTTP Output）経由で呼び出す。k1s0 基盤そのものが外部 SaaS に依存すると、C-VEND-001 制約（外部商用 SaaS 依存禁止）に抵触する。

## 段階対応と受け入れ基準サマリ

| 要件 ID | 適用段階 | 優先度 | 判定理由 |
|---|---|---|---|
| FR-EXT-DOTNET-001 | 採用初期 | MUST | パイロット業務アプリで .NET 共存検証 |
| FR-EXT-DOTNET-002 | 採用初期 | SHOULD | サイドカー不可な既存資産の救済経路 |
| FR-EXT-IDP-001 | 採用初期 | MUST | 採用検討前のデモで SSO 体験が前提 |
| FR-EXT-IDP-002 | 採用初期 | SHOULD | SAML IdP 利用部門の取り込み判定 |
| FR-EXT-MON-001 | 採用初期 | MUST | オンコール体制の継続性 |
| FR-EXT-BACKUP-001 | 採用初期 | SHOULD | 運用分担の判断次第 |

## トレーサビリティ

- FR-EXT-DOTNET-001〜002 → C-EXIST-001（既存 .NET Framework との共存）
- FR-EXT-IDP-001〜002 → C-EXIST-002（既存認証基盤との統合）
- FR-EXT-MON-001 → C-EXIST-003（既存監視・バックアップ基盤との併存）
- FR-EXT-BACKUP-001 → C-EXIST-003
- FR-EXT-DOTNET-001〜002、FR-EXT-IDP-001〜002、FR-EXT-MON-001、FR-EXT-BACKUP-001 → 企画書の差別化ポイント「レガシー .NET Framework 共存」
