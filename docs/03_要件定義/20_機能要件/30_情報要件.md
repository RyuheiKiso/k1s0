# 情報要件

本書は、k1s0 が tier1 API を通じて生成・管理する情報（論理データモデル）の要件を定義する。個別の業務データ（経費、申請、見積等）は tier2 の責務でありここでは扱わず、tier1 基盤層の共通情報モデル（ログ、監査、設定、tenant）のみを対象とする。

## 情報モデルの位置付け

情報モデルは、tier2/tier3 の業務アプリが生成するデータと、k1s0 基盤自身が生成するデータの両方を含む。本書では後者の基盤データに絞り、標準フィールドとスキーマを規定する。このフィールド標準化が、Grafana Loki 検索の再利用性、Audit API の改ざん検知、tenant 越境防止の基礎となる。

## FR-INFO-LOGMODEL-001: 構造化ログのフィールド標準

**現状**: 言語別ログライブラリのデフォルトフィールド名（`msg` / `message` / `text`、`ts` / `time` / `timestamp` 等）が揃わないと、Loki の LogQL クエリが言語別にバラつく。

**要件達成後**: 全言語 SDK で以下のフィールドを必須とする。

- `timestamp`: RFC3339 ナノ秒精度（UTC）
- `level`: `debug` / `info` / `warn` / `error` / `fatal`
- `message`: 人間可読な本文
- `service`: サービス名（`<domain>-service` 形式を推奨）
- `tenant_id`: テナント ID（JWT クレームから自動抽出）
- `trace_id`: W3C Trace Context の trace_id
- `span_id`: W3C Trace Context の span_id
- `user_id`: JWT の sub クレーム（システム処理時は null）
- `extra`: 任意の追加フィールド（map<string, any>）

**崩れた時**: Loki クエリで言語別の分岐が必要になり、運用ダッシュボードが言語ごとに作られて乱立する。

**受け入れ基準**:
- SDK 自動付与フィールドの欠落が CI でブロックされる
- Loki の LogQL で全サービス横断の検索が機能する
- JSON スキーマドキュメントが Backstage で公開

## FR-INFO-AUDITMODEL-001: 監査イベントスキーマ

**現状**: 監査イベントのスキーマが定まらないと、tier2 の監査ログ形式がバラバラとなり、監査対応のクエリコストが膨らむ。

**要件達成後**: Audit API に記録されるイベントは以下のスキーマで統一する。

- `sequence_number`: テナント内の通し番号（int64、単調増加）
- `timestamp`: RFC3339 ナノ秒精度
- `tenant_id`: テナント ID
- `user_id`: 操作者（システム操作時は `system:<service>`）
- `action`: 操作種別（`create` / `update` / `delete` / `read` / `login` / `execute` 等）
- `resource_type`: 操作対象リソース種別（例: `state.key`、`workflow.instance`、`decision.table`）
- `resource_id`: 操作対象の識別子
- `result`: `success` / `failure`
- `ip_address`: クライアント IP
- `user_agent`: クライアント User-Agent
- `previous_hash`: 前イベントの event_hash（base64）
- `event_hash`: 本イベントの SHA-256 ハッシュ（base64）
- `attributes`: 追加情報（map<string, any>、PII はマスキング済み）

**崩れた時**: 監査のクエリ・集計が部門別の独自形式となり、全社横断の不正検知ができない。

**受け入れ基準**:
- Protobuf 定義ファイルで公式スキーマを公開
- 新規 action / resource_type の追加は PR レビュー必須（予約語管理）
- attributes 内に PII 生値が含まれていないことを Kyverno / CI で検証

## FR-INFO-TENANTMODEL-001: tenant_id 必須モデル

**現状**: テナント越境を防ぐには tenant_id の一貫した取扱いが必要だが、データ層ごとにフィールド名が異なる（`tenant` / `tenant_id` / `realm` 等）と越境検知が難しい。

**要件達成後**: tier1 が扱う全データにおいて tenant_id のフィールド名は `tenant_id` で統一する。PostgreSQL は `tenant_id` 列で Row-Level Security を適用、Kafka はパーティションキーに使用、MinIO は bucket プレフィックスに使用、Valkey はキープレフィックスに使用する。tenant_id の型は文字列（最大 64 文字、英数字とハイフンのみ）。

**崩れた時**: データ層ごとにテナント識別子が異なり、越境監査が困難になる。新規データ層採用時に設計が揺れる。

**受け入れ基準**:
- Protobuf で tenant_id の共通型を定義
- PostgreSQL スキーマで全テーブル `tenant_id` 列必須（Kyverno / CI 検証）
- Kafka トピック命名規約に tenant_id 含める
- MinIO bucket プレフィックスに tenant_id 含める
- Valkey キープレフィックスに tenant_id 含める

## FR-INFO-CONFIGMODEL-001: 設定情報モデル

**現状**: tier1 基盤の設定（Dapr Component YAML、Feature Flag、Decision 決定表、Binding 設定等）が散在し、管理責任・変更フロー・監査がバラつく。

**要件達成後**: tier1 基盤の設定は以下の原則で一元管理する。

- Dapr Component YAML は `src/tier1/components/` に集中管理、Argo CD デプロイ
- Feature Flag は `src/tier1/flags/` の flagd JSON で集中管理
- Decision 決定表は `src/tier1/decisions/` の JDM JSON で集中管理
- Binding 設定は `src/tier1/bindings/` に集中管理
- 全設定変更は PR レビュー必須、Audit API に記録

**崩れた時**: 設定が散在してドキュメント鮮度が劣化、緊急変更時の影響範囲判定が困難になる。

**受け入れ基準**:
- 設定ファイルのディレクトリ構造が Backstage で可視化
- PR マージ時に Argo CD が自動同期
- 設定変更の Audit 記録（コミット hash、承認者）
- 優先度 SHOULD、Phase 1b で実装

## 情報保持期間

tier1 基盤が生成する情報の保持期間を以下に定義する。保持期間満了時は自動削除し、削除操作自体も Audit 記録する。

| データ種別 | ホット保持 | ウォーム保持 | アーカイブ保持 | 合計 |
|---|---|---|---|---|
| アプリケーションログ | Loki 7 日 | - | - | 7 日 |
| 監査ログ | Loki 90 日 | PostgreSQL 1 年 | MinIO 7 年 | 7 年 |
| メトリクス | Prometheus 15 日 | Thanos / Mimir 13 か月 | - | 13 か月 |
| 分散トレース | Tempo 7 日 | - | - | 7 日 |
| Profiling | Pyroscope 7 日 | - | - | 7 日 |
| Feature Flag 変更履歴 | Git 無期限 | - | - | 無期限 |
| Decision 決定表履歴 | Git 無期限 | - | - | 無期限 |

## スキーマ進化ルール

情報モデルのスキーマ変更は後方互換を原則とする。フィールド追加はオプショナル、フィールド削除は DEPRECATED マーク → 3 か月猶予 → 削除の手順で段階的に実施する。必須フィールドの意味変更は禁止（新フィールドの追加で代替）。

スキーマ定義は Protobuf で行い、Buf lint / Buf breaking 検出で CI ブロックする。構想設計フェーズの「src/tier1/contracts/」に集中管理し、言語別 SDK は CI で自動生成する。

## トレーサビリティ

- FR-INFO-LOGMODEL-001 → FR-T1-LOG-001、003（Log API 基本機能、tenant_id 必須）
- FR-INFO-AUDITMODEL-001 → FR-T1-AUDIT-001、002（監査ログ、改ざん検知）
- FR-INFO-TENANTMODEL-001 → 全 tier1 API、NFR-E-AC-003（tenant 越境防止）
- FR-INFO-CONFIGMODEL-001 → NFR-C-MGMT-001、002（設定管理）

## 非機能要件との接続

情報要件は個々のフィールド・スキーマを規定するが、保持期間の達成・バックアップ・アクセス制御は非機能要件（30_非機能要件）で数値化する。スキーマ互換性の維持運用は NFR-C-MNT-002 で扱う。
