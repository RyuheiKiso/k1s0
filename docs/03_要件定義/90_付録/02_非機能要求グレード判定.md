# 02. 非機能要求グレード判定

本書は IPA 非機能要求グレード（2018 年改訂版）に基づき、k1s0 プラットフォームの社会的影響度モデルを判定し、各非機能カテゴリ A〜H のレベル設定の根拠を示す。非機能要求グレードは日本国内の発注側・受注側で広く参照される非機能分類体系であり、稟議・RFP 回答・契約締結で「このシステムはグレードいくつか」が繰り返し問われる。

## 本書の位置付け

非機能要件は定性的な記述に流れやすく、「高可用性」「堅牢なセキュリティ」といった曖昧なラベルで終わると稟議や監査で突き崩される。IPA 非機能要求グレードは「①社会的影響がほとんど無いシステム」「②社会的影響が限定されるシステム」「③社会的影響が極めて大きいシステム」の三モデルを提示し、各モデル別のレベル設定指針を示す。本書は k1s0 がどのモデルに該当するかを判定し、A〜H 各カテゴリのレベルを裏付ける。

## 社会的影響度モデルの判定

k1s0 は「**モデル②社会的影響が限定されるシステム**」に該当すると判定する。判定根拠は以下の通り。

- **利用範囲**: JTC 内の従業員および限定的な取引先のみ。一般消費者向けではない。
- **障害時の社会的影響**: 業務停止による JTC 社内業務の遅延は発生するが、公共インフラや不特定多数の消費者への直接影響は限定的。
- **取扱データ**: 従業員個人情報、取引先情報、業務データ（営業機密を含む）。カード決済情報・医療情報・公共サービスデータは扱わない。
- **金銭損失規模**: 障害 1 日あたり数千万円〜億単位の機会損失想定（JTC 内業務停止）。数十億規模の直接金銭損失は想定外。

ただし将来的な扱いによってはモデル③に昇格する可能性がある。特に以下の条件下ではモデル③再判定を要する。

- テナントが金融商品取引や決済業務を扱う場合
- 個人情報保護法上の要配慮個人情報（医療・思想信条等）を大量に扱う場合
- 政府・自治体案件としての利用で公共インフラ認定を受けた場合

## カテゴリ別レベル設定

IPA 非機能要求グレードは A〜F の 6 カテゴリだが、本要件定義書では G（データ保護とプライバシー）・H（アーティファクト完整性とコンプライアンス）を追加している。以下、A〜H 各カテゴリでのレベル設定を示す。

### A. 可用性

モデル②標準レベル（レベル 3）を採用。

- **稼働率**: 99.9%（月間許容停止 43 分）、tier1 コア API は 99.95% 目標
- **RTO**: 4 時間、RPO: 秒オーダー（ログ・イベント系は秒、状態系は数分）
- **根拠**: JTC 内業務の継続性確保が目的。24/365 無停止は過剰、平日営業時間帯の可用性が厚く、夜間週末は計画停止許容
- **関連要件**: NFR-A-AVL-001〜004、NFR-A-REC-001〜003、NFR-A-CONT-001〜003

### B. 性能・拡張性

モデル②標準〜高レベル（レベル 3〜4）を採用。

- **応答時間**: tier1 API p99 < 500ms、ピーク時 1.5 倍まで許容
- **スループット**: 初期 100 RPS、Phase 2 で 1,000 RPS
- **拡張性**: HPA 自動スケール、水平分散対応
- **根拠**: 業務 UI からの呼び出しが主。ミリ秒単位のレイテンシ（HFT 等）は不要だが、ユーザー体験として 500ms 超は業務ストレスとなる
- **関連要件**: NFR-B-PERF-001〜004、NFR-B-RES-001〜003、NFR-B-WL-001〜002

### C. 運用保守性

モデル②高レベル（レベル 4）を採用。

- **自動化**: GitOps、自動バックアップ、自動スケール、自動署名検証
- **監視**: Grafana LGTM（Logs/Grafana/Tempo/Mimir）で統合観測性
- **運用コスト**: オンコール 1 名体制、bus factor 3 以上
- **根拠**: 運用工数削減と品質向上は TCO 削減の主要ドライバー。稟議書で「運用自動化で運用コスト 30% 削減」を掲げた以上、このレベルは必達
- **関連要件**: NFR-C-MNT-001〜003、NFR-C-NOP-001〜003、NFR-C-OPS-001〜002

### D. 移行性

モデル②標準レベル（レベル 3）を採用。

- **移行戦略**: サイドカー方式・API Gateway 方式の併用、既存 .NET Framework 資産との共存
- **パイロット**: 業務クリティカル度の低い社内管理系から着手、Phase 1b でパイロット合格基準達成
- **撤退計画**: NFR-D-PLN-003 で撤退計画を事前策定
- **根拠**: 既存資産保護は稟議通過の前提。全量置換は稟議不通過リスクが高いため漸進的移行を前提
- **関連要件**: NFR-D-TIM-001〜002、NFR-D-MTH-001〜002、NFR-D-OBJ-001〜003、NFR-D-PLN-001〜003

### E. セキュリティ

モデル②高レベル（レベル 4）を採用。モデル③相当の厳格さが必要な項目あり。

- **認証認可**: Keycloak SSO、mTLS 全通信、SPIFFE ワークロード ID
- **暗号化**: 保存時 AES-256（LUKS + pgcrypto）、転送時 TLS 1.3、アプリ層フィールド暗号化
- **監視**: SIEM 連携、異常検知、脆弱性スキャン（SAST/DAST/SCA）自動化
- **根拠**: 個人情報保護法対応と JTC セキュリティ規程はモデル③相当の厳格さを要求する。特にアクセス制御・暗号化は妥協不可
- **関連要件**: NFR-E-AC-001〜005、NFR-E-ENC-001〜003、NFR-E-MON-001〜002、NFR-E-LIC-001、NFR-E-SAFE-001

### F. システム環境・エコロジー

モデル②標準レベル（レベル 3）を採用。

- **オンプレミス**: JTC 自社データセンター内完結、閉域ネットワーク
- **VM ベース**: 物理 6〜10 ノード、PostgreSQL/Valkey/Kafka を含むフルスタック
- **電力効率**: HPA による負荷追従、OpenCost による可視化、CO2 換算レポート
- **根拠**: JTC データ主権要件と社内規程によりオンプレミス必須。エコロジーは ESG 目標への貢献として盛り込み
- **関連要件**: NFR-F-SYS-001〜003、NFR-F-CHR-001〜003、NFR-F-STD-001〜002、NFR-F-FAC-001〜002、NFR-F-ECO-001〜003

### G. データ保護とプライバシー

モデル②高レベル〜モデル③相当（レベル 4〜5）を採用。

- **機密度分類**: 公開/社内/機密/極秘の 4 段階、PII 自動検出
- **保管場所**: 国内保管必須（個人情報保護法・JTC データ主権）
- **ライフサイクル**: 退会後 30 日論理削除、90 日物理削除
- **漏えい通知**: T+24h 第一報、T+72h 個人情報保護委員会報告（個人情報保護法準拠）
- **根拠**: 個人情報保護法 2022 年改正の要配慮個人情報・第三者提供規制は厳格。違反時のレピュテーションリスクは甚大
- **関連要件**: NFR-G-CLS-001〜002、NFR-G-ENC-001〜003、NFR-G-AC-001〜002、NFR-G-RES-001、NFR-G-LIF-001〜002、NFR-G-PRV-001〜003、NFR-G-DES-001〜002

### H. アーティファクト完整性とコンプライアンス

モデル②高レベル（レベル 4）を採用。

- **供給チェーン**: SLSA Level 3、Sigstore 署名、SBOM CycloneDX、Rekor 透明性ログ
- **鍵管理**: OpenBao 一元管理、ライフサイクル（生成・ローテーション・失効）
- **法令対応**: 個人情報保護法、マイナンバー法、J-SOX、ISO 27001/27017
- **監査**: 監査ログ WORM、外部監査対応（年次）
- **根拠**: サプライチェーン攻撃（SolarWinds・Log4Shell 等）の社会的警戒度を踏まえ、検証可能な完整性担保は必須。監査対応は稟議で外せない項目
- **関連要件**: NFR-H-INT-001〜004、NFR-H-KEY-001、NFR-H-COMP-001〜004、NFR-H-AUD-001〜002、NFR-H-TRN-001

## 非機能項目別判定サマリ

判定結果を俯瞰するための集計を以下に示す。

- **レベル 5（モデル③相当）採用**: G の一部（漏えい通知・国内保管）
- **レベル 4（モデル②高レベル）採用**: C、E、G の多数、H
- **レベル 3（モデル②標準）採用**: A、B の多数、D、F
- **レベル 3 未満**: 該当なし（プラットフォームの性格上、下限が高い）

## 判定根拠のトレーサビリティ

稟議書・RFP 回答で「なぜこのレベル設定か」を問われた際の回答テンプレートとして、各カテゴリの「根拠」セクションを直接引用する運用を想定する。監査対応でもこの根拠記述を提示すれば、レベル設定の妥当性を示せる。

## メンテナンス

社会的影響度モデルの判定は年次で再評価する。テナント業務範囲の拡大、法令改正、JTC セキュリティ規程改訂があれば随時見直し。モデル②→③への昇格判定となった場合は全カテゴリのレベルを再設定し、要件定義書全体を見直す。
