# SEC-DAT: データ管理要件

本ファイルは、k1s0 プラットフォームの**データモデルの定義・スキーマ変更の管理・マスタデータの配信・PII カラムの明示的マーキング**を定義する。プライバシー（`05_privacy.md`）と監査（`03_audit.md`）が実効性を持つためには、そもそも「どのデータが存在し、どのカラムが PII で、どのテナントのものか」が構造化されている必要がある。本ファイルはその構造的前提を与える。

JTC 環境特有の論点として、「テナント別に物理分離するか論理分離するか」の判断がある。物理分離は安心感が高い反面運用工数が線形に増え、論理分離は Postgres RLS や ABAC で安全性を担保できるが実装ミスのリスクが残る。本ファイルでは顧客セグメント別に判断基準を明文化し、Phase 1a は論理分離既定・金融セクターは物理分離選択可の段階運用とする。

受け入れ基準は本カテゴリ標準の**検知 / 防御 / 復旧**の 3 軸で記述する。データモデルの逸脱は事後の復旧コストが極端に高いため、宣言的マーキング（declarative marking）による CI 段階での検知を重視する。

## 要件本体

### SEC-DAT-001: スキーマは Protobuf で定義し、マイグレーションを管理する

- 優先度: MUST（COM-CON-002 の帰結。Rust / Go 境界の型整合を担保）
- Phase: Phase 1a
- 関連: `COM-CON-002`、`ARC-API-*`

現状、DB スキーマと API 契約が別々に管理され、アプリコードとスキーマの乖離がランタイムエラーとして顕在化する運用は、JTC で頻発する技術的負債の典型例となっている。

本要件が満たされた世界では、tier1 の全データ型は Protobuf スキーマから生成され、DB スキーマ（PostgreSQL）は sqlx / sqldef + Atlas などのマイグレーションツールでバージョン管理される。マイグレーションは up / down ペアで記述し、各 PR で自動適用テストが走る。

崩れた場合、本番 DB と開発環境の差分が発覚し、デプロイ時の事故（unknown column / type mismatch）で業務停止に至る。

**受け入れ基準**

- 検知: Protobuf と DB スキーマの差分検知 job を CI に組み込む（schema drift 検出）
- 防御: マイグレーションは up / down 必須、本番適用前にステージング環境で自動 dry-run
- 防御: Protobuf の後方互換破壊変更（フィールド番号変更・型変更）は CI 拒否
- 復旧: マイグレーション失敗時の自動ロールバック（`down`）、復旧時間 15 分以内

**検証方法**

- `buf breaking` を CI に組み込み、後方互換破壊を検知
- 毎夜 main ブランチでマイグレーション往復テスト（up → down → up）

### SEC-DAT-002: テナント分離は論理分離を既定、物理分離は顧客選択可

- 優先度: MUST（テナント越境は契約上の重大事故）
- Phase: Phase 1a
- 関連: `QUA-TNT-*`、`SEC-IAM-002`、`SEC-RES-*`

現状、SaaS 共通 DB の論理分離実装では、`WHERE tenant_id = :x` の付与漏れが Row-Level（越境漏洩）事故の典型原因となる。一方、テナントごとに DB を分けると運用工数が線形に増える。

本要件が満たされた世界では、Phase 1a では PostgreSQL Row Level Security (RLS) による**論理分離**を既定とし、全テーブルに `tenant_id` 列を必須化する。セッション変数 `app.current_tenant` を tier1 middleware で強制注入し、RLS ポリシー `USING (tenant_id = current_setting('app.current_tenant'))` で物理的に別テナント行を読めなくする。金融・マイナンバー・医療系の顧客は**物理分離**（別 DB cluster / 別 namespace）を選択可能にし、顧客セグメント別の判断を `onboarding` 時に固定する。

崩れた場合、1 つのバグで複数テナントのデータが横断的に漏洩し、全顧客への一斉通知・契約違反賠償が発生する。

**受け入れ基準**

- 検知: 全テーブルの `tenant_id` 列存在チェックを CI で実施、欠落は PR ブロック
- 防御: RLS ポリシーは全テーブルに enforce、`BYPASSRLS` 権限はアプリロールに付与しない
- 防御: tier1 middleware が session 変数 `app.current_tenant` を必ず設定、未設定クエリは ERROR
- 復旧: 越境アクセス検知時は該当セッションを即時 revoke、監査ログで影響範囲を特定

**検証方法**

- テナント越境クエリを自動テストで全網羅（matrix）、RLS で拒否されることを確認
- 金融顧客向けの物理分離オプションを Phase 1a POC で 1 顧客ぶんデプロイ

### SEC-DAT-003: マスタデータの配信方式を定義する

- 優先度: SHOULD（Phase 1a ではファイル配布容認、Phase 1b で API 化必達）
- Phase: Phase 1b
- 関連: `ARC-T1-*`、`ARC-EVT-*`

現状、組織マスタ・商品マスタ・会計コードマスタなど JTC 横断で共有される参照データが、各アプリで独自複製されて更新遅延と整合性崩れを起こす。

本要件が満たされた世界では、マスタデータは tier1 `k1s0.Master` 配下に集約され、各 tier2 / tier3 は pull (gRPC) または push (Dapr pub-sub) で同期する。マスタ更新は CDC (Change Data Capture) で Kafka topic に publish され、subscribers は eventually consistent で追随する。

崩れた場合、マスタ不整合による業務エラー（存在しない部門コードで決済処理、無効な勘定コードで仕訳）が発生する。

**受け入れ基準**

- 検知: マスタ同期遅延の監視ダッシュボード（各 subscriber の lag）
- 防御: マスタ更新は承認ワークフロー経由、`SEC-IAM-005` と連動
- 防御: マスタ version が全 subscriber 間で 5 分以内に揃う（Phase 1b 目標）
- 復旧: 不整合検知時の再同期コマンド（強制再 snapshot）を運用 API で提供

**検証方法**

- Phase 1b で 10 subscribers 構成で配信遅延を計測、p95 < 5 分を確認

### SEC-DAT-004: PII カラムの宣言的マーキング

- 優先度: MUST（改正個人情報保護法 第 23 条「安全管理措置」の具体化）
- Phase: Phase 1a
- 関連: `SEC-PRV-002`、`SEC-AUD-001`

現状、どのカラムが個人情報に該当するかの一覧が Excel で別管理されており、スキーマ変更と手動同期する運用は高確率で乖離する。PII 漏洩事故の根本原因分析で「PII 特定が不正確」が頻繁に出てくる。

本要件が満たされた世界では、Protobuf 定義にカスタムオプション `[(k1s0.pii) = PII_NAME | PII_EMAIL | PII_PHONE | ...]` を付与することで PII を宣言する。このメタデータから以下が自動生成される。

- マスキング対象（`SEC-PRV-002`）
- 監査ログのデータアクセスイベント発火対象（`SEC-AUD-001`）
- 暗号化対象カラム（`SEC-KMS-004` の Transit Engine 適用）
- データ開示要求時の抽出対象（`SEC-PRV-004`）

崩れた場合、PII と非 PII の境界が曖昧化し、過剰暗号化で性能劣化、または見落としで漏洩リスクが並立する。

**受け入れ基準**

- 検知: `.proto` ファイルを parse して PII マーク一覧を生成、git 上で差分追跡
- 防御: PII マークのないカラムに個人情報と思われる名称（`email`、`phone`、`name` など）があれば CI 警告
- 防御: PII マーク付きカラムに対する SELECT は自動的にデータアクセスイベント（`SEC-AUD-001`）を発火
- 復旧: マーク漏れ発覚時は緊急 PR で追加し、過去の監査ログで影響範囲を事後評価

**検証方法**

- Protobuf compiler plugin で PII マーキングから auto-generated コードを生成
- 四半期ごとに全 `.proto` の PII マーク網羅性を法務レビュー

### SEC-DAT-005: データ分類（機密度レベル）の 4 段階定義

- 優先度: MUST（データ分類がないと適切な防御レベルを選択できない）
- Phase: Phase 1a
- 関連: `SEC-RES-*`、`SEC-KMS-*`、`SEC-AUD-*`

現状、すべてのデータを同一レベルで扱うと「過剰防御でコスト増」と「防御不足で漏洩リスク」が同居し、経営的合理性がない。

本要件が満たされた世界では、全データに以下 4 段階のいずれかの分類が付与され、分類ごとの最小要件が定義される。

- `PUBLIC`: 公開可能な情報（マスタコード、公開カタログ）
- `INTERNAL`: 社内一般（社内マニュアル、非公開マスタ）
- `CONFIDENTIAL`: 業務データ（取引情報、顧客データ一般）
- `RESTRICTED`: 特定機微データ（PII、マイナンバー、医療情報、金融取引詳細）

分類ごとに暗号化・アクセス制御・監査・保存場所の要件が異なり、`SEC-RES-*` の国内保管は `CONFIDENTIAL` 以上に、物理分離は `RESTRICTED` に自動適用される。

崩れた場合、防御コストと漏洩リスクの最適化ができず、FISC 安対基準 準則 7.1（情報資産の分類）に抵触する。

**受け入れ基準**

- 検知: 各テーブル・各 Protobuf message に `data_classification` オプションが付与されていること
- 防御: `RESTRICTED` データのアクセスは MFA 必須（`SEC-IAM-006`）かつ `audit.data` topic に記録
- 防御: 分類を引き下げる変更（`RESTRICTED` → `CONFIDENTIAL`）は法務承認必須
- 復旧: 分類誤りが発覚した場合、影響範囲評価と該当データの再暗号化手順を Runbook 化

**検証方法**

- 四半期ごとにデータ分類の網羅性とマッピング整合を法務・セキュリティ合同レビュー

### SEC-DAT-006: データ保持期間の宣言的定義

- 優先度: MUST（改正個人情報保護法 第 22 条「利用目的達成後の削除努力義務」対応）
- Phase: Phase 1b
- 関連: `SEC-BCK-*`、`SEC-PRV-004`

現状、保持期間が不明確なまま全データが永続化されると、GDPR や改正個人情報保護法 第 22 条の削除努力義務が履行できない。保持コストも線形に増加する。

本要件が満たされた世界では、各テーブル・各 Protobuf message に `retention_days` オプションが付与され、Cron Job が期限超過レコードを自動削除または擬似化する。削除実行は `SEC-AUD-*` に記録。

崩れた場合、不要 PII の長期保持で漏洩時のインパクト拡大、かつ個人情報保護委員会ガイドラインの趣旨に反する。

**受け入れ基準**

- 検知: 保持期間超過レコードの日次スキャン
- 防御: 削除 Job は `SEC-IAM-005` 承認経由、削除対象件数 > 10,000 は事前通知
- 防御: 削除は論理削除→物理削除→crypto-shredding の 3 段階
- 復旧: 誤削除 30 日以内は論理削除状態から復旧可能（`undelete` API）

**検証方法**

- Phase 1b で保持期間超過データの自動削除パイプラインを POC
