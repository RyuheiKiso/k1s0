# SEC-PRV: プライバシー / 個人情報保護要件

本ファイルは、k1s0 プラットフォームが**改正個人情報保護法（2022 年 4 月施行）** および関連ガイドラインを満たすための要件を定義する。プラットフォーム自身は業務データを持たないが、tier3 業務アプリが法令を満たすための**基盤機能**を提供する必要がある。企画 [`01_企画/05_法務サマリ/02_日本法令適合.md`](../../01_企画/05_法務サマリ/02_日本法令適合.md) で整理した責任分界を、本ファイルで要件として固定する。

改正個人情報保護法の要点は 4 点ある。保有個人データの**開示・訂正・削除請求**への対応（第 28〜35 条）、**漏えい等の報告・通知義務**（第 26 条）、**仮名加工情報 / 匿名加工情報**の取扱い（第 41・42 条）、**第三者提供記録**（第 29・30 条）である。tier3 業務アプリの実装負荷を最小化するため、これらすべてに対応する基盤機能を tier1 `k1s0.Privacy` に集約する。

受け入れ基準は本カテゴリ標準の**検知 / 防御 / 復旧**の 3 軸で記述する。プライバシー事故は事後復旧が特に困難であり、「漏れないこと」より「漏れた時に 72 時間以内に把握・通知できること」を優先する。

## 要件本体

### SEC-PRV-001: 改正個人情報保護法（2022 年改正）への適合基盤を提供

- 優先度: MUST（COM-CON-005 の直接の帰結）
- Phase: Phase 1a
- 関連: `COM-CON-005`、`SEC-CMP-001`

現状、各 JTC が個別に個人情報保護法対応を実装し、「開示請求に数日かかる」「漏えい報告の範囲特定ができない」などが法改正ごとに再発する。

本要件が満たされた世界では、tier1 `k1s0.Privacy` API が開示・訂正・削除・漏えい報告の共通インタフェースを提供し、tier3 は規定の API を呼ぶだけで法令要件を満たせる。法改正時は tier1 の更新で全テナントが一斉対応できる。

崩れた場合、各 tier3 が独自実装で法令違反し、個人情報保護委員会の行政指導・改善命令（第 148 条）を受けるリスクが発生する。

**受け入れ基準**

- 検知: `k1s0.Privacy` API の利用状況を Backstage でテナント別可視化
- 防御: 法改正時の影響評価プロセスを定義、改正公布から 6 ヶ月以内に対応完了
- 防御: tier3 が独自の個人情報取扱ロジックを実装する場合は法務レビュー必須
- 復旧: 対応漏れ発覚時はテナント別に優先度付けし、2 週間以内に改修

**検証方法**

- Phase 1a で `k1s0.Privacy` の最小実装（開示・削除 API）を POC
- 四半期ごとに法改正動向を法務部と棚卸し

### SEC-PRV-002: PII マスキングの自動化

- 優先度: MUST（非特権ユーザーへの過剰開示を構造的に防ぐ）
- Phase: Phase 1a
- 関連: `SEC-DAT-004`、`SEC-IAM-002`

現状、開発環境や運用ダッシュボードで本番 PII が無マスキングで閲覧でき、改正個人情報保護法 第 23 条「安全管理措置」の趣旨に反する運用がある。

本要件が満たされた世界では、`SEC-DAT-004` の PII 宣言から自動生成されるマスキングルールが、非特権ユーザーに対して透過的に適用される。`PII_NAME` は最初の 1 文字 + `*`、`PII_EMAIL` はドメイン以外を `*`、`PII_PHONE` は末尾 4 桁のみ表示など、PII 種別ごとのマスキング方針が中央管理される。

崩れた場合、「運用担当者が顧客情報をスクリーンショットで持ち出し」のような内部不正が発生し、改正個人情報保護法 第 26 条の漏えい等報告義務が発動する。

**受け入れ基準**

- 検知: マスキング解除（raw 表示）は特権ロールのみ、全件 `audit.data` に記録
- 防御: tier1 `k1s0.Mask` がアプリから呼ばれる前提で、DB 直接クエリ経路には pgAudit + Pod 境界制御で代替防御
- 防御: マスキング対象は `SEC-DAT-004` の PII 宣言から auto-generated、人手編集禁止
- 復旧: マスキングバグでの過剰マスキング（正常業務が止まる場合）も revert 手順を Runbook 化

**検証方法**

- PII カラムに対する全 SELECT で自動マスキングが適用されることを property test で網羅
- 四半期ごとに本番 DB の PII アクセス権を有する全アカウントを棚卸し

### SEC-PRV-003: 同意記録の保管と検証

- 優先度: MUST（改正個人情報保護法 第 17・18 条「利用目的の特定」「同意取得」）
- Phase: Phase 1a
- 関連: `SEC-AUD-*`、`BIZ-ONB-*`

現状、「同意を取得した」の証跡が PDF メール・Web フォームのログなどに分散保管され、監査時に突合できない。

本要件が満たされた世界では、全同意取得は tier1 `k1s0.Consent` API を経由し、以下を監査ログ（`SEC-AUD-*` の `audit.user`）に記録する。

- 同意主体（本人の識別子）
- 同意日時（ミリ秒精度 + TSA タイムスタンプ）
- 同意した利用目的（バージョン管理された同意文言の版 ID）
- 同意の証跡（電子署名 / IP + UA + フォーム入力値ハッシュ）
- 撤回可能性の明示

崩れた場合、「本人同意を得た証拠がない」状態で個人情報を利用していたことになり、第 20 条違反で勧告・命令の対象となる。

**受け入れ基準**

- 検知: 同意なしに PII 処理を行うコードパスがないか四半期スキャン
- 防御: `k1s0.Privacy.Process` API は同意検証に失敗するとエラーで呼び出し拒否
- 防御: 同意文言の版変更は新規同意取得を発火、旧同意は期限付きで有効
- 復旧: 同意取得漏れが発覚した場合、該当本人への追加同意依頼を 2 週間以内に実施

**検証方法**

- Phase 1a で同意取得の最小実装 POC、監査ログで検証可能性を確認

### SEC-PRV-004: 開示・訂正・削除請求への対応（第 28〜35 条）

- 優先度: MUST（法令期限 2 週間以内の対応不能は行政処分リスク）
- Phase: Phase 1b
- 関連: `SEC-KMS-006`、`SEC-DAT-004`

現状、開示請求に対し各 tier3 が個別に DB クエリで対応する運用は、数テナント・数業務にまたがる請求で 1 件数日を要する。

本要件が満たされた世界では、本人識別子を与えると tier1 が全 tier3 に横断的に問い合わせ、該当 PII を収集・匿名化・構造化して開示レポート（JSON/PDF）を生成する。訂正は更新 API、削除は `SEC-KMS-006` の crypto-shredding で実現する。

崩れた場合、法令期限（2 週間以内、やむを得ない事情があれば延長可）を超過し、個人情報保護委員会の勧告・命令の対象となる（第 148・149 条）。

**受け入れ基準**

- 検知: 開示請求から対応完了までの時間を全件計測、SLA 超過は経営報告
- 防御: `k1s0.Privacy.Disclose` / `Rectify` / `Erase` API を tier1 で提供、全 tier3 が参加する前提
- 防御: 対応期限は 10 営業日（法令 2 週間より前倒し、安全マージン確保）
- 復旧: 対応漏れ発覚時は追加対応と本人謝罪レターを 24 時間以内に送付

**検証方法**

- Phase 1b で擬似請求を 5 件実施し、p95 < 5 営業日で完遂することを確認

### SEC-PRV-005: 漏えい等報告・通知の即応能力（第 26 条）

- 優先度: MUST（発覚から 72 時間以内の速報義務）
- Phase: Phase 1a
- 関連: `SEC-AUD-*`、`SEC-SEC-007`

現状、漏えい発覚時に「誰の何が漏れたか」の範囲特定に数日〜数週間を要し、法令の速報期限（速やかに、目安 72 時間）を守れない。

本要件が満たされた世界では、`audit.data` topic のイベント検索で「ある時刻に異常アクセスを受けたユーザー」「そのアクセスが触れた PII カラム」を即座に抽出できる。tier1 `k1s0.Privacy.IncidentScope` が影響範囲を機械的に算出し、本人通知用の宛先リストを生成する。

崩れた場合、報告遅延で個人情報保護委員会から勧告を受け、JTC ブランドへの毀損が発生する。

**受け入れ基準**

- 検知: 異常アクセスパターンを `SEC-SEC-007`（Falco）で検知し、`k1s0.Privacy.IncidentScope` を自動起動
- 防御: 影響範囲の機械的算出で人的ミス（通知漏れ）を排除
- 防御: 本人通知文言のテンプレートを法務承認済みで tier1 が保持
- 復旧: 誤検知（実は漏えいなし）時の誤通知取消し手順も用意

**検証方法**

- 四半期テーブルトップ演習で「発覚から通知準備完了」を 72 時間以内で達成できるか計測

### SEC-PRV-006: 第三者提供記録（第 29・30 条）

- 優先度: MUST（法令要求。記録がないと提供自体が違法）
- Phase: Phase 1b
- 関連: `SEC-AUD-001`、`SEC-DAT-005`

現状、業務提携先への PII 提供記録が Excel で個別管理され、受領者側の記録と突合できない。

本要件が満たされた世界では、全第三者提供は `k1s0.Privacy.ThirdPartyTransfer` API を経由し、提供者・受領者・提供日時・提供データ項目・提供目的・本人同意 ID を監査ログに記録する。受領者側の記録義務（第 30 条）にも対応した交換フォーマット（CSV + 電子署名）を提供する。

崩れた場合、第 29 条違反となり、指導・勧告・命令の対象となる。

**受け入れ基準**

- 検知: 第三者提供イベントの不在期間（1 ヶ月以上）は法務レビューで異常チェック
- 防御: `k1s0.Privacy.ThirdPartyTransfer` 以外の経路での PII 外部送信は NetworkPolicy + Istio AuthorizationPolicy で拒否
- 防御: 記録は 3 年間保持（法令 3 年、`SEC-AUD-004` で十分）
- 復旧: 記録漏れ発覚時は法務との相談で事後記録と改善計画を個人情報保護委員会に自主申告

**検証方法**

- Phase 1b で第三者提供シナリオの E2E テスト

### SEC-PRV-007: 仮名加工情報 / 匿名加工情報の提供

- 優先度: SHOULD（分析用途の活用ニーズ対応。Phase 2 本格化）
- Phase: Phase 2
- 関連: `SEC-DAT-004`、`SEC-DAT-005`

現状、分析・AI 学習のために PII を含むデータを活用したいが、第 41・42 条の厳格な加工基準を満たす運用は実装コストが大きい。

本要件が満たされた世界では、tier1 `k1s0.Privacy.Pseudonymize` / `Anonymize` が個人情報保護法施行規則 第 31・34 条の加工基準を実装し、安全な分析データセットを提供する。再識別防止のために k-匿名性 / l-多様性チェックを自動実施する。

崩れた場合、分析用途の法令対応が各 tier3 で独自実装され、再識別化リスクが属人化する。

**受け入れ基準**

- 検知: 加工データから再識別可能性を定量評価（k-匿名性 k ≥ 5 目標）
- 防御: 加工 API の出力に再識別防止マーキングを付与
- 防御: 加工データと元データの対応表（仮名加工情報の場合）は物理的に隔離保管
- 復旧: 再識別化事故発覚時の即時提供停止と通知

**検証方法**

- Phase 2 に向けて Phase 1c で個人情報保護委員会ガイドラインを踏まえた加工仕様を策定
