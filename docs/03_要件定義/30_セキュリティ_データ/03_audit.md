# SEC-AUD: 監査ログ要件

本ファイルは、k1s0 プラットフォームの**監査ログ**——すなわち「誰が・いつ・何に・何をしたか」を事後に改ざん不能な形で再構成可能にする記録基盤——を定義する。監査ログは他のセキュリティ統制の**最後の砦**である。IAM が破られ暗号鍵が漏洩しても、改ざん不能な監査ログが残っていれば、被害範囲特定・法的責任追及・改正個人情報保護法 第 26 条の漏えい等報告が実施可能となる。

リスク台帳の `COM-RSK-007`（監査ログ改ざん検知の実装誤り）が High リスク指定されているとおり、本要件の未達は「監査ログがある」状態を装いながら実は機能していない、という最悪のシナリオを生む。したがって本ファイルでは、ハッシュチェーン（Merkle tree based append-only）の実装仕様、WORM ストレージの要件、7 年保持、外部監査への CSV 搬出、COM-RSK-007 の軽減策を具体化する。

本要件の実装は tier1 自作領域の Rust `crypto` モジュールで行う。Rust による実装は型安全性とメモリ安全性でハッシュチェーンの整合性を機械的に保証する。監査ログは改正個人情報保護法・電子帳簿保存法・金融商品取引法・FISC 安対基準の 4 法域が同時に要求する記録であり、要件定義上は「どの法令でも満たせる最も厳しい仕様」を採る。

## 要件本体

### SEC-AUD-001: 監査対象イベントの 4 分類を定義する

- 優先度: MUST（何を監査するかが定義されないと監査基盤自体が空転）
- Phase: Phase 1a
- 関連: `COM-GLO-008`、`SEC-IAM-005`、`SEC-PRV-*`

現状、「監査ログ」の範囲が曖昧で、アプリケーションログとエラーログまで混在し、数 TB のログから実際の監査対象を抽出する工数が発生する。

本要件が満たされた世界では、監査対象は以下 4 分類に明確化され、各分類ごとの Protobuf スキーマが tier1 で定義される。これ以外のログ（DEBUG / INFO）は別経路（Loki）に送られる。

- **ユーザー操作イベント**: ログイン・ログアウト・MFA challenge・セッション失効（`SEC-IAM-006/007` と連動）
- **権限変更イベント**: ロール付与・剥奪・特権昇格・承認ワークフロー結果（`SEC-IAM-005` と連動）
- **データアクセスイベント**: PII カラム読取・テナント跨ぎアクセス試行・機微データ Export（`SEC-PRV-*`・`SEC-DAT-*` と連動）
- **設定変更イベント**: NetworkPolicy 変更・RBAC 変更・Keycloak Realm 設定変更・OpenBao Policy 変更（`SEC-SEC-004`・`SEC-KMS-*` と連動）

崩れた場合、監査人から「監査対象が不明確」と指摘を受け、電子帳簿保存法 施行規則 第 3 条（真実性確保）の適合説明が成立しない。

**受け入れ基準**

- 検知: 4 分類のイベントは専用 Kafka topic (`audit.user` / `audit.auth` / `audit.data` / `audit.config`) に分離
- 防御: Protobuf スキーマが固定され、後方互換を破る変更は CI 拒否
- 防御: 監査対象 API は tier1 で宣言的マーキング（`#[audit(category="data")]`）し、アノテーション欠落は CI 検出
- 復旧: イベント欠落時の再送機構（Kafka の at-least-once）、欠番検知は SEC-AUD-002 で担保

**検証方法**

- Phase 1a で 4 分類の最小実装 POC、シーケンステストで全イベントが記録されることを確認

### SEC-AUD-002: ハッシュチェーン（Merkle 構造）による改ざん検知

- 優先度: MUST（COM-RSK-007 指定の High リスク。未達で監査応答の信用が崩壊）
- Phase: Phase 1a
- 関連: `COM-RSK-007`、`SEC-KMS-002`

現状、アプリログに追記型テーブルやクラウド WORM バケットを使うだけでは、DB 管理者による改ざん（UPDATE / DELETE）は検知できない。「監査ログがあるが改ざんされた」状態が最悪のシナリオ。

本要件が満たされた世界では、各監査イベント (i) の SHA-256 ハッシュに直前イベント (i-1) のハッシュを連鎖させる append-only Merkle tree を tier1 Rust `crypto` が実装する。1 時間ごとにルートハッシュを Ed25519 で署名し、OpenBao Transit Engine 経由で外部 Timestamping Authority (TSA、RFC 3161) にも送信する。任意時点のイベント改ざんは、以降のチェーンが破綻することで機械的に検出できる。

崩れた場合（COM-RSK-007 が発動した場合）、改ざんが検知できない監査ログは実質的に証拠能力を失い、裁判・監査・漏えい報告で根拠として使えない。

**受け入れ基準**

- 検知: 日次バッチでチェーン整合性を全件検証、不整合は即時 Alert（PagerDuty）
- 防御: 書き込みは append-only、UPDATE / DELETE は DB 権限レベルで拒否（PostgreSQL ロールと Trigger で二重防御）
- 防御: ルートハッシュの署名鍵は OpenBao Transit Engine、tier1 が `/transit/sign` 経由で署名
- 防御: プロパティベーステスト（proptest crate）で不変条件（任意位置改ざんで検知）を網羅検証
- 復旧: 改ざん検知時は当該区間のイベントを TSA 記録と突き合わせて範囲特定、影響分析と通知

**検証方法**

- 外部セキュリティレビュー（Phase 1a 完了時、COM-RSK-007 予防策）
- Phase 1a で 100 万件の監査イベント投入後、ランダム位置で UPDATE を試み、日次検証で検知されることを確認

### SEC-AUD-003: WORM ストレージによる物理的改ざん防止

- 優先度: MUST（電子帳簿保存法 施行規則 第 3 条「真実性の確保」要件の直接対応）
- Phase: Phase 1a
- 関連: `SEC-AUD-002`、`SEC-RES-*`

現状、Ext4 や S3 標準バケットはファイル削除・上書きが OS / API レベルで可能で、特権ユーザー（root / AWS admin）の内部不正が抑止できない。

本要件が満たされた世界では、監査ログ（Kafka の過去データと PostgreSQL WORM テーブル）は以下の二層 WORM で保護される。

- 第一層: PostgreSQL の `REVOKE UPDATE, DELETE ON audit_events FROM PUBLIC` + `INSERT` Trigger で論理 WORM
- 第二層: オブジェクトストレージ（MinIO Object Lock / AWS S3 Object Lock compliance モード）で物理 WORM、保持期間中は root 権限でも削除不能

二層構成により、DB 管理者の不正・Cloud admin の不正のいずれも抑止される。

崩れた場合、電子帳簿保存法 施行規則 第 3 条の「訂正又は削除の履歴の確保」が満たされず、税務調査で帳簿としての証拠能力を失う。

**受け入れ基準**

- 検知: Object Lock 設定の自動検証（日次、Terraform drift）
- 防御: PostgreSQL の UPDATE / DELETE 権限は DBA を含む全ロールに付与しない（`pg_hba.conf` + `REVOKE`）
- 防御: Object Lock は compliance モード（retention 期間中は root でも解除不能）
- 復旧: Object Lock 保持期間満了後のデータ移管手順を Runbook 化（アーカイブストレージへ自動遷移）

**検証方法**

- DBA 権限で UPDATE を試み、拒否されることを確認
- S3 Object Lock の API で compliance モードの retention が 7 年に設定されていることを確認

### SEC-AUD-004: 監査ログは 7 年間保持する（法令下限 + 1 年）

- 優先度: MUST（法令下限 6 年 + 1 年の安全マージン）
- Phase: Phase 1a
- 関連: `SEC-RES-*`、`SEC-BCK-*`

現状、多くの JTC プロジェクトでは保持期間が曖昧で、法令要求（会社法 第 432 条の 10 年、法人税法 施行規則 第 26 条の 2 の 7 年、電子帳簿保存法 施行規則の 7 年）との整合がなされていない。

本要件が満たされた世界では、監査ログは以下の階層ストレージで 7 年保持される。アクセス頻度に応じてコスト最適化しつつ、検索性と保全性を両立する。

- 0〜3 ヶ月: Kafka + PostgreSQL（ホット層、即座にクエリ可能）
- 3 ヶ月〜2 年: Object Lock 付き S3 Standard（温層、分単位でアクセス可能）
- 2〜7 年: S3 Glacier Deep Archive 相当（冷層、数時間以内に復元）

法令別の最低保持期間（法人税法 7 年、電子帳簿保存法 7 年、金融商品取引法 第 24 条の 4 の 4 の 10 年）を踏まえ、**全社横断で 7 年を既定**とし、10 年要求は顧客別 ABAC で上書き可能にする。

崩れた場合、税務調査時に監査ログが提示できず、税法上の青色申告取消しの事態に発展する。

**受け入れ基準**

- 検知: 各階層の最古レコード日付を Grafana で日次可視化
- 防御: Object Lock の retention 期間が 7 年（2555 日）以上で自動設定
- 防御: 階層遷移 job の異常終了は PagerDuty 通知
- 復旧: 顧客から過去 7 年分のデータ開示要求に対し、Glacier からの復元を 24 時間以内に開始

**検証方法**

- 人工的に 8 年前の日付でレコードを投入し、削除ポリシーが発動することを確認
- Glacier Deep Archive からの復元時間計測を半期ごとに実施

### SEC-AUD-005: 外部監査への CSV / NDJSON 搬出インタフェース

- 優先度: MUST（監査人への提供能力がないと実質的に監査不能）
- Phase: Phase 1a
- 関連: `SEC-IAM-005`（搬出権限）、`SEC-AUD-002`（改ざん検証とセット）

現状、監査人からの「過去 6 ヶ月分の全ログイン履歴」のような依頼に対し、手動 SQL 抽出で数日を要する。

本要件が満たされた世界では、tier1 が `/audit/export` API を提供し、範囲指定（日時・イベント分類・ユーザー・テナント）で CSV / NDJSON を生成する。出力には区間のハッシュチェーン検証結果（Merkle root と署名）を同梱し、「搬出後の改ざんも検知可能」な状態で監査人に引き渡せる。

崩れた場合、監査対応に 1 人月を要し、監査期間が恒常的に延長される運用負荷が発生する。

**受け入れ基準**

- 検知: `/audit/export` の呼び出し自体が監査対象（メタ監査、`audit.config` topic）
- 防御: 搬出権限は `audit-exporter` ロール保持者のみ、特権承認ワークフロー経由
- 防御: 出力 CSV の末尾に Merkle root と Ed25519 署名を添付
- 復旧: 搬出失敗時の再実行が冪等（同じクエリで同じ結果）

**検証方法**

- 四半期監査 dry-run で `/audit/export` を経由した搬出を実施し、手動検証時間を計測

### SEC-AUD-006: 監査ログの可用性（書き込み側の劣化耐性）

- 優先度: MUST（書き込み欠落は監査対象行為自体の否認を招く）
- Phase: Phase 1a
- 関連: `QUA-AVL-*`、`SEC-AUD-001`

現状、監査ログの書き込みがアプリのクリティカルパスにあると、監査 DB 障害時にアプリ全体が停止するリスクと、非同期化によるイベント欠落リスクのトレードオフが発生する。

本要件が満たされた世界では、監査イベントは Kafka に at-least-once で書き込まれ、Kafka → PostgreSQL WORM のパイプラインで恒久化される。Kafka が一時的に不通の場合もアプリ側に 10 秒のローカルバッファ（WAL）を持ち、再接続時に再送する。

崩れた場合、「監査対象の操作をしたが記録がない」状態が発生し、改正個人情報保護法 第 26 条の漏えい等報告で被疑事案の把握が不能になる。

**受け入れ基準**

- 検知: Kafka consumer lag と PostgreSQL 投入率を Grafana で秒単位監視、欠落率 > 0.01% で Alert
- 防御: アプリの監査送信失敗時はローカル WAL に 10 秒保持、再接続後 1 分以内に再送
- 防御: Kafka の replication-factor ≥ 3、min.insync.replicas ≥ 2
- 復旧: パイプライン長期断絶（> 1 時間）時は WAL からの手動復旧手順を Runbook 化

**検証方法**

- Chaos Engineering で Kafka を強制停止し、WAL 経由で復旧することを Phase 1a 完了時に確認
