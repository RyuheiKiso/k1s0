# SEC-IAM: 認証・認可要件

本ファイルは、k1s0 プラットフォームの**認証（誰であるか）** と **認可（何ができるか）** の統制を定義する。IAM が崩れると監査（`03_audit.md`）の主体が特定できず、テナント分離（`QUA-TNT-*`）が意味を失い、法令適合（`09_compliance.md`）の「本人確認・アクセス制御」条項が空洞化する。IAM は他統制の前提を提供する最上流レイヤであり、要件の漏れは下流すべてに波及する。

JTC 環境の IAM は「**社内 IdP（Active Directory / Entra ID）との連携**」と「**特権操作の承認フロー**」の 2 点で一般 SaaS と異なる。前者は稟議段階で必須の要件であり、後者は内部統制報告（J-SOX、金融商品取引法 第 24 条の 4 の 4）の実効性確保に直結する。本ファイルはこれを Keycloak ベースで実装することを前提に、RBAC + ABAC の二軸、サービスアカウント自動ローテ、MFA、セッション管理の 7 要件を整理する。

受け入れ基準は全件、本カテゴリ標準の**検知 / 防御 / 復旧**の 3 軸で記述する。

## 要件本体

### SEC-IAM-001: Keycloak を認証基盤として全コンポーネント共通化

- 優先度: MUST（認証基盤の分散は監査応答で即座に指摘される）
- Phase: Phase 1a
- 関連: `COM-GLO-005`、`SEC-SEC-001`

現状、各 JTC 社ではサービスごとに独自の認証実装（Basic 認証・独自セッション Cookie・部分的 SSO）が乱立し、退職者アカウント削除漏れが数年単位で発生している。

本要件が満たされた世界では、Keycloak (25.x 系) が OIDC / OAuth2 / SAML の全プロトコルを束ね、tier1 / tier2 / tier3 / Backstage / Grafana / Harbor などの全コンポーネントが同一 IdP で SSO される。退職時の 1 アカウント削除が全サービスに即時反映される。

崩れた場合、認証基盤の散在により「退職者が数ヶ月間 Grafana にログインできていた」ような改正個人情報保護法 第 23 条違反事例（安全管理措置の不備）が発生する。

**受け入れ基準**

- 検知: Keycloak 連携していないサービスは Backstage Software Catalog に `auth: keycloak` タグ付けで列挙、欠落は日次レポート
- 防御: 全 Ingress Gateway が `EnvoyFilter` で Keycloak JWT 検証を強制、未認証は 401
- 防御: 新規サービス追加時は Keycloak クライアント登録の Terraform モジュールを必須化
- 復旧: Keycloak 障害時の緊急バイパス（break-glass）手順を Runbook 化、利用時は監査ログに強制記録

**検証方法**

- Phase 1a POC で JWT なしリクエストが全サービスで 401 になることを確認
- 四半期ごとに社内 IdP の退職者リストと Keycloak ユーザーを突き合わせ

### SEC-IAM-002: 認可は RBAC を基本、テナント/業務属性は ABAC で表現

- 優先度: MUST（テナント越権は契約違反に直結）
- Phase: Phase 1a
- 関連: `QUA-TNT-*`、`SEC-DAT-002`、`ARC-RUL-*`

現状、単純な RBAC（role = {admin, operator, viewer}）ではテナント属性やデータ分類を表現できず、1 つの admin ロールが全テナントの全データを触れてしまう運用に陥る。

本要件が満たされた世界では、RBAC で粗粒度の機能アクセス（例: "tier3-deploy"）を定義し、ABAC（属性ベース）で `tenant_id`・`data_classification`・`time_window` などの属性条件を ZEN Engine / OPA で評価する。「A テナントの admin は B テナントの設定を閲覧できない」が構造的に保証される。

崩れた場合、テナント越権による情報漏洩が発生し、JTC 顧客の SLA 契約違反と損害賠償請求の対象となる。

**受け入れ基準**

- 検知: 認可判定ログが `SEC-AUD-*` に記録され、deny イベントが週次集計される
- 防御: 全 API コールで `subject × action × resource × environment` の 4 要素が評価される
- 防御: テナント跨ぎアクセスの試行は deny かつ異常アラートを発行
- 復旧: 認可ルール誤設定で業務停止した場合、前日 YAML へのロールバックが 10 分以内

**検証方法**

- テナント越権テストケース（matrix）を CI で自動実行
- 認可判定時間の p99 を QUA-PRF と整合させ、100ms 以内を保証

### SEC-IAM-003: サービスアカウント認証情報の自動ローテーション

- 優先度: MUST（長寿命 API キーは侵害時の被害が線形に拡大）
- Phase: Phase 1a
- 関連: `SEC-SEC-005`、`SEC-KMS-003`

現状、CI/CD やサービス間通信で使う API キーやサービストークンは作成時の静的文字列が数年間放置される運用が一般的で、1 度漏洩すると長期間悪用される。

本要件が満たされた世界では、全サービスアカウントトークンは OpenBao の dynamic credentials / Kubernetes ServiceAccount Token Volume Projection で短命化され、最大 24 時間で失効する。mTLS 証明書（ztunnel）と整合的に運用される。

崩れた場合、過去に漏洩した API キーで数ヶ月後に侵入が発生し、検知が極端に遅れる。

**受け入れ基準**

- 検知: 90 日超え未ローテのアカウントを検出する日次 job
- 防御: 静的長寿命トークンは Kyverno で作成ブロック、動的 24 時間以内のみ許可
- 防御: Workload Identity（Cloud 連携時は ESO + OIDC）で「鍵を持たないアプリ」を実現
- 復旧: 漏洩検知時、該当アカウントの全トークンを OpenBao 側で 5 分以内に revoke

**検証方法**

- 静的 token を持つ ServiceAccount を定義してデプロイ、admission で拒否されることを確認

### SEC-IAM-004: 個人アカウントは社内 IdP（AD / Entra ID）と連携

- 優先度: MUST（社内 HR 連動がないと退職者のアカウント残置が構造的に発生）
- Phase: Phase 1a
- 関連: `SEC-IAM-001`、`BIZ-ONB-*`

現状、多くの JTC では Keycloak 単独でユーザーを管理すると、AD 側の退職処理と Keycloak 側のアカウント無効化がタイムラグで乖離する。最悪、退職者が数ヶ月アクセス可能なままとなる。

本要件が満たされた世界では、Keycloak は AD / Entra ID の LDAP Federation または SAML IdP として連携し、AD 側の「アカウント無効化」が即時反映される。新規入社も AD 側の登録だけで完了する。

崩れた場合、退職者アクセスによる内部情報流出が発生する。個人情報保護委員会ガイドライン「漏えい等の対応」の事案に該当する。

**受け入れ基準**

- 検知: AD と Keycloak のユーザー差分を 1 時間ごとに sync し、差分 > 0 は監査ログに記録
- 防御: AD 側無効化から Keycloak セッション失効まで最大 5 分（業界平均 24 時間を大幅短縮）
- 防御: Keycloak 側の単独ユーザー作成は緊急用途のみ許可（承認ワークフロー経由）
- 復旧: IdP 障害時は Keycloak ローカル認証への切替（break-glass）手順を Runbook 化

**検証方法**

- 四半期ごとに退職者リストと Keycloak 有効アカウントを突合

### SEC-IAM-005: 特権アカウント操作の事前承認ワークフロー

- 優先度: MUST（内部不正防止の要、J-SOX 監査の必須項目）
- Phase: Phase 1a
- 関連: `SEC-AUD-001`、`BIZ-GOV-*`

現状、DB への直接クエリや本番 kubectl exec などの特権操作が監査なしに実行される運用があり、J-SOX 監査人から「職務分掌の不備」と指摘される典型例となっている。

本要件が満たされた世界では、特権ロール（`cluster-admin`、`db-admin`、`keycloak-admin` など）への昇格は Teleport / Backstage の承認ワークフローを経由し、承認者が 2 名以上、有効時間は最大 4 時間、全操作が監査ログに記録される。

崩れた場合、内部不正による業務データ改ざんや情報漏洩の際、行為者の特定と責任追及ができない。金融商品取引法 第 24 条の 4 の 4（内部統制報告）の適合が否定される。

**受け入れ基準**

- 検知: 特権操作の全ログが `SEC-AUD-*` に記録される
- 防御: 承認なしの特権操作は admission / RBAC で拒否
- 防御: 承認は 2 名以上、有効期限は最大 4 時間、事後レビュー必須
- 復旧: 不正操作検知時、特権トークンを即時 revoke する手順を Runbook 化

**検証方法**

- 四半期に 1 回、すべての特権操作ログを経営層へレポート

### SEC-IAM-006: 多要素認証（MFA）を全個人アカウントで必須化

- 優先度: MUST（ID/PW 単独は改正個人情報保護法 ガイドライン「安全管理措置」で不十分とされる）
- Phase: Phase 1a
- 関連: `SEC-IAM-001`、`SEC-IAM-004`

現状、単純な ID/PW 認証では fishing 攻撃や credential stuffing への耐性がない。Verizon DBIR 2024 によれば侵害の 68% が human element を含む。

本要件が満たされた世界では、Keycloak の Authentication Flow で TOTP (RFC 6238) または WebAuthn (FIDO2) を必須にし、MFA 未設定のアカウントはログインできない。AD 連携ユーザーも Keycloak 側で MFA challenge を通過する。

崩れた場合、ID/PW 漏洩時の被害が MFA ありと比較して桁違いに拡大する。

**受け入れ基準**

- 検知: MFA 未設定アカウントの日次レポート
- 防御: 初回ログイン時に MFA 登録を強制、未登録では業務操作不可
- 防御: WebAuthn（物理キー / Passkey）を推奨、TOTP は互換用途のみ
- 復旧: MFA デバイス紛失時は本人確認後の再登録フローを定義（48 時間以内）

**検証方法**

- 新規ユーザー作成直後に MFA 未設定のまま業務 API を呼び、拒否されることを確認

### SEC-IAM-007: セッション管理（有効期限 / 同時セッション数）

- 優先度: MUST（長期セッションはトークン窃取時の被害を拡大）
- Phase: Phase 1a
- 関連: `SEC-IAM-001`、`SEC-IAM-006`

現状、Web アプリの認証セッションが数日〜数週間有効のままで、端末盗難・共用端末での閉じ忘れによる不正利用リスクが残る。

本要件が満たされた世界では、Access Token は 15 分、Refresh Token は 8 時間（業務日 1 日）、Idle Timeout は 30 分に設定され、特権ロール保持時はより短くなる。同一ユーザーの同時セッションは最大 3 端末に制限される。

崩れた場合、盗難端末からの業務アクセスが長時間継続し、データ流出が拡大する。

**受け入れ基準**

- 検知: 異常な同時セッション数（> 3）を検知し、ユーザーに通知
- 防御: Access Token 15 分、Refresh Token 8 時間、Idle Timeout 30 分（業界平均 30 日 → 8 時間に短縮）
- 防御: 特権ロール時は Access Token 5 分、Refresh Token 1 時間
- 復旧: セッション即時失効 API（`/logout/all`）を管理者が呼び出し可能

**検証方法**

- Keycloak Realm 設定の CI 検証（Terraform drift 検知）
