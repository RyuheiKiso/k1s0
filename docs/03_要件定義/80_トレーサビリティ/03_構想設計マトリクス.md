# 03. 構想設計マトリクス

本書は要件定義と構想設計（[../../02_構想設計/](../../02_構想設計/)）の対応関係を明示する。要件が構想設計のどのコンポーネント・ADR・採用 OSS に結実しているかを索引化し、要件変更時の構想設計影響、構想設計変更時の要件影響を双方向で追跡する。

## 本書の位置付け

要件が追加されたら必ず該当する構想設計要素（コンポーネント／OSS／ADR）を更新する必要がある。本書は両者の対応を機械的に示し、PR 時のチェックリストとして活用する。

## 構想設計の主要要素

構想設計は以下の区分で技術判断を整理する。

- **アーキテクチャ**（[../../02_構想設計/01_アーキテクチャ/](../../02_構想設計/01_アーキテクチャ/)）: 全体構造、レイヤ図、Istio Ambient
- **tier1 設計**（[../../02_構想設計/02_tier1設計/](../../02_構想設計/02_tier1設計/)）: Dapr ファサード層、Rust 自作領域、内部通信
- **技術選定**（[../../02_構想設計/03_技術選定/](../../02_構想設計/03_技術選定/)）: OSS 選定根拠、比較検討
- **CICD と配信**（[../../02_構想設計/04_CICDと配信/](../../02_構想設計/04_CICDと配信/)）: GitOps、Progressive Delivery
- **法務とコンプライアンス**（[../../02_構想設計/05_法務とコンプライアンス/](../../02_構想設計/05_法務とコンプライアンス/)）: OSS ライセンス、AGPL 分離

### tier1 内部コンポーネント（COMP-T1-\*）

#### Pod 化しない責務（Log / Telemetry）

Log API（FR-T1-LOG-001〜004）と Telemetry API（FR-T1-TELEMETRY-001〜004）は tier1 SDK 内での構造化ログ生成・OTel span 生成に帰責し、配送はインフラ層の OpenTelemetry Collector DaemonSet に委譲する。これは NFR-B-PERF-006（計装オーバヘッド < 10ms）を守るための帰結で、tier1 アプリ層に中継 Pod を挟まない。観測基盤（ADR-OBS-001〜002 Grafana LGTM）が受け皿となる。

## 要件 → 構想設計要素（逆方向）

特定の要件が影響する構想設計要素を辿る場合は、本書を逆引きする。重要要件のみ以下に示す。

- **NFR-B-PERF-001（p99 < 500ms）** → Istio Ambient、Protobuf gRPC、Valkey キャッシュ、接続プール、HPA
- **NFR-A-CONT-001（対外 SLA 99%）** → MetalLB、Longhorn、HPA、Keycloak 冗長化
- **NFR-A-DR-001（RTO 4h）** → CloudNativePG 復旧手順、MinIO バックアップ、Velero 全体バックアップ、Runbook
- **NFR-A-DR-002（RPO 数秒）** → CloudNativePG WAL アーカイブ、MinIO
- **NFR-E-AC-001〜005（認証・認可）** → Keycloak、Istio JWT、RBAC、Kyverno、SPIFFE/SPIRE
- **NFR-G-ENC-001〜003（暗号化）** → OpenBao、cert-manager、FDE、Istio mTLS、pgcrypto
- **FR-T1-DECISION-001〜004（Decision API）** → ZEN Engine、JDM エディタ、Decision gRPC、キャッシュ

## メンテナンス

構想設計の ADR が追加・変更・Deprecated された際は本書を同時に更新する。PR で ADR 変更がある場合、本書への反映を PR テンプレートのチェックリストに含める。四半期ごとに Product Council で対応関係の整合性をレビュー。
