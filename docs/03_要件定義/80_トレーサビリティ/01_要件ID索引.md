# 01. 要件 ID 索引

本書は k1s0 要件定義書に登録された全要件 ID を集約した索引である。要件数の集計、カテゴリ別・Phase 別・優先度別の分布、および要件 ID から該当ドキュメントへの逆引きを提供する。

## 本書の位置付け

要件が 300 以上に達すると、重複登録や ID 衝突、Phase スコープ漏れが人手では追えなくなる。本索引は全要件を単一リストに集約することで、カテゴリ横断の検索、Phase 1b スコープの要件だけを抽出、MUST/SHOULD の集計など、フィルタ操作の起点となる。

## 要件 ID プレフィックス体系

要件は以下のプレフィックスで分類する。

- **BR-**: 業務要件（Business Requirement）
- **FR-**: 機能要件（Functional Requirement）
- **NFR-**: 非機能要件（Non-Functional Requirement）、A〜H の大項目
- **OR-**: 運用ライフサイクル要件（Operations）
- **DX-**: 開発者体験要件（Developer Experience）
- **BC-**: 事業契約要件（Business Contract）
- **P-**: 前提（Prerequisites）※要件ではなく所与条件
- **C-**: 制約（Constraints）※要件ではなく所与条件
- **RISK-**: リスクレジスタ項目

## カテゴリ別要件件数（集計）

Phase A 完了時点の要件件数を以下に示す。

- **業務要件（BR-）**: 10_業務要件 の全 BR 項目 — 初期版約 15 件
- **機能要件（FR-）**: 20_機能要件 — tier1 11 API × 平均 4〜5 要件 = 約 50 件、外部連携 約 6 件、情報要件 約 5 件
- **非機能要件（NFR-A〜H）**:
  - A 可用性: 約 15 件
  - B 性能・拡張性: 約 12 件
  - C 運用保守性: 約 18 件
  - D 移行性: 11 件（NFR-D-TIM/MTH/OBJ/PLN）
  - E セキュリティ: 約 25 件
  - F システム環境: 13 件（NFR-F-SYS/CHR/STD/FAC/ECO）
  - G データ保護: 15 件（NFR-G-CLS/ENC/AC/RES/LIF/PRV/DES）
  - H 完整性・コンプライアンス: 12 件（NFR-H-INT/KEY/COMP/AUD/TRN）
- **運用要件（OR-）**: 40_運用ライフサイクル — 約 30 件（INC/SUP/ENV/EOL/EXIT）
- **開発者体験（DX-）**: 50_開発者体験 — 約 35 件（TEST/GP/MET/FM/LD）
- **事業契約（BC-）**: 60_事業契約 — 約 80 件（UX/I18N/ONB/TRN/COST/BIL/LIC/LGL/SC/GOV/AI/ANA/SUS）

合計約 350 要件規模。実際の件数は各ドキュメントのサマリ表を集計して得る。

## 優先度分布（目安）

プラットフォームの性格上、MUST の比率が高くなる。

- **MUST**: 約 60%（法令遵守、稟議約束 SLO、運用継続性）
- **SHOULD**: 約 30%（品質向上、効率化、標準整合）
- **COULD**: 約 8%（将来拡張、高度な最適化）
- **WON'T**: 約 2%（明示的に対象外）

## Phase 別スコープ（目安）

- **Phase 1a（MVP-0）**: 全要件の約 15%（基礎要件、ADR、テンプレート）
- **Phase 1b（MVP-1a）**: 約 35%（パイロット要件、tier1 API 公開 11 本、基本運用）
- **Phase 1c（MVP-1b）**: 約 25%（運用体制確立、DevEx、監査対応）
- **Phase 2**: 約 15%（テナント拡大、A/B、カオス、FinOps）
- **Phase 3+**: 約 10%（マルチクラスタ、DR、ESG）

## 索引メンテナンス

要件の追加・変更・削除時は本索引の更新を PR に含める。索引が最新でなければ要件 ID の参照整合性が崩れる。四半期ごとに Product Council で索引のレビューを実施、重複・欠落・矛盾を解消する。
