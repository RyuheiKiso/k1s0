# 70. プロジェクト管理

本カテゴリは k1s0 開発プロジェクトの遂行に関わる計画・体制・品質保証・移行計画を定める。要件定義書（10〜60）は「何を作るか」を規定するのに対し、本カテゴリは「どう作るか・誰が作るか・どう品質を担保するか・どう本番に載せるか」を規定する。

## ドキュメント構成

| ファイル | 内容 |
| --- | --- |
| `01_体制と役割.md` | 役職・部門ごとの役割、RACI、意思決定体制 |
| `02_WBSと工程表.md` | OSS リリースまでの WBS と採用側運用工程の参考モデル、依存関係 |
| `03_QA計画.md` | 品質保証の方針、定量指標、レビュー・検査体制 |
| `04_テスト計画.md` | テスト種別、カバレッジ基準、環境、自動化戦略 |
| `05_移行計画.md` | レガシーからの段階移行、切替手順、並行運用、撤退 |

## なぜ独立カテゴリか

共通フレーム 2013 では「プロジェクト管理プロセス」「テストプロセス」が要件定義とは別の生存期間（ライフサイクル）で定義される。本カテゴリはそれに倣い、要件定義書の一部でありながら、後続の設計・実装でも参照され続けるドキュメント群として分離する。k1s0 を採用側組織が取り込んだ直後、即座に本カテゴリを起点にプロジェクトが走り始めるため、他の要件カテゴリよりも更新頻度が高い。

## 運用ルール

- 体制変更・工程変更は PR ベースで変更履歴を残す
- QA / テスト計画の指標（カバレッジ、欠陥密度等）は四半期レビューで更新
- 移行計画は Wave 進行に合わせて段階的に具体化する
