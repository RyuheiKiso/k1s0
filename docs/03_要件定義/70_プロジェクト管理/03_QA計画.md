# QA（品質保証）計画

本書は k1s0 プロジェクトの品質保証方針、定量指標、レビュー・検査体制を定める。テスト実施計画は [04_テスト計画.md](04_テスト計画.md) で別途詳述し、本書はそれを包含するガバナンス層を規定する。

## QA の思想

QA は「出荷直前の最終検査」ではなく、**継続的品質ガバナンス**（Continuous Quality Assurance）として設計する。内製 PaaS は毎日何度もデプロイされ、品質問題は即座に全利用部門に波及する。従って、品質確保は「特定の時点でのチェックポイント」ではなく「毎コミットごとの自動検査 + 継続的指標監視 + 四半期自己評価」の 3 層で構成する。

## 品質目標と定量指標

プロジェクト全体の品質目標は以下の 4 軸で定量化する。

### 1. 欠陥密度

- 基準値: 1 KLOC あたり 0.5 件以下（業界平均 1〜2 件）
- 測定点: 本番環境で検出された Severity 2 以上の欠陥数 / 該当リリースの追加コード行数
- 改善トリガー: 四半期で基準値を超えた場合、開発プロセス（レビュー密度、テスト密度）を見直し

### 2. DORA Four Keys（採用後の運用拡大時）

- **Deployment Frequency**: 週次 10+ 回を目標（Elite 水準）
- **Lead Time for Changes**: PR マージから本番反映まで 24 時間以内（High 水準）
- **Change Failure Rate**: 15% 以下（High 水準）
- **Mean Time to Restore**: 1 時間以内（High 水準）

DORA メトリクスは Grafana で自動集計、四半期レビューで公開。

### 3. テストカバレッジ

- tier1 コアコード: ラインカバレッジ 80% 以上
- tier1 致命的モジュール（crypto、PII マスキング、認可判定）: 95% 以上
- tier2/tier3: ラインカバレッジ 70% 以上（過度なカバレッジ強制は避け、意味あるテストを優先）

### 4. 観測性充足度

- 全公開 API に SLI 定義、SLO 設定、エラーバジェット運用が存在すること
- 欠落は四半期で棚卸し、当期で補完

## 品質保証の 3 層

### L1: 自動検査（毎コミット）

- CI パイプライン（Argo CD と連携）で以下を強制
  - 単体テスト、契約テスト（Pact）、統合テスト
  - Lint、フォーマット、static analysis（Rust clippy、golangci-lint、eslint）
  - セキュリティスキャン（Trivy、Snyk）
  - ライセンスチェック（OSS ライセンス分類、GPL 系の非同梱検証）
  - SBOM 生成、Sigstore 署名
  - Pact 契約テスト（Consumer/Provider）
  - Replay テスト（Temporal Workflow）
  - ゴールデンケース（ZEN Decision）
- いずれか失敗で PR マージブロック

### L2: レビューと人為検査（PR 単位）

- レビュアー必須: tier1 は Platform Tech Lead + 1 名、tier2/tier3 は部門 Lead + DevEx 1 名
- レビュー観点チェックリスト（[docs/00_format/review_checklist.md](../../00_format/review_checklist.md) 予定）
- ADR 変更は Technical Lead 承認必須、Product Council に 共有
- セキュリティ影響ある変更はセキュリティチーム承認

### L3: 定期的自己評価と監査（四半期・半期）

- 四半期ごとの QA レポート: 欠陥密度、DORA、SLO 達成率、脅威モデル更新
- 半期ごとの内部監査: Audit ログ確認、Runbook 鮮度、OSS ライセンス遵守
- 年次外部ペネトレーションテスト
- 年次 ISO 27001 審査（採用後の運用拡大時取得予定）

## レビュー・検査の役割分担

L1〜L3 の 3 層で検査を重ねることで、「機械で弾ける欠陥は機械に弾かせ、判断が要るものは人が見る」を徹底する。表は役割分担の索引であり、各行の実体（何を合格とするか・誰が承認するか）は下の散文で展開する。

- **コード品質**: L1 は Lint（golangci-lint / clippy / ESLint）、静的解析（gosec / cargo-audit）、カバレッジ閾値（tier1 コアは 80% 以上）を CI で強制。L2 は PR レビューで設計の妥当性を確認。L3 は四半期ごとに欠陥密度（KLoC あたり）を Backstage で集計してトレンドを読み、2 四半期連続で悪化していれば QA 計画を改訂する
- **セキュリティ**: L1 は Trivy / Snyk によるイメージスキャンと Sigstore 署名検証を Kyverno で強制。L2 はセキュリティチームが新規 OSS 採用時と本番変更時の承認レーンに入る。L3 は年次ペンテストの指摘事項のうち High 以上を次年度までに 100% 是正する
- **契約（API）**: L1 は Pact Consumer/Provider テストで IDL 互換性を確認。L2 は IDL の v1 → v1.N 追加は Tech Lead 承認、v2 破壊変更は Product Council 承認。L3 は四半期の破壊変更履歴をレビューし、DEPRECATED から削除までの最短 2 四半期ルール（00_tier1_API共通規約）の遵守を点検する
- **観測性**: L1 は SLO アラートルールの syntax / 範囲を CI で検証。L2 は Runbook 作成確認を本番デプロイ前チェックリストに組み込む。L3 は四半期の SLO / エラーバジェット会議でバーンレートと信頼性投資のバランスを決裁する
- **運用**: L1 は対象外（自動化の効く領域が薄い）。L2 は本番デプロイ承認（Argo CD Sync の人手承認）。L3 は月次ポストモーテム集約で、インシデントの根本原因を FMEA（OR-FMEA-004）に反映する
- **コンプライアンス**: L1 はライセンス自動検出（ScanCode / FOSSology）で禁止ライセンス混入を PR ブロック。L2 は新規 OSS 採用時の法務確認、特に AGPL / SSPL / BUSL は個別承認経路に乗せる。L3 は半期監査で Audit ログ鮮度・Runbook 鮮度・OSS ライセンス遵守状況を合議で点検し、個人情報保護法改正への追随を確認する

## ゴー / ノーゴー判定基準

OSS リリース判定および採用側組織での段階ゲート（リリース時点 / 採用後の運用拡大時 / 採用側のマルチクラスタ移行時）や本番リリース判定には、以下の客観基準を満たしていることを条件とする。

- Critical / High Severity 欠陥ゼロ
- L1 CI パイプライン全段階グリーン
- SLO 達成率 30 日間 99.5% 以上（採用後の運用拡大時）
- セキュリティスキャンで Critical CVE ゼロ
- 監査・法務・セキュリティの明示的承認
- Runbook 整備完了（該当コンポーネント）
- 負荷試験合格（想定トラフィックの 1.5 倍で SLO 維持）

いずれかを満たさない場合は Product Council への説明と条件付き承認が必要。

## 不具合管理

- Severity は S1 / S2 / S3 / S4 の 4 段階（OR-INC-001 参照）
- 全不具合は Backstage の Issue Tracking Plugin で追跡、Git Repo と同期
- S1/S2 は即時対応、S3 は次スプリント、S4 は四半期見直し
- 同じ根本原因が 3 回以上再発した場合は Rootcause Re-design として優先度昇格

## 品質文化の醸成

定量指標だけでは品質は確保できない。以下の文化的取り組みを並行する。

- **Blameless Postmortem**: インシデント後は個人責任を問わない文化を徹底、学びを組織資産化
- **Quality Advocate 制度**: 各 Stream-aligned Team から QA 代表を選出、定期横断会議
- **Dogfooding**: Platform Team 自身が自分たちの作る PaaS を使う（Backstage は Platform Team が最大の利用者）
- **Tech Debt Budget**: 各 Sprint の 20% を技術的負債返済に充当、バックログで可視化
