# 体制と役割

本書は k1s0 プロジェクトの組織体制、役割定義、意思決定体制、RACI を定める。人の配置が要件・品質・移行の成否を直接左右するため、要件定義書の一部として明文化する。

## 体制の設計思想

k1s0 は「内製 PaaS」であり、単なる SI 案件ではなく **恒久運用プロダクト** として扱う。SI 的プロジェクト体制（発注者・PMO・ベンダー）ではなく、**プロダクト開発体制**（Product Council、プロダクトオーナー、プラットフォームチーム、利用部門）を採用する。これは Team Topologies の "Platform Team + Stream-aligned Team" パターンに準拠する。

利用部門（tier2/tier3 開発者）は k1s0 プラットフォームの「顧客」であり、Platform Team は内部サービスプロバイダとしてふるまう。この構造により、Platform Team が自己都合で機能を押し付ける失敗パターンを回避する。

## 組織階層

プロジェクト体制は以下の 5 層構造。上位から下位にガバナンス流れ、下位から上位にフィードバック流れる。

### L0: Product Council（最上位意思決定体）

- 構成: 情報システム本部長、経営企画室長、経理部長、情報セキュリティ委員長、外部諮問委員 1〜2 名
- 頻度: 月次
- 役割: 予算承認、Phase ゲート判定、外部調達承認、重大インシデント最終責任、法令リスクの最終判断

### L1: プロダクトオーナー / テクニカルリード

- プロダクトオーナー: 全体のプロダクトビジョン責任、Product Council への説明責任、ロードマップ
- テクニカルリード: 技術選定の最終判断、アーキテクチャ、ADR 承認、技術的負債の優先順位

### L2: Platform Team（k1s0 の作り手）

- サブチーム構成:
  - **tier1 コアチーム**: Dapr ファサード、Rust 自作領域、Protobuf IDL 保守（目安 6〜8 名）
  - **インフラ / SRE**: Kubernetes、Istio Ambient、観測性、セキュリティ、SPIFFE/SPIRE、OpenBao（目安 4〜6 名）
  - **DevEx チーム**: Backstage、Golden Path、CI/CD、ドキュメント、テンプレート（目安 3〜4 名）
  - **データ基盤チーム**: CloudNativePG、Kafka、MinIO、Valkey、Longhorn（目安 3〜4 名）
- 各チームに Tech Lead 配置、週次クロスチーム同期

### L3: Stream-aligned Team（tier2/tier3 開発者 = 利用部門）

- 各業務部門から選出された開発者。k1s0 の顧客かつフィードバック元
- 専任ではなく部門業務との兼任が基本、各部門 2〜5 名

### L4: Ops / 監査 / 法務 / コンプライアンス

- SOC / オンコール: 24/7 一次対応
- 監査部門: 半期ごとに Audit ログとコンプライアンス証跡をレビュー
- 法務: OSS ライセンス、外部データ連携契約、個人情報保護法対応
- 財務: Phase 別コスト実績、予実管理

## RACI（主要意思決定）

以下は代表的な意思決定に対する RACI（Responsible / Accountable / Consulted / Informed）の割当。

| 意思決定 | Product Council | プロダクトオーナー | テクニカルリード | Platform Team | 利用部門 | 監査 / 法務 |
| --- | --- | --- | --- | --- | --- | --- |
| Phase ゲート承認 | A | R | C | C | I | C |
| 新規 ADR 採択 | I | C | A | R | C | I |
| tier1 API 新規追加 | I | A | R | R | C | I |
| 本番リリース（通常） | I | I | I | R/A | I | I |
| 本番リリース（重大変更） | A | R | C | R | C | C |
| インシデント Sev 1 対応 | I | A | C | R | I | I |
| 個人情報取扱い変更 | A | R | C | C | C | R |
| OSS ライセンス判断 | C | I | C | C | I | R/A |
| 予算超過承認 | R/A | R | I | I | I | I |
| レガシー撤退判断 | A | R | C | C | C | I |

R=実行責任 / A=最終責任 / C=相談 / I=共有

## 意思決定サイクル

- **日次**: チーム内 Stand-up、インシデント対応
- **週次**: Tech Lead 同期、PR レビュー締切
- **隔週**: Sprint Review、バックログ Refinement、Stream-aligned Team との定例
- **月次**: Product Council、予算レビュー、リスクレジスタ更新
- **四半期**: ロードマップ見直し、非機能指標レビュー、脅威モデリング見直し
- **半期**: 監査、コンプライアンス自己評価、Platform Team 組織レビュー

## 外部依存

- **OSS コミュニティ**: tier1 が依存する主要 OSS（Dapr、Istio、Temporal 等）の Issue/PR に Platform Team が参加し、Upstream 優先の方針を堅持
- **SIer / コンサル**: 初期 Phase 1 のみ限定的に支援（Platform Team 立上げ伴走、Kubernetes 運用研修）、Phase 2 以降は完全内製
- **ハードウェアベンダー**: Phase 1 物理環境調達、保守契約。ベンダーロックを避けるため抽象化層（CSI、CNI、CRI）越しにのみ依存

## スキルマップと育成

Phase 1 開始時点で、Platform Team 全体として以下のスキルを内製保有することを必須条件とする。

- Kubernetes 運用: 3 年以上経験者を最低 3 名
- Rust: 中級以上 2 名（Edition 2024 / async）
- Go: 中級以上 4 名
- SRE / 観測性: Prometheus / Grafana 運用経験者 2 名
- セキュリティ / PKI: SPIFFE / mTLS / OIDC 経験者 2 名

不足スキルは外部採用 + 既存メンバーの集中トレーニング（BC-TRN）で補填。Phase 2 開始時点でスキルマップを再評価し、ギャップが残る場合は Phase 移行判定で問題視する。
