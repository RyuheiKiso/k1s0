# 移行計画

本書は JTC 既存システム（.NET Framework 資産、主要業務システム）から k1s0 への段階移行計画を定める。ADR-MIG-001（サイドカー方式）、ADR-MIG-002（API Gateway 方式）で選定した技術手法を実際の業務システムに適用するための年度計画である。

## 移行計画の思想

レガシーマイグレーションは「一斉書換え」が最大の失敗要因。本計画は Martin Fowler の Strangler Fig Pattern に基づき、**段階的包み替え** で進める。旧系は即座に廃止せず、機能単位で k1s0 側に置き換え、旧系を逐次縮小する。2 年間の並行運用を前提とし、**撤退容易性**（Phase 移行失敗時に旧系に戻せること）を最大の原則とする。

## 移行対象の棚卸し

Phase 1 着手時に、JTC 内の業務システム全体を以下の観点で棚卸し、移行優先度を決める。

### 分類軸

- **業務クリティカル度**: S1（基幹・停止で全社影響）/ S2（部門基幹）/ S3（補助業務）/ S4（終了予定）
- **技術スタック**: .NET Framework / .NET Core+ / Java / COBOL / スクリプト言語 / パッケージ製品
- **結合度**: 他システムとの連携数、ETL / バッチ依存
- **データ量と更新頻度**: 業務データの規模、日次バッチの有無
- **撤退可能性**: 代替可能性（OSS、SaaS、k1s0 で再実装）
- **法的制約**: 監督官庁認可、契約上の変更不可

### 移行波次（Wave）分類

棚卸しの結果を以下の 4 Wave に分類する。

- **Wave 0**: 将来終了予定、移行対象外（3 年以内にサービス終了）
- **Wave 1（Phase 2 後半）**: 結合度低 + クリティカル度低 + 撤退容易の組合せ（パイロット候補）
- **Wave 2（Phase 3 前半）**: 結合度中 + クリティカル度中、移行ノウハウが Wave 1 で確立された後
- **Wave 3（Phase 3 後半以降）**: 基幹系、結合度高、法的制約あり。慎重に 1 件ずつ

## 移行パターンの選択基準

| 移行対象の特性 | 推奨パターン | ADR |
| --- | --- | --- |
| Kubernetes 化可能（Windows Container 化可）+ 機能単位で段階移行 | サイドカー方式 | ADR-MIG-001 |
| 外部公開 API の段階切替、クライアント透過 | API Gateway 方式 | ADR-MIG-002 |
| 事実上の再実装が可能 + データ移行が軽量 | 完全新規構築（旧系凍結） | — |
| 業務ロジックが軽く、データ駆動型 | Event Interception（CDC） | — |

基本は サイドカー + API Gateway の組合せ。完全新規構築は Wave 0 や Wave 1 の最小規模ケースに限る。

## パイロット計画（Phase 2 前半 = M10〜M14）

全面移行前に、必ず 2 件のパイロットを実行する。

### パイロット 1: 社内情報掲示板（Wave 1 相当）

- 目的: サイドカー方式の実運用検証
- 対象: .NET Framework Web Forms ベース、利用者 500 名
- 期間: 3 ヶ月（M10〜M12）
- 成功基準: 旧系完全撤退、k1s0 上で 30 日連続稼働、SLO 達成

### パイロット 2: 外部公開 API（Wave 1 相当）

- 目的: API Gateway 方式の実運用検証
- 対象: 取引先向け残高照会 API（旧系は Java、月間 100 万呼出）
- 期間: 4 ヶ月（M11〜M14）
- 成功基準: クライアント（取引先）に影響なく段階的 50% → 100% 切替、SLO 達成

パイロット完了後、以下を確立してから Wave 2 に進む。

- 挙動一致検証フレームワークの汎用化
- 切替手順 Runbook
- ロールバック Runbook
- 移行専門 SI パートナーとの連携体制

## Wave 2 本格移行（Phase 3 前半 = M19〜M24）

Wave 2 対象は 5〜10 システムを想定。各システムの移行は以下のステージゲートを満たす。

1. **対象分析**: 業務ヒアリング、データ構造整理、k1s0 要件マッピング（1 ヶ月）
2. **設計**: ADR-MIG-001/002 の適用方針決定、データ移行方式、切替 シナリオ（1 ヶ月）
3. **実装**: tier2 側 k1s0 コード実装、挙動一致検証、staging 動作確認（2〜4 ヶ月）
4. **並行運用**: 旧系 / 新系の両方稼働、差分モニタリング（1〜3 ヶ月）
5. **切替**: 切替日の決定、Runbook 実行、30 日監視期間
6. **撤退**: 旧系停止、ハードウェア / ライセンスの返却

## データ移行戦略

データ移行は k1s0 プロジェクト最大のリスク領域。以下の原則を守る。

### 原則 1: スキーマは原本準拠

- tier1 の共通データモデル（[20_機能要件/60_tier1_共通データモデル.md](../20_機能要件/60_tier1_共通データモデル.md)）は PaaS 層
- 業務データは tier2 側で独自スキーマを持つ。旧系スキーマの構造を尊重し、不要な再設計を避ける
- ただし PII の扱い、tenant_id 付与は k1s0 標準に合わせる

### 原則 2: データ移行は 3 段階

1. **初期バルクロード**: 旧系の全データを k1s0 にコピー（切替日の数日前、staging で検証）
2. **差分同期**: CDC（Debezium 等）で旧系変更を k1s0 に逐次反映（並行運用期間中）
3. **カットオーバー**: 旧系を停止、最終差分を反映、k1s0 を正系に昇格

### 原則 3: データ整合性検証

- 行数カウント、サム値（金額合計、件数）の一致確認
- サンプル検証（1,000 件程度を目視）
- 業務担当者の受入テスト合格必須

## 撤退戦略

全移行が成功する保証はない。各 Wave の各システムで、以下の条件いずれかが発生したら撤退判断する。

- 並行運用中の差分が 3% 以上発生
- 新系の SLO が 30 日間 95% を下回る
- パフォーマンスが旧系の 2 倍以上遅い
- セキュリティインシデント発生
- 業務担当者が受入拒否

撤退時は Runbook（[40_運用ライフサイクル/05_撤退戦略.md](../40_運用ライフサイクル/05_撤退戦略.md)）に従い、トラフィックを旧系に戻す。

## 並行運用期間

各システムは最低 1 ヶ月、クリティカル系は 3 ヶ月の並行運用を行う。並行運用中は以下を継続モニタリング。

- 業務的挙動差分（出力データ比較、承認結果、金額計算等）
- 技術的指標（レスポンスタイム、エラー率、リソース消費）
- 利用者フィードバック（業務部門）

## 法的・契約的配慮

- 監督官庁認可システムは、移行前に金融庁 / 経産省等への事前通知
- 外部取引先向け API の切替は契約変更手続き必須（場合により締結）
- 個人情報の移行は個情法の目的内であることを確認、必要なら同意取得
