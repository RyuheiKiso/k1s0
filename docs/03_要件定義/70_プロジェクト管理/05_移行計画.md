# 移行計画

本書は JTC 既存システム（.NET Framework 資産、主要業務システム）から k1s0 への段階移行計画を定める。ADR-MIG-001（サイドカー方式）、ADR-MIG-002（API Gateway 共存方式）で選定した技術手法を実際の業務システムに適用するための年度計画である。

## 本書の位置付け

移行計画は「何を作るか」ではなく「既存資産をどう段階的に置き換え、失敗時にどう戻すか」を規定する実行計画書である。30_非機能要件/D_移行性.md が「移行性の非機能品質」（NFR-D-TIM / MTH / OBJ / PLN）を定義するのに対し、本書は実際の年度別ロールアウト手順と撤退判断基準を具体化する。要件 ID は主要セクションにのみ付与（`MIG-<領域>-<通番>`）し、個別システムの移行タスクは本書ではなく WBS 側で管理する。

## MIG-STG-001: Strangler Fig 段階移行戦略

**現状**: レガシーマイグレーションは「一斉書換え」が最大の失敗要因として業界通説となっている。JTC 社内でも過去、基幹系の一斉刷新プロジェクトが複数回スケジュール破綻や予算超過を起こし、結果として旧系を延命する判断を強いられた実績がある。

**要件達成後**: 本計画は Martin Fowler の Strangler Fig Pattern に基づき、**段階的包み替え** で進める。旧系は即座に廃止せず、機能単位で k1s0 側に置き換え、旧系を逐次縮小する。2 年間の並行運用を前提とし、**撤退容易性**（Phase 移行失敗時に旧系に戻せること）を最大の原則とする。この原則は全サブ要件（MIG-WAVE、MIG-PATTERN、MIG-DATA、MIG-ROLLBACK）の設計制約として機能する。

**崩れた時**: 一斉書換えに回帰した場合、失敗時の影響範囲が全社規模に拡大し、旧系の運用ノウハウも早期に失われて復旧不能となる。JTC の過去 PJ と同様のスケジュール破綻リスクが顕在化する。

**受け入れ基準**:
- 全 Wave で旧系と新系の 並行運用期間を最低 1 か月以上確保
- 各 Wave の Runbook に撤退手順（MIG-ROLLBACK-001）を含む
- 一斉切替（Big Bang）は禁止、Product Council 承認なしには実施不可

**優先度**: MUST（失敗時の全社影響を防ぐ基本原則）

## MIG-INV-001: 移行対象の棚卸し

**現状**: JTC 内の業務システム全体像が最新化されておらず、部門ごとに保有リストが散在する。移行優先度を決める基礎情報が欠けている。

**要件達成後**: Phase 1 着手時に業務システム全体を棚卸しし、以下 6 軸で分類する。6 軸すべてで評価することで、クリティカル度だけに引きずられず、技術・データ・法的な移行難度も可視化する。

- **業務クリティカル度**: S1（基幹・停止で全社影響）/ S2（部門基幹）/ S3（補助業務）/ S4（終了予定）
- **技術スタック**: .NET Framework / .NET Core+ / Java / COBOL / スクリプト言語 / パッケージ製品
- **結合度**: 他システムとの連携数、ETL / バッチ依存
- **データ量と更新頻度**: 業務データの規模、日次バッチの有無
- **撤退可能性**: 代替可能性（OSS、SaaS、k1s0 で再実装）
- **法的制約**: 監督官庁認可、契約上の変更不可

棚卸し結果は Backstage の Service Catalog に登録し、四半期ごとに鮮度レビューする。

**崩れた時**: 棚卸しが個別判断に留まると、重複移行や移行忘れが発生し、Wave 計画の整合性が崩れる。

**受け入れ基準**:
- Phase 1a 完了時点で Wave 1 候補、Phase 1b 完了時点で Wave 2〜3 候補を確定
- 6 軸すべての値が Backstage 側で可視化されている
- 四半期ごとに棚卸し内容の差分レビューを実施

**優先度**: MUST（Wave 計画の前提条件）

## MIG-WAVE-001: 移行波次（Wave）の定義

**現状**: 移行順序が決まっていないと、ノウハウ蓄積が追いつかないまま基幹系に着手して失敗するリスクが高い。

**要件達成後**: 棚卸し（MIG-INV-001）の結果を以下 4 Wave に分類し、Wave 番号順に着手する。後続 Wave は前 Wave の成功基準達成を前提条件とする。

- **Wave 0**: 将来終了予定、移行対象外（3 年以内にサービス終了）
- **Wave 1（Phase 2 後半）**: 結合度低 + クリティカル度低 + 撤退容易の組合せ（パイロット候補）
- **Wave 2（Phase 3 前半）**: 結合度中 + クリティカル度中、移行ノウハウが Wave 1 で確立された後
- **Wave 3（Phase 3 後半以降）**: 基幹系、結合度高、法的制約あり。慎重に 1 件ずつ

**崩れた時**: Wave 順序を飛ばして基幹系から着手すると、移行ノウハウ不足のまま巨大システムに挑み、過去 PJ と同様のスケジュール破綻を繰り返す。

**受け入れ基準**:
- Wave 2 着手条件: Wave 1 で 2 件のパイロット（MIG-PILOT-001）が成功基準を満たしていること
- Wave 3 着手条件: Wave 2 で 5 件以上の Wave 2 対象システムが旧系撤退完了
- Wave スキップは Product Council 承認必須

**優先度**: MUST

## MIG-PATTERN-001: 移行パターンの選択基準

**現状**: 移行パターンが個別判断されると、同種システムでもアプローチが割れ、ノウハウが共有されない。

**要件達成後**: 対象システムの特性に応じて以下 4 パターンから選択する。基本はサイドカー（ADR-MIG-001）と API Gateway（ADR-MIG-002）の組合せで、両者が構想設計で既に ADR 化されている。完全新規構築と Event Interception（CDC）は例外パターンとして限定適用する。

パターン選択はシステムごとに設計書（Wave 2 ステージゲート 2 で作成）に記載し、選択理由を残す。

| 移行対象の特性 | 推奨パターン | 基盤 ADR | 適用例 |
| --- | --- | --- | --- |
| Kubernetes 化可能（Windows Container 化可）+ 機能単位で段階移行 | サイドカー方式 | ADR-MIG-001 | 社内情報掲示板、部門 Web アプリ |
| 外部公開 API の段階切替、クライアント透過 | API Gateway 方式 | ADR-MIG-002 | 取引先向け残高照会 API |
| 事実上の再実装が可能 + データ移行が軽量 | 完全新規構築（旧系凍結） | — | Wave 0 の小規模補助業務 |
| 業務ロジックが軽く、データ駆動型 | Event Interception（CDC） | — | レポーティング系、分析系 |

完全新規構築は Wave 0 や Wave 1 の最小規模ケースに限り、基幹系・外部 API 公開系では禁止する。CDC パターンは Debezium 等の OSS を使い、旧系への読み出し負荷が許容範囲内であることを事前検証する。

**崩れた時**: パターン選択が恣意的になると、同種システムで移行期間や成功率にバラつきが発生し、年度計画が読めなくなる。

**受け入れ基準**:
- Wave 2 以降のシステム移行設計書にパターン選択理由の記載必須
- 完全新規構築は Product Council 承認必須
- パターン別の移行期間実績を四半期ごとに集計、計画精度にフィードバック

**優先度**: MUST

## MIG-PILOT-001: パイロット計画

**現状**: サイドカー方式と API Gateway 方式は構想設計で技術検証済みだが、実運用レベルでの業務影響・切替手順・ロールバック手順が未検証。

**要件達成後**: 全面移行前（Wave 2 着手前）に 2 件のパイロットを Phase 2 前半（M10〜M14）で実行する。各パイロットはサイドカー方式・API Gateway 方式の代表例として位置付け、以降の Wave 2 展開の Runbook 雛形を獲得する。

- **パイロット 1: 社内情報掲示板（Wave 1 相当）**
  - 目的: サイドカー方式の実運用検証
  - 対象: .NET Framework Web Forms ベース、利用者 500 名
  - 期間: 3 か月（M10〜M12）
  - 成功基準: 旧系完全撤退、k1s0 上で 30 日連続稼働、SLO 達成
- **パイロット 2: 外部公開 API（Wave 1 相当）**
  - 目的: API Gateway 方式の実運用検証
  - 対象: 取引先向け残高照会 API（旧系は Java、月間 100 万呼出）
  - 期間: 4 か月（M11〜M14）
  - 成功基準: クライアント（取引先）に影響なく段階的 50% → 100% 切替、SLO 達成

パイロット完了後、以下を確立してから Wave 2 に進む。

- 挙動一致検証フレームワークの汎用化
- 切替手順 Runbook
- ロールバック Runbook
- 移行専門 SI パートナーとの連携体制

**崩れた時**: パイロット失敗や未実施で Wave 2 に進むと、基盤的な手順不備が Wave 2 全システムに波及し、移行全体が停滞する。

**受け入れ基準**:
- パイロット 2 件が成功基準をすべて達成
- Runbook（切替・ロールバック）が Git 管理、SRE 単独で実行可能
- ポストモーテムで学びを ADR-MIG-001/002 の改訂にフィードバック

**優先度**: MUST

## MIG-WAVE2-001: Wave 2 本格移行のステージゲート

**現状**: 移行プロジェクトは「何を持って次工程に進むか」が曖昧になりがちで、設計不十分のまま実装着手して手戻りが発生する。

**要件達成後**: Wave 2 対象（5〜10 システム想定）の各システムは以下 6 ステージを順守し、各ステージの成果物とゲート条件を明示する。

1. **対象分析**: 業務ヒアリング、データ構造整理、k1s0 要件マッピング（1 か月）
2. **設計**: ADR-MIG-001/002 の適用方針決定、データ移行方式、切替シナリオ（1 か月）
3. **実装**: tier2 側 k1s0 コード実装、挙動一致検証、staging 動作確認（2〜4 か月）
4. **並行運用**: 旧系 / 新系の両方稼働、差分モニタリング（1〜3 か月）
5. **切替**: 切替日の決定、Runbook 実行、30 日監視期間
6. **撤退**: 旧系停止、ハードウェア / ライセンスの返却

各ステージ完了は担当 PM + 構想設計リードのダブルサインオフで承認する。

**崩れた時**: ステージを飛ばした場合、並行運用不足による切替失敗、撤退判断の遅れによるコスト膨張、業務部門の受入拒否リスクが顕在化する。

**受け入れ基準**:
- 全 Wave 2 対象で 6 ステージを順守
- ステージ飛ばしは Product Council 承認必須
- 各ステージの成果物テンプレートを WBS 側で定義

**優先度**: MUST

## MIG-DATA-001: データ移行戦略

**現状**: データ移行は k1s0 プロジェクト最大のリスク領域。スキーマ差異、差分同期、整合性検証の 3 点でいずれかを誤ると、業務データの損失または二重管理状態が発生する。

**要件達成後**: 以下 3 原則を全データ移行で順守する。

### 原則 1: スキーマは原本準拠

- tier1 の共通データモデル（[20_機能要件/60_tier1_共通データモデル.md](../20_機能要件/60_tier1_共通データモデル.md)）は PaaS 層
- 業務データは tier2 側で独自スキーマを持つ。旧系スキーマの構造を尊重し、不要な再設計を避ける
- ただし PII の扱い、tenant_id 付与は k1s0 標準に合わせる

### 原則 2: データ移行は 3 段階

1. **初期バルクロード**: 旧系の全データを k1s0 にコピー（切替日の数日前、staging で検証）
2. **差分同期**: CDC（Debezium 等）で旧系変更を k1s0 に逐次反映（並行運用期間中）
3. **カットオーバー**: 旧系を停止、最終差分を反映、k1s0 を正系に昇格

### 原則 3: データ整合性検証

- 行数カウント、サム値（金額合計、件数）の一致確認
- サンプル検証（1,000 件程度を目視）
- 業務担当者の受入テスト合格必須

**崩れた時**: 3 原則のいずれかが崩れると、業務データの損失（受入テスト未達）、サイレント破損（整合性検証不足）、業務ロジック退行（スキーマ再設計の破綻）のいずれかが発生する。

**受け入れ基準**:
- 全移行対象で 3 段階手順（バルク・CDC・カットオーバー）を実施
- 整合性検証は行数・サム値・目視の 3 種類すべて合格
- 受入テスト不合格時は切替延期（強行禁止）

**優先度**: MUST

## MIG-ROLLBACK-001: 撤退判断基準

**現状**: 移行が失敗傾向を見せた際の撤退基準が曖昧だと、損切りが遅れて被害が拡大する。または逆に、軽微な問題で過剰に撤退してノウハウが蓄積されない。

**要件達成後**: 各 Wave の各システムで以下いずれかが発生した場合、撤退判断を起動する。起動は自動（監視トリガ）、判断は Product Council による合議とする。

- 並行運用中の差分が 3% 以上発生
- 新系の SLO が 30 日間 95% を下回る
- パフォーマンスが旧系の 2 倍以上遅い
- セキュリティインシデント発生
- 業務担当者が受入拒否

撤退時は Runbook（[40_運用ライフサイクル/05_撤退戦略.md](../40_運用ライフサイクル/05_撤退戦略.md)）に従い、トラフィックを旧系に戻す。撤退後は 30 日以内にポストモーテムを実施し、ADR または計画書に学びを反映する。

**崩れた時**: 撤退判断の遅れで業務停止が長期化する、または撤退後の再挑戦方針が立たず、同じ失敗を繰り返す。

**受け入れ基準**:
- 5 条件の監視トリガが Grafana で可視化されている
- 撤退発動時の Runbook が SRE 単独で実行可能
- 撤退後 30 日以内にポストモーテム完了

**優先度**: MUST

## MIG-PAR-001: 並行運用期間

**現状**: 並行運用期間が短すぎると切替時の差分検証が不足し、長すぎると運用コストが二重になる。

**要件達成後**: 各システムの並行運用期間を以下で規定する。

- **通常系**: 最低 1 か月
- **クリティカル系（S1/S2）**: 最低 3 か月

並行運用中は以下を継続モニタリングする。

- 業務的挙動差分（出力データ比較、承認結果、金額計算等）
- 技術的指標（レスポンスタイム、エラー率、リソース消費）
- 利用者フィードバック（業務部門）

**崩れた時**: 短期切替を強行すると、業務部門から想定外の差分指摘が出て切替直後に大規模な後戻りが発生する。逆に長期並行運用を漫然と続けると旧系の運用コストが発生し続け、TCO 試算が崩れる。

**受け入れ基準**:
- 全移行対象で最低期間を順守
- 並行運用短縮（例外）は Product Council 承認必須
- モニタリング 3 指標を Grafana で可視化

**優先度**: MUST

## MIG-LGL-001: 法的・契約的配慮

**現状**: 外部公開 API や監督官庁認可システムを移行する際、契約・通知・同意取得プロセスを飛ばすとコンプライアンス違反となる。

**要件達成後**: 以下を移行設計書の必須チェック項目とする。

- 監督官庁認可システムは、移行前に金融庁 / 経産省等への事前通知
- 外部取引先向け API の切替は契約変更手続き必須（場合により締結）
- 個人情報の移行は個情法の目的内であることを確認、必要なら同意取得

**崩れた時**: 事前通知漏れによる行政指導、契約違反による賠償、個情法違反による漏えい等報告義務発動のいずれかが発生する。

**受け入れ基準**:
- Wave 2 以降の移行設計書に法的チェックリスト必須
- 監督官庁認可系は法務リード承認必須
- 外部 API 系は調達リード承認必須

**優先度**: MUST
