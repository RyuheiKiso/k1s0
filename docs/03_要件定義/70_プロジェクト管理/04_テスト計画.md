# テスト計画

本書は k1s0 プロジェクトのテスト種別、カバレッジ基準、環境、自動化戦略を定める。QA 計画（[03_QA計画.md](03_QA計画.md)）で定めた品質指標を実現する具体実施計画として位置付ける。

## テスト戦略の思想

テストは **テストピラミッド + テストトロフィー** の両方の思想を組合せる。テストピラミッドは「単体 > 統合 > E2E」の数量比で、実行速度とフィードバック時間を最適化する。一方テストトロフィー（Kent C. Dodds）は「契約テストと統合テストに厚みを持たせる」思想で、マイクロサービスでは重要。k1s0 ではこの両方を取る。

加えて、マイクロサービス特有のテスト種別（契約テスト、Replay テスト、Chaos テスト）を必須として位置付ける。

## テスト種別と範囲

### 1. 単体テスト（Unit Test）

- 対象: 関数・クラス・モジュールレベル
- 技術: Rust `cargo test`、Go `testing`、TypeScript Vitest
- カバレッジ目標: tier1 コア 80%、致命的モジュール 95%、tier2/tier3 70%
- 実行頻度: 毎コミット（CI L1）
- 所要時間目標: 各モジュール 1 分以内

### 2. 契約テスト（Contract Test）

- 対象: tier1 API、tier2 間 API、外部連携境界
- 技術: Pact（Consumer-Driven Contract Testing）
- 方針: Consumer が期待する契約を Pact ファイルで定義、Provider が契約満足を自動検証
- 実行頻度: 毎コミット（PR マージ前に双方検証）
- 重要性: マイクロサービスでは単体テストと E2E だけでは相互作用バグが漏れる

### 3. 統合テスト（Integration Test）

- 対象: tier1 とバックエンド（PG / Kafka / MinIO 等）、tier1 同士の連携
- 技術: Testcontainers（Rust / Go / TypeScript で実コンテナ起動）
- 方針: モックではなく実コンテナで確認、本番との差分最小化
- 実行頻度: PR 時、夜間フル実行
- 所要時間: 一連フル 30 分以内

### 4. E2E テスト（End-to-End）

- 対象: tier3 UI → tier2 → tier1 → バックエンド の全経路
- 技術: Playwright（UI）、k6（API シナリオ）
- 方針: 主要ユーザストーリー 10〜20 本に絞る、量より本質性
- 実行頻度: 夜間 staging 環境、リリース前
- 所要時間: 一連 1 時間以内

### 5. 性能テスト（Load / Stress Test）

- 対象: tier1 API の SLO 確認
- 技術: k6、Gatling
- 方針: [07_負荷試験とキャパシティ.md](../40_運用ライフサイクル/07_負荷試験とキャパシティ.md) のシナリオを月次で実行
- 実行頻度: 月次、リリース前、大型機能追加時
- 判定基準: 想定トラフィックの 1.5 倍で SLO 維持

### 6. セキュリティテスト

- 対象: 脆弱性スキャン、依存ライブラリ、ペネトレーション
- 技術: Trivy、Snyk、OWASP ZAP、手動ペンテスト（年次外部）
- 実行頻度: 毎コミット（静的スキャン）、年次外部ペンテスト
- 判定基準: Critical / High ゼロ、Medium は 30 日以内対応

### 7. Replay テスト（Temporal Workflow）

- 対象: Temporal ワークフロー定義
- 技術: `temporalio/replay` ライブラリ
- 方針: 過去の実行履歴を新コードで Replay し、Determinism を検証
- 実行頻度: 毎コミット（Workflow 変更時）
- 重要性: FMEA RPN 135 最重要対策

### 8. ゴールデンケーステスト（ZEN Decision）

- 対象: JDM ルール定義
- 技術: ZEN Engine のテスト API、tier2 独自ハーネス
- 方針: 正解入出力を CSV / JSON で保持、ルール変更時に自動比較
- 実行頻度: ルール変更時、毎コミット
- 重要性: FMEA RPN 126 対策

### 9. Chaos テスト

- 対象: 障害耐性（Pod Kill、Network Partition、DB Failover）
- 技術: Chaos Mesh
- 方針: staging で GameDay 月次実施、prod では限定的に Phase 2 以降
- 実行頻度: 月次 GameDay、四半期定期
- 判定基準: RTO / RPO 目標内で自動復旧

### 10. 受入テスト（UAT: User Acceptance Test）

- 対象: 業務部門の実業務シナリオ
- 技術: 手動検証 + 補助的に Playwright
- 方針: 業務担当者と Stream-aligned Team 共同で実施、本番 Canary の前段階
- 実行頻度: 機能単位でリリース前

## テスト環境

### 環境階層

1. **local**: 開発者 PC。Testcontainers で軽量基盤
2. **dev**: Platform Team 共有、最新コード継続デプロイ
3. **staging**: prod 同等構成、本番類似データ、E2E / 性能テストの主戦場
4. **prod-canary**: 本番の 10% トラフィック、Argo Rollouts Canary
5. **prod**: 本番全トラフィック

### データ管理

- dev: 合成データ（PII を含まない）
- staging: 本番マスキング済みコピー（四半期更新、個情法対応）
- prod: 実データ（RLS による tenant 境界強制）

### 環境再現性

- 全環境は Argo CD で GitOps 同期
- Helm Chart + Kustomize overlay で環境差を最小化
- dev 環境は開発者が 15 分以内に再構築可能

## 自動化戦略

### CI/CD パイプライン

```
PR 作成時:
  ├── Lint / Format
  ├── 単体テスト
  ├── 契約テスト (Pact Consumer)
  ├── 統合テスト (Testcontainers)
  ├── セキュリティスキャン (Trivy / Snyk)
  ├── ライセンスチェック
  └── PR 承認ブロック判定

main マージ時:
  ├── Container Build
  ├── SBOM 生成
  ├── Sigstore 署名
  └── Argo CD 自動同期 → dev

staging 昇格:
  ├── E2E テスト（Playwright）
  ├── 性能テスト軽量版
  ├── 契約テスト (Pact Provider)
  └── 承認ゲート

prod 昇格:
  ├── Canary Rollout (10% → 50% → 100%)
  ├── SLO 自動監視
  ├── 逸脱時自動ロールバック
  └── Audit 記録
```

### テストインフラ

- 全テスト実行ログを Loki に集約、Grafana で可視化
- 欠陥 ID とテスト ID を Traceability Matrix で対応付け
- Flaky テスト検出、連続失敗は即 Issue 化

## カバレッジ要件と運用

カバレッジは「達成目標」であって「唯一の品質指標」ではない。以下のルールで運用する。

- カバレッジ低下時の自動 PR ブロック（既存値 -5% を許容）
- 致命的モジュール（crypto / PII / 認可）は 95% 強制
- カバレッジ計測からは「デバッグコード」「自動生成コード」「インフラ初期化」を除外
- カバレッジだけ上げるための **意味のない追加テスト** は禁止

## テストの優先順位

全種別を初日から全力実施するのは現実的でない。Phase 進行に合わせ段階的に充実させる。

- **Phase 1**: 単体 / 契約 / 統合 / E2E / 性能 / セキュリティ / Replay を必須
- **Phase 2**: Chaos / UAT を加え、GameDay 月次定着
- **Phase 3**: マルチクラスタの DR テスト、外販前提のペンテスト強化
