# BIZ-I18: 国際化 / 地域化要件

本章は、k1s0 の UI / ドキュメント / ログメッセージが扱うべき言語・タイムゾーン・地域表現を定義する。k1s0 は第一義的に JTC 国内業務で稼働するが、JTC の海外拠点・外国籍社員・業務委託の英語話者が一定割合存在することから、**日本語を既定言語とし、英語を同等品質で対応可能な状態** を最低ラインとする。多言語対応はその先の「拡張可能性」として設計上残し、Phase 進行に応じて昇格を判断する。

この章の要件は、単なる翻訳機能ではなく「法的・監査的に同一内容を保証できる翻訳運用」までを範囲に含む。免責条項や SLA の文言の翻訳差異は BIZ-LEG で契約リスクになるため、日英並存時の整合性保証が MUST となる。

## 要件本体

### BIZ-I18-001: 日本語を既定言語とし、英語を同等対応する

- 優先度: MUST（JTC の外国籍社員・英語業務委託が利用する前提。英語未対応は稟議承認後の即時論点）
- Phase: Phase 1b
- 関連: `BIZ-I18-002`, `BIZ-I18-004`, `BIZ-LEG-*`

現状、内製プラットフォームの多くは日本語のみで GUI を構築し、英語表示は「あとから足す」運用となる。後追いで英語化すると、ボタンラベル長の変化で UI がはみ出し、error_taxonomy 文言の英訳が担当者ごとに揺れ、BIZ-LEG の契約書（日本語正文／英訳参考）との齟齬が顕在化する。JTC の海外拠点パイロット（Phase 1b 想定）で英語表示が不完全だと、パイロット評価がそこで止まる。

要件達成後は、配信ポータル / Backstage / エラー画面 / ログメッセージ / 監査イベント名のすべてが日本語と英語の両方で同等品質のテキストを提供する。翻訳リソースは後述の BIZ-I18-002 の方式で管理し、新機能追加時に両言語同時にリリースされない PR は CI で検知して差し戻す。画面言語切替は Accept-Language ヘッダ → ユーザー設定 → Cookie の優先順で決定し、既定は日本語とする。

崩れた時の影響は、外国籍エンジニアが「k1s0 は日本語前提で使えない」と判断して退職オプションを行使する。JTC の採用競争上、これは単なる UX 問題ではなく、BIZ-TRN の Rust 育成パイプラインを壊すリスクに直結する。

**受け入れ基準**

- 配信ポータル / Backstage / エラー画面の全画面テキストが日英両言語で提供される
- 新規機能 PR に英訳が同梱されない場合に CI で PR をブロック
- 言語切替が Accept-Language / ユーザー設定 / Cookie の優先順で動作する

**検証方法**

- E2E テストを日英両ロケールで実行
- 翻訳抜けを検出する Lint（未翻訳キーの存在で失敗）を CI に組み込む

### BIZ-I18-002: 翻訳リソースは gettext / ICU MessageFormat で管理する

- 優先度: SHOULD（技術選定の統一。Phase 1b で固定しないと Phase 2 以降の追加コストが指数的に増える）
- Phase: Phase 1b
- 関連: `BIZ-I18-001`, `BIZ-I18-004`, `COM-XRQ-*`

現状、翻訳リソースを個別のコンポーネントが独自 JSON / YAML / TOML で抱え込むと、同一文言が複数ファイルに散らばり、翻訳の一貫性と変更追跡性が失われる。とくに「ボタン: 保存 / 保存する / セーブ」のような揺れは業務担当者の操作ミスを誘発する。

要件達成後は、k1s0 全体が **gettext (`.po` ファイル) または ICU MessageFormat** のいずれか 1 種類に統一する。フロントエンド (TypeScript / Backstage plugin) は ICU MessageFormat（Format.js）、バックエンドログ・エラーは gettext を基準とし、両者の対応表を 1 つの翻訳キー辞書で管理する。翻訳 PR は専用レビュワー（言語担当）を必須化し、語彙の揺れを PR レビューで吸収する。

崩れた時の影響は、5 年後に翻訳辞書のメンテナンスコストが人員 1 名相当まで膨れ上がり、BIZ-CST の運用コストを圧迫する。

**受け入れ基準**

- 翻訳リソースの格納形式が gettext / ICU MessageFormat のいずれかに統一される
- 翻訳辞書が単一のソースオブトゥルース（例: `infra/i18n`）で管理される
- 翻訳 PR は言語担当レビュワーの承認を必須化

**検証方法**

- 新規翻訳キー追加 PR が単一辞書経由であることを CI で検査
- 翻訳キーの参照漏れ・未使用キー検出 Lint を四半期ごとに実施

### BIZ-I18-003: タイムゾーンは内部 UTC / 表示 JST を既定とする

- 優先度: MUST（監査ログ・契約書・SLA 測定の時刻基準を固定しないと、訴訟・監査で事実認定が揺らぐ）
- Phase: Phase 1a
- 関連: `SEC-AUD-*`, `BIZ-LEG-*`, `QUA-SLA-*`

現状、監査ログ・業務データ・UI 表示が混在するシステムでは「いつの時刻か」を JST と UTC で取り違える事故が頻発する。とくに `k1s0.Audit` ([`../../01_企画/05_法務サマリ/02_日本法令適合.md`](../../01_企画/05_法務サマリ/02_日本法令適合.md) 第 2 節) のハッシュチェーンに記録される時刻が UTC と JST のどちらか判別できないと、電子帳簿保存法の「真実性」要件で証拠能力が弱くなる。

要件達成後は、k1s0 内部の全時刻は UTC で保持し、UI 表示と業務レポートは JST (Asia/Tokyo) で既定表示する。ISO 8601 形式で時刻を格納し、オフセット情報を必ず保持する。ユーザーのタイムゾーン切替は UI 設定で可能だが、監査ログ表示は常に UTC / JST の両方を併記する。

崩れた時の影響は、BIZ-LEG の SLA 測定（例: p99 500ms を月次 99.5%）が時刻基準の不整合で争点化する。監査（BIZ-LEG-004 / SEC-AUD-*）応答で「時刻は正しいか」を追加検証する余計なコストが発生する。

**受け入れ基準**

- 全コンポーネントの時刻格納が UTC・ISO 8601 形式
- UI / レポート既定表示が JST で、監査画面は UTC / JST 併記
- CI で「ローカル時刻で格納する」コードパターンを静的検査で検出

**検証方法**

- 監査ログのサンプルで時刻フィールドに UTC オフセットが含まれることを検査
- サマータイム未採用地域のエッジケースでも表示ずれが発生しないことを単体テストで確認

### BIZ-I18-004: 将来の多言語拡張性を構造的に維持する

- 優先度: SHOULD（Phase 2 以降の海外展開オプションを閉じないための拡張性確保）
- Phase: Phase 2
- 関連: `BIZ-I18-001`, `BIZ-I18-002`

現状、日英 2 言語の設計を素直に実装すると、コードに英語文字列が直書きされ、中国語・ベトナム語など将来言語の追加が「全ソースコード検索 → 置換」の長時間タスクになる。JTC の海外拠点（アジア圏）パイロットが Phase 2 で始まる想定を踏まえると、拡張性を構造で確保しておかないと、Phase 2 着手時の見積が跳ね上がる。

要件達成後は、日英どちらもコード直書きではなく翻訳キー経由で参照され、新言語追加は翻訳辞書への言語追加と CI 検査の言語リスト追加のみで完了する。RTL (アラビア語・ヘブライ語) 対応は Phase 3 以降の保留項目とし、Phase 2 までは LTR 言語のみを想定する。

崩れた時の影響は、Phase 2 の海外展開で追加工数 3〜5 人月を要し、BIZ-CST の FinOps 目標 (Phase 2 の人員増分) を超過する。

**受け入れ基準**

- 全コード上の表示文字列が翻訳キー経由で参照される
- CI で「翻訳キー経由でない日本語文字列リテラル」を静的検査で検出
- 言語追加手順が Runbook 化されている

**検証方法**

- 翻訳キー未経由の文字列リテラルを検出する Lint（日本語・英語両方）
- 新言語（仮: 韓国語）追加の POC を Phase 2 着手前に 1 回実施し、想定工数内に収まることを確認
