# BIZ-TRN: トレーニング要件

本章は、k1s0 を運用・開発・利用する各立場のトレーニング要件を定義する。対象は内部（tier1 開発者 / SRE）と外部（tier2/3 開発者 / 業務部門の配信ポータル利用者）の 2 群に分かれる。内部向けは **Rust 育成** という稀有性の高い技術領域を含むため、6 か月カリキュラムの規律化が必要となる。外部向けは **1 日〜1 週間のハンズオン** を軸にし、認定制度で品質基準を担保する。

トレーニングの失敗は、BIZ-CST の TCO 試算の人件費構造（中規模 5 FTE、[`../../01_企画/04_定量試算/01_TCO5年試算.md`](../../01_企画/04_定量試算/01_TCO5年試算.md) 第 2.2 節）を前提ごと揺るがす。「Rust が書ける人が採れなかった」という事態は、SES 単価 +30〜50% を招き ([`../../01_企画/04_定量試算/01_TCO5年試算.md`](../../01_企画/04_定量試算/01_TCO5年試算.md) 第 2.4 節)、5 年 TCO を 3.68 億から 5 億超に押し上げる。

## 要件本体

### BIZ-TRN-001: 初期オンボーディング教材（2 日間）

- 優先度: MUST（オンボーディング品質の共通ベースライン。各テナントで毎回再発明するのは不経済）
- Phase: Phase 1b
- 関連: `BIZ-ONB-003`, `BIZ-TRN-003`

現状、内製基盤の初期教育は「先輩エンジニアが資料を都度作って 3 時間説明」という構造が常態化し、資料の質と説明内容が担当者ごとに異なる。新規利用者が「配信ポータルの使い方」「Backstage からの雛形生成」「tier1 公開 API の呼び出し」を学ぶまでに、平均して 5 営業日が経過し、BIZ-ONB-004 の 5 営業日目標と衝突する。

要件達成後は、**2 日間の標準オンボーディング教材** が整備され、1 日目にアーキテクチャ概論・tier 分離思想・配信ポータル実習、2 日目に Backstage 実習・雛形 CLI 実習・受け入れテスト実習を組み込む。教材は Backstage TechDocs で配信し、バージョン管理を Git で行う。オンボーディング時は Runbook に沿って講師（内部 SRE）が進行し、受講者の完了チェックは自動テストで検証する。

崩れた時の影響は、テナント受け入れ時に初期教育で 3〜5 営業日の余計な時間を要し、BIZ-ONB-004 の 5 営業日目標の達成率が 50% 以下に落ちる。

**受け入れ基準**

- 2 日間の標準教材が Backstage TechDocs で提供される
- 教材は 1 日目（概論・配信ポータル）/ 2 日目（Backstage・雛形 CLI・受け入れテスト）で構成
- 受講者の完了チェックが自動テスト（クイズ + 実習課題）で実施される

**検証方法**

- 新規テナント受け入れ時の初回研修所要日数を記録し、平均 2 日以内
- 教材の陳腐化を防ぐため、四半期ごとに受講者フィードバックを収集し改訂

### BIZ-TRN-002: Rust 育成カリキュラム（6 か月）

- 優先度: MUST（TCO 試算の前提条件。BIZ-CST の人件費見積が Rust 採用成功に依存）
- Phase: Phase 0〜2
- 関連: `BIZ-CST-*`, `COM-XRQ-003`

現状、JTC 情シスの多くは Go または Java 経験者で構成され、Rust 実務経験者は国内採用市場全体でも希少（LinkedIn 検索で Tokyo 在住 Rust 5 年以上は 200 名未満）。Rust 未経験エンジニアが k1s0 の tier1 自作領域（crypto / 雛形 CLI / ZEN Engine 統合）を書けるようになるには所有権・ライフタイム・async の 3 概念を最低限理解する必要があり、自習だけでは 1 年超を要する。

要件達成後は、**Go 経験者を Rust 初級（自立コーディング可能レベル）まで育成する 6 か月カリキュラム** を運用する。月次の到達目標は、1 か月目（所有権・借用）/ 2 か月目（ライフタイム・trait）/ 3 か月目（Result / async / tokio）/ 4 か月目（Cargo / テスト / ベンチ）/ 5 か月目（ZEN Engine 実践課題）/ 6 か月目（tier1 機能追加の PR 提出）と設定する。育成期間中は本業稼働率 60% ([`../../01_企画/04_定量試算/01_TCO5年試算.md`](../../01_企画/04_定量試算/01_TCO5年試算.md) 第 2.4 節) とし、差分 40% の 6 か月を育成純人件費として計上する。外部メンタリング（月 1 回 × 15 万円）を組み合わせて到達確実性を高める。

崩れた時の影響は、Rust 書き手が採用できずに SES 依存度が上がり、人月単価が 100〜115 万から 130〜150 万に上振れる。5 年 TCO で +0.5〜1.0 億の上乗せが発生する。

**受け入れ基準**

- 6 か月カリキュラムが月次到達目標付きで整備される
- 受講期間中の本業稼働率 60% を運用枠として人員計画に織り込む
- 外部メンタリング枠（月 1 回 × 6 か月）が予算化される
- 最終月（6 か月目）の PR 提出が到達度評価の MUST 条件

**検証方法**

- 育成対象者が 6 か月目に tier1 自作領域への PR を提出しマージされることを確認
- 半年ごとに育成コホートの到達率（卒業 / 離脱 / 延長）を集計し、目標 70% 以上の卒業率を維持

### BIZ-TRN-003: 顧客（tier2/3 開発者）向けハンズオン

- 優先度: SHOULD（外部採用促進の基礎だが、Phase 1b のパイロット業務で必要になる時点で整備）
- Phase: Phase 1b
- 関連: `BIZ-TRN-001`, `BIZ-TRN-004`, `DEV-DEX-*`

現状、tier2/3 開発者は JTC のグループ会社や SIer 所属であり、k1s0 の tier1 公開 API・雛形 CLI・Backstage テンプレートを学ぶ機会がない。ハンズオンが整備されていないと、tier2/3 開発者は「k1s0 を使うより素の K8s + Dapr を叩く方が早い」と判断し、tier1 ファサード経由の統制（`ARC-T1-*`）がバイパスされる。

要件達成後は、**1 日ハンズオン（基礎）と 3 日ハンズオン（応用）の 2 コース** を提供する。基礎コースは tier1 公開 API の呼び出しと雛形 CLI による tier3 アプリの生成を扱い、応用コースは監査ログ統合・SLO 設定・配信ポータル連携までカバーする。教材は Backstage TechDocs で配信し、受講後のサポート窓口を明示する。

崩れた時の影響は、tier2/3 開発者が tier1 ファサードをバイパスし、監査ログ・SLO・ライセンス統制が効かない運用が蔓延する。BIZ-LEG の SLA 条項・BIZ-LIC のライセンス統制が契約上の前提を失う。

**受け入れ基準**

- 1 日（基礎）と 3 日（応用）の 2 コースが教材 + 課題付きで整備される
- 応用コース修了者は BIZ-TRN-004 の認定制度の前提条件とする
- 受講後のサポート窓口と質問対応 SLA（1 営業日以内一次応答）が定義される

**検証方法**

- Phase 1b のパイロット業務でハンズオンを試験運用し、受講後の tier1 公開 API 利用率を計測
- 修了後の技術的問い合わせ件数と種類を分類し、教材改訂の材料とする

### BIZ-TRN-004: 到達度テスト / 認定制度

- 優先度: MAY（Phase 2 以降の外部展開で効果を発揮する拡張要素）
- Phase: Phase 2
- 関連: `BIZ-TRN-003`, `BIZ-ONB-003`

現状、トレーニング受講者のスキル到達度を客観的に確認する仕組みがないと、tier2/3 開発者の受け入れ判断が「名前と所属で決める」人的判断に戻る。Phase 2 で数百名規模の tier2/3 開発者を受け入れる際、この判断コストは跳ね上がる。

要件達成後は、**k1s0 認定制度（Developer / Operator / Architect の 3 段階）** を運用する。Developer 認定は BIZ-TRN-003 基礎コース修了 + 実機課題合格、Operator 認定は応用コース + 運用 Runbook 理解テスト、Architect 認定は複数テナント対応の設計課題提出をもって付与する。認定有効期限は 2 年とし、再認定は差分テストで簡略化する。

崩れた時の影響は、Phase 2 で tier2/3 側のスキル基準が揃わず、BIZ-LEG の免責条項（`k1s0 が提供する公式ドキュメント・認定コースの範囲外での利用は免責` 想定）の運用が曖昧になる。

**受け入れ基準**

- 3 段階の認定制度が運用開始される
- 認定試験は実機課題を含み、採点基準が公開される
- 認定有効期限と再認定フローが Runbook 化される

**検証方法**

- 認定受験者数と合格率を四半期ごとに集計
- 認定保有者の利用定着率（MAU）を非保有者と比較し、認定効果を測定
