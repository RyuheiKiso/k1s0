# BIZ-CST: コスト管理要件

本章は、k1s0 の 5 年 TCO を運用段階で継続追跡するための要件を定義する。企画 [`../../01_企画/04_定量試算/01_TCO5年試算.md`](../../01_企画/04_定量試算/01_TCO5年試算.md) で宣言した「中規模 3,000 名で 5 年 TCO 3.68 億」という稟議の約束は、Phase 2 以降の運用期に実績値で検証される。実績が試算を継続的に上振れると、稟議通過の根拠が事後的に無効化され、Phase 3 以降の予算承認が停止する。

本章の要件は、TCO を「月次で見える状態」「異常を自動通知する状態」「テナント別に配賦できる状態」の 3 段で条項化する。FinOps の運用ロールを定義し、コストが運用部門の責任下で管理される構造を作る。BIZ-BIL の課金メータリングとはソースデータを共有するが、BIZ-CST は**社内の内部原価管理**、BIZ-BIL は**対外請求根拠** として役割を分離する。

## 要件本体

### BIZ-CST-001: TCO 実績の月次レポート

- 優先度: MUST（稟議承認時の TCO 約束を運用で検証する義務。未実施で Phase 2 予算承認が止まる）
- Phase: Phase 2
- 関連: `BIZ-CST-002`, `BIZ-CST-003`, `BIZ-ANL-*`

現状、内製基盤の運用コストは「請求書が届いたら経理が合算する」という事後把握で進み、稟議段階の試算と実績が乖離しても誰も気づかないまま数年が経過する。k1s0 は稟議で中規模 TCO 3.68 億 (5 年) を宣言しているため、実績の年 7,366 万 (3.68 億 ÷ 5) を月次 614 万ベースで追跡しないと、Phase 3 完了時点で事後乖離が発覚する。

要件達成後は、**TCO 実績の月次レポート** が自動生成される。レポートの内訳は人件費 / インフラ費 / ライセンス費 / 教育費の 4 項目で、各項目を試算値（企画 04_定量試算）と並置する。生成は Grafana + Prometheus（月次メトリクス集計）と経理システムの CSV 連携で行い、月次 1 営業日目の 18:00 までに自動配信する。閲覧者は情シス管理職 / FinOps / 経理 / 稟議承認時の決裁者。

崩れた時の影響は、稟議時の TCO 約束が「事後検証不能」となり、次回の Phase 予算稟議（Phase 2 → 3 の継続判断）で決裁者の信頼を失う。結果として、Phase 3 の全社ロールアウト予算が承認されず、k1s0 が Phase 2 で実質停止する。

**受け入れ基準**

- 月次レポートが自動生成され、1 営業日目 18:00 までに配信される
- レポートは人件費 / インフラ費 / ライセンス費 / 教育費の 4 項目を試算値と並置する
- 閲覧者リストが Keycloak グループで管理される

**検証方法**

- 月次レポートの配信遅延が四半期で 1 回以下
- レポート内容の整合性を経理決算値と四半期ごとに突合

### BIZ-CST-002: コスト異常検知（月次予算 +20% でアラート）

- 優先度: MUST（月次乖離を放置すると年次で四半期分の超過に拡大する）
- Phase: Phase 2
- 関連: `BIZ-CST-001`, `QUA-OBS-*`

現状、月次コストの異常値（単月の人件費急増・クラウド課金の bug / overage）を検出する仕組みがないと、異常が 3 か月目の経理精算で初めて発覚する。OSS 無償の罠 ([`../../01_企画/04_定量試算/04_OSS無償の罠を潰す.md`](../../01_企画/04_定量試算/04_OSS無償の罠を潰す.md)) で指摘される「運用費が気づかないうちに膨らむ」パターンはこの検知遅延に起因する。

要件達成後は、**月次予算の +20% をしきい値としたコスト異常検知** が Prometheus + Alertmanager で自動実行される。20% の根拠は業界標準の FinOps プラクティス（AWS Cost Anomaly Detection の既定感度が月次 +20%）で、JTC 内部基盤でも保守的な検知感度として妥当。アラートは FinOps ロール (BIZ-CST-004) と情シス管理職に通知し、72 時間以内に原因分析と対応方針を記録する。

崩れた時の影響は、単月の異常（例: Rust 育成 SES 代替で +300 万/月）が 3 か月放置され、年次で -1,000 万の予算超過として確定する。BIZ-CST-001 の月次レポートの信頼性も同時に毀損する。

**受け入れ基準**

- 月次予算 +20% 超過で Alertmanager アラートが発火
- アラート受領から 72 時間以内に原因分析結果が Runbook に記録される
- 連続 2 か月発火時はエスカレーション先（情シス本部長）が自動追加される

**検証方法**

- アラート発火から 72 時間以内対応率を四半期で計測（目標 100%）
- 誤報率（実害なしアラート / 全アラート）を月次集計し、高すぎる場合はしきい値再調整

### BIZ-CST-003: テナント別コスト配賦

- 優先度: SHOULD（全社ロールアウト段階で必須化。Phase 1b 時点は簡易配賦で可）
- Phase: Phase 2
- 関連: `BIZ-CST-001`, `BIZ-BIL-*`, `QUA-TIS-*`

現状、内部プラットフォームのコストは「情シス全体経費」として一括計上され、事業部ごとの利用量と経費が対応しない。結果として「使っていない部門が他部門のコストを負担している」と映り、Phase 2 の横展開で受け入れ拒否の理由になる。

要件達成後は、**テナント別（事業部別 / 子会社別）のコスト配賦** が月次で実施される。配賦基準は CPU 時間・メモリ使用量・ストレージ使用量・API コール数の 4 指標で、各指標の単価は企画 [`../../01_企画/04_定量試算/00_試算前提.md`](../../01_企画/04_定量試算/00_試算前提.md) 第 3 章の単価を基準とする。配賦結果は BIZ-BIL の課金メータリングとソースデータを共有するが、BIZ-CST は内部原価、BIZ-BIL は対外請求根拠として独立レポートを出力する。

崩れた時の影響は、Phase 2 で事業部ごとの k1s0 利用コストが可視化されず、事業部側の利用意思決定（内製 vs SaaS）が合理化できなくなる。

**受け入れ基準**

- 4 指標（CPU / メモリ / ストレージ / API コール）でテナント別配賦が実行される
- 配賦レポートが月次で各テナント管理者に配信される
- 配賦単価が [`../../01_企画/04_定量試算/00_試算前提.md`](../../01_企画/04_定量試算/00_試算前提.md) 第 3 章の単価と整合

**検証方法**

- 全テナントの配賦合計値が情シス全体コストと月次で一致（誤差 1% 以内）
- 配賦単価改訂時は ADR で記録

### BIZ-CST-004: FinOps ロールの定義

- 優先度: SHOULD（組織運用の基盤。Phase 2 着手時点で役割を明示）
- Phase: Phase 2
- 関連: `BIZ-CST-001`, `BIZ-CST-002`, `BIZ-GOV-*`

現状、コスト管理の責任者が不明確なまま運用が始まると、BIZ-CST-001 / 002 / 003 の各要件が「誰の業務か」の押し付け合いで機能不全になる。

要件達成後は、**FinOps ロール** が情シス組織内で明示的に定義される（兼務可、最低 0.3 FTE）。ロールの責務は、月次レポートのレビュー、異常検知アラートの一次トリアージ、テナント別配賦結果の精算依頼、四半期の TCO 実績レビュー（企画試算との整合性確認）。BIZ-GOV の RACI に組み込み、変更審議会で定期報告を行う。

崩れた時の影響は、BIZ-CST 系要件が「全員の仕事で誰の仕事でもない」状態になり、月次レポートの配信遅延が常態化する。

**受け入れ基準**

- FinOps ロールの責務が BIZ-GOV の RACI に明記される
- ロール担当者が指名され、変更履歴が Git 管理される
- 四半期 TCO レビューの議事録が公開される

**検証方法**

- FinOps 担当者不在の期間がゼロ（交代時は引継ぎ期間を Runbook で必ず確保）
- 四半期 TCO レビューの実施率 100%
