# BIZ-LIC: ライセンス要件

本章は、k1s0 自体に付与するライセンスと、依存 OSS のライセンス義務への対応を要件化する。k1s0 自身のライセンス選定は **一度公開すれば取り消せない** 性質の不可逆判断であり、Phase 0〜4 は「社内独占（Proprietary）」を既定値として選択する。これは [`../../02_構想設計/05_法務とコンプライアンス/01_内製知財戦略.md`](../../02_構想設計/05_法務とコンプライアンス/01_内製知財戦略.md) 第 2.1 節で詳細化された方針であり、稟議時点の OSS 化議論を Phase 5 以降へ段階的に先送りする構造を取る。

一方、依存 OSS（採用 40+ 本、AGPL 6 本を含む）のライセンス義務は Phase 1a から運用開始する必要がある。採用 OSS のライセンス分布は [`../../01_企画/05_法務サマリ/01_OSSライセンス適合.md`](../../01_企画/05_法務サマリ/01_OSSライセンス適合.md) 第 1 節で整理済みであり、本章は**運用上の証跡化** を要件として条項化する。

## 要件本体

### BIZ-LIC-001: k1s0 自体のライセンスは Proprietary（社内独占）を既定とする

- 優先度: MUST（ライセンス選定は不可逆。Phase 0 時点で既定値を宣言しないと開発着手できない）
- Phase: Phase 1a
- 関連: `BIZ-LIC-002`, `BIZ-LEG-004`, `BIZ-GOV-*`

現状、内製プラットフォームでライセンス方針を宣言しないまま開発を始めると、外部委託・グループ会社配布・GitHub 公開の依頼が来たときに「都度法務判断」となり、判断コストが開発速度を蝕む。k1s0 のように長期運用（10 年）と段階的展開を前提とする基盤では、この都度判断は致命的。

要件達成後は、k1s0 自体のライセンスは **Proprietary（社内独占、JTC 法人帰属、著作権法 15 条 2 項のプログラム著作物特例）** を Phase 0〜4 の既定値とする。Phase 5 以降で Apache 2.0 または BUSL 等の OSS 化を再評価する余地を残すが、Phase 4 完了までは検討自体を凍結する。初期に Apache 2.0 を選ぶと「競合他社も使える」状態を作り出し、JTC の競争優位を自ら棄てることになるため、既定値は Proprietary が妥当。BUSL は将来 OSS 化時の一次候補として検討するが、Phase 0 時点では選定確定しない（[`../../02_構想設計/05_法務とコンプライアンス/01_内製知財戦略.md`](../../02_構想設計/05_法務とコンプライアンス/01_内製知財戦略.md) 第 2.1 節）。

崩れた時の影響は、外部ベンダー・グループ会社からのアクセス要求が発生したときに、1 件ごとに法務部門のフル審査（1 件 2〜4 週間）が発動し、Phase 2 以降の展開速度が本質的に低下する。

**受け入れ基準**

- k1s0 リポジトリの `LICENSE` ファイルが Proprietary（JTC 法人帰属）を明記
- Phase 5 以降の OSS 化評価を ADR 起票タイミングとして明文化
- グループ会社配布・委託業者利用の手続きが契約雛形で定義される

**検証方法**

- リポジトリ直下の `LICENSE` ファイル存在と文言を CI で検査
- 外部配布要求を発生時に手続きログを Git 管理

### BIZ-LIC-002: 商用利用条項（グループ会社間配布 / 委託業者利用）

- 優先度: MUST（Proprietary 既定下でも許諾範囲を明示しないと業務委託開発が止まる）
- Phase: Phase 1a
- 関連: `BIZ-LIC-001`, `BIZ-LEG-004`, `BIZ-VEN-003`

現状、k1s0 が Proprietary であっても、グループ会社（例: 子会社 A）が k1s0 を内部利用するケース、業務委託エンジニアが開発期間中に k1s0 のソースコードにアクセスするケース、SES エンジニアが tier3 アプリを k1s0 上で開発するケースで、利用許諾の根拠が曖昧だと契約上の争点になる。

要件達成後は、商用利用条項として以下を明文化する: (1) JTC グループ会社は関連会社契約の範囲内で無償内部利用可 ([`../../02_構想設計/05_法務とコンプライアンス/01_内製知財戦略.md`](../../02_構想設計/05_法務とコンプライアンス/01_内製知財戦略.md) 第 2.1 節)、(2) 業務委託エンジニアは委託期間中に限りアクセス可、終了時にコード・ドキュメント・認証情報の返却 / 削除義務、(3) SES エンジニアは tier3 アプリ開発目的に限り tier1 公開 API のドキュメントにアクセス可、k1s0 内部実装コード（Rust / Go 自作領域）へのアクセス不可。

崩れた時の影響は、業務委託エンジニア終了時のコード持ち出しリスクが契約上の抑止力を持たず、BIZ-LEG の知財帰属（職務著作に基づく JTC 帰属）が事実上骨抜きになる。

**受け入れ基準**

- 商用利用条項が契約雛形に組み込まれている
- グループ会社配布時の手続きが Runbook 化されている
- 委託業者契約に返却 / 削除義務と秘密保持条項が必ず入る

**検証方法**

- 契約雛形の版数管理と法務レビュー記録を四半期ごとに確認
- 委託業者契約終了時のコード返却 / 削除証跡を全件 Git 管理

### BIZ-LIC-003: 依存 OSS のライセンス台帳（SBOM 連動）

- 優先度: MUST（AGPL 6 本の証跡は監査の一次質問。SBOM 未整備は稟議・監査両方の致命傷）
- Phase: Phase 1a
- 関連: `SEC-ART-*`, `COM-CON-004`, [`../../01_企画/05_法務サマリ/04_監査応答.md`](../../01_企画/05_法務サマリ/04_監査応答.md)

現状、OSS 依存の把握が「担当者の記憶」に依存すると、監査で「採用 OSS 一覧を出せ」と問われたときに 1〜2 週間の確認作業が発生する。[`../../01_企画/05_法務サマリ/04_監査応答.md`](../../01_企画/05_法務サマリ/04_監査応答.md) 第 2 節の典型 5 問のうち 3 問が OSS 一覧とライセンス状態を問うものであり、この回答遅延は監査評価を直撃する。

要件達成後は、**Syft によって生成される SBOM（CycloneDX 形式）が全ビルド時に自動生成** され、Harbor と `infra/compliance-reports` リポジトリに格納される。SBOM にはライセンス情報が埋め込まれ、四半期ごとに自動で法務部門向けレポートが生成される。AGPL 6 本の利用状況はこの SBOM から自動抽出され、[`../../01_企画/05_法務サマリ/05_AGPL内部利用の公開根拠.md`](../../01_企画/05_法務サマリ/05_AGPL内部利用の公開根拠.md) 第 7 節の 3 条件（未改変 / プロセス分離 / 外部境界遮断）の遵守を監査可能な形で保全する。

崩れた時の影響は、監査応答で人手調査が発生し、BIZ-LEG の監査協力条項に反する。また AGPL 利用の証跡が不十分な場合、[`../../01_企画/05_法務サマリ/05_AGPL内部利用の公開根拠.md`](../../01_企画/05_法務サマリ/05_AGPL内部利用の公開根拠.md) 第 3 節の Neo4j v. PureThink 類似ケース（特にソース提供義務判定）で反証材料を失う。

**受け入れ基準**

- 全ビルド時に Syft で SBOM が自動生成される
- SBOM が `infra/compliance-reports` に 5 年以上保管される
- 四半期自動レポートが法務部門に配信される
- Grype + allow list 違反で CI が PR をブロック

**検証方法**

- 抜き打ちで過去 5 年のうちランダムな日付を指定し、該当日の SBOM を数分以内に復元できるか検証
- allow list (Apache / MIT / BSD / PostgreSQL / ISC / MPL 2.0 / AGPL-3.0 の 7 種) 外のライセンス検出時に PR ブロックが発火することを四半期テスト

### BIZ-LIC-004: 上流 OSS への還元方針

- 優先度: SHOULD（採用 OSS の継続利用にはコミュニティ貢献が有効だが、Phase 1a では義務化しない）
- Phase: Phase 2
- 関連: `BIZ-LIC-003`, `BIZ-VEN-002`

現状、OSS 採用企業が「受益するだけ」の立場を続けると、採用 OSS のメンテナンスが先細りした際の影響をフル受けする。Redis / Terraform / Elastic のライセンス変更 ([`../../01_企画/05_法務サマリ/01_OSSライセンス適合.md`](../../01_企画/05_法務サマリ/01_OSSライセンス適合.md) 第 4 節) のような事例は、コミュニティ関与がない企業から順に選択肢を奪う。

要件達成後は、**採用 OSS の上流への還元ルート** を手続き化する。対象は、バグ発見時のレポート（全採用 OSS 対象）、軽微な修正 PR（AGPL 以外の Apache 2.0 / MIT / BSD 系で推奨）、ドキュメント修正（全採用 OSS 対象）。PR を送る際は JTC 法務部門の事前承認（職務著作に基づく OSS 公開）を経る。AGPL OSS への fork・改変は [`../../01_企画/05_法務サマリ/01_OSSライセンス適合.md`](../../01_企画/05_法務サマリ/01_OSSライセンス適合.md) 第 2 節の 5 原則に反するため、PR 送付を原則禁止する（ドキュメント修正は例外として許可）。

崩れた時の影響は、採用 OSS のバグが他社に先に PR され、自社での発見 → 回避が 1 サイクル後回しになる。長期的には BIZ-SUS の 10 年運用設計で OSS メンテナンスコストが膨張する。

**受け入れ基準**

- 上流還元手続きが Runbook 化され、法務事前承認フローが定義される
- AGPL OSS への改変 PR は原則禁止（ドキュメント修正のみ許可）
- 年次の還元実績（PR 件数 / issue 報告件数）が BIZ-GOV レビューに含まれる

**検証方法**

- 四半期ごとに還元実績をレポート
- 還元 PR の事前承認記録を Git 管理し、未承認 PR 送付が発生しないことを監査
