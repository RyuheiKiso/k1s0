# BIZ-UX: UX 要件

本カテゴリは、k1s0 の事業価値を利用者が「触って迷わない」水準にまで引き上げるための UX 要件を扱う。k1s0 の機能的強度は tier1 ファサードと配信ポータル・Backstage・エラー画面という 3 つの GUI 境界を通じて初めて体感されるため、UI が散漫だと TCO 3.68 億 ([`../../01_企画/04_定量試算/01_TCO5年試算.md`](../../01_企画/04_定量試算/01_TCO5年試算.md) 第 2.5 節) の投資が「使われないプラットフォーム」として毀損する。本章は、この毀損を契約・監査・検収で確認可能な条項に翻訳する。

要件は SHOULD 中心で構成する。UX は Phase 1b のパイロット業務フィードバックで段階的に練る性質であり、Phase 1a 承認時点では「デザインシステムの軸を一つに決める」「操作時間の目標を数値で置く」の 2 点のみを MUST とする。それ以外の細目は Phase 1b 以降の昇格候補として保留する。

## 要件本体

### BIZ-UX-001: デザインシステムの統一

- 優先度: MUST（稟議承認後の最初の設計イテレーションで凍結すべき不可逆判断。後戻しすると 2 ポータル併用固有工数 0.3 人月/年が倍増する）
- Phase: Phase 1a
- 関連: `BIZ-UX-002`, `BIZ-UX-003`, `BIZ-ACC-*`

現状、配信ポータルは独自実装、Backstage は Material Design (MUI) ベース、Grafana は独自 UI、Keycloak は AdminUI という 4 系統のデザインが並存する設計路線となる。ボタンの色、角丸、フォーカスリング、エラー文言のトーンがコンポーネントごとに異なると、利用者は「これは別システムだ」と誤認し、「1 つのプラットフォームを使っている」という心象を失う。運用工数試算の 2 ポータル併用固有工数 0.3 人月/年はこの分散の維持コストそのものである。

要件達成後の世界では、k1s0 の全 GUI は 1 種類のデザインシステム（第一候補: Material Design 3、第二候補: 自社デザインシステム化）のトークンとコンポーネント語彙を共有する。ボタン・入力・ダイアログ・トースト・エラーページまで同一のビジュアル文法で構成されるため、利用者は初めて触るツールでも「Backstage で覚えた挙動」が配信ポータルで通用する。OSS UI (Grafana / Keycloak) はブランドカラーを合わせる最低限のテーマ注入に留め、完全統一は Phase 2 以降の昇格扱いとする。

崩れた時の影響は、PoC 現場で「どの画面が正しい入り口か分からない」という一次質問が継続発生し、BIZ-ONB の 5 営業日リードタイムに食い込む。さらに OSS UI とのギャップが広がると Grafana / Backstage のメジャーアップデートで追従工数が跳ね上がり、BIZ-SUS の技術負債返済枠を食い潰す。

**受け入れ基準**

- デザインシステムの採択が ADR として記録されている（Phase 1a で起票、承認は Phase 1b 着手時点まで）
- 配信ポータルと Backstage の共通コンポーネント（ボタン / 入力 / ダイアログ / トースト / エラーページ）が同一トークンを参照する
- `BIZ-UX-004` と整合するエラー画面テンプレートが 1 種類に統一されている

**検証方法**

- デザイントークン（色・フォント・余白）のハッシュを両ポータルで比較し、差分ゼロを CI で検査する
- 新規 UI 追加 PR に対してデザインレビューを必須化し、デザインシステム外のコンポーネント使用を PR ブロックする

### BIZ-UX-002: 配信ポータルの操作時間 3 クリック以内

- 優先度: SHOULD（Phase 1b でパイロット業務開始時に達成必須。Phase 1a は目標宣言で足りる）
- Phase: Phase 1b
- 関連: `BIZ-UX-001`, `BIZ-ONB-002`, `QUA-PRF-*`

現状、OSS の開発者ポータル (Backstage 単体) は「サービス作成 → テンプレート選択 → 入力 → 実行」まで 5〜7 クリックを要する。配信ポータルが同じ深さで実装されると、業務部門が 100 端末への配信指示を出すたびに待機時間 30 秒超を全員が被り、1,000 端末規模では月換算 5 時間のムダ操作が発生する。JTC の業務担当者は「IT ツールは 1 日に数分しか触らない」ため、操作時間の UX 摩擦は定着率に直結する。

要件達成後は、配信ポータルの主要 3 操作（配信対象の選択、配信内容の指定、実行承認）が **3 クリック以内** で完結する。主要動線ごとに想定クリック数を UI フロー図で明示し、逸脱する UI 改修はデザインレビューで差し戻す。副次操作（検索・フィルタ・履歴参照）はこの 3 クリック制約の適用外とするが、主要動線を妨げない位置に配置する。

崩れた時の影響は、業務部門が「配信ポータル経由より直接 tier3 アプリを呼ぶ方が早い」と判断し、ポータル経由の監査統合 (`SEC-AUD-*`) が迂回される。監査統合が迂回されると BIZ-LEG の免責条項の前提（配信経路の単一化）が破綻する。

**受け入れ基準**

- 主要 3 動線（配信対象指定 / 配信内容指定 / 実行承認）が 3 クリック以内で完結する UI フロー図が承認されている
- 主要動線のクリック数増加を含む PR はデザインレビュー承認必須

**検証方法**

- E2E テスト（Playwright 等）で主要動線のクリック数を自動測定し、3 を超過した時点で CI 失敗
- Phase 1b のパイロット業務利用者 10 名に対し、初回操作の完了クリック数を計測する実ユーザーテスト

### BIZ-UX-003: Backstage の UX 品質ベースライン

- 優先度: SHOULD（Phase 1b 着手時点で達成が望ましいが、Backstage 上流追従により自然達成する項目も多い）
- Phase: Phase 1b
- 関連: `BIZ-UX-001`, `DEV-DEX-*`

Backstage は OSS Graduated 済みで UX 品質は上流で一定水準に保たれるが、プラグイン自作（Scaffolder テンプレート / TechDocs / カスタムエンティティ）を増やすと品質がブレやすい。内製プラグインだけが「検索でヒットしない」「ソフトウェアカタログと連動しない」状態になると、利用者は Backstage を「雛形生成の入り口」としてしか使わなくなり、BIZ-ANL の MAU 指標が Backstage で伸びない。

要件達成後は、自作 Backstage プラグインが上流と同等の UX 品質（検索ヒット / カタログ連動 / TechDocs 埋め込み / ドラフト保存）を満たす。プラグイン開発時のレビュー観点として 4 項目のチェックリストを標準化し、PR テンプレートに組み込む。

崩れた時の影響は、内製テンプレートが Backstage の Scaffolder UI ではなく独自画面に隔離され、UI 分散が拡大する。これは BIZ-UX-001 のデザイン統一と矛盾し、2 ポータル併用固有工数 0.3 人月/年 ([`../../01_企画/04_定量試算/03_運用工数試算.md`](../../01_企画/04_定量試算/03_運用工数試算.md) 第 2.7 節) が膨張する。

**受け入れ基準**

- 自作 Backstage プラグインは検索・カタログ連動・TechDocs 埋め込み・ドラフト保存の 4 機能を備える
- プラグイン PR テンプレートに上記 4 機能のチェック項目が含まれている

**検証方法**

- 自作プラグインの Backstage 検索インデックス登録を E2E で確認
- 四半期ごとに Backstage メジャーアップデート後の UX 非回帰テストを実施

### BIZ-UX-004: エラー画面の一貫性

- 優先度: SHOULD（利用者の心理的安全性に直結するが、未達でも機能は動作する。Phase 1b 着手時点で統一）
- Phase: Phase 1b
- 関連: `BIZ-UX-001`, `COM-ERR-*`（error_taxonomy）

現状、404 / 500 / 403 / 503 は各コンポーネント（Envoy Gateway / Backstage / 配信ポータル / Grafana）がそれぞれ独自の見た目で返す。利用者は「どこで何が壊れたか」を画面から判断できず、サポート窓口 (BIZ-ONB の受け入れテスト導線 / `OPS-SUP-*`) に「画面が白くなった」という曖昧な一次連絡が集中する。

要件達成後は、全 GUI が error_taxonomy ([`../../03_要件定義/00_共通/05_error_taxonomy.md`](../../03_要件定義/00_共通/05_error_taxonomy.md)) で採番されたエラーコードを画面に表示し、問い合わせ時に「エラーコード E-AUTH-401-003 です」と即伝達できる。画面のレイアウト・トーン・サポート連絡導線（問い合わせフォーム URL / 相関 ID のコピー導線）は BIZ-UX-001 のデザインシステム上の 1 種類のテンプレートに統一する。

崩れた時の影響は、サポート一次応答時間が分類不能なチケット対応で増大し、BIZ-LEG の SLA 補償条項（月額 10% 上限）のトリガ条件を満たしやすくなる。

**受け入れ基準**

- 全 GUI のエラー画面が共通テンプレートを参照する
- エラー画面に error_taxonomy コード・相関 ID・サポート導線が必ず表示される
- OSS UI (Grafana / Keycloak) はテーマ差し込みで最低限の見た目統一を図る

**検証方法**

- E2E テストで代表的エラー（404 / 500 / 403 / 503）の画面に必須要素 3 項目（コード / 相関 ID / 導線）が存在することを検証
- サポートチケット初期受電時のエラーコード記載率を月次で追跡し、90% 以上を維持
