# BIZ-ONB: テナントオンボーディング要件

本章は、新規テナント（JTC グループの事業部 / 子会社）を k1s0 の利用者として受け入れるプロセスの要件を定義する。オンボーディングは「契約」「技術セットアップ」「受け入れテスト」「利用開始研修」の 4 段を含み、各段で担当組織が異なるため、**Runbook による手順固定とリードタイム目標** で属人化を排除する必要がある。

テナントオンボーディングの質は、プラットフォームの稟議後の実質的な採用速度を決める。初期テナント 1 社に 2 か月を要する設計では、10 社展開に 20 か月かかり、Phase 2 の全社ロールアウト目標が崩壊する。本章の MUST は **5 営業日以内の受け入れ完了** という数値目標を据え、契約・技術・研修を並列化する運用体制を条項化する。

## 要件本体

### BIZ-ONB-001: オンボーディング手順の Runbook 化

- 優先度: MUST（属人化したオンボーディングは初期テナントで発覚し稟議信頼性を損なう）
- Phase: Phase 1b
- 関連: `BIZ-ONB-002`, `BIZ-ONB-004`, `OPS-OPS-*`

現状、内製プラットフォームの新規テナント受け入れは「最初のテナントは担当者が直接対応」というアドホック運用から始まり、2 社目以降で記憶頼りの手順漏れが頻発する。k1s0 のような tier 分離前提の基盤では、namespace 作成・RBAC 設定・Keycloak クライアント登録・OpenBao 鍵設定・Kafka トピック作成・監査統合の順序を 1 箇所でも誤ると、後続のセキュリティ検査 (`SEC-*`) で全件やり直しとなる。

要件達成後は、オンボーディングの全工程が Runbook として Git 管理され、Argo Workflows または Ansible Playbook で 80% 以上が自動実行される。残り 20% の手動承認ステップ（契約印押印 / IAM 初期管理者の本人確認）は Runbook にチェックリスト形式で明記する。手順変更は PR レビューを経て反映され、運用担当者の交代時にも同一品質の受け入れが保証される。

崩れた時の影響は、Phase 2 の全社ロールアウトでテナント受け入れがボトルネック化し、BIZ-CST の FinOps 運用が「受け入れ待ち案件の滞留」で計測不能になる。BIZ-TRN の顧客ハンズオンも Runbook の不在を前提に再設計する必要が出て、追加工数 1〜2 人月を要する。

**受け入れ基準**

- オンボーディング Runbook が Git 管理され、Argo Workflows / Ansible で自動実行される
- 手動承認ステップがチェックリスト形式で Runbook に含まれる
- Runbook 変更が PR レビュー経由で行われ、過去の版が Git 履歴で追跡可能

**検証方法**

- ステージング環境で Runbook を四半期ごとにドライラン実行し、所要時間と失敗ステップを記録
- 新規テナント受け入れログと Runbook ステップの対応を月次でサンプリング監査

### BIZ-ONB-002: テナント分離設定の自動化

- 優先度: MUST（手動設定ミスは SEC-TIS のテナント分離保証を直接破壊する）
- Phase: Phase 1b
- 関連: `BIZ-ONB-001`, `QUA-TIS-*`, `SEC-IAM-*`

現状、テナント分離（namespace / NetworkPolicy / RBAC / Keycloak client / OpenBao policy / Kafka ACL）を手動で設定すると、1 要素の設定漏れで他テナントへの横断アクセスが許可される事故が発生する。[`../../01_企画/05_法務サマリ/02_日本法令適合.md`](../../01_企画/05_法務サマリ/02_日本法令適合.md) 第 3 節のマイナンバー分離構造は、手動設定では維持不能な厳格さを要求する。

要件達成後は、テナントごとの分離設定が **OpenTofu + Argo Workflows** で IaC 化され、テナント ID を入力するだけで全分離要素が自動プロビジョニングされる。プロビジョニング後、Kyverno / OPA Gatekeeper による静的ポリシー検証で分離境界の網羅性を検査する。手動変更はポリシー違反として CI で検出し、Git リポジトリを唯一の真実の源とする。

崩れた時の影響は、1 件の設定漏れが発覚した時点で全テナントの分離状態を緊急監査する必要があり、BIZ-LEG の免責条項（k1s0 側の重過失除外）の認定リスクを高める。

**受け入れ基準**

- テナント分離設定が OpenTofu モジュールとして定義され、テナント ID 入力のみで完結する
- プロビジョニング後に Kyverno / OPA で 10 項目以上のポリシー検証が実行される
- 手動設定変更が Git diff として検出され、CI で警告される

**検証方法**

- テナント間の到達性を iperf / nc / gRPC で相互試験し、許可外通信がないことを確認
- Kyverno ポリシー違反を週次スキャンし、違反 0 件を KPI とする

### BIZ-ONB-003: 受け入れテストの提供

- 優先度: SHOULD（テナント側の受け入れ判断の信頼性に直結。Phase 1b 以降で提供）
- Phase: Phase 1b
- 関連: `BIZ-ONB-001`, `DEV-TST-*`

現状、テナント側は受け入れ完了の判断基準を自前で作る必要があり、「何をもって使えるとするか」が曖昧になる。結果として「動いているように見えるけど監査ログが出ていない」状態で本番利用が始まるリスクが発生する。

要件達成後は、k1s0 が **テナント受け入れテストスイート**（疎通 / 認証 / 権限 / 監査ログ / メトリクス / ログ / トレース / 分離境界の 8 項目の自動テスト）を提供する。テナントはテストを実行して結果レポートを受領することで、自組織の業務開始判断を下せる。テストは Argo Workflows で CI に組み込まれ、k1s0 プラットフォームの変更が既存テナントに対して非回帰であることも同テストで検証する。

崩れた時の影響は、テナントごとに独自の受け入れ基準を再発明することで、BIZ-ONB-004 の 5 営業日目標が守れなくなる。

**受け入れ基準**

- 受け入れテストスイートが 8 項目（疎通 / 認証 / 権限 / 監査 / メトリクス / ログ / トレース / 分離）を含む
- テスト結果が PDF / HTML レポートで出力される
- プラットフォーム変更時の非回帰テストとして CI で実行される

**検証方法**

- 新規テナント受け入れ時に本テストを必須実行し、合格後に本番利用開始を承認
- プラットフォーム PR 時に全既存テナント分の非回帰テストを実行

### BIZ-ONB-004: オンボーディングリードタイム 5 営業日以内

- 優先度: SHOULD（定量目標として宣言。Phase 1b 後半から Phase 2 で段階的に達成）
- Phase: Phase 2
- 関連: `BIZ-ONB-001`, `BIZ-ONB-002`, `BIZ-ONB-003`

現状、内製基盤の初期テナント受け入れは 1〜2 か月が常態化しており、契約 → 技術セットアップ → 研修の直列実行が長大化する。業界平均は SaaS で 3 営業日、オンプレ基盤で 10 営業日。k1s0 は JTC 内部テナントを前提とするため、契約レビューを短縮する余地が商用 SaaS より大きい。

要件達成後は、**契約書締結 → 本番利用開始まで 5 営業日以内** を標準リードタイムとする。根拠は業界平均オンプレ 10 営業日の半減であり、Runbook 自動化 (BIZ-ONB-002) と受け入れテスト (BIZ-ONB-003) による並列化で実現可能な目標である。契約・技術・研修の各段は並列実行可能とし、各段の最大所要日数（契約 2 日 / 技術 1 日 / 受け入れテスト 1 日 / 初回研修 1 日）を Runbook に明記する。

崩れた時の影響は、Phase 2 の全社ロールアウト（年間 10 事業部受け入れ想定）で所要 100 営業日（5 か月）を超過し、経営報告の年度ロードマップが崩れる。

**受け入れ基準**

- 標準リードタイムが 5 営業日以内と Runbook に明記される
- 各段の所要日数がメトリクスとして計測・公開される
- 5 営業日超過時のエスカレーション先が定義されている

**検証方法**

- テナント受け入れ案件ごとに契約〜利用開始の実績日数を記録し、月次レポートで平均値と 95 パーセンタイルを追跡
- 超過案件は原因分類（契約 / 技術 / 研修）を記録し、四半期レビューで改善
