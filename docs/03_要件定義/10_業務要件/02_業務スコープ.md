# 業務スコープ

本書は、k1s0 が対象とする業務の範囲と、意図的に対象外とする業務を明示する。スコープ境界を曖昧にすると、稟議後に「想定していた業務 X もカバーされるはず」「業務 Y は k1s0 で吸収できないのか」の追加要求が発生し、Phase 1b パイロットの範囲判定が紛糾する。

## スコープの考え方

k1s0 は「JTC 情シスが内製運用可能なマイクロサービス基盤」であり、業務アプリそのものではない。このため、対象業務は「tier2/tier3 が k1s0 を使って業務アプリを作る・回す・運用する」という開発・運用業務に絞る。エンドユーザーの業務（申請、承認、見積、在庫管理等）は k1s0 のスコープ外であり、それらは tier2/tier3 が実装する業務アプリの要件となる。

この層分離を守ることで、k1s0 は業務ドメイン非依存の汎用基盤として、異なる部署・異なる業務アプリに横展開できる。横展開を諦めて特定業務に最適化すると、結局 1 業務用の専用ツールになり、5 年 TCO の試算前提（複数業務で共有）が崩れる。

## 対象業務（In Scope）

### 開発業務（tier2/tier3 が k1s0 を使う業務）

tier2 の業務アプリ開発者と tier3 のエンドユーザー向けアプリ開発者が、k1s0 公開 API を利用して新規業務アプリを開発・改修する業務を対象とする。具体的には以下。

- 新規マイクロサービスの立ち上げ（k1s0 雛形 CLI での scaffolding、言語選択、サービス登録）
- tier1 公開 API を用いた機能実装（State 操作、PubSub、Workflow、Decision、Log、Telemetry）
- サービス間連携（gRPC コントラクト定義、Protobuf スキーマ管理、互換性確認）
- 認証・認可（Keycloak 統合、テナント ID 検証、ロールベース認可）
- 監査ログ記録（tier1 Audit API による改ざん防止ログの生成）
- Feature Flag による段階ロールアウト（flagd バックエンドの Feature API 利用）
- ローカル開発環境の立ち上げと動作確認
- CI/CD パイプラインへの組み込み（Argo CD、コンテナイメージ署名、SBOM 生成）

### 運用業務（情シス SRE が k1s0 を運用する業務）

情シス SRE チームが k1s0 基盤の日常運用を回す業務を対象とする。

- 監視とアラート対応（Prometheus / Grafana Tempo / Loki の利用、Dead Man's Switch の維持、Severity 1〜3 の応答）
- バックアップ・リストア運用（PostgreSQL PITR、etcd 定期バックアップ、MinIO スナップショット、年 2 回の復旧訓練）
- キャパシティ管理（CPU / メモリ / ディスク使用量の傾向監視、増設計画）
- インシデント対応（15 本の Runbook に基づく Severity 別対応、ポストモーテム作成）
- OSS バージョン追従（Renovate 自動 PR、影響評価、リグレッション試験）
- 証明書ローテーション（cert-manager 自動、年次点検）
- 秘密情報ローテーション（OpenBao 動的シークレット TTL 1 時間、手動ローテーション手順の整備）
- 監査ログ長期保存（1 年 / 3 年 / 7 年の保持ポリシー適用）

### 管理業務（情シスマネージャ・監査対応）

情シスマネージャと監査担当が k1s0 基盤の継続投資判断・監査対応を行う業務を対象とする。

- 年次稟議における基盤投資継続判断（人件費、VM 増設、OSS ライセンス適合の再確認）
- 監査部門への証跡提供（Audit 改ざん防止ログ、SBOM、変更管理記録、アクセス管理記録）
- ベンダー撤退シナリオの年次点検（採用 OSS の活性度、代替案の更新）
- 法令改正への対応（個人情報保護法、J-SOX、社内規程の改訂反映）
- tier2/tier3 の新規利用部門オンボーディング（ネームスペース発行、ロール設定、SDK 配布）

## 対象外業務（Out of Scope）

### エンドユーザー業務そのもの

社員の申請業務、経費精算、見積作成、在庫管理等のエンドユーザー業務は k1s0 のスコープ外。これらは tier2/tier3 が実装する業務アプリの責務。k1s0 は業務ロジックのテンプレートを提供するが、業務そのもののフロー定義は行わない。

### 業務アプリの企画・要件定義

個別業務アプリ（例: 新しい経費精算システム）の企画・要件定義・UX 設計は k1s0 のスコープ外。ただし、業務アプリの技術スタックが k1s0 を前提とすることは推奨する（強制ではない）。

### 既存レガシーシステムの再開発

.NET Framework 4.x の既存業務システムを k1s0 ネイティブに書き換える作業は個別プロジェクトであり、本基盤要件定義の対象外。k1s0 は「共存」を保証するが、「移行」は保証しない。移行は業務アプリ側の Phase 計画で判断する。

### マルチクラウド・マルチリージョン DR

現時点のオンプレ・閉域完結方針下では、マルチリージョン DR（東京 ↔ 大阪等）と完全マルチクラウドはスコープ外。Phase 5 以降の参考検討とする。

### エンドユーザーのトレーニング

エンドユーザー向けの業務アプリ操作研修は情シスの管理業務の一部だが、本基盤要件定義では扱わない。基盤層のトレーニング（tier2/tier3 開発者向け、情シス SRE 向け）は 30_非機能要件/C_運用保守性 で個別に要件化する。

### 公開クラウド事業・商用サービス展開

k1s0 基盤を社外顧客に提供することは現時点で想定しない。社外展開は Phase 5 以降の事業判断とし、本書では要件化しない。

### AI / LLM 推論基盤

社内生成 AI の推論基盤（GPU クラスタ、モデル管理、推論サービング）は Phase 3〜4 で別プロジェクトとして検討する想定であり、本書のスコープ外。ただし、将来的な RAG 利用を見据えて CloudNativePG の pgvector 拡張は構想設計で選定済み。

## Phase 別スコープ

同じ対象業務でも、Phase によって要件水準が異なる。Phase 0〜Phase 1b が本書の厳密スコープ、Phase 2 以降は参考スコープとして記述する。

### Phase 0（企画承認）

稟議通過を目的とする。業務要件は「稟議資料に記載した約束」を満たす範囲で固定化する。実装は着手しない。

### Phase 1a（MVP-0、VM 1 台デモ）

tier1 の最小機能（State、PubSub、Log、Telemetry、Decision の 5 API）を VM 1 台のオールインワン構成で動作させ、tier2 サンプルアプリで利用できることをデモする。運用・管理業務は対象外。開発業務のみ対象。

### Phase 1b（MVP-1a、VM 3 台パイロット 1 本稼働）

tier1 の主要機能（11 API のうち Workflow、Secrets、Binding、Service Invoke、Audit、Pii、Feature 追加）を 3 台構成で動作させ、パイロット業務アプリ 1 本を実業務で稼働させる。運用業務は最低限（監視、バックアップ）を対象に入れる。管理業務は対象外。

### Phase 1c（MVP-1b、運用体制化）

Phase 1b 継続。Runbook 15 本完成、バックアップ訓練、SBOM 生成率 100%、ドキュメント鮮度 90% を達成する。運用業務を全量、管理業務の一部（監査証跡提供）を対象に入れる。

### Phase 2 以降（参考スコープ）

tier2/tier3 複数部門展開、マルチクラスタ化、管理業務の全量対象化、Phase 4 で ZEN Engine JDM エディタの Backstage プラグイン化を検討する。本書では方針のみ記述し、確定水準は Phase 1c 完了時点で改訂する。

## スコープの明示とトレース

本書で定めたスコープに基づき、以下 3 種の業務要件カテゴリに ID を採番する。

- `BR-PLATUSE-*`: tier2/tier3 開発者の利用業務要件
- `BR-PLATOPS-*`: 情シス SRE の運用業務要件
- `BR-PLATGOV-*`: 情シスマネージャ・監査の管理業務要件

スコープ外の業務に対して要件 ID を付けることは禁ずる。要件定義書から外す代わりに、対象外の理由を本書に残すことで、将来の改訂時に「なぜこの業務は含めなかったか」を辿れる状態を維持する。

### 業務要件 ID の定義

業務要件 ID は散文括弧書き方式で定義する（04_業務フロー.md:99 のガイダンスに準拠）。本節で各 ID の「業務主体・達成状態・裏付けとなる技術要件」を散文として明示し、以降の業務フロー散文や要件間参照はここで定義した意味で使う。

**tier2/tier3 開発者の利用業務要件（BR-PLATUSE-*）**

k1s0 プラットフォームの価値は開発者が短時間で tier1 API を呼び出し、業務ロジック実装に集中できることにある。tier2 開発者が SDK ワンラインで State を呼び出して永続化できる状態を達成したい（BR-PLATUSE-001）。tier3 の .NET Framework 既存資産からサイドカー経由で tier1 API を呼び出し、段階的に移行できる状態を達成したい（BR-PLATUSE-002）。開発者が言語実装の内部事情（Dapr Go SDK と Rust 自作領域の混在）を意識せず、gRPC エンドポイントと SDK クライアントのみで tier1 を利用できる状態を達成したい（BR-PLATUSE-003）。イベント駆動を使って tier2 アプリ同士が疎結合に連携し、独立してリリースできる状態を達成したい（BR-PLATUSE-004）。外部通知やファイル保管を統一インタフェースで呼び出し、個別ライブラリの作り込みを避けられる状態を達成したい（BR-PLATUSE-005）。稟議・督促・遅延処理のような長期実行プロセスが Pod 再起動で失われず継続できる状態を達成したい（BR-PLATUSE-006）。ビジネスルールの変更をコードデプロイ無しで迅速に反映できる状態を達成したい（BR-PLATUSE-007）。

**情シス SRE の運用業務要件（BR-PLATOPS-*）**

tier1 プラットフォームの稼働を安定させ、稟議約束の SLO を維持するための運用業務。SRE が Grafana ダッシュボードで p99 レイテンシ・エラーバジェット・容量指標を 1 画面で俯瞰し、異常兆候を即座に検知できる状態を達成したい（BR-PLATOPS-001）。SRE がオンコール呼び出し時に Runbook を辿って復旧手順を実行でき、MTTR を抑制できる状態を達成したい（BR-PLATOPS-002）。SRE が月次のキャパシティレビューで Phase 別見積と実測の乖離を検出し、先行発注稟議を起票できる状態を達成したい（BR-PLATOPS-003）。SRE が Dead Man's Switch でメトリクス途絶を検知し、観測系自体の障害も監視対象に含められる状態を達成したい（BR-PLATOPS-004）。SRE が Phase 1a から継続的に SLO 判定の基準データを蓄積し、パイロット合否を実測で判定できる状態を達成したい（BR-PLATOPS-005）。SRE が新機能を段階ロールアウトし、障害時には即時切戻できる状態を達成したい（BR-PLATOPS-006）。既存 IAM・監視・バックアップ基盤と共存しながら運用を始められる状態を達成したい（BR-PLATOPS-007）。

**情シスマネージャ・監査の管理業務要件（BR-PLATGOV-*）**

プラットフォーム全体のガバナンスと監査対応を担う業務。マネージャが四半期ごとに要件達成率・Phase ゲート判定・OSS ライセンス遵守状況をレビューし、次四半期の投資判断ができる状態を達成したい（BR-PLATGOV-001）。マネージャが監査対応時にトレーサビリティマトリクスを即座に出力し、企画書 → 要件 → 構想設計の対応を示せる状態を達成したい（BR-PLATGOV-002）。内部監査人が tier1 の全 API 呼び出しについて改ざん検知可能な監査ログをクエリし、不正アクセスの痕跡を追跡できる状態を達成したい（BR-PLATGOV-003）。監査人が k1s0 Audit API で全操作を記録し、7 年保存ポリシーに従って長期エビデンスを取り出せる状態を達成したい（BR-PLATGOV-004）。

## スコープ判定で迷った時の指針

対象業務と対象外業務の境界が曖昧なケースは以下の指針で判定する。この指針は企画書の事業目的（[../../01_企画/企画書.md](../../01_企画/企画書.md)）から導出した。

- **tier1 が提供する汎用機能か業務固有機能か**: 汎用なら対象、業務固有なら対象外
- **tier2/tier3 の責務で吸収できるか**: 吸収可能なら対象外、不可能なら対象
- **5 年 TCO 枠内で工数確保できるか**: 確保可能なら対象、困難なら Phase 繰り延べ
- **稟議で外部約束したか**: 約束済みなら対象、未約束なら改訂プロトコル経由で追加判断
- **既存資産との共存を崩すか**: 崩さないなら対象、崩すなら対象外または合意し直し

判定が割れた場合、[00_要件定義方針/02_ステークホルダー.md](../00_要件定義方針/02_ステークホルダー.md) のエスカレーションパスに従う。
