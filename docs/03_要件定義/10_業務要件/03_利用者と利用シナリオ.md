# 利用者と利用シナリオ

本書は、k1s0 の利用者を 3 階層（一次／二次／三次）で特定し、それぞれの代表的な利用シナリオを散文で展開する。利用シナリオは機能要件の検収条件を裏打ちする「なぜこの機能が必要か」を具体化するための文書であり、表形式ではなく物語として記述する。

## 利用者階層の考え方

k1s0 はプラットフォーム製品であるため、利用者を直接触れる度合いで 3 階層に分ける。一次利用者は k1s0 公開 API を日常的に触る層、二次利用者は k1s0 基盤そのものを運用する層、三次利用者は k1s0 の存在を意識せずに業務アプリを使う層である。各階層ごとに関心と操作スキルが異なるため、機能要件の書き方も階層を指定して記述する。

## 一次利用者（k1s0 公開 API を使う開発者）

### tier2 業務アプリ開発者

JTC の社内業務システム（人事、経費、見積、在庫、稟議等）を開発する層。言語は C#（既存）、Go、Rust、Python（将来）のいずれか。役割は業務ドメインロジックの実装で、インフラ知識は最小限に抑えたい。

典型シナリオ「経費精算アプリの新規立ち上げ」: 新規サービス起票から本稼働まで、従来 3 か月を要していた工程を k1s0 で 1 か月に短縮したい。具体的には、k1s0 雛形 CLI で Go プロジェクトを scaffold し、State API で経費申請データを Valkey に保存、PubSub API で承認通知を配信、Decision API で部門・金額に基づく承認者ルールを ZEN Engine の決定表で記述、Workflow API で承認 Saga を Temporal で管理、Audit API で全承認操作をハッシュチェーン記録する。認証は Keycloak で SSO、tenant_id クレームは tier1 が自動検証する。開発者は k8s YAML を書かず、Dapr の存在も意識しない。

典型シナリオ「既存 .NET Framework 業務システムに新規機能を追加」: 既存 ASP.NET Web Forms アプリに、新しい承認フローを Go マイクロサービスとして外付けしたい。tier3 の API Gateway 経由で .NET アプリから Go マイクロサービスを呼び出し、Go 側で k1s0 Workflow API を使う。両者の認証トークンは Keycloak で統一、監査ログは tier1 Audit API に集約される。.NET Framework 側の改修は最小限（API Gateway 呼び出しコードのみ）で済む。

### tier3 アプリチーム（配信ポータル開発者）

エンドユーザー向け UI を開発する層。MAUI（クロスプラットフォーム）、React、Blazor、C# WinForms（既存）を使う。役割は UX 設計と配信運用。

典型シナリオ「社内配信ポータルの新機能を段階ロールアウト」: 新機能を全社員 3,000 名に一斉公開するとリスクが高いため、部署単位で 10% → 50% → 100% と段階展開したい。k1s0 Feature API（flagd バックエンド）でフラグを定義し、Backstage プラグイン経由で情シス管理者が画面操作でロールアウト比率を変更できる。アラート閾値超過時に自動的に 0% に戻る circuit breaker を Feature Flag ルールとして設定する。

典型シナリオ「新規業務アプリの SSO 統合」: 新規アプリを 1 日で Keycloak に統合したい。既存 AD / LDAP との Federation は tier1 で構成済みなので、tier3 の開発者は OIDC クライアント登録と PKCE フロー実装だけで済む。個別の SAML 属性マッピングは Keycloak Realm 管理画面で設定する。

## 二次利用者（k1s0 基盤そのものを運用する層）

### 情シス SRE / 運用チーム

tier1 プラットフォームの日常運用、インシデント対応、キャパ管理を担う層。Phase 1c 以降 2〜3 名体制で 8 × 5 オンコール。

典型シナリオ「Severity 1 インシデント応答」: 深夜に PagerDuty 代替の社内 Alertmanager → メール通知が SRE 担当者に届く。担当者は Grafana ダッシュボードで影響範囲を確認（該当 namespace の Pod 状態、Kafka Consumer Lag、PostgreSQL 接続数）し、Tempo の分散トレースでエラーが発生している具体的リクエストを特定する。該当 Runbook（Runbook 15 本のうち該当するもの）に従い復旧手順を実行し、30 分以内に初動応答、4 時間以内に復旧を完了する。ポストモーテムを翌営業日に作成する。

典型シナリオ「OSS バージョン追従」: Renovate が自動で PR を作成する。SRE は CVE 情報と CHANGELOG を確認し、影響度を判定。Critical CVE の場合は 48 時間以内にステージング環境で検証し、本番に Argo Rollouts で Canary 適用する。Medium 以下の場合は月次の計画的ウィンドウでまとめて適用する。

典型シナリオ「バックアップ復旧訓練」: 半年に 1 回、PostgreSQL フルクラスタ全壊シナリオで復旧訓練を実施する。手順は Runbook に記載され、事前通告なしに担当を 1 名抽出、単独で復旧完了まで実行できるかを計測する。復旧完了時間 4 時間以内（RTO 目標）を満たせなかった場合、Runbook を更新する。

### インフラ運用（k8s / ネットワーク / ストレージ）

kubeadm クラスタ、Istio Ambient、MetalLB、Longhorn 等の基盤 OSS を実運用する層。tier1 プラットフォームチームとは責務境界を分ける。

典型シナリオ「クラスタノード増設」: VM 3 台構成から 5 台構成への拡張。インフラ運用が VM 調達と kubeadm join を実施、tier1 チームが MetalLB IP プール追加・Istio Ambient ztunnel の動作確認を行う。Longhorn はレプリケーション係数を 3 のまま増設に追従。業務停止は無し（online expansion）。

典型シナリオ「ネットワーク VLAN 設定変更」: 社内 VLAN 再編に伴う k8s Service CIDR の変更。事前にステージング環境で検証、本番変更は業務時間外のメンテナンスウィンドウ（日曜 22 時〜）で実施、1 時間以内に完了。Istio Ambient の ztunnel は無停止で切り替わる（HBONE の上位層で VLAN 非依存のため）。

## 二次利用者（情シスマネージャ）

### 情シスマネージャ

k1s0 基盤の継続投資判断と対外説明責任を持つ層。年次稟議、四半期レビュー、監査対応を回す。

典型シナリオ「年次稟議」: 前年の運用実績（SLO 達成率、インシデント件数、OSS 更新件数、人件費実績）を Grafana ダッシュボードと監査ログから集計し、稟議書の根拠セクションに貼り付ける。外部約束した SLO と実績の乖離、前年比コスト変動、次年度投資計画を説明する。OpenShift 等商用 PaaS との比較も毎年更新する（OSS 価格表、クラウドベンダーの値上げ情報）。

典型シナリオ「監査対応」: 年次情報セキュリティ監査で、監査担当から「過去 12 か月間の特権操作ログ」「tenant_id 越境アクセスの発生件数」「OSS ライセンス義務の遵守状況」「秘密情報のローテーション記録」の提出を要求される。k1s0 Audit API のハッシュチェーンログを Grafana Loki から検索エクスポートし、SBOM 一覧（Phase 1c 以降 100% 生成）を CI アーカイブから抽出して提出する。提出までのリードタイムは 1 営業日以内を目標とする。

## 三次利用者（エンドユーザー）

### 社内業務担当（エンドユーザー）

tier3 の業務アプリを日常業務で利用する社員。k1s0 の存在は認識しない。関心は業務アプリの可用性と UX。

典型シナリオ「日常業務での経費申請」: 月 1 回の経費精算申請を tier3 アプリで行う。Keycloak SSO でログイン（社員コードとパスワード、Phase 3 以降は MFA）、申請画面で経費項目を入力、添付証憑をアップロード、送信する。数秒以内に受付完了メッセージが表示され、翌日までに承認通知が届く。アプリが遅い・落ちる場合は情シスヘルプデスクに連絡する。

典型シナリオ「新機能の段階的利用」: tier3 配信チームが Feature Flag で段階ロールアウトした新機能が、自分の部署にも有効化される。画面上に新しいボタンが出現、マウスホバーでツールチップが表示される。期待通り動作しない場合は問い合わせフォームから報告する（フォームデータは Audit API でハッシュチェーン記録される）。

## 利用シナリオと機能要件の対応

本書の利用シナリオは、20_機能要件/ の個別要件 ID に対応付けられる。シナリオの記述中に登場する「k1s0 Xxx API」は、20_機能要件/10_tier1_API要件/ の該当ファイルで詳述する。対応マトリクスは [80_トレーサビリティ/03_構想設計マトリクス.md](../80_トレーサビリティ/03_構想設計マトリクス.md) を参照。

シナリオと機能要件の対応が崩れた場合（シナリオが更新されたが要件は古いまま、またはその逆）、四半期ごとの整合レビューで検出する。検出責任は起案者。

## シナリオ追加のガイドライン

本書に記述したシナリオ以外で、新規利用者や新規シナリオが発見された場合の追加ルール。

シナリオは「具体的な業務行為」として記述する。抽象的な要件（「ユーザーは X を管理できる」等）はシナリオではなく機能要件として 20_機能要件 に記述する。シナリオは 100〜300 字の散文で、登場人物・入力・処理・出力・代替フローを含める。

シナリオ追加時は、影響する機能要件 ID を本書末尾（将来）のシナリオ索引に追記する。索引は Phase 1a 着手時に整備予定。過渡期は各シナリオの末尾に「想定要件 ID: FR-T1-STATE-003 等」を括弧書きで残す。
