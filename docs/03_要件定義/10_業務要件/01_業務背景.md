# 業務背景

本書は、k1s0 を必要とする JTC 情シス部門の業務環境と痛点を散文で記述する。業務要件の根拠を可視化することで、後続の機能要件・非機能要件の優先度判定が「なぜ必要か」を根拠に戻って説明できる状態をつくる。

## JTC 情シスが置かれている現状

日本の伝統的大企業（以下 JTC）の情報システム部門は、業務システムの新規開発・改修・運用・廃棄を 20〜30 年の時間軸で内製または準委任で運営してきた。オンプレ環境の VMware + Windows Server + .NET Framework + SQL Server が典型的なスタックであり、既存業務システムは 100〜300 本規模で稼働している。これらは退役期を迎えつつあるが、業務が止められないため全量書き換えは現実的でなく、部分的に新技術を取り入れながら 10 年単位で緩やかに更新するアプローチが主流である。

新技術導入は CTO レイヤの号令で始まるが、個別業務システムレベルで見ると、稟議プロセスが年次で硬直的に回っており、中規模案件でも要件定義から稟議通過まで 3〜6 か月を要する。稟議を通すには定量試算・競合比較・撤退シナリオが必須で、1 システム 1 試算の繰り返しが部門コストを圧迫している。

ここにクラウド移行の波が押し寄せる一方、日本のデータ主権、既存資産の継続利用、閉域ネットワーク要件、コスト予測可能性の観点で、全面クラウド移行に踏み切れないケースが多い。クラウド PaaS（OpenShift、Tanzu、Anthos）の採用を試みると、ライセンス費・サポート費で年 1 億円超、5 年 TCO で 5 億円超の規模になり、稟議のハードルがむしろ高くなる。結果、多くの JTC では「クラウドにも踏み切れず、オンプレ内製の改善も滞る」状態に陥っている。

## tier2 開発者の痛み

業務アプリを開発する tier2 相当の開発チームは、以下のような日常的痛みを抱える。

マイクロサービス化の恩恵を受けたいが、k8s・Istio・Dapr・分散トレース・RBAC を個別に学習・運用する余力がない。ちょっとしたアプリでも k8s YAML を 300 行書くことになり、インフラ知識が業務ロジック実装の前提になる。サービス間連携（状態共有、イベント、ワークフロー）を自前で実装するとそれだけで数人月かかり、業務ロジックに集中できない。

.NET Framework ベースの既存資産を活かしながら、部分的に Go や Rust の新機能を導入したいが、言語をまたぐ連携規約（通信プロトコル、エラー型、監査ログ形式、認証トークン伝搬）が部署ごとにバラバラで、結合テストで毎回紛糾する。結果、新規機能は部分最適で閉じ、全社的な改善が蓄積しない。

障害時の根本原因特定に時間がかかる。分散トレースを入れていても、複数サービスにまたがるリクエストの追跡で情シス SRE と開発チームが交互に仮説を立てる社内打ち合わせが続き、MTTR が数時間単位で延びる。

## tier3 アプリチームの痛み

エンドユーザー向けの社内業務アプリを開発する層は、認証統合とアプリ配信のたびに個別対応を繰り返している。既存 AD / LDAP への SAML / OIDC 接続を新規アプリごとに設計し、部署ごとの SSO テストに 2〜3 週間を要する。エンドユーザーは業務アプリごとにログイン画面が異なる体験に戸惑い、情シスヘルプデスクへの問い合わせが増える。

アプリ配信（社内配布、バージョン管理、Feature Flag による段階ロールアウト）は個別ツールで運用しており、配信チャネルの一元化がされていない。新機能の部分公開を試みると、影響範囲の切り分けに時間がかかり、結局全社一斉リリースで事故を起こすリスクを抱える。

## 運用 SRE チームの痛み

情シス運用チームは、8 × 5 オンコールで複数業務システムの面倒を見ている。インシデント対応手順（Runbook）はシステムごとに別文書に分散し、後輩への引継ぎが属人的になる。バックアップ訓練は年 1 回の定期演習でしか回せず、本番障害で実際に復旧できるかは毎回緊張を要する。

監視基盤は Zabbix / SCOM / 独自 Python スクリプトの混在で、メトリクス・ログ・トレースの相関が取れない。監視基盤自体の障害（いわゆる watcher の watchdog 問題）を検知する手段が弱く、「誰もアラートが来ていないから正常」と判断したら実は監視が停まっていた、という事故が年に数回発生する。

## 情シスマネージャと経営層の痛み

情シスマネージャは、クラウド PaaS 採択時の年次ライセンス費（年 1〜2 億円）に説明責任を負う。「OpenShift を採用したら、アプリ開発者の生産性は上がるのか」「サブスク値上げにどう対抗するのか」「ベンダー撤退時の再移行コストは」の問いに、定量で答える材料が乏しい。

経営層は、「情シスの DX 投資」が 5 年後に何を残すかを判断しなければならない。ライセンス費はフロー費用として消え、ノウハウが社内に残らないと撤退時の退路がない。内製人材への投資は採用・育成で数年単位の時間がかかり、即効性を期待できない。この両者のジレンマのなかで、意思決定が保留になりやすい。

## k1s0 導入のトリガ

k1s0 は上記の痛みを重ね合わせた結果として企画された。要件定義書が満たすべき業務的成功条件は以下に集約される。

- **開発者体験の標準化**: tier2/tier3 の開発者が k1s0 公開 API を通じて、認証・State・PubSub・Workflow・Decision・Log・Telemetry を一貫した SDK で利用できる。インフラ知識を前提としない。
- **既存資産との共存**: .NET Framework 4.x の既存アプリをリプレースせず、サイドカー方式または API Gateway 経由で k1s0 基盤に繋げる。
- **コスト予測可能性**: 5 年 TCO 3.68 億円（中規模 3,000 名想定）を稟議で外部約束した範囲内で回す。サブスク値上げのリスクを外側に持たない。
- **運用の属人化回避**: バス係数 2（Phase 1a）/ 3 以上（Phase 2）、Runbook 充足率 15/15（Phase 1c）、SBOM 生成率 100%（Phase 1c）を指標化する。
- **監査対応の軽量化**: 監査証跡・OSS ライセンス適合・個人情報保護法対応のエビデンスが、年次監査で即提出可能な状態を維持する。

## 業務の成功／失敗イメージ

要件が達成された場合、tier2 開発者は新規業務アプリの立ち上げを 3 か月から 1 か月に短縮し、認証・State・PubSub・監査ログを個別実装せず k1s0 SDK に任せる。tier3 配信チームは Feature Flag ダッシュボードで段階的ロールアウトを運用し、部分リリース事故を年数件減らす。運用 SRE は Grafana ダッシュボードと 15 本の Runbook で Severity 1 応答 30 分・MTTR 4 時間以内を維持し、情シスマネージャは年次稟議で k1s0 基盤の継続投資を前年実績で正当化する。

要件が達成されない場合、上記のいずれかが崩れる。tier2 開発者は k1s0 API の使い勝手が悪く自前 k8s YAML に逃げ、tier3 は Feature Flag が使えず一斉リリースに戻り、運用 SRE はアラート疲労で離職率が上がり、情シスマネージャは翌年の予算で OpenShift への撤退を稟議することになる。この分岐を左右するのが本書の要件水準であり、MVP-1a 実測までに数値を再確定する約束（企画書に記載の再評価タイミング）を遵守することで、外部約束のドリフトを防ぐ。
