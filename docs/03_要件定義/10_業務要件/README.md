# 10_業務要件

本カテゴリは、k1s0 が解決する「業務」を定義する。IPA 共通フレーム 2013 の「システム要件定義プロセス」における業務要件側面に対応する。k1s0 はプラットフォーム製品であるため、ここで扱う業務は *k1s0 を使って社内業務システムを開発・運用する* 業務である。エンドユーザーの業務（申請処理、見積作成等）は k1s0 の業務要件ではなく、tier2/tier3 の業務アプリ側の要件である。

## 本カテゴリの位置付け

機能要件（20_機能要件）や非機能要件（30_非機能要件）を読む前に、なぜその機能・水準が必要なのかを本カテゴリで根拠付ける。業務要件が曖昧だと、機能要件の取捨選択と非機能水準の判断が個人の感覚に引きずられる。

業務の記述は散文を主として、登場人物と役割（tier2 開発者、tier3 アプリチーム、運用 SRE、情シスマネージャ、エンドユーザー代表）が現状どのような不便を抱え、k1s0 導入後にどう変わるかを具体的な業務シーンで展開する。業務フローは drawio 図として可視化し、機能要件との対応点を明示する。

## 業務要件 ID 体系

本カテゴリの業務要件には `BR-<カテゴリ>-<番号>` の ID を付与する。カテゴリは `PLATUSE`（開発者の利用業務）、`PLATOPS`（情シス運用業務）、`PLATGOV`（情シス管理業務）の 3 種類。ID 付与は [00_要件定義方針/05_要件トレーサビリティ方針.md](../00_要件定義方針/05_要件トレーサビリティ方針.md) に従う。

## 機能要件・非機能要件への接続

本カテゴリで特定した業務ニーズは、20_機能要件・30_非機能要件の該当要件に `BR-*-*` としてリンクされる。業務要件が未達成となる場合、紐づいた機能・非機能要件の優先度判定を再評価する。双方向トレースは [80_トレーサビリティ/02_企画要件マトリクス.md](../80_トレーサビリティ/02_企画要件マトリクス.md) で維持する。
