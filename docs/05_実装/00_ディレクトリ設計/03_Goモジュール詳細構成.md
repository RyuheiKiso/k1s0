# 03. Go モジュール詳細構成

本ファイルは `src/tier1/go/` 配下の詳細ファイル配置を方式として固定化する。上流は概要設計 [DS-SW-COMP-124〜128（Go モジュール・ビルド・Docker image）](../../04_概要設計/20_ソフトウェア方式設計/01_コンポーネント方式設計/06_パッケージ構成_Rust_Go.md)・[DS-SW-COMP-020〜022（ファサード層 5 モジュール構成）](../../04_概要設計/20_ソフトウェア方式設計/01_コンポーネント方式設計/02_Daprファサード層コンポーネント.md)・[DS-SW-COMP-111（Go 公開範囲）](../../04_概要設計/20_ソフトウェア方式設計/01_コンポーネント方式設計/05_モジュール依存関係.md)で、本ファイルは `internal/` 配下を **`pods/`（Pod 実装）と `shared/`（横断基盤）の 2 階層**に分けたうえで、各 Pod を 4 層アーキテクチャ（handler / service / domain / repository）まで分解する。

## 本ファイルの位置付け

概要設計は `internal/` 直下の package 分割（`adapter/dapr/` / `common/` / `policy/` / `proto/v1/` / `otel/` / `state/` / `secret/` / `workflow/`）を方式として確定した。本ファイルはその下の 2 階層を確定させる。第 1 階層として、Pod 固有実装（`state/` / `secret/` / `workflow/`）を `internal/pods/` 配下にまとめ、Pod 横断の基盤コード（`adapter/` / `common/` / `policy/` / `otel/` / `proto/v1/`）を `internal/shared/` 配下にまとめる。第 2 階層として、各 Pod を **handler（gRPC 入口）・service（ユースケース）・domain（ドメインモデル）・repository（Dapr Adapter 呼び出し）** の 4 層に分解する。概要設計の DS-SW-COMP-020 が「ファサード層は API Router / Policy Enforcer / Dapr Adapter / Log Adapter / Metrics Emitter の 5 モジュール」と宣言したが、これは**責務の名前**であり、**Go の package 階層**ではない。本ファイルは 5 モジュールの責務を Go package に割り付ける方式を確定する。

2 階層分離の根拠は 3 点ある。第 1 に、`shared/` は全 Pod から import される横断コードで変更頻度が低く、`pods/` は機能追加で頻繁に変更されるため、変更頻度の違いを物理配置で表現することで CODEOWNERS の分割（`shared/` はアーキテクト承認必須、`pods/` は各 Pod チームで自走）と CI の path filter が自然に噛み合う。第 2 に、Phase 2 以降で新規 Pod（例: 監査ジョブ処理専用の `auditjob/` Pod）を追加する際、`pods/` 配下に追加するだけで済み、`shared/` を書き換える必要がないため追加コストが線形に収まる。第 3 に、`go-cleanarch` の層定義で `pods/*` → `shared/*` の単方向依存を宣言でき、`shared/` が特定 Pod を import する退行（例: `shared/common` が `pods/state/domain` に依存する）を CI で機械的に遮断できる。

4 層アーキテクチャ採用の根拠は 3 点。第 1 に、Dapr SDK の呼び出し（repository 層）を domain から分離することで、unit test で Dapr を丸ごと mock できる。第 2 に、handler を薄く保ち（gRPC リクエスト/レスポンスの変換のみ）、ビジネスロジックを service に寄せることで、API 契約変更時の影響を handler 層に閉じ込める。第 3 に、domain をプレーンな Go 構造体（Dapr 型に依存しない）で定義することで、Pod 間で共有できるドメイン語彙を育てやすい。

### DS-IMPL-DIR-041 Go モジュール root の構成

`src/tier1/go/` 直下には `go.mod` / `go.sum` / `README.md` / `Dockerfile.template` の 4 ファイルと、`cmd/` / `internal/` / `pkg/` / `integrationtest/` の 4 ディレクトリを配置する。それ以外のファイルを root に置くことは禁止する。`Dockerfile` は各 Pod 固有ではなく template として root に置き、`cmd/<pod>/Dockerfile` は config レイヤを加えた Pod 別 Dockerfile を配置する（DS-IMPL-DIR-049 参照）。

**確定フェーズ**: Phase 1a。**対応要件**: DX-CICD-\*、NFR-F-ENV-\*。**上流**: DS-SW-COMP-124、DS-SW-COMP-128。

### DS-IMPL-DIR-042 go.mod の module path

`go.mod` の module path は `github.com/k1s0/k1s0/src/tier1/go` とする。Phase 1b で正式なリポジトリ名が確定した時点（GitHub organization 名が最終化する）で書き換える。書き換えは全 Go ソースの import path に影響するため、ADR 起票と専用の git commit で一度に実行する（`gofmt -r` ではなく `goimports --replace` で安全に一括置換）。

**確定フェーズ**: Phase 1a（暫定）、Phase 1b（確定）。**対応要件**: DX-CICD-\*、NFR-C-NOP-001。**上流**: DS-SW-COMP-124、DS-SW-COMP-138。

### DS-IMPL-DIR-043 go.work の非使用

Go Workspace（`go.work`）は Phase 1a では使用しない。理由は `tools/` 配下の Go ツールと `src/tier1/go/` を 1 つの workspace に束ねると、依存バージョンが相互に影響して `src/tier1/go` のビルドが tools の破壊的な依存更新で壊れるためである。`tools/<name>/go.mod` と `src/tier1/go/go.mod` は独立に `go mod tidy` を実行する運用とし、`go.work` は Phase 2 以降で「複数 tier の同時開発が必要になった」場合に再評価する。

**確定フェーズ**: Phase 1a（非使用）、Phase 2（再評価）。**対応要件**: NFR-C-NOP-001、DX-CICD-\*。**上流**: DS-IMPL-DIR-008。

## cmd/ 配下の main.go

### DS-IMPL-DIR-044 cmd/<pod>/main.go の骨格

各 Pod の `main.go` は `cmd/<pod>/main.go` に配置し、100 行以内に収める。ロジックは書かず、以下の 7 ステップの起動シーケンスのみを記述する。

```go
// cmd/state/main.go の骨格（実装例ではなく構造のみ）
import (
    "github.com/k1s0/k1s0/src/tier1/go/internal/pods/state"
    "github.com/k1s0/k1s0/src/tier1/go/internal/shared/common"
    "github.com/k1s0/k1s0/src/tier1/go/internal/shared/dapr"
    "github.com/k1s0/k1s0/src/tier1/go/internal/shared/log"
    "github.com/k1s0/k1s0/src/tier1/go/internal/shared/otel"
    "github.com/k1s0/k1s0/src/tier1/go/internal/shared/policy"
)

func main() {
    cfg := common.LoadConfigFromEnv()                 // 1. 設定ロード
    ctx := context.Background()
    otelShutdown := otel.Init(ctx, cfg.PodName)       // 2. OTel 初期化
    defer otelShutdown()
    logger := log.Init(cfg.LogLevel)                  // 3. Logger 初期化
    policyEnforcer := policy.New(cfg, logger)         // 4. Policy Enforcer
    daprAdapter := dapr.New(cfg, logger)              // 5. Dapr Adapter
    app := state.Wire(cfg, logger, policyEnforcer, daprAdapter) // 6. Wire DI
    app.Run(ctx)                                      // 7. gRPC サーバ起動（blocking）
}
```

`main()` 内で初期化する依存は `cfg` / `otel` / `log` / `policy` / `dapr` の 5 つに限定し、それ以外は Wire の生成コードに任せる。`app.Run()` は graceful shutdown ハンドラ（SIGTERM で 30 秒以内に drain）を内部に持つ。import path は全て `internal/shared/` または `internal/pods/<pod>/` から解決し、`cmd/` の main 以外からの import 禁止を Go compiler の internal 可視性で強制する。

**確定フェーズ**: Phase 1a。**対応要件**: DX-GP-\*、DX-TEST-\*、NFR-A-\*。**上流**: DS-SW-COMP-125、DS-SW-COMP-020。

### DS-IMPL-DIR-045 cmd/<pod>/ の Dockerfile と起動スクリプト

各 Pod ディレクトリには `main.go` 以外に `Dockerfile` と `entrypoint.sh` を配置する（オプション）。`Dockerfile` は root の `Dockerfile.template` を継承して Pod 固有の config を注入する。`entrypoint.sh` は配置しないことを原則とし、`main.go` の `main()` で直接起動する。理由はシェルの介在が graceful shutdown の signal 伝搬を妨げるためと、bash 依存が distroless container と相性が悪いためである。

**確定フェーズ**: Phase 1a。**対応要件**: DX-CICD-\*、NFR-F-ENV-\*。**上流**: DS-SW-COMP-128。

## internal/shared/ の adapter 系配置

### DS-IMPL-DIR-046 shared/dapr/ の構造

`internal/shared/dapr/` は Dapr Go SDK をラップする adapter 群を配置する。Dapr SDK の型を `shared/dapr/` の外に漏らさず、全ての Dapr 呼び出しをここを経由させる。ファイル分割は Dapr Building Block ごと（`state.go` / `pubsub.go` / `binding.go` / `secrets.go` / `invoke.go` / `workflow.go`）とし、各ファイルは 200 行以内に収める。root の `dapr.go` には Adapter interface と Factory を定義し、3 Pod が共通 interface で Adapter を受け取れるようにする。

```go
// shared/dapr/dapr.go の interface 骨格
type Adapter interface {
    State() StateAdapter
    PubSub() PubSubAdapter
    Binding() BindingAdapter
    Secrets() SecretsAdapter
    Invoke() InvokeAdapter
    Workflow() WorkflowAdapter
}
```

interface は Pod ごとに必要なサブセットのみ使う（例: SECRET Pod は `State()` / `PubSub()` を呼ばない）。未使用サブセットの存在自体は許容する（Pod 分離の厳密化は Phase 2 以降）。`internal/pods/*/repository/` からのみ `shared/dapr/` を import でき、`internal/pods/*/service/` / `handler/` / `domain/` からの直接 import は go-cleanarch で遮断する。

**確定フェーズ**: Phase 1a。**対応要件**: DX-TEST-\*、NFR-C-NOP-002。**上流**: DS-SW-COMP-020、DS-SW-COMP-024、DS-SW-COMP-126。

### DS-IMPL-DIR-047 shared/log/ の配置

`internal/shared/log/` は `zerolog` をラップして k1s0 共通のログフォーマット（JSON / 構造化 / trace_id 埋め込み）を強制する。配下は `log.go`（Logger interface）、`zerolog.go`（zerolog 実装）、`noop.go`（テスト用 no-op 実装）、`log_test.go` の 4 ファイル構成で、総 200 行以内。Logger は Pod 起動時に 1 インスタンス生成し、全 handler / service / repository で context.Context 経由で伝搬する。

**確定フェーズ**: Phase 1a。**対応要件**: FR-T1-LOG-\*、NFR-D-MON-\*。**上流**: DS-SW-COMP-020、DS-SW-COMP-126。

### DS-IMPL-DIR-048 shared/metrics/ の配置

`internal/shared/metrics/` は `prometheus/client_golang` をラップして k1s0 共通の metric 命名規約（`k1s0_request_duration_seconds` など）を強制する。配下は `metrics.go`（Emitter interface）、`prometheus.go`（Prometheus 実装）、`noop.go`、`metrics_test.go` の 4 ファイル構成。Metric の命名規約は `k1s0-otel` 共通ライブラリ（`internal/shared/otel/`）と整合を取り、全 Pod で同じ名称を使う。

**確定フェーズ**: Phase 1a。**対応要件**: FR-T1-TELEMETRY-\*、NFR-D-MTH-\*。**上流**: DS-SW-COMP-020、DS-SW-COMP-109。

### DS-IMPL-DIR-049 shared/flagd/ の配置と Feature API 専用化

`internal/shared/flagd/` は flagd gRPC クライアントを配置する。Feature API が Dapr BB ではなく flagd 直接呼び出しを使うため（DS-SW-COMP-027）、`shared/dapr/` とは別ディレクトリにする。Phase 1c で flagd を導入するまではディレクトリのみ作成し（`.keep` ファイルで占有）、Phase 1c 時点で `flagd_client.go` を追加する。Phase 1a / 1b の段階でディレクトリ構造だけ用意しておくことで、Phase 1c の追加が既存構造を壊さず追加的に入る。

**確定フェーズ**: Phase 1a（ディレクトリのみ）、Phase 1c（実装追加）。**対応要件**: FR-T1-FEATURE-\*、DX-FM-\*。**上流**: DS-SW-COMP-027、ADR-FM-001。

## internal/shared/ の共通ライブラリ

### DS-IMPL-DIR-050 shared/common/ の配置

`internal/shared/common/` は `k1s0-common` ライブラリ（DS-SW-COMP-108）の Go 実装を配置する。配下は責務ごとに単一ファイル（`id.go` / `clock.go` / `config.go` / `context.go` / `error.go`）とし、各ファイルは 150 行以内。`id.go` は `TenantID` / `UserID` の型と生成関数、`clock.go` は時刻抽象（testclock で差し替え可能）、`config.go` は viper ラッパ、`context.go` は context.Context に tenant_id / trace_id を埋め込む拡張、`error.go` は共通エラー型を定義する。各ファイルに隣接して `*_test.go` を配置する。全 Pod の全層から import される最頻参照 package であるため、破壊的変更は ADR 必須とし、deprecation 期間を最低 1 マイナー版設ける。

### DS-IMPL-DIR-051 shared/policy/ の配置

`internal/shared/policy/` は `k1s0-policy` ライブラリ（DS-SW-COMP-110）の Go 実装を配置する。サブディレクトリで責務分割する。

- `shared/policy/jwt/` — JWT 検証（go-jwt ラッパ）
- `shared/policy/tenant/` — Tenant 境界確認（`tenant_id` 抽出・検証）

- `shared/policy/ratelimit/` — Rate Limit（token bucket、Phase 1b）

root の `policy.go` に `PolicyEnforcer` interface を定義し、`Authenticate` / `Authorize` / `RateLimit` / `Idempotent` の 4 メソッドを公開する。Phase 1a では `Authenticate` / `Authorize`（tenant 境界まで）のみ実装、`RateLimit` は Phase 1b で追加、`Idempotent` は Phase 1c で追加する。

**確定フェーズ**: Phase 1a/1b/1c。**対応要件**: NFR-E-AC-\*、DX-GP-\*。**上流**: DS-SW-COMP-110、DS-SW-COMP-020。

### DS-IMPL-DIR-052 shared/otel/ の配置

`internal/shared/otel/` は `k1s0-otel` ライブラリ（DS-SW-COMP-109）の Go 実装を配置する。配下は `otel.go`（初期化）、`trace.go`（span ユーティリティ）、`metric.go`（meter ユーティリティ）、`otel_test.go` の 4 ファイル構成で、総 250 行以内。OTel SDK（`go.opentelemetry.io/otel`）と Prometheus exporter を内包し、Pod 起動時に `otel.Init(ctx, pod_name)` で一括初期化する。span 名・metric 名の命名規約は本ディレクトリで強制する。

## internal/pods/<pod>/ の 4 層アーキテクチャ

### DS-IMPL-DIR-054 4 層の責務境界

各 Pod の `internal/pods/<pod>/` 配下は `handler/` / `service/` / `domain/` / `repository/` の 4 層で統一する。各層の責務は以下に固定する。

| 層 | 責務 | 依存方向 | 型 |
|---|---|---|---|
| handler | gRPC リクエスト/レスポンスの変換・gRPC metadata 抽出・エラーを google.rpc.Status に変換 | service を呼ぶ | proto 型 |
| service | ユースケースの実装・Policy Enforcer 呼び出し・複数 repository の orchestration | domain / repository を使う | domain 型 |
| domain | ドメインモデル（State Item / PubSub Event 等）・不変条件の表現 | なし（pure） | 自前型 |
| repository | Dapr Adapter 呼び出し・エラーの domain エラーへの変換 | domain 型を返す | Dapr → domain |

依存方向は `handler → service → {domain, repository}` の単方向で、逆方向（domain が service を呼ぶ等）は禁止する。さらに、`internal/pods/<pod>/` から `internal/pods/<other>/` への直接 import を全禁止し、Pod 間通信は必ず gRPC 経由で行う（Pod 独立性の物理的強制）。これらの制約は `go-cleanarch` で CI 検証する（DS-SW-COMP-113、DS-IMPL-DIR-074）。

**確定フェーズ**: Phase 1a。**対応要件**: DX-TEST-\*、NFR-C-NOP-002。**上流**: DS-SW-COMP-020、DS-SW-COMP-021、DS-SW-COMP-022、DS-SW-COMP-111。

### DS-IMPL-DIR-055 handler/ の配置

`internal/pods/<pod>/handler/` は gRPC handler を配置する。ファイルは API 単位で分割し、各ファイルは 200 行以内に収める。例えば COMP-T1-STATE は `state_handler.go` / `pubsub_handler.go` / `binding_handler.go` / `invoke_handler.go` / `feature_handler.go` の 5 ファイルを持つ。各 handler は Proto 定義の `ServiceName_Server` interface を実装し、内部では以下の 4 ステップを実行する。

1. proto リクエストを domain 型に変換（`fromProto()`）
2. service メソッドを呼び出す
3. domain レスポンスを proto 型に変換（`toProto()`）
4. エラーを gRPC status に変換（`toGRPCStatus()`）

handler にビジネスロジックを書くことは禁止する。条件分岐は「リクエスト型の nil チェック」と「エラー変換」のみで、それ以上のロジックは service に移す。handler は `shared/proto/v1/` を import するが、`shared/dapr/` を import することは禁止する（repository 層への迂回を防ぐ）。

**確定フェーズ**: Phase 1a。**対応要件**: DX-TEST-\*、NFR-C-NOP-002。**上流**: DS-SW-COMP-020、DS-SW-COMP-023。

### DS-IMPL-DIR-056 service/ の配置

`internal/pods/<pod>/service/` はユースケース層を配置する。ファイルは handler と 1:1 対応で分割し、各ファイルは 250 行以内。service のメソッドシグネチャは `func (s *StateService) Get(ctx context.Context, key TenantKey) (StateItem, error)` のように proto 型ではなく domain 型を受け取る。`ctx` から Policy Enforcer を呼び出して認可 / tenant 境界 / rate limit を適用する。service は複数の repository を orchestrate できるが、他 Pod の service を直接呼ぶことは禁止（DS-SW-COMP-100〜103 の依存方向）。service は `shared/policy/` と `shared/common/` を import してよいが、`shared/dapr/` は import 禁止（repository 経由のみ）。

**確定フェーズ**: Phase 1a。**対応要件**: DX-TEST-\*、NFR-C-NOP-002、NFR-E-AC-\*。**上流**: DS-SW-COMP-020、DS-SW-COMP-100〜103。

### DS-IMPL-DIR-057 domain/ の配置

`internal/pods/<pod>/domain/` はドメインモデルと不変条件を配置する。ファイルはドメイン概念単位で分割し、各ファイルは 150 行以内。例えば COMP-T1-STATE は `state_item.go`（StateItem 型）/ `pubsub_event.go`（PubSubEvent 型）/ `binding_invocation.go`（BindingInvocation 型）/ `consistency.go`（ConsistencyLevel enum）を持つ。domain は Dapr SDK 型・proto 型・SQL drive 型のいずれにも依存しない pure な Go 構造体とメソッドのみを含む。これにより domain の unit test は mocking 不要で高速に回せる。domain は `shared/common/` のみ import でき、`shared/dapr/` / `shared/proto/` / `shared/policy/` からの import は go-cleanarch で遮断する。

**確定フェーズ**: Phase 1a。**対応要件**: DX-TEST-\*、NFR-C-NOP-002。**上流**: DS-SW-COMP-020、DS-SW-COMP-025。

### DS-IMPL-DIR-058 repository/ の配置

`internal/pods/<pod>/repository/` は永続化層を配置する。各 repository は `shared/dapr/` の Adapter を呼び出して domain 型を返す。ファイル分割は handler / service と 1:1 対応で、各ファイルは 200 行以内。例えば COMP-T1-STATE は `state_repo.go`（`state` Adapter 呼び出し）、`pubsub_repo.go`（`pubsub` Adapter 呼び出し）、`binding_repo.go`（`binding` Adapter 呼び出し）を持つ。repository は Dapr SDK の error を domain error（`shared/common/error.go` で定義）に変換する責務を持つため、Dapr error 型を service 層に漏らさない。repository は `shared/dapr/` を import できる唯一の層である。

**確定フェーズ**: Phase 1a。**対応要件**: DX-TEST-\*、NFR-C-NOP-002、NFR-A-CONT-003。**上流**: DS-SW-COMP-020、DS-SW-COMP-025。

### DS-IMPL-DIR-059 wire.go と wire_gen.go の配置

各 Pod の `internal/pods/<pod>/wire.go` に Google Wire の Provider 宣言を記述し、`go generate` で `wire_gen.go` を生成する。`wire.go` は `//go:build wireinject` ビルドタグ付きで、本番ビルドには含まれない。`wire_gen.go` は生成コードで、git 管理し、`wire.go` の変更時に CI で再生成 drift を検出する。Wire を使うことで runtime reflection を避け、起動時間短縮と compile-time での依存整合検証を実現する（DS-SW-COMP-022）。`wire.go` は `shared/dapr` / `shared/policy` / `shared/log` / `shared/common` を Provider として束ね、`pods/<pod>/repository` / `service` / `handler` を順次組み立てる。

**確定フェーズ**: Phase 1a。**対応要件**: DX-GP-\*、NFR-A-\*。**上流**: DS-SW-COMP-022。

### DS-IMPL-DIR-074 go-cleanarch の設定配置

`go-cleanarch` の設定は `src/tier1/go/.go-cleanarch.yml` に配置する。層の定義は 2 段階で宣言する。第 1 段階は `internal/` 内の大枠依存方向として `cmd` → `internal/pods/*` → `internal/shared/*` の単方向を宣言し、`shared → pods` の逆方向依存を遮断する。加えて `internal/pods/<pod>/` から `internal/pods/<other>/` への Pod 間直接 import も遮断する（全て gRPC 経由）。第 2 段階は `internal/pods/<pod>/` 配下の 4 層依存方向（`handler → service → {domain, repository}`、`service → repository`、`domain → なし`）を Pod 共通設定として宣言する。違反は PR で fail させる。

```yaml
# 設定イメージ（.go-cleanarch.yml）
layers:
  - name: cmd
    depends_on: [pods]
  - name: pods
    depends_on: [shared]
    forbidden_cross_package: true  # pods/<pod> から pods/<other> への import 禁止
  - name: shared
    depends_on: []                   # shared は何にも依存しない
  - name: pod_internal
    scope: "internal/pods/*/"
    sublayers:
      - handler
      - service
      - {domain, repository}
```

**確定フェーズ**: Phase 1a。**対応要件**: DX-CICD-\*、NFR-C-NOP-002。**上流**: DS-SW-COMP-113、DS-IMPL-DIR-054。

### DS-IMPL-DIR-075 依存方向検査スクリプト

DS-SW-COMP-113 の `tools/check-deps.sh` に相当する検査を `tools/check-deps/` として Go 実装で配置する。`go list -deps -json ./...` で全 package の依存を取得し、DS-SW-COMP-100〜103 の依存方向ルール違反を検出する。具体的には以下 3 観点を検査する。

1. Pod 間横断禁止: `internal/pods/secret/*` が `internal/pods/state/*` を import していないか
2. shared 逆流禁止: `internal/shared/*` が `internal/pods/*` を import していないか
3. 4 層内部逆流禁止: `internal/shared/dapr/*` を `internal/pods/*/service/*` や `handler/*` が import していないか（repository のみ許可）

違反時は exit code 1 で CI を fail させる。go-cleanarch で表現しづらい「Pod 名一致での import 制約」は本スクリプトで補う。

**確定フェーズ**: Phase 1a。**対応要件**: DX-CICD-\*、NFR-C-NOP-002。**上流**: DS-SW-COMP-113、DS-SW-COMP-100〜103。

## CODEOWNERS と運用

### DS-IMPL-DIR-076 Go モジュールの CODEOWNERS

`src/tier1/go/` 配下の変更は `@k1s0/tier1-go-team` が responsible とする。`internal/shared/` 配下の横断モジュール（`dapr/` / `common/` / `policy/` / `otel/` / `proto/v1/` 等）は `@k1s0/tier1-architects` も承認者に加える（アーキテクチャ影響が大きく、全 Pod を破壊する変更経路となるため）。`internal/pods/<pod>/` は `@k1s0/tier1-go-team` のみで承認可能とし、Pod チームが自走できるよう権限を分散する。Phase 1a 時点では tier1-go-team が 1 〜 2 名で回すが、CODEOWNERS の構造は Phase 2 を見越して分けておく（将来 Pod 別チームへ分割する際に pods/<pod>/ のセクションを追記するだけで済むため）。

**確定フェーズ**: Phase 1a。**対応要件**: NFR-C-NOP-001、DX-CICD-\*。**上流**: DS-IMPL-DIR-018。

### DS-IMPL-DIR-077 ファイル命名規約（Go 特有）

Go のファイル命名は snake_case（例: `state_handler.go`、`pubsub_service.go`）。理由は Go の慣例（`http_server.go`、`tcp_listener.go`）に従うことと、テストファイル（`*_test.go`）の命名規約と整合させるためである。ファイル名に UpperCamelCase（`StateHandler.go`）や kebab-case（`state-handler.go`）を使うことは禁止する。

**確定フェーズ**: Phase 1a。**対応要件**: NFR-C-NOP-002。**上流**: DS-IMPL-DIR-024。

### DS-IMPL-DIR-078 package 名命名規約

Go の package 名は 1 単語の小文字（例: `handler`、`service`、`domain`、`repository`、`dapr`、`jwt`）を原則とする。2 単語以上が必要な場合は単語を連結（`ratelimit`、`idempotency`）するか、サブパッケージに分割する。アンダースコアを package 名に含めないこと（Go の慣例違反）。`pods` / `shared` という集約ディレクトリは package 名としては使用せず、ディレクトリ名のみの役割とする（`pods/state/handler` の import path は末尾の `handler` が package 名となる）。

**確定フェーズ**: Phase 1a。**対応要件**: NFR-C-NOP-002。**上流**: DS-IMPL-DIR-024。

### DS-IMPL-DIR-079 internal 可視性の活用

`internal/` ディレクトリは Go compiler が強制する公開範囲制限で、`src/tier1/go/internal/` 配下の package は `src/tier1/go/` 外からは import できない（DS-SW-COMP-111）。このため tier2 / tier3 向け公開 API は `pkg/` に置き、内部実装は全て `internal/` に閉じる。`pkg/` にビジネスロジックが漏れていないかを CI で検査する（Phase 1b の `tools/check-deps` に追加）。`pkg/k1s0sdk/` は `internal/shared/proto/v1/` からの型を再 export するが、`internal/pods/*` の domain / service 型は一切公開しない（Pod 実装の変更が tier2/3 に波及することを防ぐため）。

**確定フェーズ**: Phase 1a（internal 強制）、Phase 1b（pkg 検査）。**対応要件**: NFR-C-NOP-002。**上流**: DS-SW-COMP-111。

### DS-IMPL-DIR-080 Pod 追加時の手続

新規 Pod 追加は DS-SW-COMP-117 の依存審査に加え、本ファイルの 4 層構造を必ず採用する。新 Pod `<newpod>` は以下の手順で追加する。

1. `cmd/<newpod>/main.go` を 100 行以内で作成
2. `internal/pods/<newpod>/` に handler / service / domain / repository を作成
3. `internal/pods/<newpod>/wire.go` と `wire_gen.go` を生成
4. `integrationtest/<newpod>/` に結合テストを追加
5. `.github/workflows/pr-go.yml` の paths に `src/tier1/go/cmd/<newpod>/**` と `src/tier1/go/internal/pods/<newpod>/**` を追加
6. `CODEOWNERS` に `/src/tier1/go/cmd/<newpod>/` と `/src/tier1/go/internal/pods/<newpod>/` を追加
7. ADR 起票（`docs/02_構想設計/adr/ADR-TIER1-NNN-*.md`）

新 Pod 追加は `internal/shared/` を変更しない（変更する場合は別 PR で ADR 必須）。この手続を厳守することで、Pod 追加が線形コストで収まり、既存 Pod への副作用を遮断する。

**確定フェーズ**: 各 Phase（新 Pod 追加時）。**対応要件**: NFR-C-NOP-001、DX-CICD-\*。**上流**: DS-SW-COMP-117、DS-SW-COMP-138。
