# 07. 観測性配置

本ファイルは `infra/observability/` 配下の配置を確定する。LGTM スタック（Loki / Grafana / Tempo / Mimir）+ Pyroscope + OpenTelemetry Collector を `k1s0-obs` namespace に統合配置する。

## 観測性 4 signals

k1s0 は Log / Trace / Metrics / Profile の 4 signals を一元管理する。

- **Log**: Loki に集約。構造化 JSON ログ。保持 30 日
- **Trace**: Tempo に集約。OpenTelemetry trace。保持 14 日
- **Metrics**: Mimir に集約。Prometheus 互換。保持 1 年
- **Profile**: Pyroscope に集約。Continuous profiling。保持 14 日

これらは全て OpenTelemetry Collector が中継し、export するバックエンドを中央 Gateway で切り替え可能にする（Tempo 停止時は S3 dump に迂回等）。

## otel-collector/ の agent + gateway 分離

OpenTelemetry Collector を 2 層で配置する。

- **collector-agent**（DaemonSet）: 各 Node に配置、Pod ログ / metrics / trace / profile を収集、Gateway に転送
- **collector-gateway**（Deployment, HA）: agent からデータを受け、ルーティング・サンプリング・PII 除去を行い、Loki / Tempo / Mimir / Pyroscope に export

この 2 層化により、Node 側はシンプルな agent で済み、ルーティングロジックは Gateway に集約される。バックエンド切替（例: Tempo → Jaeger）も Gateway 側の設定変更だけで済む。

### processors/

- **k8s-attributes**: Pod namespace / pod-name / container-name / labels を自動付与
- **pii-redact**: 電話番号・メール・マイナンバーらしき文字列を正規表現でマスキング
- **sampling**: tail-based sampling（エラー全採取、正常系はサンプリング）

## loki / tempo / mimir の MinIO バックエンド

3 つともストレージバックエンドは MinIO Tenant `observability`（`infra/data/minio/tenant/observability.yaml`）を共通利用する。コスト効率と運用容易性を両立。保持期間は compactor で制御:

- Loki: 30 日
- Tempo: 14 日
- Mimir: 1 年（長期保持）

## grafana/ の構造

### datasources

4 つのデータソース（Loki / Tempo / Mimir / Pyroscope）を ConfigMap で宣言的管理。

### dashboards

主要ダッシュボード:

- **tier1-overview**: 11 公開 API の QPS / エラー率 / レイテンシ
- **tier2-services**: ドメインサービスの RED metrics
- **data-layer**: PostgreSQL / Kafka / Valkey / MinIO のリソース利用率
- **sli-slo-dashboard**: SLI / SLO / Error Budget
- **dapr-control-plane**: Dapr operator / placement / sentry の動作状況
- **istio-ambient**: ztunnel / waypoint の通信統計

ダッシュボード JSON は Git 管理、GitOps で自動デプロイ。

## pyroscope/ の Continuous Profiling

Pyroscope は eBPF collector で全 Pod の CPU / memory profile を常時収集。tier1 Rust 層 / Go 層のパフォーマンス問題を「事後分析ではなく常時監視」で検出可能。NFR-B-PERF-\* の根拠収集に貢献。

## alertmanager/ と oncall 連携

Mimir の alerting rules が発火した際、Alertmanager が以下の receiver に振り分ける。

- **pagerduty**: Sev1 / Sev2（オンコール呼び出し）
- **slack-oncall**: Sev3（Slack 通知のみ）
- **mailto-sre**: Sev4（メール、情報記録のみ）

`ops/oncall/` の運用手順と連動する。

## OpenTelemetry セマンティック規約

tier1 / tier2 / tier3 は全て OpenTelemetry 規約に従う属性を発出する。

- `service.name`: サービス名
- `service.version`: バージョン
- `deployment.environment`: dev / staging / prod
- `k1s0.tier`: tier1 / tier2 / tier3
- `k1s0.api.name`: tier1 公開 API 名（State / PubSub 等）

これらは tier1 / tier2 SDK で自動付与される（`src/sdk/go/otel/` 等）。
