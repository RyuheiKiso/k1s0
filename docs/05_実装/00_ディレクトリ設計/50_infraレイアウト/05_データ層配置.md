# 05. データ層配置

本ファイルは `infra/data/` 配下の配置を確定する。CloudNativePG（PostgreSQL）、Strimzi Kafka、Valkey、MinIO の 4 Operator を `k1s0-data` namespace に配置する。

## データ層基盤の役割

Dapr Component（`infra/dapr/components/`）の backing store として、永続データを保持する実体。tier1 公開 API が Dapr 経由で間接的に利用するため、アプリケーションコードはこれらのデータベース種別を意識しない。

- **CloudNativePG**: RDB として state（tier1 State API）、tier2 ドメインデータ、監査ログを保持
- **Strimzi Kafka**: PubSub（tier1 PubSub API）とイベントストリーム
- **Valkey**: 短時間キャッシュ、分散ロック、Pub/Sub 代替（短期用途）
- **MinIO**: Object storage（tier1 Binding API の S3 互換先、ログアーカイブ）

## cloudnativepg/ の 3 Cluster 分離

PostgreSQL クラスタを 3 個に分ける理由:

- **tier1-state**: tier1 State API の backing store。頻繁な読み書き、小サイズ
- **tier2-domain**: tier2 ドメインデータ。テーブル設計は各 bounded context 毎に schema で分離
- **audit**: 監査ログ。insert-heavy、削除禁止、長期保持

これらは同じ Operator で管理されるが、独立した Cluster として稼働する。バックアップ戦略・インスタンス数・リソース上限を独立制御できる。

### Cluster CRD の例

```yaml
# cloudnativepg/cluster/tier1-state.yaml
apiVersion: postgresql.cnpg.io/v1
kind: Cluster
metadata:
  name: tier1-state
  namespace: k1s0-data
spec:
  instances: 3
  postgresql:
    parameters:
      shared_buffers: 2GB
      max_connections: "200"
  storage:
    size: 100Gi
    storageClass: longhorn-retain
  backup:
    barmanObjectStore:
      destinationPath: s3://k1s0-pg-backup/tier1-state
      endpointURL: https://minio.k1s0-data.svc.cluster.local:9000
      s3Credentials:
        accessKeyId:
          name: minio-pg-backup
          key: access-key
        secretAccessKey:
          name: minio-pg-backup
          key: secret-key
    retentionPolicy: "30d"
```

## kafka/ の構造

### operator/ と cluster/

Strimzi Operator で Kafka CRD を管理。Kafka CRD は KRaft モード（Zookeeper 廃止）を採用、broker 3 個で起動。

### topics/

Kafka Topic を KafkaTopic CRD で宣言的に管理。Topic 名は業務ドメイン単位（`audit-events` / `feature-updates` / `domain-events.<bounded-context>`）。

### users/

KafkaUser CRD で ACL 管理。mTLS 認証のみ許可、SASL は不採用。tier1 AuditPii は producer 権限のみ、tier2 writer は特定 topic への producer 権限のみ等、最小権限で絞る。

### mirror-maker/

Phase 2 でマルチリージョン Kafka 間レプリケーションを導入予定。Phase 1 では空。

## valkey/ の役割

Redis OSS の fork である Valkey を採用。BSD-3-Clause ライセンスで OSS として維持される保証がある。

- Dapr State Store（短 TTL キャッシュ）
- tier3 BFF のセッションキャッシュ
- tier1 / tier2 の分散ロック（redlock 代替）

Cluster モード（6 node: master 3 + replica 3）で HA 化。

## minio/ の 3 Tenant 分離

MinIO は Tenant 単位で分離。

- **tier1-binding**: tier1 Binding API で使う S3 互換 object store（ファイルアップロード等）
- **audit-archive**: 監査ログの長期保存（Compliance / Legal 要件）
- **observability**: Loki / Tempo / Mimir のバックエンドストレージ

各 Tenant は独立した MinIO Pod セットで稼働、バケットライフサイクル（古いオブジェクトの削除 / archive tier 移動）を `buckets/bucket-lifecycle.yaml` で管理。

## HA / リソース配分

Phase 1b での最小構成:

- CloudNativePG: 各 Cluster 3 replica、total 9 Pod
- Kafka: 3 broker
- Valkey: 6 node
- MinIO: 各 Tenant 4 Pod（EC:2+2）、total 12 Pod

Phase 1c でスケールアウト（CloudNativePG replica 5、MinIO Tenant EC:6+3 等）を検討。
