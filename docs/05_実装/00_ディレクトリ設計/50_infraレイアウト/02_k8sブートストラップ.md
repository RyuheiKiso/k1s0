# 02. k8s ブートストラップ

本ファイルは `infra/k8s/` 配下の配置を確定する。Kubernetes クラスタの最下層、すなわち CNI / namespace / networking / storage を担当する。

## k8s/ の役割

Kubernetes クラスタに「何を最初にインストールするか」を規定する層。Argo CD も動かす前の素の状態から、全アプリケーションが動作可能になるまでの基盤を提供する。

- **CNI**: Pod 間ネットワーク（Cilium 推奨）
- **CoreDNS**: クラスタ内 DNS
- **cert-manager CRD**: TLS 証明書発行の CRD のみ先行インストール
- **namespace**: アプリケーション用 namespace（k1s0-tier1 / k1s0-data / k1s0-obs 等）
- **MetalLB / gatewayclass**: LoadBalancer Service 実体化
- **NetworkPolicy**: default-deny のベース policy
- **StorageClass**: Longhorn CSI による永続ボリューム

## bootstrap/ の設計

`bootstrap/` は、ArgoCD が起動する前の段階で、Kubeadm / k3s の直後に kubectl apply -k される。具体的手順:

1. Kubeadm or k3s で Control Plane / Worker Node を構築
2. `kubectl apply -k infra/k8s/bootstrap/` で CNI / CoreDNS / cert-manager CRD をインストール
3. Pod が全て Running になったら `kubectl apply -k infra/k8s/namespaces/` で namespace 作成
4. `kubectl apply -k infra/k8s/networking/` で MetalLB / NetworkPolicy をインストール
5. `kubectl apply -k infra/k8s/storage/` で Longhorn インストール
6. ArgoCD をインストール（`deploy/bootstrap/argocd/`）
7. ArgoCD Application で残る全層（mesh / dapr / data / security / obs / tier1-3）を宣言的に展開

bootstrap / namespaces / networking / storage の 4 段階は `ops/scripts/bootstrap-cluster.sh` に自動化として記述される。

## namespace 命名規則

- プレフィックス `k1s0-` 必須
- tier 層は `k1s0-tier1` `k1s0-tier2` `k1s0-tier3`
- 機能層は `k1s0-data`（データ層 Operator）、`k1s0-dapr`（Dapr control plane）、`k1s0-mesh`（Istio Ambient）、`k1s0-security`（Keycloak / OpenBao / SPIRE / cert-manager / Kyverno）、`k1s0-obs`（LGTM スタック）、`k1s0-platform-tools`（Backstage 等の開発者ポータル）
- テナント用は `k1s0-tenant-<id>` を 採用後の運用拡大時 で規定

namespace 名と ArgoCD AppProject 名の衝突を避けるため、Kubernetes Namespace には機能説明の suffix（`-tools` / `-tier1` 等）を付与し、AppProject 側は論理グルーピング名（`k1s0-platform`）のみとする。詳細は `60_operationレイアウト/06_Backstage_プラグイン配置.md` 参照。

## NetworkPolicy の default-deny

全 namespace で以下の default-deny ポリシーを前提とする。

```yaml
# networking/network-policies/default-deny.yaml
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: default-deny
  namespace: k1s0-tier1
spec:
  podSelector: {}
  policyTypes:
    - Ingress
    - Egress
```

アプリケーション側で必要な通信のみ明示的に allow-list する。Istio Ambient AuthorizationPolicy と併せて、L3/L4 は NetworkPolicy、L7 は Istio で管理する 2 層防御。

## StorageClass の 2 層化

- `longhorn-retain`: Prod 環境で PostgreSQL / Kafka 等のデータ永続化に使用。PVC 削除しても PV を保持
- `longhorn-delete`: 一時ボリューム（テスト用 / キャッシュ）。PVC 削除で PV も即座に削除

これは NFR-A-AVL-\* のデータ保護要件に従う。

## リリース時点 での最小構成

リリース時点 では以下のみ構築する。

- CNI: Cilium（最小構成）
- CoreDNS: 標準
- namespace: `k1s0-tier1` `k1s0-dapr` `k1s0-mesh` `k1s0-data` `k1s0-obs` の 5 個
- MetalLB: LoadBalancer Service の IP 発行
- Longhorn: 単一ノード / replica=1（開発環境）

運用蓄積後で残る namespace（tier2 / tier3 / security）と NetworkPolicy、StorageClass の 2 層化を導入する。
