# 08. 環境別パッチ配置

本ファイルは `infra/environments/` 配下の配置を確定する。dev / staging / prod の 3 環境で infra base を overlay する Kustomize パッチを規定する。

## 環境分離の必要性

`infra/` 配下の各サブディレクトリ（k8s / mesh / dapr / data / security / observability）は prod 環境をベースとする設定を記述する。dev / staging では以下が異なる。

- リソース規模（replica / PVC 容量 / CPU / memory）
- 外部連携（Let's Encrypt staging / 本番、顧客 Azure AD 連携）
- バックアップ保持期間
- フィーチャーフラグ有効化範囲
- 監視アラートの送信先

これらを overlay で吸収する。prod base + 環境パッチの Kustomize 構成とする。

## レイアウト

`patches/` はインフラ共通 overlay、`dapr-components-overlay/` は `infra/dapr/components/` を base とする Dapr Component 専用 overlay、`secrets/` は環境固有の暗号化済み secret、`values/` は Helm chart の環境 override 用 values で、用途が混在しないようサブディレクトリで分離する。

```
infra/environments/
├── README.md
├── dev/
│   ├── kustomization.yaml          # base references + patches（インフラ全体）
│   ├── patches/                    # infra/k8s / data / observability 等の patch
│   │   ├── cloudnativepg-instances-1.yaml  # instances: 3 → 1
│   │   ├── kafka-broker-1.yaml             # replicas: 3 → 1
│   │   ├── valkey-cluster-disabled.yaml    # standalone mode
│   │   ├── letsencrypt-staging.yaml        # clusterIssuer 切替
│   │   └── observability-retention-7d.yaml # 保持期間短縮
│   ├── dapr-components-overlay/    # infra/dapr/components/ への env 別 patch
│   │   ├── kustomization.yaml      # base: ../../../dapr/components
│   │   ├── state-store.patch.yaml  # dev 用 state store 接続先
│   │   └── pubsub.patch.yaml       # dev 用 Kafka broker URI
│   ├── secrets/
│   │   └── dev-oidc-client-secret.yaml.sops  # SOPS 暗号化
│   └── values/
│       └── ...                     # Helm values 上書き
├── staging/
│   ├── kustomization.yaml
│   ├── patches/
│   │   ├── cloudnativepg-instances-2.yaml
│   │   ├── kafka-broker-3.yaml             # prod と同じ
│   │   └── ...
│   ├── dapr-components-overlay/
│   │   ├── kustomization.yaml
│   │   └── state-store.patch.yaml
│   ├── secrets/
│   └── values/
└── prod/
    ├── kustomization.yaml
    ├── patches/                    # 通常は空（base が prod 基準）
    ├── dapr-components-overlay/    # 通常は空の kustomization.yaml のみ
    │   └── kustomization.yaml
    ├── secrets/
    │   └── prod-oidc-client-secret.yaml.sops
    └── values/
```

## kustomization.yaml の構造

環境 overlay は「外側の `kustomization.yaml`」と「`dapr-components-overlay/kustomization.yaml`」の 2 段構成で構築する。前者は infra 全体、後者は `infra/dapr/components/` を base とした Dapr Component 専用 overlay を適用する。

```yaml
# environments/dev/kustomization.yaml
apiVersion: kustomize.config.k8s.io/v1beta1
kind: Kustomization

resources:
  - ../../k8s/bootstrap
  - ../../k8s/namespaces
  - ../../k8s/networking
  - ../../k8s/storage
  - ../../mesh/istio-ambient
  - ../../mesh/envoy-gateway
  - ../../dapr/control-plane
  - ./dapr-components-overlay       # base: ../../dapr/components の env 別 overlay
  - ../../data/cloudnativepg
  - ../../data/kafka
  - ../../data/valkey
  - ../../data/minio
  - ../../security/keycloak
  - ../../security/openbao
  - ../../security/spire
  - ../../security/cert-manager
  - ../../security/kyverno
  - ../../observability/otel-collector
  - ../../observability/loki
  - ../../observability/tempo
  - ../../observability/mimir
  - ../../observability/grafana
  - ../../observability/pyroscope
  - ../../feature-management/flagd
  - ../../scaling/keda

patches:
  - path: patches/cloudnativepg-instances-1.yaml
  - path: patches/kafka-broker-1.yaml
  - path: patches/valkey-cluster-disabled.yaml
  - path: patches/letsencrypt-staging.yaml
  - path: patches/observability-retention-7d.yaml
```

```yaml
# environments/dev/dapr-components-overlay/kustomization.yaml
apiVersion: kustomize.config.k8s.io/v1beta1
kind: Kustomization

resources:
  - ../../../dapr/components        # base

patches:
  - path: state-store.patch.yaml
    target:
      kind: Component
      name: state-postgres
  - path: pubsub.patch.yaml
    target:
      kind: Component
      name: kafka-pubsub
```

## dev 環境の特徴

- **リソース最小化**: CloudNativePG 1 replica、Kafka 1 broker、Valkey standalone
- **Let's Encrypt staging**: 本番証明書を浪費しない
- **保持期間短縮**: ログ 7 日、トレース 3 日
- **外部連携簡易**: 顧客 Azure AD 連携は dev ではダミー IdP

開発者が単一ノードの kind / k3d クラスタでも起動可能な規模。

## staging 環境の特徴

- **HA 準拠**: 原則 prod と同じ構成
- **データは擬似プロダクション**: 実顧客データは含まない
- **負荷試験実施環境**: k6 / Gatling による負荷試験を staging で実行
- **Blue/Green で prod デプロイ前の最終検証**

## prod 環境の特徴

- **base がそのまま prod を表現**: patches/ は原則空
- **SOPS 暗号化 secret**: SOPS + AGE で暗号化、GPG key は `ops/oncall/` 管理
- **マルチリージョン**: 採用後の運用拡大時 で multi-region 化

## secrets/ の運用

環境ごとの secret（OIDC client secret、外部 API キー、S3 バックアップ先 credential 等）は SOPS で暗号化し `environments/<env>/secrets/` に配置する。

```yaml
# environments/dev/secrets/dev-oidc-client-secret.yaml.sops
apiVersion: v1
kind: Secret
metadata:
  name: tier3-web-portal-oidc
  namespace: k1s0-tier3
type: Opaque
data:
  client-secret: ENC[AES256_GCM,data:...,type:str]
```

SOPS の AGE key は `ops/oncall/sops-key/` 配下に置かれ、オンコール担当のみがアクセス可能。GitOps（ArgoCD）は SOPS plugin で復号してから apply する。

## Dev Container / local-stack との関係

開発者ローカルの kind / k3d クラスタは、`environments/dev/` を apply する。`tools/local-stack/` の bootstrap スクリプトが以下を実行する。

1. kind create cluster
2. kubectl apply -k infra/environments/dev/
3. ArgoCD サーバは不要（ローカルは kustomize 直接 apply）

staging / prod は ArgoCD ApplicationSet（`deploy/apps/application-set.yaml`）が `environments/<env>/` を target とする。
