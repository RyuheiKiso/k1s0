# 03. サービスメッシュ配置

本ファイルは `infra/mesh/` 配下の配置を確定する。k1s0 では **Istio Ambient Mesh**（sidecar-less）と **Envoy Gateway**（ingress）を採用する。

## Ambient Mesh を採用する理由

sidecar 方式は Pod 毎に Envoy が 1 個付くため、リソース消費（CPU / メモリ）が tier1 の 6 Pod + tier2 サービス + tier3 BFF で数 GB に達する。JTC での運用では NFR-B-PERF-\* の要件（CPU 10% 以下 / メモリ 50% 以下）を満たすため sidecar-less が望ましい。

Istio Ambient は ztunnel（ノード DaemonSet）が L4 暗号化・認可を担い、L7 処理が必要な場合のみ waypoint proxy を導入する。sidecar 方式より以下が優位。

- Pod 毎のメモリオーバーヘッド大幅削減（sidecar 200MB → ambient 数 MB）
- Pod 再起動なく Istio アップグレード可能
- Init container 競合の問題なし（特に Dapr sidecar との共存が容易）
- 2 名運用前提で管理オーバーヘッドを削減

## istio-ambient/ の構造

### operator/

Istio Operator を Helm chart としてインストールする values。profile は `ambient`。

```yaml
# operator/values.yaml
profile: ambient
meshConfig:
  accessLogFile: /dev/stdout
  defaultConfig:
    proxyStatsMatcher:
      inclusionRegexps:
        - .*
values:
  pilot:
    env:
      PILOT_ENABLE_AMBIENT: "true"
```

### ztunnel/

ztunnel は全 Node で DaemonSet として動作。Pod の L4 通信を透過的に捕捉し、ワークロード間 mTLS を提供する。`peerauthentication/strict-mtls.yaml` で STRICT モードを強制すれば、mTLS 不対応の通信は拒否される。

### ambient / waypoint の opt-in 境界

Istio Ambient では namespace ラベルで 2 段階の有効化を行う。k1s0 は **「k1s0-\* namespace は ambient on、legacy / kube-system 等は off」** を基本戦略とする。

- **L4 暗号化（ztunnel 経由）を有効化する namespace**: `istio.io/dataplane-mode=ambient` ラベルを付与
  - 付与対象: `k1s0-tier1`, `k1s0-tier2`, `k1s0-tier3`, `k1s0-dapr`, `k1s0-observability`
  - 付与しない: `kube-system`, `kube-public`, `argocd`, `litmus`, `cert-manager`（クラスタ運用コンポーネントは ambient 対象外）
- **L7 処理（waypoint proxy 経由）を有効化する namespace / workload**: Gateway API の `Gateway` リソースに `istio.io/waypoint-for: service` 等を指定し、対象 namespace に `istio.io/use-waypoint=<gateway-name>` ラベルを付与
  - Phase 1b: `k1s0-tier1` のみ（JWT 検証 / AuthorizationPolicy を waypoint で実行）
  - Phase 1c: `k1s0-tier2` / `k1s0-tier3` も追加

namespace ラベルの管理は `infra/environments/<env>/kustomization.yaml` の `patches:` で集中管理し、各 namespace の Application YAML 個別に散らばらないようにする。ラベル変更は ArgoCD Sync Wave の -9 〜 -5 で istio-control と同時に反映される。

### waypoint/

L7 機能（JWT 検証 / ルーティング / リトライ）が必要な namespace / workload に waypoint proxy を導入する。Phase 1b では tier1 のみに導入し、Phase 1c で tier2 / tier3 にも拡大する。

### authz/

Istio AuthorizationPolicy。tier1 公開 API へのアクセスは JWT Claim の validation を waypoint で行う。tier2 は tier1 経由しか受け付けない（deny-default）。

### peerauthentication/

PeerAuthentication で STRICT mTLS を全 namespace に強制。これによって Istio 内の全サービス間通信が暗号化される。

## envoy-gateway/ の構造

Istio Ambient と並行して Envoy Gateway を ingress として採用する。Istio Ingress Gateway ではなく Envoy Gateway を使う理由は、Gateway API（Kubernetes 標準）準拠で、将来の Ingress 層移行コストが低いため。

- `gatewayclass-envoy.yaml`: GatewayClass 定義（envoyproxy/envoy-gateway）
- `gateway-public.yaml`: インターネット公開用 Gateway（MetalLB の IP で露出）
- `gateway-internal.yaml`: VPN / Bastion 内部公開用 Gateway
- `httproute/*.yaml`: 各 tier1 API / tier3 Web アプリのルーティング

HTTPRoute は URL path 単位で tier1 API（`/api/v1/*`）と tier3 Web（`/`）を振り分ける。

```yaml
# httproute/tier1-api.yaml
apiVersion: gateway.networking.k8s.io/v1
kind: HTTPRoute
metadata:
  name: tier1-api
  namespace: k1s0-tier1
spec:
  parentRefs:
    - name: k1s0-gateway-public
      namespace: k1s0-mesh
  hostnames:
    - api.k1s0.internal
  rules:
    - matches:
        - path:
            type: PathPrefix
            value: /api/v1
      backendRefs:
        - name: tier1-facade-service-invoke
          port: 8080
```

## mesh と dapr の共存

Dapr sidecar も ztunnel もある場合、Pod の通信経路は以下のフローになる。

1. アプリコード → Dapr sidecar（localhost:3500）
2. Dapr sidecar → ztunnel（Node のソケット経由）
3. ztunnel → 対向 Node の ztunnel（mTLS）
4. 対向 ztunnel → 対向 Dapr sidecar or Service

ztunnel は Dapr 通信も含めて暗号化する。Dapr 自体の通信（control plane との通信）は `k1s0-dapr` namespace に限定されるため、PeerAuthentication を STRICT にしても Dapr Operator / Placement / Sentry の通信は mTLS で保護される。
