# 06. セキュリティ層配置

本ファイルは `infra/security/` 配下の配置を確定する。Keycloak（OIDC IdP）、OpenBao（secret 管理）、SPIRE（workload identity）、cert-manager（TLS 証明書）、Kyverno（admission policy）の 5 基盤を統合配置する。

## セキュリティ層基盤の役割

NFR-E-SEC-\* と NFR-G-PRV-\* の要件を満たす基盤層。アプリケーションはこれらを直接意識せず、Dapr / Istio / cert-manager 経由で透過的に利用する。

- **Keycloak**: OIDC / SAML の Identity Provider。tier1 / tier2 / tier3 の認証を集中管理
- **OpenBao**: HashiCorp Vault の OSS fork（ライセンス変更前の fork）。secret / dynamic secret / PKI
- **SPIRE**: SPIFFE 実装。workload identity（mTLS）を発行
- **cert-manager**: TLS 証明書の自動発行・更新。Let's Encrypt / SPIFFE Trust Domain / Istio 用
- **Kyverno**: admission policy。リソース作成時のガバナンス強制

## keycloak/ の役割

Keycloak Operator で Keycloak 本体を StatefulSet として起動。PostgreSQL バックエンドは `infra/data/cloudnativepg/` の Cluster を利用。

### realms の 2 分離

- **k1s0-internal**: 社内開発者、SRE、オンコールが使用
- **k1s0-tenant**: マルチテナント顧客ユーザ

各 realm は client / user / group / role を独立管理。realm 間の cross-authentication は禁止。

### identity-providers/ で顧客 Azure AD 連携

採用側組織の顧客の多くは Azure AD（Microsoft Entra ID）を IdP として使用。Keycloak を IdP Federation として、顧客 Azure AD にログイン委譲する構成。tier3 アプリは Keycloak のみに向き合えばよい。

## openbao/ の役割

静的 secret（API キー、DB 接続情報）と、dynamic secret（短命 DB クレデンシャル）、PKI（内部 CA）を提供。Dapr secretstore component が OpenBao を参照し、tier1 Secrets API 経由で secret を配布。

### auth-methods/

- **kubernetes**: ServiceAccount Token を使った auth。k1s0 内部アプリ向け
- **spire**: SPIFFE SVID を使った auth。Pod のワークロード ID で認証。tier1 Rust 層が採用

### secret-engines/

- **kv-v2**: 静的 secret（key-value）
- **database**: PostgreSQL / MySQL 等の dynamic credential 発行
- **pki**: 内部 CA（cert-manager と連携）

## spire/ の役割

SPIFFE workload identity を発行。各 Pod が kubelet → SPIRE Agent → SPIRE Server 経由で SVID（Short-Lived Certificate）を取得、mTLS 接続に使用。

### trust domain

k1s0 は単一 trust domain `spiffe://k1s0.internal` を採用。採用後の運用拡大時 でマルチ trust domain（環境分離）を検討。

### registration/

どの Pod にどの SPIFFE ID を発行するかを Registration Entry CRD で宣言。

```yaml
# registration/tier1-workloads.yaml
apiVersion: spire.spiffe.io/v1alpha1
kind: ClusterSPIFFEID
metadata:
  name: tier1-facade-service-invoke
spec:
  spiffeIDTemplate: spiffe://k1s0.internal/ns/{{.PodMeta.Namespace}}/sa/{{.PodSpec.ServiceAccountName}}
  podSelector:
    matchLabels:
      app: tier1-facade-service-invoke
```

### csi-driver/

SPIFFE CSI Driver で Pod に SVID を mount。アプリは `/spiffe/svid/cert.pem` を読むだけで mTLS 可能。

## cert-manager/ の役割

- **letsencrypt-prod**: 公開 Web ドメイン向け
- **internal-ca**: OpenBao PKI を Issuer として参照、社内サービス向け
- **csi-driver**: Pod に証明書を CSI mount（cert-manager CSI Driver）

## kyverno/ の役割

5 つの代表的 policy:

- `require-resource-requests`: CPU / Memory requests / limits 必須
- `require-image-signature`: Cosign 署名付き image のみ deploy 許可（Sigstore 連携）
- `disallow-privileged`: privileged Pod を全面禁止
- `require-network-policy`: 新 namespace には NetworkPolicy 必須
- `restrict-external-image-pull`: Harbor / 許可レジストリ以外から pull 禁止

これらは admission webhook として動作し、GitOps による reconcile 前に apply を拒否する。
