# 05. 既存設計との整合方針

本ファイルは `docs/05_実装/00_ディレクトリ設計/` の確定内容と、既存の概要設計 DS-SW-COMP-120 系・ADR・要件定義との整合をどのように維持するかを定める。本章のレイアウト決定は、概要設計で確定していた `src/tier1/{contracts, go, rust, infra}` 構造を昇格・分離する形で改訂するため、既存設計の一部を明示的に書き換える必要がある。

## 整合方針が必要な理由

概要設計は論理レベル、実装ディレクトリ設計は物理レベルの決定であるため、両者の間には必然的に変換関係が存在する。しかし変換関係を暗黙のままにすると、以下の問題が発生する。

- 概要設計を読んだ実装者が「`src/tier1/contracts/`」と記述を信じて実装すると、実際には `src/contracts/`（昇格後）に移動しているため、ファイルが見つからない
- DS-SW-COMP-120 の改訂履歴が残っていないと、後年の監査で「なぜ論理レベルで tier1 配下にあったのに物理レベルで tier1 配下にないのか」を説明できない
- 概要設計と実装設計の 2 つが真実を主張する状態が生まれ、どちらが最新か判別不能になる

本ファイルは整合性を維持するための 4 つのルールを定義する。

## ルール 1: 概要設計 DS-SW-COMP-120 系の改訂対象

以下の概要設計 ID は本実装ディレクトリ設計の確定に伴い改訂する。

### DS-SW-COMP-120（tier1 のトップレベル構成）

**改訂前**: `src/tier1/{contracts, go, rust, infra}` の 4 サブディレクトリ

**改訂後**: `src/tier1/{go, rust}` の 2 サブディレクトリ。contracts は `src/contracts/`（ADR-DIR-001）、infra は ルート `infra/`（ADR-DIR-002）に昇格

**対応 ADR**: ADR-DIR-001 / ADR-DIR-002

### DS-SW-COMP-121（contracts/ の内部構成）

**改訂前**: `src/tier1/contracts/v1/` 配下に tier1 11 API の .proto を集約

**改訂後**: `src/contracts/tier1/v1/` 配下に tier1 公開 11 API、`src/contracts/internal/v1/` に tier1 内部 gRPC（facade → rust）を分離。buf module 境界は `src/contracts/` 直下

**対応 ADR**: ADR-DIR-001

### DS-SW-COMP-122（Protobuf 生成コード配置）

**改訂前**: `buf generate` の Rust 出力先が `src/tier1/rust/crates/proto-gen/src/v1/`、Go 出力先が `src/tier1/go/internal/proto/v1/`

**改訂後**: 出力先は変更しない（Go の import path 維持のため）。ただし入力の `.proto` パスが `src/tier1/contracts/v1/*.proto` から `src/contracts/tier1/v1/*.proto` + `src/contracts/internal/v1/*.proto` に変更される。`buf.gen.*.yaml`（4 ファイル: go / rust / ts / csharp）の `inputs:` / `include_types:` を更新

**対応 ADR**: ADR-DIR-001

### DS-SW-COMP-132（proto-gen crate の構成）

**改訂前**: `build.rs` で `src/tier1/contracts/v1/*.proto` を tonic-build で処理

**改訂後**: `build.rs` で `src/contracts/tier1/v1/*.proto` + `src/contracts/internal/v1/*.proto` を tonic-build で処理

**対応 ADR**: ADR-DIR-001

## ルール 2: 改訂時の PR 運用

上記 4 件の DS-SW-COMP 改訂は以下の PR 運用で行う。

- 改訂 PR のタイトルに `DS-SW-COMP-120 改訂（ADR-DIR-001/002 適用）` のように対応 ID を前置
- 改訂 PR の本文に Before / After の対比を図で提示
- 対応する IMP-DIR-\* を同 PR 内で採番（例: DS-SW-COMP-120 改訂 PR で IMP-DIR-T1-021 を同時採番）
- 改訂履歴は概要設計 ID の本文内「改訂履歴」節と、[../90_トレーサビリティ/02_DS-SW-COMP_121-135_との対応.md](../90_トレーサビリティ/02_DS-SW-COMP_121-135_との対応.md) の両方に記録

## ルール 3: 論理と物理の「真実」の所在

改訂後は以下の責任分担で真実を保持する。

- **論理レベルの方式確定**: 概要設計 DS-SW-COMP-\* に記述（「tier1 は Go と Rust のハイブリッド」等）
- **物理レベルの配置確定**: 実装ディレクトリ設計 IMP-DIR-\* に記述（「src/tier1/go/ と src/tier1/rust/」等）

概要設計は「論理は変わらないが物理は変わることがある」ことを前提に記述し、物理パスを本文に含める際は「実装上は [対応 IMP-DIR] 参照」の形で相対リンクする。逆に実装ディレクトリ設計は「物理配置は論理設計 [対応 DS-SW-COMP] に基づく」と明記する。

## ルール 4: CLAUDE.md の更新

[../../../../CLAUDE.md](../../../../CLAUDE.md) の「プロジェクト構造」節は、実装段階突入時点で「実体の正」として維持する。本実装ディレクトリ設計確定と同時に CLAUDE.md の該当節を新レイアウト（`src/` + `infra/` + `deploy/` + `ops/` + `tools/` + `tests/` + `examples/` + `third_party/` + `docs/`）で書き直す。

CLAUDE.md は Claude Code / Agents が毎セッション参照するため、リリース時点 実装開始前に必ず最新化する。更新タイミングは 本プランの最終段階の範囲。

## ルール 5: ADR ステータスの維持

ADR-DIR-001 / ADR-DIR-002 / ADR-DIR-003 は本実装ディレクトリ設計の確定で Accepted となる。リリース時点 / 採用後の運用拡大時 で方針変更が必要となった場合、既存 ADR のステータスを `Superseded` にし、新規 ADR（例: ADR-DIR-004）を起票する。これは ADR テンプレート規約（[../../../00_format/ADR-xxxx.md](../../../00_format/ADR-xxxx.md)）に準拠する。

## 既存 DS-SW-COMP の中で改訂対象外

以下は本実装ディレクトリ設計で改訂しない。論理レベルの方式として維持する。

- DS-SW-COMP-119（モジュール依存関係）: 依存方向は IMP-DIR-ROOT-002 で物理に反映するが、論理方式そのものは不変
- DS-SW-COMP-123（buf 設定方針）: buf の lint / breaking 設定は不変
- DS-SW-COMP-124（Go module 全体レイアウト）: 内部構造は維持、proto の import 先のみ更新
- DS-SW-COMP-125（cmd/ 配下の Pod entrypoint）: 不変
- DS-SW-COMP-126（internal/ 配下の package 分割）: 不変
- DS-SW-COMP-127（Go ビルドターゲット）: 不変
- DS-SW-COMP-128（Go Docker image 構成）: 不変
- DS-SW-COMP-129（Cargo workspace 全体レイアウト）: 内部構造は維持、proto の入力 path のみ更新
- DS-SW-COMP-130（workspace Cargo.toml 方針）: 不変
- DS-SW-COMP-131（各 crate の内部構造）: 不変
- DS-SW-COMP-133（Rust ビルドターゲット）: 不変
- DS-SW-COMP-134（Rust Docker image 構成）: 不変
- DS-SW-COMP-135（monorepo の選択的ビルド方式）: path-filter の path 表現は更新、方針は不変
- DS-SW-COMP-136〜139: 不変
