# 04. スパースチェックアウト戦略概論

本ファイルは k1s0 モノレポで採用するスパースチェックアウト戦略の概要を説明する。詳細な cone 定義・運用手順は [../80_スパースチェックアウト運用/](../80_スパースチェックアウト運用/) 配下を参照。本ファイルは方針レベルの全体像と、なぜこの戦略を採用するかの根拠を示す。

## 概論が必要な理由

スパースチェックアウトは Git の機能として存在するが、運用方針なしに導入するとチームごとに cone 定義がばらつき、CI との整合が壊れ、新規参加者が毎回違う cone で作業する状態が発生する。本ファイルは「どのような役割で、どの粒度で cone を切り、どの場面で full cone に戻るか」を方針として先に定義する。

これにより、個々の cone 定義ファイル（`.sparse-checkout/roles/*.txt`）を書く際、コメント無しでも意図が読み取れる状態を作る。

## k1s0 が採用する戦略の全体像

### 採用機能の組み合わせ

Git の 3 つの機能を組み合わせて使う。

- **cone mode**（Git 2.27+）: ディレクトリ単位の選択的チェックアウト。個別ファイルのパターンマッチより高速
- **partial clone**（Git 2.32+）: `--filter=blob:none` で blob を遅延取得。初回クローン時間を 10 倍以上短縮
- **sparse index**（Git 2.37+）: cone 外のディレクトリを Git index で圧縮表現。`git status` 等の速度が数倍〜数十倍向上

この 3 機能の組み合わせが、本プロジェクトのリポジトリ規模（想定 60 万行超）でのスケール要件を満たす。

### 役割ベースの cone 定義

10 種類の役割別 cone 定義を `.sparse-checkout/roles/` に配置する。役割は「誰がどのレイヤーを日常的に編集するか」で切る。

- `tier1-rust-dev` : tier1 の Rust 領域（Decision / Audit / PII）を編集する開発者
- `tier1-go-dev` : tier1 の Go 領域（Dapr Facade）を編集する開発者
- `tier2-dev` : tier2 のドメインサービスを編集する開発者
- `tier3-web-dev` : tier3 の React + BFF を編集する開発者
- `tier3-native-dev` : tier3 の .NET MAUI を編集する開発者
- `platform-cli-dev` : 雛形 CLI / Backstage プラグインを編集する開発者
- `sdk-dev` : SDK 4 言語（Go / Rust / .NET / TypeScript）同格の横断開発者
- `infra-ops` : infra / deploy / ops を編集する SRE
- `docs-writer` : docs のみを編集する執筆者
- `full` : 全ディレクトリ（横断リファクタ時・インシデント調査時）

### cone 切替運用

役割変更時は以下の CLI で cone を切り替える。

```bash
tools/sparse/checkout-role.sh tier2-dev
```

このスクリプトは未コミット変更を自動 stash → cone 定義を読込 → `git sparse-checkout set --cone --stdin` を実行 → 既存変更を復元、の順で動作する。詳細実装は [../80_スパースチェックアウト運用/04_役割切替運用.md](../80_スパースチェックアウト運用/04_役割切替運用.md) 参照。

## なぜこの戦略か

### 規模に対する必然性

k1s0 の想定規模（60 万行 / 数万ファイル）では、フルチェックアウトの初期クローンが 10 分級になる可能性がある。さらに IDE のインデックス作成（rust-analyzer / gopls / TypeScript LS）は自分の担当外も対象にすると応答性が劣化する。スパースチェックアウトはこれらの両方を構造的に解決する。

### 世界トップ企業の知見

Google（CitC）、Meta（EdenFS）、Microsoft（Scalar）のモノレポ運用は、いずれもワーキングセット最小化の思想が核。k1s0 の規模では仮想ファイルシステムは過剰だが、Git 公式機能の sparse index + partial clone で同等効果を得ることが [ADR-DIR-003](../../../02_構想設計/adr/ADR-DIR-003-sparse-checkout-cone-mode.md) で確定している。

### 採用側の小規模運用との相性

NFR-C-NOP-001 で 採用側の小規模運用が前提となっている。2 人が tier1 と tier3 を掛け持ちする場面、あるいは 1 人が infra-ops と docs-writer を兼任する場面が発生する。役割切替で cone を切り替えることで、掛け持ちによる IDE 負荷分散を実現する。

### リリース時点 での必須化判定

採用検討時点では「推奨」止まり。リリース時点（運用品質確保、開発者 10 人超の想定）で必須化可否を再評価する。再評価基準は以下。

- 開発者数 10 人以上
- CI 時間が平均 20 分超
- リポジトリサイズが 1 GB 超（Git LFS 未使用時）

1 つ以上満たせば必須化、全部満たさなければ推奨継続。

## Full cone を許容する場面

スパースチェックアウトは常に最小セットが最適ではない。以下のケースでは `full.txt` への切替を推奨する。

- **リポジトリ全体に影響する横断リファクタ**（例: Protobuf スキーマの命名ルール変更）
- **新規コンポーネント追加時の影響範囲調査**（どこに既存コードが依存しているか grep する）
- **インシデント調査で全コードを grep する必要がある場面**
- **ドキュメントビルド**（docs 配下の全 md リンク整合検証）

切替は `tools/sparse/checkout-role.sh full` で即座に切り替えられる。

## submodule と Git LFS の扱い

### submodule は不採用

モノレポの atomic commit 利点を損なうため submodule は不採用。第三者 OSS のフォークは `third_party/` に vendoring で対応する（[../70_共通資産/04_third_party配置.md](../70_共通資産/04_third_party配置.md) 参照）。

### Git LFS は リリース時点 判定

リリース時点では drawio / svg / png も通常 Git 管理。100 MiB を超えるバイナリが発生した時点で ADR-DIR-004 を起票して LFS 採用可否を判定する。

## CI との整合

CI は path-filter で変更検出を行い、該当 tier のみビルドする。cone 定義と path-filter は独立に記述するが、両者の包含関係を [../80_スパースチェックアウト運用/05_CI戦略とpath_filter統合.md](../80_スパースチェックアウト運用/05_CI戦略とpath_filter統合.md) で整合検証する。

- cone 定義は「誰が編集するか」基準
- path-filter は「何が変わったか」基準
- 両者は対応表で管理（cone `tier1-rust-dev` の含むディレクトリは path-filter `rust/**` でトリガされる CI ジョブに対応）
