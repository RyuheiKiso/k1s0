# 07. ドキュメント近接配置方針

本ファイルは各主要ディレクトリの直下に配置する `README.md` の位置付け・必須項目・`docs/` との双方向リンク方針を定める。実装コードを読む開発者が「このディレクトリは何か」を口伝に頼らず理解できる状態を作るため、14 箇所の主要ディレクトリに README.md を必須とする（IMP-DIR-ROOT-004）。

## 近接配置方針が必要な理由

実装チームが IDE で `src/tier2/go/services/` を開いた際、そこから `docs/05_実装/00_ディレクトリ設計/30_tier2レイアウト/` にたどり着くには最低 6 階層の移動が必要になる。これをストレスなく行える開発者は少数であり、結果として「ディレクトリの意味はチームの暗黙知として口伝される」状態に陥る。

近接配置では、開発者が編集対象のディレクトリに対して最短距離で「このディレクトリの目的と関連 IMP-DIR」を確認できる状態を作る。同時に、`docs/05_実装/` 側の詳細設計ドキュメントは詳細な設計経緯を保持し、README.md は要約とリンクに留める。二重管理を避けるため、README.md は `docs/05_実装/` への参照を主、独自記述を従とする。

## 必須項目

各主要ディレクトリの `README.md` は以下の 5 項目を必ず含める。

1. **ディレクトリの目的**（1-2 段落の散文）
2. **サブディレクトリ構成**（ツリー表示 + 各々の役割を 1 行で）
3. **対応する docs/05_実装/ のパス**（相対リンク）
4. **CODEOWNERS に基づく責任チーム**（GitHub team 表記）
5. **参照すべき主要 IMP-DIR ID / DS-SW-COMP ID / ADR ID**

## README.md を配置する 14 箇所

以下の 14 箇所に README.md を配置する。これらはディレクトリ設計原則 IMP-DIR-ROOT-004 の最小セットである。

### ソース系（7 箇所）

- `/src/README.md` — ソースコード一次ディレクトリの俯瞰
- `/src/contracts/README.md` — Protobuf 契約の単一の真実
- `/src/tier1/README.md` — tier1 システム基盤層
- `/src/sdk/README.md` — 多言語クライアント SDK
- `/src/tier2/README.md` — ドメイン共通業務ロジック
- `/src/tier3/README.md` — エンドアプリ
- `/src/platform/README.md` — 雛形 CLI / analyzer / Backstage プラグイン

### インフラ / 運用系（3 箇所）

- `/infra/README.md` — クラスタ素構成
- `/deploy/README.md` — GitOps 配信定義
- `/ops/README.md` — 運用領域

### 共通資産系（4 箇所）

- `/tools/README.md` — 横断ツール
- `/tests/README.md` — tier 横断テスト
- `/examples/README.md` — 最小実装例
- `/third_party/README.md` — vendored OSS

### docs 系は別扱い

`/docs/README.md` は既存（`01_企画/README.md` 等のサブ README もあり）。本章の近接配置方針には含めないが、[../README.md](../README.md) と [../../README.md](../../README.md) で `docs/05_実装/` の索引は維持する。

## README.md 雛形

各ディレクトリの `README.md` は以下の雛形で書く。

```markdown
# <ディレクトリ名>

<ディレクトリの目的を散文で 1-2 段落>

## 構成

`‛`‛
<ディレクトリ名>/
├── <サブ 1>/  — <役割 1>
├── <サブ 2>/  — <役割 2>
└── <サブ 3>/  — <役割 3>
`‛`‛

## 詳細設計

- [docs/05_実装/00_ディレクトリ設計/<対応章>/](../../<相対パス>) — <章名>

## 責任チーム

- `@k1s0/<team>` — <役割>

## 主要 ID

- IMP-DIR-<サブ>-<通番> — <内容>
- DS-SW-COMP-<番号> — <内容>
- ADR-<ID> — <内容>
```

## 更新サイクル

各 README.md は以下のタイミングで更新する。

- **サブディレクトリ追加 / 削除時**: 同 PR で更新必須
- **CODEOWNERS 変更時**: 対応 README.md の「責任チーム」節を同 PR で更新
- **対応 IMP-DIR / DS-SW-COMP の変更時**: 対応 README.md の「主要 ID」節を同 PR で更新
- **半期ごとの棚卸し**: [../../../../00_format/document_standards.md](../../../../00_format/document_standards.md) の「鮮度維持」節に従い、README.md も棚卸し対象

README.md が実ディレクトリと乖離している状態は実装ディレクトリ設計の破綻と見なす。CI で README.md の存在チェックと、README.md 内の相対リンクの有効性チェックを自動化する（[../../80_スパースチェックアウト運用/05_CI戦略とpath_filter統合.md](../../80_スパースチェックアウト運用/05_CI戦略とpath_filter統合.md) 参照）。

## README.md を配置しないケース

以下のケースでは README.md を配置しない。

- 生成物ディレクトリ（`target/` / `node_modules/` / `bin/` / `obj/`）
- 小規模のサブディレクトリで、親の README.md でカバーできる範囲
- cone 外でアクセス頻度が低いケース（ただし `full.txt` でアクセス可能）

上記以外は必須。新規ディレクトリ追加 PR のレビュー観点に README.md 有無を含める。
