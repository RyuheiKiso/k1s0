# 01. リポジトリルート構成

本ファイルは k1s0 monorepo のリポジトリ root 直下に配置する全ディレクトリと全ファイルを方式として固定化する。上流は概要設計 [DS-SW-COMP-120（tier1 のトップレベル構成）](../../04_概要設計/20_ソフトウェア方式設計/01_コンポーネント方式設計/06_パッケージ構成_Rust_Go.md)で、本ファイルはそれを「リポジトリ root」の階層へ外側に拡張する。

## 本ファイルの位置付け

概要設計は `src/tier1/` の内部構造を方式として確定したが、その「外側」——リポジトリ root が `src/` の他に何を持つのか——は方式レベルで明示されていない。ルート直下の配置は一度決めると変更コストが跳ね上がるため（`docs/` と `documents/`、`tools/` と `scripts/`、`tests/` と `e2e/` がバラバラに混在すると、全 CI ワークフロー・全 IDE 設定・全 CODEOWNERS パターンに波及する）、Phase 0 の稟議通過直後に本ファイルで固定化する。

リポジトリルートに置くべきディレクトリとファイルの分類原則は 3 軸である。第 1 軸は**成果物か支援か**で、`src/` は配信される成果物、`tools/` `scripts/` `docs/` は支援であり、この区別を階層で明示する。第 2 軸は**言語独立か言語依存か**で、ルート直下に Go の `go.mod` や Rust の `Cargo.toml` を置かず、全て `src/tier1/<lang>/` の内側に閉じ込める（monorepo の複数言語を並立させるため）。第 3 軸は**CI 対象か開発者手元か**で、`.github/workflows/` は CI 対象、`.devcontainer/` と `.vscode/` は開発者手元であり、混在しないよう階層で分ける。

## 設計 ID 一覧と採番方針

本ファイルで採番する設計 ID は `DS-IMPL-DIR-001` 〜 `DS-IMPL-DIR-020` の 20 件である。

## リポジトリルートの全体構成

k1s0 リポジトリの root 直下は 7 ディレクトリ + 8 種類の root ファイルで構成する。各要素は独立した責務を持ち、混在させない。

```
k1s0/
├── README.md              # プロジェクト紹介（GitHub トップページ表示）
├── CLAUDE.md              # Claude Code 向けプロジェクト規約
├── LICENSE                # Apache-2.0 ライセンス本文
├── CODEOWNERS             # 所有権マップ（GitHub ネイティブ機能）
├── .editorconfig          # エディタ共通設定
├── .gitignore             # Git 追跡除外パターン
├── .gitattributes         # Git 属性（改行コード・LFS・linguist 等）
├── .mise.toml             # mise ツールバージョン固定
├── docs/                  # 全ドキュメント（Backstage TechDocs source）
├── src/                   # 全ソースコード（tier1 以下は tier 別に拡張）
├── tests/                 # 横断テスト（E2E / 契約 / 負荷 / フィクスチャ）
├── tools/                 # 開発支援ツール（Go / Rust 製補助バイナリ）
├── scripts/               # シェルスクリプト（CI / ローカル開発ヘルパ）
├── .github/               # GitHub 固有設定（Actions / Issue テンプレ等）
└── .devcontainer/         # Dev Container 定義（VS Code Remote Containers）
```

このレイアウトは「言語・ランタイムに依存しない命名」を root で徹底する。`src/` は Go でも Rust でも通じる一般名詞、`tools/` は言語非依存な慣習、`tests/` は横断テストの置き場として明確であり、将来 tier2（Rust / Go 以外の言語）を追加したときも破綻しない。

### DS-IMPL-DIR-001 root ディレクトリの 7 分割原則

リポジトリ root 直下には `docs/` / `src/` / `tests/` / `tools/` / `scripts/` / `.github/` / `.devcontainer/` の 7 ディレクトリのみを許可する。これ以外のディレクトリを root に追加する場合は ADR 起票を要する。背景は、rootにディレクトリが増えるほど「ここはどの責務？」という認知負荷が線形に増えることと、`CODEOWNERS` のパターンが肥大化して所有権が不鮮明になることである。`build/` `dist/` `out/` `target/` `node_modules/` 等のビルド成果物ディレクトリは全て `.gitignore` で除外し、root に現れないようにする。

**確定フェーズ**: Phase 0。**対応要件**: NFR-C-NOP-001、DX-CICD-\*。**上流**: DS-SW-COMP-120。

### DS-IMPL-DIR-002 root ファイルの 8 種類限定

リポジトリ root 直下のファイルは `README.md` / `CLAUDE.md` / `LICENSE` / `CODEOWNERS` / `.editorconfig` / `.gitignore` / `.gitattributes` / `.mise.toml` の 8 種類のみを許可する。これ以外（例: `package.json` / `go.mod` / `Cargo.toml` / `Makefile` / `docker-compose.yml`）は root に置かず、それぞれの適切なサブディレクトリに配置する。理由は複数言語 monorepo で root を「どの言語の入口」と解釈させないためで、Go 開発者が root で `go build ./...` と実行しても何も起きない（必ず `cd src/tier1/go && go build ./...` が必要）構造を意図的に作る。

**確定フェーズ**: Phase 0。**対応要件**: NFR-C-NOP-001。**上流**: DS-SW-COMP-120。

## src/ ディレクトリ

### DS-IMPL-DIR-004 src/ 直下の tier 別配置

`src/` 直下には tier 単位のサブディレクトリのみを配置する。Phase 1a 〜 1c は `src/tier1/` のみ、Phase 2 以降で `src/tier2/` および `src/tier3/` を追加する（tier2 の参照実装 / tier3 のアプリ配信ポータル）。tier 共有のライブラリ（例: Protobuf 契約の tier 横断共通型）を置きたい衝動は強いが、tier 間の結合度を低く保つため禁止する。tier 間で共有すべきは Protobuf 契約と OpenAPI 定義のみで、それらは各 tier の `contracts/` に複製する。

**確定フェーズ**: Phase 0（ルール）、Phase 1a（tier1 実体化）。**対応要件**: NFR-A-CONT-003、DX-CICD-\*。**上流**: DS-SW-COMP-120。

### DS-IMPL-DIR-005 tier1/ の 4 サブディレクトリ

`src/tier1/` 直下は概要設計 DS-SW-COMP-120 に従い、`contracts/` / `go/` / `rust/` / `infra/` の 4 サブディレクトリのみ配置する。詳細は [02_contracts詳細構成.md](02_contracts詳細構成.md) / [03_Goモジュール詳細構成.md](03_Goモジュール詳細構成.md) / [04_Rustワークスペース詳細構成.md](04_Rustワークスペース詳細構成.md) / [05_infra詳細構成.md](05_infra詳細構成.md) で展開する。`src/tier1/` 直下に `README.md` を置き、4 サブディレクトリの責務と概要設計の参照先を列挙する（150 行以内）。`src/tier1/README.md` は Backstage TechDocs の MkDocs Plugin で自動抽出し、Software Catalog に表示する。

4 サブディレクトリはそれぞれ第 2 階層で更に構造化する方式を採る。`go/internal/` は `pods/<pod>/` と `shared/<topic>/` の 2 系統（03 章 DS-IMPL-DIR-041〜044）、`rust/crates/` は `pods/<crate>/` / `shared/<crate>/` / `sdk/<crate>/` の 3 系統（04 章 DS-IMPL-DIR-081〜084）、`infra/` は `delivery/` / `runtime/` / `governance/` / `platform/` の 4 カテゴリ（05 章 DS-IMPL-DIR-121〜124）で、いずれも「変更頻度の高い Pod / crate 領域」と「横断影響の大きい共有・統治領域」を階層で分離し、CODEOWNERS / CI path filter / 所要レビュアを二段階に最適化する。`contracts/` は第 2 階層分割を持たず Protobuf / OpenAPI のフラット配置で、これは契約の一意性と参照の容易性を優先した意図的な非対称である。

**確定フェーズ**: Phase 1a。**対応要件**: DX-BS-\*、NFR-C-NOP-002。**上流**: DS-SW-COMP-120。

## tests/ ディレクトリ

### DS-IMPL-DIR-006 tests/ の横断テスト専用配置

`tests/` は `src/` 配下の言語別ユニットテスト（Go の `*_test.go`、Rust の `src/**/*.rs` の `#[cfg(test)]`）とは独立した**横断テスト専用ディレクトリ**とする。横断テストとは、単一 Pod に閉じないテスト（E2E / 契約 / 負荷）とフィクスチャの置き場を指す。`tests/` 配下は `e2e/` / `contract/` / `load/` / `fixtures/` / `golden/` の 5 サブディレクトリ固定で、詳細は [06_テストとフィクスチャ配置.md](06_テストとフィクスチャ配置.md) で展開する。言語別の結合テスト（Go: `integrationtest/`、Rust: 各 crate の `tests/`）は `src/tier1/<lang>/` 配下に閉じ、`tests/` には置かない。

**確定フェーズ**: Phase 1b。**対応要件**: DX-TEST-\*、NFR-B-PERF-\*。**上流**: DS-SW-COMP-136。

## tools/ ディレクトリ

### DS-IMPL-DIR-007 tools/ の開発支援ツール配置

`tools/` はビルド成果物には含まれない開発支援ツールを配置する。言語は Go / Rust を混在させるが、各ツールは独立したサブディレクトリで管理し、共通の `Cargo.toml` / `go.mod` に含めない。Phase 1a 〜 1c で配置するツールは以下 7 種である（詳細は [07_開発ツールとCICD配置.md](07_開発ツールとCICD配置.md) の DS-IMPL-DIR-181〜185 で展開）。

- `tools/proto-gen/` — buf CLI のラッパ（Rust 製、`tools/proto-gen/src/main.rs`）
- `tools/check-deps/` — コンポーネント依存検査（Go 製、DS-SW-COMP-113 の `tools/check-deps.sh` を Go に移植）
- `tools/check-rust-deps/` — Rust path 依存の静的検査（Rust 製、04 章 DS-IMPL-DIR-114 の `crates/pods/` ↔ `crates/shared/` ↔ `crates/sdk/` の依存方向を強制）
- `tools/k1s0-cli/` — 雛形生成 CLI（Rust 製、`k1s0 new <template>` を実装、DS-DEVX-LOCAL-\* の前提）
- `tools/mock-server/` — tier1 API モック（Go 製、DS-DEVX-LOCAL-\* の `--with-tier1=mock` を実装）
- `tools/backstage/` — Backstage TechDocs の `mkdocs.yml` と Backstage 固有設定（Phase 1c から）
- `tools/backstage-templates/` — Backstage Software Template（Phase 1b から）。`tier2/{go,csharp}/` と `tier3/{webapp-typescript,mobile-maui,bff-typescript}/` の 2 階層で tier 別・変種別に分類し、Phase 2 以降の新変種追加（例: `tier2/python/`、`tier3/desktop-tauri/`）で階層を機械的に拡張可能にする

各ツールは独立 build target を持ち、リリースには含まれない（CI でのみビルド）。`tools/` 配下の Go ツールは `tools/<name>/go.mod` を独立に持ち、src/tier1/go の module と分離する（依存バージョンが異なっても問題なくするため）。Rust ツールは `tools/<name>/Cargo.toml` を独立に持ち、同様に分離する。`backstage-templates/` は CLI バイナリではなく宣言的テンプレート集のため go.mod / Cargo.toml を持たず、`template.yaml` と `skeleton/` のみで構成する。

**確定フェーズ**: Phase 1a（proto-gen / check-deps / check-rust-deps）、Phase 1b（k1s0-cli / mock-server / backstage-templates tier2-go 先行）、Phase 1c（backstage）、Phase 2（backstage-templates tier3 系）。**対応要件**: DX-CICD-\*、DX-LD-\*、DX-GP-\*、DX-DEVEX-\*。**上流**: DS-SW-COMP-113、DS-DEVX-LOCAL-\*、DS-DEVX-DOC-\*。

### DS-IMPL-DIR-008 tools/ 配下の独立モジュール原則

`tools/` 配下の各ツールは src/tier1 とも互いとも独立した Go module / Cargo workspace とする。`tools/check-deps/go.mod` は `src/tier1/go/go.mod` と別物であり、`tools/proto-gen/Cargo.toml` は `src/tier1/rust/Cargo.toml` と別物である。理由は tools の依存更新が src/tier1 のビルドを壊さないようにする（例: `tools/k1s0-cli` が `clap 5.0` を使い、`src/tier1/rust` が `clap 4.5` を使う状況を許容）ためと、tools のリリース速度を src/tier1 から独立させるためである。ただし `k1s0-common` / `k1s0-otel-util` の共通ライブラリを tools から使いたい場合は path 依存で参照できるよう、tools 側の `Cargo.toml` に `k1s0-common = { path = "../../src/tier1/rust/crates/shared/common" }` を許可する（04 章 DS-IMPL-DIR-107 の path 深化に対応。ただし tools → src/tier1 方向のみで、逆方向は禁止）。

**確定フェーズ**: Phase 1a。**対応要件**: DX-CICD-\*、NFR-C-NOP-002。**上流**: DS-SW-COMP-111、DS-SW-COMP-112。

## scripts/ ディレクトリ

### DS-IMPL-DIR-009 scripts/ のシェルスクリプト専用配置

`scripts/` は bash / zsh で書かれた開発者向けヘルパスクリプトを配置する。`tools/` が「ビルドして実行するバイナリ」を置く場所に対し、`scripts/` は「そのまま `./script.sh` で実行できるテキスト」を置く場所とする。配置予定スクリプトは以下。

- `scripts/bootstrap.sh` — 新規開発者向け環境セットアップ（mise install → devcontainer build → pre-commit install）
- `scripts/proto-gen.sh` — `tools/proto-gen/` を呼び出して `.proto` → Go/Rust 生成を実行
- `scripts/run-e2e.sh` — `tests/e2e/` の E2E シナリオを kind クラスタ上で実行
- `scripts/lint-all.sh` — 全言語（Go / Rust / TypeScript / Helm）の lint を一括実行
- `scripts/smoke-test.sh` — Phase 1a の smoke test（OTel ping / Service Invoke 往復）

スクリプトは POSIX sh 互換を原則とせず、bash 4.0 以上（associative array 利用可）を前提とする。Windows では Git Bash 経由で動作することを CI で検証する（DX-LD-\* の Windows 動作要件）。スクリプト先頭に `#!/usr/bin/env bash` と `set -euo pipefail` を必須とし、エラー時の早期終了を徹底する。

**確定フェーズ**: Phase 1a。**対応要件**: DX-LD-\*、DX-CICD-\*。**上流**: DS-DEVX-LOCAL-\*。

### DS-IMPL-DIR-010 scripts/ と tools/ の責務分離

`scripts/` と `tools/` の境界は「50 行以内の糊コードか、それ以上の実装か」で判断する。50 行以下で収まる「既存ツールを呼び出すだけ」の処理は `scripts/*.sh`、ロジックを持つ実装は `tools/<name>/` に Go / Rust で書く。シェルスクリプトが 100 行を超えた時点で Go / Rust に移植することを検討し、200 行を超えたら ADR 起票を要する。理由は bash のデバッグ性・テスト性が低く、大規模化すると保守不能になるためである。

**確定フェーズ**: Phase 0（ルール）、各 Phase（適用）。**対応要件**: NFR-C-NOP-001、DX-CICD-\*。**上流**: DS-SW-COMP-138（変更手続）。

## .github/ ディレクトリ

### DS-IMPL-DIR-011 .github/ の GitHub 固有設定配置

`.github/` は GitHub 固有のメタデータを配置する。配下は以下の固定構造とする。

```
.github/
├── workflows/               # GitHub Actions ワークフロー（YAML）
│   ├── pr.yml              # PR 1 回あたりのパイプライン（lint / test / build を path filter で選択実行）
│   ├── main.yml            # main マージ後のビルド・push
│   ├── release.yml         # タグ push 時のリリース
│   ├── nightly.yml         # 深夜の負荷試験・脆弱性スキャン
│   └── reusable/           # 再利用ワークフロー（go-test / rust-test / helm-lint / container-scan）
├── actions/                 # カスタム composite action（setup-toolchain / buf-generate）
├── ISSUE_TEMPLATE/          # Issue テンプレート
├── PULL_REQUEST_TEMPLATE.md # PR テンプレート
├── dependabot.yml           # Dependabot 設定（Go module / Cargo / GitHub Actions / Docker の 4 系統）
└── CODEOWNERS              # （root に移設、リンク用の空ファイルは置かない）
```

`CODEOWNERS` は root の `CODEOWNERS` を正とし、`.github/CODEOWNERS` は作らない。GitHub の検索優先順位は `.github/CODEOWNERS` > `root/CODEOWNERS` > `docs/CODEOWNERS` だが、root に置くことで「所有権はプロジェクト全体の規約である」という意図を示す。

**確定フェーズ**: Phase 1a。**対応要件**: DX-CICD-\*、NFR-SUP-\*。**上流**: DS-DEVX-CICD-\*。

### DS-IMPL-DIR-012 workflows/ の 4 系統分割と再利用ワークフロー

`.github/workflows/` は `pr.yml` / `main.yml` / `release.yml` / `nightly.yml` の 4 系統に分ける。当初案では PR 系統を言語別（pr-go.yml / pr-rust.yml / pr-contracts.yml / pr-docs.yml）に分割していたが、深化後のディレクトリ構造（`internal/pods/` vs `internal/shared/`、`crates/pods/` vs `crates/shared/` vs `crates/sdk/`、`infra/{delivery,runtime,governance,platform}/`）に対しては言語軸の分割だけでは path filter の表現力が足りず、単一 `pr.yml` 内で path filter と job の `if:` 条件で選択実行する方式に変更する。共通処理（Go / Rust toolchain セットアップ、buf generate）は `workflows/reusable/*.yml`（ジョブ単位）と `actions/*/action.yml`（ステップ単位）の 2 階層で再利用し、重複を排除する。各ワークフローのジョブ構成は [07_開発ツールとCICD配置.md](07_開発ツールとCICD配置.md) の DS-IMPL-DIR-189〜194 で展開する。

**確定フェーズ**: Phase 1a。**対応要件**: DX-CICD-\*、DX-MET-\*。**上流**: DS-SW-COMP-135。

## .devcontainer/ ディレクトリ

### DS-IMPL-DIR-013 devcontainer の配置

`.devcontainer/` は VS Code / IntelliJ の Dev Container 規約（[containers.dev](https://containers.dev/)）に準拠する設定を配置する。配下は以下の固定構造とする。

```
.devcontainer/
├── devcontainer.json       # Dev Container 定義
├── Dockerfile              # Dev Container イメージ
└── post-create.sh          # コンテナ作成後の初期化スクリプト
```

`devcontainer.json` に全開発者共通の VS Code 拡張・settings を宣言し、開発者ごとの個人設定は `.vscode/settings.json` で上書きする（ただし `.vscode/settings.json` は root に置かず `.gitignore` 対象）。`Dockerfile` は `mcr.microsoft.com/devcontainers/base:ubuntu-22.04` を継承し、Rust / Go / Node.js / kubectl / helm / kind / tilt / buf / dapr CLI を追加インストールする（DS-DEVX-LOCAL-\* の定義に従う）。`post-create.sh` はコンテナ作成時に 1 度だけ実行し、`mise install` と `pre-commit install` を行う。

**確定フェーズ**: Phase 1a。**対応要件**: DX-LD-\*、DX-GP-\*。**上流**: DS-DEVX-LOCAL-\*。

### DS-IMPL-DIR-014 .vscode/ の非管理方針

`.vscode/` ディレクトリは `.gitignore` で Git 管理対象外とし、ルートリポジトリに含めない。開発者ごとの editor 設定は個人の好みであり、共通化を強制するのは Dev Container 経由のみとする。共通化したい設定（推奨拡張・フォーマッタ）は `.devcontainer/devcontainer.json` の `customizations.vscode` に寄せる。ただし Phase 2 で「リポジトリ固有の debug 設定」が必要になった場合は `.vscode/launch.json` のみを git 管理対象に含め、`settings.json` と `extensions.json` は対象外のままとする（過去事例: 複数プロジェクトで `.vscode/settings.json` の強制共通化が開発者の反発を招いたため）。

**確定フェーズ**: Phase 1a（除外ルール）、Phase 2（launch.json 判断）。**対応要件**: DX-LD-\*。**上流**: DS-DEVX-LOCAL-\*。

### DS-IMPL-DIR-018 CODEOWNERS の所有権割当

リポジトリ root の `CODEOWNERS` は GitHub ネイティブ機能で PR レビュー強制に使う。所有権の割当原則は「階層ごとに独立したオーナーを設定し、親階層のオーナーは子の default owner として機能する」とする。Phase 1a の初期割当は以下。

```
# default（root）
*                                  @k1s0/tier1-architects

# 企画・要件・設計
/docs/01_企画/                     @k1s0/product-owners
/docs/03_要件定義/                 @k1s0/tier1-architects @k1s0/product-owners
/docs/04_概要設計/                 @k1s0/tier1-architects
/docs/05_実装/                     @k1s0/tier1-architects

# 契約
/src/tier1/contracts/              @k1s0/tier1-architects @k1s0/api-leads

# 実装
/src/tier1/go/                     @k1s0/tier1-go-team
/src/tier1/rust/                   @k1s0/tier1-rust-team

# インフラ
/src/tier1/infra/                  @k1s0/infra-team

# CI/CD
/.github/workflows/                @k1s0/devex-team @k1s0/tier1-architects
/tools/                            @k1s0/devex-team
/scripts/                          @k1s0/devex-team

# セキュリティ関連（05 章 DS-IMPL-DIR-123 で governance/ 配下へ深化）
/src/tier1/infra/governance/policies/    @k1s0/security-team @k1s0/infra-team
```

チーム名（`@k1s0/...`）は Phase 0 時点では仮置きで、Phase 1a 開始時に実在の GitHub Team に紐付ける。2 名運用（NFR-C-NOP-001）下では実際のレビュアーが 1 〜 2 名になるが、CODEOWNERS のチーム構造は将来の拡張を見越して 8 チーム（`tier1-architects` / `product-owners` / `api-leads` / `tier1-go-team` / `tier1-rust-team` / `infra-team` / `devex-team` / `security-team`）で分ける。本章以降 07 章 DS-IMPL-DIR-199 までこの 8 チームのみを使用し、Pod 別・crate 別の細分チーム名（例: `@k1s0/state-team`）は作らない（チーム数の増加は 2 名運用下では管理コストに直結するため）。

上記は 01 章での初期割当で、03 / 04 / 05 / 07 章の深化構造（`pods/ vs shared/ vs sdk/` / `delivery vs runtime vs governance vs platform`）に合わせて 07 章 DS-IMPL-DIR-199 で path filter を第 2 階層まで展開する。`shared/` 層と `sdk/` 層はアーキテクト必須、`pods/` 層は言語チーム単独、`governance/policies/` はセキュリティ主管、という設計原則もそこで確定する。本章は「root 直下の骨格を CODEOWNERS に投影する」にとどめ、内部階層の展開は 07 章に委譲する。

**確定フェーズ**: Phase 0（構造）、Phase 1a（実チーム紐付け）。**対応要件**: NFR-C-NOP-001、DX-CICD-\*。**上流**: DS-DEVX-CICD-\*。

### DS-IMPL-DIR-019 設定ファイル群の配置

`.editorconfig` / `.gitignore` / `.gitattributes` / `.mise.toml` の 4 設定ファイルは root に配置し、サブディレクトリに複製しない。

- `.editorconfig`: indent_style / indent_size / end_of_line / charset / trim_trailing_whitespace / insert_final_newline の 6 項目を言語別に宣言。Rust / Go は tab / 4、TypeScript / YAML / JSON は space / 2、Markdown は space / 2（trim_trailing_whitespace=false、hard break 許可）。
- `.gitignore`: 言語別 ignore を結合（target/ / dist/ / node_modules/ / .venv/ / __pycache__/ / *.log / .DS_Store / .idea/ / .vscode/ / bin/ / build/ / out/）。
- `.gitattributes`: 改行コード強制（`* text=auto eol=lf`、ただし `*.bat text eol=crlf` / `*.ps1 text eol=crlf`）、LFS 対象（`*.svg binary`、`*.drawio text`）、linguist 除外（`docs/** linguist-documentation`、`tests/fixtures/** linguist-generated`）を宣言。
- `.mise.toml`: Go / Rust / Node.js / buf / kubectl / helm / kind / tilt / dapr-cli のバージョンを pin する。Phase 1a の pin バージョンは Go 1.22、Rust 1.85、Node.js 20 LTS、buf 1.30、kubectl 1.28、helm 3.14、kind 0.22、tilt 0.33、dapr 1.13 とする（概要設計と整合）。

設定ファイルが複数階層に散らばると、同一プロジェクト内で矛盾した設定（例: root は indent=4、subdir は indent=2）が発生するため、全て root で一元管理する。

**確定フェーズ**: Phase 1a。**対応要件**: DX-CICD-\*、DX-LD-\*、NFR-C-NOP-002。**上流**: DS-SW-COMP-124、DS-SW-COMP-130。

### DS-IMPL-DIR-020 root 直下に置かない禁止事項

以下のファイル / ディレクトリは root 直下に置くことを禁止する。違反した PR は CI の `tools/check-deps`（cross-language 層分離）と `tools/check-rust-deps`（Rust path 依存検査）で自動検出し、マージ禁止とする。

- `go.mod` / `go.sum` / `go.work` — 必ず `src/tier1/go/` か `tools/<name>/` の内側
- `Cargo.toml` / `Cargo.lock` / `rust-toolchain.toml` — 必ず `src/tier1/rust/` か `tools/<name>/` の内側
- `package.json` / `tsconfig.json` / `pnpm-lock.yaml` — Phase 2 で tier3 を追加する際は `src/tier3/` の内側
- `Makefile` / `justfile` — `scripts/` に移動し、`.sh` で書き直す（task runner を root に置かない）
- `docker-compose.yml` — 必要なら `tests/e2e/docker-compose.yml` など、明確な用途のサブディレクトリに配置
- `.env` / `.env.local` — `.gitignore` 対象とし、リポジトリに commit しない

root を「言語非依存の monorepo 入口」として保つことが、Phase 2 以降の tier2/tier3 追加時の破綻を防ぐ最大の予防策である。

**確定フェーズ**: Phase 0。**対応要件**: NFR-C-NOP-001、DX-CICD-\*。**上流**: DS-SW-COMP-120、DS-SW-COMP-138。
