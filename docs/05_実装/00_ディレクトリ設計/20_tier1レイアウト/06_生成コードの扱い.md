# 06. 生成コードの扱い

本ファイルは Protobuf 生成コード（buf generate の出力）を git commit する方針と、それに伴う CI 検証・diff drift 検出の運用を確定する。DS-SW-COMP-122 を物理配置レベルで展開する。

## 方針サマリ

生成コードは以下の方針で扱う。

1. **commit する**: 生成物を git に含める（commit 対象）
2. **手動編集禁止**: `// Code generated` ヘッダで明示、CI で検知
3. **linguist-generated**: GitHub の言語統計から除外（`.gitattributes` で指定）
4. **drift 検出**: CI で `buf generate` 後の diff が 0 であることを検証
5. **cone に含める**: スパースチェックアウトの各 cone に生成ディレクトリを含める

## なぜ commit するか

生成コードを commit せず、CI 実行時に生成する運用も可能だが、k1s0 では commit する方針を取る。理由は以下。

- **コードレビューで生成差分を確認できる**: 契約変更が各言語 stub にどう反映されたかを PR でレビュー可能
- **ローカル開発の高速化**: 開発者がクローン直後から即座に IDE の補完が効く（buf generate を走らせる必要なし）
- **CI の独立性**: 生成コードが無いと tier1 / tier2 / tier3 の CI で毎回 buf generate が必要になり、build time 増大
- **Git blame が効く**: 「この関数はいつ契約変更で追加されたか」を git blame で追跡可能

ただし commit するとリポジトリサイズが膨らむ副作用がある。k1s0 規模では許容範囲と判定（Phase 1c で再評価）。

## 生成ディレクトリ一覧

buf generate の出力先は以下の通り。言語ごとに独立した `buf.gen.<lang>.yaml`（[02_contracts配置.md](02_contracts配置.md) 参照）で指定する。

| 言語 | 出力先 | 主な用途 |
|---|---|---|
| Go（tier1） | `src/tier1/go/internal/proto/tier1/v1/` | tier1 Go Pod の gRPC サーバ実装 |
| Rust（tier1） | `src/tier1/rust/crates/proto-gen/src/`（flat） | tier1 Rust Pod の gRPC クライアント / サーバ実装 |
| C#（SDK） | `src/sdk/dotnet/src/K1s0.Sdk.Proto/Generated/` | .NET SDK 用 gRPC クライアント |
| Go（SDK） | `src/sdk/go/proto/tier1/v1/` | Go SDK 用 gRPC クライアント |
| TypeScript（SDK） | `src/sdk/typescript/packages/proto/src/` | TS SDK 用 gRPC-Web クライアント |
| Rust（SDK, Phase 2） | `src/sdk/rust/crates/k1s0-sdk-proto/src/gen/v1/` | Rust SDK 用 gRPC クライアント |

Rust（tier1）の出力が「flat」なのは tonic-build の `.out_dir("src")` が proto package 名を連結したフラットファイル（例: `k1s0.tier1.v1.rs` / `k1s0.tier1.internal.v1.rs`）を生成するためである。他言語（Go / C# / TS）は `paths=source_relative` や言語固有の plugin により階層構造を出力する。

## .gitattributes での linguist-generated 指定

`.gitattributes` に以下を追記する。

```
# Protobuf 生成コード（linguist カウント除外）
src/tier1/go/internal/proto/**                              linguist-generated=true
src/tier1/rust/crates/proto-gen/src/**                      linguist-generated=true
src/sdk/dotnet/src/K1s0.Sdk.Proto/Generated/**              linguist-generated=true
src/sdk/go/proto/**                                         linguist-generated=true
src/sdk/typescript/packages/proto/src/**                    linguist-generated=true
src/sdk/rust/crates/k1s0-sdk-proto/src/gen/**               linguist-generated=true
```

## CI での drift 検出

以下のワークフローを `.github/workflows/contracts.yaml` に実装する。

```yaml
name: contracts
on:
  pull_request:
    paths:
      - 'src/contracts/**'
      - 'src/tier1/go/internal/proto/**'
      - 'src/tier1/rust/crates/proto-gen/src/**'
      - 'src/sdk/**/generated/**'
      - 'src/sdk/**/proto/**'
  push:
    branches: [main]
jobs:
  buf-generate-drift:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: bufbuild/buf-setup-action@v1
      - name: buf lint
        working-directory: src/contracts
        run: buf lint
      - name: buf breaking
        working-directory: src/contracts
        run: buf breaking --against 'https://github.com/k1s0/k1s0.git#branch=main,subdir=src/contracts'
        continue-on-error: true  # breaking は警告のみ
      - name: buf generate
        working-directory: src/contracts
        run: |
          for tmpl in buf.gen.go.yaml buf.gen.rust.yaml buf.gen.ts.yaml buf.gen.csharp.yaml; do
            buf generate --template "$tmpl"
          done
      - name: Check drift
        run: |
          if ! git diff --exit-code \
              src/tier1/go/internal/proto \
              src/tier1/rust/crates/proto-gen/src \
              src/sdk/go/proto \
              src/sdk/rust/crates/k1s0-sdk-proto/src/gen \
              src/sdk/typescript/packages/proto/src \
              src/sdk/dotnet/src/K1s0.Sdk.Proto/Generated; then
            echo "::error::Generated code drift detected. Run 'buf generate' locally and commit the changes."
            exit 1
          fi
```

上記は雛形。Phase 1a の CI 実装時点で最適化・高速化を検討する。

## 手動編集禁止の強制

生成コードに以下のヘッダを必須とする。

- Go: `// Code generated by protoc. DO NOT EDIT.`
- Rust: `// Code generated by tonic-build. DO NOT EDIT.`
- C#: `// <auto-generated>`
- TypeScript: `// @generated by protoc-gen-connect-es. DO NOT EDIT.`

CI で各言語のリンタ（`golangci-lint` の `//lint:ignore` / Roslyn analyzer / tslint）が generated タグを認識して、手動編集と推測される diff を検出する。

## 生成コードの cone 包含

スパースチェックアウトの各 cone 定義に生成ディレクトリを含める。

- `tier1-go-dev.txt` : `src/tier1/go/internal/proto/` を含む
- `tier1-rust-dev.txt` : `src/tier1/rust/crates/proto-gen/` を含む
- `tier2-dev.txt` : `src/sdk/dotnet/src/K1s0.Sdk.Proto/Generated/` + `src/sdk/go/proto/` を含む
- `tier3-web-dev.txt` : `src/sdk/typescript/packages/proto/` を含む
- `tier3-native-dev.txt` : `src/sdk/dotnet/src/K1s0.Sdk.Proto/Generated/` を含む

cone 外だと IDE が gRPC stub を認識できず、補完が効かないため。

## 生成ツールのバージョン固定

buf plugin のバージョンは `buf.lock` に固定する。開発者の buf CLI バージョンも `rust-toolchain.toml` と同様に `.tool-versions`（asdf）または `mise.toml` で固定する方針を採用する。

```
# .tool-versions
buf 1.45.0
go 1.22.8
rust 1.85.0
node 20.18.0
dotnet 8.0.403
```

これにより「ローカルで buf generate するとリモート CI と結果が違う」事象を防止する。

## 生成コード削除時の扱い

契約変更で `.proto` のメッセージが削除された場合、生成コードからも自動削除される。ただし git から削除されるのは buf generate が出力しなくなった時点。古い生成コードが残り続ける問題を避けるため、以下の追加ルールを適用する。

- 契約変更 PR の buf generate 後、`git diff --stat` で追加と削除の両方を確認
- 削除された .proto に対応する .pb.go / .rs / .cs / .ts がすべて消えていることを CI で検証

## tier1 / SDK 2 系統生成の同一性検証

DS-SW-COMP-122 の新定義では、`src/contracts/` から tier1 内部（`src/tier1/go/internal/proto/tier1/v1/` / `src/tier1/rust/crates/proto-gen/src/`）と SDK（`src/sdk/<lang>/...`）を **4 言語独立の buf.gen.<lang>.yaml** で生成する。同じ `.proto` からの出力なので、両系統の成果物が意味論的に一致していなければならないが、言語ごとの plugin 設定（特に Go の `paths=source_relative` や optional フィールドの扱い）が僅かに異なると生成結果が乖離する。

この乖離を CI で検出する仕組みとして `tools/ci/validate-proto-parity.sh` を配置する。

```bash
#!/usr/bin/env bash
# tools/ci/validate-proto-parity.sh
# tier1 内部と SDK の Protobuf 生成物が意味論的に一致することを検証する。
set -euo pipefail

# Go 側の比較: tier1/go/internal/proto/tier1/v1/ と sdk/go/proto/tier1/v1/
go run ./tools/ci/proto-parity \
    --left  src/tier1/go/internal/proto/tier1/v1 \
    --right src/sdk/go/proto/tier1/v1 \
    --lang  go

# Rust 側の比較: tier1/rust/crates/proto-gen/src/（flat）と sdk/rust/crates/k1s0-sdk-proto/src/gen/v1/
# tier1 Rust の tonic-build 出力はフラット（k1s0.tier1.v1.rs 等）であり、
# proto-parity は FileDescriptorSet レベルでシンボル比較するためパス差は許容される
go run ./tools/ci/proto-parity \
    --left  src/tier1/rust/crates/proto-gen/src \
    --right src/sdk/rust/crates/k1s0-sdk-proto/src/gen/v1 \
    --lang  rust
```

比較対象は以下の 3 軸。

- **message / enum / service の定義集合が一致** — 片方にしか存在しない型がないこと
- **各フィールドの (name, number, type, cardinality) が一致** — wire-compatible であること
- **service の RPC シグネチャ（method name / request / response / streaming kind）が一致**

形式的には Protobuf の descriptor を両方から抽出し、`google.protobuf.FileDescriptorSet` レベルで比較する（言語固有のコード出力差分ではなく契約レベルの比較）。

CI では `ci-contracts.yml` に組み込み、必須チェックとする。乖離が検出された場合は `buf.gen.<lang>.yaml` の設定差分を調査し、両系統の plugin オプションを揃える。
