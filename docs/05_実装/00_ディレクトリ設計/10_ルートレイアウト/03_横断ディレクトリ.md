# 03. 横断ディレクトリ

本ファイルは `src/` 配下以外でリポジトリルート直下に置く 7 つの横断ディレクトリを定義する。`infra/` / `deploy/` / `ops/` / `tools/` / `tests/` / `examples/` / `third_party/` の 7 つで、いずれも `src/` 配下のソースコードと同じ責務レイヤには属さない独立領域である。

## 横断配置が必要な理由

src 配下に全てを押し込むと、以下の問題が発生する。

- Kubernetes YAML / ArgoCD Application / Runbook は tier1 / tier2 / tier3 のどれにも属さない横断資産であり、src/tier1/infra/ 等の tier 配下に押し込むと所有権が歪む（[ADR-DIR-002](../../../02_構想設計/adr/ADR-DIR-002-infra-separation.md)）
- tier 横断の E2E テストや contract テストを特定の tier 配下に置くと、テスト資産の所有権が単一 tier に縛られる
- OSS の vendoring は全 tier に影響し得る横断資産であり、どこかの tier 配下に置くべきではない

本ファイルは横断資産をルート直下の 7 ディレクトリに整理する。

## 7 横断ディレクトリ

### infra/

**役割**: Kubernetes クラスタ素構成。

**内部構造**:

```
infra/
├── k8s/                        # Kubernetes bootstrap / namespaces / networking / storage
├── mesh/                       # istio-ambient / envoy-gateway
├── dapr/                       # Dapr control-plane / components
├── data/                       # cloudnativepg / kafka / valkey / minio
├── security/                   # keycloak / openbao / spire / cert-manager / kyverno
├── observability/              # LGTM + Pyroscope + OTel Collector
├── feature-management/         # flagd
├── scaling/                    # KEDA
└── environments/               # dev / staging / prod の環境差分
```

**所有権**: `@k1s0/sre-ops`（サブディレクトリによっては security-team や tier1-go との共有）

**対応**: [50_infraレイアウト/](../50_infraレイアウト/)

### deploy/

**役割**: GitOps 配信定義。ArgoCD / Kustomize / Rollouts / OpenTofu。

**内部構造**:

```
deploy/
├── apps/                       # ArgoCD Application / ApplicationSet
├── charts/                     # 共通 Helm charts
├── kustomize/                  # base + overlays
├── rollouts/                   # Argo Rollouts AnalysisTemplate
├── opentofu/                   # ベアメタルプロビジョン
└── image-updater/              # Argo CD Image Updater 設定
```

**所有権**: `@k1s0/gitops-team` + `@k1s0/sre-ops`

**対応**: [60_operationレイアウト/](../60_operationレイアウト/)

### ops/

**役割**: 運用領域。Runbook / Chaos / DR / オンコール / 負荷試験。

**内部構造**:

```
ops/
├── runbooks/                   # Runbook 実装
├── chaos/                      # Litmus シナリオ
├── dr/                         # DR スクリプト
├── oncall/                     # オンコール運用手順
├── load/                       # 負荷試験スクリプト
└── scripts/                    # 運用スクリプト
```

**所有権**: `@k1s0/sre-ops`

**対応**: [60_operationレイアウト/](../60_operationレイアウト/)

### tools/

**役割**: 横断ツール。開発者や CI が実行する補助プログラム。

**内部構造**:

```
tools/
├── devcontainer/               # 役割別 Dev Container プロファイル
├── local-stack/                # kind / k3d ローカルクラスタ起動
├── codegen/                    # buf / openapi / scaffold 補助
├── sparse/                     # スパースチェックアウト切替 CLI
├── ci/                         # CI helper
├── git-hooks/                  # pre-commit hook
└── migration/                  # データ移行ツール
```

**所有権**: `@k1s0/platform-team`（一部は sre-ops と共有）

**対応**: [70_共通資産/01_tools配置.md](../70_共通資産/01_tools配置.md)

### tests/

**役割**: tier 横断テスト資産。tier 内部の unit test は各 tier 配下で管理する。

**内部構造**:

```
tests/
├── e2e/                        # Playwright / Cypress などの E2E
├── contract/                   # Pact による契約テスト
├── integration/                # testcontainers 統合テスト
├── fuzz/                       # Rust cargo-fuzz / Go fuzz テスト
├── golden/                     # ゴールデンファイルテスト
└── fixtures/                   # テスト用 fixture
```

**所有権**: `@k1s0/platform-team`（contract は contract-reviewers と共有）

**対応**: [70_共通資産/02_tests配置.md](../70_共通資産/02_tests配置.md)

### examples/

**役割**: 動作する最小実装例。Netflix Paved Roads に倣い、コピーして使える状態を維持する。

**内部構造**:

```
examples/
├── tier2-dotnet-service-hello/
├── tier2-go-service-hello/
├── tier3-web-hello/
├── tier3-native-hello/
└── sdk-integration-hello/
```

**所有権**: `@k1s0/platform-team` + `@k1s0/tier2-dev` + `@k1s0/tier3-web`

**対応**: [70_共通資産/03_examples配置.md](../70_共通資産/03_examples配置.md)

### third_party/

**役割**: vendored OSS。ライセンス・コンプライアンス重要領域。

**内部構造**:

```
third_party/
├── <vendor>/<project>/             # 各 OSS の vendored コピー
│   ├── LICENSE                     # 元のライセンス
│   ├── PATCHES/                    # 適用したパッチ
│   └── README.md                   # vendoring 理由・元のバージョン
```

**所有権**: `@k1s0/arch-council` + `@k1s0/compliance-team` + `@k1s0/security-team`

**対応**: [70_共通資産/04_third_party配置.md](../70_共通資産/04_third_party配置.md)

## 7 横断ディレクトリの整合

各横断ディレクトリは相互に以下の関係を持つ。

- `infra/` が宣言するクラスタリソースを `deploy/apps/` の ArgoCD Application が参照して配信する
- `deploy/charts/tier1/` が `src/tier1/` のソースをビルドした image を参照する（`values.yaml` で image tag 指定）
- `ops/runbooks/` は `infra/` のリソースや `deploy/` の Application の状態変化に対応する手順を定義する
- `tools/sparse/` は `.sparse-checkout/roles/` の cone 定義を読み込んで `git sparse-checkout set` を実行する
- `tests/e2e/` は `deploy/` で展開されたクラスタに対して E2E シナリオを実行する
- `examples/` は `src/tier2/templates/` の雛形を元にした実稼働サービスであり、Golden Path として参照される
- `third_party/` は `src/` 配下の任意の言語から参照される可能性を持つ

## 新規横断ディレクトリを追加する際の判断基準

ルート直下に新ディレクトリを追加する提案は、以下の基準で判定する。

- **tier 横断**: 特定 tier に帰属しない資産であること
- **責務明確**: 既存 7 横断ディレクトリのいずれにも属さないこと
- **CODEOWNERS 可能**: GitHub team で責任割当が可能であること
- **sparse cone 可能**: いずれかの cone 定義に属すか、独立 cone を新設できること
- **原則違反なし**: 命名衝突（IMP-DIR-ROOT-006）や最大深度（IMP-DIR-ROOT-003）に違反しないこと

5 基準のうち 1 つでも満たさない場合、追加せずに既存ディレクトリへの吸収を検討する。
