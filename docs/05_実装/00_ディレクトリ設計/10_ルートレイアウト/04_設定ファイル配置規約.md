# 04. 設定ファイル配置規約

本ファイルはリポジトリ全体の動作を決める設定ファイル（`.gitattributes` / `.gitignore` / `.editorconfig` / `.github/` 配下 / `.vscode/` / `.devcontainer/` / `.sparse-checkout/` など）の配置規約を定める。設定ファイルは目立たないがリポジトリの挙動を強く拘束するため、配置ルールを確定させないと混在と矛盾が発生しやすい。

## 本ファイルの目的

設定ファイルには以下のカテゴリがある。

- **Git 関連**: `.gitattributes` / `.gitignore` / `.git-blame-ignore-revs`
- **GitHub 関連**: `.github/` 配下
- **エディタ関連**: `.vscode/` / `.editorconfig`
- **開発環境関連**: `.devcontainer/`
- **Claude / Agents 関連**: `.claude/`
- **スパースチェックアウト関連**: `.sparse-checkout/`

これらを無秩序に配置すると、新規参加者が「どの設定が効くのか」「自分のマシンで `.vscode/` を編集してよいのか」が判らなくなる。本ファイルは配置規約を統一する。

## 設定ファイルの配置ルール

### ルール 1: 影響範囲に応じた階層選択

設定ファイルの影響範囲が以下のいずれに該当するかで配置階層を決める。

- **リポジトリ全体**: ルート直下
- **ディレクトリ固有**: 該当ディレクトリ直下
- **特定ユーザー / 端末のみ**: `.gitignore` 対象（リポジトリに含めない）

たとえば `.gitattributes` はリポジトリ全体のため最上位、`tsconfig.json` は TypeScript workspace 内に限定されるため `src/tier3/web/` 配下、個人の IDE カスタマイズは `.gitignore` に記載してリポジトリに含めない。

### ルール 2: `.github/` 配下は GitHub 固有設定のみ

`.github/` 配下に置くのは以下の 4 種類のみ。

- `workflows/` : GitHub Actions ワークフロー
- `ISSUE_TEMPLATE/` : Issue テンプレート
- `PULL_REQUEST_TEMPLATE.md` : PR テンプレート
- `CODEOWNERS` : 所有権定義（`.github/CODEOWNERS` または ルート `CODEOWNERS` のどちらかに配置。GitHub は両方を認識するが、本プロジェクトは `.github/CODEOWNERS` に統一する）

他の CI（GitLab / CircleCI）用設定は本プロジェクトでは採用しないため、.gitlab-ci.yml 等は存在しない。将来 self-hosted CI（GitHub Actions Self-Hosted Runner 以外）を採用する場合は、その時点で配置規約を追加する。

### ルール 3: `.vscode/` は共有設定のみコミット

`.vscode/` 配下は個人設定と共有設定が混在しやすい。以下のファイルのみコミット対象とし、他は `.gitignore` で除外する。

- `.vscode/settings.shared.json` : 共有設定（拡張機能の設定など）
- `.vscode/extensions.json` : 推奨拡張機能リスト
- `.vscode/launch.shared.json` : 共有デバッグ設定
- `.vscode/tasks.shared.json` : 共有タスク

`.vscode/settings.json` / `.vscode/launch.json` / `.vscode/tasks.json` は個人設定として `.gitignore` で除外。開発者は `.shared.json` から `settings.json` に手動で copy するか、wrapper script で自動適用する。

### ルール 4: `.devcontainer/` は役割別プロファイル

`.devcontainer/` 配下は役割別 Dev Container プロファイルを配置する。

```text
.devcontainer/
├── tier1-rust/
│   └── devcontainer.json
├── tier1-go/
│   └── devcontainer.json
├── tier2/
│   └── devcontainer.json
├── tier3-web/
│   └── devcontainer.json
├── tier3-native/
│   └── devcontainer.json
├── infra-ops/
│   └── devcontainer.json
└── full/
    └── devcontainer.json
```

VS Code の "Reopen in Container" 実行時に `.devcontainer/` 直下のプロファイルを選択する形式を取る。対応 cone 定義（`.sparse-checkout/roles/tier1-rust-dev.txt` 等）を自動適用する postCreateCommand を組み込む。

### ルール 5: `.claude/` は Claude Code セッション用

`.claude/` 配下には Claude Code が使うセッション設定（slash commands / custom skills 等）を配置する。既存ファイルとの整合を維持しつつ、リリース時点 以降の追加要件に応じて拡張する。

### ルール 6: `.sparse-checkout/` は cone 定義専用

`.sparse-checkout/` 配下には以下のみ配置する。

```text
.sparse-checkout/
├── README.md                   # cone 定義の読み方
└── roles/                      # 10 種類の役割別 cone 定義
    ├── tier1-rust-dev.txt
    ├── tier1-go-dev.txt
    ├── tier2-dev.txt
    ├── tier3-web-dev.txt
    ├── tier3-native-dev.txt
    ├── platform-cli-dev.txt
    ├── sdk-dev.txt
    ├── infra-ops.txt
    ├── docs-writer.txt
    └── full.txt
```

各 `.txt` ファイルは cone mode 形式のディレクトリ列挙（1 行 1 ディレクトリ）。詳細は [../80_スパースチェックアウト運用/02_役割別cone定義.md](../80_スパースチェックアウト運用/02_役割別cone定義.md) 参照。

### ルール 7: ディレクトリ固有設定の配置

以下はディレクトリ固有設定として、該当ディレクトリ直下に置く。

- `src/contracts/buf.yaml` / `buf.gen.*.yaml`（4 ファイル: go / rust / ts / csharp） / `buf.lock` : Protobuf 契約専用
- `src/tier1/rust/Cargo.toml` / `Cargo.lock` / `rust-toolchain.toml` : Rust workspace
- `src/tier1/go/go.mod` / `go.sum` : Go module
- `src/sdk/go/go.mod` : SDK Go module（独立）
- `src/sdk/dotnet/Directory.Build.props` : .NET ソリューション共通
- `src/sdk/typescript/package.json` / `pnpm-workspace.yaml` : npm workspace
- `src/tier3/web/pnpm-workspace.yaml` : Web workspace

これらをルート直下に置くと、各言語のビルドツールが全ファイルを検索対象に含めてしまい、本来ビルドしない領域も巻き込む副作用が発生する。

### ルール 8: 秘密は絶対にコミットしない

以下は `.gitignore` で確実に除外し、リポジトリに含まない。

- `.env` / `.env.*`（`.env.example` は除く）
- `credentials.json` / `token.json` / `*.key` / `*.pem`
- `kubeconfig` 実体ファイル（`kubeconfig.example` は除く）
- `.npmrc` / `.pypirc` で認証情報を含むもの

CI で secrets scanning（TruffleHog / gitleaks）を必須化し、PR で検出された場合はマージブロックする。

## 設定ファイル間の衝突解決

以下の衝突パターンは既知のため、回避ルールを定める。

### .editorconfig と言語別 formatter

`.editorconfig` はエディタ共通のインデント・改行設定を定義し、言語別 formatter（rustfmt / gofmt / prettier / dotnet-format）と整合させる。衝突時は言語別 formatter の設定を優先する（`.editorconfig` は最大公約数の設定に留める）。

### .gitattributes の linguist-generated と CODEOWNERS

生成コード（`src/tier1/go/internal/proto/**` / `src/tier1/rust/crates/proto-gen/src/**` / `src/sdk/**/proto/**` / `src/sdk/**/generated/**` 等）は `linguist-generated=true` で GitHub の言語統計から除外する。CODEOWNERS は生成コードに対しても owner を設定するが、生成コードのみの PR は自動マージを許容する CI ルールを別途設定する。

### .devcontainer/ と .vscode/ の二重管理

Dev Container 内の VS Code 設定は `.devcontainer/<role>/settings.json` で定義し、ホスト側の `.vscode/settings.shared.json` と重複する場合は Dev Container 側を優先する。重複を避けるため、共通設定は `.vscode/settings.shared.json` に集約し、Dev Container では追加のコンテナ固有設定のみ書く。
