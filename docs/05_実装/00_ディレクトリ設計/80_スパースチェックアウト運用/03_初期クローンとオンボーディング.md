# 03. 初期クローンとオンボーディング

本ファイルは新規開発者が k1s0 リポジトリを初回 clone する手順と、役割選択・開発環境起動までのオンボーディングを規定する。

## 目標

- 新規開発者が clone → role 選択 → 開発環境起動までを 30 分以内で完了
- ローカル環境と CI で同じ設定が再現される
- Windows / macOS / Linux の 3 OS で同じ手順で動作

## 初回セットアップ手順

### Step 1: リポジトリ clone

```bash
# partial clone + sparse checkout で最小構成を取得
git clone \
    --filter=blob:none \
    --sparse \
    https://github.com/k1s0/k1s0.git

cd k1s0
```

- `--filter=blob:none`: blob を遅延取得（初回 clone 時の転送量を 9 割削減）
- `--sparse`: sparse-checkout を初期化し、トップレベルのみを checkout 状態にする。後続 Step でロール別 cone を適用するまでワーキングツリーは top-level のみ

`--no-checkout` は併用しない。`--sparse` が既にトップレベルのみの最小チェックアウトを行うため、`--no-checkout` を重ねても実効差はなく、両フラグの優先関係が Git バージョン依存で意図が曖昧になるリスクを避ける。

### Step 2: sparse index 有効化

```bash
git sparse-checkout init --cone
git config core.sparseCheckoutCone true
```

sparse index（Git 2.37+）が有効になり、`.git/index` が圧縮される。

### Step 3: 役割選択

```bash
# 利用可能な役割を確認
./tools/sparse/list-roles.sh

# 例: tier1-rust-dev を選択
./tools/sparse/checkout-role.sh tier1-rust-dev
```

checkout-role.sh は内部で以下を実行:

1. `.sparse-checkout/roles/tier1-rust-dev.txt` を読み込み
2. `git sparse-checkout set --stdin` に渡す
3. 必要なディレクトリを checkout

### Step 4: Dev Container 起動

VS Code で以下を実行:

```
Ctrl+Shift+P → "Dev Containers: Reopen in Container"
→ tools/devcontainer/profiles/tier1-rust-dev/devcontainer.json を選択
```

コンテナ起動時に post-create.sh が走り、言語固有の依存解決（cargo fetch 等）が完了する。

### Step 5: 動作確認

役割毎の動作確認コマンド:

- **tier1-rust-dev**: `cargo build` / `cargo test`
- **tier1-go-dev**: `go build ./...` / `go test ./...`
- **tier2-dev**: `dotnet build` / `dotnet test`
- **tier3-web-dev**: `pnpm install` / `pnpm test`
- **tier3-native-dev**: `dotnet build`（MAUI workload）
- **infra-ops**: `kubectl cluster-info`（local-stack 起動後）

## オンボーディング Runbook

詳細手順は `ops/runbooks/daily/onboarding.md` に記述。Backstage の TechDocs からも閲覧可能。

主要な Runbook 項目:

1. 権限取得（GitHub org 招待、Keycloak ユーザ作成）
2. 開発 PC のセットアップ（Docker Desktop / Rancher Desktop / VS Code）
3. リポジトリ clone と役割選択
4. Dev Container 起動
5. Keycloak 認証設定（開発者用 client_id 発行）
6. 最初の PR 作業（チュートリアル issue）

## 自動セットアップスクリプト

手動手順をまとめた自動スクリプト `tools/sparse/bootstrap-developer.sh` を用意。

```bash
#!/usr/bin/env bash
set -euo pipefail

echo "k1s0 developer bootstrap"
echo "========================"

# 役割選択
echo "Available roles:"
ls -1 .sparse-checkout/roles/ | sed 's/\.txt$//' | nl
read -p "Select role number: " ROLE_NUM
ROLE=$(ls -1 .sparse-checkout/roles/ | sed 's/\.txt$//' | sed -n "${ROLE_NUM}p")

echo "Applying role: $ROLE"
./tools/sparse/checkout-role.sh "$ROLE"

# Dev Container 情報表示
echo ""
echo "Next step: Open VS Code in this directory and run:"
echo "  Ctrl+Shift+P → Dev Containers: Reopen in Container"
echo "  → Select: tools/devcontainer/profiles/${ROLE}/devcontainer.json"
```

## 想定所要時間

| ステップ | 所要時間 | ネットワーク帯域 |
|---|---|---|
| git clone（partial + sparse） | 1-2 分 | 100 Mbps で 200 MB |
| role 選択 & checkout | 30 秒 | - |
| Dev Container 初回ビルド | 15-20 分 | 2 GB 前後 |
| 言語依存解決（cargo / go / pnpm / dotnet） | 5-10 分 | 500 MB-1 GB |
| **合計** | **25-35 分** | **3 GB 前後** |

2 回目以降はキャッシュが効き、5 分以内に起動可能。

## Windows 環境特有の注意

- Git for Windows の `core.longpaths=true` 必須（パス長 260 文字制限回避）
- `core.autocrlf=false` + `.gitattributes` による改行管理
- Docker Desktop WSL2 バックエンド前提（Hyper-V バックエンドはサポート外）
- cone mode 内のディレクトリ名は全て小文字（NTFS の大小文字不区別回避）

## macOS 環境特有の注意

- Docker Desktop for Mac または Rancher Desktop
- Xcode Command Line Tools 必須（Git が同梱）
- MAUI 開発者は Xcode 15 も必要（iOS シミュレータ / archive ビルド）
