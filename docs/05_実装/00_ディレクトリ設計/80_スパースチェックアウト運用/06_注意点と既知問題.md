# 06. 注意点と既知問題

本ファイルは sparse-checkout cone mode 運用で遭遇する注意点と、既知問題への対処を記録する。

## submodule は原則不採用

k1s0 リポジトリは `.gitmodules` を持たない。submodule は以下の理由で不採用。

- cone mode と組合せた際の挙動が不安定（sparse-checkout で submodule directory を除外しても `.gitmodules` の entry が残る）
- partial clone と submodule の組合せは十分にテストされていない
- 依存管理は各言語パッケージマネージャで統一する方が保守容易

third_party/ で OSS フォークを取り込む際も subtree や vendoring で行い、submodule は使わない。

## Git LFS は条件付き採用（Phase 1c で判定）

大きなバイナリ（tar.gz テストフィクスチャ、drawio 以外の画像、.NET MAUI の icon）は Git LFS で管理する候補。ただし以下の制約があり Phase 1c で採用可否を判断。

- LFS と partial clone の相性: `filter=blob:none` 指定時、LFS pointer は取得されるが実体は遅延取得
- LFS と sparse-checkout の相性: sparse で除外されたディレクトリの LFS オブジェクトは checkout されないが、fetch は発生
- LFS サーバのストレージコスト: GitHub では別料金、セルフホストなら MinIO 上に実装可能

暫定方針: Phase 1a-1b は LFS 不採用、大きなバイナリは避ける（PNG は SVG 変換、tar.gz は rerun script で生成等）。

## 生成コードは全て commit

buf generate / openapi-generator / handlebars scaffold の生成コードは全て commit する。ADR-DIR-001 で決定。

`.gitattributes` で `linguist-generated=true` を付与し GitHub の PR diff から自動除外。

```
# .gitattributes（詳細は 20_tier1レイアウト/06_生成コードの扱い.md 参照）
src/tier1/go/internal/proto/**                      linguist-generated=true
src/tier1/rust/crates/proto-gen/**                   linguist-generated=true
src/sdk/dotnet/src/K1s0.Sdk.Proto/Generated/**      linguist-generated=true
src/sdk/go/proto/**                                 linguist-generated=true
src/sdk/typescript/packages/proto/src/**            linguist-generated=true
src/sdk/rust/crates/k1s0-sdk-proto/src/gen/**       linguist-generated=true
```

## Windows 大小文字不区別問題

Windows NTFS は大小文字不区別（一部設定を除く）。macOS APFS はデフォルト大小文字不区別。Linux ext4 は大小文字区別。この差で以下の事故が起きる:

- macOS で `Types.ts` と `types.ts` を別物として commit
- Windows で clone すると一方が消失

対策:

1. **全ディレクトリ・ファイル名を小文字に統一**（`src/tier1/rust` など）
2. CI で `tools/ci/case-collision-check.sh` により大小文字重複を検出
3. `.gitattributes` で `ignorecase=false` は設定しない（OS ごとの動作差は運用で管理）

## パス長制限（Windows）

Git for Windows の `core.longpaths=false`（デフォルト）だと、260 文字以上のパスで checkout 失敗。

対策:

```bash
git config --global core.longpaths true
```

開発者初回セットアップで必須設定。`tools/sparse/bootstrap-developer.sh` で自動設定。

## 改行コード（autocrlf）

Windows と Unix 混在環境で改行が LF / CRLF 混在する問題。

対策:

1. `.gitattributes` で明示的に指定:

```
* text=auto eol=lf
*.bat text eol=crlf
*.cmd text eol=crlf
*.ps1 text eol=crlf
```

2. 開発者設定:

```bash
git config --global core.autocrlf false
```

## sparse-checkout と post-checkout hook

sparse-checkout で cone を絞ると、post-checkout hook が想定外の挙動をする場合がある。例: Husky の pre-commit hook をサブディレクトリ削除で消すと以降動作しない。

対策:

- `.git/hooks/` は sparse 対象外のため削除されない（OK）
- ただし `tools/git-hooks/install.sh` で symlink を貼る時に、対象スクリプトが sparse cone 内にあることを確認

## Dev Container と sparse-checkout

Dev Container が sparse cone 外のパスを bind mount する場合、ホスト側で該当パスが存在せず起動失敗する。

対策:

- `tools/devcontainer/profiles/<role>/devcontainer.json` の mounts 指定は、当該 role の cone 内パスのみ使用
- ワイルドカードや `${localWorkspaceFolder}` を使う場合は cone 内に限定

## IDE インデックス失敗

Rust Analyzer / Go / TypeScript / C# の LSP が sparse cone 外のファイルを「存在しない」と判定し、エラー表示する場合がある（ただし実害はない）。

対処:

- LSP を再起動
- プロジェクトルートの `.vscode/settings.json` で role 毎に「workspace folder を絞る」設定
- JetBrains IntelliJ の場合は "Content Root" を role cone 内に限定

## git blame / log で sparse 外を見る

sparse cone 外のファイルも `git log -- <path>` で history を表示できる。ワーキングツリーには無いが Git は全 history を持っているため。

```bash
# sparse 外のファイルの blame
git blame --follow -- src/tier3/web/apps/portal/src/app/page.tsx
```

## 役割定義の一貫性検証（role parity）

`.sparse-checkout/roles/*.txt`（10 役）と `tools/devcontainer/profiles/*/devcontainer.json`（10 役）は、役割追加時に片方が追従されない事故が起きやすい。以下の 3 集合が完全一致することを CI で強制する。

- `.sparse-checkout/roles/*.txt` の basename 集合
- `tools/devcontainer/profiles/*/devcontainer.json` の親ディレクトリ名集合
- `docs/05_実装/00_ディレクトリ設計/80_スパースチェックアウト運用/02_役割別cone定義.md` の役割一覧表に登場する役割名集合

CI script は `tools/ci/validate-role-parity.sh`:

```bash
#!/usr/bin/env bash
# tools/ci/validate-role-parity.sh
# cone 定義・Dev Container・ドキュメントの役割集合が一致することを検証
set -euo pipefail

cone_roles=$(ls .sparse-checkout/roles/ | sed 's/\.txt$//' | sort)
dc_roles=$(ls tools/devcontainer/profiles/ | sort)
doc_roles=$(grep -oE '^\| [a-z][a-z0-9-]+ \|' \
    docs/05_実装/00_ディレクトリ設計/80_スパースチェックアウト運用/02_役割別cone定義.md \
    | tr -d '| ' | sort -u)

if [[ "$cone_roles" != "$dc_roles" ]] || [[ "$cone_roles" != "$doc_roles" ]]; then
    echo "::error::Role set mismatch detected"
    diff <(echo "$cone_roles") <(echo "$dc_roles") || true
    diff <(echo "$cone_roles") <(echo "$doc_roles") || true
    exit 1
fi
echo "Role parity check passed: $(echo "$cone_roles" | wc -l) roles in sync"
```

`.github/workflows/ci-main.yml` の `changes` job と並走させ、`.sparse-checkout/` / `tools/devcontainer/profiles/` / `02_役割別cone定義.md` のいずれかが変更された PR で必須チェックとする。
