# 06. テストとフィクスチャ配置

本ファイルはリポジトリ root 直下の `tests/` 配下の詳細ファイル配置を方式として固定化する。上流は概要設計 [50_開発者体験方式設計/01_テスト戦略方式](../../04_概要設計/50_開発者体験方式設計/01_テスト戦略方式.md)・[70_開発者体験方式設計/01_CI_CD方式](../../04_概要設計/70_開発者体験方式設計/01_CI_CD方式.md)と、要件定義 `DX-TEST-*` / `NFR-B-PERF-*` / `DX-GP-*`（ゴールデンパス）で、本ファイルは E2E シナリオ・契約検証・負荷試験・フィクスチャ・ゴールデンファイルの物理配置を確定させる。

## 本ファイルの位置付け

概要設計は「テストピラミッド（unit / integration / contract / E2E / load）」を方式として提示したが、各テストカテゴリの **物理配置** は未確定だった。一般に Go / Rust プロジェクトでは unit テストを `*_test.go` や `#[cfg(test)]` で実装ファイルに隣接させる一方、E2E や契約テストは `tests/` に集約する流儀と、`src/` 配下に混在させる流儀が並立し、チームごとに判断がぶれがちである。

本ファイルは k1s0 における分離境界を次のとおり固定する。第一に、**unit / 小さい integration テスト**は `src/tier1/<lang>/` の実装ソース隣接に置く。Go は `*_test.go`、Rust は `#[cfg(test)] mod tests` または crate 内 `tests/`。第二に、**Pod 横断で実行する E2E・Consumer-Driven Contract・負荷試験・ゴールデンファイル**は repository root 直下の `tests/` に集約する。前者は「コード変更の直後に走り、コミット前の品質ゲート」として機能し、後者は「複数 Pod を起動した状態でのみ成立する品質ゲート」として CI の nightly / release ワークフローで走る。配置を物理的に分離することで、`go test ./src/...` や `cargo test --workspace` が軽量に保たれ、ローカル開発の iteration サイクルが遅くならない。

本ファイルで採番する設計 ID は `DS-IMPL-DIR-161` 〜 `DS-IMPL-DIR-180` の 20 件である。

## tests/ 全体構造

```

│   ├── README.md
│   ├── k6/                               # k6 スクリプト（TypeScript）
│   │   ├── state_p99.ts                  # STATE の p99 測定
```

ディレクトリは 5 つの責務で分かれる。`e2e/` は複数 Pod + Dapr sidecar + バックエンド OSS を起動した統合シナリオ、`contract/` は Pact による Provider / Consumer 契約検証、`load/` は k6 / Locust による性能回帰、`fixtures/` は各テストカテゴリが共有する入力データ、`golden/` はスナップショット比較用の期待出力である。配置を分けることで、CI ワークフローはカテゴリ単位で並列起動でき、変更差分に応じて必要なカテゴリのみ実行する戦略（例: infra 変更時は `e2e/` のみ再実行）を取れる。

## DS-IMPL-DIR-161 tests/ の分離境界

`tests/` を repository root 直下に置く理由は、Go / Rust の **実装パッケージ解決から分離**するためである。Go では `tests/e2e/` を独立した Go module（`go.mod`）とし、`src/tier1/go/` の module とは別にする。これにより `go test ./src/...` を実行しても E2E テストは起動せず、開発者のローカル iteration が高速に保たれる。Rust 側は `tests/` が Cargo workspace `members` に含まれないため、`cargo test --workspace` でも自動起動しない。

一方、概要設計 [DS-DEVX-TEST-\*] で定義された unit / 小さい integration テストは実装ソース隣接に置く。Go は `src/tier1/go/internal/pods/<pod>/service/*_test.go`（03 章 DS-IMPL-DIR-043 の `pods/<pod>/` 配下）、Rust は `src/tier1/rust/crates/pods/<pod>/src/**/*.rs` の `#[cfg(test)]` または `src/tier1/rust/crates/pods/<pod>/tests/*.rs`（04 章 DS-IMPL-DIR-082 の `crates/pods/` 配下）。shared 層（Go `internal/shared/` / Rust `crates/shared/`）のテストも各 crate / パッケージ隣接に配置する。この分離により、「コミット前の unit / integration」と「nightly の E2E / load」が CI 上で別ジョブとなり、失敗の切り分けが容易になる。

**確定フェーズ**: Phase 1a。**対応要件**: DX-TEST-001、DX-TEST-002。**上流**: DS-SW-COMP-120（tier1 トップレベル）、DS-DEVX-TEST-\*。

## DS-IMPL-DIR-162 tests/ の独立 Go module

`tests/e2e/` と `tests/contract/providers/` は Go 製テストで、リポジトリ root の `tests/go.mod` を持つ独立 module とする。main module（`src/tier1/go/go.mod`）からは `replace` ディレクティブで本 repo 内パッケージを参照する。これにより以下の利点がある。

1. E2E テストだけが依存する重量 library（`sigs.k8s.io/e2e-framework`、`github.com/pact-foundation/pact-go`）を main module の依存ツリーから切り離せる
2. main module の Go version / 依存バージョンと、テスト side の必要バージョンを独立に bump できる
3. `go mod tidy` の実行時間が main module 側で短縮される

具体的には `tests/e2e/go.mod` と `tests/contract/go.mod` の 2 つに分割する。`tests/e2e/` 配下のシナリオは scenario ごとに `_test.go` とし、`go test ./tests/e2e/scenarios/...` で全シナリオを実行できるようにする。

**確定フェーズ**: Phase 1a。**対応要件**: DX-TEST-\*。**上流**: DS-DEVX-TEST-\*。

## DS-IMPL-DIR-163 e2e/framework の共通責務

`tests/e2e/framework/` は以下 4 モジュールに分割する。分割の理由は、「クラスタ起動」「Dapr sidecar 起動」「フィクスチャ読込」「アサーション」がそれぞれ独立に差し替え可能な関心事だからである。

- `cluster.go`: kind / k3d クラスタの起動・破棄、Helm Chart インストール。CI では kind、開発者ローカルでは k3d を選択可能とする
- `dapr.go`: Dapr sidecar の起動と Dapr Components の注入。Phase 1a 時点では State Store / Secret Store の 2 種類だけ
- `fixture.go`: `tests/fixtures/` からテストデータを読み込み、テスト前にバックエンド（Postgres / OpenBao）にロードする
- `assertion.go`: ゴールデンファイル比較、gRPC レスポンスの構造比較、構造化ログの検索アサーション

シナリオ側は framework API のみを使い、`k8s.io/client-go` や `github.com/dapr/go-sdk` を直接使わない。これにより framework の内部実装を差し替えても既存シナリオが壊れない。

**確定フェーズ**: Phase 1a。**対応要件**: DX-TEST-001、DX-TEST-002。**上流**: DS-DEVX-TEST-\*。

## DS-IMPL-DIR-164 e2e/scenarios のシナリオ命名と粒度

シナリオは `<api>_<観点>_test.go` の形式で命名する。1 ファイル 1 シナリオを原則とし、300 行制限を超える前にシナリオを分割する。Phase 1a 時点では 7 シナリオ（STATE CRUD / WORKFLOW 長時間 / DECISION ポリシー / AUDIT WORM / PII マスキング / SECRET ローテーション / ゴールデンパス）を用意する。

ゴールデンパス（`golden_path_test.go`）は `DX-GP-001` と 1:1 で対応する。受付 → 決裁 → 監査ログ出力の一連フローを 1 シナリオで貫通させ、「tier1 API 11 個のうち 7 個が組み合わさった時に Phase 1b が成立する」ことを CI で毎回検証する。Phase 2 以降はシナリオを追加し、Canary リリース時の AnalysisTemplate からも参照する。

**確定フェーズ**: Phase 1a。**対応要件**: DX-GP-001、DX-TEST-003。**上流**: DS-DEVX-TEST-\*、DS-BIZ-GP-\*。

## DS-IMPL-DIR-165 e2e/testdata とシナリオ固有データ

`tests/e2e/testdata/` はシナリオ固有のリクエスト / 期待レスポンスを置く。`requests/<scenario>/<step>.json` と `expected/<scenario>/<step>.json` の命名規約で、シナリオコードと 1:1 に対応させる。

fixtures（後述 DS-IMPL-DIR-169）との違いは、**testdata は特定シナリオ専用**、**fixtures は複数シナリオ / カテゴリで共有**する点。これは Go の `testdata/` 慣習（`go test` が自動的に ignore する）に従い、シナリオ作者が気軽に追加できる。一方、fixtures はバージョン管理と整合性検証の対象とし、変更には CODEOWNERS レビューを要する。

**確定フェーズ**: Phase 1a。**対応要件**: DX-TEST-\*。**上流**: Go testdata 慣習。

## DS-IMPL-DIR-166 contract/ の Provider / Consumer 分離

`tests/contract/` は Pact による Consumer-Driven Contract テストで、k1s0 tier1 は **Provider** 側である（tier2 アプリが Consumer）。ディレクトリは以下で分ける。

- `pacts/`: Pact ブローカー（Phase 1c 導入）から pull した契約 JSON。CI 実行時に取得し、git 管理しない（`.gitignore`）
- `providers/`: Provider 側検証コード。pact ファイルを再生し、tier1 API が契約どおりに応答するか検証する
- `consumers/`: tier2 側の Consumer テストがどう生成するかの参照実装と、pact publishing の手順書

Phase 1a 時点では Pact ブローカーを立てず、ファイルベースで pact JSON を `tests/contract/pacts/` にコミットする暫定運用とする（tier2 が存在しないため）。Phase 1b で tier2 が追加された時点で Pact ブローカー導入に切り替える。

**確定フェーズ**: Phase 1a（暫定）、Phase 1c（Pact ブローカー）。**対応要件**: DX-TEST-004、FR-T1-\*。**上流**: DS-DEVX-TEST-\*、DS-IMPL-DIR-021（contracts）。

## DS-IMPL-DIR-167 load/k6 と load/locust の使い分け

負荷試験ツールは目的別に 2 種類を併用する。k6（TypeScript スクリプト）は CI での性能回帰検出用、Locust（Python スクリプト）は長時間 soak 試験用である。

- `k6/`: 短時間（5 分以内）の ramp-up / steady / ramp-down。CI で nightly 実行し、p50 / p95 / p99 レイテンシと RPS を Prometheus に記録
- `locust/`: 長時間（24 時間以上）の soak 試験。メモリリーク・コネクションリーク検出用。Phase 1c 以降のリリース前に実行

k6 スクリプトは TypeScript で書き、`@types/k6` で型補完を効かせる。共通 ramp-up ロジック（`ramp_up.ts`）は各 API 個別スクリプトから import する。Locust は 1 ファイル `soak.py` で start、Phase 1c 以降に拡張する。

**確定フェーズ**: Phase 1a（k6）、Phase 1c（Locust）。**対応要件**: NFR-B-PERF-\*、NFR-SUP-\*。**上流**: DS-NFR-PERF-\*。

## DS-IMPL-DIR-168 load/results の git-lfs 管理

負荷試験結果（JSON / CSV / HTML レポート）は `tests/load/results/` に日付ディレクトリで配置する（`results/2026-04-15_phase1a_baseline/`）。サイズが数 MB〜数百 MB に及ぶため、git-lfs で管理し通常 clone で取得されないようにする。

CI では k6 結果を Prometheus remote-write で送信することで永続化し、本ディレクトリは「手動実行時のアーカイブ」と「ベースライン（Phase 1a 初回測定値）の保管」用途に限定する。ベースラインは Phase 1c のリリース判定で「Phase 1a 比で p99 が劣化していないか」の比較元になる。

**確定フェーズ**: Phase 1a。**対応要件**: NFR-B-PERF-\*。**上流**: DS-NFR-PERF-\*、DS-IMPL-DIR-157（image tag placeholder ルール）。

## DS-IMPL-DIR-169 fixtures/ の責務と共有範囲

`tests/fixtures/` は e2e / contract / load が共有する入力データを置く。5 サブディレクトリで責務を分離する。

- `auth/`: テスト用 OIDC プロバイダの発行鍵、JWT、mTLS 証明書（自己署名）
- `crypto/`: テスト用 KMS / HSM エミュレータの設定と鍵
- `pii/`: PII マスキング対象のサンプル（日本人名・マイナンバー等、全てダミー）
- `policy/`: ZEN Engine ルール（JDM 形式）のサンプル
- `workflow/`: Dapr Workflow 定義 JSON

フィクスチャは **決定論的** であり、実行順・乱数に依存しない。生成スクリプトを `fixtures/<category>/generate.sh` として配置し、再現性を担保する。`pii/` のサンプルは架空の人名ジェネレータで生成し、実在人物の姓名を含まない（`NFR-G-PROT-*` 準拠）。

**確定フェーズ**: Phase 1a。**対応要件**: DX-TEST-\*、NFR-G-PROT-\*。**上流**: DS-NFR-PROT-\*。

## DS-IMPL-DIR-170 golden/ のスナップショット比較

`tests/golden/` はスナップショット比較用のゴールデンファイルを置く。3 サブディレクトリに分ける。

- `proto/`: gRPC レスポンスの期待 protojson
- `openapi/`: RESTful 生成物（Phase 2 で tier1 外向け REST ゲートウェイを追加する場合）
- `log/`: 構造化ログの期待形式（JSON Lines）

ゴールデン更新は `GOLDEN_UPDATE=1 go test ./tests/e2e/scenarios/...` のような環境変数 flag で行い、通常実行では比較のみを行う。ゴールデン変更を伴う PR は「意図した変更か」を CODEOWNERS レビューで確認する。

ゴールデンは JSON が基本で、フィールド順序の差異に強いスキーマ比較（`gotest.tools/v3/assert/cmp`）を使う。これによりフィールド追加は backward compatible（既存ゴールデンは変更不要）、フィールド削除 / 型変更は差分として検出される。

**確定フェーズ**: Phase 1a。**対応要件**: DX-TEST-003、FR-T1-\*。**上流**: DS-IMPL-DIR-021（contracts）。

## DS-IMPL-DIR-171 Go 実装側 unit テスト配置

`src/tier1/go/internal/pods/<pod>/<layer>/*_test.go` と `src/tier1/go/internal/shared/<topic>/*_test.go` に配置する。Go 慣習に従い実装ファイルと同じディレクトリ、同じ package（ただし `package <pkg>_test` の外部テストを優先）。CI で `go test -short ./src/tier1/go/...` を PR ごとに実行し、30 秒以内に完了することを目標とする。

Pod 固有層（`internal/pods/<pod>/`）のテストは 4 層すべてで配置する。

- `handler/*_test.go`: gRPC サーバ入出力の単体テスト（ハンドラ単位）
- `service/*_test.go`: ビジネスロジックの単体テスト（mock repository）
- `domain/*_test.go`: ドメインモデルの単体テスト
- `repository/*_test.go`: sqlc 生成コードと adapter の間の単体テスト

shared 層（`internal/shared/`）のテストは topic 単位で `dapr/*_test.go`、`policy/*_test.go`、`common/*_test.go` などを隣接配置する。shared のテスト失敗は全 Pod に波及するため、CI で `test-go-unit-shared` ジョブを分離し、shared だけ差分発生時に Pod ジョブ全てを強制再実行する戦略を取れる余地を残す。

repository 単体は本番バックエンド（Postgres 等）を起動しないため mock でよい。本番バックエンドとの integration は後述 DS-IMPL-DIR-173 の `integrationtest/` で行う。

**確定フェーズ**: Phase 1a。**対応要件**: DX-TEST-001。**上流**: DS-IMPL-DIR-043（Go 4 層）、DS-DEVX-TEST-\*。

## DS-IMPL-DIR-172 Rust 実装側 unit テスト配置

Rust は Go と異なり 2 通りの配置を使い分ける。04 章 DS-IMPL-DIR-082〜084 で確定した `crates/pods/` / `crates/shared/` / `crates/sdk/` の 3 分類のいずれでも同じ配置規則を適用する。

1. **inline unit test**: `src/tier1/rust/crates/{pods,shared,sdk}/<crate>/src/**/*.rs` の末尾に `#[cfg(test)] mod tests { ... }` で配置。private 関数をテスト対象にできる。
2. **crate 内 integration test**: `src/tier1/rust/crates/{pods,shared,sdk}/<crate>/tests/*.rs` に配置。public API のみをテスト対象とし、`extern crate` 相当の使い方になる。

Phase 1a 時点の方針は、domain / service 層の純粋関数は inline、adapter / grpc 層の integration は `crates/pods/<pod>/tests/` に配置。`crates/shared/` は共有ライブラリのため大半が inline、`crates/sdk/`（Phase 2）は public API の互換性検証が主目的のため tests/ 側に寄せる。CI は `cargo test --workspace --no-fail-fast` で全実行し、`cargo nextest` を導入して並列度を上げる（Phase 1b）。

**確定フェーズ**: Phase 1a。**対応要件**: DX-TEST-001。**上流**: DS-IMPL-DIR-082（Rust 4 層）、DS-DEVX-TEST-\*。

## DS-IMPL-DIR-173 Go integrationtest/ の扱い

`src/tier1/go/integrationtest/<pod>/` は Pod 内で実バックエンド（Postgres / Valkey / OpenBao）と結合するテストを置く。E2E と異なり、Dapr sidecar は起動せず、Go SDK レベルで直接バックエンドに接続する。testcontainers-go で Postgres / Valkey を起動し、sqlc 生成コードと adapter を往復させて検証する。

`_test.go` ファイルにビルドタグ `//go:build integration` を付け、通常の `go test ./...` では実行されない設定にする。CI では `go test -tags=integration -run ./src/tier1/go/integrationtest/...` として個別ジョブで実行する。

E2E（`tests/e2e/`）との境界: integrationtest は「1 Pod + バックエンド」、E2E は「複数 Pod + Dapr sidecar + Ingress + バックエンド」。配置の分離でジョブ粒度が明確になる。

**確定フェーズ**: Phase 1a。**対応要件**: DX-TEST-002。**上流**: DS-IMPL-DIR-058（Go integrationtest）、DS-DEVX-TEST-\*。

## DS-IMPL-DIR-174 Rust crate 内 tests/ の扱い

`src/tier1/rust/crates/pods/<pod>/tests/` は各 pod crate の public API に対する integration test を置く。testcontainers-rs で Postgres を起動し、AUDIT crate の repository 層と実 DB を結合する。shared / sdk crate の integration test も同じ流儀で `crates/shared/<crate>/tests/` / `crates/sdk/<crate>/tests/` に配置するが、shared は外部バックエンドを要さない純粋層のみ、sdk は公開 API 互換性検証のみを対象とする。

配置方針: `tests/<機能名>.rs` 単位でファイルを分け、ファイル内は複数 `#[tokio::test]` を並べる。fixtures は repo root `tests/fixtures/` を共有せず、crate 内 `tests/common/` に小さなヘルパを置く（cross-crate 共有が必要な場合のみ repo root fixtures を参照）。

**確定フェーズ**: Phase 1a。**対応要件**: DX-TEST-002。**上流**: DS-IMPL-DIR-087（Rust tests/）、DS-DEVX-TEST-\*。

## DS-IMPL-DIR-175 benches/ と examples/ の位置づけ

`src/tier1/rust/crates/pods/<pod>/benches/` は criterion によるマイクロベンチを置く。CI では nightly で走らせ、回帰が起きたら Slack に通知する。短時間（5 分以内）を目標とし、soak は `tests/load/locust/` で行う。shared crate のベンチが必要な場合は `crates/shared/<crate>/benches/` に配置する（Phase 1b 以降の判断）。

`src/tier1/rust/crates/pods/<pod>/examples/` は crate の使い方を示すサンプルコードで、`cargo run --example <name>` で実行できる。Phase 2 で tier2 向けに Rust SDK を公開する際、SDK 利用例の出発点は `crates/sdk/k1s0-sdk/examples/` に配置する。Phase 1a 時点では DECISION crate（`crates/pods/decision/examples/`）のポリシー評価サンプルのみを配置する。

**確定フェーズ**: Phase 1a（decision example）、Phase 1b（全 crate）。**対応要件**: DX-TEST-\*、DX-DEVEX-\*。**上流**: DS-IMPL-DIR-088（benches）、DS-IMPL-DIR-089（examples）。

## DS-IMPL-DIR-176 テストデータの秘匿情報禁止

`tests/fixtures/` `tests/e2e/testdata/` `tests/golden/` 配下のファイルは **本番環境の秘匿情報を含めない**。具体的には以下を禁止する。

- 本番 OIDC プロバイダの発行鍵
- 本番バックエンドの接続文字列
- 実在人物の姓名 / マイナンバー / 住所
- 本番 Kafka トピックのメッセージダンプ

テスト用の OIDC 鍵はテスト実行時に生成する自己署名、接続文字列はローカル起動の testcontainers のランダムポート、人名はダミージェネレータ出力、Kafka メッセージは手書きの架空データとする。これを Kyverno 相当のレビュアーガイドに含め、PR レビュー時の確認項目とする。

**確定フェーズ**: Phase 1a。**対応要件**: NFR-G-PROT-\*、NFR-E-AC-\*。**上流**: DS-NFR-PROT-\*、DS-IMPL-DIR-158（Secret placeholder）。

## DS-IMPL-DIR-177 CI ワークフローとテスト配置の対応

`.github/workflows/` の 4 系統（PR / main / release / nightly）と本章のテスト配置は以下で対応する。詳細は [07_開発ツールとCICD配置.md](07_開発ツールとCICD配置.md) に譲るが、物理配置側の前提として固定する。

- **PR**: `go test -short ./src/tier1/go/...` + `cargo test --workspace`（unit + inline integration）
- **main**: PR と同じ + `go test -tags=integration ./src/tier1/go/integrationtest/...` + `cargo test` の `crates/<pod>/tests/`
- **release**: main と同じ + `go test ./tests/e2e/scenarios/...`（E2E 全シナリオ）+ `tests/contract/providers/` の Pact 検証
- **nightly**: release と同じ + `tests/load/k6/`（k6 負荷試験）+ benches

この 4 段階でテスト粒度とコストを段階的に上げ、早期フェーズで検出できる問題は早期に検出する。

**確定フェーズ**: Phase 1a。**対応要件**: DX-TEST-\*、DX-CICD-\*。**上流**: DS-DEVX-CICD-\*、DS-IMPL-DIR-181（`.github/workflows/`、07 章で詳細化）。

## DS-IMPL-DIR-178 tests/ 配下の CODEOWNERS

`tests/` 配下の所有権は [01 章 DS-IMPL-DIR-018](01_リポジトリルート構成.md) で定義した 8 チーム（`tier1-architects` / `product-owners` / `api-leads` / `tier1-go-team` / `tier1-rust-team` / `infra-team` / `devex-team` / `security-team`）で分ける。API 責任チームという抽象ではなく、Pod 実装言語に紐づく具体チームに割り当てることで、テスト失敗時の一次トリアージ先を物理的に確定させる。詳細な CODEOWNERS エントリは [07 章 DS-IMPL-DIR-199](07_開発ツールとCICD配置.md) に譲り、本節ではテストカテゴリと責任チームの対応を散文で述べる。

- `tests/e2e/framework/`: `@k1s0/devex-team`（E2E 基盤保守）+ `@k1s0/tier1-architects`（シナリオ抽象設計）
- `tests/e2e/scenarios/state/` `tests/e2e/scenarios/secret/` `tests/e2e/scenarios/workflow/`: `@k1s0/tier1-go-team`（Go Pod 担当）
- `tests/e2e/scenarios/audit/` `tests/e2e/scenarios/decision/` `tests/e2e/scenarios/pii/`: `@k1s0/tier1-rust-team`（Rust Pod 担当）
- `tests/e2e/scenarios/cross_pod/`（Service Invoke 等の横断シナリオ）: `@k1s0/tier1-architects`
- `tests/contract/providers/go/`: `@k1s0/tier1-go-team` + `@k1s0/api-leads`（Protobuf 契約の整合性レビュー）
- `tests/contract/providers/rust/`: `@k1s0/tier1-rust-team` + `@k1s0/api-leads`
- `tests/contract/consumers/`: `@k1s0/api-leads`（tier2 消費者視点の契約、tier1 側主導）
- `tests/load/`: `@k1s0/infra-team`（性能計測基盤の保守、Kubernetes リソース割当と直結）
- `tests/fixtures/`: `@k1s0/devex-team` + `@k1s0/security-team`（フィクスチャ全般は DevEx、秘匿情報混入確認は Security のダブルレビュー）
- `tests/golden/`: `@k1s0/api-leads`（API 応答スナップショットのため、契約レビューと同じオーナー）

scenarios / providers / golden は実装言語チーム + `api-leads` の 2 チーム承認を要求し、API 変更時に同一 PR でテストも更新する運用を徹底する。framework / load / fixtures は横断チーム（`devex-team` / `infra-team` / `security-team`）が保守する。GitHub CODEOWNERS は OR 条件のみでチーム横断の AND は表現できないため、ダブルレビュー（fixtures の DevEx + Security、providers の言語チーム + api-leads）はブランチ保護ルールの「必須レビュワー数」で担保する。

**確定フェーズ**: Phase 1a。**対応要件**: DX-TEST-\*、NFR-C-NOP-001。**上流**: DS-IMPL-DIR-018（CODEOWNERS 母集団）、DS-IMPL-DIR-199（CODEOWNERS 詳細定義）。

## DS-IMPL-DIR-179 テスト実行時間の予算

各テストカテゴリの実行時間予算を以下で固定する。予算を超えた場合は分割または並列化を行う。

- unit（PR ごと）: 合計 2 分以内（Go 30 秒 + Rust 90 秒）
- integration（main ごと）: 合計 10 分以内
- E2E（release ごと）: 合計 30 分以内、1 シナリオ 5 分以内
- load（nightly）: 合計 60 分以内（k6 ramp-up/steady/ramp-down）

2 名運用（`NFR-C-NOP-001`）で「失敗したら即ロールバック → 翌日対応」のサイクルを成立させるには、release ブランチの CI が 1 時間以内に完了することが前提である。本予算はその前提から逆算した。

**確定フェーズ**: Phase 1a。**対応要件**: DX-TEST-\*、DX-CICD-\*、NFR-C-NOP-001。**上流**: DS-DEVX-CICD-\*。

## DS-IMPL-DIR-180 テスト配置変更の ADR 起票条件

以下の変更は ADR 起票を要する。

1. `tests/` 直下のサブディレクトリの新規追加（現状 5 種: e2e / contract / load / fixtures / golden）
2. テスト category の分離境界変更（例: integrationtest を tests/ 配下に移動）
3. テスト実行時間予算の緩和（DS-IMPL-DIR-179 の閾値引き上げ）
4. Pact ブローカーの本格導入（Phase 1c）や代替ツールへの切替
5. 負荷試験ツールの変更（k6 / Locust 以外への切替）

軽微な追加（シナリオ追加、fixtures の追加、golden の更新）は ADR 不要で、CODEOWNERS レビューのみで通す。

**確定フェーズ**: Phase 0（ルール）、各 Phase（適用）。**対応要件**: NFR-C-NOP-001、DX-CICD-\*。**上流**: DS-SW-COMP-138、DS-IMPL-DIR-018。
