# 08. 命名規約と配置ルール

本ファイルは k1s0 リポジトリ全体に横断適用するファイル命名規約と配置ルールを方式として固定化する。上流は本カテゴリの 01 〜 07 章（ルート構成・contracts・Go・Rust・infra・tests・tools）の全てで、本章はそれらに共通する「名前の付け方」「1 ファイルの上限」「生成ファイルの識別」「ディレクトリの追加規則」を確定させる。

## 本ファイルの位置付け

01 〜 07 章はそれぞれ「どこに何を置くか」を確定させたが、**命名の揺れ**は各章単体では解消されない。Go の流儀（`snake_case` ファイル名 + `camelCase` 識別子）、Rust の流儀（`snake_case` 一律）、Protobuf の流儀（`snake_case` ファイル + `PascalCase` メッセージ）、Kubernetes の流儀（`kebab-case` YAML + `PascalCase` kind）、TypeScript の流儀（`kebab-case` ファイル）と、言語・エコシステムごとに慣習が異なる。

これらを重ね合わせた時、「Go の `state_store.go` は OK だが `state-store.go` は NG」「Helm の `deployment.yaml` は OK だが `deployment.yml` は NG」といった細かい判断を毎回行うことになり、PR レビューのノイズが増える。本章は (a) 言語・エコシステム別の命名を網羅表で固定、(b) 1 ファイル 300 行制限の例外規則を明確化、(c) 生成ファイルの識別マーカーを統一、(d) 新規ディレクトリ追加時の ADR 起票条件を規定、の 4 点で揺れを排除する。

## 設計 ID 一覧と採番方針

本ファイルで採番する設計 ID は `DS-IMPL-DIR-201` 〜 `DS-IMPL-DIR-220` の 20 件である。

## 命名規約の全体像

命名規約は「ファイル名」「ディレクトリ名」「識別子（関数・型・定数）」「タグ / ラベル」「リソース名（Kubernetes / Dapr）」の 5 種類に分かれる。5 種類それぞれに言語・エコシステム別の規則を当てる。全体像を下表に集約する。この表は章末の設計 ID（DS-IMPL-DIR-201 以降）のインデックスであり、各 ID で個別に規則を展開する。

| 対象 | 言語・ツール | 規則 | 例 |
|---|---|---|---|
| ファイル名 | Go | `snake_case.go` | `state_store.go` |
| ファイル名 | Rust | `snake_case.rs` | `state_store.rs` |
| ファイル名 | Protobuf | `snake_case.proto` | `state_store.proto` |
| ファイル名 | TypeScript（k6） | `kebab-case.ts` | `state-p99.ts` |
| ファイル名 | Shell | `kebab-case.sh` | `run-e2e.sh` |
| ファイル名 | Kubernetes YAML | `kebab-case.yaml` | `deployment.yaml` |
| ファイル名 | Markdown（docs 内） | `NN_日本語名.md` | `01_リポジトリルート構成.md` |
| ファイル名 | Markdown（README） | `README.md` 固定 | `README.md` |
| ディレクトリ名 | リポジトリ全般 | `kebab-case` または `snake_case` | `src/tier1/rust/crates/pods/audit/` |
| ディレクトリ名 | docs | `NN_日本語名` | `05_実装/` |
| 識別子 | Go 関数 | `camelCase` / 公開は `PascalCase` | `NewStateStore` |
| 識別子 | Rust 関数 | `snake_case` | `new_state_store` |
| 識別子 | Protobuf message | `PascalCase` | `GetStateRequest` |
| Docker image tag | 全言語 | `kebab-case:semver` | `k1s0/state:1.0.0-phase1a` |
| Kubernetes リソース名 | YAML | `kebab-case` | `tier1-state-facade` |
| Dapr Component name | YAML | `kebab-case` | `tier1-statestore-postgres` |

この表は「判断に迷った時の一次リファレンス」として使い、例外は各 DS-IMPL-DIR に根拠付きで例示する。

## DS-IMPL-DIR-201 Go ファイル命名規約

Go のソースファイルは `snake_case.go` とする。Go の公式規約（`gofmt` / `go vet`）は明示的にファイル命名を制約しないが、`golang/go` 本体と標準ライブラリの慣例に従う。

- ファイル名は内容を説明する名詞（単数形 / 複数形）: `state_store.go` `state_stores.go`
- 1 パッケージ内で複数の役割を 1 ファイルにまとめない。`state.go` に `type StateStore` と `type StateCache` を両方入れるのは禁止
- テストファイルは対応するソースファイルに `_test.go` を付ける: `state_store.go` ↔ `state_store_test.go`
- 外部パッケージテスト（black box）は `_test.go` の package 宣言を `package foo_test` とする
- build tag 指定ファイルはファイル名 suffix で示す: `main_linux.go` `main_darwin.go`（`//go:build linux` と併用）

1 Go ファイルは 300 行以内（[CLAUDE.md](../../../CLAUDE.md)）。300 行超過時は関連度の高い function / type 単位で分割し、分割ファイルは元ファイルと同じディレクトリに同じ package で置く。

**確定フェーズ**: Phase 1a。**対応要件**: DX-CICD-\*、NFR-C-NOP-001。**上流**: Go 標準慣例、CLAUDE.md 300 行ルール。

## DS-IMPL-DIR-202 Rust ファイル命名規約

Rust のソースファイルは `snake_case.rs` とする。Rust は `rustfmt` でファイル命名を強制しないが、`cargo` がモジュール解決で `snake_case` を前提にしており、大文字を含めると path 解決がぶれる。

- モジュールファイル: `state_store.rs`（親モジュールから `mod state_store;` で参照される）
- `mod.rs` は廃止（Rust 2018 以降）。サブモジュールは `state_store/mod.rs` ではなく `state_store.rs` + `state_store/` ディレクトリの組み合わせ
- テストモジュール（crate 外 integration）: `tests/<name>.rs`（[DS-IMPL-DIR-174](06_テストとフィクスチャ配置.md)）
- benchmark: `benches/<name>.rs`
- example: `examples/<name>.rs`
- binary crate のエントリ: `src/main.rs` または `src/bin/<name>.rs`

1 Rust ファイルは 300 行以内（[CLAUDE.md](../../../CLAUDE.md)）。300 行超過時は `mod` でサブモジュール化し、ディレクトリ分割する。`use` 文や `#[derive(...)]` 行もカウントに含める。

**確定フェーズ**: Phase 1a。**対応要件**: DX-CICD-\*、NFR-C-NOP-001。**上流**: Rust 2018 edition、CLAUDE.md 300 行ルール。

## DS-IMPL-DIR-203 Protobuf ファイル命名規約

Protobuf は buf の lint 規則（`DEFAULT` 準拠）に従う。

- ファイル名: `snake_case.proto`（`state_store.proto`）
- ディレクトリ: `v1/<api>/<api>.proto` または `v1/<api>/v1.proto`（[DS-IMPL-DIR-021](02_contracts詳細構成.md)）
- `package` 宣言: `k1s0.v1.<api>`（`k1s0.v1.state`）
- message 名: `PascalCase`（`GetStateRequest`）
- field 名: `snake_case`（`state_key`）
- enum 名: `PascalCase`、enum 値: `SCREAMING_SNAKE_CASE` + enum 名プレフィックス（`ConsistencyMode.CONSISTENCY_MODE_STRONG`）
- service 名: `PascalCase`（`StateService`）
- rpc 名: `PascalCase`（`GetState`）

1 `.proto` ファイルは 300 行以内。300 行超過時は「API 論理単位」で分割し、`common.proto` `state.proto` `state_admin.proto` のように suffix を付ける。

**確定フェーズ**: Phase 1a。**対応要件**: FR-T1-\*、DX-CICD-\*。**上流**: buf lint DEFAULT、DS-IMPL-DIR-021。

## DS-IMPL-DIR-204 Kubernetes / Helm / Dapr YAML 命名規約

YAML ファイルは `kebab-case.yaml`（`.yml` 拡張子は禁止）とする。

- Helm templates: `templates/<resource-kind>.yaml`（`deployment.yaml` `service.yaml` `configmap.yaml`）
- Kustomize base: `base/<resource-kind>.yaml`
- Kustomize overlays: `overlays/<env>/<purpose>-patch.yaml`（`overlays/dev/resources-patch.yaml`）
- Dapr Component: `dapr-components/<type>/<backend>.yaml`（`dapr-components/state/postgres.yaml`）
- Kyverno ClusterPolicy: `policies/<rule>.yaml`（`policies/require-cosign-signature.yaml`）

Kubernetes リソース名（`metadata.name`）は `kebab-case` で `<tier>-<component>[-<purpose>]` の形式にする。例: `tier1-state-facade`、`tier1-audit-worker`、`tier1-statestore-postgres`。長さは 63 文字以内（Kubernetes ラベル上限）。

拡張子を `.yaml` に統一する理由は、`helm lint` / `yamllint` / OPA Rego が `.yaml` を前提にするケースが多く、`.yml` との混在で静的解析が漏れるため。

**確定フェーズ**: Phase 1a。**対応要件**: NFR-C-NOP-001、DX-CICD-\*。**上流**: DS-IMPL-DIR-121〜152、Kubernetes 命名規則。

## DS-IMPL-DIR-205 シェルスクリプト命名規約

シェルスクリプトは `scripts/` 配下に置き、`kebab-case.sh` とする。

- ユーザ向け: `bootstrap.sh` `proto-gen.sh` `run-e2e.sh`
- 内部ユーティリティ（他スクリプトから source される）: `scripts/lib/<name>.sh` に分離
- CI からのみ呼ぶ: `scripts/ci/<name>.sh` に集約（Phase 1b で必要に応じて追加）

実行権限必須（`chmod +x`）、shebang は `#!/usr/bin/env bash` 固定（[DS-IMPL-DIR-188](07_開発ツールとCICD配置.md)）。1 スクリプト 200 行以内を目標とし、超過時は Go / Rust ツール化して `tools/` に移す。

**確定フェーズ**: Phase 1a。**対応要件**: DX-CICD-\*、DX-LD-\*。**上流**: DS-IMPL-DIR-009、DS-IMPL-DIR-188。

## DS-IMPL-DIR-206 TypeScript（k6）ファイル命名規約

k6 スクリプトは TypeScript で書き、`kebab-case.ts` とする。

- `tests/load/k6/<api>-<metric>.ts`（`state-p99.ts` `decision-rps.ts` `workflow-concurrency.ts`）

## DS-IMPL-DIR-207 docs 配下のファイル命名規約

`docs/` 配下は日本語ドキュメントが主で、命名規則は他ディレクトリと異なる。

- カテゴリディレクトリ: `NN_日本語名/`（`05_実装/`、`00_ディレクトリ設計/`）。番号は 2 桁
- ドキュメントファイル: `NN_日本語名.md`（`01_リポジトリルート構成.md`）
- トップの README: `README.md`（英語名で固定）

## DS-IMPL-DIR-208 ディレクトリ命名規約

ディレクトリ名は以下の優先順位で決める。

1. 言語・ツールのエコシステムに確立した慣例がある場合はそれに従う（`src/` `tests/` `crates/` `cmd/` `internal/` は Go / Rust の慣例）
2. 慣例がない場合は `kebab-case` を基本とする（`dapr-components/` `external-secrets/`）
3. docs 配下のみ `NN_日本語名` を使う（DS-IMPL-DIR-207）

ディレクトリ名には空白・大文字・ASCII 以外（docs を除く）を入れない。深さは root から 8 階層以内を目標とし、超えそうな場合は中間ディレクトリを集約する。

**確定フェーズ**: Phase 1a。**対応要件**: NFR-C-NOP-001、DX-CICD-\*。**上流**: 全章共通。

## DS-IMPL-DIR-209 1 ファイル 300 行制限の適用範囲と adaptive guideline

[CLAUDE.md](../../../CLAUDE.md) で定めた「1 ファイル 300 行以内」は **ソースコード** に適用する。docs は例外で、400 行以内を目標とする（[README.md](README.md)）。

300 行制限の **対象** は以下。

- Go `.go`、Rust `.rs`、Protobuf `.proto`、Shell `.sh`、TypeScript `.ts`、Python `.py`
- Kubernetes YAML、Helm template、Kyverno ClusterPolicy
- Dockerfile（マルチステージビルドで長くなりがちだが、300 行で構成し直す）

300 行制限の **対象外** は以下。

- Markdown（`.md`）: 400 行目標、docs 配下は柔軟に扱う
- 生成ファイル（Protobuf → Go / Rust の生成物、sqlc 生成物、wire 生成物）: 対象外
- `go.sum` `Cargo.lock`: ロックファイルは対象外
- JSON Schema（`values.schema.json`）: 対象外（機械生成相当）
- テストデータ（fixtures、golden）: 対象外

行数は空行とコメントを含めて数える。ヘッダコメント（[CLAUDE.md](../../../CLAUDE.md) 必須）とライセンスコメントを引いた「実コード」だけで判定しない（単純な行数で機械判定するため）。

**adaptive guideline — 300 行を硬い閾値にしないための緩衝帯**: 300 行という固定閾値の運用上の問題は 2 つある。(1) **意味のある分割点がない場所で機械的に分割を強制**され、結果として可読性が下がる（例: Kyverno ClusterPolicy の 1 policy が 330 行で、`match` / `mutate` / `validate` の関連ブロックを切ると policy の意図が追いにくくなる）。(2) **定義の列挙が本質の構造**に対しても同じ閾値を適用し、誤った分割を誘発する（例: 11 API × 複数 enum を持つ `errors.proto`、Kubernetes の `apiGroups` / `resources` / `verbs` を列挙する RBAC Role）。これを防ぐため、以下の**段階的ゲート**で機械判定と人手判断を組み合わせる。

- **0〜300 行**: green。無条件で通過。
- **301〜400 行**: yellow（soft warning）。CI は警告のみで fail させず、**PR description に「分割しなかった理由」を 1 行で明記**する義務を負う（理由例: 「1 Kyverno policy の意図を分断しないため」「errors.proto の `enum` 列挙が本質的な構造で単位分割が不適」）。理由は `// kisocheck: file-size-exception=<理由>` のコメントをファイル末尾に書く形式とし、`tools/check-file-size/`（Phase 1b）が理由コメントの存在を検証する。
- **401〜500 行**: red（CI fail を default とする hard warning）。`// kisocheck: file-size-exception=<理由>` が存在し、かつその理由が事前承認された allowlist カテゴリ（`kyverno-policy-single-intent` / `proto-enum-enumeration` / `rbac-role-enumeration` / `terraform-module-single-resource` / `upstream-generated-header` の 5 種。以後追加は ADR 必須）に属する場合のみ許容。新規カテゴリ追加は ADR を要する。
- **501 行以上**: black。例外なく CI fail。必ず分割する。ADR 起票してもこの閾値は緩めない（運用を 2 人で回す NFR-C-NOP-001 に対する最終防衛線）。

yellow / red の例外記述は、「次回そのファイルを触る人が 1 分で理解できる密度」を求める。例外コメントが `"仕様上必要"` のような中身のない記述の場合は CI lint で fail させる（最低 20 文字、かつ allowlist カテゴリ ID を含むことを要求）。

**例外の時限性**: yellow / red の例外コメントには `valid-until: YYYY-MM-DD` を付け、失効日以降は再度レビューを要する（CI が失効日超過の例外を fail させる）。既定の失効期間は 6 か月。これにより「昔の負債が永久に残る」経路を塞ぐ。

**確定フェーズ**: Phase 0（基本ルール）、Phase 1b（check-file-size lint、adaptive guideline 導入）。**対応要件**: NFR-C-NOP-001、NFR-C-NOP-002、DX-CICD-\*。**上流**: CLAUDE.md。

## DS-IMPL-DIR-210 生成ファイルの識別マーカー

コード生成ツールが出力するファイルは **先頭 5 行以内** に以下のいずれかのマーカーを含める。これにより `check-deps` や lint ツールが「生成ファイルをスキップ」する判定が機械的に可能になる。

- Go: `// Code generated by <tool>. DO NOT EDIT.`（Go 公式慣例）
- Rust: `// Code generated by <tool>. DO NOT EDIT.` または `// @generated`
- Protobuf 生成物: 生成ツール（protoc / buf）が自動で付与
- sqlc 生成物: sqlc が自動で付与
- wire 生成物: wire が自動で付与

マーカー行は正規表現 `^\s*(//|#)\s*(Code generated|@generated)` でマッチ可能であること。手書きファイルに誤って付けるのは禁止（`check-deps` が誤検出する）。

生成ファイルの配置は元ファイルと同じディレクトリ + 明確な suffix（`_pb.go`、`_gen.rs`、`wire_gen.go`）で区別する。

**確定フェーズ**: Phase 1a。**対応要件**: DX-CICD-\*、FR-T1-\*（契約生成）。**上流**: Go 公式慣例、DS-IMPL-DIR-021、DS-IMPL-DIR-048（wire）。

## DS-IMPL-DIR-211 テストファイル命名規約

テストファイルは言語別に以下の規則に従う。

- **Go unit / small integration（実装隣接）**: `<source>_test.go`（`state_store.go` ↔ `state_store_test.go`）
- **Go integration（integrationtest/）**: `<api>_<観点>_test.go` + ビルドタグ `//go:build integration`
- **Go E2E（tests/e2e/scenarios/）**: `<api>_<観点>_test.go`
- **Rust unit（inline）**: `#[cfg(test)] mod tests { ... }` を source 末尾
- **Rust integration（crate 内 tests/）**: `<機能名>.rs`
- **Rust repo 横断（該当なし、crate 内で完結）**: —
- **Protobuf contract**: `tests/contract/providers/<api>_provider_test.go`
- **TypeScript（k6）**: `tests/load/k6/<api>-<metric>.ts`

テスト function 命名は言語慣例に従う。Go は `func TestXxx(t *testing.T)`、Rust は `#[test] fn test_xxx()` または `#[tokio::test] async fn test_xxx()`。description は英語で、目的が一目で分かる命名（`test_get_state_returns_not_found_when_key_missing`）とする。

**確定フェーズ**: Phase 1a。**対応要件**: DX-TEST-\*。**上流**: DS-IMPL-DIR-171〜174、Go / Rust 慣例。

## DS-IMPL-DIR-212 モジュール名の階層命名規則

Go / Rust のモジュール階層は「抽象度が高い → 低い」の順で命名する。

- Go: `github.com/k1s0/k1s0/src/tier1/go/internal/<pod>/<layer>/<feature>`
  - 例: `.../internal/state/service/getset`、`.../internal/state/repository/postgres`
- Rust: `k1s0_<crate>::<layer>::<feature>`
  - 例: `k1s0_audit::service::worm`、`k1s0_decision::adapter::zen`

各モジュール名は英小文字の 1 単語で、複数単語が必要な場合は `snake_case`（Rust）または `lowercase` 連結（Go、慣例）。深さは `pod / layer / feature` の 3 階層を基本とし、4 階層目は例外的に許可する。

**確定フェーズ**: Phase 1a。**対応要件**: NFR-C-NOP-001。**上流**: DS-IMPL-DIR-043（Go 4 層）、DS-IMPL-DIR-082（Rust 4 層）。

## DS-IMPL-DIR-213 Docker image 命名とタグ規約

Docker image 名は `<registry>/<project>/<component>:<version>` の形式。registry ホスト名は本章では placeholder として `harbor.k1s0.internal` を使い、JTC 稟議通過後の実 FQDN 確定時に一括置換する（[03 章 DS-IMPL-DIR-068](03_Goモジュール詳細構成.md)、[04 章 DS-IMPL-DIR-108](04_Rustワークスペース詳細構成.md) も同じ placeholder を使う）。置換対象キーワードは本ドキュメント内で `harbor.k1s0.internal` に統一し、`grep` 一括置換で済む構造にしている。

- registry: `harbor.k1s0.internal/`（Phase 1b 以降、placeholder FQDN）、`ghcr.io/k1s0/`（Phase 1a、PoC 期間の既定レジストリ）
- project: `tier1/`
- component: Pod 名（`state` `secret` `workflow` `audit` `decision` `pii`）
- version: `<semver>-phase<N>` または `main-<sha>`（開発用）

例: `harbor.k1s0.internal/tier1/state:1.0.0-phase1a`、`harbor.k1s0.internal/tier1/audit:main-a1b2c3d`。Phase 1a 期間は `ghcr.io/k1s0/tier1-state:1.0.0-phase1a` 形式を使う（project 区切りが `/` ではなく `-` になる点に注意）。

`latest` タグは Kyverno `deny-latest-tag.yaml`（[DS-IMPL-DIR-140](05_infra詳細構成.md)、`infra/governance/policies/` 配下に配置）で禁止される。タグは immutable（同一タグの上書き push をレジストリ側の immutability policy で禁止。Harbor の Project Immutability Rule、または ghcr.io の package retention policy で実装）。

SBOM と Cosign 署名は image と同じタグで同一レジストリに併存させる（`<image>:<tag>.sig`、`<image>:<tag>.sbom`）。

**placeholder の実 FQDN への差し替え手順**（Phase 1b 初頭に実施）: (1) ADR を `docs/02_構想設計/04_CICDと配信/ADR/` に起票、(2) 本ドキュメント + `infra/delivery/helm/*/values*.yaml` + `infra/governance/policies/` の Kyverno policy + CI workflow の `harbor.k1s0.internal` を一括 `sed` 置換、(3) 旧 FQDN を Kyverno policy に残して audit モードで監視、(4) 1 sprint 監視後に旧 FQDN 参照を削除。

**確定フェーズ**: Phase 1a（ghcr 使用 + placeholder 定義）、Phase 1b（Harbor 移行時に FQDN 置換）。**対応要件**: NFR-H-INT-\*、NFR-E-NW-\*、DX-CICD-\*。**上流**: DS-IMPL-DIR-157、DS-IMPL-DIR-191。

## DS-IMPL-DIR-214 Git ブランチとタグの命名規約

Git 側の命名も本章で統一する。

- **ブランチ**: `<type>/<issue-id>-<短い説明>`
  - type: `feat` / `fix` / `docs` / `refactor` / `test` / `chore`
  - 例: `feat/k1s0-42-state-cache`、`docs/k1s0-15-impl-dir`
- **リリースタグ**: `v<semver>-phase<N>`（`v1.0.0-phase1a`）
- **RC タグ**: `v<semver>-phase<N>-rc<M>`（`v1.0.0-phase1a-rc1`）
- **nightly タグ**: 自動付与せず、image tag `main-<sha>` のみで識別

ブランチ名に日本語は入れない（CI / Git のツールで扱い難いため）。`<短い説明>` は英語・`kebab-case`・30 文字以内。

**確定フェーズ**: Phase 1a。**対応要件**: DX-CICD-\*、NFR-C-NOP-001。**上流**: DS-IMPL-DIR-192（release.yml）。

## DS-IMPL-DIR-215 環境変数とシークレット名の規約

アプリケーションが読み取る環境変数名は `K1S0_<COMPONENT>_<PURPOSE>` 形式の `SCREAMING_SNAKE_CASE` とする。

- `K1S0_STATE_DAPR_APP_ID`、`K1S0_AUDIT_POSTGRES_DSN`、`K1S0_DECISION_ZEN_RULES_PATH`
- Dapr / Kubernetes 標準の環境変数（`DAPR_HTTP_PORT`、`KUBERNETES_SERVICE_HOST` 等）は例外

シークレット参照は Dapr Secret Store 経由 or External Secrets Operator 経由とし、環境変数で平文値を直接設定しない。Kubernetes Secret 名は `<component>-<purpose>-secret`（`tier1-audit-db-secret`）の `kebab-case`。

**確定フェーズ**: Phase 1a。**対応要件**: NFR-E-AC-\*、NFR-E-ENC-\*、NFR-H-KEY-\*。**上流**: DS-IMPL-DIR-133（Dapr Secret）、DS-IMPL-DIR-144（ESO）、DS-IMPL-DIR-158（Secret placeholder）。

## DS-IMPL-DIR-216 OpenTelemetry 属性名の規約

OTel span / log 属性名は **OpenTelemetry Semantic Conventions** に準拠する。Semantic Conventions で定義された属性（`service.name`、`http.status_code`、`db.system`）はそのまま使い、k1s0 固有の属性は `k1s0.<component>.<field>` プレフィックスを付ける。

- 公式: `service.name` `service.version` `deployment.environment`
- k1s0 固有: `k1s0.tenant.id`、`k1s0.request.trace_id`、`k1s0.policy.rule_id`

全て小文字・ドット区切り。JSON ログのフィールド名も同じ規則に揃える。

**確定フェーズ**: Phase 1a。**対応要件**: NFR-D-MON-\*、NFR-D-TRACE-\*。**上流**: OpenTelemetry Semantic Conventions、DS-NFR-OBS-\*。

## DS-IMPL-DIR-217 ログレベルとログメッセージの規約

構造化ログのフィールドは最小 5 種類を必ず含める。

- `timestamp`: RFC3339 UTC（`2026-04-12T09:15:30.123Z`）
- `level`: `debug` / `info` / `warn` / `error`（小文字、syslog 相当の 5 段階は使わない）
- `message`: 英語、句点なし、動詞で始める（`"started state service"` / `"failed to connect to postgres"`）
- `trace_id` / `span_id`: OTel から取得
- `service.name`: DS-IMPL-DIR-216 の公式属性

ログメッセージは英語で書き、日本語は `message_ja` フィールドに別途含めても良い（Phase 1c 以降の運用ダッシュボードで多言語検索に使う）。PII は絶対にログに出さない（[DS-IMPL-DIR-176](06_テストとフィクスチャ配置.md) の延長）。

**確定フェーズ**: Phase 1a。**対応要件**: NFR-D-MON-\*、NFR-G-PROT-\*。**上流**: DS-NFR-OBS-\*、DS-IMPL-DIR-176。

## DS-IMPL-DIR-218 新規ディレクトリ追加時の ADR 起票条件

`src/tier1/` 直下のサブディレクトリ、`tests/` 直下のサブディレクトリ、`tools/` 直下のサブディレクトリ、`scripts/` への新規追加は、以下のいずれかに該当する場合 ADR を要する。

1. 新規言語の追加（現状: Go / Rust / TypeScript / Python のみ）
2. 新規 tier の追加（現状: tier1 のみ、tier2 / tier3 は Phase 1b 以降）
3. 新規ツールカテゴリの追加（Phase 1a 時点の 5 種: proto-gen / check-deps / k1s0-cli / mock-server / backstage 以外）
4. `tests/` 直下の新規サブディレクトリ追加（現状: e2e / contract / load / fixtures / golden 以外）
5. `.github/workflows/` の系統追加（現状: pr / main / release / nightly の 4 系統以外）

軽微な追加（既存言語の crate / package 追加、既存カテゴリ内のファイル追加）は ADR 不要。CODEOWNERS レビューで通す。

ADR が必要と判断された場合は `docs/00_format/ADR-xxxx.md` テンプレートに従い、`docs/02_構想設計/adr/ADR-<CATEGORY>-<NNN>-<kebab-case-title>.md` として起票する。`CATEGORY` は既存 ADR の運用実績に従い `TIER1` / `DATA` / `SEC` / `CICD` / `RULE` / `OBS` / `FM` / `BS` / `STOR` / `MIG` のいずれか、または無印（横断 ADR: `ADR-NNNN-<title>.md` 形式、現在の `ADR-0001`〜`ADR-0003` と同じ運用）。`NNN` はカテゴリ内連番で、カテゴリ横断での重複は許容する（カテゴリごとに独立採番）。新規カテゴリの追加は ADR を要する。

**確定フェーズ**: Phase 0。**対応要件**: NFR-C-NOP-001、DX-CICD-\*。**上流**: DS-SW-COMP-138、ADR-0001〜ADR-MIG-002（既存 26 件）。

## DS-IMPL-DIR-219 本章の適用対象外

以下の領域は本章の命名規約を **適用しない**。

- `src/tier1/contracts/v1/` 配下の生成コード（Protobuf 生成物）: 生成ツールの出力命名に従う
- `.git/` 配下: Git 自体の管理下
- `node_modules/` `target/` `dist/` 等のビルド成果物: `.gitignore` 対象
- `docs/99_壁打ち/` 配下: ブレスト用途で命名自由
- `docs/90_knowledge/` 配下: 技術学習用ドキュメントで柔軟命名（ただし `NN_日本語名.md` を推奨）
- vendor 系 OSS の取り込み（現状なし、将来取り込む場合は `vendor/` 配下で上流の命名を保持）

これら以外は全て本章の規約に従う。違反は CI の `tools/check-deps` と `shellcheck` / `yamllint` で検出する。

**確定フェーズ**: Phase 0。**対応要件**: DX-CICD-\*。**上流**: 全章共通。

## DS-IMPL-DIR-220 命名規約変更の ADR 起票条件

本章の命名規約そのものを変更する場合は、以下の条件で ADR を要する。

1. 新規言語対応の追加（DS-IMPL-DIR-218 と連動）
2. ファイル名規則の変更（`snake_case` ↔ `kebab-case` の切替、拡張子変更）
3. 1 ファイル 300 行制限の閾値変更
4. 生成ファイル識別マーカーの変更
5. 環境変数プレフィックス（`K1S0_`）の変更
6. Kubernetes リソース命名規則の変更

規約変更は全リポジトリへの影響が大きいため、ADR と同時に「既存ファイルの遡及改名 PR」を伴う。遡及改名が困難な場合（互換性を壊す場合）は、段階的移行計画を ADR に含める。軽微な改訂（新規 Pod 追加時に命名表を拡張、誤記訂正）は ADR 不要。

**確定フェーズ**: Phase 0（ルール）、各 Phase（適用）。**対応要件**: NFR-C-NOP-001。**上流**: DS-SW-COMP-138、DS-IMPL-DIR-218。
