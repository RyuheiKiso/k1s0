# 06. 依存管理

本ファイルは tier2 の NuGet パッケージ管理と Go module 管理の方針を確定する。Central Package Management（.NET）と go.mod 戦略を明記する。

## NuGet Central Package Management

.NET 側は Central Package Management（CPM）を採用する。`src/tier2/dotnet/Directory.Packages.props` で全 NuGet パッケージのバージョンを集約管理し、各 `.csproj` では `<PackageVersion>` ではなく `<PackageReference>` のみ書く（バージョンは親から継承）。

### CPM の利点

- バージョンの二重管理を排除
- CVE 対応時に 1 ファイルの変更で全サービスに影響範囲を波及
- 依存の可視化（Directory.Packages.props を見れば全利用パッケージが一目）

### ポリシー

- `<PackageVersion Include="..." Version="..." />` のバージョンは **厳密指定**（`*` や `[1.0,)` は禁止）
- バージョン UP は Renovate / Dependabot による自動 PR で管理
- Major version UP は ADR 起票相当（セキュリティ的理由を除き）
- 社内パッケージ（`K1s0.Sdk`）は GitHub Packages / 社内プライベートから取得（Phase 1b 以降）

### 雛形サンプル

`src/tier2/dotnet/Directory.Packages.props` のサンプルは [02_dotnet_solution配置.md](02_dotnet_solution配置.md) 参照。

## Go module 管理

Go 側は tier ごとの go.mod 分離を採用する（[10_ルートレイアウト/05_依存方向ルール.md](../10_ルートレイアウト/05_依存方向ルール.md) の Go module セクション参照）。

- `src/tier2/go/go.mod` は tier2 全 Go サービス共通の module
- tier2 内の全サービス（`services/stock-reconciler/` / `services/notification-hub/`）は同じ go.mod を共有
- tier1 Go の go.mod（`src/tier1/go/go.mod`）とは独立

### ポリシー

- `go mod tidy` は CI で自動実行し、手動編集で指定した `require` / `replace` との差分があれば PR でブロック
- バージョン UP は Renovate / Dependabot
- 社内パッケージ（`github.com/k1s0/k1s0/src/sdk/go`）は `replace` 指示を使わず、GitHub Packages 経由で取得（Phase 1b 以降）
- `vendor/` は原則使わない（Go module で十分、vendor が必要な場面は閉域 CI のみ）

## 依存のバージョン pin 戦略

### Lock ファイルの commit

- .NET: `packages.lock.json`（`<RestorePackagesWithLockFile>true</RestorePackagesWithLockFile>` で有効化）を commit
- Go: `go.sum` を commit
- Node.js（tier3 側だが関連）: `pnpm-lock.yaml` を commit

これらは CI で `--frozen-lockfile` / `--locked-mode` 相当を強制し、ローカルと CI で同一の依存を再現する。

### 自動更新

Renovate または Dependabot を設定する。

- Minor / Patch: 自動 PR 作成、テスト通過でマージ許可
- Major: 自動 PR 作成、手動レビューで承認必須
- Security alert: 自動 PR 作成、優先度 High

設定ファイルは `.github/renovate.json` または `.github/dependabot.yml` に配置。詳細は [../70_共通資産/05_devcontainer配置.md](../70_共通資産/05_devcontainer配置.md) の関連セクションで扱う。

## 脆弱性スキャン

依存の脆弱性を定期的にスキャンする。

- .NET: `dotnet list package --vulnerable --include-transitive`
- Go: `govulncheck ./...`
- Rust: `cargo audit`
- Node.js: `pnpm audit`

CI で週次実行し、High 以上の脆弱性が発生した場合は GitHub Issue を自動起票する。

## ライセンスチェック

`third_party/` への vendoring 判定と同様に、tier2 の依存パッケージのライセンスも定期チェックする。

- Apache 2.0 / MIT / BSD 系: 問題なし
- LGPL: 要検討（動的リンクのみ可）
- GPL / AGPL: 禁止（tier2 は商用閉域運用前提）

ツールは `dotnet-project-licenses` / `go-licenses` / `cargo-deny` / `license-checker-rseidelsohn` を候補とする。

## 社内パッケージの命名

k1s0 内部の公開パッケージは以下の命名。

- .NET: `K1s0.Sdk` / `K1s0.Sdk.Auth` / `K1s0.Testing`
- Go: `github.com/k1s0/k1s0/src/sdk/go` / `github.com/k1s0/k1s0/src/testing/go`
- TypeScript: `@k1s0/sdk` / `@k1s0/sdk-auth`
- Rust: `k1s0-sdk` / `k1s0-sdk-auth`

これらは社内パッケージであり、外部公開は Phase 1c の判定後に決定。
