# 04. サービス単位の内部構造

本ファイルは tier2 の各サービス（.NET / Go どちらも）が従う内部構造（Clean Architecture / Onion Architecture）を確定する。サービス間の実装統一により、新規参加者の学習コストを下げ、横断リファクタリングを容易にする。

## 採用するアーキテクチャ

Clean Architecture の 4 層構成（Api / Application / Domain / Infrastructure）を採用する。依存方向は以下。

```
[Api] ──→ [Application] ──→ [Domain]
                                ↑
                         [Infrastructure]
```

- Api は Application を呼び出す
- Application は Domain を使う
- Infrastructure は Domain の interface を実装する（Domain は Infrastructure を知らない = 依存性逆転）
- Api と Infrastructure の直接的な依存関係はなし

## なぜこのアーキテクチャか

tier2 は JTC 業務ロジックを実装する層であり、以下の特性が求められる。

- **テスタビリティ**: 業務ロジックを DB や外部 API から切り離して unit test できる状態
- **置換可能性**: 外部依存（DB・外部 API）を別実装に差し替えやすい（閉域運用への切替など）
- **知見の継承**: JTC 業界で Clean Architecture は広く知られており、新規参加者のオンボーディングが容易
- **.NET / Go で共通**: 両言語で同じパターンを使えることで、チーム横断の知識移転がしやすい

## 4 層の責務と配置

### Api 層

HTTP / gRPC エンドポイントを提供する。入力のバリデーションと Application 層への呼び出しに専念し、業務ロジックは含まない。

- .NET: `src/<service>/Api/Controllers/` 配下の `*Controller.cs`
- Go: `internal/api/` 配下の `handler_*.go` / `router.go`

### Application 層

ユースケース（業務フロー）を実装する。Domain の Entity / Value Object / Service を組み合わせて業務ロジックを表現する。Infrastructure の interface（Repository / Adapter）を注入され、それを通じて永続化や外部呼び出しを行う。

- .NET: `src/<service>/Application/UseCases/` + `src/<service>/Application/Services/`
- Go: `internal/application/usecases/` + `internal/application/services/`

### Domain 層

ビジネスドメインを表すモデル。Entity / Value Object / Domain Service / Domain Event / Repository interface（実装は Infrastructure）を持つ。外部依存を一切持たない。

- .NET: `src/<service>/Domain/Entities/` + `src/<service>/Domain/ValueObjects/` + `src/<service>/Domain/Events/` + `src/<service>/Domain/Interfaces/`
- Go: `internal/domain/` 配下にパッケージ分割（entity / value / event / repository）

### Infrastructure 層

Domain の Repository interface を実装する。DB アクセス（EF Core / gorm）・外部 API 呼び出し・Dapr / k1s0 SDK クライアント・ファイルストレージ など。

- .NET: `src/<service>/Infrastructure/Persistence/` + `src/<service>/Infrastructure/ExternalServices/`
- Go: `internal/infrastructure/persistence/` + `internal/infrastructure/external/`

## 4 層間の依存方向の強制

### .NET

各層の `.csproj` 内で `<ProjectReference>` の許容先を以下に制限する。

- `ApprovalFlow.Api.csproj` → Application のみ
- `ApprovalFlow.Application.csproj` → Domain のみ
- `ApprovalFlow.Domain.csproj` → 依存なし
- `ApprovalFlow.Infrastructure.csproj` → Domain のみ

CI で `dotnet list reference` の結果を検証し、許容外の参照が追加された場合はブロック。

### Go

各 internal package の import 先を自作 linter で検証。

- `internal/api/` → `internal/application/` のみ
- `internal/application/` → `internal/domain/` のみ
- `internal/domain/` → 依存なし
- `internal/infrastructure/` → `internal/domain/` のみ

## DI（依存性注入）

`main.go` / `Program.cs` で Infrastructure の実装を Domain の interface に注入する。

- .NET: `Microsoft.Extensions.DependencyInjection` の `services.AddScoped<IApprovalRepository, ApprovalRepository>();`
- Go: 手動 DI または `google/wire`

## テスト戦略

各層ごとに適切なテスト戦略を採用。

- Domain テスト: pure unit test（外部依存なし）
- Application テスト: Domain を直接使い、Infrastructure interface をモック
- Api テスト: in-process TestHost（.NET）または httptest（Go）
- Integration テスト: Testcontainers で実 DB + 実 Dapr を起動
