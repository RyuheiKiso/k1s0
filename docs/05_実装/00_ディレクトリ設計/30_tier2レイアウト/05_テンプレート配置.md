# 05. テンプレート配置

本ファイルは `src/tier2/templates/` 配下の雛形（service template）の配置を確定する。雛形は `src/platform/cli/` の k1s0 CLI（`k1s0 scaffold`）が参照し、新規サービスを自動生成する際の元になる。Netflix Paved Roads / Spotify Backstage の事例に近い思想。

## 雛形の目的

tier2 で新規サービスを作る際、開発者が毎回ゼロから `csproj` を書いたり、go.mod を構築したりするのは非効率であり、かつ品質のばらつきを生む。雛形は以下を事前に用意する。

- 標準ディレクトリ構造（Onion Architecture 準拠）
- 標準 Dockerfile（multi-stage + distroless）
- 標準 CI 設定（GitHub Actions ワークフロー）
- 標準 OpenTelemetry 組込
- 標準 tier1 SDK 組込
- 標準 unit test / integration test 雛形

これにより、新規サービス作成から初回 deploy までを 1 日以内で完了できる状態を目指す（DX-GP-\* のゴールデンパス要件）。

## .scaffold.yaml の書式

雛形メタ情報を記述する。`k1s0 CLI` が本ファイルを読み取り、placeholder 置換と target パス計算を行う。

```yaml
# src/tier2/templates/dotnet-service/.scaffold.yaml
apiVersion: scaffold.k1s0.io/v1
kind: ServiceTemplate
metadata:
  name: tier2-dotnet-service
  description: 採用側組織の業務向け .NET ドメインサービスの雛形
  language: csharp
  tier: tier2
  version: 0.1.0
spec:
  placeholders:
    - name: ServiceName
      description: サービス名（PascalCase、例: ApprovalFlow）
      required: true
      pattern: '^[A-Z][A-Za-z0-9]+$'
    - name: service-name
      description: ケバブケース版（例: approval-flow）
      required: true
      pattern: '^[a-z][a-z0-9-]+$'
    - name: ResourceName
      description: Resource 名（PascalCase、例: Approval）
      required: true
    - name: Namespace
      description: .NET namespace prefix（例: K1s0.Business.Approval）
      required: true
  target:
    root: src/tier2/dotnet/services/{{ServiceName}}/
  postSteps:
    - run: dotnet restore
    - run: dotnet build
    - run: dotnet format
```

Go service の `.scaffold.yaml` は language: go、placeholders に service-name のみ（PascalCase 不要）など Go 固有の項目を持つ。

## 雛形の placeholder 置換ルール

k1s0 CLI が以下のルールで置換する。

- `{{ServiceName}}` : PascalCase（.NET namespace / class）
- `{{service-name}}` : ケバブケース（Kubernetes resource name / URL path）
- `{{ServiceNameSnake}}` : snake_case（Go variable / package）
- `{{ResourceName}}` : PascalCase（.NET Controller 対象）
- `{{Namespace}}` : .NET namespace prefix

置換は全ファイル名・全ファイル内容を対象に行う。ただし `.scaffold.yaml` 自体は置換対象外。

## 雛形更新の方針

雛形は tier2 の Golden Path であり、変更は慎重に行う。

- **新規 placeholder 追加**: マイナーバージョン UP（0.1.0 → 0.2.0）
- **既存 placeholder の意味変更**: メジャーバージョン UP（0.x → 1.0）
- **依存ライブラリのバージョン UP**: パッチバージョン UP（0.1.0 → 0.1.1）

既存サービスの追従は雛形 UP 時に自動化されず、明示的な migration PR として行う。雛形自体の変更が既存サービスに「自動で反映される」状態は技術的に避ける（既存サービスの安定を重視）。

## 雛形テストの自動化

CI で以下を実施し、雛形が壊れていないことを検証する。

- `k1s0 scaffold service --template dotnet-service --name TestSvc --service-name test-svc --resource-name Test --namespace K1s0.Test` を実行
- 生成物が `src/tier2/dotnet/services/TestSvc/` に展開される
- `dotnet build` / `dotnet test` が通る
- 生成物を破棄（CI の ephemeral 環境内）

同様に go-service でも実施。
