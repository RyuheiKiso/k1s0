# 01. リリース原則

本ファイルは k1s0 のリリース戦略を判断する際に常に参照する 7 軸の原則を定義する。Argo CD の App / ApplicationSet 追加、Argo Rollouts の戦略変更、AnalysisTemplate の閾値調整、flagd のフラグ定義追加、rollback runbook の改訂など、リリース領域で発生する全ての設計判断は本原則との整合をレビューで確認する。

![リリース 7 原則マップ](img/リリース7原則マップ.svg)

## 原則が必要な理由

10 年保守を前提とする k1s0 では、リリースは「稀に行う大玉」ではなく「日常的に繰り返される小玉」でなければならない。「日常化」を無統制に進めると、次のような破綻が観測される。

- 緊急時に `kubectl apply -f` でクラスタへ直接反映する「裏道」が常態化し、Git と実クラスタの乖離が恒常化（GitOps の信頼残高を失う）
- Progressive Delivery を任意運用にすると、リリース担当者の裁量で「このくらいなら一気通貫で問題ない」判断が累積し、事故時の影響範囲が全ユーザーへ拡大する
- AnalysisTemplate に SLI 連動を組まないまま canary を回し、SLI が劣化しても人手でストップするまで劣化が継続する
- flagd のフラグ定義が未署名のまま配布され、悪意あるフラグ差分が admission で止まらず本番に到達する
- rollback runbook が「15 分以内」の実行時間を保証せず、事故時に 1 時間以上復旧できない
- canary を「5% → 100%」の 2 段階に短縮して運用し、中間段階の異常検知窓が消失する
- Release Notes が Slack / GitHub Releases / 個別 Wiki に分散し、image hash から変更内容を追跡する経路が欠落する

本原則は、これらの破綻を「配信経路」「進行方式」「安全弁」の各層で構造的に防止する 7 軸である。GitOps / Progressive Delivery / フィーチャーフラグ / 署名検証 / エラーバジェット連動を一つのパイプラインとして束ね、日常的に回せる最小単位のリリースを 10 年支える構造を作る。

## 原則 1: GitOps 原則を絶対化する（IMP-REL-POL-001）

**クラスタへの反映経路は Argo CD のみとし、`kubectl apply` / `helm upgrade` / `kustomize build | kubectl apply` による直接反映を一切禁止する。**

ADR-CICD-001 で Argo CD を選定した目的は、Git を単一の真実（single source of truth）とすることで「実クラスタがどうなっているか」の認識を Git の状態に一致させることにある。裏道からの反映を許すと Git と実クラスタが乖離し、ふりかえり・監査・DR の前提が崩れる。

- 本番クラスタの RBAC は「Argo CD の service account 以外が書き込めない」構成とする（Kyverno で admission 制御）
- 緊急 rollback も Git の revert コミット経由で行う（例外的な `kubectl rollout undo` も SRE リード承認＋事後 1 時間以内の Git 反映を義務付け）
- Argo CD の ApplicationSet で多環境（dev / staging / prod）を生成し、環境差分は Kustomize overlays で管理
- Git に存在しない差分（drift）は Argo CD が検出次第 self-heal するか PagerDuty 通知
- 例外的な `kubectl` 操作が必要な場合は break-glass の RBAC 昇格を使い、操作ログを監査基盤に残す

## 原則 2: Progressive Delivery を必須化する（IMP-REL-POL-002）

**本番リリースは Argo Rollouts による Progressive Delivery（PD）を必須とし、例外は内部ツール・バッチ・emergency rollback に限定する。例外は SRE リード承認を要する。**

一気通貫リリースは「事故が起きた時点で全ユーザーに波及している」状態であり、JTC の 10 年保守では許容しない。本原則は PD を Phase 0 の MUST として固定し、例外範囲を明文化する（ADR-REL-001 として本章初版策定時に起票予定）。

- PD 必須: tier1 公開 11 API / tier2 ドメインサービス / tier3 本番アプリ
- PD 対象外: 内部管理ツール / バッチジョブ / 緊急 rollback（revert commit の高速配信）
- 対象外分類は catalog-info.yaml に明記し、Backstage が可視化する
- 対象外のまま本番に出るサービスは CI で検出（catalog-info.yaml の `release.strategy: progressive` フィールド必須化）
- PD 対象外の承認履歴は catalog-info.yaml の `release.exception-approver` に明記し、四半期ごとに Security が再審査

## 原則 3: AnalysisTemplate で SLO / SLI 連動の自動判定を行う（IMP-REL-POL-003）

**canary の進行は AnalysisTemplate による SLO / SLI 連動の自動判定で決定し、手動での「次ステップへ進める」操作を禁止する。**

Argo Rollouts の AnalysisTemplate は `60_観測性設計/` で定義した SLI を Prometheus クエリとして評価し、閾値超過で自動 rollback する仕組みである。「人が画面を見て OK を押す」運用は、夜間リリースや多数並列リリースで破綻する。

- AnalysisTemplate は各サービスの catalog-info.yaml にひも付き、Scaffold が雛形を生成
- 判定対象の SLI は「可用性（成功率）」「レイテンシ（p99）」の 2 本を最小セットとし、サービス固有 SLI を追加可能
- 高 throughput サービスではエラー率 0.1% を閾値、低 throughput サービスでは絶対数（例: 5 req/5min）を閾値とする
- 判定ウィンドウは step ごとに 5〜15 分（流量に応じて調整）
- SLI 劣化検知時は自動 abort と rollback（Git revert commit 自動生成）
- AnalysisTemplate の閾値変更は SRE 承認必須（`60_観測性設計/` の SLO 改訂と同じ厳格さで扱う）

## 原則 4: flagd 定義は cosign 署名を必須とする（IMP-REL-POL-004）

**flagd に配布するフラグ定義ファイル（JSON）は cosign 署名を必須とし、未署名フラグは flagd sidecar で拒否する。**

フィーチャーフラグは「コード変更を伴わない本番挙動変更」の手段であり、署名検証なしで配布すると、悪意ある差分が admission / CI を迂回して本番挙動を書き換えられる。本原則は `80_サプライチェーン設計/` の cosign 運用とフラグ配布経路を一体化する。

- フラグ定義は `infra/feature-management/flags/` の Git 管理下に置く
- 配布経路は Argo CD 経由で flagd ConfigMap を更新し、sidecar が cosign 検証を行う
- 検証失敗時はフラグ評価を既定値で返し、違反を Incident として記録
- 署名鍵は sigstore keyless（GitHub OIDC）を採用し、長期鍵持ち出しを禁止
- フラグ定義の reviewer には DX リード / SRE リードを必須追加し、本番挙動を書き換えるリスクを常に二重チェック

## 原則 5: rollback は 15 分以内に完了する runbook を持つ（IMP-REL-POL-005）

**全本番サービスは 15 分以内に rollback 完了する runbook を持ち、SRE が四半期に 1 回は演習する。**

「rollback は自動でできる」と「15 分以内に完了する」は別の問題である。本原則は rollback の実行時間を SLI 同等の指標として扱い、超過は Incident として扱う。演習なしの自動化は本番で必ず詰まる。

- rollback runbook は `ops/runbooks/incidents/rollback-<service>/` に配置
- 演習は四半期ごとに 1 サービスを抽選し、本番相当環境（staging）で実施
- 演習結果（実時間・詰まりポイント・改善 PR）は `ops/runbooks/quarterly/rollback-drill/` に記録
- emergency 時の「Git revert + Argo CD sync」の所要時間を計測し、15 分を超過する構成は改善対象
- 15 分の内訳目安: Git revert 2 分 / Argo CD sync 3 分 / AnalysisTemplate 判定 5 分 / 安定化観測 5 分

## 原則 6: canary は最低 3 段階を必須とする（IMP-REL-POL-006）

**Argo Rollouts の canary は最低 3 段階（5% → 25% → 100%）で構成し、一気通貫（0% → 100%）や 2 段階（5% → 100%）を禁止する。**

段階が少ないと「中間段階の異常検知」の窓が消失し、PD の効果が薄れる。3 段階は「初期エラー検知」「スケール時のキャパシティ問題検知」「全トラフィックでの最終検知」の 3 種類の観測窓を提供する。

- 推奨ステップ: 5% → 25% → 50% → 100%（各段階 10〜15 分）
- 最小構成: 5% → 25% → 100%
- 例外（2 段階以下）: SRE リード＋事業責任者の両者承認＋理由の ADR 化
- ホットフィックス時も 3 段階を維持（所要時間は 30 分以内に収まる構成とする）

## 原則 7: Release Notes は Backstage カタログに自動紐付ける（IMP-REL-POL-007）

**各リリースの Release Notes は Backstage カタログのサービスエンティティに自動紐付け、catalog-info.yaml から一覧可能にする。**

Release Notes が GitHub Releases / Slack / 個別 Wiki に分散すると、「この image で何が変わったか」の追跡が不可能になり、Forensics Runbook（`80_サプライチェーン設計/`）が image hash から変更内容を逆引きできない。本原則は Release Notes を Backstage 上で一元化する。

- Release Notes は conventional commits から自動生成（release-please / git-cliff）
- Backstage の TechDocs から各サービスの Release Notes を参照可能にする
- Release Notes の identity（tag）は image hash と 1:1 対応させ、cosign 署名のサブジェクトに含める
- 非機能変更（dependency 更新 / SBOM 差分のみ）も Release Notes に含める
- Release Notes のテンプレートは Scaffold CLI が生成し、カテゴリ（feat / fix / chore / sec）を必ず含む

## 対応 IMP-REL ID

本ファイルで採番する原則レベル ID は以下とする。

- `IMP-REL-POL-001` : GitOps 原則（Argo CD 唯一経路）
- `IMP-REL-POL-002` : Progressive Delivery 必須化
- `IMP-REL-POL-003` : AnalysisTemplate による SLI 連動自動判定
- `IMP-REL-POL-004` : flagd フラグ定義の cosign 署名必須
- `IMP-REL-POL-005` : 15 分以内 rollback runbook と演習
- `IMP-REL-POL-006` : canary 最低 3 段階必須
- `IMP-REL-POL-007` : Release Notes の Backstage 自動紐付け
