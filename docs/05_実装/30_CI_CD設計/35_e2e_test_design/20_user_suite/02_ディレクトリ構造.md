# 02. user suite ディレクトリ構造

本ファイルは ADR-TEST-008 で確定した `tests/e2e/user/` 配下のディレクトリ構造を実装段階の正典として固定する。2 部位（smoke / examples）の責務分担、独立 Go module、test-fixtures 経由での fixture 利用、Golden Path examples との連携を ID として採番する。

## 本ファイルの位置付け

ADR-TEST-008 で user suite を `tests/e2e/user/` に物理分離し、go.mod を owner suite と分けることを確定した。本ファイルでは 2 部位の責務範囲・依存関係・SDK test-fixtures（ADR-TEST-010）との連携経路を実装段階の運用契約として固定する。user suite はシンプルさを優先し、smoke（k1s0 install の正常性）と examples（Golden Path examples の動作確認）の 2 部位に絞る。

## 2 部位の責務分担

owner suite は 8 部位に分かれていたが、user suite は 2 部位に絞る。利用者の検証範囲は「k1s0 が install できる」+「自分の type の app が k1s0 で動く」の 2 点に集約され、細分化すると保守工数が爆発するためである。

| 部位 | 検証対象 | 主要依存 | 想定 test 件数（運用拡大時） |
|---|---|---|---|
| `smoke/` | k1s0 install が正常か（Dapr install / tier1 facade Pod Ready / Keycloak login / SDK State.Get round-trip）| `src/sdk/<lang>/test-fixtures/` 経由（ADR-TEST-010） | 5〜10 |
| `examples/` | `examples/` 配下の Golden Path 各 type の動作確認（tier2-go-service / tier2-dotnet-service / tier3-web-portal / tier3-bff-graphql / tier3-native-maui / tier1-go-facade / tier1-rust-service） | `src/sdk/<lang>/test-fixtures/` + `examples/` 配下の binary | 7（example 数）〜 |

`smoke/` は PR 5 分予算（ADR-TEST-001）に収まる軽量検証で、最頻パス（State.Get / Audit.Record の round-trip）のみ。`examples/` は nightly 30〜45 分予算で各 example を 1 件ずつ起動 → SDK 経由の round-trip 確認 → cleanup の 1 サイクルを回す。

## go.mod 構成

`tests/e2e/user/go.mod` は単一 module として管理する。owner suite と異なり、4 言語 SDK の cross-product helper は不要で、test-fixtures（`src/sdk/go/k1s0/test-fixtures`）を direct import するシンプルな構造。

```text
tests/e2e/user/
├── go.mod                              # module github.com/k1s0/k1s0/tests/e2e/user
├── go.sum
├── helpers/                            # 部位を跨ぐ共通 helper（最小限）
│   ├── kind_lifecycle.go               # kind cluster 起動 / cleanup（test-fixtures に委譲する場合の最小 wrapper）
│   └── example_runner.go               # examples/ 配下の binary 起動 + cleanup
├── smoke/
│   ├── install_test.go                 # k1s0 install が正常か
│   ├── tier1_state_test.go             # State.Get round-trip
│   ├── tier1_audit_test.go             # Audit.Record round-trip
│   └── keycloak_login_test.go          # Keycloak OIDC login が動くか
└── examples/
    ├── tier2_go_service_test.go        # examples/tier2-go-service/ の動作確認
    ├── tier2_dotnet_service_test.go
    ├── tier3_web_portal_test.go        # tier3 web frontend → BFF → tier1 round-trip
    ├── tier3_bff_graphql_test.go
    ├── tier3_native_maui_test.go       # MAUI app の起動 + SDK 呼び出し（sim 環境）
    ├── tier1_go_facade_test.go
    └── tier1_rust_service_test.go
```

`go.mod` に含める主要依存:

- `github.com/k1s0/k1s0/src/sdk/go/k1s0/test-fixtures` — ADR-TEST-010 で確定した SDK 同梱 fixtures
- `github.com/stretchr/testify` — assertion library
- `sigs.k8s.io/kind` — kind 起動の Go API（test-fixtures が wrap）

owner suite で必要だった重量依存（chromedp / k6 / multipass orchestrator）は user suite には含めない。これは ADR-TEST-008 の 2 module 分離の主目的の 1 つで、利用者が clone する射程を絞る効果がある。

## test-fixtures との連携経路

user suite の test code は `src/sdk/go/k1s0/test-fixtures` を import し、fixture の Setup / Teardown で kind cluster の起動と minimum stack の install を 1 行で行う（ADR-TEST-010 §3 の領域 1）。

```go
// tests/e2e/user/smoke/install_test.go の典型形
package smoke

import (
    "testing"
    "github.com/k1s0/k1s0/src/sdk/go/k1s0/test-fixtures"
    "github.com/stretchr/testify/require"
)

func TestK1s0InstallNormal(t *testing.T) {
    fx := testfixtures.Setup(t, testfixtures.Options{
        KindNodes: 2,
        Stack:     testfixtures.MinimumStack,
    })
    defer fx.Teardown()

    // tier1 facade Pod Ready 確認
    require.NoError(t, fx.WaitForTier1FacadeReady())
    // SDK client init
    client := fx.NewSDKClient(t, "tenant-a")
    // State.Get round-trip
    err := client.SetState(t, "key", "value")
    require.NoError(t, err)
    val, err := client.GetState(t, "key")
    require.NoError(t, err)
    require.Equal(t, "value", val)
}
```

利用者は同パターンを自分の app repo で書ける（`@k1s0/sdk-test-fixtures` を import するだけ）。これが ADR-TEST-010 の利用者 DX の核心。

## examples/ との連携

`tests/e2e/user/examples/` の各 test は、`examples/<type>/` 配下の binary（あるいは container image）を起動 → SDK 経由で叩く → 期待 response を assert する。binary 起動は `helpers/example_runner.go` で hide し、test code 自体は SDK 呼び出しに集中する。

```go
// tests/e2e/user/examples/tier2_go_service_test.go の典型形
func TestTier2GoServiceRoundtrip(t *testing.T) {
    fx := testfixtures.Setup(t, testfixtures.Options{
        KindNodes: 2,
        Stack:     testfixtures.MinimumStack,
        AddOns:    []string{"workflow"},  // tier2-go-service が Workflow を使う場合
    })
    defer fx.Teardown()

    runner := example_runner.NewExampleRunner(t, fx, "tier2-go-service")
    runner.Start(t)
    defer runner.Stop(t)

    // tier2-go-service が tier1 を呼べることを確認
    client := fx.NewSDKClient(t, "tenant-a")
    resp, err := client.CallTier2GoService(t, "/health")
    require.NoError(t, err)
    require.Equal(t, "ok", resp)
}
```

`examples/` 配下の binary は本 repo の `examples/` ディレクトリ実体を参照する。external repo を持たず、本 repo 内で完結する設計（採用検討者がクローンするだけで全 example を起動できる DX）。

## 部位を跨ぐ test の判定

user suite では smoke / examples の境界が曖昧になる test がある。例えば「k1s0 install + tier2-go-service 起動 + round-trip 確認」は両方の責務を兼ねる。判定基準は **検証の主目的** で、

- k1s0 自体の install が正常か → smoke/
- 特定 example の動作確認 → examples/

両方が主目的の場合、smoke/ と examples/ の 2 件に分割する。smoke で「k1s0 install + 最小 round-trip」、examples で「example 個別の動作確認」を行う。重複は最小化されるが、smoke の責務（install 正常性）と examples の責務（example 動作確認）が分離されるため保守性が上がる。
