# 02. owner suite ディレクトリ構造

本ファイルは ADR-TEST-008 で確定した `tests/e2e/owner/` 配下のディレクトリ構造を実装段階の正典として固定する。8 部位（platform / observability / security / ha-dr / upgrade / sdk-roundtrip / tier3-web / perf）の責務分担、独立 Go module、helpers 共通化規約、go.mod の依存範囲を ID として採番する。

## 本ファイルの位置付け

ADR-TEST-008 で owner suite を `tests/e2e/owner/` に物理分離し、user suite と go.mod を分けることを確定した。本ファイルでは 8 部位の責務範囲・依存関係・helpers 配置を実装段階の運用契約として固定する。8 部位の独立性が崩れると、test code の保守性低下 / cross-cutting bug の発見漏れが発生するため、責務分担をディレクトリ構造で表現する。

## 8 部位の責務分担

各部位は独立した検証類型を覆い、同 part 内では機能的にまとまる。部位を跨ぐ共通 helper は `tests/e2e/owner/helpers/` に配置する。

| 部位 | 検証対象 | 主要依存 | 想定 test 件数（運用拡大時） |
|---|---|---|---|
| `platform/` | tier1 12 service の機能検証（State / Audit / PubSub / Workflow / Decision / Pii / Feature / Telemetry / Log / Binding / Secret / Invoke）| `helpers/k8s_client.go` + `tests/e2e/owner/helpers/sdk_client/`（4 言語 SDK 経由）| 60〜80 |
| `observability/` | OTLP trace 貫通 / Prometheus cardinality / log↔trace 結合 / SLO alert 発火 / dashboard goldenfile の 5 検証（ADR-TEST-009、`04_観測性5検証.md` で詳細） | `helpers/{tempo,prometheus,loki,alertmanager,grafana}_client.go` | 5（5 検証それぞれ 1 件）|
| `security/` | Kyverno deny policy 強制 / NetworkPolicy 適用 / SPIRE workload identity / Istio mTLS STRICT / Trivy CVE block | `helpers/k8s_client.go` + `helpers/cve_client.go` | 15〜20 |
| `ha-dr/` | 3CP HA 切替（control-plane 1 down で別 CP に fail-over）/ etcd snapshot 復旧 / CNPG barman-cloud 復旧 / Argo CD GitOps 完全再構築 | `helpers/k8s_client.go` + `helpers/disaster_injector.go` | 4（4 経路それぞれ 1 件）|
| `upgrade/` | kubeadm N → N+1 minor version upgrade（control-plane 各 5 分 / worker 各 3 分 / 全体 30 分以内）| `helpers/k8s_client.go` + `helpers/version_planner.go` | 1（毎月 1 回想定）|
| `sdk-roundtrip/` | 4 言語 SDK × 12 RPC = 48 cross-product（Go / Rust / .NET / TypeScript で各 12 RPC を 1 件ずつ） | `helpers/sdk_client/{go,rust,dotnet,typescript}_runner.go` | 48 |
| `tier3-web/` | tier3 Web frontend の Keycloak ログイン → BFF → tier1 round-trip（chromedp で headless Chrome 駆動、ADR-TEST-008 §7） | `helpers/chromedp_session.go` + `helpers/keycloak_oidc.go` | 5〜10 |
| `perf/` | k6 spawn を Go test がラップした perf 試験（API latency / throughput / p99 SLO 違反検出） | `helpers/k6_runner.go` + `helpers/k8s_client.go` | 5〜10 |

8 部位の合計 test 件数は運用拡大時で約 150 件、所要時間は約 60 分（並列実行可能な部位は並列）。リリース時点では各部位 1〜3 件の skeleton（Skip）で開始する。

## go.mod 構成

`tests/e2e/owner/go.mod` は単一 module として管理する。8 部位は同 module 内 sub-package で、import path は `github.com/k1s0/k1s0/tests/e2e/owner/{part}` の形を取る。

```text
tests/e2e/owner/
├── go.mod                              # module github.com/k1s0/k1s0/tests/e2e/owner
├── go.sum
├── helpers/                            # 共通 helper（部位を跨ぐ）
│   ├── k8s_client.go                   # kubectl 操作 / Pod 状態確認
│   ├── sdk_client/                     # 4 言語 SDK 呼び出し（go.mod 経由 + child process spawn）
│   │   ├── go_runner.go
│   │   ├── rust_runner.go              # cargo test 子プロセス
│   │   ├── dotnet_runner.go            # dotnet test 子プロセス
│   │   └── typescript_runner.go        # vitest 子プロセス
│   ├── tempo_client.go                 # observability/ 用
│   ├── prometheus_client.go
│   ├── loki_client.go
│   ├── alertmanager_client.go
│   ├── grafana_client.go
│   ├── chromedp_session.go             # tier3-web/ 用
│   ├── keycloak_oidc.go
│   ├── k6_runner.go                    # perf/ 用
│   ├── disaster_injector.go            # ha-dr/ 用
│   ├── version_planner.go              # upgrade/ 用
│   └── cve_client.go                   # security/ 用
├── platform/
├── observability/                      # ADR-TEST-009 の 5 検証
├── security/
├── ha-dr/
├── upgrade/
├── perf/
├── sdk-roundtrip/
└── tier3-web/
```

`go.mod` に含める主要依存:

- `github.com/stretchr/testify` — assertion library（go test 互換）
- `k8s.io/client-go` / `k8s.io/api` — k8s API direct 呼び出し（kubectl 経由ではなく）
- `github.com/chromedp/chromedp` — tier3-web/ の headless Chrome 駆動
- `go.k6.io/k6` — perf/ の k6 spawn ラッパ
- `github.com/grafana/loki/pkg/logcli` — loki_client.go
- `github.com/grafana/tempo` — tempo_client.go（HTTP API client）
- `github.com/prometheus/client_golang/api` — prometheus_client.go

これらの重量依存は user suite の `go.mod` に持ち込まない（ADR-TEST-008 の 2 module 分離の根拠）。owner module のみで完結するため、user 側の clone / build 時間は影響を受けない。

## helpers の独立性

各 helper は単一責任原則で実装する。複数の part が同 helper を import するが、helper 内で part 固有のロジックを持たないようにする。例えば `k8s_client.go` は「Pod 状態取得」を提供するが、「観測性 namespace の Pod 状態取得」は提供しない（呼び出し側が namespace を引数で渡す）。

helpers 内の cross-helper 依存は許容するが循環依存は禁止する。例えば `chromedp_session.go` が `keycloak_oidc.go` を import するのは可（OIDC handshake の前提）、`k8s_client.go` が `prometheus_client.go` を import するのは禁止（K8s 操作と観測性は独立責務）。

## 部位を跨ぐ test の判定

実装上、部位を跨ぐ test（例: 「観測性 stack を起動してから tier1 facade を叩いて metric を確認する」）が発生し得る。判定は **主要な検証対象** で行う。上記例は「metric を確認する」が主目的なら observability/、「tier1 facade の機能を確認する」が主目的なら platform/ に置く。

両方が主目的の場合は 2 件に分割する。1 件目は platform/ で「tier1 facade が想定通り動く」、2 件目は observability/ で「metric が baseline 通り出る」。1 件で両方を兼ねる test は cross-cutting bug の検出能力が落ちる。
