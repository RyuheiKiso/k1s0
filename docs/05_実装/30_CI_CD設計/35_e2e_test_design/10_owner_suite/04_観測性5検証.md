# 04. 観測性 5 検証（owner only）

本ファイルは ADR-TEST-009 で確定した観測性 E2E の 5 検証（OTLP trace 貫通 / Prometheus cardinality regression / log↔trace 結合 / SLO burn rate alert 発火 / Grafana dashboard goldenfile）を実装段階の正典として固定する。各検証の実装契約・helper client 責務・baseline JSON 配置・段階拡張を ID として採番する。

## 本ファイルの位置付け

ADR-OBS-001（Grafana LGTM）/ ADR-OBS-002（OTel Collector）/ ADR-OBS-003（インシデント分類）/ ADR-OPS-001（Runbook `runbook_url` 必須要件）の 4 ADR は、観測性スタックが「実装かつ機械検証されている」前提で意味を持つ。本 5 検証はこの実証経路を担う層で、`tests/e2e/owner/observability/` 配下の独立 Go test として配置する。kind では再現できない本番運用 fidelity（Mimir cardinality 制御 / Tempo trace TTL / Loki retention）が検証範囲に含まれるため owner suite に配置し、user suite には公開しない。

## 5 検証の実装契約

各検証は独立 Go test として実装し、`./observability/{検証 ID}_test.go` の単一ファイルで完結する。helpers は `tests/e2e/owner/helpers/{tempo,prometheus,loki,alertmanager,grafana}_client.go` を参照する。

### 検証 1: OTLP trace 貫通

OTel Collector が tier1→2→3 を貫通する trace_id を伝播するかを機械検証する。OTLP HTTP `/v1/traces` で span を送信 → Tempo HTTP API `/api/traces/<trace-id>` で取得し、span tree が `tier3-bff → tier2-service → tier1-state` の 3 段階で構成されているかを assert する。span tree の depth と service name が baseline と一致することを `helpers/tempo_client.go` の `GetSpanTree` 経由で確認する。

```text
test 構造:
1. trace_id を生成（uuid v4）
2. tier3-bff の HTTP endpoint を叩き、X-Trace-Id header に上記 trace_id を載せる
3. tier3-bff → tier2-service → tier1-state の round-trip 完了まで wait
4. Tempo に GET /api/traces/<trace-id> で取得（最大 30 秒 retry）
5. span tree を assert: depth=3 / service names = ["tier3-bff", "tier2-service", "tier1-state"] / parent-child link 正常
```

失敗時は Tempo response body を `tests/.owner-e2e/<YYYY-MM-DD>/observability/trace_propagation.json` に dump し、root cause 解析の起点とする。

### 検証 2: Prometheus cardinality regression

tier1 / tier2 が出す metric の cardinality（label combination 数）が baseline の 1.2 倍を超えないことを機械検証する。`infra/observability/cardinality/baselines/<metric_name>.json` に baseline（label combination 数の上限）を記録し、Prometheus `/api/v1/series` で取得した実数と比較する。

```text
test 構造:
1. baselines/<metric_name>.json を読み込み（前回 PASS 時の cardinality 上限）
2. Prometheus /api/v1/series?match[]=<metric_name> で series 数を取得
3. (実数 / baseline) > 1.2 で fail
4. PASS の場合、現在の cardinality を baseline JSON に上書き（commit message で正当化）
```

baseline 更新の commit message には「ADR-TEST-009 baseline 更新: <metric 名> / 更新理由」を必須記載する。`tools/audit/run.sh` で baseline 更新 commit の件数を集計して AUDIT.md で報告する仕組みは ADR-TEST-009 §5（baseline 管理）で確定済。

### 検証 3: log↔trace 結合

tier1 / tier2 / tier3 が出す log の **95% 以上に trace_id field が含まれる** ことを機械検証する。Loki LogQL `{service_name=~"k1s0-.*"} | trace_id != ""` で trace_id 付き log の件数、`{service_name=~"k1s0-.*"}` で全 log 件数を取得し、比率が 95% 以上であることを assert する。

```text
test 構造:
1. observation window: 直近 5 分（実走時点から）
2. Loki query: total = count_over_time({service_name=~"k1s0-.*"}[5m])
3. Loki query: with_trace = count_over_time({service_name=~"k1s0-.*"} | trace_id != ""[5m])
4. (with_trace / total) >= 0.95 で PASS
```

95% 閾値は SRE 業界の log↔trace 結合率の標準的な目標値で、不足する 5% は startup log や health check log で trace_id がないことを許容する範囲。

### 検証 4: SLO burn rate alert 発火（runbook_url 必須）

意図的な SLO 違反（k6 で latency 超過注入）を流し、fast burn rate alert が 5 分窓内に発火し、alert label に `runbook_url` が含まれることを機械検証する。これは ADR-OPS-001 の「全 alert に `runbook_url` 必須」要件の継続検証経路である。

```text
test 構造:
1. k6 で意図的な latency 違反を 6 分間注入（SLO=200ms に対して 1000ms に伸ばす）
2. 5 分待機（fast burn rate alert の発火 window）
3. Alertmanager /api/v2/alerts で active alert を取得
4. labels.alertname == "K1s0FastBurn" の alert が存在することを assert
5. labels.runbook_url が non-empty で URL 形式（^https?://）であることを assert
6. annotations.summary に SLO 違反量が含まれることを assert
7. k6 注入を停止、alert resolve まで待機
```

alert 名 / SLO 値 / runbook_url 形式は `infra/observability/alerts/baselines/k1s0_fast_burn.json` に baseline として記録し、変更時は commit message で正当化する。

### 検証 5: Grafana dashboard goldenfile

`infra/observability/grafana/dashboards/<dashboard_id>.json` を baseline JSON（`tests/e2e/owner/observability/baselines/dashboards/<dashboard_id>.json`）と canonical diff で比較し、panel / query / threshold の不変を機械検証する。dashboard 破壊（panel 削除 / query 変更 / threshold 不変）を検出する。

```text
test 構造:
1. Grafana HTTP API GET /api/dashboards/uid/<dashboard_id> で取得
2. response.dashboard を canonical JSON 形式（key sort + 数値正規化 + null 除去）に変換
3. baselines/dashboards/<dashboard_id>.json も同様に canonical 化
4. 2 つを diff、差分 0 で PASS
5. 差分があれば失敗、unified diff を artifact に出力
```

dashboard 更新時は本検証が fail する前提で、PR 内で `git diff` の意図を commit message に書き、baseline JSON を同 PR で更新する。本検証は他のテスト類型では検出できない独立価値（dashboard の panel / query 不変保証）を持つ。

## 段階拡張

5 検証の実装は段階展開する。リリース時点では全 5 件 skeleton（Skip）配置、採用初期で検証 1 / 5 を real 化、運用拡大時で検証 2 / 3 / 4 を real 化する（ADR-TEST-009 §3）。

| Phase | 範囲 |
|---|---|
| リリース時点 | 5 件 skeleton（test 関数枠 + `t.Skip("PHASE: release-initial, real impl from 採用初期")`）|
| 採用初期 | 検証 1（trace 貫通）+ 検証 5（dashboard goldenfile）の real 化 |
| 採用後の運用拡大時 | 検証 2 / 3 / 4 の real 化 + baseline JSON 整備 |
| 全件 real | 5 検証完備、owner full で全件 PASS が前提 |

skeleton 状態でも `t.Skip` の skip 理由は明示的に書き、AUDIT.md で skip 件数を集計することで「いつまで skeleton のまま放置されているか」が監視される。

## helpers の責務分担

5 検証は独立した OSS API を叩くため、helpers も 5 client に分かれる。各 client は `tests/e2e/owner/helpers/` 配下に配置する。

| client | 責務 | 検証 |
|---|---|---|
| `tempo_client.go` | OTLP HTTP `/v1/traces` 送信、Tempo HTTP API での span tree 取得 | 1 |
| `prometheus_client.go` | Prometheus `/api/v1/series` / `/api/v1/labels` での series / label 数取得 | 2 |
| `loki_client.go` | Loki LogQL での log 検索、trace_id field 比率計算 | 3 |
| `alertmanager_client.go` | Alertmanager `/api/v2/alerts` で active alert 取得、label 抽出 | 4 |
| `grafana_client.go` | Grafana HTTP API で dashboard JSON 取得（API key 認証） | 5 |

各 client は単一責任で独立しており、cross-client 依存は持たない。共通機能（HTTP retry / context propagation）は `helpers/common.go` に括り出す。

## baseline 管理規約

検証 2（cardinality）/ 検証 4（alert label）/ 検証 5（dashboard）の baseline JSON は以下の規約で管理する。

- 配置: 検証 2 は `infra/observability/cardinality/baselines/<metric>.json`、検証 4 は `infra/observability/alerts/baselines/<alert>.json`、検証 5 は `tests/e2e/owner/observability/baselines/dashboards/<dashboard>.json`
- 更新ルール: baseline 更新の commit message に「ADR-TEST-009 baseline 更新: <名> / 更新理由」を必須記載
- 監視: `tools/audit/run.sh` が baseline 更新 commit の件数を集計、AUDIT.md で月次報告
- 検証時の取り扱い: PR 中で baseline 更新があった場合、レビュアーは更新理由が ADR-OBS-001 / 002 / 003 の決定変更と整合するか確認
