# 05. 実装

本ディレクトリは k1s0 の実装フェーズで参照する設計ドキュメントを格納する。概要設計（`docs/04_概要設計/`）で論理レベルの方式が確定した後、実装チーム（tier1 Rust / tier1 Go / tier2 / tier3 / infra / ops / docs）が具体的な作業に入る際のガイドラインを提供する。

## 本ディレクトリの位置付け

概要設計は「何をどの方式で実現するか」を論理レベルで固定するドキュメント群である。一方で実装フェーズでは、ディレクトリ配置・ビルド単位・CI/CD 設定・運用スクリプト・スパースチェックアウトの cone 定義など、物理配置レベルの判断が大量に発生する。これらを概要設計に混ぜると、論理決定と物理配置の境界が ID 上で曖昧になり、改訂影響範囲の特定が困難になる。

本ディレクトリはその種の物理配置ドキュメントを集約する。実装 ID は接頭辞 `IMP-` を持ち、概要設計 ID `DS-` と衝突しない。詳細は [../04_概要設計/00_設計方針/03_設計ID体系とトレーサビリティ.md](../04_概要設計/00_設計方針/03_設計ID体系とトレーサビリティ.md) の「実装 ID 体系」節を参照。

```
    ├── 80_スパースチェックアウト運用/

```

Phase 0 稟議承認時点では `00_ディレクトリ設計/` のみを確定範囲とし、以降のサブディレクトリ（ビルドシステム・CI 設定・運用スクリプトの詳細設計）は Phase 1a 以降で順次追加する。

## 実装フェーズの作業原則

- すべての物理配置判断は本ディレクトリ配下のいずれかの Markdown に IMP- 接頭辞付き ID で記録する
- 物理配置が対応する概要設計 ID（`DS-` または `ADR-`）を本文末尾に明記し、双方向トレーサビリティを維持する
- 物理配置変更は概要設計の変更よりも頻度が高いことを許容するが、対応する `DS-` を変更する際は本ディレクトリの該当 IMP- も同時更新する
