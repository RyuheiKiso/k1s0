# 01. DX メトリクス原則

本ファイルは k1s0 の開発者生産性メトリクス（DORA 4 keys / SPACE / Scaffold 利用率 / time-to-first-commit）を運用する際に常に参照する 7 本の原則を定義する。稼働側の SLI（`60_観測性設計/`）とは分離し、「開発チームの健全性」を独立した数値で可視化する運用を構造レベルで規定する。

![DX メトリクス原則俯瞰](../img/95_DORA_SPACE_Scaffold利用率.svg)

## 原則が必要な理由

DX メトリクスは「開発チームの健全性」を測る指標であり、稼働 SLI（`60_観測性設計/`）とは参照者・意思決定サイクル・改善アクションが異なる。DX メトリクスと稼働 SLI を同じダッシュボードに並べた運用は、以下の破綻を生む。

- 可用性 99.9% 未達の障害対応中に、Lead Time 悪化の警告が並んで表示され、優先順位が混濁する
- Deploy Frequency を改善しようとして稼働 SLO の品質ゲートを緩める誘因が発生する
- EM が読むべき「チームの健全性」と、SRE が読むべき「プロダクトの健全性」が混ざり、どちらの意思決定も鈍化する
- 独自ダッシュボードがチームごとに乱立し、比較可能性が失われる
- DORA 4 keys のうち一部のみを恣意的に計測し、「都合の良い数字」で健全性を偽装する
- 2 名段階で SPACE を導入し、アンケート疲れでデータ自体が取れなくなる
- Scaffold 利用率の低下を計測しない結果、Paved Road が風化していることに気付けない

本原則は、これらの破綻を「稼働 SLI と DX メトリクスの物理分離 + DORA 4 keys 全計測 + Backstage Scorecards 第一主義」で構造的に防ぐ 7 本である。エンジニアリングマネージャ層（EM）が意思決定の根拠として使える精度を維持する。

## 原則 1: 稼働 SLI と DX メトリクスは別ダッシュボードに分離する（IMP-DX-POL-001）

**稼働 SLI（可用性 / レイテンシ / エラー率）と DX メトリクス（DORA 4 keys / Scaffold 利用率 / time-to-first-commit）は、Grafana の別ダッシュボード・別フォルダに配置する。同一画面への混在表示を禁止する。**

ADR-DX-001（本章初版策定時に起票予定）で本分離原則を確定する。両者は参照者（SRE vs EM）と行動の粘度（分単位 vs 月単位）が異なるため、同一画面では優先順位が混濁する。物理分離により、SRE は稼働 SLI を、EM は DX メトリクスをそれぞれ独立に判断できる状態にする。

- 稼働 SLI ダッシュボード : `infra/observability/grafana/dashboards/sli/`
- DX メトリクスダッシュボード : `infra/observability/grafana/dashboards/dx/`
- クロス参照が必要な場合（例: Change Failure Rate と MTTR）は、リンクのみとし埋め込み禁止

## 原則 2: DORA 4 keys を全て計測する（IMP-DX-POL-002）

**Lead Time for Changes / Deployment Frequency / Mean Time to Restore / Change Failure Rate の 4 つを全て計測する。一部のみの計測を禁止する。**

DORA 4 keys は「スピード（Lead Time / Deploy Frequency）」と「安定性（MTTR / Change Failure Rate）」を対で計測することで、片方だけを改善する偽装を防ぐ設計である。一部のみを計測すると、例えば Deploy Frequency 向上のために品質を犠牲にする行動が検出されない。

計測点は以下に固定する（`03_要件定義/50_開発者体験/` の DORA 要件を物理配置で満たす）。

- Lead Time : PR 作成から production デプロイ完了までの中央値
- Deploy Frequency : production へのデプロイ回数（日次 / 週次）
- MTTR : インシデント発生から復旧までの中央値（`60_観測性設計/` の Incident と連動）
- Change Failure Rate : production デプロイのうちロールバック / ホットフィックスが発生した割合

## 原則 3: Backstage Scorecards を第一手段とする（IMP-DX-POL-003）

**DX メトリクスの表示面は Backstage Scorecards を第一手段とする。サービス別・チーム別のスコアを Backstage カタログ（catalog-info.yaml）に紐付け、独自ダッシュボードの乱立を禁止する。**

ADR-BS-001 で Backstage を選定済であり、Scorecards プラグインを DX メトリクスの一次表示面とする。独自ダッシュボードは「このチームだけが見る指標」を作り出し、比較可能性を失わせる。Scorecards を通すことで、全チームが同じ指標軸で比較される状態を維持する。

- Scorecards の定義は `tools/backstage/scorecards/` に配置
- 新規指標追加は Scorecards 側で定義し、Grafana 独自ダッシュボードから昇格させる
- Grafana 詳細ダッシュボードは Scorecards からのドリルダウン先として維持（一次表示面ではない）

## 原則 4: time-to-first-commit を独立 SLI として計測する（IMP-DX-POL-004）

**新規参画者が「リポジトリ clone → ローカルで 1 コミット merge まで」に要する時間を独立 SLI として計測する。目標値を 1 営業日以内に固定する。**

time-to-first-commit は Paved Road（`50_開発者体験設計/`）の健全性を最も素直に測る指標である。Dev Container / Scaffold / ローカル CI / ドキュメントが揃っていなければ、1 営業日以内の初回コミットは成立しない。逆に言えば、本指標が悪化した時点で、DX チームは 50 章の整備状況を見直す必要がある。

計測方法は「新規参画者の入社日」を起点に、github の最初の merge されたコミットタイムスタンプまでとする。対象期間は入社後 14 日間で打ち切り、14 日超過は `>14d` として記録する。10 種類の役割別 cone（`tier1-rust-dev` 等）ごとに別集計し、特定役割の Paved Road が劣化しているケースを検出可能にする。

## 原則 5: Scaffold 利用率を Paved Road 健全性の指標とする（IMP-DX-POL-005）

**新規サービス作成時の Scaffold（`platform/scaffold/` の雛形 CLI）経由率を計測し、70% を下回った場合は Paved Road 再整備を発動する。**

Scaffold 利用率は「開発者が Paved Road を選ぶか、自力で道を作るか」を測る。低下は Paved Road が風化しているシグナルであり、テンプレートの機能不足・ドキュメント不足・新規要求との乖離を示唆する。

計測方法は新規リポジトリ / 新規サービスディレクトリ作成時のコミット履歴を走査し、Scaffold の生成物パターンが含まれるかで判定する。70% の閾値は経験則であり、Phase 1a 以降の実測値で再調整する。低下時の対応は `50_開発者体験設計/` にフィードバックする。Scaffold の golden 出力は `tests/golden/scaffold/` に snapshot 保管し、テンプレート改訂時の差分が常に可視化される状態を維持する。

## 原則 6: EM レイヤーへの定期レポートを月次で提出する（IMP-DX-POL-006）

**DORA 4 keys / Scaffold 利用率 / time-to-first-commit を月次レポートとして EM（エンジニアリングマネージャ）に提出する。レポートテンプレートは `tools/dx-report/` に固定する。**

メトリクスは「見られない限り意味がない」。EM レイヤーに定期的に届ける仕組みがなければ、ダッシュボードは徐々に形骸化する。月次という粒度は EM の意思決定サイクル（評価 / 計画）と整合し、日次・週次では粒度が細かすぎる。

レポート項目は以下に固定する。

- DORA 4 keys の月次推移（前月比・3 か月比）
- Scaffold 利用率の推移と Paved Road 整備状況
- time-to-first-commit の中央値と長期滞留者
- 改善アクションの提案（DX チームから）

## 原則 7: SPACE は 10 名以上時点で導入する（IMP-DX-POL-007）

**SPACE Framework（Satisfaction / Performance / Activity / Communication / Efficiency）はチーム規模が 10 名以上に達した時点で導入する。2 名運用段階では DORA 4 keys のみとする。**

SPACE はアンケート由来の主観指標を含むため、母集団が小さいと個人特定可能性と統計的有意性の両面で問題が出る。2 名運用段階で SPACE を導入すると、アンケート疲れでデータ自体が取れなくなり、DX メトリクス全体への信頼が崩れる。アンケートの匿名性は 10 名以上でも担保される必要があり、回答データは集約後のみ保存し個人回答の長期保管を禁止する。

- 2 名運用期（Phase 0 〜 Phase 1a 前半） : DORA 4 keys のみ
- 10 名規模到達時（Phase 1a 後半以降） : SPACE の Satisfaction / Communication を追加
- 30 名規模到達時 : SPACE 全 5 軸を導入

## 図表

```
[DX メトリクス 7 原則のレイヤ構造]
  レイヤ 1 : 分離（稼働 SLI と DX を別画面）
  レイヤ 2 : 全計測（DORA 4 keys / Scaffold / t2fc）
  レイヤ 3 : 表示面統一（Backstage Scorecards）
  レイヤ 4 : EM への配信（月次レポート）
  レイヤ 5 : 段階導入（2 名 → 10 名 → 30 名）
```

各指標と計測点・参照者・改善アクションの対応は以下となる。

- Lead Time : CI イベント → Scorecards → EM が Paved Road / CI 最適化に指示
- Deploy Frequency : Argo CD イベント → Scorecards → EM がリリース頻度計画
- MTTR : Incident event → Scorecards → SRE / EM が復旧演習の頻度再設計
- Change Failure Rate : Rollout event → Scorecards → 品質ゲート見直し
- Scaffold 利用率 : Repository scan → Scorecards → DX チームが Paved Road 整備
- time-to-first-commit : Git log → Scorecards → DX チームが Dev Container / ドキュメント改善
- SPACE（10 名以上）: Survey → 集約後 Scorecards → EM がチーム健全性判定

詳細な指標フロー図は [img/DXメトリクス原則俯瞰.drawio](../img/95_DORA_SPACE_Scaffold利用率.drawio)（Phase 0 完了時点で svg 作成）を参照。
