# 01. 依存管理原則

本ファイルは k1s0 モノレポの外部依存（crates.io / Go modules / npm / NuGet / サードパーティ vendoring）と依存更新運用に関する、常に参照する 7 軸の原則を定義する。依存追加・更新・vendoring・ライセンス判定が発生した際、本原則との整合を確認することで、2 名運用の JTC でも依存管理が破綻しないラインを維持する。

![依存管理の流路と判定点](img/依存管理流路.svg)

## 原則が必要な理由

k1s0 は ADR-0003（AGPL 分離アーキテクチャ）により、Grafana / Loki / Tempo / Pyroscope / MinIO / Renovate の AGPL OSS 6 件を「別プロセス / 別イメージ / 別ネットワーク境界」で運用することが前提となっている。この分離が依存グラフ上で崩れると、自社コードが AGPL に汚染されるリスク（AGPL 第 13 条のネットワーク公開時開示義務）が発生する。加えて 2 名運用では日々の Renovate PR を 1 件ずつ人手 merge することが業務不能であり、自動化可能な範囲とレビュー必須な範囲の線引きが運用の死活を決める。

典型的な破綻は以下である。

- AGPL ライブラリが tier1 の Rust 依存に紛れ込み、自社コードまで AGPL 公開義務を負う
- lockfile が commit されず、開発者 A と CI で依存バージョンがずれ「自分の端末では動く」が発生
- vendoring した OSS の `UPSTREAM.md` が無く、上流の新バージョンに戻る術が消える
- 依存更新 PR が 200 件溜まり、誰もレビューしないまま脆弱性を抱え続ける
- minor / major 更新が patch 扱いで自動マージされ、API 破壊変更がランタイムで露見
- SBOM が一部アーティファクトに無く、脆弱性通知が来た際に影響範囲を特定できない
- BUSL / SSPL など非互換ライセンスが知らぬ間に混入し、配布時に法務問題化

本原則は、これらの破綻を構造レベルで回避するために設けた 7 軸である。

## 原則 1: 依存更新は Renovate 経由のみ（IMP-DEP-POL-001）

**外部依存（`Cargo.toml` / `go.mod` / `package.json` / `*.csproj`）のバージョン更新は Renovate が作成する PR 経由のみとし、人手での直接編集を禁止する。**

人手更新は「なぜその版に上げたか」の記録が残らず、10 年保守の間に判断根拠が失われる。Renovate は CVE・リリースノート・破壊変更情報を PR 本文に自動添付するため、レビュー時点で判断材料が揃う。

具体的な帰結として、以下を固定する。

- 新規依存追加時のみ `Cargo.toml` / `go.mod` / `package.json` / `*.csproj` の手修正を許容
- 既存依存のバージョン変更は Renovate PR のみ受け付ける（CI lint で検出して拒否）
- `renovate.json` の設定変更は `tools/ci/` 配下の一元定義を経由する
- Renovate 運用は ADR-DEP-001（本章初版策定時に起票予定）で正式化

Renovate の実行頻度は日次、PR グループ化は minor / major 単位で週次集約とする（Phase 1a）。

## 原則 2: lockfile は commit 必須（IMP-DEP-POL-002）

**`Cargo.lock` / `go.sum` / `pnpm-lock.yaml` / `packages.lock.json` はすべて commit し、`.gitignore` で除外することを禁止する。**

lockfile が無いと、同じ `Cargo.toml` を 2 つの端末で `cargo build` した結果が異なる版を拾い、「私の端末では通る」問題の温床となる。ライブラリプロジェクトは `Cargo.lock` を commit しない慣習があるが、k1s0 は全クレートがアプリケーション側として動作するため、一律 commit する。

lockfile 運用の要件は以下とする。詳細は `20_lockfile運用/` で規定する。

- Rust: `Cargo.lock` を全 workspace で commit
- Go: `go.sum` を commit、`go.mod` の `// indirect` は整理状態を維持（`go mod tidy`）
- TypeScript: `pnpm-lock.yaml` を commit、`package-lock.json` / `yarn.lock` は削除
- .NET: `packages.lock.json` を生成する `RestorePackagesWithLockFile` を有効化して commit
- CI での `--frozen-lockfile` / `go mod verify` / `cargo install --locked` 相当を必須化

lockfile の手動編集は禁止し、Renovate PR か `cargo update --package X` 等の明示コマンドのみ許容する。

## 原則 3: vendoring は UPSTREAM と PATCHES が必須（IMP-DEP-POL-003）

**`third_party/` への OSS vendoring 時は `UPSTREAM.md`（上流 URL + コミット SHA + 分岐理由）と `PATCHES.md`（独自パッチの一覧と根拠）を必ず配置する。**

vendoring は「上流に戻れなくなった瞬間に技術的負債として恒久化」する典型パターンである。本原則は上流復帰可能性を常に開いた状態に保つための最小要件を定める。

vendoring 採用条件は以下とする。

- 上流が修正受領しない / 応答しないが当該パッチが自社運用に必須
- 上流のリリースサイクルが k1s0 のリリース要件に合わない（かつ issue / PR を上流に提出済み）
- ライセンスが copyleft 非互換で、コード取込ではなく再配布境界の設計が必要

`UPSTREAM.md` / `PATCHES.md` の必須項目は以下。

- `UPSTREAM.md` : 上流リポジトリ URL / 分岐コミット SHA / 採用日 / 採用理由 / 上流 issue / PR URL / 上流復帰の予定 Phase
- `PATCHES.md` : パッチファイル名 / 変更ファイル / 変更理由 / 上流に提出した issue / PR URL

Phase 1c 時点で全 vendored OSS について上流復帰可否を再評価する。

## 原則 4: ライセンスは SPDX 明示、BUSL / SSPL は自動拒否（IMP-DEP-POL-004）

**すべての外部依存はライセンスを SPDX 識別子で管理し、CI で非互換ライセンス（BUSL / SSPL / Commons Clause 等）を検出した PR は自動拒否する。**

ライセンスは再配布可否を決定するため、1 件の非互換ライセンス混入が k1s0 の商用配布全体を危うくする。手動チェックは 2 名運用では持続不能で、機械チェック一択となる。

SPDX 運用の要件は以下とする。詳細は `40_ライセンス監査/` で規定する。

- 許可ライセンス: MIT / Apache-2.0 / BSD-2-Clause / BSD-3-Clause / MPL-2.0 / ISC / Unlicense
- 条件付き許可: LGPL-2.1 / LGPL-3.0（動的リンクのみ、tier 境界と整合）
- 自動拒否: BUSL-1.1 / SSPL-1.0 / Commons Clause / Elastic License / Confluent Community License
- 要承認（Security + Legal）: AGPL-3.0（原則 5 の分離境界外の場合は拒否）
- 不明ライセンス: 自動拒否、依存採用 PR で SPDX 明示を要求

使用ツールは Rust: `cargo-deny` licenses / Go: `go-licenses` / TS: `license-checker` / .NET: `nuget-license`。

## 原則 5: AGPL OSS 6 件は分離境界を恒常検証（IMP-DEP-POL-005）

**Grafana / Loki / Tempo / Pyroscope / MinIO / Renovate の AGPL OSS 6 件は ADR-0003 の分離アーキテクチャで運用し、依存グラフ上で自社コードから参照しないことを CI で検証する。**

AGPL 第 13 条は「ネットワーク越しに利用されるソフトウェアを改変した場合、改変版のソースコードを利用者に開示する義務」を課す。分離運用（別プロセス / 別イメージ / 別ネットワーク境界）を維持すれば自社コードは AGPL に汚染されないが、うっかり SDK として import してしまうと自社コード全体に開示義務が発生する。

分離検証の要件は以下とする。

- 依存グラフ lint: 自社 crate / go module / npm package / csproj から AGPL 6 件を import した場合に CI を fail
- ネットワーク境界: Grafana / Loki 等は独立 namespace に配置、ServiceAccount / NetworkPolicy で境界設定（`60_観測性設計/` と連動）
- イメージ境界: 自社イメージと AGPL OSS イメージを同一コンテナに同居させない
- `LICENSE-AGPL-isolated.md` を `third_party/` に配置し、分離境界の根拠を明文化

NFR-E-NW-003（AGPL 分離）はこの運用で達成する。

## 原則 6: 自動マージは patch レベルのみ（IMP-DEP-POL-006）

**Renovate PR の自動マージは semver の patch レベルに限定し、minor / major は必ず人手レビューを経る。**

patch レベルは仕様上「後方互換なバグ修正」であり、破壊リスクが最も低い。minor は新 API 追加を含み得るため使用側のコードパスに影響することがあり、major は破壊変更そのものである。2 名運用では patch の自動化だけで依存更新業務の 70% 程度は吸収でき、残りの minor / major に人手を集中できる。

自動マージの要件は以下とする（IMP-CI-POL-007 と整合）。

- 自動マージ対象: `^1.2.3` の patch 部分のみの更新
- 条件: quality gate 全 pass + Trivy Critical 無し + cosign 署名成功 + SPDX 許可ライセンス
- 非対象: minor / major 更新 / base image tag 変更のうち OS ベース変更 / AGPL 分離境界を跨ぐ更新
- minor / major は Renovate グループ化で週次バッチ処理し、定例レビュー会で一括判定
- セキュリティ修正のみ含む minor / major は例外的に自動マージ候補とする（CVE 修正を優先）

Phase 0 では全 PR を人手レビューとし、Phase 1a で patch 自動マージを開く。

## 原則 7: SBOM は全アーティファクトに添付（IMP-DEP-POL-007）

**CI が Harbor に push するすべてのコンテナイメージ、および tier2 / tier3 が配布する SDK パッケージに対して CycloneDX 形式の SBOM を生成して署名付きで添付する。**

SBOM が無いと、新たな CVE が公表された際に「どのアーティファクトに影響するか」を特定する手段が失われる。2 名運用の JTC で数十サービスの脆弱性影響を把握するには、SBOM 照合による機械的な絞り込みが唯一現実的な手段となる。

SBOM 運用の要件は以下とする。詳細は `80_サプライチェーン設計/` で規定する。

- 形式: CycloneDX 1.5 以降（JSON）
- 生成ツール: Syft（コンテナイメージ）/ cargo-cyclonedx（Rust）/ CycloneDX for Go / cyclonedx-npm / CycloneDX for .NET
- 署名: cosign attach で SBOM をイメージ attestation として添付、keyless 署名（IMP-CI-POL-005 と整合）
- 保管: Harbor の attestation に加えて、`ops/sbom/archive/` に 5 年保管（監査要件）
- 脆弱性照合: Dependency-Track で日次スキャン、新規 CVE 検出時は Security チームに自動通知

NFR-H-INT-002（SBOM 添付）/ NFR-C-MGMT-003（SBOM 100%）はこの運用で達成する。

## 図表

```
[依存管理の流路]
  crates.io / Go modules / npm / NuGet
         │
         ▼  Renovate 日次スキャン
  Renovate PR ───► quality gate（lint / test / SPDX / Trivy）
         │                          │
         │ patch + 条件全充足        │ minor / major
         ▼                          ▼
  自動マージ（Phase 1a 以降）    人手レビュー（週次バッチ）
         │                          │
         └──────┬───────────────────┘
                ▼
         lockfile 更新 commit
                │
                ▼
  CI build → SBOM 生成 → cosign 署名 → Harbor push
```

## 対応 IMP-DEP ID

本ファイルで採番する原則レベル ID は以下とする。

- `IMP-DEP-POL-001` : 依存更新は Renovate 経由のみ
- `IMP-DEP-POL-002` : lockfile commit 必須
- `IMP-DEP-POL-003` : vendoring の UPSTREAM / PATCHES 必須
- `IMP-DEP-POL-004` : SPDX ライセンス明示と BUSL / SSPL 自動拒否
- `IMP-DEP-POL-005` : AGPL 6 件の分離境界恒常検証
- `IMP-DEP-POL-006` : 自動マージは patch レベルのみ
- `IMP-DEP-POL-007` : SBOM 全アーティファクト添付

## 対応 ADR / DS-SW-COMP / NFR

- ADR-0003（AGPL 分離アーキテクチャ）/ ADR-DEP-001（Renovate 中心運用、本章初版策定時に起票予定）
- DS-SW-COMP-129 / 130（tier1 Rust / 契約配置）
- NFR-E-NW-003（AGPL 分離）/ NFR-H-INT-002（SBOM 添付）/ NFR-C-MGMT-003（SBOM 100%）/ NFR-C-MNT-002（OSS バージョン追従）
