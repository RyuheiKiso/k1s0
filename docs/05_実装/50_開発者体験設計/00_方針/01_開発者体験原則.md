# 01. 開発者体験原則

本ファイルは k1s0 の開発者体験（DX）を設計する際に常に参照する 7 軸の原則を定義する。Dev Container のプロファイル追加、`examples/` への新 Golden Path 登録、Scaffold CLI のテンプレート更新、Backstage カタログ属性の拡張など、DX 領域で発生するあらゆる設計判断は本原則との整合をレビューで確認する。

![開発者体験 7 原則マップ](../img/50_Golden_Path_Paved_Road概観.svg)

## 原則が必要な理由

k1s0 は 採用側の小規模運用から 採用側の運用拡大期へ拡大する 10 年保守プロジェクトであり、新規参画者が「最初のコミット」に到達する時間が線形に悪化すると拡大に失敗する。DX 設計を場当たり的に放置した場合、典型的な破綻は以下として観測される。

- Golden Path が `examples/` と `docs/` の両方に重複して置かれ、どちらが正か分からず、コピー元として後者を使ったサービスが Scaffold の契約（lint / CI / Backstage 連携）を満たさない
- 「開発環境は各自で作る」運用となり、tier1 Rust 開発者と tier3 Web 開発者で Lint 設定・VS Code 拡張・devcontainer image が分散し、onboarding に 2 週間かかる
- Scaffold CLI を経由せずに手で作ったサービスが Backstage に載らず、オーナー不明・オンコール連絡先不明のまま本番に到達する
- IDE 応答性を嫌って Dev Container を外す人が増え、ローカル環境依存のバグが頻発し、Platform チームが個別対応で疲弊する
- time-to-first-commit を計測していないため、DX 悪化が数値で検知できず、気付いた時には採用歩留まりが悪化している
- Scaffold テンプレートの変更が一人レビューで通り、欠陥が全サービスに横展開してから事故で気付く

本原則は、これらの破綻を「正しい道が最短経路」となる構造として予防する Paved Road 思想の 7 軸である。Paved Road は単なるテンプレート集ではなく「舗装された道を歩く限り Platform チームが全責任を負い、外れた道は自己責任」という責任分界のモデルであり、所有権と運用責任を構造化するための制度でもある。

## 原則 1: Paved Road を一本化する（IMP-DEV-POL-001）

**正しい開発経路は `examples/` に集約した Golden Path 一本とし、代替経路を作らない。**

Netflix の Paved Road 思想は「舗装された道を歩く限り Platform チームが全責任を負う。舗装を外れた者は自己責任」という運用モデルである。k1s0 はこれを `examples/` の実稼働版集約として実装する。tier2 / tier3 の代表的な 1 ケースずつを「コピーすれば新サービスが立ち上がる」状態で `examples/` に配置し、`docs/` 側にはコードスニペットではなく設計思想のみを置く。

具体的な帰結として、以下を固定する。

- Golden Path の配置先は `examples/` のみ。`docs/` や `tools/` に重複実装を置かない
- `examples/` は常に CI で build / test / deploy を通す「動く資産」として維持する
- 代替手段（「こういう書き方もある」）は却下。必要なら Golden Path 側を改訂して一本化する
- `examples/` からの逸脱は ADR で正当化し、逸脱後のサービスは Paved Road の保証対象外と明記する
- Golden Path は「tier2 サービス（Go）」「tier3 Web（TypeScript）」「tier3 Native（MAUI）」の最低 3 ケースを リリース時点 で揃え、それぞれが Dapr / Istio / flagd / Backstage 統合の完備例となる

## 原則 2: Scaffold CLI と catalog-info.yaml を必ず経由する（IMP-DEV-POL-002）

**新規サービスは Scaffold CLI からのみ作成可能とし、catalog-info.yaml を first-class 属性として自動生成する。**

手作成サービスが増えると Backstage カタログに載らず、所有者・オンコール連絡先・ドキュメント所在が不明のまま本番へ到達する。本原則は `tools/codegen/scaffold` 経由でしかサービスが立ち上がらない構造を CI で強制する。catalog-info.yaml は Scaffold が生成し、生成物には `# Generated by scaffold. DO NOT EDIT manually.` ヘッダを付与する。

- Scaffold が出す雛形には Dockerfile / Helm chart / Rollout manifest / catalog-info.yaml / README が必ず含まれる
- 手作成された catalog-info.yaml は CI で拒否（Scaffold のバージョンヘッダ欠落で検出）
- Backstage は catalog-info.yaml の有無をソース of truth とし、カタログ外のサービスは本番 admission で Kyverno により拒否する（`80_サプライチェーン設計/` と結合）
- Scaffold の実行ログ（起動時刻 / 完了時刻 / 選択テンプレート / 採用オプション）は Backstage scaffolder が収集し、time-to-first-commit SLI の計測源泉となる

## 原則 3: 10 役別 Dev Container で IDE 応答性と onboarding を両立する（IMP-DEV-POL-003）

**Dev Container は役割別 10 プロファイルで提供し、各役割が必要な最小ツールチェインのみを含める。**

tier1 Rust 開発者に .NET SDK と pnpm と MAUI ワークロードを全部入れた 12GB image を配ると、ビルドも Rust-Analyzer も遅くなり、VS Code の IntelliSense 応答が 10 秒超に悪化する。逆に役割を無視した最小 image は「この tier の設定が入っていない」というトリアージを頻発させる。本原則は ADR-DIR-003 で定めた 10 役（`tier1-rust-dev` / `tier1-go-dev` / `tier2-dev` / `tier3-web-dev` / `tier3-native-dev` / `platform-cli-dev` / `sdk-dev` / `infra-ops` / `docs-writer` / `full`）と 1:1 対応する Dev Container プロファイルを `tools/devcontainer/profiles/` に配置する。

- `docs-writer` はルート既定の軽量 image とし、drawio / markdownlint / textlint のみを含む
- `full` は CI 検証用に限定し、個人開発での日常使用を非推奨とする（応答性悪化のため）
- Dev Container のビルドは Platform/Build チームが週次で image を更新し、digest を GitHub Container Registry で固定する
- 各プロファイルの image サイズは 4GB 以内を目標とし、起動時間（コールドスタート）を 60 秒以内に保つ

## 原則 4: time-to-first-commit を SLI 化する（IMP-DEV-POL-004）

**time-to-first-commit を `95_DXメトリクス/` 配下で SLI として計測し、採用側の小規模運用 4 時間以内、採用側の運用拡大期 2 時間以内を目標とする。**

DX の劣化は数値化しないと検知できない。本原則は新規参画者が clone から最初のマージ済み PR に到達するまでの時間を自動計測し、目標を超過した週次を Platform/Build 週次定例で必ず議題化する。計測は Backstage scaffolder の実行ログと Git の first-commit timestamp を突き合わせて算出する。

- 計測基盤は `95_DXメトリクス/` に集約し、本章は SLI 対象として定義のみを担う
- 計測は「clone から first merged PR まで」を基本定義とし、docs-only PR や typo 修正 PR は対象外
- SLI 劣化が 2 週連続した場合、Golden Path / Scaffold / Dev Container のどこが遅延源かをボトルネック分析し、改善 PR を発行する
- 入社者のヒアリング（退出時の 1on1）で「詰まった地点」を言語化し、Paved Road に反映する
- 目標値は段階ごとに再評価し、採用側の運用拡大期到達後は 90 分、20 名相当の将来段階では 60 分を検討する

## 原則 5: IDE 設定を共有資産として管理する（IMP-DEV-POL-005）

**`.vscode/` / `.idea/` / `.editorconfig` / `rustfmt.toml` / `.prettierrc` はリポジトリに commit し、各自のローカル設定で上書きしない。**

IDE 設定を個人管理に委ねると、Lint 結果・フォーマット・保存時アクションが人によって揺れ、レビューで「コードの中身ではなくフォーマット差分」を議論する時間が肥大化する。本原則は IDE 設定を共有資産として扱い、共有設定からの逸脱は PR レビューで指摘対象とする。

- `.vscode/extensions.json` で必須拡張を宣言し、Dev Container 起動時に自動インストール
- `.editorconfig` を全言語横断のフォーマット既定とし、言語固有設定（rustfmt / gofmt / prettier）と矛盾しないよう設定レビューを行う
- ローカル上書きが必要な場合は `.vscode/settings.local.json`（gitignore 対象）に分離し、共有設定は汚さない
- JetBrains IDE（IntelliJ / Rider）の `.idea/` は inspection profile / code style のみ共有し、個人固有（workspace.xml 等）は除外
- フォーマット差分を commit に含めることを禁止し、pre-commit hook で自動整形する

## 原則 6: ローカル開発は kind/k3d + Dapr Local で本番を再現する（IMP-DEV-POL-006）

**ローカル開発環境は kind または k3d + Dapr Local stack で構成し、本番 Kubernetes 構造を再現する。**

「ローカルでは動くが本番で落ちる」は、ローカルが本番と異なる実行形態（docker-compose / 素のバイナリ）を使った時に必ず発生する。tier1 の Dapr ファサード・sidecar 注入・Istio Ambient による mTLS・flagd 評価は Kubernetes 前提であり、docker-compose では再現できない。本原則は `tools/local-stack/` に kind と k3d の両プロファイルを用意し、Dapr / Istio / flagd を本番相当で起動する。

- kind は CI での差分検証用、k3d は個人開発での日常用（起動速度の差異）
- ローカルでも ArgoCD を起動し、GitOps 経由でアプリを配信（本番と同じ経路を踏む）
- `examples/` の Golden Path は `make up` 一発でローカルクラスタ上に立ち上がること
- docker-compose は ローカル DB（Postgres / Kafka）の補助起動のみ許容。アプリ層の起動には使わない

## 原則 7: Scaffold 承認ゲートは SRE + Security 二重レビュー（IMP-DEV-POL-007）

**新規 Scaffold テンプレート追加・既存テンプレート改訂は、SRE と Security の二者承認を必須とする。**

Scaffold は「一人が書いたテンプレートが全サービスに横展開される」構造であり、セキュリティ欠陥や運用欠陥を含むテンプレートが無審査で配布されると、影響が全サービスへ一斉に波及する。本原則は Scaffold テンプレート変更 PR に CODEOWNERS で SRE チームと Security チームを必須レビュアーに設定し、片方のみでのマージを禁止する。

- 審査対象は Dockerfile / Helm / Rollout / catalog-info.yaml / CI ワークフローの 5 点
- SRE は運用性（SLI 露出 / Runbook 生成 / エラーバジェット連動）を審査
- Security は sigstore 署名 / SBOM 生成 / SLSA レベル到達 / 最小権限を審査
- 承認ゲートは GitHub の environments / branch protection で強制し、CODEOWNERS だけに依存しない
- 緊急パッチ時の one-person-ship は明示的な例外申請をコメントに残し、48 時間以内に事後審査を完了する
- テンプレート変更は Golden Path（`examples/`）の CI で regression を検証し、壊れたまま merge されない

## 対応 IMP-DEV ID

本ファイルで採番する原則レベル ID は以下とする。

- `IMP-DEV-POL-001` : Paved Road を `examples/` に一本化
- `IMP-DEV-POL-002` : Scaffold CLI と catalog-info.yaml 必須経由
- `IMP-DEV-POL-003` : 10 役別 Dev Container による応答性と onboarding の両立
- `IMP-DEV-POL-004` : time-to-first-commit を SLI 化（2 名 4h / 10 名 2h）
- `IMP-DEV-POL-005` : IDE 設定を共有資産として管理
- `IMP-DEV-POL-006` : ローカル開発は kind/k3d + Dapr Local で本番再現
- `IMP-DEV-POL-007` : Scaffold 変更は SRE + Security の二重承認
