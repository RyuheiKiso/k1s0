# 01. コード生成原則

本ファイルは k1s0 モノレポの 3 経路コード生成（buf Protobuf / OpenAPI / Scaffold CLI）を運用する際に常に参照する 7 軸の原則を定義する。契約変更・生成器更新・テンプレート改訂が発生した際、本原則との整合を確認することで、手修正ドリフトと 4 言語 SDK 間の不整合を構造的に防ぐ。

![コード生成 3 経路と真実源](img/コード生成3経路.svg)

## 原則が必要な理由

k1s0 は tier1 公開 11 API と tier1 内部 gRPC の契約を Protobuf で一元管理し、そこから 4 言語 SDK（Rust / Go / C# / TypeScript）と tier1 サーバースタブを自動生成する構造になっている（ADR-DIR-001 / ADR-TIER1-002 / ADR-TIER1-003）。さらに tier2 / tier3 の新規サービス雛形は Scaffold CLI（ADR-BS-001 の Backstage Software Template 互換）で展開する。ここに手修正を許すと、10 年保守の間に不整合が蓄積し、契約と実装が別物になる。

典型的な破綻は以下である。

- 4 言語 SDK のうち 1 言語だけ Protobuf 生成コードに手修正が入り、フィールド名が微妙にずれる
- `buf breaking` を回避して互換性破壊が本番へ到達し、既存クライアントがランタイムで Unmarshal 失敗
- Scaffold CLI の出力フォーマットが生成器更新で静かに変わり、既存サービスの再生成時に破壊変更が発生
- `catalog-info.yaml` が Scaffold に含まれず、Backstage 上で新規サービスが発見できないまま運用に入る
- テンプレート変更が 1 人で merge され、全社展開後に致命的な欠陥が見つかる

本原則は、これらの破綻を構造レベルで回避するために設けた 7 軸である。

## 原則 1: 契約は `src/contracts/` が単一真実源（IMP-CODEGEN-POL-001）

**すべての Protobuf / OpenAPI スキーマは `src/contracts/` 配下に一元配置し、他の場所からの import や複製を禁止する。**

ADR-DIR-001 で `src/contracts/` を tier1 配下からルート直下に昇格させた意図は、契約を特定 tier の所有物から外し、4 言語 SDK と tier1 / tier2 / tier3 のすべてが対称的に参照できる場所に据えるためである。本原則は同じ意図をビルド設定とコード生成設定の両面で強制する。

具体的な帰結として、以下を固定する。

- tier1 公開 API: `src/contracts/tier1/v1/*.proto`
- tier1 内部 gRPC: `src/contracts/internal/v1/*.proto`
- 外部連携（BFF 等）: `src/contracts/openapi/*.yaml`
- `.proto` / `.yaml` のいかなる複製・シンボリックリンクも禁止

契約への手修正 PR は CODEOWNERS で Platform/Build + Security の 2 名必須承認とし、`30_CI_CD設計/` の branch protection で強制する。

## 原則 2: 生成は機械化し CI ステージに組込む（IMP-CODEGEN-POL-002）

**`buf generate` / OpenAPI Generator / Scaffold CLI の実行は CI の独立ステージで行い、開発者端末での手動生成物の commit を禁止する。**

手動生成では生成器のバージョン差や環境差（Windows vs Linux の改行コード等）が commit に混入し、PR 差分が膨大化する。CI の生成ステージは `tools/codegen/` 配下のラッパを呼び出す reusable workflow として実装し、生成結果と commit 内容が一致することを検証する。

具体的な CI 段配置は以下とする。詳細は `30_CI_CD設計/` の 7 段ステージ定義に従う。

- fetch 段後: `buf generate` 実行 → diff 検証
- lint 段: `buf lint` / `buf breaking`（原則 3）
- unit-test 段前: 生成物と commit 内容の一致検証（不一致なら PR 拒否）
- Scaffold CLI の golden snapshot 比較（原則 5）

開発者ローカルでも同じラッパを呼べるよう、`tools/codegen/regen.sh` を提供する（DX 観点）。

## 原則 3: Breaking change は `buf breaking` で必須検知（IMP-CODEGEN-POL-003）

**すべての `.proto` 変更 PR は `buf breaking --against '.git#branch=main'` を必須ゲートとし、フィールド削除・型変更・番号再利用を検出した場合は PR を拒否する。**

ADR-TIER1-002 で Protobuf gRPC に統一した最大の利点は後方互換性の機械的検証にある。`buf breaking` を回避すると互換性破壊が気付かれずに 4 言語 SDK を通じて本番に到達し、数週間後にフィールド欠損で顧客実装が壊れる。

例外として意図的な破壊変更が必要な場合は以下の手順を要求する。

- 新バージョンディレクトリ（`v2/`）を追加し、`v1/` は残す
- `v1/` の deprecation は最低 12 か月維持（NFR-C-MNT-003 の API 互換方針準拠）
- ADR で破壊変更の根拠と移行計画を記録

`buf breaking` の `ignore` 指定は原則禁止し、採用する場合は ADR 必須とする。

## 原則 4: 生成物は commit し「DO NOT EDIT」を徹底（IMP-CODEGEN-POL-004）

**生成されたコードはすべて `// Code generated. DO NOT EDIT.` ヘッダを持ち、commit 時点で lint / CI が手修正を検出して拒否する。**

生成物を commit する理由は IMP-BUILD-POL-007 で述べたとおり「pull 直後に build が通る」体験を保つためである。しかし生成物に手修正が入ると、次回の `buf generate` で消える変更が積み重なり、結果的に契約と実装の乖離が発生する。

以下の検出機構を CI に組み込む。

- 生成物の先頭コメントで `DO NOT EDIT` を検出し、diff が発生していれば PR 拒否
- Rust / Go の独自 linter で「生成物ファイル内に手書きコメントを含む」パターンを拒否
- `rust-analyzer` / `gopls` / `tsserver` の LSP 設定で `gen/` 配下を編集時警告

生成物の配置は IMP-BUILD-POL-007 の `gen/` サブディレクトリに従う。

## 原則 5: Scaffold 出力は golden snapshot で後方互換検証（IMP-CODEGEN-POL-005）

**Scaffold CLI の全テンプレートについて、固定入力からの生成結果を `tests/golden/` に保存し、CI で差分検出する。差分が出た PR はテンプレート変更と見なしレビュー必須とする。**

Scaffold CLI の出力は新規サービスの初期骨格になる。ここで静かな破壊変更（例: CI workflow のジョブ名変更）が入ると、既存サービスを再生成したときにリリースパイプラインが壊れる。golden snapshot は「生成結果が意図せず変わっていないこと」の不可視ゲートとして機能する。

snapshot 運用の要件は以下とする。

- 固定入力セットは `tests/golden/fixtures/scaffold/` に配置
- 期待出力は `tests/golden/scaffold/<template>/` 配下
- 差分検出は `tools/codegen/golden-diff.sh` が `diff -r` 相当を実行
- 意図的な変更時は `--update-golden` フラグで更新し、PR レビュアが差分を確認

golden 比較は Scaffold CLI のすべてのテンプレートバージョンについて実行する。

## 原則 6: Scaffold 出力は必ず `catalog-info.yaml` を含む（IMP-CODEGEN-POL-006）

**Scaffold CLI が生成する新規サービスは `catalog-info.yaml` を必ず含み、Backstage カタログへ自動登録可能な状態で出力される。**

Backstage（ADR-BS-001）によるサービス発見性は、運用フェーズで「どのサービスが存在し誰が所有しているか」を 2 名運用チームが把握するための唯一の UI となる。Scaffold が `catalog-info.yaml` を出さないと、作った人が手動で書く必要があり、記入漏れ・記入ミスがカタログ品質を侵食する。

`catalog-info.yaml` の必須フィールドは以下とする。

- `metadata.name` : tier-service 命名規約に準拠
- `metadata.owner` : CODEOWNERS チームと一致
- `spec.type` : service / library / website
- `spec.lifecycle` : experimental / production
- `spec.system` : どの業務サブシステムに属するか
- `spec.dependsOn` : 依存する他サービスの entity ref

Scaffold 出力の `catalog-info.yaml` は golden snapshot（原則 5）の対象にも含める。

## 原則 7: テンプレート変更は SRE + Security の二重承認（IMP-CODEGEN-POL-007）

**Scaffold CLI のテンプレート本体 / Protobuf 生成器設定 / OpenAPI 生成器設定の変更 PR は、SRE（B）と Security（D）の 2 名承認を branch protection で必須化する。**

テンプレートは全社で再利用される資産であり、1 件の変更が 10 年保守の間に生成される数百のサービスに波及する。ここに単一承認を許すと、セキュリティ的に問題のある初期設定（例: TLS 検証無効のクライアント生成）が全社に拡散するリスクを負う。

branch protection の要件は以下とする。

- 対象 path: `tools/codegen/**` / `src/contracts/**` / `tests/golden/fixtures/scaffold/**`
- 必須レビュアチーム: `@k1s0/sre` および `@k1s0/security`
- quality gate: golden snapshot 差分 / `buf breaking` / SPDX ライセンスチェック（`40_依存管理設計/`）

PR 作成者は変更の動機と影響範囲を PR 本文に記載する。

## 対応 IMP-CODEGEN ID

本ファイルで採番する原則レベル ID は以下とする。

- `IMP-CODEGEN-POL-001` : 契約は `src/contracts/` 単一真実源
- `IMP-CODEGEN-POL-002` : 生成の機械化と CI ステージ組込
- `IMP-CODEGEN-POL-003` : `buf breaking` による互換性検知
- `IMP-CODEGEN-POL-004` : 生成物 commit と「DO NOT EDIT」徹底
- `IMP-CODEGEN-POL-005` : Scaffold 出力の golden snapshot 検証
- `IMP-CODEGEN-POL-006` : `catalog-info.yaml` 必須生成
- `IMP-CODEGEN-POL-007` : テンプレート変更の二重承認

## 対応 ADR / DS-SW-COMP / NFR

- ADR-DIR-001（contracts 昇格）/ ADR-TIER1-002（Protobuf gRPC）/ ADR-TIER1-003（内部言語不可視）/ ADR-BS-001（Backstage）
- DS-SW-COMP-122（SDK 生成）/ 130（契約配置）/ 132（platform / scaffold）
- NFR-H-INT-001（署名付きアーティファクト）/ NFR-C-MGMT-003（SBOM 100%）/ NFR-C-MNT-003（API 互換方針）
