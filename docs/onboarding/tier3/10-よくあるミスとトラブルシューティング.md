# service tier よくあるミスとトラブルシューティング

開発中に遭遇しやすい問題を「症状 → 原因 → 解決方法」の形式でまとめた。エラーが出たらまずここを確認してほしい。

> **開発環境のトラブル（CRLF・Docker・WSL2・ポート競合等）は以下を参照:**
> [`docs/infrastructure/devenv/troubleshooting.md`](../../infrastructure/devenv/troubleshooting.md)

---

## ビルド関連

### cargo build が失敗する

**症状**: `cargo build` 実行時にコンパイルエラーが出る。

**原因 1**: system tier ライブラリが見つからない。

```
error[E0433]: failed to resolve: use of undeclared crate or module `k1s0_auth`
```

**解決方法**: sparse-checkout で system ライブラリがチェックアウトされているか確認する。

```bash
$ git sparse-checkout list
# regions/system/library/ が含まれているか確認

# 含まれていなければ追加
$ git sparse-checkout add regions/system/library
```

**原因 2**: Cargo.toml の path 指定が間違っている。

**解決方法**: 相対パスが正しいか確認する。service tier のサーバーから system ライブラリへのパスは通常 `../../../../system/library/rust/{ライブラリ名}`。

---

**原因 3**: Rust ツールチェインのバージョンが古い。

```
error[E0658]: use of unstable library feature
```

**解決方法**:

```bash
$ rustup update
$ rustc --version  # プロジェクトで指定されたバージョン以上か確認
```

---

### npm / pnpm install が失敗する

**症状**: `pnpm install` でパッケージのインストールに失敗する。

**原因 1**: Node.js のバージョンが合っていない。

**解決方法**:

```bash
$ node --version
# プロジェクトの .node-version や package.json の engines を確認
$ nvm use  # .nvmrc がある場合
```

**原因 2**: pnpm のバージョンが合っていない。

**解決方法**:

```bash
$ corepack enable
$ corepack prepare pnpm@latest --activate
```

**原因 3**: ローカルパッケージ（system-client 等）のパスが見つからない。

**解決方法**: sparse-checkout で依存パッケージがチェックアウトされているか確認する。

---

### 型エラー（TypeScript）

**症状**: `tsc` や `pnpm build` で型エラーが出る。

**原因**: API のレスポンス型と画面コンポーネントの型が不一致。

**解決方法**:

1. BFF のスキーマ（GraphQL / REST）を確認し、正しい型を定義する
2. `types/` ディレクトリの型定義を更新する
3. `strict: true` が有効になっているか `tsconfig.json` を確認する

---

### buf lint が失敗する

**症状**: `buf lint` で proto ファイルのエラーが出る。

```
order.proto:15:3: Field name "orderId" should be lower_snake_case
```

**原因**: proto の命名規約に違反している。

**解決方法**: Protocol Buffers の命名規約に従う。

| 対象 | 規約 | 例 |
| --- | --- | --- |
| フィールド名 | lower_snake_case | `order_id` |
| メッセージ名 | PascalCase | `GetOrderRequest` |
| RPC 名 | PascalCase | `GetOrder` |
| enum 値 | UPPER_SNAKE_CASE | `ORDER_STATUS_PENDING` |

## DB 関連

### PostgreSQL 接続エラー

**症状**: サーバー起動時やテスト実行時に DB 接続エラー。

```
error connecting to server: Connection refused (os error 111)
```

**原因 1**: PostgreSQL が起動していない。

**解決方法**:

```bash
# docker-compose で起動
$ docker-compose up -d postgres

# 起動確認
$ docker-compose ps
```

**原因 2**: 接続情報（URL、ポート、パスワード）が間違っている。

**解決方法**: `config.yaml` の `database.url` と、実際の PostgreSQL の設定を照合する。

```yaml
# config.yaml
database:
  url: "postgres://user:pass@localhost:5432/task_db"
```

```bash
# PostgreSQL に手動接続して確認
$ psql "postgres://user:pass@localhost:5432/task_db"
```

---

### マイグレーション失敗

**症状**: `sqlx migrate run` がエラーで止まる。

**原因 1**: 既に適用済みのマイグレーションを変更した。

**解決方法**: 既に適用済みのマイグレーションファイルは変更してはいけない。修正が必要な場合は新しいマイグレーションファイルを追加する。

```bash
# 状態確認
$ sqlx migrate info

# 開発環境のみ: DBをリセットして最初からやり直す
$ sqlx database drop
$ sqlx database create
$ sqlx migrate run
```

**原因 2**: SQL の構文エラー。

**解決方法**: マイグレーションファイルの SQL を確認する。PostgreSQL の `psql` で直接実行してエラーメッセージを確認するのが早い。

---

### sqlx コンパイル時検証エラー

**症状**: `cargo build` 時に sqlx の検証エラーが出る。

```
error: error returned from database: relation "orders" does not exist
```

**原因**: sqlx の `query!` マクロはコンパイル時に DB に接続して SQL を検証するが、DB にテーブルがない。

**解決方法**:

```bash
# 1. DBが起動しているか確認
$ docker-compose up -d postgres

# 2. マイグレーションを実行
$ sqlx migrate run

# 3. DATABASE_URL 環境変数が設定されているか確認
$ echo $DATABASE_URL

# オフラインモードを使う場合（CI等、DB接続できない環境向け）
$ cargo sqlx prepare  # sqlx-data.json を生成
```

## 認証関連

### JWT 検証エラー

**症状**: API リクエスト時に 401 Unauthorized が返る。

```
{"error": "invalid_token", "message": "JWT signature verification failed"}
```

**原因 1**: Keycloak が起動していない、または JWKS エンドポイントに到達できない。

**解決方法**:

```bash
# Keycloakの起動確認
$ docker-compose up -d keycloak

# JWKSエンドポイントの確認
$ curl http://localhost:8100/.well-known/jwks.json
```

**原因 2**: トークンの有効期限が切れている。

**解決方法**: 再ログインしてトークンを更新する。開発環境でトークンの有効期限が短すぎる場合は Keycloak の設定を確認する。

---

### ログインできない

**症状**: ログイン画面でエラーが表示される、またはリダイレクトループになる。

**原因 1**: Keycloak のクライアント設定（Redirect URI 等）が間違っている。

**解決方法**: Keycloak の管理画面（`http://localhost:8100/admin`）でクライアント設定を確認する。Redirect URI にローカル開発の URL（`http://localhost:5173/*`）が登録されているか確認する。

**原因 2**: CORS 設定が間違っている。

**解決方法**: BFF / Keycloak の CORS 設定にクライアントのオリジン（`http://localhost:5173`）が許可されているか確認する。

---

### RBAC 権限不足

**症状**: ログインはできるが、特定の API で 403 Forbidden が返る。

```
{"error": "forbidden", "message": "insufficient permissions"}
```

**原因**: ユーザーに必要なロール/権限が割り当てられていない。

**解決方法**:

1. Keycloak でユーザーのロール割り当てを確認する
2. API のハンドラーでどの権限を要求しているか確認する
3. 開発環境では管理者ロールを持つテストユーザーを使う

## gRPC / API 関連

### proto が未生成

**症状**: Rust コード内で proto から生成される型が見つからない。

```
error[E0433]: failed to resolve: could not find `UpdateOrderStatusRequest`
```

**原因**: `.proto` ファイルを変更した後にコード生成を実行していない。

**解決方法**:

```bash
# buf を使っている場合
$ buf generate

# tonic-build を使っている場合（cargo build で自動生成）
$ cargo build
# ※ build.rs に tonic_build::compile_protos の設定があるか確認
```

---

### tonic 接続エラー

**症状**: gRPC クライアントから接続できない。

```
transport error: error trying to connect: tcp connect error: Connection refused
```

**原因**: 接続先の gRPC サーバーが起動していない、またはポートが間違っている。

**解決方法**:

```bash
# 接続先サーバーが起動しているか確認
$ docker-compose ps

# config.yamlの接続先を確認
$ cat config.yaml | grep grpc

# ポートが開いているか確認
$ curl -v telnet://localhost:9090
```

---

### GraphQL スキーマエラー

**症状**: BFF 起動時に GraphQL スキーマのエラーが出る。

**原因**: `schema.graphqls` の構文エラー、またはリゾルバーの実装がスキーマと不一致。

**解決方法**:

```bash
# Go BFF: gqlgenで再生成
$ cd server/go/bff
$ go run github.com/99designs/gqlgen generate

# Rust BFF: コンパイルエラーのメッセージを確認
$ cd server/rust/bff
$ cargo build
```

## クライアント関連

### useQuery が undefined を返す

**症状**: `useQuery` の `data` が `undefined` のまま。

**原因 1**: API のエンドポイント URL が間違っている。

**解決方法**: ブラウザの開発者ツール → Network タブで実際のリクエスト URL とレスポンスを確認する。

**原因 2**: `QueryClientProvider` がコンポーネントツリーの上位に無い。

**解決方法**: `main.tsx` で `QueryClientProvider` がアプリ全体をラップしているか確認する。

**原因 3**: API がエラーを返しているが、エラー処理をしていない。

**解決方法**: `useQuery` の `isError` と `error` を確認する。

```tsx
const { data, isLoading, isError, error } = useOrders();
if (isError) {
  console.error("API エラー:", error);
}
```

---

### 状態が更新されない

**症状**: データを変更したのに画面に反映されない。

**原因 1**: `useMutation` の `onSuccess` でキャッシュを無効化していない。

**解決方法**:

```typescript
const mutation = useMutation({
  mutationFn: (data) => api.post("/api/v1/orders", data),
  onSuccess: () => {
    // これを忘れると一覧が更新されない
    queryClient.invalidateQueries({ queryKey: orderKeys.lists() });
  },
});
```

**原因 2**: Zustand ストアの更新が正しくない。

**解決方法**: Zustand の `set` 関数でイミュータブルに更新しているか確認する。

---

### CORS エラー

**症状**: ブラウザのコンソールに CORS エラーが表示される。

```
Access to fetch at 'http://localhost:8080/api/v1/orders' from origin
'http://localhost:5173' has been blocked by CORS policy
```

**原因**: BFF / サーバーの CORS 設定にクライアントのオリジンが許可されていない。

**解決方法**: サーバー側の CORS 設定を確認する。

```rust
// axumの場合
use tower_http::cors::{CorsLayer, Any};

let cors = CorsLayer::new()
    .allow_origin("http://localhost:5173".parse::<HeaderValue>().unwrap())
    .allow_methods(Any)
    .allow_headers(Any);

app.layer(cors);
```

Vite の proxy 設定で CORS を回避する方法もある。

```typescript
// vite.config.ts
export default defineConfig({
  server: {
    proxy: {
      "/api": "http://localhost:8080",
    },
  },
});
```

---

### ルーティングが動かない

**症状**: URL を直接入力してもページが表示されない、または 404 になる。

**原因 1**: TanStack Router のルートファイルが正しい場所にない。

**解決方法**: `src/routes/` のファイル名・ディレクトリ構造が URL パスと一致しているか確認する。

```
src/routes/orders/index.tsx → /orders
src/routes/orders/$orderId.tsx → /orders/:orderId
```

**原因 2**: 開発サーバー（Vite）の SPA フォールバック設定が無い。

**解決方法**: `vite.config.ts` で SPA のフォールバック設定を確認する。

---

### Zod バリデーションエラー

**症状**: フォーム送信時にバリデーションエラーが表示される、または API レスポンスの parse に失敗する。

**原因**: Zod スキーマの定義とデータの形が合っていない。

**解決方法**:

```typescript
// デバッグ: どの項目でエラーが出ているか確認
const result = schema.safeParse(data);
if (!result.success) {
  console.error(result.error.format());
  // どのフィールドがどう間違っているか詳細が出力される
}
```

## テスト関連

### テストが通らない

**症状**: `cargo test` / `pnpm test` / `flutter test` が失敗する。

**原因 1**: テスト用の環境変数が設定されていない。

**解決方法**:

```bash
# Rust: テスト用のDATABASE_URL
$ export DATABASE_URL="postgres://test:test@localhost:5432/test_db"

# React: .env.test ファイルを確認
$ cat .env.test
```

**原因 2**: テスト用の DB が起動していない（統合テストの場合）。

**解決方法**:

```bash
$ docker-compose up -d postgres
$ sqlx migrate run
```

---

### モックの書き方がわからない

**Rust（mockall）**:

```rust
// 1. リポジトリトレイトに #[automock] を付ける
#[cfg_attr(test, automock)]
pub trait TaskRepository {
    async fn find_by_id(&self, id: Uuid) -> Result<Task, RepositoryError>;
}

// 2. テストでMockTaskRepositoryを使う
let mut mock = MockTaskRepository::new();
mock.expect_find_by_id()
    .with(eq(task_id))
    .returning(|_| Ok(task));
```

**Go（gomock）**:

```go
// 1. インターフェースからモックを生成
//go:generate mockgen -destination=mock/mock_task_client.go -package=mock . TaskServiceClient

// 2. テストで使う
ctrl := gomock.NewController(t)
mock := mock.NewMockTaskServiceClient(ctrl)
mock.EXPECT().GetTask(gomock.Any(), req).Return(resp, nil)
```

**React（MSW）**:

```typescript
// msw でAPIモックを設定
const server = setupServer(
  http.get("/api/v1/orders", () => {
    return HttpResponse.json([{ id: "1", status: "pending" }]);
  }),
);
beforeAll(() => server.listen());
afterAll(() => server.close());
```

**Flutter（mocktail）**:

```dart
class MockOrderRepository extends Mock implements OrderRepository {}

final mock = MockOrderRepository();
when(() => mock.getOrder("1")).thenAnswer((_) async => order);
```

---

### MSW の設定がうまくいかない

**症状**: MSW が API リクエストをインターセプトしない。

**原因**: `setupServer` の設定が `beforeAll` で行われていない、またはハンドラーの URL パターンが合っていない。

**解決方法**:

```typescript
// テストファイルの先頭付近に記述
const server = setupServer(
  // URLパターンを正確に書く（パスパラメータは :param 形式）
  http.get("/api/v1/orders/:id", ({ params }) => {
    return HttpResponse.json({ id: params.id, status: "pending" });
  }),
);

// 必ずこの3つをセットで書く
beforeAll(() => server.listen());
afterEach(() => server.resetHandlers());
afterAll(() => server.close());
```

## Docker / ローカル環境

### docker-compose が失敗する

**症状**: `docker-compose up` でエラーが出る。

**原因 1**: Docker デーモンが起動していない。

**解決方法**:

```bash
# Docker Desktopを起動する（Windows / macOS）
# または
$ sudo systemctl start docker  # Linux
```

**原因 2**: ポートが既に使われている。

```
Error: Bind for 0.0.0.0:5432 failed: port is already allocated
```

**解決方法**:

```bash
# どのプロセスがポートを使っているか確認
$ lsof -i :5432  # macOS / Linux
$ netstat -ano | findstr :5432  # Windows

# 競合するプロセスを停止するか、docker-compose.yamlのポートマッピングを変更
```

---

### ポート競合

**症状**: サーバー起動時に「Address already in use」エラー。

**解決方法**:

```bash
# 使用中のポートを確認
$ lsof -i :8080

# 該当プロセスを停止
$ kill <PID>

# または config.yaml のポートを変更して起動
```

---

### ホットリロードが効かない

**症状**: コードを変更してもサーバー/クライアントが自動的に再起動/リビルドされない。

**Rust サーバーの場合**:

```bash
# cargo-watch を使う
$ cargo install cargo-watch
$ cargo watch -x run
```

**React クライアントの場合**: Vite の開発サーバーは自動でホットリロードされるはず。効かない場合:

```bash
# node_modules を再インストール
$ rm -rf node_modules
$ pnpm install

# Vite のキャッシュをクリア
$ rm -rf node_modules/.vite
$ pnpm dev
```

### 間違ったブランチにコミットした

**症状**: main ブランチに直接コミットしてしまった、または別のブランチにコミットしてしまった。

**解決方法（まだ push していない場合）**:

```bash
# 1. コミットを取り消す（変更内容は残す）
$ git reset --soft HEAD~1

# 2. 正しいブランチに切り替える
$ git checkout -b feat/order/correct-branch

# 3. 改めてコミット
$ git add .
$ git commit -m "feat(order): 正しいメッセージ"
```

**解決方法（既に push してしまった場合）**:

```bash
# main に push してしまった場合は、チームリーダーに相談する
# 一人で --force push しないこと
```
