## k1s0 CLI のインストール

```bash
# リポジトリをクローン後、リポジトリルートで実行
cargo install --path CLI/crates/k1s0-cli

# インストール確認
k1s0 --version
```

devcontainer 利用者はコンテナ起動時に自動インストールされるため不要。

## sparse-checkout によるサービス単位のチェックアウト

モノリポ全体をクローンする必要はない。k1s0 CLI の sparse-checkout 機能を使い、自分が担当するサービスと依存先のみをチェックアウトする。

```bash
# k1s0 CLI でプロジェクトを初期化
$ k1s0 init
# または: k1s0 → よく使う操作 > プロジェクト初期化 を選択

# sparse-checkout で必要なパスのみ取得される
# 例: task サービス担当の場合
#   regions/service/task/       ← 自サービス
#   regions/system/library/     ← system ライブラリ
#   regions/system/client/      ← system client SDK
#   regions/business/{領域}/    ← 所属領域の business tier
```

sparse-checkout の設定は k1s0 CLI が自動的に管理する。担当サービスを選択するだけで、依存する system / business tier のコードも取得される。

## docker-compose によるローカル環境

ローカル開発では docker-compose を使い、依存する system tier サービス群をコンテナとして起動する。

### 起動手順

```bash
# 依存サービス群を起動（system tier + 必要な business tier）
$ docker compose -f docker-compose.yaml -f docker-compose.dev.yaml --profile infra --profile system up -d

# 起動されるサービス例:
#   - auth-server       (認証)
#   - config-server     (設定管理)
#   - session-server    (セッション管理)
#   - PostgreSQL        (DB)
#   - Redis             (キャッシュ)
#   - Kafka + Zookeeper (メッセージング)
#   - Jaeger            (トレーシング可視化)
```

### サービスの起動

```bash
# Rust サーバー
$ cargo watch -x run

# Go BFF
$ air

# React クライアント
$ pnpm dev

# Flutter クライアント
$ flutter run
```

## k1s0 CLI によるサービスひな形生成

新しいサービスを作成する際は k1s0 CLI のひな形生成機能を使う。

```bash
$ k1s0
# メインメニュー → ひな形生成 を選択
# 対話式プロンプトで以下を指定:
#   - Tier: service
#   - サービス名: 例) task
#   - 含めるコンポーネント: server(rust), server(go/bff), client(react), client(flutter), database
```

生成されるひな形には以下が含まれる:

- サーバーの基本構造（axum ルーター、tonic gRPC サービス、sqlx 接続設定）
- BFF の基本構造（GraphQL スキーマ、リゾルバー）
- クライアントの基本構造（ルーティング、状態管理、API クライアント設定）
- DB マイグレーションの初期ファイル
- CI/CD ワークフロー（GitHub Actions）
- docker-compose.dev.yaml
- config.yaml テンプレート

## 設定ファイル（config.yaml）の扱い

各サービスは `config.yaml` で設定を管理する。system tier の config ライブラリがYAML の読み込み、環境別オーバーライド、バリデーションを提供する。

### 基本構造

```yaml
# config.yaml
server:
  host: "0.0.0.0"
  port: 8080
  grpc_port: 9090

database:
  url: "postgres://user:pass@localhost:5432/task_db"
  max_connections: 10

auth:
  jwks_url: "http://auth-server:8100/.well-known/jwks.json"

kafka:
  brokers:
    - "localhost:9092"
  consumer_group: "task-service"

telemetry:
  service_name: "task-service"
  otlp_endpoint: "http://otel-collector:4317"
```

### 環境別オーバーライド

```yaml
# config.dev.yaml — ローカル開発用のオーバーライド
database:
  url: "postgres://dev:dev@localhost:5432/task_dev"

# config.staging.yaml, config.prod.yaml も同様
```

環境変数 `K1S0_ENV` により使用する設定ファイルが切り替わる。

```bash
K1S0_ENV=dev cargo run   # config.yaml + config.dev.yaml
K1S0_ENV=prod cargo run  # config.yaml + config.prod.yaml
```
