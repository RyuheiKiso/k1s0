# service tier テストと CI/CD

## テスト戦略

service tier では以下の 3 層でテストを実施する。カバレッジ目標は **80% 以上**。

| レベル | 対象 | 実行速度 | 外部依存 |
| --- | --- | --- | --- |
| 単体テスト | ドメインロジック、ユースケース、個々のコンポーネント | 高速 | モック化 |
| 統合テスト | API エンドポイント、DB アクセス、外部サービス連携 | 中速 | テストコンテナ使用 |
| E2E テスト | ユーザーシナリオ全体（クライアント → BFF → サーバー → DB） | 低速 | 実環境に近い構成 |

### テストピラミッド

```
        /  E2E  \         ← 少数・重要シナリオのみ
       /  統合   \        ← API・DB 連携の検証
      /  単体テスト \     ← 大量・高速・ロジック中心
```

## 言語別テストツール

### Go（BFF）

| ツール | 用途 |
| --- | --- |
| testing | 標準テストパッケージ |
| testify | アサーション・テストスイート |
| gomock | インターフェースモック生成 |

```go
func TestGetOrder(t *testing.T) {
    ctrl := gomock.NewController(t)
    defer ctrl.Finish()

    mockClient := mock.NewMockOrderServiceClient(ctrl)
    mockClient.EXPECT().
        GetOrder(gomock.Any(), &pb.GetOrderRequest{Id: "order-1"}).
        Return(&pb.GetOrderResponse{
            Id:     "order-1",
            Status: "pending",
            Total:  1000,
        }, nil)

    resolver := &queryResolver{orderClient: mockClient}
    order, err := resolver.Order(context.Background(), "order-1")

    assert.NoError(t, err)
    assert.Equal(t, "order-1", order.ID)
    assert.Equal(t, "pending", order.Status)
}
```

```bash
# テスト実行
$ go test ./...

# カバレッジ付き
$ go test -coverprofile=coverage.out ./...
$ go tool cover -html=coverage.out
```

### Rust（業務サーバー・BFF）

| ツール | 用途 |
| --- | --- |
| #[cfg(test)] | 標準テストモジュール |
| mockall | トレイトモック生成 |
| testcontainers | テスト用 DB コンテナ |
| axum::test | HTTP ハンドラーテスト |

```rust
use mockall::predicate::*;

#[tokio::test]
async fn test_get_order() {
    let mut mock_repo = MockOrderRepository::new();
    mock_repo
        .expect_find_by_id()
        .with(eq(order_id))
        .returning(|_| Ok(Order {
            id: order_id,
            status: OrderStatus::Pending,
            total: Decimal::new(1000, 0),
            ..Default::default()
        }));

    let usecase = GetOrderUsecase::new(Arc::new(mock_repo));
    let result = usecase.execute(order_id).await.unwrap();

    assert_eq!(result.status, OrderStatus::Pending);
}
```

#### 統合テスト（testcontainers）

```rust
use testcontainers::runners::AsyncRunner;
use testcontainers_modules::postgres::Postgres;

#[tokio::test]
#[ignore = "requires Docker (testcontainers)"]
async fn test_order_repository_integration() {
    let container = Postgres::default().start().await.unwrap();
    let pool = PgPool::connect(&container.connection_string()).await.unwrap();
    sqlx::migrate!().run(&pool).await.unwrap();

    let repo = OrderRepository::new(pool);
    let order = repo.create(CreateOrderInput { /* ... */ }).await.unwrap();
    let found = repo.find_by_id(order.id).await.unwrap();

    assert_eq!(found.id, order.id);
}
```

```bash
# 単体テスト実行（統合テストは自動スキップされる）
$ cargo test

# 統合テスト（Docker 必須）のみ実行
$ cargo test --all -- --ignored

# カバレッジ（cargo-llvm-cov）
$ cargo llvm-cov --html
```

> **`#[ignore]` 戦略**: testcontainers を使う統合テストには `#[ignore = "requires Docker (testcontainers)"]` を付与する。
> 通常の `cargo test` ではスキップされ、CI の `integration-test.yaml` で `--ignored` フラグにより明示実行される。

### TypeScript / React

| ツール | 用途 |
| --- | --- |
| Vitest | テストランナー・アサーション |
| Testing Library | コンポーネントテスト（DOM ベース） |
| MSW (Mock Service Worker) | API モック |

```typescript
import { renderHook, waitFor } from "@testing-library/react";
import { QueryClient, QueryClientProvider } from "@tanstack/react-query";
import { http, HttpResponse } from "msw";
import { setupServer } from "msw/node";
import { useOrder } from "./use-order";

const server = setupServer(
  http.get("/api/v1/orders/:id", ({ params }) => {
    return HttpResponse.json({
      id: params.id,
      status: "pending",
      total: 1000,
    });
  })
);

beforeAll(() => server.listen());
afterAll(() => server.close());

test("useOrder はオーダー情報を取得する", async () => {
  const queryClient = new QueryClient();
  const wrapper = ({ children }: { children: React.ReactNode }) => (
    <QueryClientProvider client={queryClient}>{children}</QueryClientProvider>
  );

  const { result } = renderHook(() => useOrder("order-1"), { wrapper });

  await waitFor(() => expect(result.current.isSuccess).toBe(true));
  expect(result.current.data?.status).toBe("pending");
});
```

### Dart / Flutter

| ツール | 用途 |
| --- | --- |
| flutter_test | 標準テストパッケージ |
| mocktail | モック（コード生成不要） |
| golden_toolkit | ゴールデンテスト（任意） |

```dart
import 'package:flutter_test/flutter_test.dart';
import 'package:mocktail/mocktail.dart';

class MockOrderRepository extends Mock implements OrderRepository {}

void main() {
  late MockOrderRepository mockRepo;
  late OrderNotifier notifier;

  setUp(() {
    mockRepo = MockOrderRepository();
    notifier = OrderNotifier(mockRepo);
  });

  test('createOrder は正常に注文を作成する', () async {
    final input = CreateOrderInput(
      customerId: 'customer-1',
      items: [OrderItem(productId: 'product-1', quantity: 2)],
    );
    final expectedOrder = Order(
      id: 'order-1',
      customerId: 'customer-1',
      status: OrderStatus.pending,
      total: 2000,
      createdAt: DateTime.now(),
    );

    when(() => mockRepo.createOrder(input))
        .thenAnswer((_) async => expectedOrder);

    await notifier.createOrder(input);

    expect(notifier.state.value, expectedOrder);
    verify(() => mockRepo.createOrder(input)).called(1);
  });
}
```

```bash
# テスト実行
$ flutter test

# カバレッジ
$ flutter test --coverage
$ genhtml coverage/lcov.info -o coverage/html
```

## CI/CD パイプライン（GitHub Actions）

### パイプライン全体像

```
PR 作成/更新 → CI（lint → test → build）→ レビュー → マージ
                                                         ↓
main マージ → CD（image build → push → deploy dev → deploy staging）
                                                         ↓
                                              手動承認 → deploy prod
```

### CI ワークフロー

PR 作成・更新時に自動実行される。

```yaml
# .github/workflows/service-{サービス名}-ci.yaml
name: "service-order CI"
on:
  pull_request:
    paths:
      - "regions/service/order/**"

jobs:
  lint:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
        with:
          sparse-checkout: |
            regions/service/order
            regions/system/library
      - name: Rust lint
        run: cargo clippy --all-targets -- -D warnings
        working-directory: regions/service/order/server/rust
      - name: TypeScript lint
        run: pnpm lint
        working-directory: regions/service/order/client/react

  test:
    runs-on: ubuntu-latest
    services:
      postgres:
        image: postgres:17
        env:
          POSTGRES_DB: test_db
          POSTGRES_USER: test
          POSTGRES_PASSWORD: test
    steps:
      - uses: actions/checkout@v4
      - name: Rust test
        run: cargo test
        working-directory: regions/service/order/server/rust
      - name: Go test
        run: go test ./...
        working-directory: regions/service/order/server/go/bff
      - name: React test
        run: pnpm test --coverage
        working-directory: regions/service/order/client/react

  build:
    needs: [lint, test]
    runs-on: ubuntu-latest
    steps:
      - name: Build Docker image
        run: docker build -t order-server .
        working-directory: regions/service/order/server/rust
```

### CD ワークフロー

main ブランチへのマージ時に自動実行される。

| 環境 | デプロイ方式 | Namespace |
| --- | --- | --- |
| dev | 自動 | k1s0-service |
| staging | 自動 | k1s0-service |
| prod | 手動承認後 | k1s0-service |

### PR 作成からデプロイまでのフロー

1. **ブランチ作成**: `feature/{サービス名}/{機能名}` の命名規則
2. **実装・テスト**: ローカルでテストを通す
3. **PR 作成**: テンプレートに従い変更内容を記載
4. **CI 実行**: lint → test → build が自動実行
5. **コードレビュー**: 1 名以上の承認が必要
6. **マージ**: main ブランチへマージ
7. **CD 実行**: Docker イメージビルド → dev/staging に自動デプロイ
8. **本番デプロイ**: staging での動作確認後、手動承認で prod にデプロイ
