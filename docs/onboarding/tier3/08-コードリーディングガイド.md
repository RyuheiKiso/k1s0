# service tier コードリーディングガイド

既存コードの読み方を、「どこから読み始めるか」「何を意識して読むか」の観点で説明する。新しいサービスに参加したときや、既存機能を理解したいときに活用してほしい。

---

## サーバーサイド（Rust 業務サーバー）

### 読む順番

以下の順番で読むと、全体像から詳細へスムーズに理解が進む。

```
1. config.yaml        → サーバーの設定（何に接続しているか全体像がわかる）
2. main.rs            → サーバーの起動処理（何が初期化されるか）
3. proto/*.proto       → gRPC の API 定義（どんな機能があるか）
4. domain/            → ビジネスルール（何を守っているか）
5. application/       → ユースケース（どんな業務フローがあるか）
6. presentation/      → ハンドラー（APIをどう受け付けているか）
7. infrastructure/    → DB・外部連携の実装（具体的にどう保存・通信しているか）
```

### Step 1: config.yaml を読む

**場所**: `regions/service/{サービス名}/server/rust/config.yaml`

まずここを読むと、このサーバーが「何に接続しているか」の全体像がわかる。

```yaml
# こんな情報が書いてある
server:
  http_port: 8080          # HTTPでリッスンするポート
  grpc_port: 9090          # gRPCでリッスンするポート

database:
  url: "postgres://..."    # どのDBに接続しているか

upstream:
  auth_grpc: "..."         # 認証サーバーのアドレス
  business_api: "..."      # business tierのアドレス

kafka:
  topics:
    order_events: "..."    # どんなイベントを扱っているか
```

**読むときのポイント**: 「このサーバーは何と通信しているのか」をメモしておくと、後のコード読解が楽になる。

### Step 2: main.rs を読む

**場所**: `regions/service/{サービス名}/server/rust/src/main.rs`

サーバーの起動処理が書かれている。以下の流れが見えるはず。

```rust
// 典型的なmain.rsの流れ
#[tokio::main]
async fn main() -> Result<()> {
    // 1. 設定読み込み
    let config = Config::load("config.yaml")?;

    // 2. テレメトリ初期化（ログ・トレース）
    init_telemetry(config.telemetry.clone())?;

    // 3. DB接続プール作成
    let pool = PgPool::connect(&config.database.url).await?;

    // 4. 依存オブジェクトの組み立て（DI）
    let repo = OrderRepository::new(pool.clone());
    let usecase = OrderUsecase::new(Arc::new(repo));

    // 5. ルーター作成・サーバー起動
    let app = create_router(AppState { usecase, ... });
    // ...
}
```

**読むときのポイント**: 「どんな依存オブジェクトが作られて、どう組み立てられているか」を追う。ここを理解すると、各層のつながりがわかる。

### Step 3: proto 定義を読む

**場所**: `regions/service/{サービス名}/server/rust/proto/*.proto`

このサービスが「外部に対してどんな機能を提供しているか」がわかる。

```protobuf
// 例: order.proto
service OrderService {
  rpc GetOrder(GetOrderRequest) returns (GetOrderResponse);       // 注文取得
  rpc CreateOrder(CreateOrderRequest) returns (CreateOrderResponse); // 注文作成
  rpc ListOrders(ListOrdersRequest) returns (ListOrdersResponse);   // 注文一覧
}
```

**読むときのポイント**: `service` ブロックの RPC 定義を見ると「このサービスでできること一覧」がわかる。`message` ブロックでリクエスト/レスポンスの構造を確認する。

### Step 4: domain 層を読む

**場所**: `regions/service/{サービス名}/server/rust/src/domain/`

```
domain/
├── model/           # エンティティ・値オブジェクト
│   ├── order.rs     #   例: Order, OrderStatus, OrderItem
│   └── mod.rs
├── repository/      # リポジトリトレイト（インターフェース）
│   └── order.rs     #   例: OrderRepository trait
└── service/         # ドメインサービス（複数エンティティにまたがるロジック）
```

**読むときのポイント**:

- `model/` のエンティティを読めば「このサービスが扱うデータの構造」がわかる
- エンティティのメソッドを読めば「どんなビジネスルールがあるか」がわかる
- `repository/` のトレイト定義を読めば「どんな DB 操作が必要か」がわかる

### Step 5: application 層を読む

**場所**: `regions/service/{サービス名}/server/rust/src/application/`

```
application/
├── usecase/
│   ├── create_order.rs     # 注文作成のフロー
│   ├── get_order.rs        # 注文取得のフロー
│   └── update_status.rs    # ステータス更新のフロー
└── dto/
    ├── request.rs          # 入力DTO
    └── response.rs         # 出力DTO
```

**読むときのポイント**: 各ユースケースの `execute` メソッドが業務フローの本体。「入力を受け取り → ドメインオブジェクトを操作し → 結果を返す」という流れを追う。

### Step 6: presentation 層を読む

**場所**: `regions/service/{サービス名}/server/rust/src/presentation/`

```
presentation/
├── http/            # axum ハンドラー（REST API）
│   └── order.rs
└── grpc/            # tonic サービス実装（gRPC API）
    └── order.rs
```

**読むときのポイント**: 「外部からのリクエストを受け取り → DTO に変換 → ユースケースを呼び出し → レスポンスを返す」という流れ。エラーハンドリングの仕方にも注目する。

### Step 7: infrastructure 層を読む

**場所**: `regions/service/{サービス名}/server/rust/src/infrastructure/`

```
infrastructure/
├── repository/       # リポジトリ実装（sqlx）
│   └── order.rs      #   実際のSQL文が書かれている
├── grpc_client/      # 外部gRPCクライアント
└── kafka/            # Kafkaプロデューサー/コンシューマー
```

**読むときのポイント**: `repository/` の SQL 文を読めば「DB にどんなデータがどう保存されているか」がわかる。`sqlx::query_as!` マクロの中の SQL に注目する。

## BFF（Go: gqlgen）

### 読む順番

```
1. schema.graphqls       → GraphQLスキーマ（クライアントにどんなAPIを提供しているか）
2. gqlgen.yml            → コード生成設定
3. schema.resolvers.go   → リゾルバー実装（各APIの処理内容）
4. internal/client/      → 業務サーバーへのgRPCクライアント
5. internal/middleware/   → 認証・トレース等のミドルウェア
```

### Step 1: GraphQL スキーマを読む

**場所**: `regions/service/{サービス名}/server/go/bff/graph/schema.graphqls`

```graphql
type Query {
  order(id: ID!): Order           # 注文1件取得
  orders(filter: OrderFilter): [Order!]!  # 注文一覧
}

type Mutation {
  createOrder(input: CreateOrderInput!): Order!  # 注文作成
}

type Order {
  id: ID!
  status: OrderStatus!
  total: Float!
  items: [OrderItem!]!
}
```

**読むときのポイント**: `Query` と `Mutation` を読めば「クライアントから何ができるか」がすべてわかる。型定義で「どんなデータが返るか」も把握できる。

### Step 2: リゾルバーを読む

**場所**: `regions/service/{サービス名}/server/go/bff/graph/schema.resolvers.go`

各 Query / Mutation の実装がここにある。「スキーマの各項目がどう処理されるか」を追う。多くの場合、業務サーバーへの gRPC 呼び出しが書かれている。

## BFF（Rust: async-graphql）

### 読む順番

```
1. schema/query.rs       → Queryルート（取得系API一覧）
2. schema/mutation.rs    → Mutationルート（更新系API一覧）
3. resolver/             → 個別リゾルバー（各APIの処理内容）
4. client/               → 業務サーバーへのgRPCクライアント
```

Go BFF と同じく「どんな API が提供されているか → 各 API がどう実装されているか」の順番で読む。

## クライアント（React）

### 読む順番

```
1. package.json          → 依存ライブラリの確認
2. src/main.tsx          → エントリーポイント（Providerの設定）
3. src/app.tsx           → アプリルート
4. src/routes/           → ルーティング（URL構造 = 画面構造）
5. src/features/         → 機能モジュール（メインの実装）
   ├── api/              →   APIフック（サーバーとの通信）
   ├── components/       →   UIコンポーネント
   ├── hooks/            →   カスタムフック
   ├── stores/           →   Zustandストア（UI状態）
   └── types/            →   型定義
6. src/shared/           → 共有コンポーネント・ユーティリティ
```

### エントリーポイントから読む

**場所**: `regions/service/{サービス名}/client/react/src/main.tsx`

```tsx
// 典型的なmain.tsx
import { QueryClientProvider } from "@tanstack/react-query";
import { AuthProvider } from "system-client/auth";

createRoot(document.getElementById("root")!).render(
  <QueryClientProvider client={queryClient}>
    <AuthProvider>
      <RouterProvider router={router} />
    </AuthProvider>
  </QueryClientProvider>
);
```

**読むときのポイント**: どんな Provider でアプリ全体がラップされているかを確認する。ここで認証、データフェッチ、ルーティングの基盤が設定されている。

### ルーティングの追い方

**場所**: `regions/service/{サービス名}/client/react/src/routes/`

```
routes/
├── __root.tsx            # ルートレイアウト（ヘッダー、サイドバー等）
├── index.tsx             # / （トップページ）
└── orders/
    ├── index.tsx         # /orders （注文一覧）
    └── $orderId.tsx      # /orders/:orderId （注文詳細）
```

**読むときのポイント**: ファイル構造がそのまま URL 構造になる。各ファイルの `Route` 定義を見ると、`beforeLoad`（認証ガード）や `loader`（データ先読み）の設定がわかる。

### 状態管理の追い方

「あるデータがどこから来て、どう画面に表示されるか」を追うフロー。

```
画面コンポーネント
  → useQuery フック（TanStack Query）を使ってデータ取得
    → API 呼び出し関数（api.get / graphqlClient.request 等）
      → BFF のエンドポイント
        → レスポンスの型定義
          → 画面に表示
```

具体例:

```tsx
// 1. 画面コンポーネント（routes/orders/index.tsx）
function OrderListPage() {
  const { data: orders, isLoading } = useOrders();  // ← ここからフックを追う
  // ...
}

// 2. APIフック（features/order/api/use-orders.ts）
export function useOrders() {
  return useQuery({
    queryKey: orderKeys.lists(),
    queryFn: () => api.get<Order[]>("/api/v1/orders"),  // ← API呼び出し
  });
}

// 3. 型定義（features/order/types/order.ts）
export interface Order {
  id: string;
  status: OrderStatus;
  total: number;
  // ...
}
```

### フォーム処理の追い方

```
フォームコンポーネント
  → useForm（React Hook Form）でフォーム管理
    → zodResolver で Zod スキーマを適用（バリデーション）
      → handleSubmit で送信
        → useMutation（TanStack Query）でAPI呼び出し
```

具体例:

```tsx
// 1. Zodスキーマ（features/order/types/schemas.ts）
const createOrderSchema = z.object({
  customerId: z.string().uuid(),
  items: z.array(...).min(1, "1件以上必要"),
});

// 2. フォームコンポーネント（features/order/components/CreateOrderForm.tsx）
function CreateOrderForm() {
  const form = useForm({
    resolver: zodResolver(createOrderSchema),  // バリデーション適用
  });
  const createOrder = useCreateOrder();  // ミューテーションフック

  return (
    <form onSubmit={form.handleSubmit((data) => createOrder.mutate(data))}>
      {/* フィールド */}
    </form>
  );
}
```

## クライアント（Flutter）

### 読む順番

```
1. pubspec.yaml               → 依存パッケージの確認
2. lib/main.dart              → エントリーポイント
3. lib/app.dart               → アプリルート（ProviderScope等）
4. lib/router/app_router.dart → ルーティング（URL構造 = 画面構造）
5. lib/features/              → 機能モジュール（メインの実装）
   ├── domain/                →   モデル定義（freezed）
   ├── data/                  →   リポジトリ実装（dio）
   ├── providers/             →   Riverpodプロバイダー
   └── presentation/          →   画面・Widget
6. lib/shared/                → 共有Widget・ユーティリティ
```

### エントリーポイントから読む

**場所**: `regions/service/{サービス名}/client/flutter/lib/main.dart`

```dart
void main() {
  runApp(
    ProviderScope(  // Riverpodの初期化
      child: const App(),
    ),
  );
}
```

### 状態管理の追い方

```
画面Widget
  → ref.watch(xxxProvider) でデータ監視
    → Provider 定義（非同期データ取得）
      → Repository（dioでAPI呼び出し）
        → レスポンスの型（freezedモデル）
          → 画面に表示
```

具体例:

```dart
// 1. 画面Widget（features/order/presentation/order_list_screen.dart）
class OrderListScreen extends ConsumerWidget {
  Widget build(BuildContext context, WidgetRef ref) {
    final orders = ref.watch(orderListProvider);  // ← ここからProviderを追う
    return orders.when(
      data: (list) => ListView(...),
      loading: () => CircularProgressIndicator(),
      error: (e, _) => Text('エラー: $e'),
    );
  }
}

// 2. Provider（features/order/providers/order_providers.dart）
@riverpod
Future<List<Order>> orderList(OrderListRef ref) async {
  final repo = ref.watch(orderRepositoryProvider);
  return repo.getOrders();  // ← Repositoryを追う
}

// 3. Repository（features/order/data/order_repository.dart）
class OrderRepository {
  final Dio _dio;
  Future<List<Order>> getOrders() async {
    final response = await _dio.get('/api/v1/orders');
    return (response.data as List).map((e) => Order.fromJson(e)).toList();
  }
}

// 4. モデル（features/order/domain/order.dart）
@freezed
class Order with _$Order {
  const factory Order({
    required String id,
    required OrderStatus status,
    required double total,
  }) = _Order;
  factory Order.fromJson(Map<String, dynamic> json) => _$OrderFromJson(json);
}
```

## 共通: テストコードから仕様を理解する

テストコードは「この機能がどう動くべきか」を具体例で示した仕様書でもある。

### テストから読み取れること

| テストの種類 | 読み取れること |
| --- | --- |
| ドメインの単体テスト | ビジネスルール（「こういう入力のときにこう振る舞う」） |
| ユースケースのテスト | 業務フロー（「こういう手順で処理される」） |
| API 統合テスト | リクエスト/レスポンスの具体例 |
| コンポーネントテスト | UI の振る舞い（「このボタンを押すとこうなる」） |

### テストの探し方

```bash
# Rust: テストはソースファイル内の #[cfg(test)] モジュール、またはtests/ディレクトリ
$ find regions/service/order/server/rust -name "*.rs" | xargs grep -l "#\[test\]"

# Go: _test.go ファイル
$ find regions/service/order/server/go -name "*_test.go"

# React: .test.tsx / .test.ts ファイル
$ find regions/service/order/client/react -name "*.test.*"

# Flutter: test/ ディレクトリ
$ find regions/service/order/client/flutter/test -name "*_test.dart"
```

## 共通: 「この機能がどこに実装されているか」を探す方法

### 方法 1: API 定義から追う（サーバー側）

1. `.proto` ファイルで RPC 名を検索
2. presentation 層でハンドラーを見つける
3. ユースケース → ドメイン → リポジトリと追っていく

```bash
# 例: 「注文ステータスの更新」がどこにあるか
$ grep -r "update_status\|UpdateStatus" regions/service/order/server/rust/src/
```

### 方法 2: GraphQL スキーマから追う（BFF → サーバー）

1. `schema.graphqls` で Query / Mutation を検索
2. リゾルバーでどの gRPC メソッドを呼んでいるか確認
3. 業務サーバーの presentation 層に到達

### 方法 3: 画面から追う（クライアント側）

1. URL から `routes/` のファイルを特定
2. ページコンポーネント内の `useQuery` / `ref.watch` を見つける
3. API フック / Provider → API エンドポイントと追っていく

### 方法 4: エラーメッセージから追う

```bash
# エラーメッセージの文字列で全文検索
$ grep -r "在庫が不足しています" regions/service/order/
```

### 方法 5: DB テーブルから追う

1. `migrations/` でテーブル定義を確認
2. `infrastructure/repository/` でそのテーブルを操作する SQL を検索
3. リポジトリトレイト → ユースケースと逆に追う
