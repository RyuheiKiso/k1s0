# service tier チェックリスト

実装からマージまでの各フェーズで確認すべき項目をチェックボックス形式でまとめた。PR 提出前に一通り確認してほしい。

---

## 実装前

- [ ] 設計書を確認した（該当する機能の設計書、API 仕様、DB スキーマ等）
- [ ] 影響範囲を把握した（変更するファイル、影響を受ける機能を列挙）
- [ ] ブランチを `main` の最新から作成した
- [ ] ブランチ名が命名規則に従っている（`feat/order/xxx`、`fix/order/xxx` 等）
- [ ] 必要な依存パッケージが sparse-checkout でチェックアウトされている
- [ ] ローカル環境で DB、認証サーバー等が起動している

---

## サーバー実装中

### 層の責務

- [ ] **ドメイン層**: ビジネスルールがドメイン層に書かれている（DB やフレームワークに依存していない）
- [ ] **ユースケース層**: 業務フローがユースケースに記述されている（プレゼンテーション層に直接ロジックを書いていない）
- [ ] **プレゼンテーション層**: リクエスト → DTO 変換 → ユースケース呼び出し → レスポンス変換の流れになっている
- [ ] **インフラストラクチャ層**: DB アクセスや外部 API 呼び出しがこの層に閉じている
- [ ] 依存方向が正しい（外側 → 内側。ドメイン層がインフラ層に依存していない）

### 命名規約

- [ ] Rust: snake_case（関数名、変数名）、PascalCase（構造体名、enum 名）
- [ ] Go: camelCase（非公開）、PascalCase（公開）
- [ ] proto: lower_snake_case（フィールド名）、PascalCase（メッセージ名、RPC 名）
- [ ] ファイル名がプロジェクトの規約に従っている

### エラーハンドリング

- [ ] エラーが適切な型で返されている（`Result<T, E>` / `Status` / `AppError`）
- [ ] gRPC のステータスコードが適切（NotFound、InvalidArgument、Internal 等）
- [ ] ユーザー向けのエラーメッセージが適切（内部情報を漏らしていない）
- [ ] 予期しないエラーがパニックではなく `Err` で返されている

### ログ出力

- [ ] 重要な処理の開始・完了をログ出力している
- [ ] エラー発生時に十分な情報（ID、パラメータ等）をログに含めている
- [ ] `tracing::info!` / `tracing::error!` 等の構造化ログを使っている
- [ ] デバッグ用の `println!` / `dbg!` が残っていない

### DB 関連

- [ ] SQL インジェクション対策がされている（sqlx のプレースホルダーを使用）
- [ ] マイグレーションファイルが追加されている（スキーマ変更がある場合）
- [ ] インデックスが適切に設定されている（WHERE 句で頻繁に使うカラム）
- [ ] トランザクションが必要な場面で使われている

---

## クライアント実装中

### コンポーネント分割

- [ ] コンポーネントが適切な粒度で分割されている（1 コンポーネント 1 責務）
- [ ] 再利用可能なコンポーネントは `shared/components/` に配置している
- [ ] 機能固有のコンポーネントは `features/{機能名}/components/` に配置している

### 状態管理の選択

- [ ] サーバーからのデータは TanStack Query（React）/ Riverpod（Flutter）で管理している
- [ ] UI だけの状態は Zustand（React）/ Riverpod（Flutter）で管理している
- [ ] 不必要なグローバルステートを作っていない（ローカルの `useState` で済む場合はそちらを使う）

### アクセシビリティ

- [ ] インタラクティブ要素（ボタン、リンク、フォーム入力）に適切な `aria-label` がある
- [ ] キーボード操作でフォーカスが適切に移動する
- [ ] Radix UI のプリミティブを使っている（カスタム実装でアクセシビリティを壊していない）
- [ ] 画像に `alt` テキストが設定されている

### レスポンシブ

- [ ] モバイル画面幅（320px 以上）でレイアウトが崩れない
- [ ] Tailwind のレスポンシブプレフィックス（`sm:`, `md:`, `lg:`）を適切に使っている

### フォーム

- [ ] React Hook Form + Zod でバリデーションが設定されている
- [ ] エラーメッセージがユーザーにわかりやすい
- [ ] 送信中の二重送信が防止されている（ボタンの `disabled` 等）
- [ ] 送信成功後に適切なフィードバックがある（トースト通知、画面遷移等）

---

## PR 提出前

### テスト

- [ ] 単体テストが通っている（`cargo test` / `go test ./...` / `pnpm test` / `flutter test`）
- [ ] 追加・変更した機能にテストを書いた
- [ ] テストがドメインロジックの境界値を網羅している
- [ ] モックが適切に使われている（外部依存を差し替えている）

### lint / フォーマット

- [ ] Rust: `cargo clippy -- -D warnings` が通っている
- [ ] Rust: `cargo fmt --check` が通っている
- [ ] Go: `golangci-lint run` が通っている
- [ ] TypeScript: `pnpm lint` が通っている
- [ ] TypeScript: `pnpm format:check`（Prettier）が通っている
- [ ] Dart: `flutter analyze` が通っている

### コミットメッセージ

- [ ] Conventional Commits 形式に従っている（`feat(order): ...`）
- [ ] メッセージが変更内容を適切に表現している

### 不要コードの除去

- [ ] `console.log` / `console.error` が残っていない（React）
- [ ] `print` / `debugPrint` が残っていない（Flutter）
- [ ] `dbg!` / `println!` が残っていない（Rust）
- [ ] `fmt.Println` が残っていない（Go）
- [ ] コメントアウトしたコードが残っていない
- [ ] テスト用のハードコードされた値が残っていない

### 型安全性

- [ ] `any` 型を使っていない（TypeScript）
- [ ] `as` による型アサーションを最小限にしている（TypeScript）
- [ ] `unwrap()` を安易に使っていない（Rust。テストコード内は許容）
- [ ] `dynamic` 型を最小限にしている（Dart）

### proto / スキーマ

- [ ] `.proto` ファイルを変更した場合、コード生成を実行した
- [ ] GraphQL スキーマを変更した場合、コード生成を実行した
- [ ] `buf lint` / `buf breaking` が通っている（proto 変更時）

### その他

- [ ] 機密情報（パスワード、トークン、API キー）がコードにハードコードされていない
- [ ] 変更が自分のサービスの範囲内に収まっている（他サービスを意図せず変更していない）
- [ ] PR の説明に変更概要、テスト方法、影響範囲を記載した

---

## レビュー対応

- [ ] すべてのレビューコメントを確認した
- [ ] 指摘事項をすべて対応した（修正 or 理由を説明して合意）
- [ ] 修正を新しいコミットとして追加した（既存コミットを amend していない）
- [ ] レビュアーに返信し、再レビューを依頼した
- [ ] 修正後にテストを再実行して通過を確認した

---

## マージ後

- [ ] CI パイプラインが通過した（lint → test → build）
- [ ] dev / staging 環境へのデプロイが成功した
- [ ] デプロイ後の動作確認を行った
  - [ ] 追加・変更した機能が正しく動作する
  - [ ] 既存機能が壊れていない（リグレッションなし）
  - [ ] ログにエラーが出ていない
- [ ] 作業ブランチを削除した

```bash
# マージ済みブランチの削除
$ git checkout main
$ git pull origin main
$ git branch -d feat/order/add-status-update-api
```
