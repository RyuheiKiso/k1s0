# インフラストラクチャ

## Kubernetes クラスタ構成

k1s0 はオンプレミス Kubernetes クラスタ上で稼働する。tier ごとに Namespace を分離し、リソースの管理境界を明確にする。

### Namespace 構成

| Namespace | 用途 |
|-----------|------|
| k1s0-system | system tier の全サーバー |
| k1s0-business | business tier のサーバー |
| k1s0-service | service tier のサーバー |
| k1s0-infra | Kafka, PostgreSQL, Redis 等のミドルウェア |
| k1s0-observability | Prometheus, Grafana, Loki, Jaeger |
| k1s0-auth | Keycloak, Vault |
| istio-system | Istio コントロールプレーン |

### リソース設計

各サーバーの Pod には適切なリソース制限を設定する。

```yaml
resources:
  requests:
    cpu: "100m"
    memory: "128Mi"
  limits:
    cpu: "500m"
    memory: "512Mi"
```

### ネットワークポリシー

Namespace 間の通信は NetworkPolicy で制御する。system tier は他 tier からの受信のみ許可し、上位 tier への発信は行わない。

```yaml
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: allow-from-business
  namespace: k1s0-system
spec:
  podSelector: {}
  ingress:
    - from:
        - namespaceSelector:
            matchLabels:
              tier: business
        - namespaceSelector:
            matchLabels:
              tier: service
```

## Helm Chart 設計とデプロイパターン

### Chart 構造

各サーバーは統一された Helm Chart 構造を持つ。

```
helm/
├── Chart.yaml
├── values.yaml           # デフォルト値
├── values-dev.yaml       # 開発環境オーバーライド
├── values-staging.yaml   # ステージング環境
├── values-prod.yaml      # 本番環境
└── templates/
    ├── deployment.yaml
    ├── service.yaml
    ├── configmap.yaml
    ├── secret.yaml
    ├── hpa.yaml           # Horizontal Pod Autoscaler
    ├── pdb.yaml           # PodDisruptionBudget
    ├── serviceaccount.yaml
    └── virtualservice.yaml  # Istio
```

### デプロイコマンド

```bash
# 開発環境へのデプロイ
helm upgrade --install auth-server ./helm/ \
  -f helm/values-dev.yaml \
  --namespace k1s0-system \
  --set image.tag=$(git rev-parse --short HEAD)

# 本番環境へのデプロイ（CI/CD から実行）
helm upgrade --install auth-server ./helm/ \
  -f helm/values-prod.yaml \
  --namespace k1s0-system \
  --set image.tag=${RELEASE_TAG}
```

### 共通 Chart ライブラリ

繰り返しを避けるため、共通の Helm Chart ライブラリ（k1s0-common-chart）を提供する。各サーバーの Chart はこれを dependency として参照する。

## Terraform モジュール構成

インフラのプロビジョニングは Terraform で管理する。

### モジュール構成

```
infrastructure/terraform/
├── modules/
│   ├── kubernetes/       # K8s クラスタ構築
│   ├── keycloak/         # Keycloak Realm / Client 設定
│   ├── vault/            # Vault 設定・ポリシー
│   ├── kafka/            # Kafka トピック・ACL
│   ├── postgresql/       # DB / ユーザー / スキーマ
│   └── monitoring/       # Prometheus ルール、Grafana ダッシュボード
├── environments/
│   ├── dev/
│   │   ├── main.tf
│   │   ├── variables.tf
│   │   └── terraform.tfvars
│   ├── staging/
│   └── prod/
└── backend.tf            # State 管理
```

### 使用例

```hcl
# environments/dev/main.tf
module "keycloak" {
  source = "../../modules/keycloak"

  realm_name    = "k1s0"
  clients       = var.keycloak_clients
  roles         = var.keycloak_roles
  keycloak_url  = var.keycloak_url
}

module "kafka" {
  source = "../../modules/kafka"

  topics = [
    {
      name       = "events.tenant"
      partitions = 6
      retention  = "168h"
    },
  ]
}
```

## Docker マルチステージビルド

全サーバーはマルチステージビルドで最小限のイメージを作成する。

### Rust サーバーの Dockerfile

```dockerfile
# ステージ 1: ビルド
FROM rust:1.77-bookworm AS builder
WORKDIR /app
COPY . .
RUN cargo build --release -p auth-server

# ステージ 2: 実行
FROM debian:bookworm-slim
RUN apt-get update && apt-get install -y ca-certificates && rm -rf /var/lib/apt/lists/*
COPY --from=builder /app/target/release/auth-server /usr/local/bin/
COPY --from=builder /app/regions/system/server/rust/auth/config.yaml /etc/k1s0/
EXPOSE 50051 8080
CMD ["auth-server"]
```

### Go BFF の Dockerfile

```dockerfile
# ステージ 1: ビルド
FROM golang:1.22-bookworm AS builder
WORKDIR /app
COPY go.mod go.sum ./
RUN go mod download
COPY . .
RUN CGO_ENABLED=0 go build -o /graphql-gateway ./regions/system/server/go/graphql-gateway/

# ステージ 2: 実行
FROM gcr.io/distroless/static-debian12
COPY --from=builder /graphql-gateway /usr/local/bin/
COPY --from=builder /app/regions/system/server/go/graphql-gateway/config.yaml /etc/k1s0/
EXPOSE 8080
CMD ["graphql-gateway"]
```

### イメージサイズの目安

| サーバー種別 | ベースイメージ | 目標サイズ |
|-------------|--------------|-----------|
| Rust サーバー | debian:bookworm-slim | 30-50 MB |
| Go BFF | distroless/static | 15-30 MB |

## CI/CD パイプライン（GitHub Actions）

### パイプライン構成

```yaml
# .github/workflows/system-tier.yml
name: System Tier CI/CD

on:
  push:
    branches: [main]
    paths:
      - 'regions/system/**'
  pull_request:
    paths:
      - 'regions/system/**'

jobs:
  lint:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Rust lint
        run: cargo clippy --all-targets -- -D warnings
      - name: Go lint
        run: golangci-lint run
      - name: Proto lint
        run: buf lint

  test:
    runs-on: ubuntu-latest
    services:
      postgres:
        image: postgres:17
        env:
          POSTGRES_PASSWORD: test
      kafka:
        image: confluentinc/cp-kafka:7.6.0
    steps:
      - uses: actions/checkout@v4
      - name: Rust test
        run: cargo test --workspace
      - name: Go test
        run: go test ./...

  build:
    needs: [lint, test]
    runs-on: ubuntu-latest
    strategy:
      matrix:
        service: [auth, session, config, tenant]  # 変更されたサービスのみ
    steps:
      - uses: actions/checkout@v4
      - name: Build Docker image
        run: docker build -t ${{ matrix.service }}:${{ github.sha }} .
      - name: Push to registry
        run: docker push registry.k1s0.internal/${{ matrix.service }}:${{ github.sha }}

  deploy:
    needs: [build]
    if: github.ref == 'refs/heads/main'
    runs-on: ubuntu-latest
    steps:
      - name: Deploy to dev
        run: |
          helm upgrade --install $SERVICE ./helm/ \
            -f helm/values-dev.yaml \
            --set image.tag=${{ github.sha }}
```

### CI の最適化

- **変更検出**: 変更されたサービスのみビルド・テスト（paths フィルタ + dorny/paths-filter）
- **キャッシュ**: Cargo / Go モジュールのキャッシュ、Docker レイヤーキャッシュ
- **並列実行**: 独立したサービスのビルドを matrix で並列化
- **Breaking Change 検出**: `buf breaking` で Proto の破壊的変更を CI で検出

## サービスメッシュ（Istio）

### 機能

| 機能 | 説明 |
|------|------|
| mTLS | サービス間通信の自動暗号化 |
| トラフィック制御 | カナリアリリース、トラフィックシフト |
| リトライ・タイムアウト | 自動リトライ、タイムアウト制御 |
| サーキットブレーカー | 障害の伝播防止 |
| 可観測性 | メトリクス・トレースの自動収集 |

### VirtualService 設定例

```yaml
apiVersion: networking.istio.io/v1
kind: VirtualService
metadata:
  name: auth-server
  namespace: k1s0-system
spec:
  hosts:
    - auth-server
  http:
    - route:
        - destination:
            host: auth-server
            port:
              number: 50051
      timeout: 10s
      retries:
        attempts: 3
        perTryTimeout: 3s
        retryOn: "unavailable,resource-exhausted"
```

### DestinationRule（サーキットブレーカー）

```yaml
apiVersion: networking.istio.io/v1
kind: DestinationRule
metadata:
  name: auth-server
  namespace: k1s0-system
spec:
  host: auth-server
  trafficPolicy:
    connectionPool:
      http:
        h2UpgradePolicy: UPGRADE  # gRPC 用
    outlierDetection:
      consecutive5xxErrors: 5
      interval: 30s
      baseEjectionTime: 30s
```

## 可観測性スタック

k1s0 は OpenTelemetry を中心とした可観測性スタックを構築する。3 つの柱（トレース、メトリクス、ログ）を統合的に管理する。

### アーキテクチャ

<img src="../img/observability-stack.svg" width="800" />

### トレーシング（Jaeger）

全サービスは OpenTelemetry SDK でトレースを収集し、OTLP プロトコルで OTel Collector に送信する。

```rust
use k1s0_server_common::trace::init_tracing;

// サーバー起動時に初期化
init_tracing(&config.telemetry)?;

// 自動的に gRPC / HTTP リクエストのスパンが作成される
// カスタムスパンの追加
#[tracing::instrument(skip(self))]
async fn process_request(&self, input: Input) -> Result<Output> {
    tracing::info!(tenant_id = %input.tenant_id, "processing request");
    // ...
}
```

### メトリクス（Prometheus）

各サーバーは `/metrics` エンドポイントで Prometheus メトリクスを公開する。

標準メトリクス:
- `grpc_server_handled_total` — gRPC リクエスト数（ステータスコード別）
- `grpc_server_handling_seconds` — gRPC レイテンシ
- `http_requests_total` — HTTP リクエスト数
- `db_pool_connections` — DB コネクション数
- `kafka_consumer_lag` — Kafka コンシューマラグ

### ログ（Loki）

構造化ログ（JSON 形式）を標準出力に出力し、Loki で集約する。

```json
{
  "timestamp": "2024-01-15T10:30:00Z",
  "level": "INFO",
  "target": "auth_server::usecase",
  "message": "user authenticated",
  "trace_id": "abc123",
  "span_id": "def456",
  "tenant_id": "tenant-001",
  "user_id": "user-001"
}
```

### Grafana ダッシュボード

以下のダッシュボードが事前構築されている。

| ダッシュボード | 内容 |
|--------------|------|
| Service Overview | 全サービスのリクエスト数、エラー率、レイテンシ |
| Service Detail | 個別サービスの詳細メトリクス |
| Database | DB コネクション数、クエリレイテンシ、スロークエリ |
| Kafka | トピック別メッセージ数、コンシューマラグ |
| Infrastructure | Node CPU/Memory、Pod 状態 |

### SLO 設計

各サービスは SLO（Service Level Objective）を定義し、Grafana で監視する。

| SLI | SLO |
|-----|-----|
| 可用性（成功リクエスト率） | 99.9% |
| レイテンシ（p99） | < 500ms |
| エラー率 | < 0.1% |

SLO 違反時は Alertmanager 経由で通知が送信される。
