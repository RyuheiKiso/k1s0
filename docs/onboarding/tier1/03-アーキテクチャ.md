# アーキテクチャ

## クリーンアーキテクチャの層構成

k1s0 の全サーバー・ライブラリはクリーンアーキテクチャに基づく 4 層構成を採用する。依存の方向は**外側から内側への一方向**のみ許可される。

```
┌─────────────────────────────────────────┐
│  infrastructure（最外層）                │
│  DB接続, gRPCサーバー, Kafka, HTTP      │
│  ┌─────────────────────────────────┐    │
│  │  adapter                        │    │
│  │  ハンドラ, リポジトリ実装,       │    │
│  │  外部サービスクライアント        │    │
│  │  ┌─────────────────────────┐    │    │
│  │  │  usecase                │    │    │
│  │  │  ビジネスロジック,       │    │    │
│  │  │  ユースケース実行        │    │    │
│  │  │  ┌─────────────────┐   │    │    │
│  │  │  │  domain（最内層）│   │    │    │
│  │  │  │  エンティティ,   │   │    │    │
│  │  │  │  値オブジェクト, │   │    │    │
│  │  │  │  ドメインサービス│   │    │    │
│  │  │  └─────────────────┘   │    │    │
│  │  └─────────────────────────┘    │    │
│  └─────────────────────────────────┘    │
└─────────────────────────────────────────┘
```

### 各層の責務

| 層 | 責務 | 依存先 | 典型的な実装 |
|---|------|--------|-------------|
| domain | ビジネスルール、ドメインモデル | なし（最内層） | エンティティ、値オブジェクト、ドメインイベント、リポジトリ trait/interface |
| usecase | アプリケーション固有のビジネスロジック | domain | ユースケース実行、トランザクション制御、ドメインサービスの協調 |
| adapter | 外部との接続を変換 | usecase, domain | gRPC ハンドラ、リポジトリの DB 実装、外部 API クライアント |
| infrastructure | フレームワーク・ドライバ | adapter, usecase, domain | axum/tonic の起動、DB コネクションプール、Kafka プロデューサ |

### 依存ルール

- domain 層は他の層に一切依存しない
- usecase 層は domain 層のみに依存する
- adapter 層は usecase と domain に依存するが、infrastructure には依存しない
- infrastructure 層は全ての層に依存してよい
- **逆方向の依存は禁止**（例: domain が adapter に依存する等）

## DDD のドメインモデリング

### エンティティ

一意な識別子を持ち、ライフサイクルを通じて同一性が保たれるオブジェクト。

```rust
// Rust の例
pub struct Tenant {
    id: TenantId,          // 値オブジェクト（識別子）
    name: TenantName,      // 値オブジェクト
    status: TenantStatus,  // 列挙型
    created_at: DateTime<Utc>,
    updated_at: DateTime<Utc>,
}
```

```go
// Go の例
type Tenant struct {
    ID        TenantID
    Name      TenantName
    Status    TenantStatus
    CreatedAt time.Time
    UpdatedAt time.Time
}
```

### 値オブジェクト

不変で、値によって等価性が判断されるオブジェクト。バリデーションロジックを内包する。

```rust
// Rust の例
pub struct TenantName(String);

impl TenantName {
    pub fn new(value: &str) -> Result<Self, DomainError> {
        if value.is_empty() || value.len() > 100 {
            return Err(DomainError::InvalidTenantName);
        }
        Ok(Self(value.to_string()))
    }
}
```

### リポジトリ

domain 層でインターフェース（trait / interface）を定義し、adapter 層で具象実装を提供する。

```rust
// domain 層: trait 定義
#[async_trait]
pub trait TenantRepository: Send + Sync {
    async fn find_by_id(&self, id: &TenantId) -> Result<Option<Tenant>, DomainError>;
    async fn save(&self, tenant: &Tenant) -> Result<(), DomainError>;
    async fn delete(&self, id: &TenantId) -> Result<(), DomainError>;
}
```

```rust
// adapter 層: PostgreSQL 実装
pub struct PgTenantRepository {
    pool: PgPool,
}

#[async_trait]
impl TenantRepository for PgTenantRepository {
    async fn find_by_id(&self, id: &TenantId) -> Result<Option<Tenant>, DomainError> {
        // sqlx によるクエリ実装
    }
    // ...
}
```

## 共通設計パターン

### エラーハンドリング

system tier では統一的なエラー型を使用する。domain 層のエラーを gRPC ステータスコードに変換するパターンが標準。

```rust
// domain 層のエラー
#[derive(Debug, thiserror::Error)]
pub enum DomainError {
    #[error("not found: {0}")]
    NotFound(String),
    #[error("already exists: {0}")]
    AlreadyExists(String),
    #[error("validation error: {0}")]
    Validation(String),
    #[error("unauthorized")]
    Unauthorized,
}

// adapter 層: DomainError → gRPC Status 変換
impl From<DomainError> for tonic::Status {
    fn from(err: DomainError) -> Self {
        match err {
            DomainError::NotFound(msg) => Status::not_found(msg),
            DomainError::AlreadyExists(msg) => Status::already_exists(msg),
            DomainError::Validation(msg) => Status::invalid_argument(msg),
            DomainError::Unauthorized => Status::unauthenticated("unauthorized"),
        }
    }
}
```

### 設定管理

全サーバーは `config.yaml` から設定を読み込む。環境変数によるオーバーライドをサポートする。

```yaml
# config.yaml の基本構成
server:
  host: "0.0.0.0"
  grpc_port: 50051
  http_port: 8080

  url: "${DATABASE_URL}"
  max_connections: 10

  consumer_group: "auth-service"

  service_name: "auth"

```

### DI（依存性注入）

Rust では trait オブジェクトまたはジェネリクスによる DI、Go ではインターフェースによる DI を採用する。

```rust
// Rust: ユースケースへの DI
pub struct CreateTenantUseCase<R: TenantRepository> {
    repository: R,
}

impl<R: TenantRepository> CreateTenantUseCase<R> {
    pub fn new(repository: R) -> Self {
        Self { repository }
    }

    pub async fn execute(&self, input: CreateTenantInput) -> Result<Tenant, DomainError> {
        let tenant = Tenant::new(input.name)?;
        self.repository.save(&tenant).await?;
        Ok(tenant)
    }
}
```

### TDD の実践

全てのコードはテスト駆動で開発する。テストの粒度は以下の通り。

| テスト種別 | 対象 | 実行環境 |
|-----------|------|---------|
| ユニットテスト | domain, usecase | モック使用、DB 不要 |
| 統合テスト | adapter（リポジトリ実装等） | テスト用 PostgreSQL |
| E2E テスト | gRPC エンドポイント | docker-compose による全スタック |

```rust
// ユニットテストの例（モックリポジトリ使用）
#[cfg(test)]
mod tests {
    use super::*;
    use mockall::predicate::*;

    #[tokio::test]
    async fn test_create_tenant() {
        let mut mock_repo = MockTenantRepository::new();
        mock_repo
            .expect_save()
            .returning(|_| Ok(()));

        let usecase = CreateTenantUseCase::new(mock_repo);
        let result = usecase.execute(CreateTenantInput {
            name: "test-tenant".to_string(),
        }).await;

        assert!(result.is_ok());
    }
}
```
