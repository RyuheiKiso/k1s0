# コードリーディングガイド

「既存コードをどう読めばいいかわからない」という方のためのガイドです。実際のプロジェクトコード（`regions/business/taskmanagement/server/rust/project-master/`）を例に、読む順番とポイントを解説します。

---

## まずどこから読むか -- 5ステップの読み順

いきなり全ファイルを読もうとすると迷子になります。以下の順番で読むと、全体像から詳細へとスムーズに理解できます。

```
Step 1: config.yaml      → 「このサーバーは何をする? どこに接続する?」
Step 2: Cargo.toml        → 「どんなライブラリを使っている?」
Step 3: main.rs           → 「起動時に何が組み立てられる?」
Step 4: domain/           → 「業務データの構造は? どんな操作ができる?」
Step 5: usecase/           → 「アプリケーションの機能は何?」
（必要に応じて adapter/ と infrastructure/ も読む）
```

### Step 1: config.yaml -- 設定ファイルを読む

**ファイル**: `regions/business/taskmanagement/server/rust/project-master/config/config.yaml`

設定ファイルを最初に読むと、このサーバーの概要が一目でわかります。

```yaml
app:
  name: "k1s0-project-master-server"    # サーバーの名前
  environment: "development"

server:
  host: "0.0.0.0"
  port: 8210          # REST API のポート
  grpc_port: 50061    # gRPC のポート

database:
  host: "localhost"
  name: "k1s0"
  schema: "project_master"    # DB のスキーマ名

kafka:
  brokers:
    - "localhost:9092"
  project_type_topic: "k1s0.business.taskmanagement.projectmaster.project_type_changed.v1"
```

読み取れること:
- サーバー名は `k1s0-project-master-server`
- REST は 8210 番ポート、gRPC は 50061 番ポート
- PostgreSQL の `project_master` スキーマを使う
- Kafka にイベントを送信する

### Step 2: Cargo.toml -- 依存関係を読む

**ファイル**: `regions/business/taskmanagement/server/rust/project-master/Cargo.toml`

このファイルを見ると「何の技術を使っているか」がわかります。

```toml
[dependencies]
axum = { version = "0.8" }           # REST フレームワーク
sqlx = { version = "0.8", ... }      # DB アクセス
tonic = "0.12"                       # gRPC フレームワーク
rdkafka = { version = "0.37" }       # Kafka クライアント

# system tier のライブラリ（k1s0 プロジェクト内）
k1s0-telemetry = { path = "../../../../../system/library/rust/telemetry" }
k1s0-auth = { path = "../../../../../system/library/rust/auth" }
k1s0-server-common = { path = "../../../../../system/library/rust/server-common" }
```

注目ポイント:
- `path = "..."` で指定されているのはプロジェクト内のライブラリ（system tier のもの）
- `[dev-dependencies]` にテスト用ライブラリ（`mockall`, `axum-test` 等）がある

### Step 3: main.rs -- エントリーポイントを読む

**ファイル**: `regions/business/taskmanagement/server/rust/project-master/src/main.rs`

main.rs はサーバーの「組み立て図」です。上から順に読むと、起動時に何が行われるかがわかります。

```
main.rs の構造（上から順に）:
1. テレメトリ初期化    → ログ・トレースの設定
2. 設定読み込み        → config.yaml を読む
3. DB 接続             → PostgreSQL に接続してマイグレーション実行
4. メトリクス作成      → 監視用の数値カウンター
5. リポジトリ作成      → DB 操作の実装を作成
6. Kafka プロデューサー → メッセージ送信の準備
7. ユースケース作成    → ビジネスロジックの組み立て
8. 認証設定            → JWT 検証の準備
9. ルーター作成        → URL とハンドラーの対応付け
10. gRPC サービス      → gRPC ハンドラーの組み立て
11. サーバー起動       → REST と gRPC を並行起動
```

この流れで特に重要なのは「依存性注入」のパターンです:

```rust
// リポジトリの実装を作成して...
let project_type_repo: Arc<dyn ProjectTypeRepository> =
    Arc::new(ProjectTypePostgresRepository::new(db_pool.clone()));

// ユースケースに渡す
let manage_project_types_uc = Arc::new(
    ManageProjectTypesUseCase::new(project_type_repo.clone()),
);
```

`dyn ProjectTypeRepository` はインターフェース（trait）で、`ProjectTypePostgresRepository` はその実装です。ユースケースはインターフェースだけに依存するので、テスト時にはモック（偽物の実装）に差し替えられます。

### Step 4: domain/ -- ドメイン層を読む

ドメイン層は「このサーバーが扱うデータは何か」「どんな操作ができるか」を定義しています。

#### entity/ -- データの構造

**ファイル**: `src/domain/entity/master_project_type.rs`

```rust
pub struct MasterProjectType {
    pub id: Uuid,              // 一意な識別子
    pub code: String,          // プロジェクトタイプコード（例: "scrum"）
    pub display_name: String,  // 表示名（例: "スクラム開発"）
    pub description: Option<String>,  // 説明（あってもなくてもよい）
    pub is_active: bool,       // 有効/無効フラグ
    // ...
}
```

エンティティを読むと「このサーバーが管理するデータの全体像」が見えます。`Option<T>` はその値が「あるかもしれない・ないかもしれない」ことを意味します。

#### repository/ -- データの出入り口の定義

**ファイル**: `src/domain/repository/project_type_repository.rs`

```rust
#[async_trait]
pub trait ProjectTypeRepository: Send + Sync {
    async fn find_all(&self, active_only: bool) -> anyhow::Result<Vec<MasterProjectType>>;
    async fn find_by_code(&self, code: &str) -> anyhow::Result<Option<MasterProjectType>>;
    async fn create(&self, input: &CreateMasterProjectType, created_by: &str) -> anyhow::Result<MasterProjectType>;
    async fn update(&self, code: &str, input: &UpdateMasterProjectType) -> anyhow::Result<MasterProjectType>;
    async fn delete(&self, code: &str) -> anyhow::Result<()>;
}
```

`trait`（トレイト）は「こういうメソッドを持つべき」という約束事の定義です。実際の DB 操作コードは infrastructure 層にあります。

### Step 5: usecase/ -- ユースケースを読む

**ファイル**: `src/usecase/manage_categories.rs`

ユースケースには「アプリケーションの機能」が書かれています。

```rust
pub async fn create_project_type(&self, input: &CreateMasterProjectType, created_by: &str) -> anyhow::Result<MasterProjectType> {
    // 同じコードのプロジェクトタイプが既にあるか確認
    if let Some(_existing) = self.project_type_repo.find_by_code(&input.code).await? {
        anyhow::bail!("Duplicate code: project_type '{}' already exists", input.code);
    }
    // なければ作成
    self.project_type_repo.create(input, created_by).await
}
```

ユースケースを読むと「このサーバーでできること」の全体像がわかります。

## 4層構成 -- 各層にはどんなコードがあるか

```
src/
├── domain/              ← 業務ルール（DB や通信に依存しない）
│   ├── entity/          ← データ構造の定義
│   ├── repository/      ← データ操作のインターフェース定義（trait）
│   └── service/         ← 複数エンティティにまたがる業務ルール
├── usecase/             ← アプリケーション機能（ユースケースごとに1ファイル）
├── adapter/             ← 外部との窓口
│   ├── grpc/            ← gRPC ハンドラー（サービス間通信）
│   └── handler/         ← REST ハンドラー（外部向け API）
├── infrastructure/      ← 技術的な実装
│   ├── persistence/     ← DB 操作の実装（repository trait の実装）
│   ├── messaging/       ← Kafka との通信
│   └── config/          ← 設定ファイルの読み込み
├── proto/               ← protobuf から自動生成されたコード
├── main.rs              ← エントリーポイント
└── lib.rs               ← モジュール構成の定義
```

### 「この層にはこういうコードがある」早見表

| 層 | 含まれるコード | 依存してよいもの |
| --- | --- | --- |
| domain | struct 定義、trait 定義、バリデーションルール | 標準ライブラリ、serde、uuid、chrono のみ |
| usecase | ビジネスロジック、トランザクション制御 | domain 層のみ |
| adapter | リクエストのパース、レスポンスの構築、エラー変換 | usecase 層、domain 層 |
| infrastructure | SQL クエリ、Kafka 送信、設定ファイル読込 | domain 層（trait の実装のため） |

## 依存の追い方

### Rust の場合

```
Cargo.toml              ← 外部ライブラリの依存を確認
  ↓
use 文                   ← ファイル冒頭で何を使っているか確認
  ↓
実際の呼び出し箇所        ← メソッド呼び出しを追う
```

具体例（`manage_categories.rs` の場合）:

```rust
// 1. use 文を見る → domain 層のモデルとリポジトリを使っている
use crate::domain::entity::master_project_type::{CreateMasterProjectType, MasterProjectType, UpdateMasterProjectType};
use crate::domain::repository::project_type_repository::ProjectTypeRepository;

// 2. struct の中身を見る → リポジトリへの参照を持っている
pub struct ManageProjectTypesUseCase {
    project_type_repo: Arc<dyn ProjectTypeRepository>,  // Arc<dyn ...> = trait オブジェクト
}

// 3. メソッドの中を見る → リポジトリのメソッドを呼んでいる
self.project_type_repo.find_by_code(&input.code).await?
```

`crate::` はこのプロジェクト（クレート）自身のモジュールを指します。

### Go の場合

```
go.mod                   ← モジュール名と外部依存を確認
  ↓
import 文                ← パッケージのインポートを確認
  ↓
実際の呼び出し箇所        ← メソッド・関数呼び出しを追う
```

Go では `internal/` ディレクトリが外部パッケージからアクセスできない内部コードを意味します。

## gRPC 定義の読み方

### .proto ファイルを読む

**ファイル**: `api/proto/k1s0/business/taskmanagement/projectmaster/v1/project_master.proto`

```protobuf
service ProjectMasterService {
  rpc ListProjectTypes(ListProjectTypesRequest) returns (ListProjectTypesResponse);
  rpc CreateProjectType(CreateProjectTypeRequest) returns (CreateProjectTypeResponse);
  rpc UpdateProjectType(UpdateProjectTypeRequest) returns (UpdateProjectTypeResponse);
  // ...
}

message MasterProjectType {
  string id = 1;
  string code = 2;
  string display_name = 3;
  string description = 4;
  // ...
}
```

読み方:
- `service` ブロック: このサーバーが提供する操作（RPC）の一覧
- `rpc` 行: 「リクエスト型を受け取って、レスポンス型を返す」という定義
- `message` ブロック: データの構造。フィールド番号（`= 1`, `= 2`）は通信で使う識別子で、変更してはいけない

### 生成コードの場所

proto ファイルから生成されたコードは以下に配置されます:

```
src/proto/
├── k1s0.business.taskmanagement.projectmaster.v1.rs   ← 生成された Rust コード
├── k1s0.system.common.v1.rs                           ← 共通型の生成コード
└── mod.rs                                              ← モジュール定義
```

生成コードは直接編集しません。proto ファイルを変更して再生成します。

### proto → 生成コード → サービス実装の流れ

```
.proto ファイルで RPC を定義
    ↓  (tonic-build / buf で自動生成)
src/proto/ に Rust コードが生成される
    ↓  (生成された trait を実装する)
src/adapter/grpc/project_master_grpc.rs でハンドラーを書く
```

## テストコードの読み方

テストコードは「この機能がどう動くべきか」を示す仕様書でもあります。

### テストファイルの場所

Rust ではテストは同じファイルの末尾か、`tests/` ディレクトリに置かれます。

```rust
// ファイル末尾のテスト（単体テスト）
#[cfg(test)]
mod tests {
    use super::*;
    use mockall::predicate::*;

    #[tokio::test]
    async fn test_create_project_type_success() {
        // Arrange（準備）: モックの設定
        let mut mock_repo = MockProjectTypeRepository::new();
        mock_repo.expect_find_by_code()
            .returning(|_| Ok(None));  // 既存プロジェクトタイプなし
        mock_repo.expect_create()
            .returning(|input, _| Ok(MasterProjectType { /* ... */ }));

        // Act（実行）: テスト対象を呼ぶ
        let uc = ManageProjectTypesUseCase::new(Arc::new(mock_repo));
        let result = uc.create_project_type(&input, "test-user").await;

        // Assert（検証）: 結果を確認
        assert!(result.is_ok());
    }
}
```

テストの読み方:
1. **テスト名** を読む → 「何をテストしているか」がわかる
2. **Arrange** 部分 → 「どういう前提条件か」がわかる
3. **Act** 部分 → 「何を実行するか」がわかる
4. **Assert** 部分 → 「期待する結果は何か」がわかる

### テストから仕様を読み取る

テスト名をざっと見るだけで、そのモジュールの仕様がわかります:

```
test_create_project_type_success          → プロジェクトタイプ作成が成功するケース
test_create_project_type_duplicate_code   → 重複コードでエラーになるケース
test_update_project_type_not_found        → 存在しないプロジェクトタイプの更新でエラー
test_delete_project_type_success          → プロジェクトタイプ削除が成功するケース
```

## 「この機能がどこに実装されているか」を探す方法

### grep を使った検索パターン

```bash
# 特定の構造体がどこで定義されているか
grep -rn "pub struct MasterProjectType" regions/business/taskmanagement/

# 特定のメソッドがどこで呼ばれているか
grep -rn "create_project_type" regions/business/taskmanagement/

# 特定の SQL テーブルを操作している箇所
grep -rn "master_project_types" regions/business/taskmanagement/

# gRPC のサービス名で探す
grep -rn "ProjectMasterService" api/proto/ regions/business/taskmanagement/

# エラーメッセージから実装箇所を探す
grep -rn "already exists" regions/business/taskmanagement/

# 特定の trait の実装箇所を探す
grep -rn "impl ProjectTypeRepository" regions/business/taskmanagement/
```

### よく使う検索パターン

| 目的 | 検索コマンド |
| --- | --- |
| 構造体の定義を探す | `grep -rn "pub struct XXX"` |
| trait の定義を探す | `grep -rn "pub trait XXX"` |
| trait の実装を探す | `grep -rn "impl XXX for"` |
| 関数の定義を探す | `grep -rn "pub async fn xxx"` または `grep -rn "pub fn xxx"` |
| 設定値がどこで使われるか | `grep -rn "config.yaml のキー名"` |
| SQL テーブルの操作箇所 | `grep -rn "テーブル名" --include="*.rs"` |
| proto の RPC 名が使われている箇所 | `grep -rn "RPC名"` |

## 読む順番のまとめ

```
初めてのサーバーを読むとき:
  config.yaml → Cargo.toml → main.rs → domain/entity/ → domain/repository/ → usecase/ → adapter/

特定の機能を理解したいとき:
  proto の RPC 定義 → adapter/grpc/ のハンドラー → usecase/ → domain/ → infrastructure/

バグを追うとき:
  エラーメッセージで grep → 該当ファイル → 呼び出し元を遡る

DB のデータ構造を知りたいとき:
  database/migrations/ の SQL → domain/entity/ の構造体
```
