# チェックリスト

各作業フェーズで確認すべき項目をまとめたチェックリストです。PR を出す前やレビュー対応後に、漏れがないか確認してください。

---

## 実装前チェック

作業を始める前に確認する項目です。ここを飛ばすと、あとから大きな手戻りが発生することがあります。

- [ ] **設計書を確認した**
  - 対象サーバーの設計書（`docs/servers/` 配下）を読んだ
  - 変更するドメインの設計書（`docs/onboarding/tier2/03-ドメイン設計.md` 等）を確認した
  - proto 設計（`docs/architecture/api/proto設計.md`）の命名規約を確認した
- [ ] **影響範囲を把握した**
  - 変更するファイルの一覧を洗い出した
  - 他のサービスやライブラリへの影響がないか確認した
  - DB スキーマの変更が必要かどうか判断した
  - 既存のテストが影響を受けるか確認した
- [ ] **ブランチを正しく作成した**
  - main ブランチから分岐している
  - ブランチ名が命名規則に従っている（例: `feat/add-xxx`, `fix/xxx`）
  - main が最新の状態から分岐している（`git pull origin main` した）
- [ ] **ローカル環境が動作する**
  - `docker-compose up -d` で依存サービスが起動している
  - `cargo build` / `go build` が通る
  - 既存のテストが通る（`cargo test`）

## 実装中チェック

コーディング中に意識する項目です。定期的に振り返りましょう。

### 層の責務を守っているか

- [ ] **domain 層に技術的な依存を入れていない**
  - domain 層で `sqlx` や `tonic` を直接使っていない
  - domain 層で HTTP リクエストや DB 接続を行っていない
  - domain 層は `serde`, `uuid`, `chrono`, `thiserror` のみに依存している
- [ ] **usecase 層がリポジトリ trait 経由でデータにアクセスしている**
  - `Arc<dyn SomeRepository>` の形で依存を受け取っている
  - SQL や DB 固有の処理を書いていない
- [ ] **adapter 層でビジネスロジックを書いていない**
  - gRPC ハンドラーは「入力変換 → ユースケース呼出 → 出力変換」のみ
  - バリデーション（入力値チェック）以外の業務ルールを書いていない
- [ ] **infrastructure 層で domain 層の trait を正しく実装している**
  - `#[async_trait]` を付けている
  - 全てのメソッドを実装している

### 命名規約に従っているか

- [ ] **Rust**: `rustfmt` でフォーマットされている
  - 構造体名は `PascalCase`
  - メソッド名・変数名は `snake_case`
  - 定数は `UPPER_SNAKE_CASE`
- [ ] **Go**: `gofmt` + `golangci-lint` を通している
  - 公開する関数・型は `PascalCase`
  - プライベートな関数・型は `camelCase`
- [ ] **TypeScript**: `Prettier` + `ESLint` を通している
- [ ] **Dart**: `dart format` を通している
- [ ] **proto**: フィールド名は `lower_snake_case`、メッセージ名・サービス名は `PascalCase`

### エラーハンドリング

- [ ] **エラーを握りつぶしていない**
  - `unwrap()` を本番コードで使っていない（テストコードは OK）
  - `?` 演算子で適切にエラーを伝播している
- [ ] **エラーメッセージが具体的**
  - 「何が」「なぜ」失敗したかがメッセージからわかる
  - 例: `"Category 'account' not found"` は OK、`"not found"` は不十分
- [ ] **gRPC ステータスコードが適切**
  - 入力不正 → `INVALID_ARGUMENT`
  - 未認証 → `UNAUTHENTICATED`
  - 権限不足 → `PERMISSION_DENIED`
  - 未検出 → `NOT_FOUND`
  - 重複 → `ALREADY_EXISTS`
  - 内部エラー → `INTERNAL`

### その他

- [ ] **ログを適切に追加している**
  - 重要な操作（作成・更新・削除）にログを入れている
  - ログレベルが適切（通常操作は `info`、デバッグ用は `debug`、異常は `warn` / `error`）
- [ ] **コメントが必要な箇所にコメントを書いている**
  - 「なぜそうしたか」がわかるコメント（何をしているかはコードから読める）

## PR 提出前チェック

PR を作成する前に確認する最終チェックです。全て通っていない場合、PR は出さないでください。

### テスト

- [ ] **全てのテストが通る**
  ```bash
  cargo test
  ```
- [ ] **新しく追加した機能のテストを書いた**
  - 正常系のテストがある
  - 異常系（エラーケース）のテストがある
  - 境界値のテストがある（該当する場合）
- [ ] **DB テストが通る**（DB を使う変更の場合）
  ```bash
  cargo test --features db-tests
  ```

### lint / フォーマット

- [ ] **lint が通る**
  ```bash
  # Rust
  cargo clippy -- -D warnings

  # Go
  golangci-lint run
  ```
- [ ] **フォーマットが整っている**
  ```bash
  # Rust
  cargo fmt -- --check

  # Go
  gofmt -d .
  ```
- [ ] **proto の lint が通る**（proto を変更した場合）
  ```bash
  buf lint
  ```

### コミット

- [ ] **コミットメッセージが Conventional Commits 形式に従っている**
  - `feat:`, `fix:`, `chore:`, `docs:`, `refactor:`, `test:` のいずれかで始まる
- [ ] **1つのコミットが1つの論理的なまとまりになっている**
  - 「proto 変更」と「ハンドラー実装」が1つのコミットに混ざっていない
- [ ] **コミットに不要なファイルが含まれていない**
  ```bash
  git diff --stat origin/main...HEAD
  ```

### 不要ファイルの除外

- [ ] **デバッグ用のコードが残っていない**
  - `println!` や `dbg!` が本番コードに残っていない
  - `TODO` や `FIXME` コメントを確認した
- [ ] **機密情報が含まれていない**
  - `.env` ファイル、パスワード、API キーがコミットに含まれていない
- [ ] **自動生成ファイルが正しい状態になっている**
  - proto から生成されたコードが最新になっている

### 設計書との整合

- [ ] **proto 定義が設計書に沿っている**
- [ ] **API のリクエスト・レスポンス形式が設計書と一致している**
- [ ] **DB スキーマの変更がある場合、マイグレーションを作成した**

## レビュー対応チェック

レビューで指摘を受けた後に確認する項目です。

- [ ] **全ての指摘に対応した（または議論して合意した）**
  - 各コメントに返信している
  - 「対応しない」と判断した場合は理由を説明している
- [ ] **指摘に対応するコミットを追加した**
  - 新しいコミットとして追加している（amend していない）
  - コミットメッセージに何を修正したか書いている
- [ ] **再テストした**
  - 修正後に `cargo test` を実行して全テストが通ることを確認した
  - 修正が他の部分に影響していないか確認した
- [ ] **lint / フォーマットを再確認した**
  ```bash
  cargo fmt -- --check
  cargo clippy -- -D warnings
  ```

---

## マージ後チェック

PR がマージされた後に確認する項目です。

- [ ] **CI が通っている**
  - GitHub Actions の全てのジョブが緑（成功）になっている
  - テスト、lint、ビルドが全て通っている
- [ ] **main ブランチの動作を確認した**
  ```bash
  git checkout main
  git pull origin main
  cargo build
  cargo test
  ```
- [ ] **ローカルの作業ブランチを削除した**
  ```bash
  git branch -d feat/add-update-category-description-api
  ```
- [ ] **関連する Issue を閉じた**（該当する場合）

---

## チェックリストの使い方

1. PR を作成する前に、このページを開く
2. 該当するフェーズのチェック項目を上から順に確認する
3. 全てにチェックが入ったら PR を作成する
4. レビュー指摘対応後も、レビュー対応チェックを実行する

チェック漏れによる差し戻しを防ぐことで、自分もレビュアーも時間を節約できます。
