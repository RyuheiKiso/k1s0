# 開発環境セットアップ

## 必要なツール

### 共通ツール（全担当者必須）

| ツール | バージョン | 用途 |
| --- | --- | --- |
| Git | 最新 | バージョン管理、sparse-checkout |
| Docker / Docker Compose | 最新 | ローカル環境の構築 |
| k1s0 CLI | 最新 | ひな形生成、sparse-checkout、設定管理 |
| buf | 最新 | Protocol Buffers のリンティング・コード生成 |
| protoc | 3.x | Protocol Buffers コンパイラ |
| PostgreSQL Client | 17 | DB 接続・マイグレーション確認 |

### 言語別ツールチェイン

担当する言語に応じて以下をインストールする。

| 言語 | ツール | 補足 |
| --- | --- | --- |
| Rust | `rustup` (1.93+), `cargo`, `sqlx-cli` | `sqlx-cli` はマイグレーション管理に使用 |
| Go | `go` (1.24+), `golang-migrate` | `golang-migrate` はマイグレーション管理に使用 |
| TypeScript | `node` (22+), `pnpm` | React コンポーネント開発用 |
| Dart | `dart`, `flutter` (3.24+) | Flutter コンポーネント開発用 |

## k1s0 CLI のインストール

```bash
# リポジトリをクローン後、リポジトリルートで実行
cargo install --path CLI/crates/k1s0-cli

# インストール確認
k1s0 --version
```

devcontainer 利用者はコンテナ起動時に自動インストールされるため不要。

## sparse-checkout で必要な領域だけをチェックアウト

k1s0 はモノリポであり、全体をチェックアウトすると大量のコードが取得される。k1s0 CLI の対話式ウィザードで担当領域のみを sparse-checkout する。

### 手順

```bash
# k1s0 CLI で対話式ウィザードを起動
k1s0 init
# または: k1s0 → よく使う操作 > プロジェクト初期化
```

ウィザードで以下を選択する:
1. プロジェクト名
2. sparse-checkout の有無 → 有効にする
3. チェックアウトする Tier（business を選択）

依存する system tier のライブラリは自動的に含まれる。

### sparse-checkout の構成例（経理領域の Rust サーバー開発者）

```
regions/system/library/rust/       # system tier の Rust ライブラリ
regions/system/server/rust/        # system tier の Rust サーバー（参照用）
regions/business/accounting/       # 担当領域の全ファイル
docs/                              # 設計書
```

## docker-compose によるローカル環境

business tier の開発には system tier の依存サービスが必要になる。docker-compose で一括起動できる。

### 起動手順

```bash
# リポジトリルートで実行（認証バイパス付き開発環境）
just local-up-dev

# 起動確認
docker compose ps

# マイグレーション実行（Rust の場合）
just migrate regions/business/accounting/database/domain-master-db

# マイグレーション実行（Go の場合）
migrate -path regions/business/accounting/database/domain-master-db/migrations \
  -database "postgresql://dev:dev@localhost:5432/domain_master" up
```

### 主要な依存サービス

| サービス | ポート | 用途 |
| --- | --- | --- |
| PostgreSQL | 5432 | データベース |
| Keycloak | 8080 | 認証・認可（開発用レルム設定済み） |
| Kafka | 9092 | メッセージング |
| Schema Registry | 8081 | Kafka スキーマ管理 |
| OpenTelemetry Collector | 4317 (gRPC) | トレース・メトリクス収集 |
| Jaeger | 16686 (UI) | トレース可視化 |

## k1s0 CLI によるひな形生成

k1s0 CLI の対話式ウィザードで、新しいサーバー・ライブラリ・クライアントのひな形を生成できる。

```bash
# ひな形生成ウィザードを起動
k1s0 generate
# または: k1s0 → よく使う操作 > ひな形生成
```

対話プロンプトで以下を選択する:
1. 生成するコンポーネント（server / library / client / migration）
2. Tier（business）
3. 言語（Rust / Go / TypeScript / Dart）
4. サービス名・ドメイン名

生成されるコードは以下の規約に従う:

- クリーンアーキテクチャの4層構成（domain / usecase / adapter / infrastructure）
- gRPC サービス定義の雛形（proto ファイル）
- テストの雛形（単体テスト + 結合テスト）
- CI/CD パイプラインの設定ファイル
- Dockerfile と Helm チャート

## 設定ファイル（config.yaml）の扱い

各サーバーは `config/config.yaml` に設定を持つ。

### 設定ファイルの配置場所

```
regions/business/{領域名}/server/{lang}/{サーバー名}/config/
├── config.yaml          # デフォルト設定
├── config.dev.yaml      # 開発環境オーバーライド
└── config.test.yaml     # テスト環境オーバーライド
```

### 設定の読み込み

system tier の `config` ライブラリを使って設定を読み込む。環境変数による上書きにも対応している。

```rust
// Rust の場合
use k1s0_config::Config;

let config = Config::load("config/config.yaml")?;
```

```go
// Go の場合
import "github.com/k1s0/system/library/go/config"

cfg, err := config.Load("config/config.yaml")
```

### 設定の優先順位

1. 環境変数（`K1S0_` プレフィックス）
2. 環境別設定ファイル（`config.{env}.yaml`）
3. デフォルト設定ファイル（`config.yaml`）
4. system tier の config サーバーから取得した値
