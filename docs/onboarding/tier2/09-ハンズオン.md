# ハンズオン: ゼロから1つの機能を追加する

このドキュメントでは、taskmanagement 領域の project-master サーバーに「ステータス定義の説明文を更新する API」を追加する手順を、ステップバイステップで解説します。

---

## シナリオ

**やること**: `UpdateStatusDefinitionDescription` という gRPC RPC を追加する。既存の `UpdateStatusDefinition` はステータス定義の全フィールドを更新できるが、今回は「説明文だけを更新する」専用の軽量 API を新たに追加する。

**完成後の動作**: gRPC クライアントが `UpdateStatusDefinitionDescription` を呼び出すと、指定したステータス定義の `description` フィールドだけが更新される。

### 対象ディレクトリ

```
regions/business/taskmanagement/server/rust/project-master/   ← サーバー本体
api/proto/k1s0/business/taskmanagement/projectmaster/v1/      ← proto 定義
regions/business/taskmanagement/database/project-master-db/   ← DB マイグレーション
```

## Step 2: proto 定義に RPC を追加する

**編集ファイル**: `api/proto/k1s0/business/taskmanagement/projectmaster/v1/project_master.proto`

### 2-1. service ブロックに RPC を追加

`service ProjectMasterService` ブロックの中に、新しい RPC を追加します。

```protobuf
service ProjectMasterService {
  // ... 既存の RPC ...

  // 【追加】ステータス定義の説明文を更新する
  rpc UpdateStatusDefinitionDescription(UpdateStatusDefinitionDescriptionRequest)
      returns (UpdateStatusDefinitionDescriptionResponse);
}
```

### 2-2. リクエスト・レスポンスのメッセージ型を追加

ファイルの末尾付近に、新しいメッセージ型を追加します。

```protobuf
// ---------- ステータス定義説明文更新 ----------

message UpdateStatusDefinitionDescriptionRequest {
  string code = 1;           // 更新対象のステータス定義コード
  string description = 2;    // 新しい説明文
}

message UpdateStatusDefinitionDescriptionResponse {
  StatusDefinition status_definition = 1;  // 更新後のステータス定義
}
```

ポイント:
- フィールド番号（`= 1`, `= 2`）は、既存のメッセージ型と混同しないように注意
- 新しいメッセージ型なので、番号は 1 から始めてよい

今回のシナリオでは、説明文の更新は既存の `UpdateStatusDefinition` 構造体で対応可能です。ただし、説明文のみを更新する専用の入力型を作ると意図が明確になります。

**編集ファイル**: `regions/business/taskmanagement/server/rust/project-master/src/domain/entity/status_definition.rs`

```rust
#[derive(Debug, Clone, Serialize, Deserialize)]
pub struct UpdateStatusDefinitionDescription {
    pub description: String,
}
```

また、ドメイン層の `mod.rs` でエクスポートが必要な場合は更新します。

## Step 5: usecase 層にユースケースを追加

**編集ファイル**: `regions/business/taskmanagement/server/rust/project-master/src/usecase/manage_status_definitions.rs`

既存の `ManageStatusDefinitionsUseCase` に新しいメソッドを追加します。

```rust
use crate::domain::entity::status_definition::{
    CreateStatusDefinition, StatusDefinition, UpdateStatusDefinition,
    UpdateStatusDefinitionDescription,  // ← 追加
};

impl ManageStatusDefinitionsUseCase {
    // ... 既存のメソッド ...

    /// ステータス定義の説明文のみを更新する。
    pub async fn update_status_definition_description(
        &self,
        code: &str,
        input: &UpdateStatusDefinitionDescription,
    ) -> anyhow::Result<StatusDefinition> {
        // 1. ステータス定義が存在するか確認
        self.status_definition_repo
            .find_by_code(code)
            .await?
            .ok_or_else(|| anyhow::anyhow!("StatusDefinition '{}' not found", code))?;

        // 2. 説明文のみを更新（UpdateStatusDefinition を使い、他のフィールドは None）
        let update = UpdateStatusDefinition {
            display_name: None,
            description: Some(input.description.clone()),
            color: None,
            is_terminal: None,
            sort_order: None,
        };
        self.status_definition_repo.update(code, &update).await
    }
}
```

ポイント:
- ユースケースは domain 層のリポジトリ trait だけを使う（SQL は書かない）
- 存在チェックをユースケース側で行うのがパターン
- 既存の `update` メソッドを再利用し、説明文以外は `None` を渡す

## Step 7: adapter 層に gRPC ハンドラーを追加

**編集ファイル**: `regions/business/taskmanagement/server/rust/project-master/src/adapter/grpc/project_master_grpc.rs`

gRPC サービスの trait 実装に新しいメソッドを追加します。

```rust
use crate::domain::entity::status_definition::UpdateStatusDefinitionDescription;

// ProjectMasterService trait の実装に以下を追加
async fn update_status_definition_description(
    &self,
    request: tonic::Request<UpdateStatusDefinitionDescriptionRequest>,
) -> Result<tonic::Response<UpdateStatusDefinitionDescriptionResponse>, tonic::Status> {
    let req = request.into_inner();

    // 入力バリデーション
    if req.code.is_empty() {
        return Err(tonic::Status::invalid_argument("code is required"));
    }

    // ユースケースを呼び出す
    let input = UpdateStatusDefinitionDescription {
        description: req.description,
    };
    let status_def = self
        .manage_status_definitions_uc
        .update_status_definition_description(&req.code, &input)
        .await
        .map_err(|e| tonic::Status::internal(e.to_string()))?;

    // レスポンスを構築
    Ok(tonic::Response::new(UpdateStatusDefinitionDescriptionResponse {
        status_definition: Some(to_proto_status_definition(&status_def)),  // ドメインモデル → proto 変換
    }))
}
```

ポイント:
- `tonic::Request` からリクエストデータを取り出す
- 入力バリデーションはアダプター層で行う
- エラーは `tonic::Status` に変換する
- ドメインモデルから proto のメッセージ型に変換してレスポンスを返す

## Step 8: 単体テストを書く

**編集ファイル**: `regions/business/taskmanagement/server/rust/project-master/src/usecase/manage_status_definitions.rs`（ファイル末尾に追加）

```rust
#[cfg(test)]
mod tests {
    use super::*;
    use crate::domain::repository::status_definition_repository::StatusDefinitionRepository;
    use mockall::mock;
    use std::sync::Arc;

    // StatusDefinitionRepository のモックを作成
    mock! {
        StatusDefinitionRepo {}

        #[async_trait::async_trait]
        impl StatusDefinitionRepository for StatusDefinitionRepo {
            async fn find_all(&self, project_type_code: &str) -> anyhow::Result<Vec<StatusDefinition>>;
            async fn find_by_code(&self, code: &str) -> anyhow::Result<Option<StatusDefinition>>;
            async fn find_by_id(&self, id: uuid::Uuid) -> anyhow::Result<Option<StatusDefinition>>;
            async fn create(&self, input: &CreateStatusDefinition, created_by: &str) -> anyhow::Result<StatusDefinition>;
            async fn update(&self, code: &str, input: &UpdateStatusDefinition) -> anyhow::Result<StatusDefinition>;
            async fn delete(&self, code: &str) -> anyhow::Result<()>;
        }
    }

    fn sample_status_definition() -> StatusDefinition {
        StatusDefinition {
            id: uuid::Uuid::new_v4(),
            code: "in-progress".to_string(),
            display_name: "進行中".to_string(),
            description: Some("更新後の説明文".to_string()),
            color: Some("#0052CC".to_string()),
            is_terminal: false,
            sort_order: 2,
            created_by: "test-user".to_string(),
            created_at: chrono::Utc::now(),
            updated_at: chrono::Utc::now(),
        }
    }

    #[tokio::test]
    async fn test_update_status_definition_description_success() {
        // Arrange: モックの設定
        let mut mock_repo = MockStatusDefinitionRepo::new();

        // find_by_code が既存のステータス定義を返すように設定
        mock_repo
            .expect_find_by_code()
            .with(mockall::predicate::eq("in-progress"))
            .returning(|_| Ok(Some(sample_status_definition())));

        // update が成功するように設定
        mock_repo
            .expect_update()
            .returning(|_, _| Ok(sample_status_definition()));

        let uc = ManageStatusDefinitionsUseCase::new(Arc::new(mock_repo));

        // Act: テスト対象を実行
        let input = UpdateStatusDefinitionDescription {
            description: "更新後の説明文".to_string(),
        };
        let result = uc.update_status_definition_description("in-progress", &input).await;

        // Assert: 結果を検証
        assert!(result.is_ok());
        let status_def = result.unwrap();
        assert_eq!(status_def.code, "in-progress");
    }

    #[tokio::test]
    async fn test_update_status_definition_description_not_found() {
        // Arrange: ステータス定義が見つからない場合
        let mut mock_repo = MockStatusDefinitionRepo::new();
        mock_repo
            .expect_find_by_code()
            .returning(|_| Ok(None));  // 見つからない

        let uc = ManageStatusDefinitionsUseCase::new(Arc::new(mock_repo));

        // Act
        let input = UpdateStatusDefinitionDescription {
            description: "何かの説明".to_string(),
        };
        let result = uc.update_status_definition_description("nonexistent", &input).await;

        // Assert: エラーになること
        assert!(result.is_err());
        let err_msg = result.unwrap_err().to_string();
        assert!(err_msg.contains("not found"));
    }
}
```

## Step 9: マイグレーションを追加（必要なら）

今回のシナリオでは DB スキーマの変更は不要です（`description` カラムは既に `status_definitions` テーブルにある）。

もしスキーマ変更が必要な場合は、以下の手順で追加します。

**ディレクトリ**: `regions/business/taskmanagement/database/project-master-db/migrations/`

```bash
# 新しいマイグレーションファイルを作成
# ファイル名の番号は既存の最大番号 + 1 にする
touch regions/business/taskmanagement/database/project-master-db/migrations/003_add_some_column.up.sql
touch regions/business/taskmanagement/database/project-master-db/migrations/003_add_some_column.down.sql
```

up.sql（適用）:
```sql
ALTER TABLE project_master.status_definitions
ADD COLUMN new_column_name VARCHAR(255);
```

down.sql（取り消し）:
```sql
ALTER TABLE project_master.status_definitions
DROP COLUMN new_column_name;
```

ルール:
- up.sql と down.sql は必ずペアで作る
- down.sql を適用すると up.sql の変更が元に戻ること
- 番号は連番にする（飛ばさない）

## 作業の全体像まとめ

```
proto 定義          → API の仕様を決める
  ↓
コード生成          → proto から Rust コードを生成
  ↓
domain 層           → 入力データの型を追加
  ↓
usecase 層          → ビジネスロジックを実装
  ↓
infrastructure 層   → 必要なら DB 操作を追加
  ↓
adapter 層          → gRPC ハンドラーを実装
  ↓
テスト             → 正常系・異常系のテストを書く
  ↓
動作確認            → ローカルで実際に動かして確認
  ↓
PR                 → レビューへ
```

この流れは、新しい API を追加するときの基本パターンです。別の機能を追加する場合も、同じ流れで進められます。
