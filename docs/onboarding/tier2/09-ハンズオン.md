# ハンズオン: ゼロから1つの機能を追加する

このドキュメントでは、accounting 領域の domain-master サーバーに「カテゴリの説明文を更新する API」を追加する手順を、ステップバイステップで解説します。

---

## シナリオ

**やること**: `UpdateCategoryDescription` という gRPC RPC を追加する。既存の `UpdateCategory` はカテゴリの全フィールドを更新できるが、今回は「説明文だけを更新する」専用の軽量 API を新たに追加する。

**完成後の動作**: gRPC クライアントが `UpdateCategoryDescription` を呼び出すと、指定したカテゴリの `description` フィールドだけが更新される。

### 対象ディレクトリ

```
regions/business/accounting/server/rust/domain-master/   ← サーバー本体
api/proto/k1s0/business/accounting/domainmaster/v1/      ← proto 定義
regions/business/accounting/database/domain-master-db/   ← DB マイグレーション
```

## Step 2: proto 定義に RPC を追加する

**編集ファイル**: `api/proto/k1s0/business/accounting/domainmaster/v1/domain_master.proto`

### 2-1. service ブロックに RPC を追加

`service DomainMasterService` ブロックの中に、新しい RPC を追加します。

```protobuf
service DomainMasterService {
  // ... 既存の RPC ...

  // 【追加】カテゴリの説明文を更新する
  rpc UpdateCategoryDescription(UpdateCategoryDescriptionRequest)
      returns (UpdateCategoryDescriptionResponse);
}
```

### 2-2. リクエスト・レスポンスのメッセージ型を追加

ファイルの末尾付近に、新しいメッセージ型を追加します。

```protobuf
// ---------- カテゴリ説明文更新 ----------

message UpdateCategoryDescriptionRequest {
  string code = 1;           // 更新対象のカテゴリコード
  string description = 2;    // 新しい説明文
}

message UpdateCategoryDescriptionResponse {
  MasterCategory category = 1;  // 更新後のカテゴリ
}
```

ポイント:
- フィールド番号（`= 1`, `= 2`）は、既存のメッセージ型と混同しないように注意
- 新しいメッセージ型なので、番号は 1 から始めてよい

今回のシナリオでは、説明文の更新は既存の `UpdateMasterCategory` 構造体で対応可能です。ただし、説明文のみを更新する専用の入力型を作ると意図が明確になります。

**編集ファイル**: `regions/business/accounting/server/rust/domain-master/src/domain/entity/master_category.rs`

```rust
pub struct UpdateCategoryDescription {
    pub description: String,
}
```

また、ドメイン層の `mod.rs` でエクスポートが必要な場合は更新します。

## Step 5: usecase 層にユースケースを追加

**編集ファイル**: `regions/business/accounting/server/rust/domain-master/src/usecase/manage_categories.rs`

既存の `ManageCategoriesUseCase` に新しいメソッドを追加します。

```rust
use crate::domain::entity::master_category::{
    CreateMasterCategory, MasterCategory, UpdateMasterCategory,
    UpdateCategoryDescription,  // ← 追加
};

impl ManageCategoriesUseCase {
    // ... 既存のメソッド ...

    /// カテゴリの説明文のみを更新する。
    pub async fn update_category_description(
        &self,
        code: &str,
        input: &UpdateCategoryDescription,
    ) -> anyhow::Result<MasterCategory> {
        // 1. カテゴリが存在するか確認
        self.category_repo
            .find_by_code(code)
            .await?
            .ok_or_else(|| anyhow::anyhow!("Category '{}' not found", code))?;

        // 2. 説明文のみを更新（UpdateMasterCategory を使い、他のフィールドは None）
        let update = UpdateMasterCategory {
            display_name: None,
            description: Some(input.description.clone()),
            validation_schema: None,
            is_active: None,
            sort_order: None,
        };
        self.category_repo.update(code, &update).await
    }
}
```

ポイント:
- ユースケースは domain 層のリポジトリ trait だけを使う（SQL は書かない）
- 存在チェックをユースケース側で行うのがパターン
- 既存の `update` メソッドを再利用し、説明文以外は `None` を渡す

## Step 7: adapter 層に gRPC ハンドラーを追加

**編集ファイル**: `regions/business/accounting/server/rust/domain-master/src/adapter/grpc/domain_master_grpc.rs`

gRPC サービスの trait 実装に新しいメソッドを追加します。

```rust
use crate::domain::entity::master_category::UpdateCategoryDescription;

// DomainMasterService trait の実装に以下を追加
async fn update_category_description(
    &self,
    request: tonic::Request<UpdateCategoryDescriptionRequest>,
) -> Result<tonic::Response<UpdateCategoryDescriptionResponse>, tonic::Status> {
    let req = request.into_inner();

    // 入力バリデーション
    if req.code.is_empty() {
        return Err(tonic::Status::invalid_argument("code is required"));
    }

    // ユースケースを呼び出す
    let input = UpdateCategoryDescription {
        description: req.description,
    };
    let category = self
        .manage_categories_uc
        .update_category_description(&req.code, &input)
        .await
        .map_err(|e| tonic::Status::internal(e.to_string()))?;

    // レスポンスを構築
    Ok(tonic::Response::new(UpdateCategoryDescriptionResponse {
        category: Some(to_proto_category(&category)),  // ドメインモデル → proto 変換
    }))
}
```

ポイント:
- `tonic::Request` からリクエストデータを取り出す
- 入力バリデーションはアダプター層で行う
- エラーは `tonic::Status` に変換する
- ドメインモデルから proto のメッセージ型に変換してレスポンスを返す

## Step 8: 単体テストを書く

**編集ファイル**: `regions/business/accounting/server/rust/domain-master/src/usecase/manage_categories.rs`（ファイル末尾に追加）

```rust
#[cfg(test)]
mod tests {
    use super::*;
    use crate::domain::repository::category_repository::CategoryRepository;
    use mockall::mock;
    use std::sync::Arc;

    // CategoryRepository のモックを作成
    mock! {
        CategoryRepo {}

        #[async_trait::async_trait]
        impl CategoryRepository for CategoryRepo {
            async fn find_all(&self, active_only: bool) -> anyhow::Result<Vec<MasterCategory>>;
            async fn find_by_code(&self, code: &str) -> anyhow::Result<Option<MasterCategory>>;
            async fn find_by_id(&self, id: uuid::Uuid) -> anyhow::Result<Option<MasterCategory>>;
            async fn create(&self, input: &CreateMasterCategory, created_by: &str) -> anyhow::Result<MasterCategory>;
            async fn update(&self, code: &str, input: &UpdateMasterCategory) -> anyhow::Result<MasterCategory>;
            async fn delete(&self, code: &str) -> anyhow::Result<()>;
        }
    }

    fn sample_category() -> MasterCategory {
        MasterCategory {
            id: uuid::Uuid::new_v4(),
            code: "account".to_string(),
            display_name: "勘定科目".to_string(),
            description: Some("更新後の説明文".to_string()),
            validation_schema: None,
            is_active: true,
            sort_order: 0,
            created_by: "test-user".to_string(),
            created_at: chrono::Utc::now(),
            updated_at: chrono::Utc::now(),
        }
    }

    #[tokio::test]
    async fn test_update_category_description_success() {
        // Arrange: モックの設定
        let mut mock_repo = MockCategoryRepo::new();

        // find_by_code が既存カテゴリを返すように設定
        mock_repo
            .expect_find_by_code()
            .with(mockall::predicate::eq("account"))
            .returning(|_| Ok(Some(sample_category())));

        // update が成功するように設定
        mock_repo
            .expect_update()
            .returning(|_, _| Ok(sample_category()));

        let uc = ManageCategoriesUseCase::new(Arc::new(mock_repo));

        // Act: テスト対象を実行
        let input = UpdateCategoryDescription {
            description: "更新後の説明文".to_string(),
        };
        let result = uc.update_category_description("account", &input).await;

        // Assert: 結果を検証
        assert!(result.is_ok());
        let category = result.unwrap();
        assert_eq!(category.code, "account");
    }

    #[tokio::test]
    async fn test_update_category_description_not_found() {
        // Arrange: カテゴリが見つからない場合
        let mut mock_repo = MockCategoryRepo::new();
        mock_repo
            .expect_find_by_code()
            .returning(|_| Ok(None));  // 見つからない

        let uc = ManageCategoriesUseCase::new(Arc::new(mock_repo));

        // Act
        let input = UpdateCategoryDescription {
            description: "何かの説明".to_string(),
        };
        let result = uc.update_category_description("nonexistent", &input).await;

        // Assert: エラーになること
        assert!(result.is_err());
        let err_msg = result.unwrap_err().to_string();
        assert!(err_msg.contains("not found"));
    }
}
```

テストを実行:

## Step 9: マイグレーションを追加（必要なら）

今回のシナリオでは DB スキーマの変更は不要です（`description` カラムは既に `master_categories` テーブルにある）。

もしスキーマ変更が必要な場合は、以下の手順で追加します。

**ディレクトリ**: `regions/business/accounting/database/domain-master-db/migrations/`

```bash
# 新しいマイグレーションファイルを作成
# ファイル名の番号は既存の最大番号 + 1 にする
touch regions/business/accounting/database/domain-master-db/migrations/003_add_some_column.up.sql
touch regions/business/accounting/database/domain-master-db/migrations/003_add_some_column.down.sql
```

up.sql（適用）:
```sql
ALTER TABLE domain_master.master_categories
ADD COLUMN new_column_name VARCHAR(255);
```

down.sql（取り消し）:
```sql
ALTER TABLE domain_master.master_categories
DROP COLUMN new_column_name;
```

ルール:
- up.sql と down.sql は必ずペアで作る
- down.sql を適用すると up.sql の変更が元に戻ること
- 番号は連番にする（飛ばさない）

## 作業の全体像まとめ

```
proto 定義          → API の仕様を決める
  ↓
コード生成          → proto から Rust コードを生成
  ↓
domain 層           → 入力データの型を追加
  ↓
usecase 層          → ビジネスロジックを実装
  ↓
infrastructure 層   → 必要なら DB 操作を追加
  ↓
adapter 層          → gRPC ハンドラーを実装
  ↓
テスト             → 正常系・異常系のテストを書く
  ↓
動作確認            → ローカルで実際に動かして確認
  ↓
PR                 → レビューへ
```

この流れは、新しい API を追加するときの基本パターンです。別の機能を追加する場合も、同じ流れで進められます。
