# よくあるミスとトラブルシューティング

開発中に遭遇しやすい問題を「症状 → 原因 → 解決方法」の形式でまとめています。エラーに遭遇したら、まずここを検索してみてください。

> **開発環境のトラブル（CRLF・Docker・WSL2・ポート競合等）は以下を参照:**
> [`docs/infrastructure/devenv/troubleshooting.md`](../../infrastructure/devenv/troubleshooting.md)

## ビルド関連

### cargo build が通らない（型エラー）

**症状**:
```
error[E0308]: mismatched types
  expected `String`, found `&str`
```

**原因**: Rust では `String`（所有する文字列）と `&str`（借用する文字列スライス）は異なる型。

**解決方法**:
```rust
// &str → String に変換
let s: String = "hello".to_string();
// または
let s: String = String::from("hello");

// String → &str に変換
let s: &str = &my_string;
```

---

### cargo build が通らない（Option 関連のエラー）

**症状**:
```
error[E0599]: no method named `len` found for enum `Option<String>`
```

**原因**: `Option<String>` は「値があるかないか」を表す型で、中身の `String` のメソッドを直接呼べない。

**解決方法**:
```rust
// パターンマッチで中身を取り出す
match my_option {
    Some(value) => println!("{}", value.len()),
    None => println!("値がありません"),
}

// または if let を使う
if let Some(value) = my_option {
    println!("{}", value.len());
}

// デフォルト値を使う場合
let len = my_option.map(|v| v.len()).unwrap_or(0);
```

---

### 依存解決エラー

**症状**:
```
error: failed to select a version for `some-crate`
```

**原因**: 複数のライブラリが異なるバージョンの同じクレートを要求している。

**解決方法**:
```bash
# 依存ツリーを確認
cargo tree -d

# 問題のクレートがどこから要求されているか確認
cargo tree -i some-crate

# Cargo.toml で明示的にバージョンを指定して解決
```

---

### buf lint 失敗

**症状**:
```
api/proto/k1s0/.../domain_master.proto:42:3:
  FIELD_LOWER_SNAKE_CASE: Field name "displayName" should be lower_snake_case
```

**原因**: proto ファイルのフィールド名が命名規約に従っていない。

**解決方法**:
- proto のフィールド名は `lower_snake_case`（小文字とアンダースコア）を使う
- サービス名は `PascalCase`、RPC 名も `PascalCase`
- メッセージ名は `PascalCase`

```protobuf
// NG
string displayName = 1;

// OK
string display_name = 1;
```

## DB 関連

### PostgreSQL 接続エラー

**症状**:
```
Error: error communicating with database: Connection refused (os error 111)
```

**原因**: PostgreSQL が起動していない、または接続情報が間違っている。

**解決方法**:
```bash
# 1. PostgreSQL が起動しているか確認
docker-compose ps

# 2. 起動していなければ起動
docker-compose up -d postgres

# 3. 接続情報を確認（config.yaml と一致しているか）
# config.yaml:
#   database:
#     host: "localhost"
#     port: 5432
#     name: "k1s0"
#     user: "k1s0"
#     password: "k1s0"

# 4. 直接接続を試す
psql -h localhost -p 5432 -U k1s0 -d k1s0
```

---

### マイグレーション失敗

**症状**:
```
Error: while executing migrations: error returned from database: relation "master_categories" already exists
```

**原因**: マイグレーションが途中で失敗し、再実行時にテーブルが既に存在する。

**解決方法**:
```bash
# 1. マイグレーションの状態を確認
# sqlx-cli を使っている場合:
sqlx migrate info --source regions/business/accounting/database/domain-master-db/migrations/

# 2. DB を完全にリセットする場合（開発環境のみ!）
docker-compose down -v  # ボリュームも削除
docker-compose up -d postgres
# サーバーを再起動すればマイグレーションが最初から実行される

# 3. 特定のマイグレーションを手動で取り消す場合
psql -h localhost -p 5432 -U k1s0 -d k1s0 -f \
  regions/business/accounting/database/domain-master-db/migrations/002_create_domain_master_tables.down.sql
```

---

### sqlx のコンパイル時検証エラー

**症状**:
```
error: error returned from database: column "new_column" does not exist
  --> src/infrastructure/persistence/category_repo_impl.rs:25:13
```

**原因**: `sqlx::query!` マクロや `sqlx::query_as!` マクロは、コンパイル時に実際の DB に接続して SQL を検証する。DB のスキーマとコードが一致していない。

**解決方法**:
```bash
# 1. マイグレーションを適用して DB スキーマを最新にする
cargo run  # サーバー起動時にマイグレーションが実行される

# 2. sqlx のオフラインモードを使う場合
# 事前に sqlx-data.json を生成しておく
cargo sqlx prepare

# 3. DATABASE_URL 環境変数が正しいか確認
echo $DATABASE_URL
# postgres://k1s0:k1s0@localhost:5432/k1s0
```

## 認証関連

### JWT 検証エラー

**症状**:
```
Status { code: Unauthenticated, message: "invalid token: InvalidSignature" }
```

**原因**: JWT トークンの署名が無効。トークンが期限切れ、または Keycloak の鍵が変わった。

**解決方法**:
```bash
# 1. 新しいトークンを取得
curl -X POST http://localhost:8080/realms/k1s0/protocol/openid-connect/token \
  -d "grant_type=client_credentials" \
  -d "client_id=k1s0-business" \
  -d "client_secret=YOUR_SECRET"

# 2. トークンの中身を確認（jwt.io で復号化するか、jq を使う）
echo $TOKEN | cut -d'.' -f2 | base64 -d | jq .

# 3. 期限（exp）が現在時刻より後であることを確認
```

---

### Keycloak 接続エラー

**症状**:
```
Error: failed to fetch JWKS: error sending request: Connection refused
```

**原因**: Keycloak が起動していない、または URL が間違っている。

**解決方法**:
```bash
# 1. Keycloak が起動しているか確認
docker-compose ps keycloak

# 2. 起動していなければ起動
docker-compose up -d keycloak

# 3. Keycloak の管理画面にアクセスできるか確認
# ブラウザで http://localhost:8080 を開く

# 4. config.yaml の auth.jwks_url が正しいか確認
# auth:
#   jwks_url: "http://localhost:8080/realms/k1s0/protocol/openid-connect/certs"
```

---

### RBAC 権限不足

**症状**:
```
Status { code: PermissionDenied, message: "insufficient permissions: required action 'admin', user has 'read'" }
```

**原因**: JWT トークンに含まれるロールに、要求されたアクション（read / write / admin）の権限がない。

**解決方法**:
1. Keycloak の管理画面でユーザーのロールを確認する
2. 必要な権限を持つロールを割り当てる
3. `main.rs` の `required_action` 関数で、どの RPC にどのアクションが必要か確認する

```rust
// main.rs より
fn required_action(method: &str) -> &'static str {
    match method {
        "ListCategories" | "GetCategory" | ... => "read",
        "DeleteCategory" | "DeleteItem" | ... => "admin",
        _ => "write",  // デフォルトは write
    }
}
```

## gRPC 関連

### proto 未生成

**症状**:
```
error[E0433]: failed to resolve: could not find `UpdateCategoryDescriptionRequest` in `proto`
```

**原因**: proto ファイルを変更したが、コード生成を実行していない。

**解決方法**:
```bash
# cargo build が tonic-build を呼ぶ（build.rs 経由）
cargo build

# それでも解決しない場合、キャッシュをクリアして再ビルド
cargo clean
cargo build
```

---

### tonic 接続エラー

**症状**:
```
Error: transport error: error trying to connect: tcp connect error: Connection refused
```

**原因**: gRPC サーバーが起動していない、またはポート番号が間違っている。

**解決方法**:
```bash
# 1. サーバーが起動しているか確認
# ログに "gRPC server listening on 0.0.0.0:50061" が出ているか

# 2. ポートが使われているか確認
# Linux/macOS
lsof -i :50061
# Windows
netstat -ano | findstr 50061

# 3. grpcurl でサービス一覧を確認
grpcurl -plaintext localhost:50061 list
```

---

### gRPC ステータスコードの意味

| コード | 意味 | よくある原因 |
| --- | --- | --- |
| `OK (0)` | 成功 | - |
| `INVALID_ARGUMENT (3)` | リクエストの値が不正 | 必須フィールドが空、値の範囲外 |
| `NOT_FOUND (5)` | リソースが見つからない | 存在しない ID やコードを指定した |
| `ALREADY_EXISTS (6)` | リソースが既に存在する | 重複するコードで作成しようとした |
| `PERMISSION_DENIED (7)` | 権限不足 | RBAC のロールが不足 |
| `UNAUTHENTICATED (16)` | 未認証 | JWT トークンがない・無効・期限切れ |
| `INTERNAL (13)` | サーバー内部エラー | バグ、DB 接続失敗など |

## テスト関連

### テストが通らない

**症状**:
```
test result: FAILED. 3 passed; 1 failed; 0 ignored
```

**解決方法**:
```bash
# 失敗したテストだけを再実行（詳細出力付き）
cargo test test_update_category_description_not_found -- --nocapture

# 特定のモジュールのテストだけを実行
cargo test manage_categories

# 全テストを詳細出力で実行
cargo test -- --nocapture
```

---

### モックの書き方がわからない

**症状**: テストでリポジトリをモックにしたいが、書き方がわからない。

**解決方法**: mockall を使ったモックの基本パターン:

```rust
use mockall::mock;

// 1. モックの定義
mock! {
    CategoryRepo {}

    #[async_trait::async_trait]
    impl CategoryRepository for CategoryRepo {
        async fn find_by_code(&self, code: &str) -> anyhow::Result<Option<MasterCategory>>;
        // ... 他のメソッドも全て列挙する
    }
}

// 2. テスト内でモックを使う
#[tokio::test]
async fn test_example() {
    let mut mock = MockCategoryRepo::new();

    // 特定の引数で呼ばれたときの戻り値を設定
    mock.expect_find_by_code()
        .with(mockall::predicate::eq("account"))
        .returning(|_| Ok(Some(sample_category())));

    // 何回呼ばれてもよい場合
    mock.expect_find_by_code()
        .returning(|_| Ok(None));

    // 1回だけ呼ばれることを期待する場合
    mock.expect_find_by_code()
        .times(1)
        .returning(|_| Ok(None));
}
```

---

### DB テストの環境

**症状**: DB を使うテストを実行したいが、テスト用 DB がない。

**解決方法**:
```bash
# 1. docker-compose で PostgreSQL を起動
docker-compose up -d postgres

# 2. テスト用の環境変数を設定
export DATABASE_URL="postgres://k1s0:k1s0@localhost:5432/k1s0"

# 3. db-tests フィーチャーフラグを有効にしてテスト実行
cargo test --features db-tests

# 4. testcontainers を使っている場合は Docker が起動していれば自動で DB が用意される
```

## Docker / ローカル環境

### docker-compose up が失敗

**症状**:
```
ERROR: for postgres  Cannot start service postgres: driver failed programming external connectivity
```

**原因**: ポートが既に使われている。

**解決方法**:
```bash
# 1. 使用中のポートを確認
# Linux/macOS
lsof -i :5432
# Windows
netstat -ano | findstr 5432

# 2. 競合するプロセスを停止するか、docker-compose.yml でポートを変更

# 3. 全コンテナを停止してから再起動
docker-compose down
docker-compose up -d
```

---

### ポート競合

**症状**:
```
Error: Address already in use (os error 98)
```

**原因**: 同じポートで別のプロセスが既に起動している。

**解決方法**:
```bash
# 1. 使用中のプロセスを確認して停止
# Linux/macOS
lsof -i :8210
kill <PID>

# Windows
netstat -ano | findstr 8210
taskkill /PID <PID> /F

# 2. または config.yaml でポート番号を変更
```

---

### サービスが起動しない

**症状**: `cargo run` しても何も表示されずに終了する、またはエラーで落ちる。

**解決方法**:
```bash
# 1. 環境変数 RUST_LOG を設定して詳細ログを出す
RUST_LOG=debug cargo run

# 2. config.yaml のパスが正しいか確認
CONFIG_PATH=config/config.yaml cargo run

# 3. 依存サービス（DB, Keycloak）が起動しているか確認
docker-compose ps

# 4. 全ての依存サービスを起動
docker-compose up -d
```

## Git 関連

### コンフリクト

**症状**:
```
CONFLICT (content): Merge conflict in src/domain/entity/master_category.rs
Automatic merge failed; fix conflicts and then commit the result.
```

**解決方法**: [Git ワークフロー](./07-Gitワークフロー.md) の「コンフリクト解消の手順」を参照。

---

### 間違ったブランチにコミットしてしまった

**症状**: main ブランチにコミットしてしまった（まだ push していない）。

**解決方法**:
```bash
# 1. コミットハッシュをメモする
git log --oneline -3

# 2. main のコミットを1つ戻す（コミットは残る、変更は作業ツリーに戻る）
git reset --soft HEAD~1

# 3. 正しいブランチに切り替える
git checkout -b feat/correct-branch

# 4. 変更をコミットし直す
git add <files>
git commit -m "feat: 正しいメッセージ"
```

> **注意**: 既に push してしまった場合は、チームリーダーに相談してください。

---

### コミットの取り消し

**症状**: 直前のコミットを取り消したい（まだ push していない）。

**解決方法**:
```bash
# コミットを取り消し、変更は作業ツリーに残す
git reset --soft HEAD~1

# コミットを取り消し、変更もステージングから外す
git reset HEAD~1

# コミットを取り消し、変更も完全に破棄する（注意: 元に戻せない!）
git reset --hard HEAD~1
```

> **注意**: `--hard` は変更が完全に失われます。本当に必要な場合のみ使ってください。

## 困ったときの一般的な対処法

1. **エラーメッセージをよく読む**: Rust のエラーメッセージは非常に丁寧で、修正方法のヒントが書かれていることが多い
2. **`cargo check`**: ビルド全体より速く、型エラーだけを確認できる
3. **`RUST_LOG=debug`**: 詳細なログを出力して、どこで問題が起きているか特定する
4. **`grep` で検索**: エラーメッセージの文字列でプロジェクト内を検索し、どこから発生しているか特定する
5. **チームメンバーに聞く**: 30分以上悩んだら、遠慮なく質問する
