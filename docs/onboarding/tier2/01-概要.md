# Business Tier（領域共通層）概要

## business tier の役割と責務

business tier は、特定の業務領域（ドメイン）に閉じた共通基盤を提供する中間層である。system tier が提供する認証・設定・可観測性などの横断的な基盤の上に、業務固有のロジック・データ・UIコンポーネントを構築する。

主な責務:

- 業務領域内で共通利用するサーバー基盤の提供（ドメインマスタ管理、領域固有の業務ロジック等）
- 業務領域内で共通利用するクライアントコンポーネントの提供（共通レイアウト、ドメイン固有のUIパーツ等）
- 業務領域内で共通利用するライブラリの提供（ドメインモデル、バリデーション、ユーティリティ等）
- 業務領域固有のデータベーススキーマの管理（マスタデータ、領域内共通トランザクション等）

## 「領域」の概念

k1s0 における「領域」とは、業務ドメインの単位を指す。各領域は `regions/business/{領域名}/` として独立したディレクトリを持つ。

| 領域例 | 対象業務 | 含まれるもの |
| --- | --- | --- |
| `accounting` | 経理 | 勘定科目マスタ、仕訳共通ロジック、経理共通UI |
| `fa` | 固定資産 | 資産区分マスタ、減価償却共通ロジック、FA共通UI |

各領域は互いに独立しており、領域間の直接的な同期通信は禁止されている。領域間でデータ連携が必要な場合は Kafka による非同期メッセージングを使用する。

## 提供するもの

business tier の各領域は以下の4種別で構成される。

```
regions/business/{領域名}/
├── server/           # 領域固有の共通サーバー基盤
│   ├── rust/         #   Rust 実装（ドメインサービス）
│   └── go/           #   Go 実装（BFF 等）
├── client/           # 領域固有の共通クライアント基盤
│   ├── react/        #   React 共通コンポーネント
│   └── flutter/      #   Flutter 共通コンポーネント
├── library/          # 領域内で server・client 双方が使う共通コード
│   ├── go/
│   ├── rust/
│   ├── typescript/
│   └── dart/
└── database/         # 領域固有の共通データストア
    └── {db名}/
        └── migrations/
```

### 種別ごとの詳細

| 種別 | 役割 | 具体例 |
| --- | --- | --- |
| server | 領域固有の共通サーバー基盤 | domain-master サーバー（マスタデータCRUD、ドメインイベント配信） |
| client | 領域固有の共通クライアント基盤 | 共通レイアウト、ドメイン固有のフォーム部品、テーマ |
| library | server・client 双方で使う共通コード | ドメインモデル定義、バリデーションルール、型定義 |
| database | 領域固有の共通データストア | マスタデータ、領域内共通トランザクションデータ |

## system tier との関係（依存する側）

business tier は system tier に依存する。system tier が提供する以下の機能を利用する。

| system tier が提供するもの | 利用方式 |
| --- | --- |
| 認証トークンの検証（Keycloak + JWT） | system library を import |
| ユーザー情報の取得 | system server へ gRPC 呼出 |
| 設定情報の取得 | system library / system server へ gRPC 呼出 |
| ログ・トレース・メトリクスの送信 | system library を import |
| レート制限・ルーティング | API ゲートウェイを経由 |
| 共通UIコンポーネント・認証SDK | system client SDK を import |

system tier のデータベースへの直接アクセスは禁止されている。必要なデータは必ず system server の API を経由して取得する。

## service tier との関係（依存される側）

service tier は business tier に依存する。service tier の開発者は以下を利用する。

| business tier が提供するもの | 利用方式 |
| --- | --- |
| 領域固有のドメインモデル・バリデーション | business library を import |
| 領域共通サーバーの API | business server へ gRPC 呼出 |
| 領域共通のUIコンポーネント | business client を import |
| 領域共通のレイアウト・テーマ | business client を import |

business tier のデータベースへの直接アクセスは禁止されている。service tier は business server の API を経由してデータを取得する。

## 担当開発者に求められるスキル

### 必須スキル

| 分野 | 内容 |
| --- | --- |
| 言語 | Go / Rust / TypeScript / Dart のうち担当言語に習熟 |
| 設計 | クリーンアーキテクチャ（domain / usecase / adapter / infrastructure の4層） |
| DDD | エンティティ、値オブジェクト、集約、リポジトリ、ドメインイベントの設計・実装 |
| DB | PostgreSQL 17、マイグレーション管理（sqlx-cli / golang-migrate） |
| API | gRPC（proto / buf）、REST API の設計・実装 |
| テスト | TDD の実践、モックを活用した単体・結合テスト |

### 推奨スキル

| 分野 | 内容 |
| --- | --- |
| メッセージング | Kafka のプロデューサー・コンシューマー実装 |
| 可観測性 | OpenTelemetry によるトレース・メトリクス計装 |
| コンテナ | Docker / Kubernetes の基礎 |
| フロントエンド | React（TanStack Query, Zustand）または Flutter（Riverpod）の状態管理 |
