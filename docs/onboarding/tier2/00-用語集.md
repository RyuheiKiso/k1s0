# 用語集

このドキュメントは、k1s0 プロジェクトで使われる用語を初心者にもわかるように解説する辞書です。わからない言葉が出てきたら、まずここを引いてください。

---

## プロジェクト用語

### Tier（ティア）

- **一言で言うと**: プロジェクトの「階層」のこと。上の階層ほど共通的で、下の階層ほど個別の業務に特化する。
- **もう少し詳しく**: k1s0 では system / business / service の3つの Tier がある。上位 Tier が提供する機能を下位 Tier が使う（依存する）という一方向の関係がある。下位から上位への依存は許可されるが、同階層や上位から下位への依存は禁止。

### system tier（システムティア）

- **一言で言うと**: 全プロジェクトで共通して使う基盤部分。認証やログ収集などが含まれる。
- **もう少し詳しく**: `regions/system/` に配置される。JWT 認証、設定管理、API ゲートウェイ、可観測性（ログ・メトリクス・トレース）など、業務領域に関係なく使う機能を提供する。business tier や service tier はこの層が提供するライブラリやサーバーに依存する。

### business tier（ビジネスティア）

- **一言で言うと**: 特定の業務領域（例：経理、固定資産）の中で共通して使う基盤部分。
- **もう少し詳しく**: `regions/business/{領域名}/` に配置される。例えば経理領域なら `regions/business/accounting/` に、勘定科目マスタ管理サーバーやドメインモデルなどが置かれる。system tier の上に構築され、service tier に機能を提供する立ち位置。

### service tier（サービスティア）

- **一言で言うと**: 実際にユーザーが使う個別サービスを置く階層。
- **もう少し詳しく**: `regions/service/{サービス名}/` に配置される。business tier や system tier が提供する共通基盤の上に、具体的な業務機能（例：発注管理、在庫管理）を実装する。

### regions（リージョンズ）

- **一言で言うと**: コードが置かれる最上位のディレクトリ名。Tier ごとにサブディレクトリが切られている。
- **もう少し詳しく**: `regions/system/`、`regions/business/`、`regions/service/` の3つに分かれる。歴史的経緯でこの名前になっているが、論理的な意味は Tier と同じ。

### 領域（りょういき）

- **一言で言うと**: 業務のまとまり単位。「経理」「固定資産（FA）」などが1つの領域。
- **もう少し詳しく**: business tier の中をさらに細分化する概念。各領域は `regions/business/{領域名}/` として独立したディレクトリを持ち、server / client / library / database の4種別で構成される。領域間の直接通信は禁止されており、連携が必要な場合は Kafka（後述）による非同期メッセージングを使う。

### sparse-checkout（スパースチェックアウト）

- **一言で言うと**: モノリポの中から「自分に必要な部分だけ」をダウンロードする Git の機能。
- **もう少し詳しく**: k1s0 はモノリポ（1つの巨大なリポジトリに全コードが入っている）構成のため、全てをクローンすると非常に大きくなる。sparse-checkout を使うと、例えば `regions/business/accounting/` だけを手元に持ってくることができる。k1s0 CLI がこの設定を自動化してくれる。

### k1s0 CLI

- **一言で言うと**: プロジェクトのセットアップやコード生成を行うコマンドラインツール。
- **もう少し詳しく**: Rust 製の対話式 CLI ツール。新しいサーバーやライブラリの雛形（テンプレート）生成、sparse-checkout の設定、設定ファイルの管理などができる。`CLI/` ディレクトリに格納されている。

## アーキテクチャ用語

### クリーンアーキテクチャ

- **一言で言うと**: 「ビジネスロジック（業務の本質的なルール）を技術的な詳細（DB、通信方法など）から分離する」設計手法。
- **もう少し詳しく**: コードを4つの層（ドメイン層・ユースケース層・アダプター層・インフラストラクチャ層）に分け、内側の層が外側の層に依存しないようにする。これにより、DB を変えたり通信方法を変えたりしても、ビジネスロジック部分は影響を受けない。

### DDD（ドメイン駆動設計 / Domain-Driven Design）

- **一言で言うと**: 「業務の専門家が使う言葉をそのままコードに反映させよう」という設計アプローチ。
- **もう少し詳しく**: ソフトウェア開発で最も重要なのは「業務をどれだけ正確に理解・モデル化できるか」という考え方。業務の専門家と開発者が共通の言語（ユビキタス言語）を使い、それをそのままコードの構造体名やメソッド名に反映させる。

### TDD（テスト駆動開発 / Test-Driven Development）

- **一言で言うと**: 「先にテストを書き、そのテストが通るようにコードを書く」開発手法。
- **もう少し詳しく**: 3つのステップを繰り返す。(1) 失敗するテストを書く → (2) テストが通る最小限のコードを書く → (3) コードをきれいにする（リファクタリング）。これにより、仕様がテストとして残り、コードの品質が保たれる。

### ドメイン層

- **一言で言うと**: 業務のルールやデータ構造を定義する、最も内側の層。
- **もう少し詳しく**: `src/domain/` に配置される。エンティティ（データモデル）、値オブジェクト、リポジトリインターフェース（データの取得・保存方法の定義だけ）、ドメインサービス（複雑な業務ルール）が含まれる。外部技術（DB、HTTP）には一切依存しない。

### ユースケース層

- **一言で言うと**: 「アプリケーションとして何ができるか」を定義する層。
- **もう少し詳しく**: `src/usecase/` に配置される。ドメイン層のエンティティやリポジトリを組み合わせて、「カテゴリを作成する」「項目一覧を取得する」などのアプリケーション操作を実現する。トランザクション制御やドメインイベントの発行もここで行う。

### アダプター層

- **一言で言うと**: 外部からのリクエストを受け取り、ユースケースを呼び出す「窓口」の層。
- **もう少し詳しく**: `src/adapter/` に配置される。gRPC リクエストや REST リクエストを受け取り、ユースケース層が理解できる形に変換して呼び出す。レスポンスの変換やエラーコードの変換もここで行う。

### インフラストラクチャ層

- **一言で言うと**: DB 接続や外部サービス呼び出しなど、技術的な実装を行う層。
- **もう少し詳しく**: `src/infrastructure/` に配置される。ドメイン層で定義されたリポジトリインターフェースの「実際の実装」を書く場所。PostgreSQL への SQL 発行、Kafka へのメッセージ送信、外部 API の呼び出しなど、具体的な技術に依存するコードがここに入る。

### エンティティ

- **一言で言うと**: 一意に識別できるデータの塊。「ID を持つオブジェクト」。
- **もう少し詳しく**: 例えば `MasterCategory`（マスタカテゴリ）は `id`（UUID）を持ち、それによって一意に識別される。同じ名前のカテゴリでも ID が違えば別のエンティティとして扱う。`src/domain/entity/` に定義される。

### 値オブジェクト

- **一言で言うと**: ID を持たず、値そのものに意味があるデータ。
- **もう少し詳しく**: 例えば「金額」「メールアドレス」「日付範囲」など。2つの値オブジェクトは、中身が同じなら同一と見なす。k1s0 では `CreateMasterCategory`（カテゴリ作成リクエスト）のような入力データも値オブジェクトとして定義される。

### 集約（アグリゲート）

- **一言で言うと**: 関連するエンティティや値オブジェクトをまとめた「整合性の単位」。
- **もう少し詳しく**: 例えば「カテゴリ」と「そのカテゴリに属する項目」は強い関連がある。これらをまとめて1つの集約として扱い、データの整合性（つじつまが合っていること）を保証する。集約の外からは、集約ルート（代表となるエンティティ）を通じてのみアクセスする。

### リポジトリ

- **一言で言うと**: データの保存・取得を抽象化した「データの出入り口」。
- **もう少し詳しく**: ドメイン層では `trait CategoryRepository`（Rust）や `interface`（Go）としてインターフェースだけを定義し、実際の DB 操作はインフラストラクチャ層で実装する。これにより、テスト時にはモック（偽物のリポジトリ）に差し替えることができる。

### ドメインイベント

- **一言で言うと**: 「業務上、何かが起きた」ことを表す通知。
- **もう少し詳しく**: 例えば「カテゴリが作成された」「項目が更新された」というイベント。Kafka を使って他のサービスに通知される。イベントは過去形の名前を付ける（例: `category_changed`）。

### 境界づけられたコンテキスト（Bounded Context）

- **一言で言うと**: 「この範囲の中では、言葉の意味が統一されている」という境界。
- **もう少し詳しく**: 例えば「アカウント」という言葉は、経理領域では「勘定科目」、認証領域では「ユーザーアカウント」を意味する。同じ言葉でもコンテキスト（文脈）によって意味が異なるため、明確に境界を引いてその中では言葉の意味を統一する。k1s0 では各「領域」がおおむね1つの境界づけられたコンテキストに対応する。

### ユビキタス言語

- **一言で言うと**: チーム全員（開発者・業務担当者）が使う「共通語彙」。
- **もう少し詳しく**: コード内の変数名・メソッド名・クラス名に、業務担当者が日常使っている言葉をそのまま使う。例えば「勘定科目カテゴリ」を `MasterCategory` と名付けたら、設計書・会話・コード全てでこの名前を使う。

## 技術用語

### gRPC

- **一言で言うと**: サーバー間の高速な通信方式。
- **もう少し詳しく**: Google が開発した RPC（Remote Procedure Call = 遠隔手続き呼出し）フレームワーク。HTTP/2 を使い、Protocol Buffers でデータをバイナリ形式でやり取りするため高速。k1s0 ではサービス間通信に使われる。REST API より厳密な型定義ができる。

### REST API

- **一言で言うと**: Web ブラウザやモバイルアプリなど外部クライアント向けの通信方式。
- **もう少し詳しく**: HTTP の GET / POST / PUT / DELETE などのメソッドと URL の組み合わせでデータをやり取りする。JSON 形式でデータを送受信する。k1s0 では外部向け（フロントエンドやサードパーティ）の API に使われる。

### GraphQL

- **一言で言うと**: クライアントが「欲しいデータだけ」を指定して取得できる API の仕組み。
- **もう少し詳しく**: REST API では「決まった形のデータ」が返ってくるが、GraphQL ではクライアント側が取得したいフィールドを指定できる。k1s0 では BFF（Backend For Frontend）層で使われる場合がある。

### Protocol Buffers（protobuf / プロトバフ）

- **一言で言うと**: データの形式（スキーマ）を定義するための言語。gRPC のデータ形式に使われる。
- **もう少し詳しく**: `.proto` ファイルにメッセージの形（フィールド名・型）やサービスの定義を書く。そこからコード（Rust / Go / TypeScript など）を自動生成できる。JSON よりもコンパクトで高速。

### buf（バフ）

- **一言で言うと**: Protocol Buffers の管理・コード生成を便利にするツール。
- **もう少し詳しく**: `.proto` ファイルの lint（書き方チェック）、破壊的変更の検出、コード生成を統一的に行える。`protoc`（公式コンパイラ）の代替として使われることが多い。

### JWT（ジェーダブリューティー / JSON Web Token）

- **一言で言うと**: 「この人は誰で、何が許可されているか」をコンパクトに表現するトークン（文字列）。
- **もう少し詳しく**: Base64 エンコードされた3つの部分（ヘッダー・ペイロード・署名）からなる文字列。ペイロード部分にユーザー ID やロール（権限）が含まれ、署名によって改ざんを検知できる。k1s0 では Keycloak が発行し、各サーバーが検証する。

### OAuth 2.0（オーオース 2.0）

- **一言で言うと**: 「あるサービスのデータに、別のサービスからアクセスする許可」を安全に行う仕組み。
- **もう少し詳しく**: 認可（Authorization）のための業界標準プロトコル。ユーザーがパスワードを直接渡さずに、アクセストークンを介してリソースへのアクセスを許可する。k1s0 では Keycloak が OAuth 2.0 サーバーとして機能する。

### OIDC（OpenID Connect / オープンアイディーコネクト）

- **一言で言うと**: OAuth 2.0 の上に「この人は本当に誰か」を確認する機能を追加した仕組み。
- **もう少し詳しく**: OAuth 2.0 は「認可」（何が許可されるか）だけを扱うが、OIDC は「認証」（この人は誰か）も扱う。ID トークンという JWT を返し、ユーザーの身元情報を提供する。

### RBAC（ロールベースアクセス制御 / Role-Based Access Control）

- **一言で言うと**: 「役割（ロール）」に応じてアクセスできる機能を制限する仕組み。
- **もう少し詳しく**: ユーザーに直接権限を割り当てるのではなく、「管理者」「閲覧者」などのロールを作り、ロールに権限を紐づける。ユーザーはロールを割り当てられることで権限を得る。k1s0 では gRPC メソッドごとに `read` / `write` / `admin` のアクション区分がある。

### Keycloak（キークローク）

- **一言で言うと**: ユーザー管理と認証を行うオープンソースの認証サーバー。
- **もう少し詳しく**: ユーザー登録、ログイン画面、JWT 発行、ロール管理などを一手に引き受ける。k1s0 では `http://localhost:8080/realms/k1s0/` でアクセスし、JWT の検証に JWKS（JSON Web Key Set）エンドポイントを使う。

### Kafka（カフカ）

- **一言で言うと**: 大量のメッセージ（データ）を確実に届けるためのメッセージングシステム。
- **もう少し詳しく**: サービス間で直接通信（同期）する代わりに、Kafka にメッセージを送り（プロデュース）、必要なサービスが読み取る（コンシューム）。メッセージは一定期間保持されるため、受信側が一時的にダウンしていてもデータを失わない。

### トピック

- **一言で言うと**: Kafka でメッセージを分類する「チャンネル」のようなもの。
- **もう少し詳しく**: 例えば `k1s0.business.accounting.domainmaster.category_changed.v1` というトピックには、カテゴリが変更されたときのイベントが流れる。メッセージはトピックごとに独立して管理される。

### コンシューマー

- **一言で言うと**: Kafka からメッセージを読み取る側のプログラム。
- **もう少し詳しく**: 特定のトピックを「購読」し、新しいメッセージが届いたら処理する。複数のコンシューマーがグループを作り、メッセージを分担して処理することもできる。

### プロデューサー

- **一言で言うと**: Kafka にメッセージを送信する側のプログラム。
- **もう少し詳しく**: イベントが発生したとき（例：カテゴリが更新された）、そのイベントデータを Kafka トピックに送信する。k1s0 では `infrastructure/messaging/kafka_producer.rs` などに実装がある。

### Schema Registry（スキーマレジストリ）

- **一言で言うと**: Kafka メッセージの「データ形式」を一元管理する仕組み。
- **もう少し詳しく**: メッセージの送信者と受信者が異なるデータ形式を使うとエラーになる。Schema Registry にスキーマを登録しておくことで、互換性のあるデータ形式を保証する。

### DLQ（デッドレターキュー / Dead Letter Queue）

- **一言で言うと**: 処理に失敗したメッセージを退避させる場所。
- **もう少し詳しく**: コンシューマーがメッセージを処理中にエラーが発生した場合、何度もリトライして全体を止めるのではなく、DLQ に移動させて後から調査・再処理する。

### axum（アクサム）

- **一言で言うと**: Rust の Web フレームワーク。REST API サーバーを作るために使う。
- **もう少し詳しく**: tokio（非同期ランタイム）の上で動作する軽量な Web フレームワーク。ルーティング（URL とハンドラーの対応付け）、ミドルウェア（認証チェックなど）、レスポンス生成などを提供する。k1s0 のサーバーは axum で構築されている。

### tonic（トニック）

- **一言で言うと**: Rust の gRPC フレームワーク。gRPC サーバー・クライアントを作るために使う。
- **もう少し詳しく**: `.proto` ファイルから `tonic-build` で Rust コードを自動生成し、そのコードを使ってサーバーのメソッドを実装する。axum と同じ tokio ランタイム上で動作する。

### sqlx（エスキューエルエックス）

- **一言で言うと**: Rust から PostgreSQL に接続して SQL を実行するライブラリ。
- **もう少し詳しく**: コンパイル時に SQL 文の型チェックができる機能がある（`sqlx::query_as!` マクロ）。非同期（async/await）で DB 操作を行え、コネクションプーリング（接続の使い回し）も内蔵している。

### マイグレーション

- **一言で言うと**: データベースの構造（テーブルやカラム）を段階的に変更していく仕組み。
- **もう少し詳しく**: SQL ファイルを番号付きで管理し（例: `001_create_schema.up.sql`）、適用（up）と取り消し（down）ができる。チーム全員が同じ DB 構造を維持でき、変更履歴も残る。`regions/business/{領域名}/database/{db名}/migrations/` に格納される。

### OpenTelemetry（オープンテレメトリー）

- **一言で言うと**: アプリケーションの動作状況を「見える化」するための業界標準ツールキット。
- **もう少し詳しく**: トレース（処理の流れ）、メトリクス（数値指標）、ログの3つのシグナルを統一的に収集・送信する。k1s0 では `k1s0-telemetry` ライブラリを通じて利用する。

### トレース

- **一言で言うと**: 1つのリクエストが複数のサービスをまたいで処理される「流れ」を追跡すること。
- **もう少し詳しく**: 例えば、フロントエンド → API ゲートウェイ → domain-master サーバー → DB という流れを、1つのトレース ID で紐づけて追跡できる。どこで時間がかかっているか、どこでエラーが出たかを可視化できる。

### メトリクス

- **一言で言うと**: アプリケーションの状態を数値で表したもの（例：リクエスト数、レスポンス時間）。
- **もう少し詳しく**: Prometheus 形式で `/metrics` エンドポイントから取得し、Grafana でグラフ化する。「直近1分間のエラー率」「平均レスポンス時間」などを監視に使う。

### Helm（ヘルム）

- **一言で言うと**: Kubernetes にアプリケーションをデプロイする際の「パッケージマネージャー」。
- **もう少し詳しく**: Kubernetes のマニフェスト（設定ファイル）をテンプレート化し、環境ごとに値を変えてデプロイできる。Chart（チャート）と呼ばれるパッケージ単位で管理する。

### Kubernetes（クバネティス / K8s）

- **一言で言うと**: コンテナ化されたアプリケーションの自動デプロイ・管理を行うプラットフォーム。
- **もう少し詳しく**: 「このサーバーを3台動かして、1台壊れたら自動で新しいのを起動して」といった管理を自動化する。k1s0 では `k1s0-system` / `k1s0-business` / `k1s0-service` の Namespace に分けてデプロイする。

### Namespace（ネームスペース）

- **一言で言うと**: Kubernetes クラスタ内の「仮想的な仕切り」。
- **もう少し詳しく**: 同じ Kubernetes クラスタの中で、リソース（Pod、Service など）をグループ分けする。k1s0 では Tier ごとに Namespace を分けている（例: `k1s0-business`）。

### Pod（ポッド）

- **一言で言うと**: Kubernetes で動作するコンテナの最小単位。
- **もう少し詳しく**: 1つまたは複数のコンテナをまとめたもの。通常は1つの Pod に1つのアプリケーションコンテナが入る。Pod は一時的なもので、壊れたら新しい Pod が作られる。

### Docker（ドッカー）

- **一言で言うと**: アプリケーションを「コンテナ」と呼ばれる軽量な仮想環境にパッケージングするツール。
- **もう少し詳しく**: Dockerfile にアプリケーションの構築手順を書き、Docker イメージを作る。そのイメージからコンテナを起動すると、どの環境でも同じ動作をする。開発環境では `docker-compose` を使って複数のコンテナ（DB、Kafka 等）をまとめて起動する。

### コンテナ

- **一言で言うと**: アプリケーションとその動作に必要なもの一式を閉じ込めた「箱」。
- **もう少し詳しく**: 仮想マシン（VM）より軽量で、起動が速い。OS のカーネルを共有しつつ、プロセスやファイルシステムを隔離する。Docker で作成されたコンテナが Kubernetes 上で実行される。

### サービスメッシュ

- **一言で言うと**: サービス間通信のセキュリティ・監視・トラフィック制御を透過的に行う仕組み。
- **もう少し詳しく**: 各 Pod にサイドカープロキシ（通信を中継するコンテナ）を配置し、アプリケーションコードを変更せずに通信の暗号化（mTLS）、リトライ、タイムアウト、トラフィック分割などを実現する。

### mTLS（相互TLS認証 / mutual TLS）

- **一言で言うと**: 通信する双方が「お互いの身元を確認しあう」暗号化通信。
- **もう少し詳しく**: 通常の HTTPS ではクライアントがサーバーの証明書を検証するだけだが、mTLS ではサーバーもクライアントの証明書を検証する。サービスメッシュで自動化されるため、開発者が直接証明書を管理する必要はない。

### Vault（ヴォルト）

- **一言で言うと**: パスワードや API キーなどの機密情報を安全に管理するツール。
- **もう少し詳しく**: HashiCorp 社が開発したシークレット管理ツール。DB パスワードや API キーをソースコードに書く代わりに Vault に保存し、アプリケーションが実行時に取得する。アクセス制御や監査ログも備えている。
