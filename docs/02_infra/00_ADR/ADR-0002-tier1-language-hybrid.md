# ADR-0002: tier1 の内部言語ハイブリッド (ファサード = Go / 自作領域 = Rust)

- ステータス: Accepted
- 起票日: 2026-04-14
- 決定日: 2026-04-14
- 起票者: kiso ryuhei
- 関係者: インフラチーム / 起案者 / 決裁者

## コンテキスト

tier1 は k1s0 における実行基盤の共通 API レイヤであり、tier2 / tier3 から呼び出される building block (`k1s0.Log` / `k1s0.State` / `k1s0.PubSub` / `k1s0.Workflow` / `k1s0.Secrets` / `k1s0.Decision` / `k1s0.Feature` 等) を提供する。
tier1 の機能は大別して 2 系統ある。

1. **Dapr ファサード**: Dapr building blocks の薄いラッパ。Service Invocation / State / PubSub / Secrets を tier2/3 に提供する。
2. **自作領域**: Dapr では充足できない k1s0 固有機能。具体的には ZEN Engine (ルールエンジン) 統合、JTC 固有機能、雛形生成 CLI など。

ここで Dapr の SDK 成熟度に大きな非対称性がある。
- **Dapr Go SDK**: stable (本番運用実績が豊富)
- **Dapr Rust SDK**: Alpha (公開 API が破壊的に変わる可能性が残る)

実装言語の選択肢として「Rust 一本」「Go 一本」「Go + Rust ハイブリッド」のいずれかを採る必要がある。

## 決定

tier1 を **Go と Rust のハイブリッド構成** とする。
役割分担は以下のとおり。

| 領域 | 言語 | 理由 |
|---|---|---|
| Dapr ファサード | **Go** | Dapr Go SDK が stable。SDK の不安定性に起因するリスクをゼロにする |
| 自作領域 (ZEN Engine / JTC 固有 / CLI) | **Rust** | メモリ安全性と性能、ZEN Engine が Rust ネイティブ |

言語境界は **gRPC + Protobuf** で厳密に管理する。
**tier2 / tier3 からは内部実装言語が一切不可視** とし、利用者は単一の k1s0 API として認識する。

## 検討した選択肢

### 選択肢 A: Rust 一本

- 概要: tier1 全体を Rust で実装する。
- メリット: 言語スタックが単一でビルド/CI/Lint が一系統。スタック横断のスキル要件が小さい。
- デメリット: Dapr Rust SDK が Alpha のため、本番採用には自前の安定化レイヤ (パッチ・互換性ラッパ) が必要となる。SDK の破壊的変更を継続的に追従するコストが大きい。

### 選択肢 B: Go 一本

- 概要: tier1 全体を Go で実装する。
- メリット: Dapr Go SDK の stable をフル活用できる。エコシステムが豊富、JTC でも採用例が多く人材確保が比較的容易。
- デメリット: ZEN Engine (Rust ネイティブ) との統合に CGO や別プロセス通信が必要となり、性能・安定性を損なう。Rust が活きる領域 (メモリ安全性が長期保守に効く部分) の優位性を失う。

### 選択肢 C: Go + Rust ハイブリッド (採用)

- 概要: ファサードを Go、自作領域を Rust とし、gRPC で連携する。
- メリット: 各領域で最適な言語を選択できる。Dapr Rust SDK Alpha の問題を回避し、ZEN Engine も in-process で扱える。
- デメリット: ビルド/CI/Lint が 2 系統。言語境界の契約管理が必要。オンコール時に両言語の知識が要求される。

### 選択肢 D: 多言語混在 (Go + Rust + その他)

- 概要: 機能ごとに最適な言語を都度選定する。
- メリット: 個別最適。
- デメリット: 保守コストが爆発する。tier1 の責務範囲を超えて拡散するリスク。

## 決定理由

- **SDK 成熟度の非対称性を素直に受容する**: Dapr Go SDK が stable で Rust SDK が Alpha という現実を回避策なしで受け入れることが、最も短い経路で「動くもの」を作れる。
- **本当に Rust が活きる用途に限定する**: 自作領域に Rust を限定することで、Rust 経験者の人材確保リスクを最小化する (人数を絞れる)。一方、Dapr ファサード側は Go 経験者がアサイン可能で、JTC 内での人材プールが広がる。
- **言語境界を契約で固める**: Protobuf による型安全な契約とすることで、ハイブリッド構成のデメリット (型不整合バグ温床) を緩和する。
- **tier2/3 への影響をゼロにする**: 内部言語を不可視にすることで、将来言語構成を見直しても tier2/3 に影響しない逃げ道を確保する。

## 影響

### ポジティブな影響

- 各領域で最適な言語を選択でき、Dapr の安定性と Rust の性能/安全性を両立できる。
- 内部実装の置き換え (Go → Rust への将来的な巻き取り、または逆) が tier2/3 に影響せずに実施できる。
- 人材確保リスクが分散される (Rust 経験者は限定領域のみで足りる)。

### ネガティブな影響 / リスク

- ビルドパイプライン・CI・Lint・依存管理が言語ごとに 2 系統となる。
  - 緩和策: 雛形生成 CLI で初期構成を均一化し、開発者の認知負荷を下げる。
- 言語境界での型不整合・契約変更によるバグの温床になりうる。
  - 緩和策: gRPC + Protobuf を採用し、契約変更時の互換性チェックを CI で自動化する。
- オンコール対応時に両言語のスキルが要求される。
  - 緩和策: 構造化ログ・エラーコード・OTel 属性を言語横断で統一し、運用者が言語を意識せず根本原因にたどり着けるようにする。

### 移行・対応事項

- tier1 のリポジトリ構成を「Go プロジェクト」「Rust プロジェクト」に分離する。
- Protobuf 定義を単一の真実源として管理し、両言語向けにコード生成する。
- 雛形生成 CLI に Go / Rust 両言語のテンプレートを用意する。
- CI で Go / Rust 双方の Lint / Test を実行する。
- ログフォーマット・エラーコード体系・OTel 属性命名を ADR またはガイドラインとして別途確定する (今後の起票対象)。
