# ADR-0003: AGPL ライセンス OSS の分離アーキテクチャ

- ステータス: Accepted
- 起票日: 2026-04-19
- 決定日: 2026-04-19
- 起票者: kiso ryuhei
- 関係者: 法務部 / 調達部 / システム基盤チーム / Product Council

## コンテキスト

k1s0 が採用する OSS の中には AGPL-3.0 ライセンスのものが存在する可能性がある。AGPL は「ネットワーク経由で利用可能にする行為をも頒布とみなし、改変ソースコードの開示を義務付ける」強い Copyleft ライセンスであり、同一プロセス/同一ネットワーク境界での組込み方法を誤ると、k1s0 本体（独自実装の tier1/tier2/tier3 コード）もソース開示義務の対象となりかねない。

現時点で AGPL 該当の主要候補は以下の通り。

- **Grafana**（2021 年以降 AGPL-3.0）: 観測性ダッシュボードとして採用予定
- **MongoDB**（SSPL だが AGPL 類似の強い Copyleft）: k1s0 では採用しないが将来的な利用要求に備える
- **Neo4j Community Edition**（AGPL-3.0）: グラフ DB として将来追加候補
- **Mastodon 系**（AGPL-3.0）: 監査/通知チャネルとして検討されうる

法務部の見解は「AGPL は『プロセス境界・ネットワーク境界で明確に分離』されていれば、本体コードのソース開示は不要」である。ただし判例が乏しく、分離の証跡（アーキテクチャ図、通信境界仕様）を残しておく必要がある。万一紛争時の反論材料として、分離アーキテクチャが設計時点で意図されていたことを示す ADR が不可欠である。

k1s0 は 採用側組織向け商用プラットフォームであり、本体コードを AGPL 条件で公開することは事業上許容できない。同時に、Grafana の代替（DataDog / New Relic 等）に切替えるとベンダーロックと商用コストが発生する。つまり「AGPL OSS を使いつつ、本体コード開示を回避する」運用が必須となる。

## 決定

**AGPL-3.0 ライセンスの OSS は、k1s0 本体コードから以下の 3 条件すべてを満たす形で分離する。**

1. **別プロセス**: AGPL OSS は独立した Pod / コンテナで稼働させる。本体コードと同一プロセスに組込まない
2. **公開プロトコルでの通信**: AGPL OSS との通信は HTTP/REST、gRPC、OTLP、Prometheus Remote Write など、標準化された公開プロトコルのみ使用
3. **オプショナルな依存**: AGPL OSS が停止しても、本体コアの業務機能が継続できる設計（観測は停止するが業務は動く、等）

この 3 条件を満たす限り、AGPL OSS は「分離された独立サービス」であり、本体コードはその使用者にすぎず、AGPL の「派生物」には該当しないと法務部が解釈している。

追加で、以下の運用ルールを課す。

- 全 AGPL OSS は BC-LIC-001〜007 の依存ライセンス台帳に明記、四半期レビュー対象
- AGPL OSS への独自プラグイン・独自パッチを書く場合、それらは AGPL で公開義務（本体コードには影響しない）
- 紛争時の反論材料として、本 ADR・分離アーキテクチャ図・通信ログ設計を証跡保管

## 検討した選択肢

### 選択肢 A: 3 条件分離アーキテクチャ（採用）

- 概要: プロセス/ネットワーク/依存の 3 軸で AGPL OSS を本体から分離
- メリット:
  - 法務リスクを構造的に最小化（分離が設計時点から明示）
  - OSS コミュニティの最新機能を享受できる（Grafana 等）
  - 代替品（商用 SaaS）への切替えコスト回避
- デメリット:
  - プロセス分離のためコンテナ追加、運用対象が増える
  - 独自プラグインを書くと AGPL で公開する必要（ただし本体は影響なし）

### 選択肢 B: AGPL OSS の全面回避

- 概要: Grafana の代わりに DataDog 等商用、Neo4j の代わりに ArangoDB（Apache-2.0）等
- メリット: AGPL リスクゼロ
- デメリット:
  - 商用製品の年間コスト数千万円規模
  - 機能・コミュニティ差で開発者体験が劣化
  - オンプレ制約と矛盾する選択肢（SaaS は選べない）

### 選択肢 C: AGPL OSS を本体に組込み、本体もソース公開

- 概要: AGPL の義務を受入れ、本体コードを GitHub で公開
- メリット: 分離アーキテクチャが不要、実装が単純
- デメリット:
  - 採用側組織の事業機密が公開される（営業機密・競合優位性の喪失）
  - 採用検討の通過が困難（経営層が拒否する可能性高）

## 帰結

### ポジティブな帰結

- 法務リスクの構造的低減（分離設計の証跡が残る）
- Grafana など AGPL OSS の採用が可能
- 将来的な商用 SaaS 移行時も同じ分離設計がそのまま使える（ベンダーロック軽減）

### ネガティブな帰結

- コンテナ数の増加（運用対象が 10〜15 個増える見込み）
- 独自カスタマイズは AGPL で公開する必要（ただし本体への波及なし）
- 判例不足によるグレーゾーンが残存（四半期法務レビュー継続が必要）

## 実装タスク

- 分離アーキテクチャ図を 02_構想設計/05_法務とコンプライアンス/ 配下に作成
- AGPL 依存 OSS の台帳を BC-LIC-001 の依存リストに統合
- Grafana との通信は Prometheus Remote Write / OTLP のみ、カスタムクエリは UI 操作に限定
- 年次外部ライセンスレビュー（BC-LIC-005）で分離状態を監査
