# ADR-0002: drawio 図解レイヤ記法規約の採用

- ステータス: Accepted
- 起票日: 2026-04-19
- 決定日: 2026-04-19
- 起票者: kiso ryuhei
- 関係者: システム基盤チーム / ドキュメント全レビュアー

## コンテキスト

k1s0 のドキュメント (`docs/`) は drawio で生成した SVG を多用しており、企画書段階で `docs/01_企画/img/全体構成図.drawio` が採用判断者向けの中核図として機能している。しかし ADR-0001 (Istio Ambient Mesh 採用) の議論中に、以下の**図解起因の誤解**が発生した。

初期版の `全体構成図.drawio` では、Pod 内のコンポーネントとして `app container (灰箱) + Dapr サイドカー (青箱) + Istio サイドカー (青箱)` の 3 コンテナを視覚的にほぼ同列で描画していた。その結果、読み手 (起票者自身を含む) が Dapr サイドカーと Istio サイドカーを「同じ責務レイヤのコンポーネント」と誤認した。実際には両者は責務レイヤが明確に異なる。

- Dapr (daprd サイドカー) = **アプリ層**: アプリが `k1s0.State.Save()` のような SDK 呼び出しで**明示的に呼び出す**抽象
- Istio (Envoy / ztunnel) = **ネットワーク層**: iptables / eBPF を介してアプリに**透過的に効く**機構

この違いは採用検討資料の根幹であり、採用判断者が「なぜ両方必要なのか」を判断するための一次情報である。ところが図上の視覚的同一性 (どちらも Pod 内に並んだ色箱) が論理的差異を覆い隠し、結果として ADR-0001 の議論中に「Dapr と Istio のどちらを優先するのか」という本来成立しないはずの問い合わせが発生した。

この事象は単一図の表現ミスではなく、**drawio 図解全体における規約の欠如**に起因する構造的問題である。現時点で `docs/**/*.drawio` は 100 ファイル超存在し、今後要件定義・概要設計段階でさらに増える。規約なく運用を続ければ同じ誤解が別の図で再発する。

本 ADR は全 drawio 図解の記法を共通規約で拘束することを採択し、既存図の是正と新規作成時のチェックリストを整備する。

## 決定

**drawio 図解の作成・更新時に `docs/00_format/drawio_layer_convention.md` で定義するレイヤ記法規約に従うことを義務化する。**

規約の骨子は以下。

1. **4 レイヤ定義**: すべてのコンポーネントをアプリ層 / ネットワーク層 / インフラ層 / データ層のいずれかに分類する
2. **色システム**: 各レイヤに固定色を割り当てる (暖色 / 寒色 / 中性灰 / 紫)。同一レイヤ内の色散らばしを禁止する
3. **線種システム**: 矢印の線種で呼び出し性質を区別する (実線太=明示呼び出し / 点線細=透過捕捉 / 破線=非同期)
4. **swim lane 構造**: 大規模全体図では水平 4 レーンを基本骨格とする
5. **凡例ブロック**: すべての drawio に右下固定で凡例を配置する
6. **チェックリスト**: 作成・更新前に 7 項目の確認を実施する

規約の詳細と運用手順は `docs/00_format/drawio_layer_convention.md` を正とする。

## 検討した選択肢

### 選択肢 A: レイヤ記法規約を共通規約として定める (採用)

- 概要: 色 / 線種 / 配置の 3 軸でレイヤを明示的に分離する規約を定め、`docs/00_format/drawio_layer_convention.md` として運用する
- メリット:
  - 同じ誤解が別図で再発することを構造的に防止できる
  - 新規参画者や外部読者 (採用判断者) が規約に従った図を一貫して読める
  - 規約違反を機械的に監査できる (色コード・線種が規約値かどうかの検証が可能)
  - Kubernetes コミュニティやクラウド各社のアーキテクチャ図と視覚言語が整合する (Cloud Native Computing Foundation の landscape 図等が同様の色分類を採用)
- デメリット:
  - 既存 100 ファイル超の drawio を監査・是正する初期コストが発生する
  - ルールの学習コストが発生する (ただし規約ドキュメント + チェックリストで緩和可能)

### 選択肢 B: 規約を定めず、図ごとに最適な表現を選ぶ

- 概要: 作者の裁量に委ね、規約化しない
- メリット:
  - 初期コストがかからない
  - 図ごとに自由度が高く、特殊な表現 (例: ネットワーク層深掘り図でのパケットフロー独特の記法) に制約をかけない
- デメリット:
  - ADR-0001 で発生した誤解が**確実に再発する**
  - 図ごとに視覚言語が異なるため、複数図を横断する読解コストが増大する
  - 採用検討資料という「1 回読んで判断される図」において誤解を許容するリスクは受容できない

### 選択肢 C: 凡例のみ義務化し、色・線種は自由にする

- 概要: 各図に凡例を必ず置くが、色コードや線種は統一しない
- メリット:
  - 選択肢 A より柔軟
  - 凡例を読めば意図は伝わる (はず)
- デメリット:
  - 読み手が凡例を読まずに視覚表現から直感的に意味を読み取ろうとした瞬間、ADR-0001 と同じ誤解が発生する
  - 実際の読解行動として、読み手は凡例より先にコンポーネント本体を見る。凡例を補助情報としてのみ機能させるのは不十分
  - 「凡例はあるが色の意味は図ごとに違う」状態は、複数図読解時にかえって混乱を招く

## 決定理由

選択肢 B は「規約の不在によってすでに発生した誤解」の再発を許容することになり、採択できない。選択肢 C は凡例を読まない読み手の直感を制御できず、発生したインシデントへの対策として不十分である。

選択肢 A は初期コストが発生するが、k1s0 プロジェクトの性質上 (採用側組織の情シス向けの採用検討資料が事業価値の一次入力)、図解の誤読は事業判断の誤りに直結する。一度採択された規約は 100 以上の既存ファイルと将来の全新規ファイルに適用されるため、規約化の投資効率は極めて高い。

したがって**選択肢 A を採択する**。

## 影響

### ポジティブな影響

- ADR-0001 で発生した Dapr / Istio のレイヤ誤認が構造的に防止される
- 紹介資料・要件定義・概要設計・運用ドキュメントを通じて一貫した視覚言語が提供される
- 新規参画者のオンボーディング時、図解の読み方を規約ドキュメント 1 本で説明できる
- CLAUDE.md の drawio セクションから規約への参照が追加されており、Claude Code を含む AI 支援ツールが規約を自動的に適用する
- 既存図の監査により、本規約以外の潜在的な記法不整合 (色の混在、矢印方向の不統一等) も発見・是正される

### ネガティブな影響 / リスク

- 既存 100 超の drawio ファイルのうち、複数レイヤが登場するものは段階的に是正する必要がある。一度に全是正すると工数が膨大なため、優先順位を「企画書 > 構想設計 > 要件定義 > knowledge」の順に設定し、段階的に適用する
- 規約は絶対ルールではなく運用ガイダンスである。本規約に収まらない特殊ケース (例: 自前ネットワーク層のパケットフロー深掘り図) が登場した場合、個別図で例外記法を定義し、凡例で明示することを許容する
- 規約の更新には本 ADR を Superseded することで対応する (軽微な補足は規約ファイル自体の「履歴」節で追記)

### 移行・対応事項

1. `docs/00_format/drawio_layer_convention.md` を新設 (本 ADR 採択と同時に完了済み)
2. `CLAUDE.md` の drawio セクションに規約参照を追加 (完了済み)
3. `docs/01_企画/img/全体構成図.drawio` を規約に準拠した 4 レーン swim lane 構造で再設計
4. `docs/02_構想設計/01_アーキテクチャ/img/` にレイヤ別詳細図 4 本を新設
5. `docs/**/*.drawio` を全監査し、監査レポートを `docs/99_壁打ち/drawio_layer_audit_2026-04-19.md` に記録。優先度の高い図を先行是正
6. 全 SVG を再生成し、レイヤ分離が視覚的に機能しているか第三者視点で確認
7. 今後新規作成される drawio は、必ず規約ファイルのチェックリスト 7 項目を実施する
