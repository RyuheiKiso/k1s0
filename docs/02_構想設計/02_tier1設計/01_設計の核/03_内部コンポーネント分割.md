# tier1 内部コンポーネント分割

## 目的

本資料は、全体構成図（[`../../../01_企画/全体構成図.md`](../../../01_企画/全体構成図.md) の Row 3）で列挙される tier1 内部 6 Pod（COMP-T1-STATE / COMP-T1-SECRET / COMP-T1-AUDIT / COMP-T1-DECISION / COMP-T1-PII / COMP-T1-WORKFLOW）の存在根拠を、要件定義に登録済みの FR / NFR / BR / ADR から一意に導出可能な状態にすることを唯一の責務とする。

tier1 の内部実装は [`00_tier1のスコープ.md`](00_tier1のスコープ.md) で定めた通り tier2/tier3 から不可視のため、要件定義（[`../../../03_要件定義/20_機能要件/10_tier1_API要件/`](../../../03_要件定義/20_機能要件/10_tier1_API要件/)）は観測可能な契約面（11 公開 API）のみを扱い、Pod 分割自体は要件ではない。しかしなぜ 6 Pod に分割されているのかを要件側の記述から逆引きできない状態は、tier2 開発者・稟議決裁者・監査担当に「tier1 の中身が説明不能」という印象を与え、プラットフォーム自体の信頼性を損ねる。

本資料はこの裂け目を埋めるため、6 Pod の各々について「集約する機能要件群」「運用形態を決める非機能要件」「言語選択の ADR」を要件 ID レベルで一意に紐付け、内部コンポーネント ID（COMP-T1-\*）で呼び名を確定する。コンポーネント ID は [`../../../03_要件定義/80_トレーサビリティ/03_構想設計マトリクス.md`](../../../03_要件定義/80_トレーサビリティ/03_構想設計マトリクス.md) の「tier1 内部コンポーネント」セクションで双方向トレースの対象として管理する。

## 分割 5 原則

tier1 内部を単一の巨大 Pod にせず 6 つに分ける判断は、恣意ではなく 5 つの独立した要件駆動の強制力の合流点として導かれる。各原則は単独で Pod 分離を要請するため、「どれか 1 つでも欠けると合流地点の Pod 数が増減し、現行の 6 Pod 構成が崩れる」関係にある。

### 原則 1: データアフィニティ原則

公開 API の裏側で強結合するデータストアが異なる場合、同一 Pod に同居させると障害隔離（NFR-A-CONT-003: バックエンド障害の影響限定）が崩れる。たとえば Valkey 障害時に Secrets / Decision / Audit が同時に落ちると tier1 の対外 SLA 99%（NFR-A-CONT-001）を維持できない。ストア別に Pod を分離することで、各 API の劣化が他 API へ波及しない構造になる。

### 原則 2: 言語境界原則

ADR-TIER1-001（Dapr Go SDK + Rust 自作）で Go ファサードと Rust 自作領域の境界が確定している。両言語を同一 Pod に混在させると、ビルド・CI・Lint・依存管理が 2 系統走る運用コスト（[`02_内部言語ハイブリッド.md`](02_内部言語ハイブリッド.md) 第 6 章のトレードオフ表）を単一 Pod に押し込むことになり、Go 担当・Rust 担当の役割分離が機械的に保てない。

### 原則 3: 失敗ドメイン原則

NFR-A-CONT-003（バックエンド障害の影響限定）と NFR-A-FT-001（単一 Pod 障害時の自動復旧）を両立させるには、障害時に失敗の半径（Blast Radius）が異なる責務を同一 Pod に混ぜてはならない。Secret リース排他管理（Leader-Elected）と 高頻度 State API（HPA）を同居させると、Leader 切替で高頻度 API が一斉停止する。

### 原則 4: スケール特性原則

NFR-B-WL-001（規模別 RPS）と NFR-B-PERF-001〜007（API 別レイテンシ / スループット / 計装オーバヘッド）で、API ごとにワークロード特性が大きく異なる。State は平常時 150 RPS 中規模（NFR-B-PERF-002）だが、Decision は p99 < 1ms（NFR-B-PERF-004）を求めるため in-process キャッシュウォーム化の恩恵が独立 Pod で最大化する。Audit は書込専用で順序保証が必須のため同一 Pod にしか集約できない。

### 原則 5: 順序保証原則

FR-INFO-AUDITMODEL-001（監査イベントスキーマ、ハッシュチェーン）と FR-T1-AUDIT-001（改ざん不可能性）は `previous_hash` → `event_hash` の単方向連鎖を要求し、書込順序の乱れが改ざん検知を崩す。順序保証が要件となる処理は StatefulSet + PVC per replica に隔離しなければ、HPA スケール時の並列書込で連鎖が壊れる。

## 6 コンポーネント定義

以下 6 ブロックは「どの原則がどの方向から Pod 分離を要請しているか」を各コンポーネントごとに記録する。分割マトリクス全体は図で俯瞰する。

![tier1 内部コンポーネント分割](../img/内部コンポーネント分割.svg)

### COMP-T1-STATE（Dapr 強依存 Go ファサード集約 Pod）

公開 API のうち Dapr Building Block を薄くラップする群（Service Invoke / State / PubSub / Binding）と、Dapr 非依存だが gRPC ファサードのみで完結する軽量 API（Feature）を 1 Pod に集約する。原則 1（Valkey / Kafka / 外部 HTTP の複数ストアを扱うが、いずれも Stateless な読み書きでありストア障害が Pod 状態を汚染しない）と原則 4（中規模 RPS で HPA 3–10 の水平拡張が最適）が合流し、単一 Pod に集約することでリクエスト経路の共通ミドルウェア（Auth / Tenant / Trace / Audit / PII / ErrorMap の 6 層）を 1 度で通過させ NFR-B-PERF-001 の p99 < 500ms に寄与する。

- 集約 FR: FR-T1-INVOKE-001〜005、FR-T1-STATE-001〜005、FR-T1-PUBSUB-001〜005、FR-T1-BINDING-001〜004、FR-T1-FEATURE-001〜004
- 駆動 NFR: NFR-B-PERF-001（p99 < 500ms）、NFR-B-PERF-002（150 RPS）、NFR-A-CONT-003（影響限定）
- 参照 ADR: ADR-TIER1-001（Go ファサード）、ADR-TIER1-002（Protobuf gRPC）
- 実装言語: Go（Dapr Go SDK stable）
- 運用形態: Deployment + HPA 3〜10、Stateless、即時 reschedule 可
- 主要バックエンド: Valkey Cluster / Apache Kafka / 外部 HTTP / flagd

### COMP-T1-SECRET（OpenBao リース排他管理 Pod）

Secrets API は OpenBao の動的 Secret リース（FR-T1-SECRETS-002）で発行時刻とライフタイムを Pod 側で排他管理する必要がある。原則 3（Leader-Elected でリース排他）が単独で Pod 分離を要請する。HPA にすると複数レプリカが同一リースに対して競合しキャッシュ不整合が発生するため、active 1 / standby 2 の構成で Leader 切替時のみ standby が昇格する。これは NFR-A-CONT-002（OpenBao 障害時の degrade 稼働）と整合する。

- 集約 FR: FR-T1-SECRETS-001〜004
- 駆動 NFR: NFR-A-CONT-002（degrade 稼働）、NFR-E-ENC-001（鍵管理）、NFR-H-KEY-001（鍵ライフサイクル）
- 参照 ADR: ADR-SEC-002（OpenBao）、ADR-TIER1-001（Go ファサード）
- 実装言語: Go（Dapr Secrets Building Block ラッパ + リース管理）
- 運用形態: Deployment + Leader Election（active 1 / standby 2）
- 主要バックエンド: OpenBao（動的 Secret / PKI / Transit）

### COMP-T1-AUDIT（ハッシュチェーン永続化 Pod）

監査ログのハッシュチェーン（FR-T1-AUDIT-001）は `previous_hash` → `event_hash` の単方向連鎖で改ざん検知を成立させる（FR-INFO-AUDITMODEL-001）。原則 5（順序保証）が StatefulSet を要求し、原則 2（言語境界）が crypto / parser 用途での Rust メモリ安全性（ADR-TIER1-001）を要求する。長期保存 7 年（FR-T1-AUDIT-003 / NFR-C-NOP-003）のための WORM Append 先は各 replica の PVC に 1:1 で対応し、書込中断からの再開で順序が壊れない。

- 集約 FR: FR-T1-AUDIT-001, FR-T1-AUDIT-002, FR-T1-AUDIT-003
- 駆動 NFR: NFR-C-NOP-003（7 年保管）、NFR-A-FT-001（単 Pod 復旧）、NFR-H-INT-001〜004（完整性）
- 参照 ADR: ADR-TIER1-001（Rust 自作）、ADR-DATA-001（CloudNativePG）、ADR-DATA-003（MinIO Object Lock）
- 業務根拠: BR-PLATGOV-004（個人情報保護法・J-SOX 保管義務準拠）
- 実装言語: Rust（crypto / WORM append）
- 運用形態: StatefulSet + PVC per replica、順序保証
- 主要バックエンド: PostgreSQL WORM append-only / MinIO Object Lock

### COMP-T1-DECISION（ZEN Engine in-process Pod）

Decision API は JDM 決定表評価（FR-T1-DECISION-001）を ZEN Engine に委譲する。原則 4（p99 < 1ms、NFR-B-PERF-004）は Rust in-process 呼び出しでのみ達成可能で、gRPC over network では 1ms SLO を破る。原則 2（言語境界）により Rust 単独 Pod に隔離し、テナント別ルールのウォームキャッシュを Pod メモリに閉じ込める。HPA 2–8 のスケールアウトは評価負荷に追従する。

- 集約 FR: FR-T1-DECISION-001, FR-T1-DECISION-002, FR-T1-DECISION-003, FR-T1-DECISION-004
- 駆動 NFR: NFR-B-PERF-004（p99 < 1ms）、NFR-A-CONT-003（影響限定）
- 参照 ADR: ADR-RULE-001（ZEN Engine + JDM）、ADR-TIER1-001（Rust 自作）
- 業務根拠: BR-PLATUSE-007（ビジネスルール変更の迅速化）
- 実装言語: Rust（ZEN Engine ネイティブ統合）
- 運用形態: Deployment + HPA 2〜8、in-memory ZEN cache
- 主要バックエンド: ZEN Engine（in-process、ストア外置き無し）

### COMP-T1-PII（PII マスキング Pod）

PII マスキング（FR-T1-PII-001）は副作用なしの純関数であり、外部ストアに状態を持たない。原則 2（言語境界）で Rust の安全性を要求し（parser / crypto 系の境界違反防止）、原則 3（失敗ドメイン）で Audit の Stateful 性質から分離する。Dapr サイドカー不要の sidecar-free 構成で Pod 起動時間・メモリフットプリントを最小化し、NFR-B-PERF-006（計装オーバヘッド < 10ms）の内訳として PII 処理の寄与を 1ms 未満に抑える。

- 集約 FR: FR-T1-PII-001, FR-T1-PII-002
- 駆動 NFR: NFR-B-PERF-006（計装オーバヘッド）、NFR-G-ENC-001〜003（データ暗号化）
- 参照 ADR: ADR-TIER1-001（Rust 自作）、ADR-RULE-001（判定ルールに ZEN 併用の可能性）
- 業務根拠: BR-PLATGOV-003（漏えい時のレピュテーション保護）
- 実装言語: Rust（parser / crypto）
- 運用形態: Deployment + HPA 2〜6、Dapr サイドカーなし
- 主要バックエンド: なし（純関数、テナント設定を Decision 経由で受領）

### COMP-T1-WORKFLOW（Temporal / Dapr Workflow 併用 Pod）

長期実行ワークフローと Saga（FR-T1-WORKFLOW-001/002/003）は task queue sticky execution を要する。replica 固定 3 台は Temporal の worker pool 前提であり、HPA にすると task の偏り（hot partition）で補償トランザクションが遅延する。ADR-RULE-002 と U-005 の決着により、長期実行は Temporal、短期ワークフロー / タイマーは Dapr Workflow を採用済みとし、両系統を同一 Pod 群で運用する。Saga 永続化と実行履歴が独立ストアを使うため、Pod 分離の必要性は変わらない。

- 集約 FR: FR-T1-WORKFLOW-001, FR-T1-WORKFLOW-002, FR-T1-WORKFLOW-003, FR-T1-WORKFLOW-004, FR-T1-WORKFLOW-005
- 駆動 NFR: NFR-A-FT-003、NFR-A-DR-001〜002、NFR-A-REC-001〜002、NFR-I-SLO-006
- 参照 ADR: ADR-RULE-002（短期は Dapr Workflow、長期は Temporal）、ADR-TIER1-001（Go）
- 決着事項: U-005（2026-04-24: 長期実行は Temporal、短期は Dapr Workflow）
- 実装言語: Go（Dapr Workflow SDK / Temporal Go SDK）
- 運用形態: Deployment 固定 3 replica、task queue sticky
- 主要バックエンド: Temporal PostgreSQL（長期実行） + Valkey（短期 Dapr Workflow）

## 6 Pod に収まらない責務

Log API（FR-T1-LOG-001〜004）と Telemetry API（FR-T1-TELEMETRY-001〜004）は tier1 SDK 内での構造化ログ生成と OTel span 生成に帰責される。配送は OpenTelemetry Collector（インフラ層 DaemonSet、[`../../01_アーキテクチャ/`](../../01_アーキテクチャ/) 参照）が担うため、tier1 アプリ層に専用 Pod を置かない。原則 1（データアフィニティ）から見ると Loki / Tempo / Mimir は tier1 の専有ストアではないため、ファサード Pod を挟むと責務の二重化になる。これは NFR-B-PERF-006（計装オーバヘッド < 10ms）を守るための設計判断でもある。

## 受け入れ基準（双方向トレース）

本分割が要件定義と双方向に辿れることを以下 4 点で判定する。いずれかが崩れたときは本資料の該当コンポーネントを改訂する。

- 各 COMP-T1-\* が集約する FR 群は [`../../../03_要件定義/20_機能要件/10_tier1_API要件/00_tier1_API共通規約.md`](../../../03_要件定義/20_機能要件/10_tier1_API要件/00_tier1_API共通規約.md) の「公開 API と内部コンポーネントの収容マトリクス」と 1:1 で一致する
- 各 COMP-T1-\* が駆動される NFR は [`../../../03_要件定義/80_トレーサビリティ/03_構想設計マトリクス.md`](../../../03_要件定義/80_トレーサビリティ/03_構想設計マトリクス.md) の「tier1 内部コンポーネント」セクションから逆引き可能である
- 公開 API ファイル（01〜11）の「関連非機能要件」が本資料の COMP-T1-\* と矛盾しない（矛盾発見時は本資料と API ファイル双方で同時に改訂）
- U-005 の決着内容が反映され、COMP-T1-WORKFLOW の「主要バックエンド」が Temporal PostgreSQL（長期）と Valkey（短期）で固定されている

## 決定を見直す条件

以下のいずれかが成立したとき、本分割を再評価する。Phase 終了レトロスペクティブで定量チェックする。

1. **ADR-RULE-002 の改訂**で短期 / 長期の基盤境界が見直されたとき。COMP-T1-WORKFLOW の運用形態と言語バッジを改訂する
2. **Dapr Rust SDK が Stable 到達**で ADR-TIER1-001 の言語境界が再引きされた場合。COMP-T1-STATE / COMP-T1-SECRET を Rust に統合する選択肢が浮上する
3. **Feature API のバックエンドが flagd から別 OSS に切替**となった場合。COMP-T1-STATE 同居から COMP-T1-FEATURE 独立 Pod への分離判定が走る
4. **tier1 内部 Pod 数が 10 を超える規模拡張**で、原則 5 つの合流構造が変質した場合
