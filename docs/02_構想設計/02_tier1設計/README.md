# 02_tier1設計

k1s0 の存在意義そのものである tier1 — Dapr 隠蔽・Go+Rust ハイブリッド・多言語クライアントライブラリ・雛形生成 CLI — の設計思想を扱う章。tier1 が tier2/tier3 からどれだけの複雑性を引き受けるかが k1s0 全体の開発者体験を決める。本章は「tier1 がサボると tier2 が苦労する」前提で書かれており、引き受ける範囲・引き受けない範囲を明示することを目的にしている。

## 想定読者

tier1 実装を担うシステム基盤チーム（アーキテクト級）、クライアントライブラリ設計者、tier3 向け開発者体験の設計者。業務ロジック実装者（tier2 開発者）は [`03_開発者体験/14_tier3開発者体験設計.md`](./03_開発者体験/14_tier3開発者体験設計.md) 周辺のみ参照でよい。

## 前章と後章

- 前章: [`../01_アーキテクチャ/`](../01_アーキテクチャ/) — 本章は tier1 の内部構造。
- 後章: [`../03_技術選定/`](../03_技術選定/) — 本章の決定事項を OSS にマッピングする / [`../04_CICDと配信/`](../04_CICDと配信/) — クライアントライブラリの配布導線

## 04 章との責務境界

本章は「tier1 が何を引き受け、どう設計するか」を書く。**選定した OSS の候補比較・定量根拠・採用段階別導入計画は [`../03_技術選定/`](../03_技術選定/) の責務であり、本章では扱わない**。例えば「Go + Rust ハイブリッド」という判断そのものは [`01_設計の核/02_内部言語ハイブリッド.md`](./01_設計の核/02_内部言語ハイブリッド.md) に書くが、Go / Rust の工数・性能比較の数値は [`../03_技術選定/01_俯瞰/18_言語選定の定量分析.md`](../03_技術選定/01_俯瞰/18_言語選定の定量分析.md) に置く。読者が「なぜそう決めたか」を数値で詰めたいときは 04 章、「どう設計したか」を詰めたいときは本章、という分業である。この分業が崩れていると感じた箇所は Issue を起こしてほしい。

## 本章の構成

## なぜサブディレクトリ構造にしたか

16 本のフラットな md を同一階層に並べる旧構成では、以下の問題が顕在化していた。

- **番号と責務が一致しない** — `05_ワークフロー振り分け基準` は診断ドメイン、`06_クライアントライブラリ配布戦略` は API 契約ドメイン、`07_tier2 オンボーディング戦略` は開発者体験ドメインと、隣接する番号でドメインがジャンプする。読み手は番号順に読むと頻繁にコンテキストを切り替えざるを得ない。
- **凝集が見えない** — 「診断と補償」ドメインは `05` / `08` / `09` / `10` / `13` に離散しており、関連ファイル群がひとまとまりに見えない。
- **追加のたびに順序が崩れる** — 新規トピックが末尾の `14_`, `15_` と積まれるたびに、ドメイン内の位置づけが番号から消える。

責務別サブディレクトリに再編することで、URL パスから「どのドメインの議論か」が即座に伝わり、追加・分割の影響範囲もドメイン内に閉じる。番号は旧構成を維持して変更差分を最小化している（番号の連続性より、追跡可能性を優先した）。
