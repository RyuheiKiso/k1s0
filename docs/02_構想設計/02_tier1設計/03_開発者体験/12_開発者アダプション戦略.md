# 開発者アダプション戦略

## 目的

k1s0 の技術的合理性 (ベンダーロックイン回避、コスト削減、横断的関心事の統一) は経営層・アーキテクトの視点で記述されている。しかし日々コードを書く tier2 開発者にとって、これらの動機は**抽象的すぎて腹落ちしない**。「前のやり方で困ってなかった」開発者ほど、移行の痛みだけを感じて恩恵を感じにくい。

本資料では、tier2 開発者が**自分の日常業務が具体的に楽になる**ことを体感し、変化を主体的に受け入れるための 3 つの施策を定義する。

---

## 1. 背景となる課題

### 1.1 合理性の説明だけでは不十分な理由

[`../../../01_企画/02_競合と差別化/02_差別化ポイント.md`](../../../01_企画/02_競合と差別化/02_差別化ポイント.md) で定義している「勝ち筋 5 点」は、決裁者・アーキテクト向けの訴求としては有効である。しかし tier2 開発者には以下の心理的障壁がある。

| 障壁 | 開発者の内心 |
|---|---|
| **損失回避バイアス** | 「新しいものの利益」より「今のやり方を失うこと」への恐怖が大きい |
| **コンピテンシートラップ** | 10 年間 C# / IIS で蓄積した専門性が無価値になる不安 |
| **抽象的な利益 vs 具体的な痛み** | 「ベンダーロックイン回避」は将来の利益。「gRPC を覚える」は今日の痛み |
| **不信感** | 「上が決めたことだろう。現場を分かっていない」 |

### 1.2 解決の方向性

理屈で説得するのではなく、**体験で納得させる**。以下の 3 施策は、この原則に基づいて設計されている。

| 施策 | 対処する障壁 | 作用 |
|---|---|---|
| 痛みの追体験ワークショップ | 損失回避バイアス・不信感 | 「前のやり方がいかに非効率だったか」を体感 |
| ファーストコンタクト体験 | コンピテンシートラップ | 「新しいやり方でもすぐに成果が出せる」を実証 |
| チャンピオン制度 | 不信感 | 「上からの押し付け」ではなく「同僚の推薦」に転換 |

---

## 2. 痛みの追体験ワークショップ

### 2.1 概要

k1s0 が解決する問題を**座学ではなく追体験させる**半日 (4 時間) のハンズオンワークショップ。「前のやり方でやってみる → k1s0 でやってみる → 差を体感する」の繰り返しで構成する。

### 2.2 実施タイミング

| 対象 | タイミング | 目的 |
|---|---|---|
| 新規 tier2 開発者 | k1s0 参画前 (オンボーディング L1 開始前) | 「なぜ k1s0 があるのか」の腹落ち |
| 既存チームメンバー | 採用後の運用拡大時 のキックオフ時 | 移行への動機付け |

### 2.3 ワークショップ構成

| 時間 | セッション | 内容 | 狙い |
|---|---|---|---|
| 0:00–0:45 | **課題 1: 認証を自前実装せよ** | REST API に JWT 認証を追加する。トークン検証、リフレッシュ、ロール検証、有効期限チェックを 1 から実装 | 横断的関心事の自前実装がどれだけ面倒かを体感 |
| 0:45–1:00 | **k1s0 での同等実装** | `k1s0.Auth.RequireRole("admin")` の 1 行で終わることを実演。トークンリフレッシュも tier1 が自動管理 | 「45 分の作業が 1 行になった」 |
| 1:00–1:45 | **課題 2: ログを統一せよ** | 3 つのサービスのログフォーマットがバラバラな状態で「注文 ID: 12345 のエラー原因を特定せよ」を調査 | フォーマット不統一の調査コストを体感 |
| 1:45–2:00 | **k1s0 での同等調査** | 構造化ログ + CorrelationId を使い、Tilt ダッシュボードで 3 サービスのログを横断検索 | 「45 分が 10 秒になった」 |
| 2:00–2:45 | **課題 3: 新サービスを立ち上げよ** | Dockerfile 作成、k8s マニフェスト作成、CI 設定、認証・ログ・監視の組み込みを手動で実施 | セットアップの苦行を体感 |
| 2:45–3:00 | **k1s0 での同等作業** | `k1s0-scaffold --service-name demo --language csharp` → `k1s0 dev start` → ブラウザで動作確認 | 「45 分が 2 分になった」 |
| 3:00–4:00 | **振り返り + 質疑** | 「前のやり方に戻りたいですか?」を問い、不安や疑問を全て受け止める。ファシリテーターは全ての懸念に誠実に回答する | 合理性の腹落ちと不安の発散 |

### 2.4 ファシリテーションの原則

| 原則 | 理由 |
|---|---|
| **課題の難易度は意図的に高くしない** | 「従来のやり方は難しかった」と思わせるのではなく、「従来のやり方は**面倒だった**」と思わせる |
| **k1s0 側は最短経路のみ見せる** | 機能の網羅的説明は不要。「こんなに簡単になる」の一点突破 |
| **振り返りでは反論を歓迎する** | 「でも○○は前のほうが良かった」を封じない。正当な指摘は tier1 チームの改善タスクとして記録する |
| **ファシリテーターは tier1 チームではなくチャンピオン (4 章) が担う** | 「上からの押し付け」感を排除する |

### 2.5 教材リポジトリ

ワークショップで使用する教材は専用リポジトリで管理する。

```
k1s0-workshop/
├── challenge-1-auth/        # 課題 1: 認証自前実装の初期状態
├── challenge-2-logging/     # 課題 2: ログ不統一の 3 サービス
├── challenge-3-setup/       # 課題 3: 新サービス手動セットアップの手順書
├── solutions/               # 各課題の模範解答
└── facilitator-guide.md     # ファシリテーター向け進行ガイド
```

## 3. ファーストコンタクト体験

### 3.1 概要

新しい開発者が k1s0 に初めて触れた**最初の 60 分**で、自分が書いたコードが動くサービスを体験する。この 60 分で k8s、Dapr、Kafka、Protobuf の単語を**一度も見せない**ことを設計目標とする。

### 3.2 60 分のシナリオ

| 時間 | 操作 | ゴール | 使用ツール |
|---|---|---|---|
| 0–5 分 | `k1s0 scaffold --service-name hello --language csharp --level 1` | プロジェクトが生成されている | 雛形生成 CLI |
| 5–10 分 | `k1s0 dev start` | ブラウザでサービスが動いている | `k1s0 dev` CLI |
| 10–25 分 | 生成されたサンプル API にリクエストを送信 | レスポンスが返り、ログがダッシュボードに表示される | curl / Tilt ダッシュボード |
| 25–40 分 | `k1s0.Log.Info` で構造化ログを追加 | 自分が追加したログがダッシュボードにリアルタイム表示 | エディタ + ダッシュボード |
| 40–55 分 | 簡単なビジネスロジック (入力を加工して返す API) を追加 | ファイル保存 → 数秒で自動反映 → リクエストで動作確認 | エディタ + curl |
| 55–60 分 | `k1s0 dev status` で全体を確認 | 「自分が書いたサービスが動いている」達成感 | `k1s0 dev` CLI |

### 3.3 設計上の制約

| 制約 | 目的 |
|---|---|
| `--level 1` で生成する | L1 API (Log / Telemetry / Auth) のみが見える状態にし、情報量を最小化 |
| 最初のコード変更はログ追加 | 最も低リスクで即座にフィードバックが得られる操作から始める |
| gRPC / Protobuf は登場しない | L1 段階では生成済みクライアントを使うだけ。Proto ファイルに触れない |
| エラーは意図的に発生させない | 最初の体験は「全てがうまくいく」成功体験とする |

### 3.4 オンボーディング (L1) との関係

ファーストコンタクト体験は [`07_tier2オンボーディング戦略.md`](07_tier2オンボーディング戦略.md) の L1 レベルの**最初の 1 時間**に相当する。ファーストコンタクト完了後、L1 の残りの学習 (Auth の詳細、Telemetry の活用) に進む。

```
ファーストコンタクト体験 (60 分)
  ↓ 「動くものが作れた」達成感
L1: 基礎 (残り 1 日)
  ↓ Log / Telemetry / Auth の習得
L2: データ (2–3 日)
  ↓ ...以降は既存のオンボーディング戦略に合流
```

## 4. チャンピオン制度

### 4.1 概要

変化を上から押し付けるのではなく、**tier2 開発者の中から推進者 (チャンピオン) を育て、現場からの推進力を作る**。チャンピオンは「最初に新技術を使いこなした同僚」であり、他の開発者にとって「あの人ができるなら自分もできるかも」というロールモデルになる。

### 4.2 チャンピオンの選定

| 要素 | 内容 |
|---|---|
| **選定基準** | 新技術への関心が高い開発者。ペルソナ「[挑戦者 森田](../../../01_企画/01_背景と目的/01_ペルソナ.md)」タイプ |
| **人数** | 各チームから 1 名。全体で 3–5 名 |
| **選定方法** | 自薦 + 上長推薦。強制ではなく希望者優先 |
| **前提条件** | 現行業務の C# / REST 経験があること (既存チームへの共感を持てること) |

### 4.3 育成プログラム

チャンピオンは一般の tier2 開発者より **2 週間先行**してトレーニングを受ける。

| 週 | 内容 | 狙い |
|---|---|---|
| 1 週目 | L1–L3 の集中習得 + 痛みの追体験ワークショップ参加 | 基礎 API とイベント駆動の実践 |
| 2 週目 | L4–L5 の概念理解 + tier1 チームとの QA セッション | 高度な概念への不安を事前解消 |

育成期間中にチャンピオンが感じた不安・疑問は**全て記録**し、一般開発者の FAQ として整備する。チャンピオン自身が「最初は俺もここで詰まった」と言えることが重要。

### 4.4 チャンピオンの役割

| 役割 | 内容 | 頻度 |
|---|---|---|
| **チーム内 QA 対応** | 自チームの開発者からの質問に最初に対応 | 日常的 |
| **ペアプログラミング** | 初回の tier1 API 利用時にペアプロで支援 | 各開発者の L1 開始時 |
| **ワークショップファシリテーション** | 痛みの追体験ワークショップの進行役 | 新メンバー参画時 |
| **改善フィードバック集約** | 現場の不満・要望を集約して tier1 チームに報告 | 週次 |
| **成功事例の共有** | 「k1s0 でこれが楽になった」事例をチーム内で共有 | 随時 |

### 4.5 チャンピオンへの支援

| 支援 | 内容 |
|---|---|
| **tier1 チームへの直接エスカレーション権** | 通常のチケットフローを経由せず、Slack / Teams で直接質問可能 |
| **API 改善提案権** | tier1 公開 API の改善を直接提案でき、tier1 チームが優先的にレビュー |
| **チャンピオン同士のコミュニティ** | 月 1 回のチャンピオン定例で情報共有・困りごとの相互支援 |
| **人事評価への反映** | チャンピオン活動を「技術貢献」として人事評価に反映するよう上長と事前合意 |

### 4.6 チャンピオンの効果

> tier1 チーム: 「k1s0 を使えば楽になります」
> → 開発者: 「上が言ってるだけだろ」

> チャンピオン (同僚): 「最初は俺も不安だったけど、実際に使ったら認証とログで 3 日かかってた作業が 30 分で終わった」
> → 開発者: 「マジで?」

同じメッセージでも**誰が言うか**で受容度が変わる。経営層・アーキテクトの言葉より、隣の席の同僚の体験談のほうが強い。チャンピオン制度はこの心理的メカニズムを構造的に活用する。

## 5. 3 施策の実施順序と関係性

3 施策は独立ではなく、以下の順序で段階的に効果を発揮する。

### 5.1 実施順序

```
採用後の運用拡大時 キックオフ前:
  1. チャンピオン選定・育成 (2 週間)
     ↓
  2. 痛みの追体験ワークショップ (チャンピオンがファシリテート)
     ↓ 「楽になる」ことへの納得
採用後の運用拡大時 開始:
  3. ファーストコンタクト体験 (各開発者の初日)
     ↓ 「自分でもできる」実感
  4. オンボーディング L1–L5 (既存戦略に合流)
     ↓ チャンピオンが日常的に支援
```

### 5.2 各施策が対処する心理段階

| 心理段階 | 開発者の内心 | 対処する施策 |
|---|---|---|
| **無関心** | 「関係ないでしょ」 | ワークショップへの招待 (チャンピオンが声をかける) |
| **抵抗** | 「前のやり方でいいじゃん」 | 痛みの追体験ワークショップ |
| **不安** | 「でも自分にできるかな」 | ファーストコンタクト体験 |
| **試行** | 「やってみるか」 | オンボーディング L1–L5 + チャンピオンのペアプロ |
| **定着** | 「もう前には戻れない」 | 日常の開発で恩恵を実感 |

### 5.3 施策の効果測定

| 指標 | 計測方法 | 目標値 |
|---|---|---|
| **ワークショップ後の意識変化** | ワークショップ前後のアンケート (5 段階) | 「k1s0 を使いたい」が 4.0 以上 |
| **ファーストコンタクト完了率** | 60 分以内に動くサービスを作れた割合 | 95% 以上 |
| **L3 到達率 (1 ヶ月以内)** | オンボーディング開始から L3 到達までの期間 | 80% の開発者が 1 ヶ月以内 |
| **チャンピオンへの質問数推移** | Slack / Teams での質問カウント | 3 ヶ月後に 50% 減少 (自己解決が増える) |
| **tier1 API への改善提案数** | チャンピオン経由の提案数 | 月 3 件以上 |

## 6. 段階的導入

| 段階 | 施策 | 備考 |
|---|---|---|
| 採用後の運用拡大時 キックオフ 4 週間前 | チャンピオン候補の選定・打診 | 上長との評価反映の合意を含む |
| 採用後の運用拡大時 キックオフ 2 週間前 | チャンピオン育成開始 (L1–L5 先行トレーニング) | tier1 チームが直接指導 |
| 採用後の運用拡大時 キックオフ時 | 痛みの追体験ワークショップ実施 | チャンピオンがファシリテート |
| 採用後の運用拡大時 開始 | ファーストコンタクト体験の提供開始 | 各開発者の参画初日に実施 |
| 採用後の運用拡大時 (月次) | チャンピオン定例の実施 | 改善フィードバックの集約と共有 |
| 採用後の運用拡大時 終了時 | 効果測定 (5.3 節) | 結果に基づいて 採用側のマルチクラスタ移行時 の施策を調整 |
| 採用側のマルチクラスタ移行時 | ワークショップ教材の更新 | L4–L5 の内容を反映 |
| 採用側のマルチクラスタ移行時 | 第 2 期チャンピオンの育成 | チーム拡大に合わせて増員 |
