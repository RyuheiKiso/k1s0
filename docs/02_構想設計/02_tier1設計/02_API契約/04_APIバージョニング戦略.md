# API バージョニング戦略

## 目的

tier1 公開 API は tier2 / tier3 が依存する **唯一の接点** である。この API に破壊的変更が入った場合の影響範囲は全サービスに及ぶ。本資料では tier1 公開 API のバージョニングスキーム、破壊的変更の扱い、移行支援の仕組みを定義する。

---

## 1. バージョニングスキーム

### 1.1 Semantic Versioning (SemVer) の採用

tier1 公開 API は **SemVer 2.0.0** (`MAJOR.MINOR.PATCH`) に従う。

| 変更の種類 | SemVer の扱い | 例 |
|---|---|---|
| 破壊的変更 (後方互換性なし) | MAJOR バージョン | `k1s0.Log.Info` のシグネチャ変更 |
| 後方互換の機能追加 | MINOR バージョン | `k1s0.Decision` API の新メソッド追加 |
| バグ修正 / パフォーマンス改善 | PATCH バージョン | 内部実装の修正 (API 契約変更なし) |

### 1.2 バージョンの適用単位

| 単位 | バージョニング | 理由 |
|---|---|---|
| tier1 公開 API 全体 | **統一バージョン** | tier2 / tier3 から見える「k1s0 ライブラリ」として 1 つのバージョンを持つ |
| 各言語クライアントライブラリ | 統一バージョンに追従 | C# / Go / TS のライブラリは同一バージョン番号をリリース |
| tier1 内部実装 (Go / Rust) | 独立バージョン | 内部実装のバージョンは tier2 / tier3 に公開しない |
| gRPC Protobuf 契約 | API バージョンと同期 | `src/tier1/contracts/` の `.proto` ファイルが正の定義 |

### 1.3 バージョン表記の場所

| 場所 | 表記 |
|---|---|
| NuGet パッケージ (C#) | `K1s0.Client` パッケージのバージョン |
| Go module | `go.k1s0.dev/client` の `go.mod` バージョンタグ |
| npm パッケージ (TS) | `@k1s0/client` パッケージのバージョン |
| Protobuf package | `k1s0.api.v1` → `k1s0.api.v2` (MAJOR 変更時のみ) |
| HTTP API パス (将来) | `/api/v1/...` → `/api/v2/...` |

## 2. 破壊的変更のポリシー

### 2.1 破壊的変更の定義

以下のいずれかに該当する変更を「破壊的」と定義する。

- 既存メソッドのシグネチャ変更 (引数の型・数・順序、戻り値の型)
- 既存メソッドの削除
- 既存メソッドの挙動変更 (同一入力に対する出力の変化)
- Protobuf メッセージのフィールド型変更 / 必須フィールドの追加
- エラーコード体系の変更

### 2.2 破壊的変更を回避する設計指針

tier1 公開 API は破壊的変更を**極力避ける**設計とする。

| 指針 | 手段 |
|---|---|
| メソッドの追加は破壊的でない | 新メソッドを追加し、既存メソッドは残す |
| 引数の拡張はオプション型で | Protobuf の optional フィールド / C# のデフォルト引数 |
| 戻り値の拡張は構造体に | プリミティブ型ではなく構造体を返し、フィールド追加で対応 |
| 列挙型の追加は安全 | 新しい値の追加は後方互換 |
| 内部実装の差し替えは自由 | Dapr → 別実装への切替は API 契約を変えない限り非破壊 |

### 2.3 破壊的変更が不可避な場合のプロセス

1. **提案**: tier1 チームが RFC (Request for Comments) を `docs/` に起票する
2. **影響分析**: 全 tier2 / tier3 サービスへの影響を CI で自動検出する
3. **非推奨化 (Deprecation)**: 旧 API に `[Obsolete]` / `// Deprecated:` を付与し、最低 **1 Minor バージョン** は並行提供する
4. **移行ツール提供**: 可能な限り、旧 API → 新 API の自動書き換えツール (codemod) を提供する
5. **移行期間**: 非推奨化から削除まで最低 **3 ヶ月** の猶予を設ける
6. **削除**: 全 tier2 / tier3 サービスが移行完了したことを CI で確認後、MAJOR バージョンを上げて削除する

## 3. 並行バージョン運用

### 3.1 MAJOR バージョン共存

MAJOR バージョンが上がった場合、旧バージョンと新バージョンの tier1 サービスを**一定期間並行稼働**させる。

| 要素 | v1 | v2 |
|---|---|---|
| Protobuf package | `k1s0.api.v1` | `k1s0.api.v2` |
| tier1 gRPC エンドポイント | `/k1s0.api.v1.LogService/Info` | `/k1s0.api.v2.LogService/Info` |
| クライアントライブラリ | `K1s0.Client` v1.x | `K1s0.Client` v2.x |
| tier1 内部サービス | v1 ハンドラと v2 ハンドラを同一サービスで提供 | — |

### 3.2 並行稼働の制約

- 並行稼働期間は **最大 6 ヶ月**。超過した場合は旧バージョンを強制終了する
- 並行稼働中は旧バージョンにバグ修正のみ提供し、機能追加は新バージョンのみ
- 並行稼働のコストを最小化するため、MAJOR 変更は **年 1 回以下** を目標とする

## 4. Protobuf 契約管理

### 4.1 Protobuf を正の契約とする理由

| 観点 | Protobuf | 言語固有の interface |
|---|---|---|
| 言語非依存性 | 1 つの `.proto` から C# / Go / TS を自動生成 | 各言語で個別管理が必要 |
| 後方互換の検証 | `buf breaking` で機械的に検証可能 | 人手のレビューに依存 |
| ドキュメント生成 | `buf` / `protoc-gen-doc` で API ドキュメントを自動生成 | 別途記述が必要 |

### 4.2 契約の管理フロー

1. `src/tier1/contracts/*.proto` が全 API の正の定義
2. CI で `buf lint` (スタイル検証) + `buf breaking` (後方互換検証) を実行
3. 違反した PR は自動でブロックされる
4. 各言語のクライアントライブラリは `buf generate` で自動生成する

## 5. クライアントライブラリの配信

| 言語 | パッケージ管理 | 配信先 | 備考 |
|---|---|---|---|
| C# | NuGet | 社内 NuGet フィード (Harbor 付属 or Azure Artifacts) | `K1s0.Client` |
| Go | Go Modules | 社内 Git リポジトリ (go-import meta tag) | `go.k1s0.dev/client` |
| TypeScript | npm | 社内 npm レジストリ (Verdaccio or Harbor) | `@k1s0/client` |

ライブラリのリリースは tier1 の CI パイプラインで自動化する。Protobuf 変更 → コード生成 → テスト → パッケージ公開 を 1 本のワークフローで実行する。

---

## 6. OSS リリース時点での採用範囲

| 項目 | リリース時点 で実施 | 採用後の運用拡大時 |
|---|---|---|
| SemVer 運用開始 | 対象 (v0.x で開始、安定後に v1.0) | — |
| Protobuf 契約管理 | 対象 (`src/tier1/contracts/`) | — |
| `buf lint` / `buf breaking` CI | 対象 | — |
| C# クライアント自動生成 | 対象 (1 言語から開始) | Go / TS 追加 |
| 非推奨化プロセス | — (v0.x では破壊的変更を許容) | v1.0 以降で適用 |
| 並行バージョン運用 | — | MAJOR 変更発生時 |
| 自動 codemod | — | 採用側のマルチクラスタ移行時 |

### リリース時点 での特則

リリース時点 (v0.x) では SemVer の慣例に従い、**MINOR バージョンでの破壊的変更を許容する**。ただし変更時は CHANGELOG に明記し、tier2 / tier3 への通知を必須とする。v1.0 以降は本資料のポリシーを厳格に適用する。
