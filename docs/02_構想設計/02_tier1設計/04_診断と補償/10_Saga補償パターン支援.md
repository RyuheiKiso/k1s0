# Saga 補償パターン支援

## 目的

tier2 開発者が結果整合性と Saga パターンに対して抱く恐怖を軽減する。恐怖の根源は「補償アクションを自分で書く責任」にある。DB トランザクションでは失敗時のロールバックが自動だったが、Saga では「何を戻すか」を開発者自身が定義しなければならない。

本資料では**補償パターンを 3 段階に階層化**し、大半のケースで tier2 開発者の負担を最小化する設計を定義する。

---

## 1. 背景となる課題

tier2 開発者の想定バックグラウンドは C# + Entity Framework + SQL Server ([ペルソナ 5: 長谷川亮](../../../01_企画/01_背景と目的/01_ペルソナ.md))。従来のトランザクション処理と Saga パターンには以下の本質的な差異がある。

| 項目 | 従来 (DB トランザクション) | k1s0 (Saga パターン) |
|---|---|---|
| 失敗時の対処 | DB が自動ロールバック | 開発者が補償ロジックを定義 |
| 整合性の保証タイミング | 即時 (ACID) | 非同期 (結果整合性) |
| 対象範囲 | 単一 DB 内 | 複数サービスにまたがる |
| 精神的負荷 | 低い (「任せておけば安全」) | 高い (「自分が書かないと壊れる」) |

この差異に対して**技術的な支援なしに「Saga で書いてください」と要求する**ことは、tier2 開発者の生産性と心理的安全性を損なう。

---

## 2. 補償パターンの 3 段階階層化

補償ロジックの複雑度を 3 段階に分類し、大半のケースを tier1 側で吸収する。

### 2.1 段階定義

| 段階 | パターン名 | tier2 が書くもの | tier1 が担うもの | 適用比率 (想定) |
|---|---|---|---|---|
| **L1: 自動補償** | 冪等リトライ | 何も書かない | リトライ・ロールバックを自動実行 | 60% |
| **L2: 宣言的補償** | 補償ペア宣言 | 「この操作の逆は何か」をペアで宣言 | 実行順序・リトライ・タイムアウトを管理 | 30% |
| **L3: カスタム補償** | 完全自作 | 補償ロジック全体 | 実行タイミングの保証のみ | 10% |

### 2.2 L1: 自動補償

tier1 API の内部で自動的にリトライと巻き戻しを行う。tier2 開発者は何も書かない。

**対象となる操作**:

| API | 操作 | tier1 の自動補償動作 |
|---|---|---|
| `k1s0.State.SaveAsync` | KV ストアへの書き込み | リトライ上限超過時に例外を投げる。部分書き込みなし (原子的) |
| `k1s0.PubSub.PublishAsync` | イベント発行 | リトライ上限超過時に例外を投げる。二重発行防止 (冪等キー) |
| `k1s0.Audit.RecordAsync` | 監査ログ記録 | fire-and-forget。失敗してもビジネスロジックに影響しない |

L1 の操作は**単一リソースへの操作**であり、分散トランザクションが不要なもの。tier2 開発者はエラーハンドリング方針 ([`08_診断機能設計.md`](08_診断機能設計.md) 4 章) に従うだけでよい。

### 2.3 L2: 宣言的補償

複数サービスにまたがる操作を**補償ペア宣言**で定義する。tier2 開発者は「実行」と「その逆操作」をペアで宣言するだけであり、以下の責務は tier1 が担う。

| tier1 が担う責務 | 内容 |
|---|---|
| 実行順序管理 | 宣言されたステップを順番に実行 |
| 失敗時の逆順補償 | 失敗したステップより前の全ステップの補償を逆順に実行 |
| 各ステップのリトライ | 設定されたリトライポリシーに基づく自動リトライ |
| タイムアウト管理 | 各ステップのタイムアウト監視と切断 |
| 冪等性保証 | 補償の二重実行を防止 |
| 状態の永続化 | Saga の進行状態を Temporal に永続化 |

**API 設計**:

```csharp
// tier2 が書くコード — 補償ペア宣言
var result = await k1s0.Workflow.RunSagaAsync("order.create", saga =>
{
    saga.Step("在庫確保",
        execute: () => k1s0.Service.Invoke("inventory", "Reserve", reserveReq),
        compensate: () => k1s0.Service.Invoke("inventory", "Release", releaseReq));

    saga.Step("決済承認",
        execute: () => k1s0.Service.Invoke("payment", "Authorize", authReq),
        compensate: () => k1s0.Service.Invoke("payment", "Cancel", cancelReq));

    saga.Step("注文確定",
        execute: () => k1s0.Service.Invoke("order", "Confirm", confirmReq),
        compensate: () => k1s0.Service.Invoke("order", "Revert", revertReq));
});
```

**失敗時の動作例** (「決済承認」で失敗した場合):

| 順序 | 動作 | 結果 |
|---|---|---|
| 1 | `在庫確保` — execute | 成功 |
| 2 | `決済承認` — execute | **失敗** |
| 3 | `在庫確保` — **compensate** | tier1 が自動実行 (在庫解放) |
| 4 | Saga 全体のエラーを tier2 に返却 | `K1s0SagaFailedException` |

tier2 開発者は「在庫確保の逆は在庫解放」「決済承認の逆は決済取消」というビジネス知識を宣言するだけであり、実行制御には関与しない。

### 2.4 L3: カスタム補償

業務固有の複雑な巻き戻しが必要なケース。Temporal の Workflow / Activity を直接利用する。

**L3 が必要なケースの特徴**:

| 特徴 | 例 |
|---|---|
| 補償ロジックに条件分岐がある | 「承認済みなら上長に取消通知、未承認なら何もしない」 |
| 補償の順序が実行の逆順ではない | 「外部 API の取消は最後にまとめて実行」 |
| 補償の成否に応じた分岐がある | 「決済取消が失敗したら手動対応キューに投入」 |

**L3 の支援方針**:

L3 は tier2 開発者が単独で実装するのではなく、**tier1 チームとのペアプログラミング**で実装する。これは L3 パターンが全体の約 10% と想定されること、および誤った補償ロジックの影響が大きいことによる。

| 支援内容 | 担当 |
|---|---|
| Workflow / Activity の設計レビュー | tier1 チーム |
| 補償ロジックの実装 | tier2 開発者 (tier1 チームがペアプロで支援) |
| 補償テストの実行と確認 | tier2 開発者 (3 章のテストランナーを使用) |

## 3. Saga テストランナー

「書いた補償が本当に動くか」という不安を解消するため、ローカル環境で意図的に各ステップを失敗させるテストツールを提供する。

### 3.1 コマンド体系

```bash
# 特定のステップで失敗を注入し、補償動作を検証
k1s0-saga-test --workflow "order.create" --fail-at "決済承認"

# 全ステップで順番に失敗を注入し、全補償パスを網羅テスト
k1s0-saga-test --workflow "order.create" --fail-all-steps

# 補償実行中に二重失敗を注入し、リトライ動作を検証
k1s0-saga-test --workflow "order.create" --fail-at "決済承認" --fail-compensate "在庫確保"
```

### 3.2 出力形式

```
k1s0 Saga テスト: order.create
──────────────────────────────
  テストケース: "決済承認" で失敗を注入

  [EXEC] 在庫確保         成功  (32ms)
  [EXEC] 決済承認         失敗  (注入された失敗)
  [COMP] 在庫確保         成功  (18ms) — 在庫解放を実行

  結果: 補償完了
  全ステップの状態が初期状態に戻りました
```

全ステップ網羅テストの出力:

```
k1s0 Saga テスト: order.create (全ステップ網羅)
──────────────────────────────────────────────
  [1/3] "在庫確保" で失敗を注入
        補償対象なし (最初のステップ)                   合格

  [2/3] "決済承認" で失敗を注入
        [COMP] 在庫確保 — 成功                         合格

  [3/3] "注文確定" で失敗を注入
        [COMP] 決済承認 — 成功
        [COMP] 在庫確保 — 成功                         合格

  結果: 3/3 テストケース合格
```

### 3.3 CI への組み込み

Saga テストは CI パイプラインにも組み込み可能とする。Saga 定義を変更した PR では自動的にテストが実行される。

| 実行環境 | 実行タイミング | 目的 |
|---|---|---|
| ローカル (手動) | 開発中に随時 | 実装中の動作確認 |
| CI (自動) | Saga 定義を含むファイルが変更された PR | 補償パスの網羅性を保証 |

### 3.4 実装方針

Saga テストランナーは Temporal のテストフレームワーク (`go.temporal.io/sdk/testsuite`) を活用する。テスト用の Activity 実装を差し替え、指定されたステップで意図的にエラーを返す。

| コンポーネント | 実装 |
|---|---|
| テストランナー CLI | Rust (雛形生成 CLI と同一バイナリのサブコマンド) |
| Activity スタブ | Temporal TestSuite の mock 機能 |
| 結果レポート | 構造化 JSON + 人間可読なコンソール出力 |

## 4. DB トランザクションとの使い分け基準

結果整合性が必要なのは**サービス間**の操作であり、**単一サービス内**の DB 操作は従来通り ACID トランザクションで問題ない。この線引きを明確にすることで「全部 Saga にしなければならない」という誤解を防ぐ。

### 4.1 判断基準

| 操作の範囲 | 推奨する方法 | 理由 |
|---|---|---|
| 単一サービス内の DB 操作 | **DB トランザクション (従来通り)** | ACID で十分。Saga にする必要なし |
| 単一サービス内の DB + State 操作 | **DB トランザクション + 冪等リトライ (L1)** | State の操作は tier1 が自動補償 |
| 複数サービスにまたがる操作 | **Saga (L2: 補償ペア宣言)** | 分散トランザクションは ACID で実現不可能 |
| 複雑な業務ワークフロー | **Saga (L3: カスタム補償)** | tier1 チームと協力して実装 |

### 4.2 「迷ったら Saga を避ける」原則

| 指針 | 内容 |
|---|---|
| まずサービス分割の粒度を見直す | 2 つのサービスにまたがる操作を 1 つのサービスに統合できないか検討する |
| 非同期でよい操作を特定する | 「在庫確保」と「通知送信」は同一トランザクションに含める必要があるか？ 通知は非同期 (PubSub) で十分ではないか |
| Saga の出番自体を減らす設計 | Saga が必要な箇所を最小化する設計が最善。Saga は「最後の手段」であり「標準パターン」ではない |

### 4.3 パターン別コード対比

**パターン A: 単一サービス内 (DB トランザクション)**

```csharp
// 従来通りの書き方がそのまま使える
using var tx = await _db.BeginTransactionAsync();
var order = new Order { ... };
await _db.Orders.AddAsync(order);
await _db.SaveChangesAsync();
await tx.CommitAsync();

// tier1 API はトランザクションの外で呼ぶ
await k1s0.Log.Info("注文作成完了", new { orderId = order.Id });
await k1s0.Audit.RecordAsync("order.created", new { orderId = order.Id });
```

**パターン B: 複数サービス間 (補償ペア宣言)**

```csharp
// 複数サービスをまたぐ場合のみ Saga を使う
await k1s0.Workflow.RunSagaAsync("order.create", saga =>
{
    saga.Step("在庫確保",
        execute: () => k1s0.Service.Invoke("inventory", "Reserve", req),
        compensate: () => k1s0.Service.Invoke("inventory", "Release", req));

    saga.Step("決済承認",
        execute: () => k1s0.Service.Invoke("payment", "Authorize", req),
        compensate: () => k1s0.Service.Invoke("payment", "Cancel", req));
});
```

## 5. オンボーディングとの統合

[`07_tier2オンボーディング戦略.md`](../03_開発者体験/07_tier2オンボーディング戦略.md) の習熟度レベルとの対応関係を明示する。

| 習熟度レベル | 補償パターン段階 | tier2 開発者が学ぶこと |
|---|---|---|
| L1–L2 | **L1 (自動補償) のみ** | 補償を意識する必要なし。tier1 が自動処理 |
| L3 | **L1 + L2 の概念理解** | 「補償ペア」の考え方を学ぶ。ただし実装はまだしない |
| L4 | **L2 (補償ペア宣言) の実装** | 補償ペアを使った Saga の実装。テストランナーで検証 |
| L5 | **L3 (カスタム補償) への理解** | 複雑な補償が必要なケースの存在を知る。実装は tier1 チームと共同 |

L1–L3 の学習段階で Saga の概念に触れることはない。L4 に進んだ段階で初めて「補償ペア宣言」を実装するが、その時点で tier1 API の基本操作 (State / PubSub / Service.Invoke) に十分習熟している状態を前提とする。

## 6. 段階的導入

| フェーズ | 施策 | 備考 |
|---|---|---|
| Phase 2 前半 | L1 (自動補償) を全 API に組み込み | tier2 開発開始の前提条件 |
| Phase 2 前半 | `k1s0.Workflow.RunSagaAsync` の L2 API を実装 | Temporal バックエンド |
| Phase 2 後半 | Saga テストランナーの CLI 実装 | `--fail-at` と `--fail-all-steps` |
| Phase 2 後半 | L4 オンボーディングサンプルに補償ペア宣言の例を追加 | 橋渡しドキュメント更新 |
| Phase 3 | Saga テストの CI 組み込み | Saga 定義変更時の自動実行 |
| Phase 3 | L3 パターンの初回実装 (tier1 + tier2 ペアプロ) | 実績を蓄積してパターン集に反映 |
