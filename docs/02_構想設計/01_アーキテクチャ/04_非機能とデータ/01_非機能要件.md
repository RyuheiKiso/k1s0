# 非機能要件

## 目的

k1s0 プラットフォームの非機能要件 (パフォーマンス / 可用性 / スケーラビリティ / 運用性) を定義する。JTC 情シス部門の実情を踏まえ、過剰な要件は設定せず、Phase ごとに段階的に引き上げる。

---

## 1. 前提条件

| 項目 | 前提 |
|---|---|
| 利用者規模 (MVP) | 開発者 5〜10 名、エンドユーザー 数十名 |
| 利用者規模 (Phase 5) | 開発者 30〜50 名、エンドユーザー 数百〜千名 |
| 稼働時間 | 平日 8:00〜20:00 が主。夜間バッチは一部業務のみ |
| ネットワーク | オンプレ LAN (1〜10 Gbps)。外部接続は制限付き |
| 同時リクエスト数 (MVP) | tier1 API へのリクエスト: ピーク時 50 req/s |
| 同時リクエスト数 (Phase 5) | tier1 API へのリクエスト: ピーク時 500 req/s |

---

## 2. パフォーマンス要件

### 2.1 tier1 公開 API のレイテンシ目標

| API カテゴリ | p50 | p99 | 備考 |
|---|---|---|---|
| `k1s0.Log.*` | < 5 ms | < 50 ms | 非同期送信。呼び出し側をブロックしない |
| `k1s0.Telemetry.*` | < 5 ms | < 50 ms | 同上 |
| `k1s0.State.*` (Get/Save) | < 10 ms | < 100 ms | Valkey バックエンド前提 |
| `k1s0.PubSub.Publish` | < 20 ms | < 200 ms | Kafka へのプロデュース |
| `k1s0.Secrets.Get` | < 10 ms | < 100 ms | キャッシュ前提 |
| `k1s0.Service.Invoke` | < 50 ms + 宛先処理時間 | < 200 ms + 宛先処理時間 | Istio mTLS のオーバーヘッド含む |
| `k1s0.Decision.Evaluate` | < 20 ms | < 100 ms | ZEN Engine in-process 評価 |
| `k1s0.Workflow.Start` | < 100 ms | < 500 ms | Dapr Workflow の起動コスト |
| `k1s0.Auth.ValidateToken` | < 10 ms | < 50 ms | Keycloak JWKS キャッシュ前提 |
| `k1s0.Audit.Record` | < 10 ms | < 100 ms | 非同期永続化 |

### 2.2 infra コンポーネントのパフォーマンス目標

| コンポーネント | 指標 | 目標値 |
|---|---|---|
| Envoy Gateway | リクエストルーティング遅延 | < 5 ms (p99) |
| Istio sidecar | mTLS ハンドシェイク追加遅延 | < 3 ms (p99、接続確立済みの場合) |
| Kafka | プロデュース ACK 遅延 | < 50 ms (acks=all, p99) |
| PostgreSQL | 単純クエリ (SELECT by PK) | < 5 ms (p99) |
| Valkey | GET / SET | < 1 ms (p99) |

### 2.3 CI/CD パイプラインのパフォーマンス目標

| 指標 | 目標値 | 備考 |
|---|---|---|
| PR 時ビルド + テスト | < 10 分 | ユニットテスト + Lint + 型チェック |
| main マージ後ビルド → Harbor push | < 15 分 | コンテナビルド + スキャン含む |
| Argo CD 同期 (マニフェスト適用) | < 5 分 | GitOps 反映 |
| 雛形生成 CLI 実行 | < 5 秒 | 新規サービスの雛形出力 |

## 3. 可用性要件

### 3.1 可用性目標

| 対象 | Phase 1 (MVP) | Phase 3 | Phase 5 |
|---|---|---|---|
| tier1 公開 API | 99% (月間ダウンタイム 7.2 時間以内) | 99.5% | 99.9% |
| infra (k8s / Istio / Kafka) | 99% | 99.5% | 99.9% |
| 配信ポータル | 95% (業務時間帯のみ) | 99% | 99.5% |
| Backstage | 95% | 99% | 99.5% |
| CI/CD パイプライン | 95% | 99% | 99% |

### 3.2 可用性の実現手段

| 手段 | 対象 | 効果 |
|---|---|---|
| k8s ReplicaSet | 全ステートレスサービス | Pod 障害時の自動再起動 |
| CloudNativePG HA | PostgreSQL | プライマリ障害時のレプリカ自動昇格 |
| Kafka レプリケーション | メッセージング | ブローカー障害時のデータ保護 |
| Istio リトライ / サーキットブレーカー | サービス間通信 | 一時的な障害の自動回復 |
| PodDisruptionBudget | 重要サービス | ローリングアップデート時の最低稼働数保証 |

### 3.3 計画メンテナンスの扱い

| 項目 | 方針 |
|---|---|
| メンテナンスウィンドウ | 月 1 回、平日夜間 22:00〜翌 6:00 |
| 無停止更新 | ステートレスサービスはローリングアップデートで無停止 |
| 停止を伴う更新 | k8s / Istio / Kafka のメジャーアップデートはメンテナンスウィンドウで実施 |
| 可用性の計算 | 計画メンテナンスは可用性の計算から**除外しない** (SLA に含める) |

## 4. スケーラビリティ要件

### 4.1 水平スケーリング

| コンポーネント | スケーリング方式 | スケーリングトリガー | 上限の目安 |
|---|---|---|---|
| tier1 Go サービス | HPA | CPU / RPS | 10 Pod |
| tier1 Rust サービス | HPA | CPU | 5 Pod |
| tier2 Consumer 系 | KEDA ScaledObject | Kafka Consumer Lag | サービスごとに決定 |
| tier2 同期 API 系 | KEDA ScaledObject | Prometheus HTTP RPS | サービスごとに決定 |
| tier3 Web アプリ | KEDA ScaledObject | Prometheus HTTP RPS | サービスごとに決定 |
| Temporal Worker | KEDA ScaledObject | Task Queue バックログ | 10 Pod |
| Kafka | ブローカー追加 + パーティション増設 | — | 10 ブローカー |
| PostgreSQL | リードレプリカ追加 | — | 3 レプリカ |

tier1 はプラットフォーム基盤であり常時稼働が必須のため、KEDA の Scale-to-Zero 対象にしない。tier2 / tier3 のワークロードスケーリングに KEDA を使用し、イベント駆動の負荷に自動で追従する。KEDA の選定根拠は [`../../03_技術選定/03_周辺OSS/14_オートスケーラー.md`](../../03_技術選定/03_周辺OSS/14_オートスケーラー.md) を参照。

### 4.2 Phase 別のスケール計画

| Phase | ノード数 | 同時 Pod 数 (目安) | 備考 |
|---|---|---|---|
| Phase 1 | 3 | 30〜50 | infra + tier1 + operation |
| Phase 2 | 3〜5 | 50〜80 | tier2 サービス追加 |
| Phase 3 | 5〜8 | 80〜150 | staging / prod 分離 |
| Phase 5 | 10+ | 200+ | 全社ロールアウト |

## 5. 運用性要件

### 5.1 ログ / メトリクスの保持期間

| データ | 保持期間 (MVP) | 保持期間 (Phase 5) |
|---|---|---|
| メトリクス (Prometheus) | 15 日 | 90 日 (Thanos / Cortex で長期保管) |
| ログ (Loki) | 15 日 | 90 日 |
| 分散トレース (Grafana Tempo) | 7 日 | 30 日 |
| 監査ログ (tier1 Audit) | 365 日 | 365 日 (法的要件に応じて延長) |

### 5.2 運用自動化の目標

| 運用タスク | Phase 1 | Phase 3 以降 |
|---|---|---|
| サービスデプロイ | GitOps (Argo CD) 自動 | 同左 |
| スケーリング | 手動 | KEDA + HPA 自動 (Phase 2 から) |
| 証明書更新 | Istio CA 自動 | 同左 + cert-manager |
| バックアップ | CronJob 自動 | 同左 |
| OSS アップデート | 手動 (メンテナンスウィンドウ) | Renovate Bot で PR 自動作成 |
| アラート対応 | 手動 (Runbook 参照) | 一部自動修復 (PDB + 自動再起動) |

### 5.3 ドキュメント要件

| ドキュメント | 形式 | 更新タイミング |
|---|---|---|
| 運用手順書 (Runbook) | TechDocs (Backstage) | アラートルール変更時 |
| 障害対応記録 (Postmortem) | TechDocs | インシデント解決後 5 営業日以内 |
| 構成変更履歴 | Git コミット + ADR | 変更の都度 |

## 6. セキュリティ関連の非機能要件

セキュリティの詳細は [`04_セキュリティモデル.md`](../03_セキュリティ/01_セキュリティモデル.md) を参照。ここでは非機能的な側面のみ記載する。

| 要件 | 目標 |
|---|---|
| CVE 対応 (Critical) | 検知から 48 時間以内にパッチ適用またはワークアラウンド |
| CVE 対応 (High) | 検知から 7 日以内 |
| Secret ローテーション | 90 日ごと (Phase 3 以降で自動化) |
| 認証トークン有効期限 | アクセストークン: 5 分、リフレッシュトークン: 8 時間 |

## 7. MVP 非機能要件のサマリ

MVP (Phase 1) で達成すべき非機能要件の最小セットを整理する。

| 分類 | MVP 目標 |
|---|---|
| tier1 API p99 レイテンシ | 全 API < 500 ms |
| 可用性 | 99% (業務時間帯) |
| データ保持 | メトリクス / ログ 15 日、監査 365 日 |
| バックアップ RPO | PostgreSQL: 数秒、etcd: 24 時間 |
| バックアップ RTO | PostgreSQL: 15 分、クラスタ全壊: 4 時間 |
| CI パイプライン | PR ビルド < 10 分 |
| CVE 対応 | Critical: 48 時間以内 |

Phase 2 以降で各目標を段階的に引き上げる。
