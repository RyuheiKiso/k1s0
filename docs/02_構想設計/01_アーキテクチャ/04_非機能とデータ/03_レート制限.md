# レート制限

## 目的

tier2 / tier3 サービスのバグや過負荷が tier1 を巻き込むカスケード障害を防止する。グレースフルデグラデーション ([`08_グレースフルデグラデーション.md`](../02_可用性と信頼性/03_グレースフルデグラデーション.md)) は infra コンポーネント障害時の縮退を定義しているが、**上位層 (tier2/tier3) が下位層 (tier1) を過負荷にするシナリオ** が未対策であった。本資料でその対処を定義する。

---

## 1. 解決する問題

### 1.1 脅威シナリオ

tier2 の業務サービスに無限ループバグが混入した場合を想定する。

1. tier2 サービスが `k1s0.State.Save()` を毎秒 1000 回呼び出す
2. tier1 Go ファサードが全リクエストを Dapr → Valkey に転送
3. Valkey の CPU が 100% に到達
4. `k1s0.State` が全サービスに対して応答遅延
5. `k1s0.Settings` (同じ Valkey バックエンド) も影響
6. グレースフルデグラデーション 4.1 の「Valkey 障害」カスケードが発生

これは infra の障害ではなく、**tier2 の 1 つのバグが tier1 全体を巻き込むカスケード障害** である。

### 1.2 既存設計でカバーされない理由

グレースフルデグラデーション.md のカスケード障害マトリクス (セクション 4.1) は「障害元」を infra コンポーネント (Valkey / Kafka / PostgreSQL / Keycloak 等) として列挙している。しかし **tier2/tier3 の暴走が障害元となるパターン** が含まれていない。

---

## 2. レート制限の 3 層構造

| 層 | 配置 | 制御対象 | 効果 |
|---|---|---|---|
| **L1: Envoy Gateway** | infra namespace | 外部クライアント → k8s 内部 | エンドユーザー由来の過負荷を水際で止める |
| **L2: tier1 Go ファサード** | tier1 namespace | tier2/tier3 サービス → tier1 API | 1 つの tier2 の暴走が他サービスに波及しない |
| **L3: バックエンド保護** | tier1 内部 | tier1 → Valkey / Kafka / PostgreSQL | 共有バックエンドの容量を保護する |

### 2.1 L1: Envoy Gateway レート制限

| 項目 | 仕様 |
|---|---|
| 制限単位 | IP アドレス / 認証ユーザー |
| デフォルト制限 | 50 req/s/user |
| 超過時の応答 | HTTP 429 Too Many Requests |
| 実装 | Envoy Gateway の `RateLimitFilter` (k8s Gateway API 拡張) |
| バックエンド | Valkey (Rate Limit カウンタ保存) |
| 導入時期 | Phase 2 (Envoy Gateway 導入と同時) |

### 2.2 L2: tier1 Go ファサードのサービス単位レート制限

| 項目 | 仕様 |
|---|---|
| 制限単位 | サービスアカウント (k8s ServiceAccount) |
| デフォルト制限 | 100 req/s/service (API カテゴリ別に調整可能) |
| 超過時の応答 | gRPC `RESOURCE_EXHAUSTED` / HTTP 429 |
| 実装 | tier1 Go ファサード内のミドルウェア (token bucket アルゴリズム) |
| バックエンド | Valkey (分散 rate limiter) |
| 設定変更 | ConfigMap で宣言。サービスごとの個別制限も可能 |
| 導入時期 | Phase 2 |

### API カテゴリ別のデフォルト制限

| API カテゴリ | デフォルト制限 (req/s/service) | 根拠 |
|---|---|---|
| 観測系 (`Log`, `Telemetry`, `Audit`) | 500 | 高頻度呼び出しが正常。制限は緩め |
| 参照系 (`Auth`, `Secrets`, `Decision`, `Settings`, `Feature`) | 200 | キャッシュ併用前提。キャッシュミス時の保護 |
| 状態系 (`State`, `PubSub`, `Workflow`, `Service.Invoke`) | 100 | バックエンドへの書き込み負荷が直接的 |

### 2.3 L3: バックエンド保護

| 項目 | 仕様 |
|---|---|
| 保護対象 | Valkey / Kafka / PostgreSQL |
| 制限方式 | tier1 Go ファサード内で、バックエンド別の同時実行数を制限 (semaphore) |
| Valkey 保護 | 同時書き込み 50 ops (State + Settings の合計) |
| Kafka 保護 | 同時 Produce 20 ops |
| PostgreSQL 保護 | PgBouncer が接続数を制御 (本レート制限の範囲外。データアーキテクチャ参照) |
| 導入時期 | Phase 2 |

## 3. 超過時の振る舞い

### 3.1 クライアントライブラリの対応

tier1 クライアントライブラリが HTTP 429 / gRPC `RESOURCE_EXHAUSTED` を受けた場合の動作:

| API 分類 | 超過時の動作 |
|---|---|
| 観測系 | **破棄**: ログ / テレメトリの送信を一時停止。業務処理は継続 |
| 参照系 | **キャッシュフォールバック**: ローカルキャッシュの値で継続 (グレースフルデグラデーションと同一) |
| 状態系 | **fail-fast**: `K1s0RateLimitExceededException` を throw。呼び出し元に判断を委ねる |

### 3.2 メトリクス

| メトリクス名 | 型 | 説明 |
|---|---|---|
| `k1s0_ratelimit_exceeded_total` | Counter | レート制限超過回数 (ラベル: `api`, `service`) |
| `k1s0_ratelimit_current_rate` | Gauge | 現在のリクエストレート (ラベル: `api`, `service`) |

### 3.3 アラート

| 条件 | 重要度 | 対応 |
|---|---|---|
| `k1s0_ratelimit_exceeded_total` が 5 分間で 100 超 | Warning | 該当 tier2/tier3 サービスのバグ調査 |
| 特定サービスが継続的に制限超過 | Critical | サービスの一時停止を検討 |

## 4. カスケード障害マトリクスへの追加

グレースフルデグラデーション.md セクション 4.1 のマトリクスに以下を追加する。

| 障害元 | 直接影響を受ける tier1 API | 間接影響 |
|---|---|---|
| **tier2/tier3 の暴走 (Rate 超過)** | 該当サービスのみ制限 | 他サービスへの影響なし (Rate Limit で隔離) |

Rate Limiting が正常に機能していれば、tier2/tier3 起因のカスケード障害は L2 で遮断される。これはグレースフルデグラデーション 4.3 のバルクヘッド (障害隔離) を補完する上位防御層として機能する。

---

## 5. Phase 別の導入計画

| Phase | 施策 |
|---|---|
| **Phase 1 (MVP)** | レート制限なし。tier2/tier3 サービスが 1~2 本のみであり、暴走リスクが低いため |
| **Phase 2** | L2 (tier1 Go ファサード) を導入。L1 (Envoy Gateway) は Envoy Gateway 導入と同時。L3 (バックエンド保護) を導入 |
| **Phase 3** | 運用実績に基づき閾値を調整。Grafana ダッシュボード「tier1 Rate Limit」を追加 |

---

## 6. 禁止パターン

| パターン | 理由 |
|---|---|
| tier2/tier3 側で独自の Rate Limiting を実装 | tier1 のポリシーと二重管理になり、閾値の不整合が発生する |
| Rate Limit の閾値をコードにハードコード | ConfigMap で宣言し、再デプロイなしで変更可能にする |
| 観測系 API のレート制限超過をエラーとして扱う | 観測系は fire-and-forget。超過時は静かに破棄する |
