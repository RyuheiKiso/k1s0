# 01_アーキテクチャ

k1s0 の全体像を「tier1 → tier2 → tier3 → infra」の 4 レイヤ構造で捉え、各レイヤが上位レイヤをどう楽にするかを可視化する章。本章のゴールは、読者が「この構造なら 01 章の痛みは解ける」と納得できる地点まで抽象度を下げること。個別 OSS の選定根拠には踏み込まず、構造上の決定（依存方向・配置形態・非機能要件）に限定する。

## 想定読者

アーキテクト級の技術リード、インフラチーム、tier1 実装担当。経営層は [`01_基礎/00_概念アーキテクチャ.md`](./01_基礎/00_概念アーキテクチャ.md) の全体図と [`../全体構成図.md`](../../01_企画/全体構成図.md) だけで全体像を掴める粒度で書いている。

## なぜサブディレクトリ構造にしたか

17 本のフラットな md を同一階層に並べると、以下の問題が発生する。

- **読み順が番号依存になる** — `05_障害復旧` の次に `06_プラットフォーム自己監視` が来て `07_非機能要件` にジャンプするような、ドメインの切り替わりが番号から読み取れない。
- **関連ファイルの凝集が見えない** — `05_障害復旧` / `06_プラットフォーム自己監視` / `08_グレースフルデグラデーション` / `11_マルチクラスタ戦略` / `12_SLOとエラーバジェット` は「可用性と信頼性」という共通ドメインだが、番号が離散しているため関連が見えない。
- **番号追加のたびに順序が壊れる** — 新しいトピックが増えるたびに末尾に `13_`, `14_`, `15_` と積まれ、ドメイン内の位置づけが番号から消える。

ドメイン別サブディレクトリに再編することで、「どのドメインの議論か」が URL パスから即座に分かる。番号が衝突した場合もサブディレクトリ内に閉じるため、新規追加時の影響範囲が小さい。
