# マルチクラスタ戦略

## 目的

k1s0 の単一クラスタ構成から複数クラスタ構成への移行戦略を定義する。採用側のマルチクラスタ移行時 で staging / prod 分離を開始し、採用側の全社ロールアウト で全社ロールアウト (10+ ノード / 200+ Pod) に到達する際のクラスタ間通信・データ複製・フェイルオーバーの設計指針を示す。

---

## 1. なぜマルチクラスタが必要か

### 1.1 単一クラスタの限界

| 観点 | 単一クラスタで許容できる範囲 | 限界に到達する時点 |
|---|---|---|
| 環境分離 | namespace 分離 (infra / tier1 / tier2 / tier3 / operation) | staging と prod を同一クラスタに混在させると CRD / ClusterPolicy の影響範囲が制御不能 |
| 障害隔離 | PodDisruptionBudget / Deployment 分離 | Control Plane 障害は全 namespace に影響。prod を巻き添えにしない保証がない |
| スケール | 10+ ノード / 200+ Pod | etcd の watch 圧力・API Server のレート制限が顕在化 |
| セキュリティ | RBAC + NetworkPolicy | ClusterRole の誤設定が全 namespace に波及する blast radius |

### 1.2 マルチクラスタ化の判断基準

以下のいずれかが成立した時点で、マルチクラスタ移行を開始する。

| 条件 | 計測方法 |
|---|---|
| staging 環境のデプロイが prod の可用性に影響を与えた (インシデント発生) | Postmortem 記録 |
| Pod 数が 100 を超え、etcd の応答レイテンシが p99 で 200ms を超過 | Prometheus メトリクス `etcd_request_duration_seconds` |
| tier2 チームが 3 チーム以上に増加し、namespace 分離だけでは権限管理が煩雑 | チーム構成のレビュー |
| 採用側のマルチクラスタ移行時 のスコープ合意時点 (計画ベースでの移行開始) | 段階移行判断 |

## 2. クラスタ構成パターン

### 2.1 採用側のマルチクラスタ移行時: staging / prod 2 クラスタ分離

| クラスタ | 役割 | ノード数 | namespace 構成 |
|---|---|---|---|
| **prod** | 本番ワークロード | 3-5 | infra / tier1 / tier2 / tier3 / operation |
| **staging** | 検証・受入テスト | 2-3 | infra / tier1 / tier2 / tier3 / operation (prod と同一構成) |

staging クラスタは prod と同一のマニフェスト構成を Kustomize overlay で管理する。Argo CD の ApplicationSet で両クラスタへの同期を宣言的に制御する。

### 2.2 採用側の全社ロールアウト: 全社ロールアウト構成

| クラスタ | 役割 | ノード数 |
|---|---|---|
| **prod** | 本番ワークロード | 5-10 |
| **staging** | 検証・受入テスト | 2-3 |
| **管理クラスタ** (検討) | Argo CD / Backstage / 監視基盤の集中管理 | 2-3 |

管理クラスタの分離は 採用側の全社ロールアウト の規模に応じて判断する。採用側のマルチクラスタ移行時 時点では prod クラスタの operation namespace に Argo CD / Backstage を配置する現行方針を維持する。

## 3. クラスタ間通信

### 3.1 通信パターンの分類

| パターン | 用途 | 頻度 |
|---|---|---|
| **GitOps 同期** | Argo CD がマニフェストを各クラスタに適用 | マニフェスト変更時 |
| **イメージ配信** | Harbor からコンテナイメージを各クラスタが pull | デプロイ時 |
| **テレメトリ集約** | 各クラスタの OTel Collector → 集中 Grafana スタック | 常時 |
| **サービス間通信** | クラスタ間のサービス呼び出し (採用側の全社ロールアウト 以降で検討) | 低頻度 |

### 3.2 GitOps によるクラスタ管理

Argo CD の ApplicationSet Generator を使用し、複数クラスタへのデプロイを宣言的に管理する。

| 要素 | 方針 |
|---|---|
| Argo CD の配置 | prod クラスタ (採用側のマルチクラスタ移行時) / 管理クラスタ (採用側の全社ロールアウト 検討) |
| クラスタ登録 | `argocd cluster add` で staging クラスタを登録 |
| マニフェスト管理 | Kustomize overlay (`deploy/overlays/staging/` / `deploy/overlays/prod/`) |
| 同期戦略 | staging → prod の順序同期 (staging で検証完了後に prod を同期) |
| 認証 | ServiceAccount トークン (採用側のマルチクラスタ移行時) / OIDC (採用側の全社ロールアウト) |

### 3.3 イメージレジストリ

| 適用段階 | 方式 |
|---|---|
| 採用側のマルチクラスタ移行時 | **Harbor 単一インスタンス** (prod クラスタに配置)。staging は prod の Harbor から pull。ネットワーク帯域が許容する前提 |
| 採用側の全社ロールアウト | **Harbor レプリケーション**。管理クラスタに Harbor プライマリ、各クラスタに Harbor レプリカを配置し、イメージを自動複製。pull 時のクロスクラスタ通信を削減 |

### 3.4 テレメトリ集約

| 適用段階 | 方式 |
|---|---|
| 採用側のマルチクラスタ移行時 | 各クラスタに OTel Collector + Prometheus + Loki を配置。Grafana は prod 側で両クラスタのデータソースを参照 (Prometheus federation / Loki multi-tenancy) |
| 採用側の全社ロールアウト | 集中 Grafana スタック (管理クラスタ) に統合。各クラスタの OTel Collector Gateway が集中 Loki / Tempo に OTLP で転送 |

### 3.5 サービス間クロスクラスタ通信

採用側の全社ロールアウト までは**クラスタ間のサービス直接呼び出しを禁止**する。

理由:

- staging と prod 間の通信は本質的に不要 (環境分離の目的に反する)
- 採用側の全社ロールアウト でマルチリージョン展開が必要になった場合に初めて検討する
- クラスタ間通信の手段として Istio Multi-Primary (east-west gateway) を候補とする

## 4. データ複製戦略

### 4.1 PostgreSQL

| 適用段階 | 方式 |
|---|---|
| 採用側のマルチクラスタ移行時 | **staging 専用 PostgreSQL** (staging クラスタ内に独立インスタンス)。prod データのサブセットをマスキングして staging に投入するスクリプトを用意 |
| 採用側の全社ロールアウト | prod クラスタ内で CloudNativePG のスタンバイクラスタ (Cross-Region Replication) を検討。ただしオンプレ前提のため、物理的なサイト分離がある場合のみ |

staging の PostgreSQL は prod と完全に独立させる。staging に本番データを直接流すことは個人情報保護の観点から禁止する。

### 4.2 Kafka

| 適用段階 | 方式 |
|---|---|
| 採用側のマルチクラスタ移行時 | **staging 専用 Kafka ブローカー** (staging クラスタ内)。prod → staging のイベント複製は行わない |
| 採用側の全社ロールアウト | MirrorMaker 2 による prod → DR クラスタへのトピック複製を検討 (DR 要件が発生した場合のみ) |

### 4.3 Valkey

| 適用段階 | 方式 |
|---|---|
| 採用側のマルチクラスタ移行時 | **staging 専用 Valkey** (staging クラスタ内)。キャッシュデータの複製は不要 |
| 採用側の全社ロールアウト | 同上。Valkey はキャッシュ / 揮発データのため、クラスタ間複製の必要性は低い |

### 4.4 OpenBao

| 適用段階 | 方式 |
|---|---|
| 採用側のマルチクラスタ移行時 | **prod / staging で独立した OpenBao インスタンス**。staging 用の Secret は staging の OpenBao で管理。prod の Secret を staging にコピーすることは禁止 |
| 採用側の全社ロールアウト | 同上。OpenBao Performance Replication (Vault 互換機能) は 採用側の全社ロールアウト で DR 要件が発生した場合に検討 |

## 5. フェイルオーバーと DR

### 5.1 採用側のマルチクラスタ移行時 の DR 方針

採用側のマルチクラスタ移行時 では**クラスタレベルの自動フェイルオーバーは実装しない**。

理由:

- staging / prod の 2 クラスタ構成であり、staging はワークロードが異なるため prod の代替にならない
- 自動フェイルオーバーの実装は複雑性が高く、2-3 名体制で維持するコストに見合わない
- 採用側のマルチクラスタ移行時 の RTO 目標 (1-2 時間) は単一クラスタの再構築 (OpenTofu + GitOps) で達成可能

### 5.2 採用側の全社ロールアウト の DR 検討

採用側の全社ロールアウト で全社ロールアウトに到達し、RTO 30 分以内の要件が発生した場合、以下を検討する。

| DR パターン | 概要 | RTO |
|---|---|---|
| **Active-Passive** | prod クラスタ障害時に DR クラスタに手動切替 | 30 分 - 1 時間 |
| **Active-Active** | 2 クラスタに同一ワークロードを分散配置 | 数分 (DNS / LB 切替) |

Active-Active は PostgreSQL のマルチプライマリ / Kafka の MirrorMaker 2 / Istio Multi-Primary が前提となり、運用複雑性が極めて高い。採用側の全社ロールアウト 時点の組織規模 (5+ 名) と業務要件に基づいて判断する。

### 5.3 段階的な RTO 改善

| 適用段階 | RTO 目標 | 実現手段 |
|---|---|---|
| リリース時点 | 4 時間 | OpenTofu + GitOps によるクラスタ再構築 |
| 採用側のマルチクラスタ移行時 | 1-2 時間 | OpenTofu + GitOps + CloudNativePG バックアップ復元 + 手順書整備 |
| 採用側の全社ロールアウト | 30 分以内 | Active-Passive DR クラスタ (要件発生時のみ) |

## 6. ネットワーク設計

### 6.1 クラスタ間ネットワーク

| 要件 | 方針 |
|---|---|
| 接続方式 | オンプレ LAN 内の VLAN 分離。クラスタごとに独立したサブネット |
| ファイアウォール | クラスタ間の通信は最小限に制限 (Harbor pull / Grafana 参照 / Argo CD 管理のみ) |
| DNS | クラスタごとに独立した CoreDNS。クロスクラスタの DNS 解決は行わない |

### 6.2 CNI の選択と将来性

リリース時点 - 採用後の運用拡大時 では CNI に対する明示的な制約はないが、採用側のマルチクラスタ移行時 のマルチクラスタ移行を見据え、以下を推奨する。

| 候補 | 評価 |
|---|---|
| **Cilium** (推奨) | eBPF ベース。ClusterMesh によるマルチクラスタ接続が 採用側の全社ロールアウト で有力な選択肢になる。L7 NetworkPolicy でtier 間通信をより厳密に制御可能 |
| Calico | 実績豊富だがマルチクラスタ機能は Cilium に劣る |
| Flannel | シンプルだが NetworkPolicy の機能が限定的 |

採用後の運用拡大時 で Cilium を導入し、採用側のマルチクラスタ移行時 でのマルチクラスタ移行を円滑にすることを推奨する。

## 7. Argo CD マルチクラスタ運用

### 7.1 ApplicationSet によるクラスタ管理

```yaml
# 概念例: staging / prod への同時デプロイ管理
apiVersion: argoproj.io/v1alpha1
kind: ApplicationSet
metadata:
  name: tier2-services
  namespace: operation
spec:
  generators:
    - clusters:
        selector:
          matchLabels:
            env: staging
    - clusters:
        selector:
          matchLabels:
            env: prod
  template:
    metadata:
      name: '{{name}}-tier2-services'
    spec:
      project: default
      source:
        repoURL: https://github.com/org/k1s0-services.git
        targetRevision: HEAD
        path: 'deploy/overlays/{{metadata.labels.env}}'
      destination:
        server: '{{server}}'
        namespace: tier2
```

### 7.2 同期順序の制御

| ステップ | 対象 | 条件 |
|---|---|---|
| 1 | staging クラスタ | PR マージで自動同期 |
| 2 | staging での検証 | Litmus による自動テスト + 手動受入 |
| 3 | prod クラスタ | staging 検証完了後に手動承認で同期 |

prod への同期は Argo CD の Sync Window + 手動承認を必須とする。自動同期は staging のみ。

## 8. 段階導入計画

| 段階 | 施策 |
|---|---|
| **採用後の運用拡大時 (準備)** | CNI として Cilium を導入。Kustomize overlay を staging / prod に分離。Argo CD ApplicationSet の設計。マルチクラスタ移行計画の詳細設計開始 |
| **採用側のマルチクラスタ移行時 (実施)** | staging クラスタを構築。Argo CD から staging / prod の 2 クラスタを管理。テレメトリの集約構成。staging 用データ投入スクリプト整備 |
| **採用側の全社ロールアウト (拡張)** | DR 要件に応じて Active-Passive 構成を検討。Harbor レプリケーション。管理クラスタの分離を判断 |

---

## 9. リスクと対処

| リスク | 影響 | 対処 |
|---|---|---|
| マルチクラスタの運用複雑性が 2-3 名体制のキャパシティを超える | 運用品質の低下 | 採用側のマルチクラスタ移行時 のスコープを staging / prod 分離に限定。クラスタ間サービス呼び出しは禁止。自動フェイルオーバーは 採用側の全社ロールアウト に先送り |
| クラスタ間ネットワークの帯域不足 | Harbor からの pull 遅延 / テレメトリ欠損 | 採用側のマルチクラスタ移行時 初期にネットワーク帯域のベンチマークを実施。不足する場合は Harbor レプリケーションを 採用側のマルチクラスタ移行時 に前倒し |
| staging と prod のマニフェスト乖離 | staging で検証した構成が prod で動かない | Kustomize overlay の差分を CI で自動検証。staging 固有の設定は `kustomization.yaml` のパッチに限定 |
| Cilium の学習コスト | 採用後の運用拡大時 での導入が遅延 | 採用初期で Cilium の検証環境を構築。kubespray の CNI プラグインとして設定可能 |
