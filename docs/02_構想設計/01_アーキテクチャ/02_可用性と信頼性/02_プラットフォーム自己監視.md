# プラットフォーム自己監視

## 目的

k1s0 の可観測性基盤 (Prometheus / Loki / Grafana Tempo / Grafana Pyroscope / Grafana) は tier2 / tier3 のサービスを監視するために存在する。しかし **「監視基盤自体が障害を起こした場合に誰が気づくのか」** というブートストラップ問題が存在する。本資料ではこの問題への対処と、プラットフォーム構成コンポーネント自体の監視戦略を定義する。

---

## 1. ブートストラップ問題

### 1.1 問題の構造

| 問題 | 影響 |
|---|---|
| Prometheus が停止 | メトリクス収集が停止し、アラートが発火しなくなる |
| Loki が停止 | ログの集約が停止し、障害調査が不可能になる |
| Grafana が停止 | ダッシュボードが閲覧不能になるが、データ自体は Prometheus / Loki に残る |
| OTel Collector が停止 | 分散トレースが欠損する |
| Alertmanager が停止 | アラートの通知が届かなくなる |

これらは「監視の監視」が必要になる**メタ監視問題**であり、無限再帰に陥らないための設計が必要。

### 1.2 解決方針

**メタ監視は軽量な独立手段で行い、メインの監視スタックに依存しない。**

---

## 2. メタ監視アーキテクチャ

### 2.1 3 層の監視構造

| 層 | 監視対象 | 監視手段 | 依存先 |
|---|---|---|---|
| 第 1 層: 業務監視 | tier2 / tier3 サービス | Prometheus + Loki + Grafana Tempo + Grafana Pyroscope + Grafana | 通常の可観測性スタック (LGTMP) |
| 第 2 層: 基盤監視 | tier1 / infra コンポーネント (Kafka / PostgreSQL / Dapr 等) | 同上 + 専用ダッシュボード | 通常の可観測性スタック |
| 第 3 層: メタ監視 | 可観測性スタック自体 (Prometheus / Loki / Grafana 等) | **独立した軽量監視** | 可観測性スタックに依存しない |

### 2.2 第 3 層 (メタ監視) の実装

| 手段 | 対象 | 仕組み |
|---|---|---|
| k8s Liveness / Readiness Probe | 全 Pod | k8s が自動再起動。kubelet レベルで動作し、Prometheus 非依存 |
| kube-state-metrics | k8s オブジェクト全般 | Deployment / Pod / Node / PVC 等の k8s オブジェクト状態をメトリクスとして Prometheus に公開。Prometheus 単体では取得できない k8s 状態の監視に必須 |
| Node Problem Detector | k8s ノード | カーネル / ハードウェアの異常を検出し、Node Condition に反映 |
| 外部ヘルスチェック (CronJob) | Prometheus / Loki / Grafana の HTTP エンドポイント | 独立した CronJob が各エンドポイントを定期的に叩き、失敗時にメール / Webhook で通知 |
| Watchdog アラート | Prometheus / Alertmanager の死活 | Prometheus の `Watchdog` アラートが定期的に外部に通知。通知が途絶えたら Prometheus が死んでいると判断 (Dead Man's Switch) |
| systemd / OS レベル監視 | kubelet / containerd / etcd | VM 上の systemd が自動再起動。`journalctl` でログ確認 |

## 3. コンポーネント別の監視項目

### 3.1 infra 層

| コンポーネント | 主要監視項目 | アラート条件 |
|---|---|---|
| k8s API Server | リクエストレイテンシ、エラー率 | p99 レイテンシ > 1s、エラー率 > 1% |
| etcd | リーダー選出回数、ディスク fsync 遅延 | fsync > 100ms、リーダー変更 > 3回/h |
| Istio (istiod) | xDS 配信遅延、mTLS 証明書有効期限 | 証明書期限 < 7 日 |
| Kafka (Strimzi) | ブローカー状態、Consumer Lag、ディスク使用率 | Consumer Lag > 10000、ディスク > 80% |
| PostgreSQL (CloudNativePG) | レプリケーション遅延、接続数、デッドロック | レプリカ遅延 > 10s、接続数 > 80% |
| Kyverno | Admission リクエスト成功率、ポリシー違反数、レイテンシ | Admission 失敗率 > 0 (非ポリシー起因)、Pod Ready 数 < 2 |
| cert-manager | 証明書有効期限、発行成功率、Issuer 状態 | 証明書期限 < 14 日 (更新失敗)、Issuer Not Ready |
| kube-state-metrics | Deployment replicas 不一致、Pod CrashLoopBackOff、Node NotReady、PVC Pending | desired != available (5 分以上)、再起動 > 3回/h、Node NotReady > 1 |
| KEDA Operator | ScaledObject のスケーリングエラー数、Metrics API Server の応答遅延 | スケーリングエラー > 0 (5 分以上継続)、Metrics API 応答なし |

### 3.2 tier1 層

| コンポーネント | 主要監視項目 | アラート条件 |
|---|---|---|
| tier1 Go サービス | リクエスト成功率、レイテンシ、goroutine 数 | 成功率 < 99%、p99 > 500ms |
| tier1 Rust サービス | リクエスト成功率、レイテンシ、メモリ使用量 | 成功率 < 99%、p99 > 200ms |
| Grafana Pyroscope | ingestion 成功率、ストレージ使用量、クエリレイテンシ | ingestion エラー率 > 1%、クエリ p99 > 5s |
| Dapr Control Plane | sidecar injection 成功率、Component 状態 | injection 失敗 > 0 |
| Keycloak | 認証成功率、トークン発行レイテンシ | 認証失敗率 > 5%、レイテンシ > 2s |

### 3.3 可観測性スタック自体 (メタ監視)

| コンポーネント | 監視手段 | アラート方法 |
|---|---|---|
| Prometheus | Watchdog アラート (Dead Man's Switch) | 外部 Webhook 受信側で途絶を検知 |
| Alertmanager | Prometheus からの `AlertmanagerDown` アラート + 外部ヘルスチェック | メール / 外部 Webhook |
| Loki | Prometheus の `LokiDown` メトリクス + 外部ヘルスチェック | Prometheus 経由 + 外部 Webhook |
| Grafana | 外部ヘルスチェック (`/api/health`) | 外部 Webhook |
| OTel Collector | Prometheus の `otelcol_exporter_sent_spans` が 0 に落ちたことを検知 | Prometheus 経由 |
| Pyroscope | Prometheus の `pyroscope_ingestion_errors_total` 増加を検知 + 外部ヘルスチェック (`/ready`) | Prometheus 経由 + 外部 Webhook |

## 4. アラート戦略

### 4.1 アラートの分類

| 重要度 | 定義 | 通知先 | 応答時間 |
|---|---|---|---|
| Critical | サービス停止 / データ消失の危険 | 運用チーム (即時通知: メール + Webhook) | 15 分以内に対応開始 |
| Warning | 性能劣化 / リソース逼迫 | 運用チーム (通常通知: メール) | 翌営業日対応 |
| Info | 情報提供 (デプロイ完了等) | Backstage ダッシュボード | 確認のみ |

### 4.2 アラート通知先

| 通知手段 | 用途 | 備考 |
|---|---|---|
| メール | 採用側組織標準の通知手段 | Alertmanager → SMTP |
| Webhook (Teams / Slack) | リアルタイム通知 (導入されている場合) | Alertmanager → Webhook |
| Grafana ダッシュボード | 可視化・傾向分析 | 常時表示用モニター設置を推奨 |

### 4.3 アラート疲れの防止

| 対策 | 内容 |
|---|---|
| アラートの集約 | 同一障害の複数アラートを Alertmanager の `group_by` で集約 |
| 抑制ルール | 上位障害のアラートが発火中は下位を抑制 (`inhibit_rules`) |
| 閾値の定期見直し | 段階ごとにアラート閾値を運用実績に基づいて調整 |
| Runbook 整備 | 各 Critical アラートに対応する復旧手順書を TechDocs に用意 |

## 5. Dead Man's Switch の実装

Prometheus の Watchdog アラートを用いた Dead Man's Switch は、メタ監視の要となる仕組みである。

### 5.1 動作原理

1. Prometheus が `Watchdog` アラートを**常時発火** (正常状態で常に firing)
2. Alertmanager がこのアラートを外部の Webhook 受信サービスに定期送信
3. Webhook 受信側は「一定時間内に通知が来なかった場合」にメール等で異常を通知
4. Prometheus / Alertmanager が停止すると通知が途絶え、受信側が検知する

### 5.2 リリース時点 での簡易実装

リリース時点では専用の Dead Man's Switch サービスは構築せず、以下の簡易手段で代替する。

- 外部ヘルスチェック CronJob が Prometheus の `/api/v1/query` に定期的にリクエスト
- 失敗時に運用チームのメールアドレスに `curl` + SMTP で通知
- 採用後の運用拡大時で本格的な Dead Man's Switch サービスを検討

## 6. OSS リリース時点での採用範囲

| 項目 | リリース時点 で実施 | 採用後の運用拡大時 |
|---|---|---|
| k8s Liveness / Readiness Probe (全 Pod) | 対象 | — |
| kube-state-metrics (k8s オブジェクト状態メトリクス) | 対象 | — |
| Prometheus + Alertmanager 基本アラートルール | 対象 | ルール拡充 |
| Grafana ダッシュボード (infra + tier1) | 対象 | 業務ダッシュボード追加 |
| 外部ヘルスチェック CronJob (Prometheus / Loki / Grafana) | 対象 | — |
| Dead Man's Switch (簡易版) | 対象 | 本格実装 |
| Node Problem Detector | — | 採用後の運用拡大時 |
| Runbook (全 Critical アラート) | 主要アラートのみ | 採用後の運用拡大時 で網羅 |
