# 壊滅的障害シナリオ - 概要

## なぜ独立した章として整備するか

[`05_障害復旧とバックアップ.md`](../../02_可用性と信頼性/01_障害復旧とバックアップ.md) は「コンポーネント別の RPO / RTO とバックアップ戦略」を、[`08_グレースフルデグラデーション.md`](../../02_可用性と信頼性/03_グレースフルデグラデーション.md) は「tier1 公開 API ごとの縮退動作」を扱っている。両者で日常的な単一 Pod 障害から PostgreSQL のフェイルオーバーまではカバーできるが、**「クラスタ規模で基盤 OSS が同時に倒れる」レベルの壊滅的障害**については表内に「クラスタ全壊: 30 分」「etcd クォーラム喪失: 30 分〜1 時間」と RTO 値が並ぶだけで、Runbook 級の手順とフェイルセーフ設計の論証は欠けていた。

稟議の場では「最悪ケースで何時間止まるのか」「その間業務は本当に動くのか」という質問が必ず出る。本章はその問いに **「シナリオ別の Runbook + フェイルセーフ機構の設計根拠 + 検証方法」** で答えるために設けた。読者は本章を読むことで、以下が把握できる状態になる:

- どの基盤 OSS が倒れたとき、どこまで業務が継続するのか (= 既存 Pod は動くのか / 縮退で耐えるのか / 完全停止か)
- 完全停止までに何時間の猶予があるのか (= 既存証明書の有効期限 / キャッシュ TTL / Outbox 滞留容量)
- 復旧に必要な所要時間と人数、判断分岐
- 「机上の RTO」と「実際に達成可能な RTO」のギャップを Litmus でどう検証するか

## 取り扱う 3 シナリオ

3 つの基盤 OSS について、最も外部影響が大きい「完全停止」シナリオを取り扱う。これらを選んだ理由は、**「k1s0 全体の責任範囲のうち、自動復旧が効かず人手の判断と Runbook が必要になる領域」** だからである。CloudNativePG / Valkey / Keycloak の単体障害は Operator のフェイルオーバーや既存ライブラリのキャッシュで縮退できるため、本章の対象外とする (各々は 05/08 章に詳述)。

| シナリオ | RTO 目標 | 業務影響の特徴 | 詳細ドキュメント |
|---|---|---|---|
| **Kafka クラスタ全壊** | 60 分 | 同期処理 (State / Auth) は無影響、非同期処理のみ滞留。Outbox により欠損ゼロ | [`01_Kafka全壊.md`](./01_Kafka全壊.md) |
| **etcd 全ノード障害** | 30 分 (snapshot あり) / 4 時間 (snapshot なし) | 既存 Pod は kubelet で実行継続。新規 Pod 起動・スケジュール変更が不能 | [`02_etcd全ノード障害.md`](./02_etcd全ノード障害.md) |
| **Istio / Envoy Gateway 破損** | 5〜30 分 (レイヤ別) | 既存 Envoy は最後の xDS 設定で動作継続。証明書 TTL (24h) で猶予を確保 | [`03_Istio破損.md`](./03_Istio破損.md) |

各シナリオで RTO の差が大きい点に留意が必要である。Kafka は Operator による再構築が前提のため 60 分、etcd は snapshot の有無で経路が分岐するため二段階、Istio はレイヤごとに独立復旧できるため最短 5 分から始まる。**「全体 RTO」を一律に語ることは設計を誤らせる** ため、本章はシナリオ別に分けている。

## 共通する設計原則

### 原則 1: 「既存リソースが何もしなくても動き続ける時間」を最大化する

壊滅的障害が起きた瞬間に業務が即座に止まるのは設計失敗である。基盤 OSS が倒れても、**何分・何時間は既存リソースだけで動き続けるのか** を意識的に長くする。

- **etcd 全停止時**: kubelet が PLEG (Pod Lifecycle Event Generator) のローカルキャッシュで既存 Pod を継続実行する。新規 Pod 起動と Pod の再スケジュールはできないが、現にトラフィックを処理している Pod は動き続ける
- **Istio Control Plane 停止時**: Envoy データプレーンは最後の xDS 設定 (Listener / Route / Cluster / Endpoint) で動き続ける。新規 mTLS 証明書の発行は止まるが、既存証明書の有効期限内 (デフォルト 24h) は通信が継続する
- **Kafka 全停止時**: 同期処理は Valkey / Keycloak が独立しているため無影響。非同期処理は Outbox Pattern により Producer 側で DB に滞留させ、欠損を出さない

この「猶予時間」を稼ぐことで、人間が落ち着いて Runbook を実行できる。「即停止 → パニック対応」を強制する設計は本章で全て排除する。

### 原則 2: 復旧手段は GitOps を最大化し、手順書のステップ数を削る

復旧手順が長く複雑なほど、本番障害時に手順を踏み外すリスクが上がる。本章の Runbook はすべて以下を前提に設計する。

- k8s マニフェストはすべて Git に格納されている (Argo CD で同期)
- インフラ層 (VM / ネットワーク) は OpenTofu で宣言的に管理
- ステートフルコンポーネント (PostgreSQL / OpenBao) のバックアップは MinIO 上に自動保存
- 最悪ケースでも「Git + IaC + バックアップ」から完全再構築可能

これにより、Runbook の本質は「Git URL を指す → Argo CD を起動 → バックアップを指定」の 3 ステップに集約される。OS インストールやコンポーネントの個別設定をオペレーターに要求しない。

### 原則 3: フェイルセーフは「設計上の期待」、検証は「実装の振る舞い」

各シナリオでフェイルセーフ機構を設計しても、本当に設計通りに動くかは Chaos Engineering で継続的に検証しなければならない。本章で定義する各復旧手順とフェイルセーフ動作は、Phase 2 以降に [Litmus](../../../03_技術選定/03_周辺OSS/02_周辺OSS.md) の ChaosExperiment として実装し、週次の CronChaosEngine で自動検証する。

検証なしに Runbook だけ整備しても、本番初回実行時に「実装が設計と違う」と気付くのが最悪のシナリオである。`08_グレースフルデグラデーション.md` 7 章のテスト方針と整合させ、本章のシナリオも Litmus 対象に組み込む。

## フェイルセーフ階層の俯瞰

3 シナリオに共通するフェイルセーフは「**多層防御 + 猶予時間**」の構造である。各層が独立して動作し、上位層が倒れても下位層が一定時間業務を支える。

| 防御層 | 役割 | 猶予時間 (壊滅的障害発生から完全停止まで) |
|---|---|---|
| L1: Operator 自動復旧 | Strimzi / CloudNativePG / cert-manager による Pod 再起動・フェイルオーバー | 数十秒〜数分 |
| L2: 既存リソースの実行継続 | kubelet PLEG キャッシュ / Envoy 既存 xDS / クライアントライブラリのローカルキャッシュ | 数十分〜数時間 |
| L3: 縮退動作 | Outbox Pattern / fail-fast + caller decides / fail-open キャッシュ | 業務要件次第 (DB 容量次第) |
| L4: GitOps 全再構築 | Git + OpenTofu + Argo CD + barman-cloud | 30 分〜4 時間 |

L1 で復旧できなければ L2 で猶予を稼ぎ、L2 が尽きるまでに L4 で再構築する。L3 はその間業務継続を支える縮退レイヤとして機能する。シナリオ別の章 (01〜03) では、それぞれの基盤 OSS で L1〜L4 がどう組み合わさるかを具体化する。

## Phase 別の整備計画

| フェーズ | 整備内容 |
|---|---|
| **Phase 1 (MVP)** | 本章 Runbook の初版整備。手動での復旧手順検証 (起案者が 1 回実行)。Strimzi / etcdctl / istioctl の基本コマンドを TechDocs に登録 |
| **Phase 2** | Litmus による壊滅的障害シナリオの自動注入開始 (Kafka 全停止 / etcd quorum 喪失を月次)。復旧訓練を四半期に 1 回実施 |
| **Phase 3** | Kafka MirrorMaker 2 による DR クラスタ導入で Kafka RTO を 60 分 → 15 分に短縮。マルチクラスタ etcd フェデレーション検討 (`11_マルチクラスタ戦略.md` と統合) |
