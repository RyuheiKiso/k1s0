# リージョン戦略とデータレジデンシー

## 目的

k1s0 は閉域オンプレ前提で設計されているが、「グローバル展開する顧客が使えるか」「日本の個人情報を海外に出さない保証はあるか」は採用検討およびグループ企業展開の場で必出の論点である。最初からマルチリージョンを設計に組み込むと 採用初期 の 2 週間デモが破綻し、逆に単一リージョン固定では 採用側の全社ロールアウト の全社ロールアウト時に設計を組み直すことになる。本書は **段階別に段階的にリージョン構成を拡張する戦略** と、**データ種別ごとのレジデンシー保証** を定義する。

マルチクラスタの運用設計は [`04_マルチクラスタ戦略.md`](./04_マルチクラスタ戦略.md) に分離しており、本書はリージョン軸 (地理的配置・データ所在地) に焦点を当てる。上位の要件定義は [`../../../03_要件定義/30_非機能要件/`](../../../03_要件定義/30_非機能要件/) で採番する。

---

## 1. なぜ 段階別にリージョン構成を段階拡張するか

単一リージョンで固定するとグローバル展開の障害になり、最初からマルチリージョンで組むと リリース時点 が破綻する。この両極を避けるため、段階の進行に合わせて **1 DC → 同一リージョン内別 AZ → 同一リージョン 2 クラスタ → マルチリージョン** と 1 段ずつ広げる。各段階で RPO / RTO と運用コストのトレードオフが切り替わるため、段階ごとに判断点を設ける。

マルチリージョン展開は Keycloak / PostgreSQL / Kafka / MinIO のすべてがリージョン間同期に対応している必要があり、採用 OSS 選定時点で将来展開の可否を検証済み。採用側の全社ロールアウト で新規 OSS 探索が発生しない設計とする。

---

## 2. 段階別のリージョン構成

各段階 のリージョン構成と DR 目標値 (RPO / RTO) を以下に示す。リリース時点 のように 1 VM 構成では DR 自体が存在しない (バックアップのみ) 点に注意。

- **採用初期**: 単一 VM / 単一クラスタ / 単一リージョン。DR は PostgreSQL の論理バックアップのみ。RPO / RTO は目標値非設定 (デモ・パイロット段階)。
- **採用後の運用拡大時**: 単一クラスタだが MinIO / PostgreSQL のバックアップを **同一リージョン内の別 AZ / 別 DC** へ複製開始。RPO 目標 15 分 / RTO 4 時間。
- **採用側のマルチクラスタ移行時**: **2 クラスタ体制 (staging / prod)** を同一リージョン内で構築。Argo CD が両クラスタを管理し、prod 障害時は staging への昇格手順を Runbook 化する。自動フェイルオーバーは実装しない ([`04_マルチクラスタ戦略.md`](./04_マルチクラスタ戦略.md) 第 5.1 節)。RPO 15 分 / RTO 1〜2 時間。
- **採用側の全社ロールアウト**: **マルチリージョン (東日本 DC + 西日本 DC)** に拡張。Keycloak は master realm レプリケーション、PostgreSQL は Streaming Replication、Kafka は MirrorMaker 2、MinIO は Bucket Replication でリージョン間同期する。RPO 数分 / RTO 30 分 (要件発生時のみ)。

採用後の運用拡大時 → 採用側のマルチクラスタ移行時 の移行で初めて「クラスタ分離」が入り / → 採用側の全社ロールアウト の移行で初めて「地理的分離」が入る。2 つの分離軸を段階で分ける理由は、運用負荷の増分を 1 段ずつ検証するためであり、採用側のマルチクラスタ移行時 でクラスタ分離の運用が安定しない組織が 採用側の全社ロールアウト でリージョン分離を運用できる見込みはない。

## 3. データレジデンシーの保証

採用側組織の顧客・グループ企業の要件として最も頻出するのは「日本の個人情報を国外に出さない」「監査ログは国内で 5 年保管」の 2 つ。k1s0 はデータ種別ごとにレジデンシーを明示し、保管先を技術的に強制する。ポリシーだけでは運用中の越境を防げないため、**Kyverno で国外 MinIO / RDBMS への書込みを admission 時に拒否** する仕組みまで含めて設計する。

データ種別と保管先の対応を以下に示す。

- **個人情報 (氏名 / マイナンバー / 生体情報)**: `k1s0.Pii` API 経由のみアクセス可能。保管先は MinIO `pii-jp/` バケット固定、OpenBao Transit Engine で暗号化。国外レジデンシーの MinIO / RDBMS への書込みを Kyverno で拒否する。
- **監査ログ (`k1s0.Audit`)**: `audit-jp/` バケット固定、WORM (Write-Once Read-Many) モード、保管期間 5 年、ハッシュチェーンで改ざん検知。
- **業務データ (`k1s0.State` / `k1s0.PubSub`)**: テナントのレジデンシー設定 (Keycloak Realm 属性) に従って配置先リージョンを自動選択する。
- **運用ログ (Loki) / メトリクス (Prometheus) / トレース (Tempo)**: PII を含まない前提で自由配置。OTel Collector の `redaction` プロセッサで PII を除去してから転送する。

「自由配置」と書いたテレメトリ系も、PII が混入した時点でルール違反になる。redaction プロセッサで機械的に除去する設計は、人間の運用規律に依存しない境界作りとして重要である。

## 4. GDPR / 地方法令への対応

EU 顧客を想定する場合、GDPR 第 44 条 (第三国への転送) への回答が必要となる。k1s0 は以下 3 点で答弁する。これらは 採用側の全社ロールアウト のマルチリージョン展開時に初めて有効化する機能で、採用側のマルチクラスタ移行時 以前は「そもそも EU にデータを置かない」方針で GDPR 論点を回避する。

- **忘れられる権利 (RTBF)**: `k1s0.Pii.Delete` API が Kafka の tombstone イベントを全 tier に配信し、Postgres / MinIO / Loki から論理削除 + 物理削除 (crypto-shredding: OpenBao Transit Engine の鍵削除) を実施。削除完了は `k1s0.Audit` で証跡化する。
- **データポータビリティ**: `k1s0.Pii.Export` API が JSON-LD 形式で個人データをエクスポートする。GDPR 第 20 条の書式要件を満たす。
- **データ保護影響評価 (DPIA)**: テナント新規作成時に DPIA チェックリストを Backstage の Software Template で自動生成する。法務部門の承認をもって Realm 作成を許可するフローを組み込む。

crypto-shredding は「論理削除しても暗号化鍵を破棄することで復号不能にする」手法で、物理削除が完了するまでの間も「事実上削除済」と主張可能にする。GDPR の削除期限 (受領から 30 日以内) に対して現実的な実装として採用する。

## 5. リスクと対処

採用側の全社ロールアウト のマルチリージョン展開で想定される主なリスクと対処を以下に示す。採用側のマルチクラスタ移行時 以前はリスク自体が顕在化しないため、本章は 採用側の全社ロールアウト 移行判断時に参照する。

| リスク | 影響 | 対処 |
|---|---|---|
| リージョン間レプリケーション遅延 | RPO が目標 (数分) を超過 | Prometheus で `replication_lag_seconds` を常時監視。閾値超過で即アラート |
| Keycloak master realm レプリケーション競合 | 認証情報の整合性崩壊 | master realm はシングルプライマリ運用。セカンダリはリードレプリカ専用 |
| MinIO Bucket Replication 帯域不足 | pii-jp / audit-jp のレプリ遅延 | 帯域ベンチマーク → 専用回線契約。監査ログは WORM モードで書込み後の再転送を禁止 |
| DPIA の法務部門承認が 採用側の全社ロールアウト のスループット律速に | テナント新規作成のリードタイム増大 | DPIA チェックリストの定型化。軽微要件は Backstage の自動判定で承認スキップ |
