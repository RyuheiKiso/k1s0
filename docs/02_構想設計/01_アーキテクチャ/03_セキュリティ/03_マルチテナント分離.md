# マルチテナント分離モデル

## 目的

k1s0 は「単一の情シス部門が複数部門・複数グループ会社の業務を同居運用する」ユースケースを第一級で扱う。テナント境界が曖昧なままでは、監査で「別部門の情報が別テナントに漏れないか」「グループ企業に横展開できるか」と問われた際に即答できず、同居運用そのものが成立しない。本書はテナント分離を 4 層に分解し、どの層でどの OSS が何を保証するか、運用中に境界が破れた場合の検知と封じ込めをどう設計するかを定義する。

上位の要件定義は [`../../../03_要件定義/20_品質特性/05_tenant_isolation/`](../../../03_要件定義/20_品質特性/05_tenant_isolation/) で採番する。本書は構想設計として「どう実装するか」に踏み込むが、具体的なマニフェスト・YAML は Phase 1b の実装フェーズで決定するため本書では扱わない。

---

## 1. なぜ 4 層で分離するのか

単一層の分離 (たとえば namespace だけ) では不十分である。ネットワーク越しにサービスが呼べる状態なら namespace を分けても意味がなく、逆にネットワークを切っても同じ DB スキーマを参照していれば越境は成立する。監査で指摘される典型パターンは「どこか一層で穴が空いていた」ことに起因するため、**名前空間・ネットワーク・認可・データの 4 層すべてで同時に分離を担保する**設計とする。

4 層は独立ではなく直列防御である。どれか 1 層が破れても他の 3 層で封じ込める構造とすることで、1 つの設定ミスが即座にテナント越境に直結しない多層防御を実現する。以下、各層の担当範囲と使用 OSS を示す。

---

## 2. 4 層分離の詳細

### 2.1 名前空間分離 (k8s Namespace + ResourceQuota + LimitRange)

テナントごとに `tenant-{id}` という namespace を割当て、CPU / メモリ / Pod 数の上限を ResourceQuota で強制する。これにより、1 テナントのリソース暴走が他テナントの Pod を退避させる (ノイジーネイバー) 状況を設計段階で排除する。LimitRange は Pod 単体のリソース上限を設定し、個別の Pod 暴走も封じ込める。

namespace 作成は Argo CD + Kyverno ポリシーで強制管理する。`tenant-id` ラベルが付与されていない namespace の作成は admission で拒否する。

### 2.2 ネットワーク分離 (k8s NetworkPolicy + Istio AuthorizationPolicy)

NetworkPolicy で namespace 間の Pod-to-Pod 通信を **既定拒否** とし、tier1 公開 API への到達のみを明示的に許可する。L3/L4 の通信制御はこの層で閉じる。

Istio AuthorizationPolicy は L7 の認可を担い、mTLS の SPIFFE ID (`spiffe://cluster.local/ns/tenant-a/sa/xxx`) を principal として認可判定する。tenant-a の ServiceAccount が tenant-b のサービスに到達することを Istio 側で拒否する。NetworkPolicy が L3/L4、Istio AuthorizationPolicy が L7 と、二重に境界を張る。

### 2.3 認可分離 (Keycloak Realm + RBAC + ZEN Engine)

Keycloak は **テナントごとに Realm を分離** し、ユーザー・ロール・クライアントをテナント境界を越えて共有しない。Realm 分離により、認証情報のスコープが物理的にテナント内に閉じる。

tier1 公開 API は全リクエストで JWT の `tenant_id` クレームを検証する。さらに ZEN Engine の決定表で「この操作を、この tenant_id が、このリソースに対して行えるか」を宣言的に判定する。コードに認可ロジックを埋め込まず、決定表で業務担当が将来編集できる構造を維持する (Phase 4 で JDM Editor 導入)。

### 2.4 データ分離 (PostgreSQL スキーマ + MinIO バケット + Kafka トピック)

PostgreSQL は **スキーマ単位でテナント分離** し、Row-Level Security (RLS) で `tenant_id` 列に一致する行以外を SELECT 不可にする。スキーマ分離だけでは SQL インジェクションで越境され得るため、RLS で行レベルの二重防御を張る。

MinIO は `tenant-{id}/` プレフィックスと IAM ポリシーで分離し、Kafka は `tenant-{id}.*` トピック名前空間と ACL で分離する。監査ログ (`k1s0.Audit`) も同様に `tenant_id` を必須フィールドとし、`tenant_id=null` の書込みは tier1 が 500 を返して拒否する。

## 3. 越境検知と封じ込め

分離設計が運用中に破れる典型パターンは 3 つある。①開発者が namespace を誤って共有する、②JWT 検証のバグで `tenant_id` クレームが欠落する、③DB マイグレーションで RLS が外れる。これらは「設計時に正しくても運用中に破れる」ため、**運用中に機械的に検知する仕組み** を設計段階から組み込む。

- **Kyverno ポリシー**: Pod に `tenant-id` ラベル必須、namespace に ResourceQuota 必須を admission 時に強制する。静的検証では見逃した設定漏れを実行時に止める。
- **Istio AuthorizationPolicy のデフォルト拒否**: 明示的に許可されない tenant 間通信は全て 403 を返す。新規サービス追加時の設定漏れがフェイルセーフで止まる。
- **`k1s0.Audit` の tenant_id 必須検証**: `tenant_id=null` で監査ログを書き込んだ場合、tier1 が 500 を返し CI が落ちる。JWT 検証バグを最も近い境界で検知する。
- **Litmus Chaos 実験での越境シミュレーション**: 月次で「tenant-a の Pod が tenant-b の Secret を盗めるか」を自動検証する。設計が破れていないことを継続的に実証する。

## 4. グループ企業展開時の分離ポリシー

グループ企業・子会社に k1s0 を横展開する際は、テナント分離だけではなくクラスタ分離を選択肢として持つ。同一法人内の部門分離なら同一クラスタで足りるが、法人が異なれば監査基準・データレジデンシー要件が別になるため、境界の強度を 1 段上げる必要がある。

以下の判断基準で分離レベルを決定する。

- **同一クラスタ + テナント分離**: 同一法人内の部門・子会社。監査要件が共通で、運用チームが 1 つで済む場合。本書の 4 層分離で十分。
- **クラスタ分離 (Phase 3 マルチクラスタ前提)**: 法人が異なる / 監査基準が異なる / データレジデンシー要件が異なる場合。クラスタ境界で分離を強化する。詳細は [`../02_可用性と信頼性/04_マルチクラスタ戦略.md`](../02_可用性と信頼性/04_マルチクラスタ戦略.md) および [`../02_可用性と信頼性/07_リージョン戦略とデータレジデンシー.md`](../02_可用性と信頼性/07_リージョン戦略とデータレジデンシー.md) を参照。
