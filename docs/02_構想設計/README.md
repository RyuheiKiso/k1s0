# k1s0 構想設計ドキュメント

k1s0 の **稟議通過後に着手する技術深掘り資料群**。アーキテクチャ、tier1 設計、OSS 選定、CI/CD、法務の各論点を、実装・要件化を支える粒度で詳細化する。

---

## 本ディレクトリの位置づけ

`02_構想設計` は企画 ([`../01_企画/`](../01_企画/)) で合意した方針を、要件定義 ([`../03_要件定義/`](../03_要件定義/)) に落とす前に技術的裏付けを用意する層である。決裁者向けの薄い提案書ではなく、**開発を主導するエンジニアが「この設計で作れる / 作れない」を判断するための一次資料** として扱う。

企画書で「情シス向けマイクロサービス基盤を tier 構造で作る」と宣言した際に後ろから支える層であり、要件定義より前に位置する理由は、要件を採番する前に「どういう構造で実現するか」の合意が必要なためである。

---

## 読み方

章番号は企画 → 実装の流れに沿った論理順。初読時は以下の順で読むと、関係性が掴みやすい。

1. [`01_アーキテクチャ/`](./01_アーキテクチャ/) — 4 レイヤ構成・依存ルール・配置形態・セキュリティモデル・可用性・非機能・IoT エッジ接続・壊滅的障害シナリオ
2. [`02_tier1設計/`](./02_tier1設計/) — tier1 のスコープ・Dapr 隠蔽・Go+Rust ハイブリッド・API 設計原則・API バージョニング・診断と補償・開発者体験
3. [`03_技術選定/`](./03_技術選定/) — 選定方針・中核 OSS・周辺 OSS・ZEN Engine・OSS 長期戦略・言語選定の定量分析
4. [`04_CICDと配信/`](./04_CICDと配信/) — GHA / Argo CD パイプライン・テスト戦略・2 種類のポータル (Backstage / アプリ配信)・ローカル開発環境
5. [`05_法務とコンプライアンス/`](./05_法務とコンプライアンス/) — 採用 OSS 40+ のライセンス義務解釈 (AGPL / MPL 含む)・SBOM 監査フロー・k1s0 自体の知財戦略
