# 内製知財戦略

## 目的

k1s0 プラットフォーム自体（tier1 公開 API / 雛形生成 CLI / Dapr Component バリデータ / Rust 自作領域 / tier1 Go ファサード / 汎用ポータル部品 / 汎用ドキュメント）が生み出す知的財産について、以下を定義する。

- k1s0 に適用するライセンス方針
- 知財の所有権と利用許諾の範囲
- グループ会社間 / 委託業者との取り扱い
- 将来の外部公開（OSS 化 / コンサルティング活用）の余地

本資料が埋める空白は、[`00_OSS法務対応.md`](./00_OSS法務対応.md) が「採用 OSS 側」の法務を扱うのに対し、「k1s0 自作コード側」の知財取り扱いが未定義であった点である。k1s0 では **起案者帰属 + 所属企業への無償・非独占・取消不能の社内利用許諾** を採用し、その前提を契約と運用で閉じる。

---

## 1. k1s0 の知財範囲

k1s0 が生成・利用する知的財産を以下の 6 カテゴリに分類する。カテゴリごとに帰属と公開可否を分ける。

| カテゴリ | 具体例 | 既定の扱い |
|---|---|---|
| (A) 共通基盤コード | tier1 公開 API 仕様 (`k1s0.Log` / `k1s0.State` 等)、Go ファサード、Rust 自作領域 | **起案者帰属**、所属企業へ無償利用許諾 |
| (B) 汎用設定・テンプレート | Kustomize / Helm / OpenTofu / Kyverno の汎用テンプレート | **起案者帰属**、所属企業へ無償利用許諾 |
| (C) 開発者ツール | 雛形生成 CLI (`k1s0-scaffold`)、デバッグツール | **起案者帰属**、所属企業へ無償利用許諾 |
| (D) JTC 固有業務資産 | 申請ワークフロー JDM、権限判定 JDM、業務特化ポータル機能 | **所属企業管理**、外部公開しない |
| (E) ドキュメント・図表 | 汎用設計資料、企画資料、図版 | 汎用部分は起案者帰属、JTC 固有情報を含む版は所属企業の社内利用に限定 |
| (F) 運用ノウハウ・秘密情報 | Runbook、障害履歴、SLO 実績、Secrets、本番 values | **所属企業管理**、外部公開しない |

この分離により、「共通基盤は持ち運べる」「企業固有情報は外に出さない」を両立させる。

## 2. カテゴリ別のライセンス方針

### 2.1 (A) 共通基盤コード — **起案者帰属 + 社内無償利用**

| 観点 | 方針 |
|---|---|
| 初期ライセンス | **Proprietary（起案者帰属）** |
| 所属企業の権利 | 無償・非独占・取消不能の社内利用、改修、バックアップ、委託開発 |
| 将来の外部公開 | Phase 5 以降で Apache 2.0 / MIT / AGPL を選択肢として再評価 |
| グループ会社での利用 | 起案者と所属企業が指定した範囲で無償利用可 |

共通基盤コードを初期段階で OSS 化しない理由:

1. **まず社内価値を証明する**: Phase 0〜4 は社内での運用成立を優先する
2. **品質の担保**: 安定稼働前の公開は避ける
3. **サポート体制**: OSS 化後の Issue / PR 対応体制が未整備

### 2.2 (B) 汎用設定・テンプレート — **起案者帰属、ただし本番設定は除外**

| 観点 | 方針 |
|---|---|
| 初期ライセンス | Proprietary（起案者帰属） |
| 所属企業の権利 | 社内での複製・改変・再利用 |
| サンプル版の外部公開 | Phase 3 以降で MIT / Apache 2.0 を検討 |
| 本番 values / Secret | **外部公開対象外**（所属企業管理） |

汎用テンプレートは持ち運べるが、本番値や秘密情報は常に所属企業側で管理する。

### 2.3 (C) 開発者ツール — **起案者帰属 + 積極的な OSS 化候補**

| 観点 | 方針 |
|---|---|
| 初期ライセンス | Proprietary（起案者帰属） |
| 所属企業の権利 | 社内での実行・改変・配布 |
| 将来の外部公開 | Phase 2 完了後に Apache 2.0 での公開を積極検討 |

雛形生成 CLI やデバッグツールは他社 JTC にも直接転用しやすく、OSS 化候補として優先度が高い。

### 2.4 (D) JTC 固有業務資産 — **所属企業管理**

| 観点 | 方針 |
|---|---|
| ライセンス | 所属企業の社内利用に限定 |
| 外部公開 | 原則なし |
| 分離原則 | 共通基盤コードとリポジトリ / モジュール境界を分ける |

申請ワークフロー / 権限ポリシー / 個別業務ロジックは、企業固有の運用知見であり、共通基盤の知財スキームと分離して扱う。

### 2.5 (E) ドキュメント・図表 — **汎用部分は起案者帰属、固有情報は社内限定**

| 観点 | 方針 |
|---|---|
| 汎用設計資料 | 起案者帰属、所属企業へ社内利用許諾 |
| 固有情報を含む版 | 所属企業の社内利用に限定 |
| 将来の外部公開 | 固有情報を削った抽象版のみ、Phase 5 以降で CC BY-SA 4.0 等を検討 |

### 2.6 (F) 運用ノウハウ・秘密情報 — **所属企業管理**

| 観点 | 方針 |
|---|---|
| ライセンス | 社内限定、外部公開なし |
| 文書化 | Runbook / 監査証跡として社内保管 |

障害対応手順や SLO 達成ノウハウは、運用組織と密接に結びつくため、外部公開対象にしない。

## 3. 知財所有権の帰属

### 3.1 開発体制と所有権の原則

| 開発者種別 | 共通基盤コードの扱い | JTC 固有資産の扱い |
|---|---|---|
| 起案者 | **起案者が所有** | 所属企業管理ルールに従う |
| JTC 社員（起案者以外） | 起案者へ帰属させるか、起案者へ再利用可能な包括ライセンスを付与 | 所属企業管理 |
| 協力会社 SES（派遣） | 起案者へ帰属させるか、起案者へ包括ライセンスを付与 | 所属企業管理 |
| 業務委託（準委任 / 請負） | 契約で権利チェーンを閉じる | 所属企業管理 |
| 複数 JTC グループ会社 | 次節のライセンス許諾モデルで整理 | 次節の個別合意で整理 |

この構造を採る以上、**就業規則の一般論に依存せず、起案者-所属企業間の個別合意で明示する** ことが前提となる。曖昧なまま開発を進めない。

### 3.2 業務委託契約・利用許諾契約に必須の条項

OSS ミックス構成のプラットフォーム開発を外部委託する場合、契約書に以下を明記する。

- **利用許諾の範囲**: 所属企業と指定グループ会社に対する無償・非独占・取消不能の社内利用権
- **改変権と委託権**: 障害対応・保守・委託開発のための改変、複製、外部委託を許容
- **権利チェーン**: 共通基盤への成果物は起案者へ帰属させるか、起案者へ再利用可能な包括ライセンスを付与
- **再ライセンス制限**: 所属企業は第三者への再販・再配布・OSS 公開を単独で行わない
- **OSS 利用の義務**: [`00_OSS法務対応.md`](./00_OSS法務対応.md) の allow list 以外の OSS を無断採用しない
- **秘密保持**: 業務ロジック / 業務情報 / 認証情報の第三者開示禁止
- **業務終了後の返却 / 削除**: コード・ドキュメント・認証情報の返却または削除義務

### 3.3 グループ会社間の取り扱い

| シナリオ | 推奨対応 |
|---|---|
| 親会社で利用 → 子会社で利用 | 起案者が親会社と指定子会社へ利用許諾 |
| 複数グループ会社で共同開発 | 共通基盤は起案者帰属、各社固有資産は各社管理 |
| グループ外の取引先での利用 | 原則不可。必要時は起案者・所属企業・取引先の 3 者契約 |

## 4. AGPL 採用 OSS との構造的分離

[`00_OSS法務対応.md`](./00_OSS法務対応.md) セクション 2.4 で示した通り、AGPL OSS（Grafana / Tempo / Loki / MinIO / Pyroscope / Renovate）と k1s0 自作コードは **プロセス分離** により法的に切り離されている。これは知財戦略として以下の意味を持つ。

1. 起案者帰属の共通基盤コードを将来 OSS 化する際、AGPL の copyleft 条項に引きずられない
2. 委託開発分を起案者へ帰属 / ライセンス集約する際、AGPL OSS の義務が混線しない
3. グループ会社への配布時、AGPL OSS は各社で別途インストール運用する前提となり、共通基盤配布物に AGPL ライセンスが混入しない

この構造を維持することは知財戦略の前提条件であり、[`00_OSS法務対応.md`](./00_OSS法務対応.md) セクション 2.4 の 3 原則（プロセス分離 / ライブラリリンクなし / ABI 分離）を守ることが必須となる。

## 5. 商標とブランディング

### 5.1 "k1s0" の商標化

プラットフォーム名 "k1s0" の商標出願の是非は以下で判断する。

| シナリオ | 推奨対応 |
|---|---|
| 社内利用のみ、外部公開なし | 商標出願不要 |
| 将来 OSS 化を視野に入れる | **Phase 3 完了時点で出願主体を含めて検討** |
| 外部へのコンサルティング活用 | 商標出願を前提に再設計 |

現段階（Phase 0）は商標出願不要。Phase 3 完了時点で、起案者名義 / 所属企業名義 / 共同運用のいずれにするかを再評価する。

### 5.2 ロゴ・ブランディング資産

ロゴデザインや UI コンポーネントの意匠は、作成時に以下を明記する。

- 作成者の帰属
- 所属企業による社内利用・広報利用の範囲
- 外部公開・派生物作成の許諾範囲

これらを契約書で明記しない場合、ロゴを利用した社外プレゼン・広報資料で紛争が発生しうる。

## 6. 特許とリスク管理

### 6.1 k1s0 自体の特許出願

k1s0 のアーキテクチャ / tier 構造 / Dapr 隠蔽方式等は、既存技術の組み合わせが主であり、現時点で特許出願を積極推進しない。

- **既存技術の組み合わせが主**: 新規性が低い
- **出願コストと回収見込みの乖離**: 内部利用中心では回収困難
- **OSS 化との整合**: 将来公開する場合は特許ポリシーの明示が必要になる

ただし、監査改ざん防止やルール評価で独自性が明確に出た場合は個別検討する。

### 6.2 第三者特許の侵害リスク

k1s0 が採用する OSS（Dapr / Kubernetes / Kafka 等）は、CNCF 等の財団ガバナンス下で運営され、特許リスクは構造的に低い。共通基盤コードについては、Phase 3 完了時点で一度特許サーチを実施し、重大な侵害リスクがないことを確認する。

## 7. 公表・プレゼンテーションの扱い

### 7.1 社内プレゼンテーション

企画書本文 (`企画書.md`) および配下ドキュメントは、所属企業内部の稟議・説明会・オンボーディングで利用可能。

### 7.2 社外プレゼンテーション

技術カンファレンス等での登壇は、以下の原則で判断する。

| 公開対象 | 判断 |
|---|---|
| 技術選定の考え方（Dapr 隠蔽 / tier 構造） | **原則公開可** |
| 具体的な実装詳細（コードレベル） | Phase 2 完了後に条件付き公開可 |
| 業務ロジック（JDM 内容） | **公開不可** |
| TCO 試算の絶対額 | **公開不可** |
| TCO 試算の比率 | 公開可 |
| 採用 OSS リスト | 公開可 |
| 組織体制・人員配置 | 公開不可 |

外部公開は、**起案者が権利者として公開可否を判断し、所属企業が機密・広報・ブランド観点をレビューする** 二段階で統制する。

### 7.3 論文・書籍執筆

プロジェクトメンバーが論文 / 書籍を執筆する場合、以下を事前に確認する。

- 公開対象が上記セクション 7.2 の基準に合致するか
- 所属企業の広報・法務承認が必要か
- 共著者に協力会社 / 業務委託が含まれる場合の権利関係

## 8. 5 年タイムラインでの知財戦略サマリ

| Phase | 知財アクション |
|---|---|
| Phase 0〜1 (2026) | 起案者-所属企業間の利用許諾契約ひな形整備 + 権利チェーンの棚卸し |
| Phase 2 (2027 前半) | 雛形生成 CLI の OSS 化 RFC 起案、委託契約への帰属条項反映 |
| Phase 3 (2027 後半) | 商標方針と特許サーチを再評価、外部発表ルールを整備 |
| Phase 4 (2028) | tier1 公開 API の OSS 化可否を検討 |
| Phase 5 (2029 以降) | 共通基盤コード全体の OSS 化 / 共同運営モデルを再評価 |

各 Phase のアクションは、当該 Phase の完了時点で起案者 + 所属企業 + 法務部門で再評価する。

---

## 9. 本資料の限界

- 本資料は論点整理資料であり、法的助言ではない
- 利用許諾契約・業務委託契約の具体条項は、各社法務部門で個別設計が必要
- 知財戦略は経営戦略・広報戦略とも連動するため、所属企業の方針が優先する
- 特許・商標の実務は専門家（弁理士）への相談が前提
