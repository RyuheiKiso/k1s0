## 1. 選定の前提条件

以下を満たさない技術は選定対象から除外する。

| 条件 | 根拠 |
|---|---|
| オンプレ / クラウド VM で動作する | k1s0 の配置前提 (マネージドサービス非依存) |
| ベンダーロックインを回避できる | 5 つの勝ち筋の 1 つ |
| OSI 承認された OSS ライセンスである | 稟議ハードルをゼロに近づける |
| コミュニティが活発で長期運用に耐える | 導入後の保守性確保 |
| JTC 情報システム部門が扱える習得難易度である | 運用現実性 |

特に OSS ライセンスについては、RSALv2 / SSPL / BSL のような **「ソースは見えるが OSI 非承認」** のライセンスは対象外とする。Redis 7.4+ が採用外となる理由もここにある。

---

## 2. 判断軸

### 2.1 機能軸

- 必要な機能を OSS 単体で満たせるか
- 他の採用済み技術との統合コストが低いか (例: Istio と同じ Envoy ベースか)

### 2.2 運用軸

- JTC 情シス部門が単体で運用できるか
- 日本語情報の多寡 (相対的に影響は小さいが加点要素)
- 障害時の一次情報 (公式ドキュメント) が整備されているか

### 2.3 長期軸

- 採用組織 / 財団 (CNCF / LF / Apache Foundation 等) の存続性
- 破壊的変更の頻度
- 差し替えが必要になった時の移行コスト (ロックインの逆指標)

### 2.4 人材軸

- JTC で現地採用できるか
- 学習難易度と教材の整備度
- チーム内の既存スキルとの整合

---

## 3. 採用しないもの (除外カテゴリ)

### 3.1 クラウドマネージドサービス全般

EKS / AKS / GKE / Cloud Run / AWS Lambda / RDS 等のクラウドマネージドサービスは、「オンプレ / VM 動作」「ベンダーロックイン回避」の前提に反するため一括で対象外とする。

### 3.2 商用プラットフォーム

OpenShift / Tanzu / Rancher Prime / Gloo Platform / Mia-Platform / Humanitec 等の商用プラットフォームは、稟議ハードルとベンダーロックインの 2 点で k1s0 の原則に反する。OSS 版が利用可能な場合 (例: Rancher OSS) に限り候補とする。

### 3.3 非 OSI 承認ライセンス

RSALv2 / SSPL / BSL / Commons Clause 等の「ソース公開だが OSI 非承認」ライセンスは対象外。具体例:

| 対象 | 理由 |
|---|---|
| Redis 7.4+ | RSALv2 / SSPL デュアルライセンス |
| DragonflyDB | BSL |
| MongoDB | SSPL |

これらは Valkey のような fork 版に切り替えることで代替する。採用 OSS 全体の法務解釈 (AGPL-3.0 / MPL 2.0 を含む) と監査フローは [`../../05_法務とコンプライアンス/00_OSS法務対応.md`](../../05_法務とコンプライアンス/00_OSS法務対応.md) を参照。

---

## 4. 選定の確定プロセス

1. 候補を列挙し、本資料の前提条件と判断軸で評価
2. 次点・却下候補についても「どの軸で劣ったか」を必ず記録
3. 確定した選定は [`04_選定一覧.md`](04_選定一覧.md) に集約
4. 企画書で要点だけを語り、深掘りしたい読者を本ディレクトリに誘導

## 5. OSS リスク管理方針

### 5.1 リスク認識

OSS は以下のリスクを常に内包する。

| リスク | 事例 |
|---|---|
| ライセンス変更による有償化 | Redis (BSD → RSALv2 / SSPL)、Terraform (MPL → BSL)、Elasticsearch (Apache 2.0 → SSPL / Elastic License) |
| メンテナンス停止 | コア開発者の離脱・資金枯渇によるプロジェクト放棄 |
| 破壊的変更の強制 | メジャーバージョンアップで API 非互換が発生 |

これらはいずれも k1s0 のコントロール外で発生するため、「発生しない」前提で設計してはならない。

### 5.2 tier1 限定の原則

OSS ライブラリ・ランタイムの直接利用は **tier1 のみ** に許可する。tier2 / tier3 は tier1 公開 API を経由してのみ OSS の機能を利用する。

| 層 | OSS の直接利用 | 理由 |
|---|---|---|
| tier1 | **許可** | OSS 変更の影響を閉じ込める唯一の層 |
| tier2 | **禁止** | tier1 公開 API を経由する |
| tier3 | **禁止** | tier1 公開 API を経由する (tier2 経由が原則) |

この原則は Dapr 隠蔽方針 ([`../../02_tier1設計/01_設計の核/01_Dapr隠蔽方針.md`](../../02_tier1設計/01_設計の核/01_Dapr隠蔽方針.md)) を **OSS 全般に拡張** したものである。

### 5.3 ラッピングによるダメージ最小化

tier1 が OSS を利用する際は、必ず k1s0 固有の呼び出し用ラッパーを設けること。

- **直接呼び出しの禁止**: tier1 内部であっても、OSS の API をそのまま外部に露出してはならない
- **ラッパーの役割**: OSS の API を k1s0 の語彙で再定義し、内部実装の詳細を隠蔽する
- **差し替え容易性**: OSS が有償化・メンテ停止した場合、ラッパーの内部実装を切り替えるだけで tier2 / tier3 への影響をゼロにする

この方針は既に以下で実践されている。

| OSS | ラッパー | 差し替え時の影響範囲 |
|---|---|---|
| Dapr | `k1s0.*` 公開 API (Go ファサード) | tier1 内部のみ |
| ZEN Engine | `k1s0.Decision.*` | tier1 内部のみ |
| Temporal | `k1s0.Workflow.*` | tier1 内部のみ |
| flagd | `k1s0.Feature.*` | tier1 内部のみ |

### 5.4 新規 OSS 採用時のチェック項目

上記方針に加え、新たに OSS を採用する際は以下を確認すること。

| チェック項目 | 目的 |
|---|---|
| tier1 内に閉じ込められるか | tier2 / tier3 への波及を構造的に防ぐ |
| k1s0 公開 API でラップ可能か | OSS 固有の API が外部に漏れないことを確認する |
| 代替候補が存在するか | 有償化・メンテ停止時の差し替え先を事前に把握する |
| ライセンス変更の前例があるか | 同種のリスクが既に顕在化していないかを確認する |
