# 採用段階別 OSS 採用タイムライン

## 目的

[`04_選定一覧.md`](04_選定一覧.md) が「何を採用するか」を責務別にまとめた静的インデックスであるのに対し、本資料は「いつ導入するか」を時系列で示す。リリース時点 (Apache 2.0 公開時点) から 採用側のマルチクラスタ移行時 (全社展開) まで、各 OSS が採用側組織のどの段階で初期導入され、どの段階で本番運用水準に達するかを一枚で俯瞰できるようにする。これにより「採用初期 のデモ段階 (VM 1 台) で本当に動かせるのか」「採用初期 のパイロット段階に進むには何が増えるのか」を採用側組織内の起案の場で即答できる。

個々の OSS の選定根拠は [`04_選定一覧.md`](04_選定一覧.md) からリンクされる詳細ファイル (`01_実行基盤中核OSS.md` / `02_周辺OSS.md` / `03_ルールエンジン.md` / `07_ストレージと運用補助.md` 以降) を参照すること。本資料はタイムラインの一元インデックスに徹する。

---

## 1. 採用段階区分の前提

k1s0 はリリース時点で全機能を一気通貫で実装済みの OSS であり、k1s0 自体に開発段階の分割はない。本資料の「採用段階」は **採用側組織が k1s0 を取り込んでいく規模拡大の段階** を指す。各段階で「初期導入」「本番水準」「保留」のいずれかに分類する。

- **初期導入 (Intro)**: 稼働するが冗長化や運用手順は最低限。検証利用が主。
- **本番水準 (Prod)**: SLO 準拠・Runbook 整備済み・バックアップ検証済み。採用側組織の本番業務で依存して良い状態。
- **保留 (Hold)**: 採用確定だが導入しない。後続段階で順次解除。

採用初期 の最小構成 (VM 1 台のデモ段階) では「本番水準」に到達する OSS は存在しない。採用後の運用蓄積を経て順次昇格させる。

採用初期 はさらに以下の 3 サブ段階に分かれる。本資料の §2 / §3 / §4 ではこれらを列／行／ゲートとして区別する。

- **採用初期 (デモ / VM1)**: 起案者単独・VM 1 台で k1s0 を立ち上げ、決裁者向けデモを完走させる段階
- **採用初期 (パイロット / VM3)**: VM 3 台 + 協力者 1 名で k1s0 のコア機能を本稼働させ、新規業務アプリ 1 本を載せる段階
- **採用初期 (運用体制確立)**: パイロット稼働を継続しつつ Runbook / SLO / バックアップ検証を整備し、採用継続判断を下す段階

## 3. 採用段階別の累計 OSS 数

各段階に到達した時点で、「本番水準」に達している OSS の累計数。採用側組織の起案の場で「この段階で何が動いているか」の規模感を即答するための指標。

| 採用段階 | 本番水準 OSS 数 | Intro 段階 OSS 数 | 主な新規追加 |
|---|---|---|---|
| 採用初期 (デモ / VM1) | 0 | 11 | k8s / Dapr / Go / Prometheus / Grafana / OTel / Keycloak / CNPG / OpenTofu / Tilt / GHA runner |
| 採用初期 (パイロット / VM3) | 0 | 26 | Istio / Kafka / Rust / Loki / Tempo / Argo CD / Harbor / Valkey / ZEN Engine / Kyverno / flagd / Longhorn / MetalLB / kube-vip / Cosign/Syft/Grype |
| 採用初期 (運用体制確立) | 36 | 6 | Backstage / MinIO / Litmus / Renovate / Argo Events / pgvector / pg_partman / OpenBao / Argo Rollouts / Temporal / KEDA / ESO / Kubeshark (on-demand) |
| 採用後の運用拡大時 | 42 | 2 | Pyroscope / Tekton |
| 採用側のマルチクラスタ移行時 | 44 | 0 | 残存する Intro の本番昇格のみ |

採用初期 (運用体制確立) での本番 OSS 数 36 は「採用側組織のパイロット業務 1 つを運用できる最低水準」であり、採用後の運用拡大時 への判定の前提となる数値。それ以前の段階で 0 件なのは「動くが運用条件を満たさない」状態であることを明示する。

## 4. 採用段階間で必ず検証すること

各段階に進む前のゲート条件として、以下の検証を実施する。[`../../../01_企画/企画書.md`](../../../01_企画/企画書.md) の KPI と突き合わせる。

### 4.1 採用初期 (デモ) → 採用初期 (パイロット) ゲート

- 採用初期 (デモ) (起案者単独、VM 1 台) が決裁者に対して完走すること
- VM 3 台 + 協力者 1 名のアサイン合意が取れていること
- 採用初期 (パイロット) で新規導入する 15 OSS の Helm Chart / Operator 版数を確定していること (Renovate 未導入のため、手動版数ロック)

### 4.2 採用初期 (パイロット) → 採用初期 (運用体制確立) ゲート

- バス係数 2 の実証 (協力者が単独で環境再構築できること)
- tier1 公開 API の 5 主要 API (`Log` / `Telemetry` / `State` / `PubSub` / `Service.Invoke`) が 採用初期 (パイロット) で動作すること
- パイロット業務 1 本が配信ポータル経由で稼働していること
- SBOM 生成率 50% 以上

### 4.3 採用初期 (運用体制確立) → 採用後の運用拡大時 ゲート

- 本番水準 OSS が 36 件揃っていること (本資料第 3 節)
- Runbook 充足率 15/15 (tier1 主要障害シナリオ全件)
- フルリストア訓練が協力者単独で完了していること (起案者不在)
- ライセンス自動監査 CI が有効化されていること ([`../../05_法務とコンプライアンス/00_OSS法務対応.md`](../../05_法務とコンプライアンス/00_OSS法務対応.md) 第 4 節)

### 4.4 採用後の運用拡大時 → 採用側のマルチクラスタ移行時 ゲート

- 全段階共通 KPI がすべて「必達」水準 ([`../../../01_企画/企画書.md`](../../../01_企画/企画書.md) 第 6 節)
- システム基盤チーム以外のチームが独立して新規サービスをリリースできること (tier1 依存がブラックボックス化できる水準)

## 5. 本資料の限界

- OSS 版数は本資料で管理しない。[`17_OSS長期戦略.md`](17_OSS長期戦略.md) と Renovate / Dependabot の自動更新結果を信頼する
- 採用判断が変わった場合 (例: Kafka → NATS の切り替え検討) は [`17_OSS長期戦略.md`](17_OSS長期戦略.md) の差し替えシナリオに沿って追加の RFC が必要
- 段階境界は目安であり、KPI 未達時は前段階に留まることもあり得る
