# ルールエンジン (BRE) — ZEN Engine

## 目的

k1s0 におけるビジネスルール / 決定ロジックの宣言的記述の手段として、GoRules 社が公開する OSS ルールエンジン **ZEN Engine** を採用する。本資料はその採用根拠・役割定義・統合方針・段階導入計画を整理する。

## 採用決定の位置付け

- 採用対象: **ZEN Engine** (`gorules/zen`, MIT License)
- 採用区分: k1s0 の正式採用技術 (候補・要検討ではなく確定)
- 判断時期: 2026-04-12

---

## 1. 解決したい課題

採用側組織の情シス部門で k1s0 を運用する際、以下のような「条件分岐の塊」が必ず発生する。これらを各サービスにハードコードすると、ルール変更のたびにコード改修・PR・ビルド・デプロイが必要となり、稟議業務（業務システムの稟議）と相まって変更リードタイムが極端に長くなる。

| 領域 | 典型的な決定ロジック | 変更頻度 |
|---|---|---|
| 権限ポリシー | アプリ X は部署 Y で役職 Z 以上が利用可 | 月 |
| 申請ワークフロー | 金額 > 100 万かつ部門外 → 部長承認、それ未満 → 課長承認 | 半期 |
| 配信ポータル可視化 | アプリ表示順位を所属・利用履歴・お気に入りで決定 | 月 |
| 監査アラート | 同一ユーザーが 5 分内に 10 件以上の機密参照 → 警告 | 不定期 |
| 業務ルール (将来の tier2 サンプル) | 経費精算の自動承認しきい値、勤怠の例外判定 | 業務都合 |

これらは「コードに書くべきではない」が「ドキュメントに書いて運用するには曖昧すぎる」中間領域にあり、決定表 (Decision Table) で表現するのが最も自然である。

## 2. ZEN Engine とは

| 項目 | 内容 |
|---|---|
| 種別 | Open Source Business Rules Engine (BRE) |
| 提供元 | GoRules (商用 BRMS の評価エンジン部分を OSS 公開) |
| ライセンス | MIT |
| 実装言語 | Rust |
| バインディング | NodeJS / Python / Go (公式) |
| モデル形式 | JDM (JSON Decision Model) — DMN の思想を JSON ベースで簡素化 |
| 主要ノード種別 | Input / Output / Decision Table / Switch / Function (QuickJS) / Expression / Decision (ネスト) |
| 評価エンジン | Rust 製の単一バイナリ、サンドボックス化された QuickJS による Function ノード実行 |
| 決定表のヒットポリシー | `first` / `collect` |

JDM は DMN を実装しやすい範囲に絞り、JSON で完結する単純化された表現を提供する。Camunda DMN や Drools のような完全な DMN/DRL 実装より軽量で、エディタも GoRules が公式提供している。

## 3. 採用理由

### 3.1 tier1 の Rust 採用根拠を補強する

tier1 内部実装は **Go (Daprファサード) + Rust (自作領域)** のハイブリッド構成を採る ([`../../02_tier1設計/01_設計の核/02_内部言語ハイブリッド.md`](../../02_tier1設計/01_設計の核/02_内部言語ハイブリッド.md))。ZEN Engine は **Rust 自作領域の中核ユースケース** として位置付けられる。

ZEN Engine が Rust 側に置かれる理由:

- 単一決定の評価レイテンシが極めて低い (Rust + 事前パース済み JDM)
- メモリ安全性により長時間稼働で退行しない
- ZEN Engine が Rust ネイティブのため tier1 Rust サービスに **in-process 埋め込み** できる (Go 経由だと FFI / IPC が必要で性能・安全性ともに劣化)
- Dapr に依存しない自作領域なので Daprファサード (Go) との役割分担が綺麗

→ tier1 内部での Rust 採用根拠を「思想」から「ユースケース」に変える代表例になる。

### 3.2 「採用側組織の情シスでも運用できる」の補強

決定表は **エンジニア以外が編集可能** な数少ない技術資産である。GoRules が提供する JDM エディタ (Web ベース) を Backstage プラグインまたは独立アプリとして提供すれば以下が成立する。

- 情シス担当が採用検討ルールを画面操作で更新できる
- 業務担当が経費しきい値などを自分で調整できる
- エンジニアの介在なしにルール変更が可能になる

これは企画書の「5 つの勝ち筋」の「採用側組織の情シスが運用できる」に最も寄与するピースとなる。

### 3.3 Workflow 基盤との明確な役割分担

| 領域 | 担当 | 例 |
|---|---|---|
| オーケストレーション (状態遷移・タイマー・永続化・補償) | Temporal / Dapr Workflow | 申請 → 上長承認 → 経理確認 → 完了の長時間ワークフロー、督促メールの遅延実行 |
| 単発の決定評価 (条件分岐・スコアリング・分類) | ZEN Engine | 「この申請に必要な承認者は誰か」を 1 回の評価で返す |

両者は競合せず、Workflow の各ステップから ZEN Engine を呼び出す構成が自然。Workflow 基盤が苦手とする「複雑な if/else を YAML/コードで書きたくない」要求を ZEN Engine が引き取る。

### 3.4 ライセンスとガバナンスの整合

- MIT で完全無償。商用版との機能差なし (GoRules の商用製品 BRMS は管理 UI とコラボレーション機能が中心で、評価エンジン自体は OSS 版と同一)
- ベンダーロックイン懸念を抑えるため、**JDM ファイル (JSON) を一次資産** として Git 管理する。エンジン差し替え時は JDM → 別形式への変換ツールを書けばよい

### 3.5 OPA との棲み分け

OPA (Open Policy Agent) は CNCF Graduated でセキュリティ / 認可ポリシー領域の標準。ZEN Engine と機能領域が一見重なるが、役割は明確に分離する。

| 観点 | OPA | ZEN Engine |
|---|---|---|
| 主用途 | 認可 / アクセス制御 (k8s admission / API authz) | 業務ルール / 決定評価 |
| 言語 | Rego (宣言型) | JDM (決定表 + 式) |
| 編集者 | エンジニア / セキュリティ担当 | 業務担当 / 情シス担当も可 |
| エコシステム整合 | k8s / Istio / Envoy 直結 | tier1 内部の業務ロジック呼び出し |
| k1s0 での想定統合先 | Envoy Gateway / Istio AuthorizationPolicy / Kyverno 補完 | tier1 公開 API `k1s0.Decision.Evaluate()` |

→ **両方採用する**。OPA はインフラ寄りの認可、ZEN Engine は業務寄りの決定。重複は最小化する。

## 4. アーキテクチャ上の位置付け

### 配置レイヤ

- ZEN Engine は tier1 の内部実装 (Rust 自作領域) として配置する
- tier2 / tier3 から見えるのは tier1 公開 API `k1s0.Decision.*` のみ
- Daprファサード (Go サービス) からは独立した Rust サービスとして動作し、tier1 内部は Go ↔ Rust 間を Protobuf gRPC で通信する

### tier1 公開 API の形

tier2 / tier3 開発者が触れるのは薄いラッパーのみ。ZEN Engine / JDM の語彙は表に出さない。

- `decisionId` は JDM ファイルの論理名に対応。物理パス / バージョンは tier1 が解決する
- 入出力は型なし dictionary。型安全ラッパーは 採用側のマルチクラスタ移行時で雛形生成 CLI が提供
- バージョン固定 / カナリア評価 / シャドー評価などの高度機能はすべて tier1 側で吸収

### JDM 資産の管理

| 項目 | 方針 |
|---|---|
| 保管場所 | `src/tier1/decisions/` 配下に JDM ファイル群 (`*.json`) |
| バージョン管理 | Git。PR レビューで業務担当 + 情シス担当が承認 |
| デプロイ | tier1 サービスが起動時に Git から fetch、Valkey にキャッシュ |
| 編集 UI | 採用側のマルチクラスタ移行時 で GoRules JDM Editor を Backstage プラグインとして導入 |
| テスト | JDM 1 つにつき YAML テストケース必須 (CI で実行) |
| ロールバック | Git revert + tier1 サービスへの reload シグナル |

## 5. 段階導入計画

> **リリース時点では組み込まない**。tier1 公開 API のスタブだけ用意し、実装は段階的に進める。ただし **リリース時点 (採用初期) 中に JDM の表現力を検証する PoC を必ず実施する**。

| 段階 | 範囲 | 成果物 |
|---|---|---|
| リリース時点 | tier1 公開 API のインタフェース定義のみ。実装は未稼働 (常に「承認」を返すスタブ) | `k1s0.Decision.EvaluateAsync()` の API 契約 + 1 件の JDM サンプル |
| **リリース時点 (採用初期): JDM 表現力 PoC** | **3 件の代表ユースケースを JDM で表現し、表現力の限界を早期に検証** | **PoC 結果レポート + Go/NoGo 判断** |
| 採用後の運用拡大時 | tier1 Rust サービスに ZEN Engine を埋め込む。ファイル実体ベースで稼働開始 | tier1 内部に決定エンジンサービス、`approval.expense` 等の初期 JDM 3〜5 件 |
| 採用側のマルチクラスタ移行時 | アプリ配信ポータルの権限ポリシーと申請ワークフローで本格利用。Backstage に JDM Editor プラグインを追加 | エンドユーザー画面の権限判定が JDM で表現される |
| 採用側の全社展開期 | 業務担当が JDM Editor で直接編集。PR ベースのレビューフローを確立 | 情シス担当が業務ルール（業務システムの稟議ロジック等）を画面操作で更新できる |
| 採用側の全社ロールアウト | tier2 業務サービスで本格採用 (経費・勤怠等の自動判定) | 業務ロジックの大半が宣言的に外出しされる |

### リリース時点 JDM 表現力 PoC

ルールエンジンは tier1 の差別化要素であり、JDM の表現力が不十分な場合は 採用後の運用拡大時の設計全体に影響する。採用後の運用拡大時 で「JDM では書けない」と判明してから方針転換するのは手戻りが大きいため、リリース時点 中に早期検証を行う。

#### PoC の 3 ユースケース

| # | ユースケース | 検証ポイント |
|---|---|---|
| 1 | **経費精算の自動承認** — 金額 / 部門 / 役職のマトリクスで承認者・承認ルートを決定 | Decision Table の行数・条件組み合わせ爆発に耐えるか。`collect` ヒットポリシーで複数マッチが自然に表現できるか |
| 2 | **アプリ配信ポータルの権限ポリシー** — 所属部署 / 役職 / 利用申請状態からアプリの可視性・インストール可否を決定 | ネストした Decision ノード (サブ決定表の呼び出し) が必要になるか。Expression ノードで配列操作が必要になるか |
| 3 | **監査アラート判定** — 同一ユーザーの機密データアクセス頻度 / 時間帯 / 操作種別からアラートレベルを決定 | 時間ウィンドウの概念を JDM 単体で表現できるか (できない場合は tier1 Rust 側で前処理が必要) |

#### PoC の実施方法

| 項目 | 内容 |
|---|---|
| 実施時期 | リリース時点 の後半 2 週間 |
| 実施者 | 起案者 (メイン) + 協力者 (レビュー) |
| 環境 | ローカル環境で ZEN Engine の Rust / CLI 版を直接実行。k8s 上へのデプロイは不要 |
| 成果物 | JDM ファイル 3 件 + テストケース + PoC 結果レポート (Go/NoGo 判断) |
| 工数見積 | 3-5 人日 (既存業務のヒアリング含む) |

#### Go/NoGo 判断基準

| 判断 | 条件 |
|---|---|
| **Go** | 3 ユースケース全てが Decision Table + Expression の組み合わせで表現可能。Function ノード (JavaScript) への依存が不要 |
| **条件付き Go** | 1-2 件で Function ノードまたは tier1 Rust 側の前処理が必要だが、アーキテクチャの範囲内で対処可能 |
| **NoGo** | 3 ユースケースの過半数が JDM では自然に表現できない。代替策の検討を 採用初期で実施 |

#### NoGo 時の代替策

| 代替策 | 概要 | 判断基準 |
|---|---|---|
| **ZEN Engine + tier1 Rust 前処理** | JDM が苦手な部分 (時間ウィンドウ計算等) を tier1 Rust で前処理し、結果を JDM に渡す | 前処理のパターンが 2-3 種類以下に収まる場合 |
| **ZEN Engine + OPA 併用** | 認可寄りの複雑なポリシーは OPA (Rego) に寄せ、ZEN Engine は業務ルールに限定 | 認可と業務ルールの境界が明確な場合 |
| **ZEN Engine を段階縮小** | ZEN Engine の対象を「表形式で自然に書けるルール」に限定し、それ以外は tier1 Rust コードで実装 | JDM の表現力が想定より狭い場合 |

### リリース時点 に「API スタブだけ」入れる理由

- tier1 公開 API の形を リリース時点で決め切ることで、採用後の運用拡大時の実装差し込みがインタフェース変更なしで進む
- tier2 / tier3 サンプルが 採用後の運用拡大時 で生えてきた時に「Decision API がない」と慌てない
- 実装コストはほぼゼロ (ハードコード返却のスタブ)

## 6. リスクと対処

| リスク | 影響 | 対処 |
|---|---|---|
| GoRules 社の方向転換 / OSS 維持の停滞 | エンジン更新が止まる | JDM ファイル (JSON) を一次資産として Git 管理。評価エンジン差し替え時の移行可能性を確保 |
| コミュニティが商用 BRE より小さい | 日本語情報不足 | 公式ドキュメント (英語) を一次情報とし tier1 チームで運用ノウハウを内製 |
| Function ノードでの JavaScript 実行 | サンドボックス逸脱・性能低下 | tier1 ガバナンスとして Function ノード使用を原則禁止、決定表と Expression に限定 |
| JDM の表現力不足 | 結局コードで書き直す | 採用後の運用拡大時 開始時に「3 件の代表ユースケースで JDM が成立するか」の PoC を必ず行う |
| 業務担当が JDM Editor を使いこなせない | 結局エンジニアが書き続ける | 採用側の全社展開期 導入前に情シス担当向けの編集ガイドを整備。最初は「読めるが書けない」段階で十分価値がある |
| ルール変更の監査トレーサビリティ | 誰が何を変えたか不明 | JDM 変更は必ず Git PR 経由、tier1 監査ログに評価結果を記録 |
| ZEN Engine のバージョン破壊的変更 | tier1 サービスのリビルドが必要 | tier1 が ZEN Engine をバージョン固定。アップデートは計画的タスクとして扱う |

## 7. 採用しない選択肢の検討

ZEN Engine 採用の妥当性を裏付けるため、主要な BRE / 決定エンジン候補 8 種を k1s0 の前提条件 5 つ (オンプレ完結 / 業務担当が編集可能 / Rust 自作領域への統合性 / OSI 承認 OSS / 長期運用に耐えるコミュニティ) で同列評価する。本節は「なぜ ZEN Engine か」を「他を選ばなかった理由」から逆算する目的で構成する。

### 7.1 候補一覧と定量比較

| 候補 | ライセンス | 実装言語 | 常駐メモリ概算 | 単一決定の評価レイテンシ | 業務担当編集適性 | コミュニティ規模 | k1s0 適合度 | 採否 |
|---|---|---|---|---|---|---|---|---|
| **ZEN Engine** | MIT (OSI ◯) | **Rust** | ~30 MB (in-process 埋め込み時) | μs オーダ (事前パース済 JDM) | △ (JDM Editor を Backstage プラグイン化予定) | GitHub 1.7k stars / 2022〜、GoRules 主導 | ◎ Rust 自作領域へ in-process 埋め込み可 | **採用** |
| Drools | Apache 2.0 (OSI ◯) | Java (JVM) | ~250 MB (JVM 起動 + Rete ネットワーク) | ms オーダ (Rete アルゴリズム) | ✕ DRL は Java 風 DSL、業務担当不可 | GitHub 5.8k stars / 2001〜、RedHat 主導 | ✕ JVM 追加で Go/Rust 構成に異種運用要素 | 却下 |
| Kogito (Drools 後継) | Apache 2.0 (OSI ◯) | Java (Quarkus native) | ~80 MB (native image) | μs〜ms オーダ | ✕ DMN/DRL とも開発者向け | GitHub 0.7k stars / 2019〜、RedHat 主導 | △ Quarkus native build で軽量化可だが CI ビルド 5〜10 分 | 却下 |
| Camunda 7 (DMN モジュール単体) | Apache 2.0 (OSI ◯) | Java (JVM) | ~400 MB (Camunda Platform 全体) | ms オーダ | ◎ Camunda Modeler が成熟 | GitHub 3.4k stars / 2013〜、Camunda 主導 | ✕ **2025-04 EOL**、移行先は Camunda 8 | 却下 |
| Camunda 8 (Zeebe + DMN) | **Camunda License v1.0** (Source Available、OSI ✕) | Java | ~1 GB (Zeebe + Operate + Tasklist) | ms オーダ | ◎ Camunda Modeler | 商用主導、Community 版に機能制限 | ✕ **OSI 非承認、選定方針違反** | 却下 |
| OPA (Rego) | Apache 2.0 (OSI ◯) | Go | ~50 MB | μs オーダ | ✕ Rego は Datalog 系宣言型関数言語、学習コスト高 | CNCF **Graduated**、9.5k stars | △ 認可向け。業務ルールには表現力過多 | **別用途で併用**（k8s admission / Istio authz） |
| Easy Rules | MIT (OSI ◯) | Java | ~150 MB (JVM) | μs オーダ | ✕ Java アノテーション必須 | GitHub 4.5k stars、**2022 以降コミット停滞 (月 1 件未満)** | ✕ 実質メンテ終了 | 却下 |
| 各サービスでハードコード | — | — | — | — | ✕ コード改修 + 申請が必要 | — | ✕ 変更リードタイム長期化 | 却下 |
| 自前で決定表評価器を Rust 実装 | — | Rust | 任意 | μs オーダ | ✕ JDM Editor 相当も自作要 | — | ✕ 見積 8〜12 人月、OSS リリース時点での採用範囲外 | 却下 |

### 7.2 観点別の落選理由

各候補が k1s0 の前提条件のどこで脱落したかを散文で展開する。

#### Drools / Kogito (JVM ファミリー)

Drools は BRE 業界のデファクトであり、機能性 (Rete アルゴリズム、CEP、Workflow 統合) とコミュニティ規模 (5.8k stars / 25 年の運用実績) では ZEN Engine を圧倒する。しかし k1s0 tier1 は Go ファサード + Rust 自作領域のハイブリッドで構成されており、JVM プロセスを 1 つ追加すると以下の二次コストが連鎖する。

- JVM ランタイムの CVE 監視 / GC チューニング / ヒープダンプ解析が別スキル要員として必要になる
- メモリフットプリントが ZEN Engine の **約 8 倍** (30 MB → 250 MB)、tier1 ノードの sizing が変わる
- 起動時間 5〜10 秒。Pod 再起動 / スケールアウト時の冷起動コストが増大する
- 採用側組織の情シスは Java 運用ノウハウを持つが、Go/Rust と JVM の 3 言語混在は学習負荷が過大

Kogito は Drools 後継として Quarkus native image でフットプリントを 80 MB まで圧縮する。ただし native ビルドが CI 上で 5〜10 分かかり、CVE Critical 48 時間対応のサイクルに収まらない。

#### Camunda 7 / 8

Camunda 7 (DMN モジュール) は単体運用が技術的に可能で、Modeler の UX も優秀。しかし **2025-04 で EOL** が公式アナウンスされており、移行先の Camunda 8 は **Camunda License v1.0** (Source Available) を採用。OSI 承認外であり k1s0 選定方針「OSI 承認済み OSS のみ採用」に正面から違反する。技術評価以前に検討対象外。

#### OPA (Rego)

OPA は別カテゴリ (k8s admission / Istio authz) で採用済み。理論上は業務ルールにも転用可能だが、Rego は **Datalog 系の宣言型関数言語** で「経費精算しきい値の `>= 100000` を編集する」レベルの作業にも `some x in input.users` のような構文理解を要求する。**業務担当の編集可能性こそが ZEN Engine 採用の核心**であり、この観点で OPA は失格する。役割分担として OPA はインフラ寄りの認可、ZEN Engine は業務寄りの決定、と明確に分離する (3.5 節参照)。

#### Easy Rules

GitHub stars (4.5k) と簡潔さで一見魅力的だが、**2022 年以降のコミット頻度が月 1 件未満** に低下しており実質メンテ終了状態。10 年運用前提の k1s0 では「選定時点で死んでいる OSS」を採用できない。

#### 各サービスでハードコード / 自前実装

ハードコードは「業務ルール変更ごとに PR + ビルド + デプロイ + 業務側承認」が必須となり、採用側組織の業務承認文化と相まって変更リードタイムが**月単位**に肥大化する。自前実装は決定表エンジン本体だけなら 2〜3 人月で書けるが、JDM Editor 相当の Web UI / テスト DSL / バージョン管理 / ホットリロードまで含めると **8〜12 人月**。ZEN Engine が無償で提供する価値の再発明であり、OSS リリース時点での採用範囲縮小方針に反する。

### 7.3 結論

5 つの前提条件を同時に満たす候補は ZEN Engine のみ。Drools の機能性、OPA のエコシステム、Camunda の UI 成熟度はそれぞれ ZEN Engine を局所的に上回るが、いずれも前提条件を 1 つ以上欠落する。**業務担当が編集可能 (◎ または △) かつ Rust 自作領域に in-process 埋め込み可能** という 2 軸の交点に存在するのが ZEN Engine だけである、というのが採用の客観的根拠である。
