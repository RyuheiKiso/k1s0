# イベント駆動オートスケーラー OSS の選定

## 目的

Kafka / Kubernetes メトリクス / cron を駆動源としてワークロードを自動スケールする KEDA を整理する。HPA が扱える CPU/メモリ軸を超えて、イベントキュー長や業務時間帯に応じた縮退を実現する。対象カテゴリは以下のとおり。

- AD. イベント駆動オートスケーラー (KEDA)

---

## AD. イベント駆動オートスケーラー

| 候補 | 採否 | 評価 |
|---|---|---|
| **KEDA** | 採用 | CNCF Graduated。イベント駆動のオートスケーリング。HPA を拡張し Kafka Consumer Lag / Temporal Task Queue / Prometheus メトリクス等をスケールトリガーとして利用可能。Apache 2.0 |
| HPA (CPU / メモリ) のみ | 不十分 | CPU / メモリベースのスケーリングではイベント駆動ワークロード (Kafka Consumer / Temporal Worker) の負荷を正しく検知できない |
| HPA + Prometheus Adapter | 次点 | Prometheus のカスタムメトリクスを HPA に接続できるが、Kafka Consumer Lag や Temporal Task Queue のスケーラーが組み込まれていない。スケーラーを自前で実装する必要がある |
| Knative Serving | 却下 | Apache 2.0。サーバーレスワークロード向けの 0→N スケーリングを提供するが、Knative のインストールが大きく (Istio 依存)、k1s0 の tier 構造に適合しない |

### 背景: なぜ今決定が必要か

採用後の運用拡大時 で Kafka (Strimzi) と Temporal を導入する。これにより以下のイベント駆動ワークロードが k1s0 に追加される。

| ワークロード | 導入時期 | スケーリング指標 |
|---|---|---|
| tier2 Kafka Consumer (Dapr sidecar 経由) | 採用後の運用拡大時 | Kafka Consumer Group の Lag (未処理メッセージ数) |
| Temporal Worker (tier1 Go ファサード) | 採用後の運用拡大時 | Temporal Task Queue のバックログ深度 |
| tier2 / tier3 同期 API サービス | 採用後の運用拡大時 | Prometheus の HTTP RPS / レイテンシメトリクス |

現行の非機能要件 ([`../../01_アーキテクチャ/04_非機能とデータ/01_非機能要件.md`](../../01_アーキテクチャ/04_非機能とデータ/01_非機能要件.md)) ではスケーリング方式が HPA (CPU / RPS ベース) と定義されている。しかし CPU ベースの HPA では以下の問題が発生する。

- **Kafka Consumer**: Consumer Lag が 10 万件溜まっていても、Consumer Pod の CPU 使用率が低ければ HPA はスケールアウトしない。結果としてイベント処理の遅延が蓄積する
- **Temporal Worker**: Task Queue にタスクが滞留していても、Worker Pod はタスク待ちでアイドル状態のため CPU が低い。HPA はスケールアウトしない

### 採用理由 (KEDA)

1. **CNCF Graduated プロジェクト** — 2023 年に CNCF Graduated に昇格。Microsoft / Red Hat が主導し、コントリビューター 1,000+ で活発にメンテナンスされている。k1s0 の選定方針「CNCF プロジェクト優先」に適合する
2. **HPA の拡張として動作** — KEDA は既存の HPA を置き換えるのではなく、HPA の external metrics source として動作する。既存の HPA 設定を破壊せず、追加のスケーリングトリガーを宣言的に定義できる
3. **0→1→N のスケーリング** — HPA は minReplicas=1 が最小だが、KEDA は minReplicaCount=0 をサポートする。採用側組織の稼働時間 (平日 8:00〜20:00) 外にトラフィックのない tier2 / tier3 サービスを 0 Pod にスケールインし、リリース時点環境 (3 ノード × 16 GB) の貴重なメモリを夜間に解放できる
4. **Kafka スケーラーの組み込み** — KEDA は Apache Kafka スケーラーを標準で内蔵しており、Consumer Group の Lag に基づいてスケーリングする。Strimzi の Kafka と直接統合でき、追加のアダプターが不要
5. **Temporal スケーラーの組み込み** — KEDA は Temporal スケーラーを標準で内蔵しており、Task Queue のバックログ深度に基づいて Worker Pod をスケーリングする
6. **Prometheus スケーラー** — Prometheus のカスタムメトリクス (HTTP RPS / レイテンシ / エラー率等) をスケーリングトリガーとして利用できる。tier2 / tier3 の同期 API サービスのスケーリングに使用する
7. **ScaledObject CRD による宣言的管理** — スケーリングポリシーを ScaledObject CRD として YAML で定義する。Kustomize の overlay で環境別のスケーリング閾値を管理でき、Argo CD の GitOps フローに統合される
8. **軽量** — KEDA Operator + Metrics API Server で合計約 0.3 GB のメモリ消費。k1s0 の 採用後の運用拡大時 リソースバジェット内に収まる

### k1s0 への統合設計

#### 雛形生成 CLI との統合

雛形生成 CLI がサービス初期化時に ScaledObject CRD を自動生成する。tier2 / tier3 の開発者はスケーリング設定を自分で書かない。

| サービス種別 | 雛形が生成する ScaledObject のトリガー | デフォルト設定 |
|---|---|---|
| Kafka Consumer サービス | `kafka` (Consumer Lag) | lagThreshold: 100、maxReplicaCount: 10 |
| 同期 API サービス | `prometheus` (HTTP RPS) | threshold: 100 (req/s/pod)、maxReplicaCount: 5 |
| Temporal Worker | `temporal` (Task Queue バックログ) | activationThreshold: 5、maxReplicaCount: 10 |

デフォルト値は tier1 チームが定義し、tier2 / tier3 が業務要件に応じて overlay で調整する。

#### Dapr 隠蔽方針との整合

KEDA の ScaledObject は Kafka Consumer Group 名や Temporal Task Queue 名を参照する。これらは Dapr の抽象化の外側にある情報だが、**ScaledObject の生成は雛形生成 CLI が担い、tier2 / tier3 開発者が直接記述する必要はない**。Dapr 隠蔽方針 ([`../../02_tier1設計/01_設計の核/01_Dapr隠蔽方針.md`](../../02_tier1設計/01_設計の核/01_Dapr隠蔽方針.md)) の「tier2 / tier3 は内部実装を意識しない」原則と整合する。

#### Scale-to-Zero の運用ルール

| 対象 | minReplicaCount | 理由 |
|---|---|---|
| tier1 Go ファサード | 2 (KEDA 対象外、HPA 維持) | プラットフォーム基盤は常時稼働必須 |
| tier1 Rust サービス | 2 (KEDA 対象外、HPA 維持) | 同上 |
| tier2 Consumer 系 | 0 | イベントがなければ Pod 不要 |
| tier2 同期 API 系 | 1 | コールドスタート遅延を回避 |
| tier3 Web アプリ | 1 | 同上 |
| Temporal Worker | 0 | タスクがなければ Worker 不要 |

tier2 Consumer 系と Temporal Worker は 0→1 のスケールアップ時にコールドスタート (Pod 起動 + Dapr sidecar injection) で 5〜15 秒の遅延が発生する。非同期処理であるため許容可能。同期 API 系は初回リクエストの遅延がユーザー体験に影響するため minReplicaCount=1 を維持する。

### ScaledObject CRD の例

```yaml
apiVersion: keda.sh/v1alpha1
kind: ScaledObject
metadata:
  name: order-consumer
  namespace: tier2
spec:
  scaleTargetRef:
    name: order-consumer
  minReplicaCount: 0
  maxReplicaCount: 10
  cooldownPeriod: 300
  triggers:
    - type: kafka
      metadata:
        bootstrapServers: kafka-bootstrap.infra.svc.cluster.local:9092
        consumerGroup: order-consumer
        topic: tier2.order.order.events
        lagThreshold: "100"
```

### デプロイ構成

```text
KEDA Operator (Deployment, 2 レプリカ)
    ├── ScaledObject Controller — ScaledObject CRD を watch し HPA を自動生成・管理
    └── Metrics API Server — 外部メトリクス (Kafka Lag / Temporal Queue 等) を k8s Metrics API に公開
```

### OSS リリース時点での採用範囲

- **採用後の運用拡大時 で導入** — Kafka (Strimzi) + Temporal の導入と同時
- `infra` namespace にデプロイ (Helm Chart `kedacore/keda` でインストール)
- 採用後の運用拡大時 初期: Kafka Consumer Lag トリガー + Prometheus RPS トリガーを先行導入
- 採用後の運用拡大時 中期: Temporal Task Queue トリガーを追加 (Temporal 安定後)
- 採用側のマルチクラスタ移行時: Scale-to-Zero を夜間バッチ以外の tier2 Consumer 系に適用。CronJob で業務時間帯の minReplicaCount を動的制御

### Argo CD / Kustomize との統合

| 項目 | 方針 |
|---|---|
| KEDA Operator のインストール | Helm Chart (`kedacore/keda`) を Argo CD Application で管理 |
| ScaledObject CRD | 各サービスの `deploy/base/` に配置。雛形生成 CLI が自動生成 |
| 環境別の閾値調整 | Kustomize overlay で `lagThreshold` / `maxReplicaCount` をパッチ |
| Renovate による更新 | KEDA Helm Chart のバージョンを Renovate が自動検知し、更新 PR を生成 |

### トレードオフ

- KEDA Operator の障害時にスケールアウト/インが停止する。ただし既に稼働している Pod は影響を受けず、現在のレプリカ数で固定される。HPA 障害時と同じ挙動
- Kafka Consumer Lag ベースのスケーリングは Consumer Group のリバランスを伴う。頻繁なスケールアウト/インはリバランスコストを増大させる。`cooldownPeriod` (デフォルト 300 秒) で緩和する
- Scale-to-Zero 時のコールドスタート遅延 (5〜15 秒) は同期 API サービスには不適。同期 API 系は minReplicaCount=1 を維持する運用ルールで対処する
- KEDA の ScaledObject は Kafka の Consumer Group 名や Temporal の Task Queue 名を直接参照するため、これらの命名規則を tier1 チームが一元管理する必要がある
