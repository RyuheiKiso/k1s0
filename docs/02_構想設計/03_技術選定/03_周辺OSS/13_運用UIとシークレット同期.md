# 運用 UI とシークレット同期 OSS の選定

## 目的

k8s クラスタ運用時の GUI 可視化 (Headlamp) と API トラフィックデバッグ (Kubeshark)、そして OpenBao と k8s Secret の橋渡しを担うシークレット同期 (External Secrets Operator) を整理する。いずれも運用現場の "触って分かる" を支える領域であり、シークレット同期は [`08_シークレット管理.md`](08_シークレット管理.md) の OpenBao と対になる。対象カテゴリは以下のとおり。

- AA. Kubernetes Web UI
- AB. API トラフィックビューア
- AC. シークレット同期

---

## AA. Kubernetes Web UI

| 候補 | 採否 | 評価 |
|---|---|---|
| **Headlamp** | 採用 | CNCF Sandbox。軽量 (単一バイナリ / コンテナ)。プラグイン拡張可能。Apache 2.0 |
| Kubernetes Dashboard | 却下 | 公式だが UI が旧式。RBAC 設定が煩雑で 採用側組織の運用者が安全に利用するための追加設定が多い |
| Lens | 却下 | デスクトップアプリケーション (Electron)。各運用者の PC にインストールが必要であり、採用側組織の端末管理ポリシーと衝突する可能性が高い。ライセンスが Lens EULA (非 OSI) |
| Rancher UI | 却下 | Rancher 全体のインストールが前提。k8s ディストリビューションとして Rancher を採用していないため過剰 |
| k9s | 次点 | ターミナル UI。kubectl に慣れた開発者には有用だが、採用側組織の運用者 (石田ペルソナ) への GUI 提供が目的のため不適 |
| kubectl のみ | 却下 | 採用側組織の運用者にとって CLI ベースの運用は学習コストが高く、操作ミスのリスクが大きい |

### 採用理由 (Headlamp)

1. **採用側組織の運用者 (石田ペルソナ) に kubectl 不要の Web UI を提供** — kubectl のコマンド体系は k8s 専門家向けであり、採用側組織の運用リーダーが日常的に利用するには学習コストが高い。Headlamp はブラウザベースの Web UI で Pod 状態・ログ・イベントを視覚的に確認でき、運用の民主化を実現する
2. **CNCF Sandbox プロジェクト** — Kubernetes エコシステムの一部として CNCF に受け入れられており、k8s API との互換性が継続的に検証されている。コミュニティも活発で長期的な保守が期待できる
3. **軽量で運用負荷が極めて低い** — 単一コンテナ (Deployment 1 レプリカ) をデプロイするだけで動作する。メモリ消費は約 0.1 GB であり、採用初期 のリソース制約内で十分に収まる
4. **Keycloak OIDC 統合** — Headlamp は OIDC 認証をネイティブサポートしており、Keycloak と直接統合できる。k1s0 の SSO 統一方針に適合し、追加の認証基盤構築が不要
5. **プラグインアーキテクチャ** — TypeScript/React ベースのプラグイン機構を持ち、採用後の運用拡大時で tier1 固有のダッシュボード (Dapr Component 状態・OpenBao Secret 一覧等) を追加する拡張パスがある
6. **読み取り専用モードのサポート** — RBAC と組み合わせて運用者向けに読み取り専用ビューを提供できる。採用側組織の環境で「見るだけ」の安全な UI を確保し、誤操作リスクを排除する

### Kubernetes Dashboard との比較

| 観点 | Headlamp | Kubernetes Dashboard |
|---|---|---|
| CNCF 状態 | Sandbox | — (k8s SIG 管理) |
| UI フレームワーク | React (モダン) | AngularJS → Angular (レガシーからの移行中) |
| プラグイン拡張 | TypeScript/React プラグイン | 非対応 |
| OIDC 認証 | ネイティブサポート | ServiceAccount トークンが主流、OIDC は追加設定要 |
| マルチクラスタ | 対応 (採用側のマルチクラスタ移行時 で活用) | 単一クラスタ |
| ライセンス | Apache 2.0 | Apache 2.0 |
| メモリ | 約 0.1 GB | 約 0.2 GB |

### デプロイ構成

```text
Headlamp (Deployment, 1 レプリカ)
    ├── Keycloak OIDC 認証
    ├── k8s API Server への接続 (ServiceAccount)
    └── Envoy Gateway 経由で外部公開
```

### RBAC 設計

| ロール | 権限 | 対象ペルソナ |
|---|---|---|
| `headlamp-viewer` | 全 namespace の読み取り (get / list / watch) | 石田 (運用リーダー)、田辺 (レガシー管理者) |
| `headlamp-operator` | viewer + Pod 再起動 / Scale 操作 | 石田 (運用リーダー) |
| `headlamp-admin` | 全操作 (tier1 チーム限定) | 起案者 / 協力者 |

Keycloak のグループをベースに k8s ClusterRoleBinding を作成し、ロールを自動マッピングする。

### OSS リリース時点での採用範囲

- **採用初期 から導入**
- `operation` namespace にデプロイ (Deployment、1 レプリカ)
- Keycloak OIDC 連携で SSO ログイン
- 運用者向けに `headlamp-viewer` ロールを提供 (読み取り専用)
- Grafana と並んで「運用者が最初に開くダッシュボード」として位置付ける
- 採用後の運用拡大時 でプラグイン追加 (Argo CD 状態表示、Dapr Component 状態表示) を検討

### メモリ概算

| 構成 | メモリ |
|---|---|
| Headlamp (1 レプリカ) | 0.1 GB |

### トレードオフ

- Kubernetes Dashboard と比較してコミュニティ規模がやや小さい。ただし CNCF Sandbox への採択で成長が加速しており、リスクは許容範囲
- プラグイン開発には TypeScript/React の知識が必要。採用後の運用拡大時のカスタムプラグイン開発は tier1 チーム (Rust/Go 主体) の担当範囲外となる可能性がある。ただし 採用初期 時点ではプラグイン開発は不要
- Headlamp は k8s API Server への直接アクセスを行うため、RBAC の設定が不十分だと意図しないリソースが表示される。Kyverno (採用初期) のラベル強制と合わせて、tier 別のアクセス制御を確実にする

## AB. API トラフィックビューア

| 候補 | 採否 | 評価 |
|---|---|---|
| **Kubeshark** | 採用 | eBPF ベースのリアルタイム API トラフィックキャプチャ。HTTP / gRPC / Kafka プロトコルを可視化。Apache 2.0 |
| tcpdump + Wireshark | 却下 | パケットレベルの分析であり、HTTP/gRPC のリクエスト/レスポンスを意味的に解釈できない。運用者にとって使いづらい |
| Kiali | 次点 | Istio 専用のサービスメッシュ可視化 UI。Istio 未導入の リリース時点では利用不可。採用後の運用拡大時で併用を検討 |
| kubectl logs のみ | 却下 | ログに含まれない通信内容 (ヘッダー / ペイロード / レイテンシ) を確認できない |
| Pixie (New Relic) | 却下 | eBPF ベースだがライセンスが Apache 2.0 + Community License (一部機能制限)。New Relic エコシステムへの依存リスク |

### 採用理由 (Kubeshark)

1. **eBPF ベースのゼロインストルメンテーション** — アプリケーションへのコード変更やサイドカー追加なしで、ノードレベルの eBPF フックにより全 HTTP / gRPC / Kafka トラフィックをリアルタイムキャプチャする。tier2 / tier3 開発者がデバッグのためにコードを変更する必要がない
2. **サービスメッシュ非依存** — Istio 未導入の 採用初期 / 採用初期 でも動作する。Istio 導入後 (採用後の運用拡大時) は暗号化前のトラフィックを eBPF で取得するため、mTLS 環境でも可視性を確保できる
3. **gRPC / Kafka プロトコルのネイティブ解析** — k1s0 は tier1 公開 API に gRPC、非同期メッセージングに Kafka を採用している。Kubeshark はこれらのプロトコルをデコードし、リクエスト/レスポンスの内容をリアルタイムで表示する。tcpdump では不可能なアプリケーションレベルの可視性を提供する
4. **オンデマンドデプロイ** — 常駐ではなくデバッグ時のみ起動する運用を推奨する。`kubeshark tap` コマンドで DaemonSet を一時的にデプロイし、調査完了後に `kubeshark clean` で撤去する。常駐コンポーネントとしてのリソース消費・セキュリティ表面積をゼロにする
5. **Web UI でのリアルタイム表示** — ブラウザベースの Web UI でトラフィックをリアルタイムにフィルタリング・検索できる。開発者が「このリクエストがどのサービスにどのように到達したか」を即座に確認できる
6. **PCAP エクスポート** — キャプチャしたトラフィックを PCAP 形式でエクスポートし、Wireshark での詳細分析が可能。障害の事後分析やセキュリティ監査に活用できる

### 利用シナリオ

| シナリオ | 操作 | 効果 |
|---|---|---|
| tier1 API デバッグ | `kubeshark tap -n tier1` | tier1 Go ファサードへの gRPC リクエスト/レスポンスをリアルタイム確認 |
| tier2 サービス間通信の調査 | `kubeshark tap -n tier2` | tier2 サービス間の HTTP 呼び出しの内容・レイテンシを確認 |
| Kafka メッセージの確認 | `kubeshark tap -n infra` | Kafka ブローカーへの produce/consume メッセージを確認 (採用後の運用拡大時) |
| 障害時の緊急調査 | `kubeshark tap --all-namespaces` | 全トラフィックをキャプチャし、障害箇所を特定 |
| Dapr sidecar 通信の可視化 | `kubeshark tap -n tier1` | アプリ → Dapr sidecar → バックエンドの通信経路を確認 |

### セキュリティ上の考慮

Kubeshark は全トラフィックを平文で表示する能力を持つため、以下のアクセス制御を徹底する。

| 制御 | 内容 |
|---|---|
| 利用権限 | tier1 チーム (起案者 / 協力者) に限定。tier2 / tier3 開発者には原則非公開 |
| 実行制御 | `kubeshark` CLI は GHA runner Pod / 起案者・協力者の PC にのみ配置。Keycloak の RBAC グループで k8s RBAC を制限 |
| 常駐禁止 | 常時起動は禁止。調査完了後に必ず `kubeshark clean` で撤去 |
| PII マスキング | Kubeshark の設定でセンシティブフィールド (Authorization ヘッダー等) をマスク表示する |

### Kiali との棲み分け (採用後の運用拡大時)

| ツール | 目的 | Istio 依存 | 常駐 |
|---|---|---|---|
| Kubeshark | リアルタイムトラフィックキャプチャ・デバッグ | 不要 | オンデマンド |
| Kiali | サービスメッシュトポロジ可視化・ヘルスチェック | 必須 | 常駐 |

Kubeshark は「今この瞬間の通信内容を見る」ツール。Kiali は「サービス間の関係性と健全性を俯瞰する」ツール。採用後の運用拡大時 で Istio を導入した際に Kiali の追加採用を検討する。

### OSS リリース時点での採用範囲

- **採用初期 から利用可能** (クラスタへのデプロイはオンデマンド)
- `kubeshark` CLI を起案者・協力者の PC にインストール
- GHA runner Pod のカスタムイメージには含めない (CI での利用は想定しない)
- Runbook に「障害調査時の Kubeshark 利用手順」を記載 (採用初期)
- 採用後の運用拡大時 の Istio / Kafka 導入後に gRPC / Kafka プロトコル解析の本格活用

### メモリ概算

| 構成 | メモリ |
|---|---|
| Kubeshark (オンデマンド、DaemonSet 3 ノード) | 0.5 GB (調査時のみ) |

常駐ではないため、クラスタのメモリ概算には含めない。調査時に一時的にリソースを消費する。

### トレードオフ

- eBPF を使用するため、カーネルバージョン 4.16+ が必要。採用側組織のオンプレ VM で使用する Linux ディストリビューション (Ubuntu 22.04+ / RHEL 9+ 推奨) であれば問題なし
- 全トラフィックをキャプチャする能力があるため、セキュリティポリシーの策定が必須。上記のアクセス制御で対応
- オンデマンドデプロイのため、「障害発生前のトラフィック」は取得できない。事後分析が必要な場合は OTel Collector のログ・トレースと組み合わせる
- Kubeshark のコミュニティ規模は Kiali 等と比較して小さい。ただし eBPF ベースのトラフィック可視化ツールとしてはデファクトであり、Apache 2.0 ライセンスのためフォークも可能

## AC. シークレット同期 (External Secrets Operator)

| 候補 | 採否 | 評価 |
|---|---|---|
| **External Secrets Operator (ESO)** | 採用 | CNCF Certified。OpenBao / Vault の Secret を k8s Secret に自動同期する Operator。Apache 2.0 |
| Dapr Secret Store のみ | 不十分 | tier1 公開 API (`k1s0.Secrets`) 経由のアプリケーション層シークレット取得は Dapr で対応済みだが、**インフラ層コンポーネント (CloudNativePG / Keycloak / Argo CD / Harbor 等) が直接参照する k8s Secret** は Dapr の管轄外 |
| k8s Secrets (手動管理) | 却下 | ローテーション時に手動で `kubectl create secret` を再実行する必要がある。OpenBao の自動ローテーションの価値が失われる |
| SealedSecrets のみ | 不十分 | Git 管理された暗号化 Secret を復号して k8s Secret に展開するが、OpenBao の動的シークレットやローテーションとは連携できない |

### 旧選定 (却下) からの変更理由

[`08_シークレット管理.md`](08_シークレット管理.md) の初期選定で ESO を「OpenBao を直接 Dapr Secret Store として利用するため不要」と却下していた。以下の理由で採用に変更する。

1. **Dapr Secret Store はアプリケーション層のシークレット取得** を担うが、**インフラ層のコンポーネントは Dapr を経由しない**。CloudNativePG / Keycloak / Argo CD / Harbor / cert-manager 等は k8s Secret を直接マウントしてパスワード・TLS 証明書を取得する。これらのコンポーネントに OpenBao のシークレットを届けるには、OpenBao → k8s Secret への同期メカニズムが必要
2. **OpenBao の Database Engine で動的生成された PostgreSQL 認証情報** は TTL 付きで定期ローテーションされる。ESO の `refreshInterval` で定期同期すれば、ローテーションされた新しい認証情報が自動的に k8s Secret に反映される
3. **SealedSecrets は静的暗号化** であり、OpenBao の動的シークレットとは異なる用途。ESO が OpenBao からの動的同期を担い、SealedSecrets が Git 管理の bootstrap 設定を担う棲み分けが明確になる

### アーキテクチャ

```text
OpenBao (infra namespace)
    │
    │  ← API 経由で Secret 取得
    │
External Secrets Operator (infra namespace)
    │
    │  ← ExternalSecret CRD を watch
    │  → k8s Secret を作成/更新
    │
k8s Secret (各 namespace)
    │
    ├── CloudNativePG (PostgreSQL パスワード)
    ├── Keycloak (管理者パスワード / DB パスワード)
    ├── Argo CD (GitHub トークン / Webhook シークレット)
    ├── Harbor (管理者パスワード / DB パスワード)
    └── cert-manager (CA 秘密鍵)
```

### Dapr Secret Store との棲み分け

| 観点 | Dapr Secret Store | External Secrets Operator |
|---|---|---|
| 利用者 | tier1 公開 API (`k1s0.Secrets`) → tier2 / tier3 アプリ | インフラ層コンポーネント (CloudNativePG / Keycloak / Harbor 等) |
| 取得タイミング | アプリケーション起動時・実行時にオンデマンド | ExternalSecret CRD の `refreshInterval` で定期同期 |
| ローテーション | アプリが次回取得時に新しい値を受領 | k8s Secret が自動更新 → Pod が volume reload で反映 |
| 管理対象 | アプリケーションが使うシークレット | インフラコンポーネントが使う k8s Secret |

**両者は競合しない。Dapr はアプリケーション層、ESO はインフラ層** のそれぞれのシークレット配信を担い、OpenBao を単一の真実のソース (Single Source of Truth) として統合する。

### 採用理由 (External Secrets Operator)

1. **OpenBao を Single Source of Truth にする** — 全シークレットの格納先を OpenBao に一元化し、k8s Secret はその射影 (projection) として自動生成する。シークレットの二重管理を排除する
2. **動的シークレットの自動反映** — OpenBao Database Engine で生成された TTL 付き PostgreSQL 認証情報が、ESO の `refreshInterval` で k8s Secret に自動反映される。手動ローテーションが完全に不要になる
3. **CNCF Certified** — CNCF の認定を受けており、エコシステムとの統合が保証されている。GoDaddy が開発を開始し、現在は CNCF コミュニティが管理。コントリビューター 500+ で活発
4. **OpenBao (Vault 互換) プロバイダーが公式サポート** — ESO は HashiCorp Vault プロバイダーを公式サポートしており、OpenBao の Vault API 互換性により設定変更なしで接続できる
5. **ExternalSecret CRD による宣言的管理** — どの OpenBao パスから k8s Secret のどのキーに同期するかを YAML で宣言的に定義する。Kustomize の overlay で環境別の Secret パスを管理できる
6. **SecretStore / ClusterSecretStore の分離** — namespace 単位で接続先を分離でき、tier 間のシークレット分離を k8s ネイティブに実現する

### ExternalSecret CRD の例

```yaml
apiVersion: external-secrets.io/v1
kind: ExternalSecret
metadata:
  name: cloudnativepg-credentials
  namespace: infra
spec:
  refreshInterval: 5m
  secretStoreRef:
    name: openbao-kv
    kind: ClusterSecretStore
  target:
    name: cloudnativepg-credentials
    creationPolicy: Owner
  data:
    - secretKey: username
      remoteRef:
        key: secret/data/infra/cloudnativepg
        property: username
    - secretKey: password
      remoteRef:
        key: secret/data/infra/cloudnativepg
        property: password
```

### ClusterSecretStore の設計

| SecretStore | OpenBao パス | 対象 namespace | 用途 |
|---|---|---|---|
| `openbao-kv` | `secret/data/infra/*` | `infra` | インフラ層の静的シークレット |
| `openbao-kv-operation` | `secret/data/operation/*` | `operation` | Argo CD / Backstage / Keycloak のシークレット |
| `openbao-kv-tier1` | `secret/data/tier1/*` | `tier1` | tier1 内部サービスのシークレット |
| `openbao-db` | `database/creds/*` | `infra` / `tier1` | PostgreSQL 動的認証情報 (採用後の運用拡大時) |

### SealedSecrets / Dapr / ESO の三者整理

| 手段 | 目的 | 対象 | 導入時期 | OpenBao 導入後の役割 |
|---|---|---|---|---|
| **SealedSecrets** | Secret の暗号化 Git 管理 | OpenBao bootstrap 設定 / OpenTofu 初期値 | 採用初期 | bootstrap 設定のみに縮小 |
| **Dapr Secret Store** | アプリケーション層のシークレット取得 | tier2 / tier3 が `k1s0.Secrets` API 経由で取得 | 採用後の運用拡大時 | 変更なし (Dapr → OpenBao の経路) |
| **ESO** | インフラ層の k8s Secret 同期 | CloudNativePG / Keycloak / Argo CD / Harbor 等 | 採用初期 | OpenBao → k8s Secret の自動同期 |

### OSS リリース時点での採用範囲

- **採用初期 から導入** (OpenBao と同時)
- `infra` namespace にデプロイ (ESO Controller、1 レプリカ)
- ClusterSecretStore `openbao-kv` を作成し、OpenBao KV Engine v2 に接続
- ExternalSecret CRD で以下の k8s Secret を OpenBao から同期:
  - CloudNativePG の PostgreSQL パスワード
  - Keycloak の管理者パスワード
  - Argo CD の GitHub トークン
  - Harbor の管理者パスワード
- `refreshInterval: 5m` で定期同期
- 採用後の運用拡大時 で Database Engine の動的認証情報同期を追加

### メモリ概算

| 構成 | メモリ |
|---|---|
| ESO Controller (1 レプリカ) | 0.15 GB |
| ESO Webhook (1 レプリカ) | 0.1 GB |
| **合計** | **0.25 GB** |

### トレードオフ

- ESO Controller の障害時に k8s Secret の更新が停止する。ただし既存の k8s Secret は削除されないため、即座にサービス影響は発生しない。ESO Controller の再起動で自動復旧する
- OpenBao と ESO の二重障害時はシークレットローテーションが停止するが、既存の Secret で運用は継続可能。Prometheus アラートで ESO の同期エラーを検知する
- ExternalSecret CRD が増えるとコントローラの watch 負荷が増加する。リリース時点規模 (10〜20 ExternalSecret) では問題なし。採用側のマルチクラスタ移行時で 100+ になった場合はレプリカ数を増加する
- SealedSecrets / Dapr Secret Store / ESO の三者が存在することで、シークレット管理の経路が複雑に見える。ただし各手段の責務は明確に分離されており、運用 Runbook でフローを図示する
