# サプライチェーンセキュリティ OSS の選定

## 目的

コンテナイメージの署名検証 (Cosign) と、SBOM 生成・脆弱性スキャン (Syft + Grype) を整理する。対象カテゴリは以下のとおり。

- AE. コンテナイメージ署名 (Cosign)
- AF. SBOM 生成・脆弱性スキャン (Syft + Grype)

いずれもサプライチェーンセキュリティの強化を目的とする。[`../../01_アーキテクチャ/03_セキュリティ/01_セキュリティモデル.md`](../../01_アーキテクチャ/03_セキュリティ/01_セキュリティモデル.md) のセクション 7「サプライチェーンセキュリティ」で計画された施策の実装手段である。

## AE. コンテナイメージ署名

| 候補 | 採否 | 評価 |
|---|---|---|
| **Cosign (Sigstore)** | 採用 | OCI イメージ・アーティファクトへの署名・検証。CNCF プロジェクト (Sigstore) の一部。鍵ペア署名とキーレス署名の両方をサポート。Apache 2.0 |
| Notation (Notary v2) | 次点 | CNCF プロジェクト。OCI 署名仕様準拠だが、Kyverno の verifyImages との統合実績が Cosign より少ない。エコシステムが発展途上 |
| Docker Content Trust (Notary v1) | 却下 | Docker Hub 前提の設計。OCI 標準との乖離が大きく、Harbor / Kyverno との統合が困難 |
| 手動ダイジェスト固定のみ | 不十分 | イメージの改ざん検知は可能だが、ビルド主体の証明ができない。出所保証なし |

### 背景: なぜ今決定が必要か

[`../../01_アーキテクチャ/03_セキュリティ/01_セキュリティモデル.md`](../../01_アーキテクチャ/03_セキュリティ/01_セキュリティモデル.md) で、採用後の運用拡大時 のサプライチェーンセキュリティ対策として「Cosign 署名 + Kyverno 未署名拒否」が計画されている。また [`../../04_CICDと配信/00_CICDパイプライン.md`](../../04_CICDと配信/00_CICDパイプライン.md) の CI パイプラインツール一覧でも Cosign が 採用後の運用拡大時のイメージ署名ツールとして記載済みである。

本ドキュメントは、これらの計画に対する正式な選定根拠の記録である。

### 採用理由 (Cosign)

1. **Kyverno との成熟した統合** — Kyverno の `verifyImages` ルールは Cosign 署名の検証をネイティブにサポートする。ClusterPolicy に公開鍵を埋め込むだけで、未署名または不正署名のイメージを Admission Webhook で拒否できる。k1s0 は 採用初期 で Kyverno を導入済みであり、採用後の運用拡大時 で署名検証ポリシーを追加するだけで統合が完了する
2. **Harbor のネイティブサポート** — Harbor は Cosign 署名をアーティファクトとして保存・表示する機能を持つ。署名済みイメージには Harbor UI 上で署名バッジが表示され、運用者が視覚的に確認できる
3. **オンプレ環境での鍵ペア署名** — Cosign は鍵ペア署名 (`cosign generate-key-pair` → `cosign sign --key`) をサポートする。採用側組織のオンプレ環境ではキーレス署名 (Fulcio + Rekor) の OIDC プロバイダ依存が運用上のリスクになるため、鍵ペア署名を採用する。秘密鍵は OpenBao の Transit Engine に保管し、署名操作は OpenBao API 経由で実行する
4. **CI パイプラインへの統合が容易** — `cosign sign` コマンドを GHA ステップに 1 行追加するだけでイメージ署名が完了する。既存の CI パイプライン (`Harbor push` ステップの直後) に最小限の変更で組み込める
5. **OCI 標準準拠** — Cosign は OCI Distribution Spec に準拠した署名形式を使用する。署名はイメージとは別の OCI Artifact として Harbor に保存されるため、イメージ自体を変更しない
6. **CNCF エコシステムの中心** — Sigstore は CNCF の General Availability プロジェクトであり、Kubernetes / Helm / npm / PyPI 等の主要エコシステムで採用が進んでいる。長期的な保守と互換性が期待できる

### Notation を選ばない理由

Notation (Notary v2) は OCI 署名の標準仕様に沿ったツールであり、技術的には健全だが、以下の理由で k1s0 では Cosign を優先する。

| 比較軸 | Cosign | Notation |
|---|---|---|
| Kyverno 統合 | `verifyImages` でネイティブサポート。ドキュメント・実績ともに豊富 | サポートはあるが実績が少なく、設定パターンのドキュメントが限定的 |
| Harbor 統合 | 署名バッジ表示・署名アーティファクト保存 | 同等の機能あり |
| CLI の成熟度 | Stable。バージョン 2.x で API 安定 | GA だが破壊的変更のリスクがまだ残る |
| 採用側組織のでの認知度 | Sigstore / Cosign は CNCF レポートで頻出。採用検討資料での説明コストが低い | 認知度がやや低い |

### k1s0 への統合設計

#### 署名フロー

```text
GHA CI パイプライン
  ├── Build (Kaniko)
  ├── Push to Harbor
  ├── SBOM 生成 (Syft) → Harbor に SBOM attach
  ├── Cosign sign (秘密鍵: OpenBao Transit Engine)  ← 追加
  └── GitOps マニフェスト更新
```

CI パイプラインの `Harbor push` ステップと `GitOps マニフェスト更新` ステップの間に `cosign sign` を挿入する。

#### 鍵管理

| 項目 | 方針 |
|---|---|
| 秘密鍵の保管場所 | OpenBao Transit Engine (`cosign` キー) |
| 公開鍵の配布先 | Kyverno ClusterPolicy に埋め込み + Git リポジトリに保管 |
| 鍵のローテーション | 年 1 回。旧公開鍵は Kyverno Policy に一定期間並行登録 |
| CI からの署名操作 | GHA runner Pod が OpenBao の AppRole 認証で Transit Engine にアクセスし署名 |

OpenBao の Transit Engine は秘密鍵をエクスポートせずに署名操作を実行できるため、秘密鍵がディスク上や環境変数に平文で存在するリスクを排除する。

#### Kyverno 署名検証ポリシー

Kyverno ClusterPolicy で、`tier1` / `tier2` / `tier3` namespace へのイメージデプロイ時に Cosign 署名を検証する。

```yaml
apiVersion: kyverno.io/v1
kind: ClusterPolicy
metadata:
  name: verify-image-signature
spec:
  validationFailureAction: Enforce
  rules:
    - name: verify-cosign-signature
      match:
        any:
          - resources:
              kinds:
                - Pod
              namespaces:
                - tier1
                - tier2
                - tier3
      verifyImages:
        - imageReferences:
            - "harbor.k1s0.internal/*"
          attestors:
            - entries:
                - keys:
                    publicKeys: |-
                      -----BEGIN PUBLIC KEY-----
                      ...
                      -----END PUBLIC KEY-----
```

`infra` / `operation` namespace のサードパーティイメージ (Prometheus / Grafana 等) は署名検証の対象外とする。これらは Harbor のイメージソース制限 (採用初期 の Kyverno ポリシー) と Trivy スキャンで保護する。

### Rekor (透明性ログ) の採用判断

| 項目 | 判断 |
|---|---|
| 採用後の運用拡大時 | 不採用。鍵ペア署名で十分。透明性ログのオンプレ運用はコストに見合わない |
| 採用側の全社展開期 以降 | 再評価。複数チーム・複数クラスタで署名監査が必要になった時点で検討 |

Rekor はイメージ署名の透明性ログ (「いつ・誰が・何に署名したか」の不変記録) を提供するが、採用後の運用拡大時 時点では署名者が CI パイプラインのみであり、OpenBao の Audit Log で署名操作の追跡が可能なため、Rekor の追加価値は限定的である。

| コンポーネント | ライセンス | 備考 |
|---|---|---|
| Cosign | Apache 2.0 | Sigstore プロジェクトの一部 |

### メモリ概算

| 構成 | メモリ |
|---|---|
| Cosign (CLI、CI runner Pod 内で実行) | 0 GB (runner Pod に含まれる) |

Cosign はデーモンプロセスを持たない CLI ツールであり、CI パイプラインの GHA runner Pod 内で実行する。クラスタ上に追加のリソース消費はない。Kyverno の署名検証は既存の Kyverno Pod 内で処理される (追加メモリ: 約 0.05 GB)。

### OSS リリース時点での採用範囲

- **採用後の運用拡大時 で導入** — Kyverno (採用初期) + OpenBao Transit Engine (採用後の運用拡大時) が前提
- GHA runner のカスタムイメージに `cosign` CLI を同梱
- CI パイプラインに `cosign sign` ステップを追加 (Harbor push 直後)
- OpenBao Transit Engine に `cosign` 用の署名キーを作成
- Kyverno ClusterPolicy に `verifyImages` ルールを追加
- 採用後の運用拡大時 初期は `validationFailureAction: Audit` (警告のみ) で運用し、全イメージの署名が完了した時点で `Enforce` (拒否) に切り替え

### 段階的な強制適用

| 段階 | Kyverno validationFailureAction | 対象 |
|---|---|---|
| 採用後の運用拡大時 初期 | `Audit` (警告のみ) | 全 namespace |
| 採用後の運用拡大時 中期 | `Enforce` (拒否) | `tier1` namespace のみ |
| 採用後の運用拡大時 後期 | `Enforce` (拒否) | `tier1` / `tier2` / `tier3` 全 namespace |

### トレードオフ

- OpenBao Transit Engine が停止すると CI パイプラインでの署名が不可能になる。ただしデプロイ自体は Kyverno を `Audit` モードにすることで継続可能。Transit Engine の復旧を優先する運用 Runbook を整備する
- 鍵ペア署名は鍵のローテーション管理が必要。キーレス署名 (Fulcio) であればこの運用負荷はないが、オンプレ環境での OIDC プロバイダ依存を回避するために鍵ペア方式を選択する
- サードパーティ OCI イメージ (Prometheus / Grafana 等) は Cosign 署名検証の対象外。これらのイメージの信頼性は、Harbor のイメージミラーリング + Trivy スキャンで担保する
- Cosign v2 以降で署名形式が OCI Artifact (Referrers API) に移行する可能性がある。Harbor のバージョンと Cosign のバージョンの互換性を Renovate で追跡する

## AF. SBOM 生成・脆弱性スキャン

| 候補 | 採否 | 評価 |
|---|---|---|
| **Syft** (SBOM 生成) | 採用 | Anchore 社開発の SBOM 生成ツール。CycloneDX / SPDX / Syft JSON 形式をサポート。OCI イメージ・ファイルシステム・コンテナアーカイブに対応。Apache 2.0 |
| **Grype** (SBOM 脆弱性スキャン) | 採用 | Anchore 社開発の脆弱性スキャナ。Syft SBOM を入力として脆弱性を検出。NVD / GitHub Advisory / OS ディストリビューション DB を参照。Apache 2.0 |
| Trivy SBOM 生成モード | 次点 | Trivy にも `trivy image --format cyclonedx` で SBOM 生成機能があるが、SBOM 生成の専門ツール (Syft) に比べ検出パッケージ数が少ない場合がある |
| CycloneDX CLI | 却下 | 言語別ツール (cyclonedx-gomod, cyclonedx-rust-cargo 等) を個別管理する必要あり。統一ツールとしての一貫性が低い |
| 手動管理 | 却下 | SBOM の手動生成は現実的でなく、CI パイプラインの自動化に組み込めない |

### 背景: なぜ今決定が必要か

[`../../../01_企画/企画書.md`](../../../01_企画/企画書.md) の 採用初期 スコープに「Syft + Grype: SBOM 生成 + 脆弱性スキャン (CI ステップ)」が記載済みであり、採用初期 の完了条件 #11 として「CI パイプラインで SBOM が生成され Harbor に保管されている」が定義されている。

本ドキュメントは、これらの計画に対する正式な選定根拠の記録である。

### Grype の役割の明確化

[`02_周辺OSS.md`](02_周辺OSS.md) の Harbor 選定 (D) において Grype は「却下」と記載されている。これは **Harbor 内蔵イメージスキャナ** としての比較であり、その文脈では Trivy が Harbor との統合深度で優れていた。

本ドキュメントでの Grype の採用は **CI パイプライン上で SBOM に対する脆弱性スキャンを実行する** 用途であり、Harbor 内蔵スキャナの代替ではない。以下のように棲み分ける。

| ツール | 用途 | 実行場所 | 導入時期 |
|---|---|---|---|
| Trivy FS | ソースコードの依存パッケージスキャン | GHA runner (CI ステップ) | 採用初期 |
| Trivy (Harbor 内蔵) | コンテナイメージの脆弱性スキャン | Harbor (push 後自動) | 採用初期 |
| **Syft** | SBOM 生成 (CycloneDX 形式) | GHA runner (CI ステップ) | 採用初期 |
| **Grype** | SBOM ベースの脆弱性スキャン | GHA runner (CI ステップ) | 採用初期 |

Trivy と Grype は異なる脆弱性データベースを参照するため、両方を実行することで検出漏れを低減する (Defense in Depth)。

### 採用理由 (Syft)

1. **SBOM 生成の専門ツール** — Syft はソフトウェア構成分析 (SCA) に特化しており、コンテナイメージ内のパッケージ (OS パッケージ / 言語依存パッケージ / バイナリ) を高精度で検出する。Trivy の SBOM 生成モードと比較して、Go バイナリ内の依存関係や Rust crate の検出精度が高い
2. **CycloneDX / SPDX 両形式をサポート** — k1s0 では CycloneDX v1.5 を標準形式とする。Syft は CycloneDX / SPDX / Syft JSON のいずれでも出力可能であり、将来の形式変更にも対応できる
3. **OCI イメージへの SBOM アタッチ** — Syft が生成した SBOM を `cosign attach sbom` で OCI Artifact として Harbor のイメージに紐付けて保管する。イメージと SBOM が同一リポジトリで一元管理される
4. **Grype との統合** — Syft JSON 形式は Grype の入力として最適化されている。`syft → grype` のパイプライン処理で、SBOM 生成と脆弱性スキャンをシームレスに連結できる
5. **CI パイプラインへの統合が容易** — `syft <image> -o cyclonedx-json` の 1 コマンドで SBOM を生成。GHA ステップに追加するだけで動作する

### 採用理由 (Grype)

1. **SBOM ベースの脆弱性スキャン** — Grype は Syft が生成した SBOM を入力として脆弱性データベースと照合する。イメージ全体を再スキャンせず、SBOM からの高速スキャンが可能
2. **Trivy と異なる脆弱性データベース** — Grype は NVD / GitHub Security Advisory / Alpine SecDB / Debian Security Tracker / Amazon Linux ALAS 等を参照する。Trivy の脆弱性データベース (NVD + Trivy DB) とカバー範囲が異なるため、両方を実行することで検出漏れを低減する
3. **SBOM から再現可能なスキャン** — Harbor に保管した SBOM を後日取得し、最新の脆弱性データベースで再スキャンできる。リリース後に発見された CVE に対して、既存イメージの影響を遡及的に評価可能
4. **出力形式の柔軟性** — JSON / Table / CycloneDX VEX 形式で脆弱性レポートを出力できる。Grafana ダッシュボードへの脆弱性メトリクス統合が将来可能
5. **オフライン脆弱性データベース更新** — `grype db update` でローカルに脆弱性データベースをキャッシュする。採用側組織のネットワーク制約下でも GHA runner Pod の初期化時にデータベースを取得し、CI 実行時はオフラインで動作する

### CI パイプラインへの統合

採用初期 の CI パイプライン全体フロー内での位置付け。

```text
PR マージ後の CI パイプライン (main ブランチ)
  ├── Lint / UT / 型チェック
  ├── Protobuf 契約検証 (buf lint + buf breaking)
  ├── Testcontainers 統合テスト
  ├── Trivy FS スキャン (ソースコード依存パッケージ)
  ├── コンテナビルド (Kaniko)
  ├── Harbor push
  ├── Harbor 内蔵 Trivy (イメージスキャン、自動)
  ├── SBOM 生成 (Syft) → Harbor に SBOM attach     ← 追加
  ├── SBOM 脆弱性スキャン (Grype)                    ← 追加
  ├── Cosign sign (採用後の運用拡大時 で追加)
  └── GitOps マニフェスト更新
```

### CI ステップの詳細

| ステップ | コマンド | 失敗時の動作 |
|---|---|---|
| SBOM 生成 | `syft harbor.k1s0.internal/<project>/<image>:<tag> -o cyclonedx-json=sbom.cdx.json` | パイプライン失敗 (SBOM なしでのデプロイは許可しない) |
| SBOM attach | `cosign attach sbom --sbom sbom.cdx.json harbor.k1s0.internal/<project>/<image>:<tag>` | 警告のみ (SBOM は CI アーティファクトにも保存) |
| 脆弱性スキャン | `grype sbom:sbom.cdx.json --fail-on critical` | `CRITICAL` 検出でパイプライン失敗。`HIGH` は警告のみ |

### SBOM 形式と保管

| 項目 | 方針 |
|---|---|
| 標準形式 | CycloneDX v1.5 (JSON) |
| 保管場所 (一次) | Harbor (OCI Artifact として `cosign attach sbom` でイメージに紐付け) |
| 保管場所 (二次) | MinIO (`sbom/` バケット、CI アーティファクトとしてもアップロード) |
| 保持期間 | イメージの保持期間と同一 (段階依存: 90 日〜1 年) |
| 生成タイミング | main ブランチへのマージ後の CI パイプライン (イメージビルドのたび) |

### 採用側組織における SBOM の価値

採用側組織の情報システム部門にとって、SBOM は以下の場面で価値を発揮する。

| 場面 | SBOM の効果 |
|---|---|
| CVE 緊急対応 | 新規 CVE の発表時に「影響を受けるサービス」を SBOM から即座に特定。Grype で全 SBOM を再スキャンし影響範囲を確定 |
| 申請・監査 | 「このシステムはどのような OSS を使用しているか」への回答を CycloneDX JSON で機械的に提示。手動棚卸しが不要 |
| ライセンスコンプライアンス | SBOM にはパッケージのライセンス情報が含まれる。AGPL / GPL のコンポーネント混入を自動検知 |
| ベンダー要件 | サプライチェーンセキュリティの国際規制・ガイドライン (米国大統領令 EO 14028 等) への準拠根拠 |

### OSS リリース時点での採用範囲

- **採用初期 で導入** — Harbor (採用初期) + CI パイプライン完成が前提
- GHA runner のカスタムイメージに `syft` / `grype` CLI を同梱
- CI パイプラインに SBOM 生成 (`syft`) + 脆弱性スキャン (`grype`) ステップを追加
- `grype --fail-on critical` で Critical 脆弱性検出時にパイプラインを失敗させる
- SBOM を Harbor に OCI Artifact として attach (`cosign attach sbom` は 採用後の運用拡大時 の Cosign 導入後。採用初期 では CI アーティファクトとして MinIO に保管)
- 採用後の運用拡大時 で Cosign 導入後、SBOM attach を Harbor に移行

### 採用初期 と 採用後の運用拡大時 の境界

| 機能 | 採用初期 | 採用後の運用拡大時 |
|---|---|---|
| SBOM 生成 (Syft) | CI ステップで生成 | 同左 |
| SBOM 保管 | MinIO (CI アーティファクト) | Harbor (OCI Artifact、`cosign attach sbom`) |
| 脆弱性スキャン (Grype) | CI ステップで実行、Critical ブロック | 同左 |
| CVE 遡及スキャン | 手動 (`grype sbom:<file>`) | Argo Events で新 CVE DB 更新を検知し自動再スキャン |

### トレードオフ

- Syft + Grype (Anchore 社) と Trivy (Aqua Security 社) の両方を採用するため、脆弱性データベースの更新を 2 系統管理する必要がある。ただし両ツールとも `grype db update` / `trivy --download-db-only` で自動更新が可能であり、GHA runner カスタムイメージのビルド時に両データベースを同梱する
- Grype の脆弱性データベースは NVD ベースのため、NVD 側のデータ公開遅延が検出精度に影響する。Trivy は独自の Trivy DB で NVD の遅延を補完するため、両方の実行が相互補完として機能する
- SBOM には OS パッケージ・言語依存パッケージの全一覧が含まれるため、ファイルサイズが 1〜5 MB になる場合がある。Harbor / MinIO のストレージ消費はイメージ本体に比べて無視できる水準だが、長期的には保持期間ポリシーで管理する
- [`02_周辺OSS.md`](02_周辺OSS.md) で Grype を「却下」とした判断は Harbor 内蔵スキャナの文脈であり、本ドキュメントの CI パイプライン上の SBOM スキャナとしての採用とは矛盾しない。Harbor 内蔵スキャナは引き続き Trivy を使用する
