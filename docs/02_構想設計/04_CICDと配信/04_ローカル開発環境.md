# ローカル開発環境

## 目的

tier2 / tier3 開発者および tier1 開発者が、手元の環境で高速なフィードバックループを得るための開発体験を定義する。CI パイプラインの補完であり代替ではない。

---

## 1. 解決する問題

テスト戦略 ([`03_テスト戦略.md`](./03_テスト戦略.md)) で定義されたテスト環境は以下のとおり。

| 環境 | 用途 |
|---|---|
| ローカル (開発者 PC) | ユニットテストのみ |
| GHA runner Pod | ユニットテスト + 軽量統合テスト |
| test namespace (k8s) | 統合テスト |
| staging (Phase 3 以降) | E2E テスト |

ここに **開発中の即時確認** が欠落している。

| 問題 | 影響 |
|---|---|
| tier1 API の動作確認に CI 待ちが必要 | コード変更 → push → CI 完了 (3〜10 分) のサイクルが 1 日数十回発生 |
| ローカルで tier1 サービスを再現できない | `k1s0.Log` / `k1s0.State` / `k1s0.PubSub` の動作確認がローカルで不可能 |
| 新規参加者の立ち上がりが遅い | 環境構築手順が属人化し、開発開始まで数日かかる |
| tier1 Go ↔ Rust の連携確認が遅い | gRPC 契約変更時、CI を回すまで動作確認ができない |

k1s0 の「5 つの勝ち筋」の 1 つである **言語自由度** は、開発者が快適に開発できる環境なしには達成できない。ローカル開発環境の品質は、tier2 / tier3 開発者の k1s0 に対する第一印象を左右する。

---

## 2. 候補比較

| 候補 | 採否 | 評価 |
|---|---|---|
| **Tilt** | 採用 | Apache 2.0。k8s ネイティブのローカル開発ツール。ファイル変更検知 → 差分ビルド → Pod 更新を自動化。Starlark (Python サブセット) で柔軟に定義可能 |
| Skaffold | 次点 | Google 製 (Apache 2.0)。YAML 設定のみ。Tilt より柔軟性が低いが学習コストも低い |
| DevSpace | 次点 | Apache 2.0。開発コンテナとの同期に特化。Tilt よりも開発コンテナ指向が強い |
| Telepresence | 却下 | リモートクラスタとの通信プロキシ。MVP でリモートクラスタを前提にしたくない |
| Docker Compose | 却下 | k8s のマニフェストとの二重管理が発生する。Dapr sidecar injection の再現が困難 |
| 手動 kubectl apply | 却下 | フィードバックループが長く開発効率が低い |

---

## 3. 採用理由 (Tilt)

1. **k8s マニフェストをそのまま利用** — `deploy/base/*.yaml` を二重管理せず、ローカルの k8s に直接適用。CI / CD と同じマニフェストで開発する
2. **ファイル変更検知 + 差分ビルド** — ソースファイル保存 → 自動検知 (1〜3 秒) → コンテナ差分ビルド → Pod 更新。フィードバックループが **CI 待ち (3〜10 分) → 数秒** に短縮
3. **Starlark による柔軟な定義** — tier1 依存サービス群の起動順序制御、ヘルスチェック待機、条件分岐を宣言的に記述可能
4. **Web UI によるステータス可視化** — 各サービスのビルド状態 / ログ / ヘルスを 1 画面で確認。JTC 開発者の学習曲線を緩和
5. **Apache 2.0 ライセンス** — k1s0 の OSS ライセンス方針に完全適合
6. **Docker / Podman / k8s (Docker Desktop) に対応** — 開発者のローカル環境に合わせて選択可能

## 4. k1s0 における役割

### 4.1 開発者ワークフローの変化

**Tilt 導入前:**

| ステップ | 所要時間 | 場所 |
|---|---|---|
| コード変更 | — | ローカル |
| `git push` | 数秒 | ローカル |
| CI (Lint → UT → ビルド → スキャン) | 3〜10 分 | GHA runner |
| 結果確認 | — | GitHub |
| 失敗時: 修正 → push → 再待ち | 3〜10 分 | 繰り返し |

**Tilt 導入後:**

| ステップ | 所要時間 | 場所 |
|---|---|---|
| コード変更 → ファイル保存 | — | ローカル |
| Tilt が自動検知 → 差分ビルド → Pod 更新 | 1〜5 秒 | ローカル k8s |
| ブラウザ / ターミナルで即確認 | — | ローカル |
| 確認後に push → CI | 3〜10 分 | GHA runner |

CI は品質ゲートとして残る。Tilt は CI の前段階での **探索的開発を高速化** する。

### 4.2 ローカル環境の構成

開発者の PC 上で k8s (Docker Desktop / Rancher Desktop / 直接インストール) が動作し、その中に以下が事前構成済みで立ち上がる。

| コンポーネント | 内容 | 備考 |
|---|---|---|
| tier1 Go ファサード | `k1s0.Log` / `k1s0.State` / `k1s0.PubSub` 等 | ローカル用の軽量構成 |
| Dapr Control Plane + sidecar | ローカル用 Component (in-memory) | Kafka / Valkey への外部依存なし |
| Keycloak | 開発用 Realm + テストユーザー事前投入 | 単一インスタンス |
| PostgreSQL | 単一インスタンス (ローカル用) | CloudNativePG ではなく素の PostgreSQL |
| 開発対象サービス | 開発者が作業中のサービス | Tilt が差分ビルド・更新を自動化 |

tier2 / tier3 開発者は自分のサービスのコードだけを書く。tier1 の内部実装も Dapr の存在も Apicurio Registry も一切意識しない。

### 4.3 雛形生成 CLI との統合

雛形生成 CLI ([`../02_tier1設計/02_API契約/03_API設計原則.md`](../02_tier1設計/02_API契約/03_API設計原則.md)) が生成するファイルに Tiltfile を追加する。

| 生成ファイル (既存) | 内容 |
|---|---|
| `.github/workflows/ci.yml` | GHA ワークフロー |
| `deploy/base/*.yaml` | k8s マニフェスト |
| `catalog-info.yaml` | Backstage エントリ |

| 生成ファイル (追加) | 内容 |
|---|---|
| `Tiltfile` | Tilt の設定ファイル。tier1 依存サービスの起動 + 開発対象サービスの差分ビルド定義 |
| `tilt/tier1-deps.yaml` | tier1 サービス群のローカル構成マニフェスト |
| `tilt/dapr-components-local.yaml` | ローカル用 Dapr Component (in-memory state / in-memory pubsub) |
| `tilt/keycloak-dev-realm.json` | 開発用 Keycloak Realm エクスポート (テストユーザー含む) |

**開発者は `tilt up` の 1 コマンドで開発環境が立ち上がる。**

### 4.4 tier1 チーム自身への恩恵

tier2 / tier3 開発者だけでなく、tier1 チームも Tilt の恩恵を受ける。

| 場面 | 効果 |
|---|---|
| Go ファサードの修正 | Dapr sidecar + バックエンドとの統合をローカルで即確認 |
| Rust サービスの修正 | Go ↔ Rust の gRPC 通信をローカルで即確認 |
| gRPC 契約 (`.proto`) の変更 | 両端の生成コードを差分ビルドし、統合動作を数秒で確認 |
| 新しい tier1 API の追加 | 公開 API → 内部実装の全経路をローカルでテスト |

## 5. テスト戦略との関係

Tilt はテストの代替ではなく、**テストを書く前の探索的開発** を高速化する。

| テスト環境 | 用途 | Tilt との関係 |
|---|---|---|
| ローカル + Tilt | 開発中の即時確認・探索的開発 | Tilt が担当 |
| ローカル (Tilt 不使用) | ユニットテスト (`cargo test` / `go test`) | 変化なし |
| ローカル + Testcontainers | 実バックエンドに対する統合テスト | Tilt とは独立。Docker のみで実行 |
| GHA runner Pod (DinD) | CI (ユニット + Testcontainers 統合テスト + 軽量統合テスト) | 変化なし |
| test namespace | k8s ベース統合テスト | 変化なし |
| staging | E2E テスト | 変化なし |

Tilt は **「人間が画面を見て確認する」探索的開発** を高速化する。Testcontainers は **「機械が自動で検証する」統合テスト** を高速化する。両者は補完関係にあり競合しない。品質保証は CI / CD (Testcontainers 統合テスト含む) が担う。

## 6. 段階導入計画

| フェーズ | スコープ |
|---|---|
| MVP-1a | Tiltfile の手動作成。tier1 開発者 (2 名) がローカルで tier1 Go ファサード + Dapr の統合を即時確認。雛形生成 CLI に Tiltfile 生成を追加 |
| MVP-1b | リファレンス実装の Tiltfile を整備。協力者がローカル環境を `tilt up` で再現できることを確認 (バス係数 2 の補強) |
| Phase 2 | tier2 開発者向けの Tiltfile テンプレートを整備。tier1 依存サービス群のローカル構成を標準化。Backstage Software Templates に Tiltfile 生成を統合 |
| Phase 3 | 複数 tier2 サービスの同時起動構成。ネイティブアプリ (MAUI) のローカル開発との統合検討 |

### MVP-1a での導入理由

Tilt を Phase 2 まで遅延させると、**協力者の立ち上がりが遅延する**。MVP-1a で 2 名体制に移行した際、協力者が tier1 のコードを理解し修正するためには、CI を回さずに手元で即確認できる環境が不可欠。MVP の最重要ゴールである「バス係数 2 の実証」を加速する。

## 7. メモリ概算 (ローカル環境)

| コンポーネント | 概算メモリ |
|---|---|
| k8s (Docker Desktop) | 0.5 GB |
| Tilt | 0.1 GB |
| Dapr Control Plane | 0.3 GB |
| tier1 Go ファサード | 0.2 GB |
| Keycloak (開発用) | 0.5 GB |
| PostgreSQL (開発用) | 0.3 GB |
| 開発対象サービス | 0.2 GB |
| **合計** | **約 2.1 GB** |

開発者の PC に 8 GB 以上のメモリがあれば動作可能。JTC の標準的な業務 PC で十分。

## 8. リスクと受容判断

| リスク | 影響 | 緩和策 |
|---|---|---|
| ローカル環境と本番環境の乖離 | ローカルでは動くが本番で動かない | ローカルも k8s マニフェスト (`deploy/base/*.yaml`) を使用し乖離を最小化。CI / CD の品質ゲートは変えない |
| Dapr Component の差異 | ローカルは in-memory、本番は Kafka / Valkey | Dapr の抽象化により API は同一。統合テスト (test namespace) で本番 Component を検証 |
| 開発者 PC のスペック差 | 低スペック PC でビルドが遅い | 差分ビルド (live_update) でフルビルドを回避。最低要件を TechDocs に明記 |
| Tilt のバージョン管理 | チーム内で Tilt バージョンが異なる | 雛形生成 CLI が `.tool-versions` を生成し、Tilt バージョンを固定 |
| Tilt プロジェクトの継続性 | メンテナンスが停滞する可能性 | Docker 社が買収済みで開発は継続中。代替として Skaffold への移行パスを確保 |
