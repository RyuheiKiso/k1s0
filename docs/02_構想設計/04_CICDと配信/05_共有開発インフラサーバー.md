# 共有開発インフラサーバー

## 目的

tier2 / tier3 開発者の開発マシン負荷を最小化するため、共有開発インフラサーバーを導入する。本資料の核心的な設計判断は**開発者 PC からローカル k8s を排除する**ことにある。

従来の [`04_ローカル開発環境.md`](./04_ローカル開発環境.md) では全コンポーネントを開発者 PC 上の k8s に配置する設計だったが、共有サーバーの導入により前提が根本的に変わる。

---

## 1. ローカル k8s が不要な理由

### 1.1 従来の前提とその帰結

[`04_ローカル開発環境.md`](./04_ローカル開発環境.md) では以下の前提で設計されていた。

| 前提 | 帰結 |
|---|---|
| 全コンポーネント (tier1 サービス / Dapr / Keycloak / PostgreSQL) をローカルに配置する | **ローカル k8s が必要** |
| k8s が必要 | **Docker Desktop / Rancher Desktop / k3s のいずれかが必要** |
| k8s 上でホットリロードする | **Tilt が必要** |

結果として、開発者 PC に Docker Desktop (商用ライセンス)、Rancher Desktop、または k3s のインストールが前提条件となっていた。

### 1.2 前提の見直し

共有開発インフラサーバーに重いコンポーネントを移すと、**開発者 PC に残るのは自サービスの 1 プロセスだけ**になる。k1s0 クライアントライブラリが共有サーバー上の tier1 サービスに gRPC で接続するため、ローカルにインフラコンポーネントは一切不要。

| コンポーネント | 従来の配置 | 見直し後の配置 |
|---|---|---|
| 自サービス (tier2 / tier3) | ローカル k8s 内 | **ローカル (プロセス直接実行)** |
| tier1 Go ファサード | ローカル k8s 内 | **共有サーバー** |
| tier1 Rust サービス | ローカル k8s 内 | **共有サーバー** |
| tier1 内部コンポーネント | ローカル k8s 内 | **共有サーバー** |
| Kafka | ローカル k8s 内 (in-memory 代替) | **共有サーバー (実 Kafka)** |
| Valkey | ローカル k8s 内 (in-memory 代替) | **共有サーバー (実 Valkey)** |
| Temporal | ローカル k8s 内 | **共有サーバー** |
| Keycloak | ローカル k8s 内 | **共有サーバー** |
| PostgreSQL | ローカル k8s 内 | **共有サーバー** |
| Grafana / Loki / Tempo | なし (ローカルでは省略) | **共有サーバー** |

### 1.3 1 プロセスに k8s は過剰

k8s は複数コンテナのオーケストレーション・スケーリング・自動復旧のための基盤であり、**自サービス 1 つを動かすためだけに導入する合理性はない**。

k1s0 クライアントライブラリは、共有サーバー上の tier1 サービスに gRPC で直接接続する。tier1 の内部実装 (Dapr / Kafka / Valkey 等) は全て共有サーバー側に閉じているため、**開発者 PC にインフラコンポーネントを一切配置する必要がない**。

```bash
# tier2 開発者が実行するコマンド — これだけで開発環境が起動する
k1s0 dev start
```

このコマンドで以下が起こる。

1. `k1s0 dev` CLI が共有サーバーへの接続設定を生成する
2. 自サービスがホットリロード付きで起動する (`dotnet watch run` / `air`)
3. k1s0 クライアントライブラリが共有サーバー上の tier1 サービスに gRPC で接続する
4. tier1 の内部実装は共有サーバー内で完結し、開発者 PC には一切現れない

### 1.4 排除されるツール

ローカル k8s の排除により、以下の全てが不要になる。

| 排除されるツール | 従来の役割 | 不要になる理由 |
|---|---|---|
| **Docker Desktop** | コンテナランタイム + k8s | k8s が不要。コンテナ実行は Docker CE で代替 |
| **Rancher Desktop** | Docker Desktop の OSS 代替 | 同上 |
| **k3s / kind / minikube** | 軽量 k8s ディストリビューション | k8s 自体が不要 |
| **Tilt** | k8s 上のホットリロード | k8s が不要。言語ネイティブのホットリロードで代替 |

### 1.5 残るツール

| ツール | 用途 | インストール方法 |
|---|---|---|
| **WSL2 (Ubuntu)** | 開発環境のベース | Windows 標準機能 |
| **Docker CE** | コンテナイメージのビルド、Testcontainers による統合テスト | `apt install docker-ce` |
| **言語ランタイム** | サービスの実行 | `dotnet` / `go` のインストール |
| **`k1s0` CLI** | 開発環境の管理。**開発者が操作する唯一のツール** | 単一バイナリ配布 |

Docker CE は**日常の開発ループでは起動不要**であり、コンテナイメージのビルド (push 前の確認) や Testcontainers による統合テスト実行時にのみ使用する。tier1 の内部実装に必要なコンポーネント (Dapr 等) は全て共有サーバー側に配置されるため、**開発者 PC にはインストールしない**。

### 2.2 開発者 PC の構成

開発者 PC (Windows 11) の WSL2 上で、自サービスのみを実行する。

| コンポーネント | 実行方法 | 備考 |
|---|---|---|
| 自サービス | `dotnet watch run` (C#) / `air` (Go) | ホットリロード有効。k1s0 クライアントライブラリが共有サーバーに接続 |
| Docker CE | デーモン (必要時のみ起動) | ビルド・テスト専用 |

`k1s0 dev start` コマンドが環境設定とサービス起動を一括で実行する。tier1 の内部実装コンポーネントは開発者 PC に一切配置されない。

### 2.3 共有 infra サーバーの構成

共有サーバーは k8s クラスタとして構成し、本番環境と同等のコンポーネントを配置する。

| カテゴリ | コンポーネント |
|---|---|
| データ基盤 | Kafka (Strimzi)、Valkey、PostgreSQL、MinIO |
| 認証・制御 | Keycloak、Dapr Control Plane |
| 観測性 | Grafana、Loki、Tempo、OTel Collector |
| tier1 サービス | tier1 Go ファサード、tier1 Rust サービス |

従来のローカル環境では in-memory の Dapr Component を使用していたが、共有サーバーでは**実際の Kafka / Valkey / Temporal に接続する**ため、本番環境との乖離が大幅に縮小する。

## 3. 共有サーバーへの接続の仕組み

### 3.1 k1s0 クライアントライブラリによる透過的接続

k1s0 クライアントライブラリ ([`../02_tier1設計/02_API契約/06_クライアントライブラリ配布戦略.md`](../02_tier1設計/02_API契約/06_クライアントライブラリ配布戦略.md)) は、実行環境に応じて接続先を自動で切り替える。

| 実行環境 | 接続先の解決方法 | tier2 コードの変更 |
|---|---|---|
| k8s クラスタ (本番 / ステージング) | k8s 内 DNS (`tier1-gateway.tier1.svc.cluster.local`) | **不要** |
| ローカル開発 (共有サーバーモード) | `k1s0.config.yaml` の `dev.infra_server` | **不要** |
| ローカル開発 (ローカルモード) | `localhost` (k1s0 CLI が軽量プロキシを起動) | **不要** |

tier2 開発者のコードは `k1s0.State.SaveAsync()` と書くだけであり、接続先がどこであるかを一切意識しない。

### 3.2 tier1 内部実装の配置 (tier2 には不可視)

共有サーバー上では、tier1 の内部実装コンポーネント (Go ファサード / Rust サービス / Dapr / Kafka / Valkey 等) が稼働している。これらは [`../02_tier1設計/01_設計の核/01_Dapr隠蔽方針.md`](../02_tier1設計/01_設計の核/01_Dapr隠蔽方針.md) に基づき**開発者の視界に一切現れない**。

| 観点 | 保証される事項 |
|---|---|
| 開発者 PC へのインストール | tier1 内部コンポーネントは**一切インストールしない** |
| エラーメッセージ | tier1 内部のエラーは `k1s0 dev` CLI が翻訳し、内部実装名を露出しない |
| 設定ファイル | 共有サーバーへの接続設定は `k1s0.config.yaml` の 1 行のみ。内部コンポーネントの設定は不要 |
| ログ出力 | tier1 内部のログは共有サーバー側の Grafana / Loki で確認する。開発者 PC には流れない |

## 4. ホットリロード

### 4.1 Tilt の役割の代替

従来 Tilt が担っていた「ファイル変更検知 → 自動反映」は、言語ネイティブのホットリロードツールで代替する。

| 言語 | ツール | コマンド | 反映速度 |
|---|---|---|---|
| C# | `dotnet watch` | `dotnet watch run` | 1-3 秒 |
| Go | `air` | `air` | 1-3 秒 |

これらは Tilt と同等以上のフィードバック速度を提供する。Tilt がファイル変更 → コンテナ差分ビルド → Pod 更新の 3 段階を経るのに対し、言語ネイティブツールはファイル変更 → プロセス再起動の 2 段階で済むため、むしろ高速である。

### 4.2 `k1s0 dev` CLI がラップ

tier2 開発者は言語ツールを直接操作しない。`k1s0 dev start` が内部で適切なツールを選択・起動する。

```bash
# tier2 開発者が叩くのはこれだけ
k1s0 dev start

# 内部で実行されること:
# 1. 共有サーバーへの接続設定を環境変数としてサービスプロセスに注入
# 2. dotnet watch run / air でサービスを起動 (ホットリロード有効)
# 3. k1s0 クライアントライブラリが共有サーバーの tier1 サービスに gRPC で接続
```

## 5. 開発者 PC の要件

### 5.1 メモリ消費の比較

| 構成 | メモリ概算 | 備考 |
|---|---|---|
| 従来 (全ローカル: Docker Desktop + k8s + 全コンポーネント) | 約 2.1 GB | [`04_ローカル開発環境.md`](./04_ローカル開発環境.md) §7 |
| **本設計 (共有サーバー + ローカルに自サービスのみ)** | **約 0.2 GB** | 自サービスのプロセスのみ |

従来比で **約 90% のメモリ削減**。8 GB メモリの業務 PC でも余裕をもって開発可能。

### 5.2 ソフトウェア要件

| ソフトウェア | バージョン | 備考 |
|---|---|---|
| Windows 11 | — | WSL2 が有効であること |
| WSL2 (Ubuntu) | 22.04 以降 | 標準の Ubuntu ディストリビューション |
| Docker CE | 最新安定版 | ビルド・テスト時のみ使用 |
| .NET SDK | 8.0 以降 (C# 開発時) | `dotnet watch` を含む |
| Go | 1.22 以降 (Go 開発時) | `air` は `go install` で導入 |
| `k1s0` CLI | 最新版 | 単一バイナリ。環境管理・接続設定を担当 |

### 5.3 WSL2 ファイルシステムの注意

ソースコードは **WSL2 のネイティブファイルシステム上に配置する**こと。

| 配置場所 | ファイル I/O 速度 | ホットリロードの反応 |
|---|---|---|
| `/mnt/c/work/...` (Windows 側) | 遅い (9p プロトコル経由) | ファイル変更検知が 2-5 秒遅延する |
| **`~/work/...` (WSL2 ネイティブ)** | **速い (ext4 直接)** | **1 秒以内に検知** |

Windows 側のエディタ (VS Code) からは Remote - WSL 拡張で透過的にアクセスできるため、開発体験に影響はない。

## 6. `k1s0 dev` CLI への影響

### 6.1 コマンド体系の維持

[`../02_tier1設計/03_開発者体験/11_ローカル開発CLI設計.md`](../02_tier1設計/03_開発者体験/11_ローカル開発CLI設計.md) で定義されたコマンド体系は変更しない。内部実装のみが変わる。

| コマンド | 動作 | 従来の内部実装 | 本設計の内部実装 |
|---|---|---|---|
| `k1s0 dev start` | 開発環境を起動 | `tilt up` | 接続設定の注入 → `dotnet watch` / `air` |
| `k1s0 dev stop` | 開発環境を停止 | `tilt down` | サービスプロセス停止 |
| `k1s0 dev status` | 全体の状態表示 | `kubectl get pods` の翻訳 | 共有サーバーの Diag API + ローカルプロセスの状態確認 |
| `k1s0 dev logs <service>` | ログ表示 | `kubectl logs` | ローカルプロセスの stdout |
| `k1s0 dev restart <service>` | サービス再起動 | `kubectl rollout restart` | プロセス再起動 |
| `k1s0 dev health` | tier1 の健全性確認 | `k1s0.Diag.HealthCheckAsync()` | 共有サーバーの Diag API 呼び出し |

### 6.2 接続先の指定

共有サーバーのアドレスは `k1s0.config.yaml` または環境変数で指定する。

```yaml
# k1s0.config.yaml
service:
  name: my-service
  port: 8080

dev:
  infra_server: dev-server.internal
```

```bash
# または環境変数
export K1S0_INFRA_SERVER=dev-server.internal
k1s0 dev start
```

### 6.3 エラー翻訳の継続

[`../02_tier1設計/03_開発者体験/11_ローカル開発CLI設計.md`](../02_tier1設計/03_開発者体験/11_ローカル開発CLI設計.md) §4 で定義されたエラー自動翻訳は引き続き機能する。k8s 固有のエラー (CrashLoopBackOff 等) はローカルで発生しなくなるが、共有サーバーとの接続エラーを翻訳する。tier1 内部コンポーネント名 (Dapr / Kafka / Valkey 等) はエラーメッセージに**一切露出させない**。

| エラー (内部) | 翻訳後のメッセージ (tier2 開発者に表示) |
|---|---|
| gRPC 接続拒否 | `tier1 基盤サービスに接続できません。共有サーバーの状態を確認してください` |
| 共有サーバーへの接続タイムアウト | `開発サーバーに接続できません。ネットワークを確認してください` |
| tier1 内部の障害 | `tier1 基盤に問題が発生しています (あなたのコードの問題ではありません)` |

## 7. k8s マニフェストとの乖離対策

ローカルで k8s を使用しないことにより、k8s マニフェストの検証がローカルでは行えない。これは以下の方法で対処する。

| 懸念 | 対処 | 検証場所 |
|---|---|---|
| マニフェストの構文エラー | CI の YAML lint で検出 | GHA runner |
| 環境変数・ConfigMap の不一致 | `k1s0 dev` CLI が `k1s0.config.yaml` → 環境変数に変換してプロセスに渡す | ローカル |
| tier1 内部コンポーネントの差異 | 共有サーバーで実バックエンドに接続するため差異なし | 共有サーバー |
| ネットワークポリシー | CI の test namespace で検証 | GHA runner / test namespace |
| Kyverno ポリシー違反 | CI のドライラン検証で検出 | GHA runner |
| リソース制限 (CPU / Memory limits) | CI の test namespace でデプロイして検証 | test namespace |

本番環境との乖離で最も影響が大きいのは「in-memory 代替」による振る舞いの差異だった。本設計では共有サーバーの実バックエンドに接続するため、**従来の全ローカル構成よりも本番との乖離が小さい**。

## 8. 開発者間の分離

### 8.1 問題

複数の tier2 開発者が同一の共有サーバーに接続するため、以下の干渉が発生しうる。

| 干渉 | 例 |
|---|---|
| Kafka トピックの衝突 | 同じ `order-events` トピックに複数開発者が publish / subscribe |
| Valkey キーの衝突 | 同じ `orders:12345` キーに複数開発者が書き込み |
| Temporal ワークフロー ID の衝突 | 同じ `order.create` ワークフローを複数開発者が開始 |

### 8.2 対策: 開発者プレフィックスによる自動分離

`k1s0 dev start` の初回実行時に開発者識別子を設定し、全ての操作にプレフィックスを自動付与する。

```bash
# 初回のみ
k1s0 dev register --name tanaka

# 以降、全ての操作に自動プレフィックス
# Kafka トピック: tanaka--order-events
# Valkey キー: tanaka||orders:12345
# Temporal ワークフロー: tanaka--order.create
```

tier1 内部の設定 (キープレフィックス / トピックプレフィックス) により、tier2 開発者のコードに一切変更なく分離を実現する。分離の仕組みは tier1 内部実装であり、tier2 開発者が意識する必要はない。

## 9. 障害時のフォールバック

### 9.1 共有サーバー停止時

共有サーバーが停止した場合でも、開発を完全に止めない仕組みを用意する。

| 状況 | 影響 | フォールバック |
|---|---|---|
| 共有サーバー全体が停止 | tier1 API 呼び出しが全て失敗 | `k1s0 dev start --local` で in-memory モードに切替 |
| Kafka のみ停止 | PubSub 操作が失敗、他は正常 | PubSub のみ in-memory に自動フォールバック |
| ネットワーク一時切断 | 全接続がタイムアウト | k1s0 クライアントライブラリのリトライポリシーで自動復旧を試行 |

`--local` モードは従来の [`04_ローカル開発環境.md`](./04_ローカル開発環境.md) の in-memory 構成に相当する。`k1s0` CLI が軽量な tier1 プロキシをローカルで起動し、in-memory でデータを処理する。k8s は依然不要。

### 9.2 共有サーバーの可用性目標

| 指標 | 目標 |
|---|---|
| 可用性 (業務時間帯) | 95% |
| 計画停止の事前通知 | 1 営業日前 |
| 障害復旧時間 | 2 時間以内 |

開発用サーバーであるため本番 (99%) より低い目標とする。停止時は `--local` フォールバックで開発を継続する。

| 段階 | 施策 | 備考 |

| 採用初期 | 共有サーバー構築 (tier1 サービス + infra コンポーネント一式) | 採用初期 の VM 3 台のうち 1 台を使用 |

| 採用初期 | 開発者プレフィックスによる分離機能 | 複数開発者の同時利用に必要 |
| 採用後の運用拡大時 | `--local` フォールバックモードの実装 | 共有サーバー障害時のリスク軽減 |
| 採用側のマルチクラスタ移行時 | 共有サーバーのスペック拡張 | チーム拡大に合わせる |

## 12. [`04_ローカル開発環境.md`](./04_ローカル開発環境.md) との関係

本設計は [`04_ローカル開発環境.md`](./04_ローカル開発環境.md) の全ローカル構成を**置き換えるのではなく拡張する**。

| モード | 用途 | 使用する構成 |
|---|---|---|
| 共有サーバーモード (推奨) | 日常の tier2 / tier3 開発 | 本資料の構成 |
| ローカルモード (フォールバック) | 共有サーバー障害時、オフライン時 | [`04_ローカル開発環境.md`](./04_ローカル開発環境.md) の in-memory 構成 (k8s 不要。`k1s0` CLI が軽量プロキシを起動) |

`k1s0 dev start` のデフォルトは共有サーバーモード。`k1s0 dev start --local` でローカルモードに切り替える。
