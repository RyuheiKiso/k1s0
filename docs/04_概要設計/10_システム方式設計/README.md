# 10. システム方式設計

本カテゴリは IPA 共通フレーム 2013 の **システム方式設計プロセス（6.4.3）** に厳密対応する。k1s0 のシステム要件をハードウェア・ソフトウェア・手作業に分解し、全体方式を確立する層である。

## 本カテゴリの位置付け

要件定義書で採番された FR / NFR / BR のうち、システム全体の物理構造・論理構造・処理形態に関わるものをここで方式化する。tier1 ソフトウェアの内部構造は次章 [../20_ソフトウェア方式設計/](../20_ソフトウェア方式設計/) の担当であり、本章は tier1 を含むシステム全体のトポロジ・冗長化・処理方式・データ全体方式を扱う。

構想設計書（[../../02_構想設計/01_アーキテクチャ/](../../02_構想設計/01_アーキテクチャ/)）が 4 レイヤ構成・クラスタトポロジ・配置方式の**根拠と選定**を既に確立しているため、本章はその選定結果を**方式設計書本体**として再整理し、要件 ID との対応を明示する役割に徹する。構想設計との二重記載を避け、根拠は構想設計を参照し、本章では方式の確定事項のみを記述する。

## IPA 共通フレーム 6.4.3 との対応

共通フレーム 2013 のシステム方式設計プロセスは 6 アクティビティで構成される。本章の章立ては各アクティビティに 1:1 対応する。

- **6.4.3.1 システム要素の識別とハードウェア方式設計** → `01_ハードウェア方式設計.md`
- **6.4.3.2 ソフトウェア方式設計（全体分担）** → `02_ソフトウェア方式設計_全体分担.md`
- **6.4.3.3 システム処理方式設計** → `03_システム処理方式設計.md`
- **6.4.3.4 データベース方式設計** → `04_データベース方式設計.md`
- **6.4.3.5 システム結合テスト要件定義** → `05_システム結合テスト要件.md`
- **6.4.3.6 システム方式設計の評価・共同レビュー** → `06_システム方式設計評価レビュー.md`

## 格納予定ドキュメント

以下の順で執筆することを推奨する。

- `01_ハードウェア方式設計.md` — ノード構成、ストレージ種別、拠点配置、冗長化とフォールトトレランス方式を記述する。オンプレミス完結の JTC 前提のもと、既存データセンター資産の活用範囲も含める。
- `02_ソフトウェア方式設計_全体分担.md` — 自社開発（tier1 自作領域）、OSS 採用（Dapr / Istio / ZEN Engine など）、パッケージ利用の分担を記述する。構想設計 [../../02_構想設計/03_技術選定/](../../02_構想設計/03_技術選定/) の結果を方式として固定化する。
- `03_システム処理方式設計.md` — K8s + Istio Ambient + Dapr を軸とした処理形態を記述する。同期 gRPC / 非同期 PubSub / バッチの使い分け、エッジ IoT からの処理経路を含む。
- `04_データベース方式設計.md` — State Store（Postgres 等）、PubSub（Kafka 等）、Secret Store の全体方式と選定基準を記述する。ソフトウェア方式設計側の [../20_ソフトウェア方式設計/04_データベース最上位設計/](../20_ソフトウェア方式設計/04_データベース最上位設計/) との役割分担（本章は全体方式、次章は論理 ER・テーブル一覧）を冒頭で明示する。
- `05_システム結合テスト要件.md` — システム全体としての結合テストの要件（対象範囲、検証観点、合格基準、テスト環境要件）を定義する。テスト計画書（[../../03_要件定義/70_プロジェクト管理/04_テスト計画.md](../../03_要件定義/70_プロジェクト管理/04_テスト計画.md)）と整合させる。
- `06_システム方式設計評価レビュー.md` — 01〜05 のレビュー結果、指摘事項、是正対応、承認記録を格納する。Product Council 承認のエビデンス文書として扱う。

## 執筆方針

- 各ファイルは共通フレーム 6.4.3 の該当アクティビティ番号を冒頭に明示する。
- 方式の選定根拠は原則として構想設計の該当章を参照し、本章では再掲しない。ただし要件 ID との紐付けは本章で完結させる。
- 図は [img/](img/) に drawio と svg を配置し、md から svg を埋め込む。物理構成図・論理構成図・処理シーケンスは drawio レイヤ記法規約（[../../00_format/drawio_layer_convention.md](../../00_format/drawio_layer_convention.md)）に従う。
- 要件 ID → 設計 ID の対応は各ファイル末尾の「対応要件一覧」節で明示し、[../80_トレーサビリティ/](../80_トレーサビリティ/) に集約する。

## 他カテゴリとの関係

- **上流**: [../../03_要件定義/](../../03_要件定義/) の FR / NFR / BR を入力とする。特に NFR-A（可用性）、NFR-B（性能拡張性）、NFR-F（システム環境）は本章で方式化する主要要件。
- **下流**: 本章で確立した全体方式を前提として、[../20_ソフトウェア方式設計/](../20_ソフトウェア方式設計/) が tier1 内部構造を詳細化する。
- **横断**: [../50_非機能方式設計/](../50_非機能方式設計/) は本章と密接に連動する（可用性 DR 方式・運用方式・監視方式はシステム全体構成に依存）。重複を避けるため、本章は「全体構成としてどう並べるか」、50 章は「その構成で非機能要件をどう満たすか」の分担とする。
