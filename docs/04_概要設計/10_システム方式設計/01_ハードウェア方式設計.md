# 01. ハードウェア方式設計

本ファイルは IPA 共通フレーム 2013 の **6.4.3.1 システム要素の識別とハードウェア方式設計** に対応し、k1s0 を構成する物理・仮想ハードウェアの方式を確定する。ノード構成、ストレージ方式、ネットワーク方式、拠点配置、冗長化、フォールトトレランスを扱う。

## 本ファイルの位置付け

k1s0 は JTC 情シス部門の既存オンプレミス資産に載せるプラットフォームである。そのため「専用の高性能機材を新規調達してクラウドネイティブ前提で組む」選択肢は取れない。既存データセンターの空き VM 枠・既存 SAN ストレージ・既存 L2/L3 スイッチ・既存 DNS/NTP といった**既に稼働している資産**の上に構築する制約がある。本ファイルはこの制約を正面から受け止め、新規調達ゼロで Phase 1a〜1c を完遂できるハードウェア方式を確定する。

ハードウェア方式を曖昧にすると、性能見積もりが崩れ、運用 2 名体制で約束した工数が破綻し、企画で約束した SLA 99% が達成不能になる。したがって本ファイルでは、VM スペック・ストレージ容量・冗長化方式・障害時挙動の 4 つを具体数値で確定する。選定根拠は構想設計 [../../02_構想設計/01_アーキテクチャ/](../../02_構想設計/01_アーキテクチャ/) に譲り、本ファイルは確定結果と要件 ID 対応のみを記述する。

Phase 1a（MVP-0）は VM 1 台、Phase 1b（MVP-1a）および Phase 1c（MVP-1b）は VM 3 台、Phase 2 以降で増設・マルチ DC 化を想定する。本ファイルでは Phase 1b〜1c の 3 ノード構成を基準方式として詳細化し、Phase 1a の 1 ノード縮退構成と Phase 2 以降の拡張方針を補助的に記述する。

## ノード構成方式

Phase 1b〜1c の基準構成は **VM 3 台** とし、各 VM は **4 vCPU / 16 GB メモリ / 200 GB SSD（ローカル仮想ディスク）** を割り当てる。この諸元は要件定義書の前提条件 P-TEC-001（オンプレミス完結）および C-INF-002（既存 VM 基盤の活用）に基づき、JTC 標準の中規模 VM プロファイル（業務系 Web/AP サーバ相当）にそのまま載る範囲で選定している。

3 ノードの役割は **control-plane 兼 worker の混在型** とする。これは kubeadm のマルチマスタ HA 構成では etcd クォーラム成立のため最低 3 台が必要であり、同時に tier1 / tier2 / tier3 の Pod を動かす worker としても機能させて VM リソースを無駄にしない設計とするためである。専用 control-plane ノードを設ける案は VM 数が 5 台以上になるため Phase 1b の VM 3 台制約を逸脱する。MVP 期のワークロード規模（tier1 11 API × 各 2 レプリカ + 基盤 OSS）では役割混在でも CPU・メモリ共に余裕があり、Phase 2 で worker 専用ノードを追加する際に役割を分離する前提とする。

各ノード `k1s0-node-01` / `k1s0-node-02` / `k1s0-node-03` に割り当てる代表的な常駐 Pod の内訳は、tier1 の 11 API サービス（Go ファサード + Rust 自作領域）× 2 レプリカ、データ層（Valkey 3 / PostgreSQL 3 / Kafka 3 / OpenBao 3）、観測層（Prometheus / Loki / Tempo / Mimir / Grafana）、CI/CD 層（Argo CD / Harbor）、制御層（kube-apiserver / etcd / scheduler / controller-manager / kubelet / containerd）である。合計で 1 ノードあたり約 25〜35 Pod が常駐する見積もりで、4 vCPU のうち 1 vCPU を制御プレーンに、残り 3 vCPU をワークロードに配分する。

**設計項目 DS-SYS-HW-001 ノード基準諸元（Phase 1b〜1c）**

- 台数: 3 台（物理 VM、既存 VMware / Hyper-V / KVM いずれの仮想化基盤でも可。VM 仕様が満たせること）
- CPU: 4 vCPU / ノード（物理 CPU は Intel Xeon Silver 4210 相当以上、x86_64 アーキテクチャ）
- メモリ: 16 GB / ノード（ECC メモリ前提、ノンECC は禁止）
- ディスク: 200 GB SSD / ノード（Tier 1 SAN からの仮想ディスク、SATA SSD 以上。HDD は禁止）
- ネットワーク: 1 Gbps NIC × 2（うち 1 本は LACP 結合用、残りは管理用）
- 役割: control-plane 兼 worker（混在）

**設計項目 DS-SYS-HW-002 Phase 1a 縮退構成（MVP-0）**

Phase 1a では起案者 1 名によるデモを目的とするため、VM 3 台を確保せず **VM 1 台で single-node Kubernetes** を起動する縮退構成を取る。同一スペック（4 vCPU / 16 GB / 200 GB SSD）で kubeadm を `--skip-phases=addon/kube-proxy` なしで起動し、`kubectl taint nodes --all node-role.kubernetes.io/control-plane-` で worker 動作も許容する。HA は成立しないが、企画時点の「動くデモ」を 3〜4 週間で見せることが目的のため許容する。

**設計項目 DS-SYS-HW-003 Phase 2 拡張構成**

Phase 2 以降では worker 専用ノード 3 台を追加して計 6 台構成とし、control-plane を 3 ノードに集約、残り 3 ノードを worker 専用に配分する。Istio Ambient の waypoint proxy、Temporal、Backstage 等の追加コンポーネントを受け入れる容量を確保する。追加ノードの諸元は DS-SYS-HW-001 と同一とし、既存 VM プロファイルを流用することで調達リードタイムを 1 週間以内に抑える。

## ストレージ方式

k1s0 のストレージ要件は **ブロックストレージ**（Pod の PVC として永続化する State / Queue / WAL）と **オブジェクトストレージ**（バックアップ / ログアーカイブ / コンテナイメージ / 監査証跡のアーカイブ）の 2 種類に分解される。両者を単一の OSS で賄うと性能特性が噛み合わないため、ブロックは **Longhorn**、オブジェクトは **MinIO** を採用する。両者ともセルフマネージド OSS のみで構成し、クラウドマネージド依存を排除する。

Longhorn は各ノードのローカル 200 GB SSD を集約し、レプリカ 3 のブロックボリュームを提供する。レプリカ 3 により単一ノード障害では Read/Write 共に継続可能であり、PostgreSQL の WAL や Valkey の RDB/AOF のように失ってはならないデータを保持する用途に適合する。MinIO は同じローカルディスク上に Erasure Coding（EC 4+2 または 3+2）で構成し、1 ノード欠損でも復元可能な冗長度を確保する。

**設計項目 DS-SYS-HW-004 Longhorn ブロックストレージ方式**

- 容量配分: 各ノード 200 GB のうち OS ルート 40 GB / containerd イメージキャッシュ 20 GB / kubelet 作業領域 20 GB / Longhorn プール 120 GB とし、3 ノードで Longhorn プール合計 360 GB を確保する。
- レプリカ方針: 重要データ（PostgreSQL WAL / Valkey AOF / Kafka ログ / OpenBao Raft）は replica=3、それ以外（一時 PVC / 開発 Pod）は replica=2。
- 実効容量: replica=3 の場合、360 GB / 3 = 120 GB が実効容量。MVP 期のデータ総量見積もり（Valkey 5 GB + Kafka 30 GB + PostgreSQL 20 GB + OpenBao 1 GB + Mimir/Loki/Tempo 40 GB = 96 GB）に対して約 20% の余裕を確保する。
- スナップショット: 1 日 1 回 02:00 JST に自動スナップショット取得、7 日保持。スナップショット先は Longhorn 内部プール（オブジェクトストアへの転送は DS-SYS-HW-005 の MinIO 経由）。
- RPO: 24 時間（日次スナップショット）、Kafka および PostgreSQL は barman cloud および MirrorMaker2 により秒オーダーを別途確保（本章 [04_データベース方式設計.md](04_データベース方式設計.md) 参照）。

**設計項目 DS-SYS-HW-005 MinIO オブジェクトストレージ方式**

- 容量配分: Longhorn と論理分離するため、各ノードの 200 GB SSD とは別に、既存社内 NAS の CIFS/NFS マウント 500 GB × 3 ノード分（合計 1.5 TB）を仮想ディスクとして追加割当する。
- 配置: 3 ノードに MinIO StatefulSet をデプロイし、EC 3+2（データ 3 ドライブ + パリティ 2 ドライブ）相当の分散配置とする。1 ノード欠損でも Read/Write 可能。
- 用途: Longhorn スナップショット退避 / barman cloud（PostgreSQL バックアップ） / Kafka ログアーカイブ / Harbor コンテナイメージ / Loki チャンク / Mimir ブロック / Tempo トレースブロック / 監査証跡 WORM アーカイブ。
- 暗号化: サーバサイド暗号化 SSE-S3 有効化、鍵は OpenBao Transit エンジンから払い出し（個別の方式は [../30_共通機能方式設計/08_暗号化と鍵管理方式.md](../30_共通機能方式設計/08_暗号化と鍵管理方式.md) 参照）。
- 冗長度: 1 ノード欠損継続（EC パリティ 2 のうち 1 消費）、2 ノード欠損時は Read Only に降格。

**設計項目 DS-SYS-HW-006 Phase 1a の縮退ストレージ構成**

Phase 1a では VM 1 台のため Longhorn のレプリカ 3 は成立しない。`local-path-provisioner` によるローカル PVC のみを使用し、MinIO も standalone モード（EC なし）で起動する。データ保護は対象外（デモ用途のみ）、復旧時は再デプロイで対応する。

## ネットワーク方式

k1s0 は既存社内 LAN / VLAN / DNS / NTP を活用し、新規のネットワーク機器調達を行わない。既存 JTC データセンターの業務系 VLAN（典型的には VLAN ID 100 番台、10.0.0.0/16 等のプライベート IP 空間）に VM 3 台を収容し、既存のフロア間 L3 スイッチ経由で他のオンプレシステムおよび情シス管理拠点と接続する。

クラスタ内部の Pod ネットワークは **Calico**（VXLAN モード）を採用し、Pod CIDR は `10.244.0.0/16`、Service CIDR は `10.96.0.0/12` を割り当てる。これは kubeadm のデフォルトで kubeadm 利用者が最も多い構成であり、トラブルシューティングの情報量が最大となる。MetalLB は L2 モードで VIP を既存社内 LAN から 3 アドレス払い出し、Envoy Gateway の受け口・Keycloak の受け口・Grafana の受け口に割り当てる。

**設計項目 DS-SYS-HW-007 ネットワークセグメント設計**

- VLAN: 業務系 VLAN 1 本に 3 ノード収容（既存 VLAN を再利用）
- Pod CIDR: 10.244.0.0/16（Calico VXLAN）
- Service CIDR: 10.96.0.0/12（kubeadm デフォルト）
- MetalLB VIP プール: 既存社内 LAN から 3 アドレス払い出し（業務系 VLAN の末尾 3 アドレス）
- DNS: 既存社内 DNS に `*.k1s0.internal.example.jp` の A/CNAME を追加（BIND / AD DNS いずれでも可）
- NTP: 既存社内 NTP サーバ（time.internal.example.jp）を chrony で参照

**設計項目 DS-SYS-HW-008 ロードバランサ方式**

Kubernetes API サーバ（6443 port）への接続は **HAProxy + keepalived** の VIP を使う。VM 3 台それぞれに HAProxy を常駐させ、keepalived で VRRP ベースのアクティブ・スタンバイ VIP を構成する。Kubernetes クライアント（kubectl / Argo CD / 各種コントローラ）は VIP に対して接続し、HAProxy がバックエンド 3 ノードの kube-apiserver に分散する。HAProxy 自体の障害時は keepalived が VIP を別ノードに即座に移し、切替時間は目標 5 秒以内とする。

Envoy Gateway（tier1 の受け口）および Keycloak（OIDC プロバイダ）には MetalLB 経由で別の VIP を割り当て、kube-proxy および iptables NAT を通じて各 Pod に転送する。これにより k1s0 クライアントからの HTTPS 接続は MetalLB VIP（443）→ Envoy Gateway Service → Envoy Gateway Pod → tier1 サービスの経路で成立する。

## 冗長化とフォールトトレランス方式

k1s0 の冗長化は、**ノード障害 / ディスク障害 / ネットワーク障害** の 3 種類を想定し、それぞれに対する縮退動作を定義する。単一ノード障害ではサービス継続、2 ノード同時障害では Read Only 縮退、3 ノード障害は設計対象外（Phase 2 のマルチ DC で対応）とする。

ノード障害に対しては、Kubernetes の etcd クォーラム（3 ノードで 2 ノード以上必要）、Longhorn レプリカ 3（2 ノード以上必要）、Kafka KRaft コントローラ（3 ノードで過半数必要）、OpenBao Raft（3 ノードで過半数必要）のすべてが「1 ノード欠損で継続、2 ノード欠損で Read Only または停止」という同じ冗長度を持つ。これにより MVP 期の SLO 99.9%（月 43 分）に対して、1 ノード障害の復旧時間（kubeadm join で再参加するまで約 30 分を目標）が十分収まる。

**設計項目 DS-SYS-HW-009 ノード障害時の挙動**

- 1 ノード障害: tier1 API / 全データ層 / 観測層の全てが継続。Pod は kube-controller-manager により残存 2 ノードに再スケジュールされ、所要時間 5〜10 分。Envoy Gateway の VIP は MetalLB が即座に残存ノードに切替（目標 5 秒以内）。
- 2 ノード障害: etcd クォーラム喪失のため kube-apiserver が Read Only 扱い、新規 Pod 作成不可。既存 Pod は継続動作するが、Pod 再起動も不可。OpenBao は Read Only（Secret 参照のみ可）、Kafka は Producer 停止（Consumer のみ可）。
- 3 ノード障害: 全停止。復旧手順は Runbook（詳細は [../55_運用ライフサイクル方式設計/02_インシデント対応方式.md](../55_運用ライフサイクル方式設計/02_インシデント対応方式.md) 参照）に基づき、MinIO 上のバックアップから再構築。目標 RTO 4 時間。

**設計項目 DS-SYS-HW-010 ディスク障害時の挙動**

- 仮想ディスク 1 本障害（Longhorn レプリカ 1 喪失）: Longhorn が自動で残存 2 レプリカから読み書き継続、バックグラウンドで欠損レプリカを再構築（所要時間 30 分〜1 時間、データ量に依存）。
- 仮想ディスク 2 本障害: Longhorn レプリカ 2 喪失、残り 1 レプリカで Read Only 継続。書き込みは拒否。MinIO 上のバックアップから再構築。
- MinIO ディスク 1 本障害（EC 3+2 のパリティ 1 消費）: Read/Write 継続、バックグラウンドで Erasure Coding による自動修復。

**設計項目 DS-SYS-HW-011 ネットワーク障害時の挙動**

- 単一 NIC 障害: LACP 結合により片系継続。パケットロス 0%、スループット半減（1 Gbps → 500 Mbps）。
- ノード間ネットワーク分断（スプリットブレイン疑似）: Calico VXLAN の heartbeat により 30 秒以内に分断を検知、kube-apiserver がクォーラム側を優先。分断されたノードは NotReady となり、Pod は他ノードに再スケジュール。
- 上位 L3 スイッチ障害: VIP が到達不能となり外部アクセス停止。既存 JTC データセンターの L3 冗長化（VRRP / HSRP）に依存。

**設計項目 DS-SYS-HW-012 フォールトトレランス総合**

Phase 1b〜1c の 3 ノード構成は「single-AZ single-DC」のため、地震・電源障害・ラック全損といったデータセンター全体の障害は本設計の対象外とする。これらは Phase 2 以降のマルチ DC 構成で対応し、Phase 1 では MinIO に保存されたバックアップを**地理的に離れた JTC 別拠点の NAS に週次で複製**することで、最低限の DR を担保する（RPO 1 週間、RTO 48 時間）。バックアップ複製の具体手順は [../50_非機能方式設計/01_可用性_DR方式.md](../50_非機能方式設計/01_可用性_DR方式.md) で詳細化する。

## 拠点配置方式

Phase 1a〜1c は **単一データセンター単一ラック** に 3 ノードを配置する。JTC の既存業務系データセンターに 1U × 3 ラック相当の空き枠が確保されていることを前提とし、新規データセンター契約は行わない。ラック内の電源系統・冷却系統の冗長は JTC データセンター標準（A 系 B 系の 2 電源、HVAC 冗長空調）を活用し、k1s0 固有の物理冗長化は設計しない。

Phase 2 以降では第 2 データセンターへの拡張を想定し、MirrorMaker2 による Kafka の DC 間レプリケーション、MinIO の site-to-site レプリケーション、PostgreSQL Streaming Replication の遠隔 Standby 構築、を設計する。詳細は構想設計 [../../02_構想設計/01_アーキテクチャ/](../../02_構想設計/01_アーキテクチャ/) および [../50_非機能方式設計/01_可用性_DR方式.md](../50_非機能方式設計/01_可用性_DR方式.md) に委ねる。

**設計項目 DS-SYS-HW-013 Phase 1 の拠点配置**

- 単一 DC / 単一ラック / 3 VM（異なる物理ホスト上に配置、VMware Anti-Affinity 設定必須）
- 電源: A 系 B 系の 2 電源（ラック単位で JTC データセンターが提供）
- 冷却: HVAC 冗長空調（JTC データセンター標準）
- 物理セキュリティ: JTC データセンター標準（入退管理、監視カメラ、ラック施錠）

**設計項目 DS-SYS-HW-014 Phase 2 の拠点配置骨子**

- 第 2 DC を同一地域内に確保（距離 50km 以内、RTT 5ms 以内を目標）
- 第 2 DC に VM 3 ノード追加（計 6 ノード、2 DC 各 3 ノード）
- 第 1 DC 全損時、第 2 DC 単独で読み書き継続（RPO 60 秒、RTO 30 分を目標）

## 既存資産活用の範囲

JTC 情シス部門の既存資産を最大限活用することで、新規調達リードタイム（典型 3〜6 か月）を回避し、Phase 0 稟議通過から Phase 1a 着手までを 1 週間以内に収める。活用する資産は VM 基盤・SAN ストレージ・社内 NAS・L2/L3 スイッチ・DNS・NTP・証明書発行 CA・Active Directory・既存バックアップテープ装置の 9 項目である。

新規に導入するのは k1s0 固有の OSS（Kubernetes / Dapr / Istio Ambient / Kafka / PostgreSQL / Valkey / MinIO / Longhorn / OpenBao / Keycloak / Argo CD / Harbor / Grafana LGTM / ZEN Engine / flagd / Temporal / Backstage）のみとし、ハードウェアの新規調達は Phase 2 以降に持ち越す。

**設計項目 DS-SYS-HW-015 既存資産活用一覧**

- VM 基盤: VMware vSphere もしくは Hyper-V（どちらでも可、ゲスト OS は Ubuntu 22.04 LTS 固定）
- SAN ストレージ: ノード毎 200 GB 仮想ディスク（Tier 1 SSD プール）
- 社内 NAS: MinIO バックエンド用に 500 GB × 3 = 1.5 TB
- スイッチ: 既存 L2/L3 スイッチ（1 Gbps ポート × 2 × 3 ノード分）
- DNS: 既存社内 DNS（BIND or AD DNS）
- NTP: 既存社内 NTP
- CA: 既存社内 PKI（Subordinate CA から k1s0 クラスタ証明書を発行）
- Active Directory: Keycloak の LDAP フェデレーション元として利用
- バックアップ: 既存バックアップテープ装置（MinIO バックアップの長期保存先として月次利用）

## 物理構成の俯瞰

Phase 1b〜1c の物理構成を一言で表すと、「既存 JTC データセンターに 4 vCPU / 16 GB / 200 GB SSD の VM を 3 台払い出し、kubeadm HA で Kubernetes を起動し、全コンポーネントをその 3 ノードに封じ込める」構成である。Phase 2 での拡張余地（VM 増設・DC 増設・外部 LB 導入）を確保しつつ、Phase 1 では VM 3 台のみで 11 API / データ層 / 観測 / CI/CD / 認証 のすべてを成立させる密度設計となる。

この密度を支えるのは、Kubernetes の Pod Density（1 ノードあたり 110 Pod が上限のうち実使用 25〜35 Pod）、Longhorn のレプリカ 3 分散（単一ノード障害で継続）、Istio Ambient の sidecar レス設計（Pod あたりメモリ 50〜100 MB 節約）、である。構想設計 ADR-0001（Istio Ambient 採用）・ADR-STOR-001（Longhorn 採用）・ADR-STOR-002（MinIO 採用）の方針がここに結実する。

## Phase 2 中規模サイジング俯瞰

Phase 1b〜1c の VM 3 台構成は MVP 期の最小構成であり、3,000 名規模の本番稼働には容量・冗長度・ノード種別の分離が不足する。Phase 2 では利用者 3,000 名・tier2 ドメインサービス 10 前後・常時 Pod 数 400 程度を見込むため、ノード種別を **Control Plane / Worker / Data Plane / Observability / DR** の 5 種類に分離し、合計 16 ノードを配置する。役割ごとに CPU/RAM/ディスクの配分を変えることで、データ層の IOPS と観測層のバッファメモリを互いに干渉させない構成を取る。

下図は中規模 3,000 名を前提にした各ノード種別の諸元・配置コンポーネント・小計を一枚にまとめたものである。Phase 1b〜1c の混在型（control-plane 兼 worker）を 3 種類のノード（CP / Worker / Data）に分離する点、AGPL-3.0 OSS（Grafana LGTM / MinIO / Renovate）を Observability 専用ノード 2 台に隔離する点、DR 用の副拠点 2 ノードを本番ラックから分離する点、の 3 点が Phase 1 との主な差分である。ノード合計 116 vCPU / 560 GB RAM / 約 11 TB SSD は、JTC 標準 VM プロファイル（4〜8 vCPU の既存枠）に綺麗に載るよう設計されており、新規 HW 調達ゼロで Phase 2 に到達できる。

図の下段には、1,000 名（小規模 = Phase 1b/1c 相当）・3,000 名（中規模 = 本図の基準）・10,000 名（大規模 = Phase 3 想定）の 3 段階スケーリング差分も併記した。小規模では 3 ノード混在構成で 12 vCPU まで縮退、大規模では Worker を 6 → 16 へ線形拡張し、2 DC Active-Active と MinIO 3 リージョン複製を実稼働させる。どの規模でも同じソフトウェア構成（kubeadm + Dapr + 自作 Rust 6 コンポーネント）のまま、ノード数とレプリカ係数のみを変える設計としており、規模変更時のアプリ改修をゼロに抑える。

![HW サイジング俯瞰（中規模 3,000 名 / 16 ノード）](img/HWサイジング俯瞰.svg)

**設計項目 DS-SYS-HW-016 Phase 2 中規模サイジング（3,000 名 / 16 ノード）**

- Control Plane × 3: 4 vCPU / 16 GB / 200 GB SSD ＝ 小計 12 vCPU / 48 GB / 600 GB
- Worker × 6: 8 vCPU / 32 GB / 400 GB SSD ＝ 小計 48 vCPU / 192 GB / 2.4 TB
- Data Plane × 3: 8 vCPU / 64 GB / 1 TB NVMe + 500 GB（MinIO）＝ 小計 24 vCPU / 192 GB / 3 TB + 1.5 TB
- Observability × 2: 8 vCPU / 32 GB / 500 GB SSD ＝ 小計 16 vCPU / 64 GB / 1 TB
- DR × 2: 8 vCPU / 32 GB / 2 TB SSD（遠隔副拠点）＝ 小計 16 vCPU / 64 GB / 4 TB
- 合計: **116 vCPU / 560 GB RAM / 約 11 TB SSD**、NIC 32 本（16 × LACP 2）、MetalLB VIP 5 本 + keepalived VIP 1 本

DR 2 ノードは Phase 1b/1c では未確保で、週次のオブジェクト複製で RPO 1 週間 / RTO 48 時間を受容する。Phase 2 移行時に遠隔副拠点へ PostgreSQL Streaming Replication / MirrorMaker2 / MinIO site-to-site を配置して、RPO 60 秒 / RTO 30 分まで短縮する。詳細は [../50_非機能方式設計/01_可用性_DR方式.md](../50_非機能方式設計/01_可用性_DR方式.md) で確定する。
