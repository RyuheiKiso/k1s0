# 06. システム方式設計評価レビュー

本ファイルは IPA 共通フレーム 2013 の **6.4.3.6 システム方式設計の評価・共同レビュー** に対応する。本章（10 システム方式設計）の 01〜05 の各ファイルに対するレビュー観点・レビュア・判定基準・差戻し手順・承認記録の枠組みを定義する。

## 本ファイルの位置付け

評価レビューは、設計書のドラフトが「単なる起案者の思いつき」ではなく「複数の専門家が妥当性を確認した合意文書」に昇格するプロセスである。このプロセスを経ていない設計は、後工程の詳細設計・実装・運用で「なぜこうなっているのか分からない」状態になり、採用後の運用拡大時の拡張時に全てを再設計する羽目になる。k1s0 は 採用側の小規模運用を約束しているため、起案者が不在でも協力者がレビュー履歴を辿って設計意図を理解できる状態を作ることが、この章の最大の目的である。

本ファイルは k1s0 リリース時点では **実績が空欄** であり、承認観点チェックリスト・レビュア構成・判定基準・差戻し手順の「枠組み」のみを定義する。リリース時点 / リリース時点 の結合テスト合格後に実施される Product Council レビューで、各設計項目（DS-SYS-HW-001 〜 DS-SYS-IT-016）の承認記録が本ファイルに追記される想定である。

レビューは 2 段階で実施する。一次レビューは技術専門家 5 名（アーキテクト / インフラ責任者 / SRE リーダ / 法務 / Product Council 議長）による書面レビュー、二次レビューは一次レビュー指摘の是正後に開催する合同レビュー会議（2 時間、オフラインまたはオンライン）である。2 段階を踏むことで、個別指摘を先に潰しきってから全体合意を取るフローを成立させる。

## レビュア構成

5 名のレビュア構成を固定する。各レビュアは自身の専門領域と重複する観点を主担当とし、他観点も副担当として確認する。

**設計項目 DS-SYS-REV-007 レビュア構成**

- **アーキテクト**（主担当: DS-SYS-REV-001 / DS-SYS-REV-003）: 全体設計の整合性、構想設計との一貫性、要件カバレッジ
- **インフラ責任者**（主担当: DS-SYS-REV-001 / DS-SYS-REV-002）: VM / K8s / ストレージ / ネットワーク設計、既存 採用側組織のデータセンター資産活用
- **SRE リーダ**（主担当: DS-SYS-REV-002 / DS-SYS-REV-005）: 非機能数値達成可能性、採用側の小規模運用成立性、Runbook 整備
- **法務担当**（主担当: DS-SYS-REV-004）: AGPL 隔離運用、個人情報保護法、J-SOX、電帳法
- **Product Council 議長**（主担当: DS-SYS-REV-006）: コスト、企画整合性、段階承認最終判断

採用初期 では 採用側の小規模運用体制のため、起案者が SRE リーダを兼務、協力者がインフラ責任者を兼務する想定である。アーキテクト / 法務 / Product Council 議長は 採用側組織の情シス部門内の既存職位（または CTO 直下）から指名する。

## 判定基準

**設計項目 DS-SYS-REV-008 判定基準**

3 区分で判定する。

- **承認**: 全観点チェックリストがクリアで、かつ重大な指摘が 0 件。段階承認に進む。
- **条件付き承認**: 軽微な指摘があるが、設計本体は妥当で、指摘への是正コミットメントがある。段階承認に進むが、指摘是正を次回マイルストーンの条件にする。
- **差戻し**: 重大な指摘があり、設計本体の再検討が必要。段階承認を次回に持ち越し、再レビューを実施。

軽微 / 重大の判定は以下で区分する。

- 軽微: 文言修正 / 数値の根拠追記 / ドキュメント間リンク欠落 / レビュー対象外事項への言及不足
- 重大: 要件カバレッジ漏れ / 非機能数値達成不可能 / 構想設計 ADR 逸脱 / 法務制約違反 / 運用工数超過

## 差戻し再レビュー手順

**設計項目 DS-SYS-REV-009 差戻し再レビュー手順**

1. 差戻しのレビューコメントを GitHub Pull Request に記録（本設計の Markdown は PR ベースで更新）
2. 起案者が 1 週間以内に修正コミット、PR を更新
3. 該当指摘を担当したレビュアが再確認（書面レビューのみ）
4. 全指摘クリア後、改めて Product Council 議長に承認申請
5. Product Council 議長が最終承認 / 条件付き承認 / 再差戻しを判定

差戻しが 3 回を超える場合、段階承認自体を次々回マイルストーンに持ち越し、設計方針の再検討を行う。

## k1s0 リリース時点の条件付き承認条件

採用検討では本章 01〜05 の設計骨子を提出するが、実運用での達成確認は リリース時点 / リリース時点 でしか取れないため、以下の条件付き承認とする。

**設計項目 DS-SYS-REV-010 リリース時点 条件付き承認条件**

- 条件 1: リリース時点点で DS-SYS-IT-012（リリース時点 結合テストスコープ）の全合格基準をクリアする
- 条件 2: リリース時点点で DS-SYS-IT-013（リリース時点 結合テストスコープ）の全合格基準をクリアする
- 条件 3: リリース時点点で運用 採用側の小規模運用の Runbook が整備され、協力者が単独で障害対応を 1 回以上実施できる
- 条件 4: リリース時点点で監査ハッシュチェーン検証ツールが稼働し、改ざん試行の 100% 検出が実証される
- 条件 5: 法務レビューで AGPL 隔離運用の監査証跡が確認され、年次で法務レポートが生成される状態になる

条件未達時は 採用後の運用拡大開始を保留し、リリース時点 期間を延長して条件クリアを優先する。

## 継続レビューと改訂追跡

**設計項目 DS-SYS-REV-013 継続レビュー**

本章を含む概要設計書は **Living Document** として運用する。段階承認後も、新規要件 / 構想設計 ADR 追加 / 法令改正に追従して改訂する。

- 改訂頻度: 四半期 1 回の定期レビュー + 重大変更時の随時レビュー
- 改訂履歴: 各設計 ID 本文末尾の履歴節と、本ファイルの「改訂履歴」節に二重記録
- 追跡: Git PR 単位で改訂を管理、設計 ID 単位の影響範囲を PR 本文に明記
