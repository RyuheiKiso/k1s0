# 05. システム結合テスト要件

本ファイルは IPA 共通フレーム 2013 の **6.4.3.5 システム結合テスト要件定義** に対応する。k1s0 の infra 層 + tier1 + tier2 サンプル + tier3 サンプルを上下フルで結合した際のテスト観点・合格基準・環境要件・実施計画を方式として確定する。

## 本ファイルの位置付け

結合テストで曖昧なのは「どこまでを結合として扱うか」である。tier1 単体の自動テストはソフトウェア方式設計 [../20_ソフトウェア方式設計/06_ソフトウェア結合テスト要件.md](../20_ソフトウェア方式設計/06_ソフトウェア結合テスト要件.md) が扱い、tier1 内部の Go ⇔ Rust 境界テストは同章で閉じる。本章はそれより上の抽象で、**Kubernetes + Dapr + データ層 + tier1 全 11 API + tier2 サンプル + tier3 サンプル** を全て立ち上げた環境で、システム全体として非機能要件（性能 / 可用性 / セキュリティ / 可観測性）を満たすかを検証するテスト群の要件を定義する。

結合テストの合格は、Phase 1b / Phase 1c の完了判定条件（Product Council 承認）に直結する。合格基準を曖昧にすると、「主観的に動いているが非機能 SLO を満たしているか分からない」状態で Phase 承認が行われ、Phase 2 で数値的破綻が顕在化するリスクが高い。本章では合格基準をすべて**具体数値**で記述し、CI で再実行可能な手順で固定する。

本ファイルは結合テストの **要件** を定義する。具体的なテストケース・テストスクリプト・テストデータの実装は、テスト計画書 [../../03_要件定義/70_プロジェクト管理/04_テスト計画.md](../../03_要件定義/70_プロジェクト管理/04_テスト計画.md) および [../70_開発者体験方式設計/05_テスト戦略方式.md](../70_開発者体験方式設計/05_テスト戦略方式.md) の範囲で、本章と整合させて実装する。

## 結合テストの対象範囲

結合テストの対象範囲を、対象に含むものと含まないものの両方で明示する。範囲を事前に固定することで、テスト工数の見積もりと合格判定の客観性を確保する。

**設計項目 DS-SYS-IT-001 対象範囲**

対象に含むもの:

- infra 層: Kubernetes / Dapr control-plane / Envoy Gateway / Longhorn / MinIO / MetalLB / Calico
- tier1 層: 11 API すべての自社開発部分 + 内部 OSS（Valkey / Kafka / PostgreSQL / OpenBao / ZEN Engine / flagd）
- tier2 層: サンプルマイクロサービス 2 種（`sample-order-service` / `sample-notification-service`）
- tier3 層: サンプルエンドユーザアプリ 1 種（`sample-customer-portal`）
- operation 層: Argo CD / Harbor / Trivy / Grafana LGTM / Keycloak
- セキュリティ層: JWT 検証 / 認可 / 監査ハッシュチェーン

対象に含まないもの:

- Istio Ambient（Phase 2 以降）
- Temporal（Phase 2 以降）
- Backstage（Phase 2 以降）
- 本番データ（合成データのみ使用、個人情報 0 件）
- マルチ DC 構成（Phase 2 以降）

## 検証観点の 5 分類

結合テストの検証観点を 5 分類に整理する。各観点は独立に検証し、合格基準を個別に満たす。

**設計項目 DS-SYS-IT-002 機能結合観点**

- tier2 → tier1 の 11 API 全コールパスが成功する
- tier3 → Envoy Gateway → tier1 → tier2 → データ層 の全経路が動く
- 非同期経路（PubSub / Binding）で Producer と Consumer が正しく連携する
- ワークフロー（Dapr Workflow）が 3 ステップ以上の DAG で完走する
- Decision API が ZEN Engine の JDM ルールを評価し期待結果を返す
- Feature API が flagd 評価で期待値を返す
- 監査ログがハッシュチェーン整合性を保持しながら PostgreSQL WORM に記録される

**設計項目 DS-SYS-IT-003 性能観点**

- tier1 API 11 種すべての p99 レイテンシが 500ms 以内（k6 ベースで 10 分計測）
- PubSub Publish の p99 が 50ms 以内（k6 ベース）
- State Get の p99 が 10ms 以内
- Decision 評価の p99 が 1ms 以内
- 同時接続 1,000 セッションで全 API がエラー率 1% 未満
- スループット目標: 1,000 RPS / tier1 Pod 2 レプリカ（合計 2,000 RPS）

**設計項目 DS-SYS-IT-004 障害観点**

- Chaos Mesh で 1 ノード kill、RTO 4 時間以内に自動復旧（実測値目標 30 分）
- Valkey 1 シャード喪失、残存 Shard で Read/Write 継続
- Kafka 1 ブローカ停止、acks=all で Producer 継続
- PostgreSQL Primary 障害、Standby への failover 30 秒以内
- Envoy Gateway 1 Pod 停止、MetalLB の VIP 再配置 5 秒以内
- OpenBao 1 ノード停止、Raft クォーラム継続

**設計項目 DS-SYS-IT-005 セキュリティ観点**

- JWT なしの API 呼び出しが 401 で拒否される
- 期限切れ JWT の API 呼び出しが 401 で拒否される
- 別テナントの JWT で他テナントのリソースアクセスが 403 で拒否される
- mTLS 証明書なしの内部 Pod 間通信が拒否される（Istio Ambient 導入後）
- OpenBao シークレット読み取り権限がないトークンで Secrets API が 403 を返す
- 監査ログの UPDATE / DELETE が PostgreSQL ロールで拒否される

**設計項目 DS-SYS-IT-006 可観測性観点**

- 1 本の tier3 リクエストに対して Loki / Tempo / Mimir の全てに trace_id が一気通貫で記録される
- 監査ログの hash_chain_prev / hash_chain_curr が 10,000 件連続で整合性を保つ
- Grafana ダッシュボードで SLO エラーバジェット残量が正しく表示される
- アラートが発火して PagerDuty 互換通知経路に流れる（Phase 1c のみ）

## 合格基準

合格基準は定量値で固定する。定性的な「問題なし」「許容範囲」という表現は禁止する。

**設計項目 DS-SYS-IT-007 合格基準一覧**

| 観点 | 指標 | 目標値 | 計測方法 |
|------|------|--------|---------|
| 機能 | tier1 API 成功率 | 99.9% 以上 | k6 10 分計測 |
| 機能 | 非同期 Consumer 消費遅延 | 5 秒以内 | Kafka Consumer Lag メトリクス |
| 機能 | ワークフロー完走率 | 99% 以上 | Dapr Workflow サンプル 100 実行 |
| 性能 | tier1 API p99 | 500ms 以内 | k6 `--summary-trend-stats p99` |
| 性能 | PubSub Publish p99 | 50ms 以内 | k6 |
| 性能 | State Get p99 | 10ms 以内 | k6 |
| 性能 | Decision 評価 p99 | 1ms 以内 | Rust Criterion ベンチ |
| 性能 | 同時接続数 | 1,000 | k6 VUs=1000 |
| 性能 | スループット | 2,000 RPS | k6 rate=2000 |
| 障害 | RTO（ノード障害） | 4 時間以内（目標 30 分） | Chaos Mesh + 手動復旧 |
| 障害 | RPO（データロス） | データ層別（PG 5 分 / Kafka 60 秒 / Valkey 60 秒） | 障害前後の件数差分 |
| セキュリティ | JWT 検証率 | 100% | 10,000 リクエストのサンプル |
| セキュリティ | 脆弱性スキャン | High 以上ゼロ | Trivy コンテナスキャン |
| セキュリティ | 監査 WORM 改ざん試行 | 100% 拒否 | PostgreSQL ロールチェック |
| 可観測性 | trace_id 一気通貫率 | 95% 以上 | Tempo / Loki クロス検索 |
| 可観測性 | ハッシュチェーン整合性 | 100% | 月次検証ツール |

## テスト環境要件

**設計項目 DS-SYS-IT-008 ステージング環境**

- VM: 3 台（本番同等、各 4 vCPU / 16 GB / 200 GB SSD）
- 配置: 本番と同一データセンター、別 VLAN（業務系ではなくテスト系）
- OSS 構成: 本番と同一（Kafka / PostgreSQL / Valkey / OpenBao / MinIO / Grafana LGTM）
- バージョン: 本番と同一マイナーバージョン（パッチ差分のみ許容）
- データ: 合成データのみ、個人情報 0 件

**設計項目 DS-SYS-IT-009 テストデータ方針**

- 個人情報（氏名 / 住所 / 電話 / メール / マイナンバー）は一切使用しない
- 金銭情報（口座 / カード番号）も一切使用しない
- 代替データは Faker.js / Faker Python でシード固定生成
- データサイズ: 10 万レコード相当（本番相当の 1% 規模）
- 多言語: 日本語 / 英語 / 中国語 / 韓国語の混在データを含む（i18n 検証用）

**設計項目 DS-SYS-IT-010 負荷生成環境**

- k6（Grafana Labs、AGPL-3.0、開発環境のみ）
- 負荷生成ホスト: ステージングと別 VM（2 vCPU / 4 GB）
- シナリオ: `ramp-up 1 分 → 定常 8 分 → ramp-down 1 分` × 3 周（合計 30 分）
- 接続元: ステージング VLAN に閉じる（外部からの負荷生成は禁止）

## フェーズ別実施スコープ

結合テストは Phase ごとに実施範囲を拡大する。Phase 1a（MVP-0、2.5 人月 / 10 週間 / 1 FTE / VM 1 台）で対象となる API は `k1s0.Log` のみであり、全 11 API を対象にすると企画書の工数積算を逸脱するため、スモークテストから順次拡大する方式を取る。tier1 残 10 API の結合テストは Phase 1b 以降で段階的に実施する。

**設計項目 DS-SYS-IT-011 Phase 1a スコープ（スモーク）**

- 対象 API: `k1s0.Log` のみ（残 10 API は Phase 1b 以降）
- 機能: `k1s0.Log` への書き込みから kubeadm 単一ノード上の stdout 出力までの疎通 1 本（1 コール）
- 性能: 手動 1 コールで体感レイテンシ確認のみ（自動計測なし、Prometheus / Grafana 未導入のため）
- 障害: 対象外（VM 1 台構成であり HA 試験の前提が成立しない）
- セキュリティ: Keycloak 発行 JWT なしでの 401 拒否のみ
- 可観測性: 標準出力に現れる JSON ログの目視確認のみ（Loki / Grafana は Phase 1c から）
- 所要時間: 30 分

**設計項目 DS-SYS-IT-012 Phase 1b スコープ（機能結合）**

- 対象 API: `k1s0.Log` + `k1s0.Telemetry`（tier1 残 9 API は Phase 2）
- 機能: 2 API + サンプル tier2 / tier3 + 該当データ層（PostgreSQL HA / Prometheus / Grafana / OTel Collector）結合
- 性能: k6 で p99 測定、Log / Telemetry の合格基準（NFR-B-PERF-006 / NFR-B-PERF-007）と照合
- 障害: 1 ノード kill の Chaos 実験 1 回（kubeadm HA 構成の検証）
- セキュリティ: JWT / 認可 / テナント分離の基本ケース
- 可観測性: trace_id が Prometheus / Grafana に現れることを 100 トランザクション分検証
- 所要時間: 3 日

**設計項目 DS-SYS-IT-013 Phase 1c スコープ（運用品質追加）**

- 対象 API: `k1s0.Log` + `k1s0.Telemetry`（変更なし）
- 機能: Harbor / Trivy / Loki / Kyverno / OpenBao / MinIO など運用品質 OSS を追加した状態での再回帰
- 性能: 連続 24 時間負荷のソーク試験を追加
- 障害: Chaos Mesh で 5 シナリオ（ノード kill / ネットワーク分断 / CPU ストレス / DNS 障害 / OpenBao seal）
- セキュリティ: DAST（OWASP ZAP）/ SAST（CodeQL） / Container Scan（Trivy）/ SBOM 検証
- 可観測性: Loki へのログ集約と trace_id 相関の検証
- 所要時間: 1 週間

**設計項目 DS-SYS-IT-014 Phase 2 スコープ骨子**

- 対象 API: tier1 残 9 API（Service Invoke / State / PubSub / Secrets / Binding / Workflow / Decision / Audit-Pii / Feature）を本格結合
- Istio Ambient 追加での mTLS 検証
- Temporal ワークフロー結合 / ZEN Engine（Decision API）結合 / flagd（Feature API）結合
- 監査ハッシュチェーン改ざん試行の 100% 検出 / WORM 7 年保持検証
- マルチ DC 切替演習（RPO 60 秒検証）
- Backstage 経由のセルフサービスポータル結合
- 所要時間: 4 週間

## 実施フロー

結合テストは **自動化 + 人手確認** のハイブリッドで実施する。自動化できる範囲（API 疎通 / 性能測定 / 脆弱性スキャン）は CI で毎日実行、Chaos 実験と人手判定が必要な観点は週次で実施する。

**設計項目 DS-SYS-IT-015 実施フロー**

1. Git main ブランチへのマージ → Argo CD がステージングにデプロイ → 自動結合テスト起動
2. 自動結合テスト: k6 性能 / Trivy セキュリティ / OpenTelemetry trace 検証（10 分）
3. 週次 Chaos: 毎週火曜日 10:00 JST に Chaos Mesh シナリオ 1 つを実施
4. Phase 判定時: 全観点を一括実施、Product Council へレポート提出
5. 不合格時: 該当観点を修正 → 再実行、Phase 判定を次回に持ち越し

## 結合テスト合格と Phase 承認の関係

Phase 1b / Phase 1c の Product Council 承認は、結合テスト合格を必須条件とする。承認基準と結合テスト合格基準の対応を以下で明示する。

**設計項目 DS-SYS-IT-016 Phase 承認と結合テストの関係**

- Phase 1b 承認条件: DS-SYS-IT-012 の全合格基準クリア
- Phase 1c 承認条件: DS-SYS-IT-013 の全合格基準クリア + DS-SYS-IT-012 の再実行合格
- Phase 2 着手条件: Phase 1c 承認 + 予算承認 + Istio Ambient 導入の ADR 承認
