# 02. 設計原則と制約

本ファイルは k1s0 概要設計の全章横断で適用される設計原則と、構想設計・企画で確定した制約を列挙する。各章の執筆者は作業開始前に本ファイルを読み、設計判断の前提として参照する。

## 本ファイルの位置付け

プラットフォーム製品の設計では、章ごとに設計者が独立に判断すると章間で一貫性が崩れる。同じコンポーネント名が章 A では「State サービス」、章 B では「State ファサード」と書かれる、同じタイムアウト値が異なる章で異なる秒数で記述される、といった不整合が典型である。こうした不整合は実装段階で検出されると修正コストが跳ね上がり、場合によっては採用検討で合意した SLO を満たせない実装が完成してしまう。本ファイルは全章の設計判断の共通土台を提供することで、これらを上流で未然に防ぐ役割を担う。

原則と制約は二層で構成する。原則は k1s0 全体が守るべき設計思想（変更可能な価値観）、制約は構想設計・企画で確定した不可変の前提（変更には ADR 起票と採用再検討が必要）である。両者を混在させると、設計者が「この項目は動かせるのか動かせないのか」を判別できなくなるため明示的に分離する。

## 設計原則

設計原則は k1s0 のアーキテクチャと運用における価値判断の共通基準である。以下 10 原則を全章で共有する。各原則は複数の要件 ID にまたがって影響するため、設計判断で迷った場合はまず該当する原則に立ち返る。

### 原則 1: tier1 は複雑性を吸収する

企画書の核心思想「tier1 は tier2 を楽にするために、tier2 は tier3 を楽にするために存在する」を設計原則として昇格させる。横断的関心事（認証・ログ・監査・暗号・エラー処理・リトライ）は tier1 側で吸収し、tier2 / tier3 開発者はビジネスロジックに集中できるインタフェースを提供する。tier1 の内部複雑性は許容し、代わりに tier1 公開 API の単純性を死守する。

この原則に反する設計は禁止する。たとえば「JWT 検証を tier2 開発者に任せる」「Valkey 接続情報を tier2 が直接扱う」「Dapr Component YAML を tier2 の責任とする」といった分担は本原則違反となる。

### 原則 2: tier 間は一方向依存のみ

tier3 → tier2 → tier1 → infra の一方向依存のみを許可する。逆向き・斜め・同階層間の依存は禁止する。この依存ルールは構想設計 ADR-0001 および全体構成図で確定しており、概要設計でも厳守する。

具体的には、tier1 が tier2 / tier3 の識別子を知ることは許容するが、tier1 のコードが tier2 のコードを import する、tier2 の gRPC を呼ぶ、tier2 のイベントを購読するといった依存は禁止する。tier2 からの情報は公開 API の引数として渡される場合のみ受け入れる。

### 原則 3: 内部実装は tier2 / tier3 に不可視

tier1 内部の実装詳細（Dapr、Valkey、Kafka、OpenBao、ZEN Engine など）は tier2 / tier3 から不可視とする。tier2 / tier3 は tier1 公開 API（k1s0.State、k1s0.PubSub など）のみを呼び出し、Dapr SDK や内部ストアへの直接アクセスは禁止する。

この原則により、tier1 は内部で OSS を差し替えてもインタフェース互換を維持でき、tier2 / tier3 の改修を伴わずに保守を進められる。逆に、この原則を緩めると、tier1 の OSS バージョン更新が毎回 tier2 / tier3 の改修を巻き起こし、企画で約束した運用 採用側の小規模運用が破綻する。

### 原則 4: Protobuf gRPC を内部言語境界の唯一契約とする

tier1 内部（Go と Rust）の通信は Protobuf gRPC のみとし、HTTP/JSON・共有メモリ・ファイル渡しなどの別手段は禁止する。`.proto` ファイルを単一の真実として扱い、Go 側・Rust 側・tier2/3 クライアント側のすべてを自動生成する。

これにより、Go と Rust 間の型不整合バグが構造的に起こらなくなり、スキーマ変更が両言語で同時に反映される。CI では `buf breaking` を必須とし、非後方互換変更を自動検出する。

### 原則 5: Dapr のバージョンは tier1 内に閉じ込める

Dapr は内部実装詳細として扱い、tier2 / tier3 の契約に影響を与えない。Dapr Go SDK の破壊的変更は tier1 の Go コード内で吸収し、tier1 公開 API のシグネチャは変更しない。tier2 / tier3 開発者は Dapr の存在自体を意識しないことが望ましい状態である。

この原則の維持には、tier1 側で「公開 API → 内部変換 → Dapr SDK」の薄いアダプタ層を設ける設計が必須となる。アダプタ層の詳細は [../20_ソフトウェア方式設計/03_内部インタフェース方式設計/03_Go_Rust間言語境界方式.md](../20_ソフトウェア方式設計/03_内部インタフェース方式設計/03_Go_Rust間言語境界方式.md) で定義する。

### 原則 6: 具体数値を逃げない

設計書内の非機能目標は必ず具体数値で記述する。「高速」「大量」「早期」といった曖昧表現は禁止する。タイムアウト秒数、リトライ回数、バックオフ係数、RPS、p99 レイテンシ、RPO 秒、RTO 時間のすべてを具体的数値で書き、その根拠（積算式・実測値・業界事例）を併記する。

数値が未決の場合は「未決」と明記し、決定期限と担当者を記載する。「あとで決める」「詳細設計で決定」などの先送り表現は禁止する。

### 原則 7: 設計は段階に切り分ける

企画で合意したリリース時点（tier1 11 API 全機能本格実装）と、採用後の運用拡大時（採用側スケール拡大）の段階区分を設計書で明示する。各設計項目は「どの段階で確定するか」をメタデータとして持ち、リリース時点で確定する項目を主、運用拡大時に確定する項目を副として位置付ける。

リリース時点 段階では 採用初期 の設計は参考扱いであり、採用後の運用拡大時の設計は骨子のみを記述する。段階境界を曖昧にすると、採用検討済みスコープを超えた設計が紛れ込み、後の追加予算要求で説明責任を果たせなくなる。

### 原則 8: AGPL OSS は 運用蓄積後で隔離運用

AGPL-3.0 の OSS（Grafana / Loki / Tempo / Pyroscope / MinIO / Renovate）は 採用初期では採用しない。運用蓄積後の採用時は構想設計 ADR-0003 の 5 原則（プロセス分離 / ライブラリリンクなし / 無改変利用 / 外部境界遮断 / 監査ログ可検証性）を構造的に満たす設計とする。

具体設計例として、Grafana は独立 Pod として稼働させ、自作コードへの直接リンクをせず、HTTP API 経由のみで接続し、外部公開は Envoy Gateway + Keycloak OIDC で遮断する。隔離の詳細は [../50_非機能方式設計/05_セキュリティ方式設計.md](../50_非機能方式設計/05_セキュリティ方式設計.md) および [../75_事業運用方式設計/07_OSSライセンス運用方式.md](../75_事業運用方式設計/07_OSSライセンス運用方式.md) で記述する。

### 原則 9: 採用側の小規模運用で成立する

採用初期 では最大 採用側の小規模運用で運用できる設計とする。運用工数が 2 名の稼働時間（月次の運用工数枠、病欠・年休を差し引いて実働 280 人時）を超える設計は不可とする。自動化できないオペレーション、属人化する手順、ドキュメント化されていない暗黙知は設計段階で排除する。

この原則は企画で約束した「バス係数 2」の実現に直結する。起案者不在で協力者が Runbook を参照して障害対処を完結できること、新規協力者が 2 週間以内にキャッチアップできること、が運用設計の合格基準となる。

### 原則 10: オンプレ完結を前提とする

クラウドマネージド SaaS（EKS / AKS / GKE / RDS / MSK / Secrets Manager など）への依存は禁止する。すべての OSS はセルフマネージドで運用し、閉域ネットワーク環境でも同一構成で動作する設計とする。

この原則は 採用側組織の情シス部門の大半が閉域運用を前提とする市場特性から導出される。クラウド接続を前提とした設計は採用機会を大幅に狭め、企画で約束した差別化戦略（オンプレ完結）と矛盾する。

## 不可変制約

以下は構想設計・企画で確定した制約であり、概要設計では変更できない。制約を変更する場合は、まず構想設計 ADR または採用再検討を経由する必要がある。

### 制約 1: 採用 OSS の集合

State Store は Valkey、PubSub は Apache Kafka（Strimzi Operator）、Secret Store は OpenBao、DB は PostgreSQL（CloudNativePG）、サービスメッシュは Istio Ambient、ワークフローエンジンは Dapr Workflow と Temporal の併用、ルールエンジンは ZEN Engine、Feature Flag は flagd、観測基盤は Grafana LGTM、CI/CD は GHA + Argo CD、認証は Keycloak とする。この集合は構想設計 ADR-DATA-001〜004、ADR-SEC-001〜003、ADR-CICD-001〜003、ADR-OBS-001〜002、ADR-FM-001、ADR-RULE-001〜002 で確定している。

### 制約 2: 言語分担

Dapr ファサード層は Go、自作領域は Rust とする。Dapr ファサード層は Dapr Go SDK が stable のため Go を選択し、自作領域は ZEN Engine（Rust 製）統合・crypto 処理・監査改ざん防止の性能と安全性要件から Rust を選択する。この分担は構想設計 ADR-TIER1-001 で確定している。

### 制約 3: Protobuf gRPC 必須

tier1 内部通信は Protobuf gRPC のみ許可する。HTTP/JSON、MessagePack、独自バイナリプロトコルは禁止する。`.proto` ファイルは `src/tier1/contracts/` に一元管理し、buf CLI で breaking change を検出する。この制約は構想設計 ADR-TIER1-002 で確定している。

### 制約 4: tier 構造

tier1 / tier2 / tier3 の三層構造を基本とし、infra 層と operation 層を加えた 5 層とする。tier1 は 採用側組織の情シス基盤チームが保守、tier2 は各事業ドメインチームが保守、tier3 はアプリ開発チームが保守する。infra 層はインフラチームが保守し、operation 層は運用チームが横断で担当する。この構造は企画書で確定している。

### 制約 5: スコープ

k1s0 は起案者の個人開発 OSS としてリリース時点で tier1 11 API 全機能を本格実装する。採用側の運用は VM 3 台規模の小規模運用から開始でき、運用拡大時に採用側のチームが必要に応じて拡張する。各設計項目の確定段階（リリース時点 / 採用後の運用拡大時 など）は設計書内で明示する。

### 制約 6: 非機能数値コミット

tier1 API p99 レイテンシ 500ms 以内（業務 200ms + Dapr 80ms + OTel 20ms + 監査 50ms + NW・DB 150ms の積算）、SLA 稼働率 99%（月 7.2 時間許容）、SLO 99.9%（月 43 分許容）、RPO 秒オーダー〜24 時間（バックエンド別）、RTO 4 時間、Decision 評価 p99 1ms 以内、PubSub Publish p99 50ms 以内、State Get p99 10ms 以内、を企画で約束している。設計では各数値が構成要素にどう分配されるかを明示する。

### 制約 7: 法務制約

AGPL-3.0 OSS の採用は構想設計 ADR-0003 の 5 原則を満たすことが必須。BSL/SSPL/Commons Clause の OSS は採用禁止。個人情報保護法 2022 年改正版・J-SOX・電帳法・GDPR（国際対応時）への準拠を維持する。SBOM 自動生成と四半期ライセンスレポートを運用義務として担保する。

### 制約 8: オンプレ完結

すべての OSS はセルフマネージドで運用し、クラウド接続必須のマネージド SaaS には依存しない。VM / ベアメタル構成で動作すること、閉域ネットワークで動作すること、既存データセンター資産（VM・ストレージ・LB・DNS・NTP）と連携できることが前提となる。

### 制約 9: レガシー共存

既存の .NET Framework 資産は捨てずに共存する。tier3 拡張ポイントとしてラップ（ClickOnce / MSIX）し、配信ポータルから配信可能とする。強制移行ではなく自発的移行を促す設計とし、既存資産と k1s0 の同時稼働期間を許容する。

### 制約 10: 採用検討通過済み予算

リリース時点〜採用初期 の初期構築費 1,850 万円、採用後の運用拡大時の年間運用費は OSS 隠れコスト 2,349 万円/年（自社 5 FTE 人件費 6,900 万円は別計上、合算 9,249 万円）の範囲で設計する。ここで「2,349 万円」は VM 償却・電力・ストレージ・監査ツール・外部 TSA など OSS をセルフマネージドで回す際に生じる現金コストのみを指し、運用 採用側の小規模運用の人件費は含まない。合算 9,249 万円が競合 OpenShift 中規模 1.8 億円に対する 49% 削減コミットの分母であり、どちらかを単独で引用すると企画の数値構造が崩れるため、本設計書では原則として併記する。この予算を超える設計は採用再検討を要し、独断で進めることは禁止する。設計時点で超過が予測される場合は事前にアーキテクトと企画担当者へエスカレーションする。

## 各章の参照順

各章の執筆者は以下の順で本ファイルを参照する。第一読時は全節を通読し、執筆中は該当する原則・制約を必要に応じて参照する。

1. 章担当開始時に原則 1〜10 を通読し、章の設計思想が原則と整合することを確認する。
2. 設計判断で迷った場合、該当する制約 1〜10 を検索し、制約として確定済みか変更可能かを判別する。
3. 制約に抵触する設計が必要と判断した場合、ADR 起票の手続（[01_概要設計プロセス.md](01_概要設計プロセス.md) 参照）に従う。
4. 章レビュー時、レビュアは原則・制約と設計内容の整合を確認する。

## 対応要件一覧

本ファイルは概要設計横断の前提整理であり、特定の要件 ID への直接対応はない。ただし、以下の要件定義書側エントリと連動する。前提条件 P-BIZ-001〜003 / P-TEC-001〜005 / P-ORG-001〜003 と、制約条件 C-LEGAL-001〜003 / C-REG-001〜002 / C-INF-001〜002 / C-EXIST-001〜003 / C-VEND-001 / C-TIME-001 の全項目を設計原則・制約に反映している。設計 ID を採番する性質のファイルではないため、設計 ID 採番対象外とする。
