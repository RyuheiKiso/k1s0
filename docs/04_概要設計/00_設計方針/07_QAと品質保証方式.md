# 07. QA と品質保証方式

本ファイルは概要設計成果物そのものの品質保証方式を定める。完全性・整合性・実装可能性を継続検証する手続と自動化方式を固定化する。

## 本ファイルの位置付け

概要設計書の品質保証は、実装後のソフトウェア品質保証と性格が異なる。ソフトウェアは実行してテストできるが、設計書は実行できないため、品質保証は「人による確認」と「静的チェックの自動化」の組合せで担保するしかない。品質保証が疎かになると、設計書自体は提出されていても、要件カバレッジが漏れた・章間整合が崩れた・実装不可能な設計が紛れ込んだまま、次段階に進んでしまう。

本ファイルはこれを防ぐため、設計書の完全性・整合性・実装可能性の 3 観点を継続検証する手続を定める。自動化できる部分は CI で機械チェックし、人による判断が必要な部分は段階ゲートで明示的に確認する。

## 品質保証の 3 観点

### 観点 1: 完全性

完全性は「要件に対する設計の網羅度」を指す。以下の 4 項目で評価する。

- **要件カバレッジ 100%**: 要件定義書で採番された全要件 ID（FR / NFR / BR / OR / DX / BC / MIG）が、概要設計書内の少なくとも 1 つの設計 ID で参照されていること。
- **トレーサビリティ欠損なし**: [../80_トレーサビリティ/02_要件から設計へのマトリクス.md](../80_トレーサビリティ/02_要件から設計へのマトリクス.md) において、要件 ID → 設計 ID の対応が全要件について記載されていること。
- **IPA 共通フレーム対応なし**: IPA 共通フレーム 2013 の該当アクティビティ（6.4.3 / 7.1.2 / 6.4.10）のすべての必須成果物が、概要設計書のいずれかの章で対応していること。
- **空ファイルなし**: 概要設計書配下の `.md` ファイルがすべて内容を持ち、空ファイルが存在しないこと。

### 観点 2: 整合性

整合性は「章間・要素間の矛盾の不在」を指す。以下の 5 項目で評価する。

- **用語一致**: 章間で同一概念が同一用語で記述されること。[05_用語集と命名規約.md](05_用語集と命名規約.md) で定義した用語のみを使用すること。
- **設計 ID 衝突なし**: 章間で設計 ID の重複採番がないこと。
- **数値一致**: 非機能目標値（レイテンシ・スループット・容量）が章間で矛盾しないこと。tier1 API p99 500ms の積算内訳（業務 200ms + Dapr 80ms + OTel 20ms + 監査 50ms + NW・DB 150ms）が全章で一致すること。
- **コンポーネント名一致**: tier1 コンポーネント名（COMP-T1-STATE 等）が章間で一致すること。
- **エラーコード一致**: tier1 API のエラーコード（E-INVOKE-* 等）が章間で一致すること。

### 観点 3: 実装可能性

実装可能性は「設計が構想設計 ADR と採用 OSS で実現可能か」を指す。以下の 4 項目で評価する。

- **ADR 整合**: 構想設計 ADR（ADR-0001〜ADR-BS-001）に記載された技術選定と矛盾する設計がないこと。
- **OSS 機能整合**: 採用 OSS の機能範囲内で実現可能な設計であること。たとえば Dapr State API に存在しない機能を前提にしていないこと。
- **性能積算整合**: 設計が定めるレイテンシ割当が OSS の実測または公表値で達成可能であること。
- **法務制約整合**: AGPL OSS の扱いが構想設計 ADR-0003 の 5 原則に従っていること。

## 静的チェックの自動化

自動化可能なチェックは CI に組み込む。設計書は Markdown で管理するため、CI は以下のチェックを実行する。

### リンク切れ検出

Markdown 内の相対リンクが有効であることを `markdown-link-check` 系ツールで検証する。リンク切れを検出した場合は CI を失敗させ、PR をマージ不可とする。外部 URL のチェックは取得不能（社内ネットワーク制約）を考慮し、警告のみで PR は通す。

### 設計 ID 重複検出

`grep` / `rg` で `DS-[A-Z]+-[A-Z]+-[0-9]{3,}` を抽出し、重複があれば CI を失敗させる。同じ設計 ID が複数ファイルに登場しても、それが「採番」か「参照」かを判別するため、本文中の「### DS-」形式の見出しのみを「採番」と扱う。

### 用語ゆれ検出

[05_用語集と命名規約.md](05_用語集と命名規約.md) の「用語ゆれの禁止例」節で列挙した禁止表記（「Tier1」「tier 1」「k8s」等）を `rg` で検出し、該当があれば CI を警告とする（機械判定の誤検出が出うるため失敗ではなく警告扱い）。

### 要件 ID カバレッジ検出

要件定義書の全要件 ID を抽出し、各要件 ID が概要設計書のいずれかのファイルで参照されているかを検証する。未参照の要件 ID があれば CI を失敗させる。ただし、参照先が「対応要件一覧」節に列挙されていれば OK とする。

### 数値一致検出

tier1 API p99 500ms などの主要数値は、事前に「期待値辞書」として登録し、章間で異なる値が検出されたら CI を警告とする。数値は積算式の要素としても登場するため、単純一致ではなく文脈解析が必要で完全自動化は難しいため、レビュア判断の補助として警告レベルにとどめる。

### drawio 品質検出

drawio ファイルは XML 形式のため、白背景矩形の存在、mxGraphModel の background 属性非依存、矢印の座標と図形の座標の交差判定をスクリプトで検証する。交差があれば CI を失敗させる。詳細検証ルールは [../../00_format/drawio_layer_convention.md](../../00_format/drawio_layer_convention.md) の規約に従う。

## 人による確認の手続

自動化できない確認は以下の手続で実施する。

### 構想設計 ADR との整合確認

章執筆完了時、執筆者が構想設計 ADR 索引（[../90_付録/02_ADR索引.md](../90_付録/02_ADR索引.md)）と章内の ADR 参照箇所を突き合わせ、参照 ADR と設計方式の整合を確認する。突き合わせ結果は PR 本文に記載する。アーキテクトレビュアは同確認を再実施する。

### 採用検討スコープとの乖離確認

章執筆完了時、執筆者が企画書（[../../01_企画/企画書.md](../../01_企画/企画書.md)）の段階計画と章内の設計項目を突き合わせ、採用検討で約束した段階スコープとの乖離がないかを確認する。乖離がある場合は Product Council 承認前に段階スコープ変更の決議が必要となる。

### 法務制約との整合確認

30_共通機能方式設計、50_非機能方式設計、55_運用ライフサイクル方式設計、75_事業運用方式設計の 4 章は法務制約と密接に関わる。各章の Product Council 承認前に法務責任者の確認を必須とする。確認対象は AGPL OSS の取扱い、個人情報の取扱い、ライセンス義務の履行方式である。

### 実装プロトタイプとの整合確認

性能目標や信頼性目標は設計段階では机上値だが、リリース時点 のプロトタイプで一部を実測する。実測値が設計前提と乖離している場合は、設計書を改訂する。改訂影響範囲は設計 ID ベースで追跡する。

## 品質メトリクス

品質保証の実施状況を定量化するため、以下のメトリクスを集計する。集計は月次で PMO が実施し、Product Council へ報告する。

**設計 PR リードタイム**: 設計 PR 起票から Product Council 承認までの日数。目標は 15 日以内（段階 2〜4 の各段階で 5 日以内）。

**差戻し回数**: 設計 PR あたりの差戻し回数。目標は平均 2 回以下。3 回以上が平均化する場合は、設計観点チェックリストの強化が必要と判断する。

**要件カバレッジ**: 要件 ID のうち設計 ID で参照されている割合。目標は各段階ゲート時点で 100%。未達の場合は該当要件の設計責任者を Product Council で決議する。

**静的チェック成功率**: CI 静的チェックの成功率。目標は 95% 以上。失敗が継続する場合はチェックルール自体の見直しを検討する。

**章間整合不整合検出数**: 横串レビューで検出された章間整合不整合の件数。目標は PR あたり 3 件以下。超過する場合は章執筆前の事前打ち合わせを強化する。

## 品質保証の責任分担

品質保証の責任者を以下のように分担する。

**設計者**: 自己レビュー（段階 2）の実施、静的チェックの通過、要件カバレッジの列挙。

**ピアレビュア**: 章内論理整合のレビュー、散文品質のレビュー、図表の妥当性確認。

**アーキテクト**: 章間整合のレビュー、構想設計 ADR 整合の確認、実装可能性の確認。

**PMO**: 静的チェック自動化の維持、品質メトリクスの集計、リスクレジスタの更新、Product Council 向け報告資料の作成。

**Product Council**: 設計 PR の最終承認、採用検討スコープとの乖離判定、法務制約整合の確認。

## 継続的改善

品質保証の手続自体も継続的に改善する。四半期ごとに以下を実施する。

(1) 品質メトリクスを集計し、目標達成度を評価する。(2) 静的チェックで検出できなかった不整合が人によるレビューで検出された場合、同様の不整合を自動検出できるチェックルールを追加する。(3) 差戻し事由の分類を集計し、頻発する事由に対する予防策を検討する。(4) レビュー所要時間が長い PR の特性を分析し、PR 分割ルールを見直す。

改善内容は本ファイルに追記し、改訂履歴として残す。

## 対応要件一覧

本ファイルは設計成果物の品質保証手続を定めるため、特定要件 ID への直接対応はない。ただし、要件定義書側のプロジェクト管理カテゴリ（[../../03_要件定義/70_プロジェクト管理/](../../03_要件定義/70_プロジェクト管理/)）の QA 計画と連動する。設計 ID 採番対象外とする。
