# 03. 設計 ID 体系とトレーサビリティ

本ファイルは概要設計書で採番するすべての設計 ID の命名規則、要件 ID との対応方式、変更時の追跡ルールを定める。設計 ID はトレーサビリティの中核であり、章跨ぎの参照・改訂影響範囲の特定・監査対応のすべてが本 ID に依存する。

## 本ファイルの位置付け

プラットフォーム製品の設計書では、個別の設計項目（コンポーネント・インタフェース・制御方式・運用手順）が数百〜千を超える粒度で存在する。これらを識別する一貫した ID 体系がないと、章間参照が「前章の Saga 方式の第 3 項目を参照」のような位置依存の参照になり、章の順序変更や項目追加で参照が壊れる。また、要件 ID との対応が人間の記憶に依存すると、要件変更時の影響範囲特定が漏れる。

本ファイルはこれらの発散を防ぐため、機械可読な設計 ID 命名規則と双方向トレーサビリティのルールを固定化する。命名規則は一度決めたら長期的に動かせない性質を持つため、全章執筆開始前に本ファイルで合意を固める。

## 設計 ID 基本書式

設計 ID は以下の書式で統一する。

```
DS-<カテゴリ>-<サブ>-<通番>
```

- `DS` は Design の略で、設計 ID であることを示す固定接頭辞。要件 ID の `FR` / `NFR` / `BR` / `OR` / `DX` / `BC` と衝突しない。
- `<カテゴリ>` は概要設計書の章対応カテゴリ識別子（大文字英字 2〜5 文字）。
- `<サブ>` はカテゴリ内のサブ分類識別子（大文字英字 2〜7 文字）。
- `<通番>` は 3 桁の連番（`001` 始まり、欠番不可）。

書式例として、システム方式設計のハードウェア方式 1 番目は `DS-SYS-HW-001`、ソフトウェア方式設計のコンポーネント方式 15 番目は `DS-SW-COMP-015`、共通機能方式設計の認証方式 3 番目は `DS-CF-AUTH-003` となる。

## カテゴリ識別子の割り当て

カテゴリ識別子は概要設計書の 10〜75 章に 1:1 対応する。以下の割当を固定する。

- `SYS` : 10 システム方式設計（System）
- `SW` : 20 ソフトウェア方式設計（Software）
- `CF` : 30 共通機能方式設計（Common Function）
- `CTRL` : 40 制御方式設計（Control）
- `NFR` : 50 非機能方式設計（NFR）
- `OPS` : 55 運用ライフサイクル方式設計（Operations Lifecycle）
- `MIG` : 60 移行方式設計（Migration）
- `DEVX` : 70 開発者体験方式設計（Developer Experience）
- `BUS` : 75 事業運用方式設計（Business Operations）

80 トレーサビリティおよび 90 付録は設計 ID 採番対象外とする。これらは設計項目そのものではなく、設計項目を索引化または参照する性質の章であるためである。00 設計方針も同様に設計 ID 採番対象外とする。

## サブ分類識別子の割り当て

各カテゴリ内のサブ分類識別子は、そのカテゴリの主要トピックに対応する。以下の割当を標準とし、章執筆時に不足が生じた場合は本ファイルを更新してから新規識別子を追加する。

**SYS カテゴリ**: `HW`（ハードウェア方式）/ `SW`（ソフトウェア全体分担）/ `PROC`（システム処理方式）/ `DB`（データベース全体方式）/ `IT`（システム結合テスト要件）/ `REV`（評価レビュー）。例: `DS-SYS-HW-001`。

**SW カテゴリ**: `COMP`（コンポーネント方式）/ `EIF`（外部インタフェース方式）/ `IIF`（内部インタフェース方式）/ `DB`（データベース最上位）/ `DOC`（利用者文書暫定版）/ `IT`（結合テスト要件）/ `REV`（評価レビュー）/ `SEQ`（API シーケンス）。例: `DS-SW-COMP-001`、`DS-SW-EIF-042`。

**CF カテゴリ**: `AUTH`（認証）/ `AUTHZ`（認可）/ `LOG`（ログ）/ `AUD`（監査証跡）/ `TRACE`（分散トレーシング）/ `METRIC`（メトリクス）/ `ERR`（エラーハンドリング）/ `CRYPT`（暗号化鍵管理）/ `FM`（FeatureManagement）。例: `DS-CF-AUTH-001`。

**CTRL カテゴリ**: `SAGA`（Saga）/ `IDEM`（冪等性）/ `RETRY`（リトライとサーキットブレーカー）/ `MSG`（非同期メッセージング）/ `WF`（ワークフロー振り分け）/ `TO`（タイムアウトとバックプレッシャ）。例: `DS-CTRL-SAGA-001`。

**NFR カテゴリ**: `AVL`（可用性 DR）/ `PERF`（性能拡張）/ `OPS`（運用）/ `OBS`（監視観測性）/ `SEC`（セキュリティ）/ `CAP`（キャパシティ計画）/ `ENV`（環境エコロジー）/ `PRV`（データ保護プライバシー）/ `COMP`（完整性コンプライアンス）/ `THM`（脅威モデリング）/ `SLO`（SLI SLO エラーバジェット）。例: `DS-NFR-AVL-001`。

**OPS カテゴリ**: `SUP`（サポート階層）/ `INC`（インシデント対応）/ `ENV`（環境構成管理）/ `EOL`（非推奨 EOL）/ `EXIT`（撤退戦略）/ `FMEA`（FMEA）/ `LOAD`（負荷試験）/ `RB`（Runbook）。例: `DS-OPS-INC-001`。

**MIG カテゴリ**: `DATA`（データ移行）/ `CUT`（並行稼働切替）/ `RB`（ロールバック復旧）。例: `DS-MIG-DATA-001`。

**DEVX カテゴリ**: `CICD`（CI/CD）/ `LOCAL`（ローカル開発環境）/ `MET`（DevEx 指標）/ `BS`（Backstage ポータル）/ `TEST`（テスト戦略）/ `GP`（ゴールデンパス）。例: `DS-DEVX-CICD-001`。

**BUS カテゴリ**: `UX`（UX 方式）/ `I18N`（i18n）/ `ONB`（テナントオンボーディング）/ `TRN`（教育訓練）/ `COST`（コスト管理）/ `BIL`（課金メータリング）/ `LIC`（OSS ライセンス）/ `LGL`（法務契約）/ `SC`（サプライチェーン）/ `GOV`（ガバナンス）/ `AI`（AI ガバナンス）/ `ANA`（分析レポーティング）/ `SUS`（サステナビリティ）。例: `DS-BUS-BIL-001`。

## 通番採番ルール

通番は 3 桁の連番とする。001 から採番を開始し、同一「カテゴリ + サブ」内で欠番を生じさせない。1000 番を超える場合は 4 桁に拡張し、拡張時に [03_設計ID体系とトレーサビリティ.md](03_設計ID体系とトレーサビリティ.md)（本ファイル）を更新する。

通番の採番は章執筆時に章内で連続付与する。採番のリセットは章単位では行わず、同一カテゴリ全体で連続させる。たとえば `DS-SW-COMP-*` は 01_tier1 全体コンポーネント俯瞰から 06_パッケージ構成まで連続採番し、番号が若い方がコンポーネント全体像に近い概念であることを暗黙の慣習とする。

設計項目を削除する場合は欠番を残さず、後続の番号を詰める。ただし、一度 Product Council 承認された設計 ID は欠番扱いとし、他の項目に再利用しない（古い PR コメントや変更履歴からの参照を壊さないため）。欠番化する場合は本ファイルの「廃番一覧」節に記録する。

## 要件 ID との対応方式

設計 ID は対応する要件 ID と双方向トレース関係を保つ。対応は n:n 関係を許容する（複数設計が同一要件を分担する、複数要件を 1 つの設計でまとめて実現する、の両方を許容）。

設計 ID を採番する際、その設計が充足する要件 ID を本文末尾の「対応要件一覧」節に列挙する。要件 ID が列挙されていない設計 ID は、設計 ID 採番の根拠を失うため採番不可とする。ただし、純粋に横断的な設計（例: 用語集）で要件対応が間接的な場合は「対応要件: 間接対応」と明記する。

逆方向の対応（要件 ID → 設計 ID）は [../80_トレーサビリティ/02_要件から設計へのマトリクス.md](../80_トレーサビリティ/02_要件から設計へのマトリクス.md) で一元管理する。章執筆完了時、執筆者は同マトリクスへの追記を PR に含める。

## フェーズ属性

各設計 ID は「確定フェーズ」属性を持つ。確定フェーズは以下 5 種類から選択する。

- **Phase 0**: 稟議承認時点で確定する設計。主に 00_設計方針カテゴリと、Phase 1a 着手に必要な最小限の設計。
- **Phase 1a（MVP-0）**: デモ構成で確定する設計。tier1 API の最小サブセット、kubeadm + Dapr + Keycloak + 配信ポータル。
- **Phase 1b（MVP-1a）**: パイロット運用で確定する設計。tier1 11 API 全公開、HA 構成、基本運用。
- **Phase 1c（MVP-1b）**: 運用品質確保で確定する設計。監査・セキュリティ・バックアップ・復旧。
- **Phase 2 以降**: 基盤拡張で確定する設計。Temporal、Istio Ambient、Backstage、マルチクラスタなど。

確定フェーズは設計 ID の本文メタデータとして記載する。Phase 0 稟議承認時点では、Phase 1a〜1c の設計は参考扱い、Phase 2 以降の設計は骨子のみとする。

## 変更追跡

設計 ID の変更は以下のカテゴリに分類する。

**項目追加**: 新規に設計 ID を採番する。次の通番を用いる。追加の PR に対応要件 ID と変更根拠を明記する。

**項目内容改訂**: 既存設計 ID の本文を更新する。設計 ID は維持する。PR タイトルに設計 ID を前置する。改訂の理由が要件変更由来か、構想設計 ADR 変更由来か、実装フィードバック由来かを PR 本文で明示する。

**項目分割**: 1 つの設計 ID を 2 つ以上に分割する。元の設計 ID は廃番とし、廃番一覧に記録する。新設計 ID を新規採番する。

**項目統合**: 2 つ以上の設計 ID を 1 つに統合する。統合先の設計 ID を維持し、統合元は廃番一覧に記録する。

**項目削除**: 設計 ID を削除する。削除時は廃番一覧に記録する。設計 ID を参照している他章の記述も同 PR で更新する。

変更履歴は本書末尾の「改訂履歴」節と各設計 ID の本文内履歴節の双方に記録する。二重管理の負担はあるが、設計 ID 単位の履歴がないと影響範囲特定が困難になるため双方を維持する。

## 参照表記

他章や他設計 ID から設計 ID を参照する場合、以下の書式で統一する。

- 同一章内参照: `DS-SW-COMP-015` のように設計 ID のみを記載。
- 他章参照: `DS-SW-COMP-015`（[../20_ソフトウェア方式設計/01_コンポーネント方式設計/01_tier1全体コンポーネント俯瞰.md](../20_ソフトウェア方式設計/01_コンポーネント方式設計/01_tier1全体コンポーネント俯瞰.md) 参照）のように設計 ID と格納ファイル相対リンクを併記する。
- 複数参照: `DS-SW-COMP-015` / `DS-SW-COMP-016` / `DS-SW-COMP-017` のようにスラッシュで列挙する。

参照切れは本ファイル設計方針の [07_QAと品質保証方式.md](07_QAと品質保証方式.md) で定義する静的チェックで継続検証する。

## 廃番一覧

廃番となった設計 ID を本節で管理する。Phase 0 稟議時点では廃番は存在しないため空欄とするが、以降の改訂で廃番が発生した場合はここに追記する。記載項目は設計 ID、廃番日、廃番理由、後継設計 ID（存在する場合）、関連 PR の 5 項目とする。

## 実装 ID 体系（IMP- 接頭辞）

概要設計の設計 ID（`DS-`）は要件の実現方式を論理レベルで確定する ID 体系である。実装フェーズ（`docs/05_実装/` 配下）では、ディレクトリ配置・CI ジョブ・運用スクリプトなど、概要設計よりも具体的な配置判断と手順が大量に発生する。これらを `DS-` と同じ体系に混入させると、論理決定と物理配置が ID 上で区別できなくなり、概要設計を読む際のノイズとなる。

本節ではこれを防ぐため、実装フェーズ専用の ID 接頭辞 `IMP-` を予約する。`IMP-` は Implementation の略で、概要設計 ID `DS-` と衝突しない。

### 書式

実装 ID は以下の書式で統一する。

```
IMP-<カテゴリ>-<サブ>-<通番>
```

- `IMP` は実装 ID であることを示す固定接頭辞。
- `<カテゴリ>` は実装章対応カテゴリ識別子（大文字英字 2〜5 文字）。
- `<サブ>` はカテゴリ内のサブ分類識別子（大文字英字 2〜7 文字）。
- `<通番>` は 3 桁の連番（`001` 始まり、欠番不可）。

書式例として、ディレクトリ設計のルートレイアウト 1 番目は `IMP-DIR-ROOT-001`、tier1 レイアウト 10 番目は `IMP-DIR-T1-010`、スパースチェックアウト運用 5 番目は `IMP-DIR-SPARSE-005` となる。

### カテゴリ識別子の割り当て（初期）

Phase 0 稟議時点では `DIR`（ディレクトリ設計）を予約する。以降、実装章の追加ごとにカテゴリを追加する運用とする。

- `DIR` : ディレクトリ設計（Directory Design、`docs/05_実装/00_ディレクトリ設計/` に対応）

### DIR カテゴリのサブ分類識別子

`IMP-DIR-*` のサブ分類識別子は以下を予約する。通番帯は予想件数に応じて連続範囲を確保する。

- `ROOT`（ルートレイアウト、001-020）
- `T1`（tier1 レイアウト、021-040）
- `T2`（tier2 レイアウト、041-055）
- `T3`（tier3 レイアウト、056-070）
- `INFRA`（infra レイアウト、071-090）
- `OPS`（deploy + ops、091-110）
- `COMM`（共通資産 tools / tests / examples / third_party、111-125）
- `SPARSE`（スパースチェックアウト運用、126-145）

### DS- との関係

`IMP-DIR-*` は設計論理を具体化する物理配置 ID である。各 `IMP-DIR-*` は対応する `DS-SW-COMP-*` または関連 ADR を本文末尾に明記し、双方向トレース関係を維持する。たとえば `IMP-DIR-T1-001`（tier1 全体配置）は `DS-SW-COMP-120` を継承し、改訂影響があれば `DS-SW-COMP-120` の改訂 PR で同時に `IMP-DIR-T1-001` も更新する。

逆方向の対応（`DS-` → `IMP-`）は [../80_トレーサビリティ/02_要件から設計へのマトリクス.md](../80_トレーサビリティ/02_要件から設計へのマトリクス.md) を実装 ID 対応に拡張して管理する（Phase 1b 以降で着手）。実装フェーズ内の双方向マトリクスは [../../05_実装/00_ディレクトリ設計/90_トレーサビリティ/](../../05_実装/00_ディレクトリ設計/90_トレーサビリティ/) で管理する。

### 通番採番ルール

実装 ID の通番採番ルールは設計 ID と同一とする。通番は 3 桁連番で、「カテゴリ + サブ」内で欠番不可、削除時は後続詰めだが一度 Product Council 承認されたものは欠番扱い、という方針を継承する。

### フェーズ属性

実装 ID も `DS-` と同じ 5 種類の確定フェーズ属性を持つ。Phase 0 稟議承認時点では `IMP-DIR-*` の Phase 1a 範囲のみ確定させる。

## 対応要件一覧

本ファイルは設計 ID 命名規則そのものの定義であり、特定要件 ID への直接対応はない。ただし、要件定義書側の要件 ID 体系（[../../03_要件定義/00_要件定義方針/](../../03_要件定義/00_要件定義方針/)）との整合を保つことを前提とするため、同カテゴリの命名方針と連動する。設計 ID 採番対象外とする。
