# 04. 設計レビュー方針

本ファイルは概要設計成果物のレビュー手続を定める。レビューは設計品質の最終関門であり、稟議通過後の稼働開始までの信頼性を担保する合意プロセスである。

## 本ファイルの位置付け

設計書のレビューは設計者の能力を査定する場ではなく、成果物が稟議・要件・構想設計の約束を満たすかを客観的に判定する場である。レビュー手続が曖昧だと、設計者が個別レビュアとの非公式合意だけで設計を固め、Product Council 承認時点で過去に合意したはずの方針と矛盾する設計が露呈する、という事態が起こる。本ファイルはこれを防ぐため、レビューの段階・観点・承認基準・差戻しフローを明示的に契約化する。

[01_概要設計プロセス.md](01_概要設計プロセス.md) で 4 段階の段階ゲート（前提凍結 / 方式ドラフト / 横串レビュー / Product Council 承認）を定めたが、本ファイルは各段階内部のレビュー実施手順を詳細化する。前者がプロセス骨格、本ファイルが実施手順、という役割分担である。

## レビュー段階別の観点と参加者

各段階で誰が何を確認するかを明確化する。観点が曖昧だと、レビュアが自分の関心事だけをチェックし、抜け漏れが発生する。

### 段階 2 方式ドラフト（自己レビュー）

設計者自身が実施する。以下のチェックリストを満たさない PR は提出しない。

- 章の全サブファイルが埋まっている（空ファイルなし）。
- 各ファイル冒頭に「本ファイルの位置付け」節を置き、解決する課題を散文で説明している。
- 散文と図が両立している（表のみ、図のみ、箇条書きのみの構成はない）。
- 要件 ID カバレッジが「対応要件一覧」節に列挙され、章全体で該当要件分類（FR / NFR など）のカバー率が 100% である。
- 設計 ID が [03_設計ID体系とトレーサビリティ.md](03_設計ID体系とトレーサビリティ.md) の命名規則に従い、欠番なし・重複なしで採番されている。
- 数値コミット（レイテンシ・スループット・容量・金額）は具体値と根拠を併記している。
- 構想設計 ADR との整合が取れている（矛盾なし、参照 ADR 番号を明記）。
- drawio 図は [../../00_format/drawio_layer_convention.md](../../00_format/drawio_layer_convention.md) に従い、白背景矩形・矢印非重なり・ラベル可読性を満たす。
- 相対リンクが有効である（リンク切れなし）。

### 段階 3 横串レビュー（ピアレビュー + アーキテクトレビュー）

ピアレビュアは同カテゴリまたは隣接カテゴリの執筆経験者が担当する。アーキテクトレビュアは tier1 アーキテクトまたはインフラアーキテクトが担当する。両者の観点を分離する。

ピアレビュアの観点は以下である。章内の論理整合性、散文の品質（初見の読み手が理解できるか）、図表の妥当性、トレーサビリティの完全性、用語の統一、ファイル間のクロスリファレンスの正確さ。ピアレビュアは章の完成度を担保する。

アーキテクトレビュアの観点は以下である。章間の設計整合性、他章との設計 ID 衝突検出、構想設計 ADR との矛盾検出、公開 API 契約の非後方互換変更の有無、非機能目標値の変更の有無、言語境界設計の整合性。アーキテクトは全体最適を担保する。

両レビューは並行実施可能だが、アーキテクトレビューで差戻しが発生した場合はピアレビューの承認を取り下げる（全体整合性の修正が章内論理に影響するため）。

### 段階 4 Product Council 承認

Product Council は企画で定義した経営・情シス・法務・開発・運用の代表者で構成する月次定例会議である。設計 PR は月次定例の 1 週間前までに段階 3 までを完了させ、以下の資料を添付して提議する。

提議資料は 3 点構成とする。第一に PR 本文として章末の「対応要件一覧」と「未解決論点」を抜粋した要約、第二に稟議で約束したフェーズスコープとの乖離有無の確認表、第三に法務責任者による OSS ライセンス確認完了の証跡（該当章のみ）である。

承認判定は多数決ではなく全員合意方式とする。反対意見がある場合は次回定例へ持越し、合意案に差戻す。Product Council の判定は PR コメントに記録し、監査対応時のエビデンスとする。

## レビュー観点チェックリスト詳細

レビュアが確認する観点を詳細に定義する。チェック項目の欠落は承認の取消事由となる。

### 散文品質チェック

各章冒頭に「本ファイルの位置付け」節があり、解決する課題を 1〜3 段落の散文で説明している。表のみの章はない。各節冒頭に導入段落があり、後続の表・図・箇条書きの文脈を散文で与えている。表セルに要件本体や長い説明を詰め込んでいない。散文と図が並立し、一方に依存する構成になっていない。

### トレーサビリティチェック

章末の「対応要件一覧」節があり、該当する要件 ID を全列挙している。要件 ID は存在確認済み（要件定義書の最新版にマッチ）である。設計 ID から要件 ID への n:n 対応が漏れなく記述されている。設計 ID は [03_設計ID体系とトレーサビリティ.md](03_設計ID体系とトレーサビリティ.md) の採番ルールに準拠する。

### 数値根拠チェック

非機能数値（レイテンシ、スループット、容量、金額、時間）はすべて具体値で記述されている。各数値に根拠（積算式・実測値・業界事例・ベンダ公表値）が併記されている。業界平均と比較している場合は出典が明記されている。数値が未決の場合は「未決」と明示し、決定期限と担当者を記載している。

### 設計整合性チェック

構想設計 ADR と矛盾する方式が記述されていない。矛盾する方式が必要な場合、新規 ADR 起票の有無が PR 本文に明記されている。公開 API 契約（tier1 11 API）の非後方互換変更がない。言語分担（Go / Rust）が守られている。Protobuf gRPC 内部通信の原則が守られている。

### 図表品質チェック

drawio 図は [../../00_format/drawio_layer_convention.md](../../00_format/drawio_layer_convention.md) の 4 レイヤ記法に従う。白背景矩形が最初の要素として配置されている。矢印が他オブジェクトと重なっていない。ラベル付き矢印のラベル幅が接続間隔の 1.5 倍以下である。凡例が右下に配置されている。SVG エクスポート済みで md に埋め込まれている。

### 参照整合性チェック

相対リンクが有効である（リンク切れなし）。リンク先のアンカー（#heading）が存在する。章内の内部参照が正確である。外部 URL 参照は URL と取得日を併記している。

## 差戻し時の手続

差戻しは否定ではなく改善要求である。以下の手続で標準化する。

レビュアは PR コメントとして差戻し事由を記述する。事由は「どの要件 ID が充足されていないか」「どの設計 ID と矛盾するか」「どの数値が根拠不足か」「どの図表規約に違反しているか」のいずれかを明示する。「もっと詳しく」「いまいち」などの曖昧な事由は禁止する。

設計者は是正コミットを積み上げる形で対応する。元のコミットを上書きする `git commit --amend` や `git rebase --interactive` は原則として禁止する（レビュア側の差分確認負担が増すため）。ただし、PR 全体のコミット数が 30 を超える場合は、段階承認後にスカッシュマージしてから再レビューを依頼する例外を許容する。

差戻し回数が同一 PR で 3 回を超えた場合は、設計者・レビュア・アーキテクトの 3 者で 1 時間の対面同期を実施する。論点を箇条書きで洗い出し、合意事項を PR コメントに記録する。対面後に残る合意困難点はアーキテクトが裁定する。裁定結果に不服がある場合は Product Council に持ち込む。

## 承認記録と監査対応

承認記録は PR マージコミットのメッセージおよび [../20_ソフトウェア方式設計/07_ソフトウェア方式設計評価レビュー.md](../20_ソフトウェア方式設計/07_ソフトウェア方式設計評価レビュー.md) 等の評価レビュー文書に残す。記録項目は承認日、承認者、レビュー観点別の承認/条件付き承認/差戻しの判定、条件付き承認の場合の条件、関連 ADR 番号、次回レビュー期日である。

監査対応時は承認記録を証跡として提示する。4 半期ごとに承認記録を集計し、差戻し傾向・頻発事由を分析してレビュー観点チェックリストを改訂する（継続的改善）。

## レビュー負荷管理

レビュアの稼働が逼迫すると、レビュー品質が低下する。以下の負荷管理策を講じる。

PR は小さく分割する。1 PR あたり 500 行以内を目安とし、これを超える場合は章内でファイル単位に分割する。章全体を 1 PR でまとめることは禁止する。

レビュア 1 人あたりの同時並行 PR 数は 3 件を上限とする。これを超える依頼は別レビュアへ振り替える。レビュア不足時はアーキテクトが一時的に代行する。

レビュー所要時間は 1 PR あたり 30 分〜2 時間を目安とする。これを超える場合は PR が大きすぎる、または争点が深すぎる可能性があり、分割・同期・裁定のいずれかを発動する。

## 対応要件一覧

本ファイルはレビュー手続そのものの定義であり、特定要件 ID への直接対応はない。ただし、要件定義書側のプロジェクト管理カテゴリ（[../../03_要件定義/70_プロジェクト管理/](../../03_要件定義/70_プロジェクト管理/)）のレビュー計画と連動する。また、[../75_事業運用方式設計/10_ガバナンス運用方式.md](../75_事業運用方式設計/10_ガバナンス運用方式.md) の Product Council 運営方針と連動する。設計 ID 採番対象外とする。
