# 06. 設計リスクレジスタ

本ファイルは概要設計工程で識別した設計固有のリスクを一元管理する。各リスクには発生確率・影響度・緩和策・所有者・期限を明示し、Product Council で定期レビューする。

## 本ファイルの位置付け

設計工程のリスクは事業リスクやプロジェクトリスクとは性質が異なる。事業リスクは「申請が通らない」「協力者が確保できない」といった外部要因主体のリスクであり、要件定義書側のリスクレジスタ（[../../03_要件定義/00_要件定義方針/07_リスクレジスタ.md](../../03_要件定義/00_要件定義方針/07_リスクレジスタ.md) 想定）で管理する。対して設計リスクは「方式選定の不確実性」「他章との整合崩れ」「要件充足の残リスク」といった設計成果物固有の内部リスクであり、設計者本人が最も早く気付ける立場にある。本ファイルはこうした設計固有リスクの可視化と追跡を担う。

リスクを見えないまま進めると、採用初期 着手後に方式選定の誤りが露呈し、手戻りで 3 か月単位の遅延が発生する事例が過去の類似案件に多数ある。逆に、リスクを事前に登録して定期レビューする運用が回れば、影響度が小さいうちに緩和策を講じられる。

## リスクの採番規則

設計リスクは `DR-<カテゴリ>-<通番>` で採番する。DR は Design Risk の略。カテゴリは以下 5 種。

- `TECH`: 技術的不確実性（OSS 成熟度・パフォーマンス未達・相性問題）
- `INTG`: 設計間整合性（章間の用語・ID・契約の矛盾）
- `COV`: 要件カバレッジ（要件の設計への落とし込み漏れ）
- `SCH`: スケジュール（設計工程の遅延・手戻り）
- `RES`: リソース（レビュア不足・設計者不足・知識偏在）

通番は 001 から連続付与する。廃止リスクは廃止日と理由を記録し、番号は再利用しない。

## リスク評価軸

各リスクは発生確率と影響度の 2 軸で評価する。

**発生確率**: 今後 6 か月以内に顕在化する確率を以下 5 段階で評価する。

- **極めて高（L5）**: 90% 以上
- **高（L4）**: 60〜89%
- **中（L3）**: 30〜59%
- **低（L2）**: 10〜29%
- **極めて低（L1）**: 10% 未満

**影響度**: 顕在化した場合の設計・実装・申請への影響を以下 5 段階で評価する。

- **致命的（I5）**: 採用再検討が必要、段階進行 3 か月以上遅延
- **重大（I4）**: 設計方式の根本見直し、段階進行 1〜3 か月遅延
- **中（I3）**: 設計の部分見直し、段階進行 2 週間〜1 か月遅延
- **小（I2）**: 設計修正は軽微、段階進行 2 週間以内の遅延
- **極小（I1）**: 設計記述の微調整のみ

リスクスコアは発生確率 × 影響度で算出し、L4 × I4（16）以上を「高リスク」、L3 × I3（9）以上を「中リスク」、それ未満を「低リスク」として区分する。高リスクは毎月、中リスクは四半期、低リスクは半期ごとに Product Council でレビューする。

## 現時点で識別しているリスク

k1s0 リリース時点（2026 年 4Q 想定）で識別している設計リスクを以下に列挙する。各リスクの詳細は散文で記述し、緩和策と責任者を明示する。

### DR-TECH-001: ZEN Engine の JDM 仕様互換性

ZEN Engine は構想設計 ADR-RULE-001 で採用したルールエンジンだが、JDM 仕様は gorules 社主導の事実上標準であり、他社実装との完全互換が保証されていない。将来的に JDM 仕様が分岐した場合、k1s0 が採用した実装と業務担当者が作成した JDM ファイルの齟齬が発生する可能性がある。

発生確率は L2（低）と評価する。採用後の運用拡大時で JDM 仕様の破壊的変更が起きる可能性はあるが、その時点での実装は既に相当数の JDM ファイルで蓄積されているため、互換レイヤを設けて対応できる。影響度は I3（中）と評価する。JDM 互換レイヤの設計追加が必要だが、設計構造を根本から変える必要はない。

緩和策として、(1) 採用 JDM 仕様の版を構想設計 ADR-RULE-001 に明記する、(2) ZEN Engine のアップデートを四半期ごとに評価する、(3) 代替ルールエンジン（Drools / Camunda DMN）への移行経路を構想設計段階で素描する、の 3 点を実施する。責任者は tier1 アーキテクト、期限は リリース時点とする。

### DR-TECH-002: Dapr Workflow と Temporal の併用複雑度

構想設計 ADR-RULE-002 の未決事項として、Dapr Workflow と Temporal の 2 種ワークフローエンジンの併用が リリース時点 で確定する見込みである。2 種併用は長期実行と短期実行の使い分けで最適化できる反面、tier2 開発者が「どちらを使うべきか」を判断する負担が生じる。判断を誤ると性能またはコストで損失が生じる。

発生確率は L4（高）と評価する。併用自体は企画で約束した運用工数 0.5 人月/年の内訳に組み込まれているが、「判断迷い」は実装段階で顕在化する可能性が高い。影響度は I3（中）と評価する。判断ガイドラインの追加で緩和可能。

緩和策として、(1) 40_制御方式設計の「ワークフロー振り分け方式」でフローチャート化した振り分けロジックを整備する、(2) Backstage Software Template から自動選定するレベルまで自動化する、(3) 振り分けが誤った場合の移植コスト（Dapr Workflow → Temporal）を事前に評価する、の 3 点を実施する。責任者は tier1 開発リーダ、期限は リリース時点とする。

### DR-TECH-003: Istio Ambient と Dapr sidecar の相互作用

構想設計 ADR-0001 で採用した Istio Ambient Mesh は、Pod 内サイドカー二重注入を回避できる利点があるが、Dapr sidecar（daprd）と共存した際のトラフィック透過捕捉と明示呼び出しの境界がグレーになる懸念がある。特に mTLS / JWT 伝搬・リトライ・サーキットブレーカが Istio と Dapr の双方で有効化できてしまい、挙動の重複または抜けが発生する可能性がある。

発生確率は L3（中）と評価する。採用後の運用拡大時 で Istio Ambient を本格導入する段階で顕在化する。影響度は I4（重大）と評価する。両者の責務境界が曖昧なまま進むと、障害時の切り分けが困難になる。

緩和策として、(1) 20_ソフトウェア方式設計の内部インタフェース方式で Istio と Dapr の責務分担を表形式で固定化する、(2) 採用後の運用拡大開始時点で統合テスト環境での動作検証を必須化する、(3) 責務境界での典型的な誤設定（二重 mTLS・二重リトライなど）を検知する静的チェックを CI に組み込む、の 3 点を実施する。責任者はインフラアーキテクト、期限は 採用後の運用拡大開始時点とする。

### DR-TECH-004: Rust 人材確保の設計依存度

構想設計 ADR-TIER1-001 で Rust を自作領域の言語として確定したが、企画書でも指摘されている通り、採用側組織の情シス部門で Rust 経験者は希少である。設計書が Rust 固有の慣習（所有権・ライフタイム・async/await）に依存する記述を多用すると、設計書の読解自体が参入障壁となり、協力者確保が遅延する。

発生確率は L3（中）と評価する。リリース時点 は起案者単独でも成立するが、運用蓄積後の協力者が Rust にキャッチアップできない場合は顕在化する。影響度は I3（中）と評価する。設計書の書き直しと補足教材の追加で緩和可能。

緩和策として、(1) 本用語集・命名規約で Rust 固有概念の前提知識を明記し、読解障壁を下げる、(2) 協力者候補向けに Rust 入門教材を リリース時点 段階で整備する（`90_knowledge/` 配下想定）、(3) 設計書内の Rust コード例を最小化し、擬似コードで意図を示す、の 3 点を実施する。責任者は tier1 開発リーダ、期限は リリース時点 着手時点とする。

### DR-INTG-001: 章間のコンポーネント粒度の齟齬

20 ソフトウェア方式設計（コンポーネント単位）と 30 共通機能方式設計（機能単位）と 40 制御方式設計（制御ロジック単位）の 3 章は同じ実装を異なる切り口で記述する。粒度を統一せず個別に書くと、同じ機能が 2 つの章で異なる名称・責務で記述される齟齬が発生する。

発生確率は L4（高）と評価する。章執筆が並行するため、執筆者間の用語統一が自然には起こらない。影響度は I3（中）と評価する。横串レビューで検出可能だが、発見が遅れると差分反映で章跨ぎ PR が肥大化する。

緩和策として、(1) 用語集（[05_用語集と命名規約.md](05_用語集と命名規約.md)）で章横断の必須用語を固定化する、(2) 横串レビューを章執筆中の早期に 2 回実施する（50% 時点・完成時点）、(3) 章間参照は設計 ID を介在させ、章跨ぎ参照を機械的に検出可能にする、の 3 点を実施する。責任者はアーキテクト、期限は各章の横串レビュー実施時点とする。

### DR-COV-001: tier1 11 API の要件充足漏れ

要件定義書で採番された FR-T1-* の 49 要件をすべて tier1 設計に落とし込む際、採番側と設計側で 1:1 対応を手作業で取ると漏れが発生する。たとえば FR-T1-STATE-005（トランザクション最大 10 操作）のような制約は、Bulk 操作方式設計で見落とされやすい。

発生確率は L3（中）と評価する。機械的チェックの整備がなければ顕在化する。影響度は I3（中）と評価する。リリース時点 着手後の実装で発覚すると差戻しコストが高い。

緩和策として、(1) [../80_トレーサビリティ/02_要件から設計へのマトリクス.md](../80_トレーサビリティ/02_要件から設計へのマトリクス.md) で要件 ID → 設計 ID の全数対応表を作成する、(2) カバレッジのスクリプトチェックを CI に組み込む、(3) 章完成時に要件分類単位（FR-T1-STATE、FR-T1-PUBSUB など）でカバレッジ 100% を確認する、の 3 点を実施する。責任者は PMO、期限は各章の段階 3 レビュー実施時点とする。

### DR-COV-002: 非機能要件の数値分配検証

tier1 API p99 500ms の内訳（業務 200ms + Dapr 80ms + OTel 20ms + 監査 50ms + NW・DB 150ms）は机上積算であり、実測で全コンポーネント合計が 500ms 以内に収まるかは未検証である。設計時点で各コンポーネントが割当値を超える設計を採用してしまうと、実装後に合算値が SLO を超える。

発生確率は L3（中）と評価する。運用蓄積後の負荷試験で初めて実測されるため、設計段階での判断は机上依存である。影響度は I4（重大）と評価する。SLO 超過は採用検討で約束した数値の未達となり、Product Council への説明責任が生じる。

緩和策として、(1) 各コンポーネントの p99 割当値を設計書に明記し、設計時点で超過する設計を却下する、(2) リリース時点 着手時点でプロトタイプベンチマークを実施し、机上積算の妥当性を早期検証する、(3) 負荷試験を リリース時点 に前倒しで組み込む、の 3 点を実施する。責任者は SRE リーダ、期限は リリース時点点とする。

### DR-SCH-001: 設計完成と採用段階開始の時間整合

k1s0 リリース時点で概要設計書の全章が完成している必要はないが、採用初期の着手時点では採用初期スコープに関わる設計（10 / 20 の核心部、30 / 40 / 50 の主要部、70 ゴールデンパス、80 トレーサビリティ）が確定している必要がある。この時間整合が取れないと、採用初期を着手できない。

発生確率は L3（中）と評価する。設計者が本業と並行する前提では工数が逼迫する。影響度は I4（重大）と評価する。採用初期の着手遅延は採用検討コミットメント違反となる。

緩和策として、(1) リリース時点準備段階から設計工程の進捗を週次で Product Council に報告する、(2) 設計リソースが不足した場合の外部協力者起用予算を採用検討に含める、(3) 優先章（10 / 20 / 30）を先行執筆し、他章は並行展開する、の 3 点を実施する。責任者は PMO、期限は採用検討通過時点とする。

### DR-RES-001: アーキテクトレビュア不足

概要設計の全章（10〜75 章）の横串レビューを実施できるアーキテクトが起案者 1 名に依存する。起案者が不在・病欠・離脱した場合、レビューが停止する。

発生確率は L2（低）と評価する。企画で約束した協力者 1 名は中堅であり、アーキテクトレビュアとしての熟練は不足する。影響度は I3（中）と評価する。起案者不在期間中は設計 PR がマージされず停滞する。

緩和策として、(1) 構想設計と概要設計を併読することでアーキテクトレビュアを段階的に育成する、(2) リリース時点 着手時点で外部アーキテクトコンサルタントとの契約を検討する、(3) 起案者不在時の代行判断ルールを本ファイルで明文化する（軽微な是正は協力者裁定、方式変更は Product Council 持越し）、の 3 点を実施する。責任者は PMO、期限は リリース時点点とする。

### DR-RES-002: レガシー .NET Framework 側の協力者不在

60_移行方式設計の記述には、既存 .NET Framework 資産の保有部門の協力が不可欠である。保有部門は各事業ドメインに散在し、情シス基盤チームからの依頼だけでは動かない。協力が得られない場合、移行方式の具体化が進まない。

発生確率は L3（中）と評価する。採用側組織でセクショナリズムが働きやすい。影響度は I3（中）と評価する。移行方式が抽象的なまま 採用側の全社展開期 に突入すると現場混乱が発生する。

緩和策として、(1) 採用検討通過時点で「移行対象候補の棚卸し」を Product Council 決議事項に含める、(2) 保有部門代表を リリース時点 着手時点で Product Council に加入させる、(3) 移行成功事例を リリース時点 段階でパイロット公開し、他部門の自発的参加を促す、の 3 点を実施する。責任者は Product Council、期限は リリース時点 着手時点とする。

## リスクレビュー頻度

高リスク（L4 × I4 以上）は毎月、中リスク（L3 × I3 以上）は四半期、低リスクは半期ごとに Product Council でレビューする。レビュー時は発生確率と影響度を再評価し、緩和策の進捗を確認する。

新規リスクが識別された場合は、発見した設計者が本ファイルに追加登録する。登録には PR を必須とし、レビュアと所有者を明示する。リスクが消滅（緩和策完了または発生確率 L1 以下）した場合は、消滅日と理由を記録して廃止扱いとする。廃止したリスクの番号は再利用しない。

## 対応要件一覧

本ファイルは設計工程固有のリスクを扱うため、特定の機能・非機能要件 ID への直接対応はない。ただし、要件定義書側のリスクレジスタ（[../../03_要件定義/00_要件定義方針/07_リスクレジスタ.md](../../03_要件定義/00_要件定義方針/07_リスクレジスタ.md) 想定）の RISK-TEC（技術リスク）・RISK-OPS（運用リスク）・RISK-HUM（人材リスク）と相互参照関係を持つ。設計 ID 採番対象外とする。
