# 08. 企画整合と Phase 展開

本ファイルは企画書（[`../../01_企画/`](../../01_企画/)）と要件定義書（[`../../03_要件定義/`](../../03_要件定義/)）で固定化された定量コミット（初期構築費・年間人件費・5 年 TCO・SLA/SLO・Phase 期間）が、概要設計のどこで実装方針化されているかを 1 対 1 で突合する。Phase 0 稟議で外部監査役から「この数値の設計側根拠は」と問われた際に、即座に設計章・設計 ID まで辿れる逆引き台帳として機能する。

## 本ファイルの位置付け

稟議で約束する定量値は「数字だけ書いてあっても信じられない」。監査役や経営層の関心は「実際に 1,850 万円で収まるのか」「本当に年 2,000 万円削減できるのか」「p99 500ms は物理的に可能か」にあり、これらの疑問に設計側が即答できないと、仮に技術的に正しい設計であっても稟議は差し戻される。

本ファイルが不在の場合、次の 3 つの事故が起こる。第一に、基本設計フェーズ移行時に工数が企画試算から ±20% を超えて乖離しても検出が遅れる。第二に、Phase 1a（MVP-0）の 3〜4 週間で何を実装完了とするかが設計 ID レベルで固定されていないため、実装チームの見積りが各自バラバラになる。第三に、稟議資料で「工数根拠は企画書 [02_開発工数試算.md](../../01_企画/04_定量試算/02_開発工数試算.md) を参照」と書いても、そこから概要設計のどの章・どの設計 ID が対応するかの逆引きが手動になる。

本ファイルは企画書の定量値 → 概要設計章・設計 ID の逆引き表を作り、Phase 別の段階的展開計画と合わせて 1 枚で可視化する。

本章は方針層のため設計 ID を採番しないが、要件 ID（P-* 系前提・C-* 系制約）との紐付けは明示する。

## 工数乖離の検出と是正

企画工数と実装工数の乖離が ±20% を超えた場合、Phase 0 稟議の前提が崩れる。乖離検出は以下 3 ゲートで行う。

- **Phase Gate レビュー**: Phase 1a / 1b / 1c 完了時に、実績工数と企画想定工数を突合。乖離が ±20% 以内なら継続、20〜50% なら Product Council で是正計画策定、50% 超なら次 Phase 着手前に稟議再承認を要する。
- **月次実績集約**: 各開発チームが Jira で記録した工数を月次で集約、企画想定の月次按分値と比較。3 か月連続で乖離 ±20% 超なら Phase Gate を前倒しレビュー。
- **四半期 Product Council**: 四半期レビューで本ファイルの「企画書の定量コミット」11 項目について設計側進捗を機械集計、稟議資料での約束が守れているかを 1 枚で報告。

是正アクションは 3 段階ある。第一に、乖離が +20〜50%（超過）の場合、Phase のスコープから一部設計 ID を次 Phase に繰り越す（優先度 SHOULD / COULD を後倒し）。第二に、+50% 超の場合、稟議の前提数値（1,850 万円 / 3.68 億 / SLA 99%）を Product Council が再策定し、取締役会に再審議を起票。第三に、−20% 以上（想定より早期完了）の場合は Phase 2 スコープの一部を前倒しし、稟議価値を上積みする。

## 要件定義書・企画書との照合運用

本ファイルは Phase 0 稟議承認時点で凍結せず、以下の運用で継続更新する。

- **企画書数値変更時**: 企画書 [04_定量試算/](../../01_企画/04_定量試算/) または [03_ロードマップと体制/](../../01_企画/03_ロードマップと体制/) の数値が改訂された場合、同じ PR で本ファイルの「企画書の定量コミット」表を更新し、影響を受ける設計 ID 群の再見積りを Phase Gate 前に実施する。
