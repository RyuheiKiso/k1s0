# 08. 企画整合と Phase 展開

本ファイルは企画書（[`../../01_企画/`](../../01_企画/)）と要件定義書（[`../../03_要件定義/`](../../03_要件定義/)）で固定化された定量コミット（初期構築費・年間人件費・5 年 TCO・SLA/SLO・Phase 期間）が、概要設計のどこで実装方針化されているかを 1 対 1 で突合する。Phase 0 稟議で外部監査役から「この数値の設計側根拠は」と問われた際に、即座に設計章・設計 ID まで辿れる逆引き台帳として機能する。

## 本ファイルの位置付け

稟議で約束する定量値は「数字だけ書いてあっても信じられない」。監査役や経営層の関心は「実際に 1,850 万円で収まるのか」「本当に年 2,000 万円削減できるのか」「p99 500ms は物理的に可能か」にあり、これらの疑問に設計側が即答できないと、仮に技術的に正しい設計であっても稟議は差し戻される。

本ファイルが不在の場合、次の 3 つの事故が起こる。第一に、基本設計フェーズ移行時に工数が企画試算から ±20% を超えて乖離しても検出が遅れる。第二に、Phase 1a（MVP-0）の 3〜4 週間で何を実装完了とするかが設計 ID レベルで固定されていないため、実装チームの見積りが各自バラバラになる。第三に、稟議資料で「工数根拠は企画書 [02_開発工数試算.md](../../01_企画/04_定量試算/02_開発工数試算.md) を参照」と書いても、そこから概要設計のどの章・どの設計 ID が対応するかの逆引きが手動になる。

本ファイルは企画書の定量値 → 概要設計章・設計 ID の逆引き表を作り、Phase 別の段階的展開計画と合わせて 1 枚で可視化する。

本章は方針層のため設計 ID を採番しないが、要件 ID（P-* 系前提・C-* 系制約）との紐付けは明示する。

企画書 [03_ロードマップと体制/00_フェーズ計画.md](../../01_企画/03_ロードマップと体制/00_フェーズ計画.md) で採択された Phase 0 / 1a（MVP-0）/ 1b（MVP-1a）/ 1c（MVP-1b）/ 2（拡張）の 5 フェーズに対して、概要設計の設計 ID 群をどのフェーズで実装完了（＝ SLO 適用開始）するかを固定化する。この表は基本設計フェーズでの実装優先順位付けと、Phase Gate レビューでの Go/NoGo 判定の双方で使用される。

Phase 1a〜1c の期間・人月は企画書で以下のように固定されている。Phase 1a は **2.5 人月 / 10 週間 / 1 FTE 集中稼働**、Phase 1b は 6.8 人月 / 3.4 か月 / 2 FTE、Phase 1c は 4.4 人月 / 2.2 か月 / 2 FTE。いずれも [04_定量試算/02_開発工数試算.md](../../01_企画/04_定量試算/02_開発工数試算.md) の積算が根拠。Phase 0 は稟議承認前の準備期間（0.5 人月）、Phase 2 は Phase 1c 完了以降の継続拡張である。

| Feature | 未着手 | 未着手 | 未着手 | 本格実装（OpenFeature + flagd）| DS-SW-EIF-070, DS-CF-FM-001 |

上表の読み方は 2 点ある。第一に、Phase 1a で本格実装される API は Log 1 つのみであり、これは企画書 [02_開発工数試算.md 第 3.1 節](../../01_企画/04_定量試算/02_開発工数試算.md) の 2.5 人月積算に直結する。Log 以外の 10 API は gRPC エンドポイント定義と最小スタブ（501 Not Implemented 相当）のみで、SLO 適用対象外として扱う。第二に、Phase 1b で Telemetry が本格実装に昇格するのは、パイロット業務 1 本の SLO 監視が Phase 1b Gate の前提であるためである。Decision / Audit-Pii / Feature の 3 API は MVP-1b までは骨格のみで、本格実装は Phase 2 まで繰り越す方針で稟議に提示する。

この貫通マトリクスは Phase Gate レビューでの Go/NoGo 判定の一次資料として使う。Phase 1a 完了時には「Log が本格実装され、残 10 API がスタブとして gRPC 応答可能」の 2 条件を Gate 通過条件とする。

## Phase Gate 判定プロトコル

| Phase Gate | 判定会議開催タイミング | 判定者 | 必須提出物 | Go 判定条件（全て充足） | NoGo 時のアクション |

## 工数乖離の検出と是正

企画工数と実装工数の乖離が ±20% を超えた場合、Phase 0 稟議の前提が崩れる。乖離検出は以下 3 ゲートで行う。

- **Phase Gate レビュー**: Phase 1a / 1b / 1c 完了時に、実績工数と企画想定工数を突合。乖離が ±20% 以内なら継続、20〜50% なら Product Council で是正計画策定、50% 超なら次 Phase 着手前に稟議再承認を要する。
- **月次実績集約**: 各開発チームが Jira で記録した工数を月次で集約、企画想定の月次按分値と比較。3 か月連続で乖離 ±20% 超なら Phase Gate を前倒しレビュー。
- **四半期 Product Council**: 四半期レビューで本ファイルの「企画書の定量コミット」11 項目について設計側進捗を機械集計、稟議資料での約束が守れているかを 1 枚で報告。

是正アクションは 3 段階ある。第一に、乖離が +20〜50%（超過）の場合、Phase のスコープから一部設計 ID を次 Phase に繰り越す（優先度 SHOULD / COULD を後倒し）。第二に、+50% 超の場合、稟議の前提数値（1,850 万円 / 3.68 億 / SLA 99%）を Product Council が再策定し、取締役会に再審議を起票。第三に、−20% 以上（想定より早期完了）の場合は Phase 2 スコープの一部を前倒しし、稟議価値を上積みする。

### 月次判定プロトコル（誰が・いつ・どのツールで）

「Phase Gate まで気付かない」事故を防ぐため、乖離の月次判定を以下のプロトコルで固定する。CFO から稟議質疑で「初期構築費が 2,000 万円を超えたら、次の判定は誰が何日で行うか」と問われた際に、単一のタイムラインで即答できる状態にする。

- **月次集約の実行者と日付**: SRE チームが毎月第 1 月曜日 09:00 までに、Jira の工数ログと企画想定の月次按分値を突合したレポートを Slack `#k1s0-cost-alert` に自動投稿する。レポートは GitHub Actions の定期ジョブで生成し、元データと突合ロジックは `infra/reporting/monthly_cost_diff.yml`（Phase 1a で雛形、Phase 1b で本番化）に保存する。
- **警告しきい値抵触時のエスカレーション**: 月次乖離が ±20% を超えた月は、当月第 2 金曜日までに Product Council **臨時レビュー**を招集する（通常の四半期レビューを待たない）。招集は起案者が Slack `@product-council` メンションで起動し、情シス部長が議長となる。議題は「乖離の原因特定 / SHOULD-COULD 後倒し候補の選定 / 次月の是正計画」の 3 点に限定する。
- **SHOULD-COULD 後倒し候補リストの事前整備**: 各 Phase の設計 ID のうち SHOULD / COULD 優先度のものを「後倒し候補ポートフォリオ」として [本書末尾 Appendix A](#appendix-a-phase-別後倒し候補ポートフォリオ) に列挙する。臨時レビュー時はこのリストから後倒し候補を選定するだけで済み、判定会議時間を最短化する（目標: 60 分以内）。
- **決定の記録と稟議資料への反映**: 後倒し決定は Product Council 議事録に記録し、次 Phase の工数見積り改訂版を「参考資料」として Phase 0 稟議ドキュメント配下に追補する。これにより稟議後に「実績が企画を外れたらどう対応したか」の証跡が残る。
- **3 か月連続の ±20% 超過時の自動アラート**: 月次レポート GitHub Actions は、過去 3 か月の乖離を集計し、3 回連続で ±20% を超えた場合に `#k1s0-exit-alert` チャネルへ発報し、[`../55_運用ライフサイクル方式設計/05_撤退戦略方式.md`](../55_運用ライフサイクル方式設計/05_撤退戦略方式.md) DS-OPS-EXIT-008（KPI 常時監視ゲート）の判定プロセスへ橋渡しする。

本プロトコルは Phase 1a 完了 Gate の「必須提出物」に「月次乖離レポート 3 本」を追加することで継続運用される。Phase 1a 時点では工数実績そのものがまだ蓄積段階にあるため、Phase 1a 中は Slack 投稿のみで Product Council 臨時招集は免除するが、Phase 1b 以降は必須とする。

## Appendix A: Phase 別後倒し候補ポートフォリオ

工数乖離検出時に即座に参照できる「後倒し可能な設計 ID」の一覧である。各 Phase のスコープから SHOULD / COULD 優先度の設計 ID のみを抜き出し、次 Phase への繰り越しが可能な順に並べる。MUST 優先度は本リストから除外する（MUST を後倒す場合は稟議再承認が必須）。本リストは Phase Gate レビュー前に四半期単位で再検証する。

### Phase 1b 後倒し候補

- **Renovate 自動更新範囲の縮小**（SHOULD）: Phase 1b では tier1 SDK・基盤ライブラリのみに限定し、tier2/tier3 側の補助ライブラリ更新は Phase 1c に繰り越し可能。
- **Testcontainers の適用範囲**（SHOULD）: Phase 1b では `k1s0.Log` + `k1s0.Telemetry` のみの統合試験に限定、残 9 API のスタブ向け Testcontainers 整備は Phase 1c 可。
- **Grafana ダッシュボードの粒度**（SHOULD）: Phase 1b の必須は tier1 API レイテンシ・ノード健全性・SLO 消費率の 3 ダッシュボードに限定、詳細な Pod メトリクスや K8s 内部指標は Phase 1c に後倒し可能。

### Phase 1c 後倒し候補

- **Kyverno ポリシーの拡張適用**（SHOULD）: Phase 1c の必須は PSS Restricted + イメージソース制限のみ、NetworkPolicy 自動生成などは Phase 2 可。
- **バックアップ自動化の完全度**（SHOULD）: Phase 1c では「手動手順 + 半自動バッチ」を許容し、完全自動化（Velero + Restic）は Phase 2 に繰り越し可能。
- **手動 PostgreSQL フェイルオーバー訓練の頻度**（SHOULD）: Phase 1c 中 1 回を MUST とし、四半期反復訓練は Phase 2 で運用化。

### Phase 2 後倒し候補（拡張スコープの一部を四半期単位で繰り越し可）

Phase 2 は継続拡張のため明示的な Gate を持たないが、四半期 Product Council での優先順位再確定時に「今四半期は Backstage を優先、Chaos Mesh は次四半期」のように SHOULD 以下を可変スコープとして扱う。ここでは代表的な可変項目のみ列挙する。

- Cosign（イメージ署名）: 必要性が顕在化するまで遅延可能（Harbor の Critical 拒否で一次防御は済んでいるため）
- Pyroscope: Tempo 稼働後の追加観測で、Phase 2 の四半期優先順位判定で後送り可。
- Argo Rollouts: 段階ロールアウトは Feature Flag で代替可能な期間は後倒し可。

本ポートフォリオの運用方針は 2 点ある。第一に、臨時 Product Council での SHOULD-COULD 後倒し選定は本リストから行うことで判定時間を 60 分以内に収める。第二に、本リストの更新は Phase Gate 通過時に実施し、実装完了した項目は除外、新規 SHOULD/COULD が発生した項目は追加する。
