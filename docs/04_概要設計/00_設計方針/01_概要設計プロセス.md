# 01. 概要設計プロセス

本ファイルは k1s0 概要設計工程そのものをどう進めるかを定める。設計書を「合意された仕様書」として成立させるための入力・出力・合意者・ADR 起票条件を固定化することが目的である。

## 本ファイルの位置付け

概要設計は要件定義と詳細設計の狭間に位置し、「何を作るか」と「どう実装するか」をつなぐ層である。この層の手続が曖昧だと、後続の詳細設計・実装工程で成果物の粒度が発散し、外部ベンダー起用時の検収基準が迷走する。実際に 採用側組織の情シス部門における過去の類似案件では、概要設計工程の入出力定義が不明確だったために、設計書がアイデア集と仕様書のどちらに分類されるかが争点となり、リリース後の開発契約締結が 3 か月遅延した事例がある。本ファイルはその発散を防ぐため、工程の各アクティビティについて入力物・出力物・合意者・完了判定基準を契約レベルで固定化する。

本ファイルは IPA 共通フレーム 2013 の「システム方式設計プロセス（6.4.3）」および「ソフトウェア方式設計プロセス（7.1.2）」をベースとしつつ、k1s0 固有の拡張章（30 共通機能方式 / 40 制御方式 / 50 非機能方式 / 55 運用ライフサイクル方式 / 70 開発者体験方式 / 75 事業運用方式）への読み替えルールを定める。すべての章執筆者は作業開始前に本ファイルを参照し、入出力と合意者を明示的に確認することが求められる。

## 概要設計の全体フロー

概要設計工程は 4 段階の段階ゲートで構成する。各ゲートは前段の合意を前提として次段に進むため、ゲート飛ばしは禁止する。

段階 1 は **前提凍結** である。要件定義書（[../../03_要件定義/](../../03_要件定義/)）と構想設計書（[../../02_構想設計/](../../02_構想設計/)）の版数を固定し、概要設計はその版数に対してのみ整合性を保証する。要件定義側で新たな要件 ID が採番された場合、または構想設計側で ADR が改訂された場合は、概要設計側で影響範囲を特定したうえで取り込むことを段階ゲートで判定する。この判定を曖昧にすると、概要設計書が「どの版の要件を満たすか」という基本契約を示せなくなる。

段階 2 は **方式ドラフト** である。各章の執筆者はまず草案を作成し、章内の完全性（要件 ID カバレッジ / 設計 ID 採番 / 図表の挿入 / 用語統一）を自己検証する。この段階では章間整合は問わず、章単独で閉じた合意対象とする。ドラフトは Markdown のプルリクエスト（以下 PR）として起票し、PR タイトルに担当章番号を前置する。

段階 3 は **横串レビュー** である。章間で共通概念（コンポーネント名・データモデル名・エラーコード・設計 ID 命名）が衝突していないかをアーキテクトがレビューする。この段階で章間の用語ゆれ・番号重複・要件 ID 衝突が検出される。是正は章跨ぎの差分 PR とし、どの章を変更したかの追跡を保持する。

段階 4 は **Product Council 承認** である。企画書で定義した Product Council（経営・情シス・法務・開発・運用の代表者）が章末の「対応要件一覧」と「未解決論点」を確認し、採用検討済み段階スコープとの乖離がないかを判定する。承認されれば章は「確定」とし、以降の変更には ADR 起票を義務化する。

## アクティビティ別の入出力

概要設計の各カテゴリ（10 システム方式 / 20 ソフトウェア方式 / 30 共通機能方式 / 40 制御方式 / 50 非機能方式 / 55 運用ライフサイクル方式 / 60 移行方式 / 70 開発者体験方式 / 75 事業運用方式 / 80 トレーサビリティ / 90 付録）は IPA 共通フレーム 2013 の該当アクティビティと以下のように対応する。入力物と出力物の対応関係を PR 起票時に明示することで、成果物の欠落を検出可能にする。

10 システム方式設計は共通フレーム 6.4.3 に対応する。入力は要件定義の非機能要件 A 可用性・B 性能拡張・F システム環境と、構想設計のアーキテクチャ章である。出力はハードウェア方式・全体分担・処理方式・データベース全体方式・システム結合テスト要件・評価レビューの 6 ファイルである。合意者はアーキテクトとインフラ責任者である。

20 ソフトウェア方式設計は共通フレーム 7.1.2 に対応する。入力は要件定義の機能要件 FR-T1-* 全 49 件と外部連携 FR-EXT-*、情報要件 FR-INFO-*、構想設計の tier1 設計章である。出力はコンポーネント方式・外部 IF 方式・内部 IF 方式・データベース最上位・利用者文書暫定版・ソフトウェア結合テスト要件・評価レビュー・API シーケンス方式である。合意者はアーキテクトと tier1 開発リーダである。

30 共通機能方式設計は共通フレームに明示アクティビティを持たないが、プラットフォーム製品としての一貫性保証のため k1s0 独自章として追加する。入力は要件定義の NFR-E セキュリティ・NFR-H 完整性・FR-T1-LOG・FR-T1-AUDIT・FR-T1-PII・FR-T1-FEATURE と、構想設計のセキュリティ関連 ADR である。出力は認証・認可・ログ・監査・分散トレーシング・メトリクス・エラーハンドリング・暗号化鍵管理・FeatureManagement の 9 ファイルである。合意者はアーキテクトとセキュリティ責任者である。

40 制御方式設計は共通フレームに明示アクティビティを持たない拡張章である。入力は要件定義の FR-T1-WORKFLOW・FR-T1-PUBSUB・NFR-A 可用性・NFR-B 性能拡張と、構想設計の Temporal / Dapr Workflow / ZEN Engine 関連 ADR である。出力は Saga・冪等性・リトライとサーキットブレーカー・非同期メッセージング・ワークフロー振り分け・タイムアウトとバックプレッシャの 6 ファイルである。合意者はアーキテクトと tier1 開発リーダである。

50 非機能方式設計は要件定義の非機能要件 A〜H 全分類を設計レベルに落とし込む章である。入力は NFR-A〜NFR-I の全 129 件と、構想設計の可観測性・セキュリティ関連 ADR である。出力は可用性 DR・性能拡張・運用・監視観測性・セキュリティ・キャパシティ計画・環境エコロジー・データ保護プライバシー・完整性コンプライアンス・脅威モデリング・SLI SLO エラーバジェットの 11 ファイルである。合意者は SRE リーダ・セキュリティ責任者・法務責任者である。

55 運用ライフサイクル方式設計は要件定義 40 章に対応し、非計画的事象とライフサイクル遷移を扱う。入力は OR-INC・OR-SUP・OR-ENV・OR-EOL・OR-EXIT の全 31 件である。出力はサポート階層・インシデント対応・環境構成管理・非推奨 EOL・撤退戦略・FMEA・負荷試験・Runbook 設計・Runbook 目録の 9 ファイルである。合意者は運用リーダとカスタマーサポート責任者である。

60 移行方式設計は共通フレーム 6.4.10 に対応する。入力は要件定義の NFR-D 移行性 11 件、MIG-* 10 件と、構想設計のレガシー連携 ADR である。出力はデータ移行・並行稼働切替・ロールバック復旧の 3 ファイルである。合意者はアーキテクトとレガシー資産保有部門代表である。

70 開発者体験方式設計は要件定義 50 章に対応する。入力は DX-TEST・DX-GP・DX-MET・DX-FM・DX-LD の全 34 件と、構想設計の CI/CD 関連 ADR である。出力は CI/CD・ローカル開発環境・DevEx 指標・Backstage ポータル詳細・テスト戦略・ゴールデンパスの 6 ファイルである。合意者は開発リーダと DevEx オーナーである。

75 事業運用方式設計は要件定義 60 章に対応する。入力は BC-UX〜BC-SUS の全 84 件である。出力は UX・i18n・オンボーディング・教育訓練・コスト管理・課金メータリング・OSS ライセンス運用・法務契約運用・サプライチェーン・ガバナンス・AI ガバナンス・分析レポーティング・サステナビリティの 13 ファイルである。合意者は事業企画・法務・調達・サステナビリティ責任者である。

80 トレーサビリティは全章横断の双方向追跡を担う。入力は 10〜75 章の全設計 ID である。出力は設計 ID 索引・要件から設計へのマトリクス・詳細設計引渡し一覧・設計間依存マトリクス・構想設計 ADR 相関マトリクスの 5 ファイルである。合意者は PMO とアーキテクトである。

90 付録は監査対応エビデンスを提供する。入力は IPA 共通フレーム 2013 原典と、構想設計 ADR 索引である。出力は共通フレーム 2013 対応表・ADR 索引・参考文献の 3 ファイルである。合意者は監査責任者とアーキテクトである。

## 構想設計との役割分担

構想設計（[../../02_構想設計/](../../02_構想設計/)）と概要設計の境界を明文化する。両者を曖昧にすると、ADR 起票や改訂の責任所在が不明確になり、採用検討通過後の改変で矛盾が蓄積する。

構想設計は「**なぜその技術を選ぶか**」を扱う層である。すなわち、候補技術の比較、却下された代替案、選定の判断基準、参照した事例を含む決定記録（ADR）がその本体である。ADR は原則として構想設計側に所属し、本概要設計からは参照のみで扱う。

概要設計は「**選んだ技術をどう組み立てるか**」を扱う層である。コンポーネント分割、インタフェース契約、データモデル、制御フロー、運用手順の仕様化がその本体である。仕様化の過程で構想設計の前提が崩れた場合は、まず構想設計側で新規 ADR を起票し、承認後に概要設計へ反映する。逆向きの経路（概要設計単独で技術選定を変更する）は禁止する。

具体例を挙げる。State API の内部実装に Valkey を採用するという決定は構想設計 ADR-DATA-004 に帰属する。概要設計では Valkey Cluster の論理トポロジ、State API から Valkey への呼び出しインタフェース契約、Valkey 障害時のフェイルオーバ手順を具体化する。仮に概要設計時点で Valkey の既知不具合が発覚し Redis Stack への差し替えが必要になった場合は、まず構想設計側で ADR-DATA-004 の改訂または新規 ADR を起票し、承認後に概要設計の該当ファイルを更新する。

## ADR 起票のトリガー条件

概要設計工程で ADR 起票を要する条件を 5 つ定める。これは構想設計 ADR の追加改訂を発動する基準であり、該当条件に抵触した場合は PR を一度停止して ADR 先行で進める。

第一に、**構想設計で選定された OSS の差し替えまたは追加**である。たとえば State Store の選定先が変更される、新たな OSS を採用する、などの場合が該当する。差し替え後の OSS でも要件 ID を充足可能であることの根拠記述が必要となるため、構想設計層で先に合意する。

第二に、**公開 API 契約の非後方互換変更**である。tier1 11 API は tier2 / tier3 への公開契約であり、署名・エラーコード・レスポンスフィールドの破壊的変更は ADR 起票を要する。破壊的変更は原則として禁止し、新バージョンの並行提供と旧バージョンの非推奨化経路で対応する。この経路自体の詳細は [../20_ソフトウェア方式設計/02_外部インタフェース方式設計/02_APIバージョニング方式.md](../20_ソフトウェア方式設計/02_外部インタフェース方式設計/02_APIバージョニング方式.md) で定義する。

第三に、**内部言語境界の変更**である。Dapr ファサード層を Go、自作領域を Rust とするハイブリッド方針は構想設計 ADR-TIER1-001 で確定しており、これを変更する場合（例: 一部を Go から Rust へ移す、逆方向、新言語の追加）は ADR 起票を要する。

第四に、**非機能目標値の緩和または強化**である。tier1 API の p99 500ms、SLA 99%、RPO 秒オーダー、RTO 4 時間などの数値を変更する場合が該当する。数値は企画書の採用検討で約束した値と連動しているため、ADR を起票したうえで企画側との同期改訂が必要となる。

第五に、**法務制約への影響**である。AGPL OSS の追加、データ取扱い方針の変更、OSS ライセンス分類の変更などの法務確認を要する場合が該当する。ADR 起票後は法務責任者との確認記録を ADR に添付する。

## 段階ゲート通過条件

4 段階ゲートそれぞれの通過条件を明示する。通過条件を満たさない場合は次段に進めず、差戻しとなる。

**段階 1 前提凍結** の通過条件は、要件定義書と構想設計書の参照版数が PR 本文に明記されていること、および前段凍結時点から版数が更新されていないことである。版数が更新されている場合は、差分の影響評価をまず実施する。

**段階 2 方式ドラフト** の通過条件は、章内の全サブファイルが埋まっていること、要件 ID カバレッジが章末の「対応要件一覧」に列挙されていること、設計 ID が [../00_設計方針/03_設計ID体系とトレーサビリティ.md](03_設計ID体系とトレーサビリティ.md) の命名規約に従って採番されていること、図表が [../../00_format/drawio_layer_convention.md](../../00_format/drawio_layer_convention.md) に従って作成されていること、用語が [../00_設計方針/05_用語集と命名規約.md](05_用語集と命名規約.md) と整合していることである。

**段階 3 横串レビュー** の通過条件は、章間で設計 ID の重複がないこと、コンポーネント名・データモデル名・エラーコードの命名が章間で一致していること、要件 ID に対する設計 ID の抜け漏れがないこと、構想設計 ADR との矛盾が検出されないことである。

**段階 4 Product Council 承認** の通過条件は、段階 3 までの全差戻しが是正済みであること、章末の「未解決論点」が合意済み方針で解決されていること、採用検討で約束した段階スコープとの乖離がないこと、法務責任者による OSS ライセンス確認が完了していることである。

## 差戻し時の手続

段階ゲートで差戻しとなった場合の手続を定める。差戻しは設計者の能力ではなく成果物の不備に対する判定であり、感情的摩擦を避けるため手続を標準化する。

差戻し時は、レビュアが PR コメントとして具体的な是正要求を記述する。是正要求は「どの要件 ID が満たされていないか」「どの設計 ID との矛盾があるか」「どの数値が根拠不足か」のいずれかを必ず含めることとし、曖昧な「もっと詳しく」は禁止する。設計者は是正コミットを積み上げる形で対応し、PR 履歴を残す。

差戻し回数が同一 PR で 3 回を超えた場合は、設計者・レビュア・アーキテクトの 3 者で対面同期を行い、論点を洗い出す。書面レビューで収束しない論点は大半がスコープ境界または用語解釈の差異であり、対面で 1 時間以内に合意に至ることが多い。対面後は合意事項を PR コメントに記録する。

## 改訂と版管理

概要設計書は要件定義書と同期する版管理を行う。主要版は k1s0 リリース時点、採用初期（デモ / パイロット / 運用体制確立）の各着手時点、採用後の運用拡大開始時点とし、各時点で章の確定事項を固定化する。版更新は要件定義書と連動させ、要件 ID の採番変更や凍結スコープの変更があった場合は対応する設計章を同期改訂する。

改訂履歴は各ファイルの末尾に「改訂履歴」節を設け、版・日付・改訂者・改訂内容・関連 PR を記録する。構想設計 ADR と連動する改訂の場合は該当 ADR 番号も記録する。

## 対応要件一覧

本ファイルは概要設計工程そのものの手続定義であり、要件定義書の特定要件 ID への直接対応はない。ただし、要件定義側のプロジェクト管理カテゴリ（[../../03_要件定義/70_プロジェクト管理/](../../03_要件定義/70_プロジェクト管理/)）との整合を保つため、該当する計画書類を間接的に参照する。段階ゲート設計は同カテゴリのレビュー計画と連動し、承認者の役割定義は同カテゴリの体制表と連動する。

設計 ID を採番する性質のファイルではないため、設計 ID 採番対象外とする。
