# 05. 用語集と命名規約

本ファイルは概要設計書固有の用語定義と、ファイル・セクション・図表・ID の命名規約を定める。章横断の用語ゆれを防ぎ、設計書の読解容易性と保守性を担保することが目的である。

## 本ファイルの位置付け

プラットフォーム製品の設計書では、同じ概念が章ごとに異なる呼び方で書かれると読み手が戸惑い、さらに検索・参照・変更追跡が困難になる。たとえば「Dapr サイドカー」「Dapr sidecar」「daprd」が同一概念を指すのに混在すると、読み手は別物か否かを都度判断する必要がある。本ファイルは、こうした用語ゆれを上流で封じ込めるため、概要設計書内で用いる主要用語と命名規約を一元化する。

要件定義書側にも用語集（[../../03_要件定義/00_要件定義方針/04_用語集.md](../../03_要件定義/00_要件定義方針/04_用語集.md) 想定）が存在するため、要件定義書で定義済みの用語は参照のみとし、本ファイルでは概要設計固有の用語（粒度・構造・手続に関する用語）を補う。二重管理を避ける。

## 用語集（概要設計固有）

### コンポーネント階層の粒度用語

設計書内でコンポーネント粒度を区別するため、以下の 4 層用語を定義する。粒度が混在すると「このコンポーネント」が指す対象が読み手によって異なる解釈になるため厳密に区別する。

**システム**: k1s0 全体を指す最上位粒度。infra / tier1 / tier2 / tier3 / operation の 5 層を含む。「k1s0 システム」「k1s0 プラットフォーム」が同義。

**サブシステム**: 1 つの tier 内部で独立した責務を持つ論理単位。tier1 内でいえば「Dapr ファサード層」「自作 Rust 領域」「Golden Path CLI」などがサブシステムに該当する。

**コンポーネント**: サブシステム内で独立した実行単位（通常は 1 つ以上の Kubernetes Deployment または StatefulSet に対応）。tier1 の COMP-T1-STATE / COMP-T1-SECRET / COMP-T1-AUDIT / COMP-T1-DECISION / COMP-T1-PII / COMP-T1-WORKFLOW の 6 コンポーネントが構想設計 ADR-TIER1-001 で確定している。

**モジュール**: コンポーネント内部の Go パッケージまたは Rust crate / module に対応する論理単位。たとえば COMP-T1-STATE 内の「Dapr アダプタ」「API ルータ」「ログ連携」はそれぞれモジュールに該当する。

### tier 関連用語

**tier**: k1s0 の層構造を指す概念。小文字で `tier1`、`tier2`、`tier3` と表記する。数字を全角にする、ハイフンを入れる（`tier-1`）、英字を大文字化する（`Tier1`）ことは禁止する。

**tier1**: プラットフォーム共通基盤層。11 API を公開し、採用側組織の情シス基盤チームが保守する。

**tier2**: ドメインロジック層。各事業ドメインチームが保守する。言語は C# / Go から自由選択。

**tier3**: エンドユーザー向け UI 層。アプリ開発チームが保守する。言語は C# / Go / TypeScript / MAUI / WinUI から自由選択。

**infra**: インフラストラクチャ層。Kubernetes / Istio / 観測基盤などを含む。インフラチームが保守する。

**operation**: 横断運用層。GitOps / 監視 / リリース管理を担当し、運用チームが横断で保守する。

### API 関連用語

**tier1 API**: tier1 が tier2 / tier3 に公開する 11 個の API の総称。Service Invoke / State / PubSub / Secrets / Binding / Workflow / Log / Telemetry / Decision / Audit-Pii / Feature の 11 種を指す。

**Audit-Pii API**: Audit API と PII API は要件定義書上は別系統だが、実装 Pod は COMP-T1-AUDIT と COMP-T1-PII の 2 Pod に分離する。概要設計書では API 単位では「Audit API」「PII API」と別称し、Pod 設計では両者を包含する総称として「監査・PII 系コンポーネント」と呼ぶ。

**公開 API**: tier1 が tier2 / tier3 に公開する外部インタフェース。Protobuf gRPC で定義し、tier2/3 言語向けクライアントを自動生成する。

**内部 API**: tier1 内部のコンポーネント間通信インタフェース。Protobuf gRPC のみ許可。外部からは不可視。

**IDL**: Interface Definition Language の略。本設計書では `.proto` ファイルを IDL と同義で用いる。

### 実装関連用語

**Dapr ファサード層**: tier1 の Go 実装部分。Dapr Building Blocks（State / PubSub / Secrets / Binding / Workflow）をラップし、tier1 公開 API を提供する。

**自作 Rust 領域**: tier1 の Rust 実装部分。ZEN Engine 統合、ハッシュチェーン監査、PII 自動マスキング、Decision API を提供する。

**Daprd**: Dapr sidecar プロセスの実行バイナリ名。本設計書では「Dapr sidecar」と「daprd」を同義で用いる。文脈上プロセスを指す場合は daprd、論理的サイドカー役を指す場合は Dapr sidecar を用いる。

**ZEN Engine**: Rust 製の JDM 決定表評価エンジン。構想設計 ADR-RULE-001 で確定。Decision API の内部実装として採用。

**JDM**: JSON Decision Model の略。ZEN Engine が評価対象とする決定表の JSON 形式定義。

### プロセス・運用関連用語

**段階**: k1s0 の運用段階区分。リリース時点（tier1 11 API 全機能本格実装、起案者の個人開発 OSS として採用初期から利用可能）/ 採用後の運用拡大時（採用側のチーム拡大に伴う基盤拡張）/ 採用側のマルチクラスタ移行時（エンドユーザー体験拡充）/ 採用側の全社展開期（業務運用）/ 採用側の全社ロールアウトの 5 段階で表現する。

**バス係数**: チームの冗長性指標。1 人欠けても稼働が維持できる人数を示す。k1s0 は起案者の個人開発 OSS であり Apache 2.0 ライセンスで取消不能に公開済みのため、採用側はバス係数 1 で開始してもフォーク・自己保守が可能な状態で運用を開始できる。

**Golden Path**: tier2 / tier3 開発者が k1s0 を利用する際の標準開発経路。Backstage Software Template により雛形生成 → CI/CD 疎通 → デプロイまで 10 分以内で完結する。

**Runbook**: 運用手順書。障害対応・定常運用・変更管理の各手順を運用者が単独で実行可能な粒度で記述した文書。採用初期 までに 15 本整備する。

**ADR**: Architecture Decision Record の略。構想設計の技術選定根拠を記録する決定記録。

**Product Council**: 月次定例の経営・情シス・法務・開発・運用代表者会議。設計 PR の最終承認者。

### 非機能関連用語

**SLA**: Service Level Agreement（サービスレベル契約）。外部契約としての可用性目標。tier1 対外 SLA は 99%。

**SLO**: Service Level Objective（サービスレベル目標）。内部運用目標として SLA より厳しく設定。tier1 内部 SLO は 99.9%。

**SLI**: Service Level Indicator（サービスレベル指標）。SLO 達成状況を測定するメトリクス。

**エラーバジェット**: SLO と実績の差分として許容される失敗量。99.9% SLO の場合、月 43 分がエラーバジェット。

**RPO**: Recovery Point Objective（目標復旧時点）。障害時に許容する最大データ損失時間。

**RTO**: Recovery Time Objective（目標復旧時間）。障害検知から完全復旧までの目標時間。

## 命名規約

### ファイル命名

概要設計書のファイル名は以下の規則で統一する。

ディレクトリ名は `XX_日本語名/` の形式を用いる。XX は 2 桁の番号、日本語名はスペースなしの連続文字列。例: `00_設計方針/`、`20_ソフトウェア方式設計/`。

ファイル名は `XX_日本語名.md` の形式を用いる。XX は 2 桁の番号、日本語名はスペースなしの連続文字列、拡張子は `.md`。例: `01_概要設計プロセス.md`。

番号と日本語名の間はアンダースコア 1 個で区切る。ハイフンや全角スペースは用いない。日本語名に英字が混じる場合は先頭大文字（パスカルケース不要）でつなげる。例: `04_Backstageポータル詳細方式.md`。

drawio ファイルは `img/NN_説明.drawio` の形式を用いる。NN は画像種別 2 桁番号、説明は英字またはローマ字での簡潔な説明。例: `img/01_tier1_component_overview.drawio`。SVG は同名で拡張子を `.svg` にする。

### セクション見出し命名

Markdown 見出しは以下の規則で統一する。

`#` は各ファイルに 1 個のみ。ファイルタイトルとして用いる。

`##` は主要セクション見出しに用いる。「本ファイルの位置付け」「対応要件一覧」などは ## で統一する。

`###` は主要セクション内の小節見出しに用いる。

`####` 以下は原則として用いない。階層が深すぎる場合は、ファイル分割を検討する。

見出しの末尾に句点（。）や読点（、）は付けない。英字と日本語の混在は半角スペースで区切る。例: `### Dapr ファサード層の責務` ではなく `### Dapr ファサード層の責務`（半角スペースが `Dapr` と `ファサード` の間に入る）。

### ID 命名

設計 ID は [03_設計ID体系とトレーサビリティ.md](03_設計ID体系とトレーサビリティ.md) で定義した `DS-<カテゴリ>-<サブ>-<通番>` 形式に従う。通番は 3 桁ゼロ埋め。

要件 ID は要件定義書側の命名に従う。`FR-T1-INVOKE-001`、`NFR-A-CONT-001`、`BC-BIL-001a` などが典型例。

ADR 番号は構想設計側の命名に従う。`ADR-0001`（4 桁通番）と `ADR-TIER1-001`（領域別 3 桁通番）の 2 系統が存在する。

### コンポーネント命名

tier1 内部コンポーネントは `COMP-T1-<機能>` 形式で統一する。`COMP-T1-STATE` / `COMP-T1-SECRET` / `COMP-T1-AUDIT` / `COMP-T1-DECISION` / `COMP-T1-PII` / `COMP-T1-WORKFLOW` の 6 コンポーネントを固定する。

モジュール名は Go は `lower_snake_case` または `lowercase`、Rust は `snake_case` を用いる。パッケージ名は言語慣習に従う。

### 図表命名

図の番号は「図 X-Y」形式で統一する。X はファイル内の連番、Y は省略（シリーズ図の場合のみ 1, 2, 3...）。例: 図 1、図 2、図 3-1、図 3-2。

表の番号は「表 X」形式で統一する。例: 表 1、表 2、表 3。

図・表の説明は図の直下、表の直上に 1 行で記述する。

### 数値単位命名

レイテンシは `ms`（ミリ秒）または `s`（秒）で統一する。マイクロ秒はミリ秒換算で統一する。

スループットは `RPS`（Requests Per Second）または `TPS`（Transactions Per Second）で統一する。

容量は `KB` / `MB` / `GB` / `TB` を 2 進接頭辞（1024 ベース）として用いる。10 進接頭辞（1000 ベース）の `KiB` などは用いない。

時間は 採用初期 の期間表記で「3-4 週間」「2.5 か月」などを用いる。日数表記が必要な場合は「21 日」のように「日」を付ける。

金額は万円単位または億円単位で表記する。円単位の細かな金額は個別数値表でのみ用いる。

### 英字と日本語の混在表記

日本語の文中に英字を混在させる場合、前後に半角スペースを入れる。例: 「Go で書かれた COMP-T1-STATE コンポーネントは」のように Go と で、で と COMP-T1-STATE、COMP-T1-STATE と コンポーネント の間に半角スペース。

句読点の直後・直前には半角スペースを入れない。例: 「Go、Rust、TypeScript」のようにカンマ後にスペースなし。

## 用語ゆれの禁止例

以下の用語ゆれを明示的に禁止する。レビュー時に検出する。

- 「Dapr サイドカー」「Dapr sidecar」「daprd」の混在 → 「Dapr sidecar」に統一（daprd はプロセスを明示的に指す文脈のみ）。
- 「tier 1」「Tier1」「tier-1」「tier1」の混在 → 「tier1」に統一。
- 「サービス」「Service」「svc」の混在 → 「サービス」に統一（Kubernetes の Service リソースを指す場合のみ `Service`）。
- 「マイクロサービス」「microservice」の混在 → 「マイクロサービス」に統一。
- 「Kubernetes」「k8s」「kube」の混在 → 「Kubernetes」に統一（ファイル名・コマンド引数のみ `k8s`）。
- 「k1s0」「K1s0」「K1S0」の混在 → 「k1s0」に統一。

## 対応要件一覧

本ファイルは用語定義そのもののため、特定要件 ID への直接対応はない。ただし、要件定義書側の用語集（[../../03_要件定義/00_要件定義方針/](../../03_要件定義/00_要件定義方針/)）と整合させることを前提とする。設計 ID 採番対象外とする。
