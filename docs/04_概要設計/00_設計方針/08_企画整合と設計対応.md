# 08. 企画整合と設計対応

本ファイルは企画書（[`../../01_企画/`](../../01_企画/)）と要件定義書（[`../../03_要件定義/`](../../03_要件定義/)）で固定化された定量コミット（初期構築費・年間人件費・5 年 TCO・SLA/SLO）が、概要設計のどこで実装方針化されているかを 1 対 1 で突合する。採用検討時に判断者から「この数値の設計側根拠は」と問われた際に、即座に設計章・設計 ID まで辿れる逆引き台帳として機能する。

## 本ファイルの位置付け

採用検討で約束する定量値は「数字だけ書いてあっても信じられない」。判断者の関心は「実際に 1,850 万円で収まるのか」「本当に年 2,000 万円削減できるのか」「p99 500ms は物理的に可能か」にあり、これらの疑問に設計側が即答できないと、仮に技術的に正しい設計であっても採用検討は通過しない。

本ファイルが不在の場合、次の事故が起こる。第一に、基本設計移行時に工数が企画試算から ±20% を超えて乖離しても検出が遅れる。第二に、採用検討資料で「工数根拠は [企画書.md](../../01_企画/企画書.md) を参照」と書いても、そこから概要設計のどの章・どの設計 ID が対応するかの逆引きが手動になる。

本ファイルは企画書の定量値 → 概要設計章・設計 ID の逆引き表として機能する。本章は方針層のため設計 ID を採番しないが、要件 ID（P-* 系前提・C-* 系制約）との紐付けは明示する。

なお、k1s0 は起案者がオフタイムで個人開発した OSS をリリース時点で全機能本格実装の状態で公開している（Apache 2.0）。採用側はリリース済み成果物に対して採用検討を行うため、設計対応は「採用後の段階的構築」ではなく「リリース時点で全設計 ID が本格実装済み」という位置付けで読まれる。

## 工数乖離の検出と是正

採用後の構築・運用工数が企画想定と乖離した場合、採用判断の前提が崩れる。乖離検出は以下のゲートで行う。

- **構築完了レビュー**: 採用側の構築完了時に、実績工数と企画想定工数を突合。乖離が ±20% 以内なら継続、20〜50% なら採用側の Product Council で是正計画策定、50% 超なら採用継続可否を再決議。
- **月次実績集約**: 採用側の各チームが工数を月次で集約、企画想定の月次按分値と比較。3 か月連続で乖離 ±20% 超なら採用継続レビューを前倒し。
- **四半期レビュー**: 四半期で本ファイルの「企画書の定量コミット」11 項目について採用後実績を機械集計、採用検討時の約束が守れているかを 1 枚で報告。

是正アクションは 3 段階。第一に、乖離が +20〜50%（超過）の場合、採用側の運用優先順位を再策定し SHOULD / COULD のスコープを後倒し。第二に、+50% 超の場合、採用検討時の前提数値（1,850 万円 / 3.68 億 / SLA 99%）を採用側で再策定し、経営層に再審議を起票。第三に、−20% 以上（想定より早期到達）の場合は採用側の運用拡張スコープを前倒しし、価値を上積みする。

## 要件定義書・企画書との照合運用

本ファイルはリリース時点で凍結せず、以下の運用で継続更新する。

- **企画書数値変更時**: 企画書 [04_定量試算/](../../01_企画/) の数値が改訂された場合、同じ PR で本ファイルの「企画書の定量コミット」表を更新し、影響を受ける設計 ID 群の再見積りを実施する。
- **要件改訂時**: 要件定義書の NFR コミット値（p99 / SLA / RPO / RTO 等）が改訂された場合、本ファイルの該当行と [80_トレーサビリティ/02_要件から設計へのマトリクス.md](../80_トレーサビリティ/02_要件から設計へのマトリクス.md) を同時更新。
