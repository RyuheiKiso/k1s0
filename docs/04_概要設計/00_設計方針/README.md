# 00. 設計方針

本カテゴリは概要設計作業の手続論を定める。設計の本体ではなく「設計をどう記述・合意・検収するか」のメタ文書である。

## 本カテゴリの位置付け

要件定義書（[../../03_要件定義/](../../03_要件定義/)）が「何を作るか」を契約文書として固定化し、構想設計書（[../../02_構想設計/](../../02_構想設計/)）が「どの技術で作るか」の根拠を固定化した。概要設計書はその両者を受けて「どうやって作るかの概要」を仕様として固定化する層である。

この層で手続論を曖昧にすると、後続の詳細設計・実装フェーズで「設計書が単なるアイデア集なのか、合意された仕様なのか」が判別できなくなり、外部ベンダー起用時の検収や、社内 ステージゲートでの承認が迷走する。本カテゴリは、この発散を防ぐための手続固定化の層である。

## 格納予定ドキュメント

以下の 8 ファイルを格納する。執筆順は番号順を推奨する。

- `01_概要設計プロセス.md` — IPA 共通フレーム 2013 の 6.4.3 / 7.1.2 をどう読み替えて適用するか、各プロセスの入力・出力・合意者を定義する。構想設計との役割分担、ADR 起票のトリガー条件もここに記述する。
- `02_設計原則と制約.md` — k1s0 が全章横断で守る設計原則（tier1 内部言語不可視、gRPC 必須、行数制限、コメント規約など）と、構想設計で決まった制約（4 レイヤ、Dapr 隠蔽、Go+Rust 分担など）を前提として列挙する。各章の執筆者はここを最初に参照する。
- `03_設計ID体系とトレーサビリティ.md` — 設計 ID の採番ルール（`DS-<カテゴリ>-<サブ>-<通番>`）、要件 ID との対応方式、変更時の追跡ルールを定義する。[../80_トレーサビリティ/](../80_トレーサビリティ/) の運用基盤となる。
- `04_設計レビュー方針.md` — レビューゲート（ドラフト → ピアレビュー → アーキテクトレビュー → Product Council 承認）、レビュー観点チェックリスト、承認基準、差戻し時の手続きを定義する。
- `05_用語集と命名規約.md` — 概要設計書固有の用語定義（コンポーネント / サブコンポーネント / モジュール の粒度差など）と、ファイル・セクション・図表・ID の命名規約を定義する。要件定義書側の用語集と重複する項目は参照のみとし、二重管理を避ける。
- `06_設計リスクレジスタ.md` — 概要設計工程で識別した設計リスク（方式選定の不確実性、他章との整合リスク、要件充足の残リスクなど）を一元管理する。要件定義書側のリスクレジスタ（[../../03_要件定義/00_要件定義方針/](../../03_要件定義/00_要件定義方針/)）が事業・プロジェクトリスクを扱うのに対し、本ファイルは設計固有リスクを扱う。各リスクには発生確率・影響度・緩和策・所有者・期限を明示する。
- `07_QAと品質保証方式.md` — 概要設計成果物そのものの品質保証方式を定義する。設計書の完全性（要件カバレッジ 100%・トレーサビリティ欠損なし）、整合性（章間の用語・ID・数値の一致）、実装可能性（構想設計 ADR との整合、選定技術での実現性）を継続検証する手続きを記述する。静的チェック（リンク切れ / ID 重複 / 未定義参照）の自動化方式も含む。
- `08_企画整合とPhase展開.md` — 企画書の定量コミット 11 項目（初期構築費 1,850 万円 / 5 年 TCO 3.68 億 / SLA 99% / p99 500ms 等）と概要設計の逆引きマップ、Phase 0/1a/1b/1c/2 のスコープ確定表、工数乖離の検出 3 ゲートを固定化する。稟議で外部監査役から定量値根拠を問われた際の即応台帳。

## 執筆方針

本カテゴリは「手続論」であり、散文と表を混在させる。特に以下を守ること。

- 各ファイル冒頭に「何を解決するためのファイルか」を段落で記述する。
- IPA 共通フレームの該当節番号を必ず併記し、監査時の追跡を可能にする。
- 構想設計 ADR や要件定義書への参照は相対パスで記載し、リンク切れを継続的に検証する。
- 手続き変更が発生した場合は、PR タイトルに `[04_概要設計/00_設計方針]` のプレフィクスを付け、Product Council レビューを必須とする。

## 他カテゴリとの関係

本カテゴリで定めた手続は 10〜90 の全カテゴリに適用される。各章の執筆者は以下の順で本カテゴリを参照すること。

1. 執筆開始前に `02_設計原則と制約.md` で守るべき原則を確認
2. 設計項目採番時に `03_設計ID体系とトレーサビリティ.md` の命名ルールを適用
3. ドラフト完成時に `04_設計レビュー方針.md` のチェックリストに沿って自己レビュー
4. 新しい用語を導入する場合は `05_用語集と命名規約.md` に追記

手続き自体の変更履歴は各ファイルのフロントマターまたは末尾の改訂履歴節に記録する。
