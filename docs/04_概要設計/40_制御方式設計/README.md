# 40. 制御方式設計

本カテゴリは tier1 の信頼性を支える制御ロジックの方式を固定化する。Saga・冪等性・リトライ・サーキットブレーカー・非同期メッセージング・ワークフロー振り分けは tier1 API の横断論点であり、共通機能とは分けて独立章として設ける。

## 本カテゴリの位置付け

tier1 は複数のバックエンド（State Store / PubSub / 外部 API / ZEN Engine / Dapr Workflow）を束ねて抽象化する層である。これらの背後にある一貫性モデル（強整合 / 結果整合）や失敗モード（ネットワーク分断 / メッセージ重複 / 部分失敗）を上位層から隠蔽しつつ、SLO を満たすためには、統一された制御方式が必須となる。各 API が独自にリトライや冪等化を実装すると、挙動が API ごとに食い違い、tier2 / tier3 の開発者体験が破綻する。

構想設計書（[../../02_構想設計/02_tier1設計/04_診断と補償/](../../02_構想設計/02_tier1設計/) 等）で補償トランザクション・ワークフロー振り分け基準の**原則**が確立されているため、本章はその原則を具体的な方式（冪等キーのスキーマ、リトライ戦略の数値、バックオフ指数、サーキットブレーカー閾値など）として固定化する役割に徹する。

## 共通機能方式設計との分担

[../30_共通機能方式設計/](../30_共通機能方式設計/) と本章は隣接しており、以下の境界で分担する。

- **30 共通機能方式**: 認証 / ログ / 監査 / 暗号 / Feature Management など「全コンポーネントが同じ実装を使う横断機能」。
- **40 制御方式（本章）**: Saga / 冪等 / リトライ / ワークフロー振り分けなど「API の信頼性と一貫性を決定する制御ロジック」。

30 は「何を使うか」、40 は「どう失敗と整合性を扱うか」の違いと捉える。

## 格納予定ドキュメント

6 ファイルを格納する。執筆順はトランザクション → 冪等 → リトライ → 非同期 → ワークフロー → タイムアウトの順を推奨する。

- `01_トランザクションとSaga方式.md` — 補償トランザクション方式、Saga オーケストレーション vs コレオグラフィの使い分け、補償失敗時の escalation 方針を記述する。要件 ID `FR-T1-WORKFLOW-*` と対応する。
- `02_冪等性設計方式.md` — 冪等キーのスキーマ（クライアント生成 / サーバー生成 / ハイブリッド）、重複検出ウィンドウ、キー保持期間、衝突時の挙動を記述する。tier1 API 全体で統一する冪等化モデルを決める最重要章。
- `03_リトライとサーキットブレーカー方式.md` — リトライ対象のエラー分類、バックオフ戦略（指数 / ジッター）、最大リトライ回数、サーキットブレーカーの状態遷移閾値とハーフオープン戦略を記述する。
- `04_非同期メッセージング方式.md` — Outbox / Inbox パターンの採用方針、at-least-once / at-most-once / exactly-once の使い分け、DLQ（Dead Letter Queue）運用、メッセージ順序保証の範囲を記述する。
- `05_ワークフロー振り分け方式.md` — ZEN Engine（決定ルール）と Dapr Workflow（長時間実行オーケストレーション）の使い分け基準、振り分け境界の判定ルール、移譲ハンドオフ方式を記述する。
- `06_タイムアウトとバックプレッシャ方式.md` — 各層のタイムアウト階層（クライアント < API < バックエンド）、バックプレッシャ伝播、レート制限、過負荷時のグレースフルデグラデーション方針を記述する。

## 執筆方針

- 各方式は具体的な数値（タイムアウト秒数、リトライ回数、バックオフ係数など）まで固定化する。抽象的な方針表明で留めないこと。
- 方式 ID は `DS-CTRL-<サブカテゴリ>-<通番>` を用いる（例: `DS-CTRL-SAGA-001`、`DS-CTRL-IDEM-001`）。
- 数値の根拠は本章に併記するか、[../50_非機能方式設計/](../50_非機能方式設計/) の SLO 計算結果を参照する形で記述する。
- 冪等性・順序保証・一貫性モデルは tier2 / tier3 開発者の開発体験に直結するため、[../20_ソフトウェア方式設計/05_利用者文書_暫定版/](../20_ソフトウェア方式設計/05_利用者文書_暫定版/) で利用者向けに再編集すること。
