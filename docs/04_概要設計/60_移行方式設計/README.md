# 60. 移行方式設計

本カテゴリは IPA 共通フレーム 2013 の **移行プロセス（6.4.10）** に対応する。レガシー .NET Framework 資産からの移行と、k1s0 自体のバージョン間移行の両方を方式として固定化する層である。

## 本カテゴリの位置付け

k1s0 の事業目的は「既存 .NET Framework 資産と共存しつつベンダーロックインを回避する」ことであり、移行方式は稟議通過の前提条件でもある。要件定義書の移行計画（[../../03_要件定義/70_プロジェクト管理/05_移行計画.md](../../03_要件定義/70_プロジェクト管理/05_移行計画.md)）が「いつ・誰が・どの順で移行するか」を計画として採番したのを受け、本章はその計画を「どの仕組みで実現するか」の方式として確定させる。

加えて、k1s0 自体のバージョン間移行（tier1 API のメジャーバージョン更新、データスキーマ変更、クラスタの入れ替え）も本章の対象とする。運用継続中にクラスタ・スキーマ・プロトコルを切り替える際のダウンタイム・データ損失・切戻し手順を方式として標準化する。

## 共通フレーム 6.4.10 との対応

共通フレーム 2013 の移行プロセスは要件定義・方式設計・並行稼働・切替・旧システム撤去の各フェーズに対応するアクティビティを規定する。本章は方式設計フェーズに相当する成果物を格納する。

## 格納予定ドキュメント

3 ファイルを格納する。執筆順は上から順で完結する。

- `01_データ移行方式.md` — 既存データの変換方式、データ検証方式（整合性チェック・サンプリング検証）、移行ウィンドウと投入順序、失敗時のリトライ方式を記述する。.NET Framework 資産特有のデータ型・文字コード・エンコーディング差異も含む。
- `02_並行稼働_切替方式.md` — パイロット稼働、段階的切替（カナリーテナント → 主力テナント）、トラフィック分配方式、旧系・新系のデータ同期方式、切替判定基準（ビジネス KPI / SLO / エラー率）を記述する。
- `03_ロールバックと復旧方式.md` — 切戻し条件、切戻し手順、切戻し時のデータ整合性保全方式、切戻し発動時の stakeholder 通知経路を記述する。切戻しはビジネス継続の最終ライフラインであり、発動基準と所要時間を明示すること。

## 執筆方針

- 移行はビジネス停止リスクを伴うため、方式は「失敗したら何を壊すか」と「どう復旧するか」を必ず併記する。
- 方式 ID は `DS-MIG-<サブカテゴリ>-<通番>` を用いる（例: `DS-MIG-DATA-001`）。
- 要件定義書の移行計画に記載された日程・体制と整合させる。日程変更があれば双方を同時改訂する。
- 切戻し手順は単独で読んで実行可能な粒度まで詳細化する。他章への参照で逃げないこと。
