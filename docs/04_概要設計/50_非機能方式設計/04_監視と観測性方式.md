# 04. 監視と観測性方式設計

本ファイルは要件定義書の観測性要件（NFR-I 系 8 要件および NFR-A / NFR-B の関連要件）を受けて、k1s0 の Three Pillars（Logs / Metrics / Traces）+ Events + Synthetic の観測基盤を確定する。SLO 監視、アラート階層、ゴールデンシグナル、外形監視、Chaos Engineering までを設計する層である。

## 本ファイルの位置付け

要件定義は「SLO 消費率を可視化する」「tier1 API 全 11 個を外形監視する」といった**観測目標**を採番した。本ファイルはその目標が「どの OSS・どの指標・どのアラート閾値で実現されるか」という**方式**に翻訳する。観測方式を曖昧にすると、インシデント時に「どこを見れば良いか分からない」「アラートが多すぎて麻痺する」の 2 つの病理が発生し、2 名運用で SLO 99.9% を守ることが物理的に不可能になる。

観測は 5 つの信号（Logs / Metrics / Traces / Events / Synthetic）を統一 UI で提供する。Grafana LGTM スタックが 4 つ（Loki / Mimir / Tempo / Pyroscope）を担い、Kubernetes Events は Grafana Alloy で収集、Synthetic は Grafana Synthetic Monitoring（もしくは k6 Synthetic）で外形監視する。

## Three Pillars + 2 拡張の方式

観測性は 5 信号の統合で成り立つ。従来の Three Pillars（Logs / Metrics / Traces）に、Kubernetes ネイティブの Events と、ユーザ視点を代替する Synthetic Monitoring を加える。

**設計項目 DS-NFR-OBS-001 Logs 方式**

- 採集: OTel Collector（DaemonSet）が各ノードの container stdout / stderr を収集、OTLP で Loki に転送。
- 保管: Loki、短期 30 日（Longhorn PVC）、長期 2 年（MinIO S3 互換）。
- ラベル: `tenant_id` / `service` / `namespace` / `pod` / `container`。High cardinality ラベル（request_id 等）は構造化ログフィールドに格納、ラベル化しない。
- クエリ: LogQL、Grafana Explorer。
- 保持期間: 業務ログ 90 日、監査ログ 7 年（後者は MinIO Object Lock バケット）。
- 確定フェーズ: Phase 1b で OTel + Loki デプロイ。

**設計項目 DS-NFR-OBS-002 Metrics 方式**

- 採集: Prometheus 互換スクレイプを OTel Collector が代行、OTLP metrics 形式で Mimir に転送。
- 保管: Mimir、13 ヶ月保持（SLO レポート 1 年分を含むため）。
- 主要メトリクス: `k1s0_api_requests_total` / `k1s0_api_request_duration_seconds`（histogram）/ `k1s0_slo_burn_rate` / `k1s0_cache_hit_ratio`。
- Kubernetes 標準メトリクス: kube-state-metrics / node_exporter / cAdvisor を Mimir に集約。
- ダッシュボード: Grafana、プロビジョニング済み（Git 管理）。
- 確定フェーズ: Phase 1b で Mimir デプロイ。

**設計項目 DS-NFR-OBS-003 Traces 方式**

- 採集: tier1 Go ファサードは OpenTelemetry Go SDK、Rust 自作領域は OpenTelemetry Rust SDK で W3C Trace Context を伝播。
- Sampling: Head-based 10%（default）、Tail-based はエラー 100% / 遅延上位 5% / 特定テナント 100%。
- 保管: Tempo、30 日保持（Longhorn）、エラー系のみ 1 年（MinIO）。
- エクスポート: OTel Collector が Tempo に OTLP 送信。
- 相関: trace_id を Logs / Metrics にも付与し、Grafana 横断探索を実現。
- 確定フェーズ: Phase 1b で Tempo + サンプリング導入。

**設計項目 DS-NFR-OBS-004 Events 方式**

- Kubernetes Events（Pod Scheduled / Pulling / OOMKilled 等）を Grafana Alloy で収集。
- Loki に `type=event` ラベルで保管、Logs と統合クエリ可能。
- Argo CD のイベント（Application Sync / Health 変化）も同経路で取り込む。
- 確定フェーズ: Phase 1b で Alloy デプロイ。

**設計項目 DS-NFR-OBS-005 Synthetic 方式**

- tier1 公開 11 API 全てに対し外形監視を実装、5 分間隔で test tenant を使ってリクエスト。
- 場所: Kubernetes 外部（別 VM もしくは別 DC）から。閉域の場合は同一クラスタ内の分離 namespace から実施し、MVP 期は後者。
- メトリクス: `synthetic_probe_success` / `synthetic_probe_duration_seconds` を Mimir に書き込み。
- アラート: 2 回連続失敗で SEV2、5 回連続失敗で SEV1。
- 確定フェーズ: Phase 1b で 5 API、Phase 1c で 11 API 全てカバー。

## Grafana Unified Observability UI

Grafana を単一窓口とし、Logs / Metrics / Traces / Events / Synthetic を同一 UI から横断探索できるようにする。オンコール SRE は Grafana を開けば全ての信号にアクセスでき、複数 UI を切り替える時間を節約する。

**設計項目 DS-NFR-OBS-006 Grafana ダッシュボード体系**

- `k1s0-overview`: クラスタ全体の健康状態、Golden Signals、SLO Burn Rate。1 枚で現状把握可能。
- `k1s0-api-<api_name>`: 11 API ごとの専用ダッシュボード。Latency / Traffic / Errors / Saturation（LTES）。
- `k1s0-slo`: SLO ダッシュボード、Error Budget 消費率と Burn Rate アラート。
- `k1s0-capacity`: ノード / Pod / ストレージの容量トレンド。
- `k1s0-security`: 認証失敗率、CVE 対応状況、OpenBao 異常アクセス。
- 全ダッシュボードは Grafana Operator で Git 管理、PR 経由で変更。
- 確定フェーズ: Phase 1b で overview + 5 API、Phase 1c で完備。

## アラート階層

アラートは SEV1 / SEV2 / SEV3 の 3 階層で運用する。階層を粗く分けないとアラート疲労が発生し、SEV1 を見逃すリスクが上がる。各階層の通知経路と応答 SLA を明確化する。

**設計項目 DS-NFR-OBS-007 アラート階層定義**

- SEV1（即時対応）: サービス全停止、データ損失、セキュリティ侵害。通知経路: PagerDuty（オンコール SRE に 1 分以内）+ Slack `#k1s0-sev1`。応答 SLA 15 分、復旧 SLA 4 時間。
- SEV2（当日対応）: 部分機能停止、SLO 消費急増、パフォーマンス劣化。通知経路: Slack `#k1s0-sev2` + メール。応答 SLA 2 時間、復旧 SLA 24 時間。
- SEV3（翌営業日対応）: 警告レベル、潜在的懸念。通知経路: Slack `#k1s0-warnings` + チケット（Backstage）。応答 SLA 1 営業日、対応は次週計画。
- 各 SEV の誤発報率目標: SEV1 は月 1 回以下、SEV2 は週 2 回以下。超過時は閾値見直し。
- 確定フェーズ: Phase 1b で SEV2/SEV3、Phase 1c で PagerDuty 連携 SEV1。

**設計項目 DS-NFR-OBS-008 主要アラートルール**

- SLO Burn Rate Fast: 1 時間で月間予算 2% 消費 → SEV1。
- SLO Burn Rate Slow: 6 時間で月間予算 5% 消費 → SEV2。
- tier1 API エラー率: 5 分平均で 1% 超過 → SEV2、5% 超過 → SEV1。
- p99 レイテンシ: 10 分平均で目標超過 → SEV2。
- ノード NotReady: 5 分継続 → SEV2、10 分継続 → SEV1。
- PVC 使用率: 80% → SEV3、90% → SEV2、95% → SEV1。
- 認証失敗率: 5 分で 100 件超過（ブルートフォース疑い）→ SEV2。
- OpenBao 権限昇格検知: 即時 SEV1。
- 確定フェーズ: Phase 1b で全ルール実装。

## Golden Signals ダッシュボード

Google SRE 本の Golden Signals（Latency / Traffic / Errors / Saturation）に沿って、tier1 11 API 個別のダッシュボードを整備する。いずれの API も 4 つの信号を 1 画面で表示し、オンコール時の最初の 15 秒でボトルネック層を特定可能にする。

**設計項目 DS-NFR-OBS-009 Golden Signals ダッシュボード要件**

- Latency: p50 / p95 / p99 / p99.9 を 1 時間 / 24 時間 / 7 日のタイムスケールで。目標値との比較を色分け表示。
- Traffic: RPS を同タイムスケールで、過去 1 週間の同時刻との比較。
- Errors: 5xx 率、4xx 率（別軸）、エラー種別の上位 10（stack trace から集約）。
- Saturation: CPU / メモリ / Pod 数 / 接続プール使用率。HPA 発動しているかの可視化。
- 確定フェーズ: Phase 1b で 5 API、Phase 1c で 11 API 全て。

## SLO ダッシュボードと Burn Rate

SLO 管理は [11_SLI_SLO_エラーバジェット方式.md](11_SLI_SLO_エラーバジェット方式.md) で詳細方式を定義する。本章はその計測実装を書く。

**設計項目 DS-NFR-OBS-010 SLO 計測実装**

- SLI 計測: Envoy Gateway の access log をストリーム処理（Vector / Alloy）し、5 分間隔で `good_requests / total_requests` を Mimir に書き込み。
- Error Budget: 30 日ローリングウィンドウ、目標 99.9% に対する消費率を Mimir recording rule で事前計算。
- Burn Rate: `(1 - availability) / (1 - SLO)` を 1h / 6h / 24h ウィンドウで計算。
- Multi Burn Rate Alert: Fast burn（1h で 14.4 倍速 = 月 2% 消費）と Slow burn（6h で 6 倍速 = 月 5% 消費）の 2 レーン。
- 確定フェーズ: Phase 1b で recording rule 実装、Phase 1c で Multi Burn Rate アラート。

## Synthetic Monitoring

Synthetic は「ユーザ視点で本当に使えるか」を計測する最後の砦である。内部メトリクスが全て正常でも、Envoy Gateway の手前で障害があればユーザは使えない。このギャップを埋めるため、外形監視を必須とする。

**設計項目 DS-NFR-OBS-011 Synthetic 実装詳細**

- テストランナー: k6 を Kubernetes CronJob として 5 分ごと起動、11 API 全てにリクエスト。
- テナント: 専用の `probe_tenant` を使用、業務データに影響しない。
- 失敗判定: HTTP 5xx、タイムアウト 10 秒、レスポンス内容が想定外。
- ダッシュボード: `k1s0-synthetic` で各 API の成功率と p99 レイテンシをトレンド表示。
- アラート: 2 回連続失敗で SEV2、5 回連続失敗で SEV1。
- 確定フェーズ: Phase 1b で 5 API、Phase 1c で 11 API。

## Chaos Engineering

Chaos Engineering は計画的に本番構成同等の環境で障害を注入し、想定した復旧動作が実際に発火するか検証する営みである。Phase 2 以降の導入とし、Phase 1c までは DR 演習（手動障害注入）で代替する。

**設計項目 DS-NFR-OBS-012 Chaos Mesh 実装計画**

- 対象範囲: 本番同等の ステージング環境。本番直接の chaos 注入は Phase 3 以降検討。
- シナリオ: (1) Pod kill（tier1 API Pod ランダム 1 台）、(2) latency inject（Valkey 応答に 100ms 遅延）、(3) network partition（ノード間通信遮断 1 分）、(4) CPU saturation（特定 Pod を 100% 占有）。
- 実行頻度: 月 1 回、平日日中に実施（障害時の人的対応もテスト対象）。
- 合格基準: tier1 API の SLO を維持、ユーザからは縮退なし（Synthetic が成功率 99% 維持）。
- 確定フェーズ: Phase 2 で Chaos Mesh 導入、Phase 2 末に月次運用化。

## ダッシュボードの運用

Grafana ダッシュボードは Git 管理・PR 経由変更のみを許可する。Grafana UI 上での直接編集は監査性が失われるため禁止する。変更は Grafana Operator の CRD として PR を起票し、Argo CD が本番反映する。

**設計項目 DS-NFR-OBS-013 ダッシュボード GitOps 運用**

- 保存場所: `platform/grafana/dashboards/` に JSON として格納。
- 編集フロー: Grafana UI でプレビュー → JSON エクスポート → PR → Argo CD sync。
- UI 直接編集の無効化: Grafana の editor role を開発者には付与しない、readonly のみ。
- 緊急時の例外: SEV1 対応中の一時ダッシュボード作成は許容、復旧後 48 時間以内に PR 化。
- 確定フェーズ: Phase 1c で運用開始。

## 観測基盤自身の観測

観測基盤（Grafana / Loki / Tempo / Mimir）自身が障害になると全ての観測が盲点になる。自己監視の仕組みを二重に張る必要がある。

**設計項目 DS-NFR-OBS-014 観測基盤自己監視**

- Mimir 自身のメトリクス: kube-state-metrics と node_exporter が Mimir に送信、Mimir 自身の稼働を Mimir 内で監視（最悪ケース検知遅延あり、下記で対策）。
- 二重化: 軽量 Prometheus（single-instance）を別 namespace で起動、LGTM スタックのダウン検知専用。
- 外部到達性: Synthetic の一部を Grafana ログインに対しても実施、Grafana 自体のダウン検知。
- PagerDuty: LGTM ダウン時の SEV1 は直接 PagerDuty 経由（Grafana の Alertmanager を介さない）。
- 確定フェーズ: Phase 1b で二重化実装。
