# 07. システム環境とエコロジー方式設計

本ファイルは要件定義書 [../../03_要件定義/30_非機能要件/F_システム環境/](../../03_要件定義/30_非機能要件/F_システム環境/) の NFR-F 系 13 要件を受けて、k1s0 が稼働する物理・OS・ランタイム環境と、電力効率・カーボンフットプリントなどのエコロジー観点の方式を確定する。オンプレミス JTC データセンターの既存資産に載せつつ、環境負荷を定量的に把握する層である。

## 本ファイルの位置付け

システム環境を曖昧にすると、物理データセンター側（空調・電源・帯域）との調整が後回しになり、増設時に設置不可の事態を招く。エコロジー観点を軽視すると、JTC 企業の ESG 報告で説明できないシステム運用に陥る。本章は物理要件と環境負荷を並行して扱い、「運用可能かつ環境配慮ができている」状態を設計で担保する。

## OS とランタイム

ベース OS はサポート期間の長い LTS を採用する。Phase 1 は Ubuntu 22.04 LTS（EoL 2027 年）、Phase 2 で 24.04 LTS（EoL 2029 年）への計画的アップグレードを組む。

**設計項目 DS-NFR-ENV-001 ベース OS**

業界ではディストリビューション選定が属人的になり、EoL 間近のバージョンを選んで 2〜3 年で基盤移行を強いられる事例が稟議段階の懸念となる。本設計では Phase 1 に Ubuntu 22.04 LTS（EoL 2027 年）を採用し Phase 2 で 24.04 LTS（EoL 2029 年）へ計画アップグレードすることで、5 年 TCO 3.68 億円の試算内に OS 移行コストを収める。崩れた場合は EoL 後のセキュリティパッチ断絶により J-SOX プログラム統制監査で重大指摘を受け、SLA 99% の維持も困難となる。

- Phase 1a〜1c: Ubuntu 22.04 LTS（Jammy Jellyfish）。
- Phase 2: Ubuntu 24.04 LTS（Noble Numbat）へ計画アップグレード、Phase 2 最初の計画停止窓で実施。
- カーネル: HWE（Hardware Enablement）kernel を採用、新しいハードウェアサポートを維持。
- セキュリティ更新: unattended-upgrades で security パッケージのみ自動適用、general パッケージは Renovate + 週次人手。
- 確定フェーズ: Phase 1a で 22.04 選定、Phase 2 で 24.04。

**設計項目 DS-NFR-ENV-002 Kubernetes ディストリビューション**

業界では OpenShift / Rancher 等の商用ディストリビューションでベンダー囲い込みとライセンス費が経営層の再検討対象になり、バージョンロック状態で脱出困難な事例が稟議で問題視される。本設計では CNCF 標準の upstream Kubernetes を kubeadm でブートストラップし、マイナー 1 個遅れを維持する運用ポリシーで情シス 2 名での完全理解を実現することで、年 3,280 万円の運用費削減に貢献する。崩れた場合は商用ライセンス費が TCO を圧迫し、稟議再審議のリスクが発生する。

- ディストリビューション: 上流 Kubernetes（upstream）を kubeadm でブートストラップ。
- バージョン: Phase 1a 1.28、Phase 1b 1.29、Phase 1c 1.30、Phase 2 以降 N-1 維持。
- アップグレードポリシー: マイナー 1 個遅れを維持、patch は月次、minor は四半期。
- kubeadm 選定理由: CNCF 標準、ベンダー独自拡張がない、情シス 2 名で完全理解可能。OpenShift や Rancher は商用ライセンスや機能過多で除外。
- 確定フェーズ: Phase 1a 1.28 デプロイ。

**設計項目 DS-NFR-ENV-003 コンテナランタイム**

Docker Engine は Kubernetes v1.24 で dockershim が廃止されたにも関わらず、現場では旧運用慣性で Docker を採用し続けるケースが多く、アップグレード不能状態に陥る事例が稟議で指摘される。本設計では containerd（systemd cgroup v2 モード）と OCI 標準 runc を採用し、イメージキャッシュ 20GB を各ノードに配置して Harbor への pull-through を集約することで、ビルド時間短縮と稼働安定性を両立する。崩れた場合は Kubernetes の新バージョン追随が不能となり SLA 99% 維持に影響する。

- containerd（systemd cgroup v2 モード）。Docker Engine は Kubernetes v1.24 で廃止済みのため採用しない。
- runc（OCI 標準）、gVisor 等のサンドボックスランタイムは Phase 2 以降に再評価。
- イメージキャッシュ: 各ノード 20 GB、pull-through で Harbor に集約。
- 確定フェーズ: Phase 1a。

**設計項目 DS-NFR-ENV-004 CNI プラグイン**

CNI 選定を誤ると Phase 2 で observability / security 強化を入れる際に全 Pod 再起動が必要になり、計画停止窓が不足する事例が業界でよく聞かれる。本設計では Phase 1 は成熟度の高い Calico（VXLAN mode）で NetworkPolicy を早期に運用し、Phase 2 で eBPF ベースの Cilium に移行する二段戦略で、Ambient mesh 拡張と同じ停止窓に合わせることで追加停止を回避する。崩れた場合は停止時間超過で月次 SLA 99% を未達となり、信用毀損に直結する。

- Phase 1a〜1b: Calico（VXLAN mode）を採用。kubeadm のデフォルトから変更しやすく、NetworkPolicy サポートが成熟。
- Phase 2: Cilium（eBPF）への移行検討。eBPF により observability と security が強化される。
- 切替タイミング: Phase 2 で Ambient mesh 拡張と同時、計画停止窓で実施。
- 確定フェーズ: Phase 1a Calico、Phase 2 Cilium 再評価。

## 物理データセンター要件

k1s0 は JTC 情シス部門の既存データセンターを前提とする。新規 DC 調達は行わない。既存設備の空きラック 1/4 列（相当）に収まるサイズで設計する。

**設計項目 DS-NFR-ENV-005 ラックスペース要件**

業界では設計段階でラック占有を想定せずに調達フェーズで「設置スペースがない」と判明し、Phase キックオフが 3〜6 か月遅延する事例が珍しくない。本設計では Phase 1b〜1c は既存 VM 基盤上の VM 3 台流用で追加物理機材ゼロ、Phase 2 でも 2U × 3 台の最小増設に収め、副 DC は Phase 3 で稟議別予算に切り分けることで、5 年 TCO 3.68 億円の試算精度を保つ。崩れた場合は設備投資超過で Product Council 再稟議となり、導入スケジュール全体が崩壊する。

- Phase 1b〜1c（中規模）: 既存 VM 基盤上の VM 3 台を流用、追加物理機材なし。
- Phase 2（大規模）: 既存 VM 基盤流用が第一候補、新規調達が必要な場合は 2U ラック 3 台程度の占有を想定。
- Phase 3（マルチ DC）: 副 DC に同等スペース確保が必要、稟議と別予算で対応。
- 確定フェーズ: Phase 0〜1c は既存流用、Phase 2 以降で別途検討。

**設計項目 DS-NFR-ENV-006 電源要件**

- VM 3 台想定: 各 VM が消費するホスト物理電力は既存 VM 基盤の按分で計算。JTC 標準 VM 1 台あたり物理 100W 相当と仮定し、3 VM で 300W、年間 2,628 kWh。
- UPS: 既存 DC の UPS 給電を前提、k1s0 専用 UPS は不要。
- 二重化: A 系 / B 系電源の既存標準に準拠。
- 確定フェーズ: Phase 1b 着工時に既存 DC の電源余裕確認。

**設計項目 DS-NFR-ENV-007 空調要件**

- 発熱量: Phase 1b 中規模で 300W 相当（電力と同等）= 1,024 BTU/h。
- 温度要件: ASHRAE TC 9.9 Class A1（18〜27℃）準拠、既存 DC の空調能力内。
- 確定フェーズ: Phase 1b で既存 DC 空調余裕確認。

**設計項目 DS-NFR-ENV-008 ネットワーク帯域要件**

- 外部（tier3 クライアント ↔ Envoy Gateway）: 想定ピーク 1500 RPS × 10 KB = 約 120 Mbps、既存社内 LAN 1 Gbps で十分。
- 内部（tier1 ↔ データ層）: 同 Pod 内および同ノード内のため VXLAN オーバヘッド含めても既存 LAN で十分。
- DC 間（Phase 2 マルチ DC）: 専用線 1 Gbps 想定、既存拠点間専用線を流用。
- 確定フェーズ: Phase 1b は既存帯域で十分、Phase 2 で拠点間確認。

## 既存資産連携

k1s0 は既存社内インフラ（DNS / NTP / 認証 / LB / ストレージ）を最大限流用する。新規調達を排し、2 名運用で立ち上げる実現性を確保する。

**設計項目 DS-NFR-ENV-009 既存社内基盤連携**

- DNS: 既存社内 DNS（BIND / AD DNS）に `*.k1s0.internal.example.jp` を追加委任、内部解決のみ。
- NTP: 既存社内 NTP（time.internal.example.jp）を chrony 参照、外部 NTP は使用しない。
- 認証: Keycloak realm `k1s0-jtc` から既存 Active Directory / LDAP に連携（LDAP broker で LDAPv3）。
- ロードバランサ: 既存 F5 / Citrix / HAProxy は使用せず、MetalLB + Envoy Gateway で完結。理由は GitOps 管理下に入る運用単純化。
- 既存 SAN ストレージ: 直接は使わず、VM のローカルディスクとして見える形で活用。
- 確定フェーズ: Phase 1b で既存基盤との接続テスト。

## エコロジー方針

k1s0 は JTC 企業の ESG 報告と整合する環境負荷の計測と削減を行う。オンプレ運用はクラウドより PUE が劣ることがあるが、ベンダー物流を排除する分のカーボン削減は期待できる。定量的評価で改善サイクルを回す。

**設計項目 DS-NFR-ENV-010 PUE 目標**

- 目標 PUE: 1.5（業界平均 1.8 より 17% 改善）。
- 計測: 既存 DC の電力監視設備と連携、k1s0 占有分を按分算出。
- 改善施策: ラック間エアフロー最適化、不要 Pod の自動 idle scale down（Phase 2）。
- 確定フェーズ: Phase 1c で計測開始、Phase 2 で改善施策着手。

**設計項目 DS-NFR-ENV-011 カーボンフットプリント計測**

- 計測ツール: Kepler（Kubernetes Efficient Power Level Exporter、Phase 3 で導入）。
- MVP 期: 電力消費量 × 系統電力原単位（日本平均 0.4 kgCO2/kWh）で月次算出。
- 公開: 年次 ESG 報告に連動、Product Council レビュー。
- 確定フェーズ: Phase 1c で MVP 計算式運用、Phase 3 で Kepler 自動化。

**設計項目 DS-NFR-ENV-012 グリーン調達**

- OSS 優先: 商用ソフトウェアのベンダー物流（CD / マニュアル / 出張）を排除することで物流カーボンを削減。
- ハードウェア調達: 新規 VM 調達時は既存基盤優先、新規物理調達時は Energy Star 認証品を選定。
- ライセンス配送: 電子配送のみ、紙媒体は拒否。
- 確定フェーズ: Phase 1a から方針適用。

**設計項目 DS-NFR-ENV-013 リソース効率化**

- HPA によるアイドル Pod 削減: 夜間・休日の tier1 Pod を最小 2 までスケールダウン（Phase 2）。
- 観測データ削減: Tempo サンプリング 10%、Loki 高頻度ログのフィルタリングでストレージ削減。
- 冗長度と効率のバランス: 可用性 DR と競合するため、効率化は可用性を損なわない範囲。
- 年間 CO2 削減目標: Phase 2 末で稼働開始時比 10% 削減。
- 確定フェーズ: Phase 2 実施。

## システム制約と前提条件の方式

要件定義書 F1 章（NFR-F-SYS 系）は「オンプレ・閉域・VM ベース」という企画段階で合意された制約を、設計段階で迂回できない形に落とし込む節である。これらは k1s0 の存在理由（データ主権・コスト予測可能性・社内規程整合）を支える最上位の境界条件であり、後続設計が勝手に越境できないよう、方式レベルで違反検知ゲートを敷設する。

**設計項目 DS-NFR-ENV-014 オンプレミス完結の強制**

クラウドマネージド依存の混入を設計段階で遮断しないと、Phase 2 以降の拡張でサーバレス・マネージド DB に安易に流出し、データ主権と予算制約が同時に崩れる。全 Pod の Egress は Envoy Gateway の allowlist（ADR-CICD-003 で定めたパッケージレジストリと CVE DB のみ）を通過し、AWS / Azure / GCP の API エンドポイント（`*.amazonaws.com`、`*.azure.com`、`*.googleapis.com`）への通信は Kyverno ポリシーで CI 段階とランタイムの二段で拒否する。Phase 1a から enforcement モード、違反は SEV2 通知。

**設計項目 DS-NFR-ENV-015 閉域ネットワーク設計**

社内 LAN / VLAN 境界を越えた Ingress が JTC 社内規程に抵触するため、MetalLB の LoadBalancer Service は内部 VIP プール `10.x.x.0/24` のみを払い出し、外部公開向けの TCP/UDP ポート解放はネットワーク部門承認のもと Envoy Gateway の特定 VIP に限定する。リモートアクセスは既存 VPN（Cisco AnyConnect 想定）経由の kubeconfig 配布のみ許容し、kubectl proxy の port-forward 外部公開は Kyverno で禁止する。Phase 1a で VLAN 分離、Phase 1b で社内規程整合確認。

**設計項目 DS-NFR-ENV-016 VM ハイパーバイザ非依存**

VMware / KVM / Hyper-V のいずれでも動作させるため、ハイパーバイザ固有のデバイス（vmware-tools、Hyper-V Integration Services）への依存は入れない。ベアメタル最適化（HugePages、NUMA pinning、SR-IOV）は採用せず、汎用 VM で成立する構成に限定する。CSI は Longhorn（ハイパーバイザ非依存）、ネットワークは Calico VXLAN で統一し、既存 VM 基盤の種類変更時も再インストールなしで移設可能な構成を Phase 1a 時点から維持する。

## システム特性の方式

要件定義書 F2 章（NFR-F-CHR 系）は「Phase 別の必要リソース規模・ストレージ・ネットワーク特性」を調達承認に耐える解像度で固める節である。曖昧なまま VM 調達稟議に進むと、過少調達で性能目標未達、過剰調達で予算超過の二択になる。本節は各 Phase の最低ラインを数値で確定する。

**設計項目 DS-NFR-ENV-017 Phase 別リソース規模**

調達部門の承認を受けるため、Phase 別の VM 構成を以下の数値で確定する。Phase 1a は VM 1 台（4 vCPU / 16 GB / 200 GB SSD）のオールインワン、Phase 1b は VM 3 台（各 4 vCPU / 16 GB / 200 GB SSD）の k8s HA 最小構成、Phase 1c は Phase 1b を継続、Phase 2 は VM 5〜10 台に拡張、Phase 3 は staging + prod の二クラスタで VM 10〜20 台、Phase 5 は管理クラスタ追加で VM 15〜30 台。各 Phase キックオフ前に実測データを反映して再確定、DS-NFR-CAP-001 のキャパシティ計画と連動する。

**設計項目 DS-NFR-ENV-018 ストレージ要件の確定**

データ層の I/O ボトルネックが p99 SLO を崩さないよう、各 OSS に要求するストレージ特性を確定する。PostgreSQL（CloudNativePG）は SSD、IOPS 1,000 以上、Phase 1b 初期 100 GB。Kafka（Strimzi）は SSD 推奨、IOPS 500 以上、50 GB / ブローカー。MinIO は RAID 構成 HDD 可、500 GB〜、バックアップ集約先。Valkey はメモリ主体 + 永続化 20 GB SSD。etcd は SSD、IOPS 500 以上、10 GB。ボリューム管理は Longhorn でレプリケーション 3。四半期ごとに実使用量をモニタリングし、容量逼迫前に増設する。

**設計項目 DS-NFR-ENV-019 ネットワーク帯域と VLAN 分離**

VLAN 後付けで手戻りが発生しないよう、Phase 1b キックオフ前に情シスインフラ運用部門と構成図合意を済ませる。社内 LAN 1 Gbps 以上（Phase 1b / 1c）、10 Gbps 推奨（Phase 2+）。VLAN は tier1 インフラ / tier2 アプリ / tier3 UI の最低 3 セグメント分離、社内 NTP / DNS 参照必須、OSS パッケージ取得は既存プロキシ経由。MetalLB は Phase 1b L2 モード、Phase 3+ で BGP モードに切替。帯域計画は DS-NFR-ENV-008 の試算（1500 RPS × 10 KB ≒ 120 Mbps）と整合。

## 適合規格の方式

要件定義書 F3 章（NFR-F-STD 系）は「独自仕様でロックインを生まない」ための業界標準準拠と、社員多様性を確保するアクセシビリティを定める節である。OSS コミュニティの恩恵を受け続けるには、標準準拠は設計段階で構造的に守る必要がある。

**設計項目 DS-NFR-ENV-020 クラウドネイティブ標準準拠**

独自拡張で OSS の入替が困難にならないよう、以下の業界標準を採用時に強制する。Kubernetes（CNCF 卒業プロジェクト）、OCI Image / Distribution Spec（コンテナイメージ）、OpenTelemetry（CNCF 計装）、SPIFFE（Istio Ambient 経由のワークロード ID）、W3C Trace Context（分散トレース伝搬）、OpenFeature（flagd 経由の Feature Flag）、SPDX / CycloneDX（SBOM）。独自拡張が必要な場合は ADR で合理性を記述、Product Council 承認を必須とする。Phase 1a から全 OSS 採用時に標準準拠を確認。

**設計項目 DS-NFR-ENV-021 tier3 向けアクセシビリティガイドライン**

k1s0 tier1 自体は UI を持たないが、tier3 アプリが WCAG 2.1 AA 相当を満たせるよう、tier3 向けガイドラインを Phase 1b で配布する。内容は (1) カラーコントラスト 4.5:1 以上、(2) キーボード操作のみでの全機能到達、(3) スクリーンリーダ対応（ARIA 属性の適切な付与）、(4) 動画コンテンツへの字幕付与、(5) フォームエラーの明示的フィードバック。管理 UI の Backstage / Grafana も同基準を目標とし、年次でアクセシビリティ監査を実施する。

## 機材設置環境条件の方式

要件定義書 F4 章（NFR-F-FAC 系）は「既存 DC 環境の範囲内で完結させる」制約を定める節である。DC 改修や専用設置工事が発生すると稟議想定の予算を大きく超えるため、既存 DC の物理条件を前提条件として固定する。

**設計項目 DS-NFR-ENV-022 既存 DC 環境への適合**

JTC 既存 DC の温湿度（ASHRAE TC 9.9 Class A1、18〜27℃）・電源（A 系 / B 系二重化、既存 UPS）・ラック U 数・ケーブリング規格を前提条件として受け入れ、専用設置工事は実施しない。新規ラック増設が必要な場合は調達部門と事前合意、稟議とは別予算で対応。Phase 1b 着工時に DC の電源余裕・空調能力・ラックスペースの 3 点を情シスファシリティ部門と確認し、既存範囲内で完結することを記録する。DS-NFR-ENV-005 / 006 / 007 と一体運用。

**設計項目 DS-NFR-ENV-023 ハードウェア耐用年数管理**

保守切れ HW での運用はリスクが高いため、VM 基盤のリース期間（通常 5 年）を考慮した更新計画を維持する。調達部門と HW 更新時期を四半期ごとに共有し、リース更新時期の 6 か月前に k1s0 側の移設準備（Ansible Playbook の再実行、データ移送計画）を着手する。保守切れ HW で稼働している VM への新規デプロイは Argo CD で block、棚卸しは DS-NFR-OPS-006（構成管理）の一環で月次実施する。

## 環境マネージメントの方式

要件定義書 F5 章（NFR-F-ECO 系）は電力効率・廃棄時データサニタイズ・カーボン報告を定める節である。ESG 報告対応は JTC 経営層の説明責任に直結し、後付け困難なため Phase 初期から計測基盤を仕込む。

**設計項目 DS-NFR-ENV-024 電力効率の最適化**

業務時間外の電力消費を最小化するため、HPA / KEDA による自動スケールインを Phase 1c から運用する。夜間（22:00〜7:00）・休日の tier1 Pod は最小 2 にスケールダウン、トラフィック復帰時は HPA が 5 分以内に復旧。Phase 3+ では未使用 VM のパワーダウン手順を整備し、電力効率指標（kWh / 業務処理）を月次記録する。可用性 SLO を損なわない範囲で効率化、SLO 抵触の兆候があれば即座にスケール閾値を緩和する。DS-NFR-ENV-013 と連動。

**設計項目 DS-NFR-ENV-025 廃棄時データサニタイズ**

HW 廃棄時の情報漏えいリスクを構造的に潰すため、NIST SP 800-88 Rev.1 Purge 相当のストレージサニタイズ手順を Runbook-OPS-DISPOSE-001 で標準化する。SSD は ATA Secure Erase または暗号消去（FDE 鍵破棄）、HDD は 3 パス上書き、故障 HW は物理破壊。廃棄証明書（シリアル番号、実施日、実施者、検証者を含む）を 7 年保管し、監査時に提示可能とする。Phase 1c から運用開始、Phase 2 以降の大規模廃棄時に外部専門業者との契約条項に組込。

**設計項目 DS-NFR-ENV-026 カーボンフットプリント年次報告（Phase 5）**

Phase 5 以降の ESG 報告対応として、DC 単位の電力消費量と CO2 換算値を年次レポートに組込む。計測は Kepler（Kubernetes Efficient Power Level Exporter）で Pod 単位の電力消費を可視化、既存 DC 電力監視設備と連携してラック按分値を算出。Grafana ダッシュボードで運用中のエネルギー効率をリアルタイム可視化し、Product Council レビューに供する。Phase 1c で MVP 計算式（電力量 × 日本平均原単位 0.4 kgCO2/kWh）を運用開始、Phase 5 で Kepler 自動化 + 外部 ESG 報告フォーマット整合。優先度は COULD だが、Phase 5 の経営層説明要求に応えるため設計下地は Phase 1c から仕込む。
