# 05. セキュリティ方式設計

本ファイルは要件定義書 [../../03_要件定義/30_非機能要件/E_セキュリティ/](../../03_要件定義/30_非機能要件/E_セキュリティ/) の NFR-E 系 26 要件を受けて、k1s0 の多層防御（Defense in Depth）を確定する。Network / Identity / Application / Data の 4 層で統制を重ね、ゼロトラストを実装する方式を設計する層である。脅威そのものの採番は [10_脅威モデリング方式.md](10_脅威モデリング方式.md) に委ね、本章は「何で守るか」の統制側を扱う。

## 本ファイルの位置付け

セキュリティは単一の仕組みで成立せず、多層で重ねることで初めて機能する。Istio Ambient の mTLS だけでは JWT 盗難を防げず、Keycloak OIDC だけでは Pod 起動時の権限昇格を防げない。本章は「Network で通信を暗号化し、Identity でアクセス主体を認証し、Application で権限を強制し、Data で漏洩しても読まれないように暗号化する」という 4 層の役割分担を確定し、層ごとの方式を個別に書き下す。

もう 1 つの重要点は、セキュリティ方式が運用コスト（2 名体制）を破綻させないことである。過剰な監査ログ、過剰なスキャン、過剰な承認フローは運用負荷で折れる。本章は統制を入れると同時に、運用負荷を月 280 時間内に収める自動化方針を明示する。

## 多層防御の全体像

Defense in Depth の 4 層を以下の方針で配置する。各層は独立して機能するが、相互に補完関係にある。

**設計項目 DS-NFR-SEC-001 多層防御の層別配置**

- Network 層: Istio Ambient mTLS（サービス間通信）、NetworkPolicy（default-deny）、Envoy Gateway + WAF（外部入口）。
- Identity 層: Keycloak OIDC（人間ユーザ）、SPIFFE SVID（ワークロード）、短命トークン（15 分）。
- Application 層: OWASP Top 10 対応、Kyverno（Pod 生成制御）、Cosign（イメージ署名検証）。
- Data 層: 保存暗号化（TDE for PG / SSE-S3 for MinIO）、通信暗号化（mTLS + TLS 1.3）、PII 自動マスキング。
- 各層の責務分離: 1 層が破られても次の層が機能する設計。たとえば mTLS 破綻時も Keycloak JWT で認証失敗、JWT 盗難時も Subject の最小権限で被害限定。
- 確定フェーズ: Phase 1a で基礎層、Phase 1b で統合、Phase 1c で全層整備。

## Network 層セキュリティ

Network 層の統制は「通信の暗号化」「通信先の制限」「外部境界での防御」の 3 要素で構成する。

**設計項目 DS-NFR-SEC-002 Istio Ambient mTLS**

- 全サービス間通信を mTLS で自動暗号化（Ambient モードで Sidecar レスに ztunnel が L4 暗号、Waypoint が L7 制御）。
- 証明書: SPIFFE SVID（SPIRE が発行）、有効期限 1 時間、自動ローテーション。
- モード: STRICT（plaintext 通信を拒否）を Phase 1c から適用。Phase 1b は PERMISSIVE で段階移行。
- AuthorizationPolicy: サービス間の呼び出し許可を SPIFFE ID 単位で明示、default deny。
- 確定フェーズ: Phase 1b で Ambient デプロイ、Phase 1c で STRICT モード。
- 参照: [../../02_構想設計/01_アーキテクチャ/](../../02_構想設計/01_アーキテクチャ/) ADR-0001。

**設計項目 DS-NFR-SEC-003 NetworkPolicy**

- デフォルト: 全 namespace で default-deny ingress / egress。
- 許可ルール: tier1 → データ層、tier2 → tier1、tier3 → Envoy Gateway、Argo CD → Git（外部）、Renovate → GitHub（外部）のみ許可。
- 実装: Calico NetworkPolicy（Phase 1b）、Cilium NetworkPolicy（Phase 2 CNI 切替時）。
- 確定フェーズ: Phase 1b で default-deny 適用。

**設計項目 DS-NFR-SEC-004 Envoy Gateway + WAF**

- 外部入口: MetalLB VIP → Envoy Gateway → Istio Ambient Waypoint → tier1。
- WAF: Coraza（OWASP ModSecurity CRS 互換、Envoy フィルタ）、OWASP CRS 3.3 ルールセット。
- TLS: TLS 1.3 強制、1.2 は Phase 1c 末で無効化。cert-manager で Let's Encrypt（外部公開部分）もしくは内部 CA。
- Rate Limit: テナントあたり 1000 RPS、IP あたり 100 RPS（超過は 429）。
- 確定フェーズ: Phase 1b で Envoy Gateway、Phase 1c で WAF + Rate Limit。

## Identity 層セキュリティ

Identity 層は「誰が」「どのワークロードが」アクセスしているかを確立する。人間とワークロードは別の仕組みで認証する。

**設計項目 DS-NFR-SEC-005 人間ユーザ認証（Keycloak OIDC）**

- IdP: Keycloak、realm は `k1s0-jtc`（本番）と `k1s0-dev`（開発）。
- フロー: Authorization Code Flow + PKCE、Password Grant は禁止。
- MFA: 管理者は TOTP 必須、一般ユーザは Phase 2 でオプション提供。
- アクセストークン: JWT、有効期限 15 分。Refresh Token 8 時間。
- セッション: IdP セッション 12 時間、アイドル 2 時間で失効。
- 確定フェーズ: Phase 1a で Keycloak、Phase 1b で MFA 管理者、Phase 2 で MFA 一般ユーザ。

**設計項目 DS-NFR-SEC-006 ワークロード認証（SPIFFE）**

- SPIRE Server: k8s-workload-attestor で Pod の ServiceAccount / namespace / image hash を検証し SVID 発行。
- SVID 形式: X.509（mTLS 用）と JWT（認可用）両対応。
- TTL: 1 時間、自動ローテーション。
- trust domain: `k1s0.internal.example.jp`。
- 確定フェーズ: Phase 1b で SPIRE デプロイ、全 tier1 Pod に SVID 配布。

**設計項目 DS-NFR-SEC-007 短命トークンと権限最小化**

- アクセストークン有効期限: 15 分。これ以上長くすると盗難時の被害が拡大する。
- Refresh Token のローテーション: 使用のたびに新トークン発行、旧トークンは即時失効。
- サービスアカウント: tier1 の各 API ごとに専用 ServiceAccount、必要最小限の RBAC のみ付与。
- 管理者権限の分離: Platform SRE / SecOps / Dev で権限分離、相互の領域は read-only。
- 確定フェーズ: Phase 1b 実装。

## Application 層セキュリティ

Application 層は「何を動かしてよいか」の統制を担う。コンテナイメージの出所、Pod の権限、コード上の脆弱性の 3 側面で統制する。

**設計項目 DS-NFR-SEC-008 OWASP Top 10 対応**

- A01 Broken Access Control: tier1 内の認可を [../30_共通機能方式設計/02_認可と権限モデル方式.md](../30_共通機能方式設計/02_認可と権限モデル方式.md) で確定、全 API に tenant_id / subject の強制バインド。
- A02 Cryptographic Failures: TLS 1.3 強制、弱い暗号スイート（RC4 / 3DES / MD5）禁止。
- A03 Injection: Prepared Statement 強制、動的 SQL 禁止、gRPC Protobuf で構造化入力。
- A04 Insecure Design: 本書および脅威モデリングで対応。
- A05 Security Misconfiguration: Kyverno で secure defaults 強制（下記）。
- A06 Vulnerable Components: Renovate + Trivy（下記）。
- A07 Identification and Authentication Failures: Keycloak / SPIFFE で対応（上記）。
- A08 Software and Data Integrity Failures: Cosign イメージ署名、GitOps 変更統制。
- A09 Security Logging and Monitoring Failures: 監査ログ [../30_共通機能方式設計/04_監査証跡方式.md](../30_共通機能方式設計/04_監査証跡方式.md) で対応。
- A10 SSRF: tier1 内からの外部 URL 参照は allowlist のみ。
- 確定フェーズ: Phase 1b〜1c で全項目カバー。

**設計項目 DS-NFR-SEC-009 Kyverno による Pod 統制**

- Pod Security Standards: Restricted レベル強制（runAsNonRoot、readOnlyRootFilesystem、allowPrivilegeEscalation=false）。
- イメージ出所制限: `harbor.k1s0.internal.example.jp/*` のみ許可、docker.io 等直接 pull は拒否。
- イメージ署名検証: Cosign 署名済みイメージのみ許可（Phase 1c で全 tier1、Phase 2 で基盤 OSS）。
- リソース制限: 全 Pod に `resources.requests` / `resources.limits` 必須、未指定 Pod は admission で拒否。
- NetworkPolicy 必須: 全 namespace に default-deny NetworkPolicy がない場合、新規 Pod を拒否。
- 確定フェーズ: Phase 1b で Restricted、Phase 1c で署名検証。

**設計項目 DS-NFR-SEC-010 Cosign イメージ署名**

- 署名対象: tier1 全 6 イメージ（Go ファサード / Rust 自作領域 / Migration / Audit / Worker / Health）。
- 署名者: CI/CD（GitHub Actions OIDC 経由、keyless 署名）。
- 検証: Kyverno cluster policy で Harbor pull 時に署名検証、未署名は拒否。
- 鍵管理: keyless（Fulcio CA）を採用、鍵ファイル管理を排除。
- 確定フェーズ: Phase 1c で tier1、Phase 2 で全 OSS。

## Data 層セキュリティ

Data 層は「漏洩しても読まれない」状態を作る最後の防波堤である。保存時・通信時・処理時の 3 場面で暗号化する。

**設計項目 DS-NFR-SEC-011 保存時暗号化（at-rest）**

- PostgreSQL: TDE は PG 標準機能に存在せず、代替として CloudNativePG の Volume レベル暗号化（LUKS）を Longhorn 側で実装。
- Valkey: AOF / RDB ファイルが Longhorn PVC 上に置かれ、同じ LUKS で暗号化。
- MinIO: SSE-S3（サーバサイド暗号化）、鍵は OpenBao Transit エンジンから払い出し。
- Kafka: ディスク暗号化は Longhorn LUKS、アプリケーションレベルの暗号化は Phase 2 で検討。
- 鍵管理: 全ての at-rest 暗号化鍵は OpenBao に集約、年 1 回ローテーション。
- 確定フェーズ: Phase 1b で Longhorn LUKS、Phase 1c で MinIO SSE-S3。

**設計項目 DS-NFR-SEC-012 通信時暗号化（in-transit）**

- 外部 ↔ Envoy Gateway: TLS 1.3（cert-manager 発行）。
- Envoy Gateway ↔ tier1: mTLS（SPIFFE SVID）。
- tier1 ↔ データ層: mTLS（Istio Ambient）。
- tier1 内部 Go ↔ Rust: loopback TCP（localhost、Pod 内通信）のため暗号化省略可、ただし Protobuf gRPC で構造化。
- Kafka クライアント: SASL/SCRAM-SHA-512 + TLS。
- 確定フェーズ: Phase 1b 完成。

**設計項目 DS-NFR-SEC-013 PII 自動マスキング**

- 検出: tier1 Audit-Pii API が入出力を走査、データ辞書で PII と定義されたフィールドを検出。
- マスキング: ログ / 監査ログ出力前に自動でマスク（例: メールは `a***@example.com`、電話番号は下 4 桁のみ表示）。
- 処理: Rust 自作領域で高速実装、p99 1ms 以内で処理。
- 例外: 業務上必要な場合はマスク解除 API で明示的に要求、要求記録を監査ログ化。
- 確定フェーズ: Phase 1c で実装。
- 参照: [08_データ保護とプライバシー方式.md](08_データ保護とプライバシー方式.md) COMP-T1-PII。

## 脆弱性管理

脆弱性管理は検出・評価・対応の 3 ステップを週次サイクルで回す。OSS の CVE は日々発表されるため、人間の意思決定が介入する量を最小化する。

**設計項目 DS-NFR-SEC-014 SBOM 生成と管理**

- 生成: Syft を CI/CD パイプラインで自動実行、全イメージに SPDX / CycloneDX の SBOM を付与。
- 保管: Harbor の artifact として保管、Pod 起動時に attestation 確認。
- 更新頻度: 新規 build 毎、週次で再生成（依存 DB 更新反映）。
- 公開: 監査対応時に Product Council 経由で提供、外部公開は Phase 3 で検討。
- 確定フェーズ: Phase 1b で CI 組込、Phase 1c で全イメージ。

**設計項目 DS-NFR-SEC-015 CVE 対応 SLA**

- Critical（CVSS 9.0+）: 48 時間以内にパッチ適用。暫定緩和策（NetworkPolicy 強化 / 機能停止）は 24 時間以内。
- High（7.0〜8.9）: 7 日以内。
- Medium（4.0〜6.9）: 30 日以内。
- Low（〜3.9）: 四半期ローリングで対応。
- 未対応で SLA 超過する場合は SEV2 エスカレーション、Product Council 承認で延長。
- 確定フェーズ: Phase 1b 運用開始。

**設計項目 DS-NFR-SEC-016 Trivy による CVE スキャン**

- Harbor Image Scan: push 時に Trivy 自動実行、Critical 検出で pull ゲート（Kyverno で Pod 起動拒否）。
- CI スキャン: CI/CD で build 前に Trivy fs scan（ソース + lockfile）。
- ランタイムスキャン: 毎週月曜に稼働中 Pod の SBOM と CVE DB 突合。
- 結果: Grafana ダッシュボード `k1s0-security` で可視化、Slack 日次レポート。
- 確定フェーズ: Phase 1b で全 tier1、Phase 1c で全 OSS。

## 侵入検知

侵入検知は Phase 2 以降に本格導入する。MVP 期は認証ログ異常・監査ログ異常の基本的な検知のみで運用する。

**設計項目 DS-NFR-SEC-017 Falco（Phase 2）**

- Falco DaemonSet として全ノードに展開、eBPF でシステムコール監視。
- ルールセット: Falco 標準ルール + k1s0 独自ルール（OpenBao root トークン使用、監査 DB 直接アクセス等）。
- アラート: SEV1 ルール発火 → PagerDuty、SEV2 → Slack。
- 偽陽性抑制: Phase 2 初期 3 ヶ月はチューニング期間、アラート Slack のみ。
- 確定フェーズ: Phase 2 導入。

**設計項目 DS-NFR-SEC-018 認証ログ異常検知**

- Keycloak ログイン失敗率: 5 分で 100 件超 → SEV2（ブルートフォース疑い）。
- 深夜アクセス: 22:00〜06:00 の管理者アクセスはログ記録、翌朝レビュー。
- 未知 IP: 登録 IP 帯外からの管理者アクセスは SEV2 + MFA 追加要求。
- 確定フェーズ: Phase 1c 実装。

**設計項目 DS-NFR-SEC-019 監査ログ異常検知（Phase 3）**

- 機械学習ベースの異常検知は Phase 3 で検討。
- MVP 期はルールベース: 通常業務時間外の PII アクセス、大量削除、権限昇格の 3 パターンのみ SEV2 検知。
- 確定フェーズ: Phase 3。

## セキュリティテストと演習

セキュリティ方式の実効性を検証するため、定期的なテストを実施する。

**設計項目 DS-NFR-SEC-020 ペネトレーションテスト**

- 頻度: Phase 1c で初回実施、その後年 1 回継続。
- 委託先: 外部の第三者機関（予算は年 300 万円を企画済）。
- スコープ: tier1 公開 API 全 11 個、Envoy Gateway、Keycloak 公開部分、Grafana 公開部分。
- 報告: CVSS スコア付きレポート、Critical / High は 30 日以内修正。
- 確定フェーズ: Phase 1c 初回。

**設計項目 DS-NFR-SEC-021 Red Team 演習（Phase 3）**

- 年 1 回、内部 Red Team もしくは外部委託で実際の攻撃シナリオを実行。
- 目的: 検知と応答の実効性評価。
- 確定フェーズ: Phase 3。

## 機密データの取扱

**設計項目 DS-NFR-SEC-022 シークレット管理**

- 集約: 全てのシークレット（DB パスワード / API 鍵 / 証明書）は OpenBao に集約。
- 配布: External Secrets Operator で Kubernetes Secret に同期、Pod env / volume mount で使用。
- ローテーション: DB パスワード 90 日、API 鍵 180 日、証明書は cert-manager / SPIRE 自動。
- アクセスログ: OpenBao の全アクセスは監査ログに記録、通常業務時間外は SEV2 検知。
- 確定フェーズ: Phase 1b で OpenBao + ESO、Phase 1c で自動ローテーション。

**設計項目 DS-NFR-SEC-023 データ分類**

- 公開: 外部公開可（API 仕様など）。
- 内部: 社員のみ（運用ドキュメント）。
- 機密: 限定メンバーのみ（シークレット / 脅威モデル）。
- 個人情報: 個人情報保護法対象データ。
- 特定個人情報: マイナンバー。
- 詳細は [08_データ保護とプライバシー方式.md](08_データ保護とプライバシー方式.md) DS-NFR-PRV-003。
- 確定フェーズ: Phase 1b 分類ポリシー確定。

- 目標レベル: Phase 2 で SLSA L2、Phase 3 で L3。
- 確定フェーズ: Phase 2。

## アクセス制御の詳細方式

要件定義 NFR-E-AC-001〜005 は「JWT 認証強制 / RBAC 3 階層 / tenant_id クレーム検証 / Secret 最小権限 / MFA」を個別採番している。DS-NFR-SEC-005〜007 で大枠を設計済みだが、具体実装（Istio AuthorizationPolicy の挙動、OpenBao Policy テンプレート、MFA の実装方式）が未確定のため本節で補強する。

**設計項目 DS-NFR-SEC-027 JWT 認証強制の Istio AuthorizationPolicy**

要件定義 NFR-E-AC-001 は「全 tier1 API 呼出で JWT 強制、未添付・署名不正・有効期限切れは Unauthorized」を要求する。tier1 内部で個別に JWT 検証を書くと API 毎に実装差異が生まれるため、Istio Ambient Waypoint で L7 検証を集約する。

- JWTRule: Keycloak JWKS URL (`https://keycloak.k1s0.internal.example.jp/realms/k1s0-jtc/protocol/openid-connect/certs`) をトラストアンカーに指定、`forwardOriginalToken=true` で tier1 にも claims を渡す。
- AuthorizationPolicy: `from.source.requestPrincipals: ["*"]` で JWT 存在を必須化、未添付は 401。
- tier1 Go ファサード: Waypoint 通過後のリクエストに `X-Jwt-Claims` ヘッダが付与される前提で Claims 取出、欠損は内部エラー扱い（Waypoint 迂回検出）。
- 匿名アクセス可視化: Grafana `k1s0-security-auth` で 401 発生率を表示、100 件/5 分で SEV3。
- 確定フェーズ: Phase 1b で Waypoint、Phase 1c で AuthorizationPolicy 本番適用。

**設計項目 DS-NFR-SEC-028 RBAC 3 階層の Keycloak Role 設計**

要件定義 NFR-E-AC-002 は Realm Role / Client Role / アプリケーションロールの 3 階層管理を要求する。階層を曖昧に運用すると、「Role が多すぎて誰も棚卸しできない」という監査指摘を招く。本項で階層の切り分け基準を固定する。

- Realm Role: プラットフォーム全体の横断権限（`platform-admin`, `platform-auditor`, `platform-operator`）。4 つのみ。
- Client Role: 個別コンポーネントの操作権限（Backstage Client には `backstage-admin`, `backstage-viewer`、OpenBao Client には `vault-reader`, `vault-writer` など）。
- アプリケーションロール: 業務サービス別の業務権限（tier2 アプリ内の `hr-manager`, `finance-approver` など）。tier1 は ZEN Engine JDM でこれらを評価。
- 棚卸し: 四半期ごとに Backstage で全 Role と保持者をレポート、SRE + SecOps がレビュー。未使用 Role は自動削除候補。
- 確定フェーズ: Phase 1b で Realm/Client 確定、Phase 1c で棚卸し運用開始。

**設計項目 DS-NFR-SEC-029 tenant_id クレーム検証と越境検知**

要件定義 NFR-E-AC-003（Runbook-SEC-TENANT-001 連動）は、tenant_id の JWT claim 検証と越境試行の検知を要求する。API レベルで tenant_id 検証を忘れると一撃でテナント境界が崩壊するため、多層の強制機構を設ける。

- 層 1: Waypoint の AuthorizationPolicy で JWT claim の tenant_id が URL Path（`/api/tenants/{tenant_id}/...`）と一致しない場合 403。
- 層 2: tier1 Go ファサードが Claims の tenant_id をコンテキストに注入、下流 Rust 自作領域は Claims ベースで処理。
- 層 3: PostgreSQL RLS (Row Level Security) で全業務テーブルに `tenant_id = current_setting('app.tenant_id')::text` ポリシー適用。
- 越境検知 (Runbook-SEC-TENANT-001): Audit ログ上で `JWT.tenant_id != resource.tenant_id` を検出した場合 SEV1、該当テナントへ通知、Audit 90 日保全。
- Chaos 試験: Phase 2 で Litmus により月次越境試行（偽造 JWT）を注入、全層で阻止することを確認。
- 確定フェーズ: Phase 1b 層 1/2、Phase 1c 層 3 + Runbook、Phase 2 Chaos。

**設計項目 DS-NFR-SEC-030 OpenBao Policy による Secret 最小権限**

要件定義 NFR-E-AC-004 は「各サービスに必要な Secret のみへのアクセス」を要求する。Policy がテナント増加で線形に増える運用コストを避けるため、テンプレート + CI 検証で統制する。

- Policy テンプレート: `path "k1s0/<tenant_id>/<service_name>/*" { capabilities = ["read"] }` の形式。tenant_id と service_name はサービス作成時に埋込。
- AppRole: 各サービスに AppRole を発行、SecretID/RoleID を SPIFFE SVID 経由で配布（OpenBao の JWT auth + SPIRE 連携）。
- CI 検証: Policy ファイルを `opa test` 相当の静的検査で「広範な path (*) 使用禁止」「read 以外の capabilities 個別承認」を強制。
- 違反時: CI で PR ブロック、SecOps レビューで例外許可のみ通過。
- 確定フェーズ: Phase 1b で Policy テンプレート、Phase 1c で CI 検証自動化。

**設計項目 DS-NFR-SEC-031 MFA（TOTP / FIDO2）の段階導入**

要件定義 NFR-E-AC-005 は「Phase 3 以降、特権操作で MFA 必須化、Phase 1c は SHOULD」を採番する。Phase 1c で SHOULD を達成するため、管理者権限保有者のみ TOTP 必須化する。

- Phase 1c（SHOULD）: `platform-admin`, `platform-auditor` 保有者は Keycloak OTP Policy 強制、TOTP（Google Authenticator / Authy 互換）。
- Phase 3（MUST）: 全 tier3 ユーザへ TOTP オプション提供、Backstage 管理機能・Argo CD 本番同期・OpenBao 秘密更新で MFA 必須。Phase 3 で FIDO2 / WebAuthn 対応。
- 例外: 緊急時のバイパス手順は Product Council Chair 承認時のみ、Audit 必須。
- 確定フェーズ: Phase 1c TOTP 管理者限定、Phase 3 全面 MFA。

## 暗号化の追加方式

要件定義 NFR-E-ENC-002 / 003 はそれぞれ「データ保管暗号化」「PII マスキング・仮名化」を採番する。DS-NFR-SEC-011 / 013 で大枠は設計済みだが、鍵ローテーション自動化と Log API 連動の具体方式が未確定のため本節で補強する。

**設計項目 DS-NFR-SEC-032 保管暗号化の鍵ローテーション自動化**

要件定義 NFR-E-ENC-002 は「鍵ローテーションを年 1 回自動化」を要求する。手動ローテーションは毎年の運用負荷（2 名体制で 40 時間/回）が発生するため、OpenBao Transit の Rewrap 機能で自動化する。

- 鍵バージョン: OpenBao Transit の `min_decryption_version` を旧バージョン維持、新バージョンで暗号化。
- Rewrap ジョブ: CronJob で週次、既存データを新鍵で再暗号化（Phase 2 で全量終了を目標）。
- 対象: MinIO SSE-S3、Longhorn LUKS、PostgreSQL pgcrypto（カラム暗号化）。
- 失敗アラート: Rewrap 失敗で SEV2、Rollback 可能なバージョンを維持。
- 確定フェーズ: Phase 1c でローテーション実施、Phase 2 で Rewrap 自動化。

**設計項目 DS-NFR-SEC-033 Log API との PII マスキング連動**

要件定義 NFR-E-ENC-003 は「Log API で `pii:true` 属性のフィールドは自動マスキング」を要求する。tier1 SDK 側でマスキングを漏らすと Loki に生 PII が流入し、削除困難になる。Log API 入口で強制マスキングする。

- 属性スキーマ: Log API の構造化ログに `attributes.pii: true` フィールド可能、マスキング対象を明示。
- マスキング: 文字列の先頭 1 文字 + 末尾ドメインのみ表示（メール例: `a***@example.com`）、電話は下 4 桁のみ。
- 強制: tier1 Log API Rust 自作領域が Log Write 前に Audit-Pii API を呼出、マスキング未適用フィールドを検知したらエラーログ + Metric。
- Loki 側: ログ Pipeline Stage で二次マスキング（保険）、正規表現で PII パターンを検出したら置換。
- 監査: 四半期で Loki 検索結果のランダムサンプリング 100 件を SecOps が目視レビュー、PII 生値検出で SEV1。
- 確定フェーズ: Phase 1c で Log API マスキング、Phase 2 で Loki Pipeline 二次対策。

## セキュリティ監視の強化方式

要件定義 NFR-E-MON-001〜004 は「全特権操作の Audit 記録 / Secret 取得監査 / 操作ログ可視化 / Feature Flag・決定表変更の監査」を採番する。既存 DS-NFR-SEC-018 は認証ログ異常検知のみを扱うため、Audit 記録と可視化の具体方式を本節で補強する。

**設計項目 DS-NFR-SEC-034 特権操作の Audit 網羅**

要件定義 NFR-E-MON-001 は 10 種類の特権操作（tier1 API 認可成否、Secret 取得、Decision 評価、Workflow 実行、Binding 外部送信、Feature Flag 変更、Backstage 管理操作、Argo CD 本番同期、OpenBao 秘密更新、k8s RBAC 変更）を全て Audit API に記録することを要求する。各ソースからの記録漏れを避けるため、Audit 記録の必須性を CI で検証する。

- tier1 API: Envoy Gateway Access Log + tier1 Audit-Pii API の両経路で記録、クロスチェックで欠損検出。
- Secret 取得: OpenBao Audit Device（file + socket）を有効化、socket が tier1 Audit API へ転送。
- Decision / Workflow: tier1 Rust 自作領域が evaluate/start 毎に Audit Record を PubSub 経由で Kafka へ publish。
- Backstage / Argo CD: Backstage のカスタム Plugin で操作を hook、Argo CD は webhook + Audit Sink で連携。
- k8s RBAC 変更: audit-policy.yaml で `RoleBinding` / `ClusterRoleBinding` の CUD を全て記録、fluentd で tier1 Audit API に転送。
- 網羅検証: 週次で Chaos Monkey 的に各操作を発火、Audit 記録有無を確認。欠損で SEV2。
- 確定フェーズ: Phase 1c で全 10 種類、Phase 2 で検証自動化。

**設計項目 DS-NFR-SEC-035 Secret 取得異常パターン検知**

要件定義 NFR-E-MON-002 は「異常パターン（短時間大量取得、通常時間外取得）を Prometheus で検知、SEV2」を要求する。Secret 漏洩の経路特定を迅速化するため検知ルールを固定する。

- 異常パターン 1: 同一 AppRole で 5 分に 100 回超の Secret read → SEV2。
- 異常パターン 2: 平日 22:00〜06:00 / 休日終日の Secret read → SEV3 + 翌朝 SRE レビュー。
- 異常パターン 3: 通常利用しない path（例: 他テナントの path）への read 試行 → SEV1。
- 誤検知対策: 初回 2 週間は Slack 通知のみでチューニング、閾値を 3 σ ベースで自動調整。
- 確定フェーズ: Phase 1c で実装、Phase 2 で機械学習ベースへ移行検討。

**設計項目 DS-NFR-SEC-036 操作ログ Grafana ダッシュボード**

要件定義 NFR-E-MON-003 は「誰がいつ何をしたか」の可視化を要求する。監査者用と運用者用の 2 本のダッシュボードを用意し、それぞれの用途に最適化する。

- `k1s0-audit-viewer`（監査者用）: tenant_id / user_id / action 別フィルタ、Audit Record を時系列表示、CSV エクスポート、90 日検索。
- `k1s0-security-ops`（運用者用）: 異常パターンのアラートヒートマップ、SEV1/2 発生数、未解決インシデント一覧。
- アクセス: Keycloak Realm Role `platform-auditor` のみ監査者用にアクセス可、運用者用は SRE / SecOps。
- データソース: tier1 Audit Pii API のクエリエンドポイント（FR-T1-AUDIT-002 連携）。
- 確定フェーズ: Phase 1c で両ダッシュボード。

**設計項目 DS-NFR-SEC-037 Feature Flag・決定表変更の二重記録**

要件定義 NFR-E-MON-004 は「Feature Flag と Decision 決定表の変更を Git commit + Audit 記録の両方で残す」を要求する。ランタイム設定の不正改ざんで稟議承認が無効化されるリスクを、両方の記録でクロスチェックする。

- Feature Flag 変更: flagd の ConfigMap 変更を Git PR 経由のみ許可、Argo CD sync 時に Audit API へ Record 送出。
- Decision 決定表変更: ZEN Engine JDM ファイルの変更も Git PR 経由、Backstage UI の編集も裏で PR 作成。
- 二重記録検証: 週次で Git コミット数と Audit Record 数を突合、不一致は SEV2。
- 変更者・承認者: Git コミッタと PR Approver を Audit Record に含める。
- 確定フェーズ: Phase 1b で Git 経由化、Phase 1c で Audit 連動。

## ネットワークセキュリティの補強方式

要件定義 NFR-E-NW-001〜004 は「外部 URL allowlist / NetworkPolicy テナント隔離 / 外部境界遮断 / レート制限と接続数上限」を採番する。DS-NFR-SEC-003/004 では default-deny と Rate Limit の枠組みのみ設計済みのため、個別シナリオの具体方式を本節で補強する。

**設計項目 DS-NFR-SEC-038 Binding API の外部 URL allowlist**

要件定義 NFR-E-NW-001 は「tenant 単位の allowlist で外部 URL アクセスを制限」を要求する。SSRF 防止と不正データ流出防止の両面で、allowlist を tenant 毎に管理する。

- 管理: Binding Component YAML（Dapr の Binding Spec）で `allowedHosts: [...]` を tenant 単位に記述、Kustomize overlay で tenant overlay 分離。
- 実行時強制: tier1 Binding API が URL の host を allowlist 検証、未許可は 403 + Audit。
- egress 制御: Istio AuthorizationPolicy の egress rules で default-deny、allowlist 分のみ許可。
- 可視化: Prometheus `k1s0_egress_bytes_total` を URL 別に集計、Grafana で tenant 別表示。
- 確定フェーズ: Phase 1b で allowlist、Phase 1c で可視化。

**設計項目 DS-NFR-SEC-039 NetworkPolicy テナント隔離と Chaos 検証**

要件定義 NFR-E-NW-002 は「NetworkPolicy default-deny + Istio SPIFFE ID 検証」と「Chaos Engineering 検証」を要求する。隔離ポリシーは「書いたが効いていない」状態を検出しづらいため、Chaos での継続検証が必須。

- Kyverno Policy: tenant-id ラベル未付 Pod の生成を block、違反 PR は自動拒否。
- NetworkPolicy: 全 namespace に default-deny、tier1 → データ層 / tier2 → tier1 / tier3 → Gateway のみ許可。
- Istio AuthorizationPolicy: SPIFFE ID で呼出元サービス検証、namespace を越えた偽装を阻止。
- Chaos 検証: Phase 1c で Litmus により `tenant-a` Pod から `tenant-b` への通信を試行、全て block されることを確認。
- 確定フェーズ: Phase 1b で隔離、Phase 1c で Chaos 検証。

**設計項目 DS-NFR-SEC-040 AGPL OSS 外部境界遮断**

要件定義 NFR-E-NW-003 は「AGPL OSS（Grafana / Loki / Tempo / Pyroscope / MinIO / Renovate）の外部ネットワーク遮断」を要求する。AGPL 義務の回避（内部限定提供）を継続的に担保する。

- Istio Gateway: AGPL OSS を配置する namespace（`observability`, `infra-storage`）からの egress を全面拒否、特定の内部向け endpoint のみ allow。
- Ingress: 外部からのアクセスは Envoy Gateway の hostname allowlist で `*.k1s0.internal.example.jp` のみ受付、外部ドメインからは 404。
- プロセス分離: AGPL OSS は k1s0 本体と同 Pod に配置禁止（Kyverno で enforce）。
- 年次レビュー: 法務部門が AGPL OSS の境界遮断を年次確認、証跡として ADR-LEGAL-NNN 残し。
- 確定フェーズ: Phase 1a で境界遮断、Phase 1c で法務レビュー。

**設計項目 DS-NFR-SEC-041 レート制限と接続数上限（DoS 対策 3 層）**

要件定義 NFR-E-NW-004 は「外部境界 + サービス間 + バックエンド」の 3 層 DoS 対策を要求する。DS-NFR-SEC-004 は外部境界のみ扱うため、サービス間・バックエンド層を本項で補強する。

- 外部境界: Envoy Gateway + WAF、tenant 単位 1000 RPS、IP 単位 100 RPS（DS-NFR-SEC-004 で既に設計）。Slowloris 検知で idle 10 秒切断。
- サービス間: Istio AuthorizationPolicy の rateLimit で 呼出元 SPIFFE ID 単位 500 RPS、Circuit Breaker で連続 5 エラーで 30 秒遮断。
- バックエンド: Valkey max clients 10000、PostgreSQL max_connections 200 + PgBouncer で振分、Slow Query（5s 超）自動 kill。
- 契約プラン連動: tenant RPS 上限は 60_事業契約 の課金メータリングで契約プラン毎に設定（Basic 100 RPS / Standard 500 RPS / Enterprise 2000 RPS）。
- Chaos 試験: Phase 1c で 10 倍負荷時に 1 テナントのみ 429、他テナント正常稼働を確認。
- 確定フェーズ: Phase 1b でサービス間・バックエンド層、Phase 1c で Chaos 試験。

## リスク・脅威・インシデント対応

要件定義 NFR-E-PRE-001（準拠規格）、NFR-E-RSK-001/002（脅威モデリング・ペンテスト）、NFR-E-SIR-001〜003（インシデント対応 Runbook / 漏えい等報告義務 / フォレンジック証跡）を採番する。本節では既存 DS-NFR-SEC-020 / DS-NFR-THM-018/019 と連携しつつ、準拠規格と SIR 個別要件の方式を確定する。

**設計項目 DS-NFR-SEC-042 準拠規格の対応表と改訂サイクル**

要件定義 NFR-E-PRE-001 は「個人情報保護法 / J-SOX / ISO 27001 / NIST 800-53 / JTC 情報セキュリティ方針」への準拠を要求する。条項毎の対応方式を設計書で明示する。

- 対応表: [../../03_要件定義/90_付録/02_非機能要求グレード判定.md](../../03_要件定義/90_付録/02_非機能要求グレード判定.md) に各規格の条項と本書の DS-NFR-SEC-NNN を紐付け。
- 改訂サイクル: 法令改正時は 3 ヶ月以内に本書改訂。ISO/NIST は年次レビュー。
- 認証取得: ISO 27001 認証取得判断は Phase 5 以降、本項は準拠フレームワークとしての扱い。
- 確定フェーズ: Phase 1b で対応表整備、継続改訂。

**設計項目 DS-NFR-SEC-043 脅威モデリング ADR 化**

要件定義 NFR-E-RSK-001 は「STRIDE ベースの脅威モデリング ADR を Phase 1a で公開」を要求する。[10_脅威モデリング方式.md](10_脅威モデリング方式.md) DS-NFR-THM-001〜020 で本体設計を行い、本項は ADR 化と運用サイクルに絞って設計する。

- ADR: ADR-SEC-001（STRIDE 脅威モデル初版）、ADR-SEC-002（四半期更新）、以降四半期毎に新番号。
- スコープ更新: Phase 1b / 2 のスコープ変更時に必ず再評価、更新なしでは Phase ゲート不通。
- CVE 年次レビュー: 主要 OSS（Dapr / Istio / Keycloak / OpenBao）の CVE を年次で棚卸し、脅威モデルに反映。
- 確定フェーズ: Phase 1a で ADR-SEC-001、Phase 1c 以降定期化。

**設計項目 DS-NFR-SEC-044 年 1 回ペネトレーションテスト**

要件定義 NFR-E-RSK-002 は「Phase 1c で外部委託 or 内部 Red Team のペンテスト、重大指摘は本番前解消」を要求する。DS-NFR-SEC-020 と重複するが、RSK 側の明示マッピングのため本項で採番する。

- 頻度: Phase 1c で初回、以降年 1 回。
- 委託先: 外部第三者機関（予算 300 万円/年）。Phase 3 以降は Red Team も併用。
- スコープ: tier1 公開 11 API、Envoy Gateway、Keycloak、Backstage、Grafana 公開部分。
- 是正 SLA: Critical/High は 30 日以内、Medium は 90 日以内。
- 確定フェーズ: Phase 1c 初回、継続年 1 回。

**設計項目 DS-NFR-SEC-045 インシデント対応 Runbook 分類別整備**

要件定義 NFR-E-SIR-001 は「Severity 別 Runbook 整備、15 本中 5 本以上がセキュリティ関連」を要求する。Runbook の分類と整備計画を本項で確定する。

- SEV1 Runbook: Runbook-SEC-TENANT-001（テナント越境）、Runbook-SEC-LEAK-001（秘密情報漏洩）、Runbook-SEC-DDOS-001（DDoS）。
- SEV2 Runbook: Runbook-SEC-AUTH-001（不正ログイン・異常 Secret 取得）、Runbook-SEC-MAL-001（マルウェア検知）。
- SEV3 Runbook: Runbook-SEC-CFG-001（設定ミス・軽微認可違反）。
- 訓練: 年 2 回、Runbook の 1 つを抜き打ち実行。
- 確定フェーズ: Phase 1c で 5 本以上、Phase 2 で訓練定期化。

**設計項目 DS-NFR-SEC-046 漏えい等報告義務対応**

要件定義 NFR-E-SIR-002 は「個人情報保護法の報告義務（72 時間速報 / 30 日確報）への対応フロー」を要求する。Runbook-SEC-LEAK-001 でフローを確定する。

- 検知 → 24 時間以内通知: 情シスマネージャ + 監査担当に通知、影響範囲の初期推定。
- 72 時間以内速報: 個人情報保護委員会へオンライン報告、対象データ種別・件数・影響人数の推定を含む。
- 30 日以内確報: 確定事実、原因、再発防止策を確報として提出。
- 影響範囲特定: Audit API + tenant_id 検索のクエリテンプレート（standard queries）で迅速化。
- 確定フェーズ: Phase 1c で Runbook、Phase 2 で訓練実施。

**設計項目 DS-NFR-SEC-047 フォレンジック証跡保全（90 日）**

要件定義 NFR-E-SIR-003（要件 ID 索引上は正称、NFR-E-IR-003 は誤記）は「SEV1 発生時に Audit ログ・Pod ログ・メモリダンプを 90 日以上保全、MinIO Object Lock で改ざん拒否」を要求する。

- Audit ログ: 日次で MinIO WORM バケット（`audit-archive`）へ転送、Object Lock 保管期限 7 年（SEV1 時は最低 90 日を保証）。
- Pod ログ: SEV1 発生時に該当 Pod の Loki ログを 90 日延長保管（通常 30 日）、MinIO `forensics-<incident_id>` bucket へスナップショット。
- メモリダンプ: SEV1 発生直後に SRE が `kubectl debug` で該当 Pod の core dump を取得、MinIO WORM へ保管。
- 手順書: Runbook-SEC-FORENSIC-001。
- 確定フェーズ: Phase 1c で整備、Phase 2 で自動化。

## Web 対策と脆弱性管理の詳細

要件定義 NFR-E-WEB-001/002 と NFR-E-AV-001/002 は「OWASP Top 10 対策 / WAF / イメージスキャン / SBOM」を採番する。DS-NFR-SEC-008/014/016 で大枠は設計済みだが、tier3 向けガイドラインと SBOM 管理の具体化が不足する。

**設計項目 DS-NFR-SEC-048 OWASP Top 10 tier3 向けガイドライン**

要件定義 NFR-E-WEB-001 は「tier3 アプリ向けセキュリティガイドライン + SDK ヘルパ」を要求する。

- ガイドライン: [../../50_開発者体験/](../../50_開発者体験/) 配下に Markdown で公開、OWASP Top 10 各項目を tier3 実装者向けに具体化。
- SDK ヘルパ: CSRF トークン生成、CSP ヘッダ設定、入力検証ヘルパを tier1 SDK に同梱。
- 静的解析: tier3 アプリの CI で Semgrep / CodeQL を走らせ、OWASP ルールセットで検知。
- 確定フェーズ: Phase 1b でガイドライン、Phase 1c で SDK ヘルパ。

**設計項目 DS-NFR-SEC-049 Envoy Gateway WAF（Phase 3）**

要件定義 NFR-E-WEB-002 は「Phase 3 で WAF 導入判定、OWASP Core Rule Set 適用」を採番する。DS-NFR-SEC-004 で Phase 1c の Coraza WAF を前倒し導入しており、本項は Phase 3 の強化版を扱う。

- Phase 1c: Coraza（ModSecurity 互換）で OWASP CRS 3.3。
- Phase 3: ルールセット拡張、Bot 対策、機械学習ベースの異常検知を検討。
- 判定条件: Phase 2 末のインシデント分析で Web 攻撃頻度を評価、高頻度なら Phase 3 先行。
- 確定フェーズ: Phase 1c 基本、Phase 3 強化。

**設計項目 DS-NFR-SEC-050 コンテナイメージスキャン強化**

要件定義 NFR-E-AV-001 は「CI で Trivy 必須 + Phase 2 で Cosign + Kyverno 強制」を要求する。DS-NFR-SEC-010/016 を補強する。

- CI: 全 PR で Trivy scan（source + image）、Critical は PR ブロック、High は SecOps レビュー。
- Phase 2: Cosign 署名済みイメージ + Kyverno で未署名 Pod 起動拒否（DS-NFR-SEC-010 と整合）。
- 例外: 緊急セキュリティパッチ時の例外承認は Product Council Chair のみ。
- 確定フェーズ: Phase 1a Trivy、Phase 2 Cosign + Kyverno。

**設計項目 DS-NFR-SEC-051 SBOM 生成率 100%**

要件定義 NFR-E-AV-002 は「Phase 1c で SBOM 生成率 100%、CycloneDX/SPDX 形式、Backstage 検索可能」を要求する。DS-NFR-SEC-014 を補強する。

- 生成: Syft を CI に組込、全イメージに CycloneDX + SPDX を付与。
- Backstage: Catalog にイメージエントリを登録、SBOM を添付、CVE 検索可能。
- 検証: Pod 起動時に attestation 確認（Cosign attest と連動、Phase 2）。
- 確定フェーズ: Phase 1c 100%、Phase 2 attestation。

## ライセンス・セーフティ要件の扱い（亡霊 ID 対応）

要件定義 [../../03_要件定義/80_トレーサビリティ/01_要件ID索引.md](../../03_要件定義/80_トレーサビリティ/01_要件ID索引.md) で `NFR-E-LIC-001` および `NFR-E-SAFE-001` は「E セキュリティに存在しない亡霊 ID、ライセンス遵守は BC-LIC-001〜005 に移管済み」と明示されている。概要設計側でもこの事実を明記し、参照元の文脈を失わないよう BC-LIC への誘導を設計項目として残す。

**設計項目 DS-NFR-SEC-052 OSS ライセンス遵守の参照先移管**

要件定義 NFR-E-LIC-001 は亡霊 ID として是正済みであり、ライセンス遵守要件は BC-LIC-001〜005（60_事業契約 / 10_OSSライセンス適合）に移管されている。本設計書では [09_完整性とコンプライアンス方式.md](09_完整性とコンプライアンス方式.md) DS-NFR-COMP-* と [../../01_企画/05_法務サマリ/01_OSSライセンス適合.md](../../01_企画/05_法務サマリ/01_OSSライセンス適合.md) が扱う。

- 参照先: BC-LIC-001〜005（MIT/Apache/BSD 許容、GPL 系の扱い、AGPL の境界遮断）。
- 本書の関連設計項目: DS-NFR-SEC-040（AGPL OSS 外部境界遮断）。
- 年次レビュー: 法務部門による OSS ライセンス棚卸しを年 1 回、LICENSE ファイル自動収集 + 手動確認。
- 確定フェーズ: Phase 1b 棚卸し開始、Phase 1c 自動化。

**設計項目 DS-NFR-SEC-053 セーフティ要件の参照先移管**

要件定義 NFR-E-SAFE-001 は亡霊 ID として是正済みであり、セーフティ観点（人命影響・環境影響）は現時点の k1s0 スコープ（社内業務基盤）では直接該当しない。tier2/3 上で構築される業務アプリ固有のセーフティ要件は、当該アプリ側の BC-LIC-005（OSS 適合レビュー）と企画フェーズの稟議で扱う。

- 現状スコープ: k1s0 は社内業務基盤、人命・環境への直接影響なし。
- 参照先: tier2/3 アプリ側の個別要件、BC-LIC-005 法務レビュー。
- Phase 5 再評価: 外部公開や IoT 連携スコープ追加時に再評価。
- 確定フェーズ: Phase 1a 本書で不該当と明示、Phase 5 再評価。
