# 03. 運用方式設計

本ファイルは要件定義書 [../../03_要件定義/30_非機能要件/C_運用保守/](../../03_要件定義/30_非機能要件/C_運用保守/) の NFR-C 系 17 要件を受けて、k1s0 を日次・週次・月次・四半期の 4 サイクルで運用する方式を確定する。起案者 + 協力者の 2 名体制で月 280 実働時間以内に収める計画運用を設計する層である。インシデント対応・サポート階層・EOL・撤退戦略は [../55_運用ライフサイクル方式設計/](../55_運用ライフサイクル方式設計/) に委ね、本章は計画運用のみを扱う。

## 本ファイルの位置付け

運用方式を曖昧にすると、2 名体制で約束した運用工数が破綻する。「都度対応」「気付いた人が対応」は属人化を招き、起案者不在で全停止するバス係数 1 の状態を作る。本章は、全運用タスクを周期化して Runbook に紐付け、どの周期でどの役割が何をどう実施するかを確定する。

周期化は 4 レベルで設計する。日次（毎日自動実行、人間作業なし）、週次（月 4 回、人間作業あり）、月次（月 1 回、レビューと判断）、四半期（3 ヶ月 1 回、戦略レベルの見直し）である。この 4 レベル設計は企画書で約束した「2 名で月 280 時間」を超えないよう、各レベルに工数上限を設定する。

## 運用役割の定義

k1s0 の運用は 4 役割で分担する。起案者 + 協力者の 2 名が複数役割を兼務する前提とし、Phase 2 以降で人員増強時に役割を分離する。

**設計項目 DS-NFR-OPS-001 運用役割と責任範囲**

現状の JTC オンプレ運用は役割定義が曖昧なまま「気付いた人が対応」で回しており、稟議では起案者不在時のバス係数 1 が最大懸念となる。本項目で SRE / SecOps / Dev / Support の 4 役割を明文化し、月 280 時間の工数配分を 40/20/20/10/10 で固定する。役割が崩れると SLO 99.9% と年次監査の責任所在が不明確になり、稟議で約束した運用体制が形骸化する。

- Platform SRE: tier1 および基盤 OSS の稼働監視・SLO 管理・インシデント一次対応・Runbook 更新。Phase 1b で起案者が兼務、Phase 2 で 1 名専任化を目指す。
- SecOps: 脆弱性管理・OSS ライセンス監査・アクセス統制・監査ログレビュー。Phase 1c で協力者が兼務。
- Dev: tier2 / tier3 開発者の支援・Dapr / tier1 API の質問応答。Phase 1b で起案者が兼務。
- Support: tier3 利用者（業務担当者）からの問い合わせ 1 次受付。Phase 1b は Slack / Backstage チケット、Phase 2 で専任チーム設置。
- 工数配分目標: SRE 40% / SecOps 20% / Dev 20% / Support 10% / 教育訓練 10%（月 280 時間に対する配分）。

## 日次運用

日次運用は自動化を最大化し、人間作業を原則ゼロにする。人間が日次で手動介入する必要がある運用は設計不備とみなし、Phase 1b で自動化するか Phase 2 で自動化候補として登録する。

**設計項目 DS-NFR-OPS-002 日次自動運用タスク**

現状、日次バックアップや証明書更新を人手で回すと 2 名体制では 1 日数時間を浪費し、漏れが RPO 24h の裏付けを失わせる。本項目で barman cloud / Longhorn スナップショット / cert-manager / Trivy を全て Kubernetes CronJob 化し、毎朝 09:00 に Slack レポートのみを人間に残す。自動化が崩れると RPO 保証が事実上消滅し、復旧時の監査指摘（バックアップ証跡欠落）で稟議通過の前提が崩れる。

- PostgreSQL バックアップ: barman cloud を Kubernetes CronJob で毎日 02:00 JST 実行、MinIO に退避、7 世代保持。
- Valkey バックアップ: Longhorn PVC スナップショットを 02:30 JST 取得、7 世代保持。
- MinIO ライフサイクル: 監査ログは Object Lock で 7 年保管、Loki チャンクは 30 日で MinIO IA クラスへ遷移、90 日で削除。
- Loki / Mimir 圧縮: 毎日 03:00 JST に compactor 実行。
- Kafka ログセグメント削除: retention 7 日（業務トピック）、30 日（監査トピック）、ログクリーナー自動実行。
- イメージスキャン: Harbor Trivy Scanner が新規 push 時に自動実行（常時）。
- 証明書更新: cert-manager で SPIFFE SVID 1 時間、Envoy Gateway 公開 TLS 90 日前自動更新。
- 自動実行状況レポート: 全 CronJob の成功/失敗を毎朝 09:00 に Slack 通知。

**設計項目 DS-NFR-OPS-003 日次人間タスク（最小化）**

人間が毎日作業を持つ運用は設計不備とみなす前提で、本項目は日次人間工数を月 5 時間以内（15 分 × 20 営業日）に絞る。朝会で Slack レポートと SLO 消費率のみを確認し、異常時のみインシデント対応に遷移する Runbook 連携を敷く。日次人間タスクが肥大化すると、月 280 時間の枠が朝会だけで侵食され、週次・月次の改善サイクルに回せる時間が消える。

- 朝会 09:00〜09:15（15 分）: Slack レポート確認、前日の SLO 消費率と CronJob 失敗の有無確認。
- 異常時のみインシデント対応（[../55_運用ライフサイクル方式設計/02_インシデント対応方式.md](../55_運用ライフサイクル方式設計/02_インシデント対応方式.md) 参照）。
- 月 280 時間のうち日次作業: 15 分 × 20 営業日 = 5 時間以内。

## 週次運用

週次運用は脆弱性対応と OSS バージョン更新が中心である。Renovate（AGPL のため Phase 1b 以降）が自動 PR を生成し、人間は PR レビューとマージのみを担当する。

**設計項目 DS-NFR-OPS-004 週次セキュリティタスク**

JTC では CVE トリアージが属人化し、Critical の適用が月単位で遅延するのが一般的な現状である。本項目で Trivy 週次再スキャン・月曜朝会トリアージ・Critical 48 時間/High 7 日の適用 SLA を定義し、Grype SBOM 突合と OpenBao 監査ログレビューを週次サイクルに組み込む。この規律が崩れると、監査で「脆弱性対応遅延」指摘を受け、7 年 WORM 監査対応パックの信頼性が失われる。

- Trivy 定期スキャン: 毎週月曜 00:00 に全 Harbor イメージ再スキャン（新規 CVE 適用判定）。
- CVE トリアージ: 月曜朝会で SRE がレビュー、Critical は 48 時間以内・High は 7 日以内のパッチ適用計画。
- Grype SBOM 突合: 毎週、稼働中の Pod SBOM と最新 CVE DB の突合レポートを生成。
- OpenBao 監査ログレビュー: 週次で異常アクセス（ルート権限使用、承認外時間帯アクセス）を抽出、Slack 報告。

**設計項目 DS-NFR-OPS-005 週次 OSS 更新タスク（Phase 1b 以降）**

OSS バージョン更新を手動で追う運用では、更新遅延による既知脆弱性の滞留と、破壊的変更の突発発生が避けられない。本項目で Renovate を週次 PR 生成に組み込み、`buf breaking` と kubeval による破壊的変更検出を自動化し、レビュー工数を月 2 時間以内に抑える。更新が停滞すると OSS サポート切れイメージが稼働し続け、5 年 TCO 3.68 億円の前提となる「商用サポート不要」の経済合理性が崩れる。

- Renovate PR 生成: 毎週月曜自動実行、tier1 コード依存 / Helm チャート / コンテナイメージの PR を一括生成。
- PR レビュー: 水曜までに SRE がレビュー、マージは Argo CD auto-sync が本番適用。
- 破壊的変更検出: `buf breaking` と kubeval で失敗した PR は自動 Close、手動対応扱い。
- 工数: PR レビュー 30 分 × 週 1 = 月 2 時間以内。

**設計項目 DS-NFR-OPS-006 週次キャパシティレビュー**

キャパシティ逼迫は突発検知では手遅れで、オンプレ VM 調達リードタイム 3 か月（[06_キャパシティ計画.md](06_キャパシティ計画.md) DS-NFR-CAP-016）に間に合わない。本項目で週次 15 分のダッシュボードレビューを組み込み、CPU 70% / メモリ 80% / ディスク 70% の 3 閾値で翌週キャパシティ会議の対象を機械的に決める。レビューを怠ると四半期の容量計画に反映漏れが出て、定常 150 RPS や SLO 99.9% を物理リソース不足で割り込む。

- 前週の CPU / メモリ / ディスク利用率を Grafana ダッシュボード `k1s0-capacity-weekly` で確認。
- 閾値: CPU 70% / メモリ 80% / ディスク 70% を超過するノードがある場合、翌週の容量計画レビュー対象に追加。
- 週次レビュー 15 分 × 4 週 = 月 1 時間以内。

## 月次運用

月次は戦略レベルのレビューを中心とする。DR 演習、SLO レポート作成、セキュリティ監査報告が 3 本柱である。

**設計項目 DS-NFR-OPS-007 月次 DR 演習（Phase 1c 以降）**

現状、JTC オンプレ環境の DR 手順は書類のみで年 1 回の机上演習に留まり、実際に PG Primary を落として RTO 4 時間以内に復旧できるかは検証されていない。本項目で月 1 回 4 時間枠の実障害注入演習を 3 シナリオローテーションで行い、Runbook 更新 PR を翌週マージする閉ループに仕立てる。演習を省くと、RPO 秒〜24h / RTO 4h のコミットが実戦で通用するか不明のまま稟議期限を迎え、監査でも「DR 実効性未検証」で指摘を受ける。

- 第 1 月曜 13:00〜17:00（4 時間）に実施。
- シナリオ: 3 シナリオのローテーション。(1) PostgreSQL Primary 障害、(2) Kafka Broker 2 台喪失、(3) ノード 1 台喪失。
- 年 1 回は DC 全損シナリオ（4 時間枠に収まらないため、Phase 2 以降は休日実施）。
- 合格基準: RTO 目標時間以内に tier1 API の 11 API 全てが復帰、RPO 以内のデータ損失。
- 演習結果は Runbook 更新 PR として起票、翌週マージ。
- 工数: 4 時間 × 2 名 = 月 8 時間。

**設計項目 DS-NFR-OPS-008 月次 SLO レポート**

現状、JTC 内製運用では SLO を数値化して月次報告する文化がなく、「だいたい動いていた」で稟議の可用性コミットが曖昧に流される。本項目で Grafana の SLI / SLO / Error Budget 消費率を SRE が月 2 時間で整理し Product Council に提出する定例を敷く。レポートが欠けると SLA 99% / SLO 99.9% の遵守実績を四半期・年次で説明できず、TCO 3.68 億円の正当化根拠（可用性と運用コストの両立）が稟議上崩れる。

- 月末に前月の SLI / SLO / Error Budget 消費率を Grafana ダッシュボードから抽出。
- レポート作成: SRE が 2 時間でまとめ、Product Council に提出。
- 内容: API ごとの可用性・p99 レイテンシ・エラー予算消費、目標比較、翌月の改善アクション。
- 工数: 月 2 時間以内。

**設計項目 DS-NFR-OPS-009 月次セキュリティレビュー**

脆弱性対応やアクセスログ異常は週次で拾いつつも、月単位の集計なしでは「傾向悪化」や「特定テナントの異常パターン」を拾えない。本項目で SecOps が月 2 時間で CVE 対応件数・認証失敗率・異常アクセスを集計し、四半期で Product Council にエスカレーションする仕組みを敷く。月次レビューが欠けると、監査で「セキュリティ運用の PDCA 証跡なし」と指摘され、7 年 WORM 監査要件の実効性が疑問視される。

- 前月の脆弱性対応実績（適用済 CVE 件数、未対応件数）、アクセスログ異常件数、認証失敗率を集計。
- SecOps が 2 時間でレポート、SRE チームレビュー。
- 四半期ごとに Product Council エスカレーション。
- 工数: 月 2 時間以内。

## 四半期運用

四半期運用は戦略判断を伴う。総合キャパシティ計画、OSS ライセンス監査、脅威モデル更新の 3 本である。

**設計項目 DS-NFR-OPS-010 四半期総合キャパシティ計画**

オンプレ VM 増設は稟議 1 か月 + 調達構築 1 か月 + 検証 1 か月の計 3 か月リードタイムを要し、週次レビューだけでは間に合わない。本項目で SRE が四半期 1 回 4 時間（年 16 時間）をかけ、直近 3 か月実績から次 3 か月予測を作成し、VM 増設稟議の要否を判断する。この四半期判断が遅れると、成長予測（Phase 2 で 10 テナント）に VM 追加が間に合わず、定常 500 RPS 目標をリソース不足で達成できない。

- 直近 3 ヶ月の利用実績から次 3 ヶ月の予測を作成、リソース増強の要否判定。
- [06_キャパシティ計画.md](06_キャパシティ計画.md) の閾値と突合、VM 増設の稟議要否を判断。
- 工数: SRE 4 時間 × 四半期 1 回 = 年 16 時間。

**設計項目 DS-NFR-OPS-011 四半期 OSS ライセンス監査**

OSS ライセンスは推移的依存の更新で AGPL / BSL / SSPL / Commons Clause が混入するリスクが常にあり、発見が遅れると商用配布要件に抵触する。本項目で SecOps が四半期 4 時間（年 16 時間）で Syft の SBOM からライセンス分布レポートを生成し、年次監査対応パックに組み込む。監査が欠けると、稟議で約束した「OSS 限定で商用ライセンス費ゼロ」という年 2,000 万円超の削減根拠が、逆にライセンス違反リスクに転じて年 3,280 万円削減計画全体を毀損する。

- Syft で全 Pod の SBOM 生成、ライセンス分布レポートを作成。
- AGPL / BSL / SSPL / Commons Clause の誤混入チェック（Phase 1b 以降 AGPL は隔離運用のため通常発見されない想定）。
- 監査結果は Product Council に提出、年次監査対応パックに組み込み。
- 工数: SecOps 4 時間 × 四半期 1 回 = 年 16 時間。

**設計項目 DS-NFR-OPS-012 四半期脅威モデル更新**

脅威モデルは一度作って終わりでは業界の新規攻撃手法（サプライチェーン攻撃の高度化、新 CVSS 重大脆弱性）を反映できず、数四半期で陳腐化する。本項目で SecOps が四半期 4 時間（年 16 時間）で [10_脅威モデリング方式.md](10_脅威モデリング方式.md) の脅威リストを、過去 3 か月インシデント・業界動向・Phase 拡大を踏まえて更新する。更新を怠ると、脅威モデルが現実と乖離し、セキュリティ投資の優先度判断がブレて監査でも「脅威モデル陳腐化」指摘を受ける。

- [10_脅威モデリング方式.md](10_脅威モデリング方式.md) の脅威リストを最新化。
- 過去 3 ヶ月のインシデント、業界の新規攻撃手法、Phase 進行による範囲拡大を反映。
- 工数: SecOps 4 時間 × 四半期 1 回 = 年 16 時間。

## チェンジマネジメント

k1s0 の全変更は GitOps 単一経路で行う。kubectl による直接変更、Helm CLI による手動適用、データベース SQL の直接実行は禁止する。GitOps 以外の経路を塞ぐことで、変更履歴が自動的に Git コミットとして残り、J-SOX 統制の変更記録要件を満たす。

**設計項目 DS-NFR-OPS-013 GitOps 単一経路の強制**

JTC の従来運用では `kubectl edit` / 手動 Helm / 直接 SQL のサイドチャネルが黙認され、変更台帳が分散して J-SOX 統制の「変更記録要件」を満たせない。本項目で Argo CD + Kyverno + Flyway の組み合わせで全経路を塞ぎ、Git コミットが唯一の変更証跡となる状態を強制する。単一経路が崩れると監査で「変更統制不備」で指摘を受け、7 年 WORM 監査パックに提出する変更台帳の完全性が失われる。

- Kubernetes リソース: Argo CD が唯一のデプロイ主体。namespace 単位で `argocd.argoproj.io/managed` ラベルを付け、Kyverno で非 Argo CD 変更を拒否。
- コンテナイメージ: Harbor のみ許可、外部レジストリは NetworkPolicy で遮断。Renovate 経由の version pin 更新のみを許可。
- 設定変更: Helm values / Kustomize overlay を Git で管理、手動 `kubectl edit` は Kyverno で警告ログ記録。
- データベーススキーマ変更: Flyway マイグレーションを tier1 サービスの起動時フック、SQL 直接実行は PostgreSQL ロールで禁止。
- 確定フェーズ: Phase 1b で Kyverno 導入、Phase 1c で全経路制限完了。

**設計項目 DS-NFR-OPS-014 変更承認フロー**

変更を一律 2 名承認にすると Renovate の patch 更新で PR 渋滞が発生し、逆に緩めると hotfix の監査証跡が弱まる。本項目で low / medium / high の 3 カテゴリと承認者要件・Argo CD sync 戦略・事後レポート SLA を差別化し、Slack `#k1s0-changes` への自動通知で台帳化する。フローが崩れると緊急対応が承認待ちで間延びし、SLA 99% の復旧 SLA 4 時間を承認プロセス側で食い潰す。

- 通常変更（low risk）: PR レビュー 1 名承認で自動マージ、Argo CD auto-sync。典型は Renovate の patch 更新。
- 標準変更（medium risk）: PR レビュー 2 名、うち 1 名は SRE。Argo CD sync は手動トリガー。
- 緊急変更（high risk / hotfix）: PR 後 30 分以内のマージ、SRE + Product Council Chair 2 名承認、事後レポート 48 時間以内。
- 全変更は Slack `#k1s0-changes` に通知、変更台帳は Git ログから自動生成。
- 確定フェーズ: Phase 1b で 3 カテゴリ運用開始。

## 起動・停止・保守手順

計画停止（メンテナンス窓）は年 12 時間以内に収める前提で、無停止更新を可能な限り追求する。ローリングアップデートで可能な変更は計画停止不要とし、計画停止が必要な変更（Kubernetes メジャーアップデート、データベーススキーマ非互換変更）は年間 2〜3 回に限定する。

**設計項目 DS-NFR-OPS-015 ローリングアップデート運用**

計画停止で更新を吸収する従来運用は年間停止時間を積み上げ、SLO 99.9% の月次エラー予算 43 分を容易に食い潰す。本項目で tier1 は `maxSurge=1 / maxUnavailable=0`、PG は CloudNativePG、Valkey は Sentinel、Kafka は Strimzi rolling update で無停止更新を標準化する。ローリングが崩れると計画停止に頼らざるを得ず、年 12 時間の停止予算では Kubernetes / PG / tier1 の更新サイクルを吸収しきれず SLO 割れが常態化する。

- tier1 Pod: `maxSurge=1`、`maxUnavailable=0`。常に 3 Pod 以上稼働しながら順次更新。
- PostgreSQL: CloudNativePG の supervisor が Standby → Primary の順で無停止更新。
- Valkey: Sentinel が Replica → Master の順で順次更新、ダウンタイム 30 秒以内。
- Kafka: Strimzi rolling update、Broker 1 台ずつ再起動、ISR 維持で無停止。
- 確定フェーズ: Phase 1a で tier1、Phase 1b でデータ層全体。

**設計項目 DS-NFR-OPS-016 計画停止ウィンドウ**

無停止更新でカバーできない更新（K8s メジャーアップデート、PG メジャー更新、OpenBao マイグレーション）は、事前通知と事後レポートの枠組みなしで実施するとテナントからの信頼を失う。本項目で第 3 土曜 02:00〜06:00 の 4 時間枠を年 3 回まで固定し、72 時間前の Backstage 告知と 24 時間以内の事後レポートを定型化する。ウィンドウが曖昧だと tier2/tier3 テナントの業務に予期せぬ影響が出て、SLA 99% の年間ダウンタイム予算（約 88 時間）のうち計画停止だけで過半を食う。

- スケジュール: 第 3 土曜 02:00〜06:00 JST の 4 時間枠、年間最大 3 回利用可能。
- 用途: Kubernetes メジャーアップデート、PostgreSQL メジャーバージョンアップ、OpenBao マイグレーション。
- 事前通知: 停止 72 時間前に tier2 / tier3 全チームへ通知（Backstage 告知 + Slack）。
- 事後レポート: 停止完了 24 時間以内に結果レポート、Product Council 共有。
- 確定フェーズ: Phase 1b 運用開始。

## Runbook 参照と管理

運用手順の詳細は [../55_運用ライフサイクル方式設計/08_Runbook設計方式.md](../55_運用ライフサイクル方式設計/08_Runbook設計方式.md) に集約する。本章は Runbook の存在を前提にしつつ、各運用タスクが参照すべき Runbook ID を指定する。

**設計項目 DS-NFR-OPS-017 運用タスクと Runbook 対応**

運用タスクを定義しても、発動時にどの Runbook を開くかが曖昧だと 2 名体制は探索時間で疲弊する。本項目で日次自動タスク失敗・週次 CVE・月次 DR・計画停止・SEV1 の 5 カテゴリに Runbook ID（`RB-BACKUP`/`RB-SEC`/`RB-DR`/`RB-MAINT`/`RB-SEV1`）を機械的に対応付け、[../55_運用ライフサイクル方式設計/08_Runbook設計方式.md](../55_運用ライフサイクル方式設計/08_Runbook設計方式.md) に整備を委譲する。対応付けが崩れると RTO 4 時間の復旧 SLA が「どの手順で復旧するか」の決定遅延で達成不能になる。

- 日次自動タスク失敗時: `RB-BACKUP-001`（PG バックアップ失敗）、`RB-BACKUP-002`（Valkey スナップショット失敗）。
- 週次 CVE 対応: `RB-SEC-001`（Critical CVE パッチ手順）。
- 月次 DR 演習: `RB-DR-001`（PG Primary 障害）、`RB-DR-002`（Kafka Broker 障害）、`RB-DR-003`（ノード障害）、`RB-DR-ANN`（年次 DC 全損）。
- 計画停止: `RB-MAINT-001`（K8s アップデート）、`RB-MAINT-002`（PG メジャー更新）。
- 緊急時: `RB-SEV1-001`〜`RB-SEV1-010` の 10 パターン。
- 確定フェーズ: Phase 1b 基本 Runbook、Phase 1c で SEV1 パターン完備。
