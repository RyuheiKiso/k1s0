# 50. 非機能方式設計

本カテゴリは要件定義書で採番された非機能要件（NFR）を設計レベルに落とし込む。IPA 非機能要求グレード 2018 の 6 大項目との 1:1 対応を維持し、要件の「約束」を設計の「方式」に変換する層である。

## 本カテゴリの位置付け

[../../03_要件定義/30_非機能要件/](../../03_要件定義/30_非機能要件/) が SLO 数値・可用性目標・性能要件を**約束**として採番したのを受け、本章はその約束を「どの仕組みで実現するか」という**方式**に変換する。要件定義が「何を保証するか」なら、本章は「どう保証するか」を確定させる場である。

[../10_システム方式設計/](../10_システム方式設計/) との分担は以下のとおり明確にする。10 章は「システム全体構成としてどう並べるか」を扱い、本章は「その構成で非機能要件をどう満たすか」を扱う。例として、10 章で決めた 3 拠点マルチクラスタ構成を前提に、本章は「その構成で RPO 15 分 / RTO 1 時間を達成する具体的な DR 手順」を方式化する。

## 格納予定ドキュメント

IPA 非機能要求グレード 6 大項目に沿って 6 ファイルを格納する。

- `01_可用性_DR方式.md` — HA 構成方式（Active-Active / Active-Standby）、災害復旧手順、RPO / RTO 実装方式、フェイルオーバ自動化の範囲を記述する。要件 ID `NFR-A-*` と対応する。
- `02_性能と拡張性方式.md` — HPA / VPA のスケール方針、キャッシュ階層方式、ホットスポット回避方式、負荷分散方式を記述する。要件 ID `NFR-B-*` と対応する。
- `03_運用方式設計.md` — 起動停止手順、バックアップとリストア方式、ジョブ管理方式、オンコールと escalation 方式を記述する。要件 ID `NFR-C-*` と対応する。
- `04_監視と観測性方式.md` — SLO / エラーバジェット実装方式、アラートしきい値の設計、オンコール向けダッシュボード構成、ランブック自動リンク方式を記述する。[../30_共通機能方式設計/05_分散トレーシング方式.md](../30_共通機能方式設計/) と連動する。
- `05_セキュリティ方式設計.md` — ゼロトラストの実装方式、脆弱性管理プロセス（SBOM / CVE スキャン / パッチ適用サイクル）、侵入検知と対応方式、機密データ取扱方式を記述する。要件 ID `NFR-E-*` と対応する。
- `06_キャパシティ計画.md` — 初期サイジング根拠、成長予測、容量監視閾値、増設トリガー条件を記述する。要件 ID `NFR-B-CAP-*` と対応する。

## 執筆方針

- 各方式は対応する NFR 要件 ID を冒頭に明示し、SLO 数値を転記する。
- 数値目標を具体的な実装方式（コンポーネント配置・冗長数・タイムアウト・再試行回数）に落とし込むこと。抽象的な記述で留めないこと。
- [../10_システム方式設計/](../10_システム方式設計/) で決めた全体構成を前提として、本章は差分のみを記述する。10 章と同じことを書き直さない。
- 方式 ID は `DS-NFR-<サブカテゴリ>-<通番>` を用いる（例: `DS-NFR-AVL-001`）。

## 他カテゴリとの関係

- **上流**: [../../03_要件定義/30_非機能要件/](../../03_要件定義/30_非機能要件/) 全章が主要入力。IPA 非機能要求グレード 6 大項目（A 可用性 / B 性能拡張性 / C 運用保守性 / D 移行性 / E セキュリティ / F システム環境）と 1:1 対応。
- **隣接**: [../10_システム方式設計/](../10_システム方式設計/)（全体構成）、[../30_共通機能方式設計/](../30_共通機能方式設計/)（監査・ログ・暗号）、[../40_制御方式設計/](../40_制御方式設計/)（SLO を満たすための制御ロジック）と連動。
- **下流**: 詳細設計フェーズで、監視ダッシュボードのクエリ・アラートルール・バックアップスクリプト等の具体実装に展開される。

## 移行性（D 項目）の扱い

IPA 非機能要求グレードの D 項目（移行性）は、k1s0 では [../60_移行方式設計/](../60_移行方式設計/) で独立章として扱う。本章では D 項目の非機能要件（移行時のダウンタイム許容・データ損失許容など）のみを対象とし、移行手順そのものは 60 章に委ねる。
