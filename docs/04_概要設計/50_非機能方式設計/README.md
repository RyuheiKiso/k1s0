# 50. 非機能方式設計

本カテゴリは要件定義書で採番された非機能要件（NFR）を設計レベルに落とし込む。IPA 非機能要求グレード 2018 の 6 大項目（A〜F）に、k1s0 独自拡張の G・H を加えた 8 分類と 1:1 対応を維持し、要件の「約束」を設計の「方式」に変換する層である。

## 本カテゴリの位置付け

[../../03_要件定義/30_非機能要件/](../../03_要件定義/30_非機能要件/) が SLO 数値・可用性目標・性能要件を**約束**として採番したのを受け、本章はその約束を「どの仕組みで実現するか」という**方式**に変換する。要件定義が「何を保証するか」なら、本章は「どう保証するか」を確定させる場である。

[../10_システム方式設計/](../10_システム方式設計/) との分担は以下のとおり明確にする。10 章は「システム全体構成としてどう並べるか」を扱い、本章は「その構成で非機能要件をどう満たすか」を扱う。例として、10 章で決めた 3 拠点マルチクラスタ構成を前提に、本章は「その構成で RPO 秒〜24h（データ層別、Kafka 秒 / Postgres 5 分 / MinIO 1h / Valkey 24h）/ RTO 4h を達成する具体的な DR 手順」を方式化する。

## 要件定義書との対応 8 分類

要件定義書は IPA 標準の 6 項目（A〜F）に k1s0 独自拡張（G データ保護・プライバシー / H 完整性・コンプライアンス）を加えた 8 分類で運用されている。本章はこの 8 分類に 1:1 対応するファイルを持つ。

## 格納予定ドキュメント

- `01_可用性_DR方式.md` — HA 構成方式（Active-Active / Active-Standby）、災害復旧手順、RPO / RTO 実装方式、フェイルオーバ自動化の範囲を記述する。要件 ID `NFR-A-*` と対応する。
- `02_性能と拡張性方式.md` — HPA / VPA のスケール方針、キャッシュ階層方式、ホットスポット回避方式、負荷分散方式を記述する。要件 ID `NFR-B-*` と対応する。
- `03_運用方式設計.md` — 起動停止手順、バックアップとリストア方式、ジョブ管理方式、計画保守のウィンドウ設計を記述する。要件 ID `NFR-C-*` と対応する。インシデント対応・サポート階層・撤退戦略は [../55_運用ライフサイクル方式設計/](../55_運用ライフサイクル方式設計/) に委ね、本章は計画的運用のみを扱う。
- `04_監視と観測性方式.md` — SLO / エラーバジェット実装方式、アラートしきい値の設計、オンコール向けダッシュボード構成、ランブック自動リンク方式を記述する。[../30_共通機能方式設計/05_分散トレーシング方式.md](../30_共通機能方式設計/) と連動する。
- `05_セキュリティ方式設計.md` — ゼロトラストの実装方式、脆弱性管理プロセス（SBOM / CVE スキャン / パッチ適用サイクル）、侵入検知と対応方式、機密データ取扱方式を記述する。要件 ID `NFR-E-*` と対応する。
- `06_キャパシティ計画.md` — 初期サイジング根拠、成長予測、容量監視閾値、増設トリガー条件を記述する。要件 ID `NFR-B-CAP-*` と対応する。
- `07_システム環境とエコロジー方式.md` — 物理環境要件（温湿度・電源・空調）、消費電力最適化、カーボンフットプリント測定と削減目標を記述する。要件 ID `NFR-F-*` と対応する。サステナビリティ目標は [../75_事業運用方式設計/13_サステナビリティ方式.md](../75_事業運用方式設計/) と整合させる。
- `08_データ保護とプライバシー方式.md` — 個人データの分類方式、保存時・転送時の暗号化方式、データレジデンシー（所在地）要件、越境移転の取扱、データ主体の権利（アクセス・訂正・削除）を実現する方式を記述する。要件 ID `NFR-G-*` と対応する。[../30_共通機能方式設計/08_暗号化と鍵管理方式.md](../30_共通機能方式設計/) の暗号方式と連動する。
- `09_完整性とコンプライアンス方式.md` — 法令・業界規制（個人情報保護法 / 金融庁ガイドライン / SOX / ISO 27001 等）への準拠方式、証跡の完全性保全、コンプライアンス自動チェック方式、監査レポート生成方式を記述する。要件 ID `NFR-H-*` と対応する。[../30_共通機能方式設計/04_監査証跡方式.md](../30_共通機能方式設計/) の監査方式と連動する。
- `10_脅威モデリング方式.md` — STRIDE（Spoofing / Tampering / Repudiation / Information disclosure / Denial of service / Elevation of privilege）および PASTA による脅威モデリング方式を記述する。tier1 API 境界・tier2/tier3 境界・外部連携境界・データ境界のそれぞれで脅威を採番し、緩和策と残存リスクを明示する。`05_セキュリティ方式設計.md` が統制（コントロール）を扱うのに対し、本ファイルは脅威（何から守るか）を扱う。
- `11_SLI_SLO_エラーバジェット方式.md` — tier1 11 API および横断機能ごとの SLI 定義（可用性・レイテンシ・エラー率・鮮度）、SLO 数値、エラーバジェット運用方式（バーンレート警告・予算枯渇時の機能凍結ルール）を記述する。`04_監視と観測性方式.md` が監視の実装を扱うのに対し、本ファイルは何を監視し何を約束するかの契約層を扱う。要件 ID `NFR-A-*` / `NFR-B-*` / `NFR-I-*`（k1s0 独自の I 観測性拡張）と対応する。

## 執筆方針

- 各方式は対応する NFR 要件 ID を冒頭に明示し、SLO 数値を転記する。
- 数値目標を具体的な実装方式（コンポーネント配置・冗長数・タイムアウト・再試行回数）に落とし込むこと。抽象的な記述で留めないこと。
- [../10_システム方式設計/](../10_システム方式設計/) で決めた全体構成を前提として、本章は差分のみを記述する。10 章と同じことを書き直さない。
- 方式 ID は `DS-NFR-<サブカテゴリ>-<通番>` を用いる（例: `DS-NFR-AVL-001`）。

- **隣接**: [../10_システム方式設計/](../10_システム方式設計/)（全体構成）、[../30_共通機能方式設計/](../30_共通機能方式設計/)（監査・ログ・暗号）、[../40_制御方式設計/](../40_制御方式設計/)（SLO を満たすための制御ロジック）、[../55_運用ライフサイクル方式設計/](../55_運用ライフサイクル方式設計/)（インシデント・サポート・EOL）と連動。

## 移行性（D 項目）の扱い

IPA 非機能要求グレードの D 項目（移行性）は、k1s0 では [../60_移行方式設計/](../60_移行方式設計/) で独立章として扱う。本章では D 項目の非機能要件（移行時のダウンタイム許容・データ損失許容など）のみを対象とし、移行手順そのものは 60 章に委ねる。
