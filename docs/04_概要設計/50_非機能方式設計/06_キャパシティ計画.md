# 06. キャパシティ計画

本ファイルは要件定義書のキャパシティ関連要件を受けて、k1s0 の規模別 VM 構成・OSS リソース配分・スケール閾値・コスト予測を確定する。企画で約束した TCO（初期 1,850 万円 / 年間 2,349 万円）を超えず、かつ性能要件を満たすキャパシティ計画を設計する層である。

## 本ファイルの位置付け

キャパシティ計画を曖昧にすると、「リソース不足で SLO 割れ」と「過剰プロビジョニングで予算超過」の両方に同時に晒される。本章は規模別に具体的な VM 台数 / CPU / メモリ / ストレージを確定し、増設トリガーの閾値を数値で固定する。予算制約（Phase 0〜1c で 1,850 万円、Phase 2 以降年 2,349 万円）を超えない範囲で、性能要件（tier1 API p99 500ms、定常 150 RPS）を満たすバランスを取る。

## 規模別キャパシティモデル

k1s0 は 3 つの規模プロファイル（小・中・大）を持つ。Phase によって適用されるプロファイルが変わる。

**設計項目 DS-NFR-CAP-001 規模プロファイル定義**

- 小規模: 1 テナント、同時ユーザ 100 名、定常 50 RPS。Phase 1a（MVP-0）相当。VM 1 台。
- 中規模: 5 テナント、同時ユーザ 500 名、定常 150 RPS。Phase 1b〜1c（MVP-1a/1b）および初期商用相当。VM 3 台。
- 大規模: 20 テナント、同時ユーザ 2000 名、定常 500 RPS。Phase 2 以降の拡張商用相当。VM 6 台 + 専用ストレージ。
- 確定フェーズ: Phase 0 で企画書に記載、Phase 1b で中規模実装、Phase 2 で大規模実装。

## VM 構成

各規模プロファイルの VM 構成は以下で確定する。既存 JTC 標準 VM プロファイル（中規模 Web/AP サーバ相当）に収まる範囲で選定し、新規調達リードタイムを最小化する。

**設計項目 DS-NFR-CAP-002 小規模 VM 構成（Phase 1a）**

- VM: 1 台（single-node k8s）。
- スペック: 4 vCPU / 16 GB メモリ / 200 GB SSD。
- 役割: control-plane 兼 worker、全 OSS を 1 VM 上で稼働。
- 冗長度: なし（デモ用）。
- 年間コスト見込: VM コスト 24 万円（既存 VM 基盤流用）+ 運用 100 万円 = 124 万円。

**設計項目 DS-NFR-CAP-003 中規模 VM 構成（Phase 1b〜1c）**

- VM: 3 台。
- スペック: 各 8 vCPU / 32 GB メモリ / 500 GB SSD。
- 役割: control-plane 兼 worker 混在、HA 構成。
- 冗長度: 1 ノード喪失まで継続可。
- 年間コスト: VM 216 万円（既存基盤 72 万円 × 3）+ ストレージ追加 120 万円 + 運用 144 万円 = **480 万円**（企画の中規模 TCO 480 万円と一致）。
- 確定フェーズ: Phase 1b で稟議済予算内でデプロイ。

**設計項目 DS-NFR-CAP-004 大規模 VM 構成（Phase 2 以降）**

- VM: 6 台（control-plane 3 + worker 3 の分離構成）。
- スペック: control-plane 各 8 vCPU / 16 GB / 200 GB、worker 各 16 vCPU / 64 GB / 1 TB。
- 役割: 役割分離で大規模化対応。
- 冗長度: 2 ノード喪失まで継続可（Phase 2 改善）。
- 年間コスト: VM 576 万円（72 万円 × 3 + 144 万円 × 3）+ ストレージ 360 万円 + 運用 288 万円 + 観測基盤拡張 120 万円 = **1,344 万円**（企画の大規模 TCO 1,500 万円の範囲内）。
- 確定フェーズ: Phase 2 で稟議再承認の上でデプロイ。

## OSS リソース配分

各規模で動作させる OSS のリソース要求と上限を確定する。MVP 期（中規模）での配分は以下となる。

**設計項目 DS-NFR-CAP-005 中規模での OSS リソース配分**

tier1 関連:
- tier1 API Pod（Go ファサード + Rust 自作領域）: requests 500m CPU / 512 MB、limits 1000m / 1024 MB、レプリカ 3〜10。
- Dapr sidecar（Go）: 100m / 256 MB、limits 200m / 512 MB、1 sidecar / 1 Pod。

データ層:
- PostgreSQL（CloudNativePG）: 1000m / 8 GB、limits 2000m / 12 GB、3 Pod（Primary + Standby 2）。
- Valkey: 200m / 4 GB、limits 500m / 6 GB、3 Pod（Master + Replica 2）+ Sentinel 100m / 256 MB × 3。
- Kafka（Strimzi）: 500m / 2 GB、limits 1000m / 3 GB、3 Pod。
- OpenBao: 200m / 1 GB、limits 500m / 2 GB、3 Pod。

観測基盤:
- Grafana: 100m / 512 MB、1 Pod。
- Mimir: 500m / 4 GB、3 Pod（ingester / querier / store-gateway 各 1）。
- Loki: 500m / 2 GB、3 Pod（ingester / querier / store-gateway 各 1）。
- Tempo: 300m / 2 GB、2 Pod。
- OTel Collector: 200m / 512 MB、DaemonSet（3 ノード）。

CI/CD:
- Argo CD: 500m / 2 GB、1 Pod（Phase 2 で HA 化）。
- Harbor: 1000m / 4 GB、1 Pod。
- MinIO: 500m / 2 GB、3 Pod。

合計（requests 積算）: CPU 約 10 vCPU / メモリ 約 50 GB。3 ノード合計 24 vCPU / 96 GB に対して 42% / 52% 利用、余裕率 50% 以上を確保。

- 確定フェーズ: Phase 1b デプロイ時に確定、週次 VPA Recommender で微調整。

## ストレージ配分

**設計項目 DS-NFR-CAP-006 中規模でのストレージ配分**

各ノード 500 GB SSD のうち:
- OS + containerd: 60 GB（kubelet 作業領域含む）。
- Longhorn プール: 360 GB（3 ノード合計 1080 GB、replica=3 実効 360 GB）。
- MinIO: 80 GB（3 ノードで計 240 GB、EC 3+2 実効 144 GB）。

Longhorn プール 360 GB の用途別配分:
- PostgreSQL（Primary + Standby 2 各 60 GB）: 180 GB。
- Kafka（Broker 3 各 30 GB）: 90 GB。
- Valkey（3 各 10 GB）: 30 GB。
- OpenBao（3 各 5 GB）: 15 GB。
- その他（Prometheus local / tier1 PVC）: 45 GB。

MinIO 144 GB の用途別配分:
- PostgreSQL barman cloud バックアップ: 40 GB。
- Loki chunks（90 日）: 40 GB。
- Mimir blocks（13 ヶ月のうち 90 日ホット）: 30 GB。
- Harbor イメージキャッシュ: 20 GB。
- 監査ログアーカイブ（Object Lock）: 14 GB（90 日分、長期は外部 NAS 退避）。

- 確定フェーズ: Phase 1b デプロイ時。Phase 2 で MinIO を別ディスク 1 TB に拡張。

## スケール閾値と増設トリガー

容量増設は「事前予測 + 閾値超過」の 2 方式で判定する。事前予測は四半期キャパシティレビューで行い、閾値超過は自動監視で検知する。

**設計項目 DS-NFR-CAP-007 ノードスケール閾値**

- CPU 利用率: ノードあたり 5 分平均で 70% 超過が 1 時間継続 → Warning、3 時間継続 → SEV2。
- メモリ利用率: 80% 超過が 30 分継続 → Warning、1 時間継続 → SEV2。
- ディスク利用率: 70% → SEV3、80% → SEV2、90% → SEV1。
- Pod 配置失敗: 5 分で 10 件以上 → SEV2（容量不足の兆候）。
- 対応: Warning で次回定期レビュー、SEV2 で 72 時間以内に増設判断、SEV1 で即日対応。
- 確定フェーズ: Phase 1b でアラート実装。

**設計項目 DS-NFR-CAP-008 ClusterAutoscaler 方針**

- MVP 期（Phase 1a〜1c）: ClusterAutoscaler 無効。VM 追加は手動稟議。
- Phase 2 以降: オンプレ VM 基盤の API 対応状況を見て、必要に応じ Cluster API（CAPI）導入検討。
- 理由: オンプレ VM 基盤は自動 VM プロビジョニング API を持たないケースが多く、MVP 期は手動で十分。
- 確定フェーズ: Phase 2 で再評価。

**設計項目 DS-NFR-CAP-009 HPA スケール閾値**

- tier1 API Pod: CPU 70% で +1、カスタムメトリクス `k1s0_api_requests_per_pod` が 100 RPS で +1。
- 最小 3 / 最大 10（中規模時）、Phase 2 大規模時は最大 20。
- スケールダウン: 5 分安定期間後、1 回に 1 Pod ずつ。
- 確定フェーズ: Phase 1b 実装。

## 成長予測

Phase 進行によるテナント増・ユーザ数増の予測に基づき、3 ヶ月先の容量予測を四半期レビューで更新する。

**設計項目 DS-NFR-CAP-010 テナント増予測**

- Phase 1b（MVP-1a、1 年目）: 1 テナント（パイロット）。
- Phase 1c（MVP-1b、1.5 年目）: 3 テナント（パイロット + 初期 2 部門）。
- Phase 2（2 年目〜）: 10 テナント（JTC 社内主要部門）。
- Phase 3（3 年目〜）: 20 テナント（JTC 全社 + 一部グループ会社）。
- テナント 1 件あたりの追加負荷: ユーザ 100 名 / 定常 50 RPS / ストレージ 20 GB を仮定。
- 確定フェーズ: Phase 0 企画書記載、四半期レビューで更新。

**設計項目 DS-NFR-CAP-011 3 ヶ月先予測モデル**

- メトリクス: 直近 3 ヶ月のリソース使用率から線形回帰で 3 ヶ月先予測。
- 閾値超過予測: 3 ヶ月以内に CPU 70% / メモリ 80% / ディスク 70% を超える見込みの場合、稟議を開始。
- 稟議リードタイム: VM 増設稟議は 1 ヶ月、調達・構築は 1 ヶ月、合計 2 ヶ月。逆算で 3 ヶ月前の判断が必要。
- 確定フェーズ: Phase 1c でモデル確立、Phase 2 で自動化。

## コスト予測

企画書の TCO 試算と整合するコスト予測を維持する。想定外の超過が発生した場合は Product Council エスカレーション。

**設計項目 DS-NFR-CAP-012 規模別年間コスト**

- 小規模: 124 万円（VM 24 + 運用 100）。
- 中規模: 480 万円（VM 216 + ストレージ 120 + 運用 144）。Phase 1b〜1c で適用。
- 大規模: 1,344 万円（VM 576 + ストレージ 360 + 運用 288 + 観測拡張 120）。Phase 2 以降。
- 予算枠: 初期 1,850 万円（Phase 0〜1c）、年間 2,349 万円（Phase 2 以降）。
- コスト超過閾値: 予算の 80% 消費で Product Council 警告、95% で新規機能凍結。
- 確定フェーズ: Phase 0 稟議で確定、四半期レビューで追跡。

**設計項目 DS-NFR-CAP-013 コスト削減余地**

- VM プロファイル変更: 中規模 → 小規模 VM × 多数で合計 CPU 確保（コア単価が安いケース）。Phase 2 で検討。
- 観測基盤最適化: Loki chunk 圧縮改善、Mimir long-term storage を NAS 経由に切替で MinIO 負担軽減。
- OSS 入替: ライセンス料の発生しない OSS 限定を維持、商用版への移行検討は Phase 3 以降。
- 確定フェーズ: Phase 2 で定期レビュー項目化。

## 容量監視ダッシュボード

**設計項目 DS-NFR-CAP-014 キャパシティダッシュボード**

- `k1s0-capacity-overview`: ノード CPU / メモリ / ディスク、テナント数、API RPS のトレンド（7 日 / 30 日 / 90 日）。
- `k1s0-capacity-forecast`: 3 ヶ月予測、閾値到達日の表示。
- `k1s0-capacity-cost`: 月次コスト実績、予算消費率、超過予測。
- 週次レビュー: 毎週月曜朝会で 15 分確認、月次レビュー: 月末 30 分。
- 確定フェーズ: Phase 1c で 3 ダッシュボード完備。

## スケール閾値の運用統合

スケール閾値を数値として定義するだけでは運用に落ちない。日々の監視ダッシュボード・アラート・運用リズム（朝会・月次レビュー）と閾値を紐付け、「閾値を超えたら誰が何をするか」を機械的に決めておくことで、2 名運用でも取りこぼしが起きない仕組みに仕立てる。

**設計項目 DS-NFR-CAP-015 スケール閾値の運用プロセス統合**

- 閾値超過の検知: DS-NFR-CAP-007 で定義した SEV 値を Alertmanager のルールに 1:1 で紐付け、Slack `#k1s0-capacity` に通知。
- 朝会組込: 毎週月曜朝会 15 分枠で `k1s0-capacity-overview` を読み合わせ、Warning 状態の項目を全員で確認、対応オーナーを決める。
- 月次レビュー: 月末 30 分の Capacity Review で、直近 30 日の閾値超過件数・継続時間・原因を Product Council に報告する。3 件以上 Warning が継続した項目は翌月の改善候補に昇格。
- Runbook 紐付: 各 SEV レベルの対応手順を Runbook `RB-CAP-001 〜 005` として整備し、DS-NFR-CAP-007 の Warning / SEV2 / SEV1 の 3 レベルと 1:1 で対応。
- 確定フェーズ: Phase 1b で朝会・月次レビューを運用開始、Phase 1c で Runbook 完備。

## ハードウェア先行発注プロセス

オンプレ運用では VM / 物理機材の調達リードタイムが長く、閾値超過を検知してからでは間に合わない。稟議と調達のリードタイムを逆算し、3 か月前には増設判断を確定する先行発注プロセスを制度化する。

**設計項目 DS-NFR-CAP-016 先行発注リードタイム設計**

- 調達リードタイム: VM 増設稟議 1 か月、調達・構築 1 か月、検証 1 か月、合計 3 か月を標準とする。
- 発注トリガー: DS-NFR-CAP-011 の 3 ヶ月先予測モデルで CPU 70% / メモリ 80% / ディスク 70% のいずれかを超える見込みが検出された時点で、Product Council に発注稟議を提出する。
- 稟議テンプレート: Backstage TechDocs に `capacity-expansion-ringi-template.md` を配置し、予測グラフ・増設スペック・コスト見積・代替案検討結果の 4 項目を必須記載とする。
- 緊急時対応: 予測を超える急増（月次成長率 50% 超）が発生した場合、緊急稟議（48 時間以内承認）を発動し、既存 VM 基盤のバッファから一時的に借用する暫定対応を並走させる。
- 確定フェーズ: Phase 1c で稟議テンプレート整備、Phase 2 で完全自動化（Backstage Form + Jira Service Desk 連携）。

## 年次キャパシティ再評価

Phase 別キャパシティ見込みは初期設計時点の仮定に基づいており、運用開始後の実績と乖離する可能性が高い。年次で見込みを再評価し、企画書の TCO コミットと整合させる継続サイクルを組む。

**設計項目 DS-NFR-CAP-017 年次キャパシティ再評価サイクル**

- 実施時期: 毎年 4 月（JTC 会計年度開始月）に前年度実績を集約し、翌年度の Phase 別キャパシティ見込みを再評価する。
- 評価指標: テナント増加率、ユーザー数増加率、RPS 増加率、ストレージ増加率の 4 指標で、当初見込みとの乖離率を算出する。
- 見直しトリガー: 乖離率が +30% を超える場合、Phase 計画の繰り上げまたは TCO 見直しを Product Council に提起する。-30% を下回る場合は過剰投資の是正を検討する。
- 予算調整: 再評価結果は次年度予算稟議の根拠として稟議書に添付する。これにより、キャパシティ予測と稟議予算が年次で連動する。
- 確定フェーズ: Phase 1c で初回実施、継続運用。
