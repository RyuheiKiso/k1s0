# 08. データ保護とプライバシー方式設計

本ファイルは要件定義書 [../../03_要件定義/30_非機能要件/G_データ保護プライバシー/](../../03_要件定義/30_非機能要件/G_データ保護プライバシー/) の NFR-G 系 15 要件を受けて、個人情報保護法 2022 改正に準拠したデータ保護・プライバシー方式を確定する。PII 分類・マスキング・DSAR（Data Subject Access Request）・越境移転・保管期限を設計する層である。

## 本ファイルの位置付け

個人情報保護は法令要件であり、違反時の罰則（最大 1 億円以下の罰金）と評判毀損が経営リスクになる。「PII を暗号化する」では足りず、「どのデータが PII か」「誰が、いつ、どう扱ったか」を構造的に記録し、本人からの削除要求に 30 日以内に応答できる仕組みが必要である。本章はこれを tier1 の Audit-Pii API と監査ログの組み合わせで実現する。

個人情報保護法 2022 改正で強化された 3 点（漏えい報告義務、仮名加工情報、越境移転の明示化）に全て対応する。

## データ分類方式

データ分類は 5 区分で運用する。分類はデータ辞書 [../../03_要件定義/00_要件定義方針/07_データ辞書.md](../../03_要件定義/00_要件定義方針/07_データ辞書.md) で全フィールドに付与され、tier1 が自動で検出・扱いを決定する。

**設計項目 DS-NFR-PRV-001 データ分類 5 区分**

- 公開（Public）: 外部公開可能。API 仕様、マニュアル、製品情報。暗号化不要、誰でもアクセス可。
- 内部（Internal）: 社員限定。運用手順、システム構成。社内 LAN 内のみ、LDAP 認証必須。
- 機密（Confidential）: 限定メンバー。脅威モデル、シークレット、契約情報。OpenBao / Git private repo、監査ログ必須。
- 個人情報（Personal）: 個人情報保護法対象データ（氏名 / 住所 / 電話 / メール / 生年月日 等）。Audit-Pii API 経由必須、全アクセスを監査、マスキング自動化。
- 特定個人情報（Special）: マイナンバー（番号法適用）。専用 Vault、アクセスは番号取扱責任者のみ、二重承認フロー。

各フィールドは 1 分類のみを持つ。分類変更はデータ辞書 PR として Product Council 承認が必要。

- 確定フェーズ: Phase 1b で分類ポリシー確定、Phase 1c で全フィールドに付与完了。

## PII 自動検出と分類

**設計項目 DS-NFR-PRV-002 PII 自動検出**

- 静的検出: データ辞書で分類済みフィールド名と正規表現（メール `.+@.+\..+`、電話 `0\d{1,4}-\d{1,4}-\d{4}` 等）で検出。
- 動的検出: tier1 Audit-Pii API がリクエスト/レスポンスを走査し、未登録 PII を検出 → SEV3 アラート + データ辞書追加 PR 自動起票。
- 誤検知抑制: 業務上の非 PII 文字列（例: サポート ID でメール形式に見える）は allowlist で除外。
- 処理性能: Rust 自作領域で実装、1 レコードあたり p99 1ms。
- 確定フェーズ: Phase 1c で実装。

**設計項目 DS-NFR-PRV-003 PII 自動マスキング**

- マスキング対象: ログ出力、監査ログ出力、エラーレスポンス、Synthetic 監視の記録。
- マスキングルール: メール `a***@example.com`、電話 `***-****-1234`（下 4 桁）、氏名 `山***`（姓 1 文字 + 星印）、住所 `東京都港区***`（市区町村まで）。
- 処理場所: tier1 API の出力前、Audit-Pii API 経由で一括処理。
- マスク解除: 業務上必要な場合の API あり、要求記録を監査ログ化（誰が、いつ、なぜ解除したか）。
- 確定フェーズ: Phase 1c で実装。

## DSAR（Data Subject Access Request）対応

個人情報保護法では本人から「私の情報を開示せよ」「削除せよ」「訂正せよ」という要求（DSAR）に 30 日以内に応答する義務がある。MVP 期は手動プロセスで運用、Phase 2 で自動化する。

**設計項目 DS-NFR-PRV-004 DSAR 受付 API**

- エンドポイント: tier1 Audit-Pii API の下位機能として、本人認証済みリクエストのみ受付。
- リクエスト種別: (1) 開示請求（Access）、(2) 訂正請求（Rectification）、(3) 削除請求（Erasure / Right to be Forgotten）、(4) 処理停止請求（Restriction）。
- 本人認証: Keycloak + MFA 必須。
- 受付番号発行: 自動採番、本人へメール通知。
- 確定フェーズ: Phase 1c 基本受付、Phase 2 自動化拡張。

**設計項目 DS-NFR-PRV-005 DSAR 処理 SLA**

- 受付応答: 3 営業日以内に受付確認。
- 本処理: 30 日以内（法定期限）。
- 30 日超過見込みの場合: 延長通知（最大 60 日まで）。
- 難易度: 削除請求は他システム（レガシー DB、バックアップ）への波及確認含むため最も難易度高。
- 担当: SecOps + 情シス管理責任者の 2 名承認、個別要求ごとに監査ログ記録。
- 確定フェーズ: Phase 1c 運用開始。

**設計項目 DS-NFR-PRV-006 削除要求の波及**

- 業務 DB: tier1 経由で即時削除、復元不可にするため物理削除（論理削除ではない）。
- バックアップ: barman cloud の 7 世代バックアップはそのまま保持（期限内にローテーションで自然消滅）。法解釈上バックアップからの即時削除は必須ではなく、次回バックアップローテーションで整合。
- 監査ログ: 監査ログそのものからは削除しない（J-SOX 要件との両立）。代わりに「削除要求が行われた事実」を別レコードで追加、元データは仮名化。
- 外部連携: レガシー DB（tier3 移行対象）など外部への波及は、連携先システムごとに個別対応、Runbook 化。
- 確定フェーズ: Phase 1c で Runbook、Phase 2 で自動化。

## 越境移転

個人情報保護法 2022 改正で、EU / 米国など海外への個人情報移転は本人同意または適切な同意書面が必要となった。k1s0 は MVP 期は国内完結のため該当しないが、Phase 3 の国際展開で必要。

**設計項目 DS-NFR-PRV-007 越境移転ポリシー**

- MVP 期（Phase 1a〜Phase 2）: 国内完結、越境移転なし。Container レジストリ（Harbor）も国内 MinIO に配置。
- 例外: CI/CD で GitHub（米国）に Git リポジトリを置くため、ソースコードに PII を含めないことを徹底。Renovate（米国サービス）は tier1 基盤の CVE DB アクセスのみで個人情報を扱わない。
- Phase 3 国際展開: 越境移転が必要な場合は GDPR + 個人情報保護法の適切な同意書を本人から取得、SCC（Standard Contractual Clauses）ベースで処理。
- 確定フェーズ: Phase 3 で本格対応。

**設計項目 DS-NFR-PRV-008 データレジデンシー**

- 本番データ: 国内データセンター内のみ保持（全 OSS のストレージが国内 DC）。
- バックアップ: 同 DC 内（Phase 1c）、別 DC 副拠点（Phase 2）、いずれも国内。
- 開発環境: dummy データのみ、本番 PII は開発環境に持ち出し禁止（CI で PII 検出 + block）。
- 確定フェーズ: Phase 1b 確定、Phase 1c 監査で確認。

## 仮名加工情報

個人情報保護法 2022 改正で導入された区分。個人を特定できない形に加工した情報で、本人同意なしに内部分析で利用可能。MVP 期は未使用、Phase 2 の BI / 分析基盤で活用検討。

**設計項目 DS-NFR-PRV-009 仮名加工情報の生成**

- 仮名化方式: 氏名・メール等を SHA-256 ハッシュ（salt 付き）で置換、住所は市区町村まで保持。
- 使用範囲: 社内分析、機械学習、統計情報。
- 別 DB 分離: 仮名加工情報は別スキーマに格納、元 DB との結合禁止。
- 監査: 仮名化ジョブは月次で実行、実行記録を監査ログ化。
- 確定フェーズ: Phase 2 導入。

## 同意管理

個人同意の取得と記録は監査証跡と一体で運用する。同意の取得・変更・撤回を全て記録し、本人からの開示要求に応じて提示できる。

**設計項目 DS-NFR-PRV-010 同意管理**

- 同意記録: 利用規約・プライバシーポリシーへの同意を本人認証と組み合わせて記録、監査 DB に永続化。
- 変更履歴: 同意内容の改定（新バージョン適用）時に同意取得、古いバージョンの同意記録も保持。
- 撤回: 本人からの撤回要求は 3 営業日以内に処理、以降の処理を停止。
- 可視化: 本人が自分の同意履歴を確認できる API（Phase 2）。
- 確定フェーズ: Phase 1c 基本、Phase 2 拡張。

## データ持ち出し制御

内部からのデータ持ち出しは多くの情報漏洩事件の原因である。tier1 API 経由のみの強制、直接 DB アクセス禁止で構造的に防ぐ。

**設計項目 DS-NFR-PRV-011 直接 DB アクセス禁止**

- PostgreSQL: tier1 サービスアカウント以外は接続不可、NetworkPolicy + pg_hba.conf で制限。
- 運用 SQL: SRE が障害調査時に必要な場合は Read-Only ロールで OpenBao 経由のワンタイムパスワード発行、使用を監査ログ化。
- 一括エクスポート: tier1 API に bulk export エンドポイントを用意、制限付き（レート・サイズ）で提供。直接 `pg_dump` は禁止。
- 確定フェーズ: Phase 1b 実装。

**設計項目 DS-NFR-PRV-012 DLP 機能（Phase 3）**

- tier1 API の出力監視で、大量 PII の外部送信を検知 → 自動ブロック。
- メール・ファイル共有サービスへの PII 流出検知は Phase 3 で企業全体の DLP 製品と連携検討。
- MVP 期は代替として、Synthetic で定期的に bulk export が業務外使用されていないか監査。
- 確定フェーズ: Phase 3。

## データ保管期限

保管期限は業務要件と法令要件で決まる。古いデータを不必要に持ち続けることは漏洩リスクを上げるため、定期的な削除を自動化する。

**設計項目 DS-NFR-PRV-013 データ保管期限**

- 業務個人情報: 3 年（業務での利用実績がない場合は削除）。
- 監査ログ: 7 年（J-SOX / 会社法 要件）。
- 運用ログ（非個人情報）: 90 日。
- バックアップ: 日次 7 世代、週次 4 世代、月次 12 世代。最長 12 ヶ月。
- 削除方式: cronjob で期限超過データを検索、Audit-Pii API 経由で削除（仮名化併用）。削除結果を監査ログ化。
- 確定フェーズ: Phase 1c ポリシー実装、Phase 2 自動化完成。

**設計項目 DS-NFR-PRV-014 保管期限超過データの扱い**

- 期限超過検知: 日次バッチで検知、超過データのリストを SecOps に通知。
- 削除 or 仮名化: SecOps 判断で削除（復元不可）または仮名加工情報化（統計利用のため保持）。
- 保持延長: 法的紛争等で保持義務がある場合は特例で延長、根拠と期限を監査ログに明記。
- 確定フェーズ: Phase 1c。

## 漏えい対応

個人情報保護法 2022 改正で漏えい時の報告義務が強化された。3 日以内に個人情報保護委員会へ速報、30 日以内に確報の提出が必要となる。

**設計項目 DS-NFR-PRV-015 漏えい時報告プロセス**

- 検知: 認証ログ異常 / Falco / 監査ログ大量 PII アクセスで検知 → SEV1。
- 即応: SRE + SecOps が 1 時間以内に事象確認、Product Council エスカレーション。
- 速報: 漏えい確認から 3 日以内（72 時間以内）に個人情報保護委員会へ速報。
- 本人通知: 影響範囲確定後、7 日以内に本人通知（メール + Web 掲示）。
- 確報: 30 日以内に原因・対策・再発防止策を含む確報を提出。
- 確定フェーズ: Phase 1c で Runbook `RB-INCIDENT-LEAK-001` 整備。

## GDPR 対応（Phase 3）

Phase 3 国際展開時には EU 圏のユーザデータを扱う可能性があり、GDPR への対応が必要になる。個人情報保護法との差分を整理しておく。

**設計項目 DS-NFR-PRV-016 GDPR 対応ロードマップ（Phase 3）**

- 個人情報保護法との共通点: 同意、開示、削除、越境移転の基本枠は共通。
- 差分: GDPR は罰金が高額（売上 4% or 2000 万ユーロのいずれか高い方）、Data Protection Officer（DPO）任命義務、72 時間報告義務（個人情報保護法と同じ）。
- MVP 期は適用対象外、Phase 3 で DPO 任命、GDPR 準拠のプライバシーポリシー整備。
- 確定フェーズ: Phase 3。

## 4 段階機密度分類方式

要件定義書 G1 章（NFR-G-CLS 系）は「全データを Public / Internal / Confidential / Restricted の 4 段階に分類し、ストレージ・アクセス制御・暗号化・保管期間・バックアップを差別化する」原則を定める節である。DS-NFR-PRV-001 の 5 区分（Public / Internal / Confidential / Personal / Special）は要件定義書の 4 段階に Personal / Special を細分したものだが、要件定義書の 4 段階分類そのものを設計側で明示することで、データ辞書と整合させる。

**設計項目 DS-NFR-PRV-017 4 段階機密度分類**

データ辞書 [../../03_要件定義/00_要件定義方針/09_データ辞書.md](../../03_要件定義/00_要件定義方針/09_データ辞書.md) の全フィールドに 4 段階分類ラベル（Public / Internal / Confidential / Restricted）を付与する。Public は社外公開可能な SBOM や公開 API ドキュメント、Internal は構成情報・ADR・運用手順、Confidential は業務データ・設計書・取引情報、Restricted は個人情報・認証情報・KMS 鍵・要配慮個人情報。各段階で暗号化要件・アクセス制御ロール・保管期間・バックアップ世代数を差別化する。本設計の DS-NFR-PRV-001（5 区分）は Restricted をさらに Personal / Special に細分した運用実装であり、要件定義書の 4 段階を内包する。Phase 1a で分類ルール確定、Phase 1c 全データセットに付与完了、四半期ごとに分類レビュー。

**設計項目 DS-NFR-PRV-018 PII 検出と自動タグ付け**

個人情報の所在を可視化するため、3 層で PII を検出・タグ付けする。第一層は入力時検出で、tier1 Audit-Pii API がリクエストボディ中の氏名・メール・電話・マイナンバーの正規表現パターンを走査、検出フィールドに PII タグをメタデータ付与（DS-NFR-PRV-002 の静的・動的検出を本 ID でも強化）。第二層は月次定期スキャンで、PostgreSQL の全テーブル / MinIO の全バケット / Loki ログに対して PII スキャンジョブを実行、発見時に SEV2 アラート + データ辞書追加 PR 自動起票。第三層は PII 含有率レポートで、月次で各ストレージの PII 密度を Grafana ダッシュボードに表示、異常増加時に SecOps レビュー。Phase 1b で API 稼働、Phase 1c で定期スキャン開始。

## 暗号化方式

要件定義書 G2 章（NFR-G-ENC 系）は保存時（at-rest）・転送時（in-transit）・アプリケーション層の 3 層暗号化を定める節である。単層の暗号化ではストレージ盗難・中間者攻撃・特権者誤参照のいずれかで漏えいが発生するため、層別に鍵管理と適用範囲を固定する。

**設計項目 DS-NFR-PRV-019 保存時暗号化（at-rest）**

ストレージ盗難やマルウェア侵入時の情報漏えいを防ぐため、全ストレージに保存時暗号化を適用する。PostgreSQL（CloudNativePG）は TDE 相当またはディスク FDE（LUKS）、MinIO は Server-Side Encryption（SSE-S3 相当）で鍵は OpenBao 管理、Kafka はブローカーディスク暗号化（Phase 3+ で topic 内暗号化）、Valkey は永続化ディスク FDE、バックアップは AES-256 で暗号化し鍵を別保管、Restricted を含むログは Loki 側でも暗号化。鍵管理は OpenBao + KMS Operator（cert-manager + 自作 controller）で 90 日ローテーション、鍵ローテーション時のダウンタイム 0 を目標。Phase 1a で全 VM FDE、Phase 1b で PostgreSQL TDE / MinIO SSE 有効化。

**設計項目 DS-NFR-PRV-020 転送時暗号化（in-transit）**

通信経路の盗聴・改ざん防止のため、全通信を TLS 化する。tier1 ↔ tier2 ↔ tier3 は Istio Ambient mesh の mTLS 自動適用（SPIFFE 証明書、24 時間ローテーション）、tier1 ↔ データベースは TLS 1.3 必須、tier1 ↔ Keycloak 等外部は TLS 1.3 必須、Ingress は TLS 1.2 最低・1.3 推奨で cert-manager + Let's Encrypt 自動取得、Egress は Envoy で TLS 検証強制。TLS 1.0 / 1.1 / SSL 全版本および弱い暗号スイート（RC4、3DES、NULL cipher）は Kyverno で構造的に禁止、四半期ごとに TLS バージョン監査を実施。Phase 1a 全通信 TLS 化。

**設計項目 DS-NFR-PRV-021 アプリケーションレベル暗号化**

DBA / SRE が業務で見るべきでない要配慮個人情報を誤って閲覧しないよう、ストレージ暗号化に加えてアプリケーション層で暗号化する。対象は要配慮個人情報（健康・信条・犯罪歴）、マイナンバー、認証情報（Argon2id / bcrypt ハッシュ、平文保存禁止）。フィールドレベル暗号化は PostgreSQL の pgcrypto を活用、特定カラムのみ暗号化、鍵は OpenBao に保管してアプリは KMS API 経由でアクセス。鍵アクセスは MFA + 2 人承認フロー必須、アクセス記録は全て監査ログ化。Phase 1c で必要データに実装、年次で暗号化実装の外部監査。

## データ中心アクセス制御方式

要件定義書 G3 章（NFR-G-AC 系）は「開発者・運用者の本番データ参照権を最小化し、非本番環境では本番データを使わせない」方針を定める節である。特権アクセスは監査の焦点となるため、構造的に 2 人承認と時限付与で運用する。

**設計項目 DS-NFR-PRV-022 データ中心の最小権限原則**

内部不正・誤操作リスクを下げるため、データ閲覧権限を業務必要最小に絞る。本番データ参照は業務上必要な場合のみ 2 人承認で時限的（最大 8 時間）に付与、期限経過で自動剥奪。DBA 権限は構造変更とバックアップのみ、データ SELECT は原則禁止（障害調査時は Read-Only ロールを OpenBao 経由でワンタイム発行）。テナント越境アクセスはデフォルト禁止、例外は監査ログ必須。PII を含むログは特定ロール（SecOps）のみ閲覧可。KMS 鍵の利用は cell（暗号化単位）別に RBAC。90 日未使用の権限は自動失効、特権アカウント数を四半期レビュー。Phase 1b で承認フロー稼働。

**設計項目 DS-NFR-PRV-023 データマスキング方式**

開発者 PC 経由の本番個人情報流出を防ぐため、非本番環境では本番データのコピーを構造的に禁止する。検証・ステージング環境は合成データ（Faker 等）またはマスク済みデータのみ、本番 DB への直接 pg_dump も禁止（DS-NFR-PRV-011 と連動）。マスキング手法は氏名を仮名（日本人風のランダム氏名）、メールを `masked_<hash>@example.com`、電話を `XXX-XXXX-XXXX`、住所を都道府県のみに縮約、統計分析向けは k-匿名性（k ≥ 5）を保証。マスキングパイプラインは pgloader + カスタム変換で Phase 2 整備。非本番環境への本番データ流入を月次監査（ハッシュ比較で検知）。

## データ所在地方式

要件定義書 G4 章（NFR-G-RES 系）は「個人情報保護法 28 条の越境移転制約に抵触しないため、全データを国内保管とする」方針を定める節である。DS-NFR-PRV-008（データレジデンシー）と重なるが、要件定義書 ID への明示紐付けのため本節で独立採番する。

**設計項目 DS-NFR-PRV-024 国内保管の徹底**

全 tier1 API のデータ・バックアップ・ログ・メトリクス・Secret（KMS）を JTC 国内 DC のみで保管し、海外クラウド・海外 DC の利用を禁止する。バックアップは DR サイトも国内 DC、Harbor コンテナレジストリの MinIO も国内配置、CI/CD の GitHub（米国）にはソースコードのみで PII を一切持ち出さない。例外は Phase 4+ の海外拠点展開のみで、現地データは現地保管可とするが越境時は個人情報保護法 28 条対応（本人同意または十分性認定国）を法務と整備。四半期監査で全データ所在を確認、DS-NFR-PRV-008 と重層化。Phase 1a 所在地ポリシー確定。

## 保管期間とライフサイクル方式

要件定義書 G5 章（NFR-G-LIF 系）は「データ種別ごとに保管期間を定め、超過データを自動削除またはテナント退会時に完全削除する」方針を定める節である。不要データの蓄積は漏えい時の影響範囲を拡大させるため、削除も業務プロセスとして自動化する。

**設計項目 DS-NFR-PRV-025 標準保管期間の適用**

データ種別ごとの保管期間を以下に固定し、超過時の自動削除を運用する。業務データは業務要件に応じる（例: 取引データ 10 年、顧客データ 5 年）、監査ログは 7 年（J-SOX / 個人情報保護法 / 社内規程）、アプリケーションログは Hot 90 日 / Warm 1 年、メトリクスは高解像度 15 日 / 中 1 年 / 低 3 年、バックアップは日次 30 日 / 月次 1 年 / 年次 7 年。保管期間超過データは cronjob で検索して Audit-Pii API 経由で削除または匿名化、削除実行ログを監査ログに保全。DS-NFR-PRV-013（保管期限）を再整理し要件定義書 ID に紐付け。Phase 1c 自動削除パイプライン稼働、年次で保管期間見直し。

**設計項目 DS-NFR-PRV-026 テナント退会時の完全削除**

テナント退会後のデータ残存は個人情報保護法の目的達成後消去義務違反のため、退会から 180 日以内に全データを完全削除する。T+30 日でテナント側データ退避完了、T+90 日で名前空間内全データ削除（PostgreSQL 行の物理削除、MinIO オブジェクト削除、Valkey キー削除）、T+90 日でバックアップからのテナントデータ削除（次回バックアップ世代から不在）、T+180 日で監査ログからのテナント名前解除（監査ログ自体は 7 年保持、テナント識別のみ仮名化）。削除完了時に完全削除証明（削除対象ID / 削除日時 / 実施者の監査ログ）をテナントに送付。Phase 1c で退会削除フロー Runbook 化、年次でリハーサル、削除証明の法務レビュー。

## 本人権利対応方式

要件定義書 G6 章（NFR-G-PRV 系）は「本人からの開示・訂正・削除・利用停止請求に 30 日以内で対応する」義務を定める節である。DS-NFR-PRV-004 〜 006 で DSAR 全体を設計済みだが、NFR-G-PRV-002（訂正・削除請求）の個別 ID 紐付けを明示する。

**設計項目 DS-NFR-PRV-027 訂正・削除・利用停止請求対応**

本人からの訂正請求はデータ訂正を実施し履歴を監査ログに残す（訂正前後の値、実施日時、依頼者、実施者を記録）。削除請求は個人情報保護法の条件（目的達成・不正取得・本人同意撤回）を満たす場合に削除し、監査ログは仮名化して保持（J-SOX 要件との両立）、バックアップは次回ローテーションで自然消滅。利用停止請求（マーケティング停止等）は対象テナントに停止タグを付与して該当処理を停止、撤回は本人からの撤回通知で即時処理。受付から 30 日以内に対応、延長が必要な場合は最大 60 日で本人に延長通知。法令遵守の外部監査を年 1 回、対応実績を月次で法務部に報告。DS-NFR-PRV-005 / 006 と連携して Phase 1c 運用開始。

## プライバシー・バイ・デザイン方式

要件定義書 G7 章（NFR-G-DES 系）は「収集時点から最小化原則を適用し、同意を構造的に管理する」方針を定める節である。設計段階で最小化を仕込まないと、収集時の「念のため」が保護対象を膨張させる。

**設計項目 DS-NFR-PRV-028 最小化原則の実装**

収集・保持・利用・提供の各ステップで最小化を強制する。収集は業務目的に必要な最小限のデータのみ、テナントオンボーディング時に収集意図を Product Council がレビュー。保持は保管期間を明示（DS-NFR-PRV-025 と整合）、超過時に自動削除。利用は収集時の目的範囲内で利用、新用途は本人同意取得を必須とし監査ログに記録。提供は第三者提供を本人同意または法定要件（警察令状等）に限定、提供記録を永続保全。Phase 1b で最小化ガイドライン確定、テナントオンボーディングのレビューフローに組込、年次棚卸しで保持データの業務必要性を再確認。

**設計項目 DS-NFR-PRV-029 同意管理の構造化**

同意範囲外の処理で監督官庁指摘を受けないよう、同意の取得・変更・撤回を構造的に記録する。同意レコードは「誰が、いつ、何について、どのバージョンの利用規約に同意したか」を監査 DB に永続化（DS-NFR-PRV-010 と一体）。撤回機能は本人が Backstage のセルフサービス画面から撤回可能、撤回時は該当処理を 3 営業日以内に停止。透明性確保のため、プライバシー通知を Backstage で公開し変更時に本人通知（メール + Web 掲示）。子供の個人情報（16 歳未満）は保護者同意が必須で、該当フラグ付きレコードは別フローで承認。Phase 1c 基本実装、プライバシー通知の法務レビュー、同意撤回率を Grafana で月次モニタリング。

## 国際規制対応方式

要件定義書 G8 章（NFR-G-INT 系）は「Phase 4+ の海外拠点展開時に後付けで再設計が発生しないよう、GDPR / EU AI Act / DORA の技術要求を先行実装する」方針を定める節である。規制別に施行時期と適用トリガが異なるため、対応時期を Phase ゲートで固定する。

**設計項目 DS-NFR-PRV-030 GDPR 対応の先行実装**

Phase 4+ の EU 展開時に k1s0 基盤の再設計が発生しないよう、GDPR 技術要件を Phase 1〜3 の段階で先行実装する。データ分類（DS-NFR-PRV-017）・暗号化（DS-NFR-PRV-019〜021）・削除フロー（DS-NFR-PRV-026）・本人権利対応（DS-NFR-PRV-027）で GDPR の 80% カバレッジを Phase 1c までに確認し、Phase 4 判定時に法務レビューで残 20%（DPO 任命 / DPIA 実施 / EU 代理人指定）の対応計画を策定。組織面の要求事項は Phase 4 の海外展開判定と同時着手で、Phase 1〜3 段階では技術要件の先行実装に限定する。72 時間以内の漏えい通知は DS-NFR-PRV-015（漏えい対応）で既に対応済み。

**設計項目 DS-NFR-PRV-031 EU AI Act 対応（Decision API 高リスク用途）**

Decision API（ZEN Engine）の EU AI Act 抵触リスクを管理するため、用途申告と Audit 強化の 3 段階対応を運用する。Phase 1a は利用登録時の用途申告を必須化し、「採用・信用評価・公的サービス・医療・重要インフラ」に該当する場合は Product Council 承認、非該当用途のみ自動承認。Phase 2 は高リスク用途の申告テナント向けに EU AI Act Annex III 準拠の Audit 強化（入力・出力・ルールバージョン・評価時刻の完全記録、7 年保管）を提供。Phase 3+ で EU 域内テナント受入判定時にリスク管理システム・人間による監視・適合性評価を実装、外部専門家レビューを受ける。Decision API Audit は NFR-E-MON-001 の 7 年保管で AI Act の 6 か月要件を上回る水準を既に確保済み。

**設計項目 DS-NFR-PRV-032 DORA 対応（金融テナント受入時）**

EU 域内金融機関テナントが k1s0 を採用した場合に DORA 上の「重要 ICT 第三者サービス提供者」として対応するため、現行要件のギャップを事前に把握する。ICT リスク管理は FMEA（40_運用ライフサイクル/06_FMEA分析.md）と STRIDE で対応済み、レジリエンステストは復旧訓練年 2 回（DS-NFR-DR-007）とカオス試験で対応済み、インシデント通知は 72 時間以内の当局通知フロー（OR-INC-005）で対応済み、契約要件は BC-LGL-005 で対応済み。Phase 1c までに現行要件の DORA ギャップ分析を完了し、Phase 3+ の金融テナント受入判定時に追加契約条項（DORA 第 30 条の ICT 契約最小要件）と第三者監査受入プロセスを法務と整備。年 1 回のデジタル運用レジリエンステストを標準 Runbook（OR-INC-007）に組込み。
