# 09. 完整性とコンプライアンス方式設計

本ファイルは要件定義書 [../../03_要件定義/30_非機能要件/H_完整性コンプライアンス/](../../03_要件定義/30_非機能要件/H_完整性コンプライアンス/) の NFR-H 系 12 要件を受けて、J-SOX 内部統制・電帳法・PCI DSS スコープ・監査対応を確定する。データの完整性（整合性・改ざん耐性・証跡保全）と法令・業界規制への準拠方式を設計する層である。

## 本ファイルの位置付け

完整性・コンプライアンス方式は監査対応の生命線である。「記録がある」では足りず、「改ざんされていないことを証明できる」「誰がいつ変更したかを復元できる」「監査役の質問に数時間以内に証跡を提示できる」状態が必要である。本章は、監査ログのハッシュチェーン、GitOps による変更統制、SBOM による供給連鎖証跡、四半期レポートの自動生成を組み合わせ、JTC 情シス部門が外部監査を滞りなく通過できる方式を確定する。

監査対応で工数超過を避けるため、証跡は日常運用の副産物として自動蓄積される設計とする。監査の期ごとに手作業で集める方式は起案者 + 協力者 2 名では破綻する。

## J-SOX 内部統制対応

J-SOX（金融商品取引法 2008）は上場企業の内部統制報告を義務化している。k1s0 を採用する JTC 企業は、システムによる内部統制を外部監査人に検証される。システム面で求められる 4 統制（アクセス統制 / 変更統制 / 運用統制 / プログラム統制）に対応する。

**設計項目 DS-NFR-COMP-001 アクセス統制**

業界では権限管理が Excel 台帳で属人化し、退職者権限の残存・職務分離（SoD）違反が J-SOX 監査で毎期指摘される定番論点である。本設計では Keycloak OIDC による IdP 集約と匿名アクセス禁止、Backstage 経由の申請 → SRE 承認フロー、HR 連携による退職者即時無効化（日次ジョブ）、Kyverno による SoD 強制で構造的に統制する。崩れた場合は J-SOX アクセス統制「有効でない」判定で内部統制報告書訂正 + 金商法 24 条の 4 の 4 違反による課徴金 + 監査法人からの監査意見不表明で、上場維持要件に影響する。

- 認証: Keycloak OIDC、全人間ユーザを IdP で集約、匿名アクセス禁止。
- 認可: RBAC、最小権限原則、年次レビュー必須。
- 権限付与プロセス: Backstage 経由の申請 → SRE 承認 → 自動付与、全履歴を監査ログ化。
- 権限剥奪: 退職者は HR 連携で即時無効化、無効化ジョブは日次実行。
- SoD（職務分離）: 承認者と実行者の分離、1 人が両方を担うことを Kyverno で禁止。
- 確定フェーズ: Phase 1b で基本、Phase 1c で SoD 強制。

**設計項目 DS-NFR-COMP-002 変更統制**

業界では手動 kubectl apply や ssh 直接変更が横行し、変更履歴の追跡不能で J-SOX 変更統制が崩壊する事例が多い。本設計では GitOps 単一経路（Git PR → 承認 → Argo CD sync）を強制し、通常 1 名 / 標準 2 名 / 緊急は事後承認可だが 48 時間以内に事後レポート義務化、Git コミット履歴 + Argo CD イベントログ + Kubernetes Audit Log の 3 層で相互突合可能にする。崩れた場合は J-SOX 変更統制「有効でない」判定 + 監査法人からの内部統制監査意見不表明 + 会社法 435 条の計算書類への重要な虚偽表示リスクで、上場企業として致命的な指摘となる。

- GitOps 単一経路: 全変更は Git PR → 承認 → Argo CD sync の経路を通る。[03_運用方式設計.md](03_運用方式設計.md) DS-NFR-OPS-013 と連動。
- PR 承認: 通常 1 名、標準 2 名、緊急は事後承認可だが 48 時間以内に事後レポート。
- 変更記録: Git コミット履歴 + Argo CD イベントログ + Kubernetes Audit Log の 3 層で記録、相互突合可能。
- 証跡保管: 7 年（会社法要件）。
- 確定フェーズ: Phase 1b 運用開始。

**設計項目 DS-NFR-COMP-003 運用統制**

業界では運用作業が bash 直叩きと Slack メモで進み、J-SOX 監査時に「誰が何をいつ実施したか」の証跡が揃わず追加調査で数百工数を消費する事例が典型である。本設計では全運用タスク（バックアップ / 復旧 / CVE 対応 / 計画停止）を Runbook 単位で自動蓄積、Kubernetes Audit Log で実行主体を追跡、off-the-record な bash 直叩きは Kyverno で遮断、例外処理は事前承認 + 2 重承認で記録する。崩れた場合は J-SOX 運用統制「有効でない」判定 + 監査工数 3 倍増で年間 3,280 万円削減効果が相殺され、内部統制報告書訂正で経営層の説明責任が発生する。

- 運用記録: 全運用タスク（バックアップ / 復旧 / CVE 対応 / 計画停止）は Runbook 単位で実行記録を自動蓄積。
- 実行主体の記録: 誰が何を実行したかを Kubernetes Audit Log で追跡、bash 直叩き等の off-the-record 操作は Kyverno で遮断。
- 例外処理の記録: 手順書外の処理が必要な場合は事前承認 + 実行時記録、2 重承認必須。
- 確定フェーズ: Phase 1c。

**設計項目 DS-NFR-COMP-004 プログラム統制（開発統制）**

業界では本番リリース承認が非公式チャットで行われ後追い証跡を作るため、J-SOX プログラム統制の監査時に「承認エビデンスがない」と指摘される事例が繰り返される。本設計では GitHub Actions + Argo CD の CI/CD で全ビルドに署名済みイメージ生成、ユニット / 結合 / E2E テスト必須 + カバレッジ 80% 以上、Trivy / Grype / Semgrep で Critical 検出時は PR block、本番リリースは Product Council Chair 承認（GitHub PR の承認記録のみ有効）で統制する。崩れた場合は J-SOX プログラム統制「有効でない」判定 + 金商法内部統制評価の不適正意見 + 監査法人との契約リスクに直結する。

- CI/CD: GitHub Actions + Argo CD、全ビルドは署名済みイメージ生成。
- テスト: ユニット / 結合 / E2E が PR マージ前に必須、カバレッジ 80% 以上。
- セキュリティスキャン: Trivy / Grype / Semgrep が PR 毎実行、Critical 検出で block。
- リリース承認: 本番リリースは Product Council Chair 承認、承認記録は GitHub PR のみ有効。
- 確定フェーズ: Phase 1b 基本、Phase 1c 完備。

## 証跡と監査ログ

完整性の核心は「証跡が後から改ざんされていないことを証明できる」ことである。ハッシュチェーンで改ざん検知する。

**設計項目 DS-NFR-COMP-005 監査ログ ハッシュチェーン**

業界では監査ログが単純な append-only ファイルで保管され、DBA が事後改ざんしても検知できない構造的弱点を抱える例が多い。本設計では各監査ログレコードに前レコードの SHA-256 ハッシュを含めるハッシュチェーン + 監査専用 PostgreSQL + MinIO Object Lock の二重化で、週次ハッシュ全走査で不整合検知時に SEV1 発動、Phase 2 で TSA タイムスタンプ付与して電帳法対応を兼ねる。崩れた場合は J-SOX 監査ログ完整性の「有効でない」判定 + 金商法 193 条の 3 違反 + 監査法人による再監査要求で、監査対応コストが数千万円規模で膨張する。

- 実装: 各監査ログレコードに前レコードのハッシュ（SHA-256）を含める。
- 保管: 監査専用 PostgreSQL（業務 DB と分離）+ MinIO Object Lock バケット。
- 整合性検証: 週次でハッシュチェーン全走査、不整合検知で SEV1。
- タイムスタンプ: 時刻認証局（TSA）のタイムスタンプ付与を Phase 2 で導入、電帳法対応と連動。
- 詳細: [../30_共通機能方式設計/04_監査証跡方式.md](../30_共通機能方式設計/04_監査証跡方式.md) 参照。
- 確定フェーズ: Phase 1c ハッシュチェーン、Phase 2 TSA 連携。

**設計項目 DS-NFR-COMP-006 WORM アーカイブ**

監査対応で「7 年保管したはずの証跡が改ざん / 消失していた」事案は業界で年数件報告され、J-SOX 監査の致命傷となる。本設計では MinIO Object Lock バケット（Governance モード 7 年保持）に監査 DB から日次バッチで転送、転送後は書き換え不可を物理的に強制する（企画書 11 項目「監査 7 年 WORM」の技術実装）。崩れた場合は J-SOX 監査ログ保全義務違反 + 会社法 435 条 4 項（計算書類等の保存義務）違反 + 個人情報保護法 26 条の 2（漏えい等報告関係の記録保全）違反で、三重の法令指摘を同時に受ける。

- 保管先: MinIO Object Lock バケット、Governance モード 7 年保持。
- 転送: 監査 DB から日次バッチで WORM バケットに送信、転送後は書き換え不可。
- 検索性: 監査ログの検索は日次転送前の監査 DB で実施、過去データは WORM バケットから都度取得（検索性能は劣るが法令要件のため許容）。
- 確定フェーズ: Phase 1c。

## 電帳法対応（Phase 3）

電子帳簿保存法は 2022 改正で電子取引データの保存要件が厳格化された。JTC 企業内部の経理連携が発生する Phase 3 で対応する。

**設計項目 DS-NFR-COMP-007 電帳法ロードマップ**

電子帳簿保存法 2022 改正で電子取引データのタイムスタンプ・検索・ディスプレイ出力要件が厳格化され、未対応企業は電子取引データが税務上無効扱いとなる。本設計では MVP 期（Phase 1a〜2）は経理連携対象外とし、Phase 3 で監査ログ基盤を電帳法要件に転用 + TSA タイムスタンプ導入 + 検索 API 追加を計画する。崩れた場合は電子帳簿保存法違反で青色申告承認取消 + 税務調査時の追徴課税 + 経理連携 tier2 が利用不能となり、事業範囲が制限される。

- 真実性の要件: タイムスタンプ付与、訂正・削除履歴の保全。
- 可視性の要件: 検索機能（取引年月日 / 金額 / 取引先）、ディスプレイ出力。
- MVP 期（Phase 1a〜2）: 対象外、経理連携は Phase 3 で検討。
- Phase 3 対応: 監査ログ基盤を電帳法要件にも転用、TSA タイムスタンプ導入、検索 API 追加。
- 確定フェーズ: Phase 3 でスコープ確定。

## PCI DSS スコープ

k1s0 は決済機能を直接持たないため、tier1 は PCI DSS スコープ外とする。決済連携 tier2（クレジットカード情報を扱う tier2）が PCI DSS スコープで、tier1 はそのトークンのみを扱う。

**設計項目 DS-NFR-COMP-008 PCI DSS スコープ明示**

業界では PCI DSS スコープ境界が曖昧で、本来スコープ外の基盤まで QSA 監査対象にされて監査費用が数千万円規模に膨張する事例がある。本設計では tier1 全 API と基盤 OSS は PCI DSS スコープ外、決済連携 tier2 のみスコープ内として SAQ-D 対応、tier1 はトークンのみ扱いカード番号は一切残さないスコープ分離設計を Phase 2 で QSA 監査により検証する。崩れた場合は PCI DSS Level 1 加盟店基準違反で決済代行契約解除 + 決済事業の停止 + 監査コスト増大で、5 年 TCO 3.68 億円の試算が全面崩壊する。

- スコープ外: tier1 全 API、基盤 OSS 全て。
- スコープ内: 決済連携 tier2（Phase 2 以降導入）がクレジットカード情報を扱う部分のみ、PCI DSS SAQ-D 対応で別途監査。
- スコープ分離設計: tier1 はトークン（カード番号ではなく）を受け取り、tier2 が決済代行 API に送信。tier1 内にカード番号は一切残さない。
- 分離検証: Phase 2 で決済連携 tier2 導入時に QSA（Qualified Security Assessor）監査。
- 確定フェーズ: Phase 2（決済 tier2 の対象化）。

## ISO 27001 / SOC 2（Phase 3）

外部顧客向け事業拡大時に ISO 27001 認証取得を目指す。MVP 期は対象外だが、設計段階で取得阻害要因を作らないようにする。

**設計項目 DS-NFR-COMP-009 ISO 27001 準備**

ISO 27001 認証取得を Phase 3 で急遽着手すると過去 2〜3 年分の設計遡及コストで 6〜12 か月の遅延を招き、事業拡大機会を逸するのが業界通弊である。本設計では Phase 1a 〜 Phase 2 時点で Annex A 93 コントロールのうちセキュリティ章・本章で大部分を対応済みとし、Phase 3 で不足項目（情報セキュリティ目標の定量化 + 全社意識向上プログラム）を整備する二段戦略を採る。崩れた場合は ISO 27001 認証取得遅延 + 外部顧客事業拡大時の RFP 要件未達で、Phase 3 以降の売上拡大計画が未達となる。

- 準拠すべきコントロール: ISO 27001 Annex A 93 コントロール。
- MVP 期対応状況: 多くのコントロールが本章・セキュリティ章で既に対応済み（情報セキュリティポリシー / アクセス制御 / 暗号化 / 物理セキュリティ / 運用セキュリティ / 供給者関係 / インシデント管理）。
- 不足項目: 情報セキュリティ目標の定量的測定と改善サイクル、全社的な情報セキュリティ意識向上プログラム。Phase 3 で整備。
- 確定フェーズ: Phase 3 認証取得。

## SBOM と供給連鎖

ライセンスコンプライアンスと脆弱性対応の両方で SBOM が必要である。詳細は [05_セキュリティ方式設計.md](05_セキュリティ方式設計.md) DS-NFR-SEC-014 で設計するが、本章は監査側の活用を定める。

**設計項目 DS-NFR-COMP-010 SBOM 活用**

Log4Shell 型の重大脆弱性発生時に SBOM が未整備だと影響範囲特定に数週間を要し、その間に攻撃被害が拡大する業界事例が頻出する。本設計では CI で Syft を自動実行して SPDX / CycloneDX 形式の SBOM を全リリースで Harbor artifact として保管、四半期ごとに ORT でライセンス分布監査、AGPL / BSL / SSPL / Commons Clause / 独自ライセンス誤混入を CI で block する。崩れた場合は OSS ライセンス違反訴訟（事業停止命令 + 損害賠償）+ 脆弱性対応遅延による SLA 99% 違反 + 監査時の SBOM 不提出による J-SOX 指摘で、複合リスクが同時発生する。

- 生成: CI で Syft 自動実行、SPDX / CycloneDX 形式。
- 監査対応: 外部監査時に全リリースの SBOM を提示可能。Harbor artifact として保管。
- ライセンス分布レポート: 四半期ごとに Syft + ORT（OSS Review Toolkit）でライセンス分布、監査リスト作成。
- 禁止ライセンス検知: AGPL / BSL / SSPL / Commons Clause / 独自ライセンスの誤混入を CI で block。
- 確定フェーズ: Phase 1b で CI 組込、Phase 1c で四半期レポート。

## 監査対応パック

監査対応で毎回証跡を探す手間を削減するため、標準化された「監査対応パック」を四半期ごとに自動生成する。

**設計項目 DS-NFR-COMP-011 四半期監査対応パック**

外部監査対応で毎回エビデンスを手集めする運用は、2 名運用体制では四半期ごとに数百工数を消費して本来業務を圧迫する。本設計では四半期末から 10 営業日以内に SLO / インシデント / 変更ログ / アクセス統制 / 脆弱性対応 / SBOM / 監査ログ整合性 / DR 演習 / OSS アップグレード履歴の 9 項目を Backstage portal に PDF + CSV 自動生成、Phase 2 で完全自動化する。崩れた場合は監査対応工数超過 + 監査法人の監査意見形成遅延 + J-SOX 内部統制評価遅延で有価証券報告書提出期限（決算後 3 か月）に間に合わなくなり、上場維持に影響する。

- 生成頻度: 四半期末から 10 営業日以内。
- 内容: (1) SLO レポート、(2) インシデントレポート、(3) 変更ログ（GitOps 履歴）、(4) アクセス統制レポート（権限付与・剥奪履歴）、(5) 脆弱性対応レポート、(6) SBOM・ライセンス分布、(7) 監査ログ整合性確認結果、(8) DR 演習結果、(9) OSS アップグレード履歴。
- 形式: PDF + 生データ CSV の 2 形式、Backstage portal に保管。
- 閲覧権限: Product Council + 指定監査役のみ、アクセスログは永続記録。
- 年次パック: 四半期パック 4 件 + 年次サマリ + 翌年改善計画を統合。
- 確定フェーズ: Phase 1c 初回生成、Phase 2 で完全自動化。

## 改ざん検知と完整性監視

**設計項目 DS-NFR-COMP-012 完整性監視**

業界では監査ログの改ざんが数か月発覚せず、気づいた時には被害範囲不明で対応が後手に回る事例が報告される。本設計では監査 DB ハッシュチェーンの週次検証 + Git main ブランチの force-push 禁止・署名付き tag + Cosign 署名 + Rekor 透明性ログ + Argo CD desired state の常時差分監視で完整性を多層防御する。崩れた場合は J-SOX 監査ログ完整性「有効でない」判定 + Cosign 署名偽造による SLSA Level 降格 + 金商法 193 条の 3 違反で、上場維持要件 + サプライチェーン責任の両面で重大な経営リスクが発生する。

- 監査 DB ハッシュチェーン: 週次検証、不整合で SEV1（改ざん疑い）。
- Git 履歴: main ブランチは force-push 禁止、tag は署名付き、不正改変は GitHub 側で検知。
- コンテナイメージ: Cosign 署名 + Rekor 透明性ログで改ざん検知。
- Argo CD desired state: Git と稼働状態の差分を常時監視、勝手な変更は即座に修復。
- 確定フェーズ: Phase 1c で全要素整備。

## サプライチェーン完全性の個別方式

要件定義書 H1 章（NFR-H-INT 系）はコンテナイメージ署名・SBOM 添付・ビルド Provenance・再現可能ビルドを定める節である。DS-NFR-COMP-010（SBOM 活用）で監査側の活用を設計済みだが、要件定義書の個別 ID（INT-002 / INT-003 / INT-004）への明示紐付けとして本節に生成・検証方式を独立採番する。詳細な CI 側の設計は [05_セキュリティ方式設計.md](05_セキュリティ方式設計.md) DS-NFR-SEC-014 系と相互参照する。

**設計項目 DS-NFR-COMP-013 SBOM 自動生成と添付**

Log4Shell 型脆弱性の影響特定を数時間以内に短縮するため、全アーティファクトに SBOM を自動生成して添付する。フォーマットは CycloneDX（CNCF 標準）、生成は CI/CD のビルドステップで Syft が自動実行、OCI Artifact としてイメージに添付、Sigstore で SBOM 本体にも署名、Backstage SBOM 検索プラグインでイメージ別 / 依存パッケージ別に逆引き可能。Phase 1a で本番デプロイ対象イメージ全てに適用、Phase 1b で CI stage image と base image にも拡張、Phase 1c で tier2 アプリイメージも対象化、Phase 1b で SBOM 検索 UI 必須化。SBOM 未添付イメージは Kyverno で deploy block（Phase 1b 以降 enforcement）。DS-NFR-COMP-010 と連携して監査対応パックにも組込。

**設計項目 DS-NFR-COMP-014 ビルド Provenance（SLSA）**

ビルドサーバ侵害を検出するため、SLSA Level 3 相当の provenance を段階的に達成する。Phase 1a は SLSA Level 1（build metadata 記録、GitHub Actions / Argo Workflows のログで即時達成）、Phase 1b は SLSA Level 2（source / build の認証付き）、Phase 1c は SLSA Level 3（独立ビルド環境の ephemeral runner、provenance 署名）、Phase 3+ で Level 4（two-person review、hermetic build）を評価。Provenance 内容はビルドステップ / 入力（ソース、依存）/ ビルド環境 / ビルドツールのバージョンを in-toto Statement + SLSA Provenance v1 形式で記録、Sigstore で署名。デプロイ時の provenance 検証は Phase 1c warning mode、Phase 2 で enforcement mode（未検証イメージの deploy を block）。Rekor transparency log は Phase 1c で公的 Rekor または社内プライベート Rekor のいずれかに参加。

**設計項目 DS-NFR-COMP-015 再現可能ビルド**

同じソースから同じアーティファクトを再生成できる状態を目指し、ビルドサーバ侵害の事後検出を可能にする。決定論的ビルドのため `SOURCE_DATE_EPOCH` / `GOFLAGS` 等でタイムスタンプ・ランダム値を固定、依存は go.sum / Cargo.lock / package-lock.json でバージョン固定、ベースイメージも digest で固定、CI で同一ソースからのビルドで同一ハッシュが出ることを検証。Rust 自作領域は Phase 2 で必達、Phase 3 で全イメージ対応、依存 OSS ベースイメージの完全再現は上流依存のため Phase 3+ で SHOULD 扱い。差分発生時は原因調査プロセスを Runbook 化（コンパイラバージョン差 / 環境変数差 / パッケージミラー差の 3 パターンで原因特定）。完全再現性は難易度が高いため、主要イメージの再現性達成率を四半期でトラッキング。

## 業界ガイドライン整合方式

要件定義書 H3 章 NFR-H-COMP-004 は JTC の業種特有ガイドライン（FISC 安全対策基準 / 電子帳簿保存法 / IEC 62443 等）との整合を定める節である。業種が定まり次第個別適用するプレースホルダ要件だが、MVP 期から「どの業種ガイドラインがいつトリガされるか」を設計に仕込むことで後付け対応を回避する。

**設計項目 DS-NFR-COMP-016 業界ガイドライン整合管理**

JTC 業種が確定次第、該当業界ガイドラインを設計に反映するため、ガイドライン追跡レジストリを Phase 1a で法務部と整備する。レジストリには (1) 業種別ガイドライン一覧（FISC 安全対策基準 = 金融業、電子帳簿保存法 = 経理連携、IEC 62443 = 産業用制御システム、HIPAA = 医療、その他業界特有）、(2) 各ガイドラインの適用トリガ（金融テナント受入 / 経理連携開始 / 産業用制御システム連携 / 医療情報取扱）、(3) 現行設計での対応済み項目と追加必要項目のギャップ分析を記録する。年次で遵守状況レビュー、新規ガイドライン施行時は法務部が 3 か月以内にレジストリ更新。FISC 金融業向けは DS-NFR-COMP-008（PCI DSS スコープ）と連携、電子帳簿保存法は DS-NFR-COMP-007 で既に対応設計済み、IEC 62443 は Phase 3+ の産業連携時に評価。

## 透明性と情報公開方式

要件定義書 H5 章 NFR-H-TRN-001 は「プラットフォーム状態の透明性を公開し、テナント側の不信感を防ぐ」ことを定める節である。障害時の情報不足はテナント問い合わせ殺到と運用チーム疲弊に直結するため、情報公開を業務プロセスとして標準化する。

**設計項目 DS-NFR-COMP-017 プラットフォーム透明性公開**

テナント側の不信感と問い合わせ殺到を防ぐため、Backstage portal に Status Page とプラットフォーム情報を集約する。Status Page は主要 11 API（Service Invoke / State / PubSub / Secrets / Binding / Workflow / Log / Telemetry / Decision / Audit-Pii / Feature）の稼働状況を 1 分間隔でリアルタイム公開、信号機表示（Green / Yellow / Red）と直近 24 時間の SLO 達成率を併記。インシデント履歴は過去 Severity 1/2 事故の概要（機密情報除外、原因・対策・再発防止策を要約）を公開。計画メンテナンスは実施 2 週間前に事前公開、当日のタイムラインと影響範囲を明示。リリースノートは機能追加・変更・非推奨化を月次発行、重大脆弱性への対応状況は CVE 別に公開。社内全員がアクセス可、閲覧数と満足度アンケートを月次計測。Phase 1c で Status Page 公開、Phase 2 で満足度モニタリング。
