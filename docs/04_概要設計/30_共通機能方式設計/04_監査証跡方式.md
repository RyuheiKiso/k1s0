# 04. 監査証跡方式

本ファイルは tier1 が生成する監査証跡のデータモデル・生成経路・改ざん検知方式・保管期間・検索アクセス制御を確定させる。監査証跡は J-SOX 対応・個人情報保護法対応・情報セキュリティインシデント対応の法的根拠となる一次記録であり、改ざん・欠落・遅延のいずれも許されない。本章の方式は COMP-T1-AUDIT（Rust 実装）の実装契約となる。

## 本ファイルの位置付け

通常ログ（[03_ログ出力方式.md](03_ログ出力方式.md)）は分析用途を主目的とするため、性能優先で一部欠落を許容する。これに対し監査証跡は法的・規制的要請に応えるため、全イベントの「誰が・何を・いつ・どうした」を欠落なく記録し、事後に改ざんされていないことを証明できる状態を維持する必要がある。この性能と安全性のトレードオフを反映し、k1s0 は監査証跡を通常ログと別経路で実装する。

構想設計 ADR-TIER1-001 で COMP-T1-AUDIT を Rust 実装と決定したのは、ハッシュチェーン計算と暗号署名の性能・型安全性・panic 抑止を厳格に確保する必要があるためである。本章はその Rust コンポーネントの振る舞いを設計 ID で固定する。

## 監査対象イベント

### イベント分類

以下 6 分類のイベントを監査対象とする。監査対象を曖昧にすると「記録漏れ」が発生し、事後調査で根拠を示せなくなる。分類は J-SOX 内部統制の「職務・権限・承認・記録・報告」の 5 要素をカバーするよう設計する。

**認証イベント**は認証成功・失敗・ロックアウト・ログアウト・パスワード変更。失敗も記録することが攻撃痕跡の保全に必須である。**認可イベント**は allow / deny の両方、ロール変更、権限委譲。deny の記録は内部不正検知の要である。**管理操作イベント**は Keycloak Realm 設定変更、OpenBao 鍵操作、Feature Flag 変更、ZEN Engine 決定表変更、tier1 コンポーネント設定変更。**データアクセスイベント**は PII を含むリソースの read / write / delete、全 State API 書き込み、Secrets API 読み取り。**ワークフローイベント**は Saga 開始・完了・補償、Workflow 永続化。**インフライベント**は Kubernetes Deployment 変更、Istio Policy 変更、SPIFFE ID 発行・失効。

### 除外イベント

以下は監査対象から明示的に除外する。監査対象を広げすぎるとデータ量が爆発し、7 年保管のコストが許容範囲を超えるためである。

- Health Check / Readiness Probe
- メトリクス収集（Prometheus scrape）
- 内部 gRPC の単純 read（認可結果が allow の場合のみログ側に記録）
- 計算中間結果（Saga 中間 step、Decision 中間判定）

除外イベントも Log API 経由で通常ログには残す。調査が必要になった場合は通常ログを 90〜180 日保管している範囲で遡る。

## データモデル

### 監査レコード構造

1 監査イベントは以下のフィールドを持つ。スキーマは Protobuf で定義し、Go / Rust 両方で自動生成する。

- **audit_id**: UUIDv7（時系列ソート可能な UUID）。
- **prev_hash**: 直前の監査レコードの hash。チェーン先頭は 32 バイト 0 埋め。
- **timestamp**: RFC 3339 ナノ秒精度 UTC。
- **tenant_id**: テナント識別子。
- **actor**: 主体情報（`sub` / `roles` / `source_ip_class` / `spiffe_id` のいずれか or 複数）。
- **event_type**: 6 分類のいずれか（例: `auth.success` / `authz.denied` / `admin.role_changed`）。
- **resource**: 操作対象（`resource_type` / `resource_id` / `sensitivity_level`）。
- **action**: `read` / `write` / `delete` / `admin`。
- **outcome**: `success` / `failure`。
- **context**: API 固有の追加情報を JSON で格納（PII マスク済み）。
- **hash**: 上記全フィールドの SHA-256。本レコードが tamper-evident である証拠値。
- **signed_at**: バッチ署名のタイムスタンプ（未署名時は null）。
- **signer**: 署名者情報（OpenBao Transit Engine の鍵 ID）。
- **signature**: 日次バッチ署名値（バッチ完了後に追記）。

### ハッシュチェーン構造

各監査レコードの hash は `SHA-256(audit_id || prev_hash || timestamp || tenant_id || actor || event_type || resource || action || outcome || context)` で計算する。次レコードの prev_hash は直前レコードの hash であり、チェーン構造となる。中間レコードの改ざんは後続の hash 不一致で検知できる。

チェーンは単一グローバルチェーンとする。テナント分割チェーンも検討したが、テナント境界の監査（クロステナント試行）がチェーンから外れるリスクを避けて単一化した。単一チェーンの書き込み競合はシングルライター（COMP-T1-AUDIT Primary）で serialize する。

### 書き込み性能要件

単一ライターのスループットは 5,000 events/sec を目標とし、中規模 150 RPS 想定（イベント 8〜10/リクエスト）に対し 10 倍の余裕を確保する。書き込みレイテンシは p99 10ms 以内とし、tier1 API p99 500ms 予算に影響しない（非同期書き込み + バッファ）。バッファ満杯時は呼び出し側をブロックし、監査記録失敗よりもレスポンス遅延を選択する（監査の確実性を性能より優先する設計）。

## PII マスキングの方式化

### 設計項目 DS-CF-AUD-012 PII パターン追加の方式化

要件 FR-T1-PII-001（PII マスキング契約）は「どの項目が PII か」を時系列で拡張できることを前提とする。個人情報保護法改正・業界ガイドライン改定・新規顧客要請などにより、6 か月後に新規 PII 種別（例: 顔認証用特徴量ハッシュ、医療 ID、生体認証トークン）をマスク対象に追加する場面は不可避であり、その追加が属人運用に陥ると「ある種別は Audit で平文、別の種別はマスク」という状態が生じ、監査の信頼性が崩壊する。本節は PII マスク対象の追加を方式として固定化し、追加 PR のみで tier1 全体が一貫したマスキング状態を維持できるようにする。

### PII パターン定義の外部化

PII マスク対象は Audit / Log / Telemetry の 3 コンポーネントが共通参照する単一 YAML（`pii_patterns.yaml`）で定義する。コード内ハードコードは禁止する。YAML は以下の構造を取る。

- **pattern_id**: マスクルール識別子（`PII-EMAIL` / `PII-PHONE-JP` / `PII-MYNUMBER` などの安定 ID）。
- **applies_to**: 検出対象フィールド（JSON パス or 正規表現）。例: `context.user.email` / `context.user.phone` / 任意の文字列フィールドを全走査するなら `*`。
- **detection**: 検出ロジック（`exact_field` / `regex` / `hash_prefix`）。正規表現使用時は ReDoS 耐性を検証済みのみ許可（`*` 量指定子の後方参照禁止）。
- **masking_strategy**: マスク方式（`redact`（完全除去）/ `hash_sha256`（ハッシュ化）/ `partial`（先頭 2 文字 + 末尾 2 文字以外を `*`）/ `tokenize`（OpenBao Transit Engine の format-preserving encryption））。
- **sensitivity_level**: `P1`（最機密: マイナンバー・医療 ID 等）/ `P2`（個人識別: 氏名・電話番号等）/ `P3`（個人属性: 購買履歴等）。保管期限と暗号化要件の判定に使用。
- **added_at / added_by / justification**: 追加時点のメタデータ（監査のため必須）。

パターン定義は Protobuf スキーマで検証し、CI で「YAML パース成功 + 必須フィールド充足 + regex ReDoS チェック + sensitivity_level 値域チェック」の 4 検査を実施する。1 つでも失敗すると PR はブロックされる。

### 追加 PR のワークフロー

新規 PII パターン追加は以下 5 ステップの単一 PR で完結させる。5 ステップのいずれかが欠けると CI が Fail し、マージできない構造とする。

1. **パターン定義の追記**: `pii_patterns.yaml` に新規 entry を追加し、`pattern_id` は既存値と衝突しないことを CI で検証する。
2. **単体テストの追加**: `pii_masker_test.go` / `pii_masker_test.rs` に該当パターンのマスキング入出力を 3 ケース以上（正常系 / 境界系 / 誤検知抑制）で追加する。既存のテスト命名規約 `TestPIIMask_<pattern_id>_<case>` に従う。
3. **Audit スキーマの更新**: マスク後の値が Audit レコード `context` フィールドに埋め込まれる形式を明示し、`sensitivity_level` を `resource.sensitivity_level` に反映するロジックを Audit コンポーネントに追加する（必要時のみ）。
4. **Runbook の更新**: `55_運用ライフサイクル方式設計/08_Runbook設計方式.md` の PII インシデント Runbook（RB-PII-001 系）に新規パターンの検出時対応手順を追記する。
5. **Feature Flag による段階有効化**: 新パターンは `k1s0.pii.<pattern_id>.enabled` Flag（デフォルト false）で包み、本番投入前に Staging 環境で 1 週間観測する。誤検知率・マスキング性能劣化が基準内であることを確認後、Flag を true にする。これにより「新パターン導入直後の誤マスキングによる業務ログ欠損」を逆方向から抑制する。

### マスキング実装の拡張契約

COMP-T1-AUDIT（Rust）および Log / Telemetry SDK（Go）は `pii_patterns.yaml` を起動時に読み込み、ホットリロード（SIGHUP 受信 or Kubernetes ConfigMap 更新検知）で再読込する。ホットリロード失敗時は旧設定で動作継続し、Alertmanager に `pii_reload_failed` で発報する。無応答が 15 分続いた場合は Pod 再起動で強制的に再読込する（設定食い違いによる長時間の誤マスクを防止）。

### 監査性の担保

PII パターンの追加履歴そのものが監査対象となる。`pii_patterns.yaml` の変更は Git log で追跡可能であり、さらに「いつ誰がどの PII 種別の扱いを変えたか」を Audit イベント `pii.pattern_changed`（event_type 分類「管理操作イベント」）として記録する。これにより「リリース時点 時点では Phone を hash_sha256 でマスクしていたが、採用後の運用拡大時 で tokenize に切り替えた」という変更が 7 年保全される。

### 段階割付

- **リリース時点**: 本設計節のスケルトンのみ採用（`pii_patterns.yaml` は空 YAML、3 コンポーネントの読込コードは stub）。
- **リリース時点**: Email / Phone-JP / MyNumber の 3 基本パターンを先行投入。Log コンポーネントのマスキング本格化と同時に有効化。
- **リリース時点**: Audit コンポーネント側のマスキング本格化、`pii.pattern_changed` 監査イベントを有効化。
- **採用後の運用拡大時**: tokenize 方式（OpenBao Transit Engine FPE）と業界別パターン拡張（医療 ID / 金融口座番号等）を追加。

本節は要件 FR-T1-PII-001 と NFR-G-PRV-\* の双方を方式化する設計 ID 群（DS-CF-AUD-012）として、[../80_トレーサビリティ/02_要件から設計へのマトリクス.md](../80_トレーサビリティ/02_要件から設計へのマトリクス.md) に登録する。

## 格納先

### CloudNativePG WORM テーブル

監査レコードは CloudNativePG の専用 PostgreSQL インスタンスに格納する。通常業務 DB とは物理インスタンスを分離し、監査用クレデンシャルは COMP-T1-AUDIT のみが保持する。テーブルは append-only 制約を PostgreSQL 側で強制する。

- `audit_log` テーブル: INSERT のみ許可、UPDATE / DELETE は REVOKE で排除。
- Row Level Security を `tenant_id` ベースで有効化。ただし Audit Reader ロールは全テナント参照可能。
- `audit_id` に UNIQUE、`prev_hash` に INDEX、`timestamp` + `tenant_id` 複合 INDEX。

バックアップは pg_basebackup 日次 + WAL 継続アーカイブを標準とし、別データセンターに地理分散する（採用後の運用拡大時 でマルチクラスタ移行）。

### MinIO 冷保存（2 年経過以降）

ホット参照期間は 2 年とし、以降は MinIO Object Lock（WORM モード、Compliance Retention）で 5 年間冷保存する。Object Lock は AWS S3 Object Lock 互換で、保持期間中は rm 不可・改変不可を Object Storage 側で強制する。

冷保存移行は日次バッチで行い、2 年以上経過した監査レコードを Parquet 形式でエクスポートし MinIO に書き込む。PostgreSQL 側は冷保存完了確認後に DELETE する（WORM テーブルの例外として、アーカイブ完了後のみ DELETE 権限を付与）。

## 改ざん検知

### Nightly チェーン検証

日次 02:00 JST に全チェーン再計算バッチを実行する。チェーン先頭から順に hash を再計算し、保存 hash と一致しなければ異常判定する。

- バッチ実行時間: 100 万レコードあたり約 15 分（測定値）。
- 日次 100 万レコード想定で約 15 分、7 年蓄積 2.5 億レコードでも 60 時間以内に終わる（週次の深い検証時のみ実行）。

不一致が 1 件でも検知された場合、SEV1 アラートを PagerDuty に発報し、Platform Admin と法務担当者に Slack 通知する。検知後は改ざん対象レコードの詳細調査を Runbook 手順で実施する。

### 日次バッチ署名

前日分の監査レコード末尾 hash を OpenBao Transit Engine の秘密鍵で署名する。署名アルゴリズムは Ed25519 で、鍵ローテーション周期は 365 日、前鍵による検証は無期限保持する（過去署名の検証が常に可能な状態を維持）。

署名結果は `audit_signatures` テーブルに格納し、date / last_audit_id / signature / signer_key_id を記録する。第三者監査時は該当日のレコードと署名のみを提示すれば改ざん無しを証明できる。

## 検索 API

### 検索権限と経路

監査レコードの検索は Audit Reader ロールのみが実行可能とする。検索 API は COMP-T1-AUDIT が提供する gRPC エンドポイント `Audit.Search` を用い、tier1 公開 API として expose する。

検索条件は `tenant_id`、`event_type`、`actor.sub`、`timestamp` 範囲、`resource_id`、`outcome` の組合せで指定する。ページネーションは cursor ベース、1 ページ最大 1,000 件とする。

### 検索性能

90 日以内のレコード（ホット範囲）は p99 500ms 以内で返却。2 年以内は p99 2 秒。2 年超の冷保存レコードは MinIO から Parquet を取得し、p99 30 秒以内で返却する（オンデマンド取得）。

## 対応する要件と法令

### J-SOX 内部統制

J-SOX（金融商品取引法）の IT 統制対応として、権限付与・変更・剥奪、システムアクセス、重要データ変更の記録を 7 年保管する。本章の 7 年保管は J-SOX 要件（原則 7 年）に直接対応する。

### 個人情報保護法

2022 年改正個人情報保護法の安全管理措置義務に基づき、PII アクセス記録を保持する。PII 読み取りは `data_access` イベントとして必ず記録し、target_resource に sensitivity_level: confidential / restricted を明記する。

### 電帳法（電子帳簿保存法）

電帳法対応は本章スコープ外（tier3 業務アプリ側の責務）だが、tier3 側が電帳法対応を実装する際に基盤として本監査証跡を利用できるよう、Object Lock による改変不能性と第三者検証可能性を確保する。

## 設計 ID 一覧

- **DS-CF-AUD-001**: 監査対象イベントを 6 分類（認証 / 認可 / 管理操作 / データアクセス / ワークフロー / インフラ）に固定する。確定段階 リリース時点。
- **DS-CF-AUD-002**: Health Check / Readiness / メトリクス / 内部 read / 計算中間を明示的に除外し、通常ログでカバーする。確定段階 リリース時点。

- **DS-CF-AUD-005**: 書き込みスループット 5,000 events/sec、書き込みレイテンシ p99 10ms、バッファ満杯時 fail-closed をパラメータとする。確定段階 リリース時点。
- **DS-CF-AUD-006**: CloudNativePG 専用インスタンス + append-only + RLS で WORM テーブルを構成する。確定段階 リリース時点。
- **DS-CF-AUD-007**: 2 年以上経過レコードは MinIO Object Lock で 5 年冷保存、総保管期間 7 年。確定段階 リリース時点。
