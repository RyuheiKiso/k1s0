# 06. メトリクス収集方式

本ファイルは tier1 コンポーネント・tier2 / tier3 ワークロード・infra OSS が出力する数値メトリクスの命名規約・ラベル設計・収集経路・保管期間・ダッシュボード構成・アラート連携を確定させる。メトリクスは SLO 監視・キャパシティ計画・リソース最適化・請求の根拠となり、定義のブレはそのまま意思決定のブレに直結する。

## 本ファイルの位置付け

ログが「何が起きたか」の記録、Trace が「どう起きたか」の可視化であるのに対し、メトリクスは「どれくらい起きたか」を数値化する。SRE の実務ではメトリクスが最も頻繁に参照される一次情報であり、運用 採用側の小規模運用で基盤を運用する k1s0 において、メトリクスの品質は運用品質の上限を規定する。

構想設計 ADR-OBS-001 で Grafana Mimir を 運用蓄積後採用、リリース時点 は Prometheus standalone で運用することが確定している。本章はメトリクスのスキーマと運用パラメータを固定し、ダッシュボード・アラートの設計基盤とする。

## メトリクス命名規約

### 命名書式

Prometheus のベストプラクティスに従い、以下の書式を強制する。

```text
k1s0_<domain>_<metric>_<unit>
```

- **k1s0_**: プレフィックスを必須とし、外部 OSS（Kubernetes / Istio / PostgreSQL）のメトリクスと衝突を防ぐ。
- **`<domain>`**: `api` / `workflow` / `state` / `decision` / `pubsub` / `audit` / `auth` / `infra` 等のドメイン識別子。
- **`<metric>`**: `requests_total` / `duration_seconds` / `errors_total` 等の計測対象。
- **`<unit>`**: `seconds` / `bytes` / `count` 等の単位。Counter の場合は `_total` 末尾必須。Histogram は `_bucket` / `_sum` / `_count` が自動付与される。

例: `k1s0_api_requests_total`、`k1s0_api_duration_seconds_bucket`、`k1s0_state_bytes`。

### ラベル設計

ラベルはカーディナリティ爆発を招きやすいため、以下のホワイトリストに限定する。追加は ADR 起票を要する。

- **tenant_id**: 想定 1,000 テナント、カーディナリティ制御のため リリース時点 で per-tenant rollup を導入。
- **api**: tier1 API 11 種 + 内部 API 約 30、計 40 程度。
- **method**: gRPC メソッド名（`Get` / `Set` / `Delete` 等）。
- **status**: `ok` / `error`、あるいは gRPC status code。
- **env**: `dev` / `stg` / `prod` の 3 値。
- **service**: コンポーネント名。

高カーディナリティ値（resource_id / user_id / trace_id）はラベルに入れない。必要な場合は Exemplar（[05_分散トレーシング方式.md](05_分散トレーシング方式.md)）で Trace へ pivot する。

## 標準メトリクス

### RED 手法

tier1 API 全 11 種で以下の RED（Rate / Errors / Duration）メトリクスを必須とする。これはマイクロサービス観測のデファクト手法で、運用担当者が最初に見るべき指標である。

- **k1s0_api_requests_total** (Counter): API 呼び出し総数。ラベル `api` / `method` / `status` / `tenant_id` / `env`。
- **k1s0_api_errors_total** (Counter): API エラー総数。ラベル `api` / `method` / `error_code` / `tenant_id` / `env`。
- **k1s0_api_duration_seconds** (Histogram): API 処理時間。ラベル `api` / `method` / `tenant_id` / `env`。バケット `[0.005, 0.01, 0.025, 0.05, 0.1, 0.25, 0.5, 1, 2.5, 5, 10]`。

### USE 手法

インフラ層（Node / Pod / ネットワーク / ストレージ）で USE（Utilization / Saturation / Errors）を必須とする。USE は Kubernetes / Node Exporter / cAdvisor のメトリクスで自動取得できる範囲を標準とする。

- CPU Utilization / Saturation / Throttling
- Memory Utilization / Saturation / OOM 発生回数
- Network Bandwidth / Packet Errors
- Disk I/O Utilization / Queue Depth / Errors

### ビジネスメトリクス

tier1 API の業務的な利用状況を表すメトリクスを以下で定義する。これらは経営層・プロダクトマネージャ向けのダッシュボードで利用される。

- **k1s0_workflow_executions_total** (Counter): ワークフロー実行回数。ラベル `workflow_name` / `status` / `tenant_id`。
- **k1s0_state_bytes** (Gauge): State Store 使用バイト数。ラベル `tenant_id` / `store_type`。
- **k1s0_decision_evaluations_total** (Counter): Decision API 評価回数。ラベル `decision_id` / `outcome` / `tenant_id`。
- **k1s0_pubsub_messages_total** (Counter): PubSub メッセージ数。ラベル `topic` / `direction`（publish / consume） / `tenant_id`。
- **k1s0_audit_events_total** (Counter): 監査イベント生成数。ラベル `event_type` / `outcome` / `tenant_id`。
- **k1s0_auth_attempts_total** (Counter): 認証試行回数。ラベル `outcome` / `reason`。
- **k1s0_tenant_quota_usage** (Gauge): テナントクォータ使用率（0.0〜1.0）。ラベル `tenant_id` / `quota_type`。

## 収集経路

### スクレイピング + Remote Write のハイブリッド

tier1 コンポーネントは各 Pod で `:9090/metrics` に OpenMetrics 形式を公開する。Prometheus が定期的にスクレイピングし（15 秒間隔）、リリース時点 からは Mimir に Remote Write する。OpenTelemetry Metrics は OTel Collector 経由で Remote Write することも可能だが、初期は Prometheus スクレイピングを主経路とし、OTel Collector は予備経路として配置する。

スクレイピング方式を主経路とするのは、Prometheus コミュニティのエコシステムが成熟しており、Kubernetes ServiceMonitor / PodMonitor CR が運用 採用側の小規模運用でも破綻しにくい運用 UX を提供するためである。

### メトリクス収集レイテンシ

Prometheus からターゲットへのスクレイピング p99 100ms 以内、1 Prometheus インスタンスで 1,000 Pod を捌く設計とする。採用後の運用拡大時 の規模拡大時は Thanos Sidecar + Receive パターンで水平分割する。

## 保管期間とダウンサンプリング

### 保管ポリシー

保管は high-resolution（15 秒間隔）と down-sampled（5 分間隔）の 2 段階とする。

- **高精細（15s）**: 15 日間保管。直近障害分析と SLO 監視に使用。
- **ダウンサンプル（5m）**: 13 か月間保管。季節変動分析・容量計画に使用。
- **月次集計（1h）**: 3 年間保管。年次報告書・予算策定に使用。運用蓄積後。

Mimir の downsampling 機能で自動的に粒度変換する。ストレージ見積では 15 日高精細が支配的で、中規模 150 RPS 想定で約 500 GB（1 テナント時）、1,000 テナント時はラベル per-tenant rollup で約 5 TB を見込む。

### リリース時点 の制約

リリース時点 は Prometheus standalone、保管期間 7 日のみ（AGPL OSS 未採用のため Mimir 不可）。リリース時点 の段階で長期保管できないことを運用 Runbook に明記し、重要イベント（Deployment / インシデント）は別途スクリーンショット or CSV エクスポートでアーカイブする運用でカバーする。

## ダッシュボード構成

### ペルソナ別ダッシュボード

Grafana で以下 4 ペルソナ向けダッシュボードを標準提供する。Grafana は AGPL のため 運用蓄積後、ADR-0003 の 5 原則を満たす構造で運用する。

**SRE 向け USE ダッシュボード**: Node / Pod / コンテナ / ネットワーク / ストレージの USE 指標を階層表示する。SLO burn rate と直近 1 時間の異常検知を左上に固定配置する。

**開発者向け RED ダッシュボード**: 選択した API / サービスの Rate / Errors / Duration を表示する。p50 / p95 / p99 / p99.9 を並列比較、エラー詳細は Tempo Trace へワンクリック pivot する。

**経営層向けビジネスダッシュボード**: テナント別アクティブ利用数、API 呼び出し総数、ワークフロー実行数、PII 処理数の月次推移を表示する。

**セキュリティ担当者向けダッシュボード**: 認証失敗 / 認可 deny / 監査チェーン検証結果 / OpenBao 操作の時系列を表示する。閾値超過時は PagerDuty へ自動エスカレーション。

### ダッシュボード管理

Grafana ダッシュボードは Grafana Provisioning（ファイルベース）で管理する。Git リポジトリ `infra/grafana-dashboards` に JSON で保存し、Argo CD 経由で Grafana にデプロイする。UI 上での直接編集は prod 環境で禁止し、変更は必ず PR 経由とする。

## アラート

### Alertmanager と PagerDuty

アラートルールは PrometheusRule CR で定義し、Git 管理する。発報先は Alertmanager 経由で PagerDuty に送信し、Sev1 は即時電話、Sev2 は 15 分以内、Sev3 は 1 時間以内を SLA とする（業務時間内）。夜間（23:00-06:00 JST）は Sev1 のみ電話、Sev2 / 3 は翌朝対応とする。

### 標準アラートルール

リリース時点 の初期から以下の標準アラートを有効化する。

- **Sev1**: tier1 API 可用性 < 99%（5 分連続）、監査チェーン破損検知、Keycloak Down。
- **Sev2**: tier1 API p99 > 1 秒（5 分連続）、認証失敗急増（平時 10x）、State Store 容量 > 85%。
- **Sev3**: PubSub 消費遅延 > 5 分、Feature Flag 評価エラー急増、ログマスキング失敗 > 10 件/時。

アラートルールは リリース時点 パイロット運用で閾値を微調整する前提とし、初期は保守的（誤検知許容）な閾値から開始する。

## 設計 ID 一覧

- **DS-CF-METRIC-001**: メトリクス命名書式 `k1s0_<domain>_<metric>_<unit>`、プレフィックス必須、Prometheus ベストプラクティス準拠。確定段階 リリース時点。
- **DS-CF-METRIC-002**: ラベルは tenant_id / api / method / status / env / service の 6 種ホワイトリスト、追加は ADR 起票必須。確定段階 リリース時点。
- **DS-CF-METRIC-003**: tier1 全 API で RED メトリクス 3 種（requests_total / errors_total / duration_seconds）を必須化。Histogram バケット 11 階級。確定段階 リリース時点。
- **DS-CF-METRIC-004**: インフラ層で USE メトリクスを Node Exporter / cAdvisor で自動取得。確定段階 リリース時点。
- **DS-CF-METRIC-005**: ビジネスメトリクス 7 種（workflow / state / decision / pubsub / audit / auth / tenant_quota）を tier1 共通定義。確定段階 リリース時点。
- **DS-CF-METRIC-006**: 収集はスクレイピング（15 秒間隔）を主経路、OTel Collector Remote Write を予備経路とする。確定段階 リリース時点。
- **DS-CF-METRIC-007**: リリース時点 は Prometheus standalone 保管 7 日、運用蓄積後は Mimir で高精細 15 日 / 5 分粒度 13 か月 / 1 時間粒度 3 年の 3 段階保管。確定段階 リリース時点 / リリース時点。
- **DS-CF-METRIC-008**: AGPL OSS（Mimir / Grafana）は ADR-0003 の 5 原則を満たす構造で 運用蓄積後に限定運用。確定段階 リリース時点。
- **DS-CF-METRIC-009**: ペルソナ別ダッシュボード 4 種（SRE USE / 開発者 RED / 経営ビジネス / セキュリティ）を Grafana Provisioning で Git 管理デプロイ。確定段階 リリース時点。
- **DS-CF-METRIC-010**: Alertmanager + PagerDuty の Sev1-3 標準アラートルール 3x3 マトリクスを定義、閾値は リリース時点 パイロットで調整。確定段階 リリース時点。

- **FR-T1-TELEMETRY-001** 〜 **FR-T1-TELEMETRY-004**: tier1 Telemetry API のメトリクス部分。
- **NFR-B-PERF-001** 〜 **NFR-B-PERF-007**: 性能監視、p99 レイテンシ観測、スループット監視、計装オーバヘッド監視、Feature Flag 評価監視。
- **NFR-C-OBS-001** 〜 **NFR-C-OBS-006**: 観測性、ダッシュボード、アラート。

- **NFR-C-CAP-001**: キャパシティ計画に用いる長期メトリクス。
