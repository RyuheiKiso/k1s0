# 08. 暗号化と鍵管理方式

本ファイルは k1s0 の通信暗号化・保存時暗号化・アプリケーション層暗号化の 3 層と、これらを支える鍵管理（発行 / 保管 / ローテーション / 失効 / 漏洩対応）の方式を確定させる。暗号化は「やる」「やらない」の二択ではなく、どのレイヤで何を守るかを設計の粒度で定義する必要があり、抜けた層が実質的な漏洩経路になる。本章はそれを層別に定め、鍵管理の運用手順まで含めて固定化する。

## 本ファイルの位置付け

採用側組織の情シスがオンプレ基盤で最も頻繁に監査指摘を受けるのは「通信は暗号化しているが、データベースは平文」「秘密情報を環境変数に入れていた」「鍵ローテーションが 3 年間未実施」といった運用の欠落である。k1s0 は設計段階でこれらを構造的に防ぎ、運用 採用側の小規模運用でも鍵ローテーションが自動的に回る状態を作る必要がある。

構想設計 ADR-SEC-002（OpenBao 採用）および ADR-SEC-003（SPIFFE/SPIRE 採用）で暗号基盤の骨格は確定済みである。本章はそれを受けて、具体的な鍵階層・ローテーション周期・移行手順・HSM 連携の将来計画を定める。

## 暗号化の全体像

k1s0 の暗号化は 3 層で構成する。各層が独立に守る領域を持ち、1 層の侵害が他層へ直ちに波及しない構造とする。

**通信層（Data in transit）** は Pod 間 / 外部 - 内部の通信を暗号化する。Istio Ambient mTLS による L4 自動 mTLS、Envoy Gateway による Ingress TLS 1.3、Kafka TLS、PostgreSQL libpq TLS がこの層に属する。

**ストレージ層（Data at rest）** は永続ディスクに書き込まれたデータを暗号化する。PostgreSQL TDE、MinIO SSE-S3、Kafka ディスク LUKS、etcd encryption at rest がこの層に属する。

**アプリケーション層（Data in use の前処理）** は特定の機密フィールドを、データストアに書き込む前にアプリケーションが暗号化する。PII 項目、秘密情報、監査証跡の署名などがこの層に属し、OpenBao Transit Engine が実行エンジンとなる。

## 通信暗号化

### 内部通信（mTLS）

tier1 内部の Pod 間通信は Istio Ambient の ztunnel が HBONE プロトコルで自動 mTLS 終端する。アプリケーションコードは平文 gRPC を話すだけで、ネットワーク層で自動暗号化される。暗号化スイートは TLS 1.3、`TLS_AES_256_GCM_SHA384` を第一候補、`TLS_CHACHA20_POLY1305_SHA256` を ARM 環境向けに併記する。

証明書は SPIRE が発行する X.509 SVID で、有効期限 1 時間・先行ローテーション 30 分周期（詳細は [01_認証方式設計.md](01_認証方式設計.md)）。mTLS 接続は STRICT モードで強制し、平文通信は Istio AuthorizationPolicy で全面禁止する。

### 外部通信（Ingress TLS）

Envoy Gateway の Ingress は TLS 1.3 を必須とし、TLS 1.2 以下は無効化する。証明書は cert-manager が ACME プロトコルで発行する。リリース時点 は Let's Encrypt ACME、運用蓄積後は 採用側組織内部 CA（OpenBao PKI Secrets Engine）を用い、閉域運用でも発行可能な構成にする。

証明書の有効期限は 90 日、自動更新は残 30 日で発動する。HSTS ヘッダ（`max-age=31536000; includeSubDomains; preload`）を必須付与し、ダウングレード攻撃を防ぐ。

### Kafka 通信

Kafka のブローカー間通信・プロデューサ / コンシューマ通信は TLS 1.3 で暗号化する。Kafka ACL と組み合わせ、クライアント認証は SASL/SCRAM + mTLS のダブルチェックとする。PII を含むトピックは追加でメッセージ本体をアプリケーション層で暗号化する（後述）。

## ストレージ層暗号化

### PostgreSQL TDE

CloudNativePG が管理する PostgreSQL 17 は、以下のいずれかで at rest 暗号化する。

- **PostgreSQL 17 native TDE**（PG17 から実験的、採用後の運用拡大時 で GA 評価後に採用検討）
- **LUKS ファイルシステムレベル暗号化**（リリース時点 から採用、Kubernetes Persistent Volume レイヤで LUKS 鍵をノードに配置）

リリース時点 は LUKS での運用を基本とし、採用後の運用拡大時 で PostgreSQL 17 native TDE の成熟度を評価して移行判断する。LUKS マスター鍵は Sealed Secrets で暗号化し、OpenBao に格納する。ノード起動時に OpenBao から取得して dm-crypt に渡す。

### MinIO SSE-S3

Object Storage は MinIO を用い、SSE-S3（サーバサイド暗号化 with S3 管理鍵）を必須とする。バケット単位で暗号化強制設定を入れ、暗号化なしでの PUT は HTTP 400 で拒否する。Object-level 鍵は MinIO が自動管理、マスター鍵は OpenBao 連携で外部化する。

MinIO は AGPL OSS のため 運用蓄積後で ADR-0003 の 5 原則を満たす構造で運用する。リリース時点 は本格的 Object Storage を導入せず、PostgreSQL large object または hostPath ボリュームで代替する（監査冷保存 2 年以内のため リリース時点 時点で十分）。

### Kafka ディスク暗号化

Kafka のログディレクトリ（`/var/lib/kafka/data`）は LUKS で暗号化する。PostgreSQL と同じノード LUKS マスター鍵を共用し、運用複雑性を抑える。

### etcd encryption at rest

Kubernetes etcd は `encryption-provider-config` で AES-CBC 暗号化を有効化する。鍵は OpenBao から取得し、90 日で自動ローテーションする。

## アプリケーション層暗号

### OpenBao Transit Engine

OpenBao Transit Engine は「データを送って暗号化された結果を受け取る」keyless 対称暗号サービスである。tier1 アプリケーションは平文 → Transit Engine → 暗号文を受け取り、DB に格納する。復号も同様に Transit Engine 経由で行う。鍵の実体はアプリケーションに渡さない。

採用アルゴリズムは `AES-256-GCM` をデフォルト、後方互換性が必要な場合のみ `AES-256-CBC` を許可する。暗号化 / 復号 API のレイテンシは p99 5ms 以内（OpenBao ローカル Pod 通信）、スループットは 10,000 ops/sec を目標とする。

### PII 自動暗号化

COMP-T1-PII は PII 項目を Transit Engine 経由で自動暗号化する。PII カラムにはテナント専用の Derived Key を使い、テナント間でクロス復号できない構造とする。Derived Key は `hmac(tenant_id, master_pii_key)` で生成し、master_pii_key は OpenBao に格納される。

PII 復号はアクセスごとに監査ログに `pii.decrypt` として記録する。Rate Limit も設け、1 テナントあたり 1 分 1,000 件を超える復号リクエストは SEV3 アラート（大量抽出検知）。

### 監査証跡の署名

[04_監査証跡方式.md](04_監査証跡方式.md) のハッシュチェーン署名は、OpenBao Transit Engine の Ed25519 鍵で日次バッチ実行する。鍵は `transit/keys/k1s0-audit-signer`、ローテーション周期 365 日、過去鍵は無期限保持（過去監査の検証継続のため）。

## 鍵階層と管理

### 3 階層構造

k1s0 の鍵は以下 3 階層で管理する。階層化により、上位鍵の漏洩影響を局所化できる。

**Master Key（マスター鍵）**: OpenBao のシステム鍵。Shamir 5-of-3 分散マスターキーで 5 片に分割、3 片揃うことで復元可能。5 片は Platform Admin（複数名） + Security Officer（複数名）（採用後の運用拡大時 時点人員）に物理配布する。

**Key Encryption Key（KEK）**: Data Encryption Key を暗号化する鍵。用途別（PII / 監査 / 汎用）に分離する。ローテーション周期 365 日、OpenBao Transit Engine が内部管理する。

**Data Encryption Key（DEK）**: 実データ暗号化鍵。テナント別・用途別に派生する。ローテーション周期 90 日、Transit Engine が自動再暗号化（re-wrap）で更新する。

### ローテーション周期と運用

| 鍵種別 | ローテーション周期 | 手順 | 過去鍵保持 |
| --- | --- | --- | --- |
| Master Key | イベントベース（漏洩時 + 4 年上限） | 手動、Runbook 参照 | 切替完了まで |
| KEK | 365 日 | 自動（OpenBao scheduled） | 5 年 |
| DEK | 90 日 | 自動（re-wrap） | 無期限 |
| SVID（SPIFFE） | 1 時間 | 自動 30 分周期 | 直近 1 世代 |
| TLS 証明書（外部） | 90 日 | 自動 30 日前 | 不要 |
| LUKS Master | 年次手動 | Runbook 参照 | 切替完了まで |

上表はローテーション運用の索引である。実際の rotation 操作は OpenBao のポリシーで宣言的に定義し、自動化されたものは運用 採用側の小規模運用でも確実に実行される。手動操作が必要なものは Runbook を整備し、月次訓練で手順確認を行う。

### HSM 統合の将来計画

採用初期 は OpenBao の Shamir 分散マスターキーで十分なセキュリティを確保する。採用側のマルチクラスタ移行時（採用後の運用拡大時の拡張）で HSM（YubiHSM 2 または Nitrokey HSM 2）統合を評価し、FIPS 140-2 Level 3 相当の保証が要請される用途（金融 / 医療テナント）向けに対応する。

## 秘密情報の扱い

### 環境変数禁止

アプリケーション Pod への秘密情報注入は、以下を禁止する。

- 平文環境変数での秘密渡し（Kubernetes Secret の envFrom / env 参照も禁止）
- ConfigMap への秘密格納
- コンテナイメージへの埋め込み

代替として以下を用いる。

- **OpenBao Agent Injector**: Sidecar / Init Container が OpenBao から秘密を取得し、メモリマウントファイル `/vault/secrets/*` に書き込む。Pod 再起動で自動更新。
- **Kubernetes CSI Secret Store Driver**: OpenBao CSI Driver で tmpfs マウント経由アクセス。
- **SPIFFE SVID**: サービス認証クレデンシャルは SVID で置換。

### 鍵漏洩対応

鍵漏洩検知時の対応手順を Runbook `k1s0-ops-key-compromise` で定義する。概要は以下。

1. **即時 Revocation**: OpenBao Transit Engine で該当鍵を `min_decryption_version` 更新で旧版無効化。
2. **Rotation**: 新鍵を生成、該当暗号化データを re-wrap。
3. **Impact Assessment**: 漏洩期間中の監査ログを全量調査、該当鍵で暗号化されたデータアクセスを抽出。
4. **Notification**: 影響テナントへ通知、採用側組織の情シスセキュリティ部門へ報告、必要に応じ個人情報保護委員会へ報告（改正個人情報保護法）。
5. **Post Incident Review**: RCA（Root Cause Analysis）を 30 日以内に完了、再発防止策を ADR 化。

## 設計 ID 一覧

- **DS-CF-CRYPT-001**: 通信暗号化は Istio Ambient mTLS（TLS 1.3 / AES-256-GCM or ChaCha20-Poly1305）、Ingress Envoy TLS 1.3、Kafka TLS + SASL、すべて STRICT 強制。確定段階 リリース時点 / リリース時点。
- **DS-CF-CRYPT-002**: 外部 TLS 証明書は cert-manager + Let's Encrypt（リリース時点）または OpenBao PKI（リリース時点）で管理、90 日有効期限 + 30 日前自動更新。HSTS 必須。確定段階 リリース時点 / リリース時点。
- **DS-CF-CRYPT-003**: ストレージ at rest 暗号化は LUKS（PostgreSQL / Kafka）、MinIO SSE-S3（運用蓄積後）、etcd AES-CBC を適用。採用後の運用拡大時 で PostgreSQL 17 native TDE 評価。確定段階 リリース時点 / リリース時点 / 採用後の運用拡大時。
- **DS-CF-CRYPT-004**: アプリケーション層暗号は OpenBao Transit Engine（AES-256-GCM、keyless）で提供、p99 5ms / 10,000 ops/sec。確定段階 リリース時点。
- **DS-CF-CRYPT-005**: PII 自動暗号化はテナント別 Derived Key（HMAC 派生）でクロスデコードを構造的に防止、復号は監査記録 + Rate Limit（1 テナント 1 分 1,000 件）。確定段階 リリース時点。
- **DS-CF-CRYPT-006**: 鍵階層 3 層（Master / KEK / DEK）、ローテーション Master イベント + 4 年、KEK 365 日、DEK 90 日を固定。確定段階 リリース時点。
- **DS-CF-CRYPT-007**: Master Key は Shamir 5-of-3 分散、5 片を Platform Admin 3 + Security Officer 2 に物理配布。採用側のマルチクラスタ移行時 で HSM 統合評価。確定段階 リリース時点 / 採用側のマルチクラスタ移行時。
- **DS-CF-CRYPT-008**: Pod への秘密注入は OpenBao Agent Injector または CSI Secret Store Driver のみ、環境変数・ConfigMap・イメージ埋め込みは全面禁止。確定段階 リリース時点。
- **DS-CF-CRYPT-009**: 監査署名は OpenBao Transit Ed25519、日次バッチ、鍵ローテーション 365 日、過去鍵無期限保持。確定段階 リリース時点。
- **DS-CF-CRYPT-010**: 鍵漏洩対応 Runbook `k1s0-ops-key-compromise` を整備、Revocation / Rotation / Impact Assessment / Notification / PIR の 5 段階手順を月次訓練で確認する。確定段階 リリース時点。

## 対応要件一覧

- **NFR-E-ENC-001** 〜 **NFR-E-ENC-008**: 暗号化方式、鍵管理、ローテーション、HSM 対応。
- **NFR-E-ENC-009** 〜 **NFR-E-ENC-014**: 秘密情報管理、通信暗号化、データ at rest 暗号化、鍵漏洩対応。
