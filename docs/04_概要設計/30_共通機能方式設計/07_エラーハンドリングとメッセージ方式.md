# 07. エラーハンドリングとメッセージ方式

本ファイルは tier1 全 API および tier1 内部通信で発生するエラーの型・コード体系・分類・返却フォーマット・リトライ可否・i18n メッセージ・開発者提示と利用者提示の分離を確定させる。エラー体系がコンポーネントごとに独自定義されると、tier2 / tier3 開発者は API ごとに個別対応する必要が生じ、企画で約束した「tier1 が複雑性を吸収する」原則が破綻する。

## 本ファイルの位置付け

JTC 情シスの従来システムでは、エラーメッセージが「システムエラーが発生しました」の一言で終わり、原因切り分けが不能なことが典型的な問題であった。k1s0 は tier1 全 API で統一されたエラー型 `K1s0Error` を提供し、エラーコード・分類・リトライ可否・i18n メッセージを機械的に扱えるインタフェースを定める。

構想設計 ADR-TIER1-002（Protobuf gRPC 必須）に基づき、エラー型も Protobuf で定義し、Go / Rust / C# / TypeScript すべての言語で同一スキーマで扱う。これにより「tier2 / tier3 がエラーを受け取ったら、同じ構造で同じように扱える」状態を保証する。

## 統一エラー型 K1s0Error

### Protobuf 定義

エラー型は Protobuf で以下のように定義する。実ファイルは `src/tier1/contracts/error.proto` に配置し、buf で全言語向けに自動生成する。

```protobuf
message K1s0Error {
  string code = 1;              // E-<API>-<4桁> 形式
  string category = 2;          // input / authn / authz / conflict / rate_limit / internal / external
  string message_dev = 3;       // 開発者向け詳細メッセージ（英語固定）
  string message_user = 4;      // 利用者向けメッセージ（i18n 済み）
  string trace_id = 5;          // W3C Trace Context trace_id（サポート問い合わせ時に参照）
  bool retriable = 6;           // クライアント側リトライ可否のヒント
  int32 retry_after_seconds = 7; // リトライ推奨待機秒数（0 なら指定無し）
  map<string, string> details = 8; // 追加メタデータ（発生コンポーネント名、リソース ID 等）
}
```

gRPC エラー返却時は `Status.WithDetails` で K1s0Error を添付する。HTTP ゲートウェイを経由する場合は JSON 本体に展開してレスポンスに含める。

### エラーコード体系

エラーコードは `E-<API>-<4桁>` 書式とする。API 部分は tier1 11 API の略号（`INVOKE` / `STATE` / `PUBSUB` / `SECRETS` / `BINDING` / `WORKFLOW` / `LOG` / `TELEMETRY` / `DECISION` / `AUDIT` / `FEATURE`）を用い、4 桁は API 内部の連番で 0001 から採番する。

例: `E-STATE-0001` は State API の接続失敗、`E-INVOKE-0002` は Service Invoke のタイムアウト、`E-AUTHZ-0001` は認可 deny、`E-COMMON-9999` は分類不能な内部エラー。

コード体系は API プレフィックスで所属 API が即座に分かる設計とし、サポート問い合わせ時の切り分けを最短化する。新規エラー追加時は [../20_ソフトウェア方式設計/02_外部インタフェース方式設計/06_API別詳細方式/](../20_ソフトウェア方式設計/02_外部インタフェース方式設計/06_API別詳細方式/) 各 API ドキュメントにエラーコード表を追加する。

## エラー分類

### 7 分類

全エラーは以下 7 分類のいずれかに属する。分類を定義することで、クライアント側の対応コード（リトライするか、ユーザーに見せるか、ログのみか）を機械的に判断できる。

**input**: クライアント側の入力不正（バリデーション違反、必須欠落、形式エラー）。HTTP 400 / gRPC INVALID_ARGUMENT。リトライ不可、ユーザーに提示してクライアント側で訂正を促す。

**authn**: 認証失敗（JWT 不正、期限切れ、未提示）。HTTP 401 / gRPC UNAUTHENTICATED。詳細は [01_認証方式設計.md](01_認証方式設計.md) の 5 分類に準ずる。リトライは再認証後のみ可。

**authz**: 認可失敗（権限不足、テナント越境、deny ルール）。HTTP 403 / gRPC PERMISSION_DENIED。詳細は [02_認可と権限モデル方式.md](02_認可と権限モデル方式.md) に準ずる。リトライ不可、管理者に権限付与を依頼する必要あり。

**conflict**: リソース競合（楽観的ロック衝突、ユニーク制約違反、Saga 中の状態不整合）。HTTP 409 / gRPC ABORTED。楽観的ロック系は再取得してリトライ可能。

**rate_limit**: レート制限超過（テナントクォータ、API 個別制限、DDoS 対策）。HTTP 429 / gRPC RESOURCE_EXHAUSTED。`retry_after_seconds` に待機秒数を設定、クライアントは exponential backoff で再試行。

**internal**: tier1 内部エラー（DB 接続失敗、NullPointer、panic など）。HTTP 500 / gRPC INTERNAL。リトライ可（冪等性保証がある API に限る）、運用者調査が必要。

**external**: tier1 が依存する外部サービス（Keycloak / OpenBao / Kafka / PostgreSQL）の障害。HTTP 502 / 503 / 504 / gRPC UNAVAILABLE or DEADLINE_EXCEEDED。Circuit Breaker と組み合わせて扱う、詳細は [../40_制御方式設計/03_リトライとサーキットブレーカー方式.md](../40_制御方式設計/03_リトライとサーキットブレーカー方式.md)。

### 分類と HTTP / gRPC マッピング

| 分類 | HTTP | gRPC | retriable | 利用者提示 | 監査記録 |
| --- | --- | --- | --- | --- | --- |
| input | 400 | INVALID_ARGUMENT | false | 有（訂正指示） | 有（warn） |
| authn | 401 | UNAUTHENTICATED | 認証後のみ | 有（再認証促進） | 有（warn/error） |
| authz | 403 | PERMISSION_DENIED | false | 有（権限不足明示） | 有（warn） |
| conflict | 409 | ABORTED | 楽観的ロックのみ | 有（再取得誘導） | 有（info） |
| rate_limit | 429 | RESOURCE_EXHAUSTED | true | 有（待機時間明示） | 有（info） |
| internal | 500 | INTERNAL | 冪等のみ | 最小限 | 有（error） |
| external | 502-504 | UNAVAILABLE / DEADLINE_EXCEEDED | true | 最小限 | 有（error） |

上表はクライアント実装側の判断ロジックを機械化するための索引である。tier1 SDK はこの分類を見て自動リトライ制御を行い、tier2 / tier3 開発者は分類ロジックを書かなくて済む。

## リトライ制御

### exponential backoff

`retriable=true` かつ `rate_limit` または `external` 分類のエラーは、クライアント SDK 側で自動リトライする。アルゴリズムは exponential backoff with jitter とし、パラメータは以下固定。

- 初期待機: 100ms
- 乗数: 2.0
- 最大待機: 10 秒
- 最大試行回数: 3 回
- jitter: ±20%

`retry_after_seconds` がサーバから返された場合はその値を優先する（HTTP 429 の `Retry-After` ヘッダと同意味）。tier1 内部通信の詳細リトライポリシーは [../40_制御方式設計/03_リトライとサーキットブレーカー方式.md](../40_制御方式設計/03_リトライとサーキットブレーカー方式.md) に委譲する。

### 冪等性要件

`internal` 分類のリトライは、呼び出される API が冪等性を提供する場合のみ実施する。非冪等 API（State Increment、PubSub Publish without idempotency-key）のリトライは重複副作用のリスクがあるため、SDK は自動リトライしない。tier1 API の冪等性は [../40_制御方式設計/02_冪等性設計方式.md](../40_制御方式設計/02_冪等性設計方式.md) で個別に定義する。

## メッセージ i18n

### 2 系統のメッセージ

`K1s0Error` は `message_dev` と `message_user` の 2 系統を持つ。

**message_dev** は開発者・運用者向けの詳細メッセージで、英語固定である。エラーの技術的原因（例: `connection refused to valkey.tier1.svc.cluster.local:6379`）を記載し、ログおよび内部管理 UI のみに表示する。利用者レスポンスに含めるのは開発環境のみとする。

**message_user** はエンドユーザー向けメッセージで、Accept-Language ヘッダに応じて日本語 / 英語を切り替える。Phase 1a は日本語のみ、Phase 1b で英語、Phase 2 で中国語簡体字を追加する計画である。メッセージは技術用語を避け、ユーザーがとるべき次のアクション（「再度お試しください」「管理者にお問い合わせください」等）を必ず含める。

### メッセージカタログ管理

メッセージは Protobuf enum ではなく、外部 JSON カタログ（`src/tier1/i18n/messages-ja.json` / `messages-en.json`）で管理する。tier1 各コンポーネントは起動時にロードし、エラーコードをキーに引く。翻訳の追加・修正は開発者以外（翻訳担当）でも PR で参加可能な構造を保つ。

翻訳カタログの整合性は CI でチェックする。`message_dev` 辞書と `messages-ja.json` / `messages-en.json` のキー集合が一致しない場合 CI fail とする（翻訳漏れ検知）。

## スタックトレースの扱い

### 環境別ポリシー

- **dev**: レスポンス `details.stack` にスタックトレース全文を含める。開発者が即座に原因特定できる状態を最優先。
- **stg**: レスポンスには含めない、内部ログには含める。本番同等の挙動でテストしつつ調査可能。
- **prod**: レスポンスには含めない、内部ログにのみ含める。スタックトレース経由の情報漏洩を防ぐ。

スタックトレースに PII が混入する可能性があるため、ログ書き込み前に [03_ログ出力方式.md](03_ログ出力方式.md) の自動マスキングを適用する。

### panic / 未捕捉例外のハンドリング

Go は recover ミドルウェア、Rust は panic hook、C# は AppDomain.UnhandledException、TypeScript は `process.on('uncaughtException')` で捕捉し、`K1s0Error{code: "E-COMMON-9999", category: "internal"}` として包んで返却する。panic 発生は必ず監査ログに記録し、SEV2 アラートを発報する（panic は設計欠陥の示唆）。

## エラーメトリクス

全エラー発生は [06_メトリクス収集方式.md](06_メトリクス収集方式.md) の `k1s0_api_errors_total` にカウントする。ラベル `error_code` は高カーディナリティ化するため、API × category の 2 軸で集約し、具体的コードは Exemplar で Trace へ pivot する方式とする。

## 設計 ID 一覧

- **DS-CF-ERR-001**: 統一エラー型 `K1s0Error` を Protobuf 定義し、tier1 全 API で使用。全言語で自動生成。確定フェーズ Phase 1a。
- **DS-CF-ERR-002**: エラーコード体系 `E-<API>-<4桁>` を固定、API 別連番で 0001 から採番、API プレフィックスで所属を即判別可能とする。確定フェーズ Phase 1a。
- **DS-CF-ERR-003**: エラー分類 7 種（input / authn / authz / conflict / rate_limit / internal / external）を固定し、HTTP / gRPC ステータスマッピングを表で強制する。確定フェーズ Phase 1a。
- **DS-CF-ERR-004**: リトライは exponential backoff with jitter（初期 100ms / 乗数 2.0 / 最大 10s / 最大 3 回 / jitter ±20%）を SDK 標準で実装。確定フェーズ Phase 1b。
- **DS-CF-ERR-005**: `retry_after_seconds` がサーバから返された場合はその値を優先、rate_limit / external のみ自動リトライ対象。確定フェーズ Phase 1b。
- **DS-CF-ERR-006**: `message_dev`（英語固定、開発者向け）と `message_user`（i18n 済み、ユーザー向け）の 2 系統分離。Phase 1a 日本語、Phase 1b 英語、Phase 2 中国語。確定フェーズ Phase 1a。
- **DS-CF-ERR-007**: メッセージカタログは外部 JSON で管理、CI で `message_dev` と翻訳辞書のキー集合一致を検証する。確定フェーズ Phase 1b。
- **DS-CF-ERR-008**: スタックトレースは dev のみレスポンスに含め、stg / prod は内部ログに限定。書き込み前に PII マスキング適用。確定フェーズ Phase 1a。
- **DS-CF-ERR-009**: panic / 未捕捉例外は全言語共通ミドルウェアで捕捉し `E-COMMON-9999` で包み、監査記録 + SEV2 アラート発報する。確定フェーズ Phase 1b。
- **DS-CF-ERR-010**: エラー発生を `k1s0_api_errors_total` にカウント、ラベルは API × category の 2 軸、具体コードは Exemplar で Trace へ pivot。確定フェーズ Phase 1b。

## 対応要件一覧

- **FR-T1-COMMON-001**（想定）: tier1 API 共通エラー型。
- **NFR-C-OBS-005**: エラー分析のためのメトリクス・Trace 連携。
- **NFR-D-UX-003**: 利用者向けエラーメッセージの明確性。
- **NFR-E-SEC-011**: エラーメッセージ経由の情報漏洩防止（スタックトレース、内部構造露出）。
- **NFR-I18N-001**（想定）: i18n サポート。
