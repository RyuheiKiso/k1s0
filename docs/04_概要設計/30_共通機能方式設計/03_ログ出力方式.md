# 03. ログ出力方式

本ファイルは tier1 全コンポーネント（Go / Rust）、tier2 / tier3 SDK（Go / C# / TypeScript）、infra OSS から出力される構造化ログのスキーマ・配送経路・保管期間・PII マスキング方式を確定させる。ログは障害分析・性能分析・監査の起点となる一次情報であり、ここでスキーマが揺らぐと後続の観測基盤・監査基盤のすべてが歪む。

## 本ファイルの位置付け

採用側組織の情シスが抱える典型的な問題は「ログが場当たりで、障害分析時に grep で探すしかない」ことである。k1s0 は tier1 Log API を通じて構造化ログを強制することで、この問題を構造的に解消することを企画段階で約束している。本章はその Log API の実装契約を、tier2 / tier3 から見たインタフェースではなく、出力側の共通規約として定義する。

構造化ログのスキーマは OpenTelemetry Logs Data Model と Elastic Common Schema（ECS）v8.x を参照し、k1s0 固有の必須フィールドを追加する形で確定する。全言語・全コンポーネントで同一スキーマを強制するのは、Loki / Tempo / Mimir の相関解析を「ただ JOIN するだけ」で成立させるためである。

## ログスキーマ

### 必須フィールド

全ログエントリは以下のフィールドを必須とする。1 つでも欠けたログは tier1 Log API が受理を拒否し、送信者側に 400 Bad Request を返す（内部コンポーネントから Log API を経由しない出力はコンテナの stderr で検知し、OTel Collector 側で reject）。

- **timestamp**: RFC 3339 形式、ナノ秒精度 UTC（例: `2026-04-15T12:34:56.789012345Z`）。
- **level**: `trace` / `debug` / `info` / `warn` / `error` / `fatal` の 6 段階。
- **trace_id**: W3C Trace Context の 32 桁 hex。未計装コンテキストでは全 0 を入れる。
- **span_id**: W3C Trace Context の 16 桁 hex。同上。
- **tenant_id**: テナント識別子、正規表現 `^[a-z0-9-]{3,32}$`。システムログは `system` を固定値で入れる。
- **service**: コンポーネント名（例: `k1s0.tier1.state` / `k1s0.tier1.audit`）。
- **version**: サービスのセマンティックバージョン（例: `1.2.3`）。
- **message**: 人間可読のメッセージ本文。英語を基本、例外的に日本語も許可する。

### 任意フィールド

文脈に応じて以下を付与する。付与しない場合はフィールド自体を省略し、null を入れない（Loki 側でラベル・インデックスの膨張を避けるため）。

- **user_id**: JWT `sub` クレーム。認証コンテキストがある場合のみ。
- **api**: tier1 API 名（例: `State.Get` / `PubSub.Publish`）。
- **resource_id**: 操作対象リソースの ID。
- **error.code**: エラーコード（`E-INVOKE-0001` 形式）。
- **error.message**: エラーメッセージ（PII マスク済み）。
- **error.stack**: スタックトレース（dev / stg のみ、prod は省略）。
- **duration_ms**: 処理時間ミリ秒。
- **peer.ip**: 通信相手の IP（外部クライアントは class 化後の値、内部は SPIFFE ID）。

### スキーマの言語横断統一

Go は `log/slog`（Go 1.21+）、Rust は `tracing` crate、C# は `Microsoft.Extensions.Logging` + Serilog、TypeScript は `pino` を標準ライブラリとして採用する。各言語で上記スキーマに準拠するラッパライブラリ `k1s0-logger-<lang>` を提供し、開発者は「ロガーを初期化して `info("msg", field=value)` と呼ぶだけ」で準拠ログを出せる状態を作る。

ラッパライブラリの責務は、必須フィールドの自動補完（timestamp / service / version / trace_id / span_id / tenant_id）、JSON シリアライズ、PII 自動マスキングの 3 点である。tier2 / tier3 開発者は必須フィールドを意識せずに済む。

## ログレベル運用

### 環境別既定レベル

環境ごとに既定ログレベルを固定する。レベル変更は ConfigMap 経由で Hot Reload 可能とし、再起動を伴わない。

- **prod**: `info` 以上を出力。`debug` / `trace` は完全抑制。
- **stg**: `debug` 以上。本番と同等の性能評価のため `trace` は抑制。
- **dev**: `trace` 以上すべて出力。開発者ローカルでのデバッグに使用。

### レベルの定義

曖昧なレベル選定を避けるため各レベルの意味を以下に固定する。レベル基準がブレると、`error` が溢れて本当の障害が埋もれる、`warn` が無視される、といった運用破綻が生じる。

**trace** は関数入出力・ループ内ステップなどの微細情報。開発時のみ有効化する。**debug** は分岐判定・キャッシュヒット状況など、機能が意図通り動いたかを追跡する情報。**info** は業務的に意味のあるイベント（API 呼び出し完了・Saga 遷移など）。**warn** は即障害ではないが注意すべき状況（リトライ発生・Circuit Breaker open 手前）。**error** は処理失敗（リトライ尽きた・データ不整合検知）。**fatal** はプロセス継続不能（起動時の依存欠落・panic でのクラッシュ直前）。

## PII 自動マスキング

### マスキングの対象と範囲

ログに混入した PII（個人識別情報）は tier1 Log API が受信後、書き込み前に COMP-T1-PII に委譲して自動マスキングする。マスク対象は以下のパターンを標準で検出する。

- メールアドレス: RFC 5322 準拠パターン → `***@***.***`
- 電話番号: 国際 E.164 + JP 国内形式 → `+81-***-****-****` / `***-****-****`
- クレジットカード番号: Luhn チェックで判定 → 下 4 桁のみ残す `****-****-****-1234`
- マイナンバー: 12 桁数字の特定パターン → `************`
- 住所: 都道府県 + 市区町村 + 番地パターン → 市区町村までで truncate

加えてテナントが独自のマスキングルールを追加できる仕組みを PII API 経由で提供する。ルール追加は Tenant Admin ロール権限が必要で、ルール変更は監査ログに記録される。

### マスキングのパフォーマンス

マスキング処理のオーバーヘッドは 1 ログエントリあたり平均 0.2ms、p99 1.0ms 以内を目標とする。実装は Rust の regex crate による事前コンパイル済みパターンマッチを用い、ログ量 20GB/日（後述）のオンザフライ処理が実用レンジに入ることを リリース時点 負荷試験で検証する。

### マスキング失敗時の取り扱い

マスキング処理が例外で失敗した場合、該当ログは「fail-closed」で出力前に破棄し、代わりにメタログ（`pii.masking_failed`、trace_id のみ保持）を出す。ログを残すことよりも PII 漏洩を防ぐことを優先する設計である。破棄件数は監視メトリクスとして Mimir に送り、1 時間あたり 10 件を超えたら SEV2 アラートを発報する（マスキングルールの欠陥検知）。

## 配送経路

### リリース時点 の経路

リリース時点 では AGPL OSS を採用しないため、ログは以下経路で配送する。

- アプリケーション → stdout / stderr（JSON 1 行）
- コンテナランタイム → ノードローカルファイル `/var/log/containers/`
- Fluent Bit（DaemonSet） → ファイルローテーション + ローカル 7 日保持

リリース時点 は単一 VM 構成のため集約基盤を立てない。障害分析は VM にログインしてファイルを参照する運用で成立する（起案者 1 名体制で実運用できる設計）。

### 運用蓄積後の経路

リリース時点 で Grafana Loki を採用し、以下経路で集約する。AGPL OSS なので ADR-0003 の 5 原則（プロセス分離 / ライブラリリンクなし / 無改変利用 / 外部境界遮断 / 監査ログ可検証性）を満たす構造で導入する。

- アプリケーション → stdout / stderr
- OpenTelemetry Collector（OTLP 受信） → Loki Remote Write
- Loki → S3 互換（リリース時点 で MinIO 導入、当面はローカル PVC）

OTel Collector は CNCF 所属（Apache 2.0）で AGPL 隔離原則の影響を受けない。Loki との接続は HTTP Remote Write のみで、ライブラリリンクは一切無い。Loki 側への書き込みは OTel Collector の内部タグで集約後送信し、Loki 側のクエリは Grafana 経由（OIDC で保護された内部 UI）でのみ可能とする。

## 保管期間

ログ種別ごとに異なる保管期間を設定する。長期保管はコストと検索性能に直結するため、種別ごとに正当化された期間を設ける。

| 種別 | 保管期間 | 根拠 |
| --- | --- | --- |
| info ログ | 90 日 | 障害原因分析の現実的リミット。業界事例（AWS CloudWatch Logs 既定 90 日）に準拠。 |
| warn / error ログ | 180 日 | エラー傾向の季節変動を捉える半年分を確保。 |
| 認証・認可イベント | 365 日 | セキュリティ事後調査の法的要請を考慮し 1 年確保。 |
| 監査イベント | 7 年 | J-SOX 対応。詳細は [04_監査証跡方式.md](04_監査証跡方式.md) に委譲。 |
| debug / trace | 7 日 | 障害直後のみ参照する想定で短縮。 |

上表は保管期間設定の索引である。実装上は Loki の `retention_period` を種別ごとに設定し、超過分は自動削除する。監査イベントのみ Loki ではなく CloudNativePG の WORM テーブルに格納し、[04_監査証跡方式.md](04_監査証跡方式.md) の方式で長期保管する。

## ログボリューム予測

中規模テナント（5 テナント、合計 150 RPS）で以下のボリュームを見込む。この見積は リリース時点 のストレージ調達の根拠となる。

- 1 リクエストあたりの平均ログ行数: 8 行（入口 + 各 API 呼び出し + 結果）
- 1 ログあたり平均サイズ: 1.6 KB（JSON + 必須フィールド + PII マスク済みメッセージ）
- 日次生成量: 150 × 8 × 1.6 KB × 86,400 = 約 166 GB/日（raw）
- 圧縮後（Loki snappy 圧縮率 8 倍想定）: 約 20 GB/日
- 90 日保管時ストレージ: 1.8 TB
- 監査 7 年保管（別テーブル）: 年間 約 200 GB × 7 = 1.4 TB

この数値は リリース時点 時点の参考値であり / パイロット運用で実測し / 本番採用時に修正する前提である。見積より倍増しても 4 TB オーダーで収まる設計とし、調達時は余裕を持って 8 TB クラスのストレージを割り当てる。

## ログ APIとの関係

tier1 Log API（FR-T1-LOG-001〜004）は、本章で定義したスキーマを出力インタフェースとして受理する。tier2 / tier3 からは Log API クライアントライブラリを経由して送信し、内部では OTel Collector に OTLP で転送される。tier1 内部コンポーネントは直接 OTel Collector に送信し、Log API を経由しない（内部経路の省略による性能確保のため）。

ログ API の詳細インタフェース設計は [../20_ソフトウェア方式設計/02_外部インタフェース方式設計/06_API別詳細方式/07_Log_API方式.md](../20_ソフトウェア方式設計/02_外部インタフェース方式設計/06_API別詳細方式/07_Log_API方式.md) を参照する。本章は出力側の共通規約のみを扱う。

## 設計 ID 一覧

- **DS-CF-LOG-001**: 全ログは JSON 構造化、OpenTelemetry Logs Data Model + ECS v8.x 準拠のスキーマを採用する。確定段階 リリース時点。
- **DS-CF-LOG-002**: 必須フィールドは timestamp / level / trace_id / span_id / tenant_id / service / version / message の 8 項目、欠落ログは受理拒否する。確定段階 リリース時点。
- **DS-CF-LOG-003**: 言語別ラッパライブラリ `k1s0-logger-<lang>` を Go / Rust / C# / TypeScript で提供し、必須フィールド自動補完・PII マスク・JSON シリアライズを責務とする。確定段階 リリース時点。
- **DS-CF-LOG-004**: ログレベルは 6 段階（trace/debug/info/warn/error/fatal）、環境別既定 prod=info / stg=debug / dev=trace を固定する。確定段階 リリース時点。
- **DS-CF-LOG-005**: PII 自動マスキングを COMP-T1-PII に委譲し、email / phone / card / mynumber / address の 5 標準パターン + テナント拡張を適用する。確定段階 リリース時点。
- **DS-CF-LOG-006**: マスキング p99 1ms 以内、失敗時は fail-closed でログ破棄 + メタログ出力を必須化する。確定段階 リリース時点。
- **DS-CF-LOG-007**: リリース時点 は stdout + Fluent Bit + ローカルファイル 7 日、運用蓄積後は OTel Collector + Loki Remote Write に移行する。確定段階 リリース時点 / リリース時点。
- **DS-CF-LOG-008**: 保管期間を info 90 日 / warn-error 180 日 / authn-authz 365 日 / debug-trace 7 日に固定する。監査は別経路。確定段階 リリース時点。
- **DS-CF-LOG-009**: 中規模 150 RPS 想定で日次 20 GB（圧縮後）を基準ボリュームとし、リリース時点 パイロット実測で更新する。確定段階 リリース時点。
- **DS-CF-LOG-010**: AGPL OSS（Loki）の採用は ADR-0003 の 5 原則（プロセス分離 / 非リンク / 無改変 / 境界遮断 / 監査可検証性）を満たす構造で 運用蓄積後に限定する。確定段階 リリース時点。

## 対応要件一覧

- **FR-T1-LOG-001** 〜 **FR-T1-LOG-004**: tier1 Log API 機能要件。
- **FR-INFO-LOGMODEL**: ログ情報モデル要件、本章のスキーマが直接対応。
- **FR-T1-PII-001** 〜 **FR-T1-PII-002**: PII 自動マスキング要件（実装委譲）。
- **NFR-C-LOG-001** 〜 **NFR-C-LOG-005**: 運用ログ、相関 ID、保管期間、集約先。
- **NFR-E-MON-007**: ログに混入する PII の保護。
- **NFR-H-AUD-002**: ログと監査の分離、保管期間差別化。

構想設計 ADR は ADR-OBS-001（Grafana LGTM）、ADR-OBS-002（OTel Collector）、ADR-0003（AGPL 分離）である。分散トレーシングの trace_id / span_id については [05_分散トレーシング方式.md](05_分散トレーシング方式.md) を参照する。
