# 02. 認可と権限モデル方式

本ファイルは認証済みの主体（ユーザー・ワークロード）が tier1 公開 API 11 種に対して「何をして良いか」を判定する認可方式を確定させる。認可は認証とは独立した概念であり、認証済みであっても権限の無い操作は拒否する。本章は RBAC と ABAC をハイブリッドに組み合わせ、テナント分離と細粒度アクセス制御を同時に実現する方式を定める。

## 本ファイルの位置付け

tier1 は単一プラットフォームで複数テナントを収容するため、テナント間のアクセス制御を一切誤ってはならない。同時に、JTC 情シスの運用実態ではロール数が少数（5 種程度）で、ロール追加よりも「ロール × 属性」の掛け合わせで細粒度制御したいケースが大半となる。このため本章は純粋 RBAC でも純粋 ABAC でもなく、両者のハイブリッドを採用する。

認可ロジックは tier1 全 API で一貫して適用する必要がある。API ごとに独立実装すると、あるコンポーネントの認可バグが監査で発覚した際に全 API を再実装するコストが発生する。本章は共通 Interceptor と ZEN Engine 評価の二段階で認可を集約し、変更影響範囲を最小化する。

## 認可モデルの全体像

k1s0 の認可は「ロール（Role）」と「属性（Attribute）」の 2 軸で判定する。Role は主体が所属するコミュニティを表現する粗粒度のカテゴリ、Attribute は個別リクエスト時に評価される細粒度のコンテキストである。判定は tier1 共通 gRPC Interceptor が ZEN Engine に問い合わせ、JDM 決定表で宣言的に評価する。

判定の流れは次の通りである。認証済み JWT から `sub` / `roles` / `tenant_id` を抽出し、リクエストから `api` / `action` / `resource_type` / `resource_sensitivity` / `env` などの属性を抽出する。これらを入力として ZEN Engine の決定表（`k1s0-authz.json`）を評価し、`allow` または `deny` を得る。評価結果は必ず監査ログに記録し、deny の場合も記録する。deny を記録しないと、攻撃試行の痕跡が残らず事後分析が不能となるためである。

## ロール定義

### 標準 5 ロール

k1s0 では以下 5 種類の標準ロールを Phase 1a で定義する。ロール数を 5 に絞るのは、運用 2 名体制でロール管理を破綻させないための制約である。JTC 情シスの典型的な権限階層に照らし、以下 5 種類で全ケースの 95% 以上をカバーできる想定である。

**Platform Admin** は k1s0 基盤全体の管理者ロールであり、Keycloak Realm 管理・OpenBao 管理・Kubernetes Cluster 運用のすべての権限を持つ。tier1 API レベルでは全 API への read/write 権限を持ち、全テナント横断のアクセスが可能である。Phase 1a 時点では起案者 1 名のみが保持する。

**Tenant Admin** は個別テナント内の管理者ロールで、自テナントの設定変更・ユーザー追加・リソースクォータ調整権限を持つ。他テナントへのアクセスは不可である。tier1 API レベルでは Feature API の Flag 設定、Audit API の検索、PII API のマスキングルール設定が可能となる。

**Developer** は開発者ロールで、自テナントの State / PubSub / Secrets / Binding / Workflow / Log / Telemetry / Decision API への read/write 権限を持つ。Audit API は write 権限のみ（自動発行）、PII API は業務ロジックで自動呼び出しされる形でのみ使用可能、Feature API は read のみとする。

**Viewer** は閲覧ロールで、自テナントの全 API に対して read 権限のみを持つ。ログ参照・メトリクス確認・State の読み取りのみ可能であり、いかなる書き込みも拒否される。

**Audit Reader** は監査担当者向けロールで、全テナント横断の Audit API および Log API への read 権限を持つ。業務 API への write 権限は一切持たない。監査業務の独立性を担保するため Platform Admin とは分離する。

### ロール階層と包含関係

上記 5 ロールは単純な包含関係を持たない。Platform Admin が Developer 権限を包含すると「踏み台化」（Platform Admin で日常作業する誘惑）が生じ、最小権限原則を崩すためである。Platform Admin は管理操作専用とし、日常開発は Developer ロールでログインし直す運用を強制する。

## 属性定義

認可判定に用いる属性は以下 6 種類を標準とする。属性を多くしすぎると決定表の組合せ爆発が起きるため、6 種類を上限とし、追加する際は ADR での合意を必要とする。

- **tenant_id**: 主体とリソース両方に付与される。両者の一致が基本認可条件となる。
- **env**: `dev` / `stg` / `prod` の 3 値。prod 環境への書き込みは追加承認を要求する。
- **sensitivity_level**: リソースの機密度。`public` / `internal` / `confidential` / `restricted` の 4 値。
- **action**: `read` / `write` / `delete` / `admin` の 4 値。
- **time_of_day**: JST タイムゾーンでの時刻。深夜（23:00-06:00）の prod 書き込みを制限する際に使用。
- **source_ip_class**: `office` / `vpn` / `external` の 3 値。external からの restricted アクセスを自動 deny する際に使用。

属性は JWT クレームまたはリクエストコンテキストから抽出する。クライアント側が改ざん可能な値（User-Agent など）は属性として採用しない。source_ip_class は Envoy Gateway が信頼境界で付与するヘッダ `x-k1s0-ip-class` を用い、tier1 内部では再評価しない（集約度と性能のため）。

## 評価エンジン（ZEN Engine）

認可ルールは ZEN Engine の JDM 決定表として記述する。ZEN Engine は構想設計 ADR-RULE-001 で Decision API と共有化されており、認可も同じエンジン・同じ方式で評価する。これにより認可ルールと業務ルールが同じ手法で管理でき、運用学習コストが一本化される。

認可決定表のファイル名は `k1s0-authz.jdm` とし、OpenBao または Git 管理リポジトリ `infra/authz-policies` で管理する。Git 管理下の JSON は Argo CD でデプロイし、Kubernetes ConfigMap に反映される。ZEN Engine は ConfigMap を watch して自動リロードする（リロード遅延は p99 5 秒以内、inotify ベース）。

評価レイテンシは tier1 API p99 500ms 予算のうち 3ms 以内とする。ZEN Engine Rust 実装の in-process 評価でこの予算は余裕を持って達成可能であり、実測値では平均 0.3ms、p99 0.8ms を確認している（構想設計時プロトタイプ）。

### 決定表サンプル

決定表の概念例を以下に示す。実運用の決定表は 80 行程度を想定しているが、一部抜粋である。

```
Input:  role, action, tenant_id_match, env, sensitivity_level, source_ip_class
Output: effect (allow | deny), reason

R1: role=PlatformAdmin                        → allow, "platform-admin-allpass"
R2: role=AuditReader AND action=read         → allow, "audit-reader-read-only"
R3: role=Developer AND action=write
    AND env=prod AND sensitivity_level=restricted
    AND source_ip_class!=office              → deny, "prod-restricted-office-only"
R4: role=Viewer AND action!=read             → deny, "viewer-readonly"
R5: tenant_id_match=false                    → deny, "cross-tenant-forbidden"
R6: default                                   → deny, "fail-closed-default"
```

決定表は常に fail-closed（明示的な allow が無ければ deny）で構成する。allow ルールから外れたケースが許可される設計は絶対に禁止し、R6 の default deny を必ず最終行に配置する。

## テナント分離の強制

### クレーム検証による tenant_id 一致強制

全 tier1 API において、JWT の `tenant_id` クレームとリクエストが指定する `tenant_id`（API パラメータまたはリソース ID 内）は共通 Interceptor で強制一致検証する。不一致時は認可評価に入る前に即座に deny（403 Forbidden）とし、監査ログに `authz.cross_tenant_attempt` を記録する。

Platform Admin / Audit Reader のみクロステナントアクセスを許可するが、その場合でも `x-k1s0-tenant-override` metadata の明示指定を要求する。明示指定なしのクロステナントアクセスは Platform Admin であっても deny する。「うっかり他テナントを操作する」事故を構造的に防ぐためである。

### データストア層の tenant_id 二重防御

認可 Interceptor で deny できなかった場合の多重防御として、データストア層（Valkey / CloudNativePG / Kafka）でも tenant_id 分離を強制する。PostgreSQL は Row Level Security（RLS）で `tenant_id` カラム必須、Valkey はキー prefix `t:<tenant_id>:` 必須、Kafka はトピック名 prefix `<tenant_id>.` 必須とする。

この二重防御により、アプリケーション層の認可バグが発生しても、データストア層でクロステナントアクセスが構造的に防がれる。Kubernetes 層での NetworkPolicy によるトラフィック分離も組み合わせ、計 3 層の防御を設ける。

## API キーとサービスアカウント

### 長期クレデンシャル回避方針

従来型の API キー（静的 Secret 文字列）は原則禁止とする。長期クレデンシャルは漏洩時の影響範囲が大きく、ローテーション漏れで無効化が遅れる運用リスクがあるためである。代替として SPIFFE SVID（X.509 / JWT）を用い、最大寿命 1 時間の短期クレデンシャルで置き換える。

やむを得ず長期 API キーが必要な場合（外部 SaaS 連携など）は OpenBao の Dynamic Secrets を用い、実行時に自動発行・自動失効する。Dynamic Secret の最大寿命は 24 時間、ローテーション周期は 6 時間とする。

### Service Account の位置付け

Kubernetes の ServiceAccount は Pod 単位のワークロード ID として SPIRE Agent が Attestor に使用する。Kubernetes ServiceAccount Token そのものは tier1 認可には用いず、SPIRE が発行する SVID のみを信頼する。これにより、Kubernetes の Token 漏洩が直ちに認可侵害に直結することを防ぐ。

## 権限委譲

OAuth 2.1 Token Delegation（token exchange RFC 8693）による権限委譲を Phase 2 で導入する。Phase 1a〜1c では権限委譲は不可とし、主体の認証トークンをそのまま使う。Phase 2 導入時はダウンストリーム API に対して縮退された scope を持つ委譲トークンを発行し、権限の過剰伝播を防ぐ。

## 監査連携

認可評価の結果（allow / deny の両方）は [04_監査証跡方式.md](04_監査証跡方式.md) で定義するハッシュチェーン監査ログに記録する。記録内容は以下とする。

- `audit_id` / `timestamp` / `tenant_id` / `sub` / `api` / `action` / `resource_id`
- `effect`（allow / deny）
- `matched_rule`（ZEN Engine が一致したルール ID）
- `evaluation_latency_ms`
- `roles` / `source_ip_class` など評価属性のスナップショット

deny イベントは `authz.denied` として Log API 経由で警告ログに出し、1 時間あたり同一 `sub` の deny が 50 回を超えた場合は SEV3 アラートを発報する（攻撃試行の早期検知）。

## 設計 ID 一覧

- **DS-CF-AUTHZ-001**: RBAC（5 標準ロール） + ABAC（6 標準属性）のハイブリッドモデルを認可基本方式として採用する。確定フェーズ Phase 1a。
- **DS-CF-AUTHZ-002**: 標準ロールを Platform Admin / Tenant Admin / Developer / Viewer / Audit Reader の 5 種類に固定する。追加にはアーキテクト承認と ADR 起票を要する。確定フェーズ Phase 1a。
- **DS-CF-AUTHZ-003**: 認可評価エンジンとして ZEN Engine を採用し、決定表 `k1s0-authz.jdm` は Git 管理 + Argo CD デプロイ + ConfigMap watch で反映する。確定フェーズ Phase 1b。
- **DS-CF-AUTHZ-004**: 決定表は fail-closed 構造とし、default deny を最終行に必須配置する。確定フェーズ Phase 1a。
- **DS-CF-AUTHZ-005**: JWT `tenant_id` とリクエスト `tenant_id` の一致検証を tier1 共通 Interceptor で強制する。違反時は認可評価前に即 deny。確定フェーズ Phase 1a。
- **DS-CF-AUTHZ-006**: データストア層の tenant_id 分離（PostgreSQL RLS / Valkey key prefix / Kafka topic prefix）を二重防御として必須化する。確定フェーズ Phase 1b。
- **DS-CF-AUTHZ-007**: 長期 API キーは原則禁止、SPIFFE SVID（寿命 1 時間）または OpenBao Dynamic Secrets（最大寿命 24 時間 / ローテーション 6 時間）で代替する。確定フェーズ Phase 1b。
- **DS-CF-AUTHZ-008**: 認可評価 p99 レイテンシ 3ms 以内、ConfigMap リロード遅延 p99 5 秒以内を性能要件として固定する。確定フェーズ Phase 1b。
- **DS-CF-AUTHZ-009**: 認可結果（allow / deny 両方）を監査証跡に記録し、deny は Log API 警告出力、1 時間 50 回超過で SEV3 アラート発報する。確定フェーズ Phase 1b。
- **DS-CF-AUTHZ-010**: Phase 2 で OAuth 2.1 Token Delegation を導入する。Phase 1a〜1c は権限委譲不可。確定フェーズ Phase 2。

## 対応要件一覧

- **NFR-E-AUTHZ-001** 〜 **NFR-E-AUTHZ-006**: RBAC / ABAC モデル、テナント分離、最小権限、権限委譲。
- **NFR-E-SEC-004**: データストア層の分離による多層防御。
- **NFR-H-AUD-001**: 認可決定の監査証跡要件。
- **NFR-H-COMP-002**: J-SOX 的な職務分掌要件への対応（Platform Admin と Audit Reader の分離）。
- **FR-T1-DECISION-002**: 認可評価も ZEN Engine で処理するため Decision API 内部実装と共有。

関連構想設計 ADR は ADR-RULE-001（ZEN Engine）、ADR-SEC-001（Keycloak）、ADR-SEC-003（SPIFFE/SPIRE）である。認証の前提については [01_認証方式設計.md](01_認証方式設計.md)、エラー返却については [07_エラーハンドリングとメッセージ方式.md](07_エラーハンドリングとメッセージ方式.md)、監査記録については [04_監査証跡方式.md](04_監査証跡方式.md) を参照する。
