# 80. トレーサビリティ

本カテゴリは上流（要件定義）と下流（詳細設計以降）への紐付けを一元管理する。設計項目が独立して浮遊しないよう、すべての設計 ID を要件 ID と対応させ、設計追加・変更時の影響範囲追跡を可能にする層である。

## 本カテゴリの位置付け

概要設計書は複数のカテゴリ（10〜60）にまたがって設計 ID を採番するため、全体を横断するトレーサビリティがないと、以下の問題が発生する。

- 要件 ID 単位の実装進捗が追えない（ある要件を実装完了と宣言できない）
- 要件変更時の影響範囲がわからない（どの設計項目が影響を受けるかが不明）
- 監査時に「すべての要件が設計に落ちていること」を証明できない

本カテゴリは、これらの問題を解消する単一の情報源として機能する。要件定義書側のトレーサビリティ（[../../03_要件定義/80_トレーサビリティ/](../../03_要件定義/80_トレーサビリティ/)）が「要件 ID と企画ゴール・構想設計 ADR の紐付け」を管理するのに対し、本カテゴリは「要件 ID と設計 ID の紐付け」を管理する。両者を連結することで、企画ゴール → 要件 → 設計 → 詳細設計 の全階層トレースが成立する。

## 格納予定ドキュメント

5 ファイルを格納する。

- `01_設計ID索引.md` — すべての設計 ID を一覧化する。ID / 章番号 / ファイル名 / 概要 / 対応要件 ID / ステータス（ドラフト / レビュー中 / 承認済み）の列で管理する。設計追加時はここに行追加する。
- `02_要件から設計へのマトリクス.md` — 要件 ID（FR / NFR / BR）を縦軸、設計 ID を横軸とする双方向マトリクス。1 要件が複数の設計項目に分解される場合、1 設計項目が複数要件を満たす場合の両方を表現する。未対応要件（設計されていない要件）と過剰設計（対応要件のない設計）を検出する役割を担う。
- `03_詳細設計引渡し一覧.md` — 概要設計で確定した方式のうち、詳細設計フェーズで具体化すべき項目（実装アルゴリズム / 個別クラス / 具体クエリ / 具体スクリプト）の引渡し一覧。概要設計として「ここまでを固定化した、続きは詳細設計で」という明示的なハンドオフを残す。
- `04_設計間依存マトリクス.md` — 設計 ID 同士の依存関係を双方向で管理する。例: ある API 方式（`DS-SW-EIF-*`）は認証方式（`DS-CF-AUTH-*`）・監査方式（`DS-CF-AUD-*`）・Saga 制御（`DS-CTRL-SAGA-*`）に依存するといった関係を採番する。設計変更時の影響範囲特定と循環依存検出に用いる。要件定義書側の [../../03_要件定義/80_トレーサビリティ/](../../03_要件定義/80_トレーサビリティ/) の要件間依存マトリクスと階層が異なる（本ファイルは設計項目間）。
- `05_構想設計ADR相関マトリクス.md` — 設計 ID と構想設計 ADR（[../../02_構想設計/](../../02_構想設計/)）の対応を管理する。設計項目がどの ADR の決定を前提としているかを列挙し、ADR 変更時の影響範囲特定、および ADR との不整合検出に用いる。構想設計と概要設計の境界を明示的に文書化する役割を担う。

## 執筆方針

- 本カテゴリの 5 ファイルは、他章の設計執筆と並行して継続的に更新する。他章で設計 ID を採番したら、その日のうちに `01_設計ID索引.md` に追記する運用とする。
- マトリクスは Markdown 表で記述する。行数が 300 行制限に達した場合は、カテゴリ別にファイル分割する。
- ステータス列は `draft` / `review` / `approved` / `deprecated` の 4 値で管理する。
- 要件変更時は、該当要件 ID を参照する全設計 ID を `02_要件から設計へのマトリクス.md` で検索し、影響範囲として PR 本文に列挙する運用を徹底する。

## 双方向整合チェック CI 仕様

本カテゴリの 5 マトリクスは人手更新を前提とするが、人手更新には必ず漏れが発生する。設計 ID を新規採番した PR で `01_設計ID索引.md` への追記を忘れる、要件 ID を変更した PR で `02_要件から設計へのマトリクス.md` の対応行を更新し忘れる、削除済み設計 ID への参照リンクが他章に残存する、といった不整合は数百件規模のマトリクスでは目視レビューでの検出が困難となる。Phase 0 稟議承認以降の運用では以下 3 種類の CI スクリプトを Argo Workflows + GitHub Actions の両方で起動し、不整合を機械検出する。

スクリプト本体は Phase 1a 着手時点で `scripts/traceability/` 配下に Python 3.12 で実装する。Phase 0 稟議時点では仕様のみを本節に固定し、稟議承認後の Phase 1a 第 1 週で実装する。仕様は CI スクリプトの実装よりも先に固定しておくことで、実装着手時に「何を検出すべきか」のレビューが終わっている状態を作る。

### スクリプト 1: 重複 ID 検出（`detect_duplicate_ids.py`）

設計 ID が異なるファイルで重複採番されると、要件 → 設計の対応関係が一意に確定できなくなる。本スクリプトは以下を検出する。

- **入力**: `docs/04_概要設計/` 配下の全 `.md` ファイルから `DS-<カテゴリ>-<サブ>-<番号>` 形式の ID を正規表現で抽出
- **検出条件**: 同一 ID が 2 ファイル以上に「採番宣言」（`### 設計項目 DS-XXX-NNN` 形式の見出しまたは `**設計項目 DS-XXX-NNN`の太字宣言）として出現
- **出力**: 重複 ID / 出現ファイル一覧 / 出現行番号 / 推奨アクション（旧採番の置換 or 採番取り消し）
- **終了コード**: 重複検出時に exit 1、未検出時に exit 0
- **CI 統合**: PR 作成時の必須チェック、main マージ条件

参照（`本表は DS-XXX-NNN を参照する` 形式）は重複としてカウントしない。採番宣言と参照を区別するため、抽出時は見出し記法と太字宣言の 2 パターンに限定する。

### スクリプト 2: 未参照要件検出（`detect_orphan_requirements.py`）

要件 ID 索引（[`03_要件定義/80_トレーサビリティ/01_要件ID索引.md`](../../03_要件定義/80_トレーサビリティ/01_要件ID索引.md)）に登録された要件 ID が、概要設計のどの設計 ID からも参照されていない状態は「設計漏れ」を意味する。本スクリプトは以下を検出する。

- **入力**: 要件 ID 一覧（要件 ID 索引から正規表現抽出）と本マトリクス（`02_要件から設計へのマトリクス.md`）の「対応設計 ID」列
- **検出条件**: 要件 ID 索引に存在するが、本マトリクスの「対応設計 ID」列が空または `TBD` のままの要件 ID
- **出力**: 未参照要件 ID / 要件タイトル / 想定対応カテゴリ / Phase 0 稟議に対する重要度（MUST → BLOCKER、SHOULD → WARN、COULD → INFO）
- **終了コード**: MUST 要件の未参照を 1 件以上検出した場合は exit 1、SHOULD 以下のみの場合は exit 0（warning 報告）
- **CI 統合**: PR 作成時の MUST 要件未参照ブロック、週次 main 集計で SHOULD / COULD の進捗レポート Slack 配信

意図的な `TBD` 状態（U-001 〜 004 など）は許容リスト `scripts/traceability/allowed_tbd.yaml` に登録し、検出対象から除外する。

### スクリプト 3: リンク切れ検出（`detect_broken_references.py`）

設計 ID 間の相互参照と、設計 ID から要件 ID への参照、設計 ID から構想設計 ADR への参照のいずれかでリンク切れが発生すると、影響範囲特定が機能しなくなる。本スクリプトは以下を検出する。

- **入力**: 概要設計全 `.md` ファイル内の `DS-XXX-NNN` / `FR-XXX-NNN` / `NFR-X-XXX-NNN` / `ADR-XXX-NNN` パターンの参照
- **検出条件**: 参照先 ID が要件 ID 索引・設計 ID 索引・ADR 索引のいずれにも存在しない、または該当 ID が deprecated ステータスでマークされている
- **出力**: 参照元ファイル / 行番号 / 参照 ID / 参照先存在状況（存在しない / deprecated / draft 段階）/ 推奨アクション
- **終了コード**: 存在しない ID 参照を 1 件以上検出した場合は exit 1、deprecated 参照のみは exit 0（warning 報告）
- **CI 統合**: PR 作成時の必須チェック

参照表記の揺れ（例：`DS-SW-EIF-001` と `DS-SW-EIF-1`）も検出対象とし、ID 命名規約 [`00_設計方針/03_設計ID体系とトレーサビリティ.md`](../00_設計方針/03_設計ID体系とトレーサビリティ.md) で確定した正規形以外を warning として報告する。

## CI 統合フロー

3 スクリプトは PR ライフサイクルと main ブランチ運用の 2 経路で起動する。

| トリガー | 起動スクリプト | 失敗時の挙動 | 通知先 |
|---|---|---|---|
| PR 作成 / push | 1（重複 ID）+ 2（未参照、MUST のみ）+ 3（リンク切れ）| GitHub Actions チェック失敗 → main マージブロック | PR コメントに自動投稿 |
| main マージ後 | 1 + 2（全優先度）+ 3 | Slack `#k1s0-traceability` に集計レポート | 起案者 + 概要設計チーム |
| 週次（毎週月曜 9:00 JST）| 2（全優先度）+ 3 | Slack `#k1s0-traceability` に SHOULD / COULD 進捗 | Product Council |
| Phase Gate 直前 | 1 + 2 + 3 + マニュアルレビュー | Phase Gate 議事録の必須提出物として整合性レポートを添付 | Product Council 全員 |

PR 作成時のブロックは「MUST 要件未参照」「重複 ID」「存在しない ID 参照」の 3 条件で発動する。SHOULD / COULD の未参照は warning に留め、PR マージは許可するが Slack で起案者に通知する。Phase Gate 直前の手動レビューでは、CI スクリプトの自動検出を出発点として、人間が「警告レベルでも本当に許容してよいか」を判定する 2 段階構造を取る。

## 不整合発生時の運用フロー

CI スクリプトが不整合を検出した場合の対応フローを以下で固定する。属人的判断を排し、誰が・どの優先度で・いつまでに修正するかを事前に決めておくことで、検出から修正までの遅延を排除する。

1. **検出から 1 営業日以内**: PR 作成者が CI 失敗を確認し、修正 PR を起票（不整合の原因が当該 PR にある場合は同一 PR で修正）
2. **検出から 3 営業日以内**: 修正 PR がレビューを通過し、main マージされる。レビュー責任者は概要設計チーム + 起案者
3. **検出から 5 営業日以内に未解決の場合**: Slack `#k1s0-traceability` に自動エスカレーション通知、概要設計チームリードが直接介入
4. **2 週間以内に未解決の場合**: Product Council の月次レビューで議題化、必要に応じて該当要件の優先度見直し（MUST → SHOULD 等）または該当設計 ID の draft → deprecated 移行を決議
5. **Phase Gate 直前未解決の場合**: Phase Gate 議事録に「未解消整合性違反 N 件」を必ず明記、Product Council が Go/NoGo を判定

不整合の優先度は CI スクリプトの終了コードで自動分類されるが、人間が「これは Phase Gate を遅延させるほどではない」と判断した場合の例外承認ルートは Product Council の正式議決で個別運用する。例外承認の履歴は Confluence `k1s0-traceability-exceptions` に永続保管し、四半期レビューで例外件数の傾向を点検する。

## ローカル実行手順

開発者が PR 作成前にローカルで整合性チェックを実行できるよう、以下のコマンドラインインタフェースを提供する。Phase 1a 着手時点で `Makefile` ターゲットとして実装する。

```bash
# 重複 ID 検出
make trace-check-duplicates

# 未参照要件検出（MUST のみ）
make trace-check-orphans

# 未参照要件検出（全優先度）
make trace-check-orphans-all

# リンク切れ検出
make trace-check-references

# 全チェック一括実行（CI と同じ条件）
make trace-check-all
```

実行には Python 3.12 と PyYAML / regex / rich の 3 ライブラリが必要である。`pyproject.toml` で固定化し、`uv sync` で再現可能な環境を構築する。

## 他カテゴリとの関係

- **上流**: [../../03_要件定義/80_トレーサビリティ/](../../03_要件定義/80_トレーサビリティ/) と連結することで、企画 → 要件 → 設計の全階層トレースが成立する。
- **横断**: 10〜60 の全カテゴリからの設計 ID を集約する。各章の執筆者は本カテゴリの更新を執筆フローの一部とする。
- **下流**: 詳細設計フェーズのトレーサビリティ（未着手）が、本カテゴリを起点として詳細設計 ID を連結する。
