# 02. 要件から設計へのマトリクス

本ファイルは要件定義書で採番されたすべての要件 ID（FR / NFR / BR / OR / DX / BC / MIG / P / C / U / RISK）に対して、その要件を設計レベルで充足する概要設計書の設計 ID を紐付ける双方向トレーサビリティの台帳である。要件は「何を約束するか」を宣言し、設計は「どう実装方式として実現するか」を宣言する。本マトリクスは両者の対応関係を 1 行ごとに固定化し、要件の充足率と設計の根拠を同時に可視化する。

## 本ファイルの位置付け

要件定義書は数百の要件 ID を FR / NFR / BR / OR / DX / BC / MIG / P / C / U / RISK の 11 系統で採番した。これらは概要設計書の 9 カテゴリ（SYS / SW / CF / CTRL / NFR / OPS / MIG / DEVX / BUS）の設計 ID に落とし込む必要があるが、対応関係が文書化されていないと以下の事故が起こる。

第一に、要件が設計漏れのまま Phase 0 稟議を通過してしまう。要件 ID 単位での充足率チェックがなければ、全要件が設計カバーされているかの機械的判定ができない。第二に、要件変更時の影響範囲特定が属人化する。ある要件が変更された際、影響する設計 ID を人間の記憶で辿ることになり、見落としが発生する。第三に、監査で「この要件がどの設計で実現されているか」を問われた際、即応できない。

本マトリクスはこれら 3 つの事故を防ぐため、要件 ID を一次キーとして対応する設計 ID を列挙する。逆方向の設計 ID → 要件 ID は各設計 ID の本文末尾の「対応要件一覧」節で管理され、両方向の整合は CI の機械チェックで検証される。

本章は索引層のため設計 ID を採番しない。

## マトリクスの構造

マトリクスは以下の 7 列で構成する。各列は 1 要件の充足状況を 1 行で読めるように設計されている。

- **要件 ID**: 要件定義書で採番された一意識別子。
- **要件タイトル**: 要件定義書本文の見出しをそのまま転記。
- **対応設計 ID**: 本要件を充足する設計 ID（1:n を許容）。
- **対応設計ファイル**: 対応設計 ID が格納されている相対パス。
- **カバー状況**: `完全` / `部分` / `TBD` の 3 値。`部分` の場合は残存範囲を備考に記載。
- **担当**: 設計作成の主担当チーム（tier1 API / Infra / Sec / Ops / DevEx / Bus の 6 種）。
- **優先度**: 要件定義書側の優先度（MUST / SHOULD / COULD / WON'T）をそのまま転記。

カバー状況の判定基準は厳密に固定する。`完全` は要件本文の受け入れ基準すべてを設計 ID がカバーしている状態、`部分` は主要受け入れ基準はカバーするが詳細設計に委ねる要件箇所が残る状態、`TBD` は Phase 0 時点で設計側が未採番または未決断の状態を指す。

## BR-* 系（業務要件 — 横串リファレンス）

業務要件 BR-PLATUSE / BR-PLATOPS / BR-PLATGOV の 18 件は、要件 ID 索引で「業務要件は 10_業務要件/ で散文中に括弧書き参照する方式を採用」と定義されており、要件 ID 索引のカウント 406 件には含まれない横串要件として運用される。本概要設計マトリクスでも BR は「FR / NFR / OR / DX / BC を業務観点から貫通する横串リファレンス」として個別節を設ける。設計 ID 単位の対応は要件側 FR / NFR を経由して間接的に確定し、業務要件の影響範囲評価（要件削除時の業務影響判定 / 新規機能の業務根拠検証）に用いる。

本節の対応関係は要件側 [`03_要件定義/80_トレーサビリティ/05_業務機能マトリクス.md`](../../03_要件定義/80_トレーサビリティ/05_業務機能マトリクス.md) の BR × FR / NFR マッピングと整合させる。要件側で BR が追加・改訂された場合、同一 PR で本節も更新する CI 検証を組み込む（[80_トレーサビリティ/README.md](README.md) の双方向整合チェック対象）。

## 運用ルール

本マトリクスの継続運用は以下の 5 ルールで固定化する。要件変更・設計変更・Phase 遷移の 3 局面で本ファイルが整合を保つことを保証する。

- **要件追加時**: 要件定義書側で新規 ID が採番された場合、同じ PR で本マトリクスに行を追加する。対応設計 ID が未確定の場合は `TBD` ステータスで仮登録し、Issue として残留管理する。
- **設計追加時**: 概要設計側で新規設計 ID が採番された場合、その設計が対応する要件 ID を本マトリクスの該当行「対応設計 ID」列に追記する。追記漏れは CI の双方向整合チェックで検出する。
- **要件削除時**: 要件 ID の廃止時、本マトリクスの該当行は削除せず、要件タイトル列に「【廃止 YYYY-MM-DD】」を前置する。これは古い PR からの参照を壊さないため。
- **Phase 遷移レビュー**: Phase 1a / 1b / 1c / 2 への遷移時、カバー状況列を実績値に書き換える。Phase 切替承認の前提として `TBD` が解消していることを確認する。
- **四半期レビュー**: Product Council 四半期レビューでカバレッジサマリを再集計し、`完全` 率低下が発生している場合は是正計画を策定する。

### 4 種別分解後の未カバー実数

初版 0.3 の「4 件ゴースト一括」は、実質的に以下に整理される。

- 設計側誤読で撤回（種別 1）: 1 件 → カバー済みに再集計（FR-T1-DECISION-003 は元から完全カバー）
- 要件定義側既処理（種別 2）: 1 件 → カバレッジ分母から除外（欠番）
- 要件定義側未確定（種別 3）: 1 件 → **Phase 0 稟議時点での実質未カバー 1 件**（NFR-G-RES 系の連番確定待ち）
- 要件定義側表記揺れ（種別 4）: 1 件 → 設計側はカバー済み、申し送りは要件定義プレフィックス修正のみ

したがって概要設計カバレッジの実質値は **388 件中 387 件（99.7%）** となる。分母は当初 389 から欠番 NFR-C-MGMT-004 を除外した 388、分子は種別 3 の NFR-G-RES-00x のみ未カバーとカウントした 387 である。本ファイル上部の初版サマリ表（96.7% / 99.0%）は本改訂で 99.7% に更新された値が最新となる。
