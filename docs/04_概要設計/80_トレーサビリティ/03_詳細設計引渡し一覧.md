# 03. 詳細設計引渡し一覧

本ファイルは概要設計フェーズ完了後に詳細設計フェーズへ引き渡す項目を、設計 ID 単位で漏れなく列挙する引渡し台帳である。概要設計で確定した方式と、詳細設計で決めるべき実装細部を明示的に分離し、詳細設計チームが着手時に「何が決まっていて、何を決めるべきか」を即座に把握できる状態を保つ。

## 本ファイルの位置付け

概要設計と詳細設計の境界は、組織とチームが入れ替わる境界である。概要設計は tier1 API チーム + Infra チーム + Sec チームを中心にプラットフォーム全体を俯瞰する少数精鋭で実施するが、詳細設計は API 別・コンポーネント別に分担する多数のチームに引き渡される。引渡しが曖昧だと、詳細設計チームが概要設計の前提を再掘削する無駄が発生し、Phase 1a 着手が遅延する。

本ファイルは以下の 4 要素を設計 ID ごとに固定化し、引渡し時の情報欠損を防ぐ。

第一に、**確定事項**の明示である。概要設計で決定済みで、詳細設計で変更してはならない仕様境界。これを守らないと、詳細設計が概要設計の前提を崩し、要件充足が危うくなる。

第二に、**未決事項**の明示である。概要設計では意図的に未決とし、詳細設計で決める論点。これを明示しないと、詳細設計チームが「未決なのか、それとも読み落としたのか」を判断できない。

第三に、**関連インタフェース定義書**の特定である。proto ファイル・OpenAPI 定義・スキーマ定義など、詳細設計が参照・拡張する契約資料の格納位置。

第四に、**前提条件**の明示である。詳細設計着手前に完了している必要がある他設計 ID（環境構築・共通機能など）。

本章は索引層のため設計 ID を採番しない。

## 引渡し優先順位

詳細設計は Phase 1a / 1b / 1c / 2+ の順に段階的に進行するため、引渡しも優先順位別に 3 段階で管理する。優先順位は Phase との対応関係で機械的に決まる。

- **P1（Phase 1a MVP-0 必須）**: MVP-0 実装着手に必須の設計 ID。2026 年 6 月までに詳細設計完了が必要。
- **P2（Phase 1b/1c 必須）**: MVP-1a/1b 実装に必要な設計 ID。2026 年 12 月までに詳細設計完了が必要。
- **P3（Phase 2+ 基盤拡張）**: 基盤拡張 GA に向けて詳細設計する設計 ID。2027 年度以降。

優先順位が P1 であっても、概要設計が `TBD` ステータスの場合は引渡し不可とし、Phase 0 稟議後の追加確定を待つ。

## 未決事項の分類

未決事項は詳細設計で決めるべき性質別に 5 分類する。詳細設計チームは分類別に担当者（tier1 API エンジニア / DB エンジニア / SRE / ML エンジニア等）を配置する。

- **インタフェース内部実装詳細**: proto で契約は固定済みだが、内部実装のデータ構造・キャッシュ戦略・バッチ分割などが未決。
- **データ構造詳細**: ER 最上位は固定済みだが、インデックス設計・パーティション戦略・マテビュー構成が未決。
- **アルゴリズム選定**: Saga の補償順序、ワークフロー振り分け閾値、リトライバックオフ関数など計算ロジック選定が未決。
- **境界処理**: 異常系の境界処理（部分失敗時の整合性回復、タイムアウト後のリトライ判定）が未決。
- **負荷特性**: Phase 1a 負荷試験の結果を踏まえて詳細を決める領域（インスタンス数・バッチサイズ等）。

## 引渡し形式

詳細設計チームへの引渡しは以下の 4 点セットで行う。どの 1 点が欠けても引渡し不完全と判定する。

- **設計書本文**: 概要設計書の該当章（DS-SW-EIF-001 であれば [20_ソフトウェア方式設計/02_外部インタフェース方式設計/06_API別詳細方式/01_Service_Invoke_API方式.md](../20_ソフトウェア方式設計/02_外部インタフェース方式設計/06_API別詳細方式/01_Service_Invoke_API方式.md)）。
- **構想設計 ADR**: 関連 ADR 番号を列挙。詳細設計チームは ADR の却下代替案も参照し、再議論の無駄を省く。
- **プロトタイプコード**: 概要設計段階で作成したプロトタイプがあればリポジトリパスを明示。Phase 0 時点では tier1 INVOKE/STATE の Go+Rust プロトタイプが存在する。
- **機能要件 IDL**: [../../03_要件定義/20_機能要件/](../../03_要件定義/20_機能要件/) に格納された proto ファイル・OpenAPI 定義。詳細設計はこれを拡張するのみで、破壊的変更は禁止。

## 引渡し確認チェックリスト

引渡し会議で以下 10 項目を確認し、全項目 Yes で引渡し完了とする。Yes にならない項目は引渡し保留として課題管理する。

- 設計 ID とタイトルが概要設計書と一致している。
- 確定事項の列挙に過不足がない。
- 未決事項が 5 分類のいずれかに該当している。
- 関連 IF 定義書へのリンクが有効である。
- 前提条件の設計 ID がすべて確定済みである。
- 対応要件 ID が 02_要件から設計へのマトリクスと一致している。
- 影響する構想設計 ADR が [05_構想設計ADR相関マトリクス.md](05_構想設計ADR相関マトリクス.md) と一致している。
- 下流依存（本設計 ID を前提とする他設計 ID）が [04_設計間依存マトリクス.md](04_設計間依存マトリクス.md) と一致している。
- 詳細設計担当チームが決定している。
- 詳細設計完了予定日が Phase スケジュールと整合している。

## 運用ルール

本一覧の継続運用は以下の 4 ルールで固定化する。引渡し漏れ・再整理の無限ループを防ぐ。

- **引渡し時追記**: 引渡し会議で合意した時点で本ファイルに `引渡し済` フラグを設定。詳細設計開始後の追加未決事項は本ファイルには遡及せず、詳細設計書側で管理する。
- **差戻し時**: 詳細設計が概要設計の前提を覆す必要がある場合、本ファイルではなく概要設計書側を改訂し、改訂後に再度引渡しを行う。
- **Phase 切替時**: Phase 1a → 1b、1b → 1c、1c → 2 の切替時に全引渡し項目を棚卸しし、未完了項目を次 Phase のブロッカーとして Product Council に提示する。
- **詳細設計完了検収**: 詳細設計書が完了した時点で本ファイルの該当行に `詳細設計完了` フラグを設定。実装フェーズへの次段引渡しは別途詳細設計書側で管理する。
