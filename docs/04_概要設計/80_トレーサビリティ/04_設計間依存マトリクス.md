# 04. 設計間依存マトリクス

本ファイルは概要設計書の 9 カテゴリ（SYS / SW / CF / CTRL / NFR / OPS / MIG / DEVX / BUS）間、および個別設計 ID 間の依存関係を可視化する台帳である。ある設計 ID が他の設計 ID の前提に立つ場合、参照関係を機械可読に保つことで、設計変更時の影響範囲特定・循環依存の検出・レビュー順序の決定を自動化できる状態を目指す。

## 本ファイルの位置付け

設計項目が数百に達する規模では、章内の局所的な依存だけでなく、章を跨いだ依存（例: CF カテゴリの認証方式が SW カテゴリの全 API 仕様の前提に立つ、SYS カテゴリのハードウェア方式が NFR カテゴリの可用性設計の前提に立つ）が多数発生する。この依存が可視化されていないと、ある設計 ID の改訂が下流にどこまで影響するかが属人的な判断に委ねられる。

章跨ぎ依存の典型例として、SPIFFE SVID 採用（DS-CF-AUTHZ-001）が変更されると、tier1 全 API のセキュリティ仕様（DS-SW-EIF-* 多数）、Istio Ambient 設定（DS-SYS-PROC-003）、ゼロトラスト方式（DS-NFR-SEC-001）の 3 層に波及する。もう一つの例として、CNPG PostgreSQL 採用（DS-SYS-DB-001）が変更されると、State API 仕様（DS-SW-EIF-012）、DB 最上位設計（DS-SW-DB-001）、可用性方式（DS-NFR-AVL-001）、バックアップ運用（DS-NFR-AVL-007）の 4 層に波及する。

これらの波及を手動で辿ると見落としが発生するため、本ファイルで依存関係を章跨ぎの 9×9 マトリクスとして固定化し、さらに重要設計 ID 間の依存を個別に抽出する。本章は索引層のため設計 ID を採番しない。

## 依存の種別

設計 ID 間の依存は 4 種別に分類する。種別を明示することで、影響範囲の大きさと対処の緊急度を差別化できる。

- **仕様依存**: 下流設計が上流設計の定義を参照する依存。上流が変わると下流の記述が変わる必要がある。例: Service Invoke 仕様（DS-SW-EIF-001）が認証方式（DS-CF-AUTH-001）の JWT 検証方式を参照。
- **実装依存**: 下流設計が上流設計を前提として成立する依存。上流が利用不可なら下流は動作不可。例: 全 NFR 方式（DS-NFR-*）がハードウェア方式（DS-SYS-HW-*）の上に成立。
- **運用依存**: 下流設計の運用フローが上流設計の運用に連動する依存。例: インシデント対応（DS-OPS-INC-001）が監視観測性（DS-NFR-OBS-*）のアラート設定を前提に運用する。
- **データ依存**: 下流設計が上流設計のデータモデルを共有する依存。例: 監査ログ（DS-CF-AUD-003）が Log スキーマ（DS-CF-LOG-002）のフィールド定義を共有する。

種別の優先度は 仕様依存 ≈ 実装依存 > データ依存 > 運用依存 の順である。仕様依存と実装依存は変更が下流の書き直しを要求するため優先度が高く、運用依存は変更後の運用訓練で吸収可能な場合が多い。

## カテゴリ間依存マトリクス（9×9）

9 カテゴリ間の依存を俯瞰するマトリクスである。行がソース（依存元）、列がターゲット（依存先）を表し、セル内容は「行カテゴリの設計が列カテゴリの設計に与える影響の種別と強さ」を示す。表を左上から右下へ読むことで、基盤層（SYS）から事業運用層（BUS）へ依存がカスケードする様子が把握できる。

| ソース →<br>ターゲット ↓ | SYS | SW | CF | CTRL | NFR | OPS | MIG | DEVX | BUS |
|---|---|---|---|---|---|---|---|---|---|
| SYS | ‐ | 実装（強） | 実装（強） | 実装（中） | 実装（強） | 運用（中） | 実装（中） | 実装（弱） | ‐ |
| SW | ‐ | ‐ | 仕様（強） | 仕様（強） | 仕様（強） | 運用（中） | データ（中） | 運用（中） | ‐ |
| CF | ‐ | 仕様（強） | ‐ | 仕様（中） | 仕様（強） | 運用（中） | ‐ | 仕様（中） | データ（弱） |
| CTRL | ‐ | 仕様（強） | 仕様（弱） | ‐ | 仕様（中） | 運用（中） | ‐ | 仕様（弱） | ‐ |
| NFR | 実装（弱） | 仕様（中） | 仕様（中） | 仕様（弱） | ‐ | 運用（強） | 運用（中） | 運用（中） | 運用（中） |
| OPS | ‐ | 運用（弱） | 運用（弱） | 運用（弱） | 運用（強） | ‐ | 運用（中） | 運用（強） | 運用（中） |
| MIG | 実装（弱） | データ（中） | ‐ | ‐ | 運用（中） | 運用（強） | ‐ | ‐ | ‐ |
| DEVX | ‐ | 仕様（弱） | 仕様（弱） | ‐ | 運用（弱） | 運用（中） | ‐ | ‐ | 運用（弱） |
| BUS | ‐ | ‐ | ‐ | ‐ | 仕様（弱） | 運用（中） | ‐ | 運用（弱） | ‐ |

マトリクスの読み方: たとえば「SYS → SW」セルは `実装（強）` であり、ハードウェア方式（kubeadm 構成・Longhorn・ノードスペック）がソフトウェア方式（コンポーネント配置・インタフェース性能）に対して強い実装依存を与えることを示す。kubeadm クラスタが動かない状態では tier1 コンポーネントは実装不可能となる。

逆に「NFR → SYS」セルが `実装（弱）` なのは、NFR 方式がハードウェア方式に要件を差し戻す逆方向依存が弱く存在するからである（例: NFR 性能要件が HW スペック見直しを誘発）。このような逆方向依存はマトリクス上で `弱` と記録する。

空欄（‐）は意味のある依存が存在しないことを示す。マトリクスを対称に埋めることは目的ではなく、実存する依存のみを記録する。

## カテゴリ依存の主要経路

9×9 マトリクスの中で特に重要な依存経路を 7 本列挙する。これらはアーキテクチャ変更時に必ず辿る経路であり、変更レビューのチェックポイントとなる。

### 経路 1: SYS → SW → CF → CTRL → NFR

プラットフォームの主軸となる経路。ハードウェアの上にソフトウェアが載り、ソフトウェアに共通機能が組み込まれ、共通機能の上で制御方式が走り、最後に非機能品質が測られる流れ。SYS の変更はこの経路全体を下流へ波及する。

### 経路 2: SYS → NFR

NFR のうち可用性・性能・キャパシティは SYS のハードウェア方式と DB 方式に直接依存する。たとえば State API p99 < 50ms（DS-NFR-PERF-005）は CNPG のインスタンスサイズと Longhorn の I/O 特性に依存する。

### 経路 3: CF → SW → CTRL

共通機能（認証・ログ・観測性）が先に確定し、それを前提として SW 方式（API 仕様）が書かれ、さらに制御方式（Saga・冪等）が書かれる。CF が変わると SW と CTRL 両方に波及する。

### 経路 4: NFR → OPS → DEVX

非機能（可用性・SLO）の監視観測性が先にあり、それを前提としてインシデント対応（OPS）の Runbook が書かれ、DevEx の CI/CD が運用と統合される。NFR の SLO 変更は OPS と DEVX に波及する。

### 経路 5: MIG → SW → OPS

移行方式は既存 SW との共存を要求し、移行中の運用フローが OPS に組み込まれる。MIG の変更は SW と OPS の両方に波及する。

### 経路 6: CF → BUS（データ依存）

監査ログ・メトリクスデータは BUS の分析レポーティングと課金メータリングの入力となる。CF のスキーマ変更は BUS のレポート様式変更を要求する。

### 経路 7: OPS → BUS（運用依存）

インシデント対応・EOL・撤退戦略は BUS のガバナンスとサプライチェーンに運用フローを連鎖させる。OPS の変更は BUS の運用合意事項を更新する。

## 循環依存検出

循環依存は設計の品質欠陥の典型症状であり、検出と是正が必須である。本節は循環依存の検出ルールと、検出時の是正手順を定める。

循環依存の検出は CI ジョブで自動化する。各設計 ID の本文末尾「依存一覧」節から依存先 ID をパースし、有向グラフとして構築する。深さ優先探索でサイクルを検出した場合、PR をブロックする。

検出対象とする依存種別は 仕様依存 と 実装依存 の 2 種のみとする。データ依存はスキーマの相互参照として意図的に発生することがあり、運用依存は運用フロー上の相互連携として許容される。

検出された循環依存は以下のいずれかで是正する。第一に、依存方向の見直し（どちらか一方の依存を逆向きに書き直す）。第二に、中間層の抽出（両方が依存する共通抽象を新設計 ID として切り出す）。第三に、片方の設計 ID の粒度見直し（大きすぎる設計 ID を複数に分割し、循環を解消）。

リリース時点で検出されている循環依存は存在しない（本マトリクス作成時の棚卸し結果）。新規 PR で循環依存が発生した場合は、Product Council に提示して是正方針を合意する。

## 影響範囲分析手順

設計 ID 変更時の下流影響特定は、本マトリクスを起点に以下の 5 ステップで実施する。これはチェンジコントロール委員会の標準手順であり、属人的な判断を排除する。

- **ステップ 1**: 変更対象の設計 ID を特定する。
- **ステップ 2**: 本マトリクスの「依存元 ID」列で対象 ID を検索し、依存先 ID を列挙する。これが直接の下流影響。
- **ステップ 3**: 依存先 ID それぞれを再度「依存元 ID」として再検索し、間接の下流影響を列挙する。これを依存先が尽きるまで再帰実行（深さ 3 程度で収束）。
- **ステップ 4**: 依存種別ごとに影響度を評価する。仕様依存・実装依存は設計書本文の書き直しが必要、データ依存はスキーマ整合性確認、運用依存は Runbook 更新のみで完結することが多い。
- **ステップ 5**: 影響範囲を PR に添付し、レビュー指名を下流設計 ID の担当チームに広げる。

このプロセスをサポートするため、依存関係は本ファイルの手動台帳と、各設計 ID の本文末尾「依存一覧」節の自動抽出の 2 系統で二重管理する。二系統の差分は CI で検出される。

## 変更時のレビュー順序

複数の設計 ID を同時に変更する PR では、依存関係を辿って上流から下流の順にレビューする。上流（依存元）が不承認なら下流（依存先）のレビューは意味を持たないため、下流優先のレビューは時間の浪費となる。

本マトリクスを使ってトポロジカルソートを行い、レビュー順序を PR 作成時に自動生成する。自動生成は CI の前処理で実施し、PR 本文に「推奨レビュー順序」としてコメントされる。レビュアーはこの順序に従って承認投票する。

## 運用ルール

本マトリクスの継続運用は以下の 5 ルールで固定化する。

- **依存追記タイミング**: 設計 ID の本文に「依存一覧」節が追加・更新された時点で、同じ PR で本マトリクスに該当行を追加する。
- **依存削除タイミング**: 上流の設計 ID が廃番になった場合、下流側の依存行を削除する前に、下流の設計 ID 側の文脈を修正する。依存行削除のみの PR は無意味。
- **四半期レビュー**: Product Council 四半期レビューで本マトリクスと各設計 ID 本文の整合を確認する。差分が 10 件を超える場合は是正 PR をブロッカーとして扱う。
- **循環依存ゼロ**: 循環依存は仕様依存と実装依存に限り、0 件を常時維持する。検出時は PR マージをブロックする。
- **段階切替時**: 採用初期 → 採用後の運用拡大時 → 採用側のマルチクラスタ移行時 の切替時、全依存を棚卸ししてマトリクスを確定する。
