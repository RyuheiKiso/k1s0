# 01. 設計 ID 索引

本ファイルは概要設計書で採番したすべての設計 ID を一元管理する索引である。9 カテゴリ（SYS / SW / CF / CTRL / NFR / OPS / MIG / DEVX / BUS）に分散して採番した設計項目を、ID・タイトル・格納ファイル・確定フェーズの 4 軸で一覧化し、章を跨いだ参照と変更時の影響範囲特定の基点として機能させる。

## 本ファイルの位置付け

概要設計書は 12 カテゴリ（00〜90）に約 85 のファイルが分散し、そこで採番される設計 ID は数百に達する見込みである。章執筆者が個別に採番する場合、同一カテゴリ内で通番が重複する、逆に欠番が発生する、あるいは章統合時に ID が散逸するといった事故が起こりやすい。本索引はこれらの事故を検出するための唯一の基準点として機能する。

索引は設計 ID を保持するだけではなく、「どの設計 ID がどのフェーズで確定するか」「どのファイルに格納されているか」「対応する要件 ID は何か」という 3 属性を同一行で読める形にまとめる。これにより、稟議審議で Phase 0 の確定範囲を問われた場合、フェーズ列でフィルタすれば回答できる状態を保つ。

本章は [00_設計方針/03_設計ID体系とトレーサビリティ.md](../00_設計方針/03_設計ID体系とトレーサビリティ.md) で定めた命名規則の実装帳票であり、命名規則そのものは本ファイルで重複定義しない。索引層のため本章自体は設計 ID を採番しない。

## 採番規則の要約

採番規則の詳細は [00_設計方針/03_設計ID体系とトレーサビリティ.md](../00_設計方針/03_設計ID体系とトレーサビリティ.md) に定義されているが、索引を読む前提として要点のみ再掲する。

設計 ID は `DS-<カテゴリ>-<サブ>-<通番>` の書式で、カテゴリは概要設計章（10〜75）に 1:1 対応する。サブはカテゴリ内トピック識別子、通番は同一カテゴリ + サブ内で連続する 3 桁番号。欠番は廃番記録に残る場合のみ許容される。

Phase 属性は `Phase 0`（稟議承認時）/ `Phase 1a`（MVP-0）/ `Phase 1b`（MVP-1a）/ `Phase 1c`（MVP-1b）/ `Phase 2+`（基盤拡張）の 5 段階。本索引では全 ID に対して Phase 属性列を保持し、フェーズごとの確定範囲を一覧できる。

## 索引の機械生成運用

本索引は手動編集を主、自動抽出を副とする二段構えで維持する。これは、手動編集で「どの ID がどのフェーズで何を達成するか」の文脈を保持しつつ、自動抽出で欠番・重複・参照切れを検出するためである。

自動抽出側は CI ジョブで各章のマークダウンファイルから `DS-XXX-YYY-NNN` 形式の ID をパースし、以下の 3 系統の違反を検出する。

- **重複検出**: 同一 ID が複数ファイルで採番されている場合、ビルドエラーとして PR をブロックする。ブロック基準は章跨ぎ重複であり、同一ファイル内の参照重複は対象外とする。
- **欠番検出**: 同一カテゴリ + サブ内で通番が連続していない場合、警告を出す。廃番一覧（[00_設計方針/03_設計ID体系とトレーサビリティ.md](../00_設計方針/03_設計ID体系とトレーサビリティ.md) の廃番節）に記録された ID は欠番扱いとせず、それ以外を真の欠番とする。
- **参照切れ検出**: 他章から参照されている ID が定義側に存在しない場合、エラーとする。定義側削除と参照側残置は最も見落とされやすいため、PR マージブロックの対象とする。

手動編集側は章 PR マージ時に本ファイルへの追記を必須とし、抽出結果とのずれを週次で棚卸しする。差分が発生した場合は Product Council に提示して解消方針を合意する。

## 運用ルール

本索引の継続運用は以下の 5 ルールで固定化する。章執筆者・レビュアー・CI 担当の 3 ロールが参照する基準として機能させる。

- **追記タイミング**: 章 PR に設計 ID の新規採番が含まれる場合、同じ PR 内で本ファイルに追記する。PR の他章と本ファイルを分離することは禁止する（分離すると一時的に索引が不整合になる）。
- **削除タイミング**: 設計 ID の廃番は本ファイルから該当行を削除せず、フェーズ列を「廃番」に書き換える。削除履歴は Git の commit log で追跡可能な状態を保持する。
- **分類変更**: カテゴリまたはサブ分類を変更する場合、元 ID を廃番として残し新規 ID を採番する。理由は古い PR コメント・Slack 履歴からの参照を壊さないため。
- **四半期レビュー**: Product Council 四半期レビューで本索引と各章の整合を確認する。ずれが 5 件を超える場合は是正 PR をブロッカーとして扱う。
- **Phase 遷移レビュー**: Phase 1a / 1b / 1c / 2 への遷移時、全 ID の確定フェーズ列を実績値に書き換える。予定値のまま遷移することは禁止する。
