# 05. テスト戦略方式

本ファイルは k1s0 の自動テスト戦略を定める。対象は Test Pyramid 方針（UT 70% / 結合 20% / E2E 10%）、単体テスト、契約テスト（pact）、結合テスト（testcontainers）、E2E テスト（Playwright / k6）、カオステスト（LitmusChaos、採用後の運用拡大時、ADR-TEST-004）、セキュリティテスト（OWASP ZAP / Trivy / Grype）、CI 実行時間予算の 8 領域である。

## 本ファイルの位置付け

プラットフォーム製品では「tier1 の不具合が全 tier2 / tier3 へ波及する」ため、品質保証の経済的インパクトが大きい。自動テストが不足すると、本番障害の検出を運用現場に委ねることになり、MTTR と CFR の両方が劣化する。一方で過剰なテスト投資は CI 時間を長期化させ、Lead Time 1h の目標を壊す。本設計はこのバランスを Test Pyramid の原則で管理する。

本設計は構想設計 ADR-TEST-001（Test Pyramid + testcontainers 採用）、ADR-TIER1-002（Protobuf gRPC 契約）を前提として、PR 5 分 / 回帰 30 分の CI 時間予算に適合するテスト戦略を設計する。対応要件は DX-TEST-001〜008、NFR-E-RSK-002（セキュリティテスト / ペネトレーションテスト）、DX-MET-005（Change Failure Rate 5%）、NFR-A-REC-002（MTTR を支える Runbook 充足率）である。

## Test Pyramid の採用

Test Pyramid は Mike Cohn が提唱した階層化テスト戦略で、下層ほど数を多く・速く、上層ほど数を少なく・遅いテストを配置する、という原則である。k1s0 は以下の比率で構成する。

| 層 | 比率 | 数量目安（1 サービス） | 実行時間 | ツール |
| --- | --- | --- | --- | --- |
| UT（単体） | 70% | 300〜500 テスト | < 30 秒 | cargo test / go test / xUnit / Jest |
| 契約テスト | 5% | 20〜30 契約 | < 30 秒 | pact-go / pact-rust |
| 結合 | 20% | 80〜100 テスト | < 3 分 | testcontainers |
| E2E | 5% | 20〜30 シナリオ | < 10 分 | Playwright / k6 |

比率は「単一サービスあたりのテスト数」を基準に算出する。tier1 は 11 API に対して合計 UT 3,500 件・結合 1,000 件・E2E 150 件を目安とする。この比率を逸脱（例: E2E が 20% を超える）すると CI 時間が膨張するため、CI 統計で比率を継続監視する。

## 単体テスト（UT）

単体テストは言語ネイティブのテストフレームワークで記述する。Rust は `cargo test`（モック化は `mockall`）、Go は `go test -race`（モック化は `gomock`）、C# は `xUnit`（モック化は `Moq`）、TypeScript は `Jest`（モック化は `@golevelup/ts-jest`）を使う。

カバレッジ目標は 80%（行カバレッジ、ブランチカバレッジは 70%）とする。目標未達で CI を失敗させる硬性基準を採用し、カバレッジ低下のまま PR がマージされない設計とする。カバレッジは `cargo tarpaulin`（Rust）、`go test -cover`（Go）、`coverlet`（C#）、`jest --coverage`（TypeScript）で計測する。

単体テストの範囲は「外部依存のない純粋ロジック」に限定する。DB アクセス・HTTP 呼び出し・ファイル IO を含むコードは結合テスト層に回し、UT ではモック化する。この区分を徹底することで UT の実行時間を 30 秒以内に保てる。30 秒を超える単体テストは設計の兆候（外部依存のモック化不足、テストケースの過剰）として扱い、リファクタの対象とする。

## 契約テスト（Consumer-Driven Contract）

契約テストは「A サービスが B サービスに期待する入出力」を契約として記述し、両者の互換性を CI で検証する仕組みである。tier1 の複数マイクロサービス（Go 側ファサード / Rust 側自作領域）と tier2 クライアントの間で、Protobuf gRPC 定義とは別層の「業務意味的な期待」を担保する。

実装は pact-go（Consumer 側）と pact-rust（Provider 側）を使う。Consumer 側（tier2）が「私は State.Get に対してこのレスポンスを期待する」と契約を書き、pact ブローカーに登録する。Provider 側（tier1 Go ファサード）がブローカーから契約を取得し、自分の実装がその契約を満たすかを CI で検証する。

契約が破綻した場合、`buf breaking` と同様に CI が失敗し、互換性を壊したマージが阻止される。これにより「tier1 を変更した瞬間 tier2 の別チームが壊れる」事態を構造的に防ぐ。契約ブローカーは pact-broker 相当の自己ホスト版（リリース時点 で構築、単 Pod）で運用し、採用後の運用拡大時 で HA 化する。

## 結合テスト（testcontainers）

結合テストは「実際の依存（Valkey / Kafka / PostgreSQL / OpenBao）と組み合わせて動作するか」を検証する。モック化しないため、本番と同じバイナリ・同じ設定で検証でき、モックと本物の差異による「ローカルで動いたが本番で動かない」問題を根絶する。

testcontainers は各言語から Docker API を叩いて依存コンテナを動的起動・破棄するライブラリである。Rust は `testcontainers-rs`、Go は `testcontainers-go`、C# は `Testcontainers for .NET` を使う。テスト実行時に `TestContainer().WithImage("valkey/valkey:7.2").Start()` のようなコードで依存が立ち上がり、テスト完了で破棄される。

結合テストの範囲は以下 3 パターンとする。第 1 に「tier1 API × 単一依存」（例: State.Set → Valkey に書き込まれたか）、第 2 に「tier1 API × 複数依存」（例: PubSub.Publish → Kafka トピックに届き、かつ State.Set もされたか）、第 3 に「tier1 API × 横断機能」（例: 監査ログが確実に書き込まれたか、認可が正しく働いたか）、である。

結合テストの実行時間は 3 分以内を目標とする。並列実行（`cargo test -- --test-threads=8`、`go test -parallel=8`）とコンテナ起動のキャッシュ（Ryuk による共有セッション）で短縮する。3 分を超える場合は、テストケースの重複削減、不要な依存コンテナの削除、起動時間の短いイメージ（distroless 系）への切替で改善する。

## E2E テスト

E2E テストは「ユーザ視点で業務シナリオが成立するか」を検証する最上位テストである。UI を伴う tier3 Web アプリは Playwright で検証し、API のみの tier2 は k6 で負荷テストを兼ねた E2E 検証を行う。

Playwright は Chromium / Firefox / WebKit の 3 ブラウザで自動操作できる E2E ツールで、スクリーンショット比較・ネットワークモック・認証ハンドルを統合提供する。シナリオは「ログイン → 業務画面遷移 → データ入力 → 保存 → 確認」のような 5〜10 ステップを 1 シナリオとし、ステージング環境（リリース時点 から整備）に対して夜間バッチで実行する。失敗時はスクリーンショットと HAR ファイルを artifacts として保存する。

k6 は Go 製の負荷テスト / API テストツールで、JavaScript DSL で「100 RPS を 5 分間流し、p99 < 200ms を維持するか」などの性能要件を検証する。k1s0 では tier1 API の性能目標（State Get p99 10ms、PubSub Publish p99 50ms、tier1 API p99 500ms）を k6 で日次検証する。

E2E の実行時間は PR では実行しない（CI 時間予算超過を避けるため）。夜間バッチで全シナリオを実行し、朝会で結果を確認する運用とする。採用後の運用拡大時 では PR 時の差分シナリオのみ選択実行する「Smart Test Selection」を導入する計画である。

## カオステスト（採用後の運用拡大時）

LitmusChaos（ADR-TEST-004 で確定、構想設計 02_周辺OSS で Litmus 採用 / Chaos Mesh 次点）は CNCF Incubating の Kubernetes ネイティブなカオスエンジニアリングプラットフォームで、Pod 削除・ネットワーク遅延・ネットワーク分断・CPU 負荷などを意図的に発生させる。k1s0 の冗長性と自動復旧がシミュレーションに耐えるかを 採用後の運用拡大時 で検証する。

シナリオは以下 5 種類を標準とする。Pod Kill（tier1 の Pod をランダムに削除し、サービス継続を確認）、Network Latency（依存サービス間に 500ms の遅延を注入し、タイムアウト動作を確認）、Network Partition（Valkey / Kafka を一時切断し、フォールバック動作を確認）、CPU Stress（CPU 80% 占有して他 Pod の動作を確認）、Disk IO Stress（永続ボリュームに書き込み遅延を注入）、である。

カオス実行頻度は週次のステージング環境で 1 シナリオずつ順次実行する。実行結果は MTTR / Availability / Error Rate の 3 指標で評価し、SLO 破綻が発生したシナリオは設計見直しの起点とする。

## セキュリティテスト

セキュリティテストは SAST（静的解析）、SCA（依存スキャン）、DAST（動的スキャン）の 3 系統で構成する。

SAST は Semgrep / gosec（Go）/ clippy（Rust）で実装し、PR 時の CI で実行する。言語別の典型的脆弱性パターン（SQL Injection / XSS / 秘匿情報ハードコード）を検出する。

SCA は Trivy と Grype の 2 ツール並列実行で冗長化する。Trivy はイメージと FS をスキャンし、Grype は SBOM（Syft 生成）をスキャンする。2 ツール併用で脆弱性 DB の差分を吸収する。詳細は [01_CI_CD方式.md](01_CI_CD方式.md) で扱う。

DAST は OWASP ZAP を 採用後の運用拡大時 で導入する。ステージング環境に対して週次で実行し、Web アプリの脆弱性（XSS / CSRF / SSRF / 認証バイパス）を検出する。ZAP は Docker コンテナで実行し、Automation Framework モードで API 定義（OpenAPI）を食わせて自動探索する。

## CI 時間予算

テスト戦略全体の CI 時間予算は以下で固定する。

| テスト層 | PR 時 | main 時 | 夜間バッチ |
| --- | --- | --- | --- |
| UT | 30 秒 | 30 秒 | - |
| 契約テスト | 30 秒 | 30 秒 | - |
| 結合テスト | 3 分 | 3 分 | - |
| E2E（Playwright） | - | - | 30 分 |
| E2E（k6 性能） | - | - | 30 分 |
| カオス（採用後の運用拡大時） | - | - | 60 分（週次） |
| SAST / SCA | 1 分 | 1 分 | - |
| DAST（採用後の運用拡大時） | - | - | 60 分（週次） |

PR 時は UT + 契約 + 結合 + SAST/SCA の合計 5 分以内を目標とする。main 時は同じ構成 + Build 5 分 = 合計 10 分以内。夜間バッチは E2E + カオス + DAST で合計 2〜3 時間、朝会前（7 時）までに完了させる。回帰テスト全体（夜間バッチ相当を人手でトリガ）は 30 分以内を目標とする。

## 対応要件一覧

本ファイルは要件定義書 50_開発者体験 DX-TEST-001〜008（テスト戦略）に直接対応する。30_非機能要件 NFR-E-RSK-002（ペネトレーションテスト）、NFR-E-AV-001 / NFR-E-AV-002（コンテナイメージスキャン・SBOM）、NFR-E-WEB-001（OWASP Top 10 対策）、DX-MET-003（Lead Time）、DX-MET-005（Change Failure Rate）、NFR-A-CONT-001（tier1 対外 SLA 稼働率を支える品質ゲート）とも連動する。構想設計 ADR-TEST-001（Test Pyramid + testcontainers）、ADR-TIER1-002（Protobuf gRPC）を前提とする。tier1 の品質が全プラットフォームの品質を決める構造のため、本テスト戦略は リリース時点 時点から Test Pyramid 比率を守り、運用蓄積後で契約テスト層を充実させる計画である。
