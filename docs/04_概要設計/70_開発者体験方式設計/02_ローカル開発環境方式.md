# 02. ローカル開発環境方式

本ファイルは tier2 / tier3 開発者および tier1 開発者が自身のラップトップ上で k1s0 サービスを開発・テスト・デバッグするためのローカル開発環境の標準方式を定める。対象は雛形生成（`k1s0 new`）、Dev Container（VS Code / IntelliJ）、ローカル Kubernetes（kind / minikube）、Tilt による即時反映、tier1 API モック、OpenBao Dev Mode、ローカル観測基盤の 7 領域である。

## 本ファイルの位置付け

ローカル開発環境が劣悪だと、開発者は「動くのを確認するだけで半日」「本番と挙動が違って再現しない」状態に陥り、Golden Path 10 分ルールが空文化する。本番と同じ OSS・同じ設定・同じ依存を、軽量に開発者のマシン上で再現することが本設計の目標である。

本設計は構想設計 ADR-DEVX-001（Tilt + kind 採用）と ADR-TIER1-001（Dapr + ZEN Engine 構成）を前提として、起動から最初のコミットまで 5 分以内、ファイル変更から再反映まで 10 秒以内、tier1 API のモック切替まで 1 コマンド、の 3 つの時間予算を守る設計に落とす。対応要件は DX-LD-001〜007（ローカル開発環境）、DX-GP-001〜006（Golden Path）、DX-TEST-002（testcontainers 互換）である。

## 全体構造

ローカル開発環境は以下の層で構成する。第 1 層は Dev Container で、VS Code Remote Containers 規約に従い、開発者のマシン OS（Windows / macOS / Linux）に依存しない統一環境を提供する。第 2 層は kind または minikube で、Kubernetes 互換の軽量クラスタをラップトップ上に立ち上げる。第 3 層は Tilt で、ローカル Kubernetes 上の Pod をソースコード変更に応じてリアルタイム更新する。第 4 層は tier1 サービス（または tier1 API モック）で、実際のビジネスロジック開発対象である。

開発者が書くコードは第 4 層のみで、第 1〜3 層は雛形生成ツール `k1s0 new <template>` が自動生成する。雛形には Devcontainer 定義、Tiltfile、kind 構成、Helm Chart、Kubernetes マニフェスト、Backstage catalog-info.yaml が含まれる。詳細は [06_ゴールデンパス方式.md](06_ゴールデンパス方式.md) で扱う。

起動所要時間の目標は「git clone から動く状態まで 5 分以内」とする。内訳は Dev Container ビルド 2 分、kind クラスタ起動 1 分、Tilt 初回ビルド 1.5 分、tier1 スタブ起動 30 秒である。この予算は Phase 1b で計測し、Phase 1c で固定、Phase 2 で改善する。

## Dev Container の標準構成

Dev Container は [.devcontainer/devcontainer.json](https://containers.dev/) 規約で定義する。ベースイメージは `mcr.microsoft.com/devcontainers/base:ubuntu-22.04` を使い、以下のツールを追加インストールする。Rust toolchain（rustup + stable）、Go（公式バイナリ）、Node.js（nvm + LTS）、kubectl、helm、kind、tilt、dapr CLI、buf CLI、sqlc、grpcurl、trivy、である。

開発者が編集するソースコードはホスト側にマウントし、Dev Container の再ビルドで失われない。設定ファイル（`.vscode/settings.json`、`.devcontainer/devcontainer.json`）は Git 管理し、全開発者で共通化する。言語別の LSP 設定（rust-analyzer、gopls、typescript-language-server）も devcontainer.json の `customizations.vscode.extensions` で自動インストールする。

ローカルの秘匿情報（Harbor 認証トークン、Git 署名鍵）は Dev Container 内の `~/.config/k1s0/credentials` に手動配置し、`.gitignore` で Git 管理除外する。Phase 1c から Keycloak + OIDC による自動取得へ移行する計画である。

## ローカル Kubernetes（kind / minikube）

ローカル Kubernetes の選択肢は kind（推奨）と minikube（代替）の 2 択とする。kind は Docker-in-Docker 方式で起動が速く、マルチノード対応があり、CI との互換性が高い（GitHub Actions でも kind が動く）。minikube は VM ベースで動作が安定だが、起動が遅い。Phase 1a 標準は kind、Mac の Apple Silicon 環境で互換性問題が発生した場合のみ minikube を選ぶ。

Phase 2 では k3d を評価対象とする。k3d は k3s を Docker コンテナで動かす軽量実装で、kind よりさらに起動が速いが、tier1 の要件（Dapr Operator / Strimzi Operator / CloudNativePG Operator の互換性）を満たすかを Phase 1c 末に検証してから判断する。

クラスタ構成は以下を標準とする。Control Plane 1 ノード + Worker 1 ノード（合計 2 ノード）、Kubernetes バージョンは本番と同一のマイナー版（Phase 1a では v1.28）、CNI は kindnet（デフォルト）、Ingress は NGINX Ingress Controller（kind は NodePort、minikube はアドオン）、とする。クラスタ起動は `k1s0 dev up` コマンドでラップし、開発者は Kubernetes の詳細を意識せず使える。

## Tilt による即時反映

Tilt は「Kubernetes 向けの開発者ループ自動化ツール」で、ファイル変更を検知して即座にコンテナを再ビルドし、Kubernetes に反映する。Tiltfile（Starlark DSL）で「どのソースを監視し、どのイメージをビルドし、どのマニフェストを apply するか」を宣言する。

Tiltfile の標準例は以下の通り。

```python
# Tiltfile（雛形生成物）
load('ext://restart_process', 'docker_build_with_restart')

# Rust サービス（live_update で cargo watch 互換）
docker_build_with_restart(
    'k1s0/my-service',
    '.',
    entrypoint=['/app/my-service'],
    live_update=[
        sync('./target/release/my-service', '/app/my-service'),
    ],
)

# Kubernetes マニフェストを apply
k8s_yaml('deploy/dev/my-service.yaml')
k8s_resource('my-service', port_forwards=['8080:8080'])

# 依存サービス（Dapr / Valkey / OpenBao）
include('../../tier1/Tiltfile')
```

ファイル変更から再反映までの時間は、Rust で 10 秒以内（`cargo build --release` のインクリメンタルビルド + Live Update でバイナリ差し替え）、Go で 5 秒以内（go build 即時 + Live Update）、TypeScript で 3 秒以内（Hot Module Replacement）を目標とする。この時間予算は Phase 1b で計測し、Phase 1c で固定する。

## tier1 API モック

tier2 / tier3 開発者は tier1 の実インスタンスを起動する必要がない。tier1 API モック（`tier1-mock`）を雛形に同梱し、Tilt で tier1 実サービスの代わりに起動できる。モックは `.proto` ファイルから自動生成した gRPC スタブで、全 11 API の定義済みレスポンスを返す。

モックの切替は Tiltfile のフラグで制御する。`k1s0 dev up --with-tier1=mock`（デフォルト）でモック、`--with-tier1=real` で実 tier1 起動となる。モード切替で再ビルド不要で、Tiltfile の環境変数切替のみで完了する。

モックのレスポンス定義は YAML で宣言する。例えば `State.Get(key=foo)` に対して `value="bar"` を返す、`PubSub.Publish` に対して常に成功を返す、といった定義を `mocks/tier1-responses.yaml` に記述する。モックは contract test（pact）と同じ定義ファイルを共有し、Consumer-Driven Contract の一貫性を保つ。

Phase 2 では Record/Replay モード（実 tier1 の応答を記録して再生）を追加する。tier3 開発者が本番相当の複雑な応答パターンを再現してデバッグする用途を想定する。

## Dapr サイドカー注入

Dapr サイドカーは Tilt 経由で自動注入する。Kubernetes Deployment のアノテーション `dapr.io/enabled: "true"` を付けると、ローカル kind 上で Dapr Operator が検知し、daprd サイドカーを Pod に注入する。tier2 / tier3 開発者は Dapr の構成詳細を意識せず、SDK 呼び出しだけを書ける。

ローカルでの Dapr コンポーネント定義は以下を標準構成とする。State Store は Valkey（Docker コンテナ 1 個）、PubSub は Kafka（Strimzi Operator、Phase 1b 以降）、Secret Store は OpenBao Dev Mode、Binding は必要に応じて個別追加、である。本番同等の API を提供することが目的で、運用性や HA は不要である。

## Secret 管理（OpenBao Dev Mode）

ローカル開発では OpenBao Dev Mode を使う。Dev Mode は認証をスキップして即起動できるモードで、開発者が Secret の put / get を体感する用途に適する。Dev Mode の API は本番 OpenBao と完全互換で、開発時のコードを変更せず本番へデプロイできる。

Secret の定義は Helm Chart の values.yaml で宣言し、Tilt 起動時に OpenBao へ put する。例えば `database.password=dev-password` のようなキーを開発用に設定する。本番では同じキー名で別 Secret を配置し、アプリ側は Secret 名のみを認識するため環境差異を吸収できる。

## ローカル観測基盤

トレース・メトリクス・ログを本番同様にローカルで確認できる環境を提供する。Grafana / Tempo / Prometheus / Loki の軽量版を kind クラスタ上に同居配置する。開発者は Tilt UI から `http://localhost:3000` でダッシュボードを開き、自分のサービスが発行した OpenTelemetry トレースを即確認できる。

観測基盤の同居は本番と独立したインスタンスで、データは Pod 再起動で失われる（永続ボリューム不使用）。ローカルでの軽量化のため、Prometheus のスクレイプ間隔は 60 秒（本番は 15 秒）、Tempo の保持期間は 1 時間（本番は 7 日）とする。ログレベルは DEBUG を標準とし、本番の INFO より詳細に出力する。

観測基盤のデプロイは `k1s0 dev obs up` で一括起動する。Phase 1b でこのコマンドを整備し、Phase 1c から Backstage TechDocs から直接ガイドを参照できるようにする。
