# 70. 開発者体験方式設計

本カテゴリは要件定義書の [../../03_要件定義/50_開発者体験/](../../03_要件定義/50_開発者体験/) に 1:1 対応する方式設計章である。tier2 / tier3 開発チームが k1s0 基盤を利用して業務アプリを構築する際の生産性・自動化・ガイダンスを方式として固定化する。

## 本カテゴリの位置付け

k1s0 は「開発者にとって楽である」ことを事業上の差別化要因として掲げる。どれだけ基盤が堅牢でも、tier2 / tier3 の開発者がローカルで動かせない・CI で回らない・メトリクスが取れないなら、採用されず OpenShift などの商用 PaaS に舞い戻る圧力が生じる。本章は、この生産性レベルを契約レベルで約束するための方式文書である。

要件定義書の 50 カテゴリは tier2 / tier3 のリードエンジニアが最終合意する層であり、ここでの方式不整合は開発チームの離反に直結する。Backstage 導入・Golden Path 整備・CI/CD パイプラインといった取り組みを方式として明文化することで、「なんとなく便利」を「契約レベルの生産性保証」に昇格させる。

## 構想設計との役割分担

[../../02_構想設計/04_CICDと配信/](../../02_構想設計/04_CICDと配信/) で CI/CD パイプライン・Backstage・アプリ配信ポータル・ローカル開発環境の**骨格と選定根拠**が既に確立されている。本章はその選定を踏まえて、具体的なパイプライン定義・プラグイン構成・ローカル環境のセットアップ手順・DevEx 指標の計測方式を**方式として固定化**する。

## 格納予定ドキュメント

6 ファイルを格納する。

- `01_CI_CD方式.md` — GitHub Actions / Argo CD のパイプライン方式、ビルド・テスト・デプロイの各ステージ設計、プロモーション承認ゲート、サプライチェーン保護（署名・SBOM）の組み込み方式を記述する。要件 ID `DX-CICD-*` と対応する。
- `02_ローカル開発環境方式.md` — 開発者ノート PC 上での k1s0 ミニクラスタ方式（k3d / kind / Dapr Local 等）、依存コンポーネントのモック方式、テナントデータのローカル生成方式、オフライン開発の範囲を記述する。要件 ID `DX-LD-*` と対応する。
- `03_DevEx指標方式.md` — 開発者生産性を測定する指標（リードタイム・デプロイ頻度・変更失敗率・MTTR の DORA 4 指標、Backstage スコアカード、Golden Path 採用率）の計測方式、ダッシュボード構成、改善サイクルを記述する。要件 ID `DX-MET-*` と対応する。
- `04_Backstageポータル詳細方式.md` — Backstage のプラグイン選定、TechDocs 配信方式、Software Catalog のエンティティ設計、所有者情報の同期方式を記述する。[../20_ソフトウェア方式設計/05_利用者文書_暫定版/03_Backstageポータル利用初版.md](../20_ソフトウェア方式設計/) は利用者向け、本章は運用者向けの詳細方式。
- `05_テスト戦略方式.md` — 単体 / 契約 / 結合 / E2E / カオス / 負荷 の各テスト階層の方式を俯瞰する。[../10_システム方式設計/05_システム結合テスト要件.md](../10_システム方式設計/) と [../20_ソフトウェア方式設計/06_ソフトウェア結合テスト要件.md](../20_ソフトウェア方式設計/) が個別テスト階層の要件を扱うのに対し、本章はテスト戦略全体の俯瞰と開発者体験としての位置付けを扱う。要件 ID `DX-TEST-*` と対応する。
- `06_ゴールデンパス方式.md` — tier2 / tier3 開発者に提供する推奨実装経路（Golden Path）の方式を記述する。scaffold テンプレートの構成方針、採用する推奨アーキテクチャ（CRUD / Event-Driven / Workflow 駆動などのパターン）、言語別 SDK の推奨使い方、反推奨（アンチパターン）の明示方式、テンプレートのバージョニングと廃止ポリシー、Backstage Software Templates との接続方式を含む。Backstage（04）は配信ツールの方式、本ファイルは Golden Path そのものの設計方式であり責務が異なる。要件 ID `DX-GP-*` と対応する。

## 執筆方針

- 本章は「開発者が 1 時間以内に最初の Hello World を動かせる」水準の具体性を目指す。抽象的な方針で留めない。
- 方式 ID は `DS-DEVX-<サブカテゴリ>-<通番>` を用いる（例: `DS-DEVX-CICD-001`、`DS-DEVX-LOCAL-001`）。
- 04 Backstage と 20/05/03 Backstage 利用初版の境界を各章の冒頭で明示し、二重管理を避ける。
