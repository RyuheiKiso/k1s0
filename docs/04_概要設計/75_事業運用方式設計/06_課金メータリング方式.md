# 06. 課金メータリング方式

本章は k1s0 の課金メータリング（利用量計測と社内 chargeback / 請求書生成）の方式を定める。企画書で約束した BC-BIL-001a（基本 4 メトリック、リリース時点 MUST）と BC-BIL-001b（詳細追跡、採用後の運用拡大時）を実装に落とし込む中核章である。

## 本章の位置付け

プラットフォーム基盤を複数テナントが共用する場合、「誰がどれだけ使ったか」を計測し、社内費用分担または社外請求の根拠とする必要がある。計測がなければ「受益者負担」の原則が崩れ、重利用テナントが軽利用テナントのコストを相対的に低く見せる不公平が生じる。計測があっても誤差が大きければ信頼性が失われ、テナント側での監査対応（利用量の妥当性説明）が困難になる。

本章は「正確性（誤差 1% 未満）」「透明性（テナントがリアルタイムに自分の利用量を確認可能）」「監査性（計測ロジックが説明可能・再計算可能）」の 3 要件を同時に満たす設計を行う。リリース時点 では社内 chargeback 表示まで、採用後の運用拡大時 で月次請求書生成まで対応する段階的実装とする。

## 4 基本メトリック（BC-BIL-001a、リリース時点 MUST）

### メトリック 1: API 呼び出し数

tier1 公開 API 11 種（Service Invoke / State / PubSub / Secrets / Binding / Workflow / Log / Telemetry / Decision / Audit-Pii / Feature）への呼び出し数をテナント別に計測する。計測対象は tier1 API ゲートウェイ（Envoy Gateway）を通過する全リクエストとする。計測粒度は API メソッド単位（例: `k1s0.State/Get`、`k1s0.PubSub/Publish`）まで細分化する。

計測方法は Envoy Gateway のアクセスログと OTel Metrics の Counter（`k1s0_api_calls_total{tenant_id, api, method, status}`）を二重に取得する。二重計測は誤差検出用途であり、両者の差分が 0.5% を超える場合は計測ロジックの故障として Runbook で対処する。

### メトリック 2: State 容量（GB·h）

tier1 State サービスが保持するテナント別データ容量を時間積分で計測する。単位は GB·h（ギガバイト時）とし、1 GB のデータを 1 時間保持したら 1 GB·h とカウントする。計測対象は Valkey のテナント別 namespace 内の総データ容量とする。

計測方法は Valkey の `DBSIZE` 相当を 5 分間隔で収集し、5 分間の平均値を 5 分 / 60 分 = 1/12 に乗じて積算する。Valkey から直接取得できない場合は、Dapr state store の API 経由でテナント別キーをスキャンして総容量を計算する。5 分間隔は計測コストと精度のバランスから選定した（1 分間隔ではメトリック数が 12 倍に増える、15 分間隔では精度が落ちる）。

### メトリック 3: PubSub メッセージ数

tier1 PubSub サービスでパブリッシュおよびコンシュームされたメッセージ数をテナント別に計測する。パブリッシュ数（発行側）・コンシューム数（購読側）を独立に計測し、請求根拠は発行数とする（コンシューム数は可視化目的のみ）。

計測方法は Kafka のトピック別 offset の差分を Prometheus kafka_exporter から取得する。テナント別トピック（`tenant-<id>-<topic>`）の命名規則により、トピック名でテナント識別する。メッセージサイズは別メトリック（`k1s0_pubsub_bytes_total`）として記録し、採用後の運用拡大時の詳細課金で活用する。

### メトリック 4: Workflow 実行数

tier1 Workflow サービス（Dapr Workflow / Temporal）で実行されたワークフロー数をテナント別に計測する。1 ワークフロー実行で 1 カウントとし、ワークフロー内のアクティビティ数は 採用後の運用拡大時の詳細課金で別メトリック化する。

計測方法は Dapr Workflow の完了イベントおよび Temporal の Workflow Completed イベントを OTel Span として発行し、Tempo で集計する。Tempo からの日次バッチで tenant_id 別の完了数を PostgreSQL 集計テーブルに格納する。

## 計測実装

### 計測パイプライン

計測は「生イベント → Prometheus / Tempo → Mimir 長期保管 → 月次集計 → PostgreSQL 課金データ」の 5 段パイプラインで構成する。生イベントは Envoy アクセスログ・OTel Metrics・Kafka exporter・Tempo Span の 4 ソースから発生する。Prometheus が 1 時間スクレイプし、Mimir が 13 ヶ月分を長期保管する。月次集計（毎月 3 日）で前月分をテナント別に確定し、PostgreSQL の `billing_monthly_<yyyymm>` テーブルに格納する。

集計は冪等性を保つため同一期間で再実行可能とする。集計ジョブは Dapr Workflow で実装し、中断・リトライ・補償のいずれも Saga パターンで対処する。集計完了後、テナント別に請求データを確定する。確定後の改訂は Product Council 承認を経る（経理監査の観点から、確定後改訂は例外扱いとする）。

### tenant_id 強制検証

誤課金防止のため、すべての計測イベントには tenant_id の必須入力を強制する。tenant_id の生成は Envoy Gateway で Keycloak JWT の `tenant` claim から抽出し、下流サービスに HTTP ヘッダ `x-k1s0-tenant` として注入する。下流サービスはヘッダの有無を検証し、欠落時は 400 エラーを返却する。

tenant_id が不明なリクエスト（認証前のヘルスチェック等）は共通プール `system` としてカウントし、請求対象外とする。`system` プールの消費量を定期的にレビューし、テナント起因のリクエストが `system` に混入していないかを監査する。

### 二重計測と照合

正確性担保のため、同一イベントを 2 系統で計測する。系統 A は Envoy アクセスログベース、系統 B は OTel Metrics ベースとする。日次バッチで両系統の差分を計算し、差分が 0.5% を超える場合はアラートを発報する。差分が継続的に発生する場合は計測ロジックを修正し、修正履歴を ADR として記録する。

両系統とも故障した場合のフォールバックとして、Kafka トピック `billing-events` に直接イベントを publish するパスを備える。このパスは通常時は冗長だが、Prometheus / OTel 収集系の障害時に課金データの欠損を防ぐ最終手段となる。

## 保管

### 保管階層

計測データは以下 3 階層で保管する。(1) ホット層（Prometheus）: 直近 15 日、5 分粒度。障害対応・リアルタイム可視化用途。(2) ウォーム層（Mimir）: 直近 13 ヶ月、1 時間粒度。月次請求書生成・トレンド分析用途。(3) コールド層（PostgreSQL 課金テーブル）: 7 年保管、月次粒度。監査対応・法的保管要件。

保管期間は会計監査の要求（7 年）と法的保管要件（電帳法 7 年）に合わせる。コールド層は読み取り専用とし、改ざん検知のためハッシュチェーンで整合性を担保する（[../50_非機能方式設計/08_データ保護とプライバシー方式.md](../50_非機能方式設計/08_データ保護とプライバシー方式.md) の監査改ざん防止と連動）。

### バックアップ

PostgreSQL 課金テーブルは毎日フルバックアップ + 毎時差分バックアップを S3 互換ストレージに取得する。バックアップの保管先は本番環境とは別の物理データセンターとし、災害時の復元に備える。復元訓練は四半期ごとに実施し、復元時間 RTO 4 時間・RPO 1 時間を検証する。

## 課金モデル

### リリース時点: 社内 chargeback 表示

リリース時点 は社内コスト配賦（chargeback）の表示のみを行う。請求書は発行せず、テナント代表者が Backstage 上で「今月の利用量」「換算コスト（情報提供）」を閲覧可能とする。換算コストはコスト管理方式（[05_コスト管理方式.md](05_コスト管理方式.md)）の費目別モデルに基づき、4 メトリックをコストに変換して表示する。

リリース時点 段階では配賦額を「情報提供」として表示するに留め、経理システムとの連携や実際の社内振替は行わない。社内振替は 採用後の運用拡大時 で経理システム連携時に開始する。

### 採用後の運用拡大時: 月次請求書生成

採用後の運用拡大時 では月次請求書を自動生成する。請求書は以下の 5 情報を含む PDF ドキュメントとする。(1) テナント ID と表示名、(2) 対象期間（yyyy 年 m 月）、(3) 4 基本メトリックの使用量、(4) 換算額（単価 × 使用量）、(5) 合計金額（税込 / 税抜）。

請求書生成は月次集計完了後（毎月 5 日目標）に実行し、承認フロー（テナント代表者 → 経理責任者）を経てテナント代表者に PDF をメール送信する。承認フローは Dapr Workflow で実装する。電子署名には社内認証局の電子印影を使用し、原本性を担保する。

### 単価設定

単価は Product Council で年次決定する。初期単価案（採用後の運用拡大時 開始時）は以下とする。API 呼び出し 1 万回 100 円、State 1 GB·h 1 円、PubSub 1 万メッセージ 50 円、Workflow 1 実行 10 円。単価は市場水準（クラウド従量課金 + 間接費削減メリット）を参考に、中規模テナントの年額が 100〜500 万円となるよう調整する。

単価改定は採用検討を経て年次実施する。改定時は 90 日前告知をテナントに行う。

### 越境防止

テナント間の誤課金を防ぐため、計測ロジックは tenant_id 単位で厳密に分離する。tenant_id の注入漏れは Envoy Gateway で拒否され、下流サービスで再検証される。tenant_id が空または不明な場合は請求対象外プール `system` にカウントされる。

越境テストは CI で毎回実行する。テストは複数テナントの模擬リクエストを送信し、計測結果が各テナントに正しく帰属することを検証する。越境検出時は計測ロジックを修正し、過去の請求データを Product Council 承認の下で再計算する。

## 透明性

### テナント UI

テナント代表者が自テナントの利用量をリアルタイムに確認できる UI を Backstage プラグインとして提供する。UI の主要画面は (1) 当月サマリ（4 メトリックの使用量と前月比）、(2) 日別推移（4 メトリック × 30 日のグラフ）、(3) API 別内訳（上位 20 API の呼び出し数）、(4) 請求書履歴（採用後の運用拡大時）、(5) 異常検知アラート（通常利用との乖離 3σ 超過の通知）の 5 画面とする。

UI のデータ更新は 5 分遅延（Prometheus スクレイプ間隔と同じ）とし、リアルタイム表示と銘打つが厳密にはニア・リアルタイムとする。監査対応で正確な時刻別利用量が必要な場合は、Mimir への直接クエリで 5 分粒度の確定値を取得可能とする。

### 計測ロジック公開

計測ロジックは社内向けに全文公開する（[../50_非機能方式設計/](../50_非機能方式設計/)、および本章）。テナント側が「なぜこの請求額になったか」を追跡可能にするため、個別リクエストのトレース（Tempo）・生メトリック（Prometheus）・集計結果（PostgreSQL）を各テナントが自テナント分に限り参照可能とする。参照は Keycloak の `tenant` 属性ベース RBAC で制御する。

## 対応要件一覧

本章は以下の要件定義 60_事業契約 課金メータリング系要件と 1:1 対応する。

- BC-BIL-001: 4 基本メトリック（API 呼び出し / State 容量 / PubSub メッセージ / Workflow 実行）の計測、リリース時点 MUST。対応設計 ID: `DS-BUS-BIL-001`。
- BC-BIL-002: 計測パイプライン（OTel + Prometheus + Mimir + PostgreSQL）の実装。対応設計 ID: `DS-BUS-BIL-002`。
- BC-BIL-003: tenant_id 強制検証と二重計測による誤課金防止。対応設計 ID: `DS-BUS-BIL-003`。
- BC-BIL-004: 保管階層（ホット / ウォーム / コールド）と 7 年保管。対応設計 ID: `DS-BUS-BIL-004`。
- BC-BIL-005: リリース時点 社内 chargeback 表示と 採用後の運用拡大時 月次請求書生成。対応設計 ID: `DS-BUS-BIL-005`。
- BC-BIL-006: テナント UI でのリアルタイム使用量可視化。対応設計 ID: `DS-BUS-BIL-006`。
- BC-BIL-007: 単価設定の年次決定プロセスと 90 日前告知。対応設計 ID: `DS-BUS-BIL-007`。
