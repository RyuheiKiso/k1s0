# 05. コスト管理方式

本章は k1s0 の FinOps 運用、コスト可視化、予算管理、コスト最適化の方針を定める。企画書で約束した「中規模年額の OSS 隠れコスト 2,349 万円/年（自社 5 FTE 人件費 6,900 万円は別計上、合算 9,249 万円で OpenShift 中規模 1.8 億円比 49% 削減）」「5 年 TCO 3.68 億円以内」を維持するため、コストの可視化・監視・最適化を運用プロセスとして組み込む。以降の本章で「2,349 万円」と記載する場合は常に OSS 隠れコスト部分を指し、人件費は FinOps の別 KPI として追跡する。

## 本章の位置付け

プラットフォーム製品のコストは、ライセンス費・インフラ費・運用人件費・電力費・監査対応費の 5 項目で構成される。k1s0 は OSS セルフマネージドを採用しているためライセンス費はゼロ（OSS 採用を守り抜けば）、オンプレ前提のためインフラ費が主要費目、運用 2 名体制により人件費が抑制、オンプレ効率運用により電力費削減、自動化により監査対応費削減、という構造で 49% 削減を達成する。本章はこの構造を運用で維持する仕組みを定義する。

稟議で合意した予算を超過すれば稟議の前提が崩れ、経営判断を修正する必要が生じる。逆に予算内に収まっても使い切れていなければ「余剰」として次年度に削減圧力がかかる。どちらも避けるため、月次の予実管理と四半期の予算再調整を本章で設計する。

## FinOps 導入

### FinOps 組織

FinOps は [10_ガバナンス運用方式.md](10_ガバナンス運用方式.md) の Product Council 配下に「FinOps 分科会」として設置する。分科会メンバは情シス責任者 1 名（議長）、経理部門代表 1 名、運用責任者 1 名、各テナント代表（大規模テナントのみ）の構成とする。分科会は月次で開催し、予算消化率・コスト最適化候補・予算再配分提案を議題とする。

FinOps 分科会の意思決定権限は「月額 100 万円以下の予算再配分」「コスト最適化施策の実施承認」「テナント別 Quota 調整」の 3 項目に限定する。これを超える意思決定（年額ベース予算枠組みの変更・新規サービス導入・大規模構成変更）は Product Council のエスカレーション対象とする。

### FinOps プラクティス

FinOps Foundation の 6 原則（チーム協働・ビジネス価値重視・誰もがオーナーシップ・集中チームの運営・実績の即時可視化・クラウド可変コストモデルの活用）を k1s0 に適合させる。オンプレ前提のため「クラウド可変コストモデル」は「オンプレ可変コストモデル（VM 集約率・電力効率・ハードウェア更新サイクル）」に読み替える。

FinOps の成熟度は「Crawl（基本可視化）→ Walk（定期レビュー + 最適化）→ Run（予測・自動調整）」の 3 段階で評価する。Phase 1c までに Crawl 達成、Phase 2 で Walk 達成、Phase 3 で Run 達成を目標とする。

## コスト可視化

### 可視化粒度

コストは以下 4 軸で可視化する。(1) テナント別: 各テナントの消費リソース（CPU / Memory / Storage / Network / API 呼び出し数）とそれを金額換算した値。(2) サービス別: tier1 内部サービス（Envoy Gateway / Dapr / Valkey / Kafka / OpenBao / PostgreSQL / Grafana LGTM / Keycloak 等）の消費コスト。(3) 環境別: production / staging / development / sandbox の環境ごとのコスト。(4) 費目別: ハードウェア償却費・電力費・運用人件費・監査対応費。

### 可視化ダッシュボード

Grafana にコストダッシュボードを作成する。ダッシュボードは経営向け（サマリ）・情シス向け（詳細）・テナント向け（自テナント分のみ）の 3 種類を提供する。経営向けは月次・四半期・年次の集計、情シス向けは週次・日次の詳細、テナント向けはリアルタイム（5 分遅延）表示とする。

ダッシュボードの主要パネルは (1) 月次予実（予算 vs 実績の積み上げグラフ）、(2) 費目別推移（時系列折れ線）、(3) テナント別ランキング（上位 10 テナント）、(4) コスト効率指標（$ per 1M API 呼び出し / $ per TB Storage / $ per テナント）、(5) 前月比 / 前年同月比の 5 パネルを基本構成とする。

### コスト計算モデル

オンプレのコスト計算はクラウドのような従量課金がないため、以下のモデルで按分する。ハードウェア償却費は VM ホストの購入価格を 5 年定額償却し、VM の CPU コア数・割当メモリ・割当ストレージで按分する。電力費は VM ホストの実測電力（Kepler or サーバ BMC）にデータセンター契約単価を乗じる。運用人件費は運用 2 名の人件費をテナント数で均等按分する（Phase 2 以降、利用量比例按分に切替候補）。

計算モデルは FinOps 分科会で四半期レビューし、実態との乖離が 10% を超える場合はモデルを改訂する。モデル改訂は ADR として記録する。

## 予算管理

### 予算階層

予算は以下 3 階層で管理する。(1) 全体予算（中規模年 2,349 万円、稟議で合意）、(2) 環境別予算（production 70% / staging 15% / development 10% / sandbox 5%）、(3) テナント別予算（テナント申込時に決定）。各階層で予実対比を実施する。

### 予算アラート

予算消化率に応じて以下のアラートを発報する。80% 到達: Slack 警告通知（担当者 + 情シス責任者）。90% 到達: メール警告通知（責任者 + Product Council）。100% 到達: 自動アクション（新規リソース作成停止、既存リソースは維持、テナント代表者とエスカレーション）。120% 到達: サービス停止検討（Product Council 緊急会議で判断）。

自動停止は誤作動のリスクがあるため、Phase 1c は 100% でのアラート通知のみとし、Phase 2 で自動停止をオプトイン有効化する。100% 到達時に即座に新規リソース作成を停止するのではなく、24 時間の猶予期間を設けて責任者の判断を待つ運用とする。

### 予算申請

新規テナント申込時に年間予算を申請する。予算規模に応じて承認権限を分ける。年額 100 万円以下: 情シス責任者承認、年額 100〜500 万円: Product Council 承認、年額 500 万円超: 経営層承認。予算超過見込みは四半期レビューで発見し、次四半期以降の予算調整で対処する。

予算を使い切れない場合は、四半期末に余剰を他テナントへ再配分可能とする。余剰を繰り越すのではなく、期中で有効活用する運用とする。

## コスト最適化

### 最適化手順

コスト最適化は月次レビューで以下 5 種類の施策を実施する。

(1) Right Sizing: VM の CPU / Memory 割当を実測値に近づける。過剰割当は月次で検出し、20% 以上の余裕がある VM は縮小候補とする。(2) 未使用削除: 30 日以上アクセスのない State エントリ・PubSub トピック・Secret・バックアップを検出し、テナント代表者の同意を得て削除する。(3) VM 集約: 複数 VM を 1 台のベアメタルに集約し、オーバーヘッドを削減する。Phase 2 以降で VM → Kubernetes Node 集約を検討する。

(4) ストレージ階層化: 頻繁にアクセスするホットデータを SSD、アクセス頻度の低いウォームデータを HDD、長期保管アーカイブを S3 互換ストレージに分離する。(5) ハードウェア更新: 5 年サイクルでのハードウェア更新時、性能 / 電力効率向上を評価し、次世代ハードウェアへの移行を検討する。Phase 2 以降。

クラウド一般で用いられる Spot / Preemptible インスタンスの活用はオンプレには適用できないため N/A とする。代替として電力ピーク時の負荷シフト（バッチ処理を夜間に寄せる）を Phase 3 以降で検討する。

### 最適化効果測定

最適化施策ごとに事前事後のコストを比較し、効果を金額で記録する。効果は四半期ごとに集計し、年次レポートで経営層に報告する。効果が見込みの 50% を下回った場合は施策の有効性を再評価し、次回は別アプローチを採用する。

## 月次コストレポート

月次コストレポートは FinOps 分科会で作成し、Product Council に提出する。レポートの構成は以下 6 項目とする。(1) 全体サマリ（予実対比・前月比）、(2) 費目別詳細、(3) テナント別ランキング、(4) 最適化施策実施状況、(5) アラート発報状況、(6) 次月見込み。

レポートは Backstage Announcement で全テナントに公開する（自テナント情報のみテナント代表者が閲覧可能）。経営向けサマリは [12_分析とレポーティング方式.md](12_分析とレポーティング方式.md) の月次 Product Council レポートに組み込む。

## TCO 維持

企画書で約束した「中規模年額 2,349 万円、5 年 TCO 3.68 億円」を維持するための監視指標を定義する。月次予算消化率（100% 以内 10 ヶ月 / 12 ヶ月）、年次予算消化率（100±5% 以内）、5 年累計実績（3.68 億円以内）の 3 指標を追跡する。

TCO 超過見込みが発生した場合は、まず最適化施策で吸収を試み、それでも不足する場合は稟議再承認またはスコープ調整を行う。逆に TCO が 10% 以上下振れる場合は、余剰を拡張投資（Phase 2 / 3 の前倒し投資）または次年度予算削減に振り分ける。

## 対応要件一覧

本章は以下の要件定義 60_事業契約 コスト管理系要件と 1:1 対応する。

- BC-COST-001: FinOps 分科会の設置と月次運営。対応設計 ID: `DS-BUS-COST-001`。
- BC-COST-002: コスト可視化（テナント / サービス / 環境 / 費目）の Grafana ダッシュボード。対応設計 ID: `DS-BUS-COST-002`。
- BC-COST-003: 予算階層（全体 / 環境 / テナント）と予算アラート（80% / 100% / 120%）。対応設計 ID: `DS-BUS-COST-003`。
- BC-COST-004: コスト最適化 5 施策（Right Sizing / 未使用削除 / VM 集約 / ストレージ階層 / ハードウェア更新）の月次実施。対応設計 ID: `DS-BUS-COST-004`。
- BC-COST-005: 月次コストレポートの Product Council 提出。対応設計 ID: `DS-BUS-COST-005`。
- BC-COST-006: TCO 維持指標（月次 / 年次 / 5 年累計）の追跡と稟議予算との整合性担保。対応設計 ID: `DS-BUS-COST-006`。
