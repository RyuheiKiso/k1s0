# 10. ガバナンス運用方式

本章は k1s0 の意思決定体制、Product Council の運営、RACI マトリクス、意思決定ログ、エスカレーション、監査対応、ポリシー違反対応を定める。企画書で約束した「経営 / 情シス / 法務 / 開発 / 運用の 5 代表による Product Council 月次運営」「採用検討通過済み予算（OSS 隠れコスト 2,349 万円/年 + 自社 5 FTE 人件費 6,900 万円、合算 9,249 万円）の維持」を運用プロセスとして実現する。本章で以降「予算 2,349 万円」と略記する場合は OSS 隠れコスト分を指し、人件費予算は Product Council の別議題として並走監視する。

## 本章の位置付け

プラットフォーム製品では意思決定の遅延が事業機会喪失を、意思決定の拙速が長期的技術負債を招く。k1s0 は 採用側組織の情シス向けとして、申請プロセスで意思決定の透明性が特に重視されるため、意思決定ロジックと記録を第三者が検証可能な粒度で残す必要がある。本章はその構造を Product Council と ADR（Architectural Decision Record）の 2 軸で構築する。

ガバナンスは「誰が・何を・どのように・どの期限で」決めるかを明示することで、個人の裁量で動く属人運営を排除する。運用 採用側の小規模運用（原則 9）の維持には、意思決定プロセスが 2 名以外の関係者（経営 / 法務等）にも可視化されていることが必須となる。

## Product Council

### 設置意義

Product Council は k1s0 の最高意思決定機関として設置する。k1s0 リリース時点で発足し、それ以降は月次で定期開催する。Product Council の設置意義は (1) 横断的意思決定の一元化（アーキテクト 1 名では判断困難な事項の合議）、(2) ステークホルダ間の利害調整、(3) 採用検討通過済み合意の履行監視、(4) 企画コミットメントの進捗評価、の 4 点である。

### メンバ構成

Product Council は以下 5 代表で構成する。(1) 経営代表（情シス担当役員 or CIO、西尾決裁者ペルソナ）、(2) 情シス代表（情シス部門責任者、森田推進派ペルソナ）、(3) 法務代表（社内法務部門責任者）、(4) 開発代表（tier1 起案者 + 協力者のうち 1 名、兼 tier2 / tier3 代表として参加）、(5) 運用代表（運用責任者、石田運用ペルソナ）。必要に応じてテナント代表者・外部法律事務所・経理部門を招聘する。

### 会議の運営

会議は月次で 90 分枠を確保する。アジェンダは以下 6 セクションとする。(1) 前月 KPI サマリ（10 分、[12_分析とレポーティング方式.md](12_分析とレポーティング方式.md)）、(2) ADR 承認（15 分、新規 ADR の審議）、(3) インシデント報告（10 分、前月発生インシデントと再発防止）、(4) 予算予実（10 分、[05_コスト管理方式.md](05_コスト管理方式.md) と連動）、(5) ロードマップ進捗（15 分、段階移行判定含む）、(6) 議事と決議（30 分、新規議題の議論と意思決定）。

議事録は Backstage Discussion Forum（または GitHub Discussions）に公開する（機密情報を除く）。意思決定は多数決ではなく、5 代表の合意を基本とする。合意形成不能な場合は経営代表が裁定し、その旨を議事録に明記する。

### 意思決定権限

Product Council の意思決定権限は以下に限定する。(1) ADR 承認（技術判断）、(2) API 仕様変更（tier1 公開 API の破壊的変更）、(3) 採用段階の進行判定（採用初期 → 採用後の運用拡大時 → 採用側のマルチクラスタ移行時 → 採用側の全社展開期 の承認）、(4) 撤退判定（特定機能・テナント・OSS の廃止判断）、(5) 予算再配分（月額 500 万円超の変更、それ以下は FinOps 分科会）、(6) 法務契約改定、(7) 重大インシデント対応方針決定。

これを超える意思決定（年間予算枠の大幅変更・スコープ外の新規事業開始）は経営層 or 申請プロセスへエスカレーションする。

## RACI マトリクス

### 主要活動の RACI

RACI は Responsible（実行）・Accountable（説明責任）・Consulted（相談）・Informed（報告）の 4 区分で役割を明示する。主要活動 10 種類の RACI を以下に示す。

**ADR 起票**: 起票者（R）、Product Council（A）、法務（C、関連時）、全員（I）。
**ADR 承認**: Product Council（R/A）、法務（C、関連時）、全員（I）。
**段階移行判定**: Product Council（R/A）、経営（C）、全員（I）。
**インシデント対応**: 運用（R）、情シス責任者（A）、Product Council（I）。
**四半期ライセンスレポート確認**: 法務（R）、情シス責任者（A）、Product Council（I）、経営（I）。
**月次コストレポート確認**: FinOps 分科会（R）、Product Council（A）、経営（I）。
**テナントオンボーディング**: 情シス担当（R）、情シス責任者（A）、運用（C）、テナント代表（I）。
**セキュリティ監査**: 運用（R）、情シス責任者（A）、法務（C）、監査室（I）。
**新規 OSS 採用**: 起案者（R）、Product Council（A）、法務（C、AGPL 時）、全員（I）。
**SLA 違反対応**: 運用（R）、情シス責任者（A）、テナント代表（C）、Product Council（I）。

### RACI の維持

RACI マトリクスは本章で正本管理し、変更は Product Council 承認を経る。RACI の変更履歴は ADR として記録する。RACI の不整合（同一活動で複数の A が指定される等）は Product Council でレビューし、半年ごとに整合性を確認する。

## 意思決定ログ

### ADR

アーキテクチャ判断は ADR（Architectural Decision Record）として Git 管理する。ADR のテンプレートは `docs/00_format/adr-template.md` で提供する。主要セクションは以下 6 項目とする。(1) コンテキスト（判断の背景）、(2) 判断内容、(3) 結果（判断による利点と欠点）、(4) 代替案（検討した代替と却下理由）、(5) 関連要件 / 設計 ID、(6) 判断日 / 承認者 / レビュアー。

ADR の採番は `ADR-<通番>` とし、通番は全ドメイン横断で連続採番する。廃止された ADR は削除せず、ステータスを「Superseded by ADR-xxx」として記録する。ADR の件数は リリース時点 時点で 50〜100 件を想定する。

### GitHub Discussions

軽量な意思決定は GitHub Discussions で記録する。Discussion カテゴリは「議論（結論未決）」「決定（結論済）」「質問」「告知」の 4 つとし、「決定」のうち重要なものは ADR に昇格させる。

### Backstage Announcement

テナント代表者に向けた告知は Backstage Announcement で行う。対象は 段階移行告知、インシデント告知、料金改定告知、セキュリティ告知、法務契約改定告知の 5 種類とする。告知の重要度（Info / Warning / Critical）をラベルで明示し、Critical は必ずメール通知も併送する。

## エスカレーション

### エスカレーションパス

意思決定が Product Council で合意形成不能、または範囲外の場合、以下のエスカレーションパスを使用する。(1) 経営代表裁定（Product Council 内で経営代表が最終判断）、(2) 経営会議（経営層全体で判断、四半期）、(3) 採用再検討（1,000 万円超の予算変更・スコープ拡張）、(4) 取締役会（上場会社の場合、重大な事業戦略変更時）。

エスカレーション基準は「Product Council で 2 回連続合意不能」「Critical インシデントで Product Council 権限超え」「予算が年額 2,349 万円を 10% 超過見込み」の 3 条件のいずれかに該当する場合とする。

### 緊急意思決定

Critical インシデント（サービス全面停止、セキュリティ侵害疑い、Critical CVE）時は、Product Council の臨時会議を 24 時間以内に招集する。招集権限は情シス責任者と運用責任者にある。臨時会議は原則オンラインで実施し、5 代表の過半数（3 名）で成立する。

## 監査対応

### 内部監査

内部監査は四半期ごとに実施する。監査範囲は以下 5 領域とする。(1) J-SOX（IT 全般統制、業務処理統制）、(2) 個人情報保護法（安全管理措置）、(3) 電帳法（電子帳簿保存要件）、(4) OSS ライセンス（allow list 遵守、AGPL 5 原則）、(5) SLA 遵守。

監査結果は監査室に報告され、Product Council でレビューする。指摘事項は 30 日以内に是正計画を立案し、次四半期までに是正完了を目指す。重大指摘は経営層にも報告する。

### 外部監査

外部監査（公認会計士・外部法律事務所等）は年次で実施する。監査対象は会計監査と合わせた IT 監査とし、J-SOX 要件・情報セキュリティ要件・個人情報保護要件を重点とする。外部監査の指摘事項は優先的に対処する。

### 監査証跡

監査対応のための証跡は以下 5 種類を保管する。(1) ADR 全件（Git）、(2) 監査ログ（改ざん防止機構付き、7 年保管）、(3) 四半期ライセンスレポート（5 年保管）、(4) 月次 Product Council 議事録（7 年保管）、(5) 年次監査指摘事項とその対応（7 年保管）。これらは監査時に監査員がアクセス可能となるよう、事前に権限付与プロセスを整備する。

## ポリシー違反対応

### 違反検出

ポリシー違反は以下 4 経路で検出する。(1) CI / 自動検査（ライセンス違反、セキュリティ違反、コーディング規約違反）、(2) 監査ログ（不正アクセス、権限逸脱）、(3) 内部通報（社内通報窓口）、(4) 外部通報（テナント代表者・セキュリティ研究者等）。

### 対応段階

違反対応は以下 3 段階で実施する。(1) 教育: 軽微な違反（コーディング規約違反・手順逸脱）。対象者に違反内容と正しい手順を説明する。繰り返す場合は警告段階に昇格。(2) 警告: 反復違反・意図的違反の疑い。書面警告を発行し、違反者とマネージャが改善計画を立案する。(3) アクセス停止: 重大違反（セキュリティ違反・意図的情報漏洩・著作権侵害等）。該当ユーザのアクセス権を即時停止し、人事部門と法務部門に報告する。

### 違反記録

違反の記録は OpenBao の Secrets として機密保管する。記録項目は違反日時、違反者、違反内容、対応段階、対応結果とする。記録は人事部門責任者・法務部門責任者のみアクセス可能とする。

### 通報者保護

内部通報者・外部通報者は保護される。通報内容・通報者 ID は通報処理者以外に開示しない。報復行為は就業規則違反として処分対象とする。通報窓口は [08_法務契約運用方式.md](08_法務契約運用方式.md) の秘密情報アクセス制御と連動して運用する。

## Product Council ドキュメント

Product Council の運営文書は以下 5 種類を保管する。(1) 会議招集通知（月次、7 日前送付）、(2) アジェンダ（会議前日まで確定）、(3) 議事録（会議後 3 営業日以内に公開）、(4) 決議一覧（年次集計、全決議を時系列で一覧化）、(5) ADR 承認履歴（Git 履歴と連動）。これらは Backstage TechDocs の「governance」セクションで公開する。

本章は以下の要件定義 60_事業契約 ガバナンス系要件と 1:1 対応する。

- BC-GOV-001: Product Council 月次運営（経営 / 情シス / 法務 / 開発 / 運用の 5 代表）。対応設計 ID: `DS-BUS-GOV-001`。
- BC-GOV-002: 意思決定権限（ADR / API / 段階 / 撤退 / 予算 / 法務 / インシデント）の明文化。対応設計 ID: `DS-BUS-GOV-002`。
- BC-GOV-003: RACI マトリクス（10 活動）の維持と半年レビュー。対応設計 ID: `DS-BUS-GOV-003`。
- BC-GOV-004: 意思決定ログ（ADR / GitHub Discussions / Backstage Announcement）の運用。対応設計 ID: `DS-BUS-GOV-004`。
- BC-GOV-005: エスカレーション（経営代表 / 経営会議 / 取締役会）と緊急意思決定。対応設計 ID: `DS-BUS-GOV-005`。
- BC-GOV-006: 監査対応（四半期内部監査 + 年次外部監査）と監査証跡 7 年保管。対応設計 ID: `DS-BUS-GOV-006`。
- BC-GOV-007: ポリシー違反対応 3 段階（教育 → 警告 → アクセス停止）と通報者保護。対応設計 ID: `DS-BUS-GOV-007`。
