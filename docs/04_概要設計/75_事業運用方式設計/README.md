# 75. 事業運用方式設計

本カテゴリは要件定義書の [../../03_要件定義/60_事業契約/](../../03_要件定義/60_事業契約/) に 1:1 対応する方式設計章である。事業運営・契約・ガバナンスに関わる要件を、技術的な実現方式として固定化する層である。

## 本カテゴリの位置付け

要件定義書の 60 カテゴリは経営層・法務・調達・CSR 部門が合意する層であり、UX・i18n・テナントオンボーディング・課金・OSS ライセンス・AI ガバナンス・サステナビリティといった非技術領域を含む。しかしこれらの要件も、稼働システムで実現するには何らかの技術的方式が必要である。例えば「個人情報保護法に準拠する」という要件は、暗号化方式・アクセス制御方式・同意管理方式という技術設計に落ちる。

本章は要件定義の 60 カテゴリを「文書上の約束」から「稼働する仕組み」に翻訳する役割を持つ。他の設計章（30 共通機能 / 40 制御 / 50 非機能）が主に技術要件を扱うのに対し、本章は事業要件の技術実現を扱う点が異なる。要件定義者が「私の事業要件がどこで実現されているか」を問うた際、本章の特定ファイルに辿れる構造を保証することが目的である。

## 格納予定ドキュメント

13 ファイルを格納する。

- `01_UX方式設計.md` — 開発者ポータル・配信ポータル・CLI・エラーメッセージを通じた一貫した UX 原則、デザインシステム採用方針、アクセシビリティ対応レベル（WCAG 準拠水準）を記述する。要件 ID `BC-UX-*` と対応する。
- `02_i18n_l10n方式.md` — 国際化フレームワーク（ICU / ICU4X 等）の採用方式、翻訳ワークフロー、ロケール検出、タイムゾーン・通貨・数値フォーマット処理方式を記述する。要件 ID `BC-I18N-*` と対応する。
- `03_テナントオンボーディング方式.md` — 新規テナント受け入れのセルフサービス設計、本人確認・与信確認の組み込み方式、初期テナントスコープ・クォータ付与方式、オンボーディング完了判定を記述する。要件 ID `BC-ONB-*` と対応する。
- `04_教育訓練方式.md` — tier2 / tier3 開発者向け教育プログラム、認定制度、e-Learning 基盤との連携方式、社内トレーナー育成プロセスを記述する。要件 ID `BC-TRN-*` と対応する。
- `05_コスト管理方式.md` — インフラコストの計測・配賦・可視化方式、コスト最適化の継続プロセス、予算超過アラート、FinOps 実装方式を記述する。要件 ID `BC-COST-*` と対応する。
- `06_課金メータリング方式.md` — テナント課金のメータリング方式、使用量集計の信頼性保証（データ損失防止・改ざん防止）、請求データ連携、既存財務システムとの統合方式を記述する。要件 ID `BC-BIL-*` と対応する。
- `07_OSSライセンス運用方式.md` — 採用 OSS 40+ のライセンス義務（AGPL / MPL / Apache 2.0 等）履行方式、SBOM 生成と継続検証、ライセンス違反検知の自動化、配布物添付義務の機械化を記述する。要件 ID `BC-LIC-*` と対応する。
- `08_法務契約運用方式.md` — 利用規約・SLA 契約・NDA の機械可読化、契約条件の運用実装（SLO 違反時の自動補償など）、契約改訂プロセス、契約証跡の長期保全方式を記述する。要件 ID `BC-LGL-*` と対応する。
- `09_サプライチェーン方式.md` — ソフトウェアサプライチェーンのセキュリティ保護（署名・SBOM・依存関係監視・ビルドプロベナンス）、供給停止リスクへの対応、重要 OSS の代替案保有方式を記述する。要件 ID `BC-SC-*` と対応する。
- `10_ガバナンス運用方式.md` — Product Council・アーキテクトレビュー・ADR 承認・例外処理申請の運用方式、RACI の継続運用、ガバナンスダッシュボードを記述する。要件 ID `BC-GOV-*` と対応する。
- `11_AIガバナンス方式.md` — AI 生成コードの受入基準、モデル利用時のデータプライバシー保護、AI 由来成果物の監査証跡、バイアス・公平性評価方式を記述する。要件 ID `BC-AI-*` と対応する。
- `12_分析とレポーティング方式.md` — プラットフォーム利用状況の分析基盤、経営 KPI ダッシュボード、監査レポート生成、テナント向けレポート提供方式を記述する。要件 ID `BC-ANA-*` と対応する。
- `13_サステナビリティ方式.md` — カーボンフットプリント計測・削減目標の運用方式、電力消費最適化施策、サステナビリティレポート生成、CSR 部門連携を記述する。要件 ID `BC-SUS-*` と対応する。[../50_非機能方式設計/07_システム環境とエコロジー方式.md](../50_非機能方式設計/) の環境要件と連動する。

## 執筆方針

- 各方式は対応する BC（Business Contract）要件 ID を冒頭に明示する。
- 法務・契約・ガバナンス系は「誰が / 何を根拠に / どの頻度で / 誰に報告するか」の 4 点を必ず明示する。組織運営の明確性は法令準拠の前提である。
- 方式 ID は `DS-BUS-<サブカテゴリ>-<通番>` を用いる（例: `DS-BUS-BIL-001`、`DS-BUS-ONB-001`）。
- 各ファイル冒頭に、対応する 60_事業契約 の節番号と要件 ID を明示し、要件定義者が辿れるようにする。
