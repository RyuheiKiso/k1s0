# 09. サプライチェーン方式

本章は k1s0 のソフトウェアサプライチェーン（SSCS）運用方式を定める。OSS 調達経路の管理、依存スキャン、SLSA レベル到達、イメージ署名、脅威モデル、自動更新、Critical CVE 対応の 7 要素を通じて、サプライチェーン攻撃（典型的には typosquatting、悪意あるメンテナ、サプライヤー漏洩）からプラットフォームを守る構造を作る。

## 本章の位置付け

過去 5 年のサプライチェーン事故（SolarWinds、Codecov、Kaseya、event-stream、colors.js、ctx / phpass 等）は、OSS エコシステムに悪意ある依存を紛れ込ませる攻撃が商用利用者の防御最上位事項となったことを示している。k1s0 は OSS セルフマネージドを前提とするため、調達経路の信頼性が平均的な SaaS 利用者以上に重要である。本章はこれを「調達経路を絞る・全依存を計測する・更新を自動化する・異常を早期検出する」の 4 原則で設計する。

サプライチェーンのリスクは、漏洩時に検出するのではなく、混入前に防ぐ構造設計が有効である。本章は Phase 1c までに基本防御（依存スキャン + Harbor ミラー + Renovate）、Phase 2 で SLSA Level 2（イメージ署名）、Phase 3 で SLSA Level 3（改ざん検知）に到達する段階計画を定める。

## 調達経路

### ミラーリング必須

すべての OSS は Harbor（社内コンテナレジストリ）および社内プロキシ（Artifactory または Nexus、Phase 2 以降）を経由して取得する。GitHub / Maven Central / crates.io / NPM / Docker Hub / GitHub Container Registry などのパブリックレジストリへの直接アクセスは禁止する。

ミラー経由で取得することで以下 5 効果を得る。(1) 上流障害時の継続稼働（GitHub 障害でもビルド継続可能）、(2) 上流削除時の保存継続（left-pad 事件のような消失対策）、(3) 悪意ある変更の遡及検知（ハッシュ比較）、(4) 通信量削減、(5) 社内監査対応（全取得ログを社内保管）。

### パブリックからミラーへの同期

Harbor のレプリケーション機能で定期同期する。同期元は以下 5 種類を主要とする。Docker Hub / ghcr.io / gcr.io / registry.redhat.io / quay.io。同期は毎日深夜に実行し、差分があれば Harbor のストレージに反映する。同期対象の allow list は `infra/harbor/sync-policies.yaml` で管理し、PR レビューで変更する。

crates.io / Maven Central / NPM は Artifactory / Nexus でミラーする（Phase 2 以降）。Phase 1a〜1c は Cargo / Go / NPM のオフラインキャッシュ（`CARGO_HOME` / `GOMODCACHE` / `node_modules`）を CI レベルで活用し、Phase 2 で本格的な社内プロキシに移行する。

### 調達許可リスト

採用可能な OSS の allow list は `infra/compliance/oss-allow-list.yaml` で管理する。allow list にない OSS を新規採用する場合、Product Council 承認を要する。allow list は以下 3 カテゴリで構成する。(1) 採用済み（現在利用中）、(2) 評価中（PoC 段階）、(3) 廃止予定（非推奨化）。

allow list の更新履歴は Git で管理し、追加理由・評価結果・採用判断を各エントリに記載する。これにより新規参画者も OSS 採用経緯を追跡可能となる。

## 依存スキャン

### Grype による脆弱性スキャン

依存 OSS の脆弱性は Grype（`anchore/grype`）でスキャンする。スキャン対象は Harbor の全イメージ、CI ビルド時のアーティファクト、本番稼働中の全 Pod の 3 範囲とする。スキャン頻度は Harbor が毎日、CI が PR ごと、本番稼働が 1 時間ごととする。

Grype のデータソースは NVD（National Vulnerability Database）、GitHub Security Advisories（GHSA）、RubySec、Alpine secdb、Ubuntu USN、RedHat RHSA、SUSE SLE、Oracle ELSA、Amazon Linux の 9 種類を統合する。オフライン運用のため、脆弱性 DB は毎日 Harbor に同期してから使用する。

### 検出時の対応

Grype で Critical（CVSS 9.0 以上）検出時は即座に Slack 通知を発報する。検出されたイメージが本番稼働中なら、影響評価 → パッチ適用計画 → 48 時間以内のリリース、という対応を開始する。影響評価では「k1s0 構成で攻撃経路が成立するか」を判断し、成立しない場合は優先度を下げる。

High / Medium / Low の対応期限は [07_OSSライセンス運用方式.md](07_OSSライセンス運用方式.md) の CVE 対応セクションと共通とする。

### SBOM との連携

Syft（SBOM 生成）と Grype（脆弱性スキャン）は Anchore エコシステムで連携する。Syft で生成した SBOM を Grype が読み込むことで、コンテナイメージの全依存を脆弱性 DB と照合できる。SBOM の生成と脆弱性スキャンは CI の同一ステップで実施し、結果を PR コメントに集約する。

## SLSA Level

### SLSA の背景

SLSA（Supply-chain Levels for Software Artifacts）は Google と OpenSSF が策定したサプライチェーンセキュリティフレームワークである。Level 1〜4 の段階で、ビルドの透明性・改ざん耐性・検証可能性を高める。k1s0 は Phase 2 で Level 2、Phase 3 で Level 3 を目指す。

### Phase 1c: Level 1

Phase 1c 時点で SLSA Level 1 を達成する。Level 1 の要件は「ビルドがスクリプトで自動化されている」「ビルド過程の出所（provenance）が記録されている」の 2 点である。GitHub Actions の workflow ファイル + `actions/attest-build-provenance` で provenance を生成し、ビルドアーティファクトと共に保管する。

### Phase 2: Level 2

Phase 2 で Level 2 を達成する。Level 2 の要件は Level 1 に加えて「ビルドがホストされた環境で実行される」「provenance に署名される」の 2 点である。GitHub Actions の hosted runner を使用し、provenance は Sigstore の Cosign で署名する。Cosign は keyless 署名（OIDC ベース）を採用し、GitHub ID を直接署名主体とする。

### Phase 3: Level 3

Phase 3 で Level 3 を達成する。Level 3 の要件は Level 2 に加えて「ビルドサービスが強化されている」「改ざん耐性が担保される」の 2 点である。GitHub Actions の SLSA Level 3 対応 reusable workflow（slsa-framework/slsa-github-generator）を採用し、ビルド環境の強化（isolation、non-falsifiable provenance）を実現する。

### Level 4 は対象外

Level 4 は「ビルドの再現性（Reproducible Build）」と「双方独立レビュー」を要する。k1s0 の開発規模と運用 2 名体制では Level 4 到達は現実的でないため、Phase 2 以降の優先順位から外す。Level 4 は OpenSSF の事例でも大規模プロジェクト向けとされている。

## イメージ署名

### Cosign 採用

コンテナイメージの署名には Cosign（Sigstore プロジェクト）を採用する。Cosign は Apache 2.0 ライセンスで、keyless 署名（OIDC ベース）と key-based 署名（秘密鍵ベース）の 2 方式を提供する。k1s0 は keyless 署名を基本とし、OIDC は GitHub Actions の ID トークンを使用する。

### 検証

Kubernetes への Pod Deploy 時、Kyverno または OPA Gatekeeper でイメージ署名を検証する。署名のないイメージ・未承認 ID で署名されたイメージはデプロイをブロックする。検証ポリシーは `infra/policies/image-signing.yaml` で管理する。

検証対象は tier1 自作イメージ（最優先）、OSS イメージ（公式イメージの署名を Harbor のミラー時に保存）、テナント独自イメージ（Phase 2 以降、テナント別の署名 ID を管理）の 3 種類とする。

## 脅威モデル

### 主要脅威

サプライチェーン攻撃の主要脅威を以下 5 種類に分類する。(1) Typosquatting: パッケージ名の類似名を悪用（例: `react-dom` → `react-doom`）。(2) Malicious Maintainer: メンテナが意図的に悪意あるコードを挿入（例: colors.js / faker.js の事件）。(3) Compromised Supplier: メンテナのアカウントが侵害され、不正リリースが混入（例: event-stream）。(4) Build Infrastructure Compromise: CI / CD サービスが侵害され、ビルド時にマルウェア注入（例: Codecov）。(5) Dependency Confusion: 社内パッケージ名を公開レジストリに先行登録して誤解読させる。

### 対策

(1) Typosquatting: パッケージ名は allow list と照合、1 文字違いのタイポ検出を CI で実施。(2) Malicious Maintainer: 大規模バージョンアップ時（Major 更新）は手動レビュー必須。変更内容を diff で確認し、リリースノート未記載の変更を警戒。(3) Compromised Supplier: Renovate の自動マージは Patch / Minor のみ。Major はレビュー必須。Signed commits の確認（GitHub の Verified badge）。(4) Build Infrastructure Compromise: GitHub Actions の reusable workflow を固定 SHA でピン留め。SLSA Level 3 到達で強化。(5) Dependency Confusion: 社内パッケージは公開レジストリに登録しない。`.npmrc` / `Cargo.toml` のレジストリ指定を厳格化。

### 脅威モデリングの運用

脅威モデルは四半期ごとにレビューし、新規脅威を検出した場合は対策を追加する。脅威レビューは [../50_非機能方式設計/10_脅威モデリング方式.md](../50_非機能方式設計/10_脅威モデリング方式.md) のセキュリティ脅威レビューと統合運用する。

## Renovate 自動更新

### 運用ポリシー

Renovate（AGPL-3.0、独立運用）で依存パッケージの自動更新を行う。更新頻度は [07_OSSライセンス運用方式.md](07_OSSライセンス運用方式.md) と共通（Patch 毎日、Minor 週次、Major 月次）。

### マージポリシー

Patch / Minor の自動マージは以下 4 条件を満たす場合のみ許容する。(1) CI テスト（Unit + Integration + E2E）成功、(2) ライセンスチェック成功、(3) Grype スキャン成功（新規脆弱性の導入なし）、(4) SBOM 差分が想定内（新規パッケージ追加は手動レビュー）。Major 更新は手動レビューを必須とし、破壊的変更の影響範囲を確認する。

### Renovate 設定

Renovate の設定は `renovate.json` で管理する。主要設定は以下とする。(1) `packageRules` でライセンス別・パッケージ別の更新ポリシーを定義、(2) `schedule` で週次 / 月次の時刻を指定（運用時間外に集中、月曜朝にマージ対象一覧を確認）、(3) `reviewers` で PR レビュアを自動指定、(4) `vulnerabilityAlerts` で脆弱性修正は即時 PR 発行する設定。

## Critical CVE 48h 対応

### 対応プロセス

Critical（CVSS 9.0 以上）の CVE が検出された場合、以下の 48 時間プロセスを起動する。

Hour 0: Grype が検出、Slack 通知発報。
Hour 0〜4: 影響評価（対象 OSS の使用有無、攻撃経路の成立性判定）。
Hour 4〜12: パッチ検討（公式パッチ有無確認、回避策の検討）。
Hour 12〜24: パッチ適用テスト（staging 環境でパッチ版のテスト実施、Regression 確認）。
Hour 24〜36: 本番リリース（承認 → デプロイ → 動作確認）。
Hour 36〜48: Postmortem 作成（原因分析・再発防止策）、Backstage Announcement で公開。

### 48 時間未達時の対応

48 時間以内にパッチ適用が不可能な場合（公式パッチなし、回避策なし）、緊急 Product Council を開催する。意思決定は以下 3 選択肢から行う。(1) 該当 OSS を別 OSS に差し替え、(2) 該当機能を一時停止、(3) リスク受容（文書化した上で暫定運用継続）。いずれを選択しても、Backstage Announcement でテナント代表者に告知する。

### 訓練

Critical CVE 対応の模擬訓練を四半期ごとに実施する。仮想 CVE を Slack に発報し、48 時間プロセスのリハーサルを行う。訓練の結果は Runbook に反映する。

## 対応要件一覧

本章は以下の要件定義 60_事業契約 サプライチェーン系要件と 1:1 対応する。

- BC-SC-001: 調達経路（Harbor / Artifactory ミラー）の強制。対応設計 ID: `DS-BUS-SC-001`。
- BC-SC-002: OSS 調達 allow list の Product Council 管理。対応設計 ID: `DS-BUS-SC-002`。
- BC-SC-003: Grype + SBOM による脆弱性スキャン（毎日 / PR ごと / 1 時間ごと）。対応設計 ID: `DS-BUS-SC-003`。
- BC-SC-004: SLSA Level 到達計画（Phase 1c=L1 / Phase 2=L2 / Phase 3=L3）。対応設計 ID: `DS-BUS-SC-004`。
- BC-SC-005: Cosign によるイメージ署名と Kubernetes 検証（Phase 2）。対応設計 ID: `DS-BUS-SC-005`。
- BC-SC-006: 脅威モデル 5 種類（Typosquatting / Malicious Maintainer / Compromised Supplier / Build Infra / Dependency Confusion）と対策。対応設計 ID: `DS-BUS-SC-006`。
- BC-SC-007: Critical CVE 48 時間対応プロセスと四半期訓練。対応設計 ID: `DS-BUS-SC-007`。
