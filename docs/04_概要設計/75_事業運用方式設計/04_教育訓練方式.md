# 04. 教育訓練方式

本章は k1s0 の全ユーザカテゴリ（tier2 開発者 / tier3 開発者 / 運用者 / 管理者 / 業務部門）向けの教育訓練プログラムを設計する。k1s0 は企画書で「運用 2 名体制」と「tier2 / tier3 開発者の生産性向上」を約束しており、教育訓練の質が両コミットメントの達成可否を決める。

## 本章の位置付け

プラットフォーム製品では、利用者側の習熟が基盤側の機能実装以上に価値を左右する。tier1 API が完璧に整備されていても、tier2 / tier3 開発者がゴールデンパスを知らず独自実装を繰り返せば、コスト削減効果も品質改善効果も消える。同様に、運用者がインシデント対応 Runbook を事前に知らなければ、深夜のトラブルで 2 名体制が破綻する。本章の教育訓練設計は、これらの抽象的な懸念を具体的なカリキュラム・教材・評価方法に落とし込む。

教育訓練の方針は「自習可能を最優先」とする。集合研修は効果が高いが運用コストも高いため、原則は TechDocs・動画・ハンズオン環境による自習を前提とし、集合研修は新人研修とオンコール訓練のみに限定する。自習方式は再利用可能で拡張しやすく、「2 名運用」原則（原則 9）と整合する。

## 対象別カリキュラム

### tier2 開発者向け

tier2 開発者は基盤機能を利用してビジネスロジックサービスを構築する役割である。カリキュラムは以下 6 モジュールとする。

モジュール 1「k1s0 全体像」（1 時間）: 3 層アーキテクチャの意味、tier1 の責務範囲、tier1 公開 API 11 種の俯瞰。企画書の「tier1 は tier2 を楽にするため」の思想を身に着ける。モジュール 2「ゴールデンパス入門」（2 時間）: Backstage Scaffolder で tier2 サービス雛形を生成し、tier1 公開 API を呼び出してデプロイするまでを実機体験する。Phase 1c 以降、tier2 新規参画者の必須受講とする。

モジュール 3「tier1 公開 API 詳細」（3 時間）: State / PubSub / Secrets / Binding / Workflow / Log / Telemetry / Decision / Audit-Pii / Feature / Service Invoke の 11 API をそれぞれ 15 分単位で解説する。動画 + ハンズオン演習の組合せとする。モジュール 4「エラーハンドリングと冪等性」（2 時間）: tier1 API の共通エラーコード、リトライ戦略、Saga / 補償トランザクションの考え方を学習する。[../40_制御方式設計/](../40_制御方式設計/) と連動する。

モジュール 5「観測と運用」（2 時間）: Grafana ダッシュボードの見方、OTel ログ / メトリクス / トレースの使い分け、アラート対応の基本。モジュール 6「Feature Management と段階公開」（1 時間）: flagd 経由での Feature Flag 利用、A/B テスト、カナリア公開の実践。tier2 開発者は全 6 モジュールを通算 11 時間で修了可能とする。

### tier3 開発者向け

tier3 開発者はエンドユーザ向けアプリを構築する役割である。カリキュラムは以下 4 モジュールとする。

モジュール 1「tier3 全体像と tier2 接続」（1 時間）: tier3 がどのように tier2 を呼び出すか、tier1 を直接呼ばない理由を理解する。モジュール 2「アプリ配信ポータル活用」（2 時間）: ClickOnce / MSIX / 単体 EXE / Web アプリの 4 形態の配信手順を実機で体験する。

モジュール 3「Feature Management」（1 時間）: tier3 アプリから flagd 経由で Feature Flag を取得する方法を学ぶ。モジュール 4「既存 .NET Framework 資産との共存」（2 時間）: ClickOnce / MSIX ラップ方式を実機で体験する。tier3 開発者は 4 モジュールを通算 6 時間で修了可能とする。

### 運用者向け

運用者は Phase 1c 以降、2 名体制で k1s0 全体を運用する役割である。カリキュラムは以下 5 モジュールとする。

モジュール 1「Runbook 入門」（2 時間）: Runbook の構造、起動方法、典型的インシデント 5 種類（Dapr sidecar 障害 / Kafka 遅延 / PostgreSQL 障害 / 認証障害 / 証明書期限切れ）の対処を学ぶ。モジュール 2「Grafana と Loki でのログ調査」（2 時間）: 障害時のログフィルタ、トレース ID での相関、典型的クエリの練習。

モジュール 3「バックアップ・復元」（3 時間）: PostgreSQL / Kafka / Valkey / OpenBao のバックアップ取得と復元を実機で練習する。モジュール 4「DR 演習」（4 時間）: 四半期の DR 演習に参加し、リージョン障害からの復旧手順を実施する。モジュール 5「セキュリティ運用」（2 時間）: Keycloak の管理、OpenBao のシークレットローテーション、Istio の認可ポリシー更新。運用者は 5 モジュールを通算 13 時間で修了可能とする。

### 管理者向け

管理者は情シス部門責任者・経営層を含む。カリキュラムは以下 3 モジュールとする。

モジュール 1「k1s0 経営視点サマリ」（30 分）: KPI ダッシュボード、SLO 達成率、コスト推移、稟議効果の読み方。モジュール 2「ガバナンス運用」（1 時間）: Product Council の役割、意思決定プロセス、ADR 承認の考え方。[10_ガバナンス運用方式.md](10_ガバナンス運用方式.md) と連動する。

モジュール 3「コンプライアンス対応」（1 時間）: J-SOX / 個人情報保護法 / 電帳法の k1s0 上での対応状況、監査対応のフロー。管理者は 3 モジュールを通算 2.5 時間で修了可能とする。

### 業務部門向け

業務部門は JDM エディタでルール編集するユーザを含む。カリキュラムは以下 2 モジュールとする。モジュール 1「JDM エディタ操作入門」（2 時間）: Decision テーブル・関数式・グラフビューの使い分け、ルール作成・テスト・公開。モジュール 2「運用ガバナンス」（1 時間）: ルール変更のワークフロー、承認者、影響範囲確認。業務部門は 2 モジュールを通算 3 時間で修了可能とする。

## コンテンツ形式

### TechDocs

Backstage TechDocs が一次ソースとなる。Markdown で記述し、コード例は実行可能な形式とする。TechDocs の構成は「チュートリアル（step-by-step）」「How-To ガイド（特定タスク解決）」「リファレンス（API / 設定）」「説明（概念・背景）」の Diátaxis 4 象限に整理する。各モジュールは Diátaxis 4 象限のいずれかに分類し、章タイトルで明示する。

### 動画

動画は Phase 2 以降で順次整備する。1 動画あたり 5〜10 分に分割し、章単位で再生リストにまとめる。撮影には OBS Studio を使用し、再生ホストは社内ビデオ配信ポータル（または YouTube 非公開）とする。字幕（日本語・英語 Phase 2）を全動画に付与する。動画は新人の初回学習と継続研修の復習に活用する。

### ハンズオン環境

ハンズオン環境は Phase 1c で Kubernetes クラスタ（kind または k3d）に k1s0 最小構成を立ち上げるスクリプトを提供する。ローカル開発環境は [../70_開発者体験方式設計/02_ローカル開発環境方式.md](../70_開発者体験方式設計/02_ローカル開発環境方式.md) で定義する。ハンズオン演習は TechDocs のチュートリアルに埋め込む形式とし、1 演習 30 分以内で完走可能とする。

## 認定制度（Phase 2）

Phase 2 以降、任意参加の認定制度を導入する。認定は以下 3 段階とする。

レベル 1「k1s0 基礎」: tier2 / tier3 共通の基礎知識を問う。選択式 20 問のオンラインテスト、合格ライン 80% 以上。レベル 2「k1s0 プラクティショナ」: 対象別カリキュラム全モジュール修了、実機ハンズオンで雛形サービスを作成して合格する。レベル 3「k1s0 エキスパート」: 運用者向けで、インシデント対応シミュレーションに合格する。

認定はスキルマップ可視化とキャリアパス設計に活用する。強制ではないが、Phase 2 以降は運用者ロール就任時にレベル 2 以上を必須とする。

## オンコール訓練

オンコール訓練は運用 2 名体制の維持に直結する。四半期ごとに以下 2 種類の訓練を実施する。

シミュレータ訓練（2 時間）: 仮想インシデント（Dapr sidecar 障害 / PostgreSQL レプリケーション遅延 / Kafka コンシューマ停止 / 証明書期限切れ等）を発生させ、運用者が Runbook に従って対処する演習。訓練後は所要時間・誤操作有無・改善点を振り返る。DR 演習（4 時間）: リージョン障害からの全面復旧を実機で行う。Phase 2 以降、マルチリージョン構成で年 1 回実施する。

訓練結果は月次 Product Council で報告し、改善点は Runbook の更新に反映する。訓練未参加者には代替日程を提供し、年間 4 回全参加を運用者の必須要件とする。

## 新人研修

新規参画者向けに 2 週間のブートキャンプを実施する。ブートキャンプは以下 10 日構成とする。

Day 1: k1s0 全体像・企画書概要。Day 2: Rust 入門（tier1 開発者向け）または Go 入門（Dapr ファサード開発者向け）。Day 3〜5: tier1 公開 API ハンズオン（11 API を 3 日に分割）。Day 6〜7: Backstage / Grafana / JDM エディタ / 配信ポータルの実機体験。Day 8: Runbook による障害対応演習。Day 9: コードレビュー実習（先輩ペアで 2 件のレビュー）。Day 10: 小規模タスクへのアサインと初デプロイ。

ブートキャンプ終了後、参画者は 30 日間のメンター伴走期間（先輩 1 名が週 3 時間のレビュー）に入る。ブートキャンプとメンター伴走により、新規参画者が 2 週間 + 30 日でキャッチアップ可能な状態を作る（原則 9「バス係数 2」の直接的実現）。

## Rust 入門教材（Phase 0 整備）

構想設計で Rust 採用の潜在リスク（DR-TECH-004: Rust 人材不足）を識別した。このリスクを緩和するため、Phase 0 段階で Rust 入門教材を整備する。教材は以下 3 部構成とする。

第 1 部「Rust 言語基礎」（10 時間）: 所有権・借用・ライフタイム・エラーハンドリング・トレイトを、Rust 公式 Book のチュートリアル相当の範囲で学習する。第 2 部「k1s0 で使う Rust パターン」（5 時間）: tier1 で使用する典型パターン（tokio 非同期 / tonic gRPC / anyhow エラー / tracing ログ）をコード例と共に解説する。第 3 部「Rust 実習」（10 時間）: tier1 の小規模改修（公開 API のバリデーション追加等）を題材とした実習を 3 題実施する。

教材は社内限定で公開し、社外向けには抽象化したバージョンを Phase 2 以降で公開する。教材の継続的更新は Rust のバージョンアップに合わせ四半期ごとに実施する。

## 学習時間予算

学習時間は業務時間内に確保する。tier2 / tier3 / 運用者は月 3 日（24 時間）を学習時間予算として割り当て、管理者承認で使用可能とする。学習時間の利用実績は [12_分析とレポーティング方式.md](12_分析とレポーティング方式.md) の KPI ダッシュボードで可視化する。

学習時間の内訳は、新規モジュール受講・既存モジュール復習・社外勉強会参加・OSS コントリビュート・読書・認定取得準備、の 6 カテゴリとする。強制は行わないが、年間 24 日（288 時間）の未使用時間が発生した場合は管理者が個別相談する運用とする。

## 対応要件一覧

本章は以下の要件定義 60_事業契約 教育訓練系要件と 1:1 対応する。

- BC-TRN-001: 対象別カリキュラム（tier2 / tier3 / 運用 / 管理 / 業務）の整備。対応設計 ID: `DS-BUS-TRN-001`。
- BC-TRN-002: コンテンツ形式（TechDocs / 動画 / ハンズオン環境）の整備。対応設計 ID: `DS-BUS-TRN-002`。
- BC-TRN-003: 認定制度（Phase 2、3 段階）の運用。対応設計 ID: `DS-BUS-TRN-003`。
- BC-TRN-004: オンコール訓練（四半期、シミュレータ + DR 演習）の運用。対応設計 ID: `DS-BUS-TRN-004`。
- BC-TRN-005: 新人研修（2 週間ブートキャンプ + 30 日メンター）の運用。対応設計 ID: `DS-BUS-TRN-005`。
- BC-TRN-006: Rust 入門教材（Phase 0 整備、DR-TECH-004 緩和）と学習時間予算（3 日/月/人）。対応設計 ID: `DS-BUS-TRN-006`。
