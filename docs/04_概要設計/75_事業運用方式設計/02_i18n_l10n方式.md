# 02. i18n / l10n 方式

本章は k1s0 の多言語化（i18n）と地域化（l10n）の方針、対応言語と段階、タイムゾーン・数値・日付・通貨の書式、翻訳運用プロセス、将来拡張を定める。k1s0 は 採用側組織の情シス向けとして初期は日本国内単一言語を想定するが、企画書で約束した「グローバル展開の拡張余地を設計段階で担保する」を画面・バックエンド双方で実現する必要がある。

## 本章の位置付け

多言語対応を「後から追加する」前提で設計すると、文字列ハードコード、日付書式のローカル依存、タイムゾーン混在、数値書式の前後置空白など、数百箇所の修正が必要になる。k1s0 は リリース時点 段階から i18n 対応可能な構造を採用し、l10n（実際の翻訳物の投入）を段階的に拡張することで、追加コストを段階的に分散させる。本章は「いま翻訳しないが翻訳可能な状態」を維持する設計方針を固定する。

対応言語と書式は経営判断事項であるため、本章で確定する言語セットは構想設計 ADR への変更提案と採用再検討を経て更新する。採用検討時点では日本語のみ実投入、採用後の運用拡大時 で英語、採用側のマルチクラスタ移行時 で中文（繁体 / 簡体）、採用側の全社展開期 以降で RTL（アラビア語 / ヘブライ語）を拡張候補とする。

## i18n 基本方針

### メッセージカタログ分離

すべてのユーザ向け文字列はメッセージカタログファイルに分離し、ソースコードに直接記述することを禁止する。フロントエンドは ICU MessageFormat（`intl-messageformat` ライブラリ）を採用し、バックエンド（Go / Rust）は OS 標準の gettext ライクな wrapper（Go は `go-i18n`、Rust は `fluent-rs` または `rust-i18n`）を使用する。

カタログファイルの正本は JSON 形式で `locales/<lang>/<namespace>.json` に配置する。名前空間は画面単位（`catalog.json` / `techdocs.json` / `scaffolder.json` / `jdm-editor.json` / `delivery-portal.json` 等）で分割し、1 ファイル 300 行（キー数 100 程度）を超えた場合はさらに細分化する。

### 言語決定ロジック

ユーザの表示言語は以下の優先順位で決定する。(1) ユーザプロファイルで明示設定した言語（Keycloak 属性 `x-k1s0-locale`）、(2) ブラウザの `Accept-Language` ヘッダの最上位言語、(3) テナントのデフォルト言語（Backstage テナント設定）、(4) k1s0 システム全体のデフォルト（日本語）。優先順位の評価は tier1 の API ゲートウェイ（Envoy Gateway）で実施し、処理されたロケール情報を HTTP ヘッダ `x-k1s0-locale` として下流サービスへ渡す。

バックエンドサービス間の gRPC 通信でもロケール情報を Metadata で引き回す。エラーメッセージをレスポンスに含める際、ロケールに応じた翻訳文字列を返却する。ログメッセージは原則英語（運用チーム共通語）で記録し、ユーザ向けレスポンスのみ翻訳する。

## 対応言語と展開段階

### リリース時点〜採用初期: 日本語単一

採用検討から リリース時点 までは日本語単一言語で運用する。ただし、メッセージカタログの分離・ICU MessageFormat の採用・`Accept-Language` 引き回しの 3 点を リリース時点 時点から実装する。これにより 採用後の運用拡大時 で英語を追加する際の追加コストは翻訳作業とレビューのみとなり、アプリケーションコードの改変は不要となる。

リリース時点 までに整備する文字列は日本語のみだが、キーは英語で命名する（例: `tenant.onboarding.step1.title`）。日本語キーは翻訳管理ツールでの検索性を下げるため禁止する。

### 採用後の運用拡大時: 英語追加

採用後の運用拡大時 で英語翻訳を投入する。英語投入の対象は tier1 公開 API のエラーメッセージ、Backstage 共通画面、Grafana ダッシュボード標題、配信ポータル主要画面、TechDocs の主要ガイドの 5 範囲とする。JDM エディタは業務部門専用のため 採用後の運用拡大時 では日本語のみ、採用側のマルチクラスタ移行時 で英語化する。

英語の地域区分は `en-US` を正本とし、`en-GB` / `en-AU` / `en-IN` 等は `en-US` のフォールバック運用とする。日付書式と通貨のみ地域別に切り替える（例: `en-GB` は `dd/mm/yyyy`、`en-US` は `mm/dd/yyyy`）。

英語翻訳の投入コストは 採用後の運用拡大時 で約 150〜300 万円を見込む。試算根拠は外部翻訳業者の IT 分野相場（日英翻訳 1 ワード 8〜15 円、採用側組織A / Common Sense Advisory 2024 年調査）に対し、翻訳対象語数を tier2 / tier3 UI と主要利用者ガイドで 15,000〜25,000 ワードと見積もった結果である（Backstage 共通画面 + Grafana 標題 + 配信ポータル主要画面 + TechDocs 主要ガイド + tier1 公開 API エラーメッセージ）。ネイティブレビュー工数と Weblate 運用工数（社内 0.2 人月）を含めても年 300 万円が上限であり、企画書で約束した「年 3,280 万円削減」の 10% 未満に収まる。削減額の 5〜10% を翻訳投資に再循環する想定は、採用検討コミットメントの再算を必要としない水準である。

### 採用側のマルチクラスタ移行時: 中文追加

採用側のマルチクラスタ移行時 で中文翻訳を投入する。中文は `zh-TW`（繁体）と `zh-CN`（簡体）を独立言語として扱い、一方から他方への自動変換に依存しない（政治的・文化的理由で誤訳のリスクが高いため）。中文投入時には現地パートナー企業とのレビュー契約を締結する。

中文圏は通貨表記（元 / 人民元 / ドル）と数値区切り記号が混在するため、地域別設定を細かく持つ。タイムゾーンは `Asia/Taipei` / `Asia/Shanghai` を別扱いする。

中文翻訳の投入コストは 採用側のマルチクラスタ移行時 で約 100〜200 万円を見込む。試算根拠は日中翻訳の一般相場（1 ワード 8〜12 円、日本翻訳連盟 2024 年目安単価）に対し、採用後の運用拡大時 で確定した英訳 15,000〜25,000 ワードを日本語 → 繁体中文 + 日本語 → 簡体中文の 2 パスで翻訳する構成であり、翻訳メモリ流用で 1 ワードあたり実コストを 50% 圧縮できる前提に基づく。繁体 / 簡体を独立翻訳するため素の費用は英訳の 2 倍相当だが、採用後の運用拡大時 で整備した用語集と翻訳メモリの再利用で 100〜200 万円に収まる。現地パートナーのネイティブレビュー費（繁体 / 簡体で各 30 万円前後）を含めた総額も、英訳と合算した累計翻訳投資は 500 万円以下であり「年 3,280 万円削減」の 15% 以内に維持される。

### 採用側の全社展開期 以降: RTL 対応

採用側の全社展開期 以降でアラビア語（`ar`）・ヘブライ語（`he`）の RTL（Right-to-Left）対応を検討する。RTL 対応は CSS の論理プロパティ（`margin-inline-start` 等）への全面置換、UI コンポーネントの左右反転、数値 / 日付の BiDi 対応を要する。リリース時点 段階から CSS 論理プロパティを優先的に採用することで、採用側の全社展開期 での追加コストを削減する。

RTL 投入の時期と対応言語は市場状況に応じて判断する。採用検討時点では骨子のみとし、実装判断は 採用側のマルチクラスタ移行時 完了時の経営判断とする。

## タイムゾーン方式

### 内部表現

すべての時刻は UTC で保存する。データベース列型は `TIMESTAMP WITH TIME ZONE` を標準とし、時刻を保存する列には例外なく UTC での保存を強制する。Kafka メッセージ・監査ログ・トレースイベントはすべて UTC の ISO 8601 形式（`2026-04-12T09:30:00Z`）で記録する。

tier1 API の gRPC メッセージで時刻を扱う場合、Protobuf の `google.protobuf.Timestamp` を使用する。`Timestamp` はエポックからのナノ秒オフセットで内部的に UTC 等価となるため、タイムゾーン情報は transport で消失しない。クライアント側での UTC 変換は SDK が自動実施する。

### 表示変換

画面表示時のタイムゾーン変換は Edge（ブラウザ・tier3 クライアント）で実施する。サーバ側は UTC を返却し、クライアント側が `Intl.DateTimeFormat` でロケールとユーザ設定タイムゾーンに応じて表示変換する。ユーザ設定タイムゾーンは Keycloak 属性 `x-k1s0-timezone` に保持し、未設定時はブラウザの `Intl.DateTimeFormat().resolvedOptions().timeZone` を使用する。

日本国内ユーザの大半は `Asia/Tokyo`（JST, UTC+9）となる。監査ログ・インシデント対応で複数ユーザ間のタイムライン共有が必要な場合は UTC 表示に切り替えるオプションを提供する（Grafana のタイムゾーン切替と同様）。

### サマータイム

日本はサマータイム非採用だが、英語圏（`en-US` / `en-GB`）はサマータイム採用地域を含む。サマータイム切替日をまたぐ時刻計算は IANA タイムゾーンデータベース（tzdata）に委ね、サマータイム開始 / 終了の 2 時間内のイベントは UTC での時系列順序を正とする。k1s0 内部実装で日付加減算を行う場合は Go `time` パッケージ・Rust `chrono` / `chrono-tz` のタイムゾーン対応関数を使用し、手動オフセット計算は禁止する。

## 書式方式

### 数値書式

数値書式はロケールに応じて区切り記号と小数点記号を切り替える。日本語（`ja-JP`）は 3 桁区切りカンマ `,`、小数点ピリオド `.`。英語 US（`en-US`）は同じ `,` / `.`。ドイツ語（`de-DE`、採用側の全社展開期 候補）は 3 桁区切りピリオド `.`、小数点カンマ `,`。中文（`zh-TW` / `zh-CN`）は `,` / `.`。実装は `Intl.NumberFormat` を使用し、手動フォーマットは禁止する。

大きな数値（金額・件数）の読みやすさ確保のため、`1,000,000` 単位の表示を推奨する。日本語圏では加えて万円単位表示（`100万円` / `1,000万円`）を併記する。企画書で使用する「年額 2,349 万円」のような表記は日本語 UI 専用とし、英語 UI では `¥23.49M JPY` 形式で表記する。

### 日付書式

日付書式は ISO 8601（`yyyy-mm-dd`）を正本とし、ロケール表示時に変換する。日本語は `yyyy年m月d日`（例: `2026年4月12日`）、英語 US は `mm/dd/yyyy`（例: `04/12/2026`）、英語 GB は `dd/mm/yyyy`（例: `12/04/2026`）、ドイツ語は `dd.mm.yyyy`（例: `12.04.2026`）、中文は `yyyy年m月d日`（日本語と同じ表記慣習）とする。

日時の組合せは ISO 8601 拡張形式 `yyyy-mm-ddThh:mm:ssZ` を内部表現とし、ロケール表示時に `yyyy年m月d日 hh:mm`（日本語）等へ変換する。ログとモニタリング系の表示は国際的一貫性を優先し ISO 8601 のまま表示する（運用者の混乱を避けるため）。

### 通貨書式

通貨は ISO 4217 コード（`JPY` / `USD` / `TWD` / `CNY` / `EUR`）を内部表現とする。表示時は `Intl.NumberFormat` の通貨モードで現地記号に変換する（`JPY` → `¥`、`USD` → `$`、`CNY` → `CN¥` or `¥`）。通貨は小数点以下の桁数がロケールで異なるため（JPY は 0 桁、USD は 2 桁）、自動判定に委ねる。

課金メータリング（[06_課金メータリング方式.md](06_課金メータリング方式.md)）の請求金額は JPY 単一通貨で内部保管し、表示時のみ他通貨変換する。為替レートの参照元は日銀の公示レート（月末最終営業日）を基本とし、採用後の運用拡大時で他通貨請求対応時に確定する。

## 翻訳運用

### 翻訳管理ツール

翻訳管理には OSS の Weblate を採用する。Weblate は AGPL-3.0 だが、原則 8 に基づき独立運用（tier1 コードから切り離して翻訳者用ポータルとして運用）するため 5 原則に抵触しない。採用後の運用拡大時 で Weblate を立ち上げ、`locales/` ディレクトリを Git リポジトリとして Weblate に接続する。

Weblate は翻訳者向けの UI、翻訳メモリ（既存翻訳の再利用）、用語集、翻訳レビュー、翻訳完了率の可視化を提供する。翻訳は外部翻訳会社（専門業者）と社内バイリンガル人材の併用で進める。翻訳完了後は自動 PR が発行され、開発チームがコードレビューした上でマージする。

### 翻訳メモリと用語集

翻訳メモリはドメイン別に分離する。tier1 技術用語（State / PubSub / Secrets / Workflow 等）、tier2 業務用語（稟議 / 申請書 / 電帳法 / J-SOX 等）、tier3 アプリ固有用語の 3 ドメインを独立管理する。用語集は特に「k1s0 公式訳語」を固定する（例: `tenant` → `テナント`、`saga` → `Saga`、`circuit breaker` → `サーキットブレーカー`）。

翻訳が揺れやすい用語（`authentication` / `authorization` / `validation` 等）は用語集に強制的に登録し、Weblate の整合性チェックで翻訳揺れを検出する。採用後の運用拡大時 の英語投入前に用語集を 200 語程度まで整備する。

### 翻訳レビュー

翻訳後は必ずネイティブスピーカーによるレビューを経る。英語は外部翻訳会社のネイティブレビュー、中文は現地パートナーのネイティブレビュー、日本語は社内日本語ネイティブのレビューを経る。レビュー結果は Weblate 上でコメントとして記録し、次回翻訳時の参考とする。

翻訳の品質指標は「翻訳完了率（目標 95% 以上）」「翻訳レビュー完了率（目標 100%）」「誤訳発生率（目標 1% 未満）」「翻訳者満足度（NPS +20 以上）」の 4 指標で監視する。これらは [12_分析とレポーティング方式.md](12_分析とレポーティング方式.md) の KPI ダッシュボードに組み込む。

## 対応要件一覧

本章は以下の要件定義 60_事業契約 i18n / l10n 系要件と 1:1 対応する。

- BC-I18N-001: メッセージカタログ分離と ICU MessageFormat / go-i18n / fluent-rs の採用。対応設計 ID: `DS-BUS-I18N-001`。
- BC-I18N-002: 対応言語と段階（リリース時点 日本語 / 採用後の運用拡大時 英語 / 採用側のマルチクラスタ移行時 中文 / 採用側の全社展開期 RTL 候補）。対応設計 ID: `DS-BUS-I18N-002`。
- BC-I18N-003: タイムゾーン方式（UTC 内部保存 / Edge 表示変換 / tzdata 参照）。対応設計 ID: `DS-BUS-I18N-003`。
- BC-I18N-004: 数値 / 日付 / 通貨書式のロケール切替（Intl.NumberFormat / Intl.DateTimeFormat）。対応設計 ID: `DS-BUS-I18N-004`。
- BC-I18N-005: 翻訳管理ツール Weblate の運用（採用後の運用拡大時）。対応設計 ID: `DS-BUS-I18N-005`。
- BC-I18N-006: 翻訳メモリ・用語集・翻訳レビュー運用。対応設計 ID: `DS-BUS-I18N-006`。
