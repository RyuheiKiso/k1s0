# 08. 法務契約運用方式

本章は k1s0 のユーザ規約、サービス契約、秘密保持契約、個人情報委託契約、SLA、知財帰属、法務レビューの運用方針を定める。企画書で約束した「JTC 情シス向けとして法的リスクを最小化する」を契約・法務オペレーションで実現する。

## 本章の位置付け

プラットフォーム製品は多数のテナントと同時に契約関係を結ぶため、契約管理が属人的になると契約内容の齟齬・更新漏れ・紛争時の根拠不在という 3 種の事故を招く。とりわけ JTC 情シス向けでは、契約文言の一貫性と監査対応可能性が企業信用に直結する。本章は契約テンプレートの標準化・法務レビューの定期化・SLA 違反時のペナルティ自動執行を通じて、これらのリスクを構造的に削減する。

本章は法務専門性を要する領域のため、契約文言そのものの定義は社内法務部門および外部法律事務所との協業で実施する。本章の設計スコープは「どの契約をいつ結ぶか」「どの場面でどの文言を適用するか」「契約違反時に何が起きるか」の運用設計に限定する。

## 契約体系

### 契約階層

k1s0 のテナント向け契約は以下 5 階層で構成する。(1) 基本利用契約（Master Service Agreement、MSA）、(2) サービスレベル合意書（SLA）、(3) データ処理契約（Data Processing Agreement、DPA）、(4) 秘密保持契約（Non-Disclosure Agreement、NDA）、(5) 個別条件合意書（Order Form、オプション機能利用時）。

契約階層の適用順は (1) → (2) → (3) → (4) → (5) とし、上位契約と下位契約の文言が矛盾する場合は下位を優先する。ただし、データ保護・個人情報関連の条項は DPA が最優先とする（法的要求のため）。

### MSA（基本利用契約）

MSA はテナント新規オンボーディング時に締結する。主要条項は以下 10 項目とする。(1) 利用可能なサービスの定義、(2) テナントの権利と義務（API 使用制限・禁止行為）、(3) 情シス部門の権利と義務（SLA 保証・通知義務）、(4) 料金と支払条件（Phase 2 以降）、(5) 契約期間と更新（初期 1 年、自動更新）、(6) 解約条件（90 日前通知）、(7) 知財帰属（後述）、(8) 免責事項（間接損害の排除）、(9) 準拠法（日本国法）、(10) 紛争解決（東京地方裁判所を管轄）。

MSA テンプレートは社内法務部門で初版を作成し、外部法律事務所のレビューを経て Product Council 承認後に運用開始する。更新は年次レビューで行い、法改正・市場慣行変化に応じて改定する。

### SLA

SLA は MSA の附属書として締結する。SLA の数値コミットは制約 6 に基づき、以下を基本形とする。

稼働率 99%（月 7.2 時間の停止許容）、SLO 99.9%（月 43 分のエラーバジェット）、tier1 API p99 レイテンシ 500ms 以内、RPO 24 時間以内（バックエンド別）、RTO 4 時間以内。SLA 違反時のペナルティは、稼働率 99% 未達時に当月利用料の 10% 減額（Phase 2 以降、請求開始時）、99.5% 未達時に 25% 減額、99.0% 未達時に 50% 減額の 3 段階とする。

### DPA（データ処理契約）

DPA は個人情報を扱うテナントと締結する。個人情報保護法 2022 年改正版（個情法改正）および GDPR（国際対応時）に準拠する。主要条項は以下 8 項目とする。(1) 処理目的の制限、(2) 処理者の義務（安全管理措置、事故時通知）、(3) 再委託の条件（事前同意・同等義務の契約）、(4) データ主体の権利行使支援、(5) 国外移転の制限、(6) 保管期間と削除、(7) 監査権、(8) 契約終了時のデータ返却または削除。

DPA は個人情報を扱わないテナントでは締結不要とする。ただし、将来個人情報を扱う可能性がある場合は「オプトイン締結」を推奨し、予め準備しておく。

### NDA

NDA は tier2 / tier3 開発者が k1s0 内部設計情報にアクセスする際に締結する。k1s0 の内部実装詳細（Dapr 設定・暗号鍵管理手法・監査改ざん防止アルゴリズム等）は機密とし、第三者開示を禁止する。NDA 違反時の損害賠償額は個別交渉とするが、最低 500 万円の違約金条項を含める。

NDA の有効期間は契約終了後 3 年とする。対象情報は「書面で秘密指定されたもの」「合理的に秘密と認識できるもの」の 2 種類とし、公知情報・第三者から正当に取得した情報・独自開発情報は除外する。

### Order Form

Order Form はオプション機能利用時（高容量 State、大規模 Workflow、専用 Kafka パーティション等）に締結する。MSA の標準条件を超える利用に対して、追加料金と追加条件を規定する。Phase 2 以降で整備する。

## 知財帰属

### 職務著作

k1s0 の開発成果物は情シス部門および自社所属 tier2 / tier3 開発者の職務著作として、原則として法人（当社）に帰属する。職務著作の条件は著作権法第 15 条に従い、(1) 法人の発意に基づく、(2) 法人の業務に従事する者が作成、(3) 職務上の作成、(4) 法人名義での公表、(5) 契約 / 勤務規則に別段の定めがない、の 5 条件を満たす。

外部委託（外部ベンダへの開発委託）時は、成果物の著作権が法人に帰属する旨を明記した業務委託契約を締結する。外部ベンダの職務発明規程が不明確な場合、契約時に著作権譲渡の明文化を要求する。

### OSS 利用時の帰属

採用 OSS は元の著作者に帰属する。k1s0 の独自改変部分は法人に帰属するが、元の OSS ライセンス（Apache 2.0 / MIT / AGPL 等）の制約に従う。AGPL OSS の改変は原則禁止（ADR-0003 の原則 3）のため、改変分の著作権問題は発生しない。

### 特許

k1s0 の技術発明は職務発明として法人が承継する。特許出願の可否は知財部門の判断とし、発明者には就業規則に基づく相当の対価を支払う。OSS 由来のアイディアは特許出願対象外とする（先行技術の可能性が高く、特許無効リスクがあるため）。

## 秘密保持

### 秘密情報の分類

k1s0 における秘密情報は以下 4 分類とする。

**機密（Top Secret）**: 暗号鍵・ルート証明書・OpenBao マスター鍵。アクセス権は運用責任者および情シス部門責任者の 2 名に限定。
**秘（Confidential）**: テナント別個人情報、監査ログ、Internal API 仕様詳細。アクセス権は NDA 締結済み関係者に限定。
**社外秘（Internal）**: 設計書、Runbook、ソースコード。アクセス権は全社員。
**公開（Public）**: 公開 API 仕様、企画書、README。誰でも閲覧可能。

### アクセス制御

各分類に応じて Keycloak の Role と OpenBao の Policy を定義する。Top Secret はハードウェアトークン（YubiKey 等）を要する多要素認証、Confidential はパスワード + TOTP、Internal はパスワードのみ、Public は未認証アクセスを許容する。

アクセス履歴は監査ログに記録し、[../50_非機能方式設計/08_データ保護とプライバシー方式.md](../50_非機能方式設計/08_データ保護とプライバシー方式.md) の改ざん防止機構で保管する。

## SLA 違反対応

### SLO 違反検知

SLO 違反は Grafana アラートで自動検知する。違反検知は以下 3 段階で判定する。(1) リアルタイム検知（過去 5 分間のエラー率 / レイテンシが閾値超過）、(2) 日次集計（前日 24 時間の SLO 達成率）、(3) 月次集計（月次 SLO 達成率）。月次集計で SLA コミット（稼働率 99% / SLO 99.9%）を下回った場合、ペナルティ発動の対象となる。

### ペナルティ執行

Phase 2 以降の請求書発行時、前月の SLO 達成率に基づきペナルティを自動計算する。ペナルティは請求額からの減額として請求書に明示する（「SLA 違反による減額: ¥123,456」の形式）。ペナルティ額は Product Council 承認を要する（経理監査の観点）。

ペナルティの上限は当月利用料の 50% とする。それを超える損害賠償はテナント側からの個別請求とし、因果関係・損害額の立証責任はテナント側にある。

### インシデント報告

SLA 違反発生時、24 時間以内にテナント代表者に初報を送付する。初報には発生時刻・影響範囲・暫定対応状況を記載する。7 日以内に Postmortem（原因分析・再発防止策）を発行する。Postmortem は Backstage Announcement で公開する（機密情報を除く）。

## 法務レビュー運用

### レビュー頻度

法務レビューは以下の頻度で実施する。契約テンプレート（MSA / SLA / DPA / NDA / Order Form）の年次レビュー、法改正の適宜レビュー（改正施行前 90 日まで）、新規 AGPL 採用時の個別レビュー（[07_OSSライセンス運用方式.md](07_OSSライセンス運用方式.md)）、テナントからの契約変更要求時の個別レビュー。

### 法務担当の指定

法務レビュー担当は社内法務部門で指定する。通常案件は社内法務、複雑案件（海外テナント対応・新規法域対応・訴訟案件）は外部法律事務所を活用する。外部法律事務所との契約は年次レビューで見直す。

### 記録保管

法務判断の記録はすべて ADR として Git 管理する。ADR は公開可能な粒度で記述し、機密情報（テナント名・個別金額等）は別途管理台帳に記録する。管理台帳は OpenBao の Secrets として保管し、法務責任者のみアクセス可能とする。

## 契約更新と終了

### 年次更新

MSA は 1 年単位で自動更新する。更新 60 日前にテナント代表者に通知し、変更要求があれば 30 日間の交渉期間を設ける。条件合意後、電子署名で更新契約書を発行する。

料金改定が発生する場合は 90 日前告知を必須とする。改定内容に同意しないテナントは解約可能とする（解約時は 60 日前通知に短縮）。

### 契約終了

テナント解約時は [03_テナントオンボーディング方式.md](03_テナントオンボーディング方式.md) の 90 日廃止フローに従う。廃止期間中の料金は発生しないが、アーカイブ保管料金（Phase 2 以降で設定）のみ別途請求する。

契約終了後のデータは 90 日保管、その後法定保管期限（7 年）までアーカイブ保管、7 年経過後に完全削除する。削除時はテナント代表者に通知し、削除証明書を発行する。

- BC-LGL-002: SLA 数値コミット（稼働率 99% / SLO 99.9% / p99 500ms / RPO 24h / RTO 4h）。対応設計 ID: `DS-BUS-LGL-002`。
- BC-LGL-003: 知財帰属（職務著作・OSS ライセンス・特許）の明文化。対応設計 ID: `DS-BUS-LGL-003`。
- BC-LGL-004: 秘密情報の 4 分類とアクセス制御。対応設計 ID: `DS-BUS-LGL-004`。
- BC-LGL-005: SLA 違反時のペナルティ自動執行（当月利用料 10% / 25% / 50% 減額）。対応設計 ID: `DS-BUS-LGL-005`。
- BC-LGL-006: 法務レビュー運用（年次 / 法改正時 / 個別案件）と ADR 記録。対応設計 ID: `DS-BUS-LGL-006`。
- BC-LGL-007: 契約更新（年次自動更新）と終了（90 日廃止 + 7 年アーカイブ）の手続。対応設計 ID: `DS-BUS-LGL-007`。
