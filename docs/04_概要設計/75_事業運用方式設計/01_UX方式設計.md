# 01. UX 方式設計

本章は k1s0 のすべてのヒューマンインタフェースに共通するユーザ体験（UX）方針と、画面ごとの設計方針、アクセシビリティ、デザインレビュー運用を定める。k1s0 は tier1 基盤を画面越しに扱う複数のペルソナが存在するため、画面ごとに UX を独立判断すると基盤全体の体験が分断される。本章で横串の UX 方針を固定することで、各画面の担当者が同じ価値観で設計判断を下せるようにする。

## 本章の位置付け

k1s0 のユーザ向けインタフェースは大きく 4 面で構成される。Backstage を中心とする開発者ポータル、Grafana を中心とする観測ダッシュボード、ZEN Engine JDM エディタを中心とする業務ルール編集画面、配信ポータルを中心とするアプリ配信管理画面の 4 面である。このうち Backstage と Grafana は OSS 既製 UI を活用し、JDM エディタと配信ポータルは自作 UI とする。既製 UI と自作 UI が混在する前提では、配色・遷移・用語・アクセシビリティ水準が揃っていないと、同じテナントが日に複数面を行き来する際に操作学習コストが跳ね上がる。本章はこの学習コストを構造的に下げる UX 基盤を整える。

企画書で約束した「tier2 / tier3 開発者の生産性向上」と「運用 2 名体制」は、画面 UX の出来不出来で大きく左右される。画面の操作手順が 1 手多いだけで tier2 / tier3 が 100 人規模で使う以上、時間換算で月数百時間の損失となる。運用 2 名体制も画面上の異常検知と操作までの最短動線が確保されていなければ破綻する。本章の UX 設計は企画コミットメントを数量的に支える役割を負う。

## UX 原則

### 原則 1: 1 タスク 1 画面 1 ゴール

任意の画面は「この画面で何を達成するか」を単一の動詞で表現できることを必須とする。達成すべきタスクが複数ある画面はタブ分離またはモーダル分離で物理的に切り分ける。たとえば配信ポータルのアプリ詳細画面は「バージョン閲覧」と「新バージョン公開」をタブで分け、同時表示しない。

この原則は企画書の「tier3 担当者が単純反復作業に時間を取られない」を画面レベルで具体化する。画面内でタスクを混在させると「いまどの操作をしているか」の認知負荷が増大し、誤操作リスクも上がる。

### 原則 2: 3 クリックルール

主要タスクは最大 3 クリック以内で完了可能な動線を用意する。tier2 開発者の最頻タスク（State 登録テスト、PubSub 発行テスト、ログ参照、エラー詳細参照）、tier3 開発者の最頻タスク（アプリ配信、Feature Flag 切替、バージョン切り戻し）、運用者の最頻タスク（インシデント詳細確認、Runbook 起動、SLO 状況参照）、経営者の最頻タスク（KPI ダッシュボード閲覧、コストレポート閲覧）の 4 群について、対応する Runbook から 3 クリック到達を検証する。

3 クリックを超える動線が必要な場合は Backstage Software Catalog のショートカットピン留め機能または Grafana のブックマーク機能で短絡する。

### 原則 3: 破壊的操作は二段確認

削除・停止・公開取消・テナント廃止・鍵ローテーションなど副作用が不可逆または広範な操作は二段確認を必須とする。二段確認の実装は「確認ダイアログに対象名称のタイプ入力を要求する」パターンを基本とする（GitHub のリポジトリ削除と同様）。テナント廃止や削除のように影響が大きい操作は、加えて Backstage Announcement での 7 日前事前通知を義務付ける。

### 原則 4: 情報密度を揃える

同一タイプの画面（一覧 / 詳細 / 編集 / ダッシュボード）は情報密度を揃える。一覧画面は 1 行あたり最大 7 カラム、詳細画面はファーストビュー内に主要 5 項目、ダッシュボードは 1 画面あたり最大 9 パネルを上限とする。密度超過が必要な場合は展開ボタンで段階表示に分割する。この上限は運用 2 名が一瞬で状況把握できる情報量の実測値から逆算している。

### 原則 5: 失敗を明示的に表示する

エラー・警告・例外は必ず人間可読のメッセージと技術詳細の両方を表示する。人間可読メッセージはペルソナに応じた語彙で書き（tier3 向けは業務用語、運用者向けは技術用語）、技術詳細は畳み込み表示とする。エラーコード体系は [../50_非機能方式設計/09_エラーハンドリング方式設計.md](../50_非機能方式設計/09_エラーハンドリング方式設計.md) で定義する共通エラータクソノミに準拠する。

## 画面別 UX 方針

### Backstage ポータル

Backstage は tier2 / tier3 開発者の主要入口として機能する。Software Catalog でサービス一覧、TechDocs でドキュメント閲覧、Scaffolder でゴールデンパス（雛形生成）、Software Template でテンプレート展開、Announcement で告知を扱う。Backstage 標準 UI を最大限活用し、k1s0 固有のカスタマイズはプラグイン単位で最小限に抑える。

k1s0 固有プラグインとして、テナント別コスト表示（[05_コスト管理方式.md](05_コスト管理方式.md) と連動）、課金使用量リアルタイム表示（[06_課金メータリング方式.md](06_課金メータリング方式.md) と連動）、OSS ライセンスレポート（[07_OSSライセンス運用方式.md](07_OSSライセンス運用方式.md) と連動）の 3 プラグインを Phase 1c で整備する。プラグインは Backstage テーマ（Material UI ベース）に準拠する。

### Grafana ダッシュボード

Grafana は運用者の主要入口として機能する。ダッシュボードは「経営サマリ」「プラットフォーム全体」「テナント別」「サービス別」「インシデント対応」の 5 階層で構成し、階層間のドリルダウンリンクを必須とする。経営サマリは SLO 達成率・コスト・インシデント件数・利用量トレンドを 9 パネル以内で表示する。

Grafana は AGPL-3.0 OSS のため、原則 8 に基づき独立 Pod として稼働させ、Keycloak OIDC で認証を束ねる。Grafana UI から自作コードへのリンクは公式 Data Source Plugin 経由のみとし、Grafana 本体の改変は禁止する。ダッシュボード JSON は `infra/grafana-dashboards` リポジトリで Git 管理し、変更は PR レビュー必須とする。

### ZEN Engine JDM エディタ

JDM エディタは業務部門（田辺守護派を中心とするルール編集者）が直接操作する画面である。Decision テーブル・関数式・グラフビューの 3 モードを切替可能とし、業務部門が慣れ親しんだ Excel ライクなテーブル編集を一次体験に据える。Rust 製 ZEN Engine の WASM ビルドをブラウザで実行してリアルタイム検証を行い、保存前に結果を即時確認できるようにする。

JDM エディタは自作 UI とし、Tailwind CSS + React でフロント、Rust コアで評価を実装する。編集結果は JSON 形式で tier1 Decision API 経由で保存し、バージョン履歴は Git 風にコミット単位で管理する（Phase 1c）。差分表示・ロールバック・承認ワークフロー（Phase 2）を段階的に追加する。

### 配信ポータル

配信ポータルは tier3 アプリ開発者と業務部門双方が利用する。tier3 開発者側は「新規アプリ登録」「バージョン公開」「ターゲティング設定（Feature Flag 連動）」「ロールバック」の 4 タスクを主に行い、業務部門側は「アプリ検索」「インストール」「アップデート通知受信」を行う。双方のタスクは画面を分離し、同一 URL 上でロール別にレンダリングを切り替える。

配信対象は ClickOnce / MSIX / 単体 EXE / Web アプリ の 4 形態を Phase 1a〜1b で順次対応する。レガシー .NET Framework 資産（制約 9）を捨てずに共存するため、ClickOnce 配信を Phase 1a 段階から必須機能として整備する。

## デザインシステム

### トークンと配色

配色はアプリ層（暖色 `#fff2cc`）、ネットワーク層（寒色 `#dae8fc`）、インフラ層（中性灰 `#f5f5f5`）、データ層（薄紫 `#e1d5e7`）の 4 レイヤ規約（[../../00_format/drawio_layer_convention.md](../../00_format/drawio_layer_convention.md)）を画面でも踏襲する。ダッシュボード・画面内の構成要素が「どのレイヤか」を配色だけで判別できる状態を維持する。

Backstage テーマは Material UI のデフォルトトークンをベースに、k1s0 ブランドカラー（tier1=`#6c8ebf`、tier2=`#9673a6`、tier3=`#d79b00`）を上書きする。自作 UI（JDM エディタ / 配信ポータル）は Tailwind CSS を採用し、`tailwind.config.ts` で同じトークンを宣言する。Backstage と自作 UI でカラーパレットと spacing が揃うことで、画面遷移時の違和感を最小化する。

### タイポグラフィとアイコン

フォントは Noto Sans JP（日本語）と Inter（英数字）の組合せを基本とし、見出し 24/20/18/16 px、本文 14 px、補助 12 px の 6 段階に限定する。アイコンは Heroicons を基本セットとし、Backstage デフォルトアイコンとの重複は Heroicons 側を優先する。アイコン意味の一貫性が画面跨ぎの学習コストに直結するため、新規アイコン追加は Product Council のデザインレビューを経る。

### コンポーネント

共通コンポーネント（ボタン / 入力 / テーブル / モーダル / トースト / ローディング / 空状態 / エラー状態）は `libs/ui-kit` として単一リポジトリに集約し、Backstage プラグインと自作 UI の双方から import する。Storybook でカタログ化し、Visual Regression Test（Chromatic）を CI で必須とする。コンポーネントの破壊的変更は SemVer Major で明示し、呼び出し側の追随期限を Product Council で合意する。

## アクセシビリティ

### 準拠水準

WCAG 2.1 Level AA 準拠を全画面必須とする。コントラスト比 4.5:1 以上（大文字 3:1）、キーボード操作可能（Tab / Shift+Tab / Enter / Space / Esc）、スクリーンリーダー対応（ARIA 属性・ランドマーク・見出し階層）、ズーム 200% 対応、色のみに依存しない情報伝達、動画の字幕、の 6 項目を CI で自動検査する。自動検査ツールは axe-core を採用し、PR ごとに回帰検出する。

Level AAA は経営サマリダッシュボードと JDM エディタに限定適用する（業務部門利用が多く読み上げ依頼が発生しやすいため）。AAA 適用画面はコントラスト比 7:1、ズーム 400% 対応、手話動画（録画研修のみ）を追加要件とする。

### 障害特性別配慮

視覚障害に対してはスクリーンリーダー（NVDA / JAWS / VoiceOver）での動作確認を四半期ごとに実施し、欠陥があれば次四半期までに修正する。聴覚障害に対しては通知音のビジュアル代替（トースト通知・点滅）を必須とする。運動障害に対してはキーボード代替操作とショートカット一覧を全画面の `?` キーで表示可能とする。認知障害に対しては操作取り消し（Undo）を破壊的操作前の一段階まで復元できる仕様とする。

## ペルソナ別 UX 調整

k1s0 のユーザは企画書で定義された 4 ペルソナ（田辺守護派 / 森田推進派 / 西尾決裁者 / 石田運用）を基準とする。各ペルソナで主要タスク・情報粒度・語彙・頻度が異なるため、画面の初期設定（ダッシュボードのデフォルトパネル、一覧のデフォルト列、ログレベルのデフォルトフィルタ）をペルソナ別に事前設定する。

田辺守護派（既存資産を守りたい保守派情シス）向けには、既存 .NET Framework 資産の配信状態・互換性レポート・レガシーアプリのインシデント履歴を初期表示する。森田推進派（新技術を積極導入したい若手情シス）向けには、新規ゴールデンパス・新規サービステンプレート・Feature Flag 実験結果を初期表示する。西尾決裁者（経営層）向けには KPI サマリ・コスト推移・SLO 達成率・稟議効果（年次比較）を初期表示する。石田運用（保守運用担当）向けには SLO 違反アラート・インシデント件数・最近のデプロイ履歴・Runbook 最頻使用を初期表示する。

ペルソナ切替は Keycloak の属性（`x-k1s0-persona`）で行い、ユーザが明示的に変更することも可能とする。ペルソナを属性として保持することで、新規ユーザ招待時にペルソナを選択すれば初期画面が自動構成される。

## ユーザリサーチ

ユーザ調査を四半期ごとに実施する。調査は「利用頻度調査（画面 PV / 操作回数）」「タスク完了時間測定（主要タスク 10 種）」「満足度アンケート（NPS）」「フォーカスインタビュー（3〜5 名）」の 4 手法を組合せる。調査結果は Product Council で共有し、次四半期の UX 改善バックログに反映する。

調査で観測する KPI は次の 4 項目とする。(1) 主要タスク完了時間の中央値と p95（p95 が目標値を 20% 超過した場合は UX 再設計）、(2) エラー遭遇率（1 セッションあたりのエラー遭遇回数、目標 0.1 未満）、(3) NPS（目標 +20 以上）、(4) 新規ユーザの初回成功率（ゴールデンパス完走率、目標 80% 以上）。これらは [12_分析とレポーティング方式.md](12_分析とレポーティング方式.md) の KPI ダッシュボードに組み込む。

## デザインレビュー

新規画面・大幅変更（主要動線変更 / 配色大幅変更 / 新規プラグイン追加）は Product Council のデザインレビューを必須とする。レビュー項目は「5 原則への適合」「デザインシステム準拠」「アクセシビリティ（axe-core レポート）」「ペルソナ別初期設定」「ユーザリサーチ仮説」の 5 項目とする。レビューは月次 Product Council の前半 30 分を定常枠として確保する。

レビュー結果は Backstage の Announcement で全テナントに公開する。画面変更の事前告知は 7 日前、実装反映は 30 日以内を目安とする。破壊的変更（既存操作の削除・移動）は 90 日前告知と移行ガイド提示を必須とする。

## 対応要件一覧

本章は以下の要件定義 60_事業契約 UX 系要件と 1:1 対応する。

- BC-UX-001: UX 原則 5 項目（1 タスク 1 画面 / 3 クリック / 二段確認 / 情報密度 / 失敗明示）を全画面で順守すること。対応設計 ID: `DS-BUS-UX-001`。
- BC-UX-002: 画面別 UX 方針（Backstage / Grafana / JDM エディタ / 配信ポータル）を定めること。対応設計 ID: `DS-BUS-UX-002`。
- BC-UX-003: 共通デザインシステム（配色 / タイポグラフィ / コンポーネント）を `libs/ui-kit` として提供すること。対応設計 ID: `DS-BUS-UX-003`。
- BC-UX-004: WCAG 2.1 AA 準拠をすべての画面で必須とし、axe-core で CI 自動検査を行うこと。対応設計 ID: `DS-BUS-UX-004`。
- BC-UX-005: ペルソナ別初期設定とユーザリサーチ（四半期）を運用に組み込むこと。対応設計 ID: `DS-BUS-UX-005`。
