# k1s0 概要設計書

本ドキュメントは k1s0 プラットフォーム開発における概要設計書である。要件定義書（[../03_要件定義/](../03_要件定義/)）で合意された「何を作るか」と、構想設計（[../02_構想設計/](../02_構想設計/)）で合意された技術骨格と選定根拠を受け、IPA（独立行政法人 情報処理推進機構）が提示する共通フレーム 2013 の設計プロセスに準拠して「どう作るかの概要」を定義する。

## 本書の位置付け

k1s0 は日本大手企業（採用側組織）の情報システム部門向けに、オンプレミス完結・OSS 積み上げで構築するマイクロサービス基盤である。tier1 が State/PubSub/Workflow/RuleEngine など 11 API を公開し、tier2 のドメインロジックと tier3 のアプリ配信ポータルを支える三層構造をとる。要件定義書はこの三層構造に対して「何を約束するか」を契約文書として固定化し、構想設計書は「どの OSS をどう組み合わせるか」の骨格と選定根拠を固定化した。

概要設計書は両者を受けて、**要件を実装単位 (コンポーネント・インタフェース・データ・制御方式・運用・事業運用) に割り付ける層**である。IPA 共通フレーム 2013 の「システム方式設計プロセス」と「ソフトウェア方式設計プロセス」に相当し、後続の詳細設計・実装段階に向けた上流基準として機能する。

構想設計書との役割分担は明確にする。構想設計書は「なぜその技術選定か」という**根拠と ADR** を扱い、概要設計書は「その選定を踏まえて、どのコンポーネントがどのインタフェースで何を実現するか」という**方式設計書本体**を扱う。

## IPA 準拠の方針

本書は IPA 共通フレーム 2013（SLCP-JCF2013）の以下 3 プロセスを骨格として採用する。

- **システム方式設計プロセス（6.4.3）**: システム要件をハードウェア・ソフトウェア・手作業に分解し、全体方式を確立する。6 アクティビティ（ハードウェア方式 / ソフトウェア方式 / システム処理方式 / データベース方式 / システム結合テスト要件定義 / 評価レビュー）で構成。
- **ソフトウェア方式設計プロセス（7.1.2）**: ソフトウェアの最上位構造とコンポーネント・インタフェース・データベース・利用者文書を方式として固定化する。6 アクティビティ（コンポーネント方式 / 外部インタフェース方式 / データベース最上位設計 / 利用者文書暫定版 / ソフトウェア結合テスト要件定義 / 評価レビュー）で構成。
- **移行プロセス（6.4.10）**: レガシー資産からの移行、k1s0 内バージョン間移行の方式を確立する独立プロセス。

k1s0 はプラットフォーム製品であり、IPA が主に想定する業務システムとは性格が異なる。画面・帳票が中心の業務系設計書ではなく、tier1 API を中心とした方式設計書となる点を踏まえ、共通フレーム必須アクティビティに加えて **k1s0 固有拡張章（30 共通機能方式 / 40 制御方式 / 50 非機能方式 / 55 運用ライフサイクル方式 / 70 開発者体験方式 / 75 事業運用方式）** を設ける。これにより、要件定義書で採番された全要件 ID（FR / NFR / BR / OR / DX / BC）に対応する方式設計ファイルが 1:1 で存在することを保証する。詳細は [00_設計方針/](00_設計方針/) に記述する。

## ドキュメント構造

12 カテゴリ構造で設計を分類する。番号帯は共通フレーム 2013 の設計プロセスと要件定義書の番号帯にそれぞれ対応付け、読み順は番号順で完結するよう設計した。

```
04_概要設計/
├── README.md                        # 本書
├── 00_設計方針/                      # 概要設計プロセスの定義・横断基盤
├── 10_システム方式設計/              # 共通フレーム 6.4.3 厳密対応
├── 20_ソフトウェア方式設計/          # 共通フレーム 7.1.2 厳密対応
├── 30_共通機能方式設計/              # k1s0 固有拡張（tier1 横断機能）
├── 40_制御方式設計/                  # k1s0 固有拡張（Saga/冪等/リトライ）
├── 50_非機能方式設計/                # NFR 8 分類（A〜F + G/H 拡張）に対応
├── 55_運用ライフサイクル方式設計/    # 要件定義 40_運用ライフサイクル に対応
├── 60_移行方式設計/                  # 共通フレーム 6.4.10 対応
├── 70_開発者体験方式設計/            # 要件定義 50_開発者体験 に対応
├── 75_事業運用方式設計/              # 要件定義 60_事業契約 に対応
├── 80_トレーサビリティ/              # 要件 → 設計の双方向追跡
└── 90_付録/                          # 共通フレーム対応表・ADR 索引・参考文献
```

### 00_設計方針

概要設計作業そのものをどう進めるかの手続論を記述する。設計プロセス定義、設計原則と制約、設計 ID 体系とトレーサビリティ方針、設計レビューのゲート、用語集と命名規約を格納する。本カテゴリは設計本体ではなく「設計をどう記述・合意・検収するか」のメタ文書である。

### 10_システム方式設計

共通フレーム 2013 の 6.4.3 に厳密対応する。k1s0 のシステム要件を HW/SW/手作業に分解し、全体方式を確立する。ハードウェア方式（ノード・ストレージ・拠点・冗長化）、ソフトウェア方式（自社開発 vs OSS vs パッケージの分担）、システム処理方式（K8s + Istio + Dapr を採用した処理形態）、データベース方式（状態ストア・メッセージブローカー・シークレット管理の全体方式）、システム結合テスト要件、評価レビュー結果を格納する。

### 20_ソフトウェア方式設計

共通フレーム 2013 の 7.1.2 に厳密対応する。tier1 ソフトウェアを個別コンポーネントまで分割し、相互関係とインタフェースを方式として固定化する。Go + Rust ハイブリッド構成のコンポーネント境界、tier2/tier3 に公開する外部インタフェース（11 API・エッジ IoT・レガシー連携）、tier1 内部の gRPC とイベントスキーマ、論理 ER と最上位テーブル設計、利用者文書（暫定版）としての tier2/tier3 向け Golden Path 初版と運用者手引き初版、ソフトウェア結合テスト要件、評価レビュー結果を格納する。

### 30_共通機能方式設計

tier1 全 API にまたがる横断機能を方式として固定化する。共通フレームの明示アクティビティではないが、プラットフォーム製品として「どの API でも同じ認証・同じログ・同じエラーコード」を保証するため独立章とする。認証方式、認可と権限モデル、ログ出力方式、監査証跡方式、分散トレーシング、メトリクス収集、エラーハンドリングとメッセージ体系、暗号化と鍵管理、Feature Management を格納する。

### 40_制御方式設計

tier1 の信頼性を支える制御ロジックの方式を固定化する。Saga・冪等性・リトライ・サーキットブレーカー・非同期メッセージング・ワークフロー振り分け（ZEN Engine と Dapr Workflow の使い分け）・タイムアウトとバックプレッシャは tier1 API の横断論点であり、共通機能とは分けて独立章とする。

### 50_非機能方式設計

[../03_要件定義/30_非機能要件/](../03_要件定義/30_非機能要件/) で採番した NFR を設計レベルに落とし込む。IPA 非機能要求グレード 6 大項目（A〜F）に k1s0 独自拡張（G データ保護・プライバシー / H 完整性・コンプライアンス）を加えた 8 分類と 1:1 対応を維持し、可用性・DR、性能と拡張性、運用、監視と観測性、セキュリティ、キャパシティ計画、システム環境とエコロジー、データ保護とプライバシー、完整性とコンプライアンスを格納する。

### 55_運用ライフサイクル方式設計

[../03_要件定義/40_運用ライフサイクル/](../03_要件定義/40_運用ライフサイクル/) に 1:1 対応する。インシデント対応、サポート階層、環境構成管理、非推奨と EOL、撤退戦略など、**非計画的事象とライフサイクル遷移**の方式を格納する。50 が計画的運用を扱うのに対し、本章は予期せぬ事象と終わり方を扱う。

### 60_移行方式設計

共通フレーム 2013 の 6.4.10 移行プロセスに対応する。レガシー .NET Framework 資産からの移行方式と、k1s0 自体のバージョン間移行方式を扱う。データ移行方式、並行稼働と切替方式、ロールバックと復旧方式を格納する。

### 70_開発者体験方式設計

[../03_要件定義/50_開発者体験/](../03_要件定義/50_開発者体験/) に 1:1 対応する。tier2 / tier3 開発者が k1s0 を利用する際の生産性・自動化・ガイダンスを方式として固定化する。CI/CD、ローカル開発環境、DevEx 指標、Backstage ポータル詳細、テスト戦略を格納する。

### 75_事業運用方式設計

[../03_要件定義/60_事業契約/](../03_要件定義/60_事業契約/) に 1:1 対応する。事業運営・契約・ガバナンスに関わる要件（UX、i18n、テナントオンボーディング、教育、コスト、課金、OSS ライセンス運用、法務契約、サプライチェーン、ガバナンス、AI ガバナンス、分析、サステナビリティ）を技術実現方式として固定化する。経営・法務・調達・CSR 部門の合意事項を稼働可能な仕組みに翻訳する層。

### 80_トレーサビリティ

上流（要件定義）と下流（詳細設計以降）への紐付けを一元管理する。設計 ID 索引、要件 ID → 設計 ID の双方向マトリクス、詳細設計段階への引渡し一覧を格納する。設計追加・変更が発生した際はこのカテゴリで影響範囲を追跡する。

### 90_付録

共通フレーム 2013 設計プロセスとの対応表、構想設計 ADR との相互参照索引、参考文献を格納する。監査対応や採用検討説明時にエビデンスとして参照する。

## 設計 ID 体系

全設計項目は一意の設計 ID を付与し、要件定義・構想設計・後続詳細設計工程と双方向トレースを可能にする。ID 体系の詳細は [00_設計方針/](00_設計方針/) に記述予定。本書内の記述例は以下。

- `DS-SYS-HW-001`: 設計（Design）/ システム方式（System）/ ハードウェア / 通番
- `DS-SW-COMP-001`: 設計 / ソフトウェア方式（Software）/ コンポーネント / 通番
- `DS-SW-EIF-001`: 設計 / ソフトウェア方式 / 外部インタフェース（External Interface）/ 通番
- `DS-CF-AUTH-001`: 設計 / 共通機能（Common Function）/ 認証 / 通番
- `DS-CTRL-SAGA-001`: 設計 / 制御方式（Control）/ Saga / 通番
- `DS-NFR-AVL-001`: 設計 / 非機能方式（NFR）/ 可用性 / 通番
- `DS-OPS-INC-001`: 設計 / 運用ライフサイクル（Operations Lifecycle）/ インシデント / 通番
- `DS-MIG-DATA-001`: 設計 / 移行方式（Migration）/ データ移行 / 通番
- `DS-DEVX-CICD-001`: 設計 / 開発者体験（Developer Experience）/ CI/CD / 通番
- `DS-BUS-BIL-001`: 設計 / 事業運用（Business Operations）/ 課金メータリング / 通番

各設計 ID は対応する要件 ID（`FR-*` / `NFR-*` / `BR-*` / `OR-*` / `DX-*` / `BC-*`）を明記し、[80_トレーサビリティ/](80_トレーサビリティ/) で一元管理する。

## 構想設計との関係

[../02_構想設計/](../02_構想設計/) と本書の役割分担を以下のとおり明文化する。誤解を避けるため、該当箇所を両書に記載する。

- **構想設計書（02）**: 技術選定の**根拠と ADR**。なぜ Dapr ファサードなのか、なぜ Go+Rust ハイブリッドなのか、なぜ ZEN Engine なのか、を決定記録として残す層。
- **概要設計書（04、本書）**: 選定を踏まえた**方式設計書本体**。どのコンポーネントが何の責務を持ち、どのインタフェースで接続し、どの制御方式で信頼性を担保するか、を仕様として固定化する層。

構想設計で未決定の事項が発見された場合は、本書で新規 ADR を起票し [../02_構想設計/adr/](../02_構想設計/adr/) に追加する。逆に、本書で方式を具体化する過程で構想設計の前提が崩れた場合は、構想設計書を改訂したうえで本書を更新する。

## 改訂と合意プロセス

本書は要件定義書と同様に版を刻む。リリース時点で tier1 11 API 全機能を本格実装する前提のもと、その時点までに必要な方式と、採用後の運用拡大時に関する方式の両方を確定する。主要な変更（コンポーネント分割変更・インタフェース契約変更・制御方式の方針転換）はステークホルダーとの合意ログを残し、影響を受ける要件 ID と構想設計 ADR を連動させる。改訂ルールの詳細は [00_設計方針/](00_設計方針/) に記述予定。
