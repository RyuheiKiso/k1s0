# 55. 運用ライフサイクル方式設計

本カテゴリは要件定義書の [../../03_要件定義/40_運用ライフサイクル/](../../03_要件定義/40_運用ライフサイクル/) に 1:1 対応する方式設計章である。稼働後の運用プロセス、予期せぬインシデント、機能の非推奨化、撤退判断といった**計画外事象・ライフサイクル遷移**を方式として固定化する。

## 本カテゴリの位置付け

[../50_非機能方式設計/03_運用方式設計.md](../50_非機能方式設計/) は「計画的運用」（起動停止・バックアップ・ジョブ・計画保守）を扱うのに対し、本章は「非計画的事象と終わり方」を扱う。インシデントの発生は予定できず、機能の廃止も利用者都合で決まり、撤退判断はビジネス事由で突発する。この種の事象をどの仕組みで検知・対応・完了させるかを方式として確定させる役割を持つ。

要件定義書の 40 カテゴリは情シス運用部門が最終合意する層であり、ここでの方式不整合は運用契約の不履行に直結する。概要設計として方式を明文化することは、運用部門の SLA と開発部門の実装の間に曖昧さを残さないための契約文書である。

## 格納予定ドキュメント

9 ファイルを格納する。

- `01_サポート階層方式.md` — tier2 / tier3 開発者・情シス運用者・エンドユーザーからの問い合わせを受け止めるサポート階層（L1 / L2 / L3）の設計、チケット管理、エスカレーションルール、応答時間目標を記述する。要件 ID `OR-SUP-*` と対応する。
- `02_インシデント対応方式.md` — 障害検知から初動・トリアージ・復旧・ポストモーテムまでの一連のプロセスを方式化する。重大度分類、初動チーム編成、外部ステークホルダー通知、リカバリ手順の標準化を含む。要件 ID `OR-INC-*` と対応する。[../50_非機能方式設計/04_監視と観測性方式.md](../50_非機能方式設計/) のアラート設計と連動する。
- `03_環境構成管理方式.md` — 環境別（開発 / ステージング / 本番）構成管理、GitOps による宣言的構成、環境昇格の承認フロー、構成ドリフト検知方式を記述する。要件 ID `OR-ENV-*` と対応する。
- `04_非推奨とEOL方式.md` — tier1 API・機能・コンポーネントの非推奨化プロセス、SemVer 運用、EOL 告知からサポート終了までのタイムライン、利用者への移行ガイダンス提供方式を記述する。要件 ID `OR-EOL-*` と対応する。
- `05_撤退戦略方式.md` — k1s0 自体の廃止判断が発生した場合の撤退手順、テナントデータのエクスポート・引き継ぎ方式、既存資産の第三者移管方式、OpenShift 等への退避方式を記述する。要件 ID `OR-EXIT-*` と対応する。撤退はビジネス継続の最終ライフラインであり、発動基準と所要時間を明示する。
- `06_FMEA分析方式.md` — 故障モード影響解析（FMEA: Failure Mode and Effects Analysis）の実施方式を記述する。tier1 コンポーネント・外部依存・運用プロセスの各要素で故障モードを列挙し、発生頻度 (O) × 影響度 (S) × 検知性 (D) の RPN（Risk Priority Number）で優先順位を付け、改善アクションを採番する。02 インシデント対応が事後対処を扱うのに対し、本ファイルは事前分析を扱う。
- `07_負荷試験方式.md` — 本番投入前・定期・リリース前の負荷試験方式を記述する。ワークロードモデル（代表業務シナリオ・同時接続数・トランザクション配分）、試験ツール（k6 / Locust 等）、合否基準（SLO 遵守 / リソース使用率上限）、本番トラフィック増加時の外挿方式を含む。[../70_開発者体験方式設計/05_テスト戦略方式.md](../70_開発者体験方式設計/) のテスト戦略全体の中で、本ファイルは運用面（容量計画・リリース判定）の負荷試験を扱う。
- `08_Runbook設計方式.md` — Runbook（運用手順書）の設計方式を記述する。記載粒度（誰が読んで何分で実行できるか）、構成要素（前提条件 / 判断フロー / 実行コマンド / 検証手順 / ロールバック）、自動化度合いの判定基準、アラートからのリンク方式、版管理と陳腐化防止の仕組みを含む。
- `09_Runbook目録方式.md` — 本番運用で必要となる Runbook の目録と採番方式を記述する。インシデント種別・コンポーネント・運用イベント（起動停止 / バックアップ / リストア / 昇格）別に Runbook を網羅採番し、カバレッジを継続測定する。08 が設計方法、本ファイルは全 Runbook の索引。

## 執筆方針

- 本章は「事象が起きたときに何秒で何を判断するか」という時間軸を明示する。時間軸のない方式は運用文書として価値が低い。
- 方式 ID は `DS-OPS-<サブカテゴリ>-<通番>` を用いる（例: `DS-OPS-INC-001`、`DS-OPS-EOL-001`）。
- 運用者が単独で読んで実行可能な粒度まで方式を詳細化する。他章への参照で逃げないこと。
- 撤退戦略（05）は経営判断に直結するため、発動基準・承認者・リカバリ不能判定の閾値を明示する。
