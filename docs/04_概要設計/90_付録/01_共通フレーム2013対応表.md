# 01. 共通フレーム 2013 対応表

本ファイルは IPA（独立行政法人 情報処理推進機構）が発行する共通フレーム 2013（SLCP-JCF2013）の設計関連プロセスと、概要設計書の各章・各アクティビティとの対応関係を明示する対応表である。本書が IPA 準拠であることを監査・公共調達の場面で証明する根拠資料として機能する。

## 本ファイルの位置付け

k1s0 は日本大手企業（採用側組織）の情報システム部門向けプラットフォームであり、公共系入札や大手企業の調達プロセスでは IPA 共通フレーム 2013 への準拠が実質的な必須条件となる。準拠を主張するだけでは監査で通らず、共通フレーム側のアクティビティ ID と自社設計書の章番号が 1:1 で対応していることを示す必要がある。

本対応表は採用検討審査・監査対応・官公庁向け説明の 3 場面で即座に参照される前提で設計されている。読み手は共通フレーム側のアクティビティを知っており、「k1s0 ではどの章がこれに対応するか」を逆引きする目的で本表を参照する。したがって、共通フレーム側の用語を一次キーとし、k1s0 側の章を二次値として配置する。

本章は索引層のため設計 ID を採番しない。

## 共通フレーム 2013 設計関連プロセスの全体像

共通フレーム 2013 は「企画プロセス」「要件定義プロセス」「開発プロセス」「運用プロセス」「保守プロセス」の 5 大プロセスで構成される。本書が対象とする設計層は「開発プロセス」の中の以下 3 プロセスに相当する。

- **6.4.3 システム方式設計プロセス**: システム要件を HW / SW / 手作業に分解し、全体方式を確立するプロセス。6 アクティビティ。
- **7.1.2 ソフトウェア方式設計プロセス**: ソフトウェアの最上位構造を確立し、コンポーネント・インタフェース・データベースを方式として固定化するプロセス。6 アクティビティ。
- **6.4.10 システム移行プロセス**: 旧システムから新システムへの移行方式を確立する独立プロセス。3 アクティビティ。

これら 3 プロセスは共通フレームの必須要件を構成する。k1s0 はプラットフォーム製品の特性から、これら必須プロセスに加えて固有拡張章（30 共通機能 / 40 制御 / 50 非機能 / 55 運用ライフサイクル / 70 開発者体験 / 75 事業運用）を設けた。拡張章の位置付けも本表で明示する。

## k1s0 固有拡張章の位置付け

概要設計書には共通フレーム対応章（10 / 20 / 60）に加えて、30 / 40 / 50 / 55 / 70 / 75 の 6 つの固有拡張章がある。これらは共通フレームの必須アクティビティには直接対応しないが、以下の理由で独立章として設けられた。

第一に、プラットフォーム製品特有の要件（横断機能・制御ロジック・開発者体験）は共通フレームが主に想定する業務システムの設計書枠組みでは収まりきらない。共通フレーム 7.1.2.1 の「ソフトウェア構成要素の方式」に横断機能を押し込むと、コンポーネント粒度と横断機能粒度が混在して設計書の可読性が下がる。

第二に、IPA 非機能要求グレード 2018 は共通フレームと独立したフレームワークとして提供されており、非機能を独立章にすることで両フレームワーク準拠を同時に達成できる。

第三に、要件定義書側（[../03_要件定義/](../../03_要件定義/)）のカテゴリ構造と 1:1 対応させることで、要件 → 設計のトレーサビリティを維持できる。

以下の表は拡張章の共通フレームとの関係性を示す。対応度は「拡張」とし、共通フレーム準拠の主張には使わない。

| k1s0 章 | 章タイトル | 対応する共通フレーム関連性 | 位置付け |
|---|---|---|---|
| 30_共通機能方式設計 | tier1 横断機能 | 7.1.2.1 の下位詳細（横断視点） | プラットフォーム固有拡張 |
| 40_制御方式設計 | Saga / 冪等 / リトライ | 7.1.2.1 + 7.1.2.2 の下位詳細 | プラットフォーム固有拡張 |
| 50_非機能方式設計 | NFR 11 分類 | 非機能要求グレード 2018 準拠 | IPA 別フレームワーク準拠 |
| 55_運用ライフサイクル方式設計 | インシデント / EOL / 撤退 | 8.3 システム運用プロセス（関連） | 要件定義 40 章と 1:1 |
| 70_開発者体験方式設計 | CI/CD / DevEx | 7.1.2.4 利用者文書（関連） | 要件定義 50 章と 1:1 |
| 75_事業運用方式設計 | UX / コスト / 法務 | 1.4 取得プロセス（関連） | 要件定義 60 章と 1:1 |

## 監査時の参照経路

監査対応で「k1s0 概要設計書は共通フレーム 2013 に準拠している」と主張する際の参照経路を定義する。監査員は本表を起点に、アクティビティ ID から k1s0 対応ファイルに辿り着ける状態を保つ。

標準的な監査手順は以下 4 ステップである。第一に、監査員が共通フレームの対象アクティビティ ID を指定する（例: 7.1.2.2 ソフトウェアインタフェース方式の確立）。第二に、本表で対応 k1s0 ファイルを参照する。第三に、該当ファイルの「対応要件一覧」節で要件 ID に遡る。第四に、要件 ID から [../80_トレーサビリティ/02_要件から設計へのマトリクス.md](../80_トレーサビリティ/02_要件から設計へのマトリクス.md) で設計 ID 全体に展開する。

この 4 ステップが成立することを監査前チェックリストで確認する。途中で切断が発生している場合、監査前に解消する必要がある。

## 非機能要求グレード 2018 との関係

概要設計書の 50_非機能方式設計/ は IPA 非機能要求グレード 2018 に準拠する。非機能要求グレード側の 6 大項目（A 可用性 / B 性能・拡張性 / C 運用・保守性 / D 移行性 / E セキュリティ / F システム環境・エコロジー）に k1s0 固有拡張（G データ保護・プライバシー / H 完整性・コンプライアンス / I 観測性）を加えた 9 分類を維持する。

グレード 2018 の評価項目は 236 件に達するが、k1s0 では 採用側組織向けプラットフォームとして重要度が高い項目を重点採番している。非機能要求グレード対応の詳細は [../03_要件定義/90_付録/02_非機能要求グレード判定.md](../../03_要件定義/90_付録/) に記述されており、本対応表では概要となるグレード分類対応のみを記載する。

| A 可用性 | 50_非機能方式設計/01 可用性と DR 方式 | 完全対応 |
| B 性能・拡張性 | 50_非機能方式設計/02 性能と拡張性方式 | 完全対応 |
| C 運用・保守性 | 50_非機能方式設計/03 運用方式 + 55_運用ライフサイクル方式設計 | 完全対応 |

## 運用ルール

本対応表の継続運用は以下の 3 ルールで固定化する。

- **共通フレーム改訂追随**: IPA が共通フレームを改訂した場合、改訂内容を四半期レビューで確認し、影響があれば本表を更新する。共通フレーム 2013 の次版は 2027 年頃と見込まれており、発行時点で比較分析を実施。
- **k1s0 章構成変更**: k1s0 側の章構成を変更する場合（章追加・章統合・章分割）、変更 PR と同時に本表を更新する。
- **監査指摘対応**: 監査で本表と実際の設計書に乖離が指摘された場合、指摘事項と是正内容を本表の末尾「監査指摘履歴」節に記録する。
