# 06. ソフトウェア結合テスト要件

本ファイルは IPA 共通フレーム 2013 の **7.1.2.5 ソフトウェア結合テスト要件定義** に対応する。k1s0 tier1 の内部 6 コンポーネント群（Dapr ファサード層 Go / 自作領域 Rust / 内部データ層 OSS / gRPC 契約層 / Telemetry / Audit）が相互に結合した際の検証観点・合格基準・環境要件・実施計画を方式として確定する。

## 本ファイルの位置付け

ソフトウェア結合テストは「部品を個別に単体テストしても、組み上げた時に非機能要件を満たすとは限らない」というリスクに対処する層である。k1s0 の場合、Dapr ファサード層（Go）と自作領域（Rust）の言語境界を gRPC で跨ぐ構造、Dapr Workflow / Temporal の両対応、ZEN Engine の JDM 評価経路、OpenBao の Lease Rotation 経路、PostgreSQL WORM の hash_chain 整合性など、結合してはじめて顕在化する不整合が多い。これらを Phase 1b 完了判定までに確度高く検出することが本章の設計目標である。

本章は [../10_システム方式設計/05_システム結合テスト要件.md](../10_システム方式設計/05_システム結合テスト要件.md) と役割分担する。10 章（システム結合テスト）は infra 層 + tier1 + tier2 サンプル + tier3 サンプル の全階層を立ち上げたシステム全体の検証であるのに対し、本章は **tier1 内部のみ** を対象とする。tier1 が仕様通りの API 契約を満たし、内部 OSS・gRPC 境界・エラー伝播・観測性の全てが期待通りに動くかを、tier2 / tier3 の実コードなしで検証する層である。

結合テストの合格は Phase 1b（MVP-1a）完了判定の必須条件である。合格基準を曖昧にすると、「単体テストは通っているが結合で壊れる」状態を放置したまま Phase 承認が行われ、Phase 1c 監査対応で手戻りが発生する。本章では合格基準をすべて具体数値で記述する。

## 対象範囲と除外範囲

結合テストの対象範囲を、含むものと含まないものの両面で明示する。範囲を事前に固定することで、テスト工数の見積もりと合格判定の客観性を確保する。

**対象に含むもの**: Dapr ファサード層の Go コード（State / PubSub / Binding / Workflow / Secrets / Service Invoke の各サービス）、自作領域の Rust コード（ZEN Engine 統合 / crypto / 監査改ざん防止 / JTC 固有機能）、Go ⇔ Rust 間の gRPC 境界（`.proto` 契約・型変換・エラー伝播）、内部 OSS（Valkey / Kafka / PostgreSQL / OpenBao / ZEN Engine / flagd）の統合動作、tier1 から tier2 / tier3 へ公開される 11 API の振る舞い契約、Telemetry の OpenTelemetry 準拠性、Audit の hash_chain 整合性の 7 領域。

**対象に含まないもの**: tier2 / tier3 サンプルアプリの結合（これはシステム結合テストの範囲）、Kubernetes 層の Pod スケジューリング検証（これは infra 層の責務）、Backstage / Keycloak / Argo CD の動作検証（これは operation 層の責務）、本番環境での監査ログ長期保管検証（これは Phase 1c 以降の受け入れテスト）。

範囲境界は Phase 1b 着手時点で再確認し、Phase 中の範囲変更は Product Council 承認を必須とする。

## 設計項目 DS-SW-IT-001 検証観点の 5 分類

tier1 内部結合テストの検証観点を 5 分類に整理する。各観点は独立に検証し、合格基準を個別に満たす。

**正常系観点**: tier1 公開 11 API それぞれで、正常リクエストが期待通りの応答を返す。State.Set → Get の往復で同一値が返る、PubSub.Publish → Subscribe で全イベントが欠落なく届く、Decision.Evaluate で JDM ルールが期待値を返す、Workflow.Start で 3 ステップの DAG が完走する、Secrets.Get で OpenBao 上の値が取得できる、を代表検証とする。

**異常系観点**: 不正なリクエスト・存在しないキー・権限不足・期限切れトークン に対して tier1 が適切なエラーコード（400 / 401 / 403 / 404 / 429 / 500）を返す。エラーメッセージは PII マスキング済みで、内部実装詳細を漏らさない。クライアントが再試行可能か否かが `Retry-After` ヘッダで明示される。

**負荷観点**: 同時接続 1,000 セッションで tier1 API が p99 500ms 以内を維持する。負荷時の Pod オートスケール（HPA）が期待通り動作する。Kafka Producer が acks=all で Publish レイテンシ p99 50ms 以内を維持する。Valkey が 1 万 key 投入後も Get p99 10ms 以内を維持する。

**障害注入観点**: Chaos Mesh による障害注入で tier1 が期待通りに振る舞う。Valkey 1 シャード停止・Kafka 1 ブローカ停止・PostgreSQL Primary 障害・OpenBao 1 ノード停止 の 4 シナリオで、データ損失ゼロ・サービス継続・自動復旧の 3 項目を確認する。

**セキュリティ観点**: JWT なし・期限切れ JWT・別テナント JWT の不正アクセスが期待通り拒否される。mTLS なしの Pod 間通信が拒否される（Istio Ambient 採用時）。監査ログの UPDATE / DELETE が PostgreSQL ロールで拒否される。OpenBao シークレット読み取り権限なしで Secrets API が 403 を返す。

## 設計項目 DS-SW-IT-002 主要シナリオ 1 — State と Audit の連携

State.Set 操作時に Audit API が自動で呼ばれ、監査イベントが PostgreSQL WORM テーブルに記録される連携を検証する。このシナリオは「tier1 の内部で、複数コンポーネント間の連携が勝手に成立するか」を確認する中核である。

検証手順は以下である。第一に、tier2 相当のテストクライアントから State.Set を呼ぶ。第二に、Valkey 上に値が書かれたことを確認する。第三に、PostgreSQL の audit_events テーブルに対応イベントが書かれたことを確認する。第四に、イベントの hash_chain_prev が直前イベントの hash_chain_curr と一致することを確認する。第五に、イベントの PII 情報（user_id 等）がマスキング済みであることを確認する。

合格基準は、100 件の State.Set 連続実行で全 100 件の Audit イベントが記録され、hash_chain の整合性が 100% 維持されること。書き込み順序の逆転も禁止とし、audit_events の created_at が厳密に単調増加することを検証する。

## 設計項目 DS-SW-IT-003 主要シナリオ 2 — Decision 評価と結果キャッシュ

Decision.Evaluate 呼び出し時に ZEN Engine が JDM ルールを評価し、評価結果が Valkey にキャッシュされ、同一入力の 2 回目呼び出しはキャッシュから即応する連携を検証する。このシナリオはレスポンスタイム目標（Decision p99 1ms）の達成可否を測る重要路である。

検証手順は以下である。第一に、テストクライアントから同一入力で 100 回連続で Decision.Evaluate を呼ぶ。第二に、初回呼び出しが ZEN Engine を経由して 10ms 程度、2 回目以降がキャッシュから 1ms 以下で返ることを確認する。第三に、JDM ルールを更新した場合にキャッシュが正しく無効化され、再評価が走ることを確認する。第四に、キャッシュ TTL 満了時に再評価が走ることを確認する。

合格基準は、2 回目以降のキャッシュヒット時に p99 1ms 以内を維持すること、JDM ルール更新時のキャッシュ無効化が 5 秒以内に反映されること。キャッシュミス時でも p99 10ms 以内を維持することを副合格基準とする。

## 設計項目 DS-SW-IT-004 主要シナリオ 3 — Workflow のタスクキューと Activity

Workflow.Start 呼び出し時に Dapr Workflow がタスクキュー経由で Activity を起動し、3 ステップの DAG が完走する連携を検証する。このシナリオは非同期処理の信頼性を測る。

検証手順は以下である。第一に、テストクライアントから Workflow.Start を呼び、Activity 1 → Activity 2 → Activity 3 の DAG を開始する。第二に、各 Activity が期待順序で実行され、前段の結果が次段の入力として正しく渡ることを確認する。第三に、全 Activity 完了後に Workflow が Complete 状態となり、最終結果がクライアントに返ることを確認する。第四に、Workflow 途中でプロセス再起動を発生させ、再起動後も Workflow が中断地点から再開することを確認する。

合格基準は、100 件の Workflow 実行で 100 件が正しく完走すること、プロセス再起動テストで Workflow 状態が失われないこと（Dapr Workflow のイベントソーシングが機能していること）。

## 設計項目 DS-SW-IT-005 主要シナリオ 4 — Secret Lease と Rotation

Secrets.Get で取得した Secret が Lease 付きで返り、Lease 期限接近時に Rotation が走り、旧 Secret の利用が即時無効化される連携を検証する。このシナリオはセキュリティ要件の核である。

検証手順は以下である。第一に、テストクライアントから Secrets.Get で Secret を取得する。Lease 期限 24 時間とする。第二に、期限 1 時間前に tier1 が Rotation を発動し、新 Secret を発行することを確認する。第三に、新 Secret が払い出されて以降、旧 Secret での OpenBao 認証が拒否されることを確認する。第四に、Rotation イベントが Audit に記録されることを確認する。

合格基準は、Rotation が Lease 期限 1 時間前の範囲で確実に発動すること、旧 Secret の無効化が Rotation 後 60 秒以内に反映されること、Rotation イベントの Audit 記録漏れがゼロであること。

## 設計項目 DS-SW-IT-006 境界テスト — Go ⇔ Rust 間 gRPC 契約（Pact）

Go ファサード層と Rust 自作領域の gRPC 境界について、Pact による Consumer-Driven Contract Testing を実施する。この境界は構想設計 ADR-TIER1-002 で確定した「Protobuf gRPC のみ」の制約を実装レベルで担保する重要路である。

Pact のテスト構造は以下である。Consumer 側（Go ファサード層）が期待するリクエスト・レスポンスの契約（Pact ファイル）を生成する。Provider 側（Rust 自作領域）は Pact ファイルを読み込み、自身の実装が契約を満たすかを検証する。契約違反があれば Provider 側の CI が失敗する。

検証対象の gRPC メソッドは以下 8 つである。`Decision.Evaluate`（ZEN Engine 評価）、`Crypto.Encrypt` / `Crypto.Decrypt`（AES-GCM 暗号化）、`Audit.Append`（監査イベント追加）、`Audit.Verify`（hash_chain 整合性検証）、`Template.Generate`（CLI 雛形生成）、`JTC.Legacy.Invoke`（レガシーシステム連携）、`Feature.Evaluate`（flagd 評価結果変換）。

合格基準は、全 8 メソッドで Pact 契約テスト 100% パス、契約破壊的変更が検出された場合に自動で CI が失敗して main merge が阻止されること。`buf breaking` の静的チェックと組み合わせて、動的・静的の両面で契約変更を検出する。

## 設計項目 DS-SW-IT-007 境界テスト — tier1 公開 API の後方互換性

tier1 公開 API（11 種）の後方互換性を、旧 SDK クライアントで最新 tier1 サーバを呼ぶテストで検証する。この境界は tier2 / tier3 に対する契約であり、壊れると広範な影響が出る。

検証手順は以下である。第一に、旧バージョン（前 minor リリース）の tier1 SDK を Go / C# / TypeScript の各言語でビルドする。第二に、最新 tier1 サーバに対して旧 SDK から全 API を呼び出し、期待通りの結果が返ることを確認する。第三に、新規に追加された optional フィールドを旧 SDK が無視して正しく動作することを確認する。第四に、削除予定の deprecated API に対して deprecation warning が期待通り返ることを確認する。

合格基準は、前 minor リリースの SDK で最新 tier1 サーバの全 API が動作すること、破壊的変更を含む PR が自動検出されて main merge を阻止されること。破壊的変更が避けられない場合は major バージョンアップを必須とし、Phase 境界でのみ実施する。

## 設計項目 DS-SW-IT-008 テスト環境 — staging と負荷試験環境

結合テストは以下 3 環境で実施する。

**CI testcontainers 環境**: GitHub Actions 上で testcontainers による軽量統合テストを実行する。Valkey / Kafka / PostgreSQL / OpenBao を Docker コンテナで起動し、PR ごとに回す。実行時間は 15 分以内を目標とする。対象は正常系・異常系・境界テスト（Pact）の 3 観点。

**staging 環境**: 本番同等の OSS 構成（単一 VM × 3 の kubeadm クラスタ）で週次回帰を実施する。対象は 5 観点全て。実行時間は 2 時間以内。Chaos Mesh による障害注入は staging で実施し、本番への影響を遮断する。

**負荷試験環境**: staging と物理的に分離した専用環境（VM × 3）で月次負荷試験を実施する。k6 で RPS 勾配をかけ、tier1 の HPA 挙動と p99 レイテンシを測定する。対象は負荷観点のみ。

環境間でのテストデータは共通の合成データセット（個人情報 0 件、業務イメージを模した 10 万件規模）を使う。本番データの結合テスト環境への持ち込みは禁止である。

## 設計項目 DS-SW-IT-009 自動化範囲と実施頻度

結合テストの自動化範囲を明示する。手動実施が必要なテストは「自動化できない理由」と「実施頻度」を併記する。

- **CI testcontainers**: PR ごと、main merge ごとに自動実行。15 分以内。
- **週次回帰**: 毎週月曜 0:00 に Argo Workflows で自動起動。完了後に Slack に結果通知。
- **月次負荷試験**: 月末金曜夜 22:00 に自動起動。翌朝までに完了。結果は Grafana ダッシュボードで確認。
- **Chaos 障害注入**: 月 1 回の障害訓練日に運用者が手動で Chaos Mesh のシナリオを選択実行。手動である理由は、意図しない障害連鎖の早期検知と判断が人間に委ねられる必要があるため。
- **後方互換性テスト**: 前 minor リリース SDK が tier1 に追加されるたびに CI 自動実行。

Phase 1b 完了判定時は、上記全テストの合格を Phase Gate として固定する。合格判定は Product Council が最終承認する。

## 設計項目 DS-SW-IT-010 合格基準のまとめ

本章で規定した合格基準を集約する。以下の全ての基準を満たして初めて、ソフトウェア結合テスト合格となる。

- 契約テスト（Pact）: 8 メソッドで 100% パス
- 後方互換性テスト: 前 minor リリース SDK で全 11 API 動作
- 正常系シナリオ: 4 主要シナリオ（State+Audit / Decision+Cache / Workflow / Secret Rotation）で 100 件試行中 100 件成功
- 異常系シナリオ: 不正リクエスト 6 種で期待エラーコード 100% 返却
- 負荷試験: 11 API の p99 500ms 以内、1,000 RPS で維持
- 障害注入: 4 シナリオで自動復旧 + データ損失ゼロ
- セキュリティ: 不正 JWT / mTLS / Secret 権限 の 3 カテゴリで 100% 拒否
- Audit hash_chain 整合性: 10,000 件連続投入でエラーゼロ
- 自動化率: 正常系・異常系・境界テスト・負荷試験で 95% 以上自動

合格基準の一つでも満たさない場合、Phase 1b 完了判定は保留となる。是正対応は基盤チームが実施し、合格確認の再テストを実行する。

## 設計項目 DS-SW-IT-011 Phase 別実施計画

結合テストは Phase 進行に応じて実施範囲を拡張する。Phase 境界での合格基準を明示する。

- **Phase 0（稟議時点）**: 本章の骨子確定、テスト観点・合格基準の定義。実テスト着手前。
- **Phase 1a（MVP-0）**: 正常系と異常系の CI testcontainers テストを整備。staging は未使用。
- **Phase 1b（MVP-1a）**: 全 5 観点・4 主要シナリオ・境界テストの staging 回帰を確立。負荷試験開始。
- **Phase 1c（MVP-1b）**: Chaos 障害注入シナリオ拡充、後方互換性テスト自動化、Audit hash_chain の 10 万件検証。
- **Phase 2 以降**: Istio Ambient / Temporal / Backstage を含めた結合テスト範囲拡張。マルチクラスタ対応。

各 Phase 移行時に結合テスト計画書を Product Council に提示し、次 Phase 着手前に合格基準をレビューする。
