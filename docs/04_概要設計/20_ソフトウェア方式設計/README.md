# 20. ソフトウェア方式設計

本カテゴリは IPA 共通フレーム 2013 の **ソフトウェア方式設計プロセス（7.1.2）** に厳密対応する。tier1 ソフトウェアを個別コンポーネントまで分割し、相互関係とインタフェース・データベース最上位構造・利用者文書を方式として固定化する層である。

## 本カテゴリの位置付け

前章 [../10_システム方式設計/](../10_システム方式設計/) がシステム全体のトポロジ・処理形態・データ全体方式を確立したのを受け、本章は tier1 ソフトウェア内部をコンポーネントまで分解する。Dapr ファサード層（Go）と自作領域（Rust）のハイブリッド構成のもと、各コンポーネントが何の責務を持ち、どのインタフェースで接続するかを仕様として固定化する。

構想設計書（[../../02_構想設計/02_tier1設計/](../../02_構想設計/02_tier1設計/)）が tier1 の設計思想・Go+Rust 分担原則・API 設計原則の**根拠**を既に確立しているため、本章はその設計思想を**方式設計書本体**として具体化する。個別コンポーネント名・責務・インタフェース契約を一つずつ採番し、要件 ID との対応を確立することが本章の成果である。

## IPA 共通フレーム 7.1.2 との対応

共通フレーム 2013 のソフトウェア方式設計プロセスは 6 アクティビティで構成される。本章のサブディレクトリは各アクティビティに 1:1 対応する。

- **7.1.2.1 ソフトウェア構造とコンポーネントの方式設計** → `01_コンポーネント方式設計/`
- **7.1.2.2 外部及びコンポーネント間のインタフェース方式設計（外部側）** → `02_外部インタフェース方式設計/`
- **7.1.2.2 外部及びコンポーネント間のインタフェース方式設計（内部側）** → `03_内部インタフェース方式設計/`
- **7.1.2.3 データベースの最上位レベル設計** → `04_データベース最上位設計/`
- **7.1.2.4 利用者文書（暫定版）の作成** → `05_利用者文書_暫定版/`
- **7.1.2.5 ソフトウェア結合テスト要件定義** → `06_ソフトウェア結合テスト要件.md`
- **7.1.2.6 ソフトウェア方式設計の評価・共同レビュー** → `07_ソフトウェア方式設計評価レビュー.md`

## 格納予定ドキュメント

サブディレクトリ単位で章を構成する。各サブディレクトリは 300 行制限に収まるよう必要に応じてファイルを分割する。

### 01_コンポーネント方式設計/

tier1 を構成するコンポーネントを列挙し、責務・依存関係・パッケージ構成を固定化する。Dapr ファサード層のコンポーネント（State / PubSub / Binding / Workflow / Secrets / Service Invoke 等）と自作 Rust 領域のコンポーネント（ZEN Engine 統合 / crypto / 雛形 CLI / JTC 固有機能）の境界を明示する。格納予定ファイルは以下。

- `01_tier1全体コンポーネント俯瞰.md`
- `02_Daprファサード層コンポーネント.md`
- `03_自作Rust領域コンポーネント.md`
- `04_コンポーネント責務一覧.md`
- `05_モジュール依存関係.md`
- `06_パッケージ構成_Rust_Go.md`

### 02_外部インタフェース方式設計/

tier2 / tier3 に公開する外部インタフェースを方式として固定化する。要件定義側の [../../03_要件定義/20_機能要件/40_tier1_API契約IDL/](../../03_要件定義/20_機能要件/40_tier1_API契約IDL/) で採番された IDL を概要設計視点で束ね、バージョニング方針・エラー契約・認証契約を統合する。格納予定ファイルは以下。

- `01_tier1_11API方式概要.md`
- `02_APIバージョニング方式.md`
- `03_認証認可インタフェース方式.md`
- `04_エッジIoTインタフェース方式.md`
- `05_レガシー連携インタフェース方式.md`

### 03_内部インタフェース方式設計/

tier1 内部のコンポーネント間通信を方式として固定化する。プロジェクト規約で「tier1 内部のサービス間通信は Protobuf gRPC 必須」と定めているため、内部 gRPC 契約とイベントスキーマを統一的に管理する。格納予定ファイルは以下。

- `01_内部gRPC契約方式.md`
- `02_イベントスキーマ方式.md`
- `03_Go_Rust間言語境界方式.md`

### 04_データベース最上位設計/

論理 ER モデルとテーブル一覧を最上位レベルで固定化する。詳細な物理設計は後続の詳細設計フェーズで展開するため、本章はエンティティ・関連・主要属性・保持方針までを扱う。格納予定ファイルは以下。

- `01_論理ERモデル.md`
- `02_最上位テーブル一覧.md`
- `03_データ保持と廃棄方式.md`

### 05_利用者文書_暫定版/

共通フレーム 7.1.2.4 が明示的に要求する成果物である。k1s0 の利用者は tier2 / tier3 の開発者と情シス運用者であり、以下の文書初版を格納する。本格版は稼働直前に確定させる。格納予定ファイルは以下。

- `01_tier2_tier3開発者向けGoldenPath初版.md`
- `02_運用者手引き初版.md`
- `03_Backstageポータル利用初版.md`
- `04_アプリ配信ポータル利用初版.md`
- `05_CLI利用初版.md`

### 06_ソフトウェア結合テスト要件.md

tier1 内部コンポーネント同士の結合テスト要件を定義する。対象範囲、検証観点、合格基準、モック戦略、環境要件を含む。[../10_システム方式設計/05_システム結合テスト要件.md](../10_システム方式設計/) とは階層が異なるため、本章の結合は tier1 内部のみを対象とする。

### 07_ソフトウェア方式設計評価レビュー.md

01〜06 のレビュー結果、指摘事項、是正対応、承認記録を格納する。アーキテクトレビュー・Product Council 承認のエビデンス文書として扱う。

## 執筆方針

- 各ファイルは共通フレーム 7.1.2 の該当アクティビティ番号を冒頭に明示する。
- コンポーネント採番は `DS-SW-COMP-*`、外部 IF 採番は `DS-SW-EIF-*`、内部 IF 採番は `DS-SW-IIF-*`、DB 最上位は `DS-SW-DB-*` を用いる。
- 構想設計 ADR を参照する箇所は ADR 番号を明記する。ADR と異なる方式を採用する場合は必ず新規 ADR を起票する。
- 図は各サブディレクトリの `img/` ではなく、本カテゴリ直下の [img/](img/) に集約する。コンポーネント俯瞰図・依存関係図・シーケンス図は drawio レイヤ記法規約に従う。
