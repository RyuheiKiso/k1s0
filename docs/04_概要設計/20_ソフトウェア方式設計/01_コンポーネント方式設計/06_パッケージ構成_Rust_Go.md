# 06. パッケージ構成（Rust / Go）

本ファイルは IPA 共通フレーム 2013 の **ソフトウェア方式設計プロセス 7.1.2.1** に対応し、tier1 の Rust Cargo workspace 構成、Go module 構成、Protobuf 自動生成の配置、ビルドターゲット、container image 構成を方式として固定化する。

## 本ファイルの位置付け

[05_モジュール依存関係.md](05_モジュール依存関係.md) で依存方向が決まった前提の下、本ファイルは具体的なディレクトリ階層とビルド単位を確定させる。パッケージ構成は一度決めると変更コストが高い（Import パス変更が全 Pod のビルドに影響する）ため、Phase 0 段階でできる限り安定した形を固定する。構成を曖昧にしたまま Phase 1a に入ると、開発チームが Pod ごとにバラバラのレイアウトを採用し、結合テストで組み合わせ爆発が発生する。本ファイルはその類の発散を上流で防ぐ役割を担う。

特に Protobuf 自動生成の配置は、Go / Rust 双方で同じ `.proto` から生成される型を参照するため、単一の真実（`src/tier1/contracts/`）から両言語に派生する構造を厳密に固定する必要がある。これが乱れると Go 側が新しい型を使い、Rust 側が古い型を使う状態が発生し、内部 gRPC で型不整合エラーが runtime に発生する。

## 設計 ID 一覧と採番方針

本ファイルで採番する設計 ID は `DS-SW-COMP-120` 〜 `DS-SW-COMP-139` の 20 件である。通番は [05_モジュール依存関係.md](05_モジュール依存関係.md) の DS-SW-COMP-119 から連続し、このファイルで主要な `DS-SW-COMP-*` カテゴリの採番を終える。なお [02_Daprファサード層コンポーネント.md](02_Daprファサード層コンポーネント.md) に 6 段階多層防御統括として `DS-SW-COMP-141` を補欠採番しており（050〜140 の連番は他ファイルで全消費済みのため、最小未使用番号として 141 を充てる）、それ以降の追加採番は `DS-SW-COMP-142` から連番とする。

## ルートディレクトリ構成

tier1 のソースコードは `src/tier1/` 配下に集約し、Go・Rust の 2 サブディレクトリに分かれる。Protobuf 契約は ADR-DIR-001 により `src/contracts/` にルート昇格、Dapr / Kubernetes manifest は ADR-DIR-002 によりルート `infra/`・`deploy/`・`ops/` の 3 階層に分離された。この結果、`src/tier1/` は tier1 実装者（tier1-rust-dev / tier1-go-dev）が編集する範囲だけを含む形に整理される。

### DS-SW-COMP-120 tier1 のトップレベル構成（ADR-DIR-002 により改訂）

`src/tier1/` は以下の 2 サブディレクトリを持つ。`go/` は Go ソース（Go module root）、`rust/` は Rust ソース（Cargo workspace root）。旧構造にあった `contracts/` は `src/contracts/` にルート昇格し（[../../../05_実装/00_ディレクトリ設計/20_tier1レイアウト/02_contracts配置.md](../../../05_実装/00_ディレクトリ設計/20_tier1レイアウト/02_contracts配置.md) 参照）、`infra/` は廃止してルート `infra/`（クラスタ素構成）・`deploy/`（GitOps 配信定義）・`ops/`（Runbook/Chaos/DR）に分離された（[../../../05_実装/00_ディレクトリ設計/50_infraレイアウト/01_infra全体配置.md](../../../05_実装/00_ディレクトリ設計/50_infraレイアウト/01_infra全体配置.md) 参照）。docs / スクリプト類は従来通り `docs/` / `tools/` に配置する。

```
src/tier1/
├── go/              # Go module（Dapr ファサード層）
└── rust/            # Cargo workspace（自作領域）
```

上位参照:
- 契約: `src/contracts/`（ADR-DIR-001、IMP-DIR-T1-022）
- インフラ素構成: `infra/`（ADR-DIR-002、IMP-DIR-INFRA-071）
- GitOps 配信: `deploy/`（IMP-DIR-OPS-091）
- 運用 Runbook: `ops/`（IMP-DIR-OPS-095）

**確定フェーズ**: Phase 1a。**対応要件**: DX-CICD-\*、NFR-C-NOP-002。**改訂履歴**: ADR-DIR-001（contracts 昇格）、ADR-DIR-002（infra 分離）により 2026-04 に改訂。

## Protobuf 契約の配置

Protobuf は Go / Rust / .NET / TypeScript の 4 言語 SDK に入力される単一の真実（source of truth）として `src/contracts/` に配置する（ADR-DIR-001 により `src/tier1/contracts/` からルート昇格）。

### DS-SW-COMP-121 contracts/ の内部構成（ADR-DIR-001 により改訂）

`src/contracts/` は以下の階層で構成する。buf module 境界は `src/contracts/` 直下。tier1 公開 11 API の .proto は `src/contracts/tier1/v1/` に、tier1 内部 gRPC（Go facade → Rust）は `src/contracts/internal/v1/` に分離する。共通型 `common.proto`・エラー型 `errors.proto` は各 service 配下で共有する。`buf.yaml` / `buf.gen.yaml` / `buf.lock` は `src/contracts/` 直下に置く。

```
│       ├── feature.proto
│       ├── secrets.proto
│       ├── workflow.proto
│       ├── log.proto
│       ├── telemetry.proto
│       ├── decision.proto
│       ├── audit.proto
│       └── pii.proto
└── internal/
    └── v1/
        └── facade.proto           # Go facade → Rust 内部 gRPC
```

実装詳細は [../../../05_実装/00_ディレクトリ設計/20_tier1レイアウト/02_contracts配置.md](../../../05_実装/00_ディレクトリ設計/20_tier1レイアウト/02_contracts配置.md) 参照。

**確定フェーズ**: Phase 1a。**対応要件**: ADR-TIER1-002、DX-CICD-\*。**改訂履歴**: ADR-DIR-001（contracts 昇格）により 2026-04 に改訂。

### DS-SW-COMP-122 Protobuf 生成コード配置（ADR-DIR-001 により改訂）

`buf generate` は同一の `src/contracts/` を源として、tier1 系統と SDK 系統の 2 つに**独立して**出力する。tier1 系統は tier1 内部の gRPC と公開 API 実装用、SDK 系統は tier2 / tier3 向け配布用である。両系統が並立する理由は依存方向ルール（[../../../05_実装/00_ディレクトリ設計/10_ルートレイアウト/05_依存方向ルール.md](../../../05_実装/00_ディレクトリ設計/10_ルートレイアウト/05_依存方向ルール.md)）で tier1 は sdk を参照しないと定めているためで、tier1 が `src/sdk/<lang>/` を import すると依存方向違反となる。tier1 は自層内に生成コードを保持する。

**tier1 系統（tier1 内部の gRPC・公開 API 実装で使用）**

- Go: `src/tier1/go/internal/proto/v1/` 配下に `*.pb.go` / `*_grpc.pb.go`
- Rust: `src/tier1/rust/crates/proto-gen/src/v1/` 配下に `mod.rs` + 各 service 別モジュール

**SDK 系統（tier2 / tier3 からの消費で使用）**

- Go: `src/sdk/go/gen/` 配下に `*.pb.go` / `*_grpc.pb.go`
- Rust: `src/sdk/rust/src/gen/` 配下に `mod.rs` + 各 service 別モジュール
- .NET: `src/sdk/dotnet/K1s0.Sdk/Generated/`
- TypeScript: `src/sdk/typescript/src/gen/`

生成コードは tier1 系統・SDK 系統ともに全て git 管理する（CI での `buf generate` 結果を commit してレビュー可能にする）。生成ファイルは `// Code generated by protoc. DO NOT EDIT.` ヘッダを持ち、手動編集は禁止する。両系統の drift（同一 `src/contracts/` から生成したにもかかわらず内容が乖離する状態）は `tools/codegen/check-drift.sh` で CI 必須検証する。実装詳細は [../../../05_実装/00_ディレクトリ設計/20_tier1レイアウト/06_生成コードの扱い.md](../../../05_実装/00_ディレクトリ設計/20_tier1レイアウト/06_生成コードの扱い.md) 参照。

**確定フェーズ**: Phase 1a。**対応要件**: ADR-TIER1-002、DX-CICD-\*。**改訂履歴**: ADR-DIR-001（contracts ルート昇格）により 2026-04 に改訂。当初の改訂では「tier1 は SDK を import する」と記述していたが、依存方向ルール（tier1 ⇏ sdk）との整合のため「tier1 と SDK は同一 contracts から独立生成する 2 系統並立」の方式に再訂正（2026-04-23）。

### DS-SW-COMP-123 buf 設定方針

`buf.yaml` では以下を指定する。`version: v2`、`modules: - path: .`、`lint: use: [STANDARD]`（標準 lint ルール強制）、`breaking: use: [FILE]`（ファイル単位での破壊的変更検出）。`buf.gen.yaml` では Go 生成（`buf.build/protocolbuffers/go` + `buf.build/grpc/go`）と Rust 生成（`buf.build/community/neoeinstein-prost` + `buf.build/community/neoeinstein-tonic`）の 2 plugin を設定する。`buf dep update` で BSR（Buf Schema Registry）の外部依存（googleapis 等）を更新する。

**確定フェーズ**: Phase 1a。**対応要件**: ADR-TIER1-002、DX-CICD-\*。

## Go module 構成

Go 側は `src/tier1/go/` を Go module root（`go.mod` が直下）として、Dapr ファサード層 3 Pod の実装を集約する。

### DS-SW-COMP-124 Go module 全体レイアウト

Go module は標準的な Go project layout を踏襲する。`cmd/` 配下に Pod ごとの `main.go` を置き、`internal/` 配下に共通 package と Pod 固有実装を置く。`pkg/` は Phase 1b 以降で tier2 / tier3 用クライアントライブラリを出す予定。`go.mod` のモジュール名は `github.com/k1s0/k1s0/src/tier1/go`（Phase 1b 以降に正式リポジトリ名で確定）。

```
src/tier1/go/
├── go.mod
├── go.sum
├── cmd/
│   ├── state/
│   │   └── main.go
│   ├── secret/
│   │   └── main.go
│   └── workflow/
│       └── main.go
├── internal/
│   ├── adapter/
│   │   ├── dapr/         # Dapr Adapter モジュール
│   │   ├── log/          # Log Adapter
│   │   └── metrics/      # Metrics Emitter
│   ├── common/           # k1s0-common 相当
│   ├── policy/           # k1s0-policy 相当（Policy Enforcer）
│   ├── proto/v1/         # buf 生成コード
│   ├── otel/             # k1s0-otel 相当
│   ├── state/            # COMP-T1-STATE 固有実装
│   ├── secret/           # COMP-T1-SECRET 固有実装
│   └── workflow/         # COMP-T1-WORKFLOW 固有実装
└── pkg/                  # Phase 1b 以降の public API（予約）
```

**確定フェーズ**: Phase 1a。**対応要件**: DX-CICD-\*、NFR-C-NOP-002。

### DS-SW-COMP-125 cmd/ 配下の Pod entrypoint

各 Pod の `main.go` は 最小限の起動コードのみ含み、実装は `internal/<pod>/` に委譲する。`main.go` の構造は「設定ロード（viper） → OTel 初期化 → Policy 初期化 → Dapr Adapter 初期化 → API Router 起動 → graceful shutdown」の順で共通化する。`main.go` は 100 行以内とし、ロジックは internal/ に置く（300 行制限は docs 側のみで source code は短い方が望ましい）。

**確定フェーズ**: Phase 1a。**対応要件**: DX-GP-\*、DX-TEST-\*。

### DS-SW-COMP-126 internal/ 配下の package 分割

`internal/` は責務単位で分割する。`adapter/dapr` は Dapr Go SDK をラップし、各 Pod から共通 interface で呼び出せるようにする。`adapter/log` / `adapter/metrics` も同様。`common` / `policy` / `otel` は [05_モジュール依存関係.md](05_モジュール依存関係.md) の共通ライブラリと 1:1 対応する。`state/` / `secret/` / `workflow/` は Pod 固有実装で、ほかの Pod からは import 禁止（Go の internal 機構で強制）。

Protobuf 生成コードは `src/tier1/go/internal/proto/v1/` 配下に tier1 内部の責務として保持する（依存方向ルールにより tier1 は `src/sdk/go/` を import できないため、tier1 用の生成コードは tier1 内部に独立して持つ）。各 Pod は `github.com/k1s0/k1s0/src/tier1/go/internal/proto/v1/...` を import する。SDK Go `src/sdk/go/gen/` とは同一 `src/contracts/` を源とするが生成先は別系統であり、両者の drift は `tools/codegen/check-drift.sh` で CI 検証する（DS-SW-COMP-122 参照）。

**確定フェーズ**: Phase 1a。**対応要件**: NFR-C-NOP-002、DX-CICD-\*。

### DS-SW-COMP-127 Go ビルドターゲット

`go build` は Pod ごとに 1 バイナリを出力する。ビルドコマンドは `go build -ldflags="-s -w" -trimpath -o bin/t1-state ./cmd/state/` の形式で、静的リンク・debug info 削除・path 正規化を強制する。cross-compile は不要（Linux amd64 と arm64 のみサポート）で、arm64 対応は Phase 2 で検討する。`CGO_ENABLED=0` で pure Go ビルドとし、glibc 依存を避けて distroless container でも動作させる。

**確定フェーズ**: Phase 1a。**対応要件**: DX-CICD-\*、NFR-F-ENV-\*。

### DS-SW-COMP-128 Go Docker image 構成

各 Pod の container image は multi-stage build で作成する。第 1 stage で `golang:1.22` 上でビルド、第 2 stage で `gcr.io/distroless/static-debian12:nonroot` にバイナリのみコピーする。image サイズ目標は 30 MiB 以下で、Pod startup 時間短縮とパッチ適用範囲最小化に寄与する。image tag は semver + git commit hash（`v0.1.0-abc123`）で構成し、Phase 1b 以降の Argo Rollouts と連動する。

**確定フェーズ**: Phase 1a。**対応要件**: DX-CICD-\*、NFR-F-ENV-\*、NFR-E-ENC-001。

## Rust Cargo workspace 構成

Rust 側は `src/tier1/rust/` を Cargo workspace root として、自作 3 Pod と共通 crate を集約する。

### DS-SW-COMP-129 Cargo workspace 全体レイアウト

Cargo workspace は以下の構造で構成する。`Cargo.toml`（workspace 設定）、`rust-toolchain.toml`（stable/1.85 以上ピン）、`.cargo/config.toml`（rustflags・registry 設定）を直下に置き、各 crate は `crates/` 配下に配置する。workspace `members` は crate 名を明示列挙し、glob `crates/*` は使わない（意図しない crate 追加を防ぐ）。`proto-gen` crate は tier1 用 Protobuf 生成コードを集約する lib crate として保持する（依存方向ルールにより tier1 は `src/sdk/rust/` を `path` 依存できないため、tier1 内部で独立に contracts から生成する）。

```
src/tier1/rust/
├── Cargo.toml          # workspace 設定
├── Cargo.lock
├── rust-toolchain.toml
├── .cargo/
│   └── config.toml
└── crates/
    ├── audit/          # COMP-T1-AUDIT 本体
    ├── decision/       # COMP-T1-DECISION 本体
    ├── pii/            # COMP-T1-PII 本体
    ├── common/         # k1s0-common
    ├── proto-gen/      # tier1 用 Protobuf 生成コード
    ├── otel-util/      # k1s0-otel
    └── policy/         # k1s0-policy
```

各 bin crate（audit / decision / pii）は Cargo.toml で `proto-gen = { path = "../proto-gen" }` を dependency に加え、tier1 内部の Protobuf 生成型を取得する。`k1s0-sdk`（`src/sdk/rust/`）への `path` 依存は禁止する（依存方向ルール違反となるため、CI の cargo-deny + 自作 linter で検出）。

**確定フェーズ**: Phase 1a。**対応要件**: DX-CICD-\*、NFR-C-NOP-002。

### DS-SW-COMP-130 workspace Cargo.toml 方針

`[workspace]` セクションでは `resolver = "2"`、`members = [...]`（7 crate 明示）、`[workspace.package]` で Rust Edition 2024 / version / authors / license を共有する。`[workspace.dependencies]` で外部 crate のバージョンを集約し、子 crate は `tokio.workspace = true` の形式で参照する。これによりバージョン不整合を構造的に排除する。外部 crate は `serde 1.0`、`tokio 1.x`、`tonic 0.12`、`prost 0.13`、`tracing 0.1`、`opentelemetry 0.27`、`prometheus 0.13`、`sqlx 0.8`（AUDIT）、`rdkafka 0.36`（AUDIT）、`zen-engine 0.37`（DECISION）、`regex 1.10`（PII）、`sha2 0.10`、`aes-siv 0.7`（PII）等で pin する。

**確定フェーズ**: Phase 1a。**対応要件**: DX-CICD-\*、NFR-E-ENC-001。

### DS-SW-COMP-131 各 crate の内部構造

各 crate は標準的な Rust crate 構造を取る。`Cargo.toml` で `[package]` と `[dependencies]`、`src/lib.rs` または `src/main.rs`、`src/<module>/` で module 分割する。`audit` / `decision` / `pii` の 3 crate は bin crate（`src/main.rs`）で、それぞれ Pod のエントリポイントとなる。`common` / `proto-gen` / `otel-util` / `policy` は lib crate（`src/lib.rs`）で、bin crate から import される。test は `tests/` と `src/**/*.rs` 内の `#[cfg(test)]` の両方を使う。

**確定フェーズ**: Phase 1a。**対応要件**: DX-CICD-\*、DX-TEST-\*。

### DS-SW-COMP-132 SDK Rust crate の構成（ADR-DIR-001 により改訂）

SDK 系統の Rust 生成コードは `src/sdk/rust/` の `k1s0-sdk` crate として公開される（tier2 / tier3 から消費される配布単位）。`src/sdk/rust/build.rs` で `tonic-build` を実行し、`src/contracts/tier1/v1/*.proto` のうち tier1 公開 11 API の proto から Rust コードを生成する。`src/contracts/internal/v1/*.proto` は tier1 facade→Rust の内部 gRPC 用であり SDK には含めない（SDK を経由して tier1 内部構造が tier2 / tier3 に露出するのを防ぐ）。生成コードは `OUT_DIR` に出力され、`src/gen/mod.rs` から `include!(concat!(env!("OUT_DIR"), "/k1s0.tier1.v1.state.rs"))` で取り込む。

tier1 Rust 側の生成コード（`src/tier1/rust/crates/proto-gen/src/v1/`、DS-SW-COMP-129 参照）と SDK Rust 側の生成コード（`src/sdk/rust/src/gen/`）は同一の `src/contracts/` から独立生成される別系統である。両系統の drift は `tools/codegen/check-drift.sh` で CI 必須検証する。tier1 Rust 各 bin crate（audit / decision / pii）は tier1 内部の `proto-gen` crate を dependency に加え、`k1s0-sdk` には依存しない（依存方向ルールにより禁止、cargo-deny + 自作 linter で検出）。

**確定フェーズ**: Phase 1a。**対応要件**: ADR-TIER1-002、DX-CICD-\*。**改訂履歴**: ADR-DIR-001 により SDK Rust crate として独立公開、2026-04 改訂。当初の改訂では「tier1 は k1s0-sdk に依存する」と記述していたが、依存方向ルール（tier1 ⇏ sdk）との整合のため「tier1 は proto-gen crate を保持、SDK は独立配置」の 2 系統並立方式に再訂正（2026-04-23）。

### DS-SW-COMP-133 Rust ビルドターゲット

`cargo build --release --workspace` で全 7 crate をビルドする。`Cargo.toml` の `[profile.release]` は `lto = "thin"` / `codegen-units = 1` / `strip = true` / `panic = "abort"` を指定し、バイナリサイズと実行速度を最適化する。クロスコンパイルは不要（Linux amd64 のみ Phase 1a、arm64 は Phase 2）。`CARGO_TARGET_DIR` は CI の sccache とアラインさせる。

**確定フェーズ**: Phase 1a。**対応要件**: DX-CICD-\*、NFR-F-ENV-\*、NFR-B-PERF-\*。

### DS-SW-COMP-134 Rust Docker image 構成

各 Pod の container image は multi-stage build で作成する。第 1 stage で `rust:1.85` 上でビルド、第 2 stage で `gcr.io/distroless/cc-debian12:nonroot` にバイナリのみコピーする（`cc-debian12` は glibc 含むため `static-debian12` ではなくこちらを選択、`jemalloc` の FFI 依存があるため）。image サイズ目標は 80 MiB 以下で、Go 側（30 MiB）より大きいのは libc / jemalloc 同梱のため。image tag は semver + git commit hash で Go 側と同じ命名規約を採用する。

**確定フェーズ**: Phase 1a。**対応要件**: DX-CICD-\*、NFR-F-ENV-\*。

## CI/CD でのビルド構成

### DS-SW-COMP-135 monorepo の選択的ビルド方式（ADR-DIR-001 により path 更新）

`src/tier1/go/` と `src/tier1/rust/` の変更は独立に検知し、該当する言語側のみビルドする。`.github/workflows/` で `paths-filter` action を用い、`src/tier1/go/**` 変更時は tier1 Go ワークフロー、`src/tier1/rust/**` 変更時は tier1 Rust ワークフロー、`src/contracts/**` 変更時は tier1（Go / Rust）・SDK（4 言語）・tier2・tier3 の全系統を trigger する（旧 `src/tier1/contracts/` から昇格された path）。`src/sdk/<lang>/**` 変更時は該当言語の SDK ビルドおよび tier2 / tier3 の関連ワークフローを trigger する（tier1 は sdk に依存しないため tier1 は trigger 対象外、依存方向整合）。ビルド時間目標は Go で 3 分、Rust で 10 分（sccache あり）以内。

**確定フェーズ**: Phase 1a。**対応要件**: DX-CICD-\*、DX-MET-\*。**改訂履歴**: ADR-DIR-001 により path filter 規則を更新、2026-04 改訂。

### DS-SW-COMP-136 テスト方針

Go は `go test -race ./...` で race detector を有効化して全 unit test を実行する。Rust は `cargo nextest run` で高速並列実行する（`cargo test` より 2-3 倍高速）。Integration test は `integrationtest/` 配下（Go）と `tests/` 配下（Rust）で、Dapr / Kafka / PG を testcontainers で起動して実施する。coverage 目標は全体 80% / 公開 API 関数 100%（`cargo llvm-cov` + `go test -cover`）。

**確定フェーズ**: Phase 1a（unit）、Phase 1b（integration）、Phase 1c（coverage）。**対応要件**: DX-TEST-\*、DX-CICD-\*。

### DS-SW-COMP-137 container レジストリ

Phase 1a では GitHub Container Registry（`ghcr.io/k1s0/`）を使い、Phase 1b 以降で社内プライベート Harbor に移行する（閉域運用の前提、制約 8）。image は Cosign で署名し、Kubernetes 側で `policy-controller`（Sigstore）により署名検証を必須化する（[../../50_非機能方式設計/](../../50_非機能方式設計/) 参照）。SBOM は `syft` で各 image に同梱し、脆弱性スキャンは `grype` で定期実行する。

**確定フェーズ**: Phase 1a/1b。**対応要件**: DX-CICD-\*、NFR-E-ENC-001、NFR-H-KEY-001、制約 8。

## パッケージ構成変更の方針

### DS-SW-COMP-138 パッケージ構成変更の手続

`go.mod` module path 変更・Rust workspace 構成変更・crate 名変更は ADR 起票を要する。変更は全 Pod の再ビルドと deployment rollout を要し、Phase 1b の 2 名運用中は四半期に 1 回以下に頻度を抑える。変更時は移行ガイド（`docs/90_knowledge/` 配下）を同 PR で更新し、既存 PR に対する rebase 手順を明示する。

**確定フェーズ**: Phase 0。**対応要件**: NFR-C-NOP-001、DX-CICD-\*。

### DS-SW-COMP-139 Phase 進展に応じた構成拡張

Phase 2 以降のコンポーネント追加・新言語導入（例: Dapr Rust SDK stable 到達時の Rust 化判断）では、本ファイルを改訂して構成を再定義する。Phase 2 で想定される拡張は (a) `pkg/` 配下に tier2/3 用クライアントライブラリを Go / Rust 両言語で公開、(b) `cmd/` に新 Pod（例: COMP-T1-FEATURE 独立 Pod）追加、(c) `crates/` に新 crate（例: `k1s0-metrics-aggregator`）追加の 3 パターン。いずれも ADR 起票と monorepo 構造維持を前提とする。

**確定フェーズ**: Phase 2。**対応要件**: DS-SW-COMP-019（再評価条件）、NFR-C-NOP-002。
