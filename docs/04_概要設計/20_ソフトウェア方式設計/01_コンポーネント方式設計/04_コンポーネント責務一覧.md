# 04. コンポーネント責務一覧

本ファイルは IPA 共通フレーム 2013 の **ソフトウェア方式設計プロセス 7.1.2.1** に対応し、tier1 6 コンポーネント × 責務マトリクス、責務境界（どこまで tier1 が、どこから tier2 が担うか）、コンポーネント個別の SLA/SLO コミットメント、スケール戦略の選定理由を方式として固定化する。

## 本ファイルの位置付け

[01_tier1全体コンポーネント俯瞰.md](01_tier1全体コンポーネント俯瞰.md) が Pod の配置と運用形態を確定し、[02_Daprファサード層コンポーネント.md](02_Daprファサード層コンポーネント.md) と [03_自作Rust領域コンポーネント.md](03_自作Rust領域コンポーネント.md) が各 Pod 内部の実装方式を固定した。本ファイルはその上で「各コンポーネントが何を担い、何を担わないか」の責務境界を明示的な契約として確定させる。

責務境界の曖昧さは運用フェーズで致命的な影響を持つ。例えば「Audit-Pii API の PII 判定はどこまで tier1 が、どこから tier2 が責務を持つか」が不明確だと、規制監査時に「tier1 がマスクしていないから tier2 が悪い」という押し付け合いに陥る。本ファイルはこの類の曖昧さを設計段階で潰し、運用 Runbook・インシデント対応・監査応答の一貫性を担保する。

## 設計 ID 一覧と採番方針

本ファイルで採番する設計 ID は `DS-SW-COMP-080` 〜 `DS-SW-COMP-099` の 20 件である。通番は [03_自作Rust領域コンポーネント.md](03_自作Rust領域コンポーネント.md) の DS-SW-COMP-079 から連続し、[05_モジュール依存関係.md](05_モジュール依存関係.md) で DS-SW-COMP-100 に継続する。

## 責務マトリクスの読み方

下記の責務マトリクスは 6 コンポーネント × 11 責務カテゴリで構成される。各セルは「主担当」「副担当」「境界越境」の 3 状態を取る。主担当はそのコンポーネントが責務を完全に持ち、API 呼び出し時は必ずそのコンポーネントが処理する。副担当は他コンポーネントから内部 gRPC で呼び出される形で補助する。境界越境はそのコンポーネントから外側（tier2 / infra）に責務が出ることを示す。

### DS-SW-COMP-080 11 責務カテゴリの定義

責務カテゴリは以下 11 項目で、要件定義（FR-T1-\*）と非機能（NFR-\*）の両軸で定める。

- **R1 公開 API 受信**: gRPC エンドポイントで tier2 / tier3 のリクエストを受ける責務
- **R2 認証・認可判定**: JWT 検証・RBAC・tenant 境界確認
- **R3 バックエンド接続**: OSS（Valkey / Kafka / OpenBao / PG / MinIO）への接続と呼び出し
- **R4 業務ロジック評価**: ZEN Engine 評価、hash chain 生成、PII 検出など独自実装
- **R5 永続化**: 書込系バックエンド（PG / Kafka / MinIO）への永続化
- **R6 非同期イベント発火**: Kafka に監査・メトリクス・業務イベントを publish
- **R7 観測可能性**: OTel span / Prometheus metric / 構造化ログの生成
- **R8 ライフサイクル管理**: Pod 起動・停止・graceful shutdown・health check
- **R9 テナント設定取得**: Feature flag / Decision rule / PII rule のキャッシュ管理
- **R10 エラー変換**: 内部エラーを gRPC google.rpc.Status に変換
- **R11 レート制限・冪等性**: tenant 単位の token bucket / idempotency key

**確定フェーズ**: Phase 0。**対応要件**: FR-T1-\* 全体。

マトリクスの読み方は「縦方向でコンポーネントの全責務を読む、横方向で責務の分担を読む」の 2 視点である。縦読みで例えば COMP-T1-STATE は R3 / R5 / R6 / R9 を主担当しており、データ経路（read / write / pub/sub / feature）のハブであることが分かる。横読みで R4 を見ると業務ロジックは AUDIT / DECISION / PII の 3 Rust Pod に集中している。

**確定フェーズ**: Phase 0。**対応要件**: FR-T1-\* 全体、NFR-A-CONT-003。

## tier1 / tier2 責務境界

各コンポーネントの責務は tier1 内に閉じるが、tier2 との境界は API ごとに明確にする必要がある。境界が曖昧なまま運用に入ると、障害時の切り分け・監査対応・SLA 計算が破綻する。

### DS-SW-COMP-082 State API の責務境界

State API は KV の保存・取得・Optimistic Concurrency 制御を tier1 の責務とする。データの意味解釈（schema 検証・ビジネスルール適用）は tier2 の責務で、tier1 は byte 配列と TTL / etag のみを扱う。例えば「顧客情報の JSON スキーマが正しいか」の判定は tier1 は行わず、tier2 側で事前検証する。tier1 が責務を負うのは「Valkey に到達できなかった場合の 503 応答」「TTL 到達で自動削除されたことの保証」のみである。

**確定フェーズ**: Phase 0。**対応要件**: FR-T1-STATE-\*、NFR-A-CONT-003。

### DS-SW-COMP-083 PubSub API の責務境界

PubSub API は CloudEvents 1.0 準拠ラップ・at-least-once publish / subscribe・dead letter queue を tier1 の責務とする。イベント内容の業務意味（例: 「注文作成イベントは OrderCreated の schema に準拠すべき」）は tier2 の責務で、schema 検証は Phase 2 の Confluent Schema Registry 統合まで延期する。tier1 側では CloudEvents の必須ヘッダ（id / source / type / subject / time）のみ検証する。

**確定フェーズ**: Phase 0（境界）、Phase 2（schema 検証）。**対応要件**: FR-T1-PUBSUB-\*、NFR-A-CONT-003。

### DS-SW-COMP-084 Secrets API の責務境界

Secrets API は OpenBao 動的 Secret 発行・lease 管理・PKI 証明書発行・Transit 暗号化を tier1 の責務とする。発行された Secret の tier2 側での使用・保管・廃棄は tier2 の責務で、tier1 は発行後の追跡を行わない（PKI の失効リストは OpenBao が管理）。tier2 が誤って Secret を plain log に書いた場合、tier1 は責任を持たず、COMP-T1-PII のマスキング後段で検出されたときのみアラート発火する（ただし検出漏れのケースは tier2 の責任）。

**確定フェーズ**: Phase 0。**対応要件**: FR-T1-SECRETS-\*、NFR-H-KEY-001、NFR-G-ENC-\*。

### DS-SW-COMP-085 Decision API の責務境界

Decision API は ZEN Engine の JDM 評価・ルールセット atomic swap・評価結果 cache を tier1 の責務とする。ルールセット自体の内容（どの業務判定をルールで表現するか）は tier2 の責務で、tier1 は Backstage UI 経由でのルール CRUD の受付・保管・配布のみを行う。誤ったルールによる誤判定は tier2 の責任で、tier1 は「ルール実行後の出力が仕様通り」のみ保証する。

**確定フェーズ**: Phase 0。**対応要件**: FR-T1-DECISION-\*、NFR-B-PERF-004。

### DS-SW-COMP-086 Audit-Pii API の責務境界

Audit-Pii API は監査イベントの永続化・hash chain 生成・PII 自動検出マスキングを tier1 の責務とする。「監査すべきイベントとは何か」の定義は tier2 の責務で、tier1 は受け取ったイベントに対して改ざん防止・長期保管を担保する。tier2 が意図的に監査イベントを発火しなかった場合は tier1 で検出できないため、これは tier2 側の SAST / レビューで検査する。PII 検出のルール（どのフィールドが PII か）も tier2 が定義し、tier1 は定義されたルールに従って検出する。

**確定フェーズ**: Phase 0。**対応要件**: FR-T1-AUDIT-\*、FR-T1-PII-\*、NFR-H-INT-001。

### DS-SW-COMP-087 Workflow API の責務境界

Workflow API は workflow 定義の受付・step 実行・sticky session 維持・Saga 補償発火を tier1 の責務とする。workflow 内の業務ステップ（step 関数の中身）は tier2 の責務で、tier1 はその関数を activity として呼び出すだけである。step 内で panic / error が出た場合の error handling は tier1 が担当（Saga 補償発火）だが、step 自体の正しさは tier2 の責任である。

**確定フェーズ**: Phase 0。**対応要件**: FR-T1-WORKFLOW-\*、NFR-A-REC-001。

## コンポーネント個別 SLA/SLO

各コンポーネントは全体 SLA 99%（NFR-A-CONT-001）に寄与する個別の可用性・性能・エラー率目標を持つ。内訳が明示されていないと、障害発生時に「どのコンポーネントが SLA を食い潰したか」が判定できない。

### DS-SW-COMP-088 コンポーネント別 SLO

各コンポーネントの SLO は以下で固定する。全体 SLA 99% = 月 7.2 時間ダウン許容、SLO 99.9% = 月 43 分ダウン許容を、6 コンポーネントに分配する。配分の考え方は「バックエンド依存度の高い Pod に大きめの許容」で、Stateful な AUDIT と Leader Election の SECRET は個別 SLO を緩め、Stateless な STATE / DECISION / PII は厳しくする。

| コンポーネント | 可用性 SLO | レイテンシ SLO (p99) | エラー率 SLO |
|---|---|---|---|
| COMP-T1-STATE | 99.95%（月 22 分） | State Get 10ms / State Set 20ms / PubSub Publish 50ms | 0.01%（10 リクエスト/10 万） |
| COMP-T1-SECRET | 99.9%（月 43 分） | Secrets Get 100ms / Lease Renew 50ms | 0.05%（50 リクエスト/10 万） |
| COMP-T1-AUDIT | 99.9%（月 43 分）* | 非同期（async 書込、dedup 処理 500ms） | 0%（書込失敗は DLQ 経由で必ず再試行） |
| COMP-T1-DECISION | 99.95%（月 22 分） | Decision Evaluate 1ms | 0.01% |
| COMP-T1-PII | 99.95%（月 22 分） | PII Mask 3ms | 0.01% |
| COMP-T1-WORKFLOW | 99.9%（月 43 分） | Workflow Start 100ms / Step Execute 1s | 0.05% |

*AUDIT の可用性は「イベント受付ができるか」で測定し、書込遅延は許容する。根拠は非同期化により同期呼出 SLA に影響しないためである。詳細な SLI 定義は [../../50_非機能方式設計/11_SLI_SLO_エラーバジェット方式.md](../../50_非機能方式設計/) を参照する。

**確定フェーズ**: Phase 0（目標設定）、Phase 1c（測定開始）。**対応要件**: NFR-A-CONT-001、NFR-B-PERF-002、NFR-B-PERF-003、NFR-B-PERF-004、NFR-B-PERF-006。

### DS-SW-COMP-089 SLO 内訳の積算根拠

全体 p99 500ms（NFR-B-PERF-001）の内訳は業務 200ms + Dapr 80ms + OTel 20ms + 監査 50ms + NW・DB 150ms で積算されている（[02_設計原則と制約.md](../../00_設計方針/02_設計原則と制約.md) 制約 6）。このうち tier1 側が管理可能なのは Dapr 80ms + OTel 20ms + 監査 50ms の 150ms であり、残り 350ms は業務層（tier2 / tier3）と インフラ層（NW / DB）の責務である。tier1 側の 150ms をさらに 6 コンポーネントに分配すると、STATE 40ms / SECRET 30ms / WORKFLOW 40ms / AUDIT 非同期で同期パス含まず / DECISION 10ms / PII 10ms + その他 20ms となる。

**確定フェーズ**: Phase 0。**対応要件**: NFR-B-PERF-001。

### DS-SW-COMP-090 Error Budget の運用方針

SLO 99.9% は月 43 分の error budget を持つ。budget 消費率が閾値を超えた場合の対応は以下の 3 段階で規定する。budget 25% 消費（月 10 分）で開発速度鈍化（feature flag で新機能 rollout 停止）、budget 50% 消費（月 22 分）で Runbook 実行と RCA 優先、budget 100% 消費（月 43 分）で feature freeze と経営層報告。消費率監視は Grafana の SLO ダッシュボードで視覚化する（[../../50_非機能方式設計/](../../50_非機能方式設計/) 参照）。

**確定フェーズ**: Phase 1c。**対応要件**: NFR-A-CONT-001、DX-MET-\*、DX-RB-\*。

## スケール戦略の選定理由

全 6 コンポーネントのスケール戦略は運用形態と直結し、一度誤ると運用フェーズで修正コストが跳ね上がる。本節は各戦略の選定理由を根拠とともに記録する。

### DS-SW-COMP-091 STATE の HPA 3-10 選定理由

COMP-T1-STATE は Stateless で Valkey / Kafka / flagd / 外部 HTTP を扱うため、リクエスト負荷に比例して単純にスケールアウトできる。下限 3 は Phase 1b の 3 ノード（VM 3 台）で各ノードに 1 Pod 配置することで、単一ノード障害でも 2 Pod 稼働を保つ。上限 10 は Phase 1b の中規模想定 150 RPS × 3 倍安全係数 = 450 RPS を吸収する容量として設定する。HPA の閾値は CPU 60% で、Phase 2 で KEDA の HTTP RPS ベースに切替予定。下限を 2 に下げると Phase 1b の 3 ノード構成で偏りが生じるため 3 を維持する。

**確定フェーズ**: Phase 1a/1b。**対応要件**: NFR-B-WL-001、NFR-B-CAP-\*、NFR-A-FT-001。

### DS-SW-COMP-092 SECRET の Leader Election 選定理由

COMP-T1-SECRET は OpenBao 動的 Secret リースの排他管理を要する（FR-T1-SECRETS-002）。HPA によるマルチアクティブ構成では、複数 Pod が同一テナントのリースを並行発行すると OpenBao 側の lease_id 衝突で整合性が崩れる。Leader Election を選ぶことで active 1 Pod のみが OpenBao に書き込み、standby 2 Pod は Leader 死亡時の即時昇格に備える。replica 3 の内訳は active 1 + standby 2 で、standby 1 にすると Leader 切替中に 1 台のみになり冗長性が不足するため standby 2 が最小。

**確定フェーズ**: Phase 1b。**対応要件**: FR-T1-SECRETS-002、NFR-A-CONT-002、NFR-A-FT-001。

### DS-SW-COMP-093 AUDIT の StatefulSet 選定理由

COMP-T1-AUDIT はハッシュチェーン順序保証のため StatefulSet + PVC per replica で稼働する。Deployment だと Pod 削除時に PVC が自動解放されチェーン連続性が失われる。StatefulSet は pod-0 / pod-1 / pod-2 の固有識別子で PVC を固定し、Pod 再作成時も同じ PVC に再接続する。replica 3 は Phase 1b の中規模書込負荷（推定 30 event/s）を 3 パーティション分散で吸収しつつ、各 replica 内で順序保証を維持する。Phase 2 の大規模想定（300 event/s）では replica 5 に拡張する（DS-SW-COMP-057 参照）。

**確定フェーズ**: Phase 1b/2。**対応要件**: FR-T1-AUDIT-001、NFR-H-INT-001、NFR-C-NOP-003。

### DS-SW-COMP-094 DECISION の HPA 2-8 選定理由

COMP-T1-DECISION は in-process ZEN Engine で Pod メモリに cache を持つため、下限 2 は cache 分散による hit 率維持のために必要。上限 8 は Phase 2 の業務ルール拡大時の想定 500 RPS × 2 倍 = 1,000 RPS 相当を吸収する容量。HPA 閾値は CPU 70%（in-process cache 処理の CPU bound を前提）。下限 1 にすると Pod 再起動時の cache miss で SLO 1ms が壊れるため 2 を維持する。p99 1ms 目標は in-process なら達成可能だが、HPA が高頻度に発火すると warm cache が細切れになるため閾値 70% は慎重に設定する。

**確定フェーズ**: Phase 1b/2。**対応要件**: NFR-B-PERF-004、NFR-A-FT-001。

### DS-SW-COMP-095 PII の HPA 2-6 選定理由

COMP-T1-PII は純関数で外部ストアを持たないため、下限 2 は冗長性確保のみ。上限 6 は呼び出し元（COMP-T1-STATE の Log Adapter）が HPA 10 なので、1:1 に近い比率で追従可能な 6 とする。HPA 閾値は CPU 60%。PII 処理は 500μs / 呼び出しと軽量なため、上限 6 で十分に 10,000 RPS を吸収できる。下限 2 にしないと Pod 再起動中に PII マスキングが 0 台になる可能性があり、これは NFR-G-ENC-\* 違反に直結する。

**確定フェーズ**: Phase 1b。**対応要件**: NFR-B-PERF-006、NFR-G-ENC-\*。

### DS-SW-COMP-096 WORKFLOW の固定 3 replica 選定理由

COMP-T1-WORKFLOW は Temporal / Dapr Workflow の sticky task queue を要するため、HPA ではなく固定 replica を選ぶ。HPA でスケールアウトすると新 replica への sticky 再割当が発火し、補償トランザクションが遅延する（NFR-A-REC-001 違反リスク）。replica 3 は Phase 1b の 3 ノード構成での均等分散最小であり、1 ノード退避でも 2 ノード稼働を維持。Phase 2 で高負荷化したら replica 5 に固定拡張する（HPA 化はしない）。

**確定フェーズ**: Phase 1b/2。**対応要件**: FR-T1-WORKFLOW-\*、NFR-A-REC-001。

## 責務の運用反映

責務マトリクスは Runbook・監視ダッシュボード・監査対応の 3 系統に機械的に反映する。反映が漏れると責務境界が形骸化する。

### DS-SW-COMP-097 Runbook への責務反映方式

各インシデント Runbook は「どのコンポーネントの責務が機能不全か」を明示して書く。例えば「State API の応答遅延」の Runbook は COMP-T1-STATE の R3（バックエンド接続）を起点として「Valkey 接続状態 → Dapr sidecar → Policy Enforcer」の順に切り分ける。責務マトリクスの縦軸（コンポーネント）と Runbook の章立てを 1:1 対応させることで、責務境界を越える原因追跡を避ける。Runbook フォーマットは [../../55_運用ライフサイクル方式設計/](../../55_運用ライフサイクル方式設計/) で規定する。

**確定フェーズ**: Phase 1c。**対応要件**: DX-RB-\*、NFR-C-NOP-001。

### DS-SW-COMP-098 監視ダッシュボードへの責務反映方式

Grafana ダッシュボードは 6 コンポーネント単位で分離し、各ダッシュボードに「このコンポーネントの責務（R1-R11 のうち主担当のもの）」を最上段に表示する。これにより運用者がダッシュボードを開いた瞬間、その Pod で何を見るべきかが一目で分かる。全体俯瞰ダッシュボードは「6 Pod × 3 SLO 軸（可用性・レイテンシ・エラー率）」の 18 セル行列で、SLO 違反セルを赤くハイライトする。

**確定フェーズ**: Phase 1c。**対応要件**: DX-MET-\*、NFR-D-MON-\*。

### DS-SW-COMP-099 監査対応における責務の扱い

監査時（内部監査・J-SOX・個人情報保護法対応）の応答では、責務マトリクスを起点として「この監査要件は tier1 のどのコンポーネントが担うか」を先に確定させる。例えば「個人情報の 7 年保管」は COMP-T1-AUDIT の R5（永続化）に帰属し、tier2 の責務ではないと明示できる。逆に「個人情報の目的外利用防止」は tier2 の業務ロジック責務で、tier1 は定義されたルールの実行のみ担う。この境界を監査資料に明示することで、監査対応の時間を短縮し tier2 チームへの責任転嫁を防ぐ。

**確定フェーズ**: Phase 1c。**対応要件**: NFR-C-NOP-003、NFR-H-INT-001、BR-PLATGOV-\*。
