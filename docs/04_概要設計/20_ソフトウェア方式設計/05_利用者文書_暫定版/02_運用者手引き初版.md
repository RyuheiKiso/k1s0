# 02. 運用者手引き（暫定版）

本ファイルは IPA 共通フレーム 2013 の **7.1.2.4 利用者文書（暫定版）の作成** に対応する利用者文書のうち、JTC 情シス配下の運用者が k1s0 本番環境を維持するための手引き初版である。運用者の想定規模は企画で約束した「バス係数 2」の 2 人目、すなわち起案者不在でも本手引きを参照して運用を継続できる第 2 運用者である。

## 本ファイルの位置付け

プラットフォーム製品の運用で最も事故が多いのは、起案者が暗黙知として抱える「勘所」が文書化されていない段階での人員交代時である。k1s0 では企画段階で「バス係数 2」「2 名運用で成立する」という 2 原則を掲げており、これらを実現する運用手引きの欠如は企画コミットの不達そのものを意味する。本章は、第 2 運用者が起案者不在でも初日から稼働できる最小限の手引きを Phase 0 稟議時点で整備することを目的とする。

運用者手引きは [../../55_運用ライフサイクル方式設計/](../../55_運用ライフサイクル方式設計/) 配下の個別方式書・Runbook の「入口」として機能する。個別障害対処の詳細手順は Runbook 側にあり、本章は「何が起きたらどの Runbook を開くか」「定常運用で日次に実施すべきこと」「エスカレーション判断はどの基準で行うか」の航海図を提供する。全 Runbook を並べただけでは新規運用者が道に迷うため、本章でマクロ視点の手順を先に提示する。

暫定版として、Phase 0 時点では「日次 / 週次 / 月次 の定常手順」「障害一次対応の判断フロー」「学習リソース目録」を確定する。Phase 1b で監視ダッシュボードの画面ショットと具体的アラート値、Phase 1c で監査対応・災害復旧の拡充版を追補する。

## 対象読者と前提スキル

対象読者は JTC 情シス配下の第 2 運用者（バス係数 2 担保のための後任候補）である。前提スキルは Linux 基本操作・基本的な SQL 読解・Git の基礎・英文アラートメッセージの読解とし、Kubernetes・Dapr・Kafka・Valkey の深い知識は前提としない。これらの深堀り知識は必要になった都度、Runbook と TechDocs で補完する設計である。

読者が最初に実施すべきはオンコール訓練（[#設計項目-DS-SW-DOC-056-学習リソースとオンコール訓練](#設計項目-ds-sw-doc-056-学習リソースとオンコール訓練) 参照）であり、本章は訓練修了時点で「何を参照すればよいか」が把握できる状態を目標とする。

## 設計項目 DS-SW-DOC-050 日次手順（朝・夜の 2 サイクル）

日次手順は朝 9:00 開始の始業サイクルと 18:00 開始の終業サイクルの 2 回実施する。運用者 2 名で交代シフトとし、1 人当たり週 5 サイクル以下に抑える。サイクル所要時間は初版で各 20 分を目安とする。

**朝サイクル（9:00〜9:20）**: 前日夜 18:00 以降に発生したアラートを Grafana ダッシュボード「k1s0 Overview」で確認し、未解消のものがあれば該当 Runbook を開く。Argo CD ダッシュボードで「OutOfSync」「Degraded」状態の Application がないかを確認する。PostgreSQL のバックアップジョブが前夜 2:00 に正常完了したかを CloudNativePG の `Backup` リソースで確認する。Keycloak の SSO ログイン成功率が 99% 以上を維持しているかを確認する。

**夜サイクル（18:00〜18:20）**: 当日発生したアラートの処置記録を Backstage の Incident ページに記入する。tier1 API の p99 レイテンシが 500ms を超えた時間帯があれば、該当時刻の Tempo トレースを確認し root-cause 候補を記録する。翌日朝サイクル向けに「引き継ぎノート」を Backstage に 1 行以上書き残す。

日次手順が規定時間内に完了しない日が週 2 回以上発生した場合、運用負荷超過として [../../55_運用ライフサイクル方式設計/01_サポート階層方式.md](../../55_運用ライフサイクル方式設計/01_サポート階層方式.md) のエスカレーションパスを発動する。

## 設計項目 DS-SW-DOC-051 週次手順と月次手順

週次手順（毎週月曜 10:00〜11:00）は、バックアップリストアのランダムサンプリング検証・証明書期限の棚卸・OSS 脆弱性スキャン結果の Trivy レポート確認・Dapr Component 設定の drift チェック を実施する。バックアップリストアは週 1 件を無作為抽出して別環境にリストアし、データ整合性を SELECT ベースで確認する。復旧可能性は「定期検証されて初めて担保される」ため、検証サイクルの省略は禁止する。

月次手順（月初第 1 営業日）は、前月の SLO 達成状況（稼働率・p99 レイテンシ・エラー率）を Grafana で集計し、Product Council 向けレポートを作成する。エラーバジェット消費率が 80% を超えた領域は「機能追加凍結」の発動判断対象となる。OpenBao の root トークンローテーション、Keycloak のクライアントシークレットローテーションも月次で実施する。

週次・月次手順は Backstage の TechDocs に固定テンプレートで記録し、完了チェックを電子署名で残す。電子署名が月末時点で欠落している項目は監査不備として記録する。

## 設計項目 DS-SW-DOC-052 監視手順 — Grafana ダッシュボード体系

Grafana 上の k1s0 専用ダッシュボードは、運用者が最初に開く「k1s0 Overview」を頂点に、tier 別・機能別に階層化する。Overview ダッシュボードは以下 6 ブロックで構成する。

- **Availability**: tier1 API 11 種の稼働率（現在値・過去 24 時間・過去 30 日）
- **Performance**: p50 / p95 / p99 レイテンシの時系列（API 別）
- **Errors**: エラー率・エラー種別分布（400 / 401 / 403 / 429 / 500 系）
- **Infrastructure**: Kubernetes ノードの CPU / メモリ / ディスク使用率、Pod 再起動回数
- **Data**: Valkey / Kafka / PostgreSQL / OpenBao の各健全性指標
- **Security**: 認証失敗回数・監査ログの hash_chain 整合性エラー検知

運用者がアラート受信時に最初に開くべきは Overview であり、個別ダッシュボードは Overview からリンクで辿る。個別ダッシュボードを直接 URL でブックマークする運用は禁止する（ダッシュボード URL が変わった際に対応できないため）。

アラート通知はメール・Slack の 2 経路とし、重要度 Sev1（サービス停止）は電話発信も行う。電話発信は Phase 1c で PagerDuty 相当の自作通知基盤（Telemetry API 経由）で実装する。

## 設計項目 DS-SW-DOC-053 変更管理 — GitOps 標準手順

k1s0 の全ての設定変更は GitOps 経由で行う。手動 kubectl apply・手動 DB 変更・手動 Dapr Component 変更は禁止する。緊急時であっても、事後 30 分以内に GitOps の記録と整合させる義務がある。

変更手順は以下 4 段に固定する。第一段で GitHub リポジトリ `k1s0-gitops` に変更ブランチを切り、変更内容を commit する。第二段で PR を出し、変更の影響範囲・ロールバック手順・テスト結果を PR テンプレートに記入する。第三段で CODEOWNERS に定義された基盤チーム 2 名以上が approve する（1 名は起案者以外）。第四段で main merge 後 Argo CD が自動 sync し、Argo CD UI 上で `Synced & Healthy` になったことを確認する。

変更失敗時のロールバックは、GitHub 上で revert commit を作成することで実行する。Argo CD が revert commit を検知し、以前の状態に sync し直す。kubectl による手動ロールバックは禁止である。

PR レビュー時に「この変更で何が壊れるか」をレビュアが想像できない場合、approve せずに影響範囲分析の追記を依頼する。「動くと思う」で approve した変更が本番障害を起こす確率は業界統計で 40% 以上とされるため、根拠のない approve は禁止する。

## 設計項目 DS-SW-DOC-054 障害一次対応 — Sev1 / Sev2 のトリアージ

障害発生時のトリアージは Sev1（サービス停止または大量影響）と Sev2（部分劣化または少数影響）の 2 段で行う。Sev3 以下は日次サイクルで対応する。

**Sev1 の判定基準**: tier1 API のいずれかが 5 分以上連続でエラー率 50% 超、または可用性 SLO（99%）の月間許容時間（7.2 時間）を 1 日で 25% 以上消費する事象。判定後 5 分以内に Slack #incident チャンネルへ宣言、同時に起案者・基盤チームリーダへ電話連絡する。

**Sev2 の判定基準**: 単一 API の p99 レイテンシが 500ms を 30 分以上超過、または監査ログ hash_chain の整合性エラーが 1 件以上検出された事象。判定後 15 分以内に Slack #incident へ宣言、基盤チームリーダへ Slack メンション。

トリアージ後の初動は Runbook 目録（[#設計項目-DS-SW-DOC-056-学習リソースとオンコール訓練](#設計項目-ds-sw-doc-056-学習リソースとオンコール訓練)）から該当項目を開き、手順に従う。Runbook が存在しない事象の場合、「Runbook 欠落」として基盤チームへ起票し、事象を時系列で記録しながら暫定対処を進める。

## 設計項目 DS-SW-DOC-055 緊急時キルスイッチ — Feature Flag による即時停止

障害の原因が特定機能に絞られる場合、その機能を Feature Flag（flagd）で即時無効化することで影響を遮断する。キルスイッチは以下のケースで使用する。

- 新機能リリース後のバグ顕在化
- 特定テナントからの異常トラフィック（DoS 的挙動）
- 外部連携先（レガシーシステム）の障害に引きずられた tier1 API 劣化
- 監査ログ hash_chain 異常の暫定隔離

キルスイッチ発動は運用者 1 名で実行可能とし、事後 30 分以内に GitOps で flag 状態を固定化する。flagd の OFAC（Over-the-air Flag Change）は即時反映で、反映に 5 秒以上かからないことを Phase 1c の受け入れ試験で確認する。

キルスイッチによる過剰停止（Sev2 事象で Sev1 相当機能を停止する等）を防ぐため、キルスイッチ発動履歴は週次レビューで検証し、過剰発動が 2 回以上連続した場合は判断基準を見直す。

## 設計項目 DS-SW-DOC-056 学習リソースとオンコール訓練

新規運用者は以下の学習パスを 2 週間以内に消化する。パスの完走が、第 2 運用者としての稼働開始条件である。

**第 1 週**: [#対象読者](#対象読者と前提スキル) の前提スキル確認 → 本章の通読 → Backstage TechDocs の「運用者入門」5 本読了 → Runbook 目録の見出し通読（各 Runbook の詳細は必要時に参照）→ 監視ダッシュボード「k1s0 Overview」の全ブロック説明を受ける。

**第 2 週**: オンコール訓練を実施する。訓練は基盤チームが用意したステージング環境で行い、シナリオは 5 件（API 一部停止 / PostgreSQL failover / Dapr sidecar 再起動 / Feature Flag キルスイッチ / 監査 hash_chain 検知）とする。各シナリオは訓練者が独力で完了させ、完了時間・誤操作数を記録する。

**Runbook 目録**: 以下は Phase 1b 時点での必須 Runbook 一覧。個別詳細は [../../55_運用ライフサイクル方式設計/08_Runbook設計方式.md](../../55_運用ライフサイクル方式設計/08_Runbook設計方式.md) 参照。

- RB-001: tier1 API 障害（個別 API 単位で 11 種）
- RB-002: PostgreSQL failover
- RB-003: Kafka ブローカ障害
- RB-004: Valkey シャード障害
- RB-005: OpenBao sealed 状態からの unseal
- RB-006: Argo CD sync 失敗
- RB-007: 証明書期限切れ緊急差し替え
- RB-008: 監査 hash_chain 整合性エラー

訓練完了後、運用者は四半期に 1 回の再訓練を受ける。再訓練を 2 四半期連続で欠席した場合、オンコール担当から一時的に外れる。

## 設計項目 DS-SW-DOC-057 2 名運用の時間配分モデル

企画で約束した 2 名運用を支える時間配分を明示する。運用者 1 名の月稼働時間 170 時間（実働）とし、内訳を以下とする。

- 日次サイクル（朝・夜 × 20 分 × 20 営業日）: 約 13 時間
- 週次手順（60 分 × 4 回）: 約 4 時間
- 月次手順（4 時間 × 1 回）: 約 4 時間
- 障害対応（月間平均 Sev1: 0.5 件 × 4 時間 + Sev2: 2 件 × 2 時間）: 約 6 時間
- PR レビュー（週 5 本 × 20 分）: 約 7 時間
- オンコール拘束時間の待機補償（実働外）: 時間外カウント
- 学習・ドキュメント整備・改善提案: 約 10 時間
- 合計: 約 44 時間

残り約 126 時間は、定常運用外のプロジェクト業務（新機能の運用 onboard・DR 訓練・ライセンスレポート作成）に充てる。この時間配分が 2 四半期連続で崩れる場合、運用負荷超過として Product Council へ予算追加または機能凍結を提案する。

## 設計項目 DS-SW-DOC-058 Phase 別完成度

本章は Phase 進行に応じて以下の順で拡充する。

- **Phase 0（稟議時点）**: 本章の骨子（日次・週次・月次手順 + トリアージ + キルスイッチ + 学習パス）を確定。
- **Phase 1a（MVP-0）**: 監視ダッシュボード画面ショット・具体的アラート閾値・Runbook の初版 8 本を追加。
- **Phase 1b（MVP-1a）**: オンコール訓練シナリオの確定、PR レビュー観点チェックリスト拡充、キルスイッチ実運用訓練。
- **Phase 1c（MVP-1b）**: 監査対応・災害復旧（DR）訓練手順、PostgreSQL WORM 監査検証手順、OpenBao unseal 手順の実習版を追加。
- **Phase 2 以降**: マルチクラスタ運用、Temporal 採用後の Saga 運用、Backstage プラグイン連動の運用自動化を追加。
