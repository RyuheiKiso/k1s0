# 04. アプリ配信ポータル利用（暫定版）

本ファイルは IPA 共通フレーム 2013 の **7.1.2.4 利用者文書（暫定版）の作成** に対応する利用者文書のうち、k1s0 の tier3 自製アプリ配信ポータルの利用手引き初版である。配信ポータルは Phase 1a デモで最小版を提供し、Phase 3 で本格展開する。

## 本ファイルの位置付け

JTC 情シス部門が従業員に業務アプリを配布する際、既存の配信経路は「情シス部門のファイルサーバ経由」「社内メールに添付」「各部署が独自調達した SaaS」など統一性を欠き、配布状況・利用状況・アンインストール履歴の追跡ができない。これは企画で約束した「レガシー共存 + 段階移行」「業務アプリのガバナンス回復」の両コミットを阻害する要因となる。

アプリ配信ポータルは、この課題を解決する k1s0 tier3 の自製 PWA + Native ハイブリッド配信基盤である。PWA（Progressive Web Application）による Web ベース配信と、MSIX（Windows 11 向け）および ClickOnce（.NET Framework レガシー向け）の Native 配信を統合し、tier1 の Telemetry API を通じてインストール・起動・エラーを全量追跡する。

本章は配信ポータルの 2 系統の利用者（エンドユーザ / 管理者）に向けた手引きを暫定版として整備する。Phase 1a デモでの最小機能（PWA のみ・社内 10 名規模配信）から、Phase 3 本格展開（Native 統合・全社 1 万名規模）までの完成度を段階明示する。

## 対象読者と前提スキル

対象読者は以下 2 ペルソナである。

**エンドユーザペルソナ**（JTC 一般従業員）: 社内 AD アカウントを持つ全従業員。PC / スマートフォン / タブレット から配信ポータルにアクセスし、必要なアプリをインストール・更新する。IT スキルは Windows 基本操作・Web ブラウザ利用 のみを前提とする。

**管理者ペルソナ**（各部署の配信管理者 + 情シス）: 部署ごとに 1〜2 名の管理者がアプリ登録・配信ターゲット設定・バージョン管理を担当する。情シスは全体統括として、ポリシー・セキュリティスキャン・ライセンス管理を担う。前提スキルは管理画面の操作と、アプリパッケージング（MSIX / ClickOnce / PWA マニフェスト）の基礎知識である。

## 設計項目 DS-SW-DOC-110 配信ポータルの全体像 — PWA + Native ハイブリッド

配信ポータルは Web ブラウザでアクセスする PWA として実装され、同時に Native アプリ（MSIX / ClickOnce）の起動・更新を仲介する。これにより、新規開発アプリは PWA（Web インストール + オフライン動作）、既存 .NET Framework 資産は ClickOnce、Windows 11 ネイティブアプリは MSIX、というポートフォリオを単一のポータルから横断利用できる。

PWA 選択の理由は、JTC 情シスが長年蓄積した「配布即利用、アンインストール時の痕跡ゼロ、OS 横断」という要件を、追加ランタイムなしで満たせる点にある。ChromiumEdge / Chrome / Firefox いずれも PWA をサポートし、Service Worker によるオフライン動作も可能である。

Native 配信の理由は、既存 .NET Framework 業務アプリ（約 200 本と推定）を強制廃棄せず、ClickOnce でそのまま配信継続できる点にある。これは企画で約束した「レガシー共存 + 自発的移行」を実装レベルで担保する。Windows 11 向けの新規ネイティブアプリは MSIX で配信し、サンドボックス化・自動更新・依存関係解決の恩恵を得る。

ポータル側から見ると、PWA / MSIX / ClickOnce の区別はエンドユーザに透過的である。ユーザは「インストール」ボタンを押すだけで、配信方式はポータル側で最適解を選択する。

## 設計項目 DS-SW-DOC-111 エンドユーザ視点の利用フロー

エンドユーザは社内 AD アカウントで配信ポータルにログインし、アプリ一覧から必要なアプリを選んでインストールする。操作ステップは以下 5 段である。

**ステップ 1**: 社内ポータルのリンクから `https://apps.k1s0.internal` に遷移。Keycloak SSO が自動発動し、AD 認証済みであれば追加入力なしでログイン完了。

**ステップ 2**: 部署・役職に応じて自動フィルタされたアプリ一覧を閲覧。検索・カテゴリ・新着・人気 の 4 タブを切り替え可能。各アプリには名称・説明・配信方式（PWA / MSIX / ClickOnce）・サイズ・対応 OS・バージョンが表示される。

**ステップ 3**: インストール対象を選び「インストール」ボタンを押す。PWA の場合はブラウザの PWA 機能で即座にデスクトップ・スマートフォンのホーム画面に登録される。MSIX の場合は ms-windows-store 系ハンドラでインストーラが起動。ClickOnce の場合は `.application` ファイルが起動し、.NET Framework ランタイムがインストーラを駆動。

**ステップ 4**: インストール完了後、配信ポータル上で「インストール済み」状態となり、次回以降は「起動」ボタンが表示される。更新があれば自動的に「更新あり」バッジが付き、ユーザは任意のタイミングで更新できる。

**ステップ 5**: アンインストール時は配信ポータルの「アンインストール」ボタンから実行。ClickOnce / MSIX は OS の標準アンインストール機構を呼び出し、PWA はブラウザの PWA 削除 API を呼び出す。アンインストール履歴は tier1 Telemetry API で記録され、管理者側で追跡可能となる。

エンドユーザの操作は一貫して「配信ポータル上の 1 クリック」で完結する設計とし、OS 差異・配信方式差異をユーザから隠蔽することが本ポータルの価値である。

## 設計項目 DS-SW-DOC-112 管理者視点の運用フロー

管理者はアプリ登録・バージョン管理・配信ターゲット設定・利用状況分析 の 4 業務を配信ポータルの管理画面で行う。

**アプリ登録**は、管理者が PWA マニフェスト・MSIX パッケージ・ClickOnce マニフェストを管理画面にアップロードする形で行う。アップロード時に Trivy でウイルススキャン・脆弱性スキャンを自動実行し、Critical 脆弱性があれば登録を拒否する。署名検証も必須で、JTC コード署名証明書で署名されたパッケージのみ受け付ける。

**バージョン管理**は、同一アプリの新バージョン登録時に旧バージョンを自動アーカイブし、段階ロールアウト（1% → 10% → 100%）を設定できる。ロールアウト途中で異常検知した場合は Feature Flag 相当の即時切り戻しが可能。

**配信ターゲット設定**は、部署・役職・雇用形態・勤務地 の 4 軸で配信対象を絞り込む。AD の属性を直接参照し、組織変更時は自動的に対象が更新される。

**利用状況分析**は、インストール数・アクティブ利用者数・起動回数・エラー発生率・アンインストール率 の 5 指標を Grafana ダッシュボードで可視化する。ダッシュボードは管理画面から 1 クリックで遷移可能とし、深堀り分析は Grafana 側で行う。

## 設計項目 DS-SW-DOC-113 認証と認可 — Keycloak SSO

配信ポータルの認証は Keycloak OIDC による SSO とし、JTC 社内 AD アカウントで直接ログイン可能とする。認証フローは Backstage と同一のため、[03_Backstageポータル利用初版.md](03_Backstageポータル利用初版.md) の DS-SW-DOC-081 と整合させる。

認可は以下 4 ロールで階層化する。

- **一般ユーザ**: アプリ一覧閲覧・インストール・アンインストール
- **部署管理者**: 自部署向けアプリの登録・配信・停止
- **情シス管理者**: 全社横断のアプリ管理・ポリシー設定・ユーザ管理
- **監査者**: 全操作ログの閲覧のみ（変更権限なし）

ロール割当は AD グループと Keycloak のロールマッピングで自動化する。手動割当は情シス管理者のみが実行可能とし、割当操作は監査ログに記録される。

## 設計項目 DS-SW-DOC-114 Telemetry 連携 — tier1 Telemetry API への集約

配信ポータルは、インストール・起動・エラー・アンインストール の全イベントを tier1 Telemetry API 経由で集約する。Telemetry API は OpenTelemetry 互換の形式で受信し、Mimir（メトリクス）・Loki（ログ）・Tempo（トレース）に分配する。

イベント種別と記録内容は以下とする。

- **install**: アプリ ID、バージョン、配信方式、ユーザ ID、端末 ID、インストール開始時刻、完了時刻、成功/失敗
- **launch**: アプリ ID、起動時刻、ユーザ ID、端末 ID、起動所要時間
- **error**: アプリ ID、エラー種別、スタックトレース（PII マスキング済み）、発生時刻
- **uninstall**: アプリ ID、アンインストール時刻、ユーザ ID、端末 ID、理由（任意）

tier1 Telemetry API 経由で集約する理由は、配信ポータル独自の分析基盤を構築するのではなく、k1s0 基盤の監視基盤に統合することで運用負荷を抑える点にある。Phase 1a では Prometheus + Grafana の最小構成、Phase 1b で Grafana LGTM 全統合、Phase 3 で配信ポータル専用ダッシュボードを本格拡充する。

PII の扱いは、イベント収集時に氏名・メールアドレス・住所 などを自動マスキングする。ユーザ ID は内部 UUID（AD の SID 由来）で記録し、個人特定が必要な監査時のみ Keycloak を経由して逆引きする。逆引き操作は監査ログに残り、監査者以外は実行不可とする。

## 設計項目 DS-SW-DOC-115 PWA 配信の詳細 — Web インストールと自動更新

PWA 配信は、配信ポータルが提供する `manifest.webmanifest` を基に、ブラウザ標準の PWA インストール機構でインストールを完了する。Service Worker により、オフライン時も前回キャッシュでアプリを起動でき、ネットワーク復帰時に自動で差分同期する。

PWA の自動更新は、Service Worker の `skipWaiting + clientsClaim` パターンで実装する。新バージョン検出時はユーザに通知し、ユーザ同意のもとで即時更新する。強制更新は緊急セキュリティ修正時のみに限定し、管理画面から明示的に有効化する。

PWA 配信の利点は以下 3 点である。第一に、インストール時の管理者権限不要。第二に、アンインストール時の痕跡ゼロ（ブラウザキャッシュのみ残る）。第三に、OS 横断（Windows / macOS / iOS / Android / Linux）での統一体験。Phase 1a デモはこの PWA 配信のみで 10 アプリを提供し、基盤の動作確認を行う。

PWA 配信の制約は、OS ネイティブ機能（ファイルシステム直接操作・デバイスドライバ連携・シリアルポート）へのアクセスが限定的な点である。これらを必要とする業務アプリは Native 配信側で対応する。

## 設計項目 DS-SW-DOC-116 Native 配信の詳細 — MSIX と ClickOnce

MSIX は Windows 11 向けの新しいパッケージ形式で、サンドボックス化・自動更新・依存関係解決を標準でサポートする。新規開発の Native アプリ（.NET MAUI / WinUI 3）は MSIX で配信する。MSIX パッケージは配信ポータルの管理画面からアップロードし、JTC コード署名証明書での署名が必須である。

ClickOnce は .NET Framework 時代の配信形式で、既存資産（約 200 本と推定）をそのまま配信継続するために採用する。ClickOnce マニフェスト（`.application` ファイル）を配信ポータルが提供する URL にホストし、クライアントは URL を開くことでインストール・更新を行う。.NET Framework 4.x 向けで、Phase 3 までに段階的に MSIX または PWA へ移行することを想定するが、強制移行はしない。

両者の共通課題はセキュリティである。古い ClickOnce 配布には以下の既知脆弱性があり、本ポータルでは対策を織り込む。証明書失効時の即時配布停止、マニフェスト改ざん検知、起動前の SHA-256 整合性検証、Windows Defender AttackSurfaceReduction ルールとの連動。

## 設計項目 DS-SW-DOC-117 Phase 別完成度と段階展開

配信ポータルは Phase 1a デモで最小機能、Phase 3 で本格展開とする。各 Phase での機能完成度を以下に明示する。

- **Phase 0（稟議時点）**: 本章の骨子確定のみ。実装は着手前。
- **Phase 1a（MVP-0）**: PWA 配信のみ、10 アプリ規模、情シス管理者 1 名での運用デモ。インストール数・エラー数の基礎指標のみ計測。
- **Phase 1b（MVP-1a）**: PWA + MSIX 配信、50 アプリ規模、情シス + 各部署管理者 2〜3 名での運用。Grafana 連携本格化。
- **Phase 1c（MVP-1b）**: ClickOnce 対応追加、監査ログ hash_chain 化、脆弱性スキャン自動化、SSO Enterprise 機能拡充。
- **Phase 2**: マルチテナント対応、子会社向け配信の分離、配信ポータル自身の HA 構成。
- **Phase 3**: 全社 1 万名規模、200 アプリ規模、AI 推奨機能（ユーザの業務プロファイルに応じたアプリ推薦）、ライセンス管理自動化。

Phase 1a〜1b は情シス内の少人数運用を前提とするため、過剰機能は削ぎ落として「動く最小」を優先する。Phase 2 以降は全社展開に耐える運用自動化・管理者権限委譲を重点的に整備する。

## 設計項目 DS-SW-DOC-118 セキュリティ観点と監査対応

配信ポータルは従業員の端末にアプリをインストールする入口であるため、セキュリティ侵害の影響が全社的に波及するリスクを持つ。本節では重点対策を明示する。

配信前のマルウェアスキャンは、アップロード時に Trivy と Windows Defender の 2 系統でスキャンし、1 つでも Critical 検出があれば登録を拒否する。配布済みアプリについても週次で再スキャンし、後発的に脆弱性が発覚した場合は即時配布停止する。

コード署名は全 Native アプリで必須とし、JTC 内部 CA で発行した Code Signing 証明書での署名のみ受け付ける。証明書は OpenBao で管理し、署名鍵の直接エクスポートは禁止する。

監査ログは、アプリ登録・配布承認・配布停止・ユーザ操作（インストール・アンインストール）の全操作を tier1 Audit-Pii API で記録する。ログは PostgreSQL WORM テーブルで改ざん防止し、hash_chain で整合性を保証する。監査者は監査画面から全ログを閲覧可能だが、変更権限は一切持たない。

情報漏洩リスク対策として、配信ポータル自身の機密情報（内部 AD 属性・デバイス情報・インストール履歴）は外部アクセスから遮断する。配信ポータルは社内ネットワーク内部からのみアクセス可能とし、VPN 経由のリモートアクセスは Keycloak の MFA を必須とする。
