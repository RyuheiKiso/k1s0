# 02. API バージョニング方式

本ファイルは tier1 公開 11 API のバージョニング方式を固定化する。API は 5 年以上の運用期間でスキーマが進化し続ける資産であり、変更が tier2 / tier3 に波及しないよう「何が破壊的変更か」「どう並行提供するか」「どこで互換性を機械検証するか」を事前に決めることで、運用フェーズでの障害を構造的に防ぐ。

## 本ファイルの位置付け

tier1 は「公開 API の単純性を死守する」（設計原則 1）を最優先とする。単純性は「変更頻度の低さ」と対で成立する。頻繁なスキーマ変更は tier2 / tier3 のクライアントコードを頻繁に書き換えさせる結果となり、2 名運用の前提（設計原則 9）と矛盾する。本ファイルは「変更は避けつつ、どうしても必要な変更は構造的に安全な経路で行う」方針を規定する。

バージョニング方針を設計段階で固定しないと、実装フェーズで「フィールド追加は互換か」「enum 値追加は互換か」といった判断が属人的になり、気付けば互換性違反のスキーマが公開されるリスクがある。本ファイルはこの属人性を排除し、CI の buf breaking で機械的に違反を検出できる状態を作る。

## バージョニングの基本方針

tier1 公開 API は SemVer 2.0（major.minor.patch）を採用する。ただし SemVer の一般論をそのまま適用するのではなく、Protobuf の言語特性と業務運用の実情に合わせた具体ルールを以下で確定する。

**設計項目 DS-SW-EIF-050 SemVer 採用と Protobuf パッケージ名への埋め込み**

major バージョンは Protobuf のパッケージ名に含める（`k1s0.public.state.v1.*`、`k1s0.public.state.v2.*`）。minor と patch はパッケージ名には含めず、バージョンヘッダ `X-K1s0-Api-Version` で伝達する。パッケージ名に major を含めることで、新旧バージョンを 2 つの異なる gRPC サービスとして並行提供できる。minor / patch をパッケージ名に含めないのは、minor 追加のたびに新パッケージを切ると旧クライアントが型不一致で参照できなくなる現象を避けるためである。

major の変更は破壊的変更を許容するリリース、minor は後方互換追加、patch はバグ修正 / ドキュメント訂正のみとする。SemVer の `1.0.0` から公開を開始し、Phase 1a では `1.0.x` 系のみ、Phase 1b で `1.1.0` 系以降のマイナー更新を開始する。

**設計項目 DS-SW-EIF-051 後方互換性の定義**

後方互換性は「旧クライアントが新サーバに対して機能し続ける」ことと定める。旧サーバが新クライアントを受け入れるかは別問題（前方互換）であり、本設計では前方互換は非保証とする。これは gRPC / Protobuf の設計思想に準拠する。

後方互換とされる変更は以下に限定する。新フィールドの追加（`optional` を推奨、`repeated` / `map` 追加も可）、新メソッドの追加、新サービスの追加、enum 値の追加（未知値は `UNSPECIFIED` フォールバック必須）、`required` の緩和（Protobuf 3 では不要だが、クライアント SDK のバリデーションで緩和する）。これら以外の変更はすべて破壊的変更として扱う。

**設計項目 DS-SW-EIF-052 禁止変更の明示列挙**

破壊的変更は新 major バージョンでのみ許容し、既存 major バージョン内では禁止する。禁止例は以下。既存フィールドの削除、既存フィールドのフィールド番号の変更、フィールド型の変更（int32 → int64、string → bytes など）、フィールドの必須化（Protobuf 3 では formal には不要だが、SDK 側バリデーションで必須にする変更も禁止）、enum 値の削除、enum 値の番号変更、既存メソッドのシグネチャ変更、既存サービスの削除、意味論の変更（HTTP ステータスコードの変更、エラー条件の変更など）。

## 破壊的変更のリリース経路

破壊的変更が必要な場合、新 major バージョンで並行提供し、旧バージョンは 18 か月以上サポートする。この 18 か月は tier2 / tier3 開発チームが計画的に移行するための猶予期間であり、JTC の予算サイクル（年 1 回）を 1.5 回分カバーする根拠から導出している。

**設計項目 DS-SW-EIF-053 並行提供と段階移行のライフサイクル**

新 major リリースは以下 5 段階で進める。

1. **並行提供開始**: 新 major（v2）をリリースし、旧 major（v1）と同時公開する。エンドポイントは `api.k1s0.internal/grpc/k1s0.public.state.v1.*` と `api.k1s0.internal/grpc/k1s0.public.state.v2.*` の 2 つ。
2. **Deprecation Warning**: 旧 v1 のレスポンスヘッダに `Deprecation: true` と `Sunset: <日付>` を付与し、`X-K1s0-Upgrade-To: v2` で移行先を明示。ログに警告出力。期間 3 か月。
3. **パイロット移行**: 先行移行希望の tier2 / tier3 チームに個別支援で v2 移行を促す。Backstage のダッシュボードで移行率を可視化。期間 3 か月。
4. **段階的移行**: 全 tier2 / tier3 チームへ通知、移行期限を提示。Feature Flag で「v1 へのアクセス警告を強化」→「v1 へのアクセスを 30 秒遅延」→「v1 へのアクセス遮断」と段階進行。期間 9 か月。
5. **EOL**: v1 完全停止、Pod 撤去。期間終了後 3 か月は stub 応答（`410 Gone` + 移行ガイド URL）で残し、クライアント側の未対応を検知できるようにする。

この 5 段階で合計 18 か月（3 + 3 + 9 + 3 = 18）となる。ただし緊急セキュリティ脆弱性で v1 を即時停止する場合は、Product Council 承認で短縮する。

**設計項目 DS-SW-EIF-054 非後方互換リリースの通知プロトコル**

新 major の公開予告は公開 90 日前までに以下のチャネルで通知する。tier2 / tier3 開発チームへのメール通知、Backstage の k1s0 カタログページへの告知バナー、RSS/Slack 連携での自動通知、各 API 別詳細方式ドキュメント（[06_API別詳細方式/](06_API別詳細方式/)）への「非推奨予告」節追加。通知漏れ防止のため、MUST となる通知先のリストは [../../../03_要件定義/40_運用ライフサイクル/](../../../03_要件定義/40_運用ライフサイクル/) の非推奨と EOL 方針と同期する。

## 互換性テストの自動化

後方互換性を人間が毎回チェックすることは原則 9（2 名運用）と両立しない。CI で機械的に検出する仕組みを整え、違反はビルドブロック（ブランチプロテクション）で防ぐ。

**設計項目 DS-SW-EIF-055 buf breaking による破壊変更検出**

`.proto` の変更を含む PR は CI で `buf breaking --against '.git#branch=main'` を実行する。buf は Protobuf の後方互換性を 30 以上のルールで検証しており、フィールド削除・型変更・number 変更・enum 値削除などを静的検出する。検出時は PR がマージ不可となる。buf の設定ファイル `buf.yaml` で `FILE` カテゴリの全ルールを有効化し、`WIRE_JSON` カテゴリも追加適用する。

**設計項目 DS-SW-EIF-056 protoc-gen-validate と API コントラクトテスト**

API コントラクトは `protoc-gen-validate` でフィールド単位の制約（min_len、pattern、range など）を `.proto` 内に宣言する。CI は旧 v1 のサンプルリクエスト集を新 v1 で再実行するコントラクトテストを実施し、意味論レベルでの破壊を検出する。サンプル集はリクエスト 50 件、レスポンス期待 50 件を API 別に固定し、Git 管理下に置く。`docs/04_概要設計/20_ソフトウェア方式設計/img/` 配下の `.proto` を生成物の参照元とする。

**設計項目 DS-SW-EIF-057 SDK のバージョン固定とフローティング**

クライアント SDK（Go / Rust / TypeScript / C#）はバージョン番号を `<major>.<minor>.<patch>` 形式で公開する。tier2 / tier3 開発者は以下のポリシーで SDK バージョンを選択する。

- **フローティング推奨**: `~1.2.0`（パッチは自動追従、minor は手動昇格）。バグ修正の自動取り込みと安定性の両立。
- **固定許容**: `1.2.3`（完全固定）。特定バージョンでの動作保証が必要な業務向け。
- **major フローティング禁止**: `^1.0.0`（minor も自動追従）は禁止。minor で意図しない機能変更が入る可能性を避けるため。

Phase 1a 段階は SDK のマイナーバージョンが頻繁に上がる前提とし、tier2 / tier3 開発者には手動昇格の手順を [../05_利用者文書_暫定版/01_tier2_tier3開発者向けGoldenPath初版.md](../05_利用者文書_暫定版/01_tier2_tier3開発者向けGoldenPath初版.md) で提示する。

## 非破壊的変更の取り扱い

非破壊的変更は minor バージョン昇格で提供する。tier2 / tier3 開発者に事前予告は不要とするが、リリースノートの自動生成は必須とする。

**設計項目 DS-SW-EIF-058 minor 変更のリリース頻度と通知**

minor リリースは月次を上限とする。月次を超える頻度は tier2 / tier3 側の学習コストを圧迫するため禁止する。リリースノートは Conventional Commits（`feat:` / `fix:` / `chore:`）の Git 履歴から `git-cliff` で自動生成し、Backstage 上で公開する。

**設計項目 DS-SW-EIF-059 enum 拡張時の UNSPECIFIED 運用**

enum は全定義で `0: UNSPECIFIED` を先頭に持つことを強制する。新 enum 値を追加した際、旧クライアント SDK は未知値を `UNSPECIFIED` として解釈する。これは Protobuf の仕様に基づくが、旧 SDK 側の実装で「UNSPECIFIED 受領時のフォールバック動作」を明示化することが必須となるため、SDK 生成テンプレートに既定ハンドラを埋め込む。

## バージョン管理のガバナンス

バージョン管理は単なる技術問題ではなく、tier1 チームと tier2 / tier3 チームの協調に関わる組織問題でもある。権限と責任を明示する。

**設計項目 DS-SW-EIF-060 バージョン昇格の権限**

major 昇格は Product Council の承認を必須とする。minor 昇格は tier1 アーキテクトの承認で可とする。patch は tier1 開発者の自己承認でマージ可。この権限分離は変更の影響範囲に比例させる。

**設計項目 DS-SW-EIF-061 非推奨 API の利用状況モニタリング**

Deprecation 期間中の v1 への実アクセス量は Prometheus メトリクス `k1s0_api_request_total{version="v1"}` で計測し、Grafana ダッシュボードで可視化する。EOL 予定日 30 日前までにアクセス量が 0 に収斂しない場合、Product Council で延長判断を行う。延長は最大 6 か月まで、それを超える延長は新たに Phase 2 スコープでの承認を要する。

## フェーズ別の適用範囲

バージョニング方針は Phase 1a 時点から全 API に適用する。Phase 1a の段階で「まだ v1 しかないから」と省略すると、Phase 1b 以降で後付け導入するコストが発生するため、初期から適用する。

**設計項目 DS-SW-EIF-062 Phase 別適用スコープ**

| Phase | 適用範囲 | 根拠 |
|-------|---------|------|
| Phase 1a | `k1s0.public.*.v1.*` の生成、buf breaking の CI 必須化 | 初期から仕組みを動かす |
| Phase 1b | Deprecation Warning 機能、SDK の複数バージョン配信 | 並行提供の準備 |
| Phase 1c | 実利用者向け非推奨通知チャネル、Backstage ダッシュボード | 運用品質確保 |
| Phase 2 | 初の v2 昇格（必要時）、18 か月並行提供の実運用 | 基盤拡張 |
