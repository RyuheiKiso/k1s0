# 05. レガシー連携インタフェース方式

本ファイルは tier1 と 採用側組織内既存システム（.NET Framework 資産、SAP、オンプレ Salesforce、メインフレーム）を接続する外部インタフェースを固定化する。企画で約束した「既存資産を捨てずに共存する」（制約 9）を方式設計に落とし込み、強制移行せずに自発的移行を促す仕組みを確立する。

## 本ファイルの位置付け

採用側組織の情シス基盤では、.NET Framework で構築された業務アプリが数百本規模で稼働している。SAP、Salesforce、メインフレームなどの商用パッケージも長期契約で利用しており、これらを全廃して k1s0 に移行することは現実的でない。本ファイルは tier1 を既存資産と共存する形態で運用する方式を固定化する。

「共存」の定義は 2 段階ある。第一段階は既存資産から k1s0 の tier1 API を呼び出せること（新規機能追加時に tier1 を選択できる）。第二段階は既存資産そのものを tier3 として k1s0 配下に取り込むこと（既存 .NET アプリを ClickOnce/MSIX で配信ポータル経由に置く）。本ファイルは両者のインタフェースを並存設計する。

## 連携アーキテクチャの全体像

レガシー連携は 3 系統で構成する。REST クライアント経由の tier1 呼び出し（HTTP/JSON）、イベント駆動連携（Kafka 経由）、そして既存 .NET 資産の tier3 配信化の 3 つ。どの系統も Envoy Gateway + WAF を経由させ、一貫したセキュリティ統制下に置く。

**設計項目 DS-SW-EIF-140 3 系統連携の責務分離**

3 系統の責務は以下のとおり。

1. **REST 同期連携**: 既存 .NET Framework / SAP / Salesforce から tier1 API を HTTP/JSON で呼び出す。リクエスト単位の同期処理、短寿命な業務アクション向け。
2. **イベント駆動連携**: 既存資産が Kafka Producer として tier1 PubSub トピックへ配信、または tier1 が Output Binding で既存資産へプッシュ。長寿命な非同期ワークフロー、バッチ連携向け。
3. **tier3 配信化**: 既存 .NET アプリを ClickOnce / MSIX パッケージ化して配信ポータルから配布、tier3 拡張ポイントとして共存。新規クライアント展開・バージョン管理の統一を実現。

リリース時点で系統 1（REST 同期連携） / 系統 2（イベント駆動） / 系統 3（tier3 配信化）の 3 系統を一気通貫で提供する。3 系統の並存は採用後の運用拡大時の完了時点で本格定着する。

## REST 同期連携方式

.NET Framework 4.8 や古い商用パッケージは gRPC 公式サポートがなく、HTTP/JSON REST が事実上唯一の連携手段である。tier1 の gRPC API は Envoy Gateway の JSON Transcoder で HTTP/JSON に変換して公開する。

**設計項目 DS-SW-EIF-141 gRPC-REST 変換の方式**

Envoy Gateway に `envoy.filters.http.grpc_json_transcoder` を設定し、tier1 の全 API を HTTP/JSON としても公開する。マッピングは `.proto` ファイル内の `google.api.http` オプションで宣言する。例: `rpc Get(GetRequest) returns (GetResponse) { option (google.api.http) = { get: "/v1/state/{key}" }; }`。

HTTP/JSON 経由時の制約は以下。バイナリフィールド（`bytes`）は Base64 エンコード必須、エラーは `{"code":"...","message":"...","trace_id":"..."}` の固定形、ストリーミング（Telemetry / Log）は未対応、各エンドポイントは RESTful 命名（`/v1/state/{key}` など）を厳守する。

**設計項目 DS-SW-EIF-142 .NET Framework 4.8 からの呼び出し**

.NET Framework 4.8 は `System.Net.Http.HttpClient` で tier1 HTTP/JSON API を呼び出せる。SDK として `k1s0-sdk-netfx-http`（NuGet 配信）を提供し、型安全な DTO・自動リトライ・認証トークン管理を組み込む。

.NET Framework 4.8 は TLS 1.3 をネイティブサポートしない（TLS 1.2 までが限界）。Envoy Gateway は TLS 1.2 への fallback を 採用初期 は許容するが、リリース時点 で TLS 1.3 強制に移行する。.NET Framework 4.8 クライアントはこのタイミングで .NET 8 への移行が必要となる。移行パスは別途 [../../60_移行方式設計/](../../60_移行方式設計/) で展開する。

**設計項目 DS-SW-EIF-143 認証と AD/LDAP 連携**

既存の 採用側組織のAD（Active Directory）/ LDAP は Keycloak User Federation で連携する。ユーザは AD ドメイン（例: `EXAMPLE\user001`）でログインでき、Keycloak が内部的に LDAP Bind 検証、成功したら JWT を発行する。tier1 はその JWT を通常の認証フローで検証する。

この連携により、.NET Framework 資産の利用者は既存 AD 認証情報のままで tier1 API を使える。パスワード二重管理を避けることで、利用者の心理的抵抗を最小化する。詳細な Keycloak 構成は [../../30_共通機能方式設計/](../../30_共通機能方式設計/) で展開する。

## イベント駆動連携方式

既存資産の中には Kafka Producer / Consumer を組み込める（特に SAP、Salesforce の連携ミドルウェア経由）ものがある。これを活用し、tier1 PubSub API をイベントハブとして使う。

**設計項目 DS-SW-EIF-144 既存資産からの Kafka 配信**

既存資産は自身の Kafka クライアントで tier1 配下の Kafka Broker に直接 Publish できる。ただし制約として以下を満たす必要がある。

- SASL/SCRAM-SHA-512 認証必須。テナントごとに Kafka ACL で Publish 可能トピックを限定。
- トピック命名 `legacy/<tenant_id>/<system_id>/<event_type>` に従う。tier1 Subscriber 側は Dapr Binding Input でこの命名をフィルタリング。
- スキーマは Confluent Schema Registry 互換形式（Avro または Protobuf）で Registry に登録、Producer は Schema ID を含めて送信。

既存資産から直接 Kafka を喋れない場合は、REST-to-Kafka Bridge（Envoy Gateway の HTTP/JSON → PubSub Publish ルート）で代替する。Bridge は リリース時点 で提供。

**設計項目 DS-SW-EIF-145 tier1 から既存資産への Output Binding**

tier1 から既存資産への送出は Dapr Binding Output を用いる。対応 Binding は HTTP（REST Post）、Kafka（Topic Publish）、SMTP（メール配信）、File（NFS / S3 互換ストレージへのファイル吐き出し）の 4 種。

SAP 連携では SAP 側の iDoc 受信エンドポイントへ Output Binding で HTTP POST し、SAP 側で処理する。Salesforce 連携は Salesforce の REST API 向け Output Binding。メインフレーム連携は MQ 経由または NFS 経由のファイル連携のみ 採用後の運用拡大時 でサポート。

**設計項目 DS-SW-EIF-146 データフォーマット変換**

既存資産と tier1 はデータフォーマットが異なる。XML（SAP iDoc、SOAP）/ EDI（EDIFACT、ANSI X12）/ CSV / 固定長 / JSON / Protobuf が混在する。変換は以下の順序で行う。

- **既存資産 → tier1 方向**: エッジ（Binding Input）で XML/EDI/CSV を Protobuf に変換。変換ロジックは Rust の `serde` 互換ライブラリで実装し、変換誤り時は DLQ 転送。
- **tier1 → 既存資産方向**: Output Binding で Protobuf を XML/EDI/CSV にシリアライズ。テンプレートベース（Handlebars / Tera）で宣言的に定義。

変換テンプレートは tier2 業務ドメインチームが管理する。tier1 は変換エンジンを提供するのみで、個別テンプレートは責任範囲外とする。

## トランザクション整合性

既存資産との連携では 2 相コミット（2PC）が使えない前提で設計する。JTA / XA トランザクションは SAP など限定的な商用製品のみが対応しており、k1s0 全体の共通基盤では採用できない。

**設計項目 DS-SW-EIF-147 同期 REST + 冪等キーによる再送保証**

同期 REST 連携では冪等キー（`Idempotency-Key` ヘッダ）を必須とする。既存資産側で UUIDv7 を生成し、tier1 側で Valkey に記録、同一キー再送時は初回結果を返す。

この方式により「ネットワークエラー時の再送で重複処理される」問題を構造的に解決する。既存資産側で冪等キー生成を実装しづらい場合は、Envoy Gateway に自動生成フィルタを 採用後の運用拡大時 で導入する（リクエストボディの SHA-256 をキーとして扱う、業務的整合性は弱まるが ad-hoc 対応として有効）。

**設計項目 DS-SW-EIF-148 Saga 補償による最終整合性**

跨がりトランザクションが必要な場合（例: tier1 State 更新 + SAP iDoc 送出）は Saga 補償で最終整合性を担保する。Workflow API（Temporal または Dapr Workflow）で Saga を組み、失敗時の補償アクション（State ロールバック、SAP 取消 iDoc 送出）を明示定義する。

Saga 方式の詳細は [../../40_制御方式設計/](../../40_制御方式設計/) で展開する。本ファイルは「既存資産との連携で Saga を使う」という方針確認に留める。

## tier3 配信化（ClickOnce / MSIX）

既存の .NET Framework アプリを廃棄せず、tier3 拡張として取り込む方式を提供する。これは構想設計 [ADR-MIG-002](../../../02_構想設計/03_技術選定/) で確定している方針の実装である。

**設計項目 DS-SW-EIF-149 ClickOnce / MSIX 配信ポータル**

既存 .NET アプリを ClickOnce（.NET Framework 4.8 対応）または MSIX（.NET 8 対応）形式でパッケージ化し、k1s0 配信ポータル経由で配布する。配信ポータルは Kubernetes 上に配置した Backstage プラグインとして実装、バージョン管理・配布先ユーザ限定・起動履歴を一元管理。

この仕組みにより、既存 .NET アプリは「tier3 拡張」として k1s0 エコシステムに統合される。アプリ内から tier1 HTTP/JSON API を呼び出すことで、徐々に tier1 価値を取り込める。強制移行ではなく、開発者が自発的に tier1 を使いたくなる設計（Golden Path）が前提となる。

**設計項目 DS-SW-EIF-150 sidecar 方式による既存 exe の tier3 統合**

一部の業務 exe はパッケージ化できない（COM 依存、ネイティブ DLL 依存、GUI 複雑）。これらは「sidecar」として Kubernetes Pod 内に Windows Container で起動し、ネットワーク越しに tier1 を呼ぶ構成を 採用後の運用拡大時 で提供する。

Windows Container の運用は情シス基盤チームが管理し、tier3 開発チームは「使いたい exe を申請すれば tier3 Pod 内で起動する」という抽象で利用できる。この方式は 採用側の小規模運用原則と両立するため、運用工数の増加分（月 10 時間見込み）を事前確保する。

## 段階的移行方針

強制移行は禁止し、自発的移行を促す設計とする。既存資産と k1s0 の同時稼働期間は許容する。

**設計項目 DS-SW-EIF-151 移行パスと並存期間**

段階的移行は以下の 4 段階とする。

1. **共存期**（採用初期）: 既存資産が稼働しつつ、新機能は k1s0 ネイティブで開発。既存資産から tier1 API を呼び出せる環境を整備。
2. **統合期**（リリース時点〜採用後の運用拡大時 初期）: 既存資産を tier3 配信化、sidecar 化で k1s0 配下に取り込む。ClickOnce / MSIX 移行完了。
3. **選択的置換期**（採用後の運用拡大時 中期）: 保守困難な既存資産から優先的に tier2 / tier3 ネイティブへ書き換え。機能等価性の自動テストを整備。
4. **長期並存期**（採用後の運用拡大時）: 書き換え困難な既存資産は並存維持。完全廃止は事業判断で個別決定。

リリース時点点で「既存資産は配信ポータル経由に統一、新機能は tier1 経由」が達成されていることを目標とする。

**設計項目 DS-SW-EIF-152 段階移行のメトリクス**

移行の進捗は以下のメトリクスで Grafana 可視化する。

- `legacy_app_total`（tier3 配信化済みアプリ数 / 未対応アプリ数）
- `legacy_api_call_total`（既存資産から tier1 API への呼び出し量）
- `legacy_kafka_publish_total`（既存資産から Kafka 経由配信量）
- `native_tier2_app_total`（tier2 ネイティブで新規開発されたアプリ数）

四半期ごとに Product Council で進捗レビュー、目標達成率と資産削減額を事業報告する。
