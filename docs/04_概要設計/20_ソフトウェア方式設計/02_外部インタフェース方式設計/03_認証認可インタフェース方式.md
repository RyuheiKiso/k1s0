# 03. 認証認可インタフェース方式

本ファイルは tier1 公開 11 API の認証・認可インタフェースを固定化する。認証は「誰が呼んでいるか」を確立し、認可は「その呼び出しが許されているか」を判定する、独立した 2 レイヤである。本ファイルは両者の入口（Envoy Gateway）・トークン形式（JWT / SVID）・伝搬プロトコル（Bearer、mTLS）・エラー応答（401 / 403）を外部インタフェースとして仕様固定化する。

## 本ファイルの位置付け

認証・認可は全 11 API で共通して必須となる横断的関心事である。[01_tier1_11API方式概要.md](01_tier1_11API方式概要.md) の共通契約で全 API に JWT 必須と書き、本ファイルはその「JWT が具体的に何で、どのフィールドが必須で、どこで検証されるか」を契約として固定する。認証の内部実装（Keycloak の設定、SPIFFE / SPIRE の構成）は [../../30_共通機能方式設計/](../../30_共通機能方式設計/) の認証方式で展開し、本ファイルは外部から見えるインタフェースに絞る。

tier1 は tier2 / tier3 の複雑性を吸収する立場であり、認証・認可も「tier2 / tier3 開発者が意識しない」ことが理想状態である。ただし認証情報の伝搬は開発者に認識されざるを得ず、「JWT を付ければ後は tier1 が処理する」という最小限の契約を明示する。

## 認証アーキテクチャの全体像

認証は「人間ユーザ認証」と「サービス間認証」の 2 系統を並存させる。人間ユーザは Keycloak が発行した OIDC JWT を、サービス間は SPIFFE SVID（X.509 と JWT の両形式）を用いる。両者は同じ Envoy Gateway で検証され、tier1 アプリケーションには認証済みの `principal` 情報のみが伝搬される。

**設計項目 DS-SW-EIF-080 認証経路の 2 系統分離**

認証経路は以下の 2 系統を並存させる。

1. **人間ユーザ経路**: tier2 / tier3 アプリケーション → 開発者が持つブラウザ / CLI → OIDC Authorization Code Flow → Keycloak → JWT 発行 → tier1 API 呼び出し時に JWT 提示。
2. **サービス間経路**: tier2 / tier3 サービス Pod → SPIRE Agent → SVID 発行（X.509 mTLS + JWT）→ tier1 API 呼び出し時に mTLS + JWT 提示。

両経路は Envoy Gateway で統一的に検証する。人間ユーザは `iss=https://keycloak.k1s0.internal/realms/k1s0-jtc`、サービスは `iss=https://spire.k1s0.internal/`（SPIRE JWT-SVID Issuer）で区別する。ユースケースに応じてどちらの経路を用いるかは tier2 / tier3 側で選択する。

**設計項目 DS-SW-EIF-081 Envoy Gateway での TLS 終端と JWT 検証**

外部入口は Envoy Gateway 1 箇所に集約する。Envoy は以下 4 段階でリクエストを処理する。

1. TLS 1.3 で接続を終端（クライアント証明書検証は mTLS の場合のみ有効）。
2. `authorization` ヘッダから JWT を抽出、`iss` と `aud` に基づき JWKS（Keycloak または SPIRE）を選択し署名検証。
3. JWT の claims（`sub` / `tenant_id` / `roles` / `exp` / `nbf`）をパースし、gRPC metadata `x-k1s0-principal-*` 系に展開して tier1 へ転送。
4. tier1 API は `x-k1s0-principal-*` 系のみを信頼し、`authorization` ヘッダ自体は Envoy が剥がす（再転送不可）。

Envoy → tier1 間は内部ネットワーク（Istio Ambient の ztunnel で mTLS）であり、ヘッダ詐称は構造的に不可能となる。この「Envoy 以外で JWT を検証しない」方針は tier1 Pod の負荷軽減と一貫したポリシー適用を両立する。

## JWT 契約（人間ユーザ）

Keycloak が発行する JWT は以下フィールドを必須とする。フィールド欠落時は Envoy で 401 を返し、tier1 Pod へはルーティングしない。

**設計項目 DS-SW-EIF-082 JWT 必須クレーム**

| クレーム | 型 | 意味 | 根拠 |
|---------|---|------|------|
| `iss` | string | 発行者 URL。Keycloak realm URL と完全一致 | JWKS 選択に必須 |
| `aud` | string / array | 対象 API。`k1s0-api` を含むことが必須 | 他 realm 流用防止 |
| `sub` | string | ユーザ識別子（Keycloak の user UUID） | user_id 導出に使用 |
| `tenant_id` | string | テナント ID（UUIDv7） | RLS / key-scoping に必須 |
| `roles` | array | 権限ロール一覧 | RBAC 判定に使用 |
| `exp` | int | 有効期限（Unix 秒） | 盗難リスク最小化（15 分推奨） |
| `nbf` | int | 有効開始時刻（Unix 秒） | 未来発行トークン対策 |
| `jti` | string | トークン一意 ID（UUIDv7） | 監査ログでのトークン追跡 |

追加の claims（`email` / `preferred_username` / `groups` / `org_id`）はオプションで、必要に応じて tier1 が参照する。ただし PII（email、preferred_username）は Audit-Pii API への記録時に自動マスクされる。

**設計項目 DS-SW-EIF-083 JWT 失敗時の応答仕様**

JWT 検証失敗時は以下の応答を返す。401 は認証失敗（トークン自体の問題）、403 は認可失敗（トークンは有効だが権限不足）として厳密に区別する。

- **401 Unauthorized** + `WWW-Authenticate: Bearer realm="k1s0-api", error="invalid_token", error_description="..."`。`error` は `invalid_token` / `invalid_request` / `insufficient_scope` のいずれか（RFC 6750 準拠）。
- **403 Forbidden** + エラーボディ `{"code":"AUTHZ_DENIED","message":"...","trace_id":"..."}`。リソースの存在有無を攻撃者に推測されないよう、404 ではなく 403 を用いる（認可ポリシーで明示判定）。
- 監査ログに JWT の `jti` と失敗理由を記録。

**設計項目 DS-SW-EIF-084 匿名アクセスの全 API 禁止**

tier1 の全 11 API で匿名アクセスは禁止する。Phase 1a 段階でも例外なく、ヘルスチェック用の `/healthz` と `/readyz` 以外はすべて JWT 必須とする。ヘルスチェックは Envoy 前段で応答し、tier1 Pod の `/healthz` は kubelet / Istio からの内部呼び出しのみ受け付ける。

匿名アクセスを許容すると「開発段階の便宜」が本番に残り、セキュリティ事故の温床となる。Dev / Pre 環境でも Keycloak Dev Realm の JWT 発行を前提に、匿名アクセスを許容しない。

## mTLS 契約（サービス間）

サービス間認証は SPIFFE SVID X.509 による mTLS と、JWT-SVID の併用で行う。mTLS はサービス identity を接続レベルで確立し、JWT はリクエスト単位の delegation 情報を運ぶ。

**設計項目 DS-SW-EIF-085 SPIFFE SVID の形式と trust domain**

trust domain は `k1s0.internal.example.jp` の 1 つとする。SVID の subject は `spiffe://k1s0.internal.example.jp/ns/<namespace>/sa/<serviceaccount>` 形式で、Pod の namespace と ServiceAccount に 1:1 対応する。Phase 1b 以降は全 tier1 / tier2 / tier3 Pod に SVID を配布する。

X.509 SVID の TTL は 1 時間、自動ローテーション間隔は 30 分とする。JWT-SVID の TTL は 5 分、呼び出しごとに取得する。短命化により盗難時の被害範囲を構造的に限定する。

**設計項目 DS-SW-EIF-086 mTLS 強制の段階適用**

mTLS 強制は以下の段階で適用する。

- **Phase 1a**: mTLS は Istio Ambient の ztunnel で自動提供、ポリシーは PERMISSIVE（平文も受け付ける）。
- **Phase 1b**: Namespace 単位で STRICT（平文拒否）へ段階移行。tier1 namespace を最初に STRICT 化。
- **Phase 1c**: 全 namespace STRICT。AuthorizationPolicy で SPIFFE ID 単位のホワイトリスト適用。

この段階は運用安定性を優先し、Phase 1a で一括 STRICT 化した場合に発生する「証明書未配布の Pod が動かない」事故を避ける。

## 認可方式（RBAC + ABAC）

認可は RBAC（Role-Based Access Control）を基盤とし、細粒度の判定は ABAC（Attribute-Based Access Control）で補う。ABAC 判定は Decision API（ZEN Engine）で実装する。

**設計項目 DS-SW-EIF-087 RBAC ロール階層**

Phase 1a 時点の RBAC ロールは以下の 6 階層とする。

| ロール | 権限範囲 | 想定利用者 |
|-------|---------|----------|
| `platform-admin` | tier1 全機能管理 | 情シス基盤チーム（2 名） |
| `security-admin` | 監査ログ閲覧、Secrets 管理 | SecOps |
| `tenant-admin` | 特定テナント内の全 tier1 API | 事業ドメイン責任者 |
| `developer` | tier1 API の標準利用 | tier2 / tier3 開発者 |
| `viewer` | tier1 API の読み取り専用 | 監査人、ステークホルダ |
| `anonymous` | 利用不可（全 API 401） | 未認証アクセス |

ロールは JWT の `roles` claim に配列で入る。ユーザは複数ロールを同時に持てる（例: `["developer", "tenant-admin"]`）。権限判定は最も強い権限が採用される（union of permissions）。

**設計項目 DS-SW-EIF-088 ABAC による細粒度判定**

RBAC で「できる / できない」を粗く判定した後、ABAC で「このリクエストは本当に許可してよいか」を細粒度判定する。判定は Decision API（ZEN Engine）で行い、入力は JWT claims、リクエスト属性（API、method、resource_id）、時刻、リクエスト元 IP など。

典型例として、`developer` ロール + `tenant_id=T001` のユーザが、Audit-Pii API を「自テナントのログのみ」読み取れる制約、Secrets API を「特定命名規則のシークレットのみ」操作できる制約、などを ZEN Engine の決定テーブルで定義する。決定テーブルは [../../30_共通機能方式設計/](../../30_共通機能方式設計/) の認可方式で採番する。

**設計項目 DS-SW-EIF-089 OAuth 2.1 Token Delegation（Phase 2）**

Phase 2 では OAuth 2.1 Token Delegation（RFC 8693 Token Exchange）を実装し、tier3 アプリが tier2 経由で tier1 を呼ぶ際のサブジェクト委譲を正確に行う。具体的には tier3 の user trusted JWT から tier2 用 trusted JWT を発行し、tier1 までに「元の user」と「中継した tier2 サービス」の両 identity を追跡可能にする。この仕組みは Phase 2 スコープで Keycloak の `token_exchange` エンドポイントを利用する。Phase 1a〜1c では単一 JWT で統一する。

## Envoy Gateway の外部契約

Envoy Gateway は tier2 / tier3 から見た「唯一の tier1 入口」である。エンドポイント・プロトコル・ポートを固定化する。

**設計項目 DS-SW-EIF-090 外部エンドポイント仕様**

| 項目 | 値 | 根拠 |
|------|-----|------|
| ドメイン | `api.k1s0.internal.example.jp` | 内部 DNS で解決 |
| ポート | 443（TLS 1.3） | HTTPS 標準 |
| TLS バージョン | 1.3 のみ（Phase 1c〜）、1.2 は Phase 1a〜1b 移行期のみ | 1.2 は後方互換、1.3 で前方秘匿性強化 |
| サーバ証明書 | 内部 CA 発行、cert-manager 管理 | 閉域運用前提 |
| クライアント証明書 | サービス間経路のみ必須、人間ユーザは任意（Phase 2 MFA 連携時） | mTLS と JWT の選択 |
| ALPN | `h2`（gRPC）/ `http/1.1`（JSON） | 2 経路並存 |

**設計項目 DS-SW-EIF-091 Rate Limit による保護**

Envoy Gateway で Rate Limit を多段適用する。IP 単位 100 RPS、テナント単位 1,000 RPS、API 単位でのさらに個別制限（Decision API 3,000 RPS、State Get 5,000 RPS など）。超過時は 429 + `X-K1s0-RateLimit-Reset` で次回利用可能時刻を通知。Rate Limit バックエンドは Valkey で、テナント単位のカウンタを秒単位でウィンドウ化する。

## 認証情報の tier1 内部伝搬

Envoy が検証した認証情報は tier1 内部に metadata として渡される。tier1 側は metadata のみを信頼する。

**設計項目 DS-SW-EIF-092 tier1 内部 metadata 契約**

Envoy から tier1 への gRPC metadata は以下を付与する。

- `x-k1s0-principal-type`: `user` または `service`
- `x-k1s0-principal-sub`: JWT `sub` または SPIFFE ID
- `x-k1s0-principal-tenant-id`: JWT `tenant_id`
- `x-k1s0-principal-roles`: カンマ区切りロール一覧
- `x-k1s0-principal-jti`: JWT `jti`（監査証跡用）
- `x-k1s0-auth-method`: `oidc` / `spiffe-x509` / `spiffe-jwt`

tier1 Pod は `authorization` ヘッダを受け取らず、これら metadata のみを参照する。ヘッダ詐称は Ambient mTLS で防御されているため、metadata は信頼可能な情報源となる。
