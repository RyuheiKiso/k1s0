# 07. ソフトウェア方式設計 評価・共同レビュー

本ファイルは IPA 共通フレーム 2013 の **7.1.2.6 ソフトウェア方式設計の評価及び共同レビュー** に対応する。20_ソフトウェア方式設計カテゴリ（01_コンポーネント方式設計 〜 08_APIシーケンス方式）の全内容を対象に、完全性・整合性・実装可能性の 3 軸で評価する枠組みと、共同レビュー参加者・レビュー観点・指摘事項管理手順・次段詳細設計への引渡し条件を定める。

## 本ファイルの位置付け

ソフトウェア方式設計書は章数が多く、複数担当者が並行執筆するため、章間で用語・ID・数値の不整合が構造的に起こりやすい。例えば、tier1 の State API p99 目標値が 01 コンポーネント方式で「10ms」と書かれ、06 結合テスト要件で「15ms」と書かれていた場合、どちらが正かを実装フェーズで判断できず、手戻りが発生する。さらに、要件 ID（FR-T1-* 49 件）のカバレッジ穴が残ったまま Phase 1b に進むと、稟議で約束した機能が実装されない事態が現実に起こりうる。

本章はこれらのリスクを Phase 1a 着手前に潰すための評価・共同レビュー枠組みである。全章を対象にチェックリスト方式で評価し、指摘事項を Product Council 承認のもとで是正する。Phase 0 稟議時点では評価の「仕組み」のみを確定し、実際の評価結果（指摘事項と是正記録）は Phase 1a 着手直前の評価会で記入する。

本章は評価の実施者・実施時期・観点・記録様式を固定化する。誰が・いつ・何を・どう記録するかが曖昧だと評価自体が形骸化するため、実施前に本章で合意を固める。

## レビュー対象範囲

本章の評価対象は 20 ソフトウェア方式設計カテゴリ配下の全ファイル群である。章立てごとの分担は以下となる。

- 01 コンポーネント方式設計（6 ファイル）: tier1 全体俯瞰、Dapr ファサード層、自作 Rust 領域、責務一覧、依存関係、パッケージ構成
- 02 外部インタフェース方式設計（5 ファイル + サブディレクトリ）: 11API 概要、バージョニング、認証認可、エッジ IoT、レガシー、API 別詳細
- 03 内部インタフェース方式設計（複数ファイル）: Go/Rust 言語境界、gRPC スキーマ、内部エラー体系
- 04 データベース最上位設計（複数ファイル）: PostgreSQL スキーマ、Valkey データモデル、Kafka トピック設計
- 05 利用者文書暫定版（5 ファイル）: Golden Path / 運用者手引き / Backstage / 配信ポータル / CLI
- 06 結合テスト要件（1 ファイル）
- 08 API シーケンス方式（1 ファイル）

評価対象外は 10 システム方式設計・30 共通機能方式設計等、他カテゴリの章である。これらは各カテゴリ末尾のレビュー章で個別に評価する。ただし、20 章内の記述が他カテゴリの記述と整合する必要があるため、整合性軸の評価では他カテゴリ記述との参照突合も行う。

## 設計項目 DS-SW-REV-001 評価の 3 軸 — 完全性・整合性・実装可能性

評価は 3 軸で実施する。各軸は独立に評価し、全軸で合格して初めて Phase 1a 着手を許可する。

**完全性軸**は、要件定義書側の機能要件 FR-T1-* 49 件 と 非機能要件 NFR-* 主要項目が、ソフトウェア方式設計書のいずれかの設計項目でカバーされているかを検証する。カバレッジ穴が残ったまま Phase 1a に進むと、実装時に「設計がない」「要件が無視されている」事態が発生する。カバレッジは要件 ID → 設計 ID のマトリクス（[../80_トレーサビリティ/02_要件から設計へのマトリクス.md](../80_トレーサビリティ/02_要件から設計へのマトリクス.md)）で機械検証する。

**整合性軸**は、章間で用語・ID・数値が一貫しているかを検証する。用語統一（tier1 / Dapr sidecar / Kubernetes / k1s0 / Valkey / OpenBao が章間で同一表記）、ID 一貫性（同一概念に対する設計 ID が章間で衝突していない）、数値一貫性（p99 目標値・SLO・タイムアウト値が章間で同一）の 3 サブ軸で確認する。

**実装可能性軸**は、設計内容が構想設計 ADR の制約範囲内であり、採用 OSS の機能範囲内で実装可能かを検証する。例えば Dapr Workflow で表現できないパターンを採用していないか、Valkey で実現できない永続化要件を前提としていないか、OpenBao で提供されない Lease 機構を期待していないか、を確認する。

## 設計項目 DS-SW-REV-002 レビュア構成と役割

共同レビューは以下 5 役の参加者で構成する。各役の責任範囲を明示することで、レビュー中の責任分担の曖昧化を防ぐ。

**主査**: アーキテクト 1 名。レビュー会全体の進行、指摘事項の集約、Product Council への最終報告を担う。評価の 3 軸のうち整合性軸を主担当とする。主査は章執筆者のいずれとも兼任しない（客観性担保）。

**技術副査（tier1 Go 側）**: Dapr ファサード層を熟知する開発リーダ 1 名。01 コンポーネント方式（Dapr 部分）・02 外部 API・03 内部 gRPC の実装可能性を重点評価する。

**技術副査（tier1 Rust 側）**: 自作領域 Rust の開発リーダ 1 名。01 コンポーネント方式（Rust 部分）・02 Decision / Audit / Feature API・03 内部 gRPC の実装可能性を重点評価する。

**運用副査**: JTC 情シス運用者 1 名。05 利用者文書（運用者手引き）・06 結合テスト要件（障害注入観点）の現実性を重点評価する。運用負荷が 2 名体制に収まるかを判断する。

**Product Council オブザーバ**: 1〜2 名。稟議承認済みスコープからの逸脱がないかを俯瞰する。特に予算・Phase スコープ・非機能コミットの逸脱を監視する。

レビュアは章執筆者と異なる人物を配置し、相互レビューとする。執筆者自身が自章をレビューすることは禁止する（盲点の見逃しが起こるため）。

## 設計項目 DS-SW-REV-006 レビュー実施手順

評価・共同レビューは以下の 5 段階で実施する。各段階の所要時間目安と成果物を明示する。

**段階 1 — 事前配布（レビュー会 1 週間前）**: 主査が 20 章全ファイルを PDF 化し、全レビュアに配布。レビュアは事前に通読し、気になる箇所を付箋ツール（Hypothesis）で指摘する。所要 5 時間／人。

**段階 2 — 事前チェックリスト実施（レビュー会 3 日前）**: 各レビュアが担当軸のチェックリストを埋め、未達項目を起票する。起票は GitHub Issues（ラベル: `review-20-sw-design`）で一元管理。所要 4 時間／人。

**段階 3 — 評価会本番（対面またはリモート、4 時間）**: 主査が進行、未達項目を 1 件ずつ議論し、是正方針・担当・期限を決定する。決定事項は議事録に記録。

**段階 4 — 是正対応（レビュー会後 2 週間）**: 章執筆者が指摘事項に従って設計書を改訂。改訂 PR は主査が最終承認する。

**段階 5 — Product Council 承認（Phase 1a 着手の 1 週間前）**: 主査が指摘・是正結果をまとめて Product Council に提示。承認されれば Phase 1a 着手。

評価会は Phase 0 稟議承認後・Phase 1a 着手前の 1 回を基本とする。章大改訂時は臨時評価会を開くことができる。

## 設計項目 DS-SW-REV-007 指摘事項と是正記録の枠組み

指摘事項は以下の項目で記録する。Phase 0 時点では枠組みのみで、実記録は評価会本番で埋める。

- 指摘 ID（連番）
- 指摘日
- 指摘者（レビュア名）
- 指摘軸（完全性 / 整合性 / 実装可能性）
- 対象章・ファイル
- 指摘内容（1〜2 行で要約、詳細は本文）
- 是正方針（設計変更 / 記述修正 / 保留 / 却下のいずれか）
- 担当者
- 是正期限
- 是正完了日
- 是正内容（具体的変更箇所）
- 再確認者

指摘事項の記録は GitHub Issues（ラベル: `review-20-sw-design`）で管理する。Issue のクローズ条件は「再確認者が是正内容をレビューして OK を出す」ことであり、単純な「対応した」では閉じない。

却下となる指摘（レビュアの誤認・範囲外指摘）は「却下理由」を必ず本文に記録し、却下に対する合意を Issue コメントに残す。後から「なぜあの指摘が反映されなかったか」を追跡可能とする。

## 設計項目 DS-SW-REV-008 Phase 0 稟議時点の承認範囲

Phase 0 稟議時点では、20 章全体の「枠組み承認」を得る。個別設計項目の細部は Phase 1a 以降で確定するため、稟議時点での承認範囲を以下に明示する。

**稟議時点で承認を得る項目**: ソフトウェア方式設計全体の章構成・サブディレクトリ配置・採番体系、構想設計 ADR との整合、要件定義書との対応方針、5 利用者文書の骨子、結合テスト要件の骨子、API シーケンスの骨子、評価・レビューの仕組み（本章）。

**Phase 1a 以降で確定する項目**: 個別設計項目の本文詳細、API シーケンスの具体コード呼び出し列、結合テストの具体テストケース、利用者文書の具体コマンド系列・画面ショット、指摘事項の実際の是正記録。

**Phase 1b 以降で確定する項目**: Istio Ambient 採用時の設計差分、Temporal 採用時の Workflow 設計詳細、Backstage 本格統合時の利用者文書拡充。

**Phase 1c 以降で確定する項目**: 監査対応の実装詳細、災害復旧訓練の記録、PostgreSQL WORM の長期保管検証結果。

この範囲を稟議時に明示することで、「Phase 0 稟議では確定していない項目」を後の Phase で追加承認することが可能となる。稟議通過済みスコープを逸脱した追加は別途 Product Council 承認を要する。

## 設計項目 DS-SW-REV-009 次段（詳細設計）への引渡し条件

20 ソフトウェア方式設計の評価が完了し、Product Council 承認を得た時点で、次段の詳細設計フェーズ（IPA 共通フレーム 7.1.3 ソフトウェア詳細設計プロセス）へ引き渡す。引渡しの条件を以下に明示する。

- 完全性チェックリストの全項目が OK（要件カバレッジ 100%）
- 整合性チェックリストの全項目が OK（用語・ID・数値の章間一貫性）
- 実装可能性チェックリストの全項目が OK（採用 OSS 範囲内）
- 評価会での指摘事項が全て是正完了またはレビュー承認済み却下
- 要件 ID → 設計 ID マトリクスが最新化
- Product Council が引渡しを承認

引渡し時に、詳細設計フェーズ担当者へ以下を成果物として引き継ぐ。20 章全ファイル、設計 ID 一覧、指摘事項記録、未決事項リスト、次段で深掘りすべき論点リスト。

次段詳細設計は本評価の結果を前提として開始するため、引渡し後の方式設計変更は詳細設計への影響を伴う。変更時は詳細設計担当者への同時通知を必須とする。

## 設計項目 DS-SW-REV-010 評価の品質基準

評価そのものの品質を保つため、以下の基準を定める。評価が形骸化すると方式設計書の品質は保証されないため、評価自体のメタ評価も必要である。

- レビュア稼働時間が計画通り確保されている（事前 5 時間 + チェックリスト 4 時間 + 評価会 4 時間 = 13 時間／人）
- 指摘事項件数が章規模に対して極端に少ない（章あたり 0〜1 件）場合は、レビューが浅い可能性を疑って再実施する
- 指摘事項が特定のレビュアに偏っている場合（例: 1 人が 90% 起票）は、他レビュアの読み込みが浅い可能性を疑う
- 評価会議事録が公開され、レビュア以外（Product Council 以外の関係者）が後から辿れる
- 評価後 1 か月時点で、指摘事項の是正完了率が 90% 以上

評価のメタ評価は主査が Product Council に報告し、次回評価会の改善に反映する。評価が形骸化していると判断された場合、レビュア構成の変更・評価観点の追加を行う。
