# 01. 論理 ER モデル

本ファイルは IPA 共通フレーム 2013 の 7.1.2.3「データベースの最上位レベル設計」に対応する。tier1 が管理する全エンティティの論理 ER モデルを最上位で固定化し、エンティティ同士の関連・識別子・主要属性・PII 区分を概要設計視点で確定する。物理テーブル構造は [02_最上位テーブル一覧.md](02_最上位テーブル一覧.md) に、属性の詳細は [04_データ辞書.md](04_データ辞書.md) に委ねる。

## 本ファイルの位置付け

論理 ER モデルは物理 DB 設計の上位概念であり、「何のデータが存在し、何と何が関連するか」を物理ストアに依存せずに表現する。k1s0 では永続層が PostgreSQL（業務 / 監査）・Valkey（キャッシュ / 冪等）・Kafka（イベントログ）・MinIO（オブジェクト）・OpenBao（秘密）と 5 種に分散するため、論理モデルを先に確定してから物理配置を決める 2 段階アプローチが不可欠となる。

論理 ER モデルが固定化されていないと、「同じユーザ概念が Postgres の users テーブルと Valkey の user:* キーで別モデルで表現される」といった分裂が発生する。これは改修時の不整合の温床となるため、本ファイルで統一モデルを先に確立する。

## エンティティ一覧と責務

tier1 が管理する論理エンティティは 11 個を最上位とする。これらは要件定義 [FR-INFO-LOGMODEL/AUDITMODEL/TENANTMODEL/CONFIGMODEL](../../../03_要件定義/20_機能要件/) で挙げられた情報モデルに 1:1 対応し、tier1 11 API の責務範囲とも相関する。

**設計項目 DS-SW-DB-001 論理エンティティの 11 個列挙**

| # | エンティティ | 責務（何を表現するか） | 対応 API | 物理配置 |
|---|-----------|---------|----------|---------|
| 1 | Tenant | 採用側組織の事業単位、権限境界、課金単位 | 全 API の親エンティティ | PostgreSQL |
| 2 | User | 人間の利用者、Keycloak Subject との 1:1 対応 | 認証、Audit | PostgreSQL |
| 3 | Application | tier2 / tier3 サービスの登録情報、ServiceAccount | 全 API の呼び出し主体 | PostgreSQL |
| 4 | State | key-value の業務状態、TTL 付き | State API | Valkey Cluster（主）/ PostgreSQL（バックアップ） |
| 5 | Event | PubSub で流れるメッセージのメタデータ | PubSub API | Kafka（主）/ PostgreSQL メタ |
| 6 | Secret | 動的シークレットのリース情報（値は OpenBao） | Secrets API | PostgreSQL + OpenBao |
| 7 | FeatureFlag | 機能フラグの定義と評価ルール | Feature API | PostgreSQL（flagd 設定ストア） |
| 8 | AuditEvent | 監査証跡（WORM、ハッシュチェーン） | Audit-Pii API | PostgreSQL（7 年）+ MinIO（冷データ） |
| 9 | WorkflowExecution | 長期実行ワークフローの状態 | Workflow API | PostgreSQL / Temporal DB |
| 10 | Decision | 決定評価の履歴（サンプリング） | Decision API | PostgreSQL（サンプル）+ MinIO |
| 11 | BindingConfig | Input / Output Binding の構成 | Binding API | PostgreSQL + OpenBao（認証情報） |

継承関係は採用しない（Flat 構造）。ポリモーフィック関連も採用しない。Protobuf および SQL の両方で素朴に表現できる構造を優先する。

## 論理 ER 図

11 エンティティの関連を 1 枚の論理 ER 図で表現する。図は drawio レイヤ記法規約に従いデータ層色（`#e1d5e7` / `#9673a6`）で統一する。

![論理ERモデル](../img/論理ERモデル.svg)

図で表現している関連は以下のとおり。Tenant は User / Application / Secret / FeatureFlag / AuditEvent / WorkflowExecution / Decision の親となり、すべてのエンティティが tenant_id を外部キーで持つ（マルチテナント分離の基礎）。Application は State / Event / WorkflowExecution / BindingConfig の親となる（アプリ単位でのスコープ分離）。User は AuditEvent の関連エンティティで、監査イベントが「誰の行為か」を識別する。

関連の線種は 2 種類で表現している。実線は必須関連（外部キー NOT NULL）、破線は任意関連（外部キー NULL 許容）。Secret / FeatureFlag / Decision は Tenant に対して任意関連を持つ（プラットフォーム全体で共有される場合があるため）。Application との関連も、State は必須（app_id が必ず付与）、Event はアプリ発行者が必須、Binding は特定アプリに紐づくため必須とする。

## エンティティ間の関連詳細

図だけでは表現しきれない関連の意味論を散文で補足する。関連は単なる「外部キー制約」ではなく、業務ルールと監査要件を含む契約である。

**設計項目 DS-SW-DB-002 Tenant を頂点とする階層関連**

Tenant は全エンティティの祖先となる。User / Application / Secret / FeatureFlag / AuditEvent / WorkflowExecution / Decision / BindingConfig が tenant_id を外部キーで持ち、Tenant 削除時は ON DELETE RESTRICT で削除を阻止する。テナント削除は論理削除のみとし、物理削除は法務保留解除後に一括実行する。

State と Event は Tenant への直接関連を持たず、Application 経由で Tenant に辿る。これにより「アプリ単位でのスコープ分離」を一次、「テナント単位のスコープ分離」を二次とする階層設計となる。

**設計項目 DS-SW-DB-003 Application エンティティの意味**

Application は tier2 / tier3 サービスの登録情報を表す。各サービスは起動時に tier1 の `RegisterApplication` API を呼び、`app_id`（UUIDv7）と ServiceAccount を紐付ける。以降の tier1 API 呼び出し時、JWT の `sub`（ServiceAccount）から app_id を解決し、State / Event / Workflow / Binding のスコープを決定する。

Application は「技術的な呼び出し主体」を表し、User（人間利用者）とは明確に区別される。tier2 サービスが tier3 からの呼び出しを中継する場合、JWT の Delegation 情報で元 User を追跡可能にする（[03_認証認可インタフェース方式.md](../02_外部インタフェース方式設計/03_認証認可インタフェース方式.md) の DS-SW-EIF-089 参照）。

**設計項目 DS-SW-DB-004 AuditEvent の特殊性**

AuditEvent は WORM（Write Once Read Many）属性を持つ。Postgres レベルでは UPDATE / DELETE をトリガで禁止し、アプリケーション層でも同操作を禁止する。User との関連は nullable（システムイベントは User 不在）だが、Tenant との関連は必須とする。

hash_chain 属性が前の監査イベントの prev_hash と連鎖するため、削除・改変は構造的に不可能となる。この設計は J-SOX と個人情報保護法対応で必須となる監査証跡要件を満たす。

**設計項目 DS-SW-DB-005 State エンティティのキー構造**

State の主キーは `(tenant_id, app_id, key)` の複合キーとする。単一のグローバル key 空間ではなく、テナント × アプリ単位でスコープ分離する。この設計により、異なるアプリが同じ `key=session-xxx` を使っても衝突しない。

Valkey 物理配置時は `<tenant_id>:<app_id>:<key>` の文字列結合でキー化する。Redis Cluster のハッシュスロット分散で tenant_id をハッシュタグ化（`{<tenant_id>}:<app_id>:<key>`）することで、同一テナントのキーが同一スロットに配置され、MULTI/EXEC トランザクション可能になる。

## 主要属性と識別子方針

各エンティティの識別子（主キー）は UUIDv7 を標準採用する。タイムソートが効くため DB の INDEX 断片化が抑制され、スキーマ移行時の偶発的衝突リスクも低い。

**設計項目 DS-SW-DB-006 識別子の UUIDv7 採用**

全エンティティで主キーは UUIDv7 とする。自動採番（連番）は以下理由で禁止する。

- 連番はマルチマスタ構成で衝突する。PostgreSQL の BDR / ストリーミングレプリケーションで issue になる。
- 連番はテナント間で「どちらが先に登録されたか」が可視化される。機微情報化のリスクがある。
- UUIDv7 はタイムソート属性を持つため、連番の利点（時系列順）を継承しつつランダム性を確保する。

UUIDv7 は [RFC 9562](https://www.rfc-editor.org/rfc/rfc9562.html) 準拠の 128bit 識別子。先頭 48bit が Unix ミリ秒、続く bit がランダムで、自然にソート可能。

**設計項目 DS-SW-DB-007 タイムスタンプの UTC 統一**

全エンティティのタイムスタンプ属性は `timestamptz` 型（PostgreSQL）で UTC 保存する。アプリケーション側で JST / UTC 変換するが、DB には UTC のみ保存する。この方針はタイムゾーン混在による「23 時 59 分のイベントが次の日に記録される」バグを構造的に防ぐ。

命名規約は `<event>_at` 形式（`created_at` / `updated_at` / `deleted_at` / `published_at` / `occurred_at` など）。単純な `ts` や `date` は禁止。

**設計項目 DS-SW-DB-008 PII 区分の明示**

各エンティティ属性に PII 区分（N/A、低、中、高、マイナンバー）を付与する。User.email は「中」、User.display_name は「低」、AuditEvent.pii_masked_json は「N/A（マスク済み）」、Decision.input_hash は「N/A（ハッシュのみ）」といった区分を [04_データ辞書.md](04_データ辞書.md) で厳密に定義する。

PII 区分「中」以上の属性は PostgreSQL 側で pgcrypto 列暗号化、または MinIO 側で SSE-S3 暗号化を適用する。「マイナンバー」区分は全段階で取り扱わない方針（k1s0 はマイナンバーを保存しない）。

## エンティティライフサイクル

各エンティティのライフサイクル（作成・更新・削除・保管・廃棄）は [03_データ保持と廃棄方式.md](03_データ保持と廃棄方式.md) で詳細化する。本ファイルでは概略のみ触れる。

**設計項目 DS-SW-DB-009 論理削除と物理削除の分離**

Tenant / User / Application は論理削除（`deleted_at` 列）で開始し、法的保留解除後に物理削除する。State / Event は TTL で自動失効し、物理削除される。AuditEvent は WORM で削除不可、7 年経過後に MinIO 冷データへ移管する。

論理削除から物理削除までの期間は Tenant は 7 年、User は 3 年、Application は 1 年とし、保持期限中の参照は「削除済み」扱いとしつつ、監査要求には応答できる。
