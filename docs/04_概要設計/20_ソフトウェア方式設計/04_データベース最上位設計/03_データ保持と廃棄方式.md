# 03. データ保持と廃棄方式

本ファイルは tier1 が保持するデータのライフサイクル管理（保持期間・廃棄手順・バックアップからの削除・法的保留・DSAR 対応）を方式として固定化する。個人情報保護法 2022 改正、J-SOX、電帳法、GDPR（採用側のマルチクラスタ移行時 対応）の法令要件を満たしつつ、採用側の小規模運用原則と両立する自動化手順を設計する。

## 本ファイルの位置付け

データを「作る」設計は楽観的になりがちだが、「消す」設計を疎かにすると法的リスクと運用コストが膨れ上がる。個人情報は無期限に保管すると個人情報保護法の「必要な範囲を超える取得・利用の禁止」に違反する。監査ログは 7 年保持の法令要件があるため、早期削除すると法令違反となる。バックアップが無期限残存すると「削除したはずの個人情報がバックアップに残っている」リスクが発生する。

本ファイルはこれら保持と廃棄のポリシー衝突を整理し、データ種別ごとに「いつまで持ち、いつどう消すか」を決定する。決定は機械的に適用できる形で記述し、運用担当者の判断が入らない自動化を目指す。

## 保持期間ポリシー

データ種別ごとの保持期間を以下表で固定化する。期間の根拠は法令要件・業務要件・運用コストの 3 軸から導出する。

**設計項目 DS-SW-DB-060 データ種別別保持期間**

| データ種別 | 保持期間 | 根拠 | 物理ストア |
|----------|---------|------|----------|
| 監査ログ（AuditEvent） | 7 年 | 電帳法第 8 条、J-SOX 内部統制記録 7 年 | Postgres（24 か月ホット）→ MinIO（7 年コールド） |
| 個人情報（User、Application メタ） | 3 年 | 個人情報保護法、最終アクセスから 3 年 | Postgres |
| 操作ログ（Log API 出力） | 90 日 | 運用トラブルシュート期間、コスト最適 | Loki（90 日） |
| 分散トレース | 7 日 | デバッグ期間、ストレージ量が巨大 | Tempo |
| メトリクス | 13 か月 | 年次比較可能な最低期間 | Prometheus（30 日）→ Mimir（13 か月） |
| State（キャッシュ） | 24 時間既定、最大 7 日 | アプリ側 TTL 指定、Valkey メモリ圧 | Valkey |
| 冪等キー | 24 時間 | リトライ常識的ウィンドウ | Valkey |
| Event / Kafka | 3〜7 日 | at-least-once 配送の冗長期間 | Kafka |
| バックアップ（Postgres） | 90 日（PITR 30 日 + full 60 日） | 四半期 DR 訓練で最小必要 | MinIO |
| バックアップ（Valkey） | 7 日 | キャッシュ復旧用、短期で充分 | MinIO |
| Decision サンプル | 3 年 | 決定監査、年次 + 前年 + 前々年 | Postgres（24 か月）+ MinIO |
| Workflow 履歴 | 3 年 | 長期 Saga の後追い可能期間 | Postgres + Temporal DB |
| Feature Flag 履歴 | 無期限（論理） | 機能公開履歴の永続監査 | Postgres |

期間外データの扱いを明確にする必要がある。期間満了したデータは「読み取り可だが書き込み不可」「コールドストレージへ移管」「完全削除」の 3 状態いずれかに遷移する。遷移条件は各データ種別で個別定義する。

## 廃棄手順

廃棄は 3 段階で実行する。論理削除（deleted_at セット）→ 物理削除（DELETE）→ 暗号鍵破棄（crypto-shredding）。このカスケード方式により、誤削除時のロールバックと、正当削除時の復元不能性を両立する。

**設計項目 DS-SW-DB-061 論理削除ステップ**

第 1 段階は論理削除。`deleted_at` 列に削除時刻を記録し、以降の業務参照対象から外す。この時点ではテーブルからレコードは消えず、管理者のみがシステムテーブル経由で参照できる。

論理削除は RESTful な DELETE API で実行される標準操作。tier2 / tier3 からの削除要求は全て論理削除とし、物理削除は後段で実行する。論理削除後の期間は Tenant 7 年、User 3 年、Application 1 年、Secret 90 日とする。

**設計項目 DS-SW-DB-062 物理削除ステップ**

第 2 段階は物理削除。論理削除後の保持期間を経過したレコードを `DELETE` で物理的に除去する。Postgres では autovacuum で領域回収、MinIO では object delete、Valkey では `DEL` または TTL 期限切れで自然削除する。

物理削除は日次バッチで実行する。`DELETE FROM ... WHERE deleted_at < NOW() - INTERVAL '3 years'` のような SQL を pg_cron で定期実行する。実行ログは Audit-Pii に記録し、削除件数と対象 ID 範囲を事後確認可能にする。

**設計項目 DS-SW-DB-063 暗号鍵破棄（crypto-shredding）**

第 3 段階は暗号鍵破棄。pgcrypto で暗号化されている列、MinIO の SSE-S3 で暗号化されているオブジェクト、OpenBao 配下のシークレットは、対応する暗号鍵を破棄することで「データが物理的に残っても復号不能」な状態にする。

暗号鍵の破棄は「一度破棄したら復元不能」を成立させるため、OpenBao の Key Versioning で新鍵への rotation と旧鍵の `destroy` を組み合わせる。この手順は法的保留中のデータには適用せず、保留解除後に一括適用する。

## DSAR（Data Subject Access Request）対応

個人情報保護法 2022 改正は本人からの「開示・訂正・削除」請求を 30 日以内に処理することを求める（第 33〜36 条）。これは DSAR と呼ばれ、方式設計で応答手順を固定化する必要がある。

**設計項目 DS-SW-DB-064 開示請求の 30 日以内応答**

本人からの開示請求（自分の個人情報を出せ）は、プライバシー担当窓口 → SRE → tier1 管理コマンドで実行する。SRE は `k1s0-admin-cli dsar export --subject-id <user_id>` を実行し、User / AuditEvent / その他個人情報を含むテーブルから該当レコードを暗号化 ZIP で出力する。出力は 30 日以内（実効 15 営業日目安）に完了する。

出力フォーマットは JSON + CSV の 2 形式。暗号化は本人パスフレーズで生成された AES-256 鍵で行う。送付は 採用側組織の規程のセキュアチャネル（郵送の場合は内容証明郵便、電子の場合は S/MIME 暗号化メール）に従う。

**設計項目 DS-SW-DB-065 削除請求の 30 日以内実行**

本人からの削除請求は論理削除 → 物理削除 → 暗号鍵破棄の 3 段階を 30 日以内に完了する。削除対象は User 本体、AuditEvent の pii_masked_json フィールド、Decision の input_hash に関連する PII（tier1 内のハッシュのみ残す形で匿名化）、Workflow 履歴中の PII フィールド。

削除完了後、実行証跡を AuditEvent に `action=DSAR_DELETE, subject_id=<user_id>, executed_at=<time>` で記録する。この監査ログは削除対象となった User 本人の情報ではないため、他の AuditEvent と同じく 7 年保持する。

**設計項目 DS-SW-DB-066 訂正請求の実行**

訂正請求（誤った情報を正しく書き換えよ）は対応する API 呼び出し（`UpdateUser`）で処理する。訂正履歴は AuditEvent に旧値・新値を `diff` として記録し、訂正前の値も監査可能にする。訂正前の値そのものは Audit のハッシュ化により個人特定できないため、訂正処理の完了と見なす。

## 法的保留（Litigation Hold）

訴訟・行政調査・内部調査により「削除禁止」命令を受けたデータは、保持期間を超えても削除してはならない。法的保留の仕組みを事前整備する。

**設計項目 DS-SW-DB-067 保留メタデータの付与**

法的保留対象データは `legal_holds` テーブルで管理する。`hold_id` / `target_table` / `target_id_range` / `reason` / `imposed_at` / `expected_resolution_at` / `approver` を記録する。物理削除処理は日次バッチで `legal_holds` 参照し、保留対象を除外する。

保留の発令は 採用側組織の法務部門からの正式指示を受けた SecOps が実行する。二重承認（法務 + SecOps 管理者）を要求し、自動化スクリプト内ではハードコード承認を禁止する。

**設計項目 DS-SW-DB-068 保留解除と削除再開**

保留解除は法務部門からの解除通知後、`legal_holds.expected_resolution_at` を過去日に更新することで実行する。更新後の日次バッチで通常の物理削除・暗号鍵破棄フローへ合流する。

保留解除操作自体も AuditEvent に `action=LEGAL_HOLD_RELEASE, hold_id=..., released_by=...` で記録する。

## バックアップからの削除

個人情報を削除しても、バックアップに残存していると「実質的に削除されていない」状態となる。GDPR 17 条（right to be forgotten）対応では、バックアップからも削除する仕組みが要求される。

**設計項目 DS-SW-DB-069 バックアップの有限保管**

バックアップは無期限保管を禁止する。保持期間を明示し、期間満了で自動削除する。

- Postgres フルバックアップ: 90 日保持、91 日目に自動削除。
- Postgres インクリメンタル（WAL-G）: 30 日保持。
- Valkey RDB: 7 日保持。

期間内に DSAR 削除請求が入った場合、バックアップからの個別除去は困難（物理フォーマットの性質）のため、実施しない。代わりに「バックアップからの復旧時は削除済みデータを再削除」手順を Runbook に明示する。

**設計項目 DS-SW-DB-070 crypto-shredding によるバックアップ削除の実効**

バックアップ内の個人情報を実質的に削除する補完手段として、pgcrypto の列暗号化鍵の破棄を活用する。User.email は pgcrypto で暗号化されており、対応鍵を OpenBao 側で破棄すると、バックアップからリストアしても復号不能（暗号文のみ、実質削除）状態となる。

この方式は GDPR の「技術的に合理的な措置」要件を満たすと解釈できる。採用側のマルチクラスタ移行時（GDPR 本格対応）ではこの crypto-shredding を DSAR 削除フローに組み込み、バックアップからの個別削除の代替手段として位置付ける。

## GDPR 対応（採用側のマルチクラスタ移行時）

採用側のマルチクラスタ移行時 で国際展開する際、GDPR への対応が必要となる。リリース時点〜採用後の運用拡大時 は準備段階、採用側のマルチクラスタ移行時 で本番対応する。

**設計項目 DS-SW-DB-071 right to be forgotten 実装**

GDPR 17 条に基づく忘却権は、本人からの削除請求と同様に 30 日以内（GDPR は 1 か月）で実行する。対象データは個人情報保護法の削除請求と同等 + 「第三者提供先への通知」義務が追加される。

第三者提供先への通知は、tier1 が記録している連携先リスト（SAP / Salesforce / その他 Output Binding 先）に対して、削除要求を転送する。転送は自動化するが、受領確認は手動で行う（国際的に自動確認の標準プロトコルが未成熟なため）。

**設計項目 DS-SW-DB-072 国際データ移転の同意管理（採用側のマルチクラスタ移行時）**

GDPR 45〜50 条に基づく国際データ移転は、本人からの明示同意がある場合のみ許容する。tier1 は `consent_records` テーブル（採用側のマルチクラスタ移行時 で追加）で同意の取得時刻・撤回時刻・対象データカテゴリを記録する。

未同意のまま tier1 が国際移転を発動した場合（SAP Cloud 等への Output Binding で EU 域外サーバへ送信）、tier1 は自動で操作を拒否する。この検知は Decision API（ZEN Engine）のルール評価で行う。採用側のマルチクラスタ移行時 本格運用時に同意管理 UI を Backstage に実装する。

## 運用自動化とレビュー

保持・廃棄フローの大半は自動化するが、定期レビューは必須とする。自動化スクリプトの誤動作で「予期せず大量削除」のリスクを避ける。

**設計項目 DS-SW-DB-073 月次レビュー**

月次で以下を SRE + SecOps がレビューする。

- 物理削除実行ログ（削除件数、対象データ種別）
- DSAR 実行ログ（開示 / 削除 / 訂正 の件数、応答時間）
- 法的保留の発令・解除一覧
- バックアップ残量と自動削除の適切な実行

レビュー結果は四半期ごとに Product Council へ報告し、経営層の経営判断に利用可能な形で集約する。
