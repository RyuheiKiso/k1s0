# 04. データ辞書

本ファイルは [01_論理ERモデル.md](01_論理ERモデル.md) と [02_最上位テーブル一覧.md](02_最上位テーブル一覧.md) で参照される全エンティティ・全カラムの定義を一元管理する。カラム属性として物理名・論理名・型（PostgreSQL / Rust / Go）・長さ・NULL 可否・既定値・PII 区分・暗号化対象・意味を確定し、実装側で揺らぎが発生しないようにする。要件定義側 [../../../03_要件定義/00_要件定義方針/](../../../03_要件定義/00_要件定義方針/) のデータ辞書と連動し、概要設計では物理型・インデックス候補・暗号化対象の付加情報を上載せする。

## 本ファイルの位置付け

データ辞書は概要設計と詳細設計の境界に位置する「プロジェクト全体の共通参照」である。辞書がないまま設計を進めると、同じ `tenant_id` が A テーブルでは `uuid`、B テーブルでは `varchar(36)` で表現される、`created_at` の型が `timestamptz` と `timestamp` で混在する、といった不整合が蓄積する。これらは実装フェーズで検出されると全面修正が必要となる。

本ファイルはこの蓄積を上流で止める役割を担う。命名規約・型選択・PII 区分を属性単位で固定し、全チームが同一辞書を参照する状態を作る。辞書の更新は PR レビュー必須とし、自由追加は禁止する。

## 命名規約

命名規約は物理名・論理名・ID 名・タイムスタンプ名の 4 種で個別に定める。

**設計項目 DS-SW-DB-090 物理名の規約**

物理名（DB カラム名、Protobuf フィールド名）は `snake_case` 英小文字に統一する。例: `tenant_id`、`created_at`、`display_name`。

- 複数単語はアンダースコアで区切る。
- 英語のみ使用、略語は業界標準（id、url、http、json、uuid など）のみ許容、独自略語禁止。
- 予約語（`order`、`user`、`table` など）との衝突回避のため、テーブル名は複数形（`users`、`applications`）、カラム名は修飾（`display_name` など）を優先する。
- 長さは 30 文字以内を目安とし、超過時は短縮形を検討する。

**設計項目 DS-SW-DB-091 論理名の規約**

論理名（設計書での表記、Backstage カタログの表示）は日本語で統一する。例: 物理名 `tenant_id` に対する論理名「テナント ID」、物理名 `pii_masked_json` に対する論理名「PII マスク済み JSON」。

- 業界通称を優先し、独自造語を避ける。
- カタカナ英語（例: 「バージョン」「トークン」）は許容。
- 助詞は最小限（「の」「と」「から」のみ）。

**設計項目 DS-SW-DB-092 ID カラムの規約**

ID カラムは `<entity>_id` 形式で統一する。例: `tenant_id`、`user_id`、`app_id`、`audit_id`。

- 型は UUIDv7（128bit）、PostgreSQL では `uuid`、Rust では `uuid::Uuid`、Go では `github.com/google/uuid.UUID` または文字列。
- 自動採番（連番）は禁止（[01_論理ERモデル.md](01_論理ERモデル.md) DS-SW-DB-006）。
- 主キーは `<entity>_id` を基本とし、複合主キーは外部キー順（親 → 子）で並べる（例: `(tenant_id, app_id, key)`）。

**設計項目 DS-SW-DB-093 タイムスタンプカラムの規約**

タイムスタンプカラムは `<event>_at` 形式、型は `timestamptz`（UTC）で統一する。例: `created_at`、`updated_at`、`deleted_at`、`published_at`、`occurred_at`、`completed_at`。

- 複数形（`timestamps`）や抽象名（`ts`、`date`）は禁止。
- タイムゾーン情報を失う `timestamp`（`timestamp without time zone`）は禁止。アプリ層で JST 変換する場合も DB には UTC 保存。
- Rust では `chrono::DateTime<Utc>`、Go では `time.Time`（UTC 強制ラッパ型 `k1s0time.UTC`）で扱う。

## PII 区分

PII（Personally Identifiable Information）区分は属性単位で定義する。区分によって暗号化・マスキング・ログ出力制限が異なる。

**設計項目 DS-SW-DB-094 PII 5 区分の定義**

| 区分 | 定義 | 対応 | 例 |
|------|------|------|-----|
| N/A | 個人特定不能 | 制限なし | tenant_id、created_at、status |
| 低 | 組み合わせで個人特定の可能性 | ログ出力は hash 化 | display_name、department |
| 中 | 単体で個人特定可能、漏洩時影響中 | 列暗号化（pgcrypto）、ログ出力禁止 | email、phone_number、address |
| 高 | 機微個人情報（病歴、信条、犯罪歴等） | 列暗号化 + 追加アクセス制御 + 独立監査 | 健康診断結果、給与情報 |
| マイナンバー | マイナンバー法の特定個人情報 | k1s0 では保持しない | （該当なし） |

各区分への帰属判定は、属性単体での「個人特定可能性」と「漏洩時の影響度」の 2 軸で行う。`tenant_id` / `created_at` / `status` が N/A に該当するのは、これらが単独でも組み合わせでも個人を一意に特定できず（`tenant_id` は法人単位、`created_at` は発生時刻、`status` は列挙値）、個人情報保護法 2 条 1 項の「個人情報」に該当しないためである。`display_name` / `department` が低に該当するのは、単独では不特定多数（例: 「山田太郎」）だが `tenant_id` + `department` の組み合わせで社内特定に至る可能性があり、ログ出力時に hash 化することで組合せ攻撃を防止する必要があるためである。`email` / `phone_number` / `address` が中に該当するのは、メールアドレスや電話番号は個人情報保護法で明確に個人情報とされ、単体で特定の個人に到達可能であり、漏洩時はなりすまし・フィッシングの直接原因になるためである。高区分は個人情報保護法 2 条 3 項「要配慮個人情報」（健康診断結果・信条・犯罪歴等）に対応し、漏洩時の影響が不可逆（差別・不利益処分）であるため列暗号化に加え独立監査ログを要求する。

「マイナンバー」区分は k1s0 の全 Phase で取り扱わない方針。取り扱う必要が生じた場合は専用システム（別物理基盤）を別途構築する。

**設計項目 DS-SW-DB-095 PII 自動マスキング**

Log API / Audit-Pii API は PII 区分「低」以上の値を自動マスキングする。具体的には `email` → `a***@e***.com` 形式、`phone_number` → `090-****-****` 形式、`address` → `東京都***` 形式。マスキングは Rust の `custom-log` / `custom-audit` Pod 内で確定的に実装する。

マスキングルールは [../../30_共通機能方式設計/](../../30_共通機能方式設計/) のログ / 監査方式で詳細化する。本ファイルは「どの属性がマスク対象か」の指定に留める。

## 要件定義側データ辞書との整合ルール

要件定義側 [../../../03_要件定義/00_要件定義方針/](../../../03_要件定義/00_要件定義方針/) にもデータ辞書がある。両者の関係を明示し、差分が生じないよう運用する。

**設計項目 DS-SW-DB-107 要件定義データ辞書との同期**

要件定義側データ辞書は「業務視点での属性定義」、本ファイルは「実装視点での物理型と制約付加」。両者は名前空間を共有する（同じ物理名を指す）が、記述粒度が異なる。

同期ルールは以下。

1. 物理名・論理名・PII 区分は要件定義側が正、本ファイルが要件定義側と一致させる。不一致検出は CI で機械チェック。
2. PostgreSQL 物理型・Rust 型・Go 型は本ファイルが正、詳細設計が本ファイルと一致させる。
3. 新規エンティティ追加は要件定義側が先行、その後本ファイルが追随する。
4. 変更は PR レビュー必須。データ辞書の破壊的変更（型変更・PII 区分引き上げ）は Product Council 承認を要する。
