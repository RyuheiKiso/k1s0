# 02. 最上位テーブル一覧

本ファイルは [01_論理ERモデル.md](01_論理ERモデル.md) で確立した 11 エンティティを、物理ストア（PostgreSQL、Valkey、Kafka、MinIO、OpenBao）に配置するテーブル（相当物）として最上位で固定化する。各テーブルの主キー・主要インデックス・暗号化対象・RLS 有無・パーティション方針・バックアップ頻度を概要設計視点で確定する。詳細カラム定義は [04_データ辞書.md](04_データ辞書.md) に委ねる。

## 本ファイルの位置付け

論理 ER モデルは「何のデータが存在するか」を示すが、物理ストア配置が決まらないと性能・可用性・バックアップ方式が決定できない。k1s0 では 5 種の異なるストアにデータが分散し、各ストアで「テーブル」に相当する概念が異なる（Postgres は table、Valkey は key-space、Kafka は topic、MinIO は bucket、OpenBao は path）。本ファイルはこの差分を吸収し、論理的な「最上位テーブル一覧」として統一的に列挙する。

テーブル構造は性能 SLO と強く結びつく。State API の p99 10ms は Valkey の単一キー参照で達成する前提であり、PostgreSQL に置くと 20ms〜50ms へ伸びる。AuditEvent の WORM と 7 年保持は Postgres のパーティション + MinIO 冷データ移管の組み合わせでのみ実現可能。本ファイルはこれら物理判断の根拠を同時に記述する。

## ストア別テーブル配置の全体像

5 種ストアへの配置方針は以下のとおり。PostgreSQL は「長期 Stateful、ACID が必要なトランザクション」、Valkey は「低レイテンシ・短命キャッシュ・冪等キー」、Kafka は「イベントログ・非同期配送」、MinIO は「オブジェクト・冷データ・長期アーカイブ」、OpenBao は「秘密（パスワード、APIキー、証明書）」に責任を分ける。

**設計項目 DS-SW-DB-030 ストア別責務分離**

| ストア | 責務 | 採用根拠 | 配置される論理エンティティ |
|-------|------|---------|-------------------|
| PostgreSQL（CloudNativePG） | ACID トランザクション、長期保持 | JDBC 互換、pgcrypto、パーティション、PITR | Tenant / User / Application / AuditEvent / FeatureFlag / WorkflowExecution メタ / Decision サンプル / BindingConfig / Secret リース |
| Valkey Cluster | 低レイテンシ K/V、TTL 管理 | p99 10ms、Redis プロトコル互換 | State（主）、冪等キー、Rate Limit カウンタ |
| Kafka（Strimzi、KRaft） | イベントログ、at-least-once | at-least-once、DLQ、Consumer Group | Event（実体）、Audit Stream、Decision Stream |
| MinIO | オブジェクト、冷データ | S3 API 互換、Erasure Coding | AuditEvent 冷データ（7 年）、WorkflowExecution 長期履歴、Decision サンプル冷データ |
| OpenBao | 秘密 | Vault 互換、動的シークレット | Secret 実体（リースは Postgres） |

配置ルールは「ACID が要る → PG」「TTL が効く低レイテンシ → Valkey」「順序保証のある非同期 → Kafka」「容量と安さ重視 → MinIO」「機密 → OpenBao」のフローで決定する。同一論理エンティティが複数ストアに跨ることを許容する（例: AuditEvent は Postgres と MinIO）。

## PostgreSQL テーブル一覧

PostgreSQL 上に配置する 11 テーブルを列挙する。各テーブルは tier1 用のスキーマ `k1s0_platform` に配置し、tier2 業務テーブルは別スキーマ（テナントごとに `tenant_<uuid>`）で分離する。

**設計項目 DS-SW-DB-031 Postgres テーブル一覧**

| テーブル | 主キー | 主要 INDEX | 暗号化 | RLS | パーティション | 備考 |
|---------|-------|-----------|--------|-----|-------------|------|
| `tenants` | `tenant_id` (UUIDv7) | `name` UNIQUE | なし | 無 | なし | テナント数 <10 万想定 |
| `users` | `user_id` (UUIDv7) | `(tenant_id, subject)` UNIQUE | `email` pgcrypto | 有（tenant_id） | なし | PII 中 |
| `applications` | `app_id` (UUIDv7) | `(tenant_id, name)` UNIQUE | なし | 有 | なし | |
| `audit_events` | `audit_id` (UUIDv7) | `(tenant_id, occurred_at)`、`user_id` | なし（ハッシュ化済み） | 有 | 月次（RANGE）、pg_partman | 7 年保持、WORM |
| `feature_flags` | `flag_id` (UUIDv7) | `(tenant_id, key)` UNIQUE | なし | 有 | なし | flagd 同期元 |
| `workflow_executions` | `workflow_id` (UUIDv7) | `(tenant_id, app_id, status)` | なし | 有 | なし（Phase 2 で月次検討） | Temporal DB と併用 |
| `secret_leases` | `lease_id` (UUIDv7) | `(tenant_id, secret_name)` | なし（値は OpenBao） | 有 | なし | リース期限 TTL |
| `decisions_cache` | `decision_id` (UUIDv7) | `(tenant_id, rule_name, input_hash)` | なし | 有 | 月次（RANGE） | サンプリング 1% |
| `binding_configs` | `binding_id` (UUIDv7) | `(tenant_id, app_id)` | `endpoint_ref` pgcrypto | 有 | なし | |
| `state_durable` | `(tenant_id, app_id, key)` | `updated_at`、`ttl_expires_at` | なし | 有 | なし | Valkey 損失時のバックアップ |
| `events_meta` | `event_id` (UUIDv7) | `(tenant_id, topic, published_at)` | なし | 有 | 月次 | Kafka オフセット参照、90 日保持 |

RLS（Row Level Security）は全テーブルで tenant_id を条件に設定する。`CREATE POLICY ... USING (tenant_id = current_setting('k1s0.current_tenant_id')::uuid)` 方式。tier1 アプリケーションは接続時に `SET k1s0.current_tenant_id = '...'` を実行し、以降の SQL は自動的にテナント分離される。

パーティション有無の判断は「年間行数」と「保持期間」で決定する。`audit_events` と `events_meta` と `decisions_cache` の 3 テーブルのみ月次 RANGE パーティションを導入し、残りはパーティション非採用とする。`decisions_cache` は 1% サンプリングでも Phase 2 想定 500 RPS × 0.01 × 86,400 秒 × 30 日 = 約 1,300 万/月の行数となり、`evaluated_at` 基準の月次分割が B-tree index の膨張を抑える。`events_meta` は 90 日保持で Kafka topic 総流量の参照メタのみだが、Phase 1b で日次 500 万行（50 RPS × 86,400 秒）を超える想定のため月次 RANGE で分割する。一方 `tenants` は 10 万行以下、`feature_flags_state` は flagd 同期用で数万行規模、`applications` と `secret_leases` と `binding_configs` は tier2 アプリ数と連動し 10 万行未満、`workflow_executions` は Phase 1b で年間 3,000 万行を見込むが sticky session の高速参照が要件で Temporal 側 DB と併用のため Phase 2 で月次検討に据え置く。`state_durable` は Valkey 損失時のバックアップ用途で TTL 24 時間の短命データのため、パーティション不要かつ VACUUM で十分管理できる。

**設計項目 DS-SW-DB-032 audit_events の月次パーティション**

AuditEvent は年間数億レコードの流入が想定される（1 テナント日次 10 万、100 テナントで 1,000 万/日 = 36 億/年）。単一テーブルでは `INSERT` 時の index 更新コストと `SELECT` 時のスキャンコストが破綻するため、月次 RANGE パーティション（`occurred_at` 基準）で分割する。

パーティション管理は `pg_partman` 拡張で自動化する。前月分は read-only 化、24 か月経過後は MinIO への冷データ移管 + パーティション DROP、7 年経過後は MinIO 側も削除（法的保留解除要件）。詳細は [03_データ保持と廃棄方式.md](03_データ保持と廃棄方式.md) で展開する。

**設計項目 DS-SW-DB-033 主要テーブルの個別仕様**

主要 4 テーブルの仕様を具体化する。

- **tenants**: 行数 < 10 万。全列 in-memory キャッシュ可能。`status` 列で `active` / `suspended` / `deleted` を管理、suspended 時は全 API が 403 を返す。
- **users**: 行数 < 1,000 万。`email` は pgcrypto の pgp_sym_encrypt で暗号化、復号鍵は OpenBao で管理。検索は `email_hash` 列（SHA-256）で完全一致のみサポート。
- **audit_events**: 行数 数億〜数十億。`hash_chain` 列は前レコードの `prev_hash` と当レコード内容の SHA-256 連結。書き込みは Rust の custom-audit Pod から単一 Stream として発行、パーティション並列書き込みを避ける。
- **workflow_executions**: 行数 < 1 億。`status` 列で `running` / `completed` / `failed` / `compensated` を管理、running の一覧は 24 時間以内のみ index 利用で高速化。Temporal の DB と別に tier1 側でも軽量メタを保持。

## Valkey キー構造一覧

Valkey は table 概念がなく key space で管理する。tier1 内部の命名規約で「論理的テーブル相当」を表現する。

**設計項目 DS-SW-DB-034 Valkey キー namespace**

| namespace | キー形式 | 用途 | TTL | 備考 |
|-----------|---------|------|-----|------|
| state | `{<tenant_id>}:<app_id>:<key>` | State API | アプリ指定（既定 24h） | hashtag で同テナント同スロット |
| idem | `{<tenant_id>}:idem:<api>:<key>` | 冪等キー | 24h | アプリ呼び出しの重複排除 |
| idem_edge | `{<tenant_id>}:idem:edge:<event_id>` | エッジ事象重複排除 | 7 日 | |
| ratelimit | `rl:<scope>:<id>:<window>` | Rate Limit カウンタ | 1 秒 / 1 分 | scope = ip / tenant / api |
| session | `{<tenant_id>}:sess:<jti>` | JWT セッション拡張情報 | 15 分 | Phase 2 で SSO 拡張 |
| feature | `ff:<tenant_id>:<key>` | flagd キャッシュ | 60 秒 | flagd fallback |
| lock | `{<tenant_id>}:lock:<resource>` | 分散ロック（SET NX） | 30 秒 | 短期排他 |

hashtag `{<tenant_id>}` を使うことで、同一テナントのキーが同一 Redis Cluster スロットに配置され、MULTI/EXEC トランザクションが可能になる。テナント間のトランザクションは不要な設計（RLS と整合）とする。

## Kafka トピック一覧

Kafka トピックは「論理テーブル相当」として扱える永続イベントストアである。命名規約で tier1 内部用と tier2 / tier3 公開用を分離する。

**設計項目 DS-SW-DB-035 Kafka トピック命名**

| トピック | 生産者 | 消費者 | パーティション数 | 保持期間 | 備考 |
|---------|-------|-------|----------------|---------|------|
| `k1s0.audit.v1` | tier1 Audit-Pii | Audit-Pii Consumer → Postgres | 16 | 7 日（バッファ） | Postgres 書き込み確認後のみ ack |
| `k1s0.decision.v1` | tier1 Decision | Decision Sampler → Postgres / MinIO | 8 | 7 日 | 1% サンプリング |
| `k1s0.telemetry.v1` | tier1 Telemetry | OTel Collector | 16 | 1 日 | 非同期中継 |
| `k1s0.log.v1` | tier1 Log | Loki Ingester | 16 | 1 日 | 非同期中継 |
| `edge-events` | Dapr Binding Input（MQTT 経由） | tier2 / tier3 Consumer | 32 | 7 日 | エッジ機器事象 |
| `legacy.<system>.<event>` | 既存資産（SAP / SFDC） | tier2 Consumer | テナント毎に可変 | 7 日 | レガシー連携 |
| `<tenant>.<app>.<event>` | tier2 / tier3 PubSub Publish | tier2 / tier3 Subscribe | 動的 | 3 日 | 業務公開トピック |
| `<topic>.dlq` | Broker | SRE | 同元トピック | 30 日 | DLQ（3 回失敗で転送） |

トピック命名ポリシーは PubSub API 側で検証し、不正命名の Publish は 400 エラーで拒否する。保持期間は業務保持期間より短く、長期保持は Postgres / MinIO で別途行う設計とする。

## MinIO バケット一覧

MinIO は冷データ・オブジェクト・ログアーカイブ用途で利用する。バケット設計は用途ごとに明確に分離する。

**設計項目 DS-SW-DB-036 MinIO バケット**

| バケット | 用途 | 暗号化 | ライフサイクル |
|---------|------|--------|-------------|
| `k1s0-audit-cold` | AuditEvent 24 か月以降の冷データ | SSE-S3（KMS 連動） | 7 年で削除 |
| `k1s0-decision-cold` | Decision サンプルの 24 か月以降 | SSE-S3 | 3 年で削除 |
| `k1s0-workflow-archive` | WorkflowExecution 完了後 90 日以降 | SSE-S3 | 3 年で削除 |
| `k1s0-backup-pg` | PostgreSQL の WAL-G バックアップ | SSE-S3 + KMS | 日次 full + 4h incr、PITR 30 日 |
| `k1s0-backup-valkey` | Valkey の RDB スナップショット | SSE-S3 | 日次、7 日保持 |
| `k1s0-tier2-objects` | tier2 業務オブジェクト（テナント別プレフィクス） | SSE-S3 | テナント要件 |

バケット間のアクセス制御は IAM Policy で厳密化する。tier1 アプリは `k1s0-audit-cold` への書き込み専用、SRE は読み取り専用、WORM 属性で削除不可とする。

## OpenBao パス一覧

OpenBao は path 階層でシークレットを管理する。tier1 用とテナント用を分離する。

**設計項目 DS-SW-DB-037 OpenBao パス**

| パス | 用途 | 認証方式 |
|------|------|---------|
| `tier1/db/postgres-cnpg` | Postgres 動的認証情報 | AppRole（tier1 Pod のみ） |
| `tier1/db/valkey` | Valkey 接続情報 | AppRole |
| `tier1/kafka/saslscram` | Kafka SASL/SCRAM 認証情報 | AppRole |
| `tier1/certs/internal-ca` | 内部 CA 証明書発行 | AppRole（cert-manager 連携） |
| `tenant/<tenant_id>/custom/<name>` | テナント固有シークレット | User（JWT）+ AppRole（Pod） |

シークレットのリース情報（期限、発行履歴）は Postgres の `secret_leases` テーブルで管理し、実値は OpenBao に保持する。この 2 箇所分離により、リース管理ロジックを SQL で柔軟に操作しつつ、実値は OpenBao の強固な暗号化で保護する。

## バックアップとリカバリ

Phase 1b 時点での バックアップ頻度と RPO / RTO 目標を固定化する。

**設計項目 DS-SW-DB-038 バックアップ頻度と保持**

| ストア | 頻度 | 保持 | RPO | RTO |
|-------|------|------|-----|-----|
| PostgreSQL | WAL-G で継続的、日次 full、4h incr | PITR 30 日、full 90 日 | 5 分以内 | 1 時間 |
| Valkey | RDB 日次スナップショット | 7 日 | 24 時間（キャッシュ、復旧は冷データから） | 15 分（Cluster 再構築） |
| Kafka | MirrorMaker 2 で隣接 DC（Phase 2）| 30 日 | 秒オーダー | 2 時間 |
| MinIO | Erasure Coding（EC:6+3）で自己冗長 | 永続 | 0（WORM） | 30 分 |
| OpenBao | Raft ストレージで自己冗長、snap 日次 | 30 日 | 24 時間 | 1 時間 |

RPO / RTO は tier1 API p99 500ms / RTO 4 時間の全体 SLO から逆算した数値。Valkey の RPO 24 時間は「Valkey はキャッシュであり、State は Postgres にもバックアップされる」前提から導出する。
