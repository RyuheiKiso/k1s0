# 多言語ライブラリ提供方針

system tier が提供する共通ライブラリの多言語（Go / Rust / TypeScript / Dart）提供方針を定義する。全ライブラリが必ずしも全言語で実装される必要はなく、用途と利用者に応じた段階的な提供戦略を採用する。

---

## Tier 定義

ライブラリの多言語提供範囲を 3 段階の Tier で管理する。

| Tier | 対象言語 | 判定基準 | 例 |
|------|---------|---------|-----|
| **Tier A** | Go / Rust / TypeScript / Dart（全4言語必須） | サーバー・クライアント両方で利用され、全アプリケーションが依存する基盤ライブラリ | config, auth, telemetry, health |
| **Tier B** | Go / Rust 必須（TypeScript / Dart は実需次第） | 主にサーバーサイドで利用されるが、クライアントでも利用の可能性があるライブラリ | kafka, outbox, migration |
| **Tier C** | 実需駆動（必要になった言語から実装） | 特定のユースケースや特定言語の設計パターンに強く依存するライブラリ | bb-pubsub, bb-statestore, pubsub (Go) |

### Tier 判定フローチャート

```
ライブラリの利用者は？
├─ 全サーバー＋全クライアント → Tier A
├─ 全サーバー（クライアントは限定的） → 利用頻度は？
│   ├─ 高頻度（80%以上のサーバーが利用） → Tier A
│   └─ 中〜低頻度 → Tier B
└─ 特定サーバー／特定言語のみ → Tier C
```

### Tier C: 実需駆動（言語固有ライブラリ）

特定言語のエコシステムや設計パターンに依存するライブラリ。必要になった言語でのみ実装する。

| ライブラリ | 用途 | Go | Rust | TS | Dart |
|-----------|------|:--:|:----:|:--:|:----:|
| server-common | サーバー基盤（言語ごとに異なる実装） | ○ | ○ | ○ | ○ |
| codegen | コード生成・スキャフォールド | ○ | ○ | ○ | ○ |
| bb-ai-client | AI Gateway クライアント | ○ | ○ | ○ | ○ |
| pubsub (Go) / bb-pubsub (Rust) | PubSub Building Block 抽象化 | ○ | ○ | − | − |
| statestore (Go) / bb-statestore (Rust) | StateStore Building Block 抽象化 | ○ | ○ | − | − |
| secret-store (Go) / bb-secretstore (Rust) | SecretStore Building Block 抽象化 | ○ | ○ | − | − |
| binding (Go) / bb-binding (Rust) | Binding Building Block 抽象化 | ○ | ○ | − | − |
| bb-core (Rust) | BB 共通トレイト（Rust 専用） | − | ○ | − | − |
| telemetry-macros | トレーシングマクロ / ラッパー | ○ | ○ | − | − |

> **注記**: server-common / codegen / bb-ai-client は全4言語に存在するが、言語ごとに実装内容が大きく異なるため Tier C に分類する。TypeScript / Dart 版は building-blocks パッケージに BB 機能が統合されている。

## 新規ライブラリ追加ルール

### 1. 設計 Doc 先行

新規ライブラリの実装前に、必ず設計ドキュメントを作成する。

```
1. `docs/libraries/{category}/{lib-name}.md` に設計書を作成
2. 設計レビューで承認を得る
3. 実装を開始する
```

設計書には以下を含めること:

- **目的と背景**: なぜこのライブラリが必要か
- **公開 API**: トレイト / インターフェースの定義
- **Tier 判定結果**: どの Tier に分類されるか、その理由
- **依存関係**: 他ライブラリとの関係
- **テスト方針**: カバレッジ目標・モック戦略

### 2. Tier 判定プロセス

新規ライブラリの Tier は以下のプロセスで判定する。

| ステップ | 内容 | 判定者 |
|---------|------|--------|
| 1 | 利用者の範囲を特定（全サーバー？クライアント？特定サービス？） | 設計者 |
| 2 | Tier 判定フローチャートに従って仮 Tier を決定 | 設計者 |
| 3 | 設計レビューで Tier を確定 | レビュアー |
| 4 | `docs/libraries/_common/概要.md` の一覧に追加 | 設計者 |

### 3. 最小実装の定義

各 Tier における最小実装（MVP）の定義。

| Tier | 最小実装の範囲 |
|------|--------------|
| Tier A | 全4言語で公開 API + ユニットテスト + 統合テスト |
| Tier B | Go + Rust で公開 API + ユニットテスト + 統合テスト。TS / Dart は型定義のみ（スタブ可） |
| Tier C | 必要な言語のみで公開 API + ユニットテスト |

### 4. 実装順序

新規ライブラリの言語別実装順序。

```
1. Rust（型安全性を最初に確立し、API 設計の基準とする）
2. Go（サーバーサイドの主要言語として早期に実装）
3. TypeScript（Tier A の場合のみ。Web クライアント向け）
4. Dart（Tier A の場合のみ。Flutter クライアント向け）
```

## 破壊的変更の互換性ポリシー

### セマンティックバージョニング

全ライブラリはセマンティックバージョニング（SemVer）に従う。モノリポ内のパス依存であっても、論理的なバージョン管理を行う。

| バージョン変更 | 内容 | 例 |
|--------------|------|-----|
| **MAJOR** (x.0.0) | 破壊的変更・公開 API の削除・型変更 | トレイトメソッドのシグネチャ変更 |
| **MINOR** (0.x.0) | 後方互換な機能追加 | 新メソッドの追加・新オプションの追加 |
| **PATCH** (0.0.x) | バグ修正・パフォーマンス改善 | ロジック修正・依存更新 |

### 非推奨（Deprecation）プロセス

破壊的変更を導入する際は、以下のプロセスに従う。

```
1. 非推奨マーク: 旧 API に deprecated アノテーションを付与
   - Rust: #[deprecated(since = "x.y.z", note = "...")]
   - Go: // Deprecated: ...
   - TypeScript: @deprecated JSDoc
   - Dart: @Deprecated('...')

2. 移行ガイド: 新 API への移行方法をドキュメント化

3. 移行期間: 最低 2 マイナーバージョン（または 3 ヶ月）の並行期間

4. 削除: 移行期間終了後、次のメジャーバージョンで旧 API を削除
```

### 移行期間の詳細

| Tier | 最低移行期間 | 理由 |
|------|------------|------|
| Tier A | 3 ヶ月 または 2 マイナーバージョン | 全アプリケーションが依存するため、十分な移行時間が必要 |
| Tier B | 2 ヶ月 または 1 マイナーバージョン | サーバーサイド中心のため、比較的迅速な移行が可能 |
| Tier C | 1 ヶ月 | 利用者が限定的なため、短期間での移行が可能 |

### 全言語同時更新の原則

破壊的変更は全対象言語で同時にリリースする。特定言語のみ先行して破壊的変更を適用しない。

```
NG: Rust だけ v2.0.0 にして Go は v1.x のまま
OK: Go / Rust / TypeScript / Dart を同時に v2.0.0 にリリース
```

## 言語固有ライブラリの扱い

### `bb-` プレフィックス命名規則

Rust の Building Block 系ライブラリは `bb-` プレフィックスを付与する。これは Go 版と名前が異なる場合（Go: `pubsub` → Rust: `bb-pubsub`）に、同一概念の言語別実装であることを明示するためである。

| 命名パターン | 使用場面 | 例 |
|-------------|---------|-----|
| `bb-{name}` | Rust の Building Block 抽象化ライブラリ | bb-pubsub, bb-statestore, bb-secretstore, bb-binding |
| `bb-core` | Rust BB 共通トレイト（Component 等） | bb-core |
| `bb-ai-client` | `bb-` が機能名の一部（全言語共通） | bb-ai-client（Go / Rust / TS / Dart 共通） |
| `{name}` | Go 版の Building Block ライブラリ | pubsub, statestore, secret-store, binding |

### 言語固有ライブラリの必要性判定基準

以下の条件のいずれかを満たす場合、言語固有ライブラリとして Tier C に分類する。

| 基準 | 説明 | 例 |
|------|------|-----|
| **言語エコシステム依存** | 特定言語のフレームワーク・ツールチェインに強く依存する | server-common（Rust: axum 統合, Go: HTTP+gRPC 統合） |
| **言語機能依存** | マクロ・ジェネリクス等の言語固有機能を活用する | telemetry-macros（Rust: proc_macro, Go: 関数ラッパー） |
| **抽象化レベルの差異** | 言語ごとに公開 API の粒度や設計が大きく異なる | codegen（Go: 全機能, TS/Dart: 基本構造のみ） |
| **利用者の限定** | 特定言語のサーバー／クライアントのみが利用する | bb-core（Rust サーバー専用） |

### 言語固有ライブラリの追加手順

```
1. 既存の全言語共通ライブラリで代替できないか検討する
2. 言語固有の必要性を設計書に明記する
3. 命名規則に従い、言語固有であることを明示する
   - Rust BB: bb-{name}
   - Go 固有: そのまま（例: pubsub）
4. 概要.md の「言語固有ライブラリ」セクションに追記する
```
