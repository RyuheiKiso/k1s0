# system-library 共通実装パターン

system tier の全ライブラリで共通する実装パターンを定義する。各ライブラリ設計書ではサービス固有部分のみを記載し、共通パターンは本ドキュメントを参照する。

---

## 言語別配置パス {#言語別配置パス}

| 言語 | 配置パス |
| --- | --- |
| Go | `regions/system/library/go/{lib-name}/` |
| Rust | `regions/system/library/rust/{lib-name}/` |
| TypeScript | `regions/system/library/typescript/{lib-name}/` |
| Dart | `regions/system/library/dart/{lib-name}/` |

---

## 言語別定型ディレクトリ構成 {#定型ディレクトリ構成}

### Go

```
{lib-name}/
├── {module}.go
├── {module}_test.go
├── go.mod
└── go.sum
```

### Rust

```
{lib-name}/
├── src/
│   ├── lib.rs          # 公開 API（再エクスポート）
│   ├── {module}.rs     # 各機能モジュール
│   └── error.rs        # エラー型
├── Cargo.toml
└── README.md
```

### TypeScript

```
{lib-name}/
├── src/
│   ├── index.ts        # 公開 API
│   └── {module}.ts     # 各機能モジュール
├── tests/
│   └── {module}.test.ts
├── package.json
└── tsconfig.json
```

### Dart

```
{lib-name}/
├── lib/
│   ├── {lib_name}.dart # 公開 API
│   └── src/
│       └── {module}.dart
├── test/
│   └── {module}_test.dart
└── pubspec.yaml
```

## Cargo.toml 依存追加方法 {#cargo依存追加}

サーバーの `Cargo.toml` にライブラリ依存を追加する際の標準パターン:

```toml
# パス依存（モノリポ内）
k1s0-{lib-name} = { path = "../../system/library/rust/{lib-name}" }

# feature 有効化（テスト用モック等）
k1s0-{lib-name} = { path = "../../system/library/rust/{lib-name}", features = ["mock"] }
```

## 技術負債: HTTP クライアント統合 {#http-client-統合}

### 現状の重複

Go ライブラリ内で `http.Client` の生成が各ライブラリに個別実装されており、以下の 17 箇所で重複が発生している:

| ライブラリ | ファイル | タイムアウト |
| --- | --- | --- |
| saga | `client.go` | 30s |
| dlq-client | `client.go` | 30s |
| auth | `device_flow.go` | 30s |
| serviceauth | `client.go` | 30s |
| schemaregistry | `client.go` | 30s |
| tenant-client | `tenantclient.go` | 30s |
| vault-client | `vault_client.go` | 30s |
| graphql-client | `graphql_client.go` | 30s |
| search-client | `grpc_search_client.go` | 30s |
| scheduler-client | `grpc_client.go` | 30s |
| ratelimit-client | `ratelimit_client.go` | 30s |
| bb-ai-client | `http_client.go` | 設定可能 |
| quota-client | `http_client.go` | 設定可能 |
| file-client | `file_client.go` | 設定可能 |
| app-updater | `app_updater.go` | 設定可能 |
| webhook-client | `webhookclient.go` | 設定可能 |
| binding / building-blocks | `binding.go` / `http_binding.go` | 30s |
| health | `http_health_check.go` | 5s |

### 問題点

- `&http.Client{Timeout: 30 * time.Second}` のボイラープレートが 13 箇所以上に散在
- タイムアウト値・リトライ・トレース注入などの横断的関心事を一括変更できない
- テスト用コンストラクタ (`NewXxxWithHTTPClient`) も各ライブラリで個別に定義

### 統合方針

1. **共通 HTTP クライアントファクトリの導入**
   - `regions/system/library/go/httputil/` に共通パッケージを新設
   - デフォルトタイムアウト、リトライ、OpenTelemetry トレース注入を一元管理
2. **推奨パターン**
   ```go
   // httputil パッケージから標準クライアントを取得する
   client := httputil.NewClient(httputil.WithTimeout(30 * time.Second))
   ```
3. **移行手順**
   - 新規ライブラリは `httputil` を使用すること
   - 既存ライブラリは段階的に移行（破壊的変更を避ける）
4. **優先度**: Medium（機能的な問題はないが、保守性に影響）

## Dart: dynamic 型の使用に関する設計方針（MED-014）

### 背景

外部技術監査（2026-04-01 MED-014）において、Dart クライアントライブラリが `dynamic` 型を広範に使用していることが指摘された。
一般的に `dynamic` は型安全性を損なうため避けるべきとされているが、本プロジェクトでは意図的に使用している。

### 設計意図

Dart クライアントライブラリでは HTTP クライアントとして [Dio](https://pub.dev/packages/dio) を使用している。
Dio の `Response` 型は `Response<dynamic>` であり、JSON レスポンスのボディ型が静的に確定しない。
これは Dio ライブラリの JSON パース慣例に起因するものである。

```dart
// Dio の Response<dynamic> からデータを取り出す標準的なパターン
final Response<dynamic> response = await _client.get('/endpoint');
final Map<String, dynamic> body = response.data as Map<String, dynamic>;
```

### 許容される使用パターン

以下の場合に限り `dynamic` の使用を許容する:

| パターン | 理由 |
|---------|------|
| `Response<dynamic>` の保持 | Dio の API 仕様に準拠 |
| `Map<String, dynamic>` の JSON ボディ | Dart の JSON デシリアライズの標準パターン |
| `jsonDecode()` の戻り値 | Dart 標準ライブラリが `dynamic` を返す |

### 禁止パターン

以下の場合は `dynamic` を使用してはならない:

- ビジネスロジック内のデータ型（必ず具体的な型または `Object?` を使用する）
- 公開 API のパラメータ・戻り値の型（型安全な型を使用する）
- `dynamic` のままリストや Map に格納して後続処理に渡す（早期にキャストする）

### 推奨パターン

```dart
// 良い例: Dio 取得後すぐに型安全な型へキャストする
Future<UserProfile> fetchUser(String userId) async {
  final Response<dynamic> response = await _client.get('/users/$userId');
  // dynamic はここでキャストし、以降は型安全に扱う
  final Map<String, dynamic> json = response.data as Map<String, dynamic>;
  return UserProfile.fromJson(json);
}

// 悪い例: dynamic を後続処理に伝播させる
Future<dynamic> fetchUser(String userId) async {
  final response = await _client.get('/users/$userId');
  return response.data; // dynamic のまま返してはならない
}
```
