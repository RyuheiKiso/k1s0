## EventMetadata

全イベントに付与される共通メタデータ。`api/proto/k1s0/system/common/v1/event_metadata.proto` で定義。

| フィールド | 型 | 説明 |
| --- | --- | --- |
| event_id | string | イベント一意識別子（UUID v4） |
| event_type | string | イベント種別（`event_types.proto` の enum に対応） |
| aggregate_id | string | 集約ルートID |
| aggregate_type | string | 集約種別 |
| version | int64 | イベントバージョン番号 |
| timestamp | google.protobuf.Timestamp | イベント発生時刻 |
| correlation_id | string | リクエスト相関ID（リクエストチェーン追跡用） |
| causation_id | string | 原因イベントID（このイベントを引き起こしたイベントのID） |

### correlation_id と causation_id の違い

- **correlation_id**: 最初のリクエストから最後のレスポンスまで、一連の処理チェーン全体で同じ値を共有する。デバッグ・トレーシングに使用。
- **causation_id**: 直接の原因となったイベントのIDを格納する。イベント因果関係の追跡に使用。あるイベントが別のイベントをトリガーした場合、子イベントの `causation_id` に親イベントの `event_id` が入る。

## Outbox パターン

![Outbox パターンフロー](images/outbox-pattern-flow.svg)

トランザクショナルアウトボックスパターンにより、DBトランザクションとイベント発行の整合性を保証する。

### idempotency_key

`OutboxMessage` には `idempotency_key` フィールドが含まれる。INSERT 時に `ON CONFLICT (idempotency_key) DO NOTHING` を使用し、重複メッセージの発行を防止する。

- デフォルト: UUID v4 から自動生成
- 明示指定: `with_idempotency_key()` メソッドで業務キーを設定可能

### 並列処理

`OutboxProcessor` は `tokio::task::JoinSet` によりメッセージを並列処理する。

- デフォルト並列度: 4
- `with_concurrency(n)` で設定可能

### outbox_events テナント分離 RLS（HIGH-009 対応）

マルチテナント環境では `outbox_events` テーブルの RLS (Row Level Security) に `WITH CHECK` 句を必須とする。`WITH CHECK` がない場合、異テナント ID での INSERT が可能になりセキュリティ上の欠陥となる。

**必須パターン**（task/board/activity の `outbox_events` マイグレーションで適用済み）:

```sql
-- SET LOCAL で search_path を設定する（lessons.md: SET LOCAL を使用すること）
SET LOCAL search_path TO task_service, public;

-- FORCE RLS: テーブルオーナーにも RLS を適用する
ALTER TABLE outbox_events FORCE ROW LEVEL SECURITY;

-- AS RESTRICTIVE: 他のポリシーがあっても必ず適用される
-- USING: NULL 許可（バックグラウンドパブリッシャーは set_config 未呼出しで全テナントを読める）
-- WITH CHECK: INSERT/UPDATE 時にテナント ID を必ず一致させる（重要）
CREATE POLICY tenant_isolation ON outbox_events
    AS RESTRICTIVE
    USING (
        current_setting('app.current_tenant_id', true) IS NULL
        OR tenant_id = current_setting('app.current_tenant_id', true)::TEXT
    )
    WITH CHECK (
        tenant_id = current_setting('app.current_tenant_id', true)::TEXT
    );
```

**ルール**: Outbox テーブルを新規作成・修正する場合、必ず `FORCE ROW LEVEL SECURITY + AS RESTRICTIVE + WITH CHECK` の3点セットを適用すること。`USING` 句のみでは INSERT/UPDATE のテナント検証が行われない。

### processing_at タイムアウト検出（H-001）

`processing_at` がセットされたままポーラーがクラッシュした場合、そのイベントは通常の `processing_at IS NULL` 条件では永久に再取得されない。

**修正済みの SQL パターン**（全3サービスで適用済み）:

```sql
WHERE published_at IS NULL
  AND (processing_at IS NULL OR processing_at < NOW() - INTERVAL '5 minutes')
```

これにより、5分以上 `processing_at` がセットされたままのイベントは再取得対象になり、ポーラークラッシュからのリカバリが可能になる。

## ポイズンピル検出

DLQ Manager のコンシューマーは、メッセージのデシリアライゼーション失敗を「ポイズンピル」として検出する。

### エラー分類

| エラー種別 | 判定 | 処理 |
| --- | --- | --- |
| パースエラー（デシリアライズ失敗） | ポイズンピル | 即座に Dead 判定、リトライしない |
| 一時エラー（ネットワーク障害等） | 一時障害 | 通常のリトライフローで処理 |

ポイズンピルと判定されたメッセージは `mark_dead()` により即座に Dead 状態に移行し、無限リトライループを防止する。

## Kafka Consumer パターン（C-001）

### 概要

各サービスは `rdkafka::consumer::StreamConsumer` を使ってイベントを購読する。
Consumer は `tokio::spawn` でバックグラウンドタスクとして起動され、`tokio::sync::watch` チャネルを通じてグレースフルシャットダウンを受け付ける。

### 共通実装パターン

```rust
pub async fn run(self, mut shutdown_rx: tokio::sync::watch::Receiver<bool>) {
    loop {
        tokio::select! {
            _ = shutdown_rx.changed() => {
                if *shutdown_rx.borrow() { break; }
            }
            result = self.consumer.recv() => {
                match result {
                    Err(e) => warn!("recv error: {}", e),
                    Ok(msg) => {
                        // メッセージ処理
                        let _ = self.consumer.store_offset_from_message(&msg);
                    }
                }
            }
        }
    }
}
```

### at-least-once 保証

- `enable.auto.commit = false` で手動オフセット管理
- 処理成功後に `store_offset_from_message()` でオフセットをコミット
- 処理失敗時はコミットせず、再起動後に再処理（リトライ）

### 冪等性

各 Consumer が呼び出す UseCase 側でべき等性を確保する:

| Consumer | UseCase | 冪等性の仕組み |
|---------|---------|------------|
| Order Consumer | UpdateOrderStatusUseCase | `InvalidStatusTransition` をスキップとして扱う |
| Inventory Consumer | ReserveStockUseCase | 楽観ロック（VersionConflict）でリトライ |
| Payment Consumer | InitiatePaymentUseCase | `find_by_order_id` による重複チェック |

## Saga 補償トランザクションパターン（C-001）

### フロー

Choreography-based Saga を採用。各サービスは独立してイベントを購読し、成功・失敗に応じて次のイベントを発行する。

```
正常フロー:
  order.created
    → (inventory) ReserveStock → inventory.reserved
    → (payment)   InitiatePayment → payment.initiated → payment.completed
    → (order)     UpdateStatus(Confirmed)

補償フロー:
  payment.failed
    → (order)     UpdateStatus(Cancelled) → order.cancelled
    → (inventory) ReleaseStock
```

### サービス別購読トピックと処理

| サービス | 購読トピック | 処理 | Consumer Group |
|--------|-----------|------|--------------|
| order | `payment.completed` | OrderStatus → Confirmed | `k1s0-order-consumer` |
| order | `payment.failed` | OrderStatus → Cancelled (補償) | `k1s0-order-consumer` |
| inventory | `order.created` | 在庫確保 | `k1s0-inventory-consumer` |
| inventory | `order.cancelled` | 在庫解放 (補償) | `k1s0-inventory-consumer` |
| payment | `order.created` | 決済開始 | `k1s0-payment-consumer` |

### 注意事項

- inventory の `order.cancelled` 処理には `find_reserved_by_order_id` が必要（現在は警告ログのみ）
- `InventoryRepository` への `find_reserved_by_order_id` 追加は今後のタスク
