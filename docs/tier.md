# Tier

CLIを操作する人間は下記に分類される。
- system-framework/service/app: システム基盤エンジニア
- business-framework/service/app: 業務部門共通システム担当者
- business-service/app: 業務アプリケーション開発者
- infrastructure-operator: インフラ運用担当者

---

## Tier一覧

1. system-framework/service/app
	- 説明: システム基盤（OS、ネットワーク、ミドルウェア、共通プラットフォーム）を設計・提供するエンジニア。
	- 主な責務: 基盤設計・構築・運用、共通ライブラリ/サービス提供、セキュリティ/認証基盤の整備。
	- CLI利用目的: 基盤テンプレート生成、環境プロビジョニング、共通設定の自動化。
	- 例: プラットフォームチーム、SRE、インフラ基盤エンジニア。

2. business-framework/service/app
	- 説明: 部門固有の業務機能（特定部門向けのサービスやアダプタ等）を担う担当者。
	- 主な責務: 部門固有サービスの設計・実装・運用、部門要件に沿ったカスタマイズ、部門内での再利用コンポーネント提供。
	- CLI利用目的: 部門固有プロジェクトのスキャフォールディング、部門向けテンプレートや設定の生成、カスタマイズの簡易化。
	- 例: 営業部向けシステム担当、人事システム担当、部門専任のアプリ開発者。

3. business-service/app
	- 説明: 個別業務アプリケーションを開発するアプリケーション開発者。
	- 主な責務: ドメイン実装、API設計、テスト・CI/CDの整備、ビジネスロジックの実装。
	- CLI利用目的: 新規サービス/プロジェクトのスキャフォールディング、雛形コード・テスト・CI設定の生成。
	- 例: バックエンド開発者、フロントエンド開発者、サービステーム。

4. infrastructure-operator
	- 説明: 運用・デプロイ・監視・障害対応を専門に担当するチーム／個人。
	- 主な責務: デプロイ運用、監視とアラート設定、障害対応手順・自動化、運用ドキュメント整備。
	- CLI利用目的: デプロイ用マニフェストやランブックの生成、運用向けコマンドの実行支援。
	- 例: 運用チーム、Site Reliability、オンコール担当。

補足: 各Tierが期待するCLIの振る舞い
- 権限と安全性: 上位Tier（system/framework）は強い権限を必要とするコマンド、下位Tierは制限付きテンプレート生成が主。
- 拡張性: framework側は拡張ポイント（プラグインやテンプレート）を公開し、service/app側は簡単に採用できること。
- ドキュメント: 各Tierごとに操作ガイドと運用手順を分けて提供することで誤操作を防止する。
