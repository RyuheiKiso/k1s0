# /requirement — 要件定義エントリ起票

`docs/03_要件定義/` 配下に新しい要件エントリ（BR / FR / NFR / OR / DX / BC / RISK）を起票する。

## 引数

$ARGUMENTS

自然な日本語で「何を要件化するか」を指定する。ID 種別を先頭に指定するとより正確。

例:
- `FR: tier1 State API に idempotency-key を必須化`
- `NFR-B: tier1 公開 11 API のレイテンシ p99 200ms 以下`
- `BR: 稟議承認を 3 か月 → 1 週間に短縮`
- `RISK: 法務 AGPL 条項の解釈曖昧さ`

## 作成手順

### Step 0: 前提規約の読み込み

1. `.claude/skills/docs-delivery-principles/SKILL.md` を読み、納品品質原則を内面化する。
2. `.claude/skills/docs-requirement-writing/SKILL.md` を読み、要件書式規約を把握する。

### Step 1: ID 種別の判別

引数から ID 種別を自動判別する。

| ID 種別 | 対象 | キーワード例 | 配置先 |
|---------|------|--------------|--------|
| `BR-` | 業務要件 | 業務 / 稟議 / 承認フロー / 事業 | `10_業務要件/` |
| `FR-` | 機能要件 | API / 機能 / tier1 / 外部連携 | `20_機能要件/` |
| `NFR-A-` | 非機能: 可用性 | 可用性 / SLA / 冗長 | `30_非機能要件/A_可用性.md` |
| `NFR-B-` | 非機能: 性能 | レイテンシ / スループット / p99 | `30_非機能要件/B_性能拡張性.md` |
| `NFR-C-` | 非機能: 運用 | 運用 / 監視 / バックアップ | `30_非機能要件/C_運用保守性.md` |
| `NFR-D-` | 非機能: 移行 | 移行 / レガシー / データ移行 | `30_非機能要件/D_移行性.md` |
| `NFR-E-` | 非機能: セキュリティ | 認証 / 認可 / 暗号 / 脅威 | `30_非機能要件/E_セキュリティ.md` |
| `NFR-F-` | 非機能: エコロジー | 電力 / CO2 / 環境 | `30_非機能要件/F_システム環境エコロジー.md` |

| `NFR-I-` | SLI/SLO/エラーバジェット | SLO / SLI / エラーバジェット | `30_非機能要件/I_SLI_SLO_エラーバジェット.md` |

### Step 2: 次番号の採番

配置先ディレクトリ内の既存 ID を走査し、該当系列の最大番号 + 1 を次番号とする。

```bash
grep -rohE "^<PREFIX>-[0-9]+" docs/03_要件定義/ | sort -V | tail -1
```

新規採番時は `80_トレーサビリティ/` の ID 一覧も確認し、欠番を避ける。

### Step 3: エントリの下書き（三幕構成）

`docs-requirement-writing` Skill に従い、**現状 → 達成後 → 崩れた時**の三幕で散文展開する。

```markdown
### <ID>: <短いタイトル>

優先度 <MUST/SHOULD/COULD>（理由を 1 行） / Phase <N> / 関連 <BR-NNN / FR-NNN 等>

**現状**:

<現在どうなっているか、制約、課題を段落で>

**要件達成後**:

<この要件が満たされた世界で何が変わるか、ステークホルダ視点で>

**崩れた時の影響**:

<満たされなかった時に何が壊れるか、業務・技術・監査への影響>

**受け入れ基準**:

- [ ] <検証可能な具体条件>
- [ ] <検証可能な具体条件>

**関連**:

- 上流: <業務背景 / ステークホルダ要求>
- 下流: <実装 ADR / 関連設計書>
```

### Step 4: 引数から不足情報を補完

引数だけでは三幕が埋まらない場合、ユーザーに追加質問する:

- 「現状の課題を具体的に教えてください（数値・事例があれば）」
- 「要件達成後の世界で、誰の何が変わりますか」
- 「この要件が崩れた時、業務・監査にどんな影響がありますか」
- 「受け入れ基準として検証可能な条件を 3 件以上挙げられますか」

**自動で埋めない**。情報不足のまま三幕を書くと「表でごまかした」納品と同じ扱いになる。

### Step 5: トレーサビリティ登録

`80_トレーサビリティ/` の要件マトリクスに新規 ID を追加することをユーザーに案内する。

### Step 6: 自己検証

`docs-requirement-writing` Skill の自己チェックリスト 10 項目を通す。特に以下を重点確認:

- 表セルに要件本体を詰め込んでいないか
- 三幕が散文で展開されているか（箇条書き化していないか）
- 優先度の判定理由が記載されているか（例: 「MUST（稟議通過の前提）」）

## 記述ルール

- 本文は日本語
- 各要件は**単独で納得できる**散文に仕上げる。読み手が関連文書を参照せずとも趣旨がわかる水準。
- 数値には必ず根拠を併記
- 表は章末のサマリにのみ使用
- アスキー図禁止、図が必要なら drawio
