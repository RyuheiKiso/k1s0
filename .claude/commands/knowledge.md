# /knowledge — 技術学習用 Knowledge ドキュメント作成

第三者が技術を体系的に学べる学習用 Knowledge ドキュメントを作成する。

## 引数

$ARGUMENTS

引数の形式: `<技術名>`

- 技術名のみ指定する（例: `kubernetes`, `rust`, `tokio`）
- 技術名が不足している場合のみ対話的に確認する
- 技術カテゴリと学習トピックは Step 0 で自動判別する

## 作成手順

以下のステップを **必ず順番どおり** に実行すること。

### Step 0: 技術カテゴリ・学習トピックの自動判別

技術名のみからカテゴリとトピック一覧を自動判別する。ユーザーに確認は不要。

#### 0-1. カテゴリ判別

| カテゴリ | 判別基準 | 例 |
|----------|----------|----|
| `language` | プログラミング言語・言語仕様・言語機能 | Rust, Go, TypeScript, Python, SQL |
| `framework` | アプリケーションフレームワーク | Actix Web, Axum, React, Next.js, Django, Spring Boot |
| `library` | 特定の機能を提供するライブラリ・クレート | Tokio, Serde, ZEN Engine, pandas |
| `protocol` | 通信プロトコル・データフォーマット仕様 | gRPC, HTTP/2, GraphQL, Protocol Buffers, MQTT |
| `tool` | 開発ツール・CLI・ビルドシステム | Cargo, Docker, Git, Terraform, Helm |
| `infra` | インフラ基盤・ミドルウェア・プラットフォーム | Kubernetes, Dapr, Kafka, PostgreSQL, Redis, AWS |

判別が曖昧な場合（例: Docker はツールでもインフラでもある）は、その技術の主な学習文脈から判断する。
- CLI 操作・使い方が中心 → `tool`
- 基盤としての構成・運用が中心 → `infra`

#### 0-2. 学習トピックの自動分割

以下の手順でトピック一覧を決定する:

1. `docs/90_knowledge/` 配下を確認し、その技術の既存 Knowledge があれば重複を避ける
2. context7 MCP で対象技術の公式ドキュメント構成を取得する
3. 以下の粒度ガイドラインに従い、トピック一覧を決める

**粒度ガイドライン — カテゴリ別の典型トピック構成:**

| カテゴリ | 典型的なトピック分割 |
|----------|----------------------|
| `language` | basics（型・変数・制御構文）→ 言語固有の中核機能（例: ownership, async）→ エラー処理 → テスト |
| `framework` | basics（全体像・最小構成）→ ルーティング → ミドルウェア → DB 連携 → テスト |
| `library` | basics（導入・最小構成）→ 主要 API → 設定・カスタマイズ → 実践パターン |
| `protocol` | basics（仕組み・通信フロー）→ メッセージ設計 → 実装（クライアント/サーバ）→ 運用 |
| `tool` | basics（インストール・CLI 基本操作）→ 設定 → ワークフロー → プラグイン/拡張 |
| `infra` | basics（全体像・コアコンセプト）→ リソース定義 → ネットワーク → 運用・監視 |

**分割ルール:**
- 各トピックは 300 行以内に収まる粒度にする
- 最初は必ず `basics`（全体像・コアコンセプト）から作成する
- トピック間は相互リンクで接続し、学習順序を明示する
- 1 回のコマンド実行では **1 トピック分のみ** 作成する（全トピックを一括作成しない）
- 作成するトピックは、既存ドキュメントが無ければ `basics` を選択し、`basics` が既にあれば次の優先順のトピックを選択する

### Step 1: 情報収集

1. context7 MCP を使って対象技術の最新ドキュメントを取得する
2. 取得した情報をもとに、学習者にとって最も重要な内容を整理する

### Step 2: フォーマット確認

1. `docs/00_format/knowledge用.md` を読み、フォーマット規約を確認する
2. 既存の Knowledge ドキュメント（`docs/90_knowledge/` 配下）があれば参照し、粒度・文体を揃える

### Step 3: drawio 図の作成

1. md と同階層に `img/` ディレクトリを作成する
2. 学習に有効な図を drawio XML で作成する（最低 1 枚、推奨 2 枚）
   - アーキテクチャ図・構成図・処理フロー図・関係図など
   - 色分けで要素の種類を区別する（Control Plane 系=黄, Worker 系=緑, Pod 系=赤, アクセス系=青 など）
   - 日本語のラベルや補足を入れてよい
3. `"/c/Program Files/draw.io/draw.io.exe" --export --format svg --output <出力先.svg> <入力.drawio>` で svg を生成する
4. drawio ファイルと svg ファイルの両方を `img/` に格納する

### Step 4: マークダウン本体の作成

配置先: `docs/90_knowledge/<技術カテゴリ>/<技術名>_<学習トピック>.md`

以下のセクション構成に従う:

```
# <技術名>: <学習トピック>

- 対象読者:
- 学習目標:
- 所要時間:
- 対象バージョン:
- 最終更新日:

## 1. このドキュメントで学べること
## 2. 前提知識
## 3. 概要
## 4. 用語の整理
## 5. 仕組み・アーキテクチャ        ← drawio svg を埋め込む
## 6. 環境構築
## 7. 基本の使い方                  ← 動作する最小コード例を含む
## 8. ステップアップ
## 9. よくある落とし穴
## 10. ベストプラクティス
## 11. 演習問題（任意）
## 12. さらに学ぶには
## 13. 参考資料
```

### Step 5: 品質チェック

以下を全て満たすことを確認する:

- [ ] 300 行以内であること（`wc -l` で確認）
- [ ] 対象読者が前提知識だけで理解できる流れになっていること
- [ ] 専門用語の初出時に平易な説明を併記していること
- [ ] コード例が動作する最小構成であること
- [ ] コード例に CLAUDE.md のコメント規約（各行 1 行上に日本語コメント、先頭にファイル説明）を適用していること
- [ ] 図はアスキーアートではなく drawio svg を使っていること
- [ ] svg が正しく生成・参照されていること
- [ ] 「事実」と「解釈」が区別されていること
- [ ] 演習問題で理解度を自己検証できること

### Step 6: 300 行超過時の分割

300 行を超えた場合は、以下の方針で分割する:

1. セクション 8（ステップアップ）の各サブトピックを個別ファイルに切り出す
2. ファイル名は `<技術名>_<サブトピック>.md` とする
3. 元ファイルからリンクで接続し、学習順序を示す
4. 各分割ファイルも同一フォーマットに従う

## 記述ルール（厳守）

- 本文は日本語で記載する
- アスキーアートによる図表は禁止（drawio svg を使用）
- 表は markdown テーブル記法で記載してよい
- コード例内のコメントは CLAUDE.md のコーディング規約に従う
- 「〜すればよい」など主観表現を避け、根拠を示す
- バージョン依存の情報には対象バージョンを明記する
- 1 ファイル = 1 学習トピック
