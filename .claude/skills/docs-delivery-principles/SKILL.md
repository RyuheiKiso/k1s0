# ドキュメント納品品質原則

本 Skill は k1s0 のドキュメント作業すべてに適用される**メタ原則**。書式ルール（散文 / 表 / 図）は個別 Skill（`docs-requirement-writing` / `docs-adr-authoring` / `docs-design-spec`）で定義されているが、**作業の姿勢**に関わる原則は本 Skill に集約する。過去に本原則を逸脱した納品で複数回の叱責を受けており、再発防止のため明文化する。

## 原則 1: 部外者テスト

- ドキュメントの**納品基準**は「部外者（監査人、新規採用者、外部レビュワー）が独立して読んで理解できる」こと。
- この基準を満たさない納品は**要件未達**であり、「あとで補足する」「本人が読めばわかる」は不可。
- 納品前に**部外者テスト**をセルフで通す: 自分が書いた章を、プロジェクトの前提知識を持たないつもりで再読し、「何が書いてあるか」ではなく「**これを読んで判断や行動ができるか**」を問う。

## 原則 2: 段階対応を量の言い訳にしない（最重要）

- 「**作業量が多いから段階的に**」という提案は、品質責務の矮小化であり**禁止**。
- 過去セッション `72d524f1`（2026-04-14）で、要件定義書の品質指摘に対して「作業量が多いから段階的に改善する」と返答したところ、「**それはプロとして妥当か**」と二度目の叱責を受けた。量は受注側の都合であり、要件未達の言い訳にならない。
- 今回の作業で「Phase 1 だけやる」「段階的に進める」を**自分から提案しない**。ユーザーが明示的に分割指示した時のみ段階実施する。

## 原則 3: 品質確認としての少量先行は可

- 段階対応が完全に禁じられるわけではない。**品質確認の文脈**で少量先行するのは妥当。
- 「作業量が多いから段階的に」ではなく「**方針ズレを防ぐため、まず 1 章分で粒度と文体を確認してから横展開したい**」という提案は可。
- 違いは**誰の都合で段階化するか**: 品質担保の都合ならOK、受注側の工数都合はNG。

## 原則 4: 生成後自己検証の工程化

- Claude はドキュメント・drawio を初回生成時に検証を怠る癖がある。過去セッションで複数回発生済み（drawio の矢印交差、SVG 背景透過、レイヤ混同）。
- **生成して終わりにしない**。書式・交差・参照整合・トレーサビリティを、生成後に必ず自己検証する。
- 具体的な検証手順は各個別 Skill のチェックリストに従うこと。

## 原則 5: 図解が「後から気づく」を生まない

- 過去セッション `7694a878`（2026-04-19）で、Dapr と Istio のサイドカーを同じビジュアルで描き、ユーザーが**会話で確認するまで責務差異を認識できなかった**事例がある。
- 図は**それ単独で責務関係を表現しきる**こと。読み手が後から会話や説明を頼る状態は納品未達。
- 複数レイヤが登場する図は必ず `figure-layer-convention` Skill に従い、色・線種・配置で責務分離を視覚化する。

## 実際の運用

docs 作業を始める時、以下の順で確認する:

1. 本 Skill を読み、上記 5 原則を内面化する。
2. 作業種別に応じた個別 Skill（`docs-requirement-writing` / `docs-adr-authoring` / `docs-design-spec` / `docs-postmortem`）を読む。
3. 図を含む場合は `drawio-authoring` と（必要なら）`figure-layer-convention` を読む。
4. 作業中、段階対応を自分から提案したくなった時は、原則 2-3 に戻って「品質の都合か、工数の都合か」を自問する。
5. 納品前に部外者テストを通す（原則 1）。
6. 納品前に個別 Skill のチェックリストで自己検証する（原則 4）。
