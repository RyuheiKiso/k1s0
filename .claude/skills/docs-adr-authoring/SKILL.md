# ADR 起票規約

k1s0 の ADR は `docs/02_構想設計/adr/` に配置し、`docs/00_format/ADR-xxxx.md` テンプレートに準拠する。本 Skill はテンプレートの要点とレビュー観点を集約したもの。既存 ADR が 27 件以上あり、運用パターンが確立されているため、新規起票時は**既存系列に揃える**ことを優先する。

## ファイル命名

`ADR-DOMAIN-NNN-short-name.md` 形式。ドメイン系列は既存の運用に従うこと:

| 系列 | 主題 | 既存番号の最大値 |
|------|------|------------------|
| `ADR-000x` | 初期番号制（横断的な決定） | 0003 |
| `ADR-BS-*` | Backstage | BS-001 |
| `ADR-CICD-*` | CI/CD | CICD-003 |
| `ADR-DATA-*` | データ基盤 | DATA-004 |
| `ADR-DEP-*` | 依存管理 | DEP-001 |
| `ADR-DEV-*` | 開発者体験 | DEV-001 |
| `ADR-DIR-*` | ディレクトリ構造 | DIR-003 |
| `ADR-DX-*` | DX メトリクス | DX-001 |
| `ADR-FM-*` | Feature Management | FM-001 |
| `ADR-MIG-*` | 移行 | MIG-002 |
| `ADR-OBS-*` | 観測性 | OBS-003 |
| `ADR-POL-*` | ポリシー | POL-001 |
| `ADR-REL-*` | リリース | REL-001 |
| `ADR-RULE-*` | ルールエンジン | RULE-002 |
| `ADR-SEC-*` | セキュリティ | SEC-003 |

採番は各系列の最大番号 + 1。横断的な決定は `ADR-000x` 系に起票してもよい。short-name は**ケバブケース小文字**（例: `ADR-DATA-005-postgres-pgvector.md`）。

## 5 段構成（厳守）

```markdown
# ADR-<ID>: <決定事項のタイトル>

- ステータス: <Proposed | Accepted | Deprecated | Superseded by ADR-MMMM>
- 起票日: YYYY-MM-DD
- 決定日: YYYY-MM-DD
- 起票者: <氏名 / ロール>
- 関係者: <レビュアー、意思決定者>

## コンテキスト
## 決定
## 検討した選択肢
## 決定理由
## 影響
```

## 各セクションの必須要素

1. **コンテキスト** — 「何を悩んでいたか」が後から読んでも分かるように、**制約・前提・トレードオフ**を明記する。
2. **決定** — 採用した選択肢と、その内容を明確に記載する（1 つの ADR で 1 つの決定のみ）。
3. **検討した選択肢** — **最低 3 件**。各選択肢に「概要 / メリット / デメリット」を記載。採用しなかった選択肢にも理由を残す。
4. **決定理由** — 「**なぜ他の選択肢ではないのか**」を必ず含める。比較軸と判断基準を明示。
5. **影響** — 「ポジティブな影響」「ネガティブな影響 / リスク」「移行・対応事項」の 3 サブセクション**すべて**を記載。

## 記述ルール

- 本文は日本語。日付は `YYYY-MM-DD` 形式。
- 決定済み ADR は**原則として書き換えない**。変更が必要な場合は新規 ADR を起票し、旧 ADR のステータスを `Superseded by ADR-NNNN` に更新する。
- 図表はアスキーアート禁止。drawio で作成した SVG を埋め込む（`drawio-authoring` Skill を併用）。
- 関連 ADR / 関連要件 / 関連 Issue / PR のリンクを参考資料に列挙する。

## 運用フロー

1. `docs/00_format/ADR-xxxx.md` を複製し、該当系列の次番号を採番する。
2. ステータスを `Proposed` として PR を作成、関係者のレビューを受ける。
3. 合意が取れたらステータスを `Accepted` に更新し、決定日を記入してマージ。
4. 既存の決定を覆す場合は新規 ADR を起票し、旧 ADR を `Superseded by ADR-NNNN` に更新。

## 登録先

マージ後、以下の索引を更新する:

## 自己チェックリスト

ADR を提出する前に以下を確認する:

1. [ ] ファイル名が `ADR-DOMAIN-NNN-short-name.md` 形式で、系列の次番号になっている。
2. [ ] 5 段構成（コンテキスト / 決定 / 検討した選択肢 / 決定理由 / 影響）がすべて埋まっている。
3. [ ] 検討した選択肢が 3 件以上あり、各選択肢にメリット / デメリットがある。
4. [ ] 決定理由に「なぜ他の選択肢ではないのか」が書かれている。
5. [ ] 影響のポジティブ / ネガティブ / 移行事項の 3 サブセクションすべてに内容がある。
6. [ ] 関連 ADR / 要件 / Issue へのリンクが参考資料にある。
7. [ ] 図がある場合、drawio で作成し img/ に配置し svg を埋め込んでいる。
