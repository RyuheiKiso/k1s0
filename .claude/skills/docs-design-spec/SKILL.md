# 技術説明書・設計書規約（タイプ D）

k1s0 で最も作成頻度の高いドキュメント種別。`05_実装/` の各セクション設計書、`04_概要設計/` の方式設計書、`02_構想設計/` のアーキテクチャ資料が該当する。要件定義書（タイプ A）や ADR（タイプ B）とは書式が異なるため、本 Skill を必ず参照してから執筆する。

## 対象範囲

| ディレクトリ | 典型例 |
|--------------|--------|
| `02_構想設計/` | 01_アーキテクチャ / 02_tier1設計 / 03_技術選定 / 04_CICDと配信 / 05_法務とコンプライアンス |
| `04_概要設計/` | 10_システム方式設計 / 20_ソフトウェア方式設計（コンポーネント / 外部・内部 IF / DB / 利用者文書） / 30_共通機能 / 40_制御 / 50_非機能 / 55_運用 / 60_移行 / 70_開発者体験 / 75_事業運用 |
| `05_実装/` | 00_ディレクトリ設計 / 10_ビルド / 20_コード生成 / 30_CICD / 40_依存管理 / 50_開発者体験 / 60_観測性 / 70_リリース / 80_サプライチェーン / 85_Identity / 90_ガバナンス / 95_DX メトリクス / 99_索引 |

## 書式の骨格

### 冒頭

文書タイトルの直後に、**「何を解決するか」の導入段落**を必ず置く。部外者が 1 段落で目的を理解できる品質。

例（STRIDE 脅威モデル設計の場合）:
> 本書は k1s0 全体の脅威モデリングを STRIDE 分類に沿って体系化する。過去に脅威分析が担当者の暗黙知に留まり、新規ディレクトリ設計時に「守るべき資産」が再定義されないまま実装が進むケースがあった。本書はそれを防ぐため、STRIDE 6 軸ごとに**資産 / 脅威 / 既存対策 / 残存リスク**の 4 点を表形式で採番する。

### 章立ての基本順序

図を掲載する時は**概念図 → 構造図 → シーケンス図 → ER 図**の順序で配置する。論理から実装詳細に降りていく流れに揃えることで、読み手が全体像を掴んでから細部に入れる。

典型的な章構成:

```
# <タイトル>

<導入段落（何を解決するか）>

## 1. 背景と目的
## 2. 全体構造          ← 概念図 / 構造図
## 3. 処理フロー         ← シーケンス図
## 4. データモデル       ← ER 図（該当時のみ）
## 5. 設計判断の根拠     ← 関連 ADR へのリンク
## 6. トレーサビリティ   ← 要件 / ADR / 他設計との対応
## 7. 制約と今後の課題
```

## 論理レベルで書く

- 本書は**論理レベル**で記述し、実装詳細（具体コード / Helm の values / CI の step 定義など）は参照リンクに留める。
- 実装詳細が本書に混ざると保守時に設計書が陳腐化するため、「何を・なぜ」だけを残し「どう」は実装側へ誘導する。
- 具体的な kubectl / SQL / コードスニペットが必要な場合は別途 Runbook（タイプ C）として切り出すか、参照リンクで誘導する。

## 散文と図

- 各章・各節の冒頭に、何を解決するかの導入段落を必ず置く。
- 表の前後に「なぜこの分類なのか」「どう読むか」「重要な関係性」の散文を添える。
- 重要な概念・関係性・フロー・構造は drawio 図を作成し SVG で埋め込む（`drawio-authoring` Skill を併用）。
- アスキー図・罫線図は禁止。

## トレーサビリティ（必須）

設計書は単体で閉じず、必ず以下のリンクを持たせる:

- **上流**: 実装する要件 ID（`FR-*` / `NFR-*` 等） / 根拠 ADR
- **下流**: 関連する実装 / 他設計書 / Runbook
- **索引**: `05_実装/99_索引/` 配下の対応表（ADR 対応表 / DS-SW-COMP 対応表 / NFR 対応表 / Backstage catalog 対応）に登録

## ID 体系

| ID プレフィックス | 所在 |
|-------------------|------|
| `DS-SW-COMP-*` | `04_概要設計/20_ソフトウェア方式設計/01_コンポーネント方式設計/` |
| `IMP-DIR-*` | `05_実装/00_ディレクトリ設計/` |
| `IMP-*`（他系列） | `05_実装/` 各章下 + `99_索引/00_IMP-ID一覧/` |

新規採番時は該当索引を先に参照し、欠番を避けて次番号を使う。

## 表の使用基準

- ラベル羅列の表（例: 「○ / △ / ×」のみ）は禁止。各セルに根拠・数値・出典を含む。
- 情報密度が散文より高くなる場合のみ表を使う。
- 章末のサマリ（ID 一覧 / 段階達成度 / 優先度分布）には表を使ってよい。

## 300 行超過時の分割

- 300 行を超える場合は、節ごとにファイルを分割する。
- 分割後、元ファイルは README 的な目次 + 各節へのリンクに置き換える。
- `05_実装/` の `01_xxx設計.md` が 300 行を超える場合は、番号付きサブファイル（`02_xxx_詳細.md` 等）に分割する。

## 自己チェックリスト

1. [ ] 冒頭に「何を解決するか」の導入段落がある。
2. [ ] 図の順序が「概念 → 構造 → シーケンス → ER」になっている。
3. [ ] 論理レベルで書かれ、実装詳細は参照リンクに留められている。
4. [ ] 章・節の冒頭に導入段落がある。
5. [ ] 表の前後に解説散文がある。
6. [ ] 要件 ID / ADR / 他設計へのトレーサビリティリンクがある。
7. [ ] `05_実装/99_索引/` の該当対応表を更新した。
8. [ ] 300 行を超えていないか、超えていれば分割された。
9. [ ] drawio 図は svg でエクスポートされ、`drawio-authoring` Skill の規約を満たしている。
