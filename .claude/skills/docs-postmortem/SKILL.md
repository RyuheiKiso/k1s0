# ポストモーテム記述規約

インシデント発生後、`docs/00_format/postmortem_template.md` を複製して `docs/40_運用ライフサイクル/postmortems/` 配下に配置する。本 Skill はテンプレートの要点と守るべき原則を集約したもの。

## Blameless 原則（最重要）

- 個人責任追及の文言は**禁止**。
- 「〇〇さんがミスした」ではなく「プロセスが」「チームが」「仕組みが」と主語を組織に置き換える。
- 組織学習と再発防止に焦点を当てる。
- うまくいかなかった点を挙げる前に、**うまくいった点を先に記載**する。

## 作成 SLA

| Severity | ポストモーテム作成 | Draft 期限 | Approved 期限 |
|----------|---------------------|------------|---------------|
| S1 / S2 | 必須 | 収束から 5 営業日以内 | 10 営業日以内 |
| S3 / S4 | SRE 判断 | — | — |

## ファイル命名

`INC-{YYYY-MM-DD}-{NNN}.md`（インシデント ID と同一）。
例: `INC-2026-04-24-001.md`

## 10 セクション構成（厳守）

```markdown
# ポストモーテム: {インシデント概要}

- インシデント ID: INC-{YYYY-MM-DD}-{NNN}
- Severity: S1 / S2 / S3 / S4
- 発生日時: YYYY-MM-DD HH:MM JST
- 収束日時: YYYY-MM-DD HH:MM JST
- 影響継続時間: {MM} 分
- 主担当: {SRE オンコール名}
- 関係者: {関連チーム}
- 作成日: YYYY-MM-DD
- 作成者: {担当者名}
- ステータス: Draft / In Review / Approved

## 1. サマリ
## 2. 影響範囲
## 3. タイムライン
## 4. 根本原因
## 5. うまくいった点
## 6. うまくいかなかった点
## 7. アクションアイテム
## 8. 再発防止
## 9. 学び（Lessons Learned）
## 10. 付録
```

## 各セクションの必須要素

1. **サマリ** — 1〜2 段落で経営視点。技術者以外（Product Council、監査）が読んで理解できる水準。
2. **影響範囲** — 業務影響（件数・金額・顧客影響） / SLO 影響（エラーバジェット消費率、SLI 値） / データ損失 / 監査報告の要否。
3. **タイムライン** — Audit API から自動抽出された特権操作タイムラインを付録に添付した上で、重要イベントを時系列表で列挙（時刻 JST / イベント / 担当 / 備考）。
4. **根本原因** — 5 Whys / Fishbone で分析し、**直接原因 / 組織的原因 / 検知遅延の原因 / 対応遅延の原因**の 4 サブセクションに分ける。
5. **うまくいった点** — Blameless 原則の一環。チームが正しく動いた箇所を先に挙げる。
6. **うまくいかなかった点** — 改善余地を**事実ベース**で。個人批判なし。
7. **アクションアイテム** — 具体的かつ期限付き。担当者明示。表形式（# / アクション / 担当 / 期限 / 関連）。
8. **再発防止** — 検知強化 / Runbook 更新 / 設計変更（必要なら ADR 追加）/ トレーニング（BC-TRN 追加）。
9. **学び** — 組織として持ち帰るべき教訓。将来の設計判断に活きる水準で言語化。
10. **付録** — アラート履歴 / ログ断片 / トレース / Audit タイムライン / 関連 PR・Issue。

## 関連ドキュメントの更新

ポストモーテム作成と同時に以下を更新する:

- **Runbook** — 関連 Runbook に学びを反映。
- **FMEA** — 該当 RPN の Det 値・Sev 値を再評価。
- **ADR** — 設計変更が必要なら新規起票（`docs-adr-authoring` Skill）。
- **アラート / Prometheus ルール** — 検知強化の実装。

## 四半期レビュー

四半期に全ポストモーテムを集約レビューし、組織学習の材料にする。横断的な傾向（特定ドメインの連続障害、検知漏れパターン）があれば、ADR や設計変更に昇華する。

## 自己チェックリスト

Draft を In Review に遷移させる前に以下を確認:

1. [ ] ファイル名が `INC-YYYY-MM-DD-NNN.md` 形式で、ID と一致している。
2. [ ] ヘッダ項目（Severity / 発生日時 / 収束日時 / 影響継続時間 / 主担当 / 作成者 / ステータス）がすべて埋まっている。
3. [ ] 10 セクションすべてに内容がある（空白セクションなし）。
4. [ ] 個人名を主語にした批判的記述がない（「〇〇さんが」→「プロセスが / チームが」に置換済み）。
5. [ ] うまくいった点が、うまくいかなかった点より先に記載されている。
6. [ ] アクションアイテムに担当者と期限が明記されている。
7. [ ] 関連 Runbook / FMEA / ADR の更新リンクがある。
8. [ ] Audit API タイムラインが付録に添付されている。
