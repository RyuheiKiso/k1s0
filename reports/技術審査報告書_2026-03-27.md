## 1. エグゼクティブサマリー

k1s0 プロジェクトは、Rust/Go/TypeScript/Dart の 4 言語対応・70 以上のマイクロサービスを擁する大規模モノリポジトリである。アーキテクチャドキュメントの整備水準は高く、クリーンアーキテクチャ・DDD・TDD の原則に沿った設計が行われている。Vault によるシークレット管理、Keycloak による認証・認可、Istio によるサービスメッシュなど、エンタープライズグレードのインフラ基盤が存在することは評価に値する。

しかし本審査では、**本番環境への即時適用を阻害する CRITICAL 問題 6 件を含む、合計 50 件超の問題**を発見した。特に深刻なのは以下の 3 点である。

1. **Proto 型変更と実装の乖離によるビルド失敗**（4 サービス）: graphql-gateway・session・featureflag・tenant が現時点でビルド不能。Proto enum の String→i32 変換に実装側が追従していない。
2. **Docker-compose の初期化バグ（再現済み）**: `K1S0_DB_PASSWORD` が postgres コンテナに渡されない設計バグにより、init-db スクリプトが失敗し、複数の依存サービスが起動不能となる。
3. **本番環境 etcd の暗号化キーがプレースホルダーのまま**: `encryption-config.yaml` に `<REPLACE_WITH_BASE64_32BYTE_KEY>` が残存しており、このまま本番デプロイされると全 Kubernetes Secret が平文保存される。

これらは設計思想の欠陥ではなく、実装の未完成・テスト不足によるものであり、計画的に対処可能である。

#### [CRITICAL-DC-01] K1S0_DB_PASSWORD が postgres コンテナに未渡し

**ファイル**: `docker-compose.yaml`（postgres サービスの environment セクション）
**再現手順**: `docker compose --profile infra up -d` → postgres の init-db スクリプトが `K1S0_DB_PASSWORD must be set` で失敗
**影響**:
- `infra/docker/init-db/00-create-app-user.sh` が失敗し `k1s0` ロールが未作成
- postgres が unhealthy になり keycloak・vault など依存サービスが全て起動不能
- 修正により全サービス起動を確認済み

**修正内容**: `docker-compose.yaml` の postgres `environment` に `K1S0_DB_PASSWORD: ${K1S0_DB_PASSWORD:?K1S0_DB_PASSWORD is required}` を追加

---

#### [HIGH-DC-02] init-db スクリプトの実行順序バグ

**ファイル**: `infra/docker/init-db/16-roles.sh`
**再現手順**: postgres 起動時、`04-saga-schema.sql` 等が参照するロール `k1s0_saga_rw` 等が未作成でエラー
**影響**:
- `k1s0_saga_rw`, `k1s0_dlq_rw`, `k1s0_ratelimit_rw`, `k1s0_tenant_rw`, `k1s0_vault_rw` への GRANT が失敗
- saga, dlq-manager, ratelimit, tenant, vault サービスが DB 接続時に権限エラーとなる状態

**根本原因**: ロール作成スクリプトが `16-` 番で、ロールを参照する `04-09-` 番スキーマスクリプトより後に実行される
**修正内容**: `16-roles.sh` → `01z-create-roles.sh` にリネームして実行順序を修正

## 5. インフラ・セキュリティ審査

### 5.1 CRITICAL 問題

#### [CRITICAL-SEC-01] etcd 暗号化設定のプレースホルダー値が本番対応未完了

**ファイル**: `infra/kubernetes/security/encryption-config.yaml` (L36, L49)

```yaml
# 実際のファイル内容
- aescbc:
    keys:
      - name: key1
        secret: <REPLACE_WITH_BASE64_32BYTE_KEY>  # プレースホルダーが2箇所残存
```

**リスク**: このファイルのまま本番環境にデプロイされると etcd が暗号化されず、全 Kubernetes Secret が平文保存される。etcd への不正アクセス時に全シークレット（DB パスワード、API キー、証明書）が一度に露出する。
**対応**: CI/CD パイプラインでデプロイ前に `<REPLACE_` 残存チェックを自動化し、Vault から キーを注入するステップを組み込む。

---

#### [CRITICAL-SEC-02] Kubernetes Secret に固定パスワード（verify 環境）

**ファイル**: `infra/kubernetes/verify/keycloak-secrets.yaml` (L16-20)

```yaml
stringData:
  admin-password: "admin"
  test-admin-password: "admin123"
  test-user-password: "user123"
```

**リスク**: 弱いデフォルトパスワードがコミット履歴に記録される。verify 環境であっても誤った本番適用のリスクが存在する。
**対応**: パスワードをランダム値に変更し、External Secrets Operator または Vault Agent Injector で管理。

---

#### [CRITICAL-SEC-03] Keycloak テスト用資格情報が ConfigMap に平文埋め込み

**ファイル**: `infra/kubernetes/verify/keycloak.yaml` (L408, L418)

```yaml
# ConfigMap（デフォルトで etcd に暗号化なしで保存される）
test-admin パスワード: "admin123"
test-user パスワード: "user123"
```

**リスク**: ConfigMap は RBAC 権限があれば誰でも読取可能。etcd 暗号化が無効（CRITICAL-SEC-01 と重複）の場合は平文で保存される。
**対応**: ConfigMap から削除し、Kubernetes Secret に移設の上、etcd 暗号化を有効化。

---

#### [CRITICAL-SEC-04] docker-compose.dev.yaml での認証バイパスの広範な有効化

**ファイル**: `docker-compose.dev.yaml` (L36-42, L59-210)

```yaml
x-dev-auth-bypass: &dev-auth-bypass
  build:
    args:
      CARGO_FEATURES: "k1s0-server-common/dev-auth-bypass"
  environment:
    - ALLOW_INSECURE_NO_AUTH=true
```

**リスク**: 全システムサービスで認証バイパスが有効化されており、開発機が誤って本番相当環境で使用される場合のリスクが存在する。また、サービス間通信での認証検証が行われていないため、開発環境で発見できないセキュリティ問題が本番で顕在化する。
**評価**: フィーチャーゲートにより本番ビルドからはバイナリレベルで除去される設計は正しいが、開発環境でも完全な認証フローのテストが必要。

---

#### [CRITICAL-SEC-05] .env.dev と .gitignore の矛盾

**ファイル**: `.gitignore` (L46)、`.env.dev`

**状態**: `.gitignore` に `.env.dev` を除外するエントリがあるにもかかわらず、実際にはリポジトリに追跡されている（git に含まれている）。
**リスク**: 意図しない機密情報のリポジトリへの混入リスク。ガイドライン矛盾が開発者の誤認識を招く。
**対応**: `.env.dev` をリポジトリに含める場合は `.gitignore` から削除し、コメントで開発専用固定値のみを含むことを明記する。機密情報は含めない。

### 5.2 HIGH 問題

#### [HIGH-SEC-01] etcd バックアップの hostPath マウント

**ファイル**: `infra/kubernetes/backup/etcd-backup-cronjob.yaml` (L94-96)

```yaml
volumes:
  - name: etcd-certs
    hostPath:
      path: /etc/kubernetes/pki/etcd
```

**リスク**: etcd 証明書への直接アクセスが可能な Pod が存在する。etcd 証明書流出時は etcd への任意アクセスと全シークレット露出に直結。
**対応**: etcd Operator または外部バックアップソリューションの使用を検討。最低限、nodeSelector + taint でアクセスできるノードを制限する。

---

#### [HIGH-SEC-02] Istio mTLS が一部 PERMISSIVE（平文通信許可）

**ファイル**: `infra/istio/peerauthentication.yaml` (L28-31)

```yaml
portLevelMtls:
  "15020":
    mode: PERMISSIVE  # Prometheus スクレイプ用例外
```

**リスク**: PERMISSIVE モードでは mTLS がオプション（平文通信も許可）。Prometheus メトリクス収集のためとはいえ、このポートへの通信は盗聴可能。
**対応**: Envoy sidecar の inject 対象外 Pod を明示し、STRICT モードを維持しつつ例外を最小化する。

---

#### [HIGH-SEC-03] Kong レート制限が Redis 障害時に機能停止（DoS リスク）

**ファイル**: `infra/kong/kong.yaml` (L40-41)

```yaml
fault_tolerant: false
# コメント: "DDoS 耐性維持のため Redis 障害時もリミット強制"
```

**リスク**: コメントと実装が矛盾している。`fault_tolerant: false` は Redis 障害時にレート制限プラグインが機能停止し、リクエストを素通しさせることを意味する。Redis が障害になった瞬間にシステム全体が DDoS に無防備となる。
**対応**: Redis を HA 構成（Sentinel/Cluster）にするか、`fault_tolerant: true` に変更して local fallback を許可する。コメントを実装に一致させる。

---

#### [HIGH-SEC-04] Keycloak realm に localhost redirect URI が本番設定に混入

**ファイル**: `infra/keycloak/realm-k1s0.json` (L168-170)

```json
"redirectUris": [
  "https://app.k1s0.internal.example.com/callback",
  "http://localhost:3000/callback",  // 本番設定に localhost が残存
  "http://localhost:5173/callback"
]
```

**リスク**: 攻撃者が localhost で自前のサーバーを起動した場合、OAuth コールバックの不正受信が可能。
**対応**: 環境別 realm ファイルを分離し、本番用からは localhost を削除する。

---

#### [HIGH-SEC-05] BFF Proxy の Redis パスワードが設定ファイルに平文記載

**ファイル**: `regions/system/server/go/bff-proxy/config/config.docker.yaml` (L38-39)

```yaml
session:
  redis:
    password: "dev"
```

**リスク**: 開発用とはいえ、設定ファイルに平文パスワードが記載されており、誤った本番適用リスクがある。
**対応**: 本番設定テンプレートに Vault 参照パターンを明記する。

---

#### [HIGH-SEC-06] Keycloak webOrigins にワイルドカード CORS 許可

**ファイル**: `infra/keycloak/realm-k1s0.json` (L172-176)

```json
"webOrigins": [
  "https://*.k1s0.internal.example.com",
  "http://localhost:3000",
  "http://localhost:5173"
]
```

**リスク**: ワイルドカード（`*.k1s0.internal.example.com`）は意図しないサブドメインへのアクセスを許可する。CSRF 攻撃の表面積が広がる。
**対応**: 本番環境では明示的なドメインリストに変更する。

---

#### [HIGH-SEC-07] Keycloak start-dev コマンドが verify k8s マニフェストにも混入

**ファイル**: `infra/kubernetes/verify/keycloak.yaml` (L484)
**問題**: k8s の verify 環境でも `start-dev` コマンドが使用されており、TLS なし・セキュリティ緩和モードで動作している。docker-compose.dev.yaml のみならず k8s 環境にも波及している。
**対応**: verify 環境でも `start` コマンドを使用し、開発環境との乖離をなくす。

---

#### [HIGH-SEC-08] sslmode=disable が開発環境 DATABASE_URL に設定

**ファイル**: `docker-compose.dev.yaml`（複数サービス）
**問題**: 全サービスの `DATABASE_URL` に `?sslmode=disable` が付与。開発効率のためとはいえ、本番移行時の設定変更漏れリスクが高い。
**対応**: インフラ設定テンプレートに本番推奨値（`sslmode=require`）をコメントで明記し、CI でのチェックを追加する。

## 6. コード品質審査（Rustサービス）

### 6.1 CRITICAL 問題

#### [CRITICAL-CODE-01] graphql-gateway: unreachable! パニックリスク

**ファイル**: `regions/system/server/rust/graphql-gateway/src/infrastructure/grpc_retry.rs:63`

```rust
pub async fn with_retry<F, Fut, T>(..., max_attempts: u32) -> Result<T, Status> {
    for attempt in 1..=max_attempts {
        // ... リトライロジック ...
    }
    // max_attempts = 0 の場合、ここに到達してパニック
    unreachable!("with_retry: max_attempts must be >= 1")
}
```

**問題**: `max_attempts` の型が `u32` であり、コンパイル時に `>= 1` を保証できない。`max_attempts = 0` が渡された場合、実行時パニックが発生する。
**修正案**: `max_attempts: NonZeroU32` 型を使用するか、関数先頭で `if max_attempts == 0 { return Err(...) }` を追加する。

---

### 6.2 HIGH 問題

#### [HIGH-CODE-01] auth: Keycloak トークン expires_in のデフォルトフォールバック

**ファイル**: `regions/system/server/rust/auth/src/infrastructure/keycloak_client.rs:194`

```rust
let expires_in = body["expires_in"].as_i64().unwrap_or(300);
```

**問題**: Keycloak レスポンスのパースエラー時に 300 秒（5 分）のデフォルトが使用される。Keycloak の実際のトークン有効期限と乖離が生じた場合、期限切れのトークンがキャッシュされ続ける。
**修正案**: パース失敗時に警告ログを出力し、エラーとして伝播させるか、デフォルト値使用時に必ず warn! を出力する。

---

#### [HIGH-CODE-02] proto enum 型変更に実装が追従していない（4 サービス）

上記 BUILD エラーの根本原因として、proto の `String` → `i32`（enum）型変更が複数サービスで未対応。これは Proto スキーマとサービス実装の同期プロセスに問題があることを示す。
**対応**: Proto 型変更時の影響サービス一覧を自動生成するプロセス（`buf breaking` + 影響サービスの自動テスト）を CI に組み込む。

---

#### [HIGH-CODE-03] deprecated proto フィールドの多用（auth/graphql-gateway）

**ファイル**: `regions/system/server/rust/graphql-gateway/src/infrastructure/grpc/auth_client.rs`
**問題**: `RecordAuditLogRequest::event_type/result`、`AuditLog::event_type/result` 等の deprecated フィールドを 17 箇所で使用。ADR-0034 の dual-write パターン対応は実施済みだが、deprecated フィールドからの移行完了期限が未設定。

### 6.3 MEDIUM 問題

#### [MEDIUM-CODE-01] Service 層の Outbox 実装が未移行（3 サービス）

## 7. CLI / GUI 審査

### 7.1 CRITICAL 問題

#### [CRITICAL-GUI-01] Tauri CSP に unsafe-inline が許可

**ファイル**: `CLI/crates/k1s0-gui/tauri.conf.json` (L19)

```json
"csp": "default-src 'self'; script-src 'self'; style-src 'self' 'unsafe-inline'"
```

**問題**: `style-src 'unsafe-inline'` により CSS Injection 攻撃の表面積が広がる。スタイルベースのサイドチャネル攻撃（CSS Exfiltration）が可能になる。
**修正案**: `unsafe-inline` を削除し、必要なスタイルは外部 CSS ファイルとして `'self'` から読み込む設計に変更する。
